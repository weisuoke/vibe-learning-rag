# 核心概念1：节点函数签名与状态接收

> LangGraph 中每个节点本质上就是一个函数。理解它的签名规则——接收什么、返回什么——是掌握状态转换的第一步。

---

## 双重类比：先建立直觉

### 前端类比：React 组件接收 Props

```
React 组件                          LangGraph 节点
─────────────                       ─────────────
function Button(props) {            def my_node(state):
  const { label, onClick } = props    query = state["query"]
  // 处理逻辑                          result = search(query)
  return <button>{label}</button>     return {"result": result}
}
```

- `props` 对应 `state`：框架自动注入，你只管用
- `return JSX` 对应 `return dict`：只描述"变了什么"，框架负责合并
- 组件不需要返回整个页面，节点也不需要返回完整状态

### 日常生活类比：工厂流水线工位

想象一条汽车装配流水线：

1. 每个工位（节点）收到一辆**半成品车**（完整 State）
2. 工人只关心自己负责的部分——比如"装轮胎"工位只看底盘
3. 装完轮胎后，工人只汇报"轮胎已装好"（部分更新），不需要重新描述整辆车
4. 传送带（框架）自动把更新合并到车上，送往下一个工位

## 一、节点函数的三种基本签名

### 签名1：最基础——只接收 State

```python
from typing_extensions import TypedDict

class State(TypedDict):
    query: str
    result: str

# 最简单的节点函数：接收 State，返回 dict
def search_node(state: State) -> dict:
    """检索节点：读取 query，返回检索结果"""
    query = state["query"]           # 从完整状态中取需要的字段
    result = do_search(query)        # 执行业务逻辑
    return {"result": result}        # 只返回变化的字段！
```

关键规则：
- 输入：框架自动注入**完整的** State 对象（所有字段都能访问）
- 输出：返回一个 **dict**，只包含需要更新的字段
- 未返回的字段（如 `query`）保持不变，框架不会动它

### 签名2：带 RunnableConfig——获取运行时配置

```python
from langchain_core.runnables import RunnableConfig

def llm_node(state: State, config: RunnableConfig) -> dict:
    """LLM 节点：通过 config 获取运行时参数"""
    # config 中可以携带 API key、模型名称、用户 ID 等运行时信息
    model_name = config.get("configurable", {}).get("model", "gpt-4")
    temperature = config.get("configurable", {}).get("temperature", 0.7)

    response = call_llm(
        messages=state["messages"],
        model=model_name,
        temperature=temperature,
    )
    return {"messages": [response]}
```

`config` 是可选的第二参数：
- 不写也行，框架会自动检测函数签名来决定是否传入
- 常用于传递 `configurable` 字典（API key、模型选择、用户上下文等）
- 在 `graph.invoke(input, config={"configurable": {...}})` 时传入

### 签名3：异步签名——用于 I/O 密集型操作

```python
import httpx

async def async_search_node(state: State) -> dict:
    """异步检索节点：适合网络请求等 I/O 操作"""
    async with httpx.AsyncClient() as client:
        resp = await client.get(f"https://api.example.com/search?q={state['query']}")
        data = resp.json()
    return {"result": data["answer"]}

# 也可以同时带 config
async def async_llm_node(state: State, config: RunnableConfig) -> dict:
    """异步 LLM 节点"""
    response = await async_call_llm(state["messages"])
    return {"messages": [response]}
```

异步节点的规则和同步完全一样，只是加了 `async/await`。LangGraph 会自动识别并在异步事件循环中执行。

### 三种签名速查表

| 签名 | 适用场景 | 示例 |
|------|----------|------|
| `def f(state) -> dict` | 简单的数据处理、转换 | 文本清洗、格式化 |
| `def f(state, config) -> dict` | 需要运行时配置 | LLM 调用、API 请求 |
| `async def f(state) -> dict` | I/O 密集型操作 | 网络请求、数据库查询 |

## 二、状态接收机制：框架在幕后做了什么？

节点函数拿到的 `state` 参数不是凭空出现的。LangGraph 在编译和执行时有一套精密的机制来组装状态。

### 2.1 编译阶段：Schema → Channel 映射

当你调用 `graph.compile()` 时，`StateGraph` 会解析你的 State 类型定义，为每个字段创建对应的 Channel（通道）。

```python
from typing import Annotated
from operator import add

class State(TypedDict):
    query: str                              # → LastValue Channel（默认）
    messages: Annotated[list, add]          # → BinaryOperatorAggregate Channel（有 reducer）
    result: str                             # → LastValue Channel（默认）
```

源码中的关键函数 `_get_channels()`（位于 `graph/state.py`）：

```python
# 源码简化版 —— graph/state.py 第 1558 行
def _get_channels(schema):
    """从 State 类型定义中提取 Channel 配置"""
    type_hints = get_type_hints(schema, include_extras=True)
    all_keys = {
        name: _get_channel(name, typ)    # 为每个字段创建 Channel
        for name, typ in type_hints.items()
    }
    return (
        {k: v for k, v in all_keys.items() if isinstance(v, BaseChannel)},  # 普通 Channel
        {k: v for k, v in all_keys.items() if is_managed_value(v)},         # 托管值
        type_hints,
    )
```

每个字段的 Channel 类型由 `_get_channel()` 决定：

```python
# 源码简化版 —— graph/state.py 第 1593 行
def _get_channel(name, annotation):
    """为单个字段创建 Channel"""
    # 1. 检查是否有显式 Channel 注解
    if channel := _is_field_channel(annotation):
        return channel
    # 2. 检查是否有 reducer（二元运算符）
    if channel := _is_field_binop(annotation):
        return channel
    # 3. 默认：LastValue（只保留最后一个值）
    return LastValue(annotation)
```

### 2.2 执行阶段：Channel → State 组装

节点执行前，框架通过 `ChannelRead`（位于 `pregel/_read.py`）从所有 Channel 中读取当前值，组装成完整的 State 对象。

```python
# 源码简化版 —— pregel/_read.py 第 23 行
class ChannelRead(RunnableCallable):
    """从 Channel 中读取状态值"""

    channel: str | list[str]    # 要读取的 Channel 名称

    @staticmethod
    def do_read(config, *, select, fresh=False, mapper=None):
        """执行读取操作"""
        read = config[CONF][CONFIG_KEY_READ]   # 获取读取函数
        if mapper:
            return mapper(read(select, fresh))  # 可选：对读取结果做转换
        else:
            return read(select, fresh)          # 直接返回 Channel 值
```

整个流程可以用一张图来理解：

```
编译阶段                              执行阶段
────────                              ────────
State TypedDict                       Channel 存储
┌─────────────┐                       ┌──────────────────┐
│ query: str  │ ──→ LastValue ──────→ │ query: "RAG是什么" │
│ messages: ↓ │ ──→ BinOpAgg ──────→ │ messages: [...]   │ ──→ 组装 ──→ state
│ result: str │ ──→ LastValue ──────→ │ result: ""        │
└─────────────┘                       └──────────────────┘
                                              ↑
                                        ChannelRead 负责
```

## 三、返回值处理：节点输出如何写回状态？

### 3.1 核心规则：只返回变化的字段

```python
class State(TypedDict):
    query: str
    context: list[str]
    answer: str

def retrieve_node(state: State) -> dict:
    docs = search(state["query"])
    # 只返回 context，query 和 answer 不受影响
    return {"context": docs}

def generate_node(state: State) -> dict:
    answer = llm(state["query"], state["context"])
    # 只返回 answer，query 和 context 不受影响
    return {"answer": answer}
```

这就是源码注释中说的 `State -> Partial<State>` 签名——节点返回的是状态的"部分更新"，不是完整状态。

### 3.2 返回 None 等同于无更新

```python
def conditional_node(state: State) -> dict | None:
    """条件节点：某些情况下不需要更新状态"""
    if state["query"] == "":
        return None          # 等同于 return {}，不更新任何字段
    return {"result": process(state["query"])}
```

### 3.3 ChannelWrite：写入的幕后机制

节点返回 dict 后，框架通过 `ChannelWrite`（位于 `pregel/_write.py`）将更新写入对应的 Channel。

```python
# 源码简化版 —— pregel/_write.py 第 46 行
class ChannelWrite(RunnableCallable):
    """将节点输出写入 Channel"""

    writes: list[ChannelWriteEntry | Send]

    @staticmethod
    def do_write(config, writes):
        """执行写入操作"""
        # 1. 验证写入合法性
        for w in writes:
            if isinstance(w, ChannelWriteEntry):
                if w.channel == TASKS:
                    raise InvalidUpdateError("Cannot write to reserved channel")
        # 2. 组装写入元组
        write = config[CONF][CONFIG_KEY_SEND]
        write(_assemble_writes(writes))    # 将 dict 拆成 (channel, value) 对
```

写入时的数据流：

```
节点返回                    框架处理                     Channel 更新
────────                    ────────                     ────────────
{"context": docs}  ──→  ChannelWrite  ──→  [("context", docs)]  ──→  Channel["context"].update(docs)
                         拆成元组              写入对应 Channel          触发 reducer（如果有）
```

### 3.4 写入后的 Reducer 处理

写入 Channel 后，Channel 的 `update()` 方法决定如何处理新值：

```python
# 无 reducer 的字段（LastValue Channel）
# → 直接覆盖，且每步只能收到一个值
class LastValue:
    def update(self, values):
        if len(values) != 1:
            raise InvalidUpdateError("Can receive only one value per step")
        self.value = values[-1]

# 有 reducer 的字段（BinaryOperatorAggregate Channel）
# → 用 reducer 函数合并
class BinaryOperatorAggregate:
    def update(self, values):
        for value in values:
            self.value = self.operator(self.value, value)  # 如 add(old, new)
```

## 四、多种返回类型详解

节点函数不只能返回 `dict`，还支持 `Command` 和 `list[Command]`。

### 4.1 返回 dict（最常见）

```python
def basic_node(state: State) -> dict:
    """最常见的返回方式：只更新状态"""
    return {"messages": [AIMessage(content="Hello!")]}
```

适用场景：纯数据处理，不需要控制路由。

### 4.2 返回 Command（状态更新 + 路由控制）

```python
from langgraph.types import Command
from typing import Literal

def router_node(state: State) -> Command[Literal["search", "generate"]]:
    """同时更新状态并指定下一个节点"""
    if needs_search(state["query"]):
        return Command(
            update={"intent": "search"},    # 更新状态
            goto="search"                    # 指定下一个节点
        )
    else:
        return Command(
            update={"intent": "generate"},
            goto="generate"
        )
```

`Command` 把"更新状态"和"选择路由"合并成一个原子操作：
- `update`：等同于返回 dict 的效果
- `goto`：指定下一个要执行的节点（替代 `add_conditional_edges`）
- 类型注解 `Command[Literal["search", "generate"]]` 帮助静态分析和图可视化

### 4.3 返回 list[Command]（多目标分发）

```python
def fan_out_node(state: State) -> list[Command]:
    """向多个节点同时发送不同的更新"""
    return [
        Command(update={"task": "summarize"}, goto="summarizer"),
        Command(update={"task": "translate"}, goto="translator"),
    ]
```

这相当于"一个节点同时触发多个下游节点"，每个下游节点收到不同的状态更新。

### 返回类型速查表

| 返回类型 | 效果 | 适用场景 |
|----------|------|----------|
| `dict` | 只更新状态 | 纯数据处理 |
| `None` / `{}` | 不更新任何字段 | 条件跳过 |
| `Command` | 更新状态 + 指定路由 | 条件分支 |
| `list[Command]` | 多目标分发 | 并行扇出 |

## 五、完整实战示例：RAG 场景

把上面学到的所有知识串起来，看一个完整的 RAG 图：

```python
from typing import Annotated
from typing_extensions import TypedDict
from langchain_core.runnables import RunnableConfig
from langchain_core.messages import HumanMessage, AIMessage
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.types import Command
from typing import Literal

# ========== 1. 定义状态 ==========
class RAGState(TypedDict):
    """RAG 系统的状态定义"""
    query: str                                          # 用户查询（LastValue）
    messages: Annotated[list, add_messages]              # 消息历史（有 reducer）
    documents: list[str]                                 # 检索到的文档（LastValue）
    answer: str                                          # 生成的回答（LastValue）

# ========== 2. 定义节点函数 ==========

# 签名1：只接收 state —— 简单的检索逻辑
def retrieve(state: RAGState) -> dict:
    """检索节点：根据 query 检索相关文档

    - 接收：完整 RAGState（但只用 query 字段）
    - 返回：只更新 documents 字段
    """
    query = state["query"]
    # 模拟向量检索
    docs = vector_search(query, top_k=3)
    return {"documents": docs}

# 签名2：带 config —— 需要运行时配置的 LLM 调用
def generate(state: RAGState, config: RunnableConfig) -> dict:
    """生成节点：基于检索结果生成回答

    - 接收：完整 RAGState + 运行时配置
    - 返回：更新 answer 和 messages 两个字段
    """
    model = config.get("configurable", {}).get("model", "gpt-4")

    context = "\n".join(state["documents"])
    prompt = f"根据以下资料回答问题：\n{context}\n\n问题：{state['query']}"

    answer = call_llm(prompt, model=model)

    return {
        "answer": answer,
        "messages": [AIMessage(content=answer)],  # 追加到消息历史（触发 add_messages reducer）
    }

# 签名3：返回 Command —— 路由决策
def route_query(state: RAGState) -> Command[Literal["retrieve", "generate"]]:
    """路由节点：决定是否需要检索

    - 接收：完整 RAGState
    - 返回：Command（状态更新 + 路由指令）
    """
    if state.get("documents"):
        # 已有文档，直接生成
        return Command(goto="generate")
    else:
        # 需要先检索
        return Command(goto="retrieve")

# ========== 3. 构建图 ==========
graph = StateGraph(RAGState)

graph.add_node("route", route_query)
graph.add_node("retrieve", retrieve)
graph.add_node("generate", generate)

graph.add_edge(START, "route")
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", END)

app = graph.compile()

# ========== 4. 执行 ==========
result = app.invoke(
    {"query": "什么是 RAG？", "messages": [HumanMessage(content="什么是 RAG？")]},
    config={"configurable": {"model": "gpt-4"}},
)
print(result["answer"])
```

### 数据流追踪

```
invoke() 输入
  ↓
┌─────────────────────────────────────────────────────────┐
│ State: {query: "什么是RAG？", messages: [...], docs: [], answer: ""} │
└─────────────────────────────────────────────────────────┘
  ↓ ChannelRead 组装完整 State
  ↓
route_query(state) → Command(goto="retrieve")
  ↓ 无状态更新，路由到 retrieve
  ↓
┌─────────────────────────────────────────────────────────┐
│ State: {query: "什么是RAG？", messages: [...], docs: [], answer: ""} │
└─────────────────────────────────────────────────────────┘
  ↓ ChannelRead 组装完整 State
  ↓
retrieve(state) → {"documents": ["doc1", "doc2", "doc3"]}
  ↓ ChannelWrite 写入 documents Channel（LastValue 覆盖）
  ↓
┌──────────────────────────────────────────────────────────────────┐
│ State: {query: "什么是RAG？", messages: [...], docs: ["doc1",...], answer: ""} │
└──────────────────────────────────────────────────────────────────┘
  ↓ ChannelRead 组装完整 State
  ↓
generate(state, config) → {"answer": "RAG是...", "messages": [AIMessage(...)]}
  ↓ ChannelWrite 写入 answer Channel（LastValue 覆盖）
  ↓ ChannelWrite 写入 messages Channel（add_messages reducer 追加）
  ↓
┌──────────────────────────────────────────────────────────────────────┐
│ State: {query: "什么是RAG？", messages: [..., AI(...)], docs: [...], answer: "RAG是..."} │
└──────────────────────────────────────────────────────────────────────┘
  ↓
最终输出
```

## 六、常见陷阱与注意事项

### 陷阱1：不要返回完整状态

```python
# 错误写法：返回了所有字段
def bad_node(state: State) -> dict:
    result = process(state["query"])
    return {
        "query": state["query"],    # 多余！未变化的字段不需要返回
        "result": result,
        "context": state["context"], # 多余！
    }

# 正确写法：只返回变化的字段
def good_node(state: State) -> dict:
    result = process(state["query"])
    return {"result": result}        # 只返回变化的部分
```

返回未变化的字段不仅多余，还可能触发 LastValue 的"每步只能收到一个值"限制，导致报错。

### 陷阱2：无 reducer 的字段不能被多个节点同时更新

```python
class State(TypedDict):
    count: int                          # 无 reducer → LastValue Channel

# 如果 node_a 和 node_b 在同一步（并行）都返回 {"count": ...}
# → 报错：InvalidUpdateError: Can receive only one value per step
# 解决方案：给 count 加 reducer
class State(TypedDict):
    count: Annotated[int, lambda a, b: a + b]   # 有 reducer，可以安全并行更新
```

### 陷阱3：config 参数的位置必须是第二个

```python
# 正确：config 是第二个参数
def my_node(state: State, config: RunnableConfig) -> dict: ...

# 错误：config 不能是第一个参数
def my_node(config: RunnableConfig, state: State) -> dict: ...  # 会报错
```

LangGraph 通过检查函数签名来决定是否注入 config，它期望 state 在前、config 在后。

## 一句话总结

> 节点函数签名 `(State) -> dict` 是 LangGraph 的核心契约：框架通过 ChannelRead 组装完整状态交给你，你只需返回变化的字段，框架通过 ChannelWrite 和 Reducer 完成合并——各司其职，互不干扰。
