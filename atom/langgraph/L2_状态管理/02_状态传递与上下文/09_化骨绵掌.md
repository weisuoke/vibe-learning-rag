## 卡片1：直觉理解 - 状态传递的本质

**一句话：** LangGraph 的状态传递就像一个"共享白板"，每个节点都可以在上面读写信息，框架负责协调谁能看到什么、谁能修改什么。

**举例：**

想象一个团队协作场景：
- **白板（State）**：团队共享的工作台，记录当前任务进度
- **团队成员（Node）**：每个人负责一部分工作，完成后在白板上更新
- **协调员（Framework）**：确保更新不会冲突，按规则合并信息

```python
# 白板上的信息
state = {
    "user_query": "什么是 LangGraph？",
    "search_results": [],
    "summary": ""
}

# 成员A：搜索
def search_node(state):
    results = search(state["user_query"])
    return {"search_results": results}  # 只更新自己负责的部分

# 成员B：总结
def summary_node(state):
    summary = summarize(state["search_results"])
    return {"summary": summary}  # 只更新自己负责的部分
```

**应用：** 这种设计让每个节点只关注自己的职责，不需要管理整个状态，大大降低了系统复杂度。

**来源：** `fetch_状态传递_01.md` - Medium: State Management in LangGraph

## 卡片2：形式化定义 - Channel 机制

**一句话：** Channel 是 LangGraph 状态存储的基本单元，每个状态字段对应一个 Channel，负责存储值、更新值、序列化和恢复。

**精确定义：**

```python
# 源码：channels/base.py
class BaseChannel(Generic[Value, Update, Checkpoint]):
    """所有 Channel 的基类"""

    def get(self) -> Value:
        """获取当前值"""
        ...

    def update(self, values: Sequence[Update]) -> bool:
        """更新值（可能有多个更新）"""
        ...

    def checkpoint(self) -> Checkpoint:
        """序列化当前状态"""
        ...

    @staticmethod
    def from_checkpoint(checkpoint: Checkpoint) -> "BaseChannel":
        """从检查点恢复"""
        ...
```

**关键特性：**
1. **泛型设计**：`BaseChannel[Value, Update, Checkpoint]` 支持不同类型
2. **批量更新**：`update()` 接收多个更新，由 Reducer 合并
3. **持久化支持**：`checkpoint()` 和 `from_checkpoint()` 实现状态保存

**应用：** 在 StateGraph 初始化时，每个状态字段都会创建对应的 Channel 实例。

**来源：** `source_状态传递_01.md` - LangGraph 源码分析

## 卡片3：关键概念1 - 依赖注入模式

**一句话：** LangGraph 通过 RunnableConfig 注入读写函数，节点不直接访问状态，而是通过注入的函数读写，实现了读写分离。

**核心机制：**

```python
# 源码：pregel/_read.py
class ChannelRead(RunnableCallable):
    @staticmethod
    def do_read(config: RunnableConfig, *, select: str | list[str], ...):
        # 从 config 中获取注入的读取函数
        read: READ_TYPE = config[CONF][CONFIG_KEY_READ]
        return read(select, fresh)

# 源码：pregel/_write.py
class ChannelWrite(RunnableCallable):
    @staticmethod
    def do_write(config: RunnableConfig, writes: Sequence[...], ...):
        # 从 config 中获取注入的写入函数
        write: TYPE_SEND = config[CONF][CONFIG_KEY_SEND]
        write(_assemble_writes(writes))
```

**为什么这样设计？**
1. **解耦**：节点不依赖具体的状态存储实现
2. **可测试**：可以注入 mock 函数进行单元测试
3. **灵活性**：可以在运行时切换不同的存储后端

**应用：** 这是经典的依赖注入模式，在企业级框架中广泛使用。

**来源：** `source_状态传递_01.md` - LangGraph 源码分析

## 卡片4：关键概念2 - Reducer 合并策略

**一句话：** Reducer 函数定义了如何合并多个状态更新，支持覆盖、累加、自定义合并等策略。

**三种常见策略：**

```python
from typing import Annotated
import operator
from langgraph.graph.message import add_messages

class State(TypedDict):
    # 策略1：覆盖（默认）
    last_result: str  # 最后一个节点的值生效

    # 策略2：累加
    counter: Annotated[int, operator.add]  # 1 + 1 + 1 = 3
    search_results: Annotated[list, operator.add]  # [a] + [b] = [a, b]

    # 策略3：消息追加（智能去重）
    messages: Annotated[list, add_messages]  # 自动处理消息ID

    # 策略4：自定义
    metadata: Annotated[dict, merge_dicts]

def merge_dicts(existing: dict, update: dict) -> dict:
    """自定义合并：保留旧值，添加新值"""
    return {**existing, **update}
```

**工作流程：**
```
节点A返回: {"counter": 1, "last_result": "A"}
节点B返回: {"counter": 2, "last_result": "B"}
节点C返回: {"counter": 3}

最终状态:
{
    "counter": 6,        # 1 + 2 + 3（累加）
    "last_result": "B"   # 最后一个有值的节点（覆盖）
}
```

**应用：** 在多 Agent 系统中，使用不同的 Reducer 策略可以避免状态冲突。

**来源：** `fetch_状态传递_06.md` - Reddit: State Reducers 理解

## 卡片5：编程实现 - 完整状态流转

**一句话：** 从 StateGraph 创建到节点执行，状态经历了创建、读取、更新、合并、触发的完整生命周期。

**完整流程代码：**

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
import operator

# 1. 定义状态 Schema
class WorkflowState(TypedDict):
    query: str
    results: Annotated[list, operator.add]
    summary: str

# 2. 创建图（自动创建 Channels）
graph = StateGraph(WorkflowState)

# 3. 定义节点（读取和写入状态）
def search_node(state: WorkflowState):
    # 读取：通过 ChannelRead 从 Channel 获取
    query = state["query"]
    # 处理
    results = search_api(query)
    # 写入：返回部分更新，通过 ChannelWrite 写入
    return {"results": results}

def summary_node(state: WorkflowState):
    results = state["results"]
    summary = generate_summary(results)
    return {"summary": summary}

# 4. 添加节点和边
graph.add_node("search", search_node)
graph.add_node("summary", summary_node)
graph.add_edge(START, "search")
graph.add_edge("search", "summary")
graph.add_edge("summary", END)

# 5. 编译（绑定 Channel 读写函数）
app = graph.compile()

# 6. 执行（状态流转）
result = app.invoke({"query": "LangGraph 是什么？"})
# 内部流程：
# - 创建初始状态 → 存入 Channels
# - search_node 读取 query → 写入 results
# - Reducer 合并 results（累加）
# - 触发 summary_node
# - summary_node 读取 results → 写入 summary
# - 返回最终状态
```

**关键点：**
- 节点只返回部分更新，不需要管理完整状态
- Reducer 自动合并更新
- 框架保证原子性和一致性

**应用：** 这是构建 LangGraph 应用的标准模式。

**来源：** `context7_langgraph_01.md` - LangGraph 官方文档

## 卡片6：对比区分 - State vs Runtime Context

**一句话：** State 是可变的业务数据，Runtime Context 是不可变的运行环境，两者分离实现了关注点分离。

**对比表格：**

| 维度 | State | Runtime Context |
|------|-------|-----------------|
| **可变性** | 可变（节点可修改） | 不可变（只读） |
| **生命周期** | 随工作流执行变化 | 整个执行过程不变 |
| **内容** | 业务数据（消息、结果） | 配置信息（user_id、连接） |
| **定义方式** | `StateGraph(State)` | `context_schema=Context` |
| **访问方式** | `state["field"]` | `runtime.context.field` |
| **持久化** | 包含在 Checkpoint 中 | 不包含在 Checkpoint 中 |

**代码对比：**

```python
from dataclasses import dataclass
from langgraph.runtime import Runtime

# State：业务数据
class State(TypedDict):
    messages: list
    results: list

# Context：运行环境
@dataclass
class Context:
    user_id: str
    db_connection: Any

def node(state: State, runtime: Runtime[Context]):
    # ✅ 可以修改 state
    return {"results": [new_result]}

    # ❌ 不能修改 context
    # runtime.context.user_id = "new_id"  # TypeError

    # ✅ 可以读取 context
    user_id = runtime.context.user_id
```

**应用：** 在多租户系统中，user_id 放在 Context 中，确保不会被节点意外修改。

**来源：** `context7_langgraph_01.md` - LangGraph 官方文档

## 卡片7：进阶理解 - 多状态 Schema 设计

**一句话：** LangGraph 支持定义多个状态 Schema（InputState、OutputState、PrivateState），实现状态隔离和信息隐藏。

**四种 Schema 类型：**

```python
# 1. InputState：外部输入
class InputState(TypedDict):
    user_input: str

# 2. OutputState：外部输出
class OutputState(TypedDict):
    graph_output: str

# 3. OverallState：内部完整状态
class OverallState(TypedDict):
    foo: str
    user_input: str
    graph_output: str

# 4. PrivateState：节点私有状态
class PrivateState(TypedDict):
    bar: str  # 不暴露给外部

# 节点可以从不同 Schema 读写
def node_1(state: InputState) -> OverallState:
    return {"foo": state["user_input"] + " processed"}

def node_2(state: OverallState) -> PrivateState:
    return {"bar": state["foo"] + " internal"}

def node_3(state: PrivateState) -> OutputState:
    return {"graph_output": state["bar"] + " final"}

# 创建图时指定
builder = StateGraph(
    OverallState,
    input_schema=InputState,
    output_schema=OutputState
)
```

**优势：**
1. **信息隐藏**：PrivateState 不暴露给外部
2. **接口清晰**：InputState 和 OutputState 定义明确的输入输出
3. **灵活组合**：节点可以从不同 Schema 读写

**应用：** 在构建可复用的子图时，使用多状态 Schema 可以隐藏内部实现细节。

**来源：** `context7_langgraph_01.md` - LangGraph 官方文档

## 卡片8：高级应用 - 动态模型选择

**一句话：** 使用 Runtime Context 可以在运行时动态选择 LLM 模型，无需重新编译图。

**完整实现：**

```python
from dataclasses import dataclass
from langchain.chat_models import init_chat_model
from langgraph.graph import MessagesState, StateGraph, START, END
from langgraph.runtime import Runtime

# 1. 定义 Context Schema
@dataclass
class Context:
    model_provider: str = "anthropic"
    temperature: float = 0.7

# 2. 预初始化模型
MODELS = {
    "anthropic": init_chat_model("claude-sonnet-4"),
    "openai": init_chat_model("gpt-4"),
    "local": init_chat_model("ollama/llama3"),
}

# 3. 节点根据 Context 选择模型
def call_model(state: MessagesState, runtime: Runtime[Context]):
    # 动态选择模型
    model = MODELS[runtime.context.model_provider]
    # 使用 Context 中的参数
    response = model.invoke(
        state["messages"],
        temperature=runtime.context.temperature
    )
    return {"messages": [response]}

# 4. 创建图
builder = StateGraph(MessagesState, context_schema=Context)
builder.add_node("model", call_model)
builder.add_edge(START, "model")
builder.add_edge("model", END)
graph = builder.compile()

# 5. 不同用户使用不同模型
# 用户A：使用 Anthropic
response_a = graph.invoke(
    {"messages": [{"role": "user", "content": "hi"}]},
    context=Context(model_provider="anthropic", temperature=0.5)
)

# 用户B：使用 OpenAI
response_b = graph.invoke(
    {"messages": [{"role": "user", "content": "hi"}]},
    context=Context(model_provider="openai", temperature=0.9)
)
```

**优势：**
- 无需为每个模型编译不同的图
- 支持 A/B 测试
- 支持动态降级（主模型失败时切换备用模型）

**应用：** 在生产环境中，可以根据用户等级、成本预算、模型可用性动态选择模型。

**来源：** `context7_langgraph_01.md` - LangGraph 官方文档

## 卡片9：在实际应用中的使用 - 多 Agent 状态管理

**一句话：** 在多 Agent 系统中，使用共享状态 + Reducer 隔离 + 命名空间的混合模式，既保证协作又避免冲突。

**生产级实现：**

```python
from typing import Annotated
import operator

# 混合模式状态设计
class MultiAgentState(TypedDict):
    # 全局共享（所有 Agent 都可以读写）
    messages: Annotated[list, add_messages]
    global_context: dict

    # 命名空间隔离（每个 Agent 管理自己的字段）
    researcher_data: dict
    writer_data: dict
    critic_data: dict

    # 协调字段（主 Agent 管理）
    current_agent: str
    workflow_status: str

# Agent 1：研究员
def researcher_agent(state):
    query = state["messages"][-1].content
    research_results = do_research(query)
    return {
        "researcher_data": {
            "sources": research_results,
            "timestamp": now()
        },
        "messages": [AIMessage(f"研究完成：{len(research_results)} 个来源")]
    }

# Agent 2：写作者
def writer_agent(state):
    sources = state["researcher_data"]["sources"]
    draft = write_article(sources)
    return {
        "writer_data": {
            "draft": draft,
            "word_count": len(draft.split())
        },
        "messages": [AIMessage(f"草稿完成：{len(draft)} 字")]
    }

# Agent 3：评论家
def critic_agent(state):
    draft = state["writer_data"]["draft"]
    critique = review_article(draft)
    return {
        "critic_data": {
            "score": critique["score"],
            "suggestions": critique["suggestions"]
        },
        "messages": [AIMessage(f"评分：{critique['score']}/10")]
    }

# 协调器：决定下一个 Agent
def coordinator(state):
    status = state["workflow_status"]
    if status == "research_done":
        return "writer"
    elif status == "draft_done":
        return "critic"
    else:
        return END
```

**关键设计：**
1. **全局共享**：messages 用于 Agent 间通信
2. **命名空间**：每个 Agent 有自己的数据字段
3. **协调机制**：coordinator 控制流程

**应用：** 这是生产环境中最常用的多 Agent 状态管理模式。

**来源：** `fetch_状态传递_07.md` - Reddit: 多 Agent 状态管理

## 卡片10：总结与延伸 - 上下文工程

**一句话：** LangGraph 的状态传递机制是上下文工程的基础设施，支持 Write、Select、Compress、Isolate 四大策略。

**四大上下文工程策略：**

```python
# 1. Write（写入）：使用 State 作为 Scratchpad
class State(TypedDict):
    scratchpad: list  # 保存中间思考过程

def thinking_node(state):
    thought = "我需要先搜索，然后分析"
    return {"scratchpad": state["scratchpad"] + [thought]}

# 2. Select（选择）：使用 Reducer 选择性保留
class State(TypedDict):
    all_results: Annotated[list, keep_top_k(k=5)]

def keep_top_k(k):
    def reducer(existing, new):
        combined = existing + new
        return sorted(combined, key=lambda x: x.score)[:k]
    return reducer

# 3. Compress（压缩）：使用节点压缩上下文
def compress_node(state):
    long_history = state["messages"]
    summary = summarize(long_history)
    return {"messages": [summary]}  # 替换为摘要

# 4. Isolate（隔离）：使用 Runtime Context 隔离配置
@dataclass
class Context:
    user_id: str  # 隔离用户数据
    tenant_id: str  # 隔离租户数据

def node(state, runtime: Runtime[Context]):
    # 使用隔离的上下文
    namespace = (runtime.context.tenant_id, "memories")
    await runtime.store.aput(namespace, memory_id, data)
```

**延伸学习：**
1. **Checkpoint 机制**：状态持久化和断点续传
2. **子图状态管理**：父图与子图的状态传递
3. **并发控制**：多节点并行执行时的状态一致性
4. **性能优化**：大状态对象的优化策略

**核心价值：**
- LangGraph 的状态传递不是简单的参数传递
- 而是一套完整的上下文工程基础设施
- 支持构建复杂的、可维护的 AI 工作流

**应用：** 理解这些机制后，可以构建生产级的 Agent 系统，处理长对话、多轮交互、复杂决策等场景。

**来源：** `fetch_状态传递_02.md` - LangChain 博客: Context Engineering
