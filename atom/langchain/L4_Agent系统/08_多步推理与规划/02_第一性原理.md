## 多步推理与规划的第一性原理

### 1. 最基础的定义

**多步推理与规划 = 先想清楚要做什么，再一步步去做**

仅此而已！没有更基础的了。

这和人类解决复杂问题的方式完全一样：
- 做饭前先想好菜谱（规划），再按步骤做（执行）
- 旅行前先做攻略（规划），再按行程走（执行）
- 写论文先列提纲（规划），再逐章写（执行）

在 AI Agent 的世界里：
```python
# 规划 = 让 LLM 生成任务步骤列表
plan = planner.invoke("帮我写一篇关于 RAG 的技术报告")
# plan = ["1. 搜索RAG最新论文", "2. 总结核心技术", "3. 写引言", "4. 写主体", "5. 写结论"]

# 执行 = 逐步执行每个步骤
for step in plan:
    result = executor.invoke(step)
```

### 2. 为什么需要多步推理与规划？

**核心问题：复杂任务无法一步完成，需要分解和编排**

想象一下这个场景：

```
用户问："帮我比较 LangChain 和 LlamaIndex 在 RAG 应用中的优劣，
        写一份包含代码示例的对比报告。"
```

这个任务不可能一步完成。Agent 需要：
1. 搜索 LangChain 的 RAG 相关资料
2. 搜索 LlamaIndex 的 RAG 相关资料
3. 分析两者的架构差异
4. 编写 LangChain 的代码示例
5. 编写 LlamaIndex 的代码示例
6. 对比优劣势
7. 整合成完整报告

**没有规划能力的 Agent**：
```
想法: 我应该先搜索一下 → 调用搜索工具 → 得到结果
想法: 我还需要更多信息 → 再搜索 → 得到结果
想法: 好像还不够... → 再搜索...
（经过 15 次 LLM 调用，终于凑够了信息）
```

**有规划能力的 Agent**：
```
规划: [步骤1, 步骤2, 步骤3, 步骤4, 步骤5, 步骤6, 步骤7]
执行步骤1 → 执行步骤2 → ... → 检查 → 输出报告
（只需要 2-3 次 LLM 调用用于规划和检查）
```

### 3. 多步推理与规划的三层价值

#### 价值1：降低成本和延迟

```
ReAct 模式:
  LLM调用 → 工具 → LLM调用 → 工具 → LLM调用 → 工具 → LLM调用 → 最终答案
  （每步都要调用大模型，7次 LLM 调用）

Plan-and-Execute 模式:
  LLM规划 → 工具 → 工具 → 工具 → 工具 → LLM检查 → 最终答案
  （只在规划和检查时调用大模型，2次 LLM 调用）
```

**成本节省**：规划器用大模型（如 GPT-4），执行器用小模型或直接调工具。
大模型调用减少 60-80%，总体成本可降低 50% 以上。

#### 价值2：提升任务完成质量

```
没有全局规划          有全局规划
────────────        ────────────
步骤1 → 步骤2？      步骤1 → 步骤2 → 步骤3 → 步骤4
  ↓                  （一开始就想清楚了所有步骤）
不知道下一步该做什么
  ↓
走弯路、遗漏步骤
```

生成完整推理链是经过验证的提示技巧，能显著提升输出质量。

#### 价值3：可观测性和可控性

```python
# 有了计划，你可以：
plan = agent.get_plan()  # 查看计划
print(f"总共 {len(plan)} 步，当前执行第 {current_step} 步")

# 用户可以审核计划
if user_approves(plan):
    agent.execute(plan)
else:
    agent.replan(user_feedback)  # 根据反馈重新规划
```

### 4. 从第一性原理推导 LangChain 实现

**推理链：**

```
1. 复杂任务需要多个步骤才能完成
   ↓
2. 一次只想一步（ReAct）效率低、质量差
   ↓
3. 需要一个"规划器"先想好所有步骤
   ↓
4. 规划和执行应该分离（不同能力、不同模型）
   ↓
5. 执行过程中可能出错，需要"重新规划"能力
   ↓
6. 重新规划需要知道之前做了什么（需要状态管理）
   ↓
7. 状态管理 + 条件分支 + 循环 = 需要图结构
   ↓
8. LangGraph 的 StateGraph 完美支持这种架构
   ↓
9. Plan-and-Execute Agent = Planner节点 + Executor节点 + Replanner节点
```

在代码层面：

```python
from langgraph.graph import StateGraph

# 推导结果：构建一个有状态的图
graph = StateGraph(PlanExecuteState)

# 规划节点：LLM 生成计划
graph.add_node("planner", plan_step)

# 执行节点：执行当前步骤
graph.add_node("executor", execute_step)

# 重规划节点：检查是否完成，是否需要调整
graph.add_node("replanner", replan_step)

# 执行流程：规划 → 执行 → 重规划 → (循环或结束)
graph.add_edge("planner", "executor")
graph.add_edge("executor", "replanner")
graph.add_conditional_edges("replanner", should_continue)
```

### 5. 一句话总结第一性原理

**多步推理与规划的本质是"先想后做"——让 Agent 用大模型做全局规划，用小模型/工具做具体执行，通过规划-执行-反思-重规划的闭环持续逼近目标。**
