# 节点函数约定 - 概览

## 知识点概述

节点函数是 LangGraph 中最核心的执行单元，定义了图中每个节点的具体行为。理解节点函数的约定是掌握 LangGraph 的关键。

### 什么是节点函数？

节点函数是添加到 StateGraph 中的可执行函数，负责处理状态并返回更新。它类似于前端组件中的事件处理函数，或者工厂流水线上的工序站点。[来源: reference/context7_langgraph_01.md]

### 为什么需要约定？

LangGraph 通过约定优于配置的设计理念，让节点函数的定义更加简洁和一致：
- **自动类型推断**：从函数签名自动推断输入输出类型
- **统一错误处理**：通过 RetryPolicy 提供一致的重试机制
- **异步支持**：自动检测并处理同步/异步函数
- **灵活返回值**：支持多种返回值类型（dict、Command、None 等）

[来源: reference/source_节点函数约定_01.md]

### 核心价值

掌握节点函数约定后，你可以：
1. **快速开发**：遵循约定快速编写节点函数
2. **类型安全**：利用类型推断避免运行时错误
3. **错误恢复**：配置重试策略提高系统鲁棒性
4. **异步优化**：合理使用异步节点提升性能
5. **流程控制**：使用 Command 对象实现复杂路由

## 核心概念列表

### 1. 函数签名约定

节点函数的参数和返回值遵循特定的约定，LangGraph 会根据签名自动推断类型和行为。

**基础签名**：
```python
def node(state: State) -> dict:
    return {"key": "value"}
```

**带 Config 参数**：
```python
def node(state: State, config: RunnableConfig) -> dict:
    # 可以访问配置信息
    return {"key": "value"}
```

**带 Runtime 参数**：
```python
def node(state: State, runtime: Runtime[Context]) -> dict:
    # 可以访问运行时上下文
    return {"key": "value"}
```

**关键点**：
- 第一个参数必须是 State 类型
- LangGraph 自动从第一个参数推断 input_schema
- 支持可选的 config 和 runtime 参数

[来源: reference/source_节点函数约定_01.md]

---

### 2. 状态更新机制

节点函数返回的值会与现有状态合并，支持部分更新而非完整替换。

**部分更新**：
```python
def retrieve(state):
    question = state["question"]
    documents = retriever.invoke(question)
    # 只返回需要更新的字段
    return {"documents": documents}
```

**支持的返回值类型**：
- `dict`：部分状态更新
- `Command`：控制流程跳转
- `None`：不更新状态
- `list[Command | dict]`：多个更新
- 带 `Annotated` 的自定义类型

**Reducer 函数的作用**：
- 定义如何合并新旧值
- 默认行为是直接替换
- 可以自定义合并逻辑（如列表追加）

[来源: reference/source_节点函数约定_01.md, reference/context7_langgraph_01.md]

---

### 3. 异步节点实现

LangGraph 自动检测并处理异步节点，同步函数会被包装成异步执行。

**异步节点定义**：
```python
async def my_async_node(state: State) -> dict:
    result = await some_async_api()
    return {"result": result}
```

**自动检测机制**：
- 检测函数是否为 `async` 函数
- 同步函数使用 `run_in_executor` 在线程池中执行
- 异步函数直接使用 `await` 调用

**注意事项**：
- 推荐使用普通异步函数（`async def`）
- 避免使用异步生成器（`async def` + `yield`），可能有问题
- 处理 `CancelledError` 异常

[来源: reference/source_节点函数约定_01.md, reference/search_节点函数约定_01.md]

---

### 4. 错误处理机制

通过 RetryPolicy 配置重试策略，提供指数退避、抖动等高级功能。

**RetryPolicy 配置**：
```python
from langgraph.types import RetryPolicy

retry_policy = RetryPolicy(
    max_attempts=3,           # 最大重试次数
    initial_interval=1.0,     # 初始重试间隔（秒）
    backoff_factor=2.0,       # 退避因子（指数退避）
    max_interval=10.0,        # 最大重试间隔（秒）
    jitter=True,              # 是否添加随机抖动
    retry_on=Exception        # 重试条件
)
```

**重试条件**：
- 单个异常类型：`retry_on=ValueError`
- 多个异常类型：`retry_on=[ValueError, TypeError]`
- 自定义函数：`retry_on=lambda e: "rate limit" in str(e).lower()`

**指数退避算法**：
```python
interval = min(
    max_interval,
    initial_interval * (backoff_factor ** (attempts - 1))
)
sleep_time = interval + random.uniform(0, 1) if jitter else interval
```

[来源: reference/source_节点函数约定_02.md, reference/context7_langgraph_01.md]

### 5. 节点配置选项

添加节点时可以配置多个选项，控制节点的行为和性能。

**配置选项**：
```python
builder.add_node(
    "my_node",
    my_node_function,
    retry_policy=RetryPolicy(...),    # 重试策略
    cache_policy=CachePolicy(...),    # 缓存策略
    metadata={"description": "..."},  # 元数据
    input_schema=NodeInput,           # 输入 schema
    defer=False,                      # 是否延迟执行
    destinations=["next", "end"]      # 目标节点（用于 Command）
)
```

**常用配置**：
- `retry_policy`：配置重试策略，提高鲁棒性
- `metadata`：添加描述信息，便于调试和文档化
- `input_schema`：显式指定输入类型，增强类型安全
- `destinations`：限制 Command 可以跳转的目标节点

[来源: reference/source_节点函数约定_01.md]

## 实战场景列表

### 场景 1：基础节点函数实现

**场景描述**：实现一个简单的文档检索节点，从向量数据库中检索相关文档。

**核心代码**：
```python
from typing_extensions import TypedDict

class State(TypedDict):
    question: str
    documents: list[str]

def retrieve(state: State) -> dict:
    """检索相关文档"""
    question = state["question"]
    # 调用检索器
    documents = retriever.invoke(question)
    # 只返回需要更新的字段
    return {"documents": documents}
```

**关键点**：
- 使用 TypedDict 定义状态类型
- 只返回需要更新的字段
- 函数签名简洁明了

[来源: reference/context7_langgraph_01.md]

### 场景 2：异步节点实现

**场景描述**：实现一个异步 API 调用节点，调用外部 LLM 服务生成回答。

**核心代码**：
```python
import asyncio
from langchain_openai import ChatOpenAI

async def generate_answer(state: State) -> dict:
    """异步生成回答"""
    question = state["question"]
    documents = state["documents"]

    # 构建 prompt
    context = "\n".join(documents)
    prompt = f"Context: {context}\n\nQuestion: {question}"

    # 异步调用 LLM
    llm = ChatOpenAI()
    response = await llm.ainvoke(prompt)

    return {"answer": response.content}
```

**关键点**：
- 使用 `async def` 定义异步函数
- 使用 `await` 调用异步 API
- LangGraph 自动处理异步执行

[来源: reference/context7_langgraph_01.md, reference/search_节点函数约定_01.md]

### 场景 3：错误处理与重试

**场景描述**：实现一个不稳定的 API 调用节点，配置重试策略处理临时失败。

**核心代码**：
```python
from langgraph.types import RetryPolicy
import random

def retry_on_rate_limit(error: Exception) -> bool:
    """自定义重试条件"""
    return "rate limit" in str(error).lower()

def unreliable_api_call(state: State) -> dict:
    """模拟不稳定的 API 调用"""
    if random.random() < 0.5:
        raise Exception("Rate limit exceeded")
    return {"result": "Success!"}

# 添加节点时配置重试策略
builder.add_node(
    "api_call",
    unreliable_api_call,
    retry_policy=RetryPolicy(
        initial_interval=0.5,
        backoff_factor=2.0,
        max_interval=10.0,
        max_attempts=5,
        jitter=True,
        retry_on=retry_on_rate_limit
    )
)
```

**关键点**：
- 自定义重试条件函数
- 配置指数退避和抖动
- 限制最大重试次数

[来源: reference/source_节点函数约定_02.md, reference/context7_langgraph_01.md]

### 场景 4：Command 对象使用

**场景描述**：实现一个决策节点，根据条件动态跳转到不同的目标节点。

**核心代码**：
```python
from langgraph.types import Command
from typing import Literal

def decide_next_step(state: State) -> Command[Literal["generate", "end"]]:
    """决策下一步"""
    error = state.get("error")
    iterations = state.get("iterations", 0)

    if error == "no" or iterations >= 3:
        # 跳转到结束节点
        return Command(goto="end")
    else:
        # 跳转到生成节点，并更新状态
        return Command(
            goto="generate",
            update={"iterations": iterations + 1}
        )
```

**关键点**：
- 使用 `Command` 对象控制流程
- 使用 `Literal` 类型限制目标节点
- 可以同时更新状态和跳转

[来源: reference/source_节点函数约定_01.md, reference/context7_langgraph_01.md]

### 场景 5：Human-in-the-loop

**场景描述**：实现一个需要人工审批的节点，暂停执行等待用户输入。

**核心代码**：
```python
from langgraph.types import interrupt
from langgraph.checkpoint.memory import MemorySaver

def collect_info(state: State) -> dict:
    """收集用户信息"""
    # 暂停执行，等待用户输入
    name = interrupt("Please enter your name:")
    email = interrupt("Please enter your email:")
    return {"name": name, "email": email}

def confirm_info(state: State) -> dict:
    """确认信息"""
    confirmed = interrupt(
        f"Confirm - Name: {state['name']}, Email: {state['email']} (yes/no)"
    )
    return {"confirmed": confirmed == "yes"}

# 构建图时需要配置 checkpointer
builder = StateGraph(State)
builder.add_node("collect", collect_info)
builder.add_node("confirm", confirm_info)

# 编译时传入 checkpointer
checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)
```

**关键点**：
- 使用 `interrupt()` 函数暂停执行
- 必须配置 checkpointer 保存状态
- 使用 `Command(resume=...)` 恢复执行

[来源: reference/context7_langgraph_01.md]

### 常见误区

1. **"所有节点必须返回完整状态"**
   - 错误：节点只需返回需要更新的字段
   - 正确：利用部分更新机制

2. **"异步生成器可以作为节点函数"**
   - 错误：异步生成器可能有问题
   - 正确：使用普通异步函数

3. **"所有异常都会自动重试"**
   - 错误：需要配置 RetryPolicy
   - 正确：显式配置重试策略

4. **"CancelledError 会被正常捕获"**
   - 错误：需要特殊处理
   - 正确：单独捕获并重新抛出
