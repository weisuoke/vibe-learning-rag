## 卡片 1：直觉理解 - OpenClaw 安装是什么？

### 核心概念

**OpenClaw 安装** = 让 `openclaw` 命令在任何目录都能使用

### 类比理解

**就像在手机上安装 App：**
- 下载 App → `npm install -g openclaw`
- 安装到系统 → 复制到全局 bin 目录
- 图标出现在主屏幕 → 命令添加到 PATH
- 任何时候都能打开 → 任何目录都能运行

### 关键要点

1. **全局安装**：一次安装，随处使用
2. **PATH 配置**：系统自动查找命令
3. **符号链接**：指向实际可执行文件
4. **环境要求**：Node.js 22+

### 验证理解

```bash
# 安装后应该能在任何目录运行
cd ~
openclaw --version  # ✅ 可用

cd /tmp
openclaw --version  # ✅ 可用

cd /any/directory
openclaw --version  # ✅ 可用
```

### 一句话总结

**OpenClaw 安装就是把可执行文件放到 PATH 目录，让系统能在任何位置找到并运行它。**

## 卡片 2：形式化定义 - 安装的三要素

### 核心概念

OpenClaw 安装包含三个必需要素：

### 1. 可执行文件

```typescript
// package.json
{
  "bin": {
    "openclaw": "./dist/cli.js"
  }
}

// dist/cli.js (第一行)
#!/usr/bin/env node
```

**作用：** 告诉系统用 Node.js 执行这个文件

### 2. 全局目录

```bash
# npm 全局目录
/usr/local/lib/node_modules/openclaw/

# 全局 bin 目录
/usr/local/bin/
```

**作用：** 存储全局安装的包

### 3. PATH 环境变量

```bash
# 查看 PATH
echo $PATH
# /usr/local/bin:/usr/bin:/bin

# 系统在 PATH 中查找命令
which openclaw
# /usr/local/bin/openclaw
```

**作用：** 告诉系统在哪些目录查找命令

### 三要素关系

```
可执行文件 (openclaw)
    ↓ (npm install -g)
复制到全局目录 (/usr/local/lib/node_modules/openclaw/)
    ↓ (创建符号链接)
链接到 bin 目录 (/usr/local/bin/openclaw)
    ↓ (bin 目录在 PATH 中)
系统能找到命令 (openclaw --version)
```

### 一句话总结

**安装 = 可执行文件 + 全局目录 + PATH 配置，三者缺一不可。**

## 卡片 3：npm 全局安装机制

### 核心概念

npm 全局安装的 6 个步骤：

### 完整流程

```typescript
// 1. 下载包
npm registry → openclaw@latest.tgz

// 2. 解压到全局目录
/usr/local/lib/node_modules/openclaw/

// 3. 读取 package.json
{
  "bin": { "openclaw": "./dist/cli.js" }
}

// 4. 创建符号链接
/usr/local/bin/openclaw → ../lib/node_modules/openclaw/dist/cli.js

// 5. 设置可执行权限
chmod +x /usr/local/bin/openclaw

// 6. 验证安装
openclaw --version
```

### 关键机制

**符号链接（Symbolic Link）：**
```bash
# 查看符号链接
ls -l $(which openclaw)
# lrwxr-xr-x  1 user  admin  ../lib/node_modules/openclaw/dist/cli.js

# 符号链接的优势
# 1. 节省磁盘空间
# 2. 更新时只需更新源文件
# 3. 多个命令可以指向同一个文件
```

### 常见问题

**Q: 为什么用符号链接而不是复制文件？**
A: 符号链接节省空间，更新时只需更新一处

**Q: 为什么需要 Shebang（#!/usr/bin/env node）？**
A: 告诉操作系统用哪个解释器执行文件

### 一句话总结

**npm 全局安装 = 下载 → 解压 → 创建符号链接 → 添加到 PATH。**

## 卡片 4：pnpm vs npm 全局安装

### 核心概念

pnpm 使用**内容寻址存储**和**硬链接**，比 npm 更高效

### 存储机制对比

**npm：**
```
每个项目独立存储
/usr/local/lib/node_modules/openclaw/  (完整副本)
/project-a/node_modules/openclaw/      (完整副本)
/project-b/node_modules/openclaw/      (完整副本)
```

**pnpm：**
```
全局存储一份，项目使用硬链接
~/.pnpm-store/v3/files/openclaw@1.2.3/  (唯一副本)
/usr/local/lib/node_modules/openclaw/  (硬链接)
/project-a/node_modules/openclaw/      (硬链接)
/project-b/node_modules/openclaw/      (硬链接)
```

### 性能对比

| 指标 | npm | pnpm |
|------|-----|------|
| 磁盘占用（10个项目） | 4.87 GB | 612 MB |
| 安装速度（中型项目） | 46.1s | 12.4s |
| 节省空间 | - | 87% |

### 硬链接 vs 符号链接

```typescript
// 硬链接（pnpm）
// - 指向文件内容（inode）
// - 删除源文件，硬链接仍然有效
// - 不占用额外磁盘空间

// 符号链接（npm）
// - 指向文件路径
// - 删除源文件，符号链接失效
// - 占用少量磁盘空间
```

### 推荐使用

```bash
# 推荐：pnpm（更快、更省空间）
pnpm add -g openclaw@latest

# 备选：npm（最大兼容性）
npm install -g openclaw@latest
```

### 一句话总结

**pnpm 通过内容寻址和硬链接，比 npm 节省 87% 磁盘空间，安装速度快 3-4 倍。**

### 核心概念

OpenClaw 需要 Node.js 22+ 的 4 个关键特性

### 1. ESM 同步 require()（稳定）

```javascript
// Node.js 20: 需要 flag
node --experimental-require-module app.js

// Node.js 22: 直接支持
const esmModule = require('./esm-module.mjs');
```

**OpenClaw 使用：** 加载 ESM 配置文件

### 2. 内置 WebSocket（默认启用）

```javascript
// Node.js 20: 需要 flag 或外部库
const WebSocket = require('ws');

// Node.js 22: 内置支持
const ws = new WebSocket('ws://localhost:3000');
```

**OpenClaw 使用：** Gateway 实时通信

### 3. Maglev 编译器（性能提升）

```
CLI 启动时间对比：
Node.js 20: 450ms
Node.js 22: 315ms
性能提升: 30%
```

**OpenClaw 使用：** 提升 CLI 启动速度

### 4. V8 引擎 12.4（新语法）

```javascript
// Array.fromAsync()
const arr = await Array.fromAsync(asyncGenerator());

// Set methods
const union = setA.union(setB);
const intersection = setA.intersection(setB);
```

**OpenClaw 使用：** 现代 JavaScript 特性

### 版本检查

```typescript
const nodeVersion = process.version;
const majorVersion = parseInt(nodeVersion.slice(1).split('.')[0]);

if (majorVersion < 22) {
  console.error('需要 Node.js 22+');
  process.exit(1);
}
```

### 一句话总结

**Node.js 22+ 提供 ESM 支持、内置 WebSocket、Maglev 编译器和 V8 12.4，是 OpenClaw 的硬性要求。**

## 卡片 6：~/.openclaw/ 目录结构

### 核心概念

OpenClaw 配置存储在用户主目录的 `.openclaw/` 目录

### 标准目录结构

```
~/.openclaw/
├── settings.json          # 主配置文件（必需）
├── gateway.log            # Gateway 日志
├── sessions/              # 会话数据
│   ├── session-001.jsonl
│   └── session-002.jsonl
├── cache/                 # 缓存目录
│   ├── responses/
│   └── embeddings/
└── plugins/               # 插件目录
```

### settings.json 结构

```json
{
  "gateway": {
    "port": 3000,
    "host": "localhost",
    "mode": "development",
    "logLevel": "info"
  },
  "channels": {
    "claude": {
      "apiKey": "sk-ant-api03-...",
      "enabled": true,
      "model": "claude-3-5-sonnet-20241022"
    }
  }
}
```

### 文件权限

```bash
# 配置文件权限（只有用户可读写）
chmod 600 ~/.openclaw/settings.json

# 目录权限（只有用户可访问）
chmod 700 ~/.openclaw/
```

### 为什么在用户主目录？

1. **安全性**：保护 API keys 等敏感数据
2. **持久化**：配置在重启后仍然有效
3. **跨项目**：不同项目共享配置
4. **标准化**：遵循 Unix 惯例（~/.config/）

### 一句话总结

**~/.openclaw/ 存储配置、日志和缓存，文件权限保护敏感数据，遵循 Unix 标准。**

## 卡片 7：Onboarding Wizard 流程

### 核心概念

Onboarding Wizard 是交互式配置向导，简化首次配置

### 完整流程

```
openclaw onboard
    ↓
1. 询问 Claude API Key
   → 验证格式（sk-ant-api03-...）
   → 保存到配置
    ↓
2. 询问 Gateway 端口
   → 验证范围（1024-65535）
   → 默认值：3000
    ↓
3. 询问启用的通道
   → Claude（必需）
   → GPT（可选）
   → Gemini（可选）
    ↓
4. 生成配置文件
   → 创建 ~/.openclaw/settings.json
   → 设置文件权限（600）
    ↓
5. 提示下一步
   → openclaw gateway start
```

### 自动化配置

```typescript
// 使用环境变量（非交互式）
export CLAUDE_API_KEY=sk-ant-api03-...
export OPENCLAW_PORT=3000

// 运行配置
openclaw onboard --env
```

### 配置优先级

```
1. 命令行参数（最高）
   openclaw gateway start --port 4000

2. 环境变量
   OPENCLAW_PORT=4000

3. 配置文件
   ~/.openclaw/settings.json

4. 默认值（最低）
   port: 3000
```

### 一句话总结

**Onboarding Wizard 通过交互式提示简化配置，支持环境变量自动化，遵循配置优先级。**

## 卡片 8：Gateway 启动模式

### 核心概念

OpenClaw Gateway 支持两种启动模式：前台和守护进程

### 前台模式（开发调试）

```bash
# 启动
openclaw gateway start

# 特点
# - 日志实时显示在终端
# - Ctrl+C 停止
# - 占用终端窗口
# - 适合开发调试
```

**使用场景：**
- 本地开发
- 调试问题
- 查看实时日志

### 守护进程模式（生产环境）

```bash
# 启动
openclaw gateway start --daemon
# 或
openclaw gateway start -d

# 特点
# - 后台运行
# - 日志写入文件
# - 不占用终端
# - 适合生产环境

# 管理
openclaw gateway stop    # 停止
openclaw gateway status  # 状态
tail -f ~/.openclaw/gateway.log  # 查看日志
```

**使用场景：**
- 服务器部署
- 长期运行
- CI/CD 环境

### 进程管理

```typescript
// 检查 Gateway 是否运行
function isGatewayRunning(): boolean {
  try {
    execSync('pgrep -f "openclaw gateway"', { stdio: 'ignore' });
    return true;
  } catch {
    return false;
  }
}

// 获取进程 PID
const pid = execSync('pgrep -f "openclaw gateway"', {
  encoding: 'utf-8'
}).trim();
```

### 健康检查

```bash
# 检查 Gateway 状态
curl http://localhost:3000/health

# 预期响应
{
  "status": "ok",
  "version": "1.2.3",
  "uptime": 3600
}
```

### 一句话总结

**前台模式用于开发调试，守护进程模式用于生产环境，通过健康检查验证状态。**

## 卡片 9：环境变量优先级

### 核心概念

OpenClaw 支持多层配置，按优先级合并

### 配置优先级（从高到低）

```
1. 命令行参数（最高优先级）
   openclaw gateway start --port 4000

2. 环境变量
   export OPENCLAW_PORT=4000

3. 项目配置文件
   ./.openclawrc

4. 用户配置文件
   ~/.openclaw/settings.json

5. 默认值（最低优先级）
   { port: 3000 }
```

### 实际示例

```typescript
// 配置合并逻辑
const config = {
  ...defaultConfig,           // 5. 默认值
  ...loadUserConfig(),        // 4. 用户配置
  ...loadProjectConfig(),     // 3. 项目配置
  ...loadEnvConfig(),         // 2. 环境变量
  ...loadCLIArgs()            // 1. 命令行参数
};
```

### 常用环境变量

```bash
# API Keys
export CLAUDE_API_KEY=sk-ant-api03-...
export OPENAI_API_KEY=sk-...
export GEMINI_API_KEY=...

# Gateway 配置
export OPENCLAW_PORT=3000
export OPENCLAW_HOST=localhost
export OPENCLAW_LOG_LEVEL=info

# 运行模式
export NODE_ENV=production
```

### 多环境配置

```bash
# development.env
CLAUDE_API_KEY=sk-ant-api03-dev-...
OPENCLAW_PORT=3000
NODE_ENV=development

# production.env
CLAUDE_API_KEY=sk-ant-api03-prod-...
OPENCLAW_PORT=8080
NODE_ENV=production

# 使用
source development.env && openclaw gateway start
source production.env && openclaw gateway start
```

### 一句话总结

**配置优先级：命令行 > 环境变量 > 项目配置 > 用户配置 > 默认值，高优先级覆盖低优先级。**

## 卡片 10：故障诊断工具

### 核心概念

系统化诊断 OpenClaw 安装和运行问题

### 诊断检查清单

```typescript
// 1. 环境检查
✓ Node.js 版本 >= 22.0.0
✓ npm/pnpm 可用
✓ openclaw 命令可用

// 2. 配置检查
✓ ~/.openclaw/settings.json 存在
✓ 配置文件格式正确
✓ API keys 已配置

// 3. Gateway 检查
✓ Gateway 进程运行中
✓ 健康检查通过
✓ 端口未被占用

// 4. 日志检查
✓ 无错误日志
✓ 无警告日志
```

### 常见问题速查

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `openclaw: command not found` | PATH 未配置 | 重新安装或添加 PATH |
| `Node.js version too old` | Node.js < 22 | 升级到 Node.js 22+ |
| `Gateway failed to start` | 端口被占用 | 修改端口或停止占用进程 |
| `API key invalid` | 配置错误 | 重新运行 `openclaw onboard` |
| `Connection refused` | Gateway 未运行 | 运行 `openclaw gateway start` |

### 自动化诊断脚本

```bash

npx tsx test-openclaw.ts --diagnose

Gateway: ✅
  Running: Yes
  Healthy: Yes

Configuration: ✅
  Valid: Yes
  Channels: claude, gpt

Recommendations:
  (无问题)
```

### 日志分析

```bash
# 查看最近日志
tail -f ~/.openclaw/gateway.log

# 搜索错误
grep -i error ~/.openclaw/gateway.log

# 搜索警告

```

### 一句话总结

**系统化诊断：检查环境 → 验证配置 → 测试 Gateway → 分析日志，快速定位问题。**
