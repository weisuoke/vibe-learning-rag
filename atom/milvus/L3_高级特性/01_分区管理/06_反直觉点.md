# 反直觉点

> 避免 Milvus 分区管理的常见误区

---

## 概述

本文列出**3个最常见的误区**，帮助你避免在使用 Milvus 分区管理时犯错。

---

## 误区 1：分区越多越好 ❌

### 错误观点

"创建更多的分区可以提升性能，所以分区越多越好。"

### 为什么错？

**核心问题**：
- 分区过多会增加管理开销
- 每个分区都需要独立的元数据管理
- 分区过多会导致性能下降

**来源**：GitHub 最佳实践

**关键数据**：
- **推荐分区数**：100-200 个（Partition Key）
- **最大分区数**：1,024 个（手动分区）
- **总数控制**：建议控制在 1,000 以内

### 为什么人们容易这样错？

**心理原因**：
- 直觉认为"分区越多，数据越分散，性能越好"
- 类比数据库分表，认为分表越多越好
- 忽略了管理开销

### 正确理解

**平衡性能和管理开销**：

```python
# ❌ 错误：创建过多分区
for i in range(10000):
    collection.create_partition(f"partition_{i}")
# 问题：管理开销大，性能下降

# ✅ 正确：使用 Partition Key 自动分区
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True)  # 自动分区
    ]
)
collection = Collection("my_collection", schema)

# 配置合理的物理分区数
collection = Collection(
    name="my_collection",
    schema=schema,
    properties={"partition.num": 100}  # 100 个物理分区（推荐）
)
```

**关键建议**：
- **Partition Key**：推荐 100-200 个物理分区
- **手动分区**：控制在 1,000 以内
- **分区大小**：每个分区 20-100K 行

## 误区 2：Partition Key 会降低性能 ❌

### 错误观点

"Partition Key 自动分区会降低性能，手动分区更快。"

### 为什么错？

**核心问题**：
- Partition Key 通过自动路由避免 99% 无效扫描
- 延迟从秒级降至毫秒级
- 性能提升显著

**来源**：Twitter/X 性能优化数据

**关键数据**：
- **99% 无效扫描避免**：使用 Partition Key 可以避免扫描 99% 的无关数据
- **延迟优化**：多租户搜索延迟从秒级降至毫秒级
- **性能提升**：100 倍

### 为什么人们容易这样错？

**心理原因**：
- 认为"自动"意味着"不够优化"
- 担心哈希算法的开销
- 忽略了自动路由的性能优势

### 正确理解

**Partition Key 性能对比**：

```python
# 场景：1000 个租户，每个租户 10 万条数据，总数据量 1 亿条

# ❌ 不使用 Partition Key：扫描 1 亿条数据
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "L2", "params": {"nprobe": 10}},
    limit=10,
    expr='tenant_id == "tenant_a"'  # 需要扫描 1 亿条数据
)
# 延迟：> 1 秒

# ✅ 使用 Partition Key：只扫描 10 万条数据
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True)
    ]
)
collection = Collection("my_collection", schema)

results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "L2", "params": {"nprobe": 10}},
    limit=10,
    expr='tenant_id == "tenant_a"'  # 只扫描租户 A 的分区
)
# 延迟：< 10 毫秒
# 性能提升：100 倍
```

**关键建议**：
- **大规模多租户**：推荐使用 Partition Key
- **启用隔离**：配置 `partitionkey.isolation=True`（仅 HNSW）
- **性能监控**：监控延迟和 QPS

## 误区 3：Milvus 支持向量的更新操作 ❌

### 错误观点

"Milvus 支持直接更新向量数据，可以使用 `update()` 方法。"

### 为什么错？

**核心问题**：
- Milvus 不支持直接更新向量数据
- 需要先删除再插入（Delete + Insert）
- 或者使用 Upsert 操作

**来源**：Context7 官方文档

### 为什么人们容易这样错？

**心理原因**：
- 类比关系型数据库的 UPDATE 操作
- 认为向量数据库应该支持更新
- 忽略了向量索引的特殊性

### 正确理解

**更新向量数据的正确方法**：

```python
# ❌ 错误：尝试直接更新
# collection.update(data)  # 不存在这个方法

# ✅ 正确方法 1：先删除再插入
# 删除旧数据
collection.delete(expr="id in [1, 2, 3]")

# 插入新数据
new_data = [
    [1, 2, 3],  # id
    [[0.1]*128, [0.2]*128, [0.3]*128],  # 新的 vector
    ["new_doc1", "new_doc2", "new_doc3"]  # content
]
collection.insert(new_data)

# ✅ 正确方法 2：使用 Upsert
# Upsert 会自动处理删除和插入
new_data = [
    [1, 2, 3],  # id
    [[0.1]*128, [0.2]*128, [0.3]*128],  # 新的 vector
    ["new_doc1", "new_doc2", "new_doc3"]  # content
]
collection.upsert(new_data)
```

**关键建议**：
- **更新数据**：使用 Delete + Insert 或 Upsert
- **批量更新**：使用 Upsert 更高效
- **注意索引**：更新后需要重新建立索引

## 其他常见误区

### 误区 4：Collection 不需要创建索引就能检索 ❌

**错误观点**：
"创建 Collection 后就可以直接检索，不需要创建索引。"

**正确理解**：
- Milvus 需要创建索引才能进行高效检索
- 不创建索引会导致性能极差（暴力搜索）

```python
# ✅ 正确：创建索引
index_params = {
    "index_type": "HNSW",
    "metric_type": "L2",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="vector", index_params=index_params)

# 加载 Collection
collection.load()

# 搜索
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "L2", "params": {"ef": 100}},
    limit=10
)
```

### 误区 5：向量维度可以随时修改 ❌

**错误观点**：
"可以随时修改向量的维度。"

**正确理解**：
- 向量维度在创建 Collection 时固定
- 无法修改已创建 Collection 的向量维度
- 需要重新创建 Collection

```python
# ❌ 错误：尝试修改向量维度
# collection.alter_schema(...)  # 无法修改向量维度

# ✅ 正确：重新创建 Collection
collection.drop()

new_schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=256),  # 新维度
        FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=1024)
    ]
)
collection = Collection("my_collection", new_schema)
```

### 误区 6：Partition Key 隔离支持所有索引类型 ❌

**错误观点**：
"Partition Key 隔离支持所有索引类型。"

**正确理解**：
- Partition Key 隔离当前仅支持 HNSW 索引类型
- 不支持 IVF_FLAT、IVF_SQ8 等其他索引类型

```python
# ❌ 错误：使用 IVF_FLAT 索引 + Partition Key 隔离
collection = Collection(
    name="my_collection",
    schema=schema,
    properties={"partitionkey.isolation": True}
)

index_params = {
    "index_type": "IVF_FLAT",  # 不支持
    "metric_type": "L2",
    "params": {"nlist": 128}
}
collection.create_index(field_name="vector", index_params=index_params)

# ✅ 正确：使用 HNSW 索引 + Partition Key 隔离
collection = Collection(
    name="my_collection",
    schema=schema,
    properties={"partitionkey.isolation": True}
)

index_params = {
    "index_type": "HNSW",  # 支持
    "metric_type": "L2",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="vector", index_params=index_params)
```

## 核心要点总结

1. **分区越多越好 ❌**：分区过多会增加管理开销，推荐 100-200 个物理分区
2. **Partition Key 会降低性能 ❌**：Partition Key 通过自动路由避免 99% 无效扫描，性能提升显著
3. **Milvus 支持向量的更新操作 ❌**：需要使用 Delete + Insert 或 Upsert
4. **Collection 不需要创建索引就能检索 ❌**：需要创建索引才能进行高效检索
5. **向量维度可以随时修改 ❌**：向量维度在创建 Collection 时固定
6. **Partition Key 隔离支持所有索引类型 ❌**：当前仅支持 HNSW 索引类型
