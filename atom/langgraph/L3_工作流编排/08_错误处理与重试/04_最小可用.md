## 你只需要记住这 5 件事

### 1. 给节点加重试，一行搞定

```python
from langgraph.types import RetryPolicy

# 默认配置：最多3次尝试，指数退避，智能异常过滤
builder.add_node("my_node", my_function, retry_policy=RetryPolicy())
```

**就这一行，你的节点就有了：**
- 自动重试网络错误和服务端错误
- 不重试编程错误（ValueError、TypeError）
- 指数退避（0.5s → 1s → 2s）防止压垮服务器
- 随机抖动防止多个节点同时重试

### 2. 三个最常调整的参数

```python
# 90% 的场景只需要调这三个
RetryPolicy(
    max_attempts=5,          # 总尝试次数（含首次），默认3
    initial_interval=1.0,    # 首次重试等待秒数，默认0.5
    retry_on=ConnectionError # 只重试这种错误，默认智能判断
)
```

### 3. 不同节点用不同策略

```python
# LLM 调用：多试几次，间隔长一点
builder.add_node("call_llm", call_llm,
    retry_policy=RetryPolicy(max_attempts=5, initial_interval=1.0))

# 数据库查询：少试几次，间隔短
builder.add_node("query_db", query_db,
    retry_policy=RetryPolicy(max_attempts=2, initial_interval=0.3))

# 纯计算节点：不需要重试
builder.add_node("process", process_data)  # 不传 retry_policy
```

### 4. 函数式 API 用 @task 装饰器

```python
from langgraph.func import entrypoint, task
from langgraph.types import RetryPolicy

@task(retry_policy=RetryPolicy(max_attempts=3))
def fetch_data(query: str):
    """自动重试的数据获取任务。"""
    return api_call(query)
```

### 5. 自定义"什么错误值得重试"

```python
import httpx

def should_retry(exc: Exception) -> bool:
    """只重试网络错误和 429/5xx。"""
    if isinstance(exc, ConnectionError):
        return True
    if isinstance(exc, httpx.HTTPStatusError):
        return exc.response.status_code in [429, 500, 502, 503, 504]
    return False

policy = RetryPolicy(retry_on=should_retry)
```

## 最小可用模板

**复制这段代码到你的项目中，就有了基本的错误处理能力：**

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy

# 定义状态
class State(TypedDict):
    query: str
    result: str

# 默认重试策略（适用于大多数场景）
default_retry = RetryPolicy()

# LLM 专用策略（更多重试、更长间隔）
llm_retry = RetryPolicy(
    max_attempts=5,
    initial_interval=1.0,
    max_interval=60.0,
)

# 构建图
builder = StateGraph(State)
builder.add_node("retrieve", retrieve_docs, retry_policy=default_retry)
builder.add_node("generate", generate_answer, retry_policy=llm_retry)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

graph = builder.compile()
```

## 常见问题速答

### Q: max_attempts=3 是重试 3 次吗？

**A: 不是！是总共尝试 3 次（1 次执行 + 2 次重试）。**

```
max_attempts=3：
  执行1（首次）→ 失败 → 重试1 → 失败 → 重试2 → 失败 → 抛出异常
  总共 3 次尝试，2 次重试
```

### Q: 不传 retry_policy 会怎样？

**A: 节点不会有任何重试，异常直接抛出。** `RetryPolicy()` 不是默认行为，必须显式配置。

### Q: 所有异常都会被重试吗？

**A: 不是。** 默认的 `default_retry_on` 会排除编程错误（ValueError、TypeError 等），只重试瞬态错误（网络、5xx）。

### Q: GraphBubbleUp / GraphInterrupt 会被重试吗？

**A: 永远不会。** 它们是流程控制信号，不是错误。在重试逻辑之前就被直接抛出了。

### Q: 可以给同一个节点配多个重试策略吗？

**A: 可以！传入策略序列，第一个匹配的生效：**

```python
builder.add_node("my_node", func,
    retry_policy=(
        RetryPolicy(retry_on=ConnectionError, max_attempts=5),
        RetryPolicy(retry_on=TimeoutError, max_attempts=3),
    )
)
```

## 一张图总结

```
你的 LangGraph 节点
    │
    ├── 不传 retry_policy → 不重试，异常直接抛出
    │
    └── 传 RetryPolicy() → 启用重试
            │
            ├── 异常是 GraphBubbleUp? → 直接抛出（不重试）
            │
            ├── retry_on 匹配? → 重试
            │     ├── 尝试次数 < max_attempts? → 退避等待 → 再次执行
            │     └── 尝试次数 >= max_attempts? → 抛出异常
            │
            └── retry_on 不匹配? → 直接抛出
```
