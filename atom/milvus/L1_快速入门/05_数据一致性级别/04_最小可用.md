# 最小可用知识

> 掌握这5个核心知识点，解决80%的Milvus一致性级别使用场景

---

## 1. 默认使用Bounded，无需纠结

### 为什么

Milvus 2.6的默认一致性级别是**Bounded**（staleness bound = 0s），这是经过大量生产验证的最佳平衡点。

### 何时使用

- ✅ 大多数生产场景
- ✅ 不确定选哪个时
- ✅ 文档问答、知识检索
- ✅ 推荐系统、搜索排序

### 代码示例

```python
from pymilvus import Collection

# 创建集合时不指定，自动使用Bounded
collection = Collection(
    name="my_collection",
    schema=schema
    # consistency_level默认为"Bounded"
)

# 搜索时也不需要指定
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "L2", "params": {"nprobe": 10}},
    limit=10
    # 使用集合默认的Bounded级别
)
```

### 记住

**不指定 = Bounded = 正确选择**

## 2. 批量导入用Eventually，性能翻倍

### 为什么

批量导入场景的特点：
- 数据更新不频繁（每天/每小时一次）
- 不需要立即查询刚导入的数据
- 性能是首要考虑

Eventually一致性：
- 最低延迟
- 最高吞吐量
- 不等待WAL同步

### 何时使用

- ✅ 批量文档导入
- ✅ 离线数据处理
- ✅ 日志分析
- ✅ 数据迁移

### 代码示例

```python
from pymilvus import Collection, ConsistencyLevel

# 创建用于批量导入的集合
collection = Collection(
    name="batch_import",
    schema=schema,
    consistency_level="Eventually"  # 批量场景推荐
)

# 批量插入
entities = [
    {"vector": embedding, "text": doc}
    for embedding, doc in zip(embeddings, documents)
]
collection.insert(entities)

# 搜索时也使用Eventually
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=10,
    consistency_level="Eventually"  # 覆盖集合级别
)
```

### 性能对比

根据社区测试：
- Eventually vs Strong：延迟降低50-70%
- Eventually vs Bounded：延迟降低20-30%

### 记住

**批量导入 = Eventually = 性能优先**

## 3. 交互式应用用Session，避免困惑

### 为什么

交互式应用的典型模式：
1. 用户上传文档
2. 系统插入向量
3. 用户立即搜索

如果使用Eventually，用户可能搜不到刚上传的文档，造成困惑。

Session一致性保证：
- **同一会话内**的写入立即可见
- 低开销（比Strong快很多）
- 符合用户直觉

### 何时使用

- ✅ 用户上传后立即搜索
- ✅ 聊天机器人（对话历史）
- ✅ 实时协作应用
- ✅ 个人知识库

### 代码示例

```python
from pymilvus import Collection, ConsistencyLevel

# 创建交互式应用的集合
collection = Collection(
    name="user_docs",
    schema=schema,
    consistency_level="Session"  # 交互式应用推荐
)

# 用户上传文档
def upload_document(user_id, document, embedding):
    collection.insert([{
        "user_id": user_id,
        "vector": embedding,
        "text": document
    }])

    # 立即搜索，能看到刚上传的文档
    results = collection.search(
        data=[embedding],
        anns_field="vector",
        param={"metric_type": "COSINE", "params": {"nprobe": 10}},
        limit=10,
        expr=f"user_id == {user_id}",
        consistency_level="Session"  # 确保看到自己的写入
    )
    return results
```

### 注意事项

Session一致性是**会话级别**的：
- 同一个连接内有效
- 不同用户/连接之间不保证
- 适合单用户场景

### 记住

**用户写后立即读 = Session = 避免困惑**

## 4. 避免Strong，除非真的需要

### 为什么

Strong一致性的代价：
- **全索引加载到内存**（内存消耗大）
- **同步等待WAL**（延迟最高）
- **吞吐量降低**（并发受限）

### 何时真的需要

- ✅ 金融交易数据
- ✅ 实时库存管理
- ✅ 关键业务数据
- ✅ 多用户协作（需要全局一致性）

### 何时不需要

- ❌ 一般的RAG应用
- ❌ 文档问答系统
- ❌ 推荐系统
- ❌ 日志分析

### 代码示例

```python
from pymilvus import Collection, ConsistencyLevel

# 只在真正需要时使用Strong
collection = Collection(
    name="financial_data",
    schema=schema,
    consistency_level="Strong"  # 仅用于关键数据
)

# 搜索时也使用Strong
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "L2", "params": {"nprobe": 10}},
    limit=10,
    consistency_level="Strong"
)
```

### 性能影响

根据Reddit社区讨论：
- Strong一致性在大规模数据下可能导致内存不足
- 延迟比Bounded高50-100%
- 吞吐量降低30-50%

### 记住

**Strong = 昂贵 = 仅用于关键数据**

## 5. 可以在搜索时覆盖集合级别

### 为什么

灵活性：
- 集合级别设置默认行为
- 搜索时可以临时调整
- 不同查询有不同需求

### 使用场景

**场景1：默认Eventually，关键查询用Bounded**

```python
# 集合默认Eventually（性能优先）
collection = Collection(
    name="docs",
    schema=schema,
    consistency_level="Eventually"
)

# 普通搜索：使用Eventually
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=10
    # 使用集合默认的Eventually
)

# 关键搜索：临时提升到Bounded
critical_results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=10,
    consistency_level="Bounded"  # 临时覆盖
)
```

**场景2：默认Bounded，批量分析用Eventually**

```python
# 集合默认Bounded（平衡）
collection = Collection(
    name="knowledge_base",
    schema=schema,
    consistency_level="Bounded"
)

# 用户查询：使用Bounded
user_results = collection.search(
    data=[user_query_vector],
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=10
)

# 批量分析：降级到Eventually
analytics_results = collection.search(
    data=batch_query_vectors,
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=100,
    consistency_level="Eventually"  # 批量分析用Eventually
)
```

### 记住

**集合级别 = 默认行为，搜索级别 = 临时覆盖**

## 快速决策树

```
需要选择一致性级别？
│
├─ 是批量导入/分析？
│  └─ 是 → Eventually
│
├─ 用户写后立即读？
│  └─ 是 → Session
│
├─ 金融/关键数据？
│  └─ 是 → Strong
│
└─ 不确定/一般场景？
   └─ Bounded（默认）
```

---

## 总结：80%场景的选择

| 场景 | 推荐级别 | 理由 |
|------|----------|------|
| **默认/不确定** | Bounded | 最佳平衡，生产验证 |
| **批量导入** | Eventually | 性能翻倍 |
| **交互式应用** | Session | 避免用户困惑 |
| **关键数据** | Strong | 准确性优先 |
| **一般RAG** | Bounded | 默认即可 |
