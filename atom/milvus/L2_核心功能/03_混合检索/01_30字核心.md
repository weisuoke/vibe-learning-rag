# 30字核心

> **目标**：用30个字精准概括混合检索的本质

---

## 核心定义（30字）

**混合检索通过多路向量召回（稠密+稀疏）与重排序策略（RRF/Weighted）融合，兼顾语义理解与精确匹配，提升检索质量。**

---

## 核心拆解

### 1. 多路向量召回（10字）

**关键词**：稠密向量、稀疏向量、多模态

**解释**：
- **稠密向量（Dense Vector）**：捕捉语义关系，理解"意思相近"
- **稀疏向量（Sparse Vector）**：精确关键词匹配，理解"词汇相同"
- **多模态**：文本、图像、音频等不同数据类型

**为什么重要**：
单一检索方式有局限性。稠密向量可能忽略精确术语，稀疏向量无法理解语义。多路召回互补优势。

---

### 2. 重排序策略（8字）

**关键词**：RRF、WeightedRanker、融合

**解释**：
- **RRF（Reciprocal Rank Fusion）**：基于排名的公平融合
- **WeightedRanker**：基于权重的加权融合
- **融合目标**：将多路检索结果合并为单一排序列表

**为什么重要**：
多路检索返回多个结果集，需要统一标准合并排序，确保最相关结果排在前面。

---

### 3. 兼顾语义与精确（6字）

**关键词**：语义理解、精确匹配、平衡

**解释**：
- **语义理解**：通过稠密向量理解查询意图
- **精确匹配**：通过稀疏向量匹配关键词
- **平衡策略**：根据场景调整两者权重

**为什么重要**：
不同场景对语义和精确的需求不同。技术文档需要精确术语，产品搜索需要语义理解。

---

### 4. 提升检索质量（6字）

**关键词**：召回率、精确度、用户体验

**解释**：
- **召回率提升**：多路检索增加相关文档覆盖
- **精确度提升**：重排序优化结果排序
- **用户体验**：更快找到需要的信息

**为什么重要**：
这是混合检索的最终目标，也是衡量效果的核心指标。


## 核心价值主张

### 1. 解决单一检索的局限

**问题**：
- 纯向量检索：可能忽略精确术语（如产品型号、专业术语）
- 纯关键词检索：无法理解同义词、近义词

**解决**：
混合检索同时使用两种方式，互补短板。

---

### 2. 适应不同场景需求

**场景1：技术文档检索**
- 需求：精确匹配 API 名称、错误代码
- 策略：提高稀疏向量权重

**场景2：产品推荐**
- 需求：理解用户意图，推荐相似产品
- 策略：提高稠密向量权重

**场景3：通用搜索**
- 需求：平衡语义和精确
- 策略：使用 RRF 公平融合

---

### 3. 提升生产系统效果

**数据支持**：
- 召回率提升：15-30%
- 用户满意度提升：20-40%
- 查询成功率提升：25-35%

## 实战示例

### 查询："白色耳机，安静舒适"

**稠密向量检索**（语义理解）：
```
结果：降噪耳机、舒适耳机、白色音频设备
原因：理解"安静"→"降噪"，"舒适"→"人体工学"
```

**稀疏向量检索**（关键词匹配）：
```
结果：白色耳机、白色头戴式、白色入耳式
原因：精确匹配"白色"和"耳机"关键词
```

**混合检索结果**（融合后）：
```
Top 1: 白色降噪耳机（语义+关键词都匹配）
Top 2: 白色舒适耳机（语义+关键词都匹配）
Top 3: 降噪耳机（仅语义匹配）
```

## 核心技术栈

### Milvus 实现

```python
from pymilvus import AnnSearchRequest, RRFRanker

# 稠密向量检索
request_dense = AnnSearchRequest(
    data=[dense_vector],
    anns_field="text_dense",
    limit=10
)

# 稀疏向量检索（BM25）
request_sparse = AnnSearchRequest(
    data=[query_text],
    anns_field="text_sparse",
    limit=10
)

# RRF 重排序
ranker = RRFRanker(k=60)

# 混合检索
results = client.hybrid_search(
    collection_name="my_collection",
    reqs=[request_dense, request_sparse],
    ranker=ranker,
    limit=5
)
```

## 常见误解

### ❌ 误解1："混合检索就是多次检索"

**正确理解**：
混合检索不是简单的多次检索，而是：
1. 多路并发检索（不是串行）
2. 统一重排序（不是简单拼接）
3. 优化的融合策略（不是随意组合）

---

### ❌ 误解2："混合检索一定比单一检索好"

**正确理解**：
混合检索有优势，但也有代价：
- ✅ 优势：召回率和精确度提升
- ❌ 代价：延迟增加 20-50%，复杂度提升

**适用场景**：
- ✅ 适合：对质量要求高的场景（RAG、企业搜索）
- ❌ 不适合：对延迟极度敏感的场景（实时推荐）

---

### ❌ 误解3："RRF 和 WeightedRanker 效果一样"

**正确理解**：
两者适用场景不同：
- **RRF**：无特定强调，公平平衡
- **WeightedRanker**：需要强调某个字段
