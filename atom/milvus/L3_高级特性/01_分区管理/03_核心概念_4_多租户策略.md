# 核心概念 4：多租户策略

> 使用分区实现多租户数据隔离的四种策略

---

## 概述

**多租户**是指在单个系统中支持多个独立租户（用户、组织、应用）的能力。Milvus 提供四种多租户策略，每种策略在数据隔离、灵活性和可扩展性方面有不同的权衡。

**核心价值**：
- 数据隔离：实现租户级别的数据隔离
- 成本优化：共享基础设施，降低成本
- 可扩展性：支持从少量到百万级租户

---

## 1. 四种多租户策略概览

### 1.1 策略对比

**来源**：Context7 官方文档

| 策略 | 数据隔离 | 可扩展性 | Schema 灵活性 | RBAC 支持 | 适用场景 |
|------|----------|----------|---------------|-----------|----------|
| Database-level | 强 | 低 | 高 | 是 | 少量租户，需要强隔离 |
| Collection-level | 强 | 中 | 高 | 是 | 中等数量租户，需要独立 Schema |
| Partition-level | 中 | 中 | 低 | 否 | 中等数量租户，共享 Schema |
| Partition Key-level | 弱 | 高 | 低 | 否 | 大量租户（百万级），共享 Schema |

### 1.2 可扩展性排序

**从低到高**：
1. Database-level：支持少量租户（< 10）
2. Collection-level：支持中等数量租户（< 100）
3. Partition-level：支持最多 1,024 个租户
4. Partition Key-level：支持百万级租户（最高）

### 1.3 数据隔离排序

**从强到弱**：
1. Database-level：最强隔离（独立 Database）
2. Collection-level：强隔离（独立 Collection）
3. Partition-level：中等隔离（独立 Partition）
4. Partition Key-level：弱隔离（逻辑隔离）

---

## 2. Database-level 多租户

### 2.1 定义

**每个租户分配到一个独立的 Database**

**核心特性**：
- **最强隔离**：每个租户的数据在物理上完全隔离
- **最高灵活性**：每个租户可以有多个 Collection 和独立 Schema
- **RBAC 支持**：支持 Database 级别的 RBAC
- **独立管理**：每个租户可以独立管理所有资源

### 2.2 适用场景

- 少量租户（< 10）
- 需要最强隔离的场景
- 每个租户需要多个 Collection 的场景
- 需要 RBAC 的场景

### 2.3 优缺点

**优点**：
- ✅ 最强隔离（每个租户独立 Database）
- ✅ 最高灵活性（每个租户可以有多个 Collection）
- ✅ 支持 RBAC

**缺点**：
- ❌ 可扩展性最低（Database 数量有限制）
- ❌ 资源开销最大（每个 Database 需要独立资源）

### 2.4 示例

```python
from pymilvus import connections, db

# 连接 Milvus
connections.connect("default", host="localhost", port="19530")

# 为每个租户创建独立的 Database
db.create_database("tenant_a_db")
db.create_database("tenant_b_db")

# 切换到租户 A 的 Database
db.using_database("tenant_a_db")

# 在租户 A 的 Database 中创建 Collection
from pymilvus import FieldSchema, CollectionSchema, DataType, Collection

schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128)
    ]
)
collection_a = Collection("my_collection", schema)

# 切换到租户 B 的 Database
db.using_database("tenant_b_db")

# 在租户 B 的 Database 中创建 Collection
collection_b = Collection("my_collection", schema)
```

## 3. Collection-level 多租户

### 3.1 定义

**每个租户分配到一个独立的 Collection**

**核心特性**：
- **强隔离**：每个租户的数据在物理上完全隔离
- **Schema 灵活性**：每个租户可以有独立的 Schema
- **RBAC 支持**：支持 Collection 级别的 RBAC
- **独立管理**：每个租户可以独立管理索引、加载、释放等操作

### 3.2 适用场景

- 中等数量租户（< 100）
- 需要强隔离的场景
- 每个租户需要独立 Schema 的场景

### 3.3 优缺点

**优点**：
- ✅ 强隔离（每个租户独立 Collection）
- ✅ Schema 灵活性（每个租户可以有独立 Schema）
- ✅ 支持 RBAC

**缺点**：
- ❌ 可扩展性有限（Collection 数量有限制）
- ❌ 资源开销较大（每个 Collection 需要独立资源）

### 3.4 示例

```python
from pymilvus import connections, FieldSchema, CollectionSchema, DataType, Collection

# 连接 Milvus
connections.connect("default", host="localhost", port="19530")

# 为每个租户创建独立的 Collection
schema_a = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=1024)
    ]
)
collection_a = Collection("tenant_a_collection", schema_a)

schema_b = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=256),  # 不同维度
        FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=512)  # 不同字段
    ]
)
collection_b = Collection("tenant_b_collection", schema_b)

# 插入租户 A 的数据
data_a = [
    [1, 2, 3],
    [[0.1]*128, [0.2]*128, [0.3]*128],
    ["doc1", "doc2", "doc3"]
]
collection_a.insert(data_a)

# 搜索租户 A 的数据
query_vector = [[0.15]*128]
results = collection_a.search(
    data=query_vector,
    anns_field="vector",
    param={"metric_type": "L2", "params": {"nprobe": 10}},
    limit=10
)
```

## 4. Partition-level 多租户

### 4.1 定义

**每个租户分配到一个手动创建的分区，一个 Collection 最多可以有 1,024 个分区**

**核心特性**：
- **物理隔离**：每个租户的数据在物理上由分区分离
- **手动管理**：需要手动创建分区
- **共享 Schema**：所有租户必须共享相同的数据 Schema
- **无 RBAC 支持**：不支持 Partition 级别的 RBAC
- **灵活查询**：可以单独查询租户，也可以跨多个分区进行聚合查询
- **热冷数据管理**：可以灵活加载或释放特定租户的数据

### 4.2 适用场景

- 中等数量租户（最多 1,024 个）
- 需要物理隔离的场景
- 需要热冷数据管理的场景

### 4.3 优缺点

**优点**：
- ✅ 物理隔离（每个租户独立分区）
- ✅ 支持热冷数据管理（可以加载/释放特定分区）
- ✅ 支持跨租户聚合查询

**缺点**：
- ❌ 可扩展性有限（最多 1,024 个分区）
- ❌ 需要手动创建分区
- ❌ 所有租户必须共享相同的 Schema
- ❌ 不支持 RBAC

### 4.4 示例

```python
from pymilvus import connections, FieldSchema, CollectionSchema, DataType, Collection

# 连接 Milvus
connections.connect("default", host="localhost", port="19530")

# 创建 Collection（所有租户共享）
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=1024)
    ]
)
collection = Collection("multi_tenant_collection", schema)

# 为每个租户创建分区
collection.create_partition("tenant_a")
collection.create_partition("tenant_b")

# 插入租户 A 的数据
data_a = [
    [1, 2, 3],
    [[0.1]*128, [0.2]*128, [0.3]*128],
    ["doc1", "doc2", "doc3"]
]
collection.insert(data_a, partition_name="tenant_a")

# 只加载租户 A 的分区
collection.load(partition_names=["tenant_a"])

# 搜索租户 A 的数据
query_vector = [[0.15]*128]
results = collection.search(
    data=query_vector,
    anns_field="vector",
    param={"metric_type": "L2", "params": {"nprobe": 10}},
    limit=10,
    partition_names=["tenant_a"]
)
```

## 5. Partition Key-level 多租户（Milvus 2.6 推荐）

### 5.1 定义

**所有租户共享一个 Collection 和 Schema，但每个租户的数据会根据 Partition Key 值自动路由到 16 个物理隔离的分区中**

**核心特性**：
- **自动分区**：Milvus 自动将数据路由到 16 个物理分区
- **逻辑隔离**：虽然多个租户可能共享一个物理分区，但数据在逻辑上是分离的
- **最高可扩展性**：支持数百万租户
- **共享 Schema**：所有租户必须共享相同的数据 Schema
- **无 RBAC 支持**：不支持 Partition Key 级别的 RBAC
- **灵活查询**：可以单独查询租户，也可以跨多个分区进行聚合查询

### 5.2 适用场景

- 大量租户（百万级）
- 数据 Schema 统一的多租户系统
- 需要跨租户聚合查询或分析的场景

### 5.3 优缺点

**优点**：
- ✅ 最高可扩展性（支持数百万租户）
- ✅ 自动分区管理，无需手动创建分区
- ✅ 支持跨租户聚合查询

**缺点**：
- ❌ 数据隔离较弱（多个租户可能共享物理分区）
- ❌ 所有租户必须共享相同的 Schema
- ❌ 不支持 RBAC

### 5.4 示例

```python
from pymilvus import connections, FieldSchema, CollectionSchema, DataType, Collection

# 连接 Milvus
connections.connect("default", host="localhost", port="19530")

# 创建 Collection，使用 tenant_id 作为 Partition Key
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True),  # 标记为 Partition Key
        FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=1024)
    ]
)
collection = Collection("multi_tenant_collection", schema)

# 创建索引
index_params = {
    "index_type": "HNSW",
    "metric_type": "L2",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="vector", index_params=index_params)

# 加载 Collection
collection.load()

# 插入数据（自动路由到相应分区）
data = [
    [1, 2, 3, 4, 5],
    [[0.1]*128, [0.2]*128, [0.3]*128, [0.4]*128, [0.5]*128],
    ["tenant_a", "tenant_a", "tenant_b", "tenant_c", "tenant_c"],
    ["doc1", "doc2", "doc3", "doc4", "doc5"]
]
collection.insert(data)

# 搜索租户 A 的数据（自动只搜索相关分区）
query_vector = [[0.15]*128]
results = collection.search(
    data=query_vector,
    anns_field="vector",
    param={"metric_type": "L2", "params": {"ef": 100}},
    limit=10,
    expr='tenant_id == "tenant_a"'  # 自动路由
)
```

## 6. 策略选择建议

### 6.1 根据租户数量选择

**来源**：Context7 官方文档

| 租户数量 | 推荐策略 | 原因 |
|----------|----------|------|
| < 10 | Database-level 或 Collection-level | 需要强隔离和 RBAC |
| 10-100 | Collection-level 或 Partition-level | 平衡隔离和可扩展性 |
| 100-1,024 | Partition-level | 物理隔离 + 热冷数据管理 |
| > 1,024 | Partition Key-level | 最高可扩展性 |

### 6.2 根据隔离需求选择

| 隔离需求 | 推荐策略 | 原因 |
|----------|----------|------|
| 最强隔离 | Database-level | 独立 Database |
| 强隔离 | Collection-level | 独立 Collection |
| 中等隔离 | Partition-level | 独立 Partition |
| 弱隔离 | Partition Key-level | 逻辑隔离 |

### 6.3 根据 Schema 灵活性选择

| Schema 需求 | 推荐策略 | 原因 |
|-------------|----------|------|
| 每个租户独立 Schema | Database-level 或 Collection-level | 支持独立 Schema |
| 所有租户共享 Schema | Partition-level 或 Partition Key-level | 共享 Schema |

### 6.4 根据 RBAC 需求选择

| RBAC 需求 | 推荐策略 | 原因 |
|-----------|----------|------|
| 需要 RBAC | Database-level 或 Collection-level | 支持 RBAC |
| 不需要 RBAC | Partition-level 或 Partition Key-level | 不支持 RBAC |

## 7. 实际应用案例

### 7.1 SaaS 应用（百万级租户）

**场景**：SaaS 应用，预计有 100 万个租户

**推荐策略**：Partition Key-level

**原因**：
- 支持百万级租户
- 自动分区管理，无需手动创建分区
- 所有租户共享相同的 Schema

**实现**：
```python
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=384),
        FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True),
        FieldSchema(name="document", dtype=DataType.VARCHAR, max_length=2048)
    ]
)
collection = Collection("saas_app", schema)
```

### 7.2 企业级应用（100 个租户）

**场景**：企业级应用，有 100 个大客户，每个客户需要独立的 Schema

**推荐策略**：Collection-level

**原因**：
- 支持独立 Schema
- 强隔离
- 支持 RBAC

**实现**：
```python
# 为每个租户创建独立的 Collection
for tenant_id in range(100):
    schema = CollectionSchema(
        fields=[
            FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
            FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
            # 每个租户可以有不同的字段
            FieldSchema(name=f"custom_field_{tenant_id}", dtype=DataType.VARCHAR, max_length=512)
        ]
    )
    collection = Collection(f"tenant_{tenant_id}_collection", schema)
```

### 7.3 时间序列数据（按月份分区）

**场景**：时间序列数据，按月份分区，只查询最近的数据

**推荐策略**：Partition-level

**原因**：
- 支持热冷数据管理
- 可以独立加载/释放分区
- 物理隔离

**实现**：
```python
from datetime import datetime

collection = Collection("time_series_data")

# 按月份创建分区
for i in range(12):
    month = (datetime.now().replace(day=1) - timedelta(days=30*i)).strftime("%Y_%m")
    collection.create_partition(month)

# 只加载最近 3 个月的数据
recent_months = [
    (datetime.now() - timedelta(days=30*i)).strftime("%Y_%m")
    for i in range(3)
]
collection.load(partition_names=recent_months)
```

## 核心要点总结

1. **四种多租户策略**：Database-level、Collection-level、Partition-level、Partition Key-level
2. **可扩展性排序**：Partition Key-level > Partition-level > Collection-level > Database-level
3. **数据隔离排序**：Database-level > Collection-level > Partition-level > Partition Key-level
4. **选择建议**：根据租户数量、隔离需求、Schema 灵活性、RBAC 需求选择
5. **推荐策略**：大规模多租户场景推荐使用 Partition Key-level
