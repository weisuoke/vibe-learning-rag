## 类比1：前端开发视角

### Plan-and-Execute ≈ 前端项目管理流程

```
前端项目开发：
  PM 制定需求文档（Plan）
    ↓
  [组件A, 组件B, 页面C, 联调D]
    ↓
  前端工程师逐个开发（Execute）
    ↓
  产品验收：需要调整吗？（Replan）
    ↓
  发布上线 or 迭代修改

Plan-and-Execute Agent：
  Planner 生成计划（大模型）
    ↓
  [步骤1, 步骤2, 步骤3, 步骤4]
    ↓
  Executor 逐步执行（小模型+工具）
    ↓
  Replanner：任务完成了吗？（大模型）
    ↓
  输出结果 or 调整计划
```

### 对照表

| Agent 概念 | 前端类比 | 说明 |
|-----------|---------|------|
| **Planner** | PM/技术主管 | 看全局、定方案，不亲自写代码 |
| **Executor** | 前端工程师 | 按计划写代码，一次做一个任务 |
| **Replanner** | Code Review | 检查结果，决定是否需要修改 |
| **Plan（计划）** | 需求文档/技术方案 | 明确要做什么、按什么顺序 |
| **子目标** | 用户故事/Task | 独立可交付的最小工作单元 |
| **Re-planning** | 需求变更 | 发现新情况，调整开发计划 |
| **Reflection** | 自测/Review | 自己检查自己的代码质量 |
| **Structured Output** | TypeScript 接口定义 | 约束输出格式，避免随意输出 |
| **StateGraph** | React 状态机 (useReducer) | 状态 + 动作 → 新状态 |
| **DAG 并行执行** | Promise.all() | 无依赖的任务并行执行 |

### 具体类比展开

#### Planner ≈ 项目 PM

```
PM 的工作：
  接到需求 "做一个用户管理系统"
  → 拆分为：[用户列表页, 新增用户, 编辑用户, 删除用户, 权限管理]
  → 确定优先级和依赖关系
  → 分配给工程师

Planner 的工作：
  接到任务 "对比 Python 和 Rust 的 Web 开发"
  → 拆分为：[搜索Python信息, 搜索Rust信息, 对比分析, 写报告]
  → 确定执行顺序
  → 交给 Executor
```

#### Executor ≈ 工程师执行 Task

```
工程师：
  拿到 Task "实现用户列表页"
  → 选技术方案（用 Table 组件还是自己写）
  → 调用 API 获取数据
  → 实现组件
  → 返回结果

Executor：
  拿到步骤 "搜索 Python Web 框架信息"
  → 选工具（用 search 还是 read_doc）
  → 调用工具获取数据
  → 整理结果
  → 返回给 Replanner
```

#### Replanner ≈ 产品验收 + 需求变更

```
产品验收：
  "用户列表做好了，但缺少分页功能"
  → 新增 Task：实现分页
  → 调整后续计划

Replanner：
  "Python 信息搜到了，但缺少性能数据"
  → 新增步骤：搜索性能基准测试
  → 更新剩余计划
```

#### ReAct ≈ 没有需求文档的开发

```
ReAct 方式 = 没有需求文档：
  "做一个用户系统"
  → 先写个列表页... 嗯，还需要新增...
  → 等等，我忘了删除功能...
  → 每做一步都要思考下一步

Plan-and-Execute 方式 = 有需求文档：
  "做一个用户系统"
  → 先列出所有功能 [列表, 新增, 编辑, 删除, 权限]
  → 按顺序开发，不会遗漏
```

## 类比2：日常生活视角

### Plan-and-Execute ≈ 出国旅行规划

```
旅行规划：
  旅行社制定行程（Plan）
    ↓
  [Day1 飞机, Day2 景点A, Day3 景点B, Day4 购物, Day5 返程]
    ↓
  导游按行程执行（Execute）
    ↓
  天气不好？调整行程（Replan）
    ↓
  愉快结束旅程
```

### 对照表

| Agent 概念 | 日常生活类比 | 说明 |
|-----------|------------|------|
| **Planner** | 旅行社/攻略达人 | 提前规划全程，考虑各种因素 |
| **Executor** | 导游/司机 | 按计划带你去每个地方 |
| **Replanner** | 临时调整行程 | 下雨了改去室内景点 |
| **Plan** | 旅行攻略 | 每天去哪、怎么去、住哪 |
| **子目标** | 每天的行程 | 独立可完成的最小单元 |
| **Re-planning** | 行程变更 | 航班取消、景点关闭等意外 |
| **Reflection** | 每天晚上回顾 | 今天玩得怎么样？明天要调整吗？ |
| **Reflexion** | 旅行经验积累 | 上次去日本的经验用在这次韩国行 |
| **LLM-as-Judge** | 朋友点评你的攻略 | 独立第三方评价计划好不好 |
| **DAG 并行** | 分头行动 | 你去买东西，我去拍照，约好碰头 |

### 具体类比展开

#### 任务分解 ≈ 做一顿大餐

```
做一顿大餐（无规划）：
  "做年夜饭" → 先炒个菜... 等等，汤还没煲...
  → 忘了买葱了！→ 出去买 → 回来锅糊了 ❌

做一顿大餐（有规划）：
  Plan: [1. 列菜单, 2. 买食材, 3. 煲汤（耗时长先做）,
         4. 备菜, 5. 炒菜, 6. 装盘]
  Execute: 按计划执行，煲汤和备菜可以并行 ✅

  Re-plan: 发现没有五花肉
  → 调整：用排骨替代，修改烹饪步骤 ✅
```

#### Reflection ≈ 考试后检查试卷

```
没有 Reflection：
  写完试卷 → 直接交 → 事后发现算错了 ❌

有 Reflection：
  写完试卷 → 从头检查一遍 → 发现第3题算错了 → 改正 → 交卷 ✅

有 Reflexion（经验积累）：
  上次考试：第3题粗心算错了
  这次考试：做第3题时特别小心 → 正确 ✅
  经验记忆："做计算题要验算"
```

#### 三种 Re-planning ≈ 旅行变更

```
完全重规划 ≈ 目的地取消，整个行程重做：
  原计划去日本 → 签证没下来 → 改去韩国（全部重新规划）

增量调整 ≈ 某天景点关闭，只改当天：
  Day3 景点A维修 → Day3 改去景点A' → 其他天不变

插入补充 ≈ 发现附近有好吃的，临时加一站：
  原计划 Day2 [景点B, 午饭, 景点C]
  → 发现附近有著名拉面店
  → Day2 [景点B, 拉面店, 午饭取消, 景点C]
```

## 综合记忆图

```
                Plan-and-Execute Agent
                        │
          ┌─────────────┼─────────────┐
          │             │             │
     Planner       Executor      Replanner
     (PM/攻略)    (工程师/导游)  (验收/调整)
          │             │             │
     TypeScript    Promise.all   Code Review

          │             │             │
     旅行攻略    按行程执行      天气变了改行程
```

## 一句话类比

**Plan-and-Execute 就像项目管理——PM 先写需求文档（Planner），工程师按 Task 开发（Executor），产品验收后决定是否调整（Replanner）。ReAct 则像没有需求文档的野蛮开发，走一步看一步。**
