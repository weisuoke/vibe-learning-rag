# 核心概念4：输出格式化与后处理

> 节点函数的返回值不是随便写的——它必须匹配状态 Schema 的期望格式，并且要考虑下游 Reducer 的行为。输出格式化就是节点在"交出结果"之前的最后一道工序。

## 1. 为什么需要输出格式化？

### 问题场景

节点内部的处理逻辑往往产生"原始数据"，但状态 Schema 期望的是"规范数据"。两者之间存在格式鸿沟：

```
节点内部处理          状态 Schema 期望
─────────────        ─────────────
LLM 返回自由文本  →   需要结构化 dict
检索返回 Document →   需要纯文本列表
分数是 int        →   字段类型是 float
多个中间变量      →   只需要最终结果
```

如果不做格式化，轻则类型不匹配报错，重则 Reducer 行为异常导致数据污染。

### 双重类比

**前端类比：API 响应的 data transformer**

```
后端返回原始 JSON          前端组件期望的格式
──────────────            ──────────────
{ "user_name": "张三" }   { "name": "张三" }
{ "created_at": 16... }   { "date": "2026-02-27" }

// 中间需要一个 transformer
const transform = (raw) => ({
  name: raw.user_name,
  date: formatDate(raw.created_at)
})
```

节点函数的 return 语句就是这个 transformer——把内部处理结果转换为状态 Schema 能接受的格式。

**日常生活类比：厨师装盘**

厨师炒完菜，不能直接把锅端上桌。需要：
1. 把菜从锅里盛出来（提取有用数据）
2. 摆到盘子里（映射到正确字段）
3. 摆盘装饰（格式化数值、截断文本）
4. 擦干净盘子边缘（去掉多余信息）

"装盘"就是输出格式化——内容没变，但形式符合"上桌"的标准。

## 2. 返回值规范化：最核心的规则

### 规则一：只返回要更新的字段

```python
from typing_extensions import TypedDict, Annotated
from operator import add

class State(TypedDict):
    messages: Annotated[list, add]
    summary: str
    scores: Annotated[list[float], add]

# ✅ 正确：返回 dict，只包含要更新的字段
def good_node(state: State) -> dict:
    return {"summary": "处理完成", "scores": [0.95]}

# ❌ 错误：返回完整 State（不必要，且可能覆盖其他字段）
def bad_node(state: State) -> dict:
    return {
        "messages": state["messages"],   # 多余！你没改它就别返回
        "summary": "处理完成",
        "scores": state["scores"] + [0.95]  # 更危险！见下文
    }
```

为什么"返回完整 State"是错误的？因为框架会把你返回的每个字段都送进对应的 Channel：
- `messages` 有 `add` reducer → 框架会执行 `current + state["messages"]`，导致消息翻倍
- `scores` 有 `add` reducer → 框架会执行 `current + (current + [0.95])`，导致历史分数重复

> 源码依据：`BinaryOperatorAggregate.update()` 中 `self.value = self.operator(self.value, value)` 会对每个返回值应用 reducer，不管你是否"有意"更新该字段。
> [来源: sourcecode/langgraph/libs/langgraph/langgraph/channels/binop.py]

### 规则二：有 Reducer 的字段，返回增量值

这是初学者最常踩的坑：

```python
# ✅ 正确：有 reducer 的字段，返回增量值
def correct_reducer_return(state: State) -> dict:
    # messages 有 add reducer，返回新增的消息即可
    return {"messages": ["new message"]}  # reducer 自动执行: old + ["new message"]

# ❌ 错误：有 reducer 的字段，返回完整列表
def wrong_reducer_return(state: State) -> dict:
    # 这会导致 reducer 执行: old + (old + ["new message"]) → 消息全部重复！
    return {"messages": state["messages"] + ["new message"]}
```

**记忆口诀：有 Reducer 交增量，无 Reducer 交全量。**

### 规则三：无 Reducer 的字段，返回完整替换值

```python
class SimpleState(TypedDict):
    query: str        # 无 reducer → LastValue Channel
    result: str       # 无 reducer → LastValue Channel

def update_result(state: SimpleState) -> dict:
    # 无 reducer 的字段，返回的值会直接覆盖旧值
    return {"result": "这是新的结果"}  # 旧的 result 被完全替换
```

> 源码依据：`LastValue.update()` 直接执行 `self.value = values[-1]`，没有任何合并逻辑。
> [来源: sourcecode/langgraph/libs/langgraph/langgraph/channels/last_value.py]

## 3. 字段映射与结构转换

### 场景：LLM 输出解析

LLM 返回的是自由文本，但状态需要结构化数据。节点函数需要在 return 之前完成解析：

```python
import json
import re

class RAGState(TypedDict):
    llm_response: str           # LLM 原始输出
    answer: str                 # 结构化答案
    confidence: float           # 置信度
    sources: Annotated[list[str], add]  # 引用来源

def format_llm_output(state: RAGState) -> dict:
    """将 LLM 原始输出转换为结构化状态更新"""
    raw_output = state["llm_response"]
    parsed = _parse_json_from_text(raw_output)

    # 字段映射：LLM 输出字段 → 状态字段
    return {
        "answer": parsed.get("response", ""),
        "confidence": float(parsed.get("score", 0.0)),
        "sources": parsed.get("references", []),  # sources 有 reducer，返回增量
    }

def _parse_json_from_text(text: str) -> dict:
    """从可能包含 JSON 的文本中提取结构化数据"""
    json_match = re.search(r'\{[^{}]*\}', text, re.DOTALL)
    if json_match:
        try:
            return json.loads(json_match.group())
        except json.JSONDecodeError:
            pass
    return {"response": text, "score": 0.0, "references": []}
```

### 场景：检索结果标准化

```python
def normalize_search_results(state: dict) -> dict:
    """将不同来源的检索结果统一为标准格式"""
    raw_docs = state["raw_documents"]
    formatted = [
        {
            "content": doc.page_content if hasattr(doc, 'page_content') else str(doc),
            "source": getattr(doc, 'metadata', {}).get("source", "unknown"),
            "rank": i + 1,
        }
        for i, doc in enumerate(raw_docs)
    ]
    return {"documents": formatted, "doc_count": len(formatted)}  # 无 reducer → 完整替换
```

## 4. 列表与集合的格式化——最常见的错误来源

这个问题值得单独拿出来强调，因为它是 LangGraph 状态管理中 Bug 率最高的地方。

### 对照表：有 Reducer vs 无 Reducer

| 字段定义 | Channel 类型 | 返回值应该是 | 示例 |
|----------|-------------|-------------|------|
| `items: list[str]` | LastValue | 完整列表 | `{"items": ["a", "b", "c"]}` |
| `items: Annotated[list[str], add]` | BinaryOperatorAggregate | 新增元素 | `{"items": ["c"]}` |
| `count: int` | LastValue | 新值 | `{"count": 42}` |
| `messages: Annotated[list, add_messages]` | BinaryOperatorAggregate | 新消息 | `{"messages": [HumanMessage("hi")]}` |

### 典型错误演示

```python
class State(TypedDict):
    history: Annotated[list[str], add]   # 有 reducer
    current: list[str]                    # 无 reducer

# ─── history 字段（有 add reducer）───

# ✅ 正确
def node_a(state: State) -> dict:
    return {"history": ["step_a完成"]}
    # 效果：history = old_history + ["step_a完成"]

# ❌ 错误
def node_a_wrong(state: State) -> dict:
    return {"history": state["history"] + ["step_a完成"]}
    # 效果：history = old_history + old_history + ["step_a完成"]  ← 历史记录翻倍！

# ─── current 字段（无 reducer）───

# ✅ 正确
def node_b(state: State) -> dict:
    return {"current": ["new_item_1", "new_item_2"]}
    # 效果：current = ["new_item_1", "new_item_2"]（完全替换）

# ❌ 错误（不会报错，但语义不对）
def node_b_wrong(state: State) -> dict:
    return {"current": ["only_new"]}
    # 如果你想追加，这不会追加——它会替换。需要 reducer 才能追加。
```

## 5. 条件返回模式

节点不一定每次都更新相同的字段。根据处理结果，可以动态决定返回哪些字段：

```python
class WorkflowState(TypedDict):
    input: str
    output: str
    error: str
    status: str
    retry_count: int

def process_node(state: WorkflowState) -> dict:
    """根据处理结果决定返回哪些字段"""
    try:
        result = do_processing(state["input"])

        # 成功路径：更新 output 和 status
        return {
            "output": result,
            "status": "completed",
        }
    except Exception as e:
        # 失败路径：更新 error、status 和 retry_count
        return {
            "error": str(e),
            "status": "failed",
            "retry_count": state["retry_count"] + 1,
        }
```

### 条件返回的注意事项

```python
# ✅ 好的做法：先构建 dict，再返回
def flexible_node(state: State) -> dict:
    updates = {}
    if needs_summary(state):
        updates["summary"] = generate_summary(state["documents"])
    if needs_scoring(state):
        updates["scores"] = [calculate_score(state)]  # 有 reducer，返回增量
    updates["last_processed"] = "flexible_node"
    return updates

# ⚠️ 返回空 dict 是合法的，但状态不会有任何变化
def maybe_noop(state: State) -> dict:
    if not state.get("input"):
        return {}
    return {"output": process(state["input"])}
```

## 6. 后处理链模式

在返回之前，对数据做一系列清洗和规范化操作：

### 模式：后处理管道

```python
def answer_node(state: State) -> dict:
    """生成答案并做后处理"""
    raw_answer = call_llm(state["context"], state["query"])

    # 后处理链：依次应用多个转换
    answer = raw_answer
    answer = truncate_text(answer, max_length=2000)      # 1. 截断过长文本
    answer = format_precision(answer)                     # 2. 格式化数值精度
    answer = redact_sensitive_info(answer)                # 3. 敏感信息脱敏

    return {
        "answer": answer,
        "metadata": {
            "model": "gpt-4",
            "timestamp": datetime.now().isoformat(),      # 4. 添加元数据
            "original_length": len(raw_answer),
            "truncated": len(raw_answer) > 2000,
        },
    }
```

### 常见后处理操作

```python
from datetime import datetime
import re

def truncate_text(text: str, max_length: int = 2000) -> str:
    """截断过长文本"""
    return text if len(text) <= max_length else text[:max_length - 3] + "..."

def redact_pii(text: str) -> str:
    """敏感信息脱敏"""
    text = re.sub(r'\b\d{11}\b', '[PHONE]', text)
    text = re.sub(r'\b[\w.]+@[\w.]+\.\w+\b', '[EMAIL]', text)
    return text
```

## 7. 与 Reducer 的配合——输出格式化的核心约束

输出格式化不是孤立的——它必须考虑下游 Channel 的 Reducer 类型。这是整个知识点最关键的部分。

### Reducer 类型决定返回值格式

```
节点返回值 → Channel.update(values) → 新状态
                  ↑
            Reducer 类型决定了 update 的行为
```

| Reducer 类型 | 行为 | 节点应返回 | 源码位置 |
|-------------|------|-----------|---------|
| 无 (LastValue) | `self.value = values[-1]` | 完整替换值 | `channels/last_value.py` |
| `operator.add` | `self.value = self.value + value` | 增量列表 | `channels/binop.py` |
| `add_messages` | 智能合并消息列表 | 新消息列表 | `graph/message.py` |
| 自定义函数 | `self.value = func(self.value, value)` | 取决于函数签名 | `channels/binop.py` |

### 源码解读：BinaryOperatorAggregate.update()

```python
# channels/binop.py 核心逻辑（简化版）
def update(self, values: Sequence[Value]) -> bool:
    for value in values:
        # 检查是否是 Overwrite 类型
        is_overwrite, overwrite_value = _get_overwrite(value)
        if is_overwrite:
            self.value = overwrite_value       # 强制覆盖，跳过 reducer
        else:
            self.value = self.operator(self.value, value)  # 应用 reducer
    return True
```

关键洞察：
- `self.operator` 就是你在 `Annotated[list, add]` 中指定的 `add` 函数
- 如果 `self.value = [1, 2]`，你返回 `[3]`，结果是 `[1, 2] + [3] = [1, 2, 3]`
- 如果你错误地返回 `[1, 2, 3]`，结果是 `[1, 2] + [1, 2, 3] = [1, 2, 1, 2, 3]`

### Overwrite：绕过 Reducer 的逃生通道

当你确实需要强制覆盖一个有 Reducer 的字段时，使用 `Overwrite`：

```python
from langgraph.graph.state import Overwrite

class State(TypedDict):
    history: Annotated[list[str], add]  # 正常情况下只追加

def reset_history(state: State) -> dict:
    """特殊场景：清空历史记录"""
    # 用 Overwrite 绕过 add reducer，强制设为空列表
    return {"history": Overwrite([])}
```

> 注意：每个 super-step 中，同一个字段只能接收一个 Overwrite 值。多个 Overwrite 会抛出 `InvalidUpdateError`。
> [来源: sourcecode/langgraph/libs/langgraph/langgraph/channels/binop.py]

## 8. 在 RAG 中的应用

### 场景1：检索结果格式化

```python
from typing_extensions import TypedDict, Annotated
from operator import add

class RAGState(TypedDict):
    query: str
    raw_docs: list                              # 无 reducer → 完整替换
    formatted_context: str                      # 无 reducer → 完整替换
    sources: Annotated[list[str], add]          # 有 reducer → 增量追加

def format_retrieval_results(state: RAGState) -> dict:
    """检索后处理：统一文档格式，构建上下文字符串"""
    docs = state["raw_docs"]

    # 构建格式化的上下文（无 reducer，返回完整值）
    context_parts = []
    source_list = []
    for i, doc in enumerate(docs):
        content = doc.page_content if hasattr(doc, 'page_content') else str(doc)
        # 截断单个文档，避免上下文过长
        content = content[:500] if len(content) > 500 else content
        context_parts.append(f"[文档{i+1}] {content}")

        source = doc.metadata.get("source", "unknown") if hasattr(doc, 'metadata') else "unknown"
        source_list.append(source)

    return {
        "formatted_context": "\n\n".join(context_parts),  # 无 reducer → 完整替换
        "sources": source_list,                             # 有 reducer → 增量追加
    }
```

### 场景2：LLM 输出解析与结构化

```python
class QAState(TypedDict):
    context: str
    question: str
    raw_llm_output: str
    answer: str
    confidence: float
    needs_retry: bool

def parse_and_format_answer(state: QAState) -> dict:
    """解析 LLM 输出，提取结构化答案"""
    raw = state["raw_llm_output"]

    # 尝试解析结构化输出
    try:
        parsed = json.loads(raw)
        return {
            "answer": parsed["answer"],
            "confidence": min(max(float(parsed.get("confidence", 0.5)), 0.0), 1.0),
            "needs_retry": False,
        }
    except (json.JSONDecodeError, KeyError):
        # LLM 没有按格式输出，做降级处理
        return {
            "answer": raw.strip(),
            "confidence": 0.3,       # 低置信度
            "needs_retry": True,     # 标记需要重试
        }
```

### 场景3：上下文窗口管理

```python
def trim_context_for_llm(state: dict) -> dict:
    """确保上下文不超过 LLM 的窗口限制"""
    context = state["formatted_context"]
    max_chars = 6000  # 粗略估算：2字符≈1token，预留3000 token

    if len(context) > max_chars:
        # 按文档块截断，保留前面的高相关度文档
        docs = context.split("\n\n")
        trimmed, length = [], 0
        for doc in docs:
            if length + len(doc) > max_chars:
                break
            trimmed.append(doc)
            length += len(doc)
        context = "\n\n".join(trimmed)

    return {
        "formatted_context": context,
        "context_trimmed": len(state["formatted_context"]) > max_chars,
    }
```

## 9. 常见错误速查表

| 错误 | 原因 | 修复 |
|------|------|------|
| 列表数据翻倍 | 有 reducer 的字段返回了完整列表 | 只返回新增元素 |
| `InvalidUpdateError: Can receive only one value` | 无 reducer 的字段被多个并行节点同时更新 | 给字段加 reducer，或调整图结构 |
| 类型不匹配 | 返回 int 但字段期望 float | 在 return 前做类型转换 |
| 字段丢失 | 返回了 State 中不存在的字段名 | 检查字段名拼写，对照 Schema |
| Reducer 行为异常 | 不理解 Overwrite 的限制 | 每个 super-step 只能有一个 Overwrite |

---

## 一句话总结

**输出格式化的核心就一句话：有 Reducer 交增量，无 Reducer 交全量——在 return 之前把数据"装好盘"。**
