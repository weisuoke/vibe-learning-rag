## 卡片 1：什么是 create_agent？

### 一句话

**LangChain 1.0 的统一 Agent 工厂函数，一行代码创建 Agent。**

### 举例

```python
from langchain.agents import create_agent

# 一行代码，创建一个能搜索网络的 Agent
agent = create_agent(model="openai:gpt-4o", tools=[search])
```

### 应用场景

快速构建具有工具调用能力的 AI Agent。不管你要做文档问答、智能客服还是数据分析助手，入口都是这一个函数。

### 速记口诀

> **一个函数管所有，模型工具往里塞，返回的是编译图，invoke 一调就能跑。**

### 关键细节

| 要素 | 说明 |
|------|------|
| 输入 | `model` + `tools` + 可选参数 |
| 输出 | `CompiledStateGraph`（LangGraph 编译图） |
| 取代 | `create_react_agent`、`create_openai_functions_agent`、`initialize_agent` 等 6+ 种旧 API |
| 最简形式 | `create_agent("openai:gpt-4o", tools=[])` |

## 卡片 2：Agent 的核心循环

### 一句话

**Agent = 模型 + 工具 + 循环决策。模型决定"做什么"，工具负责"怎么做"，循环保证"做到位"。**

### 举例

```
用户提问："北京今天天气怎么样？"

循环第 1 轮：
  模型决策 → "我需要查天气" → 调用 weather_tool("北京")
  工具返回 → "晴，25°C"

循环第 2 轮：
  模型决策 → "信息够了，可以回答" → 生成最终回复
  返回 → "北京今天晴天，气温 25°C，适合出行。"
```

### 应用场景

理解 Agent 的本质行为模式。Agent 不是"一问一答"，而是"循环决策直到任务完成"。这就是它比普通 LLM 调用强大的根本原因。

### 速记口诀

> **问题进来转圈圈，模型想完工具干，信息够了才停转，不够继续再一遍。**

### 关键细节

```
┌──────────────────────────────────────┐
│            Agent 核心循环             │
│                                      │
│  用户输入 ──→ 模型决策 ──→ 需要工具？ │
│                            │    │    │
│                           是    否   │
│                            │    │    │
│                            ▼    ▼    │
│                        调用工具  返回  │
│                            │  结果   │
│                            ▼         │
│                      结果加入消息      │
│                            │         │
│                            ▼         │
│                      回到模型决策      │
└──────────────────────────────────────┘
```

## 卡片 3：model 参数的灵活性

### 一句话

**model 参数支持字符串标识符和模型实例两种方式，字符串更简洁，实例更灵活。**

### 举例

```python
# 方式 1：字符串标识符（推荐，简洁）
agent = create_agent("openai:gpt-4o", tools=[])
agent = create_agent("anthropic:claude-sonnet-4-5-20250929", tools=[])
agent = create_agent("google_genai:gemini-2.0-flash", tools=[])

# 方式 2：模型实例（需要精细控制时）
from langchain_openai import ChatOpenAI
model = ChatOpenAI(model="gpt-4o", temperature=0, max_tokens=4096)
agent = create_agent(model, tools=[])
```

### 应用场景

快速切换模型提供商。开发时用便宜的 `gpt-4o-mini`，上线时换 `gpt-4o`，只改一个字符串。需要调 temperature 等参数时，切换到实例模式。

### 速记口诀

> **字符串快速切，实例精细调。格式是 `provider:model_name`，冒号左边是厂商，右边是型号。**

### 关键细节

| 方式 | 格式 | 适用场景 |
|------|------|----------|
| 字符串 | `"openai:gpt-4o"` | 快速原型、切换模型 |
| 实例 | `ChatOpenAI(...)` | 精细控制参数 |
| 内部机制 | 字符串通过 `init_chat_model()` 转换为实例 | 两种方式最终等价 |

## 卡片 4：工具定义三要素

### 一句话

**类型注解 + docstring + 返回值 = 完整工具。普通 Python 函数就是 Agent 工具，零装饰器。**

### 举例

```python
def search_web(query: str) -> str:
    """Search the web for current information.

    Args:
        query: The search query string.
    """
    # 实际调用搜索 API
    return f"搜索结果：关于 '{query}' 的最新信息..."

# 直接传入，不需要任何包装
agent = create_agent("openai:gpt-4o", tools=[search_web])
```

### 应用场景

将任何 Python 函数变成 Agent 工具。数据库查询、API 调用、文件操作——只要是函数，就能变成工具。

### 速记口诀

> **三件套：类型注解告诉模型参数是啥，docstring 告诉模型工具干啥，返回值告诉模型结果是啥。**

### 关键细节

| 要素 | 作用 | 缺少会怎样 |
|------|------|------------|
| 类型注解 `query: str` | 告诉模型参数类型，生成正确的调用 | 模型不知道传什么参数 |
| docstring `"""..."""` | 告诉模型工具的功能描述 | 模型不知道什么时候该用这个工具 |
| 返回值 `-> str` | 告诉模型期望的返回格式 | 模型无法正确解析结果 |

### 反面教材

```python
# ❌ 错误：没有类型注解和 docstring
def search(query):
    return "结果"

# ✅ 正确：三要素齐全
def search(query: str) -> str:
    """Search the web for information.

    Args:
        query: The search query string.
    """
    return "结果"
```

## 卡片 5：Middleware 的洋葱模型

### 一句话

**多个 Middleware 像洋葱一样嵌套执行：外层先进后出，内层后进先出。**

### 举例

```python
middleware = [LoggingMiddleware, RetryMiddleware, CacheMiddleware]

# 执行顺序（洋葱模型）：
# Logging.before_model  →  进入
#   Retry.before_model  →  进入
#     Cache.before_model  →  进入
#       [模型调用]         →  核心
#     Cache.after_model   →  出来
#   Retry.after_model   →  出来
# Logging.after_model   →  出来
```

### 应用场景

理解 Middleware 的执行顺序。日志放最外层（第一个记录、最后一个记录），重试放中间层，缓存放最内层（最接近模型调用）。

### 速记口诀

> **洋葱模型记住仨：列表顺序从外到内，before 从外往里走，after 从里往外走。**

### 关键细节

```
请求方向 ──────────────────────────────→

┌─ Logging ─────────────────────────────┐
│ before                         after  │
│  ┌─ Retry ────────────────────────┐   │
│  │ before                  after  │   │
│  │  ┌─ Cache ─────────────────┐   │   │
│  │  │ before          after   │   │   │
│  │  │        [模型调用]        │   │   │
│  │  └─────────────────────────┘   │   │
│  └────────────────────────────────┘   │
└───────────────────────────────────────┘

←────────────────────────────── 响应方向
```

### 类比

就像你寄快递：
- 外层包装（Logging）最先包上、最后拆开
- 中间缓冲（Retry）第二层
- 内层保护（Cache）最贴近物品

## 卡片 6：6 种钩子速记

### 一句话

**4 个生命周期钩子 + 2 个包装器钩子 = 完整的 Agent 控制力。**

### 举例

```python
class MyMiddleware(AgentMiddleware):
    # ── 4 个生命周期钩子（在特定时刻触发）──
    def before_agent(self, state, config):     # Agent 循环开始前
        ...
    def after_agent(self, state, config):      # Agent 循环结束后
        ...
    def before_model(self, request, config):   # 每次模型调用前
        ...
    def after_model(self, response, config):   # 每次模型调用后
        ...

    # ── 2 个包装器钩子（包裹整个调用过程）──
    def wrap_model_call(self, request, config, call_next):
        # 可以修改请求、拦截响应、实现重试/缓存
        response = call_next(request, config)
        return response

    def wrap_tool_call(self, tool_call, config, call_next):
        # 可以修改工具参数、拦截工具结果
        result = call_next(tool_call, config)
        return result
```

### 应用场景

选择正确的钩子实现自定义逻辑。记住这个选择指南：

| 我想要... | 用哪个钩子 |
|-----------|-----------|
| 注入系统提示词 | `before_model` |
| 记录日志 | `before_agent` + `after_agent` |
| 实现重试 | `wrap_model_call` |
| 工具权限控制 | `wrap_tool_call` |
| 修改模型输出 | `after_model` |
| 添加对话记忆 | `before_agent` |

### 速记口诀

> **生命周期管时机，包装器管过程。before/after 只能看和改，wrap 能拦截和重试。**

### 关键区别

```
生命周期钩子：  ──before──→ [执行] ──after──→
                 只能修改输入    只能修改输出

包装器钩子：    ──wrap─────────────────────→
                 包裹整个过程，可以重试、缓存、拦截
```

## 卡片 7：AgentState 的三个字段

### 一句话

**messages（对话记录）+ jump_to（流程控制）+ structured_response（结构化输出）= Agent 的全部状态。**

### 举例

```python
# AgentState 的 TypedDict 定义
class AgentState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]  # 核心：对话历史
    jump_to: Optional[str]                                # 可选：跳转到指定节点
    structured_response: Optional[Any]                    # 可选：结构化输出

# 使用方式
result = agent.invoke({"messages": [("user", "你好")]})

# 获取最新回复
last_message = result["messages"][-1]
print(last_message.content)

# 如果设置了 response_format，获取结构化输出
structured = result.get("structured_response")
```

### 应用场景

理解 Agent 的状态管理。所有信息都在 `messages` 里流转——用户输入、模型回复、工具调用结果，全部是消息。

### 速记口诀

> **messages 是主角，99% 的时间只用它。jump_to 控流程，structured_response 管格式。**

### 关键细节

| 字段 | 类型 | 用途 | 使用频率 |
|------|------|------|----------|
| `messages` | `list[AnyMessage]` | 对话历史，Agent 的核心数据 | 每次都用 |
| `jump_to` | `Optional[str]` | 跳转到指定图节点（高级用法） | 很少用 |
| `structured_response` | `Optional[Any]` | 结构化输出结果 | 设置 response_format 时用 |

## 卡片 8：override() 不可变模式

### 一句话

**修改 ModelRequest 必须用 override()，不能直接赋值。这是函数式编程的不可变数据思想。**

### 举例

```python
def wrap_model_call(self, request, config, call_next):
    # ❌ 错误：直接修改原始请求
    request.messages.append(system_msg)  # 会影响其他 Middleware！

    # ✅ 正确：用 override() 创建新请求
    new_request = request.override(
        messages=[system_msg] + request.messages,
        model="openai:gpt-4o",  # 可以动态切换模型
        tools=request.tools + [extra_tool],  # 可以动态添加工具
    )
    return call_next(new_request, config)
```

### 应用场景

在 `wrap_model_call` 中安全修改请求。多个 Middleware 共享同一个请求对象，直接修改会导致不可预测的副作用。

### 速记口诀

> **不改原件改副本，override 出新对象。多个中间件共享数据，不可变才安全。**

### 关键细节

| 操作 | 方法 | 安全性 |
|------|------|--------|
| 修改消息列表 | `request.override(messages=[...])` | 安全 |
| 切换模型 | `request.override(model="...")` | 安全 |
| 增减工具 | `request.override(tools=[...])` | 安全 |
| 直接赋值 | `request.messages = [...]` | 危险，会影响其他 Middleware |

### 类比

就像 Git：你不会直接修改上一个 commit，而是基于它创建新 commit。`override()` 就是 Agent 世界的 `git commit`。

## 卡片 9：结构化输出三策略

### 一句话

**AutoStrategy（自动选择）、ToolStrategy（工具模式）、ProviderStrategy（原生模式）——三种策略让 Agent 返回结构化数据。**

### 举例

```python
from pydantic import BaseModel

class WeatherReport(BaseModel):
    city: str
    temperature: float
    condition: str
    suggestion: str

# 只需传入 Pydantic 模型，自动选择最佳策略
agent = create_agent(
    model="openai:gpt-4o",
    tools=[weather_tool],
    response_format=WeatherReport,  # 就这么简单
)

result = agent.invoke({"messages": [("user", "北京天气")]})
report = result["structured_response"]  # WeatherReport 实例
print(report.temperature)  # 25.0
print(report.condition)    # "晴"
```

### 应用场景

让 Agent 返回 Pydantic 模型而非纯文本。API 服务需要 JSON 响应、数据提取需要固定格式、工作流需要结构化中间结果——都用 `response_format`。

### 速记口诀

> **Auto 最省心，Tool 最兼容，Provider 最高效。不知道选哪个？用 Auto。**

### 三种策略对比

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| AutoStrategy | 自动检测模型能力，选最佳策略 | 零配置 | 可能不是最优 |
| ToolStrategy | 把输出格式伪装成工具让模型调用 | 兼容所有模型 | 多一次工具调用开销 |
| ProviderStrategy | 使用模型原生的 JSON Mode | 最高效 | 不是所有模型都支持 |

## 卡片 10：从 AgentExecutor 到 create_agent

### 一句话

**一个函数取代多种 Agent 类型，Middleware 取代参数堆叠。迁移核心：改输入格式 + 换定制方式。**

### 举例

```python
# ── 旧版（LangChain 0.x）──
from langchain.agents import AgentExecutor, create_react_agent
from langchain.prompts import PromptTemplate

prompt = PromptTemplate.from_template("...")
agent_logic = create_react_agent(llm, tools, prompt)
executor = AgentExecutor(
    agent=agent_logic,
    tools=tools,
    max_iterations=10,
    early_stopping_method="force",
    handle_parsing_errors=True,
    verbose=True,
)
result = executor.invoke({"input": "查询天气"})

# ── 新版（LangChain 1.0）──
from langchain.agents import create_agent

agent = create_agent(
    model="openai:gpt-4o",
    tools=tools,
    system_prompt="你是一个助手",
    middleware=[LoggingMiddleware, RetryMiddleware],
)
result = agent.invoke({"messages": [("user", "查询天气")]})
```

### 应用场景

迁移旧版代码到 LangChain 1.0。核心变化就三点：

### 迁移速查表

| 旧版 | 新版 | 说明 |
|------|------|------|
| `AgentExecutor(agent, tools, ...)` | `create_agent(model, tools, ...)` | 两步变一步 |
| `{"input": "问题"}` | `{"messages": [("user", "问题")]}` | 输入格式变了 |
| `max_iterations=10` | `recursion_limit=25` | 参数名变了 |
| `verbose=True` | `middleware=[LoggingMiddleware]` | 日志用中间件 |
| `handle_parsing_errors=True` | 内置自动处理 | 不需要了 |
| `Memory` 对象 | `checkpointer` 参数 | 持久化方式变了 |

### 速记口诀

> **两步变一步，input 变 messages，参数变中间件，Memory 变 checkpointer。**
