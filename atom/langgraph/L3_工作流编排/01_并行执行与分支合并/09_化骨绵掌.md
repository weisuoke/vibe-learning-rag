## 卡片1：并行执行的本质

**一句话：** 并行执行是让多个节点在同一时刻同时运行，而不是按顺序一个接一个执行。

**核心要点：**
- **串行执行**：A → B → C，总耗时 = 时间A + 时间B + 时间C
- **并行执行**：A → (B, C)，总耗时 = 时间A + max(时间B, 时间C)
- **关键价值**：减少总执行时间，提高系统吞吐量

**举例：**
```python
# 串行：6秒
task1()  # 2秒
task2()  # 2秒
task3()  # 2秒

# 并行：2秒
parallel_execute([task1, task2, task3])
```

**应用：** 在 LangGraph 中，当多个任务相互独立时（如并行调用多个 LLM、并行处理多个文档），使用并行执行可以显著提升性能。

**类比：** 就像餐厅厨房，一个厨师做三道菜需要30分钟，三个厨师同时做只需要10分钟。

## 卡片2：Send 类的核心作用

**一句话：** Send 类是 LangGraph 实现动态并行的核心工具，用于在运行时动态创建并行任务。

**核心要点：**
- **定义**：`Send(node, arg)` - 指定目标节点和输入状态
- **动态性**：运行时根据数据决定并行任务数量
- **灵活性**：每个任务可以接收不同的输入

**举例：**
```python
from langgraph.types import Send

# 为每个文档创建一个并行任务
def route(state):
    return [
        Send("summarize", {"doc": doc})
        for doc in state["documents"]
    ]
```

**应用：** 当并行任务数量取决于运行时数据时（如文档数量、用户查询数量），使用 Send API 实现动态并行。

**类比：** 就像外卖平台根据订单数量动态分配骑手，订单多就派更多骑手。

## 卡片3：Fan-out/Fan-in 模式

**一句话：** Fan-out 是从一个节点分支到多个并行节点，Fan-in 是多个并行节点汇聚到一个节点。

**核心要点：**
- **Fan-out（扇出）**：1 → N，任务分发
- **Fan-in（扇入）**：N → 1，结果汇总
- **同步点**：扇入节点等待所有并行节点完成

**举例：**
```python
# 图结构
    A
   / \
  B   C
   \ /
    D

# B 和 C 并行执行，D 等待两者都完成
```

**应用：** 并行 LLM 调用（扇出到多个 LLM，扇入汇总结果）、多数据源获取（扇出到多个 API，扇入合并数据）。

**类比：** 就像快递分拣中心，包裹从总部（扇出）分发到各个站点，再从各站点（扇入）汇总到配送员。

## 卡片4：状态 Reducer 机制

**一句话：** Reducer 函数定义如何合并多个并行节点返回的状态更新，避免状态覆盖问题。

**核心要点：**
- **问题**：多个并行节点同时更新同一字段，会覆盖
- **解决**：使用 `Annotated[type, reducer]` 指定合并策略
- **常用 Reducer**：`operator.add`（列表追加、数值相加）

**举例：**
```python
from typing import Annotated
import operator

class State(TypedDict):
    # 使用 operator.add 自动追加
    results: Annotated[list, operator.add]

# 节点1返回 {"results": ["A"]}
# 节点2返回 {"results": ["B"]}
# 自动合并为 {"results": ["A", "B"]}
```

**应用：** 所有并行执行场景都需要 Reducer，确保并行节点的结果都被保留。

**类比：** 就像多个服务员同时接收订单，订单汇总系统会把所有订单合并到一个列表，而不是只保留最后一个。

## 卡片5：动态并行 vs 静态并行

**一句话：** 静态并行在编译时确定并行度，动态并行在运行时确定并行度。

**核心要点：**

| 特性 | 静态并行 | 动态并行 |
|------|----------|----------|
| **并行度** | 编译时确定 | 运行时确定 |
| **实现方式** | 多条 `add_edge` | `Send` API |
| **性能** | 更好 | 稍慢 |
| **灵活性** | 固定 | 动态调整 |

**举例：**
```python
# 静态并行：固定3个任务
builder.add_edge(START, "task1")
builder.add_edge(START, "task2")
builder.add_edge(START, "task3")

# 动态并行：根据数据决定任务数
def route(state):
    return [Send("task", {"item": i}) for i in state["items"]]
```

**应用：** 固定任务数量用静态并行（性能更好），动态任务数量用 Send API（更灵活）。

**类比：** 静态并行像固定班次的公交车，动态并行像根据乘客数量调度的网约车。

## 卡片6：Map-Reduce 模式

**一句话：** Map-Reduce 是将数据分发到多个并行任务处理（Map），然后聚合所有结果（Reduce）的经典模式。

**核心要点：**
- **Map 阶段**：将大任务拆分成多个小任务并行执行
- **Reduce 阶段**：聚合所有小任务的结果
- **实现**：使用 Send API + Reducer 函数

**举例：**
```python
# Map：为每个主题生成笑话
def continue_to_jokes(state):
    return [
        Send("generate_joke", {"subject": s})
        for s in state["subjects"]
    ]

# Reduce：选择最佳笑话
def best_joke(state):
    return {"best": max(state["jokes"], key=len)}
```

**应用：** 文档批量摘要、多智能体协作、词频统计、批量数据处理。

**类比：** 就像工厂流水线，原材料（Map）分配给多个工人加工，最后（Reduce）质检员汇总检查所有产品。

## 卡片7：BSP 模型与超步

**一句话：** LangGraph 基于 Bulk Synchronous Parallel (BSP) 模型，将执行分为多个超步，每个超步内并行执行，超步间同步。

**核心要点：**
- **超步（Superstep）**：一个执行阶段，所有选中的节点并行执行
- **同步点（Barrier）**：超步之间的同步点，确保所有节点完成
- **保证**：同一超步内的节点一定并行，不同超步间一定串行

**举例：**
```
超步1: A 执行
       ↓ (同步点)
超步2: B, C 并行执行
       ↓ (同步点)
超步3: D 执行
```

**应用：** 理解 BSP 模型有助于预测图的执行顺序和性能特征。

**类比：** 就像接力赛，每一棒（超步）内多个选手可以并行跑，但必须等所有选手到达交接区（同步点）才能开始下一棒。

## 卡片8：条件边与动态路由

**一句话：** 条件边根据状态动态决定下一步执行哪些节点，返回 Send 对象列表时触发并行执行。

**核心要点：**
- **条件函数**：根据状态返回节点名称或 Send 对象列表
- **动态决策**：运行时根据数据决定并行策略
- **混合路由**：可以同时返回 Send 对象和节点名称

**举例：**
```python
def route(state):
    query = state["query"]

    # 根据查询内容决定并行调用哪些工具
    if "weather" in query and "news" in query:
        return [
            Send("weather_tool", {"location": "..."}),
            Send("news_tool", {"topic": "..."})
        ]
    elif "weather" in query:
        return [Send("weather_tool", {"location": "..."})]
    else:
        return "answer"  # 直接回答，不调用工具
```

**应用：** 智能路由、多工具调用、复杂决策流程。

**类比：** 就像智能导航，根据实时路况（状态）动态选择走哪些路（并行任务）。

## 卡片9：性能优化策略

**一句话：** 并行执行需要权衡并行度、内存使用和同步开销，合理控制才能获得最佳性能。

**核心要点：**
- **并行度控制**：限制最大并行任务数量，避免资源耗尽
- **批处理**：将大量小任务合并成少量大任务
- **选择策略**：小规模用静态并行，大规模用动态并行 + 批处理

**举例：**
```python
# 控制并行度
def route_with_limit(state):
    items = state["items"][:100]  # 最多100个并行任务
    return [Send("process", {"item": i}) for i in items]

# 批处理
def route_batched(state):
    items = state["items"]
    batches = [items[i:i+10] for i in range(0, len(items), 10)]
    return [Send("process_batch", {"batch": b}) for b in batches]
```

**应用：** 大规模并行任务（如处理10000个文档）时，必须使用批处理和并行度控制。

**类比：** 就像餐厅接单，不能无限接单（会爆单），要根据厨房容量（资源）限制同时处理的订单数。

## 卡片10：实际应用场景

**一句话：** 并行执行在 RAG、多智能体、数据处理等场景中都有广泛应用。

**核心场景：**

**1. 并行 LLM 调用**
```python
# 同时生成笑话、故事、诗歌
builder.add_edge(START, "generate_joke")
builder.add_edge(START, "generate_story")
builder.add_edge(START, "generate_poem")
```

**2. 文档批量处理**
```python
# 为每个文档生成摘要
def route(state):
    return [Send("summarize", {"doc": d}) for d in state["docs"]]
```

**3. 多数据源获取**
```python
# 并行获取新闻、天气、股票数据
builder.add_edge(START, "fetch_news")
builder.add_edge(START, "fetch_weather")
builder.add_edge(START, "fetch_stocks")
```

**4. 多智能体协作**
```python
# 研究员、编码员、写作员并行工作
def route(state):
    return [
        Send("research_agent", state),
        Send("code_agent", state),
        Send("writer_agent", state)
    ]
```

**应用：** 识别任务中的独立部分，使用并行执行提升性能。

**类比：** 就像项目管理，识别可以并行进行的任务（如前端和后端开发），合理分配资源加速项目进度。

### 核心公式

**并行加速比**：
```
加速比 = 串行时间 / 并行时间
理想加速比 = N（N个并行任务）
实际加速比 < N（因为同步开销）
```

**并行度选择**：
```
小规模（< 10个任务）→ 静态并行
中规模（10-100个任务）→ 动态并行
大规模（> 100个任务）→ 动态并行 + 批处理
```

### 快速检查清单

**使用并行执行前检查：**
- [ ] 任务之间是否相互独立？
- [ ] 并行能否带来性能提升？
- [ ] 是否需要合并并行结果？
- [ ] 并行度是否可控？
- [ ] 是否有足够的资源？

**选择并行方式：**
- [ ] 任务数量固定 → 静态并行
- [ ] 任务数量动态 → Send API
- [ ] 需要不同输入 → Send API
- [ ] 追求最佳性能 → 静态并行

**状态管理：**
- [ ] 是否定义了 Reducer？
- [ ] Reducer 是否正确？
- [ ] 是否避免了状态覆盖？

### 常见错误速查

**错误1：忘记定义 Reducer**
```python
# ❌ 错误：结果会被覆盖
results: list

# ✅ 正确：使用 Reducer
results: Annotated[list, operator.add]
```

**错误2：无限制并行**
```python
# ❌ 错误：可能创建10000个并行任务
return [Send("task", {"i": i}) for i in range(10000)]

# ✅ 正确：限制并行度
return [Send("task", {"i": i}) for i in range(min(100, len(items)))]
```

**错误3：Reducer 访问状态**
```python
# ❌ 错误：Reducer 不能访问完整状态
def wrong_reducer(current, new, state):  # 没有 state 参数
    return current + new

# ✅ 正确：Reducer 只接收两个参数
def correct_reducer(current, new):
    return current + new
```

### 性能优化速查

**优化策略：**
1. **小规模并行**：使用静态并行
2. **大规模并行**：使用批处理
3. **混合场景**：静态 + 动态结合
4. **内存优化**：只传递必要的状态字段
5. **监控调优**：记录执行时间，找出瓶颈

**性能对比：**
```
静态并行（3个任务）：~0ms 额外开销
动态并行（3个任务）：~5ms 额外开销
动态并行（100个任务）：~50ms 额外开销
动态并行（1000个任务）：~500ms 额外开销
```

### 实战技巧

**技巧1：渐进式并行**
```python
# 先串行验证逻辑，再改并行
# 串行版本

builder.add_edge("b", "c")

# 并行版本
```

**技巧2：错误隔离**
```python
def safe_task(state):
    try:
        result = risky_operation(state["item"])
        return {"results": [{"success": True, "data": result}]}
    except Exception as e:
        return {"results": [{"success": False, "error": str(e)}]}
```

**技巧3：进度监控**
```python
def task_with_progress(state):
    item = state["item"]
    print(f"Processing {item}...")
    result = process(item)
    print(f"Completed {item}")
    return {"results": [result]}
```
