# 核心概念 6：标量索引优化

> **目标**：掌握 Milvus 标量索引的选择策略和性能优化技巧

## 概述

标量索引是提升过滤查询性能的关键，Milvus 提供多种索引类型，合理选择和配置索引可实现 30 倍性能提升。

**核心价值**：
- 30x 性能提升
- 灵活的索引选择
- 自动索引优化
- 生产级性能保证

## 1. 索引类型概览

### 1.1 支持的索引类型

**完整列表**：
- **INVERTED**：倒排索引
- **BITMAP**：位图索引
- **STL_SORT**：排序数组
- **AUTOINDEX**：自动选择
- **TRIE**：前缀树（仅 VARCHAR）
- **NGRAM**：N-gram 索引（仅 VARCHAR）

### 1.2 数据类型支持矩阵

| 数据类型 | 支持的索引类型 |
|---------|---------------|
| **INT8/16/32/64** | INVERTED, BITMAP, STL_SORT, AUTOINDEX |
| **BOOL** | INVERTED, BITMAP, AUTOINDEX |
| **FLOAT/DOUBLE** | INVERTED, STL_SORT, AUTOINDEX |
| **VARCHAR** | INVERTED, BITMAP, TRIE, NGRAM, AUTOINDEX |
| **JSON** | INVERTED, NGRAM, AUTOINDEX |
| **ARRAY** | INVERTED, BITMAP, AUTOINDEX |

## 2. INVERTED 索引

### 2.1 基本原理

**定义**：
- 倒排索引结构
- 值到文档 ID 的映射
- 适用于精确匹配和范围查询

**数据结构**：
```
Value → Document IDs (Posting List)
25 → [1, 5, 10, 15, 20]
30 → [2, 6, 11, 16, 21]
35 → [3, 7, 12, 17, 22]
```

### 2.2 适用场景

**精确匹配**：
```python
"age == 25"
"city == '北京'"
```

**范围查询**：
```python
"age >= 25 AND age <= 35"
"price >= 100 AND price <= 500"
```

**集合查询**：
```python
"age IN [25, 30, 35]"
"city IN ['北京', '上海', '深圳']"
```

### 2.3 性能特点

**优势**：
- 查询速度快（O(log n)）
- 支持范围查询
- 适用于大多数场景

**劣势**：
- 内存占用较大
- 构建时间较长

**推荐场景**：
- 中高基数字段
- 需要范围查询
- 生产环境默认选择

[来源: reference/search_web_results.md | Milvus 2.6 官方博客]

## 3. BITMAP 索引

### 3.1 基本原理

**定义**：
- 位图索引结构
- 每个值对应一个位图
- 适用于低基数字段

**数据结构**：
```
Value → Bitmap
'active' → [1, 0, 1, 0, 1, 0, 1, 0]
'pending' → [0, 1, 0, 1, 0, 1, 0, 1]
'deleted' → [0, 0, 0, 0, 0, 0, 0, 0]
```

### 3.2 适用场景

**低基数字段**：
```python
# 状态字段（只有几个值）
"status IN ['active', 'pending', 'approved']"

# 布尔字段
"is_vip == true"

# 性别字段
"gender == 'male'"
```

### 3.3 性能特点

**优势**：
- 内存占用小
- 查询速度极快（O(1)）
- 支持高效的位运算

**劣势**：
- 仅适用于低基数字段
- 高基数字段会浪费内存

**推荐场景**：
- 基数 < 100 的字段
- 布尔字段
- 状态字段

**基数判断**：
```python
# 低基数（推荐 BITMAP）
status: ['active', 'pending', 'deleted']  # 3 个值
gender: ['male', 'female', 'other']  # 3 个值
vip_level: [1, 2, 3, 4, 5]  # 5 个值

# 高基数（不推荐 BITMAP）
user_id: [1, 2, 3, ..., 1000000]  # 100 万个值
email: ['user1@example.com', ...]  # 数百万个值
```

## 4. STL_SORT 索引

### 4.1 基本原理

**定义**：
- 排序数组结构
- 使用二分查找
- 适用于范围查询

**数据结构**：
```
Sorted Array: [10, 15, 20, 25, 30, 35, 40, 45, 50]
Binary Search: O(log n)
```

### 4.2 适用场景

**范围查询**：
```python
"age >= 25 AND age <= 35"
"price >= 100 AND price <= 500"
"timestamp > 1640995200"
```

### 4.3 性能特点

**优势**：
- 范围查询性能好
- 内存占用适中
- 实现简单

**劣势**：
- 精确匹配不如 INVERTED
- 更新成本高

**推荐场景**：
- 主要进行范围查询
- 数值类型字段
- 时间戳字段

---

## 5. AUTOINDEX 索引

### 5.1 基本原理

**定义**：
- 自动选择最优索引
- 根据数据特征决策
- 简化索引配置

**决策逻辑**：
```
IF 基数 < 100:
    选择 BITMAP
ELSE IF 数据类型 == VARCHAR AND 查询模式 == 前缀匹配:
    选择 TRIE
ELSE:
    选择 INVERTED
```

### 5.2 适用场景

**不确定索引类型**：
```python
# 让 Milvus 自动选择
index_params.add_index(
    field_name="age",
    index_type="AUTOINDEX"
)
```

### 5.3 性能特点

**优势**：
- 简化配置
- 自动优化
- 适应数据变化

**劣势**：
- 可能不是最优选择
- 缺乏控制

**推荐场景**：
- 快速原型开发
- 不确定数据特征
- 简化运维

[来源: reference/source_test_files.md | test_milvus_client_scalar_filtering.py]

## 6. TRIE 索引

### 6.1 基本原理

**定义**：
- 前缀树结构
- 适用于前缀匹配
- 仅支持 VARCHAR 类型

**数据结构**：
```
Trie Tree:
    root
    ├── J
    │   └── o
    │       └── h
    │           └── n (John, Johnny, Johnson)
    └── A
        └── d
            └── m
                └── i
                    └── n (Admin, Administrator)
```

### 6.2 适用场景

**前缀匹配**：
```python
"name LIKE 'John%'"
"email LIKE 'admin%'"
"url LIKE 'https://example.com/%'"
```

### 6.3 性能特点

**优势**：
- 前缀匹配性能极好
- 内存占用适中

**劣势**：
- 仅支持前缀匹配
- 不支持后缀或包含匹配

**推荐场景**：
- 自动补全
- 前缀搜索
- URL 匹配

---

## 7. NGRAM 索引

### 7.1 基本原理

**定义**：
- N-gram 分词索引
- 适用于模糊匹配
- 仅支持 VARCHAR 和 JSON 类型

**数据结构**：
```
Text: "Python"
Bigrams (N=2): ["Py", "yt", "th", "ho", "on"]
Trigrams (N=3): ["Pyt", "yth", "tho", "hon"]
```

### 7.2 适用场景

**模糊匹配**：
```python
"title LIKE '%Python%'"
"description LIKE '%urgent%'"
"content LIKE '%machine learning%'"
```

### 7.3 性能特点

**优势**：
- 支持模糊匹配
- 支持包含查询

**劣势**：
- 内存占用大
- 构建时间长
- 查询性能一般

**推荐场景**：
- 全文搜索
- 模糊匹配
- 内容检索

## 8. 索引选择策略

### 8.1 按查询模式选择

| 查询模式 | 推荐索引 | 示例 |
|---------|---------|------|
| **精确匹配** | INVERTED | `age == 25` |
| **范围查询** | INVERTED, STL_SORT | `age >= 25 AND age <= 35` |
| **低基数** | BITMAP | `status IN ['active', 'pending']` |
| **前缀匹配** | TRIE | `name LIKE 'John%'` |
| **模糊匹配** | NGRAM | `title LIKE '%Python%'` |
| **不确定** | AUTOINDEX | 自动选择 |

### 8.2 按数据特征选择

**基数判断**：
```python
# 低基数（< 100）→ BITMAP
基数 = DISTINCT(field) / COUNT(*)
IF 基数 < 0.01:  # 1% 以下
    使用 BITMAP

# 中高基数 → INVERTED
ELSE:
    使用 INVERTED
```

**数据类型判断**：
```python
# 布尔类型 → BITMAP
IF type == BOOL:
    使用 BITMAP

# 字符串类型 → 根据查询模式
ELIF type == VARCHAR:
    IF 查询模式 == 前缀匹配:
        使用 TRIE
    ELIF 查询模式 == 模糊匹配:
        使用 NGRAM
    ELSE:
        使用 INVERTED

# 数值类型 → INVERTED 或 STL_SORT
ELSE:
    使用 INVERTED
```

### 8.3 按性能要求选择

**高性能要求**：
```python
# 查询延迟 < 10ms
使用 BITMAP（低基数）或 INVERTED（中高基数）
```

**内存受限**：
```python
# 内存占用 < 1GB
使用 BITMAP（低基数）或 STL_SORT（中高基数）
```

**构建时间受限**：
```python
# 构建时间 < 1分钟
使用 STL_SORT 或 AUTOINDEX
```

## 9. 性能优化技巧

### 9.1 索引创建优化

**为常用字段创建索引**：
```python
# 高频查询字段
index_params.add_index("age", "INVERTED")
index_params.add_index("city", "BITMAP")

# 低频查询字段（可以不建索引）
# 不为 rare_field 创建索引
```

**批量创建索引**：
```python
# 一次性创建多个索引
index_params = client.prepare_index_params()
index_params.add_index("age", "INVERTED")
index_params.add_index("city", "BITMAP")
index_params.add_index("status", "BITMAP")
client.create_index(collection_name, index_params)
```

### 9.2 查询优化

**选择性高的条件在前**：
```python
# 好的写法（选择性高的条件在前）
"stock > 0 AND price >= 100 AND price <= 500"
# 假设 stock > 0 只保留 10% 数据

# 不好的写法（选择性低的条件在前）
"price >= 100 AND price <= 500 AND stock > 0"
# 假设价格范围保留 80% 数据
```

**使用 IN 替代多个 OR**：
```python
# 不推荐
"city == '北京' OR city == '上海' OR city == '深圳'"

# 推荐
"city IN ['北京', '上海', '深圳']"
```

### 9.3 索引维护

**定期重建索引**：
```python
# 数据量变化较大时重建索引
client.drop_index(collection_name, "age_index")
client.create_index(collection_name, index_params)
```

## 10. 性能对比

### 10.1 查询性能

**测试场景**：1百万条记录，点查询

| 索引类型 | 查询延迟 | 性能提升 |
|---------|---------|---------|
| 无索引 | 1000ms | 1x |
| INVERTED | 30ms | 33x |
| BITMAP | 10ms | 100x |
| STL_SORT | 50ms | 20x |

**结论**：
- BITMAP 性能最好（低基数）
- INVERTED 性能优秀（通用）
- 有索引比无索引快 30-100 倍

[来源: reference/search_web_results.md | Milvus 2.6 官方博客]

### 10.2 内存占用

**测试场景**：1百万条记录

| 索引类型 | 内存占用 | 额外开销 |
|---------|---------|---------|
| 无索引 | 100MB | 0% |
| INVERTED | 130MB | 30% |
| BITMAP | 110MB | 10% |
| STL_SORT | 120MB | 20% |

**结论**：
- BITMAP 内存占用最小（低基数）
- INVERTED 内存占用适中
- 索引额外开销 10-30%

### 10.3 构建时间

**测试场景**：1百万条记录

| 索引类型 | 构建时间 |
|---------|---------|
| INVERTED | 10s |
| BITMAP | 5s |
| STL_SORT | 3s |
| TRIE | 15s |
| NGRAM | 30s |

**结论**：
- STL_SORT 构建最快
- NGRAM 构建最慢
- 构建时间与索引复杂度成正比

## 12. 类比理解

### 数据库索引类比

**INVERTED = B-Tree 索引**：
```sql
-- SQL
CREATE INDEX idx_age ON users(age);
-- 适用于精确匹配和范围查询

-- Milvus
index_params.add_index("age", "INVERTED")
```

**BITMAP = 位图索引**：
```sql
-- SQL
CREATE BITMAP INDEX idx_status ON users(status);
-- 适用于低基数字段

-- Milvus
index_params.add_index("status", "BITMAP")
```

### 前端开发类比

**INVERTED = Map 查找**：
```javascript
// JavaScript Map
const ageMap = new Map();
ageMap.set(25, [1, 5, 10]);
const result = ageMap.get(25);  // O(1)

// Milvus INVERTED
index_params.add_index("age", "INVERTED")
```

**BITMAP = Set 查找**：
```javascript
// JavaScript Set
const activeUsers = new Set([1, 3, 5, 7]);
const isActive = activeUsers.has(5);  // O(1)

// Milvus BITMAP
index_params.add_index("is_active", "BITMAP")
```
