## 问题1："在 LangGraph 中，如何调试一个复杂的多步骤工作流？请描述你的调试策略。"

### 普通回答（❌ 不出彩）

"我会在节点里加 print 语句，看看每一步的输出是什么。"

**问题**：只提到了最原始的调试手段，没有展示对 LangGraph 调试体系的理解，面试官会觉得你只停留在"能跑就行"的阶段。

### 出彩回答（✅ 推荐）

> **LangGraph 的调试策略分为四个层次，从轻量到重量逐步升级：**
>
> **1. 实时流式调试**：使用 `stream_mode` 参数控制调试粒度
> - `"updates"` 追踪增量变化——哪个节点改了什么字段
> - `"values"` 查看每步后的完整状态快照
> - `"debug"` 获取最详细的执行信息——它实际上是 `checkpoints` + `tasks` 的组合（这在源码 `_loop.py` 的 `_emit()` 方法中可以看到），每个事件包含 `step`、`timestamp`、`type`、`payload` 四个字段
>
> **2. 状态快照检查**：通过 `get_state()` 获取 `StateSnapshot`
> - 检查 `values`（当前状态值）、`next`（下一步节点）、`tasks`（任务详情）
> - 可以指定 `checkpoint_id` 检查任意历史状态
> - `StateSnapshot` 是一个 8 字段的 NamedTuple，由内部的 `_prepare_state_snapshot()` 从底层 `CheckpointTuple` 构建而来
>
> **3. 历史追溯与时间旅行**：使用 `get_state_history()` 遍历完整执行历史
> - 返回按时间倒序排列的 `StateSnapshot` 迭代器
> - 定位状态异常的精确步骤——对比相邻快照就能发现哪一步出了问题
> - 通过 `update_state()` 修改历史状态后重新执行（Fork 模式），或直接从历史 checkpoint 继续执行（Replay 模式）
>
> **4. 生产级可观测性**：集成 LangSmith
> - 非侵入式追踪——只需设置 `LANGSMITH_TRACING=true` 和 `LANGSMITH_API_KEY` 两个环境变量
> - 通过 `config` 的 `tags` 和 `metadata` 关联用户和会话
> - Web UI 可视化执行路径，支持团队协作排查
>
> **在实际项目中**，我通常开发阶段用 `stream_mode="debug"` + `get_state_history()` 快速定位问题，生产环境用 LangSmith + `stream_mode="custom"` 配合 `get_stream_writer()` 输出关键业务指标。

### 代码示例——四层调试实战

```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.config import get_stream_writer

class RAGState(TypedDict):
    query: str
    documents: Annotated[list[str], add]
    answer: str

def retrieve(state: RAGState) -> dict:
    writer = get_stream_writer()
    writer({"stage": "retrieve", "status": "检索中..."})
    docs = ["向量数据库是一种专门存储向量的数据库"]
    writer({"stage": "retrieve", "found": len(docs)})
    return {"documents": docs}

def generate(state: RAGState) -> dict:
    context = "\n".join(state["documents"])
    return {"answer": f"基于 {len(state['documents'])} 篇文档: {context[:50]}"}

checkpointer = InMemorySaver()
graph = (
    StateGraph(RAGState)
    .add_node("retrieve", retrieve)
    .add_node("generate", generate)
    .add_edge(START, "retrieve")
    .add_edge("retrieve", "generate")
    .add_edge("generate", END)
    .compile(checkpointer=checkpointer)
)

config = {"configurable": {"thread_id": "debug-interview"}}

# === Level 1: 流式调试 ===
print("--- Level 1: stream_mode='updates' ---")
for chunk in graph.stream(
    {"query": "什么是向量数据库？", "documents": [], "answer": ""},
    config, stream_mode="updates"
):
    print(chunk)
# {'retrieve': {'documents': ['向量数据库是一种专门存储向量的数据库']}}
# {'generate': {'answer': '基于 1 篇文档: 向量数据库是一种专门存储向量的数据库'}}

# === Level 2: 状态快照检查 ===
print("\n--- Level 2: get_state() ---")
snapshot = graph.get_state(config)
print(f"values: {snapshot.values}")
print(f"next: {snapshot.next}")          # () 表示执行完毕
print(f"step: {snapshot.metadata['step']}")

# === Level 3: 历史追溯 ===
print("\n--- Level 3: get_state_history() ---")
for s in graph.get_state_history(config):
    step = s.metadata.get("step", "?")
    source = s.metadata.get("source", "?")
    docs_count = len(s.values.get("documents", []))
    print(f"Step {step} ({source}): {docs_count} docs, answer={bool(s.values.get('answer'))}")
```

### 为什么这个回答出彩？

1. ✅ **展示了系统性的调试思维**（四个层次，从轻到重）—— 面试官看到的是一个有方法论的工程师
2. ✅ **提到了源码级的理解**（debug = checkpoints + tasks，`_emit()` 方法）—— 证明不是只会用 API
3. ✅ **区分了开发和生产环境的策略** —— 展示工程成熟度
4. ✅ **提到了具体的 API 和数据结构**（`StateSnapshot`、`get_stream_writer()`）—— 不是空谈概念

### 追问预备

**Q: 你说 debug 模式是 checkpoints + tasks 的组合，能具体解释一下吗？**

> 在源码 `_loop.py` 的 `_emit()` 方法中，当 `stream_mode` 包含 `"debug"` 时，框架会拦截 `"checkpoints"` 和 `"tasks"` 模式的事件，将它们包装成统一格式后以 `"debug"` 模式输出。包装后的结构是：
>
> ```python
> {
>     "step": int,           # 执行步骤号
>     "timestamp": str,      # ISO 格式时间戳
>     "type": str,           # "task" | "task_result" | "checkpoint"
>     "payload": dict,       # 原始的 TaskPayload / TaskResultPayload / CheckpointPayload
> }
> ```
>
> 所以 debug 模式会产生三种事件：任务开始（`task`）、任务完成（`task_result`）、检查点创建（`checkpoint`）。一个两节点的图，完整的 debug 事件序列是：
>
> ```
> checkpoint(step=-1) → task(step=0) → task_result(step=0) → checkpoint(step=0)
>                     → task(step=1) → task_result(step=1) → checkpoint(step=1)
> ```

**Q: get_state() 必须配置 checkpointer 才能用吗？为什么？**

> 是的，`get_state()` 依赖 checkpointer。因为 LangGraph 的状态在图执行完毕后不会保留在内存中——它是通过 checkpointer 持久化到存储后端的。`get_state()` 实际上是从 checkpointer 中读取最新的 `CheckpointTuple`，然后通过 `_prepare_state_snapshot()` 构建出 `StateSnapshot` 返回给用户。没有 checkpointer，就没有持久化的状态可读，调用会抛出 `ValueError("No checkpointer set")`。
>
> 最简单的配置方式：
>
> ```python
> from langgraph.checkpoint.memory import InMemorySaver
> graph = builder.compile(checkpointer=InMemorySaver())
> ```

**Q: 如果生产环境不方便用 LangSmith，你会怎么做？**

> 两个替代方案：
>
> 1. **`stream_mode="custom"` + 结构化日志**：在关键节点中用 `get_stream_writer()` 输出业务指标，配合 Python 的 `logging` 模块将数据写入日志系统（ELK、CloudWatch 等）
>
> 2. **自定义 callback handler**：LangGraph 基于 LangChain 的 Runnable 协议，支持注入自定义回调。可以实现一个 callback handler 将执行事件发送到自己的监控平台
>
> ```python
> import logging
> from langgraph.config import get_stream_writer
>
> logger = logging.getLogger("rag_agent")
>
> def retrieve(state):
>     writer = get_stream_writer()
>     docs = do_retrieval(state["query"])
>     # 同时输出到流和日志
>     writer({"stage": "retrieve", "count": len(docs)})
>     logger.info(f"Retrieved {len(docs)} docs for query: {state['query']}")
>     return {"documents": docs}
> ```

**Q: stream_mode 可以同时使用多种模式吗？**

> 可以。传入列表即可，此时每个 chunk 变成 `(mode_name, data)` 元组：
>
> ```python
> for mode, chunk in graph.stream(input, config, stream_mode=["values", "custom"]):
>     if mode == "values":
>         print(f"[状态] {chunk}")
>     elif mode == "custom":
>         print(f"[自定义] {chunk}")
> ```
>
> 常用组合：
> - `["values", "custom"]`：状态全貌 + 自定义进度
> - `["updates", "tasks"]`：增量变化 + 任务生命周期
> - `["debug"]`：单独使用就够了，因为它已经包含了 checkpoints 和 tasks

## 问题2："LangGraph 的 stream_mode='debug' 和 stream_mode='values' 有什么区别？什么时候用哪个？"

### 普通回答（❌ 不出彩）

"debug 模式输出更多信息，values 模式输出状态值。"

**问题**：太笼统，没有说清"更多信息"具体是什么，也没有给出选择依据。面试官会追问"具体多了什么？"

### 出彩回答（✅ 推荐）

> **这两种模式的核心区别在于信息粒度和用途：**
>
> **`values` 模式**输出每步后的完整状态值，就像每一步拍一张全景照片。你看到的是状态的"全家福"——每个 chunk 是一个完整的状态字典。适合快速验证"状态对不对"。
>
> **`debug` 模式**实际上是 `checkpoints` + `tasks` 的组合（这在源码 `_loop.py` 的 `_emit()` 方法中可以看到），它不仅输出状态值，还输出：
> - **TaskPayload**：任务开始事件，包含 `id`、`name`、`input`、`triggers`
> - **TaskResultPayload**：任务结果事件，包含 `result`、`error`、`interrupts`
> - **CheckpointPayload**：检查点事件，包含 `config`、`metadata`、`values`、`next`、`tasks`
>
> 每个事件都被包装成统一格式：`{step, timestamp, type, payload}`。
>
> **选择策略：**
>
> | 场景 | 推荐模式 | 原因 |
> |------|----------|------|
> | 快速验证状态是否正确 | `values` | 直观，每步一个完整状态 |
> | 追踪执行流程和错误 | `debug` | 包含任务生命周期和错误信息 |
> | 只关心变化部分 | `updates` | 增量输出，状态大时更高效 |
> | 生产环境监控 | `custom` + LangSmith | 自定义指标 + 全链路追踪 |
>
> **一句话总结**：`values` 回答"每步状态是什么"，`debug` 回答"每步发生了什么"。

### 代码示例——直观对比

```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver

class State(TypedDict):
    messages: Annotated[list[str], add]
    count: int

def node_a(state: State) -> dict:
    return {"messages": ["A处理完毕"], "count": state["count"] + 1}

def node_b(state: State) -> dict:
    return {"messages": ["B处理完毕"], "count": state["count"] + 10}

graph = (
    StateGraph(State)
    .add_node("node_a", node_a)
    .add_node("node_b", node_b)
    .add_edge(START, "node_a")
    .add_edge("node_a", "node_b")
    .add_edge("node_b", END)
    .compile(checkpointer=InMemorySaver())
)

initial_input = {"messages": ["开始"], "count": 0}

# === values 模式：每步一个完整状态 ===
print("=== stream_mode='values' ===")
config_v = {"configurable": {"thread_id": "values-demo"}}
for chunk in graph.stream(initial_input, config_v, stream_mode="values"):
    print(chunk)
# {'messages': ['开始'], 'count': 0}
# {'messages': ['开始', 'A处理完毕'], 'count': 1}
# {'messages': ['开始', 'A处理完毕', 'B处理完毕'], 'count': 11}
# → 3 个 chunk，每个是完整状态

# === debug 模式：任务生命周期 + 检查点 ===
print("\n=== stream_mode='debug' ===")
config_d = {"configurable": {"thread_id": "debug-demo"}}
for event in graph.stream(initial_input, config_d, stream_mode="debug"):
    t = event["type"]
    step = event["step"]
    if t == "task":
        print(f"[step={step}] TASK_START: {event['payload']['name']}")
    elif t == "task_result":
        print(f"[step={step}] TASK_DONE:  {event['payload']['name']}")
        if event['payload'].get('error'):
            print(f"  ERROR: {event['payload']['error']}")
    elif t == "checkpoint":
        print(f"[step={step}] CHECKPOINT: next={event['payload']['next']}")
# [step=-1] CHECKPOINT: next=['node_a']
# [step=0]  TASK_START: node_a
# [step=0]  TASK_DONE:  node_a
# [step=0]  CHECKPOINT: next=['node_b']
# [step=1]  TASK_START: node_b
# [step=1]  TASK_DONE:  node_b
# [step=1]  CHECKPOINT: next=[]
# → 7 个事件，包含完整的执行时序
```

### 追问预备

**Q: debug 模式中 checkpoint 事件的 step 和 task 事件的 step 有什么关系？**

> 这是一个源码细节。在 `_loop.py` 的 `_emit()` 方法中：
> - task 事件的 step 是 `self.step`（当前步骤）
> - checkpoint 事件的 step 是 `self.step - 1`（上一步骤）
>
> 这是因为 checkpoint 记录的是"执行完某步后的状态"，所以它的 step 对应的是刚完成的那一步。而 task 事件记录的是"正在执行的步骤"。
>
> ```
> 时间线：
> checkpoint(step=-1)  ← 初始输入后的状态
> task(step=0)         ← node_a 开始执行
> task_result(step=0)  ← node_a 执行完毕
> checkpoint(step=0)   ← node_a 执行后的状态（step = 1-1 = 0）
> task(step=1)         ← node_b 开始执行
> task_result(step=1)  ← node_b 执行完毕
> checkpoint(step=1)   ← node_b 执行后的状态（step = 2-1 = 1）
> ```

**Q: values 模式输出的第一个 chunk 是什么？是初始输入还是第一个节点执行后的状态？**

> 第一个 chunk 是初始输入后的状态（对应 `metadata.step = -1` 到 `step = 0` 的转换）。也就是说，你传入 `invoke()` 的输入经过 channel 处理后的状态。之后每个节点执行完毕都会输出一个 chunk。
>
> 对于一个两节点的图，values 模式会输出 3 个 chunk：
> 1. 初始输入后的状态
> 2. node_a 执行后的完整状态
> 3. node_b 执行后的完整状态（最终状态）

**Q: 如果状态很大（比如包含长对话历史），values 模式会有性能问题吗？**

> 会。values 模式每步输出完整状态，如果状态中有一个 `messages: list[str]` 字段包含几百条对话记录，每个 chunk 都会重复输出这些数据。这时候更好的选择是：
>
> - `updates` 模式：只输出节点返回的增量更新，数据量小得多
> - `["updates", "custom"]` 组合：增量更新 + 自定义的关键指标
>
> ```python
> # 状态很大时，避免用 values
> # ❌ 每步输出完整的 messages 列表
> for chunk in graph.stream(input, config, stream_mode="values"):
>     print(len(str(chunk)))  # 可能很大
>
> # ✅ 只输出增量变化
> for chunk in graph.stream(input, config, stream_mode="updates"):
>     print(chunk)  # 只有当前节点的返回值
> ```

**Q: 除了 values 和 debug，还有哪些 stream_mode？分别什么时候用？**

> LangGraph 定义了 7 种 `StreamMode`（源码 `types.py`）：
>
> | 模式 | 用途 | 典型场景 |
> |------|------|----------|
> | `values` | 每步完整状态 | 快速验证 |
> | `updates` | 增量更新 | 定位"谁改了什么" |
> | `tasks` | 任务生命周期 | 监控任务执行时序 |
> | `checkpoints` | 检查点快照 | 调试持久化问题 |
> | `debug` | tasks + checkpoints | 深度排查 |
> | `custom` | 自定义数据 | 业务指标输出 |
> | `messages` | LLM token 流 | 流式输出给用户 |
>
> 其中 `messages` 专注于 LLM 的 token 级流式输出，不属于调试范畴。

## 面试加分技巧

### 回答框架：STAR-T

| 步骤 | 含义 | 示例 |
|------|------|------|
| **S** | Situation（场景） | "在一个 RAG Agent 项目中..." |
| **T** | Task（任务） | "用户反馈答案质量不稳定，需要定位问题..." |
| **A** | Action（行动） | "我用 stream_mode='debug' 追踪执行流程，发现检索节点返回了不相关文档..." |
| **R** | Result（结果） | "通过 get_state_history() 回溯到检索步骤，用 update_state() 注入正确文档后验证了生成质量..." |
| **T** | Takeaway（总结） | "最终在 Rerank 节点调整了阈值，并用 LangSmith 持续监控..." |

### 关键词清单

面试中提到这些关键词会加分：

- `stream_mode="debug"`（最详细的调试模式）
- `TaskPayload` / `TaskResultPayload` / `CheckpointPayload`（debug 模式的三种事件）
- `_emit()` 方法（debug 模式的源码实现）
- `get_state()` → `StateSnapshot`（8 字段 NamedTuple）
- `get_state_history()` → `Iterator[StateSnapshot]`（时间倒序）
- Replay vs Fork（时间旅行两种模式）
- `get_stream_writer()`（自定义流输出）
- `LANGSMITH_TRACING=true`（零侵入追踪）

### 常见陷阱

面试中容易踩的坑：

1. **只提 print 调试** —— LangGraph 提供了完整的调试工具链，只说 print 显得不专业
2. **混淆 values 和 updates** —— values 是完整状态，updates 是增量更新，区别很大
3. **不知道 debug 模式的内部实现** —— 说"debug 就是输出更多信息"太模糊，要能说出它是 checkpoints + tasks 的组合
4. **忘记 checkpointer 前提** —— `get_state()`、`get_state_history()` 都依赖 checkpointer，没有它什么都做不了
5. **不区分开发和生产环境** —— 开发用 stream_mode + get_state_history，生产用 LangSmith + custom，展示工程思维
6. **忽略实际应用场景** —— 面试官想听的是"你用这些工具解决过什么问题"，不是"它们是什么"

│  │  → values / next / tasks / interrupts                │    │
│  ├─────────────────────────────────────────────────────┤    │
│  │  Level 1: 流式观察                                    │    │
│  │  stream_mode = values | updates | tasks |            │    │
│  │               checkpoints | debug | custom           │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  开发环境: Level 1 (debug) + Level 3 (history)              │
│  生产环境: Level 1 (custom) + Level 4 (LangSmith)           │
└─────────────────────────────────────────────────────────────┘
```
