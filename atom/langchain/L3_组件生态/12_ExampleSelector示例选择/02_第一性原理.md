# ExampleSelector示例选择 - 第一性原理

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而不是通过类比或经验来推理。

在物理学中，第一性原理意味着从基本的物理定律出发推导现象。在软件工程中，第一性原理意味着从最根本的问题和需求出发，推导出解决方案。

---

## ExampleSelector 的第一性原理

### 1. 最基础的定义

**ExampleSelector = 一个根据输入动态选择最相关示例的机制**

仅此而已！没有更基础的了。

它不是：
- ❌ 一个存储示例的数据库
- ❌ 一个生成示例的工具
- ❌ 一个评估示例质量的系统

它只是：
- ✅ **输入**：用户的查询或输入
- ✅ **处理**：根据某种策略选择示例
- ✅ **输出**：最相关的 N 个示例

### 2. 为什么需要 ExampleSelector？

#### 核心问题：Few-shot Learning 的困境

让我们从最根本的问题开始思考。

**问题 1：LLM 如何理解任务？**

大语言模型（LLM）虽然强大，但它们需要理解你想要什么。有两种方式：

1. **Zero-shot**：只给指令，不给示例
   ```python
   prompt = "将以下产品描述转换为 JSON 格式：iPhone 14 售价 $799"
   ```
   - 优点：简单、快速
   - 缺点：模型可能理解错误，输出格式不一致

2. **Few-shot**：给指令 + 示例
   ```python
   prompt = """
   示例 1：
   输入：MacBook Pro 售价 $2499
   输出：{"name": "MacBook Pro", "price": 2499}

   示例 2：
   输入：AirPods 售价 $199
   输出：{"name": "AirPods", "price": 199}

   现在处理：iPhone 14 售价 $799
   """
   ```
   - 优点：模型理解更准确，输出格式一致
   - 缺点：需要提供示例

[来源: reference/fetch_example_selector_04.md | Sandgarden 教程]

**问题 2：如何选择示例？**

假设你有 100 个示例，但 LLM 的 context window 只能容纳 5 个。你会怎么选？

- **方案 A：随机选择** → 可能选到不相关的示例
- **方案 B：固定选择前 5 个** → 可能不适合当前查询
- **方案 C：根据查询动态选择最相关的 5 个** → ✅ 这就是 ExampleSelector！

**问题 3：什么是"相关"？**

这是一个哲学问题，也是技术问题。

- **语义相关**：查询"如何做披萨"，示例"如何做意大利面"比"如何修电脑"更相关
- **结构相关**：查询是长文本，示例也应该是长文本
- **多样性相关**：示例之间应该有差异，避免重复信息

### 3. ExampleSelector 的三层价值

#### 价值 1：提高 LLM 输出质量

**核心观点**：相关的示例 = 更好的输出

**为什么？**

LLM 是一个模式识别机器。当你给它看相关的示例时，它能更好地理解你想要的模式。

**类比：**
- **前端开发**：就像给 React 组件传入正确的 props，组件才能正确渲染
- **日常生活**：就像给厨师看菜谱，厨师才能做出你想要的菜

**实际案例：**

MUFG Bank 使用动态示例选择来处理 100-200 页的年度报告，将演示文稿创建时间从几小时缩短到 3-5 分钟。

[来源: reference/fetch_example_selector_05.md | Swarnendu 博客 - LangChain Best Practices 2025]

**代码示例：**

```python
from langchain_core.example_selectors import SemanticSimilarityExampleSelector
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings

# 示例池
examples = [
    {"input": "2+2", "output": "4"},
    {"input": "2+3", "output": "5"},
    {"input": "What is the capital of France?", "output": "Paris"},
    {"input": "What is the capital of Germany?", "output": "Berlin"},
]

# 创建语义相似度选择器
selector = SemanticSimilarityExampleSelector.from_examples(
    examples,
    OpenAIEmbeddings(),
    Chroma,
    k=2  # 只选择最相关的 2 个示例
)

# 查询：数学问题
math_query = {"input": "What is 5+5?"}
selected = selector.select_examples(math_query)
print("数学查询选择的示例：", selected)
# 输出：[{"input": "2+2", "output": "4"}, {"input": "2+3", "output": "5"}]

# 查询：地理问题
geo_query = {"input": "What is the capital of Italy?"}
selected = selector.select_examples(geo_query)
print("地理查询选择的示例：", selected)
# 输出：[{"input": "What is the capital of France?", "output": "Paris"}, ...]
```

#### 价值 2：优化 Token 使用和成本

**核心观点**：只选择必要的示例 = 降低成本

**为什么？**

LLM API 按 token 计费。如果你有 100 个示例，每个 50 tokens，全部放入 prompt 就是 5000 tokens。但如果只选择 5 个最相关的，就只需要 250 tokens。

**成本对比：**

| 策略 | 示例数量 | Token 数 | 成本（GPT-4） |
|------|---------|---------|--------------|
| 全部示例 | 100 | 5000 | $0.15 |
| 动态选择 | 5 | 250 | $0.0075 |
| **节省** | **95** | **4750** | **$0.1425 (95%)** |

**实际案例：**

LinkedIn 的 SQL Bot 使用 embedding-based retrieval 来检索语义相关的上下文，显著降低了 prompt 长度和成本。

[来源: reference/fetch_example_selector_05.md | Swarnendu 博客]

**代码示例：**

```python
from langchain_core.example_selectors import LengthBasedExampleSelector
from langchain_core.prompts import PromptTemplate, FewShotPromptTemplate

# 示例池（假设有很多示例）
examples = [
    {"query": "How are you?", "answer": "I'm doing well, thanks!"},
    {"query": "What time is it?", "answer": "It's time to get a watch."},
    # ... 更多示例
]

# 创建长度限制选择器
example_prompt = PromptTemplate(
    input_variables=["query", "answer"],
    template="Q: {query}\nA: {answer}"
)

selector = LengthBasedExampleSelector(
    examples=examples,
    example_prompt=example_prompt,
    max_length=100  # 限制总长度为 100 个单词
)

# 创建 Few-shot Prompt
prompt = FewShotPromptTemplate(
    example_selector=selector,
    example_prompt=example_prompt,
    prefix="You are a helpful assistant.",
    suffix="Q: {query}\nA:",
    input_variables=["query"]
)

# 短查询 → 可以包含更多示例
short_query = "Hi"
print(prompt.format(query=short_query))

# 长查询 → 自动减少示例数量
long_query = "Can you explain the theory of relativity in detail?"
print(prompt.format(query=long_query))
```

#### 价值 3：适应不同场景和任务

**核心观点**：不同的任务需要不同的示例选择策略

**为什么？**

没有一种选择策略适用于所有场景。你需要根据任务特点选择合适的策略。

**场景对比：**

| 场景 | 最佳策略 | 原因 |
|------|---------|------|
| 问答系统 | 语义相似度 | 需要找到语义相关的 QA 对 |
| Token 限制 | 长度限制 | 必须控制 prompt 长度 |
| 多样性需求 | MMR（最大边际相关性） | 避免示例过于相似 |
| 文本相似度 | N-gram 重叠 | 轻量级，不需要 embeddings |

[来源: reference/source_example_selector_01.md | LangChain 源码分析]

**实际案例：**

C.H. Robinson 使用 LangChain 的 RAG blueprint 来发现和总结大量投资数据，实现了复杂的分类任务（less-than-truckload vs full truckload shipments）。

[来源: reference/fetch_example_selector_05.md | Swarnendu 博客]

**代码示例：**

```python
from langchain_core.example_selectors import (
    SemanticSimilarityExampleSelector,
    MaxMarginalRelevanceExampleSelector,
    LengthBasedExampleSelector
)

# 场景 1：问答系统 - 使用语义相似度
qa_selector = SemanticSimilarityExampleSelector.from_examples(
    qa_examples,
    OpenAIEmbeddings(),
    Chroma,
    k=3
)

# 场景 2：多样性需求 - 使用 MMR
diverse_selector = MaxMarginalRelevanceExampleSelector.from_examples(

    k=5,
    fetch_k=20  # 先获取 20 个候选，再选择 5 个最多样的
)

# 场景 3：Token 限制 - 使用长度限制
length_selector = LengthBasedExampleSelector(
    examples=examples,
    example_prompt=example_prompt,
    max_length=500
)
```

### 4. 从第一性原理推导 ExampleSelector 的设计

让我们从最基本的需求出发，推导出 ExampleSelector 的设计。

**推理链：**

```
1. 前提：LLM 需要示例来理解任务
   ↓
2. 问题：示例太多会超过 context window
   ↓
3. 需求：需要选择最相关的示例
   ↓
4. 挑战："相关"的定义因任务而异
   ↓
5. 解决方案：需要一个抽象接口 + 多种具体策略
   ↓
6. 设计：BaseExampleSelector（抽象接口）
   ↓
7. 实现：SemanticSimilarity、Length、MMR、NGram（具体策略）
   ↓
8. 优化：支持异步、缓存、批量操作
   ↓
9. 集成：与 FewShotPromptTemplate 无缝集成
   ↓
10. 最终：一个灵活、可扩展、高性能的示例选择系统
```

#### 推导步骤详解

**步骤 1-2：问题识别**

```python
# 问题：如果有 100 个示例，全部放入 prompt 会怎样？
examples = [example_1, example_2, ..., example_100]
prompt = f"示例：{examples}\n\n查询：{query}"

# 结果：
# - Token 数量：~10,000 tokens
# - 成本：$0.30（GPT-4）
# - 延迟：~5 秒
# - 风险：可能超过 context window（如 GPT-3.5 的 4096 tokens）
```

**步骤 3-4：需求分析**

```python
# 需求：只选择最相关的 5 个示例
# 但"相关"的定义是什么？

# 定义 1：语义相关（基于 embeddings）
def semantic_relevance(query, example):
    query_embedding = get_embedding(query)
    example_embedding = get_embedding(example)
    return cosine_similarity(query_embedding, example_embedding)

# 定义 2：长度相关（基于 token 数）
def length_relevance(query, example, max_length):
    total_length = len(query) + len(example)
    return total_length <= max_length

# 定义 3：多样性相关（基于 MMR）
def diversity_relevance(query, examples, selected):
    # 平衡相关性和多样性
    return mmr_score(query, examples, selected)
```

[来源: reference/context7_langchain_01.md | LangChain 官方文档]

**步骤 5-6：抽象设计**

```python
from abc import ABC, abstractmethod

class BaseExampleSelector(ABC):
    """抽象基类：定义统一接口"""

    @abstractmethod
    def add_example(self, example: dict) -> None:
        """添加示例到存储"""
        pass

    @abstractmethod
    def select_examples(self, input_variables: dict) -> list[dict]:
        """根据输入选择示例"""
        pass
```

[来源: sourcecode/langchain/libs/core/langchain_core/example_selectors/base.py]

**步骤 7：具体实现**

```python
# 实现 1：语义相似度选择器
class SemanticSimilarityExampleSelector(BaseExampleSelector):
    def __init__(self, vectorstore, k=4):
        self.vectorstore = vectorstore
        self.k = k

    def select_examples(self, input_variables):
        query = input_variables["input"]
        # 使用向量相似度搜索
        docs = self.vectorstore.similarity_search(query, k=self.k)
        return [doc.metadata for doc in docs]

# 实现 2：长度限制选择器
class LengthBasedExampleSelector(BaseExampleSelector):
    def __init__(self, examples, max_length):
        self.examples = examples
        self.max_length = max_length

    def select_examples(self, input_variables):
        selected = []
        current_length = 0
        for example in self.examples:
            if current_length + len(str(example)) <= self.max_length:
                selected.append(example)
                current_length += len(str(example))
        return selected
```

### 5. 一句话总结第一性原理

**ExampleSelector 是一个根据输入动态选择最相关示例的抽象机制，通过统一接口支持多种选择策略（语义、长度、多样性等），从而优化 Few-shot Learning 的效果、成本和适应性。**

## 从第一性原理看 ExampleSelector 的核心设计决策

### 决策 1：为什么需要抽象接口？

**问题**：为什么不直接实现一个"最好的"选择器？

**答案**：因为没有"最好的"选择器，只有"最适合的"选择器。

- 问答系统需要语义相似度
- Token 限制场景需要长度控制
- 多样性需求需要 MMR

**设计决策**：定义 `BaseExampleSelector` 抽象接口，允许多种实现。

[来源: reference/source_example_selector_01.md | LangChain 源码分析]

---

### 决策 2：为什么选择 VectorStore 作为后端？

**问题**：为什么 `SemanticSimilarityExampleSelector` 依赖 `VectorStore`？

**答案**：因为语义相似度搜索本质上是向量相似度搜索。

- **Embeddings**：将文本转换为向量
- **VectorStore**：高效存储和检索向量
- **Similarity Search**：找到最相似的向量

**设计决策**：依赖 `VectorStore` 抽象，支持多种向量数据库（Chroma、FAISS、Pinecone 等）。

[来源: reference/source_example_selector_01.md | LangChain 源码分析]

---

### 决策 3：为什么支持异步？

**问题**：为什么需要 `aselect_examples()` 异步方法？

**答案**：因为向量搜索可能很慢，特别是在大规模数据集上。

- **同步**：阻塞主线程，影响性能
- **异步**：非阻塞，可以并发处理多个请求

**设计决策**：所有选择器都支持异步操作。

```python
# 同步
examples = selector.select_examples(input_variables)

# 异步
examples = await selector.aselect_examples(input_variables)
```

[来源: reference/source_example_selector_01.md | LangChain 源码分析]

---

### 决策 4：为什么需要 `from_examples()` 类方法？

**问题**：为什么不直接用构造函数？

**答案**：因为创建选择器需要多个步骤（创建 embeddings、创建 vectorstore、初始化选择器）。

**设计决策**：提供 `from_examples()` 类方法简化创建流程。

```python
# 复杂方式
embeddings = OpenAIEmbeddings()
texts = [example["input"] for example in examples]
vectorstore = Chroma.from_texts(texts, embeddings, metadatas=examples)
selector = SemanticSimilarityExampleSelector(vectorstore=vectorstore, k=3)

# 简化方式
selector = SemanticSimilarityExampleSelector.from_examples(
    examples,
    OpenAIEmbeddings(),
    Chroma,
    k=3
)
```

## 与 AI Agent 开发的关系

### 1. Few-shot Learning 是 Agent 的核心能力

**为什么？**

AI Agent 需要在不同场景下执行不同任务。Few-shot learning 让 Agent 能够快速适应新任务，而不需要重新训练。

**实际案例：**

- **MUFG Bank**：使用动态示例选择来处理金融文档
- **LinkedIn SQL Bot**：使用 embedding-based retrieval 来检索相关上下文
- **C.H. Robinson**：使用 RAG blueprint 来分类运输数据

[来源: reference/fetch_example_selector_05.md | Swarnendu 博客]

---

### 2. ExampleSelector 优化 Agent 的 Prompt

**为什么？**

Agent 的 Prompt 通常很长（包含工具描述、历史对话、示例等）。ExampleSelector 帮助 Agent 只选择最相关的示例，从而：

- 减少 Token 使用
- 提高响应速度
- 降低成本

**代码示例：**

```python
from langchain.agents import AgentExecutor, create_openai_functions_agent
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# 创建 Agent Prompt（包含动态示例）
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    MessagesPlaceholder(variable_name="few_shot_examples"),  # 动态示例
    MessagesPlaceholder(variable_name="chat_history"),
    ("human", "{input}"),
    MessagesPlaceholder(variable_name="agent_scratchpad"),
])

# 在 Agent 执行前，动态选择示例
def select_examples_for_agent(input_text):
    return selector.select_examples({"input": input_text})

# 集成到 Agent
agent = create_openai_functions_agent(llm, tools, prompt)
agent_executor = AgentExecutor(agent=agent, tools=tools)
```
