# 核心概念 3：状态大小控制策略

## 概述

状态大小控制是 LangGraph 生产环境中的**头号关注点**。每次 super-step 执行后，LangGraph 都会自动将完整状态序列化为 checkpoint 存入数据库。如果不加控制，消息列表和工具输出会快速累积，导致 checkpoint 膨胀、序列化变慢、数据库性能下降。本节介绍 5 种核心策略来管理状态大小。

[来源: reference/search_状态持久化准备_01.md | 社区最佳实践与生产经验]

---

## 为什么状态大小如此重要？

### 问题的根源

LangGraph 的持久化管道是这样工作的：

```
每个 super-step 执行后：
Channel.checkpoint() → SerializerProtocol.dumps_typed() → CheckpointSaver.put()
```

这意味着**每一步**都会把完整状态序列化并存储。如果你的状态包含 100 条消息，每条消息 1KB，那每个 checkpoint 至少 100KB。一个 10 步的对话就产生 1MB 的 checkpoint 数据。

[来源: reference/source_状态持久化准备_01.md | 源码分析：完整持久化管道]

### 三大影响

| 影响 | 说明 | 严重程度 |
|------|------|----------|
| **存储膨胀** | checkpoint 数据库快速增长，存储成本上升 | 中 |
| **序列化变慢** | 大状态的 msgpack 编码/解码耗时增加 | 高 |
| **数据库性能** | 大 checkpoint 的读写拖慢整体响应 | 高 |

### 生产环境的真实场景

```python
# ❌ 典型的状态膨胀场景：工具密集型 Agent
# 每次工具调用产生 2 条消息（tool_call + tool_result）
# 10 轮对话 × 3 次工具调用/轮 = 80+ 条消息
# 加上工具返回的大量 JSON 数据...checkpoint 轻松超过 1MB

from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import add_messages
from langchain_core.messages import HumanMessage, AIMessage, ToolMessage

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]  # 这个列表会无限增长！
    documents: list[dict]                     # 检索到的文档也在累积
    tool_outputs: list[str]                   # 工具输出越来越多
```

## 策略 1：消息修剪（Message Trimming）

### 核心思路

只保留最近 N 条消息，丢弃旧消息。这是最简单也最常用的策略。

### 实现方式

```python
"""
策略 1：消息修剪
保留最近 N 条消息，防止消息列表无限增长
"""
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END, add_messages
from langchain_core.messages import (
    HumanMessage, AIMessage, SystemMessage,
    trim_messages
)

# ===== 方式 A：在节点中手动修剪 =====

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

def chatbot_node(state: AgentState) -> dict:
    """聊天节点：处理前先修剪消息"""
    messages = state["messages"]

    # 保留系统消息 + 最近 10 条对话消息
    system_msgs = [m for m in messages if isinstance(m, SystemMessage)]
    non_system = [m for m in messages if not isinstance(m, SystemMessage)]

    # 只保留最近 10 条
    trimmed = system_msgs + non_system[-10:]

    # 用修剪后的消息调用 LLM
    # response = llm.invoke(trimmed)
    print(f"原始消息数: {len(messages)}, 修剪后: {len(trimmed)}")

    return {"messages": [AIMessage(content="回复内容")]}


# ===== 方式 B：使用 LangChain 的 trim_messages 工具 =====

def chatbot_with_trim(state: AgentState) -> dict:
    """使用 trim_messages 按 token 数修剪"""
    trimmed = trim_messages(
        state["messages"],
        max_tokens=4000,          # 最多保留 4000 tokens
        strategy="last",          # 保留最后的消息
        token_counter=len,        # 简单用长度计数（生产中用 tiktoken）
        include_system=True,      # 始终保留系统消息
        allow_partial=False,      # 不截断单条消息
    )

    print(f"修剪后消息数: {len(trimmed)}")
    return {"messages": [AIMessage(content="回复内容")]}
```

### 注意事项

- 系统消息（SystemMessage）通常需要始终保留
- 修剪只影响发送给 LLM 的消息，不影响已存储的 checkpoint
- 如果要减少 checkpoint 大小，需要在状态更新时就修剪

## 策略 2：外部存储卸载（External Storage Offloading）

### 核心思路

把大对象（文档、图片、长文本）移到外部存储（S3、LangGraph Store），状态中只保留引用（ID 或 URL）。

### 为什么这样做？

```
❌ 直接存储大对象：
state = {
    "documents": [{"content": "10KB的文档内容...", ...}, ...],  # 50KB
    "images": [b"大量二进制数据...", ...],                       # 500KB
}
# 每个 checkpoint 都要序列化这 550KB！

✅ 只存储引用：
state = {
    "document_ids": ["doc_001", "doc_002", "doc_003"],  # 几十字节
    "image_urls": ["s3://bucket/img1.png"],              # 几十字节
}
# 每个 checkpoint 只有几百字节
```

### 实现方式

```python
"""
策略 2：外部存储卸载
大对象存到外部，状态中只保留引用
"""
from typing_extensions import TypedDict
from typing import Annotated
from langgraph.graph import add_messages

# ===== 模拟外部存储 =====

# 生产中这里是 S3、Redis、LangGraph Store 等
_external_store: dict[str, str] = {}

def save_to_store(key: str, content: str) -> str:
    """保存大对象到外部存储，返回引用 ID"""
    _external_store[key] = content
    return key

def load_from_store(key: str) -> str:
    """从外部存储加载大对象"""
    return _external_store.get(key, "")


# ===== 状态设计：只存引用 =====

class SmartState(TypedDict):
    messages: Annotated[list, add_messages]
    document_ids: list[str]      # ✅ 只存 ID，不存内容
    current_query: str


def retrieval_node(state: SmartState) -> dict:
    """检索节点：文档存到外部，状态只保留 ID"""
    query = state["current_query"]

    # 模拟检索到 3 个大文档
    doc_ids = []
    for i in range(3):
        content = f"这是关于 '{query}' 的详细文档内容..." * 100  # 模拟大文档
        doc_id = f"doc_{query}_{i}"
        save_to_store(doc_id, content)  # 存到外部
        doc_ids.append(doc_id)

    print(f"检索到 {len(doc_ids)} 个文档，已存入外部存储")
    print(f"状态中只保留 ID: {doc_ids}")

    # ✅ 状态中只存 ID 列表
    return {"document_ids": doc_ids}


def generation_node(state: SmartState) -> dict:
    """生成节点：需要时从外部加载文档"""
    doc_ids = state["document_ids"]

    # 需要时才从外部存储加载
    documents = [load_from_store(doc_id) for doc_id in doc_ids]
    context = "\n".join(documents)

    print(f"从外部存储加载了 {len(documents)} 个文档")
    print(f"上下文总长度: {len(context)} 字符")

    # 用加载的文档生成回答
    # response = llm.invoke(f"基于以下文档回答：{context}\n问题：{state['current_query']}")

    return {"messages": [{"role": "assistant", "content": "基于文档的回答..."}]}
```

### 适用场景

| 场景 | 外部存储选择 | 说明 |
|------|-------------|------|
| 检索到的文档 | S3 / LangGraph Store | 文档内容大，但不需要每步都访问 |
| 图片/音频 | S3 / 对象存储 | 二进制数据不适合放在 checkpoint |
| 中间计算结果 | Redis | 临时数据，需要快速访问 |
| 用户上传文件 | S3 | 文件本身不应该在状态中 |

## 策略 4：压缩（Checkpoint Compression）

### 核心思路

在序列化后、存储前对 checkpoint 数据进行压缩，减少存储空间和网络传输量。

### 实现方式

LangGraph 的默认 `JsonPlusSerializer` 使用 msgpack 编码，本身就比 JSON 更紧凑。但对于特别大的 checkpoint，可以启用额外压缩。

```python

    "messages": [

        {"role": "assistant", "content": "RAG 是检索增强生成..."},
    ] * 10,  # 20 条消息
```

### 自定义压缩包装器

```python
"""
自定义压缩序列化器（概念演示）
"""
import gzip
from typing import Any

class CompressedSerializer:
    """
    包装任何序列化器，添加 gzip 压缩层

    设计模式与 EncryptedSerializer 相同：装饰器模式
    """

    def __init__(self, inner_serializer):
        self.inner = inner_serializer

    def dumps_typed(self, obj: Any) -> tuple[str, bytes]:
        # 先用内部序列化器编码
        type_tag, data = self.inner.dumps_typed(obj)
        # 再压缩
        compressed = gzip.compress(data)
        # 修改类型标签标记已压缩
        return (f"{type_tag}+gzip", compressed)

    def loads_typed(self, data: tuple[str, bytes]) -> Any:
        type_tag, raw = data
        if "+gzip" in type_tag:
            # 先解压
            raw = gzip.decompress(raw)
            type_tag = type_tag.replace("+gzip", "")
        # 再用内部序列化器解码
        return self.inner.loads_typed((type_tag, raw))


# 使用示例（概念演示）
# from langgraph.checkpoint.serde.jsonplus import JsonPlusSerializer
# compressed_serde = CompressedSerializer(JsonPlusSerializer())
# checkpointer = MemorySaver(serde=compressed_serde)

print("自定义压缩器遵循装饰器模式：")
print("  原始数据 → 序列化 → 压缩 → 存储")
print("  存储 → 解压 → 反序列化 → 原始数据")
```

## 策略 5：临时通道（Ephemeral Channels）

### 核心思路

有些数据只在当前 super-step 中有用，不需要持久化到 checkpoint。使用临时字段可以避免这些数据被序列化存储。

### 实现方式

```python
"""
策略 5：临时通道
标记不需要持久化的临时数据
"""
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END, add_messages
from langchain_core.messages import HumanMessage, AIMessage

# ===== 设计思路：区分持久化数据和临时数据 =====

class AgentState(TypedDict):
    # 需要持久化的核心数据
    messages: Annotated[list, add_messages]
    final_answer: str

    # 临时数据 - 理想情况下不应该持久化
    # 但 TypedDict 本身不支持标记"临时"
    # 解决方案：在节点中主动清理


def retrieval_node(state: AgentState) -> dict:
    """检索节点：产生临时的中间数据"""
    query = state["messages"][-1].content if state["messages"] else ""

    # 模拟检索 - 这些数据只在当前步骤需要
    raw_results = [f"文档片段 {i}: 关于 {query} 的内容..." for i in range(5)]
    scores = [0.95, 0.87, 0.82, 0.76, 0.71]

    # 处理后只保留需要的结果
    top_results = [r for r, s in zip(raw_results, scores) if s > 0.8]

    print(f"检索到 {len(raw_results)} 个结果，保留 {len(top_results)} 个")

    # ✅ 只返回精简后的数据，不把原始结果放入状态
    return {
        "messages": [AIMessage(content=f"找到 {len(top_results)} 个相关文档")],
    }


def generation_node(state: AgentState) -> dict:
    """生成节点"""
    return {
        "final_answer": "基于检索结果的回答...",
        "messages": [AIMessage(content="基于检索结果的回答...")],
    }


# 构建图
graph = StateGraph(AgentState)
graph.add_node("retrieve", retrieval_node)
graph.add_node("generate", generation_node)
graph.add_edge(START, "retrieve")
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", END)

app = graph.compile()

# 执行
result = app.invoke({
    "messages": [HumanMessage(content="什么是向量数据库？")],
    "final_answer": "",
})

print(f"\n最终状态中的消息数: {len(result['messages'])}")
print(f"最终答案: {result['final_answer'][:50]}...")
```

## 监控 Checkpoint 大小

### 实践方法

```python
"""
监控 checkpoint 大小的实用工具
"""
import sys
import json
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, START, END

# ===== 方法 1：估算状态大小 =====

def estimate_state_size(state: dict) -> dict:
    """估算状态各字段的大小"""
    sizes = {}
    total = 0

    for key, value in state.items():
        # 用 JSON 序列化估算大小（实际 msgpack 会更小）
        try:
            size = len(json.dumps(value, default=str).encode())
        except (TypeError, ValueError):
            size = sys.getsizeof(value)

        sizes[key] = size
        total += size

    return {
        "fields": sizes,
        "total_bytes": total,
        "total_kb": round(total / 1024, 2),
    }


# 测试
sample_state = {
    "messages": [{"role": "user", "content": "你好"}] * 50,
    "documents": ["长文档内容..." * 100] * 5,
    "metadata": {"thread_id": "test", "step": 10},
}

report = estimate_state_size(sample_state)
print("状态大小报告：")
for field, size in report["fields"].items():
    print(f"  {field}: {size:,} bytes ({size/1024:.1f} KB)")
print(f"  总计: {report['total_bytes']:,} bytes ({report['total_kb']} KB)")


# ===== 方法 2：设置大小告警 =====

MAX_STATE_SIZE_KB = 100  # 100KB 告警阈值

def check_state_size(state: dict, threshold_kb: float = MAX_STATE_SIZE_KB):
    """检查状态大小是否超过阈值"""
    report = estimate_state_size(state)
    if report["total_kb"] > threshold_kb:
        print(f"⚠️ 状态大小告警: {report['total_kb']} KB > {threshold_kb} KB")
        print("最大的字段：")
        sorted_fields = sorted(
            report["fields"].items(),
            key=lambda x: x[1],
            reverse=True
        )
        for field, size in sorted_fields[:3]:
            print(f"  {field}: {size/1024:.1f} KB")
        return False
    return True

check_state_size(sample_state)
```

## 策略对比总结

| 策略 | 适用场景 | 实现难度 | 效果 |
|------|----------|----------|------|
| **消息修剪** | 长对话、工具密集型 Agent | 低 | 控制消息数量 |
| **外部存储卸载** | 大文档、图片、二进制数据 | 中 | 大幅减少 checkpoint 大小 |
| **Checkpoint 修剪** | 长期运行的线程 | 低 | 减少数据库存储 |
| **压缩** | 所有场景（锦上添花） | 中 | 减少 20-40% 存储 |
| **临时通道** | 中间计算结果 | 低 | 避免不必要的持久化 |

### 推荐组合

```
开发阶段：消息修剪 + 临时通道（够用了）
生产阶段：全部 5 种策略组合使用
高流量：外部存储卸载 + 压缩 + TTL 自动过期
```

## 与 RAG 系统的关系

在 RAG 系统中，状态大小控制尤为重要：

1. **检索结果累积**：每次检索返回多个文档片段，如果全部存入状态，checkpoint 会快速膨胀
2. **工具调用频繁**：RAG Agent 经常调用搜索、数据库查询等工具，每次调用产生额外消息
3. **上下文窗口管理**：消息修剪不仅减少存储，还能确保发送给 LLM 的上下文不超过窗口限制
4. **多轮对话**：知识库问答通常是多轮的，消息累积速度快

**最佳实践**：检索到的文档存入外部存储（策略 2），消息列表定期修剪（策略 1），中间评分和排序结果不放入状态（策略 5）。
