# 【反直觉点】LangGraph 子图（Subgraph）与模块化

> 这些常见误区，90% 的初学者都会踩坑

---

## 误区1："子图不能访问父图状态" ❌

### 错误观点

> "子图是独立的 StateGraph，它和父图的状态是完全隔离的，子图无法知道父图发生了什么。"

### 为什么错

LangGraph 提供了两种机制让子图与父图通信：

1. **共享状态键**：只要子图和父图的 State 中有同名的 key，数据会自动双向传递。
2. **Command.PARENT**：子图节点可以通过 `Command(goto="parent_node", graph=Command.PARENT, update={"key": value})` 主动向父图发送状态更新和跳转指令。

```python
# 父图 State
class ParentState(TypedDict):
    messages: Annotated[list, add_messages]
    context: str

# 子图 State —— messages 同名，自动共享
class ChildState(TypedDict):
    messages: Annotated[list, add_messages]
    internal_data: str  # 这个字段父图看不到

# 子图中主动向父图通信
def child_node(state: ChildState):
    return Command(
        goto="parent_next_step",
        graph=Command.PARENT,
        update={"messages": [AIMessage(content="子图处理完成")]}
    )
```

### 为什么容易错

因为"子图"这个名字暗示了隔离和封装。大多数编程经验告诉我们，模块之间应该低耦合、不共享状态。但 LangGraph 的设计哲学是**可控的共享**——你可以选择隔离（不同名的 key 自然隔离），也可以选择共享（同名 key 自动传递）。

### 正确理解

子图和父图的状态关系是**可配置的**：同名 key 自动共享，不同名 key 自然隔离，`Command.PARENT` 提供主动通信通道。隔离是默认的，共享是可选的。

## 误区2："子图必须和父图用相同的 State schema" ❌

### 错误观点

> "要把一个子图加入父图，它们的 State 类型必须一样，否则数据传不过去。"

### 为什么错

LangGraph 支持**完全不同的 schema**，通过包装函数（wrapper function）进行状态转换。子图可以有自己独立的、与父图毫无关系的 State 定义。

```python
# 父图 State
class ParentState(TypedDict):
    user_query: str
    final_answer: str

# 子图 State —— 完全不同的结构
class ResearchState(TypedDict):
    topic: str
    sources: list[str]
    summary: str

# 包装函数：父图 State → 子图 State → 父图 State
def research_wrapper(state: ParentState) -> ParentState:
    # 转换输入
    sub_input = {"topic": state["user_query"], "sources": [], "summary": ""}
    # 调用子图
    result = research_subgraph.invoke(sub_input)
    # 转换输出
    return {"final_answer": result["summary"]}

# 在父图中使用包装函数作为节点
parent_graph.add_node("research", research_wrapper)
```

### 为什么容易错

因为共享状态键的集成方式要求同名 key，很多人以为这是唯一的集成方式。实际上，包装函数是更灵活的方案，它允许你把任何已有的子图集成到任何父图中，就像适配器模式一样。

### 正确理解

两种集成方式各有适用场景：**共享状态键**适合父子图天然有公共数据的情况（简单直接）；**包装函数**适合父子图 schema 不同、需要解耦的情况（灵活通用）。

## 误区3："子图需要单独配置 Checkpointer" ❌

### 错误观点

> "每个子图都要自己 `compile(checkpointer=MemorySaver())`，否则子图内部的状态不会被持久化。"

### 为什么错

子图**默认继承父图的 Checkpointer**。只要父图编译时配置了 Checkpointer，所有嵌套的子图都会自动使用同一个 Checkpointer，无需额外配置。

```python
from langgraph.checkpoint.memory import MemorySaver

# 子图：编译时不需要配置 checkpointer
child_graph = child_builder.compile()  # 不传 checkpointer

# 父图：只在最外层配置
parent_graph = parent_builder.compile(checkpointer=MemorySaver())

# 运行时，子图的状态也会被自动持久化
result = parent_graph.invoke(
    {"messages": [HumanMessage(content="hello")]},
    config={"configurable": {"thread_id": "1"}}
)
```

如果子图单独编译时传了 `checkpointer`，反而会导致冲突或重复持久化。

### 为什么容易错

因为在其他框架中，每个模块通常需要独立配置自己的持久化策略。而且子图可以独立编译运行（用于测试），独立运行时确实需要自己的 Checkpointer，这让人误以为集成到父图后也需要。

### 正确理解

**独立运行子图**（测试时）：需要自己配置 Checkpointer。**嵌入父图运行**（生产时）：自动继承父图 Checkpointer，子图编译时应该用 `compile()` 或 `compile(checkpointer=None)`。记住：Checkpointer 只在最外层配置一次。
