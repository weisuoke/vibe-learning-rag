# 状态管理最佳实践 - 面试必问

## 问题1："在 LangGraph 中，你会如何设计一个生产级的状态 Schema？"

### 普通回答（❌ 不出彩）：
"用 TypedDict 定义状态，加上类型注解就行了。"

### 出彩回答（✅ 推荐）：

> **生产级状态 Schema 设计需要考虑五个维度：**
>
> 1. **类型安全**：使用 TypedDict + Annotated 绑定 Reducer，简单场景用 TypedDict，需要运行时验证用 Pydantic
>
> 2. **更新策略**：每个字段选择合适的 Reducer —— 消息用 add_messages，配置用 LastValue，列表用带上限的自定义 reducer
>
> 3. **最小化原则**：只存源数据，派生数据运行时计算；临时数据用 EphemeralValue
>
> 4. **不可变性**：节点返回新字典而非修改 state 对象，确保检查点和并发安全
>
> 5. **可演进性**：新字段用 Optional + 默认值，保持向后兼容；使用 Input/Output Schema 分离内外接口
>
> **在实际项目中**，我会先用 TypedDict 快速迭代，Schema 稳定后考虑 Pydantic 验证。消息列表一定要有裁剪策略，否则状态会线性膨胀。

### 为什么这个回答出彩？
1. ✅ 展示了系统性思维（五个维度）
2. ✅ 结合了源码理解（Reducer、EphemeralValue）
3. ✅ 有实际经验（消息裁剪、渐进式选型）

## 问题2："LangGraph 中 Reducer 的设计有什么注意事项？为什么不能随便写？"

### 普通回答（❌ 不出彩）：
"Reducer 就是合并函数，把新值和旧值合在一起。"

### 出彩回答（✅ 推荐）：

> **Reducer 是状态更新的"合约"，有三个关键约束：**
>
> 1. **结合律**：(a ⊕ b) ⊕ c = a ⊕ (b ⊕ c)，因为 LangGraph 可能分批处理更新
>
> 2. **交换律**：a ⊕ b = b ⊕ a，因为并行节点的更新顺序不确定
>
> 3. **无副作用**：Reducer 不能修改外部状态，否则重放检查点时行为不一致
>
> **从源码看**，BinaryOperatorAggregate.update() 会遍历所有更新值依次应用 reducer。如果 reducer 不满足这些约束，并发执行时会产生不确定的结果。
>
> **实际建议**：优先使用内置 reducer（add_messages、operator.add），自定义时确保幂等性。需要重置状态时用 Overwrite 而非在 reducer 中加条件判断。

### 为什么这个回答出彩？
1. ✅ 理解底层约束（结合律、交换律）
2. ✅ 引用源码实现
3. ✅ 给出实际建议
