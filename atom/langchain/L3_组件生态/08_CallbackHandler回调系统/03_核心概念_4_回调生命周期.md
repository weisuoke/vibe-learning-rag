# 回调生命周期

## 1. 【30字核心】

**回调生命周期是 LangChain 中监控组件执行过程的钩子方法，通过 on_*_start/end/error 方法追踪 LLM、Chain、Tool 等组件的完整生命周期。**

---

## 2. 【第一性原理】

### 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题

### 回调生命周期的第一性原理

#### 1. 最基础的定义

**回调生命周期 = 在组件执行的关键时刻触发的钩子函数**

仅此而已！没有更基础的了。

#### 2. 为什么需要回调生命周期？

**核心问题：如何知道 LLM 调用发生了什么？**

想象你调用 OpenAI API：
- 什么时候开始调用？
- 花了多长时间？
- 用了多少 token？
- 是否出错？
- 生成了什么内容？

没有回调生命周期，这些信息都是黑盒。

#### 3. 回调生命周期的三层价值

##### 价值1：可观测性

**让黑盒变透明**

```python
# 没有回调：黑盒
result = llm.invoke("Hello")  # 不知道发生了什么

# 有回调：透明
class LogHandler(BaseCallbackHandler):
    def on_llm_start(self, serialized, prompts, **kwargs):
        print(f"开始调用 LLM: {prompts}")

    def on_llm_end(self, response, **kwargs):
        print(f"LLM 返回: {response}")

result = llm.invoke("Hello", callbacks=[LogHandler()])
```

##### 价值2：成本追踪

**实时监控 token 用量和成本**

```python
class CostHandler(BaseCallbackHandler):
    def __init__(self):
        self.total_tokens = 0

    def on_llm_end(self, response, **kwargs):
        usage = response.llm_output.get('token_usage', {})
        self.total_tokens += usage.get('total_tokens', 0)
        print(f"累计 tokens: {self.total_tokens}")
```

##### 价值3：流式输出

**实时显示生成内容**

```python
class StreamHandler(BaseCallbackHandler):
    def on_llm_new_token(self, token, **kwargs):
        print(token, end="", flush=True)
```

#### 4. 从第一性原理推导 RAG 应用

**推理链：**
```
1. RAG 系统需要调用 LLM
   ↓
2. LLM 调用是黑盒，需要监控
   ↓
3. 监控需要在关键时刻插入代码
   ↓
4. 关键时刻 = 开始、结束、错误、新 token
   ↓
5. 回调生命周期提供这些钩子
   ↓
6. 实现可观测性、成本追踪、流式输出
```

#### 5. 一句话总结第一性原理

**回调生命周期是在组件执行关键时刻触发的钩子函数，让黑盒变透明，实现可观测性、成本追踪和流式输出。**

## 3. 【核心概念】

### 核心概念1：启动回调（on_*_start）

**在组件开始执行时触发，用于记录输入和初始化状态**

```python
from langchain_core.callbacks import BaseCallbackHandler

class MyHandler(BaseCallbackHandler):
    def on_llm_start(self, serialized, prompts, **kwargs):
        """LLM 开始时触发"""
        print(f"LLM 开始: {prompts}")
        print(f"run_id: {kwargs.get('run_id')}")

    def on_chain_start(self, serialized, inputs, **kwargs):
        """Chain 开始时触发"""
        print(f"Chain 开始: {inputs}")

    def on_tool_start(self, serialized, input_str, **kwargs):
        """Tool 开始时触发"""
        print(f"Tool 开始: {input_str}")
```

**在 RAG 开发中：**
- 记录检索查询的开始时间
- 记录 LLM 调用的输入 prompt
- 追踪 Agent 工具调用的参数

---

### 核心概念2：结束回调（on_*_end）

**在组件成功完成时触发，用于记录输出和计算指标**

```python
class MyHandler(BaseCallbackHandler):
    def on_llm_end(self, response, **kwargs):
        """LLM 结束时触发"""
        print(f"LLM 输出: {response.generations}")
        usage = response.llm_output.get('token_usage', {})
        print(f"Token 用量: {usage}")

    def on_chain_end(self, outputs, **kwargs):
        """Chain 结束时触发"""
        print(f"Chain 输出: {outputs}")

    def on_tool_end(self, output, **kwargs):
        """Tool 结束时触发"""
        print(f"Tool 输出: {output}")
```

**在 RAG 开发中：**
- 计算 LLM 调用的延迟
- 统计 token 用量和成本
- 记录检索结果的数量和质量

---

### 核心概念3：错误回调（on_*_error）

**在组件执行出错时触发，用于错误处理和告警**

```python
class MyHandler(BaseCallbackHandler):
    def on_llm_error(self, error, **kwargs):
        """LLM 错误时触发"""
        print(f"LLM 错误: {error}")
        # 发送告警、记录日志

    def on_chain_error(self, error, **kwargs):
        """Chain 错误时触发"""
        print(f"Chain 错误: {error}")

    def on_tool_error(self, error, **kwargs):
        """Tool 错误时触发"""
        print(f"Tool 错误: {error}")
```

**在 RAG 开发中：**
- 捕获 API 调用失败
- 记录检索错误
- 实现重试逻辑

---

### 核心概念4：流式回调（on_llm_new_token）

**在 LLM 生成新 token 时触发，用于实时显示生成内容**

```python
class StreamHandler(BaseCallbackHandler):
    def on_llm_new_token(self, token, **kwargs):
        """每生成一个新 token 时触发"""
        print(token, end="", flush=True)
```

**在 RAG 开发中：**
- 实时显示 LLM 生成的答案
- 提升用户体验（类似 ChatGPT 的打字效果）
- 与 FastAPI 集成实现流式 API

### 4.1 监控 LLM 调用

```python
from langchain_core.callbacks import BaseCallbackHandler
from langchain_openai import ChatOpenAI

class SimpleHandler(BaseCallbackHandler):
    def on_llm_start(self, serialized, prompts, **kwargs):
        print(f"开始: {prompts[0][:50]}...")

    def on_llm_end(self, response, **kwargs):
        print(f"结束: {response.generations[0][0].text[:50]}...")

llm = ChatOpenAI(callbacks=[SimpleHandler()])
llm.invoke("Hello")
```

```python
class ChainHandler(BaseCallbackHandler):
```

## 6. 【反直觉点】

### 误区1：回调方法会自动传递给子组件 ❌

**为什么错？**
- 回调处理器不会自动传递给嵌套的组件
- 需要显式传递或使用全局配置

**为什么人们容易这样错？**
因为在前端框架（如 React）中，Context 会自动传递给子组件，所以人们以为回调也会自动传递。

**正确理解：**
```python
# ❌ 错误：以为回调会自动传递
llm = ChatOpenAI(callbacks=[handler])
chain = prompt | llm  # chain 不会自动继承 handler

# ✅ 正确：显式传递
chain = prompt | llm
chain.invoke(input, config={"callbacks": [handler]})
```

---

### 误区2：错误回调会阻止错误传播 ❌

**为什么错？**
- `on_*_error` 只是记录错误，不会阻止错误传播
- 错误仍然会抛出，需要在外层捕获

**为什么人们容易这样错？**
因为在事件监听器中，`preventDefault()` 可以阻止默认行为，所以人们以为错误回调也能阻止错误。

**正确理解：**
```python
class ErrorHandler(BaseCallbackHandler):
    def on_llm_error(self, error, **kwargs):
        print(f"记录错误: {error}")
        # 错误仍然会抛出！

try:
    llm.invoke("Hello", callbacks=[ErrorHandler()])
except Exception as e:
    print(f"外层捕获: {e}")  # 仍然需要 try-catch
```

---

### 误区3：流式回调在所有情况下都会触发 ❌

**为什么错？**
- `on_llm_new_token` 只在 `streaming=True` 时触发
- 非流式调用不会触发此方法

**为什么人们容易这样错？**
因为人们以为回调方法是"总是触发"的，忽略了流式模式的前提条件。

**正确理解：**
```python
# ❌ 错误：没有启用 streaming
llm = ChatOpenAI(callbacks=[StreamHandler()])
llm.invoke("Hello")  # on_llm_new_token 不会触发

# ✅ 正确：启用 streaming
llm = ChatOpenAI(streaming=True, callbacks=[StreamHandler()])
llm.invoke("Hello")  # on_llm_new_token 会触发
```

## 7. 【实战代码】

```python
"""
回调生命周期实战示例
演示：完整的 LLM 调用监控
"""

import time
from typing import Any
from langchain_core.callbacks import BaseCallbackHandler
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

# ===== 1. 完整的回调处理器 =====
class FullLifecycleHandler(BaseCallbackHandler):
    """监控完整生命周期的回调处理器"""

    def __init__(self):
        self.start_time = None
        self.total_tokens = 0

    def on_llm_start(self, serialized: dict, prompts: list[str], **kwargs: Any) -> None:
        """LLM 开始时触发"""
        self.start_time = time.time()
        print(f"\n{'='*50}")
        print(f"[开始] LLM 调用")
        print(f"Prompt: {prompts[0][:100]}...")
        print(f"run_id: {kwargs.get('run_id')}")

    def on_llm_new_token(self, token: str, **kwargs: Any) -> None:
        """新 token 生成时触发（流式）"""
        print(token, end="", flush=True)

    def on_llm_end(self, response, **kwargs: Any) -> None:
        """LLM 结束时触发"""
        elapsed = time.time() - self.start_time
        usage = response.llm_output.get('token_usage', {})
        self.total_tokens += usage.get('total_tokens', 0)

        print(f"\n\n[结束] LLM 调用")
        print(f"耗时: {elapsed:.2f}s")
        print(f"Token 用量: {usage}")
        print(f"累计 tokens: {self.total_tokens}")
        print(f"{'='*50}\n")

    def on_llm_error(self, error: BaseException, **kwargs: Any) -> None:
        """LLM 错误时触发"""
        print(f"\n[错误] {error}")
        print(f"run_id: {kwargs.get('run_id')}")

# ===== 2. 非流式调用 =====
print("=== 非流式调用 ===")
handler = FullLifecycleHandler()
llm = ChatOpenAI(temperature=0, callbacks=[handler])
response = llm.invoke([HumanMessage(content="What is 2+2?")])

# ===== 3. 流式调用 =====
print("\n=== 流式调用 ===")
handler = FullLifecycleHandler()
llm_streaming = ChatOpenAI(
    temperature=0,
    streaming=True,
    callbacks=[handler]
)
response = llm_streaming.invoke([HumanMessage(content="Tell me a short joke")])

# ===== 4. Chain 回调 =====
print("\n=== Chain 回调 ===")
from langchain_core.prompts import ChatPromptTemplate

class ChainHandler(BaseCallbackHandler):
    def on_chain_start(self, serialized, inputs, **kwargs):
        print(f"[Chain 开始] {inputs}")

    def on_chain_end(self, outputs, **kwargs):
        print(f"[Chain 结束] {outputs}")

prompt = ChatPromptTemplate.from_template("Tell me about {topic}")
chain = prompt | llm

chain.invoke(
    {"topic": "Python"},
    config={"callbacks": [ChainHandler()]}
)

# ===== 5. 错误处理 =====
print("\n=== 错误处理 ===")
class ErrorHandler(BaseCallbackHandler):
    def on_llm_error(self, error, **kwargs):
        print(f"[捕获错误] {type(error).__name__}: {error}")

try:
    llm_error = ChatOpenAI(
        api_key="invalid_key",
        callbacks=[ErrorHandler()]
    )
    llm_error.invoke("Hello")
except Exception as e:
    print(f"[外层捕获] {type(e).__name__}")
```

**运行输出示例：**
```

==================================================
[开始] LLM 调用
Prompt: What is 2+2?...
run_id: 12345678-1234-1234-1234-123456789012

[结束] LLM 调用
耗时: 0.85s
Token 用量: {'prompt_tokens': 10, 'completion_tokens': 5, 'total_tokens': 15}
累计 tokens: 15
==================================================
```

## 8. 【面试必问】

### 问题："如何实现完整的 LLM 调用监控？"

**普通回答（❌ 不出彩）：**
"使用回调处理器，重写 on_llm_start 和 on_llm_end 方法。"

**出彩回答（✅ 推荐）：**

> **完整的 LLM 调用监控需要覆盖三个层面：**
>
> 1. **生命周期追踪**：
>    - 使用 `on_llm_start` 记录开始时间和输入
>    - 使用 `on_llm_end` 计算延迟和 token 用量
>    - 使用 `on_llm_error` 捕获和记录错误
>
> 2. **流式输出支持**：
>    - 启用 `streaming=True`
>    - 实现 `on_llm_new_token` 实时显示生成内容
>    - 与 FastAPI 集成实现流式 API
>
> 3. **成本和性能监控**：
>    - 从 `response.llm_output['token_usage']` 提取 token 用量
>    - 根据模型定价计算成本
>    - 记录延迟、吞吐量等性能指标
>
> **与传统日志的区别**：回调处理器是非侵入式的，不需要修改业务代码，可以在初始化时或运行时动态注入。
>
> **在 RAG 系统中的应用**：可以追踪检索、重排序、生成等各个环节的性能，实现端到端的可观测性。

**为什么这个回答出彩？**
1. ✅ 分层次说明（生命周期、流式、成本）
2. ✅ 提到了与传统方法的区别
3. ✅ 联系了 RAG 实际应用
4. ✅ 展示了系统性思考

## 9. 【化骨绵掌】

### 卡片1：回调生命周期的本质

**一句话：** 回调生命周期是在组件执行关键时刻触发的钩子函数

**举例：**
```python
on_llm_start  → 开始时触发
on_llm_end    → 结束时触发
on_llm_error  → 错误时触发
```

**应用：** 在 RAG 系统中监控 LLM 调用的完整过程

---

### 卡片2：四类核心回调方法

**一句话：** 启动、结束、错误、流式四类方法覆盖完整生命周期

**举例：**
- 启动：`on_llm_start`, `on_chain_start`, `on_tool_start`
- 结束：`on_llm_end`, `on_chain_end`, `on_tool_end`
- 错误：`on_llm_error`, `on_chain_error`, `on_tool_error`
- 流式：`on_llm_new_token`

**应用：** 根据需求选择实现特定的回调方法

---

### 卡片3：回调参数标准化

**一句话：** 所有回调方法都接收 run_id、parent_run_id、tags、metadata 等标准参数

**举例：**
```python
def on_llm_start(self, serialized, prompts, *, run_id, parent_run_id, tags, **kwargs):
    pass
```

**应用：** 使用 run_id 追踪单次调用，使用 parent_run_id 追踪嵌套调用

---

### 卡片4：流式回调的前提条件

**一句话：** `on_llm_new_token` 只在 `streaming=True` 时触发

**举例：**
```python
llm = ChatOpenAI(streaming=True, callbacks=[StreamHandler()])
```

**应用：** 实现类似 ChatGPT 的打字效果

---

### 卡片5：回调处理器的传递方式

**一句话：** 回调处理器可以在初始化时或运行时传入

**举例：**
```python
# 初始化时
llm = ChatOpenAI(callbacks=[handler])

# 运行时
llm.invoke("Hello", config={"callbacks": [handler]})
```

**应用：** 运行时传入更灵活，适合动态配置

---

### 卡片6：错误回调不阻止错误传播

**一句话：** `on_*_error` 只记录错误，不会阻止错误抛出

**举例：**
```python
try:
    llm.invoke("Hello", callbacks=[ErrorHandler()])
except Exception as e:
    print(f"仍需外层捕获: {e}")
```

**应用：** 错误回调用于记录和告警，外层仍需 try-catch

---

### 卡片7：回调处理器不自动传递

**一句话：** 回调处理器不会自动传递给嵌套组件，需要显式传递
