# 核心概念3：批量处理与优化

## 概述

在实际的 RAG 应用中，我们经常需要处理大量文档的嵌入生成。单个文档逐一处理不仅效率低下，还会浪费 API 调用配额。批量处理与优化是提升 Embedding 性能的关键技术。

**为什么需要批量处理？**

1. **减少 API 调用次数**：批量请求可以将多个文本合并为一次 API 调用
2. **提高吞吐量**：并发处理可以充分利用网络带宽和 API 速率限制
3. **降低成本**：减少请求次数可以降低 API 调用成本
4. **改善用户体验**：更快的处理速度意味着更好的用户体验

**本文档涵盖的内容**：

- 批量嵌入的基础概念（batch_size, chunk_size）
- Token 管理与分块策略（embedding_ctx_length）
- 长文本处理（自动分块、加权平均）
- 异步批量处理（aembed_documents）
- 并发控制（max_concurrency）
- 进度监控（show_progress_bar）
- 性能优化技巧

## 1. 批量嵌入基础

### 1.1 核心参数

LangChain 的 Embedding 模型提供了两个关键参数来控制批量处理：

**chunk_size**：
- **定义**：每次 API 调用中包含的最大文本数量
- **默认值**：1000（OpenAI）
- **作用**：控制批量大小，避免单次请求过大

**batch_size**（部分模型支持）：
- **定义**：本地批量处理的大小
- **作用**：控制内存使用和并发处理

### 1.2 批量处理的优势

**示例对比**：

```python
# 逐个处理（低效）
embeddings = []
for text in texts:
    embedding = model.embed_query(text)
    embeddings.append(embedding)

# 批量处理（高效）
embeddings = model.embed_documents(texts)
```

**性能对比**：
- 逐个处理：1000 个文本 = 1000 次 API 调用
- 批量处理（chunk_size=100）：1000 个文本 = 10 次 API 调用

**效率提升**：约 100 倍

## 2. Token 管理与分块

### 2.1 Token 限制问题

OpenAI 的 Embedding 模型有 Token 限制：

| 模型 | Token 限制 |
|------|-----------|
| text-embedding-ada-002 | 8191 tokens |
| text-embedding-3-small | 8191 tokens |
| text-embedding-3-large | 8191 tokens |

**问题**：如果文本超过限制，API 会返回错误。

### 2.2 自动分块机制

LangChain 的 `OpenAIEmbeddings` 提供了自动分块机制：

**关键参数**：

```python
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",
    embedding_ctx_length=8191,        # Token 限制
    tiktoken_enabled=True,            # 启用 tiktoken 分词
    check_embedding_ctx_length=True   # 检查并自动分块
)
```

**工作流程**：

1. **Tokenization**：使用 tiktoken 将文本转换为 tokens
2. **长度检查**：检查 token 数量是否超过 `embedding_ctx_length`
3. **自动分块**：如果超过限制，自动将文本分成多个块
4. **独立嵌入**：每个块独立生成嵌入
5. **加权平均**：将多个嵌入合并为一个

### 2.3 非 OpenAI 提供商支持

对于不支持 tiktoken 的提供商（如 Ollama、vLLM），可以跳过 tokenization：

```python
embeddings = OpenAIEmbeddings(
    model="custom-model",
    base_url="https://your-api.com/v1",
    check_embedding_ctx_length=False  # 跳过 tokenization
)
```

**适用场景**：
- OpenAI 兼容 API（但不支持 tiktoken）
- 自定义 Embedding 服务
- 本地部署的模型

## 3. 长文本处理策略

### 3.1 自动分块算法

当文本超过 `embedding_ctx_length` 时，LangChain 会自动分块：

**分块逻辑**（源码分析）：

```python
def _tokenize(self, texts: list[str], chunk_size: int):
    """Tokenize and batch input texts."""
    
    # 1. Tokenization
    tokens = []
    for text in texts:
        if self.tiktoken_enabled:
            # 使用 tiktoken 分词
            token_ids = self.encoding.encode(text)
        else:
            # 使用 HuggingFace transformers
            token_ids = self.tokenizer.encode(text)
        
        # 2. 检查长度
        if len(token_ids) > self.embedding_ctx_length:
            # 自动分块
            chunks = self._split_tokens(token_ids)
            tokens.extend(chunks)
        else:
            tokens.append(token_ids)
    
    # 3. 批量分组
    batches = self._create_batches(tokens, chunk_size)
    return batches
```

### 3.2 加权平均合并

对于被分块的文本，LangChain 使用加权平均算法合并嵌入：

**算法原理**（源码分析）：

```python
def _process_batched_chunked_embeddings(self, embeddings, num_tokens):
    """Merge embeddings for chunked texts using weighted average."""
    
    results = []
    for text_embeddings, text_tokens in zip(embeddings, num_tokens):
        if len(text_embeddings) > 1:
            # 多个分块，需要合并
            
            # 1. 加权平均
            total_weight = sum(text_tokens)
            average = [
                sum(val * weight for val, weight in zip(embedding, text_tokens))
                / total_weight
                for embedding in zip(*text_embeddings)
            ]
            
            # 2. 归一化
            magnitude = sum(val**2 for val in average) ** 0.5
            normalized = [val / magnitude for val in average]
            
            results.append(normalized)
        else:
            # 单个分块，直接使用
            results.append(text_embeddings[0])
    
    return results
```

**关键点**：
- **权重**：基于每个分块的 token 数量
- **归一化**：保持向量的单位长度
- **用户无感知**：自动处理，用户无需关心

### 3.3 长文本处理示例

```python
from langchain_openai import OpenAIEmbeddings

# 初始化（启用自动分块）
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",
    embedding_ctx_length=8191,
    check_embedding_ctx_length=True
)

# 处理长文本（自动分块）
long_text = "..." * 10000  # 超过 8191 tokens
embedding = embeddings.embed_query(long_text)

# 结果：单个嵌入向量（自动合并）
print(f"Embedding dimension: {len(embedding)}")  # 1536
```

## 4. 异步批量处理

### 4.1 异步方法

LangChain 提供了异步版本的嵌入方法：

**同步 vs 异步**：

```python
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 同步方法
sync_embeddings = embeddings.embed_documents(texts)

# 异步方法
import asyncio

async def embed_async():
    async_embeddings = await embeddings.aembed_documents(texts)
    return async_embeddings

embeddings_result = asyncio.run(embed_async())
```

### 4.2 异步客户端

LangChain 内部使用双客户端设计：

```python
# 源码分析
class OpenAIEmbeddings(BaseModel, Embeddings):
    client: Any = Field(default=None, exclude=True)        # 同步客户端
    async_client: Any = Field(default=None, exclude=True)  # 异步客户端
    
    @model_validator(mode="after")
    def validate_environment(self) -> Self:
        # 初始化同步客户端
        self.client = openai.OpenAI(**params).embeddings
        
        # 初始化异步客户端
        self.async_client = openai.AsyncOpenAI(**params).embeddings
        
        return self
```

### 4.3 性能对比

**场景**：嵌入 1000 个文档

| 方法 | 时间 | 说明 |
|------|------|------|
| 同步逐个 | ~100s | 1000 次 API 调用 |
| 同步批量 | ~10s | 10 次 API 调用（chunk_size=100） |
| 异步批量 | ~3s | 并发处理 |

**性能提升**：异步批量比同步逐个快约 33 倍

## 5. 并发控制

### 5.1 max_concurrency 参数

LangChain 支持通过 `max_concurrency` 控制并发数：

```python
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 批量处理（限制并发数）
embeddings_result = embeddings.batch(
    texts,
    config={
        'max_concurrency': 5  # 最多 5 个并发请求
    }
)
```

### 5.2 速率限制处理

OpenAI API 有速率限制：

| 限制类型 | 限制值 |
|---------|--------|
| RPM（每分钟请求数） | 3000 |
| TPM（每分钟 Token 数） | 1000000 |

**策略**：
1. **控制并发数**：避免超过 RPM 限制
2. **自动重试**：遇到速率限制时自动重试
3. **指数退避**：重试间隔逐渐增加

### 5.3 重试机制

LangChain 内置了重试机制：

```python
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",
    max_retries=2,              # 最大重试次数
    retry_min_seconds=4,        # 最小重试间隔
    retry_max_seconds=20        # 最大重试间隔
)
```

**重试策略**：Exponential Backoff
- 第 1 次重试：等待 4 秒
- 第 2 次重试：等待 8 秒
- 第 3 次重试：等待 16 秒


### 6.2 自定义进度监控

对于更复杂的场景，可以自定义进度监控：

```python
from tqdm import tqdm

def embed_with_progress(texts, batch_size=100):
    embeddings_model = OpenAIEmbeddings(model="text-embedding-3-small")
    results = []
    
    # 分批处理
    for i in tqdm(range(0, len(texts), batch_size), desc="Embedding"):
        batch = texts[i:i+batch_size]
        batch_embeddings = embeddings_model.embed_documents(batch)
        results.extend(batch_embeddings)
    
    return results

# 使用
embeddings = embed_with_progress(large_corpus)
```

## 7. 性能优化技巧

### 7.1 批量大小调优

**原则**：
- **太小**：API 调用次数多，效率低
- **太大**：单次请求时间长，容易超时

**推荐值**：

| 场景 | chunk_size | 说明 |
|------|-----------|------|
| 短文本（<100 tokens） | 500-1000 | 可以更大 |
| 中等文本（100-500 tokens） | 100-500 | 平衡效率和稳定性 |
| 长文本（>500 tokens） | 50-100 | 避免超时 |

### 7.2 并发数调优

**原则**：
- **太小**：无法充分利用 API 速率限制
- **太大**：容易触发速率限制

**推荐值**：

| API 速率限制 | max_concurrency | 说明 |
|-------------|----------------|------|
| 3000 RPM | 5-10 | 保守策略 |
| 10000 RPM | 20-50 | 激进策略 |

### 7.3 缓存策略

对于重复的文本，使用缓存可以显著提升性能：

```python
from langchain.embeddings import CacheBackedEmbeddings
from langchain.storage import LocalFileStore
from langchain_openai import OpenAIEmbeddings

# 创建缓存存储
store = LocalFileStore("./embeddings_cache")

# 创建缓存支持的 Embeddings
cached_embeddings = CacheBackedEmbeddings.from_bytes_store(
    OpenAIEmbeddings(model="text-embedding-3-small"),
    store,
    namespace="openai-3-small"
)

# 使用（自动缓存）
embeddings = cached_embeddings.embed_documents(texts)
```

**优势**：
- 重复文本无需重新计算
- 降低 API 调用成本
- 提升响应速度

### 7.4 模型选择

根据场景选择合适的模型：

| 模型 | 维度 | 性能 | 成本 | 适用场景 |
|------|------|------|------|---------|
| text-embedding-3-small | 512 | 良好 | 低 | 大规模文档、成本敏感 |
| text-embedding-3-large | 1536 | 最好 | 高 | 高精度要求、小规模文档 |
| text-embedding-ada-002 | 1536 | 中等 | 中 | 兼容性要求 |

**降维优化**（text-embedding-3+）：

```python
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-large",
    dimensions=512  # 降低维度以节省成本
)
```

**效果**：
- 存储空间：减少 67%
- 计算成本：减少 67%
- 性能损失：约 5-10%

### 8.3 大规模文档处理

```python
from langchain_openai import OpenAIEmbeddings
from langchain.embeddings import CacheBackedEmbeddings
from langchain.storage import LocalFileStore
from tqdm import tqdm
import time

def process_large_corpus(documents, batch_size=100):
    """处理大规模文档语料库"""
    
    # 1. 创建缓存
    store = LocalFileStore("./embeddings_cache")
    base_embeddings = OpenAIEmbeddings(
        model="text-embedding-3-small",
        chunk_size=batch_size
    )
    cached_embeddings = CacheBackedEmbeddings.from_bytes_store(
        base_embeddings,
        store,
        namespace="large-corpus"
    )
    
    # 2. 分批处理
    results = []
    start_time = time.time()
    
    for i in tqdm(range(0, len(documents), batch_size), desc="Processing"):
        batch = documents[i:i+batch_size]
        batch_embeddings = cached_embeddings.embed_documents(batch)
        results.extend(batch_embeddings)
    
    elapsed_time = time.time() - start_time
    
    # 3. 统计信息
    print(f"\n处理完成！")
    print(f"文档数量：{len(documents)}")
    print(f"总耗时：{elapsed_time:.2f} 秒")
    print(f"平均速度：{len(documents)/elapsed_time:.2f} 文档/秒")
    
    return results

# 使用
large_corpus = ["文档 " + str(i) for i in range(10000)]
embeddings = process_large_corpus(large_corpus)
```

## 10. 常见问题

### Q1: chunk_size 和 batch_size 有什么区别？

**A**: 
- **chunk_size**：每次 API 调用的文本数量（网络层面）
- **batch_size**：本地批量处理的大小（内存层面）

在 LangChain 中，主要使用 `chunk_size`。

### Q2: 如何处理超长文本？

**A**: 
1. 启用 `check_embedding_ctx_length=True`
2. LangChain 会自动分块并合并
3. 或者手动分块后再嵌入

### Q3: 异步处理一定比同步快吗？

**A**: 
- 对于大规模批量处理：是的
- 对于小规模处理：差异不大
- 需要考虑并发控制和速率限制

### Q4: 如何避免触发速率限制？

**A**: 
1. 设置合理的 `max_concurrency`
2. 使用重试机制
3. 监控 API 使用情况

### Q5: 缓存会占用多少存储空间？

**A**: 
- 每个嵌入向量：约 6KB（1536 维 float32）
- 10000 个文档：约 60MB
- 建议定期清理缓存
