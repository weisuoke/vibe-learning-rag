## 什么是异常匹配策略？

**异常匹配策略 = 判断一个异常是否"值得重试"的规则。**

LangGraph 提供三种方式来定义"哪些异常触发重试"，从简单到灵活：

```
形式1：单一异常类型    → "只重试这种错误"
形式2：异常类型序列    → "重试这几种错误中的任何一种"
形式3：自定义判断函数  → "我来决定什么时候重试"
```

---

## 源码解析：`_should_retry_on()`

```python
# 来自 pregel/_retry.py
def _should_retry_on(
    retry_policy: RetryPolicy,
    exc: Exception,
) -> bool:
    """判断异常是否匹配重试策略。"""
    if isinstance(retry_policy.retry_on, type):
        # 形式1：单一异常类型
        return isinstance(exc, retry_policy.retry_on)
    if isinstance(retry_policy.retry_on, Sequence):
        # 形式2：异常类型序列
        return isinstance(exc, tuple(retry_policy.retry_on))
    if callable(retry_policy.retry_on):
        # 形式3：自定义函数
        return retry_policy.retry_on(exc)
    return False
```

**注意判断顺序：** type → Sequence → callable。这意味着如果你传入一个异常类（它既是 `type` 又是 `callable`），会按 `type` 处理。

## 形式1：单一异常类型

**适用场景：** 你明确知道只需要重试一种特定异常。

```python
from langgraph.types import RetryPolicy

# 只重试 ConnectionError
policy = RetryPolicy(retry_on=ConnectionError)

# 只重试 TimeoutError
policy = RetryPolicy(retry_on=TimeoutError)

# 只重试 ValueError（不推荐，通常是编程错误）
policy = RetryPolicy(retry_on=ValueError)
```

**匹配原理：**

```python
# 底层使用 isinstance，支持继承关系！
isinstance(ConnectionError(), ConnectionError)  # True
isinstance(ConnectionResetError(), ConnectionError)  # True ← 子类也匹配！
isinstance(TimeoutError(), ConnectionError)  # False
```

**继承匹配示例：**

```
ConnectionError
├── ConnectionResetError    ← retry_on=ConnectionError 也会匹配它！
├── ConnectionAbortedError  ← 也会匹配！
└── ConnectionRefusedError  ← 也会匹配！
```

**在 LangGraph 工作流中：**

```python
import sqlite3
from langgraph.types import RetryPolicy
from langgraph.graph import StateGraph, START, END

# 数据库操作只重试操作错误（锁、连接问题）
builder.add_node(
    "query_database",
    query_database,
    retry_policy=RetryPolicy(retry_on=sqlite3.OperationalError),
)
```

## 形式2：异常类型序列

**适用场景：** 需要重试多种不同类型的异常。

```python
from langgraph.types import RetryPolicy

# 重试网络错误或超时错误
policy = RetryPolicy(
    retry_on=[ConnectionError, TimeoutError]
)

# 重试多种 HTTP 库的异常
import httpx
import requests

policy = RetryPolicy(
    retry_on=[
        ConnectionError,
        TimeoutError,
        httpx.HTTPStatusError,
        requests.HTTPError,
    ]
)
```

**匹配原理：**

```python
# 序列被转为 tuple 用于 isinstance
isinstance(exc, tuple([ConnectionError, TimeoutError]))

# 等价于
isinstance(exc, (ConnectionError, TimeoutError))
```

**实际应用 — RAG 检索节点：**

```python
from langgraph.types import RetryPolicy

# RAG 检索可能遇到多种瞬态错误
rag_retry = RetryPolicy(
    retry_on=[
        ConnectionError,      # 网络连接问题
        TimeoutError,         # 请求超时
        ConnectionResetError, # 连接被重置
    ],
    max_attempts=3,
    initial_interval=1.0,
)

builder.add_node("retrieve", retrieve_documents, retry_policy=rag_retry)
```

## 形式3：自定义判断函数

**适用场景：** 需要根据异常的具体内容（如 HTTP 状态码）来决定是否重试。

```python
from langgraph.types import RetryPolicy
import httpx

# 只重试特定的 HTTP 状态码
def should_retry(exc: Exception) -> bool:
    """只重试可恢复的 HTTP 错误。"""
    if isinstance(exc, httpx.HTTPStatusError):
        # 429: 速率限制, 5xx: 服务端错误
        return exc.response.status_code in [429, 500, 502, 503, 504]
    if isinstance(exc, ConnectionError):
        return True  # 网络错误总是重试
    return False  # 其他错误不重试

policy = RetryPolicy(retry_on=should_retry)
```

**函数签名要求：**

```python
# 必须接受一个 Exception 参数，返回 bool
def retry_on(exc: Exception) -> bool:
    ...
```

**高级示例 — 解析 Retry-After 头：**

```python
import httpx

def smart_retry(exc: Exception) -> bool:
    """智能重试判断：解析 Retry-After 响应头。"""
    if isinstance(exc, httpx.HTTPStatusError):
        status = exc.response.status_code

        # 429 速率限制 → 总是重试
        if status == 429:
            retry_after = exc.response.headers.get("Retry-After")
            if retry_after:
                print(f"⏳ 服务器建议等待 {retry_after} 秒")
            return True

        # 5xx 服务端错误 → 重试
        if 500 <= status < 600:
            return True

        # 4xx 客户端错误 → 不重试（修复请求才行）
        return False

    # 连接错误 → 重试
    if isinstance(exc, (ConnectionError, TimeoutError)):
        return True

    return False

policy = RetryPolicy(retry_on=smart_retry)
```

## 默认的 `default_retry_on` 函数

**如果不指定 `retry_on`，LangGraph 使用内置的智能判断。**

```python
# 等价于
RetryPolicy()  # 使用 default_retry_on
```

**详细内容见：** `03_核心概念_5_默认重试行为.md`

**快速预览：**

```
✅ 重试：ConnectionError、HTTP 5xx
❌ 不重试：ValueError、TypeError、SyntaxError 等编程错误
✅ 重试：其他未知异常（保守策略，宁可多试）
```

## 三种形式的对比

| 特征 | 单一类型 | 类型序列 | 自定义函数 |
|------|---------|---------|-----------|
| **复杂度** | ⭐ | ⭐⭐ | ⭐⭐⭐ |
| **灵活度** | 低 | 中 | 高 |
| **适用场景** | 只有一种错误 | 几种错误 | 需要检查错误细节 |
| **支持继承** | ✅ | ✅ | 需手动实现 |
| **检查状态码** | ❌ | ❌ | ✅ |
| **日志/打印** | ❌ | ❌ | ✅ |

**选择建议：**

```
你知道确切的异常类型？ → 形式1（单一类型）
你有多种异常要重试？   → 形式2（类型序列）
你需要检查异常细节？   → 形式3（自定义函数）
不确定？              → 使用默认的 default_retry_on
```

## 测试用例验证

以下测试来自 LangGraph 源码（`tests/test_retry.py`），验证三种匹配方式：

```python
# 测试1：单一异常类型
def test_retry_policy_single_exception():
    policy = RetryPolicy(retry_on=ValueError, max_attempts=3)
    # ValueError → 重试 ✅
    # KeyError → 不重试 ❌（抛出）

# 测试2：异常类型序列
def test_retry_policy_sequence_exception():
    policy = RetryPolicy(retry_on=[ValueError, KeyError])
    # ValueError → 重试 ✅
    # KeyError → 重试 ✅
    # TypeError → 不重试 ❌

# 测试3：自定义函数
def test_retry_policy_callable():
    def custom_retry(exc):
        return isinstance(exc, ValueError) and "transient" in str(exc)
    policy = RetryPolicy(retry_on=custom_retry)
    # ValueError("transient error") → 重试 ✅
    # ValueError("permanent error") → 不重试 ❌
```

## 前端类比

```
异常匹配策略 ≈ 前端的错误拦截器

// 形式1：单一类型 ≈ catch 特定错误
axios.interceptors.response.use(null, (error) => {
    if (error.code === 'ECONNABORTED') retry();  // 只处理超时
});

// 形式2：类型列表 ≈ 多条件判断
if (['ECONNABORTED', 'ECONNRESET', 'ETIMEDOUT'].includes(error.code)) {
    retry();
}

// 形式3：自定义函数 ≈ 完全自定义拦截器
axios.interceptors.response.use(null, (error) => {
    if (error.response?.status === 429) retry();
    if (error.response?.status >= 500) retry();
});
```
