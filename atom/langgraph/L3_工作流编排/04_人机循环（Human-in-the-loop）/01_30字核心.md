# 人机循环（Human-in-the-loop） - 30字核心

**在节点内调用 interrupt(value) 暂停图执行等待人类决策，通过 Command(resume=value) 恢复继续运行。**

---

## 核心要素拆解

### 1. 中断（Interrupt）—— 暂停执行

**定义**: 在节点执行过程中调用 `interrupt(value)`，图立即暂停，将 value 发送给人类

**关键机制**:
- `interrupt()` 内部抛出 `GraphInterrupt` 异常
- Checkpointer 自动保存当前状态
- value 作为中断信息传递给客户端（必须 JSON 可序列化）

**价值**: 让 AI 在关键决策点"停下来问人"，而不是盲目执行

---

### 2. 恢复（Resume）—— 继续执行

**定义**: 人类做出决策后，通过 `Command(resume=value)` 恢复图的执行

**关键机制**:
- 使用相同的 `thread_id` 定位到暂停的工作流
- 节点从头重新执行（不是从 interrupt 处继续）
- `interrupt()` 在重新执行时返回 resume 值

**价值**: 将人类的判断注入 AI 工作流，实现人机协作

---

### 3. 在 LangGraph 中的价值

**安全保障**:
- 敏感操作前必须人类确认
- 防止 AI 幻觉导致的错误操作
- 满足合规审计要求

**模式支持**:
- 审批工作流（approve/reject）
- 工具调用确认（Agent 安全控制）
- 迭代反馈（人类指导 AI 修改）
- 多级审批链

**架构优势**:
- 基于 Checkpoint 的状态持久化
- 支持异步长时间等待（小时/天级别）
- 与 Web 框架无缝集成

---

## 一句话记忆

**interrupt() 暂停问人，Command(resume) 带着答案继续跑。**

---

## 核心公式

```
完整的人机循环 = interrupt(value) → 暂停 → 人类决策 → Command(resume=value) → 恢复
```

拆解每一步：

```
步骤 1: graph.invoke(input, config)          # 启动工作流
步骤 2: interrupt("需要你确认")               # 节点内暂停
步骤 3: → GraphInterrupt 异常                # 内部机制
步骤 4: → Checkpointer 保存状态              # 自动存档
步骤 5: → 返回 __interrupt__ 给客户端        # 人类看到中断信息
步骤 6: 人类做出决策                          # 等待人类
步骤 7: graph.invoke(Command(resume=决策), config)  # 恢复执行
步骤 8: → 节点从头重新执行                    # 关键！不是从中断处继续
步骤 9: → interrupt() 返回 resume 值          # 这次不暂停了
步骤 10: → 继续后续节点                       # 工作流继续
```

## 快速示例

### 最简示例：单节点中断与恢复

```python
from typing import TypedDict, Optional
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import interrupt, Command

# ========== 1. 定义状态 ==========
class State(TypedDict):
    message: str
    approved: Optional[bool]

# ========== 2. 定义节点 ==========
def ask_human(state: State):
    """这个节点会暂停，等待人类输入"""
    # interrupt() 第一次执行时暂停，恢复时返回 resume 值
    decision = interrupt({
        "question": "是否批准这条消息？",
        "content": state["message"]
    })
    return {"approved": decision}

def process_result(state: State):
    """根据人类决策处理结果"""
    if state["approved"]:
        return {"message": f"✅ 已批准: {state['message']}"}
    return {"message": f"❌ 已拒绝: {state['message']}"}

# ========== 3. 构建图 ==========
builder = StateGraph(State)
builder.add_node("ask_human", ask_human)
builder.add_node("process", process_result)

builder.add_edge(START, "ask_human")
builder.add_edge("ask_human", "process")
builder.add_edge("process", END)

# ========== 4. 编译（必须有 checkpointer！） ==========
graph = builder.compile(checkpointer=MemorySaver())

# ========== 5. 运行 ==========
config = {"configurable": {"thread_id": "thread-1"}}

# 第一次调用 → 在 ask_human 节点暂停
result = graph.invoke({"message": "发布新版本 v2.0"}, config)
print("中断了！等待人类决策...")
# 输出中会包含 __interrupt__ 信息

# 第二次调用 → 人类批准，恢复执行
final = graph.invoke(Command(resume=True), config)
print(final)
# {'message': '✅ 已批准: 发布新版本 v2.0', 'approved': True}
```

### 审批路由示例：根据决策走不同分支

```python
from typing import Literal
from langgraph.types import Command, interrupt

def approval_node(state) -> Command[Literal["execute", "cancel"]]:
    """审批节点：根据人类决策路由到不同分支"""
    decision = interrupt({
        "question": "审批转账请求",
        "amount": state["amount"],
        "recipient": state["recipient"]
    })

    if decision:
        # 批准 → 跳转到 execute 节点
        return Command(goto="execute")
    else:
        # 拒绝 → 跳转到 cancel 节点
        return Command(goto="cancel")

def execute_node(state):
    return {"status": "approved", "message": f"已转账 ${state['amount']}"}

def cancel_node(state):
    return {"status": "rejected", "message": "转账已取消"}
```

### 流式处理中断

```python
# 使用 stream 模式处理中断
config = {"configurable": {"thread_id": "stream-demo"}}

for chunk in graph.stream({"message": "测试消息"}, config):
    if "__interrupt__" in chunk:
        # 检测到中断
        interrupt_info = chunk["__interrupt__"][0].value
        print(f"需要人类决策: {interrupt_info}")
        break

# 恢复（也可以用 stream 模式）
for chunk in graph.stream(Command(resume=True), config):
    print(chunk)
```

## 核心对比

### interrupt() vs interrupt_before/after

| 特性 | interrupt() 动态中断 | interrupt_before/after 静态断点 |
|------|---------------------|-------------------------------|
| 配置位置 | 节点代码内部 | compile() 参数 |
| 灵活性 | ⭐⭐⭐⭐⭐ 支持条件逻辑 | ⭐⭐⭐ 固定节点列表 |
| 数据传递 | 可传递任意 JSON 值 | 无数据传递 |
| 中断位置 | 节点内任意位置 | 节点执行前/后 |
| 多中断点 | 支持单节点多个 interrupt | 不支持 |
| 官方推荐 | ✅ 推荐方式 | 遗留方式 |

### 四种恢复方式

| 方式 | 代码 | 适用场景 |
|------|------|----------|
| 同步 | `graph.invoke(Command(resume=v), config)` | 简单脚本 |
| 异步 | `await graph.ainvoke(Command(resume=v), config)` | Web 服务 |
| 同步流式 | `graph.stream(Command(resume=v), config)` | 实时展示 |
| 异步流式 | `graph.astream(Command(resume=v), config)` | 生产级服务 |

## 关键类比

### 前端类比：表单提交等待确认

```
interrupt()  →  用户点击"提交"后弹出确认框
                页面暂停等待用户操作
                ↓
Command()    →  用户点击"确认"或"取消"
                页面根据选择继续执行
```

就像前端的 `window.confirm("确定要删除吗？")`：
- 代码执行到这里暂停
- 弹出对话框等待用户
- 用户点击后返回 true/false
- 代码根据返回值继续

不同之处：LangGraph 的中断可以跨进程、跨时间（等待几小时甚至几天）。

### 日常生活类比：审批流程

```
interrupt()  →  员工提交报销单
                流程暂停，等待领导审批
                ↓
Command()    →  领导在 OA 系统中点击"批准"
                报销流程继续执行（财务打款）
```

就像公司的 OA 审批系统：
- 你提交一个申请（interrupt 传递信息）
- 流程暂停，等待审批人处理
- 审批人可以批准、驳回、要求修改
- 审批完成后流程自动继续下一步

关键相似点：
- 状态被保存（不会因为等待而丢失）
- 可以等待很长时间（异步）
- 审批人看到完整的申请信息
- 根据审批结果走不同流程

## 三个必须记住的规则

### 规则 1: 必须配置 Checkpointer

```python
# ❌ 错误：没有 checkpointer，interrupt 无法工作
graph = builder.compile()

# ✅ 正确：配置 checkpointer
graph = builder.compile(checkpointer=MemorySaver())
```

没有 Checkpointer，中断时状态无法保存，恢复时找不到之前的执行上下文。

### 规则 2: thread_id 必须一致

```python
config = {"configurable": {"thread_id": "my-thread"}}

# 启动（使用 thread_id）
graph.invoke(input, config)

# 恢复（必须用同一个 thread_id！）
graph.invoke(Command(resume=value), config)  # ✅

# ❌ 错误：不同的 thread_id 会找不到之前的状态
other_config = {"configurable": {"thread_id": "different-thread"}}
graph.invoke(Command(resume=value), other_config)  # 找不到中断！
```

### 规则 3: 恢复时节点从头重新执行

```python
def my_node(state):
    print("这行每次恢复都会执行！")  # ⚠️ 注意副作用
    result = interrupt("需要确认")    # 第一次暂停，恢复时返回 resume 值
    print(f"人类说: {result}")        # 只在恢复后执行到这里
    return {"answer": result}
```

恢复时不是从 `interrupt()` 处继续，而是整个节点从头执行。已匹配的 interrupt 直接返回缓存的 resume 值。

## 速记口诀

```
人机循环三步走：
interrupt 暂停问，    ← 节点内调用，传递信息给人类
checkpoint 存状态，   ← 自动保存，支持长时间等待
Command resume 跑。   ← 人类决策后，恢复继续执行

两个必须不能忘：
checkpointer 要配好，  ← 没有它中断不工作
thread_id 要一样。     ← 不一致找不到状态

一个关键要记牢：
恢复时节点重头跑。     ← 不是从 interrupt 处继续！
```

## 核心 API 速查

```python
from langgraph.types import interrupt, Command
from langgraph.checkpoint.memory import MemorySaver

# 中断：在节点内调用
result = interrupt(value)        # value 发给人类，result 是 resume 值

# 恢复：在客户端调用
graph.invoke(Command(resume=value), config)

# 恢复 + 路由：跳转到指定节点
graph.invoke(Command(resume=value, goto="next_node"), config)

# 恢复 + 更新状态
graph.invoke(Command(resume=value, update={"key": "new_value"}), config)

# 检查中断状态
snapshot = graph.get_state(config)
print(snapshot.tasks)       # 查看待执行的任务
```
