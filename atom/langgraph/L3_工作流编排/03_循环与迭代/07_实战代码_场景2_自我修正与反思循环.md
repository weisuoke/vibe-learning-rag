## 场景描述

**场景**：构建一个 Generator-Critic 自校正循环系统。Generator 节点生成内容（如技术文案），Critic 节点评估质量并给出改进意见。不合格则回到 Generator 重新生成，直到通过或达到最大修正次数。

**为什么需要自校正？**

> "Worker agents inherently suffer from deep tunnel vision when trying to solve a complex problem. They lose the forest for the trees. The Critic node serves as the 'Adult in the room.'"
> — rajatpandit.com

单次生成的内容质量往往不够理想。通过引入 Critic 节点周期性评估，Agent 可以像人类一样"写初稿 → 审阅 → 修改 → 再审阅"，逐步提升输出质量。

**特点**：
- Generator-Critic 双节点循环架构
- 条件边实现"通过/不通过"路由
- `revision_count` 计数器防止无限循环（TTL 模式）
- `critique_history` 累积帮助 Generator 逐步改进
- 不依赖外部 API，使用模拟逻辑

**图结构**：

```
START → generator → critic → [通过?] → END
                      ↑         |
                      └── [不通过] ┘
```

[来源: reference/fetch_自校正循环_01.md - Generator-Critic 架构模式]

## 完整代码

```python
"""
LangGraph 实战代码 - 场景2：自我修正与反思循环
演示：Generator-Critic 自校正模式，让 Agent 学会自我改进

图结构：
  START → generator → critic → [通过?] → END
                        ↑         |
                        └── [不通过] ┘

来源：
  - ActiveWizards: Generator-Critic Self-Correction Loop
  - rajatpandit.com: The "Critic" Node as "Adult in the Room"
  - LangGraph 源码: _branch.py L146 条件边路由机制
"""

from __future__ import annotations

import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


# ============================================================
# 1. 状态定义
# ============================================================
# Generator-Critic 模式的核心：状态中必须包含"草稿"、"评审意见"和"修正计数器"
# [来源: fetch_自校正循环_01.md - AgentState 设计]

class CorrectionState(TypedDict):
    """自校正循环的状态定义"""
    task: str                                              # 任务描述
    draft: str                                             # 当前草稿
    critique: str                                          # 最新评审意见
    critique_history: Annotated[list[str], operator.add]   # 评审历史（累积）
    revision_count: int                                    # 修正次数（覆盖写入）
    is_approved: bool                                      # 是否通过评审
    quality_score: float                                   # 质量评分 0-1


# 最大修正次数 — 防止无限循环的"断路器"
# [来源: fetch_循环优化_01.md - TTL Step Count 模式]
MAX_REVISIONS = 3


# ============================================================
# 2. Generator 节点 — 内容生成器
# ============================================================
# Generator 根据 task + critique_history 生成/改进 draft
# 首次生成时没有 critique，后续根据评审意见改进
# [来源: fetch_自校正循环_01.md - Step 2: Generator Node]

def generator_node(state: CorrectionState) -> dict:
    """
    Generator 节点：根据任务描述和评审意见生成/改进内容。

    核心逻辑：
    - 首次调用：根据 task 生成初始草稿
    - 后续调用：根据 task + critique_history 改进草稿
    """
    task = state["task"]
    critique = state.get("critique", "")
    revision = state.get("revision_count", 0)

    print(f"\n{'='*50}")
    print(f"[Generator] 第 {revision + 1} 次生成")
    print(f"{'='*50}")

    if revision == 0:
        # 首次生成：基于任务描述创建初始草稿
        print(f"[Generator] 任务: {task}")
        draft = _generate_initial_draft(task)
    else:
        # 改进生成：基于评审意见修改草稿
        print(f"[Generator] 根据评审意见改进...")
        print(f"[Generator] 评审意见: {critique}")
        draft = _improve_draft(state["draft"], critique, revision)

    print(f"[Generator] 生成草稿: {draft[:80]}...")

    return {
        "draft": draft,
        "revision_count": revision + 1,
    }


def _generate_initial_draft(task: str) -> str:
    """模拟首次内容生成（实际项目中这里调用 LLM）"""
    # 模拟一个质量一般的初始草稿
    return (
        f"关于{task}的介绍：\n"
        f"这是一个重要的技术话题。"
        f"它在很多场景中都有应用。"
    )


def _improve_draft(current_draft: str, critique: str, revision: int) -> str:
    """模拟基于评审意见的改进（实际项目中这里调用 LLM）"""
    improvements = {
        1: (
            f"{current_draft}\n\n"
            f"[改进v2] 补充了技术细节和具体示例。"
            f"增加了性能对比数据和最佳实践建议。"
        ),
        2: (
            f"{current_draft}\n\n"
            f"[改进v3] 增加了架构图说明和代码示例。"
            f"补充了常见问题解答和注意事项。"
            f"添加了参考文献和延伸阅读。"
        ),
    }
    return improvements.get(revision, f"{current_draft}\n\n[改进v{revision+1}] 进一步优化。")


# ============================================================
# 3. Critic 节点 — 质量评审器
# ============================================================
# Critic 节点评估 draft 质量，给出 critique 和 is_approved
# "The Critic exists purely to observe the ongoing transaction."
# [来源: fetch_循环优化_01.md - The "Critic" Node]

def critic_node(state: CorrectionState) -> dict:
    """
    Critic 节点：评估草稿质量，给出评审意见。

    评估维度（模拟）：
    - 内容长度是否充分
    - 是否包含具体示例
    - 是否包含技术细节
    """
    draft = state["draft"]
    revision = state["revision_count"]

    print(f"\n{'='*50}")
    print(f"[Critic] 评审第 {revision} 版草稿")
    print(f"{'='*50}")

    # 模拟质量评估
    score, critique, approved = _evaluate_draft(draft, revision)

    print(f"[Critic] 质量评分: {score:.2f}")
    print(f"[Critic] 评审结果: {'通过' if approved else '需要修改'}")
    if not approved:
        print(f"[Critic] 改进建议: {critique}")

    return {
        "critique": critique,
        "critique_history": [f"[v{revision}] {critique}"],
        "is_approved": approved,
        "quality_score": score,
    }


def _evaluate_draft(draft: str, revision: int) -> tuple[float, str, bool]:
    """
    模拟质量评估逻辑。

    实际项目中，这里可以：
    - 用更小/更便宜的 LLM 做评估（如 GPT-4o-mini）
    - 用规则引擎做初步筛选
    - 用专门的评估模型打分
    """
    score = 0.0

    # 评估维度1：内容长度
    if len(draft) > 200:
        score += 0.3
    elif len(draft) > 100:
        score += 0.2
    else:
        score += 0.1

    # 评估维度2：是否包含具体内容
    detail_keywords = ["示例", "代码", "数据", "对比", "架构"]
    for kw in detail_keywords:
        if kw in draft:
            score += 0.1

    # 评估维度3：结构完整性
    structure_keywords = ["问题", "注意", "参考", "最佳实践"]
    for kw in structure_keywords:
        if kw in draft:
            score += 0.05

    score = min(1.0, score)

    # 生成评审意见
    if score >= 0.7:
        return score, "内容质量达标，结构完整，细节充分。", True
    elif score >= 0.4:
        return score, "需要补充更多技术细节、具体示例和代码片段。", False
    else:
        return score, "内容过于简略，缺少具体示例、技术细节和结构化组织。", False


# ============================================================
# 4. 路由函数 — 循环控制的核心
# ============================================================
# 路由函数是纯 Python 逻辑，不受 LLM 幻觉影响
# "Your router is entirely immune to LLM hallucinations"
# [来源: fetch_循环优化_01.md - The Soft Landing: TTL Step Count]
# [来源: source_循环迭代_01.md - _branch.py L146 路由执行流程]

def route_after_critique(state: CorrectionState) -> Literal["generator", "__end__"]:
    """
    路由函数：决定是继续修正还是结束。

    优先级：
    1. 通过评审 → 结束（成功）
    2. 达到最大修正次数 → 结束（优雅降级）
    3. 否则 → 回到 Generator 继续修正
    """
    # 优先级1：评审通过
    if state["is_approved"]:
        print(f"\n[路由] 评审通过！质量评分: {state['quality_score']:.2f}")
        return END

    # 优先级2：达到最大修正次数（断路器）
    if state["revision_count"] >= MAX_REVISIONS:
        print(f"\n[路由] 达到最大修正次数 {MAX_REVISIONS}，优雅降级")
        print(f"[路由] 当前最佳评分: {state['quality_score']:.2f}")
        return END

    # 默认：继续修正
    print(f"\n[路由] 未通过评审，继续第 {state['revision_count'] + 1} 次修正...")
    return "generator"


# ============================================================
# 5. 构建图
# ============================================================
# [来源: fetch_自校正循环_01.md - Step 3: Define the Graph]
# [来源: context7_langgraph_01.md - add_conditional_edges 用法]

def build_correction_graph() -> StateGraph:
    """构建 Generator-Critic 自校正循环图"""

    builder = StateGraph(CorrectionState)

    # 添加节点
    builder.add_node("generator", generator_node)
    builder.add_node("critic", critic_node)

    # 定义边
    # START → generator：入口
    builder.add_edge(START, "generator")

    # generator → critic：每次生成后都要评审
    builder.add_edge("generator", "critic")

    # critic → [条件边]：根据评审结果决定下一步
    # 这里使用 add_conditional_edges，路由函数返回节点名或 END
    # [来源: source_循环迭代_01.md - state.py L839 条件边注册]
    builder.add_conditional_edges("critic", route_after_critique)

    return builder.compile()


# ============================================================
# 6. 运行示例
# ============================================================

def main():
    """运行自校正循环示例"""

    print("=" * 60)
    print("  Generator-Critic 自校正循环演示")
    print("=" * 60)

    graph = build_correction_graph()

    # 执行图
    result = graph.invoke({
        "task": "LangGraph 循环机制",
        "draft": "",
        "critique": "",
        "critique_history": [],
        "revision_count": 0,
        "is_approved": False,
        "quality_score": 0.0,
    })

    # 输出最终结果
    print("\n" + "=" * 60)
    print("  最终结果")
    print("=" * 60)
    print(f"修正次数: {result['revision_count']}")
    print(f"最终评分: {result['quality_score']:.2f}")
    print(f"是否通过: {result['is_approved']}")
    print(f"评审历史:")
    for h in result["critique_history"]:
        print(f"  - {h}")
    print(f"\n最终草稿:\n{result['draft'][:200]}...")


if __name__ == "__main__":
    main()
```

## 关键模式解析

### 模式1：revision_count 防止无限循环

```python
# 路由函数中的断路器逻辑
if state["revision_count"] >= MAX_REVISIONS:
    return END  # 优雅降级，而非抛出异常
```

这是 rajatpandit.com 提出的 "TTL Step Count" 模式的实现。关键点：

- **计数器在状态中**，不依赖 LLM 判断，完全确定性
- **优雅降级**而非硬崩溃 — 返回当前最佳结果，而非抛出 `GraphRecursionError`
- **与 `recursion_limit` 互补** — `revision_count` 是业务层面的控制，`recursion_limit` 是框架层面的兜底

[来源: fetch_循环优化_01.md - "We want a soft landing, not a hard crash"]

### 模式2：critique_history 帮助 Generator 改进

```python
class CorrectionState(TypedDict):
    critique_history: Annotated[list[str], operator.add]  # 累积所有评审意见
```

使用 `operator.add` reducer 让评审历史自动累积。Generator 可以看到所有历史评审意见，避免重复犯同样的错误。

这对应 ActiveWizards 文章中的核心设计：

> "The state holds the problem description, the generated answer, and any feedback for correction."

[来源: fetch_自校正循环_01.md - AgentState 设计]

### 模式3：Critic 作为独立评估节点

```
generator → critic → [路由] → generator（循环）
                            → END（结束）
```

Critic 节点的设计原则：
- **不执行业务逻辑**，只做评估
- **可以用更便宜的模型**（如 GPT-4o-mini 代替 GPT-4o）
- **评估维度明确**，输出结构化的评审意见

[来源: fetch_循环优化_01.md - "The Critic exists purely to observe the ongoing transaction"]

## 对比：有/无 Critic 节点

| 维度 | 无 Critic（单次生成） | 有 Critic（自校正循环） |
|------|----------------------|------------------------|
| 输出质量 | 取决于单次 LLM 调用 | 逐步提升，多轮优化 |
| 成本 | 1 次 LLM 调用 | 2-6 次 LLM 调用 |
| 可控性 | 低，结果不可预测 | 高，有明确的质量门槛 |
| 适用场景 | 简单任务、对质量要求不高 | 代码生成、报告撰写、关键内容 |
| 失败处理 | 无法自动恢复 | 优雅降级，返回最佳结果 |

## RAG 应用联系

Generator-Critic 模式在 RAG 系统中的典型应用：

1. **答案质量校验**：RAG 生成答案后，Critic 检查答案是否基于检索到的文档（防止幻觉）
2. **查询改写循环**：初始查询检索效果不好 → Critic 评估 → 改写查询 → 重新检索（见场景3）
3. **文档摘要优化**：生成摘要 → 评估是否保留了关键信息 → 改进摘要
4. **Prompt 优化**：生成 Prompt → 测试效果 → 根据反馈调整 Prompt

## 扩展思考

**如何在生产环境中使用这个模式？**

1. 将 `_generate_initial_draft` 和 `_improve_draft` 替换为真实的 LLM 调用
2. 将 `_evaluate_draft` 替换为更便宜的评估模型或规则引擎
3. 添加 `recursion_limit` 作为框架层面的兜底保护
4. 集成 LangSmith 进行可观测性监控，追踪每轮修正的质量变化

# 生产环境建议的调用方式
result = graph.invoke(
    initial_state,
    config={"recursion_limit": 10}  # 框架兜底，业务层 MAX_REVISIONS=3
)
