# 错误处理与重试 - 双重类比

> 前端开发类比 + 日常生活类比

## 类比总览

| LangGraph 概念 | 前端类比 | 日常生活类比 |
|---------------|---------|-------------|
| RetryPolicy | axios-retry 配置 | 自动重拨的电话 |
| max_attempts | 请求重试次数限制 | 最多敲3次门 |
| initial_interval | setTimeout 延迟 | 第一次等30秒再试 |
| backoff_factor | 倍增的轮询间隔 | 每次多等一倍时间 |
| max_interval | 最大轮询间隔 | 最多等5分钟 |
| jitter | Math.random() 偏移 | 随机错开排队 |
| retry_on | 条件重试判断 | 只在"占线"时重拨 |
| default_retry_on | 默认错误拦截器 | 有经验的客服判断 |
| GraphBubbleUp | throw redirect() | 红灯停车（不是故障） |
| Fallback 链 | \<picture\> 图片降级 | 餐厅没位子去隔壁 |
| 断路器 | CDN 健康检查 | 保险丝自动断开 |

## 核心类比详解

### 1. RetryPolicy ≈ axios-retry 插件

**前端类比：**

```javascript
// 前端：给 axios 配置自动重试
import axiosRetry from 'axios-retry';

axiosRetry(axios, {
    retries: 3,                          // ≈ max_attempts
    retryDelay: (retryCount) => {
        return Math.pow(2, retryCount) * 500;  // ≈ backoff_factor
    },
    retryCondition: (error) => {         // ≈ retry_on
        return error.response?.status >= 500;
    },
});
```

```python
# LangGraph：给节点配置自动重试
from langgraph.types import RetryPolicy

builder.add_node("my_node", my_func,
    retry_policy=RetryPolicy(
        max_attempts=3,           # 总尝试次数
        backoff_factor=2.0,       # 间隔倍增
        initial_interval=0.5,     # 首次间隔
    )
)
```

**相同的思路：** 声明式配置，框架自动处理重试逻辑。

**日常生活类比：**

> 打电话给客服，占线了。手机自动帮你重拨：
> - 第一次等 30 秒重拨
> - 第二次等 1 分钟重拨
> - 第三次等 2 分钟重拨
> - 三次都没通 → 提示"稍后再试"

### 2. 指数退避 ≈ 前端轮询策略

**前端类比：**

```javascript
// ❌ 固定间隔轮询（持续施压）
setInterval(() => fetch('/api/status'), 1000);

// ✅ 指数退避轮询（逐渐放松）
let delay = 1000;
function poll() {
    fetch('/api/status')
        .catch(() => {
            delay = Math.min(delay * 2, 30000);  // 指数增长，上限30s
            setTimeout(poll, delay);
        });
}
```

```python
# LangGraph：内置指数退避
RetryPolicy(
    initial_interval=0.5,    # 起始 0.5s
    backoff_factor=2.0,      # 每次翻倍
    max_interval=128.0,      # 上限 128s
)
# 退避序列：0.5s → 1s → 2s → 4s → 8s → ...
```

**日常生活类比：**

> 去银行办事，窗口排队太长：
> - ❌ 每分钟跑去问一次"轮到我了吗？"（固定间隔 → 烦死工作人员）
> - ✅ 第一次 5 分钟后问，第二次 10 分钟后，第三次 20 分钟后（指数退避 → 大家都轻松）

---

### 3. Jitter ≈ Math.random() 偏移

**前端类比：**

```javascript
// ❌ 无抖动：所有用户同时轮询
setTimeout(poll, 2000);

// ✅ 有抖动：每个用户的时间略有不同
setTimeout(poll, 2000 + Math.random() * 1000);
```

**日常生活类比：**

> 停电恢复后：
> - ❌ 所有人同时开空调 → 电网再次过载 → 再跳闸
> - ✅ 每户随机等 0~60 秒 → 电网负载平稳上升 → 正常运行
>
> 这就是"雷群效应"（Thundering Herd）的防范。

---

### 4. GraphBubbleUp ≈ throw redirect()

**前端类比：**

```javascript
// Next.js 中，redirect() 使用 throw 实现
// 它不是"错误"，是"流程控制信号"
export default function Page() {
    if (!user) {
        redirect('/login');  // ← 用 throw 实现，但不是错误
    }
    return <Dashboard />;
}

// 类似地，notFound() 也用 throw 实现
// Next.js 知道这些"异常"不应该被 error boundary 捕获
```

```python
# LangGraph 中，GraphBubbleUp 不是错误，是信号
# run_with_retry() 知道它不应该被重试

try:
    node.execute()
except GraphBubbleUp:
    raise  # ← 直接抛出，不进入重试逻辑！
except Exception:
    # ← 其他异常才进入重试
```

**日常生活类比：**

> 红灯停车 vs 发动机故障：
> - 红灯停车（GraphBubbleUp）→ 不是故障，是交通信号 → 不需要"重试"
> - 发动机故障（ConnectionError）→ 真正的故障 → 需要修复/重试

### 5. Fallback 链 ≈ \<picture\> 图片降级

**前端类比：**

```html
<!-- 前端图片降级 -->
<picture>
    <source srcset="hero.webp" type="image/webp">    <!-- 主方案：最优格式 -->
    <source srcset="hero.jpg" type="image/jpeg">      <!-- Fallback 1 -->
    <img src="placeholder.svg" alt="占位图">           <!-- Fallback 2：兜底 -->
</picture>
```

```python
# LangGraph Fallback 链
# 主方案：GPT-4（最优质量）
# Fallback 1：GPT-3.5（快速但质量低）
# Fallback 2：缓存结果（不一定准确但有响应）
# Fallback 3：静态消息（"服务暂时不可用"）
```

**日常生活类比：**

> 想吃火锅：
> - 第一选择：海底捞（排队太长）
> - Fallback 1：隔壁的呷哺呷哺
> - Fallback 2：超市买火锅底料回家自己涮
> - Fallback 3：算了，今天吃泡面
>
> 每一层都是更低成本/更低质量的替代，但确保"能吃上饭"。

### 6. 断路器 ≈ CDN 健康检查 / 保险丝

**前端类比：**

```
CDN 的健康检查机制：
┌─────────────────────────────────────────────┐
│ CDN 边缘节点                                  │
│                                              │
│ 源站正常 → 正常回源获取内容（断路器关闭）       │
│ 源站连续超时 → 停止回源，直接用缓存（断路器打开）│
│ 过一段时间 → 尝试回源一次（半开状态）           │
│ 源站恢复 → 恢复正常回源（断路器关闭）           │
└─────────────────────────────────────────────┘
```

**日常生活类比：**

> 家里的保险丝（空气开关）：
> - 正常用电 → 保险丝接通（关闭状态）
> - 电流过大 → 保险丝自动断开（打开状态）→ 防止火灾
> - 修好问题 → 手动合闸试试（半开状态）
> - 一切正常 → 保险丝恢复接通（关闭状态）
>
> 断路器的意义：**宁可暂时没电，也不要引发火灾。**

---

### 7. default_retry_on ≈ 有经验的客服判断

**日常生活类比：**

> 有经验的客服接到投诉电话：
>
> "网络不好断线了" → "帮您重新连接"（✅ 重试）
> "服务器维护中" → "请稍后再试"（✅ 重试）
> "密码输错了" → "请核实密码"（❌ 不重试，换密码再来）
> "功能还没开发" → "暂不支持"（❌ 不重试，重试也没用）
> "不知道什么错" → "帮您再试一次"（✅ 保守重试）

## 完整类比对照表

```
LangGraph 错误处理流程          前端等价物                日常生活

节点执行失败                    fetch() 请求失败          打电话没接通
  ↓                              ↓                        ↓
GraphBubbleUp?                  redirect()?              红灯?
  Yes → 直接抛出                  Yes → 路由跳转            Yes → 停车等绿灯
  No ↓                           No ↓                      No ↓
retry_on 匹配?                  retryCondition?          值得再试?
  No → 直接抛出                   No → 显示错误页            No → 放弃
  Yes ↓                          Yes ↓                     Yes ↓
退避等待                        setTimeout(delay)         等一会儿
  ↓                              ↓                        ↓
重试执行                        重新 fetch()              重新拨号
  ↓                              ↓                        ↓
max_attempts 达到?              重试次数用完?              试了3次?
  No → 继续重试                   No → 继续                  No → 再等等
  Yes → Fallback                 Yes → 显示离线内容          Yes → 换个方式联系
```
