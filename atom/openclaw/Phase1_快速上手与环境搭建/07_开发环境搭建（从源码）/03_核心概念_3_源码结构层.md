## 一、概念定位

### 1.1 在三层架构中的位置

```
┌─────────────────────────────────────┐
│     依赖管理层（pnpm workspace）      │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│     构建工具链层（TypeScript）        │
└─────────────────────────────────────┘
              ↓ 编译代码
┌─────────────────────────────────────┐
│     源码结构层（Monorepo）           │ ← 当前层
│  - src/ 核心代码                     │
│  - ui/ 用户界面                      │
│  - extensions/ 扩展插件              │
└─────────────────────────────────────┘
```

### 1.2 核心职责

源码结构层负责：
1. **代码组织**：合理组织代码文件和目录
2. **模块划分**：将功能划分为独立模块
3. **依赖管理**：管理模块之间的依赖关系
4. **扩展机制**：提供插件和扩展接口

## 二、Monorepo 组织方式

### 2.1 什么是 Monorepo？

**定义**: Monorepo（单仓库）是将多个相关项目放在一个 Git 仓库中管理的方式。

**对比**:

| 方式 | 优势 | 劣势 |
|------|------|------|
| Monorepo | 统一管理、共享代码、原子提交 | 仓库体积大、构建慢 |
| Multirepo | 独立管理、灵活部署 | 依赖管理复杂、代码重复 |

**数据来源**: `reference/07_pnpm_workspace_best_practices.md` - monorepo 最佳实践

## 三、模块依赖关系

### 3.1 依赖图

```
┌─────────────┐
│   ui/       │
└──────┬──────┘
       │ 依赖
       ↓
┌─────────────┐
│   src/      │
└──────┬──────┘
       │ 依赖
       ↓
┌─────────────┐
│ plugin-sdk/ │
└─────────────┘
       ↑
       │ 使用
┌─────────────┐
│ extensions/ │
└─────────────┘
```

### 3.2 依赖规则

**规则 1: 单向依赖**

```
ui → src → plugin-sdk
```

**禁止**:

```
src → ui  ❌
plugin-sdk → src  ❌
```

---

**规则 2: 扩展独立**

```
extensions/ 只依赖 plugin-sdk
不依赖 src/ 的内部实现
```

---

**规则 3: 工具函数共享**

```
src/utils/ 可被所有模块使用
```

---

### 3.3 模块导出

**src/index.ts**:

```typescript
// 导出核心 API
export { OpenClaw } from './core';
export { Gateway } from './gateway';

// 导出类型
export * from './types';
```

**src/plugin-sdk/index.ts**:

```typescript
// 导出插件 SDK
export { Plugin } from './plugin';
export { PluginContext } from './context';

// 导出类型
export * from './types';
```

**package.json 导出配置**:

```json
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "default": "./dist/index.js"
    },
    "./plugin-sdk": {
      "types": "./dist/plugin-sdk/index.d.ts",
      "default": "./dist/plugin-sdk/index.js"
    }
  }
}
```

## 五、扩展机制

### 5.1 插件系统

**插件接口**:

```typescript
// src/plugin-sdk/plugin.ts
export abstract class Plugin {
  abstract name: string;
  abstract version: string;

  async onLoad(): Promise<void> {}
  async onUnload(): Promise<void> {}
  async onMessage(message: Message): Promise<void> {}
}
```

**插件加载**:

```typescript
// src/plugin/loader.ts
export class PluginLoader {
  async load(pluginPath: string): Promise<Plugin> {
    const module = await import(pluginPath);
    const PluginClass = module.default;
    return new PluginClass();
  }
}
```

---

### 5.2 扩展点

**扩展点定义**:

```typescript
// src/core/extension-points.ts
export interface ExtensionPoint {
  name: string;
  register(extension: Extension): void;
  execute(...args: any[]): Promise<any>;
}
```

**扩展点使用**:

```typescript
// 注册扩展
extensionPoint.register({
  name: 'custom-command',
  execute: async (args) => {
    // 自定义逻辑
  }
});

// 执行扩展
await extensionPoint.execute(args);
```

---

### 5.3 钩子系统

**钩子定义**:

```typescript
// src/core/hooks.ts
export interface Hooks {
  beforeMessage?: (message: Message) => Promise<Message>;
  afterMessage?: (message: Message) => Promise<void>;
  onError?: (error: Error) => Promise<void>;
}
```

**钩子使用**:

```typescript
// 注册钩子
openclaw.hooks.beforeMessage = async (message) => {
  console.log('Before message:', message);
  return message;
};

// 触发钩子
if (openclaw.hooks.beforeMessage) {
  message = await openclaw.hooks.beforeMessage(message);
}
```

## 六、最佳实践

### 6.1 目录命名

**推荐**:
- 使用小写字母
- 使用连字符分隔（kebab-case）
- 名称简洁明了

**示例**:

```
✅ src/plugin-sdk/
✅ src/gateway/
✅ src/utils/

❌ src/PluginSDK/
❌ src/Gateway_Service/
❌ src/utility_functions/
```

---

### 6.2 文件命名

**推荐**:
- 使用小写字母
- 使用连字符分隔（kebab-case）
- 文件名与导出内容一致

**示例**:

```typescript
// ✅ src/plugin-sdk/plugin.ts
export class Plugin {}

// ✅ src/gateway/provider.ts
export class Provider {}

// ❌ src/plugin-sdk/Plugin.ts
// ❌ src/gateway/GatewayProvider.ts
```

---

### 6.3 模块导出

**推荐**:
- 使用 barrel exports（index.ts）
- 导出公共 API
- 隐藏内部实现

**示例**:

```typescript
// src/plugin-sdk/index.ts
export { Plugin } from './plugin';
export { PluginContext } from './context';
export * from './types';

// 不导出内部实现
// export { PluginLoader } from './loader';  ❌
```

---

### 6.4 依赖管理

**推荐**:
- 避免循环依赖
- 使用依赖注入
- 最小化依赖

**示例**:

```typescript
// ✅ 依赖注入
class Gateway {
  constructor(private provider: Provider) {}
}

// ❌ 直接依赖
class Gateway {
  private provider = new OpenAIProvider();
}
```

## 七、常见问题

### 7.1 循环依赖

**问题**: 模块 A 依赖模块 B，模块 B 依赖模块 A

**检测**:

```bash
# 使用 madge 检测循环依赖
npx madge --circular src/
```

**解决方案**:
1. 提取共享代码到独立模块
2. 使用依赖注入
3. 重新设计模块结构

---

### 7.2 模块找不到

**问题**: TypeScript 找不到模块

**可能原因**:
1. 路径映射配置错误
2. 模块未导出
3. 编译产物缺失

**解决方案**:

```json
// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "openclaw/plugin-sdk": ["./src/plugin-sdk/index.ts"]
    }
  }
}
```

---

### 7.3 类型定义缺失

**问题**: 使用模块时没有类型提示

**可能原因**:
1. 未生成 .d.ts 文件
2. package.json 中 types 字段配置错误

**解决方案**:

```json
// package.json
{
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "default": "./dist/index.js"
    }
  }
}
```

## 八、与其他层的关系

### 8.1 依赖管理层 → 源码结构层

```
pnpm workspace
    ↓ 管理依赖
源码结构
```

**关键点**:
- workspace 配置决定了哪些目录是独立包
- 源码结构决定了模块之间的依赖关系

---

### 8.2 构建工具链层 → 源码结构层

```
tsconfig.json
    ↓ 编译配置
源码结构
    ↓ 编译
dist/
```

**关键点**:
- tsconfig.json 的 include 决定了哪些文件被编译
- 源码结构决定了编译产物的结构

## 九、总结

### 9.1 核心要点

1. **Monorepo** 统一管理多个相关项目
2. **模块划分** 按功能、层次或领域组织代码
3. **依赖管理** 避免循环依赖，使用依赖注入
4. **扩展机制** 提供插件和钩子系统

### 9.2 最佳实践

1. 使用 kebab-case 命名
2. 使用 barrel exports
3. 避免循环依赖
4. 最小化依赖
