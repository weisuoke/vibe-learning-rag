# 核心概念7：字段添加机制

> 本文档详细讲解 Milvus 2.6 AddCollectionField 字段添加机制、nullable 字段、默认值设置和生产环境最佳实践

---

## 一句话定义

**AddCollectionField 是 Milvus 提供的动态添加字段到已存在 Collection 的 API,支持 nullable 字段和默认值设置,通过 Schema 版本管理实现向后兼容,适用于渐进式 Schema 演化场景。**

---

## AddCollectionField 基础

### 1. API 定义

**定义**（来源：reference/context7_milvus_02.md）:

AddCollectionField 是向已存在的 Milvus Collection 添加新的标量字段的 API。

**核心特性**:
- **必须是 nullable 字段**：添加的字段必须设置 `nullable=True`
- **不能是向量字段**：只能添加标量字段（INT64, VARCHAR, JSON 等）
- **立即可用**：字段添加后几乎立即可用，延迟极小
- **已存在实体的值**：已存在的实体该字段值为 NULL（或默认值）

---

### 2. Python API

```python
from pymilvus import MilvusClient, DataType

client = MilvusClient("http://localhost:19530")

# 添加 nullable 字段
client.add_collection_field(
    collection_name="product_catalog",
    field_name="created_timestamp",  # 新字段名称
    data_type=DataType.INT64,        # 数据类型（必须是标量类型）
    nullable=True                    # 必须为 True
)

# 添加带默认值的字段
client.add_collection_field(
    collection_name="product_catalog",
    field_name="priority_level",
    data_type=DataType.VARCHAR,
    max_length=20,
    nullable=True,
    default_value="standard"  # 默认值
)
```

## 内部实现机制

### 1. DDL 回调实现

**源码实现**（来源：reference/source_动态Schema_01.md）:

```go
func (c *Core) broadcastAlterCollectionForAddField(ctx context.Context, req *milvuspb.AddCollectionFieldRequest) error {
    // 1. 检查 Collection 是否存在
    coll, err := c.meta.GetCollectionByName(ctx, req.GetDbName(), req.GetCollectionName(), typeutil.MaxTimestamp)

    // 2. 解析并验证字段 Schema
    fieldSchema := &schemapb.FieldSchema{}
    if err = proto.Unmarshal(req.Schema, fieldSchema); err != nil {
        return errors.Wrap(err, "failed to unmarshal field schema")
    }

    // 3. 检查字段是否已存在
    for _, field := range coll.Fields {
        if field.Name == fieldSchema.Name {
            return merr.WrapErrParameterInvalidMsg("field already exists, name: %s", fieldSchema.Name)
        }
    }

    // 4. 分配新的 field ID
    fieldSchema.FieldID = nextFieldID(coll)

    // 5. 构建新的 Collection Schema（版本号 +1）
    schema := &schemapb.CollectionSchema{
        Name:               coll.Name,
        Fields:             append(coll.Fields, fieldSchema),
        Version:            coll.SchemaVersion + 1,  // Schema 版本号递增
        EnableDynamicField: coll.EnableDynamicField,
    }

    // 6. 广播 AlterCollection 消息
    msg := message.NewAlterCollectionMessageBuilderV2().
        WithHeader(&messagespb.AlterCollectionMessageHeader{
            DbId:         coll.DBID,
            CollectionId: coll.CollectionID,
        }).
        WithBody(&messagespb.AlterCollectionMessageBody{
            Updates: &messagespb.AlterCollectionMessageUpdates{
                Schema: schema,
            },
        }).
        MustBuildBroadcast()

    if _, err := broadcaster.Broadcast(ctx, msg); err != nil {
        return err
    }

    return nil
}
```

**关键特性**:
- **Schema 版本管理**：每次添加字段，Schema 版本号递增
- **字段 ID 分配**：自动分配唯一的 field ID
- **字段验证**：添加前验证字段 Schema 的合法性
- **幂等性检查**：检查字段是否已存在
- **消息广播**：通过 Streaming Node 广播 AlterCollection 消息

---

### 2. Schema 版本管理

**版本递增机制**:

```
初始 Schema (Version 1):
- id: INT64 (primary key)
- vector: FLOAT_VECTOR
- title: VARCHAR

添加字段后 (Version 2):
- id: INT64 (primary key)
- vector: FLOAT_VECTOR
- title: VARCHAR
- category: VARCHAR (nullable)  # 新增字段

再次添加字段后 (Version 3):
- id: INT64 (primary key)
- vector: FLOAT_VECTOR
- title: VARCHAR
- category: VARCHAR (nullable)
- created_at: INT64 (nullable)  # 新增字段
```

**版本管理优势**:
- 支持多版本共存
- 向后兼容
- 便于回滚

## Nullable 字段与默认值

### 1. Nullable 字段

**定义**: nullable 字段允许值为 NULL。

**必要性**: AddCollectionField 添加的字段必须是 nullable，因为已存在的实体该字段值为 NULL。

**示例**:

```python
# ✅ 正确：nullable=True
client.add_collection_field(
    collection_name="documents",
    field_name="author",
    data_type=DataType.VARCHAR,
    max_length=256,
    nullable=True  # 必须为 True
)

# ❌ 错误：nullable=False（会报错）
client.add_collection_field(
    collection_name="documents",
    field_name="author",
    data_type=DataType.VARCHAR,
    max_length=256,
    nullable=False  # 错误：AddCollectionField 不支持非 nullable 字段
)
```

---

### 2. 默认值设置

**定义**: 为新添加的字段设置默认值，确保已存在的实体和新实体都使用该默认值。

**示例**:

```python
# 添加带默认值的字段
client.add_collection_field(
    collection_name="documents",
    field_name="status",
    data_type=DataType.VARCHAR,
    max_length=32,
    nullable=True,
    default_value="active"  # 默认值
)

# 已存在的实体：status = "active"（默认值）
# 新插入的实体（未提供 status）：status = "active"（默认值）
# 新插入的实体（提供 status）：status = 提供的值
```

**优势**:
- 数据一致性：确保所有实体的该字段都有值
- 向后兼容：已存在的实体自动更新为默认值
- 简化插入：新实体可以不提供该字段

## 生产环境最佳实践

### 实践1：使用维护窗口

**建议**:
- 在低流量时段添加字段
- 停止或暂停写入操作
- 备份重要数据

**示例**:

```python
import time

# 步骤1：停止写入
print("Stopping writes...")
stop_all_writes()

# 步骤2：添加字段
print("Adding field...")
client.add_collection_field(
    collection_name="documents",
    field_name="category",
    data_type=DataType.VARCHAR,
    max_length=128,
    nullable=True,
    default_value="Uncategorized"
)

# 步骤3：重新加载 Collection
print("Reloading collection...")
client.release_collection(collection_name="documents")
time.sleep(5)
client.load_collection(collection_name="documents")

# 步骤4：验证字段
print("Verifying field...")
schema = client.describe_collection(collection_name="documents")
assert "category" in [field.name for field in schema.fields]

# 步骤5：恢复写入
print("Resuming writes...")
resume_all_writes()
```

---

### 实践2：测试所有查询类型

**建议**:
- 添加字段后，测试所有查询类型
- 特别是 HybridSearch、Search、Query、Aggregation

**示例**:

```python

    collection_name="documents",

)
```

## 常见问题

### 问题1：能否添加向量字段？

**答案**: 不能。AddCollectionField 只能添加标量字段。

**解决方案**: 如果需要添加向量字段，必须重新创建 Collection。

---

### 问题2：能否删除字段？

**答案**: 不能。Milvus 不支持删除字段。

**解决方案**: 如果需要删除字段，必须重新创建 Collection。

---

### 问题3：能否修改字段类型？

**答案**: 不能。Milvus 不支持修改字段类型。

**解决方案**: 如果需要修改字段类型，必须重新创建 Collection。

---

### 问题4：nullable 字段的性能影响？

**答案**: nullable 字段的聚合性能比非 nullable 慢 4 倍（来源：Issue #47566）。

**解决方案**: 如果字段不需要 NULL 值，避免使用 nullable。但 AddCollectionField 必须使用 nullable。
