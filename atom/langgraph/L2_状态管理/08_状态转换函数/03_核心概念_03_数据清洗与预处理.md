## 双重类比：先建立直觉

### 前端类比：表单验证 + API 响应清洗

```
前端数据清洗                              LangGraph 节点数据清洗
─────────────                            ─────────────────────
用户提交表单                              上游节点返回状态
  ↓                                        ↓
表单验证（必填、格式、范围）               输入验证（字段存在、类型、范围）
  ↓                                        ↓
数据标准化（trim、toLowerCase）            数据标准化（strip、统一格式）
  ↓                                        ↓
API 响应清洗（提取需要的字段）             输出格式化（只返回变化的字段）
  ↓                                        ↓
渲染到页面                                传递给下游节点
```

前端开发者每天都在做数据清洗——`trim()` 去空格、`parseInt()` 转类型、`?.` 防空值。LangGraph 节点里做的事情完全一样，只是对象从"用户输入"变成了"上游状态"。

### 日常生活类比：食材清洗

做菜之前，你不会直接把菜市场买回来的菜扔进锅里。先检查新不新鲜（输入验证），洗掉泥土（去除噪声），切成统一大小（类型转换），摘掉黄叶烂根（过滤无效数据）。节点函数里的数据清洗就是"洗菜切菜"——在真正的业务逻辑（炒菜）之前，先把原料处理干净。

## 1. 为什么节点内需要数据清洗？

LangGraph 的节点不是在真空中运行的。它接收的状态数据来自多个不可控的来源：

| 数据来源 | 可能的问题 | 举例 |
|----------|-----------|------|
| **上游节点** | 返回不规范的数据结构 | 检索节点返回了空列表而非预期的文档列表 |
| **LLM 输出** | 包含意外格式、幻觉内容 | GPT 返回了 Markdown 格式而非纯文本 |
| **外部 API** | 返回的数据需要标准化 | 不同搜索引擎返回的结果格式不一致 |
| **用户输入** | 包含噪声、特殊字符 | 用户输入了带前后空格的查询 |

**不做清洗的后果：**

```python
# ❌ 不做清洗：下游节点可能崩溃
def generate_node(state: State) -> dict:
    docs = state["documents"]          # 可能是 None！
    context = "\n".join(               # None 没有 join 方法 → 崩溃
        [d["content"] for d in docs]   # docs 里的 dict 可能没有 "content" key → KeyError
    )
    return {"answer": llm(context)}
```

**做了清洗的效果：**

```python
# ✅ 做了清洗：下游节点可以安心工作
def clean_then_generate(state: State) -> dict:
    docs = state.get("documents") or []
    context = "\n".join(
        str(d.get("content", "")) for d in docs if d.get("content")
    )
    if not context:
        return {"answer": "没有找到相关文档，无法回答。"}
    return {"answer": llm(context)}
```

**核心原则：每个节点应该对自己的输入负责，不要假设上游数据是完美的。**

## 2. 输入验证模式

### 2.1 基础验证：检查字段存在性与类型

输入验证是数据清洗的第一道防线。在 LangGraph 中，虽然 TypedDict 提供了类型提示，但运行时并不会强制检查类型。你需要在节点内部自己验证。

```python
from typing import Annotated
from typing_extensions import TypedDict


class State(TypedDict):
    query: str
    documents: list[dict]
    temperature: float


def validate_and_clean(state: State) -> dict:
    """输入验证与清洗节点——管道中的第一个节点"""
    updates = {}

    # ===== 1. 字符串清洗 =====
    # 检查 query 是否存在且非空
    query = state.get("query", "")
    if not query or not query.strip():
        updates["query"] = "default query"
    else:
        # 去除首尾空格，统一小写
        updates["query"] = query.strip().lower()

    # ===== 2. 列表清洗 —— 过滤无效项 =====
    docs = state.get("documents", [])
    # 只保留有实际内容的文档（content 存在且长度 > 10）
    valid_docs = [
        d for d in docs
        if isinstance(d, dict) and d.get("content") and len(d["content"]) > 10
    ]
    if valid_docs != docs:
        updates["documents"] = valid_docs

    # ===== 3. 数值范围校验 =====
    temp = state.get("temperature", 0.7)
    # 确保 temperature 在合法范围 [0.0, 2.0]
    updates["temperature"] = max(0.0, min(2.0, float(temp)))

    return updates
```

### 2.2 验证策略对比

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| **宽松验证** | 尽量修复，给默认值 | 用户输入、非关键字段 |
| **严格验证** | 不合法就报错/中断 | 关键业务字段、安全相关 |
| **静默跳过** | 过滤掉无效项，继续处理 | 批量数据、列表字段 |

```python
# 严格验证示例：关键字段不合法时写入错误状态
def strict_validate(state: State) -> dict:
    query = state.get("query", "")
    if not query or not query.strip():
        return {
            "error": "query 字段不能为空",
            "should_continue": False,
        }
    return {"query": query.strip(), "should_continue": True}
```

## 3. 类型转换与标准化

### 3.1 为什么需要标准化？

不同来源的数据格式往往不一致。比如检索节点返回的文档可能来自不同的数据源——有的有 `score` 字段，有的没有；有的 `content` 是字符串，有的是列表。标准化就是把这些"方言"统一成"普通话"。

```python
def normalize_node(state: State) -> dict:
    """类型转换与标准化节点"""
    normalized = []
    for doc in state.get("documents", []):
        normalized.append({
            # 确保 content 是字符串
            "content": str(doc.get("content", "")),
            # 确保 metadata 是字典
            "metadata": dict(doc.get("metadata", {})),
            # 确保 score 是浮点数
            "score": float(doc.get("score", 0.0)),
            # 确保 source 是字符串，缺失时给默认值
            "source": str(doc.get("source", "unknown")),
        })
    return {"documents": normalized}
```

### 3.2 常见类型转换场景

```python
def type_conversion_examples(state: State) -> dict:
    """展示常见的类型转换模式"""

    # 1. 字符串 → 列表（LLM 有时返回逗号分隔的字符串而非列表）
    tags = state.get("tags", "")
    if isinstance(tags, str):
        tag_list = [t.strip() for t in tags.split(",") if t.strip()]
    else:
        tag_list = list(tags)

    # 2. 任意类型 → 布尔值（处理 LLM 返回的各种"是/否"表达）
    is_relevant = state.get("is_relevant", None)
    if isinstance(is_relevant, str):
        is_relevant = is_relevant.lower().strip() in ("true", "yes", "是", "1")
    else:
        is_relevant = bool(is_relevant)

    return {"tags": tag_list, "is_relevant": is_relevant}
```

## 4. 空值与缺失值处理

### 4.1 空值的三种形态

在 LangGraph 状态中，"空"有多种表现形式：

```python
from typing import Optional
from typing_extensions import TypedDict


class RAGState(TypedDict, total=False):
    query: str                          # 可能根本不存在这个 key
    messages: Optional[list]            # 可能是 None
    documents: list[dict]               # 可能是空列表 []
```

| 空值形态 | 表现 | 检测方式 |
|----------|------|----------|
| **key 不存在** | `state["query"]` → KeyError | `state.get("query")` |
| **值为 None** | `state["messages"]` → None | `state.get("messages") is None` |
| **值为空容器** | `state["documents"]` → `[]` | `not state.get("documents")` |

### 4.2 防御性编程模式

```python
def safe_process(state: RAGState) -> dict:
    """安全处理，防御所有空值形态"""

    # 模式1：get + or 组合——同时处理 key 不存在和 None
    messages = state.get("messages") or []
    query = state.get("query") or ""

    # 模式2：空值检查后再处理
    if not messages:
        return {"error": "No messages to process"}

    # 模式3：安全访问对象属性（LangChain Message 对象）
    last_message = messages[-1]
    content = getattr(last_message, "content", "") or ""

    # 模式4：安全的链式访问
    metadata = state.get("metadata") or {}
    author = (metadata.get("author") or {}).get("name", "unknown")

    return {
        "processed_query": content.strip(),
        "author": author,
    }
```

### 4.3 `get + or` 的陷阱

```python
# ⚠️ 注意：0 和 False 也会被 or 替换掉！
temperature = state.get("temperature") or 0.7
# 如果 temperature 是 0.0，上面会变成 0.7（不是你想要的）

# ✅ 正确做法：用 is None 判断
temperature = state.get("temperature")
if temperature is None:
    temperature = 0.7
```

这是一个常见的 Python 陷阱。`or` 会把所有 falsy 值（`0`、`False`、`""`、`[]`）都当作"空"来处理。如果你的字段可能合法地为 `0` 或 `False`，要用 `is None` 判断。

## 5. 数据去重与过滤

### 5.1 去重节点

在 RAG 场景中，多路检索可能返回重复的文档。去重是常见的清洗操作。

```python
def deduplicate_node(state: State) -> dict:
    """去重节点：基于内容前缀去重"""
    seen = set()
    unique_docs = []

    for doc in state.get("documents", []):
        content = doc.get("content", "")
        # 用前 100 字符作为去重 key（避免长文本哈希开销）
        key = content[:100]
        if key and key not in seen:
            seen.add(key)
            unique_docs.append(doc)

    return {"documents": unique_docs}
```

### 5.2 质量过滤

```python
def quality_filter_node(state: State) -> dict:
    """质量过滤：去除低质量检索结果"""
    docs = state.get("documents", [])

    filtered = [
        doc for doc in docs
        if (
            doc.get("score", 0) >= 0.5          # 相似度阈值
            and len(doc.get("content", "")) > 50  # 最小内容长度
            and doc.get("source") != "deprecated"  # 排除废弃来源
        )
    ]

    # 按分数降序排列
    filtered.sort(key=lambda d: d.get("score", 0), reverse=True)

    return {"documents": filtered}
```

### 5.3 去重策略对比

| 策略 | 去重 key | 优点 | 缺点 |
|------|----------|------|------|
| **内容前缀** | `content[:100]` | 简单快速 | 可能误判相似但不同的文档 |
| **完整哈希** | `hash(content)` | 精确 | 长文本计算开销大 |
| **ID 去重** | `doc["id"]` | 最快 | 依赖文档有唯一 ID |
| **语义去重** | embedding 相似度 | 能识别语义重复 | 需要额外计算 |

## 6. 错误处理与降级策略

### 6.1 核心原则：节点不应该崩溃

一个节点崩溃会导致整个图执行失败。数据清洗节点应该尽可能"吞掉"异常，返回安全的降级结果。

```python
def resilient_clean_node(state: State) -> dict:
    """带错误处理的清洗节点"""
    errors = []

    # ===== 1. 安全的类型转换 =====
    try:
        temperature = float(state.get("temperature", 0.7))
        temperature = max(0.0, min(2.0, temperature))
    except (TypeError, ValueError) as e:
        errors.append(f"temperature 转换失败: {e}")
        temperature = 0.7  # 降级到默认值

    # ===== 2. 安全的文档处理 =====
    try:
        docs = state.get("documents", [])
        if not isinstance(docs, list):
            raise TypeError(f"documents 应为 list，实际为 {type(docs)}")
        clean_docs = [
            {
                "content": str(d.get("content", "")),
                "score": float(d.get("score", 0.0)),
            }
            for d in docs
            if isinstance(d, dict)
        ]
    except Exception as e:
        errors.append(f"documents 处理失败: {e}")
        clean_docs = []  # 降级到空列表

    # ===== 3. 记录错误到状态 =====
    result = {
        "temperature": temperature,
        "documents": clean_docs,
    }
    if errors:
        result["clean_errors"] = errors

    return result
```

### 6.2 降级策略对照表

| 场景 | 降级策略 | 类比 |
|------|----------|------|
| 类型转换失败 | 使用默认值 | 收银机扫码失败，手动输入价格 |
| 列表为空 | 返回空结果 + 提示信息 | 图书馆没找到书，告诉读者"暂无结果" |
| 外部 API 超时 | 使用缓存数据 | 外卖送不到，吃冰箱里的存货 |
| LLM 输出格式错误 | 重试或返回原始文本 | 翻译软件翻错了，直接看原文 |

## 7. 在 RAG 中的典型应用

### 7.1 查询清洗节点

```python
import re


def clean_query_node(state: State) -> dict:
    """RAG 查询清洗：标准化用户输入"""
    query = state.get("query", "")

    # 1. 去除首尾空格
    query = query.strip()

    # 2. 去除特殊字符（保留中文、英文、数字、空格）
    query = re.sub(r"[^\w\s\u4e00-\u9fff]", " ", query)

    # 3. 合并多余空格
    query = re.sub(r"\s+", " ", query).strip()

    # 4. 长度截断（避免超长查询）
    if len(query) > 500:
        query = query[:500]

    # 5. 空查询降级
    if not query:
        query = "请提供您的问题"

    return {"query": query}
```

### 7.2 检索结果清洗节点

```python
def clean_retrieval_node(state: State) -> dict:
    """RAG 检索结果清洗：标准化 + 去重 + 过滤"""
    raw_docs = state.get("documents", [])

    # 第一步：标准化格式
    normalized = []
    for doc in raw_docs:
        if not isinstance(doc, dict):
            continue
        normalized.append({
            "content": str(doc.get("content", "")).strip(),
            "source": str(doc.get("source", "unknown")),
            "score": float(doc.get("score", 0.0)),
        })

    # 第二步：去重
    seen = set()
    unique = []
    for doc in normalized:
        key = doc["content"][:100]
        if key not in seen and key:
            seen.add(key)
            unique.append(doc)

    # 第三步：质量过滤
    filtered = [d for d in unique if d["score"] >= 0.3 and len(d["content"]) > 20]

    # 第四步：按分数排序，取 Top-K
    filtered.sort(key=lambda d: d["score"], reverse=True)
    top_k = filtered[:5]

    return {"documents": top_k}
```

### 7.3 LLM 输出清洗节点

```python
import json


def clean_llm_output_node(state: State) -> dict:
    """清洗 LLM 输出：提取结构化信息"""
    raw_output = state.get("llm_response", "")

    # 尝试从 LLM 输出中提取 JSON（LLM 常在 JSON 前后加 ```json ... ```）
    structured = None
    try:
        json_match = re.search(r"```json\s*(.*?)\s*```", raw_output, re.DOTALL)
        if json_match:
            structured = json.loads(json_match.group(1))
        else:
            structured = json.loads(raw_output)
    except (json.JSONDecodeError, AttributeError):
        structured = {"answer": raw_output.strip(), "confidence": 0.0}

    answer = str(structured.get("answer", raw_output))
    confidence = max(0.0, min(1.0, float(structured.get("confidence", 0.5))))

    return {"answer": answer, "confidence": confidence}
```

## 8. 最佳实践速查表

> 把上面所有模式浓缩成一张表，方便日常查阅。

| 实践 | 做法 | 反模式 |
|------|------|--------|
| 字段访问 | `state.get("key", default)` | `state["key"]`（可能 KeyError） |
| 空值处理 | `value or default`（注意 0/False 陷阱） | 不检查直接使用 |
| 类型转换 | `try/except` 包裹 + 默认值 | 裸调用 `float(x)` |
| 列表清洗 | 列表推导式过滤 | 原地修改输入状态 |
| 错误记录 | 写入状态的 `errors` 字段 | `print()` 或静默忽略 |
| 去重 | 基于内容前缀的 set 去重 | 嵌套循环逐一比较 |
| 降级 | 返回安全默认值 + 错误信息 | 抛出异常让图崩溃 |

---

## 9. 关键要点总结

1. **每个节点对自己的输入负责**：不要假设上游数据是完美的
2. **用 `state.get()` 而非 `state[]`**：防止 KeyError
3. **类型转换要包 try/except**：LLM 输出和外部 API 都不可信
4. **空值有三种形态**：key 不存在、值为 None、值为空容器，要分别处理
5. **降级优于崩溃**：清洗节点应该"吞掉"异常，返回安全的默认值
6. **错误信息写入状态**：方便下游节点或最终输出中展示问题
7. **清洗节点放在管道前端**：越早清洗，下游越安心
