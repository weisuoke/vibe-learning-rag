# 03_核心概念_4 - 编排者-工作者模式（Orchestrator-Worker）

> 由一个"指挥官"节点用 LLM 动态分解任务，分配给多个"工人"节点并行执行，最后汇总结果的高级工作流模式

---

## 概念定义

**编排者-工作者模式是由编排者（Orchestrator）用 LLM 动态分解任务，将子任务分配给工作者（Worker）并行执行，最后由合成者（Synthesizer）汇总结果的工作流模式。**

这是六大工作流模式中最接近"真实团队协作"的一个。就像项目经理拿到一个大需求后，先拆解成多个子任务，分配给不同的开发者并行开发，最后由技术负责人合并代码并做集成测试。

与简单并行模式的关键区别：**任务分解是动态的**。并行模式在编译时就确定了有几个分支，而编排者-工作者模式在运行时由 LLM 决定拆成几个子任务、每个子任务是什么。

---

## 模式原理

### 核心结构

```
START → Orchestrator → [Worker_1 | Worker_2 | ... | Worker_N] → Synthesizer → END
                            ↑ 数量和内容由 LLM 动态决定
```

整个流程分三个阶段：

1. **规划阶段**：Orchestrator 用 LLM 分析输入，生成子任务列表
2. **执行阶段**：多个 Worker 并行执行各自的子任务
3. **合成阶段**：Synthesizer 汇总所有 Worker 的结果，生成最终输出

### 三个核心角色

| 角色 | 职责 | 类比 |
|------|------|------|
| 编排者（Orchestrator） | 用 LLM 分析任务，拆解为子任务列表 | 项目经理拆需求 |
| 工作者（Worker） | 执行具体的子任务，返回结果 | 开发者写代码 |
| 合成者（Synthesizer） | 汇总所有工作者的结果，生成最终输出 | 技术负责人合并代码 |

### 与简单并行模式的区别

| 特征 | 简单并行模式 | 编排者-工作者模式 |
|------|-------------|-----------------|
| 任务数量 | 编译时确定（固定 N 个分支） | 运行时动态决定 |
| 任务内容 | 预定义的固定节点 | LLM 根据输入动态生成 |
| 分配方式 | 硬编码的 `add_edge` | LLM 驱动的任务分解 |
| 灵活性 | 低（改分支要改代码） | 高（同一图处理不同任务） |
| 适用场景 | 已知的固定并行任务 | 未知的、需要智能拆解的任务 |

## LangGraph 两种实现方式

### 方式 1：Functional API 实现（推荐）

**最直观的方式，用 `@task` 和 `@entrypoint` 实现编排者-工作者模式。**

```python
"""
编排者-工作者模式 - Functional API 实现
场景：研究报告生成系统
  - 编排者：分析主题，规划报告章节
  - 工作者：并行撰写每个章节
  - 合成者：汇总章节，生成完整报告

图拓扑：
    START → orchestrator → [worker_1 | worker_2 | ... | worker_N] → synthesizer → END
"""
from pydantic import BaseModel, Field
from langgraph.func import entrypoint, task


# ========== 1. 定义数据结构 ==========

class Section(BaseModel):
    """报告章节的结构化定义"""
    title: str = Field(description="章节标题")
    description: str = Field(description="章节应该覆盖的内容要点")


class ReportPlan(BaseModel):
    """报告规划：包含多个章节"""
    sections: list[Section] = Field(description="报告的章节列表")


class SectionResult(BaseModel):
    """章节撰写结果"""
    title: str
    content: str


# ========== 2. 定义三个核心角色 ==========

@task
def orchestrator(topic: str) -> list[Section]:
    """
    编排者：用 LLM 分析主题，动态规划报告章节。
    关键点：章节数量和内容完全由 LLM 根据输入决定。
    """
    # 实际项目中用 LLM + structured output：
    # planner = llm.with_structured_output(ReportPlan)
    # plan = planner.invoke(f"为主题「{topic}」规划一份研究报告的章节")
    # return plan.sections

    # 模拟 LLM 的动态规划结果
    print(f"[编排者] 分析主题: {topic}")
    sections = [
        Section(title="概述与背景", description=f"{topic}的发展历程和核心定义"),
        Section(title="核心技术原理", description=f"{topic}的关键技术和实现机制"),
        Section(title="实践应用案例", description=f"{topic}在实际项目中的应用"),
        Section(title="未来趋势展望", description=f"{topic}的发展方向和挑战"),
    ]
    print(f"[编排者] 规划了 {len(sections)} 个章节: {[s.title for s in sections]}")
    return sections


@task
def worker(section: Section) -> SectionResult:
    """
    工作者：撰写单个章节。
    每个 worker 独立执行，互不依赖，天然支持并行。
    """
    # 实际项目中用 LLM 生成章节内容：
    # content = llm.invoke(f"撰写章节「{section.title}」，要点：{section.description}")

    # 模拟 LLM 撰写
    content = f"## {section.title}\n\n{section.description}的详细分析内容..."
    print(f"[工作者] 完成章节: {section.title}")
    return SectionResult(title=section.title, content=content)


@task
def synthesizer(topic: str, results: list[SectionResult]) -> str:
    """
    合成者：汇总所有章节，生成完整报告。
    可以用 LLM 做润色、添加引言和结论。
    """
    # 实际项目中可以用 LLM 做最终润色：
    # report = llm.invoke(f"将以下章节整合为完整报告：{results}")

    # 模拟合成
    report_parts = [f"# 研究报告：{topic}\n"]
    for r in results:
        report_parts.append(r.content)
    report = "\n\n".join(report_parts)
    print(f"[合成者] 生成完整报告，共 {len(results)} 个章节")
    return report


# ========== 3. 编排入口 ==========

@entrypoint()
def orchestrator_worker(topic: str) -> str:
    """
    编排者-工作者模式入口：
    1. 编排者规划章节（动态数量）
    2. 工作者并行撰写各章节
    3. 合成者汇总为完整报告
    """
    # 第一步：编排者规划
    sections = orchestrator(topic).result()

    # 第二步：工作者并行执行
    # 关键：先启动所有 task，再统一收集结果
    futures = [worker(section) for section in sections]
    results = [f.result() for f in futures]

    # 第三步：合成者汇总
    report = synthesizer(topic, results).result()
    return report


# ========== 4. 执行 ==========
result = orchestrator_worker.invoke("RAG 检索增强生成技术")
print(f"\n{'='*50}")
print(result)
# [编排者] 分析主题: RAG 检索增强生成技术
# [编排者] 规划了 4 个章节: ['概述与背景', '核心技术原理', '实践应用案例', '未来趋势展望']
# [工作者] 完成章节: 概述与背景
# [工作者] 完成章节: 核心技术原理
# [工作者] 完成章节: 实践应用案例
# [工作者] 完成章节: 未来趋势展望
# [合成者] 生成完整报告，共 4 个章节
```

#### Functional API 的并行关键

```python
# 关键技巧：先启动所有 task，再统一 .result()
futures = [worker(section) for section in sections]   # 全部启动（不阻塞）
results = [f.result() for f in futures]               # 统一等待结果

# 错误写法：逐个等待，变成串行
results = [worker(section).result() for section in sections]  # 每次都阻塞等待！
```

`worker(section)` 返回的是一个 Future 对象，调用 `.result()` 才会阻塞等待。先批量启动再统一收集，LangGraph 会自动并行调度。

### 方式 2：Graph API + Send 实现

**使用 StateGraph + Send API 实现动态工作者派发。适合需要更精细控制图结构的场景。**

```python
"""
编排者-工作者模式 - Graph API + Send 实现
场景：代码审查系统
  - 编排者：分析代码仓库结构，识别需要审查的模块
  - 工作者：并行审查每个模块
  - 合成者：汇总审查意见，生成审查报告

关键：使用 Send API 在运行时动态创建工作者实例
"""
import operator
from typing_extensions import TypedDict
from pydantic import BaseModel
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send


# ========== 1. 定义 State ==========

class ReviewTask(TypedDict):
    """单个审查任务的输入（Send 传递给 worker 的数据）"""
    module_name: str
    review_focus: str


class ReviewState(TypedDict):
    """主图的 State"""
    repo_url: str                          # 输入：仓库地址
    modules: list[dict]                    # 编排者输出：模块列表
    review_results: list[str]              # 工作者输出：审查结果（用 reducer 聚合）
    final_report: str                      # 合成者输出：最终报告


# 自定义 State，review_results 使用 add reducer 聚合并行结果
class ReviewStateWithReducer(TypedDict):
    repo_url: str
    modules: list[dict]
    review_results: list[str]  # 需要在 StateGraph 中配置 reducer
    final_report: str


# ========== 2. 定义节点函数 ==========

def orchestrator_node(state: ReviewState) -> dict:
    """编排者：分析仓库，识别需要审查的模块"""
    repo = state["repo_url"]
    # 实际项目中用 LLM 分析代码结构
    modules = [
        {"module_name": "auth", "review_focus": "安全性和权限控制"},
        {"module_name": "api", "review_focus": "接口设计和错误处理"},
        {"module_name": "database", "review_focus": "查询性能和数据一致性"},
    ]
    print(f"[编排者] 分析仓库 {repo}，识别 {len(modules)} 个模块")
    return {"modules": modules}


def worker_node(state: ReviewTask) -> dict:
    """工作者：审查单个模块"""
    name = state["module_name"]
    focus = state["review_focus"]
    # 实际项目中用 LLM 做代码审查
    result = f"[{name}] 审查完成 - 关注点: {focus}，发现 2 个问题"
    print(f"[工作者] {result}")
    return {"review_results": [result]}


def synthesizer_node(state: ReviewState) -> dict:
    """合成者：汇总审查结果"""
    results = state["review_results"]
    report = f"代码审查报告 ({len(results)} 个模块):\n" + "\n".join(results)
    print(f"[合成者] 生成审查报告")
    return {"final_report": report}


# ========== 3. 动态派发函数（核心） ==========

def dispatch_workers(state: ReviewState) -> list[Send]:
    """
    根据编排者的输出，动态创建 Send 指令。
    每个 Send 会创建一个 worker_node 实例。
    """
    return [
        Send("worker", {"module_name": m["module_name"], "review_focus": m["review_focus"]})
        for m in state["modules"]
    ]


# ========== 4. 构建图 ==========

# 注意：review_results 需要 reducer 来聚合多个 worker 的输出
from typing import Annotated

class AggregatedReviewState(TypedDict):
    repo_url: str
    modules: list[dict]
    review_results: Annotated[list[str], operator.add]  # add reducer 聚合
    final_report: str


builder = StateGraph(AggregatedReviewState)

# 添加节点
builder.add_node("orchestrator", orchestrator_node)
builder.add_node("worker", worker_node)
builder.add_node("synthesizer", synthesizer_node)

# 添加边
builder.add_edge(START, "orchestrator")
# 核心：用条件边 + Send 实现动态派发
builder.add_conditional_edges("orchestrator", dispatch_workers, ["worker"])
builder.add_edge("worker", "synthesizer")
builder.add_edge("synthesizer", END)

# 编译并执行
graph = builder.compile()
result = graph.invoke({
    "repo_url": "https://github.com/example/rag-project",
    "modules": [],
    "review_results": [],
    "final_report": ""
})
print(f"\n{result['final_report']}")
# [编排者] 分析仓库 ..., 识别 3 个模块
# [工作者] [auth] 审查完成 - 关注点: 安全性和权限控制，发现 2 个问题
# [工作者] [api] 审查完成 - 关注点: 接口设计和错误处理，发现 2 个问题
# [工作者] [database] 审查完成 - 关注点: 查询性能和数据一致性，发现 2 个问题
# [合成者] 生成审查报告
```

#### Send API 的工作原理

```python
# Send 的签名
Send(node_name: str, arg: dict)

# dispatch_workers 返回的 Send 列表：
[
    Send("worker", {"module_name": "auth", "review_focus": "安全性"}),
    Send("worker", {"module_name": "api", "review_focus": "接口设计"}),
    Send("worker", {"module_name": "database", "review_focus": "查询性能"}),
]
# 每个 Send 创建一个 worker 实例，三个 worker 并行执行
# 所有 worker 的输出通过 Annotated[list[str], operator.add] 自动聚合
```

Send 在底层创建 PUSH 类型的任务，通过 Topic 通道路由到目标节点。多个 Send 指向同一个节点时，该节点会被并行实例化多次。

## 两种实现方式对比

| 维度 | Functional API | Graph API + Send |
|------|---------------|-----------------|
| 代码量 | 少（Pythonic 写法） | 多（显式定义图结构） |
| 并行控制 | 自动（批量启动 task） | 显式（Send API） |
| State 管理 | 函数参数传递 | TypedDict + Reducer |
| 可视化 | 支持 | 支持 |
| 灵活性 | 高（普通 Python 逻辑） | 最高（完全控制图拓扑） |
| 适用场景 | 大多数编排者-工作者场景 | 需要与其他图模式组合 |

**选择建议**：优先用 Functional API，除非需要与 StateGraph 的其他特性（如 checkpointing、human-in-the-loop）深度集成。

## 适用场景

| 场景 | 编排者做什么 | 工作者做什么 | 合成者做什么 |
|------|------------|------------|------------|
| 报告生成 | 规划章节结构 | 并行撰写各章节 | 整合为完整报告 |
| 代码审查 | 分析代码结构 | 并行审查各模块 | 汇总审查意见 |
| 多文档问答 | 分解问题为子问题 | 并行检索各子问题 | 综合子答案 |
| 竞品分析 | 确定分析维度 | 并行分析各维度 | 生成对比报告 |
| 翻译校对 | 拆分文档段落 | 并行翻译各段落 | 合并并统一术语 |

---

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| 任务分解动态灵活，同一图处理不同输入 | 编排者的 LLM 调用增加延迟和成本 |
| 工作者并行执行，总延迟 ≈ 最慢的子任务 | 任务分解质量依赖 LLM 能力 |
| 天然支持不同数量的子任务 | 工作者之间无法通信（独立执行） |
| 合成阶段可以做质量把控 | 子任务间有依赖时不适用 |

### 延迟分析

```
传统串行：T = T_编排 + T_任务1 + T_任务2 + T_任务3 + T_合成
编排者-工作者：T = T_编排 + max(T_任务1, T_任务2, T_任务3) + T_合成

例：编排 2s，三个任务分别 5s/3s/4s，合成 2s
串行：2 + 5 + 3 + 4 + 2 = 16s
并行：2 + max(5,3,4) + 2 = 9s  （节省 44%）
```

## 在 RAG 中的典型应用

### 复杂问题分解 → 并行子问题检索 → 综合回答

当用户提出一个复杂问题时，编排者将其分解为多个子问题，工作者并行检索每个子问题的答案，合成者综合所有子答案生成最终回答。

```python
"""
RAG 多子问题并行检索 - 编排者-工作者模式
场景：用户问 "对比 LangChain 和 LlamaIndex 在 RAG 场景下的优劣"
  - 编排者：拆解为多个子问题
  - 工作者：并行检索每个子问题
  - 合成者：综合回答
"""
from langgraph.func import entrypoint, task


@task
def decompose_question(question: str) -> list[str]:
    """编排者：将复杂问题分解为子问题"""
    # 实际项目中用 LLM：
    # sub_questions = llm.invoke(f"将以下问题分解为独立的子问题：{question}")
    sub_questions = [
        "LangChain 的 RAG 实现方式和核心特点是什么？",
        "LlamaIndex 的 RAG 实现方式和核心特点是什么？",
        "LangChain 和 LlamaIndex 在性能上有什么差异？",
        "LangChain 和 LlamaIndex 的社区生态和文档质量如何？",
    ]
    print(f"[编排者] 分解为 {len(sub_questions)} 个子问题")
    return sub_questions


@task
def retrieve_and_answer(sub_question: str) -> dict:
    """工作者：检索并回答单个子问题"""
    # 实际项目中：
    # docs = vector_store.similarity_search(sub_question, k=3)
    # answer = llm.invoke(f"基于以下文档回答：{docs}\n问题：{sub_question}")
    answer = f"关于「{sub_question[:15]}...」的检索结果和回答"
    print(f"[工作者] 完成子问题: {sub_question[:20]}...")
    return {"question": sub_question, "answer": answer}


@task
def synthesize_answer(original_question: str, sub_answers: list[dict]) -> str:
    """合成者：综合所有子答案"""
    # 实际项目中用 LLM 综合：
    # final = llm.invoke(f"综合以下子答案回答原始问题：{sub_answers}")
    parts = [f"- {a['answer']}" for a in sub_answers]
    final = f"综合回答「{original_question}」：\n" + "\n".join(parts)
    print(f"[合成者] 综合 {len(sub_answers)} 个子答案")
    return final


@entrypoint()
def rag_orchestrator_worker(question: str) -> str:
    """RAG 编排者-工作者：复杂问题 → 子问题并行检索 → 综合回答"""
    # 1. 分解问题
    sub_questions = decompose_question(question).result()

    # 2. 并行检索（先启动所有 task，再统一收集）
    futures = [retrieve_and_answer(q) for q in sub_questions]
    sub_answers = [f.result() for f in futures]

    # 3. 综合回答
    final = synthesize_answer(question, sub_answers).result()
    return final


# 执行
answer = rag_orchestrator_worker.invoke("对比 LangChain 和 LlamaIndex 在 RAG 场景下的优劣")
print(f"\n{answer}")
```

这个模式特别适合 RAG 中的复杂问题。单次检索很难覆盖多维度的问题，拆解后并行检索能显著提升回答的全面性。

## 最佳实践

### 1. 用 Structured Output 约束编排者输出

```python
from pydantic import BaseModel, Field

class TaskPlan(BaseModel):
    """编排者的输出必须是结构化的"""
    tasks: list[str] = Field(description="子任务列表")
    priority: list[int] = Field(description="优先级排序")

# 用 with_structured_output 确保 LLM 输出格式正确
planner = llm.with_structured_output(TaskPlan)
plan = planner.invoke("分析这个需求并拆解子任务...")
```

不要让编排者返回自由文本，用 Pydantic 模型约束输出格式，避免下游解析失败。

### 2. 工作者要完全独立

```python
# 推荐：每个 worker 只依赖自己的输入
@task
def worker(section: Section) -> SectionResult:
    # 只用 section 的数据，不依赖其他 worker 的结果
    return SectionResult(title=section.title, content="...")

# 不推荐：worker 之间有依赖
@task
def worker_b(section: Section, worker_a_result: str) -> SectionResult:
    # 依赖 worker_a 的结果，破坏了并行性
    ...
```

### 3. 合成者要做质量把控

```python
@task
def synthesizer(topic: str, results: list[SectionResult]) -> str:
    # 不只是简单拼接，要做质量检查
    valid_results = [r for r in results if len(r.content) > 50]
    if len(valid_results) < len(results):
        print(f"[合成者] 警告：{len(results) - len(valid_results)} 个章节质量不达标")
    # 用 LLM 做最终润色和一致性检查
    ...
```

### 4. 控制子任务数量

编排者可能生成过多子任务，导致并行开销过大。建议在 Pydantic 模型中限制：

## 常见误区

### 误区 1："编排者-工作者就是 Map-Reduce"

两者的关键区别在于任务来源：
- **编排者-工作者**：任务由 LLM 动态生成（智能分解）
- **Map-Reduce**：任务来自已有数据的映射（机械拆分）

编排者-工作者的"编排"是智能的，Map-Reduce 的"映射"是机械的。

### 误区 2："工作者越多越快"

并行度受限于 LLM API 的并发限制和 rate limit。10 个工作者同时调用 LLM，可能触发限流反而更慢。实际部署时要考虑 API 并发上限。

### 误区 3："编排者只需要调用一次"

复杂场景下，编排者可能需要多轮规划。比如第一轮粗分，根据工作者反馈再细分。这时可以在编排者-工作者外面套一个评估者-优化者循环。

## 与其他模式的关系

```
编排者-工作者 = 顺序链式 + 动态并行

与其他模式的组合：
  + 评估者-优化者 → 迭代式编排（编排 → 执行 → 评估 → 重新编排）
  + 路由模式     → 不同类型的子任务路由到不同的专业工作者
  + Map-Reduce   → 工作者内部可以用 Map-Reduce 处理批量数据
```
