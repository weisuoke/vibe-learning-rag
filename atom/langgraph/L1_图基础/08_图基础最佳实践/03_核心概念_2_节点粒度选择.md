# 核心概念 2：节点粒度选择

> 本文档是"08_图基础最佳实践"知识点的核心概念之一，详细讲解 LangGraph 中节点粒度选择的原则和实践。

---

## 什么是节点粒度？

**节点粒度**指的是单个节点承担的功能范围和复杂度。

在 LangGraph 中，节点是图的基本执行单元。节点粒度的选择直接影响：
- 图的可读性和可维护性
- 节点的可测试性和可复用性
- 图的执行性能
- 错误处理的精细度

---

## 单一职责原则

### 核心理念

**每个节点应该只做一件事，并把这件事做好。**

这是节点粒度选择的首要原则，源自软件工程的单一职责原则（Single Responsibility Principle）。

### 好的节点设计示例

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    query: str
    search_results: list
    validated_results: list
    formatted_output: str

# ✅ 好的设计：每个节点职责单一
def search_node(state: State) -> dict:
    """只负责搜索"""
    results = perform_search(state["query"])
    return {"search_results": results}

def validate_node(state: State) -> dict:
    """只负责验证"""
    validated = [r for r in state["search_results"] if is_valid(r)]
    return {"validated_results": validated}

def format_node(state: State) -> dict:
    """只负责格式化"""
    formatted = format_results(state["validated_results"])
    return {"formatted_output": formatted}

# 构建图
graph = StateGraph(State)
graph.add_node("search", search_node)
graph.add_node("validate", validate_node)
graph.add_node("format", format_node)
graph.add_edge(START, "search")
graph.add_edge("search", "validate")
graph.add_edge("validate", "format")
graph.add_edge("format", END)
```

### 不好的节点设计示例

```python
# ❌ 不好的设计：节点做太多事情
def search_validate_format_node(state: State) -> dict:
    """搜索 + 验证 + 格式化 + 日志 + 错误处理"""
    try:
        # 搜索
        results = perform_search(state["query"])
        log_search(results)

        # 验证
        validated = [r for r in results if is_valid(r)]
        log_validation(validated)

        # 格式化
        formatted = format_results(validated)
        log_formatting(formatted)

        # 保存到数据库
        save_to_db(formatted)

        return {
            "search_results": results,
            "validated_results": validated,
            "formatted_output": formatted
        }
    except Exception as e:
        log_error(e)
        return {"error": str(e)}
```

**问题**：
- 难以测试（需要模拟多个依赖）
- 难以复用（功能耦合在一起）
- 难以维护（修改一个功能可能影响其他功能）
- 难以调试（错误发生在哪个步骤不清楚）

## 何时拆分节点

### 拆分原则

当满足以下条件时，应该考虑拆分节点：

#### 1. 功能独立且可复用

```python
# ✅ 拆分：搜索功能可以在多个地方复用
def search_documents(state: State) -> dict:
    """通用的文档搜索节点"""
    results = vector_store.search(state["query"], top_k=10)
    return {"search_results": results}

def search_web(state: State) -> dict:
    """通用的网络搜索节点"""
    results = web_search_api.search(state["query"])
    return {"web_results": results}

# 这两个节点可以在不同的图中复用
```

#### 2. 需要单独测试

```python
# ✅ 拆分：每个节点可以独立测试
def extract_entities(state: State) -> dict:
    """提取实体"""
    entities = nlp_model.extract_entities(state["text"])
    return {"entities": entities}

def classify_intent(state: State) -> dict:
    """分类意图"""
    intent = classifier.predict(state["text"])
    return {"intent": intent}

# 测试时可以分别验证实体提取和意图分类的正确性
```

#### 3. 可能需要重试或错误处理

```python
from langgraph.types import RetryPolicy

# ✅ 拆分：API 调用需要重试策略
def call_llm(state: State) -> dict:
    """调用 LLM"""
    response = llm.invoke(state["prompt"])
    return {"llm_response": response}

def call_external_api(state: State) -> dict:
    """调用外部 API"""
    data = external_api.fetch(state["query"])
    return {"api_data": data}

# 为不同节点配置不同的重试策略
graph.add_node(
    "llm",
    call_llm,
    retry_policy=RetryPolicy(max_attempts=3)
)
graph.add_node(
    "api",
    call_external_api,
    retry_policy=RetryPolicy(max_attempts=5, initial_interval=1.0)
)
```

#### 4. 逻辑复杂度高

```python
# ✅ 拆分：复杂的数据处理逻辑
def preprocess_data(state: State) -> dict:
    """数据预处理"""
    cleaned = clean_text(state["raw_data"])
    return {"cleaned_data": cleaned}

def transform_data(state: State) -> dict:
    """数据转换"""
    transformed = apply_transformations(state["cleaned_data"])
    return {"transformed_data": transformed}

def aggregate_data(state: State) -> dict:
    """数据聚合"""
    aggregated = aggregate_results(state["transformed_data"])
    return {"final_data": aggregated}
```

## 何时合并节点

### 合并原则

当满足以下条件时，应该考虑合并节点：

#### 1. 紧密耦合的操作

```python
# ✅ 合并：紧密耦合的操作
def load_and_parse_document(state: State) -> dict:
    """加载并解析文档（紧密耦合）"""
    # 加载和解析必须一起进行
    with open(state["file_path"], "r") as f:
        content = f.read()
    parsed = parse_document(content)
    return {"document": parsed}

# ❌ 不必要的拆分
def load_document(state: State) -> dict:
    with open(state["file_path"], "r") as f:
        return {"raw_content": f.read()}

def parse_document_node(state: State) -> dict:
    parsed = parse_document(state["raw_content"])
    return {"document": parsed}
```

#### 2. 性能考虑

```python
# ✅ 合并：减少状态传递开销
def quick_validation(state: State) -> dict:
    """快速验证（合并简单操作）"""
    # 这些操作都很快，合并可以减少状态传递开销
    is_not_empty = len(state["text"]) > 0
    is_valid_length = len(state["text"]) < 10000
    has_required_fields = all(k in state for k in ["text", "user_id"])

    return {
        "is_valid": is_not_empty and is_valid_length and has_required_fields
    }

# ❌ 过度拆分导致性能问题
def check_not_empty(state: State) -> dict:
    return {"is_not_empty": len(state["text"]) > 0}

def check_length(state: State) -> dict:
    return {"is_valid_length": len(state["text"]) < 10000}

def check_fields(state: State) -> dict:
    return {"has_fields": all(k in state for k in ["text", "user_id"])}
```

#### 3. 简单的转换操作

```python
# ✅ 合并：简单的数据转换
def prepare_prompt(state: State) -> dict:
    """准备提示词（简单转换）"""
    prompt = f"User query: {state['query']}\nContext: {state['context']}"
    return {"prompt": prompt}

# ❌ 不必要的拆分
def format_query(state: State) -> dict:
    return {"formatted_query": f"User query: {state['query']}"}

def add_context(state: State) -> dict:
    return {"prompt": f"{state['formatted_query']}\nContext: {state['context']}"}
```

## 节点粒度权衡

### 可读性 vs 性能

**可读性优先原则**：在大多数情况下，应该优先考虑可读性和可维护性，而不是性能。

```python
# ✅ 推荐：可读性好
def extract_keywords(state: State) -> dict:
    """提取关键词"""
    keywords = keyword_extractor.extract(state["text"])
    return {"keywords": keywords}

def rank_keywords(state: State) -> dict:
    """关键词排序"""
    ranked = sorted(state["keywords"], key=lambda x: x.score, reverse=True)
    return {"ranked_keywords": ranked}

# ❌ 不推荐：为了性能牺牲可读性
def extract_and_rank_keywords(state: State) -> dict:
    """提取并排序关键词（性能优化）"""
    keywords = keyword_extractor.extract(state["text"])
    ranked = sorted(keywords, key=lambda x: x.score, reverse=True)
    return {"ranked_keywords": ranked}
```

**何时考虑性能优化**：
- 通过性能分析工具（profiling）确认存在瓶颈
- 状态传递开销确实很大（大对象、高频调用）
- 优化后仍然保持合理的可读性

### 模块化 vs 简洁性

**使用子图实现模块化**：

```python
from langgraph.graph import StateGraph

# ✅ 使用子图组织相关节点
def create_preprocessing_subgraph():
    """数据预处理子图"""
    subgraph = StateGraph(State)
    subgraph.add_node("clean", clean_data)
    subgraph.add_node("normalize", normalize_data)
    subgraph.add_node("validate", validate_data)
    subgraph.add_edge(START, "clean")
    subgraph.add_edge("clean", "normalize")
    subgraph.add_edge("normalize", "validate")
    subgraph.add_edge("validate", END)
    return subgraph.compile()

# 主图中使用子图
main_graph = StateGraph(State)
main_graph.add_node("preprocess", create_preprocessing_subgraph())
main_graph.add_node("analyze", analyze_data)
main_graph.add_edge(START, "preprocess")
main_graph.add_edge("preprocess", "analyze")
main_graph.add_edge("analyze", END)
```

## 节点命名规范

### 命名原则

**使用描述性的动词短语**：

```python
# ✅ 好的命名
def search_documents(state: State) -> dict: ...
def validate_results(state: State) -> dict: ...
def format_response(state: State) -> dict: ...
def call_llm(state: State) -> dict: ...

# ❌ 不好的命名
def node1(state: State) -> dict: ...
def process(state: State) -> dict: ...
def do_stuff(state: State) -> dict: ...
def handler(state: State) -> dict: ...
```

### 命名模式

```python
# 1. 动词 + 名词
def extract_entities(state: State) -> dict: ...
def classify_intent(state: State) -> dict: ...
def generate_response(state: State) -> dict: ...

# 3. 特定领域术语
```

## 完整示例：节点粒度实践

```python
"""
节点粒度选择实战示例
演示：如何合理拆分和组织节点
"""

from typing_extensions import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy

# ===== 1. 状态定义 =====
class RAGState(TypedDict):
    query: str
    documents: Annotated[list, add]
    reranked_docs: list
    context: str
    response: str
    metadata: dict

# ===== 2. 节点定义（合理粒度）=====

def retrieve_documents(state: RAGState) -> dict:
    """
    检索文档节点
    职责：从向量数据库检索相关文档
    """
    print(f"检索文档: {state['query']}")
    # 模拟检索
    docs = [
        {"content": "文档1内容", "score": 0.9},
        {"content": "文档2内容", "score": 0.8},
        {"content": "文档3内容", "score": 0.7},
    ]
    return {"documents": docs}

def rerank_documents(state: RAGState) -> dict:
    """
    重排序节点
    职责：使用重排序模型优化文档顺序
    """
    print("重排序文档")
    # 模拟重排序
    reranked = sorted(
        state["documents"],
        key=lambda x: x["score"],
        reverse=True
    )[:2]
    return {"reranked_docs": reranked}

def build_context(state: RAGState) -> dict:
    """
    构建上下文节点
    职责：将文档组合成上下文字符串
    """
    print("构建上下文")
    context = "\n\n".join([
        f"文档 {i+1}: {doc['content']}"
        for i, doc in enumerate(state["reranked_docs"])
    ])
    return {"context": context}

def generate_response(state: RAGState) -> dict:
    """
    生成回复节点
    职责：调用 LLM 生成最终回复
    """
    print("生成回复")
    # 模拟 LLM 调用
    prompt = f"问题: {state['query']}\n\n上下文:\n{state['context']}"
    response = f"基于上下文的回复: {state['query']}"

    metadata = {
        "num_docs": len(state["reranked_docs"]),
        "context_length": len(state["context"])
    }

    return {
        "response": response,
        "metadata": metadata
    }

# ===== 3. 构建图 =====

# 创建图
graph = StateGraph(RAGState)

# 添加节点
graph.add_node("retrieve", retrieve_documents)
graph.add_node("rerank", rerank_documents)
graph.add_node("build_context", build_context)
graph.add_node(
    "generate",
    generate_response,
    retry_policy=RetryPolicy(max_attempts=3)  # LLM 调用需要重试
)

# 添加边
graph.add_edge(START, "retrieve")
graph.add_edge("retrieve", "rerank")
graph.add_edge("rerank", "build_context")
graph.add_edge("build_context", "generate")
graph.add_edge("generate", END)

# 编译图
app = graph.compile()

# ===== 4. 执行 =====

if __name__ == "__main__":
    result = app.invoke({
        "query": "什么是 LangGraph？",
        "documents": [],
        "reranked_docs": [],
        "context": "",
        "response": "",
        "metadata": {}
    })

    print("\n=== 执行结果 ===")
    print(f"查询: {result['query']}")
    print(f"检索文档数: {len(result['documents'])}")
    print(f"重排序后文档数: {len(result['reranked_docs'])}")
    print(f"上下文长度: {len(result['context'])}")
    print(f"回复: {result['response']}")
    print(f"元数据: {result['metadata']}")
```

**运行输出**：
```
检索文档: 什么是 LangGraph？
重排序文档
构建上下文
生成回复

=== 执行结果 ===
查询: 什么是 LangGraph？
检索文档数: 3
重排序后文档数: 2
上下文长度: 45
回复: 基于上下文的回复: 什么是 LangGraph？
元数据: {'num_docs': 2, 'context_length': 45}
```

## 节点粒度对比示例

### 过度拆分的问题

```python
# ❌ 过度拆分：导致图过于复杂
def load_file(state: State) -> dict:
    return {"raw_content": open(state["path"]).read()}

def decode_content(state: State) -> dict:
    return {"decoded": state["raw_content"].decode("utf-8")}

def strip_whitespace(state: State) -> dict:
    return {"stripped": state["decoded"].strip()}

def split_lines(state: State) -> dict:
    return {"lines": state["stripped"].split("\n")}

def filter_empty_lines(state: State) -> dict:
    return {"filtered": [l for l in state["lines"] if l]}

# 图变得非常复杂
graph.add_edge(START, "load")
graph.add_edge("load", "decode")
graph.add_edge("decode", "strip")
graph.add_edge("strip", "split")
graph.add_edge("split", "filter")
graph.add_edge("filter", END)
```

### 合理粒度的设计

```python
# ✅ 合理粒度：简洁清晰
def load_and_preprocess_file(state: State) -> dict:
    """加载并预处理文件"""
    with open(state["path"], "r", encoding="utf-8") as f:
        content = f.read()

    # 预处理
    lines = [
        line.strip()
        for line in content.split("\n")
        if line.strip()
    ]

    return {"lines": lines}

# 图简洁清晰
graph.add_edge(START, "load_and_preprocess")
graph.add_edge("load_and_preprocess", "process")
graph.add_edge("process", END)
```

## 最佳实践总结

### 节点粒度选择清单

- [ ] 每个节点是否只负责一个明确的功能？
- [ ] 节点函数是否简短易懂（< 50 行）？
- [ ] 节点是否可以独立测试？
- [ ] 节点是否可以在其他图中复用？
- [ ] 节点命名是否清晰描述其功能？
- [ ] 是否避免了过度拆分导致图过于复杂？
- [ ] 是否避免了过度合并导致节点职责不清？
- [ ] 是否为需要重试的节点配置了 RetryPolicy？
- [ ] 是否使用子图组织相关节点？
- [ ] 性能优化是否基于实际测量？

### 核心原则

1. **单一职责**：每个节点只做一件事
2. **可读性优先**：优先考虑可读性和可维护性
3. **合理拆分**：功能独立、可测试、可复用时拆分
4. **合理合并**：紧密耦合、简单操作时合并
5. **描述性命名**：使用清晰的动词短语命名
6. **使用子图**：组织相关节点，保持主图简洁
7. **性能优化**：基于实际测量，不过早优化
8. **错误处理**：为不同节点配置合适的重试策略

## 在 LangGraph 中的应用

### 与状态化工作流的关系

节点粒度选择直接影响状态化工作流的设计：

- **细粒度节点**：状态更新更频繁，Checkpoint 更多，便于断点续传
- **粗粒度节点**：状态更新较少，性能更好，但恢复粒度较粗

### 与人机循环的关系

```python
from langgraph.types import interrupt

def review_node(state: State) -> dict:
    """人工审核节点（细粒度）"""
    # 细粒度节点便于在特定步骤插入人工审核
    approved = interrupt("请审核结果")
    return {"approved": approved}
```

### 与并行执行的关系

```python
# 细粒度节点便于并行执行
graph.add_edge("start", "search_docs")
graph.add_edge("start", "search_web")  # 并行执行
graph.add_edge("search_docs", "merge")
graph.add_edge("search_web", "merge")
```

---

**记住**：节点粒度选择没有绝对的标准，需要根据具体场景权衡可读性、可维护性、性能和复杂度。优先考虑可读性和可维护性，在确认存在性能瓶颈时再进行优化。
