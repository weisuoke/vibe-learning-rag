## 基础概念题（初级）

### Q1: OpenClaw CLI 的 5 个核心命令是什么？各自的作用是什么？

**答案**：

```bash
1. openclaw gateway    # 启动和管理 Gateway 控制平面
2. openclaw message    # 发送消息到多个通道
3. openclaw agent      # 与 AI Agent 交互
4. openclaw channels   # 管理和监控通道状态
5. openclaw config     # 配置管理（读取/设置/删除）
```

**详细说明**：
- `gateway`: Gateway 是整个系统的控制平面，负责消息路由、Agent 集成、Session 管理
- `message`: 统一的消息发送接口，自动路由到所有配置的通道
- `agent`: 调用 AI Agent（基于 pi-agent-core），支持思考级别和自动发送
- `channels`: 查看通道列表、状态、日志，添加/删除通道
- `config`: 管理配置文件（`~/.openclaw/config/settings.json`）

**考察点**：
- 是否理解 OpenClaw 的核心功能
- 是否知道每个命令的基本用途

## 架构设计题（中级）

### Q4: OpenClaw Gateway 在架构中扮演什么角色？为什么需要 Gateway？

**答案**：

**角色**：Gateway 是完整的控制平面（Control Plane），不是简单的消息转发。

**核心功能**：
1. **消息路由引擎**：根据目标地址路由到对应通道
2. **Agent 系统集成**：集成 pi-agent-core，提供 AI 能力
3. **Session 管理**：维护对话历史（JSONL 存储）
4. **配置热重载**：监听配置文件变化，自动重载
5. **安全策略执行**：DM 配对审批、权限控制
6. **扩展系统加载**：加载 Extension、Plugin、Hook

**为什么需要 Gateway**：

**问题**：
- 每个通道（WhatsApp、Telegram、Slack）有不同的 API
- 直接调用需要处理各种认证、协议、数据格式
- 难以统一管理和监控

**解决方案**：
- Gateway 提供统一的 REST API
- CLI 只需要知道一个接口：`http://localhost:18789`
- Gateway 负责转换和路由

**类比**：
- Gateway = Nginx 反向代理 + API Gateway
- Channels = 微服务后端
- CLI = HTTP 客户端

**考察点**：
- 是否理解 Gateway 的完整职责
- 是否能类比到后端架构

### Q5: OpenClaw CLI 命令是同步还是异步的？为什么？

**答案**：

**大部分命令是异步的**。

**原因**：
1. Gateway 使用事件驱动架构
2. 消息发送是后台任务，不阻塞 CLI
3. CLI 只是提交任务，不等待完成

**示例**：
```bash
# 命令立即返回，消息在后台发送
openclaw message send --to +1234567890 --message "Hello"
# 输出: Message queued (ID: abc123)

# 实际发送可能需要几秒钟
# 可以通过日志查看发送状态
openclaw channels logs
```

**同步命令**（少数）：
```bash
openclaw config get model     # 直接读取配置文件
openclaw --version            # 显示版本
openclaw gateway status       # 查询 Gateway 状态
```

**考察点**：
- 是否理解异步模型
- 是否知道如何验证任务完成

**配置文件结构**：

### Q7: 如何实现多通道消息广播？

**答案**：

```bash
# 方式 1: 使用 @all 目标
openclaw message send --to @all --message "系统维护通知"

# 方式 2: 使用脚本批量发送
for channel in telegram slack discord; do
  openclaw message send --channel $channel --message "通知"

# 方式 3: 使用 TypeScript 脚本
```

**TypeScript 实现**：
```typescript
import { exec } from 'child_process';
import { promisify } from 'util';

const execAsync = promisify(exec);

async function broadcastMessage(message: string) {
  const { stdout } = await execAsync(
    `openclaw message send --to @all --message "${message}"`
  );
  console.log(stdout);
}

broadcastMessage('系统上线通知');
```

**考察点**：
- 是否知道 `@all` 目标

### Q8: 如何配置 Agent 的思考级别？不同级别有什么区别？

**答案**：

**思考级别**：
```bash
# 低思考级别（快速响应）
openclaw agent --message "总结任务" --thinking low

# 中等思考级别（默认）
openclaw agent --message "总结任务" --thinking medium

# 高思考级别（深度分析）
openclaw agent --message "总结任务" --thinking high
```

**区别**：

| 级别 | 响应时间 | Token 消耗 | 适用场景 |
|------|---------|-----------|---------|
| low | 1-2秒 | 少 | 简单问答、快速查询 |
| medium | 3-5秒 | 中等 | 一般任务、代码生成 |
| high | 5-10秒 | 多 | 复杂分析、深度思考 |

**实现原理**：
- 思考级别影响 LLM 的 `temperature` 和 `max_tokens`
- 高思考级别会使用更多的上下文

**考察点**：
- 是否理解思考级别的作用
- 是否能根据场景选择合适的级别

### Q9: 如何防止 Agent 响应自动发送到通道？

**答案**：

```bash
# 使用 --deliver=false 参数
openclaw agent --message "调试信息" --deliver=false
```

**使用场景**：
1. **本地调试**：不想打扰用户
2. **测试 Agent**：验证响应内容
3. **敏感信息**：避免泄露

**默认行为**：
- 默认 `--deliver=true`
- Agent 响应会自动发送到所有配置的通道

**TypeScript 实现**：
```typescript
async function debugAgent(message: string) {
  const { stdout } = await execAsync(
    `openclaw agent --message "${message}" --deliver=false`
  );
  console.log('Agent 响应（仅本地）:', stdout);
}
```

**考察点**：
- 是否知道 `--deliver` 参数
- 是否理解默认行为

## 安全与故障排查题（高级）

### Q10: OpenClaw 的 DM 安全策略是什么？如何配置？

**答案**：

**默认策略**：`pairing`（配对审批）

**三种策略**：
```bash
1. pairing（默认）：需要审批配对请求
2. accept：自动接受所有 DM
3. reject：拒绝所有 DM
```

**配置方式**：
```bash
# 查看当前策略
openclaw config get dmPolicy

# 修改为自动接受（不推荐）
openclaw config set dmPolicy accept

# 修改为拒绝所有
openclaw config set dmPolicy reject
```

**配对审批流程**：
```bash
# 1. 用户发送消息，触发配对请求
# 2. 查看待审批请求
openclaw pairing list

# 3. 审批配对
openclaw pairing approve telegram 123456

# 4. 用户可以正常发送消息
```

**安全建议**：
- 生产环境使用 `pairing` 策略
- 测试环境可以使用 `accept`
- 敏感场景使用 `reject` + 白名单

**考察点**：
- 是否理解安全策略
- 是否知道配对审批流程

### Q11: Gateway 启动失败，如何排查？

**答案**：

**常见原因和解决方案**：

**1. 端口被占用**
```bash
# 检查端口占用
lsof -i :18789

# 解决方案 1: 停止占用进程
kill -9 <PID>

# 解决方案 2: 使用其他端口
openclaw gateway --port 18790
```

**2. 配置文件错误**
```bash
# 检查配置文件
cat ~/.openclaw/config/settings.json

# 验证 JSON 格式
jq . ~/.openclaw/config/settings.json

# 重置配置
openclaw onboard --reset
```

**3. API Key 未配置**
```bash
# 检查 API Key
openclaw config get provider
openclaw config get model

# 设置 API Key
export ANTHROPIC_API_KEY=sk-ant-...
openclaw config set provider anthropic
```

**4. 权限问题**
```bash
# 检查文件权限
ls -la ~/.openclaw/

# 修复权限
chmod 700 ~/.openclaw/
chmod 600 ~/.openclaw/config/settings.json
```

**5. 依赖缺失**
```bash
# 检查 Node.js 版本
node --version  # 应该 >= 22.0.0

# 重新安装
npm install -g openclaw@latest
```

**诊断工具**：
```bash
# 使用 doctor 命令
openclaw doctor

# 查看详细日志
openclaw gateway --verbose
```

**考察点**：
- 是否掌握系统性排查方法
- 是否熟悉常见错误和解决方案

### Q12: 如何实现 CLI 命令的自动化和脚本化？

**答案**：

**方式 1: Bash 脚本**
```bash
#!/bin/bash

# 启动 Gateway
openclaw gateway &
GATEWAY_PID=$!

# 等待 Gateway 启动
sleep 3

# 发送消息
openclaw message send --to @all --message "系统启动"

# 清理
trap "kill $GATEWAY_PID" EXIT
```

**方式 2: TypeScript 脚本**
```typescript
import { exec } from 'child_process';
import { promisify } from 'util';

const execAsync = promisify(exec);

async function automateOpenClaw() {
  // 启动 Gateway
  exec('openclaw gateway > /dev/null 2>&1 &');
  await new Promise(resolve => setTimeout(resolve, 3000));

  // 发送消息
  await execAsync('openclaw message send --to @all --message "启动"');

  // Agent 交互
  const { stdout } = await execAsync(
    'openclaw agent --message "总结任务" --json'
  );
  const response = JSON.parse(stdout);
  console.log(response);
}

automateOpenClaw();
```

**方式 3: 使用管道**
```bash
# 批量处理任务
cat tasks.txt | while read task; do
  openclaw agent --message "$task" --json | jq '.response'
done

# 结合其他工具
openclaw channels status --json | jq '.channels[] | select(.status == "offline")'
```

**方式 4: Cron 定时任务**
```bash
# 每天 9 点发送通知
0 9 * * * openclaw agent --message "今日任务总结"

# 每小时检查 Gateway 状态
0 * * * * openclaw gateway health || openclaw gateway restart
```

**考察点**：
- 是否能编写自动化脚本
- 是否了解 JSON 输出格式
- 是否能结合其他工具使用

## 高级架构题（高级）

### Q13: OpenClaw CLI 如何与 Gateway 通信？使用什么协议？

**答案**：

**通信协议**：HTTP/REST API

**通信流程**：
```
CLI → HTTP POST → Gateway (localhost:18789) → Channel API
```

**示例**：
```bash
# CLI 命令
openclaw message send --to +1234567890 --message "Hello"

# 等价的 HTTP 请求
curl -X POST http://localhost:18789/api/message \
  -H "Content-Type: application/json" \
  -d '{
    "to": "+1234567890",
    "message": "Hello"
  }'
```

**CLI 内部实现**（简化）：
```typescript
class OpenClawCLI {
  private gatewayUrl = 'http://localhost:18789';

  async sendMessage(to: string, message: string) {
    const response = await fetch(`${this.gatewayUrl}/api/message`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ to, message }),
    });

    return response.json();
  }
}
```

**考察点**：
- 是否理解 CLI 和 Gateway 的通信机制
- 是否能用 HTTP 客户端替代 CLI

### Q14: 如何扩展 OpenClaw CLI 的功能？

**答案**：

**方式 1: 使用 Extension API**
```typescript
// my-extension.ts
import { Extension } from 'openclaw';

export class MyExtension implements Extension {
  name = 'my-extension';

  async onMessage(message: Message) {
    // 自定义消息处理逻辑
    console.log('收到消息:', message);
  }

  async onAgentResponse(response: AgentResponse) {
    // 自定义 Agent 响应处理
    console.log('Agent 响应:', response);
  }
}
```

**方式 2: 使用 Hook 系统**
```bash
# 安装 Hook
openclaw hooks install my-hook

# 启用 Hook
openclaw hooks enable my-hook
```

**方式 3: 使用 Plugin SDK**
```typescript
// my-plugin/index.ts
export default {
  name: 'my-plugin',
  commands: {
    'my-command': async (args) => {
      // 自定义命令逻辑
      console.log('执行自定义命令:', args);
    },
  },
};
```

**方式 4: 编写自定义脚本**
```typescript
// 使用 OpenClaw 作为库
import { Gateway, Agent } from 'openclaw';

const gateway = new Gateway();
await gateway.start();

const agent = new Agent(gateway);
const response = await agent.process('总结任务');
console.log(response);
```

**考察点**：
- 是否了解扩展机制
- 是否能编写自定义扩展
