# 实战代码场景2：RAG元数据扩展

> 本文档演示在 RAG 系统中使用动态 Schema 灵活扩展文档元数据

---

## 一、场景描述

### 1.1 业务需求

**场景**：构建企业级 RAG 知识库系统，需要存储来自多个业务部门的文档，每个部门有不同的元数据需求。

**核心挑战**：
- 不同部门的文档元数据字段不同（技术部门需要版本号，法务部门需要审批状态）
- 元数据需求随业务发展不断变化
- 需要支持元数据过滤查询（按部门、按状态、按时间范围）
- 需要支持多版本元数据共存（旧文档保留旧元数据，新文档使用新元数据）

**业务价值**：
- 提高文档检索精度（通过元数据过滤）
- 支持业务快速迭代（无需修改 Schema）
- 降低维护成本（统一的 Collection 管理）

### 1.2 技术目标

1. **动态元数据扩展**：支持任意元数据字段的添加
2. **元数据过滤查询**：基于元数据进行精准过滤
3. **多版本共存**：旧文档和新文档的元数据可以不同
4. **元数据更新策略**：支持元数据的增量更新

### 1.3 预期效果

- 不同部门可以添加自己的元数据字段
- 元数据过滤查询性能满足需求
- 支持元数据的增量更新
- 多版本元数据可以共存

## 二、技术方案

### 2.1 架构设计

```
┌─────────────────────────────────────────────────────────┐
│                    业务层                                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ 技术部门  │  │ 法务部门  │  │ 财务部门  │              │
│  └──────────┘  └──────────┘  └──────────┘              │
└─────────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│                  RAG 知识库系统                          │
│  ┌─────────────────────────────────────────────────┐   │
│  │  固定字段（核心元数据）：                         │   │
│  │  - id, vector, text                             │   │
│  │  - department (部门)                            │   │
│  │  - created_at (创建时间)                        │   │
│  │  - updated_at (更新时间)                        │   │
│  └─────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────┐   │
│  │  动态字段（业务元数据）：                         │   │
│  │  - 技术部门: version, tech_stack, api_version   │   │
│  │  - 法务部门: approval_status, legal_category    │   │
│  │  - 财务部门: fiscal_year, budget_code           │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 2.2 元数据设计策略

**固定字段**（高频访问，需要索引）：
- `id`: INT64, 主键
- `vector`: FLOAT_VECTOR, 768维
- `text`: VARCHAR, 文档内容
- `department`: VARCHAR, 部门（高频过滤）
- `created_at`: INT64, 创建时间（高频过滤）
- `updated_at`: INT64, 更新时间

**动态字段**（业务元数据，灵活扩展）：
- 技术部门：`version`, `tech_stack`, `api_version`, `language`
- 法务部门：`approval_status`, `legal_category`, `contract_type`
- 财务部门：`fiscal_year`, `budget_code`, `cost_center`

### 2.3 元数据过滤策略

**策略1：单一元数据过滤**
```python
filter='department == "tech" and version == "v2.0"'
```

**策略2：复合元数据过滤**
```python
filter='department == "legal" and approval_status == "approved" and created_at > 1640000000'
```

**策略3：范围过滤**
```python
filter='department == "finance" and fiscal_year >= 2024 and fiscal_year <= 2026'
```

### 2.4 多版本共存策略

**版本1（初始版本）**：
- 只有基础元数据：department, created_at

**版本2（扩展版本）**：
- 添加业务元数据：version, approval_status, fiscal_year

**版本3（完整版本）**：
- 添加更多元数据：tech_stack, legal_category, budget_code

**共存机制**：
- 旧文档保留旧元数据（缺失新字段）
- 新文档使用新元数据（包含所有字段）
- 查询时安全处理缺失字段

### 3.2 创建RAG知识库Collection

```python
def create_rag_collection(client: MilvusClient) -> None:
    """
    创建RAG知识库Collection（启用动态Schema）

    [来源: reference/search_RAG多租户_03.md]
    """
    print("=" * 60)
    print("步骤1：创建RAG知识库Collection")
    print("=" * 60)

    # 删除已存在的Collection
    if client.has_collection(COLLECTION_NAME):
        client.drop_collection(COLLECTION_NAME)
        print(f"已删除旧Collection: {COLLECTION_NAME}")

    # 创建Schema（启用动态字段）
    schema = client.create_schema(
        auto_id=False,
        enable_dynamic_field=True  # 启用动态元数据
    )

    # 固定字段（核心元数据）
    schema.add_field(field_name="id", datatype=DataType.INT64, is_primary=True)
    schema.add_field(field_name="vector", datatype=DataType.FLOAT_VECTOR, dim=VECTOR_DIM)
    schema.add_field(field_name="text", datatype=DataType.VARCHAR, max_length=65535)

    # 高频过滤字段（固定）
    schema.add_field(field_name="department", datatype=DataType.VARCHAR, max_length=64)
    schema.add_field(field_name="created_at", datatype=DataType.INT64)
    schema.add_field(field_name="updated_at", datatype=DataType.INT64)

    # 创建索引
    index_params = client.prepare_index_params()
    index_params.add_index(field_name="vector", index_type="AUTOINDEX", metric_type="COSINE")
    index_params.add_index(field_name="department", index_type="INVERTED")
    index_params.add_index(field_name="created_at", index_type="INVERTED")

    # 创建Collection
    client.create_collection(
        collection_name=COLLECTION_NAME,
        schema=schema,
        index_params=index_params
    )

    print(f"✓ Collection创建成功: {COLLECTION_NAME}")
    print(f"✓ 固定字段: id, vector, text, department, created_at, updated_at")
    print(f"✓ 动态字段: 任意业务元数据（自动存储在$meta中）")
```

def verify_multi_version_coexistence(client: MilvusClient) -> None:
    """
    验证多版本元数据共存

    [来源: reference/search_RAG多租户_03.md]
    """

    print("步骤5：多版本元数据共存验证")
    print("=" * 60)

    query_vector = np.random.randn(VECTOR_DIM).tolist()

    # 查询所有技术部门文档（包含版本1和版本2）
    print("\n5.1 查询所有技术部门文档")
    results = client.search(
        collection_name=COLLECTION_NAME,
        data=[query_vector],
        filter='department == "tech"',
        output_fields=["id", "text", "created_at", "version", "tech_stack"],
        limit=10
    )

    print(f"\n找到 {len(results[0])} 条技术部门文档")

    v1_count = 0
    v2_count = 0

    for hit in results[0]:
        version = hit.get('version', None)
        if version is None:
            v1_count += 1
            print(f"  ID: {hit['id']} - 版本1（无version字段）")
        else:
            v2_count += 1
            print(f"  ID: {hit['id']} - 版本2（version={version}, tech_stack={hit.get('tech_stack', 'N/A')}）")

    print(f"\n统计:")
    print(f"  - 版本1文档: {v1_count}条（只有基础元数据）")
    print(f"  - 版本2文档: {v2_count}条（包含业务元数据）")
    print(f"  - 说明: 旧文档和新文档可以共存，查询时安全处理缺失字段")

def update_metadata_strategy(client: MilvusClient) -> None:
    """
    演示元数据更新策略

    [来源: reference/search_RAG多租户_03.md]
    """

    # 策略1：使用upsert更新元数据

        "id": 0,
        "vector": np.random.randn(VECTOR_DIM).tolist(),
        "text": "Technical document 0: API documentation and usage guide (UPDATED)",

        "tech_stack": "Python",
        "api_version": "2024.2",

        collection_name=COLLECTION_NAME,

    if results[0]:
        hit = results[0][0]

### 4.4 故障排查

**问题1：元数据过滤返回空结果**
```
原因: 该文档没有该元数据字段（版本1文档）
解决: 使用复合过滤条件，或查询时安全处理缺失字段
```

**问题2：元数据更新失败**
```
原因: upsert操作需要提供所有固定字段
解决: 确保upsert数据包含所有固定字段（id, vector, text等）
```

**问题3：查询性能慢**
```
原因: 动态字段无法创建索引
解决: 将高频过滤字段定义为固定字段，创建索引
```

2. **多版本共存**：旧文档和新文档可以有不同的元数据
3. **过滤查询有效**：基于元数据的过滤查询正常工作
4. **更新策略简单**：使用upsert可以更新元数据

### 5.2 最佳实践

1. **核心元数据固定**：将高频过滤字段（department, created_at）定义为固定字段
2. **业务元数据动态**：将业务特定字段定义为动态字段
3. **安全访问**：使用 `dict.get()` 方法安全访问动态字段
4. **版本管理**：在元数据中记录版本信息，便于追踪

### 5.3 生产建议

1. **适用场景**：多部门协作、元数据需求多样化、快速迭代
2. **性能优化**：为固定字段创建索引，减少动态字段过滤
3. **数据一致性**：使用upsert更新元数据，确保数据一致性
4. **监控指标**：查询延迟、元数据字段数量、版本分布
