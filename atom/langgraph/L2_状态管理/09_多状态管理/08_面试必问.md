## 问题1："LangGraph 中如何管理复杂工作流的多种状态？"

### 普通回答（不出彩）

"可以用 TypedDict 定义状态，然后在节点中读写。"

**问题在哪？** 只说了最基础的用法，完全没有体现对多状态管理的理解。面试官会觉得你只用过单一 Schema 的简单图，没有处理过复杂工作流。

### 出彩回答（推荐）

> **LangGraph 的多状态管理有三个层次：**
>
> **第一层：Schema 分离——控制图的边界**
>
> 通过 `input_schema` / `output_schema` 控制图的入口和出口，内部用 `OverallState` 管理完整状态。这样外部调用者只看到干净的接口，内部实现细节完全隐藏：
>
> ```python
> class InputState(TypedDict):
>     question: str                    # 外部只需传这一个
>
> class OutputState(TypedDict):
>     answer: str                      # 外部只能拿到这一个
>
> class OverallState(InputState, OutputState):
>     raw_chunks: list[str]            # 内部中间状态，对外不可见
>     rerank_scores: list[float]
>
> graph = StateGraph(
>     OverallState,
>     input_schema=InputState,         # 入口白名单
>     output_schema=OutputState,       # 出口白名单
> )
> ```
>
> **第二层：节点级分组——每个节点只看自己需要的**
>
> 每个节点通过函数参数的类型注解自动推断 `input_schema`，只接收需要的状态子集。源码中 `add_node()` 会检查函数第一个参数的类型提示，如果是带 type hints 的类，就自动作为该节点的 input_schema：
>
> ```python
> class PrivateState(TypedDict):
>     bar: str
>
> def node_2(state: OverallState) -> PrivateState:
>     return {"bar": state["foo"] + " is"}
>
> def node_3(state: PrivateState) -> OutputState:
>     return {"graph_output": state["bar"] + " Lance"}
> ```
>
> **第三层：子图隔离——子模块有独立的状态空间**
>
> 子图有完全独立的 Channel 集合，通过共享键或函数包装与父图通信。命名空间机制（`NS_SEP="|"`, `NS_END=":"`）确保每个子图有独立的 checkpoint 空间。
>
> **底层实现上**，所有 Schema 共享同一个 Channel 池。`_add_schema()` 方法将多个 Schema 的 Channel 合并到 `self.channels` 统一注册表中，Schema 只是 Channel 的过滤视图——同名字段必须兼容，不同 Reducer 的同名字段会抛出错误。
>
> 此外还有 `context_schema`，用于传递不可变的运行时配置（如 API keys），它不创建 Channel，通过 `Runtime` 对象访问：
>
> ```python
> def node(state: State, runtime: Runtime[ContextSchema]):
>     api_key = runtime.context["api_key"]
> ```
>
> **在实际 RAG 项目中**，我用三层状态管理构建检索增强生成系统：InputState 只接收用户问题，OverallState 管理检索中间结果和 Prompt 模板，OutputState 只返回最终答案和来源。检索子图和生成子图各自有独立的状态空间，互不干扰。

### 为什么这个回答出彩？

1. 从三个层次（Schema 分离、节点分组、子图隔离）系统性回答，展示了完整的知识体系
2. 提到了底层 Channel 池合并机制和 `_add_schema()` 方法，展示源码级理解
3. 区分了 `context_schema` 和 `state_schema` 的本质差异（是否创建 Channel）
4. 提到了命名空间常量 `NS_SEP` 和 `NS_END`，展示对子图隔离机制的深入理解
5. 联系了实际 RAG 项目应用，展示实战经验

### 追问方向

- **"input_schema 和 output_schema 不设置会怎样？"**
  答：默认都等于 `state_schema`。这意味着外部可以传入所有字段、拿到所有字段，没有任何边界控制。对于简单图没问题，但复杂系统建议显式设置。

- **"同名字段在不同 Schema 中有不同 Reducer 会怎样？"**
  答：`_add_schema()` 合并时会检查兼容性。LastValue Channel 与任何 Channel 兼容（会被覆盖），但两个不同的 BinaryOperatorAggregate Channel 会抛出错误。

- **"节点的 input_schema 推断机制是怎么工作的？"**
  答：`add_node()` 检查函数第一个参数的类型注解，如果是带 type hints 的类（如 TypedDict），就自动作为该节点的 input_schema。也可以通过 `input_schema` 参数显式指定。

## 问题2："子图和父图之间如何通信？有哪些方式？"

### 普通回答（不出彩）

"子图可以访问父图的状态。"

**问题在哪？** 这个回答是错误的。子图和父图的状态是完全隔离的，子图不能直接访问父图的状态。面试官会立刻判断你没有真正用过子图。

### 出彩回答（推荐）

> **子图和父图的状态是完全隔离的——每个子图有自己独立的 Channel 集合和 checkpoint 空间。有三种通信方式：**
>
> **方式1：共享键自动传递**
>
> 将编译后的子图直接添加为节点，同名键自动在父子图间传递：
>
> ```python
> class ParentState(TypedDict):
>     foo: str
>
> class SubgraphState(TypedDict):
>     foo: str    # 与父图同名 → 自动传递
>     bar: str    # 子图私有 → 不传递
>
> subgraph = subgraph_builder.compile()
>
> # 直接添加编译后的子图为节点
> parent_builder.add_node("sub", subgraph)
> # 调用时：父图的 foo 自动传入子图，子图的 foo 自动传回父图
> # bar 是子图私有的，父图看不到
> ```
>
> 这是最简单的方式，适用于父子图有共同字段的场景。
>
> **方式2：函数包装手动转换**
>
> 用函数包装子图调用，手动做状态映射。适用于父子图 Schema 完全不同的场景：
>
> ```python
> class ParentState(TypedDict):
>     query: str
>     result: str
>
> class SubgraphState(TypedDict):
>     bar: str
>     baz: str
>
> subgraph = subgraph_builder.compile()
>
> def call_subgraph(state: ParentState):
>     # 手动转换：父图状态 → 子图输入
>     response = subgraph.invoke({"bar": state["query"]})
>     # 手动转换：子图输出 → 父图状态
>     return {"result": response["bar"]}
>
> parent_builder.add_node("sub", call_subgraph)
> ```
>
> **方式3：Command(graph=Command.PARENT)**
>
> 子图节点可以显式发送更新到父图状态，适用于子图需要"越级汇报"的场景：
>
> ```python
> from langgraph.types import Command
>
> def subgraph_node(state: SubgraphState):
>     result = process(state["bar"])
>     return Command(
>         update={"result": result},
>         graph=Command.PARENT    # 更新发送到父图
>     )
> ```
>
> **命名空间机制保证隔离**
>
> 源码中定义了两个常量：
> - `NS_SEP = "|"` — 分隔层级
> - `NS_END = ":"` — 分隔命名空间和 task_id
>
> 命名空间格式：`{node_name}:{task_id}|{child_node_name}:{child_task_id}`
>
> 示例：
> - `""` — 根图
> - `"agent:abc123"` — 一级子图
> - `"agent:abc123|researcher:def456"` — 二级子图
>
> 每个子图在 checkpoint 中有独立的命名空间，状态互不干扰。
>
> **在实际 RAG 项目中**，我用方式1（共享键）让检索子图和父图共享 `query` 字段，用方式2（函数包装）调用一个 Schema 完全不同的外部评估子图，用方式3（Command.PARENT）让子图中的质量检查节点直接更新父图的 `needs_retry` 标志。

### 为什么这个回答出彩？

1. 先纠正了"子图可以访问父图状态"的常见误解，展示准确理解
2. 系统性地列出三种通信方式，每种都有代码示例和适用场景
3. 提到了命名空间机制的具体实现（NS_SEP、NS_END），展示源码级理解
4. 区分了三种方式的适用场景，展示实战判断力
5. 联系了实际 RAG 项目中三种方式的混合使用

### 追问方向

- **"共享键方式和函数包装方式有什么性能差异？"**
  答：共享键方式由框架自动处理状态传递，子图作为 Pregel 子任务执行，可以利用框架的 checkpoint 和 stream 机制。函数包装方式是手动调用 `invoke()`，本质上是嵌套执行，灵活性更高但失去了部分框架优化。

- **"Command.PARENT 可以跨多层传递吗？"**
  答：`Command(graph=Command.PARENT)` 只能向上传递一层。如果需要跨多层，每层都需要显式转发。

- **"子图的 checkpoint 和父图是什么关系？"**
  答：通过 config keys 共享同一个 checkpointer 实例，但命名空间不同。父图的 checkpoint 在根命名空间 `""`，子图在 `"node_name:task_id"` 命名空间下，互不干扰。

## 面试准备建议

### 回答框架

面试中回答 LangGraph 多状态管理相关问题时，建议按这个框架组织：

1. **先说结论**：一句话概括核心答案（多状态管理 = 给数据画边界线）
2. **分层展开**：从 Schema 分离 → 节点分组 → 子图隔离，由浅入深
3. **说底层机制**：提到 Channel 池、`_add_schema()`、命名空间
4. **举实例**：用 RAG 场景举一个具体例子
5. **提注意事项**：主动说出常见陷阱（如子图状态不能直接访问）

### 核心知识点速记

| 概念 | 关键点 |
|------|--------|
| 四种 Schema | `state_schema`（内部完整）、`input_schema`（入口）、`output_schema`（出口）、`context_schema`（不可变上下文） |
| 默认行为 | input/output 默认等于 state_schema，context 默认 None |
| Channel 池 | 所有 Schema 合并到统一的 `self.channels`，Schema 是过滤视图 |
| 节点推断 | 函数第一个参数的类型注解自动作为节点 input_schema |
| 子图隔离 | 独立 Channel 集合，通过共享键/函数包装/Command.PARENT 通信 |
| 命名空间 | `NS_SEP="\|"` 分隔层级，`NS_END=":"` 分隔命名空间和 task_id |
| context_schema | 不创建 Channel，通过 `Runtime` 对象访问，不可变 |

### 加分项

- 能说出 `_add_schema()` 的合并规则（同名字段兼容性检查）
- 能画出四层状态边界：输入接口 → 内部完整状态 → 节点级子集 → 子图私有状态
- 能对比三种子图通信方式的适用场景
- 能解释命名空间格式 `"agent:abc123|researcher:def456"` 的含义
- 能区分 `context_schema` 和 `state_schema` 的本质差异
