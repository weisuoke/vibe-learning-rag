## 卡片1：什么是状态转换函数？

**一句话**：状态转换函数就是 LangGraph 节点内部的数据处理逻辑，接收完整状态、返回部分更新。

**类比**：就像餐厅厨师——收到完整的点单（State），只做自己负责的那道菜（转换逻辑），然后把做好的菜交出去（返回 dict）。厨师不需要管其他桌的菜，也不需要管传菜流程。

**代码直觉**：

```python
from typing import TypedDict

class State(TypedDict):
    query: str
    context: list[str]
    answer: str

def retrieve_node(state: State) -> dict:
    """检索节点：接收完整状态，只更新 context 字段"""
    query = state["query"]              # 1. 从状态中读取需要的数据
    docs = search_documents(query)      # 2. 执行转换逻辑
    return {"context": docs}            # 3. 只返回要更新的字段
    # query 和 answer 保持不变，框架自动处理
```

**核心原则**：每个节点就是一个状态转换函数，是图执行的基本单元。节点不需要关心"状态怎么合并"（那是 Reducer 的事），只需要关心"拿到数据后怎么加工"。

**2 分钟记住**：状态转换函数 = 接收原料 → 加工处理 → 输出成品。

## 卡片2：节点函数签名——三种合法写法

**一句话**：签名是 `State -> dict`，接收完整状态，返回部分更新。同步、异步、带配置三种写法都合法。

**三种签名**：

```python
from langchain_core.runnables import RunnableConfig

# 写法1：同步函数（最常用）
def sync_node(state: State) -> dict:
    result = process(state["input"])
    return {"output": result}

# 写法2：异步函数（调用异步 API 时使用）
async def async_node(state: State) -> dict:
    result = await async_llm_call(state["query"])
    return {"answer": result}

# 写法3：带配置参数（需要访问运行时配置时使用）
def config_node(state: State, config: RunnableConfig) -> dict:
    user_id = config["configurable"].get("user_id", "anonymous")
    return {"user_id": user_id, "output": process(state["input"])}
```

**怎么选？**

| 场景 | 选择 | 原因 |
|------|------|------|
| 纯数据处理 | 同步 | 简单直接，没有 I/O 等待 |
| 调用 LLM / 外部 API | 异步 | 避免阻塞，提高并发性能 |
| 需要 thread_id / user_id | 带配置 | 从 RunnableConfig 获取运行时参数 |

**注意**：不管哪种写法，返回值都是 `dict`（部分更新），不是完整的 State。

**2 分钟记住**：同步处理数据、异步调用 API、带配置读参数——三种签名覆盖所有场景。

## 卡片3：部分返回的秘密——只更新你关心的字段

**一句话**：只返回要更新的字段，未返回的字段保持不变。这是 LangGraph 状态管理的核心设计。

**举例**：

```python
class State(TypedDict):
    a: str
    b: int
    c: list
    d: dict

# 假设当前状态：a="hello", b=42, c=[1,2], d={"x": 1}

def node(state: State) -> dict:
    return {"a": "world"}
    # 执行后：a="world", b=42, c=[1,2], d={"x": 1}
    # 只有 a 变了，其他字段原封不动
```

**为什么这样设计？** 传统方式返回完整状态，每个节点都要知道所有字段，节点之间高度耦合。LangGraph 方式返回部分更新，每个节点只关心自己的字段，新增字段不影响现有节点。

**前端类比**：就像 React 的 `setState`——你只传要改的字段，框架帮你合并。

**常见错误**：

```python
# ❌ 错误：返回完整状态（多余且危险）
def bad_node(state: State) -> dict:
    return {"a": "new", "b": state["b"], "c": state["c"], "d": state["d"]}

# ✅ 正确：只返回要更新的
def good_node(state: State) -> dict:
    return {"a": "new"}
```

**2 分钟记住**：只返回变化的字段，框架负责合并——这让每个节点只关注自己的职责。

## 卡片4：有 Reducer vs 无 Reducer——覆盖还是合并？

**一句话**：无 Reducer 的字段直接覆盖旧值，有 Reducer 的字段通过函数合并新旧值。

**对比示例**：

```python
from typing import Annotated
import operator

class State(TypedDict):
    name: str                                    # 无 Reducer → 覆盖
    items: Annotated[list[str], operator.add]    # 有 Reducer → 追加

# 假设当前状态：name="Alice", items=["a", "b"]

def node(state: State) -> dict:
    return {"name": "Bob", "items": ["c"]}

# 执行后：
# name = "Bob"          ← 直接覆盖（LastValue Channel）
# items = ["a", "b", "c"]  ← 追加合并（BinaryOperatorAggregate Channel）
```

**这对转换函数意味着什么？**

```python
# 对于覆盖字段：返回最终值
def update_name(state: State) -> dict:
    return {"name": "Bob"}  # 直接变成 "Bob"

# 对于 Reducer 字段：返回增量值（不是完整值！）
def add_items(state: State) -> dict:
    return {"items": ["new_item"]}  # 追加 "new_item"，不是替换整个列表

# ❌ 常见错误：对 Reducer 字段手动合并
def bad_add_items(state: State) -> dict:
    old = state["items"]
    return {"items": old + ["new_item"]}  # 会导致重复！
    # Reducer 会再追加一次，结果变成 ["a", "b", "a", "b", "new_item"]
```

**RAG 场景中的典型搭配**：

| 字段 | 策略 | 原因 |
|------|------|------|
| `query` | 覆盖 | 当前查询只有一个 |
| `messages` | Reducer（add_messages） | 对话历史需要累积 |
| `documents` | Reducer（operator.add） | 检索结果需要追加 |
| `answer` | 覆盖 | 最终答案只有一个 |
| `step` | 覆盖 | 当前步骤标记 |

**2 分钟记住**：覆盖字段返回最终值，Reducer 字段返回增量值——搞混了就会数据重复。

## 卡片5：数据清洗节点——垃圾进垃圾出的防线

**一句话**：在图中专门设置清洗节点，确保数据质量，防止脏数据污染下游节点。

**为什么需要清洗？** 用户输入 `"  什么是 RAG？？？  \n\n "` 直接送给 LLM，多余空格和换行符会干扰输出质量。清洗后变成 `"什么是RAG"`，LLM 收到干净输入，输出更稳定。

**清洗节点模板**：

```python
import re

def clean_node(state: State) -> dict:
    raw_query = state.get("query", "")

    # Step 1: 基础清洗
    cleaned = raw_query.strip()                          # 去首尾空白
    cleaned = " ".join(cleaned.split())                  # 合并多余空格
    cleaned = re.sub(r'[？！。，]{2,}', lambda m: m.group()[0], cleaned)  # 去重复标点

    # Step 2: 空值防御
    if not cleaned:
        return {"query": cleaned, "error": "空查询", "is_valid": False}

    # Step 3: 长度限制
    if len(cleaned) > 500:
        cleaned = cleaned[:500]

    return {"query": cleaned, "is_valid": True}
```

**2 分钟记住**：清洗节点是 RAG 系统的第一道防线——垃圾进垃圾出，干净进干净出。

## 卡片6：输出格式化——把 LLM 的"自由发挥"变成结构化数据

**一句话**：节点返回前，将内部数据转换为状态 Schema 期望的格式，确保下游节点能正确消费。

**为什么需要格式化？**

LLM 返回的是自由文本，但下游节点通常需要结构化数据。比如 LLM 返回一大段带参考来源的文本，下游需要的是 `{"answer": "...", "sources": [...]}` 这样的结构。

**格式化节点模板**：

```python
def format_node(state: State) -> dict:
    raw_answer = state.get("raw_answer", "")
    docs = state.get("documents", [])

    # Step 1: 清理 LLM 输出
    answer = raw_answer.strip()
    if not answer:
        answer = "抱歉，无法生成回答。"

    # Step 2: 截断过长内容
    if len(answer) > 2000:
        answer = answer[:2000] + "..."

    # Step 3: 构建结构化输出
    sources = [{"content": d["text"][:200], "score": float(d.get("score", 0))}
               for d in docs if isinstance(d, dict)]

    return {
        "answer": answer,
        "sources": sources,
        "metadata": {"source_count": len(sources), "truncated": len(raw_answer) > 2000}
    }
```

**JSON 解析兜底**（LLM 返回 JSON 时）：

```python
import json, re

def parse_json_node(state: State) -> dict:
    raw = state["llm_output"]
    try:
        parsed = json.loads(raw)
    except json.JSONDecodeError:
        match = re.search(r'```json\s*(.*?)\s*```', raw, re.DOTALL)
        parsed = json.loads(match.group(1)) if match else {"raw_text": raw, "parse_failed": True}
    return {"structured_output": parsed}
```

**2 分钟记住**：格式化节点把 LLM 的"自由发挥"变成结构化数据，是连接 LLM 和业务逻辑的桥梁。

## 卡片7：Command API——状态更新和路由决策一步到位

**一句话**：Command 让节点同时完成状态更新和路由决策，不再需要单独的条件边。

**传统方式 vs Command 方式**：

```python
# 传统方式：节点只更新状态，路由靠条件边
def classify_node(state: State) -> dict:
    return {"category": "technical" if is_technical(state["query"]) else "general"}

def route_fn(state: State) -> str:
    return "tech_node" if state["category"] == "technical" else "general_node"

builder.add_conditional_edges("classify", route_fn, {"tech_node": "tech", "general_node": "general"})

# Command 方式：节点同时更新状态 + 决定路由
from langgraph.types import Command
from typing import Literal

def classify_node(state: State) -> Command[Literal["tech", "general"]]:
    if is_technical(state["query"]):
        return Command(update={"category": "technical"}, goto="tech")
    return Command(update={"category": "general"}, goto="general")
```

**Command 的核心参数**：

```python
Command(
    update={"field": "value"},   # 状态更新（和普通 dict 返回一样）
    goto="next_node",            # 下一个节点（替代条件边）
    graph=Command.PARENT         # 可选：发送到父图（子图场景）
)
```

**RAG 场景实战**：

```python
def query_router(state: State) -> Command[Literal["retrieve", "direct_answer", "clarify"]]:
    query = state["query"]
    confidence = analyze_query(query)

    if confidence > 0.8:
        return Command(update={"route": "retrieve"}, goto="retrieve")
    elif confidence > 0.4:
        return Command(update={"route": "direct"}, goto="direct_answer")
    else:
        return Command(update={"route": "clarify", "clarify_reason": "查询不够明确"}, goto="clarify")
```

**2 分钟记住**：Command = update + goto，一个返回值搞定状态更新和路由决策。

## 卡片8：管道模式——每个节点只做一件事

**一句话**：多个节点串联形成转换管道，每步做一件事，像工厂流水线一样依次处理。

**反面教材 vs 正面教材**：

```python
# ❌ "上帝节点"：清洗 + 检索 + 生成 + 格式化全塞一起
def god_node(state: State) -> dict:
    query = state["query"].strip().lower()
    docs = retriever.search(query, top_k=5)
    context = "\n".join(d.content for d in docs)
    answer = llm.generate(f"Context: {context}\nQ: {query}")
    return {"context": docs, "answer": answer.strip()[:2000]}
```

```python
# ✅ 管道模式：每个节点只做一件事
def clean_node(state: State) -> dict:
    return {"cleaned_query": state["query"].strip().lower()}

def retrieve_node(state: State) -> dict:
    docs = retriever.search(state["cleaned_query"], top_k=5)
    return {"documents": [d.content for d in docs]}

def filter_node(state: State) -> dict:
    return {"documents": [d for d in state["documents"] if len(d) > 50]}

def generate_node(state: State) -> dict:
    context = "\n".join(state["documents"])
    return {"answer": llm.generate(f"Context: {context}\nQ: {state['cleaned_query']}")}

# 串联：clean → retrieve → filter → generate → END
```

**管道模式的好处**：

| 维度 | 上帝节点 | 管道模式 |
|------|----------|----------|
| 可测试性 | 难以单独测试某个步骤 | 每个节点独立测试 |
| 可调试性 | 出错了不知道哪一步 | 状态快照精确定位 |
| 可复用性 | 整个节点不可复用 | clean_node 可用于多个图 |
| 可维护性 | 改一处可能影响全部 | 改一个节点不影响其他 |

**2 分钟记住**：管道模式 = 流水线，每个工人（节点）只做一道工序，出了问题一眼就能定位。

## 卡片9：错误处理策略——让图优雅地失败

**一句话**：在节点内用 try/except 捕获错误，将错误信息写入状态，而不是让异常炸掉整个图。

**为什么不能直接抛异常？**

```python
# ❌ 异常直接抛出：整个图执行中断，无法恢复
def bad_node(state: State) -> dict:
    result = call_llm(state["query"])  # 如果 LLM 超时，整个图崩溃
    return {"answer": result}

# ✅ 错误写入状态：图可以继续执行，下游节点可以处理错误
def safe_node(state: State) -> dict:
    try:
        result = call_llm(state["query"])
        return {"answer": result, "status": "success", "error": None}
    except TimeoutError:
        return {"answer": "", "status": "failed", "error": "LLM 调用超时"}
    except Exception as e:
        return {"answer": "", "status": "failed", "error": str(e)}
```

**错误处理 + Command 路由**：

```python
from langgraph.types import Command
from typing import Literal

def safe_retrieve(state: State) -> Command[Literal["generate", "fallback"]]:
    try:
        docs = retriever.search(state["query"], top_k=5)
        if not docs:
            return Command(update={"error": "未找到相关文档"}, goto="fallback")
        return Command(update={"documents": docs, "status": "success"}, goto="generate")
    except Exception as e:
        return Command(update={"error": f"检索失败: {str(e)}"}, goto="fallback")
```

**2 分钟记住**：错误信息写入状态而非抛出异常——让图优雅地失败，而不是粗暴地崩溃。

## 卡片10：最佳实践总结——好的状态转换函数长什么样？

**一句话**：好的状态转换函数是纯函数思维、单一职责、防御性编程的结合体。

**五条核心原则**：

```
原则1：只返回要更新的字段
  → 不要返回完整状态，让框架处理合并

原则2：Reducer 字段返回增量
  → 有 operator.add 的字段，返回新增部分，不要手动拼接旧值

原则3：做好空值防御
  → 用 state.get("key", default) 而不是 state["key"]

原则4：错误信息写入状态
  → try/except 捕获异常，把错误信息作为状态字段返回

原则5：每个节点只做一件事
  → 清洗、检索、生成、格式化分开，不要写"上帝节点"
```

**检查清单——写完一个节点后问自己**：

```
□ 是否只返回了需要更新的字段？
□ Reducer 字段是否返回的是增量而非完整值？
□ 是否处理了输入为空/None 的情况？
□ 是否用 try/except 包裹了可能失败的操作？
□ 这个节点是否只做了一件事？
□ 能否不依赖外部状态独立测试这个函数？
```

**综合示例**（所有原则的体现）：

```python
from typing import TypedDict, Annotated
import operator

class RAGState(TypedDict):
    query: str
    documents: Annotated[list[str], operator.add]
    answer: str
    status: str
    error: str

def retrieve_node(state: RAGState) -> dict:
    """检索节点：遵循所有最佳实践"""
    query = state.get("query", "")              # 原则3：空值防御
    if not query:
        return {"status": "error", "error": "查询为空"}  # 原则4：错误入状态

    try:
        docs = retriever.search(query, top_k=5)
        return {
            "documents": [d.content for d in docs],  # 原则2：增量返回
            "status": "retrieved"                      # 原则1：只返回要更新的
        }
    except Exception as e:
        return {"status": "error", "error": f"检索失败: {str(e)}"}
    # 原则5：只做检索，不做清洗也不做生成
```

**2 分钟记住**：五条原则——部分返回、增量更新、空值防御、错误入状态、单一职责。遵循这些，你的 LangGraph 图会更可靠、可测试、可维护。
