# 核心概念 2：泛型函数与泛型结构体

## 一句话定义

泛型让你用类型参数 `<T>` 编写一次代码，编译器自动为每种具体类型生成专用版本（单态化）。

> **TypeScript 开发者注意**：Rust 的泛型在编译时就"消失"了——编译器会为 `foo<i32>` 和 `foo<String>` 各生成一份独立的机器码。TypeScript 的泛型只是类型检查工具，运行时完全擦除。这是本质区别。

---

## 泛型函数

### 基础语法

```rust
// ===== Rust =====
fn largest<T: PartialOrd>(a: T, b: T) -> T {
    if a > b { a } else { b }
}

fn main() {
    let x = largest(10, 20);           // 编译器推断 T = i32
    let y = largest(3.14, 2.71);       // 编译器推断 T = f64
    let z = largest("hello", "world"); // 编译器推断 T = &str
    println!("{x}, {y}, {z}");         // 20, 3.14, world
}
```

```typescript
// ===== TypeScript 对照 =====
// 语法几乎一样，但运行时没有任何泛型痕迹
function largest<T>(a: T, b: T): T {
    return a > b ? a : b; // TS 这里会报错，因为 T 没有约束
}
```

**关键差异**：

| 对比项 | Rust `<T: PartialOrd>` | TypeScript `<T>` |
|--------|----------------------|------------------|
| 约束方式 | Trait 约束（编译器强制） | `extends` 约束（可选） |
| 运行时 | 单态化，生成多份代码 | 类型擦除，只有一份代码 |
| 性能 | 零开销，等同手写 | 无影响（类型只在编译时） |

### 单态化（Monomorphization）

这是 Rust 泛型的核心机制。编译器看到你调用 `largest(10, 20)` 和 `largest(3.14, 2.71)` 后，会自动生成两个函数：

```rust
// 编译器在幕后生成的代码（你看不到，但实际执行的就是这个）
fn largest_i32(a: i32, b: i32) -> i32 {
    if a > b { a } else { b }
}

fn largest_f64(a: f64, b: f64) -> f64 {
    if a > b { a } else { b }
}
```

**前端类比**：想象 Webpack 的 Tree Shaking 反过来——不是删除没用的代码，而是为每种用到的类型"复制"一份专用代码。结果是：运行时零开销，但二进制文件会变大。

**日常生活类比**：你写了一份"万能菜谱模板"，厨师（编译器）根据客人点的菜（具体类型），提前把每道菜的详细步骤都写好贴在墙上。上菜时不用再查模板，直接照墙上的做，速度最快。

### ZeroClaw 实例：`impl Into<String>` 语法糖

ZeroClaw 源码中大量使用 `impl Into<String>` 作为泛型的简写形式：

```rust
// ===== ZeroClaw 源码：ChatMessage 构造 =====
// 来源：src/providers/traits.rs
impl ChatMessage {
    pub fn system(content: impl Into<String>) -> Self {
        ChatMessage {
            role: "system".to_string(),
            content: content.into(),
        }
    }
}

// 调用时可以传 &str 或 String，都行
let msg1 = ChatMessage::system("你是一个助手");          // &str
let msg2 = ChatMessage::system(String::from("你是一个助手")); // String
```

```typescript
// ===== TypeScript 对照 =====
// TS 中 string 字面量和 String 对象没有所有权区别，不需要这种模式
class ChatMessage {
    static system(content: string): ChatMessage {
        return { role: "system", content };
    }
}
```

**为什么 ZeroClaw 这样写？** 因为 Rust 中 `&str`（借用的字符串切片）和 `String`（拥有所有权的字符串）是不同类型。`impl Into<String>` 让调用者传哪种都行，函数内部统一转成 `String`。这是 Rust 中极其常见的人体工程学模式。

再看一个例子：

```rust
// ===== ZeroClaw 源码：SendMessage 构造 =====
// 来源：src/channels/traits.rs
impl SendMessage {
    pub fn new(content: impl Into<String>, recipient: impl Into<String>) -> Self {
        SendMessage {
            content: content.into(),
            recipient: recipient.into(),
        }
    }
}

// 两个参数都可以灵活传入
let msg = SendMessage::new("Hello!", "user_123");
```

## 泛型结构体

### 基础语法

```rust
// ===== Rust =====
struct Point<T> {
    x: T,
    y: T,
}

let int_point = Point { x: 5, y: 10 };       // Point<i32>
let float_point = Point { x: 1.0, y: 4.0 };  // Point<f64>
```

```typescript
// ===== TypeScript 对照 =====
interface Point<T> {
    x: T;
    y: T;
}

const intPoint: Point<number> = { x: 5, y: 10 };
const floatPoint: Point<number> = { x: 1.0, y: 4.0 };
// 注意：TS 的 number 不区分整数和浮点数，Rust 区分
```

### ZeroClaw 实例 1：`HookResult<T>` 泛型枚举

```rust
// ===== ZeroClaw 源码：HookResult<T> =====
// 来源：src/hooks/traits.rs
pub enum HookResult<T> {
    Continue(T),      // 继续执行，携带（可能被修改的）数据
    Cancel(String),   // 取消执行，附带原因
}
```

```typescript
// ===== TypeScript 对照 =====
// 类似一个带泛型的联合类型
type HookResult<T> =
    | { kind: "continue"; value: T }
    | { kind: "cancel"; reason: string };
```

**这解决什么问题？** ZeroClaw 的 Hook 系统允许插件在 Agent 执行的各个阶段介入。Hook 可以：
- 放行并可能修改数据（`Continue(T)`）——比如修改即将发送的消息
- 拦截并取消操作（`Cancel(String)`）——比如安全策略阻止危险命令

`T` 在不同 Hook 中代表不同类型：消息、工具调用参数、内存条目等。一个枚举定义，处处复用。

### ZeroClaw 实例 2：`ChatRequest<'a>` 生命周期泛型

```rust
// ===== ZeroClaw 源码：ChatRequest<'a> =====
// 来源：src/providers/traits.rs
pub struct ChatRequest<'a> {
    pub messages: &'a [ChatMessage],    // 借用消息切片
    pub tools: Option<&'a [ToolSpec]>,  // 可选借用工具列表
}
```

这里的 `'a` 不是类型参数，而是**生命周期参数**——它告诉编译器"这个结构体借用的数据至少活到 `'a` 这么久"。

```typescript
// ===== TypeScript 对照 =====
// TS 没有生命周期概念，直接引用就行
interface ChatRequest {
    messages: ChatMessage[];          // 没有借用/所有权区别
    tools?: ToolSpec[];
}
```

**为什么用借用而不是拥有？** `ChatRequest` 只是临时打包数据发给 Provider，不需要拥有这些数据。借用避免了不必要的内存拷贝——对于可能包含大量消息历史的请求，这很重要。

### ZeroClaw 实例 3：`ConfigWrapper<T: ChannelConfig>` 带约束的泛型 + PhantomData

```rust
// ===== ZeroClaw 源码：ConfigWrapper =====
// 来源：src/config/traits.rs
struct ConfigWrapper<T: ChannelConfig>(std::marker::PhantomData<T>);

impl<T: ChannelConfig> ConfigHandle for ConfigWrapper<T> {
    fn section_name(&self) -> &'static str {
        T::SECTION_NAME  // 调用 T 的关联常量
    }
}
```

```typescript
// ===== TypeScript 对照 =====
// TS 中不需要 PhantomData，泛型参数可以只用于类型层面
class ConfigWrapper<T extends ChannelConfig> implements ConfigHandle {
    sectionName(): string {
        // TS 无法在运行时访问泛型参数的静态属性
        // 这是 Rust 泛型比 TS 泛型更强大的地方
        throw new Error("TS 做不到这个");
    }
}
```

**`PhantomData<T>` 是什么？** 结构体没有真正存储 `T` 类型的字段，但需要告诉编译器"我在逻辑上和 `T` 有关"。`PhantomData` 是零大小的标记类型，不占内存，只影响类型检查。

**日常生活类比**：`PhantomData` 就像你在快递单上写了"易碎品"标签——箱子里并没有多一个"易碎品"实体，但快递员（编译器）会因为这个标签改变处理方式。

## impl 块中的泛型

### 为泛型类型实现方法

```rust
// ===== Rust =====
// 为所有 HookResult<T> 实现通用方法
impl<T> HookResult<T> {
    pub fn is_cancel(&self) -> bool {
        matches!(self, HookResult::Cancel(_))
    }

    pub fn unwrap(self) -> T {
        match self {
            HookResult::Continue(val) => val,
            HookResult::Cancel(reason) => panic!("called unwrap on Cancel: {reason}"),
        }
    }
}
```

注意 `impl<T>` 中的 `<T>` ——这是在告诉编译器"我要为**任意类型** T 的 `HookResult<T>` 实现方法"。

```typescript
// ===== TypeScript 对照 =====
// TS 中方法直接写在类/接口里，不需要单独的 impl 块
function isCancel<T>(result: HookResult<T>): boolean {
    return result.kind === "cancel";
}
```

### 为特定类型特化

```rust
// ===== Rust =====
struct Wrapper<T> {
    value: T,
}

// 通用实现：所有 T 都有
impl<T> Wrapper<T> {
    fn new(value: T) -> Self {
        Wrapper { value }
    }
}

// 特化实现：只有 T = String 时才有这个方法
impl Wrapper<String> {
    fn is_empty(&self) -> bool {
        self.value.is_empty()
    }
}

fn main() {
    let w1 = Wrapper::new("hello".to_string());
    println!("{}", w1.is_empty()); // OK

    let w2 = Wrapper::new(42);
    // w2.is_empty(); // 编译错误！i32 版本没有 is_empty 方法
}
```

```typescript
// ===== TypeScript 对照 =====
// TS 无法为泛型类的特定类型参数添加额外方法
// 最接近的做法是类型守卫 + 独立函数
class Wrapper<T> {
    constructor(public value: T) {}
}

// 无法做到：class Wrapper<string> { isEmpty() { ... } }
// 只能用独立函数
function isEmpty(w: Wrapper<string>): boolean {
    return w.value.length === 0;
}
```

**这是 Rust 泛型比 TypeScript 强大的地方之一**：你可以为同一个泛型结构体的不同类型参数提供不同的方法集。编译器会根据实际类型决定哪些方法可用。

## impl Trait 语法糖

`impl Trait` 是 Rust 中一个重要的语法糖，在参数位置和返回位置有不同含义。

### 参数位置：泛型的简写

```rust
// 这两种写法完全等价
fn print_it(x: impl std::fmt::Display) {
    println!("{x}");
}

fn print_it_verbose<T: std::fmt::Display>(x: T) {
    println!("{x}");
}
```

**什么时候用哪个？**
- `impl Trait`：参数类型只出现一次，简洁优先
- `<T: Trait>`：需要多个参数是同一类型，或需要在函数体中引用 `T`

```rust
// 必须用显式泛型：两个参数必须是同一类型
fn add<T: std::ops::Add<Output = T>>(a: T, b: T) -> T {
    a + b
}

// 用 impl Trait 的话，两个参数可以是不同类型（通常不是你想要的）
fn print_two(a: impl Display, b: impl Display) {
    println!("{a}, {b}"); // a 和 b 可以是不同类型，这里没问题
}
```

### 返回位置：隐藏具体类型

```rust
// ===== ZeroClaw 源码：Gateway handler =====
// 来源：src/gateway/routes.rs
async fn handle_health(State(state): State<AppState>) -> impl IntoResponse {
    // 返回类型是 impl IntoResponse
    // 调用者只知道"返回值可以变成 HTTP 响应"，不知道具体类型
    Json(serde_json::json!({ "status": "ok" }))
}
```

```typescript
// ===== TypeScript 对照 =====
// TS 没有直接等价物，最接近的是返回接口类型
async function handleHealth(): Response {
    return new Response(JSON.stringify({ status: "ok" }));
}
```

**返回位置 `impl Trait` 的限制**：函数内所有返回路径必须返回**同一个**具体类型。

```rust
// 编译错误！两个分支返回不同类型
fn make_display(use_int: bool) -> impl Display {
    if use_int {
        42          // i32
    } else {
        "hello"     // &str  -- 类型不同，编译失败！
    }
}
// 解决方案：使用 Box<dyn Display>（详见 05_动态分发与Trait_Object）
```

## 单态化 vs 动态分发（预告）

这是理解 Rust 性能模型的关键对比，将在 `05_动态分发与Trait_Object` 中详细展开。

```rust
// 方式 1：泛型（静态分发 / 单态化）
fn process_static(tool: impl Tool) {
    // 编译器为每种 Tool 实现生成一份 process_static
    // 运行时：直接调用，零开销
    println!("{}", tool.name());
}

// 方式 2：Trait Object（动态分发）
fn process_dynamic(tool: &dyn Tool) {
    // 只有一份 process_dynamic 代码
    // 运行时：通过虚表（vtable）查找方法，有微小开销
    println!("{}", tool.name());
}
```

| 对比项 | 泛型（单态化） | `dyn Trait`（动态分发） |
|--------|--------------|----------------------|
| 代码生成 | 每种类型一份 | 只有一份 |
| 运行时开销 | 零 | 虚表查找（约几纳秒） |
| 二进制大小 | 较大（代码膨胀） | 较小 |
| 灵活性 | 编译时确定类型 | 运行时可切换类型 |
| ZeroClaw 用法 | `impl Into<String>` 参数 | `Box<dyn Provider>` 存储 |

**ZeroClaw 的选择**：核心 Trait（Provider、Channel、Tool、Memory）全部使用 `Box<dyn Trait>` 动态分发，因为需要在运行时根据配置文件选择具体实现。而构造函数参数（如 `impl Into<String>`）使用泛型静态分发，因为类型在编译时就确定了。

## 小结

| 概念 | Rust 语法 | TypeScript 对照 | ZeroClaw 实例 |
|------|----------|----------------|--------------|
| 泛型函数 | `fn foo<T>(x: T)` | `function foo<T>(x: T)` | `with_connection<T>(...)` |
| 泛型结构体 | `struct Foo<T> { f: T }` | `interface Foo<T> { f: T }` | `HookResult<T>` |
| 生命周期泛型 | `struct Foo<'a> { r: &'a T }` | 无对应 | `ChatRequest<'a>` |
| 带约束泛型 | `struct Foo<T: Trait>` | `interface Foo<T extends X>` | `ConfigWrapper<T: ChannelConfig>` |
| impl 块泛型 | `impl<T> Foo<T> { ... }` | 类方法直接写 | `impl<T> HookResult<T>` |
| 特化 impl | `impl Foo<String> { ... }` | 无对应 | - |
| impl Trait 参数 | `fn foo(x: impl Trait)` | 无直接对应 | `impl Into<String>` |
| impl Trait 返回 | `fn foo() -> impl Trait` | 返回接口类型 | `-> impl IntoResponse` |
| 单态化 | 编译器自动完成 | 不存在（类型擦除） | 所有泛型函数 |

**核心记忆点**：
1. Rust 泛型 = 编译时代码生成（单态化），零运行时开销
2. TypeScript 泛型 = 编译时类型检查，运行时完全擦除
3. `impl Trait` 是泛型的语法糖，让代码更简洁
4. ZeroClaw 中 `impl Into<String>` 是最常见的泛型用法——让 `&str` 和 `String` 都能传入
