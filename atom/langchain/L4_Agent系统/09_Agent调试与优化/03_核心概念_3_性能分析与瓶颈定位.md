# 核心概念3：性能分析与瓶颈定位

> 知道 Agent「慢在哪里」比「让它更快」更重要——通过 Token 监控、延迟分析和并发控制，精准定位性能瓶颈。

---

## 概述

Agent 的瓶颈和传统程序不同——不在 CPU/内存，而在 **LLM API 延迟**、**Token 消耗**和**推理循环次数**。一次 Agent 调用可能触发 3-10 次 LLM 请求（每次 1-5 秒），还可能陷入无限循环。

**核心思路：** 先测量，再优化。用回调系统采集指标，用数据定位瓶颈。

---

## 1. Agent 性能的四个关键指标

| 指标 | 含义 | 典型值 | 为什么重要 |
|------|------|--------|-----------|
| **延迟** | 端到端响应时间 | 3-30s | 用户体验 |
| **Token 消耗** | 输入/输出 Token 总量 | 1000-5000/次 | API 费用 |
| **工具效率** | 工具调用次数/成功率/耗时 | 0.1-10s/次 | 执行质量 |
| **循环次数** | Agent 推理步数 | 正常 2-5 步 | 失控风险 |

Agent 的 Token 消耗比普通 Chain 高得多——每轮循环都要带上之前的推理历史：

```
普通 Chain: Prompt 500 + Output 200 = 700 Token

Agent（3 轮循环）:
  第1轮: Prompt 800  + Output 200 = 1000
  第2轮: Prompt 1200 + Output 150 = 1350   ← 累加历史
  第3轮: Prompt 1500 + Output 300 = 1800
  总计: 4150 Token（普通 Chain 的 6 倍！）
```

## 2. Token 使用监控

### 自定义 Token 统计回调

```python
"""Token 使用统计回调 - 通过 on_llm_end 收集每次 LLM 的 Token 用量"""
from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.outputs import LLMResult
from typing import Any


class TokenTracker(BaseCallbackHandler):
    """追踪 Agent 执行过程中的 Token 消耗"""

    def __init__(self):
        self.total_tokens = 0
        self.prompt_tokens = 0
        self.completion_tokens = 0
        self.llm_calls = 0
        self.per_call_usage = []

    def on_llm_end(self, response: LLMResult, **kwargs: Any) -> None:
        self.llm_calls += 1
        if hasattr(response, "llm_output") and response.llm_output:
            usage = response.llm_output.get("token_usage", {})
            prompt = usage.get("prompt_tokens", 0)
            completion = usage.get("completion_tokens", 0)
            total = usage.get("total_tokens", 0)
            self.prompt_tokens += prompt
            self.completion_tokens += completion
            self.total_tokens += total
            self.per_call_usage.append({
                "call": self.llm_calls,
                "prompt": prompt, "completion": completion, "total": total,
            })

    def report(self) -> str:
        cost = (self.prompt_tokens * 2.5 + self.completion_tokens * 10) / 1_000_000
        lines = [
            f"📊 Token Report | LLM Calls: {self.llm_calls}",
            f"   Prompt: {self.prompt_tokens} | Completion: {self.completion_tokens} "
            f"| Total: {self.total_tokens}",
            f"   💰 Est. Cost (GPT-4o): ${cost:.4f}",
        ]
        for c in self.per_call_usage:
            lines.append(f"   #{c['call']}: in={c['prompt']} out={c['completion']}")
        report = "\n".join(lines)
        print(report)
        return report
```

**使用方式：**

```python
tracker = TokenTracker()
result = agent_executor.invoke(
    {"input": "北京今天天气怎么样？"},
    config={"callbacks": [tracker]}
)
tracker.report()
# 📊 Token Report | LLM Calls: 3
#    Prompt: 2450 | Completion: 380 | Total: 2830
#    💰 Est. Cost (GPT-4o): $0.0441
#    #1: in=650 out=120
#    #2: in=850 out=95
#    #3: in=950 out=165
```

### Token 增长模式——快速识别异常

```
正常模式（稳步增长）:        异常模式1（爆炸增长→工具返回太大）:
  #1: in=650  ████           #1: in=650   ████
  #2: in=850  █████          #2: in=5200  ██████████████████████  ← 问题!
  #3: in=950  ██████         #3: in=5800  ████████████████████████

异常模式2（线性增长→Agent 兜圈子）:
  #1~#6: 650→900→1150→1400→1650→1900  ← 还在涨，循环了！
```

## 3. 延迟分析

### 自定义计时回调

```python
"""延迟分析回调 - 记录每个组件的耗时"""
import time
from langchain_core.callbacks import BaseCallbackHandler
from typing import Dict


class LatencyTracker(BaseCallbackHandler):
    def __init__(self):
        self.timings: Dict[str, dict] = {}
        self.completed: list = []

    def on_llm_start(self, serialized, prompts, *, run_id, **kwargs):
        name = serialized.get("name", serialized.get("id", ["LLM"])[-1])
        self.timings[str(run_id)] = {"start": time.time(), "name": f"LLM:{name}", "type": "llm"}

    def on_llm_end(self, response, *, run_id, **kwargs):
        self._record(run_id)

    def on_tool_start(self, serialized, input_str, *, run_id, **kwargs):
        self.timings[str(run_id)] = {
            "start": time.time(), "name": f"Tool:{serialized.get('name','?')}", "type": "tool"
        }

    def on_tool_end(self, output, *, run_id, **kwargs):
        self._record(run_id)

    def _record(self, run_id):
        key = str(run_id)
        if key in self.timings:
            elapsed = time.time() - self.timings[key]["start"]
            entry = {**self.timings[key], "elapsed_s": round(elapsed, 3)}
            del entry["start"]
            self.completed.append(entry)
            print(f"⏱️  {entry['name']}: {elapsed:.2f}s")

    def waterfall(self):
        """打印文字版瀑布图"""
        if not self.completed:
            return
        total = sum(x["elapsed_s"] for x in self.completed)
        print("\n🌊 执行瀑布图")
        print("=" * 60)
        for item in self.completed:
            ratio = item["elapsed_s"] / total if total > 0 else 0
            bar = "█" * int(ratio * 30) + "░" * (30 - int(ratio * 30))
            icon = {"llm": "🤖", "tool": "🔧"}.get(item["type"], "❓")
            print(f"  {icon} {item['name'][:18]:18s} |{bar}| {item['elapsed_s']:.2f}s ({ratio:.0%})")
        print(f"  总计: {total:.2f}s")
```

  🔧 Tool:search       |████████░░░░░░░░░░░░░░░░░░░░░░| 0.85s (14%)

→ LLM 占 85% 时间 → 换更快模型或加缓存

## 4. 并发控制（源码解析）

### max_concurrency 参数

`RunnableConfig` 中控制并行执行的关键参数：

```python
# 源码定义（runnables/config.py）
class RunnableConfig(TypedDict, total=False):
    max_concurrency: int | None     # 控制并行执行数量
    recursion_limit: int            # 默认 25，防止无限循环
    # ...其他字段省略
```

```python
# 场景1：控制并行工具执行
result = agent_executor.invoke(
    {"input": "对比北京和上海的天气"},
    config={"max_concurrency": 3}  # 最多同时执行 3 个工具
)

# 场景2：批处理并发控制
results = agent_executor.batch(
    [{"input": "Q1"}, {"input": "Q2"}, {"input": "Q3"}, {"input": "Q4"}],
    config={"max_concurrency": 2}  # 每次并行 2 个
)
# 执行过程：[Q1,Q2] 并行 → [Q3,Q4] 并行 → 合并结果
```

### ContextThreadPoolExecutor 原理

LangChain 用自定义线程池替代标准 `ThreadPoolExecutor`，解决 **ContextVar 跨线程传播**的问题：

```python
class ContextThreadPoolExecutor(ThreadPoolExecutor):
    """在线程间保持上下文变量（如 trace_id）"""
    def submit(self, func, *args, **kwargs):
        ctx = contextvars.copy_context()      # 捕获当前上下文
        return super().submit(ctx.run, func, *args, **kwargs)  # 子线程用捕获的上下文
```

```
没有 Context 传播:                    有 Context 传播:
主线程: trace_id = "abc"             主线程: trace_id = "abc"
  ├─ 子线程1: trace_id = None ❌      ├─ 子线程1: trace_id = "abc" ✅
  └─ 子线程2: trace_id = None ❌      └─ 子线程2: trace_id = "abc" ✅
→ 并行工具追踪信息丢失!              → 所有子任务关联同一 trace
```

### 并发度调优速查

| 场景 | 推荐值 | 原因 |
|------|--------|------|
| 开发调试 | 1 | 方便逐步观察 |
| API 有速率限制 | RPM/60 | 不超限流 |
| 多工具并行 | 3-5 | 平衡速度和稳定性 |
| 批量处理 | 5-10 | 充分利用 IO 等待 |

## 5. 递归限制

### recursion_limit vs max_iterations

```python
# recursion_limit: RunnableConfig 级别，默认 25
result = agent_executor.invoke(input, config={"recursion_limit": 10})

# max_iterations: AgentExecutor 级别，默认 15
executor = AgentExecutor(agent=agent, tools=tools, max_iterations=8)
```

| 对比 | recursion_limit | max_iterations |
|------|----------------|----------------|
| 作用范围 | 整个 Runnable 链 | 仅 Agent 循环 |
| 超出行为 | 抛出 RecursionError | 优雅返回提示 |
| 推荐用途 | 通用安全防护 | 精细控制步数 |

### 常见无限循环场景与解决

```python
# 场景1：工具返回不明确 → Agent 反复调同一工具
# 场景2：Prompt 不够明确 → Agent 过度验证不停止
# 场景3：工具静默失败 → Agent 不断重试

class LoopDetector(BaseCallbackHandler):
    """检测 Agent 循环"""
    def __init__(self, warn_at: int = 5):
        self.steps = 0
        self.tool_calls = []

    def on_agent_action(self, action, *, run_id, **kwargs):
        self.steps += 1
        self.tool_calls.append(action.tool)
        if self.steps == warn_at:
            print(f"⚠️ Agent 已执行 {self.steps} 步，可能循环")
        # 检测连续 3 次调同一工具
        if len(self.tool_calls) >= 3 and len(set(self.tool_calls[-3:])) == 1:
            print(f"🔄 连续 3 次调用 {self.tool_calls[-1]}，疑似循环")
```

**解决策略速查：**

| 策略 | 实现 | 适用场景 |
|------|------|---------|
| recursion_limit | `config={"recursion_limit": 10}` | 通用防护 |
| max_iterations | `AgentExecutor(max_iterations=8)` | 精细控制 |
| 改进 Prompt | 明确告知何时停止 | 根本解决 |
| 超时控制 | `max_execution_time=30` | 兜底保护 |

## 6. 综合性能分析器

将 Token、延迟、循环三维指标整合为一个监控器：

```python
"""综合性能分析器"""
import time
from typing import Any, Dict, List
from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.outputs import LLMResult


class AgentPerformanceMonitor(BaseCallbackHandler):
    def __init__(self):
        self.start_time = time.time()
        self.total_tokens = 0
        self.prompt_tokens = 0
        self.completion_tokens = 0
        self.llm_calls = 0
        self.timings: Dict[str, dict] = {}
        self.llm_latencies: List[float] = []
        self.tool_latencies: Dict[str, List[float]] = {}
        self.agent_steps = 0
        self.tool_sequence: List[str] = []

    def on_llm_start(self, serialized, prompts, *, run_id, **kwargs):
        self.timings[str(run_id)] = {"start": time.time()}

    def on_llm_end(self, response: LLMResult, *, run_id, **kwargs):
        key = str(run_id)
        if key in self.timings:
            self.llm_latencies.append(time.time() - self.timings[key]["start"])
        self.llm_calls += 1
        if hasattr(response, "llm_output") and response.llm_output:
            u = response.llm_output.get("token_usage", {})
            self.prompt_tokens += u.get("prompt_tokens", 0)
            self.completion_tokens += u.get("completion_tokens", 0)
            self.total_tokens += u.get("total_tokens", 0)

    def on_tool_start(self, serialized, input_str, *, run_id, **kwargs):
        self.timings[str(run_id)] = {"start": time.time(), "name": serialized.get("name", "?")}

    def on_tool_end(self, output, *, run_id, **kwargs):
        key = str(run_id)
        if key in self.timings:
            name = self.timings[key]["name"]
            self.tool_latencies.setdefault(name, []).append(
                time.time() - self.timings[key]["start"]
            )

    def on_agent_action(self, action, **kwargs):
        self.agent_steps += 1
        self.tool_sequence.append(action.tool)

    def report(self):
        elapsed = time.time() - self.start_time
        llm_total = sum(self.llm_latencies)
        cost = (self.prompt_tokens * 2.5 + self.completion_tokens * 10) / 1_000_000

        print(f"\n{'='*50}")
        print(f"  📊 Agent 性能报告")
        print(f"{'='*50}")
        print(f"  ⏱️  总时间: {elapsed:.2f}s | 步数: {self.agent_steps} | LLM: {self.llm_calls}次")
        print(f"  📝 Token: {self.total_tokens} (in={self.prompt_tokens} out={self.completion_tokens})")
        if self.llm_latencies:
            print(f"  ⏳ LLM延迟: {llm_total:.2f}s ({llm_total/elapsed*100:.0f}%) "
                  f"avg={llm_total/len(self.llm_latencies):.2f}s")
        for name, times in self.tool_latencies.items():
            print(f"  🔧 {name}: {sum(times):.2f}s ({len(times)}次)")
        print(f"  💰 费用: ${cost:.4f}")

        # 自动诊断
        if self.llm_latencies and llm_total / elapsed > 0.7:
            print(f"  ⚠️ LLM 占 {llm_total/elapsed:.0%} 时间 → 考虑缓存/快模型")
        if self.agent_steps > 5:
            print(f"  ⚠️ {self.agent_steps} 步 → 优化 Prompt 减少循环")
        print(f"{'='*50}\n")
```

## 7. 常见瓶颈速查表

| 瓶颈 | 症状 | 诊断 | 解决 |
|------|------|------|------|
| LLM 延迟 | 每步 2-5s | LatencyTracker | 缓存/快模型/流式 |
| Token 浪费 | 高费用 | TokenTracker | Prompt 精简/截断返回值 |
| 工具超时 | 某步特别慢 | Tool 计时 | 超时设置/异步/缓存 |
| 无限循环 | 步数>10 | LoopDetector | recursion_limit/改 Prompt |
| Context 爆炸 | Token 暴增 | 增长模式分析 | 限制工具返回长度 |
| API 限流 | 429 错误 | 错误日志 | max_concurrency/退避 |

### 快速决策树

```
Agent 性能不好？
├─ 太慢 → LLM 慢 → 换快模型/加缓存    Tool 慢 → 超时/异步/缓存
├─ 太贵 → 循环多 → 减步数/优化 Prompt   Prompt 长 → 精简/摘要
└─ 不稳 → 随机 → 重试+退避             必然 → 调试修复
```

---

## 小结

1. **先测量再优化**：Token + 延迟 + 步数，三维指标采集
2. **Token 随循环累积增长**，是费用的主要来源
3. **瀑布图直观定位瓶颈**，通常 80% 时间在 LLM
4. **max_concurrency** 平衡速度和 API 限流
5. **recursion_limit** 是防止无限循环的安全阀（默认 25）
6. **综合监控器**让瓶颈自动浮现，优于手动排查

> **一句话：** Agent 性能优化 = 测量（Token+延迟+步数）→ 定位（最慢/最贵的步骤）→ 解决（缓存/并发/限循环）
