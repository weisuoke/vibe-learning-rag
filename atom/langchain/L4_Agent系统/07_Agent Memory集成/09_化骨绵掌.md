## 卡片 1：Checkpointer 一分钟入门

```
什么是 Checkpointer？
  → Agent 的「对话录音机」

怎么用？
  1. 创建：checkpointer = InMemorySaver()
  2. 传入：create_agent(..., checkpointer=checkpointer)
  3. 标识：config = {"configurable": {"thread_id": "xxx"}}

核心规则：
  同一 thread_id → 对话连续
  不同 thread_id → 全新对话

一句话记忆：
  checkpointer + thread_id = Agent 记忆
```

## 卡片 3：SummarizationMiddleware

```
问题：对话太长 → Token 超限 → 报错

解决：自动把旧消息压缩为摘要

配置（3 个参数记住就行）：
  SummarizationMiddleware(
      model="gpt-4.1-mini",     # 1. 用什么模型做摘要
      trigger=("tokens", 4000), # 2. 什么时候触发
      keep=("messages", 20),    # 3. 保留多少条最近的
  )

效果：
  [100条消息] → [1条摘要 + 20条最近消息]
  Token: 8000 → 2000 ✓

类比：3小时会议 → 1页会议纪要
```

## 卡片 4：Store（长期记忆）

```
Checkpointer 的局限：
  thread-001 记得你叫张三
  thread-002 不知道你是谁  ← 新会话就忘了

Store 解决这个问题：
  Store 里存：张三 = Python 开发者
  任何 thread 都能查到  ✓

核心操作：
  store.put(namespace, key, value)    # 存
  store.search(namespace, query)      # 语义搜索
  store.get(namespace, key)           # 精确获取

Checkpointer = 聊天记录（每段独立）
Store = 通讯录备注（全局共享）
```

## 卡片 5：thread_id 设计

```
thread_id = 决定谁的记忆和谁隔离

设计模式：

  模式 1：每用户一个长期对话
    thread_id = f"user-{user_id}"
    → 适合个人助手

  模式 2：每会话独立
    thread_id = f"user-{user_id}-session-{session_id}"
    → 适合客服工单

  模式 3：按场景分离
    thread_id = f"user-{user_id}-{scenario}"
    → 适合多功能助手

注意：
  ❌ 所有人用同一个 thread_id → 记忆串了
  ✅ 每人独立 thread_id → 记忆隔离
```

## 卡片 6：RemoveMessage

```
作用：删除指定的消息

什么时候用？
  - 工具返回了 3000 字的文档 → 太长了，删掉旧的
  - 对话已经结合了工具结果 → 原始结果不需要了

怎么用？
  from langchain.messages import RemoveMessage

  @after_model
  def cleanup(state, runtime):
      messages = state["messages"]
      to_remove = []
      for msg in messages[:-5]:  # 保留最近5条
          if msg.type == "tool" and len(msg.content) > 1000:
              to_remove.append(RemoveMessage(id=msg.id))
      return {"messages": to_remove} if to_remove else None

对比 SummarizationMiddleware：
  RemoveMessage → 直接删除（免费但信息丢失）
  Summarization → 压缩保留（有成本但保留语义）
```

## 卡片 7：Middleware 执行顺序

```
middleware = [A, B, C]

执行顺序取决于类型：

  before_model 类型（模型调用前）：
    A.before → B.before → C.before → 模型推理

  after_model 类型（模型调用后）：
    模型推理 → A.after → B.after → C.after

推荐顺序：
  middleware=[
      token_monitor,              # 1. 先监控
      inject_long_term_memory,    # 2. 注入长期记忆
      SummarizationMiddleware(),  # 3. 压缩旧消息
      extract_user_info,          # 4. 提取用户信息
  ]

记忆口诀：先看后注，先压后存
```

## 卡片 8：三种记忆的生命周期

```
工作记忆（Scratchpad）：
  ├── 生命周期：单次推理（几秒~几分钟）
  ├── 内容：工具调用 + 结果
  ├── 管理：Agent 循环自动
  └── 类比：做数学题的草稿纸

短期记忆（Checkpointer）：
  ├── 生命周期：单次会话（几分钟~几天）
  ├── 内容：对话消息列表
  ├── 管理：自动保存，thread_id 隔离
  └── 类比：今天跟某人的聊天记录

长期记忆（Store）：
  ├── 生命周期：跨会话（几天~永久）
  ├── 内容：用户偏好、事实
  ├── 管理：显式读写，语义搜索
  └── 类比：通讯录备注

关系：
  工作记忆 ⊂ 短期记忆 ⊂ 长期记忆
  （时间跨度越来越长，信息越来越精炼）
```

## 卡片 10：生产部署 5 要素

```
要素 1：持久化
  开发 → InMemory
  生产 → PostgresSaver
  代码只改一行 ✓

要素 2：多进程
  InMemory → 每个进程独立记忆（❌）
  PostgreSQL → 所有进程共享记忆（✓）

要素 3：Token 控制
  必须加 SummarizationMiddleware
  否则长对话必崩

要素 4：定时清理
  30天未活跃的 thread → 清理
  过大的工具输出 → 清理
  防止数据库无限增长

要素 5：监控
  跟踪：消息数量、Token 使用、数据库大小
  告警：连接池满、存储超限

口诀：持久化、多进程、控 Token、定期清、要监控
```
