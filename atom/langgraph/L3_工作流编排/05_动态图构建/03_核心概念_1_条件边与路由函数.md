# 核心概念1：条件边与路由函数

## 一句话定义

**条件边（Conditional Edges）是 LangGraph 动态图的基础构建块，通过路由函数在运行时根据状态数据决定下一步执行哪个节点。**

---

## 什么是条件边？

在静态图中，边是固定的：A 之后一定是 B。但在动态图中，A 之后去哪取决于当前的状态数据——这就是条件边。

```python
# 静态边：A → B，永远不变
graph.add_edge("A", "B")

# 条件边：A → ???，运行时才知道
graph.add_conditional_edges("A", routing_function)
```

**前端类比：** 静态边就像 `<a href="/about">` 写死的链接，条件边就像 `<Navigate to={isLoggedIn ? "/dashboard" : "/login"} />` 根据状态动态跳转。

**日常生活类比：** 静态边就像固定的公交路线——每天走同一条路。条件边就像出租车——根据乘客说的目的地（状态数据）决定走哪条路。

---

## add_conditional_edges 的工作原理

### 方法签名

```python
def add_conditional_edges(
    self,
    source: str,           # 源节点名称
    path: callable,        # 路由函数（决定去哪）
    path_map: dict = None, # 路径映射（可选，返回值 → 节点名）
) -> None:
```

[来源: sourcecode/langgraph/graph/state.py - add_conditional_edges 方法]

三个参数的含义：

| 参数 | 作用 | 必需？ |
|------|------|--------|
| `source` | 从哪个节点出发 | 是 |
| `path` | 路由函数，接收状态，返回目标节点名 | 是 |
| `path_map` | 将路由函数的返回值映射到实际节点名 | 否 |

### 最简单的用法

```python
from langgraph.graph import StateGraph, START, END
from typing import TypedDict

class State(TypedDict):
    user_type: str
    message: str

def classify_user(state: State):
    """分类节点：判断用户类型"""
    # 假设已经通过某种方式判断了用户类型
    return {"user_type": state.get("user_type", "normal")}

def handle_vip(state: State):
    return {"message": "VIP 专属服务"}

def handle_normal(state: State):
    return {"message": "普通服务"}

# ===== 路由函数 =====
def route_by_user_type(state: State) -> str:
    """根据用户类型决定下一步"""
    if state["user_type"] == "vip":
        return "vip_handler"
    return "normal_handler"

# ===== 构建图 =====
graph = StateGraph(State)
graph.add_node("classifier", classify_user)
graph.add_node("vip_handler", handle_vip)
graph.add_node("normal_handler", handle_normal)

graph.add_edge(START, "classifier")
graph.add_conditional_edges("classifier", route_by_user_type)
graph.add_edge("vip_handler", END)
graph.add_edge("normal_handler", END)

app = graph.compile()
```

执行流程：

```
START → classifier → route_by_user_type(state) → "vip_handler" 或 "normal_handler" → END
```

## path 函数（路由函数）详解

### 函数签名

路由函数的签名很简单：

```python
def routing_function(state: State) -> str | list[str]:
    """
    参数：当前状态
    返回：目标节点名（字符串）或多个目标节点名（列表）
    """
    ...
```

[来源: sourcecode/langgraph/graph/_branch.py - BranchSpec 类]

### 返回值类型

路由函数可以返回三种类型的值：

#### 类型1：返回字符串 → 路由到单个节点

```python
def route_single(state: State) -> str:
    if state["score"] > 80:
        return "pass_node"
    return "fail_node"
```

#### 类型2：返回字符串列表 → 同时路由到多个节点（并行）

```python
from typing import Sequence

def route_multiple(state: State) -> Sequence[str]:
    """同时路由到多个节点，并行执行"""
    if state["which"] == "both":
        return ["node_b", "node_c"]  # 同时去 B 和 C
    return ["node_b"]                # 只去 B
```

[来源: Context7 官方文档 - 多路返回]

这是一个很强大的特性——一个条件边可以同时激活多个下游节点。

#### 类型3：返回 Send 对象列表 → 动态并行（数量运行时决定）

```python
from langgraph.types import Send

def route_dynamic(state: State):
    """动态创建并行任务"""
    return [Send("worker", {"item": item}) for item in state["items"]]
```

### 返回值总结

| 返回类型 | 效果 | 适用场景 |
|----------|------|----------|
| `str` | 路由到单个节点 | 简单分支 |
| `list[str]` | 并行路由到多个节点 | 固定并行 |
| `list[Send]` | 动态创建并行任务 | Map-Reduce |

## 路径映射（path_map）的使用

### 为什么需要 path_map？

路由函数的返回值不一定要和节点名完全一致。`path_map` 提供了一层映射，让路由函数的返回值和实际节点名解耦。

```python
# 不用 path_map：路由函数必须返回精确的节点名
def route(state):
    return "vip_handler"  # 必须和 add_node 时的名字一模一样

# 用 path_map：路由函数返回语义化的值，path_map 做映射
def route(state):
    return "vip"  # 返回语义化的值

graph.add_conditional_edges(
    "classifier",
    route,
    {
        "vip": "vip_handler",       # "vip" → 实际节点 "vip_handler"
        "normal": "normal_handler", # "normal" → 实际节点 "normal_handler"
    }
)
```

### 实际应用：与 END 配合

`path_map` 最常见的用途是将路由结果映射到 `END`：

```python
from langgraph.graph import END

def should_continue(state: State) -> str:
    if state["done"]:
        return "end"
    return "continue"

graph.add_conditional_edges(
    "check_node",
    should_continue,
    {
        "end": END,              # "end" → 结束图执行
        "continue": "next_node", # "continue" → 继续到下一个节点
    }
)
```

[来源: Context7 官方文档 - 带路径映射的条件边]

### 工具调用场景的经典用法

LangGraph 官方的工具调用模式大量使用 path_map：

```python
from langgraph.prebuilt import tools_condition

workflow.add_conditional_edges(
    "generate_query_or_respond",
    tools_condition,           # 内置路由函数：检查是否有工具调用
    {
        "tools": "retrieve",   # 有工具调用 → 去检索节点
        END: END,              # 没有工具调用 → 结束
    },
)
```

[来源: Context7 官方文档 - LLM 驱动的路由]

**前端类比：** `path_map` 就像 React Router 的路由配置表——URL pattern 映射到组件。

```javascript
// 前端类比
const routes = {
  "/vip": VipPage,       // path_map: "vip" → "vip_handler"
  "/normal": NormalPage,  // path_map: "normal" → "normal_handler"
};
```

## 多路返回：同时路由到多个节点

### 基本用法

路由函数返回一个序列（列表），可以同时激活多个下游节点：

```python
from typing import Sequence
from typing_extensions import TypedDict, Annotated
import operator

class State(TypedDict):
    which: str
    results: Annotated[list, operator.add]  # reducer：合并并行结果

def route_bc_or_cd(state: State) -> Sequence[str]:
    """根据状态同时路由到多个节点"""
    if state["which"] == "cd":
        return ["c", "d"]  # 同时去 C 和 D
    return ["b", "c"]      # 同时去 B 和 C

graph.add_conditional_edges("a", route_bc_or_cd)
```

[来源: Context7 官方文档 - 多路返回]

### 并行结果的合并

当多个节点并行执行时，需要用 **Reducer** 来合并结果，否则后执行的节点会覆盖先执行的节点的输出：

```python
import operator
from typing import Annotated

class State(TypedDict):
    # 没有 reducer：后写入的覆盖先写入的（危险！）
    # bad_results: list

    # 有 reducer：并行结果自动合并（安全）
    results: Annotated[list, operator.add]
```

**日常生活类比：** 多路返回就像老板同时给多个部门下达任务——"市场部做调研，技术部做原型，设计部做 UI"。三个部门并行工作，最后结果汇总到一起。Reducer 就是那个汇总报告的人。

## 源码层面：BranchSpec 如何工作

### BranchSpec 的角色

当你调用 `add_conditional_edges()` 时，LangGraph 内部创建了一个 `BranchSpec` 对象：

```python
# 简化的内部流程
class BranchSpec:
    path: Runnable    # 路由函数（被包装为 Runnable）
    ends: dict | None # 路径映射（path_map）
```

[来源: sourcecode/langgraph/graph/_branch.py - BranchSpec 类]

### 执行流程

```
1. 源节点执行完毕，输出新状态
   ↓
2. BranchSpec 的 path（路由函数）被调用，传入当前状态
   ↓
3. 路由函数返回目标节点名（或列表、或 Send 对象）
   ↓
4. 如果有 path_map（ends），将返回值映射为实际节点名
   ↓
5. 目标节点被加入下一步的执行计划
   ↓
6. Pregel 引擎执行目标节点
```

### 编译时 vs 运行时

```python
# ===== 编译时发生的事 =====
graph = StateGraph(State)
graph.add_conditional_edges("A", route_fn, {"x": "B", "y": "C"})
app = graph.compile()
# 此时 BranchSpec 被创建，route_fn 被注册，但还没执行

# ===== 运行时发生的事 =====
result = app.invoke({"input": "hello"})
# 1. 节点 A 执行
# 2. route_fn(state) 被调用 → 返回 "x"
# 3. path_map 映射 "x" → "B"
# 4. 节点 B 被执行
```

## 完整代码示例：多级条件路由

```python
"""条件边实战：客服分流系统（意图路由 + 优先级路由）"""
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class CustomerState(TypedDict):
    query: str
    intent: str
    priority: str
    response: str

# ===== 节点定义 =====
def classify_intent(state: CustomerState) -> dict:
    query = state["query"].lower()
    if "退款" in query or "退货" in query:
        intent = "refund"
    elif "投诉" in query or "不满" in query:
        intent = "complaint"
    elif "怎么" in query or "如何" in query:
        intent = "question"
    else:
        intent = "general"
    print(f"[分类] '{state['query']}' → {intent}")
    return {"intent": intent}

def assess_priority(state: CustomerState) -> dict:
    query = state["query"].lower()
    priority = "high" if ("紧急" in query or "立刻" in query) else "normal"
    print(f"[优先级] {priority}")
    return {"priority": priority}

def handle_refund(state):
    return {"response": "已启动退款流程，预计 3-5 个工作日到账。"}

def handle_complaint_high(state):
    return {"response": "已转接高级客服专员，请稍候。"}

def handle_complaint_normal(state):
    return {"response": "感谢反馈，我们会在 24 小时内处理。"}

def handle_question(state):
    return {"response": "请查看帮助中心，或描述您的具体问题。"}

def handle_general(state):
    return {"response": "您好，请问有什么可以帮您？"}

# ===== 路由函数 =====
def route_by_intent(state: CustomerState) -> str:
    return {"refund": "refund", "complaint": "complaint",
            "question": "question"}.get(state["intent"], "general")

def route_by_priority(state: CustomerState) -> str:
    return "high" if state["priority"] == "high" else "normal"

# ===== 构建图 =====
graph = StateGraph(CustomerState)
graph.add_node("classify", classify_intent)
graph.add_node("assess_priority", assess_priority)
graph.add_node("refund_handler", handle_refund)
graph.add_node("complaint_high", handle_complaint_high)
graph.add_node("complaint_normal", handle_complaint_normal)
graph.add_node("question_handler", handle_question)
graph.add_node("general_handler", handle_general)

graph.add_edge(START, "classify")

# 第一级条件边：意图路由
graph.add_conditional_edges("classify", route_by_intent, {
    "refund": "refund_handler",
    "complaint": "assess_priority",   # 投诉先评估优先级
    "question": "question_handler",
    "general": "general_handler",
})

# 第二级条件边：优先级路由
graph.add_conditional_edges("assess_priority", route_by_priority, {
    "high": "complaint_high",
    "normal": "complaint_normal",
})

for node in ["refund_handler", "complaint_high", "complaint_normal",
             "question_handler", "general_handler"]:
    graph.add_edge(node, END)

app = graph.compile()

# ===== 测试 =====
for query in ["我要退款", "紧急投诉，服务太差了", "怎么修改密码"]:
    result = app.invoke({"query": query, "intent": "", "priority": "", "response": ""})
    print(f"回复: {result['response']}\n")
```

**预期输出：**

```
[分类] '我要退款' → refund
回复: 已启动退款流程，预计 3-5 个工作日到账。

[分类] '紧急投诉，服务太差了' → complaint
[优先级] high
回复: 已转接高级客服专员，请稍候。

[分类] '怎么修改密码' → question
回复: 请查看帮助中心，或描述您的具体问题。
```

## 与静态边的对比

| 维度 | 静态边 `add_edge` | 条件边 `add_conditional_edges` |
|------|-------------------|-------------------------------|
| 决策时机 | 编译时 | 运行时 |
| 目标节点 | 固定一个 | 动态选择（单个/多个/Send） |
| 路由逻辑 | 无 | 路由函数 |
| 可视化 | 实线箭头 | 虚线箭头（带条件标签） |
| 使用场景 | 顺序流程、固定管道 | 分支决策、AI 路由 |

用静态边：流程固定不变（预处理 → 推理 → 后处理）。用条件边：需要根据数据做分支决策。

## 常见陷阱

### 陷阱1：路由函数返回值与 path_map 不匹配

```python
# 错误：路由函数返回 "vip"，但 path_map 里没有 "vip"
def route(state):
    return "vip"

graph.add_conditional_edges("A", route, {
    "VIP": "vip_handler",  # 大小写不匹配！
})
# 运行时会报错：找不到 "vip" 对应的节点
```

### 陷阱2：忘记给并行分支的状态字段加 Reducer

```python
# 错误：并行节点写入同一个字段，后者覆盖前者
class State(TypedDict):
    results: list  # 没有 reducer！

# 正确：使用 Annotated + operator.add
class State(TypedDict):
    results: Annotated[list, operator.add]  # 并行结果自动合并
```

### 陷阱3：条件边和静态边混用导致意外行为

```python
# 注意：同一个源节点不要同时有静态边和条件边
# 静态边总是会执行，条件边也会执行，两者不互斥
graph.add_edge("A", "B")                          # 静态边：A → B
graph.add_conditional_edges("A", route, {...})     # 条件边：A → ???
# 结果：B 一定会执行，条件边选中的节点也会执行！
```
