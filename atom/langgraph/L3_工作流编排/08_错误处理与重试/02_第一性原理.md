# 错误处理与重试 - 第一性原理

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题。

---

## 错误处理与重试的第一性原理

### 1. 最基础的定义

**错误处理 = 程序遇到异常时的应对策略**
**重试 = 失败后自动再次尝试**

仅此而已！没有更基础的了。

把这两个概念合在一起：**当工作流中的某个步骤失败时，系统能识别错误类型，并对可恢复的错误自动重新执行。**

### 2. 为什么需要错误处理与重试？

**核心问题：分布式系统中，失败是常态而非例外。**

在 LangGraph 构建的 AI 工作流中，每个节点都可能调用外部服务：

```
用户请求 → [LLM调用] → [向量检索] → [数据库查询] → [结果生成]
                ↓            ↓             ↓
           可能超时      可能不可用      可能连接断开
```

**关键认知转变**：
- ❌ 传统思维：假设一切正常运行，出错时处理
- ✅ 分布式思维：假设一切都会失败，设计如何恢复

### 3. 错误处理与重试的三层价值

#### 价值1：系统可用性

**没有重试**：一次网络波动 = 整个工作流失败
**有重试**：自动重新尝试，用户甚至不知道发生了故障

```python
# 前端类比：fetch 请求的自动重试
async function fetchWithRetry(url, retries = 3) {
    for (let i = 0; i < retries; i++) {
        try {
            return await fetch(url);
        } catch (e) {
            if (i === retries - 1) throw e;
            await sleep(1000 * Math.pow(2, i)); // 指数退避
        }
    }
}
```

#### 价值2：故障隔离

**没有错误分类**：所有错误都一样处理
**有错误分类**：网络错误重试，编程错误立即报告

```
错误发生
  ├─ 瞬态错误（网络超时、速率限制）→ 值得重试 ✅
  └─ 永久错误（参数错误、权限不足）→ 重试无意义 ❌
```

#### 价值3：资源保护

**没有退避策略**：所有客户端同时重试 → 服务器雪崩
**有退避策略**：错开重试时间 → 给服务器恢复的喘息空间

```
日常生活类比：
- ❌ 电话占线后疯狂重拨 → 线路更拥堵
- ✅ 等一会儿再拨，每次多等一点 → 最终打通
```

### 4. 从第一性原理推导 LangGraph 重试机制

**推理链：**

```
1. AI 工作流依赖外部服务（LLM API、向量数据库、搜索引擎）
   ↓
2. 外部服务必然会遇到瞬态故障（网络波动、速率限制、临时过载）
   ↓
3. 瞬态故障的特征：稍后重试大概率会成功
   ↓
4. 因此需要自动重试机制，避免人工干预
   ↓
5. 但并非所有错误都值得重试（编程错误重试100次也不会成功）
   ↓
6. 因此需要异常分类：区分"可重试"和"不可重试"
   ↓
7. 同时重试可能加重服务器负担（重试风暴）
   ↓
8. 因此需要退避策略：间隔越来越长 + 随机抖动
   ↓
9. 重试次数必须有限制，否则可能永远卡住
   ↓
10. 重试耗尽后需要降级方案（fallback）
    ↓
11. 最终形成完整的弹性架构：
    RetryPolicy（重试）+ 异常分类 + 退避 + Fallback
```

### 5. LangGraph 的设计决策

从第一性原理出发，LangGraph 做出了以下设计决策：

| 需求 | LangGraph 的解决方案 |
|------|---------------------|
| 声明式重试 | `RetryPolicy` NamedTuple |
| 异常分类 | `retry_on` 参数（支持类型/列表/函数） |
| 退避策略 | 指数退避 + 可选 Jitter |
| 重试限制 | `max_attempts` 参数 |
| 不可重试的异常 | `GraphBubbleUp` 直接跳过重试 |
| 节点级粒度 | `add_node(retry_policy=...)` |
| 状态保护 | 每次重试前清除上次写入 |

### 6. 一句话总结第一性原理

**错误处理与重试的本质是"接受失败是常态"，通过智能分类、渐进退避和有限重试，将不可靠的外部服务封装成可靠的工作流节点。**
