# 第一性原理

> 回到事物最基本的真理，从源头思考 Partition 的本质

---

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而不是通过类比或经验。

**应用到 Partition**：
- 不是"Partition 像数据库分区"
- 而是"Partition 是什么？为什么需要它？它解决了什么根本问题？"

---

## Partition 的第一性原理

### 1. 最基础的定义

**Partition = Collection 内部的数据子集**

仅此而已！没有更基础的了。

**形式化定义**：
```
Collection C = {entity₁, entity₂, ..., entityₙ}
Partition P ⊆ C
```

**核心特征**：
- Partition 是 Collection 的子集
- 所有 Partition 的并集 = Collection
- Partition 之间可以不相交（推荐）

---

### 2. 为什么需要 Partition？

**核心问题：向量检索的性能瓶颈**

#### 问题场景

假设你有一个多租户 SaaS 应用：
- 1000 个租户
- 每个租户 10 万条向量数据
- 总数据量：1 亿条向量

**不使用 Partition 的问题**：
```python
# 搜索租户 A 的数据
results = client.search(
    collection_name="all_tenants",
    data=[query_vector],
    filter='tenant_id == "tenant_a"',  # 需要扫描 1 亿条数据
    limit=10
)
# 性能：扫描 1 亿条数据，延迟 > 1 秒
```

**问题分析**：
1. **全表扫描**：即使只需要租户 A 的数据，也要扫描所有 1 亿条数据
2. **标量过滤开销**：在向量检索后再进行标量过滤，效率低
3. **内存浪费**：所有数据都需要加载到内存

**根本原因**：
- 向量检索是计算密集型操作（O(n) 或 O(log n)）
- 数据量越大，检索越慢
- 标量过滤无法减少向量检索的计算量

---

### 3. Partition 的三层价值

#### 价值 1：减少检索范围

**原理**：将数据分区，只检索相关分区

**示例**：
```python
# 使用 Partition Key 自动分区
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True)  # 标记为 Partition Key
    ]
)

# 搜索租户 A 的数据
results = client.search(
    collection_name="all_tenants",
    data=[query_vector],
    filter='tenant_id == "tenant_a"',  # 只扫描租户 A 的分区
    limit=10
)
# 性能：只扫描 10 万条数据，延迟 < 10 毫秒
# 性能提升：100 倍
```

**价值量化**：
- 数据量减少：1 亿 → 10 万（减少 99.9%）
- 延迟降低：1 秒 → 10 毫秒（降低 99%）

#### 价值 2：数据隔离

**原理**：不同租户的数据物理或逻辑隔离

**示例**：
```python
# 手动分区：物理隔离
collection.create_partition("tenant_a")
collection.create_partition("tenant_b")

# 插入数据到不同分区
collection.insert(data_a, partition_name="tenant_a")
collection.insert(data_b, partition_name="tenant_b")

# 搜索时只搜索特定分区
results = collection.search(
    data=[query_vector],
    partition_names=["tenant_a"],  # 只搜索租户 A 的分区
    limit=10
)
```

**价值量化**：
- 数据隔离：租户 A 的数据不会被租户 B 访问
- 安全性：物理隔离提供更强的安全保障

#### 价值 3：资源优化

**原理**：只加载需要的分区，节省内存

**示例**：
```python
# 只加载租户 A 的分区
collection.load_partitions(["tenant_a"])

# 搜索租户 A 的数据（只有租户 A 的数据在内存中）
results = collection.search(
    data=[query_vector],
    partition_names=["tenant_a"],
    limit=10
)

# 释放租户 A 的分区，加载租户 B 的分区
collection.release_partitions(["tenant_a"])
collection.load_partitions(["tenant_b"])
```

**价值量化**：
- 内存节省：只加载 10 万条数据，而非 1 亿条
- 成本降低：内存使用减少 99.9%

---

### 4. 从第一性原理推导 Partition Key

**推理链**：

```
1. 前提：向量检索的性能瓶颈是数据量
   ↓
2. 推导：需要减少检索的数据量
   ↓
3. 推导：将数据分区，只检索相关分区
   ↓
4. 推导：需要一个字段来标识数据属于哪个分区
   ↓
5. 推导：这个字段就是 Partition Key
   ↓
6. 推导：Milvus 根据 Partition Key 的值自动将数据路由到不同分区
   ↓
7. 最终：Partition Key 自动分区机制
```

**Partition Key 的本质**：
- **自动路由**：根据 Partition Key 的值自动将数据分配到不同分区
- **哈希分布**：使用哈希算法将 Partition Key 值映射到 16 个物理分区
- **逻辑隔离**：虽然多个租户可能共享一个物理分区，但数据在逻辑上是分离的

**为什么是 16 个物理分区？**
- **平衡性能和管理开销**：16 个分区足以支持百万级租户，同时管理开销可控
- **哈希分布**：16 个分区可以均匀分布数据，避免数据倾斜
- **可配置**：可以根据实际需求调整分区数量（100-200 个推荐）

---

### 5. 一句话总结第一性原理

**Partition 是通过数据分区减少检索范围，从而提升性能、实现数据隔离和资源优化的机制。**

## 从第一性原理推导多租户策略

### 推理链

```
1. 前提：多租户系统需要数据隔离和性能优化
   ↓
2. 推导：需要选择合适的隔离级别
   ↓
3. 推导：隔离级别越高，可扩展性越低
   ↓
4. 推导：需要在隔离和可扩展性之间权衡
   ↓
5. 推导：四种多租户策略
   ↓
6. 最终：根据租户数量和隔离需求选择策略
```

### 四种多租户策略

| 策略 | 数据隔离 | 可扩展性 | 适用场景 |
|------|----------|----------|----------|
| Database-level | 最强 | 最低 | 少量租户（< 10） |
| Collection-level | 强 | 低 | 中等数量租户（< 100） |
| Partition-level | 中 | 中 | 中等数量租户（< 1,024） |
| Partition Key-level | 弱 | 最高 | 大量租户（百万级） |

**选择建议**：
- **少量租户**：Database-level 或 Collection-level
- **中等数量租户**：Partition-level
- **大量租户**：Partition Key-level（推荐）

## 核心洞察

### 1. Partition 的本质

**Partition 不是技术实现，而是一种思维方式**：
- 将大问题分解为小问题
- 只处理相关的数据
- 避免不必要的计算

### 2. Partition Key 的创新

**Partition Key 是 Milvus 2.6 的核心创新**：
- 自动管理：无需手动创建分区
- 高可扩展性：支持百万级租户
- 简化开发：开发者只需标记字段

### 3. 性能优化的根本

**性能优化的根本是减少计算量**：
- 不是优化算法（虽然也重要）
- 而是减少需要处理的数据量
- Partition 通过数据分区实现这一目标

## 实际应用

### 场景 1：SaaS 应用

**问题**：1000 个租户，每个租户 10 万条数据

**解决方案**：
```python
# 使用 Partition Key 自动分区
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True)
    ]
)
```

**效果**：
- 性能提升：100 倍
- 延迟降低：1 秒 → 10 毫秒
- 支持百万级租户

### 场景 2：时间序列数据

**问题**：按时间分区，只查询最近的数据

**解决方案**：
```python
# 手动分区：按月份分区
collection.create_partition("2026_01")
collection.create_partition("2026_02")

# 只加载最近的分区
collection.load_partitions(["2026_02"])

# 搜索最近的数据
results = collection.search(
    data=[query_vector],
    partition_names=["2026_02"],
    limit=10
)
```

**效果**：
- 内存节省：只加载最近的数据
- 性能提升：只检索相关时间段的数据
