# 所有权与借用 —— 反直觉点

> **一句话定位：** 从 TypeScript 转向 Rust 时，最容易掉进的 6 个思维陷阱和 4 个违背直觉但确实如此的事实。

---

## 初学者三大误区

### 误区 1："所有权太复杂了，不如用 GC"

#### 为什么人们这样想

写了三年 TypeScript，你对内存管理的全部经验就是——**不用管它**。然后来到 Rust，编译器劈头盖脸告诉你 `let copy = user;` 之后 `user` 就不能用了。你的第一反应是："GC 不好吗？"

#### 为什么是错的

所有权系统只有 **3 条规则**：

```
1. 每个值有且仅有一个所有者
2. 同一时刻只能有一个所有者
3. 所有者离开作用域，值被 drop
```

而 GC 调优呢？Java 有超过 10 种 GC 算法，每种几十个参数；Node.js 的 `--max-old-space-size` 需要根据业务调整；Go 的 GC 暂停在高频交易场景仍然是性能杀手。

**3 条确定性规则 vs 几十个需要凭经验调整的 GC 参数——哪个更简单？**

#### 正确理解

所有权是**编译时的 GC**。写代码时多想一步（"这个值归谁管"），运行时零开销。

| 维度 | GC（JS/Java/Go） | 所有权（Rust） |
|------|-----------------|---------------|
| 谁来释放？ | 运行时 GC 线程 | 编译器自动插入 `drop()` |
| 何时释放？ | 不确定（GC 说了算） | 确定（作用域结束时） |
| 运行时开销 | 有（GC 暂停、引用图维护） | 零 |
| 学习曲线 | 前期低，后期高（调优） | 前期高，后期低（规则固定） |

#### 数据支持

| 指标 | ZeroClaw (Rust) | 等价 Node.js 实现 |
|------|----------------|-------------------|
| 空载内存 | < 5 MB | 50 - 200 MB |
| 释放时机 | 确定、可预测 | 依赖 GC 启发式算法 |
| 启动速度 | 毫秒级 | 秒级（V8 预热） |

5 MB vs 200 MB，40 倍差距。在嵌入式和云原生场景，这就是"能不能部署"的区别。

### 误区 2："每次编译器报错都要 clone()"

#### 为什么人们这样做

编译器说 `name` 已经 move 了，你搜 StackOverflow 看到答案：**加 `.clone()`**。编译通过了！于是你养成了习惯——只要编译器报所有权错误，就加 `.clone()`。

#### 为什么是反模式

`.clone()` 对 `String` 和 `Vec` 意味着**深拷贝**——分配新内存，复制全部数据。就像你不会在 TypeScript 里每次传参都 `JSON.parse(JSON.stringify(obj))`，Rust 里也不该每次都 clone。

```rust
let data = load_huge_file();       // 10MB 字符串
let copy1 = data.clone();          // +10MB
let copy2 = data.clone();          // +10MB
// 40MB 内存，其中 30MB 是不必要的
```

#### 正确做法：先用 `&` 借用

90% 的情况下，函数不需要拥有数据，只需要读取它：

```rust
// 反模式：获取所有权（多余）
fn process(data: String) { println!("{}", data); }

// 正确：用不可变借用
fn process(data: &str) { println!("{}", data); }

fn main() {
    let name = String::from("ZeroClaw");
    process(&name);           // 借给 process 用
    println!("{}", name);     // name 仍然有效，零额外开销
}
```

ZeroClaw 的真实代码也遵循这个模式：

```rust
pub fn create_memory(
    config: &MemoryConfig,        // 借用，不获取所有权
    workspace_dir: &Path,         // 借用
    api_key: Option<&str>,        // 借用
) -> anyhow::Result<Box<dyn Memory>> { /* ... */ }
```

[来源: reference/source_ownership_01.md | ZeroClaw src/memory/mod.rs]

#### 但也别过度优化

**初学时用 clone 是完全可以的。** 优化优先级：先让代码编译通过（clone 没问题）→ 让逻辑正确 → 再用 `&` 或 `Arc` 替换不必要的 clone。

> **类比：** clone 就是 Rust 里的 `any`——安全，但不够精炼。先跑起来再优化。

[来源: reference/search_ownership_01.md]

### 误区 3："`&mut` 就像 TypeScript 里的普通赋值"

#### 为什么人们这样想

TypeScript 里修改对象随时随地，没有任何限制。你以为 `&mut` 只是告诉编译器"我要修改这个值"。

#### 为什么是错的

`&mut` 不仅意味着"可修改"，还意味着**独占**：

```rust
let mut data = vec![1, 2, 3];
let r1 = &data;           // 不可变借用 —— OK
let r2 = &data;           // 多个 & 共存 —— OK
let w = &mut data;         // 可变借用 —— r1/r2 必须已结束！
// println!("{:?}", r1);   // 编译错误！& 和 &mut 不能共存

let mut data = vec![1, 2, 3];
let w1 = &mut data;
let w2 = &mut data;        // 编译错误！两个 &mut 不能共存
```

#### 正确理解：数据库锁的类比

`&mut` 像**排它锁（Exclusive Lock）**，`&` 像**共享锁（Shared Lock）**：

| 操作 | Rust | 数据库 | TypeScript |
|------|------|--------|-----------|
| 多个读者同时读 | 多个 `&T` 共存 | 多个共享锁 | 随便读 |
| 一个写者独占写 | 一个 `&mut T` | 排它锁 | 随便写（危险!） |
| 读者和写者同时 | 编译错误 | 死锁/串行化 | 可以但可能出 bug |

**为什么这么严格？** "同时读写同一数据"是几乎所有并发 bug 的根源。TypeScript 允许你这么做，但 bug 要等到运行时才暴露。Rust 在编译时就拦住了。

[来源: reference/context7_rust_book_01.md]

## 进阶三大误区

> 以下三个误区来自 pretzelhammer 的 [Common Rust Lifetime Misconceptions](https://github.com/pretzelhammer/rust-blog/blob/master/posts/common-rust-lifetime-misconceptions.md)，Rust 社区公认的经典总结。

### 误区 4："`T: 'static` 意味着值永远存在"

#### 错误理解

看到 `'static` 就想到字符串字面量 `let s: &'static str = "hello";`，以为 `T: 'static` 意味着"T 必须永远存在"。

#### 正确理解

`T: 'static` 的真正含义是：**T 不包含任何非 `'static` 的引用。** 它包括所有拥有型类型：

```rust
// 以下全部满足 T: 'static
let a: i32 = 42;                          // 基础类型
let b: String = String::from("hello");     // 拥有型字符串
let c: Vec<u8> = vec![1, 2, 3];            // 拥有型向量
// String 满足 'static 不是因为它永远存在，而是因为它不借用外部数据
```

#### 关键区分

| 概念 | 含义 | 例子 |
|------|------|------|
| `&'static T` | 引用，指向的数据活到程序结束 | `"hello"`（字符串字面量） |
| `T: 'static` | 类型内部不含非 `'static` 引用 | `String`、`Vec<i32>`、`i32` |

```rust
fn spawn_thread<T: Send + 'static>(val: T) {
    std::thread::spawn(move || {
        // val 被移动到新线程，不借用主线程数据，安全
    });
}
spawn_thread(String::from("hello"));   // OK：String 满足 'static
// spawn_thread(&local_string);        // 可能编译错误：引用不满足 'static
```

> **前端类比：** `T: 'static` 就像 `Serializable`——能序列化的对象可以发送到 Web Worker；包含 DOM 引用的对象不能。

[来源: reference/fetch_lifetime_misconceptions_01.md | 误解 2]

### 误区 5："如果编译通过，生命周期注解就是正确的"

#### 错误理解

"编译器这么严格，能通过编译的代码肯定是对的吧？"

#### 正确理解

编译器只验证**内存安全**，不验证**你的意图**。

#### 经典案例：ByteIter

```rust
struct ByteIter<'remainder> {
    remainder: &'remainder [u8],
}

impl<'remainder> ByteIter<'remainder> {
    // 省略后编译器推断为：fn next<'a>(&'a mut self) -> Option<&'a u8>
    // 返回值绑定到 &mut self 而非底层切片 'remainder —— 语义错误！
    fn next(&mut self) -> Option<&u8> {
        if self.remainder.is_empty() { return None; }
        let byte = &self.remainder[0];
        self.remainder = &self.remainder[1..];
        Some(byte)
    }
}

// 后果：不能同时持有两个返回的字节
let mut iter = ByteIter { remainder: b"hello" };
let first = iter.next().unwrap();
let second = iter.next().unwrap();  // 编译错误：first 还占着 &mut self
```

正确写法——显式标注生命周期：

```rust
impl<'remainder> ByteIter<'remainder> {
    fn next(&mut self) -> Option<&'remainder u8> {
        //                         ^^^^^^^^^^ 绑定到切片生命周期，而非 self
        if self.remainder.is_empty() { return None; }
        let byte = &self.remainder[0];
        self.remainder = &self.remainder[1..];
        Some(byte)
    }
}
// 现在 first 和 second 可以同时存在了
```

> **前端类比：** TypeScript 类型推断有时推出"能编译但不是你想要的类型"，比如推断出 `string` 而你需要 `"success" | "error"`。编译通过不代表语义正确。

[来源: reference/fetch_lifetime_misconceptions_01.md | 误解 5]

### 误区 6："闭包和函数遵循相同的生命周期省略规则"

#### 错误理解

函数能省略生命周期，闭包应该也一样：

```rust
fn function(x: &i32) -> &i32 { x }     // OK
let closure = |x: &i32| -> &i32 { x };  // 编译错误！
```

#### 正确理解

这不是 bug，是**历史遗留的设计差异**。闭包中输入和输出引用被赋予不同的生命周期。

#### 解决方案

```rust
// 方案 1：通过 trait bound 让编译器正确推断
fn apply<F: Fn(&i32) -> &i32>(f: F, x: &i32) -> &i32 { f(x) }
let result = apply(|x| x, &42);

// 方案 2：写成函数再引用
fn identity(x: &i32) -> &i32 { x }
let refs: Vec<&i32> = items.iter().map(identity).collect();

// 方案 3：避免在闭包中返回引用
items.iter().for_each(|x| println!("{}", x));
```

> **前端类比：** 就像箭头函数和 `function` 声明对 `this` 的绑定规则不同——语法相似，底层行为有区别。

[来源: reference/fetch_lifetime_misconceptions_01.md | 误解 10]

## 反直觉但真实的 4 个事实

### 事实 1：`move` 不移动数据

```rust
let s1 = String::from("hello");
// 栈上：{ ptr: 0x1234, len: 5, cap: 5 }，堆上 0x1234: [h,e,l,l,o]

let s2 = s1;
// 栈上：{ ptr: 0x1234, len: 5, cap: 5 } ← 复制了 24 字节栈数据
// 堆上 0x1234: [h,e,l,l,o]              ← 堆数据完全没动！
// s1 被编译器标记为无效
```

**"move" 实际上只是：** (1) 复制栈上元数据（24 字节） (2) 标记原变量无效 (3) 堆数据纹丝不动。编译器甚至可能把栈上的复制也优化掉。

### 事实 2：`Arc::clone()` 很便宜

```rust
let config = Arc::new(Config::load());   // Config 有 100KB
let config2 = Arc::clone(&config);
// 只是原子计数器 +1（一条 CPU 指令），100KB 数据零拷贝
```

| 操作 | 成本 |
|------|------|
| `String::clone()` | 分配新堆内存 + 复制全部字符 |
| `Vec::clone()` | 分配新堆内存 + 复制全部元素 |
| `Arc::clone()` | 原子计数器 +1（约 1 纳秒） |

ZeroClaw 中所有工具都通过 `Arc::clone()` 共享 `SecurityPolicy`，开销几乎为零。

> **社区建议：** 写 `Arc::clone(&config)` 而非 `config.clone()`，明确告诉读者这不是深拷贝。

[来源: reference/source_ownership_01.md]

### 事实 3：`String` 是 `Vec<u8>` 的特化

```rust
// String 的源码定义（简化）
pub struct String { vec: Vec<u8> }  // 就是一个字节向量！

// 比 Vec<u8> 多了什么？UTF-8 验证 + 字符串方法。仅此而已。
let s = String::from("hello");
let bytes: Vec<u8> = s.into_bytes();        // 零开销转换
let s2 = String::from_utf8(bytes).unwrap();  // 验证 UTF-8
```

理解这一点后就不再困惑：`String::len()` 返回字节数而非字符数；索引按字节而非字符。

### 事实 4：`Box<T>` 和 `T` 的运行时行为可能完全相同

LLVM 的逃逸分析可以将未逃逸的 `Box<T>` 堆分配降级为栈分配。`Box<T>` 是 Rust 中唯一获得编译器特殊优化的智能指针。

`Box` 的真正价值：(1) Trait 对象 `Box<dyn Tool>` (2) 递归类型 (3) 大型数据避免撑爆栈。

[来源: reference/source_ownership_01.md]

## 误区速查表

| 误区 | 直觉反应 | 正确理解 |
|------|---------|---------|
| "所有权太复杂" | 不如用 GC | 只有 3 条规则，比 GC 调优简单 |
| "编译器报错就 clone" | clone 是万能药 | 先用 `&` 借用，clone 是最后手段 |
| "`&mut` = 普通赋值" | 随时修改没限制 | `&mut` 是排它锁，独占访问 |
| "`T: 'static` = 永远存在" | 只能是字面量 | 包括所有拥有型类型 |
| "编译通过 = 正确" | 编译器比我聪明 | 编译器只验证安全，不验证意图 |
| "闭包 = 函数" | 省略规则相同 | 闭包的省略规则不同（历史原因） |
| "`move` = 搬运数据" | 堆数据被复制 | 只复制栈上元数据，堆数据不动 |
| "`Arc::clone()` = 深拷贝" | 很昂贵 | 只增加原子计数器，约 1 纳秒 |

## 从 TypeScript 到 Rust 的思维转换清单

```
TypeScript 思维                    Rust 思维
─────────────────────────────     ─────────────────────────────
"GC 会处理内存"               →  "谁拥有这个值？什么时候释放？"
"随便赋值，对象是共享的"        →  "赋值 = 转移所有权，用 & 借用"
"随时修改对象"                 →  "&mut 是独占的，一个人用完其他人才能用"
"不用想生命周期"               →  "引用不能活过被引用的数据"
"编译器说啥就改啥"             →  "编译器建议不一定是最优解"
"clone 解决一切"              →  "先想 & 借用，再想 Arc 共享"
```
