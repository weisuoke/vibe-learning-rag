## 卡片 1：三种工具创建方式速查

**一句话：** LangChain 提供三种工具创建方式，从简单到复杂依次是 @tool 装饰器、StructuredTool、BaseTool 类继承。

**速查表：**

| 方式 | 代码模板 | 适用场景 | 优势 | 劣势 |
|------|----------|----------|------|------|
| **@tool** | `@tool`<br>`def func(x: str) -> str:` | 简单无状态工具 | 最简洁 | 无状态管理 |
| **StructuredTool** | `StructuredTool.from_function(`<br>`  func=my_func,`<br>`  args_schema=MySchema)` | 包装现有 API | 显式控制 | 稍显冗长 |
| **BaseTool** | `class MyTool(BaseTool):`<br>`  def _run(self, ...):` | 复杂状态管理 | 最大控制权 | 学习曲线高 |

**应用：** 90% 的场景使用 @tool，需要状态管理时才考虑 BaseTool。

## 卡片 2：@tool 装饰器语法速查

**一句话：** @tool 装饰器有三种使用模式：无参、带参、函数调用，支持自动 schema 推断和 docstring 解析。

**语法速查：**

```python
# 模式 1：无参装饰器（最简单）
@tool
def simple_tool(query: str) -> str:
    """Tool description."""
    return f"Result: {query}"

# 模式 2：带参装饰器（自定义配置）
@tool(
    "custom_name",
    description="Custom description",
    parse_docstring=True,
    response_format="content_and_artifact",
    handle_tool_error=True,
)
def advanced_tool(query: str) -> str:
    """Tool with custom config."""
    return f"Result: {query}"

# 模式 3：函数调用模式（动态创建）
def existing_function(query: str) -> str:
    return f"Result: {query}"

tool_instance = tool("my_tool", description="...")(existing_function)
```

**关键参数：**
- `name_or_callable`: 工具名称或被装饰的函数
- `description`: 工具描述（覆盖 docstring）
- `parse_docstring`: 是否解析 Google-style docstring
- `args_schema`: 显式指定输入 schema
- `response_format`: "content" 或 "content_and_artifact"
- `handle_tool_error`: 错误处理策略

**应用：** 优先使用模式 1，需要自定义配置时使用模式 2。

## 卡片 3：Schema 定义速查

**一句话：** 使用 Pydantic BaseModel 定义工具输入 schema，通过 Field 添加描述和约束。

**常用约束速查：**

```python
from pydantic import BaseModel, Field
from typing import Literal, List, Optional

class ToolInput(BaseModel):
    # 必需字段
    query: str = Field(description="搜索关键词")

    # 数值约束
    limit: int = Field(default=10, ge=1, le=100, description="结果数量")

    # 字符串长度约束
    username: str = Field(min_length=3, max_length=20, description="用户名")

    # 正则表达式约束
    email: str = Field(pattern=r'^[\w\.-]+@[\w\.-]+\.\w+$', description="邮箱")

    # 枚举值约束
    sort_by: Literal["relevance", "date"] = Field(default="relevance")

    # 可选字段
    filters: Optional[List[str]] = Field(default=None, description="过滤条件")

    # 可变默认值（使用 default_factory）
    tags: List[str] = Field(default_factory=list, description="标签列表")
```

**Field 参数速查：**
- `description`: 字段描述（LLM 会读取）
- `default`: 默认值
- `default_factory`: 默认值工厂函数（用于可变对象）
- `ge/le/gt/lt`: 数值范围约束
- `min_length/max_length`: 字符串/列表长度约束
- `pattern`: 正则表达式约束
- `min_items/max_items`: 列表元素数量约束

**应用：** 始终使用 Field 提供详细描述，使用 Literal 限制枚举值。

## 卡片 4：参数注入机制速查

**一句话：** 使用 InjectedToolArg 标记参数，LangChain 会自动注入上下文信息，不会出现在 JSON Schema 中。

**注入类型速查：**

```python
from langchain_core.tools import tool, InjectedToolArg, InjectedToolCallId
from langchain_core.runnables import RunnableConfig
from langchain_core.callbacks import CallbackManagerForToolRun
from typing import Annotated, Optional

@tool
def my_tool(
    # 业务参数（LLM 传递）
    query: str,

    # 注入参数（系统自动注入）
    config: Annotated[RunnableConfig, InjectedToolArg] = None,
    tool_call_id: Annotated[str, InjectedToolCallId] = None,
    callbacks: Annotated[Optional[CallbackManagerForToolRun], InjectedToolArg] = None,
) -> str:
    # 从 config 中提取上下文
    user_id = config.get("configurable", {}).get("user_id")
    session_id = config.get("configurable", {}).get("session_id")

    # 使用 tool_call_id 记录日志
    logger.info(f"[{tool_call_id}] User {user_id} query: {query}")

    return f"Result for {query}"
```

**使用场景：**
- **RunnableConfig**: 访问用户 ID、会话 ID、元数据
- **InjectedToolCallId**: 获取工具调用 ID，用于追踪和日志
- **Callbacks**: 访问回调处理器，用于流式输出

**应用：** 多租户系统中自动注入 tenant_id，确保数据隔离。

## 卡片 5：异步工具速查

**一句话：** 异步工具通过 async/await 实现非阻塞 I/O，在 Agent 系统中可以并发执行多个工具，显著提升性能。

**三种实现方式：**

```python
# 方式 1：@tool + async 函数（推荐）
@tool
async def async_tool(query: str) -> str:
    async with httpx.AsyncClient() as client:
        response = await client.get(f"https://api.example.com?q={query}")
        return response.text

# 方式 2：StructuredTool + coroutine 参数
async def async_func(query: str) -> str:
    # 异步实现
    pass

tool = StructuredTool.from_function(
    coroutine=async_func,  # 使用 coroutine 而非 func
    name="my_tool",
    args_schema=MySchema,
)

# 方式 3：BaseTool + _arun 方法
class MyTool(BaseTool):
    def _run(self, query: str, config=None) -> str:
        # 同步版本（回退方案）
        pass

    async def _arun(self, query: str, config=None) -> str:
        # 异步版本（推荐）
        pass
```

**并发执行：**

```python
import asyncio

# 串行执行（慢）
result1 = await tool1.ainvoke({"query": "test1"})
result2 = await tool2.ainvoke({"query": "test2"})

# 并发执行（快）
results = await asyncio.gather(
    tool1.ainvoke({"query": "test1"}),
    tool2.ainvoke({"query": "test2"}),
)
```

**应用：** HTTP API 调用、数据库查询、向量检索都应该使用异步工具。

## 卡片 6：响应格式控制速查

**一句话：** content_and_artifact 响应格式允许工具返回摘要（给 LLM）和完整数据（给程序），节省 tokens 和成本。

**两种格式对比：**

```python
# 格式 1：content（默认）
@tool
def simple_tool(query: str) -> str:
    """返回字符串，LLM 直接看到"""
    return "Result: ..."

# 格式 2：content_and_artifact（推荐）
@tool(response_format="content_and_artifact")
def advanced_tool(query: str) -> Tuple[str, Dict[str, Any]]:
    """返回 (摘要, 完整数据)"""
    summary = "Found 100 results"  # 给 LLM 看（10 tokens）
    artifact = {"results": [...]}  # 给程序用（不占 LLM tokens）
    return summary, artifact
```

**使用示例：**

```python
# 调用工具
message = advanced_tool.invoke({"query": "test"})

# 访问结果
print(message.content)    # "Found 100 results"
print(message.artifact)   # {"results": [...]}
```

**成本对比（GPT-4 @ $0.03/1K tokens）：**
- content 模式：20K tokens × $0.03 = **$0.60/次**
- content_and_artifact 模式：10 tokens × $0.03 = **$0.0003/次**
- **节省 99.95%**

**应用：** 搜索工具、数据库查询、RAG 检索都应该使用 content_and_artifact。

## 卡片 7：错误处理速查

**一句话：** ToolException 标记可恢复错误，不会中断 Agent 循环；普通异常标记致命错误，会立即停止 Agent。

**错误分类决策树：**

```
工具执行失败
    ↓
是否可以恢复？
    ├─ 否 → 抛出普通异常（中断 Agent）
    │       例如：API 密钥未配置、权限不足
    │
    └─ 是 → 抛出 ToolException（不中断）
            例如：API 临时不可用、数据不存在
                ↓
            需要定制错误消息？
                ├─ 否 → handle_tool_error=True
                └─ 是 → handle_tool_error=custom_func
```

**三种错误处理策略：**

```python
from langchain.tools import tool, ToolException

# 策略 1：默认行为（传播异常）
@tool
def tool1(param: str) -> str:
    raise ValueError("Fatal error")  # 中断 Agent

# 策略 2：简单捕获
@tool(handle_tool_error=True)
def tool2(param: str) -> str:
    raise ValueError("Error")  # 返回 "Error: ..."

# 策略 3：自定义处理
def custom_handler(error: Exception) -> str:
    if isinstance(error, requests.Timeout):
        return "API 超时，请稍后重试"
    elif isinstance(error, requests.HTTPError):
        return f"API 错误 {error.response.status_code}"
    return f"未知错误: {str(error)}"

@tool(handle_tool_error=custom_handler)
def tool3(param: str) -> str:
    # 所有异常都会被 custom_handler 处理
    pass
```

**错误消息编写技巧：**
- ❌ 不好：`raise ToolException("Error")`
- ✅ 好：`raise ToolException(f"未找到城市 '{location}'，请检查拼写或使用英文名称")`
  - 包含：问题描述 + 上下文 + 解决建议

**应用：** 生产环境使用策略 3，提供定制化错误消息。

## 卡片 8：自定义验证器速查

**一句话：** 使用 Pydantic 的 @validator 装饰器实现复杂的参数验证逻辑。

**验证器模板：**

```python
from pydantic import BaseModel, Field, validator

class ToolInput(BaseModel):
    query: str = Field(description="搜索查询")
    tags: List[str] = Field(default_factory=list)
    min_score: float = Field(default=0.0, ge=0.0, le=1.0)

    @validator('query')
    def validate_query(cls, v):
        """验证查询字符串"""
        if len(v.strip()) == 0:
            raise ValueError("查询不能为空")
        if len(v) > 500:
            raise ValueError("查询过长（最多 500 字符）")
        return v.strip()

    @validator('tags')
    def validate_tags(cls, v):
        """验证标签列表"""
        if len(v) > 10:
            raise ValueError("标签过多（最多 10 个）")
        # 去重并转小写
        return list(set(tag.lower() for tag in v))

    @validator('min_score')
    def validate_score(cls, v, values):
        """验证分数范围（可以访问其他字段）"""
        if v < 0 or v > 1:
            raise ValueError("分数必须在 0-1 之间")
        return v
```

**常见验证场景：**
- 字符串清理（去除空格、转小写）
- 列表去重
- 白名单验证
- 跨字段验证（如日期范围）
- 业务规则验证

**应用：** 数据库查询工具中验证表名白名单，防止 SQL 注入。

## 卡片 9：工具测试速查

**一句话：** 使用 pytest 测试工具的正常场景、错误场景和边界条件。

**测试模板：**

```python
import pytest
from langchain.tools import ToolException

def test_tool_success():
    """测试成功场景"""
    result = my_tool.invoke({"query": "test"})
    assert "Result" in result
    assert isinstance(result, str)

def test_tool_error_handling():
    """测试错误处理"""
    with pytest.raises(ToolException) as exc_info:
        my_tool.invoke({"query": "invalid"})
    assert "不在白名单中" in str(exc_info.value)

def test_tool_schema():
    """测试 schema 生成"""
    schema = my_tool.args_schema.model_json_schema()
    assert "query" in schema["properties"]
    assert schema["properties"]["query"]["type"] == "string"

@pytest.mark.asyncio
async def test_async_tool():
    """测试异步工具"""
    result = await async_tool.ainvoke({"query": "test"})
    assert result is not None

def test_tool_with_config():
    """测试参数注入"""
    result = my_tool.invoke(
        {"query": "test"},
        config={"configurable": {"user_id": "user123"}}
    )
    assert "user123" in result
```

**测试覆盖清单：**
- [ ] 正常输入场景
- [ ] 边界值测试（最小值、最大值）
- [ ] 错误输入场景
- [ ] Schema 验证
- [ ] 异步调用（如果适用）
- [ ] 参数注入（如果适用）
- [ ] 响应格式（content vs content_and_artifact）

**应用：** 每个工具至少编写 3 个测试用例（成功、失败、边界）。

## 卡片 10：最佳实践速查

**一句话：** 遵循最佳实践可以提高工具的可靠性、性能和可维护性。

**最佳实践清单：**

**1. 工具创建**
- ✅ 优先使用 @tool 装饰器
- ✅ 启用 parse_docstring=True 确保参数描述传递给 LLM
- ✅ 使用 Literal 限制枚举值
- ✅ 提供合理的默认值

**2. Schema 设计**
- ✅ 使用 Field() 提供详细描述
- ✅ 使用 default_factory 处理可变默认值
- ✅ 参数名称与函数签名一致
- ✅ 保持 schema 简单清晰

**3. 异步支持**
- ✅ I/O 密集型工具使用异步实现
- ✅ 使用 httpx 而非 requests
- ✅ 复用连接池（在 BaseTool 中管理）
- ✅ 设置超时避免无限等待

**4. 响应格式**
- ✅ 大数据返回使用 content_and_artifact
- ✅ 摘要简洁（< 200 tokens）
- ✅ Artifact 包含完整数据和元数据
- ✅ 错误信息在两个部分都提供

**5. 错误处理**
- ✅ 可恢复错误使用 ToolException
- ✅ 不可恢复错误使用普通异常
- ✅ 错误消息包含问题描述 + 上下文 + 解决建议
- ✅ 记录详细错误日志（包括堆栈）

**6. 参数注入**
- ✅ 使用 InjectedToolArg 标记注入参数
- ✅ 始终提供默认值（= None）
- ✅ 安全地访问嵌套字段（使用 .get()）
- ✅ 不要在 docstring 中提及注入参数

**7. 测试**
- ✅ 编写单元测试覆盖正常和错误场景
- ✅ 测试 schema 生成
- ✅ 测试异步调用（如果适用）
- ✅ 测试参数注入（如果适用）

**8. 性能优化**
- ✅ 使用异步工具提升并发性能
- ✅ 使用 content_and_artifact 节省 tokens
- ✅ 缓存工具结果（如果适用）
- ✅ 设置合理的超时时间

**9. 安全性**
- ✅ 参数白名单验证
- ✅ 防止路径遍历攻击
- ✅ 使用参数化查询防止 SQL 注入
- ✅ 不要在错误消息中泄露敏感信息

**10. 文档**
- ✅ 编写清晰的 docstring
- ✅ 提供使用示例
- ✅ 说明适用场景和限制
- ✅ 记录依赖的环境变量

**应用：** 将此清单作为代码审查的检查项。
