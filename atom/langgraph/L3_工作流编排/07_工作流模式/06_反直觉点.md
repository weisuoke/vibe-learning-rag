# 07_工作流模式 - 反直觉点

> LangGraph L3_工作流编排 | 维度6

---

## 误区1："工作流越复杂越好" ❌

### 为什么错？

很多人学了六大模式后，恨不得每个项目都用上编排者-工作者 + Map-Reduce + 评估者-优化者的组合。实际上，**大多数场景用顺序链式就够了**。

LangGraph 官方文档明确指出：

> "Start with the simplest pattern that solves your problem."

### 为什么人们容易这样错？

学了新工具就想用——这是"锤子综合症"（手里有锤子，看什么都像钉子）。复杂模式看起来更"高级"，但每增加一层复杂度，就增加了调试难度、延迟和成本。

### 正确理解

```python
# ❌ 过度设计：简单的 RAG 查询用了编排者-工作者模式
@entrypoint()
def over_engineered_rag(query: str):
    subtasks = orchestrator(query).result()  # 不需要！
    results = [worker(t) for t in subtasks]  # 不需要！
    return synthesizer([r.result() for r in results]).result()

# ✅ 简单直接：顺序链式就够了
@entrypoint()
def simple_rag(query: str):
    docs = retrieve(query).result()
    answer = generate(query, docs).result()
    return answer
```

**选择模式的原则**：能用简单模式解决的，绝不用复杂模式。只有当简单模式明显不够时，才升级。

## 误区2："并行一定比顺序快" ❌

### 为什么错？

并行模式确实能降低总延迟，但有两个常被忽略的成本：

1. **状态合并开销**：Reducer 需要处理并发写入
2. **API 限流**：同时发 10 个 LLM 请求，很可能触发 rate limit

```python
# 看起来更快，但可能更慢
# 10 个并行 LLM 调用 → 触发 rate limit → 部分请求被限流 → 重试 → 总时间反而更长

def fan_out(state):
    return [Send("worker", {"item": i}) for i in state["items"]]  # 100个并行任务！
```

### 为什么人们容易这样错？

直觉上"同时做"一定比"排队做"快。但这忽略了资源竞争和外部限制。就像高速公路——车少时并行快，车多时反而堵车。

### 正确理解

```python
# ✅ 控制并行度，分批处理
def fan_out_batched(state):
    items = state["items"]
    batch_size = 5  # 每批最多5个并行
    # 分批 Send，而不是一次性全部 Send
    batch = items[:batch_size]
    return [Send("worker", {"item": i}) for i in batch]
```

**经验法则**：并行度控制在 3-5 个 LLM 调用以内，超过就分批。

## 误区3："路由模式需要 LLM 做路由" ❌

### 为什么错？

很多人以为路由模式必须用 LLM 的 structured output 来做分类决策。实际上，**简单的规则路由往往更好**：

- 更快（省一次 LLM 调用）
- 更可靠（规则不会"幻觉"）
- 更便宜（零 token 消耗）

### 为什么人们容易这样错？

LangGraph 官方示例用了 LLM 路由，给人一种"路由就该用 LLM"的印象。但官方示例是为了展示能力，不是最佳实践。

### 正确理解

```python
# ❌ 杀鸡用牛刀：用 LLM 做简单分类
def llm_router(state):
    route = llm.with_structured_output(Route).invoke(state["input"])
    return route.step  # 一次 LLM 调用，200ms + 费用

# ✅ 规则路由：简单场景用规则
def rule_router(state):
    text = state["input"]
    if text.startswith("/help"):
        return "help_handler"
    if "投诉" in text or "退款" in text:
        return "complaint_handler"
    return "general_handler"  # 0ms，零成本

# ✅ LLM 路由：只在规则搞不定时用
def smart_router(state):
    # 先试规则
    if text.startswith("/"):
        return command_router(state)
    # 规则搞不定，再用 LLM
    return llm_router(state)
```

**原则**：规则能搞定的用规则，规则搞不定的才用 LLM。

## 误区4："评估者-优化者模式总能提升质量" ❌

### 为什么错？

评估者-优化者模式的效果取决于**评估器的质量**。如果评估器本身不靠谱，循环再多次也没用——甚至可能越改越差。

```
生成好笑话 → 评估器误判"不好笑" → 带错误反馈修改 → 改成不好笑的 → 评估器说"好笑了" → 输出垃圾
```

### 为什么人们容易这样错？

"迭代改进"听起来很合理——多改几次总会更好吧？但这假设了评估器是准确的。用同一个 LLM 既生成又评估，容易陷入"自我欺骗"。

### 正确理解

```python
# ❌ 用同一个模型生成和评估
generator = ChatOpenAI(model="gpt-4o-mini")
evaluator = ChatOpenAI(model="gpt-4o-mini")  # 同一个模型！

# ✅ 用更强的模型评估，或用确定性指标
evaluator = ChatOpenAI(model="gpt-4o")  # 更强的模型做评估

# ✅✅ 最好用确定性评估（不依赖 LLM）
def evaluate(state):
    joke = state["joke"]
    # 确定性检查：长度、格式、关键词
    has_punchline = "？" in joke or "！" in joke
    is_long_enough = len(joke) > 20
    if has_punchline and is_long_enough:
        return {"grade": "pass"}
    return {"grade": "fail", "feedback": "缺少反转或太短"}
```

**原则**：评估器要比生成器更可靠。能用确定性指标就不用 LLM 评估。

## 误区5："Send API 和普通并行是一回事" ❌

### 为什么错？

它们看起来都是"多个节点并行执行"，但底层机制完全不同：

| 维度 | 普通并行 | Send API |
|------|---------|----------|
| 分支数量 | 编译时确定 | 运行时动态 |
| Worker 状态 | 共享全局 State | 每个 Worker 独立输入 |
| 任务类型 | PULL（被通道触发） | PUSH（被 Send 推送） |
| 图结构 | 固定拓扑 | 动态拓扑 |

### 为什么人们容易这样错？

执行效果看起来一样——都是多个节点同时跑。但"固定3个并行节点"和"根据数据动态创建N个并行任务"是本质不同的。

### 正确理解

```python
# 普通并行：图结构里写死了3个分支
builder.add_edge("a", "b")  # 永远是3个
builder.add_edge("a", "c")
builder.add_edge("a", "d")

# Send API：运行时决定几个分支
def fan_out(state):
    # 可能是3个，可能是30个，取决于数据
    return [Send("worker", {"item": i}) for i in state["items"]]
```

**选择标准**：分支数量固定 → 普通并行；分支数量动态 → Send API。

## 反直觉总结

| 直觉 | 现实 |
|------|------|
| 复杂模式更强大 | 简单模式解决大多数问题 |
| 并行一定更快 | 受限于 API 限流和资源竞争 |
| 路由必须用 LLM | 规则路由更快更可靠 |
| 迭代一定能改进 | 取决于评估器质量 |
| Send 和并行一样 | 底层机制和适用场景不同 |
