# Agent最佳实践 - 核心概念 4：上下文管理模式

> 上下文管理 = 对话摘要（SummarizationMiddleware）+ 上下文编辑（ContextEditingMiddleware）+ 中间步骤裁剪（trim_intermediate_steps）三重手段，让 Agent 在长对话中保持高效推理且不超出上下文窗口。

---

## 为什么 Agent 需要上下文管理？

Agent 在多步推理中面临一个物理限制：**LLM 的上下文窗口是有限的**。

| 模型 | 上下文窗口 | 约等于 |
|------|-----------|--------|
| GPT-4.1 | 128K Token | ~96,000 个汉字 |
| Claude 3.5 Sonnet | 200K Token | ~150,000 个汉字 |

听起来很大？但 Agent 的上下文消耗**远超**普通聊天：

```
普通聊天（1 轮）：
  用户消息 + LLM 回复 ≈ 500 Token

Agent 执行（10 步循环）：
  System Prompt:          500 Token
  用户消息:               200 Token
  LLM 推理 #1:           300 Token
  工具调用 #1 + 结果:     2,000 Token  ← 工具返回的数据很长！
  LLM 推理 #2:           300 Token
  工具调用 #2 + 结果:     3,000 Token
  ...
  LLM 推理 #10:          300 Token
  工具调用 #10 + 结果:    5,000 Token
  ────────────────────────────────
  总计 ≈ 30,000+ Token     ← 单次执行就消耗大量上下文！

多轮对话（5 轮 Agent 执行）：
  30,000 × 5 = 150,000 Token  ← 已经超出 GPT-4.1 的窗口！
```

**不做上下文管理的后果：**

```
┌─────────────────────────────────────────┐
│ 上下文窗口 (128K Token)                    │
│                                           │
│ ████████████████████████████████████████ │ ← 填满了历史消息
│ ████████████████████████████████████████ │
│ ████████████████████████████████████████ │
│ ████████████████████████ ← 没空间了！      │
│                                           │
│ 后果 1: LLM 报错 → Agent 崩溃            │
│ 后果 2: LLM 性能急剧下降（注意力稀释）     │
│ 后果 3: Token 费用飙升（按 Token 计费）    │
└─────────────────────────────────────────┘
```

## 第一重手段：SummarizationMiddleware（对话摘要）

### 核心机制

当对话历史超过阈值时，自动将旧消息压缩为摘要，释放上下文空间。

```python
from langchain.agents.middleware import SummarizationMiddleware

summarization = SummarizationMiddleware(
    trigger=("tokens", 4000),    # 触发条件：消息超过 4000 Token
    keep=5,                      # 保留最近 5 条消息不摘要
)
```

### 三种触发条件

| 触发类型 | 配置示例 | 含义 |
|---------|---------|------|
| `"tokens"` | `trigger=("tokens", 4000)` | 消息总 Token 数超过 4000 时触发 |
| `"messages"` | `trigger=("messages", 20)` | 消息数量超过 20 条时触发 |
| `"fraction"` | `trigger=("fraction", 0.75)` | 消息占上下文窗口 75% 时触发 |

### 摘要过程

```
触发摘要前的消息列表：
  [System Prompt]
  [用户消息 #1]  ←─┐
  [LLM 回复 #1]    │
  [工具调用 #1]    │  这些旧消息被压缩为摘要
  [工具结果 #1]    │
  [LLM 回复 #2]    │
  [用户消息 #2]    │
  [LLM 回复 #3]  ←─┘
  [用户消息 #3]  ←─┐
  [LLM 回复 #4]    │  keep=3 → 保留最近 3 条
  [用户消息 #4]  ←─┘

触发摘要后的消息列表：
  [System Prompt]
  [摘要消息: "之前的对话中，用户询问了 X，Agent 通过 Y 工具查询了 Z..."]
  [用户消息 #3]
  [LLM 回复 #4]
  [用户消息 #4]
```

### 源码解析

```python
# langchain_v1/agents/middleware/summarization.py — 摘要核心逻辑

class SummarizationMiddleware(AgentMiddleware):
    """当对话历史超过阈值时，自动压缩为摘要。"""

    def before_model(self, request: ModelRequest) -> ModelRequest:
        """在 LLM 调用前检查是否需要摘要。"""
        messages = request.messages

        # 1. 检查是否触发摘要
        if not self._should_summarize(messages):
            return request

        # 2. 分离：保留的消息 vs 需要摘要的消息
        to_summarize = messages[:-self.keep]    # 前面的旧消息
        to_keep = messages[-self.keep:]         # 最近的消息保留

        # 3. 使用 LLM 生成摘要
        summary = self._generate_summary(to_summarize)

        # 4. 重新组装消息列表
        summary_message = SystemMessage(
            content=f"[对话摘要] {summary}"
        )
        request.messages = [
            request.messages[0],  # 保留原始 System Prompt
            summary_message,       # 插入摘要
            *to_keep,             # 保留最近的消息
        ]
        return request

    def _should_summarize(self, messages: list) -> bool:
        """根据触发条件判断是否需要摘要。"""
        trigger_type, threshold = self.trigger
        if trigger_type == "tokens":
            total_tokens = sum(
                estimate_tokens(m.content) for m in messages
            )
            return total_tokens > threshold
        elif trigger_type == "messages":
            return len(messages) > threshold
        elif trigger_type == "fraction":
            total_tokens = sum(
                estimate_tokens(m.content) for m in messages
            )
            return total_tokens > self.model_context_size * threshold
        return False

    def _generate_summary(self, messages: list) -> str:
        """调用 LLM 生成消息摘要。"""
        summary_prompt = (
            "请将以下对话历史压缩为简洁摘要，"
            "保留关键信息和上下文：\n\n"
            + "\n".join(m.content for m in messages)
        )
        return self.summary_model.invoke(summary_prompt).content
```

**关键设计**：
1. 摘要在 `before_model` 钩子中执行，在 LLM 调用前压缩消息
2. 始终保留 System Prompt（`messages[0]`）
3. `keep` 参数确保最近的消息不被摘要，保持上下文连贯性

### 触发条件选择指南

| 触发条件 | 优点 | 缺点 | 推荐场景 |
|---------|------|------|---------|
| `"tokens"` | 精确控制 Token 预算 | 需要 Token 估算 | 成本敏感场景 |
| `"messages"` | 简单直观 | 不同消息长度差异大 | 快速配置 |
| `"fraction"` | 自适应模型窗口大小 | 需要知道模型上下文大小 | 多模型部署 |

## 第二重手段：ContextEditingMiddleware（上下文编辑）

### 核心机制

比摘要更精细的控制——允许自定义编辑逻辑来处理上下文。

```python
from langchain.agents.middleware import ContextEditingMiddleware

context_editing = ContextEditingMiddleware(
    edit_fn=custom_edit_function,    # 自定义编辑函数
)
```

### 典型编辑策略

```python
# 策略 1：移除工具调用的详细结果，只保留摘要
def remove_verbose_tool_results(request):
    for msg in request.messages:
        if isinstance(msg, ToolMessage):
            if len(msg.content) > 1000:
                msg.content = msg.content[:200] + "\n[... 结果已截断 ...]"
    return request

# 策略 2：移除中间的思考过程，只保留关键决策
def keep_only_decisions(request):
    filtered = []
    for msg in request.messages:
        if isinstance(msg, AIMessage) and msg.tool_calls:
            filtered.append(msg)      # 保留有工具调用的决策
        elif not isinstance(msg, AIMessage):
            filtered.append(msg)      # 保留非 AI 消息
    request.messages = filtered
    return request

# 策略 3：按类型限制消息数量
def limit_by_type(request):
    tool_messages = [m for m in request.messages if isinstance(m, ToolMessage)]
    if len(tool_messages) > 10:
        # 只保留最近 10 条工具结果
        to_remove = tool_messages[:-10]
        request.messages = [m for m in request.messages if m not in to_remove]
    return request
```

### SummarizationMiddleware vs ContextEditingMiddleware

| 维度 | SummarizationMiddleware | ContextEditingMiddleware |
|------|------------------------|-------------------------|
| **处理方式** | 自动调用 LLM 生成摘要 | 自定义函数编辑消息 |
| **精度** | 粗粒度（整体摘要） | 细粒度（按消息类型/内容编辑） |
| **额外成本** | 有（需要额外 LLM 调用） | 无（纯代码逻辑） |
| **信息保留** | 有损压缩（LLM 摘要可能丢信息） | 可精确控制保留什么 |
| **适用场景** | 长对话压缩 | 精细的上下文裁剪 |

## 第三重手段：中间步骤裁剪（经典模式）

### AgentExecutor 的 trim_intermediate_steps

这是 AgentExecutor（经典模式）中管理上下文增长的机制：

```python
# langchain_classic/agents/agent.py — 中间步骤裁剪

def _prepare_intermediate_steps(self, intermediate_steps):
    # 策略 1：保留最近 N 步
    if isinstance(self.trim_intermediate_steps, int):
        if self.trim_intermediate_steps > 0:
            return intermediate_steps[-self.trim_intermediate_steps:]

    # 策略 2：自定义裁剪函数
    if callable(self.trim_intermediate_steps):
        return self.trim_intermediate_steps(intermediate_steps)

    # 默认：不裁剪（保留所有步骤）
    return intermediate_steps
```

### 使用示例

```python
# 经典模式
agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    trim_intermediate_steps=5,    # 只保留最近 5 步
)

# 自定义裁剪（按 Token 数限制）
def token_budget_trim(steps):
    """保留最近的步骤，直到 Token 预算用完。"""
    budget = 4000
    kept = []
    for step in reversed(steps):
        action, observation = step
        tokens = estimate_tokens(str(action) + str(observation))
        if budget - tokens < 0:
            break
        budget -= tokens
        kept.insert(0, step)
    return kept

agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    trim_intermediate_steps=token_budget_trim,
)
```

**设计启示：** 长时间运行的 Agent 需要管理中间步骤的增长，否则每次 LLM 调用时传递的 `agent_scratchpad` 会持续膨胀，最终超出上下文窗口。

## 生产环境的上下文管理策略

### 推荐组合

```python
from langchain.agents import create_agent
from langchain.agents.middleware import (
    SummarizationMiddleware,
    ContextEditingMiddleware,
    ModelCallLimitMiddleware,
)

# 策略：摘要 + 编辑 + 限制 三重保障
agent = create_agent(
    "gpt-4.1",
    tools=tools,
    middleware=[
        # 第 1 层：全局限制（最外层保护）
        ModelCallLimitMiddleware(run_limit=25),

        # 第 2 层：上下文编辑（先裁剪冗余内容，减少摘要成本）
        ContextEditingMiddleware(
            edit_fn=remove_verbose_tool_results,  # 截断过长的工具结果
        ),

        # 第 3 层：对话摘要（对裁剪后仍然过长的内容进行摘要）
        SummarizationMiddleware(
            trigger=("fraction", 0.6),  # 超过 60% 上下文窗口时触发
            keep=5,                      # 保留最近 5 条消息
        ),
    ],
)
```

### 执行流程

```
对话进行到第 15 轮，消息列表很长：

1. ContextEditingMiddleware.before_model()
   → 扫描工具结果：3 条超过 1000 字符
   → 截断为 200 字符 + "[已截断]"
   → Token 从 80,000 降到 60,000

2. SummarizationMiddleware.before_model()
   → 检查 Token 数：60,000 / 128,000 = 46.9%
   → 未超过 60% 阈值 → 不触发摘要
   → 继续执行

3. 对话继续到第 25 轮...
   → ContextEditingMiddleware 裁剪后仍有 90,000 Token
   → SummarizationMiddleware 检查：90,000 / 128,000 = 70.3%
   → 超过 60% 阈值 → 触发摘要 ✅
   → 保留最近 5 条，其余压缩为摘要
   → Token 从 90,000 降到 15,000
```

### 不同场景的上下文管理策略

| 场景 | 上下文压力 | 推荐策略 |
|------|-----------|---------|
| **简单问答 Agent** | 低 | 仅 Limit(25) 即可 |
| **RAG 文档问答** | 中（检索结果较长） | ContextEditing(截断) + Limit(30) |
| **智能客服** | 高（多轮长对话） | Summarization(0.6) + ContextEditing + Limit(20) |
| **代码助手** | 高（代码输出很长） | ContextEditing(截断代码) + Summarization(0.5) + Limit(50) |
| **数据分析 Agent** | 极高（SQL 结果巨大） | ContextEditing(只保留前10行) + Summarization(tokens:4000) |

## 经典模式 vs Middleware 模式对比

| 维度 | AgentExecutor (经典) | Middleware (新) |
|------|---------------------|----------------|
| **上下文管理工具** | `trim_intermediate_steps` | `SummarizationMiddleware` + `ContextEditingMiddleware` |
| **触发方式** | 每次循环前自动裁剪 | `before_model` 钩子中检查 |
| **灵活性** | 只能裁剪中间步骤 | 可以编辑任何消息（系统、用户、AI、工具） |
| **摘要能力** | 无 | 自动调用 LLM 生成摘要 |
| **与其他能力组合** | 难以与重试/安全等组合 | 自然地与其他 Middleware 堆叠 |

---

## 关键设计原则

1. **先裁剪后摘要**：先用 ContextEditing 去掉冗余信息，再用 Summarization 压缩，减少摘要 LLM 调用成本
2. **始终保留 System Prompt**：摘要时不能丢失系统指令
3. **保留最近消息**：`keep` 参数确保当前对话的连贯性
4. **选择合适的触发条件**：`fraction` 最通用（自适应模型窗口），`tokens` 最精确（成本控制），`messages` 最简单
5. **设置 CallLimit 作为兜底**：即使上下文管理失效，调用限制也能防止无限循环
