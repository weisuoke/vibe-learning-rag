# 自定义Tool开发 - 30字核心

> 用最精炼的语言抓住本质

---

## 30字核心概念

**用装饰器或类继承将Python函数封装为Agent可调用的工具，通过Schema定义参数让LLM理解如何使用。**

（恰好 30 个汉字）

---

## 核心概念拆解

让我们逐字理解这 30 个字背后的深意：

### 1. "用装饰器或类继承"

**关键词**：创建方式的多样性

这揭示了 LangChain 提供的三种工具创建方式：

- **装饰器**：`@tool` - 最简洁的方式
- **类继承**：`BaseTool` - 最灵活的方式
- **（隐含）工厂方法**：`StructuredTool.from_function()` - 中间方式

**为什么有多种方式？**
- 简单场景不需要复杂的类结构
- 复杂场景需要状态管理和生命周期控制
- 不同开发者有不同的编程习惯

**类比**：
- 前端：函数组件 vs 类组件（React）
- 日常：快餐 vs 定制餐（根据需求选择）

---

### 2. "将Python函数封装"

**关键词**：函数是基础单元

工具的本质是**可执行的函数**：
- 输入：参数（来自 LLM 的决策）
- 处理：业务逻辑（你的代码）
- 输出：结果（返回给 LLM）

**为什么是函数？**
- 函数是最小的可复用单元
- 函数有明确的输入输出契约
- 函数易于测试和调试

**封装的意义**：
- 添加元数据（名称、描述）
- 添加 Schema（参数类型、约束）
- 集成到 LangChain 生态（Runnable 协议）

**类比**：
- 前端：将业务逻辑封装为 API 端点
- 日常：将菜谱（函数）装订成菜单（工具）

---

### 3. "为Agent可调用的工具"

**关键词**：Agent 是调用者

工具不是孤立存在的，它是为 Agent 服务的：

**Agent 调用工具的流程**：
```
1. LLM 分析用户问题
2. LLM 决定需要调用哪个工具
3. LLM 生成工具调用参数（JSON）
4. Agent 执行工具调用
5. 工具返回结果
6. LLM 基于结果继续推理
```

**"可调用"的技术含义**：
- 工具必须实现 `Runnable` 接口
- 工具必须能被序列化（传递给 LLM）
- 工具必须有标准的输入输出格式

**类比**：
- 前端：组件必须符合框架的接口规范才能被渲染
- 日常：电器必须符合插座标准才能使用

---

### 4. "通过Schema定义参数"

**关键词**：Schema 是契约

Schema 是工具和 LLM 之间的"通信协议"：

**Schema 包含的信息**：
- 参数名称（`location`）
- 参数类型（`str`, `int`, `bool`）
- 参数描述（"City name or coordinates"）
- 是否必需（`required`）
- 默认值（`default="celsius"`）
- 约束条件（`Literal["celsius", "fahrenheit"]`）

**为什么需要 Schema？**
- LLM 需要知道工具接受什么参数
- LLM 需要知道参数的含义和格式
- 运行时需要验证参数的合法性

**Schema 的两种来源**：
1. **自动推断**：从函数签名和类型提示推断
2. **显式定义**：使用 Pydantic BaseModel 定义

**类比**：
- 前端：TypeScript 接口定义（API 契约）
- 日常：点餐单（菜品名称、价格、备注）

---

### 5. "让LLM理解如何使用"

**关键词**：描述是关键

工具的描述决定了 LLM 是否会正确调用它：

**好的描述应该包含**：
- **功能说明**：这个工具做什么？
- **使用时机**：什么情况下应该调用？
- **参数说明**：每个参数的含义和示例
- **返回值说明**：返回什么格式的数据？

**示例对比**：

❌ **差的描述**：
```python
@tool
def search(q: str) -> str:
    """Search."""
    return search_api(q)
```

✅ **好的描述**：
```python
@tool
def search_products(query: str, max_results: int = 10) -> str:
    """Search the product database by keywords.

    Use this when the user asks about product availability,
    pricing, or specifications.

    Args:
        query: Search keywords (e.g., "laptop", "iPhone 15")
        max_results: Maximum number of results to return (1-100)

    Returns:
        JSON string with product details including name, price, and stock.
    """
    return search_api(query, max_results)
```

**为什么描述如此重要？**
- LLM 通过描述理解工具的用途
- 描述不清晰会导致工具被误用或不被使用
- 描述是 LLM 做决策的唯一依据

**类比**：
- 前端：API 文档（开发者通过文档理解如何调用）
- 日常：说明书（用户通过说明书理解如何使用产品）

## 与 AI Agent 开发的联系

### 工具在 Agent 系统中的角色

```
用户问题
   ↓
LLM 推理 ← 工具描述（Schema + Description）
   ↓
决定调用工具
   ↓
生成调用参数 ← Schema 验证
   ↓
执行工具函数
   ↓
返回结果
   ↓
LLM 继续推理
   ↓
生成回复
```

### 工具开发的核心挑战

1. **描述清晰性**：让 LLM 准确理解工具用途
2. **Schema 完整性**：让 LLM 知道如何传参
3. **错误处理**：工具失败不应中断 Agent
4. **性能优化**：工具调用可能成为瓶颈

## 实际应用示例

### 示例 1：简单工具（@tool 装饰器）

```python
from langchain.tools import tool

@tool
def get_current_time(timezone: str = "UTC") -> str:
    """Get current time in specified timezone.

    Args:
        timezone: Timezone name (e.g., "UTC", "America/New_York")
    """
    from datetime import datetime
    import pytz

    tz = pytz.timezone(timezone)
    current_time = datetime.now(tz)
    return current_time.strftime("%Y-%m-%d %H:%M:%S %Z")
```

**30字核心的体现**：
- ✅ 用装饰器（@tool）
- ✅ 封装 Python 函数（get_current_time）
- ✅ Agent 可调用（自动集成到 LangChain）
- ✅ Schema 定义参数（timezone: str）
- ✅ LLM 理解使用（docstring 描述）

---

### 示例 2：复杂工具（BaseTool 继承）

```python
from langchain.tools import BaseTool
from pydantic import BaseModel, Field
from typing import Any

class WeatherInput(BaseModel):
    location: str = Field(description="City name")
    units: str = Field(default="metric", description="Temperature units")

class WeatherTool(BaseTool):
    name = "get_weather"
    description = "Get current weather for a location"
    args_schema = WeatherInput

    # 工具状态
    api_client: Any = None

    def __init__(self, api_key: str, **kwargs):
        super().__init__(**kwargs)
        self.api_client = WeatherAPIClient(api_key)

    def _run(self, location: str, units: str = "metric", config: RunnableConfig = None) -> str:
        """Synchronous execution."""
        weather_data = self.api_client.get_weather(location, units)
        return f"Weather in {location}: {weather_data['temp']}°{units[0].upper()}"

    async def _arun(self, location: str, units: str = "metric", config: RunnableConfig = None) -> str:
        """Asynchronous execution."""
        weather_data = await self.api_client.aget_weather(location, units)
        return f"Weather in {location}: {weather_data['temp']}°{units[0].upper()}"
```

**30字核心的体现**：
- ✅ 用类继承（BaseTool）
- ✅ 封装 Python 函数（_run 方法）
- ✅ Agent 可调用（实现 Runnable 接口）
- ✅ Schema 定义参数（WeatherInput）
- ✅ LLM 理解使用（description + args_schema）

## 常见误解澄清

### 误解 1："工具就是函数"
❌ **错误**：工具 = 函数
✅ **正确**：工具 = 函数 + 元数据（名称、描述、Schema）

**解释**：
- 函数只是实现逻辑
- 工具还包含 LLM 理解所需的描述信息

---

### 误解 2："Schema 是可选的"
❌ **错误**：不定义 Schema 也能工作
✅ **正确**：Schema 是 LLM 理解工具的关键

**解释**：
- 没有 Schema，LLM 不知道如何传参
- 自动推断的 Schema 可能不够准确
- 显式定义 Schema 提升可靠性

---

### 误解 3："描述越长越好"
❌ **错误**：写很长的描述让 LLM 更理解
✅ **正确**：描述要清晰简洁，抓住关键信息

**解释**：
- LLM 的 context window 有限
- 过长的描述会浪费 token
- 关键是说清楚"何时用"和"如何用"

---

### 误解 4："所有工具都要用 BaseTool"
❌ **错误**：BaseTool 最灵活，应该都用它
✅ **正确**：优先用 @tool，需要时才用 BaseTool

**解释**：
- @tool 更简洁，适合 80% 场景
- BaseTool 适合需要状态管理的复杂工具
- 过度设计会增加维护成本
