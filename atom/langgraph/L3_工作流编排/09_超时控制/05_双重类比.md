# 双重类比

超时控制很容易被误解成“报错开关”。

为了把它真正理解成“分层预算系统”，我们用 **前端开发类比** + **日常生活类比** 双线说明。

---

## 类比 1：`step_timeout`

**前端类比：**
像 React 页面里一次批量状态更新的 commit 窗口。

如果一批异步依赖里有一个迟迟不返回，这一轮界面就可能迟迟不能稳定提交。`step_timeout` 就像你为这轮“批处理提交”设了总等待上限。

**日常生活类比：**
像旅行团集合发车时间。

导游不会因为一个人迟到就让整车无限等下去。等到 8:30 还没到，就必须做决定：继续等、联系、改签、或者先走。

```python
graph.step_timeout = 8
```

**对应关系：**
- 一轮并发节点 = 一车人集合
- step deadline = 发车时间
- 超时取消 = 不再继续等

---

## 类比 2：节点内 HTTP timeout

**前端类比：**
像浏览器里 `fetch()` 自己的超时控制，而不是只靠页面级 loading。

如果你只在页面层写“30 秒后提示失败”，但 `fetch()` 本身没有 abort，网络请求仍可能在底层继续悬挂。

**日常生活类比：**
像打客服电话时，你给每一个菜单步骤都设等待上限。

你不会只说“整个事情最多 30 分钟”，而应该细分：
- 接通客服最多等 2 分钟；
- 转人工最多等 3 分钟；
- 回调最多等 10 分钟。

```python
timeout = httpx.Timeout(10.0, connect=2.0, read=6.0, write=4.0, pool=2.0)
```

---

## 类比 3：`interrupt` 不是 timeout，而是“暂停”

**前端类比：**
像草稿自动保存后关闭弹窗，用户稍后回来继续填写。

这不是“页面一直卡着等待用户输入”，而是明确暂停交互，保存状态，等用户回来再继续。

**日常生活类比：**
像银行柜台办到一半，需要补材料，先取号暂停，准备好再回来继续办理。

```python
approved = interrupt("Do you approve this action?")
```

**关键区别：**
- timeout：等到上限还没来，就触发兜底策略
- interrupt：主动暂停，承认这不是“现在就该完成”的等待

---

## 类比 4：流式背压

**前端类比：**
像前端正在消费 SSE / WebSocket 消息，但 UI 渲染太慢，结果消息处理链反向积压，最终让整个交互表现得像“服务端超时”。

**日常生活类比：**
像厨房上菜很快，但前厅收桌、摆盘、传菜太慢，最后客人觉得“怎么还没上齐”。

慢的不一定是后厨，也可能是中间的消费环节。

```python
async for chunk in graph.astream(...):
    await asyncio.sleep(0.6)  # 模拟消费端处理慢
```

---

## 类比 5：平台 timeout

**前端类比：**
像 Vercel / Cloudflare 的函数执行窗口。你的代码逻辑没报错，不代表平台愿意无限等下去。

**日常生活类比：**
像商场停车场 15 分钟免费接人。你车里的人还在处理事情，不代表停车场不会开始计费或让你离场。

这就是为什么：
- 图没 timeout；
- 平台却 timeout；
- 前端却报 network error；

三件事可以同时成立。

---

## 类比总结表

| 目标概念 | 前端类比 | 日常生活类比 | 核心含义 |
|---|---|---|---|
| `step_timeout` | 一轮批处理提交窗口 | 旅行团发车时间 | 限制一轮 orchestration 等待 |
| I/O timeout | `fetch()` / abort deadline | 客服各环节等待上限 | 限制单次外部调用 |
| `interrupt` | 草稿保存后稍后继续 | 银行补材料后继续办理 | 主动暂停，不是超时失败 |
| 流式背压 | SSE 消费端渲染变慢 | 前厅传菜太慢 | 慢可能出在消费端 |
| 平台 timeout | Serverless 执行窗口 | 停车场免费等待窗口 | 交互层/平台层 budget |

---

## 记忆口诀
