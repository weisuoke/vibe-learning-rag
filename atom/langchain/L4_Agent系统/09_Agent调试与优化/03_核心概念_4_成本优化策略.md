# 核心概念4：成本优化策略

> Agent 调用 LLM 和工具都要花钱——通过 LLM 缓存、提示缓存、批处理、模型路由和工具调用缓存，在不牺牲质量的前提下大幅降低运行成本。

---

## 概述

### 为什么成本优化如此重要？

一个典型的 Agent 执行流程可能包含：

```
用户提问 "帮我分析这篇论文并总结要点"
  ↓
🧠 LLM 调用 #1：理解意图、规划步骤        → $0.002
🔧 工具调用 #1：读取论文内容              → $0.001
🧠 LLM 调用 #2：分析论文结构              → $0.005
🧠 LLM 调用 #3：提取关键观点              → $0.008
🧠 LLM 调用 #4：生成总结                  → $0.006
                                          ─────────
                                    单次请求: $0.022
                                    日均1000次: $22/天
                                    月成本: $660
```

如果不加优化，成本会随用户量线性增长。但通过合理的优化策略，可以降低 **50%-90%** 的成本。

### 成本优化的五大策略

| 策略 | 原理 | 预期降幅 | 适用场景 |
|------|------|---------|---------|
| **LLM 缓存** | 相同输入不重复调 API | 30-80% | 重复性问题多的场景 |
| **提示缓存** | 缓存静态前缀，只传增量 | 60-90% | 长系统提示的 Agent |
| **批处理** | 合并多个请求为批量调用 | ~50% | 非实时批量任务 |
| **模型路由** | 简单任务用便宜模型 | 40-70% | 任务复杂度差异大 |
| **工具调用缓存** | 缓存工具返回结果 | 20-60% | 工具结果有时效性 |

## 1. LLM 缓存（LLM Cache）

### 核心思想

> 相同的 Prompt + 相同的模型配置 = 相同的结果，为什么要重复调 API？

LangChain 的缓存系统源码定义（`langchain_core/caches.py`）：

```python
class BaseCache(ABC):
    """LLM 缓存的抽象基类"""
    
    @abstractmethod
    def lookup(self, prompt: str, llm_string: str) -> RETURN_VAL_TYPE | None:
        """查找缓存：根据 prompt + llm_string 匹配"""
    
    @abstractmethod
    def update(self, prompt: str, llm_string: str, return_val: RETURN_VAL_TYPE) -> None:
        """更新缓存：存储新的结果"""
    
    @abstractmethod
    def clear(self, **kwargs) -> None:
        """清除缓存"""
```

**缓存键 = prompt + llm_string（模型配置的序列化）**

### 使用 InMemoryCache

```python
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache
from langchain_openai import ChatOpenAI

# ===== 1. 启用内存缓存 =====
set_llm_cache(InMemoryCache())

llm = ChatOpenAI(model="gpt-4o-mini")

# ===== 2. 第一次调用（缓存未命中）=====
import time

start = time.time()
result1 = llm.invoke("什么是 LangChain？")
time1 = time.time() - start
print(f"第一次调用: {time1:.2f}s")  # 约 1-2 秒

# ===== 3. 第二次调用（缓存命中）=====
start = time.time()
result2 = llm.invoke("什么是 LangChain？")
time2 = time.time() - start
print(f"第二次调用: {time2:.4f}s")  # 约 0.001 秒（几乎瞬间）

print(f"速度提升: {time1/time2:.0f}x")
print(f"结果一致: {result1.content == result2.content}")  # True
```

### SQLite 持久化缓存

```python
from langchain_community.cache import SQLiteCache

# 持久化到本地文件，重启后缓存仍然有效
set_llm_cache(SQLiteCache(database_path=".langchain_cache.db"))

# 使用方式完全相同
result = llm.invoke("什么是 LangChain？")
```

### 缓存适用场景

| 场景 | 是否适合缓存 | 原因 |
|------|-------------|------|
| FAQ 问答 | ✅ 非常适合 | 问题重复率高 |
| 文档摘要 | ✅ 适合 | 相同文档的摘要结果固定 |
| 对话机器人 | ⚠️ 部分适合 | 上下文不同导致缓存命中率低 |
| 创意写作 | ❌ 不适合 | 需要多样性，不该返回缓存结果 |
| 实时数据分析 | ❌ 不适合 | 数据实时变化，缓存会过时 |

## 2. 提示缓存（Prompt Caching）

### 核心思想

> Agent 的系统提示通常很长且固定。每次都传完整提示是浪费——缓存静态前缀，只传增量部分。

2025-2026 年，主流 LLM 提供商（Anthropic、OpenAI、Google）都支持了提示缓存：

```
传统方式（每次传完整 Prompt）：
┌──────────────────────────────────────┐
│ 系统提示（2000 tokens）              │  ← 每次都传，每次都收费
│ 工具定义（500 tokens）               │
│ 历史对话（300 tokens）               │
│ 用户问题（50 tokens）                │
└──────────────────────────────────────┘
  总计：2850 tokens × $0.003/1K = $0.00855

提示缓存方式：
┌──────────────────────────────────────┐
│ 系统提示（2000 tokens）  ← 已缓存   │  ← 首次缓存，后续免费/低价
│ 工具定义（500 tokens）   ← 已缓存   │
├──────────────────────────────────────┤
│ 历史对话（300 tokens）               │  ← 每次传，正常收费
│ 用户问题（50 tokens）                │
└──────────────────────────────────────┘
  缓存命中后：350 tokens × $0.003/1K = $0.00105
  成本降低：88%！
```

### Anthropic 提示缓存示例

```python
from langchain_openai import ChatOpenAI

# Anthropic 的提示缓存通过 cache_control 参数启用
llm = ChatOpenAI(
    model="claude-sonnet-4-20250514",
    base_url="https://api.anthropic.com/v1",
)

# 在消息中标记可缓存的部分
messages = [
    {
        "role": "system",
        "content": [
            {
                "type": "text",
                "text": "你是一个专业的 LangChain 开发助手...(很长的系统提示)",
                "cache_control": {"type": "ephemeral"}  # 标记为可缓存
            }
        ]
    },
    {"role": "user", "content": "如何调试 Agent？"}
]

# 第一次调用：缓存写入（略贵 25%）
# 后续调用：缓存命中（便宜 90%）
result = llm.invoke(messages)
```

### 提示缓存最佳实践

1. **把不变的内容放在前面**：系统提示、工具定义、few-shot 示例
2. **把变化的内容放在后面**：用户输入、对话历史
3. **监控缓存命中率**：通过 LangSmith 追踪查看

## 3. 批处理（Batch Processing）

### 核心思想

> 10 个请求逐个发送 vs 打包一次发送，后者更便宜（约 50% 折扣）。

OpenAI 和 Anthropic 都提供了批处理 API，适用于非实时场景：

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")

# ===== 单个调用（贵）=====
results = []
for question in questions:
    result = llm.invoke(question)  # 每次独立调用
    results.append(result)

# ===== batch 调用（更便宜、更快）=====
results = llm.batch(
    questions,
    config={"max_concurrency": 5}  # 并行 5 个
)
```

### langasync 批处理工具（2026 新）

```python
# langasync 将 LCEL 链包装为批处理模式
# 适用于评估、标注等离线任务
from langasync import wrap_chain

chain = prompt | llm | parser

# 包装为批处理模式
async_chain = wrap_chain(chain, batch_size=10)

# 批量执行（成本降低约 50%）
results = await async_chain.abatch(inputs)
```

### 批处理适用场景

| 场景 | 适合批处理？ | 原因 |
|------|-------------|------|
| 批量评估 | ✅ 非常适合 | 不需要实时返回 |
| 数据标注 | ✅ 非常适合 | 大量相似任务 |
| 报告生成 | ✅ 适合 | 可以接受延迟 |
| 实时对话 | ❌ 不适合 | 用户在等待 |
| 流式输出 | ❌ 不适合 | 需要逐字显示 |

## 4. 模型路由（Model Routing）

### 核心思想

> 不是所有任务都需要最强的模型。简单问题用便宜模型，复杂问题才用贵模型。

```python
from langchain_openai import ChatOpenAI
from langchain_core.runnables import RunnableBranch, RunnableLambda

# ===== 定义不同级别的模型 =====
cheap_llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)    # $0.15/1M tokens
expensive_llm = ChatOpenAI(model="gpt-4o", temperature=0)      # $2.50/1M tokens

# ===== 简单的复杂度评估 =====
def assess_complexity(input_dict):
    """根据输入长度和关键词判断复杂度"""
    text = str(input_dict)
    
    complex_keywords = ["分析", "对比", "策略", "架构", "设计", "优化"]
    has_complex_keyword = any(kw in text for kw in complex_keywords)
    is_long = len(text) > 500
    
    return has_complex_keyword or is_long

# ===== 构建路由链 =====
routed_chain = RunnableBranch(
    (lambda x: assess_complexity(x), expensive_llm),  # 复杂 → 贵模型
    cheap_llm  # 简单 → 便宜模型（默认）
)

# 使用
result = routed_chain.invoke({"question": "你好"})  # → gpt-4o-mini
result = routed_chain.invoke({"question": "请分析 LangChain 的架构设计并给出优化策略"})  # → gpt-4o
```

### 进阶：用 LLM 自动判断复杂度

```python
classifier_prompt = """判断以下问题的复杂度，只回答 "simple" 或 "complex"：

问题：{question}
复杂度："""

# 用最便宜的模型做分类
classifier = ChatOpenAI(model="gpt-4o-mini")
complexity = classifier.invoke(classifier_prompt.format(question=user_question))

if "simple" in complexity.content.lower():
    result = cheap_llm.invoke(user_question)
else:
    result = expensive_llm.invoke(user_question)
```

### 成本对比

```
场景：日均 1000 次请求（70% 简单、30% 复杂）

不路由（全用 gpt-4o）：
  1000 × $0.005 = $5.00/天

路由后：
  700 × $0.0003 = $0.21（简单 → gpt-4o-mini）
  300 × $0.005  = $1.50（复杂 → gpt-4o）
  总计 = $1.71/天

节省：66%！
```

## 5. 工具调用缓存

### 核心思想

> 同样的搜索查询，5 分钟内结果不会变。缓存工具返回值，避免重复调用外部 API。

```python
import time
import hashlib
import json
from functools import wraps

def cached_tool(ttl_seconds=300):
    """工具调用缓存装饰器，支持 TTL"""
    cache = {}
    
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            # 生成缓存键
            key = hashlib.md5(
                json.dumps({"args": args, "kwargs": kwargs}, sort_keys=True).encode()
            ).hexdigest()
            
            # 检查缓存
            if key in cache:
                result, timestamp = cache[key]
                if time.time() - timestamp < ttl_seconds:
                    print(f"  📦 缓存命中: {func.__name__}")
                    return result
            
            # 缓存未命中，执行工具
            print(f"  🔧 执行工具: {func.__name__}")
            result = func(*args, **kwargs)
            cache[key] = (result, time.time())
            return result
        
        return wrapper
    return decorator

# ===== 使用示例 =====
@cached_tool(ttl_seconds=600)  # 10 分钟缓存
def search_web(query: str) -> str:
    """搜索网络（模拟）"""
    # 实际调用搜索 API
    return f"搜索结果: {query} 的相关信息..."

# 第一次调用：执行工具
result1 = search_web("LangChain debugging")   # 🔧 执行工具

# 第二次调用：缓存命中
result2 = search_web("LangChain debugging")   # 📦 缓存命中
```

### 在 LangChain Tool 中使用缓存

```python
from langchain_core.tools import tool

@tool
@cached_tool(ttl_seconds=300)
def get_weather(city: str) -> str:
    """获取城市天气"""
    # 模拟 API 调用
    return f"{city}: 晴天，25°C"

# Agent 多次调用同一工具时自动走缓存
```

## 6. 综合成本监控

### 成本追踪回调

```python
from langchain_core.callbacks import BaseCallbackHandler

# 定价表（美元/百万 tokens，2026 年）
PRICING = {
    "gpt-4o": {"input": 2.50, "output": 10.00},
    "gpt-4o-mini": {"input": 0.15, "output": 0.60},
    "claude-sonnet-4-20250514": {"input": 3.00, "output": 15.00},
}

class CostTracker(BaseCallbackHandler):
    """追踪 LLM 调用成本"""
    
    def __init__(self):
        self.total_cost = 0.0
        self.calls = []
    
    def on_llm_end(self, response, **kwargs):
        if hasattr(response, 'llm_output') and response.llm_output:
            usage = response.llm_output.get('token_usage', {})
            model = response.llm_output.get('model_name', 'unknown')
            
            pricing = PRICING.get(model, {"input": 0, "output": 0})
            input_cost = usage.get('prompt_tokens', 0) / 1_000_000 * pricing["input"]
            output_cost = usage.get('completion_tokens', 0) / 1_000_000 * pricing["output"]
            call_cost = input_cost + output_cost
            
            self.total_cost += call_cost
            self.calls.append({
                "model": model,
                "tokens": usage.get('total_tokens', 0),
                "cost": call_cost
            })
    
    def report(self):
        print(f"\n💰 成本报告")
        print(f"{'='*40}")
        print(f"总调用次数: {len(self.calls)}")
        print(f"总成本: ${self.total_cost:.6f}")
        for i, call in enumerate(self.calls, 1):
            print(f"  #{i} {call['model']}: {call['tokens']} tokens, ${call['cost']:.6f}")

# 使用
tracker = CostTracker()
result = agent.invoke(input, config={"callbacks": [tracker]})
tracker.report()
```

## 7. 成本优化决策树

```
你的 Agent 成本高吗？
│
├── 是 → 重复性请求多吗？
│        ├── 是 → 启用 LLM 缓存（InMemoryCache / SQLiteCache）
│        └── 否 → 系统提示很长吗？
│                 ├── 是 → 启用提示缓存（Prompt Caching）
│                 └── 否 → 所有请求都需要实时响应吗？
│                          ├── 否 → 使用批处理（Batch API）
│                          └── 是 → 所有请求都很复杂吗？
│                                   ├── 否 → 启用模型路由
│                                   └── 是 → 优化 Prompt 减少 Token
│
└── 否 → 🎉 保持现状
```

## 8. 优化效果汇总

| 策略 | 实现难度 | 预期成本降幅 | 对质量的影响 | 推荐优先级 |
|------|---------|-------------|-------------|-----------|
| LLM 缓存 | ⭐ 简单 | 30-80% | 无（完全一致） | 🥇 第一 |
| 模型路由 | ⭐⭐ 中等 | 40-70% | 简单任务无影响 | 🥈 第二 |
| 提示缓存 | ⭐⭐ 中等 | 60-90% | 无 | 🥉 第三 |
| 批处理 | ⭐ 简单 | ~50% | 无（增加延迟） | 第四 |
| 工具缓存 | ⭐⭐ 中等 | 20-60% | 需考虑时效性 | 第五 |

## 小结

| 要点 | 说明 |
|------|------|
| **核心原则** | 在保持输出质量的前提下，最小化 Token 消耗和 API 调用次数 |
| **最简单的优化** | `set_llm_cache(InMemoryCache())` 一行代码，立刻生效 |
| **最大的收益** | 提示缓存（90% 降幅），但需要提供商支持 |
| **最被忽视的** | 模型路由——不是所有任务都需要 GPT-4 |
| **生产环境必备** | CostTracker 回调 + LangSmith 成本追踪 |

**一句话记忆**：缓存减少重复、路由降低单价、批处理打折扣——三管齐下省 50-90%。
