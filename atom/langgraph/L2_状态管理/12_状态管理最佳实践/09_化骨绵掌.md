# 化骨绵掌 - 10个2分钟知识卡片

> 将 LangGraph 状态管理最佳实践拆解为 10 个独立卡片，每个 2 分钟可读完，从"状态是什么"到"生产级演进策略"形成递进

---

## 卡片1：状态是什么？

**一句话：** 状态是 LangGraph 图中所有节点共享的数据容器，它决定了工作流能"记住"什么。

**核心理解：**

LangGraph 的每个节点不能直接互相传参。它们通过一个共享的状态字典通信：

```python
from typing import TypedDict, Annotated
from langgraph.graph import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]  # 对话历史
    query: str                                # 当前查询
```

节点读取状态、处理数据、返回更新——这就是 LangGraph 的全部数据流模型。

**应用：** 每个工作流的第一步就是设计状态。状态设计得好，后续开发顺畅；设计得差，后面全是补丁。就像盖房子先画图纸——状态 Schema 就是你的图纸。

## 卡片2：Schema 设计的第一原则 - 类型安全

**一句话：** 用 `TypedDict` + `Annotated` 让每个字段都有明确的类型和更新规则，把错误拦在编译期。

**两种选型：**

| 场景 | 选择 | 原因 |
|------|------|------|
| 快速原型、内部工具 | `TypedDict` | 轻量、零开销 |
| 生产系统、外部输入 | `Pydantic BaseModel` | 运行时验证、自动文档 |

**代码示例：**

```python
# TypedDict —— 轻量快速
class State(TypedDict):
    messages: Annotated[list, add_messages]
    query: str
    doc_ids: list[str]

# Pydantic —— 严格验证
from pydantic import BaseModel, Field

class State(BaseModel):
    messages: Annotated[list, add_messages]
    query: str = Field(min_length=1, description="用户查询")
    temperature: float = Field(ge=0.0, le=2.0, default=0.7)
```

**应用：** 开发阶段用 TypedDict 快速迭代，上线前切换到 Pydantic 加上验证。类型安全不是负担，是保险——它帮你在凌晨三点少接一个告警电话。

## 卡片3：Reducer 是状态更新的合约

**一句话：** Reducer 定义了"新值如何与旧值合并"，选错 Reducer 会导致数据丢失或无限膨胀。

**三种模式：**

```python
from typing import Annotated
from langgraph.graph import add_messages

class State(TypedDict):
    # 1. LastValue（默认）—— 新值直接覆盖旧值
    query: str

    # 2. add_messages —— 消息累积追加
    messages: Annotated[list, add_messages]

    # 3. 自定义 Reducer —— 完全控制合并逻辑
    score: Annotated[float, lambda old, new: max(old, new)]
```

**选型速查：**

| 数据特征 | 推荐 Reducer | 示例 |
|----------|-------------|------|
| 每次覆盖 | LastValue（默认） | `query`, `current_step` |
| 持续累积 | `add_messages` / `operator.add` | `messages`, `logs` |
| 条件合并 | 自定义函数 | 取最大值、去重合并 |

**应用：** 最常见的 bug 是该累积的字段用了覆盖（丢消息），或该覆盖的字段用了累积（列表爆炸）。定义 Schema 时，每个字段都要问自己：这个字段是"替换"还是"追加"？

## 卡片4：不可变更新 - 永远返回新字典

**一句话：** 节点不能直接修改 state 对象，必须返回一个新的更新字典，这是检查点正确和并发安全的基石。

**错误 vs 正确：**

```python
# ❌ 错误：直接修改状态（突变）
def bad_node(state: State):
    state["count"] += 1           # 直接改了原对象
    state["items"].append("new")  # 原地修改列表
    return state

# ✅ 正确：返回新的更新字典
def good_node(state: State):
    return {
        "count": state["count"] + 1,
        "items": [*state["items"], "new"],
    }
```

**为什么这么重要？**

1. 检查点保存的是快照——如果你改了原对象，历史快照也被污染了
2. 并发节点共享同一个状态引用——直接修改会产生竞态条件
3. 时间旅行调试依赖不可变性——改了原对象就没法"回到过去"

**应用：** 把这条规则刻进肌肉记忆：节点函数永远 `return {"field": new_value}`，永远不写 `state["field"] = xxx`。

## 卡片5：状态最小化 - 只存必要数据

**一句话：** 派生数据运行时计算，不存入状态；大对象存引用，不存内容。这是性能优化的第一原则。

**反面教材 vs 正确做法：**

```python
# ❌ 状态膨胀：什么都往状态里塞
class BadState(TypedDict):
    documents: list[str]          # 完整文档内容（可能几MB）
    doc_count: int                # 派生数据：len(documents)
    has_documents: bool           # 派生数据：len(documents) > 0
    formatted_output: str         # 中间产物

# ✅ 最小化状态：只存必要数据
class GoodState(TypedDict):
    doc_ids: list[str]            # 只存 ID，需要时再查
    query: str                    # 原始输入
    answer: str                   # 最终输出
```

**判断标准：**

| 问题 | 如果"是" | 如果"否" |
|------|---------|---------|
| 多个节点需要这个数据？ | 存入状态 | 节点内部变量 |
| 能从其他字段计算出来？ | 运行时计算 | 存入状态 |
| 数据量大于 1KB？ | 存引用/ID | 考虑是否必要 |
| 只在相邻两个节点间传递？ | 考虑 EphemeralValue | 存入状态 |

**应用：** 每次往状态里加字段前，问自己："这个字段不存行不行？"如果能从已有字段推导出来，就别存。状态越小，序列化越快，检查点越轻。

## 卡片6：消息列表必须有上限

**一句话：** 无限增长的消息列表是最常见的状态膨胀原因，长对话场景必须设置裁剪策略。

**问题场景：**

用户和 Agent 聊了 200 轮，`messages` 列表里有 400 条消息。每次检查点序列化都要处理这 400 条，LLM 调用也会因为 context window 超限而报错。

**解决方案：**

```python
from langchain_core.messages import trim_messages

def trimmed_messages(left: list, right: list) -> list:
    """自定义 Reducer：追加后裁剪，保留最近 20 条"""
    merged = left + right
    # 保留第一条系统消息 + 最近 19 条
    if len(merged) > 20:
        return [merged[0]] + merged[-19:]
    return merged

class State(TypedDict):
    messages: Annotated[list, trimmed_messages]
```

**裁剪策略选择：**

| 策略 | 适用场景 | 实现方式 |
|------|---------|---------|
| 保留最近 N 条 | 通用对话 | 切片 `messages[-N:]` |
| 保留首条 + 最近 N 条 | 有系统提示的对话 | `[messages[0]] + messages[-N:]` |
| Token 预算裁剪 | 精确控制 LLM 输入 | `trim_messages(max_tokens=4000)` |
| 摘要压缩 | 超长对话 | 旧消息 → LLM 摘要 → 替换 |

**应用：** 任何生产级对话系统都必须有消息裁剪。不是"要不要做"的问题，是"什么时候做"的问题——答案是现在。

## 卡片7：EphemeralValue - 用完即弃

**一句话：** 临时数据不需要持久化到检查点，用完即弃的通道可以显著减少序列化开销。

**什么是临时数据？**

- 节点间传递的中间计算结果（如原始检索结果，后续节点只需要排序后的 top-k）
- 调试信息、日志数据
- 一次性的配置参数

**实现方式：**

```python
from langgraph.channels import EphemeralValue

class State(TypedDict):
    # 持久化字段 —— 每个检查点都保存
    messages: Annotated[list, add_messages]
    answer: str

    # 临时字段 —— 不写入检查点，节点执行完就清除
    raw_docs: Annotated[list, EphemeralValue(list)]
    debug_info: Annotated[dict, EphemeralValue(dict)]
```

**持久化 vs 临时的判断：**

```
这个数据在图执行完后还需要吗？
├── 需要 → 持久化字段（普通 Reducer）
└── 不需要 → 临时字段（EphemeralValue）
    ├── 只在下一个节点用 → EphemeralValue
    └── 只在当前节点用 → 节点内部变量，别放状态里
```

**应用：** 一个 RAG 管道检索了 50 篇文档，但最终只用了 5 篇。把 50 篇都存进检查点是浪费。用 EphemeralValue 存原始检索结果，只把最终的 5 篇 ID 存入持久状态。

## 卡片8：Input/Output Schema 分离

**一句话：** 内部状态不应该暴露给外部调用者，用 Input/Output Schema 分离实现接口隔离。

**问题：** 你的状态有 10 个字段，但调用者只需要传入 `query`，只关心 `answer`。如果暴露全部字段，调用者会困惑，也可能误传内部字段。

**解决方案：**

```python
# 内部完整状态
class InternalState(TypedDict):
    messages: Annotated[list, add_messages]
    query: str
    doc_ids: list[str]
    retrieval_score: float
    answer: str

# 外部输入 Schema —— 调用者只需要提供这些
class InputSchema(TypedDict):
    query: str

# 外部输出 Schema —— 调用者只能看到这些
class OutputSchema(TypedDict):
    answer: str

# 构建图时指定三个 Schema
graph = StateGraph(
    InternalState,
    input=InputSchema,
    output=OutputSchema,
)
```

**三层隔离：**

| 层 | Schema | 谁能看到 |
|----|--------|---------|
| 输入层 | `InputSchema` | 外部调用者 |
| 内部层 | `InternalState` | 图内节点 |
| 输出层 | `OutputSchema` | 外部调用者 |

**应用：** 这就像 API 设计中的 DTO（Data Transfer Object）——你不会把数据库表结构直接暴露给前端。Input/Output Schema 让你的图有清晰的"接口契约"，内部重构不影响外部调用。

## 卡片9：Schema 演进 - 向后兼容优先

**一句话：** 新字段用 `Optional` + 默认值，避免破坏已有检查点，这是生产环境平滑升级的关键。

**场景：** 你的 RAG 系统已经上线，数据库里有几万个检查点。现在要给状态加一个 `language` 字段。

**安全的做法：**

```python
# v1 —— 已上线的 Schema
class StateV1(TypedDict):
    messages: Annotated[list, add_messages]
    query: str
    answer: str

# v2 —— 新增字段，向后兼容
class StateV2(TypedDict):
    messages: Annotated[list, add_messages]
    query: str
    answer: str
    language: str  # 新字段

# 节点中安全读取新字段
def generate_node(state: StateV2):
    # 用 .get() 提供默认值，兼容旧检查点
    language = state.get("language", "zh")
    # ...
```

**演进规则：**

| 操作 | 安全性 | 处理方式 |
|------|--------|---------|
| 新增可选字段 | 安全 | `state.get("field", default)` |
| 新增必填字段 | 危险 | 必须提供迁移脚本 |
| 删除字段 | 中等 | 先标记废弃，下个版本再删 |
| 修改字段类型 | 危险 | 新建字段 + 迁移，不要改原字段 |

**应用：** 永远假设数据库里有"旧版本"的检查点。每次改 Schema，先问自己："旧检查点加载到新 Schema 会不会报错？"如果会，就需要迁移策略。

## 卡片10：Overwrite - 紧急逃生通道

**一句话：** 当需要绕过 Reducer 直接重置状态时，使用 `Overwrite` 包装器——这是你的紧急逃生通道。

**问题场景：**

`messages` 字段绑定了 `add_messages` Reducer，正常情况下只能追加。但有时你需要清空消息列表（比如错误恢复、会话重置）。

**解决方案：**

```python
from langgraph.graph import Overwrite

def reset_node(state: State):
    """错误恢复：清空消息，重新开始"""
    return {
        "messages": Overwrite(value=[]),  # 绕过 add_messages，直接设为空列表
        "query": state["query"],          # 保留原始查询
    }

def trim_node(state: State):
    """裁剪：只保留最近 10 条消息"""
    recent = state["messages"][-10:]
    return {
        "messages": Overwrite(value=recent),  # 绕过累积，直接替换
    }
```

**使用原则：**

- 正常流程用 Reducer（累积、合并）
- 异常流程用 Overwrite（重置、裁剪）
- Overwrite 是"破窗"操作，不要在正常业务逻辑中滥用

**应用：** 把 Overwrite 想象成数据库的 `TRUNCATE TABLE`——你不会在正常业务里用它，但当数据出了问题需要紧急修复时，它是救命的工具。
