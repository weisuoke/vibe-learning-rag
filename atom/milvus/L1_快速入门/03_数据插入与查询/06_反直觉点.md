# 反直觉点

> **避开常见误区：Milvus 数据插入与查询中的反直觉现象**

---

## 概述

在使用 Milvus 进行数据插入与查询时，有许多行为与直觉相悖，容易导致错误或性能问题。本文总结了最常见的反直觉点，帮助你避开这些陷阱。

---

## 反直觉点 1: Insert 不检查主键重复

### 直觉认为

"数据库的 Insert 操作应该检查主键重复，防止插入重复数据。"

### 实际情况

**Milvus 的 Insert 操作不检查主键重复！**

```python
# 第一次插入
client.insert("my_collection", [
    {"id": 1, "document": "text1", "dense": [0.1, 0.2]}
])

# 第二次插入相同主键（不会报错！）
client.insert("my_collection", [
    {"id": 1, "document": "text2", "dense": [0.3, 0.4]}
])

# 结果：Collection 中有两条 id=1 的记录！
```

### 为什么这样设计？

**性能考虑**: 检查主键重复需要查询现有数据，会显著降低插入性能。Milvus 优先保证高吞吐量。

### 正确做法

**使用 Upsert 避免重复**:
```python
# Upsert 会自动检查并更新
client.upsert("my_collection", [
    {"id": 1, "document": "text", "dense": [0.1, 0.2]}
])
```

### 影响

- ❌ **错误使用**: 可能导致数据重复，查询结果不一致
- ✅ **正确使用**: 新数据用 Insert，更新数据用 Upsert

## 反直觉点 2: Embedding Functions 不处理动态字段

### 直觉认为

"Embedding Functions 应该能处理所有字段，包括动态添加的字段。"

### 实际情况

**Embedding Functions 只处理 Schema 中显式定义的字段！**

```python
# 启用动态字段
schema.enable_dynamic_field = True

# 配置 Embedding Function
schema.add_function(Function(
    input_field_names=["document"],  # 只处理 document 字段
    output_field_names=["dense"]
))

# 插入数据
client.insert("my_collection", [
    {
        "document": "text",      # ✅ 会被嵌入
        "extra_field": "value"   # ❌ 不会被嵌入（动态字段）
    }
])
```

### 为什么这样设计？

**明确性**: Embedding Functions 需要明确知道输入和输出字段，动态字段无法在 Schema 定义时确定。

### 正确做法

**在 Schema 中显式定义所有需要嵌入的字段**:
```python
schema.add_field("document", DataType.VARCHAR, max_length=5000)
schema.add_field("title", DataType.VARCHAR, max_length=500)

# 为每个字段配置 Embedding Function
schema.add_function(Function(
    input_field_names=["document"],
    output_field_names=["doc_vector"]
))

schema.add_function(Function(
    input_field_names=["title"],
    output_field_names=["title_vector"]
))
```

## 反直觉点 3: Search 的 distance 不是"距离"

### 直觉认为

"Search 返回的 distance 是向量之间的距离，越小越相似。"

### 实际情况

**distance 的含义取决于 metric_type！**

| Metric Type | distance 含义 | 相似度规则 | 范围 |
|-------------|---------------|------------|------|
| **L2** | 欧氏距离 | 越小越相似 | [0, ∞) |
| **IP** | 内积 | 越大越相似 | [-1, 1] |
| **COSINE** | 余弦相似度 | 越大越相似 | [-1, 1] |

```python
# 使用 COSINE
results = client.search(
    data=[query],
    search_params={"metric_type": "COSINE"}
)

# distance 越大越相似！
for hit in results[0]:
    print(f"distance: {hit['distance']}")  # 0.95 比 0.5 更相似
```

### 为什么这样设计？

**统一接口**: 不同度量类型有不同的数学定义，Milvus 保留原始值而非统一转换。

### 正确做法

**根据 metric_type 理解 distance**:
- L2: distance 小 = 相似度高
- IP/COSINE: distance 大 = 相似度高

## 反直觉点 4: 向量维度不能随意改变

### 直觉认为

"我可以先用 768 维向量，后来改用 1536 维向量。"

### 实际情况

**Collection 创建后，向量维度不能改变！**

```python
# 创建 Collection（768 维）
schema.add_field("dense", DataType.FLOAT_VECTOR, dim=768)
client.create_collection("my_collection", schema)

# 尝试插入 1536 维向量（会报错！）
client.insert("my_collection", [
    {"id": 1, "dense": [0.1] * 1536}  # ❌ 维度不匹配
])
# 错误: vector dimension mismatch
```

### 为什么这样设计？

**索引依赖**: 向量索引结构依赖于固定的维度，改变维度需要重建整个索引。

### 正确做法

**创建新 Collection**:
```python
# 方案1: 创建新 Collection
client.create_collection("my_collection_v2", new_schema)

# 方案2: 使用支持可变维度的模型（如 OpenAI）
params={
    "provider": "openai",
    "model_name": "text-embedding-3-small",
    "dim": 768  # 可以在 256-1536 之间调整
}
```

## 反直觉点 5: limit 不是"最多返回多少条"

### 直觉认为

"Search 的 limit=10 表示最多返回 10 条结果。"

### 实际情况

**在混合搜索中，limit 是每个子搜索的候选数量！**

```python
# 混合搜索
results = client.hybrid_search(
    search_requests=[req1, req2],
    ranker=RRFRanker(),
    limit=10  # ✅ 最终返回 10 条
)

# 但每个子搜索的 limit 是候选数量
req1 = AnnSearchRequest(
    data=[query],
    limit=100  # ❌ 不是最终返回 100 条，而是候选 100 条
)
```

### 为什么这样设计？

**重排序需要**: 混合搜索需要足够的候选结果才能有效重排序。

### 正确做法

**区分候选数量和最终返回数量**:
```python
# 候选数量（每个子搜索）
req1 = AnnSearchRequest(..., limit=100)  # 候选 100 条
req2 = AnnSearchRequest(..., limit=100)  # 候选 100 条

# 最终返回数量（混合搜索）
results = client.hybrid_search(
    search_requests=[req1, req2],
    limit=10  # 最终返回 10 条
)
```

## 反直觉点 6: Query 不返回向量字段

### 直觉认为

"Query 应该返回所有字段，包括向量字段。"

### 实际情况

**Query 默认不返回向量字段（除非显式指定）！**

```python
# Query 查询
results = client.query(
    collection_name="my_collection",
    filter='id == 1',
    output_fields=["*"]  # 返回所有字段
)

print(results[0].keys())
# 输出: ['id', 'document']  # 没有 'dense' 向量字段！
```

### 为什么这样设计？

**性能优化**: 向量字段通常很大（768-1536 维），默认不返回可以减少数据传输量。

### 正确做法

**显式指定向量字段**:
```python
results = client.query(
    collection_name="my_collection",
    filter='id == 1',
    output_fields=["document", "dense"]  # 显式指定向量字段
)
```

## 反直觉点 7: 分区不会自动平衡数据

### 直觉认为

"Milvus 会自动在分区之间平衡数据，保证每个分区大小相近。"

### 实际情况

**分区是手动管理的，不会自动平衡！**

```python
# 创建分区
client.create_partition("my_collection", "partition_a")
client.create_partition("my_collection", "partition_b")

# 所有数据都插入 partition_a
for i in range(1000000):
    client.insert("my_collection", [data], partition_name="partition_a")

# 结果：partition_a 有 100 万条数据，partition_b 为空
# Milvus 不会自动平衡！
```

### 为什么这样设计？

**用户控制**: 分区通常按业务逻辑划分（如按类别、时间），自动平衡会破坏业务语义。

### 正确做法

**按业务逻辑合理分配数据**:
```python
def insert_with_partition(data):
    # 根据业务逻辑选择分区
    if data["category"] == "tech":
        partition = "tech_partition"
    elif data["category"] == "business":
        partition = "business_partition"
    else:
        partition = "_default"

    client.insert("my_collection", [data], partition_name=partition)
```

## 反直觉点 8: Embedding Functions 的 API 调用是同步的

### 直觉认为

"Embedding Functions 会异步调用嵌入 API，不会阻塞插入操作。"

### 实际情况

**Embedding Functions 的 API 调用是同步的！**

```python
# 插入 1000 条数据
start = time.time()
client.insert("my_collection", documents)  # 会等待所有嵌入 API 调用完成
elapsed = time.time() - start

print(f"耗时: {elapsed}s")  # 可能需要 10-30 秒
```

### 为什么这样设计？

**一致性保证**: 同步调用确保数据插入时向量已生成，避免数据不一致。

### 正确做法

**批量插入以提升性能**:
```python
# ❌ 逐条插入（慢）
for doc in documents:
    client.insert("my_collection", [doc])  # 每次调用一次 API

# ✅ 批量插入（快）
client.insert("my_collection", documents)  # Milvus 批量调用 API
```

## 反直觉点 9: ANN 搜索不保证找到真正最相似的向量

### 直觉认为

"Search 操作会找到真正最相似的 Top-K 向量。"

### 实际情况

**ANN 搜索是近似的，可能错过真正最相似的向量！**

```python
# ANN 搜索
results = client.search(data=[query], limit=10)

# 返回的 10 个结果可能不是真正最相似的 10 个
# 召回率通常在 95-99%，意味着可能错过 1-5% 的真正结果
```

### 为什么这样设计？

**性能权衡**: 精确搜索（kNN）在大规模数据下太慢，ANN 牺牲少量精度换取速度。

### 正确做法

**理解并接受近似性**:
- 对于大多数应用，95-99% 的召回率已经足够
- 如果需要更高召回率，调整索引参数（如增大 nprobe）
- 如果需要 100% 精确，使用小数据集或接受较慢的查询速度

## 反直觉点 10: 标量过滤会降低搜索性能

### 直觉认为

"添加标量过滤条件应该能加速搜索，因为减少了搜索范围。"

### 实际情况

**标量过滤可能降低搜索性能！**

```python
# 无过滤（快）
results = client.search(data=[query], limit=10)  # ~10ms

# 有过滤（可能更慢）
results = client.search(
    data=[query],
    filter='category == "tech"',  # 需要额外的过滤操作
    limit=10
)  # ~15-20ms
```

### 为什么这样设计？

**过滤开销**: 标量过滤需要额外的计算，特别是在预过滤（pre-filtering）模式下。

### 正确做法

**合理使用过滤**:
- 如果过滤条件能显著减少候选集（如过滤掉 90% 数据），性能会提升
- 如果过滤条件只过滤少量数据（如过滤掉 10% 数据），性能可能下降
- 考虑使用分区代替标量过滤

## 反直觉点 11: offset 越大，查询越慢

### 直觉认为

"分页查询的性能应该是恒定的，无论 offset 多大。"

### 实际情况

**offset 越大，查询越慢！**

```python
# 第 1 页（快）
page1 = client.search(data=[query], limit=10, offset=0)  # ~10ms

# 第 100 页（慢）
page100 = client.search(data=[query], limit=10, offset=990)  # ~50ms

# 第 1000 页（很慢）
page1000 = client.search(data=[query], limit=10, offset=9990)  # ~200ms
```

### 为什么这样设计？

**实现机制**: Milvus 需要先找到前 N+offset 个结果，然后跳过前 offset 个，返回后 N 个。

### 正确做法

**避免深度分页**:
```python
# ❌ 深度分页
results = client.search(limit=10, offset=10000)

# ✅ 基于游标的分页（使用 QueryIterator）
iterator = client.query_iterator(
    filter='...',
    batch_size=10
)
```

## 反直觉点 12: 混合搜索不一定比单一搜索更好

### 直觉认为

"混合搜索结合了多种搜索策略，结果一定比单一搜索更好。"

### 实际情况

**混合搜索可能比单一搜索更差！**

```python
# 场景：纯语义查询
query = "What is the meaning of life?"

# Dense 搜索（好）
dense_results = client.search(data=[query], anns_field="dense")
# 返回哲学相关的文档

# Dense + Sparse 混合搜索（可能更差）
hybrid_results = client.hybrid_search(
    search_requests=[dense_req, sparse_req],
    ranker=RRFRanker()
)
# Sparse 可能匹配到无关的包含 "life" 的文档，降低整体质量
```

### 为什么这样设计？

**场景依赖**: 混合搜索适合需要多维度匹配的查询，但对于单一维度的查询可能引入噪音。

### 正确做法

**根据查询类型选择策略**:
- **语义查询**: 使用 Dense 搜索
- **关键词查询**: 使用 Sparse 搜索
- **混合查询**: 使用混合搜索
- **实验验证**: 在实际数据上对比效果

## 反直觉点 13: 向量归一化不是自动的

### 直觉认为

"使用 COSINE 或 IP 度量时，Milvus 会自动归一化向量。"

### 实际情况

**Milvus 不会自动归一化向量！**

```python
# 插入未归一化的向量
vector = [1.0, 2.0, 3.0]  # 模长 = sqrt(14) ≈ 3.74

client.insert("my_collection", [
    {"id": 1, "dense": vector}  # Milvus 不会归一化
])

# 使用 COSINE 搜索
results = client.search(
    data=[[1.0, 2.0, 3.0]],  # 查询向量也未归一化
    search_params={"metric_type": "COSINE"}
)
# 结果可能不准确！
```

### 为什么这样设计？

**用户控制**: 归一化可能改变向量的语义，Milvus 让用户自己决定是否归一化。

### 正确做法

**手动归一化向量**:
```python
import numpy as np

def normalize(vector):
    norm = np.linalg.norm(vector)
    return (vector / norm).tolist() if norm > 0 else vector

# 插入归一化的向量
vector = normalize([1.0, 2.0, 3.0])
client.insert("my_collection", [{"id": 1, "dense": vector}])

# 查询时也要归一化
query = normalize([1.0, 2.0, 3.0])
results = client.search(data=[query], ...)
```

## 反直觉点 14: Collection 的 Schema 不能修改

### 直觉认为

"我可以随时修改 Collection 的 Schema，添加或删除字段。"

### 实际情况

**Collection 创建后，Schema 基本不能修改！**

```python
# 创建 Collection
schema.add_field("id", DataType.INT64, is_primary=True)
schema.add_field("document", DataType.VARCHAR, max_length=1000)
client.create_collection("my_collection", schema)

# 尝试添加新字段（不支持！）
# 无法直接修改 Schema
```

### 为什么这样设计？

**数据一致性**: 修改 Schema 需要重新处理所有现有数据，代价很高。

### 正确做法

**使用动态字段或创建新 Collection**:
```python
# 方案1: 启用动态字段（创建时）
schema.enable_dynamic_field = True

# 方案2: 创建新 Collection
client.create_collection("my_collection_v2", new_schema)

# 方案3: 数据迁移
# 从旧 Collection 导出数据，插入到新 Collection
```

## 反直觉点 15: 删除数据不会立即释放空间

### 直觉认为

"删除数据后，磁盘空间应该立即释放。"

### 实际情况

**删除数据后，空间不会立即释放！**

```python
# 删除数据
client.delete("my_collection", filter='id < 1000')

# 磁盘空间没有立即释放
# 需要等待 Compaction（压缩）操作
```

### 为什么这样设计？

**性能优化**: 立即释放空间需要重写数据文件，代价很高。Milvus 使用延迟压缩策略。

### 正确做法

**手动触发 Compaction**:
```python
from pymilvus import utility

# 触发压缩
utility.compact("my_collection")

# 等待压缩完成
utility.wait_for_compaction_completed("my_collection")
```

## 总结

### 核心反直觉点速记

| 反直觉点 | 直觉 | 实际 | 正确做法 |
|----------|------|------|----------|
| **Insert 重复** | 检查重复 | 不检查 | 用 Upsert |
| **动态字段嵌入** | 会嵌入 | 不嵌入 | 显式定义 |
| **distance 含义** | 越小越相似 | 取决于 metric | 看 metric_type |
| **向量维度** | 可以改 | 不能改 | 创建新 Collection |
| **混合搜索 limit** | 最终返回数 | 候选数 | 区分候选和最终 |
| **Query 返回向量** | 会返回 | 不返回 | 显式指定 |
| **分区平衡** | 自动平衡 | 不平衡 | 手动分配 |
| **API 调用** | 异步 | 同步 | 批量插入 |
| **ANN 精确性** | 精确 | 近似 | 接受近似 |
| **标量过滤性能** | 加速 | 可能降速 | 合理使用 |
| **offset 性能** | 恒定 | 越大越慢 | 避免深度分页 |
| **混合搜索效果** | 一定更好 | 不一定 | 根据场景选择 |
| **向量归一化** | 自动 | 不自动 | 手动归一化 |
| **Schema 修改** | 可以改 | 不能改 | 用动态字段 |
| **删除释放空间** | 立即 | 延迟 | 手动 Compaction |

### 避坑指南

1. **数据插入**: 新数据用 Insert，更新用 Upsert
2. **Embedding Functions**: 只处理 Schema 定义的字段
3. **搜索理解**: 理解 distance 含义，接受 ANN 近似性
4. **性能优化**: 批量操作，避免深度分页，合理使用过滤
5. **Schema 设计**: 提前规划，启用动态字段，向量维度不可变
