## 卡片1：循环的本质 -- 边指向已有节点

**一句话：** LangGraph 中的循环就是"边指向图中已经存在的节点"，形成有向环。

**举例：**

```python
from langgraph.graph import StateGraph, START, END

builder = StateGraph(State)
builder.add_node("a", node_a)
builder.add_node("b", node_b)

builder.add_edge(START, "a")
builder.add_edge("a", "b")
builder.add_edge("b", "a")  # 回边！b 执行完后回到 a，形成循环

graph = builder.compile()
```

这段代码中 `add_edge("b", "a")` 就是回边。`add_edge` 本身不阻止循环——它只禁止 `END -> X` 和 `X -> START`，但允许任意节点之间的回连（源码 `state.py` L785）。

**应用：** ReAct Agent 的经典循环就是这个模式：`agent` 节点调用 LLM 决策 → `tool` 节点执行工具 → 回到 `agent` 节点继续推理，直到 LLM 决定不再调用工具。

```
ReAct 循环结构：
START → agent → tool → agent → tool → ... → agent → END
         ↑       │
         └───────┘  (回边)
```

**类比：** 就像流水线上的质检环节——产品从生产线出来，质检不合格就送回生产线返工，合格才放行出厂。

**关键区别：** 与 DAG（有向无环图）框架不同，LangGraph 从设计之初就支持有向环。这是它区别于 LangChain 传统 Chain 的核心能力——Chain 只能从头到尾走一遍，Graph 可以反复循环。

## 卡片2：条件边 -- 循环的开关

**一句话：** 条件边是控制"继续循环还是结束"的开关，路由函数返回节点名继续，返回 `END` 结束。

**举例：**

```python
from typing import Literal
from langgraph.graph import END

def route(state: State) -> Literal["b", "__end__"]:
    if len(state["aggregate"]) < 7:
        return "b"       # 继续循环
    else:
        return "__end__"  # 退出循环

builder.add_conditional_edges("a", route)
builder.add_edge("b", "a")  # 回边
```

条件边的路由函数在运行时被调用，根据当前状态决定下一步走向。它是 `BranchSpec` 对象（`_branch.py` L83），通过 `path.invoke(state)` 执行路由函数，返回值决定写入哪个节点的触发 channel。

**应用：** 质量检查循环中的通过/不通过判断。比如 RAG 系统中，检索结果相关性评分 > 0.8 就生成回答，否则改写查询重新检索。

**注意事项：**
- 路由函数不能返回 `None` 或 `START`，否则抛出 `ValueError`
- 路由函数的返回值必须是节点名或 `END`（`__end__`）
- 路由函数在 Python 中执行，不在 LLM 中执行，所以它是确定性的

**类比：** 就像考试阅卷——答对了（返回 END）就通过，答错了（返回节点名）就回去重做。

## 卡片3：递归限制 -- 循环的安全阀

**一句话：** `recursion_limit` 是防止无限循环的最后防线，默认 25，超出时抛出 `GraphRecursionError`。

**举例：**

```python
# 设置递归限制
result = graph.invoke(
    {"aggregate": []},
    config={"recursion_limit": 10}  # 最多 10 个 superstep
)

# 精确公式（源码 _loop.py L1120）：
# stop = step + recursion_limit + 1
# 当 step > stop 时，status = "out_of_steps"
```

当 `tick()` 检测到 `step > stop` 时，循环停止，主循环抛出 `GraphRecursionError`（继承自 Python 内置 `RecursionError`）。

**应用：** 生产环境必须设置合理的递归限制。正如 [Optimizing LangGraph Cycles](https://rajatpandit.com/optimizing-langgraph-cycles) 所说："The problem with recursion_limit is that a hard exception is not a user-experience strategy." 它是安全网，不是解决方案。

**类比：** 就像电路中的保险丝——正常情况下不应该触发，但万一短路了，保险丝能防止整个电路烧毁。

## 卡片4：Superstep -- 循环的计数单位

**一句话：** 一个 superstep 可以并行执行多个节点，但只消耗 1 次递归计数。

**举例：**

```
图结构：A → B → (C, D) → A

superstep 1: A 执行           ← 消耗 1 次
superstep 2: B 执行           ← 消耗 1 次
superstep 3: C 和 D 并行执行  ← 只消耗 1 次！
superstep 4: A 再次执行       ← 消耗 1 次

一轮循环 = 3 个 superstep（不是 4 个节点执行）
```

源码中，`tick()` 方法（`_loop.py` L459）一次性通过 `prepare_next_tasks()` 准备当前 superstep 的所有可执行任务，`runner.tick()` 并行执行它们，然后 `after_tick()` 中 step 只递增 1 次。

**应用：** 设置 `recursion_limit` 时，要按"图的拓扑深度 x 循环次数"估算。如果你的循环体有 3 层深度，想循环 5 次，至少需要 `recursion_limit = 15`。

**类比：** 就像数据库的批量操作——一次 `INSERT INTO ... VALUES (...), (...), (...)` 插入 3 行，但只算 1 次操作。

## 卡片5：RemainingSteps -- 优雅降级的秘密武器

**一句话：** `RemainingSteps` 让节点感知剩余步数，在接近限制时主动退出，而不是等框架崩溃。

**举例：**

```python
from langgraph.managed import RemainingSteps

class State(TypedDict):
    aggregate: Annotated[list, operator.add]
    remaining_steps: RemainingSteps  # 框架自动注入，无需手动维护

def route(state: State) -> Literal["b", "__end__"]:
    if state["remaining_steps"] <= 2:
        return "__end__"  # 快到限制了，优雅退出
    return "b"            # 还有余量，继续循环
```

底层实现（`is_last_step.py`）非常简洁：`RemainingSteps = stop - step`。它是一个 ManagedValue，由 `PregelScratchpad` 自动计算，节点通过类型注解注入即可使用。

**应用：** 在接近限制时路由到 fallback 节点，返回部分结果而非崩溃。比如 Agent 在第 8 步发现只剩 2 步了，就返回"基于目前信息的最佳回答"，而不是继续尝试然后硬崩溃。

**对比两种处理方式：**

```python
# ❌ 被动处理：等框架崩溃再 try/catch
try:
    result = graph.invoke(inputs, {"recursion_limit": 10})
except GraphRecursionError:
    result = "处理超时"  # 丢失所有中间状态

# ✅ 主动处理：用 RemainingSteps 提前感知
def route(state):
    if state["remaining_steps"] <= 2:
        return "fallback"  # 优雅降级，保留中间结果
    return "agent"
```

**类比：** 就像手机电量提醒——电量低于 20% 时提醒你省电模式，而不是等到 0% 直接关机。

## 卡片6：Command(goto) -- 节点内部的自路由

**一句话：** Command 让节点在返回值中直接指定下一个节点，同时更新状态，无需预定义条件边。

**举例：**

```python
from langgraph.types import Command
from typing import Literal

def review(state: State) -> Command[Literal["review", "__end__"]]:
    if state["quality_score"] >= 0.9:
        return Command(
            update={"status": "approved"},
            goto="__end__"       # 通过，结束
        )
    return Command(
        update={"revision_count": state["revision_count"] + 1},
        goto="review"           # 自循环，继续审查
    )
```

Command 的 `goto` 最终写入 `branch:to:{node}` channel（`state.py` L1492 `_control_branch()`），与条件边的底层机制完全相同。区别在于 Command 从节点内部发起，且能同时更新状态。

**应用：** 多 Agent 交接场景。比如 researcher Agent 完成调研后，通过 `Command(update={"research": data}, goto="coder")` 把研究结果传给 coder Agent，一步完成状态传递和路由。

**注意：** 必须声明返回类型 `Command[Literal["node_a", "node_b"]]`，否则图无法正确渲染。

**Command 的三个陷阱：**
1. **不覆盖静态边**：如果 node_a 有静态边到 node_b，同时返回 `Command(goto="node_c")`，node_b 和 node_c 都会执行
2. **必须类型注解**：缺少 `Command[Literal[...]]` 返回类型会导致图渲染失败
3. **自循环需要终止条件**：`Command(goto="self")` 如果没有退出路径，会无限循环

## 卡片7：Send -- 并行迭代的利器

**一句话：** Send 向同一节点发送多个不同输入，实现 Map-Reduce 模式的并行迭代。

**举例：**

```python
from langgraph.types import Send

def dispatch(state: State):
    """为每个文档创建一个并行处理任务"""
    return [
        Send("summarize", {"doc": doc, "index": i})
        for i, doc in enumerate(state["documents"])
    ]

builder.add_conditional_edges("dispatch", dispatch)
```

Send 在 `_control_branch()` 中被写入 TASKS channel（而非 `branch:to:{node}` channel），由 `prepare_next_tasks()` 在下一个 superstep 中创建对应的并行任务。每个 Send 携带独立的输入状态，目标节点接收的是 Send 的 `arg`，而非全局状态。

**应用：** 批量文档处理（10 篇文档并行生成摘要）、并行 API 调用（同时查询多个数据源）、多角度分析（同一问题从不同视角并行分析）。

**类比：** 就像外卖平台——一个订单来了 5 份餐，平台同时派 5 个骑手去不同餐厅取餐，而不是一个骑手跑 5 趟。

## 卡片8：TTL 计数器 -- 比 recursion_limit 更精准的控制

**一句话：** 在状态中维护 `steps` 计数器，比框架级 `recursion_limit` 更灵活，可以为不同循环设置不同的最大次数。

**举例：**

```python
class AgentState(TypedDict):
    messages: Annotated[list, operator.add]
    steps: int  # 业务级计数器

def agent_node(state: AgentState):
    return {
        "messages": [process(state)],
        "steps": state["steps"] + 1  # 每次递增
    }

def router(state: AgentState):
    if state["steps"] > 5:
        return "fallback"  # 超过 5 次，优雅降级
    if task_done(state):
        return "__end__"
    return "agent"
```

TTL 计数器的优势：
- **确定性**：纯 Python 逻辑，不受 LLM 幻觉影响
- **灵活性**：不同循环可以有不同的限制（检索循环 3 次，推理循环 10 次）
- **可观测性**：计数器在状态中，可以通过 checkpoint 查看

**应用：** 生产环境中，TTL 计数器是第二道防线（业务终止条件之后，recursion_limit 之前）。它确保即使 LLM 永远不说"完成"，系统也能在可控的步数内优雅退出。

**类比：** 就像银行的每日转账限额——不是等你把账户清空才阻止，而是到了限额就提醒你"今天够了"。

## 卡片9：Generator-Critic -- 自我修正的循环模式

**一句话：** 生成器产出内容 → 评审器评估 → 不合格回到生成器修改，直到通过或达到最大次数。

**举例：**

```python
# Generator-Critic 循环的核心结构
workflow = StateGraph(AgentState)
workflow.add_node("generator", generator_node)
workflow.add_node("critic", critic_node)

workflow.add_edge(START, "generator")
workflow.add_edge("generator", "critic")

def route_after_critique(state):
    if state["critique_history"][-1] == "APPROVED":
        return "__end__"
    if state["revision_number"] >= 3:
        return "__end__"  # 最多修改 3 次
    return "generator"    # 回到生成器重新生成

workflow.add_conditional_edges("critic", route_after_critique)
```

这个模式来自 [ActiveWizards 的深度分析](https://activewizards.com/blog/a-deep-dive-into-langgraph-for-self-correcting-ai-agents)。核心思想是：单独的 Critic 节点提供"旁观者清"的视角，打破 Generator 的隧道视野。

**应用：**
- **代码生成**：生成代码 → 运行测试 → 测试失败则修改代码 → 再测试
- **内容创作**：写初稿 → 审查质量 → 不达标则修改 → 再审查
- **数据清洗**：清洗数据 → 验证质量 → 不合格则重新清洗

**进阶技巧**：Critic 可以用更小更便宜的模型（如 GPT-4o-mini），因为评估比生成简单得多。这样既保证了质量把控，又控制了成本。

**完整的 Generator-Critic 流程图：**

```
START → generator → critic → 条件边判断
                      │
                      ├── "APPROVED" → END
                      ├── revision < 3 → generator（循环回去）
                      └── revision >= 3 → END（强制退出）
```

**生产环境注意事项：**
- 每次循环都是一次 LLM 调用，要控制成本
- Generator 和 Critic 的 prompt 要精心设计，减少不必要的循环
- 建议在状态中记录每次 critique 的内容，供 Generator 参考改进方向

## 卡片10：循环的底层统一模型

**一句话：** 所有循环方式最终都通过 `branch:to:{node}` channel 触发节点，理解这个统一模型就能理解所有循环行为。

**举例：**

```
四种循环方式的底层归一：

1. add_edge("b", "a")
   → 编译时注册：b 的 writers 写入 branch:to:a channel

2. add_conditional_edges("a", route)
   → route 返回 "b" 时，写入 branch:to:b channel

3. Command(goto="a")
   → _control_branch() 写入 branch:to:a channel

4. Send("worker", arg)
   → 写入 TASKS channel（特殊情况，创建新任务）
```

完整的迭代流程：

```
tick() → 检查 step > stop → prepare_next_tasks()
  → runner.tick() → 并行执行所有任务
  → after_tick() → apply_writes() → 写入 channels
  → _put_checkpoint() → step += 1
  → 回到 tick()
```

循环的形成条件：某个节点的 writes 触发了图中已经执行过的节点的 trigger channel。循环的终止条件：没有新任务被触发（`status="done"`）或超出步数限制（`status="out_of_steps"`）。

**应用：** 当你遇到循环行为不符合预期时，回到这个统一模型来分析：检查哪些 channel 被写入了，哪些节点被触发了。这是调试循环问题的终极方法。

**类比：** 就像理解了 HTTP 协议，就能理解所有 Web 框架的请求处理——无论 Express、Django 还是 FastAPI，底层都是 HTTP 请求/响应。
