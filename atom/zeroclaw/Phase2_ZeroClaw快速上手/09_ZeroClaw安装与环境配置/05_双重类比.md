# 05 双重类比 — 用你熟悉的世界理解 ZeroClaw 安装与配置

> **核心思路：** 每个概念同时给出**前端开发类比** + **日常生活类比**，让零 Rust 基础的前端开发者秒懂。
>
> **目标受众：** 有 TypeScript/前端基础、零 Rust 基础的开发者

---

## 为什么需要双重类比？

ZeroClaw 的安装与配置涉及 Rust 编译、TOML 配置、Feature Flags 等对前端开发者来说"陌生"的概念。但这些概念的**底层逻辑**并不新鲜——你在前端开发和日常生活中早已接触过类似的思维模式。

双重类比的目的：

| 类比类型 | 优势 | 适合谁 |
|---------|------|--------|
| 前端开发类比 | 精确映射技术概念，理解"原来是这个意思" | 有前端经验的开发者 |
| 日常生活类比 | 直觉理解，降低认知门槛 | 完全零基础的读者 |

> **阅读建议：** 如果你是前端开发者，重点看"前端类比"部分；如果你帮朋友解释 ZeroClaw，用"日常生活类比"效果更好。

## 类比 1：安装方式 — 5 种获取 ZeroClaw 的路径

### ZeroClaw 概念

ZeroClaw 提供 5 种安装方式，适配不同用户场景：

| 方式 | 命令 | 适用场景 |
|------|------|---------|
| 一键脚本 | `curl ... \| bash` | 首次体验、快速上手 |
| Homebrew | `brew install zeroclaw` | macOS/Linux 日常开发 |
| Cargo Install | `cargo install zeroclaw` | Rust 开发者、定制编译 |
| Docker | `docker compose up -d` | 服务器部署、隔离环境 |
| 预编译二进制 | 下载 tar.gz 解压 | 离线环境、CI/CD |

### 前端类比：npm 生态的 5 种安装体验

你在前端世界早就用过"同一个工具、多种安装方式"了：

```
ZeroClaw 安装方式          →  前端对应方式
─────────────────────────────────────────────────────
一键脚本                   →  npx create-react-app
  curl ... | bash             一条命令，自动检测环境，开箱即用
                              你不需要知道 webpack 怎么配

Homebrew                   →  brew install node / nvm install
  brew install zeroclaw       用系统包管理器统一管理版本
                              brew update 一键升级

Cargo Install              →  npm install -g typescript
  cargo install zeroclaw      从源码编译安装（Cargo）
                              ≈ npm 拉取源码 + 构建 native 模块

Docker                     →  docker run -p 3000:3000 my-app
  docker compose up -d        容器化运行，环境完全隔离
                              不污染宿主机

预编译二进制                →  直接下载 VS Code / Electron 应用
  下载 tar.gz 解压             不需要任何工具链
                              解压即用
```

**关键区别：** `npm install -g` 安装的是 JS 源码 + 依赖树，运行时仍需 Node.js 解释执行。`cargo install` 安装的是编译好的原生二进制——就像你下载 VS Code 一样，双击就跑，不需要任何运行时。

```bash
# 前端世界
npm install -g typescript
which tsc
# /usr/local/bin/tsc → 这其实是一个 Node.js 脚本！
head -1 $(which tsc)
# #!/usr/bin/env node  ← 需要 Node.js 运行时

# ZeroClaw 世界
cargo install zeroclaw
which zeroclaw
# ~/.cargo/bin/zeroclaw → 这是一个原生二进制！
file $(which zeroclaw)
# zeroclaw: Mach-O 64-bit executable arm64  ← 直接是机器码
```

### 日常生活类比：买家具的 5 种方式

想象你要给新家添一张书桌：

```
安装方式        →  买家具方式              →  体验对比
──────────────────────────────────────────────────────────────
一键脚本        →  全屋定制上门服务          量尺寸 → 设计 → 送货 → 安装
                   打一个电话全搞定          你只需要"开门"
                   = curl ... | bash

Homebrew        →  宜家配送服务              从宜家目录选好型号
                   宜家帮你送到家门口        统一渠道，定期有新品
                   = brew install            brew update = 宜家上新

Cargo Install   →  买材料自己做              自己买木板、五金件
                   用自己的工具从头做        最灵活，但需要技能和时间
                   = cargo install           可以完全定制

Docker          →  租用共享办公桌            不买，租一个
                   用完归还，不占家里空间    独立空间，互不干扰
                   = docker run             容器退出 = 退租

预编译二进制    →  买二手成品家具            淘宝/咸鱼买别人做好的
                   送到就能用，不需要工具    不能定制，但最快最省事
                   = 下载 tar.gz            解压 = 搬进家
```

**核心相同点：** 不管你选哪种方式，最终你得到的都是"一张能用的书桌"——类比到 ZeroClaw，就是 PATH 里多了一个可执行的 `zeroclaw` 二进制文件。

## 类比 2：环境变量 — ZEROCLAW_* 体系

### ZeroClaw 概念

ZeroClaw 通过 `ZEROCLAW_*` 前缀的环境变量控制运行时行为：

```bash
# 设置 API Key（最常用）
export ZEROCLAW_API_KEY="sk-or-v1-abc123"

# 切换 Provider
export ZEROCLAW_DEFAULT_PROVIDER="openrouter"

# 指定模型
export ZEROCLAW_DEFAULT_MODEL="anthropic/claude-sonnet-4-6"

# 设置 Gateway 端口
export ZEROCLAW_PORT=42617
```

特点：进程级生效、不写入磁盘、退出即消失。

### 前端类比：process.env + .env 文件 + dotenv

**你一定用过这个：**

```typescript
// React 项目中
const apiUrl = process.env.REACT_APP_API_URL;
const apiKey = process.env.REACT_APP_API_KEY;
```

```bash
# .env.local（不提交到 Git）
REACT_APP_API_URL=https://api.example.com
REACT_APP_API_KEY=sk-abc123
```

ZeroClaw 的环境变量体系和前端的 `.env` 生态**几乎完全对等**：

```
前端世界                          ZeroClaw 世界
────────────────────────────────────────────────────────────
process.env.REACT_APP_*         = ZEROCLAW_* 环境变量前缀
.env / .env.local               = ~/.bashrc 中的 export 语句
.env.production                 = 生产服务器的环境变量
dotenv 包加载 .env              = shell 自动读取 export
GitHub Secrets                  = CI/CD 中注入 ZEROCLAW_API_KEY
process.env 覆盖 .env           = 环境变量优先于 config.toml
```

**为什么 API Key 要走环境变量而不是配置文件？**

```typescript
// 前端：你绝对不会这样做！
// ❌ config.ts
export const API_KEY = "sk-abc123";  // 提交到 Git = 泄露

// ✅ .env.local（.gitignore 排除）
REACT_APP_API_KEY=sk-abc123
```

```bash
# ZeroClaw：同样的道理
# ❌ config.toml（可能被提交到 Git）
api_key = "sk-or-v1-abc123"

# ✅ 环境变量（不落盘）
export ZEROCLAW_API_KEY="sk-or-v1-abc123"
```

### 日常生活类比：手机系统设置

环境变量就像手机的**系统设置**（设置 App）：

```
手机系统设置                      ZeroClaw 环境变量
────────────────────────────────────────────────────────────
WiFi 开关                       = ZEROCLAW_DEFAULT_PROVIDER
  连哪个网络？                      用哪个 AI Provider？
  影响所有需要联网的 App            影响所有 zeroclaw 命令

蓝牙配对                        = ZEROCLAW_API_KEY
  配对密码（不显示明文）            API 密钥（不写入文件）
  一旦断开需要重新配对              进程退出需要重新 export

亮度调节                        = ZEROCLAW_PORT
  全局影响所有界面显示              全局影响 Gateway 监听端口
  随时调，不需要重启               随时改，不需要重装

语言设置                        = ZEROCLAW_DEFAULT_MODEL
  全局影响所有 App 的显示语言       全局影响所有命令使用的模型
  改了之后所有 App 都跟着变         改了之后所有 agent 都用新模型
```

**核心相同点：**
- 📱 手机系统设置 = **全局级**配置，所有 App 共享
- 📱 某个 App 内部设置 = **应用级**配置，只影响自己
- 🔧 ZeroClaw 环境变量 = **全局级**，所有 zeroclaw 命令共享
- 🔧 ZeroClaw config.toml = **持久化级**，写入磁盘

**关键直觉：** 环境变量就是"不需要打开文件就能调的旋钮"，系统设置就是"不需要打开 App 就能改的选项"。

## 类比 3：配置文件 config.toml — 分节管理

### ZeroClaw 概念

`~/.zeroclaw/config.toml` 是 ZeroClaw 的核心配置文件，用 TOML 格式分节管理：

```toml
# 顶层字段（全局设置）
default_provider = "openrouter"
default_model = "anthropic/claude-sonnet-4-6"

[gateway]           # Gateway 网关配置
port = 42617
host = "[::]"

[security]          # 安全策略
allowed_directories = ["/home/user/projects"]

[memory]            # 记忆后端
backend = "sqlite"
path = "~/.zeroclaw/memory.db"

[agent]             # Agent 编排
max_iterations = 20
temperature = 0.7
```

### 前端类比：next.config.js + tsconfig.json + .eslintrc 合体

前端项目通常要维护**一堆**配置文件：

```
前端项目根目录：
├── next.config.js      ← 框架运行时配置
├── tsconfig.json       ← 编译器配置
├── .eslintrc.js        ← 代码规范配置
├── tailwind.config.js  ← 样式配置
├── postcss.config.js   ← CSS 处理配置
├── jest.config.js      ← 测试配置
└── .prettierrc         ← 格式化配置
```

**ZeroClaw 把所有这些"合体"到了一个文件：**

```
前端配置文件                →  config.toml 对应节
───────────────────────────────────────────────────────
next.config.js             →  顶层字段 + [gateway]
  运行时行为配置                服务监听端口、Provider 选择

tsconfig.json              →  [agent]
  编译/执行选项                 Agent 运行参数（温度、迭代次数）

.eslintrc.js               →  [security]
  代码规范 = 安全边界           允许访问的目录、安全策略

tailwind.config.js         →  [memory] + [tools]
  扩展功能配置                  记忆后端、工具扩展

jest.config.js             →  [channel]
  输入输出通道配置              交互通道（CLI/WebSocket/Matrix）
```

**为什么 ZeroClaw 用一个文件而不是多个？**

```
前端：7 个配置文件 × 不同格式（JS/JSON/YAML）= 😵
ZeroClaw：1 个 config.toml × 分节管理 = 😊
```

就像 monorepo 用一个 `package.json` 管理多个子包的依赖，ZeroClaw 用一个 `config.toml` 的多个 `[section]` 管理所有模块的配置。

```toml
# config.toml 的 [section] ≈ package.json 的顶层字段
# [gateway] ≈ "scripts"
# [security] ≈ "engines"
# [memory] ≈ "dependencies"
# [agent] ≈ "main" + "type"
```

### 日常生活类比：汽车仪表盘设置菜单

`config.toml` 就像汽车的**中控大屏设置菜单**：

```
汽车设置菜单                  config.toml
──────────────────────────────────────────────────────
🚗 驾驶设置                 = [agent]
  ├── 运动模式 / 经济模式       temperature（创意 vs 精准）
  ├── 最高车速限制              max_iterations（最大执行步数）
  └── 自动巡航速度              timeout（超时时间）

❄️ 空调设置                 = [gateway]
  ├── 温度 26°C                port = 42617
  ├── 风量 3 档                host = "[::]"
  └── 内循环 / 外循环          protocol（通信协议）

🗺️ 导航设置                 = [memory]
  ├── 记住常去地点              backend = "sqlite"（记忆存储）
  ├── 历史路线                  path = "~/.zeroclaw/memory.db"
  └── 离线地图                  （本地 vs 远程存储）

🔒 安全设置                 = [security]
  ├── 车门自动锁               allowed_directories（允许访问的目录）
  ├── 碰撞预警                 sandbox_mode（沙箱模式）
  └── 儿童锁                   （安全边界）

📻 娱乐设置                 = [channel]
  ├── 蓝牙连接手机              channel-matrix（Matrix 协议）
  ├── FM 电台                  CLI（命令行通道）
  └── USB 播放                 WebSocket（Web 通道）
```

**核心相同点：**
- 🚗 所有设置在**一个中控屏**管理，但分成不同的菜单页（节/section）
- 🚗 每个菜单页管理一个**功能域**，互不干扰
- 🚗 设置保存后**下次启动仍然生效**（持久化）
- 🔧 config.toml 也是一个文件、多个 `[section]`、持久化保存

## 类比 4：Feature Flags — 编译时功能选择

### ZeroClaw 概念

Feature Flags 是 Rust 编译时的"功能开关"，决定哪些模块被编译进最终二进制：

```bash
# 默认安装：只有核心功能（最小构建）
cargo install zeroclaw

# 启用硬件控制 + Matrix 聊天通道
cargo install zeroclaw --features hardware,channel-matrix

# 全部启用（不推荐！）
cargo install zeroclaw --all-features
```

```toml
# Cargo.toml 中的定义
[features]
default = []                    # 默认零 Feature！

# 可选功能
hardware = ["dep:serialport", "dep:rusb"]
channel-matrix = ["dep:matrix-sdk"]
channel-discord = ["dep:serenity"]
memory-redis = ["dep:redis"]
# ... 共 12 个可选 Feature
```

### 前端类比：Webpack tree-shaking + code-splitting

你写过 React 项目吗？想想 Webpack 的 tree-shaking：

```typescript
// 你的代码只 import 了 Button
import { Button } from 'antd';

// Webpack tree-shaking 会把 antd 中你没用到的组件"摇掉"
// Table、Modal、DatePicker... 都不会进入最终 bundle

// 最终 bundle 只包含 Button 相关代码
// bundle size: 50KB  而不是 1.2MB（全量 antd）
```

**ZeroClaw Feature Flags 就是"编译时的 tree-shaking"：**

```
Webpack tree-shaking              ZeroClaw Feature Flags
──────────────────────────────────────────────────────────────
import { Button } from 'antd'  = --features hardware
  只打包 Button 相关代码           只编译硬件控制相关代码

没 import 的组件被"摇掉"        = 没启用的 Feature 不被编译
  Table → 不在 bundle 中           channel-matrix → 不在二进制中

bundle size 变小                = 二进制体积变小
  50KB vs 1.2MB                    8.8MB vs 15MB+

构建时决定                      = 编译时决定
  webpack.config.js                cargo build --features "..."
```

**但有一个关键区别：**

```
Webpack tree-shaking：自动的（分析 import 语句）
  → 你不需要手动选，bundler 帮你决定

ZeroClaw Feature Flags：手动的（你指定 --features）
  → 你主动决定要哪些功能

为什么不能自动？因为 Rust 编译不像 JS 打包——
Feature 可能引入系统级依赖（USB 驱动、加密库），
这些依赖是否存在取决于操作系统，不能自动判断。
```

另一个前端类比是 **Next.js 的动态导入 + code-splitting**：

```typescript
// Next.js：运行时按需加载（动态导入）
const HeavyChart = dynamic(() => import('./HeavyChart'), {
  loading: () => <p>Loading...</p>,
});

// ZeroClaw：编译时按需包含（Feature Flags）
// cargo install zeroclaw --features hardware
// → 只有启用了 hardware 才会编译 USB/串口代码
```

| 对比维度 | Next.js 动态导入 | ZeroClaw Feature Flags |
|---------|-----------------|----------------------|
| 时机 | **运行时**加载 | **编译时**包含 |
| 粒度 | 组件级 | 功能模块级 |
| 未使用的代码 | 在服务器上，按需下载 | 根本不存在于二进制中 |
| 好处 | 首屏加载更快 | 二进制更小、依赖更少 |

### 日常生活类比：自助餐选菜

Feature Flags 就像自助餐的**选菜策略**：

```
自助餐模式                        ZeroClaw Feature Flags
──────────────────────────────────────────────────────────────
基础套餐（88元）                = default = []（零 Feature 默认）
  ├── 白米饭 ✅                     CLI 交互 ✅
  ├── 3 个家常菜 ✅                 Agent 核心循环 ✅
  └── 汤 ✅                         基础工具集 ✅
  够吃饱了！                        够跑起来了！

加菜（每道 +18元）              = --features hardware
  ├── 龙虾 🦞                       USB/串口控制
  ├── 和牛 🥩                       Matrix 聊天通道
  └── 鲍鱼 🫧                       Redis 记忆后端
  按需添加，按量付费                按需启用，编译时间增加

豪华全选（388元）               = --all-features
  全部菜品都上！                    所有 12 个功能全编译！
  ├── 吃不完浪费 ❌                  二进制臃肿 ❌
  ├── 桌子摆不下 ❌                  编译时间暴增 ❌
  └── 胃受不了 ❌                    引入不必要的系统依赖 ❌
```

**核心相同点：**
- 🍽️ 基础套餐足够吃饱 = 默认零 Feature 足够用（CLI + Agent 核心）
- 🍽️ 按需加菜 = 按需启用 Feature（只加你真正需要的）
- 🍽️ 全选不是最优解 = `--all-features` 不是推荐做法
- 🍽️ 每加一道菜，花更多钱/等更久 = 每启用一个 Feature，多一些编译时间/体积

**另一个生活类比——乐高积木：**

```
乐高基础包（200 块基础砖）      = ZeroClaw 默认安装
  能搭出一个小房子                 能运行基础 Agent

乐高扩展包（汽车轮子套件）      = --features hardware
  给小房子加个车库                 给 Agent 加硬件控制能力

乐高超级全家桶（5000 块）       = --all-features
  ├── 很多砖你永远用不上            很多功能你永远用不到
  ├── 箱子超重搬不动                二进制体积膨胀
  └── 找特定零件更困难              排查问题更复杂
```

## 类比 5：配置优先级链 — 谁说了算？

### ZeroClaw 概念

当多个地方设置了同一个配置项时，ZeroClaw 按以下优先级决定"听谁的"：

```
优先级从高到低：

1. 环境变量 (ZEROCLAW_*)        ──→ 最高！临时覆盖一切
2. 命令行参数 (--model xxx)     ──→ 本次执行有效
3. config.toml 配置文件         ──→ 持久化设置
4. 代码默认值                   ──→ 最低，兜底方案
```

```bash
# 实际示例：model 字段的优先级
# config.toml 中设置了 default_model = "gpt-4"
# 但你在命令行指定了 --model claude-sonnet-4-6
# 还设置了环境变量 export ZEROCLAW_DEFAULT_MODEL="gemini-pro"

# 最终用哪个？→ gemini-pro（环境变量优先级最高）
```

### 前端类比：CSS specificity（特异性）

这个概念前端开发者再熟悉不过了！

```css
/* CSS 优先级规则 */
p { color: black; }                    /* 元素选择器 → 最低 */
.text { color: blue; }                 /* class 选择器 */
#title { color: green; }              /* ID 选择器 */
<p style="color: red">                 /* inline style */
p { color: purple !important; }        /* !important → 最高 */
```

**完美映射到 ZeroClaw 配置优先级：**

```
CSS 优先级                    ZeroClaw 配置优先级
──────────────────────────────────────────────────────────────
元素选择器 (p)              = 代码默认值
  最基础的样式                  最基础的默认配置
  优先级最低                    优先级最低

class 选择器 (.text)        = config.toml
  可复用的样式规则              可复用的持久化配置
  写在 .css 文件中              写在配置文件中

ID 选择器 (#title)          = 命令行参数 (--model xxx)
  针对特定元素                  针对本次执行
  优先级较高                    优先级较高

inline style (style="")     = 环境变量 (ZEROCLAW_*)
  直接写在元素上                直接设在 shell 中
  覆盖几乎所有规则              覆盖几乎所有配置

!important                  = （无对应）
  强制覆盖，不推荐使用          ZeroClaw 没有这一层
  CSS 的"核武器"               环境变量已经是最高层
```

**代码对照：**

```css
/* CSS：4 层覆盖 */
p { color: black; }              /* ← 默认值 */
.my-class { color: blue; }      /* ← config.toml */
#my-id { color: green; }        /* ← --model flag */
style="color: red"              /* ← 环境变量 */

/* 最终颜色 = red（inline style 优先级最高） */
```

```bash
# ZeroClaw：4 层覆盖
# 默认值:    model = "gpt-3.5-turbo"        (代码内置)
# 配置文件:  default_model = "gpt-4"         (config.toml)
# 命令行:    --model claude-sonnet-4-6           (本次执行)
# 环境变量:  ZEROCLAW_DEFAULT_MODEL="gemini" (export)

# 最终模型 = gemini（环境变量优先级最高）
```

另一个前端开发类比是 **Next.js 的环境变量分层**：

```bash
# Next.js 环境变量优先级（从低到高）
.env                    # 所有环境共享        ← 代码默认值
.env.production         # 生产环境            ← config.toml
.env.local              # 本地覆盖（不提交）  ← 命令行参数
NEXT_PUBLIC_*=xxx       # 命令行直接设置      ← 环境变量
```

### 日常生活类比：快递地址优先级

想象你在电商平台下单：

```
快递地址优先级                    ZeroClaw 配置优先级
──────────────────────────────────────────────────────────────
平台默认地址                    = 代码默认值
  注册时填的地址                    编译时写死的默认值
  很久没更新了                      最基础的兜底方案
  优先级最低                        优先级最低

常用收货地址簿                  = config.toml
  你设置的几个常用地址              你写好的配置文件
  公司 / 家 / 父母家               不同项目 / 不同场景
  持久化保存                        持久化保存

最近一次下单地址                = 命令行参数 (--model)
  系统自动记住上次的选择            本次执行时指定的参数
  比地址簿优先                      比配置文件优先
  但只影响"当前这单"               只影响"当前这次执行"

手写修改地址                    = 环境变量
  下单时手动输入新地址              在终端手动 export
  优先级最高                        优先级最高
  覆盖一切                          覆盖一切
```

**场景故事：**

> 小明的地址簿里存了"公司"和"家"两个地址（= config.toml 的两个配置项），他通常发到公司（= config.toml 的默认配置）。
>
> 有一天他在手机上下单，系统显示上次发到了"家"（= 命令行参数覆盖了默认配置）。
>
> 但这次他要发到酒店（临时出差），于是**手写了新地址**（= 环境变量，最高优先级，覆盖一切）。
>
> 最终快递送到了酒店。

```bash
# 对应 ZeroClaw 命令
# config.toml: default_model = "gpt-4"          ← 公司地址
# 上次使用: --model claude-sonnet-4-6               ← 家的地址（不影响，命令行不持久化）
export ZEROCLAW_DEFAULT_MODEL="gemini-pro"       # ← 酒店地址（环境变量覆盖一切）
zeroclaw agent -m "Hello"
# 最终使用: gemini-pro ✅
```

## 类比总结对照表

### 完整对照表

| ZeroClaw 概念 | 前端开发类比 | 日常生活类比 | 核心相似点 |
|--------------|-------------|-------------|-----------|
| **5 种安装方式** | npm/npx/yarn/docker/下载 | 买家具的 5 种方式 | 殊途同归，结果一样 |
| ├ 一键脚本 | `npx create-react-app` | 全屋定制上门服务 | 一条命令，全自动 |
| ├ Homebrew | `brew install node` | 宜家配送 | 包管理器统一管理 |
| ├ Cargo Install | `npm install -g` | 自己买材料做 | 从源码构建 |
| ├ Docker | `docker run` | 租共享办公桌 | 隔离环境 |
| └ 预编译二进制 | 下载 VS Code | 买二手成品 | 即下即用 |
| **环境变量** | `process.env.*` + `.env` | 手机系统设置 | 全局影响、不持久化 |
| ├ ZEROCLAW_API_KEY | REACT_APP_API_KEY | 蓝牙配对密码 | 安全敏感，不落盘 |
| ├ ZEROCLAW_DEFAULT_MODEL | NEXT_PUBLIC_API_URL | 语言设置 | 全局影响所有操作 |
| └ ZEROCLAW_PORT | PORT=3000 | 亮度调节 | 随时调整 |
| **config.toml 分节** | next.config + tsconfig 合体 | 汽车中控设置菜单 | 一处管理、分区独立 |
| ├ [gateway] | next.config.js | 空调设置 | 运行时服务配置 |
| ├ [security] | .eslintrc | 安全设置 | 规则与边界 |
| ├ [memory] | 数据库配置 | 导航/记忆设置 | 数据存储 |
| └ [agent] | tsconfig.json | 驾驶模式设置 | 执行行为参数 |
| **Feature Flags** | Webpack tree-shaking | 自助餐选菜 | 按需选择、最小化 |
| ├ default = [] | 最小 bundle | 基础套餐 | 默认最精简 |
| ├ --features xxx | import 特定组件 | 单独加菜 | 按需启用 |
| └ --all-features | import 全量库 | 豪华全选 | 不推荐 |
| **配置优先级** | CSS specificity | 快递地址优先级 | 高优先级覆盖低优先级 |
| ├ 代码默认值 | 元素选择器 | 平台默认地址 | 最低优先级 |
| ├ config.toml | class 选择器 | 常用地址簿 | 持久化配置 |
| ├ 命令行参数 | ID 选择器 | 上次下单地址 | 本次有效 |
| └ 环境变量 | inline style | 手写地址 | 最高优先级 |

### 速记口诀

> **安装像买家具**（5 种方式，结果都是一张桌子）
>
> **环境变量像手机设置**（全局影响，随时调整）
>
> **config.toml 像汽车中控**（一屏管理，分区设置）
>
> **Feature Flags 像自助餐**（基础够用，按需加菜）
>
> **优先级像 CSS 特异性**（inline style 覆盖 class 覆盖 element）
