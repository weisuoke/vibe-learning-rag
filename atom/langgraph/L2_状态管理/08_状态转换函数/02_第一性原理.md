# 状态转换函数 - 第一性原理

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题。

不是"别人怎么做我就怎么做"，而是"这件事最本质的需求是什么？从这个需求出发，最合理的解决方案是什么？"

对于状态转换函数，我们要问的第一性问题是：

> **如何让图中的每个节点安全、可预测地修改共享状态？**

---

## 状态转换函数的第一性原理

### 1. 最基础的定义

**状态转换函数 = 接收当前状态，返回状态变更的纯函数**

```python
def node_function(state: State) -> dict:
    # 读取状态 → 处理逻辑 → 返回部分更新
    return {"field": new_value}
```

仅此而已！没有更基础的了。

就像工厂流水线上的一个工位：零件（状态）传过来，工人（节点函数）看一眼
零件当前的样子，做自己负责的那道工序，然后把改动的部分标记出来，传给下一
个工位。工人不需要知道整条流水线长什么样，只需要专注自己的工序。

再换一个角度理解——你在填一张表格。表格有很多栏（状态字段），每个审批人
（节点）拿到完整的表格，但只填写自己负责的那几栏，然后把表格传给下一个人。
没有人会去改别人已经填好的内容。

**状态转换函数就是 LangGraph 中每个节点的"工作合同"：拿到什么、做什么、
交出什么，清清楚楚。**

把这个概念放到 LangGraph 中：

```
用户输入 → 节点A(state) → {更新A} → 节点B(state') → {更新B} → 节点C(state'') → 最终输出
              ↑                          ↑                          ↑
         接收完整状态              接收合并后的状态            接收再次合并后的状态
         返回部分更新              返回部分更新                返回部分更新
```

每个节点执行完后，LangGraph 框架自动把节点返回的部分更新合并回完整状态，
然后把更新后的完整状态传给下一个节点。节点本身不需要关心合并逻辑。

---

### 2. 为什么需要状态转换函数？

**核心问题：多个节点共享同一个状态对象，如何确保更新是可控的、可追踪的？**

想象一下没有状态转换函数的世界：

```python
# ❌ 危险的做法：节点直接修改共享状态
shared_state = {"query": "", "docs": [], "answer": ""}

def bad_retrieve_node(state):
    state["docs"] = search(state["query"])  # 直接修改！
    state["answer"] = ""                     # 顺手改了不该改的字段！
    del state["query"]                       # 甚至删除了字段！

def bad_generate_node(state):
    state["answer"] = llm(state["docs"])    # 直接修改！
    state["docs"] = []                       # 清空了检索结果！
```

这种做法有三个致命问题：

1. **不可追踪**：谁改了什么？什么时候改的？无从得知
2. **不可预测**：节点可能修改任意字段，产生意想不到的副作用
3. **不可测试**：节点的行为依赖外部状态，无法独立测试

**状态转换函数的解决方案：**

```python
# ✅ 安全的做法：节点返回部分更新
def good_retrieve_node(state: State) -> dict:
    docs = search(state["query"])
    return {"docs": docs}  # 只返回要更新的字段

def good_generate_node(state: State) -> dict:
    answer = llm(state["docs"])
    return {"answer": answer}  # 只返回要更新的字段
```

节点不能直接修改输入状态，只能通过返回值"申请"修改。框架负责审核和执行
这些修改。这就像银行柜员不能直接改账本，只能通过系统提交交易申请。

---

### 3. 状态转换函数的三层价值

#### 价值1：数据隔离与安全

**节点只能通过返回值修改状态，不能直接修改输入状态。**

这是一种 immutable pattern（不可变模式）。节点拿到的 `state` 参数是只读的，
所有修改都必须通过返回一个新的字典来表达。

```python
def process_node(state: State) -> dict:
    # ❌ 不要这样做（虽然技术上可以，但违反约定）
    # state["docs"].append(new_doc)

    # ✅ 正确做法：返回新值
    return {"docs": state["docs"] + [new_doc]}
```

**类比：银行柜员只能通过系统操作，不能直接改账本。** 每笔交易都有记录，
可以审计追踪。如果柜员能直接改账本，银行就乱套了。

#### 价值2：可组合性

**每个节点是独立的转换单元，可以自由组合、重排、替换。**

因为每个节点的接口都是统一的 `State -> Partial<State>`，所以节点之间
没有直接依赖，只通过状态间接通信。

```python
# 节点可以自由组合
graph.add_node("retrieve", retrieve_node)
graph.add_node("rerank", rerank_node)
graph.add_node("generate", generate_node)

# 今天的流程：检索 → 重排序 → 生成
graph.add_edge("retrieve", "rerank")
graph.add_edge("rerank", "generate")

# 明天想跳过重排序？直接改边就行，节点代码不用动
# graph.add_edge("retrieve", "generate")
```

**类比：乐高积木。** 每块积木（节点）有标准的接口（凸起和凹槽），可以
和任何其他积木拼接。你可以随意组合出不同的造型，而不需要修改积木本身。

#### 价值3：可测试性

**纯函数，给定输入必有确定输出，可以独立单元测试每个节点。**

```python
# 单元测试一个节点，完全不需要启动整个图
def test_retrieve_node():
    # 构造输入状态
    test_state = {"query": "什么是 RAG？", "docs": [], "answer": ""}

    # 调用节点函数
    result = retrieve_node(test_state)

    # 验证输出
    assert "docs" in result
    assert len(result["docs"]) > 0
    assert isinstance(result["docs"], list)

def test_generate_node():
    test_state = {
        "query": "什么是 RAG？",
        "docs": ["RAG 是检索增强生成...", "RAG 结合了检索和生成..."],
        "answer": ""
    }

    result = generate_node(test_state)

    assert "answer" in result
    assert len(result["answer"]) > 0
```

**类比：数学函数 f(x) = y。** 给定 x=2，f(2) 永远等于同一个 y。不会因为
你在哪里调用、什么时候调用而改变结果。这种确定性让测试变得简单可靠。

**三层价值的关系：**

```
价值1：数据隔离     → "谁能改什么？"     → 通过返回值控制
价值2：可组合性     → "怎么拼接节点？"   → 统一的函数签名
价值3：可测试性     → "怎么验证正确？"   → 纯函数可独立测试

三层层层递进，从安全到灵活到可靠，构成状态转换函数的核心价值。
```

### 4. 从第一性原理推导 LangGraph 状态转换

**推理链：**

```
1. 需要多步骤处理复杂任务
   ↓
2. 将任务分解为多个处理节点
   ↓
3. 节点之间需要共享数据（状态）
   ↓
4. 每个节点需要读取状态、处理、返回更新
   ↓
5. 这就是状态转换函数：State -> Partial<State>
   ↓
6. 框架负责将部分更新合并回完整状态
   ↓
7. 下一个节点接收更新后的完整状态
```

让我们逐步展开这条推理链：

**第 1-3 步：问题的本质**

一个 RAG 系统需要完成多个步骤：理解问题、检索文档、重排序、生成回答。
这些步骤不能塞进一个函数里——那样会变成一团意大利面条代码。所以我们把
每个步骤拆成独立的节点。

但节点之间需要传递数据：检索节点的输出是生成节点的输入。最自然的方式
就是用一个共享的状态对象来承载这些数据。

```python
# 状态就是节点之间的"共享白板"
class RAGState(TypedDict):
    query: str           # 用户问题（理解节点写入）
    docs: list[str]      # 检索结果（检索节点写入）
    answer: str          # 最终回答（生成节点写入）
```

**第 4-5 步：状态转换函数的诞生**

既然有了共享状态，每个节点的工作就很清晰了：

```python
# 每个节点都遵循同一个模式：
# 1. 接收完整状态
# 2. 读取自己需要的字段
# 3. 执行处理逻辑
# 4. 返回要更新的字段

def understand_node(state: RAGState) -> dict:
    refined = refine_query(state["query"])
    return {"query": refined}

def retrieve_node(state: RAGState) -> dict:
    docs = vector_search(state["query"])
    return {"docs": docs}

def generate_node(state: RAGState) -> dict:
    answer = llm_generate(state["query"], state["docs"])
    return {"answer": answer}
```

注意：每个节点返回的不是完整状态，而是**部分更新**（Partial State）。
这是一个关键的设计决策——节点只声明自己改了什么，不需要复制整个状态。

**第 6-7 步：框架的合并机制**

节点返回部分更新后，LangGraph 框架负责合并：

```python
# 框架内部的合并逻辑（简化示意）
current_state = {"query": "什么是 RAG？", "docs": [], "answer": ""}

# 检索节点返回 {"docs": ["文档1", "文档2"]}
# 框架合并后：
current_state = {"query": "什么是 RAG？", "docs": ["文档1", "文档2"], "answer": ""}

# 生成节点返回 {"answer": "RAG 是检索增强生成..."}
# 框架合并后：
current_state = {"query": "什么是 RAG？", "docs": ["文档1", "文档2"], "answer": "RAG 是检索增强生成..."}
```

合并规则取决于字段是否配置了 reducer：
- **无 reducer**：直接覆盖（LastValue 语义）
- **有 reducer**：调用 reducer 函数合并（如 `operator.add` 追加列表）

### 5. 源码验证

LangGraph 源码中，状态转换函数的核心设计体现在 `graph/state.py` 的
`StateGraph` 类文档字符串中：

```python
# langgraph/graph/state.py
class StateGraph:
    """A graph whose nodes communicate by reading and writing to a shared state.

    The signature of each node is State -> Partial<State>.

    Each state key can optionally be annotated with a reducer function that
    will be used to aggregate the values of that key received from multiple nodes.
    The signature of a reducer function is (Value, Value) -> Value.
    """
```

这段文档精确地表达了状态转换函数的第一性原理：
- 节点通过**读写共享状态**来通信
- 每个节点的签名是 `State -> Partial<State>`
- 每个字段可以配置 reducer 来控制合并行为

**状态更新的完整链路（源码级）：**

```
节点执行 → 返回部分状态更新（dict）
    ↓
ChannelWrite 收集写入条目
    ↓
apply_writes() 按 Channel 分组
    ↓
channel.update(values) 应用更新
    ├── LastValue: 直接覆盖（只允许一个值）
    ├── BinaryOperatorAggregate: 应用 reducer 函数
    └── EphemeralValue: 临时值（步骤结束后清除）
    ↓
下一个节点接收更新后的完整状态
```

**关键源码片段——Channel 更新机制：**

```python
# channels/last_value.py - 无 reducer 时的默认行为
class LastValue:
    def update(self, values):
        if len(values) != 1:
            raise InvalidUpdateError(
                "Can receive only one value per step. "
                "Use an Annotated key to handle multiple values."
            )
        self.value = values[-1]  # 直接覆盖

# channels/binop.py - 有 reducer 时的行为
class BinaryOperatorAggregate:
    def update(self, values):
        for value in values:
            self.value = self.operator(self.value, value)  # 应用 reducer
```

这两个类是状态转换函数的"执行引擎"。当节点返回 `{"docs": new_docs}` 时：
- 如果 `docs` 字段没有 reducer → `LastValue.update()` → 直接覆盖
- 如果 `docs` 字段有 `operator.add` reducer → `BinaryOperatorAggregate.update()` → 列表追加

### 6. 状态转换函数与 RAG 开发的关系

状态转换函数在 RAG 开发中无处不在，因为 RAG 管道本身就是一个多步骤的
状态转换过程：

**一个典型的 RAG 管道就是一系列状态转换函数的组合：**

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

class RAGState(TypedDict):
    query: str
    docs: list[str]
    answer: str
    messages: Annotated[list, add_messages]  # 对话历史用 reducer 追加

# 每个节点都是一个状态转换函数
def retrieve_docs(state: RAGState) -> dict:
    """检索相关文档"""
    docs = vector_store.search(state["query"], top_k=5)
    return {"docs": docs}

def generate_answer(state: RAGState) -> dict:
    """基于检索结果生成回答"""
    context = "\n\n".join(state["docs"])
    answer = llm.invoke(f"根据以下内容回答：{context}\n问题：{state['query']}")
    return {"answer": answer}

# 组装图
graph = StateGraph(RAGState)
graph.add_node("retrieve", retrieve_docs)
graph.add_node("generate", generate_answer)
graph.add_edge(START, "retrieve")
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", END)
```

每个节点只关心自己的职责，通过状态转换函数的统一接口串联起来。如果你想
替换检索策略、增加重排序步骤、或者修改生成逻辑，只需要修改对应的节点函数
或调整边的连接，其他节点完全不受影响。

---

### 7. 一句话总结第一性原理

**状态转换函数是 LangGraph 的原子操作单元，通过"接收完整状态 → 返回部分
更新"的模式，实现了安全、可组合、可测试的数据流转。**

记住这个核心公式：

```
状态转换函数 = State -> Partial<State>
            = 接收完整状态 → 返回部分更新
            = 纯函数（无副作用）+ 部分更新（只改需要改的）+ 框架合并（自动处理）
```

## 本节要点回顾

1. **状态转换函数的本质**：接收完整状态，返回部分更新的纯函数
2. **为什么需要**：多节点共享状态时，需要可控、可追踪的更新机制
3. **三层价值**：数据隔离（安全）→ 可组合性（灵活）→ 可测试性（可靠）
4. **推导链**：多步骤任务 → 节点分解 → 共享状态 → 部分更新 → 框架合并
5. **源码验证**：`StateGraph` 文档字符串明确定义 `State -> Partial<State>`
6. **合并机制**：`LastValue`（覆盖）和 `BinaryOperatorAggregate`（reducer）
7. **RAG 应用**：RAG 管道天然就是一系列状态转换函数的组合
