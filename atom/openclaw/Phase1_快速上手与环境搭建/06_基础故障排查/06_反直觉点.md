# OpenClaw 基础故障排查 - 反直觉点

**列出 OpenClaw 基础故障排查中最常见的 3 个误区，帮助你避免常见错误。**

---

## 误区 1：doctor 能修复所有问题 ❌

### 为什么错？

**错误观点**：
"运行 `openclaw doctor --repair` 就能解决所有问题"

**正确理解**：
doctor 只能修复**配置相关**的问题，无法解决：
- 网络连接问题
- API 密钥无效
- 端口被其他进程占用
- 权限不足（需要手动授权）
- 通道服务器故障

**来源**: `reference/source_doctor_command.md`

### 为什么人们容易这样错？

**心理原因**：
1. **工具万能心理**：看到"自动修复"就认为能解决一切
2. **懒惰倾向**：希望一个命令解决所有问题
3. **文档误读**：只看到"自动修复"，没看到"配置问题"的限定

**类比**：
- **后端开发类比**：就像 `npm install` 能安装依赖，但无法修复代码逻辑错误
- **日常生活类比**：就像感冒药能治感冒，但无法治疗骨折

### 正确做法

```bash
# 1. 先运行 doctor 检查配置
openclaw doctor

# 2. 如果是配置问题，使用 repair
openclaw doctor --repair

# 3. 如果不是配置问题，查看日志
openclaw logs --follow

# 4. 根据日志特征定位问题类型
# - EADDRINUSE → 端口冲突，需要停止其他进程
# - unauthorized → 认证问题，检查 API 密钥
# - network error → 网络问题，检查连接
```

**来源**: `reference/source_troubleshooting_main.md`

## 误区 2：日志越多越好 ❌

### 为什么错？

**错误观点**：
"应该开启所有日志级别，记录所有信息"

**正确理解**：
- 过多的日志会**淹没关键信息**
- 日志文件过大会**影响性能**
- 调试日志包含**敏感信息**（API 密钥、用户数据）

**推荐做法**：
- 生产环境：`info` 或 `warn` 级别
- 开发环境：`debug` 级别
- 故障排查：临时开启 `debug`，问题解决后关闭

**来源**: `reference/source_debugging_tools.md`

### 为什么人们容易这样错？

**心理原因**：
1. **信息焦虑**：担心错过关键信息
2. **过度准备**：认为记录越多越安全
3. **不了解成本**：不知道日志过多的副作用

**类比**：
- **后端开发类比**：就像数据库查询，`SELECT *` 看似方便，但会影响性能
- **日常生活类比**：就像录像监控，24小时录制会占满硬盘，关键时刻反而找不到

### 正确做法

```typescript
// 根据环境设置日志级别
const logLevel = process.env.NODE_ENV === 'production' ? 'info' : 'debug';

// 故障排查时临时开启 debug
openclaw logs --follow --level debug

// 问题解决后恢复正常级别
openclaw config set logging.level info
```

**来源**: `reference/source_debugging_tools.md`

## 误区 3：重启能解决一切 ❌

### 为什么错？

**错误观点**：
"遇到问题就重启 Gateway，不需要诊断"

**正确理解**：
- 重启只是**临时缓解**，不是根本解决
- 不诊断就重启会**掩盖真正问题**
- 频繁重启会**丢失状态**（会话、配对等）
- 某些问题重启后**会立即复现**

**应该做的**：
1. 先诊断问题根源
2. 尝试针对性修复
3. 只有在必要时才重启

**来源**: `reference/search_openclaw_troubleshooting.md`

### 为什么人们容易这样错？

**心理原因**：
1. **Windows 思维**：习惯了"重启解决90%问题"
2. **避免思考**：重启比诊断简单
3. **立竿见影**：重启后问题暂时消失，产生"解决了"的错觉

**类比**：
- **后端开发类比**：就像服务器内存泄漏，重启后暂时正常，但问题会再次出现
- **日常生活类比**：就像汽车发动机异响，熄火重启后暂时消失，但故障依然存在

### 正确做法

```bash
# ❌ 错误做法：直接重启
openclaw gateway restart

# ✅ 正确做法：先诊断
# 1. 检查状态
openclaw gateway status

# 2. 查看日志
openclaw logs --follow

# 3. 识别问题类型
# - 配置错误 → doctor --repair
# - 端口冲突 → 停止其他实例
# - 内存泄漏 → 需要修复代码

# 4. 针对性修复后再重启
openclaw doctor --repair
openclaw gateway restart
```

**来源**: `reference/source_gateway_troubleshooting.md`

## 其他常见误区

### 误区 4：配置越复杂越好

**错误观点**：
"应该配置所有可选项，让系统更强大"

**正确理解**：
- 配置越多，出错概率越高
- 大部分默认值已经优化
- 只配置必需的选项

**来源**: `reference/source_doctor_command.md`

### 误区 5：所有通道都要配置

**错误观点**：
"应该配置所有 8 个通道，以备不时之需"

**正确理解**：
- 只配置实际使用的通道
- 未使用的通道会增加复杂度
- 按需添加通道

**来源**: `reference/source_channels_troubleshooting.md`

### 误区 6：升级后不需要检查

**错误观点**：
"升级后直接使用，不需要运行 doctor"

**正确理解**：
- 升级后**必须**运行 `openclaw doctor --repair`
- 配置格式可能变化
- 新版本可能有新的必需配置

**来源**: `reference/search_doctor_command.md`
