# 核心概念 05：状态组织与模块化

> 通过嵌套状态、子图隔离、Managed Values 和 Input/Output Schema 分离，将复杂工作流的状态拆分为职责清晰、互不干扰的模块。

## 一句话定义

**状态组织与模块化是将复杂工作流的状态按职责拆分——用嵌套结构分组、用子图隔离边界、用 Managed Values 分离运行时上下文、用 Input/Output Schema 控制对外接口——让每个模块只关心自己需要的数据。**

---

## 为什么需要状态模块化？

### 问题：状态膨胀与混乱

当工作流变复杂时，状态会迅速膨胀：

```
一个 RAG + 多轮对话 + 工具调用系统的状态：
├── query, messages, session_id          # 对话相关
├── chunks, embeddings, scores, context  # 检索相关
├── tool_name, tool_args, tool_result    # 工具相关
├── draft, refined_answer, citations     # 生成相关
├── db_conn, api_client, config          # 运行时资源
├── user_id, permissions                 # 用户上下文
└── retry_count, error_log, step_count   # 流程控制
```

如果把这 20+ 个字段全部平铺在一个 TypedDict 里：

1. **认知负担**：每个节点函数都看到全部字段，不知道哪些和自己有关
2. **命名冲突**：检索的 `scores` 和重排序的 `scores` 容易撞名
3. **状态污染**：运行时资源（db_conn）混在业务数据里，序列化时出问题
4. **接口模糊**：外部调用者不知道该传哪些字段、能拿到哪些结果

### 解决方案：四层模块化策略

```
策略                    解决的问题              对应机制
──────────────────    ──────────────────    ──────────────
1. 嵌套状态分组        字段太多、认知负担        Pydantic 嵌套模型
2. 子图状态隔离        模块间互相干扰            子图独立 Schema
3. Managed Values     运行时资源污染状态        ManagedValueSpec
4. Input/Output 分离  对外接口不清晰            独立的输入输出 Schema
```

## 1. 扁平 vs 嵌套状态

### 扁平状态：简单工作流的首选

当字段少于 8-10 个时，扁平结构最直观：

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import add_messages

# ✅ 简单工作流：扁平就够了
class SimpleRAGState(TypedDict):
    query: str
    context: str
    answer: str
    messages: Annotated[list, add_messages]
```

**优点**：读写直接、无嵌套访问开销、序列化简单。

### 嵌套状态：复杂领域的组织方式

当字段超过 10 个，且能按领域分组时，用 Pydantic 嵌套模型：

```python
from pydantic import BaseModel, Field
from typing import Annotated
from langgraph.graph import add_messages

# 按领域分组的嵌套模型
class ConversationInfo(BaseModel):
    """对话相关状态"""
    session_id: str = ""
    turn_count: int = 0
    summary: str = ""

class RetrievalInfo(BaseModel):
    """检索相关状态"""
    chunks: list[str] = Field(default_factory=list)
    scores: list[float] = Field(default_factory=list)
    context: str = ""

class GenerationInfo(BaseModel):
    """生成相关状态"""
    draft: str = ""
    answer: str = ""
    citations: list[str] = Field(default_factory=list)

# 顶层状态：清晰的分组
class ComplexRAGState(BaseModel):
    query: str = ""
    messages: Annotated[list, add_messages]
    conversation: ConversationInfo = Field(default_factory=ConversationInfo)
    retrieval: RetrievalInfo = Field(default_factory=RetrievalInfo)
    generation: GenerationInfo = Field(default_factory=GenerationInfo)
```

**节点访问嵌套字段**：

```python
def retrieve_node(state: ComplexRAGState):
    """检索节点：只关心 retrieval 分组"""
    query = state.query
    # 更新嵌套字段——返回完整的嵌套对象
    return {
        "retrieval": RetrievalInfo(
            chunks=[f"chunk about {query}"],
            scores=[0.95],
            context=f"context for {query}",
        )
    }

def generate_node(state: ComplexRAGState):
    """生成节点：读 retrieval，写 generation"""
    context = state.retrieval.context
    return {
        "generation": GenerationInfo(
            draft=f"基于 {context} 的草稿",
            answer=f"基于 {context} 的最终答案",
            citations=["source_1"],
        )
    }
```

### 扁平 vs 嵌套：选择指南

```
维度              扁平状态                    嵌套状态
──────          ──────────                  ──────────
字段数量          < 10 个                     > 10 个
领域边界          模糊（都是一个流程）          清晰（多个子领域）
Reducer 需求      每个字段独立 reducer          嵌套对象整体替换
序列化成本        低                           中（嵌套对象需要递归序列化）
代码可读性        字段少时好                    字段多时好
典型场景          简单 RAG、单轮问答            多轮对话 + 检索 + 工具调用
```

**关键权衡**：嵌套状态的 Reducer 通常是整体替换（LastValue），无法对嵌套对象的单个字段做增量更新。如果需要对嵌套内的列表做 append 操作，考虑把该字段提升到顶层。

## 2. 子图状态隔离

> 详细原理见 `03_核心概念_03_子图状态隔离.md`，这里聚焦模块化设计视角。

### 核心思想

每个子图是一个独立的"状态模块"，有自己的 Schema：

```
父图 (OrchestratorState)
├── query, context, answer          ← 对外接口
│
├── 检索子图 (RetrievalState)
│   ├── query, context              ← 共享键（与父图通信）
│   └── chunks, scores, embeddings  ← 私有键（父图看不到）
│
└── 生成子图 (GenerationState)
    ├── query, context, answer      ← 共享键
    └── draft, revision_notes       ← 私有键
```

### 代码示例：模块化 RAG 系统

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# ========== 检索模块（独立子图）==========
class RetrievalState(TypedDict):
    query: str           # 共享：从父图接收
    context: str         # 共享：返回给父图
    chunks: list[str]    # 私有：内部中间结果
    scores: list[float]  # 私有：内部中间结果

def embed_and_search(state: RetrievalState):
    query = state["query"]
    chunks = [f"文档片段A关于{query}", f"文档片段B关于{query}"]
    scores = [0.95, 0.82]
    return {"chunks": chunks, "scores": scores}

def rerank_and_format(state: RetrievalState):
    # 只取高分片段
    top_chunks = [
        c for c, s in zip(state["chunks"], state["scores"]) if s > 0.9
    ]
    return {"context": " | ".join(top_chunks)}

retrieval_builder = StateGraph(RetrievalState)
retrieval_builder.add_node("search", embed_and_search)
retrieval_builder.add_node("rerank", rerank_and_format)
retrieval_builder.add_edge(START, "search")
retrieval_builder.add_edge("search", "rerank")
retrieval_builder.add_edge("rerank", END)
retrieval_graph = retrieval_builder.compile()

# ========== 生成模块（独立子图）==========
class GenerationState(TypedDict):
    query: str       # 共享
    context: str     # 共享
    answer: str      # 共享
    draft: str       # 私有

def draft_answer(state: GenerationState):
    return {"draft": f"[草稿] 根据 '{state['context']}' 回答 '{state['query']}'"}

def refine_answer(state: GenerationState):
    return {"answer": state["draft"].replace("[草稿] ", "")}

gen_builder = StateGraph(GenerationState)
gen_builder.add_node("draft", draft_answer)
gen_builder.add_node("refine", refine_answer)
gen_builder.add_edge(START, "draft")
gen_builder.add_edge("draft", "refine")
gen_builder.add_edge("refine", END)
generation_graph = gen_builder.compile()

# ========== 协调器（父图）==========
class OrchestratorState(TypedDict):
    query: str      # 输入
    context: str    # 中间传递
    answer: str     # 输出

parent = StateGraph(OrchestratorState)
parent.add_node("retrieve", retrieval_graph)
parent.add_node("generate", generation_graph)
parent.add_edge(START, "retrieve")
parent.add_edge("retrieve", "generate")
parent.add_edge("generate", END)

app = parent.compile()

# 执行
result = app.invoke({"query": "什么是向量数据库？", "context": "", "answer": ""})
print(result["answer"])
# 根据 '文档片段A关于什么是向量数据库？' 回答 '什么是向量数据库？'
```

**模块化收益**：
- 检索子图可以独立测试：`retrieval_graph.invoke({"query": "test", ...})`
- 替换检索实现不影响生成模块
- 父图只有 3 个字段，极其简洁

## 3. Managed Values（运行时上下文）

### 什么是 Managed Values？

Managed Values 是一种特殊的状态字段，用于提供**运行时上下文**——它们不是用户数据，而是执行环境信息。

**核心特征**：
- 由框架在每一步自动计算，节点只读
- **不会被序列化到 Checkpoint**——不占存储空间
- 不能出现在 Input/Output Schema 中（源码强制校验）
- 通过 `Annotated[T, ManagedValue子类]` 声明

### 源码实现

```python
# 来源: libs/langgraph/langgraph/managed/base.py
class ManagedValue(ABC, Generic[V]):
    @staticmethod
    @abstractmethod
    def get(scratchpad: PregelScratchpad) -> V: ...

ManagedValueSpec = type[ManagedValue]
```

每个 `ManagedValue` 子类实现 `get()` 方法，从 `PregelScratchpad`（执行上下文）中提取值。

### 内置 Managed Values

```python
# 来源: libs/langgraph/langgraph/managed/is_last_step.py

class IsLastStepManager(ManagedValue[bool]):
    @staticmethod
    def get(scratchpad: PregelScratchpad) -> bool:
        return scratchpad.step == scratchpad.stop - 1

IsLastStep = Annotated[bool, IsLastStepManager]

class RemainingStepsManager(ManagedValue[int]):
    @staticmethod
    def get(scratchpad: PregelScratchpad) -> int:
        return scratchpad.stop - scratchpad.step

RemainingSteps = Annotated[int, RemainingStepsManager]
```

### 使用示例

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.managed.is_last_step import IsLastStep, RemainingSteps
from langgraph.graph import StateGraph, START, END

class AgentState(TypedDict):
    query: str
    answer: str
    # Managed Values：运行时上下文，不存入 Checkpoint
    is_last_step: IsLastStep
    remaining_steps: RemainingSteps

def agent_node(state: AgentState):
    """节点可以读取运行时上下文"""
    if state["is_last_step"]:
        # 最后一步了，直接返回当前结果
        return {"answer": f"[最终] 回答 {state['query']}"}

    remaining = state["remaining_steps"]
    if remaining > 3:
        # 还有足够步数，可以做更深入的检索
        return {"answer": f"[深度检索] 回答 {state['query']}"}
    else:
        # 步数不多了，快速返回
        return {"answer": f"[快速] 回答 {state['query']}"}

builder = StateGraph(AgentState)
builder.add_node("agent", agent_node)
builder.add_edge(START, "agent")
builder.add_edge("agent", END)
graph = builder.compile()

result = graph.invoke({"query": "什么是RAG？", "answer": ""})
print(result)
# {'query': '什么是RAG？', 'answer': '[深度检索] 回答 什么是RAG？'}
# 注意：is_last_step 和 remaining_steps 不会出现在输出中
```

### 为什么不直接放在普通状态里？

```
普通状态字段                          Managed Values
──────────────                      ──────────────
✗ 会被序列化到 Checkpoint             ✓ 不占存储空间
✗ 需要节点手动更新                    ✓ 框架自动计算
✗ 可能被节点错误修改                  ✓ 只读，不可修改
✗ 外部调用者可能误传                  ✓ 不在 Input Schema 中
```

### 源码校验：Managed Values 不能出现在 Input/Output Schema

```python
# 来源: libs/langgraph/langgraph/graph/state.py
def _add_schema(self, schema, /, allow_managed=True):
    channels, managed, type_hints = _get_channels(schema)
    if managed and not allow_managed:
        names = ", ".join(managed)
        schema_name = getattr(schema, "__name__", "")
        raise ValueError(
            f"Invalid managed channels detected in {schema_name}: {names}."
            " Managed channels are not permitted in Input/Output schema."
        )
```

在 `__init__` 中，`input_schema` 和 `output_schema` 调用 `_add_schema` 时传入 `allow_managed=False`，确保 Managed Values 只能出现在内部状态中。

## 4. Input/Output Schema 分离

### 为什么需要分离？

内部状态往往比对外接口复杂得多。如果把所有内部字段都暴露给调用者：
- 调用者不知道该传哪些字段
- 返回值包含大量无关的中间数据
- API 接口不稳定（内部重构会影响外部）

**Input/Output Schema 分离**就是给图定义"对外接口"——调用者只看到精简的输入输出，内部状态随便折腾。

### 源码机制

```python
# 来源: libs/langgraph/langgraph/graph/state.py
class StateGraph:
    def __init__(
        self,
        state_schema,
        *,
        input_schema=None,
        output_schema=None,
    ):
        self.state_schema = state_schema
        # 如果不指定，默认等于 state_schema
        self.input_schema = input_schema or state_schema
        self.output_schema = output_schema or state_schema

        self._add_schema(self.state_schema)
        self._add_schema(self.input_schema, allow_managed=False)
        self._add_schema(self.output_schema, allow_managed=False)
```

### 代码示例

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END, add_messages

# ========== 输入 Schema：调用者需要提供的 ==========
class InputSchema(TypedDict):
    query: str

# ========== 输出 Schema：调用者能拿到的 ==========
class OutputSchema(TypedDict):
    answer: str
    sources: list[str]

# ========== 内部状态：包含所有中间数据 ==========
class InternalState(TypedDict):
    query: str                                      # 来自输入
    answer: str                                     # 输出字段
    sources: list[str]                              # 输出字段
    messages: Annotated[list, add_messages]          # 内部：对话历史
    chunks: list[str]                               # 内部：检索片段
    scores: list[float]                             # 内部：相关性分数
    draft: str                                      # 内部：草稿

def retrieve(state: InternalState):
    query = state["query"]
    return {
        "chunks": [f"片段1: {query}相关内容", f"片段2: {query}补充内容"],
        "scores": [0.95, 0.78],
    }

def generate(state: InternalState):
    top_chunks = [c for c, s in zip(state["chunks"], state["scores"]) if s > 0.8]
    context = "\n".join(top_chunks)
    return {
        "draft": f"基于以下内容回答：\n{context}",
        "answer": f"回答：{context}",
        "sources": [f"source_{i}" for i in range(len(top_chunks))],
    }

# 关键：指定三个不同的 Schema
builder = StateGraph(
    InternalState,                # 内部状态（完整）
    input_schema=InputSchema,     # 输入接口（精简）
    output_schema=OutputSchema,   # 输出接口（精简）
)
builder.add_node("retrieve", retrieve)
builder.add_node("generate", generate)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

graph = builder.compile()

# 调用者只需要传 query
result = graph.invoke({"query": "什么是向量数据库？"})
print(result)
# {'answer': '回答：片段1: 什么是向量数据库？相关内容', 'sources': ['source_0']}
# 注意：chunks, scores, draft, messages 都不会出现在输出中！
```

### 三层 Schema 的关系

```
                    ┌─────────────────────────────┐
                    │       InternalState          │
                    │  (内部状态：所有字段)          │
                    │                             │
                    │  query ←── InputSchema       │
                    │  messages                    │
                    │  chunks                      │
                    │  scores                      │
                    │  draft                       │
                    │  answer ──→ OutputSchema     │
                    │  sources ──→ OutputSchema    │
                    └─────────────────────────────┘

InputSchema ⊆ InternalState    （输入字段必须是内部状态的子集）
OutputSchema ⊆ InternalState   （输出字段必须是内部状态的子集）
```

## 5. 状态分组策略

### 策略一：按领域分组

将状态按业务领域划分，每个领域一个嵌套对象或子图：

```
按领域分组
├── 用户信息    → user: UserInfo (session_id, permissions, preferences)
├── 对话管理    → conversation: ConversationInfo (messages, turn_count, summary)
├── 检索结果    → retrieval: RetrievalInfo (chunks, scores, context)
├── 生成控制    → generation: GenerationInfo (draft, answer, citations)
└── 运行时      → Managed Values (is_last_step, remaining_steps)
```

**适用场景**：多领域交叉的复杂系统（RAG + 对话 + 工具调用）。

### 策略二：按生命周期分组

将状态按数据的生命周期划分：

```
按生命周期分组
├── 输入层      → InputSchema (query, user_id)
├── 处理层      → InternalState (chunks, scores, draft, ...)
├── 输出层      → OutputSchema (answer, sources)
└── 运行时层    → Managed Values (is_last_step)
```

**适用场景**：流水线式工作流，数据从输入到输出单向流动。

### 策略三：混合策略（推荐）

实际项目中，通常结合两种策略：

```python
from typing import Annotated
from typing_extensions import TypedDict
from pydantic import BaseModel, Field
from langgraph.graph import add_messages
from langgraph.managed.is_last_step import IsLastStep

# --- 按领域分组的嵌套模型 ---
class RetrievalResult(BaseModel):
    chunks: list[str] = Field(default_factory=list)
    scores: list[float] = Field(default_factory=list)
    context: str = ""

# --- 按生命周期分层的 Schema ---
class InputSchema(TypedDict):
    """对外输入接口"""
    query: str

class OutputSchema(TypedDict):
    """对外输出接口"""
    answer: str

class WorkflowState(TypedDict):
    """内部完整状态"""
    # 输入层
    query: str
    # 处理层（按领域嵌套）
    retrieval: RetrievalResult
    messages: Annotated[list, add_messages]
    draft: str
    # 输出层
    answer: str
    # 运行时层
    is_last_step: IsLastStep
```

### 命名规范

```
规则                              示例                          说明
──────────────────              ──────────                    ──────
顶层字段用 snake_case            query, answer                 Python 标准
嵌套对象用名词                    retrieval, conversation        表示"一组数据"
布尔字段用 is_/has_ 前缀         is_complete, has_context       语义清晰
列表字段用复数                    chunks, scores, messages       表示"多个"
私有中间字段加 _ 前缀（可选）      _draft, _raw_chunks           暗示"不对外暴露"
Managed Values 用描述性名称       is_last_step, remaining_steps  框架约定
```

## 6. 模块化设计清单

在设计复杂工作流的状态时，按以下清单逐项检查：

### 字段分类

- [ ] 列出所有需要的状态字段
- [ ] 将每个字段标记为：输入 / 输出 / 内部 / 运行时
- [ ] 输入字段 → 放入 `input_schema`
- [ ] 输出字段 → 放入 `output_schema`
- [ ] 运行时资源 → 使用 Managed Values 或 `RunnableConfig`
- [ ] 内部字段 → 只在 `state_schema` 中

### 分组决策

- [ ] 字段 < 10 个？→ 扁平结构
- [ ] 字段 > 10 个？→ 按领域嵌套分组
- [ ] 有独立可测试的子流程？→ 拆为子图
- [ ] 子图之间需要通信？→ 定义共享键

### 子图边界

- [ ] 每个子图的共享键是否最小化？（只暴露必要的字段）
- [ ] 子图是否可以独立测试？
- [ ] 子图的 Schema 是否自包含？（不依赖父图的特定字段名）

### 安全检查

- [ ] Managed Values 没有出现在 Input/Output Schema 中？
- [ ] 运行时资源（db_conn、api_client）没有放在普通状态中？
- [ ] 嵌套对象的更新方式是否明确？（整体替换 vs 增量更新）

## 7. 源码引用：Schema 处理流程

### `_add_schema()` 的完整逻辑

```python
# 来源: libs/langgraph/langgraph/graph/state.py
def _add_schema(self, schema, /, allow_managed=True):
    if schema not in self.schemas:
        _warn_invalid_state_schema(schema)
        # 1. 解析 Schema 的所有字段
        channels, managed, type_hints = _get_channels(schema)

        # 2. 校验 Managed Values 是否允许
        if managed and not allow_managed:
            names = ", ".join(managed)
            raise ValueError(
                f"Invalid managed channels: {names}. "
                "Not permitted in Input/Output schema."
            )

        # 3. 注册到 schemas 字典
        self.schemas[schema] = {**channels, **managed}

        # 4. 合并 channels（检查冲突）
        for key, channel in channels.items():
            if key in self.channels:
                if self.channels[key] != channel:
                    if isinstance(channel, LastValue):
                        pass  # LastValue 允许重复定义
                    else:
                        raise ValueError(
                            f"Channel '{key}' already exists with a different type"
                        )
            else:
                self.channels[key] = channel

        # 5. 合并 managed values（检查冲突）
        for key, mv in managed.items():
            if key in self.managed:
                if self.managed[key] != mv:
                    raise ValueError(
                        f"Managed value '{key}' already exists with a different type"
                    )
            else:
                self.managed[key] = mv
```

### `_get_channels()` 的字段分类逻辑

```python
# 来源: libs/langgraph/langgraph/graph/state.py
def _get_channels(schema):
    type_hints = get_type_hints(schema, include_extras=True)
    all_keys = {
        name: _get_channel(name, typ)
        for name, typ in type_hints.items()
        if name != "__slots__"
    }
    return (
        # 普通 Channel（LastValue, BinaryOperatorAggregate 等）
        {k: v for k, v in all_keys.items() if isinstance(v, BaseChannel)},
        # Managed Values（IsLastStep, RemainingSteps 等）
        {k: v for k, v in all_keys.items() if is_managed_value(v)},
        # 原始类型提示
        type_hints,
    )
```

**关键洞察**：`_get_channels()` 自动将 `Annotated[T, ManagedValue子类]` 识别为 Managed Value，将其从普通 Channel 中分离出来。这就是为什么 Managed Values 不会被序列化——它们根本不在 Channel 系统中。

## 小结

1. **扁平 vs 嵌套**：字段少用扁平，字段多按领域嵌套分组，注意嵌套对象的 Reducer 是整体替换
2. **子图隔离**：每个子图有独立 Schema，通过共享键或函数包装与父图通信，私有字段完全不可见
3. **Managed Values**：运行时上下文（`IsLastStep`、`RemainingSteps`）不存入 Checkpoint，不能出现在 Input/Output Schema
4. **Input/Output 分离**：用独立的 Schema 定义对外接口，隐藏内部实现细节
5. **分组策略**：按领域分组 + 按生命周期分层，实际项目中混合使用
6. **设计清单**：字段分类 → 分组决策 → 子图边界 → 安全检查
