# 核心概念5：可观测性与调试

> LangChain 管道一旦超过三个组件，出了问题你根本不知道错在哪。可观测性就是给管道装上"X 光机"——通过 Callbacks 回调、Tracing 追踪、Graph 可视化和 LangSmith 集成，让每一步的输入输出、耗时、错误全部透明可见。本文基于 `langchain_core.callbacks`、`langchain_core.tracers` 和 `RunnableGraph` 模块源码分析。

---

## 一句话定义

**可观测性与调试是通过 Callbacks 回调、Tracing 追踪、Graph 可视化和 LangSmith 集成，让 LangChain 管道的运行过程透明可监控的能力。**

把它想成汽车的仪表盘——你不需要打开引擎盖就能知道速度、油量、发动机温度。没有仪表盘的车能开，但出了问题你只能靠猜。

---

## 为什么可观测性重要？

LLM 应用的调试和传统应用完全不同，面临四个独特挑战：

| 挑战 | 传统应用 | LLM 应用 |
|------|----------|----------|
| **黑盒问题** | 函数输入输出确定 | LLM 输出不可预测，同样的输入可能得到不同结果 |
| **多步骤链路** | 调用栈清晰 | 错误可能发生在 Retriever、Prompt、LLM、Parser 任何环节 |
| **性能瓶颈** | profiler 一目了然 | 不知道是检索慢、模型慢还是后处理慢 |
| **成本追踪** | 计算资源固定 | 每次调用消耗不同数量的 token，直接关联费用 |

---

## 工具1：Callbacks 回调系统

### 原理

Callbacks 是 LangChain 的事件通知机制。每个 Runnable 在执行的关键节点触发回调，你只需注册 Handler 就能监听。就像 DOM 事件监听——不需要修改组件代码，`addEventListener` 即可。

### 关键事件

```
组件生命周期事件：
├── Chain:     on_chain_start → on_chain_end / on_chain_error
├── LLM:       on_llm_start → on_llm_new_token → on_llm_end / on_llm_error
├── Tool:      on_tool_start → on_tool_end / on_tool_error
└── Retriever: on_retriever_start → on_retriever_end / on_retriever_error
```

每个事件携带 `run_id`（唯一标识）、`parent_run_id`（父级调用）和 `serialized`（组件元信息），形成完整调用树。

### 代码示例：计时回调

```python
from langchain_core.callbacks import BaseCallbackHandler
import time

class TimingCallback(BaseCallbackHandler):
    """记录每个步骤的执行时间"""

    def __init__(self):
        self.timings = {}
        self._start_times = {}

    def on_chain_start(self, serialized, inputs, *, run_id, **kwargs):
        self._start_times[run_id] = time.time()
        print(f"  [START] {serialized.get('name', 'unknown')}")

    def on_chain_end(self, outputs, *, run_id, **kwargs):
        if run_id in self._start_times:
            elapsed = time.time() - self._start_times[run_id]
            self.timings[str(run_id)] = elapsed
            print(f"  [DONE]  {elapsed:.3f}s")

    def on_llm_end(self, response, *, run_id, **kwargs):
        if run_id in self._start_times:
            elapsed = time.time() - self._start_times[run_id]
            if response.llm_output:
                usage = response.llm_output.get("token_usage", {})
                print(f"  [LLM]   {elapsed:.3f}s | tokens: {usage}")

    def on_chain_error(self, error, *, run_id, **kwargs):
        print(f"  [ERROR] {type(error).__name__}: {error}")

# ===== 使用回调 =====
timing = TimingCallback()

# 方式1：通过 config 传入（推荐，仅本次调用生效）
result = chain.invoke(
    {"concept": "RAG"},
    config={"callbacks": [timing]},
)

# 方式2：构造时绑定（所有调用都生效）
chain_with_cb = chain.with_config(callbacks=[timing])
```

## 工具2：Tracing 追踪

### 追踪 vs 回调

| 维度 | Callbacks | Tracing |
|------|-----------|---------|
| 粒度 | 单个事件 | 完整调用树 |
| 存储 | 内存中，用完即弃 | 持久化到 LangSmith |
| 可视化 | 需要自己实现 | LangSmith 控制台自带 |
| 适用场景 | 开发调试 | 生产监控 + 评估 |

Tracing 本质上是特殊的 Callback——`LangChainTracer` 继承自 `BaseCallbackHandler`，把所有事件组织成树状结构并上报到 LangSmith。

### LangSmith 集成

```python
import os

# 只需设置环境变量，LangChain 自动启用追踪
os.environ["LANGSMITH_TRACING"] = "true"
os.environ["LANGSMITH_API_KEY"] = "your-api-key"
os.environ["LANGSMITH_PROJECT"] = "my-rag-project"

# 之后所有 LangChain 调用自动上报到 https://smith.langchain.com
result = chain.invoke({"concept": "RAG"})
```

### @traceable 装饰器

```python
from langsmith import traceable

@traceable(name="document_preprocessing")
def preprocess_documents(docs: list[str]) -> list[str]:
    """自定义函数也能被追踪，输入输出出现在 LangSmith 追踪树中"""
```

## 工具4：astream_events 流式事件

`astream_events` 以流式方式输出管道执行过程中的所有事件，实时观察每个组件的行为。

```python
async for event in chain.astream_events(
    {"concept": "RAG"}, version="v2",
):
    kind = event["event"]
    if kind == "on_chat_model_stream":
        # LLM 流式输出的每个 token
        content = event["data"]["chunk"].content
        if content:
            print(content, end="", flush=True)
    elif kind == "on_retriever_end":
        docs = event["data"]["output"]
        print(f"\n[检索] 返回 {len(docs)} 个文档")
    elif kind == "on_chain_error":
        print(f"\n[错误] {event['data']}")
```

过滤特定事件：

```python
async for event in chain.astream_events(
    {"concept": "RAG"}, version="v2",
    include_names=["ChatOpenAI"],   # 只看特定组件
    include_types=["on_llm_end"],   # 只看特定事件类型
):
    print(event)
```

## 调试最佳实践

### 实践1：分步调试——先拆后合

```python
prompt = ChatPromptTemplate.from_template("总结：{text}")
model = ChatOpenAI(model="gpt-4o-mini")
parser = StrOutputParser()
test_input = {"text": "LangChain 是一个 LLM 应用开发框架..."}

# 拆开每一步单独执行，检查类型和内容
prompt_result = prompt.invoke(test_input)
print(f"Prompt 输出: {type(prompt_result)} -> {prompt_result}")

model_result = model.invoke(prompt_result)
print(f"Model 输出: {type(model_result)} -> {model_result}")

parser_result = parser.invoke(model_result)
print(f"Parser 输出: {type(parser_result)} -> {parser_result}")

# 确认每步正常后再组合
chain = prompt | model | parser
```

### 实践2：verbose / debug 全局开关

```python
from langchain.globals import set_verbose, set_debug

set_verbose(True)   # 打印每个组件的输入输出
set_debug(True)     # 打印更详细的内部信息

result = chain.invoke({"text": "测试输入"})

# 调试完记得关闭
set_verbose(False)
set_debug(False)
```

### 实践3：RunnableLambda 插桩

```python
from langchain_core.runnables import RunnableLambda

def inspect(data, label=""):
    """插桩函数：打印中间数据但不修改它"""
    print(f"[INSPECT {label}] type={type(data).__name__}, value={str(data)[:200]}")
    return data  # 原样返回，不影响管道

# 在任意位置插入探针
chain_debug = (
    prompt
    | RunnableLambda(lambda x: inspect(x, "after_prompt"))
    | model
    | RunnableLambda(lambda x: inspect(x, "after_model"))
    | parser
)
```

### 实践4：异常兜底 with_fallbacks

```python
main_model = ChatOpenAI(model="gpt-4o")
fallback_model = ChatOpenAI(model="gpt-4o-mini")

# gpt-4o 调用失败时自动切换到 gpt-4o-mini
safe_model = main_model.with_fallbacks([fallback_model])
chain = prompt | safe_model | parser
```

## 在 RAG 开发中的应用

### 检索质量监控

```python
class RetrievalMonitor(BaseCallbackHandler):
    def on_retriever_end(self, documents, *, run_id, **kwargs):
        print(f"[检索监控] 返回 {len(documents)} 个文档")
        for i, doc in enumerate(documents):
            source = doc.metadata.get("source", "unknown")
            print(f"  [{i+1}] source={source}, {doc.page_content[:80]}...")
        if not documents:
            print("  [警告] 未检索到任何文档！")
```

### 端到端 RAG 追踪树（LangSmith 视图）

```
rag_chain (3.2s)
├── retriever (0.8s)
│   ├── embedding (0.3s)      <- query 向量化
│   └── vector_search (0.5s)  <- 向量检索
├── prompt (0.01s)            <- 组装上下文
└── llm (2.4s)                <- 生成回答
    └── tokens: 1,247
```

一眼看出：LLM 调用占 75% 时间，是主要瓶颈。

### 成本控制

```python
class CostGuard(BaseCallbackHandler):
    def __init__(self, max_cost_usd: float = 0.10):
        self.max_cost = max_cost_usd

    def on_llm_end(self, response, *, run_id, **kwargs):
        if response.llm_output:
            usage = response.llm_output.get("token_usage", {})
            total = usage.get("total_tokens", 0)
            estimated_cost = total * 0.00000015  # gpt-4o-mini 估算
            if estimated_cost > self.max_cost:
                raise RuntimeError(f"成本超限: ${estimated_cost:.4f}")
```

```

│   ├── 流式观察 → astream_events

│
└── 生产阶段
    ├── 全链路追踪 → LangSmith Tracing
    ├── 成本监控   → get_openai_callback / CostGuard
    ├── 质量评估   → LangSmith Evaluation
    └── 异常兜底   → with_fallbacks
```

## 小结

可观测性不是"锦上添花"，而是 LLM 应用从"能跑"到"能用"的关键一步。四个工具各有分工：Callbacks 管事件监听，Tracing 管全链路追踪，Graph 管结构可视化，astream_events 管实时流式观察。开发阶段用 verbose + 分步调试快速定位问题，生产阶段用 LangSmith 做持续监控和评估。
