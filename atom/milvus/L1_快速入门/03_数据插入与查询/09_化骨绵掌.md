## 卡片 1: Embedding Functions

### 核心要点
Milvus 2.6 的 Data-in, Data-out 模式，直接插入原始文本，自动调用嵌入 API。

### 记忆口诀
```
文本进，文本出，
向量化，库来做，
五步变三步，
RAG更简单。
```

### 关键代码
```python
# 配置 Embedding Function
schema.add_function(Function(
    name="openai_embedding",
    input_field_names=["document"],
    output_field_names=["dense"],
    params={"provider": "openai", "model_name": "text-embedding-3-small"}
))

# 直接插入原始文本
client.insert({"document": "原始文本"})

# 直接用文本查询
results = client.search(data=["查询文本"])
```

### 常见陷阱
- ❌ Embedding Functions 不处理动态字段
- ❌ 输入字段必须是 VARCHAR 类型
- ❌ 输入字段不能为 NULL

## 卡片 2: Insert vs Upsert

### 核心要点
Insert 不检查主键重复，Upsert 检查并更新。

### 记忆口诀
```
Insert 快但可能重复，
Upsert 慢但保证唯一，
新数据用 Insert，
更新用 Upsert。
```

### 关键代码
```python
# Insert: 不检查重复
client.insert("collection", [
    {"id": 1, "document": "text"}
])

# Upsert: 检查并更新
client.upsert("collection", [
    {"id": 1, "document": "updated text"}
])
```

### 常见陷阱
- ❌ Insert 可能创建重复主键的记录
- ❌ 忘记使用 Upsert 导致数据重复
- ❌ 在不需要更新时使用 Upsert 降低性能

## 卡片 3: Search vs Query

### 核心要点
Search 是向量相似度搜索，Query 是标量字段精确查询。

### 记忆口诀
```
Search 找相似，
Query 找精确，
Search 用向量，
Query 用过滤。
```

### 关键代码
```python
# Search: 向量相似度搜索
results = client.search(
    data=[query_vector],
    anns_field="dense",
    limit=10
)

# Query: 标量字段查询
results = client.query(
    filter='category == "tech"',
    limit=10
)
```

### 常见陷阱
- ❌ 混淆 Search 和 Query 的使用场景
- ❌ Query 默认不返回向量字段
- ❌ Search 的 distance 含义取决于 metric_type

## 卡片 4: ANN vs kNN

### 核心要点
ANN 是近似搜索，牺牲 1-5% 精度换取 10,000 倍速度。

### 记忆口诀
```
kNN 精确但慢，
ANN 近似但快，
大数据用 ANN，
小数据用 kNN。
```

### 关键对比
| 特性 | kNN | ANN |
|------|-----|-----|
| **精度** | 100% | 95-99% |
| **速度** | O(n) | O(log n) |
| **适用** | 小数据 | 大数据 |

### 常见陷阱
- ❌ 期望 ANN 返回 100% 精确结果
- ❌ 在小数据集上过度优化 ANN 参数
- ❌ 不理解召回率的含义

## 卡片 5: 混合搜索

### 核心要点
Dense + Sparse 混合搜索，结合语义理解和关键词匹配。

### 记忆口诀
```
Dense 捕捉语义，
Sparse 匹配关键词，
混合搜索最佳，
RRF 重排序。
```

### 关键代码
```python
# Dense 搜索请求
dense_req = AnnSearchRequest(
    data=[query],
    anns_field="dense_vector",
    limit=100  # 候选数量
)

# Sparse 搜索请求
sparse_req = AnnSearchRequest(
    data=[query],
    anns_field="sparse_vector",
    limit=100
)

# 混合搜索
results = client.hybrid_search(
    search_requests=[dense_req, sparse_req],
    ranker=RRFRanker(),
    limit=10  # 最终返回数量
)
```

### 常见陷阱
- ❌ 混淆候选数量和最终返回数量
- ❌ 候选数量设置过小影响重排序效果
- ❌ 认为混合搜索一定比单一搜索好

## 卡片 6: 批量操作

### 核心要点
批量操作比逐条操作快 10 倍以上。

### 记忆口诀
```
批量插入快十倍，
批量搜索省时间，
推荐批次一千条，
性能提升看得见。
```

### 关键代码
```python
# ❌ 逐条插入（慢）
for doc in docs:
    client.insert([doc])

# ✅ 批量插入（快）
client.insert(docs)  # 推荐批次 1000-5000

# ❌ 逐个搜索（慢）
for query in queries:
    results = client.search(data=[query])

# ✅ 批量搜索（快）
results = client.search(data=queries)
```

### 常见陷阱
- ❌ 逐条插入导致性能极差
- ❌ 批次过大导致内存溢出
- ❌ 批次过小无法充分利用性能

## 卡片 7: 分区策略

### 核心要点
按业务逻辑分区，减少搜索范围，提升查询效率。

### 记忆口诀
```
按类别分区，
按时间分区，
只搜相关区，
性能提十倍。
```

### 关键代码
```python
# 创建分区
client.create_partition("collection", "tech_docs")

# 插入到指定分区
client.insert(
    "collection",
    data,
    partition_name="tech_docs"
)

# 只搜索指定分区
results = client.search(
    "collection",
    data=[query],
    partition_names=["tech_docs"]  # 只搜索这个分区
)
```

### 常见陷阱
- ❌ 分区不会自动平衡数据
- ❌ 分区过多导致管理复杂
- ❌ 分区过少无法提升性能

## 卡片 8: 度量类型

### 核心要点
COSINE 推荐用于文本嵌入，distance 含义取决于 metric_type。

### 记忆口诀
```
COSINE 用于文本，
L2 用于通用，
IP 用于归一化，
大小看类型。
```

### 关键对比
| Metric | 相似度规则 | 范围 | 适用场景 |
|--------|------------|------|----------|
| **COSINE** | 越大越相似 | [-1, 1] | 文本嵌入（推荐） |
| **L2** | 越小越相似 | [0, ∞) | 通用场景 |
| **IP** | 越大越相似 | [-1, 1] | 归一化向量 |

### 常见陷阱
- ❌ 混淆 distance 的含义
- ❌ 使用 COSINE 但向量未归一化
- ❌ 使用错误的度量类型导致结果不准

## 卡片 9: RAG 应用流程

### 核心要点
RAG = 检索 (Milvus) + 生成 (LLM)，Top-K 通常设置为 3-5。

### 记忆口诀
```
插入文档自动嵌入，
查询文本自动检索，
Top-K 设置三到五，
上下文传给 LLM。
```

### 关键代码
```python
# 1. 插入文档
client.insert({"document": "Milvus is..."})

# 2. 检索相关文档
results = client.search(
    data=["What is Milvus?"],
    limit=3  # Top-3
)

# 3. 提取上下文
contexts = [hit['entity']['document'] for hit in results[0]]

# 4. 调用 LLM
prompt = f"基于以下上下文回答：\n{contexts}\n\n问题：{question}"
answer = llm.generate(prompt)
```

### 常见陷阱
- ❌ Top-K 设置过大导致上下文过长
- ❌ Top-K 设置过小导致信息不足
- ❌ 忘记使用 Embedding Functions 简化流程

## 卡片 10: 性能优化清单

### 核心要点
批量操作、分区搜索、合理 limit、简化过滤。

### 记忆口诀
```
批量操作是关键，
分区搜索减范围，
limit 设置要合理，
过滤条件要简单。
```

### 优化清单
```
✅ 批量插入（1000-5000 条）
✅ 批量搜索（多个查询一次）
✅ 使用分区减少搜索范围
✅ 合理设置 limit（通常 5-10）
✅ 简化过滤条件（等值判断最快）
✅ 使用 AUTOINDEX（自动优化）
✅ 避免深度分页（offset 过大）
✅ 使用 Upsert 避免重复
✅ 向量归一化（COSINE/IP）
✅ 选择合适的度量类型
```

### 常见陷阱
- ❌ 逐条操作导致性能差
- ❌ 搜索所有分区浪费时间
- ❌ limit 设置过大浪费资源
- ❌ 复杂过滤条件降低性能
- ❌ 深度分页越来越慢

## 速记表

### 核心操作对照表

| 操作 | 用途 | 关键参数 | 性能提示 |
|------|------|----------|----------|
| **Insert** | 插入新数据 | data, partition_name | 批量插入 1000-5000 |
| **Upsert** | 插入或更新 | data, partition_name | 比 Insert 慢，按需使用 |
| **Search** | 相似度搜索 | data, anns_field, limit | limit=5-10，使用分区 |
| **Query** | 精确查询 | filter, limit | 简化过滤条件 |
| **Get** | 主键获取 | ids | 最快的查询方式 |
| **Hybrid Search** | 混合搜索 | search_requests, ranker | 候选 50-100，返回 10 |

### 度量类型速查表

| Metric | 相似度 | 范围 | 场景 | 归一化 |
|--------|--------|------|------|--------|
| **COSINE** | 越大越相似 | [-1, 1] | 文本嵌入 | 推荐 |
| **L2** | 越小越相似 | [0, ∞) | 通用 | 可选 |
| **IP** | 越大越相似 | [-1, 1] | 归一化向量 | 必须 |

### 批次大小推荐表

| 数据量 | 推荐批次 | 方法 |
|--------|----------|------|
| < 1万 | 100-500 | Insert |
| 1万-10万 | 500-1000 | Insert |
| 10万-100万 | 1000-5000 | Insert |
| > 100万 | 文件导入 | BulkInsert |

## 记忆技巧

### 1. 联想记忆
- **Embedding Functions** = 自动挡汽车（自动处理）
- **Insert vs Upsert** = 往书架放书 vs 更新通讯录
- **Search vs Query** = 找相似的书 vs 通讯录筛选
- **ANN vs kNN** = 字典查词 vs 逐页翻书
- **混合搜索** = 找餐厅综合评分

### 2. 口诀记忆
```
文本进，文本出，向量化交给库，
Insert 快，Upsert 准，批量操作是关键，
Search 找相似，Query 找精确，
ANN 近似快，kNN 精确慢，
Dense 加 Sparse，混合搜索最佳，
分区减范围，limit 要合理。
```

### 3. 对比记忆
- Insert ↔ Upsert（快 vs 准）
- Search ↔ Query（相似 vs 精确）
- ANN ↔ kNN（近似 vs 精确）
- Dense ↔ Sparse（语义 vs 关键词）
- 批量 ↔ 逐条（快 vs 慢）

### 4. 场景记忆

- **数据导入**: 批量 Insert + 分区
- **数据更新**: Upsert
