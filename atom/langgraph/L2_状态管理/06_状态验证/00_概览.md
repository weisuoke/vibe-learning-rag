# 状态验证（State Validation）- 知识点概览

## 知识点简介

**状态验证是 LangGraph 中通过 Pydantic BaseModel 替代 TypedDict 定义状态，在运行时自动执行类型检查、数据转换和业务约束验证的机制。**

在 LangGraph 中，状态（State）是节点之间共享数据的核心载体。当图的复杂度增长、外部输入不可控时，没有验证机制的状态就像没有安检的机场——任何"危险品"都可能混入系统，导致难以调试的运行时错误。状态验证就是为 LangGraph 的数据流加上一道安检门。

## 为什么需要状态验证？

### 没有验证时会发生什么？

```python
from typing_extensions import TypedDict

class State(TypedDict):
    query: str
    temperature: float
    max_results: int

# 这些错误数据会静默通过，直到下游节点崩溃
bad_state = {
    "query": 123,           # 应该是字符串
    "temperature": "hot",   # 应该是浮点数
    "max_results": -5       # 应该是正整数
}
```

TypedDict 只在静态类型检查（如 mypy）时起作用，运行时完全不做任何验证。这意味着错误数据会在图中一路传播，直到某个节点因为类型不匹配而崩溃——而此时你已经很难追溯错误的源头。

### 状态验证解决三个核心问题

| 问题 | 说明 | 示例 |
|------|------|------|
| **输入安全** | 确保外部输入符合预期格式 | 用户传入的 query 必须是非空字符串 |
| **类型正确** | 自动进行类型转换和检查 | `"42"` 自动转为 `42`，`"abc"` 报错 |
| **业务约束** | 验证业务规则 | temperature 必须在 0.0-2.0 之间 |

### 一个直观的对比

```python
# ❌ TypedDict：运行时不验证
class UnsafeState(TypedDict):
    temperature: float

# ✅ Pydantic BaseModel：运行时自动验证
from pydantic import BaseModel, Field

class SafeState(BaseModel):
    temperature: float = Field(ge=0.0, le=2.0)
```

当你用 `SafeState(temperature=3.0)` 时，Pydantic 会立即抛出 `ValidationError`，而不是让这个非法值在图中传播。

## 核心概念速览

本知识点包含 6 个核心概念：

| 编号 | 概念 | 一句话说明 | 重要程度 |
|------|------|-----------|----------|
| 1 | Pydantic BaseModel 状态定义 | 用 Pydantic 替代 TypedDict 定义状态，启用运行时验证 | ★★★★★ |
| 2 | field_validator 字段验证器 | 对单个字段的自定义验证逻辑（before/after 模式） | ★★★★☆ |
| 3 | model_validator 模型验证器 | 跨字段的整体验证逻辑（密码匹配、互斥字段等） | ★★★★☆ |
| 4 | 运行时类型强制转换 | Pydantic 自动将 `"42"`→`42`、`"true"`→`True` 等类型转换 | ★★★☆☆ |
| 5 | 验证错误处理 | ValidationError 捕获、结构化错误信息、优雅降级 | ★★★★☆ |
| 6 | TypedDict vs Pydantic 选型策略 | 内部用 TypedDict（轻量）、边界用 Pydantic（严格） | ★★★★★ |

### 场景1 预览：基础 Pydantic 状态验证

```python
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph, START, END

class SearchState(BaseModel):
    query: str = Field(min_length=1, description="搜索查询")
    temperature: float = Field(default=0.7, ge=0.0, le=2.0)
    max_results: int = Field(default=5, ge=1, le=100)
    results: list[str] = Field(default_factory=list)

def search_node(state: SearchState) -> dict:
    return {"results": [f"Result for: {state.query}"]}

builder = StateGraph(SearchState)
builder.add_node("search", search_node)
builder.add_edge(START, "search")
builder.add_edge("search", END)
graph = builder.compile()

# ✅ 合法输入
result = graph.invoke({"query": "LangGraph tutorial", "temperature": 0.5})

# ❌ 非法输入 → ValidationError
# graph.invoke({"query": "", "temperature": 3.0})
```

### 验证触发流程

```
graph.invoke(input_data)
    ↓
StateGraph._coerce_state(input_data)
    ↓
判断 schema 是否为 Pydantic BaseModel
    ↓
是 → Pydantic 验证（类型检查 + 约束验证）
    ↓                    ↓
验证通过 → 继续执行    验证失败 → ValidationError
    ↓
否 → TypedDict（无运行时验证，直接通过）
```

## 学习建议

### 1. 动手实践

- 先跑通场景 1 的基础示例，感受 Pydantic 验证的效果
- 故意传入错误数据，观察 ValidationError 的报错信息
- 对比同一个状态用 TypedDict 和 Pydantic 定义的行为差异

### 2. 循序渐进

- 先掌握 `Field()` 的基础约束（这能覆盖 80% 的验证需求）
- 再学 `field_validator`（处理复杂的单字段逻辑）
- 最后学 `model_validator`（处理跨字段关系）

### 3. 关注选型

- 不要盲目全用 Pydantic——内部状态用 TypedDict 更轻量
- 不要盲目全用 TypedDict——边界输入必须验证
- 混合架构是生产环境的最佳实践

### 4. 常见陷阱

- **陷阱 1**：Pydantic v1 和 v2 的 API 差异很大，确保使用 v2 语法
- **陷阱 2**：`field_validator` 的 `mode="before"` 和 `mode="after"` 行为不同
- **陷阱 3**：Pydantic 状态与 Annotated Reducer 的组合需要特殊处理
- **陷阱 4**：验证错误会中断整个图的执行，需要提前规划错误处理
