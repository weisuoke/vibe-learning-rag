# Retriever高级策略 - 30字核心

> **一句话精髓**：掌握 LangChain 中五大高级 Retriever 策略的原理与组合应用

---

## 核心陈述

**Retriever高级策略通过混合检索、重排序和多路召回，将基础检索器组合成生产级高精度检索管道。**

---

## 逐词拆解

### 1.「Retriever高级策略」— 不是一个类，而是一组策略

基础 Retriever 只有一个 `VectorStoreRetriever`，通过 `as_retriever()` 创建。高级策略是 LangChain 提供的 5 个专用 Retriever 子类，每个解决一个特定的检索痛点：

```python
# 基础 Retriever — 单路向量检索
from langchain_core.vectorstores import VectorStoreRetriever

# 高级策略 — 5 个专用 Retriever
from langchain.retrievers import EnsembleRetriever          # 混合检索
from langchain.retrievers import ContextualCompressionRetriever  # 重排序
from langchain.retrievers import MultiQueryRetriever         # 查询扩展
from langchain.retrievers import ParentDocumentRetriever     # 父文档检索
from langchain.retrievers import SelfQueryRetriever          # 自查询
```

它们都继承自 `BaseRetriever`，都实现 `Runnable` 接口，都能无缝接入 LCEL 管道。

### 2.「混合检索」— 多路召回，互补短板

单一检索方式有盲区：向量检索擅长语义匹配但对关键词不敏感，BM25 擅长精确匹配但不理解语义。混合检索通过 RRF（Reciprocal Rank Fusion）融合多路结果：

```python
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever

# BM25 关键词检索 + 向量语义检索
bm25_retriever = BM25Retriever.from_documents(docs, k=10)
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 10})

# RRF 融合：score(doc) = Σ weight_i / (rank_i + 60)
ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6]  # 语义检索权重更高
)
```

### 3.「重排序」— 粗筛之后精排

第一轮检索（召回）追求"不漏"，返回较多候选文档。重排序用更精确的模型（Cross-Encoder）对候选文档重新打分，只保留最相关的 Top-N：

```python
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import CrossEncoderReranker
from langchain_community.cross_encoders import HuggingFaceCrossEncoder

# Cross-Encoder 逐对打分，精度远高于 Bi-Encoder
reranker = CrossEncoderReranker(
    model=HuggingFaceCrossEncoder(model_name="BAAI/bge-reranker-base"),
    top_n=5
)

# 装饰器模式：包装任意基础检索器
compression_retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=ensemble  # 可以包装上面的混合检索器
)
```

### 4.「多路召回」— 不只是多个检索器

"多路召回"在这里有三层含义：

| 层次 | 策略 | 实现方式 |
|------|------|---------|
| 多检索器 | EnsembleRetriever | BM25 + 向量 + 其他检索器并行 |
| 多查询 | MultiQueryRetriever | LLM 生成多角度查询，同一检索器多次检索 |
| 多粒度 | ParentDocumentRetriever | 小块匹配，返回大块上下文 |

```python
from langchain.retrievers import MultiQueryRetriever

# LLM 自动生成 3 个查询变体
multi_query = MultiQueryRetriever.from_llm(
    retriever=vector_retriever,
    llm=llm
)
# 用户问 "LangChain 怎么做 RAG"
# LLM 生成：
#   1. "LangChain RAG 架构是什么"
#   2. "如何用 LangChain 实现检索增强生成"
#   3. "LangChain 检索器和生成器如何配合"
# 三个查询分别检索，结果去重合并
```

### 5.「基础检索器组合成」— 组合模式是核心设计

所有高级策略都不是从零开始，而是包装和组合基础检索器。这是装饰器模式和策略模式的典型应用：

```python
# 层层包装，逐步增强
# 第 1 层：基础向量检索
base = vectorstore.as_retriever(search_kwargs={"k": 20})

# 第 2 层：混合检索（加入 BM25）
ensemble = EnsembleRetriever(
    retrievers=[bm25, base],
    weights=[0.4, 0.6]
)

# 第 3 层：重排序（精度提升）
final = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=ensemble
)

# 最终使用 — 接口完全一致
docs = final.invoke("什么是 RAG？")
```

### 6.「生产级高精度检索管道」— 目标是生产可用

"生产级"意味着：
- **高精度**：Ensemble + Rerank 组合可将检索准确率提升 15-25%
- **高召回**：MultiQuery 扩展查询角度，减少漏检
- **低延迟**：异步并行执行，合理控制候选数量
- **可观测**：所有策略都支持 LangSmith 追踪和 Callback

```python
# 生产级管道 = LCEL 组合
from langchain_core.runnables import RunnablePassthrough

rag_chain = (
    {"context": final | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# 支持异步、批量、流式
answer = await rag_chain.ainvoke("什么是向量数据库？")
```

## 核心价值分解

### 价值 1：召回率提升 — 不再漏检

**问题**：单路向量检索对精确关键词匹配弱，用户搜"Python 3.12 新特性"可能漏掉包含"Python 3.12 new features"的文档。

**方案**：EnsembleRetriever 融合 BM25（关键词精确匹配）+ 向量检索（语义理解），互补盲区。

```python
# BM25 能精确匹配 "Python 3.12"
# 向量检索能理解 "新特性" ≈ "new features"
# RRF 融合两者的排名，综合得分最高的文档排在前面
ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6]
)
```

### 价值 2：精度提升 — 去除噪声

**问题**：召回 20 篇文档中可能只有 5 篇真正相关，噪声文档会干扰 LLM 生成。

**方案**：ContextualCompression 用 Cross-Encoder 重新打分，只保留最相关的 Top-5。

```python
# Cross-Encoder 对 (query, doc) 对逐一打分
# 精度远高于 Bi-Encoder 的余弦相似度
reranker = CrossEncoderReranker(model=cross_encoder, top_n=5)
compressed = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=ensemble  # 从 20 篇中精选 5 篇
)
```

### 价值 3：查询理解增强 — 理解用户真实意图

**问题**：用户查询往往角度单一或表述模糊，一次检索可能错过相关文档。

**方案**：MultiQueryRetriever 让 LLM 从多个角度重述查询，SelfQueryRetriever 让 LLM 提取结构化过滤条件。

```python
# MultiQuery：一个问题变三个
# "RAG 性能优化" →
#   "如何提升 RAG 系统的检索精度"
#   "RAG 延迟优化方法有哪些"
#   "RAG 生产环境性能调优最佳实践"

# SelfQuery：自然语言 → 结构化查询
# "2024年发布的关于 Python 的文档" →
#   query: "Python 文档"
#   filter: year == 2024
```

### 价值 4：上下文完整性 — 小搜大返回

**问题**：小块文本匹配精度高，但返回给 LLM 的上下文太短，缺少必要背景信息。

**方案**：ParentDocumentRetriever 用小块做搜索索引，命中后返回对应的父文档（大块）。

```python
from langchain.retrievers import ParentDocumentRetriever
from langchain_text_splitters import RecursiveCharacterTextSplitter

parent_retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,       # 存小块的嵌入
    docstore=InMemoryStore(),      # 存父文档原文
    child_splitter=RecursiveCharacterTextSplitter(chunk_size=200),
    parent_splitter=RecursiveCharacterTextSplitter(chunk_size=1000),
)
# 搜索时匹配 200 字的小块
# 返回时给出 1000 字的父块 — 上下文更完整
```

### 价值 5：统一接口 — 组合无摩擦

**所有高级策略都是 BaseRetriever**，接口完全一致：

```python
# 无论用哪种策略，调用方式完全相同
docs = ensemble.invoke("query")           # 混合检索
docs = compressed.invoke("query")         # 重排序
docs = multi_query.invoke("query")        # 查询扩展
docs = parent_retriever.invoke("query")   # 父文档
docs = self_query.invoke("query")         # 自查询

# 都能无缝接入 LCEL 管道
chain = {"context": any_retriever} | prompt | llm
```

## 问题-方案对照表

| 检索痛点 | 具体表现 | 对应策略 | 核心原理 |
|----------|---------|---------|---------|
| 召回不全 | 关键词匹配和语义匹配各有盲区 | EnsembleRetriever | BM25+向量 RRF 融合 |
| 结果不准 | Top-K 中混入大量无关文档 | ContextualCompression | Cross-Encoder 精排 |
| 查询太窄 | 用户表述单一，错过相关文档 | MultiQueryRetriever | LLM 生成多角度查询 |
| 上下文不足 | 小块匹配准但信息不完整 | ParentDocumentRetriever | 小搜大返回 |
| 无法精确过滤 | 自然语言无法表达结构化条件 | SelfQueryRetriever | LLM 解析为结构化查询 |
| 延迟太高 | 多策略串行执行太慢 | asyncio.gather 并行 | 异步并行 + 合理 Top-K |
| 难以调试 | 不知道哪个环节出了问题 | LangSmith + Callbacks | 全链路追踪 |

## 为什么这 30 个字重要？

### 1. 它定义了从"能用"到"好用"的跨越

基础 `as_retriever()` 让你的 RAG 系统"能检索"。但生产环境中，单路向量检索的准确率通常只有 60-70%。加上混合检索 + 重排序，准确率可以提升到 85-90%。这 30 个字描述的就是这个跨越。

### 2. 它揭示了检索优化的三个维度

```
召回率（Recall）  ← 混合检索、多路召回
精确率（Precision）← 重排序、压缩过滤
查询理解          ← 查询扩展、自查询
```

任何检索优化都逃不出这三个维度。理解了这一点，面对新的检索需求你就知道该从哪个维度入手。

### 3. 它强调了"组合"而非"替换"

高级策略不是替换基础检索器，而是在基础检索器之上层层包装。这意味着：
- 你可以渐进式地优化检索管道
- 每一层都可以独立测试和替换
- 组合方式灵活，适应不同场景

### 4. 它指向了生产级的标准

"生产级"三个字意味着不只是 demo 能跑，而是要考虑：延迟、吞吐、可观测性、降级策略。这 30 个字提醒你，学习高级策略的目标是构建真正可上线的检索管道。

## 一句话记忆法

**"混排多父自，五策略组合管道"**

- **混**：混合检索（EnsembleRetriever）
- **排**：重排序（ContextualCompression）
- **多**：多查询（MultiQueryRetriever）
- **父**：父文档（ParentDocumentRetriever）
- **自**：自查询（SelfQueryRetriever）
- **组合管道**：层层包装，LCEL 串联
