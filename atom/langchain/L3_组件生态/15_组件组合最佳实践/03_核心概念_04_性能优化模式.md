# 核心概念4：性能优化模式

> 性能优化模式是通过流式传播、批处理并行、缓存策略和并发执行，让 LangChain 管道在保证正确性的同时达到最优性能的技术集合。本文基于 `langchain_core.runnables.base.py`（RunnableSequence 的 batch/stream 实现）和 `langchain.cache`（caches.py）源码分析。

---

## 一句话定义

**性能优化模式是通过流式传播、批处理并行、缓存策略和并发执行，让 LangChain 管道在保证正确性的同时达到最优性能的技术集合。**

把它想成高速公路的交通管理——流式是让车辆不等红灯直接上匝道，批处理是多车道并行，缓存是记住常走的路线不用重新导航，并行执行是同时派多辆车去不同目的地取货。

---

## 优化维度1：流式传播（Streaming）

### 原理

流式传播让用户在 LLM 生成第一个 token 时就开始看到输出，而不是等待完整响应。

```
传统模式：[等待全部生成.....................] → 一次性显示
流式模式：[你] [好] [，] [春] [天] [很] [美] → 逐字显示
```

### 源码洞察

从 `base.py` 中 `RunnableSequence` 的流式实现：

- 如果所有组件实现 `transform` 方法，整个序列可以端到端流式传输
- RunnableLambda 默认不支持 transform，会阻断流式传播
- 流式传播是逐步骤的：每个步骤处理完一个 chunk 就传给下一步

```python
# RunnableSequence.transform() 的核心逻辑（简化）
def transform(self, input, config, **kwargs):
    final_pipeline = self.first.transform(input, ...)
    for step in self.middle:
        final_pipeline = step.transform(final_pipeline, ...)
    yield from self.last.transform(final_pipeline, ...)
```

### 代码示例

```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template("写一首关于{topic}的诗")
model = ChatOpenAI(model="gpt-4o", streaming=True)
parser = StrOutputParser()

chain = prompt | model | parser

# 同步流式
for chunk in chain.stream({"topic": "春天"}):
    print(chunk, end="", flush=True)

# 异步流式
async for chunk in chain.astream({"topic": "春天"}):
    print(chunk, end="", flush=True)

# 流式事件（获取中间步骤信息）
async for event in chain.astream_events({"topic": "春天"}, version="v2"):
    if event["event"] == "on_chat_model_stream":
        print(event["data"]["chunk"].content, end="")
```

### 流式传播的阻断与解决

```python
from langchain_core.runnables import RunnableLambda

# ❌ RunnableLambda 阻断流式
chain = prompt | model | RunnableLambda(lambda x: x.content.upper()) | parser
# model 的流式输出会被 lambda 阻断

# ✅ 使用 RunnableGenerator 保持流式
from langchain_core.runnables import RunnableGenerator

def upper_stream(input_stream):
    for chunk in input_stream:
        yield chunk.upper()

chain = prompt | model | RunnableGenerator(upper_stream)
```

## 优化维度2：批处理（Batching）

### 原理

batch 方法对多个输入并行处理，利用线程池优化 IO 密集型操作。

### 源码洞察

从 `base.py` 中 `RunnableSequence.batch` 的实现：

- RunnableSequence.batch 对每个步骤调用 batch，而不是对每个输入调用 invoke
- 默认使用 ContextThreadPoolExecutor 并行执行
- 可通过 config 的 max_concurrency 控制并发数

```python
# RunnableSequence.batch() 的核心逻辑（简化）
def batch(self, inputs, config=None, *, return_exceptions=False, **kwargs):
    # 关键：对每个步骤调用 batch，而不是对每个输入调用 invoke
    results = self.first.batch(inputs, ...)
    for step in self.middle:
        results = step.batch(results, ...)
    results = self.last.batch(results, ...)
    return results
```

### 代码示例

```python
# 基础批处理
questions = [
    {"topic": "春天"},
    {"topic": "夏天"},
    {"topic": "秋天"},
    {"topic": "冬天"},
]
results = chain.batch(questions)

# 控制并发数
results = chain.batch(
    questions,
    config={"max_concurrency": 2},  # 最多同时处理 2 个
)

# 异步批处理
results = await chain.abatch(questions)

# 批处理 + 异常处理
results = chain.batch(
    questions,
    return_exceptions=True,  # 失败的返回异常对象而不是抛出
)
for result in results:
    if isinstance(result, Exception):
        print(f"失败: {result}")
    else:
        print(f"成功: {result}")
```

### batch vs 循环 invoke 的性能对比

```python
import time

# ❌ 慢：逐个 invoke
start = time.time()
results_slow = [chain.invoke(q) for q in questions]
print(f"逐个 invoke: {time.time() - start:.2f}s")  # ~8s

# ✅ 快：batch 并行
start = time.time()
results_fast = chain.batch(questions)
print(f"batch 并行: {time.time() - start:.2f}s")    # ~2s
```

## 优化维度3：缓存策略（Caching）

### 原理

缓存避免重复的 LLM 调用，对相同输入直接返回缓存结果。

### 源码洞察

从 `caches.py` 的缓存架构：

```python
class BaseCache(ABC):
    @abstractmethod
    def lookup(self, prompt: str, llm_string: str) -> Optional[RETURN_VAL_TYPE]:
        """查找缓存：prompt + llm配置 作为 key"""

    @abstractmethod
    def update(self, prompt: str, llm_string: str, return_val: RETURN_VAL_TYPE):
        """更新缓存"""
```

关键设计：缓存 key = prompt + llm_string，同一个 prompt 用不同模型不会混淆；通过 `set_llm_cache()` 全局生效，对业务代码完全透明。

### 代码示例

```python
from langchain.cache import InMemoryCache
from langchain.globals import set_llm_cache

# 内存缓存
set_llm_cache(InMemoryCache())

# SQLite 缓存（持久化）
from langchain_community.cache import SQLiteCache
set_llm_cache(SQLiteCache(database_path=".langchain.db"))

# 语义缓存（相似问题也能命中）
from langchain_community.cache import RedisSemanticCache
set_llm_cache(RedisSemanticCache(
    redis_url="redis://localhost:6379",
    embedding=OpenAIEmbeddings(),
))
```

### 缓存策略选择

| 缓存类型 | 适用场景 | 优点 | 缺点 |
|----------|----------|------|------|
| InMemoryCache | 开发/测试 | 快速，无依赖 | 重启丢失 |
| SQLiteCache | 单机生产 | 持久化，简单 | 不支持分布式 |
| RedisCache | 分布式生产 | 高性能，分布式 | 需要 Redis |
| SemanticCache | 语义相似查询 | 相似问题命中 | 需要 Embedding |

### 缓存注意事项

- 缓存命中时流式输出会一次性返回，可能导致用户体验不一致
- InMemoryCache 没有 TTL，需要手动清理
- 不适合缓存的场景：实时信息查询、需要随机性的生成、输入高度多样化

## 优化维度4：并行执行（Parallel Execution）

### 原理

RunnableParallel 并发执行多个独立的 Runnable，将结果合并为字典。

### 代码示例

```python
from langchain_core.runnables import RunnableParallel

# 并行检索 + 并行生成
chain = RunnableParallel(
    web_results=web_retriever | format_docs,
    db_results=db_retriever | format_docs,
    history=memory_retriever | format_history,
) | merge_context | prompt | model | parser
```

### 并行 vs 串行的时间对比

```
串行执行（总时间 = 各步骤之和）：
  网络搜索 [====2s====]
                      数据库搜索 [===1.5s===]
                                            历史搜索 [==1s==]
  总时间: 4.5s

并行执行（总时间 = 最慢步骤）：
  网络搜索   [====2s====]
  数据库搜索 [===1.5s===]
  历史搜索   [==1s==]
  总时间: 2s
```

## 优化维度6：重试与降级

### 代码示例

```python
from langchain_openai import ChatOpenAI

# 重试：遇到异常自动重试
model_with_retry = ChatOpenAI(model="gpt-4o").with_retry(
    stop_after_attempt=3,
    wait_exponential_jitter=True,
)

# 降级：主模型失败时切换到备用模型
model_with_fallback = ChatOpenAI(model="gpt-4o").with_fallbacks(
    [ChatOpenAI(model="gpt-4o-mini")],
)

# 组合：重试 + 降级
robust_model = (
    ChatOpenAI(model="gpt-4o")
    .with_retry(stop_after_attempt=2)
    .with_fallbacks([
        ChatOpenAI(model="gpt-4o-mini").with_retry(stop_after_attempt=2)
    ])
)
# 执行顺序：gpt-4o 尝试2次 → gpt-4o-mini 尝试2次
```

## 性能优化决策树

```
需要优化什么？
├── 用户体验（首字延迟）→ 流式传播
├── 吞吐量（批量处理）→ batch + 并行
├── 成本（API 费用）→ 缓存 + 模型分级
├── 延迟（端到端时间）→ 并行 + 缓存
└── 稳定性（错误率）→ 重试 + 降级
```

## 在 RAG 开发中的应用

### 完整的 RAG 性能优化示例

```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableParallel, RunnablePassthrough, RunnableLambda
from langchain.cache import InMemoryCache
from langchain.globals import set_llm_cache

# 1. 启用缓存
set_llm_cache(InMemoryCache())

# 2. 模型配置（重试 + 降级）
model = (
    ChatOpenAI(model="gpt-4o", streaming=True)
    .with_retry(stop_after_attempt=2)
    .with_fallbacks([ChatOpenAI(model="gpt-4o-mini", streaming=True)])
)

# 3. 并行多路检索
def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

parallel_retrieval = RunnableParallel(
    semantic_results=semantic_retriever | format_docs,
    keyword_results=keyword_retriever | format_docs,
)

# 4. 合并上下文
def merge_results(inputs: dict) -> str:
    return f"语义检索：\n{inputs['semantic_results']}\n\n关键词检索：\n{inputs['keyword_results']}"

# 5. 完整链（支持流式输出）
prompt = ChatPromptTemplate.from_template(
    "根据以下上下文回答问题。\n\n{context}\n\n问题：{question}"
)

rag_chain = (
    RunnableParallel(
        context=RunnableLambda(lambda x: x["question"])
                | parallel_retrieval
                | RunnableLambda(merge_results),
        question=RunnablePassthrough() | RunnableLambda(lambda x: x["question"]),
    )
    | prompt
    | model
    | StrOutputParser()
)

# 6. 流式输出给用户
for chunk in rag_chain.stream({"question": "什么是向量检索？"}):
    print(chunk, end="", flush=True)

# 7. 批量处理
questions = [{"question": q} for q in ["问题1", "问题2", "问题3"]]
results = rag_chain.batch(questions, config={"max_concurrency": 3})
```

### RAG 各阶段优化策略总结

| RAG 阶段 | 优化策略 | 具体手段 |
|----------|----------|----------|
| 文档索引 | 批处理 | `embeddings.embed_documents()` 批量生成向量 |
| 检索阶段 | 并行 | `RunnableParallel` 多路召回 |
| 生成阶段 | 流式 | `chain.stream()` 逐 token 输出 |
| 重复查询 | 缓存 | `SemanticCache` 语义缓存 |
| 成本控制 | 模型分级 | 简单问题用 `gpt-4o-mini` |
| 稳定性 | 重试降级 | `with_retry` + `with_fallbacks` |

## 性能优化的常见误区

| 误区 | 真相 |
|------|------|
| "所有地方都用流式" | 流式增加复杂度，只在面向用户的输出端使用 |
| "并发数越高越好" | 过高的并发会触发 API 限流，反而更慢 |
| "缓存万能" | 输入多样化时缓存命中率低，白白浪费内存 |
| "batch 一定比循环快" | CPU 密集型任务中，batch 的线程开销可能抵消并行收益 |
| "优化要一步到位" | 先测量瓶颈，再针对性优化，避免过早优化 |

---

## 小结

性能优化核心原则：先测量再优化，选择合适的工具（流式、批处理、缓存、并行各有适用场景），从最大瓶颈开始，保持简单。
