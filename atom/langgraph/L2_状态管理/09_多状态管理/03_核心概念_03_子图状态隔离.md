# 核心概念 03：子图状态隔离

> 每个子图拥有独立的状态 Schema 和 Channel 集合，父子图之间通过"共享键"或"函数包装"通信——子图的内部状态对父图完全不可见。

## 一句话定义

**子图状态隔离是指每个子图拥有完全独立的 StateGraph、Channel 集合和命名空间，父图无法直接访问子图的内部状态，两者只通过明确定义的接口（共享键或函数包装）交换数据。**

---

## 为什么需要这个？

### 问题场景

假设你在构建一个复杂的 AI Agent 系统：

```
主图（协调器）
├── RAG 子图（检索增强生成）
│   ├── 内部状态：chunks, embeddings, rerank_scores, ...
├── 工具调用子图（执行外部工具）
│   ├── 内部状态：tool_name, tool_args, retry_count, ...
└── 对话管理子图（管理多轮对话）
    ├── 内部状态：session_id, turn_count, summary, ...
```

如果所有子图共享同一个状态：
1. **命名冲突**：RAG 子图的 `scores` 和工具调用子图的 `scores` 会互相覆盖
2. **状态污染**：一个子图的 bug 可能破坏另一个子图的数据
3. **耦合过紧**：修改一个子图的状态定义，可能影响所有其他子图

### 解决方案

每个子图有自己独立的状态空间，父图把子图当作"黑盒"：

```python
# RAG 子图：有自己的状态
class RAGState(TypedDict):
    query: str          # 与父图共享
    chunks: list[str]   # 子图私有
    scores: list[float] # 子图私有
    answer: str         # 与父图共享

# 工具子图：有自己的状态
class ToolState(TypedDict):
    query: str          # 与父图共享
    tool_name: str      # 子图私有
    tool_args: dict     # 子图私有
    result: str         # 与父图共享

# 父图：只看到共享的字段
class ParentState(TypedDict):
    query: str
    answer: str
    result: str
```

## 双重类比：先建立直觉

### 前端类比：微前端架构

```
微前端架构                              LangGraph 子图
─────────────                           ─────────────
主应用（Shell）                          父图
├── 订单微前端                           ├── RAG 子图
│   ├── 自己的 Redux Store              │   ├── 自己的 Channel 集合
│   ├── 自己的路由                       │   ├── 自己的状态 Schema
│   └── 通过 CustomEvent 通信           │   └── 通过共享键通信
├── 用户微前端                           ├── 工具子图
│   ├── 自己的 Redux Store              │   ├── 自己的 Channel 集合
│   └── 完全独立的状态                   │   └── 完全独立的状态
```

- 每个微前端有自己的状态管理（Redux Store），互不干扰
- 微前端之间通过约定的接口（CustomEvent、URL 参数）通信
- 主应用不需要知道微前端内部的状态结构

### 日常生活类比：公司的部门隔离

- **总经理**（父图）只看到每个部门的"汇报接口"：营收数据、项目进度
- **研发部**（子图A）内部有自己的 Jira、代码仓库、技术债务——总经理不需要知道
- **市场部**（子图B）内部有自己的投放数据、渠道分析——研发部看不到
- 部门之间通过"共享文档"（共享键）或"会议汇报"（函数包装）交换信息

## 详细解释

### 1. 子图的完全隔离

每个子图在编译时会创建自己独立的：
- **Channel 集合**：子图的字段和父图的字段在不同的 Channel 中
- **命名空间**：通过 `NS_SEP` 和 `NS_END` 分隔，确保 Checkpoint 不冲突
- **执行上下文**：子图有自己的执行循环

```python
# 来源: libs/langgraph/langgraph/_internal/_constants.py
NS_SEP = "|"   # 分隔层级：graph|subgraph|subsubgraph
NS_END = ":"   # 分隔命名空间和 task_id
```

命名空间示例：
- `""` — 根图（父图）
- `"rag_subgraph:abc123"` — 一级子图
- `"rag_subgraph:abc123|retriever:def456"` — 二级子图（子图的子图）

### 2. 两种子图使用方式

#### 方式一：直接添加编译后的子图（共享键自动通信）

```python
subgraph = subgraph_builder.compile()
parent_builder.add_node("sub", subgraph)
```

LangGraph 会自动找到父图和子图 Schema 中**同名的字段**（共享键），在调用子图时：
1. 将父图中共享键的值传给子图
2. 子图执行完毕后，将共享键的值写回父图
3. 非共享键对父图不可见

#### 方式二：函数包装（手动状态转换）

```python
def call_subgraph(state: ParentState):
    # 手动将父图状态转换为子图输入
    response = subgraph.invoke({"bar": state["foo"]})
    # 手动将子图输出转换为父图更新
    return {"foo": response["bar"]}

parent_builder.add_node("sub", call_subgraph)
```

这种方式更灵活——父图和子图的字段名可以完全不同。

### 3. 共享键的自动发现机制

当使用方式一（直接添加子图）时，LangGraph 在编译阶段会：

```python
# 来源: libs/langgraph/langgraph/graph/state.py（概念简化版）
def _find_shared_keys(parent_schema, subgraph):
    """找到父图和子图之间的共享键"""
    parent_keys = set(get_type_hints(parent_schema).keys())
    sub_input_keys = set(get_type_hints(subgraph.input_schema).keys())
    sub_output_keys = set(get_type_hints(subgraph.output_schema).keys())

    # 共享输入键：父图有 且 子图输入也有
    shared_input = parent_keys & sub_input_keys
    # 共享输出键：父图有 且 子图输出也有
    shared_output = parent_keys & sub_output_keys

    return shared_input, shared_output
```

数据流：

```
父图状态                    子图执行                    父图状态更新
────────                    ────────                    ──────────
{foo: "hello",    ──→    子图只收到共享键:    ──→    {foo: 子图返回的 foo,
 bar: "world"}           {foo: "hello"}              bar: "world"}  ← 不变
                         子图内部:
                         {foo: "hello",
                          baz: "private"}  ← 父图看不到
```

### 4. 父子图通信的四种路径

```
路径                          方向              说明
──────────────────────────    ──────            ──────
1. 共享键自动传递              父 → 子 → 父      同名字段自动通信
2. 函数包装手动转换            父 → 子 → 父      不同字段名，手动映射
3. Command(graph=PARENT)      子 → 父           子图显式更新父图状态
4. Config keys 共享资源        双向              checkpointer, store 等
```

### 5. 命名空间与 Checkpoint 隔离

每个子图在 Checkpoint 中有自己的命名空间，确保状态持久化不冲突：

```python
# 来源: libs/langgraph/langgraph/pregel/main.py（概念简化版）
# 子图执行时，命名空间会被拼接
child_namespace = f"{parent_namespace}{NS_SEP}{node_name}{NS_END}{task_id}"

# 示例：
# 父图命名空间: ""
# 子图命名空间: "rag_node:task_001"
# 孙图命名空间: "rag_node:task_001|retriever:task_002"
```

这意味着：
- 父图和子图的 Checkpoint 存储在不同的命名空间下
- 即使父图和子图有同名字段，Checkpoint 也不会冲突
- 可以独立回溯父图或子图的状态历史

## 代码示例

### 示例 1：直接添加子图（共享键自动通信）

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# ========== 子图定义 ==========
class SubgraphState(TypedDict):
    foo: str   # 与父图共享的键
    bar: str   # 子图私有的键

def sub_node_1(state: SubgraphState):
    """子图节点1：设置私有字段"""
    return {"bar": "子图内部数据"}

def sub_node_2(state: SubgraphState):
    """子图节点2：用私有字段更新共享字段"""
    return {"foo": state["foo"] + " + " + state["bar"]}

# 构建子图
sub_builder = StateGraph(SubgraphState)
sub_builder.add_node("sub_1", sub_node_1)
sub_builder.add_node("sub_2", sub_node_2)
sub_builder.add_edge(START, "sub_1")
sub_builder.add_edge("sub_1", "sub_2")
sub_builder.add_edge("sub_2", END)
subgraph = sub_builder.compile()

# ========== 父图定义 ==========
class ParentState(TypedDict):
    foo: str   # 与子图共享的键

def parent_node(state: ParentState):
    """父图节点：设置初始值"""
    return {"foo": "来自父图"}

# 构建父图
parent_builder = StateGraph(ParentState)
parent_builder.add_node("parent_node", parent_node)
parent_builder.add_node("sub", subgraph)  # 直接添加编译后的子图！
parent_builder.add_edge(START, "parent_node")
parent_builder.add_edge("parent_node", "sub")
parent_builder.add_edge("sub", END)

graph = parent_builder.compile()

# ========== 执行 ==========
result = graph.invoke({"foo": ""})
print(result)
# {'foo': '来自父图 + 子图内部数据'}
```

**数据流解析**：

```
invoke({"foo": ""})
  ↓
parent_node → {"foo": "来自父图"}
  ↓
父图状态: {foo: "来自父图"}
  ↓ 共享键 foo 传入子图
  ↓
子图收到: {foo: "来自父图", bar: ???}  ← bar 未初始化
  ↓
sub_node_1 → {"bar": "子图内部数据"}
  ↓
子图状态: {foo: "来自父图", bar: "子图内部数据"}
  ↓
sub_node_2 → {"foo": "来自父图 + 子图内部数据"}
  ↓
子图状态: {foo: "来自父图 + 子图内部数据", bar: "子图内部数据"}
  ↓ 共享键 foo 写回父图（bar 不写回，因为父图没有 bar）
  ↓
父图状态: {foo: "来自父图 + 子图内部数据"}
```

### 示例 2：函数包装（不同 Schema 的子图）

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# ========== 子图定义（完全不同的 Schema）==========
class SubgraphState(TypedDict):
    bar: str    # 子图用 bar
    baz: str    # 子图私有

def subgraph_node_1(state: SubgraphState):
    return {"baz": "中间结果"}

def subgraph_node_2(state: SubgraphState):
    return {"bar": state["bar"] + " → " + state["baz"] + " → 完成"}

sub_builder = StateGraph(SubgraphState)
sub_builder.add_node("s1", subgraph_node_1)
sub_builder.add_node("s2", subgraph_node_2)
sub_builder.add_edge(START, "s1")
sub_builder.add_edge("s1", "s2")
sub_builder.add_edge("s2", END)
subgraph = sub_builder.compile()

# ========== 父图定义 ==========
class ParentState(TypedDict):
    foo: str    # 父图用 foo（和子图的 bar 不同名！）

def pre_process(state: ParentState):
    return {"foo": "处理后的 " + state["foo"]}

def call_subgraph(state: ParentState):
    """函数包装：手动做状态转换"""
    # 父图的 foo → 子图的 bar
    response = subgraph.invoke({"bar": state["foo"]})
    # 子图的 bar → 父图的 foo
    return {"foo": response["bar"]}

parent_builder = StateGraph(ParentState)
parent_builder.add_node("pre", pre_process)
parent_builder.add_node("sub", call_subgraph)  # 用函数包装，不是直接传子图
parent_builder.add_edge(START, "pre")
parent_builder.add_edge("pre", "sub")
parent_builder.add_edge("sub", END)

graph = parent_builder.compile()

result = graph.invoke({"foo": "原始输入"})
print(result)
# {'foo': '处理后的 原始输入 → 中间结果 → 完成'}
```

### 示例 3：多层嵌套子图

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# ========== 第三层：孙图 ==========
class GrandchildState(TypedDict):
    value: str

def grandchild_node(state: GrandchildState):
    return {"value": state["value"] + " → 孙图处理"}

gc_builder = StateGraph(GrandchildState)
gc_builder.add_node("gc", grandchild_node)
gc_builder.add_edge(START, "gc")
gc_builder.add_edge("gc", END)
grandchild = gc_builder.compile()

# ========== 第二层：子图 ==========
class ChildState(TypedDict):
    value: str       # 与父图和孙图共享
    child_data: str  # 子图私有

def child_node(state: ChildState):
    return {"value": state["value"] + " → 子图处理", "child_data": "子图私有数据"}

child_builder = StateGraph(ChildState)
child_builder.add_node("child", child_node)
child_builder.add_node("grandchild", grandchild)  # 嵌套孙图
child_builder.add_edge(START, "child")
child_builder.add_edge("child", "grandchild")
child_builder.add_edge("grandchild", END)
child_graph = child_builder.compile()

# ========== 第一层：父图 ==========
class ParentState(TypedDict):
    value: str

def parent_node(state: ParentState):
    return {"value": "起点"}

parent_builder = StateGraph(ParentState)
parent_builder.add_node("parent", parent_node)
parent_builder.add_node("child", child_graph)  # 嵌套子图
parent_builder.add_edge(START, "parent")
parent_builder.add_edge("parent", "child")
parent_builder.add_edge("child", END)

graph = parent_builder.compile()

result = graph.invoke({"value": ""})
print(result)
# {'value': '起点 → 子图处理 → 孙图处理'}
```

**命名空间追踪**：

```
命名空间                              执行的节点
──────────                            ──────────
""                                    parent_node
"child:task_001"                      child_node
"child:task_001|grandchild:task_002"  grandchild_node
```

### 示例 4：RAG 场景——检索子图 + 生成子图

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# ========== 检索子图 ==========
class RetrievalState(TypedDict):
    query: str           # 共享键
    chunks: list[str]    # 私有
    scores: list[float]  # 私有
    context: str         # 共享键

def embed_query(state: RetrievalState):
    return {"chunks": [f"chunk about {state['query']}"]}

def search_and_rerank(state: RetrievalState):
    return {
        "scores": [0.95],
        "context": " | ".join(state["chunks"]),
    }

retrieval_builder = StateGraph(RetrievalState)
retrieval_builder.add_node("embed", embed_query)
retrieval_builder.add_node("search", search_and_rerank)
retrieval_builder.add_edge(START, "embed")
retrieval_builder.add_edge("embed", "search")
retrieval_builder.add_edge("search", END)
retrieval_graph = retrieval_builder.compile()

# ========== 生成子图 ==========
class GenerationState(TypedDict):
    query: str       # 共享键
    context: str     # 共享键
    draft: str       # 私有
    answer: str      # 共享键

def draft_answer(state: GenerationState):
    return {"draft": f"草稿: 基于 '{state['context']}' 回答 '{state['query']}'"}

def refine_answer(state: GenerationState):
    return {"answer": state["draft"].replace("草稿: ", "")}

gen_builder = StateGraph(GenerationState)
gen_builder.add_node("draft", draft_answer)
gen_builder.add_node("refine", refine_answer)
gen_builder.add_edge(START, "draft")
gen_builder.add_edge("draft", "refine")
gen_builder.add_edge("refine", END)
generation_graph = gen_builder.compile()

# ========== 父图：协调器 ==========
class OrchestratorState(TypedDict):
    query: str      # 共享键
    context: str    # 共享键
    answer: str     # 共享键

parent_builder = StateGraph(OrchestratorState)
parent_builder.add_node("retrieve", retrieval_graph)    # 检索子图
parent_builder.add_node("generate", generation_graph)   # 生成子图
parent_builder.add_edge(START, "retrieve")
parent_builder.add_edge("retrieve", "generate")
parent_builder.add_edge("generate", END)

graph = parent_builder.compile()

result = graph.invoke({"query": "什么是RAG？", "context": "", "answer": ""})
print(result)
# {'query': '什么是RAG？',
#  'context': 'chunk about 什么是RAG？',
#  'answer': "基于 'chunk about 什么是RAG？' 回答 '什么是RAG？'"}
```

**隔离效果**：
- 父图看不到检索子图的 `chunks` 和 `scores`
- 父图看不到生成子图的 `draft`
- 只有 `query`、`context`、`answer` 在父子图之间流动

## 源码解析

### 关键源码 1：子图作为节点的编译处理

当编译后的子图被添加为节点时，LangGraph 会特殊处理：

```python
# 来源: libs/langgraph/langgraph/graph/state.py（概念简化版）
def add_node(self, node_name, action, ...):
    if isinstance(action, CompiledGraph):
        # action 是编译后的子图
        # 1. 找到共享键
        parent_keys = set(self.schemas[self.state_schema].keys())
        sub_input_keys = set(get_type_hints(action.input_schema).keys())
        sub_output_keys = set(get_type_hints(action.output_schema).keys())

        shared_input = parent_keys & sub_input_keys
        shared_output = parent_keys & sub_output_keys

        # 2. 创建输入映射：只传共享键给子图
        # 3. 创建输出映射：只从子图取共享键
```

### 关键源码 2：命名空间拼接

子图执行时，命名空间会被层层拼接：

```python
# 来源: libs/langgraph/langgraph/pregel/main.py（概念简化版）
def _execute_subgraph(self, subgraph, input_data, config):
    # 拼接命名空间
    parent_ns = config.get("namespace", "")
    child_ns = f"{parent_ns}{NS_SEP}{node_name}{NS_END}{task_id}" if parent_ns else f"{node_name}{NS_END}{task_id}"

    # 子图在自己的命名空间下执行
    child_config = {**config, "namespace": child_ns}
    return subgraph.invoke(input_data, child_config)
```

### 关键源码 3：Channel 完全独立

每个 StateGraph 实例有自己的 `channels` 字典：

```python
# 来源: libs/langgraph/langgraph/graph/state.py
class StateGraph:
    def __init__(self, state_schema, ...):
        self.channels: dict[str, BaseChannel] = {}  # 每个图实例独立的 Channel 池
        self.schemas: dict[type, dict] = {}
        self._add_schema(state_schema)
```

父图的 `channels` 和子图的 `channels` 是完全不同的字典对象——即使有同名字段，它们也是不同的 Channel 实例。

## 两种方式的选择指南

| 维度 | 直接添加子图 | 函数包装 |
|------|-------------|---------|
| 代码量 | 少（自动映射） | 多（手动转换） |
| 灵活性 | 低（必须有同名键） | 高（任意字段映射） |
| 适用场景 | 父子图 Schema 相似 | 父子图 Schema 完全不同 |
| 调试难度 | 低（框架处理） | 中（需要检查转换逻辑） |
| 子图复用 | 子图必须有匹配的键名 | 子图可以完全独立设计 |

**经验法则**：
- 如果子图是专门为这个父图设计的 → 直接添加（共享键）
- 如果子图是通用模块，可能被多个父图复用 → 函数包装

## 常见问题

### Q1：子图可以修改父图的状态吗？

通过共享键可以。子图返回的共享键值会自动写回父图。此外，子图还可以通过 `Command(graph=Command.PARENT)` 显式更新父图状态。

### Q2：父图可以访问子图的私有字段吗？

不能。子图的非共享字段对父图完全不可见。这是设计上的隔离保证。

### Q3：子图的 Checkpoint 和父图的 Checkpoint 是分开的吗？

是的。通过命名空间机制，子图的 Checkpoint 存储在独立的命名空间下。可以独立查看和回溯子图的状态历史。

### Q4：直接添加子图时，如果没有共享键会怎样？

如果父图和子图没有任何同名字段，子图会收到空输入，执行完毕后也不会写回任何数据到父图。这通常意味着配置有误。

### Q5：子图可以嵌套多少层？

理论上没有限制。每层嵌套都会在命名空间中增加一个层级。但实际中建议不超过 3 层，否则调试和理解成本会显著增加。

## 小结

1. **完全隔离**：每个子图有独立的 StateGraph、Channel 集合和命名空间
2. **两种通信方式**：直接添加（共享键自动通信）或函数包装（手动状态转换）
3. **共享键机制**：父图和子图中同名的字段自动在两者之间传递
4. **命名空间隔离**：通过 `NS_SEP` (`|`) 和 `NS_END` (`:`) 拼接，确保 Checkpoint 不冲突
5. **核心价值**：模块化、可复用、可独立测试——把复杂系统拆成互不干扰的子系统
