# 条件分支策略 - 反直觉点

> 条件分支策略中有三个极易踩坑的误区，每一个都可能让你的工作流产生意想不到的行为。本节用代码实证逐一击破。

---

## 误区1："条件边只能有两个分支" ❌

### 为什么错？

很多人以为条件边就是 if-else 二选一。

**事实是：条件边可以有任意多个分支，路由函数可以返回任何已注册的节点名。**

条件边的本质不是"二选一的开关"，而是"多路分发的路由器"。路由函数的返回值决定了走哪条路，而返回值可以是任意节点名——甚至可以返回一个列表，同时激活多个节点。

来看三种不同的分支模式：

1. **二选一**：`return "A" or "B"`（最常见，但只是最简单的用法）
2. **多选一**：`return "A" or "B" or "C" or "D" or ...`（任意多个分支）
3. **多选多**：`return ["A", "B", "C"]`（同时激活多个节点）

### 为什么人们容易这样错？

**if-else 二元思维的惯性：** 编程入门时，条件判断几乎都是 if-else 二选一。即使学了 switch/case，大多数人的第一反应仍然是"条件 = 二选一"。

**教程偏差：** 绝大多数 LangGraph 入门教程只展示最简单的二选一示例：

```python
# 教程里最常见的例子（二选一）
def route(state):
    if state["score"] > 0.8:
        return "approve"
    return "reject"
```

看多了这种例子，自然会以为条件边只能有两个分支。

**"条件"这个词的误导：** "条件"在日常语言中暗示"是或否"，但 LangGraph 的 `add_conditional_edges` 更准确的名字应该是 `add_dynamic_routing_edges`——它是一个动态路由器，不是一个布尔开关。

### 正确理解

```python
"""
误区1 实证：条件边支持任意多个分支
"""
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    query: str
    intent: str
    result: str


def classify_intent(state: State) -> dict:
    """模拟意图分类：5种意图"""
    query = state["query"].lower()
    if "搜索" in query or "查找" in query:
        intent = "search"
    elif "代码" in query or "编程" in query:
        intent = "code"
    elif "聊天" in query or "你好" in query:
        intent = "chat"
    elif "计算" in query or "数学" in query:
        intent = "math"
    else:
        intent = "unknown"
    return {"intent": intent}


# ✅ 路由函数可以返回 5 个不同的节点名
def route_by_intent(state: State) -> Literal["search", "code", "chat", "math", "__end__"]:
    """5 路分支，不是只有 2 路！"""
    intent = state["intent"]
    if intent == "unknown":
        return "__end__"  # 直接结束
    return intent


def search_node(state: State) -> dict:
    return {"result": f"搜索结果: {state['query']}"}

def code_node(state: State) -> dict:
    return {"result": f"代码生成: {state['query']}"}

def chat_node(state: State) -> dict:
    return {"result": f"对话回复: {state['query']}"}

def math_node(state: State) -> dict:
    return {"result": f"数学计算: {state['query']}"}


builder = StateGraph(State)
builder.add_node("classifier", classify_intent)
builder.add_node("search", search_node)
builder.add_node("code", code_node)
builder.add_node("chat", chat_node)
builder.add_node("math", math_node)

builder.add_edge(START, "classifier")
# ✅ 一个条件边，5 个可能的目标
builder.add_conditional_edges("classifier", route_by_intent)
for node in ["search", "code", "chat", "math"]:
    builder.add_edge(node, END)

graph = builder.compile()

# 测试 5 种不同意图
test_queries = [
    "帮我搜索 Python 教程",
    "写一段快速排序代码",
    "你好，今天天气怎么样",
    "计算 123 × 456",
    "随便说点什么",  # unknown → 直接结束
]

for query in test_queries:
    result = graph.invoke({"query": query, "intent": "", "result": ""})
    print(f"'{query}' → {result.get('result', '(直接结束)')}")
```

**运行输出：**
```
'帮我搜索 Python 教程' → 搜索结果: 帮我搜索 Python 教程
'写一段快速排序代码' → 代码生成: 写一段快速排序代码
'你好，今天天气怎么样' → 对话回复: 你好，今天天气怎么样
'计算 123 × 456' → 数学计算: 计算 123 × 456
'随便说点什么' → (直接结束)
```

更进一步——路由函数还可以返回列表，同时激活多个节点：

```python
"""
进阶：路由函数返回列表，同时激活多个节点
"""
from typing import TypedDict, Annotated, Sequence
import operator
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    query: str
    results: Annotated[list[str], operator.add]


def multi_route(state: State) -> Sequence[str]:
    """根据需求同时激活多个处理节点"""
    query = state["query"]
    targets = []
    if "分析" in query:
        targets.append("analyze")
    if "翻译" in query:
        targets.append("translate")
    if "摘要" in query:
        targets.append("summarize")
    return targets if targets else ["default"]


def analyze(state: State) -> dict:
    return {"results": ["分析完成"]}

def translate(state: State) -> dict:
    return {"results": ["翻译完成"]}

def summarize(state: State) -> dict:
    return {"results": ["摘要完成"]}

def default(state: State) -> dict:
    return {"results": ["默认处理"]}

def collect(state: State) -> dict:
    print(f"汇总: {state['results']}")
    return {}


builder = StateGraph(State)
for name, fn in [("analyze", analyze), ("translate", translate),
                 ("summarize", summarize), ("default", default),
                 ("collect", collect)]:
    builder.add_node(name, fn)

builder.add_conditional_edges(START, multi_route)
for node in ["analyze", "translate", "summarize", "default"]:
    builder.add_edge(node, "collect")
builder.add_edge("collect", END)

graph = builder.compile()

# 同时触发"分析"和"翻译"两个节点
result = graph.invoke({"query": "请分析并翻译这段文字", "results": []})
print(f"结果: {result['results']}")
# 汇总: ['分析完成', '翻译完成']
```

### 记忆口诀

> **条件边不是"开关"，是"路由器"——开关只有开和关，路由器可以通向任意多个目的地。**

## 误区2："Send 和条件边返回多个节点名是一样的" ❌

### 为什么错？

表面上看，Send 和条件边返回列表都能"同时执行多个节点"。但它们有一个本质区别：

| 特性 | 条件边返回列表 | Send |
|------|--------------|------|
| 每个节点收到的状态 | **相同**（共享完整图状态） | **不同**（每个 Send 携带独立状态） |
| 并行数量 | 运行时决定，但目标节点必须预先注册 | 运行时决定，可以多次 Send 到同一个节点 |
| 典型场景 | 同时做分析 + 翻译 + 摘要（不同任务） | 对 N 篇文档做相同处理（相同任务，不同数据） |

**关键区别：Send 创建的是独立的任务实例，每个实例有自己的输入状态。条件边返回列表只是激活已有节点，所有节点共享同一份状态。**

### 为什么人们容易这样错？

**表面效果相似：** 两者都能"同时执行多个节点"，运行结果看起来差不多。

**忽略了状态传递的差异：** 如果你不关心每个节点收到什么状态，两者确实看起来一样。但在实际应用中，"每个节点收到不同的状态"是 Send 最核心的能力。

**类比混淆：** 就像"给三个人发同一封邮件"和"给三个人发不同的邮件"——动作看起来一样（都是发邮件），但内容完全不同。

### 正确理解

```python
"""
误区2 实证：条件边多路 vs Send 的本质区别
"""
import operator
from typing import TypedDict, Annotated, Sequence
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send


# ===== 方案1：条件边返回列表（所有节点收到相同状态）=====
print("=" * 60)
print("方案1：条件边返回列表 - 所有节点收到【相同】状态")
print("=" * 60)


class SharedState(TypedDict):
    document: str  # 所有节点共享同一个文档
    results: Annotated[list[str], operator.add]


def multi_route(state: SharedState) -> Sequence[str]:
    """返回多个节点名 → 所有节点收到相同的 state"""
    return ["node_a", "node_b"]


def node_a(state: SharedState) -> dict:
    # 收到完整的 state，包含 document
    print(f"  [A] 收到文档: '{state['document']}'")
    return {"results": [f"A处理了'{state['document']}'"]}


def node_b(state: SharedState) -> dict:
    # 也收到完整的 state，和 A 收到的一模一样
    print(f"  [B] 收到文档: '{state['document']}'")
    return {"results": [f"B处理了'{state['document']}'"]}


g1 = StateGraph(SharedState)
g1.add_node("node_a", node_a)
g1.add_node("node_b", node_b)
g1.add_conditional_edges(START, multi_route)
g1.add_edge("node_a", END)
g1.add_edge("node_b", END)

app1 = g1.compile()
r1 = app1.invoke({"document": "同一份报告", "results": []})
print(f"结果: {r1['results']}\n")
# A 和 B 收到的是同一份文档！


# ===== 方案2：Send（每个节点收到不同状态）=====
print("=" * 60)
print("方案2：Send - 每个节点收到【不同】状态")
print("=" * 60)


class MainState(TypedDict):
    documents: list[str]  # 多个文档
    results: Annotated[list[str], operator.add]


class WorkerState(TypedDict):
    doc_name: str  # 每个 worker 只收到自己的文档
    results: Annotated[list[str], operator.add]


def dispatch(state: MainState):
    """每个 Send 携带不同的文档"""
    return [
        Send("worker", {"doc_name": doc, "results": []})
        for doc in state["documents"]
    ]


def worker(state: WorkerState) -> dict:
    # 每个 worker 只收到自己的文档，不是全部文档
    print(f"  [Worker] 收到文档: '{state['doc_name']}'")
    return {"results": [f"处理了'{state['doc_name']}'"]}


g2 = StateGraph(MainState)
g2.add_node("worker", worker)
g2.add_conditional_edges(START, dispatch)
g2.add_edge("worker", END)

app2 = g2.compile()
r2 = app2.invoke({
    "documents": ["报告A", "报告B", "报告C"],
    "results": [],
})
print(f"结果: {r2['results']}")
# 每个 worker 收到的是不同的文档！
```

**运行输出：**
```
============================================================
方案1：条件边返回列表 - 所有节点收到【相同】状态
============================================================
  [A] 收到文档: '同一份报告'
  [B] 收到文档: '同一份报告'
结果: ["A处理了'同一份报告'", "B处理了'同一份报告'"]

============================================================
方案2：Send - 每个节点收到【不同】状态
============================================================
  [Worker] 收到文档: '报告A'
  [Worker] 收到文档: '报告B'
  [Worker] 收到文档: '报告C'
结果: ["处理了'报告A'", "处理了'报告B'", "处理了'报告C'"]
```

看到区别了吗？

- **条件边多路**：A 和 B 都收到 `"同一份报告"`——它们共享状态
- **Send**：三个 worker 分别收到 `"报告A"`、`"报告B"`、`"报告C"`——每个有独立状态

### 什么时候用哪个？

| 场景 | 选择 | 原因 |
|------|------|------|
| 同时做分析 + 翻译 + 摘要 | 条件边返回列表 | 不同任务，处理同一份数据 |
| 对 N 篇文档做相同处理 | Send | 相同任务，处理不同数据 |
| 多个 Agent 协作处理同一问题 | 条件边返回列表 | 不同角色，共享上下文 |
| 批量处理用户请求 | Send | 相同流程，不同输入 |

### 记忆口诀

> **条件边多路是"群发同一封邮件"，Send 是"给每个人写不同的信"——看起来都是发邮件，但内容天差地别。**

## 误区3："Command 可以完全替代 add_conditional_edges" ❌

### 为什么错？

Command 确实很强大——它能在节点内部同时完成状态更新和路由控制。但这不意味着它可以完全替代 `add_conditional_edges`。两者有三个关键差异：

**1. 可视化差异（最重要）**

`add_conditional_edges` 在图的 Mermaid 可视化中能显示所有可能的路径，Command 做不到。

```python
# 条件边：Mermaid 图能显示所有分支
builder.add_conditional_edges("router", route_fn, {
    "search": "search_node",
    "code": "code_node",
    "chat": "chat_node",
})
# Mermaid 图会画出 router → search_node, router → code_node, router → chat_node

# Command：Mermaid 图只能看到类型注解中的目标
def router(state) -> Command[Literal["search_node", "code_node", "chat_node"]]:
    return Command(goto="search_node")
# Mermaid 图能显示可能的目标，但不如条件边直观
```

**2. 定义位置差异**

- `add_conditional_edges`：路由逻辑定义在**图的边**上，与节点逻辑分离
- `Command`：路由逻辑定义在**节点内部**，与业务逻辑耦合

**3. 适用场景差异**

- 条件边：适合**静态已知**的分支（编译时就知道有哪些可能路径）
- Command：适合**动态决策**的场景（运行时才知道要去哪里，或者需要同时更新状态）

### 为什么人们容易这样错？

**"新 API 一定比旧 API 好"的思维定式：** Command 是 LangGraph 较新引入的 API，功能更强大。很多人自然会想"既然 Command 能做条件边的事，那就全用 Command 好了"。

**过度追求灵活性：** Command 确实更灵活，但灵活性是有代价的——代码可读性下降，图结构不透明，调试更困难。

**忽略了团队协作：** 当你一个人写代码时，Command 的灵活性很爽。但当团队成员需要理解你的工作流时，条件边的显式声明比 Command 的隐式路由更容易理解。

### 正确理解

```python
"""
误区3 实证：条件边 vs Command 的适用场景对比
"""
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.types import Command


class State(TypedDict):
    query: str
    category: str
    confidence: float
    result: str


# ===== 场景1：静态分支 → 用条件边（推荐）=====
print("=" * 60)
print("场景1：静态分支 → add_conditional_edges（推荐）")
print("=" * 60)


def classifier(state: State) -> dict:
    """纯粹的分类逻辑，不涉及路由"""
    query = state["query"]
    if "搜索" in query:
        return {"category": "search", "confidence": 0.95}
    elif "代码" in query:
        return {"category": "code", "confidence": 0.88}
    return {"category": "chat", "confidence": 0.75}


def route_fn(state: State) -> Literal["search", "code", "chat"]:
    """路由逻辑独立于业务逻辑，清晰明了"""
    return state["category"]


def search(state: State) -> dict:
    return {"result": "搜索结果"}

def code(state: State) -> dict:
    return {"result": "代码结果"}

def chat(state: State) -> dict:
    return {"result": "对话结果"}


g1 = StateGraph(State)
g1.add_node("classifier", classifier)
g1.add_node("search", search)
g1.add_node("code", code)
g1.add_node("chat", chat)

g1.add_edge(START, "classifier")
# ✅ 条件边：所有可能路径一目了然
g1.add_conditional_edges("classifier", route_fn, {
    "search": "search",
    "code": "code",
    "chat": "chat",
})
for node in ["search", "code", "chat"]:
    g1.add_edge(node, END)

app1 = g1.compile()

# 打印 Mermaid 图 → 能看到所有分支
print("Mermaid 图（条件边版本）：")
print(app1.get_graph().draw_mermaid())

result1 = app1.invoke({
    "query": "帮我搜索资料",
    "category": "", "confidence": 0.0, "result": "",
})
print(f"结果: {result1['result']}\n")


# ===== 场景2：动态决策 → 用 Command（推荐）=====
print("=" * 60)
print("场景2：动态决策 → Command（推荐）")
print("=" * 60)


class DynamicState(TypedDict):
    query: str
    retry_count: int
    confidence: float
    result: str


def smart_processor(state: DynamicState) -> Command[Literal["refine", "output", "__end__"]]:
    """
    需要同时更新状态 + 决定路由 → Command 更合适
    这里的路由决策依赖于处理过程中产生的中间结果
    """
    # 模拟处理
    confidence = 0.3 + state["retry_count"] * 0.3

    if confidence >= 0.8:
        # 质量达标：更新结果 + 路由到输出
        return Command(
            update={"confidence": confidence, "result": "高质量结果"},
            goto="output",
        )
    elif state["retry_count"] < 3:
        # 质量不够：更新重试计数 + 路由回自己
        return Command(
            update={"confidence": confidence, "retry_count": state["retry_count"] + 1},
            goto="refine",
        )
    else:
        # 重试次数用完：更新降级结果 + 结束
        return Command(
            update={"confidence": confidence, "result": "降级结果"},
            goto="__end__",
        )


def refine(state: DynamicState) -> Command[Literal["smart_processor"]]:
    """优化节点：调整后重新处理"""
    print(f"  第 {state['retry_count']} 次优化，当前置信度: {state['confidence']:.1f}")
    return Command(goto="smart_processor")


def output(state: DynamicState) -> dict:
    print(f"  输出结果，置信度: {state['confidence']:.1f}")
    return {}


g2 = StateGraph(DynamicState)
g2.add_node("smart_processor", smart_processor)
g2.add_node("refine", refine)
g2.add_node("output", output)

g2.add_edge(START, "smart_processor")
# 注意：不需要 add_conditional_edges，Command 自带路由
g2.add_edge("output", END)

app2 = g2.compile()

result2 = app2.invoke({
    "query": "复杂问题",
    "retry_count": 0, "confidence": 0.0, "result": "",
})
print(f"最终结果: {result2['result']}")
```

**运行输出：**
```
============================================================
场景1：静态分支 → add_conditional_edges（推荐）
============================================================
Mermaid 图（条件边版本）：
（显示 classifier → search/code/chat 的完整分支图）
结果: 搜索结果

============================================================
场景2：动态决策 → Command（推荐）
============================================================
  第 1 次优化，当前置信度: 0.3
  第 2 次优化，当前置信度: 0.6
  输出结果，置信度: 0.9
最终结果: 高质量结果
```

### 选型决策表

| 判断条件 | 选择 | 原因 |
|----------|------|------|
| 分支在编译时已知 | `add_conditional_edges` | 可视化友好，结构清晰 |
| 需要同时更新状态 + 路由 | `Command` | 原子操作，避免不一致 |
| 团队需要理解工作流 | `add_conditional_edges` | 显式声明，易于理解 |
| 路由依赖处理中间结果 | `Command` | 灵活，可在节点内决策 |
| 需要循环重试 | `Command` | 可以 goto 回自己或上游节点 |
| 简单的意图分类路由 | `add_conditional_edges` | 简单场景用简单方案 |

### 最佳实践

```python
# ✅ 最佳实践：条件边用于静态分支，Command 用于动态决策

# 静态分支：意图分类 → 不同处理节点
builder.add_conditional_edges("classifier", route_fn, {
    "search": "search_node",
    "code": "code_node",
    "chat": "chat_node",
})

# 动态决策：质量评估 → 重试或通过
def evaluator(state) -> Command[Literal["retry", "output"]]:
    if state["quality"] < 0.8 and state["retries"] < 3:
        return Command(
            update={"retries": state["retries"] + 1},
            goto="retry",
        )
    return Command(goto="output")
```

### 记忆口诀

> **条件边是"公开的路线图"，Command 是"司机的临场判断"——路线图让所有人看得懂，临场判断只有司机自己知道。优先画路线图，必要时才让司机自由发挥。**

## 三个误区总结

| 误区 | 错误直觉 | 正确理解 |
|------|----------|----------|
| 条件边只能两个分支 | 条件 = if-else 二选一 | 条件边是路由器，支持任意多路分发 |
| Send = 条件边返回列表 | 都是并行，效果一样 | Send 每个任务状态独立，条件边共享状态 |
| Command 完全替代条件边 | 新 API 一定更好 | 各有适用场景，优先条件边，必要时用 Command |

### 在 RAG 开发中的启示

这三个误区在 RAG 工作流中尤其重要：

1. **误区1（条件边多分支）：** RAG 的意图路由经常需要 5+ 个分支（搜索、问答、摘要、翻译、代码生成等）。不要因为"条件边只能二选一"的错误认知而写出一堆嵌套的二选一路由。直接用一个多路路由函数，干净利落。

2. **误区2（Send vs 条件边）：** RAG 中的文档批量处理（chunking、embedding、摘要）天然适合 Send 模式——每篇文档需要独立的处理状态。而混合检索（同时做向量检索 + 关键词检索 + 知识图谱检索）适合条件边返回列表——不同检索方式共享同一个查询状态。

3. **误区3（Command vs 条件边）：** RAG 的质量评估-重试循环（Self-RAG）适合用 Command（需要更新重试计数 + 动态路由）。而前端的意图分类路由适合用条件边（静态已知的分支，可视化友好）。
