# 核心概念1：回调系统与调试模式

> 回调系统是 LangChain 调试的基础设施——通过 Observer 模式，让 Agent 执行过程中的每一步都变得可观测。

---

## 概述

回调系统（Callback System）是 LangChain 内置的**事件监听机制**。当 Agent 执行时，每一个关键动作——LLM 调用、Tool 执行、Chain 运行——都会触发对应的回调事件。你可以注册"回调处理器"来监听这些事件，实现日志记录、性能监控、实时调试等功能。

Agent 的执行过程是一个"黑盒"，回调系统的作用就是**把黑盒变成白盒**：

```
没有回调                          有回调
┌──────────┐                    ┌──────────┐
│  输入     │                    │  输入     │
│   ↓      │                    │   ↓      │
│  ？？？  │   ──────────→     │ on_chain_start()
│  ？？？  │                    │ on_llm_new_token() × N
│   ↓      │                    │ on_agent_action()
│  输出     │                    │ on_tool_end()
└──────────┘                    │ on_agent_finish()
                                │   ↓
                                │  输出
                                └──────────┘
```

**前端类比：** 就像浏览器 DevTools 的 Network 面板，让你看到每一个 HTTP 请求的细节。

## 1. 全局调试开关

LangChain 提供两个全局开关（定义在 `langchain_core/globals.py`）：

```python
_verbose: bool = False  # 关键节点摘要
_debug: bool = False    # 一切内部细节
```

### set_debug() vs set_verbose()

```python
from langchain_core.globals import set_debug, set_verbose

# verbose：输出人类可读的执行摘要（推荐日常使用）
set_verbose(True)

# debug：输出所有内部细节，包括完整 prompt、token 数
set_debug(True)
```

### 使用场景对比

| 特性 | `set_verbose(True)` | `set_debug(True)` |
|------|---------------------|-------------------|
| **输出量** | 适中，关键节点 | 海量，每个细节 |
| **可读性** | 人类友好 | 信息密集，需要筛选 |
| **适用场景** | 日常开发、快速定位 | 深度排查、源码级调试 |
| **显示内容** | Chain 名称、Agent 动作 | 完整 prompt、序列化信息、token 统计 |
| **前端类比** | `console.log()` | Chrome DevTools 全量日志 |

### 实际开发建议

```python
import os
from langchain_core.globals import set_debug, set_verbose

def setup_debug_mode():
    """根据环境自动配置调试级别"""
    env = os.getenv("ENV", "development")
    if env == "development":
        set_verbose(True); set_debug(False)
    elif env == "debugging":
        set_verbose(True); set_debug(True)
    else:  # production
        set_verbose(False); set_debug(False)
```

## 2. 回调处理器架构

### Callback Mixin 设计（源码解析）

LangChain 使用 **Mixin 模式** 组织回调事件，每个 Mixin 对应一类组件：

```python
# 源码位置：langchain_core/callbacks/base.py

class RetrieverManagerMixin:   # 检索器事件
    def on_retriever_end(self, documents, *, run_id, **kwargs): ...
    def on_retriever_error(self, error, *, run_id, **kwargs): ...

class LLMManagerMixin:         # LLM 事件
    def on_llm_new_token(self, token, *, run_id, **kwargs): ...
    def on_llm_end(self, response, *, run_id, **kwargs): ...
    def on_llm_error(self, error, *, run_id, **kwargs): ...

class ChainManagerMixin:       # Chain + Agent 事件
    def on_chain_start(self, serialized, inputs, *, run_id, **kwargs): ...
    def on_chain_end(self, outputs, *, run_id, **kwargs): ...
    def on_agent_action(self, action, *, run_id, **kwargs): ...  # Agent 专属
    def on_agent_finish(self, finish, *, run_id, **kwargs): ...  # Agent 专属

class ToolManagerMixin:        # Tool 事件
    def on_tool_end(self, output, *, run_id, **kwargs): ...
    def on_tool_error(self, error, *, run_id, **kwargs): ...
```

`BaseCallbackHandler` 继承所有 Mixin，每个方法有默认空实现，你只需覆盖感兴趣的方法。

### 回调事件生命周期

```
Agent 执行生命周期与回调事件
═══════════════════════════════════════════════════
on_chain_start()                          ← Agent 开始执行
  │
  ├── 🧠 思考阶段（Thought）
  │     ├── on_llm_start()                ← LLM 开始推理
  │     ├── on_llm_new_token() × N        ← 逐 token 流式输出
  │     └── on_llm_end() / on_llm_error()
  │
  ├── 🔧 行动阶段（Action）
  │     ├── on_agent_action()             ← Agent 决定调用 Tool
  │     │     action.tool = "search"
  │     │     action.log = "我需要搜索..."  ← LLM 的推理过程！
  │     ├── on_tool_start()
  │     └── on_tool_end() / on_tool_error()
  │
  ├── 🔄 循环（可重复多次）
  │
  └── ✅ 完成
        ├── on_agent_finish()             ← 最终答案 + 推理过程
        └── on_chain_end()
```

**关键：** `action.log` 和 `finish.log` 包含 LLM 的推理过程，是理解 Agent "为什么这么做"的关键。

### 关键参数

| 参数 | 类型 | 作用 | 前端类比 |
|------|------|------|----------|
| `run_id` | UUID | 唯一标识本次运行 | `requestId` |
| `parent_run_id` | UUID | 父级运行 ID，形成调用链 | `traceId` + `spanId` |
| `tags` | list[str] | 分类标签，用于过滤 | CSS `class` |
| `metadata` | dict | 任意上下文信息 | HTTP headers |

通过 `run_id` / `parent_run_id` 可还原完整调用树：

```
run_id: aaa (AgentExecutor)
  ├── run_id: bbb, parent: aaa (LLM Call #1)
  ├── run_id: ccc, parent: aaa (Tool: search)
  └── run_id: ddd, parent: aaa (LLM Call #2)
```

## 3. 内置回调处理器

### StdOutCallbackHandler

打印关键执行信息到控制台，使用 ANSI 颜色高亮：

```python
from langchain_core.callbacks import StdOutCallbackHandler

handler = StdOutCallbackHandler()
result = agent_executor.invoke(
    {"input": "北京天气怎么样？"},
    config={"callbacks": [handler]}
)
# 输出：> Entering new AgentExecutor chain...
#       Action: weather_search → Observation: 晴 25°C
#       > Finished chain.
```

### StreamingStdOutCallbackHandler

实时逐 token 输出 LLM 响应，实现打字机效果：

```python
from langchain_core.callbacks import StreamingStdOutCallbackHandler

llm = ChatOpenAI(model="gpt-4o-mini", streaming=True,
                 callbacks=[StreamingStdOutCallbackHandler()])
```

### FileCallbackHandler

将所有事件持久化到文件：

```python
from langchain_core.callbacks import FileCallbackHandler
file_handler = FileCallbackHandler("agent_debug.log")
```

| 处理器 | 输出目标 | 适用场景 |
|--------|----------|----------|
| `StdOutCallbackHandler` | 控制台 | 日常开发调试 |
| `StreamingStdOutCallbackHandler` | 控制台（流式） | 观察 LLM 实时推理 |
| `FileCallbackHandler` | 文件 | 持久化日志、事后分析 |

## 4. 自定义回调处理器

### 基础自定义回调

```python
from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.agents import AgentAction, AgentFinish
from uuid import UUID
from typing import Any

class AgentDebugHandler(BaseCallbackHandler):
    """专门用于调试 Agent 的回调处理器"""

    def __init__(self):
        self.step_count = 0

    def on_chain_start(self, serialized: dict, inputs: dict,
                       *, run_id: UUID, **kwargs) -> None:
        print(f"🚀 Agent 开始执行 | 输入: {inputs.get('input', 'N/A')}")

    def on_agent_action(self, action: AgentAction,
                        *, run_id: UUID, **kwargs) -> None:
        self.step_count += 1
        print(f"🔧 第 {self.step_count} 步 | 工具: {action.tool}")
        print(f"   参数: {action.tool_input}")
        print(f"   推理: {action.log.strip()}")

    def on_tool_error(self, error: BaseException,
                      *, run_id: UUID, **kwargs) -> None:
        print(f"❌ 工具出错: {error}")

    def on_agent_finish(self, finish: AgentFinish,
                        *, run_id: UUID, **kwargs) -> None:
        print(f"✅ 完成（共 {self.step_count} 步）| 答案: {finish.output}")

# 使用
result = agent_executor.invoke(
    {"input": "帮我查一下今天的新闻"},
    config={"callbacks": [AgentDebugHandler()]}
)
```

### 高级：计时回调

```python
import time
from langchain_core.callbacks import BaseCallbackHandler
from uuid import UUID

class TimingCallbackHandler(BaseCallbackHandler):
    """记录每个步骤耗时，定位性能瓶颈"""

    def __init__(self):
        self.timings: dict[UUID, dict] = {}

    def on_llm_start(self, serialized, prompts, *, run_id, **kwargs):
        self.timings[run_id] = {"type": "llm", "start": time.time()}

    def on_llm_end(self, response, *, run_id, **kwargs):
        if run_id in self.timings:
            elapsed = time.time() - self.timings[run_id]["start"]
            print(f"⏱️  LLM 调用耗时: {elapsed:.2f}s")

    def on_tool_start(self, serialized, input_str, *, run_id, **kwargs):
        self.timings[run_id] = {"type": "tool", "name": serialized.get("name", "?"), "start": time.time()}

    def on_tool_end(self, output, *, run_id, **kwargs):
        if run_id in self.timings:
            info = self.timings[run_id]
            print(f"⏱️  工具 [{info.get('name')}] 耗时: {time.time() - info['start']:.2f}s")
```

### 高级：Token 统计回调

```python
from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.outputs import LLMResult

class TokenStatsHandler(BaseCallbackHandler):
    """统计 Token 用量和成本"""

    PRICING = {
        "gpt-4o-mini": {"input": 0.15 / 1e6, "output": 0.60 / 1e6},
        "gpt-4o": {"input": 2.50 / 1e6, "output": 10.00 / 1e6},
    }

    def __init__(self, model_name="gpt-4o-mini"):
        self.model_name = model_name
        self.total_input_tokens = 0
        self.total_output_tokens = 0
        self.llm_call_count = 0

    def on_llm_end(self, response: LLMResult, *, run_id, **kwargs):
        self.llm_call_count += 1
        if response.llm_output and "token_usage" in response.llm_output:
            usage = response.llm_output["token_usage"]
            self.total_input_tokens += usage.get("prompt_tokens", 0)
            self.total_output_tokens += usage.get("completion_tokens", 0)

    def print_summary(self):
        pricing = self.PRICING.get(self.model_name, {"input": 0, "output": 0})
        cost = self.total_input_tokens * pricing["input"] + self.total_output_tokens * pricing["output"]
        print(f"📈 LLM调用 {self.llm_call_count}次 | 输入 {self.total_input_tokens} tokens | 输出 {self.total_output_tokens} tokens | 费用 ${cost:.6f}")
```

## 5. CallbackManager 工作原理（源码解析）

### 事件分发机制

```python
# 源码简化版：manager.py 的核心逻辑
def handle_event(handlers, event_name, ignore_condition_name, *args, **kwargs):
    for handler in handlers:
        try:
            getattr(handler, event_name)(*args, **kwargs)
        except Exception as e:
            logger.warning("Error in %s.%s: %s", handler.__class__.__name__, event_name, repr(e))
            if handler.raise_error:  # 可选：严格模式下向上传播错误
                raise
```

**关键设计：** 默认情况下回调错误不会中断 Agent 执行（容错优先）。设置 `raise_error=True` 可切换为严格模式。

### 回调继承

回调处理器**自动向下传递**给子组件：

```
AgentExecutor（callbacks=[handler_A, handler_B]）
  ├── LLM 调用 → 自动继承 handler_A, handler_B ✅
  ├── Tool 调用 → 自动继承 ✅
  │     └── Tool 内部的 Chain → 也继承 ✅
  └── 你不需要手动传递回调给每个子组件！
```

源码中通过 `configure()` 方法区分 `inheritable_callbacks`（可继承）和 `local_callbacks`（仅本级）。

### 线程安全

```python
class ContextThreadPoolExecutor(ThreadPoolExecutor):
    """保持上下文的线程池执行器"""
    def submit(self, func, *args, **kwargs):
        ctx = copy_context()  # 捕获当前线程上下文
        return super().submit(ctx.run, func, *args, **kwargs)  # 新线程中恢复
```

保证并行调用多个 Tool 时，回调上下文正确隔离。

## 6. 实际应用：Agent 调试工作流

### 开发阶段调试清单

```
Step 1️⃣  set_verbose(True) → 查看基本执行流程
Step 2️⃣  on_agent_action().log → Agent 在"想什么"？选对工具了吗？
Step 3️⃣  on_tool_start/end → 参数传对了吗？返回值合理吗？
Step 4️⃣  set_debug(True) → 检查完整 Prompt 是否包含正确上下文
Step 5️⃣  TokenStatsHandler → 是否超出 context window？
Step 6️⃣  LangSmith → 可视化完整调用链（见核心概念2）
```

### 常见调试场景

| 问题现象 | 排查方法 |
|----------|----------|
| Agent 返回错误答案 | 检查 `on_agent_action().log` 每步推理 |
| Agent 选错了 Tool | 检查 Tool 的 description 是否清晰 |
| Agent 陷入循环 | 步数统计 + 检查是否在重复相同操作 |
| Tool 执行报错 | `on_tool_error()` 检查 `tool_input` 格式 |
| 响应速度慢 | `TimingCallbackHandler` 定位耗时步骤 |
| 费用过高 | `TokenStatsHandler` 分析 token 用量 |

### 组合使用多个回调

```python
handlers = [
    AgentDebugHandler(),                   # Agent 行为跟踪
    TimingCallbackHandler(),               # 性能计时
    TokenStatsHandler("gpt-4o-mini"),      # Token 统计
    FileCallbackHandler("debug.log"),      # 持久化日志
]

result = agent_executor.invoke(
    {"input": "分析这份报告的关键数据"},
    config={"callbacks": handlers}
)
```

## 小结

| 要点 | 说明 |
|------|------|
| **全局开关** | `set_verbose(True)` 日常用，`set_debug(True)` 深度排查 |
| **Mixin 模式** | 4 类 Mixin 覆盖 LLM/Chain/Tool/Retriever 全部事件 |
| **Agent 专属事件** | `on_agent_action()` 和 `on_agent_finish()` 是调试核心 |
| **`action.log`** | 包含 LLM 推理过程，理解 Agent 决策的关键线索 |
| **内置处理器** | StdOut（控制台）、Streaming（流式）、File（文件） |
| **自定义处理器** | 继承 `BaseCallbackHandler`，覆盖你关心的方法即可 |
| **回调继承** | 只需在最外层注册，子组件自动继承 |
| **容错设计** | 默认不中断执行，`raise_error=True` 切换为严格模式 |

**一句话记住：** 回调系统是 Agent 的"监控摄像头"——看关键画面（verbose）、看全部录像（debug），或安装自定义摄像头（自定义 Handler）对准你最关心的环节。
