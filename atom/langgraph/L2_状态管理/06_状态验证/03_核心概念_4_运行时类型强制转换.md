# 核心概念 4：运行时类型强制转换（Type Coercion）

## 概述

类型强制转换（Type Coercion）是 Pydantic 自动将兼容类型转换为目标类型的能力。当你传入 `"42"` 给一个 `int` 字段时，Pydantic 不会报错，而是悄悄帮你转成 `42`。在 LangGraph 中，这个特性至关重要——因为 LLM 的输出几乎都是字符串，而你的状态字段可能是 `int`、`float`、`bool`、`datetime` 等各种类型。Pydantic 的类型转换充当了"LLM 输出"和"结构化状态"之间的自动翻译层。

[来源: reference/context7_pydantic_01.md | Pydantic 官方文档]

## 为什么 LLM 场景特别需要类型转换？

在传统 Web 开发中，前后端之间有明确的 API 契约，数据类型通常是确定的。但在 LLM 应用中，情况完全不同：

### 1. LLM 输出天然是字符串

LLM 本质上是一个"文本生成器"，它的输出永远是字符串。即使你让它返回一个数字，拿到的也是 `"42"` 而不是 `42`：

```python
# LLM 返回的 JSON 字符串解析后
llm_output = json.loads('{"count": "42", "active": "true", "score": "0.95"}')
# llm_output = {"count": "42", "active": "true", "score": "0.95"}
# 所有值都是字符串！
```

### 2. JSON 解析的类型模糊

JSON 规范中只有 `number` 类型，没有 `int` 和 `float` 的区分。解析后 `42` 可能是 `int` 也可能是 `float`，取决于解析器的实现。更常见的情况是，LLM 把数字用引号包起来变成了字符串。

### 3. API 返回的数据类型不精确

外部 API（搜索引擎、数据库、第三方服务）返回的数据格式各异。同一个字段在不同来源中可能是 `"true"`、`1`、`True` 或 `"yes"`。

**没有类型转换时**，你需要在每个节点里手动处理这些类型差异：

```python
# 😩 没有类型转换，每个节点都要手动转换
def process_node(state):
    count = int(state["count"]) if isinstance(state["count"], str) else state["count"]
    active = state["active"] in ("true", "True", "1", True)
    score = float(state["score"]) if isinstance(state["score"], str) else state["score"]
    # ... 真正的业务逻辑
```

**有了 Pydantic 类型转换**，这些全部自动完成。

## 常见转换规则速查表

Pydantic 内置了一套完整的类型转换规则。以下是 LangGraph 开发中最常遇到的转换：

| 输入值 | 目标类型 | 转换结果 | 说明 |
|--------|----------|----------|------|
| `"42"` | `int` | `42` | 字符串 → 整数 |
| `"3.14"` | `float` | `3.14` | 字符串 → 浮点数 |
| `"true"` | `bool` | `True` | 字符串 → 布尔值 |
| `"false"` | `bool` | `False` | 字符串 → 布尔值 |
| `1` | `bool` | `True` | 非零整数 → True |
| `0` | `bool` | `False` | 零 → False |
| `"2024-01-01"` | `datetime` | `datetime(2024,1,1)` | 字符串 → 日期时间 |
| `42` | `float` | `42.0` | 整数 → 浮点数 |
| `3.14` | `str` | `"3.14"` | 数字 → 字符串 |
| `{"key": "val"}` | `Model` | `Model(key="val")` | 字典 → Pydantic 模型 |

### 快速验证

```python
from pydantic import BaseModel
from datetime import datetime

class DemoState(BaseModel):
    count: int
    score: float
    active: bool
    created_at: datetime

# 全部用字符串传入
state = DemoState(
    count="42",
    score="3.14",
    active="true",
    created_at="2024-01-01"
)

print(f"count: {state.count} ({type(state.count).__name__})")
# count: 42 (int)
print(f"score: {state.score} ({type(state.score).__name__})")
# score: 3.14 (float)
print(f"active: {state.active} ({type(state.active).__name__})")
# active: True (bool)
print(f"created_at: {state.created_at} ({type(state.created_at).__name__})")
# created_at: 2024-01-01 00:00:00 (datetime)
```

**注意**：并非所有转换都能成功。`"hello"` 无法转为 `int`，`"maybe"` 无法转为 `bool`。Pydantic 只处理"合理的"转换。

## 在 LangGraph 中的完整示例

### 基础示例：字符串输入自动转换

```python
from langgraph.graph import StateGraph, START, END
from pydantic import BaseModel

class CoercionState(BaseModel):
    count: int
    active: bool
    score: float

def inspect_node(state: CoercionState):
    """检查类型转换结果"""
    print(f"count: {state.count} (type: {type(state.count).__name__})")
    print(f"active: {state.active} (type: {type(state.active).__name__})")
    print(f"score: {state.score} (type: {type(state.score).__name__})")
    return {}

builder = StateGraph(CoercionState)
builder.add_node("inspect", inspect_node)
builder.add_edge(START, "inspect")
builder.add_edge("inspect", END)
graph = builder.compile()

# 字符串输入会被自动转换
result = graph.invoke({"count": "42", "active": "true", "score": "0.95"})
# 输出：
# count: 42 (type: int)
# active: True (type: bool)
# score: 0.95 (type: float)
```

这里发生了什么？当 `graph.invoke()` 接收到字典 `{"count": "42", ...}` 时，LangGraph 内部调用 `CoercionState(**input)`，Pydantic 自动将 `"42"` 转为 `42`、`"true"` 转为 `True`、`"0.95"` 转为 `0.95`。

### RAG 实战示例：处理 LLM 输出

在真实的 RAG 场景中，LLM 返回的结构化数据经常需要类型转换：

```python
import json
from pydantic import BaseModel
from langgraph.graph import StateGraph, START, END

class RAGState(BaseModel):
    query: str
    num_results: int = 5
    confidence: float = 0.0
    is_relevant: bool = False
    answer: str = ""

def llm_parse_node(state: RAGState):
    """模拟 LLM 返回的 JSON 解析结果（所有值都是字符串）"""
    llm_response = '{"num_results": "3", "confidence": "0.87", "is_relevant": "true"}'
    parsed = json.loads(llm_response)
    # parsed = {"num_results": "3", "confidence": "0.87", "is_relevant": "true"}
    # 全是字符串！但 Pydantic 会自动转换
    return parsed

def generate_node(state: RAGState):
    """此时 state 中的字段已经是正确类型"""
    assert isinstance(state.num_results, int)       # ✅ 已转为 int
    assert isinstance(state.confidence, float)       # ✅ 已转为 float
    assert isinstance(state.is_relevant, bool)       # ✅ 已转为 bool
    return {"answer": f"找到 {state.num_results} 条结果，置信度 {state.confidence}"}

builder = StateGraph(RAGState)
builder.add_node("parse", llm_parse_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "parse")
builder.add_edge("parse", "generate")
builder.add_edge("generate", END)
graph = builder.compile()

result = graph.invoke({"query": "什么是向量数据库？"})
print(result)
```

## 源码解析：类型转换的触发机制

LangGraph 通过两个关键函数实现类型转换。理解它们能帮你预判转换行为。

### `_pick_mapper`：决定是否启用转换

```python
# langgraph/graph/state.py (简化版)
from functools import partial
from inspect import isclass

def _pick_mapper(state_keys, schema):
    # 情况1：根模型，不需要转换
    if state_keys == ["__root__"]:
        return None
    # 情况2：schema 是 dict 的子类（如 TypedDict），不需要转换
    if isclass(schema) and issubclass(schema, dict):
        return None
    # 情况3：Pydantic BaseModel，启用转换
    return partial(_coerce_state, schema)
```

### `_coerce_state`：实际执行转换

```python
# langgraph/graph/state.py
def _coerce_state(schema, input):
    return schema(**input)  # 触发 Pydantic 的完整验证 + 类型转换
```

**核心逻辑**：`schema(**input)` 就是 `CoercionState(**{"count": "42", ...})`，等价于 `CoercionState(count="42", ...)`。Pydantic 在构造实例时自动执行类型转换。

### 完整调用链

```
graph.invoke({"count": "42", "active": "true", "score": "0.95"})
    │
    ▼
_pick_mapper(state_keys, CoercionState)
    │  CoercionState 不是 dict 子类
    ▼
返回 partial(_coerce_state, CoercionState)
    │
    ▼
_coerce_state(CoercionState, {"count": "42", "active": "true", "score": "0.95"})
    │
    ▼
CoercionState(count="42", active="true", score="0.95")
    │  Pydantic 内部：
    │  "42" → int(42) ✅
    │  "true" → bool(True) ✅
    │  "0.95" → float(0.95) ✅
    ▼
返回 CoercionState 实例，传入第一个节点
```

## 转换失败的处理

当输入值无法转换为目标类型时，Pydantic 会抛出 `ValidationError`。在 LangGraph 中，这意味着 `graph.invoke()` 会直接抛出异常。

### 基础错误捕获

```python
from langgraph.graph import StateGraph, START, END
from pydantic import BaseModel, ValidationError

class CoercionState(BaseModel):
    count: int
    active: bool
    score: float

def inspect_node(state: CoercionState):
    return {}

builder = StateGraph(CoercionState)
builder.add_node("inspect", inspect_node)
builder.add_edge(START, "inspect")
builder.add_edge("inspect", END)
graph = builder.compile()

# "not-a-number" 无法转为 int
try:
    graph.invoke({"count": "not-a-number", "active": "true", "score": "0.95"})
except Exception as e:
    print(f"转换失败: {e}")
    # 转换失败: 1 validation error for CoercionState
    # count
    #   Input should be a valid integer, unable to parse string as an integer
```

### 常见的转换失败场景

```python
from pydantic import BaseModel, ValidationError

class TestState(BaseModel):
    count: int
    active: bool
    score: float

# ❌ "abc" 无法转为 int → int_parsing 错误
# ❌ "maybe" 无法转为 bool（只有 "true"/"false"/"1"/"0" 可以）
# ❌ "高" 无法转为 float → float_parsing 错误
```

实践建议：为可能接收 LLM 输出的字段提供默认值，减少转换失败的影响。

## strict 模式：禁用类型转换

有时候你不希望 Pydantic 自动转换类型——你想要严格匹配，传入 `"42"` 就应该报错，因为你期望的是 `int` 而不是 `str`。这时可以启用 strict 模式。

### 全局 strict 模式

```python
from pydantic import BaseModel, ConfigDict, ValidationError

class StrictState(BaseModel):
    model_config = ConfigDict(strict=True)

    count: int      # 只接受 int，不接受 "42"
    active: bool    # 只接受 bool，不接受 "true"
    score: float    # 只接受 float/int，不接受 "0.95"

# ✅ 精确类型，通过
state = StrictState(count=42, active=True, score=0.95)
print(state)

# ❌ 字符串输入，即使可以转换也会报错
try:
    StrictState(count="42", active="true", score="0.95")
except ValidationError as e:
    print(f"strict 模式拒绝: {e.error_count()} 个错误")
    for err in e.errors():
        print(f"  - {err['loc'][0]}: {err['msg']}")
    # strict 模式拒绝: 3 个错误
    #   - count: Input should be a valid integer
    #   - active: Input should be a valid boolean
    #   - score: Input should be a valid number
```

### 字段级 strict 模式

如果只想对部分字段启用 strict，可以使用 `Field` 的 `strict` 参数：

```python
from pydantic import BaseModel, Field

class MixedState(BaseModel):
    # 这个字段严格模式：只接受 int
    count: int = Field(strict=True)
    # 这个字段宽松模式：接受 "true" 等字符串
    active: bool = Field(default=False)
    # 这个字段宽松模式：接受 "0.95" 等字符串
    score: float = Field(default=0.0)

# ✅ count 是 int，其他字段用字符串
state = MixedState(count=42, active="true", score="0.95")
print(state)  # count=42 active=True score=0.95

# ❌ count 用字符串会报错
try:
    MixedState(count="42", active="true", score="0.95")
except ValidationError as e:
    print(f"count 字段拒绝字符串: {e.errors()[0]['msg']}")
```

### 什么时候用 strict 模式？

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 接收 LLM 输出 | 宽松（默认） | LLM 输出多为字符串，需要自动转换 |
| 内部节点间传递 | 宽松（默认） | 减少样板代码 |
| 安全敏感字段 | strict | 防止意外的隐式转换 |
| 金融/计算精度 | strict | 避免 `"1"` 被转为 `True`（bool 是 int 子类） |
| 外部 API 输入 | strict | 强制调用方传入正确类型 |

## 与 TypedDict 的对比：零转换 vs 自动转换

TypedDict 不做任何运行时转换，数据原样传入、原样使用。

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class TypedDictState(TypedDict):
    count: int
    active: bool
    score: float

def inspect_node(state):
    print(f"count: {state['count']} (type: {type(state['count']).__name__})")
    return {}

builder = StateGraph(TypedDictState)
builder.add_node("inspect", inspect_node)
builder.add_edge(START, "inspect")
builder.add_edge("inspect", END)
graph = builder.compile()

# 传入字符串，拿到的还是字符串！
result = graph.invoke({"count": "42", "active": "true", "score": "0.95"})
# count: 42 (type: str)  ← 没有转换，还是字符串
```

同样的输入，用 Pydantic BaseModel 定义状态时，`"42"` 会自动变成 `42`（int）。

### 对比总结

| 行为 | TypedDict | Pydantic BaseModel |
|------|-----------|-------------------|
| `"42"` → `int` 字段 | 保持 `"42"`（str） | 转为 `42`（int） |
| `"true"` → `bool` 字段 | 保持 `"true"`（str） | 转为 `True`（bool） |
| `"abc"` → `int` 字段 | 保持 `"abc"`（str） | 抛出 `ValidationError` |
| 运行时开销 | 几乎为零 | 有少量开销 |
| 类型安全性 | 无保障 | 强保障 |

**一句话总结**：TypedDict 是"信任模式"——相信调用方传入正确类型；Pydantic 是"验证模式"——不信任任何输入，自动转换或报错。

## 进阶：嵌套模型的类型转换

Pydantic 的类型转换能递归处理嵌套模型。传入字典，自动转换为模型实例。

```python
from pydantic import BaseModel
from langgraph.graph import StateGraph, START, END

class SearchResult(BaseModel):
    title: str
    score: float
    relevant: bool

class RAGState(BaseModel):
    query: str
    results: list[SearchResult] = []

def search_node(state: RAGState):
    # 返回字典列表，Pydantic 会自动转为 SearchResult 列表
    return {
        "results": [
            {"title": "向量数据库入门", "score": "0.95", "relevant": "true"},
            {"title": "RAG 架构设计", "score": "0.87", "relevant": 1},
        ]
    }

builder = StateGraph(RAGState)
builder.add_node("search", search_node)
builder.add_edge(START, "search")
builder.add_edge("search", END)
graph = builder.compile()

result = graph.invoke({"query": "什么是向量数据库？"})
# result["results"][0] 是 SearchResult 实例
# "0.95" → float, "true" → bool, 1 → True 全部自动转换
```

双重转换同时发生：外层字典列表 → `list[SearchResult]`，内层 `"0.95"` → `float`。

## 常见陷阱

### 陷阱 1：bool 是 int 的子类

```python
from pydantic import BaseModel

class State(BaseModel):
    count: int

state = State(count=True)
print(state.count)  # 1（不是 True！bool 是 int 子类，不会报错）
```

如果字段是 `int`，传入 `True` 会变成 `1`。在 LLM 场景中可能导致语义错误。

### 陷阱 2：空字符串无法转换

```python
# State(count="") → ValidationError
# 空字符串不能转为 int/float/bool
```

### 陷阱 3：浮点数精度

```python
state = State(score="0.1")  # score: float
print(f"{state.score:.20f}")  # 0.10000000000000000555...
# 浮点数精度问题依然存在，Pydantic 不会帮你解决
```

## 小结

类型强制转换是 Pydantic 在 LangGraph 中最"润物细无声"的能力。你可能从未注意到它的存在，但它一直在帮你把 LLM 输出的字符串转换为结构化的类型。

关键要点：
- Pydantic BaseModel 默认启用类型转换，TypedDict 不做任何转换
- LangGraph 通过 `_coerce_state` → `schema(**input)` 触发 Pydantic 验证和转换
- LLM 场景特别依赖类型转换，因为 LLM 输出天然是字符串
- 需要严格类型匹配时，使用 `ConfigDict(strict=True)` 或 `Field(strict=True)`
- 转换失败会抛出 `ValidationError`，应在调用层做好错误处理
