> **目标**：模拟 ZeroClaw 配置加载流程，掌握 .context() 错误上下文注入与错误链

---

## 场景说明

ZeroClaw 启动时需要加载配置文件（读取文件 → 解析 JSON → 验证字段）。每一步都可能失败，而且失败原因各不相同。如果只看底层错误（如 "os error 2"），根本不知道是哪个文件出了问题。

**`.context()` 的作用**：给每层错误附加一句人类可读的描述，形成**错误链**。排查问题时从外到内逐层定位。

### 涉及的 ZeroClaw 源码模式

| 模式 | ZeroClaw 源文件 | 本场景对应 |
|------|----------------|-----------|
| `.context("msg")?` 读文件 | `src/security/secrets.rs` | `read_config_file()` |
| `.context("msg")?` 数据库 | `src/memory/sqlite.rs` | `parse_config()` |
| `.with_context(\|\| format!(...))` | `src/security/secrets.rs` | `read_config_file()` |
| `bail!` / `ensure!` 验证 | `src/tools/web_fetch.rs` | `validate_config()` |
| `{:#}` 格式化错误链 | anyhow 标准用法 | `main()` 中的错误输出 |

[来源: reference/source_error_handling_01.md - .context() 添加上下文, bail! 提前返回]

### src/main.rs

```rust
// ============================================================
// 场景2：模拟 ZeroClaw 配置加载流程
// ============================================================
//
// 对应 ZeroClaw 源码：
//   - src/security/secrets.rs → fs::read_to_string(&path).context("...")?
//   - src/memory/sqlite.rs    → Connection::open(&path).context("...")?
//   - src/tools/web_fetch.rs  → bail! / ensure! 验证配置值
//   - src/config/schema.rs    → Config 结构体 + Option<T> 字段
//
// 核心学习点：
//   1. .context() vs .with_context() 的区别
//   2. 错误链的层层嵌套
//   3. {:#} 格式化输出完整错误链
// ============================================================

use anyhow::{Result, bail, ensure, Context};
use serde::Deserialize;
use std::fs;

// ----------------------------------------------------------
// 1. Config 结构体（模拟 src/config/schema.rs）
// ----------------------------------------------------------
// ZeroClaw 的配置包含必填字段和可选字段。
// 必填字段：String（反序列化时 null/缺失 → Err）
// 可选字段：Option<String>（null/缺失 → None）

#[derive(Debug, Deserialize)]
struct Config {
    provider: String,              // 必填：LLM 提供商名称
    api_key: Option<String>,       // 可选：Ollama 不需要
    api_url: Option<String>,       // 可选：不设置用默认值
    model: Option<String>,         // 可选：不设置用默认模型
    max_tokens: Option<u32>,       // 可选：不设置用默认值
}

// ----------------------------------------------------------
// 2. read_config_file — 读取文件 + .context()
// ----------------------------------------------------------
// 对应 ZeroClaw：
//   src/security/secrets.rs → fs::read_to_string(&self.key_path)
//                                .context("Failed to read secret key file")?
//
// .context("msg") 给底层 io::Error 加一层描述。
// .with_context(|| format!("...")) 需要格式化时用闭包（惰性求值）。

fn read_config_file(path: &str) -> Result<String> {
    // ★ .context()：固定字符串，大多数场景够用
    // 底层错误：No such file or directory (os error 2)
    // 加上 context 后：Failed to read config file → Caused by: os error 2

    fs::read_to_string(path)
        .with_context(|| format!("Failed to read config file '{}'", path))
    //   ^^^^^^^^^^^^^^
    //   .with_context() vs .context() 的区别：
    //   - .context("固定字符串")      → 无论成功失败都会创建字符串（但编译器可能优化）
    //   - .with_context(|| format!()) → 只在 Err 时才执行闭包（惰性求值）
    //   当需要 format! 拼接变量时，用 .with_context() 更合适
}

// ----------------------------------------------------------
// 3. parse_config — 解析 JSON + .context()
// ----------------------------------------------------------
// 对应 ZeroClaw：
//   src/memory/sqlite.rs → Connection::open(&path_buf)
//                             .context("SQLite failed to open database")?
//
// serde_json 的错误信息很技术化（如 "expected value at line 1 column 1"），
// 加上 .context() 让调用者知道"是在解析配置文件时出错的"。

fn parse_config(content: &str) -> Result<Config> {
    let config: Config = serde_json::from_str(content)
        .context("Failed to parse config as JSON")?;
    //       ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    //       底层错误：expected value at line 1 column 1
    //       加上 context：Failed to parse config as JSON
    //                     Caused by: expected value at line 1 column 1
    Ok(config)
}

// ----------------------------------------------------------
// 4. validate_config — 验证配置 + bail!/ensure!
// ----------------------------------------------------------
// 对应 ZeroClaw：
//   src/tools/web_fetch.rs → bail!("URL cannot be empty")
//   src/security/secrets.rs → ensure!(blob.len() > NONCE_LEN, "...")

fn validate_config(config: &Config) -> Result<()> {
    // ★ ensure!：provider 不能为空
    ensure!(
        !config.provider.is_empty(),
        "Config validation failed: 'provider' cannot be empty"
    );

    // ★ ensure!：provider 必须是已知的
    let known = ["openai", "anthropic", "ollama"];
    ensure!(
        known.contains(&config.provider.as_str()),
        "Unknown provider '{}'. Supported: {}",
        config.provider,
        known.join(", ")
    );

    // ★ bail!：非 ollama 必须有 api_key
    if config.provider != "ollama" && config.api_key.is_none() {
        bail!(
            "Provider '{}' requires 'api_key' but it was not set",
            config.provider
        );
    }

    // ★ ensure!：max_tokens 范围检查（如果设置了的话）
    if let Some(max_tokens) = config.max_tokens {
        ensure!(
            max_tokens >= 1 && max_tokens <= 128_000,
            "max_tokens must be 1..=128000, got {}", max_tokens
        );
    }

    Ok(())
}

// ----------------------------------------------------------
// 5. load_config — 组合以上步骤，展示错误链
// ----------------------------------------------------------
// 这是顶层组合函数。每一步都用 .context() 加一层描述，
// 最终形成多层错误链：
//   load_config 失败
//     → read_config_file 失败
//       → io::Error（底层原因）

fn load_config(path: &str) -> Result<Config> {
    let content = read_config_file(path)
        .context("Configuration loading failed")?;
    //       ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    //       再加一层 context：现在错误链有 3 层
    //       0: Configuration loading failed
    //       1: Failed to read config file 'xxx'
    //       2: No such file or directory (os error 2)

    let config = parse_config(&content)
        .context("Configuration loading failed")?;

    validate_config(&config)
        .context("Configuration loading failed")?;

    Ok(config)
}

// ----------------------------------------------------------
// 6. 辅助函数：展示错误链的不同输出格式
// ----------------------------------------------------------

fn print_error_formats(label: &str, err: &anyhow::Error) {
    // 格式 1：{} — 只打印最外层错误
    println!("  {{}}  = {}", err);

    // 格式 2：{:#} — 打印完整错误链（用 : 连接）
    println!("  {{:#}} = {:#}", err);

    // 格式 3：{:?} — Debug 格式（开发调试用）
    println!("  {{:?}} = {:?}", err);

    // 格式 4：手动遍历 .chain()（最详细）
    println!("  chain:");
    for (i, cause) in err.chain().enumerate() {
        println!("    {}: {}", i, cause);
    }
}

// ----------------------------------------------------------
// 7. main 函数：测试多种失败场景
// ----------------------------------------------------------

fn main() {
    // =====================================================
    // 场景 A：文件不存在
    // =====================================================
    println!("=== 场景 A：文件不存在 ===");
    match load_config("/tmp/nonexistent_zeroclaw_config.json") {
        Ok(config) => println!("  Loaded: {:?}", config),
        Err(e) => print_error_formats("file not found", &e),
    }

    // =====================================================
    // 场景 B：JSON 格式错误
    // =====================================================
    println!("\n=== 场景 B：JSON 格式错误 ===");
    // 先写一个格式错误的文件
    let bad_json_path = "/tmp/zeroclaw_bad_config.json";
    fs::write(bad_json_path, "{ invalid json !!!").ok();
    match load_config(bad_json_path) {
        Ok(config) => println!("  Loaded: {:?}", config),
        Err(e) => print_error_formats("bad json", &e),
    }

    // =====================================================
    // 场景 C：JSON 合法但缺少必填字段
    // =====================================================
    println!("\n=== 场景 C：缺少必填字段 provider ===");
    let missing_field_path = "/tmp/zeroclaw_missing_field.json";
    fs::write(missing_field_path, r#"{ "api_key": "sk-123" }"#).ok();
    match load_config(missing_field_path) {
        Ok(config) => println!("  Loaded: {:?}", config),
        Err(e) => print_error_formats("missing field", &e),
    }

    // =====================================================
    // 场景 D：配置值不合法（未知 provider）
    // =====================================================
    println!("\n=== 场景 D：未知 provider ===");
    let bad_provider_path = "/tmp/zeroclaw_bad_provider.json";
    fs::write(
        bad_provider_path,
        r#"{ "provider": "gemini", "api_key": "key123" }"#,
    ).ok();
    match load_config(bad_provider_path) {
        Ok(config) => println!("  Loaded: {:?}", config),
        Err(e) => print_error_formats("bad provider", &e),
    }

    // =====================================================
    // 场景 E：非 ollama 但缺少 api_key
    // =====================================================
    println!("\n=== 场景 E：openai 缺少 api_key ===");
    let no_key_path = "/tmp/zeroclaw_no_key.json";
    fs::write(no_key_path, r#"{ "provider": "openai" }"#).ok();
    match load_config(no_key_path) {
        Ok(config) => println!("  Loaded: {:?}", config),
        Err(e) => print_error_formats("no api_key", &e),
    }

    // =====================================================
    // 场景 F：正常配置
    // =====================================================
    println!("\n=== 场景 F：正常配置 ===");
    let ok_path = "/tmp/zeroclaw_ok_config.json";
    fs::write(
        ok_path,
        r#"{
            "provider": "openai",
            "api_key": "sk-test-key-123",
            "model": "gpt-4",
            "max_tokens": 4096
        }"#,
    ).ok();
    match load_config(ok_path) {
        Ok(config) => {
            println!("  Loaded successfully!");
            println!("  provider: {}", config.provider);
            println!("  api_key:  {}", config.api_key.as_deref().unwrap_or("(none)"));
            println!("  api_url:  {}", config.api_url.as_deref().unwrap_or("(default)"));
            println!("  model:    {}", config.model.as_deref().unwrap_or("(default)"));
            println!("  max_tok:  {}", config.max_tokens.map_or("(default)".into(), |n| n.to_string()));
        }
        Err(e) => print_error_formats("unexpected", &e),
    }

    // =====================================================
    // 场景 G：ollama 不需要 api_key（验证通过）
    // =====================================================
    println!("\n=== 场景 G：ollama 无需 api_key ===");
    let ollama_path = "/tmp/zeroclaw_ollama.json";
    fs::write(ollama_path, r#"{ "provider": "ollama" }"#).ok();
    match load_config(ollama_path) {
        Ok(config) => println!("  Loaded: provider={}, api_key={:?}", config.provider, config.api_key),
        Err(e) => print_error_formats("ollama", &e),
    }
}
```

## 逐模式解析

### 模式 1：.context() — 固定字符串上下文

```rust
// ZeroClaw 源码：src/memory/sqlite.rs
Connection::open(&path_buf)
    .context("SQLite failed to open database")?;

// 本场景对应：
serde_json::from_str(content)
    .context("Failed to parse config as JSON")?;
```

`.context("msg")` 在 `Err` 时给错误包一层描述，`Ok` 时什么都不做。底层错误被保留在错误链中。

[来源: reference/source_error_handling_01.md - .context() 添加上下文]

### 模式 2：.with_context() — 惰性格式化上下文

```rust
// ZeroClaw 源码：src/security/secrets.rs
fs::read_to_string(&self.key_path)
    .context("Failed to read secret key file")?;

// 本场景对应（需要拼接路径时用 with_context）：
fs::read_to_string(path)
    .with_context(|| format!("Failed to read config file '{}'", path))?;
```

**`.context()` vs `.with_context()` 选择**：

| 场景 | 用哪个 | 理由 |
|------|--------|------|
| 固定字符串 | `.context("msg")` | 简洁，大多数场景够用 |
| 需要 format! | `.with_context(\|\| format!(...))` | 闭包只在 Err 时执行，避免无谓的字符串分配 |

```typescript
// TypeScript 等价：Error cause（ES2022）
try {
  fs.readFileSync(path);
} catch (e) {
  throw new Error(`Failed to read config file '${path}'`, { cause: e });
}
```

[来源: reference/context7_anyhow_01.md - Context trait]

### 模式 3：错误链的三种输出格式

```rust
let err: anyhow::Error = /* ... */;

// {} — 只打印最外层（给用户看）
println!("{}", err);
// → "Configuration loading failed"

// {:#} — 完整错误链，用 : 连接（给日志看）
println!("{:#}", err);
// → "Configuration loading failed: Failed to read config file '...': No such file or directory (os error 2)"

// chain() — 逐层遍历（给开发者调试看）
for (i, cause) in err.chain().enumerate() {
    println!("  {}: {}", i, cause);
}
// → 0: Configuration loading failed
//   1: Failed to read config file '...'
//   2: No such file or directory (os error 2)
```

每一层 `.context()` 回答一个问题（以文件不存在为例）：

```
层 0: "Configuration loading failed"          ← 在做什么时失败了？
层 1: "Failed to read config file '/tmp/...'" ← 具体哪一步失败了？
层 2: "No such file or directory (os error 2)" ← 底层原因是什么？
```

[来源: reference/context7_anyhow_01.md - 错误链遍历]

## 小结

本场景覆盖了 ZeroClaw 配置加载的 4 个核心模式：

| 模式 | 用途 | 一句话记忆 |
|------|------|-----------|
| `.context("msg")?` | 给错误加固定描述 | "出错时告诉调用者在做什么" |
| `.with_context(\|\| format!(...))?` | 给错误加动态描述 | "需要拼接变量时用闭包" |
| `bail!` / `ensure!` | 业务规则验证 | "不满足条件就返回错误" |
| `{:#}` / `.chain()` | 输出完整错误链 | "从外到内逐层定位问题" |

**核心原则**：每个 `.context()` 回答一个问题——"在做什么时失败了？"。多层 context 形成错误链，从业务含义到底层原因，逐层递进，排查问题时不用猜。
