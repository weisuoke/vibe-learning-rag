# Agent调试与优化 - 反直觉点

> 3 个常见误区，帮你避开 Agent 调试与优化中最容易踩的坑。

---

## 误区1：「日志打得越多，调试越容易」 ❌

### 为什么错？

**过多的日志反而会让调试更困难**。当你同时开启 `set_debug(True)` + 多个回调 + LangSmith 追踪时，你看到的不是"清晰的线索"，而是"信息海啸"：

```python
# ❌ 错误做法：什么都开
from langchain_core.globals import set_debug, set_verbose
set_debug(True)
set_verbose(True)

# 再加上自定义回调
handler1 = StdOutCallbackHandler()
handler2 = FileCallbackHandler("debug.log")
handler3 = MyCustomHandler()

# 结果：控制台几百行输出，根本找不到关键信息
agent.invoke(input, config={"callbacks": [handler1, handler2, handler3]})
```

一个复杂 Agent 单次执行可能产生 **50+ 个回调事件**。如果全部输出，你会得到：
- LLM 的完整 prompt（可能几千个 token）× 多次调用
- 每个工具的输入输出
- 每个中间步骤的序列化数据
- 大量重复信息

### 为什么人们容易这样错？

来自传统编程的习惯——"不确定哪里出了问题？那就到处加 `console.log`"。在前端调试中，这招通常管用，因为前端执行路径短、数据量小。但 Agent 不同：

- 一次 Agent 调用可能包含 **5-20 次 LLM 调用 + 多次工具调用**
- 每次 LLM 调用的 prompt 可能有 **几千个 token**
- 回调事件之间有复杂的父子嵌套关系

### 正确理解

**分层调试法：先粗后细，逐步缩小范围**。

```python
# ✅ 正确做法：分层调试

# 第1层：先开 verbose，看概要（执行了什么）
from langchain_core.globals import set_verbose
set_verbose(True)
# → 看到：Agent 调用了哪些工具、什么顺序

# 第2层：发现可疑步骤后，用 tags 精准追踪
result = agent.invoke(
    input,
    config={
        "tags": ["debug-tool-search"],  # 打标签过滤
        "metadata": {"debug_session": "issue-123"}
    }
)

# 第3层：用 LangSmith 深入分析单个可疑 Run
from langchain_core.tracers.context import collect_runs
with collect_runs() as cb:
    result = agent.invoke(input)
    # 只查看特定 run 的详细信息
    print(cb.traced_runs[0].inputs)
    print(cb.traced_runs[0].outputs)
```

**口诀**：「先看轮廓，再看细节。先 verbose，再 debug，最后 LangSmith」。

## 误区2：「缓存开了就万事大吉，能省很多钱」 ❌

### 为什么错？

LLM 缓存只在**完全相同的输入**才会命中。在实际 Agent 中，缓存命中率通常**低得惊人**：

```python
# ❌ 以为能命中缓存的场景（实际上不会！）

# 调用 1：用户问 "什么是 RAG？"
# 调用 2：用户问 "什么是 RAG?"  ← 全角问号 vs 半角，不命中！
# 调用 3：用户问 "什么是RAG？"  ← 少了空格，不命中！

# Agent 场景更严重：
# Agent 的 prompt 包含对话历史，每轮对话后 prompt 都不同
# 即使用户问同样的问题，上下文不同 → prompt 不同 → 不命中

# 示例：Agent 的实际 prompt 长这样
"""
System: 你是一个助手...
Human: 什么是RAG？           ← 第一次调用
"""

"""
System: 你是一个助手...
Human: 什么是RAG？           ← 第二次调用
AI: RAG是检索增强生成...
Human: 能详细说说吗？         ← 上下文变了，整个 prompt 不同！
"""
```

### 为什么人们容易这样错？

因为 HTTP 缓存给了错误的直觉。HTTP 缓存可以用 URL 做 key，URL 通常是稳定的。但 LLM 缓存的 key 是**完整的 prompt 文本**，而 Agent 的 prompt 几乎每次都不同。

### 正确理解

**缓存适合的场景**：
1. **开发测试**：反复调试同一个固定输入 → 命中率高
2. **批量处理**：处理标准化的输入列表 → 有重复则命中
3. **无上下文的单次调用**：FAQ 问答等 → 相同问题会命中

**缓存不适合的场景**：
1. **多轮对话 Agent**：上下文不断变化 → 几乎不命中
2. **带记忆的 Agent**：记忆导致 prompt 每次不同
3. **实时数据查询**：结果应该不同，缓存反而有害

```python
# ✅ 正确的成本优化策略（不要只依赖缓存）

# 策略1：模型路由 - 简单任务用便宜模型（效果最好）
from langchain_openai import ChatOpenAI
cheap = ChatOpenAI(model="gpt-4o-mini")    # $0.15/M tokens
expensive = ChatOpenAI(model="gpt-4o")     # $2.5/M tokens

# 策略2：精简 prompt - 减少不必要的上下文
# 策略3：限制工具调用次数 - max_iterations
# 策略4：缓存只在合适的场景开启
```

**口诀**：「缓存是锦上添花，不是救命稻草。模型路由才是成本优化第一招」。

## 误区3：「Agent 出错就是 LLM 的问题，换个更强的模型就好」 ❌

### 为什么错？

Agent 系统中，LLM 只是链条上的一环。大量"看起来像 LLM 问题"的 Bug，根因都不在 LLM：

```
Agent 执行失败的常见原因分布（社区经验）：
─────────────────────────────────
40%  工具定义不清晰 → LLM 不知道该调哪个工具
20%  Prompt 设计不当 → LLM 收到了错误/不足的指令
15%  工具本身有 Bug → 工具返回错误结果，LLM 据此做了错误推理
10%  输出解析失败   → LLM 输出格式不符合解析器预期
10%  上下文过长     → 超出 context window 被截断
 5%  LLM 能力不足   → 真正需要更强模型
```

```python
# ❌ 错误：Agent 调错工具了，换 GPT-4o 试试

# ✅ 正确：先检查工具定义
# 对比：
# 差的工具描述
tool_bad = Tool(
    name="search",
    description="搜索",  # 太模糊！LLM 不知道什么时候该用
    func=search_func
)

# 好的工具描述
tool_good = Tool(
    name="web_search",
    description="在互联网上搜索实时信息。当用户问到最新新闻、天气、"
                "股价等需要实时数据的问题时使用。输入应该是简洁的搜索关键词。"
                "不要用于已知的常识性问题。",
    func=search_func
)
```

### 为什么人们容易这样错？

因为 LLM 是 Agent 中最"神秘"的部分，也是最容易甩锅的对象。工具代码是你写的、Prompt 是你写的，但 LLM 是个黑盒——"一定是黑盒的问题"。这是一种**确认偏误**。

### 正确理解

**调试 Agent 的正确顺序**：

```
Agent 出错了
│
├── 1. 先看工具定义 ← 40% 的问题在这
│   └── description 是否清晰？
│   └── 参数类型是否正确？
│
├── 2. 再看 Prompt ← 20% 的问题在这
│   └── system prompt 是否有明确指令？
│   └── 是否给了足够的例子？
│
├── 3. 然后看工具实现 ← 15% 的问题在这
│   └── 工具是否正确执行？
│   └── 返回值格式是否合理？
│
├── 4. 检查输出解析 ← 10% 的问题在这
│   └── handle_parsing_errors=True 了吗？
│
├── 5. 检查上下文长度 ← 10% 的问题在这
│   └── 是否超出 context window？
│
└── 6. 最后才考虑换模型 ← 只有 5% 需要
```

```python
# ✅ 正确的排查流程示例

# 1. 先用 verbose 看 Agent 决策过程
from langchain_core.globals import set_verbose
set_verbose(True)

# 2. 检查工具是否被正确调用
from langchain_core.callbacks import BaseCallbackHandler

class ToolDebugHandler(BaseCallbackHandler):
    def on_agent_action(self, action, **kwargs):
        print(f"Agent 选择了工具: {action.tool}")
        print(f"传入参数: {action.tool_input}")
        # 如果 Agent 总是选错工具 → 改工具 description
        # 如果参数总是不对 → 改工具的参数说明

    def on_tool_end(self, output, **kwargs):
        print(f"工具返回: {output[:200]}")
        # 如果工具返回了错误数据 → 修工具实现

result = agent.invoke(input, config={"callbacks": [ToolDebugHandler()]})
```

**口诀**：「Agent 出 Bug，先查工具再查 Prompt，最后才怀疑模型」。

## 误区速查表

| 误区 | 正确理解 | 行动建议 |
|------|---------|---------|
| 日志越多越好 | 分层调试，逐步深入 | verbose → debug → LangSmith |
| 缓存能省很多钱 | Agent 场景缓存命中率低 | 优先用模型路由降成本 |
| 换模型就能修 Bug | 大多数问题在工具和 Prompt | 按排查顺序逐步检查 |
