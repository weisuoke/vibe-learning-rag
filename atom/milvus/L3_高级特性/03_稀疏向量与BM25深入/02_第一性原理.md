## 稀疏向量与 BM25 的第一性原理

### 1. 最基础的定义

**稀疏向量 = 只存储非零元素的向量表示**

仅此而已！没有更基础的了。

**数学定义：**
```
稀疏向量 V = {(position_i, value_i) | value_i ≠ 0, i ∈ [0, n)}
```

**示例：**
```python
# 稠密向量（10000 维）
dense_vector = [0, 0, 0.5, 0, 0, ..., 0.8, ..., 0, 0.3, ..., 0]
# 存储空间：10000 × 4 字节 = 40KB

# 稀疏向量（只存储 3 个非零元素）
sparse_vector = {
    2: 0.5,      # 位置 2，值 0.5
    1000: 0.8,   # 位置 1000，值 0.8
    5000: 0.3    # 位置 5000，值 0.3
}
# 存储空间：3 × 8 字节 = 24 字节
# 压缩比：40KB / 24B = 1666:1
```

**BM25 = 改进的 TF-IDF 算法，用于全文搜索**

**数学定义：**
```
BM25(q, d) = Σ IDF(qi) × (f(qi, d) × (k1 + 1)) / (f(qi, d) + k1 × (1 - b + b × |d| / avgdl))
```

**核心改进：**
1. **饱和函数**：高频词不会无限增长
2. **文档长度归一化**：长文档不会占优势

---

### 2. 为什么需要稀疏向量与 BM25？

**核心问题：如何高效地表示和检索文本信息？**

#### 问题1：稠密向量的存储和计算成本高

**场景：**
- 10 亿个文档
- 每个文档 1536 维稠密向量
- 存储空间：10 亿 × 1536 × 4 字节 = 6TB

**解决方案：**
- 使用稀疏向量，只存储非零元素
- 存储空间：10 亿 × 100 × 8 字节 = 800GB（压缩 87.5%）

#### 问题2：语义检索对关键词匹配不敏感

**场景：**
- 用户查询："Kubernetes 集群配置"
- 稠密向量检索：找到语义相关的文档，但可能不包含"Kubernetes"关键词
- 用户期望：精确匹配"Kubernetes"关键词

**解决方案：**
- 使用 BM25 进行关键词匹配
- 结合稠密向量（语义）和稀疏向量（关键词）
- 混合检索提升准确率 20-40%

#### 问题3：传统 TF-IDF 算法的局限性

**TF-IDF 的问题：**
1. **高频词问题**：词频线性增长，高频词权重过高
2. **文档长度问题**：长文档词频高，占优势

**BM25 的改进：**
1. **饱和函数**：`f(qi, d) × (k1 + 1) / (f(qi, d) + k1)`
   - 词频增长趋于饱和
   - 参数 k1 控制饱和速度（默认 1.2）

2. **文档长度归一化**：`1 - b + b × |d| / avgdl`
   - 参数 b 控制归一化程度（默认 0.75）
   - 长文档不会占优势

### 3. 稀疏向量与 BM25 的三层价值

#### 价值1：存储效率（工程价值）

**稀疏向量的存储优势：**

```python
# 示例：10000 维向量，只有 100 个非零元素（1% 密度）

# 稠密向量存储
dense_storage = 10000 * 4  # 40KB

# 稀疏向量存储（positions + values）
sparse_storage = 100 * 8   # 800 字节

# 压缩比
compression_ratio = dense_storage / sparse_storage  # 50:1
```

**实际应用：**
- 10 亿文档 × 10000 维 = 40TB（稠密）
- 10 亿文档 × 100 非零元素 = 800GB（稀疏）
- 节省 98% 存储空间

---

#### 价值2：计算效率（性能价值）

**稀疏向量的计算优势：**

```python
# 向量内积计算

# 稠密向量：O(n)
def dense_dot_product(v1, v2):
    return sum(v1[i] * v2[i] for i in range(len(v1)))
    # 时间复杂度：O(10000)

# 稀疏向量：O(k)，k 是非零元素数量
def sparse_dot_product(v1, v2):
    result = 0
    for pos, val in v1.items():
        if pos in v2:
            result += val * v2[pos]
    return result
    # 时间复杂度：O(100)
```

## 从第一性原理理解核心概念

### 概念1：稀疏向量数据结构

**第一性原理：**
- 向量 = 有序的数字列表
- 稀疏 = 大部分元素为零
- 稀疏向量 = 只存储非零元素的向量

**Milvus 实现：**
```python
# 数据结构（来源：reference/source_sparse_bm25_01.md）
positions = [2, 1000, 5000]      # uint32 数组
values = [0.5, 0.8, 0.3]         # float32 数组

# 序列化格式：8 字节/元素
# [position_0 (4B)] [value_0 (4B)] [position_1 (4B)] [value_1 (4B)] ...

# 维度计算
dim = max(positions) + 1  # 5001
```

**为什么这样设计？**
1. **内存效率**：只存储非零元素
2. **计算效率**：跳过零元素
3. **序列化效率**：固定 8 字节/元素

---

### 概念2：BM25 算法原理

**第一性原理：**
- 文本检索 = 找到包含查询词的文档
- 相关性 = 词频 × 逆文档频率
- BM25 = 改进的 TF-IDF（饱和 + 归一化）

**BM25 公式推导：**

```
1. TF-IDF（基础）
   Score = TF(qi, d) × IDF(qi)

2. 引入饱和函数（解决高频词问题）
   TF_saturated = f(qi, d) × (k1 + 1) / (f(qi, d) + k1)

3. 引入文档长度归一化（解决长文档问题）
   k = k1 × (1 - b + b × |d| / avgdl)

4. BM25（最终）
   BM25(q, d) = Σ IDF(qi) × (f(qi, d) × (k1 + 1)) / (f(qi, d) + k)
```

**参数含义：**
- `k1`：词频饱和参数（默认 1.2）
  - k1 越大，词频影响越大
  - k1 = 0：退化为布尔检索
  - k1 = ∞：退化为 TF-IDF

- `b`：文档长度归一化参数（默认 0.75）
  - b = 0：不考虑文档长度
  - b = 1：完全归一化
  - b = 0.75：平衡

**数据来源：** `reference/context7_pymilvus_01.md` - BM25 参数说明

### 概念3：WAND 算法优化

**第一性原理：**
- top-k 查询 = 找到分数最高的 k 个文档
- 暴力方法 = 计算所有文档的分数，排序取 top-k
- WAND = 通过上界估计跳过不可能进入 top-k 的文档

**WAND 算法原理：**

```
1. 维护上界信息
   - 每个词项的最大分数
   - 每个块的最大分数

2. 跳跃式遍历
   - 计算当前文档的上界
   - 如果上界 < 第 k 个文档的分数，跳过
   - 否则，计算精确分数

3. Block Max-WAND 优化
   - 将倒排列表分块
   - 维护每个块的最大分数
   - 跳过整个块
```

**性能提升：**
- Block Max-WAND 实现 4 倍加速
- 适用于大规模检索场景

**数据来源：** `reference/search_wand_algorithm_01.md` - WAND 算法详解

### 概念4：混合检索融合策略

**第一性原理：**
- 多个检索结果 = 多个排序列表
- 融合 = 将多个排序列表合并为一个
- 目标 = 最大化检索准确率

**融合策略：**

#### RRFRanker（Reciprocal Rank Fusion）

```python
# 公式
RRF(d) = Σ 1 / (k + rank_i(d))

# 参数
k = 60  # 默认值

# 示例
# 文档 A 在结果1中排名第1，在结果2中排名第3
RRF(A) = 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323

# 文档 B 在结果1中排名第2，在结果2中排名第2
RRF(B) = 1/(60+2) + 1/(60+2) = 0.0161 + 0.0161 = 0.0322

# 文档 A 排名更高
```

**优点：**
- 无需手动调整权重
- 鲁棒性好
- 适合快速上手

#### WeightedRanker（加权融合）

```python
# 公式
Score = w1 × score1 + w2 × score2

# 参数
weights = [0.7, 0.3]  # 70% 稠密向量，30% 稀疏向量

# 示例
# 文档 A
score_A = 0.7 × 0.9 + 0.3 × 0.6 = 0.63 + 0.18 = 0.81

# 文档 B
score_B = 0.7 × 0.8 + 0.3 × 0.8 = 0.56 + 0.24 = 0.80

# 文档 A 排名更高
```

**优点：**
- 精确控制权重
- 适合明确知道各字段重要性的场景

**数据来源：** `reference/search_hybrid_weight_01.md` - 混合检索权重调优

## 实战代码：从第一性原理实现混合检索

```python
"""
从第一性原理实现 Milvus 2.6 混合检索
演示：稀疏向量 + BM25 + 混合检索
"""

from pymilvus import connections, FieldSchema, CollectionSchema, DataType, Collection
from pymilvus import Function, FunctionType, AnnSearchRequest, RRFRanker

# ===== 1. 连接 Milvus 2.6 =====
connections.connect(host="localhost", port="19530")

# ===== 2. 定义 Schema（第一性原理：数据结构） =====
fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=65535),
    # 稠密向量：语义理解
    FieldSchema(name="dense_vector", dtype=DataType.FLOAT_VECTOR, dim=768),
    # 稀疏向量：关键词匹配（BM25 自动生成）
    FieldSchema(name="sparse_vector", dtype=DataType.SPARSE_FLOAT_VECTOR)
]

# ===== 3. 创建 BM25 Function（第一性原理：自动生成稀疏向量） =====
bm25_function = Function(
    name="bm25_fn",
    input_field_names=["content"],
    output_field_names="sparse_vector",
    function_type=FunctionType.BM25
)

schema = CollectionSchema(fields=fields, functions=[bm25_function])
collection = Collection(name="hybrid_search_demo", schema=schema)

# ===== 4. 创建索引（第一性原理：加速检索） =====
# 稠密向量索引
collection.create_index(
    field_name="dense_vector",
    index_params={
        "index_type": "HNSW",
        "metric_type": "L2",
        "params": {"M": 16, "efConstruction": 200}
    }
)

# 稀疏向量索引（WAND 算法）
collection.create_index(
    field_name="sparse_vector",
    index_params={
        "index_type": "SPARSE_INVERTED_INDEX",  # 或 SPARSE_WAND
        "metric_type": "BM25",
        "params": {"bm25_k1": 1.2, "bm25_b": 0.75}
    }
)

# ===== 5. 插入数据（第一性原理：自动生成稀疏向量） =====
import numpy as np

data = [
    {
        "content": "如何配置 Kubernetes 集群？",
        "dense_vector": np.random.rand(768).tolist()
    },
    {
        "content": "Docker 容器化部署指南",
        "dense_vector": np.random.rand(768).tolist()
    },
    {
        "content": "Kubernetes 网络配置详解",
        "dense_vector": np.random.rand(768).tolist()
    }
]

collection.insert(data)
collection.load()

# ===== 6. 混合检索（第一性原理：融合多个检索结果） =====
query_text = "Kubernetes 集群配置"
query_dense = np.random.rand(768).tolist()

# 稠密向量检索请求
dense_req = AnnSearchRequest(
    data=[query_dense],
    anns_field="dense_vector",
    param={"metric_type": "L2", "params": {"ef": 100}},
    limit=10
)

# 稀疏向量检索请求（BM25）
sparse_req = AnnSearchRequest(
    data=[query_text],
    anns_field="sparse_vector",
    param={"metric_type": "BM25"},
    limit=10
)

# 混合检索（RRFRanker 融合）
results = collection.hybrid_search(
    reqs=[dense_req, sparse_req],
    ranker=RRFRanker(k=60),
    limit=5,
    output_fields=["content"]
)

print("混合检索结果：")
for i, hit in enumerate(results[0]):
    print(f"{i+1}. {hit.entity.get('content')} (score: {hit.score})")
```

**运行输出示例：**
```
混合检索结果：
1. Kubernetes 网络配置详解 (score: 0.0323)
2. 如何配置 Kubernetes 集群？ (score: 0.0315)
3. Docker 容器化部署指南 (score: 0.0102)
```

## 关键洞察

### 洞察1：稀疏向量不是"压缩"，而是"选择性存储"

**常见误解：**
- 稀疏向量是对稠密向量的压缩

**第一性原理：**
- 稀疏向量是只存储非零元素的向量表示
- 不是"压缩"，而是"选择性存储"
- 本质上是不同的数据结构

---

### 洞察2：BM25 不是"更好的 TF-IDF"，而是"解决 TF-IDF 问题的算法"

**常见误解：**
- BM25 是 TF-IDF 的改进版

**第一性原理：**
- TF-IDF 有两个根本问题：高频词问题、文档长度问题
- BM25 通过饱和函数和归一化解决这两个问题
- 不是"改进"，而是"解决问题"

---

### 洞察3：混合检索不是"可选项"，而是"必需品"

**常见误解：**
- 混合检索是可选的优化

**第一性原理：**
- 稠密向量和稀疏向量各有优势
- 单一检索方式无法满足所有场景
- 混合检索是 2026 年 RAG 系统的标准配置
