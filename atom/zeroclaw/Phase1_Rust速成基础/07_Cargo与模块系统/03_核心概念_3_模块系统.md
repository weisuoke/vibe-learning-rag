# Cargo 与模块系统 - 核心概念 3：模块系统

> **知识点**: Cargo 与模块系统
> **层级**: Phase1_Rust速成基础
> **维度**: 核心概念（mod 声明、use 导入、pub 可见性控制与模块组织）
> **目标受众**: 有 TypeScript/前端经验但零 Rust 经验的开发者

---

## 一句话定义

> **模块系统通过 `mod` 声明、`use` 导入、`pub` 可见性控制，将代码组织成层级命名空间——Rust 版的 ES Modules，但默认私有、显式声明。**

---

## 一、mod 声明：定义模块

Rust 的模块不会被自动发现——**每个模块必须用 `mod` 显式声明**。这是从 TypeScript 转过来最先踩的坑 [^search1]。

### 1.1 TypeScript 自动发现 vs Rust 显式声明

```typescript
// TypeScript：文件存在就能 import，无需"声明"
import { formatDate } from './utils';  // src/utils.ts 存在 → 直接 import
```

```rust
// Rust：文件存在还不够，必须在父模块中 mod 声明！
// src/lib.rs
mod utils;  // ← 没有这行，utils.rs 即使存在也不会被编译！
use crate::utils::format_date;
```

**这是 Rust 初学者最容易困惑的地方！** 创建了 `src/utils.rs`，编译器却说找不到——原因就是忘了 `mod utils;` [^search1]。

### 1.2 四种 mod 声明方式

**方式 1：内联模块**（代码少于 50 行时）

```rust
mod config {
    pub struct AppConfig { pub name: String, pub port: u16 }
}
```

**方式 2：文件模块**（最常用——一个模块对应一个文件）

```rust
// src/lib.rs
mod config;    // ← 编译器去找 src/config.rs

// src/config.rs
pub struct AppConfig { pub name: String, pub port: u16 }
```

**方式 3：目录模块·旧风格**（mod foo; → foo/mod.rs）

```rust
// src/lib.rs
mod tools;            // ← 找 src/tools/mod.rs
// src/tools/mod.rs   ← 旧风格入口
pub mod shell;        // ← 找 src/tools/shell.rs
pub mod fetch;
```

**方式 4：目录模块·新风格**（foo.rs + foo/ 目录共存，社区推荐）

```rust
// src/lib.rs
mod tools;            // ← 找 src/tools.rs
// src/tools.rs       ← 新风格：文件名 = 模块名
pub mod shell;        // ← 找 src/tools/shell.rs
pub mod fetch;
```

**社区推荐新风格**——编辑器标签页不会全是 `mod.rs`，一眼能看出文件属于哪个模块 [^search1]。

### 1.3 映射规则与 crate root

```
声明方式                     编译器查找路径                    TypeScript 类比
─────────────────────────────────────────────────────────────────────────────
mod foo { ... }              （内联，无文件）                   namespace { }
mod foo;（在 lib.rs）        src/foo.rs 或 src/foo/mod.rs      import './foo'
mod bar;（在 foo.rs）        src/foo/bar.rs                    import './foo/bar'
```

**⚠️ `foo.rs` 和 `foo/mod.rs` 不能同时存在**——编译器会报错 [^context7_module1]。

`lib.rs` 是库的根模块（crate root），`main.rs` 是二进制入口。项目可以**同时有两者**——ZeroClaw 就是这样：`lib.rs` 定义所有模块，`main.rs` 调用它们启动应用 [^source1]。

```rust
// src/main.rs — 引用 lib.rs 的公共 API
use zeroclaw::{Agent, Config};

fn main() {
    let config = Config::load();
    Agent::new(config).run();
}
```

## 二、use 导入：引用其他模块的项

`use` 是 Rust 的 `import`——把其他模块的项带入当前作用域。但它**只是起别名**，模块的加载在 `mod` 声明时就完成了 [^context7_module1]。

### 2.1 三种路径前缀

```rust
use crate::config::AppConfig;   // 绝对路径（推荐）≈ import from '@/config'
use super::AppConfig;            // 父模块        ≈ import from '..'
use self::helper::format;        // 当前模块      ≈ import from './helper'
```

### 2.2 各种 use 语法

```rust
use std::collections::HashMap;           // 基本导入
use std::{io, fs, path::Path};           // 嵌套导入（一行导入多个）
use std::io::*;                          // 通配符（⚠️ 不推荐，污染命名空间）
use std::io::Result as IoResult;         // 别名（解决命名冲突）
use std::io::{self, Read, Write};        // 同时导入模块本身和其中的项
```

### 2.3 TypeScript 对照

```typescript
// TypeScript                           // Rust
import { useState } from 'react';       use react::use_state;
import * as React from 'react';         use react;
import { Result as IoRes } from './io'; use io::Result as IoRes;
import type { Config } from './config'; // Rust 不区分——use 统一导入类型和值
```

## 三、pub 可见性：访问控制

**Rust 默认一切私有**——这和 TypeScript 完全相反 [^context7_module1] [^search1]。

### 3.1 五种可见性级别

```rust
mod outer {
    fn private_fn() { }                           // 1. 默认私有
    pub fn fully_public() { }                     // 2. 完全公开
    pub(crate) fn crate_internal() { }            // 3. crate 内公开（最常用！）
    pub(super) fn parent_only() { }               // 4. 父模块公开
    pub(in crate::outer) fn specific_path() { }   // 5. 指定路径公开
}
```

### 3.2 对照表

```
Rust 语法              可见范围                TypeScript 大致等价
──────────────────────────────────────────────────────────────────────
（无修饰符）            当前模块 + 子模块       不 export 的函数
pub                    所有人                  export
pub(crate)             当前 crate 内           不 export（包级别私有）
pub(super)             父模块                  TypeScript 无精确等价
pub(in path)           指定路径                TypeScript 无等价
```

**`pub(crate)` 是最实用的中间态**——ZeroClaw 大量使用它来隐藏实现细节 [^source1]。

### 3.3 struct 字段的独立可见性

**结构体公开 ≠ 字段公开**——这是 Rust 独特设计 [^context7_module1]：

```rust
pub struct User {
    pub name: String,          // 公开字段
    pub(crate) email: String,  // crate 内可见
    password_hash: String,     // 私有！外部无法直接访问或构造
}

impl User {
    pub fn new(name: String, email: String, password: &str) -> Self {
        User { name, email, password_hash: hash(password) }
    }
}
```

有私有字段的 struct **不能被外部直接用字面量构造**，必须通过构造函数。这保护了内部实现自由度。

### 3.4 子模块可访问父模块的私有成员

```rust
mod outer {
    fn secret() { println!("secret!"); }  // 私有

    mod inner {
        pub fn reveal() {
            super::secret();  // ✅ 子模块可以访问父模块的私有成员！
        }
    }
}
// outer::secret();        // ❌ 外部不能访问
// outer::inner::reveal(); // ❌ inner 模块本身也是私有的
```

TS 中文件之间不存在"子模块可以看父模块私有成员"的概念。

## 四、pub use（Re-export）：清洁公共 API

`pub use` 把内部项重新导出到更短的路径——和 TypeScript 的 barrel file 模式完全一样 [^context7_module1]：

```typescript
// TypeScript barrel file                // Rust re-export
// src/index.ts                          // src/lib.rs
export { Disk } from './storage/disk';   pub use storage::disk::Disk;
export { Net } from './storage/net';     pub use storage::net::Net;
// 用户：import { Disk } from 'pkg'     // 用户：use pkg::{Disk, Net};
```

### 4.1 为什么需要 Re-export

```
不使用 re-export                              使用 re-export
──────────────────────                        ──────────────────────
use zeroclaw::tools::shell::ShellTool;        use zeroclaw::ShellTool;
use zeroclaw::providers::openai::OpenAI;      use zeroclaw::OpenAI;
                                              ↑ 隐藏内部结构，API 更干净
```

**好处**：隐藏内部结构（自由重构）、缩短导入路径、明确公共 API 边界。

### 4.2 ZeroClaw 的 Re-export 模式

```rust
// src/lib.rs [^source1]
pub mod agent;
pub mod tools;
pub mod channels;
pub mod providers;
pub mod config;

// Re-export 核心类型到顶层
pub use config::Config;
pub use agent::{Agent, AgentBuilder};
pub use tools::Tool;
pub use channels::Channel;
pub use providers::Provider;
```

用户既可以 `use zeroclaw::Agent;`（短路径），也可以 `use zeroclaw::agent::Agent;`（完整路径）。

## 五、模块与文件系统的映射规则

### 5.1 完整映射表

```
代码声明                        文件系统位置                       说明
──────────────────────────────────────────────────────────────────────────
crate（根模块）                 src/lib.rs 或 src/main.rs          crate root
mod foo;（在 lib.rs 中）        src/foo.rs 或 src/foo/mod.rs       顶层模块
mod bar;（在 foo.rs 中）        src/foo/bar.rs                     嵌套模块
mod inline { ... }              （内联，无文件）                    内联模块
```

**所有 `mod` 声明都从 crate root 开始**——如果一个模块没有从根出发的 `mod` 链条连接到它，编译器就不会编译它 [^context7_module1]。

### 5.2 ZeroClaw src/lib.rs 中 30+ 模块的组织

```rust
// src/lib.rs — 每个功能域一个顶层模块 [^source1]
pub mod agent;          // 代理循环         pub mod memory;         // 存储后端
pub mod channels;       // 多通道消息       pub mod observability;  // 可观测性
pub mod config;         // 配置管理         pub mod providers;      // LLM 提供商
pub mod gateway;        // HTTP API        pub mod rag;            // RAG 管道
pub mod hardware;       // 硬件发现         pub mod tools;          // 40+ 工具
// ... 还有 20+ 其他模块
```

### 5.3 最佳实践：保持浅层级

社区共识——**模块嵌套不超过 2-3 层**。太深应拆成独立 crate [^search1]：

```
✅ 推荐：浅层级                    ❌ 避免：过深嵌套
src/                               src/
├── lib.rs                         ├── lib.rs
├── tools.rs          (1 层)       ├── tools/
├── tools/                         │   ├── io/
│   ├── shell.rs      (2 层)       │   │   ├── file/
│   └── fetch.rs                   │   │   │   └── csv.rs  (4 层！)
```

ZeroClaw 把 `robot-kit` 拆成了 `crates/robot-kit/` 独立 crate，获得更清晰的边界和并行编译 [^source1]。

## 六、ZeroClaw 的模块组织实例

### 6.1 tools/mod.rs — Trait + 工厂函数 + feature 门控

```rust
// src/tools/mod.rs [^source1]
mod shell;                              // 私有子模块
mod fetch;
mod traits;

#[cfg(feature = "hardware")]            // feature 门控
mod hardware_scan;

pub use traits::Tool;                   // re-export trait
pub use shell::ShellTool;
pub use fetch::FetchTool;

pub fn default_tools() -> Vec<Box<dyn Tool>> {
    vec![Box::new(ShellTool::new()), Box::new(FetchTool::new())]
}

pub fn all_tools() -> Vec<Box<dyn Tool>> {
    let mut tools = default_tools();
    #[cfg(feature = "hardware")]
    tools.push(Box::new(hardware_scan::HardwareScanTool::new()));
    tools
}
```

TypeScript 等价就是 `index.ts` barrel file + 工厂函数。

### 6.2 channels/mod.rs — pub mod vs mod 的精确控制

```rust
// src/channels/mod.rs [^source1]
mod traits;                   // ← 私有模块
pub use traits::Channel;      // ← 但 re-export 了 trait

pub mod telegram;             // ← 公开：外部可用 zeroclaw::channels::telegram::*
pub mod discord;
pub mod cli;

#[cfg(feature = "channel-matrix")]
pub mod matrix;               // ← 条件编译的公开模块
```

**`mod traits;` + `pub use traits::Channel;`** = 隐藏模块，暴露类型。

### 6.3 providers/mod.rs — 工厂模式封装

```rust
// src/providers/mod.rs [^source1]
mod traits;
pub use traits::Provider;

mod openai;                   // ← 全部私有！外部看不到具体实现
mod anthropic;
mod ollama;

mod reliable;
pub use reliable::ReliableProvider;

pub fn create_provider(name: &str) -> Box<dyn Provider> {
    match name {
        "openai"    => Box::new(openai::OpenAIProvider::new()),
        "anthropic" => Box::new(anthropic::AnthropicProvider::new()),
        "ollama"    => Box::new(ollama::OllamaProvider::new()),
        _ => panic!("Unknown provider: {}", name),
    }
}
```

**精髓**：内部模块全部 `mod`（私有），外部只看到 `Box<dyn Provider>`。内部重构不影响外部 API。

## 七、Rust 与 TypeScript 模块系统全景对比

```
维度              TypeScript/ES Modules              Rust Modules
──────────────────────────────────────────────────────────────────────────────
默认可见性         模块内公开，export 对外             默认私有，pub 对外
模块发现           自动（文件存在即可 import）         显式（必须 mod 声明）
导入语法           import { x } from './path'         use crate::path::x;
导出语法           export function x() {}             pub fn x() {}
Re-export          export { x } from './mod'          pub use mod::x;
Barrel file        index.ts                           mod.rs 或同名 .rs
别名              import { x as y }                   use x as y;
类型导入           import type { T }                   不区分（use 统一导入）
循环依赖           运行时才报错                        编译时禁止循环 mod
```

## 八、常见陷阱

### 陷阱 1：创建文件后忘了 mod 声明

```rust
// src/lib.rs 中没有 mod utils; → 编译器：unresolved import
// 解决：在父模块中加 mod utils;
```

### 陷阱 2：混淆 mod 和 use

```rust
mod foo;      // 声明/加载模块（让编译器编译 foo.rs）
use foo::bar; // 导入项（创建快捷方式）
// 两者功能不同，缺一不可！
```

### 陷阱 3：pub struct 的私有字段

```rust
pub struct Config { name: String }  // 字段是私有的！
let c = Config { name: "test".into() }; // ❌ 不能直接构造
// 解决：pub name: String 或提供 pub fn new()
```

### 陷阱 4：foo.rs 和 foo/mod.rs 不能共存

编译器会报 `file found for module at both paths` 错误。选一种风格，推荐新风格 [^search1]。

## 总结

```
Rust 模块系统核心要点：

  mod 声明（定义模块）：
    mod foo { ... }          → 内联模块
    mod foo;                 → 文件模块（foo.rs 或 foo/mod.rs）
    ⚠️ 文件存在不等于模块存在——必须 mod 声明！

  use 导入（引用项）：
    use crate::foo::Bar;     → 绝对路径（推荐）
    use super::Bar;          → 父模块
    use std::{io, fs};       → 嵌套导入
    use io::Result as Res;   → 别名

  pub 可见性（访问控制）：
    （默认）                  → 私有（仅模块内 + 子模块）
    pub                      → 完全公开
    pub(crate)               → crate 内公开（最常用的中间态）
    ⚠️ struct 字段有独立可见性——pub struct ≠ 字段公开！

  pub use re-export（清洁 API）：
    pub use internal::Type;  → 缩短导入路径，隐藏内部结构
    ≈ TypeScript: export { Type } from './internal'

  与 TypeScript 最大区别：
    ✅ 默认私有（TS 默认模块内公开）
    ✅ 显式 mod 声明（TS 文件自动发现）
    ✅ struct 字段独立可见性
    ✅ 子模块可看父模块私有成员
```
