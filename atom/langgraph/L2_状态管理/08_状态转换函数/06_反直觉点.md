## 误区1："节点函数必须返回完整状态" ❌

### 错误观点

"节点函数接收完整 State，所以也应该返回完整 State。不返回的字段会丢失或变成 None。"

```python
from typing import Annotated, TypedDict
from operator import add

class State(TypedDict):
    query: str
    answer: str
    items: Annotated[list, add]

# ❌ 错误写法：返回完整状态
def bad_node(state: State) -> dict:
    result = process(state["query"])
    return {
        "query": state["query"],   # 多余！未变化不需要返回
        "answer": result,          # 这个才是真正需要更新的
        "items": state["items"],   # 危险！items 有 add reducer，会重复追加！
    }
```

### 为什么错？

**核心事实：** LangGraph 节点只需返回要更新的字段（`Partial<State>`），未返回的字段保持不变。

返回完整状态会引发两个严重问题：

**问题1：有 Reducer 的字段会被重复累加**

```python
# 假设当前 state["items"] = ["a", "b"]

# ❌ 返回完整状态
def bad_node(state: State) -> dict:
    return {"items": state["items"]}
    # add(["a", "b"], ["a", "b"]) = ["a", "b", "a", "b"]  ← 重复了！

# ✅ 只返回增量
def good_node(state: State) -> dict:
    return {"items": ["c"]}
    # add(["a", "b"], ["c"]) = ["a", "b", "c"]  ← 正确追加
```

**问题2：无 Reducer 的字段在并行节点中会冲突**

```python
class State(TypedDict):
    count: int  # 无 reducer → LastValue Channel

# 两个并行节点都返回 {"count": ...}
# → InvalidUpdateError: Can receive only one value per step
```

**源码层面：** 在 `channels/last_value.py` 中，当节点不返回某个字段时，该 Channel 的 `update()` 不会被调用，值自然保持不变——**不写入 = 不变化**。

[来源: reference/source_状态转换函数_01.md | channels/last_value.py]

### 为什么人们容易这样错？

- **Redux reducer 经验**：Redux 要求返回完整 state，否则字段变 `undefined`
- **函数式编程习惯**：纯函数"输入完整、输出完整"的思维定式
- **安全感驱动**：担心不返回的字段会"消失"

```
Redux reducer                    LangGraph 节点
─────────────                    ──────────────
必须返回完整 state               只返回变化的字段
不返回 = 字段丢失                不返回 = 字段不变
return { ...state, key: val }    return { "key": val }
```

### 正确理解

```python
# ✅ 正确：只返回要更新的字段
def retrieve_node(state: State) -> dict:
    docs = search(state["query"])
    return {"context": docs}       # query、answer 保持不变

# ✅ 不需要更新时
def passthrough_node(state: State) -> dict:
    log(state["query"])
    return {}                      # 空字典 = 不更新任何字段
```

| 场景 | 应该返回什么 |
|------|-------------|
| 更新部分字段 | `{"field": value}` |
| 不需要更新 | `{}` 或 `None` |
| 有 reducer 的字段 | 只返回增量（新增部分） |
| 无 reducer 的字段 | 返回完整新值（覆盖旧值） |

**记忆口诀：** "只说变了什么，没变的别提——Partial 就是部分更新"

## 误区2："Command 的 goto 会替代静态边" ❌

### 错误观点

"如果节点 A 有静态边 `add_edge('A', 'B')`，同时返回 `Command(goto='C')`，那么只会执行 C，不会执行 B。"

```python
graph.add_edge("A", "B")  # 静态边

def node_a(state: State) -> Command[Literal["C"]]:
    return Command(update={"x": 1}, goto="C")

# ❌ 错误预期：只有 C 会执行
# ✅ 实际结果：B 和 C 都会执行！
```

### 为什么错？

**核心事实：** Command 的 `goto` 是**添加**额外的边，不是**替换**已有的边。静态边和 Command 是叠加关系。

**实际验证：**

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import Command
from typing import Literal, Annotated
from typing_extensions import TypedDict
import operator

class State(TypedDict):
    x: int
    log: Annotated[list[str], operator.add]

def node_a(state: State) -> Command[Literal["C"]]:
    return Command(update={"x": 1, "log": ["A executed"]}, goto="C")

def node_b(state: State) -> dict:
    return {"log": ["B executed"]}

def node_c(state: State) -> dict:
    return {"log": ["C executed"]}

builder = StateGraph(State)
builder.add_node("A", node_a)
builder.add_node("B", node_b)
builder.add_node("C", node_c)
builder.add_edge(START, "A")
builder.add_edge("A", "B")  # 静态边
builder.add_edge("B", END)
builder.add_edge("C", END)

graph = builder.compile()
result = graph.invoke({"x": 0, "log": []})
print(result["log"])
# ['A executed', 'B executed', 'C executed']  ← B 和 C 都执行了！
```

**源码层面：** 在 `pregel/_algo.py` 中，下一步节点从静态边和 Command 两个来源**合并**收集，是 `set.add()` 的关系——合并，不是覆盖。

[来源: reference/source_状态转换函数_01.md | pregel/_algo.py]

### 为什么人们容易这样错？

- **if/else 思维**：选了一条路就不走另一条
- **"goto" 的语义暗示**：让人觉得是唯一目的地
- **条件边经验**：`add_conditional_edges` 确实是互斥选择，容易混淆

```
条件边（add_conditional_edges）     Command 的 goto
──────────────────────────         ─────────────────
选择一条路走（互斥）                添加一条额外的路（叠加）
替代默认路由                        补充默认路由
```

### 正确理解

**需要互斥路由时，不要混用静态边和 Command。**

```python
# ✅ 方案1：全用 Command（推荐）
# 不添加 A → B 的静态边，完全由 Command 控制
def node_a(state: State) -> Command[Literal["B", "C"]]:
    if state["x"] > 0:
        return Command(update={"log": ["go to C"]}, goto="C")
    else:
        return Command(update={"log": ["go to B"]}, goto="B")

# ✅ 方案2：全用条件边
def route_from_a(state: State) -> str:
    return "C" if state["x"] > 0 else "B"

builder.add_conditional_edges("A", route_from_a, ["B", "C"])
```

**记忆口诀：** "Command 是加法不是替换——goto 添加路，不删旧路"

## 误区3："数据清洗应该在图外面做" ❌

### 错误观点

"数据清洗是预处理步骤，应该在 `graph.invoke()` 之前完成。图内部只放核心业务逻辑。"

```python
# ❌ 错误做法：在图外面清洗
def handle_request(user_input: str):
    cleaned = user_input.strip().lower()
    cleaned = re.sub(r'[^\w\s]', '', cleaned)
    result = graph.invoke({"query": cleaned})
    return result
```

### 为什么错？

**核心事实：** 数据清洗是状态转换的核心部分，应该作为图中的节点。图外清洗有 4 个致命缺陷：

| 缺陷 | 说明 |
|------|------|
| 不可追踪 | 清洗过程不被 checkpoint 记录，无法回溯"原始输入 → 清洗后" |
| 不可重试 | 清洗失败 → 整个请求失败；图内清洗失败 → 只重试清洗节点 |
| 不可替换 | 清洗逻辑和调用代码耦合，换策略需要改调用方 |
| 不可测试 | 清洗逻辑散落在业务代码中，难以单独单元测试 |

```python
# ✅ 图内清洗 → 每一步都有 checkpoint
# get_state_history 可以看到：
# Step 0: query = "  什么是 RAG？？？  "  (原始输入)
# Step 1: cleaned_query = "什么是rag"     (清洗后)
# Step 2: context = [...]                  (检索结果)
```

[来源: reference/search_状态转换函数_01.md | 社区最佳实践]

### 为什么人们容易这样错？

- **ETL 经验**：清洗是"进系统之前"的事
- **Web 开发习惯**：表单验证在前端做，后端只处理"干净数据"
- **低估 LLM 输出的不确定性**：传统系统输入可控，但 LLM 输出格式不稳定

```
传统系统                          LLM/RAG 系统
─────────                        ─────────────
输入可控（表单、API）              输入不可控（自然语言）
清洗一次就够                      每个节点的输出都可能需要清洗
```

### 正确理解

**清洗就是转换，转换就是节点。放进图里，享受框架的全部能力。**

```python
from langgraph.graph import StateGraph, START, END
import re

# ✅ 清洗作为独立节点
def clean_input(state: RAGState) -> dict:
    """输入清洗节点：标准化用户查询"""
    raw = state["query"]
    cleaned = " ".join(raw.split())
    cleaned = re.sub(r'[？！。，]+$', '', cleaned).strip().lower()
    if not cleaned:
        cleaned = "请提供一个有效的问题"
    return {"cleaned_query": cleaned}

def retrieve(state: RAGState) -> dict:
    docs = vector_search(state["cleaned_query"], top_k=5)
    return {"context": docs}

def generate(state: RAGState) -> dict:
    answer = llm_generate(state["cleaned_query"], state["context"])
    return {"raw_answer": answer}

# ✅ 输出格式化也作为独立节点
def format_output(state: RAGState) -> dict:
    raw = state["raw_answer"].strip()
    if len(raw) > 2000:
        raw = raw[:2000] + "..."
    return {"formatted_answer": raw}

# 构建图：清洗 → 检索 → 生成 → 格式化
builder = StateGraph(RAGState)
builder.add_node("clean_input", clean_input)
builder.add_node("retrieve", retrieve)
builder.add_node("generate", generate)
builder.add_node("format_output", format_output)
builder.add_edge(START, "clean_input")
builder.add_edge("clean_input", "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", "format_output")
builder.add_edge("format_output", END)
```

**记忆口诀：** "清洗也是转换，转换就是节点——放进图里，全程可控"

[来源: reference/source_状态转换函数_01.md | sourcecode/langgraph]

## 额外提醒：容易忽略的细节

### 细节1：返回空 dict `{}` 和返回 `None` 效果相同

```python
def node_a(state: State) -> dict:
    return {}      # 不更新任何字段

def node_b(state: State) -> None:
    return None    # 效果完全一样

def node_c(state: State):
    pass           # 隐式返回 None，也一样
```

三种写法都被视为"无更新"。推荐 `return {}` 更明确。

### 细节2：有 reducer 的字段返回增量，无 reducer 的字段返回完整值

这是最常见的 bug 来源：

```python
class State(TypedDict):
    name: str                              # 无 reducer → LastValue
    items: Annotated[list[str], add]       # 有 reducer → BinaryOperatorAggregate

def my_node(state: State) -> dict:
    return {
        "name": "新名字",          # 无 reducer：覆盖旧值
        "items": ["新元素"],       # 有 reducer：追加到旧列表
    }
    # name: "旧名字" → "新名字"
    # items: ["旧元素"] → ["旧元素", "新元素"]
```

### 细节3：异步节点和同步节点的返回值规则完全相同

```python
def sync_node(state: State) -> dict:
    return {"result": do_something(state["query"])}

async def async_node(state: State) -> dict:
    return {"result": await do_something_async(state["query"])}
```

`async def` vs `def` 只影响执行方式，不影响返回值处理。

## 误区总结表

| 误区 | 错误观点 | 正确理解 | 记忆口诀 |
|------|----------|----------|----------|
| 误区1 | 节点必须返回完整状态 | 只返回变化的字段（Partial） | "只说变了什么，没变的别提" |
| 误区2 | Command goto 会替代静态边 | goto 是添加额外的边，不是替换 | "Command 是加法不是替换" |
| 误区3 | 数据清洗应该在图外面做 | 清洗是节点，放进图里享受全部能力 | "清洗也是转换，转换就是节点" |

---

## 避免误区的检查清单

- [ ] 节点返回值只包含需要更新的字段，不包含未变化的字段
- [ ] 有 reducer 的字段只返回增量，不返回 `state["field"]`（会导致重复）
- [ ] 使用 Command 时，确认是否有静态边会同时生效
- [ ] 需要互斥路由时，不混用静态边和 Command，选择其一
- [ ] 数据清洗逻辑放在图内作为独立节点，而非图外预处理
- [ ] 返回 `{}` 或 `None` 表示不更新，不要返回 `state` 本身
