## 10个2分钟知识卡片

---

### 卡片1：工作流模式是什么？

**一句话：** 工作流模式是"节点+边"的不同连接方式，解决不同类型的复杂任务。

**举例：**
```
顺序：A → B → C（流水线）
并行：A → [B, C] → D（同时做）
路由：A → B 或 C（选一条路）
```

**应用：** 就像乐高积木的拼法——同样的积木（节点），不同的拼法（模式）造出不同的东西。LangGraph 的六大模式覆盖了 95% 的工作流需求。

---

### 卡片2：顺序链式——最基础的模式

**一句话：** 步骤按顺序执行，前一步输出是后一步输入。

**举例：**
```python
# 三种写法，效果一样
# 写法1：逐条加边
builder.add_edge(START, "step1")
builder.add_edge("step1", "step2")
builder.add_edge("step2", END)

# 写法2：快捷方式
builder.add_sequence([step1, step2])

# 写法3：Functional API
@entrypoint()
def chain(x):
    a = step1(x).result()
    return step2(a).result()
```

**应用：** RAG 管道（检索→重排→生成）、ETL 流程、文档处理链。80% 的场景从这里开始。

---

### 卡片3：路由——智能分拣员

**一句话：** 根据输入特征，动态选择走哪条路。

**举例：**
```python
def route(state):
    if state["type"] == "qa":
        return "qa_handler"
    return "chat_handler"

builder.add_conditional_edges("router", route, {
    "qa_handler": "qa_handler",
    "chat_handler": "chat_handler",
})
```

**应用：** 客服分流、查询分类、模型选择。能用规则路由就不用 LLM 路由——更快更可靠。

---

### 卡片4：并行——空间换时间

**一句话：** 多个独立任务同时执行，用 reducer 合并结果。

**举例：**
```python
class State(TypedDict):
    results: Annotated[list, operator.add]  # 关键：reducer

# Fan-out
builder.add_edge(START, "task_a")
builder.add_edge(START, "task_b")
# Fan-in
builder.add_edge("task_a", "merge")
builder.add_edge("task_b", "merge")
```

**应用：** 多源检索、多维分析、多模型投票。记住：没有 reducer，并行结果会互相覆盖。

---

### 卡片5：Map-Reduce——动态并行

**一句话：** 用 Send API 在运行时创建任意数量的并行任务。

**举例：**
```python
from langgraph.types import Send

def fan_out(state):
    # 数据有多少条，就创建多少个并行任务
    return [Send("worker", {"item": i}) for i in state["items"]]

builder.add_conditional_edges("splitter", fan_out, ["worker"])
```

**应用：** 批量文档处理、多主题内容生成、批量翻译。和普通并行的区别：分支数量是数据决定的，不是写死的。

---

### 卡片6：编排者-工作者——智能项目经理

**一句话：** LLM 动态分解任务，分配给多个工作者并行执行。

**举例：**
```python
@task
def orchestrator(topic):
    # LLM 决定需要哪些子任务
    return planner.invoke(f"规划{topic}的章节")

@entrypoint()
def workflow(topic):
    subtasks = orchestrator(topic).result()
    futures = [worker(t) for t in subtasks]
    return synthesizer([f.result() for f in futures]).result()
```

**应用：** 报告生成、代码审查、复杂问题分解。本质是"智能版 Map-Reduce"——Splitter 由 LLM 驱动。

---

### 卡片7：评估者-优化者——自我改进循环

**一句话：** 生成→评估→反馈→改进，循环直到满意。

**举例：**
```python
builder.add_edge(START, "generator")
builder.add_edge("generator", "evaluator")
builder.add_conditional_edges("evaluator", should_continue, {
    "done": END,
    "retry": "generator",  # 回到生成器
})
```

**应用：** 内容优化、代码生成、Prompt 调优。三个必须：结构化反馈、终止条件、评估器比生成器更可靠。

---

### 卡片8：模式组合——真实项目的样子

**一句话：** 实际项目很少只用一种模式，通常是多种模式嵌套组合。

**举例：**
```
智能客服系统 = 路由 + 并行 + 顺序
  路由：分流到技术/投诉/闲聊
  并行：技术问题同时查多个知识库
  顺序：投诉走 情感分析→安抚→工单

自适应 RAG = 路由 + 顺序 + 评估者-优化者
  路由：查询分类
  顺序：检索→生成
  评估者-优化者：检查幻觉→重新生成
```

**应用：** 组合的原则是"由外到内"——先确定整体模式（通常是路由），再确定每条路径内部的模式。

---

### 卡片9：Graph API vs Functional API

**一句话：** 两种 API 风格，同样的模式，不同的写法。

**举例：**
```python
# Graph API：显式定义图结构
builder = StateGraph(State)
builder.add_node("a", func_a)
builder.add_edge(START, "a")
builder.add_edge("a", END)
graph = builder.compile()

# Functional API：更 Pythonic
@entrypoint()
def workflow(x):
    return func_a(x).result()
```

| 场景 | 推荐 API |
|------|---------|
| 复杂图结构 | Graph API |
| 简单链式 | Functional API |
| 需要可视化 | Graph API |
| 快速原型 | Functional API |

**应用：** 不用纠结选哪个——先用 Functional API 快速验证，复杂了再切 Graph API。

---

### 卡片10：选择模式的黄金法则

**一句话：** 从最简单的模式开始，只有明显不够时才升级。

**决策顺序：**
```
1. 先试顺序链式 → 够用就停
2. 需要分类？加路由
3. 有独立子任务？加并行
4. 子任务数量动态？用 Send/Map-Reduce
5. 需要 LLM 分解任务？用编排者-工作者
6. 需要迭代改进？用评估者-优化者
```

**应用：** 过度设计是最常见的错误。一个简单的 RAG 管道用顺序链式就够了，不需要编排者-工作者 + Map-Reduce 的组合。**简单就是美。**
