## 卡片1: 直觉理解 - 一致性级别是什么

### 核心概念

**一致性级别 = 数据新鲜度保证**

在分布式向量数据库中，数据存储在多个节点上。当你写入数据后，不同节点看到这个数据的时间不同。一致性级别定义了"读取操作能看到多新的数据"。

### 日常类比

想象你在微信群里发了一条消息：
- **Strong一致性**：所有人立即看到（但发送慢）
- **Bounded一致性**：大多数人几秒内看到（平衡）
- **Eventually一致性**：最终所有人都会看到（发送快）

### 为什么需要

分布式系统的物理约束：
1. 网络传输需要时间
2. 节点同步需要时间
3. 不能既快又准

### 关键洞察

**一致性级别不是"越强越好"，而是"够用就好"。**

## 卡片2: 形式化定义 - CAP定理视角

### CAP定理

```
C (Consistency): 所有节点看到相同的数据
A (Availability): 所有请求都能得到响应
P (Partition Tolerance): 系统在网络分区时仍能工作

定理：在网络分区时，只能保证C或A，不能同时保证
```

### 三种选择

| 一致性级别 | CAP选择 | 权衡 |
|-----------|---------|------|
| **Strong** | CP | 牺牲可用性，保证一致性 |
| **Bounded** | 平衡 | 在C和A之间取得平衡 |
| **Eventually** | AP | 牺牲一致性，保证可用性 |

### 数学表达

```
Strong:     GuaranteeTs = 最新时间戳
Bounded:    GuaranteeTs = 最新时间戳 - staleness_bound
Eventually: GuaranteeTs = 0（无保证）
```

### 记忆口诀

**CP = 准确但慢，AP = 快但可能旧，平衡 = Bounded**

## 卡片3: Strong一致性 - 原理与应用

### 核心原理

**等待所有节点同步到最新时间戳**

```
写入流程：
1. 写入主节点
2. 同步复制到所有节点
3. 等待所有节点确认
4. 返回成功

读取流程：
1. 获取最新时间戳
2. 等待所有节点同步到最新
3. 读取数据
```

### 性能代价

```
延迟：120ms（基准）
吞吐量：1000 QPS
内存：400GB（全索引加载）
```

### 适用场景

✅ **应该使用**：
- 金融交易数据
- 实时库存管理
- 关键业务数据
- 法律文档RAG

❌ **不应该使用**：
- 一般文档问答
- 推荐系统
- 日志分析
- 批量数据处理

### 代码示例

```python
collection = Collection(
    name="financial_data",
    schema=schema,
    consistency_level="Strong"
)

results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "L2", "params": {"nprobe": 10}},
    limit=10,
    consistency_level=ConsistencyLevel.Strong
)
```

### 记住

**Strong = 准确性优先 = 高代价 = 仅用于关键场景**

## 卡片4: Bounded一致性 - 原理与应用

### 核心原理

**等待时间窗口内的时间戳**

```
时间窗口 = staleness_bound（默认0秒）

读取流程：
1. 计算目标时间戳 = 当前时间 - staleness_bound
2. 等待节点同步到目标时间戳
3. 读取数据
```

### 性能特点

```
延迟：60ms（比Strong快2倍）
吞吐量：1500 QPS（比Strong高50%）
内存：100GB（部分索引加载）
```

### 为什么是默认

1. 经过大量生产验证
2. 适合70%的场景
3. 性能和一致性的最佳平衡
4. 在大多数情况下接近Strong

### 适用场景

✅ **应该使用**（默认选择）：
- 文档问答系统
- 推荐系统
- 实时搜索
- 知识库检索
- 不确定用哪个时

### 代码示例

```python
# 创建集合（默认Bounded）
collection = Collection(
    name="docs",
    schema=schema,
    consistency_level="Bounded"  # 默认级别
)

# 搜索时使用默认级别
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=10
    # 使用集合默认的Bounded级别
)
```

### 记住

**Bounded = 最佳默认选择 = 70%场景的最优解**

## 卡片5: Eventually一致性 - 原理与应用

### 核心原理

**不等待同步，直接读取**

```
写入流程：
1. 写入主节点
2. 立即返回（不等待复制）
3. 后台异步复制到其他节点

读取流程：
1. 直接从本地缓存读取
2. 不检查时间戳
3. 可能读到旧数据
```

### 性能优势

```
延迟：40ms（比Strong快3倍）
吞吐量：2000 QPS（比Strong高100%）
内存：100GB（按需加载）
```

### 行业标准

根据社区讨论：
> **大多数向量数据库都使用Eventually一致性**

原因：
- 向量检索是读多写少
- 用户对几秒延迟不敏感
- 性能比实时一致性更重要

### 适用场景

✅ **应该使用**：
- 批量数据导入
- 日志分析
- 大规模推荐系统
- 离线文档处理
- 高吞吐量场景

### 代码示例

```python
# 批量导入场景
collection = Collection(
    name="batch_import",
    schema=schema,
    consistency_level="Eventually"
)

# 批量搜索（最高性能）
results = collection.search(
    data=batch_query_vectors,
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=10,
    consistency_level=ConsistencyLevel.Eventually
)
```

### 常见误区

❌ **误区**："Eventually不可靠"
✅ **真相**：Eventually是行业标准，最终一定一致

### 记住

**Eventually = 最高性能 = 行业标准 = 批量场景首选**

## 卡片8: Milvus 2.6实现 - Streaming Node + Woodpecker WAL

### 架构概览

```
┌─────────────┐
│ Coordinator │  生成全局版本化查询视图
└──────┬──────┘
       │
       ├──────────────┬──────────────┐
       ↓              ↓              ↓
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│Streaming N1 │ │Streaming N2 │ │ Query Node  │
└─────────────┘ └─────────────┘ └─────────────┘
       │              │              │
       └──────────────┴──────────────┘
              跨节点状态机
           维护一致性保证
```

### Streaming Node

**核心机制**：
1. 协调器生成全局版本化查询视图
2. 同步到所有streaming node和query node
3. 跨节点状态机维护一致性

**一致性实现**：
```go
func (w *WAL) Read(ctx context.Context, level ConsistencyLevel) (*Record, error) {
    switch level {
    case Strong:
        return w.readWithSync(ctx)  // 等待最新
    case Bounded:
        return w.readWithBoundedStaleness(ctx, w.maxStaleness)  // 等待窗口
    case Eventually:
        return w.readAsync(ctx)  // 不等待
    }
}
```

### Woodpecker WAL

**核心机制**：
- 基于对象存储的WAL
- Checkpoint机制跟踪同步进度
- 后台同步进程保证最终一致

**Checkpoint管理**：
```go
// Strong: 等待checkpoint更新到最新
func (cm *CheckpointManager) WaitForCheckpoint(ctx context.Context, targetCp uint64) error

// Bounded: 等待checkpoint在时间窗口内
func (cm *CheckpointManager) WaitForBoundedCheckpoint(ctx context.Context, stalenessBound time.Duration) error

// Eventually: 不等待checkpoint
func (cm *CheckpointManager) ReadWithoutWait(ctx context.Context) (*Data, error)
```

### GuaranteeTs机制

**时间戳保证**：
```
Strong:     GuaranteeTs = 最新时间戳
Bounded:    GuaranteeTs = 最新时间戳 - staleness_bound
Eventually: GuaranteeTs = 0
```

### 记住

**Milvus 2.6通过Streaming Node + Woodpecker WAL实现了灵活的一致性级别控制**

## 卡片9: RAG场景选择 - 决策框架

### 场景映射表

| RAG场景 | 推荐级别 | 理由 |
|---------|---------|------|
| **文档问答** | Bounded | 平衡性能与准确性 |
| **实时知识库** | Session | 用户立即看到更新 |
| **批量处理** | Eventually | 最高性能 |
| **交互式聊天** | Session | 用户体验优先 |
| **金融/法律RAG** | Strong | 准确性优先 |
| **大规模推荐** | Eventually | 吞吐量优先 |

### 决策框架

```python
def choose_consistency_level(scenario, requirements):
    # 关键数据优先使用Strong
    if requirements['data_criticality'] == 'critical':
        return 'Strong'

    # 根据场景类型选择
    if scenario == 'document_qa':
        return 'Bounded'  # 默认
    elif scenario == 'realtime_kb':
        if requirements['update_frequency'] == 'high':
            return 'Session'  # 高频更新
        else:
            return 'Bounded'
    elif scenario == 'batch_processing':
        return 'Eventually'  # 性能优先
    elif scenario == 'interactive_chat':
        return 'Session'  # 用户体验
    else:
        return 'Bounded'  # 默认
```

### 实际案例

**案例1：文档问答系统**
```python
# 使用Bounded（默认）
collection = Collection(
    name="document_qa",
    schema=schema,
    consistency_level="Bounded"
)

# 查询时使用默认级别
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=5
)
```

**案例2：批量文档导入**
```python
# 使用Eventually（性能优先）
collection = Collection(
    name="batch_import",
    schema=schema,
    consistency_level="Eventually"
)

# 批量插入
for batch in batches:
    collection.insert(batch)
```

**案例3：实时知识库**
```python
# 使用Session（用户体验）
collection = Collection(
    name="realtime_kb",
    schema=schema,
    consistency_level="Session"
)

# 更新后立即查询
collection.upsert([{"kb_id": "001", "vector": embedding, "content": "..."}])
results = collection.search(...)  # 能看到刚更新的内容
```

### 快速决策

```
批量导入 → Eventually
交互式应用 → Session
一般RAG → Bounded
关键数据 → Strong
```

## 卡片10: 总结与延伸 - 生产环境最佳实践

### 核心要点总结

1. **一致性级别 = 数据新鲜度 vs 查询速度的权衡**
2. **Strong = 准确但慢，Bounded = 平衡，Eventually = 快但可能旧**
3. **默认使用Bounded，适合70%的场景**
4. **批量场景用Eventually，交互式用Session**
5. **关键数据用Strong，但要承受性能代价**

### 生产环境最佳实践

#### 1. 默认配置

```python
# 推荐：使用Bounded作为默认
collection = Collection(
    name="my_collection",
    schema=schema,
    consistency_level="Bounded"  # 默认级别
)
```

#### 2. 场景化配置

```python
# 根据场景临时调整
def search_with_scenario(query_vector, scenario):
    if scenario == "critical":
        level = ConsistencyLevel.Strong
    elif scenario == "batch":
        level = ConsistencyLevel.Eventually
    else:
        level = ConsistencyLevel.Bounded

    return collection.search(
        data=[query_vector],
        anns_field="vector",
        param={"metric_type": "COSINE", "params": {"nprobe": 10}},

        consistency_level=level
```

#### 3. 性能监控

```python
import time
import numpy as np

def monitor_consistency_performance(collection):
    levels = ["Strong", "Bounded", "Eventually"]
    results = {}

    for level in levels:
        latencies = []
        for _ in range(100):
            start = time.time()
            collection.search(
                data=[np.random.rand(128).tolist()],
                anns_field="vector",
                param={"metric_type": "L2", "params": {"nprobe": 10}},
                limit=10,
                consistency_level=level
            )
            latencies.append(time.time() - start)

        results[level] = {
            "mean": np.mean(latencies),
            "p95": np.percentile(latencies, 95),
            "p99": np.percentile(latencies, 99)
        }

    return results
```

#### 4. 成本优化

```
优化策略：
1. 默认使用Bounded（平衡）
2. 批量场景降级到Eventually（性能）
3. 关键场景升级到Strong（准确性）

预期效果：
- 性能提升：30-50%
- 成本降低：50-90%
- 用户体验：无明显差异
```
