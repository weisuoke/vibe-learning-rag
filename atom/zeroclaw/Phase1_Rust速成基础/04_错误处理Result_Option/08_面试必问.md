## 为什么准备这两个问题？

这两个问题覆盖了 Rust 错误处理的两个核心维度：

1. **语言机制**：Result vs try/catch——理解 Rust 错误处理的本质

面试官通过这两个问题，能判断你是"背了语法"还是"真正理解了设计哲学"。

## 问题1："Rust 的 Result 和 TypeScript 的 try/catch 有什么区别？"

### 普通回答（不出彩）

> "Result 是一个枚举，有 Ok 和 Err 两个变体。try/catch 是异常机制。Result 用模式匹配处理，try/catch 用 catch 块处理。本质上都是错误处理，只是语法不同。"

### 出彩回答（推荐）

> **三层本质区别：**
>
> **1. 类型层面——编译时强制 vs 运行时可选**
>
> Result 是类型系统的一部分。函数签名 `fn read_file(path: &str) -> Result<String, io::Error>` 明确告诉调用者"这个函数可能失败，失败时错误类型是 io::Error"。编译器强制你处理这个 Result——不处理就编译不过。
>
> TypeScript 的 try/catch 是运行时机制。函数签名 `function readFile(path: string): string` 完全看不出它会抛异常。你可以不写 try/catch，编译器不管，运行时才崩溃。
>
> ```rust
> // Rust：不处理 Result → 编译器警告
> let content = fs::read_to_string("config.json");
> //  ^^^^^^^ unused `Result` that must be used
> ```
>
> ```typescript
> // TypeScript：不写 try/catch → 编译通过，运行时崩溃
> const content = fs.readFileSync("config.json", "utf-8");
> // 文件不存在？运行时 throw，没人接住 → 进程可能崩溃
> ```
>
> **2. 传播层面——显式 vs 隐式**
>
> Rust 用 `?` 运算符显式传播错误，每个函数签名都标注了"我可能失败"。你看调用链就知道哪些地方可能出错。
>
> TypeScript 的 throw 隐式冒泡。一个函数调了十层深的函数，任何一层都可能 throw，但你从函数签名完全看不出来。
>
> ```rust
> // Rust：? 让错误传播路径一目了然
> fn init() -> anyhow::Result<()> {
>     let config = load_config("app.json")?;  // 可能失败，显式标注
>     let db = connect_db(&config.db_url)?;    // 可能失败，显式标注
>     Ok(())
> }
> ```
>
> **3. 组合层面——函数式链 vs 嵌套 catch**
>
> Result 支持 `map`、`and_then`、`unwrap_or` 等函数式组合，可以优雅地链式处理。try/catch 只能嵌套或者 `.catch()` 链。
>
> ```rust
> // Rust：组合子链
> let port = config.get("port")
>     .and_then(|v| v.as_u64())
>     .unwrap_or(8080);
> ```
>
> **在 ZeroClaw 中的体现**：所有 trait 方法返回 `anyhow::Result<T>`，配合 `.context()` 构建完整错误链。每个 `?` 都是一个显式的"这里可能失败"标记，代码审查时一眼就能看到所有错误路径。

### 面试官可能的追问

**追问 A："那 Rust 的 panic 和 TypeScript 的 throw 是不是一样的？"**

> 不一样。panic 是 Rust 中"不可恢复的 bug"，类似于 C++ 的 abort——整个进程终止。它不是正常的错误处理机制。正常错误用 Result，只有逻辑 bug（比如数组越界）才 panic。
>
> TypeScript 的 throw 既用于"正常错误"（文件不存在）也用于"bug"（类型错误），没有区分。Rust 把这两种情况分开了：Result 处理预期错误，panic 处理不可恢复的 bug。

**追问 B："? 运算符和 TypeScript 的 ?. 可选链有什么关系？"**

> 形式相似但语义完全不同：
> - TypeScript `?.`：遇到 null/undefined 就短路返回 undefined（值层面）
> - Rust `?`：遇到 Err/None 就提前 return 错误到调用者（控制流层面）
>
> ```typescript
> const name = user?.profile?.name;  // 短路返回 undefined
> ```
> ```rust
> let name = get_user()?.profile()?.name()?;  // 短路 return Err(...)
> ```

[来源: reference/search_error_handling_01.md - Rust 的 ? 运算符类似 TS 的 optional chaining]

### 为什么这个回答出彩？

1. **分层递进**：不是笼统说"不一样"，而是从类型、传播、组合三个层面逐步深入
2. **双语代码对比**：每个层面都有 Rust vs TypeScript 的代码对照，面试官能直观感受差异
3. **联系实际项目**：提到 ZeroClaw 的 trait 设计，展示你理解这些差异如何影响架构决策
4. **抓住本质**：核心洞察是"编译时强制 vs 运行时可选"——这不是语法差异，是安全保证级别的根本不同

## 问题2："什么时候用 anyhow，什么时候用 thiserror？"

### 普通回答（不出彩）

> "anyhow 用于应用程序，thiserror 用于库。anyhow 简单方便，thiserror 可以定义具体的错误类型。"

### 出彩回答（推荐）

> **核心判断标准：调用者是否需要 match 具体错误变体。**
>
> **1. thiserror——调用者需要区分错误类型**
>
> 当你写的是库代码（给别人用的 crate），调用者可能需要根据不同错误做不同处理。这时用 thiserror 定义具体的错误枚举：
>
> ```rust
> #[derive(thiserror::Error, Debug)]
> pub enum StorageError {
>     #[error("not found: {0}")]
>     NotFound(String),
>     #[error("connection failed")]
>     Connection(#[from] io::Error),
>     #[error("permission denied: {0}")]
>     PermissionDenied(String),
> }
>
> // 调用者可以 match 做不同恢复策略
> match storage.get("key") {
>     Err(StorageError::NotFound(_)) => use_default(),
>     Err(StorageError::Connection(_)) => retry(),
>     Err(StorageError::PermissionDenied(_)) => escalate(),
>     Ok(v) => use_value(v),
> }
> ```
>
> **2. anyhow——调用者只需要知道"失败了"**
>
> 当你写的是应用层代码（main 函数、HTTP handler、CLI 入口），错误最终是打印给用户或写入日志的，不需要精确区分类型。这时用 anyhow，重点是附加上下文信息方便调试：
>
> ```rust
> fn run() -> anyhow::Result<()> {
>     let config = load_config("app.json")
>         .context("Failed to load config")?;
>     let db = connect_db(&config.db_url)
>         .context("Failed to connect database")?;
>     Ok(())
> }
> // 错误输出：Failed to connect database → Connection refused (os error 111)
> // 人类可读，调试友好，不需要 match
> ```
>
> **3. 混合使用——2025-2026 年 Rust 社区最佳实践**
>
> 实际项目中最常见的模式是混合使用：
> - 内部模块用 thiserror 定义精确错误
> - 顶层用 anyhow 包装，附加上下文
> - thiserror 的类型自动实现 `std::error::Error`，`?` 可以无缝转成 `anyhow::Error`
>
> ```rust
> // 内部模块：thiserror 精确定义
> #[derive(thiserror::Error, Debug)]
> pub enum ConfigError {
>     #[error("file not found: {0}")]
>     FileNotFound(String),
>     #[error("parse failed")]
>     Parse(#[from] serde_json::Error),
> }
>
> // 顶层：anyhow 统一包装
> fn main() -> anyhow::Result<()> {
>     let config = load_config("app.json")
>         .context("initialization failed")?;
>     //  ConfigError → anyhow::Error（自动转换）
>     Ok(())
> }
> ```
>
> **ZeroClaw 的选择**：所有 trait 方法统一返回 `anyhow::Result<T>`。原因有三：
> 1. ZeroClaw 是应用层代码，不是给别人用的库
> 2. 不同 Tool 实现遇到的错误类型完全不同（Shell 有超时、Web 有网络错误、File 有 IO 错误），用枚举穷举不现实
> 3. Trait 统一性比错误类型精确性更重要——`anyhow::Result<T>` 让所有 trait 方法签名一致

### 为什么这个回答出彩？

1. **给出判断标准**：不是死记"库用 thiserror、应用用 anyhow"，而是给出核心判断依据——"调用者是否需要 match"
2. **三层递进**：thiserror 单独用 → anyhow 单独用 → 混合使用，展示完整的工程视角
3. **代码说话**：每个场景都有代码示例，不是空谈理论

### 面试官可能的追问

**追问 A："如果 ZeroClaw 将来要把 Tool 模块拆成独立 crate，错误处理怎么改？"**

> 拆成独立 crate 后，Tool 模块就变成了"库代码"——调用者（ZeroClaw 主程序）可能需要 match 具体错误做不同处理。这时应该：
>
> 1. 在 `zeroclaw-tools` crate 中用 thiserror 定义 `ToolError` 枚举
> 2. ZeroClaw 主程序在调用处用 anyhow 包装
>
> ```rust
> // zeroclaw-tools/src/error.rs（库层）
> #[derive(thiserror::Error, Debug)]
> pub enum ToolError {
>     #[error("tool '{0}' not found")]
>     NotFound(String),
>     #[error("execution timed out")]
>     Timeout,
>     #[error(transparent)]
>     Other(#[from] anyhow::Error),
> }
>
> // zeroclaw/src/main.rs（应用层）
> match tool.execute(args).await {
>     Err(ToolError::Timeout) => retry_with_longer_timeout(),
>     Err(ToolError::NotFound(name)) => suggest_similar(&name),
>     Err(e) => eprintln!("Tool error: {e}"),
>     Ok(r) => display(r),
> }
> ```
>
> 这就是"混合模式"的典型应用——模块边界决定了用哪个库。

[来源: reference/source_error_handling_01.md - 统一错误类型：anyhow::Result<T>]

**追问 B："anyhow 和 Box\<dyn Error\> 有什么区别？"**

> 功能上类似——都是类型擦除的错误容器。但 anyhow 额外提供了：
> - `.context()` / `.with_context()` 附加上下文
> - `bail!()` / `ensure!()` 便捷宏
> - `.chain()` 遍历错误链
> - `.downcast_ref()` 类型恢复
> - 自动 backtrace 支持
>
> `Box<dyn Error>` 是标准库的"裸容器"，anyhow 是"带工具箱的容器"。实际项目中几乎没有理由用 `Box<dyn Error>` 而不用 anyhow。

### anyhow vs thiserror 速查对比

| 维度 | anyhow | thiserror |
|------|--------|-----------|
| **定位** | 应用层错误容器 | 库层错误定义工具 |
| **核心能力** | 类型擦除 + 上下文链 | derive 宏生成 Display/Error/From |
| **返回类型** | `anyhow::Result<T>` | `Result<T, MyError>` |
| **调用者能力** | 打印、日志、downcast | match 具体变体 |
| **便捷宏** | `anyhow!`, `bail!`, `ensure!` | `#[error]`, `#[from]`, `#[source]` |
| **错误链** | `.context()` 逐层附加 | `#[source]` 标记底层原因 |
| **ZeroClaw 用法** | 所有 trait 方法返回类型 | 未来拆分 crate 时使用 |
| **TypeScript 类比** | `catch(e: unknown)` | `class XxxError extends Error` |
