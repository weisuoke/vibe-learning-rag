# 条件分支策略 - 30字核心

**条件分支策略是 LangGraph 通过路由函数在运行时动态决定下一节点的机制，实现工作流的智能决策与分支控制。**

---

## 核心要素拆解

### 1. "路由函数" —— 决策的载体

路由函数是条件分支的核心：接收当前状态 `state`，返回目标节点名（字符串）。

```python
def route_function(state) -> Literal["node_a", "node_b", "node_c"]:
    """路由函数 = 一个纯函数，输入状态，输出目标节点名"""
    if state["score"] > 0.8:
        return "node_a"
    elif state["score"] > 0.5:
        return "node_b"
    return "node_c"
```

三种路由载体：
- **路由函数** → `add_conditional_edges(source, route_fn)` → 最常用
- **Send 对象** → `return [Send("worker", arg)]` → 动态并行
- **Command 对象** → `return Command(goto="target")` → 状态+路由一体化

### 2. "运行时动态决定" —— 编译时 vs 运行时

```
编译时（compile）：定义所有可能的路径
  ├── "node_a" 是可能的目标 ✓
  ├── "node_b" 是可能的目标 ✓
  └── "node_c" 是可能的目标 ✓

运行时（invoke）：根据实际状态选择一条路径
  └── state["score"] = 0.9 → 走 "node_a" ✓
```

**关键区分**：图结构在 `compile()` 后不可变，"动态"指的是路径选择，而非修改图本身。

### 3. "智能决策与分支控制" —— 不只是 if-else

条件分支策略覆盖的决策模式：

| 模式 | 机制 | 场景 |
|------|------|------|
| 简单分支 | 条件边 + if-else | 意图分类路由 |
| 多路并行 | 条件边返回列表 | 同时走多条路 |
| 动态并行 | Send API | 运行时决定并行数量 |
| 状态+路由 | Command API | 一次操作改状态+选路径 |
| 回退重试 | 条件边 + 循环 | 质量不达标时回退 |
| 降级处理 | 条件边 + 计数器 | 重试耗尽后降级 |

## 关键词提取

| 关键词 | 含义 |
|--------|------|
| 条件分支 | 根据条件走不同路径（区别于固定路径的静态边） |
| 路由函数 | 接收 state、返回节点名的纯函数，是决策逻辑的载体 |
| 运行时动态 | 路径选择发生在 invoke 时，而非 compile 时 |
| path_map | 路由值到节点名的映射表，解耦路由逻辑和节点命名 |
| Literal 推断 | 通过类型注解自动识别所有可能的目标节点 |
| Send | 运行时创建动态并行分支的 API |
| Command | 同时更新状态和控制路由的一体化 API |
| 回退机制 | 条件不满足时路由回之前的节点重试 |
| 循环守卫 | recursion_limit / 计数器，防止无限循环 |

## 与 RAG/Agent 开发的关联

条件分支策略在 RAG 和 Agent 系统中无处不在：

**RAG 场景**:
- 检索质量评估 → 质量达标走生成，不达标走查询改写（Self-RAG）
- 文档类型判断 → PDF 走 PDF 解析器，网页走 HTML 解析器
- 多策略检索 → 向量检索失败时回退到关键词检索

**Agent 场景**:
- 工具选择 → LLM 决定调用搜索、计算还是代码执行（ReAct）
- 多 Agent 调度 → Supervisor 决定下一个执行的 Agent
- 任务完成判断 → 结果满意则输出，不满意则继续迭代

## 一句话记忆

**条件边选路径，Send 开并行，Command 管全局，回退保底线 —— 四招让工作流在每个岔路口都做出最优决策。**

## 核心公式

```
条件分支策略 = 路由函数(state) → 目标节点名
             + 编译时定义可能性
             + 运行时选择路径
             + 回退与守卫保障
```
