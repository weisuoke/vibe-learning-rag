## 卡片 1：Plan-and-Execute 一句话

```
Plan-and-Execute = PM + 工程师

PM（Planner）定需求 → 工程师（Executor）写代码 → 产品验收（Replanner）

三个角色，各司其职：
  Planner：大模型定计划（What to do）
  Executor：小模型做任务（How to do）
  Replanner：大模型评进度（Done or Adjust?）

记忆锚点：先规划，后执行，最后检查。
```

---

## 卡片 2：P&E vs ReAct 速记

```
ReAct = 走一步看一步（没有需求文档的野蛮开发）
P&E = 先列计划再做（有需求文档的正规开发）

什么时候选谁？
  1-2 步 → ReAct（快、开销小）
  3+ 步 → P&E（不遗漏、有全局视角）
  探索性 → ReAct（灵活适应）

核心差异：Token 消耗
  ReAct → 每步带全部历史，指数增长
  P&E → 执行器只看当前步骤，线性增长
```

## 卡片 3：任务分解的最佳实践

```
规则：3-7 步，粗粒度

❌ 过度分解（20步）：
  搜索A → 记录A版本 → 搜索A特性 → 记录A特性...
  → 慢、贵、失去灵活性

✅ 适度分解（5步）：
  搜索A和B → 对比分析 → 场景匹配 → 给出建议 → 生成报告
  → 每步有灵活度

关键点：
  - 规划只管"做什么"，执行管"怎么做"
  - 用 Structured Output 约束步骤格式
  - 最后一步强制是"验证/总结"
```

## 卡片 4：重规划不是 Bug，是 Feature

```
GPS 走错路 → 重新规划路线 → 到达目的地 ✅
Agent 信息变了 → 重新规划步骤 → 完成任务 ✅

触发重规划的时机：
  1. 步骤失败（工具报错）
  2. 结果偏离预期（搜错了）
  3. 发现新信息（A被B替代了）
  4. 所有步骤完成后评估

防护措施：
  replan_count >= 3 → 强制终止
  （3次还解决不了 = 规划器要优化了）

经验参考：
  0次重规划 → 任务太简单，可能不需要 P&E
  1-2次 → 正常范围
  3+次 → 优化 Prompt
```

## 卡片 5：Structured Output 是规划的基石

```
没有 Structured Output：
  LLM → "我觉得可以先搜索一下..."
  → 模糊、不可解析、不可靠

有 Structured Output：
  class Plan(BaseModel):
      steps: List[str]

  LLM → {"steps": ["搜索...", "分析...", "总结..."]}
  → 确定、直接可用、类型安全

实现方式：
  planner = llm.with_structured_output(Plan)
  plan = planner.invoke("分解这个任务...")

类比：TypeScript 之于 JavaScript
  给 LLM 输出加上"类型系统"
```

## 卡片 6：三种规划架构对比

```
           Plan-and-Execute    ReWOO          LLMCompiler
规划方式    步骤列表           带变量的步骤   DAG 有向图
执行方式    串行              串行           并行
特色       简单可靠           $1引用变量     自动并行

选型速查：
  初学/简单 → Plan-and-Execute
  步骤间有数据传递 → ReWOO
  需要高性能并行 → LLMCompiler

99% 的场景用 Plan-and-Execute 就够了。
```

## 卡片 7：反思三兄弟

```
Reflection（反思）= 写完试卷检查一遍
  → 只评估当前结果
  → 不合格就重做
  → 不记忆经验

Reflexion（反射）= 错题本 + 考前复习
  → 记录失败经验
  → 下次避免同样错误
  → 越用越好

LLM-as-Judge（评委）= 同行评审
  → 独立第三方评价
  → 比自我反思更客观
  → 多一次 LLM 调用成本

生产中最常用：Reflection + 质量门控
  评分 >= 7 → 通过
  评分 < 7 → 重新生成（最多3轮）
```

## 卡片 8：LangGraph 实现 P&E 的核心代码

```
# 状态
class State(TypedDict):
    input: str
    plan: List[str]
    step_index: int
    past_steps: Annotated[List[tuple], operator.add]  # ← 自动累积
    response: str

# 三个节点
graph = StateGraph(State)
graph.add_node("planner", plan_node)     # 生成计划
graph.add_node("executor", execute_node)  # 执行步骤
graph.add_node("replanner", replan_node)  # 评估进度

# 路由
graph.set_entry_point("planner")
graph.add_edge("planner", "executor")
graph.add_conditional_edges("executor", route, {
    "execute": "executor",   # 还有步骤
    "replan": "replanner",   # 步骤完了
    "end": END,              # 有结果了
})

# 记住：Annotated + operator.add 是状态累积的关键技巧
```

## 卡片 9：模型分工省钱秘籍

```
规划 = 战略层 → 需要强推理 → GPT-4o（贵）
执行 = 战术层 → 按指令做 → GPT-4o-mini（便宜）

5步计划的成本对比：
  全用 GPT-4o → 6次大模型调用 → $$$$$$
  分工模式 → 1次大模型 + 5次小模型 → $$$
  节省约 60%

为什么可行？
  → 计划已经明确了每步该做什么
  → 执行不需要强推理，只需要正确调用工具
  → 就像 PM 定好需求后，初级工程师也能完成具体编码

代码一行搞定：
  planner_llm = ChatOpenAI(model="gpt-4o")
  executor_llm = ChatOpenAI(model="gpt-4o-mini")
```

## 卡片 10：三种场景递进关系

```
场景1：基础 Plan-and-Execute
  → 会规划、会执行、会重规划
  → 入门必学

场景2：自适应规划系统
  → 会动态调整、会失败重试、会任务插入
  → 生产必需

场景3：带反思的多步推理
  → 会自我评估、会自我批评、会迭代改进
  → 高质量保障

递进关系：
  基础 P&E → 自适应规划 → 反思增强
  （能做事） → （能应变） → （能出精品）

一句话：
  Plan-and-Execute 是骨架，
  自适应是肌肉，
  反思是大脑。
```
