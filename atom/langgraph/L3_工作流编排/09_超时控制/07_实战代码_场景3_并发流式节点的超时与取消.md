# 实战代码 · 场景 3：并发流式节点的超时与取消

> **目标**：复现“快节点先产出 chunk，但慢节点最终被取消，`astream()` 抛 timeout”的场景
> **适用**：前端流式 UI、SSE、后台增量输出、观察背压问题

---

## 场景说明

这个例子直接借鉴源码测试思路：

- `fast_node()` 很快返回；
- `slow_node()` 明显更慢；
- `graph.step_timeout = 1.0`；
- 消费端故意在 `async for` 里 `sleep(0.6)`，制造背压；
- 最终观察 `slow_node` 被取消、流式执行抛 `asyncio.TimeoutError`。

---

## 完整代码

```python
"""
场景3：并发流式节点的超时与取消

运行方式：
    python example_stream_backpressure_timeout.py

依赖：
    pip install langgraph
"""

from __future__ import annotations

import asyncio
from typing import Any

from typing_extensions import TypedDict

from langgraph.graph import StateGraph


class State(TypedDict):
    hello: str


async def slow_node(state: State) -> dict[str, str]:
    try:
        print("[slow_node] 开始执行")
        await asyncio.sleep(1.5)
        print("[slow_node] 完成")
        return {"hello": "slow done"}
    except asyncio.CancelledError:
        print("[slow_node] 收到取消信号")
        raise


async def fast_node(state: State) -> dict[str, str]:
    print("[fast_node] 开始执行")
    await asyncio.sleep(0.3)
    print("[fast_node] 完成")
    return {"hello": "fast done"}


def build_graph():
    builder = StateGraph(State)
    builder.add_node(slow_node)
    builder.add_node(fast_node)

    # 关键：让入口点同时触发两个节点，制造同一 step 并发
    builder.set_conditional_entry_point(lambda _: ["slow_node", "fast_node"])

    graph = builder.compile()
    graph.step_timeout = 1.0
    return graph


async def consume_stream_with_backpressure(graph, payload: State) -> None:
    async for chunk in graph.astream(payload, stream_mode="updates"):
        print(f"[consumer] 收到 chunk: {chunk}")

        # 故意变慢，模拟 UI 渲染 / 日志写入 / SSE 处理开销
        await asyncio.sleep(0.6)
        print("[consumer] chunk 处理完成")


async def main() -> None:
    graph = build_graph()

    try:
        await consume_stream_with_backpressure(graph, {"hello": "world"})
    except asyncio.TimeoutError as exc:
        print("\n[graph timeout]")
        print("异常类型:", type(exc).__name__)
        print("异常信息:", str(exc))
        print("解释：当前 step 预算耗尽，未完成任务被取消。")


if __name__ == "__main__":
    asyncio.run(main())
```

## 这个示例最值钱的地方

### 1. 快节点成功并不代表 step 完成

这是理解 `step_timeout` 的关键。

### 2. 消费端自己也会影响 timeout 体验

如果你在流式消费里做重计算、重渲染、重日志，就会制造背压。

### 3. 被取消的不一定是“出错的节点”

`slow_node` 被取消，不一定说明它逻辑错误，只是当前 step 的预算已经耗尽。

## 真实业务里的对应关系

### Web 前端
- 收到 token / chunk 后做复杂富文本渲染；
- 每次更新都触发昂贵状态同步；
- 导致流式消费本身变慢。

### 中间层服务
- 每个 chunk 都写 DB / Redis / Kafka；
- 处理链积压。

### 数据处理系统
- 流式输出先到聚合器；
- 聚合器本身吞吐不够，反向拖慢整体。

---

## 如何优化？

1. `async for` 内只做最小工作；
2. 慢处理异步化或入队；
3. 把慢节点拆出单独 step；
4. 调整 step budget 前，先确认是否是消费端背压。

## 变体实现：参数化辅助层

```python
from __future__ import annotations

import asyncio
import time
from dataclasses import dataclass
from typing import Any

@dataclass
class TimeoutBudgets:
    node_timeout_s: float = 1.0
    step_timeout_s: float = 2.0
    consumer_timeout_s: float = 0.5

async def stream_backpressure_demo_remote_call(delay_s: float) -> dict[str, Any]:
    await asyncio.sleep(delay_s)
    return {'ok': True, 'delay_s': delay_s}

async def stream_backpressure_demo_guarded_call(delay_s: float, budgets: TimeoutBudgets) -> dict[str, Any]:
    started_at = time.perf_counter()
    try:
        result = await asyncio.wait_for(stream_backpressure_demo_remote_call(delay_s), timeout=budgets.node_timeout_s)
        result["timed_out"] = False
        result["elapsed_s"] = round(time.perf_counter() - started_at, 3)
        return result
    except asyncio.TimeoutError:
        return {
            "ok": False,
            "timed_out": True,
            "delay_s": delay_s,
            "elapsed_s": round(time.perf_counter() - started_at, 3),
        }

async def stream_backpressure_demo_main() -> None:
    budgets = TimeoutBudgets()
    result = await stream_backpressure_demo_guarded_call(delay_s=1.8, budgets=budgets)
    if result["timed_out"]:
        print("[并发流式节点的超时与取消] 使用 fallback 或降级路径")
    else:
        print("[并发流式节点的超时与取消] 使用主路径结果")
    print(result)

if __name__ == "__main__":
    asyncio.run(stream_backpressure_demo_main())
```
