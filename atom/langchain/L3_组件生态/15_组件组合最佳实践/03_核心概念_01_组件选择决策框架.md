# 核心概念1：组件选择决策框架

## 一句话定义

**组件选择决策框架是根据应用场景、性能需求和复杂度，系统性地选择最合适的 LangChain 组件和组合方式的方法论。**

---

## 为什么需要决策框架？

LangChain 生态组件众多，初学者最常犯的错误不是"不会用"，而是"选错了"。

### 没有框架时的典型问题

```
场景：写一个简单的翻译功能

新手做法：
  1. 创建 Agent
  2. 给 Agent 配 Tool
  3. 加上 Memory
  4. 用 LangGraph 管理状态
  → 200 行代码，调试 3 小时

老手做法：
  prompt | model | StrOutputParser()
  → 3 行代码，5 分钟搞定
```

**核心问题：** 不是工具不好用，是你用了大炮打蚊子。

### 决策框架的价值

决策框架帮你回答三个问题：
1. **该不该用 LangChain？** —— 简单任务直接用 SDK 更好
2. **用哪些组件？** —— 不同场景对应不同组件组合
3. **怎么组合？** —— 串行、并行、条件、降级的选择

---

## 决策维度

### 维度1：应用复杂度

这是最重要的第一判断维度。复杂度决定了你需要的"武器等级"。

| 复杂度 | 特征 | 推荐方案 | 代码量 |
|--------|------|----------|--------|
| 简单 | 单次 LLM 调用，固定流程 | 直接用 SDK 或简单 LCEL 链 | 5-20 行 |
| 中等 | 多步骤，有检索/解析 | LCEL 链 (RunnableSequence) | 20-80 行 |
| 复杂 | 条件分支，多路径 | LCEL + RunnableBranch | 80-200 行 |
| 高级 | 循环、状态、多代理 | LangGraph | 200+ 行 |

#### 各复杂度的代码对比

```python
# ===== 简单：直接用 SDK =====
# 单次 LLM 调用，没有检索、没有分支、没有记忆
from openai import OpenAI

client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "把 Hello World 翻译成中文"}],
)

# ===== 中等：LCEL 链 =====
# 多步骤处理，有明确的数据流向
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4o-mini")
    | StrOutputParser()
)

# ===== 复杂：LCEL + 条件路由 =====
# 根据输入动态选择不同处理路径
route_chain = RunnableBranch(
    (lambda x: "代码" in x["question"], code_qa_chain),
    (lambda x: "文档" in x["question"], doc_qa_chain),
    general_qa_chain,  # 默认分支
)

# ===== 高级：LangGraph =====
# 需要循环、持久化状态、多 Agent 协作、人机交互审批流程
# → 超出 LCEL 能力范围，用 LangGraph
```

### 维度2：执行模式选择

选对执行模式直接影响用户体验和系统吞吐量。

| 需求 | 推荐模式 | 对应方法 | 适用场景 |
|------|----------|----------|----------|
| 同步单次调用 | invoke | `chain.invoke()` | 后端 API、脚本 |
| 异步单次调用 | ainvoke | `chain.ainvoke()` | FastAPI 异步路由 |
| 批量处理 | batch/abatch | `chain.batch()` | 离线分析、数据处理 |
| 实时流式 | stream/astream | `chain.stream()` | 聊天界面、实时反馈 |
| 流式事件 | astream_events | `chain.astream_events()` | 复杂链的中间状态监控 |

#### 执行模式决策树

```
你的场景是什么？
  │
  ├─ 用户在聊天界面等待回复
  │   → stream / astream（逐字输出，体验好）
  │
  ├─ 后端 API 处理请求
  │   ├─ 用了 async 框架（FastAPI）→ ainvoke
  │   └─ 同步框架（Flask）→ invoke
  │
  ├─ 批量处理一堆文档/问题
  │   → batch / abatch（自动并行，吞吐高）
  │
  └─ 需要监控链中每个步骤的输出
      → astream_events（事件级粒度）
```

#### 关键源码洞察

从 `base.py` 源码（第 2817 行 RunnableSequence）可以看到：

```python
# RunnableSequence.batch 的优化逻辑：
# 对序列中的每个步骤调用 batch，利用线程池并行处理
# 这意味着 batch 不是简单的循环调用 invoke，而是有真正的并行优化

# 流式传播规则：
# 如果序列中所有组件都实现了 transform 方法，整个序列可以端到端流式
# 注意：RunnableLambda 默认不支持 transform，会阻断流式传播
```

### 维度3：组件类型选择

#### Model 选择

```
你的任务需要什么级别的模型？
  │
  ├─ 简单任务（分类、提取、翻译）
  │   → gpt-4o-mini / claude-haiku（便宜快速）
  │
  ├─ 复杂推理（分析、创作、多步推理）
  │   → gpt-4o / claude-sonnet（更强大）
  │
  ├─ 需要本地部署（隐私/离线）
  │   → Ollama + llama3 / qwen2
  │
  └─ 成本敏感的批量任务
      → 批处理 API（OpenAI Batch API 半价）
```

```python
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic

cheap_model = ChatOpenAI(model="gpt-4o-mini", temperature=0)   # 简单任务
strong_model = ChatOpenAI(model="gpt-4o", temperature=0)       # 复杂推理
# 带降级：主力挂了自动切换
model_with_fallback = strong_model.with_fallbacks([ChatAnthropic(model="claude-sonnet-4-20250514")])
```

#### Retriever 选择

| 场景 | 推荐组件 | 原因 |
|------|----------|------|
| 基础语义检索 | `VectorStoreRetriever` | 最简单，够用就行 |
| 混合检索（精确+语义） | `EnsembleRetriever` | BM25 + 向量互补 |
| 需要重排序 | `ContextualCompressionRetriever` | 召回后精排提升精度 |
| 查询不够好 | `MultiQueryRetriever` | 自动改写查询，多角度召回 |
| 需要父文档上下文 | `ParentDocumentRetriever` | 小块检索，大块返回 |

```python
# 基础：直接用 VectorStore
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# 进阶：混合检索
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever

ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6],
)

# 高级：混合检索 + 重排序
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import CrossEncoderReranker

retriever_with_rerank = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=ensemble,
)
```

#### Memory / 对话历史选择

| 场景 | 推荐方案 | 说明 |
|------|----------|------|
| 单轮问答 | 不需要 Memory | 别加没用的复杂度 |
| 简单多轮对话 | `RunnableWithMessageHistory` | 官方推荐的新方式 |
| 需要持久化 | `RunnableWithMessageHistory` + Redis/DB | 外部存储 |
| 超长对话 | 摘要 + 滑动窗口 | 避免 context window 溢出 |

```python
# 2025-2026 推荐方式：RunnableWithMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_community.chat_message_histories import ChatMessageHistory

store = {}

def get_session_history(session_id: str):
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

chain_with_history = RunnableWithMessageHistory(
    chain, get_session_history,
    input_messages_key="question", history_messages_key="history",
)
```

#### OutputParser 选择

| 需求 | 推荐组件 | 说明 |
|------|----------|------|
| 纯文本输出 | `StrOutputParser` | 最常用，直接提取文本 |
| JSON 结构 | `JsonOutputParser` | 解析 JSON 字符串 |
| Pydantic 模型 | `PydanticOutputParser` | 类型安全的结构化输出 |
| 流式 JSON | `SimpleJsonOutputParser` | 边生成边解析 |
| 列表输出 | `CommaSeparatedListOutputParser` | 逗号分隔列表 |

```python
from langchain_core.output_parsers import StrOutputParser, JsonOutputParser
from langchain_core.pydantic_v1 import BaseModel, Field

# 最常用：纯文本
chain = prompt | model | StrOutputParser()

# 结构化输出：Pydantic
class Answer(BaseModel):
    answer: str = Field(description="回答内容")
    confidence: float = Field(description="置信度 0-1")
    sources: list[str] = Field(description="引用来源")

parser = JsonOutputParser(pydantic_object=Answer)
chain = prompt | model | parser
```

## 决策流程图

把上面三个维度串起来，形成完整的决策流程：

```
开始：我要构建一个 LLM 应用
  │
  ├─ 需要循环/状态/多代理？ → 是 → LangGraph
  │   否 ↓
  ├─ 需要条件分支？ → 是 → LCEL + RunnableBranch
  │   否 ↓
  ├─ 需要并行处理？ → 是 → LCEL + RunnableParallel
  │   否 ↓
  ├─ 简单单次调用？ → 是 → 直接 SDK 或简单链
  │   否 ↓
  └─ LCEL RunnableSequence（默认选择）
```

## 实际应用场景

### 场景1：文档问答系统

**需求分析：** 用户上传文档，提问获取答案。无循环、无分支、有检索。

**组件选择：**
- Retriever: `VectorStoreRetriever`（Chroma）—— 文档量不大，内存向量库够用
- Model: `ChatOpenAI`（gpt-4o-mini）—— 问答任务不需要最强模型
- Parser: `StrOutputParser` —— 纯文本回答
- 组合: `RunnableParallel`（并行获取 context + question）+ `RunnableSequence`
- 执行模式: `astream`（聊天界面逐字输出）

### 场景2：多语言翻译（带降级）

**需求分析：** 翻译任务，需要高可用。主模型挂了要自动切换。

**组件选择：**
- Model: `ChatOpenAI` + `with_fallbacks` —— 主力 + 备用
- Prompt: `ChatPromptTemplate` —— 简单模板
- Parser: `StrOutputParser` —— 纯文本

```python
"""场景2：多语言翻译 —— 带降级的组件组合"""
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# 源码 fallbacks.py：RunnableWithFallbacks 按顺序尝试，第一个成功就返回
model = ChatOpenAI(model="gpt-4o-mini").with_fallbacks(
    [ChatAnthropic(model="claude-haiku-4-20250414")]
)

prompt = ChatPromptTemplate.from_template(
    "将以下文本翻译成{target_language}：\n\n{text}"
)
translation_chain = prompt | model | StrOutputParser()
```

### 场景3：智能客服（条件路由）

**需求分析：** 根据用户意图分流到不同处理链。有分支，无循环。

**组件选择：**
- Router: `RunnableBranch` —— 条件路由
- 多个专业链: 售前/售后/技术支持 —— 各自独立的 LCEL 链
- Memory: `RunnableWithMessageHistory` —— 多轮对话

```python
"""场景3：智能客服 —— 条件路由组合"""
from langchain_core.runnables import RunnableBranch, RunnablePassthrough

# 意图分类链
classify_chain = classify_prompt | model | StrOutputParser()

# 条件路由：源码 branch.py 按顺序检查条件，第一个为 True 的被选中
router = RunnableBranch(
    (lambda x: "售前" in x["intent"], presale_chain),
    (lambda x: "售后" in x["intent"], aftersale_chain),
    tech_chain,  # 默认走技术支持
)

# 完整链：分类 → 路由 → 处理
full_chain = RunnablePassthrough.assign(intent=classify_chain) | router
```

## 反模式（不要这样做）

### 反模式1：过度使用 LangChain

```python
# ❌ 简单任务套了一层 LangChain，增加了依赖和复杂度
chain = ChatPromptTemplate.from_template("{text}") | ChatOpenAI() | StrOutputParser()

# ✅ 直接用 SDK
response = OpenAI().chat.completions.create(
    model="gpt-4o-mini", messages=[{"role": "user", "content": "你好"}]
)
```

**判断标准：** 如果你的"链"只有 `prompt | model | parser`，而且不需要流式、批处理、降级等特性，直接用 SDK 更简单。

### 反模式2：组件选择不匹配

```python
# ❌ 没有工具的 Agent 没有意义
agent = create_react_agent(model, tools=[])  # tools 为空，退化为普通 LLM 调用

# ✅ 简单任务用 Chain
chain = prompt | model | parser
```

### 反模式3：忽略执行模式

```python
# ❌ 实时对话用 batch，用户要等所有问题处理完
results = chain.batch(questions)
# ✅ 实时对话用 stream，逐字输出
async for chunk in chain.astream(question): yield chunk

# ❌ 批量处理用循环 invoke，串行慢
results = [chain.invoke(q) for q in questions]
# ✅ 批量处理用 batch，自动并行
results = chain.batch(questions, config={"max_concurrency": 5})
```

### 反模式4：重试粒度太大

```python
# ❌ 对整个链重试，prompt 和 parser 不会失败，重试它们没意义
chain = (prompt | model | parser).with_retry(max_attempt_number=3)
# ✅ 只对可能失败的组件重试（源码 retry.py 的最佳实践）
chain = prompt | model.with_retry(max_attempt_number=3) | parser
```

## 在 RAG 开发中的应用

RAG 系统是组件选择决策框架的最佳实践场景，因为它天然涉及多个组件的协作。

### RAG 组件选择决策表

| 决策点 | 选项 A | 选项 B | 选择依据 |
|--------|--------|--------|----------|
| 向量库 | Chroma（内存） | Milvus/Pinecone（分布式） | 文档量：<10万用 A，>10万用 B |
| 检索策略 | 纯向量检索 | 混合检索 + ReRank | 速度优先用 A，精度优先用 B |
| 执行模式 | stream | batch | 实时对话用 A，批量分析用 B |
| 对话历史 | 无 Memory | RunnableWithMessageHistory | 单轮问答用 A，多轮对话用 B |
| 模型选择 | gpt-4o-mini | gpt-4o | 简单问答用 A，复杂分析用 B |
| 错误处理 | with_retry | with_fallbacks | 瞬态错误用 A，服务不可用用 B |

### 典型 RAG 管道的组件组合

```python
"""生产级 RAG 管道：展示组件选择决策的完整应用"""
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# Model：问答用 mini，带降级保证可用性
model = ChatOpenAI(model="gpt-4o-mini").with_fallbacks(
    [ChatOpenAI(model="gpt-4o")]
)

# Retriever：文档量不大，Chroma 够用
vectorstore = Chroma(embedding_function=OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# Prompt + Parser
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个知识库助手。根据上下文回答问题，不要编造信息。"),
    ("human", "上下文：\n{context}\n\n问题：{question}"),
])

def format_docs(docs):
    return "\n\n---\n\n".join(doc.page_content for doc in docs)

# 组合：RunnableParallel + RunnableSequence
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt | model | StrOutputParser()
)

# 执行模式根据场景选择：API → ainvoke / 聊天 → astream / 批量 → batch
```

## 组件选择速查卡

遇到选择困难时，按这个顺序快速决策：

```
1. 复杂度 → 决定用不用 LangChain / LangGraph
2. 执行模式 → 决定 invoke / stream / batch
3. Model → 决定用哪个模型（成本 vs 能力）
4. Retriever → 决定检索策略（如果有检索需求）
5. 错误处理 → 决定 retry / fallback（生产环境必须有）
6. Memory → 决定是否需要对话历史（多轮才需要）
```

---

## 小结

| 要点 | 说明 |
|------|------|
| 三个决策维度 | 复杂度、执行模式、组件类型 |
| 第一原则 | 能简单就别复杂，直接 SDK > 简单 LCEL > 复杂 LCEL > LangGraph |
| 执行模式 | 实时用 stream，批量用 batch，API 用 ainvoke |
| 错误处理 | retry 针对瞬态错误，fallback 针对服务不可用，粒度要小 |
| RAG 特有 | 向量库选型看数据量，检索策略看精度需求，模型选型看任务复杂度 |
| 核心反模式 | 过度封装、组件不匹配、执行模式错误、重试粒度太大 |
