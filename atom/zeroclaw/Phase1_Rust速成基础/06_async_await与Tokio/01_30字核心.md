## 30 字核心

---

## 一、关键词拆解

### 1.1 `async fn` — 异步函数声明

`async fn` 在语法上和 TypeScript 的 `async function` 几乎一样，但含义完全不同：

```rust
// Rust — async fn 声明
async fn fetch_data(url: &str) -> Result<String> {
    let resp = reqwest::get(url).await?;
    resp.text().await
}
```

```typescript
// TypeScript — 对比
async function fetchData(url: string): Promise<string> {
    const resp = await fetch(url);
    return resp.text();
}
```

**关键差异**：Rust 的 `async fn` 不返回 `Promise`，而是返回一个实现了 `Future` trait 的**匿名类型**（编译器自动生成的状态机结构体）。

### 1.2 编译为惰性状态机 — 最核心的差异

**"惰性"**：调用 `async fn` 不会执行任何代码，只是创建一个 Future 实例。

```rust
let future = fetch_data("https://api.example.com");
// ↑ 此时什么都没发生！没有发送任何 HTTP 请求
// future 只是一个状态机实例，等待被驱动

future.await?;
// ↑ 现在才开始真正执行：发送请求、等待响应
```

**"状态机"**：编译器将 `async fn` 中每个 `.await` 点转化为一个状态：

```rust
// 编译器生成的心智模型（简化）
enum FetchDataFuture {
    Start { url: String },             // 初始状态
    WaitingForResponse { ... },         // 等待 HTTP 响应
    WaitingForText { resp: Response },  // 等待读取 body
    Done,                               // 完成
}
```

### 1.3 Tokio 调度执行 — 运行时驱动

Future 是惰性的，需要有人来**驱动（poll）**它。Tokio 就是那个驱动者：

- **Tokio 运行时** = 多线程工作窃取调度器
- 它反复调用 Future 的 `poll()` 方法，推动状态机前进
- 当 Future 返回 `Pending`（还没完成，在等 IO），Tokio 去执行别的任务
- 当 IO 就绪，Tokio 通过 `Waker` 回来继续 poll

```rust
#[tokio::main]  // ← 启动 Tokio 运行时
async fn main() {
    // 在 Tokio 运行时内，.await 才有意义
    let data = fetch_data("https://api.example.com").await.unwrap();
}
```

### 1.4 ZeroClaw 1939 个异步函数的底层引擎

ZeroClaw 的四大核心 Trait 全部是异步的：

```rust
#[async_trait]
pub trait Provider: Send + Sync {
    async fn chat(&self, messages: Vec<Message>) -> Result<Message>;
    // ... 12 个方法
}

#[async_trait]
pub trait Tool: Send + Sync {
    async fn execute(&self, args: Value) -> Result<ToolResult>;
    // ... 5 个方法
}

#[async_trait]
pub trait Memory: Send + Sync {
    async fn store(&self, content: &str) -> Result<()>;
    async fn recall(&self, query: &str) -> Result<Vec<MemoryEntry>>;
    // ... 8 个方法
}

#[async_trait]
pub trait Channel: Send + Sync {
    async fn send(&self, message: ChannelMessage) -> Result<()>;
    async fn listen(&self, tx: mpsc::Sender<ChannelMessage>) -> Result<()>;
    // ... 13 个方法
}
```

每一个方法调用的背后，都是 `async fn` → 状态机 → Tokio 调度的完整链路。

## 二、30 字核心的三层含义

### 第一层：它是什么（机制）

`async fn` 是 Rust 编译器的一种**代码变换**——把看似顺序执行的代码转换为一个可以暂停/恢复的状态机。`.await` 是暂停点，Tokio 是恢复者。

### 第二层：为什么需要（问题）

高并发 IO 场景下（LLM 调用、网络通信、数据库读写），同步阻塞意味着浪费。async 让程序在等待 IO 时去做别的事，一个线程可以驱动成千上万个任务。

### 第三层：在 ZeroClaw 中的价值（实践）

Agent 同时需要：调用 LLM、执行工具、收发消息、监控组件健康——全部依赖 async/await + Tokio 实现非阻塞并发。没有 async，ZeroClaw 就只能一件事做完再做下一件。

## 三、TypeScript 对照理解

```
TypeScript                              Rust
───────────────────────────────────────────────────────────
async function → Promise<T>             async fn → impl Future<Output = T>
Promise 立即执行                         Future 惰性，.await 才执行
V8 事件循环驱动                          Tokio 运行时驱动
单线程                                   多线程（工作窃取调度器）
取消需要 AbortController                  drop Future 即取消
Promise.all()                            tokio::join!
Promise.race()                           tokio::select!
```

## 四、记忆口诀

> **async 造状态机，await 是暂停点，Tokio 来调度，IO 不阻塞。**

- async 造状态机：`async fn` 编译为 enum 状态机
- await 是暂停点：每个 `.await` 都是一个可暂停/恢复的位置
- Tokio 来调度：多线程运行时驱动 Future 执行
- IO 不阻塞：等待 IO 时去做别的任务，提高吞吐量

## 总结

```
  async fn 编译为惰性状态机，Tokio 调度执行，
  是 ZeroClaw 1939 个异步函数的底层引擎。

  async fn    ────→ 编译为状态机 enum，不是 Promise
  惰性执行    ────→ 调用不执行，.await 或 poll 才执行
  Tokio 调度  ────→ 多线程运行时，驱动 Future 完成
  底层引擎    ────→ 四大 Trait × 全部 async = 1939 个 async fn
```
