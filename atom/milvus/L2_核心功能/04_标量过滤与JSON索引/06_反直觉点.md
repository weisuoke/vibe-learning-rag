## 1. 索引相关误区

### 误区 1："索引越多越好"

**直觉认知**：
```
更多索引 = 更快查询
所以应该为所有字段创建索引
```

**实际情况**：
```
索引有成本：
- 存储空间：额外 20-50%
- 构建时间：初始化慢
- 更新成本：插入/删除变慢
- 内存占用：索引需要加载到内存
```

**正确做法**：
```python
# ❌ 错误：为所有字段创建索引
index_params.add_index("id", "INVERTED")
index_params.add_index("name", "INVERTED")
index_params.add_index("age", "INVERTED")
index_params.add_index("email", "INVERTED")
index_params.add_index("phone", "INVERTED")
# ... 100个字段

# ✅ 正确：只为常用查询字段创建索引
index_params.add_index("id", "INVERTED")  # 主键，必须
index_params.add_index("email", "INVERTED")  # 常用查询
index_params.add_index("status", "BITMAP")  # 常用过滤
```

**原因**：
- 大部分字段很少被查询
- 索引维护成本高
- 内存占用增加
- 写入性能下降

### 误区 2："标量过滤会显著降低性能"

**直觉认知**：
```
标量过滤 = 额外计算
所以会显著降低查询性能
```

**实际情况**：
```
有索引：性能提升 30 倍
无索引：确实会慢

关键：是否创建索引
```

**性能对比**：
```python
# 无索引
查询延迟：1000ms

# 有索引
查询延迟：30ms

# 性能提升：33 倍
```

**正确认知**：
- 标量过滤本身不慢
- 关键是索引优化
- 合理的索引可以显著提升性能

[来源: reference/search_web_results.md | Milvus 2.6 官方博客]

---

### 误区 3："BITMAP 索引适用于所有场景"

**直觉认知**：
```
BITMAP 索引性能最好
所以应该优先使用 BITMAP
```

**实际情况**：
```
BITMAP 只适用于低基数字段（< 100个值）
高基数字段使用 BITMAP 会浪费内存
```

**示例**：
```python
# ❌ 错误：高基数字段使用 BITMAP
index_params.add_index("user_id", "BITMAP")  # 100万个值
index_params.add_index("email", "BITMAP")  # 数百万个值

# ✅ 正确：根据基数选择索引
index_params.add_index("status", "BITMAP")  # 3个值（低基数）
index_params.add_index("gender", "BITMAP")  # 3个值（低基数）
index_params.add_index("user_id", "INVERTED")  # 100万个值（高基数）
index_params.add_index("email", "INVERTED")  # 数百万个值（高基数）
```

**判断标准**：
```python
基数 = DISTINCT(field) / COUNT(*)

IF 基数 < 0.01:  # 1% 以下
    使用 BITMAP
ELSE:
    使用 INVERTED
```

## 2. JSON 相关误区

### 误区 4："JSON 查询一定很慢"

**直觉认知**：
```
JSON 嵌套访问 = 慢
所以应该避免使用 JSON
```

**实际情况**：
```
无索引：确实很慢（10000ms）
JSON Path Index：100 倍性能提升（100ms）

关键：是否创建 JSON Path Index
```

**性能对比**：
```python
# 无索引
查询延迟：10000ms

# JSON Path Index
查询延迟：100ms

# 性能提升：100 倍
```

**正确做法**：
```python
# 为常用 JSON 路径创建索引
index_params.add_index(
    field_name="metadata",
    index_type="INVERTED",
    params={
        "json_path": "metadata['category']",
        "json_cast_type": "varchar"
    }
)
```

[来源: reference/search_web_results.md | Milvus 2.6 官方博客]

---

### 误区 5："JSON Path 使用点号语法"

**直觉认知**：
```javascript
// JavaScript
const brand = product.metadata.product.brand;

// Milvus（错误）
"metadata.product.brand"
```

**实际情况**：
```python
# ❌ 错误：使用点号语法
"metadata.product.brand"  # Milvus 不支持

# ✅ 正确：使用方括号语法
"metadata['product']['brand']"
```

**原因**：
- Milvus 使用 JSON Pointer 规范
- 不支持 JavaScript 风格的点号语法
- 必须使用方括号

---

### 误区 6："JSON 字段不需要指定类型"

**直觉认知**：
```
JSON 是动态类型
所以不需要指定 json_cast_type
```

**实际情况**：
```python
# ❌ 错误：不指定 json_cast_type
index_params.add_index(
    field_name="metadata",
    index_type="INVERTED",
    params={
        "json_path": "metadata['price']"
        # 缺少 json_cast_type
    }
)

# ✅ 正确：必须指定 json_cast_type
index_params.add_index(
    field_name="metadata",
    index_type="INVERTED",
    params={
        "json_path": "metadata['price']",
        "json_cast_type": "double"  # 必须指定
    }
)
```

**原因**：
- 索引需要知道数据类型
- 类型转换需要明确指定
- 避免类型不匹配错误

## 3. 查询相关误区

### 误区 7："选择性低的条件应该放在前面"

**直觉认知**：
```
先执行简单条件
再执行复杂条件
```

**实际情况**：
```
选择性高的条件应该放在前面
可以快速减少候选集
```

**示例**：
```python
# ❌ 错误：选择性低的条件在前
"price >= 100 AND price <= 500 AND stock > 0"
# 价格范围保留 80% 数据（选择性低）
# stock > 0 只保留 10% 数据（选择性高）

# ✅ 正确：选择性高的条件在前
"stock > 0 AND price >= 100 AND price <= 500"
# stock > 0 先执行，快速过滤掉 90% 数据
```

**性能差异**：
```
错误写法：需要处理 80% 数据
正确写法：只需要处理 10% 数据
性能提升：8 倍
```

---

### 误区 8："OR 操作符比 IN 更快"

**直觉认知**：
```
OR 是基础操作符
应该比 IN 更快
```

**实际情况**：
```
IN 操作符性能更好
支持索引优化
```

**示例**：
```python
# ❌ 不推荐：多个 OR
"city == '北京' OR city == '上海' OR city == '深圳' OR city == '广州'"

# ✅ 推荐：使用 IN
"city IN ['北京', '上海', '深圳', '广州']"
```

**原因**：
- IN 操作符有专门的优化
- 支持索引批量查找
- 代码更简洁易读

---

### 误区 9："NOT 操作符性能很好"

**直觉认知**：
```
NOT 只是简单的取反
应该很快
```

**实际情况**：
```
NOT 可能导致全表扫描
性能较差
```

**示例**：
```python
# ❌ 不推荐：使用 NOT
"NOT (age >= 60)"  # 可能全表扫描

# ✅ 推荐：使用直接操作符
"age < 60"  # 可以使用索引
```

**原因**：
- NOT 需要先获取所有结果，再取反
- 无法直接使用索引
- 优先使用 `!=`、`<`、`>` 等直接操作符

## 4. 混合查询误区

### 误区 10："Pre-filtering 总是更快"

**直觉认知**：
```
先过滤再检索
减少搜索空间
所以总是更快
```

**实际情况**：
```
高选择性（< 50%）：Pre-filtering 更快
低选择性（> 50%）：Post-filtering 更快

关键：过滤选择性
```

**性能对比**：
```python
# 高选择性（10%）
Pre-filtering: 50ms
Post-filtering: 200ms
# Pre-filtering 更快

# 低选择性（90%）
Pre-filtering: 250ms
Post-filtering: 110ms
# Post-filtering 更快
```

**原因**：
- Pre-filtering 在候选集小时有优势
- Post-filtering 在候选集大时有优势
- Milvus 会自动选择最优策略

### 误区 11："混合查询会显著增加延迟"

**直觉认知**：
```
标量过滤 + 向量检索 = 双倍延迟
```

**实际情况**：
```
有索引：延迟增加 20-50%
无索引：延迟增加 10 倍

关键：索引优化
```

**性能对比**：
```python
# 纯向量检索
延迟：100ms

# 混合查询（有索引）
延迟：120-150ms（增加 20-50%）

# 混合查询（无索引）
延迟：1000ms（增加 10 倍）
```

**正确认知**：
- 合理的索引可以控制延迟增加
- 混合查询的召回率提升 15-30%
- 性价比很高

## 5. 数据类型误区

### 误区 12："字符串数字可以直接比较"

**直觉认知**：
```python
"99" < "100"  # 应该为 true
```

**实际情况**：
```python
# 字符串比较（字典序）
"99" > "100"  # true（字典序）

# 数值比较
99 < 100  # true（数值）
```

**正确做法**：
```python
# ❌ 错误：字符串存储数字
schema.add_field("price", DataType.VARCHAR)
data = {"price": "99.99"}

# ✅ 正确：使用数值类型
schema.add_field("price", DataType.DOUBLE)
data = {"price": 99.99}

# 或者使用类型转换
index_params.add_index(
    field_name="metadata",
    index_type="INVERTED",
    params={
        "json_path": "metadata['price']",
        "json_cast_type": "double",
        "json_cast_function": "STRING_TO_DOUBLE"
    }
)
```

---

### 误区 13："空字符串等于 NULL"

**直觉认知**：
```python
"" == NULL  # 应该为 true
```

**实际情况**：
```python
# 空字符串不等于 NULL
"" != NULL

# NULL 表示未定义
NULL 表示字段值不存在
```

**示例**：
```python
# 空字符串
{"description": ""}  # 有值，但是空

# NULL
{"description": null}  # 无值

# 字段不存在
{}  # 字段不存在
```

**查询差异**：
```python
# 查找空字符串
"description == ''"

# 查找 NULL
"description IS NULL"

# 查找非空
"description IS NOT NULL AND description != ''"
```

## 6. 性能优化误区

### 误区 14："AUTOINDEX 总是最优"

**直觉认知**：
```
AUTOINDEX 自动选择
所以总是最优
```

**实际情况**：
```
AUTOINDEX 基于启发式规则
可能不是最优选择
```

**示例**：
```python
# 场景：低基数字段（3个值）
# AUTOINDEX 可能选择 INVERTED
# 但 BITMAP 更优

# ❌ 使用 AUTOINDEX
index_params.add_index("status", "AUTOINDEX")
# 可能选择 INVERTED（不是最优）

# ✅ 手动指定 BITMAP
index_params.add_index("status", "BITMAP")
# 性能更好
```

**建议**：
- 了解数据特征时，手动指定索引类型
- 不确定时，使用 AUTOINDEX
- 生产环境建议手动优化

---

### 误区 15："索引构建后不需要维护"

**直觉认知**：
```
索引一次构建
永久有效
```

**实际情况**：
```
数据分布变化后
索引可能不再最优
需要定期重建
```

**示例**：
```python
# 初始数据：status 只有 2 个值（低基数）
# 使用 BITMAP 索引

# 数据增长后：status 有 100 个值（中等基数）
# BITMAP 不再最优，应该使用 INVERTED

# 重建索引
client.drop_index(collection_name, "status_index")
index_params.add_index("status", "INVERTED")
client.create_index(collection_name, index_params)
```

**建议**：
- 定期检查数据分布
- 数据量变化较大时重建索引
- 监控索引使用情况

## 7. 常见陷阱

### 陷阱 1：过度优化

**问题**：
```python
# 为每个字段创建多个索引
index_params.add_index("age", "INVERTED")
index_params.add_index("age", "BITMAP")
index_params.add_index("age", "STL_SORT")
```

**后果**：
- 存储空间浪费
- 构建时间长
- 维护成本高

**正确做法**：
```python
# 只创建一个最优索引
index_params.add_index("age", "INVERTED")
```

---

### 陷阱 2：忽略数据分布

**问题**：
```python
# 不考虑数据分布，盲目使用 BITMAP
index_params.add_index("user_id", "BITMAP")  # 100万个值
```

**后果**：
- 内存占用大
- 性能不佳

**正确做法**：
```python
# 先分析数据分布
基数 = DISTINCT(user_id) / COUNT(*)
IF 基数 < 0.01:
    使用 BITMAP
ELSE:
    使用 INVERTED
```

---

### 陷阱 3：忽略查询模式

**问题**：
```python
# 不考虑查询模式，盲目创建索引
index_params.add_index("rare_field", "INVERTED")  # 很少被查询
```

**后果**：
- 索引浪费
- 维护成本高

**正确做法**：
```python
# 只为常用查询字段创建索引
# 监控查询日志，识别高频字段
```

## 8. 记忆技巧

### 口诀

```
索引不是越多越好，
JSON查询不一定慢，
选择性高的条件在前，
IN比OR性能更好。

BITMAP只适合低基数，
NOT操作要谨慎用，
混合查询延迟可控，
空字符串不等于NULL。
```

### 关键原则

1. **索引有成本**：只为常用字段创建索引
2. **类型很重要**：根据基数选择索引类型
3. **顺序影响性能**：选择性高的条件在前
4. **简化优于复杂**：使用 IN 替代多个 OR
5. **监控很关键**：定期检查索引使用情况
