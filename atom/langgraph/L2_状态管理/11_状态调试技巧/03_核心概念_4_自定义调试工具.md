# 自定义调试工具

> 内置 API 不够用时，自己造轮子——从 print 到装饰器到 StreamWriter，逐步构建适合你项目的调试体系

---

## 一句话定义

**自定义调试工具是在 LangGraph 内置调试 API（stream、get_state）之外，开发者根据项目需求自行构建的日志、装饰器、差异对比等辅助手段，用于更灵活地追踪状态变化和定位问题。**

---

## 为什么需要自定义调试工具？

LangGraph 提供了 `stream_mode="debug"`、`get_state()`、`get_state_history()` 等内置调试手段，已经很强大了。但实际开发中你会遇到这些场景：

- **快速验证**：改了一行代码，想立刻看到某个变量的值，不想搭建完整的流式管道
- **结构化日志**：生产环境需要按级别过滤日志，print 满足不了
- **状态差异追踪**：节点前后状态到底改了什么？内置 API 给你完整快照，但不会帮你做 diff
- **批量调试**：想给所有节点统一加上调试逻辑，不想一个个改
- **自定义数据流**：需要在节点内部向外部推送特定的调试信息

**前端类比：** 就像前端开发——浏览器 DevTools 已经很强了，但你还是会用 `console.log`、自定义 Logger 类、Redux DevTools 中间件。不同层次的工具解决不同层次的问题。

**日常生活类比：** 医院有 CT、核磁等高级设备（内置 API），但医生日常还是会用听诊器、体温计（print/logging）。简单工具响应快，复杂工具信息全，两者配合使用。

---

## 工具一：节点内 print 调试

### 最简单也最有效的方法

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    query: str
    docs: list[str]
    answer: str

def retrieve_node(state: State):
    print(f"[retrieve] 输入 query: {state['query']}")

    # 模拟检索
    docs = [f"文档_{i}: 关于{state['query']}的内容" for i in range(3)]

    print(f"[retrieve] 检索到 {len(docs)} 篇文档")
    print(f"[retrieve] 第一篇: {docs[0][:50]}...")
    return {"docs": docs}

def generate_node(state: State):
    print(f"[generate] 收到 {len(state['docs'])} 篇文档")
    print(f"[generate] query: {state['query']}")

    answer = f"基于 {len(state['docs'])} 篇文档的回答"

    print(f"[generate] 生成答案: {answer[:80]}...")
    return {"answer": answer}

# 构建图
builder = StateGraph(State)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

graph = builder.compile()
result = graph.invoke({"query": "什么是RAG", "docs": [], "answer": ""})
```

**输出：**
```
[retrieve] 输入 query: 什么是RAG
[retrieve] 检索到 3 篇文档
[retrieve] 第一篇: 文档_0: 关于什么是RAG的内容...
[generate] 收到 3 篇文档
[generate] query: 什么是RAG
[generate] 生成答案: 基于 3 篇文档的回答...
```

### 适用场景

- 适合：本地开发、快速验证、图结构简单（3-5 个节点以内）
- 不适合：生产环境（无法过滤/持久化）、节点多调用频繁（输出刷屏）

命名技巧：给每条 print 加 `[节点名]` 前缀，多节点图中能快速定位来源。

## 工具二：Python logging 集成

### 从 print 升级到 logging

`logging` 是 Python 标准库，零依赖。相比 print，它支持日志级别、格式化、多输出目标。

```python
import logging
from typing import TypedDict

# ========== 配置 logging ==========
logger = logging.getLogger("langgraph.debug")
logger.setLevel(logging.DEBUG)

handler = logging.StreamHandler()
handler.setFormatter(logging.Formatter(
    "[%(asctime)s] %(levelname)-8s %(name)s | %(message)s",
    datefmt="%H:%M:%S"
))
logger.addHandler(handler)


# ========== 在节点中使用 ==========
class State(TypedDict):
    query: str
    docs: list[str]
    answer: str

def retrieve_node(state: State):
    logger.info(f"开始检索, query={state['query']}")

    docs = [f"doc_{i}" for i in range(3)]
    logger.debug(f"检索结果详情: {docs}")

    if len(docs) == 0:
        logger.warning("检索结果为空，可能影响生成质量")

    return {"docs": docs}

def generate_node(state: State):
    logger.info(f"开始生成, 文档数={len(state['docs'])}")

    try:
        answer = f"基于 {len(state['docs'])} 篇文档的回答"
        logger.debug(f"生成结果: {answer}")
        return {"answer": answer}
    except Exception as e:
        logger.error(f"生成失败: {e}", exc_info=True)
        return {"answer": "生成失败"}
```

### 日志级别速查

| 级别 | 用途 | 示例 |
|------|------|------|
| `DEBUG` | 详细的变量值、中间结果 | `logger.debug(f"docs={docs}")` |
| `INFO` | 关键流程节点 | `logger.info("开始检索")` |
| `WARNING` | 异常但不致命的情况 | `logger.warning("检索结果为空")` |
| `ERROR` | 错误，需要关注 | `logger.error(f"API 调用失败: {e}")` |
| `CRITICAL` | 系统级故障 | `logger.critical("向量库连接断开")` |

### 环境切换技巧

```python
import os

# 通过环境变量控制日志级别
log_level = os.getenv("LANGGRAPH_LOG_LEVEL", "INFO")
logger.setLevel(getattr(logging, log_level))

# 开发环境：LANGGRAPH_LOG_LEVEL=DEBUG python app.py
# 生产环境：LANGGRAPH_LOG_LEVEL=WARNING python app.py
```

## 工具三：StreamWriter 自定义输出

### 什么是 StreamWriter？

`get_stream_writer()` 是 LangGraph 提供的 API，允许你在节点内部向外部推送自定义数据。它和 `stream_mode="custom"` 配合使用，让你在流式输出中插入任意调试信息。

**前端类比：** 就像 WebSocket 的 `send()`——节点内部主动向外推送消息，外部监听者实时接收。

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.config import get_stream_writer
from langgraph.checkpoint.memory import InMemorySaver

class State(TypedDict):
    query: str
    docs: list[str]
    answer: str

def retrieve_node(state: State):
    writer = get_stream_writer()

    # 推送调试信息：检索开始
    writer({
        "event": "retrieve_start",
        "query": state["query"],
        "timestamp": "节点开始执行"
    })

    docs = [f"doc_{i}" for i in range(3)]

    # 推送调试信息：检索完成
    writer({
        "event": "retrieve_done",
        "doc_count": len(docs),
        "doc_ids": [d for d in docs]
    })

    return {"docs": docs}

def generate_node(state: State):
    writer = get_stream_writer()

    writer({"event": "generate_start", "input_docs": len(state["docs"])})

    answer = f"基于 {len(state['docs'])} 篇文档的回答"

    writer({"event": "generate_done", "answer_length": len(answer)})

    return {"answer": answer}

# 构建图
builder = StateGraph(State)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 使用 stream_mode="custom" 接收自定义数据
config = {"configurable": {"thread_id": "debug-1"}}
for chunk in graph.stream(
    {"query": "什么是RAG", "docs": [], "answer": ""},
    config=config,
    stream_mode="custom"
):
    print(f"[自定义流] {chunk}")
```

**输出：**
```
[自定义流] {'event': 'retrieve_start', 'query': '什么是RAG', 'timestamp': '节点开始执行'}
[自定义流] {'event': 'retrieve_done', 'doc_count': 3, 'doc_ids': ['doc_0', 'doc_1', 'doc_2']}
[自定义流] {'event': 'generate_start', 'input_docs': 3}
[自定义流] {'event': 'generate_done', 'answer_length': 14}
```

### 多模式组合

你可以同时使用多种 stream_mode，把自定义数据和状态更新一起接收：

```python
for mode, chunk in graph.stream(
    {"query": "什么是RAG", "docs": [], "answer": ""},
    config=config,
    stream_mode=["updates", "custom"]
):
    if mode == "custom":
        print(f"[调试] {chunk}")
    elif mode == "updates":
        print(f"[状态更新] {chunk}")
```

### StreamWriter vs print

| 对比项 | print | StreamWriter |
|--------|-------|-------------|
| 输出目标 | 标准输出（终端） | 流式管道（可编程消费） |
| 结构化 | 纯文本 | 任意 Python 对象 |
| 生产可用 | 不推荐 | 推荐 |
| 外部集成 | 困难 | 容易（API 端点、前端 UI） |
| 异步支持 | 无 | 原生支持 |

## 工具四：调试装饰器模式

### 问题：每个节点都写 print/logging 太重复

当图有 10+ 个节点时，每个节点都手动加调试代码，既重复又容易遗漏。装饰器模式一次定义，处处复用。

### 基础版装饰器

```python
import functools
import time

def debug_node(func):
    """通用节点调试装饰器：记录输入输出和耗时"""
    @functools.wraps(func)
    def wrapper(state):
        node_name = func.__name__
        print(f">>> 进入节点: {node_name}, 输入键: {list(state.keys())}")

        start = time.time()
        result = func(state)
        elapsed = time.time() - start

        print(f"<<< 离开节点: {node_name} (耗时 {elapsed:.3f}s), 更新键: {list(result.keys())}")
        return result
    return wrapper

# 使用：加一行装饰器就行
@debug_node
def retrieve_node(state: State):
    docs = [f"doc_{i}" for i in range(3)]
    return {"docs": docs}

@debug_node
def generate_node(state: State):
    return {"answer": f"基于 {len(state['docs'])} 篇文档的回答"}
```

### 进阶版：可开关的装饰器

```python
import os
import functools
import logging

logger = logging.getLogger("langgraph.debug")

def debug_node(func=None, *, enabled=None, level="DEBUG"):
    """
    可配置的调试装饰器

    用法：
        @debug_node                          # 默认，跟随环境变量
        @debug_node(enabled=True)            # 强制开启
        @debug_node(level="INFO")            # 指定日志级别
    """
    if func is None:
        return lambda f: debug_node(f, enabled=enabled, level=level)

    @functools.wraps(func)
    def wrapper(state):
        # 判断是否启用调试
        is_enabled = enabled
        if is_enabled is None:
            is_enabled = os.getenv("LANGGRAPH_DEBUG", "0") == "1"

        if not is_enabled:
            return func(state)

        node_name = func.__name__
        log_func = getattr(logger, level.lower(), logger.debug)

        log_func(f">>> [{node_name}] 输入键: {list(state.keys())}")
        result = func(state)
        log_func(f"<<< [{node_name}] 输出键: {list(result.keys())}")

        return result
    return wrapper

# 使用示例
@debug_node
def my_node(state: State):
    return {"answer": "hello"}

# 开发时启用：LANGGRAPH_DEBUG=1 python app.py
# 生产时关闭：python app.py（默认不输出调试信息）
```

## 工具五：状态差异对比工具

### 问题：状态快照太大，看不出改了什么

`get_state()` 返回完整状态，当状态字段多、值复杂时，肉眼对比两个快照非常痛苦。

### 状态 diff 工具

```python
def diff_states(before: dict, after: dict) -> dict:
    """对比两个状态快照，返回 added/removed/changed/unchanged"""
    added = {k: after[k] for k in after if k not in before}
    removed = {k: before[k] for k in before if k not in after}
    changed = {}
    unchanged = []

    for k in before:
        if k in after:
            if before[k] != after[k]:
                changed[k] = {"before": before[k], "after": after[k]}
            else:
                unchanged.append(k)

    return {
        "added": added,
        "removed": removed,
        "changed": changed,
        "unchanged": unchanged
    }

def print_diff(diff: dict):
    """格式化打印状态差异"""
    for k, v in diff.get("added", {}).items():
        print(f"  [+] 新增 {k} = {repr(v)[:80]}")
    for k, v in diff.get("removed", {}).items():
        print(f"  [-] 删除 {k} = {repr(v)[:80]}")
    for k, info in diff.get("changed", {}).items():
        print(f"  [~] 变化 {k}: {repr(info['before'])[:40]} -> {repr(info['after'])[:40]}")
    if diff.get("unchanged"):
        print(f"  [=] 未变: {diff['unchanged']}")
```

### 配合 get_state_history 使用

```python
from langgraph.checkpoint.memory import InMemorySaver

# 假设图已经运行完毕
config = {"configurable": {"thread_id": "1"}}
history = list(graph.get_state_history(config))

# history[0] 是最新状态，history[-1] 是最早状态
# 逐步对比每次状态变化
for i in range(len(history) - 1):
    newer = history[i]
    older = history[i + 1]

    node_name = newer.metadata.get("source", "unknown")
    print(f"\n--- 步骤: {node_name} ---")

    diff = diff_states(older.values, newer.values)
    print_diff(diff)
```

**输出示例：**
```
--- 步骤: retrieve ---
  [~] 变化 docs: [] -> ['doc_0', 'doc_1', 'doc_2']
  [=] 未变: ['query', 'answer']

--- 步骤: generate ---
  [~] 变化 answer: '' -> '基于 3 篇文档的回答'
  [=] 未变: ['query', 'docs']
```

## 工具六：条件断点

### 在特定条件下暂停执行

有时候 bug 只在特定条件下出现。条件断点让你精准拦截，而不是每次都暂停。

```python
import pdb

class StateValidationError(Exception):
    """状态校验失败"""
    pass

def validate_state(state: dict, node_name: str):
    """通用状态校验——可在任何节点入口调用"""
    if "query" in state and not state["query"]:
        raise StateValidationError(f"[{node_name}] query 字段为空")
    if "docs" in state and len(state.get("docs", [])) > 100:
        print(f"[{node_name}] WARNING: 文档数量异常 ({len(state['docs'])})")

def generate_node(state: State):
    validate_state(state, "generate")

    # pdb 条件断点：只在异常情况下触发交互式调试
    if len(state.get("docs", [])) == 0:
        pdb.set_trace()  # 程序暂停，可检查变量

    return {"answer": f"基于 {len(state['docs'])} 篇文档的回答"}
```

两种思路：`pdb.set_trace()` 适合本地交互式排查；`raise` 自定义异常适合自动化测试中捕获状态问题。

## 最佳实践总结

### 按环境选择工具

| 环境 | 推荐工具 | 原因 |
|------|----------|------|
| 本地开发 | print + 条件断点 | 快速、直接、零配置 |
| 团队开发 | logging + 调试装饰器 | 可配置级别、统一格式 |
| 集成测试 | 状态 diff + get_state_history | 精确追踪每步变化 |
| 生产环境 | StreamWriter + LangSmith | 结构化、可监控、不影响性能 |

### 调试工具组合策略

```
开发阶段 → @debug_node 装饰器 + print
测试阶段 → logging(INFO) + diff_states
生产阶段 → StreamWriter + LangSmith + logging(WARNING)
```

### 三条黄金法则

1. **调试代码要能一键关闭**：用环境变量或装饰器参数控制，别让调试代码污染生产日志
2. **结构化优于纯文本**：StreamWriter 输出 dict 比 print 字符串更容易被程序消费
3. **先用简单工具**：能用 print 解决的问题，不要上 LangSmith。复杂度应该匹配问题的复杂度
