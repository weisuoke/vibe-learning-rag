## 为什么需要循环？

### 现实问题

想象你正在构建一个 AI 代码审查助手：

```python
# ❌ 单次执行：只能审查一轮，发现不了深层问题
def code_review(code):
    issues = llm.find_issues(code)      # 找到 3 个问题
    fixed = llm.fix_issues(code, issues) # 修复了 3 个问题
    return fixed                         # 但修复可能引入新问题！
```

真实场景中，修复一个 bug 可能引入新的 bug。你需要的是**反复审查、反复修正**，直到代码质量达标：

```python
# ✅ 循环迭代：持续改进直到满意
while not quality_ok:
    issues = llm.find_issues(code)
    if not issues:
        break  # 没问题了，退出循环
    code = llm.fix_issues(code, issues)
```

**这就是循环与迭代的价值！**

### AI Agent 的核心模式都依赖循环

循环不是可选的高级特性——它是 AI Agent 的**基础能力**：

| Agent 模式 | 循环行为 | 为什么需要循环 |
|-----------|---------|--------------|
| **ReAct** | 思考 → 行动 → 观察 → 思考... | Agent 需要根据工具返回结果继续推理 |
| **自我修正** | 生成 → 检查 → 修正 → 检查... | 第一次生成的结果往往不完美 |
| **迭代检索** | 检索 → 评估 → 改写查询 → 再检索... | 一次检索可能找不到最佳答案 |
| **多轮对话** | 提问 → 回答 → 追问 → 回答... | 复杂问题需要多轮交互才能解决 |
| **规划执行** | 规划 → 执行 → 评估 → 调整规划... | 计划赶不上变化，需要动态调整 |

### LangGraph 的解决方案

LangGraph 中的循环本质是：**边（edge）指向图中已经存在的节点，形成有向环**。

```python
# 最简单的循环：b 执行完后回到 a
builder.add_edge(START, "a")
builder.add_conditional_edges("a", route)  # 条件决定继续还是结束
builder.add_edge("b", "a")                 # 回边：形成循环
```

与传统的 `while` 循环不同，LangGraph 的循环：
- 每一步都有 **checkpoint**（可以回溯、恢复）
- 有内置的 **递归限制**（防止无限循环）
- 支持 **人工介入**（在循环中暂停等待审批）
- 基于 **图结构**（可视化、可调试）

## 核心概念速览

### 1. 循环边与回路构建 - 四种方式创建循环

**一句话**: LangGraph 提供四种构建循环的机制，底层都归结为 channel 触发

LangGraph 中构建循环有四种方式，但底层机制完全统一——节点写入 `branch:to:{node}` channel 触发目标节点：

```python
# 方式 1: 静态边 + 条件边（最常用）
builder.add_edge("b", "a")                    # 回边
builder.add_conditional_edges("a", route)      # 条件决定是否继续

# 方式 2: 纯条件边
builder.add_conditional_edges("a", lambda s: "b" if need_more(s) else END)

# 方式 3: Command 自路由
def node_a(state):
    if need_more(state):
        return Command(goto="b", update={"count": state["count"] + 1})

# 方式 4: Send 动态迭代
def route(state):
    return [Send("process", {"item": i}) for i in state["items"]]
```

**关键特性**:
- `add_edge()` 不阻止循环，只禁止 `END → X` 和 `X → START`
- 条件边是循环的核心构建方式，`path` 函数在运行时决定下一个节点
- Command 允许节点在返回值中直接指定下一个节点，无需预先定义条件边
- 所有方式最终都通过 `branch:to:{node}` channel 触发节点

**类比**: 就像 JavaScript 中 `while`、`for`、`do-while`、`递归` 都能实现循环，选择哪种取决于场景

---

### 2. 终止条件设计 - 何时停止循环

**一句话**: 条件边返回 `END` 是最常用的终止方式，配合 `RemainingSteps` 实现优雅降级

```python
# 基于业务逻辑的终止
def route(state: State) -> Literal["continue", "__end__"]:
    if state["quality_score"] >= 0.9:  # 质量达标
        return END
    if state["iteration"] >= 5:        # 最大迭代次数
        return END
    return "continue"

# 基于剩余步数的优雅降级
def route(state: State) -> Literal["agent", "__end__"]:
    if state["remaining_steps"] <= 2:  # 快到递归限制了
        return END                      # 优雅退出，而非抛异常
    return "agent"
```

**三种终止策略**:

| 策略 | 方法 | 推荐度 |
|------|------|--------|
| 条件终止 | 路由函数返回 `END` | ⭐⭐⭐⭐⭐ 最常用 |
| 主动监控 | `RemainingSteps` 管理值 | ⭐⭐⭐⭐ 推荐，优雅降级 |
| 被动限制 | `recursion_limit` 配置 | ⭐⭐⭐ 兜底保护 |

---

### 3. 递归限制与安全机制 - 防止无限循环

**一句话**: `recursion_limit` 是防止无限循环的安全阀，默认 25 步，超出抛出 `GraphRecursionError`

```python
# 设置递归限制
result = graph.invoke(
    {"messages": [("user", "帮我写一篇文章")]},
    config={"recursion_limit": 50}  # 最多 50 个超步
)

# 使用 RemainingSteps 主动感知（推荐）
from langgraph.managed.is_last_step import RemainingSteps

class State(TypedDict):
    messages: Annotated[list, operator.add]
    remaining_steps: RemainingSteps  # 自动注入剩余步数

def agent(state: State):
    if state["remaining_steps"] <= 2:
        return {"messages": ["接近限制，返回当前最佳结果"]}
    # 正常处理...
```

**关键机制**:
- `stop = step + recursion_limit + 1`，实际允许 `limit + 1` 个超步
- `IsLastStep`: 布尔值，判断是否为最后一步
- `RemainingSteps`: 整数，返回剩余步数
- 超出限制时抛出 `GraphRecursionError`（继承自 Python `RecursionError`）

---

### 4. Command 自循环模式 - 节点内部自路由

**一句话**: Command 让节点在返回值中同时完成状态更新和路由决策，无需预定义条件边

```python
from langgraph.types import Command

def reviewer(state: State) -> Command[Literal["writer", "__end__"]]:
    review = llm.review(state["draft"])
    if review["needs_revision"]:
        return Command(
            update={"feedback": review["comments"]},
            goto="writer"  # 路由回写作节点
        )
    return Command(
        update={"final_draft": state["draft"]},
        goto=END
    )
```

**关键特性**:
- 同时完成状态更新 + 路由决策（条件边只能路由，不能更新状态）
- 必须添加返回类型注解 `Command[Literal["node_a", "node_b"]]`
- Command 不会覆盖静态边（两者会同时生效）
- 适合多智能体交接场景

**何时用 Command vs 条件边？**
- **Command**: 需要同时更新状态并路由（如 Agent 交接）
- **条件边**: 只需要条件路由，不需要更新状态

---

### 5. Send 动态迭代 - Map-Reduce 模式

**一句话**: Send 在条件边或 Command 中向同一节点发送多个不同输入，实现并行 map 操作

```python
from langgraph.types import Send

# 动态创建多个并行任务
def fan_out(state: State):
    return [
        Send("process_chunk", {"chunk": chunk, "index": i})
        for i, chunk in enumerate(state["chunks"])
    ]

builder.add_conditional_edges("splitter", fan_out)
builder.add_edge("process_chunk", "aggregator")
```

**关键特性**:
- 运行时动态确定迭代次数（不需要编译时确定）
- 每个 Send 可以携带独立的输入状态
- 多个 Send 到同一节点会并行执行
- 结果通过 Reducer 自动合并

**类比**: 就像 Python 的 `map()` 函数，对列表中每个元素应用同一个函数

---

### 6. 迭代状态累积 - 跨迭代的状态管理

**一句话**: 使用 Reducer（如 `operator.add`）在每次迭代中累积状态，而非覆盖

```python
import operator
from typing import Annotated

class State(TypedDict):
    # 每次迭代追加，不覆盖
    messages: Annotated[list, operator.add]
    # 每次迭代覆盖（保留最新值）
    current_draft: str
    # 迭代计数器
    iteration: int
```

**关键特性**:
- `Annotated[list, operator.add]`: 追加模式，保留所有历史
- 普通字段: 覆盖模式，只保留最新值
- 每次迭代都会保存 checkpoint，支持回溯
- 状态累积是 ReAct 等模式的基础（消息历史不断增长）

## 实战场景预览

### 场景 1: ReAct Agent 循环

**问题**: Agent 需要反复调用工具并根据结果推理

**解决方案**: 条件边控制的 Agent-Tool 循环

```python
def should_continue(state: State) -> Literal["tools", "__end__"]:
    last_message = state["messages"][-1]
    if last_message.tool_calls:
        return "tools"     # 有工具调用，继续循环
    return END             # 没有工具调用，结束

builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", should_continue)
builder.add_edge("tools", "agent")  # 回边：工具执行后回到 Agent
```

**适用场景**: 通用 AI Agent、智能助手、自动化工作流

---

### 场景 2: 自我修正循环

**问题**: LLM 生成的代码/文本需要反复检查和修正

**解决方案**: 生成-检查-修正的迭代循环

```python
def check_quality(state: State) -> Literal["revise", "__end__"]:
    if state["quality_score"] < 0.9 and state["iteration"] < 5:
        return "revise"
    return END

builder.add_edge(START, "generate")
builder.add_edge("generate", "evaluate")
builder.add_conditional_edges("evaluate", check_quality)
builder.add_edge("revise", "evaluate")  # 修正后重新评估
```

**适用场景**: 代码生成、文章写作、数据清洗

---

### 场景 3: 迭代检索增强（Iterative RAG）

**问题**: 一次检索可能找不到最佳答案，需要改写查询重新检索

**解决方案**: 检索-评估-改写的迭代循环

```python
def assess_retrieval(state: State) -> Literal["rewrite_query", "generate"]:
    if state["relevance_score"] < 0.7 and state["retrieval_count"] < 3:
        return "rewrite_query"  # 改写查询，重新检索
    return "generate"           # 检索质量足够，生成答案

builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "assess")
builder.add_conditional_edges("assess", assess_retrieval)
builder.add_edge("rewrite_query", "retrieve")  # 回到检索
builder.add_edge("generate", END)
```

**适用场景**: 知识库问答、文档检索、智能搜索

---

### 场景 4: 多轮规划执行

**问题**: 复杂任务需要分步规划、逐步执行、动态调整

**解决方案**: 规划-执行-评估的迭代循环

```python
def evaluate_progress(state: State) -> Literal["replan", "execute", "__end__"]:
    if state["all_tasks_done"]:
        return END
    if state["need_replan"]:
        return "replan"
    return "execute"

builder.add_edge(START, "plan")
builder.add_edge("plan", "execute")
builder.add_edge("execute", "evaluate")
builder.add_conditional_edges("evaluate", evaluate_progress)
builder.add_edge("replan", "execute")
```

**适用场景**: 项目管理 Agent、研究助手、自动化测试

---

### 场景 5: 带安全降级的循环

**问题**: 循环可能因为 LLM 幻觉或逻辑错误而无法终止

**解决方案**: 使用 `RemainingSteps` 实现优雅降级

```python
from langgraph.managed.is_last_step import RemainingSteps

class State(TypedDict):
    messages: Annotated[list, operator.add]
    remaining_steps: RemainingSteps

def agent(state: State):
    if state["remaining_steps"] <= 2:
        return {"messages": ["接近步数限制，返回当前最佳结果"]}
    # 正常 Agent 逻辑...

def route(state: State) -> Literal["agent", "fallback", "__end__"]:
    if state["remaining_steps"] <= 2:
        return "fallback"  # 降级到 fallback 节点
    if need_continue(state):
        return "agent"
    return END
```

**适用场景**: 生产环境 Agent、高可靠性系统、用户面向的应用

## 快速开始

### 5 分钟快速体验

```python
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# 1. 定义状态
class State(TypedDict):
    aggregate: Annotated[list, operator.add]

# 2. 定义节点
def node_a(state: State):
    print(f'Node A sees {state["aggregate"]}')
    return {"aggregate": ["A"]}

def node_b(state: State):
    print(f'Node B sees {state["aggregate"]}')
    return {"aggregate": ["B"]}

# 3. 定义终止条件
def route(state: State) -> Literal["b", "__end__"]:
    if len(state["aggregate"]) < 7:
        return "b"       # 继续循环
    return END           # 终止

# 4. 构建图
builder = StateGraph(State)
builder.add_node("a", node_a)
builder.add_node("b", node_b)

builder.add_edge(START, "a")
builder.add_conditional_edges("a", route)
builder.add_edge("b", "a")  # 回边：形成循环！

# 5. 编译并运行
graph = builder.compile()
result = graph.invoke({"aggregate": []})
print(result)
# {'aggregate': ['A', 'B', 'A', 'B', 'A', 'B', 'A']}
```

**运行结果**: A 和 B 交替执行，直到列表长度达到 7 时终止。

## 关键要点总结

1. **循环是 AI Agent 的基础能力**，ReAct、自我修正、迭代检索都依赖循环
2. **四种构建方式**：静态边+条件边、纯条件边、Command、Send
3. **底层统一机制**：所有循环都通过 `branch:to:{node}` channel 触发
4. **三种终止策略**：条件终止、主动监控（RemainingSteps）、被动限制（recursion_limit）
5. **递归限制是安全阀**：默认 25 步，超出抛出 `GraphRecursionError`
6. **状态累积**：使用 Reducer 在迭代间保留历史，使用普通字段保留最新值
