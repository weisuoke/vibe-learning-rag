## 问题1："请介绍 LangChain 1.0 的 create_agent，它和旧版 AgentExecutor 有什么区别？"

### 普通回答（不出彩）

"create_agent 是 LangChain 1.0 新的 Agent 创建函数，取代了 AgentExecutor。它更简单，支持 Middleware。"

**为什么不出彩：** 只说了"是什么"，没说"为什么"和"怎么不同"。面试官听完会追问："具体哪里不同？为什么要换？"

### 出彩回答（推荐）

> **create_agent 的本质是架构范式的转变，不仅仅是 API 的更新。**
>
> **1. 设计理念层面：**
> 旧版 AgentExecutor 是"Agent + Executor"分离模式——你先创建一个 Agent（决策逻辑），再用 Executor 包装执行。这导致了两个问题：一是需要在 6+ 种 Agent 类型中选择（`create_react_agent`、`create_openai_functions_agent`、`create_tool_calling_agent` 等），二是自定义需要通过参数堆叠（`max_iterations`、`handle_parsing_errors`、`early_stopping_method` 等）。
>
> `create_agent` 采用"一个函数，一个图"的设计——`model` + `tools` + `middleware` 输入，`CompiledStateGraph` 输出。所有自定义通过 Middleware 组合实现，而非参数堆叠。
>
> **2. 技术实现层面：**
> 旧版基于 Chain 的 while 循环，新版基于 LangGraph 的 StateGraph。这意味着 `create_agent` 天然支持状态持久化（checkpointer）、中断恢复（interrupt）、子图嵌套（name 参数）等生产级特性。
>
> **3. 扩展性层面：**
> Middleware 系统借鉴了 Web 服务器中间件模式，提供 6 种钩子（`before_agent`、`before_model`、`after_model`、`after_agent`、`wrap_model_call`、`wrap_tool_call`），支持装饰器和类两种定义方式。19 个内置 Middleware 覆盖了重试、限流、人工审核、摘要等常见场景。
>
> **在实际工作中：** 我们团队从 AgentExecutor 迁移到 create_agent 后，Agent 的可维护性显著提升——每个自定义逻辑都是独立的 Middleware，可以单独测试和复用。

### 为什么这个回答出彩？

1. 从三个层面（设计理念、技术实现、扩展性）展开，展示系统性思维
2. 提到了具体的技术细节（StateGraph、6 种钩子、19 个内置 Middleware）
3. 联系了实际工作经验，不是纸上谈兵

### 面试官可能的追问

**追问 A："CompiledStateGraph 是什么？为什么返回图而不是 Agent 对象？"**

> `CompiledStateGraph` 是 LangGraph 的编译状态图。返回图而非对象有三个好处：
> 1. 可以直接作为子图嵌入更大的 LangGraph 工作流（多 Agent 协作）
> 2. 天然获得 LangGraph 的所有能力（持久化、中断、流式输出）
> 3. 统一调用接口：`.invoke()`、`.stream()`、`.ainvoke()`

**追问 B："旧版代码怎么迁移？"**

> LangChain 提供了 `langchain-classic` 包做向后兼容。迁移的核心变化是：
> - `AgentExecutor` 的参数 → 对应的 Middleware
> - `max_iterations` → `ModelCallLimitMiddleware`
> - `handle_parsing_errors` → `ModelRetryMiddleware`
> - `memory` → `checkpointer` 参数
> - 输入从 `{"input": "问题"}` 变为 `{"messages": [...]}`

## 问题2："LangChain 的 Middleware 是什么？和 LangGraph 的自定义节点有什么区别？"

### 普通回答（不出彩）

"Middleware 是 LangChain 1.0 的新特性，可以在 Agent 循环中添加自定义逻辑。"

**为什么不出彩：** 没有说清楚 Middleware 解决了什么问题，也没有和 LangGraph 做对比。

### 出彩回答（推荐）

> **Middleware 解决的核心问题是"上下文工程的可组合性"。**
>
> LangChain 官方博客明确指出：Agent 难以带入生产的根本原因是上下文工程（Context Engineering）——模型的输入决定输出质量，但简单的 Agent 循环不提供足够的控制点。
>
> 过去两年 LangChain 团队不断给 Agent 加参数：动态 prompt、pre/post model hook、动态模型切换……结果参数越来越多，参数之间有依赖关系，难以组合。Middleware 就是对这个问题的终极解答——不再加参数，而是加"层"。
>
> **Middleware vs LangGraph 自定义节点：**
>
> | 维度 | Middleware | LangGraph 自定义节点 |
> |------|-----------|---------------------|
> | 抽象层级 | 高层（声明式） | 低层（命令式） |
> | 使用场景 | 标准 Agent 循环的自定义 | 完全自定义的工作流 |
> | 学习成本 | 低（6 种钩子） | 高（图论概念） |
> | 灵活性 | 中等（在 Agent 循环内） | 极高（任意图结构） |
> | 执行模型 | 洋葱模型（进入顺序，返回逆序） | 图的拓扑排序 |
> | 复用性 | 高（独立 Middleware 可跨项目复用） | 低（节点通常和业务耦合） |
>
> **关键洞察：** `create_agent` 底层就是 LangGraph，所以 Middleware 本质上是对 LangGraph 节点的高层封装。当 Middleware 不够用时，可以无缝降级到 LangGraph。
>
> **实际选择建议：**
> - 80% 的场景用 `create_agent` + Middleware
> - 需要非标准循环（如多 Agent 协作、条件分支、DAG 工作流）时用 LangGraph
> - 两者可以混用：`create_agent` 返回的图可以作为 LangGraph 的子图

### 为什么这个回答出彩？

1. 引用了官方博客的核心观点（上下文工程），展示你读过一手资料
2. 用对比表清晰展示区别，结构化表达
3. 给出了实际选择建议，展示工程判断力

### 面试官可能的追问

**追问 A："Middleware 的 6 种钩子分别在什么时候触发？"**

> 6 种钩子覆盖 Agent 循环的完整生命周期：
>
> ```
> Agent 开始
>   │
>   ▼
> before_agent()          ← Agent 循环开始前（只执行一次）
>   │
>   ▼
> ┌─ before_model()       ← 每轮模型调用前
> │  wrap_model_call()    ← 包装模型调用（可重试、降级）
> │  after_model()        ← 每轮模型调用后
> │  │
> │  ▼
> │  wrap_tool_call()     ← 包装工具调用（可拦截、修改）
> │  │
> └──┘ （循环直到模型不再调用工具）
>   │
>   ▼
> after_agent()           ← Agent 循环结束后（只执行一次）
> ```
>
> 其中 `before_agent` / `after_agent` 只在整个 Agent 生命周期的首尾各执行一次，而 `before_model` / `after_model` / `wrap_model_call` / `wrap_tool_call` 在每轮循环中都会执行。

**追问 B："多个 Middleware 的执行顺序是什么？"**

> 遵循洋葱模型——进入时按列表顺序执行，返回时逆序执行：
>
> ```python
> middleware=[A, B, C]
>
> # 执行顺序：
> # A.before_model → B.before_model → C.before_model
> #                    模型调用
> # C.after_model → B.after_model → A.after_model
> ```
>
> 这和 Express.js 中间件、Koa 洋葱模型是同一个设计模式。

## 问题3（加分题）："如果让你设计一个生产级 Agent，你会怎么配置 create_agent？"

### 普通回答（不出彩）

"我会用 create_agent 传入模型和工具，加一些 Middleware 就行了。"

**为什么不出彩：** 太笼统，没有展示生产环境的思考深度。

### 出彩回答（推荐）

> 我会从六个维度来设计：模型选择、工具设计、Middleware 栈、持久化、错误处理、可观测性。
>
> **以一个企业知识库问答 Agent 为例：**

```python
from langchain.agents import create_agent
from langchain.agents.middleware import (
    ModelRetryMiddleware,
    ModelFallbackMiddleware,
    ModelCallLimitMiddleware,
    ToolCallLimitMiddleware,
    SummarizationMiddleware,
    HumanInTheLoopMiddleware,
    PIIMiddleware,
)
from langchain_core.tools import tool
from langgraph.checkpoint.postgres import PostgresSaver

# ===== 1. 模型选择 =====
# 主模型用 GPT-4o（推理能力强），备用模型用 GPT-4o-mini（成本低、稳定）
# 用字符串标识符，create_agent 内部通过 init_chat_model 自动初始化

# ===== 2. 工具设计 =====
@tool
def search_knowledge_base(query: str) -> str:
    """搜索企业知识库，返回最相关的文档片段"""
    # 实际实现：向量检索 + 重排序
    ...

@tool
def query_database(sql: str) -> str:
    """执行只读 SQL 查询，获取业务数据"""
    # 实际实现：只允许 SELECT，有超时限制
    ...

@tool
def create_ticket(title: str, description: str) -> str:
    """创建工单（需要人工审批）"""
    ...

# ===== 3. 组装生产级 Agent =====
agent = create_agent(
    # --- 模型 ---
    model="openai:gpt-4o",

    # --- 工具 ---
    tools=[search_knowledge_base, query_database, create_ticket],

    # --- System Prompt ---
    system_prompt="""你是企业知识库助手。规则：
    1. 回答必须基于知识库检索结果，不要编造
    2. 涉及数据查询时使用 SQL 工具
    3. 用户要求创建工单时，先确认信息再创建
    4. 用中文回答，语气专业友好""",

    # --- Middleware 栈（顺序很重要） ---
    middleware=[
        # 第一层：PII 脱敏（最外层，确保敏感信息不泄露）
        PIIMiddleware(),

        # 第二层：模型稳定性
        ModelRetryMiddleware(max_retries=2),
        ModelFallbackMiddleware(fallback_model="openai:gpt-4o-mini"),

        # 第三层：成本控制
        ModelCallLimitMiddleware(max_calls=15),
        ToolCallLimitMiddleware(max_calls=30),

        # 第四层：长对话优化
        SummarizationMiddleware(
            max_messages=50,
            model="openai:gpt-4o-mini",  # 用便宜模型做摘要
        ),

        # 第五层：高风险操作审批
        HumanInTheLoopMiddleware(
            tools=[create_ticket],  # 只拦截创建工单
        ),
    ],

    # --- 持久化 ---
    checkpointer=PostgresSaver.from_conn_string(
        "postgresql://user:pass@localhost:5432/agent_db"
    ),

    # --- 调试与标识 ---
    debug=False,
    name="knowledge-assistant",  # LangSmith 中的标识
)
```

> **六个维度的设计考量：**
>
> **维度 1 - 模型选择：**
> 主模型选 GPT-4o，推理能力强，适合复杂问答。备用模型选 GPT-4o-mini，成本低且稳定，作为降级方案。用字符串标识符 `"openai:gpt-4o"` 而非手动初始化，代码更简洁。
>
> **维度 2 - 工具设计：**
> 工具按风险等级分类——只读工具（search、query）不需要审批，写入工具（create_ticket）需要人工确认。工具的 docstring 要写清楚，因为 LLM 靠它来决定调用哪个工具。
>
> **维度 3 - Middleware 栈：**
> 顺序遵循"安全 → 稳定 → 成本 → 优化 → 审批"的原则。PII 脱敏放最外层确保敏感信息不泄露；重试和降级保证可用性；调用限制控制成本；摘要优化长对话；人工审批放最内层只拦截高风险操作。
>
> **维度 4 - 持久化：**
> 生产环境用 `PostgresSaver` 而非 `InMemorySaver`。`thread_id` 用 `user_id + session_id` 组合生成，确保唯一性。
>
> **维度 5 - 错误处理：**
> 三层防护——`ModelRetryMiddleware` 处理偶发 API 错误，`ModelFallbackMiddleware` 处理主模型不可用，`ModelCallLimitMiddleware` 防止无限循环。
>
> **维度 6 - 可观测性：**
> `name` 参数在 LangSmith 中标识这个 Agent。生产环境接入 LangSmith 后，可以看到每轮循环的模型输入输出、工具调用参数和结果、Middleware 的执行情况、Token 消耗和延迟。

### 为什么这个回答出彩？

1. 从六个维度系统性展开，展示架构设计能力
2. 代码完整可运行，不是伪代码
3. 解释了 Middleware 顺序的设计原则，展示工程经验
4. 考虑了成本、安全、可用性等生产环境的真实问题

### 面试官可能的追问

**追问 A："Middleware 的顺序为什么这样排？换一下行不行？"**

> 顺序很重要，因为洋葱模型意味着外层 Middleware 先执行。
>
> - `PIIMiddleware` 必须在最外层：确保任何进入模型的数据都已脱敏
> - `ModelRetryMiddleware` 要在 `ModelFallbackMiddleware` 前面：先重试，重试全部失败再降级
> - `SummarizationMiddleware` 要在 `ModelCallLimitMiddleware` 后面：因为摘要本身会消耗一次模型调用，如果放前面可能提前触发限制
> - `HumanInTheLoopMiddleware` 放最内层：只在真正要执行工具时才拦截，避免不必要的人工干预

**追问 B："如果要支持多租户怎么办？"**

> 两种方案：
> 1. **Partition 模式**：每个租户一个 `thread_id` 前缀，通过 `checkpointer` 隔离对话状态
> 2. **Context 模式**：用 `context_schema` 传入租户信息，Middleware 根据租户动态调整行为（如不同租户用不同模型、不同工具集）
>
> ```python
> from typing import TypedDict
>
> class TenantContext(TypedDict):
>     tenant_id: str
>     plan: str  # "free" | "pro" | "enterprise"
>
> agent = create_agent(
>     model="openai:gpt-4o",
>     tools=[...],
>     context_schema=TenantContext,
>     middleware=[TenantRoutingMiddleware()],
> )
>
> # 调用时传入租户上下文
> result = agent.invoke(
>     {"messages": [...]},
>     config={"configurable": {"thread_id": f"tenant-A-session-001"}},
>     context={"tenant_id": "A", "plan": "enterprise"},
> )
> ```

## 面试技巧总结

### 回答框架：WHY → WHAT → HOW

| 步骤 | 内容 | 示例 |
|------|------|------|
| WHY | 为什么需要这个东西？ | "旧版参数爆炸，难以组合" |
| WHAT | 它是什么？核心设计是什么？ | "一个工厂函数，返回 CompiledStateGraph" |
| HOW | 怎么用？有什么最佳实践？ | "model + tools + middleware，按需组合" |

### 关键数字速记

面试中提到具体数字会显得你对技术有深入了解：

| 数字 | 含义 |
|------|------|
| **1** | 1 个统一入口（`create_agent`），取代 10+ 种旧函数 |
| **6** | 6 种 Middleware 钩子（4 个生命周期 + 2 个包装器） |
| **19** | 19 个内置 Middleware（开箱即用） |
| **2** | 2 种 Middleware 定义方式（装饰器 + 类） |
| **3** | 3 种结构化输出策略（ToolStrategy、ProviderStrategy、AutoStrategy） |

### 加分话术

**展示一手资料阅读：**
- "LangChain 官方博客在 2025 年 9 月的 Agent Middleware 文章中提到……"
- "我看过 `factory.py` 的源码，create_agent 内部是 1800+ 行，核心是构建 LangGraph StateGraph……"

**展示工程判断力：**
- "80% 的场景用 create_agent + Middleware 就够了，只有需要非标准循环时才降级到 LangGraph"
- "Middleware 不是越多越好，生产环境的最小配置是 ModelRetryMiddleware + ModelCallLimitMiddleware"

**展示迁移经验：**
- "从 AgentExecutor 迁移到 create_agent，核心变化是输入格式从 `{input: ...}` 变为 `{messages: [...]}`"
- "旧版的 `max_iterations` 对应新版的 `ModelCallLimitMiddleware`"

### 常见陷阱

| 陷阱 | 正确说法 |
|------|----------|
| "create_agent 返回一个 Agent 对象" | 返回 `CompiledStateGraph`，是 LangGraph 的编译状态图 |
| "Middleware 只在开始和结束时执行" | `before_model` / `after_model` 在每轮循环中都执行 |
| "create_agent 是 AgentExecutor 的简单替代" | 是架构级重新设计，从 Chain 到 Graph |
| "所有内置 Middleware 都应该加上" | 按需选择，少即是多 |
| "system_prompt 可以在 invoke 时动态修改" | 编译时固定，动态修改需要用 Middleware |
