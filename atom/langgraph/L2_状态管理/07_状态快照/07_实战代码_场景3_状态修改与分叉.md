# 实战代码 - 场景3：状态修改与分叉

## 场景说明

本场景演示 `update_state()` 修改状态并从历史分叉出新的执行路径。你将看到：

1. 执行图得到初始结果
2. 获取历史快照，选择一个中间步骤
3. 使用 `update_state()` 修改该快照的状态
4. 从修改后的状态继续执行（分叉）
5. 对比原始执行和分叉执行的结果
6. 理解 Reducer 规则在 `update_state()` 中的作用

`update_state()` 是时间旅行的核心操作——它不会修改已有的 checkpoint，而是基于指定的 checkpoint 创建一个全新的 checkpoint。

[来源: reference/context7_langgraph_01.md | LangGraph 官方文档]

## 完整代码

```python
"""
LangGraph 状态快照实战 - 场景3：状态修改与分叉
演示：update_state() 修改状态 + 从历史分叉 + Reducer 规则

核心知识点：
- graph.update_state(config, values) 创建新 checkpoint
- as_node 参数指定"以哪个节点的身份"写入
- Reducer 规则：有 reducer 的字段追加，无 reducer 的字段覆盖
- 从历史快照分叉并继续执行
- 原始执行路径不受影响

运行环境：Python 3.13+, langgraph
安装依赖：uv add langgraph
"""

from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver


# ============================================================
# 1. 定义状态 —— 注意哪些字段有 Reducer
# ============================================================

class DocState(TypedDict):
    """文档处理管道状态

    messages 有 add reducer → update_state 时会追加
    其他字段无 reducer → update_state 时会覆盖
    """
    query: str                             # 覆盖模式
    messages: Annotated[list[str], add]    # 追加模式
    docs: list[str]                        # 覆盖模式
    score: float                           # 覆盖模式


# ============================================================
# 2. 定义节点
# ============================================================

def retrieve(state: DocState) -> dict:
    """检索文档 —— 模拟返回质量一般的结果"""
    query = state["query"]
    # 模拟检索：返回一些相关度不高的文档
    docs = [
        f"文档A: {query}的基础介绍（相关度低）",
        f"文档B: {query}的历史背景（相关度中）",
        f"文档C: {query}的技术细节（相关度高）",
    ]
    return {
        "messages": [f"检索完成: 找到 {len(docs)} 篇文档"],
        "docs": docs,
        "score": 0.6,  # 整体相关度评分
    }


def generate(state: DocState) -> dict:
    """生成回答 —— 基于检索到的文档"""
    docs = state["docs"]
    score = state["score"]
    # 模拟生成：质量取决于文档质量
    if score >= 0.8:
        answer = f"高质量回答（基于 {len(docs)} 篇高相关文档）"
    elif score >= 0.5:
        answer = f"中等质量回答（基于 {len(docs)} 篇文档，相关度 {score}）"
    else:
        answer = f"低质量回答（文档相关度不足 {score}）"
    return {
        "messages": [f"生成完成: {answer}"],
    }


# ============================================================
# 3. 构建并执行图
# ============================================================

builder = StateGraph(DocState)
builder.add_node("retrieve", retrieve)
builder.add_node("generate", generate)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

memory = InMemorySaver()
graph = builder.compile(checkpointer=memory)

config = {"configurable": {"thread_id": "fork-demo-001"}}

print("=" * 60)
print("=== 第一步：原始执行 ===")
print("=" * 60)

result = graph.invoke(
    {"query": "LangGraph 状态管理", "messages": [], "docs": [], "score": 0.0},
    config,
)

print(f"原始结果:")
print(f"  messages: {result['messages']}")
print(f"  docs: {result['docs']}")
print(f"  score: {result['score']}")


# ============================================================
# 4. 查看历史，找到检索步骤的快照
# ============================================================

print()
print("=" * 60)
print("=== 第二步：查看历史，定位检索步骤 ===")
print("=" * 60)

all_states = list(graph.get_state_history(config))
print(f"\n总快照数: {len(all_states)}")

# 找到 retrieve 节点执行后的快照（writes 中包含 "retrieve"）
retrieve_snapshot = None
for state in all_states:
    writes = state.metadata.get("writes", {})
    if "retrieve" in writes:
        retrieve_snapshot = state
        break

if retrieve_snapshot:
    print(f"\n找到检索步骤快照:")
    print(f"  step: {retrieve_snapshot.metadata.get('step')}")
    print(f"  docs: {retrieve_snapshot.values['docs']}")
    print(f"  score: {retrieve_snapshot.values['score']}")
    print(f"  next: {retrieve_snapshot.next}")
    print(f"  checkpoint_id: {retrieve_snapshot.config['configurable']['checkpoint_id'][:12]}...")


# ============================================================
# 5. update_state —— 修改检索结果（创建分叉）
# ============================================================

print()
print("=" * 60)
print("=== 第三步：修改检索结果（update_state） ===")
print("=" * 60)

# 用更好的文档替换原始检索结果
better_docs = [
    "文档X: LangGraph 状态管理核心原理（高相关度）",
    "文档Y: StateGraph 与 Reducer 详解（高相关度）",
    "文档Z: Checkpoint 持久化实战（高相关度）",
]

# update_state 的关键参数：
# - config: 基于哪个快照修改（这里用检索步骤的快照）
# - values: 要修改的字段值
# - as_node: 以哪个节点的身份写入（影响 next 的计算）
new_config = graph.update_state(
    retrieve_snapshot.config,  # 基于检索步骤的快照
    values={
        "docs": better_docs,           # 覆盖模式：直接替换
        "score": 0.95,                 # 覆盖模式：直接替换
        "messages": ["[手动修改] 替换为高质量文档"],  # 追加模式：会追加到列表
    },
    as_node="retrieve",  # 以 retrieve 节点的身份写入
)

print(f"update_state 返回的新 config:")
print(f"  thread_id: {new_config['configurable']['thread_id']}")
print(f"  checkpoint_id: {new_config['configurable']['checkpoint_id'][:12]}...")

# 验证修改后的状态
modified_snapshot = graph.get_state(new_config)
print(f"\n修改后的快照:")
print(f"  docs: {modified_snapshot.values['docs']}")
print(f"  score: {modified_snapshot.values['score']}")
print(f"  messages: {modified_snapshot.values['messages']}")
print(f"  next: {modified_snapshot.next}")
print(f"  metadata.source: {modified_snapshot.metadata.get('source')}")


# ============================================================
# 6. 从分叉点继续执行
# ============================================================

print()
print("=" * 60)
print("=== 第四步：从分叉点继续执行 ===")
print("=" * 60)

# 从修改后的状态继续执行
# invoke(None, config) 表示不提供新输入，从 checkpoint 恢复执行
forked_result = graph.invoke(None, config=new_config)

print(f"分叉执行结果:")
print(f"  messages: {forked_result['messages']}")
print(f"  docs: {forked_result['docs']}")
print(f"  score: {forked_result['score']}")


# ============================================================
# 7. 对比原始执行和分叉执行
# ============================================================

print()
print("=" * 60)
print("=== 第五步：对比原始 vs 分叉 ===")
print("=" * 60)

print(f"\n{'字段':<12} {'原始执行':<40} {'分叉执行'}")
print("-" * 90)
print(f"{'score':<12} {result['score']:<40} {forked_result['score']}")
print(f"{'docs数量':<12} {len(result['docs']):<40} {len(forked_result['docs'])}")
print(f"{'最后消息':<12} {result['messages'][-1]:<40} {forked_result['messages'][-1]}")


# ============================================================
# 8. Reducer 规则验证
# ============================================================

print()
print("=" * 60)
print("=== 第六步：Reducer 规则在 update_state 中的作用 ===")
print("=" * 60)

# 重新获取修改后的快照来验证 Reducer 行为
print("\n--- messages 字段（有 add reducer → 追加） ---")
print(f"  修改前 messages 数量: {len(retrieve_snapshot.values['messages'])}")
print(f"  修改后 messages 数量: {len(modified_snapshot.values['messages'])}")
print(f"  修改前: {retrieve_snapshot.values['messages']}")
print(f"  修改后: {modified_snapshot.values['messages']}")
print(f"  说明: '[手动修改]...' 被追加到了列表末尾")

print("\n--- docs 字段（无 reducer → 覆盖） ---")
print(f"  修改前: {retrieve_snapshot.values['docs'][0][:30]}...")
print(f"  修改后: {modified_snapshot.values['docs'][0][:30]}...")
print(f"  说明: 整个列表被替换为新的文档列表")

print("\n--- score 字段（无 reducer → 覆盖） ---")
print(f"  修改前: {retrieve_snapshot.values['score']}")
print(f"  修改后: {modified_snapshot.values['score']}")
print(f"  说明: 值被直接覆盖")


# ============================================================
# 9. as_node 参数的作用
# ============================================================

print()
print("=" * 60)
print("=== 第七步：as_node 参数的作用 ===")
print("=" * 60)

# as_node 决定了 update_state 创建的 checkpoint 的 next 字段
# 如果 as_node="retrieve"，next 就是 retrieve 之后的节点（generate）
# 如果 as_node="generate"，next 就是 generate 之后的节点（END → 空元组）

print("\n--- as_node='retrieve' 的效果 ---")
config_test = {"configurable": {"thread_id": "fork-demo-002"}}
graph.invoke(
    {"query": "测试", "messages": [], "docs": [], "score": 0.0},
    config_test,
)
states = list(graph.get_state_history(config_test))
retrieve_snap = None
for s in states:
    writes = s.metadata.get("writes", {})
    if "retrieve" in writes:
        retrieve_snap = s
        break

if retrieve_snap:
    # as_node="retrieve" → next 应该是 ("generate",)
    new_cfg_a = graph.update_state(
        retrieve_snap.config,
        values={"score": 0.99},
        as_node="retrieve",
    )
    snap_a = graph.get_state(new_cfg_a)
    print(f"  as_node='retrieve' → next={snap_a.next}")

    # as_node="generate" → next 应该是 ()（END）
    new_cfg_b = graph.update_state(
        retrieve_snap.config,
        values={"score": 0.99},
        as_node="generate",
    )
    snap_b = graph.get_state(new_cfg_b)
    print(f"  as_node='generate' → next={snap_b.next}")

    print(f"\n  说明: as_node 决定了从哪个节点之后继续执行")
    print(f"  - as_node='retrieve' → 下一步执行 generate")
    print(f"  - as_node='generate' → 图已结束，不会继续执行")


# ============================================================
# 10. 验证原始执行不受影响
# ============================================================

print()
print("=" * 60)
print("=== 第八步：验证原始执行不受影响 ===")
print("=" * 60)

# 用原始 config 获取最新快照
original_latest = graph.get_state(config)
print(f"\n原始线程 (fork-demo-001) 的最新快照:")
print(f"  score: {original_latest.values['score']}")
print(f"  messages 数量: {len(original_latest.values['messages'])}")

# 查看原始线程的历史长度（分叉会增加快照）
original_history = list(graph.get_state_history(config))
print(f"  历史快照总数: {len(original_history)}")
print(f"\n  说明: 分叉操作在同一个 thread 下创建了新的 checkpoint，")
print(f"  但原始执行路径的快照仍然存在，可以通过 checkpoint_id 访问")
```

## 关键知识点解析

### 1. update_state 的工作原理

```
update_state(config, values, as_node)
  ↓
1. 从 config 指定的 checkpoint 加载状态
2. 按 Reducer 规则合并 values
   - 有 reducer 的字段 → 调用 reducer（如 add → 追加）
   - 无 reducer 的字段 → 直接覆盖
3. 创建新的 checkpoint（不修改原有的）
4. 根据 as_node 计算 next 字段
5. 返回新 checkpoint 的 config
```

### 2. Reducer 规则对照表

| 字段定义 | update_state 行为 | 示例 |
|---------|------------------|------|
| `messages: Annotated[list, add]` | 追加 | `["a"] + ["b"] → ["a", "b"]` |
| `docs: list[str]` | 覆盖 | `["old"] → ["new"]` |
| `score: float` | 覆盖 | `0.6 → 0.95` |
| `count: Annotated[int, add]` | 相加 | `3 + 2 → 5` |

### 3. as_node 参数的选择策略

| 场景 | as_node 值 | 效果 |
|------|-----------|------|
| 修改检索结果，重新生成 | `"retrieve"` | next=("generate",)，继续执行 generate |
| 直接修改最终结果 | `"generate"` | next=()，不再执行任何节点 |
| 修改输入，重新执行全部 | `"__start__"` | next=("retrieve",)，从头开始 |

### 4. 分叉 vs 修改

`update_state()` 不是"修改"已有的 checkpoint，而是"分叉"出一个新的：

```
原始链: [input] → [retrieve] → [generate] → [END]
                       ↑
                  update_state 在这里分叉
                       ↓
分叉链: [input] → [retrieve] → [modified] → [generate'] → [END']
```

原始链的所有 checkpoint 仍然存在，可以通过 checkpoint_id 访问。

## 在 RAG 系统中的应用

状态修改与分叉在 RAG 系统中的典型用途：

- **检索结果修正**：当自动检索的文档不够好时，手动替换为更好的文档，重新生成回答
- **参数调优**：修改 temperature、top_k 等参数，对比不同参数下的生成效果
- **人工干预**：在 Human-in-the-loop 场景中，审核员修改中间状态后继续执行
- **A/B 测试**：从同一个检索结果分叉，用不同的 prompt 生成回答，对比效果
