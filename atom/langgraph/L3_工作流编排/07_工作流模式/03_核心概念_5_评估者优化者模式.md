# 07_工作流模式 - 核心概念5：评估者-优化者模式（Evaluator-Optimizer）

> LangGraph L3_工作流编排 | 核心概念 5/6

---

## 一句话定义

**评估者-优化者模式是"生成→评估→反馈→重新生成"的迭代循环模式，通过 LLM 自我评估不断提升输出质量。**

---

## 模式原理

### 核心思想

一次生成的结果往往不够好。让一个 LLM 生成，另一个 LLM 评估，如果不合格就带着反馈重新生成，直到满意为止。

```
┌──────────────────────────────────┐
│                                  │
▼                                  │
生成器 → 评估器 ──不合格+反馈──→ ┘
              │
              └──合格──→ END
```

### 前端类比

就像代码审查（Code Review）循环：
- 开发者提交 PR（生成）
- 审查者评审（评估）
- 如果有问题，带着评审意见修改后重新提交（反馈循环）
- 直到审查通过（终止条件）

### 日常生活类比

论文修改：写初稿 → 导师审阅 → 根据批注修改 → 再次审阅 → 直到导师满意。

---

## 核心特征

| 特征 | 说明 |
|------|------|
| **生成-评估循环** | 两个角色交替执行 |
| **结构化反馈** | 评估器输出具体的改进建议，而非简单的"好/坏" |
| **条件终止** | 通过条件边控制循环何时结束 |
| **质量递增** | 每次迭代都基于反馈改进 |

## LangGraph 实现

### 方式1：Graph API 实现

```python
"""
评估者-优化者模式实战：自动优化笑话生成
生成笑话 → 评估是否好笑 → 不好笑则带反馈重新生成
"""
from typing import Literal
from typing_extensions import TypedDict
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph, START, END
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")


# ===== 1. 定义状态 =====
class State(TypedDict):
    topic: str
    joke: str
    feedback: str
    grade: str
    attempts: int


# ===== 2. 定义评估输出结构 =====
class Feedback(BaseModel):
    grade: Literal["funny", "not funny"] = Field(
        description="判断笑话是否好笑"
    )
    feedback: str = Field(
        description="如果不好笑，给出改进建议"
    )


# 用 structured output 确保评估结果格式化
evaluator = llm.with_structured_output(Feedback)


# ===== 3. 定义节点 =====
def generate_joke(state: State):
    """生成器：生成或改进笑话"""
    attempts = state.get("attempts", 0) + 1

    if state.get("feedback"):
        # 带反馈重新生成
        msg = llm.invoke(
            f"写一个关于{state['topic']}的笑话。"
            f"之前的版本不够好，请根据反馈改进：{state['feedback']}"
        )
    else:
        # 首次生成
        msg = llm.invoke(f"写一个关于{state['topic']}的笑话")

    print(f"[生成器] 第{attempts}次尝试: {msg.content[:50]}...")
    return {"joke": msg.content, "attempts": attempts}


def evaluate_joke(state: State):
    """评估器：评估笑话质量"""
    result = evaluator.invoke(f"评估这个笑话的质量：{state['joke']}")
    print(f"[评估器] 评分: {result.grade}, 反馈: {result.feedback[:30]}...")
    return {"grade": result.grade, "feedback": result.feedback}


# ===== 4. 定义路由函数 =====
def should_continue(state: State):
    """决定是否继续优化"""
    if state["grade"] == "funny":
        return "accepted"
    # 最多尝试3次，防止无限循环
    if state.get("attempts", 0) >= 3:
        return "max_attempts"
    return "retry"


# ===== 5. 构建图 =====
builder = StateGraph(State)

builder.add_node("generator", generate_joke)
builder.add_node("evaluator", evaluate_joke)

builder.add_edge(START, "generator")
builder.add_edge("generator", "evaluator")

# 条件边：评估后决定下一步
builder.add_conditional_edges(
    "evaluator",
    should_continue,
    {
        "accepted": END,        # 合格 → 结束
        "retry": "generator",   # 不合格 → 带反馈重新生成
        "max_attempts": END,    # 超过最大次数 → 强制结束
    },
)

graph = builder.compile()

# ===== 6. 执行 =====
result = graph.invoke({"topic": "程序员", "attempts": 0})
print(f"\n最终笑话: {result['joke']}")
print(f"总尝试次数: {result['attempts']}")
```

**运行输出示例：**
```
[生成器] 第1次尝试: 程序员最怕什么？Bug...
[评估器] 评分: not funny, 反馈: 太老套了，需要更有创意...
[生成器] 第2次尝试: 为什么程序员总是分不清万圣节和圣诞节...
[评估器] 评分: funny, 反馈: 很有创意的双关...

最终笑话: 为什么程序员总是分不清万圣节和圣诞节？因为 Oct 31 == Dec 25
总尝试次数: 2
```

### 方式2：Functional API 实现

```python
from langgraph.func import task, entrypoint


@task
def generate(topic: str, feedback: str = "") -> str:
    """生成笑话"""
    if feedback:
        msg = llm.invoke(f"改进关于{topic}的笑话，反馈：{feedback}")
    else:
        msg = llm.invoke(f"写一个关于{topic}的笑话")
    return msg.content


@task
def evaluate(joke: str) -> Feedback:
    """评估笑话"""
    return evaluator.invoke(f"评估：{joke}")


@entrypoint()
def evaluator_optimizer(topic: str) -> str:
    feedback = ""
    for attempt in range(3):  # 最多3次
        joke = generate(topic, feedback).result()
        result = evaluate(joke).result()

        if result.grade == "funny":
            return joke
        feedback = result.feedback

    return joke  # 超过次数返回最后一版
```

> Functional API 的循环用 Python `for` 循环实现，比 Graph API 的条件边更直观。

## 关键设计要素

### 1. 结构化评估输出

```python
class Feedback(BaseModel):
    grade: Literal["pass", "fail"]  # 明确的通过/失败
    feedback: str                    # 具体改进建议
    score: float                     # 可选：数值评分
```

用 `with_structured_output` 确保评估结果可靠解析，而非自由文本。

### 2. 终止条件设计

**必须设置终止条件**，否则可能无限循环：

| 终止策略 | 实现方式 |
|---------|---------|
| 质量达标 | `grade == "pass"` |
| 最大次数 | `attempts >= max_attempts` |
| 分数阈值 | `score >= 0.8` |
| 改进停滞 | 连续两次分数无提升 |

```python
def should_continue(state):
    if state["grade"] == "pass":
        return "done"
    if state["attempts"] >= 3:
        return "done"  # 防止无限循环
    return "retry"
```

### 3. 反馈传递

评估器的反馈必须传递给生成器，否则重新生成只是"再试一次"而非"改进"：

```python
# ❌ 没有反馈：每次都是盲目重试
msg = llm.invoke(f"写一个关于{topic}的笑话")

# ✅ 带反馈：基于评估意见改进
msg = llm.invoke(
    f"写一个关于{topic}的笑话。"
    f"上一版的问题：{feedback}。请针对性改进。"
)
```

## 与 03_循环与迭代 的区别

| 维度 | 03_循环与迭代 | 本节：评估者-优化者模式 |
|------|-------------|---------------------|
| **视角** | 循环机制（How） | 设计模式（When/Why） |
| **重点** | 条件边实现循环、终止条件 | 生成-评估-反馈的完整模式 |
| **角色** | 单一节点循环 | 生成器 + 评估器两个角色 |
| **反馈** | 不一定有反馈 | 结构化反馈是核心 |

---

## 适用场景

| 场景 | 生成器 | 评估器 | 终止条件 |
|------|--------|--------|---------|
| **内容创作** | 写文章/笑话 | 评估质量 | 评分达标 |
| **代码生成** | 生成代码 | 运行测试 | 测试通过 |
| **Prompt 优化** | 生成 prompt | 评估效果 | 准确率达标 |
| **RAG 答案优化** | 生成回答 | 检查幻觉 | 无幻觉 |
| **翻译优化** | 翻译文本 | 评估流畅度 | 评分 ≥ 0.9 |

---

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| 输出质量显著提升 | 延迟成倍增加（每次循环 = 2次 LLM 调用） |
| 自动化质量控制 | 评估器本身可能不准确 |
| 可控的迭代次数 | Token 消耗随迭代次数增长 |
| 适合高质量要求场景 | 不适合实时响应场景 |

---

## 在 RAG 中的应用

### 自校正 RAG（Self-Corrective RAG）

```
用户提问 → 检索文档 → 生成回答 → 评估回答
                                    │
                          ┌─────────┴─────────┐
                          │                   │
                     有幻觉/不准确          准确
                          │                   │
                     重新检索+生成          返回答案
```

这就是 CRAG（Corrective RAG）的核心思想：生成回答后自动检查质量，不合格就重新检索和生成。

---

## 小结

评估者-优化者模式的本质是**用 LLM 评估 LLM 的输出，通过结构化反馈驱动迭代改进**。关键三要素：结构化评估输出、明确的终止条件、反馈传递机制。适合对输出质量有高要求的场景，但要注意延迟和成本的权衡。
