# 核心概念 5：静态分发 vs 动态分发

> **一句话概括：** Rust 独特地提供两种多态分发机制——泛型/`impl Trait`（静态分发，编译时展开，零运行时开销）和 `dyn Trait`（动态分发，vtable 查找，运行时灵活）。理解何时用哪个，是写出兼顾性能和灵活性的 Rust 代码的关键。

---

## 1. 静态分发（Static Dispatch）

### 1.1 机制：单态化（Monomorphization）

静态分发通过**泛型**或 **`impl Trait`** 实现。编译器在编译时**为每个具体类型生成一份专用代码**——这个过程叫做**单态化**。

```rust
// 泛型写法
fn process<T: Tool>(tool: &T) {
    println!("Tool: {}", tool.name());
    let result = tool.execute("hello");
    println!("Result: {}", result);
}

// impl Trait 写法（等价于上面的泛型）
fn process(tool: &impl Tool) {
    println!("Tool: {}", tool.name());
    let result = tool.execute("hello");
    println!("Result: {}", result);
}

// 当你这样调用时：
process(&shell_tool);    // T = ShellTool
process(&file_tool);     // T = FileReadTool
```

编译器实际生成的代码（概念上）：

```rust
// 编译器生成的 ShellTool 专用版本
fn process_ShellTool(tool: &ShellTool) {
    // 直接调用 ShellTool::name()，无需 vtable
    println!("Tool: {}", ShellTool::name(tool));
    // 直接调用 ShellTool::execute()，可以内联
    let result = ShellTool::execute(tool, "hello");
    println!("Result: {}", result);
}

// 编译器生成的 FileReadTool 专用版本
fn process_FileReadTool(tool: &FileReadTool) {
    println!("Tool: {}", FileReadTool::name(tool));
    let result = FileReadTool::execute(tool, "hello");
    println!("Result: {}", result);
}

// 调用时直接选择对应版本——编译期已确定！
process_ShellTool(&shell_tool);
process_FileReadTool(&file_tool);
```

### 1.2 图解

```
编译前（源码）:
┌─────────────────────────────┐
│ fn process<T: Tool>(t: &T)  │  ← 一份泛型代码
│ {                           │
│     t.name();               │
│     t.execute("hello");     │
│ }                           │
└─────────────────────────────┘

编译后（机器码）:
┌─────────────────────────────────────┐
│ fn process_ShellTool(t: &ShellTool) │  ← 专用版本 1
│ {                                   │
│     ShellTool::name(t);             │  ← 直接调用，可内联
│     ShellTool::execute(t, "hello"); │
│ }                                   │
├─────────────────────────────────────┤
│ fn process_FileReadTool(t: &FileRd) │  ← 专用版本 2
│ {                                   │
│     FileReadTool::name(t);          │  ← 直接调用，可内联
│     FileReadTool::execute(t, "hi"); │
│ }                                   │
├─────────────────────────────────────┤
│ fn process_WebSearchTool(t: &WebSr) │  ← 专用版本 3
│ {                                   │     ...每种类型一份！
│     ...                             │
│ }                                   │
└─────────────────────────────────────┘
```

### 1.3 性能优势

```rust
// 编译器可以做以下优化：
fn process(tool: &impl Tool) {
    tool.name();
    //   ^^^^^^^
    // 1. 内联（Inlining）：直接把 name() 的函数体塞进来
    //    → 减少函数调用开销
    //
    // 2. 常量传播（Constant Propagation）：
    //    如果 name() 返回 "shell"（字面量），编译器直接用 "shell"
    //
    // 3. 死代码消除（Dead Code Elimination）：
    //    如果某个分支在该类型下永远不执行，编译器直接删除
    //
    // 4. 循环展开（Loop Unrolling）：
    //    知道具体类型后，编译器可以更激进地优化循环
}
```

**总结：** 静态分发 = **编译时多一点工作，运行时零额外开销**。

## 2. 动态分发（Dynamic Dispatch）

### 2.1 机制：vtable 查找

动态分发通过 `dyn Trait` 实现。编译器**只生成一份函数代码**，运行时通过 **vtable 间接调用**正确的方法实现。

```rust
fn process(tool: &dyn Tool) {
    println!("Tool: {}", tool.name());
    let result = tool.execute("hello");
    println!("Result: {}", result);
}

// 调用时：
process(&shell_tool as &dyn Tool);   // 运行时查 vtable
process(&file_tool as &dyn Tool);    // 运行时查 vtable
```

编译器生成的代码（概念上）：

```rust
// 只生成一份代码！
fn process(tool: &dyn Tool) {
    // tool 是胖指针：(data_ptr, vtable_ptr)

    // 1. 从 vtable 中加载 name 函数指针
    let name_fn = tool.vtable_ptr[3];  // 偏移量 3 = name 方法
    // 2. 通过函数指针调用
    let name = name_fn(tool.data_ptr);
    println!("Tool: {}", name);

    // 3. 从 vtable 中加载 execute 函数指针
    let exec_fn = tool.vtable_ptr[4];
    // 4. 通过函数指针调用
    let result = exec_fn(tool.data_ptr, "hello");
    println!("Result: {}", result);
}
```

### 2.2 图解

```
编译后（机器码）:
┌──────────────────────────────────┐
│ fn process(tool: &dyn Tool)      │  ← 只有一份代码
│ {                                │
│   let fn_ptr = vtable[offset];   │  ← 运行时查表
│   fn_ptr(data_ptr);              │  ← 间接调用
│ }                                │
└──────────────────────────────────┘

运行时调用流程:
┌─────────────┐
│ tool.name() │
└──────┬──────┘
       │
       ▼
┌──────────────────┐
│ 读取 vtable_ptr   │ ← 额外一次指针解引用
└──────┬───────────┘
       │
       ▼
┌──────────────────────┐
│ vtable[3] = name_fn  │ ← 加载函数指针
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│ 调用 name_fn(data)   │ ← 间接调用
└──────────────────────┘
```

### 2.3 性能代价

每次方法调用的额外开销：

```
vtable 间接调用:
  1. 加载 vtable 指针        ~ 0.5ns（L1 cache hit）
  2. 加载函数指针             ~ 0.5ns（vtable 通常在 cache 中）
  3. 间接跳转                 ~ 1-3ns（可能的分支预测失败）
  ───────────────────────
  总计:                     ~ 1-5ns per call

  + 无法内联 → 失去后续优化机会（常量传播、死代码消除等）
```

**关键：** 1-5ns 的开销在绝大多数场景下可以忽略。ZeroClaw 中一次 LLM API 调用 100-500ms，vtable 查找的 5ns 相当于在马拉松赛道上多走了一步。

## 3. 性能对比表

| 维度 | 静态分发（泛型/impl Trait） | 动态分发（dyn Trait） |
|------|--------------------------|---------------------|
| **运行时开销** | 零——直接调用 | ~1-5ns/call vtable 查找 |
| **二进制大小** | 较大——每种类型生成一份代码 | 较小——只有一份代码 + vtable |
| **内联优化** | ✅ 可以完全内联 | ❌ 无法内联（间接调用） |
| **编译时间** | 较慢——单态化展开 | 较快——无需展开 |
| **异构集合** | ❌ 不支持 | ✅ `Vec<Box<dyn Trait>>` |
| **运行时选择** | ❌ 编译时确定 | ✅ 配置/输入驱动 |
| **类型信息** | 完全保留 | 擦除为 Trait 接口 |
| **代码膨胀** | 有风险 | 无 |
| **典型场景** | 热路径、已知类型 | 插件系统、依赖注入 |

### 数字直觉

```
                    时间线（对数刻度）
    ──────────────────────────────────────────────
    │                                             │
    1ns     5ns    100ns    1μs    1ms    100ms
    │        │       │       │      │       │
    ├─vtable─┤       │       │      │       │
    │ lookup │       │       │      │       │
    │        │       ├──Box──┤      │       │
    │        │       │ alloc │      │       │
    │        │       │       │      ├─LLM───┤
    │        │       │       │      │  API  │
    │        │       │       │      │ call  │
```

在 I/O 密集型应用（如 ZeroClaw）中，动态分发的开销完全可以忽略。只有在**每秒数百万次调用的 CPU 密集型热路径**中，才需要考虑静态分发。

## 4. 何时用哪个——决策树

```
你的场景是什么？

    需要异构集合？
    （不同类型放同一个 Vec/HashMap）
        │
        ├─ Yes ──→ dyn Trait（Box<dyn> / Arc<dyn>）
        │           例：Vec<Box<dyn Tool>>
        │
        └─ No
            │
            需要运行时选择实现？
            （根据配置/用户输入决定用哪个类型）
                │
                ├─ Yes ──→ dyn Trait
                │           例：config.toml 决定用 OpenAI 还是 Anthropic
                │
                └─ No
                    │
                    类型集合是封闭的吗？
                    （你知道所有可能的类型）
                        │
                        ├─ Yes ──→ enum（最佳性能）
                        │           例：enum Message { User(..), Assistant(..) }
                        │
                        └─ No
                            │
                            默认 ──→ 泛型 / impl Trait（静态分发）
                            例：fn process(tool: &impl Tool)
```

### 决策总结表

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 函数参数，类型已知 | `impl Trait` | 零开销，编译器优化 |
| 返回值，类型已知 | `impl Trait` | 调用者获得具体类型 |
| 结构体字段，可插拔 | `Box<dyn Trait>` | 运行时替换实现 |
| 多组件共享 | `Arc<dyn Trait>` | 共享所有权 + 动态分发 |
| 异构集合 | `Vec<Box<dyn Trait>>` | 不同类型同一容器 |
| 封闭类型枚举 | `enum` + `match` | 最佳性能，无间接调用 |
| 热路径性能关键 | 泛型 / enum | 零开销 |
| API 边界 | `dyn Trait` | 稳定的接口，灵活的实现 |

## 5. ZeroClaw 的混合策略

ZeroClaw 在同一个项目中**同时使用静态和动态分发**——架构边界用动态，内部热路径用静态。

### 5.1 架构边界：动态分发

```rust
// [来源: sourcecode/zeroclaw] Agent 的可插拔组件
struct Agent {
    // 所有架构边界使用 dyn Trait：
    provider: Box<dyn Provider>,         // 动态：运行时选择 LLM 提供商
    tools: Vec<Box<dyn Tool>>,           // 动态：异构工具集合
    memory: Arc<dyn Memory>,             // 动态：可插拔存储后端
    runtime: Arc<dyn RuntimeAdapter>,    // 动态：可插拔运行时
    hooks: Box<dyn HookHandler>,         // 动态：生命周期钩子
    observer: Arc<dyn Observer>,         // 动态：可观测性
}

// 为什么用动态分发？
// 1. 可插拔：用户通过配置选择实现（config.toml）
// 2. 异构集合：Vec<Box<dyn Tool>> 可以放不同类型的工具
// 3. 开放扩展：新增 impl Tool for X 无需修改 Agent 代码
// 4. 性能无关：这些组件不在热路径上（LLM API 调用是瓶颈）
```

### 5.2 内部热路径：静态分发

```rust
// 泛型工厂函数——接受闭包参数
pub fn create_memory_with_builders<F, G>(
    backend_name: &str,
    sqlite_builder: F,      // 静态分发：闭包类型在编译时确定
    workspace_dir: G,       // 静态分发：同上
) -> anyhow::Result<Box<dyn Memory>>
//   ^^^^^^^^^^^^^^^^^^^^
//   返回值是 dyn（动态分发）
where
    F: FnOnce() -> anyhow::Result<SqliteMemory>,
    G: Into<PathBuf>,
{
    match classify_memory_backend(backend_name) {
        MemoryBackendKind::Sqlite => {
            Ok(Box::new(sqlite_builder()?))
            //   ^^^^^^^^
            // 闭包调用在编译时已知具体类型
            // 编译器可以内联 sqlite_builder()
        }
        MemoryBackendKind::Markdown => {
            Ok(Box::new(MarkdownMemory::new(workspace_dir.into())))
        }
        MemoryBackendKind::None => {
            Ok(Box::new(NoneMemory::new()))
        }
    }
}

// impl Into<String> —— 常见的静态分发模式
fn set_name(&mut self, name: impl Into<String>) {
    self.name = name.into();
    // impl Into<String> 在编译时展开为具体类型
    // 可以接受 &str、String、Cow<str> 等——零运行时开销
}
```

### 5.3 混合策略图解

```
ZeroClaw 架构的分发策略：

外层（架构边界）──── 动态分发 ────────────────────────
│                                                    │
│  ┌─Agent─────────────────────────────────────────┐ │
│  │                                               │ │
│  │  provider: Box<dyn Provider>  ── 动态 ──      │ │
│  │  tools: Vec<Box<dyn Tool>>    ── 动态 ──      │ │
│  │  memory: Arc<dyn Memory>      ── 动态 ──      │ │
│  │                                               │ │
│  │  ┌─内部实现──────────────────────────────────┐│ │
│  │  │                                           ││ │
│  │  │  闭包参数: F: FnOnce()      ── 静态 ──   ││ │
│  │  │  转换 trait: impl Into<T>   ── 静态 ──   ││ │
│  │  │  迭代器: impl Iterator      ── 静态 ──   ││ │
│  │  │  Builder 参数: 泛型 <T>     ── 静态 ──   ││ │
│  │  │                                           ││ │
│  │  └─内层（热路径）── 静态分发 ────────────────┘│ │
│  │                                               │ │
│  └───────────────────────────────────────────────┘ │
│                                                    │
─────────────────────────────────────────────────────
```

**设计原则：** 在**用户可见的接口边界**使用动态分发（灵活性），在**内部实现细节**使用静态分发（性能）。

## 6. enum_dispatch 替代方案（预告）

当类型集合是**封闭的**（你知道所有可能的类型），可以用 `enum` 替代 `dyn Trait`，获得 **5-10x 性能提升**。

### 6.1 核心思想

```rust
// ===== 动态分发：dyn Trait =====
trait Shape {
    fn area(&self) -> f64;
}

struct Circle { radius: f64 }
struct Rectangle { width: f64, height: f64 }

impl Shape for Circle {
    fn area(&self) -> f64 { std::f64::consts::PI * self.radius * self.radius }
}
impl Shape for Rectangle {
    fn area(&self) -> f64 { self.width * self.height }
}

let shapes: Vec<Box<dyn Shape>> = vec![
    Box::new(Circle { radius: 5.0 }),
    Box::new(Rectangle { width: 3.0, height: 4.0 }),
];
// 每次 shape.area() → vtable 间接调用 + 无法内联


// ===== enum 替代：零间接调用 =====
enum ShapeEnum {
    Circle(Circle),
    Rectangle(Rectangle),
}

impl ShapeEnum {
    fn area(&self) -> f64 {
        match self {
            ShapeEnum::Circle(c) => c.area(),
            ShapeEnum::Rectangle(r) => r.area(),
            // match 分支在编译时已知 → 编译器可以内联
        }
    }
}

let shapes: Vec<ShapeEnum> = vec![
    ShapeEnum::Circle(Circle { radius: 5.0 }),
    ShapeEnum::Rectangle(Rectangle { width: 3.0, height: 4.0 }),
];
// 每次 shape.area() → match 跳转 + 可内联 → 快 5-10x
```

### 6.2 为什么 enum 更快

| 方面 | `dyn Trait` | `enum` |
|------|------------|--------|
| 方法调用 | vtable 间接跳转（不可预测） | match 分支（CPU 可预测） |
| 内存布局 | 堆上分散（每个 Box 独立分配） | 连续数组（cache 友好） |
| 内联 | 不可能 | 编译器可以内联 |
| 堆分配 | 每个元素一次 Box::new() | 零堆分配（enum 在栈/Vec 内） |

### 6.3 限制

```rust
// enum 的缺点：
// 1. 封闭性：新增类型必须修改 enum 定义（破坏开闭原则）
// 2. 大小由最大变体决定：如果某个类型特别大，浪费内存
// 3. 不适合插件系统：无法在运行时添加新类型

// 所以 ZeroClaw 不用 enum 做 Tool/Provider：
// - 用户需要自定义新工具（开放扩展）
// - 工具列表由配置决定（运行时选择）
// - 工具大小差异较大（ShellTool vs DelegateTool）
```

> **详细的 enum_dispatch 实战对比将在 [07_实战代码_场景4_enum_dispatch对比](./07_实战代码_场景4_enum_dispatch对比.md) 中展开。**

## 7. TypeScript 对照：TS 没有静态分发

```typescript
// TypeScript / JavaScript 中 —— 所有方法调用都是动态分发

interface Tool {
  name(): string;
  execute(input: string): string;
}

function process(tool: Tool): void {
  // V8 引擎在运行时通过隐藏类（Hidden Class）查找方法
  // 类似于 Rust 的 vtable，但更复杂
  console.log(tool.name());
  console.log(tool.execute("hello"));
}

// TypeScript 的泛型在编译后完全消失（类型擦除）：
function processGeneric<T extends Tool>(tool: T): void {
  // 编译为 JavaScript 后 ↓
  // function processGeneric(tool) { ... }
  // T 消失了！运行时仍然是动态查找
  console.log(tool.name());
}

// 对比：
// Rust  泛型 → 编译时展开为具体类型 → 直接调用 → 零运行时开销
// TS    泛型 → 编译时擦除 → 运行时动态查找 → 有运行时开销
```

| 方面 | Rust | TypeScript |
|------|------|-----------|
| 默认分发 | 静态（泛型） | 动态（所有调用） |
| 可选分发 | 动态（`dyn Trait`） | 无选择 |
| 泛型实现 | 单态化（代码生成） | 类型擦除（运行时消失） |
| 性能控制 | 开发者精确控制 | V8 JIT 自动优化 |
| 内联可能性 | 静态分发可内联 | JIT 可能内联热点 |

**V8 的优化策略：** 虽然 TS/JS 没有静态分发，V8 的 JIT 编译器会通过**内联缓存（Inline Cache）**和**隐藏类（Hidden Class）**优化热点路径，效果接近（但不等于）Rust 的静态分发。

## 8. 完整对比代码

同一个功能，分别用静态分发和动态分发实现，展示差异：

```rust
use std::time::Instant;

// ===== Trait 定义 =====
trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn execute(&self, input: &str) -> String;
}

struct ShellTool;
impl Tool for ShellTool {
    fn name(&self) -> &str { "shell" }
    fn execute(&self, input: &str) -> String {
        format!("[shell] {}", input)
    }
}

struct FileReadTool;
impl Tool for FileReadTool {
    fn name(&self) -> &str { "file_read" }
    fn execute(&self, input: &str) -> String {
        format!("[file] {}", input)
    }
}

// ===== 静态分发版本 =====
fn process_static(tool: &impl Tool) -> String {
    // 编译时已知具体类型
    // 编译器为 ShellTool 和 FileReadTool 各生成一份代码
    // 可以内联 name() 和 execute()
    format!("{}: {}", tool.name(), tool.execute("benchmark"))
}

// 等价的泛型写法
fn process_static_generic<T: Tool>(tool: &T) -> String {
    format!("{}: {}", tool.name(), tool.execute("benchmark"))
}

// ===== 动态分发版本 =====
fn process_dynamic(tool: &dyn Tool) -> String {
    // 运行时通过 vtable 查找
    // 只有一份代码，通过函数指针间接调用
    // 不能内联
    format!("{}: {}", tool.name(), tool.execute("benchmark"))
}

// ===== 异构集合（只能用动态分发） =====
fn process_collection(tools: &[Box<dyn Tool>]) -> Vec<String> {
    // 这里只能用 dyn，因为 Vec 中元素类型不同
    tools.iter()
        .map(|tool| format!("{}: {}", tool.name(), tool.execute("batch")))
        .collect()
}

// ===== enum 替代（封闭集合，最高性能） =====
enum ToolEnum {
    Shell(ShellTool),
    FileRead(FileReadTool),
}

impl ToolEnum {
    fn name(&self) -> &str {
        match self {
            ToolEnum::Shell(t) => t.name(),
            ToolEnum::FileRead(t) => t.name(),
        }
    }

    fn execute(&self, input: &str) -> String {
        match self {
            ToolEnum::Shell(t) => t.execute(input),
            ToolEnum::FileRead(t) => t.execute(input),
        }
    }
}

fn process_enum(tool: &ToolEnum) -> String {
    format!("{}: {}", tool.name(), tool.execute("benchmark"))
}

// ===== 性能基准测试 =====
fn main() {
    let shell = ShellTool;
    let file = FileReadTool;
    let iterations = 1_000_000;

    // 测试 1：静态分发
    let start = Instant::now();
    for _ in 0..iterations {
        let _ = process_static(&shell);
        let _ = process_static(&file);
    }
    let static_time = start.elapsed();

    // 测试 2：动态分发
    let tools: Vec<Box<dyn Tool>> = vec![
        Box::new(ShellTool),
        Box::new(FileReadTool),
    ];
    let start = Instant::now();
    for _ in 0..iterations {
        for tool in &tools {
            let _ = process_dynamic(tool.as_ref());
        }
    }
    let dynamic_time = start.elapsed();

    // 测试 3：enum 分发
    let enum_tools = vec![
        ToolEnum::Shell(ShellTool),
        ToolEnum::FileRead(FileReadTool),
    ];
    let start = Instant::now();
    for _ in 0..iterations {
        for tool in &enum_tools {
            let _ = process_enum(tool);
        }
    }
    let enum_time = start.elapsed();

    println!("===== 性能对比（{} 次迭代）=====", iterations);
    println!("静态分发 (impl Trait): {:?}", static_time);
    println!("动态分发 (dyn Trait):  {:?}", dynamic_time);
    println!("enum 分发 (match):     {:?}", enum_time);
    println!();
    println!("动态 vs 静态: {:.1}x 慢",
        dynamic_time.as_nanos() as f64 / static_time.as_nanos() as f64);
    println!("动态 vs enum:  {:.1}x 慢",
        dynamic_time.as_nanos() as f64 / enum_time.as_nanos() as f64);
}
```

**预期输出（实际数字因硬件而异）：**

```
===== 性能对比（1000000 次迭代）=====
静态分发 (impl Trait): 45.2ms
动态分发 (dyn Trait):  68.7ms
enum 分发 (match):     46.1ms

动态 vs 静态: 1.5x 慢
动态 vs enum:  1.5x 慢
```

> **注意：** 实际性能差距取决于函数复杂度。函数体越大，vtable 调用开销占比越小。在 ZeroClaw 这样的 I/O 密集型应用中，差异几乎不可测量。

## 9. 核心记忆点

```
静态 vs 动态分发速记：

  静态分发（泛型/impl Trait）:
    ✅ 零运行时开销
    ✅ 可内联优化
    ❌ 二进制较大（代码膨胀）
    ❌ 不能异构集合
    ❌ 编译时确定类型

  动态分发（dyn Trait）:
    ✅ 运行时灵活
    ✅ 异构集合
    ✅ 二进制较小
    ❌ vtable 间接调用（~1-5ns）
    ❌ 无法内联

  enum 替代（封闭集合）:
    ✅ 接近静态分发性能
    ✅ 可放同一容器
    ❌ 封闭——不能运行时扩展
    ❌ 大小由最大变体决定

ZeroClaw 策略:
    架构边界 → dyn Trait（灵活性优先）
    内部热路径 → 泛型/闭包（性能优先）
    封闭枚举  → enum（如 Message、Event）
```

---

## 10. 一句话总结

> **Rust 的两种分发机制是它的独特优势——静态分发（泛型/`impl Trait`）给你 C++ 级别的零成本性能，动态分发（`dyn Trait`）给你 TypeScript 级别的运行时灵活性。ZeroClaw 的架构完美展示了如何混合使用：架构边界用 `Box<dyn Trait>` / `Arc<dyn Trait>` 实现可插拔设计，内部用泛型和闭包保持零开销。理解"何时用哪个"是从 TypeScript 转向 Rust 最重要的思维转变之一。**
