# 08_面试必问

## 问题1："LangGraph 中的 Annotated 字段是什么？如何使用？为什么需要它？"

### 普通回答（❌ 不出彩）

"Annotated 是 Python 的类型注解，可以在 LangGraph 中给状态字段添加 reducer 函数，用来控制状态更新的方式。比如用 `operator.add` 可以累加列表。"

**问题**：回答太浅，没有展示深度理解。

---

### 出彩回答（✅ 推荐）

> **Annotated 字段在 LangGraph 中有三层含义：**
>
> **1. 语法层面**：它是 Python 3.9+ 的类型注解特性，格式是 `Annotated[type, metadata]`，可以在类型上附加元数据。LangGraph 利用这个特性，将 reducer 函数作为元数据附加到状态字段上。
>
> **2. 机制层面**：LangGraph 在初始化 StateGraph 时，会通过 `get_type_hints(include_extras=True)` 解析状态类的类型注解，提取 `__metadata__` 属性，验证 reducer 函数签名是否符合 `(old, new) -> merged` 的要求，然后创建 `BinaryOperatorAggregate` Channel 来管理状态更新。
>
> **3. 应用层面**：它解决了状态累积的问题。默认情况下，LangGraph 的状态更新是覆盖模式，但在对话系统、多步推理等场景中，我们需要累积消息、步骤等数据。Annotated 提供了声明式的状态管理方式，避免手动合并状态。
>
> **与普通状态字段的区别**：
> - 无 Annotated：`messages: list` → 每次更新直接覆盖
> - 有 Annotated：`messages: Annotated[list, operator.add]` → 每次更新累加到列表末尾
>
> **在实际工作中的应用**：
> - 对话系统：使用 `Annotated[list[BaseMessage], add_messages]` 管理对话历史，支持按 ID 更新消息
> - ReAct Agent：使用 `Annotated[list[ToolResult], operator.add]` 分离工具结果，避免 token 膨胀
> - Plan-and-Execute：使用 `Annotated[list[str], operator.add]` 累积执行步骤

**为什么这个回答出彩？**

1. ✅ **多层次解释**：从语法、机制、应用三个层面完整解释
2. ✅ **技术深度**：提到了 `get_type_hints(include_extras=True)`、`BinaryOperatorAggregate`、`__metadata__` 等实现细节
3. ✅ **对比说明**：清晰对比有无 Annotated 的区别
4. ✅ **实际应用**：给出三个具体的生产环境使用场景

## 问题2："如何自定义 Reducer 函数？有哪些注意事项？"

### 普通回答（❌ 不出彩）

"自定义 Reducer 就是写一个函数，接收两个参数，返回合并后的值。比如 `lambda x, y: x + y`。"

**问题**：没有说明签名要求、常见陷阱、最佳实践。

---

### 出彩回答（✅ 推荐）

> **自定义 Reducer 函数有三个关键点：**
>
> **1. 签名要求**：必须是 `(old_value, new_value) -> merged_value`，恰好两个位置参数。LangGraph 会用 `inspect.signature()` 验证签名，不符合会抛出 `ValueError`。
>
> **2. 实现方式**：
> - Lambda 函数：`lambda x, y: y or x`（优先使用新值）
> - 普通函数：`def merge_dedup(old, new): return old + [x for x in new if x not in set(old)]`
> - 工厂函数：`def merge_with_limit(max_size): return lambda old, new: (old + new)[-max_size:]`
>
> **3. 常见陷阱**：
> - ❌ **可变对象共享**：`old.extend(new); return old` 会修改原始状态，导致 checkpoint 错误
> - ✅ **创建新对象**：`return old + new` 或 `return [*old, *new]`
> - ❌ **签名错误**：`def bad(value): return value` 只有一个参数，会被拒绝
> - ✅ **正确签名**：`def good(old, new): return old + new`
>
> **在实际工作中的应用**：
> - 去重合并：`lambda old, new: old + [x for x in new if x not in set(old)]`
> - 限制大小：`lambda old, new: (old + new)[-100:]` 保留最近 100 条
> - 条件覆盖：`lambda x, y: y if y is not None else x` 只在新值非空时覆盖
> - 字典深度合并：自定义函数递归合并嵌套字典

**为什么这个回答出彩？**

1. ✅ **签名验证机制**：说明了 LangGraph 如何验证 reducer 签名
2. ✅ **多种实现方式**：展示了 lambda、普通函数、工厂函数三种方式
3. ✅ **常见陷阱**：指出可变对象共享、签名错误等问题
4. ✅ **实战模式**：提供了四种常用的自定义 reducer 模式

## 延伸问题

**面试官可能追问**：

1. "Annotated 的 `__metadata__` 属性是如何被解析的？"
   - 答：通过 `get_type_hints(schema, include_extras=True)` 获取类型提示，然后访问 `typ.__metadata__` 获取元数据元组，最后一个元素就是 reducer 函数

2. "为什么 LangGraph 推荐 TypedDict 而不是 Pydantic？"
   - 答：TypedDict 更轻量级，支持部分状态更新，性能更好（~0.1ms vs ~1.0ms per update）

3. "如何处理 token 限制问题？"
   - 答：分离不同类型的数据到独立字段，使用滑动窗口保留最近消息，或使用摘要技术压缩历史
