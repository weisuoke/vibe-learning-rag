# 03_核心概念_2 - 终止条件设计

> LangGraph 循环的核心挑战：如何优雅地停下来

---

## 概念定义

**终止条件是 LangGraph 循环中决定"何时停止迭代"的逻辑，通过条件边路由函数返回 `END` 或路由到终止节点来实现。**

LangGraph 的循环本质是边指向图中已存在的节点形成有向环。没有终止条件的循环会一直执行直到触发 `GraphRecursionError`。好的终止条件设计需要考虑：

1. 什么条件下应该停止（业务逻辑）
2. 停止时走哪条路径（成功/失败/超时）
3. 接近限制时如何优雅降级（安全机制）

---

## 条件边终止 — 最基础的终止模式

### 核心原理

条件边的路由函数（`path` 函数）在运行时检查状态，返回 `END` 时循环终止。这是 LangGraph 中最常用的终止方式。

源码层面，条件边通过 `add_conditional_edges()` 注册（`state.py` L839-L887），路由函数的返回值在 `_branch.py` 的 `_finish()` 方法中被解析为目标节点或 `END`：

```python
# _branch.py L192
def _finish(self, writer, input, result, config):
    if not isinstance(result, (list, tuple)):
        result = [result]
    if self.ends:
        destinations = [r if isinstance(r, Send) else self.ends[r] for r in result]
    else:
        destinations = cast(Sequence[Send | str], result)
    # 验证：不能返回 None 或 START
    if any(dest is None or dest == START for dest in destinations):
        raise ValueError("Branch did not return a valid destination")
```

当路由函数返回 `END` 时，不会写入任何节点的触发 channel，下一轮 `tick()` 中 `prepare_next_tasks()` 找不到待执行的任务，循环自然结束。

### 基础模式：计数器终止

```python
"""
最基础的循环终止模式：基于计数器
当 aggregate 列表长度达到阈值时停止
"""
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    aggregate: Annotated[list, operator.add]


def node_a(state: State):
    print(f'Node A sees {state["aggregate"]}')
    return {"aggregate": ["A"]}


def node_b(state: State):
    print(f'Node B sees {state["aggregate"]}')
    return {"aggregate": ["B"]}


# === 关键：路由函数决定是否继续循环 ===
def route(state: State) -> Literal["b", "__end__"]:
    if len(state["aggregate"]) < 7:
        return "b"       # 继续循环
    else:
        return END       # 终止循环


builder = StateGraph(State)
builder.add_node("a", node_a)
builder.add_node("b", node_b)

builder.add_edge(START, "a")
builder.add_conditional_edges("a", route)  # a 执行后，由 route 决定去向
builder.add_edge("b", "a")                 # b 执行后回到 a，形成循环

graph = builder.compile()

result = graph.invoke({"aggregate": []})
print(f"最终结果: {result['aggregate']}")
```

**运行输出：**
```
Node A sees []
Node B sees ['A']
Node A sees ['A', 'B']
Node B sees ['A', 'B', 'A']
Node A sees ['A', 'B', 'A', 'B']
Node B sees ['A', 'B', 'A', 'B', 'A']
Node A sees ['A', 'B', 'A', 'B', 'A', 'B']
最终结果: ['A', 'B', 'A', 'B', 'A', 'B', 'A']
```

**执行流程：**
```
START → a (aggregate=[]) → route: len=1 < 7 → b → a → route: len=3 < 7 → b → a
→ route: len=5 < 7 → b → a → route: len=7 >= 7 → END
```

## 状态条件判断 — 多种终止标准

实际项目中，终止条件远比"计数器达到阈值"复杂。以下是四种常见的终止标准模式。

### 模式 1：计数器终止

最简单直接，适合固定迭代次数的场景。

```python
def route_by_counter(state: State) -> Literal["process", "__end__"]:
    """基于迭代计数器的终止"""
    max_iterations = 5
    if state["iteration_count"] >= max_iterations:
        return END
    return "process"
```

### 模式 2：质量分数终止

适合自校正（self-correction）场景，如代码生成、文本优化。

```python
def route_by_quality(state: State) -> Literal["refine", "__end__"]:
    """基于质量评分的终止 — 达标即停"""
    threshold = 0.85
    if state["quality_score"] >= threshold:
        return END    # 质量达标，停止优化
    return "refine"   # 质量不够，继续迭代
```

### 模式 3：任务完成终止

适合 Agent 场景，任务完成标志由 LLM 判断。

```python
def route_by_completion(state: State) -> Literal["agent", "__end__"]:
    """基于任务完成状态的终止"""
    if state.get("task_complete", False):
        return END
    return "agent"
```

### 模式 4：复合条件终止

实际生产中最常用 — 组合多个条件，任一满足即终止。

```python
"""
复合终止条件示例：质量门控 + 迭代上限 + 任务完成
三个条件任一满足即终止循环
"""
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class RefineState(TypedDict):
    content: str
    quality_score: float
    iteration_count: int
    task_complete: bool
    history: Annotated[list[str], operator.add]


def evaluate_node(state: RefineState):
    """评估当前内容质量"""
    iteration = state["iteration_count"]

    # 模拟质量评分递增（实际中调用 LLM 评估）
    scores = [0.3, 0.5, 0.7, 0.85, 0.92]
    score = scores[min(iteration, len(scores) - 1)]

    return {
        "quality_score": score,
        "history": [f"第{iteration + 1}轮评估: 质量={score}"],
    }


def refine_node(state: RefineState):
    """优化内容"""
    iteration = state["iteration_count"]
    return {
        "content": f"优化后的内容 v{iteration + 1}",
        "iteration_count": iteration + 1,
        "history": [f"第{iteration + 1}轮优化完成"],
    }


# === 复合终止条件 ===
def should_continue(state: RefineState) -> Literal["refine", "__end__"]:
    """
    三重终止条件（任一满足即停止）：
    1. 质量达标 (>= 0.85)
    2. 迭代次数达到上限 (>= 5)
    3. 任务标记为完成
    """
    # 条件 1：质量达标
    if state["quality_score"] >= 0.85:
        print(f"  → 终止原因: 质量达标 ({state['quality_score']})")
        return END

    # 条件 2：迭代上限
    if state["iteration_count"] >= 5:
        print(f"  → 终止原因: 达到最大迭代次数")
        return END

    # 条件 3：任务完成
    if state.get("task_complete", False):
        print(f"  → 终止原因: 任务标记完成")
        return END

    return "refine"


builder = StateGraph(RefineState)
builder.add_node("evaluate", evaluate_node)
builder.add_node("refine", refine_node)

builder.add_edge(START, "evaluate")
builder.add_conditional_edges("evaluate", should_continue)
builder.add_edge("refine", "evaluate")  # 优化后重新评估，形成循环

graph = builder.compile()

result = graph.invoke({
    "content": "初始内容",
    "quality_score": 0.0,
    "iteration_count": 0,
    "task_complete": False,
    "history": [],
})

print("\n执行历史:")
for entry in result["history"]:
    print(f"  {entry}")
print(f"最终质量: {result['quality_score']}")
```

**运行输出：**
```
  → 终止原因: 质量达标 (0.85)

执行历史:
  第1轮评估: 质量=0.3
  第1轮优化完成
  第2轮评估: 质量=0.5
  第2轮优化完成
  第3轮评估: 质量=0.7
  第3轮优化完成
  第4轮评估: 质量=0.85
最终质量: 0.85
```

## 多出口设计 — 不同退出路径

循环不一定只有"成功退出"一条路。根据终止原因的不同，可以走不同的后续处理路径。

### 设计思路

```
                    ┌─── 质量达标 ──→ success_handler ──→ END
                    │
evaluate ──route──→├─── 迭代超限 ──→ timeout_handler ──→ END
                    │
                    ├─── 出现错误 ──→ error_handler ───→ END
                    │
                    └─── 继续优化 ──→ refine ──→ evaluate (循环)
```

### 完整实现

```python
"""
多出口循环设计：根据终止原因走不同的处理路径
- 成功出口：质量达标，执行成功处理
- 超时出口：迭代超限，执行降级处理
- 错误出口：出现异常，执行错误处理
"""
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class MultiExitState(TypedDict):
    content: str
    quality_score: float
    iteration_count: int
    exit_reason: str
    has_error: bool
    history: Annotated[list[str], operator.add]


def process_node(state: MultiExitState):
    """处理节点：模拟内容生成和评估"""
    iteration = state["iteration_count"] + 1

    # 模拟：第3轮可能出错
    if iteration == 3 and state.get("has_error", False):
        return {
            "iteration_count": iteration,
            "quality_score": -1.0,  # 错误标记
            "history": [f"第{iteration}轮: 处理出错"],
        }

    # 模拟质量递增
    score = min(0.2 * iteration + 0.1, 0.95)
    return {
        "iteration_count": iteration,
        "quality_score": score,
        "history": [f"第{iteration}轮: 质量={score:.2f}"],
    }


# === 多出口路由函数 ===
def multi_exit_route(
    state: MultiExitState,
) -> Literal["process", "success_handler", "timeout_handler", "error_handler"]:
    """根据不同条件路由到不同的出口"""

    # 出口 1：错误退出
    if state["quality_score"] < 0:
        return "error_handler"

    # 出口 2：成功退出
    if state["quality_score"] >= 0.85:
        return "success_handler"

    # 出口 3：超时退出
    if state["iteration_count"] >= 5:
        return "timeout_handler"

    # 继续循环
    return "process"


def success_handler(state: MultiExitState):
    return {
        "exit_reason": "success",
        "history": [f"✅ 成功完成! 最终质量: {state['quality_score']:.2f}"],
    }


def timeout_handler(state: MultiExitState):
    return {
        "exit_reason": "timeout",
        "history": [f"⏰ 迭代超限，返回当前最佳结果 (质量: {state['quality_score']:.2f})"],
    }


def error_handler(state: MultiExitState):
    return {
        "exit_reason": "error",
        "history": ["❌ 处理出错，执行错误恢复"],
    }


builder = StateGraph(MultiExitState)

# 添加节点
builder.add_node("process", process_node)
builder.add_node("success_handler", success_handler)
builder.add_node("timeout_handler", timeout_handler)
builder.add_node("error_handler", error_handler)

# 添加边
builder.add_edge(START, "process")
builder.add_conditional_edges("process", multi_exit_route)

# 三个出口都通向 END
builder.add_edge("success_handler", END)
builder.add_edge("timeout_handler", END)
builder.add_edge("error_handler", END)

graph = builder.compile()

# 测试正常流程（质量递增，最终成功退出）
print("=== 正常流程 ===")
result = graph.invoke({
    "content": "初始内容",
    "quality_score": 0.0,
    "iteration_count": 0,
    "exit_reason": "",
    "has_error": False,
    "history": [],
})
for entry in result["history"]:
    print(f"  {entry}")
print(f"退出原因: {result['exit_reason']}")
```

**运行输出：**
```
=== 正常流程 ===
  第1轮: 质量=0.30
  第2轮: 质量=0.50
  第3轮: 质量=0.70
  第4轮: 质量=0.90
  ✅ 成功完成! 最终质量: 0.90
退出原因: success
```

## 优雅降级 — 接近限制时的安全处理

### 问题背景

当循环接近 `recursion_limit` 时，如果不做处理，LangGraph 会抛出 `GraphRecursionError`。这对用户体验很差 — 用户看到的是一个异常，而不是一个有用的结果。

优雅降级的核心思想：**在接近限制时主动检测，返回部分结果而不是崩溃。**

### 使用 RemainingSteps 实现优雅降级

`RemainingSteps` 是 LangGraph 提供的管理值（Managed Value），自动注入到状态中，返回当前剩余的可执行步数。

源码定义（`is_last_step.py`）：

```python
# is_last_step.py
class RemainingStepsManager(ManagedValue[int]):
    @staticmethod
    def get(scratchpad: PregelScratchpad) -> int:
        return scratchpad.stop - scratchpad.step

RemainingSteps = Annotated[int, RemainingStepsManager]
```

### Fallback 节点模式

这是 Context7 官方文档推荐的模式：当剩余步数不足时，路由到一个 fallback 节点返回"尽力而为"的结果。

```python
"""
优雅降级模式：使用 RemainingSteps + Fallback 节点
当接近递归限制时，主动路由到 fallback 节点返回部分结果
"""
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.managed.is_last_step import RemainingSteps


class GracefulState(TypedDict):
    messages: Annotated[list[str], operator.add]
    remaining_steps: RemainingSteps  # 管理值，自动注入


def reasoning_node(state: GracefulState):
    """推理节点：正常处理逻辑"""
    remaining = state["remaining_steps"]

    # 即使在正常处理中，也可以感知剩余步数
    if remaining <= 2:
        return {"messages": [f"[推理] 步数不足({remaining})，快速总结..."]}

    return {"messages": [f"[推理] 深度思考中... (剩余{remaining}步)"]}


def route_decision(
    state: GracefulState,
) -> Literal["reasoning_node", "fallback_node"]:
    """路由决策：剩余步数不足时转向 fallback"""
    if state["remaining_steps"] <= 2:
        return "fallback_node"
    return "reasoning_node"


def fallback_node(state: GracefulState):
    """Fallback 节点：返回尽力而为的结果"""
    return {
        "messages": ["[Fallback] 已达复杂度上限，返回当前最佳结果"]
    }


builder = StateGraph(GracefulState)
builder.add_node("reasoning_node", reasoning_node)
builder.add_node("fallback_node", fallback_node)

builder.add_edge(START, "reasoning_node")
builder.add_conditional_edges("reasoning_node", route_decision)
builder.add_edge("fallback_node", END)

graph = builder.compile()

# 设置较小的递归限制来观察优雅降级
result = graph.invoke(
    {"messages": []},
    config={"recursion_limit": 5},
)

print("执行过程:")
for msg in result["messages"]:
    print(f"  {msg}")
```

**运行输出：**
```
执行过程:
  [推理] 深度思考中... (剩余6步)
  [推理] 深度思考中... (剩余4步)
  [推理] 步数不足(2)，快速总结...
  [Fallback] 已达复杂度上限，返回当前最佳结果
```

**关键点：**
- `RemainingSteps` 是自动注入的，不需要手动维护
- `remaining_steps <= 2` 是推荐的阈值（留 1 步给 fallback 节点执行，1 步给 END）
- Fallback 节点直接连接到 `END`，确保不会再进入循环

## 终止条件设计的最佳实践

### 1. 始终提供明确的 END 出口

```python
# ✅ 正确：路由函数有明确的 END 返回
def route(state) -> Literal["continue", "__end__"]:
    if state["done"]:
        return END
    return "continue"

# ❌ 危险：路由函数可能永远不返回 END
def route(state) -> Literal["step_a", "step_b"]:
    if state["score"] > 0.5:
        return "step_a"
    return "step_b"  # 没有 END 出口！
```

### 2. 组合多层防护

```python
def safe_route(state) -> Literal["process", "__end__"]:
    """三层防护：业务条件 + 计数器 + RemainingSteps"""
    # 第一层：业务条件
    if state["task_complete"]:
        return END

    # 第二层：迭代计数器
    if state["iteration_count"] >= 10:
        return END

    # 第三层：系统级安全阀
    if state["remaining_steps"] <= 2:
        return END

    return "process"
```

### 3. 不同场景的推荐策略

| 场景 | 推荐终止策略 | 说明 |
|------|-------------|------|
| ReAct Agent | 任务完成 + 迭代上限 | LLM 判断任务完成，上限兜底 |
| 自校正循环 | 质量达标 + 迭代上限 | 质量分数达到阈值即停 |
| 数据处理管道 | 计数器 | 固定迭代次数 |
| 多轮对话 | 用户指令 + 超时 | 用户说"结束"或超时 |
| 搜索优化 | 结果不再改善 + 上限 | 连续两轮分数无提升即停 |

### 4. 避免常见陷阱

```python
# ❌ 陷阱 1：终止条件永远不满足
def route(state):
    if state["score"] == 1.0:  # 浮点数精确比较，可能永远不等于 1.0
        return END
    return "continue"

# ✅ 修复：使用 >= 而非 ==
def route(state):
    if state["score"] >= 0.95:
        return END
    return "continue"

# ❌ 陷阱 2：忘记更新状态中的计数器
def process(state):
    return {"content": "new content"}  # 没有更新 iteration_count！

# ✅ 修复：每次迭代都更新计数器
def process(state):
    return {
        "content": "new content",
        "iteration_count": state["iteration_count"] + 1,
    }
```

## 在 RAG 开发中的应用

### 场景 1：RAG 自校正检索

```python
def rag_retrieval_route(state) -> Literal["rewrite_query", "__end__"]:
    """RAG 检索质量不够时，改写查询重新检索"""
    # 检索结果相关性足够高
    if state["relevance_score"] >= 0.8:
        return END

    # 已经改写了 3 次，不再尝试
    if state["rewrite_count"] >= 3:
        return END

    return "rewrite_query"
```

### 场景 2：Agent 工具调用循环

```python
def agent_route(state) -> Literal["call_tool", "generate_answer", "__end__"]:
    """Agent 决定是否需要继续调用工具"""
    last_message = state["messages"][-1]

    # LLM 认为信息足够，生成最终答案
    if not last_message.tool_calls:
        return "generate_answer"

    # 工具调用次数过多，强制生成答案
    if state["tool_call_count"] >= 5:
        return "generate_answer"

    return "call_tool"
```
