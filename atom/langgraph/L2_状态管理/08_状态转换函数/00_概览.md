# 状态转换函数（State Transition Functions）- 知识点概览

## 知识点简介

**状态转换函数是 LangGraph 节点内部将输入状态转换为输出更新的核心逻辑，遵循 `State -> Partial<State>` 签名，负责数据清洗、格式化、业务计算等转换操作，是节点"做什么事"的本质定义。**

在 LangGraph 中，每个节点本质上就是一个状态转换函数。它接收完整的当前状态，执行某种转换逻辑（调用 LLM、清洗数据、格式化输出等），然后返回一个字典，只包含需要更新的字段。框架负责把这个"部分更新"合并回完整状态——合并的规则由 Reducer 定义（那是 01_Reducer 的内容），而转换函数本身关注的是：**拿到数据后，怎么加工它？**

### 与 Reducer 的关键区分

| 维度 | 状态转换函数（本节） | Reducer 函数（01_Reducer） |
|------|---------------------|--------------------------|
| 核心问题 | 节点内部如何转换数据？ | 多个更新如何合并？ |
| 执行位置 | 节点函数内部 | 框架层（Channel 机制） |
| 开发者角色 | 你来写转换逻辑 | 你来声明合并策略 |
| 类比 | 厨师炒菜的过程 | 多道菜怎么摆盘上桌 |
| 签名 | `State -> dict` | `(Value, Value) -> Value` |

> 一句话区分：**转换函数是"节点做了什么"，Reducer 是"结果怎么合并"。**

## 为什么需要学习状态转换函数？

### 没有转换思维时会发生什么？

```python
# 常见的"意大利面条"节点——什么都塞在一起
def bad_node(state):
    raw = state["user_input"]
    # 清洗、转换、调用 LLM、格式化全部混在一起
    cleaned = raw.strip().lower()
    if not cleaned:
        cleaned = "default query"
    response = call_llm(cleaned)
    formatted = f"## 回答\n\n{response}"
    tokens = len(response.split())
    return {
        "cleaned_input": cleaned,
        "llm_response": response,
        "formatted_output": formatted,
        "token_count": tokens,
    }
```

这段代码能跑，但不可测试、不可复用、不可维护。状态转换函数的学习目标就是让你写出**清晰、可组合、可测试**的节点逻辑。

### 状态转换函数解决三个核心问题

| 问题 | 说明 | 示例 |
|------|------|------|
| **节点职责不清** | 一个节点做太多事，难以调试 | 清洗 + LLM 调用 + 格式化全在一个函数里 |
| **数据质量不可控** | 脏数据直接传给 LLM，结果不稳定 | 用户输入有多余空格、特殊字符 |
| **输出格式不统一** | 不同节点返回格式不一致，下游节点难以处理 | 有的返回字符串，有的返回字典 |

## 核心概念预览

### 概念 1：节点函数签名与状态接收

LangGraph 的 `StateGraph` 源码中明确定义了节点函数的签名约定：

> "The signature of each node is `State -> Partial<State>`. Each state key can optionally be annotated with a reducer function."
> — `graph/state.py` StateGraph 类文档

```python
from typing import TypedDict

class State(TypedDict):
    query: str
    context: list[str]
    answer: str

# 节点接收完整 State，返回只包含要更新字段的 dict
def retrieve_node(state: State) -> dict:
    query = state["query"]           # 读取需要的字段
    docs = search_documents(query)   # 执行转换逻辑
    return {"context": docs}         # 只返回要更新的字段
    # query 和 answer 保持原值不变
```

**关键点：** 节点拿到的是完整状态的只读快照，返回的是部分更新字典。框架通过 Channel 机制（`LastValue` 用于简单覆盖，`BinaryOperatorAggregate` 用于 Reducer 字段）处理合并。

### 概念 2：状态转换管道模式

把复杂的转换逻辑拆成清晰的步骤：

```python
def rag_node(state: State) -> dict:
    # Step 1: 预处理
    query = state["query"].strip().lower()

    # Step 2: 核心逻辑
    docs = retriever.search(query, top_k=5)
    context = "\n".join(doc.content for doc in docs)
    answer = llm.generate(f"基于以下内容回答：\n{context}\n\n问题：{query}")

    # Step 3: 后处理
    answer = answer.strip()
    if len(answer) > 2000:
        answer = answer[:2000] + "..."

    return {"context": [doc.content for doc in docs], "answer": answer}
```

### 概念 3：数据清洗与预处理

在数据进入核心逻辑之前，确保它是干净的：

```python
def preprocess_node(state: State) -> dict:
    raw_query = state["query"]

    # 去除多余空白
    cleaned = " ".join(raw_query.split())
    # 移除特殊字符
    cleaned = re.sub(r'[^\w\s\u4e00-\u9fff？！。，]', '', cleaned)
    # 空查询兜底
    if not cleaned:
        cleaned = "请提供一个有效的问题"

    return {"cleaned_query": cleaned}
```

### 概念 4：输出格式化与后处理

LLM 返回的原始文本往往需要加工才能交给用户或下游节点：

```python
def format_node(state: State) -> dict:
    raw_answer = state["raw_answer"]
    sources = state["context"]

    # 结构化输出
    formatted = {
        "answer": raw_answer.strip(),
        "sources": [s[:200] for s in sources],  # 截断过长的来源
        "confidence": calculate_confidence(raw_answer, sources),
    }

    return {"formatted_result": formatted}
```

### 概念 5：Command API 状态转换

`Command` 对象让你在一次返回中同时完成状态更新和路由控制：

```python
from langgraph.types import Command

def router_node(state: State) -> Command:
    if state["needs_retrieval"]:
        return Command(
            update={"step": "retrieving"},
            goto="retrieve_node"
        )
    else:
        return Command(
            update={"step": "generating"},
            goto="generate_node"
        )
```

## 双重类比速览

### 前端开发类比

状态转换函数就像 React 组件中的事件处理函数：

```
React 组件                    LangGraph 节点
─────────────                ──────────────
props（只读输入）       ←→    state（完整状态）
事件处理函数            ←→    节点转换函数
setState({...})         ←→    return {"field": value}
中间件（数据转换）      ←→    预处理/后处理管道
```

你不会在 React 的 `onClick` 里直接修改 `props`，同样你也不应该在 LangGraph 节点里直接修改 `state`。你只是"描述"你想要的变化，框架帮你执行。

### 日常生活类比

状态转换函数就像餐厅厨师的工作流程：

```
餐厅厨房                      LangGraph 节点
─────────                    ──────────────
收到点单（完整订单）     ←→    接收完整 State
备料（清洗、切配）       ←→    数据预处理
烹饪（核心操作）         ←→    调用 LLM / 检索
装盘（摆盘、装饰）       ←→    输出格式化
出菜（只出这一道）       ←→    返回 Partial<State>
```

厨师不需要管其他桌的菜（其他字段），只专注做好自己这道菜（更新自己负责的字段），然后交给传菜员（框架）送到正确的位置。

### 场景 1 预览：基础节点转换模式

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
import operator

class RAGState(TypedDict):
    query: str
    cleaned_query: str
    context: Annotated[list[str], operator.add]
    answer: str

def clean_node(state: RAGState) -> dict:
    """预处理节点：清洗用户输入"""
    raw = state["query"]
    cleaned = raw.strip().lower()
    return {"cleaned_query": cleaned}

def retrieve_node(state: RAGState) -> dict:
    """检索节点：根据清洗后的查询检索文档"""
    query = state["cleaned_query"]
    # 模拟检索
    docs = [f"文档片段：关于 {query} 的内容"]
    return {"context": docs}

def generate_node(state: RAGState) -> dict:
    """生成节点：基于上下文生成回答"""
    context = "\n".join(state["context"])
    answer = f"基于检索到的内容，关于 '{state['cleaned_query']}' 的回答是..."
    return {"answer": answer}

# 构建图
builder = StateGraph(RAGState)
builder.add_node("clean", clean_node)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "clean")
builder.add_edge("clean", "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

graph = builder.compile()
result = graph.invoke({"query": "  什么是 RAG？  "})
print(result["answer"])
```

## 常见误区预警

### 误区 1：直接修改输入状态

```python
# ❌ 错误：直接修改 state
def bad_node(state):
    state["query"] = state["query"].strip()  # 直接修改！
    return state

# ✅ 正确：返回新的更新字典
def good_node(state):
    return {"cleaned_query": state["query"].strip()}
```

### 误区 2：混淆"转换"和"合并"

```python
# ❌ 错误理解：在节点里手动合并列表
def bad_node(state):
    old_docs = state["documents"]
    new_docs = search(state["query"])
    return {"documents": old_docs + new_docs}  # 手动合并

# ✅ 正确理解：让 Reducer 处理合并
# State 定义：documents: Annotated[list[str], operator.add]
def good_node(state):
    new_docs = search(state["query"])
    return {"documents": new_docs}  # 只返回新的，Reducer 会追加
```

### 误区 3：一个节点做太多事

```python
# ❌ 错误：一个节点包含清洗 + 检索 + 生成 + 格式化
def god_node(state):
    # 200 行代码...
    pass

# ✅ 正确：拆分为多个职责单一的节点
# clean_node -> retrieve_node -> generate_node -> format_node
```
