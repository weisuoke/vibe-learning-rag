# 09_超时控制 - 概览

> **定位**：L3_工作流编排 的第 9 个知识点
> **一句话**：把“等多久、在哪里等、等不到怎么办”从隐性约定变成显式设计。

---

## 这节到底在学什么？

在 LangGraph 里，超时控制不是一个孤立参数，而是一整套“预算系统”：

- 图运行时的 `step_timeout`，约束一个 orchestration step 最多等多久；
- 节点内部的 I/O timeout，约束 HTTP、数据库、工具调用、模型 SDK；
- 客户端 / 平台 timeout，约束 HTTP 请求、部署就绪、run 等待窗口；
- 业务上的 fallback、retry、interrupt、resume，决定超时后如何恢复。

如果你只记住“把 `graph.step_timeout = 30` 设上就行”，那在真正的生产场景里几乎一定会踩坑。

这节的目标是把这些边界拆开，让你知道：

1. **LangGraph 原生到底提供了什么 timeout 能力；**
2. **哪些 timeout 必须自己在节点里实现；**
3. **为什么前端超时、平台超时、图超时经常不是一回事；**
4. **如何把 timeout 和 retry / fallback / checkpoint / async polling 组合起来。**

## 你会遇到的四类超时问题

### 1. 节点自己太慢

最典型的情况：
- 一个 node 里调用外部 API；
- API 卡住 40 秒；
- 你没有给 HTTP client 配 timeout；
- 最后只能等到外层 `step_timeout` 或平台 timeout 来兜底。

这种情况的本质不是“图 timeout 配太小”，而是“节点内部没有 deadline”。

### 2. 同一个 step 里有并发任务，慢的拖住快的

LangGraph 的 Pregel 模型按 step 推进。一个 step 里如果同时起多个任务，往往要等这一轮都结束，下一轮状态更新才算提交。

所以常见现象是：
- 快节点 0.5 秒完成；
- 慢节点 8 秒才结束；
- 整个 step 还是要等慢节点；
- `step_timeout` 触发时，慢节点会被取消。

[来源: sourcecode/langgraph/libs/langgraph/langgraph/pregel/_runner.py]

### 3. 流式消费端自己慢，反向拖住执行

很多人会默认以为只有 node 慢才会 timeout。

但源码测试表明，在 `astream()` 场景中：
- 快节点先产出 chunk；
- 消费端处理 chunk 很慢；
- 慢节点还在继续；
- 最终整个 step 仍可能因为 deadline 耗尽而抛 `asyncio.TimeoutError`。

这就是“背压”问题。

[来源: atom/langgraph/L3_工作流编排/09_超时控制/reference/source_流式平台_03.md]

### 4. 图没超时，前端 / 平台先超时了

最真实的生产问题往往发生在这里：
- 图还在跑；
- 用户前端已经 network error；
- API gateway 已断开；
- 平台部署窗口已超过 300 秒；
- 刷新页面后又能看到最终结果。

这意味着超时发生在 **交互层或平台层**，不是图执行器层。

## 本知识点的核心地图

### 第一层：图运行时超时
- `step_timeout`
- `TimeoutError` / `asyncio.TimeoutError`
- 取消同 step 内未完成任务

### 第二层：节点内部 deadline
- `asyncio.wait_for()`
- `httpx.Timeout(...)`
- 工具 / SDK / DB 驱动自带 timeout

### 第三层：恢复策略
- `RetryPolicy`
- fallback 节点
- checkpoint / resume
- `interrupt`

### 第四层：平台与交互层
- SDK `timeout`
- run-level `timeout` 状态
- 前端等待窗口
- 异步任务 + polling

## 一张最重要的心智图

```text
用户请求
  ↓
前端 / API Gateway timeout
  ↓
LangGraph SDK timeout
  ↓
图运行时 step_timeout
  ↓
节点内 I/O timeout
  ↓
重试 / fallback / interrupt / resume
```

你要做的不是只改其中一层，而是给每一层都配上合理预算。

## 先记住结论

**LangGraph 的 timeout 控制，本质上不是“给图设一个秒数”，而是“给不同层级各自设预算，并设计超时后的恢复路径”。**
