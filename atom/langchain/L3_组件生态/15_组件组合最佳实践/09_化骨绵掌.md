## 为什么叫"化骨绵掌"？

化骨绵掌的精髓在于"看似绵软，实则刚猛"。组件组合也是如此——表面上只是用 `|` 连接几个组件，背后却蕴含着数据流设计、错误处理、性能优化等深层功力。

这10张卡片覆盖了组件组合中最容易被忽略、但在生产环境中至关重要的技巧。

## 10个2分钟知识卡片

### 卡片1：组合的本质 - 数据流图

**一句话：** LangChain 的组件组合本质上是在构建一个有向数据流图。

**举例：**

```python
# 这个链就是一个线性数据流图
chain = prompt | model | parser
# prompt → model → parser
#
# ┌────────┐    ┌───────┐    ┌────────┐
# │ prompt │ →  │ model │ →  │ parser │
# └────────┘    └───────┘    └────────┘
```

不同的组合原语对应不同的图结构：

| 组合原语 | 图结构 | 类比 |
|---------|--------|------|
| `RunnableSequence` (`\|`) | 线性链 | 流水线 |
| `RunnableParallel` | 分叉-汇合 | 并行车道 |
| `RunnableBranch` | 条件分支 | 十字路口 |
| `RunnablePassthrough.assign()` | 旁路增强 | 主路+辅路 |

**应用：** 理解了数据流图，就能理解为什么 RunnableParallel 是"分叉"，RunnableBranch 是"选择路径"。

### 卡片2：管道操作符 `|` 的魔法

**一句话：** `|` 操作符创建 RunnableSequence，自动展平嵌套序列。

**举例：**

```python
# 这两个是等价的
chain1 = a | b | c
chain2 = RunnableSequence(a, b, c)

# 嵌套会自动展平
chain3 = (a | b) | c  # 等价于 a | b | c，不会嵌套
```

**为什么这很重要？**

- 不用担心括号导致嵌套层级增加
- 调试时看到的是扁平结构，更容易理解
- 性能不会因为括号方式不同而有差异

**应用：** 不用担心括号导致嵌套，LangChain 会自动处理。

---

### 卡片3：Dict 字面量的自动转换

**一句话：** 在 `|` 链中使用 dict 字面量会自动转换为 RunnableParallel。

**举例：**

```python
# dict 自动变成 RunnableParallel
chain = step1 | {"a": step2, "b": step3} | step4
# 等价于
chain = step1 | RunnableParallel(a=step2, b=step3) | step4
```

**RAG 中的经典用法：**

```python
# 这里的 dict 就是自动转换的 RunnableParallel
rag_chain = (
    {"question": RunnablePassthrough(), "context": retriever}
    | prompt
    | model
    | parser
)
```

**注意：** dict 必须在管道 `|` 中才会自动转换，单独赋值给变量只是普通 dict。

**应用：** RAG 链中 `{"question": ..., "context": ...}` 就是利用了这个特性。

### 卡片4：RunnablePassthrough.assign() 的妙用

**一句话：** assign() 保留原始输入的所有字段，同时添加新字段。

**举例：**

```python
chain = RunnablePassthrough.assign(
    context=retriever,
    timestamp=lambda x: "2026-02-27",
)
# 输入: {"question": "test"}
# 输出: {"question": "test", "context": [...], "timestamp": "2026-02-27"}
```

**对比 RunnableParallel：**

```python
# RunnableParallel 需要显式传递 question
chain_parallel = RunnableParallel(
    question=RunnablePassthrough(),  # 必须手动传递
    context=retriever,
)

# assign() 自动保留所有原始字段
chain_assign = RunnablePassthrough.assign(context=retriever)
# question 自动保留，不需要显式传递
```

**链式 assign：**

```python
# 可以链式调用，逐步丰富数据
chain = (
    RunnablePassthrough.assign(context=retriever)
    | RunnablePassthrough.assign(summary=summarizer)
    | RunnablePassthrough.assign(score=scorer)
)
# 每一步都保留前面所有字段，同时添加新字段
```

**应用：** 比 RunnableParallel 更方便，不需要显式传递已有字段。

### 卡片5：Retry 的粒度原则

**一句话：** 重试范围越小越好，只在最可能失败的组件上添加重试。

**举例：**

```python
# ✅ 好：只重试 model
chain = prompt | model.with_retry() | parser

# ❌ 差：重试整个链
chain = (prompt | model | parser).with_retry()

# ❌ 更差：重试范围包含有副作用的步骤
chain = (save_to_db | model | parser).with_retry()
# 重试时 save_to_db 会重复执行！
```

**重试粒度决策表：**

| 组件类型 | 是否需要重试 | 原因 |
|---------|------------|------|
| Prompt 模板 | 否 | 纯计算，不会失败 |
| LLM API 调用 | 是 | 网络波动、限流 |
| 输出解析器 | 视情况 | 格式不对可重试 |
| 向量检索 | 视情况 | 数据库连接可能断 |
| 自定义函数 | 视情况 | 取决于是否有副作用 |

**应用：** LLM API 调用是最可能失败的环节，重试应该加在这里。

### 卡片6：Fallback 的 __getattr__ 代理

**一句话：** 对 fallback 对象调用方法时，会自动对所有 fallback 调用相同方法。

**举例：**

```python
model = gpt4.with_fallbacks([claude])
# 调用 bind_tools 会同时对 gpt4 和 claude 调用
model.bind_tools([my_tool])
# 等价于
gpt4.bind_tools([my_tool]).with_fallbacks([claude.bind_tools([my_tool])])
```

**这意味着你可以先配置 fallback，再配置其他选项：**

```python
model = (
    gpt4
    .with_fallbacks([gpt4_mini])
    .bind(temperature=0.7)        # 自动应用到所有 fallback
    .bind_tools([search_tool])     # 自动应用到所有 fallback
)
```

> 注意：只有 Runnable 接口的方法会被代理，自定义方法不会。

**应用：** 不需要手动对每个 fallback 模型单独配置工具。

### 卡片7：流式传播的阻断点

**一句话：** RunnableLambda 不支持 transform，会阻断流式传播。

**举例：**

```python
# ❌ lambda 阻断流式
chain = prompt | model | RunnableLambda(process) | parser
# model 的流式输出会被 RunnableLambda 阻断
# 用户看到的是：等待...等待...一次性输出全部结果

# ✅ RunnableGenerator 保持流式
def process_stream(input_stream):
    for chunk in input_stream:
        yield transform(chunk)

chain = prompt | model | RunnableGenerator(process_stream)
# 用户看到的是：逐字输出，体验流畅
```

**流式传播能力对照表：**

| 组件 | 支持流式传播 | 说明 |
|------|------------|------|
| ChatModel | ✅ | 原生支持 |
| StrOutputParser | ✅ | 逐块解析 |
| RunnableLambda | ❌ | 等待全部输入 |
| RunnableGenerator | ✅ | 逐块处理 |
| RunnableParallel | ✅ | 并行流式 |
| PromptTemplate | ❌ | 但通常不影响（瞬时完成） |

**应用：** 需要流式 + 自定义处理时，用 RunnableGenerator 替代 RunnableLambda。

### 卡片8：batch 的智能重试

**一句话：** RunnableRetry 的 batch 只重试失败的元素，已成功的不会重复执行。

**举例：**

```python
# 假设 batch 3 个输入，第2个失败
results = chain.with_retry().batch([input1, input2, input3])
# 重试时只重新执行 input2，input1 和 input3 的结果保留
```

**为什么这很重要？**

```
场景：批量处理 1000 个文档的 embedding
- 第 500 个因为网络波动失败了
- 如果重试整个 batch → 重新计算 999 个已成功的 = 浪费钱
- 智能重试 → 只重新计算第 500 个 = 省钱省时间
```

| 策略 | 1000个输入，1个失败 | 额外成本 |
|------|-------------------|---------|
| 重试整个 batch | 重新执行 1000 个 | 约 $1.00 |
| 智能重试 | 只重新执行 1 个 | 约 $0.001 |

**应用：** 批量处理时不用担心重试导致重复计算和额外成本。

### 卡片9：ConfigurableField 运行时切换

**一句话：** configurable_fields 让你在不重建链的情况下切换组件参数。

**举例：**

```python
model = ChatOpenAI(model="gpt-4o-mini").configurable_fields(
    model_name=ConfigurableField(id="model_name")
)

# 运行时切换
chain.invoke(input, config={"configurable": {"model_name": "gpt-4o"}})
```

**实际应用场景：**

```python
# 场景1：A/B 测试
def handle_request(user_input, experiment_group):
    model_name = "gpt-4o" if experiment_group == "A" else "gpt-4o-mini"
    return chain.invoke(
        user_input,
        config={"configurable": {"model_name": model_name}},
    )

# 场景2：按用户等级选择模型
def handle_by_tier(user_input, user_tier):
    model_map = {"free": "gpt-4o-mini", "pro": "gpt-4o"}
    return chain.invoke(
        user_input,
        config={"configurable": {"model_name": model_map[user_tier]}},
    )
```

**应用：** A/B 测试、按用户等级使用不同模型、动态调整温度。

### 卡片10：Graph 可视化验证管道

**一句话：** 用 get_graph() 可视化管道结构，验证组合是否正确。

**举例：**

```python
chain.get_graph().print_ascii()  # 终端 ASCII 图
chain.get_graph().draw_mermaid()  # Mermaid 图
```

**调试技巧：**

```python
# 检查节点数量是否符合预期
graph = chain.get_graph()
print(f"节点数: {len(graph.nodes)}")
print(f"边数: {len(graph.edges)}")

# 检查输入输出 schema
print(f"输入: {chain.input_schema.schema()}")
print(f"输出: {chain.output_schema.schema()}")
```

**应用：** 复杂管道容易出错，可视化是最快的验证方式。

## 组合技：卡片联合使用

### 组合技1：生产级 RAG 管道（卡片 4 + 5 + 6）

```python
# Fallback 模型（卡片6）
model = ChatOpenAI(model="gpt-4o").with_fallbacks(
    [ChatOpenAI(model="gpt-4o-mini")]
)

# 只在模型上加重试（卡片5）
model = model.with_retry(stop_after_attempt=3)

# 用 assign 逐步丰富数据（卡片4）
chain = (
    RunnablePassthrough.assign(context=retriever)
    | prompt
    | model
    | parser
)
```

### 组合技2：可配置的流式管道（卡片 7 + 9 + 10）

```python
# 可配置模型（卡片9）
model = ChatOpenAI(model="gpt-4o-mini").configurable_fields(
    model_name=ConfigurableField(id="model_name"),
)

# 流式处理（卡片7）
def format_stream(chunks):
    for chunk in chunks:
        yield f"[AI] {chunk.content}" if hasattr(chunk, "content") else chunk

chain = prompt | model | RunnableGenerator(format_stream)

# 可视化验证（卡片10）
chain.get_graph().print_ascii()
```
