# 最小可用

掌握下面 5 个点，你就已经能把 LangGraph 的 timeout 设计从“能跑”提升到“基本靠谱”。

---

## 4.1 先配 `step_timeout`，但别把它当万能方案

```python
graph = builder.compile()
graph.step_timeout = 10
```

这一步解决的是：
- 一个 step 里并发任务无限等待；
- 某个 node 卡死后把整轮 orchestration 拖住。

但它解决不了：
- 节点内部 HTTP 请求读超时；
- 平台 / SDK / 前端等待窗口；
- 人工审批类长等待。

**实际应用：**
- 给内部工具调用图加一个 5~20 秒的步级预算；
- 在测试环境更激进一些，尽早暴露卡顿。

[来源: atom/langgraph/L3_工作流编排/09_超时控制/reference/source_步级超时_01.md]

---

## 4.2 节点内部 I/O 一定要单独配 timeout

```python
import asyncio

async def call_api():
    return await asyncio.wait_for(fake_remote_call(), timeout=1.5)
```

或者：

```python
import httpx

timeout = httpx.Timeout(10.0, connect=2.0, read=6.0, write=4.0, pool=2.0)
client = httpx.AsyncClient(timeout=timeout)
```

**为什么必须这样做？**

因为 `step_timeout` 是外层 step budget，I/O timeout 是内层 resource budget。

**实际应用：**
- LLM 调用：给 read timeout 更宽；
- 搜索 API：connect timeout 更短；
- 内网服务：pool timeout 也要配。

[来源: atom/langgraph/L3_工作流编排/09_超时控制/reference/context7_httpx_02.md]

---

## 4.3 timeout 后要立即决定：重试、降级还是失败

```python
async def tool_node(state):
    try:
        result = await asyncio.wait_for(fake_tool(), timeout=1.0)
        return {"result": result, "timed_out": False}
    except asyncio.TimeoutError:
        return {"result": "fallback", "timed_out": True}
```

如果不做这一步，timeout 只会把错误从“慢失败”变成“快失败”。

**实际应用：**
- 搜索超时 → 走缓存结果；
- 模型调用超时 → 切备用模型；
- 非关键步骤超时 → 返回部分结果。

---

## 4.4 长运行等待不要强塞进同步请求

官方文档和社区案例都说明：
- 人类等待用 `interrupt`；
- 长任务用 durable execution / resume；
- 前端用 async run + polling；
- 不要让浏览器挂着等几分钟。

```python
from langgraph.types import interrupt

def approval_node(state):
    approved = interrupt("Approve?")
    return {"approved": approved}
```

**实际应用：**
- 报表生成、审批、人工复核；
- 外部批处理任务完成通知；
- 多分钟级 agent workflow。

[来源: atom/langgraph/L3_工作流编排/09_超时控制/reference/context7_langgraph_01.md]

## 4.5 把 timeout 和 retry / recursion limit 一起设计

```python
from langgraph.types import RetryPolicy

builder.add_node(
    "search",
    search_node,
    retry_policy=RetryPolicy(max_attempts=3, initial_interval=1.0),
)
```

同时你还应该考虑：
- `graph.step_timeout`
- `RetryPolicy.max_attempts`
- `recursion_limit`
- fallback route

**为什么？**

因为：
- 只有 timeout 没 retry，瞬态故障恢复不了；
- 只有 retry 没 timeout，失败会拖很久；
- 只有 recursion limit 没 timeout，单步仍可卡住。

[来源: atom/langgraph/L3_工作流编排/09_超时控制/reference/search_超时控制_01.md]

## 这些知识已经足够你做什么？

- 给 LangGraph 图加上步级兜底 timeout；
- 在节点内部为网络调用设置 deadline；
- 为 timeout 设计 fallback 分支；
- 识别“前端 timeout”和“图 timeout”不是同一个问题；
- 用 interrupt / polling 替代超长同步等待。

---

## 最小检查清单

上线前至少问自己 6 个问题：

1. 图有 `step_timeout` 吗？
2. 节点内网络调用有 timeout 吗？
3. timeout 后是 retry、fallback 还是直接失败？
4. 长任务是否应该改为 async run + polling？
5. 是否区分了 SDK / 平台 / 图 / 节点四层 timeout？
6. 是否记录了 timeout 指标和日志？

如果这 6 个问题有 3 个以上答不上来，说明 timeout 设计还没达标。
