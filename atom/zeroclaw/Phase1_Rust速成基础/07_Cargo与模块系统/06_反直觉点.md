> **知识点**: Cargo 与模块系统
> **层级**: Phase1_Rust速成基础
> **维度**: 反直觉点（5 个常见误区）

## 概述

从 TypeScript/npm 转到 Rust/Cargo，有 5 个**几乎人人都会踩的坑**。它们的共同特点是：你在 TypeScript 里养成的直觉，在 Rust 里会把你带沟里。

本节用 **❌ 错误观点 → 🔍 为什么错 → ✅ 正确理解** 的三段式，帮你一次性避坑。

## 误区1：文件创建了就自动变成模块 ❌

### 错误观点

> "我在 `src/` 下创建了 `utils.rs`，就可以直接 `use crate::utils` 了"

### 为什么错？

Rust 的模块系统需要**显式声明**。仅仅创建文件是不够的，你必须在 `lib.rs` 或 `main.rs` 中用 `mod utils;` 告诉编译器"这个模块存在"。

这与 TypeScript 的自动发现机制**完全不同**。

### 为什么人们容易这样想？

TypeScript/Python/Go 都是**文件 = 模块**，创建文件就能 import。你的本能反应是"创建文件 → 直接 use"。

### 错误示范

```
src/
├── main.rs
└── utils.rs    ← 创建了文件
```

```rust
// src/utils.rs
pub fn greet(name: &str) -> String {
    format!("Hello, {}!", name)
}

// src/main.rs
// ❌ 编译错误！编译器不知道 utils 模块的存在
use crate::utils::greet;

fn main() {
    println!("{}", greet("World"));
}
```

编译器报错：`error[E0433]: could not find 'utils' in the crate root`

### 正确做法

```rust
// src/main.rs
mod utils;  // ← 必须显式声明！告诉编译器去找 src/utils.rs

use crate::utils::greet;  // 现在才能用 ✅

fn main() {
    println!("{}", greet("World"));
}
```

### 对比总结

| | TypeScript | Rust |
|---|---|---|
| 创建 `utils.ts` / `utils.rs` | 立即可以 `import` | ❌ 编译错误 |
| 额外步骤 | 无 | 需要在根文件中 `mod utils;` |
| 忘记声明？ | 不可能 | **最常见的新手错误** |

### 为什么 Rust 选择显式声明？

**可预测性**——看 `lib.rs` 就知道项目有哪些模块，不用扫描文件系统。**安全性**——不会意外把测试文件、临时文件编译进去。**编译性能**——编译器只处理声明过的文件。

> 💡 **ZeroClaw 的 `src/lib.rs` 就是整个项目的"目录页"**——30+ 个 `pub mod` 声明清晰地列出了所有模块。看一眼就知道项目的全部结构，不需要 `ls src/` [^1]。

## 误区2：pub 等于 export，不加 pub 等于 private ❌

### 错误观点

> "Rust 的 `pub` 和 TypeScript 的 `export` 一模一样"

### 为什么错？

这个类比在表面上成立，但有 **3 个关键区别** 会让你掉进坑里：

**区别1：`pub` 有多个级别**

TypeScript 只有两种：`export`（公开）或不 `export`（私有）。Rust 有 **5 个级别**：

```rust
// Rust 的 5 级可见性
pub fn everyone_sees_me() {}        // 完全公开（所有 crate 可见）
pub(crate) fn crate_only() {}       // 仅本 crate 内可见
pub(super) fn parent_only() {}      // 仅父模块可见
pub(in crate::api) fn api_only() {} // 仅指定路径可见
fn truly_private() {}               // 仅当前模块（及子模块）可见
```

TypeScript 没有 `pub(crate)` 的等价物——你要么 `export`，要么不 `export`，没有"只在这个 npm 包内部可见"这回事 [^2]。

**区别2：struct 的字段有独立的可见性**

这是**最坑人的**。在 TypeScript 里，`export` 一个 class，它的所有属性都跟着暴露了。Rust 不是这样：

```rust
// Rust —— struct 是 pub 的，但字段不一定是！
pub struct User {
    pub name: String,       // ✅ 外部可以访问
    pub email: String,      // ✅ 外部可以访问
    password_hash: String,  // 🔒 私有！外部不能访问，也不能直接构造
}
```

```typescript
// TypeScript —— export class，所有属性默认暴露
export class User {
    name: string;         // 都可以访问
    passwordHash: string; // 也可以访问（private 只是约定，不是编译时强制）
}
```

**Rust 的坑在这里**：如果 struct 有任何私有字段，外部代码就**无法直接构造**这个 struct：

```rust
// 在另一个模块里 —— ❌ 编译错误！password_hash 是私有的
let user = User {
    name: "Alice".to_string(),
    email: "alice@example.com".to_string(),
    password_hash: "abc123".to_string(),  // 无法设置私有字段
};
```

正确做法——提供构造函数：

```rust
impl User {
    pub fn new(name: String, email: String, raw_password: &str) -> Self {
        Self {
            name,
            email,
            password_hash: hash_password(raw_password),  // 内部处理
        }
    }
}
```

**区别3：子模块可以访问父模块的私有成员**

这在 TypeScript 里是不可能的，但 Rust 允许：

```rust
mod outer {
    fn secret() -> &'static str { "隐藏数据" }  // 私有函数

    pub mod inner {
        pub fn reveal() {
            // ✅ 子模块可以通过 super:: 访问父模块的私有成员！
            println!("秘密是: {}", super::secret());
        }
    }
}

fn main() {
    outer::inner::reveal();  // ✅ 输出: 秘密是: 隐藏数据
    // outer::secret();      // ❌ 编译错误：secret 是私有的
}
```

```typescript
// TypeScript —— 不可能！文件模块之间没有父子访问特权
// outer/index.ts
function secret(): string { return "隐藏数据"; }  // 没有 export
// outer/inner.ts
// ❌ 无法访问 secret()，因为它没有 export
```

### 可见性完整对比

| 场景 | TypeScript | Rust |
|------|-----------|------|
| 完全公开 | `export` | `pub` |
| 包内可见 | ❌ 不存在 | `pub(crate)` |
| 父级可见 | ❌ 不存在 | `pub(super)` |
| 完全私有 | 不 `export` | 不加修饰符 |
| 字段可见性 | 跟随 class | **独立控制** |
| 子访问父私有 | ❌ 不可以 | ✅ 可以 |

> 💡 **记住**：`pub` ≈ `export`，但 Rust 的可见性是**多级的、细粒度的**。特别是 struct 字段的独立可见性——这是 TypeScript 开发者最容易忽略的点 [^2]。

## 误区3：Feature Flags 可以互斥使用 ❌

### 错误观点

> "我可以定义两个互斥的 feature，让用户选一个。比如 `backend-postgres` 和 `backend-sqlite`，用户只能二选一"

### 为什么错？

Rust 的 feature flags 有一条**铁律**：**Feature 必须是可加的（additive）**。

什么意思？如果开启 feature A 能编译，开启 feature B 也能编译，那么**同时开启 A + B 也必须能编译**。

为什么会同时开启？因为 Cargo 有 **feature unification（特性统一）** 机制：

```
你的项目
├── 依赖 crate-x（需要 my-lib 的 feature "postgres"）
└── 依赖 crate-y（需要 my-lib 的 feature "sqlite"）

→ Cargo 自动合并：my-lib 同时开启 postgres + sqlite！
```

在 workspace 中也一样——不同成员 crate 依赖同一个库的不同 feature，Cargo 会**取并集** [^3]。

### 错误示范

```toml
# Cargo.toml —— ❌ 错误设计！互斥 feature
[features]
backend-postgres = ["dep:sqlx-postgres"]
backend-sqlite = ["dep:sqlx-sqlite"]
```

```rust
// src/db.rs —— ❌ 互斥的条件编译
#[cfg(feature = "backend-postgres")]
pub use postgres_impl::connect;

#[cfg(feature = "backend-sqlite")]
pub use sqlite_impl::connect;  // ❌ 两个 feature 同时开启时，重复定义 connect！
```

### 正确做法1：设计为可加的

```rust
// src/db.rs —— ✅ 正确设计：feature 可加
#[cfg(feature = "backend-postgres")]
pub mod postgres;  // PostgresConnection

#[cfg(feature = "backend-sqlite")]
pub mod sqlite;    // SqliteConnection

// 两个模块有不同的类型名，不冲突
// 同时开启 = 同时可用，用户按需选择
```

### 正确做法2：用 compile_error! 友好报错

如果你的场景真的不支持同时开启，用 `compile_error!` 宏给出清晰的错误信息：

```rust
// src/lib.rs —— ✅ 如果不得已需要互斥，给出清晰报错
#[cfg(all(feature = "backend-postgres", feature = "backend-sqlite"))]
compile_error!(
    "Feature `backend-postgres` 和 `backend-sqlite` 不能同时启用。\
     请只选择一个数据库后端。\
     例如: cargo build --features backend-postgres"
);
```

### TypeScript 对比

```typescript
// TypeScript / Webpack —— 运行时判断，可以做互斥
if (process.env.DB_BACKEND === "postgres") {
    const { connect } = await import("./postgres");
} else {
    const { connect } = await import("./sqlite");
}
```

Rust 的 feature 是**编译时**的——要么编译，要么不编译，没有"打包进去但不用"这回事。

### Feature Flags 三条铁律

| 规则 | 说明 |
|------|------|
| **必须可加** | A 能编译 + B 能编译 → A+B 必须能编译 |
| **不能互斥** | workspace 中 feature 会被自动合并（取并集） |
| **默认最小** | `default = []` 让用户按需启用 |

> 💡 **ZeroClaw 的 15+ feature flags 全部遵循可加原则**：`hardware` + `channel-matrix` + `rag-pdf` 可以任意组合，不冲突。因为每个 feature 只引入独立的模块和依赖，互不干扰 [^1][^3]。

## 误区4：Cargo.lock 应该放在 .gitignore 里 ❌

### 错误观点

> "Cargo.lock 就像 node_modules 一样，是生成的产物，不应该提交到 git"

### 为什么错？

首先，Cargo.lock 不是 node_modules——它更像 **package-lock.json**。但规则不完全一样：

| 项目类型 | 是否提交 Cargo.lock | 原因 |
|----------|-------------------|------|
| **应用程序**（bin crate） | ✅ **必须提交** | 确保所有环境用完全相同的依赖版本 |
| **库**（lib crate） | ❌ 不提交 | 库的用户有自己的 Cargo.lock |

### 为什么应用程序必须提交？

```
场景：你的团队在开发一个命令行工具

没有 Cargo.lock：
  你：    cargo build → 拉到 serde 1.0.193
  同事：  cargo build → 拉到 serde 1.0.210（一周后发布的新版本）
  CI：    cargo build → 拉到 serde 1.0.215（又更新了）
  → 三个人构建出不同的二进制！可能有人能跑，有人 bug。

有 Cargo.lock：
  你：    cargo build → serde 1.0.193 ← 锁定
  同事：  cargo build → serde 1.0.193 ← 相同
  CI：    cargo build → serde 1.0.193 ← 相同
  → 可重复构建（Reproducible Build）✅
```

### 为什么库不提交？

库的 Cargo.lock 对用户没用——Cargo 会忽略依赖库的 lockfile，统一用用户自己的 Cargo.lock 解析版本。所以库提交 Cargo.lock 毫无意义。

### TypeScript 对比

| | npm / pnpm | Cargo |
|---|---|---|
| lockfile | `package-lock.json` | `Cargo.lock` |
| 应用要提交吗 | ✅ 是 | ✅ 是 |
| 库要提交吗 | 争议（通常不提交） | ❌ 官方明确不提交 |
| 产物目录 | `node_modules/`（不提交） | `target/`（不提交） |

### 正确的 .gitignore

```gitignore
# .gitignore for a Rust 应用程序（bin crate）
/target/
# 注意：Cargo.lock 不在这里！应该提交！

# .gitignore for a Rust 库（lib crate）
/target/
Cargo.lock    # 库不提交 lockfile
```

> 💡 **一句话总结**：`target/` ≈ `node_modules/`（永远不提交）。`Cargo.lock` ≈ `package-lock.json`（应用提交，库不提交）[^3]。

## 误区5：模块越深越好，文件夹嵌套越多越清晰 ❌

### 错误观点

> "按照 Java 的包命名习惯，深度嵌套更清晰。比如 `crate::services::user::auth::oauth::google::callback`"

### 为什么错？

深层嵌套在 Rust 中带来的问题远比好处多：

**问题1：路径太长，写起来痛苦**

```rust
// ❌ 深度嵌套 —— 6 层
use crate::services::user::auth::oauth::google::callback;
use crate::services::user::auth::oauth::github::callback as gh_callback;

// ✅ 浅层级 —— 最多 2 层
use crate::auth::google;
use crate::auth::github;
```

**问题2：每一层都需要 mod 声明，维护成本高**

6 层嵌套 = 每层都要一个 `mod.rs`，每个 `mod.rs` 都要声明子模块。2 层只需要 `src/auth/mod.rs → pub mod google; pub mod github;`，完了！

**问题3：编译单位是 crate，不是模块**

在 TypeScript/Node.js 里，你可能把功能分到很深的目录，因为 bundler 会处理。但在 Rust 里，**模块不是编译单位——crate 才是**。深嵌套不会给你更好的编译隔离或更快的编译速度。

### 正确做法：拆分 crate 而不是嵌套模块

当模块太大时，Rust 社区的共识是**拆成独立 crate**，而不是继续嵌套 [^3]：

```
# ✅ 推荐结构：浅模块 + 多 crate
my-project/
├── Cargo.toml                 # workspace
├── crates/
│   ├── auth/                  # 独立 crate
│   │   ├── Cargo.toml
│   │   └── src/
│   │       ├── lib.rs
│   │       ├── google.rs      # 最多 1 层
│   │       └── github.rs
│   └── notifications/         # 独立 crate
│       ├── Cargo.toml
│       └── src/
│           ├── lib.rs
│           └── email.rs
└── src/
    ├── lib.rs
    └── main.rs
```

### 为什么拆 crate 比嵌套模块好？

| | 深嵌套模块 | 拆分 crate |
|---|---|---|
| 编译速度 | 改一个文件，整个 crate 重新编译 | 只重新编译改动的 crate ✅ |
| 隐私边界 | `pub(crate)` 可以跨模块访问 | crate 之间有**硬隔离** ✅ |
| 可复用性 | 只能在本项目用 | 可以独立发布到 crates.io ✅ |
| 路径长度 | `crate::a::b::c::d::e` | `auth::google` ✅ |

### 社区推荐的层级数

```
模块层级         推荐程度
─────────────────────────
1 层              ✅ 最佳（src/auth.rs）
2 层              ✅ 常见（src/auth/google.rs）
3 层              ⚠️ 考虑拆 crate
4+ 层             ❌ 几乎一定应该拆 crate
```

> 💡 **ZeroClaw 的做法**：30+ 个顶层模块（`pub mod agent`、`pub mod tools` 等）几乎都是 **1-2 层**。当功能足够独立时，直接拆成 `crates/robot-kit` 作为独立 crate——而不是嵌套成 `src/hardware/robot/kit/` [^1][^3]。

## 5 个误区速查表

| 误区 | 直觉（来自 TS/JS） | Rust 的真相 |
|------|-------------------|-------------|
| **文件 = 模块** | 创建文件就能 import | 必须 `mod xxx;` 显式声明 |
| **pub = export** | 二元的（公开/私有） | 5 级可见性 + 字段独立控制 |
| **Feature 可互斥** | 运行时条件判断 | 编译时必须可加（additive） |
| **不提交 Cargo.lock** | 类似 node_modules | 应用必须提交，库不提交 |
| **模块越深越好** | 深嵌套 + path alias | 最多 2 层，大了就拆 crate |

## 避坑检查清单

每次写 Rust 项目时，对照检查：

- [ ] 新建 `.rs` 文件后，是否在 `lib.rs` / `main.rs` / `mod.rs` 中添加了 `mod xxx;`？
- [ ] struct 有私有字段时，是否提供了 `pub fn new()` 构造函数？
- [ ] feature flags 是否都是可加的？同时开启所有 feature 能否编译？
- [ ] 应用程序的 Cargo.lock 是否已提交到 git？
- [ ] 模块嵌套是否超过 2 层？超过了就考虑拆成独立 crate。
