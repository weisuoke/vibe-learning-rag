# 节点函数约定 - 30字核心

## 30字核心定义

**节点函数是 LangGraph 中定义状态转换逻辑的函数，遵循特定签名约定，是构建状态化工作流的基本单元。**

[来源: reference/source_节点函数约定_01.md, reference/context7_langgraph_01.md]

---

## 核心定义的详细解释

### 1. 什么是节点函数？

节点函数（Node Function）是 LangGraph 图中的**执行单元**，每个节点函数负责：

1. **接收当前状态**：从图的状态流中获取输入
2. **执行业务逻辑**：处理数据、调用 API、执行计算等
3. **返回状态更新**：将处理结果返回给图，更新状态

**类比理解**：
- **前端类比**：节点函数就像 React 组件函数，接收 props（状态），执行渲染逻辑，返回新的 UI 状态
- **日常生活类比**：节点函数就像流水线上的工序站点，接收半成品（当前状态），执行加工操作，输出加工后的产品（新状态）

[来源: reference/context7_langgraph_01.md]

### 2. 为什么需要"特定签名约定"？

LangGraph 对节点函数的签名有严格约定，这是为了：

#### 2.1 类型安全与自动推断

```python
# LangGraph 会自动推断节点函数的输入输出类型
def my_node(state: State) -> dict:
    return {"key": "value"}

# 源码中的类型推断逻辑
if (
    isfunction(action)
    or ismethod(action)
    or ismethod(getattr(action, "__call__", None))
) and (
    hints := get_type_hints(getattr(action, "__call__"))
    or get_type_hints(action)
):
    if input_schema is None:
        first_parameter_name = next(
            iter(
                inspect.signature(
                    cast(FunctionType, action)
                ).parameters.keys()
            )
        )
        if input_hint := hints.get(first_parameter_name):
            if isinstance(input_hint, type) and get_type_hints(input_hint):
                inferred_input_schema = input_hint
```

[来源: reference/source_节点函数约定_01.md]

**关键点**：
- LangGraph 通过 Python 的类型注解自动推断节点的输入输出类型
- 第一个参数的类型被推断为 `input_schema`
- 这使得图可以在编译时进行类型检查，避免运行时错误

#### 2.2 统一的执行模型

所有节点函数遵循相同的签名约定，使得 LangGraph 可以：
- **统一调度**：用相同的方式调用所有节点
- **统一错误处理**：用相同的机制处理所有节点的异常
- **统一重试逻辑**：用相同的策略重试失败的节点

```python
# 源码中的节点执行逻辑（同步版本）
def run_with_retry(
    task: PregelExecutableTask,
    retry_policy: Sequence[RetryPolicy] | None,
    configurable: dict[str, Any] | None = None,
) -> None:
    """Run a task with retries."""
    retry_policy = task.retry_policy or retry_policy
    attempts = 0
    config = task.config
    if configurable is not None:
        config = patch_configurable(config, configurable)
    while True:
        try:
            # clear any writes from previous attempts
            task.writes.clear()
            # run the task
            return task.proc.invoke(task.input, config)
        except Exception as exc:
            # 统一的错误处理和重试逻辑
            if not retry_policy:
                raise
            # ... 重试逻辑
```

[来源: reference/source_节点函数约定_02.md]

#### 2.3 灵活的参数支持

节点函数支持多种参数组合：

```python
# 基础签名：只接收状态
def node1(state: State) -> dict:
    return {"result": state["x"] + 1}

# 带 Config 参数：访问运行时配置
def node2(state: State, config: RunnableConfig) -> dict:
    model = config.get("configurable", {}).get("model", "gpt-4")
    return {"model": model}

# 带 Runtime 参数：访问运行时上下文
def node3(state: State, runtime: Runtime[Context]) -> dict:
    r = runtime.context.get("r", 1.0)
    return {"r": r}
```

[来源: reference/source_节点函数约定_01.md]

**关键点**：
- `state`：必需参数，接收当前图状态
- `config`：可选参数，访问运行时配置（如模型选择、API 密钥等）
- `runtime`：可选参数，访问运行时上下文（如共享资源、全局变量等）

### 3. 为什么是"状态化工作流的基本单元"？

#### 3.1 状态转换的原子操作

每个节点函数代表一个**原子的状态转换操作**：

```
状态 S1 --[节点函数 N1]--> 状态 S2 --[节点函数 N2]--> 状态 S3
```

**类比**：
- **前端类比**：就像 Redux 中的 reducer，每个 reducer 负责一个状态切片的更新
- **日常生活类比**：就像做菜的步骤，每个步骤（洗菜、切菜、炒菜）都是一个独立的操作，但组合起来完成整道菜

[来源: reference/context7_langgraph_01.md]

#### 3.2 可组合性

节点函数可以通过边（Edge）连接，形成复杂的工作流：

```python
# 简单的线性流程
builder.add_node("step1", step1_func)
builder.add_node("step2", step2_func)
builder.add_node("step3", step3_func)
builder.add_edge("step1", "step2")
builder.add_edge("step2", "step3")

# 条件分支流程
builder.add_conditional_edges(
    "step1",
    decide_next_step,  # 路由函数
    {
        "path_a": "step2a",
        "path_b": "step2b"
    }
)
```

[来源: reference/context7_langgraph_01.md]

#### 3.3 独立性与可测试性

每个节点函数是独立的，可以单独测试：

```python
# 测试单个节点函数
def test_my_node():
    state = {"x": 1}
    result = my_node(state)
    assert result == {"x": 2}
```

**优势**：
- **单元测试友好**：每个节点可以独立测试
- **调试方便**：可以单独运行某个节点查看输出
- **复用性强**：同一个节点函数可以在多个图中使用

## 为什么这个定义重要？

### 1. 理解 LangGraph 的核心抽象

节点函数是 LangGraph 的**核心抽象**，理解节点函数约定是掌握 LangGraph 的第一步。

**类比**：
- 就像学习 React 必须先理解组件（Component）
- 就像学习 Redux 必须先理解 reducer
- 就像学习 LangChain 必须先理解 Runnable

如果不理解节点函数约定，你会遇到：
- ❌ 不知道如何定义节点函数的参数和返回值
- ❌ 不理解为什么有些节点函数报错
- ❌ 不知道如何处理异步操作和错误

[来源: reference/context7_langgraph_01.md]

### 2. 避免常见错误

理解节点函数约定可以避免很多常见错误：

#### 错误1：返回值类型不正确

```python
# ❌ 错误：返回完整状态对象
def my_node(state: State) -> State:
    state["x"] = state["x"] + 1
    return state  # 错误：不应该返回完整状态

# ✅ 正确：返回部分状态更新
def my_node(state: State) -> dict:
    return {"x": state["x"] + 1}  # 正确：只返回需要更新的字段
```

[来源: reference/context7_langgraph_01.md]

**为什么？**
- LangGraph 使用 Reducer 函数合并状态更新
- 返回完整状态会覆盖其他节点的更新
- 返回部分更新可以与其他节点的更新合并

#### 错误2：使用异步生成器

```python
# ❌ 不推荐：使用异步生成器（可能有问题）
async def my_async_generator_node(state: State):
    async for item in some_async_generator():
        yield {"item": item}

# ✅ 推荐：使用普通异步函数
async def my_async_node(state: State) -> dict:
    result = await some_async_api()
    return {"result": result}
```

[来源: reference/search_节点函数约定_01.md]

**为什么？**
- 异步生成器节点在某些情况下函数体不执行
- yield 流式输出可能失效
- 错误处理机制可能失效

#### 错误3：不处理异常

```python
# ❌ 错误：不处理异常
def my_node(state: State) -> dict:
    result = risky_operation()  # 可能抛出异常
    return {"result": result}

# ✅ 正确：配置 RetryPolicy
builder.add_node(
    "my_node",
    my_node,
    retry_policy=RetryPolicy(
        max_attempts=3,
        initial_interval=1.0,
        backoff_factor=2.0,
        retry_on=lambda e: isinstance(e, (ConnectionError, TimeoutError))
    )
)

# ✅ 或者：在节点内部处理异常
def my_node_with_error_handling(state: State) -> dict:
    try:
        result = risky_operation()
        return {"result": result, "error": None}
    except Exception as e:
        return {"result": None, "error": str(e)}
```

[来源: reference/source_节点函数约定_02.md, reference/search_节点函数约定_01.md]

### 3. 构建健壮的工作流

理解节点函数约定可以帮助你构建更健壮的工作流：

#### 3.1 错误恢复

```python
# 配置重试策略
retry_policy = RetryPolicy(
    max_attempts=5,
    initial_interval=0.5,
    backoff_factor=2.0,
    max_interval=10.0,
    jitter=True,
    retry_on=lambda e: "rate limit" in str(e).lower()
)

builder.add_node("api_call", api_call_node, retry_policy=retry_policy)
```

[来源: reference/source_节点函数约定_02.md]

**关键点**：
- `max_attempts`：最大重试次数
- `initial_interval`：初始重试间隔（秒）
- `backoff_factor`：退避因子（指数退避）
- `max_interval`：最大重试间隔（秒）
- `jitter`：是否添加随机抖动（避免雷鸣群效应）
- `retry_on`：重试条件（异常类型或自定义函数）

#### 3.2 人机协作

```python
from langgraph.types import interrupt

def collect_info(state: FormState) -> dict:
    """Collect user information with interrupts."""
    name = interrupt("Please enter your name:")
    email = interrupt("Please enter your email:")
    return {"name": name, "email": email}
```

[来源: reference/context7_langgraph_01.md]

**关键点**：
- `interrupt()` 函数暂停图执行
- 需要 checkpointer 保存状态
- 使用 `Command(resume=...)` 恢复执行

#### 3.3 动态路由

```python
from langgraph.types import Command
from typing import Literal

def my_command_node(state: State) -> Command[Literal["next", "end"]]:
    if state["x"] > 10:
        return Command(goto="end")
    return Command(goto="next", update={"x": state["x"] + 1})
```

[来源: reference/source_节点函数约定_01.md]

**关键点**：
- 使用 `Command` 对象控制流程跳转
- 使用 `Literal` 类型注解指定可能的目标节点
- 可以同时更新状态和跳转节点

## 与其他知识点的关系

### 1. 与 StateGraph 的关系

**依赖关系**：节点函数是 StateGraph 的组成部分

```python
from langgraph.graph import StateGraph

# StateGraph 定义图结构
builder = StateGraph(State)

# 节点函数定义状态转换逻辑
def my_node(state: State) -> dict:
    return {"x": state["x"] + 1}

# 将节点函数添加到图中
builder.add_node("my_node", my_node)
```

**关系说明**：
- StateGraph 是容器，节点函数是内容
- StateGraph 定义图的拓扑结构，节点函数定义每个节点的行为
- 没有节点函数，StateGraph 只是一个空壳

[来源: reference/source_节点函数约定_01.md]

### 2. 与状态管理的关系

**协作关系**：节点函数通过 Reducer 函数更新状态

```python
from typing import Annotated
from operator import add

class State(TypedDict):
    messages: Annotated[list[str], add]  # 使用 add reducer
    count: int

def my_node(state: State) -> dict:
    # 返回部分状态更新
    return {
        "messages": ["new message"],  # 会被 add reducer 追加到列表
        "count": state["count"] + 1   # 直接覆盖
    }
```

**关系说明**：
- 节点函数返回部分状态更新
- Reducer 函数决定如何合并更新
- 不同字段可以使用不同的 Reducer 函数

[来源: reference/source_节点函数约定_01.md]

### 3. 与边和条件路由的关系

**协作关系**：节点函数通过边连接，形成工作流

```python
# 普通边：固定连接
builder.add_edge("node1", "node2")

# 条件边：动态路由
def decide_next_step(state: State) -> str:
    """路由函数：返回下一个节点名称"""
    if state["error"]:
        return "error_handler"
    return "next_step"

builder.add_conditional_edges(
    "node1",
    decide_next_step,
    {
        "error_handler": "error_node",
        "next_step": "node2"
    }
)
```

**关系说明**：
- 节点函数定义"做什么"
- 边定义"怎么连接"
- 条件路由定义"如何选择下一步"

[来源: reference/context7_langgraph_01.md]

### 4. 与持久化和检查点的关系

**协作关系**：节点函数的执行状态可以被持久化

```python
from langgraph.checkpoint.memory import InMemorySaver

# 配置 checkpointer
checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 执行图（自动保存检查点）
result = graph.invoke(
    {"x": 1},
    config={"configurable": {"thread_id": "thread-1"}}
)

# 从检查点恢复执行
result = graph.invoke(
    None,  # 不需要输入，从检查点恢复
    config={"configurable": {"thread_id": "thread-1"}}
)
```

**关系说明**：
- 节点函数执行后，状态会被保存到检查点
- 如果节点函数失败，可以从上一个检查点恢复
- 支持断点续传和人机协作

[来源: reference/context7_langgraph_01.md]

### 5. 与异步执行的关系

**协作关系**：节点函数可以是同步或异步的

```python
# 同步节点函数
def sync_node(state: State) -> dict:
    result = sync_operation()
    return {"result": result}

# 异步节点函数
async def async_node(state: State) -> dict:
    result = await async_operation()
    return {"result": result}

# LangGraph 自动检测并处理
builder.add_node("sync_node", sync_node)
builder.add_node("async_node", async_node)
```

**关系说明**：
- LangGraph 自动检测 `async` 函数
- 同步函数会被包装成异步执行（使用 `run_in_executor`）
- 异步节点可以并行执行（如果没有依赖关系）

[来源: reference/source_节点函数约定_01.md]

## 快速参考

### 节点函数签名模板

```python
# 基础签名
def node(state: State) -> dict:
    return {"key": "value"}

# 带 Config
def node(state: State, config: RunnableConfig) -> dict:
    return {"key": "value"}

# 带 Runtime
def node(state: State, runtime: Runtime[Context]) -> dict:
    return {"key": "value"}

# 异步节点
async def node(state: State) -> dict:
    result = await async_operation()
    return {"result": result}

# 返回 Command
def node(state: State) -> Command[Literal["next", "end"]]:
    if condition:
        return Command(goto="end")
    return Command(goto="next", update={"key": "value"})
```

### 返回值类型

| 返回值类型 | 说明 | 示例 |
|-----------|------|------|
| `dict` | 部分状态更新 | `{"x": 1}` |
| `Command` | 控制流程跳转 | `Command(goto="next")` |
| `None` | 不更新状态 | `None` |
| `list[Command \| dict]` | 多个更新 | `[{"x": 1}, Command(goto="next")]` |

## 总结

**节点函数约定是 LangGraph 的核心抽象**，它定义了：
1. **输入约定**：接收 State、Config、Runtime 参数
2. **输出约定**：返回 dict、Command、None 或 list
3. **执行约定**：支持同步/异步、错误处理、重试
4. **组合约定**：通过边连接，形成工作流

理解节点函数约定是掌握 LangGraph 的第一步，也是构建健壮状态化工作流的基础。
