> LangGraph 工作流编排的核心能力：让图的结构在运行时根据数据和 LLM 决策动态变化，而非编译时完全固定

## 🎯 为什么要学这个？

### 现实问题

想象你构建了一个 AI 工作流，所有路径在编译时就写死了：

```python
# ❌ 静态图：路径固定，无法适应变化
builder.add_edge("input", "process")
builder.add_edge("process", "output")
# 如果需要根据输入内容走不同路径呢？
# 如果并行任务数量在运行前未知呢？
# 如果需要 LLM 自己决定下一步呢？
```

在真实的 AI 应用中，工作流必须是动态的：
- 用户问的是技术问题还是闲聊？需要路由到不同处理节点
- 一篇文档要拆成多少个 chunk 并行处理？编译时不知道
- Agent 决定调用哪个工具？取决于 LLM 的实时判断
- 多个子 Agent 之间如何切换？需要运行时导航

### LangGraph 的解决方案

```python
# ✅ 动态图：运行时决定路径
from langgraph.types import Send, Command

# 方式1：条件边 - 根据状态选择路径
def route_by_type(state):
    if state["query_type"] == "technical":
        return "tech_handler"
    return "chat_handler"

builder.add_conditional_edges("classifier", route_by_type)

# 方式2：Send - 运行时创建并行任务
def fan_out(state):
    return [Send("process_chunk", {"chunk": c}) for c in state["chunks"]]

# 方式3：Command - 节点内部同时更新状态 + 控制路由
def smart_node(state) -> Command[Literal["next_a", "next_b"]]:
    return Command(update={"score": 0.9}, goto="next_a")
```

**这就是动态图构建的价值：让工作流像水一样，根据实际情况自动流向正确的方向。**

## 🔑 核心概念速览

### 1. 条件边与路由函数 - 运行时路径选择

**一句话**: 通过 `add_conditional_edges()` + 路由函数，根据当前状态在运行时决定下一个节点

```python
from langgraph.graph import StateGraph, START, END

def route_by_intent(state):
    """根据用户意图路由到不同处理节点"""
    if state["intent"] == "question":
        return "qa_handler"
    elif state["intent"] == "summary":
        return "summary_handler"
    return "default_handler"

builder = StateGraph(State)
builder.add_conditional_edges("classifier", route_by_intent)
```

**关键特性**:
- 路由函数接收当前状态，返回目标节点名（字符串）
- 支持返回列表实现多路并行：`return ["node_a", "node_b"]`
- 可选 `path_map` 参数做路径名映射
- 编译时定义路由逻辑，运行时根据状态动态选择

**类比**: 就像前端路由器（React Router），URL 不同走不同组件

---

### 2. Send API 与 Map-Reduce - 运行时动态并行

**一句话**: 用 `Send(node, arg)` 在运行时动态创建任意数量的并行任务，实现 map-reduce 模式

```python
from langgraph.types import Send

def fan_out_to_workers(state):
    """运行时决定并行任务数量和内容"""
    return [
        Send("process_item", {"item": item})
        for item in state["items"]  # 数量在运行时才知道
    ]

builder.add_conditional_edges("splitter", fan_out_to_workers)
```

**关键特性**:
- `Send(node, arg)` 创建一个指向目标节点的动态边
- 每个 Send 可以携带不同的状态参数
- 并行任务数量完全由运行时数据决定
- 配合 `Annotated[list, operator.add]` reducer 合并结果

**类比**: 就像 JavaScript 的 `Promise.all(items.map(fn))`，并行处理后汇总

---

### 3. Command API 与动态导航 - 状态更新 + 路由控制

**一句话**: 节点返回 `Command(update=..., goto=...)` 在一次操作中同时更新状态和决定下一步

```python
from langgraph.types import Command
from typing import Literal

def decision_node(state) -> Command[Literal["approve", "reject"]]:
    """同时更新状态 + 控制路由"""
    score = evaluate(state["content"])
    if score > 0.8:
        return Command(update={"score": score}, goto="approve")
    return Command(update={"score": score}, goto="reject")
```

**关键特性**:
- `update` 更新状态，`goto` 指定下一个节点
- 支持 `graph=Command.PARENT` 导航到父图（多代理切换）
- `goto` 可以是字符串、Send 对象或列表
- 类型注解 `Command[Literal["a", "b"]]` 帮助图可视化

**类比**: 就像 Redux 的 dispatch，一个 action 同时改状态和触发副作用

---

### 4. LLM 驱动的智能路由 - AI 决策驱动

**一句话**: 用 LLM 的结构化输出作为路由函数的决策依据，实现 AI 自主选择执行路径

```python
from pydantic import BaseModel

class Route(BaseModel):
    step: Literal["search", "calculate", "respond"]

router = llm.with_structured_output(Route)

def llm_router(state):
    decision = router.invoke([
        SystemMessage(content="根据用户问题选择处理方式"),
        HumanMessage(content=state["query"]),
    ])
    return decision.step

builder.add_conditional_edges("router", llm_router)
```

**关键特性**:
- 结构化输出确保 LLM 返回有效的路由值
- Pydantic BaseModel 定义可选路径的枚举
- 条件边 + LLM 决策 = 智能路由
- 适用于 Agent 工具选择、意图分类等场景

**类比**: 就像智能客服的 IVR 系统，AI 听懂你说的话后自动转接对应部门

---

### 5. 图构建器模式与编译 - StateGraph Builder Pattern

**一句话**: `StateGraph` 采用 builder 模式逐步添加节点和边，`compile()` 冻结结构生成可执行图

```python
from langgraph.graph import StateGraph, START, END

# Builder 阶段：灵活组装
builder = StateGraph(State)
builder.add_node("a", node_a)
builder.add_node("b", node_b)
builder.add_edge(START, "a")
builder.add_conditional_edges("a", router)
builder.add_edge("b", END)

# 编译阶段：冻结结构
graph = builder.compile(checkpointer=checkpointer)
# 编译后不能再修改图结构
```

**关键特性**:
- `add_node()` 支持函数、Runnable、子图等多种节点类型
- `add_edge()` 创建静态边，`add_conditional_edges()` 创建动态边
- `add_sequence()` 快速创建顺序链
- `compile()` 后图结构不可变，但运行时路径是动态的
- 支持 `checkpointer`、`interrupt_before/after` 等编译选项

**类比**: 就像 Docker 的 build → run，构建时定义结构，运行时动态执行

## 🎬 实战场景预览

### 场景 1: 基础条件路由

**问题**: 根据用户输入类型路由到不同处理节点

**解决方案**: `add_conditional_edges` + 路由函数

```python
def classify_and_route(state):
    query = state["query"]
    if "代码" in query or "bug" in query:
        return "code_helper"
    elif "总结" in query or "摘要" in query:
        return "summarizer"
    return "general_chat"

builder.add_conditional_edges("classifier", classify_and_route)
```

**适用场景**: 意图分类、多技能 Agent、请求分发

---

### 场景 2: Send Map-Reduce

**问题**: 需要对未知数量的文档并行处理后汇总

**解决方案**: Send API 动态创建并行任务

```python
def split_documents(state):
    """运行时决定并行任务数"""
    return [
        Send("process_doc", {"doc": doc, "index": i})
        for i, doc in enumerate(state["documents"])
    ]

builder.add_conditional_edges("splitter", split_documents)
```

**适用场景**: 批量文档处理、多源数据聚合、并行 API 调用

---

### 场景 3: Command 多代理切换

**问题**: 多个专业 Agent 之间需要动态切换

**解决方案**: Command + `graph=Command.PARENT` 跨图导航

```python
def research_agent(state) -> Command[Literal["writer_agent"]]:
    result = do_research(state["topic"])
    return Command(
        update={"research": result},
        goto="writer_agent",
        graph=Command.PARENT  # 导航到父图的其他子图
    )
```

**适用场景**: 多代理协作、专家系统、任务委派

---

### 场景 4: LLM 智能路由系统

**问题**: 需要 AI 自主决定工作流走向

**解决方案**: 结构化输出 + 条件边

```python
class TaskRoute(BaseModel):
    action: Literal["search", "calculate", "generate", "done"]
    reason: str

def llm_decide(state):
    route = llm.with_structured_output(TaskRoute).invoke(state["messages"])
    return route.action

builder.add_conditional_edges("planner", llm_decide)
```

**适用场景**: 自主 Agent、ReAct 循环、动态任务规划

### 常见学习误区

❌ **误区 1**: "动态图 = 运行时修改图结构"
- 图结构在 compile() 后就固定了
- "动态"指的是运行时选择哪条路径，而非修改图本身

❌ **误区 2**: "Command 的 goto 会替代静态边"
- 如果节点同时有静态边和 Command goto，两者都会执行
- 使用 Command 时通常不应该给该节点添加静态边

❌ **误区 3**: "Send 和普通并行是一样的"
- 普通并行（fan-out）在编译时确定分支数量
- Send 在运行时动态决定分支数量和内容
- Send 的每个实例可以携带不同的状态

## 🚀 快速开始

### 5 分钟快速体验

```python
from typing import TypedDict, Literal, Annotated
from langgraph.graph import StateGraph, START, END
import operator

# 1. 定义状态
class State(TypedDict):
    query: str
    category: str
    result: str

# 2. 定义节点
def classifier(state: State):
    query = state["query"]
    if "天气" in query:
        return {"category": "weather"}
    elif "新闻" in query:
        return {"category": "news"}
    return {"category": "general"}

def weather_handler(state: State):
    return {"result": f"天气查询结果: {state['query']}"}

def news_handler(state: State):
    return {"result": f"新闻搜索结果: {state['query']}"}

def general_handler(state: State):
    return {"result": f"通用回答: {state['query']}"}

# 3. 路由函数
def route_by_category(state: State) -> str:
    return state["category"] + "_handler"

# 4. 构建图
builder = StateGraph(State)
builder.add_node("classifier", classifier)
builder.add_node("weather_handler", weather_handler)
builder.add_node("news_handler", news_handler)
builder.add_node("general_handler", general_handler)

builder.add_edge(START, "classifier")
builder.add_conditional_edges("classifier", route_by_category)
builder.add_edge("weather_handler", END)
builder.add_edge("news_handler", END)
builder.add_edge("general_handler", END)

# 5. 编译并运行
graph = builder.compile()

# 测试不同输入 → 走不同路径
result1 = graph.invoke({"query": "今天天气怎么样"})
print(result1["result"])  # 天气查询结果: 今天天气怎么样

result2 = graph.invoke({"query": "最新新闻有哪些"})
print(result2["result"])  # 新闻搜索结果: 最新新闻有哪些
```

**运行结果**: 同一个图，不同输入自动走不同路径。这就是动态图构建的核心价值。

## 💡 关键要点总结

1. **动态图构建是 LangGraph 实现智能工作流的核心能力**，让图在运行时根据数据自适应
2. **三种核心机制**：条件边（路由选择）、Send（动态并行）、Command（状态+路由）
3. **图结构编译后不可变**，"动态"指的是运行时路径选择，而非修改图结构
4. **条件边最常用**，适合大多数路由场景
5. **Send 解决运行时并行**，任务数量在编译时未知的场景
6. **Command 最灵活**，一次操作同时更新状态和控制路由，支持跨图导航
