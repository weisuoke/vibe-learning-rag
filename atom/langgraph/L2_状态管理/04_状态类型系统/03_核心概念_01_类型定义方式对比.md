# 核心概念：类型定义方式对比

> TypedDict vs Pydantic vs dataclass - LangGraph 状态类型系统的三种定义方式

---

## 概述

LangGraph 支持三种状态类型定义方式：**TypedDict**、**Pydantic BaseModel** 和 **dataclass**。每种方式都有其独特的优势和适用场景。

**核心问题**：
- 为什么 LangGraph 支持多种类型定义方式？
- 如何选择最适合的类型定义方式？
- 三者在性能、功能、易用性上有何差异？

**本文目标**：
- 深入对比三种类型定义方式
- 理解每种方式的优缺点
- 掌握选择建议和最佳实践
- 通过代码示例理解实际应用

---

## 1. TypedDict：轻量级类型定义

### 1.1 基本概念

TypedDict 是 Python 的内置类型，用于定义字典的类型结构。它是 LangGraph **官方推荐的主要方式**。

[来源: reference/search_状态类型系统_01.md | LangChain 官方文档]

**核心特性**：
- 轻量级：没有运行时验证开销
- 快速：直接使用 Python 字典
- 灵活：支持 `Annotated` 绑定 Reducer
- 兼容：与 Python 类型系统无缝集成

### 1.2 基础用法

```python
from typing_extensions import TypedDict, Annotated, NotRequired
from langgraph.graph import StateGraph

# 基础 TypedDict 定义
class State(TypedDict):
    messages: list[str]
    user_id: str
    count: int

# 使用 total=False 使所有字段可选
class OptionalState(TypedDict, total=False):
    messages: list[str]
    user_id: str

# 混合使用 Required 和 NotRequired
class MixedState(TypedDict):
    messages: list[str]  # 必需
    user_id: NotRequired[str]  # 可选
    count: NotRequired[int]  # 可选

graph = StateGraph(State)
```

[来源: reference/context7_typing_extensions_01.md | typing_extensions 官方文档]

### 1.3 Annotated 与 Reducer 绑定

TypedDict 的强大之处在于可以通过 `Annotated` 绑定 Reducer 函数：

```python
from typing_extensions import TypedDict, Annotated

def add_messages(left: list, right: list) -> list:
    """Reducer 函数：合并消息列表"""
    return left + right

class State(TypedDict):
    # 绑定 Reducer 函数到字段
    messages: Annotated[list[str], add_messages]
    user_id: str

# LangGraph 会自动使用 add_messages 合并 messages 字段
```

[来源: reference/source_状态类型系统_01.md | sourcecode/langgraph/.../state.py]

**Reducer 函数要求**：
- 必须是二元操作符（接受两个参数）
- 第一个参数是当前值，第二个参数是新值
- 返回合并后的值

### 1.4 TypedDict 的优势

**性能优势**：
- 零运行时开销：TypedDict 只是类型提示，不影响运行时性能
- 直接使用字典：没有额外的对象包装
- 快速访问：字典访问速度极快

**灵活性**：
- 支持 `total=True/False` 控制字段必需性
- 支持 `Required`/`NotRequired` 精细控制
- 支持 `ReadOnly` 标记只读字段（Python 3.13+）

[来源: reference/context7_typing_extensions_01.md | typing_extensions 官方文档]

### 1.5 TypedDict 的局限

**缺少运行时验证**：
- 不会检查字段类型是否正确
- 不会检查必需字段是否存在
- 需要手动验证数据

**缺少默认值**：
- TypedDict 本身不支持默认值
- 需要在代码中手动处理默认值

**缺少类型转换**：
- 不会自动转换类型（如字符串转整数）
- 需要手动处理类型转换

## 2. Pydantic BaseModel：强类型验证

### 2.1 基本概念

Pydantic 是一个数据验证库，提供运行时类型验证和自动类型转换。

[来源: reference/context7_pydantic_01.md | Pydantic 官方文档]

**核心特性**：
- 运行时验证：自动检查字段类型
- 类型转换：自动转换兼容类型
- 丰富约束：支持字段约束（如 `min_length`、`gt`）
- 清晰错误：提供详细的验证错误消息

### 2.2 基础用法

```python
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph

# 基础 Pydantic 模型
class State(BaseModel):
    messages: list[str]
    user_id: str
    count: int = 0  # 支持默认值

# 使用 Field 添加验证和约束
class ValidatedState(BaseModel):
    messages: list[str] = Field(default_factory=list)
    user_id: str = Field(..., min_length=1, max_length=100)
    count: int = Field(default=0, ge=0)  # >= 0
    age: int | None = Field(default=None, ge=0, le=150)

graph = StateGraph(ValidatedState)
```

[来源: reference/context7_pydantic_01.md | Pydantic 官方文档]

### 2.3 自动类型转换

Pydantic 的强大之处在于自动类型转换：

```python
from pydantic import BaseModel
from datetime import datetime

class State(BaseModel):
    id: int
    name: str
    created_at: datetime

# 自动类型转换
data = {
    'id': '123',  # 字符串自动转换为整数
    'name': 'Alice',
    'created_at': '2025-01-01 12:00:00'  # 字符串自动转换为 datetime
}

state = State(**data)
print(state.id)  # 123 (int)
print(state.created_at)  # datetime(2025, 1, 1, 12, 0)
```

[来源: reference/context7_pydantic_01.md | Pydantic 官方文档]

### 2.4 LangGraph 中的特殊处理

LangGraph 对 Pydantic 模型有特殊处理，使用 `model_fields_set` 跟踪显式设置的字段：

```python
from pydantic import BaseModel

class State(BaseModel):
    x: int
    y: str = "default"

# 只有显式设置的字段会被更新
state = State(x=1)
print(state.model_fields_set)  # {'x'}
# 'y' 不在 model_fields_set 中，因为使用了默认值
```

[来源: reference/source_状态类型系统_01.md | sourcecode/langgraph/.../\_fields.py]

**LangGraph 更新逻辑**：
```python
# 从源码中提取的逻辑
def get_update_as_tuples(input: Any, keys: Sequence[str]) -> list[tuple[str, Any]]:
    if isinstance(input, BaseModel):
        keep = input.model_fields_set  # 跟踪显式设置的字段
        defaults = {k: v.default for k, v in type(input).model_fields.items()}

    # 只更新与默认值不同的字段或在 model_fields_set 中的字段
    return [
        (k, value)
        for k in keys
        if (value := getattr(input, k, MISSING)) is not MISSING
        and (
            value is not None
            or defaults.get(k, MISSING) is not None
            or (keep is not None and k in keep)
        )
    ]
```

[来源: reference/source_状态类型系统_01.md | sourcecode/langgraph/.../\_fields.py]

### 2.5 Pydantic 的优势

**强大的验证能力**：
- 自动类型检查
- 丰富的字段约束（`min_length`、`max_length`、`gt`、`ge`、`lt`、`le`）
- 自定义验证器
- 递归验证嵌套模型

**清晰的错误消息**：
```python
from pydantic import BaseModel, Field, ValidationError

class State(BaseModel):
    age: int = Field(..., ge=0, le=150)

try:
    state = State(age=200)
except ValidationError as e:
    print(e)
    # 输出详细的错误信息，包括字段路径和错误原因
```

**IDE 支持**：
- 类型提示
- 自动补全
- 字段文档

### 2.6 Pydantic 的局限

**性能开销**：
- 运行时验证有性能成本
- 对于高并发场景，性能影响明显

[来源: reference/search_状态类型系统_01.md | 社区讨论]

**复杂性**：
- 学习曲线较陡
- 配置选项较多
- 版本迁移（v1 vs v2）

## 3. dataclass：平衡之选

### 3.1 基本概念

dataclass 是 Python 3.7+ 的内置装饰器，用于简化类的定义。

**核心特性**：
- 支持默认值
- 支持字段元数据
- 自动生成 `__init__`、`__repr__` 等方法
- 轻量级，性能接近 TypedDict

### 3.2 基础用法

```python
from dataclasses import dataclass, field
from langgraph.graph import StateGraph

# 基础 dataclass 定义
@dataclass
class State:
    messages: list[str]
    user_id: str
    count: int = 0  # 支持默认值

# 使用 field 添加元数据和默认工厂
@dataclass
class AdvancedState:
    messages: list[str] = field(default_factory=list)
    user_id: str = field(default="")
    count: int = field(default=0, metadata={"description": "计数器"})

graph = StateGraph(State)
```

### 3.3 字段默认值推断

LangGraph 支持从 dataclass 的字段定义中推断默认值：

```python
from dataclasses import dataclass, field

@dataclass
class State:
    x: int
    y: str = "default"
    z: list = field(default_factory=list)

# LangGraph 会自动识别：
# - x 是必需字段（没有默认值）
# - y 是可选字段（有默认值 "default"）
# - z 是可选字段（有默认工厂 list）
```

[来源: reference/source_状态类型系统_01.md | sourcecode/langgraph/.../\_fields.py]

### 3.4 dataclass 的优势

**平衡性能和功能**：
- 性能接近 TypedDict
- 支持默认值和默认工厂
- 支持字段元数据

**简洁性**：
- 语法简洁
- 自动生成常用方法
- 易于理解和维护

### 3.5 dataclass 的局限

**缺少运行时验证**：
- 不会检查字段类型
- 不会自动转换类型

**功能有限**：
- 不支持复杂的字段约束
- 不支持自定义验证器

## 4. 三者对比

### 4.1 功能对比表

| 特性 | TypedDict | Pydantic BaseModel | dataclass |
|------|-----------|-------------------|-----------|
| **运行时验证** | ✗ | ✓ | ✗ |
| **类型转换** | ✗ | ✓ | ✗ |
| **默认值** | ✗ | ✓ | ✓ |
| **字段约束** | ✗ | ✓ | ✗ |
| **性能** | 最快 | 较慢 | 快 |
| **IDE 支持** | ✓ | ✓ | ✓ |
| **序列化** | 手动 | ✓ (model_dump) | 手动 |
| **Reducer 支持** | ✓ | ✓ | ✓ |
| **学习曲线** | 低 | 中 | 低 |
| **官方推荐** | ✓ | 边界验证 | - |

[来源: reference/search_状态类型系统_01.md | 社区讨论]

### 4.2 性能对比

**基准测试结果**（相对性能）：

| 操作 | TypedDict | dataclass | Pydantic |
|------|-----------|-----------|----------|
| 创建对象 | 1.0x | 1.1x | 3.5x |
| 字段访问 | 1.0x | 1.0x | 1.0x |
| 更新字段 | 1.0x | 1.0x | 1.2x |
| 序列化 | 1.0x | 1.0x | 2.0x |

[来源: reference/search_状态类型系统_01.md | 社区讨论]

**结论**：
- TypedDict 性能最优
- dataclass 性能接近 TypedDict
- Pydantic 有明显的性能开销（但提供了强大的验证功能）

### 4.3 使用场景对比

**TypedDict 适用场景**：
- 内部状态管理（轻量、快速）
- 不需要运行时验证的场景
- 性能敏感的应用
- 需要 Reducer 函数的场景

**Pydantic 适用场景**：
- API 边界验证（输入、输出）
- 需要递归验证的复杂数据结构
- 需要严格类型检查的场景
- 生产环境的可靠性要求

**dataclass 适用场景**：
- 需要默认值的简单状态
- 不需要运行时验证的场景
- 追求简洁性和可读性

[来源: reference/search_状态类型系统_01.md | 社区讨论]

## 5. 选择建议

### 5.1 官方推荐

**LangChain 官方文档明确指出**：
- TypedDict 是主要推荐方式
- Pydantic 性能较低，仅在需要递归验证时使用
- dataclass 作为第三种选择

[来源: reference/search_状态类型系统_01.md | LangChain 官方文档]

### 5.2 决策树

```
开始
  |
  ├─ 需要运行时验证？
  |   ├─ 是 → 使用 Pydantic
  |   └─ 否 → 继续
  |
  ├─ 需要默认值？
  |   ├─ 是 → 使用 dataclass
  |   └─ 否 → 使用 TypedDict
  |
  └─ 性能敏感？
      ├─ 是 → 使用 TypedDict
      └─ 否 → 使用 dataclass 或 Pydantic
```

### 5.3 最佳实践

**混合使用策略**：
```python
from typing_extensions import TypedDict
from pydantic import BaseModel, Field

# 外部 API 使用 Pydantic（边界验证）
class APIInput(BaseModel):
    query: str = Field(..., min_length=1, max_length=1000)
    user_id: str = Field(..., pattern=r'^[a-zA-Z0-9_-]+$')

# 内部状态使用 TypedDict（性能优先）
class InternalState(TypedDict):
    messages: list[str]
    context: dict

def process_input(api_input: APIInput) -> InternalState:
    # 验证后转换为内部状态
    return {
        "messages": [api_input.query],
        "context": {"user_id": api_input.user_id}
    }
```

[来源: reference/search_状态类型系统_01.md | 社区最佳实践]

**原则**：
1. **内部状态用 TypedDict**：追求性能
2. **API 边界用 Pydantic**：确保数据可靠性
3. **简单场景用 dataclass**：平衡性能和功能

## 6. 实战代码示例

### 6.1 TypedDict 实战

```python
from typing_extensions import TypedDict, Annotated, NotRequired
from langgraph.graph import StateGraph

def add_messages(left: list, right: list) -> list:
    return left + right

def merge_context(left: dict, right: dict) -> dict:
    return {**left, **right}

class State(TypedDict):
    # 必需字段
    user_id: str

    # 带 Reducer 的字段
    messages: Annotated[list[str], add_messages]
    context: Annotated[dict, merge_context]

    # 可选字段
    error: NotRequired[str]
    metadata: NotRequired[dict]

# 创建图
graph = StateGraph(State)

def node1(state: State) -> State:
    return {
        "messages": ["Hello"],
        "context": {"step": "node1"}
    }

def node2(state: State) -> State:
    return {
        "messages": ["World"],
        "context": {"step": "node2"}
    }

graph.add_node("node1", node1)
graph.add_node("node2", node2)
graph.add_edge("node1", "node2")
graph.set_entry_point("node1")
graph.set_finish_point("node2")

app = graph.compile()

# 运行
result = app.invoke({"user_id": "123"})
print(result["messages"])  # ["Hello", "World"]
print(result["context"])  # {"step": "node2"}
```

### 6.2 Pydantic 实战

```python
from pydantic import BaseModel, Field, validator
from langgraph.graph import StateGraph

class State(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=100)
    messages: list[str] = Field(default_factory=list)
    count: int = Field(default=0, ge=0)

    @validator('user_id')
    def validate_user_id(cls, v):
        if not v.isalnum():
            raise ValueError('user_id must be alphanumeric')
        return v

# 创建图
graph = StateGraph(State)

def node1(state: State) -> State:
    return State(
        user_id=state.user_id,
        messages=state.messages + ["Hello"],
        count=state.count + 1
    )

graph.add_node("node1", node1)
graph.set_entry_point("node1")
graph.set_finish_point("node1")

app = graph.compile()

# 运行（自动验证）
try:
    result = app.invoke({"user_id": "user123", "messages": []})
    print(result)
except Exception as e:
    print(f"Validation error: {e}")
```

### 6.3 dataclass 实战

```python
from dataclasses import dataclass, field
from langgraph.graph import StateGraph

@dataclass
class State:
    user_id: str
    messages: list[str] = field(default_factory=list)
    count: int = 0
    metadata: dict = field(default_factory=dict)

# 创建图
graph = StateGraph(State)

def node1(state: State) -> dict:
    return {
        "messages": state.messages + ["Hello"],
        "count": state.count + 1,
        "metadata": {"step": "node1"}
    }

graph.add_node("node1", node1)
graph.set_entry_point("node1")
graph.set_finish_point("node1")

app = graph.compile()

# 运行
initial_state = State(user_id="123")
result = app.invoke(initial_state)
print(result)
```

### 6.4 混合使用实战

```python
from typing_extensions import TypedDict
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph

# API 输入验证（Pydantic）
class APIInput(BaseModel):
    query: str = Field(..., min_length=1, max_length=1000)
    user_id: str = Field(..., pattern=r'^[a-zA-Z0-9_-]+$')
    max_results: int = Field(default=10, ge=1, le=100)

# 内部状态（TypedDict）
class InternalState(TypedDict):
    messages: list[str]
    context: dict
    results: list[dict]

# 转换函数
def api_to_internal(api_input: APIInput) -> InternalState:
    return {
        "messages": [api_input.query],
        "context": {
            "user_id": api_input.user_id,
            "max_results": api_input.max_results
        },
        "results": []
    }

# 创建图
graph = StateGraph(InternalState)

def process_node(state: InternalState) -> InternalState:
    # 处理逻辑
    return {
        "results": [{"id": 1, "score": 0.9}]
    }

graph.add_node("process", process_node)
graph.set_entry_point("process")
graph.set_finish_point("process")

app = graph.compile()

# 使用
try:
    # 1. 验证输入（Pydantic）
    api_input = APIInput(query="test", user_id="user123")

    # 2. 转换为内部状态（TypedDict）
    internal_state = api_to_internal(api_input)

    # 3. 运行图（高性能）
    result = app.invoke(internal_state)

    print(result)
except Exception as e:
    print(f"Error: {e}")
```

## 7. 常见问题

### Q1: 为什么 TypedDict 是推荐方式？

**A**:
- 性能：TypedDict 没有运行时验证开销
- 简单：语法简洁，易于理解
- 灵活：支持 Annotated reducer
- 兼容：与 Python 类型系统无缝集成

[来源: reference/search_状态类型系统_01.md | 社区讨论]

### Q2: 什么时候必须使用 Pydantic？

**A**:
- 需要递归验证复杂数据结构
- API 边界需要严格验证
- 需要自定义验证器
- 需要数据转换和清洗

### Q3: 如何处理 Python < 3.12 的兼容性问题？

**A**:
- 使用 `typing_extensions.TypedDict` 而非 `typing.TypedDict`
- 确保 `typing_extensions` 版本 >= 4.0.0
- 避免混合使用 `typing.TypedDict` 和 Pydantic

[来源: reference/search_状态类型系统_01.md | GitHub Issue #2198]

### Q4: dataclass 和 TypedDict 如何选择？

**A**:
- 需要默认值 → dataclass
- 追求极致性能 → TypedDict
- 需要 Reducer → TypedDict（更灵活）
- 简单场景 → dataclass（更简洁）

## 8. 总结

### 8.1 核心要点

1. **TypedDict**：官方推荐，性能最优，适合内部状态
2. **Pydantic**：强大验证，适合 API 边界
3. **dataclass**：平衡之选，适合简单场景

### 8.2 选择原则

- **性能优先** → TypedDict
- **验证优先** → Pydantic
- **简洁优先** → dataclass
- **混合使用** → 内部 TypedDict + 边界 Pydantic

### 8.3 最佳实践

1. 内部状态使用 TypedDict
2. API 边界使用 Pydantic
3. 简单场景使用 dataclass
4. 根据实际需求灵活选择
