# 第一性原理

> **目标**：从底层逻辑理解标量过滤与JSON索引的本质

---

## 核心问题

### 为什么需要标量过滤？

**第一性思考**：向量检索解决的是"语义相似"问题，但实际业务需要"语义相似 + 业务规则"。

**本质矛盾**：
```
向量检索：找到语义最相似的结果
业务需求：在满足业务规则的结果中找到语义最相似的

例如：
- 推荐商品：语义相似 + 价格范围 + 有库存
- 文档检索：语义相关 + 用户权限 + 部门范围
- 智能客服：语义匹配 + 用户等级 + 服务时间
```

**结论**：标量过滤是将业务规则引入向量检索的必然需求。

---

### 为什么需要JSON索引？

**第一性思考**：现代应用的元数据越来越复杂，扁平的标量字段无法满足需求。

**演进路径**：
```
阶段1：简单标量字段
price, stock, category

阶段2：多个标量字段
price, stock, category, brand, color, size

阶段3：嵌套结构（JSON）
metadata: {
  product: { brand, model, specs },
  inventory: { stock, warehouse },
  pricing: { base, discount, final }
}
```

**本质需求**：
- 灵活性：无需预定义schema，动态添加字段
- 表达力：支持复杂的嵌套结构和数组
- 查询能力：高效查询嵌套字段

**结论**：JSON索引是应对复杂元数据的必然选择。

## 底层原理

### 1. 标量过滤的执行时机

**核心问题**：过滤应该在向量检索前还是后？

#### Pre-filtering（前置过滤）

**原理**：
```
1. 先应用标量过滤，得到候选集
2. 在候选集中进行向量检索
```

**适用场景**：
- 过滤条件选择性高（过滤掉大部分数据）
- 例如：`stock > 0 AND price < 100`（只保留10%数据）

**优势**：
- 减少向量检索的搜索空间
- 降低计算成本

**劣势**：
- 如果候选集太小，可能召回不足

#### Post-filtering（后置过滤）

**原理**：
```
1. 先进行向量检索，得到Top-K结果
2. 对结果应用标量过滤
```

**适用场景**：
- 过滤条件选择性低（保留大部分数据）
- 例如：`created_at > '2020-01-01'`（保留90%数据）

**优势**：
- 保证召回率
- 向量检索不受影响

**劣势**：
- 可能需要检索更多结果才能满足limit

#### Hybrid（混合策略）

**原理**：
```
Milvus自动根据过滤选择性选择策略
- 选择性高 → Pre-filtering
- 选择性低 → Post-filtering
```

### 2. JSON Path Index的实现原理

**核心问题**：如何高效查询嵌套JSON？

#### 方案1：全表扫描（Naive）

```python
# 每次查询都解析所有JSON
for doc in documents:
    json_obj = parse_json(doc.metadata)
    if json_obj['category'] == 'electronics':
        results.append(doc)
```

**问题**：
- 每次查询都要解析JSON
- 无法利用索引
- 性能随数据量线性下降

#### 方案2：JsonFlatIndex（扁平索引）

**原理**：
```cpp
// 构建时提取JSON路径值
for each document:
    value = json.at_pointer("metadata['category']")
    index[value].append(document_id)

// 查询时直接查找
results = index['electronics']
```

**特点**：
- 简单直接
- 适用于简单查询
- 内存占用小

[来源: reference/source_json_index.md | JsonFlatIndex.cpp]

#### 方案3：JsonInvertedIndex（倒排索引）

**原理**：
```cpp
// 构建倒排索引
for each document:
    for each json_path in paths:
        value = extract_value(document, json_path)
        inverted_index[json_path][value].append(document_id)

// 查询时使用倒排索引
results = inverted_index["metadata['category']"]['electronics']
```

**特点**：
- 支持复杂查询
- 100倍性能提升
- 适用于生产环境

**关键技术**：
- **simdjson**：高性能JSON解析（SIMD优化）
- **Tantivy**：Rust实现的倒排索引引擎
- **位图操作**：高效的集合运算

[来源: reference/source_json_index.md | JsonInvertedIndex.cpp]

### 3. 索引优化的本质

**核心问题**：为什么索引能提升30-100倍性能？

#### 数据结构的选择

**无索引（线性扫描）**：
```
时间复杂度：O(n)
空间复杂度：O(1)

查询100万条记录：100万次比较
```

**INVERTED索引（倒排索引）**：
```
时间复杂度：O(log n) + O(k)  // k是结果数量
空间复杂度：O(n)

查询100万条记录：log(100万) ≈ 20次比较 + 结果提取
```

**性能提升**：
```
1,000,000 / 20 = 50,000倍（理论）
实际：30-100倍（考虑索引构建、内存访问等开销）
```

#### 索引类型对比

| 索引类型 | 数据结构 | 适用场景 | 时间复杂度 |
|---------|---------|---------|-----------|
| **INVERTED** | 倒排索引 | 精确匹配、范围查询 | O(log n) |
| **BITMAP** | 位图 | 低基数字段（如性别、状态） | O(1) |
| **STL_SORT** | 排序数组 | 范围查询 | O(log n) |
| **TRIE** | 前缀树 | 前缀匹配 | O(m) m=字符串长度 |
| **NGRAM** | N-gram索引 | 模糊匹配 | O(k) k=匹配数量 |

## 设计权衡

### 1. 索引 vs 无索引

**权衡点**：
```
索引优势：
+ 查询快（30-100倍）
+ 支持复杂查询

索引代价：
- 存储空间（额外20-50%）
- 构建时间（初始化慢）
- 更新成本（插入/删除变慢）
```

**决策原则**：
- 高频查询字段：必须建索引
- 低频查询字段：可以不建索引
- 写多读少：谨慎建索引

### 2. Pre-filtering vs Post-filtering

**权衡点**：
```
Pre-filtering：
+ 减少搜索空间
+ 降低计算成本
- 可能召回不足

Post-filtering：
+ 保证召回率
+ 向量检索不受影响
- 可能需要检索更多结果
```

**决策原则**：
- 选择性 > 50%：Pre-filtering
- 选择性 < 50%：Post-filtering
- 不确定：让Milvus自动选择

[来源: reference/context7_pymilvus_scalar_filtering.md | PyMilvus官方文档]

### 3. JsonFlatIndex vs JsonInvertedIndex

**权衡点**：
```
JsonFlatIndex：
+ 简单直接
+ 内存占用小
- 性能一般

JsonInvertedIndex：
+ 100倍性能提升
+ 支持复杂查询
- 内存占用大（额外30-50%）
- 构建时间长
```

**决策原则**：
- 简单查询、小数据量：JsonFlatIndex
- 复杂查询、大数据量：JsonInvertedIndex
- 生产环境：JsonInvertedIndex

## 性能模型

### 1. 标量过滤性能公式

```
查询延迟 = 过滤延迟 + 向量检索延迟

无索引：
过滤延迟 = O(n) × 比较成本

有索引：
过滤延迟 = O(log n) × 索引查找成本 + O(k) × 结果提取成本

其中：
n = 总数据量
k = 过滤后结果数量
```

**实际数据**（1百万记录）：
```
无索引：1000ms
INVERTED索引：30ms
性能提升：33倍
```

[来源: reference/search_web_results.md | Milvus 2.6官方博客]

### 2. JSON Path Index性能公式

```
JSON查询延迟 = JSON解析 + 路径提取 + 值比较

无索引：
延迟 = n × (解析成本 + 路径提取成本 + 比较成本)

JSON Path Index：
延迟 = 索引查找成本 + k × 结果提取成本

其中：
n = 总数据量
k = 匹配结果数量
```

**实际数据**（嵌套JSON查询）：
```
无索引：10000ms
JSON Path Index：100ms
性能提升：100倍
```

## 实现细节

### 1. 布尔表达式执行

**表达式树**：
```
filter = "age > 25 AND (city == '北京' OR city == '上海')"

解析为表达式树：
        AND
       /   \
    age>25  OR
           /  \
    city==北京 city==上海
```

**执行流程**：
```cpp
// 递归执行表达式树
VectorPtr eval(ExprNode* node) {
    if (node->type == COMPARE) {
        return eval_compare(node);
    } else if (node->type == AND) {
        auto left = eval(node->left);
        auto right = eval(node->right);
        return bitwise_and(left, right);
    } else if (node->type == OR) {
        auto left = eval(node->left);
        auto right = eval(node->right);
        return bitwise_or(left, right);
    }
}
```

**位图优化**：
```cpp
// 使用位图表示结果
TargetBitmap result(n, false);  // n个文档，初始为false

// AND操作：位图与运算
result = left_bitmap & right_bitmap;

// OR操作：位图或运算
result = left_bitmap | right_bitmap;

// NOT操作：位图取反
result = ~bitmap;
```

[来源: reference/source_expression.md | LogicalBinaryExpr.cpp]

### 2. JSON Path解析

**路径格式**：
```python
# 单层访问
"metadata['category']"

# 嵌套访问
"metadata['product']['brand']"

# 数组访问
"metadata['tags'][0]"

# 混合访问
"metadata['reviews'][0]['rating']"
```

**解析流程**：
```cpp
// 使用simdjson解析
auto doc = json.doc();
auto result = doc.at_pointer("metadata['category']");

// 处理错误
if (result.error() != simdjson::SUCCESS) {
    // 路径不存在或类型不匹配
    handle_error();
}
```

**类型转换**：
```cpp
// 支持类型转换函数
params = {
    "json_path": "metadata['price']",
    "json_cast_type": "double",
    "json_cast_function": "STRING_TO_DOUBLE"
}

// 将字符串"99.99"转换为double 99.99
```

## 类比理解

### 前端开发类比

**标量过滤 = SQL WHERE子句**：
```javascript
// 前端过滤
const filtered = products.filter(p =>
    p.price >= 100 && p.price <= 500 && p.stock > 0
);

// Milvus标量过滤
filter = "price >= 100 AND price <= 500 AND stock > 0"
```

**JSON Path = JavaScript对象访问**：
```javascript
// JavaScript
const brand = product.metadata.product.brand;

// Milvus JSON Path
json_path = "metadata['product']['brand']"
```

**索引 = 数据库索引**：
```javascript
// 无索引：全表扫描
products.find(p => p.id === 123);  // O(n)

// 有索引：快速查找
productsById[123];  // O(1)
```

### 生活类比

**标量过滤 = 筛选条件**：
```
找餐厅：
- 向量检索：找口味相似的餐厅
- 标量过滤：价格100-500元、距离<3km、营业中
```

**JSON Path = 商品详情页**：
```
商品信息：
- 基本信息：名称、价格、库存
- 详细规格（JSON）：
  - 尺寸：长×宽×高
  - 重量：净重、毛重
  - 材质：外壳、内胆
```

**索引 = 图书馆目录**：
```
无索引：逐本翻书找内容
有索引：查目录直接定位
```

## 常见误解

### ❌ 误解1："标量过滤会显著降低性能"

**正确理解**：
- 有索引：性能提升30倍
- 无索引：确实会慢
- 关键：为常用字段创建索引

### ❌ 误解2："JSON查询一定很慢"

**正确理解**：
- JSON Path Index：100倍性能提升
- 全表扫描：确实很慢
- 关键：使用JSON Path Index

### ❌ 误解3："索引越多越好"

**正确理解**：
- 索引有成本：存储空间、构建时间、更新成本
- 只为高频查询字段建索引
- 权衡查询性能和写入性能
