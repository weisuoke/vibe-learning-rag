# 核心概念 2：_call 主循环与 _iter_next_step 单步执行

> AgentExecutor 的心脏是两个方法：_call 控制"转多少圈"，_iter_next_step 控制"每圈做什么"

---

## 整体架构

AgentExecutor 的执行逻辑分为两层：

```
┌──────────────────────────────────────────────────┐
│  _call() —— 外层 while 循环                      │
│  职责：循环控制、计时、early stopping              │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │  _iter_next_step() —— 单步执行             │  │
│  │  职责：调用 LLM、解析输出、执行工具          │  │
│  │                                            │  │
│  │  ┌──────────────────────────────────────┐  │  │
│  │  │  _perform_agent_action()             │  │  │
│  │  │  职责：查找工具、执行工具、处理异常    │  │  │
│  │  └──────────────────────────────────────┘  │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

**前端类比：** 就像 React 的渲染机制 —— `_call` 是 reconciler（决定要不要重新渲染），`_iter_next_step` 是单次 render（执行一次组件函数），`_perform_agent_action` 是处理副作用的 useEffect。

## _call 主循环：源码逐行解读

`_call` 是 AgentExecutor 的入口方法，定义在 `agent.py:1570`。

```python
# 来源: langchain_classic/agents/agent.py 第 1570 行
def _call(
    self,
    inputs: dict[str, str],
    run_manager: CallbackManagerForChainRun | None = None,
) -> dict[str, Any]:
    """运行 Agent 的文本到完成的映射"""

    # ① 构建工具查找表：{工具名: 工具对象}
    name_to_tool_map = {tool.name: tool for tool in self.tools}

    # ② 为每个工具分配颜色（终端输出用）
    color_mapping = get_color_mapping(
        [tool.name for tool in self.tools],
        excluded_colors=["green", "red"]
    )

    # ③ 初始化状态
    intermediate_steps: list[tuple[AgentAction, str]] = []
    iterations = 0
    time_elapsed = 0.0
    start_time = time.time()

    # ④ 主循环：持续执行直到应该停止
    while self._should_continue(iterations, time_elapsed):

        # ⑤ 执行一步（核心！）
        next_step_output = self._take_next_step(
            name_to_tool_map,
            color_mapping,
            inputs,
            intermediate_steps,
            run_manager,
        )

        # ⑥ 如果 LLM 返回最终答案，结束循环
        if isinstance(next_step_output, AgentFinish):
            return self._return(
                next_step_output, intermediate_steps, run_manager
            )

        # ⑦ 否则，记录这一步的结果
        intermediate_steps.extend(next_step_output)

        # ⑧ 检查 return_direct：工具是否要求直接返回
        if len(next_step_output) == 1:
            tool_return = self._get_tool_return(next_step_output[0])
            if tool_return is not None:
                return self._return(
                    tool_return, intermediate_steps, run_manager
                )

        # ⑨ 更新计数器
        iterations += 1
        time_elapsed = time.time() - start_time

    # ⑩ 循环结束但没有 AgentFinish → 执行 early stopping
    output = self._action_agent.return_stopped_response(
        self.early_stopping_method, intermediate_steps, **inputs
    )
    return self._return(output, intermediate_steps, run_manager)
```

### 逐步拆解

#### ① name_to_tool_map —— 工具查找表

```python
name_to_tool_map = {tool.name: tool for tool in self.tools}
# 例如: {"get_weather": <GetWeatherTool>, "search": <SearchTool>}
```

这是一个字典推导式，把工具列表转成 `{名称: 对象}` 的映射。后续 LLM 返回 `AgentAction(tool="get_weather", ...)` 时，通过这个字典 O(1) 查找对应的工具对象。

**为什么不直接遍历列表？** 性能。如果有 20 个工具，每次查找都遍历列表是 O(n)，字典查找是 O(1)。在多轮循环中差距会放大。

#### ② color_mapping —— 终端颜色

```python
color_mapping = get_color_mapping(
    [tool.name for tool in self.tools],
    excluded_colors=["green", "red"]
)
# 例如: {"get_weather": "blue", "search": "yellow"}
```

纯粹是为了终端输出好看。绿色留给最终答案，红色留给错误，其他颜色分配给各个工具。在 verbose=True 时，不同工具的输出用不同颜色区分。

#### ③ 初始化状态

```python
intermediate_steps: list[tuple[AgentAction, str]] = []  # 历史步骤
iterations = 0          # 已执行轮数
time_elapsed = 0.0      # 已用时间（秒）
start_time = time.time()  # 开始时间戳
```

`intermediate_steps` 是整个循环的"记忆"。每轮执行完，新的 `(action, observation)` 对被追加进去，下一轮 LLM 能看到之前所有的操作和结果。

#### ④ _should_continue —— 循环守卫

```python
# 来源: agent.py 第 1235 行
def _should_continue(self, iterations: int, time_elapsed: float) -> bool:
    if self.max_iterations is not None and iterations >= self.max_iterations:
        return False
    if self.max_execution_time is not None and time_elapsed >= self.max_execution_time:
        return False
    return True
```

两个刹车条件，任一触发就停止：
- `max_iterations`：默认 15，防止无限循环
- `max_execution_time`：默认 None（不限时），生产环境建议设置

#### ⑥⑦⑧ 三种退出路径

```
_take_next_step() 的返回值
  │
  ├── AgentFinish → 直接返回最终答案（最常见的退出方式）
  │
  ├── [AgentStep] 且 return_direct=True → 工具结果直接作为最终答案
  │
  └── [AgentStep] 且 return_direct=False → 继续循环
```

#### ⑩ Early Stopping —— 超时兜底

```python
output = self._action_agent.return_stopped_response(
    self.early_stopping_method,  # "force" 或 "generate"
    intermediate_steps,
    **inputs
)
```

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| `"force"` | 返回固定字符串 "Agent stopped due to iteration limit or time limit." | 开发调试 |
| `"generate"` | 让 LLM 基于已有步骤生成一个总结性答案 | 生产环境 |

## _iter_next_step：单步执行的完整逻辑

`_iter_next_step` 是每一轮循环的核心，定义在 `agent.py:1301`。它是一个生成器函数，yield 出 AgentFinish 或 AgentStep。

```python
# 来源: agent.py 第 1301 行（简化版，保留核心逻辑）
def _iter_next_step(
    self,
    name_to_tool_map: dict[str, BaseTool],
    color_mapping: dict[str, str],
    inputs: dict[str, str],
    intermediate_steps: list[tuple[AgentAction, str]],
    run_manager: CallbackManagerForChainRun | None = None,
) -> Iterator[AgentFinish | AgentStep]:

    # ① 裁剪 intermediate_steps（防止上下文过长）
    intermediate_steps = self._prepare_intermediate_steps(
        intermediate_steps
    )

    # ② 调用 LLM 获取下一步决策
    try:
        output = self.agent.plan(
            intermediate_steps,
            callbacks=run_manager.get_child() if run_manager else None,
            **inputs,
        )
    except OutputParserException as e:
        # ③ 解析失败的处理（详见下方）
        # ...处理逻辑...
        yield AgentStep(action=error_action, observation=observation)
        return

    # ④ 如果 LLM 返回 AgentFinish，循环结束
    if isinstance(output, AgentFinish):
        yield output
        return

    # ⑤ 如果返回 AgentAction，逐个执行工具
    actions: list[AgentAction]
    if isinstance(output, AgentAction):
        actions = [output]
    else:
        actions = output  # MultiActionAgent 返回列表

    for agent_action in actions:
        yield self._perform_agent_action(
            name_to_tool_map, color_mapping,
            agent_action, run_manager
        )
```

### 关键步骤详解

#### ① _prepare_intermediate_steps —— 裁剪历史

```python
def _prepare_intermediate_steps(
    self, intermediate_steps: list[tuple[AgentAction, str]]
) -> list[tuple[AgentAction, str]]:
    if (
        isinstance(self.trim_intermediate_steps, int)
        and self.trim_intermediate_steps > 0
    ):
        return intermediate_steps[-self.trim_intermediate_steps:]
    elif callable(self.trim_intermediate_steps):
        return self.trim_intermediate_steps(intermediate_steps)
    else:
        return intermediate_steps
```

**为什么要裁剪？** LLM 的 Context Window 有限。如果 Agent 已经执行了 10 轮，每轮的 action + observation 可能有几百 token，全部塞进去会超出上下文限制。

| trim_intermediate_steps 值 | 行为 |
|---------------------------|------|
| `-1`（默认） | 不裁剪，保留全部历史 |
| `5` | 只保留最近 5 步 |
| `callable` | 自定义裁剪函数 |

#### ② agent.plan() —— 调用 LLM 决策

这是整个循环中最关键的一步。`plan()` 方法：

1. 把用户输入 + intermediate_steps 组装成 Prompt
2. 调用 LLM 获取响应
3. 用 OutputParser 解析 LLM 的文本输出
4. 返回 `AgentAction`（需要工具）或 `AgentFinish`（有答案了）

```python
# agent.plan() 的内部逻辑（简化）
def plan(self, intermediate_steps, **kwargs):
    # 组装 prompt
    prompt = self.prompt_template.format(
        input=kwargs["input"],
        agent_scratchpad=self._format_steps(intermediate_steps),
        tools=self._format_tools(),
    )
    # 调用 LLM
    llm_output = self.llm.invoke(prompt)
    # 解析输出
    return self.output_parser.parse(llm_output)
```

`agent_scratchpad` 就是把 intermediate_steps 格式化成文本，让 LLM 看到之前的推理轨迹：

```
Thought: 我需要查天气
Action: get_weather
Action Input: {"city": "北京"}
Observation: 晴天, 22°C
Thought: 现在我知道天气了...
```

#### ③ 解析错误处理

当 LLM 输出格式不符合预期时（比如没有按 `Action: xxx` 格式输出），OutputParser 会抛出 `OutputParserException`：

```python
except OutputParserException as e:
    if isinstance(self.handle_parsing_errors, bool):
        if not self.handle_parsing_errors:
            raise e  # 直接抛出，循环终止
        observation = "Invalid or incomplete response"
    elif isinstance(self.handle_parsing_errors, str):
        observation = self.handle_parsing_errors  # 使用自定义错误信息
    elif callable(self.handle_parsing_errors):
        observation = self.handle_parsing_errors(e)  # 调用自定义处理函数

    # 构造一个特殊的 AgentAction，工具名为 "_Exception"
    output = AgentAction("_Exception", observation, text)
    # 用 ExceptionTool 处理
    observation = ExceptionTool().run(
        output.tool_input,
        verbose=self.verbose,
        color=None,
        callbacks=run_manager.get_child() if run_manager else None,
    )
    # 将错误作为一步反馈给 LLM，让它自我修正
    yield AgentStep(action=output, observation=observation)
    return
```

**这是 Agent 自修复能力的关键。** 当 `handle_parsing_errors=True` 时，解析错误不会终止循环，而是作为一个 observation 反馈给 LLM："你的输出格式不对"，LLM 在下一轮会尝试修正格式。

## _perform_agent_action：工具执行

定义在 `agent.py:1380`，负责查找并执行工具：

```python
# 来源: agent.py 第 1380 行（简化版）
def _perform_agent_action(
    self,
    name_to_tool_map: dict[str, BaseTool],
    color_mapping: dict[str, str],
    agent_action: AgentAction,
    run_manager: CallbackManagerForChainRun | None = None,
) -> AgentStep:

    # ① 在工具表中查找
    if agent_action.tool in name_to_tool_map:
        tool = name_to_tool_map[agent_action.tool]
        color = color_mapping[agent_action.tool]

        # ② 执行工具
        observation = tool.run(
            agent_action.tool_input,
            verbose=self.verbose,
            color=color,
            callbacks=run_manager.get_child() if run_manager else None,
        )
    else:
        # ③ 工具不存在 → 使用 InvalidTool
        observation = InvalidTool().run(
            {
                "requested_tool_name": agent_action.tool,
                "available_tool_names": list(name_to_tool_map.keys()),
            },
            verbose=self.verbose,
            color=None,
            callbacks=run_manager.get_child() if run_manager else None,
        )

    # ④ 返回 AgentStep
    return AgentStep(action=agent_action, observation=observation)
```

### InvalidTool —— 优雅处理工具不存在

当 LLM 幻觉出一个不存在的工具名时，不是直接报错，而是用 `InvalidTool` 返回一条友好的错误信息：

```python
# InvalidTool 的 run 方法返回类似这样的信息：
# "get_wether is not a valid tool, try one of [get_weather, search]."
```

这条信息作为 observation 反馈给 LLM，LLM 在下一轮通常能修正工具名（比如把 `get_wether` 改成 `get_weather`）。

### return_direct —— 工具直接返回

有些工具的结果就是最终答案，不需要 LLM 再加工：

```python
# 在 _call 主循环中
if len(next_step_output) == 1:
    tool_return = self._get_tool_return(next_step_output[0])
    if tool_return is not None:
        return self._return(tool_return, intermediate_steps, run_manager)
```

```python
# _get_tool_return 检查工具是否设置了 return_direct
def _get_tool_return(self, next_step_output):
    agent_action = next_step_output.action
    tool_name = agent_action.tool
    if tool_name in self.name_to_tool_map:
        tool = self.name_to_tool_map[tool_name]
        if tool.return_direct:
            return AgentFinish(
                {"output": next_step_output.observation},
                ""
            )
    return None
```

**使用场景：** 比如一个"生成图片"工具，它的输出（图片 URL）就是最终答案，不需要 LLM 再总结一遍。

```python
@tool(return_direct=True)
def generate_image(prompt: str) -> str:
    """生成图片并返回 URL"""
    url = call_dalle_api(prompt)
    return f"![生成的图片]({url})"
# 这个工具执行后，结果直接返回给用户，跳过 LLM 的最终推理
```

## 异步执行：_acall 与并行工具调用

`_acall` 是 `_call` 的异步版本，定义在 `agent.py:1624`。核心区别在于工具的并行执行：

```python
# 来源: agent.py 第 1624 行（简化版）
async def _acall(self, inputs, run_manager=None):
    # ... 初始化逻辑与 _call 相同 ...

    try:
        async with asyncio_timeout(self.max_execution_time):
            while self._should_continue(iterations, time_elapsed):
                next_step_output = await self._atake_next_step(
                    name_to_tool_map, color_mapping,
                    inputs, intermediate_steps, run_manager,
                )
                # ... 后续逻辑与 _call 相同 ...
    except TimeoutError:
        # 超时 → early stopping
        output = self._action_agent.return_stopped_response(
            self.early_stopping_method, intermediate_steps, **inputs
        )
        return self._return(output, intermediate_steps, run_manager)
```

### asyncio.gather —— 并行执行多个工具

当 MultiActionAgent 一次返回多个 AgentAction 时，异步版本可以并行执行：

```python
# _aiter_next_step 中的并行执行逻辑
async def _aiter_next_step(self, ...):
    # ... agent.aplan() 获取 actions ...

    if isinstance(output, AgentAction):
        actions = [output]
    else:
        actions = output

    # 关键：用 asyncio.gather 并行执行所有工具
    result = await asyncio.gather(
        *[
            self._aperform_agent_action(
                name_to_tool_map, color_mapping, agent_action, run_manager
            )
            for agent_action in actions
        ]
    )
    for agent_step in result:
        yield agent_step
```

**同步 vs 异步对比：**

```
同步 _call（串行执行）:
  工具A ████████ → 工具B ████████ → 工具C ████████
  总耗时: 3 × T

异步 _acall（并行执行）:
  工具A ████████
  工具B ████████
  工具C ████████
  总耗时: 1 × T
```

## 完整执行流程图

把所有方法串在一起：

```
executor.invoke({"input": "..."})
    │
    ▼
_call(inputs)
    │
    ├── name_to_tool_map = {tool.name: tool}
    ├── intermediate_steps = []
    ├── iterations = 0
    │
    ▼
while _should_continue(iterations, time_elapsed):
    │
    ▼
  _take_next_step(...)  →  _iter_next_step(...)
    │
    ├── _prepare_intermediate_steps()  ← 裁剪历史
    │
    ├── agent.plan(intermediate_steps, **inputs)
    │     │
    │     ├── 成功 → AgentAction 或 AgentFinish
    │     │
    │     └── OutputParserException
    │           ├── handle_parsing_errors=True → 错误作为 observation
    │           └── handle_parsing_errors=False → 抛出异常
    │
    ├── AgentFinish? → yield → _call 中 return self._return(...)
    │
    └── AgentAction? → _perform_agent_action(...)
          │
          ├── tool 存在 → tool.run() → observation
          │
          └── tool 不存在 → InvalidTool().run() → 错误提示
          │
          └── yield AgentStep(action, observation)
    │
    ▼
intermediate_steps.extend(next_step_output)
iterations += 1
    │
    ▼
[回到 while 循环顶部]
    │
    ▼ (超出限制)
return_stopped_response(early_stopping_method)
```

## 用代码模拟完整循环

手写一个简化版的 AgentExecutor 循环，帮助理解底层原理：

```python
"""
手写简化版 AgentExecutor 循环
目的：理解 _call 和 _iter_next_step 的核心逻辑
"""
from dataclasses import dataclass


@dataclass
class AgentAction:
    tool: str
    tool_input: dict
    log: str


@dataclass
class AgentFinish:
    return_values: dict
    log: str


@dataclass
class AgentStep:
    action: AgentAction
    observation: str


def simple_agent_loop(
    agent_plan_fn,       # 模拟 agent.plan()
    tools: dict,         # {工具名: 工具函数}
    inputs: dict,        # 用户输入
    max_iterations: int = 15,
):
    """简化版 AgentExecutor._call()"""

    intermediate_steps: list[AgentStep] = []
    iterations = 0

    # === 主循环（对应 _call 的 while 循环）===
    while iterations < max_iterations:

        # === 单步执行（对应 _iter_next_step）===

        # 1. 调用 LLM 决策
        output = agent_plan_fn(intermediate_steps, **inputs)

        # 2. 如果是最终答案，返回
        if isinstance(output, AgentFinish):
            print(f"[完成] {output.return_values}")
            return output.return_values

        # 3. 如果是动作，执行工具
        action = output
        print(f"[第{iterations+1}轮] 调用工具: {action.tool}({action.tool_input})")

        # 4. 查找并执行工具（对应 _perform_agent_action）
        if action.tool in tools:
            observation = tools[action.tool](action.tool_input)
        else:
            observation = f"工具 '{action.tool}' 不存在，可用: {list(tools.keys())}"

        print(f"[第{iterations+1}轮] 结果: {observation}")

        # 5. 记录这一步
        step = AgentStep(action=action, observation=observation)
        intermediate_steps.append(step)
        iterations += 1

    # 超出限制
    print(f"[超时] 达到最大迭代次数 {max_iterations}")
    return {"output": "Agent stopped due to iteration limit."}


# === 使用示例 ===

# 模拟工具
def get_weather(params):
    return f"{params.get('city', '未知')}: 晴天, 22°C"

def search_hotels(params):
    return f"{params.get('city', '未知')}推荐: 亚朵酒店 ¥400/晚"

tools = {"get_weather": get_weather, "search_hotels": search_hotels}

# 模拟 agent.plan()：预设的决策序列
call_count = 0
def mock_plan(intermediate_steps, **kwargs):
    global call_count
    call_count += 1
    if call_count == 1:
        return AgentAction("get_weather", {"city": "北京"}, "需要查天气")
    elif call_count == 2:
        return AgentAction("search_hotels", {"city": "北京"}, "需要查酒店")
    else:
        return AgentFinish(
            {"output": "北京晴天22°C，推荐亚朵酒店¥400/晚"},
            "信息收集完毕"
        )

result = simple_agent_loop(mock_plan, tools, {"input": "北京出差"})
# 输出:
# [第1轮] 调用工具: get_weather({'city': '北京'})
# [第1轮] 结果: 北京: 晴天, 22°C
# [第2轮] 调用工具: search_hotels({'city': '北京'})
# [第2轮] 结果: 北京推荐: 亚朵酒店 ¥400/晚
# [完成] {'output': '北京晴天22°C，推荐亚朵酒店¥400/晚'}
```

## 与 AI Agent 开发的关系

理解 `_call` 和 `_iter_next_step` 的意义：

1. **自定义循环控制**：知道 `_should_continue` 的逻辑后，你可以合理设置 `max_iterations` 和 `max_execution_time`，在成本和效果之间找到平衡

2. **调试 Agent 行为**：当 Agent 行为异常时，打开 `verbose=True` 或 `return_intermediate_steps=True`，可以看到每一轮的 action 和 observation，精确定位问题

3. **迁移到新架构**：LangChain v1 的 `create_agent` 底层用 LangGraph 实现了类似的循环，但用状态图替代了 while 循环。理解旧版原理，迁移时就能理解新版的设计动机

4. **RAG Agent 优化**：在 RAG 场景中，Agent 可能需要多轮检索。理解循环机制后，你可以通过 `trim_intermediate_steps` 控制上下文长度，通过 `max_iterations` 限制检索次数，避免成本失控

## 常见问题

**Q1: _take_next_step 和 _iter_next_step 是什么关系？**

`_take_next_step` 是对 `_iter_next_step` 的包装。`_iter_next_step` 是生成器（yield），`_take_next_step` 把生成器的结果收集成列表返回。这样 `_call` 中可以直接用 `isinstance` 判断返回类型。

**Q2: intermediate_steps 会无限增长吗？**

如果不设置 `trim_intermediate_steps`，是的。每轮循环追加一个 `(AgentAction, observation)` 元组。15 轮循环后可能有几千 token。生产环境建议设置裁剪策略。

**Q3: 为什么用生成器（yield）而不是直接 return？**

因为 `AgentExecutorIterator` 需要逐步迭代。生成器让调用方可以在每一步之间插入自定义逻辑（比如日志、回调、人工审核），而不是等整个循环跑完。
