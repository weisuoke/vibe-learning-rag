# 实战代码 - 场景4：TypedDict + Pydantic 混合架构

## 场景说明

本场景演示 LangGraph 中最推荐的生产级状态管理模式——混合架构。你将看到：

1. 用 Pydantic `BaseModel` 作为 `input` schema，严格验证外部输入
2. 用 `TypedDict` + Reducer 作为内部状态，支持多节点结果累积
3. 用 Pydantic `BaseModel` 作为 `output` schema，规范对外输出
4. 三层 schema 如何在 `StateGraph` 构造函数中组合

核心思想：**边界处严格验证，内部流转保持轻量**。这不是折中方案，而是两种工具各取所长。

[来源: reference/context7_langgraph_01.md | LangGraph 官方文档]
[来源: reference/search_状态验证_01.md]

## 完整代码

```python
"""
LangGraph 状态验证实战 - 场景4：TypedDict + Pydantic 混合架构
演示：input_schema 用 Pydantic 验证，内部状态用 TypedDict + Reducer

运行环境：Python 3.13+, langgraph, pydantic
安装依赖：uv add langgraph pydantic
"""

from pydantic import BaseModel, Field, field_validator, ValidationError
from typing import Annotated
from typing_extensions import TypedDict
from operator import add
from langgraph.graph import StateGraph, START, END


# ============================================================
# 1. 输入验证层（Pydantic）—— 守住外部边界
# ============================================================

class UserInput(BaseModel):
    """用户输入 schema - 严格验证

    这一层的职责：
    - 拦截非法输入（空查询、超长文本、越界参数）
    - 自动类型转换（字符串 "3" → 整数 3）
    - 数据清洗（去除首尾空白）

    只定义"从外部进来"的字段，不包含内部处理字段。
    """
    query: str = Field(
        min_length=1,
        max_length=500,
        description="用户查询，1~500 字符",
    )
    top_k: int = Field(
        default=5,
        ge=1,
        le=20,
        description="返回结果数量，1~20",
    )

    @field_validator("query")
    @classmethod
    def clean_query(cls, v: str) -> str:
        """去除首尾空白"""
        return v.strip()


# ============================================================
# 2. 内部状态（TypedDict + Reducer）—— 轻量流转
# ============================================================

class InternalState(TypedDict):
    """内部状态 schema - 零验证开销

    这一层的职责：
    - 在节点之间传递数据（字典访问，无属性开销）
    - 使用 Reducer 累积多个节点的输出（search_results, processing_log）
    - 包含所有字段：输入字段 + 中间结果 + 最终输出

    关键点：
    - Annotated[list, add] 表示多个节点返回的列表会自动合并
    - 这是 TypedDict 独有的优势，Pydantic BaseModel 不直接支持 Reducer
    """
    # 从输入继承的字段
    query: str
    top_k: int
    # 中间结果（使用 Reducer 累积）
    search_results: Annotated[list, add]
    processing_log: Annotated[list[str], add]
    # 最终输出
    final_answer: str


# ============================================================
# 3. 输出验证层（Pydantic）—— 规范对外接口
# ============================================================

class OutputState(BaseModel):
    """输出 schema - 只暴露需要的字段

    这一层的职责：
    - 过滤内部字段（search_results、processing_log 不对外暴露）
    - 确保输出格式稳定（即使内部实现变化，对外接口不变）
    - 提供默认值（sources 默认为空列表）
    """
    final_answer: str
    sources: list[str] = []


# ============================================================
# 4. 定义节点函数
# ============================================================

def search_node(state: InternalState):
    """检索节点 - 模拟向量检索

    注意：
    - 参数类型是 InternalState（TypedDict），用字典方式访问
    - 返回的 search_results 是列表，会通过 Reducer(add) 累积
    - 返回的 processing_log 也是列表，同样累积
    """
    query = state["query"]
    top_k = state["top_k"]

    # 模拟检索结果
    mock_results = [
        {"title": f"关于「{query}」的文档_{i+1}", "score": round(0.95 - i * 0.1, 2)}
        for i in range(top_k)
    ]

    return {
        "search_results": mock_results,  # Reducer: 追加到列表
        "processing_log": [f"检索完成: 找到 {len(mock_results)} 条结果"],
    }


def rerank_node(state: InternalState):
    """重排序节点 - 模拟 ReRank

    演示 Reducer 的累积效果：
    - 这个节点也返回 search_results，会和 search_node 的结果合并
    - processing_log 同样累积
    """
    results = state["search_results"]
    # 模拟重排序：按 score 降序
    sorted_results = sorted(results, key=lambda x: x["score"], reverse=True)
    top_result = sorted_results[:3] if len(sorted_results) >= 3 else sorted_results

    return {
        # 注意：这里返回的列表会 ADD 到已有的 search_results 上
        # 如果你只想替换，不要用 Reducer，或者用不同的字段名
        "search_results": [],  # 不追加新结果，只记录日志
        "processing_log": [f"重排序完成: 保留 top {len(top_result)} 条"],
    }


def generate_node(state: InternalState):
    """生成节点 - 模拟 LLM 生成回答"""
    results = state["search_results"]
    query = state["query"]

    # 模拟基于检索结果生成回答
    context_summary = ", ".join(r["title"] for r in results[:3])
    answer = f"基于 {len(results)} 条检索结果回答「{query}」: 参考了 [{context_summary}]"

    return {
        "final_answer": answer,
        "processing_log": [f"生成完成: 回答长度 {len(answer)} 字符"],
    }


# ============================================================
# 5. 构建混合架构图
# ============================================================

# 关键：StateGraph 的三个参数分别指定三层 schema
builder = StateGraph(
    InternalState,       # 内部状态 schema（TypedDict，支持 Reducer）
    input=UserInput,     # 输入 schema（Pydantic，严格验证）
    output=OutputState,  # 输出 schema（Pydantic，规范输出）
)

builder.add_node("search", search_node)
builder.add_node("rerank", rerank_node)
builder.add_node("generate", generate_node)

builder.add_edge(START, "search")
builder.add_edge("search", "rerank")
builder.add_edge("rerank", "generate")
builder.add_edge("generate", END)

graph = builder.compile()


# ============================================================
# 6. 测试：正常调用
# ============================================================

print("=" * 60)
print("=== 测试 1：正常调用 ===")
print("=" * 60)

result = graph.invoke({"query": "什么是向量数据库？", "top_k": 3})
print(f"输出类型: {type(result)}")
print(f"结果: {result}")
# 注意：输出只包含 OutputState 定义的字段（final_answer, sources）
# search_results、processing_log、query、top_k 都被过滤掉了


# ============================================================
# 7. 测试：输入验证拦截
# ============================================================

print()
print("=" * 60)
print("=== 测试 2：空查询 → Pydantic 输入验证拦截 ===")
print("=" * 60)

try:
    graph.invoke({"query": ""})
except ValidationError as e:
    print(f"验证失败，共 {e.error_count()} 个错误:")
    for err in e.errors():
        field = ".".join(str(loc) for loc in err["loc"])
        print(f"  [{field}] {err['msg']}")


# ============================================================
# 8. 测试：自动类型转换
# ============================================================

print()
print("=" * 60)
print("=== 测试 3：字符串 top_k → Pydantic 自动转换为 int ===")
print("=" * 60)

result = graph.invoke({"query": "测试类型转换", "top_k": "3"})
print(f"结果: {result}")


# ============================================================
# 9. 测试：top_k 越界
# ============================================================

print()
print("=" * 60)
print("=== 测试 4：top_k 越界 → Pydantic 约束拦截 ===")
print("=" * 60)

try:
    graph.invoke({"query": "测试", "top_k": 100})
except ValidationError as e:
    print(f"验证失败:")
    for err in e.errors():
        field = ".".join(str(loc) for loc in err["loc"])
        print(f"  [{field}] {err['msg']}")


# ============================================================
# 10. 测试：只传必填字段（默认值生效）
# ============================================================

print()
print("=" * 60)
print("=== 测试 5：只传 query，top_k 使用默认值 5 ===")
print("=" * 60)

result = graph.invoke({"query": "默认参数测试"})
print(f"结果: {result}")
```

## 关键知识点解析

### 1. StateGraph 的三个 schema 参数

```python
StateGraph(
    InternalState,       # 第一个参数：内部状态 schema
    input=UserInput,     # input 关键字参数：输入验证 schema
    output=OutputState,  # output 关键字参数：输出过滤 schema
)
```

| 参数 | 类型 | 触发时机 | 职责 |
|------|------|---------|------|
| 第一个参数 | TypedDict | 节点间数据流转 | 定义所有字段 + Reducer |
| `input` | Pydantic | `graph.invoke()` 接收输入时 | 验证 + 转换 + 清洗 |
| `output` | Pydantic | `graph.invoke()` 返回结果时 | 过滤字段 + 格式规范 |

### 2. Reducer 为什么只能用 TypedDict？

Pydantic BaseModel 的字段赋值会触发验证，而 Reducer 需要在运行时动态合并值（如 `existing_list + new_list`）。TypedDict 本质是普通字典，LangGraph 可以自由操作它的值，不受验证约束。

```python
# TypedDict：Reducer 正常工作
class State(TypedDict):
    results: Annotated[list, add]  # add = lambda a, b: a + b

# Pydantic：Reducer 与验证机制冲突
# 不推荐在 Pydantic BaseModel 中使用 Reducer
```

### 3. 输出过滤的实际效果

`output=OutputState` 的作用是**只返回 OutputState 中定义的字段**：

- `final_answer` → 保留（OutputState 中有定义）
- `sources` → 保留（OutputState 中有定义，默认 `[]`）
- `query` → 过滤掉
- `top_k` → 过滤掉
- `search_results` → 过滤掉
- `processing_log` → 过滤掉

这意味着内部实现细节不会泄露给调用方，API 接口保持稳定。

### 4. 混合架构 vs 纯 Pydantic 的取舍

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 简单图（2-3 个节点） | 纯 Pydantic | 代码量少，不需要 Reducer |
| 复杂图（多节点累积结果） | 混合架构 | 需要 Reducer，内部状态频繁读写 |
| API 服务 | 混合架构 | 输入验证 + 输出过滤都需要 |
| 快速原型 | 纯 TypedDict | 最快上手，后续再加验证 |

## RAG 开发中的应用

混合架构天然适配 RAG 管道的三段式结构：

```
用户查询 ──► [Pydantic 验证] ──► 检索 ──► 重排序 ──► 生成 ──► [Pydantic 输出]
                                   │         │         │
                                   ▼         ▼         ▼
                              TypedDict + Reducer 累积中间结果
```

- **输入层**：验证查询长度、top_k 范围、temperature 约束
- **内部层**：检索结果用 `Annotated[list, add]` 累积，多路检索自动合并
- **输出层**：只返回 `answer` 和 `sources`，隐藏 embedding、score 等内部细节
