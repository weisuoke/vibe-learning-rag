## 快速参考

### 推荐配置（2025-2026）

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 通用配置（推荐）
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,  # 1000 characters
    chunk_overlap=200,  # 20% 重叠
    add_start_index=True  # 便于溯源
)

# 基准测试最佳配置
text_splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
    model_name="gpt-4",
    chunk_size=512,  # 512 tokens（基准测试最佳）
    chunk_overlap=100
)
```

### 常用分块器对比

| 分块器 | 适用场景 | 优点 | 缺点 |
|--------|----------|------|------|
| RecursiveCharacterTextSplitter | 通用文本 | 语义完整性好 | 性能略低 |
| CharacterTextSplitter | 简单文本 | 性能高 | 灵活性低 |
| TokenTextSplitter | 成本敏感场景 | 精确控制 token | 需要 tiktoken |
| HTMLHeaderTextSplitter | 网页内容 | 保留结构 | 仅限 HTML |
| MarkdownHeaderTextSplitter | 技术文档 | 保留层级 | 仅限 Markdown |
| PythonCodeTextSplitter | 代码分析 | 语法感知 | 仅限代码 |

### 关键参数速查

**chunk_size（块大小）**：
- 小文档（< 1000 字符）：500
- 中等文档（1000-5000 字符）：1000
- 大文档（> 5000 字符）：1500-2000

**chunk_overlap（块重叠）**：
- 通常设置为 chunk_size 的 15-20%
- 最小值：100 字符
- 最大值：chunk_size 的 30%

**length_function（长度计算）**：
- 默认：`len`（字符数）
- Token 计数：`from_tiktoken_encoder()`
- 自定义：传入自定义函数

## 常见问题

### Q1: 应该使用哪个分块器？
**A**: 大多数情况下使用 `RecursiveCharacterTextSplitter`。它是 LangChain 推荐的默认分块器，平衡了语义完整性和性能。

### Q2: chunk_size 应该设置多大？
**A**:
- 通用推荐：1000 characters 或 512 tokens
- 根据 LLM 的 context window 调整
- 通过基准测试找到最佳值

### Q3: chunk_overlap 有什么用？
**A**:
- 保持上下文连续性
- 避免在句子或段落中间切断
- 提高检索召回率
- 通常设置为 chunk_size 的 15-20%

### Q4: 如何处理代码文件？
**A**: 使用 `RecursiveCharacterTextSplitter.from_language(Language.PYTHON)` 或 `PythonCodeTextSplitter`，它们会根据代码语法进行分块。

### Q5: 如何优化成本？
**A**: 使用 `from_tiktoken_encoder()` 精确控制 token 数量，避免超出 LLM 的 context window。
