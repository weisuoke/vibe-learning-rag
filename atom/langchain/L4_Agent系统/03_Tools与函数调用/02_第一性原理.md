# Tools 与函数调用 - 第一性原理

> 从"LLM 只能生成文本"这一基本事实，推导出 Tool 系统的完整设计

---

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而非依赖类比和经验。

物理学家费曼说："把事物拆解到最基本的真理，然后从那里开始推理。"

对 Tools 与函数调用，第一性原理意味着：
- 不去看"LangChain 怎么封装 Tool"
- 而是从"LLM 为什么需要 Tool"出发
- 推导出 Tool 系统必须是什么样的

---

## Tools 与函数调用的第一性原理

### 1. 最基础的定义

**Tool = 函数 + 描述 + Schema**

LLM 本身只能生成文本，无法执行代码或访问外部系统。Tool 就是给 LLM 一个"说明书"，告诉它有哪些函数可以调用、每个函数需要什么参数、返回什么结果。

拆开来看：
- **函数**：真正干活的代码（查天气、算数学、发邮件）
- **描述**：告诉 LLM 这个函数能做什么（自然语言）
- **Schema**：告诉 LLM 这个函数需要什么参数、什么类型（JSON Schema）

```python
# Tool 的三要素，用最原始的方式表达
tool_spec = {
    "name": "get_weather",                    # 函数名
    "description": "查询指定城市的天气",        # 描述（给 LLM 看的）
    "parameters": {                            # Schema（参数规范）
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "城市名称"}
        },
        "required": ["city"]
    }
}

# 真正执行的函数
def get_weather(city: str) -> str:
    return f"{city}今天晴，25°C"
```

这三者缺一不可：

| 缺少什么 | 后果 | 结论 |
|----------|------|------|
| 缺少函数 | 有说明书但没人干活 | 必须保留 |
| 缺少描述 | LLM 不知道这个工具能做什么，无法决策 | 必须保留 |
| 缺少 Schema | LLM 不知道该传什么参数，调用必然出错 | 必须保留 |

LangChain 源码中的 `BaseTool`（`langchain_core/tools/base.py`）正是对这三要素的抽象：`name` + `description` + `args_schema`。所有 Tool 的创建方式——`@tool` 装饰器、`StructuredTool.from_function()`、`BaseTool` 子类——本质上都是在填充这三个字段。

---

### 2. 为什么需要 Tool？

**核心问题：LLM 是一个"只会说话的大脑"，如何让它"动手做事"？**

LLM 的能力边界非常清晰：
```
LLM 能做的：
  ✅ 理解自然语言
  ✅ 推理和规划
  ✅ 生成文本回复
  ✅ 分析已有信息

LLM 做不到的：
  ❌ 访问实时数据（天气、股价、新闻）
  ❌ 执行精确计算（大数乘法、统计分析）
  ❌ 操作外部系统（发邮件、查数据库、调 API）
  ❌ 读写文件、访问网络
```

**没有 Tool 会怎样？**
```python
# 没有 Tool：LLM 只能"编"答案
response = llm.invoke("今天北京天气怎么样？")
# → "我无法获取实时天气信息，但北京这个季节通常..."  （瞎猜）

response = llm.invoke("帮我给张三发一封邮件")
# → "好的，以下是邮件草稿..."  （只能写草稿，发不出去）
```

**有了 Tool：LLM 变成"能做事的大脑"**
```python
# 有 Tool：LLM 决策 + 工具执行
response = llm_with_tools.invoke("今天北京天气怎么样？")
# → AIMessage(tool_calls=[{"name": "get_weather", "args": {"city": "北京"}}])
# → 执行 get_weather("北京") → "晴，25°C"
# → "北京今天天气晴朗，气温25°C，适合出行。"  （真实数据）
```

Tool 是 LLM 从"对话者"变成"执行者"的关键。这不是锦上添花，而是 Agent 系统存在的根本前提。

---

### 3. Tool 的三层价值

#### 价值 1：能力扩展 — 让 LLM 突破文本生成的边界

**定义**：Tool 把 LLM 的能力从"生成文本"扩展到"与真实世界交互"。

**为什么重要**：
- LLM 的训练数据有截止日期，Tool 让它获取实时信息
- LLM 的数学能力有限，Tool 让它调用计算器
- LLM 无法执行副作用，Tool 让它操作外部系统

```python
# 能力扩展的典型场景
@tool
def search_web(query: str) -> str:
    """搜索互联网获取实时信息"""  # LLM 获得了"上网"的能力
    ...

@tool
def calculate(expression: str) -> float:
    """精确计算数学表达式"""  # LLM 获得了"精确计算"的能力
    return eval(expression)

@tool
def send_email(to: str, subject: str, body: str) -> str:
    """发送电子邮件"""  # LLM 获得了"发邮件"的能力
    ...
```

**类比**：
- 就像给手机装 App —— 手机本身只能打电话，装了微信能聊天，装了支付宝能付款
- 就像给前端页面加 API 调用 —— 纯 HTML 只能展示静态内容，加了 fetch 就能获取动态数据

#### 价值 2：标准化接口 — 统一的 Tool 协议让任何函数都能被 LLM 调用

**定义**：不管你的函数是查天气、算数学还是发邮件，只要符合 Tool 协议（名称 + 描述 + Schema），LLM 就能调用它。

**为什么重要**：
- 开发者不需要为每个函数写特殊的 LLM 集成代码
- LLM 不需要为每种工具学习不同的调用方式
- 新工具可以即插即用

LangChain 源码中的 `convert_to_openai_tool()`（`function_calling.py`）就是这个标准化的体现——它能把 `BaseTool`、`Pydantic BaseModel`、`TypedDict`、普通函数等多种输入统一转换为 OpenAI 的 Tool 格式。不管你用什么方式定义 Tool，最终都走同一条转换链路。

**类比**：
- 就像 USB 接口 —— 不管是键盘、鼠标还是U盘，只要是 USB 接口就能插上用
- 就像 REST API 规范 —— 不管后端用什么语言，前端只需要知道 URL 和参数格式

#### 价值 3：安全可控 — 通过 Schema 验证确保 LLM 传递正确的参数

**定义**：Schema 不只是"说明书"，更是"安全检查站"。LLM 生成的参数必须符合 Schema 定义的类型和约束。

**为什么重要**：
- LLM 可能生成错误类型的参数（把数字写成字符串）
- LLM 可能遗漏必填参数或传入危险的参数值
- Schema 验证在执行前拦截这些问题

```python
from pydantic import BaseModel, Field

class DatabaseQuery(BaseModel):
    """数据库查询参数 — Schema 就是安全边界"""
    table: str = Field(description="表名", pattern="^[a-zA-Z_]+$")  # 防止 SQL 注入
    limit: int = Field(description="返回条数", ge=1, le=100)         # 限制范围
    filters: dict = Field(description="过滤条件")

# LLM 生成的参数会经过 Pydantic 验证
# 如果 table="users; DROP TABLE users" → 验证失败，不会执行
# 如果 limit=10000 → 验证失败，超出范围
```

LangChain 源码中，`BaseTool` 的 `args_schema` 属性就是用 Pydantic BaseModel 定义的。调用 Tool 时，参数会先经过 Schema 验证，验证通过才执行函数。

**类比**：
- 就像 TypeScript 的类型检查 —— 编译时就能发现参数类型错误，不用等到运行时
- 就像机场安检 —— 不管你是谁，行李都要过 X 光机

### 4. 从第一性原理推导 Agent 工具调用

现在我们从最基本的事实出发，一步步推导出完整的工具调用协议。

#### 第一步：LLM 只能生成文本

```
事实：LLM 的输出是文本（token 序列）
限制：文本本身不能执行任何操作
```

```python
response = llm.invoke("帮我查一下北京天气")
# response = "我无法访问实时天气数据..."
# 纯文本，没有任何实际效果
```

#### 第二步：需要一种方式让 LLM "描述想做什么"

```
事实：LLM 不能直接执行函数
解法：让 LLM 用结构化文本"描述"它想调用什么函数
```

```python
# 最原始的方案：让 LLM 输出特定格式的文本
response = llm.invoke("""
你有以下工具可用：
- get_weather(city): 查询城市天气

用户问题：北京天气怎么样？
请用 JSON 格式输出你想调用的工具。
""")
# LLM 输出：{"tool": "get_weather", "args": {"city": "北京"}}
```

这就是函数调用的雏形——LLM 不执行函数，而是**输出一个调用请求**。

#### 第三步：函数调用协议标准化

```
事实：不同模型输出格式不同，手动解析容易出错
解法：定义标准化的函数调用协议
```

OpenAI 率先定义了 Function Calling 协议，LangChain 将其抽象为 `tool_calls`：

```python
# 标准化的 tool_calls 格式（LangChain 源码 messages/tool.py）
class ToolCall(TypedDict):
    name: str       # 工具名称
    args: dict      # 参数字典
    id: str         # 调用 ID（用于匹配请求和响应）
    type: str       # 固定为 "tool_call"

# 模型返回的 AIMessage 中包含 tool_calls
response = model_with_tools.invoke("北京天气怎么样？")
response.tool_calls
# → [{"name": "get_weather", "args": {"city": "北京"}, "id": "call_abc123", "type": "tool_call"}]
```

#### 第四步：执行函数，返回结果

```
事实：LLM 输出了调用请求，但请求本身不会自动执行
解法：外部系统解析请求，执行函数，把结果封装为 ToolMessage
```

```python
from langchain_core.messages import ToolMessage

# 执行工具
for tool_call in response.tool_calls:
    result = get_weather(**tool_call["args"])

    # 把结果封装为 ToolMessage
    tool_message = ToolMessage(
        content=result,                    # 工具返回的内容
        tool_call_id=tool_call["id"],      # 关联到哪个调用请求
    )
```

`tool_call_id` 是关键——它把"请求"和"响应"关联起来。就像快递单号，寄件和收件靠它匹配。

#### 第五步：LLM 根据结果继续推理

```
事实：工具结果需要反馈给 LLM，它才能生成最终回答
解法：把 ToolMessage 加入对话历史，让 LLM 继续推理
```

```python
messages = [
    HumanMessage("北京天气怎么样？"),
    AIMessage(tool_calls=[{"name": "get_weather", "args": {"city": "北京"}, "id": "call_abc123"}]),
    ToolMessage(content="北京今天晴，25°C", tool_call_id="call_abc123"),
]

# LLM 看到工具结果后，生成最终回答
final_response = model.invoke(messages)
# → "北京今天天气晴朗，气温25°C，非常适合户外活动。"
```

#### 第六步：封装成循环 — 这就是 Agent

```
事实：复杂任务可能需要多次工具调用
解法：把上面的流程放进循环，直到 LLM 不再请求工具调用
```

```python
# Agent 的核心循环（伪代码）
messages = [HumanMessage(user_question)]

while True:
    response = model_with_tools.invoke(messages)
    messages.append(response)

    # 没有 tool_calls → 任务完成
    if not response.tool_calls:
        return response.content

    # 有 tool_calls → 执行工具，继续循环
    for tool_call in response.tool_calls:
        result = execute_tool(tool_call)
        messages.append(ToolMessage(content=result, tool_call_id=tool_call["id"]))
```

**推导总结图**：

```
1. LLM 只能生成文本
   ↓
2. 但实际任务需要执行动作（搜索、计算、API 调用）
   ↓
3. 需要一种方式让 LLM "描述想做什么"
   ↓
4. 函数调用协议：LLM 输出结构化的 JSON（工具名 + 参数）
   ↓
5. 外部系统执行函数，返回 ToolMessage
   ↓
6. LLM 根据结果继续推理或生成最终回答
   ↓
7. 这就是 Agent 的核心循环：思考 → 调用工具 → 观察结果 → 继续思考
```

每一步都是从一个不可回避的事实出发，推导出一个必要的设计决策。

---

### 5. 一句话总结第一性原理

**Tool 是 LLM 从"只会说"到"能做事"的桥梁，通过标准化的函数描述（description + Schema）和调用协议（tool_calls + ToolMessage），让 AI 能够安全、可控地与外部世界交互。**

## LangChain Tool 系统的设计推导

### 从第一性原理到源码架构

理解了"为什么需要 Tool"，我们可以推导出 LangChain Tool 系统的架构为什么是这样的：

```
需求                          LangChain 的设计决策
──────────────────────────────────────────────────────
Tool 需要函数+描述+Schema  →  BaseTool(name, description, args_schema)
创建 Tool 要简单           →  @tool 装饰器（自动推断 Schema）
复杂场景需要自定义 Schema  →  StructuredTool.from_function()
最大灵活性                 →  BaseTool 子类（完全控制）
Tool 要能被模型调用        →  bind_tools() 绑定到模型
不同模型格式不同           →  convert_to_openai_tool() 统一转换
调用结果要回传给模型       →  ToolMessage(content, tool_call_id)
```

### 三种创建方式的推导

为什么 LangChain 提供三种创建 Tool 的方式？因为不同场景对简洁性和灵活性的需求不同：

```
@tool 装饰器        StructuredTool        BaseTool 子类
  最简洁     ←──────────────────────────→   最灵活
  自动推断 Schema    手动指定 Schema       完全自定义
  适合简单函数       适合需要精确控制参数    适合复杂工具
```

- **@tool 装饰器**：80% 的场景用这个就够了，Schema 从类型注解自动推断
- **StructuredTool.from_function()**：需要自定义 Pydantic Schema 时使用
- **BaseTool 子类**：需要维护状态、异步执行、自定义错误处理时使用

详细用法见 [03_核心概念_1_Tool定义三种方式.md](./03_核心概念_1_Tool定义三种方式.md)。

## 与 RAG 开发的关系

在 RAG 系统中，Tool 让 Agent 从"只能检索"变成"能检索、能搜索、能计算、能操作"：

```python
# RAG Agent 的典型工具集
@tool
def search_knowledge_base(query: str) -> str:
    """在向量知识库中搜索相关文档"""
    results = vector_store.similarity_search(query, k=5)
    return "\n".join([doc.page_content for doc in results])

@tool
def search_web(query: str) -> str:
    """当知识库中没有答案时，搜索互联网"""
    ...

# Agent 根据问题类型，自动选择合适的工具
model_with_tools = model.bind_tools([search_knowledge_base, search_web])
```

这就是从 Naive RAG 到 Agentic RAG 的关键一步——给 LLM 配上工具，让它自主决定何时检索、何时搜索、何时直接回答。

## 总结

### Tools 与函数调用的第一性原理

1. **最基础定义**：Tool = 函数 + 描述 + Schema，三者缺一不可
2. **核心问题**：LLM 只能生成文本，Tool 让它能"做事"
3. **三层价值**：能力扩展、标准化接口、安全可控
4. **调用协议**：LLM 输出 tool_calls → 外部执行 → ToolMessage 回传 → 继续推理
5. **设计推导**：简洁性与灵活性的平衡催生了三种创建方式

### 关键洞察

- **Tool 不是 LLM 的附属品，而是 Agent 系统的基石** —— 没有 Tool，Agent 就是空谈
- **Schema 不只是文档，更是安全边界** —— 它在执行前验证参数，防止错误和危险操作
- **标准化协议是关键** —— tool_calls + ToolMessage 的配对机制让任何函数都能被任何模型调用
- **从简单到复杂** —— @tool 覆盖 80% 场景，StructuredTool 和 BaseTool 处理剩下的 20%
