# 超时异常与取消传播

> **核心概念 2/6**
> **关键词**：cancel、TimeoutError、asyncio.TimeoutError、panic_or_proceed、异常优先级

---

## 为什么“超时后会怎样”比“超时值是多少”更重要？

因为很多系统不是死在“超时太短”，而是死在“超时之后留下了一地鸡毛”：

- 任务没取消干净；
- 状态半写入；
- 更高语义异常被吞掉；
- 调用方收到的错误和真实原因不一致。

LangGraph 在这块的关键点，是 `_panic_or_proceed()`。

---

## 核心定义

**超时异常与取消传播，指的是当前 step 超出 deadline 后，LangGraph 如何取消未完成任务、保留高优先级语义信号，并向上层抛出合适异常。**

---

## `_panic_or_proceed()` 做了什么？

核心逻辑可以概括成四步：

1. 把 future 分成 `done` 和 `inflight`；
2. 先检查 `done` 里有没有异常；
3. 如果异常是 `GraphInterrupt`，先收集起来；
4. 如果最终还有 `inflight`，就 cancel 并抛 timeout。

对应源码：

```python
for fut in futs:
    if fut.cancelled():
        continue
    elif fut.done():
        done.add(fut)
    else:
        inflight.add(fut)
```

```python
if exc := _exception(fut):
    while inflight:
        inflight.pop().cancel()
    if panic:
        if isinstance(exc, GraphInterrupt):
            interrupts.append(exc)
        elif fut not in SKIP_RERAISE_SET:
            raise exc
```

```python
if inflight:
    while inflight:
        inflight.pop().cancel()
    raise timeout_exc_cls("Timed out")
```

---

## 这段代码真正表达的顺序是什么？

### 顺序一：先尊重已经发生的高优先级语义

如果某个已完成 future 抛的是 `GraphInterrupt`，LangGraph 不会先急着说“超时了”，而是会收集 interrupt。

### 顺序二：真的还有没完成的任务，才算 timeout

只有在 `inflight` 仍然存在时，才会进入 timeout 分支。

### 顺序三：timeout 之前先取消

LangGraph 不会只抛异常，不管后台任务死活。它会先 `cancel()` 未完成任务。

这很关键，因为如果不 cancel，timeout 只是“调用方以为结束了”，后台实际还在跑。

---

## 同步和异步有什么差别？

### 同步
- 默认用 `TimeoutError`

### 异步
- 在 `runner.atick()` 中显式传入 `asyncio.TimeoutError`

源码：

```python
_panic_or_proceed(
    futures.done.union(...),
    timeout_exc_cls=asyncio.TimeoutError,
    panic=reraise,
)
```

所以：
- `invoke()` 常见的是 `TimeoutError`
- `ainvoke()` / `astream()` 常见的是 `asyncio.TimeoutError`

这两个异常类型不同，但语义上都表示“当前 step 超过预算”。

---

## 为什么“取消传播”很关键？

因为 timeout 不是纯逻辑问题，它还影响资源释放：

- 协程是否继续挂在 event loop 上；
- 工具调用是否继续占连接；
- 子图是否继续推进；
- 流式消费者是否还会收到后续 chunk。

异步测试里有个非常直观的标记：

```python
except asyncio.CancelledError:
    inner_task_cancelled = True
    raise
```

最后断言：

```python
assert inner_task_cancelled
```

说明 LangGraph 确实把 timeout 转化成了对内部任务的取消。

## 为什么 timeout 不应该吞掉控制流？

看似矛盾：

- 一方面 timeout 要强势中止；
- 另一方面某些语义信号又不能被中止逻辑覆盖。

这正是异常优先级的意义。

比如：
- `GraphInterrupt` 表示“流程应该暂停给外部处理”；
- `ParentCommand` 表示“控制权应该回到父图”；

这些都不是普通失败，更不是“因为慢所以失败”。

如果 timeout 直接盖掉它们，工作流语义就乱了。

---

## 一个直觉图

```text
step 执行结束后
  ↓
有没有 done 的 future 抛出更高优先级异常？
  ├─ 有 GraphInterrupt / 语义信号 → 先传播
  └─ 没有
      ↓
还有 inflight future 吗？
  ├─ 有 → cancel → raise timeout
  └─ 没有 → 正常继续
```

这张图能帮助你快速理解为什么某些复杂场景下不会直接报 timeout。

## 一个最小示例：取消感知

```python
import asyncio
from typing_extensions import TypedDict
from langgraph.graph import StateGraph


class State(TypedDict):
    value: int


async def slow_node(state: State):
    try:
        await asyncio.sleep(5)
        return {"value": state["value"] + 1}
    except asyncio.CancelledError:
        print("slow_node 被取消")
        raise


builder = StateGraph(State)
builder.add_node("slow", slow_node)
builder.set_entry_point("slow")

graph = builder.compile()
graph.step_timeout = 1

async def main():
    try:
        await graph.ainvoke({"value": 1})
    except asyncio.TimeoutError:
        print("graph timeout")

asyncio.run(main())
```

这个例子能帮助你验证：
- 图会抛 `asyncio.TimeoutError`；
- node 内会收到 `CancelledError`。

---

## 这对业务代码意味着什么？

### 1. 长操作应尽量可取消

如果 node 内部是纯阻塞、不可取消的外部库调用，那么 LangGraph 即便发起 cancel，也不一定能立刻让底层停下来。

### 2. 清理逻辑要写在 `CancelledError` 路径

例如：
- 关闭临时连接；
- 写部分诊断信息；
- 标记状态为 aborted / timed_out。

### 3. 不要把所有异常都吞成普通返回

如果你把 `CancelledError` / timeout 也无脑 `except Exception` 后吃掉，外层图会失去准确控制。

---

## 最容易忽视的一个问题

**超时异常不一定说明“最慢的那个任务坏了”。**

有时只是：
- 预算设置太紧；
- 流式消费者太慢；
- step 里并发组合不合理；
- 某个上游信号比 timeout 更该优先处理。

所以排查 timeout，别只盯着“哪个函数睡了最久”，还要看执行结构。

---

## 结论

**LangGraph 的 timeout 机制不是单纯抛异常，而是“先分类已完成任务的语义，再取消未完成任务，最后用正确异常向上冒泡”。**
