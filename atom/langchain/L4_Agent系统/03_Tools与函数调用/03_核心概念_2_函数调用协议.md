# 核心概念 2：函数调用协议

> `bind_tools()` 把工具"告诉"模型，模型返回 `tool_calls` 而非直接回答——这套协议是 LLM 从"只会说"到"能做事"的通信标准。

---

## 概述

**函数调用协议**解决的核心问题是：LLM 和 Tool 之间如何"对话"？

具体来说，它回答三个问题：
1. **怎么告诉模型有哪些工具可用？** → `bind_tools()`
2. **模型怎么表达"我要调用某个工具"？** → `AIMessage.tool_calls`
3. **不同模型提供商的格式不同怎么办？** → `convert_to_openai_tool()` 等格式转换函数

```
完整链路：

定义 Tool → bind_tools() → 模型推理 → AIMessage(tool_calls) → 执行 Tool → ToolMessage → 模型继续推理
              ↑                          ↑                                      ↑
          协议入口                    协议核心                              协议闭环
```

---

## 1. bind_tools() — 将工具绑定到模型

### 一句话定义

**`bind_tools()` 把 Tool 的 Schema 转换为模型能理解的格式，返回一个"带工具能力"的新模型。**

### 源码位置

`langchain_core/language_models/chat_models.py`，`BaseChatModel` 类的方法。

### 函数签名

```python
def bind_tools(
    self,
    tools: Sequence[Union[Dict[str, Any], Type, Callable, BaseTool]],
    *,
    tool_choice: Optional[Union[str, Dict]] = None,
    **kwargs: Any,
) -> Runnable:
    ...
```

### 参数详解

| 参数 | 类型 | 说明 |
|------|------|------|
| `tools` | Sequence | 工具列表，支持多种输入格式 |
| `tool_choice` | str / dict / None | 控制模型如何选择工具 |
| `**kwargs` | Any | 传递给底层模型的额外参数 |

### tools 参数支持的输入类型

`bind_tools()` 非常灵活，接受 4 种输入格式：

```python
from langchain_core.tools import tool
from pydantic import BaseModel
from langchain_openai import ChatOpenAI

model = ChatOpenAI(model="gpt-4o-mini")

# ===== 格式 1: BaseTool 实例（最常用，90% 场景用这个）=====
@tool
def search(query: str) -> str:
    """搜索互联网"""
    return f"搜索结果: {query}"

model.bind_tools([search])

# ===== 格式 2: Pydantic BaseModel 类 =====
class SearchInput(BaseModel):
    """搜索互联网的参数"""
    query: str

model.bind_tools([SearchInput])

# ===== 格式 3: Python 字典（OpenAI 原始格式）=====
model.bind_tools([{"type": "function", "function": {"name": "search", "description": "搜索互联网",
    "parameters": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}}}])

# ===== 格式 4: 普通函数（自动转换）=====
def search_func(query: str) -> str:
    """搜索互联网"""
    return f"搜索结果: {query}"

model.bind_tools([search_func])
```

### tool_choice 参数详解

`tool_choice` 控制模型在什么情况下调用工具：

| 值 | 含义 | 使用场景 |
|------|------|----------|
| `None`（默认） | 等同于 `"auto"` | 大多数场景 |
| `"auto"` | 模型自行决定是否调用工具 | 通用对话 + 工具混合 |
| `"any"` | 模型必须调用至少一个工具 | 确保工具被使用 |
| `"none"` | 模型不能调用任何工具 | 临时禁用工具 |
| `"search"` | 模型必须调用指定名称的工具 | 强制使用特定工具 |

```python
from langchain_openai import ChatOpenAI

model = ChatOpenAI(model="gpt-4o-mini")

# 自动决定（默认）
model.bind_tools([search], tool_choice="auto")

# 必须调用工具
model.bind_tools([search], tool_choice="any")

# 禁止调用工具
model.bind_tools([search], tool_choice="none")

# 强制调用指定工具
model.bind_tools([search, calculator], tool_choice="search")
```

**在 RAG 中的典型用法**：

```python
# 场景：强制 Agent 先检索再回答
retriever_model = model.bind_tools(
    [search_knowledge_base],
    tool_choice="search_knowledge_base"  # 强制检索
)

# 场景：让 Agent 自主决定是否需要检索
flexible_model = model.bind_tools(
    [search_knowledge_base, search_web],
    tool_choice="auto"  # 自动判断
)
```

### bind_tools() 的返回值

`bind_tools()` 返回一个新的 `Runnable` 对象，不会修改原模型：

```python
model = ChatOpenAI(model="gpt-4o-mini")

# bind_tools 返回新对象，原 model 不变
model_with_tools = model.bind_tools([search])

# model 仍然是普通模型（不带工具）
# model_with_tools 是带工具的模型
```

这符合 LangChain 的不可变设计原则——每次操作都返回新对象，不产生副作用。

## 2. 模型如何生成 tool_calls

### 核心机制

当模型绑定了工具后，它在推理时会做一个决策：

```
用户输入 → 模型思考 → 决策分支：
                        ├─ 需要工具 → 返回 AIMessage(tool_calls=[...])
                        └─ 不需要工具 → 返回 AIMessage(content="直接回答")
```

### tool_calls 的数据结构

```python
from langchain_core.messages import AIMessage

# 模型决定调用工具时，返回的 AIMessage
response = AIMessage(
    content="",  # 调用工具时，content 通常为空
    tool_calls=[
        {
            "name": "get_weather",        # 工具名称
            "args": {"city": "北京"},      # 参数字典（已解析的 JSON）
            "id": "call_abc123",          # 唯一调用 ID
            "type": "tool_call"           # 固定值
        }
    ]
)
```

**四个字段的作用**：

| 字段 | 作用 | 类比 |
|------|------|------|
| `name` | 指定调用哪个工具 | 快递单上的"收件人" |
| `args` | 传递给工具的参数 | 快递包裹里的"内容" |
| `id` | 唯一标识这次调用 | 快递单号 |
| `type` | 固定为 `"tool_call"` | 标记这是一个工具调用请求 |

### 模型可以一次调用多个工具

```python
# 用户问："北京和上海今天天气怎么样？"
# 模型可能返回两个 tool_calls（并行调用）
response = model_with_tools.invoke("北京和上海今天天气怎么样？")
print(len(response.tool_calls))  # 可能是 2

# response.tool_calls = [
#     {"name": "get_weather", "args": {"city": "北京"}, "id": "call_001", "type": "tool_call"},
#     {"name": "get_weather", "args": {"city": "上海"}, "id": "call_002", "type": "tool_call"},
# ]
```

### 判断模型是否调用了工具

```python
response = model_with_tools.invoke("你好，今天心情怎么样？")

if response.tool_calls:
    # 模型决定调用工具
    for tc in response.tool_calls:
        print(f"调用工具: {tc['name']}({tc['args']})")
else:
    # 模型直接回答
    print(f"直接回答: {response.content}")
```

## 3. Schema 格式转换

### 为什么需要格式转换？

不同的模型提供商对工具描述的 JSON 格式要求不同：

```
OpenAI:     {"type": "function", "function": {"name": ..., "parameters": ...}}
Anthropic:  {"name": ..., "description": ..., "input_schema": ...}
Bedrock:    {"toolSpec": {"name": ..., "inputSchema": {"json": ...}}}
```

LangChain 通过 `langchain_core/utils/function_calling.py` 中的转换函数，将统一的 Tool 定义转换为各提供商的格式。开发者只需定义一次 Tool，就能在所有模型上使用。

### 核心转换函数

#### convert_to_openai_tool()

**最常用的转换函数**，将任意 Tool 定义转换为 OpenAI 格式：

```python
from langchain_core.utils.function_calling import convert_to_openai_tool
from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """查询城市天气"""
    return f"{city}今天晴"

openai_tool = convert_to_openai_tool(get_weather)
# → {"type": "function", "function": {"name": "get_weather", "description": "查询城市天气",
#     "parameters": {"type": "object", "properties": {"city": {"type": "string"}}, "required": ["city"]}}}
```

#### convert_to_openai_function()

转换为 OpenAI 的旧版 function 格式（不含外层 `type: "function"` 包装）：

```python
from langchain_core.utils.function_calling import convert_to_openai_function

openai_func = convert_to_openai_function(get_weather)
# → {"name": "get_weather", "description": "...", "parameters": {...}}
```

#### 支持的输入类型

转换函数接受多种输入，内部自动识别：

```python
from pydantic import BaseModel
from langchain_core.utils.function_calling import convert_to_openai_tool

convert_to_openai_tool(get_weather)          # 1. BaseTool 实例

class WeatherInput(BaseModel):
    """查询城市天气"""
    city: str
convert_to_openai_tool(WeatherInput)         # 2. Pydantic BaseModel 类

convert_to_openai_tool({"type": "function",  # 3. Python 字典（直接透传）
    "function": {"name": "get_weather"}})

def my_func(x: int) -> str:
    """做某事"""
    return str(x)
convert_to_openai_tool(my_func)              # 4. 普通函数
```

### 不同提供商的格式对比

| 提供商 | 外层结构 | Schema 字段名 | 示例 |
|--------|----------|---------------|------|
| OpenAI | `{"type": "function", "function": {...}}` | `parameters` | 最常见格式 |
| Anthropic | `{"name": ..., "description": ..., ...}` | `input_schema` | 扁平结构 |
| Bedrock | `{"toolSpec": {"name": ..., ...}}` | `inputSchema.json` | 嵌套结构 |

**关键点**：开发者不需要关心这些格式差异。`bind_tools()` 内部会根据模型类型自动选择正确的转换函数。定义一次 Tool，就能在 OpenAI、Anthropic、Bedrock 等任何模型上使用。

### strict 模式

OpenAI 支持 `strict` 模式，保证模型输出严格匹配 JSON Schema：

```python
# 启用 strict 模式
model_with_tools = model.bind_tools(
    [get_weather],
    strict=True  # 模型输出必须严格匹配 Schema
)
```

启用 strict 模式后，`convert_to_openai_tool()` 会在 Schema 中添加 `"strict": true`，并确保所有属性都标记为 `required`、`additionalProperties` 设为 `false`。这在生产环境中非常有用，可以避免模型生成不符合预期的参数。

## 4. 完整的函数调用流程

### 六步流程图

```
步骤1: 定义 Tool
    @tool
    def get_weather(city: str) -> str: ...
        │
        ▼
步骤2: bind_tools() 绑定到模型
    model_with_tools = model.bind_tools([get_weather])
        │
        ▼
步骤3: 模型推理，生成 tool_calls
    response = model_with_tools.invoke("北京天气？")
    → AIMessage(tool_calls=[{name, args, id}])
        │
        ▼
步骤4: 执行 Tool
    result = get_weather.invoke({"city": "北京"})
    → "北京今天晴，25°C"
        │
        ▼
步骤5: 构造 ToolMessage 回传
    ToolMessage(content=result, tool_call_id=id)
        │
        ▼
步骤6: 模型基于结果生成最终回答
    → "北京今天天气晴朗，气温25°C。"
```

### 完整可运行代码

```python
"""
函数调用协议 - 完整流程演示
"""
import os
from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langchain_core.messages import HumanMessage, ToolMessage

load_dotenv()

# ===== 步骤 1: 定义 Tool =====
@tool
def get_weather(city: str) -> str:
    """查询指定城市的实时天气信息"""
    weather_data = {
        "北京": "晴，25°C，湿度 40%",
        "上海": "多云，28°C，湿度 65%",
    }
    return weather_data.get(city, f"{city}：暂无天气数据")

# ===== 步骤 2: bind_tools() 绑定到模型 =====
model = ChatOpenAI(model="gpt-4o-mini")
model_with_tools = model.bind_tools([get_weather])

# ===== 步骤 3: 模型推理，生成 tool_calls =====
messages = [HumanMessage(content="北京今天天气怎么样？")]
response = model_with_tools.invoke(messages)
print(f"tool_calls: {response.tool_calls}")

# ===== 步骤 4 & 5: 执行 Tool，构造 ToolMessage =====
messages.append(response)
for tc in response.tool_calls:
    result = get_weather.invoke(tc["args"])
    print(f"工具结果: {result}")
    messages.append(ToolMessage(content=result, tool_call_id=tc["id"]))

# ===== 步骤 6: 模型基于结果生成最终回答 =====
final_response = model_with_tools.invoke(messages)
print(f"最终回答: {final_response.content}")
```

**预期输出**：
```
tool_calls: [{'name': 'get_weather', 'args': {'city': '北京'}, 'id': 'call_abc123', 'type': 'tool_call'}]
工具结果: 晴，25°C，湿度 40%
最终回答: 北京今天天气晴朗，气温25°C，湿度40%，非常适合户外活动。
```

### 消息链路可视化

```
[HumanMessage("北京天气？")]
    → [+ AIMessage(tool_calls=[{get_weather, {city:"北京"}, call_001}])]
    → [+ ToolMessage("晴，25°C", tool_call_id="call_001")]
    → [+ AIMessage("北京今天天气晴朗，气温25°C...")]
```

这就是完整的 `HumanMessage → AIMessage(tool_calls) → ToolMessage → AIMessage(最终回答)` 消息链路。

## 5. 在 RAG / Agent 中的应用

### RAG 场景：检索工具 + 函数调用

```python
@tool
def search_knowledge_base(query: str) -> str:
    """在公司知识库中搜索相关文档，用于回答公司内部问题"""
    return "根据知识库：Q3 营收 5.2 亿元，同比增长 15%"

@tool
def search_web(query: str) -> str:
    """搜索互联网获取实时信息，用于回答知识库中没有的问题"""
    return "根据网络搜索：今日 A 股大盘上涨 1.2%"

# Agent 自动决定用哪个工具
model_with_tools = model.bind_tools(
    [search_knowledge_base, search_web],
    tool_choice="auto"  # 问公司问题→知识库，问实时新闻→网络，简单问题→直接回答
)
```

### Agent 场景：多工具协作

Agent 可以在一次对话中多步调用不同工具——先检索、再计算、最后总结：

```python
model_with_tools = model.bind_tools(
    [retriever, calculator, summarizer],
    tool_choice="auto"
)
```

---

## 总结

函数调用协议是 Tool 系统的"通信标准"：

1. **`bind_tools()`** 是入口——把工具 Schema 告诉模型，支持 4 种输入格式和 4 种 tool_choice 策略
2. **`tool_calls`** 是核心——模型用结构化 JSON 表达"我要调用什么工具、传什么参数"
3. **格式转换** 是桥梁——`convert_to_openai_tool()` 等函数屏蔽了不同提供商的格式差异
4. **完整流程** 是闭环——定义 → 绑定 → 调用 → 执行 → 回传 → 最终回答

掌握这套协议，你就能让任何 LLM 调用任何工具，构建真正能"做事"的 Agent。
