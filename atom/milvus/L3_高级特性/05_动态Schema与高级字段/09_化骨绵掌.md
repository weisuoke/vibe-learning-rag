# 09_化骨绵掌

## 卡片1：直觉理解——什么是动态Schema

**一句话：** 动态Schema让你无需预定义所有字段，就能往Collection里插入任意额外数据。

**举例：**

```python
# 启用动态Schema
schema = client.create_schema(
    auto_id=False,
    enable_dynamic_field=True  # 就这一行
)

# 插入时随意添加字段
data = [{"id": 1, "vector": [0.1]*768,
         "author": "Alice",      # 没预定义，但能存
         "tags": ["AI", "ML"]}]  # 没预定义，但能存
```

**应用：** RAG系统中不同文档类型有不同元数据（PDF有页码，网页有URL），动态Schema让它们共存于一个Collection。

---

## 卡片2：形式化定义——$meta字段的秘密

**一句话：** 所有动态字段都存储在一个隐式的 `$meta` JSON 字段中，本质是一个大JSON对象。

**举例：**

```
你插入的数据：
  id=1, vector=[...], author="Alice", tags=["AI"]

Milvus实际存储：
  id=1, vector=[...], $meta={"author":"Alice","tags":["AI"]}
```

查询动态字段时，Milvus 需要解析 `$meta` 的 JSON，所以**无法为动态字段创建索引**。

**应用：** 理解了 $meta 机制，就明白为什么高频过滤字段不应该放在动态字段里——每次查询都要做 JSON 解析。

---

## 卡片3：关键概念——动态字段数据操作

**一句话：** 动态字段的插入、查询、过滤和固定字段语法完全一样，但性能差距很大。

**举例：**

```python
# 插入：和固定字段一样
client.insert("docs", [{"id": 1, "vector": v, "author": "Alice"}])

# 过滤查询：语法一样
results = client.search(
    collection_name="docs",
    data=[query_vec],
    filter='author == "Alice"',       # 动态字段过滤
    output_fields=["author", "tags"]  # 返回动态字段
)

# 但注意：动态字段过滤 = 全表扫描 + JSON解析
# 固定字段过滤（有索引）= O(log n)
```

**应用：** 开发阶段用动态字段快速迭代，上线后把高频查询字段转为固定字段加索引。

## 卡片4：关键概念——Int8向量量化

**一句话：** Int8向量通过SQ8标量量化，将Float32的4字节/维度压缩为1字节/维度，节省75%内存。

**举例：**

```python
import numpy as np

# Float32: 每个维度 4 bytes
float32_vec = np.array([0.123, -0.456, 0.789], dtype=np.float32)
# 768维 = 768 × 4 = 3072 bytes

# Int8: 每个维度 1 byte（SQ8线性映射，不是简单截断）
# 768维 = 768 × 1 = 768 bytes（节省75%）

# ⚠️ 维度必须是8的倍数（SIMD优化要求）
# ✅ dim=128, 256, 768
# ❌ dim=100, 300
```

**应用：** Reddit上有RAG创业公司用RaBitQ+SQ8组合量化，成本直降75%。百万级向量场景下，Int8是标配。

## 卡片5：编程实现——动态Schema + Int8向量

**一句话：** 动态Schema和Int8向量可以组合使用，同时获得灵活性和内存优化。

**举例：**

```python
from pymilvus import MilvusClient, DataType

client = MilvusClient("http://localhost:19530")

# 创建Schema：动态字段 + Int8向量
schema = client.create_schema(auto_id=False, enable_dynamic_field=True)
schema.add_field("id", DataType.INT64, is_primary=True)
schema.add_field("vector", DataType.INT8_VECTOR, dim=128)  # Int8向量
schema.add_field("text", DataType.VARCHAR, max_length=1024)

# 插入数据（带动态字段）
data = [{"id": 1,
         "vector": [12, -46, 79, 33, -88, 56, -23, 91] * 16,  # 128维Int8
         "text": "RAG document",
         "source": "pdf",        # 动态字段
         "page_num": 42}]        # 动态字段
client.insert("docs", data)
```

**应用：** RAG知识库中，向量用Int8节省内存，文档元数据用动态字段灵活扩展，两全其美。

## 卡片6：对比区分——三种Schema扩展方式

**一句话：** 动态Schema、AddCollectionField、JSON字段是三种不同的扩展方式，各有优劣。

**举例：**

| 特性 | 动态Schema | AddCollectionField | JSON字段 |
|------|-----------|-------------------|---------|
| 灵活性 | ⭐⭐⭐ 最高 | ⭐⭐ 中等 | ⭐⭐⭐ 高 |
| 查询性能 | ⭐ 无索引 | ⭐⭐⭐ 可加索引 | ⭐⭐ JSON Path Index |
| 使用门槛 | 低 | 中（需停写入） | 低 |
| 嵌套结构 | ❌ 不支持 | ❌ 不支持 | ✅ 支持 |
| 适用场景 | 快速迭代 | 字段稳定后 | 复杂元数据 |

**应用：** 典型RAG系统演化路径：动态Schema（原型期）→ AddCollectionField（稳定期）→ JSON字段（复杂元数据）。

## 卡片7：进阶理解——AddCollectionField内部机制

**一句话：** AddCollectionField通过Schema版本号递增和消息广播实现字段热添加。

**举例：**

源码中的核心流程（`ddl_callbacks_alter_collection_add_field.go`）：

```
1. 检查Collection是否存在
2. 解析并验证字段Schema
3. 检查字段名是否重复
4. 分配新的 field ID
5. Schema版本号 +1（Version: coll.SchemaVersion + 1）
6. 通过 Streaming Node 广播 AlterCollection 消息
7. 所有节点更新本地Schema缓存
```

**关键限制：**
- 必须 `nullable=True`
- 只能添加标量字段（不能加向量字段）
- 已存在数据的新字段值为 NULL 或默认值

**应用：** 理解内部机制后，就明白为什么添加字段后需要 reload Collection——各节点需要同步新的Schema版本。

## 卡片8：高级应用——多租户元数据管理

**一句话：** 根据租户数量选择隔离策略：Collection隔离、Partition隔离、元数据过滤。

**举例：**

```python
# 小规模（<100租户）：Collection隔离
client.create_collection(f"tenant_{tenant_id}", dimension=768)

# 中规模（100-10K）：Partition隔离
client.create_partition("docs", f"tenant_{tenant_id}")
client.search("docs", data=[vec], partition_names=[f"tenant_{tid}"])

# 大规模（>10K）：元数据过滤
client.search("docs", data=[vec], filter=f"tenant_id == {tid}")
```

**应用：** SaaS型RAG平台通常从Partition隔离起步，随着租户增长迁移到元数据过滤。Milvus 2.6支持100K collections，Collection隔离的上限也提高了。

## 卡片9：RAG实战——生产级Schema管理

**一句话：** 生产环境的Schema管理有三大坑：group_by bug、nullable性能、并发冲突。

**举例：**

```
坑1：group_by + 动态字段
  → 缺失字段被视为有效分组（Issue #47438）
  → 解决：聚合字段必须用固定Schema

坑2：nullable字段聚合
  → 比非nullable慢4倍（Issue #47566）
  → 解决：高频聚合字段避免nullable

坑3：AddCollectionField + 并发insert
  → 报 schema mismatch 错误（Issue #41858）
  → 解决：添加字段前暂停写入，添加后reload Collection

坑4：AddCollectionField + HybridSearch
  → 添加字段后HybridSearch可能失败（Issue #43003）
  → 解决：添加字段后测试所有查询类型
```

**应用：** 生产环境添加字段的标准流程：暂停写入 → 添加字段 → reload → 测试所有查询 → 恢复写入。

## 卡片10：总结与延伸——Schema演化路线图

**一句话：** Schema设计是一个演化过程：从灵活到稳定，从动态到固定，从原型到生产。

**举例：**

```
阶段1：原型期
  → enable_dynamic_field=True
  → 所有元数据都是动态字段
  → 快速迭代，验证需求

阶段2：稳定期
  → 高频字段用 AddCollectionField 转正
  → 创建 INVERTED 索引
  → 性能优化

阶段3：优化期
  → Float32 → Int8 向量（节省75%内存）
  → 多租户隔离策略调整
  → JSON Path Index 优化复杂查询

未来：
  → 空间数据类型（GIS函数，源码中已有实现）
  → Float16/BFloat16 向量（平衡精度和内存）
  → EmbList 多向量场景
```

**应用：** 不要一开始就过度设计Schema。用动态Schema快速验证，用数据驱动Schema演化，这才是RAG系统的正确姿势。
