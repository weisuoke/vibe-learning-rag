## 卡片 1：条件边是什么？

**一句话：** 条件边让图在运行时根据状态选择不同路径，就像代码中的 if-else。

### 核心概念

普通边（`add_edge`）是固定路径：A 永远走到 B。条件边（`add_conditional_edges`）是动态路径：A 走到哪里，取决于运行时的状态。

```python
from langgraph.graph import StateGraph, START, END

# 固定边：永远走同一条路
builder.add_edge("input", "output")

# 条件边：运行时决定走哪条路
builder.add_conditional_edges("input", route_function)
```

### 最小示例

```python
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    has_tool_call: bool


def check_tool(state: State) -> Literal["use_tool", "__end__"]:
    if state["has_tool_call"]:
        return "use_tool"
    return "__end__"


builder = StateGraph(State)
# ... 添加节点 ...
builder.add_conditional_edges("agent", check_tool)
```

### RAG 应用

Agent 调用 LLM 后，判断输出中是否包含工具调用。有则路由到工具执行节点，无则直接结束——这就是最经典的条件边场景。

## 卡片 2：路由函数的写法

**一句话：** 路由函数接收状态，返回下一个节点的名称。

### 核心规则

路由函数的签名很简单：

```python
def route(state: State) -> str:
    # 根据状态判断
    # 返回节点名称
    ...
```

### 三种返回方式

```python
# 方式1：返回节点名称字符串
def route_v1(state: State) -> str:
    if state["score"] > 0.8:
        return "accept"
    return "reject"

# 方式2：返回 END 常量（结束图执行）
from langgraph.graph import END

def route_v2(state: State) -> str:
    if state["done"]:
        return END  # 等价于 "__end__"
    return "next_step"

# 方式3：用 Literal 注解（推荐，后面卡片详解）
def route_v3(state: State) -> Literal["accept", "reject"]:
    if state["score"] > 0.8:
        return "accept"
    return "reject"
```

### 关键约束

- 路由函数必须是**纯判断**，不应修改状态
- 返回值必须是已注册的节点名称或 `"__end__"`
- 函数应该快速执行，不要在里面调用 LLM

### RAG 应用

根据 LLM 输出决定下一步：输出包含 `tool_calls` 就路由到工具节点，否则路由到生成回答节点。路由函数只做判断，不做处理。

## 卡片 3：path_map 路径映射

**一句话：** path_map 将路由函数的返回值映射到实际节点名，实现解耦。

### 为什么需要 path_map？

路由函数返回的字符串不一定要和节点名完全一致。path_map 充当翻译层：

```python
# 不用 path_map：路由返回值 = 节点名（耦合）
def route(state):
    if state["approved"]:
        return "send_email_node"  # 必须和节点名一模一样
    return "manual_review_node"

# 用 path_map：路由返回值 ≠ 节点名（解耦）
def route(state):
    if state["approved"]:
        return "yes"  # 语义清晰的返回值
    return "no"

builder.add_conditional_edges(
    "checker",
    route,
    {"yes": "send_email_node", "no": "manual_review_node"}  # path_map
)
```

### 解耦的好处

```python
# 场景：重命名节点
# 不用 path_map → 改节点名 + 改路由函数
# 用 path_map → 只改 path_map 字典

# 重构前
{"yes": "send_email_node", "no": "manual_review_node"}

# 重构后（只改这里，路由函数不动）
{"yes": "notification_service", "no": "human_in_loop"}
```

### RAG 应用

路由逻辑返回 `"relevant"` / `"irrelevant"`，path_map 映射到实际的 `"generate_answer"` / `"web_search"` 节点。当你重构节点命名时，路由逻辑完全不用改。

## 卡片 4：Literal 类型推断

**一句话：** 用 Literal 注解返回类型，LangGraph 自动推断所有可能路径。

### 核心机制

```python
from typing import Literal

# ✅ 推荐：Literal 注解
def route(state: State) -> Literal["search", "chat", "__end__"]:
    if state["need_search"]:
        return "search"
    if state["is_chat"]:
        return "chat"
    return "__end__"

# LangGraph 读取 Literal 注解，自动知道有 3 条可能的边
# 不需要手动传 path_map
builder.add_conditional_edges("router", route)
```

### 对比：没有 Literal 的情况

```python
# ❌ 没有 Literal：LangGraph 不知道有哪些路径
def route(state: State) -> str:
    ...

# 必须手动传 path_map 告诉它所有可能的路径
builder.add_conditional_edges("router", route, {
    "search": "search",
    "chat": "chat",
    "__end__": "__end__"
})
```

### Literal vs path_map 选择

| 场景 | 推荐方式 |
|------|----------|
| 返回值 = 节点名 | Literal（最简洁） |
| 返回值 ≠ 节点名 | path_map（需要映射） |
| 需要图可视化 | Literal（自动推断边） |

### RAG 应用

图可视化（`graph.get_graph().draw_mermaid()`）依赖类型推断来画边。用 Literal 注解后，可视化图会自动显示所有分支路径，方便调试和文档化。

## 卡片 5：Send 动态路由

**一句话：** Send 向指定节点发送自定义状态，实现动态并行。

### 条件边 vs Send

条件边选择**一条**路径走。Send 可以同时向**多个**节点发送任务，而且每个任务携带不同的状态。

```python
from langgraph.types import Send

# 条件边：选一条路走
def route(state) -> Literal["a", "b"]:
    return "a"  # 只能走 a 或 b

# Send：同时向多个节点发送任务
def fan_out(state):
    return [
        Send("worker", {"task": t})
        for t in state["tasks"]
    ]

builder.add_conditional_edges("dispatcher", fan_out)
```

### Map-Reduce 模式

```python
class State(TypedDict):
    topics: list[str]
    results: Annotated[list[str], operator.add]  # 自动合并


def dispatch(state: State):
    """Map 阶段：为每个主题创建一个并行任务"""
    return [Send("research", {"topic": t}) for t in state["topics"]]


def research(state) -> dict:
    """每个 worker 独立处理一个主题"""
    return {"results": [f"研究结果: {state['topic']}"]}


builder.add_conditional_edges("start", dispatch)
# research 节点会并行执行 N 次，N = len(topics)
```

### 关键特性

- Send 的数量在运行时动态决定
- 每个 Send 携带独立的状态片段
- 所有 Send 的结果通过 reducer 自动合并

### RAG 应用

对 N 篇检索到的文档并行做摘要：`[Send("summarize", {"doc": d}) for d in docs]`。文档数量在检索后才知道，Send 完美处理这种动态并行。

## 卡片 6：Command 高级路由

**一句话：** Command 让节点同时控制路由和状态更新，无需条件边。

### 传统方式 vs Command

```python
# 传统方式：节点更新状态 + 条件边做路由（两步）
def my_node(state):
    return {"result": "done"}  # 只能更新状态

def route(state):
    return "next"  # 单独的路由逻辑

builder.add_conditional_edges("my_node", route)

# Command 方式：节点同时做路由 + 状态更新（一步）
from langgraph.types import Command

def my_node(state) -> Command[Literal["next", "retry"]]:
    if state["ok"]:
        return Command(goto="next", update={"result": "done"})
    return Command(goto="retry", update={"retries": state["retries"] + 1})

# 不需要 add_conditional_edges，直接 add_edge 或什么都不加
```

### Command 的三个能力

```python
# 1. goto：控制下一个节点
Command(goto="target_node")

# 2. update：更新状态
Command(goto="next", update={"key": "value"})

# 3. goto + update：同时做两件事
Command(goto="retry", update={"attempt": state["attempt"] + 1})
```

### 什么时候用 Command？

| 场景 | 用条件边 | 用 Command |
|------|---------|-----------|
| 路由逻辑独立于节点 | ✅ | |
| 路由和状态更新紧耦合 | | ✅ |
| 需要在节点内部决定去向 | | ✅ |
| 多个节点共享路由逻辑 | ✅ | |

### RAG 应用

检索节点判断结果质量：质量达标则 `Command(goto="generate", update={"docs": results})`，不达标则 `Command(goto="rewrite_query", update={"retries": n+1})`。决策和状态更新在同一个地方，逻辑更内聚。

## 卡片 7：循环守卫

**一句话：** 用计数器防止条件边导致的无限循环。

### 问题：条件边 + 循环 = 潜在死循环

```python
# 危险：如果 LLM 一直返回工具调用，就会无限循环
def should_continue(state) -> Literal["tools", "__end__"]:
    if state["messages"][-1].tool_calls:
        return "tools"  # 又回到 agent → 又调工具 → 又回来...
    return "__end__"
```

### 解决方案：计数器守卫

```python
MAX_STEPS = 10

def should_continue(state) -> Literal["tools", "__end__"]:
    # 守卫条件：超过最大步数强制停止
    if state.get("steps", 0) >= MAX_STEPS:
        return "__end__"

    if state["messages"][-1].tool_calls:
        return "tools"
    return "__end__"


def agent_node(state):
    result = llm.invoke(state["messages"])
    return {
        "messages": [result],
        "steps": state.get("steps", 0) + 1  # 递增计数器
    }
```

### 三层防护体系

```python
# 第1层：业务逻辑守卫（推荐）
if state["steps"] >= MAX_STEPS:
    return "__end__"

# 第2层：LangGraph 内置 recursion_limit
app = graph.compile()
result = app.invoke(input, {"recursion_limit": 25})

# 第3层：超时保护
import asyncio
result = await asyncio.wait_for(
    app.ainvoke(input),
    timeout=60.0
)
```

### RAG 应用

Agent 调用搜索工具后发现结果不够好，重写 query 再搜索——这个循环必须有上限。设置 `max_retries=3`，超过就用当前最佳结果生成回答，避免无限重试。

## 卡片 8：错误路由与回退

**一句话：** 在节点中捕获异常，通过条件边路由到错误处理或降级节点。

### 基本模式：try-except + 状态标记

```python
def call_llm(state: State) -> dict:
    try:
        response = llm.invoke(state["messages"])
        return {"response": response, "error": None}
    except Exception as e:
        return {"response": None, "error": str(e)}


def route_after_llm(state) -> Literal["process", "fallback", "__end__"]:
    if state["error"] is None:
        return "process"
    if state.get("retries", 0) < 3:
        return "fallback"
    return "__end__"
```

### 三级回退策略

```python
def fallback_node(state: State) -> dict:
    retries = state.get("retries", 0)

    if retries == 0:
        # 第1级：重试（可能是临时网络问题）
        return {"retries": 1, "error": None}

    elif retries == 1:
        # 第2级：降级（换一个更便宜/更稳定的模型）
        backup_response = backup_llm.invoke(state["messages"])
        return {"response": backup_response, "retries": 2, "error": None}

    else:
        # 第3级：人工介入
        return {
            "response": "抱歉，系统暂时无法处理，已转人工。",
            "retries": retries + 1,
            "needs_human": True
        }
```

### 错误路由的设计原则

1. **节点负责捕获异常**，不要让异常直接抛出到图引擎
2. **用状态字段传递错误信息**（`error`、`retries`）
3. **路由函数根据错误状态决定走向**
4. **总有一个兜底路径**，确保图不会卡死

### RAG 应用

检索服务超时 → 重试一次 → 还是超时 → 降级到关键词搜索 → 关键词搜索也失败 → 返回"暂时无法检索，请稍后重试"。每一级回退都通过条件边路由。

## 卡片 9：Supervisor 路由模式

**一句话：** 中央调度节点分类请求，路由到专家代理。

### 模式结构

```
用户请求 → Supervisor → 专家A（搜索）
                      → 专家B（计算）
                      → 专家C（写作）
                      → 结束
```

### 实现代码

```python
from typing import Literal

SPECIALISTS = {
    "search": "search_agent",
    "calculate": "calc_agent",
    "write": "write_agent",
}


def supervisor(state: State) -> dict:
    """中央调度：用 LLM 判断请求类型"""
    classification = llm.invoke(
        f"将以下请求分类为 search/calculate/write/done：\n{state['query']}"
    )
    return {"next_action": classification.content.strip()}


def route_to_specialist(state) -> Literal[
    "search_agent", "calc_agent", "write_agent", "__end__"
]:
    action = state["next_action"]
    if action == "done":
        return "__end__"
    return SPECIALISTS.get(action, "__end__")


builder.add_conditional_edges("supervisor", route_to_specialist)

# 每个专家完成后回到 supervisor，形成循环
for agent_name in SPECIALISTS.values():
    builder.add_edge(agent_name, "supervisor")
```

### Supervisor 的关键设计

```python
# 1. 分类提示词要明确
CLASSIFY_PROMPT = """你是一个请求分类器。
可选类别：search, calculate, write, done
只返回类别名称，不要解释。"""

# 2. 循环守卫不能少
def supervisor(state):
    if state.get("rounds", 0) >= 5:
        return {"next_action": "done"}
    # ... 正常分类逻辑 ...
    return {"next_action": action, "rounds": state.get("rounds", 0) + 1}

# 3. 结果聚合
class State(TypedDict):
    query: str
    next_action: str
    rounds: int
    results: Annotated[list[str], operator.add]  # 每个专家的结果自动合并
```

### RAG 应用

多 Agent 协作的 RAG 系统：Supervisor 判断用户问题需要"检索知识库"、"查询数据库"还是"直接回答"，路由到对应的专家 Agent。每个专家完成后回到 Supervisor 决定是否需要进一步处理。

## 卡片 10：选择策略总结

**一句话：** 静态分支用条件边，动态并行用 Send，运行时决策用 Command。

### 三种机制对比

| 特性 | add_conditional_edges | Send | Command |
|------|----------------------|------|---------|
| 路径数量 | 编译时确定 | 运行时动态 | 运行时动态 |
| 并行能力 | 单选一条路 | 多路并行 | 单选一条路 |
| 状态控制 | 路由函数不改状态 | 每个 Send 携带独立状态 | 同时路由 + 更新状态 |
| 定义位置 | 图构建时 | 路由函数内 | 节点内部 |
| 使用频率 | ~80% | ~15% | ~5% |

### 决策流程图

```
需要分支路由？
├── 路径在编译时已知？
│   ├── 是 → add_conditional_edges + Literal
│   └── 否 → 需要并行？
│       ├── 是 → Send
│       └── 否 → Command
├── 路由和状态更新紧耦合？
│   └── 是 → Command
└── 需要 Map-Reduce？
    └── 是 → Send
```

### 组合使用示例

```python
# 一个图中可以混合使用三种机制

# 1. 入口路由：条件边（路径已知）
builder.add_conditional_edges("classifier", route_by_intent)

# 2. 并行处理：Send（动态数量）
builder.add_conditional_edges("dispatcher", fan_out_to_workers)

# 3. 质量检查：Command（路由 + 状态更新）
def quality_check(state) -> Command[Literal["output", "retry"]]:
    if state["score"] > 0.8:
        return Command(goto="output", update={"final": True})
    return Command(goto="retry", update={"attempt": state["attempt"] + 1})
```

### 记忆口诀

```
条件边管分叉，Literal 标路径；
Send 管并行，Map-Reduce 是拿手；
Command 管决策，路由更新一把抓；
循环加守卫，回退要三级；
八成条件边，一成五 Send，剩下 Command 填。
```

### RAG 应用中的典型组合

```
用户提问
  ↓ [条件边] 意图分类
  ├── 知识问答 → [Send] 并行检索多个知识库
  │                ↓ [条件边] 质量检查
  │                ├── 达标 → 生成回答
  │                └── 不达标 → [Command] 重写 query + 重试
  ├── 闲聊 → 直接回答
  └── 复杂任务 → [Supervisor] 多 Agent 协作
```
