# 核心概念3：错误处理策略

> LLM 应用不是"调通就完事"——网络抖动、API 限流、模型输出格式错误，这些在生产环境中是家常便饭。本文基于 `langchain_core.runnables.fallbacks` (RunnableWithFallbacks) 和 `langchain_core.runnables.retry` (RunnableRetry) 的源码分析，讲清楚 LangChain 如何让你的管道在异常面前"优雅降级"而不是"直接崩溃"。

---

## 一句话定义

**错误处理策略是通过 Fallback 降级、Retry 重试和异常分类，让 LangChain 管道在面对 LLM 调用失败、网络超时等异常时仍能优雅运行的机制。**

把它想成飞机的冗余系统——主引擎熄火了，备用引擎自动接管；一次点火失败，自动重试几次。你的 LLM 管道也需要同样的"生存能力"。

---

## 为什么错误处理如此重要？

LLM 应用面临的异常远比传统 Web 应用多：

| 异常场景 | 频率 | 影响 |
|----------|------|------|
| **API 限流** (429 Too Many Requests) | 高频，尤其是高并发场景 | 请求被拒绝，用户无响应 |
| **网络超时** (连接超时、读取超时) | 中频，依赖网络质量 | 长时间等待后失败 |
| **服务端错误** (500 Internal Server Error) | 低频，但不可预测 | 服务完全不可用 |
| **模型输出异常** (格式错误、幻觉、拒绝回答) | 中频，模型行为不确定 | 下游解析失败 |
| **Token 超限** (超过 context window) | 中频，长文档场景常见 | 请求被拒绝 |
| **认证失败** (API Key 过期或无效) | 低频，配置问题 | 所有请求失败 |

**核心矛盾**：LLM 调用本质上是**不确定的远程服务调用**——你无法保证每次都成功，但你可以保证失败时有预案。

---

## 策略1：Fallback 降级（RunnableWithFallbacks）

### 原理

Fallback 的核心思想很简单：**主方案失败了，自动切换到备选方案**。

RunnableWithFallbacks 在主 Runnable 抛出异常时，按顺序尝试 fallback 列表中的备选方案，直到某个成功或全部失败。

```
主方案 ──失败──→ 备选1 ──失败──→ 备选2 ──失败──→ 抛出最后的异常
  │                │                │
  成功              成功              成功
  ↓                ↓                ↓
  返回结果          返回结果          返回结果
```

### 源码洞察（fallbacks.py）

从 `fallbacks.py` line 36 的 `RunnableWithFallbacks` 类：

```python
class RunnableWithFallbacks(RunnableSerializable):
    """带降级方案的 Runnable 包装器"""
    runnable: Runnable                          # 主 Runnable
    fallbacks: Sequence[Runnable]               # 备选方案列表（按优先级排序）
    exceptions_to_handle: tuple[type[BaseException], ...] = (Exception,)  # 处理哪些异常
    exception_key: Optional[str] = None         # 是否将异常传递给 fallback
```

**四个关键设计：**

1. **`exceptions_to_handle`**：默认捕获所有 `Exception`，但你可以精确指定只处理 `RateLimitError` 等特定异常，其他异常直接抛出
2. **`exception_key`**：设置后，异常对象会作为字典的一个键传递给 fallback，让 fallback 知道"为什么主方案失败了"
3. **`__getattr__` 代理**：当你对 fallback 对象调用 `bind_tools()` 等方法时，会自动对主 Runnable 和所有 fallback 都调用相同方法——不需要手动逐个配置
4. **batch 优化**：批量调用时，只对失败的输入尝试 fallback，已成功的不会重复执行

### 代码示例

#### 模型级 Fallback（最常用）

```python
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic

# GPT-4o 失败时，依次尝试 Claude → GPT-4o-mini
model = ChatOpenAI(model="gpt-4o").with_fallbacks(
    [
        ChatAnthropic(model="claude-sonnet-4-5-20250929"),  # 备选1：换供应商
        ChatOpenAI(model="gpt-4o-mini"),                    # 备选2：降级到小模型
    ]
)

# 使用方式和普通 model 完全一样
result = model.invoke("你好")
```

这里的降级逻辑：GPT-4o 可用就用（最佳质量）→ GPT-4o 挂了用 Claude（跨供应商容灾）→ Claude 也挂了用 GPT-4o-mini（至少能响应）。

#### 链级 Fallback

```python
from langchain_core.runnables import RunnableLambda
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template("回答问题：{question}")
model = ChatOpenAI(model="gpt-4o")
parser = StrOutputParser()

# 主链
primary_chain = prompt | model | parser

# 兜底方案：返回固定响应
def hardcoded_response(inputs):
    return "抱歉，服务暂时不可用，请稍后重试。"

# 整条链失败时，返回友好提示
chain = primary_chain.with_fallbacks(
    [RunnableLambda(hardcoded_response)]
)
```

#### 带异常传递的 Fallback

通过 `exception_key` 参数，可以把异常对象传递给 fallback，让它知道"为什么主方案失败了"：

```python
chain = primary_chain.with_fallbacks(
    [RunnableLambda(smart_fallback)],
    exception_key="error",  # 异常对象注入到输入字典
)
# fallback 收到: {"question": "原始问题", "error": <异常对象>}
```

> **注意**：使用 `exception_key` 时，主链的输入必须是 dict 类型。

#### 指定异常类型

```python
from openai import RateLimitError, APITimeoutError

# 只在限流和超时时降级，其他异常（如认证错误）直接抛出
model = ChatOpenAI(model="gpt-4o").with_fallbacks(
    [ChatAnthropic(model="claude-sonnet-4-5-20250929")],
    exceptions_to_handle=(RateLimitError, APITimeoutError),
)
```

**为什么要指定异常类型？** `AuthenticationError` 降级也没用（Key 错了），`InvalidRequestError` 换模型也解决不了。只有 `RateLimitError` 这类"换个供应商就好"的异常才适合降级。

> **__getattr__ 代理**：对 fallback 对象调用 `bind_tools()` 等方法时，会自动对所有 fallback 都调用——不需要手动逐个配置。

## 策略2：Retry 重试（RunnableRetry）

### 原理

Retry 解决的是**瞬态故障**——那些"再试一次可能就好了"的问题。

RunnableRetry 基于 Python 的 `tenacity` 库实现，支持指数退避 + 随机抖动，避免"重试风暴"。

```
第1次调用 ──失败──→ 等待 1s ──→ 第2次调用 ──失败──→ 等待 2s ──→ 第3次调用 ──成功──→ 返回结果
                                                                              ──失败──→ 抛出异常
```

### 源码洞察（retry.py）

从 `retry.py` line 48 的 `RunnableRetry` 类：

```python
class RunnableRetry(RunnableBindingLike):
    """带重试能力的 Runnable 包装器"""
    retry_exception_types: tuple[type[BaseException], ...] = (Exception,)
    wait_exponential_jitter: bool = True       # 指数退避 + 随机抖动
    max_attempt_number: int = 3                # 最大重试次数
    # 底层使用 tenacity.Retrying / tenacity.AsyncRetrying
```

**五个关键设计：**

1. **基于 tenacity**：不是自己造轮子，而是用 Python 生态中最成熟的重试库
2. **默认策略合理**：3 次重试 + 指数退避 + 随机抖动，开箱即用
3. **重试范围要小**：源码注释明确建议"只重试可能失败的组件，不要重试整个链"
4. **stream/transform 不支持重试**：因为"重试一个流"的语义不明确——已经发出去的 chunk 怎么办？
5. **batch 智能重试**：批量调用时，只重试失败的元素，已成功的不会重复执行

### 代码示例

#### 基础重试

```python
from langchain_openai import ChatOpenAI

# 默认：3 次重试，指数退避 + 随机抖动
model_with_retry = ChatOpenAI(model="gpt-4o").with_retry()

# 等价于
model_with_retry = ChatOpenAI(model="gpt-4o").with_retry(
    stop_after_attempt=3,
    wait_exponential_jitter=True,
)
```

#### 自定义重试参数

```python
from openai import RateLimitError, APITimeoutError

model_with_retry = ChatOpenAI(model="gpt-4o").with_retry(
    retry_if_exception_type=(RateLimitError, APITimeoutError),  # 只重试这两种异常
    stop_after_attempt=5,                                        # 最多重试 5 次
    wait_exponential_jitter=True,                                # 指数退避 + 抖动
)
```

**指数退避 + 随机抖动**：每次失败后等待时间翻倍（1s → 2s → 4s → 8s），加上随机偏移避免多个客户端同时重试造成"重试风暴"。

#### 重试的最佳实践：粒度要小

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI

prompt = ChatPromptTemplate.from_template("回答：{question}")
model = ChatOpenAI(model="gpt-4o")
parser = StrOutputParser()

# ✅ 正确：只在 model 上重试（它是最可能失败的组件）
chain = prompt | model.with_retry() | parser

# ❌ 错误：重试整个链（prompt 和 parser 不会失败，重试它们没意义）
chain = (prompt | model | parser).with_retry()
```

**为什么粒度要小？** `prompt.invoke()` 和 `parser.invoke()` 是纯本地操作，不会因网络问题失败。只有 `model.invoke()` 涉及远程 API 调用，才需要重试。重试整个链还会浪费时间重新执行 prompt 格式化。

## 策略3：Fallback + Retry 组合

### 推荐模式：先重试，重试失败后降级

这是生产环境中最常用的组合：

```python
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic

# 构建带完整容错的模型
model = (
    ChatOpenAI(model="gpt-4o")
    .with_retry(stop_after_attempt=3)           # 先重试 3 次
    .with_fallbacks([
        ChatAnthropic(model="claude-sonnet-4-5-20250929")
            .with_retry(stop_after_attempt=2),  # 备选也重试 2 次
        ChatOpenAI(model="gpt-4o-mini"),        # 最后的兜底，不重试
    ])
)
```

**执行流程：**

```
GPT-4o 第1次 ──失败──→ GPT-4o 第2次 ──失败──→ GPT-4o 第3次 ──失败
                                                                ↓
Claude 第1次 ──失败──→ Claude 第2次 ──失败
                                       ↓
GPT-4o-mini 第1次 ──失败──→ 抛出异常
                   ──成功──→ 返回结果
```

### 为什么这个顺序？

| 层级 | 策略 | 原因 |
|------|------|------|
| 第一层 | GPT-4o + 重试3次 | 主力模型，瞬态故障重试即可恢复 |
| 第二层 | Claude + 重试2次 | 跨供应商容灾，OpenAI 整体故障时切换 |
| 第三层 | GPT-4o-mini 不重试 | 最后的兜底，能响应就行，不追求质量 |

### 反模式：不要在 fallback 外层再加 retry

```python
# ❌ 外层 retry 会导致整个 fallback 链被重复执行
model = (
    ChatOpenAI(model="gpt-4o")
    .with_fallbacks([ChatAnthropic(model="claude-sonnet-4-5-20250929")])
    .with_retry(stop_after_attempt=3)
)

# ✅ 每个模型独立控制重试
model = (
    ChatOpenAI(model="gpt-4o").with_retry(stop_after_attempt=3)
    .with_fallbacks([
        ChatAnthropic(model="claude-sonnet-4-5-20250929").with_retry(stop_after_attempt=2),
    ])
)
```

## 策略4：输出验证与自动修复

模型输出格式不对也是一种"错误"，需要处理。

### 解析失败时重试

```python
from langchain_core.output_parsers import PydanticOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from pydantic import BaseModel

class Answer(BaseModel):
    answer: str
    confidence: float
    sources: list[str]

parser = PydanticOutputParser(pydantic_object=Answer)

prompt = ChatPromptTemplate.from_template(
    "回答问题并按格式输出。\n{format_instructions}\n问题：{question}"
).partial(format_instructions=parser.get_format_instructions())

model = ChatOpenAI(model="gpt-4o")

# 解析失败时重试整个 model + parser（因为需要模型重新生成）
chain = prompt | (model | parser).with_retry(
    retry_if_exception_type=(Exception,),
    stop_after_attempt=2,
)
```

### 解析失败时降级到原始文本

```python
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableLambda

# 主链：尝试结构化解析
structured_chain = prompt | model | parser

# 降级链：返回原始文本
fallback_chain = prompt | model | StrOutputParser()

# 组合：解析失败就返回原始文本
chain = structured_chain.with_fallbacks([fallback_chain])
```

## 异常分类指南

不同异常需要不同的处理策略，盲目重试或降级反而会让问题更严重：

| 异常类型 | 处理策略 | 原因 |
|----------|----------|------|
| `RateLimitError` (429) | Retry + 指数退避 | 临时限流，等一等就好了 |
| `APITimeoutError` | Retry (2-3次) | 网络波动，重试大概率成功 |
| `APIConnectionError` | Retry + Fallback | 可能是持续性网络问题 |
| `AuthenticationError` (401) | **直接抛出** | 配置错误，重试 100 次也没用 |
| `InvalidRequestError` (400) | **直接抛出** | 输入有问题，需要修复代码 |
| `InternalServerError` (500) | Retry + Fallback | 服务端问题，可能持续一段时间 |
| `OutputParserException` | Retry 或 Fallback | 模型输出格式不对，重新生成可能修复 |
| `ContextWindowExceeded` | **直接抛出** | 输入太长，需要分块处理，重试无意义 |

**核心原则：只重试"再试一次可能成功"的异常。**

### 代码实现

```python
from openai import RateLimitError, APITimeoutError, APIConnectionError

RETRYABLE = (RateLimitError, APITimeoutError)
FALLBACKABLE = (RateLimitError, APITimeoutError, APIConnectionError)

model = (
    ChatOpenAI(model="gpt-4o")
    .with_retry(
        retry_if_exception_type=RETRYABLE,
        stop_after_attempt=3,
        wait_exponential_jitter=True,
    )
    .with_fallbacks(
        [ChatOpenAI(model="gpt-4o-mini").with_retry(stop_after_attempt=2)],
        exceptions_to_handle=FALLBACKABLE,
    )
)
# AuthenticationError、BadRequestError 会直接抛出，不浪费时间
```

## 在 RAG 开发中的应用

RAG 系统有多个可能失败的环节，每个环节需要不同的错误处理策略。

### 完整 RAG 错误处理示例

```python
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableLambda

# 1. Embedding 层：OpenAIEmbeddings 内部已有重试机制
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 2. 检索层：fallback 到关键词搜索
def keyword_search_fallback(question: str) -> str:
    return "（关键词检索结果）相关文档片段..."

retriever_with_fallback = retriever.with_fallbacks(
    [RunnableLambda(keyword_search_fallback)]
)

# 3. LLM 层：重试 + 模型降级
llm = (
    ChatOpenAI(model="gpt-4o")
    .with_retry(stop_after_attempt=3)
    .with_fallbacks([
        ChatOpenAI(model="gpt-4o-mini").with_retry(stop_after_attempt=2),
    ])
)

# 4. 组装完整 RAG 链
prompt = ChatPromptTemplate.from_template(
    "根据以下上下文回答问题。\n上下文：{context}\n问题：{question}"
)

rag_chain = (
    {
        "context": retriever_with_fallback,
        "question": RunnablePassthrough(),
    }
    | prompt
    | llm
    | StrOutputParser()
)
```

### RAG 错误处理层次总结

| 层级 | 组件 | 推荐策略 | 原因 |
|------|------|----------|------|
| Embedding | OpenAIEmbeddings | Retry (内置) | 网络问题，重试即可 |
| 检索 | VectorStore Retriever | Fallback → 关键词搜索 | 向量库挂了还能用关键词兜底 |
| LLM 生成 | ChatOpenAI | Retry + Fallback | 最关键也最不稳定的环节 |
| 输出解析 | OutputParser | Retry + Fallback → 原始文本 | 格式错误可以重试，实在不行返回原文 |

## 常见陷阱

### 陷阱1：重试所有异常

```python
# ❌ 危险：认证错误也会被重试 3 次，浪费时间
model = ChatOpenAI(model="gpt-4o").with_retry(stop_after_attempt=3)

# ✅ 安全：只重试值得重试的异常
model = ChatOpenAI(model="gpt-4o").with_retry(
    retry_if_exception_type=(RateLimitError, APITimeoutError),
    stop_after_attempt=3,
)
```

### 陷阱2：stream 场景下的重试

```python
# ⚠️ RunnableRetry 不支持 stream/transform 重试
model = ChatOpenAI(model="gpt-4o").with_retry(stop_after_attempt=3)
for chunk in model.stream("你好"):  # 重试不会生效
    print(chunk.content, end="")

# 需要流式 + 容错，用 fallback 代替（fallback 在 stream 模式下正常工作）
model = ChatOpenAI(model="gpt-4o").with_fallbacks(
    [ChatOpenAI(model="gpt-4o-mini")]
)
```

## 决策速查

当你不确定该用什么策略时：

- **Retry** = "再试一次可能就好了"（限流、超时）
- **Fallback** = "这条路走不通，换条路"（服务宕机、跨供应商容灾）
- **直接抛出** = "重试和换路都解决不了"（认证错误、参数错误）

---

## 小结

错误处理不是"锦上添花"，而是 LLM 应用从 demo 走向生产的**必经之路**。LangChain 通过 `with_retry()` 和 `with_fallbacks()` 两个方法，让你用最少的代码实现完整的容错策略。

记住三个原则：
1. **重试粒度要小**——只重试可能失败的组件
2. **异常要分类**——不是所有异常都值得重试
3. **先重试后降级**——给主方案足够的机会，实在不行再切换
