# 子图（Subgraph）与模块化 - 面试必问

> 精选 2 道 LangGraph 子图模块化的核心面试题，覆盖集成方式选型与多代理 Handoff 设计

---

## 问题1："LangGraph 中子图（Subgraph）有哪些集成方式？各自的适用场景是什么？"

### 普通回答（不出彩）

"子图有两种集成方式：共享状态键和状态转换。共享键就是父子图用同名字段自动传递，状态转换就是用函数手动映射。"

### 出彩回答（推荐）

> **子图集成有三个层次，选择取决于父子图之间的耦合程度：**
>
> **1. 共享状态键（自动传递）**
> 父子图 Schema 中同名键自动双向传递。适用于父子图由同一团队维护、Schema 设计一致的场景。优点是零代码映射；缺点是父子图 Schema 耦合，改名就会破坏传递。
>
> ```python
> # 父子图都有 query 和 result，自动共享
> parent.add_node("child", compiled_child)
> ```
>
> **2. 状态转换（手动包装函数）**
> 用包装函数在父图状态和子图状态之间做显式映射。适用于复用第三方子图、父子图 Schema 不同的场景。通过 `input_schema` / `output_schema` 定义清晰的接口契约。
>
> ```python
> def call_child(state: ParentState):
>     result = compiled_child.invoke({"text": state["query"]})
>     return {"answer": result["output"]}
> parent.add_node("child", call_child)
> ```
>
> **3. Command.PARENT 跨图通信**
> 子图通过 `Command(goto="target", graph=Command.PARENT, update={...})` 向父图发送路由指令和状态更新。适用于多代理 Handoff、子图需要决定父图下一步走向的场景。这是唯一让子图能"控制"父图路由的机制。
>
> **实际项目选择策略：**
> - 同团队简单组合 -> 共享键
> - 跨团队/第三方集成 -> 状态转换 + input/output_schema
> - 多代理切换 -> Command.PARENT
>
> 在我做的多代理客服系统中，每个专业代理是独立子图，用 Command.PARENT 实现代理间 Handoff，父图只负责调度，子图各自维护私有状态。

### 为什么这个回答出彩？

1. 分层递进，覆盖三种方式而非仅两种
2. 每种方式给出了明确的适用场景和权衡
3. 提到 input_schema/output_schema 的接口设计思维
4. 结合实际项目经验说明选型逻辑

## 问题2："如何在 LangGraph 中实现多代理 Handoff？"

### 普通回答（不出彩）

"用条件边判断当前应该由哪个代理处理，然后路由到对应节点就行了。"

### 出彩回答（推荐）

> **多代理 Handoff 的核心是让每个代理作为独立子图，通过 Command.PARENT 实现切换：**
>
> **1. 每个代理是独立子图**
> 每个代理有自己的 StateGraph、私有状态和独立逻辑。编译后作为父图的一个节点。这样代理可以独立开发和测试。
>
> **2. 使用 Command(graph=Command.PARENT) 触发切换**
> 当代理判断需要转交时，返回 `Command(goto="target_agent", graph=Command.PARENT, update={"messages": [...]})`。`graph=Command.PARENT` 让路由指令冒泡到父图层级执行。
>
> ```python
> def agent_node(state: AgentState):
>     response = llm.invoke(state["messages"])
>     if needs_handoff(response):
>         return Command(
>             goto="specialist_agent",
>             graph=Command.PARENT,
>             update={"messages": [response]}
>         )
>     return Command(update={"messages": [response]})
> ```
>
> **3. 父图状态需要 reducer**
> 父图的 `messages` 字段必须用 `Annotated[list, add_messages]`，因为多个代理子图都会向其追加消息。没有 reducer，消息会被覆盖而非累积。
>
> **4. Checkpointer 策略**
> - **继承模式**（默认）：子图共用父图的 checkpointer，对话历史统一管理
> - **独立模式**：子图有自己的 checkpointer，适合需要独立恢复的场景
> - **禁用模式**：子图传 `checkpointer=False`，适合无状态的工具调用代理
>
> **5. 与 LangGraph Swarm 的对比**
> LangGraph 提供了 `create_swarm()` 工具函数，它封装了上述模式。Swarm 适合标准化的代理切换场景；手动实现适合需要自定义路由逻辑、状态转换或分层 Handoff 的场景。

### 为什么这个回答出彩？

1. 架构清晰：独立子图 + Command.PARENT 的分层设计
2. 细节到位：提到 reducer 的必要性和 checkpointer 策略
3. 代码示例直观展示 Handoff 核心机制
4. 有对比视角：手动实现 vs Swarm 的选型建议
5. 体现了对生产环境的思考（状态管理、持久化）
