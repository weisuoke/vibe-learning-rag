## 为什么要学这个？

### 现实问题

你已经学会了 LangGraph 的各种零件——并行执行、循环迭代、条件分支。但面对一个真实需求时，第一个问题往往是：

**"我该用什么模式来组织这些零件？"**

```python
# 场景1：用户提问 → 检索文档 → 生成回答
# 这是顺序链式？还是需要路由？

# 场景2：同时搜索 5 个数据源 → 合并结果
# 这是并行模式？还是 Map-Reduce？

# 场景3：生成内容 → 评估质量 → 不合格就重写
# 这是评估者-优化者？还是简单循环？

# 场景4：把大任务拆成子任务 → 分配给不同 Agent
# 这是编排者-工作者？还是路由模式？
```

没有模式意识，你会：
- 用顺序链式硬写本该并行的逻辑，性能差 3-5 倍
- 把路由和循环混在一起，代码难以维护
- 遇到新需求不知道从哪个模式开始设计

### LangGraph 的解决方案：六大工作流模式

Anthropic 和 LangChain 团队总结了 AI 应用中最常见的六种工作流模式，每种模式解决一类特定问题：

```
┌─────────────────────────────────────────────────────────────┐
│                    六大工作流模式                              │
│         "从需求到实现的设计蓝图"                                │
├─────────────┬─────────────┬─────────────────────────────────┤
│  基础模式    │  组合模式    │  高级模式                        │
├─────────────┼─────────────┼─────────────────────────────────┤
│ 1.顺序链式  │ 3.并行模式   │ 4.编排者-工作者                   │
│ 2.路由模式  │ 6.Map-Reduce │ 5.评估者-优化者                  │
└─────────────┴─────────────┴─────────────────────────────────┘
```

**工作流模式就是 AI 应用的"设计模式"——不用从零发明轮子，直接套用经过验证的架构。**

## 六大工作流模式速览

### 模式 1: 顺序链式模式 (Prompt Chaining)

**一句话**: 多个步骤按顺序执行，前一步输出作为后一步输入，中间可插入 Gate 检查点

```
[输入] → [步骤1] → [Gate检查] → [步骤2] → [步骤3] → [输出]
                       ↓ (不通过)
                     [终止]
```

```python
# 典型实现
builder.add_edge(START, "generate")
builder.add_conditional_edges("generate", quality_gate)  # Gate 检查
builder.add_edge("refine", "polish")
builder.add_edge("polish", END)
```

**适用场景**: 任务可分解为明确的线性步骤，每步有清晰的输入输出
**RAG 应用**: 查询理解 → 检索 → 重排序 → 生成 → 格式化

---

### 模式 2: 路由模式 (Routing)

**一句话**: 根据输入特征动态选择执行路径，通常由 LLM 结构化输出驱动路由决策

```
              ┌→ [处理器A] ─┐
[输入] → [路由器] → [处理器B] ─┼→ [输出]
              └→ [处理器C] ─┘
```

```python
# 典型实现：LLM 驱动路由
def llm_router(state) -> Literal["story", "joke", "poem"]:
    result = llm.with_structured_output(RouteSchema).invoke(state["input"])
    return result.route_type

builder.add_conditional_edges("classifier", llm_router)
```

**适用场景**: 不同类型的输入需要不同的处理逻辑
**RAG 应用**: 问题类型分类 → 事实查询走检索、闲聊走直接回复、计算走工具调用

---

### 模式 3: 并行模式 (Parallelization)

**一句话**: 将任务 Fan-out 到多个并行分支同时执行，再 Fan-in 汇聚结果

```
           ┌→ [分支A] ─┐
[输入] → [分发] → [分支B] ─┼→ [汇聚] → [输出]
           └→ [分支C] ─┘
```

```python
# 典型实现：静态并行 + defer 同步
builder.add_edge("dispatch", "branch_a")
builder.add_edge("dispatch", "branch_b")
builder.add_edge("dispatch", "branch_c")
builder.add_edge(["branch_a", "branch_b", "branch_c"], "aggregate")
```

**适用场景**: 多个独立子任务可同时执行，最后合并结果
**RAG 应用**: 同时查询多个知识库、并行生成多个候选答案后投票

---

### 模式 4: 编排者-工作者模式 (Orchestrator-Worker)

**一句话**: 编排者动态分解任务并分配给工作者，工作者并行执行后由合成器汇总

```
[输入] → [编排者] → [工作者1] ─┐
            ↓       [工作者2] ─┼→ [合成器] → [输出]
         (动态分解)  [工作者N] ─┘
```

```python
# 典型实现：Functional API
@task
def orchestrate(state):
    subtasks = llm.invoke(f"将以下任务分解为子任务: {state['task']}")
    results = [worker(t) for t in subtasks]  # 动态并行
    return synthesize(results)
```

**适用场景**: 任务复杂度未知，需要 LLM 动态决定如何分解
**RAG 应用**: 复杂问题拆解为多个子问题，分别检索后综合回答

---

### 模式 5: 评估者-优化者模式 (Evaluator-Optimizer)

**一句话**: 生成结果后由评估者打分，不合格则带反馈返回优化者改进，形成迭代循环

```
[输入] → [生成器] → [评估者] ──(合格)──→ [输出]
              ↑          │
              └──(不合格+反馈)──┘
```

```python
# 典型实现：条件边控制循环
def evaluate_gate(state) -> Literal["output", "revise"]:
    if state["score"] >= 0.8:
        return "output"
    return "revise"  # 带反馈回到生成器

builder.add_conditional_edges("evaluator", evaluate_gate)
builder.add_edge("revise", "generator")  # 循环回去
```

**适用场景**: 输出质量有明确评估标准，需要迭代改进
**RAG 应用**: 生成答案 → 检查幻觉/相关性 → 不合格则改写查询重新检索

---

### 模式 6: Map-Reduce 模式

**一句话**: 使用 Send API 在运行时动态创建任意数量的并行任务，结果通过 reducer 聚合

```
[输入] → [拆分器] ──Send──→ [工作者] ×N → [聚合器] → [输出]
                  (数量运行时决定)
```

```python
# 典型实现：Send API
def fan_out(state):
    return [Send("worker", {"item": i}) for i in state["items"]]

builder.add_conditional_edges("splitter", fan_out)
```

**适用场景**: 并行任务数量在编译时未知，由运行时数据决定
**RAG 应用**: 对 N 篇检索文档分别生成摘要，N 由检索结果决定

## 六大模式对比总览

| 模式 | 核心机制 | 数据流 | 复杂度 | 典型场景 |
|------|----------|--------|--------|----------|
| 顺序链式 | `add_edge` + Gate | 线性 | ⭐ | 多步处理管道 |
| 路由 | `add_conditional_edges` | 分支 | ⭐⭐ | 意图分类分发 |
| 并行 | Fan-out/Fan-in + `defer` | 扇形 | ⭐⭐ | 多源并行查询 |
| 编排者-工作者 | `@task` + 动态分解 | 星形 | ⭐⭐⭐ | 复杂任务拆解 |
| 评估者-优化者 | 条件边 + 循环 | 环形 | ⭐⭐⭐ | 质量迭代优化 |
| Map-Reduce | `Send` API + reducer | 动态扇形 | ⭐⭐⭐ | 批量并行处理 |

### 模式选型决策树

```
你的任务需要什么？
│
├─ 步骤固定，按顺序执行？
│  └─→ 顺序链式模式
│
├─ 不同输入走不同路径？
│  └─→ 路由模式
│
├─ 多个子任务可以同时跑？
│  ├─ 任务数量编译时已知？
│  │  └─→ 并行模式
│  └─ 任务数量运行时才知道？
│     └─→ Map-Reduce 模式
│
├─ 任务需要 LLM 动态拆解？
│  └─→ 编排者-工作者模式
│
└─ 输出需要迭代改进？
   └─→ 评估者-优化者模式
```

## 实战场景预览

### 场景 1: 模式选型决策树

**问题**: 面对新需求，不知道该用哪种工作流模式
**方案**: 基于任务特征的决策树 + 模式组合策略
**适用**: 架构设计阶段、技术选型评审

---

### 场景 2: RAG 多步推理管道

**问题**: 复杂问题需要多步检索和推理，不同类型问题走不同路径
**方案**: 顺序链式（主管道）+ 路由（问题分类）组合
**适用**: 知识库问答、文档分析、研究助手

---

### 场景 3: 多 Agent 协作系统

**问题**: 复杂任务需要多个专家 Agent 协作完成
**方案**: 编排者动态分解任务 + Map-Reduce 并行执行
**适用**: 研究报告生成、代码审查、多领域分析

---

### 场景 4: 自优化内容生成

**问题**: 生成内容质量不稳定，需要自动评估和改进
**方案**: 评估者-优化者模式，生成 → 评估 → 反馈 → 改进循环
**适用**: 文案生成、代码生成、翻译优化

---

### 场景 5: 智能客服分流系统

**问题**: 不同类型的客户问题需要路由到不同处理流程
**方案**: 路由模式（意图分类）+ 并行模式（多源信息查询）
**适用**: 客服系统、工单分发、服务台

---

### 场景 6: 批量文档处理管道

**问题**: 大量文档需要并行处理，文档数量运行时才确定
**方案**: Map-Reduce（动态并行处理）+ 顺序链式（每篇文档的处理流程）
**适用**: 文档摘要、批量翻译、数据提取

## 核心 API 速查

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send, Command
from langgraph.func import entrypoint, task
from typing import Literal, Annotated, TypedDict
import operator

# ===== 模式1: 顺序链式 =====
builder.add_edge(START, "step1")
builder.add_edge("step1", "step2")
builder.add_edge("step2", END)
# 快捷方式
builder.add_sequence(["step1", "step2", "step3"])

# ===== 模式2: 路由 =====
def router(state) -> Literal["handler_a", "handler_b"]:
    return classify(state["input"])
builder.add_conditional_edges("classifier", router)

# ===== 模式3: 并行 =====
builder.add_edge("start", "branch_a")
builder.add_edge("start", "branch_b")
builder.add_edge(["branch_a", "branch_b"], "merge")

# ===== 模式4: 编排者-工作者 (Functional API) =====
@task
def worker(item):
    return process(item)

@entrypoint()
def orchestrator(inputs):
    subtasks = plan(inputs["task"])
    results = [worker(t) for t in subtasks]
    return synthesize(results)

# ===== 模式5: 评估者-优化者 =====
def eval_gate(state) -> Literal["output", "revise"]:
    return "output" if state["score"] >= 0.8 else "revise"
builder.add_conditional_edges("evaluator", eval_gate)
builder.add_edge("revise", "generator")  # 循环

# ===== 模式6: Map-Reduce =====
def fan_out(state):
    return [Send("worker", {"item": i}) for i in state["items"]]
builder.add_conditional_edges("splitter", fan_out)

class State(TypedDict):
    results: Annotated[list, operator.add]  # reducer 聚合
```

## 快速开始

### 5 分钟快速体验：路由模式

```python
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END

# 1. 定义状态
class State(TypedDict):
    topic: str
    route: str
    result: str

# 2. 定义节点
def classify(state: State):
    topic = state["topic"].lower()
    if "代码" in topic or "编程" in topic:
        return {"route": "coder"}
    elif "写作" in topic or "文章" in topic:
        return {"route": "writer"}
    return {"route": "generalist"}

def coder(state: State):
    return {"result": f"[代码专家] 正在处理: {state['topic']}"}

def writer(state: State):
    return {"result": f"[写作专家] 正在处理: {state['topic']}"}

def generalist(state: State):
    return {"result": f"[通用助手] 正在处理: {state['topic']}"}

# 3. 路由函数
def route_by_type(state: State) -> Literal["coder", "writer", "generalist"]:
    return state["route"]

# 4. 构建图
builder = StateGraph(State)
builder.add_node("classify", classify)
builder.add_node("coder", coder)
builder.add_node("writer", writer)
builder.add_node("generalist", generalist)

builder.add_edge(START, "classify")
builder.add_conditional_edges("classify", route_by_type)
builder.add_edge("coder", END)
builder.add_edge("writer", END)
builder.add_edge("generalist", END)

graph = builder.compile()

# 5. 测试
r1 = graph.invoke({"topic": "帮我写一段 Python 代码"})
print(r1["result"])  # [代码专家] 正在处理: 帮我写一段 Python 代码

r2 = graph.invoke({"topic": "帮我写一篇文章"})
print(r2["result"])  # [写作专家] 正在处理: 帮我写一篇文章
```

**运行结果**: 同一个图，不同输入自动路由到不同专家。这就是路由模式——六大工作流模式中最直观的一个。

## 关键要点总结

1. **六大工作流模式是 AI 应用的设计蓝图**，覆盖从简单管道到复杂多 Agent 系统的全部编排需求
2. **基础模式（顺序 + 路由）覆盖 60% 场景**，先掌握这两个再学其他
3. **并行模式和 Map-Reduce 的区别**：前者任务数编译时已知，后者运行时才知道
4. **编排者-工作者适合"不知道怎么拆"的任务**，让 LLM 动态决定分解方案
5. **评估者-优化者是质量保证的核心模式**，RAG 中的 Self-RAG 就是典型应用
6. **模式可以组合使用**，真实系统往往是多种模式的嵌套组合
