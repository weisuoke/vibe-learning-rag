# 核心概念 5：中间步骤管理

> Agent 每走一步都会留下"脚印"——intermediate_steps 就是这些脚印的有序记录。管理好这些脚印，决定了 Agent 能走多远、走多稳。

---

## 什么是 intermediate_steps？

在 AgentExecutor 的执行循环中，`intermediate_steps` 是一个列表，记录了 Agent 从开始到当前的所有"动作-观察"对：

```python
# 类型定义
intermediate_steps: list[tuple[AgentAction, str]]

# 每个元素是一个二元组：
# - AgentAction: Agent 决定执行的动作（包含 tool 名称、tool_input、log）
# - str: 工具执行后返回的 observation（观察结果）
```

用一个具体例子来理解：

```python
# 用户问："北京今天天气怎么样？"
# Agent 执行过程中，intermediate_steps 逐步积累：

intermediate_steps = [
    # 第 1 步：Agent 决定查天气，得到结果
    (
        AgentAction(tool="weather_api", tool_input="北京", log="我需要查询北京的天气"),
        "北京今天晴，气温 25°C，湿度 40%"
    ),
    # 第 2 步：Agent 决定查空气质量，得到结果
    (
        AgentAction(tool="air_quality", tool_input="北京", log="再查一下空气质量"),
        "北京 AQI 85，轻度污染"
    ),
]
```

## intermediate_steps 如何在循环中积累

回顾主循环的核心代码（agent.py:1570）：

```python
def _call(self, inputs, run_manager=None):
    name_to_tool_map = {tool.name: tool for tool in self.tools}
    intermediate_steps = []          # ① 初始化为空列表
    iterations = 0
    time_elapsed = 0.0
    start_time = time.time()

    while self._should_continue(iterations, time_elapsed):
        # ② 执行一步：调用 LLM → 执行工具
        next_step_output = self._take_next_step(
            name_to_tool_map, color_mapping,
            inputs, intermediate_steps, run_manager  # ③ 传入当前步骤
        )

        # ④ 如果 Agent 决定结束
        if isinstance(next_step_output, AgentFinish):
            return self._return(next_step_output, intermediate_steps, run_manager)

        # ⑤ 将新步骤追加到列表
        intermediate_steps.extend(next_step_output)

        iterations += 1
        time_elapsed = time.time() - start_time

    # ⑥ 超出限制，early stopping
    output = self._action_agent.return_stopped_response(
        self.early_stopping_method, intermediate_steps, **inputs
    )
    return self._return(output, intermediate_steps, run_manager)
```

关键流程：

```
初始化: intermediate_steps = []
│
▼ 循环开始
├─ _take_next_step(inputs, intermediate_steps)
│   ├─ _prepare_intermediate_steps(intermediate_steps)  ← 可能裁剪
│   ├─ agent.plan(intermediate_steps, **inputs)         ← LLM 决策
│   └─ tool.run(action.tool_input)                      ← 工具执行
│
├─ 返回 AgentFinish? → _return(output, intermediate_steps) → 结束
├─ 返回 AgentStep?  → intermediate_steps.extend(steps)  → 继续
│
▼ iterations += 1 → 回到循环开始
```

每一轮循环，`intermediate_steps` 都会增长。LLM 在下一轮决策时，能看到之前所有步骤的结果，从而做出更好的判断。

## _prepare_intermediate_steps：步骤裁剪

随着循环进行，`intermediate_steps` 会越来越长。每一步的 observation 可能包含大量文本（比如搜索结果、API 返回值），这些内容都会被拼接到 Prompt 中发送给 LLM。

问题来了：
- **Context Window 溢出**：LLM 的上下文窗口有限，步骤太多会超出限制
- **成本失控**：每次调用 LLM 的 token 数量随步骤增长线性增加
- **注意力稀释**：LLM 在超长上下文中可能"忘记"早期重要信息

`trim_intermediate_steps` 就是解决方案。

### 源码实现（agent.py:1718）

```python
def _prepare_intermediate_steps(
    self, intermediate_steps: list[tuple[AgentAction, str]]
) -> list[tuple[AgentAction, str]]:
    # 模式一：整数模式 - 只保留最后 N 步
    if (
        isinstance(self.trim_intermediate_steps, int)
        and self.trim_intermediate_steps > 0
    ):
        return intermediate_steps[-self.trim_intermediate_steps :]

    # 模式二：函数模式 - 自定义裁剪逻辑
    if callable(self.trim_intermediate_steps):
        return self.trim_intermediate_steps(intermediate_steps)

    # 默认：不裁剪（trim_intermediate_steps = -1）
    return intermediate_steps
```

### 模式一：整数模式（保留最后 N 步）

```python
from langchain.agents import AgentExecutor

executor = AgentExecutor(
    agent=my_agent,
    tools=my_tools,
    trim_intermediate_steps=3,  # 只保留最后 3 步
)

# 假设 Agent 已经执行了 5 步：
# intermediate_steps = [step1, step2, step3, step4, step5]
# 传给 LLM 的只有: [step3, step4, step5]
```

适用场景：Agent 的早期探索步骤不重要，只需要最近的上下文。

### 模式二：函数模式（自定义裁剪）

```python
def smart_trim(steps: list[tuple]) -> list[tuple]:
    """智能裁剪：保留第一步（初始意图）+ 最后 2 步（最新上下文）"""
    if len(steps) <= 3:
        return steps
    return [steps[0]] + steps[-2:]

executor = AgentExecutor(
    agent=my_agent,
    tools=my_tools,
    trim_intermediate_steps=smart_trim,
)
```

更高级的裁剪策略：

```python
def token_aware_trim(steps: list[tuple], max_tokens: int = 2000) -> list[tuple]:
    """基于 token 数量的裁剪：从最新步骤开始保留，直到达到 token 上限"""
    result = []
    total_tokens = 0
    for action, observation in reversed(steps):
        # 粗略估算 token 数（1 个中文字 ≈ 2 tokens）
        step_tokens = len(action.log + observation) * 2
        if total_tokens + step_tokens > max_tokens:
            break
        result.insert(0, (action, observation))
        total_tokens += step_tokens
    return result

executor = AgentExecutor(
    agent=my_agent,
    tools=my_tools,
    trim_intermediate_steps=token_aware_trim,
)
```

### 裁剪时机

裁剪发生在 `_iter_next_step` 的最开始，也就是每次调用 LLM 之前：

```python
# agent.py:1301 _iter_next_step（简化）
def _iter_next_step(self, name_to_tool_map, color_mapping,
                     inputs, intermediate_steps, run_manager):
    # ① 先裁剪
    intermediate_steps = self._prepare_intermediate_steps(intermediate_steps)

    # ② 再调用 LLM
    output = self.agent.plan(intermediate_steps, callbacks=..., **inputs)
    # ...
```

注意：裁剪只影响传给 LLM 的内容，不影响 `intermediate_steps` 列表本身。原始列表始终保持完整。

## _return：返回结果处理

当 Agent 决定结束（返回 AgentFinish）或超出限制时，`_return` 方法负责组装最终输出。

### 源码实现（agent.py:1240）

```python
def _return(
    self,
    output: AgentFinish,
    intermediate_steps: list[tuple[AgentAction, str]],
    run_manager: CallbackManagerForChainRun | None = None,
) -> dict[str, Any]:
    # ① 触发 on_agent_finish 回调
    if run_manager:
        run_manager.on_agent_finish(output, color="green", verbose=self.verbose)

    # ② 提取返回值
    final_output = output.return_values

    # ③ 如果配置了 return_intermediate_steps，附加步骤记录
    if self.return_intermediate_steps:
        final_output["intermediate_steps"] = intermediate_steps

    return final_output
```

### return_intermediate_steps 的作用

默认情况下，AgentExecutor 只返回最终答案。开启 `return_intermediate_steps=True` 后，可以获取完整的推理轨迹：

```python
executor = AgentExecutor(
    agent=my_agent,
    tools=my_tools,
    return_intermediate_steps=True,  # 暴露推理过程
)

result = executor.invoke({"input": "北京今天天气怎么样？"})

# 最终答案
print(result["output"])
# "北京今天晴，气温 25°C，空气质量轻度污染，建议外出佩戴口罩。"

# 推理轨迹
for action, observation in result["intermediate_steps"]:
    print(f"工具: {action.tool}")
    print(f"输入: {action.tool_input}")
    print(f"结果: {observation}")
    print("---")
# 工具: weather_api
# 输入: 北京
# 结果: 北京今天晴，气温 25°C，湿度 40%
# ---
# 工具: air_quality
# 输入: 北京
# 结果: 北京 AQI 85，轻度污染
# ---
```

这对调试和可观测性至关重要——你能看到 Agent "是怎么想的"。

## _consume_next_step：迭代器输出转换

`_take_next_step` 内部调用 `_iter_next_step`（一个生成器），然后通过 `_consume_next_step` 将生成器输出收集为列表或 AgentFinish：

```python
# agent.py:1280（简化）
def _consume_next_step(
    self, values: NextStepOutput
) -> Union[AgentFinish, list[tuple[AgentAction, str]]]:
    try:
        output = next(values)  # 获取第一个输出
    except StopIteration:
        return []

    if isinstance(output, AgentFinish):
        return output

    # 收集所有 AgentStep，转换为 (action, observation) 元组
    steps = [output]
    for step in values:
        steps.append(step)

    return [
        (step.action, step.observation) for step in steps
    ]
```

为什么用生成器？因为 `_iter_next_step` 可能产出多个步骤（当 Agent 是 MultiActionAgent 时，一次决策可以请求执行多个工具）。生成器模式让每个步骤可以独立触发回调，而不是等所有工具执行完才通知。

## AgentExecutorIterator：流式中间步骤

除了一次性执行，AgentExecutor 还支持通过迭代器逐步获取中间结果：

```python
# agent_iterator.py 提供了 AgentExecutorIterator

executor = AgentExecutor(
    agent=my_agent,
    tools=my_tools,
    return_intermediate_steps=True,
)

# 方式一：一次性执行
result = executor.invoke({"input": "分析这篇文章"})

# 方式二：逐步迭代（流式）
for step in executor.iter({"input": "分析这篇文章"}):
    if "intermediate_step" in step:
        action, observation = step["intermediate_step"]
        print(f"[进行中] 使用工具: {action.tool}")
        print(f"[进行中] 观察结果: {observation[:100]}...")
    elif "output" in step:
        print(f"[完成] {step['output']}")
```

### 迭代器内部机制

```python
# AgentExecutorIterator 的核心逻辑（简化）
class AgentExecutorIterator:
    def __iter__(self):
        self.intermediate_steps = []
        self.iterations = 0
        return self

    def __next__(self):
        # 检查是否应该继续
        if not self.agent_executor._should_continue(
            self.iterations, self.time_elapsed
        ):
            raise StopIteration

        # 执行一步
        next_output = self.agent_executor._take_next_step(
            self.name_to_tool_map,
            self.color_mapping,
            self.inputs,
            self.intermediate_steps,
            self.run_manager,
        )

        if isinstance(next_output, AgentFinish):
            # 返回最终结果并停止迭代
            result = self.agent_executor._return(
                next_output, self.intermediate_steps
            )
            raise StopIteration

        # 追加步骤并返回中间结果
        self.intermediate_steps.extend(next_output)
        self.iterations += 1
        return {"intermediate_step": next_output[0]}
```

迭代器模式的价值：
- **实时反馈**：用户可以看到 Agent 正在做什么，而不是等待最终结果
- **提前终止**：外部代码可以在任意步骤中断迭代
- **流式 UI**：前端可以逐步展示 Agent 的推理过程

## 回调系统：on_agent_action 与 on_agent_finish

intermediate_steps 的每一步都会触发回调，这是 LangChain 可观测性的基础。

### 回调触发时机

```python
# _iter_next_step 中的回调触发（简化）
def _iter_next_step(self, ..., run_manager):
    # LLM 返回 AgentAction 后
    output = self.agent.plan(intermediate_steps, **inputs)

    if isinstance(output, AgentFinish):
        yield output  # _return 中触发 on_agent_finish
        return

    # 对每个 action 触发回调
    for agent_action in actions:
        # ① 触发 on_agent_action
        if run_manager:
            run_manager.on_agent_action(agent_action, color="green")

        # ② 执行工具
        observation = tool.run(agent_action.tool_input)

        # ③ 产出 AgentStep
        yield AgentStep(action=agent_action, observation=observation)
```

### 自定义回调处理器

```python
from langchain.callbacks.base import BaseCallbackHandler

class AgentStepLogger(BaseCallbackHandler):
    """记录 Agent 每一步的动作和结果"""

    def on_agent_action(self, action, **kwargs):
        print(f"[ACTION] 工具={action.tool}, 输入={action.tool_input}")
        print(f"[THOUGHT] {action.log}")

    def on_agent_finish(self, finish, **kwargs):
        print(f"[FINISH] {finish.return_values}")

    def on_tool_start(self, serialized, input_str, **kwargs):
        print(f"[TOOL START] {serialized.get('name', 'unknown')}")

    def on_tool_end(self, output, **kwargs):
        print(f"[TOOL END] 结果长度={len(str(output))}")

# 使用回调
executor = AgentExecutor(
    agent=my_agent,
    tools=my_tools,
    callbacks=[AgentStepLogger()],
    verbose=False,  # 关闭默认输出，用自定义回调替代
)
```

### 回调与 intermediate_steps 的关系

```
intermediate_steps 积累过程:
│
├─ Step 1: agent.plan() → AgentAction
│   ├─ on_agent_action(action)          ← 回调触发
│   ├─ tool.run() → observation
│   ├─ on_tool_end(observation)         ← 回调触发
│   └─ intermediate_steps.append((action, observation))
│
├─ Step 2: agent.plan() → AgentAction
│   ├─ on_agent_action(action)          ← 回调触发
│   ├─ tool.run() → observation
│   └─ intermediate_steps.append((action, observation))
│
└─ Step 3: agent.plan() → AgentFinish
    └─ on_agent_finish(finish)          ← 回调触发
```

### 技巧 1：打印完整推理链

```python
executor = AgentExecutor(
    agent=my_agent,
    tools=my_tools,
    return_intermediate_steps=True,
    verbose=True,  # 开启详细输出

result = executor.invoke({"input": "帮我查一下 Python 3.13 的新特性"})

for i, (action, obs) in enumerate(result["intermediate_steps"]):
    print(f"\n=== 第 {i+1} 步 ===")
    print(f"思考: {action.log}")
    print(f"工具: {action.tool}({action.tool_input})")
    print(f"观察: {obs[:200]}...")
```

### 技巧 2：检测循环行为

Agent 有时会陷入"死循环"——反复调用同一个工具。通过 intermediate_steps 可以检测：

```python
def detect_loop(steps: list[tuple], threshold: int = 3) -> bool:
    """检测 Agent 是否在重复调用同一工具"""
    if len(steps) < threshold:
        return False
    recent_tools = [action.tool for action, _ in steps[-threshold:]]
    return len(set(recent_tools)) == 1  # 最近 N 步都是同一个工具

def loop_aware_trim(steps: list[tuple]) -> list[tuple]:
    """检测到循环时，只保留最后一步 + 提示信息"""
    if detect_loop(steps):
        # 可以在这里注入提示，让 Agent 换个思路
        print("[WARNING] 检测到 Agent 循环行为")
    return steps[-3:]  # 保留最后 3 步

executor = AgentExecutor(
    agent=my_agent,
    tools=my_tools,
    trim_intermediate_steps=loop_aware_trim,
)
```

### 技巧 3：成本估算

```python
def estimate_cost(steps: list[tuple]) -> dict:
    """估算 intermediate_steps 的 token 消耗"""
    total_chars = 0
    for action, observation in steps:
        total_chars += len(action.log) + len(str(action.tool_input)) + len(observation)

    estimated_tokens = total_chars // 2  # 粗略估算
    # GPT-4 价格：$30/1M input tokens
    estimated_cost = (estimated_tokens / 1_000_000) * 30

    return {
        "steps": len(steps),
        "total_chars": total_chars,
        "estimated_tokens": estimated_tokens,
        "estimated_cost_usd": round(estimated_cost, 4),
    }

result = executor.invoke({"input": "..."})
if "intermediate_steps" in result:
    cost = estimate_cost(result["intermediate_steps"])
    print(f"本次执行: {cost['steps']} 步, ~{cost['estimated_tokens']} tokens, ~${cost['estimated_cost_usd']}")
```

### 4. 裁剪策略要匹配业务场景

| 场景 | 推荐策略 | 原因 |
|------|----------|------|
| 简单问答 | `trim=3` | 步骤少，保留最近即可 |
| 多步研究 | `trim=5` | 需要更多上下文 |
| 长文档分析 | 自定义函数（基于 token） | observation 可能很长 |
| 数据分析 | 自定义函数（保留首尾） | 第一步的数据加载很重要 |

## 与 RAG 的关联

在 RAG Agent 中，intermediate_steps 的管理尤为重要：

1. **检索结果很长**：每次向量检索返回的文档片段可能有数千字，几步下来就会撑爆 context window
2. **多轮检索**：Agent 可能需要多次检索不同关键词，每次的 observation 都是大段文本
3. **成本敏感**：RAG 场景下 token 消耗主要来自检索结果，裁剪直接影响成本

```python
def rag_aware_trim(steps: list[tuple], max_total_tokens: int = 4000) -> list[tuple]:
    """RAG 场景专用裁剪：限制检索结果的总 token 数"""
    result = []
    total = 0
    for action, observation in reversed(steps):
        obs_tokens = len(observation) // 2  # 粗略估算
        if total + obs_tokens > max_total_tokens:
            # 截断 observation 而非丢弃整个步骤
            remaining = max_total_tokens - total
            truncated_obs = observation[:remaining * 2]
            result.insert(0, (action, truncated_obs + "\n[...已截断]"))
            break
        result.insert(0, (action, observation))
        total += obs_tokens
    return result
```

## 小结

intermediate_steps 是 AgentExecutor 的"飞行记录仪"：

- **积累**：每轮循环追加 `(AgentAction, observation)` 元组
- **裁剪**：`_prepare_intermediate_steps` 在每次 LLM 调用前裁剪，支持整数模式和函数模式
- **返回**：`_return` 根据 `return_intermediate_steps` 决定是否暴露推理轨迹
- **流式**：`AgentExecutorIterator` 支持逐步获取中间结果
- **回调**：`on_agent_action` / `on_agent_finish` 提供实时可观测性

一句话：**intermediate_steps 是 Agent 的推理日志，trim 控制 LLM 看到多少历史，return 控制用户看到多少过程。**
