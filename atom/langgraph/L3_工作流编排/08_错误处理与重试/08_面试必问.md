## Q1: LangGraph 的 RetryPolicy 默认会重试所有异常吗？

**❌ 错误答案：** "是的，默认重试所有异常。"

**✅ 正确答案：**

不是。RetryPolicy 的默认行为（`default_retry_on`）会**智能过滤**异常：

**会重试：**
- `ConnectionError` - 网络连接错误
- HTTP 5xx 错误 - 服务端临时故障
- 其他未知异常（保守策略）

**不会重试：**
- `ValueError`, `TypeError` - 编程错误
- `ImportError`, `NameError` - 代码错误
- `OSError`, `RuntimeError` - 系统错误
- HTTP 4xx 错误 - 客户端错误

**原理：** 编程错误重试也不会成功，应该直接抛出让开发者修复。

```python
# 源码：default_retry_on 函数
def default_retry_on(exc: Exception) -> bool:
    if isinstance(exc, ConnectionError):
        return True  # 重试网络错误
    if isinstance(exc, (ValueError, TypeError)):
        return False  # 不重试编程错误
    return True  # 其他异常保守重试
```

## Q2: max_attempts=3 是重试 3 次还是总共尝试 3 次？

**❌ 错误答案：** "重试 3 次，所以总共 4 次尝试。"

**✅ 正确答案：**

**总共尝试 3 次**（1 次执行 + 2 次重试）。

```python
RetryPolicy(max_attempts=3)

# 执行流程：
# 尝试 1（首次执行）→ 失败
# 尝试 2（第 1 次重试）→ 失败
# 尝试 3（第 2 次重试）→ 失败
# → 抛出异常
```

**记忆技巧：** `max_attempts` = "最多尝试次数"，不是"最多重试次数"。

## Q3: 指数退避的计算公式是什么？

**❌ 错误答案：** "每次等待时间翻倍。"

**✅ 正确答案：**

```python
interval = min(
    max_interval,
    initial_interval * (backoff_factor ** (attempts - 1))
)

# 如果启用 jitter：
sleep_time = interval + random.uniform(0, 1)
```

**示例（默认参数）：**
```
initial_interval = 0.5s
backoff_factor = 2.0
max_interval = 128.0s

尝试 1 → 尝试 2: 0.5 * (2 ** 0) = 0.5s
尝试 2 → 尝试 3: 0.5 * (2 ** 1) = 1.0s
尝试 3 → 尝试 4: 0.5 * (2 ** 2) = 2.0s
尝试 4 → 尝试 5: 0.5 * (2 ** 3) = 4.0s
...
上限: 128.0s
```

**为什么需要 max_interval？** 防止等待时间无限增长。

## Q4: 什么是 Jitter？为什么需要它？

**❌ 错误答案：** "Jitter 是随机延迟，让重试更随机。"

**✅ 正确答案：**

**Jitter（抖动）** 是在退避时间上添加随机偏移，防止**重试风暴**。

**问题场景：**
```
100 个客户端同时请求 → 服务端崩溃
→ 所有客户端同时等待 2 秒
→ 2 秒后同时重试
→ 服务端再次崩溃
→ 恶性循环
```

**Jitter 解决方案：**
```python
# 不启用 jitter：所有客户端等待 2.0s
sleep(2.0)

# 启用 jitter：每个客户端等待 2.0s + [0, 1)s
sleep(2.0 + random.uniform(0, 1))
# 客户端 A: 2.3s
# 客户端 B: 2.7s
# 客户端 C: 2.1s
# → 请求分散，避免同时重试
```

**LangGraph 的 jitter 实现：** 在计算的间隔上加 `random.uniform(0, 1)`。

## Q5: GraphBubbleUp 异常会被重试吗？

**❌ 错误答案：** "会，因为它是异常。"

**✅ 正确答案：**

**永远不会被重试。** `GraphBubbleUp` 是**流程控制信号**，不是错误。

**源码逻辑：**
```python
def run_with_retry(task, retry_policy):
    while True:
        try:
            return task.run()
        except ParentCommand as exc:
            # 命名空间路由，不重试
            raise
        except GraphBubbleUp as exc:
            # 流程控制，不重试
            raise
        except Exception as exc:
            # 其他异常才进入重试逻辑
            if should_retry(exc):
                wait_and_retry()
            else:
                raise
```

**其他不重试的信号：**
- `ParentCommand` - 子图命令
- `GraphInterrupt` - 人工中断

## Q6: 如何为不同节点配置不同的重试策略？

**❌ 错误答案：** "在 StateGraph 上设置全局策略。"

**✅ 正确答案：**

在 `add_node()` 时为每个节点单独配置：

```python
# 数据库节点：快速重试
db_retry = RetryPolicy(max_attempts=3, initial_interval=0.1)

# LLM 节点：慢速重试
llm_retry = RetryPolicy(max_attempts=5, initial_interval=2.0)

builder = StateGraph(State)
builder.add_node("query_db", query_db, retry_policy=db_retry)
builder.add_node("call_llm", call_llm, retry_policy=llm_retry)
builder.add_node("process", process_data)  # 不重试
```

**优先级：** 节点级策略 > 全局策略（如果有）

## Q7: 如何只重试特定类型的异常？

**❌ 错误答案：** "在节点函数里 try-catch 特定异常。"

**✅ 正确答案：**

使用 `retry_on` 参数：

```python
# 方式 1：单一异常类型
RetryPolicy(retry_on=ConnectionError)

# 方式 2：多个异常类型
RetryPolicy(retry_on=(ConnectionError, TimeoutError))

# 方式 3：自定义判断函数
def should_retry(exc: Exception) -> bool:
    if isinstance(exc, httpx.HTTPStatusError):
        return exc.response.status_code >= 500
    return False

RetryPolicy(retry_on=should_retry)
```

**最佳实践：** 使用自定义函数，可以根据异常的具体属性（如 HTTP 状态码）判断。

## Q8: 什么是 Fallback 模式？与重试有什么区别？

**❌ 错误答案：** "Fallback 就是重试的另一种说法。"

**✅ 正确答案：**

**重试（Retry）** 和 **降级（Fallback）** 是两种不同的容错策略：

| 维度 | 重试 (Retry) | 降级 (Fallback) |
|------|-------------|----------------|
| **目标** | 重复执行同一操作 | 切换到备用方案 |
| **适用场景** | 瞬态错误（网络抖动） | 持续故障（服务宕机） |
| **质量** | 保持原有质量 | 可能降低质量 |
| **示例** | 重试 API 调用 3 次 | 主模型失败 → 备用模型 |

**组合使用：**
```python
# 主库：重试 3 次
try:
    return primary_db.search(query)  # 自动重试
except:
    # 重试失败后，Fallback 到备库
    return backup_db.search(query)
```

**Fallback 链：**
```
主模型 → 备用模型 → 缓存结果 → 默认响应
```

## Q9: 如何实现断路器模式？

**❌ 错误答案：** "用 try-catch 捕获异常就行。"

**✅ 正确答案：**

断路器需要**跟踪失败次数**和**状态转换**：

```python
class CircuitBreaker:
    def __init__(self, failure_threshold=5, timeout=60):
        self.state = "CLOSED"  # CLOSED, OPEN, HALF_OPEN
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.timeout = timeout
        self.last_failure_time = None

    def call(self, func, *args):
        # OPEN 状态：拒绝请求
        if self.state == "OPEN":
            if time.time() - self.last_failure_time >= self.timeout:
                self.state = "HALF_OPEN"  # 超时后进入半开
            else:
                raise Exception("断路器打开，拒绝请求")

        try:
            result = func(*args)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

    def _on_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = "OPEN"  # 打开断路器

    def _on_success(self):
        if self.state == "HALF_OPEN":
            self.state = "CLOSED"  # 恢复
        self.failure_count = 0
```

**状态转换：**
```
CLOSED → OPEN: 失败次数超过阈值
OPEN → HALF_OPEN: 超时后允许测试
HALF_OPEN → CLOSED: 测试成功
HALF_OPEN → OPEN: 测试失败
```

## Q10: 生产环境的重试策略应该如何配置？

**❌ 错误答案：** "所有节点都用默认配置。"

**✅ 正确答案：**

根据**操作特性**和**业务需求**差异化配置：

```python
# 1. 数据库查询（快速操作）
db_retry = RetryPolicy(
    max_attempts=3,        # 少量尝试
    initial_interval=0.1,  # 快速重试
    max_interval=1.0,      # 短超时
    retry_on=sqlite3.OperationalError  # 只重试锁冲突
)

# 2. 向量检索（中速操作）
retrieval_retry = RetryPolicy(
    max_attempts=4,
    initial_interval=0.5,
    max_interval=10.0,
    retry_on=ConnectionError
)

# 3. LLM 调用（慢速操作）
llm_retry = RetryPolicy(
    max_attempts=5,        # 多次尝试
    initial_interval=2.0,  # 慢速重试
    max_interval=60.0,     # 长超时
    jitter=True,           # 启用抖动
    retry_on=lambda e: isinstance(e, (RateLimitError, APITimeoutError))
)

# 4. 纯计算（不重试）
# 不传 retry_policy
```

**配置原则：**
1. **快速操作** → 快速重试，少量尝试
2. **慢速操作** → 慢速重试，多次尝试
3. **编程错误** → 不重试
4. **启用 jitter** → 防止重试风暴
5. **设置 max_interval** → 防止等待过长

**监控指标：**
- 重试率 < 10%：正常
- 重试率 > 20%：需要告警
- 平均重试次数 < 1.5：健康

### 如果面试官追问："如何测试重试逻辑？"

**答案：**

```python
# 1. 模拟失败场景
def flaky_function(state):
    global attempt_count
    attempt_count += 1
    if attempt_count < 3:
        raise ConnectionError("模拟失败")
    return {"result": "success"}

# 2. 验证重试次数
attempt_count = 0
result = graph.invoke(state)
assert attempt_count == 3  # 验证重试了 2 次

# 3. 验证退避时间
start_time = time.time()
result = graph.invoke(state)
elapsed = time.time() - start_time
# 验证总时间 ≈ 0.5s + 1.0s = 1.5s
assert 1.4 <= elapsed <= 1.6

# 4. 验证异常过滤
def programming_error(state):
    raise ValueError("编程错误")

# 应该直接抛出，不重试
with pytest.raises(ValueError):
    graph.invoke(state)
```
