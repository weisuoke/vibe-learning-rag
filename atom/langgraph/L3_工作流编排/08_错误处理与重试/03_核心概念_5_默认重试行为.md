# 默认重试行为

## 什么是默认重试行为？

**默认重试行为 = 当你使用 `RetryPolicy()` 而不指定 `retry_on` 时，LangGraph 自动判断哪些异常值得重试的内置逻辑。**

它就像一个经验丰富的运维工程师，看一眼错误就知道"这个等等再试" 还是 "这个没救了别浪费时间"。

---

## 源码解析：`default_retry_on()`

```python
# 来自 langgraph/_internal/_retry.py
def default_retry_on(exc: Exception) -> bool:
    # ✅ 重试：网络连接错误
    if isinstance(exc, ConnectionError):
        return True

    # ✅ 重试：HTTP 5xx 服务端错误
    if isinstance(exc, httpx.HTTPStatusError):
        return 500 <= exc.response.status_code < 600
    if isinstance(exc, requests.HTTPError):
        return (
            500 <= exc.response.status_code < 600
            if exc.response is not None
            else True
        )

    # ❌ 不重试：编程错误和不可恢复错误
    if isinstance(exc, (
        ValueError,           # 值错误
        TypeError,            # 类型错误
        ArithmeticError,      # 算术错误（含 ZeroDivisionError）
        ImportError,          # 导入错误
        LookupError,          # 查找错误（含 KeyError、IndexError）
        NameError,            # 名称未定义
        SyntaxError,          # 语法错误
        RuntimeError,         # 运行时错误（含 GraphRecursionError）
        ReferenceError,       # 弱引用错误
        StopIteration,        # 迭代结束
        StopAsyncIteration,   # 异步迭代结束
        OSError,              # 操作系统错误
    )):
        return False

    # ✅ 重试：其他未知异常（保守策略）
    return True
```

## 判断流程图

```
异常发生
  │
  ├─ ConnectionError? ──────────────── Yes → ✅ 重试
  │
  ├─ httpx.HTTPStatusError?
  │   └─ status_code >= 500? ───────── Yes → ✅ 重试
  │                                    No  → ❌ 不重试（4xx 客户端错误）
  │
  ├─ requests.HTTPError?
  │   └─ response 存在?
  │       ├─ Yes → status >= 500? ──── Yes → ✅ 重试
  │       │                            No  → ❌ 不重试
  │       └─ No  → ✅ 重试（无法判断，保守重试）
  │
  ├─ 编程错误类异常? ──────────────── Yes → ❌ 不重试
  │   (ValueError/TypeError/...)
  │
  └─ 其他未知异常 ─────────────────── ✅ 重试（保守策略）
```

## 为什么这样分类？

### ✅ ConnectionError → 重试

```
原因：网络问题通常是瞬态的
- DNS 解析暂时失败
- TCP 连接被重置
- 服务器临时不可达

现实类比：电话没打通 → 过一会儿再打，大概率能通
```

### ✅ HTTP 5xx → 重试

```
原因：服务端临时故障，不是你的问题
- 500 Internal Server Error：服务端临时异常
- 502 Bad Gateway：代理/负载均衡器问题
- 503 Service Unavailable：服务过载或维护中
- 504 Gateway Timeout：上游服务超时

现实类比：餐厅说"厨房暂时出了点问题" → 等几分钟再点
```

### ❌ HTTP 4xx → 不重试

```
原因：客户端错误，是你的问题，重试也不会变好
- 400 Bad Request：请求格式错误
- 401 Unauthorized：认证无效
- 403 Forbidden：权限不足
- 404 Not Found：资源不存在

现实类比：密码错了 → 再输同样的密码100次还是错
```

### ❌ ValueError/TypeError → 不重试

```
原因：编程错误，不会自己恢复
- ValueError("invalid literal"): 传了错误的参数
- TypeError("expected str, got int"): 类型不匹配

现实类比：把盐当糖放 → 重新放一次还是盐
```

### ✅ 未知异常 → 重试

```
原因：宁可多试一次，也不要因为未知错误导致系统崩溃
- 可能是第三方库的自定义异常
- 可能是临时性的奇怪问题

设计理念：保守策略 → 不确定就试试
```

## 特殊设计解读

### 为什么 429 不在 default_retry_on 里？

```
疑问：429 (Rate Limit) 明显应该重试，为什么源码里没特殊处理？

答案：429 本身就是 HTTP 4xx，会走 httpx.HTTPStatusError 判断，
      status_code=429 不满足 >= 500 的条件，所以默认不重试。

但是！OpenAI/Anthropic 等 SDK 自带 429 重试机制！
所以 LangGraph 不需要重复处理。

如果你需要在 LangGraph 层面处理 429，使用自定义 retry_on：
```

```python
def retry_with_429(exc: Exception) -> bool:
    if isinstance(exc, httpx.HTTPStatusError):
        return exc.response.status_code in [429, 500, 502, 503, 504]
    return default_retry_on(exc)

policy = RetryPolicy(retry_on=retry_with_429)
```

### 为什么 RuntimeError 不重试？

```
RuntimeError 家族包含：
- GraphRecursionError（图步数超限）→ 重试也会再次超限
- RecursionError（Python 递归超限）→ 代码逻辑问题
- NotImplementedError → 功能未实现

这些都是系统性问题，不会因为重试而消失。
```

### 为什么 OSError 不重试？

```
看起来矛盾：OSError 包含网络错误啊？

但注意：ConnectionError 已经被单独判断了！
Python 的异常层次是：
  OSError
  ├── ConnectionError  ← 已经被第一个 if 捕获（重试 ✅）
  ├── FileNotFoundError ← 文件不存在，重试无意义
  ├── PermissionError   ← 权限不足，重试无意义
  └── TimeoutError      ← 这里有争议...

所以 OSError 的排除主要影响的是文件系统相关错误，
网络连接错误已经被 ConnectionError 提前拦截了。
```

## 三种覆盖默认行为的方式

### 方式1：更严格 — 只重试特定异常

```python
# 只重试网络错误，其他一律不重试
policy = RetryPolicy(retry_on=ConnectionError)
```

### 方式2：更宽松 — 添加更多可重试异常

```python
def lenient_retry(exc: Exception) -> bool:
    # 先用默认逻辑
    if default_retry_on(exc):
        return True
    # 额外重试 TimeoutError
    if isinstance(exc, TimeoutError):
        return True
    return False

policy = RetryPolicy(retry_on=lenient_retry)
```

### 方式3：完全自定义

```python
def my_retry(exc: Exception) -> bool:
    """完全自定义的重试判断。"""
    # 只重试这些特定的异常
    if isinstance(exc, (ConnectionError, TimeoutError)):
        return True
    if isinstance(exc, httpx.HTTPStatusError):
        return exc.response.status_code in [429, 500, 502, 503, 504]
    return False

policy = RetryPolicy(retry_on=my_retry)
```

## 默认行为分类总结表

```
异常类型                  默认行为   原因
─────────────────────────────────────────────
ConnectionError          ✅ 重试    网络瞬态错误

HTTP 4xx                 ❌ 不重试  客户端错误
ValueError               ❌ 不重试  编程错误
TypeError                ❌ 不重试  编程错误
ArithmeticError          ❌ 不重试  计算错误
ImportError              ❌ 不重试  环境问题
LookupError              ❌ 不重试  编程错误
NameError                ❌ 不重试  编程错误
SyntaxError              ❌ 不重试  编程错误

OSError                  ❌ 不重试  文件/系统错误
StopIteration            ❌ 不重试  迭代逻辑
```

## 前端类比

```
default_retry_on ≈ Axios 的默认错误处理

// Axios 默认不重试任何错误
// 但你可以配置 axios-retry：
axiosRetry(axios, {
    retries: 3,
    retryCondition: (error) => {
        // ← 这就是 default_retry_on 的角色
        if (error.code === 'ECONNABORTED') return true;  // 超时
        if (error.response?.status >= 500) return true;   // 5xx
        return false;                                      // 其他不重试
    },
});
```
