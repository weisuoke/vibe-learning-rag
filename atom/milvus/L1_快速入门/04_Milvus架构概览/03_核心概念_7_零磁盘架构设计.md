## 一、原理讲解

### 1.1 什么是零磁盘架构？

**零磁盘架构（Zero-Disk Architecture）** 是 Milvus 2.6 的核心创新，指的是 **Worker 节点不依赖本地磁盘存储任何持久化数据**，所有数据直接写入对象存储（MinIO/S3）。

**核心理念：**
```
传统架构（有磁盘）：
数据写入 → 本地磁盘缓存 → 对象存储
         ↑ 需要管理磁盘空间、故障恢复

零磁盘架构（无磁盘）：
数据写入 → 直接写入对象存储
         ↑ 无需本地磁盘，完全云原生
```

### 1.2 为什么需要零磁盘架构？

#### 1.2.1 传统架构的问题

在 Milvus 2.5 及之前，系统依赖 **Kafka/Pulsar** 作为消息队列（MQ），存在以下问题：

```
传统架构（2.5 及之前）：
┌──────────┐
│  Client  │
└──────────┘
     ↓ 写入数据
┌──────────────┐
│ Proxy Node   │
└──────────────┘
     ↓ 发送到 MQ
┌──────────────┐
│ Kafka/Pulsar │ ← 需要本地磁盘存储日志
│ (消息队列)   │   - 磁盘空间管理复杂
└──────────────┘   - 故障恢复困难
     ↓ 消费数据      - 运维成本高
┌──────────────┐
│  Data Node   │ ← 从 MQ 读取数据
└──────────────┘
     ↓ 持久化
┌──────────────┐
│Object Storage│
└──────────────┘
```

**问题总结：**

1. **磁盘空间管理**
   - Kafka/Pulsar 需要本地磁盘存储日志
   - 磁盘空间不足时需要扩容或清理
   - 增加了运维复杂度

2. **故障恢复困难**
   - 磁盘故障导致数据丢失
   - 需要复杂的备份和恢复机制
   - 恢复时间长

3. **运维成本高**
   - 需要单独部署和维护 Kafka/Pulsar 集群
   - 监控磁盘使用情况
   - 处理磁盘故障

4. **资源浪费**
   - 数据在 MQ 和对象存储中重复存储
   - 磁盘空间占用大
   - 成本高

#### 1.2.2 零磁盘架构的优势

Milvus 2.6 引入 **Woodpecker**，实现零磁盘架构：

```
零磁盘架构（2.6）：
┌──────────┐
│  Client  │
└──────────┘
     ↓ 写入数据
┌──────────────┐
│ Proxy Node   │
└──────────────┘
     ↓ 直接写入对象存储
┌──────────────┐
│Streaming Node│ ← Woodpecker：零磁盘 WAL
│ (Woodpecker) │   - 日志直接写入对象存储
└──────────────┘   - 无需本地磁盘
     ↓ 订阅 WAL      - 简化运维
┌──────────────┐
│  Data Node   │ ← 从对象存储读取 WAL
└──────────────┘
     ↓ 已在对象存储
┌──────────────┐
│Object Storage│ ← 单一数据源
└──────────────┘
```

**优势总结：**

- ✅ **无需本地磁盘**：Worker 节点完全无状态
- ✅ **简化运维**：无需管理磁盘空间和故障恢复
- ✅ **降低成本**：无需重复存储数据
- ✅ **提升可靠性**：对象存储多副本，数据不丢失
- ✅ **云原生**：完全适配云环境，弹性扩展

### 1.3 Woodpecker：零磁盘 WAL 的实现

#### 1.3.1 什么是 Woodpecker？

**Woodpecker** 是 Milvus 2.6 引入的新组件，替代 Kafka/Pulsar，实现零磁盘的 Write-Ahead Log（WAL）。

**核心特性：**
- **直接写入对象存储**：WAL 日志直接写入 MinIO/S3
- **无需本地磁盘**：完全依赖对象存储
- **高性能**：利用对象存储的并行写入能力
- **高可靠**：对象存储多副本保证数据不丢失

#### 1.3.2 Woodpecker 的架构

```
Woodpecker 架构：
┌─────────────────────────────────────────────────────────┐
│                  Streaming Node (Woodpecker)            │
│                                                          │
│  ┌────────────────────────────────────────────────┐    │
│  │              WAL Manager                        │    │
│  │  - 接收写入请求                                 │    │
│  │  - 分配 WAL 序列号                              │    │
│  │  - 批量写入对象存储                             │    │
│  └────────────────────────────────────────────────┘    │
│                         ↓                                │
│  ┌────────────────────────────────────────────────┐    │
│  │              Object Storage Writer              │    │
│  │  - Multipart Upload（分片上传）                 │    │
│  │  - 并行写入多个 WAL 文件                        │    │
│  │  - 自动重试和错误处理                           │    │
│  └────────────────────────────────────────────────┘    │
│                         ↓                                │
│  ┌────────────────────────────────────────────────┐    │
│  │              Subscription Manager               │    │
│  │  - 管理 Data Node 的订阅                        │    │
│  │  - 推送 WAL 更新通知                            │    │
│  │  - 处理订阅者的读取请求                         │    │
│  └────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
                          ↓
        ┌─────────────────┴─────────────────┐
        ↓                                   ↓
┌──────────────────┐              ┌──────────────────┐
│ Object Storage   │              │   Data Node      │
│ (WAL 文件)       │              │ (订阅 WAL)       │
└──────────────────┘              └──────────────────┘
```

#### 1.3.3 WAL 文件格式

```
对象存储中的 WAL 文件结构：
s3://milvus-bucket/wal/
├── collection_1/
│   ├── vchannel_0/
│   │   ├── wal_0000000001.log  # WAL 文件
│   │   ├── wal_0000000002.log
│   │   └── wal_0000000003.log
│   └── vchannel_1/
│       ├── wal_0000000001.log
│       └── wal_0000000002.log
└── collection_2/
    └── vchannel_0/
        └── wal_0000000001.log
```

**WAL 文件内容：**
```
WAL 文件格式（简化）：

│ - Magic Number: 0x4D494C5655        │

```

### 1.4 零磁盘架构的数据流

#### 1.4.1 写入流程

```
数据写入流程（零磁盘）：
┌──────────┐
│  Client  │
└──────────┘
     ↓ insert(vectors)
┌──────────────┐
│ Proxy Node   │ ← 1. 接收请求，路由到 VChannel
└──────────────┘
     ↓ 2. 发送到 Streaming Node
┌──────────────┐
│Streaming Node│ ← 3. 分配序列号，批量缓冲
│ (Woodpecker) │
└──────────────┘
     ↓ 4. 直接写入对象存储（无本地磁盘）
┌──────────────┐
│Object Storage│ ← 5. WAL 文件持久化
│ (WAL 文件)   │
└──────────────┘
     ↓ 6. 通知 Data Node
┌──────────────┐
│  Data Node   │ ← 7. 从对象存储读取 WAL
└──────────────┘
     ↓ 8. 处理数据，生成 Binlog
┌──────────────┐
│Object Storage│ ← 9. Binlog 持久化
│ (Binlog)     │
└──────────────┘
```

**关键步骤：**

1. **Proxy Node 路由**：根据 VChannel 分片规则路由请求
2. **Streaming Node 缓冲**：在内存中批量缓冲写入请求
3. **直接写入对象存储**：使用 Multipart Upload 并行写入
4. **通知订阅者**：通知 Data Node 有新的 WAL 数据
5. **Data Node 消费**：从对象存储读取 WAL 并处理

#### 1.4.2 读取流程

```
数据读取流程（零磁盘）：
┌──────────┐
│  Client  │
└──────────┘
     ↓ search(vector)
┌──────────────┐
│ Proxy Node   │ ← 1. 接收请求，路由到 Query Node
└──────────────┘
     ↓ 2. 分发查询
┌──────────────┐
│ Query Node   │ ← 3. 查询历史数据（已索引）
└──────────────┘
     ↓ 4. 从对象存储加载索引（无本地磁盘）
┌──────────────┐
│Object Storage│ ← 5. 索引文件
│ (Index)      │
└──────────────┘
     ↓ 6. 查询实时数据
┌──────────────┐
│Streaming Node│ ← 7. 查询 Growing Segment
│ (Woodpecker) │
└──────────────┘
     ↓ 8. 从对象存储读取 WAL（无本地磁盘）
┌──────────────┐
│Object Storage│ ← 9. WAL 文件
│ (WAL)        │
└──────────────┘
     ↓ 10. 合并结果
┌──────────────┐
│ Proxy Node   │ ← 11. 返回结果
└──────────────┘
     ↓ 12. 返回给客户端
┌──────────┐
│  Client  │
└──────────┘
```

### 1.5 零磁盘架构 vs 传统架构

| 维度 | 传统架构（2.5） | 零磁盘架构（2.6） |
|------|----------------|------------------|
| **消息队列** | Kafka/Pulsar | Woodpecker |
| **本地磁盘** | 需要（存储 MQ 日志） | 不需要 |
| **数据存储** | MQ + 对象存储（重复） | 对象存储（单一） |
| **运维复杂度** | 高（需管理 MQ 集群） | 低（无需 MQ） |
| **故障恢复** | 复杂（磁盘故障） | 简单（对象存储多副本） |
| **成本** | 高（磁盘 + MQ） | 低（仅对象存储） |
| **扩展性** | 受限（磁盘容量） | 无限（对象存储） |
| **云原生** | 部分（依赖磁盘） | 完全（无磁盘依赖） |

### 1.6 零磁盘架构的权衡

#### 1.6.1 优势

1. **简化运维**
   - 无需管理本地磁盘
   - 无需部署和维护 Kafka/Pulsar
   - 监控和故障排查更简单

2. **降低成本**
   - 无需购买和维护磁盘
   - 数据不重复存储
   - 对象存储按需付费

3. **提升可靠性**
   - 对象存储多副本
   - 无磁盘故障风险
   - 数据持久性更高

4. **完全云原生**
   - Worker 节点完全无状态
   - 可随意扩缩容
   - 适配所有云平台

#### 1.6.2 权衡

1. **网络延迟**
   - 直接写入对象存储，网络延迟略高于本地磁盘
   - 通过批量写入和 Multipart Upload 缓解

2. **对象存储依赖**
   - 完全依赖对象存储的可用性
   - 需要选择高可用的对象存储服务

3. **成本模型变化**
   - 从磁盘成本转为对象存储成本
   - 需要根据实际使用情况评估

## 二、手写实现：简化版零磁盘 WAL

下面我们用 Python 实现一个简化版的零磁盘 WAL，展示 Woodpecker 的核心原理。

### 2.1 基础数据结构

```python
from dataclasses import dataclass
from typing import List, Dict, Optional
from enum import Enum
import time
import json
import os

class OperationType(Enum):
    """操作类型"""
    INSERT = "insert"
    DELETE = "delete"
    UPDATE = "update"

@dataclass
class WALRecord:
    """WAL 记录"""
    sequence_number: int
    timestamp: int
    operation: OperationType
    collection_id: int
    vchannel_id: int
    data: Dict

@dataclass
class WALFile:
    """WAL 文件元数据"""
    file_path: str
    start_sequence: int
    end_sequence: int
    num_records: int
    size_bytes: int
```

### 2.2 零磁盘 WAL Writer

```python
class ZeroDiskWALWriter:
    """零磁盘 WAL 写入器：直接写入对象存储"""

    def __init__(self, object_storage, collection_id: int, vchannel_id: int):
        self.object_storage = object_storage
        self.collection_id = collection_id
        self.vchannel_id = vchannel_id
        self.next_sequence = 1
        self.buffer: List[WALRecord] = []
        self.buffer_size_limit = 100  # 批量写入阈值
        self.current_file_id = 1

        print(f"[ZeroDiskWAL] Initialized for collection {collection_id}, vchannel {vchannel_id}")
        print(f"[ZeroDiskWAL] ✅ Zero-Disk: No local disk required!")

    def write(self, operation: OperationType, data: Dict) -> int:
        """写入 WAL 记录"""
        # 1. 创建 WAL 记录
        record = WALRecord(
            sequence_number=self.next_sequence,
            timestamp=int(time.time() * 1000),
            operation=operation,
            collection_id=self.collection_id,
            vchannel_id=self.vchannel_id,
            data=data
        )

        self.next_sequence += 1

        # 2. 添加到缓冲区
        self.buffer.append(record)

        print(f"[ZeroDiskWAL] Buffered record #{record.sequence_number} ({operation.value})")

        # 3. 检查是否需要 Flush
        if len(self.buffer) >= self.buffer_size_limit:
            self.flush()

        return record.sequence_number

    def flush(self):
        """Flush 缓冲区到对象存储（零磁盘）"""
        if not self.buffer:
            return

        print(f"\n[ZeroDiskWAL] Flushing {len(self.buffer)} records to object storage...")
        print(f"[ZeroDiskWAL] ✅ Zero-Disk: Writing directly to object storage (no local disk)")

        # 1. 序列化 WAL 记录
        wal_data = {
            "header": {
                "magic": "MILVUS",
                "version": "2.6",
                "collection_id": self.collection_id,
                "vchannel_id": self.vchannel_id
            },
            "records": [
                {
                    "sequence_number": r.sequence_number,
                    "timestamp": r.timestamp,
                    "operation": r.operation.value,
                    "data": r.data
                }
                for r in self.buffer
            ]
        }

        wal_json = json.dumps(wal_data, indent=2)
        wal_bytes = wal_json.encode('utf-8')

        # 2. 生成对象存储路径
        wal_key = f"wal/collection_{self.collection_id}/vchannel_{self.vchannel_id}/wal_{self.current_file_id:010d}.log"

        # 3. 直接写入对象存储（零磁盘！）
        start_seq = self.buffer[0].sequence_number
        end_seq = self.buffer[-1].sequence_number

        self.object_storage.put_object(wal_key, wal_bytes)

        # 4. 创建 WAL 文件元数据
        wal_file = WALFile(
            file_path=f"s3://milvus/{wal_key}",
            start_sequence=start_seq,
            end_sequence=end_seq,
            num_records=len(self.buffer),
            size_bytes=len(wal_bytes)
        )

        print(f"[ZeroDiskWAL] ✅ Flushed to {wal_file.file_path}")
        print(f"[ZeroDiskWAL]    Sequence range: {start_seq} - {end_seq}")
        print(f"[ZeroDiskWAL]    Size: {len(wal_bytes) / 1024:.2f} KB")

        # 5. 清空缓冲区
        self.buffer = []
        self.current_file_id += 1
```

### 2.3 零磁盘 WAL Reader

```python
class ZeroDiskWALReader:
    """零磁盘 WAL 读取器：从对象存储读取"""

    def __init__(self, object_storage, collection_id: int, vchannel_id: int):
        self.object_storage = object_storage
        self.collection_id = collection_id
        self.vchannel_id = vchannel_id

        print(f"[ZeroDiskWAL] Reader initialized for collection {collection_id}, vchannel {vchannel_id}")
        print(f"[ZeroDiskWAL] ✅ Zero-Disk: Reading directly from object storage (no local disk)")

    def read_wal_file(self, file_id: int) -> Optional[List[WALRecord]]:
        """从对象存储读取 WAL 文件"""
        # 1. 生成对象存储路径
        wal_key = f"wal/collection_{self.collection_id}/vchannel_{self.vchannel_id}/wal_{file_id:010d}.log"

        print(f"\n[ZeroDiskWAL] Reading WAL file from object storage...")
        print(f"[ZeroDiskWAL] ✅ Zero-Disk: No local disk cache needed")

        # 2. 从对象存储读取（零磁盘！）
        wal_bytes = self.object_storage.get_object(wal_key)

        if not wal_bytes:
            print(f"[ZeroDiskWAL] WAL file not found: {wal_key}")
            return None

        # 3. 反序列化
        wal_json = wal_bytes.decode('utf-8')
        wal_data = json.loads(wal_json)

        # 4. 解析 WAL 记录
        records = []
        for r in wal_data["records"]:
            record = WALRecord(
                sequence_number=r["sequence_number"],
                timestamp=r["timestamp"],
                operation=OperationType(r["operation"]),
                collection_id=self.collection_id,
                vchannel_id=self.vchannel_id,
                data=r["data"]
            )
            records.append(record)

        print(f"[ZeroDiskWAL] ✅ Read {len(records)} records from object storage")
        return records

    def subscribe(self, start_sequence: int = 1):
        """订阅 WAL（从指定序列号开始）"""
        print(f"\n[ZeroDiskWAL] Subscribing to WAL from sequence {start_sequence}")

        file_id = 1
        while True:
            records = self.read_wal_file(file_id)

            if not records:
                break

            # 过滤出需要的记录
            filtered_records = [r for r in records if r.sequence_number >= start_sequence]

            for record in filtered_records:
                yield record

            file_id += 1

### 2.4 完整示例

```python
# 需要先导入 ObjectStorage（从之前的 Storage 层实现）
from typing import List
import numpy as np

def zero_disk_wal_demo():
    """演示零磁盘 WAL 的完整工作流程"""

    print("=" * 60)
    print("Zero-Disk WAL Demo (Milvus 2.6 Woodpecker)")
    print("=" * 60)

    # 1. 初始化对象存储（模拟）
    from dataclasses import dataclass
    import os

    class SimpleObjectStorage:
        def __init__(self):
            self.storage = {}

        def put_object(self, key: str, data: bytes):
            self.storage[key] = data
            return f"s3://milvus/{key}"

        def get_object(self, key: str):
            return self.storage.get(key)

    object_storage = SimpleObjectStorage()

    # 2. 创建零磁盘 WAL Writer
    print("\n[Step 1] Creating Zero-Disk WAL Writer")
    print("-" * 60)

    collection_id = 1
    vchannel_id = 0

    wal_writer = ZeroDiskWALWriter(object_storage, collection_id, vchannel_id)

    # 3. 写入数据（模拟插入操作）
    print("\n[Step 2] Writing Data to Zero-Disk WAL")
    print("-" * 60)

    for i in range(150):  # 超过缓冲区限制，触发 Flush
        data = {
            "id": i,
            "vector": [float(x) for x in np.random.rand(128)],
            "metadata": {"text": f"document_{i}"}
        }
        wal_writer.write(OperationType.INSERT, data)

    # 4. 手动 Flush 剩余数据
    print("\n[Step 3] Flushing Remaining Data")
    print("-" * 60)
    wal_writer.flush()

    # 5. 创建零磁盘 WAL Reader
    print("\n[Step 4] Creating Zero-Disk WAL Reader")
    print("-" * 60)

    wal_reader = ZeroDiskWALReader(object_storage, collection_id, vchannel_id)

    # 6. 订阅并读取 WAL
    print("\n[Step 5] Subscribing to Zero-Disk WAL")
    print("-" * 60)

    record_count = 0
    for record in wal_reader.subscribe(start_sequence=1):
        if record_count < 5:  # 只打印前 5 条
            print(f"  Record #{record.sequence_number}: {record.operation.value}, ID={record.data.get('id')}")
        record_count += 1

    print(f"\n  Total records read: {record_count}")

    # 7. 显示零磁盘优势
    print("\n[Step 6] Zero-Disk Architecture Benefits")
    print("-" * 60)
    print("✅ No local disk required")
    print("✅ All data stored in object storage")
    print("✅ Worker nodes are completely stateless")
    print("✅ Simplified operations and maintenance")
    print("✅ Lower cost (no disk management)")

    print("\n" + "=" * 60)
    print("Zero-Disk WAL Demo Completed")
    print("=" * 60)


if __name__ == "__main__":
    zero_disk_wal_demo()

### 3.2 场景 2：RAG 系统的实时数据写入

**问题：** RAG 系统需要实时插入新文档，零磁盘架构如何保证性能？

**零磁盘架构的作用：**
- **批量写入**：Woodpecker 在内存中批量缓冲，减少对象存储写入次数
- **Multipart Upload**：大文件并行上传，提升吞吐量
- **异步持久化**：写入对象存储不阻塞客户端响应

**实际代码示例：**

```python
from pymilvus import connections, Collection
import time

# 连接到 Milvus 2.6（零磁盘架构）
connections.connect(host="localhost", port="19530")

def benchmark_zero_disk_insert(collection_name: str, num_docs: int):
    """测试零磁盘架构的写入性能"""

    collection = Collection(collection_name)

    # 准备测试数据
    documents = [
        {
            "text": f"Document {i}",
            "embedding": [float(x) for x in np.random.rand(768)]
        }
        for i in range(num_docs)
    ]

    # 批量插入
    print(f"Inserting {num_docs} documents...")
    start_time = time.time()

    batch_size = 1000
    for i in range(0, num_docs, batch_size):
        batch = documents[i:i+batch_size]
        texts = [doc["text"] for doc in batch]
        embeddings = [doc["embedding"] for doc in batch]

        collection.insert([texts, embeddings])

    elapsed = time.time() - start_time
    throughput = num_docs / elapsed

    print(f"✅ Inserted {num_docs} documents in {elapsed:.2f} seconds")
    print(f"   Throughput: {throughput:.2f} docs/sec")
    print(f"   Zero-Disk: All data written directly to object storage")

    return throughput

# 示例：插入 10 万文档
benchmark_zero_disk_insert("rag_knowledge_base", 100000)
```

**零磁盘架构的优势：**
- 批量写入减少对象存储的写入次数
- Multipart Upload 提升大文件上传速度
- 无需担心本地磁盘空间不足

### 3.4 场景 4：RAG 系统的成本优化

**问题：** RAG 系统如何降低存储成本？

**零磁盘架构的作用：**
- **无需磁盘**：节省磁盘购买和维护成本
- **对象存储分层**：热数据和冷数据使用不同存储类别
- **按需付费**：只为实际使用的存储付费

**实际代码示例：**

```python
def calculate_cost_savings():
    """计算零磁盘架构的成本节省"""

    print("Cost Comparison: Traditional vs Zero-Disk")
    print("=" * 60)

    # 假设场景：10 个 Worker 节点，每个节点 1TB 数据
    num_nodes = 10
    data_per_node_tb = 1

    # 传统架构成本
    print("\n[Traditional Architecture (2.5)]")
    print("-" * 60)

    # 1. 本地磁盘成本
    disk_cost_per_tb = 100  # USD/月
    kafka_disk_cost = num_nodes * data_per_node_tb * disk_cost_per_tb
    print(f"  Kafka/Pulsar disk: ${kafka_disk_cost}/month")

    # 2. 对象存储成本（重复存储）
    object_storage_cost_per_tb = 23  # AWS S3 标准存储
    duplicate_storage_cost = num_nodes * data_per_node_tb * object_storage_cost_per_tb
    print(f"  Object storage (duplicate): ${duplicate_storage_cost}/month")

    # 3. 运维成本
    ops_cost = 500  # 磁盘管理、监控、故障处理
    print(f"  Operations: ${ops_cost}/month")

    traditional_total = kafka_disk_cost + duplicate_storage_cost + ops_cost
    print(f"\n  Total: ${traditional_total}/month")

    # 零磁盘架构成本
    print("\n[Zero-Disk Architecture (2.6)]")
    print("-" * 60)

    # 1. 无本地磁盘成本
    print(f"  Local disk: $0/month (✅ Zero-Disk)")

    # 2. 对象存储成本（单一存储）
    single_storage_cost = num_nodes * data_per_node_tb * object_storage_cost_per_tb
    print(f"  Object storage (single): ${single_storage_cost}/month")

    # 3. 运维成本（降低）
    reduced_ops_cost = 100  # 无需磁盘管理
    print(f"  Operations: ${reduced_ops_cost}/month")

    zero_disk_total = single_storage_cost + reduced_ops_cost
    print(f"\n  Total: ${zero_disk_total}/month")

    # 成本节省
    print("\n[Cost Savings]")
    print("-" * 60)
    savings = traditional_total - zero_disk_total
    savings_percent = (savings / traditional_total) * 100

    print(f"  Monthly savings: ${savings}")
    print(f"  Savings percentage: {savings_percent:.1f}%")
    print(f"  Annual savings: ${savings * 12}")

# 示例
calculate_cost_savings()

## 四、总结

### 4.1 零磁盘架构的核心价值

1. **完全云原生**：Worker 节点无状态，适配所有云平台
2. **简化运维**：无需管理本地磁盘，降低运维复杂度
3. **降低成本**：无需磁盘购买和维护，成本降低 80%+
4. **提升可靠性**：数据存储在对象存储，多副本保证不丢失
5. **快速恢复**：节点故障后可立即替换，无需数据迁移

### 4.2 Milvus 2.6 的创新：Woodpecker

- **替代 Kafka/Pulsar**：无需部署和维护消息队列
- **直接写入对象存储**：WAL 日志直接写入 MinIO/S3
- **Multipart Upload**：大文件并行上传，提升吞吐量
- **零磁盘设计**：Worker 节点完全无状态

### 4.3 对 RAG 开发的影响

- **更简单的部署**：无需配置磁盘，快速启动
- **更低的成本**：无需磁盘管理，成本降低 80%+
- **更高的可靠性**：数据在对象存储，故障不丢失
- **更快的恢复**：节点故障后立即替换，无需数据迁移

### 4.4 类比总结

**前端类比：**
- 传统架构 = 本地存储（LocalStorage）

- 传统架构 = 在家存储文件（需要管理硬盘）
