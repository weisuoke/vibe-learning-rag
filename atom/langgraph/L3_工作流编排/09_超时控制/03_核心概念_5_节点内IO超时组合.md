# 节点内 I/O 超时组合

> **核心概念 5/6**
> **关键词**：asyncio.wait_for、httpx.Timeout、inner deadline、outer budget

---

## 这是最容易被低估、却最有生产价值的概念

真正把系统从“会 timeout”变成“timeout 后仍然可控”的关键，几乎都在 node 内部。

因为工作流的慢，常常不是 LangGraph 自己慢，而是它在等：
- HTTP 服务；
- LLM SDK；
- 数据库；
- 搜索引擎；
- 队列结果；
- 第三方工具。

---

## 一句话定义

**节点内 I/O 超时组合，就是在 node 内先为具体外部调用设置细粒度 deadline，再由 `step_timeout` 在外层兜住整轮 orchestration。**

---

## 为什么不能只靠 `step_timeout`？

因为 `step_timeout` 只知道：

> “当前 step 已经等太久了。”

它并不知道：
- 是 connect 卡住了；
- 还是 read 卡住了；
- 还是连接池没资源；
- 还是某个异步工具链中的第二步太慢。

这些都必须由 node 内的更细粒度 timeout 来表达。

---

## 第一层：`httpx.Timeout`

官方 HTTPX 文档给出了四类关键预算：

```python
timeout = httpx.Timeout(
    10.0,
    connect=5.0,
    read=30.0,
    write=10.0,
    pool=5.0,
)
```

这意味着你可以区分：
- 建链慢；
- 读响应慢；
- 写请求慢；
- 池资源不足。

这比一个粗糙的“总 timeout=10”信息量高得多。

[来源: atom/langgraph/L3_工作流编排/09_超时控制/reference/context7_httpx_02.md]

---

## 第二层：`asyncio.wait_for()`

如果 node 内不是单次 HTTP，而是一串协程逻辑，推荐再包一层：

```python
result = await asyncio.wait_for(do_complex_async_work(), timeout=8.0)
```

这样控制的是：

- 一个逻辑动作；
- 一个工具组合；
- 一个多步外部调用子流程。

它不关心底层是 HTTP 还是其他 awaitable，只关心“这整件事最多值不值得等 8 秒”。

---

## 第三层：`step_timeout`

有了 node 内 deadline 之后，再给图加上外层：

```python
graph.step_timeout = 10
```

这时预算层级变成：

```text
http connect/read/write/pool < node coroutine budget < graph step budget
```

这是一种典型的“内紧外松”结构。

---

## 一个合理组合长什么样？

### 示例：检索节点

```python
timeout = httpx.Timeout(6.0, connect=1.5, read=4.5, write=2.0, pool=1.0)

async with httpx.AsyncClient(timeout=timeout) as client:
    response = await asyncio.wait_for(
        client.get(url),
        timeout=7.0,
    )

graph.step_timeout = 9.0
```

解释：
- HTTP 细粒度 timeout 最早；
- 整个检索动作再有一个协程 budget；
- 整个 step 再给一点兜底空间。

---

## 为什么要“内紧外松”？

因为你希望最接近根因的层最先报错。

### 错误顺序应该是：
1. 连接建立异常慢 → `connect timeout`
2. 读取响应异常慢 → `read timeout`
3. 整个检索子流程太慢 → `wait_for timeout`
4. 整个 step 都拖住了 → `step_timeout`

如果顺序反过来，你会得到很多“看起来是 graph timeout，实际上是底层 I/O 问题”的模糊异常。

---

## fallback 应该放在哪一层？

通常推荐放在 node 层或 node 之后的路由层，而不是等整个 graph timeout 之后再想办法。

### 原因
- node 层最清楚失败原因；
- graph 层知道“超时了”，但往往不知道应降级成什么；
- 越早 fallback，越能保住用户体验。

### 一个简单模式

```python
async def search_node(state):
    try:
        result = await asyncio.wait_for(remote_search(), timeout=2.0)
        return {"result": result, "timed_out": False}
    except asyncio.TimeoutError:
        return {"result": "use_cache", "timed_out": True}
```

然后由条件边决定走正常路径还是缓存路径。

## `runtime.context` 可以做什么？

本地源码测试表明，LangGraph 支持 `context_schema`，可以把 timeout 等预算信息放进 `runtime.context`。

这不等于 LangGraph 会自动执行这些 timeout，但它非常适合：
- 把预算从代码硬编码中抽离；
- 每次调用动态传入；
- 让节点内逻辑按上下文决定用哪个 timeout。

例如：

```python
class Context(BaseModel):
    api_key: str
    timeout: int = 30
```

node 内读取：

```python
runtime.context.timeout
```

## 常见错误搭配

### 错误搭配 1：只有 `step_timeout`
后果：问题暴露太晚。

### 错误搭配 2：HTTP timeout 比 graph timeout 更宽
后果：外层经常先超时，定位困难。

### 错误搭配 3：timeout 后没有 fallback
后果：系统只会更快失败，不会更稳。

### 错误搭配 4：把 timeout 硬编码在 5 个地方
后果：调参和排障都很痛苦。

## 推荐实战模板

```python
import asyncio
import httpx


async def robust_call(url: str, node_timeout: float) -> dict:
    timeout = httpx.Timeout(6.0, connect=1.5, read=4.0, write=2.0, pool=1.0)

    async with httpx.AsyncClient(timeout=timeout) as client:
        response = await asyncio.wait_for(client.get(url), timeout=node_timeout)
        response.raise_for_status()
        return response.json()
```

这个模板的价值在于，它明确区分：
- HTTP 分段预算；
- node 总预算。

外层 graph 还可以再有 `step_timeout`。

## 一句话收束

**真正稳的 LangGraph timeout 设计，不是把预算全堆给 `step_timeout`，而是让每一层只负责最适合它负责的等待。**
