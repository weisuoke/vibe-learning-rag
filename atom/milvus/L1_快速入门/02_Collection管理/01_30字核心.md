# Collection管理 - 30字核心

---

## 核心定义

**Collection 是 Milvus 中存储向量数据的基本单位，定义了数据的结构（Schema）和组织方式，是向量检索的基础容器。**

---

## 为什么这30字很重要？

### 1. 抓住本质
Collection 不是简单的"表"，而是**专为向量检索优化的数据容器**：
- 支持向量字段（FLOAT_VECTOR、FLOAT16_VECTOR等）
- 支持标量字段（INT、VARCHAR、JSON、ARRAY等）
- 定义了数据的结构和约束

### 2. 理解核心价值
Collection 是 Milvus 的**数据组织单元**：
- **结构定义**：通过 Schema 定义字段类型和约束
- **数据隔离**：不同 Collection 的数据完全隔离
- **检索单位**：检索操作在 Collection 级别执行

### 3. 联系 RAG 应用
在 RAG 系统中，Collection 是**知识库的物理载体**：
- 文档问答系统：一个 Collection 存储所有文档的向量
- 多租户知识库：每个租户一个 Collection（Milvus 2.6 支持 100K collections）
- 多模态检索：一个 Collection 存储文本向量 + 图像向量

---

## Milvus 2.6 核心特性

### 100K Collections
- **能力**：单个 Milvus 实例支持 10 万个 Collection
- **场景**：大规模多租户 SaaS 应用
- **价值**：每个用户/租户独立 Collection，数据完全隔离

### Dynamic Schema
- **能力**：运行时动态添加字段，无需重建 Collection
- **场景**：需求变化频繁的应用
- **价值**：灵活应对业务变化，降低维护成本

### 新字段类型
- **FLOAT16_VECTOR / BFLOAT16_VECTOR**：低精度向量，节省 50% 存储
- **ARRAY**：数组类型，存储多值字段（如标签列表）
- **增强 JSON**：支持 JSON Path Index，高效查询嵌套字段

## 快速记忆

```
Collection = 数据容器
           ├── Schema（结构定义）
           ├── 向量字段（核心）
           ├── 标量字段（元数据）
           └── 索引（加速检索）
```

---

## 一句话类比

**前端类比：** Collection 就像 TypeScript 的接口定义 + MongoDB 的 Collection，既定义了数据结构，又是数据的存储容器。

**日常类比：** Collection 就像图书馆的一个书架，书架有固定的格式（书名、作者、ISBN），每本书都按这个格式存放。
