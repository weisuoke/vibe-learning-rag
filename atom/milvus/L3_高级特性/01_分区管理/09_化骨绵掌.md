## 卡片 1：Partition 的直觉理解

**一句话**：Partition 是 Collection 的子集，就像图书馆的书架分层，只在特定层找书，不需要翻遍整个书架。

**举例**：
```python
# Collection = 图书馆的书架
# Partition = 书架的不同层
collection = Collection("library")
collection.create_partition("2026_books")  # 第1层：2026年的书
collection.create_partition("2025_books")  # 第2层：2025年的书

# 只在第1层找书
results = collection.search(
    data=[query_vector],
    partition_names=["2026_books"]  # 只搜索第1层
)
```

**应用**：多租户场景、时间序列数据、热冷数据管理

---

## 卡片 2：Partition 的形式化定义

**一句话**：Partition 是 Collection 的子集，所有 Partition 的并集等于 Collection。

**形式化**：
```
Collection C = {entity₁, entity₂, ..., entityₙ}
Partition P ⊆ C

特性：
- Partition 是 Collection 的子集
- 所有 Partition 的并集 = Collection
- Partition 之间可以不相交（推荐）
```

**应用**：理解分区的数学本质，设计分区策略

---

## 卡片 3：Partition Key 自动分区

**一句话**：Partition Key 通过标记字段为 `is_partition_key=True`，Milvus 自动根据该字段值进行分区，支持百万级租户。

**举例**：
```python
# 标记 tenant_id 为 Partition Key
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True)  # 自动分区
    ]
)

# Milvus 自动将数据路由到 16 个物理分区
# 支持数百万个 tenant_id 值
```

**应用**：大规模多租户场景（> 1,024 个租户）

---

## 卡片 4：手动分区 vs Partition Key

**一句话**：手动分区提供物理隔离和热冷数据管理，Partition Key 提供自动管理和最高可扩展性。

**对比**：
| 维度 | 手动分区 | Partition Key |
|------|----------|---------------|
| 管理方式 | 手动创建 | 自动管理 |
| 最大数量 | 1,024 个 | 支持百万租户 |
| 数据隔离 | 物理隔离 | 逻辑隔离 |
| 热冷数据管理 | 支持 | 不支持 |

**应用**：根据租户数量和隔离需求选择合适的策略

---

## 卡片 5：分区的生命周期

**一句话**：分区的生命周期包括创建、加载、使用、释放、删除五个阶段。

**流程**：
```python
# 1. 创建
collection.create_partition("partition_a")

# 2. 加载（数据需要加载到内存才能检索）
collection.load(partition_names=["partition_a"])

# 3. 使用（搜索、查询）
results = collection.search(data=[query_vector], partition_names=["partition_a"])

# 4. 释放（从内存中释放）
collection.release(partition_names=["partition_a"])

# 5. 删除（删除分区及其数据）
collection.drop_partition("partition_a")
```

**应用**：热冷数据管理、资源优化

---

## 卡片 6：多租户策略选择

**一句话**：根据租户数量、隔离需求、Schema 灵活性、RBAC 需求选择合适的多租户策略。

**决策树**：
```
租户数量 < 10 且需要 RBAC
  → Database-level

租户数量 < 100 且需要独立 Schema
  → Collection-level

租户数量 < 1,024 且需要热冷数据管理
  → Partition-level

租户数量 > 1,024 或需要自动管理
  → Partition Key-level（推荐）
```

**应用**：SaaS 应用、RAG 系统、时间序列数据

---

## 卡片 7：性能优化关键数据

**一句话**：使用 Partition Key 可以避免 99% 无效扫描，延迟从秒级降至毫秒级。

**关键数据**：
- **99% 无效扫描避免**：只扫描相关分区
- **延迟优化**：从秒级降至毫秒级
- **推荐分区数**：100-200 个物理分区（Partition Key）
- **推荐分区大小**：20-100K 行（手动分区）
- **最大分区数**：1,024 个（手动分区）

**应用**：性能优化、容量规划

---

## 卡片 8：Partition Key 隔离特性

**一句话**：启用 Partition Key 隔离后，Milvus 为每个 Partition Key 值创建独立的 HNSW 图，进一步提升性能。

**配置**：
```python
# 启用 Partition Key 隔离
collection = Collection(
    name="my_collection",
    schema=schema,
    properties={"partitionkey.isolation": True}
)

# 限制：
# 1. 仅支持 HNSW 索引
# 2. 搜索时必须包含单个 Partition Key 值
```

**应用**：大规模多租户场景的性能优化

---

## 卡片 9：热冷数据管理

**一句话**：通过手动分区实现热冷数据分离，只加载热数据到内存，节省资源。

**示例**：
```python
# 按月份分区
collection.create_partition("2026_02")  # 热数据
collection.create_partition("2026_01")  # 热数据
collection.create_partition("2025_12")  # 冷数据

# 只加载热数据
collection.load(partition_names=["2026_02", "2026_01"])

# 按需加载冷数据
collection.load(partition_names=["2025_12"])
# 使用后释放
collection.release(partition_names=["2025_12"])
```

**应用**：时间序列数据、日志数据、归档数据

---

## 卡片 10：分区最佳实践

**一句话**：选择合适的分区策略，配置合理的分区数量，监控性能并及时调优。

**最佳实践**：
1. **策略选择**：根据租户数量选择 Partition Key 或手动分区
2. **分区数量**：Partition Key 推荐 100-200 个物理分区，手动分区控制在 1,000 以内
3. **分区大小**：每个分区 20-100K 行
4. **性能优化**：启用 Partition Key 隔离，使用 HNSW 索引
5. **监控调优**：监控 QPS、延迟、内存使用、分区负载

**应用**：生产环境部署、性能调优
