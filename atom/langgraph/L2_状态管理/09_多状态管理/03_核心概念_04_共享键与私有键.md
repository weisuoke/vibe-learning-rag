# 核心概念4：共享键与私有键

> 父子图通过"同名键"自动通信——名字相同的字段自动传递，名字不同的字段各管各的。这是 LangGraph 子图通信的核心机制。

## 1. 什么是共享键与私有键？

### 核心问题：父子图的状态怎么传递？

当你把一个编译好的子图直接作为节点添加到父图时，两个图有各自独立的 State 定义。问题来了：

```
父图 State: { foo: str }
子图 State: { foo: str, bar: str }

子图怎么拿到父图的 foo？
子图的 bar 会不会污染父图？
子图修改了 foo，父图能看到吗？
```

LangGraph 的答案非常简洁：**看名字**。

### 一句话定义

**共享键是父子图 Schema 中同名的字段，自动双向传递；私有键是只存在于子图中的字段，对父图不可见。**

### 双重类比

**前端类比：React 父子组件的 props 传递**

```javascript
// 父组件有 { name, age }
// 子组件声明 props: { name, color }
// name 是共享的 → 自动从父传到子
// color 是子组件私有的 → 父组件看不到
// age 没被子组件声明 → 子组件拿不到
```

父组件通过 props 把同名属性传给子组件，子组件内部的 state 对父组件不可见。LangGraph 的共享键机制几乎一模一样。

**日常生活类比：总公司和分公司的共享文件夹**

总公司和分公司各有自己的文件系统。但有一个"共享文件夹"（同名键），总公司放进去的文件分公司能看到，分公司修改后总公司也能看到。而分公司自己的内部文件夹（私有键），总公司完全看不到。

## 2. 基本用法：直接添加子图

### 完整示例

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


# ===== 第一步：定义子图 =====
class SubgraphState(TypedDict):
    foo: str  # 共享键：与父图同名
    bar: str  # 私有键：只在子图内部使用


def subgraph_node_1(state: SubgraphState):
    """子图节点1：写入私有键"""
    return {"bar": "bar"}


def subgraph_node_2(state: SubgraphState):
    """子图节点2：读取私有键，写入共享键"""
    # bar 是子图内部的中间结果
    # foo 是要传回父图的最终结果
    return {"foo": state["foo"] + state["bar"]}


# 构建并编译子图
subgraph_builder = StateGraph(SubgraphState)
subgraph_builder.add_node(subgraph_node_1)
subgraph_builder.add_node(subgraph_node_2)
subgraph_builder.add_edge(START, "subgraph_node_1")
subgraph_builder.add_edge("subgraph_node_1", "subgraph_node_2")
subgraph_builder.add_edge("subgraph_node_2", END)
subgraph = subgraph_builder.compile()


# ===== 第二步：定义父图 =====
class ParentState(TypedDict):
    foo: str  # 与子图的 foo 同名 → 共享键


def node_1(state: ParentState):
    return {"foo": "hi! " + state["foo"]}


# 构建父图，直接把编译后的子图作为节点
builder = StateGraph(ParentState)
builder.add_node("node_1", node_1)
builder.add_node("node_2", subgraph)  # 关键：直接传入编译后的子图
builder.add_edge(START, "node_1")
builder.add_edge("node_1", "node_2")
builder.add_edge("node_2", END)
graph = builder.compile()

# ===== 第三步：执行 =====
result = graph.invoke({"foo": "foo"})
print(result)
# {'foo': 'hi! foobar'}
```

### 执行流程详解

```
输入: {"foo": "foo"}
  │
  ▼
node_1: foo = "hi! " + "foo" = "hi! foo"
  │
  ▼ 进入子图（共享键 foo 自动传入）
  │
  ├─ subgraph_node_1: bar = "bar"        ← 写入私有键
  │
  ├─ subgraph_node_2: foo = "hi! foo" + "bar" = "hi! foobar"
  │                                        ← 读取私有键 + 写入共享键
  │
  ▼ 退出子图（共享键 foo 自动传回，私有键 bar 丢弃）
  │
输出: {"foo": "hi! foobar"}
```

### 关键观察

1. **进入子图时**：父图的 `foo` 值自动传给子图的 `foo`
2. **子图内部**：`bar` 只在子图内部流转，父图完全不知道它的存在
3. **退出子图时**：子图的 `foo` 值自动传回父图，`bar` 被丢弃
4. **零配置**：不需要任何额外代码来处理传递，同名就自动传

## 3. 共享键的自动传递机制

### 传递规则

```
┌─────────────────────────────────────────────────┐
│                    父图                          │
│  ParentState: { foo: str, baz: str }            │
│                                                  │
│  ┌───────────────────────────────────────────┐  │
│  │              子图（作为节点）               │  │
│  │  SubState: { foo: str, bar: str }         │  │
│  │                                            │  │
│  │  foo ←──── 共享键 ────→ foo               │  │
│  │  bar ←──── 私有键（子图独有）              │  │
│  │                                            │  │
│  └───────────────────────────────────────────┘  │
│  baz ←──── 父图独有（子图看不到）               │
└─────────────────────────────────────────────────┘
```

| 字段 | 父图有？ | 子图有？ | 类型 | 传递方向 |
|------|----------|----------|------|----------|
| `foo` | 有 | 有 | 共享键 | 父→子（进入时），子→父（退出时） |
| `bar` | 无 | 有 | 私有键 | 不传递，子图内部使用 |
| `baz` | 有 | 无 | 父图独有 | 不传递，子图看不到 |

### 源码层面的实现

当你调用 `builder.add_node("node_2", subgraph)` 时，LangGraph 在编译阶段会：

```python
# 源码逻辑（简化）：libs/langgraph/langgraph/graph/state.py

# 1. 获取子图的 input_schema 和 output_schema
sub_input_fields = get_type_hints(subgraph.input_schema)
sub_output_fields = get_type_hints(subgraph.output_schema)

# 2. 找出与父图 Schema 的交集 → 这就是共享键
parent_fields = get_type_hints(parent_state_schema)
shared_input_keys = set(sub_input_fields) & set(parent_fields)
shared_output_keys = set(sub_output_fields) & set(parent_fields)

# 3. 进入子图时：只传递共享键的值
sub_input = {k: parent_state[k] for k in shared_input_keys}

# 4. 退出子图时：只取回共享键的值
parent_update = {k: sub_output[k] for k in shared_output_keys}
```

### 类型兼容性要求

共享键的 Channel 类型必须兼容：

```python
# ✅ 兼容：两边都是 str
class ParentState(TypedDict):
    foo: str

class SubState(TypedDict):
    foo: str


# ✅ 兼容：一边有 Reducer，一边是 LastValue（默认）
class ParentState(TypedDict):
    messages: Annotated[list, add]  # 有 Reducer

class SubState(TypedDict):
    messages: list  # LastValue（默认）


# ❌ 不兼容：两边有不同的 Reducer
class ParentState(TypedDict):
    data: Annotated[list, add]  # Reducer = add

class SubState(TypedDict):
    data: Annotated[list, lambda a, b: b]  # Reducer = 覆盖
```

## 4. 两种子图添加方式的对比

LangGraph 支持两种方式将子图添加为节点，它们在共享键行为上有本质区别。

### 方式一：直接添加编译后的子图（推荐）

```python
# 直接传入编译后的子图对象
builder.add_node("sub", subgraph)  # subgraph 是 compile() 的结果
```

**特点：**
- 共享键自动传递（同名字段）
- 子图有独立的 checkpoint 命名空间
- 支持 `stream_mode="subgraphs"` 观察子图内部
- 子图的私有键完全隔离

### 方式二：在包装函数中手动调用子图

```python
def call_subgraph(state: ParentState):
    """手动调用子图，需要自己处理状态转换"""
    # 手动构造子图输入
    sub_input = {"bar": state["foo"]}

    # 手动调用子图
    sub_output = subgraph.invoke(sub_input)

    # 手动提取子图输出
    return {"foo": sub_output["bar"]}

builder.add_node("sub", call_subgraph)
```

**特点：**
- 完全手动控制输入输出映射
- 父子图 Schema 可以完全不同（字段名不同也行）
- 子图没有独立的 checkpoint 命名空间
- 无法用 `stream_mode="subgraphs"` 观察子图内部

### 对比表

| 特性 | 直接添加 | 包装函数 |
|------|----------|----------|
| 共享键自动传递 | 自动 | 手动 |
| Schema 可以不同 | 必须有同名键 | 完全自由 |
| 独立 checkpoint | 有 | 无 |
| 子图流式观察 | 支持 | 不支持 |
| 代码量 | 少 | 多 |
| 灵活性 | 低 | 高 |

### 选择建议

```
父子图有同名字段？
  ├─ 是 → 直接添加（简单、自动、有 checkpoint）
  └─ 否 → 包装函数（灵活、手动映射）
```

## 5. Command.PARENT：显式更新父图状态

除了共享键的自动传递，子图还可以通过 `Command(graph=Command.PARENT)` 显式更新父图状态。

### 基本用法

```python
from langgraph.types import Command
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


# ===== 子图 =====
class SubState(TypedDict):
    task: str
    internal_result: str


def process(state: SubState):
    """子图内部处理"""
    return {"internal_result": f"已处理: {state['task']}"}


def report_to_parent(state: SubState) -> Command:
    """通过 Command 显式更新父图状态"""
    return Command(
        update={"parent_result": state["internal_result"]},
        graph=Command.PARENT  # 指定更新目标是父图
    )


sub_builder = StateGraph(SubState)
sub_builder.add_node("process", process)
sub_builder.add_node("report", report_to_parent)
sub_builder.add_edge(START, "process")
sub_builder.add_edge("process", "report")
sub_builder.add_edge("report", END)
sub = sub_builder.compile()


# ===== 父图 =====
class ParentState(TypedDict):
    task: str           # 共享键
    parent_result: str  # 只在父图中，通过 Command.PARENT 更新


def prepare(state: ParentState):
    return {"task": "分析文档"}


parent_builder = StateGraph(ParentState)
parent_builder.add_node("prepare", prepare)
parent_builder.add_node("sub", sub)
parent_builder.add_edge(START, "prepare")
parent_builder.add_edge("prepare", "sub")
parent_builder.add_edge("sub", END)
parent_graph = parent_builder.compile()

result = parent_graph.invoke({"task": "", "parent_result": ""})
print(result)
# {'task': '分析文档', 'parent_result': '已处理: 分析文档'}
```

### Command.PARENT vs 共享键

| 特性 | 共享键 | Command.PARENT |
|------|--------|----------------|
| 传递方式 | 自动（同名字段） | 显式（代码指定） |
| 可更新的字段 | 只能更新同名字段 | 可更新父图任意字段 |
| 代码量 | 零额外代码 | 需要写 Command |
| 适用场景 | 简单数据传递 | 复杂跨图通信 |
| 可读性 | 隐式，需要对照 Schema | 显式，一目了然 |

### 什么时候用 Command.PARENT？

```python
# 场景1：子图需要更新父图中不存在于子图 Schema 的字段
# 子图没有 parent_result 字段，但需要更新父图的 parent_result
Command(update={"parent_result": "done"}, graph=Command.PARENT)

# 场景2：子图需要同时更新自己的状态和父图的状态
Command(
    update={"parent_status": "completed"},  # 更新父图
    graph=Command.PARENT,
    goto="next_parent_node"  # 还可以指定父图的下一个节点
)

# 场景3：子图需要触发父图的路由决策
Command(
    update={"route_decision": "fallback"},
    graph=Command.PARENT,
    goto="fallback_handler"  # 直接跳转到父图的特定节点
)
```

## 6. 多层嵌套中的共享键

共享键机制在多层嵌套中同样适用，每一层只与直接相邻的层通信。

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


# ===== 第三层：孙子图 =====
class GrandchildState(TypedDict):
    shared: str       # 与子图共享
    gc_private: str   # 孙子图私有


def gc_node(state: GrandchildState):
    return {
        "gc_private": "gc_internal",
        "shared": state["shared"] + " → grandchild"
    }


gc_builder = StateGraph(GrandchildState)
gc_builder.add_node("gc_node", gc_node)
gc_builder.add_edge(START, "gc_node")
gc_builder.add_edge("gc_node", END)
grandchild = gc_builder.compile()


# ===== 第二层：子图 =====
class ChildState(TypedDict):
    shared: str       # 与父图和孙子图都共享
    child_private: str  # 子图私有


def child_node(state: ChildState):
    return {
        "child_private": "child_internal",
        "shared": state["shared"] + " → child"
    }


child_builder = StateGraph(ChildState)
child_builder.add_node("child_node", child_node)
child_builder.add_node("grandchild", grandchild)  # 嵌套孙子图
child_builder.add_edge(START, "child_node")
child_builder.add_edge("child_node", "grandchild")
child_builder.add_edge("grandchild", END)
child = child_builder.compile()


# ===== 第一层：父图 =====
class ParentState(TypedDict):
    shared: str  # 与子图共享


parent_builder = StateGraph(ParentState)
parent_builder.add_node("child", child)
parent_builder.add_edge(START, "child")
parent_builder.add_edge("child", END)
parent = parent_builder.compile()

result = parent.invoke({"shared": "root"})
print(result)
# {'shared': 'root → child → grandchild'}
```

### 数据流向

```
父图 shared="root"
  │
  ▼ 共享键 shared 传入子图
子图 shared="root", child_private=""
  │
  ├─ child_node: shared="root → child"
  │
  ▼ 共享键 shared 传入孙子图
孙子图 shared="root → child", gc_private=""
  │
  ├─ gc_node: shared="root → child → grandchild"
  │
  ▼ 共享键 shared 传回子图
子图 shared="root → child → grandchild"
  │
  ▼ 共享键 shared 传回父图
父图 shared="root → child → grandchild"
```

### 关键规则

- **逐层传递**：父图不能直接与孙子图通信，必须经过子图
- **每层独立**：每层只看自己和直接子图的同名字段
- **私有键隔离**：`child_private` 和 `gc_private` 各自隔离，互不可见

## 7. 常见陷阱与最佳实践

### 陷阱1：共享键名字拼错

```python
# ❌ 拼写不一致 → 变成两个私有键，数据不传递
class ParentState(TypedDict):
    result: str  # 注意：result

class SubState(TypedDict):
    results: str  # 注意：results（多了个 s）

# 父图的 result 和子图的 results 不是同名键！
# 数据不会自动传递
```

### 陷阱2：共享键类型不匹配

```python
# ❌ 类型不兼容会导致运行时错误
class ParentState(TypedDict):
    data: Annotated[list[str], add]  # Reducer: add

class SubState(TypedDict):
    data: Annotated[list[str], lambda a, b: b]  # Reducer: 覆盖

# 两个不同的 Reducer 作用于同一个共享键 → 冲突
```

### 陷阱3：以为私有键会传回父图

```python
class SubState(TypedDict):
    foo: str   # 共享键
    bar: str   # 私有键

def sub_node(state: SubState):
    return {
        "foo": "shared_result",
        "bar": "private_result"  # ← 这个值不会传回父图
    }

# 父图只能看到 foo 的更新，bar 在子图退出时被丢弃
```

### 最佳实践

```python
# ✅ 实践1：明确标注哪些是共享键
class SubState(TypedDict):
    # === 共享键（与父图通信）===
    query: str       # 从父图接收查询
    result: str      # 向父图返回结果

    # === 私有键（子图内部）===
    chunks: list     # 内部分块结果
    scores: list     # 内部评分结果


# ✅ 实践2：子图的 input/output schema 精确控制共享范围
class SubInput(TypedDict):
    query: str  # 只接收 query

class SubOutput(TypedDict):
    result: str  # 只返回 result

class SubInternal(SubInput, SubOutput):
    chunks: list
    scores: list

sub_builder = StateGraph(
    SubInternal,
    input_schema=SubInput,    # 精确控制输入
    output_schema=SubOutput   # 精确控制输出
)
```

## 8. 在 RAG 开发中的应用

### 场景：检索子图与生成子图的状态隔离

```python
from typing import Annotated
from typing_extensions import TypedDict
from operator import add
from langgraph.graph import StateGraph, START, END


# ===== 检索子图 =====
class RetrievalState(TypedDict):
    query: str                              # 共享键：从父图接收
    retrieved_docs: list[str]               # 共享键：传回父图
    # 私有键：检索内部使用
    embedding: list[float]
    candidate_ids: list[int]
    rerank_scores: list[float]


def embed_query(state: RetrievalState):
    """将查询转为向量（内部步骤）"""
    return {"embedding": [0.1, 0.2, 0.3]}  # 模拟 embedding


def vector_search(state: RetrievalState):
    """向量检索（内部步骤）"""
    return {"candidate_ids": [1, 2, 3, 4, 5]}


def rerank(state: RetrievalState):
    """重排序并输出最终文档（写入共享键）"""
    return {
        "rerank_scores": [0.9, 0.8, 0.7],
        "retrieved_docs": ["文档A", "文档B", "文档C"]  # 共享键
    }


retrieval_builder = StateGraph(RetrievalState)
retrieval_builder.add_node("embed", embed_query)
retrieval_builder.add_node("search", vector_search)
retrieval_builder.add_node("rerank", rerank)
retrieval_builder.add_edge(START, "embed")
retrieval_builder.add_edge("embed", "search")
retrieval_builder.add_edge("search", "rerank")
retrieval_builder.add_edge("rerank", END)
retrieval_subgraph = retrieval_builder.compile()


# ===== 父图 =====
class RAGState(TypedDict):
    query: str              # 共享键
    retrieved_docs: list[str]  # 共享键
    answer: str


def generate(state: RAGState):
    """基于检索结果生成回答"""
    docs = "\n".join(state["retrieved_docs"])
    return {"answer": f"基于 {len(state['retrieved_docs'])} 篇文档生成的回答"}


rag_builder = StateGraph(RAGState)
rag_builder.add_node("retrieve", retrieval_subgraph)  # 直接添加子图
rag_builder.add_node("generate", generate)
rag_builder.add_edge(START, "retrieve")
rag_builder.add_edge("retrieve", "generate")
rag_builder.add_edge("generate", END)
rag = rag_builder.compile()

result = rag.invoke({
    "query": "什么是向量数据库？",
    "retrieved_docs": [],
    "answer": ""
})
print(result["answer"])
# 基于 3 篇文档生成的回答
```

### 为什么这样设计？

- **父图不关心检索细节**：embedding、candidate_ids、rerank_scores 都是检索子图的内部实现，父图只需要最终的 `retrieved_docs`
- **子图可以独立演进**：检索策略从向量检索改为混合检索，只需要修改子图内部，父图代码不变
- **状态不会膨胀**：私有键在子图退出时丢弃，父图的状态保持精简

## 小结

| 要点 | 说明 |
|------|------|
| 共享键 | 父子图 Schema 中同名的字段，自动双向传递 |
| 私有键 | 只存在于子图中的字段，对父图不可见 |
| 自动传递 | 进入子图时传入共享键，退出时传回共享键 |
| Command.PARENT | 显式更新父图状态，可更新任意父图字段 |
| 类型兼容 | 共享键的 Channel 类型（Reducer）必须兼容 |
| 逐层传递 | 多层嵌套中，每层只与直接相邻层通信 |
| 最佳实践 | 用注释标注共享键/私有键，用 input/output schema 精确控制 |
