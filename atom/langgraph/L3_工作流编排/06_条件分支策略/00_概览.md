## 为什么要学这个？

### 现实问题

在真实的 AI 应用中，工作流不可能是一条直线。每一步都可能面临"接下来该走哪条路"的决策：

```python
# ❌ 静态图：所有路径写死，无法应对变化
builder.add_edge("input", "process")
builder.add_edge("process", "output")
# 如果检索质量不达标，需要回退重试呢？
# 如果用户意图有 5 种，需要路由到不同专家呢？
# 如果要对 N 篇文档并行处理，N 在运行前未知呢？
# 如果 LLM 判断需要调用不同工具呢？
```

这些场景的共同特征：**下一步走哪里，取决于运行时的状态，而非编译时的固定配置。**

### LangGraph 的解决方案：三种条件分支机制

```python
# ✅ 方式1：add_conditional_edges - 路由函数决定路径
def route_by_quality(state):
    if state["retrieval_score"] > 0.8:
        return "generate"       # 质量够好，直接生成
    return "re_retrieve"        # 质量不够，回退重试

builder.add_conditional_edges("evaluator", route_by_quality)

# ✅ 方式2：Send - 运行时创建动态并行分支
def fan_out_chunks(state):
    return [Send("process_chunk", {"chunk": c}) for c in state["chunks"]]

builder.add_conditional_edges("splitter", fan_out_chunks)

# ✅ 方式3：Command - 节点内部同时更新状态 + 控制路由
def smart_evaluator(state) -> Command[Literal["approve", "revise", "reject"]]:
    score = evaluate(state["draft"])
    if score > 0.9:
        return Command(update={"quality": "high"}, goto="approve")
    elif score > 0.5:
        return Command(update={"quality": "medium", "revision_count": state.get("revision_count", 0) + 1}, goto="revise")
    return Command(update={"quality": "low"}, goto="reject")
```

**条件分支策略就是让工作流拥有"大脑"——在每个岔路口根据实际情况做出最优决策。**

## 核心概念速览

### 概念 1: add_conditional_edges 基础

**一句话**: `add_conditional_edges(source, path, path_map, then)` 是条件分支的基础 API，通过路由函数在运行时决定下一个节点

```python
from langgraph.graph import StateGraph, START, END

# 基础用法：路由函数返回节点名
def route_function(state):
    if state["score"] > 0.8:
        return "high_quality"
    return "low_quality"

builder.add_conditional_edges("evaluator", route_function)

# 带 path_map：路由值 → 节点名映射
builder.add_conditional_edges("evaluator", route_function, {
    "good": "high_quality_handler",
    "bad": "low_quality_handler",
})

# 带 then：所有分支执行完后统一去某个节点
builder.add_conditional_edges("classifier", route_function, then="aggregator")
```

**关键特性**:
- `path` 参数是路由函数，接收 state，返回字符串或字符串列表
- `path_map` 可选，将路由函数返回值映射为实际节点名
- `then` 可选，指定所有分支完成后的汇聚节点
- 内部通过 `BranchSpec` 封装路由逻辑

---

### 概念 2: Literal 类型推断与路径映射

**一句话**: LangGraph 通过路由函数的 `Literal` 返回类型注解自动推断所有可能的目标节点，省去手动编写 path_map

```python
from typing import Literal

# ✅ 自动推断：LangGraph 从 Literal 类型注解中提取目标节点
def route_by_intent(state) -> Literal["search", "calculate", "respond"]:
    if state["intent"] == "search":
        return "search"
    elif state["intent"] == "calculate":
        return "calculate"
    return "respond"

builder.add_conditional_edges("classifier", route_by_intent)
# LangGraph 自动知道可能的目标是 ["search", "calculate", "respond"]

# ✅ 显式映射：当路由值和节点名不同时使用 path_map
def route_by_score(state) -> Literal["good", "bad"]:
    return "good" if state["score"] > 0.8 else "bad"

builder.add_conditional_edges("evaluator", route_by_score, {
    "good": "high_quality_handler",   # "good" → 实际节点名
    "bad": "low_quality_handler",     # "bad" → 实际节点名
})
```

**关键特性**:
- Literal 类型注解让图可视化工具能正确显示所有可能的分支
- 没有 Literal 注解也没有 path_map 时，可视化可能不完整
- Command 也支持类型注解：`Command[Literal["a", "b"]]`

---

### 概念 3: 多路分支与优先级路由

**一句话**: 路由函数返回列表可同时激活多个分支，结合优先级逻辑可实现 Supervisor 模式的智能调度

```python
# 多路并行：路由函数返回列表，同时走多条路
def multi_route(state):
    targets = []
    if state["needs_search"]:
        targets.append("search_node")
    if state["needs_calculation"]:
        targets.append("calc_node")
    return targets if targets else ["default_node"]

builder.add_conditional_edges("dispatcher", multi_route)

# Supervisor 模式：中心调度节点决定下一个执行的 Agent
def supervisor_route(state) -> Literal["researcher", "writer", "reviewer", "__end__"]:
    """Supervisor 根据当前进度决定下一个 Agent"""
    if not state.get("research_done"):
        return "researcher"
    elif not state.get("draft_done"):
        return "writer"
    elif not state.get("review_done"):
        return "reviewer"
    return "__end__"
```

**关键特性**:
- 返回列表时，所有目标节点并行执行
- Supervisor 模式是多 Agent 系统的经典架构
- 优先级路由通过 if-elif 链实现，先匹配的优先

---

### 概念 4: Send 动态路由与 Map-Reduce

**一句话**: `Send(node, arg)` 在运行时创建任意数量的并行分支，每个分支携带独立状态，实现 map-reduce 模式

```python
from langgraph.types import Send
from typing import Annotated
import operator

class OverallState(TypedDict):
    documents: list[str]
    summaries: Annotated[list[str], operator.add]  # reducer 合并并行结果

class WorkerState(TypedDict):
    document: str

def fan_out_documents(state: OverallState):
    """运行时动态创建 N 个并行任务"""
    return [
        Send("summarize_doc", {"document": doc})
        for doc in state["documents"]  # 数量运行时才知道
    ]

def summarize_doc(state: WorkerState):
    return {"summaries": [f"摘要: {state['document'][:50]}..."]}

builder.add_conditional_edges("splitter", fan_out_documents)
```

**关键特性**:
- 每个 Send 实例创建一个独立的并行任务
- 任务数量完全由运行时数据决定（编译时未知）
- 并行结果通过 reducer（如 `operator.add`）合并到主状态
- 每个 Send 可以携带不同的状态参数（状态隔离）

---

### 概念 5: Command 高级路由控制

**一句话**: 节点返回 `Command(update=..., goto=...)` 在一次操作中同时更新状态和指定下一个节点，支持跨图导航

```python
from langgraph.types import Command
from typing import Literal

# 基础用法：update + goto
def evaluate_node(state) -> Command[Literal["approve", "revise"]]:
    score = state["quality_score"]
    if score > 0.8:
        return Command(update={"status": "approved"}, goto="approve")
    return Command(update={"status": "needs_revision"}, goto="revise")

# 高级用法：跨图导航（多代理切换）
def research_agent(state) -> Command[Literal["writer_agent"]]:
    result = do_research(state["topic"])
    return Command(
        update={"research_result": result},
        goto="writer_agent",
        graph=Command.PARENT  # 导航到父图中的其他子图
    )

# goto 支持 Send 对象（Command + Send 组合）
def dispatcher(state) -> Command:
    tasks = state["pending_tasks"]
    return Command(
        update={"dispatched": True},
        goto=[Send("worker", {"task": t}) for t in tasks]
    )
```

**关键特性**:
- `update` 更新状态（等价于普通节点返回值）
- `goto` 指定下一个节点（字符串、Send 对象或列表）
- `graph=Command.PARENT` 实现跨子图导航
- 使用 Command 时不要给该节点添加静态出边（否则两者都会执行）

---

### 概念 6: 回退机制与循环守卫

**一句话**: 通过条件边实现错误回退路由，配合 `recursion_limit` / 计数器防止无限循环，构建健壮的降级策略

```python
# 回退路由：质量不达标时回退重试
def quality_gate(state) -> Literal["output", "retry", "fallback"]:
    if state["quality_score"] > 0.8:
        return "output"          # 质量达标，输出
    elif state["retry_count"] < 3:
        return "retry"           # 未达标但还有重试次数
    return "fallback"            # 重试耗尽，降级处理

# 循环守卫：编译时设置递归上限
graph = builder.compile()
# 运行时设置 recursion_limit 防止无限循环
result = graph.invoke(input_data, {"recursion_limit": 25})

# 节点内计数器守卫
def retry_node(state):
    count = state.get("retry_count", 0) + 1
    if count > 3:
        return {"status": "max_retries_exceeded", "retry_count": count}
    # 执行重试逻辑...
    return {"retry_count": count, "result": new_result}
```

**关键特性**:
- 条件边可以路由回已执行过的节点，形成循环
- `recursion_limit` 是全局安全阀，防止无限循环（默认 25）
- 节点内计数器提供更精细的循环控制
- 降级策略：重试 → 简化处理 → 返回默认值 → 报错

## 实战场景预览

### 场景 1: 基础条件路由

**问题**: 根据用户输入类型路由到不同处理节点
**方案**: `add_conditional_edges` + 简单路由函数 + Literal 类型推断
**适用**: 意图分类、请求分发、简单分支

---

### 场景 2: 多路分支与 Supervisor

**问题**: 多个专家 Agent 需要中心调度，按优先级依次执行
**方案**: Supervisor 节点 + 条件边循环 + 完成条件判断
**适用**: 多 Agent 协作、任务编排、流水线调度

---

### 场景 3: Send 并行 Map-Reduce

**问题**: 对未知数量的文档并行处理后汇总结果
**方案**: Send API 动态创建并行任务 + reducer 合并结果
**适用**: 批量文档处理、多源数据聚合、Tree of Thoughts

---

### 场景 4: Command 动态路由

**问题**: 节点需要同时更新状态和决定下一步走向
**方案**: Command(update=..., goto=...) 一体化控制
**适用**: 评估-决策节点、状态机转换、审批流程

---

### 场景 5: 错误处理与回退

**问题**: 工作流执行失败时需要重试、降级或回退
**方案**: 条件边 + 重试计数器 + recursion_limit + 降级节点
**适用**: 生产级容错、API 调用重试、质量保证循环

---

### 场景 6: RAG 条件路由

**问题**: RAG 系统中检索质量不达标时需要自适应调整策略
**方案**: Self-RAG / Adaptive RAG 模式，评估检索质量后路由
**适用**: 检索质量判断、查询改写触发、多策略检索切换

---

### 场景 7: 人机循环条件分支

**问题**: 关键决策需要人工审批，审批结果决定后续路径
**方案**: interrupt + Command 实现审批分支
**适用**: 内容审核、高风险操作确认、人工标注

---

### 场景 8: 多 Agent 协作路由

**问题**: 多个 Agent 之间需要根据执行结果动态切换
**方案**: Command + graph=Command.PARENT 实现 Agent 间 handoff
**适用**: 研究-写作-审核流水线、专家系统、任务委派

---

### 场景 9: LLM 驱动智能路由

**问题**: 需要 LLM 自主判断下一步执行什么操作
**方案**: 结构化输出 + 条件边，LLM 决策驱动路由
**适用**: ReAct Agent、工具选择、自主任务规划

## 核心 API 速查

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send, Command
from typing import Literal, Annotated
import operator

# ===== 1. add_conditional_edges =====
# 基础：路由函数返回节点名
builder.add_conditional_edges("source", route_function)

# 带 path_map：路由值映射为节点名
builder.add_conditional_edges("source", route_fn, {
    "option_a": "node_a",
    "option_b": "node_b",
})

# 带 then：分支完成后汇聚
builder.add_conditional_edges("source", route_fn, then="merge_node")

# 多路并行：路由函数返回列表
def multi_route(state) -> list[str]:
    return ["node_a", "node_b"]

# ===== 2. Literal 类型推断 =====
# 自动推断目标节点（推荐）
def router(state) -> Literal["a", "b", "c"]:
    ...

# Command 类型注解
def node(state) -> Command[Literal["x", "y"]]:
    return Command(goto="x")

# ===== 3. Send API =====
def fan_out(state):
    return [Send("worker", {"item": x}) for x in state["items"]]

builder.add_conditional_edges("splitter", fan_out)

# 状态需要 reducer
class State(TypedDict):
    results: Annotated[list, operator.add]

# ===== 4. Command API =====
# 基础：update + goto
def node(state) -> Command[Literal["a", "b"]]:
    return Command(update={"key": "val"}, goto="a")

# 跨图导航
def subgraph_node(state) -> Command:
    return Command(goto="other_agent", graph=Command.PARENT)

# Command + Send 组合
def node(state) -> Command:
    return Command(goto=[Send("w", {"t": t}) for t in tasks])

# ===== 5. 循环守卫 =====
# 编译时设置
graph = builder.compile()
# 运行时限制递归深度
result = graph.invoke(data, {"recursion_limit": 25})
```

## 快速开始

### 5 分钟快速体验

```python
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END

# 1. 定义状态
class State(TypedDict):
    query: str
    intent: str
    response: str

# 2. 定义节点
def classify(state: State):
    query = state["query"].lower()
    if "价格" in query or "多少钱" in query:
        return {"intent": "pricing"}
    elif "怎么用" in query or "教程" in query:
        return {"intent": "tutorial"}
    return {"intent": "general"}

def pricing_handler(state: State):
    return {"response": f"[价格专家] {state['query']}"}

def tutorial_handler(state: State):
    return {"response": f"[教程专家] {state['query']}"}

def general_handler(state: State):
    return {"response": f"[通用助手] {state['query']}"}

# 3. 路由函数（带 Literal 类型注解）
def route_by_intent(state: State) -> Literal["pricing_handler", "tutorial_handler", "general_handler"]:
    return f"{state['intent']}_handler"

# 4. 构建图
builder = StateGraph(State)
builder.add_node("classify", classify)
builder.add_node("pricing_handler", pricing_handler)
builder.add_node("tutorial_handler", tutorial_handler)
builder.add_node("general_handler", general_handler)

builder.add_edge(START, "classify")
builder.add_conditional_edges("classify", route_by_intent)  # 条件分支！
builder.add_edge("pricing_handler", END)
builder.add_edge("tutorial_handler", END)
builder.add_edge("general_handler", END)

# 5. 编译并运行
graph = builder.compile()

# 测试：不同输入走不同路径
r1 = graph.invoke({"query": "这个产品多少钱？"})
print(r1["response"])  # [价格专家] 这个产品多少钱？

r2 = graph.invoke({"query": "怎么用这个功能？"})
print(r2["response"])  # [教程专家] 怎么用这个功能？

r3 = graph.invoke({"query": "你好"})
print(r3["response"])  # [通用助手] 你好
```

**运行结果**: 同一个图，不同输入自动路由到不同专家节点。这就是条件分支策略的核心价值。

## 关键要点总结

1. **条件分支策略是 LangGraph 工作流的决策引擎**，让图在每个岔路口根据状态做出最优选择
2. **三种核心机制互补**：条件边（路由选择）、Send（动态并行）、Command（状态+路由一体化）
3. **Literal 类型注解**让图可视化工具正确显示所有可能的分支路径
4. **回退机制 + 循环守卫**是生产级工作流的必备组件
5. **Supervisor 模式**是多 Agent 系统中最常用的条件分支架构
6. **条件边最常用**（覆盖 80% 场景），Send 解决动态并行，Command 处理复杂控制流
