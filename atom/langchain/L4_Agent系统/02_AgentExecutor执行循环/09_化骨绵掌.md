## 卡片1：执行循环的本质

**一句话：** AgentExecutor 就是一个 while 循环，不断重复"思考→行动→观察"直到得出答案。

**核心逻辑：**

```
while 未完成 and 未超限:
    决策 = agent.plan(历史步骤, 用户输入)
    if 决策 == "我有答案了":
        return 最终答案
    执行工具，记录结果
    把结果加入历史步骤
```

**为什么重要：** 这个模式叫 ReAct（Reasoning + Acting），是目前所有 AI Agent 系统的基础范式。不管是 LangChain、LlamaIndex 还是 AutoGPT，底层都是这个循环。理解了它，就理解了 Agent 的本质。

**关键记忆：** Agent 不是一次性给答案，而是像人一样"想一步做一步"，每一步都基于之前的结果来决定下一步。

🔗 **RAG 关联：** RAG Agent 的检索-判断-再检索循环，就是这个 while 循环的具体应用。Agent 检索一次发现信息不够，会自动决定换个关键词再检索。

## 卡片2：三个核心数据类型

**一句话：** AgentExecutor 用三个数据类型描述整个执行过程——要做什么、做完了、做的结果。

**三兄弟：**

| 类型 | 含义 | 关键字段 | 类比 |
|------|------|----------|------|
| `AgentAction` | 我要做什么 | tool, tool_input, log | 任务工单 |
| `AgentFinish` | 我做完了 | return_values, log | 完工报告 |
| `AgentStep` | 做完的结果 | action, observation | 执行记录 |

**数据流转：**

```
agent.plan() → AgentAction("search", "天气")
                    ↓
              执行 search 工具
                    ↓
AgentStep(action=上面的action, observation="晴天25度")
                    ↓
              加入 intermediate_steps
                    ↓
agent.plan() → AgentFinish(return_values={"output": "今天晴天"})
```

**关键记忆：** `AgentAction` 是"意图"，`AgentStep` 是"意图+结果"的配对。`intermediate_steps` 就是一个 `AgentStep` 的列表，记录了完整的推理轨迹。

🔗 **RAG 关联：** 在 RAG Agent 中，AgentAction 可能是"用向量检索查找相关文档"，observation 就是检索返回的文档片段。

## 卡片3：agent.plan() — Agent 的大脑

**一句话：** `plan()` 是 Agent 的决策中心，接收历史步骤和用户输入，决定下一步做什么或直接给出答案。

**函数签名：**

```python
def plan(
    self,
    intermediate_steps: list[tuple[AgentAction, str]],  # 历史轨迹
    **kwargs  # 包含用户输入 input 等
) -> AgentAction | AgentFinish:  # 继续行动 or 结束
```

**决策过程：**

```
plan() 内部做了什么：
1. 把 intermediate_steps 格式化成文本（之前做了什么）
2. 组装完整 prompt（系统指令 + 工具描述 + 历史 + 用户问题）
3. 调用 LLM 生成回复
4. 解析 LLM 输出 → AgentAction 或 AgentFinish
```

**关键记忆：** `plan()` 的本质就是一次 LLM 调用。它把"我之前做了什么"和"用户问了什么"拼成 prompt，让 LLM 决定下一步。LLM 的输出格式决定了是继续行动还是给出最终答案。

**MultiActionAgent 的区别：** 普通 Agent 的 `plan()` 返回单个 `AgentAction`，而 `MultiActionAgent` 可以返回 `list[AgentAction]`，一次规划多个动作。

🔗 **RAG 关联：** RAG Agent 的 plan() 会根据已检索的文档判断——信息够了就生成答案（AgentFinish），不够就决定再检索一次（AgentAction）。

## 卡片4：_iter_next_step — 单步执行引擎

**一句话：** `_iter_next_step()` 是执行循环的单次迭代，负责调用 plan()、执行工具、处理错误，是循环体的核心实现。

**执行流程：**

```
_iter_next_step():
    ┌─ 1. 调用 agent.plan()
    │      ↓
    ├─ 2a. 返回 AgentFinish？→ yield AgentFinish → 结束
    │      ↓ (返回 AgentAction)
    ├─ 2b. 返回 AgentAction(s)
    │      ↓
    ├─ 3. 工具名是 "_Exception"？→ yield AgentStep(错误信息)
    │      ↓
    ├─ 4. 在 tool_map 中查找工具
    │      ↓ (找不到)
    ├─ 4a. 找不到？→ 用 InvalidTool 返回错误提示
    │      ↓ (找到了)
    ├─ 5. 执行工具：tool.run(action.tool_input)
    │      ↓
    └─ 6. yield AgentStep(action, observation)
```

**错误处理细节：**

```python
# 解析失败时（LLM 输出格式不对）
if handle_parsing_errors:
    observation = "Invalid or incomplete response"  # 错误变成 observation
    # 继续循环，让 LLM 看到错误后自我修正

# 工具不存在时
if tool_name not in tool_map:
    observation = f"{tool_name} is not a valid tool, try one of [{valid_tools}]"
    # 告诉 LLM 可用的工具列表
```

**关键记忆：** `_iter_next_step` 是一个生成器（yield），它可以产出多个 `AgentStep`（当 MultiActionAgent 返回多个动作时）。每个动作独立执行，独立产出结果。

🔗 **RAG 关联：** 当 RAG Agent 调用检索工具时，`_iter_next_step` 负责实际执行检索并把结果（文档片段）作为 observation 返回给 Agent。

## 卡片5：循环控制三板斧

**一句话：** AgentExecutor 用三个参数防止 Agent 失控——最多循环几次、最多跑多久、超限后怎么办。

**三个控制参数：**

| 参数 | 默认值 | 作用 | 类比 |
|------|--------|------|------|
| `max_iterations` | 15 | 最多循环次数 | 考试时间限制 |
| `max_execution_time` | None | 最大运行秒数 | 闹钟 |
| `early_stopping_method` | "force" | 超限后的处理方式 | 交卷策略 |

**_should_continue() 判断逻辑：**

```python
def _should_continue(self, iterations, time_elapsed):
    # 检查次数限制
    if self.max_iterations is not None and iterations >= self.max_iterations:
        return False
    # 检查时间限制
    if self.max_execution_time is not None and time_elapsed > self.max_execution_time:
        return False
    return True
```

**超限处理（early_stopping_method）：**

```python
# "force" 模式：直接返回固定文本
return AgentFinish(
    return_values={"output": "Agent stopped due to iteration limit or time limit."},
    log=""
)

# "generate" 模式：让 LLM 基于已有信息生成最终答案
# 把当前的 intermediate_steps 传给 LLM，让它总结一个答案
```

**关键记忆：** 生产环境必须设置 `max_iterations`。默认值15对大多数场景够用，但复杂任务可能需要调高。`"generate"` 模式比 `"force"` 更优雅，因为它会尝试给出一个有意义的答案而不是硬停。

🔗 **RAG 关联：** RAG Agent 如果检索-判断循环太多次（比如一直找不到相关文档），max_iterations 会强制停止，避免无限消耗 token。

## 卡片6：错误自修复机制

**一句话：** AgentExecutor 把错误变成 observation 反馈给 LLM，让 Agent 自己修正错误，而不是直接崩溃。

**handle_parsing_errors 三种模式：**

```python
# 模式1：布尔值 - 使用默认错误消息
agent_executor = AgentExecutor(
    agent=agent, tools=tools,
    handle_parsing_errors=True  # "Invalid or incomplete response"
)

# 模式2：字符串 - 自定义错误消息
agent_executor = AgentExecutor(
    agent=agent, tools=tools,
    handle_parsing_errors="请按正确格式输出，包含 Action 和 Action Input"
)

# 模式3：回调函数 - 自定义处理逻辑
def my_handler(error: OutputParserException) -> str:
    return f"输出格式错误: {error}。请重试。"

agent_executor = AgentExecutor(
    agent=agent, tools=tools,
    handle_parsing_errors=my_handler
)
```

**自修复流程：**

```
LLM 输出: "我觉得答案是42"  （格式不对，没有 Action/Final Answer）
    ↓ 解析失败
observation = "Invalid or incomplete response"
    ↓ 加入 intermediate_steps
LLM 再次推理，看到错误提示
    ↓ 自我修正
LLM 输出: "Final Answer: 答案是42"  （格式正确）
    ↓ 解析成功
返回 AgentFinish
```

**InvalidTool 机制：**

```
LLM 输出: Action: google_search  （但工具列表里没有这个）
    ↓
observation = "google_search is not a valid tool, try one of [search, calculator]"
    ↓
LLM 看到提示后改用正确的工具名
```

**关键记忆：** 错误自修复的核心思想是"把错误当作一种 observation"。LLM 看到错误信息后，就像人看到报错一样，会尝试换个方式重来。这比直接抛异常优雅得多。

🔗 **RAG 关联：** RAG Agent 调用检索工具时如果参数格式错误，自修复机制会让 Agent 自动修正查询参数，而不是整个流程崩溃。

## 卡片7：intermediate_steps 的秘密

**一句话：** `intermediate_steps` 是 Agent 的"工作记忆"，记录了完整的推理轨迹，可以裁剪、返回、用于调试。

**数据结构：**

```python
# intermediate_steps 的类型
intermediate_steps: list[tuple[AgentAction, str]]

# 实际内容示例
[
    (AgentAction(tool="search", tool_input="LangChain是什么", log="..."),
     "LangChain是一个LLM应用开发框架..."),
    (AgentAction(tool="search", tool_input="LangChain最新版本", log="..."),
     "LangChain最新版本是0.3..."),
]
```

**三个关键操作：**

```python
# 1. 返回给用户（便于调试和展示推理过程）
executor = AgentExecutor(
    agent=agent, tools=tools,
    return_intermediate_steps=True  # 默认 False
)
result = executor.invoke({"input": "..."})
print(result["intermediate_steps"])  # 可以看到每一步

# 2. 裁剪历史（节省 token，防止上下文溢出）
def trim_steps(steps):
    """只保留最近3步"""
    return steps[-3:]

executor = AgentExecutor(
    agent=agent, tools=tools,
    trim_intermediate_steps=trim_steps
)

# 3. 在回调中监控
class StepCallback(BaseCallbackHandler):
    def on_agent_action(self, action, **kwargs):
        print(f"执行工具: {action.tool}({action.tool_input})")
    def on_tool_end(self, output, **kwargs):
        print(f"工具结果: {output}")
```

**为什么需要裁剪：**

```
第1步: plan() prompt = 系统指令 + 工具描述 + 0步历史 + 用户问题  → 2000 tokens
第5步: plan() prompt = 系统指令 + 工具描述 + 4步历史 + 用户问题  → 5000 tokens
第10步: plan() prompt = 系统指令 + 工具描述 + 9步历史 + 用户问题 → 10000 tokens
                                                                    ↑ 可能超出上下文窗口！
```

**关键记忆：** `intermediate_steps` 会随着循环不断增长，每一步都会被完整传给 LLM。步数越多，token 消耗越大。裁剪策略是生产环境的必备优化。

🔗 **RAG 关联：** RAG Agent 的 intermediate_steps 记录了每次检索的查询词和返回的文档片段，这些信息对于调试"为什么检索结果不好"非常有价值。

## 卡片8：异步与并行执行

**一句话：** AgentExecutor 支持异步执行，最大的优势是 MultiActionAgent 可以并行执行多个工具调用。

**同步 vs 异步：**

```python
# 同步执行（阻塞）
result = executor.invoke({"input": "查询天气和新闻"})

# 异步执行（非阻塞）
result = await executor.ainvoke({"input": "查询天气和新闻"})
```

**并行工具执行（关键优势）：**

```python
# 同步：串行执行，总时间 = 工具1时间 + 工具2时间
for action in actions:
    observation = tool.run(action.tool_input)  # 一个一个来

# 异步：并行执行，总时间 = max(工具1时间, 工具2时间)
async def _aiter_next_step():
    actions = agent.plan(...)  # 返回多个 AgentAction
    # 所有工具同时执行
    observations = await asyncio.gather(*[
        tool.arun(action.tool_input) for action in actions
    ])
```

**异步超时控制：**

```python
# 使用 asyncio 的超时机制
try:
    async with asyncio_timeout(self.max_execution_time):
        # 执行循环
        ...
except asyncio.TimeoutError:
    # 超时处理
    ...
```

**关键记忆：** 异步的最大价值不是"非阻塞"，而是"并行"。当 Agent 一次规划了多个工具调用时，异步模式可以同时执行它们，大幅减少总耗时。

🔗 **RAG 关联：** RAG Agent 可能需要同时从多个数据源检索（向量库 + 关键词搜索 + 知识图谱），异步并行可以显著提升检索速度。

## 卡片9：从 AgentExecutor 到 create_agent

**一句话：** LangChain 正在用 LangGraph 的 `create_react_agent` 替代 `AgentExecutor`，核心循环不变，架构更灵活。

**演进对比：**

```
AgentExecutor（旧）          create_react_agent（新）
┌──────────────────┐      ┌──────────────────────┐
│  while 循环       │  →   │  StateGraph 状态机     │
│  参数控制行为     │  →   │  中间件控制行为        │
│  固定执行流程     │  →   │  可自定义节点和边      │
│  Chain 继承       │  →   │  Graph 组合           │
└──────────────────┘      └──────────────────────┘
```

**核心循环对比（本质相同）：**

```python
# AgentExecutor 的循环
while self._should_continue(iterations, time_elapsed):
    output = self._take_next_step(...)  # plan + execute
    if isinstance(output, AgentFinish):
        return output

# create_react_agent 的循环（LangGraph 实现）
# 节点: agent_node（调用LLM） → tool_node（执行工具）
# 边: should_continue 条件边（检查是否有工具调用）
graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", tool_node)
graph.add_conditional_edges("agent", should_continue, {
    "continue": "tools",
    "end": END
})
```

**迁移建议：**

```python
# 旧方式
from langchain.agents import AgentExecutor, create_openai_functions_agent
agent = create_openai_functions_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, max_iterations=10)

# 新方式
from langgraph.prebuilt import create_react_agent
app = create_react_agent(llm, tools)
# 通过中间件添加控制逻辑
```

**关键记忆：** 不管用哪种方式，核心都是"调用 LLM 决策 → 执行工具 → 把结果反馈给 LLM → 重复"。AgentExecutor 用 while 循环实现，LangGraph 用状态图实现，但本质是同一个模式。

🔗 **RAG 关联：** LangGraph 版本的 RAG Agent 可以更灵活地定义检索策略，比如"先粗检索→再精检索→判断是否需要补充检索"的复杂流程。

## 卡片10：生产环境检查清单

**一句话：** 把 AgentExecutor 部署到生产环境前，必须检查这些配置，否则迟早出事。

**必须配置（不配会出事）：**

```python
executor = AgentExecutor(
    agent=agent,
    tools=tools,

    # 1. 防止无限循环（必须设置）
    max_iterations=15,          # 根据任务复杂度调整
    max_execution_time=120,     # 秒，防止单次请求太久

    # 2. 防止解析崩溃（必须开启）
    handle_parsing_errors=True, # 或自定义错误消息

    # 3. 便于调试（强烈建议）
    return_intermediate_steps=True,
    verbose=True,               # 开发环境开启
)
```

**建议配置（提升质量）：**

```python
executor = AgentExecutor(
    # ...基础配置...

    # 4. 节省 token（长对话必备）
    trim_intermediate_steps=lambda steps: steps[-5:],

    # 5. 优雅停止（比 force 更好）
    early_stopping_method="generate",

    # 6. 回调监控（生产必备）
    callbacks=[LoggingCallback(), MetricsCallback()],
)
```

**常见故障与解决：**

| 故障现象 | 原因 | 解决方案 |
|----------|------|----------|
| Agent 无限循环 | 未设 max_iterations | 设置合理的迭代上限 |
| 直接报错崩溃 | 未开 handle_parsing_errors | 开启错误处理 |
| Token 超限 | intermediate_steps 太长 | 使用 trim 裁剪 |
| 响应太慢 | 串行执行多工具 | 使用异步并行 |
| 答案质量差 | early_stopping="force" | 改用 "generate" |
| 无法复现问题 | 未记录中间步骤 | 开启 return_intermediate_steps |

**关键记忆：** 生产环境的 AgentExecutor 配置核心就三个词——**限制**（max_iterations）、**容错**（handle_parsing_errors）、**可观测**（return_intermediate_steps）。

🔗 **RAG 关联：** RAG Agent 在生产环境中尤其需要关注 token 消耗，因为检索返回的文档片段会快速填满上下文窗口，trim_intermediate_steps 是必备优化。
