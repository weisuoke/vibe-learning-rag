# 实战代码 - 场景6：RAG 条件路由

> 本文档演示 Adaptive RAG 系统中的条件分支策略，根据检索质量动态路由到不同处理路径

---

## 场景描述

**场景**：构建一个 Adaptive RAG 系统，根据检索结果质量动态决定下一步操作。

**核心挑战**：传统 RAG 是「检索 → 生成」的固定管道，但实际场景中：
- 检索结果可能不相关 → 需要改写查询重新检索
- 向量库可能没有答案 → 需要回退到 Web 搜索
- 生成结果可能有幻觉 → 需要重新生成或拒绝回答

**Adaptive RAG 路由模式**：

```
                    ┌──────────┐
                    │  用户提问  │
                    └─────┬────┘
                          │
                    ┌─────▼────┐
                    │  向量检索  │
                    └─────┬────┘
                          │
                    ┌─────▼────┐
                    │ 质量评估   │
                    └──┬──┬──┬─┘
                       │  │  │
              高质量    │  │  │  低质量
             (≥0.7)    │  │  │ (<0.4)
                       │  │  │
                  生成   │ 改写 │ Web搜索
                       │  │  │
                       ▼  │  ▼
                    ┌─────▼────┐
                    │  生成回答  │
                    └─────┬────┘
                          │
                    ┌─────▼────┐
                    │ 幻觉检测   │
                    └──┬─────┬─┘
                       │     │
                    通过    幻觉
                       │     │
                    输出   重新生成
```

[来源: reference/search_conditional_branching_01.md, reference/fetch_best_practices_01.md]

## 完整代码

```python
"""
条件分支策略 - 实战场景6：RAG 条件路由
演示：Adaptive RAG 检索质量判断路由

路由策略：
  检索质量高 (≥0.7) → 直接生成
  检索质量中 (0.4~0.7) → 改写查询后重新检索
  检索质量低 (<0.4) → 回退到 Web 搜索
  生成后检测幻觉 → 通过则输出，否则重新生成
"""

import random
from typing import TypedDict, Literal, Annotated
from langgraph.graph import StateGraph, START, END
import operator


# ============================================================
# 1. 状态定义
# ============================================================

class RAGState(TypedDict):
    """Adaptive RAG 状态"""
    question: str                # 用户原始问题
    rewritten_question: str      # 改写后的问题
    documents: list[str]         # 检索到的文档片段
    generation: str              # 生成的回答
    retrieval_score: float       # 检索质量分数 (0~1)
    generation_score: float      # 生成质量分数 (0~1)
    retry_count: int             # 重试计数（循环守卫）
    max_retries: int             # 最大重试次数
    source: str                  # 数据来源: vectordb / web / cache
    route_log: list[str]         # 路由决策日志


# ============================================================
# 2. 检索节点
# ============================================================

def retrieve_from_vectordb(state: RAGState) -> dict:
    """
    向量数据库检索。

    实际项目中，这里调用 ChromaDB / Milvus / FAISS 等。
    这里用模拟数据演示路由逻辑。
    """
    question = state.get("rewritten_question") or state["question"]
    log = list(state.get("route_log", []))

    print(f"  [检索] 向量检索: '{question}'")

    # 模拟检索结果和质量分数
    # 实际项目中：score = 检索结果与 query 的相似度均值
    score = random.uniform(0.2, 1.0)
    docs = [
        f"[向量库] 文档片段1: 关于'{question}'的相关内容...",
        f"[向量库] 文档片段2: '{question}'的补充说明...",
    ]

    print(f"  [检索] 检索到 {len(docs)} 个片段，质量分数: {score:.2f}")

    return {
        "documents": docs,
        "retrieval_score": score,
        "source": "vectordb",
        "route_log": log + [f"向量检索: score={score:.2f}"],
    }


def web_search(state: RAGState) -> dict:
    """
    Web 搜索回退。

    当向量库检索质量太低时，回退到 Web 搜索。
    实际项目中，这里调用 Tavily / SerpAPI / Bing Search 等。
    """
    question = state.get("rewritten_question") or state["question"]
    log = list(state.get("route_log", []))

    print(f"  [Web] 搜索: '{question}'")

    # Web 搜索通常能找到更广泛的信息
    docs = [
        f"[Web] 搜索结果1: {question} 的详细解释...",
        f"[Web] 搜索结果2: {question} 的最新资料...",
    ]

    print(f"  [Web] 找到 {len(docs)} 个结果")

    return {
        "documents": docs,
        "retrieval_score": 0.75,  # Web 搜索通常质量还行
        "source": "web",
        "route_log": log + ["Web 搜索回退"],
    }


# ============================================================
# 3. 查询改写节点
# ============================================================

def rewrite_query(state: RAGState) -> dict:
    """
    查询改写 —— 当检索质量中等时，改写查询后重新检索。

    实际项目中，这里调用 LLM 改写查询：
    - 扩展关键词
    - 分解复合问题
    - 添加上下文限定
    """
    question = state.get("rewritten_question") or state["question"]
    retry_count = state.get("retry_count", 0)
    log = list(state.get("route_log", []))

    # 模拟查询改写策略
    strategies = [
        f"请详细解释: {question}",
        f"{question} 的原理和应用场景",
        f"用简单的语言解释 {question}",
    ]
    rewritten = strategies[retry_count % len(strategies)]

    print(f"  [改写] '{question}' → '{rewritten}'")

    return {
        "rewritten_question": rewritten,
        "retry_count": retry_count + 1,
        "route_log": log + [f"查询改写: '{rewritten}'"],
    }


# ============================================================
# 4. 生成节点
# ============================================================

def generate_answer(state: RAGState) -> dict:
    """
    基于检索结果生成回答。

    实际项目中，这里调用 LLM，将检索到的文档作为上下文注入 prompt。
    """
    question = state.get("rewritten_question") or state["question"]
    docs = state.get("documents", [])
    source = state.get("source", "unknown")
    log = list(state.get("route_log", []))

    print(f"  [生成] 基于 {source} 的 {len(docs)} 个文档生成回答...")

    # 模拟生成和质量评估
    # 实际项目中：调用 LLM 生成，然后用另一个 LLM 评估是否有幻觉
    gen_score = random.uniform(0.4, 1.0)

    context_preview = docs[0][:30] if docs else "无上下文"
    answer = (
        f"基于{source}检索的回答: "
        f"关于'{question}'，根据检索到的资料（{context_preview}...），"
        f"答案是..."
    )

    print(f"  [生成] 生成完成，质量分数: {gen_score:.2f}")

    return {
        "generation": answer,
        "generation_score": gen_score,
        "route_log": log + [f"生成回答: score={gen_score:.2f}"],
    }


# ============================================================
# 5. 路由函数
# ============================================================

def route_by_retrieval_quality(
    state: RAGState,
) -> Literal["generate", "rewrite_query", "web_search"]:
    """
    根据检索质量路由 —— Adaptive RAG 的核心决策点。

    决策逻辑：
      score ≥ 0.7  → 直接生成（检索质量足够好）
      0.4 ≤ score < 0.7 → 改写查询重新检索（可能是 query 不够好）
      score < 0.4  → Web 搜索（向量库可能没有相关内容）

    循环守卫：改写重试超过上限 → 强制进入生成
    """
    score = state.get("retrieval_score", 0)
    retry_count = state.get("retry_count", 0)
    max_retries = state.get("max_retries", 2)

    # 高质量：直接生成
    if score >= 0.7:
        print(f"  [路由] 检索质量高 ({score:.2f} ≥ 0.7) → 直接生成")
        return "generate"

    # 中等质量 + 未超重试上限：改写查询
    if score >= 0.4 and retry_count < max_retries:
        print(f"  [路由] 检索质量中 ({score:.2f})，第{retry_count+1}次改写")
        return "rewrite_query"

    # 低质量 或 改写次数用完：Web 搜索
    if score < 0.4:
        print(f"  [路由] 检索质量低 ({score:.2f} < 0.4) → Web 搜索")
        return "web_search"

    # 改写次数用完但质量仍不够：强制生成（循环守卫）
    print(f"  [路由] 改写{retry_count}次仍不够，强制生成")
    return "generate"


def route_by_generation_quality(
    state: RAGState,
) -> Literal["output", "regenerate", "refuse"]:
    """
    根据生成质量路由 —— 幻觉检测。

    决策逻辑：
      score ≥ 0.7  → 输出（质量合格）
      0.4 ≤ score < 0.7 → 重新生成（可能有幻觉）
      score < 0.4  → 拒绝回答（严重幻觉）

    循环守卫：重新生成最多1次
    """
    score = state.get("generation_score", 0)
    retry_count = state.get("retry_count", 0)

    if score >= 0.7:
        print(f"  [路由] 生成质量高 ({score:.2f} ≥ 0.7) → 输出")
        return "output"

    if score >= 0.4 and retry_count < 3:
        print(f"  [路由] 生成质量中 ({score:.2f})，重新生成")
        return "regenerate"

    print(f"  [路由] 生成质量低 ({score:.2f} < 0.4) → 拒绝回答")
    return "refuse"


# ============================================================
# 6. 输出节点
# ============================================================

def output_answer(state: RAGState) -> dict:
    """输出最终回答"""
    log = list(state.get("route_log", []))
    print(f"  [输出] 回答已生成，来源: {state.get('source', 'unknown')}")
    return {
        "route_log": log + [f"输出回答 (来源: {state.get('source', 'unknown')})"],
    }


def refuse_answer(state: RAGState) -> dict:
    """拒绝回答（幻觉风险太高）"""
    log = list(state.get("route_log", []))
    print(f"  [拒绝] 无法生成可靠回答")
    return {
        "generation": "抱歉，我无法基于现有信息给出可靠的回答。建议您查阅官方文档或咨询专家。",
        "route_log": log + ["拒绝回答: 幻觉风险过高"],
    }


# ============================================================
# 7. 构建 Adaptive RAG 图
# ============================================================

def build_adaptive_rag() -> StateGraph:
    """构建 Adaptive RAG 工作流"""
    builder = StateGraph(RAGState)

    # 添加节点
    builder.add_node("retrieve", retrieve_from_vectordb)
    builder.add_node("web_search", web_search)
    builder.add_node("rewrite_query", rewrite_query)
    builder.add_node("generate", generate_answer)
    builder.add_node("output", output_answer)
    builder.add_node("refuse", refuse_answer)

    # 入口 → 向量检索
    builder.add_edge(START, "retrieve")

    # ✅ 核心路由1：检索质量决策
    builder.add_conditional_edges(
        "retrieve",
        route_by_retrieval_quality,
        {
            "generate": "generate",           # 高质量 → 直接生成
            "rewrite_query": "rewrite_query",  # 中等 → 改写查询
            "web_search": "web_search",        # 低质量 → Web 搜索
        },
    )

    # 查询改写后 → 重新检索
    builder.add_edge("rewrite_query", "retrieve")

    # Web 搜索后 → 直接生成
    builder.add_edge("web_search", "generate")

    # ✅ 核心路由2：生成质量决策（幻觉检测）
    builder.add_conditional_edges(
        "generate",
        route_by_generation_quality,
        {
            "output": "output",         # 质量合格 → 输出
            "regenerate": "generate",   # 可能有幻觉 → 重新生成
            "refuse": "refuse",         # 严重幻觉 → 拒绝回答
        },
    )

    # 输出 / 拒绝 → END
    builder.add_edge("output", END)
    builder.add_edge("refuse", END)

    return builder.compile()


# ============================================================
# 8. 测试运行
# ============================================================

def main():
    """测试 Adaptive RAG"""
    print("=" * 60)
    print("条件分支策略 - 场景6：Adaptive RAG 条件路由")
    print("=" * 60)

    graph = build_adaptive_rag()

    # 测试不同场景
    test_cases = [
        {"question": "什么是 RAG？", "seed": 42},
        {"question": "LangGraph 的 Send API 怎么用？", "seed": 100},
        {"question": "量子计算对密码学的影响", "seed": 77},
    ]

    for tc in test_cases:
        print(f"\n{'─' * 60}")
        print(f"问题: {tc['question']} (seed={tc['seed']})")
        print(f"{'─' * 60}")

        random.seed(tc["seed"])

        initial_state: RAGState = {
            "question": tc["question"],
            "rewritten_question": "",
            "documents": [],
            "generation": "",
            "retrieval_score": 0.0,
            "generation_score": 0.0,
            "retry_count": 0,
            "max_retries": 2,
            "source": "",
            "route_log": [],
        }

        result = graph.invoke(initial_state)

        print(f"\n  最终回答: {result['generation'][:80]}...")
        print(f"  数据来源: {result['source']}")
        print(f"  检索分数: {result['retrieval_score']:.2f}")
        print(f"  生成分数: {result['generation_score']:.2f}")
        print(f"  路由日志:")
        for entry in result["route_log"]:
            print(f"    - {entry}")


if __name__ == "__main__":
    main()
```

## 代码详解

### 1. 两个核心路由点

Adaptive RAG 的精髓在于两个条件分支：

```python
# 路由点1：检索质量 → 决定信息来源
builder.add_conditional_edges("retrieve", route_by_retrieval_quality, {
    "generate": "generate",            # 高质量 → 生成
    "rewrite_query": "rewrite_query",  # 中等 → 改写
    "web_search": "web_search",        # 低质量 → Web
})

# 路由点2：生成质量 → 决定是否输出
builder.add_conditional_edges("generate", route_by_generation_quality, {
    "output": "output",        # 合格 → 输出
    "regenerate": "generate",  # 幻觉 → 重试
    "refuse": "refuse",        # 严重 → 拒绝
})
```

### 2. 循环守卫的位置

```python
# 改写循环守卫：在 route_by_retrieval_quality 中
if score >= 0.4 and retry_count < max_retries:
    return "rewrite_query"  # 还能改写
# else → 强制进入 generate 或 web_search

# 生成循环守卫：在 route_by_generation_quality 中
if score >= 0.4 and retry_count < 3:
    return "regenerate"  # 还能重试
# else → refuse
```

### 3. 完整路由流程图

```
START
  │
  ▼
retrieve (向量检索)
  │
  ├── score ≥ 0.7 ──────────────────────→ generate
  │
  ├── 0.4 ≤ score < 0.7 ──→ rewrite_query ──→ retrieve (循环)
  │   (retry < max)                              │
  │                                              └── 循环守卫：
  │                                                  retry ≥ max → generate
  │
  └── score < 0.4 ──────────→ web_search ──→ generate
                                                │
                                                ▼
                                          generate (生成)
                                                │
                                          ├── score ≥ 0.7 → output → END
                                          │
                                          ├── 0.4~0.7 → regenerate (循环)
                                          │   (retry < 3)
                                          │
                                          └── score < 0.4 → refuse → END
```

## 与 LangGraph 官方 Adaptive RAG 的对比

LangGraph 官方教程中的 Adaptive RAG 示例（langgraph/examples/rag）使用了类似的模式：

| 对比项 | 官方示例 | 本场景 |
|--------|----------|--------|
| 检索评估 | 用 LLM 打分（structured output） | 模拟分数（便于演示路由逻辑） |
| 查询改写 | LLM 改写 | 模拟改写策略 |
| Web 搜索 | Tavily Search API | 模拟搜索结果 |
| 幻觉检测 | LLM 判断（grounded in documents?） | 模拟分数 |
| 循环守卫 | 依赖 recursion_limit | 显式 retry_count 计数器 |
| 路由方式 | add_conditional_edges | add_conditional_edges（相同） |

**关键差异**：

1. 官方示例依赖 `recursion_limit` 作为全局循环守卫，本场景使用显式的 `retry_count` 计数器，更精细可控。

2. 官方示例的检索评估用 LLM 做 structured output（返回 "yes"/"no"），本场景用数值分数，支持更细粒度的路由。

3. 本场景将「拒绝回答」作为独立路径，官方示例通常没有这个分支。

## 生产化改造要点

### 1. 用真实的检索评估替换模拟分数


### 1. 检索质量阈值要根据业务调整

```python
# 高精度场景（医疗、法律）：提高阈值
HIGH_THRESHOLD = 0.85   # 只有非常相关才直接生成
```

### 2. 改写次数不宜过多

```python
    return "refuse"  # 明确告诉用户无法回答
```

## 总结

**Adaptive RAG 的条件分支核心**：

1. 两个路由点：检索质量路由 + 生成质量路由
2. 三条检索路径：直接生成 / 改写重检索 / Web 搜索回退
3. 三条生成路径：输出 / 重新生成 / 拒绝回答
4. 循环守卫：改写次数和重生成次数都有上限

**与场景5的关系**：场景5的三级回退是通用模式，场景6将其应用到 RAG 的具体场景中——检索失败时的回退（向量库 → 改写 → Web）和生成失败时的回退（重试 → 拒绝）。
