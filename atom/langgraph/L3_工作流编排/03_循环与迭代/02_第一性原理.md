## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而不是通过类比或经验来理解。

在物理学中，第一性原理意味着从基本的物理定律出发推导现象。在工程中，意味着从最基础的需求和约束出发设计系统。

对于 LangGraph 的循环与迭代，我们要问：**什么是循环的本质？为什么 AI 工作流需要循环？如何从零构建循环机制？**

## 循环与迭代的第一性原理

### 1. 最基础的定义

**循环 = 边指向图中已存在的节点，形成有向环**

仅此而已！没有更基础的了。

在 LangGraph 中，图由节点（node）和边（edge）组成。正常的边从一个节点指向下游节点，形成有向无环图（DAG）。而循环的本质就是：**某条边的终点是图中已经执行过的节点**，让执行流"回头"，形成有向环。

```
DAG（无循环）:          有向环（循环）:
A → B → C → END        A → B → C
                              ↑     ↓
                              └─────┘
```

这个定义包含三个关键要素：
1. **边（Edge）**：连接两个节点的有向通道
2. **已存在的节点**：目标节点已经在图中定义过
3. **有向环**：执行流可以回到之前经过的节点

从源码角度看，LangGraph 对循环没有任何特殊限制。`add_edge()` 只禁止两件事：`END` 不能作为起点，`START` 不能作为终点。除此之外，任何节点之间都可以连边，包括"回边"（back edge）。

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L785-L837]
def add_edge(self, start_key, end_key):
    if start_key == END:
        raise ValueError("END cannot be a start node")
    if end_key == START:
        raise ValueError("START cannot be an end node")
    # 没有其他限制！允许 B → A 这样的回边
    self.edges.add((start_key, end_key))
```

**关键洞察**：循环不是 LangGraph 的"特殊功能"，而是图结构的自然属性。只要边可以指向任意节点，循环就自然存在。LangGraph 的设计哲学是：**不限制图的拓扑结构，而是通过递归限制来保证安全性**。

### 2. 为什么需要循环？

**核心问题：AI 任务很少一次就能完成，需要反复尝试和改进。**

让我们从最基本的场景思考：

#### 场景 1：一次性执行的局限

```python
# 一次性执行：输入 → 处理 → 输出
def generate_code(requirement):
    code = llm.invoke(f"Write code for: {requirement}")
    return code

# 问题：生成的代码可能有 bug，但没有机会修正
result = generate_code("写一个排序算法")
# 如果代码有错误？只能重新调用整个流程
```

**问题**：
- 质量不可控：LLM 的输出不确定，一次生成可能不够好
- 无法自我修正：没有反馈机制，错误无法被发现和修复
- 复杂任务无法分步：需要多轮推理的任务无法在单次调用中完成

#### 场景 2：循环带来的质量飞跃

```python
# 循环执行：输入 → 生成 → 检查 → (不合格) → 重新生成 → 检查 → (合格) → 输出
def generate_and_refine(requirement, max_attempts=3):
    for attempt in range(max_attempts):
        code = llm.invoke(f"Write code for: {requirement}")
        errors = check_code(code)
        if not errors:
            return code  # 通过检查，返回结果
        requirement = f"{requirement}\n\nPrevious errors: {errors}\nPlease fix."
    return code  # 达到最大尝试次数，返回最后结果
```

**优势**：
- 质量可控：通过反复检查和修正，输出质量显著提升
- 自我修正：错误被发现后可以在下一轮修复
- 渐进式改进：每一轮都在上一轮的基础上优化

**但普通的 for 循环还不够！** 在复杂的 AI 工作流中，我们还需要：
1. **动态决策**：根据中间结果决定是否继续循环、走哪条路径
2. **状态持久化**：每一轮的中间结果需要被保存，支持断点恢复
3. **安全边界**：防止无限循环消耗资源
4. **人机协作**：在循环中插入人工审核节点

这就是为什么需要 **LangGraph 的循环与迭代机制**——它不是简单的 for 循环，而是一个完整的状态化循环执行引擎。

### 3. 循环的三层价值

#### 价值 1：迭代优化 - 反复改进输出质量

**从"一次生成"到"渐进式改进"**

这是循环最直观的价值。AI 的输出具有不确定性，通过"生成 → 评估 → 修正"的循环，可以逐步逼近理想结果。

```python
# 自校正循环：Generator-Critic 模式
#
#   ┌──────────────┐
#   │   generate    │ ← 生成内容
#   └──────┬───────┘
#          ↓
#   ┌──────────────┐
#   │   evaluate    │ ← 评估质量
#   └──────┬───────┘
#          ↓
#   ┌──────────────┐     不合格
#   │   route       │ ──────────→ 回到 generate
#   └──────┬───────┘
#          ↓ 合格
#        [END]

def route(state) -> Literal["generate", END]:
    if state["score"] < 0.8:
        return "generate"  # 回到生成节点，形成循环
    return END             # 质量达标，结束
```

**实际案例**：
- **代码生成与修复**：生成代码 → 运行测试 → 发现 bug → 修复代码 → 再次测试
- **文档摘要优化**：生成摘要 → 检查覆盖率 → 补充遗漏 → 再次检查
- **RAG 答案精炼**：检索 → 生成答案 → 评估相关性 → 调整检索策略 → 重新检索

#### 价值 2：动态决策 - 根据中间结果调整策略

**从"固定流程"到"自适应工作流"**

循环不仅仅是"重复做同一件事"，更重要的是在每一轮中根据当前状态做出不同的决策。

```python
# ReAct 循环：reason → act → observe → reason
#
#   ┌──────────────┐
#   │   reason      │ ← 分析当前状态，决定下一步行动
#   └──────┬───────┘
#          ↓
#   ┌──────────────┐
#   │   act         │ ← 执行行动（调用工具、查询数据库等）
#   └──────┬───────┘
#          ↓
#   ┌──────────────┐
#   │   observe     │ ← 观察行动结果
#   └──────┬───────┘
#          ↓
#   ┌──────────────┐     需要更多信息
#   │   decide      │ ──────────────→ 回到 reason
#   └──────┬───────┘
#          ↓ 信息充足
#        [END]

def decide(state) -> Literal["reason", END]:
    if state["has_enough_info"]:
        return END
    return "reason"  # 继续推理-行动循环
```

**实际案例**：
- **多步推理 Agent**：分析问题 → 选择工具 → 执行 → 分析结果 → 决定是否需要更多工具
- **RAG 自适应检索**：检索 → 评估结果 → 改写查询 → 重新检索 → 评估 → 满意则生成答案
- **多轮对话**：理解用户意图 → 提问澄清 → 获取回答 → 判断是否需要更多信息

#### 价值 3：自主代理 - Agent 自主决定何时停止

**从"人工控制"到"自主决策"**

循环的最高级价值是让 Agent 具备自主性——它自己决定什么时候任务完成了，而不是由外部硬编码的条件来控制。

```python
# Agent 自主循环
#
#   ┌──────────────┐
#   │   agent       │ ← LLM 分析状态，决定行动
#   └──────┬───────┘
#          ↓
#   ┌──────────────┐
#   │   tools       │ ← 执行 Agent 选择的工具
#   └──────┬───────┘
#          ↓
#   ┌──────────────┐     有 tool_calls
#   │   should_     │ ──────────────→ 回到 agent
#   │   continue    │
#   └──────┬───────┘
#          ↓ 无 tool_calls（Agent 认为任务完成）
#        [END]

def should_continue(state) -> Literal["agent", END]:
    last_message = state["messages"][-1]
    if last_message.tool_calls:
        return "agent"  # Agent 还想调用工具，继续循环
    return END          # Agent 没有工具调用，认为任务完成
```

**关键区别**：
- 价值 1（迭代优化）：循环由**质量评估**驱动——"结果够好了吗？"
- 价值 2（动态决策）：循环由**信息充分性**驱动——"信息够了吗？"
- 价值 3（自主代理）：循环由 **Agent 自身判断**驱动——"我觉得完成了吗？"

**实际案例**：
- **ReAct Agent**：Agent 自主决定调用哪些工具、调用几次、何时给出最终答案
- **研究助手**：Agent 自主搜索、阅读、总结，直到认为研究充分
- **代码助手**：Agent 自主编写、测试、调试，直到所有测试通过

### 4. 从第一性原理推导 LangGraph 的循环应用

让我们从最基础的需求出发，逐步推导出 LangGraph 的循环机制：

**推理链：**

```
1. 【基础需求】AI 任务需要反复尝试和改进
   ↓
   问题：如何让执行流"回头"？

2. 【回边机制】允许边指向已存在的节点
   ↓
   解决方案：add_edge("B", "A") 或条件边返回已存在的节点名
   底层实现：写入 branch:to:{node} channel 触发目标节点

3. 【终止条件】循环必须有退出机制，否则无限循环
   ↓
   解决方案：条件边函数在满足条件时返回 END
   def route(state): return END if done else "loop_node"

4. 【安全边界】条件可能永远不满足怎么办？
   ↓
   解决方案：递归限制（recursion_limit）
   graph.invoke(inputs, {"recursion_limit": 25})

5. 【优雅降级】达到限制时不应该直接崩溃
   ↓
   解决方案：RemainingSteps 管理值
   节点可以感知剩余步数，在最后几步返回部分结果

6. 【状态持久化】循环中的中间状态需要保存
   ↓
   解决方案：每个 superstep 后自动保存 checkpoint
   支持断点恢复、时间旅行

7. 【人机协作】某些循环需要人工介入
   ↓
   解决方案：interrupt_before / interrupt_after
   在指定节点前/后暂停，等待人工输入

8. 【动态路由】循环中的路径不是固定的
   ↓
   解决方案：
   - 条件边：add_conditional_edges(source, router_fn)
   - Command：节点返回 Command(goto="target")
   - Send：动态创建并行任务

9. 【复杂场景】多个循环嵌套、循环中包含并行
   ↓
   解决方案：
   - 子图（Subgraph）：每个子图有独立的循环
   - 等待边：add_edge(["c", "d"], "a") 多节点同步后再循环
   - BSP 模型：superstep 机制天然支持复杂拓扑

10. 【最终应用】构建自主 AI Agent
    - ReAct 循环：推理-行动-观察
    - 自校正循环：生成-评估-修正
    - 多 Agent 协作循环：分配-执行-汇总-再分配
    - RAG 迭代检索：查询-检索-评估-改写-重新检索
```

**关键洞察**：

1. **循环的底层统一模型**
   无论使用哪种方式构建循环（静态边、条件边、Command），底层都归结为同一个机制：节点执行后写入 `branch:to:{node_name}` channel，触发目标节点在下一个 superstep 中执行。
   [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L1297]

2. **PregelLoop 是循环的执行引擎**
   LangGraph 的执行引擎 PregelLoop 通过 `tick() → runner.tick() → after_tick()` 的循环驱动整个图的执行。每次 `tick()` 检查是否有待执行的任务，有则继续，无则结束。
   [来源: sourcecode/langgraph/libs/langgraph/langgraph/pregel/_loop.py L459]

3. **递归限制是唯一的安全阀**
   LangGraph 不会在编译时检测循环（因为条件边的路由是运行时决定的），而是通过 `step > stop` 的运行时检查来防止无限循环。默认 `recursion_limit=25`，即最多执行 26 个 superstep。
   [来源: sourcecode/langgraph/libs/langgraph/langgraph/pregel/_loop.py L1119-L1120]

4. **循环 ≠ 重复，循环 = 状态驱动的迭代**
   每一轮循环中，状态都在变化。节点看到的是更新后的状态，因此每一轮的行为可能完全不同。这是 LangGraph 循环与简单 for 循环的本质区别。

---

### 5. 一句话总结第一性原理

**循环与迭代是通过边指向已存在节点形成有向环，由 PregelLoop 引擎驱动 superstep 迭代执行，配合条件边实现动态终止、递归限制保证安全边界的机制，使 AI 工作流具备迭代优化、动态决策和自主代理的能力。**

## 核心洞察

### 洞察 1：循环是 LangGraph 区别于 DAG 框架的核心能力

**为什么这很重要？**

大多数工作流引擎（如 Airflow、Prefect）只支持 DAG（有向无环图），不允许循环。这意味着所有的执行路径必须在编译时确定，无法实现"根据结果决定是否重试"这样的动态行为。

LangGraph 从设计之初就支持循环，这是它被创建的核心原因之一。正如 LangGraph 官方博客所说：

> "LangGraph is specifically designed for creating cyclic graphs, supporting the for-loop style iteration common in agent runtimes."

**启示**：
- 如果你的工作流不需要循环，可能不需要 LangGraph
- 如果你需要 Agent 自主决策、迭代优化、多轮交互，LangGraph 的循环能力是关键

### 洞察 2：递归限制不是性能限制，而是安全机制

**常见误解**：递归限制是为了节省计算资源。

**真相**：递归限制是为了防止无限循环。在 AI 工作流中，条件边的路由由 LLM 决定，而 LLM 的输出不可预测。如果 LLM 始终返回"继续"，没有递归限制就会无限循环。

```python
# 危险：没有安全边界的循环
def route(state):
    # 如果 LLM 始终认为需要更多信息...
    if llm.invoke("Do we need more info?") == "yes":
        return "search"  # 永远不会到达 END！
    return END

# 安全：使用 RemainingSteps 主动监控
def route(state):
    if state["remaining_steps"] <= 2:
        return END  # 快到限制了，优雅退出
    if needs_more_info(state):
        return "search"
    return END
```

**最佳实践**：
- 始终设置合理的 `recursion_limit`
- 使用 `RemainingSteps` 实现优雅降级，而不是等到 `GraphRecursionError`
- 在条件边中加入多重退出条件

### 洞察 3：每一轮循环都是一个完整的 superstep

**这意味着什么？**

LangGraph 基于 Pregel 的 BSP（Bulk Synchronous Parallel）模型。每一轮循环不是简单的"再执行一次节点"，而是一个完整的 superstep：

1. `tick()`：检查步数限制，准备待执行的任务
2. `runner.tick()`：并行执行所有任务
3. `after_tick()`：应用写入到 channels，保存 checkpoint，递增 step

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/pregel/main.py L2643-L2674]
while loop.tick():                    # 1. 检查是否继续
    for _ in runner.tick(tasks):      # 2. 执行任务
        yield from _output(...)
    loop.after_tick()                 # 3. 应用写入，保存状态

if loop.status == "out_of_steps":
    raise GraphRecursionError(...)    # 超出限制则报错
```

**实际影响**：
- 每一轮循环后状态都被持久化（checkpoint）
- 可以在任意一轮循环后暂停和恢复
- 循环中的并行任务在同一个 superstep 内执行
- 调试时可以查看每一轮循环的状态快照

## 思维模型

### 模型 1：自校正循环（Generator-Critic）

```
┌──────────────────────────────────────────┐
│                                          │
│   ┌──────────┐    ┌──────────┐          │
│   │ Generator │───→│  Critic  │          │
│   │ (生成器)  │    │ (评审器) │          │
│   └──────────┘    └────┬─────┘          │
│        ↑               │                │
│        │          ┌────┴─────┐          │
│        │          │  Route   │          │
│        │          └────┬─────┘          │
│        │    不合格     │     合格        │
│        └───────────────┘     │          │
│                              ↓          │
│                           [END]         │
└──────────────────────────────────────────┘
```

### 模型 2：ReAct 循环（Reason-Act-Observe）

```
┌──────────────────────────────────────────┐
│                                          │
│   ┌──────────┐    ┌──────────┐          │
│   │  Reason   │───→│   Act    │          │
│   │ (推理)    │    │ (行动)   │          │
│   └──────────┘    └────┬─────┘          │
│        ↑               │                │
│        │          ┌────┴─────┐          │
│        │          │ Observe  │          │
│        │          │ (观察)   │          │
│        │          └────┬─────┘          │
│        │    继续       │     完成        │
│        └───────────────┘     │          │
│                              ↓          │
│                           [END]         │
└──────────────────────────────────────────┘
```

### 模型 3：PregelLoop 执行模型

```
PregelLoop.__enter__()
  ├── step = checkpoint_step + 1
  ├── stop = step + recursion_limit + 1
  │
  ├─→ tick()
  │     ├── step > stop? → "out_of_steps" → GraphRecursionError
  │     ├── prepare_next_tasks() → 根据 channel 确定待执行节点
  │     ├── 没有任务? → "done" → 正常结束
  │     └── return True（继续）
  │           │
  │     runner.tick() → 执行所有任务
  │           │
  │     after_tick()
  │           ├── apply_writes() → 写入 channels
  │           │     └── 写入 "branch:to:X" → X 节点下一轮被触发
  │           ├── _put_checkpoint() → 保存状态, step += 1
  │           └── 检查 interrupt
  │           │
  └───────────┘ (回到 tick)
```

## 实践指南

### 何时使用循环？

**适合循环的场景**：
1. 输出质量需要迭代优化（自校正、代码修复）
2. 任务需要多步推理（ReAct Agent、研究助手）
3. 需要根据中间结果动态调整策略（自适应检索）
4. Agent 需要自主决定何时完成（工具调用循环）

**不适合循环的场景**：
1. 任务是确定性的，一次就能完成（简单的数据转换）
2. 流程是固定的，不需要动态决策（ETL 管道）
3. 没有质量评估机制（无法判断是否需要重试）

### 循环终止的三种策略

| 策略 | 方法 | 推荐度 | 适用场景 |
|------|------|--------|----------|
| 条件终止 | 路由函数返回 `END` | 最常用 | 有明确终止条件 |
| 主动监控 | `RemainingSteps` 管理值 | 推荐 | 需要优雅降级 |
| 被动限制 | `recursion_limit` 配置 | 兜底 | 所有循环都应设置 |

```python
# 推荐：三种策略组合使用
def route(state) -> Literal["agent", END]:
    # 策略 1：条件终止 - 任务完成
    if state["task_completed"]:
        return END

    # 策略 2：主动监控 - 快到限制了
    if state["remaining_steps"] <= 2:
        return END  # 优雅退出，返回部分结果

    # 继续循环
    return "agent"

# 策略 3：被动限制 - 兜底保护
result = graph.invoke(inputs, {"recursion_limit": 25})
```
