## 卡片 1: Tool 是什么？

**一句话：** Tool = 函数 + 描述 + Schema，让 LLM 能"动手做事"。

**直觉理解：**

LLM 是一个只会说话的大脑——它能分析问题、做出决策，但不能上网搜索、不能查数据库、不能调 API。Tool 就是给这个大脑配的"手"，让它从"只会说"变成"能做事"。

```
没有 Tool:  用户问天气 → LLM 凭记忆瞎猜 → "北京大概 20 度吧"
有了 Tool:  用户问天气 → LLM 调用 get_weather("北京") → "北京今天晴，25°C"
```

**Tool 的三要素：**

```python
from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:    # ← 函数：实际执行逻辑
    """查询指定城市的实时天气信息"""    # ← 描述：告诉 LLM 这个工具干什么
    return call_weather_api(city)       # ← Schema：从类型注解自动生成参数说明
```

| 要素 | 作用 | 给谁看 |
|------|------|--------|
| 函数 | 实际执行逻辑 | 框架执行 |
| 描述 (description) | 告诉 LLM 什么时候用 | LLM 决策 |
| Schema (args_schema) | 告诉 LLM 需要什么参数 | LLM 填参 |

**应用：** 在 RAG 系统中，向量检索器、搜索引擎、计算器都可以包装成 Tool，让 Agent 自主决定何时调用。Tool 是 Agent 循环中 "Act" 步骤的具体实现——没有 Tool，Agent 就是空转。

## 卡片 2: @tool 装饰器 — 一行代码创建 Tool

**一句话：** @tool + 类型注解 + docstring = 完整的 Tool，90% 场景用这个就够了。

**最简示例：**

```python
from langchain_core.tools import tool

@tool
def multiply(a: int, b: int) -> int:
    """计算两个整数的乘积"""
    return a * b

# 验证 Tool 已创建
print(multiply.name)         # "multiply"
print(multiply.description)  # "计算两个整数的乘积"
print(multiply.args_schema.model_json_schema())
# → {"properties": {"a": {"type": "integer"}, "b": {"type": "integer"}}, ...}

# 直接调用
result = multiply.invoke({"a": 3, "b": 7})
print(result)  # 21
```

**三个关键点：**

1. **类型注解必须有**：`a: int, b: int` 告诉 LLM 参数类型，没有注解 LLM 不知道传什么
2. **docstring 就是 description**：LLM 靠这段文字决定是否调用这个工具
3. **返回值自动变成字符串**：Tool 的输出最终会放进 ToolMessage.content，所以会被转为 str

**自定义名称和返回类型：**

```python
@tool("search_engine", response_format="content_and_artifact")
def search(query: str) -> tuple[str, list[dict]]:
    """搜索互联网获取最新信息"""
    results = do_search(query)
    summary = f"找到 {len(results)} 条结果"
    return summary, results  # 摘要给模型，完整数据给用户
```

**应用：** @tool 是日常开发中最常用的 Tool 定义方式。写一个函数、加一行装饰器，就能让 Agent 调用任意外部服务。

## 卡片 3: bind_tools() — 把工具交给模型

**一句话：** bind_tools() 将工具的 JSON Schema 注入模型上下文，让模型知道"我有哪些工具可用"。

**核心流程：**

```python
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

@tool
def search(query: str) -> str:
    """搜索互联网获取最新信息"""
    return do_search(query)

@tool
def calculate(expression: str) -> str:
    """计算数学表达式"""
    return str(eval(expression))

# 关键一步：把工具绑定到模型
llm = ChatOpenAI(model="gpt-4o")
llm_with_tools = llm.bind_tools([search, calculate])
```

**bind_tools() 做了什么？**

```
绑定前：LLM 只能生成文本回答
绑定后：LLM 可以选择"调用工具"或"直接回答"

底层原理：
  @tool search → JSON Schema → 注入到 API 请求的 tools 参数
  @tool calculate → JSON Schema → 注入到 API 请求的 tools 参数
```

模型收到的信息类似：

```json
{
  "tools": [
    {"type": "function", "function": {"name": "search", "description": "搜索互联网获取最新信息", "parameters": {...}}},
    {"type": "function", "function": {"name": "calculate", "description": "计算数学表达式", "parameters": {...}}}
  ]
}
```

**tool_choice 控制调用行为：**

```python
# 模型自行决定是否调用（默认）
llm.bind_tools(tools, tool_choice="auto")

# 必须调用至少一个工具
llm.bind_tools(tools, tool_choice="any")

# 必须调用指定工具
llm.bind_tools(tools, tool_choice="search")
```

**应用：** bind_tools() 是函数调用的起点。LangChain 自动处理 OpenAI / Anthropic / Google 等不同提供商的格式差异，你只需要写一次 bind_tools()。

## 卡片 4: tool_calls — 模型的调用请求

**一句话：** 模型不直接执行工具，而是返回结构化的调用请求——"我想调用 X 工具，参数是 Y"。

**关键认知：**

```
❌ 错误理解：LLM 调用工具 = LLM 执行了函数
✅ 正确理解：LLM 调用工具 = LLM 输出了一个 JSON 请求，由框架代为执行
```

**实际代码：**

```python
llm_with_tools = ChatOpenAI(model="gpt-4o").bind_tools([search, calculate])

response = llm_with_tools.invoke("北京今天天气怎么样？")

# response 是 AIMessage，不是天气结果！
print(response.content)      # ""（空字符串，模型选择调用工具而非直接回答）
print(response.tool_calls)
# [
#   {
#     "name": "search",
#     "args": {"query": "北京今天天气"},
#     "id": "call_abc123",
#     "type": "tool_call"
#   }
# ]
```

**tool_calls 的三个字段：**

| 字段 | 含义 | 示例 |
|------|------|------|
| `name` | 要调用哪个工具 | `"search"` |
| `args` | 传什么参数 | `{"query": "北京今天天气"}` |
| `id` | 本次调用的唯一标识 | `"call_abc123"` |

**多工具调用：** 模型可以一次返回多个 tool_calls，表示"我需要同时调用多个工具"：

```python
response = llm_with_tools.invoke("北京和上海今天天气分别怎么样？")
# response.tool_calls 可能包含两个调用请求
```

**应用：** 理解 tool_calls 是理解整个函数调用机制的关键。模型只负责"决策"，框架负责"执行"——这种分离让系统更安全、更可控。

## 卡片 5: ToolMessage — 工具执行结果回传

**一句话：** 工具执行后，通过 ToolMessage 把结果传回模型，模型再基于结果生成最终回答。

**完整消息链路：**

```
HumanMessage("北京天气？")
    ↓
AIMessage(tool_calls=[{name: "search", args: {...}, id: "call_123"}])
    ↓
ToolMessage(content="北京今天晴，25°C", tool_call_id="call_123")  ← 这一步
    ↓
AIMessage("北京今天天气晴朗，气温 25°C，适合出行。")
```

**代码示例：**

```python
from langchain_core.messages import HumanMessage, ToolMessage

# 1. 用户提问
messages = [HumanMessage("北京今天天气怎么样？")]

# 2. 模型返回 tool_calls
ai_msg = llm_with_tools.invoke(messages)
messages.append(ai_msg)

# 3. 执行工具，构造 ToolMessage
for tc in ai_msg.tool_calls:
    result = tool_map[tc["name"]].invoke(tc["args"])
    messages.append(ToolMessage(
        content=str(result),
        tool_call_id=tc["id"],  # 必须与 tool_call 的 id 对应！
    ))

# 4. 模型基于工具结果生成最终回答
final = llm_with_tools.invoke(messages)
print(final.content)  # "北京今天天气晴朗，气温 25°C..."
```

**关键细节：** `tool_call_id` 必须与对应的 `tool_calls[i]["id"]` 一致。这是模型关联"请求"和"响应"的唯一依据。如果 id 不匹配，模型会困惑。

**应用：** 这个 4 步消息链路（Human → AI(tool_calls) → Tool → AI(最终回答)）是所有 Tool 调用的标准流程。AgentExecutor 和 LangGraph 都是在自动化这个循环。

## 卡片 6: Schema — 工具的"说明书"

**一句话：** Schema 告诉 LLM 工具需要什么参数、每个参数是什么类型和含义，是 LLM 正确填参的依据。

**自动推断 vs 手动定义：**

```python
# 方式 1：从类型注解自动推断（简单场景）
@tool
def search(query: str, top_k: int = 5) -> str:
    """搜索知识库"""
    ...

# 自动生成的 Schema：
# {"query": {"type": "string"}, "top_k": {"type": "integer", "default": 5}}
```

```python
# 方式 2：用 Pydantic BaseModel 手动定义（需要详细描述）
from pydantic import BaseModel, Field

class SearchInput(BaseModel):
    query: str = Field(description="自然语言搜索查询")
    top_k: int = Field(default=5, description="返回结果数量，1-20")
    category: str = Field(default="all", description="搜索类别：all/tech/hr/finance")

@tool(args_schema=SearchInput)
def search(query: str, top_k: int = 5, category: str = "all") -> str:
    """在公司知识库中搜索文档"""
    ...
```

**方式 3：从 docstring 自动提取参数描述：**

```python
@tool(parse_docstring=True)
def search(query: str, top_k: int = 5) -> str:
    """在公司知识库中搜索文档。

    Args:
        query: 自然语言搜索查询，描述要查找的信息
        top_k: 返回结果数量，默认 5
    """
    ...
# parse_docstring=True 会自动把 Args 中的描述提取到 Schema
```

**Schema 质量直接影响调用准确率：**

| Schema 质量 | LLM 行为 |
|-------------|----------|
| 无参数描述 | LLM 猜测参数含义，可能传错 |
| 有类型无描述 | LLM 知道类型，但不确定语义 |
| 类型 + 描述 + 约束 | LLM 精准填参，调用成功率最高 |

**应用：** 在 RAG 场景中，检索工具的 Schema 设计直接影响检索质量。比如 `top_k` 的描述中写明"推荐 3-5"，模型就不会传一个 100 进来。

## 卡片 7: 三种定义方式对比

**一句话：** @tool（简洁）→ StructuredTool（自定义）→ BaseTool（完全控制），复杂度递增，灵活性递增。

**对比表格：**

| 维度 | @tool 装饰器 | StructuredTool | BaseTool 子类 |
|------|-------------|----------------|---------------|
| 代码量 | 最少（3 行） | 中等（10 行） | 最多（20+ 行） |
| 灵活性 | 中等 | 较高 | 最高 |
| 自定义 Schema | 通过 args_schema 参数 | 原生支持 | 原生支持 |
| 状态管理 | 不支持 | 不支持 | 支持（类属性） |
| 异步支持 | 自动（async def） | 需要 coroutine 参数 | 实现 _arun |
| 适用场景 | 90% 日常开发 | 需要自定义 Schema | 复杂工具、需要状态 |

**三种方式代码对比：**

```python
# 方式 1: @tool — 最简洁，日常首选
@tool
def search(query: str) -> str:
    """搜索互联网"""
    return do_search(query)

# 方式 2: StructuredTool — 需要自定义 Schema 时
from langchain_core.tools import StructuredTool
search_tool = StructuredTool.from_function(
    func=do_search,
    name="search",
    description="搜索互联网",
    args_schema=SearchInput,
)

# 方式 3: BaseTool — 需要状态管理或完全控制时
from langchain_core.tools import BaseTool

class SearchTool(BaseTool):
    name: str = "search"
    description: str = "搜索互联网"
    api_key: str  # 可以持有状态

    def _run(self, query: str) -> str:
        return do_search(query, api_key=self.api_key)
```

**选择决策树：**

```
需要定义 Tool？
├─ 简单函数包装 → @tool（90% 的情况）
├─ 需要自定义参数 Schema → @tool(args_schema=...) 或 StructuredTool
└─ 需要持有状态 / 复杂初始化 → BaseTool 子类
```

**应用：** 实际项目中，先用 @tool 快速实现，遇到瓶颈再升级到 StructuredTool 或 BaseTool。不要过度设计。

## 卡片 8: 工具选择策略

**一句话：** 工具太多时，用策略动态筛选，让模型从"大海捞针"变成"精准点菜"。

**问题：** 给模型 20 个工具，选择准确率可能降到 60%。每个工具 Schema 约 100-200 tokens，20 个就是 2000-4000 tokens 的额外成本。

**四层解决方案：**

| 层级 | 方案 | 适用场景 |
|------|------|----------|
| L1 | `tool_choice` 参数 | 强制/禁止调用特定工具 |
| L2 | 手动过滤工具列表 | 基于规则的简单筛选 |
| L3 | 自定义 Middleware | 基于上下文动态过滤 |
| L4 | `LLMToolSelectorMiddleware` | 用小模型智能选择 |

**LLMToolSelectorMiddleware 示例：**

```python
from langchain.agents import create_agent
from langchain.agents.middleware import LLMToolSelectorMiddleware

agent = create_agent(
    model="gpt-4.1",
    tools=all_20_tools,  # 注册 20 个工具
    middleware=[
        LLMToolSelectorMiddleware(
            model="gpt-4.1-mini",  # 用便宜的小模型做预筛选
            max_tools=3,            # 每次最多暴露 3 个给主模型
        ),
    ],
)
# 用户问 "请假制度是什么？"
# → mini 模型从 20 个中选出 search_hr
# → 主模型只看到 search_hr，精准调用
```

**决策流程：**

```
你的 Agent 有多少工具？
├─ ≤ 5 个 → 直接全部暴露，不需要选择策略
├─ 6-15 个 → 用 LLMToolSelectorMiddleware 动态筛选
└─ > 15 个 → 工具分组 + 路由，或拆分成多个专用 Agent
```

**应用：** 在 RAG 系统中，你可能有多个知识库检索工具（HR 文档、技术文档、财务文档）。工具选择策略确保用户问"请假制度"时，模型不会去搜技术文档。

## 卡片 9: 高级特性速览

**一句话：** response_format、InjectedToolArg、extras 等高级特性解决生产环境中的实际问题。

**特性 1：response_format — 内容与工件分离**

```python
@tool(response_format="content_and_artifact")
def rag_search(query: str) -> tuple[str, list[dict]]:
    """搜索知识库"""
    docs = search_vector_store(query)
    summary = f"找到 {len(docs)} 条相关文档"
    return summary, docs  # 摘要给模型（省 token），完整数据给用户
```

模型只看到 `"找到 5 条相关文档"`，而不是 5 条完整文档内容。

**特性 2：InjectedToolArg — 隐藏参数注入**

```python
from langchain_core.tools import tool, InjectedToolArg
from typing import Annotated

@tool
def get_user_data(
    query: str,
    user_id: Annotated[str, InjectedToolArg],  # LLM 看不到这个参数
) -> str:
    """查询用户数据"""
    return fetch_data(query, user_id=user_id)

# LLM 只看到 query 参数，user_id 由框架在运行时注入
```

**特性 3：return_direct — 跳过模型润色**

```python
@tool(return_direct=True)
def get_stock_price(symbol: str) -> str:
    """查询股票实时价格"""
    return f"{symbol}: $150.23"
# 结果直接返回用户，不经过模型二次加工 → 更快、更省 token
```

**特性 4：handle_tool_error — 错误容错**

```python
@tool(handle_tool_error="搜索服务暂时不可用，请稍后重试")
def search(query: str) -> str:
    """搜索互联网"""
    return do_search(query)  # 如果抛异常，模型会收到错误提示而非崩溃
```

**应用：** 这些特性在生产环境中不可或缺。response_format 节省 token 成本，InjectedToolArg 保障安全，handle_tool_error 防止 Agent 崩溃。

## 卡片 10: 总结与延伸

**一句话：** Tool 是 Agent 的"手"，函数调用协议是 Agent 的"神经系统"。

**核心知识回顾：**

```
Tool 系统的完整链路：

定义 Tool          绑定到模型         模型决策           执行工具          结果回传
@tool          → bind_tools()   → tool_calls      → tool.invoke()  → ToolMessage
函数+描述+Schema   注入 JSON Schema   返回调用请求        框架代为执行       通过 id 关联
```

**10 个卡片的知识地图：**

| 卡片 | 核心知识 | 一句话 |
|------|----------|--------|
| 1 | Tool 是什么 | 函数 + 描述 + Schema = 让 LLM 动手做事 |
| 2 | @tool 装饰器 | 一行代码创建 Tool，90% 场景够用 |
| 3 | bind_tools() | 把工具的 Schema 注入模型上下文 |
| 4 | tool_calls | 模型不执行工具，只输出调用请求 |
| 5 | ToolMessage | 工具结果通过 tool_call_id 回传模型 |
| 6 | Schema | 参数说明书，决定 LLM 填参准确率 |
| 7 | 三种定义方式 | @tool → StructuredTool → BaseTool |
| 8 | 工具选择策略 | 工具多时用 Middleware 动态筛选 |
| 9 | 高级特性 | response_format、InjectedToolArg 等 |
| 10 | 总结 | Tool 是 Agent 与外部世界交互的全部秘密 |

**与 Agent 系统的关系：**

```
01_create_agent标准抽象  →  定义 Agent 怎么创建
02_AgentExecutor执行循环  →  定义 Think→Act→Observe 循环
03_Tools与函数调用       →  定义 "Act" 具体怎么执行 ← 你在这里
04_Agent推理与决策       →  定义 "Think" 具体怎么推理 ← 下一步
```

**下一步学习：**

1. **04_Agent推理与决策**：理解 Agent 如何选择工具、规划步骤
2. **实战练习**：用 @tool 包装一个真实 API，跑通完整的 4 步消息链路
3. **生产进阶**：学习 LLMToolSelectorMiddleware 和 InjectedToolArg 的实际应用

**记住这条链路：** `@tool 定义能力 → bind_tools() 告知模型 → tool_calls 模型决策 → ToolMessage 结果回传`。掌握它，就掌握了 Agent 与外部世界交互的全部秘密。

## 快速复习

### 核心概念

| 概念 | 定义 | 关键点 |
|------|------|--------|
| Tool | 函数 + 描述 + Schema | LLM 靠 description 选工具 |
| bind_tools() | 将工具 Schema 注入模型 | 自动处理多提供商格式差异 |
| tool_calls | 模型的结构化调用请求 | 包含 name、args、id |
| ToolMessage | 工具执行结果消息 | 通过 tool_call_id 关联请求 |
| Schema | 参数类型和描述 | Pydantic BaseModel 自动生成 |

### 三种定义方式速记

| 方式 | 一句话 | 何时用 |
|------|--------|--------|
| @tool | 最简洁 | 日常开发（90%） |
| StructuredTool | 自定义 Schema | 需要精细参数控制 |
| BaseTool | 完全控制 | 需要状态管理 |

### 高级特性速记

| 特性 | 解决什么问题 |
|------|-------------|
| response_format | 摘要给模型，完整数据给用户 |
| InjectedToolArg | 隐藏参数不暴露给 LLM |
| return_direct | 跳过模型润色，直接返回 |
| handle_tool_error | 工具出错不崩溃 |
