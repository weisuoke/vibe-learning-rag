# 反直觉点

很多 timeout 设计问题，不是因为你没见过 timeout，而是因为你带着错误直觉去理解它。

下面这 6 个误区，是 LangGraph 工作流里最常见、也最容易让系统在生产里出问题的地方。

---

## 误区 1：给图设了 `step_timeout`，节点内部就自动有 timeout 了 ❌

**为什么错？**

`step_timeout` 是运行时在等待当前 step 完成时使用的外层 deadline。

它并不会自动把 timeout 传进：
- `httpx.Client`
- 数据库连接
- 三方 SDK
- 你的 `await some_tool()`

也就是说，节点内部调用仍然可能长时间悬挂，直到外层 step budget 被耗尽。

**为什么人们容易这样错？**

因为很多框架里的“timeout”看起来像全局开关，人会本能地以为“一处设置，全局生效”。

但在实际系统里，图调度器和 I/O 客户端根本不是一层。

**正确理解：**

```python
timeout = httpx.Timeout(8.0, connect=2.0, read=5.0, write=3.0, pool=2.0)
client = httpx.AsyncClient(timeout=timeout)

graph.step_timeout = 10
```

正确做法是：**节点内有 I/O timeout，图外层再有 step timeout。**

---

## 误区 2：`step_timeout` 控制的是某个节点执行时长 ❌

**为什么错？**

从源码看，`step_timeout` 是传给 `runner.tick()` / `runner.atick()` 的，用来约束当前 step 内所有 future 的等待窗口。

它真正控制的是：

> “这一轮 step 整体最多等多久。”

如果一个 step 内有并发节点，那么快节点完成并不代表 step 完成。

**为什么人们容易这样错？**

因为大家看见“节点 A 很慢，所以 step_timeout 触发了”，就误以为 timeout 和 node 是一一对应的。

实际上它是 **step-budget**，不是 **node-budget**。

**正确理解：**
- node-level → 用 `wait_for` / `httpx.Timeout`
- step-level → 用 `graph.step_timeout`

---

## 误区 3：timeout 一旦发生，真正重要的控制流也会被吞掉 ❌

**为什么错？**

源码测试明确覆盖了 `ParentCommand` 在 timeout 打开时仍需正常传播的场景。

这说明 LangGraph 的 timeout 不是“高于一切的覆盖器”，而是“没有更高优先级语义信号时的兜底器”。

**为什么人们容易这样错？**

因为很多系统里的 timeout 处理写得非常粗暴：一旦超时就统一报错，其他上下文全部丢失。

LangGraph 在这点上更精细。

**正确理解：**
- `ParentCommand` / `GraphInterrupt` 这类语义信号需要保留；
- timeout 只负责处理真正“还未结束”的 pending work。

---

## 误区 4：前端报 network error，就说明图执行失败了 ❌

**为什么错？**

前端超时可能只是：
- 浏览器请求窗口耗尽；
- API gateway 断开；
- SSE / 流式连接被中间层关闭；
- 前端自己消费太慢。

而后端图可能还在运行，甚至最终成功并写入状态。

**为什么人们容易这样错？**

因为从用户视角看，“看不到结果”很容易被等同为“后端失败”。

但生产系统是分层的：前端看见的是连接状态，不一定是工作流状态。

**正确理解：**
- 用 run status / thread state / checkpoint 去确认工作流是否真的失败；
- 不要只盯着浏览器控制台。

## 误区 5：长任务最直接的办法，就是把 timeout 调大 ❌

**为什么错？**

如果等待本质上是：
- 人工审批；
- 外部批任务；
- 多分钟级生成；
- 需要跨会话恢复；

那“调大 timeout”只是在让一个不适合长时间占用的交互路径继续占用资源。

**为什么人们容易这样错？**

因为改参数比改架构简单。

但 LangGraph 官方文档更强调的是：
- `interrupt`
- checkpoint
- resume
- durable execution

这说明正确做法往往是 **改等待方式**，而不是 **无限延长等待时间**。

---

## 误区 6：只要有 retry，就能弥补 timeout 问题 ❌

**为什么错？**

retry 解决的是“值得再试一次”的瞬态失败；
timeout 解决的是“不能继续无限等”的预算边界。

二者不是互相替代，而是互相补位。

没有 timeout 的 retry，可能变成长时间反复拖延；
没有 retry 的 timeout，可能对瞬态故障过于脆弱。

**为什么人们容易这样错？**

因为两者最后都可能表现成“程序没有一次成功”，表面上像同类问题。

**正确理解：**

```text
deadline 负责“最多等多久”
retry    负责“值不值得再来一次”
fallback 负责“再不行怎么保底”
```

---

## 反直觉总结

### 你真正要反过来记住的是：

1. timeout 不是单层参数，而是多层预算；
2. 图 timeout 不是节点 timeout；
3. 前端 timeout 不是图失败；
4. 长任务的正确方案常常不是加大 timeout，而是改成 pause / resume / polling；
5. retry 不是 timeout 的替代品。

---

## 一句话收束

**最危险的不是 timeout 太小，而是你根本没搞清楚超时发生在哪一层。**
