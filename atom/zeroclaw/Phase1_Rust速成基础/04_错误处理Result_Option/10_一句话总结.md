# 错误处理 Result/Option - 一句话总结

> 维度 10/10 | Phase 1 第 4 个知识点 | 全文总结与学习路线图

## 一句话总结

**Result<T,E> 和 Option<T> 是 Rust 用枚举将"可能失败"和"可能为空"编码为编译时类型约束的错误处理机制，通过 ? 运算符简洁传播，配合 anyhow 的 context/bail/ensure 构建完整错误链，ZeroClaw 以此实现了零 panic、安全优先的 Trait 驱动错误处理架构。**

---

## 逐层拆解

### "用枚举将错误编码为编译时类型约束"

Rust 没有 try/catch，错误不是运行时异常，而是类型系统的一部分。函数签名 `-> Result<T, E>` 明确告诉调用者"我可能失败"，编译器强制你处理每个 Result：

```rust
// 函数签名就是契约：我可能失败，你必须处理
async fn execute(&self, args: Value) -> anyhow::Result<ToolResult>;

// 不处理 Result → 编译器警告：unused Result that must be used
// 不处理 Option → 无法直接使用内部值，编译错误
```

```typescript
// TypeScript 对照：函数签名看不出会不会抛异常
async function execute(args: object): Promise<ToolResult> {
  // 可能 throw，但签名里看不出来
}
```

[来源: reference/context7_rust_01.md, reference/search_error_handling_01.md]

### "通过 ? 运算符简洁传播"

`?` 是 match 的语法糖——Ok 时解包，Err 时提前返回。一个字符替代五行模板代码，让错误传播既显式又简洁：

```rust
// 每个 ? 都是一个显式的错误传播点
let conn = Connection::open(&path).context("SQLite failed to open database")?;
let key_bytes = self.load_or_create_key()?;
let content = std::fs::read_to_string(path).context("failed to read config")?;
```

[来源: reference/source_error_handling_01.md]

### "配合 anyhow 的 context/bail/ensure 构建完整错误链"

anyhow 三件套覆盖 90% 的错误处理场景，`.context()` 层层嵌套形成从业务到系统的完整追踪路径：

```rust
// bail! —— 立即返回错误
bail!("URL cannot be empty");

// ensure! —— 条件断言
ensure!(blob.len() > NONCE_LEN, "Encrypted value too short");

// .context() —— 附加上下文，形成错误链
// 输出：Error: failed to initialize agent
//       Caused by: failed to read config file
//       Caused by: No such file or directory (os error 2)
```

[来源: reference/context7_anyhow_01.md]

### "零 panic、安全优先的 Trait 驱动错误处理架构"

ZeroClaw 四大原则确保 AI Agent 运行时在任何错误下优雅降级：

```rust
// 所有 Trait 方法统一返回 anyhow::Result<T>
trait Tool:     async fn execute(&self, args: Value) -> anyhow::Result<ToolResult>;
trait Memory:   async fn store(&self, entry: MemoryEntry) -> anyhow::Result<()>;
trait Channel:  async fn receive(&mut self) -> anyhow::Result<ChannelMessage>;
trait Provider: async fn chat(&self, messages: Vec<Message>) -> anyhow::Result<Response>;
```

| # | 概念 | 一句话 |
|---|------|--------|
| 1 | Result 枚举 | `Ok(T)` 装成功值，`Err(E)` 装错误，编译器强制你拆开看 |
| 2 | Option 枚举 | `Some(T)` 有值，`None` 没有，替代 null/undefined 的编译时安全方案 |
| 3 | ? 运算符 | Ok 解包继续，Err 提前返回，一个字符替代五行 match |

| 6 | thiserror | 库层错误定义：`#[derive(Error)]` + `#[from]` + `#[error(transparent)]` |

## 决策速查

遇到错误处理选择时，按这个流程走：

```
函数可能失败？
├── 是 ──────────────────→ 返回 Result<T, E>
│   ├── 应用层代码？ ────→ anyhow::Result<T>（ZeroClaw 标准）
│   └── 库层代码？ ──────→ thiserror 自定义枚举
│
值可能不存在？
├── 是 ──────────────────→ Option<T>
│   ├── 有合理默认值？ ──→ .unwrap_or(default)
│   ├── 缺失是错误？ ────→ .ok_or_else(|| anyhow!("msg"))?
│   └── 需要链式处理？ ──→ .and_then(|v| ...).map(|v| ...)
│
拿到 Result 怎么处理？
├── 传播给调用者 ────────→ ?
├── 传播并加上下文 ──────→ .context("msg")?
├── 需要分别处理 ────────→ match Ok/Err
├── 转换错误类型 ────────→ .map_err(|e| ...)
│
需要主动报错？
├── 无条件报错 ──────────→ bail!("msg")
├── 条件不满足时报错 ────→ ensure!(cond, "msg")
├── 构造临时错误 ────────→ Err(anyhow!("msg"))
│
以上都不行？
└── 绝对不可能失败 ──────→ .expect("为什么不可能失败")
    （ZeroClaw 中尽量避免，仅限测试代码）
```

[来源: reference/source_error_handling_01.md, reference/context7_anyhow_01.md]

## 常见错误速查

| 你写的代码 | 问题 | 应该写 |
|-----------|------|--------|
| `args["command"].as_str().unwrap()` | 键缺失或非字符串 → panic | `args.get("command").and_then(\|v\| v.as_str()).ok_or_else(\|\| anyhow!("..."))?` |
| `fs::read_to_string(path)?` | 错误信息只有 IO 错误，缺少业务上下文 | `fs::read_to_string(path).context("failed to read config")?` |
| `let v = opt.unwrap();` | None → panic，进程终止 | `let v = opt.ok_or_else(\|\| anyhow!("value missing"))?;` |
| `if bad { return Err(anyhow!("...")); }` | 冗长 | `if bad { bail!("..."); }` |
| `if !valid { return Err(anyhow!("...")); }` | 冗长 | `ensure!(valid, "...");` |
| `result.unwrap_or(String::new())` | 吞掉了错误，不知道失败原因 | 先 log 错误再用默认值，或直接 `?` 传播 |

**规律总结：**
- `?` 运算符无处不在，是最基础的错误传播方式
- 每个源文件倾向于使用 1-2 种主要模式，而非全部混用
- `.context()` 集中在 IO/数据库操作（需要业务上下文的地方）
- `bail!`/`ensure!` 集中在输入验证（需要主动报错的地方）
- Option 组合子集中在 JSON 参数提取（从 `serde_json::Value` 取值）

## 三个反直觉点回顾

| 误区 | 真相 | 一句话 |
|------|------|--------|
| "Result 和 try/catch 本质一样" | Result 是编译时强制的类型约束，try/catch 是运行时可选的控制流 | 不处理 Result → 编译不过；不写 catch → 运行时崩溃 |
| "unwrap() 到处用没问题" | unwrap 遇到 Err/None 会 panic，整个进程终止 | ZeroClaw 原则：生产代码零 unwrap |
| "Option 和 null 差不多" | Option 是显式的编译时检查，null 是隐式的运行时炸弹 | `String` 一定有值，`Option<String>` 可能没有，两者是不同类型 |

[来源: reference/search_error_handling_01.md]

## TypeScript 开发者速查对照

| 你想做的事 | TypeScript | Rust |
|-----------|-----------|------|
| 表示可能失败 | `throw new Error()` | `-> Result<T, E>` |
| 处理错误 | `try { } catch(e) { }` | `match r { Ok(v) => ..., Err(e) => ... }` |
| 传播错误 | throw 自动冒泡 | `?` 显式传播 |
| 表示可能为空 | `T \| null \| undefined` | `Option<T>` |
| 安全取值 | `x?.y?.z` | `x.and_then(\|v\| v.y()).and_then(\|v\| v.z())` |
| 默认值 | `x ?? defaultValue` | `x.unwrap_or(default)` |
| 空值转错误 | `if (!x) throw new Error()` | `x.ok_or_else(\|\| anyhow!("msg"))?` |
| 条件断言 | `if (!cond) throw new Error()` | `ensure!(cond, "msg")` |
| 立即报错 | `throw new Error("msg")` | `bail!("msg")` |
| 附加错误上下文 | 手动拼接 message | `.context("msg")?` |
| 自定义错误类型 | `class MyError extends Error` | `#[derive(Error)] enum MyError` |
| 通用错误处理 | 无标准方案 | `anyhow::Result<T>` |
