## 概述

Tokio 提供三种时间控制工具，每一种都有精确的前端类比：

| 工具 | 用途 | JS 类比 | 一句话说明 |
|------|------|---------|------------|
| `sleep` | 延迟/暂停 | `setTimeout` + `await` | 等一会儿再继续 |
| `interval` | 周期性执行 | `setInterval` | 每隔 N 秒做一次 |
| `timeout` | 超时保护 | `Promise.race([fn(), setTimeout()])` | 限时完成，否则报错 |

本节通过 4 个完整可运行示例，掌握这三种工具在实际项目（特别是 ZeroClaw 风格的 AI Agent 服务）中的用法。

## 1. sleep 实战：指数退避重试

**场景**：调用外部 API 可能临时失败（网络波动、限流），需要自动重试并逐步增加等待时间。

### Cargo.toml

```toml
[package]
name = "timer-patterns"
version = "0.1.0"
edition = "2021"

[dependencies]
tokio = { version = "1", features = ["full"] }
anyhow = "1"
```

### 代码

```rust
use std::time::Duration;
use anyhow::{Result, bail};

/// 模拟一个不稳定的 API：前 3 次调用失败，第 4 次成功
async fn unreliable_api(attempt: u32) -> Result<String> {
    tokio::time::sleep(Duration::from_millis(100)).await; // 模拟网络延迟

    if attempt < 4 {
        bail!("服务暂时不可用（第 {} 次尝试）", attempt)
    }
    Ok("API 返回成功数据".to_string())
}

/// 带指数退避的重试函数
///
/// 对应 ZeroClaw 源码中的模式：
/// ```
/// tokio::time::sleep(Duration::from_secs(backoff)).await;
/// backoff = backoff.saturating_mul(2).min(max_backoff);
/// ```
async fn retry_with_backoff<F, Fut, T>(
    operation_name: &str,
    max_retries: u32,
    initial_backoff_ms: u64,
    max_backoff_ms: u64,
    mut operation: F,
) -> Result<T>
where
    F: FnMut(u32) -> Fut,
    Fut: std::future::Future<Output = Result<T>>,
{
    let mut backoff = initial_backoff_ms;

    for attempt in 1..=max_retries {
        println!(
            "  🔄 [{}] 第 {}/{} 次尝试...",
            operation_name, attempt, max_retries
        );

        match operation(attempt).await {
            Ok(result) => {
                println!(
                    "  ✅ [{}] 第 {} 次尝试成功！",
                    operation_name, attempt
                );
                return Ok(result);
            }
            Err(e) => {
                println!(
                    "  ❌ [{}] 失败: {}",
                    operation_name, e
                );

                if attempt < max_retries {
                    println!(
                        "  ⏳ [{}] 等待 {}ms 后重试（下次退避: {}ms）",
                        operation_name,
                        backoff,
                        (backoff * 2).min(max_backoff_ms)
                    );

                    // 核心：sleep 等待退避时间
                    tokio::time::sleep(Duration::from_millis(backoff)).await;

                    // 指数退避：翻倍，但不超过上限
                    backoff = (backoff * 2).min(max_backoff_ms);
                }
            }
        }
    }

    bail!("[{}] 重试 {} 次后仍然失败", operation_name, max_retries)
}

#[tokio::main]
async fn main() -> Result<()> {
    println!("═══ sleep 实战：指数退避重试 ═══\n");

    // 重试策略：最多 5 次，初始退避 200ms，最大退避 2000ms
    let result = retry_with_backoff(
        "调用 AI API",
        5,        // max_retries
        200,      // initial_backoff_ms
        2000,     // max_backoff_ms
        |attempt| async move { unreliable_api(attempt).await },
    ).await?;

    println!("\n📋 最终结果: {}", result);

    // 退避时间线：
    // 尝试1 → 失败 → 等 200ms
    // 尝试2 → 失败 → 等 400ms    (200 × 2)
    // 尝试3 → 失败 → 等 800ms    (400 × 2)
    // 尝试4 → 成功！
    // 总等待时间: 200 + 400 + 800 = 1400ms

    Ok(())
}
```

## 2. interval 实战：心跳与健康检查

**场景**：每 30 秒检查一次所有组件的健康状态，对应 ZeroClaw Daemon 的状态刷新机制。

```rust
use std::time::{Duration, Instant};
use tokio::time::MissedTickBehavior;

/// 模拟健康检查（可能耗时不同）
async fn health_check(component: &str) -> bool {
    // 模拟不同组件不同的检查延迟
    let delay = match component {
        "database" => 50,
        "redis" => 20,
        "llm_api" => 200,  // LLM API 检查比较慢
        _ => 10,
    };
    tokio::time::sleep(Duration::from_millis(delay)).await;

    // 模拟偶尔不健康
    let healthy = rand_simple() % 10 > 2; // 80% 概率健康
    healthy
}

fn rand_simple() -> u64 {
    use std::time::SystemTime;
    SystemTime::now()
        .duration_since(SystemTime::UNIX_EPOCH)
        .unwrap()
        .subsec_nanos() as u64
}

#[tokio::main]
async fn main() {
    println!("═══ interval 实战：心跳健康检查 ═══\n");

    let components = vec!["database", "redis", "llm_api"];

    // ─── 创建 interval（对应 ZeroClaw 的心跳机制） ───
    // ZeroClaw 源码模式：
    //   let mut interval = tokio::time::interval(Duration::from_secs(STATUS_FLUSH_SECONDS));
    //   interval.set_missed_tick_behavior(tokio::time::MissedTickBehavior::Skip);
    let mut interval = tokio::time::interval(Duration::from_secs(2));
    interval.set_missed_tick_behavior(MissedTickBehavior::Skip);

    let start = Instant::now();

    // 只检查 5 轮（演示用，实际项目是无限循环）
    for round in 1..=5 {
        // 等待下一个 tick
        interval.tick().await;

        println!(
            "🔍 第 {} 轮健康检查 [{:.1}s]",
            round,
            start.elapsed().as_secs_f64()
        );

        // 检查所有组件
        for component in &components {
            let healthy = health_check(component).await;
            let icon = if healthy { "✅" } else { "❌" };
            println!("  {} {}", icon, component);
        }
        println!();
    }
}
```

### MissedTickBehavior 三种模式详解

```
假设 interval = 1 秒，但某次 tick 的处理花了 2.5 秒：

正常情况：
  tick0    tick1    tick2    tick3    tick4
  |--------|--------|--------|--------|
  0s       1s       2s       3s       4s

tick1 处理花了 2.5 秒（从 1s 到 3.5s），该怎么办？

─── Burst（默认）───────────────────────────────────────
立即补发所有错过的 tick，然后恢复正常节奏：

  tick0   tick1         tick2 tick3   tick4    tick5
  |-------|=============|-----|-------|--------|
  0s      1s            3.5s 3.5s    4.5s     5.5s
                         ↑    ↑
                         立即补发两个错过的 tick！

  适用场景：每次 tick 都很重要（如数据同步），不能漏

─── Delay ───────────────────────────────────────────
从当前时间重新开始计时：

  tick0   tick1              tick2         tick3
  |-------|=================|-------------|-------------|
  0s      1s                3.5s          4.5s          5.5s
                             ↑
                             从 3.5s 开始重新算 1 秒

  适用场景：保持固定间隔，不管过去错过了多少（如 UI 刷新）

─── Skip（ZeroClaw 使用这个）──────────────────────────
跳过错过的 tick，在下一个对齐的时间点恢复：

  tick0   tick1              tick3         tick4
  |-------|=================|-------------|-------------|
  0s      1s                3.5s→4s       5s            6s
                             ↑
                             跳过 tick2，直接等 tick3（4s 时刻）

  适用场景：周期性状态刷新，错过的不需要补（如健康检查）
  这是最常用的！ZeroClaw 也选择了这个。
```

### interval vs JS setInterval 对比

```typescript
// ─── JS setInterval ───
// const id = setInterval(() => {
//     console.log("每 2 秒执行");
// }, 2000);
// clearInterval(id);  // 停止

// ─── Rust tokio interval ───
// let mut interval = tokio::time::interval(Duration::from_secs(2));
// loop {
//     interval.tick().await;  // ← 异步等待，不阻塞线程
//     println!("每 2 秒执行");
// }
// // 退出循环即停止（或 drop interval）

// 区别：
// 1. JS setInterval 是回调式 → Rust interval 是 await 式
// 2. JS setInterval 返回 ID 用于 clear → Rust drop 即停止
// 3. JS 没有 MissedTickBehavior 的概念
// 4. Rust interval 的第一个 tick 立即完成（不等待）
```

## 3. timeout 实战：API 调用超时保护

**场景**：调用 LLM API 可能很慢，需要设置超时保护。对应 ZeroClaw 的：

```rust
// ZeroClaw 源码模式：
// let result = tokio::time::timeout(
//     Duration::from_secs(10),
//     channel.health_check()
// ).await;
```

### 代码

```rust
use std::time::Duration;
use anyhow::Result;

/// 模拟 LLM API 调用（可能很慢）
async fn call_llm_api(prompt: &str, delay_ms: u64) -> Result<String> {
    println!("  🤖 调用 LLM API: \"{}\"（预计 {}ms）", prompt, delay_ms);
    tokio::time::sleep(Duration::from_millis(delay_ms)).await;
    Ok(format!("LLM 回复: 关于「{}」的回答", prompt))
}

/// 带超时保护的 LLM 调用
async fn call_llm_with_timeout(
    prompt: &str,
    timeout_secs: u64,
    actual_delay_ms: u64,
) {
    println!(
        "\n--- 超时设置: {}s | 实际耗时: {}ms ---",
        timeout_secs, actual_delay_ms
    );

    // tokio::time::timeout 返回 Result<T, Elapsed>
    //   Ok(inner_result) → 在超时前完成
    //   Err(Elapsed)     → 超时了
    match tokio::time::timeout(
        Duration::from_secs(timeout_secs),
        call_llm_api(prompt, actual_delay_ms),
    ).await {
        // 三层结果解析：
        Ok(Ok(response)) => {
            // 未超时 + API 成功
            println!("  ✅ 成功: {}", response);
        }
        Ok(Err(api_error)) => {
            // 未超时 + API 返回错误
            println!("  ❌ API 错误: {}", api_error);
        }
        Err(_elapsed) => {
            // 超时了！内部的 Future 被自动取消（drop）
            println!("  ⏰ 超时！LLM API 未在 {}s 内响应", timeout_secs);
            println!("  💡 建议：重试或降级到更快的模型");
        }
    }
}

#[tokio::main]
async fn main() {
    println!("═══ timeout 实战：API 调用超时保护 ═══");

    // 场景1：快速响应（200ms < 2s 超时）
    call_llm_with_timeout("你好", 2, 200).await;

    // 场景2：超时（3000ms > 2s 超时）
    call_llm_with_timeout("写一篇长文", 2, 3000).await;

    // 场景3：刚好在超时边界
    call_llm_with_timeout("简单问题", 1, 950).await;

    // ─── 封装成实用工具函数 ───
    println!("\n═══ 封装：带超时 + 重试的 LLM 调用 ═══");

    match call_llm_with_retry_and_timeout(
        "解释量子力学",
        2,     // 超时 2 秒
        3,     // 最多重试 3 次
    ).await {
        Ok(response) => println!("\n  📋 最终结果: {}", response),
        Err(e) => println!("\n  💥 最终失败: {}", e),
    }
}

/// 结合 timeout + 重试（sleep 退避）的实用函数
async fn call_llm_with_retry_and_timeout(
    prompt: &str,
    timeout_secs: u64,
    max_retries: u32,
) -> Result<String> {
    let mut backoff_ms: u64 = 500;

    for attempt in 1..=max_retries {
        println!(
            "\n  🔄 尝试 {}/{}（超时: {}s）",
            attempt, max_retries, timeout_secs
        );

        // 模拟：前两次超时，第三次成功
        let actual_delay = if attempt < 3 { 3000 } else { 500 };

        match tokio::time::timeout(
            Duration::from_secs(timeout_secs),
            call_llm_api(prompt, actual_delay),
        ).await {
            Ok(Ok(response)) => return Ok(response),
            Ok(Err(e)) => {
                println!("  ❌ API 错误: {}", e);
            }
            Err(_) => {
                println!("  ⏰ 超时！");
            }
        }

        if attempt < max_retries {
            println!("  ⏳ 等待 {}ms 后重试...", backoff_ms);
            tokio::time::sleep(Duration::from_millis(backoff_ms)).await;
            backoff_ms *= 2;
        }
    }

    anyhow::bail!("LLM 调用在 {} 次尝试后仍然失败", max_retries)
}
```

### timeout 的本质

```
tokio::time::timeout(duration, future)

它实际上等价于：

tokio::select! {
    result = future => Ok(result),        // Future 先完成
    _ = tokio::time::sleep(duration) => Err(Elapsed),  // 超时先触发
}

所以 timeout 本质上是 select! 的一个语法糖。
但 timeout 更简洁、更安全、表意更清晰——优先使用 timeout。
```

## 4. 综合场景：带超时的心跳健康检查

**场景**：ZeroClaw Daemon 风格——每 5 秒检查一次所有组件，每个组件检查有独立的超时。

```rust
use std::time::Duration;
use tokio::time::MissedTickBehavior;
use anyhow::Result;

/// 组件健康状态
#[derive(Debug)]
struct HealthStatus {
    name: String,
    healthy: bool,
    message: String,
    latency_ms: u64,
}

/// 模拟检查单个组件的健康（可能慢或失败）
async fn check_component(name: &str) -> Result<String> {
    let delay = match name {
        "database" => 100,
        "redis" => 50,
        "llm_api" => 800,     // LLM API 比较慢
        "vector_db" => 200,
        _ => 30,
    };

    tokio::time::sleep(Duration::from_millis(delay)).await;

    // 偶尔失败
    if name == "llm_api" && rand_simple() % 3 == 0 {
        anyhow::bail!("LLM API 连接超时")
    }

    Ok(format!("{} 运行正常（{}ms）", name, delay))
}

/// 带超时的健康检查
async fn check_component_with_timeout(
    name: &str,
    timeout_ms: u64,
) -> HealthStatus {
    let start = std::time::Instant::now();

    match tokio::time::timeout(
        Duration::from_millis(timeout_ms),
        check_component(name),
    ).await {
        Ok(Ok(msg)) => HealthStatus {
            name: name.to_string(),
            healthy: true,
            message: msg,
            latency_ms: start.elapsed().as_millis() as u64,
        },
        Ok(Err(e)) => HealthStatus {
            name: name.to_string(),
            healthy: false,
            message: format!("错误: {}", e),
            latency_ms: start.elapsed().as_millis() as u64,
        },
        Err(_) => HealthStatus {
            name: name.to_string(),
            healthy: false,
            message: format!("超时（>{}ms）", timeout_ms),
            latency_ms: timeout_ms,
        },
    }
}

fn rand_simple() -> u64 {
    use std::time::SystemTime;
    SystemTime::now()
        .duration_since(SystemTime::UNIX_EPOCH)
        .unwrap()
        .subsec_nanos() as u64
}

#[tokio::main]
async fn main() {
    println!("═══ 综合场景：ZeroClaw 风格健康检查 ═══\n");

    let components = vec!["database", "redis", "llm_api", "vector_db"];

    // ─── interval：每 3 秒检查一次 ───
    let mut interval = tokio::time::interval(Duration::from_secs(3));
    interval.set_missed_tick_behavior(MissedTickBehavior::Skip);

    // 只执行 3 轮（演示用）
    for round in 1..=3 {
        interval.tick().await;

        println!("╔══════════════════════════════════════════╗");
        println!("║  健康检查 第 {} 轮                        ║", round);
        println!("╠══════════════════════════════════════════╣");

        // ─── join!：并行检查所有组件 + 每个组件独立 timeout ───
        // 这里组件数量固定，可以用 join!
        let (db, redis, llm, vector) = tokio::join!(
            check_component_with_timeout("database", 500),
            check_component_with_timeout("redis", 500),
            check_component_with_timeout("llm_api", 500),     // 500ms 超时
            check_component_with_timeout("vector_db", 500),
        );

        // 打印结果
        for status in [db, redis, llm, vector] {
            let icon = if status.healthy { "✅" } else { "❌" };
            println!(
                "║ {} {:<12} | {:>4}ms | {}",
                icon, status.name, status.latency_ms, status.message
            );
        }

        println!("╚══════════════════════════════════════════╝\n");
    }

    println!("🏁 健康检查演示结束");
}
```

### 三种工具组合使用总结

```
综合场景中三种工具的角色：

interval（每 3 秒）─── 驱动循环，决定"什么时候检查"
    │
    ▼
join!（并行）──── 同时检查所有组件，决定"怎么并行"
    │
    ├── timeout（500ms）── 保护单个组件检查不超时
    │       └── check_component("database")
    ├── timeout（500ms）
    │       └── check_component("redis")
    ├── timeout（500ms）
    │       └── check_component("llm_api")    ← 可能超时
    └── timeout（500ms）
            └── check_component("vector_db")
```

## 5. TypeScript 对照

```typescript
// ─── sleep = setTimeout + Promise ───

// Rust:
//   tokio::time::sleep(Duration::from_secs(1)).await;

// TypeScript:
await new Promise(resolve => setTimeout(resolve, 1000));

// ─── interval = setInterval（但行为不同） ───

// Rust:
//   let mut interval = tokio::time::interval(Duration::from_secs(2));
//   loop {
//       interval.tick().await;
//       doWork();
//   }

// TypeScript（类似但不完全一样）:
setInterval(() => {
    doWork();
}, 2000);

// 更接近的 TS 实现：
async function rustStyleInterval(periodMs: number) {
    while (true) {
        await doWork();
        await new Promise(r => setTimeout(r, periodMs));
    }
}
// 注意：JS 版本不精确——如果 doWork 花了 500ms，
// 实际间隔是 period + 500ms。Rust interval 会补偿这个差异。

// ─── timeout = Promise.race ───

// Rust:
//   tokio::time::timeout(Duration::from_secs(5), api_call()).await

// TypeScript:
function withTimeout<T>(promise: Promise<T>, ms: number): Promise<T> {
    return Promise.race([
        promise,
        new Promise<never>((_, reject) =>
            setTimeout(() => reject(new Error("Timeout")), ms)
        ),
    ]);
}
await withTimeout(apiCall(), 5000);

// 关键区别：
// Rust timeout 超时后会 DROP future（真正取消）
// JS Promise.race 超时后原 Promise 继续运行（资源浪费！）

// ─── 指数退避重试 ───

// Rust:
//   tokio::time::sleep(Duration::from_millis(backoff)).await;
//   backoff = (backoff * 2).min(max_backoff);

// TypeScript:
let backoff = 200;
const maxBackoff = 2000;
for (let i = 0; i < maxRetries; i++) {
    try {
        return await apiCall();
    } catch (e) {
        await new Promise(r => setTimeout(r, backoff));
        backoff = Math.min(backoff * 2, maxBackoff);
    }
}
```

## 6. 速查表

```
┌───────────────┬───────────────────────────────────────────────────┐
│ 需求           │ 代码                                              │
├───────────────┼───────────────────────────────────────────────────┤
│ 等 1 秒        │ tokio::time::sleep(Duration::from_secs(1)).await │
│ 等 500ms      │ tokio::time::sleep(Duration::from_millis(500))   │
│               │     .await                                       │
│ 每 5 秒执行    │ let mut iv = interval(Duration::from_secs(5));   │
│               │ loop { iv.tick().await; do_work(); }              │
│ 跳过错过的 tick│ iv.set_missed_tick_behavior(Skip);               │
│ 5 秒内完成     │ timeout(Duration::from_secs(5), fut).await       │
│ 超时结果解析   │ Ok(Ok(v))=成功, Ok(Err(e))=业务错误,            │
│               │ Err(_)=超时                                       │
│ 指数退避       │ sleep(backoff).await;                            │
│               │ backoff = (backoff * 2).min(max);                 │
└───────────────┴───────────────────────────────────────────────────┘
```

### 一句话总结

> **sleep 管"等多久"，interval 管"每隔多久"，timeout 管"最多等多久"**——它们组合起来就是 ZeroClaw 中心跳检查（interval + timeout）和故障重试（sleep + 指数退避）的全部基础。记住 Rust 的 timeout 会真正取消内部 Future，比 JS 的 Promise.race 更安全。
