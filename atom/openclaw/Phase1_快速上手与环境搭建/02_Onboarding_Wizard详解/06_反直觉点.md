**本文档揭示 Onboarding Wizard 中容易误解的设计决策和意外行为，帮助你避免常见陷阱。**

2. ✅ 常见误区及其真相

## 反直觉点 1：QuickStart 也会生成 Token（即使是 Loopback）

### 误区

> "Loopback (127.0.0.1) 只能本机访问，不需要认证，为什么 QuickStart 还要生成 Token？"

### 真相

**即使是 Loopback 绑定，QuickStart 也会自动生成 Token。**

**原因**：

1. **安全默认值**：即使是本地访问，也应该有认证保护
2. **未来扩展性**：后续可能需要远程访问，Token 已经准备好
3. **多用户环境**：本机可能有多个用户，Token 防止未授权访问
4. **一致性**：所有模式都使用相同的认证机制，简化代码逻辑

**实际影响**：

```bash
# QuickStart 生成的配置
{
  gateway: {
    port: 18789,
    bind: "loopback",
    auth: {
      mode: "token",
      token: "oc_1a2b3c4d5e6f7g8h9i0j"  # 自动生成
    }
  }
}
```

**为什么这是好设计**：

- 防止本地恶意软件访问 Gateway
- 防止其他用户（共享机器）访问
- 符合"安全默认值"原则

## 反直觉点 2：向导不会覆盖已有配置（除非明确选择 Reset）

### 误区

> "重新运行 `openclaw onboard` 会清空我的配置和数据。"

### 真相

**向导会检测已有配置，提供三种选项：保留、更新、重置。**

**实际行为**：

```bash
$ openclaw onboard

◇  Existing config detected
│  Gateway: port 18789, bind loopback, auth token
│  Workspace: ~/.openclaw/workspace
│  Channels: WhatsApp, Telegram
│
◆  Config handling
│  ● Use existing values
│  ○ Update values
│  ○ Reset
```

**选项说明**：

| 选项 | 行为 | 数据保留 |
|------|------|---------|
| **Use existing values** | 保留所有现有配置 | ✅ 完全保留 |
| **Update values** | 更新部分配置，保留其他 | ✅ 选择性保留 |
| **Reset** | 清空配置，重新开始 | ❌ 根据 Reset Scope |

**Reset Scope 选项**：

```
◆  Reset scope
│  ● Config only                           # 只删除配置文件
│  ○ Config + creds + sessions             # 删除配置 + 凭证 + 会话
│  ○ Full reset (config + creds + sessions + workspace)  # 全部删除
```

**为什么这是好设计**：

- 防止误操作导致数据丢失
- 支持增量配置调整
- 提供明确的重置选项

## 反直觉点 3：systemd 安装失败不是 Bug（Linux 环境）

### 误区

> "向导提示 'Systemd user services are unavailable'，这是 OpenClaw 的 Bug。"

### 真相

**这是 Linux 环境配置问题，不是 OpenClaw 的 Bug。**

**根本原因**：

```bash
# 错误信息
Failed to connect to bus: $DBUS_SESSION_BUS_ADDRESS and $XDG_RUNTIME_DIR not defined
```

**原因分析**：

1. **缺少 dbus-user-session**：systemd 用户服务需要 D-Bus 会话总线
2. **未启用 lingering**：用户登出后 systemd 会话被终止
3. **SSH 环境问题**：`su` 或 `sudo -iu` 不创建完整用户会话

**解决方案**：

```bash
# 1. 安装 dbus-user-session
sudo apt install -y dbus-user-session

# 2. 启用 lingering
sudo loginctl enable-linger $USER

# 3. 重新登录（重要！）
exit
# SSH 重新连接

# 4. 验证
systemctl --user status

# 5. 重新运行向导
openclaw onboard
```

**为什么这是合理设计**：

- OpenClaw 无法修复系统级配置问题
- 提供清晰的错误信息和跳过选项
- 用户可以手动运行 Gateway（不依赖 systemd）

**详细排查**：查看 `temp/troubleshooting/systemd_issue_1818.md`

## 反直觉点 4：macOS Gateway "未就绪"可能是 CLI 不在 PATH

### 误区

> "Gateway 进程在运行（`ps` 可见），但向导一直显示 'Gateway did not become ready'。"

### 真相

**Gateway 启动失败，但错误信息被隐藏了。**

**根本原因**：

```
enableLaunchdGateway() 失败
    ↓
status = .failed(reason: "openclaw CLI not found in PATH")
    ↓
waitForGatewayReady() 继续轮询 WebSocket
    ↓
12 秒超时后显示通用错误："Gateway did not become ready"
```

**真实问题**：

- `openclaw` CLI 不在 PATH 中
- LaunchAgent 无法找到 `openclaw` 命令
- Gateway 根本没有启动（虽然 `ps` 显示进程）

**解决方案**：

```bash
# 1. 确认 CLI 在 PATH 中
which openclaw
# 应该输出：/usr/local/bin/openclaw 或类似路径

# 2. 如果未找到，添加到 PATH
export PATH="/usr/local/bin:$PATH"

# 3. 验证
openclaw --version

# 4. 重新运行向导
openclaw onboard
```

**为什么这是设计缺陷**：

- 错误信息被隐藏（已在 PR #8260 修复）
- 应该立即失败并显示真实原因
- 不应该继续轮询 WebSocket

**详细排查**：查看 `temp/troubleshooting/macos_gateway_readiness_issue_6156.md`

## 反直觉点 5：Model/Auth 步骤可能被跳过（版本 Bug）

### 误区

> "向导应该总是提示配置模型和 API 密钥。"

### 真相

**在某些版本（2026.2.12+）中，Model/Auth 步骤被意外跳过。**

**症状**：

```bash
$ openclaw onboard

◇  Onboarding mode
│  QuickStart
│
◆  How do you want to hatch your bot?  # 直接跳到这里！
│  ● Hatch in TUI
│  ○ Open Dashboard
```

**后果**：

- 默认使用 `anthropic/claude-opus-4-6`（付费模型）
- 没有配置 API 密钥
- Agent 完全无响应（无错误提示）

**临时解决方案**：

```bash
# 方式 1：手动配置 API 密钥
openclaw config set env.ANTHROPIC_API_KEY "sk-ant-..."
openclaw gateway restart

# 方式 2：降级到工作版本
npm uninstall -g openclaw
npm install -g openclaw@2026.2.9
openclaw onboard
```

**为什么这是 Bug**：

- 违反了向导的核心设计（必须配置模型）
- 导致新用户困惑（Agent 无响应）
- 已在 GitHub Issue #16134 报告

**详细排查**：查看 `temp/troubleshooting/model_auth_skip_issue_16134.md`

## 反直觉点 6：Tailscale Funnel 强制使用 Password 认证

### 误区

> "我可以使用 Token 认证 + Tailscale Funnel。"

### 真相

**Tailscale Funnel 强制要求 Password 认证。**

**原因**：

```
Tailscale Funnel = 公网暴露
    ↓
Token 可能泄露（URL、日志、浏览器历史）
    ↓
Password 更安全（不会出现在 URL 中）
```

**向导行为**：

```bash
◆  Tailscale exposure
│  Funnel (public internet)
│
◇  Note
│  Tailscale funnel requires password auth.
│
◆  Gateway auth
│  Password  # 自动切换到 Password
```

**为什么这是好设计**：

- Token 在 URL 中可见：`http://example.com?token=oc_abc123`
- Password 使用 Basic Auth（HTTP Header，不在 URL）
- 防止 Token 泄露到浏览器历史、日志、分析工具

**相关约束**：

| Tailscale 模式 | 允许的认证方式 |
|---------------|--------------|
| **Off** | Token 或 Password |
| **Serve** | Token 或 Password |
| **Funnel** | 仅 Password |

## 反直觉点 7：Tailscale 模式强制 Loopback 绑定

### 误区

> "我可以使用 Tailscale + LAN 绑定同时暴露两个网络。"

### 真相

**Tailscale Serve/Funnel 强制 Gateway 绑定到 Loopback。**

**原因**：

```
Tailscale Serve/Funnel = 反向代理
    ↓
Gateway 绑定到 Loopback (127.0.0.1)
    ↓
Tailscale 代理 Loopback → Tailnet/Internet
    ↓
防止同时暴露 LAN + Tailnet（安全风险）
```

**向导行为**：

```bash
◆  Gateway bind
│  LAN (0.0.0.0)
│
◆  Tailscale exposure
│  Serve
│
◇  Note
│  Tailscale requires bind=loopback. Adjusting bind to loopback.
│
◇  Gateway bind
│  Loopback (127.0.0.1)  # 自动调整
```

**为什么这是好设计**：

- 防止意外暴露到 LAN（安全风险）
- Tailscale 已经提供网络访问，不需要 LAN 绑定
- 符合"最小权限"原则

## 反直觉点 8：`--json` 不等于 `--non-interactive`

### 误区

> "`--json` 参数应该自动启用非交互模式。"

### 真相

**`--json` 只改变输出格式，不影响交互性。**

**实际行为**：

```bash
# 错误用法（仍然会提示交互）
openclaw onboard --json

# 正确用法（非交互 + JSON 输出）
openclaw onboard --non-interactive --json \
  --auth-choice anthropic-api-key \
  --anthropic-api-key "$ANTHROPIC_API_KEY"
```

**为什么这是合理设计**：

- `--json` 和 `--non-interactive` 是正交的关注点
- 用户可能想要交互式配置 + JSON 输出（用于日志）
- 明确分离输出格式和交互模式

---

## 反直觉点 9：非交互模式仍然需要环境变量或参数

### 误区

> "非交互模式应该自动检测所有配置。"

### 真相

**非交互模式需要明确提供所有必需参数。**

**错误示例**：

```bash
# 缺少 API 密钥
openclaw onboard --non-interactive \
  --auth-choice anthropic-api-key
# 错误：Missing required parameter: --anthropic-api-key
```

**正确示例**：

```bash
# 方式 1：命令行参数
openclaw onboard --non-interactive \
  --auth-choice anthropic-api-key \
  --anthropic-api-key "sk-ant-..."

# 方式 2：环境变量
export ANTHROPIC_API_KEY="sk-ant-..."
openclaw onboard --non-interactive \
  --auth-choice anthropic-api-key
```

**为什么这是合理设计**：

- 非交互模式用于自动化，必须明确所有配置
- 防止意外使用默认值（可能不安全）
- 符合"显式优于隐式"原则

## 反直觉点 10：向导不会自动启动 Gateway（某些情况）

### 误区

> "向导完成后 Gateway 应该自动运行。"

### 真相

**如果跳过 Daemon 安装，Gateway 不会自动启动。**

**场景 1：跳过 Daemon 安装**

```bash
◆  Install Gateway service (recommended)
│  No  # 用户选择跳过
│
└  Setup complete! 🎉

# Gateway 未启动！需要手动运行
$ openclaw status
Gateway: Not running
```

**场景 2：Daemon 安装失败**

```bash
◇  Gateway service
│  Gateway service install failed.
│
└  Setup complete! 🎉

# Gateway 未启动！需要手动运行
$ openclaw gateway run
```

**手动启动方式**：

```bash
# 方式 1：前台运行（调试）
openclaw gateway run --foreground

# 方式 2：后台运行（生产）
openclaw gateway start

# 方式 3：安装 Daemon（推荐）
openclaw gateway install
```

**为什么这是合理设计**：

- 用户明确选择跳过 Daemon，应该尊重选择
- Daemon 安装失败时，不应该强制启动（可能有环境问题）
- 提供手动启动选项，给用户完全控制

## 反直觉点 11：配置热重载不适用于 Gateway 服务器设置

### 误区

> "修改 `openclaw.json` 后，所有配置都会自动生效。"

### 真相

**Gateway 服务器设置（端口、绑定、认证）需要重启。**

**热重载支持**：

| 配置类别 | 热重载 | 需要重启 |
|---------|-------|---------|
| Channels | ✅ | ❌ |
| Agent & Models | ✅ | ❌ |
| Automation (hooks, cron) | ✅ | ❌ |
| Sessions & Messages | ✅ | ❌ |
| Tools & Media | ✅ | ❌ |
| **Gateway Server** (port, bind, auth) | ❌ | ✅ |
| **Infrastructure** (discovery, plugins) | ❌ | ✅ |

**示例**：

```bash
# 修改 Channel 配置（热重载）
openclaw config set channels.telegram.botToken "new-token"
# 立即生效，无需重启

# 修改 Gateway 端口（需要重启）
openclaw config set gateway.port 18790
# 需要重启：openclaw gateway restart
```

**为什么这是合理设计**：

- 端口、绑定、认证是底层网络配置，无法热重载
- 其他配置（Channels、Models）可以热重载，提升体验
- 符合大多数服务器软件的行为（Nginx、Apache 也需要重启）

## 反直觉点 12："Configure later" 在 macOS App 中不可逆（旧版本）

### 误区

> "选择 'Configure later' 后，可以稍后重新运行设置。"

### 真相

**在旧版本中，选择 'Configure later' 后无法重新进入设置。**

**问题**：

- 没有 Settings 菜单
- 没有菜单栏项
- 重启 App 不会重新显示设置向导

**解决方案**：

```bash
# 方式 1：使用 CLI 向导
openclaw onboard

# 方式 2：升级到新版本（已修复）
# PR #8260 添加了 "Run Setup..." 菜单项
```

**为什么这是设计缺陷**：

- 违反了"可恢复性"原则
- 用户无法撤销错误选择
- 已在 PR #8260 修复

## 常见误区总结

| 误区 | 真相 |
|------|------|
| Loopback 不需要认证 | 即使 Loopback 也生成 Token（安全默认值） |
| 重新运行向导会清空配置 | 提供保留/更新/重置三种选项 |
| systemd 安装失败是 Bug | 是 Linux 环境配置问题 |
| Gateway "未就绪"是网络问题 | 可能是 CLI 不在 PATH |
| Model/Auth 总是会提示 | 某些版本有 Bug，会跳过 |
| Token + Tailscale Funnel | Funnel 强制 Password |
| Tailscale + LAN 绑定 | Tailscale 强制 Loopback |
| `--json` = 非交互 | `--json` 只改变输出格式 |
| 非交互模式自动检测配置 | 需要明确提供所有参数 |
| 向导完成后 Gateway 自动运行 | 跳过 Daemon 则不会启动 |
| 所有配置都热重载 | Gateway 服务器设置需要重启 |
| "Configure later" 可逆 | 旧版本不可逆（已修复） |

## 避免陷阱的建议

### 1. 首次使用

```bash
# 推荐：使用 QuickStart 快速体验
openclaw onboard --flow quickstart

# 不要跳过 Daemon 安装（除非有特殊原因）
```

### 2. Linux VPS 部署

```bash
# 先检查 systemd 可用性
systemctl --user status

# 如果失败，先修复环境
sudo apt install -y dbus-user-session
sudo loginctl enable-linger $USER
exit  # 重新登录

# 然后运行向导
openclaw onboard
```

### 3. macOS 部署

```bash
# 先确认 CLI 在 PATH
which openclaw

# 如果未找到，添加到 PATH
export PATH="/usr/local/bin:$PATH"

# 然后运行向导
openclaw onboard
```

### 4. 自动化脚本

```bash
# 明确提供所有参数
openclaw onboard --non-interactive \
  --auth-choice anthropic-api-key \
  --anthropic-api-key "$ANTHROPIC_API_KEY" \
  --gateway-bind loopback \
  --gateway-port 18789 \
  --install-daemon
```

### 5. 配置调整

```bash
# 热重载配置（Channels、Models）
openclaw config set channels.telegram.botToken "new-token"

# 需要重启的配置（Gateway 服务器）
openclaw config set gateway.port 18790
openclaw gateway restart
```

## 总结

Onboarding Wizard 的许多"反直觉"设计实际上是深思熟虑的安全和可用性决策：

1. **安全默认值**：即使 Loopback 也生成 Token
2. **防止数据丢失**：重新运行不会覆盖配置
3. **环境依赖**：systemd 问题不是 OpenClaw 的 Bug
4. **明确约束**：Tailscale Funnel 强制 Password
5. **显式配置**：非交互模式需要明确参数

理解这些设计决策，可以帮助你更好地使用 Onboarding Wizard，避免常见陷阱。
