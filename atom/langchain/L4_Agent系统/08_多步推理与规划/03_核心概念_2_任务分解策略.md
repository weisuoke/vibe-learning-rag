# 核心概念 2：任务分解策略

> LangChain Agent 系统 · 第 8 个知识点 · 核心概念 2/5

---

## 一句话定义

**任务分解是将一个复杂任务拆分为多个可独立执行的子任务的过程，是 Plan-and-Execute Agent 的核心能力。**

---

## 为什么任务分解如此关键？

```
没有任务分解：
  "写一份RAG技术对比报告" → LLM 尝试一次性完成 → 质量差、容易幻觉

有任务分解：
  "写一份RAG技术对比报告" → [搜索资料, 分析架构, 对比优劣, 编写报告]
                           → 每个子任务独立、聚焦 → 质量高
```

---

## 三种主要分解策略

### 策略1：LLM 驱动分解（最常用）

**原理**：直接让 LLM 将任务分解为步骤列表。

```python
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field
from typing import List

class TaskDecomposition(BaseModel):
    """LLM 生成的任务分解结果"""
    objective: str = Field(description="原始目标")
    steps: List[str] = Field(description="分解后的步骤列表")
    reasoning: str = Field(description="分解的理由")

llm = ChatOpenAI(model="gpt-4o", temperature=0)
decomposer = llm.with_structured_output(TaskDecomposition)

# 分解任务
result = decomposer.invoke("""
请将以下任务分解为可执行的步骤。
每个步骤应该是具体的、可独立执行的。
步骤之间按依赖顺序排列。

任务：帮我调研 LangChain 和 LlamaIndex 的最新版本差异，
     并给出在 RAG 项目中选择哪个框架的建议。
""")

print(f"目标: {result.objective}")
for i, step in enumerate(result.steps, 1):
    print(f"  步骤{i}: {step}")
print(f"理由: {result.reasoning}")
```

**输出示例**：
```
目标: 调研 LangChain 和 LlamaIndex 的最新版本差异
  步骤1: 搜索 LangChain 最新版本的核心特性和变更
  步骤2: 搜索 LlamaIndex 最新版本的核心特性和变更
  步骤3: 对比两个框架在 RAG 场景下的架构差异
  步骤4: 分析两个框架的社区活跃度和生态成熟度
  步骤5: 根据项目需求给出选择建议
理由: 先分别了解每个框架，再从多维度对比，最后给出结论
```

**优点**：简单直接，利用 LLM 的理解能力
**缺点**：分解质量取决于 LLM 能力和 Prompt 设计

### 策略2：递归分解（适合深层任务）

**原理**：将每个子任务进一步分解，直到足够细粒度。

```python
from typing import Optional

class TaskNode(BaseModel):
    """任务树节点"""
    task: str = Field(description="任务描述")
    subtasks: Optional[List["TaskNode"]] = Field(
        default=None,
        description="子任务列表，如果任务足够简单则为 None"
    )
    is_leaf: bool = Field(description="是否是叶子节点（可直接执行）")

class RecursiveDecomposition(BaseModel):
    """递归分解结果"""
    root: TaskNode = Field(description="任务树根节点")

decomposer = llm.with_structured_output(RecursiveDecomposition)

def recursive_decompose(task: str, depth: int = 0, max_depth: int = 3):
    """递归分解任务"""
    if depth >= max_depth:
        return TaskNode(task=task, is_leaf=True)

    result = decomposer.invoke(f"""
    将以下任务分解为 2-4 个子任务。
    如果任务足够简单可以直接执行，标记为叶子节点。
    当前深度：{depth}/{max_depth}

    任务：{task}
    """)
    return result.root

# 使用递归分解
task_tree = recursive_decompose("构建一个完整的 RAG 问答系统")
```

**可视化**：
```
构建一个完整的 RAG 问答系统
├── 数据准备
│   ├── 收集文档 ← 叶子节点
│   ├── 文档解析和清洗 ← 叶子节点
│   └── 文本分块 ← 叶子节点
├── 向量化和存储
│   ├── 选择 Embedding 模型 ← 叶子节点
│   └── 构建向量数据库 ← 叶子节点
├── 检索和生成
│   ├── 实现检索器 ← 叶子节点
│   └── 设计 Prompt 模板 ← 叶子节点
└── 测试和优化
    ├── 编写测试用例 ← 叶子节点
    └── 评估和调优 ← 叶子节点
```

**优点**：层次清晰，复杂任务也能处理
**缺点**：LLM 调用次数多，耗时较长

### 策略3：迭代分解（适合探索性任务）

**原理**：不一次性分解所有步骤，每执行完一步后再决定下一步。

```python
class NextStep(BaseModel):
    """下一步决策"""
    next_task: str = Field(description="下一步要执行的任务")
    is_final: bool = Field(description="是否是最后一步")
    reasoning: str = Field(description="为什么选择这个下一步")

step_planner = llm.with_structured_output(NextStep)

def iterative_decompose(objective: str, max_steps: int = 10):
    """迭代式任务分解"""
    completed_steps = []

    for i in range(max_steps):
        next_step = step_planner.invoke(f"""
        目标：{objective}
        已完成步骤：{completed_steps}
        请决定下一步应该做什么。
        如果目标已经完成，标记 is_final=True。
        """)

        if next_step.is_final:
            print(f"✅ 任务完成！共 {len(completed_steps)} 步")
            break

        print(f"步骤 {i+1}: {next_step.next_task}")
        print(f"  理由: {next_step.reasoning}")

        # 执行当前步骤（这里简化为直接记录）
        result = execute(next_step.next_task)
        completed_steps.append((next_step.next_task, result))

    return completed_steps
```

**优点**：灵活，可根据中间结果调整方向
**缺点**：无全局视角，LLM 调用次数不确定

---

## 三种策略对比

| 维度 | LLM驱动 | 递归分解 | 迭代分解 |
|------|---------|---------|---------|
| **生成方式** | 一次性生成所有步骤 | 逐层递归展开 | 每步动态决定 |
| **全局视角** | ✅ 有 | ✅ 有 | ❌ 无 |
| **灵活性** | 中等 | 低 | 高 |
| **LLM 调用** | 1 次 | 多次（按深度） | N 次（按步骤） |
| **适用场景** | 大多数任务 | 层次化复杂任务 | 探索性任务 |
| **典型应用** | Plan-and-Execute | 项目管理 | 研究探索 |

## Structured Output：分解的关键技术

任务分解的质量高度依赖**结构化输出**。LangChain 提供了 `with_structured_output` 方法。

### 设计好的分解 Schema

```python
from pydantic import BaseModel, Field
from typing import List, Literal
from enum import Enum

class TaskPriority(str, Enum):
    HIGH = "high"
    MEDIUM = "medium"
    LOW = "low"

class SubTask(BaseModel):
    """子任务定义"""
    id: int = Field(description="任务编号")
    title: str = Field(description="任务标题，简洁明确")
    description: str = Field(description="任务详细描述")
    tool_needed: str = Field(description="需要使用的工具名称")
    dependencies: List[int] = Field(
        default_factory=list,
        description="依赖的前置任务编号列表"
    )
    priority: TaskPriority = Field(
        default=TaskPriority.MEDIUM,
        description="任务优先级"
    )
    estimated_complexity: Literal["simple", "medium", "complex"] = Field(
        description="预估复杂度"
    )

class TaskPlan(BaseModel):
    """完整的任务计划"""
    objective: str = Field(description="原始目标")
    tasks: List[SubTask] = Field(description="子任务列表")
    total_estimated_steps: int = Field(description="预估总步骤数")

# 使用结构化输出生成计划
planner = llm.with_structured_output(TaskPlan)
plan = planner.invoke("""
请将以下任务分解为可执行的子任务。
每个子任务需要指定：编号、标题、描述、所需工具、依赖关系、优先级和复杂度。

任务：搭建一个客服知识库问答系统
可用工具：search（网络搜索）、read_doc（读取文档）、write_code（编写代码）、
         run_test（运行测试）、deploy（部署）
""")
```

## 分解质量的提升技巧

### 1. 提供分解示例（Few-shot）

```python
decompose_prompt = """
请将任务分解为可执行的步骤。

示例：
任务：写一篇关于量子计算的科普文章
步骤：
1. 搜索量子计算的基本原理
2. 搜索量子计算的最新应用案例
3. 编写引言（什么是量子计算）
4. 编写主体（原理 + 应用）
5. 编写结论（未来展望）
6. 审校和修改

现在请分解以下任务：
任务：{task}
"""
```

### 2. 限制步骤数量

```python
decompose_prompt = """
请将任务分解为 3-7 个步骤。
太少的步骤会导致每步太复杂，太多的步骤会导致执行效率低。
每个步骤应该可以在 1-2 个工具调用内完成。
"""
```

### 3. 指定可用工具

```python
decompose_prompt = """
你可以使用以下工具：
- search: 搜索网络信息
- calculator: 数学计算
- write_file: 写入文件
- read_file: 读取文件

请只使用这些工具来分解任务，不要假设有其他工具。
"""
```

## 常见陷阱

| 陷阱 | 说明 | 解决方案 |
|------|------|---------|
| **分解太细** | 20个步骤，执行效率极低 | 限制步骤数（3-7个） |
| **分解太粗** | 每步仍然很复杂 | 使用递归分解 |
| **步骤不可执行** | "理解用户需求"这种步骤无法执行 | 强调每步必须对应具体工具调用 |
| **忽略依赖** | 步骤间有依赖但没标注 | 显式标注依赖关系 |
| **缺少验证** | 没有检查步骤的步骤 | 在末尾加入验证/审查步骤 |

---

## 关键要点

1. **三种策略**：LLM 驱动（一次性）、递归（逐层）、迭代（逐步）
2. **Structured Output 是关键**：用 Pydantic 模型约束分解格式
3. **步骤数量 3-7 个最佳**：太多太少都有问题
4. **每步要对应工具调用**：确保步骤可执行
5. **标注依赖关系**：让执行器知道顺序
