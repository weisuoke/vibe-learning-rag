## 为什么需要多步推理与规划？

### ReAct 的局限性

在 04_Agent推理与决策 中，我们学习了 ReAct 模式（Thought→Action→Observation 循环）。ReAct 是 Agent 推理的基础，但面对复杂任务时有明显短板：

| 维度 | ReAct（逐步推理） | Plan-and-Execute（先规划后执行） |
|------|-------------------|-------------------------------|
| 思考方式 | 走一步看一步，每步都重新决策 | 先制定完整计划，再分步执行 |
| LLM 调用 | 每次工具调用都需要大模型参与 | 仅规划和总结时用大模型，执行可用小模型 |
| 全局视野 | 只关注当前步，容易偏离目标 | 看到全局，子任务聚焦 |
| 成本 | 高（每步都调大模型） | 低（规划一次，执行用轻量模型） |
| 适合场景 | 简单问答、单步检索 | 研究报告、多步数据分析、复杂工作流 |

**一句话总结**：ReAct 像"边走边问路"，Plan-and-Execute 像"先看地图再出发"。

### 典型场景对比

```
场景：用户要求 "帮我调研 Python Web 框架，对比 FastAPI、Django、Flask，写一份选型报告"

=== ReAct 方式（逐步推理）===
Thought: 我需要先搜索 FastAPI 的信息
Action: search("FastAPI 特点")
Observation: FastAPI 是高性能的...
Thought: 好的，现在搜索 Django
Action: search("Django 特点")
Observation: Django 是全功能的...
Thought: 还需要搜 Flask...
（每一步都调用大模型重新推理，成本高、速度慢）

=== Plan-and-Execute 方式（先规划后执行）===
Plan:
  1. 分别搜索 FastAPI、Django、Flask 的核心特点
  2. 收集每个框架的性能基准数据
  3. 整理各框架的适用场景和生态
  4. 综合对比并生成选型建议
Execute:
  Step 1: search("FastAPI 特点") → 记录结果
  Step 2: search("Django 特点") → 记录结果
  ...
  Step 4: synthesize(所有结果) → 生成报告
（规划一次，执行可并行或用小模型，高效精准）
```

```

用户输入 ──→ ┌─────────────┐
             │   Planner   │  ← 规划器：LLM 生成任务计划
             │  (大模型)    │
             └──────┬──────┘
                    │
                    ▼
             ┌─────────────┐
             │  Task List  │  ← 任务列表：结构化的子目标
             │ [T1,T2,T3]  │
             └──────┬──────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │Executor │ │Executor │ │Executor │  ← 执行器：可用小模型/工具
   │  Task1  │ │  Task2  │ │  Task3  │
   └────┬────┘ └────┬────┘ └────┬────┘
        │           │           │
        ▼           ▼           ▼
   ┌─────────────────────────────────┐
   │         Results Pool            │  ← 结果汇总
   └──────────────┬──────────────────┘
                  │
                  ▼
             ┌─────────────┐
             │  Replanner  │  ← 重规划器：评估并决定下一步

     需要调整计划      任务全部完成
     → 回到 Planner   → 输出最终答案
```

## 核心概念速览

本知识点包含 **5 个核心概念**，覆盖多步推理与规划的完整知识面：

### 概念 1: Plan-and-Execute 模式

Agent 先通过 Planner 生成完整的任务计划，再由 Executor 逐步执行，执行后由 Replanner 决定是否调整计划。三组件协作，将复杂任务拆解为可管理的子步骤。这是与 ReAct 最大的区别。

### 概念 2: 任务分解策略

如何将一个大任务拆解成多个小任务？主要策略包括：递归分解（大问题拆成小问题，小问题再拆）、迭代分解（一步步细化）、LLM 驱动分解（用 Structured Output 让 LLM 直接输出结构化计划）。

### 概念 3: 子目标规划与动态调整

子目标的设定（SMART 原则）、优先级排序、以及最关键的——Re-planning（动态调整）。当执行过程中发现计划有误或出现新信息时，Agent 需要能够重新规划。

### 概念 4: 执行监控与自我反思

Reflection（反思）：Agent 对自己的输出进行评估和改进。Reflexion：在反思基础上加入外部验证和经验记忆。LLM-as-Judge：使用 LLM 评估执行质量。这些机制让 Agent 能够自我纠错。

### 概念 5: LangGraph 规划工作流

用 LangGraph 的 StateGraph 实现规划工作流，包括 `@task` 和 `@entrypoint` Functional API、Orchestrator-Worker 模式、TodoListMiddleware 任务规划中间件、以及中断恢复（Human-in-the-Loop）。

## 2025-2026 技术趋势

### 三代规划架构演进

| 阶段 | 架构 | 代表 | 特点 |
|------|------|------|------|
| 第一代 | ReAct 逐步推理 | AgentExecutor | 简单但低效 |
| 第二代 | Plan-and-Execute | LangGraph Plan-Execute | 规划与执行分离 |
| 第三代 | Deep Agents | TodoListMiddleware + SubAgent | 原生规划 + 子代理协作 |

### 2026 前沿方向

1. **TodoListMiddleware**：LangChain 原生的任务规划中间件，自动提供 `write_todos` 工具
2. **Deep Agents SDK**：内置规划、文件系统、子代理三大能力
3. **Task-Decoupled Planning (TDP)**：任务解耦规划，性能提升 82%
4. **Human-in-the-Loop 规划**：人机协作的规划审批流程

## 与 RAG 和 AI Agent 开发的关系

### 多步推理是复杂 RAG Agent 的必备能力

```
简单 RAG:     用户问题 → 检索 → 生成 → 回答（单步）

复杂 RAG:     用户问题 → 规划检索策略
              → 分步检索多个知识库
              → 交叉验证检索结果
              → 如果信息不足，调整检索策略
              → 综合所有信息生成回答（多步规划）
```

### 实际应用场景

| 场景 | 多步推理的价值 | 规划方式 |
|------|--------------|---------|
| 研究报告生成 | 分步搜索、整理、写作 | Plan→Execute→Replan |
| 多源数据分析 | 分别查询不同数据源，交叉分析 | 任务并行执行 |
| 复杂问答 | 拆解复合问题为子问题 | 递归任务分解 |
| 代码重构 | 分析→规划→修改→测试→验证 | 带反思的迭代执行 |
| 智能客服升级 | 多轮理解→方案规划→分步解决 | 动态调整计划 |
