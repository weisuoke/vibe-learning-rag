# 实战代码 - 场景4：生产级调试方案

## 场景说明

本场景演示如何构建一个生产级的 LangGraph 调试方案——把前面学过的 `StreamWriter`、结构化日志、错误追踪和 LangSmith 配置组合在一起，形成一套从开发到生产都能用的调试体系。

核心要点：
1. **结构化日志**：用 JSON 格式输出日志，方便日志聚合系统（ELK、CloudWatch）解析
2. **traced_node 装饰器**：一次定义，所有节点自动获得追踪能力，业务代码零侵入
3. **StreamWriter 自定义事件**：通过 `stream_mode="custom"` 向外部推送节点生命周期事件
4. **错误追踪与恢复**：节点异常时自动记录上下文，支持错误状态写入和后续分析
5. **LangSmith 集成**：两个环境变量开启全链路追踪，与本地调试互补

> 本场景不依赖外部 API，所有代码可直接运行。LangSmith 部分为可选配置。

## 核心知识点

| 知识点 | 说明 |
|--------|------|
| `StructuredFormatter` | 自定义 logging Formatter，输出 JSON 格式日志 |
| `traced_node` 装饰器 | 包装节点函数，自动注入 StreamWriter 事件和日志 |
| `get_stream_writer()` | LangGraph API，在节点内部向外推送自定义数据 |
| `stream_mode="custom"` | 接收 StreamWriter 推送的自定义事件 |
| `stream_mode=["updates", "custom"]` | 多模式组合，同时接收状态更新和自定义事件 |
| `tags` / `metadata` | LangSmith 追踪配置，用于分类筛选和业务关联 |

## 完整代码

```python
"""
LangGraph 状态调试 - 生产级调试方案
演示：结构化日志 + traced_node 装饰器 + StreamWriter + 错误追踪 + LangSmith 配置

运行环境：Python 3.13+, langgraph
运行方式：python examples/langgraph/production_debug.py
"""

import json
import logging
import functools
import time
from datetime import datetime, timezone
from typing import TypedDict, Annotated
from operator import add

from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.config import get_stream_writer


# ============================================================
# 第1部分：结构化日志配置
# ============================================================
# 生产环境的日志必须是结构化的（JSON 格式），原因：
# - 日志聚合系统（ELK、CloudWatch、Datadog）能自动解析
# - 可以按字段筛选（按节点名、按错误级别）
# - 比纯文本 print 更容易做告警规则

class StructuredFormatter(logging.Formatter):
    """JSON 格式的日志 Formatter"""

    def format(self, record: logging.LogRecord) -> str:
        log_data = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "node": getattr(record, "node", "system"),
            "message": record.getMessage(),
        }
        # 如果有异常信息，也加进去
        if record.exc_info and record.exc_info[0] is not None:
            log_data["exception"] = self.formatException(record.exc_info)
        return json.dumps(log_data, ensure_ascii=False)


# 创建 logger
logger = logging.getLogger("langgraph_prod")
logger.setLevel(logging.DEBUG)
# 清除已有 handler，避免重复输出
logger.handlers.clear()

handler = logging.StreamHandler()
handler.setFormatter(StructuredFormatter())
logger.addHandler(handler)


# ============================================================
# 第2部分：traced_node 装饰器
# ============================================================
# 核心思路：用装饰器把调试逻辑和业务逻辑分离
# - 业务代码里不需要写任何调试代码
# - 装饰器自动记录：节点开始/结束、耗时、输入输出摘要、异常

def traced_node(node_name: str):
    """
    生产级节点追踪装饰器

    功能：
    1. 通过 StreamWriter 推送节点生命周期事件（start/end/error）
    2. 通过结构化日志记录执行信息
    3. 自动计算节点耗时
    4. 异常时记录错误上下文并重新抛出

    用法：
        @traced_node("my_node")
        def my_node(state: State) -> dict:
            return {"result": "hello"}
    """
    def decorator(func):
        @functools.wraps(func)
        def wrapper(state):
            writer = get_stream_writer()
            start_time = time.time()
            timestamp = datetime.now(timezone.utc).isoformat()

            # --- 发送开始事件 ---
            writer({
                "event": "node_start",
                "node": node_name,
                "timestamp": timestamp,
                "input_keys": list(state.keys()),
            })
            logger.info(
                f"节点开始执行, 输入键: {list(state.keys())}",
                extra={"node": node_name}
            )

            try:
                # --- 执行业务逻辑 ---
                result = func(state)
                elapsed = time.time() - start_time

                # --- 发送成功事件 ---
                writer({
                    "event": "node_end",
                    "node": node_name,
                    "success": True,
                    "elapsed_ms": round(elapsed * 1000, 2),
                    "output_keys": list(result.keys()) if result else [],
                })
                logger.info(
                    f"节点执行完成, 耗时: {elapsed:.3f}s, 输出键: {list(result.keys()) if result else []}",
                    extra={"node": node_name}
                )
                return result

            except Exception as e:
                elapsed = time.time() - start_time

                # --- 发送错误事件 ---
                writer({
                    "event": "node_error",
                    "node": node_name,
                    "error_type": type(e).__name__,
                    "error_message": str(e),
                    "elapsed_ms": round(elapsed * 1000, 2),
                })
                logger.error(
                    f"节点执行失败: {type(e).__name__}: {e}",
                    extra={"node": node_name},
                    exc_info=True
                )
                # 重新抛出，让 LangGraph 的错误处理机制接管
                raise

        return wrapper
    return decorator


# ============================================================
# 第3部分：定义状态和业务节点
# ============================================================

class State(TypedDict):
    query: str
    steps: Annotated[list[str], add]
    result: str
    error: str


@traced_node("analyze")
def analyze(state: State) -> dict:
    """分析节点：解析用户查询"""
    query = state["query"]
    analysis = f"分析完成: '{query}' 是一个关于技术实践的问题"
    return {"steps": [f"[analyze] {analysis}"], "result": analysis}


@traced_node("retrieve")
def retrieve(state: State) -> dict:
    """检索节点：模拟文档检索"""
    # 模拟检索耗时
    time.sleep(0.05)
    docs = ["文档1: RAG架构设计", "文档2: 向量检索优化", "文档3: Prompt工程"]
    return {
        "steps": [f"[retrieve] 检索到 {len(docs)} 篇文档"],
        "result": f"检索结果: {', '.join(docs)}"
    }


@traced_node("generate")
def generate(state: State) -> dict:
    """生成节点：基于检索结果生成回答"""
    answer = f"基于检索结果的回答: {state['result'][:50]}..."
    return {
        "steps": [f"[generate] 生成回答完成"],
        "result": answer
    }


# ============================================================
# 第4部分：构建图
# ============================================================

builder = StateGraph(State)
builder.add_node("analyze", analyze)
builder.add_node("retrieve", retrieve)
builder.add_node("generate", generate)

builder.add_edge(START, "analyze")
builder.add_edge("analyze", "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

graph = builder.compile(checkpointer=MemorySaver())


# ============================================================
# 第5部分：执行 - 使用 custom 模式捕获自定义事件
# ============================================================

print("=" * 60)
print("=== 演示1：stream_mode='custom' 捕获追踪事件 ===")
print("=" * 60)

config = {
    "configurable": {"thread_id": "prod-debug-1"},
    # LangSmith 追踪配置（可选，设置环境变量后自动生效）
    "tags": ["production", "v1.0"],
    "metadata": {
        "user_id": "user_123",
        "environment": "staging",
        "feature": "rag_pipeline"
    }
}

print("\n--- 自定义追踪事件 ---")
for event in graph.stream(
    {"query": "RAG最佳实践", "steps": [], "result": "", "error": ""},
    config,
    stream_mode="custom"
):
    # event 就是 StreamWriter 推送的 dict
    print(json.dumps(event, ensure_ascii=False, indent=2))


# ============================================================
# 第6部分：多模式组合 - 同时看状态更新和自定义事件
# ============================================================

print()
print("=" * 60)
print("=== 演示2：多模式组合 (updates + custom) ===")
print("=" * 60)

config2 = {
    "configurable": {"thread_id": "prod-debug-2"},
}

print("\n--- 状态更新 + 追踪事件 ---")
for mode, chunk in graph.stream(
    {"query": "向量数据库选型", "steps": [], "result": "", "error": ""},
    config2,
    stream_mode=["updates", "custom"]
):
    if mode == "custom":
        # 自定义追踪事件：只打印关键信息
        node = chunk.get("node", "?")
        event = chunk.get("event", "?")
        elapsed = chunk.get("elapsed_ms", "")
        elapsed_str = f" ({elapsed}ms)" if elapsed else ""
        print(f"  [追踪] {node}: {event}{elapsed_str}")
    elif mode == "updates":
        # 状态更新：打印节点名和更新的键
        for node_name, update in chunk.items():
            print(f"  [更新] {node_name} → 更新键: {list(update.keys())}")


# ============================================================
# 第7部分：错误场景演示
# ============================================================

print()
print("=" * 60)
print("=== 演示3：错误追踪 ===")
print("=" * 60)


# 定义一个会出错的节点
@traced_node("risky_process")
def risky_process(state: State) -> dict:
    """模拟一个可能出错的节点"""
    if "危险" in state["query"]:
        raise ValueError("检测到不安全的查询内容")
    return {"result": "处理完成"}


# 构建包含错误节点的图
error_builder = StateGraph(State)
error_builder.add_node("analyze", analyze)
error_builder.add_node("risky_process", risky_process)
error_builder.add_edge(START, "analyze")
error_builder.add_edge("analyze", "risky_process")
error_builder.add_edge("risky_process", END)

error_graph = error_builder.compile(checkpointer=MemorySaver())

config3 = {"configurable": {"thread_id": "error-debug-1"}}

print("\n--- 触发错误，观察追踪输出 ---")
try:
    for mode, chunk in error_graph.stream(
        {"query": "危险操作测试", "steps": [], "result": "", "error": ""},
        config3,
        stream_mode=["updates", "custom"]
    ):
        if mode == "custom":
            event_type = chunk.get("event", "")
            if "error" in event_type:
                print(f"  [错误事件] {json.dumps(chunk, ensure_ascii=False)}")
            else:
                print(f"  [追踪] {chunk.get('node')}: {chunk.get('event')}")
        elif mode == "updates":
            for node_name, update in chunk.items():
                print(f"  [更新] {node_name} → {list(update.keys())}")
except Exception as e:
    print(f"\n  捕获异常: {type(e).__name__}: {e}")

# 错误后检查状态：看执行到了哪一步
print("\n--- 错误后状态检查 ---")
error_state = error_graph.get_state(config3)
print(f"  已完成的步骤: {error_state.values.get('steps', [])}")
print(f"  下一个节点: {error_state.next}")
print(f"  说明: 错误发生在 risky_process，analyze 的结果已保存")


# ============================================================
# 第8部分：LangSmith 配置说明
# ============================================================

print()
print("=" * 60)
print("=== 附录：LangSmith 生产配置 ===")
print("=" * 60)

print("""
LangSmith 是非侵入式的追踪方案，只需设置环境变量：

  export LANGSMITH_TRACING=true
  export LANGSMITH_API_KEY=<your-key>
  export LANGSMITH_PROJECT=my-rag-production

设置后，上面所有的 graph.stream() 和 graph.invoke() 调用
都会自动上报到 LangSmith Web UI，包括：
  - 每个节点的输入/输出
  - LLM 调用详情（token 数、延迟）
  - 完整的执行路径和耗时
  - tags 和 metadata（用于筛选和关联）

本地调试工具（StreamWriter + 结构化日志）和 LangSmith 互补：
  - StreamWriter：实时流式输出，适合开发和测试
  - 结构化日志：持久化到文件/日志系统，适合运维
  - LangSmith：可视化 + 团队协作 + 历史对比，适合生产
""")
```

## 代码解析

### 1. 生产调试架构总览

```
应用层:  @traced_node 装饰器 → 业务代码零侵入
         ├── StreamWriter → stream_mode="custom" → 实时监控面板
         └── logging(JSON) → 日志聚合系统(ELK) → 告警 + 分析
追踪层:  LangSmith（可选）→ 环境变量启用 → Web UI → 团队协作
存储层:  Checkpointer → 状态历史 → 错误后恢复
```

### 2. 为什么用装饰器而不是在每个节点里写调试代码？

```python
# ❌ 不推荐：调试代码和业务代码混在一起
def analyze(state):
    writer = get_stream_writer()
    writer({"event": "start", "node": "analyze"})
    logger.info("开始分析")
    start = time.time()

    result = f"分析: {state['query']}"  # ← 业务逻辑只有这一行

    elapsed = time.time() - start
    writer({"event": "end", "node": "analyze", "elapsed": elapsed})
    logger.info(f"分析完成, 耗时: {elapsed}")
    return {"result": result}

# ✅ 推荐：装饰器分离关注点
@traced_node("analyze")
def analyze(state):
    result = f"分析: {state['query']}"  # ← 只有业务逻辑
    return {"result": result}
```

好处：
- 业务代码干净，只关注业务逻辑
- 调试逻辑统一维护，改一处全局生效
- 可以通过环境变量控制开关（扩展装饰器即可）

### 3. StreamWriter 事件设计规范

生产环境建议统一事件格式，包含：`event`（类型：node_start/node_end/node_error）、`node`（节点名）、`timestamp`、`elapsed_ms`（耗时）、`success`（是否成功）、`error_type`/`error_message`（异常信息）、`input_keys`/`output_keys`（状态键列表）。统一格式让监控系统能自动解析和告警。

### 4. 错误后的状态检查

LangGraph 的 checkpointer 在每个节点成功后保存状态。节点异常时，之前节点的输出已保存在 checkpoint 中。通过 `get_state()` 可以看到 `values`（已保存的状态）和 `next`（失败的节点），修复问题后可以从该 checkpoint 恢复执行。

## 关键要点

1. **结构化日志是生产环境的基础**：JSON 格式让日志系统能自动解析、筛选、告警，比 print 强一个量级
2. **装饰器模式实现关注点分离**：业务代码不需要知道调试逻辑的存在，`@traced_node` 一行搞定
3. **StreamWriter 是可编程的调试通道**：输出的是结构化数据（dict），可以被前端 UI、监控系统、测试框架消费
4. **多模式组合是最佳实践**：`stream_mode=["updates", "custom"]` 同时获取状态变化和调试事件
5. **错误追踪要记录上下文**：不只是错误信息，还要记录哪个节点、耗时多少、输入是什么
6. **LangSmith 是锦上添花**：本地调试工具解决 80% 的问题，LangSmith 解决剩下的 20%（团队协作、历史对比、可视化）

---

## 扩展思考

- **调试开关**：给 `traced_node` 加 `enabled` 参数，通过环境变量 `LANGGRAPH_TRACE=0` 关闭追踪
- **日志量控制**：生产环境可以采样（只追踪 10% 的请求），在装饰器里加随机数判断
- **接入监控**：在 `stream_mode="custom"` 消费端把事件推送到 Prometheus/Kafka，接入 Grafana 看板
- **异步节点**：`traced_node` 需要异步版本，用 `async def wrapper(state)` 包装，逻辑一样
