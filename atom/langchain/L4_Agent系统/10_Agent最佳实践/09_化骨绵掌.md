# Agent最佳实践 - 化骨绵掌

> 化骨绵掌 = 深入骨髓的理解。不只是知道「怎么做」，更要理解「为什么这样做」以及「还能怎么做」。

---

## 1. Middleware 洋葱模型的本质是"信任边界"

### 表面理解

Middleware 按声明顺序形成洋葱模型，外层先执行。

### 深层理解

**每一层 Middleware 定义了一个信任边界。** 越外层的 Middleware 信任度越低，防护越严格。

```
外 ──→ 内

[不信任层] ModelCallLimit  → "我不信任 Agent 能自行停止"
[半信任层] ModelRetry      → "我信任 API 大概率能恢复"
[半信任层] ModelFallback   → "我信任总有模型可用"
[信任层]   Summarization   → "我信任摘要能保留关键信息"
[核心]     LLM 调用         → 最内层，被所有层保护

对应现实世界的安全架构：
  防火墙（不信任外部流量）
    → 认证网关（半信任已认证用户）
      → 业务逻辑（信任已授权请求）
        → 数据库（最内层，被层层保护）
```

**为什么限制必须在最外层？** 因为如果你不信任 Agent 能自行停止循环，那么任何内层的重试、降级行为都应该在这个「不信任」的管控之下。

## 2. 重试 vs 降级：解决的是不同时间尺度的故障

### 表面理解

重试处理临时故障，降级处理持久故障。

### 深层理解

```
故障的时间尺度：

毫秒级故障（网络抖动）      → 重试一次就好了
秒级故障（API 限流 429）    → 重试 + 指数退避（等一等就好了）
分钟级故障（服务重启）      → 重试 3 次都失败 → 降级到备用模型
小时级故障（供应商宕机）    → 跨供应商降级
天级故障（模型下线）        → 需要人工介入更新配置

重试和降级的关系不是"二选一"，而是"时间阶梯"：
  短时间内用重试撑住 → 撑不住了用降级兜底
```

**工程决策启示：** `max_retries × 平均退避时间` 应该小于用户可接受的等待时间。如果重试总时间超过用户耐心，那么重试没有意义——应该更早触发降级。

## 3. PII 脱敏的"测不准原理"

### 表面理解

用正则匹配检测 PII，mask/redact 处理。

### 深层理解

**PII 检测存在一个基本矛盾：检测越严格，误报越多；检测越宽松，漏报越多。**

```
过严格的问题：
  "我的订单编号是 13812345678"
  → 正则匹配到手机号格式 → 脱敏为 138****5678
  → 但实际上这是订单号，不是手机号！用户困惑

过宽松的问题：
  "请联系 zhang.san@company.com"
  → 正则没匹配到（因为邮箱前缀含点号不在简单模式中）
  → PII 泄露！

这就是为什么生产级 PII 检测通常需要：
  1. 正则匹配（快速初筛）
  2. NER 模型（语义理解）
  3. 上下文判断（"订单号"前缀 → 不脱敏）
  4. 人工配置白名单（已知的非 PII 格式）
```

**思考延伸：** PIIMiddleware 的 `strategy="block"` 策略表面上最安全，但如果误报率高，Agent 会频繁被阻断，反而降低可用性。`strategy="mask"` 是更好的平衡——即使误报，用户也能看到部分内容并理解意思。

## 4. 人工审核的"审核疲劳"问题

### 表面理解

HumanInTheLoop 让关键操作需要人工审核，更安全。

### 深层理解

**如果什么都审核，等于什么都不审核。**

```
反模式：所有操作都需要审核
  interrupt_on={
      "search_kb": {"always": True},       # 搜索也审核？
      "query_order": {"always": True},      # 查询也审核？
      "transfer_funds": {"always": True},   # 转账审核 ← 只有这个是必要的
  }

  后果：
  → 审核人每分钟收到 10+ 个审核请求
  → 审核人开始不看直接点"批准"
  → 真正危险的操作也被"习惯性批准"
  → 审核机制形同虚设

正确做法：只审核高风险操作
  interrupt_on={
      "transfer_funds": {"always": True},              # 转账：必须审核
      "send_notification": {"condition": "channel == 'sms'"},  # 短信：有条件审核
      # search_kb, query_order → 不审核（读操作，低风险）
  }
```

**设计原则：** 审核操作应遵循 **Signal-to-Noise Ratio（信噪比）**——真正需要审核的操作（信号）和不需要审核的操作（噪声）的比例。信噪比太低，审核机制就会失效。

## 5. 上下文摘要的"有损压缩"本质

### 表面理解

SummarizationMiddleware 把旧消息压缩成摘要，节省 Token。

### 深层理解

**摘要本质上是有损压缩——你不知道它丢失了什么。**

```
原始对话（10 条消息）：
  msg1: 用户问了退货政策的细节
  msg2: Agent 查了知识库
  msg3: 用户说"第 3 条不太合理"
  msg4: Agent 解释了第 3 条的背景
  msg5: 用户表示理解但还是不满意
  msg6: Agent 建议创建工单
  msg7: 用户同意
  msg8: Agent 创建了工单
  msg9: 用户问工单进度
  msg10: Agent 告知 1 小时内有人跟进

摘要后：
  "用户咨询退货政策，对第 3 条不满意，已创建工单。"

丢失了什么？
  → "第 3 条不太合理" → 用户的具体不满点
  → "Agent 解释了背景" → 解释的具体内容
  → 如果用户后续追问"你之前解释了什么"，Agent 已经不记得了
```

**工程决策：**
- `keep` 参数的选择取决于"用户多大概率会回顾之前的对话"
- 客服场景 `keep=5` 合理（用户通常关注最近的交流）
- 数据分析 `keep=3` 更合适（中间步骤不太会被回顾）
- 教学场景 `keep=10` 更安全（学生可能反复回顾）

## 6. 工具 description 就是给 LLM 的"API 文档"

### 表面理解

工具描述要清晰。

### 深层理解

**LLM 对工具的理解方式和人类阅读 API 文档一样——它在"查文档"。**

```
人类程序员选择用哪个 API：
  1. 看 API 名称 → "search" 还是 "query"
  2. 看 API 文档 → 功能描述、参数说明、返回值
  3. 看示例代码 → 什么场景下使用
  4. 看限制说明 → 什么情况下不该用

LLM 选择用哪个工具：
  1. 看 name → search_kb
  2. 看 description → "搜索知识库获取产品信息..."
  3. 看 args_schema → {"query": str, "limit": int}
  4. 看 description 中的限制 → "只能搜索已有内容"

结论：好的 description = 好的 API 文档
      差的 description = 没有文档的 API → 谁敢调用？
```

**延伸思考：** 这也解释了为什么 `args_schema` 中每个字段的 `description` 也很重要——它相当于 API 文档中每个参数的注释。

## 7. "从单 Agent 开始"不只是建议，是工程约束

### 表面理解

先简单后复杂，渐进式开发。

### 深层理解

**单 Agent 是你验证"问题定义是否正确"的最快方式。**

```
常见错误路径：
  "这个问题很复杂" → 直接设计多 Agent 系统
  → 花 2 周搭建多 Agent 架构
  → 发现其中一个 Agent 的 prompt 不对
  → 调试困难（是 prompt 的问题？还是 Agent 通信的问题？）
  → 又花 1 周排查
  → 最终发现单 Agent + 5 个工具就能解决

正确路径：
  "这个问题很复杂" → 先用单 Agent + 少量工具验证
  → 3 天搭建，跑通核心流程
  → 发现确实需要拆分（工具 > 15 个 / 权限冲突 / 需要并行）
  → 有了清晰的拆分依据再升级到多 Agent
  → 升级时已经知道每个子 Agent 的职责边界
```

**多 Agent 是优化手段，不是设计起点。**

## 8. 错误转化为观察是 Agent 自愈的核心机制

### 表面理解

ToolException 会被 handle_tool_error 处理。

### 深层理解

**LangChain Agent 最精妙的设计之一：把错误变成学习材料。**

```
传统软件的错误处理：
  函数调用 → 异常 → 捕获 → 返回错误码 → 结束

Agent 的错误处理：
  工具调用 → 异常
  → 不是直接结束！而是：
  → 将错误信息作为"观察"(observation)添加到中间步骤
  → 下一轮 plan() 时 LLM 能看到这条错误记录
  → LLM 理解自己犯了什么错 → 自行纠正 → 选择正确的工具/参数

这本质上是在做 few-shot learning：
  错误记录 = 一个"负面示例"
  LLM 看到负面示例后，避免重复同样的错误
```

**这也是为什么 `handle_parsing_errors=True` 是生产必备**——OutputParserException 被转化为 `_Exception` 工具的调用，错误信息进入 scratchpad，让 LLM 看到自己的格式错误并自我纠正。

## 9. 安全不是一个层，是每一层都要考虑的

### 表面理解

加一个 PIIMiddleware 就安全了。

### 深层理解

**安全是纵深防御（Defense in Depth），不是单点防护。**

```
单点防护的思维：
  "加了 PIIMiddleware，PII 安全问题解决了 ✅"
  → 但如果攻击者通过 prompt injection 让 LLM
    把 PII 编码为 base64 呢？正则检测不到 base64 编码的邮箱！

纵深防御的思维：
  Layer 1: System Prompt → 明确禁止输出 PII
  Layer 2: PIIMiddleware → 正则 + NER 自动检测
  Layer 3: ToolCallLimit → 限制数据查询次数
  Layer 4: HumanInTheLoop → 高风险操作人工审核
  Layer 5: Callbacks + LangSmith → 全链路审计

  每一层都不完美，但攻击者需要同时突破所有层才能得逞。
  突破的概率从 10% 变成 10% × 10% × 10% × 10% × 10% = 0.001%
```

## 10. Agent 最佳实践的元原则

### 所有最佳实践都指向同一个元原则：**让 Agent 在不确定的环境中做出确定的保护**

```
LLM 的输出是不确定的     → 但 Middleware 的保护是确定的
API 的可用性是不确定的   → 但重试和降级的行为是确定的
用户输入是不确定的       → 但 PII 检测的规则是确定的
任务步数是不确定的       → 但调用限制是确定的
对话长度是不确定的       → 但摘要触发的阈值是确定的

Agent 最佳实践 = 在不确定性中构建确定性的保护壳
```

这就是为什么 Middleware 模式比 AgentExecutor 更优秀——它把"确定的保护"从分散的配置项变成了**可组合、可插拔、声明式**的保护层。
