# 自定义Tool开发 - 第一性原理

> 从根本问题出发，推导出工具设计的必然性

---

## 什么是第一性原理？

第一性原理思维是一种从最基本的真理出发，逐步推导出结论的思考方式。在自定义 Tool 开发中，我们要回答：

**核心问题**：为什么 LLM 需要工具？工具的本质是什么？如何设计才能让 LLM 正确使用工具？

---

## 第一性原理推导链

### 问题 1：LLM 的根本局限是什么？

**基本事实**：LLM 是一个**文本生成模型**

```
输入：文本（Prompt）
处理：神经网络计算
输出：文本（Response）
```

**推导出的局限**：
1. **无法执行代码**：LLM 只能生成代码文本，不能运行代码
2. **无法访问外部系统**：LLM 无法直接调用 API、查询数据库、读取文件
3. **无法获取实时信息**：LLM 的知识截止于训练时间
4. **无法执行确定性计算**：复杂数学计算可能出错

**类比**：
- 前端：浏览器只能渲染 HTML/CSS/JS，无法直接访问文件系统（需要 File API）
- 日常：大脑只能思考，无法直接搬运重物（需要工具：起重机）

**结论 1**：LLM 需要"手"和"眼睛"来与外部世界交互，这就是工具的起源。

---

### 问题 2：如何让 LLM 调用外部函数？

**基本事实**：LLM 只能输出文本，不能直接执行函数

**挑战**：如何让 LLM "告诉" 系统它想调用哪个函数？

**解决方案演进**：

#### 方案 1：让 LLM 生成代码（❌ 不可行）
```python
# LLM 输出
"execute: get_weather('Beijing')"
```

**问题**：
- 安全风险：`eval()` 可以执行任意代码
- 不可控：LLM 可能生成错误的代码
- 难以解析：代码格式多样

#### 方案 2：让 LLM 生成结构化数据（✅ 可行）
```json
{
  "tool_name": "get_weather",
  "arguments": {
    "location": "Beijing",
    "units": "celsius"
  }
}
```

**优势**：
- 安全：只是数据，不是可执行代码
- 可解析：标准 JSON 格式
- 可验证：可以检查参数类型和值

**类比**：
- 前端：用户点击按钮（事件）→ 触发回调函数
- 日常：顾客点菜（菜单）→ 厨师做菜

**结论 2**：LLM 通过生成结构化的"调用请求"（JSON）来间接调用函数。

---

### 问题 3：LLM 如何知道有哪些函数可以调用？

**基本事实**：LLM 没有"内置"的函数列表

**挑战**：如何让 LLM 知道系统提供了哪些工具？

**解决方案**：在 Prompt 中提供工具列表

```
System: You have access to the following tools:

1. get_weather
   Description: Get current weather for a location
   Parameters:
     - location (string, required): City name
     - units (string, optional): "celsius" or "fahrenheit"

2. search_database
   Description: Search the product database
   Parameters:
     - query (string, required): Search keywords
     - limit (integer, optional): Max results (default: 10)

User: What's the weather in Beijing?
```

**LLM 的推理过程**：
1. 分析用户问题：需要获取天气信息
2. 查看可用工具：发现 `get_weather` 工具
3. 生成调用请求：`{"tool_name": "get_weather", "arguments": {"location": "Beijing"}}`

**类比**：
- 前端：API 文档（开发者通过文档了解可用接口）
- 日常：餐厅菜单（顾客通过菜单了解可点的菜品）

**结论 3**：工具必须有清晰的"说明书"（名称 + 描述 + 参数列表），LLM 才能理解如何使用。

---

### 问题 4：如何确保 LLM 传递正确的参数？

**基本事实**：LLM 可能生成错误的参数（类型错误、缺少必需参数、超出范围）

**挑战**：如何验证 LLM 生成的参数是否合法？

**解决方案**：定义参数 Schema（类型系统）

```python
# Schema 定义
{
  "location": {
    "type": "string",
    "description": "City name",
    "required": True
  },
  "units": {
    "type": "string",
    "enum": ["celsius", "fahrenheit"],
    "default": "celsius",
    "required": False
  }
}
```

**验证流程**：
```
LLM 生成参数
    ↓
Schema 验证
    ↓
验证通过 → 执行函数
验证失败 → 返回错误给 LLM
```

**类比**：
- 前端：TypeScript 类型检查（编译时验证参数类型）
- 日常：机场安检（检查行李是否符合规定）

**结论 4**：工具必须有严格的参数 Schema，确保调用安全性和正确性。

---

### 问题 5：如何让开发者方便地创建工具？

**基本事实**：手动编写工具定义（名称、描述、Schema、函数）非常繁琐

**挑战**：如何简化工具创建流程？

**解决方案演进**：

#### 方案 1：手动定义所有字段（❌ 繁琐）
```python
tool = {
    "name": "get_weather",
    "description": "Get current weather",
    "parameters": {
        "type": "object",
        "properties": {
            "location": {"type": "string", "description": "City name"},
            "units": {"type": "string", "enum": ["celsius", "fahrenheit"]}
        },
        "required": ["location"]
    },
    "function": get_weather_impl
}
```

#### 方案 2：从函数签名自动推断（✅ 简洁）
```python
from langchain.tools import tool

@tool
def get_weather(location: str, units: str = "celsius") -> str:
    """Get current weather for a location.

    Args:
        location: City name
        units: Temperature units
    """
    return f"Weather in {location}: 22°{units[0].upper()}"
```

**自动推断的内容**：
- 工具名称：从函数名推断（`get_weather`）
- 参数 Schema：从类型提示推断（`location: str`）
- 参数描述：从 docstring 推断（`Args` 部分）
- 函数实现：直接使用原函数

**类比**：
- 前端：React Hooks（自动推断依赖）vs 手动管理生命周期
- 日常：自动挡汽车 vs 手动挡汽车

**结论 5**：工具创建应该尽可能自动化，减少开发者的重复劳动。

---

### 问题 6：如何支持复杂的工具需求？

**基本事实**：不同工具有不同的复杂度需求

**挑战**：如何在简洁性和灵活性之间取得平衡？

**解决方案**：提供多层次的抽象

#### 层次 1：装饰器（简单场景）
```python
@tool
def simple_tool(param: str) -> str:
    """Simple tool."""
    return f"Result: {param}"
```

**适用场景**：
- 无状态工具
- 简单的函数包装
- 快速原型开发

#### 层次 2：工厂方法（中等复杂度）
```python
from langchain.tools import StructuredTool

tool = StructuredTool.from_function(
    name="custom_name",
    description="Custom description",
    func=existing_function,
    args_schema=CustomSchema,
)
```

**适用场景**：
- 包装现有 API 客户端
- 需要自定义工具元数据
- 显式控制 Schema

#### 层次 3：类继承（最大控制）
```python
from langchain.tools import BaseTool

class ComplexTool(BaseTool):
    name = "complex_tool"
    description = "Complex tool with state"

    # 工具状态
    connection_pool: Any = None

    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self.connection_pool = create_pool()

    def _run(self, param: str, config: RunnableConfig) -> str:
        # 使用状态
        with self.connection_pool.get_connection() as conn:
            return conn.query(param)

    async def _arun(self, param: str, config: RunnableConfig) -> str:
        # 异步实现
        async with self.connection_pool.get_async_connection() as conn:
            return await conn.query(param)
```

**适用场景**：
- 需要维护状态（连接池、会话）
- 复杂的初始化和清理逻辑
- 需要重写多个生命周期方法

**类比**：
- 前端：函数组件 vs 类组件 vs 自定义 Hooks
- 日常：快餐 vs 点菜 vs 私人定制

**结论 6**：工具系统应该提供多层次的抽象，让开发者根据需求选择合适的复杂度。

### 问题 7：如何让工具与 LangChain 生态集成？

**基本事实**：LangChain 有统一的 Runnable 协议

**挑战**：如何让工具无缝集成到 LCEL 链和 Agent 中？

**解决方案**：工具继承自 RunnableSerializable

```python
class BaseTool(RunnableSerializable[Union[str, Dict, ToolCall], Any]):
    """Base class for tools."""
```

**集成效果**：

#### 1. 支持 LCEL 管道操作符
```python
chain = prompt | llm | tool | parser
```

#### 2. 支持标准调用接口
```python
# 同步调用
result = tool.invoke({"param": "value"})

# 异步调用
result = await tool.ainvoke({"param": "value"})

# 批量调用
results = tool.batch([{"param": "value1"}, {"param": "value2"}])

# 流式调用
for chunk in tool.stream({"param": "value"}):
    print(chunk)
```

#### 3. 支持配置传递
```python
result = tool.invoke(
    {"param": "value"},
    config={"configurable": {"user_id": "123"}}
)
```

**类比**：
- 前端：React 组件必须返回 JSX（统一接口）
- 日常：所有电器必须符合插座标准（统一接口）

**结论 7**：工具必须实现统一的接口协议，才能与框架无缝集成。

## 三层架构的必然性

通过以上推导，我们得出工具设计的三层架构：

### 第一层：函数实现（Function）
```python
def get_weather_impl(location: str, units: str) -> str:
    """实际的业务逻辑"""
    api_response = weather_api.get(location, units)
    return format_response(api_response)
```

**职责**：执行具体的业务逻辑

### 第二层：参数 Schema（Schema）
```python
class WeatherInput(BaseModel):
    location: str = Field(description="City name")
    units: Literal["celsius", "fahrenheit"] = Field(default="celsius")
```

**职责**：
- 定义参数类型和约束
- 提供参数描述（给 LLM 看）
- 验证参数合法性

### 第三层：工具对象（Tool Object）
```python
@tool(args_schema=WeatherInput)
def get_weather(location: str, units: str = "celsius") -> str:
    """Get current weather for a location."""
    return get_weather_impl(location, units)
```

**职责**：
- 封装函数和 Schema
- 提供工具元数据（名称、描述）
- 实现 Runnable 接口
- 处理错误和配置

**架构图**：
```
┌─────────────────────────────────────────┐
│         Tool Object (工具对象)           │
│  - name: "get_weather"                  │
│  - description: "Get weather..."        │
│  - 实现 Runnable 接口                    │
│  - 错误处理                              │
│  ┌───────────────────────────────────┐  │
│  │    Schema (参数定义)               │  │
│  │  - location: str (required)       │  │
│  │  - units: Literal[...] (optional) │  │
│  │  - 参数验证                        │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │    Function (函数实现)             │  │
│  │  - 调用 API                        │  │
│  │  - 处理数据                        │  │
│  │  - 返回结果                        │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

## LangChain 设计的合理性

### 1. @tool 装饰器的设计

**为什么需要装饰器？**
- 最小化样板代码
- 自动推断 Schema
- 保持函数的可读性

**为什么支持多种使用方式？**
```python
# 无参装饰器
@tool
def simple_tool(param: str) -> str:
    """Simple tool."""
    return param

# 带参装饰器
@tool("custom_name", parse_docstring=True)
def advanced_tool(param: str) -> str:
    """Advanced tool.

    Args:
        param: Parameter description.
    """
    return param

# 函数调用
my_tool = tool(existing_function)
```

**设计理由**：
- 无参装饰器：最简洁，适合 80% 场景
- 带参装饰器：需要自定义配置
- 函数调用：包装现有函数

**来源**：[源码分析 convert.py | @tool 装饰器实现]

### 2. StructuredTool 的设计

**为什么需要 StructuredTool？**
- 显式控制工具元数据
- 包装现有 API 客户端
- 分离函数实现和工具定义

**为什么使用 from_function() 类方法？**
```python
tool = StructuredTool.from_function(
    name="send_email",
    description="Send email via API",
    func=api_wrapper.send,
    args_schema=EmailInput,
)
```

**设计理由**：
- 工厂模式：创建逻辑集中管理
- 类型安全：显式指定所有参数
- 灵活性：可以包装任何可调用对象

**来源**：[Context7 文档 | StructuredTool 用法]

### 3. BaseTool 的设计

**为什么需要 BaseTool？**
- 最大控制权
- 状态管理
- 生命周期控制

**为什么分离 _run() 和 _arun()？**
```python
class MyTool(BaseTool):
    def _run(self, param: str, config: RunnableConfig) -> str:
        """Synchronous implementation."""
        return sync_operation(param)

    async def _arun(self, param: str, config: RunnableConfig) -> str:
        """Asynchronous implementation."""
        return await async_operation(param)
```

**设计理由**：
- 同步/异步分离：避免强制异步
- 自动回退：没有 _arun() 时自动使用线程池
- 性能优化：异步实现性能更好

**来源**：[源码分析 base.py | BaseTool 基类设计]

## 核心设计原则总结

### 1. 安全性原则
- LLM 不能直接执行代码
- 所有参数必须经过 Schema 验证
- 使用 ToolException 处理可恢复错误

### 2. 简洁性原则
- 优先使用装饰器（最少代码）
- 自动推断 Schema（减少重复）
- 合理的默认值（开箱即用）

### 3. 灵活性原则
- 三层抽象（装饰器、工厂、继承）
- 支持同步和异步
- 支持状态管理

### 4. 可组合性原则
- 实现 Runnable 接口
- 支持 LCEL 管道操作符
- 支持配置传递

### 5. 可观测性原则
- 支持 callbacks
- 支持 RunnableConfig
- 支持错误处理和日志

## 实际应用示例

### 示例 1：从问题到解决方案

**问题**：构建一个天气查询 Agent

**第一性原理推导**：

1. **LLM 局限**：无法直接获取实时天气数据
2. **解决方案**：创建 get_weather 工具
3. **Schema 需求**：location (必需), units (可选)
4. **实现选择**：简单工具，使用 @tool 装饰器

**代码实现**：
```python
from langchain.tools import tool
from typing import Literal
import httpx

@tool
async def get_weather(
    location: str,
    units: Literal["celsius", "fahrenheit"] = "celsius"
) -> str:
    """Get current weather for a location.

    Use this when the user asks about weather conditions,
    temperature, or forecast for a specific city.

    Args:
        location: City name (e.g., "Beijing", "New York")
        units: Temperature units preference

    Returns:
        Current weather description with temperature
    """
    api_key = os.getenv("WEATHER_API_KEY")
    async with httpx.AsyncClient() as client:
        response = await client.get(
            "https://api.weather.com/v1/current",
            params={"q": location, "units": units, "key": api_key}
        )
        data = response.json()
        return f"Weather in {location}: {data['temp']}°{units[0].upper()}, {data['condition']}"
```

**为什么这样设计？**
- ✅ 使用 @tool：代码简洁
- ✅ 异步函数：性能更好
- ✅ Literal 类型：限制可选值
- ✅ 详细描述：帮助 LLM 理解何时调用
- ✅ 类型提示：自动推断 Schema

### 示例 2：复杂工具的设计

**问题**：构建一个数据库查询工具，需要连接池管理

**第一性原理推导**：

1. **状态需求**：需要维护数据库连接池
2. **生命周期**：需要初始化和清理逻辑
3. **实现选择**：复杂工具，使用 BaseTool 继承

**代码实现**：
```python
from langchain.tools import BaseTool
from pydantic import BaseModel, Field, validator
from typing import Any
import sqlalchemy

class QueryInput(BaseModel):
    table: str = Field(description="Table name")
    columns: list[str] = Field(description="Columns to select")
    limit: int = Field(default=10, ge=1, le=100)

    @validator('table')
    def validate_table(cls, v):
        # 白名单验证
        allowed_tables = ['users', 'products', 'orders']
        if v not in allowed_tables:
            raise ValueError(f"Table must be one of {allowed_tables}")
        return v

class DatabaseTool(BaseTool):
    name = "database_query"
    description = "Execute safe SQL queries on the database"
    args_schema = QueryInput

    # 工具状态
    engine: Any = None

    def __init__(self, connection_string: str, **kwargs):
        super().__init__(**kwargs)
        self.engine = sqlalchemy.create_engine(
            connection_string,
            pool_size=10,
            max_overflow=20
        )

    def _run(
        self,
        table: str,
        columns: list[str],
        limit: int,
        config: RunnableConfig = None
    ) -> str:
        """Synchronous execution."""
        # 参数化查询防止 SQL 注入
        cols = ", ".join(columns)
        query = f"SELECT {cols} FROM {table} LIMIT :limit"

        with self.engine.connect() as conn:
            result = conn.execute(
                sqlalchemy.text(query),
                {"limit": limit}
            )
            rows = result.fetchall()
            return str(rows)

    async def _arun(
        self,
        table: str,
        columns: list[str],
        limit: int,
        config: RunnableConfig = None
    ) -> str:
        """Asynchronous execution."""
        # 使用异步引擎
        async with self.engine.begin() as conn:
            result = await conn.execute(
                sqlalchemy.text(f"SELECT {', '.join(columns)} FROM {table} LIMIT :limit"),
                {"limit": limit}
            )
            rows = await result.fetchall()
            return str(rows)

    def __del__(self):
        """清理资源"""
        if self.engine:
            self.engine.dispose()
```

**为什么这样设计？**
- ✅ 使用 BaseTool：需要状态管理
- ✅ 连接池：提升性能
- ✅ 白名单验证：防止 SQL 注入
- ✅ 参数化查询：安全性
- ✅ 同步/异步双实现：灵活性
- ✅ 资源清理：避免泄漏

## 与 AI Agent 架构的关系

### Agent 执行循环中的工具调用

```
┌─────────────────────────────────────────────────────┐
│                  Agent 执行循环                      │
│                                                     │
│  1. 接收用户输入                                     │
│     ↓                                               │
│  2. LLM 推理（基于 Prompt + 工具列表）               │
│     ↓                                               │
│  3. 决策：回复 or 调用工具？                         │
│     ├─ 回复 → 返回结果                              │
│     └─ 调用工具 → 继续                              │
│        ↓                                            │
│  4. 解析工具调用请求（JSON）                         │
│     ↓                                               │
│  5. Schema 验证                                     │
│     ├─ 验证失败 → 返回错误给 LLM                    │
│     └─ 验证通过 → 继续                              │
│        ↓                                            │
│  6. 执行工具函数                                     │
│     ↓                                               │
│  7. 获取工具结果                                     │
│     ↓                                               │
│  8. 将结果注入 Prompt                                │
│     ↓                                               │
│  9. LLM 继续推理（基于工具结果）                     │
│     ↓                                               │
│  10. 循环直到生成最终回复                            │
└─────────────────────────────────────────────────────┘
```

### 工具在 Agent 中的作用

1. **扩展能力边界**：让 Agent 能做 LLM 做不到的事
2. **提供实时信息**：突破知识截止日期限制
3. **执行确定性操作**：避免 LLM 的不确定性
4. **集成外部系统**：连接企业内部系统

## 常见误区与澄清

### 误区 1："工具就是函数"
❌ **错误理解**：工具 = 函数

✅ **正确理解**：工具 = 函数 + Schema + 元数据 + Runnable 接口

**为什么？**
- 函数只是实现逻辑
- Schema 让 LLM 理解如何调用
- 元数据让 LLM 理解何时调用
- Runnable 接口让工具集成到框架

---

### 误区 2："LLM 可以直接执行代码"
❌ **错误理解**：LLM 生成代码 → 直接执行

✅ **正确理解**：LLM 生成调用请求（JSON）→ 系统执行工具

**为什么？**
- 安全性：不能让 LLM 执行任意代码
- 可控性：必须经过验证和审核
- 可靠性：工具执行是确定性的

---

### 误区 3："Schema 是可选的"
❌ **错误理解**：不定义 Schema 也能工作

✅ **正确理解**：Schema 是 LLM 理解工具的关键

**为什么？**
- 没有 Schema，LLM 不知道如何传参
- 自动推断的 Schema 可能不够准确
- 显式定义 Schema 提升可靠性

---

### 误区 4："所有工具都要用 BaseTool"
❌ **错误理解**：BaseTool 最灵活，应该都用它

✅ **正确理解**：优先用 @tool，需要时才用 BaseTool

**为什么？**
- @tool 更简洁，适合 80% 场景
- BaseTool 适合需要状态管理的复杂工具
- 过度设计会增加维护成本

## 总结：从第一性原理到实践

### 核心推导链

```
LLM 只能生成文本
    ↓
需要工具与外部世界交互
    ↓
LLM 通过生成 JSON 调用工具
    ↓
工具必须有清晰的说明书（Schema）
    ↓
Schema 必须验证参数合法性
    ↓
工具创建应该尽可能自动化
    ↓
提供多层次抽象（装饰器、工厂、继承）
    ↓
工具必须实现统一接口（Runnable）
    ↓
三层架构：Function + Schema + Tool Object
```

### 设计原则

1. **安全性**：Schema 验证、参数化查询、ToolException
2. **简洁性**：装饰器、自动推断、合理默认值
3. **灵活性**：三层抽象、同步/异步、状态管理
4. **可组合性**：Runnable 接口、LCEL 集成、配置传递
5. **可观测性**：callbacks、RunnableConfig、错误处理

### 实践建议

1. **优先使用 @tool 装饰器**：简洁高效
2. **显式定义 Schema**：提升可靠性
3. **提供详细描述**：帮助 LLM 理解
4. **处理错误情况**：使用 ToolException
5. **考虑异步实现**：提升性能
