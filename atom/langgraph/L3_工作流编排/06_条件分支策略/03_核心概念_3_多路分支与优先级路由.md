# 核心概念3：多路分支与优先级路由

## 一句话定义

**多路分支让一个路由函数同时激活多个下游节点并行执行，优先级路由则按条件优先级顺序匹配第一个满足的路径——两者结合构成了复杂工作流的核心调度能力。**

---

## 多路分支基础

### 什么是多路分支？

普通条件边是"多选一"——根据状态选择一个目标节点。多路分支是"多选多"——根据状态同时选择多个目标节点，让它们并行执行。

```python
# 单路分支：选一个
def route_single(state) -> str:
    return "node_a"  # 只去一个节点

# 多路分支：选多个
def route_multiple(state) -> list[str]:
    return ["node_a", "node_b", "node_c"]  # 同时去三个节点
```

**前端类比：** 单路分支就像 `if-else`，只走一条路。多路分支就像 `Promise.all()`，同时发起多个异步请求。

**日常生活类比：** 单路分支是"你去买菜"。多路分支是"你去买菜，他去买肉，她去买水果"——同时派出多个人，最后汇总结果。

### 基本用法

```python
import operator
from typing import TypedDict, Annotated, Sequence
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    input: str
    results: Annotated[list[str], operator.add]  # reducer：合并并行结果

def analyzer(state: State):
    return {"input": state["input"]}

def sentiment_check(state: State):
    return {"results": [f"情感分析: {state['input']}"]}

def keyword_extract(state: State):
    return {"results": [f"关键词提取: {state['input']}"]}

def summary_gen(state: State):
    return {"results": [f"摘要生成: {state['input']}"]}

def aggregator(state: State):
    """汇总所有并行结果"""
    combined = " | ".join(state["results"])
    return {"results": [f"汇总: {combined}"]}

# 多路分支：同时激活多个分析节点
def route_to_analyzers(state: State) -> Sequence[str]:
    """返回列表 → 并行执行多个节点"""
    return ["sentiment", "keyword", "summary"]

# 构建图
graph = StateGraph(State)
graph.add_node("analyzer", analyzer)
graph.add_node("sentiment", sentiment_check)
graph.add_node("keyword", keyword_extract)
graph.add_node("summary", summary_gen)
graph.add_node("aggregator", aggregator)

graph.add_edge(START, "analyzer")
graph.add_conditional_edges("analyzer", route_to_analyzers)

# 所有并行节点完成后汇总
graph.add_edge("sentiment", "aggregator")
graph.add_edge("keyword", "aggregator")
graph.add_edge("summary", "aggregator")
graph.add_edge("aggregator", END)

app = graph.compile()
result = app.invoke({"input": "LangGraph 真好用", "results": []})
print(result["results"])
```

执行流程：

```
                    ┌→ sentiment ─┐
START → analyzer ──→├→ keyword  ──├→ aggregator → END
                    └→ summary  ──┘
                    （三个节点并行执行）
```

### 关键点：必须用 Reducer

并行节点会同时写入状态。如果没有 Reducer，后完成的节点会覆盖先完成的节点的结果：

```python
# 错误：没有 reducer，并行结果会互相覆盖
class BadState(TypedDict):
    results: list[str]  # 没有 Annotated + operator.add

# 正确：用 reducer 合并并行结果
class GoodState(TypedDict):
    results: Annotated[list[str], operator.add]  # 自动合并
```

### 多路分支 vs Send：什么时候用哪个？

这是一个常见的困惑点。两者都能实现并行，但适用场景不同：

| 维度 | 多路分支（返回列表） | Send 动态并行 |
|------|---------------------|--------------|
| 目标节点 | 不同的节点 | 同一个节点的多个实例 |
| 并行数量 | 编码时已知 | 运行时动态决定 |
| 输入状态 | 共享同一个状态 | 每个实例可以有不同状态 |
| 典型场景 | 多维度分析 | Map-Reduce、批量处理 |

```python
# 多路分支：去不同的节点
def route(state) -> list[str]:
    return ["sentiment", "keyword", "summary"]
    # → 三个不同的节点并行执行，共享状态

# Send：去同一个节点的多个实例
from langgraph.types import Send
def route(state):
    return [Send("worker", {"item": item}) for item in state["items"]]
    # → 同一个 worker 节点被创建多个实例，每个实例有不同的输入
```

**日常生活类比：**
- 多路分支 = 同时派不同部门做不同的事（市场部调研、技术部开发、设计部出图）
- Send = 同时派多个快递员送不同的包裹（同一个角色，不同的任务数据）

## Supervisor 路由模式

### 什么是 Supervisor 模式？

Supervisor（监督者）模式是多 Agent 系统中最常见的架构：一个中央调度节点（Supervisor）接收所有请求，根据请求类型分发给不同的专家 Agent 处理。

```
                    ┌→ search_agent ──┐
用户请求 → supervisor ├→ chat_agent   ──├→ 汇总响应
                    └→ code_agent   ──┘
```

**前端类比：** Supervisor 就像 API Gateway——所有请求先到网关，网关根据 URL 路径分发到不同的微服务。

**日常生活类比：** Supervisor 就像医院的分诊台——所有患者先到分诊台，护士根据症状分配到不同科室（内科、外科、眼科）。

### 基础实现

```python
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END

class AgentState(TypedDict):
    query: str
    intent: str
    response: str

# ===== Supervisor 节点：分类意图 =====
def supervisor(state: AgentState):
    """中央调度：分析用户意图"""
    query = state["query"].lower()
    if "搜索" in query or "查找" in query:
        intent = "search"
    elif "写代码" in query or "编程" in query:
        intent = "code"
    else:
        intent = "chat"
    return {"intent": intent}

# ===== 专家 Agent 节点 =====
def search_agent(state: AgentState):
    return {"response": f"[搜索Agent] 正在搜索: {state['query']}"}

def chat_agent(state: AgentState):
    return {"response": f"[聊天Agent] 回复: {state['query']}"}

def code_agent(state: AgentState):
    return {"response": f"[代码Agent] 生成代码: {state['query']}"}

# ===== 路由函数 =====
SPECIALISTS = {
    "search": "search_agent",
    "chat": "chat_agent",
    "code": "code_agent",
}

def route_to_specialist(state: AgentState) -> Literal["search_agent", "chat_agent", "code_agent"]:
    """根据意图路由到专家 Agent"""
    intent = state["intent"]
    return SPECIALISTS.get(intent, "chat_agent")

# ===== 构建图 =====
graph = StateGraph(AgentState)
graph.add_node("supervisor", supervisor)
graph.add_node("search_agent", search_agent)
graph.add_node("chat_agent", chat_agent)
graph.add_node("code_agent", code_agent)

graph.add_edge(START, "supervisor")
graph.add_conditional_edges("supervisor", route_to_specialist, SPECIALISTS)
graph.add_edge("search_agent", END)
graph.add_edge("chat_agent", END)
graph.add_edge("code_agent", END)

app = graph.compile()

# 测试
for query in ["帮我搜索 Python 教程", "写一个排序算法", "今天天气怎么样"]:
    result = app.invoke({"query": query, "intent": "", "response": ""})
    print(f"查询: {query}")
    print(f"响应: {result['response']}\n")
```

**预期输出：**

```
查询: 帮我搜索 Python 教程
响应: [搜索Agent] 正在搜索: 帮我搜索 Python 教程

查询: 写一个排序算法
响应: [代码Agent] 生成代码: 写一个排序算法

查询: 今天天气怎么样
响应: [聊天Agent] 回复: 今天天气怎么样
```

### Supervisor 模式的变体：带回环

更实际的 Supervisor 模式中，专家 Agent 处理完后会回到 Supervisor，由 Supervisor 决定是否需要继续处理：

```python
from typing import TypedDict, Literal, Annotated
import operator
from langgraph.graph import StateGraph, START, END

class LoopState(TypedDict):
    query: str
    steps: Annotated[list[str], operator.add]
    iteration: int
    done: bool

def supervisor(state: LoopState):
    """Supervisor 检查是否需要继续"""
    iteration = state.get("iteration", 0) + 1
    done = iteration >= 3  # 最多 3 轮
    return {"iteration": iteration, "done": done}

def worker(state: LoopState):
    step = f"第{state['iteration']}轮处理: {state['query']}"
    return {"steps": [step]}

def route_supervisor(state: LoopState) -> Literal["worker", "__end__"]:
    if state.get("done", False):
        return END
    return "worker"

graph = StateGraph(LoopState)
graph.add_node("supervisor", supervisor)
graph.add_node("worker", worker)

graph.add_edge(START, "supervisor")
graph.add_conditional_edges("supervisor", route_supervisor)
graph.add_edge("worker", "supervisor")  # 回环：worker → supervisor

app = graph.compile()
result = app.invoke({"query": "分析数据", "steps": [], "iteration": 0, "done": False})
print(result["steps"])
# ['第1轮处理: 分析数据', '第2轮处理: 分析数据']
```

执行流程：

```
START → supervisor → worker → supervisor → worker → supervisor → END
         (轮1)      (处理)    (轮2)       (处理)    (轮3,done)
```

## 优先级路由策略

### 什么是优先级路由？

优先级路由是一种按优先级顺序检查条件的模式——第一个匹配的条件决定路由方向，后续条件不再检查。这和 `if-elif-else` 链的逻辑完全一致。

```python
# 优先级路由的本质就是 if-elif-else 链
def priority_route(state) -> str:
    # 优先级1（最高）：紧急情况
    if state.get("is_emergency"):
        return "emergency_handler"
    # 优先级2：VIP 用户
    if state.get("is_vip"):
        return "vip_handler"
    # 优先级3：已认证用户
    if state.get("is_authenticated"):
        return "auth_handler"
    # 默认回退（最低优先级）
    return "default_handler"
```

**前端类比：** 优先级路由就像 CSS 的选择器优先级——`!important` > 内联样式 > ID > class > 标签。第一个匹配的规则生效。

**日常生活类比：** 优先级路由就像急诊室的分诊——心脏骤停 > 大出血 > 骨折 > 感冒。最严重的优先处理。

### 完整实现

```python
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END

class RequestState(TypedDict):
    query: str
    is_emergency: bool
    is_vip: bool
    is_authenticated: bool
    response: str

# ===== 分类节点 =====
def classify_request(state: RequestState):
    """分析请求属性"""
    query = state["query"].lower()
    return {
        "is_emergency": "紧急" in query or "崩溃" in query,
        "is_vip": "vip" in query.lower(),
        "is_authenticated": "已登录" in query,
    }

# ===== 处理节点 =====
def emergency_handler(state: RequestState):
    return {"response": "[紧急通道] 已启动应急响应流程"}

def vip_handler(state: RequestState):
    return {"response": "[VIP通道] 专属客服已接入"}

def auth_handler(state: RequestState):
    return {"response": "[认证通道] 正在处理您的请求"}

def default_handler(state: RequestState):
    return {"response": "[默认通道] 请先登录或描述您的问题"}

# ===== 优先级路由函数 =====
def priority_route(state: RequestState) -> Literal[
    "emergency_handler", "vip_handler", "auth_handler", "default_handler"
]:
    """按优先级顺序检查条件，第一个匹配的决定路由"""
    if state.get("is_emergency"):
        return "emergency_handler"     # 优先级1：紧急
    if state.get("is_vip"):
        return "vip_handler"           # 优先级2：VIP
    if state.get("is_authenticated"):
        return "auth_handler"          # 优先级3：已认证
    return "default_handler"           # 默认回退

# ===== 构建图 =====
graph = StateGraph(RequestState)
graph.add_node("classify", classify_request)
graph.add_node("emergency_handler", emergency_handler)
graph.add_node("vip_handler", vip_handler)
graph.add_node("auth_handler", auth_handler)
graph.add_node("default_handler", default_handler)

graph.add_edge(START, "classify")
graph.add_conditional_edges("classify", priority_route)
for handler in ["emergency_handler", "vip_handler", "auth_handler", "default_handler"]:
    graph.add_edge(handler, END)

app = graph.compile()

# 测试
test_cases = [
    {"query": "紧急！系统崩溃了", "is_emergency": False, "is_vip": False,
     "is_authenticated": False, "response": ""},
    {"query": "VIP用户，已登录，查询订单", "is_emergency": False, "is_vip": False,
     "is_authenticated": False, "response": ""},
    {"query": "已登录，修改密码", "is_emergency": False, "is_vip": False,
     "is_authenticated": False, "response": ""},
    {"query": "随便问问", "is_emergency": False, "is_vip": False,
     "is_authenticated": False, "response": ""},
]

for case in test_cases:
    result = app.invoke(case)
    print(f"查询: {case['query']}")
    print(f"响应: {result['response']}\n")
```

**预期输出：**

```
查询: 紧急！系统崩溃了
响应: [紧急通道] 已启动应急响应流程

查询: VIP用户，已登录，查询订单
响应: [VIP通道] 专属客服已接入

查询: 已登录，修改密码
响应: [认证通道] 正在处理您的请求

查询: 随便问问
响应: [默认通道] 请先登录或描述您的问题
```

### 优先级路由的关键设计原则

1. **条件互斥性：** 理想情况下，每个请求只匹配一个条件。如果条件有重叠（如"紧急 VIP 用户"），优先级顺序决定走哪条路。
2. **默认回退：** 永远要有一个默认路径，避免路由函数返回未注册的节点名。
3. **条件检查顺序：** 最严格/最紧急的条件放在最前面。

## 条件链模式

### 什么是条件链？

条件链是指多个条件边串联使用，形成多级决策树。每一级根据不同的维度做路由决策。

```
                                    ┌→ vip_refund
                    ┌→ refund ──→ ──┤
                    │               └→ normal_refund
用户请求 → classify ─┤
                    │               ┌→ urgent_complaint
                    └→ complaint ──→┤
                                    └→ normal_complaint
```

### 实现：两级条件路由

```python
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END

class TicketState(TypedDict):
    query: str
    category: str
    priority: str
    response: str

# ===== 第一级：分类 =====
def classify_ticket(state: TicketState):
    query = state["query"].lower()
    if "退款" in query:
        category = "refund"
    elif "投诉" in query:
        category = "complaint"
    else:
        category = "inquiry"
    return {"category": category}

# ===== 第二级：优先级评估 =====
def assess_refund(state: TicketState):
    priority = "high" if "紧急" in state["query"] else "normal"
    return {"priority": priority}

def assess_complaint(state: TicketState):
    priority = "high" if "严重" in state["query"] else "normal"
    return {"priority": priority}

# ===== 终端处理节点 =====
def handle_urgent_refund(state: TicketState):
    return {"response": "紧急退款：已加急处理，24小时内到账"}

def handle_normal_refund(state: TicketState):
    return {"response": "普通退款：3-5个工作日处理"}

def handle_urgent_complaint(state: TicketState):
    return {"response": "严重投诉：已转高级主管"}

def handle_normal_complaint(state: TicketState):
    return {"response": "普通投诉：48小时内回复"}

def handle_inquiry(state: TicketState):
    return {"response": "咨询：请查看帮助中心"}

# ===== 路由函数 =====
def route_category(state: TicketState) -> Literal["assess_refund", "assess_complaint", "inquiry_handler"]:
    return {
        "refund": "assess_refund",
        "complaint": "assess_complaint",
    }.get(state["category"], "inquiry_handler")

def route_refund_priority(state: TicketState) -> Literal["urgent_refund", "normal_refund"]:
    return "urgent_refund" if state["priority"] == "high" else "normal_refund"

def route_complaint_priority(state: TicketState) -> Literal["urgent_complaint", "normal_complaint"]:
    return "urgent_complaint" if state["priority"] == "high" else "normal_complaint"

# ===== 构建图 =====
graph = StateGraph(TicketState)

# 添加所有节点
graph.add_node("classify", classify_ticket)
graph.add_node("assess_refund", assess_refund)
graph.add_node("assess_complaint", assess_complaint)
graph.add_node("urgent_refund", handle_urgent_refund)
graph.add_node("normal_refund", handle_normal_refund)
graph.add_node("urgent_complaint", handle_urgent_complaint)
graph.add_node("normal_complaint", handle_normal_complaint)
graph.add_node("inquiry_handler", handle_inquiry)

# 第一级条件边：按类别路由
graph.add_edge(START, "classify")
graph.add_conditional_edges("classify", route_category)

# 第二级条件边：按优先级路由
graph.add_conditional_edges("assess_refund", route_refund_priority)
graph.add_conditional_edges("assess_complaint", route_complaint_priority)

# 所有终端节点 → END
for node in ["urgent_refund", "normal_refund", "urgent_complaint",
             "normal_complaint", "inquiry_handler"]:
    graph.add_edge(node, END)

app = graph.compile()

# 测试
for query in ["紧急退款，订单有误", "投诉服务态度", "怎么修改地址"]:
    result = app.invoke({"query": query, "category": "", "priority": "", "response": ""})
    print(f"查询: {query} → {result['response']}")
```

### 条件链 vs 单层路由

| 维度 | 单层路由 | 条件链（多级） |
|------|---------|---------------|
| 路由函数数量 | 1 个 | 多个（每级一个） |
| 决策维度 | 单一维度 | 多个维度逐级细化 |
| 可维护性 | 简单场景好 | 复杂场景好 |
| 可视化 | 扁平 | 树状，层次清晰 |
| 适用场景 | 3-5 个分支 | 分支有层级关系 |

## 实际应用场景

### 场景1：客服系统意图路由

```python
from typing import Literal

INTENT_MAP = {
    "billing": "billing_agent",
    "technical": "tech_agent",
    "sales": "sales_agent",
    "general": "general_agent",
}

def route_customer_intent(state) -> Literal[
    "billing_agent", "tech_agent", "sales_agent", "general_agent"
]:
    """LLM 分类用户意图，路由到对应的专家 Agent"""
    intent = state["classified_intent"]
    return INTENT_MAP.get(intent, "general_agent")

graph.add_conditional_edges("intent_classifier", route_customer_intent, INTENT_MAP)
```

### 场景2：多模型选择路由

根据任务复杂度选择不同的 LLM 模型，优化成本和性能：

```python
def route_to_model(state) -> Literal["fast_model", "standard_model", "powerful_model"]:
    """根据任务复杂度选择模型"""
    complexity = state["task_complexity"]
    token_count = state["estimated_tokens"]

    # 优先级路由：复杂度高的用强模型
    if complexity == "high" or token_count > 4000:
        return "powerful_model"    # GPT-4 / Claude Opus
    elif complexity == "medium":
        return "standard_model"    # GPT-3.5 / Claude Sonnet
    else:
        return "fast_model"        # 轻量模型，低成本

graph.add_conditional_edges("complexity_analyzer", route_to_model)
```

### 场景3：RAG 质量评估后的分支

检索结果质量不同，走不同的处理路径：

```python
def route_by_retrieval_quality(state) -> Literal["direct_answer", "rewrite_query", "web_search", "__end__"]:
    """根据检索质量决定下一步"""
    relevance_score = state["relevance_score"]
    has_results = len(state.get("retrieved_docs", [])) > 0

    if not has_results:
        return "web_search"           # 没有结果 → 网络搜索
    elif relevance_score < 0.3:
        return "rewrite_query"        # 相关度低 → 改写查询重试
    elif relevance_score < 0.7:
        return "web_search"           # 相关度中等 → 补充网络搜索
    else:
        return "direct_answer"        # 相关度高 → 直接生成答案

graph.add_conditional_edges("quality_checker", route_by_retrieval_quality)
```

这就是 Self-RAG / Adaptive RAG 的核心思路——根据检索质量动态调整策略。

### 场景4：多路分支 + 优先级路由组合

有时候需要同时使用两种模式：

```python
import operator
from typing import Annotated, Sequence

class AnalysisState(TypedDict):
    document: str
    analysis_types: list[str]
    results: Annotated[list[str], operator.add]

def route_analyses(state: AnalysisState) -> Sequence[str]:
    """根据文档类型决定需要哪些分析（多路分支）"""
    doc = state["document"].lower()
    routes = []

    # 优先级逻辑决定哪些分析需要执行
    if "合同" in doc or "协议" in doc:
        routes.extend(["legal_check", "risk_assessment"])
    if "财务" in doc or "金额" in doc:
        routes.append("financial_review")
    if not routes:
        routes.append("general_review")

    return routes

graph.add_conditional_edges("classifier", route_analyses)
```

## 常见陷阱

### 陷阱1：优先级路由没有默认回退

```python
# 错误：如果所有条件都不满足，返回 None
def bad_route(state):
    if state["type"] == "a":
        return "node_a"
    if state["type"] == "b":
        return "node_b"
    # 如果 type 是 "c"？返回 None → 运行时报错！

# 正确：永远有默认回退
def good_route(state) -> Literal["node_a", "node_b", "default_node"]:
    if state["type"] == "a":
        return "node_a"
    if state["type"] == "b":
        return "node_b"
    return "default_node"  # 默认回退
```

### 陷阱2：多路分支忘记 Reducer

```python
# 错误：并行节点写入同一字段，结果被覆盖
class BadState(TypedDict):
    results: list[str]  # 没有 reducer

# 正确：用 Annotated + operator.add
class GoodState(TypedDict):
    results: Annotated[list[str], operator.add]
```

### 陷阱3：Supervisor 路由映射不完整

```python
# 错误：SPECIALISTS 缺少某个意图的映射
SPECIALISTS = {"search": "search_agent", "code": "code_agent"}

def route(state):
    intent = state["intent"]
    return SPECIALISTS[intent]  # 如果 intent="chat" → KeyError!

# 正确：使用 .get() 提供默认值
def route(state) -> Literal["search_agent", "code_agent", "chat_agent"]:
    intent = state["intent"]
    return SPECIALISTS.get(intent, "chat_agent")  # 默认回退到 chat
```

### 陷阱4：条件链层级过深

```python
# 不推荐：超过 3 级的条件链
# classify → assess_type → assess_priority → assess_region → handler
# 太深了，难以理解和维护

# 推荐：扁平化，用组合条件
def route(state) -> Literal["handler_a", "handler_b", "handler_c", "default"]:
    """一个路由函数处理多个维度"""
    if state["type"] == "refund" and state["priority"] == "high":
        return "handler_a"
    if state["type"] == "complaint":
        return "handler_b"
    return "default"
```

## 总结

| 模式 | 核心思想 | 适用场景 |
|------|---------|---------|
| 多路分支 | 返回列表，同时激活多个节点 | 多维度并行分析 |
| Supervisor 路由 | 中央调度，分发到专家节点 | 多 Agent 系统 |
| 优先级路由 | if-elif-else 链，第一个匹配的生效 | 分级处理、权限控制 |
| 条件链 | 多级条件边串联，逐级细化 | 复杂决策树 |
| 多路 + 优先级组合 | 先决定做哪些分析，再并行执行 | 文档多维度处理 |
