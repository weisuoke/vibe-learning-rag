## 为什么需要 DocumentLoader?

### RAG 应用的第一步

在构建 RAG (Retrieval-Augmented Generation) 应用时,第一步就是从各种数据源加载文档:

```
数据源 (PDF/Word/HTML/...)
    ↓ DocumentLoader
Document 对象 (统一格式)
    ↓ TextSplitter
Document 分块
    ↓ Embedding
向量化
    ↓ VectorStore
存储和检索
```

**没有 DocumentLoader 的问题**:
- 每种格式需要不同的解析库 (PyPDF2, python-docx, BeautifulSoup...)
- 元数据管理混乱
- 无法统一处理大文件的内存问题
- 难以复用解析逻辑

**有了 DocumentLoader**:
- 统一的 `BaseLoader` 接口
- 自动处理懒加载和内存管理
- 标准化的 `Document` 格式
- 可组合的 BlobLoader + BlobParser 架构

### 实际应用场景

**场景 1: 企业知识库**
```python
# 加载公司文档库
from langchain_community.document_loaders import DirectoryLoader

loader = DirectoryLoader(
    "company_docs/",
    glob="**/*.pdf",
    loader_cls=PyPDFLoader
)
docs = loader.load()  # 统一加载所有 PDF
```

**场景 2: 网页内容抓取**
```python
# 加载网页内容
from langchain_community.document_loaders import WebBaseLoader

loader = WebBaseLoader("https://docs.langchain.com")
docs = loader.load()  # 自动解析 HTML
```

**场景 3: 代码仓库分析**
```python
# 加载 GitHub 仓库
from langchain_community.document_loaders import GitLoader

loader = GitLoader(
    repo_path="./langchain",
    file_filter=lambda x: x.endswith(".py")
)
docs = loader.load()  # 加载所有 Python 文件
```

## 核心架构

### 三层抽象设计

LangChain 的 DocumentLoader 采用了清晰的三层抽象:

```
┌─────────────────────────────────────────┐
│           BaseLoader                    │  ← 文档加载器
│  - load() / lazy_load()                 │
│  - aload() / alazy_load()               │
│  - load_and_split()                     │
└─────────────────────────────────────────┘
              ↑
              │ 可以组合
              ↓
┌─────────────────────────────────────────┐
│         BlobLoader                      │  ← 原始数据加载
│  - yield_blobs()                        │
└─────────────────────────────────────────┘
              +
┌─────────────────────────────────────────┐
│       BaseBlobParser                    │  ← 数据解析
│  - lazy_parse(blob)                     │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│      Blob → Document                    │  ← 数据转换
└─────────────────────────────────────────┘
```

**设计优势**:
1. **职责分离**: 加载和解析分开,可以独立复用
2. **懒加载优先**: 避免大文件一次性加载到内存
3. **可组合性**: 同一个 Parser 可以用于不同的 Loader

### 核心数据结构

**Blob - 原始数据**:
```python
class Blob:
    data: bytes | str | None      # 原始数据
    path: PathLike | None          # 文件路径
    mimetype: str | None           # MIME 类型
    encoding: str = "utf-8"        # 编码
    metadata: dict                 # 元数据
```

**Document - 结构化文档**:
```python
class Document:
    page_content: str              # 文本内容
    metadata: dict                 # 元数据
    id: str | None                 # 唯一标识符
```

**关键区别**:
- Blob: 原始数据,可能是二进制,用于加载阶段
- Document: 文本数据,用于检索和处理阶段

## 常见问题

### Q1: DocumentLoader 和直接读文件有什么区别?

**直接读文件**:
```python
with open("file.txt") as f:
    content = f.read()  # 一次性加载到内存
```

**使用 DocumentLoader**:
```python
loader = TextLoader("file.txt")
for doc in loader.lazy_load():  # 流式加载
    process(doc)
```

**优势**:
- 统一的 Document 格式
- 自动元数据管理
- 懒加载支持
- 与 LangChain 生态集成

### Q2: 什么时候用 load() vs lazy_load()?

**load()** - 小文件或需要全部数据:
```python
docs = loader.load()  # 返回 list[Document]
print(len(docs))      # 可以直接获取数量
```

**lazy_load()** - 大文件或流式处理:
```python
for doc in loader.lazy_load():  # 返回 Iterator[Document]
    process(doc)  # 逐个处理,节省内存
```

### Q3: 如何选择合适的 PDF Loader?

| Loader | 优势 | 劣势 | 适用场景 |
|--------|------|------|----------|
| PyPDFLoader | 快速,轻量 | 复杂布局支持差 | 简单文本 PDF |
| PDFPlumberLoader | 表格支持好 | 速度较慢 | 包含表格的 PDF |
| UnstructuredPDFLoader | 布局识别强 | 依赖多,慢 | 复杂布局 PDF |

### Q4: 如何处理编码问题?

```python
# 指定编码
loader = TextLoader("file.txt", encoding="gbk")

# 自动检测编码
from langchain_community.document_loaders import TextLoader
loader = TextLoader("file.txt", autodetect_encoding=True)
```
