# 【30字核心】Tools 与函数调用

> **Tool 是 LangChain 中将函数封装为可被 LLM 调用的标准接口，是 Agent 与外部世界交互的桥梁。**

---

## 关键词解析

| 关键词 | 含义 |
|--------|------|
| **Tool** | LangChain 中对"一个可执行动作"的抽象，包含名称、描述、输入 Schema、执行函数四要素 |
| **函数封装** | 把普通 Python 函数包装成 Tool 对象，让 LLM 能理解"这个函数是干什么的、需要什么参数" |
| **标准接口** | 不管底层是搜索 API、数据库查询还是计算器，对 LLM 来说都是统一的 `name + description + args_schema` |
| **LLM 调用** | LLM 不直接执行代码，而是输出结构化的 `tool_calls`（工具名 + 参数），由框架代为执行 |
| **外部世界交互** | LLM 本身只能生成文本，Tool 赋予它搜索、计算、读写数据库、调用 API 等实际能力 |

---

## 为什么这30个字重要

### LLM 的根本局限

LLM 是一个"纯文本生成器"。它不能上网搜索，不能查数据库，不能执行代码，不能调用任何外部服务。没有 Tool，LLM 就是一个只会说话、不会动手的大脑。

### Tool 解决了什么

Tool 机制解决的是 **LLM 能力边界扩展** 问题：

```
没有 Tool:  用户提问 → LLM 凭记忆回答（可能过时、可能幻觉）
有了 Tool:  用户提问 → LLM 决定调用哪个工具 → 获取真实数据 → 基于事实回答
```

这不是锦上添花，而是 Agent 能否"做事"的前提条件。AgentExecutor 的 Think→Act→Observe 循环中，**Act 这一步就是调用 Tool**。没有 Tool，循环无法运转。

### 标准接口的价值

Tool 的核心设计哲学是**对 LLM 屏蔽实现细节**。LLM 不需要知道搜索是用的 Google 还是 Bing，不需要知道数据库是 MySQL 还是 PostgreSQL。它只需要看到：

```
工具名: search
描述: 搜索互联网获取最新信息
参数: {"query": "搜索关键词"}
```

这种标准化让你可以随时替换底层实现，而不影响 Agent 的决策逻辑。

## 与 RAG/Agent 开发的关系

在 RAG 系统中，Tool 是**检索能力的载体**：

| RAG 场景 | 对应的 Tool |
|----------|------------|
| 向量检索 | `retriever_tool`——封装 VectorStore 的相似度搜索 |
| 混合检索 | 同时绑定向量检索 + 关键词检索两个 Tool |
| 多知识库 | 每个知识库一个 Tool，LLM 自主选择查哪个库 |
| 实时数据 | 搜索引擎 Tool，补充知识库中没有的最新信息 |

**没有 Tool，Agent 就是空转的循环；有了 Tool，Agent 才能真正与世界交互。**
