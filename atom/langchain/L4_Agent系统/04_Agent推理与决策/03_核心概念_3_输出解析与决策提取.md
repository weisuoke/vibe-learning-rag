# 核心概念 3：输出解析与决策提取

> OutputParser 是 Agent 推理循环的"翻译官"——将 LLM 的自由文本输出解析为结构化的 AgentAction 或 AgentFinish 决策

---

## 为什么需要输出解析？

LLM 的输出是**自由文本**，但 Agent 循环需要的是**结构化决策**：

```
LLM 输出（自由文本）：
"Thought: 我需要查询北京的天气
Action: weather_tool
Action Input: 北京"

Agent 循环需要（结构化决策）：
AgentAction(tool="weather_tool", tool_input="北京", log="...")
```

**OutputParser 就是连接这两者的桥梁。** 没有它，Agent 循环就不知道 LLM 想做什么。

**前端类比：** 就像 `JSON.parse()` 把 API 返回的字符串解析为 JavaScript 对象。LLM 的输出是"字符串"，OutputParser 把它解析为"可操作的对象"。

**日常生活类比：** 就像翻译官。LLM 用"自然语言"表达决策，OutputParser 把它翻译成"机器能执行的指令"。

---

## 两种决策类型：AgentAction vs AgentFinish

所有 OutputParser 的输出都是这两种类型之一：

```
┌─────────────────────────────────────────────────┐
│              LLM 输出文本                         │
│                                                  │
│  OutputParser 解析后，只有两种可能：               │
│                                                  │
│  ┌──────────────────┐  ┌──────────────────────┐ │
│  │   AgentAction    │  │    AgentFinish       │ │
│  │                  │  │                      │ │
│  │ tool: str        │  │ return_values: dict  │ │
│  │ tool_input: Any  │  │ log: str             │ │
│  │ log: str         │  │                      │ │
│  │                  │  │ 含义：任务完成，       │ │
│  │ 含义：需要调用   │  │ 返回最终答案          │ │
│  │ 工具，循环继续   │  │ 循环结束              │ │
│  └──────────────────┘  └──────────────────────┘ │
└─────────────────────────────────────────────────┘
```

### AgentAction —— "我还需要更多信息"

```python
from langchain.agents import AgentAction

# 当 LLM 决定调用工具时，解析器返回 AgentAction
action = AgentAction(
    tool="weather_tool",       # 要调用的工具名
    tool_input="北京",          # 传给工具的参数
    log="Thought: 我需要查询北京天气\nAction: weather_tool\nAction Input: 北京"
)

# AgentExecutor 收到 AgentAction 后：
# 1. 找到名为 "weather_tool" 的工具
# 2. 调用 weather_tool.invoke("北京")
# 3. 将结果作为 Observation 追加到 scratchpad
# 4. 继续下一轮循环
```

### AgentFinish —— "我已经有答案了"

```python
from langchain.agents import AgentFinish

# 当 LLM 认为可以回答时，解析器返回 AgentFinish
finish = AgentFinish(
    return_values={"output": "北京今天5°C，多云，建议穿厚外套"},
    log="Thought: 我已经有了天气信息，可以回答了\nFinal Answer: 北京今天5°C..."
)

# AgentExecutor 收到 AgentFinish 后：
# 1. 停止循环
# 2. 返回 return_values["output"] 作为最终结果
```

### 关键理解

**Agent 循环的本质就是不断问 LLM "你想做什么"，然后用 OutputParser 把回答翻译成 AgentAction 或 AgentFinish。** 翻译成 Action 就继续循环，翻译成 Finish 就停止。

---

## 5 种 OutputParser 详解

LangChain 提供 5 种 OutputParser，适用于不同的推理策略：

```
┌─────────────────────────────────────────────────────────┐
│                   5 种 OutputParser                       │
│                                                          │
│  文本解析类（从 LLM 文本输出中提取决策）                    │
│  ├─ 1. ReActSingleInputOutputParser  （正则匹配）         │
│  ├─ 2. ReActJsonSingleInputOutputParser（JSON 提取）      │
│  ├─ 3. XMLAgentOutputParser          （XML 标签）         │
│  └─ 4. MRKLOutputParser             （继承 ReAct）        │
│                                                          │
│  结构化解析类（从模型原生结构中提取决策）                    │
│  └─ 5. ToolsAgentOutputParser        （tool_calls 提取）  │
└─────────────────────────────────────────────────────────┘
```

---

### 解析器 1: ReActSingleInputOutputParser

**适用场景：** 经典 ReAct 文本推理，LLM 输出 `Action:` 和 `Action Input:` 格式

**解析原理：**

```python
# LLM 输出示例：
llm_output = """Thought: 我需要搜索北京的天气信息
Action: weather_tool
Action Input: 北京"""

# 解析器的工作流程：
# 1. 先检查是否包含 "Final Answer:"
#    → 如果有，返回 AgentFinish
# 2. 用正则匹配 "Action:" 和 "Action Input:"
#    → 如果匹配成功，返回 AgentAction
# 3. 如果都不匹配，抛出 OutputParserException
```

**源码核心逻辑（简化版）：**

```python
import re
from langchain.agents import AgentAction, AgentFinish
from langchain.agents.output_parsers import ReActSingleInputOutputParser

class ReActSingleInputOutputParser:
    """从 ReAct 格式文本中提取 Action 和 Action Input"""

    def parse(self, text: str):
        # 第一步：检查是否是最终答案
        if "Final Answer:" in text:
            answer = text.split("Final Answer:")[-1].strip()
            return AgentFinish(
                return_values={"output": answer},
                log=text
            )

        # 第二步：正则匹配 Action 和 Action Input
        # 关键正则：匹配 "Action: xxx" 和 "Action Input: xxx"
        action_match = re.search(
            r"Action\s*:\s*(.+?)\s*\n"    # 匹配工具名
            r"Action\s*Input\s*:\s*(.+)",  # 匹配工具输入
            text,
            re.DOTALL
        )

        if action_match:
            tool = action_match.group(1).strip()
            tool_input = action_match.group(2).strip()
            return AgentAction(
                tool=tool,
                tool_input=tool_input,
                log=text
            )

        # 第三步：解析失败，抛出异常
        raise OutputParserException(
            f"Could not parse LLM output: {text}",
            send_to_llm=True  # 关键！让 LLM 自我修正
        )
```

**关键细节：**
- 优先检查 `Final Answer:`，确保不会把最终答案误解析为 Action
- 正则使用 `re.DOTALL` 支持多行匹配
- `send_to_llm=True` 实现自我修正（后面详解）

### 解析器 2: ReActJsonSingleInputOutputParser

**适用场景：** JSON 格式的 ReAct 推理，LLM 在 markdown 代码块中输出 JSON

**解析原理：**

```python
# LLM 输出示例：
llm_output = """Thought: 我需要搜索北京的天气
```json
{
    "action": "weather_tool",
    "action_input": "北京"
}
```"""

# 解析器的工作流程：
# 1. 先检查 "Final Answer:" → AgentFinish
# 2. 从 markdown 代码块中提取 JSON
# 3. 解析 JSON 中的 "action" 和 "action_input"
# 4. 返回 AgentAction
```

**源码核心逻辑（简化版）：**

```python
import json
import re

class ReActJsonSingleInputOutputParser:
    """从 JSON 代码块中提取 action 和 action_input"""

    def parse(self, text: str):
        # 检查最终答案
        if "Final Answer:" in text:
            return AgentFinish(
                return_values={"output": text.split("Final Answer:")[-1].strip()},
                log=text
            )

        # 从 markdown 代码块提取 JSON
        # 匹配 ```json ... ``` 或 ``` ... ```
        json_match = re.search(r"```(?:json)?\s*(.*?)```", text, re.DOTALL)
        if json_match:
            json_str = json_match.group(1).strip()
            data = json.loads(json_str)
            return AgentAction(
                tool=data["action"],
                tool_input=data.get("action_input", ""),
                log=text
            )

        raise OutputParserException(
            f"Could not parse JSON from: {text}",
            send_to_llm=True
        )
```

**与 ReAct 解析器的区别：**

| 维度 | ReAct 正则解析 | ReAct JSON 解析 |
|------|---------------|----------------|
| 输入格式 | `Action: xxx\nAction Input: xxx` | `{"action": "xxx", "action_input": "xxx"}` |
| 解析方式 | 正则表达式 | JSON.parse |
| 鲁棒性 | 对格式敏感，空格/换行可能导致失败 | JSON 格式更严格但更可靠 |
| 适用模型 | 所有模型 | 擅长生成 JSON 的模型（GPT-4、Claude） |

### 解析器 3: ToolsAgentOutputParser

**适用场景：** 使用模型原生函数调用（tool_calls），这是 2025-2026 的主流方式

**这是最可靠的解析器**，因为它不需要从文本中"猜"决策，而是直接从模型的结构化输出中提取。

**解析原理：**

```python
# 模型输出（不是文本，而是结构化的 AIMessage）：
from langchain_core.messages import AIMessage

ai_message = AIMessage(
    content="",  # 内容为空！决策在 tool_calls 中
    tool_calls=[
        {
            "name": "weather_tool",
            "args": {"city": "北京"},
            "id": "call_abc123"
        }
    ]
)

# 解析器的工作流程：
# 1. 检查 AIMessage.tool_calls 是否为空
#    → 为空：返回 AgentFinish（模型认为不需要调用工具）
#    → 不为空：返回 AgentAction（或多个 AgentAction）
```

**源码核心逻辑（简化版）：**

```python
from langchain_core.messages import AIMessage
from langchain.agents import AgentAction, AgentFinish

class ToolsAgentOutputParser:
    """从 AIMessage.tool_calls 直接提取决策"""

    def parse_result(self, result):
        # result 是 AIMessage 列表
        ai_message = result[0]

        # 没有 tool_calls → 任务完成
        if not ai_message.tool_calls:
            return AgentFinish(
                return_values={"output": ai_message.content},
                log=str(ai_message)
            )

        # 有 tool_calls → 提取所有工具调用
        actions = []
        for tool_call in ai_message.tool_calls:
            actions.append(AgentAction(
                tool=tool_call["name"],
                tool_input=tool_call["args"],
                log=str(ai_message)
            ))

        # 支持多工具并行！
        # 如果只有一个 tool_call → 返回单个 AgentAction
        # 如果有多个 tool_calls → 返回 List[AgentAction]
        if len(actions) == 1:
            return actions[0]
        return actions  # MultiAction！
```

**为什么这是最可靠的解析器？**

| 维度 | 文本解析（ReAct） | 结构化解析（ToolsAgent） |
|------|-------------------|------------------------|
| 解析方式 | 正则/JSON 从文本提取 | 直接读取 tool_calls 字段 |
| 失败率 | 较高（格式不对就失败） | 极低（模型原生支持） |
| 多工具支持 | 不支持（单次单工具） | 支持（多个 tool_calls 并行） |
| 自我修正 | 需要 send_to_llm 机制 | 几乎不需要 |
| 适用模型 | 所有模型 | 支持函数调用的模型（GPT-4、Claude 等） |

### 解析器 4: XMLAgentOutputParser

**适用场景：** 使用 XML 标签格式的推理，特别适合 Claude 模型

**解析原理：**

```python
# LLM 输出示例（Claude 风格）：
llm_output = """我需要查询北京的天气。

<tool>weather_tool</tool>
<tool_input>北京</tool_input>"""

# 或者最终答案：
llm_output_final = """根据查询结果，北京今天5°C。

<final_answer>北京今天5°C，多云，建议穿厚外套。</final_answer>"""

# 解析器的工作流程：
# 1. 检查是否包含 <final_answer> 标签 → AgentFinish
# 2. 检查是否包含 <tool> 和 <tool_input> 标签 → AgentAction
```

**源码核心逻辑（简化版）：**

```python
import re

class XMLAgentOutputParser:
    """从 XML 标签中提取决策"""

    def parse(self, text: str):
        # 检查最终答案
        final_match = re.search(
            r"<final_answer>(.*?)</final_answer>",
            text, re.DOTALL
        )
        if final_match:
            return AgentFinish(
                return_values={"output": final_match.group(1).strip()},
                log=text
            )

        # 提取工具调用
        tool_match = re.search(r"<tool>(.*?)</tool>", text, re.DOTALL)
        input_match = re.search(r"<tool_input>(.*?)</tool_input>", text, re.DOTALL)

        if tool_match:
            return AgentAction(
                tool=tool_match.group(1).strip(),
                tool_input=input_match.group(1).strip() if input_match else "",
                log=text
            )

        raise OutputParserException(
            f"Could not parse XML output: {text}",
            send_to_llm=True
        )
```

**为什么 Claude 适合 XML 格式？**

Claude 在训练数据中大量接触了 XML 格式，对 XML 标签的生成非常稳定。相比正则匹配 `Action:` 这种纯文本格式，XML 标签有明确的开始和结束标记，解析更可靠。

### 解析器 5: MRKLOutputParser

**适用场景：** MRKL（Modular Reasoning, Knowledge and Language）系统

**关键理解：MRKLOutputParser 直接继承自 ReActSingleInputOutputParser，解析逻辑完全相同。**

```python
# 源码（真的就这么简单）：
class MRKLOutputParser(ReActSingleInputOutputParser):
    """MRKL 输出解析器，继承 ReAct 解析器"""
    pass  # 没有任何额外逻辑！
```

**那 MRKL 和 ReAct 的区别在哪？**

区别不在解析器，而在 **Prompt**：

| 维度 | ReAct | MRKL |
|------|-------|------|
| Prompt 结构 | 包含 few-shot 推理示例 | prefix + tools + format_instructions + suffix |
| 推理风格 | 详细的 Thought 推理过程 | 更简洁，侧重工具选择 |
| 工具描述 | 嵌入在示例中 | 独立的 tools 段落 |
| 解析器 | ReActSingleInputOutputParser | MRKLOutputParser（继承前者） |

## 自我修正机制：send_to_llm=True

这是所有文本解析器共享的**容错机制**。当解析失败时，不是直接报错，而是把错误信息发回给 LLM，让它重新生成格式正确的输出。

### 工作流程

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ LLM 输出 │───→│ OutputParser │───→│ 解析成功？    │
│ 自由文本 │    │ 尝试解析     │    │              │
└──────────┘    └──────────────┘    └──────┬───────┘
                                          │
                                    ┌─────┴─────┐
                                    │           │
                                   是          否
                                    │           │
                                    ▼           ▼
                              返回决策    send_to_llm=True?
                              (Action     ┌─────┴─────┐
                              或 Finish)  │           │
                                         是          否
                                          │           │
                                          ▼           ▼
                                    把错误信息    直接抛出
                                    发回 LLM     异常
                                    让它重试
```

### 代码示例

```python
from langchain_core.exceptions import OutputParserException

# 当解析失败时，抛出带 send_to_llm=True 的异常
raise OutputParserException(
    # 这段文字会被发送给 LLM，作为"纠正提示"
    f"Could not parse LLM output: `{text}`",
    observation="Invalid Format: Missing 'Action:' or 'Final Answer:'."
                " Please use the correct format.",
    llm_output=text,
    send_to_llm=True  # 关键标志！
)

# AgentExecutor 捕获这个异常后：
# 1. 将 observation 作为新的 Observation 追加到 scratchpad
# 2. 让 LLM 重新生成输出
# 3. 再次尝试解析
# 这就实现了"自我修正"
```

### 自我修正的实际效果

```
第 1 轮：
LLM: "我觉得应该查一下天气"  ← 没有按格式输出
Parser: 解析失败！发送纠正提示给 LLM
Observation: "Invalid Format: Missing 'Action:'. Please use the correct format."

第 2 轮：
LLM: "Thought: 我需要查天气\nAction: weather_tool\nAction Input: 北京"  ← 格式正确了
Parser: 解析成功！返回 AgentAction
```

## 解析器选型指南

### 决策流程图

```
你的模型支持原生函数调用（tool_calls）？
├── 是 → 使用 ToolsAgentOutputParser（最可靠，推荐）
└── 否 → 你的模型是 Claude？
    ├── 是 → 使用 XMLAgentOutputParser（Claude 对 XML 最稳定）
    └── 否 → 你需要 JSON 格式的输出？
        ├── 是 → 使用 ReActJsonSingleInputOutputParser
        └── 否 → 使用 ReActSingleInputOutputParser（经典通用）
```

### 对比总结表

| 解析器 | 输入格式 | 解析方式 | 多工具 | 可靠性 | 推荐场景 |
|--------|----------|----------|--------|--------|----------|
| ReActSingleInput | `Action: xxx` | 正则 | 否 | 中 | 经典 ReAct |
| ReActJsonSingleInput | `{"action": "xxx"}` | JSON | 否 | 中高 | JSON 格式推理 |
| ToolsAgent | `tool_calls` 字段 | 直接提取 | 是 | 最高 | 现代主流 |
| XMLAgent | `<tool>xxx</tool>` | XML 标签 | 否 | 高 | Claude 模型 |
| MRKL | `Action: xxx` | 正则 | 否 | 中 | MRKL 系统 |

### 2025-2026 趋势

**ToolsAgentOutputParser 正在成为事实标准。** 原因：
1. 主流模型（GPT-4、Claude、Gemini）都支持原生函数调用
2. 不需要文本格式约束，解析失败率极低
3. 支持多工具并行调用，效率更高
4. LangGraph 的 `create_react_agent` 默认使用这种方式

但 ReAct 文本解析器在以下场景仍有价值：
- 需要**显式推理过程**（可解释性要求高）
- 使用**不支持函数调用的模型**（开源小模型）
- **调试和教学**场景（能看到完整的 Thought 过程）

## 实战：手动使用 OutputParser

```python
from langchain.agents.output_parsers import ReActSingleInputOutputParser

parser = ReActSingleInputOutputParser()

# 测试 1：解析 AgentAction
text1 = """Thought: 我需要查询天气
Action: weather_tool
Action Input: 北京"""

result1 = parser.parse(text1)
print(type(result1))       # AgentAction
print(result1.tool)        # "weather_tool"
print(result1.tool_input)  # "北京"

# 测试 2：解析 AgentFinish
text2 = """Thought: 我已经有了所有信息
Final Answer: 北京今天5°C，建议穿厚外套"""

result2 = parser.parse(text2)
print(type(result2))                    # AgentFinish
print(result2.return_values["output"])  # "北京今天5°C，建议穿厚外套"

# 测试 3：解析失败 → 自我修正
text3 = "我觉得应该查一下天气"  # 没有按格式输出

try:
    parser.parse(text3)
except Exception as e:
    print(f"解析失败: {e}")
    print(f"send_to_llm: {e.send_to_llm}")  # True
```

## 常见陷阱

### 1. Final Answer 优先级

ReAct 解析器**优先检查 Final Answer**。如果 LLM 输出中同时包含 `Action:` 和 `Final Answer:`，会返回 AgentFinish 而非 AgentAction。

```python
# 这段输出会被解析为 AgentFinish，而不是 AgentAction！
text = """Thought: 让我查一下天气
Action: weather_tool
Action Input: 北京
Final Answer: 我不确定，但大概5度吧"""
# → AgentFinish（因为 Final Answer 优先）
```

### 2. Stop Sequence 的重要性

ReAct Agent 会绑定 `stop=["\nObservation"]`，防止 LLM 自己"编造" Observation：

```python
# 没有 stop sequence 时，LLM 可能输出：
"""Thought: 我需要查天气
Action: weather_tool
Action Input: 北京
Observation: 北京今天25°C，晴天"""  ← LLM 自己编的！不是真实工具返回

# 有 stop sequence 时，LLM 输出到 "Observation" 就停止：
"""Thought: 我需要查天气
Action: weather_tool
Action Input: 北京"""  ← 到这里就停了，等待真实工具返回
```

### 3. ToolsAgentOutputParser 的空 content

使用函数调用时，AIMessage 的 `content` 通常为空字符串，决策信息全在 `tool_calls` 中。不要试图从 `content` 中提取信息。

## 小结

| 要点 | 内容 |
|------|------|
| 核心作用 | 将 LLM 自由文本输出解析为 AgentAction 或 AgentFinish |
| 5 种解析器 | ReAct正则、ReActJSON、ToolsAgent、XML、MRKL |
| 最推荐 | ToolsAgentOutputParser（原生函数调用，最可靠） |
| 容错机制 | send_to_llm=True 实现自我修正 |
| 关键区分 | AgentAction = 继续循环，AgentFinish = 停止循环 |
