## 4.1 定义和实现 Trait

### 最简语法

```rust
// ===== Rust =====
// 第一步：定义 Trait（声明"能做什么"）
trait Greeter {
    fn greet(&self) -> String;
}

// 第二步：为具体类型实现 Trait（承诺"怎么做"）
struct EnglishGreeter;
impl Greeter for EnglishGreeter {
    fn greet(&self) -> String {
        "Hello!".to_string()
    }
}

struct ChineseGreeter;
impl Greeter for ChineseGreeter {
    fn greet(&self) -> String {
        "你好！".to_string()
    }
}

fn main() {
    let en = EnglishGreeter;
    let cn = ChineseGreeter;
    println!("{}", en.greet()); // Hello!
    println!("{}", cn.greet()); // 你好！
}
```

```typescript
// ===== TypeScript 对照 =====
interface Greeter {
    greet(): string;
}

class EnglishGreeter implements Greeter {
    greet(): string { return "Hello!"; }
}

class ChineseGreeter implements Greeter {
    greet(): string { return "你好！"; }
}
```

### 默认实现

Trait 方法可以有默认实现，实现者可以选择覆盖或直接使用：

```rust
trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;

    // 有默认实现——大多数实现者不需要覆盖
    fn spec(&self) -> ToolSpec {
        ToolSpec {
            name: self.name().to_string(),
            description: self.description().to_string(),
        }
    }
}
```

```typescript
// TypeScript 没有 interface 默认实现，需要用 abstract class
abstract class Tool {
    abstract name(): string;
    abstract description(): string;

    // 默认实现
    spec(): ToolSpec {
        return { name: this.name(), description: this.description() };
    }
}
```

### 记住这个公式

```
Rust:       trait Foo { ... }     +  impl Foo for Bar { ... }
TypeScript: interface Foo { ... } +  class Bar implements Foo { ... }
```

**唯一关键区别**：Rust 的 trait 可以有默认实现，TypeScript 的 interface 不行（需要 abstract class）。

## 4.2 泛型函数的基本写法

### 基本模式

```rust
// ===== Rust =====
use std::fmt::Display;

// <T: Display> 意思是：T 可以是任何类型，只要它实现了 Display
fn print_item<T: Display>(item: T) {
    println!("Item: {item}");
}

fn main() {
    print_item(42);        // T = i32
    print_item("hello");   // T = &str
    print_item(3.14);      // T = f64
}
```

```typescript
// ===== TypeScript 对照 =====
// <T extends Displayable> 几乎等价
function printItem<T extends { toString(): string }>(item: T): void {
    console.log(`Item: ${item}`);
}
```

**核心映射**：`<T: SomeTrait>` = `<T extends SomeInterface>`

### `impl Trait` 语法糖（ZeroClaw 最常见用法）

在 ZeroClaw 源码中，你会频繁看到 `impl Into<String>` 这种写法：

```rust
// ===== Rust =====
// 这两种写法完全等价：
fn send_message_v1<T: Into<String>>(content: T) { /* ... */ }
fn send_message_v2(content: impl Into<String>) { /* ... */ }

// 调用时可以传 &str 或 String，都能自动转换
fn main() {
    send_message_v2("hello");              // &str → String
    send_message_v2(String::from("hi"));   // String 直接用
}
```

**ZeroClaw 实例**（`src/channels/traits.rs`）：

```rust
// SendMessage 的构造函数
pub fn new(content: impl Into<String>, recipient: impl Into<String>) -> Self {
    Self {
        content: content.into(),
        recipient: recipient.into(),
    }
}
```

**记住**：看到 `impl Into<String>` 就理解为"接受任何能转成 String 的东西"——等价于 TypeScript 函数参数写 `string` 但同时接受能隐式转换的类型。

### 多重约束

```rust
// T 必须同时满足 Display 和 Clone
fn process<T: Display + Clone>(item: T) {
    let copy = item.clone();
    println!("{item}, {copy}");
}
```

```typescript
// TypeScript 用交叉类型
function process<T extends Displayable & Cloneable>(item: T): void { /* ... */ }
```

**核心映射**：`T: A + B` = `T extends A & B`

## 4.3 读懂 where 子句

### 核心规则：where 只是把约束搬到后面

这两种写法**完全等价**：

```rust
// 写法 1：内联约束（约束少时用）
fn foo<T: Display + Clone, U: Debug>(t: T, u: U) { /* ... */ }

// 写法 2：where 子句（约束多时用，更易读）
fn foo<T, U>(t: T, u: U)
where
    T: Display + Clone,
    U: Debug,
{ /* ... */ }
```

**看到 where 不要慌**，它只是为了可读性把约束从 `<>` 里搬到了函数签名后面。

### 逐行阅读技巧

遇到复杂的 where 子句，按这个步骤读：

1. 先看函数名和参数，忽略 where
2. 再逐行读 where，每行一个约束
3. 把约束翻译成中文

### ZeroClaw 实例解读

```rust
// 源码：src/memory/mod.rs
fn create_memory_with_builders<F, G>(
    backend_name: &str,
    workspace_dir: &Path,
    mut sqlite_builder: F,
    mut postgres_builder: G,
    unknown_context: &str,
) -> anyhow::Result<Box<dyn Memory>>
where
    F: FnMut() -> anyhow::Result<SqliteMemory>,
    G: FnMut() -> anyhow::Result<Box<dyn Memory>>,
```

逐行翻译：

```
fn create_memory_with_builders<F, G>(  ← 函数名，两个泛型参数 F 和 G
    backend_name: &str,                ← 后端名称（字符串引用）
    workspace_dir: &Path,              ← 工作目录路径
    mut sqlite_builder: F,             ← SQLite 构建器（类型 F）
    mut postgres_builder: G,           ← Postgres 构建器（类型 G）
    unknown_context: &str,             ← 错误上下文信息
) -> anyhow::Result<Box<dyn Memory>>  ← 返回一个 Memory trait object
where
    F: FnMut() -> anyhow::Result<SqliteMemory>,  ← F 是一个闭包，调用后返回 SqliteMemory
    G: FnMut() -> anyhow::Result<Box<dyn Memory>>, ← G 是一个闭包，调用后返回 Memory trait object
```

用 TypeScript 理解：

```typescript
// 等价的 TypeScript 写法
function createMemoryWithBuilders(
    backendName: string,
    workspaceDir: string,
    sqliteBuilder: () => SqliteMemory,    // F
    postgresBuilder: () => Memory,         // G
    unknownContext: string,
): Memory { /* ... */ }
```

**关键洞察**：`FnMut()` 就是 TypeScript 的 `() =>`（箭头函数/闭包）。`FnMut() -> Result<T>` 就是 `() => T`（可能抛错的版本）。

## 4.4 理解 Send + Sync 约束

### 一句话理解

看到 `trait Foo: Send + Sync`，意思是：**实现这个 Trait 的类型必须能安全地跨线程使用**。

### ZeroClaw 的所有核心 Trait 都有这个约束

```rust
// 四大核心 Trait 全部要求 Send + Sync
pub trait Provider: Send + Sync { /* ... */ }
pub trait Channel:  Send + Sync { /* ... */ }
pub trait Tool:     Send + Sync { /* ... */ }
pub trait Memory:   Send + Sync { /* ... */ }
```

**为什么？** 因为 ZeroClaw 是异步多任务的——一个 Agent 可能同时处理多个对话、调用多个工具。所有核心组件必须能安全地在多个异步任务间共享。

### 前端类比

```typescript
// TypeScript 中没有 Send/Sync 的概念，因为 JS 是单线程的
// 最接近的类比是 Web Worker 的 postMessage：
// - 不是所有对象都能 postMessage（不是所有类型都 Send）
// - 只有 Transferable 和可序列化的对象才行

worker.postMessage(data);  // data 必须是可转移的 ≈ Send
```

### 99% 的情况你不需要特别处理

Rust 编译器会**自动推导** Send 和 Sync。只要你的 struct 的所有字段都是 Send + Sync 的，你的 struct 就自动是 Send + Sync 的。

```rust
// 这个 struct 自动满足 Send + Sync，因为 String 和 f64 都满足
struct MyTool {
    name: String,
    temperature: f64,
}

// 直接 impl Tool 就行，不需要额外处理 Send + Sync
#[async_trait]
impl Tool for MyTool {
    fn name(&self) -> &str { &self.name }
    // ...
}
```

**什么时候会出问题？** 只有当你使用了 `Rc`（非线程安全引用计数）或裸指针时。解决方案：用 `Arc` 替代 `Rc`，用 `Mutex` 包裹可变状态。

```rust
// ❌ 编译错误：Rc 不是 Send
use std::rc::Rc;
struct BadTool { data: Rc<String> }

// ✅ 正确：Arc 是 Send + Sync
use std::sync::Arc;
struct GoodTool { data: Arc<String> }
```

## 4.5 识别 #[async_trait] 模式

### 为什么需要它？

Rust 原生的 trait 对 `async fn` 的支持还不完整（特别是与 `dyn Trait` 配合时）。`#[async_trait]` 宏是一个"补丁"，让 trait 中可以定义异步方法。

### 识别模式

看到 `#[async_trait]` 出现在两个地方：

```rust
use async_trait::async_trait;

// 1. Trait 定义上
#[async_trait]
pub trait Provider: Send + Sync {
    async fn chat(&self, request: ChatRequest<'_>) -> anyhow::Result<ChatResponse>;
    // ...
}

// 2. Trait 实现上（也要加！）
#[async_trait]
impl Provider for OpenAIProvider {
    async fn chat(&self, request: ChatRequest<'_>) -> anyhow::Result<ChatResponse> {
        // 可以用 .await
        let response = self.client.post(&self.url).send().await?;
        // ...
    }
}
```

### 前端类比

```typescript
// 类似于早期 JS 需要 Promise polyfill
// 浏览器原生不支持 Promise 时：
import 'promise-polyfill';

// Rust 原生 async trait 支持不完整时：
use async_trait::async_trait;
```

### 三条规则

1. **Trait 定义**加了 `#[async_trait]` → **实现也必须加**
2. 加了 `#[async_trait]` 的 trait 里可以写 `async fn`
3. `async fn` 的方法体里可以用 `.await`

### ZeroClaw 中的使用频率

```
#[async_trait] 的 Trait：Provider, Channel, Tool, Memory, HookHandler, Sandbox
非 async 的 Trait：     RuntimeAdapter, Observer, ChannelConfig, ConfigHandle
```

**判断标准**：如果 Trait 的方法需要做 I/O（网络请求、数据库查询、文件读写），就需要 `#[async_trait]`。

## 速查卡片（一页纸总结）

```
┌─────────────────────────────────────────────────────────────┐
│                  Trait 与泛型速查卡片                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  定义 Trait:     trait Foo { fn bar(&self); }               │
│  实现 Trait:     impl Foo for MyType { fn bar(&self) {...} }│
│  默认实现:       trait Foo { fn bar(&self) { /* 默认 */ } } │
│                                                             │
│  泛型约束:       fn foo<T: Display>(x: T)                   │
│  语法糖:         fn foo(x: impl Display)     ← 等价        │
│  多重约束:       fn foo<T: Display + Clone>(x: T)           │
│  where 子句:     fn foo<T>(x: T) where T: Display + Clone   │
│                                                             │
│  Send + Sync:    trait Foo: Send + Sync { ... }             │
│                  → "实现者必须线程安全"                        │
│                  → 99% 自动满足，不用管                       │
│                                                             │
│  #[async_trait]: 定义和实现都要加                              │
│                  → 方法里可以用 async fn + .await             │
│                                                             │
│  impl Into<T>:   fn foo(s: impl Into<String>)               │
│                  → "接受任何能转成 String 的东西"              │
│                                                             │
│  TS 映射表:                                                  │
│    trait         →  interface                                │
│    impl for      →  implements                               │
│    <T: Foo>      →  <T extends Foo>                          │
│    T: A + B      →  T extends A & B                          │
│    impl Into<S>  →  参数类型 string（自动转换）               │
│    where         →  （TS 没有，约束直接写在 <> 里）           │
│    Send + Sync   →  （TS 没有，JS 单线程不需要）              │
│    #[async_trait] →  （TS 原生支持 async，不需要）            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
