# 实战代码 场景3：性能优化

> LangChain 管道性能优化实战：流式输出、批处理、缓存、并行执行与成本控制

---

## 场景描述

RAG 系统上线后，性能是绕不开的话题。用户等 5 秒没响应就关页面，批量处理 1000 条文档跑一晚上，相同问题反复调 API 烧钱——这些都是真实痛点。

本场景聚焦 LangChain 管道的 6 大性能优化技巧：

| 优化方向 | 解决的问题 | 核心手段 |
|----------|-----------|----------|
| 流式输出 | 用户等待焦虑 | `stream()` / `astream()` |
| 流式后处理 | 自定义处理打断流式 | `RunnableGenerator` |
| 批处理 | 大量任务吞吐低 | `batch()` + `max_concurrency` |
| 缓存 | 重复查询浪费 API 调用 | `InMemoryCache` / `SQLiteCache` |
| 并行执行 | 独立子任务串行慢 | `RunnableParallel` |
| 成本控制 | API 费用过高 | 模型分级 + `RunnableBranch` |

## 完整可运行代码

```python
"""
性能优化实战
演示：流式输出、批处理、缓存策略、并行执行、成本优化
"""

import os
import time
import asyncio
from dotenv import load_dotenv
load_dotenv()

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import (
    RunnableParallel,
    RunnablePassthrough,
    RunnableLambda,
    RunnableGenerator,
)

# ===== 1. 流式输出优化 =====
print("=== 1. 流式输出优化 ===")

prompt = ChatPromptTemplate.from_template("写一首关于{topic}的短诗（4行）")
model = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)
parser = StrOutputParser()

chain = prompt | model | parser

# 测量首字延迟
start = time.time()
first_chunk = True
for chunk in chain.stream({"topic": "春天"}):
    if first_chunk:
        print(f"首字延迟: {time.time() - start:.2f}s")
        first_chunk = False
    print(chunk, end="", flush=True)
total_time = time.time() - start
print(f"\n总耗时: {total_time:.2f}s")


# ===== 2. RunnableGenerator 保持流式 =====
print("\n\n=== 2. RunnableGenerator 保持流式 ===")

# ❌ RunnableLambda 会阻断流式
# 它会等上游全部输出完毕，再一次性处理，用户看到的是"卡住→突然全出来"
# chain = prompt | model | RunnableLambda(lambda x: x.content.upper())

# ✅ RunnableGenerator 保持流式
# 它是一个生成器函数，逐 chunk 接收、逐 chunk 输出
def add_prefix_stream(input_stream):
    """流式添加前缀——第一个 chunk 前插入 emoji"""
    yield "🌸 "
    for chunk in input_stream:
        yield chunk

streaming_chain = (
    prompt
    | model
    | StrOutputParser()
    | RunnableGenerator(add_prefix_stream)
)

for chunk in streaming_chain.stream({"topic": "秋天"}):
    print(chunk, end="", flush=True)
print()


# ===== 3. 批处理优化 =====
print("\n=== 3. 批处理优化 ===")

topics = [{"topic": t} for t in ["春天", "夏天", "秋天", "冬天"]]

# 方式1：逐个调用（慢）
start = time.time()
results_sequential = [chain.invoke(t) for t in topics]
sequential_time = time.time() - start
print(f"逐个调用: {sequential_time:.2f}s")

# 方式2：batch 并行（快）
# max_concurrency 控制最大并发数，避免触发 API 限流
start = time.time()
results_batch = chain.batch(topics, config={"max_concurrency": 4})
batch_time = time.time() - start
print(f"batch 并行: {batch_time:.2f}s")
print(f"加速比: {sequential_time / batch_time:.1f}x")

# 方式3：异步批处理（适合已有 async 上下文的场景）
async def async_batch():
    start = time.time()
    results = await chain.abatch(topics, config={"max_concurrency": 4})
    elapsed = time.time() - start
    print(f"async batch: {elapsed:.2f}s")
    return results

# 在 async 环境中使用：
# asyncio.run(async_batch())


# ===== 4. 缓存策略 =====
print("\n=== 4. 缓存策略 ===")

from langchain.cache import InMemoryCache
from langchain.globals import set_llm_cache

# 启用内存缓存
set_llm_cache(InMemoryCache())

# 第一次调用（缓存 miss）
start = time.time()
result1 = chain.invoke({"topic": "月亮"})
time1 = time.time() - start
print(f"第1次调用: {time1:.2f}s")

# 第二次调用（缓存 hit）——相同输入直接返回缓存结果，不调 API
start = time.time()
result2 = chain.invoke({"topic": "月亮"})
time2 = time.time() - start
print(f"第2次调用: {time2:.4f}s (缓存命中)")
print(f"加速比: {time1 / max(time2, 0.001):.0f}x")
print(f"结果一致: {result1 == result2}")

# 用完记得清除，避免内存泄漏
set_llm_cache(None)


# ===== 5. 并行执行优化 =====
print("\n=== 5. 并行执行优化 ===")

# 同时从多个角度分析同一个主题
# RunnableParallel 会并发执行所有分支，总耗时 ≈ 最慢的那个分支
analysis_chain = RunnableParallel(
    summary=(
        ChatPromptTemplate.from_template("用一句话总结{topic}")
        | model
        | parser
    ),
    pros=(
        ChatPromptTemplate.from_template("列出{topic}的3个优点")
        | model
        | parser
    ),
    cons=(
        ChatPromptTemplate.from_template("列出{topic}的3个缺点")
        | model
        | parser
    ),
)

start = time.time()
result = analysis_chain.invoke({"topic": "远程办公"})
parallel_time = time.time() - start
print(f"并行分析耗时: {parallel_time:.2f}s")
print(f"总结: {result['summary'][:80]}...")
print(f"优点: {result['pros'][:80]}...")
print(f"缺点: {result['cons'][:80]}...")


# ===== 6. 成本优化：模型分级 =====
print("\n=== 6. 成本优化：模型分级 ===")

from langchain_core.runnables import RunnableBranch

# 核心思路：简单问题用便宜模型，复杂问题用强模型
# gpt-4o-mini 价格约为 gpt-4o 的 1/30，大部分简单问题它就够了

def classify_complexity(question: str) -> dict:
    """简单的复杂度分类（生产环境可以用 LLM 做分类）"""
    complex_keywords = ["分析", "比较", "设计", "架构", "优化"]
    is_complex = any(kw in question for kw in complex_keywords)
    return {
        "question": question,
        "complexity": "complex" if is_complex else "simple",
    }

simple_chain = (
    ChatPromptTemplate.from_template("简洁回答：{question}")
    | ChatOpenAI(model="gpt-4o-mini")  # 便宜
    | StrOutputParser()
)

complex_chain = (
    ChatPromptTemplate.from_template("详细分析：{question}")
    | ChatOpenAI(model="gpt-4o")  # 更强
    | StrOutputParser()
)

cost_optimized_chain = (
    RunnableLambda(classify_complexity)
    | RunnableBranch(
        (lambda x: x["complexity"] == "simple", simple_chain),
        complex_chain,  # 默认分支
    )
)

# 简单问题 → 路由到 gpt-4o-mini
result = cost_optimized_chain.invoke("什么是Python？")
print(f"简单问题: {result[:80]}...")

# 复杂问题 → 路由到 gpt-4o
result = cost_optimized_chain.invoke("分析微服务架构的优缺点")
print(f"复杂问题: {result[:80]}...")

print("\n✅ 性能优化演示完成")
```

## 逐段解析

### 1. 流式输出——降低体感延迟

```python
for chunk in chain.stream({"topic": "春天"}):
    print(chunk, end="", flush=True)
```

关键点：
- `stream()` 返回一个迭代器，模型每生成一个 token 就 yield 一次
- 首字延迟通常在 0.3-0.8s，而 `invoke()` 要等全部生成完才返回
- 对用户来说，"立刻看到文字在跳"比"等 3 秒突然出一大段"体验好得多

RAG 场景：检索完成后，生成阶段用 `stream()` 让用户边看边等。

### 2. RunnableGenerator——流式不断流

```python
def add_prefix_stream(input_stream):
    yield "🌸 "
    for chunk in input_stream:
        yield chunk
```

这是一个容易踩的坑：

| 方式 | 流式是否保持 | 原因 |
|------|-------------|------|
| `RunnableLambda` | ❌ 阻断 | 等上游全部完成再处理 |
| `RunnableGenerator` | ✅ 保持 | 逐 chunk 接收、逐 chunk 输出 |

只要你的后处理逻辑是"来一个处理一个"，就用 `RunnableGenerator`。

### 3. batch——吞吐量翻倍

```python
results = chain.batch(topics, config={"max_concurrency": 4})
```

- `batch()` 内部用线程池并发执行，4 个请求同时发出
- `max_concurrency` 控制并发上限，防止触发 API 限流（OpenAI 默认 RPM 限制）
- 典型加速比：3-4x（受 API 并发限制影响）

生产建议：批量处理文档时，先 chunk → 再 batch embedding，效率提升显著。

### 4. 缓存——相同问题不重复花钱

```python
set_llm_cache(InMemoryCache())
```

- 相同的 prompt + 参数 → 直接返回缓存结果，不调 API
- `InMemoryCache`：进程内缓存，重启丢失，适合开发调试
- `SQLiteCache`：持久化缓存，适合生产环境高频重复查询

注意：`temperature > 0` 时缓存命中意味着每次返回相同结果，如果需要多样性就不要缓存。

### 5. RunnableParallel——独立任务并行跑

```python
RunnableParallel(
    summary=chain_a,
    pros=chain_b,
    cons=chain_c,
)
```

- 三个分支同时发请求，总耗时 ≈ max(a, b, c) 而不是 a + b + c
- 适合"多角度分析"、"多源检索"等场景
- RAG 中常见用法：同时查向量库 + 关键词库 + 知识图谱

### 6. 模型分级——省钱的艺术

```python
RunnableBranch(
    (lambda x: x["complexity"] == "simple", simple_chain),
    complex_chain,
)
```

- gpt-4o-mini 价格约为 gpt-4o 的 1/30
- 80% 的用户问题其实是简单问题，用 mini 就够了
- 只有真正需要深度分析的才路由到贵模型

## 性能优化速查表

| 优化目标 | 技术 | 适用场景 | 预期效果 |
|----------|------|----------|----------|
| 首字延迟 | `stream()` / `astream()` | 实时对话 | 0.3s 内首字 |
| 吞吐量 | `batch()` + `max_concurrency` | 批量处理 | 3-4x 加速 |
| 重复查询 | `InMemoryCache` / `SQLiteCache` | 高频相同查询 | 100x+ 加速 |
| 多任务 | `RunnableParallel` | 独立子任务 | 耗时取最慢分支 |
| 成本 | 模型分级 + `RunnableBranch` | 混合复杂度 | 节省 50-80% |
| 流式后处理 | `RunnableGenerator` | 流式自定义转换 | 保持流式体验 |

---

## 常见陷阱

1. **batch 不是万能的**：如果 API 有严格的 RPM 限制，`max_concurrency` 设太高会被限流，反而更慢
2. **缓存 + temperature > 0**：缓存会让每次返回完全相同的结果，需要多样性时别用缓存
3. **RunnableLambda 断流**：在流式链中插入 `RunnableLambda` 会阻断流式，改用 `RunnableGenerator`
4. **并行不等于更快**：如果子任务之间有依赖关系，强行并行会出错，只有真正独立的任务才适合并行
5. **模型分级的分类器本身也有成本**：如果用 LLM 做复杂度分类，分类本身也要花钱，简单规则往往更划算

---

## 与 RAG 的关系

在 RAG 系统中，这些优化技巧的典型应用：

- **流式输出**：检索完成后，生成阶段流式返回，用户不用干等
- **批处理**：文档入库时批量 embedding，比逐条快 3-4 倍
- **缓存**：热门问题缓存答案，节省 API 调用
- **并行检索**：同时查向量库和关键词库，结果合并后 rerank
- **模型分级**：简单问答用 mini，复杂分析用 4o，月省几百刀
