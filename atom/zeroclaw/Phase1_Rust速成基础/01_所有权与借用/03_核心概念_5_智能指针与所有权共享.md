# 核心概念 5：智能指针与所有权共享

> **一句话概括：** 当所有权三规则"每个值只有一个所有者"不够用时，智能指针（Box/Rc/Arc）让你在编译时安全地实现堆分配、共享所有权和跨线程数据传递。

---

## 1. 为什么需要智能指针

### 1.1 所有权三规则的限制

回顾 Rust 所有权核心规则：每个值有且仅有**一个**所有者；赋值或传参会**移动**所有权；所有者离开作用域时值被**自动释放**。

在真实项目中，你会遇到这些规则无法覆盖的场景：

- **编译时无法确定数据大小** -- trait 对象，不同实现结构体大小不同
- **多个组件需要读取同一份数据** -- 安全策略被多个工具共享
- **递归数据结构** -- 链表、树节点，类型定义里包含自身
- **跨线程共享状态** -- 多个异步任务共用同一份配置

### 1.2 类比：Google Docs vs 本地文件

| 操作模式 | 类比 | Rust 对应 |
|----------|------|-----------|
| 独占编辑一个本地文件 | 只有你能打开 | 普通所有权 `let x = value` |
| 给别人**复制**一份 | 各自修改互不影响 | `Clone` 深拷贝 |
| 给别人一个**只读链接** | 你持有原件，别人只能看 | 不可变借用 `&T` |
| 创建一个**共享文档** | 多人拥有访问权，最后关闭的人"清理" | `Rc<T>` / `Arc<T>` |

### 1.3 TypeScript 对比

```typescript
// TypeScript: 所有对象都是引用，自动 GC
const config = { maxRetries: 3 };
const tool1 = { config };  // 自动共享引用
const tool2 = { config };  // GC 在所有引用消失后清理
```

```rust
// Rust: 用 Arc 显式实现多所有者共享
let config = Arc::new(Config { max_retries: 3 });
let tool1_config = config.clone();  // 引用计数 +1
let tool2_config = config.clone();  // 引用计数 +1
// 当所有 Arc 被 drop 时，Config 才被释放
```

## 2. Box\<T\> -- 堆分配与单一所有权

`Box<T>` 是最简单的智能指针：在**堆**上分配数据，`Box` 本身（一个指针）在**栈**上。仍然是**单一所有权**，只是把数据放到堆上。

### 2.1 三个核心使用场景

**场景 1：Trait 对象（编译时大小未知）**

```rust
// 不同 Tool 实现有不同大小，编译器不知道分配多少栈空间
// Box 把数据放到堆上，Box 本身大小固定（胖指针 = 数据指针 + vtable）
let tool: Box<dyn Tool> = Box::new(ShellTool::new(security.clone()));
```

**场景 2：大数据避免栈溢出**

```rust
// 栈空间有限（通常 2-8 MB），800 万字节可能栈溢出
let big_array = Box::new([0u8; 8_000_000]);  // 放到堆上
```

**场景 3：递归类型**

```rust
enum List {
    Cons(i32, Box<List>),  // Box 打断无限递归的大小计算
    Nil,
}
```

### 2.2 ZeroClaw 实战：Vec\<Box\<dyn Tool\>\> 工具注册表

```rust
// [来源: sourcecode/zeroclaw/src/tools/mod.rs, 第 148-160 行]
pub fn default_tools_with_runtime(
    security: Arc<SecurityPolicy>,
    runtime: Arc<dyn RuntimeAdapter>,
) -> Vec<Box<dyn Tool>> {
    vec![
        Box::new(ShellTool::new(security.clone(), runtime)),
        Box::new(FileReadTool::new(security.clone())),
        Box::new(FileWriteTool::new(security.clone())),
        Box::new(FileEditTool::new(security.clone())),
        Box::new(GlobSearchTool::new(security.clone())),
        Box::new(ContentSearchTool::new(security)),
    ]
}
```

`Vec<Box<dyn Tool>>` 就是「异构集合」-- 不同类型统一为固定大小的堆指针，类似 TypeScript 的 `Tool[]`。

### 2.3 ZeroClaw 实战：Box\<dyn Memory\> 工厂模式

```rust
// [来源: sourcecode/zeroclaw/src/memory/mod.rs, 第 40-69 行]
fn create_memory_with_builders<F, G>(
    backend_name: &str, workspace_dir: &Path, /*...*/
) -> anyhow::Result<Box<dyn Memory>> {
    match classify_memory_backend(backend_name) {
        MemoryBackendKind::Sqlite => Ok(Box::new(sqlite_builder()?)),
        MemoryBackendKind::Markdown => Ok(Box::new(MarkdownMemory::new(workspace_dir))),
        MemoryBackendKind::None => Ok(Box::new(NoneMemory::new())),
        // ...
    }
}
```

工厂返回 `Box<dyn Trait>` 是 Rust 中"策略模式"的标准写法 -- 调用方只通过 trait 接口操作。

## 3. Rc\<T\> -- 单线程共享所有权

`Rc` = **R**eference **C**ount（引用计数），让多个变量"共同拥有"同一份数据。

```rust
use std::rc::Rc;

let data = Rc::new(vec![1, 2, 3]);
let data2 = Rc::clone(&data);  // 引用计数 +1（不复制数据！）
let data3 = Rc::clone(&data);  // 引用计数 +1
drop(data3);                    // 引用计数 -1
// 最后一个 Rc 被 drop 时，Vec 才释放
```

### 3.1 Rc::clone() vs Clone

```rust
let shared = Rc::clone(&data);      // O(1)，纳秒级，只增加计数
let copied = vec![1, 2, 3].clone(); // O(n)，深拷贝所有元素
```

社区惯例用 `Rc::clone(&x)` 而非 `x.clone()` 来强调"这是廉价操作"。

### 3.2 Weak\<T\> 打破循环引用

两个 `Rc` 互相引用 = 引用计数永远不归零 = 内存泄漏。

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;

struct Node {
    parent: RefCell<Weak<Node>>,       // 弱引用，不增加引用计数
    children: RefCell<Vec<Rc<Node>>>,  // 强引用
}
// Weak 可能指向已被释放的数据，使用时需 upgrade() 检查
```

### 3.3 Rc 的限制：不能跨线程

```rust
let data = Rc::new(42);
std::thread::spawn(move || {
    println!("{}", data);  // 编译错误！Rc 不实现 Send
});
```

## 4. Arc\<T\> -- 线程安全共享所有权

`Arc` = **A**tomic **R**eference **C**ount。与 Rc 唯一区别：引用计数用**原子指令**。

| 对比项 | `Rc<T>` | `Arc<T>` |
|--------|---------|----------|
| 引用计数操作 | 普通递增/递减 | 原子递增/递减 |
| 线程安全 | 否（不实现 `Send`） | 是（`Send + Sync`） |
| 性能开销 | 极小（~1-2ns） | 略大（~5-10ns，原子操作） |
| 使用场景 | 单线程共享 | 多线程/异步共享 |

实际应用中，Arc 的开销完全可以忽略 -- ZeroClaw 的瓶颈是 LLM API 调用（百毫秒级）。

**`Arc<T>` 只提供不可变访问。** 需要修改共享数据时配合 `Mutex` 或 `RwLock`：

```rust
let shared_state = Arc::new(Mutex::new(vec![1, 2, 3]));
let state_clone = Arc::clone(&shared_state);
tokio::spawn(async move {
    let mut data = state_clone.lock().unwrap();
    data.push(4);  // 安全地修改共享数据
});
```

## 5. ZeroClaw 实战实例

### 5.1 Arc\<SecurityPolicy\> -- 多工具共享安全策略

ZeroClaw 中最普遍的 Arc 模式。安全策略创建后被所有工具共享读取：

```rust
// [来源: sourcecode/zeroclaw/src/tools/mod.rs]
pub fn default_tools_with_runtime(
    security: Arc<SecurityPolicy>,
    runtime: Arc<dyn RuntimeAdapter>,
) -> Vec<Box<dyn Tool>> {
    vec![
        Box::new(ShellTool::new(security.clone(), runtime)), // clone = 引用计数 +1
        Box::new(FileReadTool::new(security.clone())),
        // ... 每个工具都 clone 一份 Arc ...
        Box::new(ContentSearchTool::new(security)), // 最后一个直接 move
    ]
}
```

为什么用 `Arc<SecurityPolicy>` 而不是 `&SecurityPolicy`？工具跨越异步边界、存储在结构体中，引用的生命周期注解会非常复杂。

```rust
// [来源: sourcecode/zeroclaw/src/tools/pdf_read.rs]
pub struct PdfReadTool {
    security: Arc<SecurityPolicy>,  // 存储共享引用
}
// 使用时: self.security.is_path_allowed(path)
```

### 5.2 Arc\<HashMap\<String, DelegateAgentConfig\>\> -- 委托配置

```rust
// [来源: sourcecode/zeroclaw/src/tools/delegate.rs, 第 23-36 行]
pub struct DelegateTool {
    agents: Arc<HashMap<String, DelegateAgentConfig>>,
    security: Arc<SecurityPolicy>,
    depth: u32,
    parent_tools: Arc<Vec<Arc<dyn Tool>>>,  // 嵌套 Arc
    // ...
}
```

### 5.3 Arc\<Vec\<Arc\<dyn Tool\>\>\> -- 嵌套共享

拆解 `parent_tools: Arc<Vec<Arc<dyn Tool>>>`：

1. `dyn Tool` -- trait 对象，大小未知
2. `Arc<dyn Tool>` -- 线程安全的共享 trait 对象
3. `Vec<Arc<dyn Tool>>` -- 工具列表
4. `Arc<Vec<...>>` -- 列表本身也共享（多个 DelegateTool 共用）

```rust
// [来源: sourcecode/zeroclaw/src/tools/mod.rs, 第 327 行]
let parent_tools = Arc::new(tool_arcs.clone());
// Vec<Arc<dyn Tool>> 的 clone 只复制 Arc 指针（计数 +1），不复制工具本身！
let delegate_tool = DelegateTool::new_with_options(/*...*/)
    .with_parent_tools(parent_tools);
```

### 5.4 Arc 到 Box 的桥接

ZeroClaw 内部用 `Arc` 管理工具（支持共享给子代理），对外 API 返回 `Box`（简单的单一所有权）：

```rust
// [来源: sourcecode/zeroclaw/src/tools/mod.rs, 第 108-140 行]
#[derive(Clone)]
struct ArcDelegatingTool {
    inner: Arc<dyn Tool>,
}

impl ArcDelegatingTool {
    fn boxed(inner: Arc<dyn Tool>) -> Box<dyn Tool> {
        Box::new(Self { inner })  // 用 Box 包装 Arc
    }
}

// 所有方法都委托给 inner
impl Tool for ArcDelegatingTool {
    fn name(&self) -> &str { self.inner.name() }
    async fn execute(&self, args: Value) -> Result<ToolResult> {
        self.inner.execute(args).await
    }
}

fn boxed_registry_from_arcs(tools: Vec<Arc<dyn Tool>>) -> Vec<Box<dyn Tool>> {
    tools.into_iter().map(ArcDelegatingTool::boxed).collect()
}
```

### 5.5 Arc\<dyn Memory\> -- 内存后端共享

```rust
// [来源: sourcecode/zeroclaw/src/tools/mod.rs, 第 167 行]
pub fn all_tools(memory: Arc<dyn Memory>, /*...*/) -> Vec<Box<dyn Tool>> {
    // 三个内存工具共享同一个后端实例
    Arc::new(MemoryStoreTool::new(memory.clone(), security.clone())),
    Arc::new(MemoryRecallTool::new(memory.clone())),
    Arc::new(MemoryForgetTool::new(memory, security.clone())),
}
```

`Arc<dyn Memory>` = 运行时多态（Sqlite/Markdown/Postgres）+ 线程安全共享。

## 6. 智能指针选择决策树

```
需要在堆上分配吗？
+-- 不需要 -> 普通值或引用 &T
+-- 需要
    +-- 单一所有权？ -> Box<T>
    |   +-- trait 对象 -> Box<dyn Trait>
    |   +-- 大数据 -> Box<[u8; N]>
    |   +-- 递归类型 -> Box<Self>
    +-- 多个所有者？
        +-- 单线程？ -> Rc<T>
        |   +-- 需要可变？ -> Rc<RefCell<T>>
        |   +-- 循环引用？ -> Weak<T>
        +-- 跨线程？ -> Arc<T>
            +-- 需要可变？ -> Arc<Mutex<T>> 或 Arc<RwLock<T>>
            +-- 只读？ -> Arc<T>
```

**ZeroClaw 使用频率：**

| 指针类型 | 频率 | 典型场景 |
|---------|------|---------|
| `Arc<T>` | 极高 | `Arc<SecurityPolicy>`、`Arc<Config>`、`Arc<dyn Tool>` |
| `Box<dyn Trait>` | 高 | `Box<dyn Tool>`、`Box<dyn Memory>`、`Box<dyn Provider>` |
| `Arc<Mutex<T>>` | 中等 | 共享可变状态 |
| `Rc<T>` | 低 | ZeroClaw 是异步多线程架构，很少用 Rc |

## 7. 与 TypeScript 对比总结

| 概念 | TypeScript/JavaScript | Rust |
|------|----------------------|------|
| 对象分配位置 | 全部在堆上（自动） | 默认栈上，`Box` 放堆上 |
| 内存回收 | GC（不可预测暂停） | 所有者 drop（确定性释放） |
| 多变量引用同一对象 | 天然支持 | 需要 `Rc` 或 `Arc` |
| 弱引用 | `WeakRef` | `Weak<T>` + `upgrade()` |
| 循环引用 | GC 能处理 | 必须用 `Weak<T>` 手动打破 |
| 线程安全 | 单线程 + Web Worker | `Arc<T>` + `Mutex`/`RwLock` |
| 接口/多态 | `interface` + 对象引用 | `dyn Trait` + `Box`/`Arc` |

**心智模型转换要点：**

1. **"免费"变"显式"** -- TS 的自动共享，Rust 需要 `Arc`/`Rc` 显式声明
2. **编译器是盟友** -- 忘了 `Arc` 或用错 `Rc`（跨线程），编译器立刻拦住
3. **性能确定** -- 无 GC 暂停，引用计数开销纳秒级且可预测
4. **`dyn` = "我不知道具体类型"** -- 类似 TS 接口引用，但需 `Box`/`Arc` 包装

## 8. 常见错误与反直觉点

**Arc::clone 不是深拷贝：** `Arc::clone(&data)` 只增加引用计数，两个变量指向同一份数据。

**不是所有地方都需要 Arc：** 只读访问用 `&T` 就够，只有需要"拥有"且"共享"时才用 Arc。

```rust
// 反模式
fn process(data: Arc<String>) { /* ... */ }
// 正确
fn process(data: &str) { /* ... */ }
```

**Arc\<Mutex\<T\>\> 不是万能药：** 只锁需要可变的部分，不要把整个状态锁起来。

**循环引用 = 内存泄漏：** Rust 没有 GC 检测循环。两个 `Rc`/`Arc` 互相引用，内存永不释放。用 `Weak<T>` 打破。

## 9. 速记卡片

| 智能指针 | 一句话记忆 | 搭配 |
|---------|----------|------|
| `Box<T>` | 堆上的单一所有者 | trait 对象、递归类型 |
| `Rc<T>` | 单线程共享文档 | `RefCell` 实现可变，`Weak` 打破循环 |
| `Arc<T>` | 多线程共享文档 | `Mutex`/`RwLock` 实现可变 |
| `Weak<T>` | "可能已删除的文档链接" | `upgrade() -> Option<Rc/Arc>` |

---

## 10. 一句话总结

> **`Box` 把数据放到堆上（单一所有权），`Rc`/`Arc` 让多个所有者共享同一份数据（引用计数），三者的选择取决于"谁拥有数据"和"是否跨线程" -- 编译器会在你选错时立刻报错。**
