# 多状态管理 - 第一性原理

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题。

不是"别人怎么做我就怎么做"，而是"这件事最本质的需求是什么？从这个需求出发，最合理的解决方案是什么？"

对于多状态管理，我们要问的第一性问题是：

> **当一个图变得复杂——有多个阶段、多个子模块、多种角色——如何让每个部分只看到自己该看的数据、只改自己该改的字段？**

---

## 多状态管理的第一性原理

### 1. 最基础的定义

**多状态管理 = 为图的不同层级和不同阶段划定独立的状态边界**

```python
# 一句话版本：
# 不是所有节点都需要看到所有状态，不是所有状态都需要暴露给外部

# 源码层面：StateGraph 接受四个 Schema 参数
graph = StateGraph(
    state_schema=OverallState,      # 内部完整状态
    input_schema=InputState,        # 外部输入接口
    output_schema=OutputState,      # 外部输出接口
    context_schema=ContextSchema,   # 运行时不可变上下文
)
```

仅此而已！多状态管理的本质就是**给数据画边界线**。

想象你在一家公司工作。公司有一个巨大的数据库（完整状态），但不是每个人都能
看到所有数据：
- 前台只能看到访客登记信息（input_schema）
- 客户只能拿到最终报告（output_schema）
- 各部门有自己的内部文档（节点级状态分组）
- 子公司有独立的账本（子图状态隔离）
- 公司章程人人可读但没人能改（context_schema）

**多状态管理就是 LangGraph 的"权限管理系统"：控制谁能看到什么、谁能改什么。**

---

### 2. 为什么需要多状态管理？

**核心问题：当所有节点共享同一个巨大的状态对象时，系统会变得脆弱、混乱、不可维护。**

想象一下只有单一状态的世界：

```python
# ❌ 单一状态：所有字段对所有人可见
class EverythingState(TypedDict):
    # 用户输入
    user_query: str
    # 检索中间结果（内部实现细节）
    raw_chunks: list[str]
    rerank_scores: list[float]
    embedding_cache: dict
    # 生成中间结果（内部实现细节）
    prompt_template: str
    llm_raw_response: str
    token_count: int
    # 最终输出
    answer: str
    sources: list[str]
```

这种做法有四个致命问题：

**问题 1：信息泄露**
```python
# 调用者能看到所有内部细节
result = graph.invoke({"user_query": "什么是 RAG？"})
print(result["embedding_cache"])    # 暴露了内部缓存！
print(result["rerank_scores"])      # 暴露了排序分数！
print(result["prompt_template"])    # 暴露了 Prompt 模板！
```

**问题 2：接口混乱**
```python
# 调用者不知道该传什么、能拿什么
# 这 9 个字段里，哪些是输入？哪些是输出？哪些是中间状态？
graph.invoke({
    "user_query": "问题",
    "raw_chunks": [],        # 需要传吗？
    "rerank_scores": [],     # 需要传吗？
    "embedding_cache": {},   # 需要传吗？
})
```

**问题 3：节点耦合**
```python
# 每个节点都能看到所有字段，容易产生隐式依赖
def generate_node(state: EverythingState):
    # 这个节点本来只需要 raw_chunks 和 user_query
    # 但因为能看到 rerank_scores，就"顺手"用了
    if state["rerank_scores"][0] > 0.9:  # 隐式依赖！
        ...
```

**问题 4：子模块不可复用**
```python
# 如果想把检索模块抽成独立子图复用，
# 它被迫要知道整个 EverythingState 的结构
# 改一个字段名，所有子模块都要跟着改
```

**多状态管理的解决方案：**

```python
# ✅ 多状态管理：清晰的边界
class InputState(TypedDict):
    user_query: str                    # 外部只需要传这一个

class OutputState(TypedDict):
    answer: str                        # 外部只能拿到这两个
    sources: list[str]

class OverallState(InputState, OutputState):
    raw_chunks: list[str]              # 内部中间状态
    rerank_scores: list[float]         # 对外不可见
    prompt_template: str

graph = StateGraph(
    OverallState,
    input_schema=InputState,           # 入口白名单
    output_schema=OutputState,         # 出口白名单
)

# 调用者的体验：干净、清晰
result = graph.invoke({"user_query": "什么是 RAG？"})
# result == {"answer": "RAG 是...", "sources": ["doc1", "doc2"]}
# 看不到 raw_chunks、rerank_scores 等内部细节
```

就像一栋办公楼：访客只能进大厅（input_schema），只能从前台拿到会议纪要
（output_schema），各楼层有门禁（节点级状态分组），子公司在另一栋楼
（子图隔离）。不是不信任，而是**边界清晰才能高效协作**。

---

### 3. 多状态管理的三层价值

#### 价值1：边界控制 —— Input/Output Schema 分离

**图的输入和输出应该是简洁、稳定的接口，与内部实现解耦。**

这是多状态管理最基础的价值。通过 `input_schema` 和 `output_schema`，你可以：

1. **隐藏内部实现**：调用者不需要知道图内部有多少中间字段
2. **稳定外部接口**：内部重构不影响调用方
3. **类型安全**：编译时就能检查输入输出是否合法

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class InputState(TypedDict):
    question: str

class OutputState(TypedDict):
    answer: str

class OverallState(InputState, OutputState):
    # 这些字段对外不可见
    retrieved_docs: list[str]
    confidence_score: float

builder = StateGraph(
    OverallState,
    input_schema=InputState,      # 外部只能传 question
    output_schema=OutputState,    # 外部只能拿 answer
)
```

**类比：餐厅的菜单和出餐口。** 顾客（调用者）通过菜单（input_schema）点菜，
从出餐口（output_schema）取餐。厨房里的备料、烹饪、摆盘过程（内部状态）
顾客完全不需要知道。菜单可以一直不变，但厨房可以随时换厨师、换食材供应商。

**源码验证：** `input_schema` 和 `output_schema` 默认等于 `state_schema`：

```python
# langgraph/graph/state.py
def __init__(self, state_schema, context_schema=None, *, input_schema=None, output_schema=None):
    self.input_schema = input_schema or state_schema   # 默认 = 完整状态
    self.output_schema = output_schema or state_schema  # 默认 = 完整状态
```

这意味着如果你不指定，图的输入输出就是完整状态——这对简单场景够用，但对
复杂系统来说就是"把厨房敞开给顾客看"。

#### 价值2：关注点分离 —— 节点级状态分组与子图隔离

**每个节点只关心自己需要的数据，子图拥有完全独立的状态空间。**

这一层解决的是"内部复杂度"问题。即使内部状态有 20 个字段，每个节点也只
需要看到其中 3-5 个。

**节点级状态分组：** 每个节点可以声明自己的 input_schema，只接收状态的子集。

```python
class OverallState(TypedDict):
    user_input: str
    foo: str
    graph_output: str

class PrivateState(TypedDict):
    bar: str

# node_1 只接收 InputState（只有 user_input）
def node_1(state: InputState) -> OverallState:
    return {"foo": state["user_input"] + " name"}

# node_2 接收 OverallState（有 foo）
def node_2(state: OverallState) -> PrivateState:
    return {"bar": state["foo"] + " is"}

# node_3 只接收 PrivateState（只有 bar）
def node_3(state: PrivateState) -> OutputState:
    return {"graph_output": state["bar"] + " Lance"}
```

LangGraph 会自动从函数的类型注解推断每个节点的 input_schema：

```python
# langgraph/graph/state.py - 自动推断机制
def add_node(self, node, action=None, *, input_schema=None, ...):
    if input_schema is not None:
        # 显式指定
        self.nodes[node] = StateNodeSpec(..., input_schema=input_schema)
    elif inferred_input_schema is not None:
        # 从函数第一个参数的类型提示自动推断
        self.nodes[node] = StateNodeSpec(..., input_schema=inferred_input_schema)
```

**子图状态隔离：** 子图有完全独立的 Channel 集合，与父图是两个"世界"。

```python
# 子图有自己的状态定义
class SubgraphState(TypedDict):
    bar: str    # 与父图的 foo 无关
    baz: str    # 子图私有

# 父图有自己的状态定义
class ParentState(TypedDict):
    foo: str

# 父子图通过显式调用通信
def parent_node(state: ParentState):
    response = subgraph.invoke({"bar": state["foo"]})  # 显式传入
    return {"foo": response["bar"]}                     # 显式取出
```

**类比：公司的部门制度。** 每个部门（节点）有自己的工作台，上面只放自己需要
的文件。子公司（子图）有独立的办公室和文件柜，与总公司通过正式的文件传递
（invoke 调用）沟通，而不是随意翻看对方的文件。

#### 价值3：可组合性 —— 子图独立状态带来的模块化

**子图因为有独立状态，可以像乐高积木一样在不同父图中复用。**

这是多状态管理最高层的价值。当子图的状态与父图完全隔离时：

1. **独立开发**：子图团队不需要知道父图的状态结构
2. **独立测试**：子图可以单独 invoke 测试，不依赖父图
3. **跨项目复用**：同一个子图可以嵌入不同的父图

```python
# 一个可复用的"文档检索"子图
class RetrievalState(TypedDict):
    query: str
    docs: list[str]

retrieval_subgraph = build_retrieval_graph()  # 独立构建、独立测试

# 在 RAG 项目中使用
rag_builder = StateGraph(RAGState)
rag_builder.add_node("retrieve", retrieval_subgraph)

# 在客服项目中也能使用——同一个子图，不同的父图
support_builder = StateGraph(SupportState)
support_builder.add_node("search", retrieval_subgraph)
```

**共享键自动通信：** 当子图作为编译图直接添加为节点时，父子图中同名的键
会自动传递，无需手动转换：

```python
class SubgraphState(TypedDict):
    foo: str    # 与父图共享的键
    bar: str    # 子图私有的键

class ParentState(TypedDict):
    foo: str    # 同名键 → 自动传递给子图

builder = StateGraph(ParentState)
builder.add_node("child", compiled_subgraph)  # foo 自动共享
```

**命名空间隔离：** LangGraph 通过命名空间系统确保子图的 Checkpoint 不会
与父图冲突：

```python
# 命名空间格式（源码常量）
NS_SEP = "|"   # 层级分隔符
NS_END = ":"   # 命名空间与 task_id 分隔符

# 示例：
# ""                                    → 根图
# "agent:abc123"                        → 一级子图
# "agent:abc123|researcher:def456"      → 二级子图（嵌套）
```

**类比：连锁餐厅的加盟模式。** 总部（父图）定义了品牌标准（共享键），
每家加盟店（子图）有自己的员工、库存、账本（独立状态）。加盟店可以开在
不同城市（不同父图），只要遵守品牌标准就行。总部通过统一的报表系统
（命名空间）追踪每家店的运营数据，互不混淆。

**三层价值的关系：**

```
价值1：边界控制       → "外部能看到什么？"     → Input/Output Schema
价值2：关注点分离     → "内部谁看到什么？"     → 节点分组 + 子图隔离
价值3：可组合性       → "模块能复用吗？"       → 独立状态 + 命名空间

三层层层递进：从外部接口到内部分工到跨项目复用，构成多状态管理的完整价值链。
```

### 4. 从第一性原理推导多状态管理

**推理链：**

```
1. 复杂工作流有多个阶段 → 每个阶段需要不同数据
   ↓
2. 不同阶段的数据不应互相干扰 → 需要状态隔离
   ↓
3. 图的输入输出应该简洁 → 需要 Schema 分离
   ↓
4. 子图应该可复用 → 需要独立状态 + 命名空间
   ↓
5. 运行时配置不应污染状态 → 需要 context_schema
```

让我们逐步展开这条推理链：

**第 1 步：复杂工作流的本质需求**

一个生产级 RAG 系统不是简单的"检索 + 生成"。它可能包含：查询理解、查询
改写、多路检索、重排序、上下文压缩、回答生成、幻觉检测、引用标注……

每个阶段需要的数据完全不同：
- 查询改写需要：原始查询、对话历史
- 向量检索需要：改写后的查询、embedding 模型配置
- 重排序需要：候选文档列表、排序模型配置
- 回答生成需要：精选文档、Prompt 模板、LLM 配置

如果把所有字段塞进一个 State，就像把所有工具塞进一个工具箱——找个螺丝刀
都要翻半天。

**第 2 步：状态隔离的必要性**

不同阶段的数据不仅"不同"，而且可能"冲突"。比如：
- 检索阶段的 `scores` 是 BM25 分数
- 重排序阶段的 `scores` 是 cross-encoder 分数
- 如果共用同一个字段，就会互相覆盖

状态隔离让每个阶段有自己的"工作空间"，互不干扰。

**第 3 步：Schema 分离的价值**

图的调用者不应该关心内部有多少中间字段。就像你用一个 API，只需要知道
请求参数和响应格式，不需要知道服务器内部的数据库表结构。

```python
# input_schema 和 output_schema 就是图的"API 文档"
class InputState(TypedDict):
    question: str           # 调用者只需要传这个

class OutputState(TypedDict):
    answer: str             # 调用者只能拿到这个
    sources: list[str]
```

**第 4 步：子图独立状态与命名空间**

当你想把"检索模块"抽成独立子图在多个项目中复用时，子图必须有自己的状态
定义。否则，子图就被绑死在某个特定父图的状态结构上了。

LangGraph 通过命名空间系统确保子图的 Checkpoint 数据不会与父图混淆：

```
根图的 Checkpoint:     namespace = ""
子图的 Checkpoint:     namespace = "retriever:task_001"
孙图的 Checkpoint:     namespace = "retriever:task_001|reranker:task_002"
```

每一层都有独立的存储空间，就像文件系统的目录结构。

**第 5 步：context_schema 的独特定位**

有些数据是"运行时配置"——比如 API 密钥、模型名称、温度参数。这些数据：
- 不应该出现在状态中（它们不是工作流数据）
- 不应该被节点修改（它们是只读配置）
- 应该在整个执行过程中保持不变

`context_schema` 专门解决这个问题：

```python
from langgraph.runtime import Runtime

class ContextSchema(TypedDict):
    model_name: str
    temperature: float

class State(TypedDict):
    query: str
    answer: str

def generate_node(state: State, runtime: Runtime[ContextSchema]):
    # 从 runtime.context 读取配置，不污染状态
    model = runtime.context["model_name"]
    temp = runtime.context["temperature"]
    answer = call_llm(state["query"], model=model, temperature=temp)
    return {"answer": answer}

builder = StateGraph(State, context_schema=ContextSchema)
```

关键区别：`context_schema` **不创建 Channel**。它通过 `Runtime` 对象传递
给节点，是完全独立于状态系统的"旁路通道"。

### 5. 源码验证

LangGraph 源码中，多状态管理的核心设计体现在 `graph/state.py` 的
`StateGraph.__init__()` 方法中：

```python
# langgraph/graph/state.py - StateGraph 构造函数
class StateGraph:
    def __init__(
        self,
        state_schema: type[StateT],
        context_schema: type[ContextT] | None = None,
        *,
        input_schema: type[InputT] | None = None,
        output_schema: type[OutputT] | None = None,
    ) -> None:
        ...
        self.input_schema = input_schema or state_schema
        self.output_schema = output_schema or state_schema
```

四个参数，四种角色：

| 参数 | 默认值 | 创建 Channel？ | 用途 |
|------|--------|---------------|------|
| `state_schema` | 必填 | 是 | 内部完整状态，所有字段的超集 |
| `input_schema` | = state_schema | 是（子集） | 图的输入接口，START 节点只写入这些字段 |
| `output_schema` | = state_schema | 是（子集） | 图的输出接口，返回值只包含这些字段 |
| `context_schema` | None | **否** | 运行时不可变上下文，通过 Runtime 传递 |

**Schema 到 Channel 的映射机制：**

```python
# langgraph/graph/state.py - _get_channels() 函数
def _get_channels(schema: type[dict]):
    """将 Schema 的类型注解转换为 Channel"""
    type_hints = get_type_hints(schema, include_extras=True)
    all_keys = {
        name: _get_channel(name, typ)
        for name, typ in type_hints.items()
        if name != "__slots__"
    }
    return (
        {k: v for k, v in all_keys.items() if isinstance(v, BaseChannel)},  # Channel
        {k: v for k, v in all_keys.items() if is_managed_value(v)},         # ManagedValue
        type_hints,
    )
```

每个 Schema 的每个字段都会被转换为一个 Channel。多个 Schema 的 Channel
通过 `_add_schema()` 方法合并到统一的 Channel 池中。同名字段必须兼容——
如果两个 Schema 对同一个字段定义了不同的 Reducer，编译时会报错。

**命名空间系统的源码常量：**

```python
# langgraph/_internal/_constants.py
NS_SEP = "|"   # 层级分隔符：graph|subgraph|subsubgraph
NS_END = ":"   # task_id 分隔符：node_name:task_id
```

命名空间格式：`{node_name}:{task_id}|{child_node_name}:{child_task_id}`

这个格式确保了：
- 每个子图实例有唯一标识（通过 task_id）
- 多层嵌套不会冲突（通过 `|` 分隔层级）
- Checkpoint 存储可以按命名空间隔离

**子图通信的四条路径（源码级）：**

```
1. 调用时传入输入        → subgraph.invoke({"bar": state["foo"]})
2. 返回值取出输出        → return {"foo": response["bar"]}
3. Command 显式更新父图  → Command(graph=Command.PARENT, update={...})
4. Config keys 共享资源  → checkpointer, store, cache, stream
```

### 6. 多状态管理与 RAG 开发的关系

多状态管理在 RAG 开发中至关重要，因为生产级 RAG 系统天然就是多阶段、
多模块的复杂工作流。

**一个典型的多状态 RAG 管道：**

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

# ---- 边界 Schema：对外接口 ----
class RAGInput(TypedDict):
    question: str

class RAGOutput(TypedDict):
    answer: str
    sources: list[str]

# ---- 内部完整状态：包含所有中间字段 ----
class RAGInternalState(RAGInput, RAGOutput):
    refined_query: str              # 查询改写结果
    raw_docs: list[str]             # 原始检索结果
    reranked_docs: list[str]        # 重排序后的文档
    confidence: float               # 回答置信度
    messages: Annotated[list, add_messages]  # 对话历史

# ---- 节点函数：每个只关心自己需要的字段 ----
def rewrite_query(state: RAGInput) -> dict:
    """查询改写：只需要 question"""
    refined = improve_query(state["question"])
    return {"refined_query": refined}

def retrieve_docs(state: RAGInternalState) -> dict:
    """文档检索：需要 refined_query"""
    docs = vector_search(state["refined_query"], top_k=20)
    return {"raw_docs": docs}

def rerank_docs(state: RAGInternalState) -> dict:
    """重排序：需要 raw_docs 和 refined_query"""
    reranked = cross_encoder_rerank(state["refined_query"], state["raw_docs"])
    return {"reranked_docs": reranked[:5]}

def generate_answer(state: RAGInternalState) -> dict:
    """生成回答：需要 reranked_docs 和 question"""
    context = "\n\n".join(state["reranked_docs"])
    answer = llm.invoke(f"根据以下内容回答：{context}\n问题：{state['question']}")
    sources = extract_sources(state["reranked_docs"])
    return {"answer": answer, "sources": sources}

# ---- 组装图：使用多状态管理 ----
builder = StateGraph(
    RAGInternalState,
    input_schema=RAGInput,       # 外部只传 question
    output_schema=RAGOutput,     # 外部只拿 answer + sources
)
builder.add_node("rewrite", rewrite_query)
builder.add_node("retrieve", retrieve_docs)
builder.add_node("rerank", rerank_docs)
builder.add_node("generate", generate_answer)
builder.add_edge(START, "rewrite")
builder.add_edge("rewrite", "retrieve")
builder.add_edge("retrieve", "rerank")
builder.add_edge("rerank", "generate")
builder.add_edge("generate", END)

graph = builder.compile()

# 调用者的体验：干净简洁
result = graph.invoke({"question": "什么是向量数据库？"})
# result == {"answer": "向量数据库是...", "sources": ["doc1.pdf", "doc2.pdf"]}
# 看不到 refined_query、raw_docs、reranked_docs、confidence 等内部细节
```

**为什么 RAG 特别需要多状态管理？**

1. **阶段多**：查询理解 → 检索 → 重排序 → 生成 → 后处理，每个阶段的数据需求不同
2. **模块可复用**：检索模块、重排序模块可以在不同 RAG 项目中复用（子图隔离）
3. **接口稳定**：用户只关心"问题进去、答案出来"，内部优化不影响调用方
4. **配置灵活**：不同场景用不同模型、不同温度（context_schema）

---

### 7. 一句话总结第一性原理

**多状态管理是 LangGraph 的"权限与边界系统"，通过 Schema 分离、节点分组、
子图隔离和运行时上下文四层机制，让复杂工作流的每个部分只看到自己该看的数据、
只改自己该改的字段。**

记住这个核心公式：

```
多状态管理 = Schema 分离（外部边界）
           + 节点分组（内部分工）
           + 子图隔离（模块独立）
           + context_schema（配置旁路）

StateGraph(
    state_schema,      # 内部全景    → "公司所有数据"
    input_schema,      # 入口白名单  → "前台能接收什么"
    output_schema,     # 出口白名单  → "客户能拿到什么"
    context_schema,    # 只读配置    → "公司章程"
)
```

## 本节要点回顾

1. **多状态管理的本质**：为图的不同层级和阶段划定独立的状态边界
2. **为什么需要**：单一状态导致信息泄露、接口混乱、节点耦合、模块不可复用
3. **三层价值**：边界控制（外部接口）→ 关注点分离（内部分工）→ 可组合性（跨项目复用）
4. **四个 Schema**：state_schema（完整状态）、input_schema（输入接口）、output_schema（输出接口）、context_schema（运行时配置）
5. **推导链**：多阶段 → 不同数据需求 → 状态隔离 → Schema 分离 → 子图独立 → context 旁路
6. **命名空间**：`{node}:{task_id}|{child}:{child_task_id}` 格式确保子图 Checkpoint 隔离
7. **RAG 应用**：生产级 RAG 天然需要多状态管理——阶段多、模块可复用、接口要稳定
