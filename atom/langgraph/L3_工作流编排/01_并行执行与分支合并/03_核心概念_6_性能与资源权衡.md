## 1. 核心权衡：动态 vs 静态

### 1.1 两种并行模式对比

LangGraph 提供两种并行执行模式，各有优劣：

| 维度 | 动态并行（Send API） | 静态并行（多条边） |
|------|---------------------|-------------------|
| **灵活性** | ⭐⭐⭐⭐⭐ 运行时确定 | ⭐⭐ 编译时固定 |
| **性能** | ⭐⭐⭐ 有额外开销 | ⭐⭐⭐⭐⭐ 最优性能 |
| **内存使用** | ⭐⭐ 动态分配 | ⭐⭐⭐⭐ 预分配 |
| **代码复杂度** | ⭐⭐⭐⭐ 较复杂 | ⭐⭐⭐ 简单直观 |
| **适用场景** | 任务数量不确定 | 任务数量固定 |

### 1.2 动态并行（Send API）

**优势**：
- 运行时动态确定并行任务数量
- 灵活的状态分发
- 支持 Map-Reduce 模式
- 适合数据驱动的并行

**劣势**：
- 每个 Send 对象有创建开销
- 动态调度增加同步成本
- 内存使用不可预测
- 调试相对困难

**示例**：
```python
from langgraph.graph import StateGraph
from langgraph.types import Send
from typing_extensions import TypedDict, Annotated
import operator

class State(TypedDict):
    items: list[str]
    results: Annotated[list[str], operator.add]

def fan_out(state: State):
    # 动态创建并行任务
    return [Send("process", {"item": item}) for item in state["items"]]

def process(state: State):
    # 处理单个项目
    return {"results": [f"Processed: {state['item']}"]}

builder = StateGraph(State)
builder.add_node("process", process)
builder.add_conditional_edges("start", fan_out, ["process"])
```

### 1.3 静态并行（多条边）

**优势**：
- 编译时确定，性能最优
- 内存使用可预测
- 调试简单直观
- 同步开销最小

**劣势**：
- 并行度固定，不够灵活
- 无法根据数据动态调整
- 代码重复（需要定义多个节点）

**示例**：
```python
from langgraph.graph import StateGraph
from typing_extensions import TypedDict, Annotated
import operator

class State(TypedDict):
    results: Annotated[list[str], operator.add]

def task_a(state: State):
    return {"results": ["Result A"]}

def task_b(state: State):
    return {"results": ["Result B"]}

def task_c(state: State):
    return {"results": ["Result C"]}

builder = StateGraph(State)
builder.add_node("task_a", task_a)
builder.add_node("task_b", task_b)
builder.add_node("task_c", task_c)

# 静态并行：从 start 扇出到三个任务
builder.add_edge("start", "task_a")
builder.add_edge("start", "task_b")
builder.add_edge("start", "task_c")

# 扇入到 end
builder.add_edge("task_a", "end")
builder.add_edge("task_b", "end")
builder.add_edge("task_c", "end")
```

## 2. 并行度控制策略

### 2.1 为什么需要控制并行度？

**问题场景**：
- 处理 1000 个文档，每个文档创建一个并行任务
- 同时发起 1000 个 LLM 调用
- 内存占用激增，系统崩溃

**核心原则**：
> 并行度不是越高越好，要根据系统资源和外部服务限制合理控制

### 2.2 批处理策略

**方法1：固定批次大小**

```python
from langgraph.graph import StateGraph
from langgraph.types import Send
from typing_extensions import TypedDict, Annotated
import operator

class State(TypedDict):
    documents: list[str]
    results: Annotated[list[str], operator.add]

def batch_fan_out(state: State):
    """将文档分批处理，控制并行度"""
    batch_size = 10  # 每批处理 10 个文档
    documents = state["documents"]

    # 创建批次任务
    sends = []
    for i in range(0, len(documents), batch_size):
        batch = documents[i:i + batch_size]
        sends.append(Send("process_batch", {"batch": batch}))

    return sends

def process_batch(state: State):
    """处理一批文档"""
    results = []
    for doc in state["batch"]:
        # 处理单个文档
        result = f"Processed: {doc}"
        results.append(result)

    return {"results": results}

builder = StateGraph(State)
builder.add_node("process_batch", process_batch)
builder.add_conditional_edges("start", batch_fan_out, ["process_batch"])
```

**方法2：动态批次大小**

```python
import os

def adaptive_batch_fan_out(state: State):
    """根据系统资源动态调整批次大小"""
    documents = state["documents"]
    total_docs = len(documents)

    # 根据文档数量动态调整批次大小
    if total_docs < 50:
        batch_size = 5
    elif total_docs < 200:
        batch_size = 10
    else:
        batch_size = 20

    # 创建批次任务
    sends = []
    for i in range(0, total_docs, batch_size):
        batch = documents[i:i + batch_size]
        sends.append(Send("process_batch", {"batch": batch}))

    return sends
```

### 2.3 限流策略

**方法：使用信号量控制并发**

```python
import asyncio
from typing import List

class RateLimitedState(TypedDict):
    items: list[str]
    results: Annotated[list[str], operator.add]
    semaphore: asyncio.Semaphore  # 控制并发数

async def process_with_limit(state: RateLimitedState):
    """使用信号量限制并发"""
    async with state["semaphore"]:
        # 模拟 API 调用
        await asyncio.sleep(0.1)
        return {"results": [f"Processed: {state['item']}"]}

def fan_out_with_limit(state: RateLimitedState):
    """创建限流的并行任务"""
    # 最多同时执行 5 个任务
    semaphore = asyncio.Semaphore(5)

    sends = []
    for item in state["items"]:
        sends.append(Send("process", {
            "item": item,
            "semaphore": semaphore
        }))

    return sends
```

## 3. 内存使用与同步开销

### 3.1 内存使用分析

**动态并行的内存开销**：

1. **Send 对象创建**：每个 Send 对象占用内存
2. **状态复制**：每个并行任务可能复制部分状态
3. **结果缓存**：所有并行任务的结果需要缓存直到合并

**示例：内存使用估算**

```python
import sys

def estimate_memory_usage(num_tasks: int, state_size_kb: int):
    """估算并行任务的内存使用"""
    # Send 对象大小（估算）
    send_object_size = 0.5  # KB

    # 状态复制大小
    state_copy_size = state_size_kb * 0.3  # 假设复制 30% 的状态

    # 结果缓存大小（假设每个结果 1KB）
    result_cache_size = 1  # KB

    # 总内存使用
    total_memory = num_tasks * (
        send_object_size +
        state_copy_size +
        result_cache_size
    )

    return total_memory

# 示例：1000 个任务，每个状态 10KB
memory_mb = estimate_memory_usage(1000, 10) / 1024
print(f"预估内存使用: {memory_mb:.2f} MB")
# 输出: 预估内存使用: 4.39 MB
```

### 3.2 同步开销

**Bulk Synchronous Parallel (BSP) 模型**：

LangGraph 使用 BSP 模型，每个超步（Superstep）结束时有同步点：

```
超步1: [Task1, Task2, Task3] 并行执行
       ↓
    同步点（等待所有任务完成）
       ↓
超步2: [Reduce] 合并结果
```

**同步开销来源**：
1. **等待最慢任务**：木桶效应，最慢的任务决定整体速度
2. **状态合并**：Reducer 函数的执行时间
3. **调度开销**：任务分发和结果收集

**优化策略**：
```python
def balanced_fan_out(state: State):
    """平衡任务负载，减少同步等待"""
    items = state["items"]

    # 按任务复杂度排序（简单任务优先）
    sorted_items = sorted(items, key=lambda x: estimate_complexity(x))

    # 创建并行任务
    return [Send("process", {"item": item}) for item in sorted_items]

def estimate_complexity(item: str) -> int:
    """估算任务复杂度"""
    # 简单示例：根据字符串长度估算
    return len(item)
```

## 4. 大规模并行任务优化

### 4.1 分层并行策略

**问题**：1000 个文档需要并行处理

**解决方案**：两层并行

```python
from langgraph.graph import StateGraph
from langgraph.types import Send
from typing_extensions import TypedDict, Annotated
import operator

class State(TypedDict):
    documents: list[str]
    results: Annotated[list[str], operator.add]

def level1_fan_out(state: State):
    """第一层：分成 10 个批次"""
    documents = state["documents"]
    batch_size = len(documents) // 10

    sends = []
    for i in range(0, len(documents), batch_size):
        batch = documents[i:i + batch_size]
        sends.append(Send("level2_fan_out", {"batch": batch}))

    return sends

def level2_fan_out(state: State):
    """第二层：每个批次再分成 10 个子任务"""
    batch = state["batch"]
    sub_batch_size = max(1, len(batch) // 10)

    sends = []
    for i in range(0, len(batch), sub_batch_size):
        sub_batch = batch[i:i + sub_batch_size]
        sends.append(Send("process", {"items": sub_batch}))

    return sends

def process(state: State):
    """处理子批次"""
    results = [f"Processed: {item}" for item in state["items"]]
    return {"results": results}

builder = StateGraph(State)
builder.add_node("level2_fan_out", level2_fan_out)
builder.add_node("process", process)
builder.add_conditional_edges("start", level1_fan_out, ["level2_fan_out"])
builder.add_conditional_edges("level2_fan_out", level2_fan_out, ["process"])
```

### 4.2 流式处理策略

**问题**：等待所有任务完成才能看到结果

**解决方案**：流式输出中间结果

```python
from langgraph.graph import StateGraph
from typing import AsyncIterator

async def streaming_process(state: State) -> AsyncIterator[dict]:
    """流式处理并输出中间结果"""
    for item in state["items"]:
        # 处理单个项目
        result = f"Processed: {item}"

        # 立即输出结果
        yield {"results": [result]}

        # 模拟处理延迟
        await asyncio.sleep(0.1)

# 使用流式节点
builder.add_node("process", streaming_process)
```

### 4.3 资源监控与自适应调整

```python
import psutil
import time

class AdaptiveState(TypedDict):
    items: list[str]
    results: Annotated[list[str], operator.add]
    max_parallel: int

def adaptive_fan_out(state: AdaptiveState):
    """根据系统资源自适应调整并行度"""
    # 获取系统资源使用情况
    cpu_percent = psutil.cpu_percent(interval=0.1)
    memory_percent = psutil.virtual_memory().percent

    # 根据资源使用情况调整并行度
    if cpu_percent > 80 or memory_percent > 80:
        # 资源紧张，减少并行度
        max_parallel = 5
    elif cpu_percent < 50 and memory_percent < 50:
        # 资源充足，增加并行度
        max_parallel = 20
    else:
        # 中等资源，适中并行度
        max_parallel = 10

    # 分批创建任务
    items = state["items"]
    batch_size = max(1, len(items) // max_parallel)

    sends = []
    for i in range(0, len(items), batch_size):
        batch = items[i:i + batch_size]
        sends.append(Send("process", {"batch": batch}))

    return sends
```

## 5. 性能基准测试

### 5.1 测试框架

```python
import time
from typing import Callable

def benchmark_parallel_strategy(
    strategy: Callable,
    num_items: int,
    num_runs: int = 3
) -> dict:
    """性能基准测试"""
    results = {
        "avg_time": 0,
        "min_time": float('inf'),
        "max_time": 0,
        "memory_peak": 0
    }

    for _ in range(num_runs):
        # 记录开始时间和内存
        start_time = time.time()
        start_memory = psutil.Process().memory_info().rss / 1024 / 1024  # MB

        # 执行策略
        strategy(num_items)

        # 记录结束时间和内存
        end_time = time.time()
        end_memory = psutil.Process().memory_info().rss / 1024 / 1024  # MB

        # 更新统计
        elapsed = end_time - start_time
        memory_used = end_memory - start_memory

        results["avg_time"] += elapsed
        results["min_time"] = min(results["min_time"], elapsed)
        results["max_time"] = max(results["max_time"], elapsed)
        results["memory_peak"] = max(results["memory_peak"], memory_used)

    results["avg_time"] /= num_runs

    return results

# 测试动态并行
dynamic_results = benchmark_parallel_strategy(
    lambda n: dynamic_parallel_strategy(n),
    num_items=100
)

# 测试静态并行
static_results = benchmark_parallel_strategy(
    lambda n: static_parallel_strategy(n),
    num_items=100
)

print(f"动态并行: {dynamic_results}")
print(f"静态并行: {static_results}")
```

## 6. 最佳实践总结

### 6.1 选择决策树

```
是否需要运行时确定并行度？
├─ 是 → 使用 Send API（动态并行）
│   ├─ 任务数量 < 50 → 直接并行
│   ├─ 任务数量 50-200 → 批处理（batch_size=10）
│   └─ 任务数量 > 200 → 分层并行 + 批处理
│
└─ 否 → 使用多条边（静态并行）
    ├─ 任务数量 < 10 → 直接扇出扇入
    └─ 任务数量 >= 10 → 考虑重构为动态并行
```

### 6.2 性能优化清单

- [ ] 控制并行度（批处理、限流）
- [ ] 平衡任务负载（避免木桶效应）
- [ ] 监控资源使用（CPU、内存）
- [ ] 使用流式处理（减少等待时间）
- [ ] 分层并行（大规模任务）
- [ ] 缓存中间结果（避免重复计算）
- [ ] 异步执行（提高吞吐量）

### 6.3 常见陷阱

1. **过度并行**：创建过多任务导致内存溢出
2. **忽略同步开销**：最慢任务拖累整体性能
3. **状态复制过多**：每个任务复制大量状态
4. **缺乏错误处理**：一个任务失败导致整体失败
5. **忽略外部限制**：API 限流、数据库连接数
