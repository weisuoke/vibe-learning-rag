## 卡片1：为什么需要状态调试？

**一句话：** LangGraph 图的执行是非确定性的多步工作流，传统断点调试不够用，你需要专门的状态调试手段。

**核心问题：**

传统程序是线性的：A → B → C，出错了加个断点就能定位。但 LangGraph 图有三个特殊性：

1. **多步骤**：一个图可能经过 5-10 个节点，每个节点都会修改状态
2. **条件分支**：路由节点根据状态动态决定走哪条路，每次执行路径可能不同
3. **状态累积**：Annotated reducer（如 `operator.add`）让状态不是简单覆盖，而是累积合并

```
传统调试：在某一行设断点 → 看变量值
LangGraph 调试：在某一步看完整状态快照 → 追踪状态如何一步步变化
```

**应用：** 当你的 RAG Agent 返回了错误答案，你需要知道是检索节点没找到正确文档，还是生成节点忽略了上下文——状态调试让你精确定位到出问题的那一步。

## 卡片2：stream_mode 是什么？

**一句话：** `stream_mode` 是 `graph.stream()` 的参数，控制输出什么级别的调试信息，6 种模式从粗到细覆盖所有观测需求。

**6 种调试相关模式：**

| 模式 | 输出内容 | 信息粒度 |
|------|----------|----------|
| `values` | 每步后的完整状态 | 粗 |
| `updates` | 仅节点的增量更新 | 中 |
| `tasks` | 任务开始/完成事件 | 中 |
| `checkpoints` | 检查点完整快照 | 细 |
| `debug` | checkpoints + tasks | 最细 |
| `custom` | 你手动写入的数据 | 自定义 |

**基本用法：**

```python
# 只需要改一个参数
for chunk in graph.stream(input_data, config, stream_mode="updates"):
    print(chunk)
```

**应用：** 不同的调试场景选不同的模式——快速排查用 `updates`，深度分析用 `debug`，生产监控用 `custom`。

## 卡片3：values vs updates 模式

**一句话：** `values` 输出每步后的完整状态（全景照片），`updates` 只输出变化部分（差异对比），大多数情况用 `updates` 就够了。

**对比示例：**

```python
# values 模式 —— 每步输出完整状态
# 输出: {'messages': ['开始'], 'count': 0}
# 输出: {'messages': ['开始', 'A处理完毕'], 'count': 1}
# 输出: {'messages': ['开始', 'A处理完毕', 'B处理完毕'], 'count': 11}

# updates 模式 —— 只输出节点的更新
# 输出: {'node_a': {'messages': ['A处理完毕'], 'count': 1}}
# 输出: {'node_b': {'messages': ['B处理完毕'], 'count': 11}}
```

**前端类比：** `values` 像 React 的 `state`（每次拿到完整状态），`updates` 像 Redux 的 `action`（只看发生了什么变化）。

**选择建议：**
- 状态字段少（< 5 个）→ 用 `values`，直观
- 状态字段多或包含长列表 → 用 `updates`，避免信息过载
- 不确定 → 先用 `updates`，需要全貌时切 `values`

## 卡片4：debug 模式的秘密

**一句话：** `debug` 模式 = `checkpoints` + `tasks`，它同时输出检查点事件和任务事件，是信息量最大的调试模式。

**三种事件类型：**

| type | 触发时机 | 包含信息 |
|------|----------|----------|
| `"task"` | 节点开始执行 | 节点名、输入数据、触发器 |
| `"task_result"` | 节点执行完毕 | 节点名、结果、错误、中断 |
| `"checkpoint"` | 状态保存时 | 完整状态值、下一步节点、元数据 |

**源码实现（简化）：**

```python
# langgraph/pregel/_loop.py
# debug 模式复用 checkpoints 和 tasks 的事件，额外包装 step + timestamp + type
{
    "step": 0,
    "timestamp": "2026-02-27T10:30:45+00:00",
    "type": "task",          # 或 "task_result" 或 "checkpoint"
    "payload": { ... }       # 原始的 TaskPayload / CheckpointPayload
}
```

**应用：** 当 `updates` 模式信息不够时，切到 `debug` 模式。它能告诉你节点收到了什么输入、返回了什么结果、是否有错误，以及每个检查点的完整快照。

## 卡片5：get_state() 快照检查

**一句话：** `get_state(config)` 返回 `StateSnapshot`，一个调用就能拿到图执行的完整快照，是调试的第一工具。

**前提条件：** 图必须配置了 checkpointer（`compile(checkpointer=InMemorySaver())`）。

**最常用的 4 个字段：**

```python
snapshot = graph.get_state(config)

snapshot.values          # 当前状态的所有字段值
snapshot.next            # 下一步要执行的节点（空元组 = 执行完毕）
snapshot.metadata["step"]    # 执行到第几步
snapshot.metadata["source"]  # 快照来源：input / loop / update
```

**快速判断执行状态：**

```python
snapshot = graph.get_state(config)
if not snapshot.next:
    print("图已执行完毕")
else:
    print(f"图卡在了: {snapshot.next}")
```

**应用：** 调试的第一步永远是 `get_state()` —— 先看看现在是什么情况，再决定下一步怎么查。就像医生先量体温，再决定做什么检查。

## 卡片6：get_state_history() 历史追溯

**一句话：** `get_state_history()` 返回按时间倒序排列的 StateSnapshot 迭代器，让你遍历完整执行历史，定位问题发生的精确步骤。

**基本用法：**

```python
for snapshot in graph.get_state_history(config):
    step = snapshot.metadata.get("step", -1)
    source = snapshot.metadata.get("source", "")
    print(f"Step {step} [{source}]: next={snapshot.next}")
    print(f"  values={snapshot.values}")
```

**读懂历史记录：**

| step | source | 含义 |
|------|--------|------|
| `-1` | `input` | 用户输入刚进来，还没执行任何节点 |
| `0` | `loop` | 第一个节点执行完毕 |
| `1` | `loop` | 第二个节点执行完毕 |
| `N` | `update` | 手动调用了 `update_state()` |

**实用技巧：**

```python
# 限制数量，避免历史太长
for snapshot in graph.get_state_history(config, limit=5):
    ...

# 对比相邻步骤，找到状态突变点
history = list(graph.get_state_history(config))
for i in range(len(history) - 1):
    curr, prev = history[i].values, history[i + 1].values
    for key in curr:
        if curr.get(key) != prev.get(key):
            print(f"Step {history[i].metadata['step']}: '{key}' 发生变化")
```

**应用：** 当最终结果不对，用 `get_state_history()` 像 `git log` 一样回溯，找到"从哪一步开始不对"。

## 卡片7：时间旅行调试

**一句话：** 通过 `checkpoint_id` 回到任意历史状态，用 `update_state()` 修改后重新执行——这是 LangGraph 最强大的调试能力。

**核心流程：**

```
发现问题 → 遍历历史找到出错前的状态 → 修改状态 → 从那里重新执行
```

**代码示例：**

```python
# 1. 遍历历史，找到目标状态
target_config = None
for snapshot in graph.get_state_history(config):
    if snapshot.next == ("step_c",):  # 找到 step_c 执行前的状态
        target_config = snapshot.config
        break

# 2. 修改历史状态（创建分叉）
graph.update_state(
    target_config,
    {"answer": "修正后的回答"},  # 修改 answer 字段
    as_node="step_b",            # 伪装成 step_b 的输出
)

# 3. 从修改后的状态继续执行
new_result = graph.invoke(None, target_config)
```

**日常生活类比：** 就像游戏里的"读档"——存档点就是 checkpoint，读档后你可以做出不同的选择，走出不同的结局。

**应用：** 发现 RAG 管道中检索节点返回了错误文档？回到检索前，手动注入正确文档，验证后续节点是否能正常工作。

## 卡片8：自定义调试工具

**一句话：** `StreamWriter` 发送自定义数据到流中，调试装饰器分离调试代码与业务代码，状态差异对比工具自动发现变化。

**三种方案：**

**方案1：StreamWriter（推荐生产使用）**

```python
from langgraph.config import get_stream_writer

def retrieve(state):
    writer = get_stream_writer()
    writer({"stage": "retrieval", "found_docs": 3})
    # ... 业务逻辑
    return result
```

**方案2：调试装饰器（分离关注点）**

```python
def debug_node(func):
    def wrapper(state):
        print(f"[{func.__name__}] 输入: {state}")
        result = func(state)
        print(f"[{func.__name__}] 输出: {result}")
        return result
    return wrapper

@debug_node
def my_node(state):
    return {"answer": "处理完成"}
```

**方案3：状态差异对比**

```python
def diff_states(before: dict, after: dict) -> dict:
    changes = {}
    for key in set(list(before.keys()) + list(after.keys())):
        if before.get(key) != after.get(key):
            changes[key] = {"before": before.get(key), "after": after.get(key)}
    return changes
```

**应用：** 开发阶段用 print/装饰器快速排查，生产阶段用 StreamWriter 输出结构化调试数据。

## 卡片9：LangSmith 生产监控

**一句话：** 设置 `LANGSMITH_TRACING=true` 即可启用，零侵入提供执行路径可视化、性能指标、错误追踪。

**最简启用：**

```bash
export LANGSMITH_TRACING=true
export LANGSMITH_API_KEY=<your-api-key>
```

不需要修改任何业务代码。你的每一次 `invoke()` 和 `stream()` 都会自动被追踪。

**追踪配置：**

```python
config = {
    "configurable": {"thread_id": "user-001"},
    "tags": ["production", "v2.1"],
    "metadata": {
        "user_id": "user_123",
        "session_id": "session_456",
    }
}
result = graph.invoke(input_data, config=config)
```

**LangSmith 自动记录：**
- 每个节点的输入/输出和耗时
- LLM 调用的 token 用量和延迟
- 完整的错误堆栈和异常上下文
- 执行路径的可视化展示

**前端类比：** 本地调试像 `console.log()`，LangSmith 像 Sentry/DataDog——有完整的监控面板、历史记录，团队成员都能看到。

**应用：** 用户反馈"昨天的回答不对"，在 LangSmith 中按 `user_id` + 时间范围筛选，精确找到那次执行的完整追踪。

## 卡片10：调试策略总结

**一句话：** 不同阶段用不同工具，从快速排查到生产监控形成完整的调试体系。

**三阶段策略：**

```
开发阶段（快速迭代）
├── print() / logging          ← 最快，即时反馈
├── stream_mode="updates"      ← 追踪每步变化
├── stream_mode="debug"        ← 需要完整信息时
└── get_state()                ← 检查具体状态

测试阶段（问题定位）
├── get_state_history()        ← 历史追溯
├── 时间旅行（update_state）    ← 修改历史重新执行
└── 状态差异对比               ← 自动发现变化

生产阶段（监控运维）
├── LangSmith                  ← 全链路追踪
├── StreamWriter               ← 自定义监控数据
└── 结构化日志                 ← 持久化调试信息
```

**调试决策树（遇到问题时）：**

| 步骤 | 操作 | 目的 |
|------|------|------|
| 第1步 | `get_state(config)` | 看最终状态对不对 |
| 第2步 | `stream_mode="updates"` | 追踪每步变化，找到出错的节点 |
| 第3步 | `get_state_history()` | 对比相邻步骤，精确定位问题 |
| 第4步 | 节点内加 print/logging | 看节点的输入和输出细节 |
| 第5步 | 时间旅行 | 回到出错前，修改状态验证修复方案 |
| 第6步 | `stream_mode="debug"` | 查看任务级别的完整信息 |

**核心原则：** 从粗到细，从简单到复杂。先用最轻量的工具，不够再升级。
