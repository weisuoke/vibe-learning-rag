## 30 字核心

> **Rust 通过编译期所有权检查，零成本保证内存安全，无需垃圾回收。**

这句话浓缩了 Rust 内存管理的全部哲学。下面逐词拆解。

---

## 一、为什么是"编译期"？

### 1.1 编译期 vs 运行时：两种检查策略

| 维度 | 运行时检查（GC 方案） | 编译期检查（Rust 方案） |
|------|----------------------|----------------------|
| **代表语言** | JavaScript/TS, Go, Java | Rust |
| **检查时机** | 程序运行时，GC 线程后台扫描 | 编译时，编译器静态分析 |
| **性能代价** | GC 暂停 + 额外内存开销 | 零运行时开销 |
| **开发体验** | 不用管内存，偶遇 GC 卡顿 | 编译器会"骂你"，但跑起来飞快 |
| **出错时机** | 运行时才发现内存问题 | 编译阶段直接报错 |

**前端类比：** TypeScript 的类型检查就是一种"编译期检查"。Rust 的所有权检查类似，但检查的不是类型，而是"这块内存谁在用、用多久、能不能改"。

### 1.2 一个直观对比

```typescript
// TypeScript —— 运行时 GC 管内存
function createUser() {
  const user = { name: "Alice" }; // 堆上分配
  return user;                     // 引用传出去
}
// user 对象什么时候释放？等 GC 扫，不确定
```

```rust
// Rust —— 编译期就决定好了
fn create_user() -> String {
    let user = String::from("Alice"); // 堆上分配
    user  // 所有权转移给调用方，释放时机确定
}
```

## 二、什么是"所有权检查"？

### 2.1 所有权三规则

> 1. **每个值都有一个所有者**（owner）
> 2. **同一时刻只能有一个所有者**
> 3. **当所有者离开作用域，值被自动释放**（drop）

[来源: reference/context7_rust_book_01.md]

**生活类比：** 把"值"想成一本实体书——每本书只有一个人"持有"，不能同时让两人拥有，持有者离开图书馆就归还（释放）。

### 2.2 移动语义（Move）

这是 TypeScript 开发者最容易踩坑的地方：

```rust
let s1 = String::from("hello");
let s2 = s1;          // 所有权从 s1 转移到 s2
// println!("{}", s1); // 编译错误！s1 已失效
println!("{}", s2);    // OK
```

```typescript
// TypeScript —— 赋值只是复制引用，GC 追踪
const s1 = "hello";
const s2 = s1;  // s1 和 s2 都有效，GC 负责追踪
```

**关键区别：** TS 赋值是"复制引用 + GC 追踪"，Rust 赋值是"转移所有权 + 编译器追踪"。

[来源: reference/context7_rust_book_01.md]

### 2.3 借用（Borrow）—— 临时使用不夺权

| 借用类型 | 语法 | 类比 | 限制 |
|---------|------|------|------|
| **不可变借用** `&T` | `&s` | 图书馆借书看 | 可同时借给多人看 |
| **可变借用** `&mut T` | `&mut s` | 借走修改 | 同一时刻只能借给一个人改 |

```rust
fn print_length(s: &String) {  // 借用，不夺走所有权
    println!("长度: {}", s.len());
}

let s = String::from("hello");
print_length(&s);   // 借给函数用
println!("{}", s);   // s 还在，没被拿走
```

```rust
// 多个不可变借用 —— OK
let s = String::from("hello");
let r1 = &s;  // OK
let r2 = &s;  // OK，多人同时看没问题

// 可变 + 不可变借用不能共存 —— 编译错误
let mut s = String::from("hello");
let r1 = &s;
// let r2 = &mut s;  // 错误！有人在看时不能改
```

**前端类比：** 这就像数据库的读写锁——多线程可同时读，但写必须独占。

[来源: reference/context7_rust_book_01.md]

### 2.4 函数签名中的三种语义

在 ZeroClaw 源码中你会频繁看到：

| 签名 | 含义 | 类比 |
|------|------|------|
| `fn do_it(self)` | 夺走所有权 | "这本书给你了" |
| `fn do_it(&self)` | 不可变借用，只读 | "给你看看" |
| `fn do_it(&mut self)` | 可变借用，可改 | "借你改改" |

```rust
// ZeroClaw SafetyMonitor 中的不可变借用
pub async fn can_move(&self) -> bool {
    self.state.can_move.load(Ordering::SeqCst)
}
```

## 三、什么是"零成本"？

### 3.1 Zero-Cost Abstractions

> **你用不到的特性，不为它付出任何代价。**
> **你用到的特性，不可能有比手写更高效的实现。**

| 特性 | C 语言 | TypeScript/JS | Rust |
|------|--------|---------------|------|
| 内存管理 | 手动 malloc/free | GC 运行时开销 | 所有权（编译期，零运行时开销） |
| 生成的机器码 | 直接操作内存 | V8 + GC 代码 | 与手写 C 等效 |
| 内存安全 | 不保证 | GC 兜底 | 编译器兜底 |

### 3.2 直觉理解

- **GC 方案**（TS）：包裹放传送带上，有专人盯着没人要的包裹再回收——"盯着的人"消耗资源
- **手动管理**（C）：你自己决定什么时候扔包裹——快但容易扔早了或忘了扔
- **所有权方案**（Rust）：包裹上贴标签"属于张三，张三走了就自动销毁"——不需要额外的人盯着

## 四、什么是"内存安全"？

所有权系统在编译期消灭四类内存 Bug：

```
┌──────────────────────────────────────────────────────┐
│  检查 1: 每个值只有一个所有者  ──→  防止 Double-Free   │
│  检查 2: 移动后不能再使用      ──→  防止 Use-After-Free │
│  检查 3: 可变/不可变借用互斥   ──→  防止 Data Race     │
│  检查 4: 引用不能比数据活得久  ──→  防止 Dangling Ref  │
│                                                       │
│         全部在编译期完成，运行时零开销                    │
└──────────────────────────────────────────────────────┘
```

**Bug 1 — Use-After-Free：** 移动后不能使用，编译器直接报错。在 C 中可能读到垃圾数据或被黑客利用。

**Bug 2 — Double-Free：** 只有一个所有者 = 只释放一次。规则 2 的直接结果。

**Bug 3 — Data Race：** 借用规则保证"有人读时不能写"，编译期就杜绝了并发冲突。

**Bug 4 — Dangling Reference：** 编译器禁止返回局部变量的引用。

```rust
// 编译错误：悬垂引用
fn dangle() -> &String {
    let s = String::from("hello");
    &s  // s 函数结束就释放了，引用会悬空
}

// 正确做法：返回所有权
fn no_dangle() -> String {
    let s = String::from("hello");
    s  // 移动出去
}
```

## 五、为什么"无需垃圾回收"？

### 5.1 各语言 GC 对比

| 语言 | 内存管理 | GC 暂停 | 额外开销 |
|------|---------|---------|---------|
| **JavaScript/TS** | Mark-Sweep + Generational | 1-50ms | ~30% 额外内存 |
| **Go** | Concurrent Mark-Sweep | ~0.5ms | GC 元数据 |
| **Java** | G1/ZGC 等 | 1-200ms | 堆设为需求的 2-4 倍 |
| **Rust** | 所有权 + RAII（无 GC） | **0ms** | **零额外开销** |

### 5.2 RAII —— 确定性释放

```rust
{
    let s = String::from("hello");  // 分配
    // 使用 s ...
}   // 编译器自动插入 drop(s)，确定性释放
```

### 5.3 TypeScript GC 概念对应

| JS/TS 概念 | Rust 对应 | 说明 |
|-----------|-----------|------|
| `WeakRef` | 生命周期 `'a` | 不延长对象生命 |
| `FinalizationRegistry` | `Drop` trait | 销毁时执行清理 |
| `null` / `undefined` | `Option<T>` | 值可能不存在 |
| 闭包捕获变量 | `move` 闭包 | 数据是借用还是移动 |

## 六、在 ZeroClaw 中的体现

### 6.1 Arc\<SecurityPolicy\> —— 共享所有权避免深拷贝

ZeroClaw 的安全策略需要被 20+ 个工具共享。用 `Arc`（原子引用计数）而非深拷贝：

```rust
pub fn default_tools_with_runtime(
    security: Arc<SecurityPolicy>,
    runtime: Arc<dyn RuntimeAdapter>,
) -> Vec<Box<dyn Tool>> {
    vec![
        Box::new(ShellTool::new(security.clone(), runtime)),  // 引用计数 +1
        Box::new(FileReadTool::new(security.clone())),         // 引用计数 +1
        Box::new(FileWriteTool::new(security.clone())),        // 引用计数 +1
        Box::new(ContentSearchTool::new(security)),            // 最后一个，移动
    ]
}
```

[来源: sourcecode/zeroclaw/src/tools/mod.rs]

每个工具只保存一个 8 字节指针，不深拷贝整个策略数据：

```rust
pub struct FileReadTool {
    security: Arc<SecurityPolicy>,  // 8 字节指针，不是完整副本
}
```

[来源: sourcecode/zeroclaw/src/tools/file_read.rs]

### 6.2 SafetyMonitor —— 并发中的所有权实战

```rust
pub struct SafetyMonitor {
    config: SafetyConfig,           // 独占所有权
    state: Arc<SafetyState>,        // 多任务共享状态
    event_tx: broadcast::Sender<SafetyEvent>,
    shutdown: AtomicBool,
}
```

异步任务中通过 `clone` + `move` 安全共享状态：

```rust
tokio::spawn({
    let state = self.state.clone();       // Arc 引用计数 +1
    let event_tx = self.event_tx.clone(); // 给新任务一份
    async move {
        // 新任务拥有 state 和 event_tx 的共享所有权
        tokio::time::sleep(Duration::from_secs(2)).await;
        state.can_move.store(true, Ordering::SeqCst);
    }
});
```

## 七、Copy 与 Clone 的区别

```rust
// Copy（自动，栈上简单类型）
let x: i32 = 42;
let y = x;           // 复制，x 依然有效

// Move（默认，堆上复杂类型）
let s1 = String::from("hello");
let s2 = s1;          // 移动，s1 失效

// Clone（显式深拷贝）
let s1 = String::from("hello");
let s2 = s1.clone();  // 深拷贝，s1 有效
```

| 操作 | 成本 | 场景 |
|------|------|------|
| Copy（自动） | 极低（栈上几个字节） | i32, bool, f64 等 |
| Clone（显式） | 可能很高（堆内存深拷贝） | 需要独立副本 |
| Arc::clone | 极低（引用计数 +1） | 共享同一数据 |
| Move（默认） | 零（转移指针） | 转让所有权 |

[来源: reference/context7_rust_book_01.md]

## 八、非词法生命周期（NLL）

编译器分析引用的"最后使用点"，而非作用域末尾：

```rust
let mut s = String::from("hello");
let r1 = &s;
let r2 = &s;
println!("{} {}", r1, r2);  // r1, r2 最后使用在此
// r1, r2 生命周期已结束

let r3 = &mut s;             // OK！不可变借用已结束
println!("{}", r3);
```

## 十、常见误区

**误区 1："Rust 不能有多个变量指向同一数据"**
事实：可以。用 `&T` 借用或 `Arc<T>` 共享所有权。规则限制的是"同时修改"。

**误区 2："所有权让 Rust 代码很难写"**
事实：大部分情况只用 `&self`（不可变借用），体验接近 TypeScript。

**误区 3："Move 意味着数据被物理移动了"**
事实：Move 通常只是编译器层面的概念，编译器会优化掉不必要的拷贝。

**误区 4："Clone 和 Copy 是一样的"**
事实：`Copy` 是隐式的栈复制。`Clone` 是显式的，可能涉及昂贵的堆分配。

## 总结

```
  Rust 通过编译期所有权检查，零成本保证内存安全，无需垃圾回收。

  编译期 ──→ 不是运行时，性能零损耗
  所有权 ──→ 三规则 + 借用规则
  零成本 ──→ 生成的代码和手写 C 一样快
  内存安全 ─→ 消灭 UAF / DF / DR / Dangling 四大 bug
  无需 GC ──→ 确定性释放，无暂停

  ZeroClaw 因此实现：< 5MB RAM | < 10ms 启动 | 0ms GC 暂停
```
