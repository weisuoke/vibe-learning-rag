## 基础概念类

### Q1: OpenClaw 使用什么包管理器？为什么？

**答案**：

OpenClaw 使用 **pnpm**，原因：

1. **内容寻址存储**：
   - 全局 store 存储所有包
   - 硬链接到项目 node_modules
   - 节省磁盘空间（相比 npm 节省 50-70%）

2. **严格依赖管理**：
   - 避免幽灵依赖
   - 只能访问 package.json 中声明的依赖
   - 提升项目可靠性

3. **Monorepo 支持**：
   - workspace 协议
   - 依赖提升
   - 跨包引用

4. **性能优势**：
   - 安装速度快 2-3 倍
   - 并行安装
   - 增量安装

**追问**：pnpm 和 npm 的底层架构有什么区别？

**答案**：
- **npm**：每个项目完整复制依赖到 node_modules
- **pnpm**：全局 store + 硬链接 + 符号链接

### Q2: OpenClaw 使用什么构建工具？为什么不用 tsc？

**答案**：

OpenClaw 使用 **tsdown**（基于 Rolldown），原因：

1. **Bundler vs Compiler**：
   - tsc：逐文件编译，保留模块结构
   - tsdown：整体打包，tree-shaking，代码分割

2. **性能优势**：
   - 构建速度比 tsup 快 2 倍
   - 生成声明文件快 8 倍
   - 基于 Rust 实现

3. **优化能力**：
   - Tree-shaking：移除未使用代码
   - Code splitting：按需加载
   - Minification：压缩代码

4. **多入口支持**：
   - OpenClaw 有 8 个入口点
   - tsdown 可以高效处理

**追问**：OpenClaw 的 8 个入口点是什么？

**答案**：
1. src/index.ts - 主入口
2. src/entry.ts - 备用入口
3. src/cli/daemon-cli.ts - CLI 守护进程
4. src/infra/warning-filter.ts - 警告过滤
5. src/plugin-sdk/index.ts - Plugin SDK
6. src/plugin-sdk/account-id.ts - Account ID
7. src/extensionAPI.ts - Extension API
8. src/hooks/bundled/*/handler.ts - Hook 处理器

### Q3: OpenClaw 有多少个测试配置？为什么需要多个？

**答案**：

OpenClaw 有 **5 个独立的测试配置**：

1. **vitest.unit.config.ts**：
   - Pool: threads
   - 用途：单元测试
   - 特点：快速，共享内存

2. **vitest.e2e.config.ts**：
   - Pool: vmForks
   - 用途：E2E 测试
   - 特点：完全隔离，避免状态污染

3. **vitest.live.config.ts**：
   - maxWorkers: 1
   - 用途：Live 测试
   - 特点：单线程，避免外部服务并发冲突

4. **vitest.gateway.config.ts**：
   - Pool: threads
   - 用途：Gateway 测试
   - 特点：快速，针对 Gateway 模块

5. **vitest.extensions.config.ts**：
   - Pool: threads
   - 用途：扩展测试
   - 特点：快速，针对扩展模块

**为什么需要多个配置**：
- 不同测试类型需要不同的隔离级别
- 不同测试类型需要不同的性能特性
- 不同测试类型需要不同的超时设置

**追问**：threads 和 vmForks 的区别是什么？

**答案**：
- **threads**：共享内存，快速，但可能状态污染
- **vmForks**：完全隔离，慢速，但绝对安全

## 实战操作类

### Q4: 如何从零开始构建 OpenClaw？

**答案**：

完整流程（约 20 分钟）：

```bash
# 1. 克隆仓库
git clone https://github.com/openclaw/openclaw.git
cd openclaw

# 2. 检查环境
node --version  # 需要 22+
pnpm --version  # 需要 10.23.0+

# 3. 安装依赖（10-15 分钟）
pnpm install

# 4. 构建项目（2-3 分钟）
pnpm build

# 5. 运行测试（可选）
pnpm test:fast

# 6. 启动 Gateway
pnpm gateway:dev
```

**关键点**：
- 依赖安装最耗时（原生依赖编译）
- 构建流程有 9 步
- 验证每个步骤的输出

**追问**：如果依赖安装失败怎么办？

**答案**：
1. 检查 Node.js 版本（需要 22+）
2. 检查编译工具链（macOS: Xcode Command Line Tools, Linux: build-essential, Windows: Visual Studio Build Tools）
3. 清理缓存：`pnpm store prune`
4. 重新安装：`rm -rf node_modules && pnpm install`

### Q6: 如何优化 OpenClaw 的构建速度？

**答案**：

多种优化策略：

1. **启用 tsdown 缓存**：
```typescript
export default defineConfig({
  cache: true,
  cacheDir: '.tsdown-cache',
  incremental: true
})
```

2. **优化 TypeScript 配置**：
```json
{
  "compilerOptions": {
    "skipLibCheck": true,
    "skipDefaultLibCheck": true,
    "incremental": true
  }
}
```

3. **减少入口点**：
   - 合并相关入口点
   - 从 8 个减少到 3 个
   - 提升 36% 构建速度

4. **使用 esbuild（开发环境）**：
```bash
esbuild src/index.ts --bundle --platform=node --outfile=dist/index.js
# 提升 98% 构建速度
```

**效果**：
- 首次构建：2m 5s
- 增量构建（无修改）：5s（提升 96%）
- 增量构建（修改单个文件）：15s（提升 88%）

## 高级概念类

### Q13: 什么是 workspace 协议？OpenClaw 如何使用？

**答案**：

**workspace 协议**：pnpm 的本地包引用机制

**OpenClaw 的使用**：
```json
{
  "dependencies": {
    "openclaw": "workspace:*"
  }
}
```

**行为**：
- **开发时**：使用符号链接指向本地包
- **发布时**：自动替换为实际版本号

**示例**：
```json
// 开发时
{
  "dependencies": {
    "openclaw": "workspace:*"
  }
}

// 发布时（自动转换）
{
  "dependencies": {
    "openclaw": "2026.2.22"
  }
}
```

**追问**：如果忘记更新版本号会怎样？

**答案**：
- 发布时会报错：`Cannot find package openclaw@workspace:*`
- 解决方案：使用 `pnpm publish --dry-run` 检查

### Q14: OpenClaw 如何处理内存不足的问题？

**答案**：

多种策略：

1. **限制并发**：
```bash
export OPENCLAW_TEST_WORKERS=1
export OPENCLAW_TEST_MAX_OLD_SPACE_SIZE_MB=2048
```

2. **增加 Node.js 内存限制**：
```bash
NODE_OPTIONS='--max-old-space-size=4096' pnpm build
```

3. **使用 --no-frozen-lockfile**：
```bash
pnpm install --no-frozen-lockfile
```

4. **流式处理大文件**：
```typescript
// ❌ 不好：一次性读取
const content = fs.readFileSync(path, 'utf-8')

// ✅ 好：流式处理
const fileStream = fs.createReadStream(path)
```

**GitHub Issue #1868**：
- 环境: 4GB RAM
- 错误: OOM (Out of Memory)
- 原因: pnpm 反复完整构建
- 解决: 限制并发 + 增加内存限制

## 核心洞察

### 1. 工具链选择很重要

**表面**：用什么工具都一样

**实际**：
- pnpm vs npm：架构完全不同
- tsdown vs tsc：bundler vs compiler
- oxlint vs ESLint：性能差异 50-100 倍

---

### 2. 测试配置需要分离

**表面**：一个配置文件就够了

**实际**：
- 不同测试类型需要不同隔离级别
- threads vs vmForks 的权衡
- 性能 vs 安全的平衡

---

### 3. 原生依赖是最大挑战

**表面**：`pnpm install` 就能搞定

**实际**：
- 需要编译工具链
- 跨平台兼容性差
- 可能需要 10-15 分钟
