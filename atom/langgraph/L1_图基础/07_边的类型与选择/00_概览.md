# 07_边的类型与选择 - 知识点概览

## 知识点简介

边（Edge）是 LangGraph 中连接节点的核心机制，决定了工作流的执行路径和控制流。LangGraph 提供了五种边类型：普通边用于固定路由、条件边实现动态决策、等待边处理并行同步、序列边简化线性流程、动态路由 Send 支持 fan-out 模式。掌握边的类型与选择策略是构建复杂状态化工作流的关键。本知识点深入源码实现，结合官方文档和社区实践，系统讲解边的内部机制、使用场景、选择策略和常见陷阱，帮助你在实际开发中做出正确的边类型选择。

**学习时长**: 2-3 小时
**难度级别**: ⭐⭐⭐ (中级)
**前置知识**: StateGraph 与节点定义、State Schema 与类型化状态

---

## 核心概念列表

本知识点包含 **5 个核心概念**，按学习顺序排列：

### 1. 普通边（Normal Edge）
**定义**: 通过 `add_edge()` 添加的固定路由边，直接连接两个节点
**核心特征**: 确定性流程、单起始或多起始（等待边）
**使用场景**: 简单的线性流程、确定性的节点连接
**API**: `builder.add_edge(start, end)` 或 `builder.add_edge([start1, start2], end)`

### 2. 条件边（Conditional Edge）
**定义**: 通过 `add_conditional_edges()` 添加的动态路由边，根据路由函数返回值决定下一个节点
**核心特征**: 动态决策、支持多分支、可返回 Send 对象
**使用场景**: 需要根据状态进行决策的场景、条件分支、循环控制
**API**: `builder.add_conditional_edges(source, path_func, path_map)`

### 3. 等待边（Waiting Edge）
**定义**: 通过 `add_edge([node1, node2], target)` 添加的同步边，等待多个节点全部完成后才执行目标节点
**核心特征**: 多节点同步、AND 逻辑、使用 NamedBarrierValue 通道
**使用场景**: 需要等待多个并行任务完成的场景、汇聚点
**API**: `builder.add_edge([node1, node2, ...], target)`

### 4. 序列边（Sequence Edge）
**定义**: 通过 `add_sequence()` 添加的语法糖，自动添加节点和边，简化线性流程定义
**核心特征**: 语法糖、自动推断节点名称、内部使用普通边
**使用场景**: 简化顺序节点的添加、线性流程
**API**: `builder.add_sequence([node1, node2, ...])`

### 5. 动态路由（Send）
**定义**: 条件边返回 `Send` 对象实现动态并行执行，可以向目标节点发送自定义状态
**核心特征**: 动态 fan-out、自定义状态传递、支持 map-reduce 模式
**使用场景**: 动态并行执行、map-reduce 模式、批量处理
**API**: `Send(node, state)` 在条件边的路由函数中返回

## 边的类型对比表

| 边类型 | API | 路由方式 | 使用场景 | 性能 | 复杂度 |
|--------|-----|----------|----------|------|--------|
| **普通边** | `add_edge(start, end)` | 固定路由 | 简单线性流程 | 高 | 低 |
| **条件边** | `add_conditional_edges(source, path, path_map)` | 动态路由 | 条件分支、循环控制 | 中 | 中 |
| **等待边** | `add_edge([node1, node2], target)` | 固定路由 + 同步 | 并行等待、汇聚点 | 中 | 中 |
| **序列边** | `add_sequence([node1, node2, ...])` | 固定路由 | 线性流程语法糖 | 高 | 低 |
| **动态路由** | `Send(node, state)` | 动态路由 + 自定义状态 | 动态并行、map-reduce | 中 | 高 |

## 边的选择策略

### 决策树

```
需要动态决策？
├─ 是 → 需要动态并行？
│   ├─ 是 → 使用条件边 + Send
│   └─ 否 → 使用条件边
└─ 否 → 需要等待多个节点？
    ├─ 是 → 使用等待边
    └─ 否 → 是线性流程？
        ├─ 是 → 使用序列边
        └─ 否 → 使用普通边
```

### 选择原则

1. **简单优先**: 优先使用普通边和序列边，除非需要动态决策
2. **类型一致**: 避免条件边和普通边同时指向同一节点
3. **性能考虑**: 条件边和动态路由有额外开销，仅在必要时使用
4. **可维护性**: 使用类型注解提升代码可读性和可维护性

## 常见误区预警

### 误区1: 条件边和普通边混用
**错误示例**:
```python
builder.add_edge("A", "B")  # 普通边
builder.add_conditional_edges("A", route_func, {"path1": "B", "path2": "C"})  # 条件边
```
**问题**: 节点 B 可能被执行两次
**正确做法**: 只使用条件边或只使用普通边

### 误区2: Command 覆盖预定义边
**错误理解**: 认为 Command 可以覆盖所有预定义边
**正确理解**: Command 无法覆盖通过 `add_edge` 显式定义的边

### 误区3: 条件边路由函数的状态类型
**错误理解**: 认为路由函数总是接收完整状态
**正确理解**: 路由函数接收的状态类型取决于 input_schema

### 误区4: defer 节点与条件边组合
**错误理解**: 认为 defer 节点可以与任何边类型组合
**正确理解**: defer 节点与条件边组合时需要特别注意

### 误区5: 条件边的可视化
**错误理解**: 认为条件边的可视化会自动正确
**正确理解**: 需要类型注解或 path_map 才能正确可视化
