# 实战代码 - 场景2：多节点状态流转

> **学习目标**：通过实战代码理解状态在多个节点间的传递、累积和转换机制

---

## 场景概述

**实战场景**：文章生成工作流

模拟一个完整的内容创作流程，包含多个顺序执行的节点：
1. **主题生成节点**：生成文章主题
2. **大纲规划节点**：根据主题生成大纲
3. **内容撰写节点**：根据大纲撰写内容
4. **质量审核节点**：审核并改进内容

**核心要点**：
- 状态在节点间顺序传递
- 每个节点只返回部分状态更新
- 状态自动累积和合并
- 完整的执行链路追踪

**来源**：基于 Context7 官方文档和 Reddit 社区实践案例

---

## 双重类比

### 前端开发类比

```javascript
// 前端：Redux 数据流
const initialState = { topic: null, outline: null, content: null };

// 类似 LangGraph 的节点函数
const generateTopic = (state) => ({ ...state, topic: "AI 技术" });
const generateOutline = (state) => ({ ...state, outline: ["引言", "正文"] });
const writeContent = (state) => ({ ...state, content: "文章内容..." });

// 顺序执行
let state = initialState;
state = generateTopic(state);    // 更新 topic
state = generateOutline(state);  // 更新 outline
state = writeContent(state);     // 更新 content
```

**对应关系**：
- LangGraph State ≈ Redux Store
- 节点函数 ≈ Action Creator
- add_edge ≈ 中间件链
- 部分状态返回 ≈ Partial Update

### 日常生活类比

**类比：餐厅后厨流程**

```
原材料准备站 → 切配站 → 烹饪站 → 摆盘站 → 出餐
    ↓            ↓        ↓        ↓        ↓
  准备食材    切好配菜   炒好菜品   装盘完成   上桌
```

**对应关系**：
- 每个工作站 = 一个节点
- 菜品状态 = State（从生到熟的转变）
- 传菜员 = 边（Edge）
- 每站只负责自己的工作 = 部分状态更新

## 完整实战代码

**来源**：基于 Context7 官方文档示例改编

```python
"""
多节点状态流转实战：文章生成工作流
演示：状态在多个节点间的顺序传递和累积

环境要求：
- Python 3.13+
- langgraph
- langchain-openai (可选，用于真实 LLM 调用)
"""

import os
from typing import TypedDict, NotRequired, Literal
from typing_extensions import Annotated
from langgraph.graph import StateGraph, START, END
from datetime import datetime


# ===== 1. 定义状态 Schema =====
class ArticleState(TypedDict):
    """
    文章生成状态

    来源：Context7 官方文档 - State 定义规范
    """
    # 主题（第一个节点生成）
    topic: NotRequired[str]

    # 大纲（第二个节点生成）
    outline: NotRequired[list[str]]

    # 内容（第三个节点生成）
    content: NotRequired[str]

    # 审核结果（第四个节点生成）
    review: NotRequired[dict]

    # 执行日志（所有节点累积）
    logs: Annotated[list[str], lambda x, y: x + y]  # 使用 reducer 累积日志


# ===== 2. 定义节点函数 =====

def generate_topic(state: ArticleState) -> dict:
    """
    节点1：生成文章主题

    输入：空状态或初始状态
    输出：部分状态更新（只返回 topic 和 logs）

    来源：Context7 - 节点函数定义规范
    """
    print("\n[节点1] 正在生成主题...")

    # 模拟主题生成（实际可调用 LLM）
    topic = "LangGraph 状态管理深度解析"

    # 只返回部分状态更新
    return {
        "topic": topic,
        "logs": [f"[{datetime.now().strftime('%H:%M:%S')}] 生成主题: {topic}"]
    }


def generate_outline(state: ArticleState) -> dict:
    """
    节点2：根据主题生成大纲

    输入：包含 topic 的状态
    输出：部分状态更新（只返回 outline 和 logs）

    来源：Context7 - 节点间状态传递示例
    """
    print(f"\n[节点2] 正在为主题 '{state['topic']}' 生成大纲...")

    # 基于主题生成大纲
    outline = [
        "1. StateGraph 核心概念",
        "2. State Schema 定义",
        "3. 节点函数协议",
        "4. 状态传递机制",
        "5. 实战案例分析"
    ]

    return {
        "outline": outline,
        "logs": [f"[{datetime.now().strftime('%H:%M:%S')}] 生成大纲: {len(outline)} 个章节"]
    }


def write_content(state: ArticleState) -> dict:
    """
    节点3：根据大纲撰写内容

    输入：包含 topic 和 outline 的状态
    输出：部分状态更新（只返回 content 和 logs）

    来源：Context7 - 多节点协作示例
    """
    print(f"\n[节点3] 正在撰写内容...")

    # 基于大纲生成内容
    content_parts = []
    for section in state["outline"]:
        content_parts.append(f"\n## {section}\n\n这是关于 {section} 的详细内容...\n")

    content = "".join(content_parts)

    return {
        "content": content,
        "logs": [f"[{datetime.now().strftime('%H:%M:%S')}] 撰写内容: {len(content)} 字符"]
    }


def review_content(state: ArticleState) -> dict:
    """
    节点4：审核内容质量

    输入：包含完整状态（topic, outline, content）
    输出：部分状态更新（只返回 review 和 logs）

    来源：Reddit 社区 - 质量审核模式
    """
    print(f"\n[节点4] 正在审核内容...")

    # 模拟内容审核
    review = {
        "quality_score": 85,
        "suggestions": [
            "增加代码示例",
            "补充实战案例",
            "优化章节结构"
        ],
        "approved": True
    }

    return {
        "review": review,
        "logs": [f"[{datetime.now().strftime('%H:%M:%S')}] 审核完成: 质量分 {review['quality_score']}"]
    }


# ===== 3. 构建图 =====

def build_article_workflow() -> StateGraph:
    """
    构建文章生成工作流图

    来源：Context7 - StateGraph 构建模式
    """
    # 创建 StateGraph
    builder = StateGraph(ArticleState)

    # 添加节点
    builder.add_node("generate_topic", generate_topic)
    builder.add_node("generate_outline", generate_outline)
    builder.add_node("write_content", write_content)
    builder.add_node("review_content", review_content)

    # 添加边：定义顺序执行流程
    # 来源：Context7 - 顺序执行模式
    builder.add_edge(START, "generate_topic")
    builder.add_edge("generate_topic", "generate_outline")
    builder.add_edge("generate_outline", "write_content")
    builder.add_edge("write_content", "review_content")
    builder.add_edge("review_content", END)

    return builder


# ===== 4. 执行工作流 =====

def main():
    """主函数：执行完整的文章生成工作流"""

    print("=" * 60)
    print("多节点状态流转实战：文章生成工作流")
    print("=" * 60)

    # 构建并编译图
    builder = build_article_workflow()
    graph = builder.compile()

    # 执行图（初始状态为空）
    print("\n开始执行工作流...\n")
    result = graph.invoke({"logs": []})  # 初始化 logs 为空列表

    # 输出最终状态
    print("\n" + "=" * 60)
    print("执行完成！最终状态：")
    print("=" * 60)

    print(f"\n【主题】\n{result['topic']}")

    print(f"\n【大纲】")
    for item in result['outline']:
        print(f"  - {item}")

    print(f"\n【内容预览】\n{result['content'][:200]}...")

    print(f"\n【审核结果】")
    print(f"  质量分: {result['review']['quality_score']}")
    print(f"  是否通过: {'✅ 通过' if result['review']['approved'] else '❌ 未通过'}")
    print(f"  改进建议:")
    for suggestion in result['review']['suggestions']:
        print(f"    - {suggestion}")

    print(f"\n【执行日志】")
    for log in result['logs']:
        print(f"  {log}")

    print("\n" + "=" * 60)


# ===== 5. 状态流转可视化 =====

def visualize_state_flow():
    """
    可视化状态在节点间的流转过程

    来源：Twitter 最佳实践 - 状态追踪模式
    """
    print("\n" + "=" * 60)
    print("状态流转可视化")
    print("=" * 60)

    builder = build_article_workflow()
    graph = builder.compile()

    # 逐步执行并追踪状态变化
    state = {"logs": []}

    print("\n初始状态:")
    print(f"  {state}")

    # 手动模拟节点执行（实际由 LangGraph 自动处理）
    nodes = [
        ("generate_topic", generate_topic),
        ("generate_outline", generate_outline),
        ("write_content", write_content),
        ("review_content", review_content)
    ]

    for node_name, node_func in nodes:
        print(f"\n执行节点: {node_name}")
        update = node_func(state)
        state.update(update)
        print(f"  状态更新: {list(update.keys())}")
        print(f"  当前状态字段: {list(state.keys())}")


if __name__ == "__main__":
    # 执行主工作流
    main()

    # 可视化状态流转
    visualize_state_flow()
```

## 核心知识点解析

### 1. 部分状态返回机制

**来源**：Context7 官方文档 - State 更新规范

```python
# ❌ 错误：返回完整状态
def node(state: State) -> State:
    return {
        "field1": state["field1"],  # 不必要的重复
        "field2": state["field2"],  # 不必要的重复
        "field3": "new_value"       # 实际更新
    }

# ✅ 正确：只返回需要更新的字段
def node(state: State) -> dict:
    return {
        "field3": "new_value"  # 只返回更新的字段
    }
```

**关键点**：
- 节点函数只需返回需要更新的字段
- LangGraph 自动合并到完整状态
- 减少代码冗余，提高可维护性

### 2. Reducer 函数累积状态

**来源**：Context7 官方文档 - Annotated 类型使用

```python
from typing_extensions import Annotated

class State(TypedDict):
    # 使用 lambda 定义 reducer：累积列表
    logs: Annotated[list[str], lambda x, y: x + y]
```

**工作原理**：
```python
# 初始状态
state = {"logs": ["log1"]}

# 节点1 返回
update1 = {"logs": ["log2"]}
# 合并后：logs = ["log1"] + ["log2"] = ["log1", "log2"]

# 节点2 返回
update2 = {"logs": ["log3"]}
# 合并后：logs = ["log1", "log2"] + ["log3"] = ["log1", "log2", "log3"]
```

### 3. 顺序执行模式

**来源**：Context7 官方文档 - 边的连接模式

```python
# 线性执行链
builder.add_edge(START, "node1")
builder.add_edge("node1", "node2")
builder.add_edge("node2", "node3")
builder.add_edge("node3", END)
```

**执行流程**：
```
START → node1 → node2 → node3 → END
  ↓       ↓       ↓       ↓       ↓
 初始   更新1   更新2   更新3   最终
```

### 4. NotRequired 字段

**来源**：Context7 官方文档 - TypedDict 最佳实践

```python
from typing import NotRequired

class State(TypedDict):
    # 可选字段：初始状态可以不包含
    topic: NotRequired[str]
    outline: NotRequired[list[str]]
```

**优势**：
- 初始状态可以为空
- 节点逐步填充字段
- 类型安全检查

## 实战要点总结

### 1. 状态传递规则

**来源**：Twitter 最佳实践

| 规则 | 说明 | 示例 |
|------|------|------|
| **只读输入** | 节点接收完整状态 | `state['topic']` |
| **部分输出** | 节点只返回更新字段 | `return {"topic": "..."}` |
| **自动合并** | LangGraph 自动合并状态 | 无需手动合并 |
| **类型安全** | TypedDict 提供类型检查 | IDE 自动补全 |

### 2. 节点设计原则

**来源**：Reddit 社区最佳实践

1. **单一职责**：每个节点只做一件事
2. **无副作用**：不修改输入状态
3. **幂等性**：相同输入产生相同输出
4. **可测试性**：节点函数独立可测试

### 3. 常见陷阱

**来源**：Reddit 社区常见问题

| 陷阱 | 问题 | 解决方案 |
|------|------|----------|
| **返回完整状态** | 代码冗余 | 只返回更新字段 |
| **修改输入状态** | 副作用 | 返回新字典 |
| **忘记 NotRequired** | 类型错误 | 使用 NotRequired |
| **忽略 Reducer** | 状态覆盖 | 使用 Annotated |
