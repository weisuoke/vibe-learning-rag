# 第一性原理

> 从"模型 + 工具 + 循环"推导 create_agent 的统一抽象设计

---

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而非依赖类比和经验。

物理学家费曼说："把事物拆解到最基本的真理，然后从那里开始推理。"

在 Agent 系统设计中，第一性原理意味着：
- 不去看"别人怎么做 Agent"
- 而是从"Agent 到底要解决什么问题"出发
- 推导出最优的 API 设计

LangChain 团队正是用这种思维，从三年的 Agent 抽象演进中提炼出了 `create_agent()` 这个统一方案。

---

## create_agent 的第一性原理

### 1. 最基础的定义

**Agent = 模型 + 工具 + 循环决策**

`create_agent()` 将这三者组合成一个可执行的状态图。仅此而已，没有更基础的了。

拆开来看：
- **模型**：负责"思考"，决定下一步做什么
- **工具**：负责"执行"，与外部世界交互
- **循环决策**：模型不断判断"任务完成了吗？"，没完成就继续调用工具

```
用户输入
  ↓
┌─────────────────────────────┐
│  模型思考：需要调用工具吗？    │ ←──────┐
└─────────┬───────────────────┘        │
          │                            │
     ┌────┴─────┐                      │
     │ 有tool_calls │                    │
     └────┬─────┘                      │
          ↓                            │
   ┌───────────────┐                   │
   │  执行工具，返回结果 │ ──────────────┘
   └───────────────┘
          │
     ┌────┴─────┐
     │ 无tool_calls │
     └────┬─────┘
          ↓
      最终输出
```

这个循环就是 Agent 的全部本质。`create_agent()` 做的事情，就是把这个循环封装成一个 LangGraph 状态图。

---

### 2. 为什么需要 create_agent？

**核心问题：上下文工程（Context Engineering）**

LangChain 博客原文指出：

> "模型的输入决定输出质量。简单的 Agent 状态和循环适合入门，但随着复杂度增加，开发者需要更多控制力。"

LLM 本身只能"说话"，不能"做事"。要让它做事，需要：
1. 告诉它有哪些工具可用
2. 让它决定调用哪个工具
3. 执行工具并把结果反馈给它
4. 让它继续决策，直到任务完成

这个需求催生了 Agent 系统。但在 LangChain 1.0 之前，实现 Agent 的方式极度碎片化：

```python
# 旧版：10+ 种创建方式，每种有不同的参数和行为
from langchain.agents import initialize_agent          # 最早的方式
from langchain.agents import create_react_agent         # ReAct 模式
from langchain.agents import create_openai_functions_agent  # OpenAI Functions
from langchain.agents import create_tool_calling_agent  # Tool Calling
from langchain.agents import create_structured_chat_agent   # 结构化聊天
from langchain.agents import create_json_agent          # JSON 模式
from langchain.agents import create_xml_agent           # XML 模式
# ... 还有更多
```

**旧方案的演进与困境**：

LangChain 团队在过去两年做了大量改进（按时间顺序）：
- 允许用户指定运行时配置
- 允许任意状态 schema
- 允许函数返回 prompt（动态 prompt）
- 允许函数返回消息列表（完全控制）
- 允许 pre model hook（模型调用前）
- 允许 post model hook（模型调用后）
- 允许函数返回模型（动态模型切换）

**问题暴露了**：参数越来越多，参数之间有依赖关系，难以组合。

| 问题 | 具体表现 |
|------|----------|
| API 碎片化 | 10+ 种 `create_xxx_agent()`，选哪个？ |
| 参数爆炸 | 每次加新功能就加新参数，函数签名越来越长 |
| 难以组合 | 想同时用"动态 prompt"和"模型降级"？参数之间可能冲突 |
| 缺乏统一抽象 | 每种 Agent 的参数、行为、返回值都不同 |

**推理链**：

```
旧方案用参数堆叠解决定制需求
  ↓
参数越来越多 → 参数爆炸
  ↓
参数之间有依赖 → 难以组合
  ↓
需要一种新的扩展机制 → 不是加参数，而是加"层"
  ↓
借鉴 Web 服务器的 Middleware 模式
  ↓
新方案：create_agent() + Middleware → 可组合、可复用
```

**新方案：一个函数 + middleware 组合 = 无限可能**

```python
# 新版：一个入口，解决所有问题
from langchain.agents import create_agent

agent = create_agent(
    model="openai:gpt-4o",
    tools=[search, calculator],
    middleware=[LoggingMiddleware(), RateLimitMiddleware()],
)
```

### 3. create_agent 的三层价值

#### 价值 1：统一入口（降低认知负担）

**定义**：整个 Agent 系统只暴露一个公开 API —— `create_agent()`。

**为什么重要**：
- 不用纠结"该用哪个 create 函数"
- 新手 5 分钟就能创建第一个 Agent
- 团队代码风格统一，review 成本低
- 模型无关：支持字符串标识符如 `"openai:gpt-4o"`，不需要手动初始化模型对象

```python
# 以前：需要根据模型能力选择不同的创建函数
# "我用的是 GPT-4，该用 create_openai_functions_agent 还是 create_tool_calling_agent？"

# 现在：一个函数搞定，模型用字符串指定
agent = create_agent(model="openai:gpt-4o", tools=[...])
```

**类比**：
- 就像 React 从 `createClass` / `React.Component` / `PureComponent` 统一到函数组件 + Hooks
- 就像餐厅从"中餐部、西餐部、日料部"合并成一个"全品类厨房"

#### 价值 2：Middleware 可组合性（灵活定制）

**定义**：通过 middleware 拦截 Agent 循环的每个步骤，实现任意定制。

**为什么重要**：
- 不改核心代码就能添加功能
- middleware 可以自由组合、复用
- 像 Express.js 中间件一样，进入时顺序执行，返回时逆序执行
- 每个 middleware 独立、可复用
- 21 个内置 middleware 覆盖常见需求

```python
from langchain.agents.middleware import (
    SummarizationMiddleware,    # 长对话自动摘要
    HumanInTheLoopMiddleware,   # 人工审批
    ModelFallbackMiddleware,    # 模型降级
    PIIMiddleware,              # PII 脱敏
)

agent = create_agent(
    model="openai:gpt-4o",
    tools=[...],
    middleware=[
        PIIMiddleware(),              # 第一层：脱敏
        SummarizationMiddleware(),    # 第二层：摘要
        HumanInTheLoopMiddleware(),   # 第三层：人工审批
        ModelFallbackMiddleware(),    # 第四层：模型降级
    ],
)
```

**类比**：
- 就像 Express.js 的 `app.use(middleware)` —— 请求经过一层层中间件
- 就像快递包裹经过"打包 → 安检 → 分拣 → 运输"的流水线

#### 价值 3：LangGraph 运行时（生产级能力）

**定义**：底层基于 LangGraph StateGraph，返回 `CompiledStateGraph`。

**为什么重要**：
- 天然支持循环（Agent 的核心就是循环）
- 自动获得持久化、检查点、中断等能力
- 支持 human-in-the-loop（暂停/恢复）
- 可以无缝降级到 LangGraph 获取更多控制
- 可视化调试

```python
from langgraph.checkpoint.memory import InMemorySaver

agent = create_agent(
    model="openai:gpt-4o",
    tools=[...],
    checkpointer=InMemorySaver(),  # 状态持久化
)

# 第一轮对话
result1 = agent.invoke(
    {"messages": [{"role": "user", "content": "我叫小明"}]},
    config={"configurable": {"thread_id": "user-001"}},
)

# 第二轮对话 —— Agent 记得你是谁
result2 = agent.invoke(
    {"messages": [{"role": "user", "content": "我叫什么？"}]},
    config={"configurable": {"thread_id": "user-001"}},
)
# Agent 回答："你叫小明"
```

**类比**：
- 就像 Redux 的状态管理 —— 每个 action 都有记录，可以回溯
- 就像医院的病历系统 —— 每次就诊都有记录，下次来不用重新描述病情

### 4. 从第一性原理推导 create_agent 设计

**完整推理链**：

```
1. LLM 能理解自然语言
   ↓
2. 给 LLM 工具描述，它能决定调用哪个工具
   ↓
3. 执行工具，获取真实世界的结果
   ↓
4. 将结果反馈给 LLM，让它继续判断
   ↓
5. 重复 2-4，直到 LLM 认为任务完成
   ↓ → 这就是 Agent 循环
6. Agent 需要循环 → 需要状态管理 → AgentState
   ↓
7. 不同场景需要不同控制 → 需要扩展点 → Middleware hooks
   ↓
8. 控制需要可组合 → 需要中间件模式 → AgentMiddleware
   ↓
9. 需要生产级能力 → 需要持久化运行时 → LangGraph
   ↓
10. 需要简单入口 → 需要工厂函数 → create_agent()
```

**逐步推导**：

**步骤 1**：LLM 只能生成文本
```python
# LLM 的本质能力
response = llm.invoke("今天北京天气怎么样？")
# → "我无法获取实时天气信息..."  （它只能"说"，不能"做"）
```

**步骤 2**：给 LLM 工具描述
```python
# 告诉 LLM 它有一个天气查询工具
tools = [{"name": "get_weather", "description": "查询城市天气", "parameters": {...}}]
response = llm.bind_tools(tools).invoke("今天北京天气怎么样？")
# → AIMessage(tool_calls=[{"name": "get_weather", "args": {"city": "北京"}}])
```

**步骤 3**：执行工具
```python
# 根据 LLM 的决策，执行对应工具
tool_result = get_weather(city="北京")
# → "北京今天晴，25°C"
```

**步骤 4**：反馈结果，继续决策
```python
# 把工具结果告诉 LLM
messages.append(ToolMessage(content="北京今天晴，25°C"))
response = llm.invoke(messages)
# → "北京今天天气晴朗，气温25°C，适合出行。"  （没有 tool_calls → 任务完成）
```

**步骤 5**：封装成循环
```python
# 这就是 create_agent 内部做的事
while True:
    response = model.invoke(messages)
    if not response.tool_calls:  # 没有工具调用 → 结束
        break
    for tool_call in response.tool_calls:
        result = execute_tool(tool_call)
        messages.append(result)
```

**步骤 6**：加上状态管理和 middleware
```python
# create_agent 把上面的循环变成 LangGraph 状态图
# 并在每个步骤插入 middleware 拦截点
agent = create_agent(
    model=model,
    tools=tools,
    middleware=middlewares,      # 拦截点
    checkpointer=checkpointer,  # 状态持久化
)
```

**推导总结图**：

```
┌──────────────────────────────────────────────────────────┐
│                    第一性原理推导                          │
│                                                          │
│  LLM 能理解语言 ──→ 给它工具描述 ──→ 它能决策            │
│       ↓                                                  │
│  决策 + 执行 + 反馈 = 循环 ──→ 需要状态管理              │
│       ↓                                                  │
│  不同场景需要不同控制 ──→ 需要可组合的扩展机制            │
│       ↓                                                  │
│  参数堆叠行不通 ──→ 借鉴 Middleware 模式                 │
│       ↓                                                  │
│  最终方案：create_agent() + Middleware + LangGraph       │
└──────────────────────────────────────────────────────────┘
```

### 5. 一句话总结第一性原理

**create_agent 是对"模型 + 工具 + 循环"这一 Agent 本质的标准化封装，通过 Middleware 模式解决了上下文工程的可组合性问题。**

## 与旧版 Agent 系统的对比

### 设计哲学的转变

| 维度 | 旧版（0.x） | 新版（1.0+） |
|------|-------------|-------------|
| 创建方式 | 10+ 种 `create_xxx_agent()` | 1 个 `create_agent()` |
| 定制方式 | 参数堆叠 + 继承重写 | Middleware 组合 |
| 运行时 | AgentExecutor（线性） | LangGraph StateGraph（图） |
| 状态管理 | Memory 对象（有限） | Checkpointer（完整持久化） |
| 类型安全 | 弱 | 强（Generic + TypedDict） |
| 扩展机制 | 加参数 | 加 Middleware |

### 从碎片化到统一

```python
# ===== 旧版：选择困难症 =====
# "我该用哪个？它们有什么区别？"
agent1 = initialize_agent(tools, llm, agent=AgentType.ZERO_SHOT_REACT_DESCRIPTION)
agent2 = create_react_agent(llm, tools, prompt)
agent3 = create_openai_functions_agent(llm, tools, prompt)
agent4 = create_tool_calling_agent(llm, tools, prompt)

# ===== 新版：一个就够了 =====
agent = create_agent(model="openai:gpt-4o", tools=tools)
```

## Middleware 的洋葱模型

Middleware 的执行顺序遵循洋葱模型 —— 外层先进后出：

```
请求进入 →
  ┌─ Middleware A: before_model ─┐
  │  ┌─ Middleware B: before_model ─┐
  │  │                              │
  │  │     模型调用（核心）          │
  │  │                              │
  │  └─ Middleware B: after_model ──┘
  └─ Middleware A: after_model ──┘
← 响应返回
```

这意味着：
- `before_model` 按 A → B 顺序执行
- `after_model` 按 B → A 顺序执行
- 每个 middleware 都能拦截请求和响应

```python
# 6 对钩子，覆盖 Agent 循环的每个阶段
class AgentMiddleware:
    def before_agent(self, state, runtime): ...   # Agent 开始前
    def after_agent(self, state, runtime): ...    # Agent 结束后
    def before_model(self, state, runtime): ...   # 模型调用前
    def after_model(self, state, runtime): ...    # 模型调用后
    def wrap_model_call(self, request, handler): ...  # 包装模型调用
    def wrap_tool_call(self, request, handler): ...   # 包装工具调用
```

## 实践建议

### 从第一性原理思考 Agent 设计

**问题**：我需要构建一个客服 Agent，应该怎么设计？

**第一性原理思考**：
1. 客服的本质是什么？→ 理解用户问题，查找答案，给出回复
2. 需要哪些工具？→ 知识库检索、工单系统、订单查询
3. 需要哪些定制？→ PII 脱敏、人工转接、对话摘要
4. 需要状态管理吗？→ 是，多轮对话需要记住上下文

**结论**：

```python
agent = create_agent(
    model="openai:gpt-4o",
    tools=[search_kb, query_order, create_ticket],
    system_prompt="你是一个专业的客服助手...",
    middleware=[
        PIIMiddleware(),
        SummarizationMiddleware(max_messages=20),
        HumanInTheLoopMiddleware(handoff_keywords=["转人工"]),
    ],
    checkpointer=PostgresSaver(),  # 持久化对话状态
)
```

## 总结

### create_agent 的第一性原理

1. **最基础定义**：Agent = 模型 + 工具 + 循环决策
2. **核心问题**：上下文工程 —— 模型的输入决定输出质量，需要细粒度控制
3. **旧方案困境**：参数堆叠 → 参数爆炸 → 难以组合
4. **新方案突破**：Middleware 模式 → 可组合、可复用、职责单一
5. **三层价值**：统一入口、Middleware 组合、LangGraph 驱动

### 关键洞察

- **统一优于碎片**：一个 `create_agent()` 胜过十个 `create_xxx_agent()`
- **组合优于继承**：middleware 组合比子类重写更灵活
- **加层优于加参数**：Middleware 模式比参数堆叠更可维护
- **状态图优于线性执行**：LangGraph 天然支持 Agent 的循环本质
