# 动态图构建 - 30字核心

**动态图构建是通过条件边、Send 和 Command 让工作流在运行时自主选择执行路径的能力。**

---

## 核心要素拆解

### 1. 条件边（Conditional Edges）—— 运行时路径选择

**定义**: 通过 `add_conditional_edges(source, path_function)` 注册一个路由函数，在运行时根据当前状态返回目标节点名

**关键机制**:
- 路由函数接收 `state` 参数，返回字符串（节点名）或字符串列表（多路并行）
- 编译时注册所有可能的目标节点，运行时由函数动态选择
- 内部通过 `BranchSpec` 封装路由逻辑，编译后成为 Pregel 执行图的一部分

**价值**: 最常用的动态路由方式，覆盖 80% 的"根据条件走不同路径"需求

```python
from langgraph.graph import StateGraph, START, END

def route_function(state):
    """运行时决定下一个节点"""
    if state["score"] > 0.8:
        return "high_quality"
    elif state["score"] > 0.5:
        return "medium_quality"
    return "low_quality"

builder = StateGraph(State)
builder.add_conditional_edges("evaluator", route_function)
```

---

### 2. Send API —— 运行时动态并行

**定义**: `Send(node_name, arg)` 在运行时创建指向目标节点的动态边，每个 Send 实例携带独立的状态参数

**关键机制**:
- 路由函数返回 `Send` 对象列表，每个对象触发一个并行任务
- 任务数量完全由运行时数据决定（编译时未知）
- 通过 Pregel 的 TASKS 通道调度，遵循 BSP（批量同步并行）执行模型
- 并行结果通过 reducer（如 `operator.add`）合并到主状态

**价值**: 解决"编译时不知道要并行多少个任务"的问题，是 map-reduce 模式的核心

```python
from langgraph.types import Send

def fan_out(state):
    """运行时创建 N 个并行任务"""
    return [
        Send("worker", {"task": item})
        for item in state["items"]  # 数量运行时才知道
    ]
```

---

### 3. Command API —— 状态更新 + 路由控制

**定义**: 节点返回 `Command(update=..., goto=...)` 在一次操作中同时更新状态和指定下一个节点

**关键机制**:
- `update` 字段更新图状态（等价于普通节点返回值）
- `goto` 字段指定下一个节点（字符串、Send 对象或列表）
- `graph=Command.PARENT` 可以导航到父图（跨子图通信）
- 类型注解 `Command[Literal["a", "b"]]` 帮助图可视化工具识别可能的目标

**价值**: 最灵活的动态控制方式，一个 API 同时解决"改什么"和"去哪里"两个问题

```python
from langgraph.types import Command
from typing import Literal

def smart_node(state) -> Command[Literal["approve", "reject"]]:
    score = evaluate(state["content"])
    if score > 0.8:
        return Command(update={"score": score}, goto="approve")
    return Command(update={"score": score}, goto="reject")
```

### 4. 在 LangGraph 中的价值

**为什么工作流需要"动态"？**

静态图（所有路径编译时固定）无法应对真实 AI 应用的需求：

| 场景 | 静态图的局限 | 动态图的解决方案 |
|------|-------------|-----------------|
| 意图分类 | 无法根据用户输入走不同路径 | 条件边 + 路由函数 |
| 批量处理 | 无法处理未知数量的并行任务 | Send API map-reduce |
| Agent 决策 | 无法让 LLM 自主选择下一步 | LLM + 结构化输出 + 条件边 |
| 多代理切换 | 无法在子图间动态导航 | Command + PARENT |
| 错误回退 | 无法根据执行结果动态调整 | 条件边 + 循环 |

**架构优势**:
- 编译时定义所有可能性 → 可验证、可可视化
- 运行时选择实际路径 → 灵活、自适应
- 三种机制互补 → 覆盖从简单路由到复杂编排的全部场景

## 一句话记忆

**条件边选路径，Send 开并行，Command 管全局 —— 三板斧让图"活"起来。**

---

## 核心公式

```
动态图构建 = 编译时定义可能性 + 运行时选择路径
```

三种机制的定位：

```
条件边（Conditional Edges）
  ├── 定位：最常用的路由方式
  ├── 输入：state → 返回节点名
  ├── 场景：if/else 分支、意图路由、LLM 决策
  └── 特点：编译时注册，运行时选择

Send API
  ├── 定位：动态并行（map-reduce）
  ├── 输入：state → 返回 Send 对象列表
  ├── 场景：批量处理、多文档并行、动态 fan-out
  └── 特点：任务数量运行时决定

Command API
  ├── 定位：状态 + 路由一体化控制
  ├── 输入：节点内部返回 Command 对象
  ├── 场景：多代理切换、复杂决策、跨图导航
  └── 特点：update + goto + graph 三合一
```

## 快速示例

### 最简示例：条件边实现动态路由

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

# ========== 1. 定义状态 ==========
class State(TypedDict):
    query: str
    intent: str
    response: str

# ========== 2. 定义节点 ==========
def classify(state: State):
    """分类节点：判断用户意图"""
    query = state["query"].lower()
    if "价格" in query or "多少钱" in query:
        return {"intent": "pricing"}
    elif "怎么用" in query or "教程" in query:
        return {"intent": "tutorial"}
    return {"intent": "general"}

def pricing_handler(state: State):
    return {"response": f"关于价格的回答: {state['query']}"}

def tutorial_handler(state: State):
    return {"response": f"关于教程的回答: {state['query']}"}

def general_handler(state: State):
    return {"response": f"通用回答: {state['query']}"}

# ========== 3. 路由函数 ==========
def route_by_intent(state: State) -> str:
    """根据意图路由到对应处理节点"""
    intent_map = {
        "pricing": "pricing_handler",
        "tutorial": "tutorial_handler",
        "general": "general_handler",
    }
    return intent_map.get(state["intent"], "general_handler")

# ========== 4. 构建图 ==========
builder = StateGraph(State)

# 添加节点
builder.add_node("classify", classify)
builder.add_node("pricing_handler", pricing_handler)
builder.add_node("tutorial_handler", tutorial_handler)
builder.add_node("general_handler", general_handler)

# 添加边
builder.add_edge(START, "classify")
builder.add_conditional_edges("classify", route_by_intent)  # 动态路由！
builder.add_edge("pricing_handler", END)
builder.add_edge("tutorial_handler", END)
builder.add_edge("general_handler", END)

# ========== 5. 编译并运行 ==========
graph = builder.compile()

# 测试：不同输入走不同路径
r1 = graph.invoke({"query": "这个产品多少钱？"})
print(r1["response"])  # 关于价格的回答: 这个产品多少钱？

r2 = graph.invoke({"query": "怎么用这个功能？"})
print(r2["response"])  # 关于教程的回答: 怎么用这个功能？

r3 = graph.invoke({"query": "你好"})
print(r3["response"])  # 通用回答: 你好
```

### Send 示例：动态并行处理

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send
import operator

class OverallState(TypedDict):
    subjects: list[str]
    jokes: Annotated[list[str], operator.add]  # reducer 合并并行结果

class WorkerState(TypedDict):
    subject: str

def generate_topics(state: OverallState):
    return {"subjects": ["猫", "程序员", "咖啡"]}

def fan_out_to_workers(state: OverallState):
    """运行时动态创建并行任务"""
    return [
        Send("generate_joke", {"subject": s})
        for s in state["subjects"]
    ]

def generate_joke(state: WorkerState):
    return {"jokes": [f"关于{state['subject']}的笑话"]}

# 构建图
builder = StateGraph(OverallState)
builder.add_node("generate_topics", generate_topics)
builder.add_node("generate_joke", generate_joke)

builder.add_edge(START, "generate_topics")
builder.add_conditional_edges("generate_topics", fan_out_to_workers)
builder.add_edge("generate_joke", END)

graph = builder.compile()
result = graph.invoke({"subjects": [], "jokes": []})
print(result["jokes"])
# ['关于猫的笑话', '关于程序员的笑话', '关于咖啡的笑话']
```

### Command 示例：状态更新 + 路由控制

```python
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.types import Command

class State(TypedDict):
    value: int
    status: str
    result: str

def evaluate(state: State) -> Command[Literal["pass_node", "fail_node"]]:
    """同时更新状态 + 决定路由"""
    if state["value"] >= 60:
        return Command(
            update={"status": "passed"},
            goto="pass_node"
        )
    return Command(
        update={"status": "failed"},
        goto="fail_node"
    )

def pass_node(state: State):
    return {"result": f"通过！分数: {state['value']}"}

def fail_node(state: State):
    return {"result": f"未通过。分数: {state['value']}，需要补考"}

builder = StateGraph(State)
builder.add_node("evaluate", evaluate)
builder.add_node("pass_node", pass_node)
builder.add_node("fail_node", fail_node)

builder.add_edge(START, "evaluate")
# 注意：使用 Command 时不需要 add_edge，路由由 Command.goto 控制
builder.add_edge("pass_node", END)
builder.add_edge("fail_node", END)

graph = builder.compile()

r1 = graph.invoke({"value": 85})
print(r1["result"])   # 通过！分数: 85
print(r1["status"])   # passed

r2 = graph.invoke({"value": 45})
print(r2["result"])   # 未通过。分数: 45，需要补考
print(r2["status"])   # failed
```

## 核心对比

### 三种动态机制对比

| 特性 | 条件边 | Send API | Command API |
|------|--------|----------|-------------|
| 注册方式 | `add_conditional_edges()` | 条件边内返回 Send 列表 | 节点返回 Command 对象 |
| 路由决策位置 | 独立的路由函数 | 独立的路由函数 | 节点内部 |
| 并行支持 | 返回列表可多路并行 | 天然支持动态并行 | goto 可以是列表 |
| 状态更新 | 不更新（纯路由） | 每个 Send 携带独立状态 | update 字段更新状态 |
| 跨图导航 | 不支持 | 不支持 | `graph=Command.PARENT` |
| 典型场景 | if/else 分支 | map-reduce 并行 | 多代理切换 |
| 复杂度 | ⭐⭐ 低 | ⭐⭐⭐ 中 | ⭐⭐⭐⭐ 高 |
| 使用频率 | 最高 | 中等 | 按需使用 |

### 静态边 vs 动态边

| 特性 | 静态边 `add_edge()` | 动态边 `add_conditional_edges()` |
|------|---------------------|--------------------------------|
| 路径确定时机 | 编译时 | 运行时 |
| 灵活性 | 固定路径 | 根据状态动态选择 |
| 可预测性 | 完全可预测 | 取决于路由函数 |
| 适用场景 | 顺序流程、固定管道 | 分支决策、动态路由 |
| 可视化 | 实线箭头 | 虚线箭头（多个可能目标） |

## 关键类比

### 前端类比：React Router 动态路由

```
条件边        →  React Router 的路由守卫（beforeEnter）
                 根据用户状态（登录/未登录）跳转不同页面
                 ↓
Send API      →  Promise.all(items.map(fetchData))
                 运行时决定并行请求数量
                 ↓
Command API   →  Redux dispatch + navigate 组合
                 一个操作同时改状态和跳页面
```

就像前端的路由系统：
- 编译时定义所有可能的路由（`/home`, `/login`, `/dashboard`）
- 运行时根据用户状态决定实际跳转哪个页面
- 路由守卫函数 = LangGraph 的路由函数

### 日常生活类比：导航 App

```
条件边        →  导航 App 根据实时路况选择路线
                 堵车走高速，不堵走国道
                 ↓
Send API      →  外卖平台同时派多个骑手取餐
                 骑手数量取决于当前订单量
                 ↓
Command API   →  快递分拣中心
                 扫码（读取状态）→ 贴标签（更新状态）→ 放到对应传送带（路由）
                 一个动作完成三件事
```

关键相似点：
- 路线（路径）在出发前就规划好了所有可能性
- 实际走哪条路取决于实时路况（运行时状态）
- 目的地不变，但到达方式是动态的

## 三个必须记住的规则

### 规则 1: 图结构编译后不可变

```python
builder = StateGraph(State)
builder.add_node("a", node_a)       # ✅ 编译前：可以添加
graph = builder.compile()            # 编译！结构冻结

# ❌ 编译后不能修改图结构
# graph.add_node("b", node_b)       # 不存在这个方法

# ✅ 但运行时路径是动态的
graph.invoke({"input": "走路径A"})   # 可能走 A → B → D
graph.invoke({"input": "走路径B"})   # 可能走 A → C → D
```

"动态"不是修改图，而是在已有的图中选择路径。

### 规则 2: Command 的 goto 不会替代静态边

```python
# ⚠️ 危险：静态边 + Command goto 会同时执行！
builder.add_edge("node_a", "node_b")  # 静态边 → 总是去 node_b

def node_a(state) -> Command[Literal["node_c"]]:
    return Command(goto="node_c")      # Command → 也去 node_c

# 结果：node_b 和 node_c 都会执行！

# ✅ 正确做法：使用 Command 时不要给该节点添加静态边
```

### 规则 3: Send 需要配合 reducer

```python
# ❌ 没有 reducer，并行结果会互相覆盖
class BadState(TypedDict):
    results: list  # 没有 reducer！

# ✅ 使用 reducer 合并并行结果
class GoodState(TypedDict):
    results: Annotated[list, operator.add]  # operator.add 合并列表
```

Send 创建的并行任务各自独立运行，结果需要通过 reducer 正确合并到主状态。

## 速记口诀

```
动态图构建三板斧：
条件边选路走，      ← add_conditional_edges + 路由函数
Send 开并行，       ← 运行时创建 N 个并行任务
Command 管全局。    ← update 改状态，goto 定方向

一个核心要记牢：
编译定可能，运行选路径。  ← 图结构不变，路径动态选

两个坑别踩：
Command 别配静态边，      ← 两者会同时执行
Send 结果要 reducer 合。  ← 否则并行结果互相覆盖
```

## 核心 API 速查

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send, Command
from typing import Literal, Annotated
import operator

# ===== 条件边 =====
# 基础用法：路由函数返回节点名
builder.add_conditional_edges("source", route_function)

# 带路径映射
builder.add_conditional_edges("source", route_function, {
    "option_a": "node_a",
    "option_b": "node_b",
})

# 多路并行（路由函数返回列表）
def multi_route(state):
    return ["node_a", "node_b"]  # 同时走两条路

# ===== Send API =====
# 动态创建并行任务
def fan_out(state):
    return [Send("worker", {"item": x}) for x in state["items"]]

builder.add_conditional_edges("splitter", fan_out)

# 状态需要 reducer
class State(TypedDict):
    results: Annotated[list, operator.add]

# ===== Command API =====
# 状态更新 + 路由
def node(state) -> Command[Literal["next_a", "next_b"]]:
    return Command(update={"key": "value"}, goto="next_a")

# 跨图导航（多代理切换）
def subgraph_node(state) -> Command[Literal["other_agent"]]:
    return Command(goto="other_agent", graph=Command.PARENT)
```
