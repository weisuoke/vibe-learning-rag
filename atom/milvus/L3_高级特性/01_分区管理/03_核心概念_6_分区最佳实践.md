# 核心概念 6：分区最佳实践

> 生产环境中的分区使用建议和优化策略

---

## 概述

**分区最佳实践**是指在生产环境中使用分区功能的推荐方法和优化策略。基于 Milvus 官方文档、社区实践和性能测试结果，本文总结了分区管理的最佳实践。

**核心价值**：
- 避免常见陷阱
- 提升系统性能
- 降低运维成本
- 保证系统稳定性

---

## 1. 自动分区 vs 手动分区选择

### 1.1 选择建议

**来源**：GitHub 最佳实践

**推荐策略**：

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 大规模多租户（> 1,024 个） | Partition Key 自动分区 | 支持百万级租户 |
| 中等数量租户（100-1,024 个） | 手动分区 | 物理隔离 + 热冷数据管理 |
| 少量租户（< 100 个） | Collection-level 多租户 | 强隔离 + Schema 灵活性 |
| 时间序列数据 | 手动分区（按时间） | 热冷数据管理 |
| 需要 RBAC | Collection-level 或 Database-level | 支持 RBAC |

### 1.2 自动分区（Partition Key）最佳实践

**推荐配置**：
```python
from pymilvus import FieldSchema, CollectionSchema, DataType, Collection

# 定义 Schema
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=384),
        FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True),
        FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=2048)
    ]
)

# 创建 Collection，配置物理分区数和启用隔离
collection = Collection(
    name="my_collection",
    schema=schema,
    properties={
        "partition.num": 100,  # 配置 100 个物理分区（推荐）
        "partitionkey.isolation": True  # 启用 Partition Key 隔离（仅 HNSW）
    }
)

# 创建 HNSW 索引（支持 Partition Key 隔离）
index_params = {
    "index_type": "HNSW",
    "metric_type": "L2",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="vector", index_params=index_params)
```

**关键配置**：
- **物理分区数**：100-200 个（生产环境推荐）
- **Partition Key 隔离**：启用（仅支持 HNSW 索引）
- **索引类型**：HNSW（支持 Partition Key 隔离）

### 1.3 手动分区最佳实践

**推荐配置**：
```python
from pymilvus import Collection

collection = Collection("my_collection")

# 按业务逻辑创建分区
for tenant_id in range(1, 101):
    collection.create_partition(f"tenant_{tenant_id}")

# 只加载需要的分区（热数据）
active_tenants = [f"tenant_{i}" for i in range(1, 11)]
collection.load(partition_names=active_tenants)
```

**关键配置**：
- **分区大小**：20-100K 行（推荐）
- **分区数量**：控制在 1,000 以内
- **热冷数据管理**：只加载需要的分区

## 2. 分区数量建议

### 2.1 Partition Key 物理分区数

**来源**：GitHub 最佳实践

**推荐配置**：

| 租户数量 | 推荐物理分区数 | 原因 |
|----------|----------------|------|
| < 1 万 | 16 个（默认） | 足够支持小规模场景 |
| 1-10 万 | 64 个 | 平衡性能和管理开销 |
| > 10 万 | 100-200 个 | 支持大规模场景 |

**配置方法**：
```python
collection = Collection(
    name="my_collection",
    schema=schema,
    properties={"partition.num": 100}  # 配置 100 个物理分区
)
```

**注意事项**：
- 物理分区数过多会增加系统压力
- 物理分区数过少会影响过滤效率
- 推荐 100-200 个物理分区

### 2.2 手动分区数量

**来源**：GitHub 最佳实践

**推荐配置**：

| 场景 | 推荐分区数 | 原因 |
|------|-----------|------|
| 多租户 | 100-200 个 | 平衡隔离和性能 |
| 时间序列 | 12-24 个（按月份） | 热冷数据管理 |
| 业务分类 | 10-50 个 | 按业务逻辑分区 |

**关键建议**：
- **总数控制**：建议控制在 1,000 以内
- **分区大小**：每个分区 20-100K 行
- **避免过多**：分区过多会导致性能下降

## 3. 分区大小建议

### 3.1 推荐分区大小

**来源**：GitHub 最佳实践

**推荐配置**：
- **最小**：10K 行
- **推荐**：20-100K 行
- **最大**：500K 行

**原因**：
- 分区过小：管理开销大，性能提升有限
- 分区过大：检索范围大，性能提升有限
- 推荐范围：平衡性能和管理开销

### 3.2 分区大小监控

**监控方法**：
```python
from pymilvus import utility, Collection

collection = Collection("my_collection")

# 获取每个分区的统计信息
for partition in collection.partitions:
    stats = utility.get_partition_stats("my_collection", partition.name)
    row_count = stats.get("row_count", 0)
    print(f"Partition: {partition.name}, Row Count: {row_count}")

    # 检查分区大小是否在推荐范围内
    if row_count < 10000:
        print(f"  ⚠️ Warning: Partition too small (< 10K rows)")
    elif row_count > 500000:
        print(f"  ⚠️ Warning: Partition too large (> 500K rows)")
    else:
        print(f"  ✅ OK: Partition size in recommended range")
```

## 4. 性能监控和调优

### 4.1 监控指标

**关键指标**：
- **QPS**：每秒查询数
- **延迟**：查询响应时间（P50, P95, P99）
- **内存使用**：加载的分区占用的内存
- **分区负载**：每个分区的数据量和访问频率

### 4.2 监控方法

**使用 Milvus 内置监控**：
```python
from pymilvus import utility, Collection
import time

collection = Collection("my_collection")

# 监控查询性能
def monitor_query_performance(query_vector, partition_names=None):
    start_time = time.time()

    results = collection.search(
        data=[query_vector],
        anns_field="vector",
        param={"metric_type": "L2", "params": {"nprobe": 10}},
        limit=10,
        partition_names=partition_names
    )

    end_time = time.time()
    latency = (end_time - start_time) * 1000  # 毫秒

    print(f"Query latency: {latency:.2f} ms")
    print(f"Results count: {len(results[0])}")

    return latency

# 对比不同分区策略的性能
print("=== 搜索所有分区 ===")
latency_all = monitor_query_performance(query_vector)

print("\n=== 搜索特定分区 ===")
latency_partition = monitor_query_performance(query_vector, partition_names=["partition_a"])

print(f"\n性能提升: {(latency_all / latency_partition):.2f}x")
```

### 4.3 性能调优策略

**策略 1：增加物理分区数**
```python
# 如果分区负载不均衡，增加物理分区数
collection = Collection(
    name="my_collection",
    schema=schema,
    properties={"partition.num": 200}  # 从 100 增加到 200
)
```

**策略 2：调整分区大小**
```python
# 如果分区过大，拆分分区
# 例如：按月份拆分为按周
collection.create_partition("2026_02_week1")
collection.create_partition("2026_02_week2")
collection.create_partition("2026_02_week3")
collection.create_partition("2026_02_week4")
```

**策略 3：优化索引参数**
```python
# 根据实际场景调整索引参数
index_params = {
    "index_type": "HNSW",
    "metric_type": "L2",
    "params": {
        "M": 32,  # 增加 M 值，提升召回率
        "efConstruction": 400  # 增加 efConstruction，提升索引质量
    }
}
```

## 5. 热冷数据管理

### 5.1 热冷数据分离策略

**场景**：按时间分区，只加载最近的数据

**最佳实践**：
```python
from pymilvus import Collection
from datetime import datetime, timedelta

collection = Collection("time_series_data")

# 创建按月份的分区
def create_monthly_partitions(months=12):
    for i in range(months):
        month = (datetime.now() - timedelta(days=30*i)).strftime("%Y_%m")
        if not collection.has_partition(month):
            collection.create_partition(month)

create_monthly_partitions(12)

# 只加载最近 3 个月的数据（热数据）
def load_hot_data(months=3):
    hot_partitions = [
        (datetime.now() - timedelta(days=30*i)).strftime("%Y_%m")
        for i in range(months)
    ]
    collection.load(partition_names=hot_partitions)
    print(f"Loaded hot data: {hot_partitions}")

load_hot_data(3)

# 定期释放冷数据
def release_cold_data(months_to_keep=3):
    all_partitions = [p.name for p in collection.partitions if p.name != "_default"]
    hot_partitions = [
        (datetime.now() - timedelta(days=30*i)).strftime("%Y_%m")
        for i in range(months_to_keep)
    ]
    cold_partitions = [p for p in all_partitions if p not in hot_partitions]

    if cold_partitions:
        collection.release(partition_names=cold_partitions)
        print(f"Released cold data: {cold_partitions}")

release_cold_data(3)
```

### 5.2 按需加载策略

**场景**：临时查询历史数据

**最佳实践**：
```python
def query_historical_data(query_vector, month):
    # 检查分区是否已加载
    partition = collection.partition(month)
    if not partition.is_loaded:
        print(f"Loading partition: {month}")
        collection.load(partition_names=[month])

    # 查询数据
    results = collection.search(
        data=[query_vector],
        anns_field="vector",
        param={"metric_type": "L2", "params": {"nprobe": 10}},
        limit=10,
        partition_names=[month]
    )

    # 查询完成后释放（可选）
    # collection.release(partition_names=[month])

    return results

# 查询 2025 年 12 月的数据
results = query_historical_data(query_vector, "2025_12")
```

## 6. 生产环境配置建议

### 6.1 推荐配置

**大规模多租户场景**：
```python
# 使用 Partition Key 自动分区
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=384),
        FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True),
        FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=2048)
    ]
)

collection = Collection(
    name="multi_tenant_collection",
    schema=schema,
    properties={
        "partition.num": 100,  # 100 个物理分区
        "partitionkey.isolation": True  # 启用 Partition Key 隔离
    }
)

# 创建 HNSW 索引
index_params = {
    "index_type": "HNSW",
    "metric_type": "COSINE",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="vector", index_params=index_params)

# 加载 Collection
collection.load()
```

**时间序列数据场景**：
```python
# 使用手动分区
```

### 6.2 避免的问题

**问题 1：分区过多**
```python
# ❌ 错误：创建过多分区
for i in range(10000):
    collection.create_partition(f"partition_{i}")
# 问题：管理开销大，性能下降

# ✅ 正确：使用 Partition Key 自动分区
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=128),
        FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64,
                   is_partition_key=True)
    ]
)
```

**问题 2：分区过大**
```python
# ❌ 错误：单个分区包含过多数据
collection.insert(data_1M, partition_name="partition_a")  # 100 万条数据
# 问题：检索范围大，性能提升有限

# ✅ 正确：拆分分区
collection.create_partition("partition_a_1")
collection.create_partition("partition_a_2")
collection.insert(data_500K, partition_name="partition_a_1")
collection.insert(data_500K, partition_name="partition_a_2")
```

**问题 3：未配置监控**
```python
# ❌ 错误：未配置监控和告警
# 问题：无法及时发现性能问题

# ✅ 正确：配置监控和告警
def monitor_partition_health():
    for partition in collection.partitions:
        stats = utility.get_partition_stats("my_collection", partition.name)
        row_count = stats.get("row_count", 0)

        if row_count > 500000:
            print(f"⚠️ Alert: Partition {partition.name} too large ({row_count} rows)")
        elif row_count < 10000:
            print(f"⚠️ Alert: Partition {partition.name} too small ({row_count} rows)")

monitor_partition_health()
```

## 7. 实际应用案例

## 核心要点总结

1. **自动分区 vs 手动分区**：根据租户数量和场景选择
2. **分区数量建议**：Partition Key 推荐 100-200 个物理分区，手动分区控制在 1,000 以内
3. **分区大小建议**：20-100K 行（推荐）
4. **性能监控**：监控 QPS、延迟、内存使用、分区负载
5. **热冷数据管理**：只加载需要的分区，节省内存
6. **生产环境配置**：根据场景选择合适的配置
