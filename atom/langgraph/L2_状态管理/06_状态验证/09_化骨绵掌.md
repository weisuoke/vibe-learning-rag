## 卡片1：什么是状态验证？

**一句话**：在数据进入系统前检查它是否符合预期。

**类比**：快递分拣中心的入口检查——包裹进入传送带之前，先过安检机，超重的退回、违禁品拦截、地址不清的打回重写。

**为什么需要？**

LangGraph 的工作流由多个节点组成，状态在节点间流转。如果入口处放进了脏数据：
- 字符串传给了期望整数的字段 → 节点报错
- 空查询传给了检索器 → 返回无意义结果
- 负数的 top_k → 向量数据库异常

**核心原则**：在入口处严格检查，内部节点就可以放心使用数据。

**代码直觉**：

```python
# 没有验证 —— 随时可能爆炸
state = {"query": "", "top_k": -1}  # 没人拦你

# 有验证 —— 入口就拦住
state = RAGState(query="", top_k=-1)  # ValidationError!
```

**2 分钟记住**：状态验证 = 系统的免疫系统，在入口处消灭病毒。

## 卡片2：TypedDict vs Pydantic

**一句话**：TypedDict 是告示牌（提醒你该放什么），Pydantic 是安检门（实际检查你放了什么）。

**对比表**：

| 特性 | TypedDict | Pydantic BaseModel |
|------|-----------|-------------------|
| 运行时验证 | ❌ 不检查 | ✅ 严格检查 |
| 类型转换 | ❌ 不转换 | ✅ 自动转换 |
| IDE 提示 | ✅ 有 | ✅ 有 |
| 性能开销 | 几乎为零 | 有验证开销 |
| Reducer 支持 | ✅ 原生支持 | ⚠️ 需要额外处理 |
| 适用场景 | 内部状态传递 | 外部输入验证 |

**代码对比**：

```python
# TypedDict —— 告示牌
from typing import TypedDict

class State(TypedDict):
    query: str
    top_k: int

state: State = {"query": 123, "top_k": "abc"}  # Python 不报错！

# Pydantic —— 安检门
from pydantic import BaseModel

class State(BaseModel):
    query: str
    top_k: int

state = State(query=123, top_k="abc")  # query 被转为 "123"，top_k 报错！
```

**选型决策**：
- 处理用户输入 / LLM 输出 → **Pydantic**
- 内部节点间传递 → **TypedDict**
- 生产环境 → **混合使用**

**2 分钟记住**：TypedDict 管"看"，Pydantic 管"查"。

## 卡片3：Pydantic BaseModel 基础

**一句话**：继承 BaseModel 就能获得运行时验证能力。

**最小示例**：

```python
from pydantic import BaseModel
from typing import Optional

class RAGState(BaseModel):
    query: str                          # 必填，字符串
    top_k: int = 5                      # 可选，默认 5
    threshold: float = 0.7              # 可选，默认 0.7
    results: Optional[list] = None      # 可选，默认 None
```

**自动获得的能力**：

```python
# 1. 类型验证
state = RAGState(query="什么是RAG", top_k="abc")  # ValidationError

# 2. 类型转换
state = RAGState(query="什么是RAG", top_k="5")    # top_k 自动转为 int 5

# 3. 默认值
state = RAGState(query="什么是RAG")               # top_k=5, threshold=0.7

# 4. 序列化
state.model_dump()  # {'query': '什么是RAG', 'top_k': 5, ...}
```

**在 LangGraph 中使用**：

```python
from langgraph.graph import StateGraph

# 直接替代 TypedDict
graph = StateGraph(RAGState)
```

**2 分钟记住**：继承 BaseModel = 自动获得验证 + 转换 + 序列化。

## 卡片4：Field 约束

**一句话**：用 `Field()` 声明字段的约束条件，不写一行验证逻辑就能限制输入。

**常用约束**：

```python
from pydantic import BaseModel, Field

class SearchState(BaseModel):
    query: str = Field(min_length=1, max_length=500, description="用户查询")
    top_k: int = Field(default=5, ge=1, le=100, description="返回结果数")
    threshold: float = Field(default=0.7, ge=0.0, le=1.0, description="相似度阈值")
    model_name: str = Field(default="gpt-4", pattern=r"^(gpt|claude|deepseek)-")
```

**约束速查表**：

| 约束 | 含义 | 适用类型 |
|------|------|----------|
| `min_length` | 最小长度 | str, list |
| `max_length` | 最大长度 | str, list |
| `ge` | 大于等于 | int, float |
| `le` | 小于等于 | int, float |
| `gt` | 大于 | int, float |
| `lt` | 小于 | int, float |
| `pattern` | 正则匹配 | str |

**2 分钟记住**：Field() = 声明式约束，零逻辑代码实现输入限制。

## 卡片5：field_validator

**一句话**：对单个字段的自定义验证逻辑，Field 约束搞不定的用它。

**基本语法**：

```python
from pydantic import BaseModel, field_validator

class RAGState(BaseModel):
    query: str
    top_k: int = 5

    @field_validator('query')
    @classmethod
    def clean_query(cls, v: str) -> str:
        # v 是字段的值
        v = v.strip()                    # 去首尾空格
        if not v:
            raise ValueError('查询不能为空')
        return v                         # 返回处理后的值
```

**两种模式**：

```python
# mode='before'：在类型转换之前执行
@field_validator('top_k', mode='before')
@classmethod
def parse_top_k(cls, v):
    if isinstance(v, str) and v.isdigit():
        return int(v)           # 手动转换
    return v

# mode='after'（默认）：在类型转换之后执行
@field_validator('top_k')
@classmethod
def check_top_k(cls, v: int) -> int:
    if v > 50:
        return 50               # 超过上限就截断
    return v
```

**实际应用 —— 清洗 LLM 输出**：

```python
@field_validator('llm_response')
@classmethod
def clean_response(cls, v: str) -> str:
    # LLM 有时会在回答前加 "Answer:" 前缀
    if v.startswith('Answer:'):
        v = v[7:]
    return v.strip()
```

**2 分钟记住**：field_validator = 单字段的自定义清洗 + 验证。

## 卡片6：model_validator

**一句话**：对整个模型的跨字段验证，当一个字段的合法性取决于另一个字段时使用。

**基本语法**：

```python
from pydantic import BaseModel, model_validator

class PipelineState(BaseModel):
    mode: str = "simple"
    api_key: str = ""
    top_k: int = 5

    @model_validator(mode='after')
    def check_advanced_mode(self):
        if self.mode == "advanced" and not self.api_key:
            raise ValueError('高级模式需要提供 api_key')
        return self
```

**before vs after**：

```python
# mode='before'：接收原始 dict，字段还没解析
@model_validator(mode='before')
@classmethod
def set_defaults(cls, data: dict):
    if data.get('mode') == 'advanced' and 'top_k' not in data:
        data['top_k'] = 20
    return data

# mode='after'：接收已验证的模型实例，最常用
@model_validator(mode='after')
def validate_combination(self):
    ...
    return self
```

**执行顺序**：`model_validator(before)` → `field_validator(before)` → 类型转换 → `field_validator(after)` → `model_validator(after)`

**2 分钟记住**：model_validator = 跨字段的关联验证，after 模式最常用。

## 卡片7：类型强制转换

**一句话**：Pydantic 自动将兼容类型转换为目标类型，比如 `"42"` → `42`。

**转换规则**：

```python
from pydantic import BaseModel

class State(BaseModel):
    count: int
    score: float
    active: bool

# 自动转换成功
State(count="42", score="0.8", active="true")
# → count=42, score=0.8, active=True

# 转换失败
State(count="abc", score=0.8, active=True)  # ValidationError
```

**为什么这在 LangGraph 中很重要？**

LLM 的输出通常是字符串，即使你要求它返回数字：

```python
llm_output = {"top_k": "5", "threshold": "0.7"}

# 没有 Pydantic → 手动转换每个字段
top_k = int(llm_output["top_k"])

# 有 Pydantic → 一行搞定
state = RAGState(**llm_output)
```

**严格模式**（关闭自动转换）：

```python
from pydantic import ConfigDict

class StrictState(BaseModel):
    model_config = ConfigDict(strict=True)
    count: int

StrictState(count="42")  # ValidationError! 不允许 str → int
```

**2 分钟记住**：Pydantic 的类型转换 = LLM 输出的自动适配器。

## 卡片8：ValidationError

**一句话**：验证失败时抛出的结构化异常，包含所有错误的详细信息。

**基本用法**：

```python
from pydantic import BaseModel, Field, ValidationError

class RAGState(BaseModel):
    query: str = Field(min_length=1)
    top_k: int = Field(ge=1, le=100)

try:
    state = RAGState(query="", top_k=200)
except ValidationError as e:
    print(e.error_count())   # 2（两个错误）

    for error in e.errors():
        print(error)
        # {'type': 'string_too_short', 'loc': ('query',), ...}
        # {'type': 'less_than_equal', 'loc': ('top_k',), ...}
```

**错误结构**：

```python
# 每个错误包含：type（错误类型）、loc（字段路径）、msg（消息）、input（原始值）
{'type': 'string_too_short', 'loc': ('query',),
 'msg': 'String should have at least 1 character', 'input': ''}
```

**在 LangGraph 中的错误处理**：

```python
def invoke_graph(user_input: dict):
    try:
        return {"status": "ok", "data": graph.invoke(user_input)}
    except ValidationError as e:
        messages = [f"字段 '{err['loc'][0]}': {err['msg']}" for err in e.errors()]
        return {"status": "error", "message": "输入验证失败", "details": messages}
```

**2 分钟记住**：ValidationError = 结构化的错误报告，用 `.errors()` 提取详情。

## 卡片9：混合架构

**一句话**：输入用 Pydantic 严格验证，内部用 TypedDict 轻量传递——这是生产环境的最佳实践。

**架构图**：

```
用户输入 → [Pydantic InputState 验证] → 节点1 → 节点2 → 节点3 → 输出
              ↑ 严格检查                    ↑ TypedDict 轻量传递
```

**代码实现**：

```python
from typing import TypedDict, Annotated
from pydantic import BaseModel, Field, field_validator
from langgraph.graph import StateGraph
import operator

# 外部输入：Pydantic 严格验证
class InputState(BaseModel):
    query: str = Field(min_length=1, max_length=500)
    top_k: int = Field(default=5, ge=1, le=50)

    @field_validator('query')
    @classmethod
    def clean_query(cls, v):
        return v.strip()

# 内部状态：TypedDict + Reducer
class InternalState(TypedDict):
    query: str
    top_k: int
    documents: Annotated[list, operator.add]  # Reducer 支持
    answer: str

# 混合架构
graph = StateGraph(InternalState, input=InputState)
```

**为什么这样设计？**

| 关注点 | InputState (Pydantic) | InternalState (TypedDict) |
|--------|----------------------|--------------------------|
| 验证 | ✅ 运行时验证 | ❌ 不需要 |
| 转换 | ✅ 自动类型转换 | ❌ 不需要 |
| Reducer | ❌ 不需要 | ✅ 原生支持 |
| 性能 | 有开销（仅一次） | 零开销 |

**2 分钟记住**：混合架构 = Pydantic 守门 + TypedDict 干活。

## 卡片10：验证的边界

**一句话**：Pydantic 验证仅在图的入口处触发，不要过度依赖它保护内部状态。

**验证触发点**：

```
graph.invoke(input)
    │
    ▼
_coerce_state(input)  ← 验证在这里发生
    │
    ▼
节点1 → 节点2 → 节点3  ← 这里没有验证
    │
    ▼
输出
```

**源码证据**（LangGraph 内部）：

```python
# langgraph 内部简化逻辑
def _coerce_state(self, input_data):
    if issubclass(self.input_schema, BaseModel):
        # 只在这里触发 Pydantic 验证
        validated = self.input_schema(**input_data)
        return validated.model_dump()
    return input_data
```

**这意味着什么？**

```python
def node_a(state):
    # 这里修改状态不会被 Pydantic 验证
    state["top_k"] = -999       # 不会报错！
    state["query"] = ""         # 不会报错！
    return state
```

**如何保护内部状态？**

```python
# 方案1：在关键节点手动验证
def retrieval_node(state):
    assert state["top_k"] > 0, "top_k 必须为正数"
    assert state["query"], "query 不能为空"
    # ... 执行检索

# 方案2：用条件边做路由检查
def should_continue(state):
    if not state.get("query"):
        return "error_handler"
    return "next_node"
```

**2 分钟记住**：Pydantic 只管入口，内部安全靠自己——用 assert 或条件边补充防护。
