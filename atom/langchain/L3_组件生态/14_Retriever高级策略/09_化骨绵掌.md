## 卡片1：什么是高级检索策略？

**一句话：** 通过组合多种检索方法弥补单一检索器的局限性，让 RAG 系统同时拥有高召回率和高精确率。

**举例：**

单一检索器的困境：

| 检索方式 | 擅长 | 不擅长 |
|----------|------|--------|
| 向量检索 | 语义相似（"快乐" → "开心"） | 精确关键词（"Error 404"） |
| BM25 关键词 | 精确匹配（"Error 404"） | 同义词理解（"快乐" ≠ "开心"） |
| 混合检索 | 两者兼得 | 需要调权重 |

```python
# 单一向量检索 → 搜 "Python 3.12 release date" 可能返回 Python 3.11 的内容
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# 高级策略 → 混合检索 + 重排序，精确又全面
ensemble = EnsembleRetriever(retrievers=[bm25_retriever, vector_retriever])
compression = ContextualCompressionRetriever(base_compressor=reranker, base_retriever=ensemble)
```

**应用：** RAG 系统检索优化的第一步。生产环境中几乎没有人只用单一向量检索，混合策略是标配。

## 卡片2：RRF 算法直觉

**一句话：** 每个文档的最终分数 = 各检索器中排名倒数的加权和，排名越靠前贡献越大。

**举例：**

RRF 公式：`score(doc) = Σ weight_i / (rank_i + c)`，其中 c=60 是论文标准默认值。

```
文档 A：
  - 向量检索排第 2 → 贡献 = 0.6 / (2 + 60) = 0.00968
  - BM25 排第 5    → 贡献 = 0.4 / (5 + 60) = 0.00615
  - 总分 = 0.01583

文档 B：
  - 向量检索排第 8 → 贡献 = 0.6 / (8 + 60) = 0.00882
  - BM25 排第 1    → 贡献 = 0.4 / (1 + 60) = 0.00656
  - 总分 = 0.01538

结果：文档 A > 文档 B（两个检索器都认可的文档排更前）
```

**为什么 c=60？** 这个常数让排名差距被"压缩"——第 1 名和第 10 名的分数差距不会太大，避免某个检索器的结果完全主导最终排名。

**应用：** EnsembleRetriever 的核心算法。理解 RRF 才能理解为什么混合检索有效——它不是简单地合并结果，而是用排名投票机制让多个检索器"民主决策"。

## 卡片3：EnsembleRetriever 用法

**一句话：** 把多个检索器装进一个列表，设好权重，EnsembleRetriever 自动用 RRF 融合结果。

**举例：**

```python
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings

# 准备两个检索器
docs = [...]  # Document 列表
bm25 = BM25Retriever.from_documents(docs, k=10)
vectorstore = Chroma.from_documents(docs, OpenAIEmbeddings())
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 10})

# 一行代码搞定混合检索
ensemble = EnsembleRetriever(
    retrievers=[bm25, vector_retriever],
    weights=[0.4, 0.6],  # BM25 权重 0.4，向量 0.6
    c=60,                 # RRF 常数，一般不用改
)

results = ensemble.invoke("Python 异步编程最佳实践")
```

**权重经验值：**

| 场景 | BM25 权重 | 向量权重 | 原因 |
|------|-----------|----------|------|
| 通用问答 | 0.4 | 0.6 | 语义理解更重要 |
| 技术文档 | 0.5 | 0.5 | 关键词和语义同等重要 |
| 法律/医疗 | 0.6 | 0.4 | 精确术语匹配更重要 |

**应用：** 生产级 RAG 的默认检索方案。2025-2026 年社区共识是 `EnsembleRetriever(BM25 + Vector)` 作为起点。

## 卡片4：重排序的两阶段范式

**一句话：** 先用便宜的检索器粗筛 20 个候选，再用昂贵的重排序模型精选 Top-5。

**举例：**

```
两阶段检索 = 招聘流程

第一阶段（粗筛）：HR 快速浏览 100 份简历 → 筛出 20 份
  ↓ 速度快、成本低、召回高
第二阶段（精排）：技术主管逐份评估 20 份 → 选出 5 人面试
  ↓ 速度慢、成本高、精确高
```

```python
# 第一阶段：粗筛（向量检索，k=20）
base_retriever = vectorstore.as_retriever(search_kwargs={"k": 20})

# 第二阶段：精排（CrossEncoder 重排序，top_n=5）
from langchain.retrievers.document_compressors import CrossEncoderReranker
from langchain_community.cross_encoders import HuggingFaceCrossEncoder

model = HuggingFaceCrossEncoder(model_name="cross-encoder/ms-marco-MiniLM-L-6-v2")
reranker = CrossEncoderReranker(model=model, top_n=5)
```

**为什么不直接精确检索？**
- 向量检索 Top-4：可能漏掉好结果（召回低）
- 向量检索 Top-20：噪声多，塞进 prompt 浪费 token
- 粗筛 20 + 精排 5：召回高 + 精确高，两全其美

**应用：** 所有生产级 RAG 系统的标准架构。重排序是提升检索质量性价比最高的手段。

## 卡片5：ContextualCompressionRetriever 用法

**一句话：** 装饰器模式——把任意重排序器/压缩器"套"在任意基础检索器外面，一行代码升级检索质量。

**举例：**

```python
from langchain.retrievers import ContextualCompressionRetriever
from langchain_community.document_compressors import FlashrankRerank

# 基础检索器（可以是任何 Retriever）
base_retriever = ensemble  # 上面的 EnsembleRetriever

# 重排序器（FlashRank：免费、本地、快速）
compressor = FlashrankRerank(top_n=5)

# 一行代码套上重排序
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base_retriever,
)

# 使用方式完全一样
results = compression_retriever.invoke("什么是 RAG？")
# 返回的是重排序后的 Top-5 文档
```

**重排序器选型：**

| 重排序器 | 成本 | 速度 | 质量 | 适用场景 |
|----------|------|------|------|----------|
| FlashrankRerank | 免费 | 快 | 中 | 开发/小项目 |
| CrossEncoderReranker | 免费 | 中 | 高 | 本地部署 |
| CohereRerank | API 付费 | 快 | 高 | 生产环境 |
| JinaRerank | API 付费 | 快 | 高 | 多语言场景 |

**应用：** 装饰器模式的优雅之处在于——你可以随时给任何检索器"加一层"重排序，不需要改动原有代码。

## 卡片6：MultiQueryRetriever 查询扩展

**一句话：** 用 LLM 把一个查询改写成 3 个不同角度的变体，分别检索后合并去重，提高召回率。

**举例：**

```python
from langchain.retrievers.multi_query import MultiQueryRetriever
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(temperature=0)
multi_query = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(search_kwargs={"k": 5}),
    llm=llm,
    include_original=True,  # 也保留原始查询
)

results = multi_query.invoke("RAG 系统如何处理长文档？")
```

**LLM 内部生成的查询变体：**

```
原始查询：RAG 系统如何处理长文档？

变体 1：What are the strategies for handling long documents in RAG pipelines?
变体 2：长文档分块和检索的最佳实践是什么？
变体 3：RAG 架构中如何优化大文件的检索效率？
```

每个变体分别检索 5 个文档 → 合并去重 → 最终可能返回 8-15 个唯一文档。

**什么时候用？**
- 用户查询模糊或过于简短（"RAG 优化"）
- 需要从多个角度覆盖一个话题
- 单次检索召回率不够

**什么时候不用？**
- 查询已经很明确（"Python 3.12 发布日期"）
- 对延迟敏感（多了一次 LLM 调用）

**应用：** 经典 IR 技术"查询扩展"的 LLM 版本。适合知识库问答、研究助手等需要高召回的场景。

## 卡片7：ParentDocumentRetriever 小到大

**一句话：** 用小块做精准检索，用大块（父文档）喂给 LLM，解决"检索粒度"和"上下文完整性"的矛盾。

**举例：**

核心矛盾：
- 小块（200 字）→ 检索精准，但上下文不完整，LLM 看不懂
- 大块（2000 字）→ 上下文完整，但检索不精准，噪声多

ParentDocumentRetriever 的解法：两层存储。

```python
from langchain.retrievers import ParentDocumentRetriever
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.storage import InMemoryStore
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings

# 子块分割器（用于检索）
child_splitter = RecursiveCharacterTextSplitter(chunk_size=200)

# 父块分割器（用于返回给 LLM）
parent_splitter = RecursiveCharacterTextSplitter(chunk_size=2000)

# 两层存储
vectorstore = Chroma(embedding_function=OpenAIEmbeddings())
docstore = InMemoryStore()

retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,
    docstore=docstore,
    child_splitter=child_splitter,
    parent_splitter=parent_splitter,
)

# 添加文档（自动分割 + 建立父子关系）
retriever.add_documents(docs)

# 检索：搜索小块 → 返回对应的父块
results = retriever.invoke("什么是注意力机制？")
# 返回的是 2000 字的父块，包含完整上下文
```

**三层 vs 两层：**

```
三层（有 parent_splitter）：原始文档 → 父块(2000字) → 子块(200字)
两层（无 parent_splitter）：原始文档(即父) → 子块(200字)
```

**应用：** 技术文档、学术论文等需要完整上下文的场景。检索精度和上下文完整性不再是二选一。

## 卡片8：SelfQueryRetriever 自查询

**一句话：** 用 LLM 把自然语言查询自动拆分成"语义查询 + 元数据过滤条件"，实现结构化检索。

**举例：**

用户说："找一篇 2024 年发表的关于 Transformer 的论文"

普通检索器：把整句话做向量检索 → 可能返回 2020 年的论文
SelfQueryRetriever：LLM 自动拆分 →
- 语义查询：`"Transformer"`
- 元数据过滤：`year == 2024`

```python
from langchain.retrievers.self_query.base import SelfQueryRetriever
from langchain.chains.query_constructor.schema import AttributeInfo

# 定义元数据字段
metadata_field_info = [
    AttributeInfo(name="year", description="论文发表年份", type="integer"),
    AttributeInfo(name="author", description="论文作者", type="string"),
    AttributeInfo(name="category", description="论文类别", type="string"),
]

retriever = SelfQueryRetriever.from_llm(
    llm=ChatOpenAI(temperature=0),
    vectorstore=vectorstore,
    document_content_description="学术论文摘要",
    metadata_field_info=metadata_field_info,
)

# 自然语言查询 → 自动生成结构化过滤
results = retriever.invoke("找一篇 2024 年 John 写的关于 RAG 的论文")
# LLM 内部生成：query="RAG", filter: year==2024 AND author=="John"
```

**支持 20+ 向量存储后端：** Chroma, Pinecone, Milvus, Qdrant, Elasticsearch, Weaviate 等。

**应用：** 电商搜索（"200 元以下的蓝色运动鞋"）、文档管理（"2024 年 Q3 的财务报告"）、知识库（"Python 相关的入门教程"）。

## 卡片9：生产级组合策略

**一句话：** 生产环境的黄金管道是 `MultiQuery → Ensemble(BM25+Vector) → Rerank → Top-K`，四层递进提升检索质量。

**举例：**

```python
from langchain.retrievers import EnsembleRetriever, ContextualCompressionRetriever
from langchain.retrievers.multi_query import MultiQueryRetriever
from langchain_community.retrievers import BM25Retriever
from langchain_community.document_compressors import FlashrankRerank
from langchain_chroma import Chroma
from langchain_openai import ChatOpenAI, OpenAIEmbeddings

# === 第 1 层：基础检索器 ===
docs = [...]
bm25 = BM25Retriever.from_documents(docs, k=15)
vector = Chroma.from_documents(
    docs, OpenAIEmbeddings()
).as_retriever(search_kwargs={"k": 15})

# === 第 2 层：混合检索（RRF 融合） ===
ensemble = EnsembleRetriever(
    retrievers=[bm25, vector],
    weights=[0.4, 0.6],
)

# === 第 3 层：查询扩展（可选，增加召回） ===
multi_query = MultiQueryRetriever.from_llm(
    retriever=ensemble,
    llm=ChatOpenAI(temperature=0),
)

# === 第 4 层：重排序（精选 Top-5） ===
reranker = FlashrankRerank(top_n=5)
final_retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=multi_query,
)

# 一行调用，四层管道自动执行
results = final_retriever.invoke("如何优化 RAG 系统的检索质量？")
```

**每一层的作用：**

```
用户查询
  ↓
[MultiQuery] 生成 3 个查询变体 → 提高召回率
  ↓
[Ensemble] BM25 + 向量混合检索 → 语义 + 关键词双保险
  ↓
[Rerank] CrossEncoder 精排 → 过滤噪声，精选 Top-5
  ↓
最终 5 个高质量文档 → 送入 LLM 生成回答
```

**应用：** 这是 2025-2026 年社区推荐的生产级 RAG 检索管道。根据实际需求可以增减层数——简单场景去掉 MultiQuery，成本敏感去掉 Rerank。

## 卡片10：检索策略选型指南

**一句话：** 没有万能策略，根据数据特点、延迟要求和成本预算选择合适的组合。

**举例：**

**决策树：**

```
你的场景是什么？
│
├── 简单问答（FAQ、客服）
│   └── 推荐：Vector + BM25 混合检索
│       └── EnsembleRetriever(weights=[0.4, 0.6])
│
├── 知识库检索（技术文档、论文）
│   └── 推荐：混合检索 + 重排序
│       └── Ensemble → Rerank(top_n=5)
│
├── 长文档场景（书籍、报告）
│   └── 推荐：父文档检索 + 重排序
│       └── ParentDocumentRetriever → Rerank
│
├── 结构化数据（有明确过滤条件）
│   └── 推荐：自查询检索
│       └── SelfQueryRetriever
│
└── 高召回要求（研究、分析）
    └── 推荐：查询扩展 + 混合 + 重排序
        └── MultiQuery → Ensemble → Rerank
```

**成本 vs 质量权衡：**

| 策略组合 | 延迟 | 成本 | 质量 | 复杂度 |
|----------|------|------|------|--------|
| 纯向量检索 | 低 | 低 | 中 | 低 |
| Ensemble(BM25+Vector) | 低 | 低 | 中高 | 低 |
| Ensemble + Rerank | 中 | 中 | 高 | 中 |
| MultiQuery + Ensemble + Rerank | 高 | 高 | 很高 | 高 |
| ParentDoc + Rerank | 中 | 中 | 高 | 中 |

**黄金法则：**
1. 先用 `Ensemble(BM25+Vector)` 建立基线
2. 加 Rerank 看质量提升是否值得延迟增加
3. 如果召回不够，加 MultiQuery
4. 如果有结构化过滤需求，用 SelfQuery
5. 如果文档很长，用 ParentDocument

**应用：** 没有银弹。从最简单的方案开始，用评估指标（Recall@K, MRR, NDCG）驱动优化决策。
