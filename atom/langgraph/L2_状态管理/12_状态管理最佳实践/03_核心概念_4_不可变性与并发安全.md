## 1. 一句话定义

**不可变性与并发安全确保多个节点同时读写状态时，数据始终一致、不会出现竞态条件。**

这是 LangGraph 状态管理中最容易被忽视、但出了问题最难调试的一环。

---

## 2. 为什么不可变性很重要？

### 2.1 LangGraph 的并行执行模型

LangGraph 中，多个节点可以在同一个 step 内并行执行：

```
         ┌──→ [Node A] ──┐
[Start] ─┤               ├──→ [Merge] → [End]
         └──→ [Node B] ──┘
```

Node A 和 Node B 同时运行，同时读取状态，同时产出更新。如果状态是可变的，灾难就来了。

### 2.2 可变状态的三大灾难

#### 灾难 1：竞态条件（Race Condition）

```python
# ❌ 假设状态是可变的引用
def node_a(state):
    state["counter"] += 1  # 读到 0，写入 1
    return state

def node_b(state):
    state["counter"] += 1  # 也读到 0，也写入 1
    return state

# 期望: counter = 2 → 实际: counter = 1（丢失更新）
```

#### 灾难 2：不可预测的执行顺序

```python
# ❌ 可变状态 + 并行 = 结果不确定
def node_a(state):
    state["items"].append("a")
    return state

def node_b(state):
    print(len(state["items"]))  # 0 还是 1？取决于谁先跑
    state["items"].append("b")
    return state
```

#### 灾难 3：调试地狱

可变状态的 Bug 三大特点：**不可复现**（取决于时序）、**不可追踪**（中间状态被覆盖）、**不可定位**（多节点都可能是元凶）。

## 3. LangGraph 的不可变性设计

### 3.1 StateSnapshot — 不可变的状态快照

LangGraph 使用 `NamedTuple` 定义 `StateSnapshot`，天然不可变：

```python
# 来源: libs/langgraph/langgraph/types.py
from typing import NamedTuple, Any

class StateSnapshot(NamedTuple):
    """状态快照 — 不可变设计"""
    values: dict[str, Any]          # 当前状态值
    next: tuple[str, ...]           # 下一步节点
    config: dict                    # 配置信息
    metadata: dict | None           # 元数据
    created_at: str | None          # 创建时间
    parent_config: dict | None      # 父配置
    tasks: tuple                    # 待执行任务

# ✅ 可以读取
# snapshot.values → {"counter": 0}
# ❌ 不能修改
# snapshot.values = {"counter": 1}  → AttributeError!
```

### 3.2 节点接收副本，而非引用

每个节点拿到的是状态的**副本**，不是原始引用：

```python
from typing import TypedDict, Annotated
import operator

class State(TypedDict):
    counter: Annotated[int, operator.add]
    items: Annotated[list[str], operator.add]

def node_a(state: State) -> dict:
    # state 是副本，但最佳实践是：不要修改它，只返回更新
    return {"counter": 1, "items": ["from_a"]}

def node_b(state: State) -> dict:
    # 不管 node_a 做了什么，这里的 state 都是原始值
    return {"counter": 1, "items": ["from_b"]}
```

### 3.3 更新收集 → 原子应用

节点不直接修改状态，返回更新字典，由框架统一应用：

```
  ┌─────────────────────────────────────────────┐
  │ Step N                                       │
  │  Node A 读取副本 → 计算 → 返回 {"x": 1}     │
  │  Node B 读取副本 → 计算 → 返回 {"x": 2}     │
  │  ──── 所有节点完成后 ────                     │
  │  框架收集更新: [{"x": 1}, {"x": 2}]          │
  │  通过 Reducer 原子合并: x = reducer(1, 2)    │
  │  ✅ 新状态一次性生效，无中间状态泄露           │
  └─────────────────────────────────────────────┘
```

## 4. 并发安全的 Reducer 要求

### 4.1 两个数学性质

更新到达 Reducer 的顺序不确定，因此 Reducer 必须满足：

#### 结合律（Associativity）：`(a ⊕ b) ⊕ c = a ⊕ (b ⊕ c)`

```python
# ✅ 加法满足结合律
(1 + 2) + 3 == 1 + (2 + 3)  # 6 == 6 ✓
# ❌ 减法不满足
(10 - 3) - 2 != 10 - (3 - 2)  # 5 != 9 ✗
```

#### 交换律（Commutativity）：`a ⊕ b = b ⊕ a`

```python
# ✅ 加法满足交换律
1 + 2 == 2 + 1  # ✓
# ✅ 集合并集满足交换律
{1, 2} | {3} == {3} | {1, 2}  # ✓
# ❌ 列表拼接不满足交换律
["a"] + ["b"] != ["b"] + ["a"]  # ✗
```

### 4.2 为什么这两个性质至关重要？

```python
# Node A、B、C 并行执行，分别返回 a, b, c
# 框架收到更新的顺序不确定：[a,b,c] / [b,a,c] / [c,a,b] ...
# 满足结合律+交换律 → 任何顺序结果一致 ✅
# 不满足 → 不同顺序不同结果 → 不确定性 Bug ❌
```

### 4.3 安全 vs 不安全的 Reducer

```python
import operator
from typing import Annotated, TypedDict

# ✅ 安全的 Reducer（满足结合律 + 交换律）

class SafeState(TypedDict):
    total: Annotated[int, operator.add]          # 加法

def union_reducer(current: set, new: set) -> set:
    return current | new                          # 集合并集

def max_reducer(current: int, new: int) -> int:
    return max(current, new)                      # 取最大值

def or_reducer(current: bool, new: bool) -> bool:
    return current or new                         # 布尔 OR


# ❌ 不安全的 Reducer

# 列表拼接 — 不满足交换律（如果业务依赖顺序就是 Bug）
class UnsafeState1(TypedDict):
    log: Annotated[list[str], operator.add]

# 字符串拼接 — 不满足交换律
def concat_reducer(current: str, new: str) -> str:
    return current + new  # "hello"+"world" ≠ "world"+"hello"

# 减法 — 结合律和交换律都不满足
def sub_reducer(current: int, new: int) -> int:
    return current - new  # 完全不安全！
```

### 4.4 列表拼接的特殊情况

`operator.add` 用于列表时不满足交换律，但 LangGraph 中广泛使用：

```python
class ChatState(TypedDict):
    messages: Annotated[list, operator.add]
```

**看业务逻辑**：只关心"收集了哪些" → 安全；依赖精确顺序 → 不安全。实践中并行节点产出的消息通常独立，顺序不影响语义。

## 5. 原子更新模式

### 5.1 BinaryOperatorAggregate 的原子性

```python
# 来源: libs/langgraph/langgraph/channels/binop.py
class BinaryOperatorAggregate:
    def update(self, values: Sequence[Value]) -> bool:
        """原子更新 — 一次处理所有值"""
        if not values:
            return False
        if self.value is MISSING:
            self.value = values[0]
            values = values[1:]
        for value in values:
            self.value = self.operator(self.value, value)
        return True  # 所有值在单次调用中处理完毕
```

`update()` 接收**序列**，所有更新在一次方法调用中完成，不存在"处理了一半"的中间状态。

### 5.2 原子更新流程示例

```python
class State(TypedDict):
    score: Annotated[int, operator.add]
    tags: Annotated[list[str], operator.add]

def node_a(state: State) -> dict:
    return {"score": 10, "tags": ["fast"]}

def node_b(state: State) -> dict:
    return {"score": 20, "tags": ["accurate"]}

# 框架内部：
# score_channel.update([10, 20])  → 一次调用，原子操作
# tags_channel.update([["fast"], ["accurate"]])  → 一次调用
# 新状态一次性可见：{"score": 30, "tags": ["fast", "accurate"]}
# 其他节点永远看不到 score=10 但 tags 还没更新的中间状态
```

### 5.3 EmptyChannelError — 未初始化保护

```python
# 来源: libs/langgraph/langgraph/channels/base.py
class EmptyChannelError(Exception):
    """尝试读取未初始化的 Channel 时抛出"""
    pass

class BaseChannel:
    def is_available(self) -> bool:
        """检查 Channel 是否有值可读"""
        ...

# 使用模式：先检查再读取
if channel.is_available():
    value = channel.get()
else:
    raise EmptyChannelError(f"Channel {name} has no value")
```

## 6. 常见并发陷阱

### 陷阱 1：直接修改输入状态（最常见！）

```python
# ❌ 反模式
def bad_node(state: State) -> dict:
    state["items"].append("new_item")  # 修改了输入！
    state["counter"] += 1
    return state

# ✅ 正确模式
def good_node(state: State) -> dict:
    return {
        "items": ["new_item"],   # Reducer 负责合并
        "counter": 1             # Reducer 负责累加
    }
```

**为什么危险？** 浅拷贝下 `state["items"]` 是共享引用，`append` 会污染并行节点看到的数据。

### 陷阱 2：非确定性 Reducer

```python
import random
from datetime import datetime

# ❌ Reducer 中使用随机数
def bad_reducer(current: list, new: list) -> list:
    combined = current + new
    random.shuffle(combined)  # 每次结果不同！
    return combined

# ❌ Reducer 中使用时间戳
def bad_reducer_2(current: dict, new: dict) -> dict:
    return {**current, **new, "updated_at": datetime.now()}

# ✅ 纯函数 Reducer
def good_reducer(current: list, new: list) -> list:
    return sorted(set(current + new))  # 确定性去重排序
```

### 陷阱 3：共享可变对象

```python
# ❌ 模块级可变对象
shared_cache = {}

def node_a(state):
    state["cache"]["key_a"] = "value_a"  # 修改共享对象！
    return {"cache": state["cache"]}

# ✅ 每次创建新对象
def good_node_a(state) -> dict:
    return {"cache": {**state["cache"], "key_a": "value_a"}}
```

### 陷阱 4：Reducer 中产生副作用

```python
# ❌ Reducer 有副作用
def bad_logging_reducer(current: int, new: int) -> int:
    open("reducer.log", "a").write(f"{current}+{new}\n")  # 副作用！
    return current + new

# ✅ Reducer 是纯函数，日志放在节点中
def pure_add(current: int, new: int) -> int:
    return current + new

def node_with_logging(state) -> dict:
    import logging
    logging.getLogger(__name__).info(f"Score: {state['score'] + 10}")
    return {"score": 10}
```

## 7. 防御性编程模式

### 7.1 深拷贝保护

```python
import copy

# 不需要深拷贝 — 基本类型不可变
def node_simple(state) -> dict:
    return {"counter": state["counter"] + 1}  # int 不可变，安全

# 不需要深拷贝 — 创建全新对象
def node_new(state) -> dict:
    return {"items": ["brand_new"]}  # 全新 list，安全

# 需要深拷贝 — 修改嵌套可变对象
def node_nested(state) -> dict:
    config = copy.deepcopy(state["config"])
    config["nested"]["key"] = "new_value"
    return {"config": config}
```

### 7.2 Frozen Dataclass 模式

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class SearchResult:
    """不可变的搜索结果"""
    query: str
    score: float
    content: str
    # frozen=True → result.score = 0.9 会抛 FrozenInstanceError

class RAGState(TypedDict):
    results: Annotated[list[SearchResult], operator.add]

def search_node(state: RAGState) -> dict:
    result = SearchResult(query="LangGraph", score=0.95, content="...")
    return {"results": [result]}  # ✅ 不可变对象，安全
```

### 7.3 带验证的 Reducer

```python
def validated_add(current: list[str], new: list[str]) -> list[str]:
    """带验证的 Reducer"""
    if not isinstance(new, list):
        raise TypeError(f"Expected list, got {type(new)}")
    for item in new:
        if not isinstance(item, str):
            raise TypeError(f"Expected str items, got {type(item)}")
    combined = current + new
    if len(combined) > 1000:
        raise ValueError(f"State too large: {len(combined)} items")
    return combined

class SafeState(TypedDict):
    documents: Annotated[list[str], validated_add]
```

## 8. 完整实战：并发安全的 RAG 多路检索

```python
"""并发安全的 RAG 多路检索系统"""
from typing import TypedDict, Annotated
from dataclasses import dataclass
import operator

# ── 不可变数据结构 ──

@dataclass(frozen=True)
class RetrievedDoc:
    content: str
    source: str
    score: float

# ── 并发安全的 Reducer ──

def merge_docs(
    current: list[RetrievedDoc],
    new: list[RetrievedDoc]
) -> list[RetrievedDoc]:
    """合并检索结果，去重并按分数排序"""
    seen = set()
    merged = []
    for doc in current + new:
        key = (doc.source, doc.content[:50])
        if key not in seen:
            seen.add(key)
            merged.append(doc)
    return sorted(merged, key=lambda d: d.score, reverse=True)

def merge_errors(current: list[str], new: list[str]) -> list[str]:
    """合并错误信息，去重（set 保证交换律）"""
    return list(set(current + new))

# ── 状态定义 ──

class RAGState(TypedDict):
    query: str
    docs: Annotated[list[RetrievedDoc], merge_docs]
    errors: Annotated[list[str], merge_errors]
    search_count: Annotated[int, operator.add]

# ── 并行检索节点 ──

def vector_search(state: RAGState) -> dict:
    """向量检索 — 与 keyword_search 并行"""
    return {
        "docs": [
            RetrievedDoc("向量结果1", "vector_db", 0.95),
            RetrievedDoc("向量结果2", "vector_db", 0.87),
        ],
        "search_count": 1
    }

def keyword_search(state: RAGState) -> dict:
    """关键词检索 — 与 vector_search 并行"""
    return {
        "docs": [RetrievedDoc("关键词结果1", "elastic", 0.91)],
        "search_count": 1
    }

# 不管执行顺序如何：
# merge_docs 内部做了排序和去重 → 两种顺序最终结果一致 ✅
```

## 9. 速查表：安全 vs 危险操作

| 操作 | 安全性 | 说明 |
|------|--------|------|
| `return {"key": new_value}` | ✅ 安全 | 返回增量更新 |
| `state["list"].append(x)` | ❌ 危险 | 直接修改输入 |
| `return {**state, "key": v}` | ⚠️ 浅拷贝 | 嵌套对象仍共享引用 |
| `operator.add` (int) | ✅ 安全 | 满足结合律+交换律 |
| `operator.add` (list) | ⚠️ 顺序不确定 | 不满足交换律 |
| `max(a, b)` | ✅ 安全 | 满足结合律+交换律 |
| `a - b` | ❌ 危险 | 不满足结合律 |
| `frozen dataclass` | ✅ 安全 | 不可变对象 |
| 模块级可变变量 | ❌ 危险 | 共享状态污染 |
| Reducer 中写日志/random | ❌ 危险 | 副作用/非确定性 |

## 10. 核心原则总结

**三条铁律：**

1. **永远不要修改输入 state** — 只返回增量更新字典
2. **Reducer 必须是纯函数** — 无副作用、确定性、满足结合律和交换律
3. **使用不可变数据结构** — frozen dataclass、tuple 替代 list、frozenset 替代 set

**一句话记忆：**

> 节点只读状态、只写更新；Reducer 只做数学、不搞副作用。
