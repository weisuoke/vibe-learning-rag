## 卡片1：什么是状态快照？

**一句话**：状态快照是图在某个执行步骤的完整状态记录，包含当前值、下一步动作、配置信息等。

**类比**：就像游戏的自动存档——每过一关，系统自动保存你的血量、装备、位置。你可以随时读档回到任何一关重新打。

**为什么需要？**

LangGraph 的图执行是多步骤的，每个节点处理完后状态会变化。如果没有快照：
- 出错了不知道哪一步出的问题
- 想回到之前的状态重新执行做不到
- 无法实现 human-in-the-loop（人工审核后继续）

**核心原则**：每个 checkpoint 都是不可变的，修改状态会创建新的 checkpoint，而不是覆盖旧的。

**代码直觉**：

```python
from langgraph.checkpoint.memory import InMemorySaver

# 必须配置 checkpointer 才能使用快照
graph = builder.compile(checkpointer=InMemorySaver())

# 执行后，每一步都会自动保存快照
config = {"configurable": {"thread_id": "1"}}
result = graph.invoke({"query": "什么是RAG"}, config=config)

# 获取当前快照
snapshot = graph.get_state(config)
```

**2 分钟记住**：状态快照 = 图执行的自动存档系统，每一步都有记录。

## 卡片2：StateSnapshot 的 8 个字段

**一句话**：StateSnapshot 是一个 NamedTuple，8 个字段各司其职，覆盖了"当前状态是什么、接下来做什么、怎么来的"三大问题。

**字段速查表**：

| 字段 | 类型 | 含义 | 类比 |
|------|------|------|------|
| `values` | dict | 当前所有 channel 的值 | 游戏存档里的角色属性 |
| `next` | tuple[str] | 下一步要执行的节点名 | 下一关的名字 |
| `config` | RunnableConfig | 获取此快照的配置 | 存档的文件路径 |
| `metadata` | CheckpointMetadata | 元数据（来源、步骤等） | 存档的备注信息 |
| `created_at` | str | 创建时间戳 | 存档时间 |
| `parent_config` | RunnableConfig | 父快照的配置 | 上一个存档的路径 |
| `tasks` | tuple[PregelTask] | 当前步骤的任务列表 | 当前关卡的任务 |
| `interrupts` | tuple[Interrupt] | 待处理的中断 | 暂停等待玩家操作 |

**代码示例**：

```python
snapshot = graph.get_state(config)

print(snapshot.values)       # {'query': '什么是RAG', 'answer': '...'}
print(snapshot.next)         # () 表示执行完毕，('retrieve',) 表示下一步是检索
print(snapshot.metadata)     # {'source': 'loop', 'step': 2, ...}
print(snapshot.created_at)   # '2026-02-27T10:30:00+00:00'
```

**2 分钟记住**：values 看状态，next 看下一步，metadata 看来源——三个字段解决 90% 的调试需求。

## 卡片3：Checkpoint 底层存储

**一句话**：Checkpoint 是 StateSnapshot 的底层存储格式，是一个 TypedDict，关注 channel 级别的版本管理。

**StateSnapshot vs Checkpoint**：

```
用户看到的：StateSnapshot（高层、可读）
    ↑ _prepare_state_snapshot() 构建
底层存储的：Checkpoint（底层、版本管理）
```

**Checkpoint 核心字段**：

```python
class Checkpoint(TypedDict):
    v: int                    # 版本号（当前为 4）
    id: str                   # 唯一 ID（单调递增）
    ts: str                   # ISO 8601 时间戳
    channel_values: dict      # channel 名 → 值
    channel_versions: dict    # channel 名 → 版本号
    versions_seen: dict       # 节点 ID → 已见的 channel 版本
```

**关键区别**：

| 维度 | StateSnapshot | Checkpoint |
|------|--------------|------------|
| 类型 | NamedTuple | TypedDict |
| 面向 | 用户 | 存储引擎 |
| 包含 | values, next, tasks | channel_values, versions_seen |
| 用途 | 查看和调试 | 持久化和版本管理 |

**为什么要分两层？** 因为用户不需要关心 channel 版本管理这些内部细节，而存储引擎不需要知道"下一步执行哪个节点"这种高层信息。各管各的，职责清晰。

**2 分钟记住**：Checkpoint 是数据库里的原始记录，StateSnapshot 是查询后返回的业务对象。

## 卡片4：配置 Checkpointer

**一句话**：必须在 `compile()` 时配置 checkpointer，否则 get_state / get_state_history / update_state 全部不可用。

**最小配置**：

```python
from langgraph.graph import StateGraph
from langgraph.checkpoint.memory import InMemorySaver

# 定义图
builder = StateGraph(State)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge("retrieve", "generate")
builder.set_entry_point("retrieve")
builder.set_finish_point("generate")

# 关键：配置 checkpointer
graph = builder.compile(checkpointer=InMemorySaver())
```

**常见 Checkpointer 选型**：

| Checkpointer | 适用场景 | 特点 |
|--------------|----------|------|
| `InMemorySaver` | 开发调试 | 内存存储，重启丢失 |
| `SqliteSaver` | 单机生产 | 文件持久化 |
| `PostgresSaver` | 分布式生产 | 多实例共享 |

**忘记配置的后果**：

```python
# 没有 checkpointer
graph = builder.compile()  # 注意：没传 checkpointer

graph.get_state(config)  # ValueError: No checkpointer set!
```

**调用时必须传 thread_id**：

```python
# ❌ 错误：没有 thread_id
graph.invoke({"query": "什么是RAG"})

# ✅ 正确：指定 thread_id
config = {"configurable": {"thread_id": "user-123"}}
graph.invoke({"query": "什么是RAG"}, config=config)
```

**2 分钟记住**：checkpointer + thread_id = 快照功能的两个前提条件，缺一不可。

## 卡片5：get_state() 获取快照

**一句话**：通过 thread_id 获取最新快照，通过 checkpoint_id 获取特定历史快照。

**基本用法**：

```python
config = {"configurable": {"thread_id": "1"}}

# 获取最新快照
snapshot = graph.get_state(config)
print(snapshot.values)   # 当前状态值
print(snapshot.next)     # 下一步节点（空 tuple 表示已完成）
```

**获取特定历史快照**：

```python
# 指定 checkpoint_id
config = {
    "configurable": {
        "thread_id": "1",
        "checkpoint_id": "1ef663ba-28fe-6528-8002-5a559208592c"
    }
}
snapshot = graph.get_state(config)
```

**内部流程**（简化）：

```
get_state(config)
    → checkpointer.get_tuple(config)     # 从存储获取 CheckpointTuple
    → _prepare_state_snapshot(tuple)      # 构建 StateSnapshot
    → 返回 StateSnapshot
```

**实际调试场景**：

```python
snapshot = graph.get_state(config)

# 检查执行是否完成
if not snapshot.next:
    print("图已执行完毕")
    print(f"最终答案: {snapshot.values.get('answer')}")
else:
    print(f"执行暂停，下一步: {snapshot.next}")
    print(f"当前步骤: {snapshot.metadata.get('step')}")
```

**2 分钟记住**：get_state = 读取存档，不传 checkpoint_id 就读最新的，传了就读指定的。

## 卡片6：get_state_history() 遍历历史

**一句话**：逆时间顺序返回所有快照，支持 filter、before、limit 三个参数精确筛选。

**基本用法**：

```python
config = {"configurable": {"thread_id": "1"}}

# 获取所有历史快照（最新的在前面）
for snapshot in graph.get_state_history(config):
    print(f"步骤 {snapshot.metadata.get('step')}: "
          f"来源={snapshot.metadata.get('source')}, "
          f"下一步={snapshot.next}")
```

**三个筛选参数**：

```python
# 1. filter：按 metadata 字段筛选
# 只看手动更新的快照
for s in graph.get_state_history(config, filter={"source": "update"}):
    print(s)

# 2. before：只返回某个 checkpoint 之前的快照
for s in graph.get_state_history(config, before=some_config):
    print(s)

# 3. limit：限制返回数量
# 只看最近 5 个快照
for s in graph.get_state_history(config, limit=5):
    print(s)
```

**实际应用——找到出问题的步骤**：

```python
# RAG Agent 生成了错误答案，回溯找原因
for snapshot in graph.get_state_history(config):
    step = snapshot.metadata.get("step", "?")
    source = snapshot.metadata.get("source", "?")
    docs = snapshot.values.get("documents", [])
    print(f"Step {step} ({source}): 检索到 {len(docs)} 篇文档")

    # 找到检索结果为空的步骤
    if not docs and step > 0:
        print(f"  → 问题在这里！检索结果为空")
        break
```

**注意**：`get_state_history()` 返回的是迭代器，不是列表。如果需要多次遍历，先转成 `list()`。

**2 分钟记住**：get_state_history = 查看所有存档记录，filter/before/limit 帮你快速定位。

## 卡片7：update_state() 修改状态

**一句话**：创建新 checkpoint 而非修改现有的，遵循 reducer 规则——有 reducer 的字段走合并，没有的走覆盖。

**基本用法**：

```python
config = {"configurable": {"thread_id": "1"}}

# 修改状态
new_config = graph.update_state(
    config,
    values={"query": "改进后的查询"}
)
# new_config 包含新的 checkpoint_id
```

**Reducer 规则**：

```python
from typing import Annotated, TypedDict
import operator

class State(TypedDict):
    query: str                                  # 无 reducer → 覆盖
    documents: Annotated[list, operator.add]    # 有 reducer → 追加

# 假设当前状态：query="原始查询", documents=["doc1"]

graph.update_state(config, {"query": "新查询", "documents": ["doc2"]})
# 结果：query="新查询"（覆盖），documents=["doc1", "doc2"]（追加）
```

**指定"以哪个节点的身份"更新**：

```python
# as_node 参数：模拟某个节点的输出
new_config = graph.update_state(
    config,
    values={"documents": [new_doc]},
    as_node="retrieve"  # 假装是 retrieve 节点的输出
)
```

**为什么 as_node 重要？** 因为 LangGraph 需要知道"这次更新是谁做的"，才能正确计算下一步应该执行哪个节点。如果你以 `retrieve` 的身份更新，下一步就会执行 `retrieve` 之后的节点。

**2 分钟记住**：update_state = 创建新存档（不覆盖旧的），reducer 决定是覆盖还是合并。

## 卡片8：Replay 重放

**一句话**：使用 checkpoint_id 恢复到某个历史状态继续执行，LangGraph 智能跳过已执行的步骤。

**操作步骤**：

```python
# 1. 找到要回到的时间点
history = list(graph.get_state_history(config))
target = history[3]  # 选择第 4 个快照

# 2. 用该快照的 config 继续执行
result = graph.invoke(None, config=target.config)
# 注意：传 None 作为输入，表示"从存档继续"
```

**"智能跳过"是什么意思？**

```
原始执行：节点A → 节点B → 节点C → 节点D
                          ↑
                    Replay 从这里开始

实际执行：跳过A → 跳过B → 重新执行C → 重新执行D
```

LangGraph 通过比较 `versions_seen` 和 `channel_versions` 来判断哪些步骤可以跳过。如果某个节点的输入 channel 没有变化，就不需要重新执行。

**适用场景**：
- 重现 bug：回到出问题的步骤，观察执行过程
- 重试失败的步骤：网络超时导致某个节点失败，回到该步骤重试
- 验证修复：修改了节点逻辑后，从某个历史状态重新执行看结果

**2 分钟记住**：Replay = 读档重打，但已经打过的关会自动跳过。

## 卡片9：Fork 分叉

**一句话**：update_state() + invoke() 组合，先修改历史状态再继续执行，探索"如果当时不一样会怎样"。

**操作步骤**：

```python
# 1. 找到要修改的历史快照
history = list(graph.get_state_history(config))
target = history[3]

# 2. 修改状态，创建新分支
new_config = graph.update_state(
    target.config,                          # 基于历史快照
    values={"query": "更精确的查询词"}       # 修改某个字段
)

# 3. 从新分支继续执行
result = graph.invoke(None, config=new_config)
```

**Fork vs Replay 的区别**：

```
Replay：  A → B → C → D
                  ↑ 从这里原样继续

Fork：    A → B → C → D（原始路径）
                  ↑
                  C' → D'（新分支，C 的状态被修改了）
```

**RAG 场景实战**：

```python
# 场景：RAG Agent 检索到的文档不够好，想手动替换

# 1. 找到检索步骤的快照
for s in graph.get_state_history(config):
    if "retrieve" in s.next:
        retrieve_snapshot = s
        break

# 2. 手动注入更好的文档
better_docs = [{"content": "RAG 是检索增强生成...", "source": "manual"}]
new_config = graph.update_state(
    retrieve_snapshot.config,
    values={"documents": better_docs},
    as_node="retrieve"
)

# 3. 从新分支继续（跳过检索，直接进入生成）
result = graph.invoke(None, config=new_config)
print(result["answer"])  # 基于手动注入的文档生成的答案
```

**2 分钟记住**：Fork = 修改存档再重打，探索不同选择的结果。

## 卡片10：在 RAG 中的应用

**一句话**：状态快照让 RAG Agent 变得可观测、可调试、可恢复——从黑盒变成白盒。

**三大应用场景**：

**场景1：调试检索质量**

```python
# 回溯检查每一步的检索结果
for snapshot in graph.get_state_history(config):
    docs = snapshot.values.get("documents", [])
    if docs:
        print(f"Step {snapshot.metadata['step']}: "
              f"检索到 {len(docs)} 篇文档")
        for doc in docs[:2]:
            print(f"  - {doc.get('content', '')[:50]}...")
```

**场景2：优化 Prompt**

```python
# Fork 到生成步骤，尝试不同的 prompt
target = find_snapshot_before_generate(config)

# 尝试 prompt A
new_config_a = graph.update_state(
    target.config,
    values={"system_prompt": "你是一个严谨的技术专家，只基于提供的文档回答"}
)
result_a = graph.invoke(None, config=new_config_a)

# 尝试 prompt B
new_config_b = graph.update_state(
    target.config,
    values={"system_prompt": "你是一个友好的助手，用通俗语言解释技术概念"}
)
result_b = graph.invoke(None, config=new_config_b)

# 对比两种 prompt 的效果
print(f"Prompt A: {result_a['answer'][:100]}")
print(f"Prompt B: {result_b['answer'][:100]}")
```

**场景3：A/B 测试不同策略**

```python
# 在检索步骤分叉，测试不同的检索策略
target = find_snapshot_before_retrieve(config)

# 策略 A：top_k=3
config_a = graph.update_state(target.config, values={"top_k": 3})
result_a = graph.invoke(None, config=config_a)

# 策略 B：top_k=10
config_b = graph.update_state(target.config, values={"top_k": 10})
result_b = graph.invoke(None, config=config_b)
```

**状态快照在 RAG 开发中的价值**：

| 能力 | 没有快照 | 有快照 |
|------|----------|--------|
| 调试 | 只能看最终结果 | 可以看每一步的中间状态 |
| 重试 | 从头重跑整个流程 | 从失败步骤继续 |
| 优化 | 改代码 → 重跑 → 对比 | Fork → 修改 → 立即对比 |
| 恢复 | 崩溃后数据丢失 | 从最近的 checkpoint 恢复 |

**2 分钟记住**：状态快照让 RAG Agent 从"跑一次看结果"变成"随时回溯、随时分叉、随时对比"。
