## 4.1 Input/Output Schema 分离

**一句话：** 用 `input_schema` 控制图接收什么，用 `output_schema` 控制图返回什么，`OverallState` 是内部的完整状态

这是多状态管理最核心、最常用的能力。它解决的问题很直接：调用方不需要看到图内部的中间状态。

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# === 第1步：定义三个 Schema ===

class InputState(TypedDict):
    """图的入口：调用方只需要提供 question"""
    question: str

class OutputState(TypedDict):
    """图的出口：调用方只会看到 answer"""
    answer: str

class OverallState(TypedDict):
    """图的内部：包含所有字段，包括中间状态"""
    question: str
    answer: str
    context: str       # 中间状态：检索到的上下文
    confidence: float  # 中间状态：置信度评分

# === 第2步：定义节点 ===

def retrieve_node(state: OverallState) -> dict:
    """检索节点：读取 question，写入 context"""
    query = state["question"]
    # 模拟检索
    context = f"关于 '{query}' 的相关文档内容..."
    return {"context": context, "confidence": 0.85}

def generate_node(state: OverallState) -> dict:
    """生成节点：读取 question + context，写入 answer"""
    return {"answer": f"根据检索结果回答：{state['context']}"}

# === 第3步：构建图，指定 input/output schema ===

builder = StateGraph(
    OverallState,                    # 内部完整状态
    input_schema=InputState,         # 入口过滤器
    output_schema=OutputState,       # 出口过滤器
)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

graph = builder.compile()

# === 第4步：调用 ===

# 输入：只需要 question（InputState 的字段）
result = graph.invoke({"question": "什么是 RAG？"})

# 输出：只包含 answer（OutputState 的字段）
print(result)
# {'answer': '根据检索结果回答：关于 \'什么是 RAG？\' 的相关文档内容...'}

# context 和 confidence 不会出现在返回值中！
print("context" in result)  # False
print("confidence" in result)  # False
```

**关键点：**
- `OverallState` 是图内部的"全量状态"，包含所有字段
- `InputState` 是入口过滤器——调用方只需要提供这些字段
- `OutputState` 是出口过滤器——调用方只会看到这些字段
- `context`、`confidence` 等中间状态对外完全不可见
- 如果不指定 `input_schema` 和 `output_schema`，默认都等于 `OverallState`

**最常见的写法——继承方式：**

```python
# 方式1：OverallState 继承 Input + Output（推荐，简洁）
class InputState(TypedDict):
    question: str

class OutputState(TypedDict):
    answer: str

class OverallState(InputState, OutputState):
    context: str       # 额外的内部字段
    confidence: float  # 额外的内部字段

# 方式2：OverallState 独立定义（更灵活，字段名可以不同）
class OverallState(TypedDict):
    question: str      # 与 InputState 共享
    answer: str        # 与 OutputState 共享
    context: str       # 内部独有
    confidence: float  # 内部独有
```

## 4.2 节点自动推断 Input Schema

**一句话：** 节点函数的第一个参数类型注解会被自动推断为该节点的 input schema，节点只接收注解类型中定义的字段

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# === 定义多个状态类型 ===

class OverallState(TypedDict):
    user_input: str
    foo: str
    graph_output: str

class PrivateState(TypedDict):
    bar: str

# === 节点通过参数类型注解声明自己需要的状态子集 ===

def node_1(state: OverallState) -> dict:
    """接收 OverallState 的字段"""
    return {"foo": state["user_input"] + " name"}

def node_2(state: OverallState) -> dict:
    """接收 OverallState 的字段，写入 PrivateState 的字段"""
    return {"bar": state["foo"] + " is"}

def node_3(state: PrivateState) -> dict:
    """只接收 PrivateState 的字段（bar）"""
    # 这里只能看到 bar，看不到 user_input 和 foo
    return {"graph_output": state["bar"] + " Lance"}
```

**自动推断的工作原理：**

```python
# 当你这样写：
def my_node(state: SomeState) -> dict:
    ...

# LangGraph 在 add_node 时会检查函数签名：
# 1. 获取第一个参数的类型注解 → SomeState
# 2. 如果 SomeState 有 type hints（是 TypedDict），自动设为该节点的 input_schema
# 3. 节点执行时，框架只传入 SomeState 中定义的字段
```

**完整可运行示例：**

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class InputState(TypedDict):
    user_input: str

class OutputState(TypedDict):
    graph_output: str

class OverallState(TypedDict):
    user_input: str
    foo: str
    graph_output: str

class PrivateState(TypedDict):
    bar: str

def node_1(state: InputState) -> dict:
    """自动推断：只接收 user_input"""
    return {"foo": state["user_input"] + " name"}

def node_2(state: OverallState) -> dict:
    """自动推断：接收 OverallState 的所有字段"""
    return {"bar": state["foo"] + " is"}

def node_3(state: PrivateState) -> dict:
    """自动推断：只接收 bar"""
    return {"graph_output": state["bar"] + " Lance"}

builder = StateGraph(
    OverallState,
    input_schema=InputState,
    output_schema=OutputState,
)
builder.add_node("node_1", node_1)
builder.add_node("node_2", node_2)
builder.add_node("node_3", node_3)
builder.add_edge(START, "node_1")
builder.add_edge("node_1", "node_2")
builder.add_edge("node_2", "node_3")
builder.add_edge("node_3", END)

graph = builder.compile()

# 输入只需要 user_input
result = graph.invoke({"user_input": "My"})
# 输出只包含 graph_output
print(result)  # {'graph_output': 'My name is Lance'}
```

**关键点：**
- 类型注解不只是文档——LangGraph 真的会读取它来决定传什么给节点
- 所有 Schema 的字段共享同一个 Channel 池，`PrivateState` 的 `bar` 和 `OverallState` 的 `foo` 在同一个池子里
- 节点可以写入任何 Channel 的字段，不受 input schema 限制——input schema 只控制"读"

## 4.3 子图状态隔离

**一句话：** 子图有完全独立的状态空间，父子图通过"共享键"或"显式传参"通信

子图状态隔离有两种实现方式，选择取决于父子图的 Schema 是否有同名字段。

### 方式1：函数包装（Schema 不同时推荐）

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# === 子图：有自己独立的 Schema ===

class SubgraphState(TypedDict):
    bar: str   # 子图独有
    baz: str   # 子图独有

def subgraph_node_1(state: SubgraphState) -> dict:
    return {"baz": "baz"}

def subgraph_node_2(state: SubgraphState) -> dict:
    return {"bar": state["bar"] + state["baz"]}

subgraph_builder = StateGraph(SubgraphState)
subgraph_builder.add_node("sub_1", subgraph_node_1)
subgraph_builder.add_node("sub_2", subgraph_node_2)
subgraph_builder.add_edge(START, "sub_1")
subgraph_builder.add_edge("sub_1", "sub_2")
subgraph_builder.add_edge("sub_2", END)
subgraph = subgraph_builder.compile()

# === 父图：通过函数包装调用子图 ===

class ParentState(TypedDict):
    foo: str

def call_subgraph(state: ParentState) -> dict:
    """手动转换：父图状态 → 子图输入 → 子图输出 → 父图状态"""
    # 1. 父图字段 → 子图字段（foo → bar）
    response = subgraph.invoke({"bar": state["foo"]})
    # 2. 子图字段 → 父图字段（bar → foo）
    return {"foo": response["bar"]}

builder = StateGraph(ParentState)
builder.add_node("node_1", lambda state: {"foo": "hi! "})
builder.add_node("node_2", call_subgraph)
builder.add_edge(START, "node_1")
builder.add_edge("node_1", "node_2")
builder.add_edge("node_2", END)

graph = builder.compile()
result = graph.invoke({"foo": ""})
print(result)  # {'foo': 'hi! baz'}
```

### 方式2：直接添加编译子图（有共享键时推荐）

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# === 子图：与父图有同名字段 foo ===

class SubgraphState(TypedDict):
    foo: str   # 与父图共享的键
    bar: str   # 子图私有

def sub_node_1(state: SubgraphState) -> dict:
    return {"bar": "bar"}

def sub_node_2(state: SubgraphState) -> dict:
    return {"foo": state["foo"] + state["bar"]}

subgraph_builder = StateGraph(SubgraphState)
subgraph_builder.add_node("sub_1", sub_node_1)
subgraph_builder.add_node("sub_2", sub_node_2)
subgraph_builder.add_edge(START, "sub_1")
subgraph_builder.add_edge("sub_1", "sub_2")
subgraph_builder.add_edge("sub_2", END)
subgraph = subgraph_builder.compile()

# === 父图：直接添加编译后的子图 ===

class ParentState(TypedDict):
    foo: str

builder = StateGraph(ParentState)
builder.add_node("node_1", lambda state: {"foo": "hi! "})
builder.add_node("node_2", subgraph)  # 直接传入编译后的子图！
builder.add_edge(START, "node_1")
builder.add_edge("node_1", "node_2")
builder.add_edge("node_2", END)

graph = builder.compile()
result = graph.invoke({"foo": ""})
print(result)  # {'foo': 'hi! bar'}
```

**两种方式对比：**

| 特性 | 函数包装 | 直接添加子图 |
|------|----------|-------------|
| Schema 关系 | 可以完全不同 | 需要有同名共享键 |
| 数据转换 | 手动转换（灵活） | 自动通过共享键传递 |
| 代码量 | 多一个包装函数 | 更简洁 |
| 适用场景 | 字段名不同、需要转换 | 字段名相同、直接传递 |

## 4.4 context_schema 传递运行时配置

**一句话：** `context_schema` 定义不可变的运行时上下文，通过 `Runtime` 对象在节点中访问，不参与状态更新和持久化

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.runtime import Runtime

# === 定义 context schema ===

class MyContext(TypedDict):
    model_name: str
    temperature: float

class State(TypedDict):
    query: str
    answer: str

# === 节点通过 runtime 参数访问 context ===

def generate_node(state: State, runtime: Runtime[MyContext]) -> dict:
    """runtime.context 包含运行时配置"""
    model = runtime.context["model_name"]
    temp = runtime.context["temperature"]
    # 模拟使用不同模型和温度生成回答
    answer = f"[{model}, temp={temp}] 回答：{state['query']}"
    return {"answer": answer}

# === 构建图，指定 context_schema ===

builder = StateGraph(State, context_schema=MyContext)
builder.add_node("generate", generate_node)
builder.add_edge(START, "generate")
builder.add_edge("generate", END)

graph = builder.compile()

# === 调用时通过 context 参数传入 ===

result = graph.invoke(
    {"query": "什么是 RAG？"},
    context={"model_name": "gpt-4", "temperature": 0.7},
)
print(result["answer"])
# [gpt-4, temp=0.7] 回答：什么是 RAG？

# 换一个配置再调用
result = graph.invoke(
    {"query": "什么是 RAG？"},
    context={"model_name": "claude-3", "temperature": 0.3},
)
print(result["answer"])
# [claude-3, temp=0.3] 回答：什么是 RAG？
```

**关键点：**
- `context_schema` 不创建 Channel——它的数据不在状态中，不会被 checkpoint 保存
- 节点通过第二个参数 `runtime: Runtime[ContextType]` 访问
- context 是只读的，节点不能修改它
- 适合放：API keys、模型配置、用户身份信息等"每次调用可能不同但执行过程中不变"的数据
- 不适合放：需要在节点间传递和修改的数据（那应该放在 State 里）

**context vs state 的选择标准：**

| 数据特征 | 放在 State | 放在 Context |
|----------|-----------|-------------|
| 节点间需要传递和修改 | 是 | |
| 每次调用可能不同 | | 是 |
| 执行过程中不变 | | 是 |
| 需要被 checkpoint 保存 | 是 | |
| 包含敏感信息（API key） | | 是（不会被持久化） |

## 这些知识足以

掌握以上 4 个要点，你就能：

1. **构建有清晰边界的图** — Input/Output Schema 分离，调用方只看到需要的字段
2. **让不同节点只看到需要的状态** — 通过类型注解自动推断节点的 input schema
3. **使用子图实现模块化** — 函数包装或共享键，两种方式灵活选择
4. **传递运行时配置而不污染状态** — context_schema 提供只读、不持久化的运行时上下文

## 快速参考

| 需求 | 方案 | 关键代码 |
|------|------|----------|
| 控制图的输入字段 | `input_schema` | `StateGraph(S, input_schema=InputState)` |
| 控制图的输出字段 | `output_schema` | `StateGraph(S, output_schema=OutputState)` |
| 节点只接收部分状态 | 类型注解推断 | `def node(state: SubState)` |
| 子图独立 Schema | 函数包装 | `subgraph.invoke({"bar": state["foo"]})` |
| 子图共享键 | 直接添加子图 | `builder.add_node("sub", compiled_subgraph)` |
| 运行时配置 | `context_schema` | `runtime.context["key"]` |
| 敏感数据传递 | context（不持久化） | `graph.invoke(input, context={...})` |
