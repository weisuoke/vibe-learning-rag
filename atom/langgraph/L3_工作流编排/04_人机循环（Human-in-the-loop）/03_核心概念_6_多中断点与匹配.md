# 03_核心概念_6 - 多中断点与匹配

> 单个节点内可以设置多个 interrupt() 调用，LangGraph 按顺序将 resume 值与中断配对，实现多步人机交互

---

## 概念定义

**多中断点允许在单个节点内设置多个 interrupt() 调用，LangGraph 按调用顺序将 resume 值与中断配对，已匹配的中断直接返回缓存值，未匹配的中断抛出新的 GraphInterrupt。**

这个机制让你可以在一个节点内实现多步表单、多轮确认等复杂交互，而不需要拆分成多个节点。

一句话理解：**多个 interrupt() 就像排队叫号，每个 resume 值按顺序"领走"对应的中断。**

---

## 顺序匹配机制

### 核心规则

多中断点的匹配遵循严格的顺序规则：

```
节点内的 interrupt() 调用：
    interrupt_0("问题A")  ← 第 0 个中断
    interrupt_1("问题B")  ← 第 1 个中断
    interrupt_2("问题C")  ← 第 2 个中断

resume 值列表：
    [resume_0, resume_1, resume_2]

匹配规则：
    interrupt_0 ↔ resume_0
    interrupt_1 ↔ resume_1
    interrupt_2 ↔ resume_2
```

### 匹配过程详解

每次恢复执行时，节点从头重新运行，interrupt() 的行为取决于是否有匹配的 resume 值：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/types.py:420]
# interrupt() 内部逻辑（简化）：
#
# 1. 检查当前任务的 resume 值列表
# 2. 如果当前 interrupt 的序号 < len(resume_values)：
#      → 直接返回 resume_values[序号]（不暂停）
# 3. 如果当前 interrupt 的序号 >= len(resume_values)：
#      → 抛出 GraphInterrupt（暂停执行）
```

### 四种状态

| 状态 | 条件 | interrupt() 行为 |
|------|------|-----------------|
| 首次执行 | 无任何 resume 值 | 第一个 interrupt 就暂停 |
| 部分恢复 | 有部分 resume 值 | 已匹配的返回缓存值，第一个未匹配的暂停 |
| 完全恢复 | 所有 interrupt 都有 resume 值 | 全部返回缓存值，节点正常完成 |
| 按 ID 恢复 | 使用字典形式 resume | 精确匹配指定 ID 的中断 |

## 完整执行流程图

以一个三步表单（姓名 → 年龄 → 确认）为例：

```
═══════════════════════════════════════════════════════════════
第一次执行：resume_values = []
═══════════════════════════════════════════════════════════════

form_node 开始执行
    │
    ├─ name = interrupt("请输入姓名")
    │       序号=0, len(resume_values)=0
    │       0 >= 0 → 抛出 GraphInterrupt ❌ 暂停！
    │
    ├─ age = interrupt(...)        ← 不会执行到
    └─ confirm = interrupt(...)    ← 不会执行到

→ 返回: __interrupt__ = [Interrupt(value="请输入姓名")]

═══════════════════════════════════════════════════════════════
第一次恢复：Command(resume="张三") → resume_values = ["张三"]
═══════════════════════════════════════════════════════════════

form_node 从头重新执行
    │
    ├─ name = interrupt("请输入姓名")
    │       序号=0, len(resume_values)=1
    │       0 < 1 → 返回 "张三" ✅ 继续！
    │
    ├─ age = interrupt("张三，请输入年龄")
    │       序号=1, len(resume_values)=1
    │       1 >= 1 → 抛出 GraphInterrupt ❌ 暂停！
    │
    └─ confirm = interrupt(...)    ← 不会执行到

→ 返回: __interrupt__ = [Interrupt(value="张三，请输入年龄")]

═══════════════════════════════════════════════════════════════
第二次恢复：Command(resume=25) → resume_values = ["张三", 25]
═══════════════════════════════════════════════════════════════

form_node 从头重新执行
    │
    ├─ name = interrupt("请输入姓名")
    │       序号=0 < 2 → 返回 "张三" ✅
    │
    ├─ age = interrupt("张三，请输入年龄")
    │       序号=1 < 2 → 返回 25 ✅
    │
    └─ confirm = interrupt("确认：张三, 25岁？")
            序号=2, len(resume_values)=2
            2 >= 2 → 抛出 GraphInterrupt ❌ 暂停！

→ 返回: __interrupt__ = [Interrupt(value="确认：张三, 25岁？")]

═══════════════════════════════════════════════════════════════
第三次恢复：Command(resume=True) → resume_values = ["张三", 25, True]
═══════════════════════════════════════════════════════════════

form_node 从头重新执行
    │
    ├─ name = interrupt(...)     → 返回 "张三" ✅
    ├─ age = interrupt(...)      → 返回 25 ✅
    └─ confirm = interrupt(...)  → 返回 True ✅

→ 节点正常完成，返回 {"result": "注册成功：张三, 25岁"}
```

## 任务作用域

### resume 值列表的隔离

resume 值列表是按**任务（task）**隔离的，不会跨任务共享：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/types.py]
# resume 值的作用域说明：
#
# - 每个节点执行对应一个"任务"（PregelTask）
# - resume 值列表绑定在该任务上
# - 不同节点的 interrupt 互不影响
# - 同一节点的不同执行实例（如并行 Send）也互不影响
```

### 作用域示例

```python
from typing import TypedDict
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, START, END
from langgraph.types import interrupt, Command


class State(TypedDict):
    result_a: str
    result_b: str


def node_a(state: State):
    """节点 A 有自己的 resume 值列表"""
    name = interrupt("节点A：请输入姓名")      # A 的第 0 个中断
    role = interrupt("节点A：请输入角色")      # A 的第 1 个中断
    return {"result_a": f"{name} ({role})"}


def node_b(state: State):
    """节点 B 有独立的 resume 值列表"""
    city = interrupt("节点B：请输入城市")      # B 的第 0 个中断（独立编号）
    return {"result_b": city}


# 构建图：A → B（串行）
builder = StateGraph(State)
builder.add_node("node_a", node_a)
builder.add_node("node_b", node_b)
builder.add_edge(START, "node_a")
builder.add_edge("node_a", "node_b")
builder.add_edge("node_b", END)

graph = builder.compile(checkpointer=MemorySaver())
config = {"configurable": {"thread_id": "scope-demo"}}

# 执行流程：
# 1. 触发 node_a 的第一个 interrupt
graph.invoke({"result_a": "", "result_b": ""}, config)

# 2. 恢复 node_a 的第一个 interrupt → 触发第二个
graph.invoke(Command(resume="张三"), config)

# 3. 恢复 node_a 的第二个 interrupt → node_a 完成 → 触发 node_b 的 interrupt
graph.invoke(Command(resume="工程师"), config)

# 4. 恢复 node_b 的 interrupt → 全部完成
result = graph.invoke(Command(resume="北京"), config)
print(result)
# {"result_a": "张三 (工程师)", "result_b": "北京"}
```

**关键点**：node_b 的 `interrupt("节点B：请输入城市")` 是它自己的第 0 个中断，与 node_a 的中断编号无关。

## 节点重新执行的影响

### 核心行为

恢复时，节点从**开头**重新执行所有代码，不是从上次中断处继续：

```python
call_count = 0  # 模块级变量，用于演示

def demo_node(state):
    global call_count
    call_count += 1
    print(f"节点执行次数: {call_count}")

    # 假设有 3 个 interrupt
    a = interrupt("问题1")   # 恢复后直接返回缓存值
    print(f"a = {a}")        # 每次恢复都会打印

    b = interrupt("问题2")   # 恢复后直接返回缓存值
    print(f"b = {b}")        # 每次恢复都会打印

    c = interrupt("问题3")
    print(f"c = {c}")

    return {"result": f"{a}-{b}-{c}"}
```

**执行时序**：

```
第一次执行（call_count=1）：
    打印 "节点执行次数: 1"
    interrupt("问题1") → 暂停

第一次恢复（call_count=2）：
    打印 "节点执行次数: 2"
    interrupt("问题1") → 返回缓存值
    打印 "a = ..."
    interrupt("问题2") → 暂停

第二次恢复（call_count=3）：
    打印 "节点执行次数: 3"
    interrupt("问题1") → 返回缓存值
    打印 "a = ..."
    interrupt("问题2") → 返回缓存值
    打印 "b = ..."
    interrupt("问题3") → 暂停

第三次恢复（call_count=4）：
    打印 "节点执行次数: 4"
    所有 interrupt 返回缓存值
    节点完成
```

### 副作用警告

```python
# ❌ 危险：interrupt 之前有副作用操作
def dangerous_node(state):
    send_email("开始处理")     # 每次恢复都会重复发送！
    charge_payment(100)        # 每次恢复都会重复扣款！

    result = interrupt("确认？")
    return {"result": result}


# ✅ 安全：副作用放在 interrupt 之后，或使用幂等操作
def safe_node(state):
    # 读取操作是幂等的，重复执行没问题
    data = fetch_data(state["id"])

    result = interrupt({"data": data, "question": "确认？"})

    # 副作用放在 interrupt 之后（只执行一次）
    if result:
        send_email("处理完成")
    return {"result": result}
```

## 按 ID 恢复

### 精确匹配模式

除了按顺序匹配，LangGraph 还支持通过中断 ID 精确恢复特定中断：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/types.py:368]
# Command.resume 支持两种形式：
#   - 单个值：Command(resume="value") → 按顺序匹配
#   - 字典：Command(resume={"interrupt-id": "value"}) → 按 ID 匹配
```

### 获取中断 ID

```python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, START, END
from langgraph.types import interrupt, Command
from typing import TypedDict


class State(TypedDict):
    result: str


def multi_interrupt_node(state: State):
    name = interrupt("请输入姓名")
    age = interrupt("请输入年龄")
    return {"result": f"{name}, {age}岁"}


builder = StateGraph(State)
builder.add_node("form", multi_interrupt_node)
builder.add_edge(START, "form")
builder.add_edge("form", END)

graph = builder.compile(checkpointer=MemorySaver())
config = {"configurable": {"thread_id": "id-demo"}}

# 第一次执行：触发第一个 interrupt
graph.invoke({"result": ""}, config)

# 获取中断状态，查看中断 ID
snapshot = graph.get_state(config)
for task in snapshot.tasks:
    for intr in task.interrupts:
        print(f"中断 ID: {intr.id}")
        print(f"中断值: {intr.value}")
        # 中断 ID 类似: "a1b2c3d4e5f6..."
```

### 按 ID 恢复示例

```python
# 获取中断 ID
snapshot = graph.get_state(config)
interrupt_id = snapshot.tasks[0].interrupts[0].id

# 按 ID 精确恢复
graph.invoke(
    Command(resume={interrupt_id: "张三"}),
    config
)
```

**按 ID 恢复的优势**：
- 不依赖中断顺序，更加健壮
- 适合中断顺序可能变化的动态场景
- 可以选择性恢复特定中断

## 完整代码示例：多步表单

```python
"""
完整示例：多步表单（姓名 → 年龄 → 确认）
演示多中断点的顺序匹配机制
"""

from typing import TypedDict
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, START, END
from langgraph.types import interrupt, Command


class FormState(TypedDict):
    result: str


def registration_form(state: FormState):
    """多步注册表单 —— 单节点内 3 个 interrupt"""

    # 第 0 个中断：收集姓名
    name = interrupt({
        "step": 1,
        "total_steps": 3,
        "field": "name",
        "prompt": "请输入你的姓名",
        "type": "text",
    })

    # 第 1 个中断：收集年龄（注意：prompt 使用了上一步的 name）
    age = interrupt({
        "step": 2,
        "total_steps": 3,
        "field": "age",
        "prompt": f"{name}，请输入你的年龄",
        "type": "number",
    })

    # 第 2 个中断：确认信息
    confirmed = interrupt({
        "step": 3,
        "total_steps": 3,
        "field": "confirm",
        "prompt": f"请确认注册信息：\n姓名: {name}\n年龄: {age}",
        "type": "boolean",
    })

    if confirmed:
        return {"result": f"注册成功: {name}, {age}岁"}
    else:
        return {"result": "注册已取消"}


# 构建图
builder = StateGraph(FormState)
builder.add_node("form", registration_form)
builder.add_edge(START, "form")
builder.add_edge("form", END)

graph = builder.compile(checkpointer=MemorySaver())
config = {"configurable": {"thread_id": "registration-001"}}


# ========== 模拟完整交互流程 ==========

# 步骤 1：启动表单 → 触发第一个 interrupt
print("=== 步骤 1：启动表单 ===")
result = graph.invoke({"result": ""}, config)

# 检查中断状态
snapshot = graph.get_state(config)
for task in snapshot.tasks:
    for intr in task.interrupts:
        print(f"中断信息: {intr.value}")
# 输出: {"step": 1, "field": "name", "prompt": "请输入你的姓名", ...}


# 步骤 2：输入姓名 → 触发第二个 interrupt
print("\n=== 步骤 2：输入姓名 ===")
result = graph.invoke(Command(resume="张三"), config)

snapshot = graph.get_state(config)
for task in snapshot.tasks:
    for intr in task.interrupts:
        print(f"中断信息: {intr.value}")
# 输出: {"step": 2, "field": "age", "prompt": "张三，请输入你的年龄", ...}


# 步骤 3：输入年龄 → 触发第三个 interrupt
print("\n=== 步骤 3：输入年龄 ===")
result = graph.invoke(Command(resume=25), config)

snapshot = graph.get_state(config)
for task in snapshot.tasks:
    for intr in task.interrupts:
        print(f"中断信息: {intr.value}")
# 输出: {"step": 3, "field": "confirm", "prompt": "请确认注册信息：\n姓名: 张三\n年龄: 25", ...}


# 步骤 4：确认 → 表单完成
print("\n=== 步骤 4：确认注册 ===")
result = graph.invoke(Command(resume=True), config)
print(f"最终结果: {result['result']}")
# 输出: 注册成功: 张三, 25岁
```

## 常见陷阱

### 陷阱 1：中断顺序变化导致匹配错误

```python
# ❌ 危险：interrupt 的顺序依赖运行时条件
def unstable_node(state):
    if state.get("need_name", True):
        name = interrupt("请输入姓名")
    else:
        name = "默认用户"

    age = interrupt("请输入年龄")
    return {"result": f"{name}, {age}"}
```

**问题**：如果第一次执行时 `need_name=True`，interrupt("请输入姓名") 是第 0 个中断。但如果恢复时状态变化导致 `need_name=False`，那么 interrupt("请输入年龄") 变成了第 0 个中断，之前为"姓名"提供的 resume 值会错误地匹配到"年龄"。

```python
# ✅ 安全：保持 interrupt 顺序固定
def stable_node(state):
    # 始终按固定顺序调用 interrupt
    name = interrupt("请输入姓名")
    age = interrupt("请输入年龄")

    # 条件逻辑放在 interrupt 之后
    if not state.get("need_name", True):
        name = "默认用户"

    return {"result": f"{name}, {age}"}
```

### 陷阱 2：循环中的 interrupt 数量不固定

```python
# ❌ 危险：循环次数可能变化
def loop_node(state):
    results = []
    for item in state["items"]:  # items 数量可能变化！
        decision = interrupt(f"处理 {item}？")
        if decision:
            results.append(item)
    return {"results": results}
```

**问题**：如果恢复时 `state["items"]` 的内容或顺序发生变化，interrupt 的匹配就会错乱。

```python
# ✅ 安全：确保循环内容在恢复时不变
def safe_loop_node(state):
    # 使用固定的列表（不依赖可能变化的状态）
    items = state["items_snapshot"]  # 在第一次执行时快照
    results = []
    for item in items:
        decision = interrupt(f"处理 {item}？")
        if decision:
            results.append(item)
    return {"results": results}
```

### 陷阱 3：忘记节点会重新执行

```python
# ❌ 问题：API 调用在每次恢复时都会重复
def expensive_node(state):
    # 这个 API 调用会在每次恢复时重复执行！
    api_result = call_expensive_api(state["query"])

    decision = interrupt({
        "result": api_result,
        "question": "使用这个结果吗？"
    })
    return {"result": api_result if decision else None}


# ✅ 改进：缓存昂贵操作的结果
def cached_node(state):
    # 如果状态中已有结果，直接使用（幂等）
    if state.get("api_result"):
        api_result = state["api_result"]
    else:
        api_result = call_expensive_api(state["query"])

    decision = interrupt({
        "result": api_result,
        "question": "使用这个结果吗？"
    })
    return {
        "api_result": api_result,  # 缓存到状态中
        "result": api_result if decision else None,
    }
```

## 与前端开发的类比

| 多中断点概念 | 前端类比 |
