## 一句话定义

**状态可维护性与演进是确保 LangGraph 工作流能随需求变化而安全升级 Schema，同时保持已有检查点有效可恢复的设计策略。**

---

## 为什么状态演进很重要？

### 生产系统的现实

生产环境中的 LangGraph 工作流不是一次性脚本——它们 7×24 运行，数据库里积累了成千上万个检查点。当业务需求变化时，你必须修改状态 Schema，但不能让旧检查点变成废数据。

```
第 1 周：上线 v1 Schema → 积累 10,000 个检查点
第 3 周：需求变更，要加新字段 → 旧的 10,000 个检查点怎么办？
第 6 周：字段类型要改 → 新旧检查点如何共存？
第 12 周：要删掉废弃字段 → 依赖这些字段的节点怎么处理？
```

### 三个核心挑战

| 挑战 | 说明 | 后果 |
|------|------|------|
| 检查点兼容性 | 旧检查点用旧 Schema 保存，新代码用新 Schema 读取 | 反序列化失败，工作流崩溃 |
| 运行中任务 | 有些工作流正在执行到一半，状态是旧格式 | 节点收到意外数据，逻辑出错 |
| 回溯能力 | 时间旅行需要恢复任意历史状态 | 无法回到旧版本的检查点 |

### 前端类比：API 版本管理

前端开发者对这个问题不陌生——后端 API 升级时，你不能让所有旧版客户端同时崩溃：

```
REST API 版本管理                    LangGraph 状态演进
─────────────────                   ─────────────────
/api/v1/users → v2 加了新字段       State v1 → v2 加了新字段
旧客户端还在用 v1                    旧检查点还是 v1 格式
v1 和 v2 要同时工作                  新代码要能读旧检查点
废弃 v1 前要给迁移期                 废弃字段前要给过渡期
```

## Schema 变更策略

### 策略 1：添加新字段（安全）

添加新字段是最安全的变更——只要提供默认值，旧检查点恢复时缺少的字段会自动填充。

```python
"""
场景：RAG 系统上线后，需要新增 rerank_scores 字段
"""
from typing import TypedDict, Annotated, Optional
from operator import add

# ===== V1：初始版本 =====
class RAGStateV1(TypedDict):
    query: str
    documents: Annotated[list[str], add]
    answer: str

# ===== V2：添加新字段 =====
class RAGStateV2(TypedDict):
    query: str
    documents: Annotated[list[str], add]
    answer: str
    # 新增字段：使用 NotRequired 或在节点中处理缺失情况
    rerank_scores: list[float]       # 新字段
    retrieval_strategy: str          # 新字段

# ===== 节点中安全处理缺失字段 =====
def rerank_node(state: RAGStateV2) -> dict:
    """安全处理：旧检查点可能没有 rerank_scores"""
    scores = state.get("rerank_scores", [])  # 用 .get() 提供默认值
    strategy = state.get("retrieval_strategy", "dense")  # 默认策略

    if not scores:
        # 旧检查点没有分数，执行首次排序
        scores = [1.0 / (i + 1) for i in range(len(state["documents"]))]

    return {
        "rerank_scores": scores,
        "retrieval_strategy": strategy,
    }
```

**关键原则：** 新字段永远要有默认值处理逻辑，因为旧检查点不包含这些字段。

### 策略 2：删除字段（谨慎）

删除字段比添加危险得多——如果有节点还在读取被删除的字段，会直接报错。

```python
"""
场景：废弃 debug_info 字段，分三步走
"""

# ===== 第一步：标记废弃（保留字段，停止写入） =====
class RAGStateStep1(TypedDict):
    query: str
    documents: Annotated[list[str], add]
    answer: str
    debug_info: dict    # 标记为废弃，节点不再写入新数据

def generate_node_step1(state: RAGStateStep1) -> dict:
    """停止写入 debug_info，但不删除字段"""
    return {
        "answer": f"Answer to: {state['query']}",
        # 不再返回 debug_info
    }

# ===== 第二步：移除读取（确认没有节点读取该字段） =====
def check_no_readers():
    """在代码库中搜索，确认没有节点读取 debug_info"""
    # grep -r "debug_info" --include="*.py" src/
    # 确认所有引用都已移除
    pass

# ===== 第三步：从 Schema 中移除 =====
class RAGStateStep3(TypedDict):
    query: str
    documents: Annotated[list[str], add]
    answer: str
    # debug_info 已完全移除
```

**三步删除法：** 停止写入 → 移除读取 → 删除字段。每步之间留足时间，确保没有遗漏。

### 策略 3：修改字段类型（危险）

类型变更是最危险的操作——旧检查点中的数据类型和新 Schema 不匹配，反序列化可能失败或产生错误结果。

```python
"""
场景：documents 从 list[str] 改为 list[Document]
需要编写迁移函数
"""
from pydantic import BaseModel

class Document(BaseModel):
    content: str
    source: str
    score: float = 0.0

# ===== 迁移函数 =====
def migrate_documents(old_docs) -> list[Document]:
    """将旧格式文档迁移为新格式"""
    if not old_docs:
        return []

    # 检查是否已经是新格式
    if old_docs and isinstance(old_docs[0], Document):
        return old_docs

    # 旧格式是 list[str]，迁移为 list[Document]
    return [
        Document(content=doc, source="legacy", score=0.0)
        for doc in old_docs
    ]

# ===== 在节点入口处做迁移 =====
def retrieve_node(state: dict) -> dict:
    """节点入口处处理类型迁移"""
    raw_docs = state.get("documents", [])
    documents = migrate_documents(raw_docs)

    # 后续逻辑使用迁移后的数据
    return {"documents": documents}
```

### 策略 4：重命名字段（需要迁移）

```python
"""
场景：将 user_query 重命名为 question
"""

# ===== 过渡期：两个字段同时存在 =====
class RAGStateTransition(TypedDict):
    user_query: str     # 旧名称（废弃中）
    question: str       # 新名称

def normalize_node(state: RAGStateTransition) -> dict:
    """过渡节点：将旧字段值复制到新字段"""
    question = state.get("question", "") or state.get("user_query", "")
    return {"question": question}
```

### 变更安全等级速查

| 变更类型 | 安全等级 | 是否需要迁移 | 建议 |
|----------|---------|-------------|------|
| 添加字段（有默认值） | 安全 | 否 | 直接添加，节点用 `.get()` |
| 添加字段（无默认值） | 中等 | 否 | 必须提供默认值 |
| 删除字段 | 中等 | 否 | 三步删除法 |
| 修改字段类型 | 危险 | 是 | 编写迁移函数 |
| 重命名字段 | 危险 | 是 | 过渡期双字段共存 |
| 修改 Reducer | 危险 | 可能 | 确保新旧 Reducer 兼容 |

## 检查点版本管理

### LangGraph 的版本迁移机制

LangGraph 内部通过 `_migrate_checkpoint()` 函数管理检查点版本的向后兼容。检查点从 v1 演进到 v4，每个版本增加新的元数据字段。

[来源: sourcecode/langgraph/libs/checkpoint/langgraph/checkpoint/base/__init__.py]

```python
"""
LangGraph 检查点版本迁移路径（源码简化版）

检查点结构随版本演进：
- v1: 基础结构 (id, ts, channel_values, channel_versions, versions_seen)
- v2: 新增 pending_sends（待发送消息队列）
- v3: 将 pending_sends 从 checkpoint 移到独立存储
- v4: 未来扩展预留
"""

# 源码中的迁移逻辑（简化展示）
def _migrate_checkpoint(checkpoint: dict) -> dict:
    """将旧版本检查点迁移到最新版本"""
    version = checkpoint.get("v", 1)

    if version == 1:
        # v1 → v2: 添加 pending_sends 字段
        checkpoint["pending_sends"] = []
        checkpoint["v"] = 2
        version = 2

    if version == 2:
        # v2 → v3: pending_sends 移到独立存储
        # 从 checkpoint 中移除，由 CheckpointSaver 单独管理
        del checkpoint["pending_sends"]
        checkpoint["v"] = 3
        version = 3

    # 每个版本升级都是增量的，保证向后兼容
    return checkpoint
```

### Channel 的序列化与恢复

每个 Channel 通过 `checkpoint()` 和 `from_checkpoint()` 方法实现状态的保存与恢复：

```python
"""
Channel 序列化机制（源码简化版）

[来源: sourcecode/langgraph/libs/langgraph/langgraph/channels/base.py]
"""
from typing import Any

# MISSING 哨兵值：表示 Channel 尚未初始化
class _MISSING_TYPE:
    """标记空 Channel 的哨兵对象"""
    pass

MISSING = _MISSING_TYPE()

class BaseChannel:
    def checkpoint(self) -> Any:
        """将当前 Channel 状态序列化为可存储的值"""
        # 如果 Channel 为空（未初始化），返回 MISSING
        if self.value is MISSING:
            raise EmptyChannelError()
        return self.value

    def from_checkpoint(self, checkpoint: Any) -> "BaseChannel":
        """从检查点数据恢复 Channel 状态"""
        if checkpoint is MISSING:
            # 空检查点 → 空 Channel
            self.value = MISSING
        else:
            self.value = checkpoint
        return self
```

### 版本追踪：channel_versions 与 versions_seen

LangGraph 用两个字典追踪状态变化，实现增量检查点：

```python
"""
版本追踪机制

channel_versions: 每个 Channel 的当前版本号
versions_seen:    每个节点看到的 Channel 版本号

这两个字典配合，让 LangGraph 知道：
1. 哪些 Channel 发生了变化（需要保存）
2. 哪些节点需要重新执行（依赖的 Channel 有更新）
"""

# 概念示例
channel_versions = {
    "query": 3,           # query 被更新了 3 次
    "documents": 5,       # documents 被更新了 5 次
    "answer": 1,          # answer 只被写入 1 次
}

versions_seen = {
    "retrieve_node": {    # retrieve_node 最后看到的版本
        "query": 3,       # 已经看到 query 的第 3 版
        "documents": 3,   # 但只看到 documents 的第 3 版（落后了）
    },
    "generate_node": {
        "query": 3,
        "documents": 5,
        "answer": 1,
    },
}

# 判断节点是否需要重新执行
def needs_rerun(node_name: str) -> bool:
    """如果节点依赖的 Channel 有更新，需要重新执行"""
    seen = versions_seen.get(node_name, {})
    for channel, version in channel_versions.items():
        if channel in seen and seen[channel] < version:
            return True
    return False
```

### 自定义迁移逻辑

当你的业务状态 Schema 发生重大变更时，可以编写自定义迁移器：

```python
"""
自定义状态迁移器：处理业务 Schema 的版本演进
"""
from typing import Any
from datetime import datetime

class StateMigrator:
    """状态 Schema 迁移管理器"""

    def __init__(self):
        self._migrations: dict[tuple[int, int], callable] = {}

    def register(self, from_version: int, to_version: int):
        """注册迁移函数"""
        def decorator(fn):
            self._migrations[(from_version, to_version)] = fn
            return fn
        return decorator

    def migrate(self, state: dict, from_version: int, to_version: int) -> dict:
        """执行迁移链：v1 → v2 → v3 → ... → target"""
        current = from_version
        while current < to_version:
            next_ver = current + 1
            key = (current, next_ver)
            if key not in self._migrations:
                raise ValueError(f"缺少迁移路径: v{current} → v{next_ver}")
            state = self._migrations[key](state)
            state["_schema_version"] = next_ver
            current = next_ver
        return state


# ===== 使用示例 =====
migrator = StateMigrator()

@migrator.register(1, 2)
def migrate_v1_to_v2(state: dict) -> dict:
    """v1 → v2: 添加 rerank_scores 字段"""
    state["rerank_scores"] = []
    state["retrieval_strategy"] = "dense"
    return state

@migrator.register(2, 3)
def migrate_v2_to_v3(state: dict) -> dict:
    """v2 → v3: documents 从 list[str] 改为 list[dict]"""
    old_docs = state.get("documents", [])
    state["documents"] = [
        {"content": doc, "source": "legacy", "score": 0.0}
        if isinstance(doc, str) else doc
        for doc in old_docs
    ]
    return state

# 迁移旧状态
old_state = {
    "_schema_version": 1,
    "query": "什么是 RAG？",
    "documents": ["doc1", "doc2"],
    "answer": "",
}

new_state = migrator.migrate(old_state, from_version=1, to_version=3)
print(f"迁移后版本: v{new_state['_schema_version']}")
print(f"文档格式: {new_state['documents'][0]}")
# 输出:
# 迁移后版本: v3
# 文档格式: {'content': 'doc1', 'source': 'legacy', 'score': 0.0}
```

## 向后兼容设计

### 原则 1：始终提供默认值

```python
"""
所有新字段必须有默认值处理，这是向后兼容的基石
"""
from typing import TypedDict, Annotated, Optional
from operator import add

class EvolvableState(TypedDict, total=False):
    """使用 total=False 让所有字段变为可选"""
    query: str
    documents: Annotated[list[str], add]
    answer: str
    # v2 新增
    confidence_score: float
    # v3 新增
    source_urls: list[str]

# 安全读取模式
def safe_read_node(state: EvolvableState) -> dict:
    """防御性读取：每个字段都有兜底值"""
    query = state.get("query", "")
    docs = state.get("documents", [])
    confidence = state.get("confidence_score", 0.0)
    urls = state.get("source_urls", [])

    return {
        "answer": f"处理了 {len(docs)} 篇文档",
        "confidence_score": confidence or 0.5,
    }
```

### 原则 2：使用 Optional 类型标记新字段

```python
"""
Optional 类型明确表达"这个字段可能不存在"
"""
from typing import TypedDict, Optional

class RAGStateV3(TypedDict):
    # 核心字段（始终存在）
    query: str
    answer: str

    # 可选字段（可能不存在于旧检查点）
    rerank_model: Optional[str]          # None 表示未配置
    max_tokens: Optional[int]            # None 表示使用默认值
    metadata: Optional[dict]             # None 表示无元数据

def process_with_optional(state: RAGStateV3) -> dict:
    """处理可选字段"""
    model = state.get("rerank_model") or "cross-encoder/ms-marco"
    max_tokens = state.get("max_tokens") or 4096
    metadata = state.get("metadata") or {}

    return {"answer": f"使用 {model}，最大 {max_tokens} tokens"}
```

### 原则 3：废弃模式（Deprecation Pattern）

```python
"""
字段废弃的标准流程：标记 → 警告 → 迁移 → 移除
"""
import warnings
from typing import TypedDict

class RAGState(TypedDict):
    query: str
    answer: str
    # 废弃字段
    user_query: str      # DEPRECATED: 使用 query 代替
    debug_mode: bool     # DEPRECATED: 使用 config 传递

def deprecated_field_handler(state: dict) -> dict:
    """处理废弃字段，发出警告并自动迁移"""
    updates = {}

    # 检查废弃字段 user_query
    if "user_query" in state and state["user_query"]:
        if not state.get("query"):
            updates["query"] = state["user_query"]
        warnings.warn(
            "字段 'user_query' 已废弃，请使用 'query'。"
            "将在 v4.0 中移除。",
            DeprecationWarning,
            stacklevel=2,
        )

    # 检查废弃字段 debug_mode
    if "debug_mode" in state:
        warnings.warn(
            "字段 'debug_mode' 已废弃，请通过 config 传递调试选项。",
            DeprecationWarning,
            stacklevel=2,
        )

    return updates
```

### 原则 4：特性开关（Feature Flags）

```python
"""
用状态中的特性开关控制新旧逻辑的切换
"""
from typing import TypedDict, Annotated
from operator import add

class RAGState(TypedDict):
    query: str
    documents: Annotated[list[str], add]
    answer: str
    # 特性开关
    features: dict  # {"use_rerank": True, "use_hybrid_search": False}

def retrieve_node(state: RAGState) -> dict:
    """根据特性开关选择检索策略"""
    features = state.get("features", {})

    if features.get("use_hybrid_search", False):
        # 新逻辑：混合检索
        docs = hybrid_search(state["query"])
    else:
        # 旧逻辑：纯向量检索
        docs = vector_search(state["query"])

    return {"documents": docs}

def rerank_node(state: RAGState) -> dict:
    """根据特性开关决定是否重排序"""
    features = state.get("features", {})

    if features.get("use_rerank", False):
        docs = rerank(state["documents"], state["query"])
        return {"documents": docs}

    # 特性未开启，直接跳过
    return {}

# 占位函数
def hybrid_search(query): return [f"hybrid:{query}"]
def vector_search(query): return [f"vector:{query}"]
def rerank(docs, query): return sorted(docs)
```

### 脱敏模式（Redaction Pattern）

```python
"""
如果必须在状态中记录用户输入，使用脱敏处理
"""
import re
from typing import TypedDict

class AuditState(TypedDict):
    query: str
    raw_input: str           # 用户原始输入（可能含敏感信息）
    redacted_input: str      # 脱敏后的输入

def redact_sensitive_data(text: str) -> str:
    """脱敏处理：替换常见敏感信息模式"""
    patterns = {
        # 邮箱
        r'[\w.-]+@[\w.-]+\.\w+': '[EMAIL]',
        # 手机号（中国）
        r'1[3-9]\d{9}': '[PHONE]',
        # 身份证号
        r'\d{17}[\dXx]': '[ID_CARD]',
        # 银行卡号
        r'\d{16,19}': '[CARD_NUMBER]',
        # API Key 模式
        r'sk-[a-zA-Z0-9]{20,}': '[API_KEY]',
    }
    result = text
    for pattern, replacement in patterns.items():
        result = re.sub(pattern, replacement, result)
    return result

def input_node(state: AuditState) -> dict:
    """入口节点：对用户输入做脱敏"""
    raw = state.get("raw_input", state["query"])
    return {
        "redacted_input": redact_sensitive_data(raw),
    }

# 测试
test_input = "我的邮箱是test@example.com，手机13812345678"
print(redact_sensitive_data(test_input))
# 输出: 我的邮箱是[EMAIL]，手机[PHONE]
```

### 合规考虑

| 法规 | 要求 | 状态设计影响 |
|------|------|-------------|
| GDPR | 用户有权要求删除个人数据 | 检查点中的用户数据必须可定位、可删除 |
| CCPA | 用户有权知道收集了哪些数据 | 状态 Schema 需要文档化哪些字段含个人信息 |
| HIPAA | 医疗数据需要加密存储 | 如果状态含医疗信息，Checkpointer 需要加密 |

```python
"""
GDPR 合规：支持用户数据删除
"""
def delete_user_data(checkpointer, user_id: str):
    """删除指定用户的所有检查点数据"""
    # 1. 查找该用户的所有 thread
    threads = find_threads_by_user(user_id)

    # 2. 删除每个 thread 的检查点
    for thread_id in threads:
        checkpointer.delete({"configurable": {"thread_id": thread_id}})

    # 3. 记录删除操作（审计日志）
    log_deletion(user_id, len(threads))

# 占位函数
def find_threads_by_user(user_id): return []
def log_deletion(user_id, count): pass
```

## 可维护性清单

在每次修改状态 Schema 前，过一遍这个清单：

### 变更前检查

```
[ ] 1. 明确变更类型（添加/删除/修改/重命名）
[ ] 2. 评估安全等级（安全/中等/危险）
[ ] 3. 检查是否有节点依赖被变更的字段
[ ] 4. 确认新字段有默认值处理
[ ] 5. 如果是类型变更，编写迁移函数
```

### 变更中检查

```
[ ] 6. 更新 Schema 定义（TypedDict / Pydantic）
[ ] 7. 更新所有读取该字段的节点（使用 .get() 防御性读取）
[ ] 8. 更新所有写入该字段的节点
[ ] 9. 编写迁移测试（旧状态 → 新状态）
[ ] 10. 更新 Schema 文档和变更日志
```

### 变更后检查

```
[ ] 11. 运行完整测试套件
[ ] 12. 用旧检查点数据做恢复测试
[ ] 13. 验证时间旅行功能正常
[ ] 14. 在 staging 环境验证迁移
[ ] 15. 制定回滚计划
```

### 自动化检查脚本

```python
"""
状态 Schema 健康检查工具
"""
from typing import get_type_hints, TypedDict

def check_state_health(state_class: type, sample_old_state: dict) -> list[str]:
    """检查状态 Schema 的健康度"""
    issues = []
    hints = get_type_hints(state_class)

    # 检查 1：旧状态是否缺少新字段
    for field_name in hints:
        if field_name not in sample_old_state:
            issues.append(
                f"⚠️  字段 '{field_name}' 在旧状态中不存在，"
                f"确保节点使用 .get('{field_name}', default) 读取"
            )

    # 检查 2：旧状态是否有多余字段
    for field_name in sample_old_state:
        if field_name not in hints and not field_name.startswith("_"):
            issues.append(
                f"⚠️  旧状态包含已移除的字段 '{field_name}'，"
                f"确认没有节点依赖此字段"
            )

    # 检查 3：敏感字段检测
    sensitive_patterns = ["password", "secret", "token", "api_key", "credential"]
    for field_name in hints:
        for pattern in sensitive_patterns:
            if pattern in field_name.lower():
                issues.append(
                    f"🔴 字段 '{field_name}' 疑似敏感数据，"
                    f"不应存储在状态中"
                )

    if not issues:
        issues.append("✅ 状态 Schema 健康检查通过")

    return issues


# ===== 使用示例 =====
class CurrentState(TypedDict):
    query: str
    documents: list[dict]
    answer: str
    rerank_scores: list[float]

# 模拟一个 v1 的旧检查点
old_checkpoint = {
    "query": "什么是 RAG？",
    "documents": ["doc1", "doc2"],  # 旧格式是 list[str]
    "answer": "",
    "debug_info": {"step": 1},     # 已被移除的字段
}

results = check_state_health(CurrentState, old_checkpoint)
for r in results:
    print(r)
# 输出:
# ⚠️  字段 'rerank_scores' 在旧状态中不存在，确保节点使用 .get('rerank_scores', default) 读取
# ⚠️  旧状态包含已移除的字段 'debug_info'，确认没有节点依赖此字段
```

## 关键要点总结

1. **添加字段最安全**：始终提供默认值，用 `.get()` 防御性读取
2. **删除字段要三步走**：停止写入 → 移除读取 → 删除定义，每步之间留缓冲期
3. **类型变更最危险**：必须编写迁移函数，做往返测试
4. **LangGraph 内部有版本管理**：`_migrate_checkpoint()` 处理 v1→v2→v3→v4 的迁移，`channel_versions` / `versions_seen` 追踪增量变化
5. **敏感数据不进状态**：API 密钥、密码、令牌通过 config 传递，用户输入做脱敏处理
6. **文档化是投资**：Schema 变更日志和字段文档，在团队协作中价值巨大
7. **自动化检查**：用健康检查脚本在 CI 中验证 Schema 兼容性
