# 一句话总结

> **目标**：用最精炼的语言总结标量过滤与 JSON 索引的核心要点

---

## 核心总结

**标量过滤与 JSON 索引是 Milvus 2.6 的核心特性，通过为标量字段和 JSON 路径创建索引（INVERTED、BITMAP、STL_SORT 等），实现精确条件筛选与向量检索的高效结合，标量索引可实现 30 倍性能提升，JSON Path Index 可实现 100 倍性能提升，合理选择索引类型（BITMAP 用于低基数，INVERTED 用于中高基数）和查询优化（选择性高的条件在前，使用 IN 替代 OR）是构建生产级 RAG 应用的关键。**

---

## 关键要点

### 1. 核心概念

```
标量过滤：
- 定义：对结构化字段进行精确条件筛选
- 操作符：==, !=, >, <, >=, <=, IN, LIKE, IS NULL
- 布尔运算：AND, OR, NOT
- 时间复杂度：O(log n)（有索引）

JSON Path Index：
- 定义：为 JSON 字段的特定路径创建索引
- 语法：metadata['product']['category']
- 类型转换：json_cast_type（varchar, double, int64）
- 性能提升：100 倍
```

### 2. 索引类型

```
INVERTED（倒排索引）：
- 适用：通用场景，中高基数字段
- 性能：O(log n)
- 推荐：生产环境默认选择

BITMAP（位图索引）：
- 适用：低基数字段（< 100个值）
- 性能：O(1)
- 推荐：状态字段、布尔字段

STL_SORT（排序数组）：
- 适用：范围查询
- 性能：O(log n)
- 推荐：数值类型、时间戳

AUTOINDEX（自动索引）：
- 适用：不确定时
- 性能：自动选择
- 推荐：快速原型开发

TRIE（前缀树）：
- 适用：前缀匹配
- 性能：O(m)
- 推荐：自动补全、URL 匹配

NGRAM（N-gram 索引）：
- 适用：模糊匹配
- 性能：O(n)
- 推荐：全文搜索
```

### 3. 执行策略

```
Pre-filtering（前置过滤）：
- 流程：标量过滤 → 向量检索
- 适用：高选择性（> 50%）
- 性能：减少搜索空间

Post-filtering（后置过滤）：
- 流程：向量检索 → 标量过滤
- 适用：低选择性（< 50%）
- 性能：保证召回率

Hybrid（混合策略）：
- 流程：自动选择最优策略
- 适用：所有场景
- 性能：平衡性能和召回率
```

### 4. 性能优化

```
索引优化：
- 为常用查询字段创建索引
- 根据基数选择索引类型
- 避免为所有字段创建索引

查询优化：
- 选择性高的条件在前
- 使用 IN 替代多个 OR
- 避免 NOT 操作符

数据建模：
- 扁平化高频访问字段
- 预计算常用条件
- 标准化字符串

批量操作：
- 批量插入
- 批量查询
- 使用缓存
```

## 核心公式

### 性能提升公式

```
标量索引性能提升：
无索引延迟 / 有索引延迟 = 30x

JSON Path Index 性能提升：
无索引延迟 / 有索引延迟 = 100x

混合查询延迟：
总延迟 = 标量过滤延迟 + 向量检索延迟

Pre-filtering 成本：
O(log n + log m)，其中 m = 候选集大小

Post-filtering 成本：
O(log n + K')，其中 K' = 预检索数量
```

### 索引选择公式

```
基数计算：
基数 = DISTINCT(field) / COUNT(*)

索引选择：
IF 基数 < 0.01:
    使用 BITMAP
ELSE:
    使用 INVERTED

选择性计算：
选择性 = 过滤后数据量 / 总数据量

策略选择：
IF 选择性 > 50%:
    使用 Pre-filtering
ELSE:
    使用 Post-filtering
```

## 常见误区

### 误区1：索引越多越好

```
❌ 错误认知：为所有字段创建索引
✅ 正确做法：只为常用查询字段创建索引

原因：
- 索引有成本（存储、构建、维护）
- 大部分字段很少被查询
- 内存占用增加
```

### 误区2：JSON 查询一定很慢

```
❌ 错误认知：JSON 嵌套访问慢
✅ 正确做法：创建 JSON Path Index

性能数据：
- 无索引：10000ms
- 有索引：100ms
- 性能提升：100倍
```

### 误区3：BITMAP 索引适用于所有场景

```
❌ 错误认知：BITMAP 性能最好
✅ 正确做法：根据基数选择索引

判断标准：
- 低基数（< 100个值）→ BITMAP
- 中高基数 → INVERTED
```

### 误区4：选择性低的条件应该放在前面

```
❌ 错误认知：先执行简单条件
✅ 正确做法：选择性高的条件在前

示例：
❌ "price >= 100 AND price <= 500 AND stock > 0"
✅ "stock > 0 AND price >= 100 AND price <= 500"
```

## 快速参考

### 创建索引

```python
# 标量索引
index_params.add_index("age", "INVERTED")
index_params.add_index("status", "BITMAP")

# JSON Path Index
index_params.add_index(
    field_name="metadata",
    index_name="category_index",
    index_type="INVERTED",
    params={
        "json_path": "metadata['product']['category']",
        "json_cast_type": "varchar"
    }
)
```

### 查询示例

```python
# 标量过滤
filter = "age > 30 AND city == '北京'"
```
