## 问题 1：Plan-and-Execute 和 ReAct 有什么区别？分别适用于什么场景？

### 考察意图

理解两种主流 Agent 推理模式的本质差异和适用条件。

### 优秀回答

```
核心区别是「规划与执行的关系」：

ReAct：交替进行
  Thought → Action → Observation → Thought → Action → ...
  每一步都基于上一步的结果做决策，走一步看一步。

Plan-and-Execute：先规划后执行
  Plan: [步骤1, 步骤2, 步骤3] → Execute(步骤1) → Execute(步骤2) → ...
  先有全局计划，再逐步执行。

三个关键差异：

1. Token 效率
   ReAct：每步都带上完整历史 → Token 消耗随步骤指数增长
   P&E：执行器只关注当前步骤 + 结果摘要 → 线性增长

2. 模型分工
   ReAct：整个过程用同一个模型
   P&E：规划用强模型（GPT-4o），执行用小模型（GPT-4o-mini）→ 降低成本

3. 全局视角
   ReAct：只看到局部，容易陷入死循环或遗漏步骤
   P&E：先看全局再执行，不容易遗漏

适用场景：
  ReAct → 1-2 步简单任务、高度探索性任务
  P&E → 3步以上的复杂任务、可预期的多步工作流
  不是谁更高级，而是谁更合适。
```

### 加分点

- 提到 P&E 的重规划（Replan）机制解决了"计划赶不上变化"的问题
- 提到 LLMCompiler 把两者优势结合：DAG 并行 + 动态调度
- 提到实际生产中常用 P&E 作为外层框架，ReAct 作为单步执行器

## 问题 2：如何实现任务分解？LLM 分解任务有什么常见问题？

### 考察意图

理解多步推理中任务分解的实现方式和工程挑战。

### 优秀回答

```
任务分解的三种策略：

1. LLM 驱动分解（最常用）
   用 Structured Output 让 LLM 直接输出计划：

   class Plan(BaseModel):
       steps: List[str] = Field(description="3-7个执行步骤")

   planner = llm.with_structured_output(Plan)
   plan = planner.invoke("对比 ChromaDB 和 Milvus 的性能")

   优点：灵活，能处理各种任务
   缺点：质量依赖 Prompt 和模型能力

2. 递归分解
   大任务 → 子任务 → 子子任务，直到每个足够简单
   适合：层级很深的复杂任务

3. 迭代分解
   只规划下一步，执行完再规划下一步
   适合：探索性任务，无法预判全貌

LLM 分解的常见问题：

1. 步骤太多太细 → 解决：在 Prompt 中限制 "3-7步"
2. 线性排列，不识别可并行任务 → 解决：要求标注依赖关系
3. 遗漏验证步骤 → 解决：强制要求最后一步是验证/审查
4. 与可用工具不匹配 → 解决：在 Prompt 中列出可用工具

经验法则：
  规划负责"做什么"（What），执行负责"怎么做"（How）。
  粗粒度规划 + 细粒度执行 = 最优策略。
```

### 加分点

- 提到 ReWOO 用变量引用（$1, $2）解决步骤间数据传递
- 提到 LLMCompiler 构建 DAG 识别并行任务
- 提到可以用 Pydantic 的 SubTask 模型定义依赖关系

## 问题 3：什么是重规划（Re-planning）？什么时候该触发重规划？

### 考察意图

考察对动态规划和错误恢复的理解。

### 优秀回答

```
重规划是 Plan-and-Execute 中应对不确定性的核心机制。
不是"规划失败的补救"，而是"根据新信息做更好决策"——
就像 GPS 导航走错路会重新规划路线，这是特性不是 bug。

重规划的触发时机：

1. 步骤执行失败（API 超时、工具报错）
   → 先重试，重试失败则重规划替代方案

2. 步骤结果与预期不符
   → 搜索"Python性能"，结果返回的是 Python 教程
   → 需要调整搜索策略

3. 发现新信息改变方向
   → 调研框架A，发现A已被B替代
   → 后续应该转向研究B

4. 所有步骤完成后评估
   → 检查目标是否达成
   → 可能需要补充步骤

三种重规划模式：

1. 完全重规划：目标发生根本变化，整个计划重做
2. 增量调整：某步骤失败，只修改该步骤和后续
3. 插入补充：发现遗漏，在计划中插入新步骤

实现中的关键防护：
  replan_count >= 3 → 强制终止，防止无限重规划
  这个上限是工程经验值，3次重规划还解决不了通常是规划器本身需要优化。
```

### 加分点

- 提到 LangGraph 中用 `conditional_edges` 实现重规划路由
- 提到重规划次数的经验参考：0次=任务太简单，1-2次=正常，3+次=需要优化Prompt
- 提到自适应规划系统中 `ReplanAction` 的 Structured Output 设计

## 问题 4：Reflection 和 Reflexion 有什么区别？

### 考察意图

区分两个容易混淆的反思机制。

### 优秀回答

```
Reflection（反思）：
  执行后自我评估当前结果的质量。
  就像考试写完后检查一遍试卷。

  特点：
  - 只看当前输出
  - 每次独立评估
  - 不积累经验

  实现：
    assessor = llm.with_structured_output(QualityAssessment)
    assessment = assessor.invoke("评估这份报告的质量...")
    if assessment.overall < 7:
        # 重新生成

Reflexion（反射/经验反思）：
  不仅评估当前结果，还将失败经验存入"经验记忆"，
  下次遇到类似任务时主动参考，避免重复犯错。
  就像考试后记录错题本，下次考试前复习。

  特点：
  - 跨任务积累经验
  - 有经验记忆存储
  - 越用越好

  实现：
    memory = ReflexionMemory()
    past_lessons = memory.search("类似任务的经验")
    # 带着历史经验执行当前任务
    result = execute_with_experience(task, past_lessons)
    # 记录本次经验
    memory.add("这次学到的教训")

关键区别总结：
  Reflection = 考完检查试卷（单次）
  Reflexion = 错题本 + 考前复习（跨次积累）

第三种机制 LLM-as-Judge：
  用独立的 LLM 评估输出质量，类似"同行评审"。
  比自我反思更客观，但成本多一次 LLM 调用。
```

### 加分点

- 提到 Self-Critique 机制：主动挑战自己的结论，找出偏见和弱点
- 提到质量门控的具体维度：完整性、准确性、相关性、深度
- 提到反思循环需要最大轮次限制（如 3 轮），防止无限循环

## 问题 5：用 LangGraph 实现 Plan-and-Execute 的核心步骤是什么？

### 考察意图

考察 LangGraph 实际编码能力。

### 优秀回答

```
核心 5 步：

1. 定义状态（State）
   class PlanExecuteState(TypedDict):
       input: str                                        # 用户输入
       plan: List[str]                                   # 计划步骤
       step_index: int                                   # 当前步骤索引
       past_steps: Annotated[List[tuple], operator.add]  # 已完成步骤（自动累积）
       response: str                                     # 最终响应

   关键点：Annotated + operator.add 让 past_steps 自动追加而不是覆盖。

2. 定义节点函数
   - plan_node：用 Structured Output 生成计划
   - execute_node：执行当前步骤（调用工具）
   - replan_node：评估进度，决定继续/调整/完成

3. 构建图
   graph = StateGraph(PlanExecuteState)
   graph.add_node("planner", plan_node)
   graph.add_node("executor", execute_node)
   graph.add_node("replanner", replan_node)

4. 设置路由边
   graph.set_entry_point("planner")
   graph.add_edge("planner", "executor")
   graph.add_conditional_edges("executor", should_continue, {
       "execute": "executor",    # 继续执行下一步
       "replan": "replanner",    # 所有步骤完成，评估
       "end": END,               # 有最终响应，结束
   })
   graph.add_conditional_edges("replanner", should_continue, {
       "execute": "executor",    # 调整后继续
       "replan": "replanner",
       "end": END,
   })

5. 编译运行
   app = graph.compile()
   result = await app.ainvoke({"input": "用户的复杂任务"})

模型分工的设计哲学：
  planner_llm = ChatOpenAI(model="gpt-4o")       # 规划用强模型
  executor_llm = ChatOpenAI(model="gpt-4o-mini")  # 执行用小模型

  规划需要强推理，执行需要高效快速 → 分工降低成本。
```

### 加分点

- 提到 LangGraph 2026 的 Functional API（@task/@entrypoint）作为替代方案
- 提到 `interrupt()` 实现 Human-in-the-Loop，在规划后暂停等人审批
- 提到 Orchestrator-Worker 模式用于多 Agent 协作场景
- 提到 StateGraph 和 React 的 useReducer 类似：状态 + 动作 → 新状态

## 问题 6：如何评估 Agent 的多步推理质量？有哪些指标？

### 考察意图

考察对 Agent 系统评估和质量保障的理解。

### 优秀回答

```
评估多步推理质量需要从三个维度入手：

1. 规划质量
   - 步骤完整性：是否覆盖了达成目标所需的所有关键步骤
   - 步骤合理性：顺序是否符合逻辑依赖
   - 步骤粒度：是否在 3-7 步的最佳范围内
   - 工具匹配度：每步是否选择了最合适的工具

   衡量方式：人工评审 或 LLM-as-Judge 打分

2. 执行质量
   - 步骤成功率：多少步骤成功执行，多少失败
   - 重试次数：失败后重试的频率（过多说明工具不稳定）
   - 重规划次数：0次可能太简单，1-2次正常，3+次需优化
   - Token 消耗：与完成质量的性价比

   衡量方式：运行日志统计

3. 输出质量
   - 完整性：是否回答了用户问题的所有方面
   - 准确性：信息是否正确
   - 相关性：是否与用户意图匹配
   - 深度：分析是否足够深入

   衡量方式：QualityAssessment 多维度评分（1-10分制）
   综合分 >= 7 视为通过，低于 7 触发改进循环。

实际生产中的做法：
  1. 带反思的 Agent 自带质量门控（Reflection 节点）
  2. 关键场景加 Human-in-the-Loop 人工审核
  3. 记录所有中间步骤用于事后分析
  4. 设置 Token 预算上限防止失控
```

### 加分点

- 提到 Self-Critique 检测偏见和单一视角
- 提到 Reflexion 的经验积累可以渐进提升整体质量
- 提到端到端评估：给定输入，直接评估最终输出的正确性

## 容易被追问的细节

### "Structured Output 在规划中为什么重要？"

```
没有 Structured Output：
  LLM 返回："我觉得可以先搜索一下，然后分析..."
  → 需要额外解析，格式不确定，容易出错

有 Structured Output：
  class Plan(BaseModel):
      steps: List[str]

  LLM 返回：{"steps": ["搜索...", "分析...", "总结..."]}
  → 直接拿到结构化的步骤列表，100%可解析

本质：Structured Output 是 Agent 系统的"类型系统"，
     就像 TypeScript 之于 JavaScript，把运行时错误前移到编译时。
```

### "如何防止 Agent 无限循环？"

```
三道防线：
1. 重规划次数上限（replan_count >= 3 → 强制终止）
2. 反思轮次上限（reflection_count >= 3 → 使用最佳版本）
3. 总步骤/Token 预算（超出预算 → 停止并返回当前最佳结果）

实际代码：
  if replan_count >= 3:
      print("达到最大重规划次数，强制完成")
      return {"response": generate_best_answer(state)}
```

### "P&E 架构中为什么要用两个不同的模型？"

```
经济学原理：
  规划 = 战略决策 → 需要强推理能力 → GPT-4o（贵但准确）
  执行 = 战术操作 → 按指令调工具 → GPT-4o-mini（便宜且快）

实际效果：
  - 5步计划 → 1次 GPT-4o + 5次 GPT-4o-mini
  - 比 6次全用 GPT-4o 省约 60% 成本
  - 执行质量几乎不受影响（步骤已经明确，执行不需要强推理）
```
