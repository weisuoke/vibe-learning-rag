# 核心概念 1：所有权三规则与移动语义

> **一句话定位**：所有权是 Rust 内存安全的根基 -- 理解了所有权三规则和移动语义，你就理解了 Rust 和其他语言最本质的区别。

---

## 1. 所有权三规则

Rust 的所有权系统建立在三条简洁的规则之上。这三条规则由编译器在编译期强制执行，不需要垃圾回收器，也不需要手动 `free`。

[来源: reference/context7_rust_book_01.md | The Rust Programming Language]

### 规则 1：每个值有且仅有一个所有者（owner）

在 Rust 中，每一个值都被绑定到一个变量上，这个变量就是该值的"所有者"。

```rust
fn main() {
    let name = String::from("ZeroClaw");  // name 是 "ZeroClaw" 这个 String 值的所有者
    println!("{}", name);
}
```

**TypeScript 对比：**

```typescript
// TypeScript 中没有"所有者"概念，多个变量可以指向同一个对象
let name = { value: "ZeroClaw" };
let alias = name;  // alias 和 name 都指向同一个对象 -- 两个"主人"
alias.value = "Modified";
console.log(name.value);  // "Modified" -- 被别人改了你可能不知道
```

TypeScript 的对象默认是引用共享的，多个变量可以同时持有对同一个对象的引用。这在方便的同时也带来了"谁在改我的数据？"这种难以追踪的 bug。Rust 通过所有权规则从根源上消除了这类问题。

---

### 规则 2：在任一时刻，只能有一个所有者

当你把一个值赋给另一个变量时，所有权会**转移**（move），原变量不再有效。

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;  // 所有权从 s1 转移到 s2
    // println!("{}", s1);  // 编译错误！s1 已经不再是所有者
    println!("{}", s2);     // OK，s2 是当前所有者
}
```

**TypeScript 对比：**

```typescript
let s1 = "hello";
let s2 = s1;  // 复制了值（基本类型）或复制了引用（对象类型）
console.log(s1);  // 完全没问题 -- s1 仍然可用
console.log(s2);  // 也没问题
```

在 TypeScript 中，赋值不会让原变量失效。但在 Rust 中，赋值堆数据（如 `String`）意味着所有权的转移 -- 这就是"移动语义"，后面会详细讲。

---

### 规则 3：当所有者离开作用域，值被 drop（释放）

Rust 中的每个值，当它的所有者变量离开作用域时，会自动调用 `drop` 函数来释放资源。

```rust
fn main() {
    {
        let s = String::from("hello");
        println!("{}", s);  // s 在这里有效
    }  // s 离开作用域 -> 自动调用 drop -> 堆内存被释放

    // println!("{}", s);  // 编译错误！s 已经不存在了
}
```

**TypeScript 对比：**

```typescript
function example() {
    {
        let s = "hello";
        console.log(s);
    }
    // s 在这里不可访问了（块级作用域），但内存何时释放？
    // 答案：等垃圾回收器"心情好了"再来清理 -- 你无法控制
}
```

在 TypeScript/JavaScript 中，内存释放交给 GC（垃圾回收器），时机不确定。Rust 的规则 3 保证了**确定性释放** -- 离开作用域就立即释放，没有 GC 暂停，没有内存泄漏。

---

### 三规则总结表

| 规则 | 含义 | TypeScript 差异 |
|------|------|-----------------|
| 规则 1 | 每个值有且仅有一个所有者 | TS 对象可被多个变量共享引用 |
| 规则 2 | 任意时刻只能有一个所有者 | TS 赋值不会使原变量失效 |
| 规则 3 | 所有者离开作用域，值被 drop | TS 依赖 GC，释放时机不确定 |

## 2. 移动语义（Move Semantics）

### 什么是移动？

"移动"是 Rust 中赋值堆数据时的默认行为。当你把一个 `String` 赋给另一个变量时，Rust 会：

1. 复制栈上的指针、长度和容量（极快，固定大小）
2. **使原变量无效化**（这一步是关键！）
3. **不复制**堆上的实际数据

[来源: reference/context7_rust_book_01.md | The Rust Programming Language]

### 为什么 Rust 默认移动而非复制？

答案是为了防止**双重释放（double free）**。

假设 Rust 不移动，而是像 TypeScript 那样让两个变量指向同一块堆内存：

```
s1 ──┐
     ├──> 堆上的 "hello" 数据
s2 ──┘

当 s1 离开作用域 -> drop -> 释放堆内存
当 s2 离开作用域 -> drop -> 再次释放同一块内存 !!!  -> 双重释放 bug
```

Rust 的解决方案：移动后，原变量失效。只有一个变量能 drop 这块内存。

```
s1 (已失效)
s2 ──> 堆上的 "hello" 数据

只有 s2 离开作用域时会 drop -> 安全！
```

---

### 移动 vs 浅拷贝 vs 深拷贝

| 操作 | 复制栈数据 | 复制堆数据 | 原变量有效？ | Rust 中的方式 |
|------|-----------|-----------|-------------|---------------|
| 移动（Move） | 是 | 否 | 否 | 默认赋值 `let s2 = s1;` |
| 浅拷贝（Shallow Copy） | 是 | 否 | 是 | Rust 中不存在（会导致双重释放） |
| 深拷贝（Deep Copy） | 是 | 是 | 是 | `.clone()` |

> **关键洞察**：Rust 的"移动"在底层操作上和"浅拷贝"一样（只复制栈数据），但多了一步"使原变量无效"。这一步让 Rust 在零运行时开销下保证了内存安全。

---

### 代码示例：String 赋值时的移动

```rust
fn main() {
    let s1 = String::from("hello");
    // String 在栈上存储：指针 + 长度(5) + 容量(5)
    // 指针指向堆上的实际字节数据 "hello"

    let s2 = s1;
    // 栈上的 (指针, 长度, 容量) 被复制给 s2
    // s1 被标记为无效

    println!("{}", s2);  // OK: "hello"
    // println!("{}", s1);  // 编译错误：value borrowed here after move
}
```

内存变化图解：

```
赋值前：                        赋值后（移动）：
┌─────┐                       ┌─────┐
│ s1  │                       │ s1  │ (已失效)
│ ptr ──────┐                 │     │
│ len:5│    │                 └─────┘
│ cap:5│    │
└─────┘    │                 ┌─────┐
           ▼                 │ s2  │
        ┌──────┐             │ ptr ──────┐
        │hello │             │ len:5│    │
        └──────┘             │ cap:5│    │
                             └─────┘    │
                                        ▼
                                     ┌──────┐
                                     │hello │
                                     └──────┘
```

### 代码示例：函数传参时的移动

```rust
fn main() {
    let s = String::from("hello");
    takes_ownership(s);     // s 的所有权移动到函数参数
    // println!("{}", s);   // 编译错误！s 已经被移走了
}

fn takes_ownership(some_string: String) {
    println!("{}", some_string);
}  // some_string 离开作用域 -> drop -> 堆内存被释放
```

[来源: reference/context7_rust_book_01.md | The Rust Programming Language]

**TypeScript 对比：**

```typescript
function takesValue(someString: string) {
    console.log(someString);
}

let s = "hello";
takesValue(s);
console.log(s);  // 完全没问题 -- 但如果是对象，函数内部的修改会影响外部
```

### 编译错误示例：use after move

这是 Rust 初学者最常见的编译错误之一：

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;  // s1 的值被移动到 s2

    println!("{}", s1);  // 编译错误！
}
```

编译器错误信息：

```
error[E0382]: borrow of moved value: `s1`
 --> src/main.rs:5:20
  |
2 |     let s1 = String::from("hello");
  |         -- move occurs because `s1` has type `String`
3 |     let s2 = s1;
  |              -- value moved here
5 |     println!("{}", s1);
  |                    ^^ value borrowed here after move
```

> **学习技巧**：Rust 编译器的错误信息非常友好，它会告诉你值在哪里被移走了（`value moved here`）以及你在哪里试图使用它（`value borrowed here after move`）。养成认真阅读编译错误的习惯。

## 3. Copy 与 Clone

### Copy trait：自动复制的类型

有些类型在赋值时不会移动，而是**自动复制**。这些类型实现了 `Copy` trait。

```rust
fn main() {
    let x = 42;       // i32 实现了 Copy
    let y = x;        // x 被复制，不是移动
    println!("x = {}, y = {}", x, y);  // 两个都有效！

    let a = true;     // bool 实现了 Copy
    let b = a;
    println!("a = {}, b = {}", a, b);  // 两个都有效！

    let pi = 3.14;    // f64 实现了 Copy
    let pi2 = pi;
    println!("pi = {}, pi2 = {}", pi, pi2);  // 两个都有效！
}
```

**哪些类型自动实现了 Copy？**

| 类型 | 示例 | 原因 |
|------|------|------|
| 所有整数类型 | `i32`, `u64`, `isize` | 固定大小，存储在栈上 |
| 布尔类型 | `bool` | 1 字节，栈上 |
| 浮点类型 | `f32`, `f64` | 固定大小，栈上 |
| 字符类型 | `char` | 4 字节 Unicode，栈上 |
| 纯 Copy 元组 | `(i32, bool)` | 所有元素都是 Copy |

**不实现 Copy 的类型：**

| 类型 | 原因 |
|------|------|
| `String` | 包含堆分配的数据 |
| `Vec<T>` | 包含堆分配的数据 |
| `HashMap<K, V>` | 包含堆分配的数据 |
| 包含非 Copy 字段的元组 | 如 `(String, i32)` |

> **核心规律**：能 Copy 的类型都是**纯栈数据** -- 大小在编译时已知，复制开销极低（就是复制几个字节）。包含堆数据的类型不能 Copy，因为"复制"意味着要复制整块堆内存，这个开销不应该被隐式执行。

[来源: reference/context7_rust_book_01.md | The Rust Programming Language]

---

### Clone trait：显式深拷贝

当你确实需要复制堆数据时，使用 `.clone()` 方法。这是一个**显式**操作，告诉读代码的人："这里有一次潜在的昂贵复制"。

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone();  // 显式深拷贝：堆上的 "hello" 也被完整复制了一份

    println!("s1 = {}, s2 = {}", s1, s2);  // 两者都有效，各自拥有独立的堆数据
}
```

内存变化图解：

```
clone 之后：
┌─────┐          ┌──────┐
│ s1  │          │hello │  <- s1 拥有的堆数据
│ ptr ──────────>│      │
│ len:5│         └──────┘
│ cap:5│
└─────┘

┌─────┐          ┌──────┐
│ s2  │          │hello │  <- s2 拥有的独立堆数据（深拷贝）
│ ptr ──────────>│      │
│ len:5│         └──────┘
│ cap:5│
└─────┘
```

---

### 什么时候用 Clone？什么时候应该避免？

**适合用 Clone 的场景：**

```rust
// 1. 需要保留原值，同时传递给需要所有权的函数
fn main() {
    let config_name = String::from("production");
    log_name(config_name.clone());  // clone 一份给函数
    println!("当前配置: {}", config_name);  // 原值仍可用
}

fn log_name(name: String) {
    println!("日志: {}", name);
}

// 2. 数据量小，clone 开销可忽略
fn main() {
    let short_str = String::from("ok");
    let backup = short_str.clone();  // 几个字节，开销可忽略
}
```

**应该避免 Clone 的场景：**

```rust
// 不好的做法：大数据频繁 clone
fn process_data(data: Vec<u8>) {
    // 如果 data 是 10MB 的文件内容，每次 clone 都要复制 10MB！
}

fn main() {
    let big_data: Vec<u8> = vec![0u8; 10_000_000];  // 10MB
    for _ in 0..100 {
        process_data(big_data.clone());  // 100 次 * 10MB = 1GB 的无意义复制！
    }
}

// 好的做法：用借用（&）代替 clone -- 下一节详细讲
fn process_data_better(data: &Vec<u8>) {
    // 只传递引用，零复制开销
}
```

### `derive(Copy, Clone)` vs 只有 `Clone`

```rust
// 纯栈数据的结构体 -> 可以同时实现 Copy + Clone
#[derive(Copy, Clone, Debug)]
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let p1 = Point { x: 1.0, y: 2.0 };
    let p2 = p1;  // 自动复制（Copy），p1 仍然有效
    println!("p1 = {:?}, p2 = {:?}", p1, p2);
}
```

```rust
// 包含堆数据的结构体 -> 只能 Clone，不能 Copy
#[derive(Clone, Debug)]
struct User {
    name: String,    // String 不是 Copy
    age: u32,
}

fn main() {
    let user1 = User { name: String::from("Alice"), age: 30 };
    // let user2 = user1;        // 移动！user1 失效
    let user2 = user1.clone();   // 必须显式 clone
    println!("user1 = {:?}", user1);  // clone 后两者都有效
}
```

**规则总结：**
- `Copy` 要求所有字段都是 `Copy` 类型 -- 纯栈数据
- `Clone` 没有这个限制 -- 任何类型都可以实现
- 实现了 `Copy` 的类型必须同时实现 `Clone`（Copy 是 Clone 的子集）
- 如果你不确定，**只实现 Clone**，需要复制时显式调用 `.clone()`

## 4. 函数与所有权

### 函数参数获取所有权

当你把一个堆数据类型传给函数时，所有权被移动到函数参数中：

```rust
fn main() {
    let s = String::from("hello");
    takes_ownership(s);
    // s 在这里已经无效了
    // 如果再使用 s，编译器会报错

    let x = 5;
    makes_copy(x);
    // x 是 i32（Copy 类型），所以这里仍然有效
    println!("x = {}", x);  // OK
}

fn takes_ownership(some_string: String) {
    println!("{}", some_string);
}  // some_string 被 drop，内存释放

fn makes_copy(some_integer: i32) {
    println!("{}", some_integer);
}  // some_integer 是 Copy 的，离开作用域没有特殊操作
```

[来源: reference/context7_rust_book_01.md | The Rust Programming Language]

---

### 函数返回值转移所有权

函数可以通过返回值把所有权"还给"调用者：

```rust
fn main() {
    let s1 = gives_ownership();         // 函数返回值的所有权转移给 s1
    println!("s1 = {}", s1);

    let s2 = String::from("hello");
    let s3 = takes_and_gives_back(s2);  // s2 移入函数，函数又把所有权返回给 s3
    // println!("{}", s2);  // 编译错误！s2 已经被移走了
    println!("s3 = {}", s3);  // OK
}

fn gives_ownership() -> String {
    let some_string = String::from("yours");
    some_string  // 返回值 -> 所有权转移给调用者
}

fn takes_and_gives_back(a_string: String) -> String {
    a_string  // 直接返回 -> 所有权归还给调用者
}
```

[来源: reference/context7_rust_book_01.md | The Rust Programming Language]

---

### 所有权的"借而不取"问题

如果每次调用函数都要转移所有权，然后再通过返回值拿回来，代码会变得很麻烦：

```rust
// 很笨拙的做法：先移入再移出
fn calculate_length(s: String) -> (String, usize) {
    let length = s.len();
    (s, length)  // 不得不把 s 也返回，否则调用者就丢失了所有权
}

fn main() {
    let s1 = String::from("hello");
    let (s2, len) = calculate_length(s1);
    // s1 无效了，必须用 s2
    println!("'{}' 的长度是 {}", s2, len);
}
```

这种模式太繁琐了。Rust 为此提供了一个优雅的解决方案 -- **借用（Borrowing）**，即通过引用传递数据而不转移所有权。这将在下一节「核心概念 2：借用与引用」中详细讲解。

```rust
// 预告：借用的优雅写法
fn calculate_length(s: &String) -> usize {  // &String = 借用，不获取所有权
    s.len()
}

fn main() {
    let s1 = String::from("hello");
    let len = calculate_length(&s1);  // 只是借出引用
    println!("'{}' 的长度是 {}", s1, len);  // s1 仍然有效！
}
```

## 5. ZeroClaw 实战实例

以下示例来自 ZeroClaw 项目的真实代码，展示所有权和移动语义在生产代码中的实际运用。

### 示例 1：DelegateTool::new() -- 获取 HashMap 所有权后包装进 Arc

```rust
pub fn new(
    agents: HashMap<String, DelegateAgentConfig>,  // 获取所有权（移动）
    fallback_credential: Option<String>,            // 获取所有权（移动）
    security: Arc<SecurityPolicy>,                  // Arc 的 clone 只增加引用计数
) -> Self {
    Self {
        agents: Arc::new(agents),   // agents 被移动到 Arc::new() 中，包装成共享指针
        security,                   // security 已经是 Arc，赋值只移动 Arc 的栈数据
        fallback_credential,
        // ...
    }
}
```

[来源: sourcecode/zeroclaw/src/tools/delegate.rs]

**解析：**
- `agents: HashMap<String, DelegateAgentConfig>` -- 函数参数接收 `HashMap` 的**所有权**。调用者传入后，原变量失效。
- `Arc::new(agents)` -- `agents` 的所有权被移动到 `Arc` 内部。`Arc`（Atomic Reference Counted）是 Rust 中安全共享所有权的方式。
- 调用者不需要（也不能）在调用 `new()` 之后再使用原来的 `agents` 变量。

---

### 示例 2：tools.into_iter() -- 消费 Vec 获取所有权

```rust
fn boxed_registry_from_arcs(tools: Vec<Arc<dyn Tool>>) -> Vec<Box<dyn Tool>> {
    tools.into_iter().map(ArcDelegatingTool::boxed).collect()
//  ^^^^^^^^^^^^^^^^^
//  into_iter() 消费 tools（获取所有权），之后 tools 不再可用
//  每次迭代产生的 Arc<dyn Tool> 也被移动到 map 闭包中
}
```

[来源: sourcecode/zeroclaw/src/tools/mod.rs]

**解析：**
- `.into_iter()` vs `.iter()` 的关键区别：
  - `.into_iter()` 消费集合，获取每个元素的**所有权**（移动语义）
  - `.iter()` 只借用集合，产生每个元素的**引用**
- 这里使用 `into_iter()` 是因为我们需要把每个 `Arc<dyn Tool>` 的所有权移动到新的 `ArcDelegatingTool` 结构体中。

---

### 示例 3：Arc::clone() -- 低成本共享

```rust
// 在 all_tools_with_runtime 中
let mut tool_arcs: Vec<Arc<dyn Tool>> = vec![
    Arc::new(ShellTool::new(security.clone(), runtime)),
    //                      ^^^^^^^^^^^^^^^^
    //                      Arc<SecurityPolicy> 的 clone -- 只增加引用计数
    Arc::new(FileReadTool::new(security.clone())),
    Arc::new(FileWriteTool::new(security.clone())),
    // ... 更多工具，每个都 clone 了 security
];

let parent_tools = Arc::new(tool_arcs.clone());
// tool_arcs.clone() -- 复制 Vec，但每个元素是 Arc，
// Arc 的 clone 只增加引用计数（原子操作），不深拷贝工具本身
```

[来源: sourcecode/zeroclaw/src/tools/mod.rs]

**解析：**
- `Arc::clone()` 是一种特殊的 clone -- 它**不会**深拷贝内部数据
- 它只做一件事：将引用计数 +1（原子操作，线程安全）
- 成本极低：一次原子整数加法，无内存分配
- 这是 Rust 中在多个所有者之间共享数据的惯用方式

**TypeScript 对比：**

```typescript
// TypeScript 中多个变量默认就是共享引用
const security = new SecurityPolicy();
const tool1 = new ShellTool(security);  // 共享引用
const tool2 = new FileReadTool(security);  // 共享同一个对象

// 问题：谁都可以修改 security，可能导致竞态条件
// Rust 的 Arc 在编译期保证：只有不可变引用可以共享，
```

## 本节要点回顾

| 概念 | 核心要点 |
|------|---------|
| 所有权三规则 | 每个值一个所有者 / 任意时刻一个所有者 / 离开作用域自动 drop |
| 移动语义 | 堆数据赋值默认移动，原变量失效 -- 防止双重释放 |
| Copy | 栈数据（i32, bool, f64, char）自动复制，赋值后两者都有效 |
| Clone | 堆数据显式深拷贝（.clone()），两者独立 |
| 函数传参 | 传入堆数据 = 移动所有权；传入 Copy 类型 = 自动复制 |
| 函数返回 | 返回值把所有权转移给调用者 |
| Arc | 低成本共享所有权，clone 只增加引用计数 |

> **下一节预告**：所有权转移太"霸道"了，每次都要把数据"给出去"。有没有一种方式，只是"借来看看"而不拿走所有权？这就是 Rust 的**借用与引用**（`&T` 和 `&mut T`），下一节详细讲解。
