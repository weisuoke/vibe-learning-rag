# 03_核心概念_1 - 循环边与回路构建

> LangGraph 循环的构建方式：从静态边到条件边，从 Command 到等待边，底层统一为 channel 触发机制

---

## 概念定义

**循环边是指向图中已存在节点的边，使执行流形成有向环。LangGraph 提供四种构建循环的方式，底层统一通过 `branch:to:{node}` channel 触发机制实现。**

理解循环边的构建方式是掌握 LangGraph 迭代机制的基础。无论是简单的 A → B → A 循环，还是复杂的多节点条件循环，都归结为同一个底层模型。

---

## 四种循环构建方式

### 方式 1：静态边回连 — `add_edge("B", "A")`

**最简单的循环构建方式：用普通边将下游节点连回上游节点。**

#### 源码分析

`add_edge()` 本身不区分"正向边"和"回边"。它只做两个检查：`END` 不能作为起点，`START` 不能作为终点。除此之外，任何节点之间都可以连边。

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L785-L837]
def add_edge(self, start_key: str | list[str], end_key: str) -> Self:
    if isinstance(start_key, str):
        if start_key == END:
            raise ValueError("END cannot be a start node")
        if end_key == START:
            raise ValueError("START cannot be an end node")
        # 没有循环检测！直接添加边
        self.edges.add((start_key, end_key))
        return self
```

编译时，`attach_edge()` 将边转化为 channel 写入：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L1297-L1305]
def attach_edge(self, starts: str | Sequence[str], end: str) -> None:
    if isinstance(starts, str):
        if end != END:
            # 向目标节点的触发 channel 写入
            self.nodes[starts].writers.append(
                ChannelWrite(
                    (ChannelWriteEntry(
                        _CHANNEL_BRANCH_TO.format(end), None
                    ),)
                )
            )
```

**关键机制**：每个节点有一个 `branch:to:{node_name}` 的 `EphemeralValue` channel 作为触发器。当上游节点写入这个 channel 时，下游节点在下一个 superstep 中被激活。循环就是让下游节点写入上游节点的触发 channel。

#### 完整示例

```python
"""
静态边回连示例：A → B → A 循环
必须配合条件边提供退出路径，否则无限循环

图拓扑：
    START → a → (条件判断)
                  ↓ 继续     ↓ 结束
                  b → a      END
"""
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    aggregate: Annotated[list, operator.add]


def node_a(state: State):
    print(f'Node A sees {state["aggregate"]}')
    return {"aggregate": ["A"]}


def node_b(state: State):
    print(f'Node B sees {state["aggregate"]}')
    return {"aggregate": ["B"]}


# 终止条件：aggregate 列表长度达到 7 时结束
def route(state: State) -> Literal["b", "__end__"]:
    if len(state["aggregate"]) < 7:
        return "b"
    return END


builder = StateGraph(State)
builder.add_node("a", node_a)
builder.add_node("b", node_b)

builder.add_edge(START, "a")
builder.add_conditional_edges("a", route)
builder.add_edge("b", "a")  # 回边！B 完成后回到 A

graph = builder.compile()
result = graph.invoke({"aggregate": []})
print(result)
# {'aggregate': ['A', 'B', 'A', 'B', 'A', 'B', 'A']}
```

#### 执行流程

```
superstep 1: a 执行 → aggregate = ["A"]
             route 判断: len < 7 → 去 b
superstep 2: b 执行 → aggregate = ["A", "B"]
superstep 3: a 执行 → aggregate = ["A", "B", "A"]  (回边触发)
             route 判断: len < 7 → 去 b
superstep 4: b 执行 → aggregate = ["A", "B", "A", "B"]
superstep 5: a 执行 → aggregate = ["A", "B", "A", "B", "A"]
             route 判断: len < 7 → 去 b
superstep 6: b 执行 → aggregate = ["A", "B", "A", "B", "A", "B"]
superstep 7: a 执行 → aggregate = ["A", "B", "A", "B", "A", "B", "A"]
             route 判断: len = 7, 不 < 7 → END
```

#### 注意事项

- 静态边回连**必须**配合条件边使用，否则会无限循环直到触发 `GraphRecursionError`
- 静态边是无条件的：只要源节点执行完毕，目标节点一定会被触发
- 适合简单的固定循环模式，如 A → B → A → B → ...

### 方式 2：条件边回连 — `add_conditional_edges()`

**最常用的循环构建方式：通过路由函数在运行时决定是否回到已执行过的节点。**

#### 源码分析

条件边的核心是 `BranchSpec` 类，它包装了路由函数的逻辑：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/_branch.py L83]
class BranchSpec(NamedTuple):
    path: Runnable[Any, Hashable | list[Hashable]]  # 路由函数
    ends: dict[Hashable, str] | None                 # 路由映射表
    input_schema: type[Any] | None = None
```

路由执行流程分为两步：`_route()` 调用路由函数，`_finish()` 将结果写入 channel：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/_branch.py L146-L225]
def _route(self, input, config, *, reader, writer):
    if reader:
        value = reader(config)  # 读取最新 state
    else:
        value = input
    result = self.path.invoke(value, config)  # 调用路由函数
    return self._finish(writer, input, result, config)

def _finish(self, writer, input, result, config):
    if not isinstance(result, (list, tuple)):
        result = [result]
    if self.ends:
        destinations = [r if isinstance(r, Send) else self.ends[r] for r in result]
    else:
        destinations = cast(Sequence[Send | str], result)
    # 验证：不能返回 None 或 START
    if any(dest is None or dest == START for dest in destinations):
        raise ValueError("Branch did not return a valid destination")
    entries = writer(destinations, False)  # 写入 channel 触发目标节点
```

编译时，`attach_branch()` 将条件边挂载到源节点：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L1323-L1370]
def attach_branch(self, start, name, branch, *, with_reader=True):
    def get_writes(packets, static=False):
        writes = [
            (ChannelWriteEntry(
                p if p == END else _CHANNEL_BRANCH_TO.format(p), None
            ) if not isinstance(p, Send) else p)
            for p in packets
        ]
        return writes
    # 将 branch runner 添加到源节点的 writers
    self.nodes[start].writers.append(branch.run(get_writes, reader))
```

**关键点**：路由函数返回的节点名被转化为 `branch:to:{node_name}` channel 写入。如果返回的是已执行过的节点名，就形成了循环。

#### 完整示例

```python
"""
条件边回连示例：自校正循环
节点 a 自己决定是否需要重新执行

图拓扑：
    START → generate → evaluate → (条件判断)
                ↑                    ↓ 不合格    ↓ 合格
                └────────────────────┘           END
"""
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    draft: str
    score: float
    attempt: int
    feedback: str


def generate(state: State):
    attempt = state.get("attempt", 0) + 1
    feedback = state.get("feedback", "")

    if feedback:
        # 根据反馈改进
        draft = f"Improved draft (attempt {attempt}), fixing: {feedback}"
    else:
        draft = f"Initial draft (attempt {attempt})"

    return {"draft": draft, "attempt": attempt}


def evaluate(state: State):
    attempt = state["attempt"]
    # 模拟评估：每次尝试分数递增
    score = min(0.3 * attempt, 1.0)
    feedback = "" if score >= 0.8 else f"Score {score:.1f}, needs improvement"
    return {"score": score, "feedback": feedback}


def should_retry(state: State) -> Literal["generate", "__end__"]:
    """条件边：决定是否回到 generate 节点"""
    if state["score"] >= 0.8:
        return END       # 质量达标，结束
    return "generate"    # 回到 generate，形成循环


builder = StateGraph(State)
builder.add_node("generate", generate)
builder.add_node("evaluate", evaluate)

builder.add_edge(START, "generate")
builder.add_edge("generate", "evaluate")
# 条件边：evaluate 后决定是否循环
builder.add_conditional_edges("evaluate", should_retry)

graph = builder.compile()
result = graph.invoke({"draft": "", "score": 0.0, "attempt": 0, "feedback": ""})
print(f"Final: {result['draft']} (score: {result['score']})")
```

#### 条件边 vs 静态边回连

| 特性 | 静态边回连 | 条件边回连 |
|------|-----------|-----------|
| 循环控制 | 无条件回连，需要另一个条件边退出 | 自身包含退出逻辑 |
| 灵活性 | 低，固定路径 | 高，运行时动态决策 |
| 适用场景 | 简单固定循环 | 大多数循环场景 |
| 代码清晰度 | 需要两处定义（回边 + 条件边） | 一处定义（条件边包含回连和退出） |

### 方式 3：等待边 — `add_edge(["c", "d"], "a")`

**多节点同步后再循环：等待多个并行节点全部完成后，再触发目标节点。**

#### 源码分析

当 `add_edge()` 的第一个参数是列表时，使用 `NamedBarrierValue` channel 实现同步：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L785-L837]
def add_edge(self, start_key: str | list[str], end_key: str) -> Self:
    if isinstance(start_key, str):
        # 单源边（前面已分析）
        ...
    else:
        # 多源等待边
        channel_name = f"join:{'+'.join(start_key)}:{end_key}"
        for key in start_key:
            self.waiting_edges.add((key, channel_name, end_key))
        return self
```

等待边使用 `NamedBarrierValue` channel：只有当所有指定的源节点都写入后，目标节点才会被触发。这在循环中非常有用——当循环体包含并行分支时，需要等待所有分支完成后再回到循环起点。

#### 完整示例

```python
"""
等待边 + 循环示例：并行分支完成后回到循环起点

图拓扑：
    START → a → (条件判断)
                  ↓ 继续          ↓ 结束
                  b → c (并行)    END
                  b → d (并行)
                  [c, d] → a (等待边，同步后回到 a)
"""
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    aggregate: Annotated[list, operator.add]


def node_a(state: State):
    print(f'Node A sees {state["aggregate"]}')
    return {"aggregate": ["A"]}


def node_b(state: State):
    print(f'Node B sees {state["aggregate"]}')
    return {"aggregate": ["B"]}


def node_c(state: State):
    print(f'Node C sees {state["aggregate"]}')
    return {"aggregate": ["C"]}


def node_d(state: State):
    print(f'Node D sees {state["aggregate"]}')
    return {"aggregate": ["D"]}


def route(state: State) -> Literal["b", "__end__"]:
    if len(state["aggregate"]) < 7:
        return "b"
    return END


builder = StateGraph(State)
builder.add_node("a", node_a)
builder.add_node("b", node_b)
builder.add_node("c", node_c)
builder.add_node("d", node_d)

builder.add_edge(START, "a")
builder.add_conditional_edges("a", route)
# b 完成后并行触发 c 和 d
builder.add_edge("b", "c")
builder.add_edge("b", "d")
# 等待边：c 和 d 都完成后才回到 a
builder.add_edge(["c", "d"], "a")

graph = builder.compile()
result = graph.invoke({"aggregate": []})
print(result)
# {'aggregate': ['A', 'B', 'C', 'D', 'A', 'B', 'C', 'D', 'A']}
```

#### 执行流程

```
superstep 1: a 执行 → ["A"]
             route: len=1 < 7 → 去 b
superstep 2: b 执行 → ["A", "B"]
superstep 3: c 和 d 并行执行 → ["A", "B", "C", "D"]
             等待边：c ✓ d ✓ → 触发 a
superstep 4: a 执行 → ["A", "B", "C", "D", "A"]
             route: len=5 < 7 → 去 b
superstep 5: b 执行 → ["A", "B", "C", "D", "A", "B"]
superstep 6: c 和 d 并行执行 → ["A", "B", "C", "D", "A", "B", "C", "D"]
             等待边：c ✓ d ✓ → 触发 a
superstep 7: a 执行 → ["A", "B", "C", "D", "A", "B", "C", "D", "A"]
             route: len=9 ≥ 7 → END
```

#### 使用场景

等待边在循环中的典型场景：
- **多 Agent 协作循环**：多个 Agent 并行工作，全部完成后由协调者决定是否继续
- **并行验证循环**：多个验证器并行检查，全部通过后才进入下一轮
- **Map-Reduce 循环**：并行处理后聚合，根据聚合结果决定是否再次 Map

### 方式 4：Command 的 goto — 节点内部自路由

**节点在返回值中直接指定下一个要执行的节点，无需预先定义条件边。**

#### 源码分析

`Command` 是一个数据类，包含 `goto` 字段用于指定下一个节点：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/types.py L367-L417]
@dataclass
class Command(Generic[N]):
    graph: str | None = None
    update: Any | None = None       # 状态更新
    resume: dict | Any | None = None
    goto: Send | Sequence[Send | N] | N = ()  # 下一个节点
```

节点返回 Command 时，`_control_branch` 函数处理 goto：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L1492-L1518]
def _control_branch(value):
    commands = []
    if isinstance(value, Command):
        commands.append(value)
    rtn = []
    for command in commands:
        goto_targets = (
            [command.goto] if isinstance(command.goto, (Send, str))
            else command.goto
        )
        for go in goto_targets:
            if isinstance(go, Send):
                rtn.append((TASKS, go))
            elif isinstance(go, str) and go != END:
                # 写入目标节点的触发 channel（与边的机制完全相同）
                rtn.append((_CHANNEL_BRANCH_TO.format(go), None))
    return rtn
```

**关键点**：Command 的 goto 最终也是写入 `branch:to:{node_name}` channel，与边的机制完全一致。

#### 完整示例

```python
"""
Command goto 循环示例：节点内部决定是否循环

图拓扑：
    START → processor → (Command 决定)
                ↑            ↓ goto="processor"    ↓ goto=END 或无 goto
                └────────────┘                     END

注意：Command 只添加动态边，不会覆盖静态边
"""
from typing import Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import Command


class State(TypedDict):
    value: int
    history: list[int]


def processor(state: State) -> Command[Literal["processor"]]:
    """节点内部决定是否循环回自身"""
    value = state["value"] + 1
    history = state.get("history", []) + [value]

    if value < 5:
        # 继续循环：更新状态并回到自身
        return Command(
            update={"value": value, "history": history},
            goto="processor"  # 回到自身，形成循环
        )
    else:
        # 结束：只更新状态，不指定 goto
        return Command(
            update={"value": value, "history": history}
        )


builder = StateGraph(State)
builder.add_node("processor", processor)
builder.add_edge(START, "processor")
# 不需要 add_conditional_edges！Command 自带路由

graph = builder.compile()
result = graph.invoke({"value": 0, "history": []})
print(result)
# {'value': 5, 'history': [1, 2, 3, 4, 5]}
```

#### Command vs 条件边

| 特性 | Command goto | 条件边 |
|------|-------------|--------|
| 路由定义位置 | 节点函数内部 | 图构建时（外部） |
| 状态更新 | 可以同时更新状态和路由 | 只负责路由 |
| 类型注解 | 需要 `Command[Literal["node"]]` | 不需要 |
| 与静态边关系 | 不覆盖静态边（两者都执行） | 替代静态边 |
| 适用场景 | 路由逻辑与节点逻辑紧密耦合 | 路由逻辑独立于节点 |

**重要警告**：Command 不会覆盖静态边。如果节点同时有静态边和 Command goto，两者都会执行：

```python
# 注意：node_a 会同时去 node_b（静态边）和 node_c（Command）
builder.add_edge("node_a", "node_b")

def node_a(state) -> Command[Literal["node_c"]]:
    return Command(goto="node_c")
# 结果：node_b 和 node_c 都会在下一个 superstep 执行！
```

## 底层统一模型

### 所有边都归结为 channel 写入

无论使用哪种方式构建循环，底层都归结为同一个机制：

```
┌─────────────────────────────────────────────────────────┐
│                    底层统一模型                           │
│                                                         │
│  add_edge("B", "A")                                     │
│    → B.writers.append(                                  │
│        ChannelWrite("branch:to:A", None)                │
│      )                                                  │
│                                                         │
│  add_conditional_edges("B", router_fn)                  │
│    → router_fn 返回 "A"                                 │
│    → ChannelWrite("branch:to:A", None)                  │
│                                                         │
│  Command(goto="A")                                      │
│    → _control_branch 处理                               │
│    → ChannelWrite("branch:to:A", None)                  │
│                                                         │
│  add_edge(["C", "D"], "A")                              │
│    → NamedBarrierValue channel                          │
│    → C 和 D 都完成后                                     │
│    → ChannelWrite("branch:to:A", None)                  │
│                                                         │
│  ─────────────────────────────────────                  │
│  最终都是：写入 branch:to:{node} channel                 │
│  → 目标节点在下一个 superstep 被触发                      │
└─────────────────────────────────────────────────────────┘
```

### channel 名称格式

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L90]
_CHANNEL_BRANCH_TO = "branch:to:{0}"

# 示例：
# 触发节点 "agent" → channel 名 = "branch:to:agent"
# 触发节点 "tools" → channel 名 = "branch:to:tools"
```

### 触发流程

```
节点 B 执行完毕
    ↓
B 的 writers 执行
    ↓
写入 "branch:to:A" channel (EphemeralValue)
    ↓
after_tick() → apply_writes() 应用到 channels
    ↓
下一轮 tick() → prepare_next_tasks()
    ↓
检测到 "branch:to:A" channel 有值
    ↓
节点 A 被加入待执行任务列表
    ↓
runner.tick() 执行节点 A
```

## 循环拓扑图谱

### 拓扑 1：简单二节点循环

```
    START → A ──→ B
            ↑     │
            └─────┘

构建方式：
  builder.add_edge(START, "a")
  builder.add_edge("a", "b")
  builder.add_edge("b", "a")  # 回边
  # 需要在 a 或 b 上加条件边退出
```

### 拓扑 2：自循环（节点回到自身）

```
    START → A ──→ END
            ↑  │
            └──┘

构建方式：
  builder.add_edge(START, "a")
  builder.add_conditional_edges("a", route)
  # route 返回 "a" 形成自循环，返回 END 退出
```

### 拓扑 3：带并行分支的循环

```
    START → A ──→ B ──→ C
            ↑         ↗   (并行)
            │    B ──→ D
            │         │
            └─────────┘  (等待 C 和 D)

构建方式：
  builder.add_edge(START, "a")
  builder.add_conditional_edges("a", route)  # route 返回 "b" 或 END
  builder.add_edge("b", "c")
  builder.add_edge("b", "d")
  builder.add_edge(["c", "d"], "a")  # 等待边
```

### 拓扑 4：多出口循环

```
    START → A ──→ B ──→ (条件判断)
            ↑              ↓ 回到A    ↓ 去C    ↓ 结束
            └──────────────┘          C → END   END

构建方式：
  builder.add_edge(START, "a")
  builder.add_edge("a", "b")
  builder.add_conditional_edges("b", route)
  # route 返回 "a"（循环）、"c"（另一条路径）或 END
  builder.add_edge("c", END)
```

### 拓扑 5：嵌套循环（通过子图）

```
    START → [子图: A ↔ B] → C → (条件判断)
                  ↑                  ↓ 回到子图    ↓ 结束
                  └──────────────────┘              END

构建方式：
  # 内层循环：子图
  inner = StateGraph(State)
  inner.add_node("a", node_a)
  inner.add_node("b", node_b)
  inner.add_edge("a", "b")
  inner.add_conditional_edges("b", inner_route)  # 内层循环

  # 外层循环：主图
  outer = StateGraph(State)
  outer.add_node("inner", inner.compile())
  outer.add_node("c", node_c)
  outer.add_edge(START, "inner")
  outer.add_edge("inner", "c")
  outer.add_conditional_edges("c", outer_route)  # 外层循环
```

## 在 LangGraph 工作流中的应用

### 应用场景 1：ReAct Agent 循环

```python
"""
ReAct Agent：最经典的 LangGraph 循环模式
agent 节点调用 LLM，如果 LLM 返回 tool_calls 则执行工具，否则结束
"""
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class AgentState(TypedDict):
    messages: Annotated[list, lambda x, y: x + y]


def agent(state: AgentState):
    """调用 LLM，可能返回 tool_calls"""
    # 模拟 LLM 调用
    messages = state["messages"]
    if len(messages) < 3:
        return {"messages": [{"role": "assistant", "tool_calls": [{"name": "search"}]}]}
    return {"messages": [{"role": "assistant", "content": "Final answer"}]}


def tools(state: AgentState):
    """执行工具调用"""
    return {"messages": [{"role": "tool", "content": "Tool result"}]}


def should_continue(state: AgentState) -> Literal["tools", "__end__"]:
    last_msg = state["messages"][-1]
    if isinstance(last_msg, dict) and last_msg.get("tool_calls"):
        return "tools"
    return END


builder = StateGraph(AgentState)
builder.add_node("agent", agent)
builder.add_node("tools", tools)

builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", should_continue)
builder.add_edge("tools", "agent")  # 工具执行后回到 agent

graph = builder.compile()
```

### 应用场景 2：RAG 迭代检索

```python
"""
RAG 迭代检索：检索 → 评估 → 不满意则改写查询重新检索
"""
from typing import Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class RAGState(TypedDict):
    query: str
    documents: list[str]
    relevance_score: float
    attempt: int


def retrieve(state: RAGState):
    """检索文档"""
    query = state["query"]
    # 模拟检索
    docs = [f"Doc about {query} (attempt {state.get('attempt', 0) + 1})"]
    return {"documents": docs, "attempt": state.get("attempt", 0) + 1}


def evaluate(state: RAGState):
    """评估检索质量"""
    score = min(0.3 * state["attempt"], 1.0)
    return {"relevance_score": score}


def rewrite_query(state: RAGState):
    """改写查询"""
    return {"query": f"refined: {state['query']}"}


def route(state: RAGState) -> Literal["rewrite_query", "__end__"]:
    if state["relevance_score"] >= 0.8:
        return END
    return "rewrite_query"


builder = StateGraph(RAGState)
builder.add_node("retrieve", retrieve)
builder.add_node("evaluate", evaluate)
builder.add_node("rewrite_query", rewrite_query)

builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "evaluate")
builder.add_conditional_edges("evaluate", route)
builder.add_edge("rewrite_query", "retrieve")  # 回边：重新检索

graph = builder.compile()
```

## 关键特性总结

### 四种方式对比

| 方式 | 源码位置 | 适用场景 | 特点 |
|------|----------|----------|------|
| 静态边回连 | `state.py` L785 | 简单固定循环 | 无条件回连，需配合条件边退出 |
| 条件边回连 | `state.py` L839 | 大多数循环 | 最常用，路由函数决定是否循环 |
| 等待边 | `state.py` L785 (list) | 并行分支后循环 | 多节点同步后再触发 |
| Command goto | `types.py` L367 | 路由与状态更新耦合 | 节点内部自路由，不覆盖静态边 |

### 底层统一

所有方式最终都写入 `branch:to:{node_name}` 的 `EphemeralValue` channel，由 PregelLoop 的 `prepare_next_tasks()` 检测并触发目标节点执行。

## 最佳实践

### 1. 优先使用条件边

```python
# ✅ 推荐：条件边，路由逻辑清晰
builder.add_conditional_edges("evaluate", should_retry)

# ⚠️ 谨慎：Command goto，路由逻辑隐藏在节点内部
def evaluate(state) -> Command[Literal["generate"]]:
    if state["score"] < 0.8:
        return Command(goto="generate")
```

### 2. 每个循环必须有明确的退出路径

```python
# ✅ 正确：有 END 退出路径
def route(state) -> Literal["agent", "__end__"]:
    if done:
        return END
    return "agent"

# ❌ 危险：没有退出路径
def route(state) -> Literal["agent"]:
    return "agent"  # 永远不会结束！
```

### 3. 组合使用多种终止策略

```python
from langgraph.managed.is_last_step import RemainingSteps

class State(TypedDict):
    result: str
    remaining_steps: RemainingSteps  # 感知剩余步数

def route(state) -> Literal["agent", "__end__"]:
    # 策略 1：业务条件终止
    if state.get("task_done"):
        return END
    # 策略 2：主动监控递归限制
    if state["remaining_steps"] <= 2:
        return END  # 优雅退出
    return "agent"

# 策略 3：被动递归限制（兜底）
graph.invoke(inputs, {"recursion_limit": 25})
```

### 4. 避免 Command 与静态边冲突

```python
# ⚠️ 注意：Command 不覆盖静态边
builder.add_edge("a", "b")  # 静态边

def node_a(state) -> Command[Literal["c"]]:
    return Command(goto="c")
# 结果：b 和 c 都会执行！

# ✅ 如果只想去 c，不要定义静态边
# 或者使用条件边替代 Command
```
