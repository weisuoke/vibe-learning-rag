# 反直觉点

> 避免 Milvus 多向量检索的常见误区

---

## 概述

本文列出**3个最常见的误区**，帮助你避免在使用 Milvus 多向量检索时犯错。

---

## 误区 1：多向量检索就是多次调用 search() 然后手动合并结果

### 错误观点

"多向量检索就是对每个向量字段分别调用 search()，然后手动合并结果。"

### 为什么错？

**核心问题**：
- Milvus 提供了专门的 `hybrid_search` API 和 Ranker 机制
- 手动合并结果无法保证融合算法的正确性
- 性能低下，无法利用 Milvus 的优化

**来源**：Context7 官方文档 + 源码分析

**关键数据**：
- **hybrid_search API**：专门用于多向量混合检索
- **Ranker 机制**：RRFRanker 和 WeightedRanker 提供标准融合算法
- **性能优化**：Milvus 内部优化了多向量检索的执行流程

### 为什么人们容易这样错？

**心理原因**：
- 直觉认为"多向量检索 = 多次检索 + 合并"
- 类比传统数据库的多表查询
- 不了解 Milvus 的 hybrid_search API

### 正确理解

**使用 hybrid_search API 和 Ranker**：

```python
from pymilvus import AnnSearchRequest, RRFRanker

# ❌ 错误：手动多次调用 search() 然后合并
dense_results = collection.search(
    data=[dense_vector],
    anns_field="dense_vector",
    param={"metric_type": "IP"},
    limit=10
)

sparse_results = collection.search(
    data=[sparse_vector],
    anns_field="sparse_vector",
    param={"metric_type": "IP"},
    limit=10
)

# 手动合并结果（算法不标准，性能差）
merged_results = manual_merge(dense_results, sparse_results)

# ✅ 正确：使用 hybrid_search API
from pymilvus import AnnSearchRequest, RRFRanker

# 1. 定义多个检索请求
dense_req = AnnSearchRequest(
    data=[dense_vector],
    anns_field="dense_vector",
    param={"metric_type": "IP"},
    limit=10
)

sparse_req = AnnSearchRequest(
    data=[sparse_vector],
    anns_field="sparse_vector",
    param={"metric_type": "IP"},
    limit=10
)

# 2. 使用 RRFRanker 自动融合
results = collection.hybrid_search(
    reqs=[dense_req, sparse_req],
    rerank=RRFRanker(),
    limit=10,
    output_fields=["text"]
)
```

**关键建议**：
- **使用 hybrid_search**：专门的 API，性能更好
- **选择合适的 Ranker**：RRFRanker（不确定权重）或 WeightedRanker（已知权重）
- **避免手动合并**：无法保证算法正确性

## 误区 2：RRFRanker 和 WeightedRanker 的权重需要归一化

### 错误观点

"使用 WeightedRanker 时，权重必须加起来等于 1，需要手动归一化。"

### 为什么错？

**核心问题**：
- WeightedRanker 的权重可以是任意正数
- Milvus 内部会自动处理权重
- 不需要手动归一化

**来源**：Context7 官方文档

**关键数据**：
- **权重范围**：任意正数
- **自动处理**：Milvus 内部自动归一化
- **灵活配置**：可以使用 0.5, 1.0 或 50, 100 等任意比例

### 为什么人们容易这样错？

**心理原因**：
- 类比机器学习中的权重归一化
- 认为权重必须是概率分布（和为1）
- 不了解 Milvus 的内部处理机制

### 正确理解

**权重可以是任意正数**：

```python
from pymilvus import WeightedRanker

# ❌ 错误：手动归一化权重
sparse_weight = 0.7 / (0.7 + 1.0)  # 0.4117
dense_weight = 1.0 / (0.7 + 1.0)   # 0.5882

ranker = WeightedRanker(sparse_weight, dense_weight)

# ✅ 正确：直接使用任意正数
ranker = WeightedRanker(0.7, 1.0)  # 稀疏向量权重 0.7，稠密向量权重 1.0

# ✅ 也可以使用其他比例
ranker = WeightedRanker(70, 100)   # 等价于 0.7, 1.0
ranker = WeightedRanker(7, 10)     # 等价于 0.7, 1.0
ranker = WeightedRanker(0.5, 1.0)  # 稀疏向量权重 0.5，稠密向量权重 1.0

# 执行混合检索
results = collection.hybrid_search(
    reqs=[sparse_req, dense_req],
    rerank=ranker,
    limit=10,
    output_fields=["text"]
)
```

**关键建议**：
- **任意正数**：权重可以是任意正数，不需要归一化
- **相对比例**：重要的是权重之间的相对比例
- **灵活调整**：可以根据实际效果调整权重

## 误区 3：只需要为主向量字段创建索引

### 错误观点

"多向量检索时，只需要为主要的向量字段创建索引，其他向量字段可以不创建索引。"

### 为什么错？

**核心问题**：
- 所有向量字段都必须创建索引
- 如果只为部分向量字段创建索引，LoadCollection 会失败
- 这是 Milvus 的强制要求

**来源**：源码分析（hybridsearch_test.go）

**关键数据**：
- **强制要求**：所有向量字段都必须创建索引
- **LoadCollection 失败**：部分索引会导致加载失败
- **源码证据**：测试代码明确验证了这一点

### 为什么人们容易这样错？

**心理原因**：
- 认为"主向量字段"更重要，其他字段可以忽略
- 类比关系型数据库的可选索引
- 不了解 Milvus 的索引要求

### 正确理解

**所有向量字段都必须创建索引**：

```python
from pymilvus import Collection, FieldSchema, CollectionSchema, DataType

# 定义多个向量字段
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
        FieldSchema(name="dense_vector", dtype=DataType.FLOAT_VECTOR, dim=768),
        FieldSchema(name="sparse_vector", dtype=DataType.SPARSE_FLOAT_VECTOR),
        FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=1000)
    ]
)

collection = Collection("my_collection", schema)

# ❌ 错误：只为 dense_vector 创建索引
index_params = {
    "index_type": "HNSW",
    "metric_type": "IP",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="dense_vector", index_params=index_params)

# 尝试加载 Collection
collection.load()  # ❌ 失败！因为 sparse_vector 没有索引

# ✅ 正确：为所有向量字段创建索引
# 1. 为 dense_vector 创建索引
dense_index_params = {
    "index_type": "HNSW",
    "metric_type": "IP",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="dense_vector", index_params=dense_index_params)

# 2. 为 sparse_vector 创建索引
sparse_index_params = {
    "index_type": "SPARSE_INVERTED_INDEX",
    "metric_type": "IP",
    "params": {"drop_ratio_build": 0.2}
}
collection.create_index(field_name="sparse_vector", index_params=sparse_index_params)

# 3. 加载 Collection
collection.load()  # ✅ 成功！
```

**关键建议**：
- **所有向量字段**：必须为所有向量字段创建索引
- **不同索引类型**：不同类型的向量字段使用不同的索引类型
- **先索引后加载**：索引创建完成后才能加载 Collection

## 其他常见误区

### 误区 4：BM25 检索需要手动生成稀疏向量

**错误观点**：
"使用 BM25 混合检索时，需要手动调用 BM25 算法生成稀疏向量。"

**正确理解**：
- Milvus 提供了 BM25 函数，可以自动将文本转换为稀疏向量
- 检索时直接传入文本即可，不需要手动生成向量

```python
from pymilvus import Function, FunctionType, AnnSearchRequest

# ✅ 正确：使用 BM25 函数自动生成稀疏向量
# 1. 定义 BM25 函数
bm25_function = Function(
    name="bm25_fn",
    function_type=FunctionType.BM25,
    input_field_names=["text"],
    output_field_names="sparse_vector",
    params={}
)
schema.add_function(bm25_function)

# 2. 检索时直接传入文本
query = "what is hybrid search"

sparse_req = AnnSearchRequest(
    data=[query],  # 直接传入文本，不需要手动生成向量
    anns_field="sparse_vector",
    param={"metric_type": "BM25"},
    limit=10
)

dense_req = AnnSearchRequest(
    data=[query_embedding],  # 稠密向量需要手动生成
    anns_field="dense_vector",
    param={"metric_type": "IP"},
    limit=10
)

# 3. 混合检索
results = collection.hybrid_search(
    reqs=[sparse_req, dense_req],
    rerank=RRFRanker(),
    limit=10,
    output_fields=["text"]
)
```

### 误区 5：多向量检索的 limit 参数是总结果数

**错误观点**：
"AnnSearchRequest 的 limit 参数是最终返回的结果数量。"

**正确理解**：
- 每个 AnnSearchRequest 的 limit 是该向量字段的候选数量
- 最终结果数量由 hybrid_search 的 limit 参数控制
- 候选数量应该大于等于最终结果数量

```python
# ✅ 正确：理解 limit 参数的含义
dense_req = AnnSearchRequest(
    data=[dense_vector],
    anns_field="dense_vector",
    param={"metric_type": "IP"},
    limit=20  # 从 dense_vector 字段检索 20 个候选结果
)

sparse_req = AnnSearchRequest(
    data=[sparse_vector],
    anns_field="sparse_vector",
    param={"metric_type": "IP"},
    limit=20  # 从 sparse_vector 字段检索 20 个候选结果
)

# 最终返回 10 个结果（从 20+20=40 个候选中融合）
results = collection.hybrid_search(
    reqs=[dense_req, sparse_req],
    rerank=RRFRanker(),
    limit=10,  # 最终返回 10 个结果
    output_fields=["text"]
)
```

**关键建议**：
- **候选数量**：AnnSearchRequest 的 limit 是候选数量
- **最终结果**：hybrid_search 的 limit 是最终结果数量
- **推荐配置**：候选数量 = 最终结果数量 × 2（提高融合质量）

### 误区 6：RRFRanker 比 WeightedRanker 性能更好

**错误观点**：
"RRF 算法更先进，性能比 WeightedRanker 更好。"

**正确理解**：
- RRFRanker 和 WeightedRanker 适用场景不同
- RRFRanker 适合不确定权重的场景，自动平衡多个检索结果
- WeightedRanker 适合已知权重的场景，可精确控制各向量字段的重要性

```python
# ✅ 场景 1：不确定权重，使用 RRFRanker
# 适用于：探索阶段、不确定各向量字段的重要性
results = collection.hybrid_search(
    reqs=[dense_req, sparse_req],
    rerank=RRFRanker(),  # 自动平衡
    limit=10,
    output_fields=["text"]
)

# ✅ 场景 2：已知权重，使用 WeightedRanker
# 适用于：生产环境、已经调优过权重
results = collection.hybrid_search(
    reqs=[dense_req, sparse_req],
    rerank=WeightedRanker(0.7, 1.0),  # 精确控制权重
    limit=10,
    output_fields=["text"]
)
```

**关键建议**：
- **RRFRanker**：探索阶段、不确定权重
- **WeightedRanker**：生产环境、已知权重
- **性能对比**：两者性能相近，主要区别在于适用场景

### 误区 7：多向量字段必须有相同的维度

**错误观点**：
"多向量检索时，所有向量字段必须有相同的维度。"

**正确理解**：
- 不同向量字段可以有不同的维度
- 每个向量字段独立配置维度和类型
- 支持 FloatVector、BinaryVector、SparseFloatVector、BFloat16Vector 等多种类型

```python
# ✅ 正确：不同向量字段可以有不同的维度
schema = CollectionSchema(
    fields=[
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
        FieldSchema(name="dense_vector_768", dtype=DataType.FLOAT_VECTOR, dim=768),  # 768 维
        FieldSchema(name="dense_vector_1536", dtype=DataType.FLOAT_VECTOR, dim=1536),  # 1536 维
        FieldSchema(name="sparse_vector", dtype=DataType.SPARSE_FLOAT_VECTOR),  # 稀疏向量（动态维度）
        FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=1000)
    ]
)

collection = Collection("my_collection", schema)

# 为每个向量字段创建索引
collection.create_index(
    field_name="dense_vector_768",
    index_params={"index_type": "HNSW", "metric_type": "IP", "params": {"M": 16}}
)

collection.create_index(
    field_name="dense_vector_1536",
    index_params={"index_type": "HNSW", "metric_type": "IP", "params": {"M": 16}}
)

collection.create_index(
    field_name="sparse_vector",
    index_params={"index_type": "SPARSE_INVERTED_INDEX", "metric_type": "IP"}
)
```

## 核心要点总结

1. **多向量检索就是多次调用 search() 然后手动合并结果**：使用 hybrid_search API 和 Ranker 机制
2. **RRFRanker 和 WeightedRanker 的权重需要归一化**：权重可以是任意正数，不需要归一化
3. **只需要为主向量字段创建索引**：所有向量字段都必须创建索引
4. **BM25 检索需要手动生成稀疏向量**：使用 BM25 函数自动生成
5. **多向量检索的 limit 参数是总结果数**：AnnSearchRequest 的 limit 是候选数，hybrid_search 的 limit 是最终结果数
6. **RRFRanker 比 WeightedRanker 性能更好**：两者适用场景不同，性能相近
7. **多向量字段必须有相同的维度**：不同向量字段可以有不同的维度
