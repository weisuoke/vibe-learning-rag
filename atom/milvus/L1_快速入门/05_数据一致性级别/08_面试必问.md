## 面试题：Milvus的三种一致性级别有什么区别？如何选择？

### ❌ 普通回答（60分）

"Milvus有Strong、Bounded、Eventually三种一致性级别。Strong最强，Eventually最弱。Strong保证数据立即可见，Eventually有延迟。根据需求选择，需要准确性就用Strong，需要性能就用Eventually。"

### ✅ 优秀回答（90分）

#### 1. 本质理解

"Milvus的一致性级别源于CAP定理。在分布式向量数据库中，数据通过WAL异步复制到多个节点，一致性级别决定了读取操作需要等待多久才能看到最新写入。"

#### 2. 三种级别详解

**Strong一致性（CP选择）**：
- **保证**：所有写入立即可见，使用最新时间戳作为GuaranteeTs
- **实现**：QueryNode等待ServiceTime满足GuaranteeTs才执行搜索
- **代价**：
  - 全索引加载到内存（内存消耗大）
  - 同步等待WAL（延迟最高）
  - 吞吐量降低30-50%
- **适用**：金融交易、实时库存、关键业务数据

**Bounded一致性（平衡选择，默认）**：
- **保证**：时间窗口内的写入可见，默认staleness bound = 0s
- **实现**：允许一定延迟但保证在时间窗口内
- **优势**：
  - 性能比Strong好20-30%
  - 一致性比Eventually强
  - 70%生产环境的选择
- **适用**：大多数生产场景、文档问答、推荐系统

**Eventually一致性（AP选择）**：
- **保证**：最终一致，无时间保证
- **实现**：完全异步，不等待WAL同步
- **优势**：
  - 延迟最低（比Strong快50-70%）
  - 吞吐量最高
  - 向量数据库行业标准
- **适用**：批量导入、日志分析、高吞吐场景

#### 3. 实际选择标准

**决策树**：
```
1. 是否是金融/关键数据？
   是 → Strong
   否 → 继续

2. 是否是批量/分析场景？
   是 → Eventually
   否 → 继续

3. 其他场景
   → Bounded（默认）
```

**RAG场景推荐**：
- 批量文档导入：Eventually（性能优先）
- 交互式问答：Session（用户体验）
- 实时知识库：Bounded（平衡选择）
- 金融/法律RAG：Strong（准确性优先）

#### 4. 性能对比

| 级别 | 延迟 | 吞吐量 | 内存 | 一致性 |
|------|------|--------|------|--------|
| Strong | 120ms | 1000 QPS | 400GB | 100% |
| Bounded | 60ms | 1500 QPS | 100GB | 99% |
| Eventually | 40ms | 2000 QPS | 100GB | 95% |

#### 5. 代码示例

```python
from pymilvus import Collection, ConsistencyLevel

# 创建集合时指定默认级别
collection = Collection(
    name="docs",
    schema=schema,
    consistency_level="Bounded"  # 默认级别
)

# 搜索时可以覆盖集合级别
results = collection.search(
    data=[query_vector],
    anns_field="vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=10,
    consistency_level="Eventually"  # 临时降级
)
```

#### 6. 常见误区

**误区1**：一致性级别越强越好
- **真相**：Strong代价高，大多数场景不需要

**误区2**：Eventually不可靠
- **真相**：Eventually是行业标准，最终一定一致

**误区3**：Bounded是鸡肋
- **真相**：Bounded是最佳默认，70%生产环境的选择

#### 7. 生产经验

"在实际项目中，我们使用Bounded作为默认级别，覆盖了90%的场景。对于批量数据导入，临时降级到Eventually以提升性能。只有在处理金融数据时才使用Strong。这样的配置在保证数据准确性的同时，性能提升了30%以上。"

#### 8. 底层实现机制

**Milvus 2.6的实现**：

1. **Streaming Node架构**：
   - 使用跨节点状态机维护一致性
   - 协调器生成全局版本化查询视图
   - 同步到所有streaming node和query node

2. **Woodpecker WAL**：
   - 基于对象存储的WAL
   - Checkpoint机制跟踪同步进度
   - 锁管理确保写入一致性

3. **GuaranteeTs机制**：
   - Strong：使用最新时间戳
   - Bounded：最新时间戳 - staleness bound
   - Eventually：无时间戳保证

**代码层面**（Golang实现）：

```go
// 源码参考：internal/streamingnode/server/wal/consistency.go
type ConsistencyLevel int

const (
    Strong ConsistencyLevel = iota
    Bounded
    Eventually
)

func (w *WAL) Read(ctx context.Context, level ConsistencyLevel) (*Record, error) {
    switch level {
    case Strong:
        return w.readWithSync(ctx)  // 同步等待
    case Bounded:
        return w.readWithBoundedStaleness(ctx, w.maxStaleness)  // 时间窗口
    case Eventually:
        return w.readAsync(ctx)  // 异步读取
    }
}
```

#### 10. 监控和调优

**生产环境监控指标**：

```python
import time
import numpy as np
from pymilvus import Collection

def monitor_consistency_performance(collection):
    """监控不同一致性级别的性能"""
    levels = ["Strong", "Bounded", "Eventually"]
    results = {}

    for level in levels:
        latencies = []
        for _ in range(100):
            start = time.time()
            collection.search(
                data=[np.random.rand(1024).tolist()],
                anns_field="vector",
                param={"metric_type": "L2", "params": {"nprobe": 10}},
                limit=10,
                consistency_level=level
            )
            latencies.append(time.time() - start)

        results[level] = {
            "mean": np.mean(latencies),
            "p95": np.percentile(latencies, 95),
            "p99": np.percentile(latencies, 99)
        }

    return results

# 输出示例：
# Strong:     mean=120ms, p95=180ms, p99=250ms
# Bounded:    mean=60ms,  p95=90ms,  p99=120ms
# Eventually: mean=40ms,  p95=60ms,  p99=80ms
```

**调优建议**：
1. 监控P95/P99延迟，不只是平均值
2. 根据业务SLA选择合适级别
3. 使用A/B测试验证性能提升
4. 定期review一致性级别配置

### 1. 实际项目经验

"在我们的RAG项目中，初期使用Strong一致性，发现P99延迟达到300ms，用户体验不佳。分析后发现，文档更新频率是每小时一次，完全不需要Strong。切换到Bounded后，P99延迟降到100ms，用户满意度提升20%。对于批量数据导入，使用Eventually，吞吐量提升了50%。"

"如果遇到'搜索不到刚插入的数据'问题，首先检查一致性级别。如果使用Eventually，这是正常行为。解决方案：
1. 切换到Session（交互式场景）
2. 切换到Bounded（一般场景）
3. 添加延迟等待（批量场景）"
