> 条件分支策略的 2 个高频面试问题，每个包含普通回答和出彩回答

## 问题1：LangGraph 中条件边和普通边有什么区别？什么时候用条件边？

### 普通回答（不出彩）

"条件边可以根据条件选择不同的下一个节点，普通边是固定的。当需要根据状态做判断时用条件边。"

**问题**：只说了表面区别，没有展示对 LangGraph 架构的深入理解。

### 出彩回答（推荐）

**条件边有三层含义，从浅到深：**

**第一层：控制流层面**

普通边是编译时确定的 A→B，条件边是运行时决策的 A→{B,C,D}。

```python
# 普通边：确定性，A 执行完一定到 B
builder.add_edge("process", "output")

# 条件边：运行时决策，A 执行完可能到 B、C 或 D
builder.add_conditional_edges("process", route_function)
```

**第二层：实现机制层面**

条件边通过路由函数实现，LangGraph 支持三种路径推断方式：

| 方式 | 写法 | 适用场景 |
|------|------|----------|
| Literal 类型注解 | `def route(s) -> Literal["a", "b"]:` | 推荐默认写法，自动推断路径 |
| 显式 path_map | `add_conditional_edges(src, fn, {"x": "a", "y": "b"})` | 路由返回值与节点名不同时 |
| 无注解 | `def route(s): return ...` | 不推荐，图可视化会全连接 |

三种方式的核心区别在于 LangGraph 如何知道"有哪些可能的目标节点"——Literal 从类型注解推断，path_map 从字典的 values 推断，无注解则无法推断。

**第三层：架构设计层面**

条件边是 Agent 智能行为的基础。在 ReAct 模式中，LLM 的输出决定下一步行动，这个"决定"就是通过条件边实现的：

```python
# LLM 输出 action → 条件边路由 → 对应工具节点
def route_action(state) -> Literal["search", "calculate", "answer"]:
    return state["llm_decision"]  # LLM 的决策驱动路由
```

没有条件边，就没有 Agent 的"自主决策"能力。

**底层实现原理（加分项）：**

LangGraph 编译时，条件边会被转换为 `Branch` 对象。`Branch` 内部持有路由函数和 path_map，在图执行时被调用：

```python
# LangGraph 内部简化逻辑
class Branch:
    def __init__(self, path, path_map, then=None):
        self.path = path          # 路由函数
        self.path_map = path_map  # 路径映射表

    def run(self, state):
        result = self.path(state)           # 调用路由函数
        target = self.path_map[result]      # 查找目标节点
        return target                       # 返回下一个要执行的节点
```

这意味着条件边的开销非常小——就是一次函数调用 + 一次字典查找。

**何时使用条件边：**
- 需要根据状态做决策时（if-else 逻辑）
- Agent 需要决定是否调用工具、调用哪个工具
- 需要错误处理和回退路径（成功 → 下一步，失败 → 重试）
- 需要循环控制（继续 vs 停止）

**何时不需要条件边：**
- 线性流程（用 `add_edge` 或 `add_sequence`）
- 所有路径都会执行（用 `Send` 做并行分发）
- 路由决策和状态更新需要原子操作（用 `Command`）

## 问题2：解释 LangGraph 中 Send、Command 和 add_conditional_edges 的区别和使用场景

### 普通回答（不出彩）

"Send 用于并行，Command 用于动态路由，add_conditional_edges 用于条件分支。它们各有各的用途。"

**问题**：只是复述了文档的一句话定义，没有展示对三者关系的系统理解。

### 出彩回答（推荐）

三者是 LangGraph 路由能力的三个层次，解决不同维度的问题：

**核心对比表：**

| 维度 | add_conditional_edges | Send | Command |
|------|----------------------|------|---------|
| 定义位置 | 边定义（图结构层） | 条件边的返回值 | 节点的返回值 |
| 决策时机 | 路由函数执行时 | 路由函数执行时 | 节点函数执行时 |
| 状态传递 | 共享图状态 | 自定义子状态（隔离） | 可同时更新图状态 |
| 并行能力 | 不支持（选一条路） | 核心能力（返回多个 Send） | 通过 goto 列表实现 |
| 可视化 | 友好（显示所有可能路径） | 部分友好（显示目标节点） | 不友好（运行时才知道去哪） |
| 跨图导航 | 不支持 | 不支持 | 支持（`graph=Command.PARENT`） |

**三者的本质区别用一句话概括：**

- `add_conditional_edges`：在边上做选择——"走哪条路？"
- `Send`：在边上做复制——"同时走多条路，每条路带不同行李"
- `Command`：在节点里做决策——"我自己决定去哪，顺便更新一下状态"

**代码对比：**

```python
# ---- add_conditional_edges：选一条路走 ----
def route(state) -> Literal["a", "b"]:
    return "a" if state["score"] > 0.8 else "b"
builder.add_conditional_edges("node", route)

# ---- Send：同时走多条路 ----
def fan_out(state):
    return [Send("process", {"item": x}) for x in state["items"]]
builder.add_conditional_edges("split", fan_out)

# ---- Command：节点内部决定去哪 ----
def smart_node(state) -> Command[Literal["a", "b"]]:
    if state["score"] > 0.8:
        return Command(update={"quality": "high"}, goto="a")
    return Command(update={"quality": "low"}, goto="b")
```

**选择原则（决策树）：**

```
需要路由吗？
├── 是 → 需要并行吗？
│   ├── 是 → 运行时才知道并行数量？
│   │   ├── 是 → Send（动态 Map-Reduce）
│   │   └── 否 → 多个 add_edge 即可
│   └── 否 → 路由时需要同时更新状态？
│       ├── 是 → Command（原子操作：路由 + 状态更新）
│       └── 否 → add_conditional_edges（最简单）
└── 否 → add_edge（固定边）
```

**实际项目中的使用比例**：大约 80% 用 `add_conditional_edges`，15% 用 `Send`，5% 用 `Command`。大多数场景用条件边就够了，Send 在批量处理场景（文档摘要、并行评估）中不可替代，Command 主要用在需要跨子图通信或需要路由与状态更新原子化的高级场景。

**真实项目举例（加分项）：**

```python
# 场景1：RAG 质量路由 → add_conditional_edges
# 检索后评估质量，决定是直接生成还是重新检索
def quality_route(state) -> Literal["generate", "re_retrieve"]:
    return "generate" if state["score"] > 0.8 else "re_retrieve"
builder.add_conditional_edges("evaluate", quality_route)

# 场景2：批量文档处理 → Send
# 10 篇文档并行生成摘要，数量运行前未知
def distribute(state):
    return [Send("summarize", {"doc": d}) for d in state["docs"]]
builder.add_conditional_edges("loader", distribute)

# 场景3：子图向父图报告结果 → Command
# 子 Agent 完成任务后，通知父 Agent 并更新全局状态
def sub_agent_done(state) -> Command:
    return Command(
        update={"result": state["output"]},
        goto="coordinator",
        graph=Command.PARENT,  # 跨图导航
    )
```

## 面试加分技巧

### 回答条件分支相关问题时的通用策略

1. **分层回答**：先说"是什么"（API 层面），再说"为什么这样设计"（架构层面），最后说"实际怎么选"（工程层面）
2. **画图辅助**：条件分支天然适合用流程图解释，面试时在白板上画出 `节点 → 条件边 → 多个目标` 的结构
3. **提到循环守卫**：大多数候选人只会说条件边的基本用法，主动提到"条件边 + 循环 = 需要循环守卫"会显得你有生产经验
4. **对比传统方案**：提到 LangGraph 的条件边 vs 传统 if-else 的区别——条件边是声明式的、可可视化的、可序列化的，而 if-else 是命令式的、不可观测的

### 容易被追问的点

面试官在你回答完上述问题后，可能会追问以下问题，提前准备好：

**追问1："条件边的路由函数可以是异步的吗？"**

> 可以。LangGraph 支持 `async def route(state)` 作为路由函数。在需要调用外部服务做路由决策时（比如调用 LLM 判断意图），异步路由函数可以避免阻塞。但要注意，路由函数应该尽量轻量——如果路由逻辑很重，考虑把它放到前一个节点里，让路由函数只做简单的状态读取。

**追问2："条件边可以返回 END 吗？"**

> 可以。路由函数返回 `END`（即 `"__end__"`）会让图在该分支直接结束。这在错误处理中很常见——如果检测到不可恢复的错误，直接路由到 END 而不是继续执行。

```python
from langgraph.graph import END

def route_with_early_exit(state) -> Literal["process", "__end__"]:
    if state.get("fatal_error"):
        return END          # 直接结束
    return "process"        # 继续处理
```

**追问3："多个条件边可以从同一个节点出发吗？"**

> 不可以。一个源节点只能有一组条件边（一次 `add_conditional_edges` 调用）。如果需要多层路由逻辑，应该拆成多个节点，每个节点负责一层决策。
