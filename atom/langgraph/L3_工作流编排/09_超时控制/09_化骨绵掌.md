# 化骨绵掌

这一部分不再讲“是什么”，而是讲 **真正做系统时怎么下手**。

你可以把它理解为：把“超时控制”从概念压缩成行动框架。

---

## 一、先画超时责任图，不要先填秒数

很多团队的做法是：

1. 线上慢了；
2. 找到一个 timeout 参数；
3. 从 30 改成 60；
4. 继续上线。

这通常只是在延后下一次事故。

真正应该先做的是画出责任图：

```text
哪一层负责什么等待？
哪一层有权决定停止等待？
停止等待后谁负责恢复？
```

例如：

| 层级 | 负责对象 | timeout 策略 | 超时后动作 |
|---|---|---|---|
| HTTP client | 单次网络调用 | `httpx.Timeout` | retry / fallback |
| node | 某个异步工具流程 | `asyncio.wait_for` | 返回降级结果 |
| graph step | 一轮并发执行 | `step_timeout` | cancel inflight |
| SDK | API 请求窗口 | client timeout | 返回调用方错误 |
| 前端 | 用户交互等待 | loading / polling | 提示后台处理中 |

这张表不画，timeout 再多也是盲配。

---

## 二、把“等待”拆成三种，再决定策略

### 类型 1：短等待
- 几百毫秒到几秒
- 典型例子：DB 查询、缓存读取、单次 API
- 处理方式：节点内 timeout + retry

### 类型 2：中等待
- 5~30 秒
- 典型例子：复杂检索、模型生成、多步工具链
- 处理方式：节点预算 + step 预算 + 流式进度

### 类型 3：长等待
- 分钟级甚至更久
- 典型例子：人工审批、外部批任务、报表生成
- 处理方式：interrupt / checkpoint / async run / polling

超时设计最大的错误之一，就是把三种等待都当成第一种处理。

---

## 三、为每层预算建立“内紧外松”的顺序

一个常见的好配置是：

```text
单次 HTTP connect 2s
单次 HTTP read    8s
单次 node budget  10s
step_timeout       12s
SDK request        20s
前端请求窗口       25s
```

为什么建议内紧外松？

因为你希望：
- 问题越靠近根源，越早暴露；
- 越外层只是兜底，不要先于内层失控；
- 最外层给系统留一点“清理与返回错误”的时间。

如果顺序反过来，例如：
- 前端 10 秒；
- graph step 30 秒；

那用户只会看见 network error，而后端还在无意义地忙碌。

---

## 四、超时之后别只会抛错，要有三板斧

### 第一斧：Retry
适用于瞬态错误。

### 第二斧：Fallback
适用于业务可降级。

### 第三斧：Pause / Resume
适用于长等待和人工环节。

你可以把 timeout 后的动作理解成下面这棵树：

```text
超时
 ├─ 值得重试？ → retry
 ├─ 可降级？   → fallback
 ├─ 可暂停？   → interrupt / resume
 └─ 都不行？   → fail fast + 告警
```

如果团队里 timeout 后的唯一动作是 `raise`，那系统韧性会很差。

---

## 五、遇到长任务，优先想“换交互模型”

社区里关于长任务 timeout 的大量案例，本质上都不是 LangGraph 不行，而是交互模式不合适。

典型错误模式：
- 浏览器发一个请求；
- 后端开始跑一个 2 分钟工作流；
- 前端静静等；
- 中间几乎不发状态；
- 90 秒后 network error；
- 刷新又发现结果其实在。

这时正确动作往往不是：
- 把 step timeout 从 60 改 180；

而是：
- 让 run 在后台继续；
- 前端轮询 run status；
- 必要时展示中间进度；
- 工作流状态写 checkpoint。

## 六、把 timeout 当成“架构探针”

超时出现在哪一层，其实就在告诉你哪里需要重构：

### 如果经常是 node I/O timeout
- 依赖不稳定；
- 请求粒度太大；
- 连接池配置有问题。

### 如果经常是 graph step timeout
- 一个 step 做的事过多；
- 并发任务差异过大；
- 慢任务没有拆 step。

### 如果经常是前端 / SDK timeout
- 同步交互模型不适配；
- 需要 polling / streaming / async run。

### 如果经常是 platform timeout
- 说明问题已超出图代码本身，进入部署架构层。

所以 timeout 不是坏消息，它是最诚实的系统反馈。

## 七、一个非常实用的落地模板

### 1. 先定义预算对象

```python
budgets = {
    "connect": 2.0,
    "read": 8.0,
    "node": 10.0,
    "step": 12.0,
    "sdk": 20.0,
}
```

### 2. 节点内先吃掉最贴近根因的超时

```python
result = await asyncio.wait_for(call_remote(), timeout=budgets["node"])
```

### 3. 图外层再兜底

```python
graph.step_timeout = budgets["step"]
```

### 4. timeout 后路由到 fallback 或 error node

```python
if state["timed_out"]:
    return "fallback"
return "next"
```

### 5. 长任务改成后台 run

```text
POST /runs  -> 返回 run_id
GET /runs/{id} -> 查询 status / partial result
```

这就是一个 80% 场景够用的生产骨架。

## 八、最容易被忽略的两个细节

### 细节 1：timeout 也需要 observability

没有日志与指标，你只会看到“超时很多”，却不知道：
- connect 超时多；
- read 超时多；
- graph step 超时多；
- 还是前端断连多。

### 细节 2：timeout 预算不要和业务 SLA 脱节

比如用户对某个按钮的预期就是 3 秒内出结果，你给整个链路都配 30 秒，其实已经违背产品体验了。

预算不只是技术问题，也是产品问题。

---

## 九、高手和新手的分水岭

### 新手会问
“这个 timeout 参数该填多少？”

### 高手会先问
- 这是哪一层的 timeout？
- 这层 timeout 的责任边界是什么？
- 超时之后谁接手？
- 这是应该失败、重试、降级，还是暂停？

一旦你的问题变成后者，timeout 设计就进入成熟阶段了。

---

## 十、最后的化骨绵掌

把所有内容压成一句操作原则：

**离根因越近，timeout 越早处理；离用户越近，恢复策略越友好。**

这句话几乎能指导你完成绝大多数 LangGraph timeout 设计。
