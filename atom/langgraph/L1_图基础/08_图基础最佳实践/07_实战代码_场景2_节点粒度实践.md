# 实战代码 - 场景2：节点粒度实践

> 本文档通过对比示例展示节点粒度选择的最佳实践，帮助你理解何时拆分、何时合并节点。

---

## 场景概述

**实战目标**：构建一个数据处理管道，对比不同节点粒度设计

**核心对比**：
- 反模式1：过度拆分（节点太细）
- 反模式2：过度合并（节点太粗）
- 最佳实践：合理粒度（单一职责 + 可读性）

**涉及最佳实践**：
- 单一职责原则
- 节点粒度权衡
- 使用子图模块化
- 性能与可维护性平衡

---

## 反模式1：过度拆分

### 问题描述

将简单操作拆分成过多节点，导致图结构复杂，难以理解和维护。

### 代码示例

```python
"""
反模式示例：过度拆分节点
问题：图结构过于复杂，难以理解
"""

from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    text: str
    cleaned: str
    lowercased: str
    stripped: str
    words: list[str]
    filtered_words: list[str]
    result: str

# ❌ 过度拆分：每个微小操作都是一个节点
def remove_punctuation(state: State) -> dict:
    """只移除标点符号"""
    import string
    cleaned = state["text"].translate(str.maketrans("", "", string.punctuation))
    return {"cleaned": cleaned}

def convert_to_lowercase(state: State) -> dict:
    """只转换小写"""
    lowercased = state["cleaned"].lower()
    return {"lowercased": lowercased}

def strip_whitespace(state: State) -> dict:
    """只去除首尾空格"""
    stripped = state["lowercased"].strip()
    return {"stripped": stripped}

def split_into_words(state: State) -> dict:
    """只分词"""
    words = state["stripped"].split()
    return {"words": words}

def filter_empty_words(state: State) -> dict:
    """只过滤空字符串"""
    filtered = [w for w in state["words"] if w]
    return {"filtered_words": filtered}

def join_words(state: State) -> dict:
    """只拼接单词"""
    result = " ".join(state["filtered_words"])
    return {"result": result}

# 构建过度拆分的图
graph = StateGraph(State)
graph.add_node("remove_punctuation", remove_punctuation)
graph.add_node("convert_to_lowercase", convert_to_lowercase)
graph.add_node("strip_whitespace", strip_whitespace)
graph.add_node("split_into_words", split_into_words)
graph.add_node("filter_empty_words", filter_empty_words)
graph.add_node("join_words", join_words)

# 线性连接所有节点
graph.add_edge(START, "remove_punctuation")
graph.add_edge("remove_punctuation", "convert_to_lowercase")
graph.add_edge("convert_to_lowercase", "strip_whitespace")
graph.add_edge("strip_whitespace", "split_into_words")
graph.add_edge("split_into_words", "filter_empty_words")
graph.add_edge("filter_empty_words", "join_words")
graph.add_edge("join_words", END)

app = graph.compile()

# 测试
result = app.invoke({
    "text": "Hello, World! This is a TEST.",
    "cleaned": "", "lowercased": "", "stripped": "",
    "words": [], "filtered_words": [], "result": ""
})
print(f"结果: {result['result']}")
# 输出: hello world this is a test
```

### 问题分析

1. **图结构复杂**：6个节点处理简单的文本清理
2. **状态膨胀**：需要维护多个中间状态字段
3. **性能开销**：每个节点都需要状态序列化和传递
4. **难以理解**：需要追踪多个步骤才能理解整体逻辑
5. **难以测试**：需要为每个微小操作编写测试

## 反模式2：过度合并

### 问题描述

将多个不相关的功能合并到一个节点，违反单一职责原则。

### 代码示例

```python
"""
反模式示例：过度合并节点
问题：节点职责不清，难以维护和复用
"""

from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    raw_data: str
    processed_result: dict

# ❌ 过度合并：一个节点做所有事情
def process_everything(state: State) -> dict:
    """
    处理所有事情：
    1. 数据清理
    2. 数据验证
    3. 数据转换
    4. 数据存储
    5. 发送通知
    """
    import string
    import json
    
    # 1. 清理数据
    text = state["raw_data"]
    cleaned = text.translate(str.maketrans("", "", string.punctuation))
    cleaned = cleaned.lower().strip()
    
    # 2. 验证数据
    if len(cleaned) < 5:
        raise ValueError("数据太短")
    
    # 3. 转换数据
    words = cleaned.split()
    word_count = len(words)
    unique_words = len(set(words))
    
    # 4. 存储数据（模拟）
    data = {
        "text": cleaned,
        "word_count": word_count,
        "unique_words": unique_words
    }
    # db.save(data)  # 假设的数据库操作
    
    # 5. 发送通知（模拟）
    # send_email("admin@example.com", f"Processed {word_count} words")
    
    # 6. 生成报告
    report = f"处理完成: {word_count} 个单词, {unique_words} 个唯一单词"
    
    return {
        "processed_result": {
            "data": data,
            "report": report
        }
    }

# 构建过度合并的图
graph = StateGraph(State)
graph.add_node("process_everything", process_everything)
graph.add_edge(START, "process_everything")
graph.add_edge("process_everything", END)

app = graph.compile()

# 测试
result = app.invoke({
    "raw_data": "Hello, World! This is a TEST.",
    "processed_result": {}
})
print(f"报告: {result['processed_result']['report']}")
```

### 问题分析

1. **职责不清**：一个节点负责清理、验证、转换、存储、通知
2. **难以测试**：需要模拟多个依赖（数据库、邮件服务）
3. **难以复用**：无法单独复用其中某个功能
4. **难以维护**：修改一个功能可能影响其他功能
5. **难以调试**：错误发生时难以定位具体步骤

## 最佳实践：合理粒度

### 设计原则

1. **单一职责**：每个节点只负责一个明确的功能
2. **合理聚合**：紧密相关的操作可以合并
3. **可读性优先**：图结构清晰易懂
4. **便于测试**：每个节点可以独立测试

### 完整代码示例

```python
"""
最佳实践：合理的节点粒度
场景：数据处理管道

特点：
- 节点职责单一但不过度拆分
- 紧密相关的操作合并
- 图结构清晰易懂
- 便于测试和维护
"""

from typing_extensions import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy

# ===== 1. 状态定义 =====

class DataPipelineState(TypedDict):
    """数据处理管道状态"""
    raw_data: str
    cleaned_data: str
    validation_result: dict
    analysis_result: dict
    storage_result: dict

# ===== 2. 节点定义（合理粒度）=====

def clean_data_node(state: DataPipelineState) -> dict:
    """
    数据清理节点
    职责：清理和标准化输入数据
    
    合理聚合：
    - 移除标点符号
    - 转换小写
    - 去除空格
    这些操作紧密相关，合并到一个节点
    """
    import string
    
    text = state["raw_data"]
    
    # 合并紧密相关的清理操作
    cleaned = text.translate(str.maketrans("", "", string.punctuation))
    cleaned = cleaned.lower().strip()
    cleaned = " ".join(cleaned.split())  # 规范化空格
    
    print(f"[清理数据] 原始: {len(text)} 字符 -> 清理后: {len(cleaned)} 字符")
    
    return {"cleaned_data": cleaned}


def validate_data_node(state: DataPipelineState) -> dict:
    """
    数据验证节点
    职责：验证数据质量
    
    单一职责：只负责验证，不做其他操作
    """
    cleaned = state["cleaned_data"]
    
    # 验证逻辑
    is_valid = True
    errors = []
    
    if len(cleaned) < 5:
        is_valid = False
        errors.append("数据太短（少于5个字符）")
    
    if not cleaned:
        is_valid = False
        errors.append("数据为空")
    
    words = cleaned.split()
    if len(words) < 2:
        is_valid = False
        errors.append("单词数量太少（少于2个）")
    
    validation_result = {
        "is_valid": is_valid,
        "errors": errors,
        "word_count": len(words)
    }
    
    print(f"[验证数据] 验证结果: {'通过' if is_valid else '失败'}")
    if errors:
        print(f"[验证数据] 错误: {', '.join(errors)}")
    
    return {"validation_result": validation_result}


def analyze_data_node(state: DataPipelineState) -> dict:
    """
    数据分析节点
    职责：分析数据特征
    
    单一职责：只负责分析，不做存储或通知
    """
    cleaned = state["cleaned_data"]
    words = cleaned.split()
    
    # 分析逻辑
    analysis = {
        "total_words": len(words),
        "unique_words": len(set(words)),
        "avg_word_length": sum(len(w) for w in words) / len(words) if words else 0,
        "longest_word": max(words, key=len) if words else "",
        "shortest_word": min(words, key=len) if words else ""
    }
    
    print(f"[分析数据] 总单词: {analysis['total_words']}, 唯一单词: {analysis['unique_words']}")
    
    return {"analysis_result": analysis}


def store_data_node(state: DataPipelineState) -> dict:
    """
    数据存储节点
    职责：存储处理结果
    
    单一职责：只负责存储
    可以配置重试策略处理存储失败
    """
    # 模拟存储操作
    data_to_store = {
        "cleaned_data": state["cleaned_data"],
        "validation": state["validation_result"],
        "analysis": state["analysis_result"]
    }
    
    # 实际应用中这里会调用数据库
    # db.insert("processed_data", data_to_store)
    
    storage_result = {
        "stored": True,
        "record_id": "mock_id_12345",
        "timestamp": "2026-02-25T12:00:00Z"
    }
    
    print(f"[存储数据] 存储成功, ID: {storage_result['record_id']}")
    
    return {"storage_result": storage_result}


# ===== 3. 构建图 =====

def create_data_pipeline():
    """创建数据处理管道"""
    
    graph = StateGraph(DataPipelineState)
    
    # 配置重试策略（只为可能失败的节点）
    storage_retry_policy = RetryPolicy(
        max_attempts=3,
        initial_interval=1.0,
        backoff_factor=2.0
    )
    
    # 添加节点
    graph.add_node("clean_data", clean_data_node)
    graph.add_node("validate_data", validate_data_node)
    graph.add_node("analyze_data", analyze_data_node)
    graph.add_node(
        "store_data",
        store_data_node,
        retry_policy=storage_retry_policy  # 存储操作需要重试
    )
    
    # 添加边（线性流程）
    graph.add_edge(START, "clean_data")
    graph.add_edge("clean_data", "validate_data")
    graph.add_edge("validate_data", "analyze_data")
    graph.add_edge("analyze_data", "store_data")
    graph.add_edge("store_data", END)
    
    return graph


# ===== 4. 执行示例 =====

def main():
    """主函数"""
    
    print("=" * 80)
    print("节点粒度最佳实践示例")
    print("场景：数据处理管道")
    print("=" * 80)
    
    # 创建并编译图
    graph = create_data_pipeline()
    app = graph.compile()
    
    # 准备输入
    initial_state = {
        "raw_data": "Hello, World! This is a TEST. LangGraph is AWESOME!",
        "cleaned_data": "",
        "validation_result": {},
        "analysis_result": {},
        "storage_result": {}
    }
    
    # 执行
    print("\n开始执行数据处理管道...")
    print("-" * 80)
    
    result = app.invoke(initial_state)
    
    # 输出结果
    print("\n" + "=" * 80)
    print("执行结果")
    print("=" * 80)
    
    print(f"\n原始数据: {result['raw_data']}")
    print(f"清理后数据: {result['cleaned_data']}")
    
    print(f"\n验证结果:")
    print(f"  是否有效: {result['validation_result']['is_valid']}")
    print(f"  单词数量: {result['validation_result']['word_count']}")
    
    print(f"\n分析结果:")
    for key, value in result['analysis_result'].items():
        print(f"  {key}: {value}")
    
    print(f"\n存储结果:")
    print(f"  已存储: {result['storage_result']['stored']}")
    print(f"  记录ID: {result['storage_result']['record_id']}")
    
    print("\n" + "=" * 80)


if __name__ == "__main__":
    main()
```

## 使用子图模块化

### 场景说明

当图结构变得复杂时，使用子图组织相关节点，保持主图简洁。

### 代码示例

```python
"""
使用子图进行模块化设计
"""

from langgraph.graph import StateGraph, START, END

# ===== 创建子图：数据预处理 =====

def create_preprocessing_subgraph():
    """数据预处理子图"""
    
    subgraph = StateGraph(DataPipelineState)
    
    # 子图节点
    subgraph.add_node("clean", clean_data_node)
    subgraph.add_node("validate", validate_data_node)
    
    # 子图边
    subgraph.add_edge(START, "clean")
    subgraph.add_edge("clean", "validate")
    subgraph.add_edge("validate", END)
    
    return subgraph.compile()


# ===== 创建子图：数据分析 =====

def create_analysis_subgraph():
    """数据分析子图"""
    
    subgraph = StateGraph(DataPipelineState)
    
    # 子图节点
    subgraph.add_node("analyze", analyze_data_node)
    subgraph.add_node("store", store_data_node)
    
    # 子图边
    subgraph.add_edge(START, "analyze")
    subgraph.add_edge("analyze", "store")
    subgraph.add_edge("store", END)
    
    return subgraph.compile()


# ===== 主图：组合子图 =====

def create_modular_pipeline():
    """使用子图的模块化管道"""
    
    main_graph = StateGraph(DataPipelineState)
    
    # 添加子图作为节点
    main_graph.add_node("preprocessing", create_preprocessing_subgraph())
    main_graph.add_node("analysis", create_analysis_subgraph())
    
    # 连接子图
    main_graph.add_edge(START, "preprocessing")
    main_graph.add_edge("preprocessing", "analysis")
    main_graph.add_edge("analysis", END)
    
    return main_graph.compile()


# 使用模块化管道
modular_app = create_modular_pipeline()
result = modular_app.invoke({
    "raw_data": "Test data",
    "cleaned_data": "",
    "validation_result": {},
    "analysis_result": {},
    "storage_result": {}
})
```

## 节点粒度对比总结

### 对比表

| 维度 | 过度拆分 | 过度合并 | 合理粒度 |
|------|----------|----------|----------|
| **节点数量** | 过多（6+） | 过少（1-2） | 适中（3-5） |
| **单个节点复杂度** | 极简单 | 极复杂 | 适中 |
| **图结构复杂度** | 高 | 低 | 适中 |
| **可读性** | 差 | 差 | 好 |
| **可测试性** | 差 | 差 | 好 |
| **可复用性** | 差 | 差 | 好 |
| **性能** | 差（状态传递开销大） | 好 | 好 |
| **维护性** | 差 | 差 | 好 |

### 选择指南

**何时拆分节点**：
- 功能独立且可复用
- 需要单独测试
- 可能需要重试或错误处理
- 逻辑复杂度高（> 50 行）

**何时合并节点**：
- 紧密耦合的操作（如清理 + 标准化）
- 简单的转换操作（< 10 行）
- 性能考虑（减少状态传递）

**何时使用子图**：
- 相关节点超过 3 个
- 需要在多个地方复用
- 需要独立测试和维护

## 最佳实践清单

### 节点设计

- [ ] 每个节点职责单一明确
- [ ] 节点函数简短易懂（< 50 行）
- [ ] 节点命名清晰描述功能
- [ ] 节点可以独立测试
- [ ] 避免过度拆分和过度合并

### 图结构

- [ ] 图结构清晰易懂
- [ ] 节点数量适中（3-7 个为佳）
- [ ] 使用子图组织相关节点
- [ ] 避免过深的嵌套

### 性能优化

- [ ] 基于实际测量进行优化
- [ ] 合并紧密相关的简单操作
- [ ] 为可能失败的节点配置重试
- [ ] 监控节点执行时间

---

## 总结

节点粒度选择的核心原则：

1. **单一职责**：每个节点只做一件事
2. **可读性优先**：优先考虑可读性和可维护性
3. **合理聚合**：紧密相关的操作可以合并
4. **使用子图**：组织相关节点，保持主图简洁
5. **性能优化**：基于实际测量，不过早优化

记住：好的节点粒度设计应该让图结构清晰易懂，每个节点职责明确，便于测试和维护。
