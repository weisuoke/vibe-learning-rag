## 卡片1：调试的第一步 — 开启全局 Debug

**一句话：** `set_debug(True)` 是 LangChain 调试的"万能开关"，一行代码看到所有内幕。

**举例：**

```python
from langchain_core.globals import set_debug
set_debug(True)

# 之后所有 LangChain 操作都会打印：
# - 完整的 Prompt（发给 LLM 的完整内容）
# - LLM 的原始返回
# - 工具调用的输入和输出
# - 每一步的耗时
```

**注意区分：**
- `set_debug(True)` → 输出全部信息（开发用）
- `set_verbose(True)` → 输出简化信息（快速浏览用）
- 两者可以同时使用，debug 包含 verbose 的所有信息

**应用：** 开发阶段第一时间用 debug 模式定位问题，确认 LLM 收到了什么、返回了什么。

## 卡片2：回调系统 — 观察者模式的精髓

**一句话：** LangChain 的回调系统是基于观察者模式的事件总线，不修改代码就能监控一切。

**举例：**

```python
from langchain_core.callbacks import BaseCallbackHandler

class MyHandler(BaseCallbackHandler):
    def on_llm_start(self, serialized, prompts, **kwargs):
        print(f"📤 LLM 收到: {prompts[0][:50]}...")

    def on_tool_end(self, output, **kwargs):
        print(f"🔧 工具返回: {output[:50]}...")

    def on_agent_action(self, action, **kwargs):
        print(f"🤔 Agent 决定用: {action.tool}")
```

**核心事件（必须记住的6个）：**
| 事件 | 触发时机 |
|------|---------|
| `on_llm_start` | LLM 调用前 |
| `on_llm_end` | LLM 调用后 |
| `on_tool_start` | 工具执行前 |
| `on_tool_end` | 工具执行后 |
| `on_agent_action` | Agent 决定用工具 |
| `on_agent_finish` | Agent 完成任务 |

**应用：** 自定义回调是最灵活的调试手段，可以记录任何你关心的信息。

## 卡片3：LangSmith — Agent 的 X 光机

**一句话：** LangSmith 把 Agent 的每次运行展开成可视化的树状结构，是理解复杂行为的最佳工具。

**举例：**

```python
# 只需3个环境变量即可启用
import os
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_API_KEY"] = "lsv2_pt_xxxxx"
os.environ["LANGCHAIN_PROJECT"] = "my-agent"

# 之后所有操作自动上报到 LangSmith
# 在 https://smith.langchain.com 查看追踪
```

**LangSmith 追踪能看到什么：**
- 🌳 完整的调用链树状图（LLM → 工具 → LLM → ...）
- 📝 每个节点的输入和输出
- ⏱️ 每一步的耗时
- 💰 Token 消耗和成本
- 🔄 不同运行之间的对比

**应用：** Agent 行为不可预测时，LangSmith 是唯一能让你"回放"整个过程的工具。

## 卡片4：RunnableConfig — 调试的配置中心

**一句话：** `RunnableConfig` 是每次运行的"配置背包"，控制回调、并发、标签和递归深度。

**举例：**

```python
from langchain_core.runnables import RunnableConfig

config = RunnableConfig(
    tags=["production", "user-123"],     # 标签：用于过滤追踪
    metadata={"session_id": "abc123"},   # 元数据：附加信息
    callbacks=[my_handler],              # 回调：自定义监控
    max_concurrency=5,                   # 并发控制
    recursion_limit=25,                  # 递归深度限制
    run_name="customer-support-agent",   # 运行名称
)

result = chain.invoke(input, config=config)
```

**关键点：**
- `tags` 和 `metadata` 可在 LangSmith 中用于过滤和搜索
- `recursion_limit` 默认 25，是防止无限递归的安全网
- Config 会自动传播到子组件（通过 ContextVar）

**应用：** 用标签区分不同来源的请求（如 `["api", "web"]`），在 LangSmith 中快速定位特定类型的问题。

## 卡片5：LLM 缓存 — 避免重复付费

**一句话：** LangChain 的缓存系统用 prompt + 模型参数作为 key，完全相同的请求直接返回缓存结果。

**举例：**

```python
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache

# 开启内存缓存
set_llm_cache(InMemoryCache())

# 第一次调用 - 走 API（慢）
result1 = llm.invoke("什么是 Python？")  # ~1000ms

# 第二次完全相同的调用 - 走缓存（快）
result2 = llm.invoke("什么是 Python？")  # ~1ms
```

**缓存类型选择：**

| 类型 | 持久化 | 适用场景 |
|------|-------|---------|
| `InMemoryCache` | ❌ 重启丢失 | 开发测试 |
| `SQLiteCache` | ✅ 文件持久化 | 单机生产 |
| Redis 缓存 | ✅ 分布式 | 多机生产 |

**反直觉：** Agent 的缓存命中率通常很低（<10%），因为包含上下文的 prompt 每次都不同。缓存更适合简单的独立 Chain。

**应用：** 对于 FAQ 类应用（问题固定），缓存可以节省 90%+ 成本。

## 卡片6：防止死循环 — 两个安全阀

**一句话：** `max_iterations` 和 `recursion_limit` 是 Agent 的两个紧急刹车，防止 LLM 陷入无限循环。

**举例：**

```python
# 安全阀1：Agent 层面
agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,  # 最多10轮工具调用
    # 达到上限时返回超时消息而不是崩溃
)

# 安全阀2：Runnable 层面（更底层）
result = chain.invoke(
    input,
    config={"recursion_limit": 25}  # 默认25
)
```

**死循环的信号：**
- 连续 3+ 次调用相同工具且参数相似
- Token 消耗急剧增长但没有有效输出
- 延迟异常增长

**应用：** 生产环境务必设置 `max_iterations`，建议值 10-15。宁可让 Agent 提前返回"无法完成"，也不要让它无限循环烧钱。

## 卡片7：模型路由 — 用对的模型做对的事

**一句话：** 不是所有任务都需要最强模型，智能路由可以用 20% 的成本覆盖 80% 的场景。

**举例：**

```python
def route_to_model(query: str) -> str:
    """根据任务复杂度选择模型"""
    complex_keywords = ["分析", "推理", "对比", "策略", "为什么"]

    if any(kw in query for kw in complex_keywords):
        return "gpt-4o"       # $2.50/M 输入
    else:
        return "gpt-4o-mini"  # $0.15/M 输入（便宜16倍！）

# 实际使用
model_name = route_to_model(user_query)
llm = ChatOpenAI(model=model_name)
```

**成本对比：**
- 假设每天 10,000 次调用
- 全用 GPT-4o：~$25/天
- 路由后（80% mini + 20% 4o）：~$7/天
- **节省 72%**

**应用：** 先从简单的关键词路由开始，后续可以用一个小模型做分类器实现更精准的路由。

## 卡片8：错误处理三件套 — 重试、降级、解析修复

**一句话：** `max_retries` 处理瞬时错误，`with_fallbacks` 实现模型降级，`handle_parsing_errors` 让 LLM 自我修复。

**举例：**

```python
from langchain_openai import ChatOpenAI

# 1. 自动重试（处理网络抖动、限流）
llm = ChatOpenAI(model="gpt-4o", max_retries=3)

# 2. 模型降级（主模型不可用时切换）
llm_safe = llm.with_fallbacks(
    [ChatOpenAI(model="gpt-4o-mini")]
)

# 3. 输出解析修复（Agent 专用）
agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    handle_parsing_errors=True,
    # LLM 输出格式不对时，把错误信息反馈给 LLM 重新生成
)
```

**三件套何时触发：**
- `max_retries` → API 超时、429 限流、网络错误
- `with_fallbacks` → 主模型完全不可用
- `handle_parsing_errors` → LLM 输出格式不符合预期

**应用：** 生产环境必须同时配置这三个，缺一不可。它们互补而不重叠。

## 卡片9：结构化日志 — 生产环境的生命线

**一句话：** 生产环境的 Agent 日志必须是结构化 JSON，而不是随意的 print 输出。

**举例：**

```python
import json
import logging

logger = logging.getLogger("agent")

# ❌ 错误做法 — 无法被日志平台解析
print(f"LLM 调用完成，耗时 {duration}ms")

# ✅ 正确做法 — 结构化 JSON 日志
log_entry = {
    "event": "llm_complete",
    "duration_ms": 1234,
    "tokens": 500,
    "model": "gpt-4o-mini",
    "cost_usd": 0.0003,
    "run_id": "abc-123",
}
logger.info(json.dumps(log_entry))
```

**为什么必须结构化？**
- ELK/Datadog 等平台可以自动解析 JSON 字段
- 可以基于字段做聚合查询（如"过去1小时平均延迟"）
- 可以基于字段设置告警规则（如"cost_usd > 0.01 时告警"）

**应用：** 在自定义回调处理器的每个事件方法中，输出结构化日志，这是生产监控的基础。

## 卡片10：监控的终极目标 — 从被动到主动

**一句话：** 最好的监控不是出了问题才发现，而是在问题影响用户之前自动检测和修复。

**三个成熟度等级：**

```
等级1：被动调试（大部分团队在这里）
├── 用户报告问题 → 开发者看日志 → 定位修复
└── 问题发现时间：小时级

等级2：主动监控（推荐目标）
├── 告警系统检测异常 → 通知开发者 → 快速修复
├── 自动评估检测质量下降
└── 问题发现时间：分钟级

等级3：自动修复（未来方向）
├── 错误率过高 → 自动切换备用模型
├── Token 超预算 → 自动降级到便宜模型
├── 质量下降 → 自动回滚到上一个 Prompt 版本
└── 问题发现时间：秒级
```

**从等级1到等级2的最小行动：**
1. 添加 `ProductionMonitorHandler`（收集指标）
2. 设置错误率告警阈值（如 >10%）
3. 添加慢请求告警（如 >10s）
4. 配置 LangSmith 采样追踪（10%）

**应用：** 大多数团队卡在等级1，投入1-2天实现等级2的基本监控，就能显著减少生产事故的影响范围和修复时间。
