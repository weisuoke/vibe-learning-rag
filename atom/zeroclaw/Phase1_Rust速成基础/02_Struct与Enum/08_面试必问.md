## Q1：Rust 的 enum 和 C/TypeScript 的 enum 有什么区别？

### 标准答案

Rust 的 enum 是 **tagged union**（带标签的联合体），每个变体可以携带不同类型和数量的数据。C/TypeScript 的 enum 只是整数常量的命名集合。

```rust
// Rust：变体携带不同数据
enum Message {
    Quit,                       // 无数据
    Text(String),               // 一个 String
    Move { x: i32, y: i32 },   // 两个命名字段
}
```

```typescript
// TypeScript：只是数字别名
enum Direction { Up, Down, Left, Right }  // 0, 1, 2, 3
```

### 加分回答

- Rust enum 等价于 TypeScript 的 **discriminated union**，但标签由编译器管理，不需要手动维护 `type` 字段
- Rust enum 可以有 `impl` 块，直接给变体集合添加方法
- `match` 强制穷尽检查，新增变体时编译器会标记所有未更新的 match

## Q2：Struct 的三种形式分别是什么？各自适用场景？

### 标准答案

```rust
// 1. 命名字段 struct（最常用）
struct Agent { name: String, model: String }

// 2. 元组 struct（少量字段、语义包装）
struct UserId(u64);
struct Rgb(u8, u8, u8);

// 3. 单元 struct（零大小类型标记）
struct Validated;
```

| 形式 | 适用场景 |
|------|---------|
| 命名字段 | 3+ 字段、需要可读性 |
| 元组 | Newtype 包装、1-3 个字段 |
| 单元 | 类型状态标记、trait 实现载体 |

### 加分回答

元组 struct 的 Newtype 模式可以为外部类型实现 trait（绕过孤儿规则）：

```rust
struct Meters(f64);
struct Seconds(f64);
// Meters + Seconds 编译错误——类型安全，零运行时开销
```

## Q3：Copy 和 Clone 的区别？什么类型能 derive(Copy)？

### 标准答案

| | Copy | Clone |
|--|------|-------|
| 触发 | 隐式（赋值/传参自动） | 显式（`.clone()`） |
| 实现 | 按位拷贝（memcpy） | 可自定义深拷贝 |
| 要求 | 所有字段都是 Copy | 所有字段都是 Clone |

能 Copy 的：`i32`、`f64`、`bool`、`char`、`&T`、元组/数组（如果元素都 Copy）

不能 Copy 的：`String`、`Vec<T>`、`Box<T>`——含堆数据

```rust
let a = 42;
let b = a;        // Copy，a 仍可用

let s = String::from("hi");
let t = s;        // Move！s 不可用
let u = t.clone(); // Clone，t 仍可用
```

### 加分回答

Copy 是 Clone 的子 trait（`trait Copy: Clone {}`）。derive(Copy) 必须同时 derive(Clone)。设计 API 时，小的值类型（坐标、ID、枚举标记）应该 derive(Copy)，大的或含堆数据的类型只 derive(Clone)。

## Q4：match 的穷尽性检查有什么实际价值？

### 标准答案

当你新增 enum 变体时，编译器会在**所有** match 表达式处报错，强制你更新处理逻辑。这在大型项目中防止遗漏。

```rust
enum Level { ReadOnly, Supervised, Full }

fn check(l: &Level) -> &str {
    match l {
        Level::ReadOnly => "safe",
        Level::Supervised => "caution",
        Level::Full => "danger",
    }
}

// 新增 Level::Admin 后，上面的 match 立即编译报错
// TypeScript 的 switch 不会报错，Admin 被默默忽略
```

### 加分回答

- `#[non_exhaustive]` 属性可以让库作者在不破坏下游的情况下新增变体——下游必须有 `_ =>` 分支
- `_ =>` 通配符是双刃剑：方便但会吞掉新变体。安全关键代码应写全所有分支
- `matches!` 宏用于只需要 bool 判断的场景，避免写完整 match

## Q5：Option\<T\> 和 null/undefined 的本质区别？

### 标准答案

`Option<T>` 是一个普通的 enum（`Some(T)` 或 `None`），和 `T` 是**完全不同的类型**。编译器强制你在使用前处理 None 的情况。

```rust
let x: Option<String> = Some("hello".into());

// ❌ 不能直接当 String 用
// x.to_uppercase();

// ✓ 必须先解包
let upper = x.map(|s| s.to_uppercase());        // Option<String>
let upper = x.unwrap_or_default().to_uppercase(); // String
```

TypeScript 的 `?` 可选标记不阻止你直接访问 undefined 值（运行时炸）。Rust 在编译期就阻止了。

### 加分回答

Option 的零开销：`Option<&T>` 和 `Option<Box<T>>` 利用 **niche optimization**，None 用空指针表示，不额外占内存。`size_of::<Option<&str>>() == size_of::<&str>()`。

## Q6：什么时候用 struct 组合，什么时候用 enum？

### 标准答案

**判断标准：AND 还是 OR。**

```rust
// AND（同时存在）→ struct
struct Agent {
    name: String,       // 有 name
    security: Policy,   // 且有 security
    tools: Vec<Tool>,   // 且有 tools
}

// OR（互斥状态）→ enum
enum State {
    Idle,               // 空闲
    Running(Task),      // 或运行中
    Failed(String),     // 或失败
}
```

| 问自己 | 选择 |
|--------|------|
| 这些数据同时存在？ | struct |
| 同一时刻只有一种？ | enum |
| 需要状态机？ | enum |
| 需要组合多个子系统？ | struct 嵌套 struct |

### 加分回答

实际项目中经常组合使用：struct 的字段类型是 enum，enum 的变体携带 struct。ZeroClaw 的 `Agent` struct 包含 `AutonomyLevel` enum 字段，`ConversationMessage` enum 的变体携带 `ChatMessage` struct。

## Q7：derive(Debug) 和手动实现 Display 的区别？

### 标准答案

```rust
#[derive(Debug)]
struct Agent { name: String }

let a = Agent { name: "bot".into() };
println!("{:?}", a);   // Debug: Agent { name: "bot" }
println!("{:#?}", a);  // Debug pretty-print

// Display 需要手动实现
use std::fmt;
impl fmt::Display for Agent {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "Agent({})", self.name)
    }
}
println!("{}", a);     // Display: Agent(bot)
```

| | Debug | Display |
|--|-------|---------|
| 用途 | 开发调试 | 面向用户输出 |
| 格式符 | `{:?}` | `{}` |
| 可 derive | 是 | 否，必须手动实现 |
| 输出风格 | 包含类型名和字段名 | 自定义 |

### 加分回答

`Debug` 是几乎所有类型都应该 derive 的。`Display` 只在需要面向用户的格式化时才实现。实现了 `Display` 会自动获得 `ToString` trait（`.to_string()` 方法）。

## Q8：Builder 模式中 `mut self` 和 `&mut self` 的区别？

### 标准答案

```rust
// mut self：消费所有权，返回新 self（推荐）
fn name(mut self, n: &str) -> Self {
    self.name = Some(n.into());
    self  // 所有权转移回调用者
}
// 用法：builder.name("a").model("b").build()

// &mut self：借用修改，返回 &mut Self
fn name(&mut self, n: &str) -> &mut Self {
    self.name = Some(n.into());
    self
}
// 用法：builder.name("a"); builder.model("b"); builder.build()
```

| | `mut self` | `&mut self` |
|--|-----------|-------------|
| 所有权 | 每次转移 | 始终在调用者 |
| 链式调用 | 自然流畅 | 需要分号分隔或 `&mut` 链 |
| build() | 消费 builder（一次性） | builder 可复用 |
| 推荐场景 | 一次性构建 | 需要复用 builder |

### 加分回答

Rust 社区惯例是 `mut self`（消费式），因为大多数 Builder 只用一次。`build()` 消费 self 还能防止"构建后继续修改"的 bug。标准库的 `std::process::Command` 用 `&mut self` 是因为需要多次 `.arg()` 后复用。
