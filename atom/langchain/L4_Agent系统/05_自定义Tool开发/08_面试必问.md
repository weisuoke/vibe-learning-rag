## 问题 1："LangChain 中有哪几种创建自定义工具的方式？它们的区别是什么？"

### 普通回答（❌ 不出彩）

"有三种方式：@tool 装饰器、StructuredTool 和 BaseTool。@tool 最简单，BaseTool 最复杂。"

### 出彩回答（✅ 推荐）

> **LangChain 提供三种工具创建方式，各有适用场景：**
>
> **1. @tool 装饰器（推荐首选）**
> - 最简洁的方式，自动从函数签名推断 schema
> - 支持 Google-style docstring 解析（parse_docstring=True）
> - 底层委托给 StructuredTool.from_function()
> - 适合 90% 的场景，特别是简单无状态工具
>
> **2. StructuredTool.from_function()（显式控制）**
> - 适合包装现有 API 客户端，无需修改原函数
> - 可以显式指定 args_schema，提供更详细的参数描述
> - 支持 coroutine 参数，灵活处理异步函数
> - 适合需要精确控制 schema 或动态创建工具的场景
>
> **3. BaseTool 类继承（最大控制权）**
> - 适合需要管理复杂状态的工具（如数据库连接池）
> - 可以在 __init__() 中初始化资源，在 __del__() 中清理
> - 必须实现 _run() 方法，可选实现 _arun() 方法
> - 注意：在 LangGraph 中访问状态受限，推荐用 @tool + InjectedToolArg
>
> **选择决策树**：
> - 简单工具 → @tool
> - 包装现有 API → StructuredTool
> - 复杂状态管理 → BaseTool
> - 需要访问 LangGraph 状态 → @tool + InjectedToolArg

**为什么这个回答出彩？**
1. ✅ 结构清晰，分三层讲解（定义、特点、适用场景）
2. ✅ 提供决策树，帮助面试官理解你的思考过程
3. ✅ 提到了实际开发中的注意事项（LangGraph 状态访问）
4. ✅ 展示了对源码的理解（@tool 委托给 StructuredTool）

## 问题 2："什么是参数注入机制？为什么需要它？"

### 普通回答（❌ 不出彩）

"参数注入就是自动传递一些参数给工具，比如 config。"

### 出彩回答（✅ 推荐）

> **参数注入机制是 LangChain 工具系统的高级特性，解决了上下文信息传递的问题。**
>
> **核心问题**：
> 在 Agent 系统中，有些参数（如用户 ID、会话状态）是运行时上下文，不应该由 LLM 决定。如果让 LLM 传递这些参数，会导致：
> - LLM 不知道当前用户是谁
> - 安全风险（LLM 可能伪造用户 ID）
> - schema 污染（业务参数和上下文参数混在一起）
>
> **解决方案**：
> 使用 `InjectedToolArg` 标记参数，LangChain 会自动注入，不会出现在 JSON Schema 中：
>
> ```python
> from langchain_core.tools import tool, InjectedToolArg
> from langchain_core.runnables import RunnableConfig
> from typing import Annotated
>
> @tool
> def personalized_search(
>     query: str,  # LLM 传递
>     config: Annotated[RunnableConfig, InjectedToolArg]  # 系统注入
> ) -> str:
>     user_id = config.get("configurable", {}).get("user_id")
>     return f"为用户 {user_id} 搜索 '{query}'"
> ```
>
> **支持的注入类型**：
> 1. **RunnableConfig** - 访问用户 ID、会话 ID、元数据
> 2. **InjectedToolCallId** - 获取工具调用 ID，用于追踪和日志
> 3. **Callbacks** - 访问回调处理器，用于流式输出
>
> **实际应用场景**：
> - 多租户系统：自动注入 tenant_id，确保数据隔离
> - 权限控制：注入用户角色，工具内部验证权限
> - 分布式追踪：注入 tool_call_id，关联日志和监控

**为什么这个回答出彩？**
1. ✅ 从问题出发，解释为什么需要这个机制
2. ✅ 提供代码示例，展示实际用法
3. ✅ 列举三种注入类型，展示全面理解
4. ✅ 联系实际应用场景（多租户、权限、追踪）

## 问题 3："异步工具和同步工具有什么区别？什么时候应该使用异步工具？"

### 普通回答（❌ 不出彩）

"异步工具用 async/await，同步工具不用。异步工具更快。"

### 出彩回答（✅ 推荐）

> **异步工具通过 Python 的 async/await 机制实现非阻塞 I/O，在 Agent 系统中有显著性能优势。**
>
> **性能对比**：
> 假设 Agent 需要调用 3 个工具（天气 API 2秒、数据库查询 1秒、搜索引擎 3秒）：
> - 同步执行：2 + 1 + 3 = **6 秒**
> - 异步执行：max(2, 1, 3) = **3 秒**（50% 性能提升）
>
> **三种实现方式**：
>
> **1. @tool + async 函数（推荐）**
> ```python
> @tool
> async def get_weather(city: str) -> str:
>     async with httpx.AsyncClient() as client:
>         response = await client.get(f"https://api.weather.com?q={city}")
>         return response.text
> ```
>
> **2. StructuredTool + coroutine 参数**
> ```python
> weather_tool = StructuredTool.from_function(
>     coroutine=async_fetch_weather,  # 异步函数
>     name="get_weather",
>     args_schema=WeatherInput,
> )
> ```
>
> **3. BaseTool + _arun 方法**
> ```python
> class WeatherTool(BaseTool):
>     async def _arun(self, city: str, config=None) -> str:
>         # 异步实现
> ```
>
> **何时使用异步？**
> - ✅ HTTP API 调用
> - ✅ 数据库查询
> - ✅ 文件 I/O（大文件）
> - ✅ 向量数据库检索
> - ❌ CPU 密集型计算（如矩阵运算）
> - ❌ 纯内存操作（如列表排序）
>
> **自动回退机制**：
> 如果工具只提供同步实现，LangChain 会自动使用线程池执行，不会阻塞事件循环。但性能不如原生异步实现。
>
> **最佳实践**：
> - 使用 httpx 而非 requests（支持 async/await）
> - 复用连接池（在 BaseTool 中管理 AsyncClient）
> - 设置超时（避免工具无限等待）

**为什么这个回答出彩？**
1. ✅ 用具体数字展示性能差异（6秒 vs 3秒）
2. ✅ 列举三种实现方式，展示全面掌握
3. ✅ 提供清晰的使用场景判断（何时用/不用）
4. ✅ 提到自动回退机制，展示对源码的理解
5. ✅ 给出最佳实践建议

## 问题 4："什么是 content_and_artifact 响应格式？它解决了什么问题？"

### 普通回答（❌ 不出彩）

"content_and_artifact 可以返回两个值，一个给 LLM，一个给程序。"

### 出彩回答（✅ 推荐）

> **content_and_artifact 是 LangChain 工具的高级响应格式，解决了大数据返回的 token 浪费问题。**
>
> **核心问题**：
> 假设搜索工具返回 100 条结果，每条 200 tokens：
> - 全部返回给 LLM：20000 tokens（浪费 context window，增加成本）
> - 只返回摘要：程序无法访问完整数据，需要重新调用 API
>
> **解决方案**：
> ```python
> @tool(response_format="content_and_artifact")
> def search(query: str) -> Tuple[str, Dict[str, Any]]:
>     results = api.search(query, limit=100)
>
>     # 摘要（给 LLM 看，10 tokens）
>     summary = f"Found {len(results)} results. Top: {results[0]['title']}"
>
>     # 完整数据（给程序用，不占用 LLM tokens）
>     artifact = {
>         "results": results,
>         "metadata": {"query_time_ms": 123, "api_version": "v2"},
>         "statistics": {"avg_score": 0.85}
>     }
>
>     return summary, artifact
> ```
>
> **ToolMessage 结构**：
> - `message.content`：摘要（LLM 可见）
> - `message.artifact`：完整数据（程序可访问）
> - `message.tool_call_id`：工具调用 ID
>
> **成本对比**（GPT-4 @ $0.03/1K tokens）：
> - content 模式：20K tokens × $0.03 = **$0.60/次**
> - content_and_artifact 模式：10 tokens × $0.03 = **$0.0003/次**
> - **节省 99.95%**
>
> **适用场景**：
> - ✅ 搜索工具（返回大量结果）
> - ✅ 数据库查询（返回多行数据）
> - ✅ RAG 检索（返回多个文档）
> - ✅ API 调用（需要保留元数据）
> - ❌ 简单计算器（返回值很小）
>
> **最佳实践**：
> - 摘要简洁（< 200 tokens），包含关键信息
> - Artifact 结构化，包含完整数据和元数据
> - 错误信息在两个部分都提供

**为什么这个回答出彩？**
1. ✅ 用具体数字展示成本节省（99.95%）
2. ✅ 提供完整代码示例，展示实际用法
3. ✅ 对比两种模式的优劣
4. ✅ 列举适用和不适用场景
5. ✅ 给出最佳实践建议

## 问题 5："ToolException 和普通异常有什么区别？如何设计工具的错误处理？"

### 普通回答（❌ 不出彩）

"ToolException 不会中断 Agent，普通异常会中断。"

### 出彩回答（✅ 推荐）

> **ToolException 是 LangChain 的特殊异常，允许工具报告错误而不中断 Agent 循环，让 Agent 有机会自我修正。**
>
> **错误分类原则**：
>
> **可恢复错误（使用 ToolException）**：
> - 用户输入错误（参数不合法）
> - 外部服务临时不可用（API 超时）
> - 数据不存在（404）
> - 速率限制（429）
>
> **不可恢复错误（使用普通异常）**：
> - 配置错误（API 密钥缺失）
> - 权限不足（403）
> - 代码逻辑错误
> - 系统资源耗尽
>
> **错误处理流程对比**：
>
> **ToolException 流程**：
> ```
> User: "查询 admin 表"
>   ↓
> Agent: 调用 query_database(table="admin")
>   ↓
> Tool: 抛出 ToolException("表名不在白名单中")
>   ↓
> Agent: 收到错误观察 → 重新思考
>   ↓
> Agent: 调用 query_database(table="users", filter="role='admin'")
>   ↓
> Tool: 返回成功结果
> ```
>
> **普通异常流程**：
> ```
> Agent: 调用工具
>   ↓
> Tool: 抛出 ValueError("API_KEY 未配置")
>   ↓
> Agent: 立即停止，向用户报告配置错误
> ```
>
> **三种错误处理策略**：
>
> **1. 默认行为（传播异常）**
> ```python
> @tool
> def tool1(param: str) -> str:
>     raise ValueError("Fatal error")  # 中断 Agent
> ```
>
> **2. 简单捕获（handle_tool_error=True）**
> ```python
> @tool(handle_tool_error=True)
> def tool2(param: str) -> str:
>     raise ValueError("Error")  # 返回 "Error: ..."
> ```
>
> **3. 自定义处理（handle_tool_error=func）**
> ```python
> def custom_handler(error: Exception) -> str:
>     if isinstance(error, requests.Timeout):
>         return "API 超时，建议使用缓存或稍后重试"
>     elif isinstance(error, requests.HTTPError):
>         if error.response.status_code == 429:
>             return "API 速率限制，请等待 60 秒"
>         return f"API 错误 {error.response.status_code}"
>     return f"未知错误: {str(error)}"
>
> @tool(handle_tool_error=custom_handler)
> def tool3(query: str) -> str:
>     # 所有异常都会被 custom_handler 处理
> ```
>
> **错误消息编写技巧**：
> - ❌ 不好：`raise ToolException("Error")`（太模糊）
> - ❌ 不好：`raise ToolException(str(e))`（技术细节过多）
> - ✅ 好：`raise ToolException(f"未找到城市 '{location}'，请检查拼写或使用英文名称")`
>   - 包含：问题描述 + 上下文 + 解决建议
>
> **在 RAG 系统中的应用**：
> ```python
> @tool
> def retrieve_documents(query: str) -> str:
>     try:
>         results = vector_store.similarity_search(query)
>         if not results:
>             raise ToolException(
>                 f"未找到与 '{query}' 相关的文档，请尝试更通用的关键词"
>             )
>         return format_results(results)
>     except ConnectionError:
>         raise ToolException("向量数据库连接失败，请稍后重试")
> ```
>
> Agent 可以根据错误调整策略：
> - 如果检索失败 → 改写查询
> - 如果数据库不可用 → 使用缓存

**为什么这个回答出彩？**
1. ✅ 清晰的错误分类原则（可恢复 vs 不可恢复）
2. ✅ 用流程图展示两种异常的处理差异
3. ✅ 列举三种错误处理策略，展示全面掌握
4. ✅ 提供错误消息编写技巧（对比好坏示例）
5. ✅ 联系 RAG 实际应用场景

## 问题 6："如何在工具中定义复杂的输入 schema？Pydantic 有哪些高级用法？"

### 普通回答（❌ 不出彩）

"用 Pydantic BaseModel 定义 schema，用 Field 添加描述。"

### 出彩回答（✅ 推荐）

> **Pydantic schema 是工具开发的基础，直接影响 LLM 对工具的理解和调用准确性。**
>
> **基础 Schema 定义**：
> ```python
> from pydantic import BaseModel, Field
> from typing import Literal, List, Optional
>
> class SearchInput(BaseModel):
>     """搜索工具输入 schema"""
>
>     # 必需字段
>     query: str = Field(description="搜索关键词")
>
>     # 数值约束
>     limit: int = Field(
>         default=10,
>         ge=1,  # greater than or equal
>         le=100,  # less than or equal
>         description="返回结果数量（1-100）"
>     )
>
>     # 枚举值约束
>     sort_by: Literal["relevance", "date", "popularity"] = Field(
>         default="relevance",
>         description="排序方式"
>     )
>
>     # 可选字段
>     filters: Optional[List[str]] = Field(
>         default=None,
>         description="可选过滤条件"
>     )
> ```
>
> **高级用法 1：自定义验证器**
> ```python
> from pydantic import BaseModel, Field, validator
>
> class AdvancedInput(BaseModel):
>     query: str = Field(description="搜索查询")
>     tags: List[str] = Field(default_factory=list)
>
>     @validator('query')
>     def validate_query(cls, v):
>         """验证查询字符串"""
>         if len(v.strip()) == 0:
>             raise ValueError("查询不能为空")
>         if len(v) > 500:
>             raise ValueError("查询过长（最多 500 字符）")
>         return v.strip()
>
>     @validator('tags')
>     def validate_tags(cls, v):
>         """验证标签列表"""
>         if len(v) > 10:
>             raise ValueError("标签过多（最多 10 个）")
>         # 去重并转小写
>         return list(set(tag.lower() for tag in v))
> ```
>
> **高级用法 2：嵌套对象**
> ```python
> class Address(BaseModel):
>     street: str = Field(description="街道地址")
>     city: str = Field(description="城市")
>     country: str = Field(description="国家")
>
> class UserProfile(BaseModel):
>     name: str = Field(description="姓名")
>     age: int = Field(ge=0, le=150, description="年龄")
>     addresses: List[Address] = Field(
>         default_factory=list,
>         description="地址列表"
>     )
> ```
>
> **高级用法 3：联合类型**
> ```python
> from typing import Union
>
> class TextQuery(BaseModel):
>     type: Literal["text"] = "text"
>     query: str
>
> class ImageQuery(BaseModel):
>     type: Literal["image"] = "image"
>     image_url: str
>
> class MultiModalInput(BaseModel):
>     query: Union[TextQuery, ImageQuery] = Field(
>         description="查询可以是文本或图像"
>     )
> ```
>
> **常见陷阱**：
>
> **1. 可变默认值**
> ```python
> # ❌ 错误：所有实例共享同一个列表
> class BadSchema(BaseModel):
>     items: List[str] = Field(default=[])
>
> # ✅ 正确：使用 default_factory
> class GoodSchema(BaseModel):
>     items: List[str] = Field(default_factory=list)
> ```
>
> **2. 参数名称不匹配**
> ```python
> # ❌ 错误：schema 和函数参数名不一致
> class MySchema(BaseModel):
>     search_query: str
>
> @tool(args_schema=MySchema)
> def my_tool(query: str) -> str:  # 参数名不匹配
>     pass
>
> # ✅ 正确：参数名一致
> class MySchema(BaseModel):
>     query: str
>
> @tool(args_schema=MySchema)
> def my_tool(query: str) -> str:
>     pass
> ```
>
> **最佳实践**：
> - 使用 Field() 提供详细描述（LLM 会读取）
> - 使用 Literal 限制枚举值
> - 使用 default_factory 处理可变默认值
> - 使用自定义验证器实现复杂验证逻辑
> - 保持 schema 简单清晰，避免过度嵌套

**为什么这个回答出彩？**
1. ✅ 从基础到高级，层层递进
2. ✅ 提供三种高级用法（验证器、嵌套、联合类型）
3. ✅ 列举常见陷阱并提供解决方案
4. ✅ 给出最佳实践建议
