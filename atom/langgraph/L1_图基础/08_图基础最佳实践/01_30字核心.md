## 应用场景

### 场景1: 生产级 RAG 系统

**需求**: 构建一个稳定、高性能的 RAG 问答系统

**应用最佳实践**:
```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy

# 1. 状态设计: 使用 TypedDict + Reducer
class RAGState(TypedDict):
    query: str                                    # 用户查询
    rewritten_query: str                          # 改写后的查询
    search_results: Annotated[list, add]          # 检索结果(聚合)
    reranked_results: list                        # 重排序结果
    generated_answer: str                         # 生成的答案
    metadata: dict                                # 元数据

# 2. 节点粒度: 单一职责
def query_rewrite_node(state: RAGState) -> dict:
    """只负责查询改写"""
    rewritten = rewrite_query(state["query"])
    return {"rewritten_query": rewritten}

def search_node(state: RAGState) -> dict:
    """只负责检索"""
    results = search(state["rewritten_query"])
    return {"search_results": results}

def rerank_node(state: RAGState) -> dict:
    """只负责重排序"""
    reranked = rerank(state["search_results"])
    return {"reranked_results": reranked}

def generate_node(state: RAGState) -> dict:
    """只负责生成"""
    answer = generate(state["reranked_results"])
    return {"generated_answer": answer}

# 3. 错误处理: 配置 RetryPolicy
retry_policy = RetryPolicy(
    max_attempts=3,
    initial_interval=1.0,
    backoff_factor=2.0,
    jitter=True,
)

# 4. 构建图
graph = StateGraph(RAGState)
graph.add_node("rewrite", query_rewrite_node, retry_policy=retry_policy)
graph.add_node("search", search_node, retry_policy=retry_policy)
graph.add_node("rerank", rerank_node)
graph.add_node("generate", generate_node, retry_policy=retry_policy)

graph.add_edge(START, "rewrite")
graph.add_edge("rewrite", "search")
graph.add_edge("search", "rerank")
graph.add_edge("rerank", "generate")
graph.add_edge("generate", END)

app = graph.compile()
```

**最佳实践应用**:
- ✅ 状态设计: TypedDict + Annotated[list, add] 聚合检索结果
- ✅ 节点粒度: 每个节点只负责一个功能
- ✅ 错误处理: 为 API 调用节点配置 RetryPolicy
- ✅ 性能优化: 可以并行执行多个检索源

### 场景2: 多步推理代理

**需求**: 构建一个能够进行多步推理的代理系统

**应用最佳实践**:
```python
from typing import TypedDict, Annotated, Literal
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy

# 1. 状态设计
class ReasoningState(TypedDict):
    task: str                                     # 任务描述
    plan: list[str]                               # 推理计划
    steps: Annotated[list[dict], add]             # 执行步骤(聚合)
    current_step: int                             # 当前步骤
    results: dict                                 # 中间结果
    final_answer: str                             # 最终答案
    is_valid: bool                                # 验证结果

# 2. 节点粒度: 单一职责
def plan_node(state: ReasoningState) -> dict:
    """只负责规划"""
    plan = create_plan(state["task"])
    return {"plan": plan, "current_step": 0}

def execute_node(state: ReasoningState) -> dict:
    """只负责执行当前步骤"""
    step = state["plan"][state["current_step"]]
    result = execute_step(step, state["results"])
    return {
        "steps": [{"step": step, "result": result}],
        "current_step": state["current_step"] + 1,
        "results": {**state["results"], step: result}
    }

def validate_node(state: ReasoningState) -> dict:
    """只负责验证"""
    is_valid = validate_results(state["results"])
    return {"is_valid": is_valid}

def synthesize_node(state: ReasoningState) -> dict:
    """只负责综合答案"""
    answer = synthesize_answer(state["results"])
    return {"final_answer": answer}

# 3. 条件路由
def should_continue(state: ReasoningState) -> Literal["execute", "validate"]:
    """判断是否继续执行"""
    if state["current_step"] < len(state["plan"]):
        return "execute"
    return "validate"

def should_retry(state: ReasoningState) -> Literal["plan", "synthesize"]:
    """判断是否需要重新规划"""
    if state["is_valid"]:
        return "synthesize"
    return "plan"

# 4. 构建图
graph = StateGraph(ReasoningState)

retry_policy = RetryPolicy(max_attempts=3, initial_interval=1.0)

graph.add_node("plan", plan_node)
graph.add_node("execute", execute_node, retry_policy=retry_policy)
graph.add_node("validate", validate_node)
graph.add_node("synthesize", synthesize_node)

graph.add_edge(START, "plan")
graph.add_conditional_edges("plan", should_continue)
graph.add_conditional_edges("execute", should_continue)
graph.add_conditional_edges("validate", should_retry)
graph.add_edge("synthesize", END)

app = graph.compile()
```

**最佳实践应用**:
- ✅ 状态设计: 使用 Annotated[list[dict], add] 聚合执行步骤
- ✅ 节点粒度: 规划、执行、验证、综合分离
- ✅ 错误处理: 为执行节点配置重试策略
- ✅ 条件路由: 使用 Literal 类型提示明确路由选项

### 场景3: 人机协作审批流程

**需求**: 构建一个需要人工审批的工作流系统

**应用最佳实践**:
```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import interrupt

# 1. 状态设计
class ApprovalState(TypedDict):
    request: dict                                 # 审批请求
    auto_check_result: dict                       # 自动检查结果
    approval_status: str                          # 审批状态
    approver: str                                 # 审批人
    comments: str                                 # 审批意见
    final_decision: str                           # 最终决策

# 2. 节点设计
def auto_check_node(state: ApprovalState) -> dict:
    """自动检查"""
    result = auto_check(state["request"])
    return {"auto_check_result": result}

def human_approval_node(state: ApprovalState) -> dict:
    """人工审批 - 使用 interrupt"""
    # 等待人工审批
    approval = interrupt({
        "type": "approval_required",
        "request": state["request"],
        "auto_check": state["auto_check_result"]
    })

    return {
        "approval_status": approval["status"],
        "approver": approval["approver"],
        "comments": approval["comments"]
    }

def decision_node(state: ApprovalState) -> dict:
    """最终决策"""
    decision = make_decision(
        state["auto_check_result"],
        state["approval_status"]
    )
    return {"final_decision": decision}

# 3. 构建图 - 使用 Checkpointer
checkpointer = MemorySaver()

graph = StateGraph(ApprovalState)
graph.add_node("auto_check", auto_check_node)
graph.add_node("human_approval", human_approval_node)
graph.add_node("decision", decision_node)

graph.add_edge(START, "auto_check")
graph.add_edge("auto_check", "human_approval")
graph.add_edge("human_approval", "decision")
graph.add_edge("decision", END)

app = graph.compile(checkpointer=checkpointer)

# 4. 使用
config = {"configurable": {"thread_id": "approval-001"}}

# 第一次执行 - 会在 human_approval 节点暂停
result = app.invoke({"request": {...}}, config)

# 人工审批后恢复执行
result = app.invoke(
    {"approval_status": "approved", "approver": "Alice", "comments": "LGTM"},
    config
)
```

**最佳实践应用**:
- ✅ 状态设计: 清晰的审批流程状态
- ✅ 节点粒度: 自动检查、人工审批、最终决策分离
- ✅ 人机协作: 使用 interrupt 实现人工审批
- ✅ 持久化: 使用 Checkpointer 实现断点续传

## 常见反模式对照

### 反模式1: 在状态中存储大对象 ❌

**错误示例**:
```python
class State(TypedDict):
    documents: list[str]  # 存储完整文档内容(可能很大)
    embeddings: list[list[float]]  # 存储所有向量(占用大量内存)
```

**正确做法** ✅:
```python
class State(TypedDict):
    document_ids: list[str]  # 只存储文档 ID
    embedding_ids: list[str]  # 只存储向量 ID
    # 大对象存储在外部(数据库、向量库)
```

### 反模式2: 在 interrupt 前执行副作用 ❌

**错误示例**:
```python
def node(state: State):
    # ❌ 副作用在 interrupt 前执行
    db.append_to_history(state["user_id"], "approval_requested")

    approved = interrupt("Approve this change?")
    return {"approved": approved}
```

**正确做法** ✅:
```python
from langgraph.func import task

@task
def write_to_db(user_id: str, message: str):
    db.append_to_history(user_id, message)

def node(state: State):
    # ✅ 副作用封装在 task 中
    write_to_db(state["user_id"], "approval_requested").result()

    approved = interrupt("Approve this change?")
    return {"approved": approved}
```

### 反模式3: 抑制意外错误 ❌

**错误示例**:
```python
def node(state: State):
    try:
        result = api_call(state["data"])
        return {"result": result}
    except Exception:
        pass  # ❌ 静默忽略错误
```

**正确做法** ✅:
```python
from langgraph.types import RetryPolicy

retry_policy = RetryPolicy(max_attempts=3)

def node(state: State):
    # ✅ 让错误向上传播,由 RetryPolicy 处理
    result = api_call(state["data"])
    return {"result": result}

graph.add_node("api_node", node, retry_policy=retry_policy)
```

### 反模式4: 过度复杂的图设计 ❌

**错误示例**:
```python
# ❌ 单个图包含 20+ 个节点,难以理解
graph = StateGraph(State)
graph.add_node("step1", ...)
graph.add_node("step2", ...)
# ... 20+ 个节点
```

**正确做法** ✅:
```python
# ✅ 使用子图模块化
preprocessing_graph = create_preprocessing_graph()
processing_graph = create_processing_graph()
postprocessing_graph = create_postprocessing_graph()

main_graph = StateGraph(State)
main_graph.add_node("preprocessing", preprocessing_graph)
main_graph.add_node("processing", processing_graph)
main_graph.add_node("postprocessing", postprocessing_graph)
```

## 核心原则总结

### 1. 状态设计原则

**黄金法则**:
- 使用 TypedDict 明确定义类型
- 使用 Annotated[type, reducer] 定义聚合字段
- 状态中只存储必要的数据
- 避免存储大对象,使用引用

**类比**:
- **前端**: Redux store 设计原则
- **日常**: 工作台只放必需工具

### 2. 节点粒度原则

**黄金法则**:
- 单一职责原则
- 可读性优先
- 合理拆分与合并
- 使用子图组织相关节点

**类比**:
- **前端**: React 组件拆分原则
- **日常**: 流水线工序划分

### 3. 错误处理原则

**黄金法则**:
- 使用 RetryPolicy 处理临时性错误
- 让意外错误向上传播
- 在关键节点添加错误处理
- 实施完善的监控和告警

**类比**:
- **前端**: try-catch + error boundary
- **日常**: 质量检查站

### 4. 性能优化原则

**黄金法则**:
- 合理设计状态结构
- 避免存储大对象
- 使用缓存策略
- 监控性能指标

**类比**:
- **前端**: React 性能优化
- **日常**: 生产线效率优化

### 5. 反模式避免原则

**黄金法则**:
- 不在 interrupt 前执行副作用
- 避免非确定性工作流
- 不抑制意外错误
- 避免过度复杂的图设计

**类比**:
- **前端**: 代码坏味道
- **日常**: 生产事故隐患
