# 反直觉点

> 关于 create_agent 最常见的认知误区——你以为的 Agent，可能和真正的 Agent 完全不同

---

## 概述

`create_agent` 是 LangChain 1.0 的核心 API，但它的设计哲学与旧版 AgentExecutor 截然不同。本文列出 3 个深度误区 + 2 个快速提醒，帮你避开最常见的坑。

---

## 误区1："create_agent 只是 AgentExecutor 的简单替代" ❌

### 为什么错？

`create_agent` 不是 API 重命名，而是**架构级别的重新设计**。

底层发生了根本性变化：

| 维度 | 旧版 AgentExecutor | 新版 create_agent |
|------|-------------------|-------------------|
| 架构模式 | Chain 模式（线性管道） | LangGraph StateGraph（状态图） |
| 创建流程 | 两步：先创建 Agent 决策逻辑，再包装 Executor | 一步：直接返回完整的编译图 |
| 返回类型 | `AgentExecutor` 实例 | `CompiledStateGraph` 实例 |
| 输入格式 | `agent.run("问题")` 或 `agent({"input": "问题"})` | `agent.invoke({"messages": [...]})` |
| 状态管理 | `AgentAction` / `AgentFinish` 枚举 | `AgentState` TypedDict（messages 为核心） |
| 定制方式 | 参数爆炸（`max_iterations`、`early_stopping_method`...） | Middleware 组合模式 |
| 持久化 | 需要额外实现 Memory | 内置 `checkpointer` 参数 |

```python
# ===== 旧版：两步创建，概念分离 =====
from langchain.agents import create_react_agent, AgentExecutor

# 第一步：创建决策逻辑（Agent 只负责"想"）
agent = create_react_agent(llm, tools, prompt)

# 第二步：包装执行引擎（Executor 负责"跑"）
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,           # 参数散落在 Executor 上
    handle_parsing_errors=True,  # 错误处理也是参数
    early_stopping_method="generate",
    verbose=True,
)
result = executor.invoke({"input": "查一下北京天气"})


# ===== 新版：一步到位，统一抽象 =====
from langchain.agents import create_agent
from langchain.agents.middleware import ModelRetryMiddleware, ModelCallLimitMiddleware

# 一个函数搞定一切
agent = create_agent(
    model="openai:gpt-4o",
    tools=tools,
    system_prompt="你是一个天气助手",
    middleware=[
        ModelRetryMiddleware(max_retries=2),      # 错误处理 → Middleware
        ModelCallLimitMiddleware(max_calls=10),    # 循环限制 → Middleware
    ],
)
result = agent.invoke({"messages": [{"role": "user", "content": "查一下北京天气"}]})
```

### 为什么人们容易这样错？

1. **迁移指南的误导**：官方迁移文档把旧版函数和新版函数一一对应，让人觉得只是"换个写法"。但对应关系掩盖了底层架构的根本变化。

2. **表面相似性**：两者都是"传入 model + tools，得到一个可以 invoke 的东西"。但就像汽油车和电动车都能开，底层动力系统完全不同。

3. **渐进式迁移的错觉**：你可以用 `create_agent` 写出和旧版几乎一样的代码，这让人误以为"没什么变化"。但当你需要 checkpointer、interrupt、子图嵌套时，才会发现新架构的真正威力。

### 正确理解

**create_agent 是从 Chain 范式到 Graph 范式的跃迁。**

旧版的心智模型是"管道"——数据从左到右流过一系列处理步骤。新版的心智模型是"状态图"——节点之间根据条件跳转，状态在图中流转。

这不是换了个函数名，而是换了整个思维方式。

## 误区2："Middleware 越多越好，应该把所有内置 Middleware 都加上" ❌

### 为什么错？

LangChain 1.0 提供了 19+ 个内置 Middleware，但这不意味着你应该全部使用。

**每个 Middleware 都有成本：**

1. **执行开销**：每个 Middleware 的 `before_model` / `after_model` 钩子在**每轮** LLM 调用中都会执行。Agent 循环 5 次 × 8 个 Middleware × 2 个钩子 = 80 次额外函数调用。

2. **消息膨胀**：某些 Middleware 会向消息列表注入内容（如 `TodoListMiddleware` 注入待办列表），增加 Token 消耗和成本。

3. **冲突风险**：Middleware 之间可能产生意想不到的交互。

```python
# ❌ 错误：把能想到的 Middleware 全加上
agent = create_agent(
    "openai:gpt-4o",
    tools=[search_tool, calc_tool],
    middleware=[
        SummarizationMiddleware(model="openai:gpt-4o-mini"),
        ContextEditingMiddleware(),
        ModelRetryMiddleware(max_retries=3),
        ModelCallLimitMiddleware(max_calls=25),
        ToolRetryMiddleware(max_retries=3),
        PIIMiddleware(),
        HumanInTheLoopMiddleware(),
        TodoListMiddleware(),
        LLMToolSelectorMiddleware(),
    ],
)
# 问题1：SummarizationMiddleware 可能删除 ContextEditingMiddleware 需要的历史消息
# 问题2：HumanInTheLoopMiddleware 在每次工具调用前暂停，开发体验极差
# 问题3：LLMToolSelectorMiddleware 额外调用一次 LLM 来选工具，2 个工具根本不需要
# 问题4：每轮循环 9 个 Middleware × 多个钩子 = 大量额外开销
```

### 常见冲突场景

| Middleware A | Middleware B | 冲突描述 |
|-------------|-------------|---------|
| `SummarizationMiddleware` | `ContextEditingMiddleware` | 摘要可能删除编辑上下文需要的原始消息 |
| `HumanInTheLoopMiddleware` | `ToolRetryMiddleware` | 工具失败后重试，但每次重试都要人工确认 |
| `LLMToolSelectorMiddleware` | 只有 1-2 个工具时 | 额外 LLM 调用毫无意义，工具少时模型自己就能选对 |
| `ModelCallLimitMiddleware` | `SummarizationMiddleware` | 摘要本身消耗一次模型调用，可能提前触发限制 |

### 为什么人们容易这样错？

1. **"防御性编程"心态**：写后端时习惯加各种中间件（日志、鉴权、限流、CORS），觉得多加保护总没错。但 Agent Middleware 不同于 Web Middleware——它在**循环中**执行，成本是乘法而非加法。

2. **内置 = 推荐的错觉**：看到 19 个内置 Middleware，以为都是"最佳实践"。实际上它们是**工具箱**，你只需要拿出当前需要的那几把扳手。

3. **不了解执行模型**：不知道 Middleware 钩子在每轮循环中都会执行，以为只在 Agent 开始和结束时各执行一次。

### 正确理解

**按场景选择，少即是多。**

```python
# ✅ 正确：生产环境的最小推荐配置
agent = create_agent(
    "openai:gpt-4o",
    tools=[search_tool, calc_tool],
    middleware=[
        # 必选：防止 API 偶发错误导致整个流程失败
        ModelRetryMiddleware(max_retries=2),
        # 必选：防止 Agent 陷入工具调用死循环
        ModelCallLimitMiddleware(max_calls=10),
    ],
)
```

**Middleware 选择决策树：**

```
开始
 │
 ├─ 所有场景 → ModelRetryMiddleware + ModelCallLimitMiddleware（必选）
 │
 ├─ 对话超过 20 轮？ → 加 SummarizationMiddleware
 │
 ├─ 有不可逆操作（删除/发送）？ → 加 HumanInTheLoopMiddleware
 │
 ├─ 工具超过 10 个？ → 加 LLMToolSelectorMiddleware
 │
 ├─ 处理用户敏感数据？ → 加 PIIMiddleware
 │
 └─ 其他 → 不加，保持简单
```

**类比理解：**

- **前端类比**：你不会在每个 React 组件里都加 `useMemo`、`useCallback`、`React.memo`。过度优化反而让代码更慢更难维护。Middleware 也一样——只在确实需要时才加。
- **日常生活类比**：出门不需要同时带雨伞、防晒霜、暖手宝、急救包。看天气预报，带你需要的就行。

## 误区3："create_agent 返回的是一个 Agent 对象" ❌

### 为什么错？

`create_agent()` 返回的不是一个 Agent 对象，而是一个 **`CompiledStateGraph`**——LangGraph 的编译状态图。

这意味着：

- **没有** `.run()` 方法
- **没有** `AgentExecutor` 包装
- **没有** Agent 类的实例属性（如 `agent.memory`、`agent.tools`）
- 它是一个**状态机**，通过 `.invoke()`、`.stream()`、`.ainvoke()` 驱动执行

```python
from langchain.agents import create_agent

agent = create_agent("openai:gpt-4o", tools=[])

# 看看它到底是什么
print(type(agent))
# <class 'langgraph.graph.state.CompiledStateGraph'>

# ❌ 这些都不存在
# agent.run("你好")           → AttributeError
# agent.memory                → AttributeError
# agent.agent_executor        → AttributeError

# ✅ 正确的调用方式
result = agent.invoke({"messages": [{"role": "user", "content": "你好"}]})

# ✅ 流式输出
for chunk in agent.stream({"messages": [{"role": "user", "content": "你好"}]}):
    print(chunk)

# ✅ 异步调用
result = await agent.ainvoke({"messages": [{"role": "user", "content": "你好"}]})
```

### 为什么人们容易这样错？

1. **函数名误导**：函数叫 `create_agent`，自然以为返回的是 Agent。但实际上它创建的是一个"能执行 Agent 行为的状态图"——Agent 是行为模式，不是对象类型。

2. **旧版心智模型**：LangChain 0.x 有明确的 `Agent` 类和 `AgentExecutor` 类，形成了"Agent 是一个对象"的思维定式。

3. **其他框架的影响**：CrewAI 有 `Agent` 类，AutoGen 有 `AssistantAgent` 类，这些框架确实返回 Agent 对象。但 LangChain 1.0 选择了不同的抽象——图。

### 正确理解

**Agent 不是对象，是图的执行模式。**

这个设计选择带来了巨大的好处——因为返回的是 `CompiledStateGraph`，你可以：

```python
# 1. 当作独立 Agent 使用
result = agent.invoke({"messages": [...]})

# 2. 作为子图嵌入到更大的 LangGraph 工作流
from langgraph.graph import StateGraph
workflow = StateGraph(...)
workflow.add_node("researcher", researcher_agent)  # Agent 作为节点
workflow.add_node("writer", writer_agent)           # 另一个 Agent 作为节点

# 3. 利用 LangGraph 的所有能力
agent_with_memory = create_agent(
    "openai:gpt-4o",
    tools=[my_tool],
    checkpointer=MemorySaver(),  # 状态持久化
)
# 获取状态快照
snapshot = agent_with_memory.get_state({"configurable": {"thread_id": "abc"}})
```

**类比理解：**

- **前端类比**：Vue 3 的 `createApp()` 返回的不是"应用对象"，而是一个"应用实例"，行为由内部的响应式系统驱动。`create_agent` 同理——返回的是一个可执行的运行时，不是一个数据容器。
- **日常生活类比**：你去 4S 店"买一辆车"，拿到的不是设计图纸，而是一辆组装好可以开的车。`CompiledStateGraph` 就是那辆组装好的车——你只需要踩油门（`.invoke()`）。

## 额外误区（快速提醒）

### 误区4："tools=[] 和不传 tools 效果完全一样" ⚠️

语义上有微妙区别，但结果确实相同：

```python
# 这两种写法的最终效果一样：Agent 只有 model node，没有工具循环
agent_a = create_agent("openai:gpt-4o")                # tools 默认为 None
agent_b = create_agent("openai:gpt-4o", tools=[])      # 明确传空列表

# 但语义不同：
# None  = "未指定工具"（可能后续通过 Middleware 的 tools 属性注入）
# []    = "明确没有工具"
```

真正需要注意的是：**Middleware 可以通过 `tools` 类属性注入额外工具**。如果你传了 `tools=[]` 但某个 Middleware 定义了 `tools = [my_tool]`，最终 Agent 仍然会有工具可用。

### 误区5："system_prompt 可以在每次 invoke 时动态修改" ⚠️

`system_prompt` 在 `create_agent()` 调用时就固定了，编译进了状态图。后续 `invoke` 无法修改它。

```python
# ❌ 这样不行
agent = create_agent("openai:gpt-4o", system_prompt="你是助手A")
# 没有办法在 invoke 时改成 "你是助手B"

# ✅ 方案1：用 Middleware 的 wrap_model_call 动态修改
from langchain.agents.middleware.types import AgentMiddleware

class DynamicPromptMiddleware(AgentMiddleware):
    def wrap_model_call(self, request, handler):
        # 根据运行时上下文动态修改 system_message
        user_role = request.runtime.context.get("role", "assistant")
        new_request = request.override(
            system_message=SystemMessage(content=f"你是{user_role}")
        )
        return handler(new_request)

# ✅ 方案2：用 context_schema 传递运行时参数
from pydantic import BaseModel

class MyContext(BaseModel):
    role: str = "助手"

agent = create_agent(
    "openai:gpt-4o",
    context_schema=MyContext,
    middleware=[DynamicPromptMiddleware()],
)
```

## 总结

### 三个核心反直觉点

| 误区 | 真相 | 记忆口诀 |
|------|------|---------|
| 简单替代 AgentExecutor | 架构级重新设计（Chain → Graph） | 不是换名字，是换引擎 |
| Middleware 越多越好 | 按需选择，少即是多 | 两个必选，其余按需 |
| 返回 Agent 对象 | 返回 `CompiledStateGraph` | Agent 是图，不是对象 |

### 正确的思维方式

- `create_agent` 返回的是一个**可执行的状态图**，用 `.invoke()` / `.stream()` 驱动
- Middleware 是**工具箱**，不是**必选清单**——`ModelRetryMiddleware` + `ModelCallLimitMiddleware` 是生产环境的最小配置
- 复杂功能通过**参数组合**和 **Middleware 组合**实现，不需要切换到不同的 Agent 类型

### 一句话记住

> **create_agent 返回图不返回对象，Middleware 按需不贪多，简单入口藏着无限可能。**
