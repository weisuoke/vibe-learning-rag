# Agent最佳实践 - 概览

> Agent 最佳实践是让 Agent 从"能跑"升级为"能打"的系统工程——通过 Middleware 架构实现关注点分离，用重试/降级/限制三重保护确保可靠性，用 PII/HITL/沙箱守住安全底线，用摘要/编辑/预算管好上下文，最终构建出一个可靠、安全、可维护的生产级 Agent。

## 为什么需要 Agent 最佳实践？

### 从 09 到 10：从"优化"到"工程化"

在上一个知识点 `09_Agent调试与优化` 中，我们学会了看清 Agent（回调/追踪）、加速 Agent（缓存/批处理）、控制成本（模型路由/Token优化）、处理异常（错误回调/重试）。但这些都是**单点优化**。

**10_Agent最佳实践** 的核心问题是：**如何将这些单点能力组合成一个可靠、安全、可维护的生产系统？**

| 维度 | 09_调试与优化（单点） | 10_最佳实践（系统） |
|------|---------------------|-------------------|
| **关注点** | 某个缓存怎么配、某个错误怎么处理 | 整体架构如何设计、多个关注点如何组合 |
| **核心机制** | Callbacks / LangSmith / Cache | **Middleware** 架构 + 组合模式 |
| **可靠性** | 单个重试策略 | 重试 + 降级 + 限制 三重保护 |
| **安全性** | 基本的错误处理 | PII 检测 + 人工审核 + 沙箱隔离 |
| **上下文** | Token 计数 | 对话摘要 + 上下文编辑 + 预算管理 |
| **部署** | 开发环境调试 | 生产环境架构设计 |

### 典型场景对比

```
=== 没有最佳实践的 Agent ===

用户：帮我查一下张三的银行卡号
Agent：好的，张三的银行卡号是 6228-1234-5678-9012  ← PII 泄露！

API 偶尔超时
Agent：（直接崩溃，用户看到 500 错误）  ← 无重试！

长对话后
Agent：（把 10 万 Token 的历史全发给 LLM）  ← 上下文爆炸！

=== 有最佳实践的 Agent ===

用户：帮我查一下张三的银行卡号
Agent：张三的银行卡号是 6228-****-****-9012  ← PII 自动脱敏

API 偶尔超时
Agent：（自动重试 → 如果还失败 → 降级到备用模型 → 成功响应）  ← 三重保护

长对话后
Agent：（自动摘要旧对话，只保留最近 4000 Token）  ← 智能压缩
```

**一句话总结**：最佳实践 = 用 Middleware 架构将可靠性、安全性、上下文管理等关注点**解耦并组合**，让 Agent 从"能用"进化为"可靠可控"。

```
                    Agent 最佳实践 — Middleware 架构

用户输入 ──→ ┌─────────────────────────────────────────────────┐
             │                create_agent()                   │
             │                                                 │
             │  middleware=[                                    │
             │    ModelRetryMiddleware(max_retries=3),          │ ← 可靠性
             │    ModelFallbackMiddleware(fallbacks=[...]),     │ ← 降级
             │    ModelCallLimitMiddleware(run_limit=25),       │ ← 限制
             │    PIIMiddleware(strategy="mask"),               │ ← 安全
             │    HumanInTheLoopMiddleware(...),                │ ← 合规
             │    SummarizationMiddleware(trigger=("tokens",4000)), │ ← 上下文
             │  ]                                              │
             │                                                 │
             │  checkpointer=InMemorySaver()  ← 状态持久化      │
             │  store=InMemoryStore()         ← 长期记忆        │
             └──────────────────────┬──────────────────────────┘
                                   │
                                   ▼
                            Agent 执行循环
                    (LangGraph StateGraph 底层驱动)
                                   │
                                   ▼
                            响应用户 ──→ 输出
```

### Middleware 的四大钩子

```
                     Middleware 钩子机制

before_model()  ──→  [LLM 调用]  ──→  after_model()
                         │
                    wrap_model_call()
                    (环绕整个 LLM 调用)
                         │
                    wrap_tool_call()
                    (环绕工具执行)
```

| 钩子 | 时机 | 典型用途 |
|------|------|---------|
| `before_model` | LLM 调用前 | PII 检测、上下文编辑、Token 预算检查 |
| `after_model` | LLM 调用后 | 输出过滤、日志记录、结果缓存 |
| `wrap_model_call` | 环绕 LLM 调用 | 重试、降级、调用计数 |
| `wrap_tool_call` | 环绕工具执行 | 工具重试、人工审核、执行限制 |

## 核心概念速览

本知识点包含 **5 个核心概念**，覆盖 Agent 最佳实践的完整知识面：

### 概念 1: Middleware 架构模式

LangChain 1.0（`langchain_v1` 包）的核心创新。`create_agent()` 通过 `middleware=[]` 参数接受一系列中间件，每个中间件负责一个关注点（如重试、安全、摘要）。中间件按声明顺序**堆叠执行**（类似洋葱模型），支持自定义开发。14 个内置中间件覆盖了生产环境的主要需求。

```python
from langchain.agents import create_agent
from langchain.agents.middleware import ModelRetryMiddleware, PIIMiddleware

agent = create_agent(
    "gpt-4.1",
    tools=tools,
    middleware=[
        ModelRetryMiddleware(max_retries=3),
        PIIMiddleware(strategy="mask"),
    ],
)
```

### 概念 2: 可靠性设计模式

三重保护机制：**ModelRetryMiddleware**（指数退避 + 随机抖动重试）、**ModelFallbackMiddleware**（主模型失败后按优先级切换备用模型）、**ModelCallLimitMiddleware + ToolCallLimitMiddleware**（限制调用次数防止无限循环和成本失控）。这三者组合使用，构成了 Agent 的"防崩"护甲。

### 概念 3: 安全与合规模式

四大安全中间件：**PIIMiddleware**（检测并处理个人敏感信息，支持 redact/mask/hash/block 四种策略）、**HumanInTheLoopMiddleware**（关键操作需人工审批，支持 approve/edit/reject）、**ShellToolMiddleware**（沙箱执行 Shell 命令）、以及序列化安全（`allowed_objects` + `secretsFromEnv` 变更）。

### 概念 4: 上下文管理模式

解决长对话的 Token 爆炸问题：**SummarizationMiddleware**（当对话超过阈值时自动摘要旧消息，支持 fraction/tokens/messages 三种触发方式）、**ContextEditingMiddleware**（编辑上下文窗口内容）、以及 Token 预算管理策略。

### 概念 5: 工具设计与 Agent 架构最佳实践

生产环境的综合指导：工具描述优化（`description` 是 LLM 选择工具的关键）、工具 Schema 设计（Pydantic 强类型）、Agent 测试策略（单元测试 → 集成测试 → E2E → 红队测试）、版本迁移（`create_react_agent` → `create_agent`）、多代理架构模式（Subagents/Handoffs/Skills/Router）。

## 与 RAG 和 AI Agent 开发的关系

### 最佳实践是所有 AI 应用的共同需求

```
RAG 系统:     检索 → 生成 → 回答
              每一步都可能失败，都需要重试和降级
              用户输入可能包含 PII，需要脱敏
              长文档可能超出上下文窗口，需要管理

Agent 系统:   推理 → 工具调用 → 再推理 → ...
              循环更多，失败点更多，保护更重要
              工具操作涉及安全，需要审核和沙箱
              多步推理积累大量历史，上下文管理更关键

共同需求:     可靠（重试+降级） + 安全（PII+审核） + 可控（限制+管理）
```

### 实际应用场景

| 场景 | 可靠性需求 | 安全需求 | 上下文需求 |
|------|-----------|---------|-----------|
| 智能客服 Agent | API 重试 + 模型降级 | PII 脱敏（客户信息） | 长对话摘要 |
| 文档问答 RAG | 检索重试 + 生成降级 | 文档中的敏感信息过滤 | 大文档上下文管理 |
| 代码助手 Agent | 代码执行重试 | Shell 命令沙箱隔离 | 代码上下文裁剪 |
| 数据分析 Agent | SQL 执行重试 | 数据访问权限控制 | 查询结果摘要 |
| 金融合规 Agent | 高可用多模型降级 | 人工审核交易操作 | 合规记录管理 |

## 关键术语表

| 术语 | 英文 | 含义 |
|------|------|------|
| 中间件 | Middleware | 在 Agent 执行流程中插入的可组合处理器 |
| 钩子 | Hook | Middleware 的执行时机点（before/after/wrap） |
| 模型重试 | Model Retry | LLM 调用失败后自动重试（含退避策略） |
| 模型降级 | Model Fallback | 主模型不可用时切换到备用模型 |
| 调用限制 | Call Limit | 限制 LLM/工具的最大调用次数 |
| PII 检测 | PII Detection | 检测个人可识别信息（邮箱/信用卡/IP 等） |
| 人工审核 | Human-in-the-Loop (HITL) | 关键操作需要人工确认 |
| 对话摘要 | Conversation Summarization | 将长对话压缩为摘要以控制 Token 消耗 |
| 上下文工程 | Context Engineering | 精心管理传递给 LLM 的上下文信息 |
| 工具白名单 | Tool Allowlisting | 明确定义 Agent 可使用的工具列表 |
| 沙箱 | Sandbox | 在受限环境中执行不可信代码 |
| 指数退避 | Exponential Backoff | 每次重试等待时间指数增长的策略 |
| 随机抖动 | Jitter | 在重试延迟中添加随机偏移，避免雷群效应 |
