## 30 字核心

> **Result\<T,E\> 和 Option\<T\> 是 Rust 用枚举替代异常和空值的编译时安全错误处理机制，ZeroClaw 全局使用 anyhow::Result 统一错误传播。**

这句话浓缩了 Rust 错误处理的全部哲学。下面逐词拆解。

---

## 一、"枚举替代异常" -- 错误不再是隐形炸弹

### 1.1 TypeScript 的异常：隐式的、不可见的

```typescript
// TypeScript -- 你根本看不出这个函数会抛什么错
function parseConfig(raw: string): Config {
  return JSON.parse(raw);  // 可能抛 SyntaxError
}

// 调用方完全不知道要处理错误
const config = parseConfig(input);  // 编译通过，运行时爆炸
```

TypeScript 的异常有三个致命问题：

1. **函数签名不体现错误**：看 `parseConfig` 的类型签名，完全不知道它会抛错
2. **编译器不强制处理**：不写 try/catch 也能编译通过
3. **catch 里的 error 是 any**：你不知道捕获到的是什么类型的错误

### 1.2 Rust 的 Result：显式的、类型安全的

```rust
// Rust -- 函数签名明确告诉你"这个操作可能失败"
fn parse_config(raw: &str) -> Result<Config, serde_json::Error> {
    serde_json::from_str(raw)  // 返回 Result，不是抛异常
}

// 调用方必须处理错误，否则编译不通过
let config = parse_config(input)?;  // 显式传播错误
```

Result 解决了 TypeScript 异常的全部问题：

| 问题 | TypeScript 异常 | Rust Result |
|------|----------------|-------------|
| 函数签名体现错误？ | 不体现 | `-> Result<T, E>` 明确标注 |
| 编译器强制处理？ | 不强制 | 不处理就编译不过 |
| 错误有类型？ | `any`（无类型） | 泛型 `E`（类型安全） |

### 1.3 Result 的本质：一个只有两个变体的枚举

```rust
// Result 的完整定义（标准库源码）
enum Result<T, E> {
    Ok(T),   // 成功，携带成功值
    Err(E),  // 失败，携带错误值
}
```

就这么简单。没有魔法，没有运行时机制，就是一个普通的枚举。

**前端类比**：想象 TypeScript 强制你用 discriminated union 替代 try/catch：

```typescript
// 如果 TypeScript 有 Result 类型
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

const result = parseConfig(input);
if (result.ok) {
  console.log(result.value);   // TS 知道这里是 Config
} else {
  console.error(result.error); // TS 知道这里是 ParseError
}
```

Rust 的 Result 就是这个思路，但由编译器原生支持，语法更简洁。

## 二、"枚举替代空值" -- 告别 null/undefined

### 2.1 TypeScript 的 null 问题

```typescript
// TypeScript -- null/undefined 是运行时最常见的崩溃原因
interface User {
  name: string;
  email?: string;        // string | undefined
  phone: string | null;  // string | null
}

function getEmailDomain(user: User): string {
  // 忘记检查 → TypeError: Cannot read properties of undefined
  return user.email.split("@")[1];  // 编译可能通过（取决于 strict 配置）
}
```

JavaScript/TypeScript 中 `Cannot read properties of undefined` 是最常见的运行时错误，被称为"十亿美元的错误"（Tony Hoare 语）。

### 2.2 Rust 的 Option：编译期消灭空值

```rust
// Rust -- 没有 null，用 Option 表示"可能没有值"
struct User {
    name: String,
    email: Option<String>,   // 明确表示"可能没有"
    phone: Option<String>,   // 明确表示"可能没有"
}

fn get_email_domain(user: &User) -> Option<String> {
    // 编译器强制你处理 None 的情况
    user.email.as_ref()?.split('@').nth(1).map(String::from)
}
```

Option 的定义同样简单：

```rust
// Option 的完整定义（标准库源码）
enum Option<T> {
    Some(T),  // 有值
    None,     // 没有值
}
```

### 2.3 ZeroClaw 中的 Option 使用

ZeroClaw 的配置系统大量使用 Option 表示可选字段：

```rust
// src/config/schema.rs
pub struct ProviderConfig {
    pub api_key: Option<String>,    // API 密钥可能未配置
    pub api_url: Option<String>,    // 自定义 URL 可选
    pub model: Option<String>,      // 模型名可选，有默认值
}

// src/memory/traits.rs
pub struct MemoryEntry {
    pub session_id: Option<String>, // 会话 ID 可选
    pub score: Option<f64>,         // 相似度分数可选
}
```

[来源: reference/source_error_handling_01.md]

**生活类比**：Option 就像快递柜 -- `Some(包裹)` 表示柜子里有东西，`None` 表示柜子是空的。你必须先打开看看，才能决定下一步做什么。不像 TypeScript 的 null，你以为柜子里有东西就直接伸手去拿，结果摸了个空。

## 三、"编译时安全" -- 编译器是你的安全网

### 3.1 编译器强制你处理每一种情况

```rust
fn process(result: Result<String, Error>) {
    // 编译错误！match 必须覆盖所有变体
    match result {
        Ok(value) => println!("{}", value),
        // 缺少 Err 分支 → 编译不通过
    }
}
```

```rust
fn process(opt: Option<String>) {
    // 编译错误！match 必须覆盖所有变体
    match opt {
        Some(value) => println!("{}", value),
        // 缺少 None 分支 → 编译不通过
    }
}
```

这就是 Rust 的核心哲学：**把运行时错误变成编译时错误**。

### 3.2 对比：编译时 vs 运行时

```
TypeScript 的错误处理时间线：
  编写代码 → 编译通过 → 部署上线 → 用户触发 → 运行时崩溃 → 半夜被叫醒修 bug
                                                    ↑
                                              错误在这里才暴露

Rust 的错误处理时间线：
  编写代码 → 编译报错 → 修复 → 编译通过 → 部署上线 → 稳定运行
               ↑
         错误在这里就被拦住了
```

**前端类比**：TypeScript 的类型检查已经帮你在编译期拦住了很多类型错误。Rust 的 Result/Option 更进一步，把"错误处理"和"空值检查"也纳入了编译期检查。

## 四、"anyhow::Result 统一错误传播" -- ZeroClaw 的选择

### 4.1 为什么 ZeroClaw 选择 anyhow？

Rust 标准库的 `Result<T, E>` 要求你指定具体的错误类型 `E`。但在应用层，一个函数可能遇到多种不同类型的错误：

```rust
// 不用 anyhow -- 需要定义自己的错误枚举，很繁琐
fn load_config(path: &str) -> Result<Config, ConfigError> {
    let content = fs::read_to_string(path)?;  // io::Error
    let config: Config = serde_json::from_str(&content)?;  // serde_json::Error
    Ok(config)
    // 问题：io::Error 和 serde_json::Error 是不同类型！
    // 需要手动定义 ConfigError 枚举并实现 From trait
}

// 用 anyhow -- 一行搞定，所有错误自动转换
fn load_config(path: &str) -> anyhow::Result<Config> {
    let content = fs::read_to_string(path)
        .context("failed to read config file")?;
    let config: Config = serde_json::from_str(&content)
        .context("failed to parse config")?;
    Ok(config)
    // anyhow 自动接受任何实现了 std::error::Error 的类型
}
```

### 4.2 anyhow 的核心能力

```rust
use anyhow::{Result, bail, ensure, Context, anyhow};

// 1. bail! -- 提前返回错误（类似 throw）
fn validate_url(url: &str) -> Result<()> {
    if url.is_empty() {
        bail!("URL cannot be empty");  // 等价于 return Err(anyhow!("..."))
    }
    Ok(())
}

// 2. ensure! -- 条件断言（类似 assert 但不 panic）
fn decrypt(blob: &[u8]) -> Result<Vec<u8>> {
    ensure!(blob.len() > 12, "Encrypted value too short");
    // ...
    Ok(vec![])
}

// 3. .context() -- 给错误添加上下文
fn open_db(path: &str) -> Result<Connection> {
    Connection::open(path).context("SQLite failed to open database")
}
```

[来源: reference/context7_anyhow_01.md]

### 4.3 anyhow vs thiserror：何时用哪个？

| 场景 | 选择 | 原因 |
|------|------|------|
| **应用层**（ZeroClaw 主程序） | `anyhow` | 灵活，不需要定义错误枚举 |
| **库层**（给别人用的 crate） | `thiserror` | 调用者需要 match 具体错误类型 |
| **混合使用** | 内部 `thiserror`，顶层 `anyhow` | 最佳实践 |

```rust
// thiserror -- 定义具体错误类型（库层）
#[derive(thiserror::Error, Debug)]
pub enum DataError {
    #[error("validation failed: {0}")]
    Validation(String),
    #[error("not found: {id}")]
    NotFound { id: u64 },
    #[error("I/O error")]
    Io(#[from] std::io::Error),  // #[from] 自动实现 From trait
}

// anyhow -- 在应用层包装（ZeroClaw 的做法）
fn process() -> anyhow::Result<()> {
    do_something().context("processing failed")?;  // 任何错误都能接
    Ok(())
}
```

## 五、一图胜千言

### 5.1 Result 与 Option 的结构

```
Result<T, E>                    Option<T>
┌─────────────────┐             ┌─────────────────┐
│  Ok(T)          │             │  Some(T)         │
│  ┌───────────┐  │             │  ┌───────────┐  │
│  │ 成功的值  │  │             │  │ 存在的值  │  │
│  └───────────┘  │             │  └───────────┘  │
├─────────────────┤             ├─────────────────┤
│  Err(E)         │             │  None            │
│  ┌───────────┐  │             │                  │
│  │ 错误信息  │  │             │  （什么都没有）  │
│  └───────────┘  │             │                  │
└─────────────────┘             └─────────────────┘

用途：操作可能失败               用途：值可能不存在
类比：try/catch                  类比：null/undefined
```

### 5.2 错误传播流程（? 运算符）

```
函数 A                    函数 B                    函数 C
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│ let x = b()?;│────────>│ let y = c()?;│────────>│ 实际操作     │
│              │  Ok(v)  │              │  Ok(v)  │              │
│ 继续执行...  │<────────│ 继续执行...  │<────────│ return Ok(v) │
│              │         │              │         │              │
│ return Err(e)│<────────│ return Err(e)│<────────│ return Err(e)│
└──────────────┘  Err(e) └──────────────┘  Err(e) └──────────────┘

? 运算符的作用：
  Ok(value)  → 解包 value，继续执行
  Err(error) → 立即返回 Err(error)，错误自动冒泡
```

### 5.3 TypeScript vs Rust 错误处理对照

```
TypeScript                          Rust
──────────                          ────
try {                               fn do_work() -> Result<Data> {
  const a = stepA();                    let a = step_a()?;
  const b = stepB(a);                   let b = step_b(a)?;
  const c = stepC(b);                   let c = step_c(b)?;
  return c;                             Ok(c)
} catch (e) {                       }
  console.error(e);
  throw e;                          // 调用方
}                                   match do_work() {
                                        Ok(data) => use(data),
                                        Err(e) => log::error!("{:#}", e),
                                    }
```

## 七、记忆口诀

### 核心三句话

> **Result = 成功或失败（Ok / Err）**
> **Option = 有或没有（Some / None）**
> **? = 错误冒泡（自动传播）**

### 对比记忆

> **TS 靠 try/catch 兜底，Rust 靠 Result 明示。**
> **TS 的 null 是地雷，Rust 的 Option 是路标。**
> **TS 的错误是隐式的，Rust 的错误是显式的。**

### ZeroClaw 口诀

> **anyhow 统一错误，? 号到处传播，context 注入上下文，bail 提前返回。**

## 八、常见误区

### 误区 1："Result 和 try/catch 差不多，只是语法不同"

**事实**：本质区别在于 **编译时 vs 运行时**。try/catch 是运行时机制，忘记写 catch 程序照样编译通过。Result 是编译时机制，不处理就编译不过。这不是语法糖的区别，是安全保证级别的区别。

### 误区 2："unwrap() 就是 Rust 版的不写 try/catch"

**事实**：`unwrap()` 在遇到 Err/None 时会 **panic（程序崩溃）**，比 TypeScript 的未捕获异常更严重。ZeroClaw 的原则是生产代码零 `unwrap()`，只在测试中使用。

```rust
// 危险！生产代码不要这样写
let value = some_result.unwrap();  // Err 时直接 panic

// 正确做法
let value = some_result?;                    // 传播错误
let value = some_result.unwrap_or(default);  // 提供默认值
let value = some_result.context("msg")?;     // 传播 + 上下文
```

### 误区 3："Option 和 Result 可以互换使用"

**事实**：语义不同。`Option` 表示"值可能不存在"（正常情况），`Result` 表示"操作可能失败"（错误情况）。配置字段用 `Option`，函数返回用 `Result`。

```rust
// Option -- 值可能不存在（这是正常的）
struct Config {
    api_key: Option<String>,  // 没有 API key 不是错误，只是没配置
}

// Result -- 操作可能失败（这是需要处理的）
fn connect(url: &str) -> Result<Connection> {  // 连接失败是错误
    // ...
}
```

### 误区 4："anyhow 就是把所有错误吞掉了"

**事实**：anyhow 不吞错误，它 **保留完整的错误链**。每个 `.context()` 都会在错误链上添加一层，调试时可以看到完整的错误传播路径。

```rust
// 错误链示例
Error: Configuration loading failed        // 最外层 context
  Caused by:
    0: failed to read config file           // 中间层 context
    1: No such file or directory (os error 2) // 原始错误
```

## 九、总结

```
  Result<T,E> 和 Option<T> 是 Rust 用枚举替代异常和空值的
  编译时安全错误处理机制，ZeroClaw 全局使用 anyhow::Result 统一错误传播。

  枚举替代异常 ──→ Result<T,E> = Ok(成功) | Err(失败)
  枚举替代空值 ──→ Option<T>   = Some(有值) | None(没有)
  编译时安全   ──→ 不处理就编译不过，运行时不会有意外
  ? 运算符     ──→ 错误自动冒泡，代码简洁
  anyhow       ──→ ZeroClaw 统一错误类型，bail!/ensure!/context()
  thiserror    ──→ 库层自定义错误，#[derive(Error)]

  ZeroClaw 四原则：零 panic | 安全优先 | 上下文丰富 | Trait 一致
```
