# Milvus架构概览 - 核心概念1：整体架构与分层设计

> 理解 Milvus 2.6 的三层架构设计，掌握存算分离的核心理念

---

## 原理讲解

### 1. 架构设计的第一性原理

**核心问题：** 如何构建一个可扩展、高性能、低成本的向量数据库？

**设计理念：**
```
向量检索的本质需求：
1. 海量数据存储（TB/PB 级）
2. 高性能计算（毫秒级响应）
3. 弹性扩展（应对流量波动）
4. 成本优化（降低运营成本）

解决方案：
存算分离 + 微服务架构 + 云原生设计
```

### 2. 三层架构详解

Milvus 2.6 采用经典的三层架构设计：

```
┌─────────────────────────────────────────────────────────────────┐
│                        Access Layer (接入层)                      │
│                                                                   │
│  ┌─────────┐                                                     │
│  │  Proxy  │  ← 负载均衡、请求路由、结果聚合                      │
│  └─────────┘                                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                    Coordinator Layer (协调层)                     │
│                                                                   │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐             │
│  │ Root Coord  │  │ Data Coord  │  │ Query Coord │             │
│  │ (元数据管理) │  │ (数据协调)   │  │ (查询协调)   │             │
│  └─────────────┘  └─────────────┘  └─────────────┘             │
│                                                                   │
│  2.6 新特性：Coord Merge（小规模部署时可合并）                    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      Worker Layer (工作层)                        │
│                                                                   │
│  ┌────────────┐  ┌───────────┐  ┌────────────┐  ┌─────────────┐│
│  │Query Node  │  │Data Node  │  │Index Node  │  │Streaming    ││
│  │(查询执行)   │  │(数据写入) │  │(索引构建)   │  │Node (2.6)   ││
│  └────────────┘  └───────────┘  └────────────┘  └─────────────┘│
│                                                                   │
│  特点：无状态、可水平扩展、故障隔离                                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                     Storage Layer (存储层)                        │
│                                                                   │
│  ┌──────────────┐  ┌──────────────────┐  ┌──────────────────┐  │
│  │Meta Storage  │  │Object Storage    │  │Message Queue     │  │
│  │(etcd)        │  │(S3/MinIO)        │  │(Woodpecker WAL)  │  │
│  │元数据存储     │  │向量数据/索引存储  │  │预写日志(2.6)     │  │
│  └──────────────┘  └──────────────────┘  └──────────────────┘  │
│                                                                   │
│  特点：持久化、高可用、低成本                                      │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 各层职责详解

#### 3.1 Access Layer (接入层)

**Proxy 组件：**
- **职责**：作为系统的统一入口，处理所有客户端请求
- **功能**：
  - 请求路由：将请求分发到合适的 Coordinator
  - 负载均衡：在多个 Worker 之间分配任务
  - 结果聚合：收集并合并多个 Worker 的结果
  - 连接管理：维护客户端连接池

**类比：** Proxy 就像餐厅的前台，接待客人，分配座位，协调服务

#### 3.2 Coordinator Layer (协调层)

**Root Coordinator：**
- **职责**：管理全局元数据
- **功能**：
  - Collection 的创建、删除、修改
  - Partition 的管理
  - Schema 的定义和验证
  - 全局 ID 分配
  - 时间戳服务（TSO）

**Data Coordinator：**
- **职责**：协调数据写入和管理
- **功能**：
  - Segment 的分配和管理
  - 数据刷新（Flush）调度
  - 索引构建任务调度
  - 数据压缩（Compaction）调度
  - 垃圾回收（GC）

**Query Coordinator：**
- **职责**：协调查询任务
- **功能**：
  - 数据加载到 Query Node
  - 查询任务分配
  - 负载均衡
  - 故障转移

**2.6 新特性：Coord Merge**
- 小规模部署时，三个 Coordinator 可以合并为一个进程
- 降低资源消耗，简化部署
- 适合单机或小集群场景

#### 3.3 Worker Layer (工作层)

**Query Node：**
- **职责**：执行向量检索
- **功能**：
  - 加载索引到内存
  - 执行向量相似度计算
  - 标量过滤
  - 结果排序和返回

**Data Node：**
- **职责**：处理数据写入
- **功能**：
  - 接收 Streaming Node 的数据流
  - 数据格式转换
  - 数据持久化到对象存储
  - Binlog 生成

**Index Node：**
- **职责**：构建向量索引
- **功能**：
  - 从对象存储读取原始向量
  - 构建索引（HNSW、IVF 等）
  - 索引持久化到对象存储

**Streaming Node（2.6 新增）：**
- **职责**：统一流式数据处理
- **功能**：
  - 接收所有数据写入请求
  - 数据流转和分发
  - 与 Woodpecker WAL 集成
  - 替代外部消息队列

#### 3.4 Storage Layer (存储层)

**Meta Storage (etcd)：**
- **存储内容**：
  - Collection/Partition 元数据
  - Schema 定义
  - 索引配置
  - Segment 信息
  - 节点状态

**Object Storage (S3/MinIO)：**
- **存储内容**：
  - 原始向量数据
  - 索引文件
  - Binlog 文件
  - Snapshot 文件

**Woodpecker WAL (2.6)：**
- **存储内容**：
  - 预写日志（Write-Ahead Log）
  - 数据变更记录
  - 事务日志

## 手写实现

### 简化版架构模拟

```python
"""
Milvus 2.6 架构的简化模拟实现
展示三层架构的核心交互逻辑
"""

from typing import List, Dict, Any
from dataclasses import dataclass
import time

# ===== Storage Layer =====

class MetaStorage:
    """元数据存储（模拟 etcd）"""
    def __init__(self):
        self.collections = {}
        self.segments = {}
    
    def create_collection(self, name: str, schema: Dict):
        self.collections[name] = {
            "schema": schema,
            "created_at": time.time()
        }
        print(f"[MetaStorage] Collection '{name}' created")
    
    def get_collection(self, name: str) -> Dict:
        return self.collections.get(name)

class ObjectStorage:
    """对象存储（模拟 S3/MinIO）"""
    def __init__(self):
        self.data = {}
        self.indexes = {}
    
    def write_data(self, segment_id: str, vectors: List):
        self.data[segment_id] = vectors
        print(f"[ObjectStorage] Data written to segment {segment_id}")
    
    def read_data(self, segment_id: str) -> List:
        return self.data.get(segment_id, [])
    
    def write_index(self, segment_id: str, index: Any):
        self.indexes[segment_id] = index
        print(f"[ObjectStorage] Index written for segment {segment_id}")

class WoodpeckerWAL:
    """预写日志（2.6 新增）"""
    def __init__(self, object_storage: ObjectStorage):
        self.object_storage = object_storage
        self.log = []
    
    def append(self, operation: Dict):
        # 直接写入对象存储（零磁盘架构）
        self.log.append(operation)
        print(f"[WoodpeckerWAL] Operation logged: {operation['type']}")

# ===== Worker Layer =====

class StreamingNode:
    """流式节点（2.6 核心组件）"""
    def __init__(self, wal: WoodpeckerWAL):
        self.wal = wal
        self.buffer = []
    
    def receive_data(self, data: Dict):
        """接收数据写入请求"""
        print(f"[StreamingNode] Received data: {data['id']}")
        # 写入 WAL
        self.wal.append({"type": "insert", "data": data})
        # 缓存数据
        self.buffer.append(data)
        return True
    
    def flush_to_data_node(self, data_node):
        """刷新数据到 Data Node"""
        if self.buffer:
            data_node.process_batch(self.buffer)
            self.buffer = []

class DataNode:
    """数据节点"""
    def __init__(self, object_storage: ObjectStorage):
        self.object_storage = object_storage
        self.segment_id = 0
    
    def process_batch(self, batch: List[Dict]):
        """处理批量数据"""
        print(f"[DataNode] Processing batch of {len(batch)} records")
        # 生成 Segment ID
        self.segment_id += 1
        segment_id = f"seg_{self.segment_id}"
        # 提取向量
        vectors = [item["vector"] for item in batch]
        # 写入对象存储
        self.object_storage.write_data(segment_id, vectors)
        return segment_id

class IndexNode:
    """索引节点"""
    def __init__(self, object_storage: ObjectStorage):
        self.object_storage = object_storage
    
    def build_index(self, segment_id: str):
        """构建索引"""
        print(f"[IndexNode] Building index for segment {segment_id}")
        # 从对象存储读取数据
        vectors = self.object_storage.read_data(segment_id)
        # 构建索引（简化：直接使用原始数据）
        index = {"vectors": vectors, "type": "HNSW"}
        # 写入对象存储
        self.object_storage.write_index(segment_id, index)
        return index

class QueryNode:
    """查询节点"""
    def __init__(self, object_storage: ObjectStorage):
        self.object_storage = object_storage
        self.loaded_indexes = {}
    
    def load_index(self, segment_id: str):
        """加载索引到内存"""
        print(f"[QueryNode] Loading index for segment {segment_id}")
        index = self.object_storage.indexes.get(segment_id)
        if index:
            self.loaded_indexes[segment_id] = index
    
    def search(self, query_vector: List[float], top_k: int = 10):
        """执行向量检索"""
        print(f"[QueryNode] Searching with top_k={top_k}")
        # 简化：返回模拟结果
        results = []
        for segment_id, index in self.loaded_indexes.items():
            # 模拟相似度计算
            for i, vec in enumerate(index["vectors"][:top_k]):
                results.append({
                    "id": f"{segment_id}_{i}",
                    "distance": 0.95 - i * 0.01
                })
        return sorted(results, key=lambda x: x["distance"], reverse=True)[:top_k]

# ===== Coordinator Layer =====

class RootCoordinator:
    """Root Coordinator"""
    def __init__(self, meta_storage: MetaStorage):
        self.meta_storage = meta_storage
    
    def create_collection(self, name: str, schema: Dict):
        """创建 Collection"""
        print(f"[RootCoord] Creating collection '{name}'")
        self.meta_storage.create_collection(name, schema)

class DataCoordinator:
    """Data Coordinator"""
    def __init__(self):
        self.segments = []
    
    def schedule_flush(self, streaming_node, data_node):
        """调度数据刷新"""
        print(f"[DataCoord] Scheduling flush")
        streaming_node.flush_to_data_node(data_node)
    
    def schedule_index_build(self, segment_id: str, index_node):
        """调度索引构建"""
        print(f"[DataCoord] Scheduling index build for {segment_id}")
        index_node.build_index(segment_id)

class QueryCoordinator:
    """Query Coordinator"""
    def __init__(self):
        pass
    
    def load_collection(self, segment_ids: List[str], query_node):
        """加载 Collection 到 Query Node"""
        print(f"[QueryCoord] Loading {len(segment_ids)} segments")
        for segment_id in segment_ids:
            query_node.load_index(segment_id)

# ===== Access Layer =====

class Proxy:
    """Proxy（接入层）"""
    def __init__(self, root_coord, data_coord, query_coord):
        self.root_coord = root_coord
        self.data_coord = data_coord
        self.query_coord = query_coord
    
    def create_collection(self, name: str, schema: Dict):
        """创建 Collection"""
        return self.root_coord.create_collection(name, schema)
    
    def insert(self, collection: str, data: List[Dict], streaming_node):
        """插入数据"""
        print(f"[Proxy] Inserting {len(data)} records to '{collection}'")
        for item in data:
            streaming_node.receive_data(item)
    
    def search(self, collection: str, query_vector: List[float], 
               top_k: int, query_node):
        """执行检索"""
        print(f"[Proxy] Searching in '{collection}'")
        return query_node.search(query_vector, top_k)

# ===== 完整流程演示 =====

def demo_milvus_architecture():
    """演示 Milvus 2.6 架构的完整流程"""
    
    print("=" * 60)
    print("Milvus 2.6 架构演示")
    print("=" * 60)
    
    # 1. 初始化 Storage Layer
    print("\n[1] 初始化 Storage Layer")
    meta_storage = MetaStorage()
    object_storage = ObjectStorage()
    wal = WoodpeckerWAL(object_storage)
    
    # 2. 初始化 Worker Layer
    print("\n[2] 初始化 Worker Layer")
    streaming_node = StreamingNode(wal)
    data_node = DataNode(object_storage)
    index_node = IndexNode(object_storage)
    query_node = QueryNode(object_storage)
    
    # 3. 初始化 Coordinator Layer
    print("\n[3] 初始化 Coordinator Layer")
    root_coord = RootCoordinator(meta_storage)
    data_coord = DataCoordinator()
    query_coord = QueryCoordinator()
    
    # 4. 初始化 Access Layer
    print("\n[4] 初始化 Access Layer")
    proxy = Proxy(root_coord, data_coord, query_coord)
    
    # 5. 创建 Collection
    print("\n[5] 创建 Collection")
    schema = {
        "fields": [
            {"name": "id", "type": "int64"},
            {"name": "vector", "type": "float_vector", "dim": 128}
        ]
    }
    proxy.create_collection("documents", schema)
    
    # 6. 插入数据
    print("\n[6] 插入数据")
    data = [
        {"id": i, "vector": [0.1 * i] * 128}
        for i in range(100)
    ]
    proxy.insert("documents", data, streaming_node)
    
    # 7. 刷新数据
    print("\n[7] 刷新数据到 Data Node")
    data_coord.schedule_flush(streaming_node, data_node)
    
    # 8. 构建索引
    print("\n[8] 构建索引")
    segment_id = "seg_1"
    data_coord.schedule_index_build(segment_id, index_node)
    
    # 9. 加载数据
    print("\n[9] 加载数据到 Query Node")
    query_coord.load_collection([segment_id], query_node)
    
    # 10. 执行检索
    print("\n[10] 执行向量检索")
    query_vector = [0.5] * 128
    results = proxy.search("documents", query_vector, top_k=5, query_node=query_node)
    
    print("\n检索结果：")
    for i, result in enumerate(results):
        print(f"  {i+1}. ID: {result['id']}, Distance: {result['distance']:.4f}")
    
    print("\n" + "=" * 60)
    print("演示完成！")
    print("=" * 60)

if __name__ == "__main__":
    demo_milvus_architecture()
```

### 场景1：文档问答系统的架构映射

```python
"""
RAG 系统如何利用 Milvus 2.6 的三层架构
"""

# RAG 系统的数据流
"""
1. 文档入库流程：
   用户上传文档
     ↓
   Proxy 接收请求
     ↓
   Streaming Node 处理文档流
     ↓
   Data Node 持久化向量
     ↓
   Index Node 构建索引
     ↓

2. 查询流程：
   用户提问
     ↓
   Proxy 接收查询
     ↓
   Query Node 检索相关文档
     ↓
   返回 Top-K 结果
     ↓
   LLM 生成答案

# 架构优势在 RAG 中的体现
"""
1. 存算分离：
   - 文档库增长 → 只扩展对象存储
   - 查询增多 → 只扩展 Query Node
   - 成本优化 → 使用廉价对象存储

2. 微服务架构：
   - 文档入库慢 → 优化 Data Node
   - 检索慢 → 优化 Query Node
   - 索引构建慢 → 优化 Index Node

3. 零磁盘架构：
   - 降低部署成本
   - 简化运维
   - 提高可靠性
"""
```

## 架构设计的关键洞察

### 1. 为什么是三层而不是两层或四层？

**两层架构的问题：**
- 协调逻辑和执行逻辑耦合
- 难以独立优化
- 扩展性差

**四层架构的问题：**
- 过度设计
- 增加复杂度
- 性能开销大

**三层架构的优势：**
- 职责清晰：协调、执行、存储
- 易于扩展：每层独立扩展
- 性能平衡：减少网络跳转

### 2. 为什么 Worker 层要无状态？

**有状态的问题：**
- 扩展困难（需要数据迁移）
- 故障恢复慢（需要恢复状态）
- 负载均衡难（状态绑定节点）

**无状态的优势：**
- 快速扩展（启动即可用）
- 快速恢复（替换节点）
- 灵活调度（任意节点处理）

### 3. 为什么要存算分离？

**成本对比（100TB 数据，1000 QPS）：**

```
传统架构（存算耦合）：
  - 需要 20 台高配机器（SSD + 高 CPU）
  - 成本：$50,000/月
  - 资源利用率：50%（夜间浪费）

存算分离架构：
  - 对象存储：$2,300/月
  - 计算节点：白天 10 台，夜间 2 台
  - 成本：$2,300 + $15,000 = $17,300/月
  - 节省：65%
```
