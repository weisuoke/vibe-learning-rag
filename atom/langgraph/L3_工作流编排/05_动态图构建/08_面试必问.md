# 动态图构建 - 面试必问

> 两道高频面试题，覆盖 LangGraph 动态路由机制的核心区别和多代理系统设计。每道题都有"普通回答"和"出彩回答"的对比，帮你在面试中脱颖而出。

---

## 问题1："LangGraph 中条件边、Send 和 Command 三种动态路由机制有什么区别？分别适用什么场景？"

### 普通回答（不出彩）

> 条件边是根据状态选择路径，Send 是动态创建并行任务，Command 是在节点内部路由。三者都能实现动态路由，但用法不同。

这个回答的问题：只说了"是什么"，没有说"为什么"和"怎么选"。面试官听完还是不知道你是否真正理解了它们的设计意图。

### 出彩回答（推荐）

> 这三种机制解决的是动态路由中三个不同层次的问题，核心区别在于**路由逻辑的位置**和**运行时的灵活度**。

#### 第一层：条件边——编译时定义规则，运行时选择路径

条件边是最基础的动态路由。路由规则在图编译时就确定了（哪些路径可选），但具体走哪条路径是运行时根据状态决定的。

```python
# 路由规则在编译时定义：只能去 "pdf"、"word" 或 "fallback"
graph.add_conditional_edges(
    "classify",
    route_by_type,
    path_map={"pdf": "process_pdf", "word": "process_word", "unknown": "fallback"},
)
```

适用场景：分支数量固定、路径在编译时已知。比如文档类型路由（PDF/Word/Image）、LLM 意图分类（FAQ/技术/计费）。

关键特征：路由逻辑在**边上**（路由函数），节点只负责计算和状态更新。

#### 第二层：Send——运行时动态创建并行任务

Send 解决的是"编译时不知道要创建多少个任务"的问题。它允许在运行时动态生成任意数量的并行任务，每个任务可以携带不同的状态。

```python
from langgraph.types import Send

def fan_out_to_workers(state: OverallState):
    # 运行时才知道有多少个 subject，每个都创建一个独立任务
    return [Send("generate_joke", {"subject": s}) for s in state["subjects"]]

graph.add_conditional_edges("collect_subjects", fan_out_to_workers)
```

适用场景：Map-Reduce 模式——输入一个列表，对每个元素并行处理，最后汇总结果。比如批量文档处理、多查询并行检索、多角度分析。

关键特征：任务数量是**运行时动态的**，不是编译时固定的。

[来源: Context7 官方文档 - Send API]

#### 第三层：Command——节点内部同时更新状态和控制路由

Command 把路由决策从"边"移到了"节点内部"，让状态更新和路由控制成为一个原子操作。

```python
from langgraph.types import Command
from typing import Literal

def smart_router(state: State) -> Command[Literal["agent_a", "agent_b"]]:
    if state["task"] == "coding":
        return Command(
            update={"assigned_to": "agent_a", "priority": "high"},
            goto="agent_a",
        )
    return Command(
        update={"assigned_to": "agent_b", "priority": "normal"},
        goto="agent_b",
    )
```

适用场景：需要在路由的同时更新状态的场景，特别是多代理系统中的代理切换（通过 `graph=Command.PARENT` 跨子图导航）。

关键特征：路由逻辑在**节点内部**，状态更新和路由是**原子操作**。

[来源: Medium - A Beginner's Guide to Dynamic Routing in LangGraph with Command()]

#### 三者对比总结

| 维度 | 条件边 | Send | Command |
|------|--------|------|---------|
| 路由逻辑位置 | 边上（路由函数） | 边上（返回 Send 列表） | 节点内部 |
| 路径数量 | 编译时固定 | 运行时动态 | 编译时固定（类型注解） |
| 并行能力 | 可返回多个路径 | 天然并行，每个 Send 独立 | 单一目标 |
| 状态更新 | 节点更新，边只路由 | 每个 Send 携带独立状态 | 路由 + 更新一体化 |
| 跨图导航 | 不支持 | 不支持 | 支持（Command.PARENT） |
| 典型场景 | if-else 分支 | map-reduce 并行 | 多代理切换 |

#### 选型决策树

```
需要动态路由？
├── 分支数量编译时已知？
│   ├── 是 → 需要同时更新状态？
│   │   ├── 是 → Command
│   │   └── 否 → 条件边
│   └── 否 → Send（map-reduce）
└── 需要跨子图导航？
    └── 是 → Command + Command.PARENT
```

> 面试加分点：提到"路由逻辑的位置不同"这个核心区别，说明你理解了 LangGraph 的设计哲学——条件边把路由逻辑放在边上实现关注点分离，Command 把路由逻辑放在节点内部实现原子性操作，两种设计各有取舍。

## 问题2："如何在 LangGraph 中实现一个可扩展的多代理系统？"

### 普通回答（不出彩）

> 可以用子图来实现多代理系统，每个代理是一个子图，主图负责路由。用 Command 可以在代理之间切换。

这个回答太笼统了。面试官想听的是具体的架构设计、状态管理策略、以及你在实际项目中遇到的权衡。

### 出彩回答（推荐）

> 可扩展的多代理系统需要解决三个核心问题：**代理隔离**、**状态共享**、**路由控制**。我会用三层架构来设计。

#### 第一层：主图层——Router + 子图节点

主图负责接收用户请求、路由到合适的代理、收集最终结果。

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import Command
from typing import Literal, Annotated
from typing_extensions import TypedDict
import operator


# 主图状态：只包含跨代理共享的信息
class OrchestratorState(TypedDict):
    user_request: str
    current_agent: str
    messages: Annotated[list[str], operator.add]  # reducer 确保消息不丢失
    final_result: str


def orchestrator(state: OrchestratorState) -> Command[
    Literal["code_agent", "research_agent", "writing_agent"]
]:
    """主路由器：分析请求并路由到对应代理

    用 Command 实现路由，因为需要同时更新 current_agent 状态。
    Command.PARENT 不在这里用——这里是主图，不是子图。
    """
    request = state["user_request"].lower()

    if any(kw in request for kw in ["代码", "函数", "实现"]):
        target = "code_agent"
    elif any(kw in request for kw in ["调研", "搜索", "查找"]):
        target = "research_agent"
    else:
        target = "writing_agent"

    return Command(
        update={
            "current_agent": target,
            "messages": [f"[Orchestrator] 路由到 {target}"],
        },
        goto=target,
    )
```

关键设计决策：主图用 `Command` 而不是条件边，因为路由的同时需要更新 `current_agent` 状态。

#### 第二层：子图层——每个代理独立的 StateGraph

每个代理是一个独立的 `StateGraph`，有自己的内部状态和处理逻辑。代理完成任务后，通过 `Command(graph=Command.PARENT)` 返回主图。

```python
# 代码代理的内部状态
class CodeAgentState(TypedDict):
    user_request: str          # 从主图继承
    messages: Annotated[list[str], operator.add]  # 共享键
    code_output: str           # 代理私有状态
    test_passed: bool          # 代理私有状态


def code_plan(state: CodeAgentState) -> dict:
    """代码代理：规划步骤"""
    return {
        "messages": [f"[CodeAgent] 分析需求: {state['user_request']}"],
        "code_output": "",
    }


def code_generate(state: CodeAgentState) -> dict:
    """代码代理：生成代码"""
    return {
        "code_output": f"def solution(): pass  # 基于: {state['user_request']}",
        "messages": ["[CodeAgent] 代码生成完成"],
    }


def code_review(state: CodeAgentState) -> Command[Literal["__end__"]]:
    """代码代理：审查并返回主图

    关键：用 Command(graph=Command.PARENT) 返回主图。
    update 中只包含主图状态中存在的共享键。
    """
    return Command(
        update={
            "messages": [f"[CodeAgent] 审查完成，代码: {state['code_output'][:50]}"],
            "final_result": state["code_output"],
        },
        goto="__end__",           # 结束子图
        graph=Command.PARENT,     # 导航到父图
    )


# 构建代码代理子图
def build_code_agent() -> StateGraph:
    agent = StateGraph(CodeAgentState)
    agent.add_node("plan", code_plan)
    agent.add_node("generate", code_generate)
    agent.add_node("review", code_review)
    agent.add_edge(START, "plan")
    agent.add_edge("plan", "generate")
    agent.add_edge("generate", "review")
    return agent  # 注意：返回未编译的 StateGraph，由主图编译
```

[来源: Context7 官方文档 - Command.PARENT 父图导航]

关键设计决策：
- `code_output` 和 `test_passed` 是代理私有状态，主图看不到
- `messages` 是共享键，通过 reducer（`operator.add`）确保主图和子图的消息合并
- `Command(graph=Command.PARENT)` 让子图能"跳出"自己，回到主图

#### 第三层：状态层——共享键用 Reducer，私有键各自管理

这是多代理系统中最容易出错的地方。状态设计的核心原则：

```python
# 主图状态（共享层）
class OrchestratorState(TypedDict):
    user_request: str                                    # 只读共享
    current_agent: str                                   # 主图管理
    messages: Annotated[list[str], operator.add]          # 共享 + reducer
    final_result: str                                    # 子图写入

# 子图状态（私有层）
class CodeAgentState(TypedDict):
    user_request: str                                    # 从主图继承（同名键）
    messages: Annotated[list[str], operator.add]          # 共享键（必须同 reducer）
    code_output: str                                     # 私有键
    test_passed: bool                                    # 私有键
```

状态隔离 vs 共享的权衡：

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 完全隔离 | 代理互不干扰 | 信息传递麻烦 | 独立任务 |
| 完全共享 | 信息流通顺畅 | 状态冲突风险 | 紧密协作 |
| 混合（推荐） | 平衡隔离和共享 | 设计复杂度高 | 生产环境 |

混合策略的规则：
1. **只读数据**（如 `user_request`）：通过同名键自动继承
2. **追加数据**（如 `messages`）：用 `Annotated[list, operator.add]` reducer
3. **私有数据**（如 `code_output`）：只在子图状态中定义

#### 完整组装示例

```python
def build_multi_agent_system():
    """组装完整的多代理系统"""
    main_graph = StateGraph(OrchestratorState)

    # 主路由器
    main_graph.add_node("orchestrator", orchestrator)

    # 子图作为节点（传入未编译的 StateGraph）
    main_graph.add_node("code_agent", build_code_agent())
    main_graph.add_node("research_agent", build_research_agent())
    main_graph.add_node("writing_agent", build_writing_agent())

    # 入口
    main_graph.add_edge(START, "orchestrator")
    # Command 自带路由，不需要额外的条件边

    # 子图完成后 → 结束
    main_graph.add_edge("code_agent", END)
    main_graph.add_edge("research_agent", END)
    main_graph.add_edge("writing_agent", END)

    return main_graph.compile()
```

#### 可扩展性设计

添加新代理只需要三步：

```python
# 1. 定义新代理的子图
def build_debug_agent() -> StateGraph:
    agent = StateGraph(DebugAgentState)
    # ... 添加节点和边 ...
    return agent

# 2. 在主图中注册
main_graph.add_node("debug_agent", build_debug_agent())
main_graph.add_edge("debug_agent", END)

# 3. 在路由器的 Command 类型注解中添加
def orchestrator(state) -> Command[
    Literal["code_agent", "research_agent", "writing_agent", "debug_agent"]
]:
    # ... 添加路由逻辑 ...
```

不需要修改现有代理的代码，符合开闭原则。

> 面试加分点：提到"共享键必须在主图和子图中使用相同的 reducer"这个细节。很多人在实际开发中会踩这个坑——子图的 `messages` 用了 `operator.add`，但主图忘了加 reducer，导致消息被覆盖而不是追加。

## 面试技巧总结

### 回答动态图相关问题的框架

1. **先说核心区别**（一句话定位）
2. **再给代码示例**（证明你写过，不是背的）
3. **最后说实际经验**（踩过什么坑、做过什么权衡）

### 常见追问及应对

| 追问 | 应对要点 |
|------|---------|
| "条件边和 Command 能混用吗？" | 可以，但要注意 Command 的 goto 不会阻止静态边执行。如果节点同时有静态边和 Command goto，两者都会执行 |
| "Send 的并行任务如何汇总结果？" | 通过 reducer（如 `operator.add`）自动合并。所有 Send 任务完成后，下游节点才会执行 |
| "Command.PARENT 能跨多层吗？" | 只能导航到最近的父图。如果需要跨多层，需要逐层传递 |
| "动态图的调试怎么做？" | 用 `graph.get_graph().draw_mermaid()` 可视化图结构，用 LangSmith 追踪运行时路径 |
