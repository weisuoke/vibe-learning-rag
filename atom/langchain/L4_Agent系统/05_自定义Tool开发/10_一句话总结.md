# 10_一句话总结

> 用一句话精准概括自定义 Tool 开发的核心本质

---

## 核心总结

**自定义 Tool 开发是通过装饰器或类继承将 Python 函数封装为 Agent 可调用的工具,用 Schema 定义参数让 LLM 理解如何使用,用错误处理让 Agent 能够自我修正,从而扩展 LLM 的能力边界。**

（60 个汉字）

---

## 核心总结拆解

让我们逐字理解这句话背后的深意:

### 1. "通过装饰器或类继承"
**关键词**: 创建方式的多样性

- **@tool 装饰器**: 80% 场景的首选,简洁高效
- **StructuredTool**: 需要显式控制时使用
- **BaseTool 继承**: 复杂状态管理时使用

**设计哲学**: 简单场景简单做,复杂场景有工具

---

### 2. "将 Python 函数封装为 Agent 可调用的工具"
**关键词**: 函数是基础,封装是手段

- **函数**: 业务逻辑的最小单元
- **封装**: 添加元数据(名称、描述、Schema)
- **Agent 可调用**: 实现 Runnable 接口,集成到 LangChain 生态

**本质**: 工具 = 函数 + 元数据 + 接口协议

---

### 3. "用 Schema 定义参数让 LLM 理解如何使用"
**关键词**: Schema 是桥梁

- **Schema 定义**: 参数类型、约束、描述
- **LLM 理解**: 通过 Schema 知道如何传参
- **运行时验证**: 确保参数合法性

**核心价值**: Schema 是 LLM 和工具之间的"通信协议"

---

### 4. "用错误处理让 Agent 能够自我修正"
**关键词**: 容错机制

- **ToolException**: 可恢复错误,不中断 Agent
- **普通异常**: 致命错误,中断 Agent
- **自我修正**: Agent 可以尝试其他策略

**设计理念**: 工具失败不应导致整个对话失败

---

### 5. "从而扩展 LLM 的能力边界"
**关键词**: 工具的终极价值

- **突破局限**: LLM 只能生成文本,工具让它能"行动"
- **实时信息**: 突破知识截止日期
- **确定性操作**: 避免 LLM 的不确定性
- **系统集成**: 连接企业内部系统

**核心意义**: 工具是 Agent 从"聊天机器人"进化为"行动者"的关键

## 核心设计原则

### 1. 简洁性原则
**优先使用 @tool 装饰器**

```python
# ✅ 推荐: 简洁高效
@tool
def search(query: str) -> str:
    """搜索工具"""
    return search_api(query)

# ❌ 过度设计: 简单工具使用 BaseTool
class SearchTool(BaseTool):
    name = "search"
    def _run(self, query: str) -> str:
        return search_api(query)
```

---

### 2. 安全性原则
**Schema 验证 + 参数化查询**

```python
class QueryInput(BaseModel):
    table: Literal["users", "products"] = Field(description="表名")
    columns: list[str] = Field(description="列名")

@tool(args_schema=QueryInput)
def query_db(table: str, columns: list[str]) -> str:
    """安全的数据库查询"""
    # 白名单验证 + 参数化查询
    cols = ", ".join(columns)
    query = f"SELECT {cols} FROM {table}"
    return db.execute(query)
```

---

### 3. 容错性原则
**使用 ToolException 处理可恢复错误**

```python
@tool
def api_call(query: str) -> str:
    """API 调用"""
    try:
        response = requests.get(f"https://api.example.com?q={query}")
        response.raise_for_status()
        return response.text
    except requests.Timeout:
        raise ToolException("API 超时,请稍后重试")  # 不中断 Agent
```

---

### 4. 可观测性原则
**清晰的描述 + 详细的日志**

```python
@tool
def search_documents(query: str, top_k: int = 5) -> str:
    """从知识库检索相关文档.

    当用户询问产品信息、技术文档、政策规定时使用此工具.

    Args:
        query: 搜索查询,用自然语言描述需求
        top_k: 返回结果数量(1-10)
    """
    logger.info(f"Searching for: {query}, top_k={top_k}")
    results = vector_store.similarity_search(query, k=top_k)
    logger.info(f"Found {len(results)} documents")
    return format_results(results)
```

## 常见误区总结

### 误区 1: "@tool 功能有限"
**错误**: 认为 @tool 只是语法糖,功能不如 BaseTool

**正确**: @tool 支持几乎所有高级特性(异步、注入、错误处理)

**决策**: 99% 场景使用 @tool,1% 场景使用 BaseTool

---

### 误区 2: "Schema 越详细越好"
**错误**: 过度详细的 Schema 会困扰 LLM

**正确**: 简洁清晰的 Schema 更有效

**原则**: 描述 1-2 句话,示例 1-2 个,约束 2-3 个

---

### 误区 3: "异步总是更快"
**错误**: 所有工具都使用异步

**正确**: 只在 I/O 密集型任务中使用异步

**决策**: 有 await(等待 I/O) → 异步,纯计算 → 同步

---

### 误区 4: "ToolException 降低可靠性"
**错误**: 避免使用 ToolException

**正确**: ToolException 提高系统容错能力

**分类**: 致命错误 → 普通异常,可恢复错误 → ToolException

## 快速参考卡

### 三种创建方式对比

| 方式 | 代码量 | 灵活性 | 适用场景 |
|------|--------|--------|----------|
| **@tool** | 最少 | 中等 | 80% 场景 |
| **StructuredTool** | 中等 | 高 | 包装现有 API |
| **BaseTool** | 最多 | 最高 | 状态管理 |

---

### 决策树

```
需要创建工具?
├─ 需要管理持久化状态?
│  ├─ 是 → BaseTool 继承
│  └─ 否 → 继续
│
└─ 需要包装现有 API?
   ├─ 是 → StructuredTool.from_function()
   └─ 否 → @tool 装饰器
```

### 代码模板

```python
# 最简单的工具
@tool
def my_tool(param: str) -> str:
    """工具描述.

    Args:
        param: 参数描述
    """
    return f"Result: {param}"

# 复杂参数工具
class MyInput(BaseModel):
    param: str = Field(description="参数描述")

@tool(args_schema=MyInput)
def complex_tool(param: str) -> str:
    """工具描述"""
    return f"Result: {param}"

# 异步工具
@tool
async def async_tool(param: str) -> str:
    """异步工具"""
    async with httpx.AsyncClient() as client:
        response = await client.get(f"https://api.example.com?q={param}")
        return response.text
```

## 最后的话

**自定义 Tool 开发的本质**:

> 工具是 Agent 的"手"和"眼睛",让 LLM 从"思考者"变为"行动者"。通过简洁的装饰器语法、清晰的 Schema 定义、健壮的错误处理,我们赋予 Agent 与外部世界交互的能力,从而构建真正实用的 AI 应用。

**核心要点**:
- 优先使用 @tool(简洁高效)
- Schema 是 LLM 理解工具的关键
- ToolException 让 Agent 能够自我修正
- 工具设计要考虑安全性和可观测性

**记住**: 好的工具设计 = 简洁的接口 + 清晰的描述 + 健壮的错误处理
