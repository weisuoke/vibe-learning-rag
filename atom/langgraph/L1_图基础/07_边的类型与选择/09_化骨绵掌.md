## 卡片1：边是什么？直觉理解

**一句话：** 边是图中连接节点的路径，决定了工作流的执行顺序和分支逻辑。

**举例：**

想象你在使用导航 App：
- **普通边**：从家到公司的固定路线（每天都走这条路）
- **条件边**：根据路况选择不同路线（堵车走高速，不堵走市区）
- **等待边**：等所有家人都准备好才出门（AND 逻辑）

在 LangGraph 中：
```python
# 普通边：固定路径
builder.add_edge("A", "B")  # A 完成后总是执行 B

# 条件边：动态路径
builder.add_conditional_edges(
    "A",
    route_function,  # 根据状态决定下一步
    {"path1": "B", "path2": "C"}
)
```

**应用：** 边是构建 LangGraph 工作流的核心，决定了节点的执行顺序和分支逻辑。理解边的类型是设计复杂工作流的基础。

## 卡片2：边的形式化定义

**一句话：** 边是有向图中的有向连接，定义了从源节点到目标节点的转移规则。

**形式化表示：**

```
边 = (源节点, 目标节点, 转移条件)

其中：
- 源节点：边的起点（必须是已定义的节点）
- 目标节点：边的终点（可以是节点或 END）
- 转移条件：决定是否执行该边的规则
  - 固定转移：总是执行（普通边）
  - 条件转移：根据状态决定（条件边）
  - 同步转移：等待多个源节点（等待边）
```

**分类标准：**

1. **按转移规则**：固定边 vs 条件边
2. **按源节点数量**：单源边 vs 多源边
3. **按目标节点数量**：单目标边 vs 多目标边
4. **按状态传递**：标准边 vs Send 边

**应用：** 形式化定义帮助我们精确理解边的行为，在设计复杂工作流时避免歧义。

## 卡片3：普通边（Normal Edge）

**一句话：** 普通边是固定路由的边，源节点完成后总是执行目标节点。

**核心特征：**
- API：`add_edge(start, end)`
- 转移规则：确定性，无条件执行
- 使用场景：线性流程、固定步骤

**代码示例：**

```python
from langgraph.graph import StateGraph, START, END

builder = StateGraph(State)
builder.add_node("load", load_documents)
builder.add_node("chunk", chunk_documents)
builder.add_node("embed", embed_documents)

# 普通边：固定流程
builder.add_edge(START, "load")      # 开始 → 加载
builder.add_edge("load", "chunk")    # 加载 → 分块
builder.add_edge("chunk", "embed")   # 分块 → 向量化
builder.add_edge("embed", END)       # 向量化 → 结束
```

**内部实现：**
- 存储在 `self.edges` 集合中
- 编译时写入 `branch:to:{target}` 通道
- 源节点完成后立即触发目标节点

**应用：** 适合 RAG 文档处理流程、数据预处理管道等确定性场景。

## 卡片4：条件边（Conditional Edge）

**一句话：** 条件边根据路由函数的返回值动态决定下一个节点，实现分支逻辑。

**核心特征：**
- API：`add_conditional_edges(source, path, path_map)`
- 转移规则：动态，根据状态决定
- 使用场景：需要决策的场景

**代码示例：**

```python
from typing import Literal

def route_by_quality(state: State) -> Literal["high", "low"]:
    """根据文档质量路由"""
    if state["quality_score"] > 0.8:
        return "high"
    return "low"

builder.add_conditional_edges(
    "grade_documents",
    route_by_quality,
    {
        "high": "generate",      # 高质量 → 直接生成
        "low": "web_search"      # 低质量 → 网络搜索
    }
)
```

**路由函数设计：**
- 必须是纯函数（只读状态，不修改）
- 使用 `Literal` 类型注解提升可维护性
- 返回值必须在 `path_map` 中定义

**应用：** 适合 RAG 质量评估、查询路由、错误处理等需要决策的场景。

## 卡片5：等待边（Waiting Edge）

**一句话：** 等待边等待多个源节点全部完成后才执行目标节点，实现 AND 同步逻辑。

**核心特征：**
- API：`add_edge([node1, node2, ...], target)`
- 转移规则：同步，等待所有源节点
- 使用场景：并行任务汇聚

**代码示例：**

```python
builder = StateGraph(State)
builder.add_node("retrieve_vector", retrieve_from_vector_db)
builder.add_node("retrieve_keyword", retrieve_from_keyword_db)
builder.add_node("merge_results", merge_and_rerank)

# 等待边：等待两个检索任务都完成
builder.add_edge(START, "retrieve_vector")
builder.add_edge(START, "retrieve_keyword")
builder.add_edge(
    ["retrieve_vector", "retrieve_keyword"],  # 等待这两个节点
    "merge_results"                            # 才执行合并
)
```

**内部实现：**
- 使用 `NamedBarrierValue` 通道同步
- 创建 `join:node1+node2:target` 通道
- 所有源节点完成后才触发目标节点

**应用：** 适合混合检索、多模型并行推理、分布式任务汇聚等场景。

## 卡片6：序列边（Sequence Edge）

**一句话：** 序列边是语法糖，自动添加节点和边，简化线性流程的定义。

**核心特征：**
- API：`add_sequence([node1, node2, ...])`
- 转移规则：自动创建普通边
- 使用场景：简化顺序节点添加

**代码示例：**

```python
# 传统方式：逐个添加节点和边
builder.add_node("load", load_documents)
builder.add_node("chunk", chunk_documents)
builder.add_node("embed", embed_documents)
builder.add_edge("load", "chunk")
builder.add_edge("chunk", "embed")

# 序列边：一行搞定
builder.add_sequence([
    ("load", load_documents),
    ("chunk", chunk_documents),
    ("embed", embed_documents)
])
```

**内部实现：**
- 内部调用 `add_node()` 和 `add_edge()`
- 自动推断节点名称或使用手动指定
- 按顺序连接所有节点

**应用：** 适合 RAG 文档处理流程、数据预处理管道等线性场景，减少样板代码。

## 卡片7：动态路由（Send）

**一句话：** Send 对象允许条件边向目标节点发送自定义状态，实现动态并行执行（fan-out）。

**核心特征：**
- API：条件边返回 `Send(node, state)`
- 转移规则：动态并行，每个 Send 独立执行
- 使用场景：Map-Reduce 模式

**代码示例：**

```python
from langgraph.types import Send

def fan_out_by_topics(state: State) -> list[Send]:
    """为每个主题创建并行任务"""
    return [
        Send("summarize", {"topic": topic})
        for topic in state["topics"]
    ]

builder.add_conditional_edges(START, fan_out_by_topics)
builder.add_node("summarize", summarize_topic)
builder.add_edge("summarize", END)

# 执行：
result = graph.invoke({
    "topics": ["AI", "ML", "Deep Learning"],
    "summaries": []
})
# 三个 summarize 节点并行执行
```

**关键点：**
- 每个 Send 可以发送不同的状态
- 目标节点会被执行多次（每个 Send 一次）
- 适合动态数量的并行任务

**应用：** 适合批量文档处理、多查询并行检索、动态代理调度等场景。

## 卡片8：5种边类型对比与选择

**一句话：** 根据流程特征选择合适的边类型，避免混用导致的问题。

**对比表格：**

| 边类型 | API | 转移规则 | 源节点数 | 目标节点数 | 使用场景 |
|--------|-----|----------|----------|------------|----------|
| 普通边 | `add_edge(A, B)` | 固定 | 1 | 1 | 线性流程 |
| 条件边 | `add_conditional_edges()` | 动态 | 1 | 1或多 | 分支决策 |
| 等待边 | `add_edge([A,B], C)` | 同步 | 多 | 1 | 并行汇聚 |
| 序列边 | `add_sequence([...])` | 固定 | 1 | 1 | 简化线性 |
| Send边 | 返回 `Send()` | 动态并行 | 1 | 动态多 | Map-Reduce |

**选择策略：**

1. **简单流程** → 普通边或序列边
2. **条件分支** → 条件边
3. **并行等待** → 等待边
4. **动态并行** → 条件边 + Send

**常见陷阱：**
- ❌ 条件边和普通边同时指向同一节点（导致重复执行）
- ❌ Command 无法覆盖 `add_edge` 定义的边
- ❌ 路由函数修改状态（应该是纯函数）

**应用：** 理解边的选择策略可以避免 90% 的工作流设计问题。

## 卡片9：RAG 工作流中的边使用实战

**一句话：** RAG 工作流结合多种边类型实现复杂的检索-生成逻辑。

**Self-RAG 工作流示例：**

```python
workflow = StateGraph(State)

# 节点定义
workflow.add_node("retrieve", retrieve_documents)
workflow.add_node("grade_documents", grade_relevance)
workflow.add_node("transform_query", rewrite_query)
workflow.add_node("web_search", search_web)
workflow.add_node("generate", generate_answer)

# 边定义
workflow.add_edge(START, "retrieve")                    # 普通边：开始检索
workflow.add_edge("retrieve", "grade_documents")        # 普通边：评估文档

# 条件边1：根据文档质量决策
workflow.add_conditional_edges(
    "grade_documents",
    decide_to_generate,
    {
        "transform_query": "transform_query",  # 质量差 → 改写查询
        "generate": "generate"                 # 质量好 → 直接生成
    }
)

workflow.add_edge("transform_query", "retrieve")        # 普通边：重新检索

# 条件边2：根据生成质量决策
workflow.add_conditional_edges(
    "generate",
    grade_generation,
    {
        "not supported": "generate",           # 不支持 → 重新生成
        "useful": END,                         # 有用 → 结束
        "not useful": "transform_query"        # 无用 → 改写查询
    }
)
```

**关键设计：**
- 使用条件边实现质量评估和决策
- 使用普通边连接固定步骤
- 形成循环以支持迭代优化

**应用：** 这是生产级 RAG 系统的标准模式，结合了检索、评估、改写、生成的完整流程。

## 卡片10：总结与延伸

**一句话：** 掌握5种边类型和选择策略，就能设计出高效、可维护的 LangGraph 工作流。

**核心要点总结：**

1. **5种边类型**：
   - 普通边：固定路由
   - 条件边：动态决策
   - 等待边：并行同步
   - 序列边：简化线性
   - Send边：动态并行

2. **设计原则**：
   - 路由函数是纯函数
   - 使用类型注解提升可维护性
   - 避免边类型混用
   - 优先使用高级抽象（序列边、Send）

3. **常见误区**：
   - 条件边和普通边混用导致重复执行
   - Command 无法覆盖显式边
   - 路由函数接收的状态类型取决于 input_schema
   - defer 节点与条件边组合需谨慎

**延伸学习：**

1. **进阶主题**：
   - 边的内部实现（通道系统）
   - 边的编译过程
   - 边的可视化优化

2. **实战练习**：
   - 实现一个带循环的 RAG 工作流
   - 使用 Send 实现批量文档处理
   - 设计一个多代理协作系统

3. **最佳实践**：
   - 阅读 LangGraph 官方示例
   - 研究社区的生产级工作流

```
   └─ 否 → 条件边
```
