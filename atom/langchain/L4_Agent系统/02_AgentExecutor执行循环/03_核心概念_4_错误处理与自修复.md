# 核心概念 4：错误处理与自修复

> Agent 不怕犯错，怕的是犯了错不知道怎么修。AgentExecutor 的错误处理机制让 LLM 能"看到"自己的错误，然后自我纠正——这就是 AI Agent 的自修复能力。

---

## 为什么错误处理对 Agent 至关重要？

在 Agent 执行循环中，错误是常态而非异常：

```
Agent 执行中可能出现的错误：

1. LLM 输出格式错误（最常见）
   LLM 返回: "我觉得应该搜索一下"
   期望格式: Action: search\nAction Input: "关键词"
   → OutputParserException

2. 工具名称不存在
   LLM 请求: Action: google_search
   可用工具: [search, calculator]
   → InvalidTool

3. 工具执行失败
   工具: database_query
   错误: ConnectionRefusedError
   → ToolException

4. 工具输入格式错误
   工具期望: {"query": "...", "limit": 10}
   LLM 给的: "帮我查一下"
   → ValidationError
```

如果不处理这些错误，Agent 会直接崩溃。但更聪明的做法是：**把错误信息反馈给 LLM，让它自己修正。**

这就是 AgentExecutor 的自修复模式。

---

## OutputParserException：LLM 输出解析失败

### 什么时候会发生？

Agent 的工作流程是：LLM 生成文本 → OutputParser 解析出 AgentAction 或 AgentFinish。

当 LLM 的输出不符合预期格式时，OutputParser 会抛出 `OutputParserException`：

```python
# ReAct 格式期望的输出
"""
Thought: 我需要搜索这个问题
Action: search
Action Input: "Python 3.13 新特性"
"""

# LLM 实际可能输出的"坏格式"
"""
我觉得应该搜索一下 Python 3.13 的新特性，让我来查查看。
"""
# → 没有 Action/Action Input，解析器无法提取工具调用
# → 抛出 OutputParserException
```

常见的解析失败场景：

| 场景 | LLM 输出示例 | 原因 |
|------|-------------|------|
| 缺少 Action | "让我想想..." | LLM 没有按格式输出 |
| Action 格式错误 | "Action：search"（中文冒号） | 格式细节不对 |
| 多余内容 | "好的！Action: search\n顺便说一下..." | 输出了额外文本 |
| JSON 格式错误 | `{"action": "search", input: "..."}` | JSON 语法错误 |
| 幻觉工具名 | "Action: google_it" | 工具名不在可用列表中 |

## handle_parsing_errors：三种处理模式

### 源码定义（agent.py:1012）

```python
class AgentExecutor(Chain):
    handle_parsing_errors: bool | str | Callable[[OutputParserException], str] = False
```

三种模式对应三种处理策略：

### 模式一：`False`（默认）——不处理，直接崩溃

```python
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    handle_parsing_errors=False,  # 默认值
)

# LLM 输出格式错误时：
# → 直接抛出 OutputParserException
# → Agent 执行终止
# → 用户看到一个错误
```

适用场景：开发调试阶段，你需要看到原始错误来修复 Prompt。

### 模式二：`True`——自动修复

```python
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    handle_parsing_errors=True,
)

# LLM 输出格式错误时：
# → 捕获 OutputParserException
# → 把错误信息作为 observation 反馈给 LLM
# → LLM 在下一轮看到错误，尝试修正输出格式
```

这是最常用的模式。LLM 看到的反馈类似：

```
Observation: Invalid or incomplete response.
Output: 我觉得应该搜索一下 Python 3.13 的新特性
Error: Could not parse LLM output: `我觉得应该搜索一下 Python 3.13 的新特性`
```

LLM 收到这个反馈后，通常会在下一轮修正格式：

```
Thought: 我之前的输出格式不对，让我重新按正确格式输出
Action: search
Action Input: "Python 3.13 新特性"
```

### 模式三：`str`——自定义错误提示

```python
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    handle_parsing_errors="输出格式错误，请严格按照以下格式回复：\n"
                          "Thought: 你的思考\n"
                          "Action: 工具名\n"
                          "Action Input: 工具输入",
)

# LLM 输出格式错误时：
# → 捕获异常
# → 把你自定义的字符串作为 observation 反馈给 LLM
# → LLM 看到明确的格式指导
```

适用场景：当默认错误信息不够清晰，LLM 反复犯同样的格式错误时。

### 模式四：`Callable`——自定义处理函数

```python
def custom_error_handler(error: OutputParserException) -> str:
    """自定义错误处理：根据错误类型给出不同提示"""
    error_text = str(error)

    if "Could not parse LLM output" in error_text:
        return (
            "你的回复格式不正确。请使用以下格式：\n"
            "Thought: [你的思考过程]\n"
            "Action: [工具名称]\n"
            "Action Input: [工具输入]\n"
            "或者如果你已经有了最终答案：\n"
            "Thought: 我已经有了答案\n"
            "Final Answer: [最终答案]"
        )
    elif "action_input" in error_text:
        return "工具输入格式错误，请确保 Action Input 是有效的字符串或 JSON。"
    else:
        return f"发生了解析错误：{error_text}。请重新组织你的回复。"

executor = AgentExecutor(
    agent=agent,
    tools=tools,
    handle_parsing_errors=custom_error_handler,
)
```

适用场景：需要根据不同错误类型给出精确指导的生产环境。

## 源码解析：错误处理的完整流程

### _iter_next_step 中的错误处理（agent.py:1313）

```python
def _iter_next_step(
    self, name_to_tool_map, color_mapping,
    inputs, intermediate_steps, run_manager
):
    try:
        # ① 裁剪中间步骤
        intermediate_steps = self._prepare_intermediate_steps(intermediate_steps)

        # ② 调用 LLM 进行决策（可能抛出 OutputParserException）
        output = self.agent.plan(
            intermediate_steps,
            callbacks=run_manager.get_child() if run_manager else None,
            **inputs,
        )

    except OutputParserException as e:
        # ③ 捕获解析异常

        if isinstance(self.handle_parsing_errors, bool):
            if not self.handle_parsing_errors:
                raise e  # False：不处理，直接抛出
            # True：使用异常自带的 observation 或 str(e)
            observation = str(e.observation) if hasattr(e, "observation") else str(e)

        elif isinstance(self.handle_parsing_errors, str):
            # 字符串：使用自定义错误信息
            observation = self.handle_parsing_errors

        elif callable(self.handle_parsing_errors):
            # 函数：调用自定义处理函数
            observation = self.handle_parsing_errors(e)

        else:
            raise ValueError("handle_parsing_errors 类型不合法")

        # ④ 把错误包装成一个特殊的 AgentAction
        output = AgentAction(
            tool="_Exception",        # 特殊工具名
            tool_input=observation,    # 错误信息作为输入
            log=str(e),               # 原始错误日志
        )

        # ⑤ 使用 ExceptionTool 处理
        if isinstance(output, AgentAction):
            # ... 后续会执行 ExceptionTool
            observation = ExceptionTool().run(output.tool_input)
            yield AgentStep(action=output, observation=observation)
            return

    # 正常流程继续...
```

### 关键设计：错误变成 observation

这是整个自修复机制的核心思想：

```
正常流程：
  LLM → AgentAction(tool="search", input="...") → tool.run() → observation

错误流程：
  LLM → OutputParserException → AgentAction(tool="_Exception", input=错误信息)
      → ExceptionTool.run() → observation（= 错误信息）

两条路径最终都产出 observation，都会被追加到 intermediate_steps。
LLM 在下一轮决策时，能看到这个 observation（错误信息），从而修正行为。
```

## ExceptionTool：把错误变成工具调用

### 源码实现

```python
class ExceptionTool(BaseTool):
    """当 Agent 输出无法解析时使用的特殊工具"""
    name: str = "_Exception"
    description: str = "Exception tool"

    def _run(self, query: str, run_manager=None) -> str:
        # 直接返回错误信息，不做任何处理
        return query

    async def _arun(self, query: str, run_manager=None) -> str:
        return query
```

ExceptionTool 的实现极其简单——它只是把输入原样返回。但它的存在让错误处理流程与正常工具调用流程保持一致：

```
正常工具调用：
  AgentAction(tool="search") → search_tool.run() → "搜索结果..."

异常处理：
  AgentAction(tool="_Exception") → ExceptionTool.run() → "错误信息..."

对 intermediate_steps 来说，两者的结构完全一样：
  (AgentAction, observation_string)
```

这种设计的优雅之处在于：**LLM 不需要知道发生了错误，它只是看到了一个"工具返回了错误信息"的 observation，然后自然地调整策略。**

## InvalidTool：处理不存在的工具

### 什么时候触发？

当 LLM 请求调用一个不在可用工具列表中的工具时：

```python
# 可用工具
tools = [search_tool, calculator_tool]
# 工具名映射: {"search": search_tool, "calculator": calculator_tool}

# LLM 输出
# Action: google_search    ← 这个工具不存在！
# Action Input: "Python 教程"
```

### 源码处理（agent.py:1313 _iter_next_step 中）

```python
# 在 _iter_next_step 中，执行工具前的检查
for agent_action in actions:
    if agent_action.tool in name_to_tool_map:
        # 工具存在，正常执行
        tool = name_to_tool_map[agent_action.tool]
        observation = tool.run(agent_action.tool_input, ...)
    else:
        # 工具不存在，使用 InvalidTool
        observation = InvalidTool().run(
            requested_tool_name=agent_action.tool,
            available_tool_names=list(name_to_tool_map.keys()),
        )
    yield AgentStep(action=agent_action, observation=observation)
```

### InvalidTool 的实现

```python
class InvalidTool(BaseTool):
    """当 Agent 请求不存在的工具时返回错误信息"""
    name: str = "_Invalid_Tool"
    description: str = "Called when tool name is invalid."

    def _run(self, requested_tool_name, available_tool_names, **kwargs):
        return (
            f"{requested_tool_name} is not a valid tool, "
            f"try one of [{', '.join(available_tool_names)}]."
        )
```

LLM 看到的 observation：

```
google_search is not a valid tool, try one of [search, calculator].
```

LLM 在下一轮通常会修正：

```
Thought: google_search 不可用，我应该用 search 工具
Action: search
Action Input: "Python 教程"
```

## 工具执行错误的处理

除了 LLM 输出解析错误，工具本身执行也可能失败。

### ToolException

```python
from langchain_core.tools import ToolException

class DatabaseQueryTool(BaseTool):
    name = "db_query"
    description = "查询数据库"

    def _run(self, query: str) -> str:
        try:
            result = database.execute(query)
            return str(result)
        except ConnectionError:
            raise ToolException("数据库连接失败，请稍后重试")
        except Exception as e:
            raise ToolException(f"查询执行错误: {str(e)}")
```

### handle_tool_error

工具级别也有错误处理配置：

```python
from langchain_core.tools import BaseTool, ToolException

# 方式 1：工具自带错误处理
class SafeSearchTool(BaseTool):
    name = "search"
    description = "搜索工具"
    handle_tool_error = True  # 工具错误自动转为 observation

    def _run(self, query: str) -> str:
        # 如果抛出 ToolException，错误信息会作为 observation 返回
        raise ToolException("搜索服务暂时不可用")

# 方式 2：自定义错误处理函数
def handle_search_error(error: ToolException) -> str:
    return f"搜索失败了（{error}），请尝试换个关键词或使用其他工具。"

class SearchTool(BaseTool):
    name = "search"
    description = "搜索工具"
    handle_tool_error = handle_search_error
```

### 错误处理层级

```
错误处理的三道防线：

第一道：工具内部 try/except
  → 工具自己处理已知错误，返回友好信息

第二道：handle_tool_error（工具级别）
  → ToolException 被捕获，转为 observation

第三道：handle_parsing_errors（AgentExecutor 级别）
  → OutputParserException 被捕获，转为 observation

如果三道防线都没拦住：
  → 异常冒泡到调用方，Agent 执行终止
```

### 带监控的完整方案

```python
from langchain.callbacks.base import BaseCallbackHandler

class ErrorMonitor(BaseCallbackHandler):
    """监控 Agent 错误频率"""

    def __init__(self):
        self.error_count = 0
        self.total_steps = 0

    def on_agent_action(self, action, **kwargs):
        self.total_steps += 1
        if action.tool == "_Exception":
            self.error_count += 1
            logger.warning(
                f"Agent 解析错误 ({self.error_count}/{self.total_steps}): "
                f"{action.tool_input[:200]}"
            )

    def on_agent_finish(self, finish, **kwargs):
        error_rate = self.error_count / max(self.total_steps, 1)
        if error_rate > 0.3:
            logger.error(
                f"Agent 错误率过高: {error_rate:.0%} "
                f"({self.error_count}/{self.total_steps})"
            )

monitor = ErrorMonitor()
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    handle_parsing_errors=True,
    callbacks=[monitor],
    max_iterations=10,
)
```

## 常见错误模式与解决方案

### 模式 1：LLM 反复格式错误

```
症状：Agent 连续 3-4 轮都是 _Exception
原因：Prompt 中的格式说明不够清晰，或模型能力不足
```

```python
# 解决方案：使用更强的格式指导
def strict_format_handler(error):
    return (
        "FORMAT ERROR. You MUST respond in EXACTLY this format:\n"
        "Thought: <your reasoning>\n"
        "Action: <tool name from the list>\n"
        "Action Input: <input for the tool>\n"
        "DO NOT include any other text."
    )
```

### 模式 2：工具名幻觉

```
症状：LLM 反复请求不存在的工具
原因：Prompt 中的工具描述不够突出，或工具名不直观
```

```python
# 解决方案：改善工具命名和描述
from langchain.tools import Tool

# ❌ 不好的命名
bad_tool = Tool(name="t1", description="搜索", func=search_fn)

# ✅ 好的命名
good_tool = Tool(
    name="web_search",
    description="搜索互联网获取实时信息。输入应该是搜索关键词。",
    func=search_fn,
)
```

### 模式 3：错误-修复死循环

```
症状：Agent 在"错误 → 修复 → 又错误"之间循环
原因：LLM 的修复尝试本身也有格式问题
```

```python
# 解决方案：限制重试次数 + 自定义裁剪
def error_aware_trim(steps):
    """如果连续 3 次都是错误，只保留最后一次错误"""
    error_streak = 0
    for action, _ in reversed(steps):
        if action.tool == "_Exception":
            error_streak += 1
        else:
            break

    if error_streak >= 3:
        # 只保留最后一个错误步骤，避免 LLM 被大量错误信息淹没
        return [steps[-1]]
    return steps[-5:]

executor = AgentExecutor(
    agent=agent,
    tools=tools,
    handle_parsing_errors=True,
    trim_intermediate_steps=error_aware_trim,
    max_iterations=8,  # 严格限制，防止错误循环消耗太多
)
```

```python
        if not results:
            # 不抛异常，但给 LLM 明确的信号
            return "未找到相关文档。建议：1) 换个关键词重试 2) 尝试更宽泛的查询"
        return "\n\n".join([doc.page_content for doc in results])

# 2. Embedding 服务不可用
class RobustRetrieverTool(BaseTool):
    name = "retrieve"
    description = "从知识库检索相关文档"
    handle_tool_error = True  # 自动处理工具错误

    def _run(self, query: str) -> str:
        try:
            results = vector_store.similarity_search(query, k=3)
            return "\n\n".join([doc.page_content for doc in results])
        except Exception as e:
            raise ToolException(
                f"检索服务暂时不可用({type(e).__name__})，"
                "请尝试直接回答用户问题，或告知用户稍后重试。"
            )
```

**RAG Agent 的错误处理原则：**

1. 检索为空不是错误，但要给 LLM 明确的"换个方式试试"的信号
2. Embedding 服务故障要优雅降级，不能让整个 Agent 崩溃
3. 文档解析错误（PDF 损坏等）要跳过而非终止
4. 错误信息要对 LLM 友好——告诉它"发生了什么"和"可以怎么做"

---

## 小结

AgentExecutor 的错误处理机制构建了一个自修复闭环：

- `OutputParserException`：LLM 输出格式不对时触发
- `handle_parsing_errors`：三种模式（`bool` / `str` / `callable`），控制如何把错误反馈给 LLM
- `ExceptionTool`：把错误包装成标准的工具调用，保持 intermediate_steps 结构一致
- `InvalidTool`：处理不存在的工具名，告诉 LLM 可用的工具列表
- 自修复闭环：错误 → observation → LLM 看到错误 → 修正行为 → 重试

一句话：**错误不是终点，而是 Agent 学习的机会——handle_parsing_errors 把"崩溃"变成"反馈"，让 LLM 自己修正错误，这就是 AI Agent 的韧性所在。**
