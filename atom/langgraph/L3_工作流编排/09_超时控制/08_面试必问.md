# 面试必问

下面这些问题，不只是为了面试，更是为了逼你真正把“超时控制”想清楚。

---

## 1. LangGraph 的 `step_timeout` 控制的到底是什么？

**标准回答：**

`step_timeout` 控制的是一个 Pregel step 的最长等待时间，不是整个 run 的总时长，也不是单个 node 的内部 I/O 时间。

它会在执行器等待当前 step 内 future 完成时生效；如果 deadline 到了，还有 inflight 任务没结束，就会取消这些任务并抛超时异常。

**追问点：**
- 为什么不是 node timeout？
- 并发 step 里快节点和慢节点怎么互动？

## 2. `step_timeout` 和 `asyncio.wait_for()` 有什么区别？

**标准回答：**

`step_timeout` 是图调度层预算，面向整个 step；`asyncio.wait_for()` 是节点内部协程级预算，面向某个具体 awaitable。

最好的做法不是二选一，而是把它们叠起来：
- 节点内部先约束单次远程调用；
- 图外层再约束整个 step。

---

## 3. 为什么说“给图设了 timeout”不等于“系统就不会卡住”？

**标准回答：**

因为 timeout 是分层的。图可能有 timeout，但前端、SDK、平台、HTTP client、数据库驱动都可能有各自的等待窗口。

如果只配了一层，其他层仍然可能先出问题。

例如：
- 图还没 timeout；
- 前端已经 network error；
- 平台已经断开请求；
- 或者 HTTP 客户端还在底层读响应。

---

## 4. timeout 发生时，LangGraph 会怎么处理还没结束的任务？

**标准回答：**

运行时会取消 inflight tasks，然后抛出 `TimeoutError` 或 `asyncio.TimeoutError`。

这是 `_panic_or_proceed()` 明确实现的行为：
- 先识别 done / inflight；
- 如果仍有 inflight，则全部 cancel；
- 再 raise timeout exception。

---

## 5. `ParentCommand` 会不会被 timeout 吞掉？

**标准回答：**

不应该。源码测试明确验证了在 timeout 打开时，`ParentCommand` 仍然要被正确传播。

这说明 timeout 在 LangGraph 中是调度兜底，不应该吞掉更高语义层的控制信号。

---

## 6. 子图和父图都设置 `step_timeout` 时，怎么理解它们的关系？

**标准回答：**

它们代表不同层级的预算：
- 子图 timeout 是局部 step budget；
- 父图 timeout 是外层 orchestration budget。

在复杂工作流里，推荐让越靠近慢操作的层级越早超时，越外层的 timeout 作为更宽的兜底。

---

## 7. 为什么人类审批不应该靠把 timeout 调大来解决？

**标准回答：**

因为这类等待本质上不是“当前请求里的同步等待”，而是“流程暂停”。

LangGraph 官方更推荐：
- `interrupt`
- checkpoint
- resume

也就是保存状态、归还控制权、稍后继续，而不是让系统一直占着资源等用户回来。

## 8. 为什么 `astream()` 的消费端变慢也可能导致 timeout？

**标准回答：**

因为流式执行不是单向“后端推、前端收”这么简单。消费端慢会形成背压，而 step 的整体 deadline 还在继续倒计时。

异步测试已经证明，消费端在 `async for` 中慢处理 chunk 时，慢节点未完成，最终可能导致 `asyncio.TimeoutError`，并触发内部任务取消。

---

## 9. `RunStatus = timeout` 和 Python 抛 `TimeoutError` 是一回事吗？

**标准回答：**

不是同一层。

- Python 抛 `TimeoutError`：更接近当前进程内的执行异常；
- `RunStatus = timeout`：是平台 / server 侧把运行结果提升成显式状态。

前者偏运行时细节，后者偏产品化接口语义。

---

## 10. 生产环境里 timeout 应该怎么分层配置？

**标准回答：**

一个推荐顺序是：

1. 单次 I/O timeout（HTTP / DB / SDK）
2. 节点内部协程 timeout（`wait_for`）
3. 图 step timeout（`step_timeout`）
4. retry / max_attempts / recursion limit
5. SDK 请求 timeout
6. 平台 / gateway / 前端等待窗口

每一层都要有清晰责任。

---

## 11. timeout 和 retry 的关系是什么？

**标准回答：**

timeout 负责“最多等多久”，retry 负责“是否再试一次”。

两者必须一起设计：
- timeout 没 retry → 瞬态故障恢复能力弱；
- retry 没 timeout → 可能越试越久。

---

## 12. 如果前端经常 timeout，但后端最终能成功，应该优先改哪里？

**标准回答：**

优先改交互模型，而不是只改图 timeout。

常见手段：
- 改成后台任务；
- run status 轮询；
- 中间状态流式回传；
- checkpoint 后 resume。

因为问题大概率在同步交互窗口，而不是图逻辑本身。

---

## 13. 什么时候应该失败得更快，而不是等更久？

**标准回答：**

当操作满足以下任一条件时：
- 依赖服务明显不稳定；
- 用户感知延迟很敏感；
- 有明确 fallback；
- 重试收益低，资源消耗高；
- 平台窗口本来就很短。

快失败不是悲观，而是为了更快进入可恢复路径。

---

## 14. timeout 指标应该怎么监控？

**标准回答：**

至少分层监控：
- node timeout 次数；
- graph step timeout 次数；
- SDK timeout 次数；
- platform timeout / gateway 断连次数；
- timeout 后 fallback / retry 的命中率。

否则你只能知道“有人超时了”，不知道超时发生在哪一层。

---

## 15. 最后一句话，如何定义一个“设计良好的 timeout 系统”？

**标准回答：**

一个设计良好的 timeout 系统，不是只有一个全局数字，而是：

- 有分层预算；
- 有可解释边界；
- 有超时后的恢复策略；
- 有观测手段；
- 有对长任务的异步化方案。

---

## 面试收束

如果面试里只能说一句最值钱的话，我会说：

**LangGraph 的超时控制重点不在参数值，而在“你是否把等待拆成了不同层级，并为每一层设计了预算与恢复路径”。**
