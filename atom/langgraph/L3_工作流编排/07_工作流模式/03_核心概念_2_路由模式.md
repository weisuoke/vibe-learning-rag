# 07_工作流模式 - 核心概念2：路由模式（Routing）

> LangGraph L3_工作流编排 | 核心概念 2/6

---

## 概念定义

**路由模式是根据输入特征动态选择执行路径的工作流模式，让 LLM 充当"智能分拣员"。**

想象一个医院的分诊台：护士根据患者症状把人分流到不同科室——发烧去内科、骨折去骨科、皮疹去皮肤科。路由模式做的就是同样的事：一个 Router 节点分析输入，然后把请求派发到最合适的处理节点。

```
START → Router → [Handler_A | Handler_B | Handler_C] → END
```

与顺序链式模式的"一条路走到底"不同，路由模式在运行时才决定走哪条路。

---

## 模式原理

### 核心结构

```
                    ┌→ Handler_A → END
START → Router ─────┼→ Handler_B → END
                    └→ Handler_C → END
```

Router 节点是整个模式的核心。它接收输入，做出分类决策，然后通过 `add_conditional_edges` 把请求路由到对应的 Handler。

### 三个核心特征

| 特征 | 说明 |
|------|------|
| 动态路径选择 | 运行时根据输入内容决定走哪条路，而非编译时固定 |
| LLM 驱动路由 | 可以用 structured output 让 LLM 做分类决策 |
| 多路分支 | 支持任意数量的目标节点，每个分支处理不同类型的请求 |

### 与顺序链式模式的关键区别

顺序链式模式中的 Gate 是"继续还是终止"的二选一；路由模式是"去 A、去 B、还是去 C"的多选一。Gate 是红绿灯，路由是立交桥匝道。

---

## LangGraph 三种实现方式

### 方式 1：基于函数的路由

**最直接的方式，用 Python 函数做分类判断，适合规则明确的场景。**

```python
"""
路由模式 - 基于函数的路由
场景：根据用户查询类型路由到不同处理器

图拓扑：
    START → classify → [factual_handler | analytical_handler | creative_handler] → END
"""
from typing import Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class QueryState(TypedDict):
    query: str           # 用户查询
    query_type: str      # 查询类型
    response: str        # 最终回答


def classify(state: QueryState) -> dict:
    """分类节点：分析查询内容，打上类型标签"""
    query = state["query"].lower()
    # 简单的关键词规则分类
    if any(kw in query for kw in ["是什么", "定义", "多少", "哪年"]):
        query_type = "factual"
    elif any(kw in query for kw in ["为什么", "分析", "比较", "影响"]):
        query_type = "analytical"
    else:
        query_type = "creative"
    print(f"[classify] 查询类型: {query_type}")
    return {"query_type": query_type}


# --- 路由函数：根据 State 中的 query_type 决定下一步 ---
def route_query(state: QueryState) -> Literal[
    "factual_handler", "analytical_handler", "creative_handler"
]:
    """路由函数：返回目标节点名称"""
    return f"{state['query_type']}_handler"


def factual_handler(state: QueryState) -> dict:
    """事实型查询：直接检索知识库"""
    print("[factual] 执行精确检索...")
    return {"response": f"[事实回答] 针对「{state['query']}」的精确检索结果"}


def analytical_handler(state: QueryState) -> dict:
    """分析型查询：多步推理"""
    print("[analytical] 执行多步推理...")
    return {"response": f"[分析回答] 针对「{state['query']}」的深度分析"}


def creative_handler(state: QueryState) -> dict:
    """创意型查询：开放生成"""
    print("[creative] 执行创意生成...")
    return {"response": f"[创意回答] 针对「{state['query']}」的创意内容"}


# --- 构建图 ---
builder = StateGraph(QueryState)

# 添加节点
builder.add_node("classify", classify)
builder.add_node("factual_handler", factual_handler)
builder.add_node("analytical_handler", analytical_handler)
builder.add_node("creative_handler", creative_handler)

# 添加边
builder.add_edge(START, "classify")
# 核心：条件边实现路由
builder.add_conditional_edges("classify", route_query)
# 所有 handler 都指向 END
builder.add_edge("factual_handler", END)
builder.add_edge("analytical_handler", END)
builder.add_edge("creative_handler", END)

graph = builder.compile()

# 测试不同类型的查询
for q in ["Python 是什么", "为什么要用 RAG", "帮我写首诗"]:
    result = graph.invoke({"query": q, "query_type": "", "response": ""})
    print(f"  → {result['response']}\n")
# [classify] 查询类型: factual
# [factual] 执行精确检索...
#   → [事实回答] 针对「Python 是什么」的精确检索结果
#
# [classify] 查询类型: analytical
# [analytical] 执行多步推理...
#   → [分析回答] 针对「为什么要用 RAG」的深度分析
#
# [classify] 查询类型: creative
# [creative] 执行创意生成...
#   → [创意回答] 针对「帮我写首诗」的创意内容
```

#### 源码层面发生了什么

`add_conditional_edges` 在编译时创建 `BranchSpec`，运行时调用路由函数获取目标节点名：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/_branch.py]
class BranchSpec(NamedTuple):
    path: Runnable          # 路由函数（如 route_query）
    ends: dict | None       # 目标映射（如 {"factual": "factual_handler", ...}）
    input_schema: type | None
```

执行流程：classify 完成 → 调用 `route_query(state)` → 返回 `"factual_handler"` → 写入 `branch:to:factual_handler` channel → factual_handler 被触发。

### 方式 2：LLM Structured Output 路由

**最智能的方式，用 LLM 做分类决策，适合规则模糊、需要语义理解的场景。**

这是路由模式最强大的用法。不再用 if/elif 硬编码规则，而是让 LLM 理解用户意图后做出路由决策。

```python
"""
路由模式 - LLM Structured Output 路由
场景：智能客服分流系统（问答 / 投诉 / 通用咨询）

图拓扑：
    START → llm_router → [qa_handler | complaint_handler | general_handler] → END
"""
from typing import Literal
from typing_extensions import TypedDict
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph, START, END

# 如果要实际运行，取消下面的注释并配置 API Key
# from langchain_openai import ChatOpenAI
# llm = ChatOpenAI(model="gpt-4o-mini")


# --- Pydantic 模型定义路由决策的结构 ---
class RouteDecision(BaseModel):
    """LLM 的路由决策输出"""
    route: Literal["qa", "complaint", "general"] = Field(
        description="用户意图分类：qa=知识问答, complaint=投诉反馈, general=通用咨询"
    )
    confidence: float = Field(
        description="分类置信度，0-1 之间",
        ge=0.0, le=1.0
    )
    reasoning: str = Field(
        description="分类理由，简要说明为什么这样分类"
    )


class CustomerState(TypedDict):
    message: str          # 用户消息
    route: str            # 路由决策
    confidence: float     # 置信度
    response: str         # 最终回复


def llm_router(state: CustomerState) -> dict:
    """LLM 路由节点：用 structured output 做智能分类"""
    message = state["message"]

    # --- 实际项目中使用 LLM ---
    # structured_llm = llm.with_structured_output(RouteDecision)
    # decision = structured_llm.invoke(
    #     f"请分析以下客户消息的意图：\n\n{message}"
    # )

    # --- 模拟 LLM 输出（演示用） ---
    if "退款" in message or "投诉" in message or "差评" in message:
        decision = RouteDecision(
            route="complaint", confidence=0.92, reasoning="包含投诉相关关键词"
        )
    elif "怎么" in message or "如何" in message or "什么" in message:
        decision = RouteDecision(
            route="qa", confidence=0.88, reasoning="包含疑问词，属于知识问答"
        )
    else:
        decision = RouteDecision(
            route="general", confidence=0.75, reasoning="无法明确分类，归入通用咨询"
        )

    print(f"[router] 路由决策: {decision.route} (置信度: {decision.confidence})")
    print(f"         理由: {decision.reasoning}")
    return {
        "route": decision.route,
        "confidence": decision.confidence,
    }


def route_by_decision(state: CustomerState) -> Literal[
    "qa_handler", "complaint_handler", "general_handler"
]:
    """路由函数：根据 LLM 的决策选择目标节点"""
    return f"{state['route']}_handler"


def qa_handler(state: CustomerState) -> dict:
    """知识问答处理器：检索知识库回答"""
    print("[qa] 检索知识库...")
    return {"response": f"[知识问答] 关于「{state['message']}」的回答：..."}


def complaint_handler(state: CustomerState) -> dict:
    """投诉处理器：记录工单 + 安抚"""
    print("[complaint] 创建投诉工单...")
    return {"response": f"[投诉处理] 已记录您的反馈，客服经理将在24小时内联系您。"}


def general_handler(state: CustomerState) -> dict:
    """通用咨询处理器"""
    print("[general] 通用回复...")
    return {"response": f"[通用咨询] 感谢您的咨询，以下是相关信息：..."}


# --- 构建图 ---
builder = StateGraph(CustomerState)

builder.add_node("llm_router", llm_router)
builder.add_node("qa_handler", qa_handler)
builder.add_node("complaint_handler", complaint_handler)
builder.add_node("general_handler", general_handler)

builder.add_edge(START, "llm_router")
builder.add_conditional_edges("llm_router", route_by_decision)
builder.add_edge("qa_handler", END)
builder.add_edge("complaint_handler", END)
builder.add_edge("general_handler", END)

graph = builder.compile()

# 测试
for msg in ["怎么重置密码？", "我要投诉快递太慢！", "你好，想了解一下会员权益"]:
    result = graph.invoke({
        "message": msg, "route": "", "confidence": 0.0, "response": ""
    })
    print(f"  → {result['response']}\n")
# [router] 路由决策: qa (置信度: 0.88)
#          理由: 包含疑问词，属于知识问答
# [qa] 检索知识库...
#   → [知识问答] 关于「怎么重置密码？」的回答：...
#
# [router] 路由决策: complaint (置信度: 0.92)
#          理由: 包含投诉相关关键词
# [complaint] 创建投诉工单...
#   → [投诉处理] 已记录您的反馈，客服经理将在24小时内联系您。
#
# [router] 路由决策: general (置信度: 0.75)
#          理由: 无法明确分类，归入通用咨询
# [general] 通用回复...
#   → [通用咨询] 感谢您的咨询，以下是相关信息：...
```

#### `with_structured_output` 的关键作用

```python
# 让 LLM 返回结构化的 Pydantic 对象，而非自由文本
structured_llm = llm.with_structured_output(RouteDecision)
decision = structured_llm.invoke("分析用户意图：...")
# decision.route → "qa" | "complaint" | "general"
# decision.confidence → 0.92
# decision.reasoning → "包含投诉关键词"
```

这比让 LLM 返回自由文本再解析可靠得多。Pydantic 模型保证了输出格式，`Literal` 类型约束了路由只能是预定义的几个值。

### 方式 3：Functional API 路由

**最 Pythonic 的方式，用 if/elif 控制流实现路由，代码读起来就像普通 Python 函数。**

```python
"""
路由模式 - Functional API 实现
场景：RAG 查询路由（根据问题类型选择不同检索策略）
"""
from langgraph.func import entrypoint, task


@task
def classify_query(query: str) -> str:
    """分类用户查询类型"""
    query_lower = query.lower()
    if any(kw in query_lower for kw in ["是什么", "定义", "列举"]):
        return "factual"
    elif any(kw in query_lower for kw in ["为什么", "原因", "分析"]):
        return "analytical"
    else:
        return "creative"


@task
def factual_search(query: str) -> str:
    """事实型：精确关键词检索"""
    print(f"[factual] 精确检索: {query}")
    return f"精确检索结果：{query} 的定义是..."


@task
def analytical_search(query: str) -> str:
    """分析型：多轮推理检索"""
    print(f"[analytical] 多步推理: {query}")
    return f"深度分析结果：{query} 的原因包括..."


@task
def creative_generate(query: str) -> str:
    """创意型：开放式生成"""
    print(f"[creative] 创意生成: {query}")
    return f"创意内容：关于 {query} 的全新视角..."


@entrypoint()
def rag_router(query: str) -> str:
    """RAG 路由入口：分类 → 路由到对应处理器"""
    query_type = classify_query(query).result()

    # 用普通 Python if/elif 实现路由，简洁直观
    if query_type == "factual":
        return factual_search(query).result()
    elif query_type == "analytical":
        return analytical_search(query).result()
    else:
        return creative_generate(query).result()


# 测试
result = rag_router.invoke("Transformer 是什么")
print(f"结果: {result}")
# [factual] 精确检索: Transformer 是什么
# 结果: 精确检索结果：Transformer 是什么 的定义是...
```

#### Functional API 路由的特点

Functional API 的路由不需要 `add_conditional_edges`，直接用 Python 的 `if/elif` 就行。LangGraph 会根据实际执行路径自动构建图结构。代码更直观，但失去了图的静态可视化能力（因为路径是动态的）。

---

## 三种方式对比

| 维度 | 函数路由 | LLM Structured Output | Functional API |
|------|----------|----------------------|----------------|
| 路由智能度 | 低（硬编码规则） | 高（LLM 语义理解） | 低-高（取决于实现） |
| 代码量 | 中 | 多（需定义 Pydantic 模型） | 少 |
| 可靠性 | 高（确定性规则） | 中（LLM 可能误判） | 高 |
| 延迟 | 低（无 LLM 调用） | 高（额外一次 LLM 调用） | 低 |
| 可视化 | 支持 | 支持 | 部分支持 |
| 适用场景 | 规则明确的分类 | 需要语义理解的分类 | 快速原型 |

**选择建议**：
- 分类规则可以用关键词/正则表达 → 函数路由
- 分类需要理解语义、意图模糊 → LLM Structured Output
- 快速验证想法、不需要可视化 → Functional API

## 与 06_条件分支策略 的区别

条件分支策略（06）和路由模式（本文）都用到了 `add_conditional_edges`，但关注点不同：

| 维度 | 06_条件分支策略 | 本文：路由模式 |
|------|----------------|---------------|
| 关注点 | **机制**：条件边怎么实现的 | **模式**：什么时候用路由、为什么用 |
| 核心问题 | `add_conditional_edges` 的 API 用法 | 如何设计一个好的路由系统 |
| 内容侧重 | path_map、Literal 类型、END 处理 | LLM 路由、Pydantic 模型、场景选型 |
| 类比 | 铁路道岔的机械结构 | 医院分诊台的工作流程 |

简单说：06 讲的是"条件边这个零件怎么用"，本文讲的是"用这个零件搭建路由系统的设计模式"。

## 适用场景

| 场景 | Router 逻辑 | Handler 节点 |
|------|-------------|-------------|
| 智能客服分流 | LLM 分析用户意图 | 问答、投诉、咨询、转人工 |
| 多模型选择 | 根据任务复杂度选模型 | GPT-4o（复杂）、GPT-4o-mini（简单） |
| RAG 查询分类 | 分析问题类型 | 精确检索、多步推理、开放生成 |
| 多语言处理 | 检测输入语言 | 中文处理器、英文处理器、翻译器 |
| 内容审核 | 检测内容类型 | 文本审核、图片审核、视频审核 |

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| 关注点分离：每个 Handler 只处理一类请求 | Router 是单点故障，路由错误影响全局 |
| 易于扩展：新增类型只需加 Handler + 更新路由 | LLM 路由增加一次额外调用的延迟和成本 |
| 资源高效：只执行匹配的分支，不浪费算力 | 路由边界模糊时分类准确率下降 |
| 可观测性好：路由决策可记录、可分析 | 路由类别过多时维护成本上升 |

### 路由失败的兜底策略

```python
def route_with_fallback(state) -> str:
    """带兜底的路由函数"""
    route = state["route"]
    confidence = state["confidence"]
    # 置信度太低时走兜底路径
    if confidence < 0.6:
        print(f"[fallback] 置信度 {confidence} 过低，走通用处理")
        return "general_handler"
    return f"{route}_handler"
```

当 LLM 路由的置信度不够时，应该有一个 fallback 路径，避免把请求送到错误的 Handler。

## 在 RAG 中的应用

RAG 系统中，路由模式最典型的应用是**查询分类路由**：

```
用户查询 → 查询分类器 → ┬→ 事实型 → 直接向量检索 → 生成
                        ├→ 分析型 → 多步检索 + 推理链 → 生成
                        └→ 对话型 → 历史上下文 + 检索 → 生成
```

不同类型的查询需要不同的检索策略：
- **事实型**（"Python 发布于哪年"）：单次精确检索就够了
- **分析型**（"为什么 RAG 比微调更适合知识密集型任务"）：需要检索多篇文档 + 多步推理
- **对话型**（"你刚才说的那个方法具体怎么用"）：需要结合对话历史做检索

用路由模式可以为每种查询类型定制最优的处理管道，而不是用一个通用管道处理所有查询。这就像医院不会让所有患者都走同一个科室一样。

---

## 最佳实践

### 1. 路由类别不要太多

```python
# 推荐：3-5 个类别，边界清晰
route: Literal["qa", "complaint", "general"]

# 不推荐：类别太多，LLM 容易混淆
route: Literal["qa", "complaint", "refund", "shipping",
               "account", "billing", "technical", "feedback"]
# 类别多时，考虑用两级路由（先粗分再细分）
```

### 2. 路由决策要带置信度

```python
class RouteDecision(BaseModel):
    route: Literal["qa", "complaint", "general"]
    confidence: float  # 关键：有了置信度才能做兜底
```

### 3. 所有 Handler 输出格式统一

```python
# 所有 Handler 都返回相同的 State 字段
def qa_handler(state) -> dict:
    return {"response": "..."}      # 统一返回 response

def complaint_handler(state) -> dict:
    return {"response": "..."}      # 统一返回 response
```

这样无论走哪条路径，下游节点都能用同样的方式读取结果。

### 4. 记录路由决策用于分析

把每次路由的决策（类型、置信度、理由）记录到 State 中，方便后续分析路由准确率和优化分类策略。

---

## 常见误区

### 误区 1："路由模式只能用 LLM 做分类"

不是。简单场景用关键词匹配、正则表达式、甚至消息长度判断都可以。LLM 路由适合需要语义理解的复杂场景，但它增加了延迟和成本。

### 误区 2："路由类别越细越好"

类别太细会导致 LLM 分类准确率下降，边界模糊的请求容易被误分。建议先用 3-5 个粗粒度类别，必要时再用两级路由细分。

### 误区 3："路由后只能走一条路"

`add_conditional_edges` 的路由函数可以返回列表，实现一对多路由（同时触发多个 Handler）。但这已经进入并行模式的范畴了。
