> **维度**: 核心概念（Cargo 依赖声明、版本规范、来源控制与 features 机制）

## 一句话定义

> **依赖管理是 Cargo 的核心功能，通过 Cargo.toml 声明和 Cargo.lock 锁定来确保可重复的构建——类似 package.json + package-lock.json 但更严格。**

---

## 一、三种依赖类型

Rust 将依赖按**用途**分成三类，和 npm 的分类几乎一一对应 [^context7_cargo1] [^search1]：

### 1.1 `[dependencies]` — 运行时依赖

你的程序运行时真正需要的库。编译产物中会包含这些代码。

```toml
[dependencies]
tokio = { version = "1.42", features = ["rt-multi-thread", "macros"] }
serde = { version = "1.0", features = ["derive"] }
reqwest = { version = "0.12", default-features = false, features = ["rustls-tls", "json"] }
```

### 1.2 `[dev-dependencies]` — 开发/测试依赖

**仅在 `cargo test` 和 `cargo bench` 时编译**，不会进入最终产物。

```toml
[dev-dependencies]
pretty_assertions = "1.4"       # 美化测试断言输出
tempfile = "3.10"                # 临时文件（测试用）
tokio = { version = "1.42", features = ["test-util"] }  # 测试工具
```

### 1.3 `[build-dependencies]` — 构建脚本依赖

**仅在 `build.rs` 构建脚本中使用**——`build.rs` 在编译你的代码**之前**执行，用于代码生成、C 绑定等。npm 没有直接对应物。

```toml
[build-dependencies]
cc = "1.0"                       # 编译 C/C++ 代码
prost-build = "0.13"             # 从 .proto 文件生成 Rust 代码
```

### 1.4 三种类型对照表

```
类型                    npm 对应物              何时编译         进入最终产物？
──────────────────────────────────────────────────────────────────────────────
[dependencies]          dependencies            始终              ✅ 是
[dev-dependencies]      devDependencies         cargo test 时     ❌ 否
[build-dependencies]    （无直接对应）           cargo build 前    ❌ 否（仅构建脚本用）
```

**关键区别**：npm 的 `devDependencies` 主要影响 `npm install --production` 是否安装。而 Rust 的 `[dev-dependencies]` 更严格——编译器根本不会把它们编译进 release 产物 [^context7_cargo1]。

## 二、版本规范详解

Cargo 使用语义化版本（SemVer），语法和 npm 相似但有微妙差异 [^context7_cargo1]。

### 2.1 各种版本写法

```toml
[dependencies]
# 1. 精确版本——只接受这个版本
exact_ver = "=1.2.3"            # npm: "1.2.3"（npm 也用 = 前缀表示精确）

# 2. 兼容版本（^ 是默认行为）——允许非 breaking 更新
compat_ver = "1.2.3"            # 等同于 ^1.2.3 → ≥1.2.3, <2.0.0
compat_ver2 = "^1.2.3"          # 显式写法，效果相同

# 3. 最小版本（~）——只允许 patch 更新
tilde_ver = "~1.2.3"            # ≥1.2.3, <1.3.0（npm 的 ~ 含义相同！）

# 4. 通配符（*）——匹配某位置上的任意数字
wildcard_ver = "1.2.*"          # ≥1.2.0, <1.3.0
wildcard_ver2 = "1.*"           # ≥1.0.0, <2.0.0

# 5. 范围比较——灵活组合
range_ver = ">=1.0, <2.0"       # 和 ^1.0 等效
range_ver2 = ">=0.5.0, <=0.8.0" # 限制在某个范围
```

### 2.2 版本规范对照表

```
Cargo 写法         含义                         npm 等价写法
─────────────────────────────────────────────────────────────
"1.2.3"            ^1.2.3 (≥1.2.3, <2.0.0)     "^1.2.3"
"^1.2.3"           同上（显式写法）               "^1.2.3"
"~1.2.3"           ≥1.2.3, <1.3.0               "~1.2.3"
"=1.2.3"           精确匹配                       "1.2.3"
"1.2.*"            ≥1.2.0, <1.3.0               "~1.2.0"
"*"                任意版本                       "*"
">=1, <2"          范围表达式                     ">=1.0.0 <2.0.0"
```

### 2.3 ⚠️ 0.x 版本的特殊规则

这是 Cargo 最容易让人困惑的地方：

```toml
# 1.x 版本——major 是 breaking change
serde = "1.0.195"           # ≥1.0.195, <2.0.0 ✅ 符合直觉

# 0.x 版本——MINOR 被视为 breaking change！
rand = "0.8.5"              # ≥0.8.5, <0.9.0  ← 注意上界是 0.9，不是 1.0！
```

**规则**：在 `^` 语义下，最左边的非零数字被视为 major 版本。0.x 表示 API 不稳定，所以 0.8 → 0.9 可能是 breaking。这和 npm 的行为**一致** [^context7_cargo1]。

## 三、依赖来源

依赖不一定来自 crates.io。Cargo 支持三种来源 [^context7_cargo1] [^search1]：

### 3.1 crates.io（默认仓库）

```toml
[dependencies]
serde = "1.0"                   # 从 crates.io 下载（默认行为）
```

crates.io 之于 Rust，就像 npmjs.com 之于 Node.js。但有一个重要区别——**crate 名字一旦发布不可删除**（npm 则有 72 小时窗口可 unpublish）。

### 3.2 Git 仓库依赖

```toml
[dependencies]
# 使用 Git 仓库（默认拉 default branch 最新提交）
my-lib = { git = "https://github.com/user/my-lib" }

# 指定分支
my-lib = { git = "https://github.com/user/my-lib", branch = "next" }

# 指定 tag
my-lib = { git = "https://github.com/user/my-lib", tag = "v1.0.0" }

# 指定 commit（最精确）
my-lib = { git = "https://github.com/user/my-lib", rev = "abc1234" }
```

**何时用 Git 依赖？**
- 依赖还没发布到 crates.io
- 需要上游的最新 fix（还没发版）
- Fork 了一个库做了修改

**TypeScript 类比**：npm 也支持 `"my-lib": "github:user/my-lib#branch"`，但 Cargo 的 Git 依赖在社区中更常见。

### 3.3 本地路径依赖

```toml
[dependencies]
robot-kit = { path = "crates/robot-kit" }    # 相对于 Cargo.toml 的路径
```

**这是 workspace 内最常见的依赖方式**——同一仓库内的 crate 互相引用。

**ZeroClaw 示例**：主 crate 依赖 robot-kit 子 crate [^source1]：

```toml
# zeroclaw/Cargo.toml
[dependencies]
robot-kit = { path = "crates/robot-kit" }

# 使用时就像普通依赖一样
# use robot_kit::SafetyMonitor;
```

**TypeScript 类比**：类似 pnpm workspace 中 `"@zeroclaw/robot-kit": "workspace:*"` 或 TypeScript 的 `paths` 映射。

### 3.4 三种来源对比

```
来源            语法                                  类比                 适用场景
─────────────────────────────────────────────────────────────────────────────────────
crates.io      version = "1.0"                       npm install pkg     ✅ 生产首选
Git 仓库       git = "https://..."                   npm install github: 上游未发版
本地路径       path = "crates/xxx"                   workspace:*         monorepo 内部
```

## 四、Cargo.lock 的角色

### 4.1 为什么需要 lock 文件

Cargo.toml 只记录**版本范围**，比如 `serde = "1.0"`。今天解析可能是 1.0.217，下个月可能是 1.0.230。Cargo.lock 记录的是**精确版本 + 哈希校验**，确保团队所有人、CI 环境都用完全相同的版本 [^context7_cargo1]。

```
Cargo.toml（你的意图）         Cargo.lock（实际快照）
──────────────────────         ──────────────────────
serde = "1.0"                  serde 1.0.217 (checksum: abc123...)
tokio = "1.42"                 tokio 1.42.0 (checksum: def456...)
                               tokio-macros 2.4.0 (传递依赖)
                               proc-macro2 1.0.86 (传递依赖)
```

### 4.2 何时提交 Cargo.lock

这个规则和 npm 生态**完全一致** [^context7_cargo1] [^search1]：

```
项目类型        提交 Cargo.lock？     为什么？
─────────────────────────────────────────────────────────────────────
应用程序        ✅ 必须提交            确保所有人构建出一模一样的产物
库（library）   ❌ 不提交（.gitignore）  让下游应用决定具体版本
```

**类比**：
- 应用 → 你做一道菜，食谱写死用"某品牌 500ml 酱油" → `package-lock.json` 提交
- 库 → 你写菜谱给别人，只说"适量酱油" → 让用你菜谱的人自己选品牌

### 4.3 cargo update 命令

```bash
cargo update                      # 更新所有依赖到 Cargo.toml 允许的最新版本
cargo update serde                # 只更新 serde（及其传递依赖）
cargo update serde --precise 1.0.210  # 更新 serde 到指定版本
```

**和 npm 对比**：

```bash
# Cargo                          # npm
cargo update                     npm update
cargo update serde               npm update serde
# Cargo.lock 自动更新             # package-lock.json 自动更新
```

## 五、依赖的 Features 控制

Features 是 Rust 依赖管理中最独特的部分——**编译时的功能开关**，让你只编译需要的代码。npm 没有对应概念 [^context7_cargo1] [^source1]。

### 5.1 启用特定 features

```toml
[dependencies]
# 启用 serde 的 derive feature（派生宏）
serde = { version = "1.0", features = ["derive"] }

# 启用 tokio 的多线程运行时和宏
tokio = { version = "1.42", features = ["rt-multi-thread", "macros", "fs", "io-util"] }

# 一个简写：如果只需要 version，可以用字符串
rand = "0.8"   # 等同于 { version = "0.8" }，使用默认 features
```

### 5.2 禁用默认 features

许多 crate 有 `default` features 会拉入额外的依赖。你可以关掉它们：

```toml
[dependencies]
# reqwest 默认使用系统 OpenSSL，我们关掉默认特性改用 rustls
reqwest = { version = "0.12", default-features = false, features = ["rustls-tls", "json"] }
```

**为什么要禁用默认 features？**

```
场景                              原因
─────────────────────────────────────────────────────────────────────
交叉编译（如 ARM Linux）           默认依赖可能不支持目标平台
嵌入式/WebAssembly                 需要 no_std 环境
减少编译时间                       少编译 = 更快
减小二进制体积                     不用的功能不编译进去
```

**ZeroClaw 的真实例子**：reqwest 默认使用系统 OpenSSL，但 ZeroClaw 部署到树莓派时，交叉编译 OpenSSL 很痛苦。改用 `rustls-tls`（纯 Rust TLS 实现）就没有 C 依赖了 [^source1]。

### 5.3 features 语法总结

```toml
# 完整语法
[dependencies]
my-crate = {
    version = "1.0",                    # 版本约束
    features = ["feat-a", "feat-b"],    # 启用的 features
    default-features = false,           # 是否使用默认 features
    optional = true,                    # 是否是可选依赖（仅当 feature 启用时编译）
}
```

### 5.4 命令行控制 features

```bash
cargo build                                       # 仅 default features
cargo build --features "hardware,channel-matrix"   # 启用指定 features
cargo build --all-features                         # 启用所有 features
cargo build --no-default-features                  # 禁用默认 features
cargo build --no-default-features --features "minimal"  # 组合使用
```

## 六、ZeroClaw 依赖分析

分析 ZeroClaw 真实项目的依赖选型和组织方式 [^source1]。

### 6.1 核心依赖分类

```
分类              依赖名                   用途                  为什么选它
──────────────────────────────────────────────────────────────────────────────
异步运行时        tokio                    异步 I/O + 多线程      Rust 异步事实标准
序列化           serde + serde_json        JSON/TOML 序列化       生态最广，derive 宏极方便
HTTP 客户端      reqwest                   API 调用              最流行的 HTTP 客户端
CLI 解析         clap                      命令行参数             derive 宏，自动生成 help
日志与追踪       tracing                   结构化日志             异步友好，比 log 更强大
错误处理         anyhow / thiserror        错误类型               anyhow 做应用，thiserror 做库
```

### 6.2 可选依赖（feature 门控）

```toml
# 仅当 hardware feature 启用时才编译
nusb = { version = "0.1", optional = true }            # USB 设备枚举
tokio-serial = { version = "5.4", optional = true }    # 串口通信

# 仅当 channel-matrix feature 启用时
matrix-sdk = { version = "0.7", optional = true }      # Matrix 协议

# 仅当 observability-otel feature 启用时
opentelemetry = { version = "0.25", optional = true }  # OpenTelemetry
```

**这就是 features 的真正价值**——ZeroClaw 支持 15+ 功能模块，但最小构建只编译核心部分。用户根据部署场景按需启用 [^source1]：

```bash
# 树莓派机器人——需要硬件 + 安全
cargo build --release --features "hardware,peripheral-rpi,sandbox-landlock"

# 云服务器聊天机器人——只需要通道
cargo build --release --features "channel-matrix,channel-lark"

# 开发调试——全部功能
cargo build --all-features
```

### 6.3 依赖数量与编译时间的关系

```
依赖情况                        首次编译时间（估算）    增量编译
──────────────────────────────────────────────────────────────
最小构建（default = []）         ~2 分钟               ~3 秒
部分 features                   ~4 分钟               ~5 秒
--all-features                  ~8 分钟               ~8 秒
```

**为什么这么重要？** Rust 编译是**编译时做更多事（类型检查、借用检查、单态化、优化）**换取运行时零开销。dependencies 越多 → 编译的代码越多 → 时间越长 [^search1]。

**优化建议**：
1. **用 `optional = true`** 把非必需依赖变可选——不启用就不编译
2. **拆分 crate** 利用并行编译——ZeroClaw 把 robot-kit 拆成独立 crate
3. **用 `cargo check` 代替 `cargo build`**——日常开发只检查类型，不生成二进制

## 七、依赖管理实战速查

### 7.1 添加依赖的三种方式

```bash
# 方式 1：命令行（推荐日常使用）
cargo add serde --features derive           # 添加带 feature 的依赖
cargo add --dev pretty_assertions            # 添加开发依赖
cargo add --build cc                         # 添加构建依赖

# 方式 2：手动编辑 Cargo.toml（更精细控制）
# 直接在 [dependencies] 下添加

# 方式 3：指定 Git 源
cargo add my-lib --git https://github.com/user/my-lib --branch main
```

### 7.2 查看依赖树

```bash
cargo tree                       # 显示完整依赖树（包括传递依赖）
cargo tree -d                    # 只显示重复的依赖（有多个版本的）
cargo tree -e features           # 显示 feature 启用情况
cargo tree -i serde              # 反向查找：谁依赖了 serde？
```

**TypeScript 类比**：`cargo tree` ≈ `npm ls`，但输出更详细。

## 八、常见陷阱与最佳实践

### 陷阱 1：feature 不是互斥开关

```toml
# ❌ 错误设计：两个 feature 互斥
[features]
backend-postgres = ["dep:postgres"]
backend-sqlite = ["dep:sqlite"]
# 用户如果同时启用两个会怎样？→ 两个都编译进去！
```

**Features 是可加的（additive）**——启用 A + 启用 B = 启用 A 和 B。不要设计互斥 features [^search1]。

### 陷阱 2：传递依赖的 feature 统一

```toml
# crate A 依赖 tokio with features = ["rt"]
# crate B 依赖 tokio with features = ["macros"]
# 最终 tokio 被编译为 features = ["rt", "macros"]（取并集）
```

这意味着你不能假设某个 feature 没被启用——workspace 中另一个 crate 可能启用了它 [^search1]。

### 陷阱 3：`cargo add` 不下载

```bash
cargo add serde     # ← 只修改了 Cargo.toml，还没下载！
cargo build         # ← 这一步才下载 + 编译
# 不像 npm install 一步完成
```

### 最佳实践总结

```
实践                                原因
──────────────────────────────────────────────────────────────────
精确指定 features                   减少编译时间和产物体积
default-features = false 后按需加    避免引入不需要的系统依赖
path 依赖用于 workspace 内部        monorepo 内部引用
Git 依赖只用于临时/未发布的库        生产环境尽量用 crates.io 版本
定期 cargo update                   获取安全补丁
cargo tree -d 检查重复依赖           多版本共存会增加编译时间
```

## 总结

```
Cargo 依赖管理核心要点：

  三种依赖类型：
    [dependencies]       → 运行时需要       → npm dependencies
    [dev-dependencies]   → 仅测试需要       → npm devDependencies
    [build-dependencies] → 仅构建脚本需要   → npm 无对应

  版本规范：
    "1.2.3" = ^1.2.3    → 兼容更新（默认）
    "~1.2.3"            → 仅 patch 更新
    "=1.2.3"            → 精确锁定
    ⚠️ 0.x 版本：minor 视为 breaking

  三种来源：
    crates.io            → 生产首选（默认）
    git = "https://..."  → 未发布的库
    path = "crates/xxx"  → workspace 内部

  Features 控制：
    features = ["derive"]          → 启用特定功能
    default-features = false       → 关掉默认功能
    optional = true                → feature 门控的可选依赖

  与 npm 最大的区别：
    ✅ Features 是编译时功能开关——不用的代码根本不编译
    ✅ 没有 node_modules/——依赖缓存在 ~/.cargo/registry/
    ✅ 更严格的版本解析——0.x 的 minor 升级是 breaking
```
