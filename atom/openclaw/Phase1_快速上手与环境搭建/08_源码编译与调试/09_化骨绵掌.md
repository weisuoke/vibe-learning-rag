## 核心本质

### 本质 1：pnpm 不是包管理器，是依赖图管理器

**表面理解**：pnpm 是更快的 npm

**深层本质**：

pnpm 管理的是**依赖关系图**，而不是简单的文件复制：

```
传统 npm 思维：
package.json → 下载包 → 复制到 node_modules

pnpm 真实工作方式：
package.json → 构建依赖图 → 内容寻址存储 → 硬链接 + 符号链接
```

**为什么重要**：
- 理解依赖图才能理解幽灵依赖
- 理解内容寻址才能理解空间节省
- 理解链接机制才能理解跨平台问题

**实际影响**：
```typescript
// 代码中使用了未声明的依赖
import lodash from 'lodash'  // package.json 中没有

// npm: 可能能跑（其他包依赖了 lodash）
// pnpm: 报错（严格依赖管理）
```

---

### 本质 2：tsdown 不是编译器，是代码优化器

**表面理解**：tsdown 是更快的 tsc

**深层本质**：

tsdown 的核心是**代码优化**，而不是简单的类型擦除：

```
tsc 工作方式：
TypeScript → 类型检查 → 擦除类型 → JavaScript

tsdown 工作方式：
TypeScript → 依赖分析 → Tree-shaking → Code splitting → Minification → JavaScript
```

**为什么重要**：
- 理解优化才能理解为什么快
- 理解 bundling 才能理解入口点配置
- 理解 tree-shaking 才能理解未使用代码移除

**实际影响**：
```typescript
// utils.ts
export function used() { return 'used' }
export function unused() { return 'unused' }

// index.ts
import { used } from './utils'

// tsc 输出: 包含 used() 和 unused()
// tsdown 输出: 只包含 used()（tree-shaking）
```

---

### 本质 3：Vitest 不是测试框架，是隔离执行环境

**表面理解**：Vitest 是更快的 Jest

**深层本质**：

Vitest 的核心是**执行环境隔离**，而不是简单的测试运行：

```
Jest 工作方式：
测试文件 → 加载到内存 → 执行 → 清理

Vitest 工作方式：
测试文件 → 选择隔离级别 → 创建隔离环境 → 执行 → 销毁环境
```

**为什么重要**：
- 理解隔离才能理解 pool 模式选择
- 理解环境才能理解状态污染
- 理解销毁才能理解内存管理

**实际影响**：
```typescript
// test1.test.ts
test('test1', () => {
  globalState.value = 1
})

// test2.test.ts
test('test2', () => {
  expect(globalState.value).toBe(undefined)
})

// threads: 可能失败（共享状态）
// vmForks: 成功（完全隔离）
```

## 深层原理

### 原理 1：内容寻址存储的数学基础

**表面**：pnpm 使用哈希存储

**深层**：

内容寻址存储基于**密码学哈希函数**：

```
文件内容 → SHA-256 → 哈希值 → 存储路径

例如：
lodash@4.17.21/index.js
  ↓ SHA-256
  a1b2c3d4e5f6...
  ↓ 存储路径
  ~/.pnpm-store/v3/files/a1/b2c3d4e5f6...
```

**数学性质**：
1. **确定性**：相同内容 → 相同哈希
2. **唯一性**：不同内容 → 不同哈希（碰撞概率 < 10^-60）
3. **不可逆**：哈希 → 无法还原内容

**实际应用**：
- 去重：相同内容只存储一次
- 验证：哈希不匹配 = 内容被篡改
- 分布式：多机器共享 store

---

### 原理 2：Tree-shaking 的静态分析

**表面**：移除未使用代码

**深层**：

Tree-shaking 基于**静态程序分析**：

```
1. 构建调用图（Call Graph）
   entry.ts → function A → function B
                        → function C

2. 标记可达节点（Reachability Analysis）
   entry.ts: 可达
   function A: 可达
   function B: 可达
   function C: 可达
   function D: 不可达 ← 移除

3. 移除不可达代码（Dead Code Elimination）
```

**限制**：
- 只能分析静态导入（`import`）
- 无法分析动态导入（`require()`, `import()`）
- 无法分析运行时行为

**实际应用**：
```typescript
// ✅ 可以 tree-shake
import { used } from './utils'

// ❌ 无法 tree-shake
const moduleName = 'utils'
const module = require(`./${moduleName}`)
```

### 原理 3：进程隔离的操作系统机制

**表面**：vmForks 完全隔离

**深层**：

进程隔离基于**操作系统虚拟内存**：

```
threads（线程）:
  进程 A
    ├── 线程 1 → 共享内存
    ├── 线程 2 → 共享内存
    └── 线程 3 → 共享内存

forks（进程）:
  进程 A → 独立内存空间
  进程 B → 独立内存空间
  进程 C → 独立内存空间

vmForks（VM 进程）:
  VM 1 → 进程 A → 独立内存 + 独立文件系统
  VM 2 → 进程 B → 独立内存 + 独立文件系统
  VM 3 → 进程 C → 独立内存 + 独立文件系统
```

**性能代价**：
- threads: 快速，但可能状态污染
- forks: 中等，进程创建开销
- vmForks: 慢速，VM 创建开销

**实际应用**：
- 单元测试：threads（快速）
- E2E 测试：vmForks（隔离）
- Live 测试：单线程（避免并发）

## 反模式识别

### 反模式 1：过度依赖缓存

**错误做法**：
```bash
# 缓存所有东西
pnpm install --prefer-offline
tsdown --cache
vitest --cache
```

**问题**：
- 缓存失效导致难以调试
- 缓存污染导致构建不一致
- 缓存过期导致使用旧版本

**正确做法**：
```bash
# 开发环境：使用缓存
pnpm install --prefer-offline

# CI/CD 环境：清理缓存
pnpm install --frozen-lockfile
rm -rf .tsdown-cache
```

---

### 反模式 2：忽略原生依赖

**错误做法**：
```json
{
  "dependencies": {
    "sharp": "^0.33.0"
  }
}
```

**问题**：
- 跨平台编译失败
- 缺少编译工具链
- 二进制不兼容

**正确做法**：
```json
{
  "dependencies": {
    "sharp": "^0.33.0"
  },
  "optionalDependencies": {
    "@img/sharp-darwin-arm64": "^0.33.0",
    "@img/sharp-linux-x64": "^0.33.0",
    "@img/sharp-win32-x64": "^0.33.0"
  }
}
```

---

### 反模式 3：测试配置混用

**错误做法**：
```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    include: ['**/*.test.ts'],  // 所有测试
    pool: 'threads'  // 单一 pool 模式
  }
})
```

**问题**：
- 单元测试和 E2E 测试混在一起
- 隔离级别不匹配
- 性能和安全无法平衡

**正确做法**：
```typescript
// vitest.unit.config.ts
export default defineConfig({
  test: {
    include: ['**/*.test.ts'],
    pool: 'threads'  // 快速
  }
})

// vitest.e2e.config.ts
export default defineConfig({
  test: {
    include: ['**/*.e2e.test.ts'],
    pool: 'vmForks'  // 隔离
  }
})
```

## 系统思维

### 思维 1：依赖管理是图论问题

**传统思维**：依赖是树形结构

**系统思维**：依赖是有向无环图（DAG）

```
传统思维（树）:
  A
  ├── B
  │   └── D
  └── C
      └── D  ← 重复

系统思维（DAG）:
  A
  ├── B ──┐
  └── C ──┤
          ↓
          D  ← 共享
```

**实际应用**：
- pnpm 使用 DAG 去重
- workspace 协议管理内部依赖
- 循环依赖检测

---

### 思维 2：构建是数据流转换

**传统思维**：构建是文件转换

**系统思维**：构建是数据流转换

```
传统思维：
  .ts 文件 → .js 文件

系统思维：
  源码（数据）
    ↓ 解析（Parser）
  AST（数据）
    ↓ 转换（Transformer）
  优化后 AST（数据）
    ↓ 生成（Generator）
  目标代码（数据）
```

**实际应用**：
- tsdown 的 pipeline 设计
- 插件系统的数据流
- 增量构建的缓存策略

---

### 思维 3：测试是状态机验证

**传统思维**：测试是输入输出验证

**系统思维**：测试是状态机验证

```
传统思维：
  输入 → 函数 → 输出

系统思维：
  初始状态
    ↓ 操作 1
  中间状态 1
    ↓ 操作 2
  中间状态 2
    ↓ 操作 3
  最终状态
```

**实际应用**：
- E2E 测试的状态管理
- 并发测试的状态隔离
- 测试夹具的状态重置

## 工程权衡

### 权衡 1：性能 vs 隔离

**场景**：选择 Vitest pool 模式

**权衡矩阵**：
```
         性能    隔离    内存    适用场景
threads  ★★★★★  ★☆☆☆☆  ★★★★★  单元测试
forks    ★★★☆☆  ★★★★☆  ★★★☆☆  集成测试
vmForks  ★☆☆☆☆  ★★★★★  ★☆☆☆☆  E2E 测试
```

**决策原则**：
- 单元测试：优先性能（threads）
- E2E 测试：优先隔离（vmForks）
- 集成测试：平衡（forks）

---

### 权衡 2：构建速度 vs 产物质量

**场景**：选择构建工具

**权衡矩阵**：
```
         速度    优化    类型    适用场景
esbuild  ★★★★★  ★★☆☆☆  ★☆☆☆☆  开发构建
tsdown   ★★★★☆  ★★★★☆  ★★★★☆  生产构建
tsc      ★★☆☆☆  ★☆☆☆☆  ★★★★★  类型检查
```

**决策原则**：
- 开发环境：esbuild（快速迭代）
- 生产环境：tsdown（优化产物）
- 类型检查：tsc（准确性）

---

### 权衡 3：磁盘空间 vs 安装速度

**场景**：选择包管理器

**权衡矩阵**：
```
      空间    速度    严格性  适用场景
npm   ★☆☆☆☆  ★★☆☆☆  ★☆☆☆☆  传统项目
yarn  ★★☆☆☆  ★★★☆☆  ★★☆☆☆  Monorepo
pnpm  ★★★★★  ★★★★☆  ★★★★★  现代项目
```

**决策原则**：
- 磁盘受限：pnpm（节省 50-70%）
- 速度优先：pnpm 或 yarn
- 严格依赖：pnpm

## 元认知

### 认知 1：工具链是生态系统

**表面认知**：工具是独立的

**深层认知**：工具链是相互依赖的生态系统

```
pnpm（依赖管理）
  ↓ 提供依赖
tsdown（构建）
  ↓ 生成产物
Vitest（测试）
  ↓ 验证质量
Docker（部署）
  ↓ 运行环境
```

**实际影响**：
- 工具版本需要兼容
- 配置需要协调
- 优化需要整体考虑

---

### 认知 2：调试是假设验证

**表面认知**：调试是找错误

**深层认知**：调试是科学方法

```
1. 观察现象
   ↓
2. 提出假设
   ↓
3. 设计实验（断点、日志）
   ↓
4. 验证假设
   ↓
```

**实际应用**：
