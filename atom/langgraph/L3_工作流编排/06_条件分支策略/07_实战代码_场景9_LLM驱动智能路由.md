# 实战代码 - 场景9：LLM 驱动智能路由

> 使用 LLM 输出驱动条件路由，实现 ReAct Agent 模式——LLM 决定是否调用工具、调用哪个工具、还是直接回答，工具执行后回到 LLM 继续推理，直到 LLM 判断可以给出最终答案。

---

## 场景说明

**业务场景：ReAct Agent（推理-行动循环）**

用户提出一个问题，Agent 通过"思考 → 行动 → 观察"的循环来解决问题：

1. LLM 分析当前状态，决定下一步行动（调用搜索？调用计算器？直接回答？）
2. 如果需要工具，执行工具并将结果反馈给 LLM
3. LLM 根据工具结果继续推理，直到信息充足后给出最终答案

**这是 Agent 最核心的模式**——LLM 不再是被动的"问一答一"，而是主动规划、调用工具、迭代推理。

**为什么这是条件分支的终极场景？**
- 路由决策完全由 LLM 输出驱动（不是硬编码规则）
- 存在循环（工具执行后回到 LLM）
- 分支数量可扩展（新增工具只需新增节点）
- 需要循环守卫防止无限调用

**核心流程：**

```
START
  ↓
llm_node（LLM 思考 + 决策）◄──────────────┐
  ↓                                        │
route_action（条件路由）                     │
  ├── "search"    → search_tool ───────────┘  （工具执行后回到 LLM）
  ├── "calculate" → calc_tool ─────────────┘
  └── "final_answer" → answer_node → END
```

**本场景覆盖的核心要点：**
1. LLM 节点生成结构化输出（包含 action 字段）
2. 条件边根据 LLM 输出的 action 路由到不同工具
3. 工具执行后回到 LLM（形成循环）
4. LLM 决定 `"final_answer"` 时退出循环
5. 循环守卫防止 LLM 无限调用工具

## 完整可运行代码

### 基础版：ReAct Agent 模式

```python
"""
条件分支策略 - 实战场景9：LLM 驱动智能路由
演示：ReAct Agent 模式

核心模式：
- LLM 节点输出结构化决策（action 字段）
- 条件边根据 action 路由到对应工具
- 工具执行后回到 LLM（推理-行动循环）
- LLM 输出 "final_answer" 时退出循环
"""

from typing import TypedDict, Literal, Annotated
import operator
from langgraph.graph import StateGraph, START, END


# ===== 1. 状态定义 =====

class AgentState(TypedDict):
    """ReAct Agent 状态

    字段说明：
    - input: 用户原始问题
    - messages: 完整的推理轨迹（思考 + 工具结果）
    - current_action: LLM 当前决策的行动
    - tool_results: 所有工具执行结果的累积
    - final_answer: 最终回答
    - steps: 已执行步数（用于循环守卫）
    """
    input: str
    messages: Annotated[list[dict], operator.add]
    current_action: str       # "search" | "calculate" | "final_answer"
    tool_results: Annotated[list[str], operator.add]
    final_answer: str
    steps: int


# ===== 2. LLM 决策节点（Agent 的"大脑"） =====

def llm_node(state: AgentState) -> dict:
    """模拟 LLM 的推理与决策

    在真实项目中，这里会调用 OpenAI / Anthropic API，
    让 LLM 根据当前上下文决定下一步行动。

    模拟逻辑：
    - 第一步：搜索相关信息
    - 第二步：如果已有搜索结果，进行计算
    - 第三步及以后：给出最终回答
    """
    steps = state.get("steps", 0)
    tool_results = state.get("tool_results", [])

    # ---- 模拟 LLM 的思考过程 ----
    if steps == 0:
        # 第一步：需要搜索信息
        action = "search"
        thought = "我需要先搜索相关信息来回答这个问题"
    elif steps == 1 and "搜索结果" in str(tool_results):
        # 第二步：有了搜索结果，需要计算
        action = "calculate"
        thought = "已获取信息，需要进行计算来得出结论"
    else:
        # 信息充足，给出最终回答
        action = "final_answer"
        thought = "信息已经足够，可以给出最终回答了"

    print(f"  [LLM] 步骤 {steps + 1} | 思考: {thought} | 行动: {action}")

    return {
        "current_action": action,
        "messages": [{
            "role": "assistant",
            "content": f"思考: {thought}\n行动: {action}"
        }],
        "steps": steps + 1,
    }


# ===== 3. 路由函数（条件边的核心） =====

def route_action(state: AgentState) -> Literal["search_tool", "calc_tool", "answer"]:
    """根据 LLM 输出的 action 路由到对应节点

    关键点：
    - 返回值类型是 Literal，LangGraph 自动推断所有可能路径
    - 路由逻辑极简——只读取 LLM 的决策，不做额外判断
    - LLM 负责"想"，路由函数只负责"转发"
    """
    action = state["current_action"]
    if action == "search":
        return "search_tool"
    elif action == "calculate":
        return "calc_tool"
    return "answer"


# ===== 4. 工具节点 =====

def search_tool(state: AgentState) -> dict:
    """搜索工具：模拟搜索外部信息

    真实项目中这里会调用搜索 API（Google、Bing、向量数据库等）
    """
    query = state["input"]
    # 模拟搜索结果
    result = f"搜索结果: 关于「{query}」的信息——今日气温 25°C，晴转多云，东南风 3 级"
    print(f"  [搜索工具] 查询: {query} → 返回结果")

    return {
        "tool_results": [result],
        "messages": [{"role": "tool", "content": f"[搜索] {result}"}],
    }


def calc_tool(state: AgentState) -> dict:
    """计算工具：模拟数值计算

    真实项目中这里会调用计算引擎或代码解释器
    """
    # 模拟计算
    result = "计算结果: 体感温度 = 25°C × 0.7 + 湿度修正 = 约 22°C"
    print(f"  [计算工具] 执行计算 → 返回结果")

    return {
        "tool_results": [result],
        "messages": [{"role": "tool", "content": f"[计算] {result}"}],
    }


# ===== 5. 最终回答节点 =====

def answer_node(state: AgentState) -> dict:
    """汇总所有工具结果，生成最终回答

    真实项目中这里会再次调用 LLM，
    让它基于所有工具结果生成自然语言回答。
    """
    results = state.get("tool_results", [])
    steps = state.get("steps", 0)

    # 汇总所有工具结果
    evidence = "\n  ".join(results) if results else "无工具调用"
    final = f"经过 {steps} 步推理，综合以下证据：\n  {evidence}\n最终回答：今天天气晴好，气温 25°C，体感约 22°C，适合户外活动。"

    print(f"  [回答] 生成最终回答（基于 {len(results)} 条工具结果）")

    return {
        "final_answer": final,
        "messages": [{"role": "assistant", "content": f"[最终回答] {final}"}],
    }


# ===== 6. 构建图 =====

def build_react_agent():
    """构建 ReAct Agent 图

    核心结构：
    - START → llm（入口）
    - llm → 条件路由 → {search_tool, calc_tool, answer}
    - search_tool → llm（循环回到 LLM）
    - calc_tool → llm（循环回到 LLM）
    - answer → END（退出）
    """
    builder = StateGraph(AgentState)

    # 添加节点
    builder.add_node("llm", llm_node)
    builder.add_node("search_tool", search_tool)
    builder.add_node("calc_tool", calc_tool)
    builder.add_node("answer", answer_node)

    # 入口边
    builder.add_edge(START, "llm")

    # 条件边：LLM 输出驱动路由
    # LangGraph 从 Literal["search_tool", "calc_tool", "answer"] 自动推断路径
    builder.add_conditional_edges("llm", route_action)

    # 工具执行后回到 LLM（形成循环）
    builder.add_edge("search_tool", "llm")
    builder.add_edge("calc_tool", "llm")

    # 最终回答后结束
    builder.add_edge("answer", END)

    return builder.compile()


# ===== 7. 运行测试 =====

if __name__ == "__main__":
    graph = build_react_agent()

    print("=" * 60)
    print("ReAct Agent - LLM 驱动智能路由")
    print("=" * 60)

    result = graph.invoke({
        "input": "今天天气怎么样？",
        "messages": [],
        "current_action": "",
        "tool_results": [],
        "final_answer": "",
        "steps": 0,
    })

    print("\n" + "=" * 60)
    print("执行结果")
    print("=" * 60)
    print(f"最终回答: {result['final_answer']}")
    print(f"总步数: {result['steps']}")
    print(f"\n完整推理轨迹:")
    for i, msg in enumerate(result["messages"]):
        print(f"  {i+1}. [{msg['role']}] {msg['content'][:80]}")
```

### 进阶版：带循环守卫的 ReAct Agent

基础版有一个隐患：如果 LLM 一直不输出 `"final_answer"`，Agent 会无限循环。生产环境必须加循环守卫。

```python
"""
条件分支策略 - 实战场景9（进阶版）：带循环守卫的 ReAct Agent
演示：max_steps 防止无限循环 + 强制终止机制
"""

from typing import TypedDict, Literal, Annotated
import operator
import random
from langgraph.graph import StateGraph, START, END


# 最大允许步数（循环守卫的核心参数）
MAX_STEPS = 5


class AgentState(TypedDict):
    input: str
    messages: Annotated[list[dict], operator.add]
    current_action: str
    tool_results: Annotated[list[str], operator.add]
    final_answer: str
    steps: int


def llm_node_with_guard(state: AgentState) -> dict:
    """带循环守卫的 LLM 决策节点

    关键改进：
    1. 检查步数是否超过 MAX_STEPS
    2. 超过时强制输出 final_answer
    3. 模拟更真实的 LLM 行为（随机选择工具）
    """
    steps = state.get("steps", 0)
    tool_results = state.get("tool_results", [])

    # ---- 循环守卫：强制终止 ----
    if steps >= MAX_STEPS - 1:
        action = "final_answer"
        thought = f"已达到最大步数 {MAX_STEPS}，强制给出回答"
        print(f"  [LLM] ⚠️ 循环守卫触发！强制终止")
    # ---- 正常 LLM 决策 ----
    elif steps == 0:
        action = "search"
        thought = "需要搜索相关信息"
    elif len(tool_results) < 2:
        # 模拟 LLM 随机选择工具（真实场景中由 LLM 决定）
        action = random.choice(["search", "calculate"])
        thought = f"还需要更多信息，决定使用 {action}"
    else:
        action = "final_answer"
        thought = "信息充足，可以回答"

    print(f"  [LLM] 步骤 {steps + 1}/{MAX_STEPS} | 行动: {action}")

    return {
        "current_action": action,
        "messages": [{"role": "assistant", "content": f"[步骤{steps+1}] {thought} → {action}"}],
        "steps": steps + 1,
    }


def route_action(state: AgentState) -> Literal["search_tool", "calc_tool", "answer"]:
    """路由函数（与基础版相同）"""
    action = state["current_action"]
    if action == "search":
        return "search_tool"
    elif action == "calculate":
        return "calc_tool"
    return "answer"


def search_tool(state: AgentState) -> dict:
    query = state["input"]
    result = f"搜索结果(步骤{state['steps']}): 关于「{query}」的信息"
    return {
        "tool_results": [result],
        "messages": [{"role": "tool", "content": result}],
    }


def calc_tool(state: AgentState) -> dict:
    result = f"计算结果(步骤{state['steps']}): 数值分析完成"
    return {
        "tool_results": [result],
        "messages": [{"role": "tool", "content": result}],
    }


def answer_node(state: AgentState) -> dict:
    results = state.get("tool_results", [])
    steps = state.get("steps", 0)
    is_forced = steps >= MAX_STEPS

    status = "（循环守卫强制终止）" if is_forced else "（正常完成）"
    final = f"经过 {steps} 步推理{status}，基于 {len(results)} 条证据生成回答。"

    return {
        "final_answer": final,
        "messages": [{"role": "assistant", "content": f"[最终回答] {final}"}],
    }


# 构建图
builder = StateGraph(AgentState)
builder.add_node("llm", llm_node_with_guard)
builder.add_node("search_tool", search_tool)
builder.add_node("calc_tool", calc_tool)
builder.add_node("answer", answer_node)

builder.add_edge(START, "llm")
builder.add_conditional_edges("llm", route_action)
builder.add_edge("search_tool", "llm")
builder.add_edge("calc_tool", "llm")
builder.add_edge("answer", END)

graph = builder.compile()

if __name__ == "__main__":
    print("=" * 60)
    print("ReAct Agent（带循环守卫）- MAX_STEPS =", MAX_STEPS)
    print("=" * 60)

    result = graph.invoke({
        "input": "分析一下最近的市场趋势",
        "messages": [],
        "current_action": "",
        "tool_results": [],
        "final_answer": "",
        "steps": 0,
    })

    print(f"\n最终回答: {result['final_answer']}")
    print(f"总步数: {result['steps']}")
```

## 关键设计解析

### 1. LLM 驱动路由的核心模式

```python
# 模式的本质：LLM 输出 → 结构化决策 → 条件路由

# 步骤1：LLM 节点输出包含 action 字段的结构化结果
def llm_node(state) -> dict:
    # 真实项目中：调用 LLM API，要求输出 JSON 格式
    # response = openai.chat.completions.create(
    #     model="gpt-4",
    #     messages=state["messages"],
    #     response_format={"type": "json_object"},  # 强制 JSON 输出
    # )
    # decision = json.loads(response.choices[0].message.content)
    # return {"current_action": decision["action"]}
    ...

# 步骤2：条件边读取 action，路由到对应节点
def route_action(state) -> Literal["search_tool", "calc_tool", "answer"]:
    return state["current_action"]  # 极简——只转发 LLM 的决策

# 步骤3：工具节点执行后，边指回 LLM（形成循环）
builder.add_edge("search_tool", "llm")  # 关键：工具 → LLM
builder.add_edge("calc_tool", "llm")    # 关键：工具 → LLM
```

**核心洞察：路由函数本身不做决策，它只是 LLM 决策的"传声筒"。真正的智能在 LLM 节点里。**

### 2. 循环守卫的三种实现方式

```python
# ---- 方式1：节点内计数器（本场景使用） ----
# 优点：灵活，可以在强制终止前做清理工作
# 缺点：需要在状态中维护 steps 字段
def llm_node(state):
    if state["steps"] >= MAX_STEPS:
        return {"current_action": "final_answer"}  # 强制退出
    ...

# ---- 方式2：路由函数中检查（推荐简单场景） ----
# 优点：路由逻辑集中
# 缺点：路由函数承担了过多职责
def route_with_guard(state) -> Literal["search_tool", "calc_tool", "answer"]:
    if state["steps"] >= MAX_STEPS:
        return "answer"  # 强制路由到终点
    return {"search": "search_tool", "calculate": "calc_tool"}.get(
        state["current_action"], "answer"
    )

# ---- 方式3：LangGraph recursion_limit（全局兜底） ----
# 优点：零代码，全局生效
# 缺点：触发时抛异常，不够优雅
graph = builder.compile()
result = graph.invoke(input_state, {"recursion_limit": 25})
```

**生产建议：方式1 + 方式3 组合使用。方式1 做优雅降级，方式3 做最后兜底。**

### 3. 与 LangGraph prebuilt `create_react_agent` 的对比

LangGraph 提供了开箱即用的 ReAct Agent，和我们手写的有什么区别？

```python
# ---- LangGraph prebuilt 版本 ----
from langgraph.prebuilt import create_react_agent

# 一行代码创建 ReAct Agent
agent = create_react_agent(
    model=ChatOpenAI(model="gpt-4"),
    tools=[search_tool, calc_tool],
)

# ---- 手写版本（本场景） ----
# 需要自己定义状态、节点、路由、循环守卫...
```

| 维度 | `create_react_agent` | 手写 ReAct |
|------|---------------------|------------|
| 代码量 | 1-5 行 | 50-100 行 |
| 灵活性 | 受限于预设结构 | 完全自定义 |
| 工具调用 | 自动处理 tool_calls | 需要手动解析 |
| 循环守卫 | 内置 `recursion_limit` | 需要手动实现 |
| 状态定制 | 有限（可通过 `state_schema` 扩展） | 完全自由 |
| 多 Agent | 不直接支持 | 可自由组合 |
| 适用场景 | 标准 ReAct 模式 | 复杂自定义 Agent |

**选择原则：**
- 标准 ReAct 模式 → 用 `create_react_agent`，省时省力
- 需要自定义推理逻辑、多 Agent 协作、特殊状态管理 → 手写

### 4. 扩展工具的方式

新增工具只需三步，不需要修改现有代码：

```python
# 步骤1：新增工具节点
def weather_tool(state: AgentState) -> dict:
    return {
        "tool_results": ["天气数据: 晴天 25°C"],
        "messages": [{"role": "tool", "content": "天气查询完成"}],
    }

# 步骤2：注册节点
builder.add_node("weather_tool", weather_tool)

# 步骤3：添加回到 LLM 的边
builder.add_edge("weather_tool", "llm")

# 步骤4：更新路由函数的 Literal 类型和分支逻辑
def route_action(state) -> Literal["search_tool", "calc_tool", "weather_tool", "answer"]:
    action = state["current_action"]
    if action == "weather":
        return "weather_tool"
    ...
```

## 常见陷阱

### 陷阱1：忘记循环守卫

```python
# ❌ 危险：LLM 可能永远不输出 "final_answer"
def llm_node(state):
    action = call_llm(state)  # 如果 LLM 一直返回 "search"...
    return {"current_action": action}

# ✅ 安全：加上步数限制
def llm_node(state):
    if state["steps"] >= MAX_STEPS:
        return {"current_action": "final_answer"}  # 强制退出
    action = call_llm(state)
    return {"current_action": action}
```

### 陷阱2：路由函数返回值与节点名不匹配

```python
# ❌ 错误：路由返回 "search" 但节点名是 "search_tool"
def route_action(state):
    return state["current_action"]  # 返回 "search"
builder.add_node("search_tool", search_tool)  # 节点名是 "search_tool"
# → 运行时报错：找不到节点 "search"

# ✅ 正确：路由返回值必须匹配节点名
def route_action(state) -> Literal["search_tool", "calc_tool", "answer"]:
    action_map = {"search": "search_tool", "calculate": "calc_tool"}
    return action_map.get(state["current_action"], "answer")
```

### 陷阱3：工具节点忘记连回 LLM

```python
# ❌ 错误：工具执行后直接结束
builder.add_edge("search_tool", END)  # 搜索完就结束了，没有后续推理

# ✅ 正确：工具执行后回到 LLM 继续推理
builder.add_edge("search_tool", "llm")  # 搜索完回到 LLM 决定下一步
```

## 一句话总结

**ReAct Agent 的本质就是"LLM 决策 → 条件路由 → 工具执行 → 回到 LLM"的循环，条件边是这个循环的调度器，循环守卫是它的刹车。**
