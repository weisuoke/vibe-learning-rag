# 实战代码 - 场景2：数据清洗管道

## 场景说明

本场景演示如何用 LangGraph 构建一个多步骤数据清洗管道，将脏数据通过一系列节点逐步清洗为干净、规范的输出。

核心知识点：
1. 线性管道模式——多个清洗节点串联执行
2. Reducer 字段的累积效果——`errors` 和 `pipeline_steps` 跨节点自动追加
3. 验证失败的传播——通过 `is_valid` 标志让下游节点跳过处理
4. 每个节点只做一件事——单一职责的清洗步骤

这是状态转换管道模式的典型应用：输入 → 验证 → 去噪 → 分词 → 格式化 → 输出，每一步都是一个独立的状态转换函数。

## 完整代码

```python
"""
LangGraph 状态转换函数 - 数据清洗管道
演示：多步骤数据清洗与格式化

核心知识点：
- 线性管道：validate → denoise → tokenize → format
- Reducer 累积：errors 和 pipeline_steps 跨节点自动追加
- 验证失败传播：is_valid=False 时下游节点跳过处理
- 单一职责：每个节点只做一步清洗

运行环境：Python 3.13+, langgraph
安装依赖：uv add langgraph
"""

from typing import Annotated
from typing_extensions import TypedDict
from operator import add
from langgraph.graph import StateGraph, START, END
import re


# ============================================================
# 1. 状态定义
# ============================================================

class CleaningState(TypedDict):
    """数据清洗管道状态

    无 Reducer 字段（LastValue Channel）：
    - raw_input, cleaned_text, tokens, word_count, is_valid
      → 每次返回直接覆盖旧值

    有 Reducer 字段（BinaryOperatorAggregate Channel）：
    - errors → 使用 operator.add，每个节点返回的错误列表会追加
    - pipeline_steps → 使用 operator.add，记录每个节点的处理日志
    """
    # 无 Reducer：直接覆盖
    raw_input: str              # 原始输入
    cleaned_text: str           # 清洗后文本
    tokens: list[str]           # 分词结果
    word_count: int             # 词数统计
    is_valid: bool              # 是否通过验证

    # 有 Reducer：跨节点累积追加
    errors: Annotated[list[str], add]           # 错误日志
    pipeline_steps: Annotated[list[str], add]   # 处理步骤记录


# ============================================================
# 2. 清洗节点——每个节点只做一步清洗
# ============================================================

def validate_input(state: CleaningState) -> dict:
    """步骤1：输入验证——检查原始输入是否合法"""
    raw = state.get("raw_input", "")

    if not raw or not raw.strip():
        return {
            "is_valid": False,
            "errors": ["输入为空"],
            "pipeline_steps": ["validate_input: FAILED - 空输入"],
        }

    if len(raw) > 10000:
        return {
            "is_valid": False,
            "errors": ["输入超过10000字符限制"],
            "pipeline_steps": ["validate_input: FAILED - 超长"],
        }

    return {
        "is_valid": True,
        "pipeline_steps": [f"validate_input: OK ({len(raw)} chars)"],
    }


def remove_noise(state: CleaningState) -> dict:
    """步骤2：去除噪声——清洗 HTML 标签、多余空白、特殊字符"""
    if not state.get("is_valid", False):
        return {"pipeline_steps": ["remove_noise: SKIPPED"]}

    text = state["raw_input"]

    # 去除 HTML 标签
    text = re.sub(r'<[^>]+>', '', text)
    # 合并多余空白为单个空格
    text = re.sub(r'\s+', ' ', text).strip()
    # 去除特殊字符（保留中文、英文、数字、基本标点）
    text = re.sub(r'[^\w\s\u4e00-\u9fff.,!?;:，。！？；：]', '', text)

    return {
        "cleaned_text": text,
        "pipeline_steps": [f"remove_noise: 清洗完成 ({len(text)} chars)"],
    }


def tokenize_text(state: CleaningState) -> dict:
    """步骤3：分词——将清洗后文本拆分为 token 列表"""
    if not state.get("is_valid", False):
        return {"pipeline_steps": ["tokenize: SKIPPED"]}

    text = state.get("cleaned_text", "")
    # 按空格和中英文标点分割
    tokens = [
        t.strip()
        for t in re.split(r'[\s,;.!?，。！？；：]+', text)
        if t.strip()
    ]

    return {
        "tokens": tokens,
        "word_count": len(tokens),
        "pipeline_steps": [f"tokenize: {len(tokens)} tokens"],
    }


def final_format(state: CleaningState) -> dict:
    """步骤4：最终格式化——将 token 列表重新组合为标准化文本"""
    if not state.get("is_valid", False):
        return {
            "cleaned_text": "",
            "pipeline_steps": ["final_format: 返回空结果"],
        }

    tokens = state.get("tokens", [])
    cleaned = " ".join(tokens)

    return {
        "cleaned_text": cleaned,
        "pipeline_steps": ["final_format: 完成"],
    }


# ============================================================
# 3. 构建清洗管道——线性串联
# ============================================================

graph = StateGraph(CleaningState)

# 添加节点
graph.add_node("validate", validate_input)
graph.add_node("denoise", remove_noise)
graph.add_node("tokenize", tokenize_text)
graph.add_node("format", final_format)

# 线性串联：validate → denoise → tokenize → format
graph.add_edge(START, "validate")
graph.add_edge("validate", "denoise")
graph.add_edge("denoise", "tokenize")
graph.add_edge("tokenize", "format")
graph.add_edge("format", END)

# 编译
pipeline = graph.compile()


# ============================================================
# 4. 测试用例
# ============================================================

def make_initial_state(raw_input: str) -> CleaningState:
    """构造初始状态——所有字段都需要初始值"""
    return {
        "raw_input": raw_input,
        "cleaned_text": "",
        "tokens": [],
        "word_count": 0,
        "is_valid": False,
        "errors": [],
        "pipeline_steps": [],
    }


def print_result(title: str, result: dict):
    """格式化打印结果"""
    print(f"\n{'=' * 50}")
    print(f"  {title}")
    print(f"{'=' * 50}")
    print(f"  有效: {result['is_valid']}")
    print(f"  清洗结果: '{result['cleaned_text']}'")
    print(f"  词数: {result['word_count']}")
    print(f"  tokens: {result['tokens']}")
    if result['errors']:
        print(f"  错误: {result['errors']}")
    print(f"  管道步骤:")
    for step in result['pipeline_steps']:
        print(f"    → {step}")


print("=== LangGraph 数据清洗管道 ===")

# --- 测试1：正常输入（含 HTML 标签和多余空白）---
result1 = pipeline.invoke(
    make_initial_state("  <b>Hello</b>   World!  这是一个  测试   ")
)
print_result("测试1: 正常输入（含 HTML + 多余空白）", result1)

# --- 测试2：空输入 ---
result2 = pipeline.invoke(make_initial_state("   "))
print_result("测试2: 空输入", result2)

# --- 测试3：含特殊字符的输入 ---
result3 = pipeline.invoke(
    make_initial_state("价格是 $100 & 折扣 <50%>！！！快来★抢购☆吧")
)
print_result("测试3: 含特殊字符", result3)

# --- 测试4：纯中文输入 ---
result4 = pipeline.invoke(
    make_initial_state("LangGraph是一个强大的框架，用于构建多Agent系统。")
)
print_result("测试4: 纯中文输入", result4)
```

## 关键知识点解析

### 1. Reducer 字段的累积效果

这个管道中最值得关注的是 `errors` 和 `pipeline_steps` 两个字段。它们使用了 `Annotated[list[str], add]`，意味着每个节点返回的列表不会覆盖旧值，而是追加到已有列表。

```python
# 状态定义中的 Reducer 字段
errors: Annotated[list[str], add]           # operator.add 作为 Reducer
pipeline_steps: Annotated[list[str], add]   # operator.add 作为 Reducer
```

执行流程中的累积过程：

```
初始状态:  pipeline_steps = []
validate:  返回 ["validate_input: OK"]     → 累积为 ["validate_input: OK"]
denoise:   返回 ["remove_noise: 清洗完成"]  → 累积为 ["validate_input: OK", "remove_noise: 清洗完成"]
tokenize:  返回 ["tokenize: 4 tokens"]     → 累积为 [..., "tokenize: 4 tokens"]
format:    返回 ["final_format: 完成"]      → 累积为 [..., "final_format: 完成"]
```

对比普通字段 `cleaned_text`（无 Reducer）：

```
初始状态:  cleaned_text = ""
denoise:   返回 "Hello World 这是一个 测试"  → 覆盖为 "Hello World 这是一个 测试"
format:    返回 "Hello World 这是一个 测试"  → 覆盖为 "Hello World 这是一个 测试"
```

这就是 Reducer 和非 Reducer 字段的核心区别：Reducer 字段跨节点累积，普通字段每次覆盖。

### 2. 验证失败的传播模式

管道中的错误传播依赖一个简单的 `is_valid` 标志：

```python
# validate_input 设置 is_valid=False
# 后续每个节点都检查这个标志
def remove_noise(state: CleaningState) -> dict:
    if not state.get("is_valid", False):
        return {"pipeline_steps": ["remove_noise: SKIPPED"]}
    # ... 正常处理
```

这种模式的好处：
- 验证失败时，后续节点不会崩溃，而是优雅跳过
- 每个跳过的节点仍然会记录到 `pipeline_steps`（因为是 Reducer 字段）
- 最终可以从 `pipeline_steps` 中看到完整的执行轨迹

### 3. 管道模式 vs 单节点处理

为什么要拆成 4 个节点，而不是写一个大函数？

| 优势 | 说明 |
|------|------|
| 可观测性 | 通过 `stream` 可以看到每一步的中间状态 |
| 可复用性 | `remove_noise` 节点可以在其他管道中复用 |
| 可测试性 | 每个节点可以独立单元测试 |
| 可扩展性 | 需要新的清洗步骤？插入一个新节点即可 |
| 错误定位 | `pipeline_steps` 精确记录哪一步出了问题 |

### 4. 初始状态的完整性

构造初始状态时，所有字段都需要提供初始值：

```python
def make_initial_state(raw_input: str) -> CleaningState:
    return {
        "raw_input": raw_input,
        "cleaned_text": "",     # 普通字段：给空字符串
        "tokens": [],           # 普通字段：给空列表
        "word_count": 0,        # 普通字段：给 0
        "is_valid": False,      # 普通字段：给 False
        "errors": [],           # Reducer 字段：给空列表（累积的起点）
        "pipeline_steps": [],   # Reducer 字段：给空列表（累积的起点）
    }
```

Reducer 字段的初始值是累积的起点。如果 `errors` 初始为 `["预置错误"]`，那么 validate 节点返回 `["输入为空"]` 后，最终 `errors` 会是 `["预置错误", "输入为空"]`。

## 进阶：用 stream 观察管道执行过程

```python
"""
使用 stream 逐步观察每个节点的状态变化
这是管道模式的核心优势之一——完整的可观测性
"""

print("\n=== Stream 模式：逐步观察 ===\n")

for event in pipeline.stream(
    make_initial_state("  <b>Hello</b>   World!  "),
    stream_mode="updates",
):
    for node_name, updates in event.items():
        print(f"[{node_name}] 返回的更新:")
        for key, value in updates.items():
            print(f"  {key}: {value}")
        print()
```

预期输出：

```
=== Stream 模式：逐步观察 ===

[validate] 返回的更新:
  is_valid: True
  pipeline_steps: ['validate_input: OK (25 chars)']

[denoise] 返回的更新:
  cleaned_text: Hello World
  pipeline_steps: ['remove_noise: 清洗完成 (11 chars)']

[tokenize] 返回的更新:
  tokens: ['Hello', 'World']
  word_count: 2
  pipeline_steps: ['tokenize: 2 tokens']

[format] 返回的更新:
  cleaned_text: Hello World
  pipeline_steps: ['final_format: 完成']
```

通过 `stream_mode="updates"`，可以清楚看到每个节点返回了哪些字段更新。注意 `pipeline_steps` 每次只返回当前节点的一条记录，但在最终状态中会累积为完整列表——这就是 Reducer 在幕后做的事。

## 常见问题

### Q1: 为什么 Reducer 字段要用列表包裹单个元素？

```python
# ✅ 正确：返回列表
return {"errors": ["输入为空"]}

# ❌ 错误：返回字符串
return {"errors": "输入为空"}
# TypeError: operator.add 无法将 str 追加到 list
```

因为 `operator.add` 对列表执行的是 `list.__add__`（列表拼接），所以两边都必须是列表。

### Q2: 如果某个节点不返回 Reducer 字段会怎样？

```python
def some_node(state):
    # 没有返回 errors 和 pipeline_steps
    return {"cleaned_text": "hello"}
```

完全没问题。Reducer 字段不要求每个节点都返回。如果某个节点没有返回 `errors`，那 `errors` 的值保持不变（不会追加任何内容）。

## 总结

本场景展示了 LangGraph 状态转换函数在数据清洗管道中的应用：

1. 线性管道 `validate → denoise → tokenize → format`，每个节点单一职责
2. Reducer 字段 `errors` 和 `pipeline_steps` 跨节点自动累积，提供完整的执行轨迹
3. `is_valid` 标志实现验证失败的优雅传播，下游节点跳过而非崩溃
4. `stream` 模式提供逐节点的可观测性，方便调试和监控

这个模式可以直接迁移到 RAG 场景：查询清洗 → 检索 → 结果去重 → 格式化注入，管道中的每一步都是一个独立的状态转换函数。
