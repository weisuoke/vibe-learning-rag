## 卡片 1：Result 直觉理解

**一句话：** Result 是一个"双通道信封"——Ok 通道装成功值，Err 通道装错误信息，编译器强制你拆开看是哪个通道。

**代码速览：**

```rust
// Result 的定义（标准库内置，不需要 import）
enum Result<T, E> {
    Ok(T),   // 成功通道：携带类型为 T 的值
    Err(E),  // 失败通道：携带类型为 E 的错误
}

// 使用：函数签名明确告诉你"我可能失败"
fn parse_port(s: &str) -> Result<u16, String> {
    s.parse::<u16>().map_err(|e| format!("无效端口: {e}"))
}

let port = parse_port("8080");  // Ok(8080)
let bad  = parse_port("abc");   // Err("无效端口: ...")
```

```typescript
// TypeScript 对照：函数签名看不出会不会抛异常
function parsePort(s: string): number {
  const n = parseInt(s);
  if (isNaN(n)) throw new Error("无效端口");  // 隐式！
  return n;
}
```

**ZeroClaw 应用：** 每个 Trait 方法都返回 `anyhow::Result<T>`。`Tool::execute` 返回 `Result<ToolResult>`，`Memory::store` 返回 `Result<()>`，`Channel::receive` 返回 `Result<ChannelMessage>`。不理解 Result 就看不懂任何一个方法签名。

[来源: reference/source_error_handling_01.md]

**记住：** Result = 双通道信封，编译器强制你拆。

## 卡片 2：Option 直觉理解

**一句话：** Option 是 Rust 版的"可能没有"——Some 表示有值，None 表示没有，编译器强制你处理"没有"的情况。

**代码速览：**

```rust
// Option 的定义（标准库内置）
enum Option<T> {
    Some(T),  // 有值
    None,     // 没有值（替代 null/undefined）
}

// 使用：类型签名明确告诉你"这个值可能不存在"
fn find_user(id: u64) -> Option<String> {
    if id == 1 { Some("Alice".to_string()) } else { None }
}

let user = find_user(1);   // Some("Alice")
let ghost = find_user(99); // None
```

```typescript
// TypeScript 对照：null/undefined 是隐式的运行时炸弹
function findUser(id: number): string | null {
  return id === 1 ? "Alice" : null;
}
// 忘记检查 null → TypeError: Cannot read properties of null
```

**ZeroClaw 应用：** 配置系统大量使用 Option 区分"必填"和"可选"：

```rust
// src/config/schema.rs
pub struct ProviderConfig {
    pub name: String,              // 必填，绝不可能为空
    pub api_key: Option<String>,   // 可选，用 Ollama 时不需要
    pub api_url: Option<String>,   // 可选，不设置用默认值
}
```

[来源: reference/source_error_handling_01.md]

**记住：** Option = 编译器强制的 null 检查。`String` 一定有值，`Option<String>` 可能没有。

## 卡片 3：? 运算符——错误传播的语法糖

**一句话：** `?` 是 match 的语法糖——Ok 时解包继续，Err 时提前返回，一个字符替代 5 行模板代码。

**展开前后对比：**

```rust
// ===== 没有 ? 的写法（冗长）=====
fn read_config(path: &str) -> Result<String, std::io::Error> {
    let content = match std::fs::read_to_string(path) {
        Ok(c) => c,
        Err(e) => return Err(e),  // 手动提前返回
    };
    Ok(content)
}

// ===== 用 ? 的写法（简洁）=====
fn read_config(path: &str) -> Result<String, std::io::Error> {
    let content = std::fs::read_to_string(path)?;  // 一个 ? 搞定
    Ok(content)
}
```

```typescript
// TypeScript 没有等价物。最接近的是 ?. 但那是空值链，不是错误传播
// TS 的错误传播是隐式的（throw 自动冒泡），不需要也无法显式标记
```

**ZeroClaw 应用：** 几乎每个函数体都有 `?`，它是 ZeroClaw 代码中出现频率最高的符号之一：

```rust
// src/memory/sqlite.rs —— 连续 ? 传播
let conn = Connection::open(&path).context("SQLite failed to open database")?;
conn.execute_batch(CREATE_TABLE_SQL).context("Failed to create tables")?;
```

[来源: reference/source_error_handling_01.md, reference/context7_rust_01.md]

**记住：** `?` = "如果出错就返回错误，否则解包继续"。一个字符，五行功能。

## 卡片 4：match 模式匹配处理错误

**一句话：** match 是 Rust 的"穷尽 switch"——必须覆盖 Ok/Err（或 Some/None）的每种情况，漏掉任何一种编译不过。

**基础 match：**

```rust
fn handle_result(r: Result<i32, String>) {
    match r {
        Ok(value) => println!("成功: {value}"),
        Err(e)    => println!("失败: {e}"),
        // 漏掉任何一个分支 → 编译错误
    }
}
```

**ZeroClaw 嵌套 match——shell.rs 超时模式：**

```rust
// src/tools/shell.rs —— 超时 + 执行结果的嵌套 match
match tokio::time::timeout(Duration::from_secs(SHELL_TIMEOUT_SECS), cmd.output()).await {
    Ok(Ok(output)) => {
        // 第一层 Ok：没超时；第二层 Ok：命令执行成功
        let stdout = String::from_utf8_lossy(&output.stdout);
        ToolResult { success: true, output: stdout.to_string(), .. }
    },
    Ok(Err(e)) => {
        // 没超时，但命令执行失败
        ToolResult { success: false, error: Some(e.to_string()), .. }
    },
    Err(_) => {
        // 超时了
        ToolResult { success: false, error: Some("Command timed out".into()), .. }
    },
}
```

这个模式的类型拆解：`timeout()` 返回 `Result<Result<Output, io::Error>, Elapsed>`，外层是超时判断，内层是命令执行结果。

[来源: reference/source_error_handling_01.md]

**记住：** match = 穷尽的 switch。嵌套 Result 用嵌套 match，编译器保证你不会漏掉任何情况。

## 卡片 5：组合子链——函数式错误处理

**一句话：** `.get().and_then().ok_or_else()` 是 Option/Result 的链式操作——像 Promise 的 `.then().catch()` 一样串联处理，避免嵌套 match。

**核心组合子速查：**

| 组合子 | 作用 | 类比 |
|--------|------|------|
| `.map(f)` | 转换 Ok/Some 内的值 | `Promise.then(v => transform(v))` |
| `.and_then(f)` | 链式操作（f 也返回 Result/Option） | `Promise.then(v => fetchMore(v))` |
| `.unwrap_or(default)` | 失败时用默认值 | `value ?? defaultValue` |
| `.ok_or_else(f)` | Option 转 Result（None 变 Err） | 无直接对应 |

**ZeroClaw 参数提取——最常见的组合子链：**

```rust
// src/tools/shell.rs —— 从 JSON 参数中安全提取字符串
let command = args.get("command")       // Option<&Value>  —— 键可能不存在
    .and_then(|v| v.as_str())           // Option<&str>    —— 值可能不是字符串
    .ok_or_else(|| anyhow!("Missing 'command' parameter"))?;  // &str —— None 转 Err 并传播
//  ^^^^^^^^^^^                         Option → Result 的桥梁

// 对比不用组合子的写法（嵌套 match 地狱）：
let command = match args.get("command") {
    Some(v) => match v.as_str() {
        Some(s) => s,
        None => return Err(anyhow!("'command' is not a string")),
    },
    None => return Err(anyhow!("Missing 'command' parameter")),
};
```

```typescript
// TypeScript 对照：optional chaining + nullish coalescing
const command = args?.command ?? (() => { throw new Error("Missing command"); })();
// 但 TS 版本没有类型安全保证，args.command 可能是任何类型
```

[来源: reference/source_error_handling_01.md]

**记住：** 组合子链 = 函数式管道。`.get().and_then().ok_or_else()?` 是 ZeroClaw 中最常见的参数提取模式。

## 卡片 6：anyhow 三件套——bail!/ensure!/context

**一句话：** `bail!` 立即返回错误，`ensure!` 条件不满足时返回错误，`.context()` 给错误附加上下文——三个工具覆盖 90% 的错误处理场景。

**三件套用法：**

```rust
use anyhow::{bail, ensure, Context, Result};

fn validate_and_load(url: &str, data: &[u8]) -> Result<String> {
    // bail! —— 立即返回错误（等价于 return Err(anyhow!(...))）
    if url.is_empty() {
        bail!("URL cannot be empty");
    }

    // ensure! —— 条件断言（条件为 false 时返回 Err）
    ensure!(!data.is_empty(), "input data cannot be empty");
    ensure!(data.len() < 10_000_000, "data too large: {} bytes", data.len());

    // .context() —— 给已有错误附加可读上下文
    let content = std::fs::read_to_string(url)
        .context("failed to read config file")?;

    Ok(content)
}
```

**ZeroClaw 对应源码：**

```rust
// bail! —— src/tools/web_fetch.rs
anyhow::bail!("Only http:// and https:// URLs are allowed");

// ensure! —— src/security/secrets.rs
anyhow::ensure!(blob.len() > NONCE_LEN, "Encrypted value too short");

// .context() —— src/memory/sqlite.rs
Connection::open(&path_buf).context("SQLite failed to open database")?;
```

[来源: reference/context7_anyhow_01.md, reference/source_error_handling_01.md]

**记住：** bail! = 立即报错，ensure! = 条件断言，.context() = 附加说明。三件套在手，错误处理不愁。

## 卡片 7：thiserror vs anyhow——何时用哪个

**一句话：** thiserror 用于库代码（定义具体错误类型，调用者可 match），anyhow 用于应用代码（类型擦除，灵活附加上下文）。

**对比表：**

| 维度 | thiserror | anyhow |
|------|-----------|--------|
| **定位** | 库代码（被别人调用） | 应用代码（最终程序） |
| **错误类型** | 具体枚举 `enum MyError { ... }` | 类型擦除 `anyhow::Error` |
| **调用者能否 match** | 能，每个变体都可匹配 | 不能（需要 downcast） |
| **上下文附加** | 手动在 `#[error("...")]` 中写 | `.context("msg")` 链式附加 |
| **使用复杂度** | 需要定义枚举 + derive | 直接用 `Result<T>` |
| **TypeScript 类比** | 自定义 Error 子类 | 通用 Error + message |

**thiserror 示例：**

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum ConfigError {
    #[error("file not found: {path}")]
    NotFound { path: String },
    #[error("parse error")]
    Parse(#[from] serde_json::Error),  // #[from] 自动实现 From trait
    #[error(transparent)]
    Other(#[from] anyhow::Error),      // 透明委托
}
```

**ZeroClaw 的选择：** 全局使用 anyhow，因为 ZeroClaw 是应用层代码（AI Agent 运行时），不是被其他 crate 调用的库。所有 Trait 方法统一返回 `anyhow::Result<T>`，不使用自定义错误枚举。

[来源: reference/context7_thiserror_01.md, reference/source_error_handling_01.md]

**记住：** 写库用 thiserror（让调用者 match），写应用用 anyhow（灵活省事）。ZeroClaw 是应用 → 用 anyhow。

## 卡片 8：错误链——追踪错误根因

**一句话：** `.context()` 层层嵌套形成错误链，从最外层的业务描述到最内层的系统错误，像洋葱一样一层层剥开找根因。

**错误链构建：**

```rust
use anyhow::{Context, Result};

fn load_agent_config(path: &str) -> Result<Config> {
    let raw = std::fs::read_to_string(path)
        .context("failed to read config file")?;          // 第 3 层：IO 错误
    let config: Config = serde_json::from_str(&raw)
        .context("failed to parse config JSON")?;          // 第 2 层：解析错误
    Ok(config)
}

fn start_agent(config_path: &str) -> Result<()> {
    let config = load_agent_config(config_path)
        .context("failed to initialize agent")?;           // 第 1 层：业务错误
    Ok(())
}
```

**错误链输出示例：**

```
Error: failed to initialize agent

Caused by:
    0: failed to read config file
    1: No such file or directory (os error 2)
```

从上到下：业务含义 → 操作描述 → 系统根因。排查问题时从第一行知道"什么业务出错了"，从最后一行知道"底层原因是什么"。

**ZeroClaw 中的错误链：**

```rust
// src/memory/sqlite.rs —— 数据库操作的错误链
Connection::open(&path_buf)
    .context("SQLite failed to open database")?;
// 如果文件权限不足，错误链输出：
// Error: SQLite failed to open database
// Caused by: unable to open database file (os error 13: Permission denied)
```

[来源: reference/context7_anyhow_01.md, reference/source_error_handling_01.md]

**记住：** 每个 `.context()` 加一层洋葱皮。错误链 = 从业务到系统的完整追踪路径。

## 卡片 9：ZeroClaw 四大错误处理原则

**一句话：** 零 panic、安全优先、上下文丰富、Trait 一致——四条原则让 AI Agent 运行时在任何错误下都能优雅降级而非崩溃。

**四大原则详解：**

| 原则 | 含义 | 违反后果 |
|------|------|---------|
| **零 panic** | 生产代码不用 `unwrap()`，仅测试中使用 | Agent 进程终止，所有用户断线 |
| **安全优先** | 错误信息不泄露敏感数据（密钥、路径等） | 安全漏洞，密钥泄露 |
| **上下文丰富** | 每个错误都附带 `.context("描述")` | 排查困难，只看到底层错误 |
| **Trait 一致性** | 所有 trait 方法统一返回 `anyhow::Result<T>` | 接口混乱，实现者困惑 |

**源码示例——四原则的体现：**

```rust
// 原则 1：零 panic —— 用 ? 和组合子替代 unwrap
let command = args.get("command")
    .and_then(|v| v.as_str())
    .ok_or_else(|| anyhow!("Missing 'command' parameter"))?;
// 而不是：args["command"].as_str().unwrap()

// 原则 2：安全优先 —— 错误信息不暴露密钥内容
cipher.encrypt(&nonce, plaintext.as_bytes())
    .map_err(|e| anyhow!("Encryption failed: {e}"))?;
// 只说"加密失败"，不打印密钥或明文

// 原则 3：上下文丰富 —— 每层操作都加 context
Connection::open(&path_buf).context("SQLite failed to open database")?;

// 原则 4：Trait 一致性 —— 所有 trait 方法返回 anyhow::Result<T>
async fn execute(&self, args: Value) -> anyhow::Result<ToolResult>;
async fn store(&self, entry: MemoryEntry) -> anyhow::Result<()>;
async fn receive(&mut self) -> anyhow::Result<ChannelMessage>;
```

[来源: reference/source_error_handling_01.md]

**记住：** 四条原则的核心是一个字——**稳**。AI Agent 不能因为一个参数缺失就整个崩溃。

## 卡片 10：从 TypeScript 到 Rust 的思维转换

**一句话：** 从"错误是意外"转变为"错误是类型"——TypeScript 把错误当异常抛出，Rust 把错误当返回值传递。

**完整对照表：**

| 维度 | TypeScript 思维 | Rust 思维 |
|------|----------------|-----------|
| 错误本质 | 异常（运行时意外） | 类型（编译时约束） |
| 错误处理 | try/catch（可选的） | Result + ?（强制的） |
| 空值处理 | null/undefined（隐式） | Option（显式） |
| 错误传播 | throw 自动冒泡 | `?` 显式传播 |
| 错误类型 | catch(e) 的 e 是 unknown | Err(E) 的 E 有具体类型 |
| 不处理的后果 | 运行时崩溃 | 编译不过 |
| 错误上下文 | 手动拼 message | `.context("msg")` 链式 |
| 空值安全 | `?.` optional chaining | Option 的 `map`/`and_then` |
| 错误库 | 无标准方案 | anyhow（应用）+ thiserror（库） |
| 设计哲学 | 先跑起来再说 | 编译通过就稳了 |

**思维转换的关键时刻：**

```
第 1 天：为什么编译器不让我用这个值？→ 因为它是 Result，你还没处理错误
第 3 天：为什么到处都要写 ?         → 因为每个可能失败的操作都要显式传播
第 7 天：哦，编译通过就意味着错误都处理了 → 恭喜，你已经转变了思维
第 14 天：回去写 TS 觉得 try/catch 好不安全 → 你已经是 Rust 思维了
```

**最终心法：** Rust 的错误处理不是"更麻烦的 try/catch"，而是"把错误变成类型系统的一部分"。前期多花 10% 时间写 Result/Option/context，换来运行时零意外崩溃。ZeroClaw 作为 AI Agent 运行时，这个交换绝对值得。
