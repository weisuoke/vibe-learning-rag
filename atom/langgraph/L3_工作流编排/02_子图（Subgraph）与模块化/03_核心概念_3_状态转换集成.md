# 核心概念 3：状态转换集成

> 当父图和子图的 state schema 完全不同时，通过包装函数手动转换状态，实现松耦合的模块化集成

---

## 概述

**状态转换集成是什么？**

状态转换集成是通过包装函数（wrapper function）手动转换父子图之间不同 state schema 的集成方式。与共享状态键的"自动传递"不同，状态转换是一种"手动翻译"机制：你编写一个普通函数，负责把父图状态翻译成子图能理解的输入，调用子图执行，再把子图的输出翻译回父图能理解的状态更新。

**为什么需要状态转换？**

在实际项目中，子图往往是通用模块（如情感分析、文本摘要、数据清洗），它们的 schema 是独立设计的，不会为某个特定父图定制。这时父子图的字段名、结构可能完全不同，无法依赖共享键自动传递。状态转换就是解决这个问题的桥梁。

---

## 一句话定义

**状态转换集成是用包装函数将父图状态映射为子图输入、将子图输出映射回父图状态更新的手动集成方式。**

---

## 详细解释

### 适用场景

状态转换集成适用于以下情况：

1. **父图和子图的 schema 完全不同**：字段名不同、结构不同、甚至类型不同
2. **子图是通用模块**：被多个不同父图复用，不能为某个父图修改 schema
3. **需要对输入/输出做额外处理**：比如从父图状态中提取部分字段、对子图返回值做格式转换或过滤
4. **跨团队协作**：子图由另一个团队开发，schema 设计独立

### 与共享状态键的核心区别

共享状态键是**自动的**——LangGraph 框架根据同名字段自动传递数据。状态转换是**手动的**——你完全控制数据如何在父子图之间流动。

```
共享状态键：父图 State.foo → [自动传递] → 子图 State.foo
状态转换：  父图 State.foo → [包装函数手动映射] → 子图 State.bar
```

### 包装函数的职责

包装函数是状态转换的核心，它承担五步工作：

```
步骤1: 接收父图状态
步骤2: 从父图状态中提取数据，构造子图输入
步骤3: 调用 subgraph.invoke() 执行子图
步骤4: 从子图输出中提取数据，转换为父图状态更新
步骤5: 返回父图状态更新字典
```

用代码表示：

```python
def call_subgraph(state: ParentState) -> dict:
    # 步骤1-2: 接收父图状态，构造子图输入
    subgraph_input = {"bar": state["foo"]}

    # 步骤3: 调用子图
    subgraph_output = subgraph.invoke(subgraph_input)

    # 步骤4-5: 转换并返回父图状态更新
    return {"foo": subgraph_output["bar"]}
```

### 类比理解

**前端类比**：
- 状态转换就像 API 适配层（adapter layer）
- 前端组件需要的数据格式和后端返回的不同
- 你写一个适配函数把后端数据转换成组件 props
- 包装函数 = API adapter，父图 = 前端页面，子图 = 后端服务

**日常生活类比**：
- 状态转换就像翻译官
- 中国商人（父图）说中文，法国供应商（子图）说法语
- 翻译官（包装函数）把中文翻译成法语传给供应商，再把法语回复翻译成中文给商人
- 双方不需要学对方的语言，翻译官负责一切

## 何时选择状态转换

### 选择状态转换的信号

| 信号 | 说明 |
|------|------|
| 子图是通用模块 | 被多个不同父图复用，schema 不能为某个父图定制 |
| 父子图 schema 独立设计 | 字段名、结构、类型都不同 |
| 需要输入/输出预处理 | 从父图提取部分字段、对子图返回值做过滤或合并 |
| 跨团队开发 | 子图和父图由不同团队维护，schema 演进独立 |
| 多层嵌套场景 | parent → child → grandchild，每层有完全独立的状态 |

### 选择共享状态键的信号

| 信号 | 说明 |
|------|------|
| 父子图有自然共享的字段 | 比如都需要 `messages`、`user_id` |
| 子图是父图的"子流程" | 逻辑上是同一个工作流的一部分 |
| 希望最小化样板代码 | 不想写包装函数，让框架自动处理 |

## 完整代码示例

### 场景 1：简单状态转换（父图 foo → 子图 bar）

这是最基础的状态转换模式，父图有 `foo` 字段，子图有 `bar` 字段，两者完全不同。

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END


# ============================================================
# 第一步：定义父图和子图的状态（schema 完全不同）
# ============================================================

class ParentState(TypedDict):
    """父图状态：使用 foo 字段"""
    foo: str


class ChildState(TypedDict):
    """子图状态：使用 bar 字段，与父图无任何共享"""
    bar: str


# ============================================================
# 第二步：构建子图
# ============================================================

def child_node(state: ChildState) -> dict:
    """子图节点：对 bar 字段做处理"""
    return {"bar": state["bar"] + " -> processed by child"}


child_builder = StateGraph(ChildState)
child_builder.add_node("child_node", child_node)
child_builder.add_edge(START, "child_node")
child_builder.add_edge("child_node", END)
child_subgraph = child_builder.compile()


# ============================================================
# 第三步：编写包装函数（状态转换的核心）
# ============================================================

def call_child_subgraph(state: ParentState) -> dict:
    """包装函数：负责父图 ↔ 子图的状态转换

    职责：
    1. 从父图状态提取数据 → 构造子图输入
    2. 调用子图
    3. 从子图输出提取数据 → 构造父图状态更新
    """
    # 父图 foo → 子图 bar
    subgraph_input = {"bar": state["foo"]}

    # 调用子图执行
    subgraph_output = child_subgraph.invoke(subgraph_input)

    # 子图 bar → 父图 foo
    return {"foo": subgraph_output["bar"]}


# ============================================================
# 第四步：构建父图，将包装函数作为节点添加
# ============================================================

parent_builder = StateGraph(ParentState)
parent_builder.add_node("call_child", call_child_subgraph)  # 注意：添加的是包装函数，不是子图
parent_builder.add_edge(START, "call_child")
parent_builder.add_edge("call_child", END)
parent_graph = parent_builder.compile()


# ============================================================
# 第五步：运行
# ============================================================

result = parent_graph.invoke({"foo": "hello"})
print(result)
# 输出: {'foo': 'hello -> processed by child'}
```

**关键点**：

- `add_node("call_child", call_child_subgraph)` 添加的是包装函数，不是编译后的子图对象
- 包装函数的签名接收 `ParentState`，LangGraph 把它当作普通节点函数处理
- 子图的调用发生在包装函数内部，对父图来说是透明的

### 场景 2：多层嵌套（parent → child → grandchild）

支持多层嵌套，每层有独立的状态 schema，每层通过包装函数转换。

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END


# ============================================================
# 三层独立状态定义
# ============================================================

class GrandChildState(TypedDict):
    """最内层：使用 grandbaz 字段"""
    grandbaz: str


class ChildState(TypedDict):
    """中间层：使用 bar 字段"""
    bar: str


class ParentState(TypedDict):
    """最外层：使用 foo 字段"""
    foo: str


# ============================================================
# 第一层：构建 grandchild 子图
# ============================================================

def grandchild_node(state: GrandChildState) -> dict:
    return {"grandbaz": state["grandbaz"] + " -> grandchild处理"}


grandchild_builder = StateGraph(GrandChildState)
grandchild_builder.add_node("gc_node", grandchild_node)
grandchild_builder.add_edge(START, "gc_node")
grandchild_builder.add_edge("gc_node", END)
grandchild_subgraph = grandchild_builder.compile()


# ============================================================
# 第二层：构建 child 子图（内部调用 grandchild）
# ============================================================

def call_grandchild(state: ChildState) -> dict:
    """child → grandchild 的状态转换"""
    # child.bar → grandchild.grandbaz
    gc_output = grandchild_subgraph.invoke({"grandbaz": state["bar"]})
    # grandchild.grandbaz → child.bar
    return {"bar": gc_output["grandbaz"]}


def child_node(state: ChildState) -> dict:
    return {"bar": state["bar"] + " -> child处理"}


child_builder = StateGraph(ChildState)
child_builder.add_node("child_node", child_node)
child_builder.add_node("call_grandchild", call_grandchild)
child_builder.add_edge(START, "child_node")
child_builder.add_edge("child_node", "call_grandchild")
child_builder.add_edge("call_grandchild", END)
child_subgraph = child_builder.compile()


# ============================================================
# 第三层：构建 parent 图（内部调用 child）
# ============================================================

def call_child(state: ParentState) -> dict:
    """parent → child 的状态转换"""
    # parent.foo → child.bar
    child_output = child_subgraph.invoke({"bar": state["foo"]})
    # child.bar → parent.foo
    return {"foo": child_output["bar"]}


parent_builder = StateGraph(ParentState)
parent_builder.add_node("call_child", call_child)
parent_builder.add_edge(START, "call_child")
parent_builder.add_edge("call_child", END)
parent_graph = parent_builder.compile()


# ============================================================
# 运行：数据流经三层
# ============================================================

result = parent_graph.invoke({"foo": "原始数据"})
print(result)
# 输出: {'foo': '原始数据 -> child处理 -> grandchild处理'}
```

**数据流路径**：

```
parent.foo="原始数据"
  → [call_child 包装函数] → child.bar="原始数据"
    → [child_node] → child.bar="原始数据 -> child处理"
    → [call_grandchild 包装函数] → grandchild.grandbaz="原始数据 -> child处理"
      → [gc_node] → grandchild.grandbaz="原始数据 -> child处理 -> grandchild处理"
    → child.bar="原始数据 -> child处理 -> grandchild处理"
  → parent.foo="原始数据 -> child处理 -> grandchild处理"
```

**关键点**：

- 每层只能访问自己的 State：grandchild 不知道 parent 和 child 的存在
- 包装函数是层与层之间唯一的桥梁
- 嵌套深度没有硬性限制，但建议控制在 3 层以内

### 场景 3：对子图输出做额外处理（合并、过滤、转换）

实际项目中，包装函数不仅仅做简单的字段映射，还需要对子图的输出做额外处理。

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
import operator


# ============================================================
# 状态定义
# ============================================================

class ParentState(TypedDict):
    """父图状态：包含查询、结果列表、处理状态"""
    query: str
    results: Annotated[list[str], operator.add]  # 使用 reducer 累加
    status: str


class AnalyzerState(TypedDict):
    """分析子图状态：完全独立的 schema"""
    input_text: str
    analysis_result: str
    confidence: float


# ============================================================
# 构建分析子图
# ============================================================

def analyze_node(state: AnalyzerState) -> dict:
    """模拟文本分析"""
    text = state["input_text"]
    # 模拟分析逻辑
    result = f"分析结果: {text[:20]}..."
    confidence = 0.85 if len(text) > 10 else 0.3
    return {
        "analysis_result": result,
        "confidence": confidence,
    }


analyzer_builder = StateGraph(AnalyzerState)
analyzer_builder.add_node("analyze", analyze_node)
analyzer_builder.add_edge(START, "analyze")
analyzer_builder.add_edge("analyze", END)
analyzer_subgraph = analyzer_builder.compile()


# ============================================================
# 包装函数：带额外处理逻辑
# ============================================================

def call_analyzer_subgraph(state: ParentState) -> dict:
    """包装函数：不仅做映射，还做过滤和格式化

    额外处理：
    1. 构造子图输入时从 query 提取关键信息
    2. 对子图输出做置信度过滤
    3. 格式化结果后返回父图
    """
    # === 输入转换 ===
    subgraph_input = {
        "input_text": state["query"],
        "analysis_result": "",
        "confidence": 0.0,
    }

    # === 调用子图 ===
    output = analyzer_subgraph.invoke(subgraph_input)

    # === 输出处理：置信度过滤 ===
    if output["confidence"] < 0.5:
        # 置信度太低，标记为不可靠
        return {
            "results": [f"[低置信度] {output['analysis_result']}"],
            "status": "low_confidence",
        }

    # === 输出处理：格式化结果 ===
    formatted = f"[置信度:{output['confidence']:.0%}] {output['analysis_result']}"
    return {
        "results": [formatted],
        "status": "completed",
    }


# ============================================================
# 构建父图
# ============================================================

parent_builder = StateGraph(ParentState)
parent_builder.add_node("analyze", call_analyzer_subgraph)
parent_builder.add_edge(START, "analyze")
parent_builder.add_edge("analyze", END)
parent_graph = parent_builder.compile()


# ============================================================
# 运行
# ============================================================

result = parent_graph.invoke({
    "query": "LangGraph 子图的状态转换机制是什么？",
    "results": [],
    "status": "pending",
})
print(result)
# 输出示例:
# {
#     'query': 'LangGraph 子图的状态转换机制是什么？',
#     'results': ['[置信度:85%] 分析结果: LangGraph 子图的状态转换机...'],
#     'status': 'completed'
# }
```

**关键点**：

- 包装函数中可以做任何 Python 逻辑：条件判断、数据过滤、格式转换
- 子图返回值的防御性检查（如置信度过滤）应在包装函数中完成
- 使用 Reducer（如 `operator.add`）可以在父图中累积多次子图调用的结果

## 与共享状态键的对比表

| 维度 | 共享状态键 | 状态转换 |
|------|-----------|---------|
| **集成方式** | `add_node("name", compiled_subgraph)` | `add_node("name", wrapper_function)` |
| **添加的对象** | 编译后的子图对象 | 普通 Python 函数 |
| **状态传递** | 自动（框架根据同名键传递） | 手动（你写代码控制映射） |
| **schema 要求** | 父子图必须有同名共享键 | 父子图 schema 可以完全不同 |
| **耦合度** | 中等（需要约定共享键名） | 低（完全解耦） |
| **灵活性** | 低（只能传递同名键的值） | 高（可以做任意转换和处理） |
| **样板代码** | 少（无需包装函数） | 多（需要编写包装函数） |
| **子图复用性** | 一般（依赖特定键名） | 强（不依赖父图 schema） |
| **调试难度** | 低（数据流自动且透明） | 中（需要检查包装函数逻辑） |
| **流式输出** | 自动支持 `subgraphs=True` | 需要在包装函数中手动处理 |
| **适用场景** | 父子图是同一工作流的一部分 | 子图是通用模块，被多处复用 |
| **典型例子** | 对话流程中的子流程 | 通用的文本分析/情感分析模块 |

**选择建议**：

- 如果父子图本来就该共享某些字段（如 `messages`），用共享状态键
- 如果子图是独立设计的通用模块，或者父子图 schema 差异大，用状态转换
- 不确定时，优先尝试共享状态键（更简单），不满足再改为状态转换

## 最佳实践

### 1. 包装函数保持简单，只做转换不做业务逻辑

```python
# 正确：包装函数只负责状态映射
def call_analyzer(state: ParentState) -> dict:
    output = analyzer.invoke({"text": state["query"]})
    return {"result": output["analysis"]}


# 错误：在包装函数里写业务逻辑
def call_analyzer(state: ParentState) -> dict:
    # 不要在这里做复杂的业务处理
    cleaned = state["query"].strip().lower()
    tokens = cleaned.split()
    if len(tokens) > 100:
        cleaned = " ".join(tokens[:100])
    # 这些逻辑应该放在父图的独立节点中
    output = analyzer.invoke({"text": cleaned})
    return {"result": output["analysis"]}
```

**原则**：如果转换逻辑超过 5 行，考虑把预处理拆成父图的独立节点。

### 2. 命名清晰

```python
# 推荐：call_xxx_subgraph 或 invoke_xxx 命名模式
def call_sentiment_subgraph(state: ParentState) -> dict: ...
def call_summarizer_subgraph(state: ParentState) -> dict: ...
def invoke_rag_pipeline(state: ParentState) -> dict: ...

# 不推荐：模糊命名
def process(state: ParentState) -> dict: ...
def handle(state: ParentState) -> dict: ...
def step2(state: ParentState) -> dict: ...
```

### 3. 对子图返回值做防御性检查

```python
def call_subgraph_safe(state: ParentState) -> dict:
    """带防御性检查的包装函数"""
    try:
        output = subgraph.invoke({"bar": state["foo"]})
    except Exception as e:
        # 子图执行失败时返回安全的默认值
        return {"foo": state["foo"], "error": str(e)}

    # 检查返回值是否包含预期的键
    if "bar" not in output:
        return {"foo": state["foo"], "error": "子图未返回 bar 字段"}

    # 检查返回值类型
    if not isinstance(output["bar"], str):
        return {"foo": str(output["bar"])}

    return {"foo": output["bar"]}
```

### 4. 子图 invoke 使用完整的输入

```python
# 推荐：提供子图 schema 需要的所有字段
def call_subgraph(state: ParentState) -> dict:
    output = subgraph.invoke({
        "bar": state["foo"],
        "config": "default",  # 子图需要的其他字段也要提供
    })
    return {"foo": output["bar"]}


# 风险：只提供部分字段，可能导致子图内部 KeyError
def call_subgraph(state: ParentState) -> dict:
    output = subgraph.invoke({"bar": state["foo"]})  # 如果子图还需要 config 字段呢？
    return {"foo": output["bar"]}
```

### 5. 多层嵌套控制深度

```python
# 推荐：最多 3 层嵌套
# parent → child → grandchild（最多到这里）

# 不推荐：超过 3 层
# parent → child → grandchild → great_grandchild → ...
# 如果需要这么多层，考虑重新设计架构，把中间层合并
```

## 常见错误

### 错误 1：把编译后的子图直接添加但 schema 不兼容

```python
# 错误：父子图没有共享键，直接添加不会报错，但数据不会传递
class ParentState(TypedDict):
    foo: str

class ChildState(TypedDict):
    bar: str  # 与父图没有共享键

child = child_builder.compile()
parent_builder.add_node("child", child)  # 能添加，但 bar 字段不会收到 foo 的值
```

**解决**：使用包装函数做状态转换。

### 错误 2：包装函数忘记返回父图状态更新

```python
# 错误：没有返回值，父图状态不会更新
def call_subgraph(state: ParentState):
    subgraph.invoke({"bar": state["foo"]})
    # 忘记 return 了！


# 正确：必须返回父图状态更新字典
def call_subgraph(state: ParentState) -> dict:
    output = subgraph.invoke({"bar": state["foo"]})
    return {"foo": output["bar"]}
```

### 错误 3：在包装函数中修改原始状态

```python
# 错误：直接修改 state 对象
def call_subgraph(state: ParentState) -> dict:
    state["foo"] = "modified"  # 不要直接修改！
    output = subgraph.invoke({"bar": state["foo"]})
    return {"foo": output["bar"]}


# 正确：只通过返回值更新状态
def call_subgraph(state: ParentState) -> dict:
    output = subgraph.invoke({"bar": state["foo"]})
    return {"foo": output["bar"]}  # 通过返回值让框架更新状态
```

## 总结

| 要点 | 说明 |
|------|------|
| **本质** | 包装函数做父子图之间的"翻译官" |
| **核心操作** | 接收父图状态 → 构造子图输入 → invoke 子图 → 转换输出 → 返回父图更新 |
| **适用场景** | 父子图 schema 完全不同、子图是通用模块需要复用 |
| **与共享键区别** | 共享键是自动传递同名字段，状态转换是手动映射任意字段 |
| **最佳实践** | 包装函数只做转换、命名清晰、防御性检查、控制嵌套深度 |
| **注意事项** | 流式输出需要额外处理、不要在包装函数中写业务逻辑 |

---

**记住**：状态转换集成的核心价值是**解耦**——让子图不需要知道父图的存在，让父图不需要适配子图的 schema。包装函数是它们之间唯一的桥梁，保持这座桥简单清晰，是写好状态转换的关键。
