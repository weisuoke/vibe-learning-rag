# 07_工作流模式 - 核心概念3：并行模式（Parallelization）

> LangGraph L3_工作流编排 | 核心概念 3/6

---

## 一句话定义

**并行模式是将任务分发到多个节点同时执行，再汇聚结果的工作流模式，用"空间换时间"提升效率。**

---

## 模式原理

### 核心思想

当多个任务之间**没有依赖关系**时，可以同时执行它们，而不是排队等待。

```
顺序执行（3秒）：  A → B → C → 汇总
                   1s   1s   1s

并行执行（1秒）：  ┌→ A ─┐
                   ├→ B ─┤→ 汇总
                   └→ C ─┘
                    1s
```

### 两个关键动作

1. **Fan-out（扇出）**：一个节点触发多个并行节点
2. **Fan-in（扇入）**：多个并行节点汇聚到一个节点

### 前端类比

就像 `Promise.all()` —— 同时发起多个请求，等所有请求都返回后再处理结果：

```javascript
// 前端并行请求
const [users, orders, products] = await Promise.all([
  fetchUsers(),
  fetchOrders(),
  fetchProducts()
]);
```

### 日常生活类比

餐厅厨房：主厨同时安排三个厨师分别做凉菜、热菜、汤，全部做好后一起上桌。

---

## 核心特征

| 特征 | 说明 |
|------|------|
| **Fan-out/Fan-in** | 分发任务到多个节点，汇聚结果到一个节点 |
| **Reducer 聚合** | 通过 `Annotated[list, operator.add]` 合并并行结果 |
| **Superstep 并行** | 同一超步（superstep）内的节点自动并行执行 |
| **defer 同步** | `defer=True` 让节点等待所有并行分支完成后再执行 |

## LangGraph 实现

### 方式1：Graph API 基础并行（Fan-out/Fan-in）

```python
"""
并行模式实战：多维度文档分析
同时进行情感分析、关键词提取、内容摘要，最后汇总
"""
import operator
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


# ===== 1. 定义状态 =====
class State(TypedDict):
    text: str
    # operator.add 作为 reducer：新值会追加到列表，而不是覆盖
    results: Annotated[list, operator.add]
    summary: str


# ===== 2. 定义并行节点 =====
def analyze_sentiment(state: State):
    """情感分析节点"""
    text = state["text"]
    # 实际项目中这里调用 LLM
    sentiment = "positive" if "好" in text else "neutral"
    print(f"[情感分析] 结果: {sentiment}")
    return {"results": [{"type": "sentiment", "value": sentiment}]}


def extract_keywords(state: State):
    """关键词提取节点"""
    text = state["text"]
    # 简单模拟关键词提取
    keywords = [w for w in text.split() if len(w) > 2][:5]
    print(f"[关键词提取] 结果: {keywords}")
    return {"results": [{"type": "keywords", "value": keywords}]}


def generate_summary(state: State):
    """摘要生成节点"""
    text = state["text"]
    summary = text[:50] + "..." if len(text) > 50 else text
    print(f"[摘要生成] 结果: {summary}")
    return {"results": [{"type": "summary", "value": summary}]}


# ===== 3. 定义汇聚节点 =====
def aggregate(state: State):
    """汇总所有并行结果"""
    results = state["results"]
    report = "\n".join(
        f"- {r['type']}: {r['value']}" for r in results
    )
    print(f"[汇总] 收到 {len(results)} 个结果")
    return {"summary": f"分析报告:\n{report}"}


# ===== 4. 构建图 =====
builder = StateGraph(State)

# 添加节点
builder.add_node("sentiment", analyze_sentiment)
builder.add_node("keywords", extract_keywords)
builder.add_node("summarize", generate_summary)
builder.add_node("aggregate", aggregate)

# Fan-out：START 同时触发三个分析节点
builder.add_edge(START, "sentiment")
builder.add_edge(START, "keywords")
builder.add_edge(START, "summarize")

# Fan-in：三个分析节点都汇聚到 aggregate
builder.add_edge("sentiment", "aggregate")
builder.add_edge("keywords", "aggregate")
builder.add_edge("summarize", "aggregate")

builder.add_edge("aggregate", END)

graph = builder.compile()

# ===== 5. 执行 =====
result = graph.invoke({"text": "LangGraph 是一个很好的工作流框架"})
print(result["summary"])
```

**运行输出：**
```
[情感分析] 结果: positive
[关键词提取] 结果: ['LangGraph', '工作流框架']
[摘要生成] 结果: LangGraph 是一个很好的工作流框架
[汇总] 收到 3 个结果
分析报告:
- sentiment: positive
- keywords: ['LangGraph', '工作流框架']
- summary: LangGraph 是一个很好的工作流框架
```

> **关键点**：三个分析节点在同一个 superstep 内并行执行，执行顺序不确定，但 `aggregate` 一定在三个都完成后才执行。

### 方式2：不等长分支 + defer

当并行分支长度不同时，短分支会先到达汇聚点。使用 `defer=True` 确保等待所有分支完成：

```python
"""
不等长分支示例：
分支1: A → B → B2 → D（3步）
分支2: A → C → D（2步）
D 需要等待两个分支都完成
"""
import operator
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    aggregate: Annotated[list, operator.add]


def node_a(state: State):
    print(f'添加 "A" 到 {state["aggregate"]}')
    return {"aggregate": ["A"]}

def node_b(state: State):
    print(f'添加 "B" 到 {state["aggregate"]}')
    return {"aggregate": ["B"]}

def node_b2(state: State):
    print(f'添加 "B2" 到 {state["aggregate"]}')
    return {"aggregate": ["B2"]}

def node_c(state: State):
    print(f'添加 "C" 到 {state["aggregate"]}')
    return {"aggregate": ["C"]}

def node_d(state: State):
    print(f'添加 "D" 到 {state["aggregate"]}')
    return {"aggregate": ["D"]}


builder = StateGraph(State)
builder.add_node("a", node_a)
builder.add_node("b", node_b)
builder.add_node("b2", node_b2)
builder.add_node("c", node_c)
# defer=True：D 会等待所有到达它的分支都完成
builder.add_node("d", node_d, defer=True)

builder.add_edge(START, "a")
# 分支1：a → b → b2 → d（长分支）
builder.add_edge("a", "b")
builder.add_edge("b", "b2")
builder.add_edge("b2", "d")
# 分支2：a → c → d（短分支）
builder.add_edge("a", "c")
builder.add_edge("c", "d")
builder.add_edge("d", END)

graph = builder.compile()
result = graph.invoke({"aggregate": []})
print(f"最终结果: {result['aggregate']}")
```

**运行输出：**
```
添加 "A" 到 []
添加 "B" 到 ['A']
添加 "C" 到 ['A']
添加 "B2" 到 ['A', 'B', 'C']
添加 "D" 到 ['A', 'B', 'C', 'B2']
最终结果: ['A', 'B', 'C', 'B2', 'D']
```

> **没有 `defer=True` 时**，D 会在 C 完成后就执行（因为 C→D 的边已满足），此时 B2 还没完成。`defer=True` 确保 D 等待所有入边的分支都完成。

### 方式3：Functional API 并行

```python
"""
Functional API 并行模式
更 Pythonic 的写法，自动推断图结构
"""
from langgraph.func import task, entrypoint


@task
def analyze_sentiment_task(text: str) -> dict:
    """情感分析"""
    return {"sentiment": "positive" if "好" in text else "neutral"}


@task
def extract_keywords_task(text: str) -> dict:
    """关键词提取"""
    return {"keywords": text.split()[:3]}


@task
def summarize_task(text: str) -> dict:
    """摘要生成"""
    return {"summary": text[:50]}


@task
def aggregate_task(results: list[dict]) -> str:
    """汇总结果"""
    return "\n".join(f"- {k}: {v}" for r in results for k, v in r.items())


@entrypoint()
def parallel_analysis(text: str) -> str:
    # 同时启动三个任务（返回 Future）
    sentiment_fut = analyze_sentiment_task(text)
    keywords_fut = extract_keywords_task(text)
    summary_fut = summarize_task(text)

    # 等待所有结果（类似 Promise.all）
    results = [
        sentiment_fut.result(),
        keywords_fut.result(),
        summary_fut.result(),
    ]

    return aggregate_task(results).result()


# 调用
report = parallel_analysis.invoke("LangGraph 是一个很好的工作流框架")
print(report)
```

## Reducer 深入理解

Reducer 是并行模式的核心机制。没有 reducer，多个节点写同一个字段会互相覆盖。

```python
# ❌ 没有 reducer：后写入的覆盖先写入的
class BadState(TypedDict):
    result: str  # 只保留最后一个节点的结果

# ✅ 有 reducer：所有结果都保留
class GoodState(TypedDict):
    results: Annotated[list, operator.add]  # 追加合并
```

### 常用 Reducer

| Reducer | 效果 | 适用场景 |
|---------|------|---------|
| `operator.add` | 列表追加 | 收集多个结果 |
| 自定义函数 | 灵活合并 | 去重、排序、加权 |

```python
# 自定义 reducer：去重合并
def unique_merge(existing: list, new: list) -> list:
    return list(set(existing + new))

class State(TypedDict):
    keywords: Annotated[list, unique_merge]
```

## 适用场景

| 场景 | 并行内容 | 汇聚方式 |
|------|---------|---------|
| **多维度文档分析** | 情感 + 关键词 + 摘要 | 合并为分析报告 |
| **多模型投票** | 同一问题问 GPT/Claude/Gemini | 投票选最佳答案 |
| **RAG 多源检索** | 同时查多个向量库 | 合并 + 去重 + 重排 |
| **多语言翻译** | 同时翻译成多种语言 | 收集所有翻译结果 |
| **A/B 测试** | 同时运行两个 prompt 版本 | 对比评估 |

---

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| 大幅降低总延迟 | 状态合并需要 reducer 设计 |
| 充分利用计算资源 | 并行节点间不能通信 |
| 提高系统吞吐量 | 调试时执行顺序不确定 |
| 天然适合独立子任务 | 错误处理更复杂 |

## 在 RAG 中的应用

### 1. 多源并行检索

```python
builder.add_edge(START, "search_docs")      # 文档库
builder.add_edge(START, "search_faq")       # FAQ库
builder.add_edge(START, "search_history")   # 历史对话库
# 汇聚后重排序
builder.add_edge("search_docs", "rerank")
builder.add_edge("search_faq", "rerank")
builder.add_edge("search_history", "rerank")
```

### 2. 多角度问题分析

```python
# 同时从不同角度分析用户问题
builder.add_edge(START, "intent_analysis")    # 意图分析
builder.add_edge(START, "entity_extraction")  # 实体提取
builder.add_edge(START, "context_analysis")   # 上下文分析
```

## 最佳实践

1. **确保节点独立**：并行节点之间不应有数据依赖
2. **设计好 Reducer**：根据业务需求选择合适的合并策略
3. **使用 defer 处理不等长分支**：避免汇聚节点过早执行
4. **控制并行度**：过多并行节点可能导致 API 限流
5. **错误隔离**：一个并行节点失败不应影响其他节点

---

## 小结

并行模式的本质是**识别任务中的独立子任务，同时执行以换取更低的延迟**。在 LangGraph 中，通过 fan-out/fan-in 边结构 + reducer 聚合机制实现。选择并行模式的判断标准很简单：**子任务之间是否有依赖？没有就并行。**
