# 核心概念 2：短期记忆与对话历史

> Buffer / Window / Token 三种经典策略 + Checkpointer 现代方案

---

## 一句话定义

**短期记忆是 Agent 在单次对话中记住之前说过什么的能力，是实现多轮对话的基础，通过 Checkpointer 或 ConversationBuffer* 系列类实现。**

---

## 为什么 Agent 需要短期记忆？

LLM 本身是**无状态**的 —— 每次调用都是独立的：

```python
# 没有记忆的 Agent（每次都是独立调用）
response_1 = llm.invoke("我叫张三")        # → "你好张三！"
response_2 = llm.invoke("我叫什么名字？")   # → "我不知道你叫什么"  ← 忘了！

# 有记忆的 Agent（自动保存对话历史）
agent.invoke("我叫张三", config)            # → "你好张三！"
agent.invoke("我叫什么名字？", config)       # → "你叫张三"  ← 记住了！
```

**短期记忆 = 把历史消息一起发给 LLM，让它「看到」之前的对话。**

---

## 核心原理：消息列表就是记忆

```
没有记忆时，LLM 收到的消息：
┌──────────────────────────┐
│ System: 你是一个助手       │
│ Human: 我叫什么名字？      │  ← 只有当前问题
└──────────────────────────┘

有记忆时，LLM 收到的消息：
┌──────────────────────────┐
│ System: 你是一个助手       │
│ Human: 我叫张三           │  ← 历史消息
│ AI: 你好张三！             │  ← 历史消息
│ Human: 我叫什么名字？      │  ← 当前问题
└──────────────────────────┘
```

**问题**：对话越来越长，消息列表会超过 LLM 的 Context Window 限制。

**解决方案**：三种经典策略 + 现代 Checkpointer。

---

## 经典方案一：ConversationBufferMemory（全量存储）

### 原理

最简单的策略 —— **存储所有对话，不做任何裁剪**。

```
对话1: Human → AI
对话2: Human → AI
对话3: Human → AI
   ...全部保留...
对话N: Human → AI     ← 可能超出 Context Window！
```

### 代码示例

```python
from langchain.memory import ConversationBufferMemory

# 创建全量缓冲记忆
memory = ConversationBufferMemory(
    memory_key="chat_history",   # prompt 中的变量名
    return_messages=True,        # True: 返回消息对象列表
                                 # False: 返回格式化字符串
)

# 手动模拟对话
memory.save_context(
    {"input": "我叫张三"},
    {"output": "你好张三！很高兴认识你。"}
)
memory.save_context(
    {"input": "我喜欢吃披萨"},
    {"output": "披萨是很多人的最爱！你喜欢什么口味？"}
)

# 加载记忆
vars = memory.load_memory_variables({})
print(vars["chat_history"])
# [HumanMessage("我叫张三"), AIMessage("你好张三！..."),
#  HumanMessage("我喜欢吃披萨"), AIMessage("披萨是很多人的...")]
```

[来源: sourcecode/langchain/libs/langchain/langchain_classic/memory/buffer.py]

### 适用场景

- ✅ 短对话（< 20 轮）
- ✅ 原型开发 / 快速验证
- ❌ 长对话（会超出 Token 限制）
- ❌ 生产环境高并发

---

## 经典方案二：ConversationBufferWindowMemory（滑动窗口）

### 原理

只保留**最近 k 轮**对话，旧的自动丢弃：

```
k = 3（保留最近3轮）

对话1: Human → AI    ← 丢弃
对话2: Human → AI    ← 丢弃
对话3: Human → AI    ← 保留
对话4: Human → AI    ← 保留
对话5: Human → AI    ← 保留（最新）
```

### 代码示例

```python
from langchain.memory import ConversationBufferWindowMemory

memory = ConversationBufferWindowMemory(
    k=5,                          # 保留最近5轮对话
    memory_key="chat_history",
    return_messages=True,
)

# 模拟10轮对话
for i in range(10):
    memory.save_context(
        {"input": f"问题{i+1}"},
        {"output": f"回答{i+1}"}
    )

# 只有最近5轮
vars = memory.load_memory_variables({})
messages = vars["chat_history"]
print(f"记忆中的消息数: {len(messages)}")  # 10 (5轮 × 2条/轮)
print(messages[0].content)                 # "问题6"（前5轮已丢弃）
```

[来源: sourcecode/langchain/libs/langchain/langchain_classic/memory/buffer_window.py]

### k 值怎么选？

| 场景 | 推荐 k 值 | 理由 |
|------|-----------|------|
| 简单问答 | 3-5 | 只需最近上下文 |
| 客服对话 | 5-10 | 需要回溯问题细节 |
| 技术讨论 | 10-20 | 需要完整的技术上下文 |
| Agent 多步任务 | 15-30 | 需要看到完整的工具调用历史 |

### 适用场景

- ✅ 长对话（确保不超 Token）
- ✅ 对话焦点在近期
- ❌ 需要记住很久以前的信息
- ❌ 需要精确 Token 控制

## 经典方案三：ConversationTokenBufferMemory（Token 限制）

### 原理

不按轮数裁剪，而是按 **Token 总量** 裁剪 —— 更精确地控制上下文大小：

```
max_token_limit = 2000

消息1 (200 tokens) ← 丢弃（总量超限时）
消息2 (300 tokens) ← 丢弃
消息3 (400 tokens) ← 保留
消息4 (500 tokens) ← 保留
消息5 (600 tokens) ← 保留
                     合计: 1500 tokens < 2000 ✓
```

### 代码示例

```python
from langchain.memory import ConversationTokenBufferMemory
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4.1-mini")  # 需要 LLM 来计算 Token

memory = ConversationTokenBufferMemory(
    llm=llm,                      # 用于计算 Token 数
    max_token_limit=2000,          # 最大 Token 限制
    memory_key="chat_history",
    return_messages=True,
)

# 保存一些对话
memory.save_context(
    {"input": "请详细解释一下量子计算的原理..."  },  # 长消息
    {"output": "量子计算基于量子力学的叠加态和纠缠态..." }  # 长回复
)

# 当累计 Token 超过 2000 时，自动删除最旧的消息
vars = memory.load_memory_variables({})
```

[来源: sourcecode/langchain/libs/langchain/langchain_classic/memory/token_buffer.py]

### 适用场景

- ✅ 需要精确控制 Context Window 使用量
- ✅ 消息长度差异大的场景
- ❌ 不能容忍丢失任何历史信息

## 三种经典策略对比

| 维度 | Buffer | Window | Token Buffer |
|------|--------|--------|-------------|
| **裁剪方式** | 不裁剪 | 按轮数 | 按 Token |
| **需要 LLM** | 否 | 否 | 是（计算Token） |
| **精确度** | — | 粗粒度 | 细粒度 |
| **信息丢失** | 无（但会爆） | 丢旧轮 | 丢旧消息 |
| **适用** | 短对话 | 一般对话 | 精确控制 |
| **推荐度** | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |

## 现代方案：Checkpointer（推荐）

### 为什么 Checkpointer 更好？

经典方案的 3 个痛点 Checkpointer 全部解决：

| 痛点 | 经典方案 | Checkpointer |
|------|----------|--------------|
| 持久化 | 默认在内存，重启丢失 | 自动持久化到存储后端 |
| 多用户 | 需手动管理多个 Memory 实例 | thread_id 原生支持 |
| 状态回溯 | 不支持 | 可回到任意历史状态 |

### 基本用法

```python
from langchain.agents import create_agent
from langgraph.checkpoint.memory import InMemorySaver

# 创建带短期记忆的 Agent
agent = create_agent(
    model="gpt-4.1",
    tools=[],
    checkpointer=InMemorySaver(),  # ← 这就是短期记忆
)

# 使用 thread_id 标识对话
config = {"configurable": {"thread_id": "user-zhangsan"}}

# 多轮对话 — 同一 thread_id 自动保持上下文
result1 = agent.invoke(
    {"messages": [{"role": "user", "content": "我叫张三"}]},
    config
)
print(result1["messages"][-1].content)
# → "你好张三！很高兴认识你。"

result2 = agent.invoke(
    {"messages": [{"role": "user", "content": "我叫什么名字？"}]},
    config
)
print(result2["messages"][-1].content)
# → "你叫张三。"
```

### 多用户支持

```python
# 不同用户用不同 thread_id，互不干扰
config_user_a = {"configurable": {"thread_id": "user-A"}}
config_user_b = {"configurable": {"thread_id": "user-B"}}

agent.invoke({"messages": "我叫 Alice"}, config_user_a)
agent.invoke({"messages": "我叫 Bob"}, config_user_b)

# 各自独立的对话历史
agent.invoke({"messages": "我叫什么？"}, config_user_a)  # → "Alice"
agent.invoke({"messages": "我叫什么？"}, config_user_b)  # → "Bob"
```

### Checkpointer 如何控制记忆长度？

Checkpointer 本身**不裁剪消息**。需要配合 **Middleware** 实现：

```python
from langchain.agents import create_agent
from langchain.agents.middleware import SummarizationMiddleware, after_model
from langchain.messages import RemoveMessage
from langgraph.checkpoint.memory import InMemorySaver

# 方案 A：自动摘要（替代 ConversationSummaryBufferMemory）
agent = create_agent(
    model="gpt-4.1",
    tools=[],
    middleware=[
        SummarizationMiddleware(
            model="gpt-4.1-mini",
            trigger=("tokens", 4000),   # 超过 4000 token 触发摘要
            keep=("messages", 20)        # 保留最近 20 条
        )
    ],
    checkpointer=InMemorySaver(),
)

# 方案 B：手动删除旧消息（替代 ConversationBufferWindowMemory）
@after_model
def trim_old_messages(state, runtime):
    """保留最近 10 条消息"""
    messages = state["messages"]
    if len(messages) > 10:
        to_remove = messages[:-10]
        return {"messages": [RemoveMessage(id=m.id) for m in to_remove]}
    return None

agent = create_agent(
    model="gpt-4.1",
    tools=[],
    middleware=[trim_old_messages],
    checkpointer=InMemorySaver(),
)
```

## 生产环境的 Checkpointer 选型

| Checkpointer | 适用场景 | 特点 |
|---------------|----------|------|
| `InMemorySaver` | 开发/测试 | 零配置，重启丢失 |
| `SqliteSaver` | 轻量生产 | 单文件，简单部署 |
| `PostgresSaver` | 企业生产 | 高可用，支持并发 |
| `RedisSaver` | 高性能 | 亚毫秒延迟，支持 TTL |

```python
# 开发环境
from langgraph.checkpoint.memory import InMemorySaver
checkpointer = InMemorySaver()

# 生产环境（PostgreSQL）
from langgraph.checkpoint.postgres import PostgresSaver
checkpointer = PostgresSaver(conn_string="postgresql://user:pass@host/db")

# 高性能场景（Redis）
from langgraph.checkpoint.redis import RedisSaver
checkpointer = RedisSaver(url="redis://localhost:6379")
```

## 经典 vs 现代方案：迁移速查

| 经典 Memory 类 | 现代替代 | 迁移方式 |
|----------------|----------|----------|
| `ConversationBufferMemory` | `Checkpointer` | 直接替换 |
| `ConversationBufferWindowMemory` | `Checkpointer + trim middleware` | 加 middleware |
| `ConversationTokenBufferMemory` | `Checkpointer + SummarizationMiddleware` | 加 middleware |
| 手动 `memory_key` 配置 | 自动管理 | 不需要了 |
| 手动 `return_messages` | 始终消息对象 | 不需要了 |

| RAG 场景 | 短期记忆用途 | 推荐方案 |
|----------|-------------|----------|
| **多轮文档问答** | 记住用户之前问了什么 | Checkpointer |
| **追问细节** | 基于上轮回答深入 | Checkpointer |
| **上下文检索优化** | 用对话历史改写 query | Checkpointer + Query改写 |
| **Agent 工具链** | 记住中间步骤 | Checkpointer（自动） |

## 本节小结

> **短期记忆的本质就是把历史消息一起发给 LLM。**
> - 经典三策略：Buffer（全存）、Window（按轮裁）、Token（按量裁）
> - 现代方案：Checkpointer 自动管理 + Middleware 控制策略
> - **新项目推荐 Checkpointer**，简单、自动、支持持久化和多用户
