# 核心概念 5：推理策略对比与选型

> 三种推理策略各有所长——ReAct 文本推理可解释性强，Tool Calling 工具调用最可靠，XML 推理适合特定模型。选对策略，事半功倍。

---

## 三种推理策略概览

LangChain 支持三种主流的 Agent 推理策略，它们解决的是同一个问题：**让 LLM 能够分步推理并调用工具**，但实现方式截然不同。

```
┌─────────────────────────────────────────────────────────┐
│                  三种推理策略                              │
│                                                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐ │
│  │ ReAct 文本  │  │ Tool Calling│  │ XML 推理        │ │
│  │ 推理（经典）│  │ 工具调用    │  │ （特定模型）    │ │
│  │             │  │ （现代主流）│  │                 │ │
│  │ Thought:    │  │ tool_calls  │  │ <tool>xxx</tool>│ │
│  │ Action:     │  │ [{"name":   │  │ <tool_input>    │ │
│  │ Action Input│  │   "xxx"}]   │  │   xxx           │ │
│  └─────────────┘  └─────────────┘  │ </tool_input>   │ │
│                                     └─────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

---

## 策略 1: ReAct 文本推理（经典）

### 核心原理

LLM 通过 Prompt 约束，按 `Thought→Action→Action Input` 的文本格式输出推理过程。OutputParser 用正则表达式从文本中提取决策。

### 实现方式

```python
from langchain.agents import create_react_agent
from langchain_core.prompts import PromptTemplate

# ReAct 的 LCEL 链：
# RunnablePassthrough.assign(agent_scratchpad)
#   | prompt
#   | llm.bind(stop=["\nObservation"])  ← Stop Sequence 防幻觉
#   | ReActSingleInputOutputParser()

agent = create_react_agent(
    llm=llm,
    tools=tools,
    prompt=react_prompt  # 包含 Thought/Action/Observation 格式说明
)
```

### LLM 输出示例

```
Thought: 用户问北京天气，我需要调用天气工具
Action: weather_tool
Action Input: 北京
```

### 优势

| 优势 | 说明 |
|------|------|
| 可解释性强 | Thought 展示了完整推理过程，方便调试和审计 |
| 模型兼容性好 | 任何能生成文本的 LLM 都能用，包括开源小模型 |
| 灵活性高 | 可以自定义 Prompt 格式、Few-Shot 示例 |
| 教学价值 | 推理过程可视化，适合学习和理解 Agent 原理 |

### 劣势

| 劣势 | 说明 |
|------|------|
| 格式遵循不稳定 | LLM 可能不严格按格式输出，导致解析失败 |
| 不支持多工具并行 | 每次只能输出一个 Action |
| 需要 Stop Sequence | 必须防止 LLM 编造 Observation |
| Token 消耗较高 | Thought 文本占用额外 Token |

## 策略 2: Tool Calling 工具调用（现代主流）

### 核心原理

利用模型的**原生函数调用能力**（如 OpenAI Function Calling、Anthropic Tool Use）。工具信息通过 API 参数传递，模型直接输出结构化的 `tool_calls`，不需要文本格式约束。

### 实现方式

```python
from langchain.agents import create_tool_calling_agent
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# Tool Calling 的 Prompt 更简洁——不需要格式说明
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    ("human", "{input}"),
    MessagesPlaceholder("agent_scratchpad"),
])

agent = create_tool_calling_agent(
    llm=llm,          # 必须支持 bind_tools()
    tools=tools,
    prompt=prompt
)

# 内部实现：
# llm_with_tools = llm.bind_tools(tools)  ← 工具信息通过 API 传递
# prompt | llm_with_tools | ToolsAgentOutputParser()
```

### LLM 输出示例

```python
# 不是文本！而是结构化的 AIMessage
AIMessage(
    content="",  # 内容为空
    tool_calls=[
        {
            "name": "weather_tool",
            "args": {"city": "北京"},
            "id": "call_abc123"
        }
    ]
)
```

### Scratchpad 格式差异

```python
# ReAct 文本推理的 Scratchpad（字符串）：
scratchpad = """Thought: 我需要查天气
Action: weather_tool
Action Input: 北京
Observation: 5°C，多云
Thought: """

# Tool Calling 的 Scratchpad（消息列表）：
scratchpad = [
    AIMessage(content="", tool_calls=[{"name": "weather_tool", "args": {"city": "北京"}, "id": "call_123"}]),
    ToolMessage(content="5°C，多云", tool_call_id="call_123"),
]
# 注意：用的是 ToolMessage，不是 HumanMessage！
```

### 优势

| 优势 | 说明 |
|------|------|
| 最可靠 | 模型原生支持，不依赖文本格式解析 |
| 支持多工具并行 | 一次可以输出多个 tool_calls |
| 无需 Stop Sequence | 模型不会编造工具返回值 |
| Token 效率高 | 不需要 Thought 文本，节省 Token |
| 参数类型安全 | 模型按 Schema 生成参数，类型更准确 |

### 劣势

| 劣势 | 说明 |
|------|------|
| 模型要求高 | 必须支持函数调用（GPT-4、Claude、Gemini 等） |
| 推理过程不可见 | 默认不输出 Thought，调试困难 |
| 灵活性较低 | 工具格式由 API 规范决定，自定义空间小 |

### 混合模式：CoT + Tool Calling

想要 Tool Calling 的可靠性，又想要推理过程的可见性？可以在 system prompt 中要求 LLM 先思考再调用工具：

```python
prompt = ChatPromptTemplate.from_messages([
    ("system", """You are a helpful assistant.
Before calling any tool, always explain your reasoning in the response content.
Think step by step about what information you need and which tool to use."""),
    ("human", "{input}"),
    MessagesPlaceholder("agent_scratchpad"),
])

# 这样 LLM 会在 content 中写推理过程，同时在 tool_calls 中输出工具调用
# AIMessage(
#     content="用户问北京天气，我需要调用天气工具来获取实时数据。",
#     tool_calls=[{"name": "weather_tool", "args": {"city": "北京"}, "id": "..."}]
# )
```

## 策略 3: XML 推理（特定模型）

### 核心原理

使用 XML 标签格式输出推理和工具调用。特别适合 Claude 模型，因为 Claude 对 XML 标签的生成非常稳定。

### 实现方式

```python
from langchain.agents import create_xml_agent

agent = create_xml_agent(
    llm=llm,  # 推荐 Claude 模型
    tools=tools,
    prompt=xml_prompt
)
```

### LLM 输出示例

```xml
我需要查询北京的天气信息。

<tool>weather_tool</tool>
<tool_input>北京</tool_input>
```

### Observation 格式

```xml
<!-- 系统将工具返回值包装在 <observation> 标签中 -->
<observation>5°C，多云，北风3级</observation>
```

### 优势与劣势

| 维度 | 说明 |
|------|------|
| 优势 | Claude 对 XML 生成极其稳定，解析可靠 |
| 优势 | 标签有明确边界，不需要复杂正则 |
| 优势 | 推理过程可见（标签外的文本就是 Thought） |
| 劣势 | 主要适合 Claude，其他模型效果不佳 |
| 劣势 | 不支持多工具并行 |
| 劣势 | 随着 Claude 支持原生 Tool Use，XML 推理的必要性降低 |

## 三种策略全面对比

### 核心维度对比表

| 维度 | ReAct 文本推理 | Tool Calling | XML 推理 |
|------|---------------|-------------|----------|
| **实现方式** | Prompt 约束 + 正则解析 | API 原生支持 | Prompt 约束 + XML 解析 |
| **工具信息传递** | 在 Prompt 文本中 | 在 API 参数中 | 在 Prompt 文本中 |
| **输出格式** | `Action: xxx` | `tool_calls: [...]` | `<tool>xxx</tool>` |
| **解析方式** | 正则表达式 | 直接读取字段 | XML 标签匹配 |
| **可靠性** | 中 | 最高 | 高（Claude） |
| **可解释性** | 最高（Thought 可见） | 低（默认无 Thought） | 高（标签外文本） |
| **多工具并行** | 不支持 | 支持 | 不支持 |
| **模型要求** | 任何 LLM | 支持函数调用的模型 | 推荐 Claude |
| **Stop Sequence** | 需要 | 不需要 | 可选 |
| **Token 效率** | 较低 | 最高 | 中等 |
| **LangChain 函数** | `create_react_agent` | `create_tool_calling_agent` | `create_xml_agent` |

### Scratchpad 格式对比

```
ReAct 文本推理（字符串拼接）：
┌──────────────────────────────────────────┐
│ "Thought: 我需要查天气                    │
│  Action: weather_tool                     │
│  Action Input: 北京                       │
│  Observation: 5°C                         │
│  Thought: "                               │
└──────────────────────────────────────────┘

Tool Calling（消息列表）：
┌──────────────────────────────────────────┐
│ [                                         │
│   AIMessage(tool_calls=[{...}]),          │
│   ToolMessage(content="5°C", id="..."),   │
│ ]                                         │
└──────────────────────────────────────────┘

XML 推理（XML 标签字符串）：
┌──────────────────────────────────────────┐
│ "<tool>weather_tool</tool>                │
│  <tool_input>北京</tool_input>            │
│  <observation>5°C</observation>"          │
└──────────────────────────────────────────┘
```

## 选型决策树

```
你的场景需要显式推理过程（可解释性）？
│
├── 是 → 你的模型支持原生函数调用？
│   │
│   ├── 是 → 【推荐】Tool Calling + System Prompt 要求 CoT
│   │         （混合模式：可靠性 + 可解释性兼得）
│   │
│   └── 否 → 【推荐】ReAct 文本推理
│             （经典方案，任何模型都能用）
│
└── 否 → 你的模型支持原生函数调用？
    │
    ├── 是 → 【强烈推荐】Tool Calling
    │         （最可靠、最高效、支持多工具并行）
    │
    └── 否 → 你的模型是 Claude？
        │
        ├── 是 → 【推荐】XML 推理
        │         （Claude 对 XML 最稳定）
        │
        └── 否 → 模型擅长生成 JSON？
            │
            ├── 是 → ReAct JSON 推理
            │
            └── 否 → ReAct 文本推理（兜底方案）
```

### 快速选型表

| 你的情况 | 推荐策略 | 原因 |
|----------|----------|------|
| 用 GPT-4 / GPT-4o | Tool Calling | 原生支持，最可靠 |
| 用 Claude 3.5+ | Tool Calling | Claude 也支持原生 Tool Use |
| 用 Claude 但需要可解释性 | XML 推理 或 Tool Calling + CoT | Claude 对 XML 极稳定 |
| 用开源模型（Llama、Qwen） | ReAct 文本推理 | 大多数开源模型不支持函数调用 |
| 需要调试推理过程 | ReAct 文本推理 | Thought 完全可见 |
| 需要多工具并行 | Tool Calling | 唯一支持多 tool_calls 的方案 |
| 教学/演示场景 | ReAct 文本推理 | 推理过程最直观 |

## 经典 AgentExecutor vs LangGraph

除了推理策略的选择，还有一个重要的架构选择：用经典的 AgentExecutor 还是新的 LangGraph？

### 架构对比

```
经典 AgentExecutor：
┌─────────────────────────────────────┐
│  while not finished:                 │
│    action = agent.plan(steps)        │
│    result = execute_tool(action)     │
│    steps.append((action, result))    │
│  return final_answer                 │
└─────────────────────────────────────┘
简单的 while 循环，线性执行

LangGraph ReAct Agent：
┌─────────────────────────────────────┐
│  StateGraph:                         │
│    agent_node → should_continue?     │
│      ├── has tool_calls → tool_node  │
│      └── no tool_calls → END         │
│    tool_node → agent_node            │
└─────────────────────────────────────┘
状态图，支持条件路由和并行执行
```

### 对比表

| 维度 | AgentExecutor | LangGraph |
|------|--------------|-----------|
| 复杂度 | 简单，开箱即用 | 较复杂，需要定义状态图 |
| 灵活性 | 有限，固定循环模式 | 极高，自定义状态转换 |
| 并行执行 | 不支持 | 支持（多节点并行） |
| 人机交互 | 有限（handle_parsing_errors） | 原生支持（interrupt_before） |
| 状态管理 | intermediate_steps 列表 | MessagesState 状态对象 |
| 持久化 | 不支持 | 原生支持（checkpointer） |
| 推荐场景 | 简单 Agent、快速原型 | 复杂 Agent、生产环境 |
| 官方态度 | 维护中，不再新增功能 | 主推方向 |

### LangGraph 中的 ReAct 模式

```python
from langgraph.prebuilt import create_react_agent

# LangGraph 版本的 ReAct Agent
# 内部使用 Tool Calling（不是文本推理）
agent = create_react_agent(
    model=llm,
    tools=tools,
    # 可选：添加 system prompt
    prompt="You are a helpful assistant."
)

# 核心模式：
# LLM 节点 → 检查 tool_calls → 有则执行工具 → 回到 LLM 节点
# 条件路由：has tool_calls → tool_node, else → END
```

**注意命名混淆：** LangGraph 的 `create_react_agent` 虽然叫 "ReAct"，但内部使用的是 **Tool Calling** 模式（原生 tool_calls），不是经典的文本 Thought/Action 格式。名字中的 "ReAct" 指的是"推理+行动"的理念，而非具体的文本格式。

## 2025-2026 趋势分析

### 趋势 1: 函数调用成为主流

原生函数调用（Tool Calling）正在全面替代文本推理：
- GPT-4、Claude 3.5+、Gemini 都支持原生函数调用
- 开源模型（Llama 3、Qwen 2）也开始支持
- LangChain 官方推荐 `create_tool_calling_agent` 作为默认选择

### 趋势 2: 混合模式兴起

CoT + 函数调用的混合模式越来越流行：
- GPT-4.1 的 system prompt 最佳实践建议"先思考再调用工具"
- 兼顾可靠性（原生函数调用）和可解释性（CoT 推理过程）
- 不需要额外的 OutputParser，推理过程在 content 中，工具调用在 tool_calls 中

### 趋势 3: LangGraph 主导复杂场景

复杂推理场景正在从 AgentExecutor 迁移到 LangGraph：
- 多 Agent 协作需要状态图编排
- 人机交互（Human-in-the-loop）需要中断和恢复
- 生产环境需要持久化和可观测性

### 趋势 4: ReAct 文本推理的定位

ReAct 文本推理不会消失，但定位在变化：
- 从"主流方案"变为"特定场景方案"
- 在需要可解释性、审计追踪的场景中仍有价值
- 在教学和调试场景中不可替代
- 在不支持函数调用的开源模型上仍是首选

### 趋势 5: 评估体系成熟

Agent 推理质量的评估工具越来越完善：
- `agentevals`：评估 Agent 的工具调用准确率
- `openevals`：评估推理过程的质量
- 标准化的 benchmark 让策略选型有据可依

## 实战：三种策略的代码对比

### ReAct 文本推理

```python
from langchain.agents import create_react_agent, AgentExecutor
from langchain_core.prompts import PromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

@tool
def search(query: str) -> str:
    """搜索互联网获取信息"""
    return f"搜索结果：关于 '{query}' 的信息..."

llm = ChatOpenAI(model="gpt-4o")

# ReAct Prompt（简化版）
react_prompt = PromptTemplate.from_template("""Answer the following questions using the available tools.

Tools:
{tools}

Format:
Thought: your reasoning
Action: tool name, one of [{tool_names}]
Action Input: input for the tool
Observation: tool result
... (repeat as needed)
Thought: I now know the final answer
Final Answer: your answer

Question: {input}
Thought:{agent_scratchpad}""")

agent = create_react_agent(llm, [search], react_prompt)
executor = AgentExecutor(agent=agent, tools=[search], verbose=True)
```

### Tool Calling 工具调用

```python
from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    ("human", "{input}"),
    MessagesPlaceholder("agent_scratchpad"),
])

agent = create_tool_calling_agent(llm, [search], prompt)
executor = AgentExecutor(agent=agent, tools=[search], verbose=True)
```

### XML 推理

```python
from langchain.agents import create_xml_agent, AgentExecutor

xml_prompt = PromptTemplate.from_template("""You are a helpful assistant. Use these tools:

{tools}

To use a tool:
<tool>tool_name</tool>
<tool_input>input</tool_input>

When done:
<final_answer>your answer</final_answer>

Question: {input}
{agent_scratchpad}""")

agent = create_xml_agent(llm, [search], xml_prompt)
executor = AgentExecutor(agent=agent, tools=[search], verbose=True)
```

## 小结

| 要点 | 内容 |
|------|------|
| 三种策略 | ReAct 文本推理、Tool Calling 工具调用、XML 推理 |
| 2026 主流 | Tool Calling（原生函数调用）是默认推荐 |
| ReAct 价值 | 可解释性、教学、开源模型场景 |
| 混合模式 | CoT + Tool Calling 兼顾可靠性和可解释性 |
| 架构选择 | 简单场景用 AgentExecutor，复杂场景用 LangGraph |
| 选型原则 | 优先 Tool Calling，需要可解释性时加 CoT，不支持函数调用时用 ReAct |
