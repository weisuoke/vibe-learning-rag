# Milvus架构概览 - 最小可用

## 4.1 整体架构：存算分离的三层设计

**核心理念：** Milvus 2.6 采用云原生的存算分离架构，将系统分为三层

```
┌─────────────────────────────────────────────────────────────┐
│                    Access Layer (接入层)                      │
│                      SDK / RESTful API                       │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                  Coordinator Layer (协调层)                   │
│   Root Coord  │  Data Coord  │  Query Coord  │ (可合并)      │
│   (元数据管理)  │  (数据协调)   │  (查询协调)   │               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    Worker Layer (工作层)                      │
│  Query Node  │  Data Node  │  Index Node  │ Streaming Node  │
│  (查询执行)   │  (数据写入)  │  (索引构建)   │  (流式处理)      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                   Storage Layer (存储层)                      │
│   Meta Storage (etcd)  │  Object Storage (S3/MinIO)         │
│   WAL (Woodpecker)     │  Message Queue (Pulsar/Kafka)      │
└─────────────────────────────────────────────────────────────┘
```

**为什么这样设计？**
- **存算分离**：计算节点无状态，可以独立扩缩容
- **微服务架构**：每个组件职责单一，故障隔离
- **云原生**：充分利用云存储的低成本和高可用性

**在 RAG 系统中的价值：**
- 查询高峰时可以快速扩展 Query Node
- 数据写入高峰时可以独立扩展 Data Node
- 存储成本低（使用对象存储）

## 4.2 Streaming Node：2.6 核心创新

**一句话：** Streaming Node 是 Milvus 2.6 的核心组件，统一处理所有流式数据操作

```python
# Streaming Node 的作用
"""
传统架构（2.5 及之前）：
  写入请求 → Data Node → Pulsar/Kafka → Query Node
  问题：依赖外部消息队列，增加延迟和成本

2.6 架构：
  写入请求 → Streaming Node → Query Node
  优势：内置流式处理，减少依赖，降低延迟
"""
```

**核心特性：**
1. **统一流式处理**：所有数据流转都通过 Streaming Node
2. **零磁盘架构**：配合 Woodpecker WAL，不需要本地磁盘
3. **降低成本**：减少对外部消息队列的依赖

**实际应用场景：**
- 实时数据写入（如用户反馈实时入库）
- 增量索引更新（新数据立即可搜索）
- 多租户数据隔离（每个租户独立流）

---

## 4.3 Woodpecker WAL：零磁盘架构的基石

**一句话：** Woodpecker WAL 是 Milvus 2.6 的预写日志系统，实现零磁盘架构

```
传统 WAL（Write-Ahead Log）：
  写入请求 → 本地磁盘 WAL → 内存 → 对象存储
  问题：需要本地磁盘，增加成本和复杂度

Woodpecker WAL（2.6）：
  写入请求 → 对象存储 WAL → 内存 → 对象存储
  优势：完全无磁盘，降低成本，简化部署
```

**核心优势：**
1. **零磁盘**：不需要本地磁盘，降低成本
2. **高可用**：WAL 存储在对象存储，天然高可用
3. **搜索新鲜度优化**：新数据更快可见

**在 RAG 系统中的价值：**
- 降低部署成本（不需要高性能本地磁盘）
- 提高数据可靠性（对象存储天然多副本）
- 加快数据可见性（新文档更快可搜索）

## 4.4 数据流转过程：从写入到检索

**完整流程：**

```
1. 数据写入流程：
   Client → Proxy → Streaming Node → Data Node → Object Storage
                                   ↓
                              Woodpecker WAL

2. 索引构建流程：
   Data Coord → Index Node → Object Storage (索引文件)

3. 数据加载流程：
   Query Coord → Query Node → 从 Object Storage 加载数据和索引到内存

4. 查询流程：
   Client → Proxy → Query Node (内存检索) → 返回结果
```

**关键理解：**
- **写入路径**：数据先写入 Streaming Node，再持久化到对象存储
- **查询路径**：Query Node 从内存中检索，不访问磁盘
- **索引构建**：异步进行，不阻塞写入和查询

## 4.5 Coordinator 层：大脑中枢

**三个核心 Coordinator：**

| Coordinator | 职责 | 类比 |
|-------------|------|------|
| **Root Coord** | 管理 Collection、Partition 等元数据 | 图书馆馆长（管理书架布局） |
| **Data Coord** | 协调数据写入、Segment 管理、索引构建 | 仓库管理员（管理货物入库） |
| **Query Coord** | 协调查询任务、负载均衡、数据加载 | 调度中心（分配查询任务） |

**2.6 新特性：Coord Merge**
- 小规模部署时，可以将三个 Coordinator 合并为一个进程
- 降低资源消耗，简化部署

**在 RAG 系统中的作用：**
- Root Coord：管理多个知识库（Collection）
- Data Coord：协调文档入库和索引构建
- Query Coord：分配查询任务到不同的 Query Node
