# 多步推理与规划 - 30字核心

> **多步推理与规划是让 Agent 先制定完整计划再分步执行的架构模式，是构建复杂任务 Agent 的核心能力。**

---

## 关键词解析

| 关键词 | 含义 |
|--------|------|
| **先制定完整计划** | 与 ReAct 的"走一步看一步"不同，Plan-and-Execute 模式要求 Agent 在执行前先用 LLM 生成包含所有子任务的完整计划（Task List），建立全局视野 |
| **分步执行** | 计划生成后，Executor 逐步（或并行）执行每个子任务。子任务的执行可以使用更小的模型或专用工具，无需每步都调用大模型，降低成本提升效率 |
| **架构模式** | 这不是一个单一技巧，而是一整套架构：Planner（规划器）+ Executor（执行器）+ Replanner（重规划器）三组件协作，并支持动态调整和自我反思 |
| **复杂任务** | 简单问答用 ReAct 足够，但研究报告、多源分析、多步检索等复杂任务，需要规划能力才能可靠完成。规划是 Agent 从"玩具"到"生产力工具"的分水岭 |

---

## 为什么这30字重要

### ReAct 的根本局限

ReAct（Thought→Action→Observation 循环）让 Agent 能够逐步推理，但它有两个致命问题：

```
问题1：每步都调大模型 → 成本高、速度慢
问题2：只看当前步    → 缺乏全局视野，容易走弯路

用户: "调研并对比 3 个数据库的性能"

ReAct 方式:
  Thought: 我先查第一个数据库吧 → Action → Observation
  Thought: 嗯...再查第二个       → Action → Observation
  Thought: 还有第三个            → Action → Observation
  Thought: 等等，我忘了查性能指标了，重来... ❌（浪费了前面的努力）

Plan-and-Execute 方式:
  Plan: [查DB1性能, 查DB2性能, 查DB3性能, 统一对比]
  Execute: 高效执行每一步 ✅（不会遗漏，不会走弯路）
```

### 规划给 Agent 带来的三大能力

1. **全局视野**：看到所有需要完成的步骤，避免遗漏和重复
2. **成本优化**：规划只调用一次大模型，执行可用轻量模型
3. **可靠性**：Re-planning 机制在出现意外时动态调整，不会一条道走到黑

---

## 核心公式

```
多步推理与规划 = Planner（规划） + Executor（执行） + Replanner（调整）

具体展开：
Planner    → LLM 生成完整任务计划（子目标列表）
Executor   → 逐步执行每个子目标（可用小模型/工具）
Replanner  → 评估执行结果，决定：继续执行 or 调整计划 or 输出答案
```

## 与相关知识点的关系

| 知识点 | 关系 |
|--------|------|
| **04_Agent推理与决策 (ReAct)** | ReAct 是"逐步推理"，Plan-and-Execute 是"先规划后执行"。两者互补——规划内的子任务执行仍可使用 ReAct |
| **02_AgentExecutor执行循环** | AgentExecutor 提供循环执行框架，Plan-and-Execute 在此基础上增加了"规划层"和"重规划层" |
| **07_Agent Memory集成** | 规划状态需要持久化存储（checkpointer），失败后才能恢复。长期记忆（store）可以存储历史规划经验 |
| **03_Tools与函数调用** | 规划中的每个子任务最终通过 Tool 执行。规划决定"做什么"，Tools 负责"怎么做" |

## 记忆技巧

### 口诀

**"先想后做，做完回看，不对就改"**

- **先想**：Planner 生成计划
- **后做**：Executor 分步执行
- **回看**：Replanner 评估结果
- **不对就改**：Re-planning 动态调整

### 场景联想

想象你在组织一场团建活动：
- **ReAct 方式**：到了再说，先去一个地方，不好玩再换 → 低效混乱
- **Plan-and-Execute 方式**：先列出行程（上午登山→中午野餐→下午团队游戏），然后按计划执行，下雨了就调整计划 → 高效可靠
