## 为什么 Agent 需要记忆？

### 无记忆 Agent vs 有记忆 Agent

想象你去一家餐厅，服务员每次来都不记得你之前说了什么：

| 场景 | 无记忆 Agent ❌ | 有记忆 Agent ✅ |
|------|----------------|----------------|
| 第1轮 | 用户：我叫小明 → Agent：你好小明！ | 用户：我叫小明 → Agent：你好小明！ |
| 第2轮 | 用户：我叫什么？→ Agent：我不知道 | 用户：我叫什么？→ Agent：你叫小明！ |
| 第3轮 | 用户：帮我查订单 → Agent：什么订单？ | 用户：帮我查订单 → Agent：小明，我来帮你查 |

**核心问题**：LLM 本身是**无状态**的——每次调用都是一张白纸。Agent 要想"记住"之前的对话，就必须有一套记忆系统，把历史信息主动注入到每次请求中。

### 记忆对 Agent 的三大价值

1. **对话连贯性**：多轮对话不再"失忆"，用户体验质的飞跃
2. **上下文感知**：Agent 能结合历史信息做出更准确的决策和工具调用
3. **个性化服务**：记住用户偏好，跨会话提供定制化体验

## 2025-2026 架构演进

Agent Memory 经历了一次重大架构演进，理解这个演进是掌握本知识点的关键：

### 经典方式（LangChain 0.1-0.2）

```python
# ❌ 旧方式：使用独立的 Memory 类
from langchain.memory import ConversationBufferMemory

memory = ConversationBufferMemory()
agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    memory=memory,        # Memory 作为参数传入
    verbose=True
)
```

**特点**：Memory 是独立对象，通过 `memory=` 参数挂载到 AgentExecutor。

### 现代方式（LangChain 0.3+ / LangGraph）

```python
# ✅ 新方式：使用 checkpointer + store + middleware
from langchain.agents import create_agent
from langchain.agents.middleware import SummarizationMiddleware
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.store.memory import InMemoryStore

agent = create_agent(
    "gpt-4.1",
    tools=tools,
    middleware=[
        SummarizationMiddleware(model="gpt-4.1-mini", trigger=("tokens", 4000))
    ],
    checkpointer=InMemorySaver(),   # 短期记忆
    store=InMemoryStore(),           # 长期记忆
)
```

**特点**：Memory 通过 checkpointer（短期）+ store（长期）+ middleware（策略）三位一体管理。

### 演进对照表

| 维度 | 经典方式 | 现代方式 |
|------|---------|---------|
| 短期记忆 | `ConversationBufferMemory` | `Checkpointer` |
| 记忆压缩 | `ConversationSummaryMemory` | `SummarizationMiddleware` |
| 长期记忆 | `VectorStoreRetrieverMemory` | `Store` + 语义搜索 |
| 集成方式 | `AgentExecutor(memory=...)` | `create_agent(checkpointer=..., store=...)` |
| 状态管理 | Memory 类自行管理 | LangGraph 框架统一管理 |
| 持久化 | 需自行实现 | 内置多种后端（SQLite/Postgres/Redis） |

> **学习建议**：本知识点**两种方式都会讲**。经典方式帮你理解原理，现代方式是生产推荐。

## 记忆类型全景

```
Agent 记忆体系
│
├── 短期记忆（对话历史）
│   │  作用：记住本次会话的对话内容
│   │  生命周期：单次会话（session/thread）
│   │
│   ├── 经典实现                           现代实现
│   │   ├─ ConversationBufferMemory       ├─ Checkpointer（自动保存全量）
│   │   ├─ ConversationBufferWindowMemory ├─ RemoveMessage（手动裁剪）
│   │   └─ ConversationTokenBufferMemory  └─ SummarizationMiddleware（智能压缩）
│   │
│   └── 核心问题：对话越长，Token 消耗越大 → 需要压缩策略
│
├── 工作记忆（思考过程）
│   │  作用：Agent 的"草稿纸"，记录中间推理步骤
│   │  生命周期：单次任务执行
│   │
│   ├── Agent Scratchpad（中间步骤）
│   ├── Intermediate Steps（工具调用记录）
│   └── 上下文工程（Context Engineering）
│   │
│   └── 核心问题：推理步骤太多会撑爆 Context Window
│
└── 长期记忆（跨会话）
    │  作用：记住用户跨多次会话的信息
    │  生命周期：永久 / 用户级
    │
    ├── 经典实现                           现代实现
    │   ├─ VectorStoreRetrieverMemory     ├─ Store（键值存储）
    │   └─ ConversationEntityMemory       ├─ Store + 语义搜索（向量索引）
    │                                     └─ LangMem（自动记忆提取）
    │
    └── 核心问题：存什么？怎么检索？如何防过期？
```

## 与 RAG 和 AI Agent 开发的关系

Agent Memory 与 RAG 系统有天然的联系：

### 记忆即检索

```
RAG 系统:   用户问题 → 检索知识库 → 注入上下文 → LLM 生成答案
Agent Memory: 用户问题 → 加载记忆 → 注入上下文 → LLM 推理决策
```

本质上，Memory 的"加载"动作就是一种特殊的"检索"——从记忆存储中检索相关的历史信息。

### 长期记忆 ≈ 个人知识库

长期记忆的 Store + 语义搜索，本质上就是一个**用户私有的 RAG 系统**：
- 存储用户偏好和历史 → 相当于"用户文档"
- 语义搜索匹配相关记忆 → 相当于 RAG 的向量检索
- 注入到 System Prompt → 相当于 RAG 的上下文注入

### 实际应用场景

| 场景 | Memory 需求 | RAG 关联 |
|------|------------|---------|
| 智能客服 | 记住用户诉求和上下文 | 结合知识库检索产品信息 |
| 文档问答 | 记住用户已问过的问题 | RAG 检索文档 + Memory 避免重复 |
| 个人助手 | 记住用户偏好和习惯 | 长期记忆 ≈ 个人知识库 |
| 代码助手 | 记住项目上下文和代码风格 | RAG 检索代码库 + Memory 追踪修改 |
