## 4.1 用 TypedDict 或 Pydantic BaseModel 定义状态，坚持可序列化类型

**一句话：** 状态中的每个字段都必须能被序列化为字节——如果不能序列化，就不能持久化

这是持久化的第一条铁律。LangGraph 的 `JsonPlusSerializer` 能处理大多数 Python 类型，但不是所有类型。

### 安全的类型（直接用）

```python
from typing import Annotated
from typing_extensions import TypedDict
from operator import add
from pydantic import BaseModel

# === TypedDict 方式（推荐，轻量） ===
class State(TypedDict):
    # 基础类型 — 完全安全
    query: str
    count: int
    score: float
    is_done: bool

    # 容器类型 — 完全安全（内部元素也要可序列化）
    messages: Annotated[list[str], add]
    metadata: dict[str, str]
    tags: set[str]

    # 特殊类型 — JsonPlusSerializer 内置支持
    # UUID, datetime, Decimal, Path, Enum 等都可以

# === Pydantic 方式（需要验证时用） ===
class PydanticState(BaseModel):
    query: str
    score: float = 0.0
    messages: list[str] = []

    # Pydantic v2 模型会被 JsonPlusSerializer 正确处理
    # 通过扩展类型 5 编码
```

### 危险的类型（避免使用）

```python
# ❌ 不可序列化的类型 — 不要放在状态中！

class BadState(TypedDict):
    # ❌ 数据库连接 — 不可序列化
    # db_conn: psycopg2.connection

    # ❌ 文件句柄 — 不可序列化
    # file_handle: io.TextIOWrapper

    # ❌ Lambda 函数 — 不可序列化
    # callback: Callable

    # ❌ 自定义类（没有序列化支持）— 默认不可序列化
    # custom_obj: MyCustomClass

    # ❌ 线程锁 — 不可序列化
    # lock: threading.Lock

    pass

# ✅ 正确做法：只存储数据，不存储连接/句柄/函数
class GoodState(TypedDict):
    db_url: str              # 存 URL，不存连接
    file_path: str           # 存路径，不存句柄
    callback_name: str       # 存函数名，不存函数
    custom_data: dict        # 存数据字典，不存自定义对象
```

### 快速验证：你的状态能序列化吗？

```python
from langgraph.checkpoint.serde.jsonplus import JsonPlusSerializer

serde = JsonPlusSerializer()

# 测试你的状态值能否被序列化
test_state = {
    "query": "hello",
    "messages": ["hi", "there"],
    "score": 0.95,
}

try:
    type_tag, data = serde.dumps_typed(test_state)
    restored = serde.loads_typed((type_tag, data))
    print(f"序列化成功！类型标签: {type_tag}, 大小: {len(data)} 字节")
    print(f"恢复后: {restored}")
    assert restored == test_state, "数据不一致！"
    print("往返验证通过")
except Exception as e:
    print(f"序列化失败: {e}")
```

**预期输出：**
```
序列化成功！类型标签: msgpack, 大小: 42 字节
恢复后: {'query': 'hello', 'messages': ['hi', 'there'], 'score': 0.95}
往返验证通过
```

## 4.2 保持状态小——只存储你需要的

**一句话：** 状态大小爆炸是生产环境的头号问题，只在状态中存储必要的数据

社区反复强调的教训：**消息列表和工具输出会快速累积，导致 checkpoint 越来越大，最终拖慢整个系统。**

### 状态膨胀的典型场景

```python
from typing import Annotated
from typing_extensions import TypedDict
from operator import add

# ❌ 危险：消息无限累积
class DangerousState(TypedDict):
    messages: Annotated[list[dict], add]  # 每轮对话追加，永远增长
    documents: Annotated[list[str], add]  # 每次检索追加，永远增长
    tool_outputs: Annotated[list[dict], add]  # 工具输出追加

# 10 轮对话后，messages 可能有 50+ 条消息
# 每条消息可能包含大量工具输出
# checkpoint 大小：几 KB → 几 MB → 几十 MB
```

### 控制状态大小的策略

```python
from typing import Annotated
from typing_extensions import TypedDict
from operator import add

# ✅ 策略1：限制消息数量
def keep_last_n(existing: list, new: list, n: int = 20) -> list:
    """只保留最近 n 条消息"""
    combined = existing + new
    return combined[-n:]  # 只保留最后 n 条

class SmartState(TypedDict):
    # 用自定义 Reducer 限制消息数量
    messages: Annotated[list[dict], keep_last_n]
    # 只存储当前轮次需要的数据，不累积
    current_docs: list[str]  # 用 LastValue，每次覆盖
    final_answer: str

# ✅ 策略2：大对象存外部，状态只存引用
class EfficientState(TypedDict):
    query: str
    # 不存完整文档内容，只存引用 ID
    doc_ids: list[str]           # 引用 ID，几十字节
    # 不存完整的 embedding 向量
    # embedding: list[float]     # ❌ 1536 维 = 12KB
    embedding_id: str            # ✅ 引用 ID，几十字节
    answer: str

# ✅ 策略3：区分"过程数据"和"结果数据"
class CleanState(TypedDict):
    query: str
    answer: str
    # 过程数据用 LastValue（每次覆盖，不累积）
    current_context: str         # 当前上下文，每次覆盖
    current_score: float         # 当前评分，每次覆盖
    # 只有真正需要追踪历史的才用 add Reducer
    key_decisions: Annotated[list[str], add]  # 关键决策记录
```

### 监控状态大小

```python
from langgraph.checkpoint.serde.jsonplus import JsonPlusSerializer
import sys

serde = JsonPlusSerializer()

def check_state_size(state: dict, warn_threshold_kb: int = 100):
    """检查状态序列化后的大小"""
    type_tag, data = serde.dumps_typed(state)
    size_kb = len(data) / 1024

    print(f"状态大小: {size_kb:.1f} KB (类型: {type_tag})")

    if size_kb > warn_threshold_kb:
        print(f"  ⚠ 警告：超过 {warn_threshold_kb} KB 阈值！")
        # 找出最大的字段
        for key, value in state.items():
            _, field_data = serde.dumps_typed(value)
            field_kb = len(field_data) / 1024
            if field_kb > 1:
                print(f"  - {key}: {field_kb:.1f} KB")
    else:
        print(f"  状态大小正常")

# 使用示例
check_state_size({
    "query": "hello",
    "messages": [{"role": "user", "content": "x" * 10000}] * 50,
    "answer": "world",
})
```

## 4.3 开发用 InMemorySaver，生产用 PostgresSaver

**一句话：** 不同阶段用不同的 checkpointer，切换只需要改一行代码

```python
from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict

class State(TypedDict):
    query: str
    answer: str

def my_node(state: State) -> dict:
    return {"answer": f"回答了 {state['query']}"}

builder = StateGraph(State)
builder.add_node("node", my_node)
builder.add_edge(START, "node")
builder.add_edge("node", END)

# === 开发环境 ===
from langgraph.checkpoint.memory import InMemorySaver
checkpointer = InMemorySaver()
# 优点：零配置，速度快
# 缺点：进程结束数据丢失

# === 本地测试 ===
# from langgraph.checkpoint.sqlite import SqliteSaver
# checkpointer = SqliteSaver.from_conn_string("dev_checkpoints.db")
# 优点：数据持久化到文件，重启不丢失
# 缺点：不适合多进程/多实例

# === 生产环境 ===
# from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver
# checkpointer = await AsyncPostgresSaver.from_conn_string(DATABASE_URL)
# 优点：高可用、支持多实例、可扩展
# 缺点：需要维护数据库

# 编译时传入 checkpointer — 这是唯一需要改的地方
graph = builder.compile(checkpointer=checkpointer)

# 使用方式完全一样
config = {"configurable": {"thread_id": "user-123"}}
result = graph.invoke({"query": "什么是 LangGraph？"}, config)
print(result)
```

**选择指南：**

| 场景 | Checkpointer | 理由 |
|------|-------------|------|
| 本地开发 | `InMemorySaver` | 零配置，快速迭代 |
| 集成测试 | `SqliteSaver(":memory:")` | 内存 SQLite，测试后自动清理 |
| 本地持久化 | `SqliteSaver("file.db")` | 重启不丢失，方便调试 |
| 生产部署 | `AsyncPostgresSaver` | 高可用，多实例共享 |
| 高吞吐场景 | Redis Checkpointer | 低延迟，去规范化存储 |

## 4.4 敏感数据启用 EncryptedSerializer

**一句话：** 如果状态中包含用户 PII 或敏感信息，用 EncryptedSerializer 加密 checkpoint

默认情况下，checkpoint 中的数据是明文存储的（虽然是 msgpack 二进制格式，但不是加密的）。如果你的状态包含用户的个人信息、对话内容等敏感数据，应该启用加密。

```python
import os

# === 第1步：设置加密密钥 ===
# 生产环境通过环境变量传入
os.environ["LANGGRAPH_AES_KEY"] = "your-32-byte-secret-key-here!!"
# 密钥长度：16 字节 = AES-128，24 字节 = AES-192，32 字节 = AES-256

# === 第2步：创建加密序列化器 ===
from langgraph.checkpoint.serde.encrypted import EncryptedSerializer

serde = EncryptedSerializer.from_pycryptodome_aes()
# 自动从 LANGGRAPH_AES_KEY 环境变量读取密钥
# 使用 AES-EAX 模式（认证加密：机密性 + 完整性）

# === 第3步：传给 checkpointer ===
from langgraph.checkpoint.sqlite import SqliteSaver
import sqlite3

conn = sqlite3.connect("encrypted_checkpoints.db")
checkpointer = SqliteSaver(conn, serde=serde)

# 之后正常使用，加密对业务代码完全透明
# graph = builder.compile(checkpointer=checkpointer)
```

**什么时候需要加密？**

| 状态内容 | 是否需要加密 |
|----------|-------------|
| 用户对话内容 | 建议加密 |
| 用户个人信息（姓名、邮箱） | 必须加密 |
| API 密钥、Token | 不应放在状态中（用 context） |
| 纯技术中间状态（分数、ID） | 通常不需要 |
| 检索到的公开文档 | 通常不需要 |

**注意：** Pydantic 的 `SecretStr` 不能保护 checkpoint 中的数据！`JsonPlusSerializer` 会调用 `get_secret_value()` 提取明文。要真正保护数据，必须用 `EncryptedSerializer`。

## 4.5 生产环境监控和修剪 Checkpoint

**一句话：** checkpoint 会持续增长，生产环境必须设置监控和清理策略

```python
# === 监控：定期检查 checkpoint 数量和大小 ===

async def monitor_checkpoints(checkpointer, thread_id: str):
    """监控指定线程的 checkpoint 状况"""
    config = {"configurable": {"thread_id": thread_id}}

    count = 0
    for cp in checkpointer.list(config):
        count += 1

    print(f"线程 {thread_id}: {count} 个 checkpoint")

    if count > 100:
        print(f"  ⚠ checkpoint 数量过多，建议修剪")

# === 修剪：删除旧的 checkpoint ===

# 方式1：使用内置的 prune 方法（如果 checkpointer 支持）
# checkpointer.prune(config, strategy="keep_latest", n=10)

# 方式2：手动删除旧 checkpoint
def prune_old_checkpoints(checkpointer, thread_id: str, keep_latest: int = 10):
    """只保留最近 N 个 checkpoint"""
    config = {"configurable": {"thread_id": thread_id}}
```
