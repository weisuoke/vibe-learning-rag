# 多状态管理（Multi-State Management）- 知识点概览

> **一个图只用一个 State 就够了？当图变复杂——有子图、有多阶段、有敏感数据——你需要多个 Schema 协作，让每个模块只看到它该看到的状态。**

---

## 知识点简介

**多状态管理是 LangGraph 通过 Input/Output Schema 分离、节点级类型注解、子图状态隔离和命名空间机制，实现"不同模块看到不同状态视图"的架构能力。**

当你的 LangGraph 应用从一个简单的线性流程成长为包含子图、多阶段 Pipeline、多租户隔离的复杂系统时，一个"大而全"的 State 会带来三个问题：

1. **信息泄露**：图的调用者能看到所有内部中间状态（比如 LLM 的原始输出、内部评分）
2. **耦合过紧**：每个节点都依赖完整的 State 类型，改一个字段所有节点都受影响
3. **子图冲突**：父图和子图如果有同名但含义不同的字段，会互相覆盖

多状态管理就是解决这些问题的。它让你可以：
- 用 `input_schema` / `output_schema` 控制图的"对外接口"
- 用节点参数类型注解让每个节点只接收它需要的字段
- 用子图实现完全独立的状态空间
- 用共享键让父子图优雅通信
- 用命名空间隔离不同子图的 Checkpoint

### 与单一 State 的关键区分

| 维度 | 单一 State（之前学的） | 多状态管理（本节） |
|------|----------------------|-------------------|
| Schema 数量 | 1 个 TypedDict | 多个 TypedDict 协作 |
| 节点视图 | 所有节点看到完整状态 | 每个节点看到状态子集 |
| 图的边界 | 输入输出都是完整 State | 输入输出可以是不同 Schema |
| 子图关系 | 没有子图概念 | 父子图状态完全隔离 |
| 适用场景 | 简单线性流程 | 复杂多模块系统 |

### 场景 1 预览：基础多 Schema 管理

```python
from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict

# 定义多个 Schema
class InputState(TypedDict):
    question: str

class OutputState(TypedDict):
    answer: str

class OverallState(TypedDict):
    question: str
    answer: str
    internal_context: str  # 内部字段，调用者不可见

def retrieve_node(state: InputState) -> dict:
    """检索节点：只需要 question"""
    return {"internal_context": f"关于 '{state['question']}' 的检索结果"}

def generate_node(state: OverallState) -> dict:
    """生成节点：需要 question + internal_context"""
    return {"answer": f"基于 {state['internal_context']} 的回答"}

# 构建图：指定输入输出 Schema
builder = StateGraph(OverallState, input_schema=InputState, output_schema=OutputState)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

graph = builder.compile()

# 调用：只需传入 InputState 的字段
result = graph.invoke({"question": "什么是 RAG？"})
print(result)
# {'answer': '基于 关于 \'什么是 RAG？\' 的检索结果 的回答'}
# 注意：internal_context 不会出现在输出中！
```

### 核心数据流

```
StateGraph(OverallState, input_schema=InputState, output_schema=OutputState)
    ↓
_add_schema() → 将多个 Schema 的字段合并到统一 Channel 池
    ↓
compile() → 生成 CompiledGraph
    ↓
invoke({"question": "..."})
    ↓
START 节点 → 只写入 input_schema 的字段到 Channel
    ↓
各节点执行 → 根据函数参数类型从 Channel 读取对应字段
    ↓
END → 只返回 output_schema 的字段给调用者
```

## 常见问题

### Q1：什么时候需要多状态管理？什么时候一个 State 就够了？

如果你的图是简单的线性流程（3-5 个节点），一个 State 完全够用。当你遇到以下情况时考虑多状态管理：
- 需要隐藏内部状态（不想让调用者看到中间结果）
- 需要模块化（子图独立开发、测试、复用）
- 不同节点需要的字段差异很大（避免每个节点都依赖完整 State）

### Q2：子图的状态和父图完全隔离，那怎么传数据？

两种方式：
1. **共享键**：父图和子图 Schema 中同名的字段会自动传递（推荐）
2. **显式转换**：在调用子图的节点中手动做数据映射（灵活但更啰嗦）

### Q3：`context_schema` 和普通 State 字段有什么区别？

`context_schema` 定义的字段是不可变的运行时上下文，通过 `Runtime[ContextSchema]` 在节点中访问。它不参与状态更新（不能被节点修改），适合放 API key、模型配置等"只读"信息。普通 State 字段是可变的，会随着节点执行而更新。

### Q4：多个 Schema 中有同名字段怎么办？

同名字段必须兼容。具体规则：如果一个 Schema 中该字段没有 Reducer（默认 LastValue），另一个有 Reducer，以有 Reducer 的为准。如果两个 Schema 对同名字段定义了不同的 Reducer，会抛出错误。
