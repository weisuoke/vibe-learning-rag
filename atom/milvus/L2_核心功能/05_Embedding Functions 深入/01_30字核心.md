# 30字核心

**Milvus Embedding Functions是内置的文本向量化功能，支持10个主流提供商，自动处理embedding API调用和批量数据。**

---

## 核心定义拆解

### 1. 内置的文本向量化功能

**含义**：
- 不需要手动调用embedding API
- 在数据插入和查询时自动执行向量化
- 与Milvus collection schema深度集成

**对比传统方式**：

```python
# ❌ 传统方式：手动调用embedding API
from openai import OpenAI
client = OpenAI()

# 1. 手动获取embedding
texts = ["doc1", "doc2", "doc3"]
embeddings = []
for text in texts:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    embeddings.append(response.data[0].embedding)

# 2. 手动插入Milvus
collection.insert([
    {"text": texts[0], "vector": embeddings[0]},
    {"text": texts[1], "vector": embeddings[1]},
    {"text": texts[2], "vector": embeddings[2]}
])

# ✅ Embedding Functions：自动化
collection.insert([
    {"text": "doc1"},
    {"text": "doc2"},
    {"text": "doc3"}
])
# Milvus自动调用embedding API并插入向量
```

### 2. 支持10个主流提供商

**10个提供商列表**：
1. OpenAI
2. Azure OpenAI
3. Cohere
4. AWS Bedrock
5. Google VertexAI
6. VoyageAI
7. 阿里云DashScope
8. SiliconFlow
9. Hugging Face TEI
10. Zilliz Cloud Pipelines

**统一接口设计**：
```python
# 所有提供商使用相同的配置方式
Function(
    name="my_ef",
    function_type=FunctionType.TEXTEMBEDDING,
    input_field_names=["text"],
    output_field_names=["vector"],
    params={
        "provider": "openai",  # 或 "cohere", "voyageai", ...
        "model_name": "text-embedding-3-small",
        "api_key": "sk-xxx"
    }
)
```

### 3. 自动处理embedding API调用

**自动化内容**：
- **批量处理**：自动按MaxBatch分批调用API
- **并行执行**：多个embedding函数并行处理
- **错误处理**：自动重试和错误聚合
- **凭证管理**：支持多种凭证来源（参数/YAML/环境变量）

**批量处理示例**：
```python
# 插入10000条数据
collection.insert([{"text": f"doc{i}"} for i in range(10000)])

# Milvus自动处理：
# - OpenAI: 10000 / 128 = 79批API调用
# - Cohere: 10000 / 96 = 105批API调用
# - Bedrock: 10000 / 1 = 10000批API调用（无批处理）
```

### 4. 批量数据处理

**批量大小对比**：

| 提供商 | MaxBatch | 10000条数据需要API调用次数 |
|--------|----------|---------------------------|
| OpenAI | 128 | 79次 |
| VoyageAI | 128 | 79次 |
| VertexAI | 128 | 79次 |
| Cohere | 96 | 105次 |
| Zilliz | 64 | 157次 |
| SiliconFlow | 32 | 313次 |
| TEI | 32 | 313次 |
| DashScope | 25 | 400次 |
| Bedrock | 1 | 10000次 ⚠️ |

**性能影响**：
- 批量大小直接影响API调用次数
- API调用次数影响总延迟和成本
- 选择批量大的提供商可显著提升性能

## 在RAG开发中的价值

### 1. 简化开发流程

**传统RAG开发**：
```python
# 需要管理3个组件
embedding_client = OpenAI()  # 1. Embedding API客户端
vector_store = MilvusClient()  # 2. 向量数据库客户端
llm_client = OpenAI()  # 3. LLM客户端

# 需要手动协调
def add_documents(texts):
    embeddings = embedding_client.embed(texts)
    vector_store.insert(texts, embeddings)

def search(query):
    query_embedding = embedding_client.embed(query)
    results = vector_store.search(query_embedding)
    return results
```

**使用Embedding Functions**：
```python
# 只需管理2个组件
vector_store = MilvusClient()  # 1. 向量数据库（内置embedding）
llm_client = OpenAI()  # 2. LLM客户端

# 自动协调
def add_documents(texts):
    vector_store.insert([{"text": t} for t in texts])

def search(query):
    results = vector_store.search(data=[query])
    return results
```

### 2. 生产级特性

**并行执行**：
```python
# 多个embedding函数并行处理
schema.add_function(openai_ef)  # 英文embedding
schema.add_function(dashscope_ef)  # 中文embedding

# 插入时并行调用两个API
collection.insert([{"text": "Hello 你好"}])
# → 同时生成英文向量和中文向量
```

**错误处理**：
```python
# 自动处理常见错误
try:
    collection.insert(data)
except MilvusException as e:
    if "empty string" in str(e):
        # 自动过滤空字符串
        pass
    elif "batch size exceeded" in str(e):
        # 自动减小批量大小
        pass
```

### 3. 多提供商灵活切换

**开发到生产的切换**：
```python
# 开发环境：使用OpenAI（稳定可靠）
dev_function = Function(
    params={"provider": "openai", "model_name": "text-embedding-3-small"}
)

# 生产环境：切换到VoyageAI（高性能）
prod_function = Function(
    params={"provider": "voyageai", "model_name": "voyage-3-large"}
)

# 只需修改配置，代码无需改动
```

## 核心优势总结

### 1. 开发效率提升

**减少代码量**：
- 传统方式：~50行代码（embedding + 插入 + 查询）
- Embedding Functions：~20行代码
- **代码量减少60%**

**减少错误点**：
- 无需手动管理embedding API客户端
- 无需手动处理批量数据
- 无需手动同步文本和向量

### 2. 性能优化

**自动批处理**：
- 自动按MaxBatch分批
- 减少API调用次数
- 降低网络延迟

**并行执行**：
- 多个embedding函数并行
- 提升整体吞吐量
- 充分利用系统资源

### 3. 生产可靠性

**错误处理**：
- 自动重试机制
- 错误聚合和报告
- 优雅降级

**凭证管理**：
- 多种凭证来源
- 优先级机制
- 安全性保障

## 与传统方案对比

### 对比表

| 维度 | 传统方案 | Embedding Functions |
|------|----------|---------------------|
| **代码复杂度** | 高（需手动管理） | 低（自动化） |
| **API调用管理** | 手动 | 自动 |
| **批量处理** | 手动实现 | 自动优化 |
| **错误处理** | 手动实现 | 内置机制 |
| **多提供商支持** | 需重写代码 | 修改配置即可 |
| **并行执行** | 需手动实现 | 自动支持 |
| **凭证管理** | 手动管理 | 多来源支持 |
| **维护成本** | 高 | 低 |

### 适用场景

**适合使用Embedding Functions**：
- ✅ 新项目开发
- ✅ 需要快速原型验证
- ✅ 需要支持多个embedding提供商
- ✅ 需要生产级可靠性
- ✅ 团队规模较小，希望减少维护成本

**可能不适合**：
- ❌ 需要极致定制化的embedding流程
- ❌ 已有成熟的embedding管理系统
- ❌ 需要使用不支持的embedding提供商
- ❌ 需要在embedding前后插入复杂的预处理/后处理逻辑
