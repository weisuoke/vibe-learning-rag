# 状态持久化准备 - 30字核心

**状态持久化准备是在设计状态时确保可序列化、控制大小、保护敏感数据，为 Checkpoint 存储扫清障碍。**

---

## 关键词解析

### 状态持久化准备 = 行李安检

在你把行李（状态）交给快递公司（Checkpointer）之前，你需要先检查三件事：

- **可序列化**：行李里不能有液体炸药（数据库连接、文件句柄、Lambda 函数）——这些东西过不了安检（序列化器）
- **大小控制**：行李不能超重（消息列表无限累积、大文档直接塞进状态）——超重要加钱或被拒收（性能下降或存储失败）
- **敏感数据保护**：贵重物品要上锁（API 密钥、用户隐私信息）——不能明文放在行李外面（需要加密序列化）

**做好这三件事，你的状态就能安全、高效地被持久化到任何存储后端。**

---

## 核心公式

```
状态持久化准备 = 可序列化设计 + 大小控制 + 敏感数据保护
```

展开来看：

```
可序列化设计：
  ✅ 基础类型：str, int, float, bool, None
  ✅ 容器类型：list, dict, tuple, set
  ✅ 扩展类型：UUID, datetime, Decimal, Enum, Path, Pydantic, NumPy
  ❌ 运行时对象：数据库连接、文件句柄、Socket、Lambda、线程锁

大小控制：
  ✅ 消息修剪（只保留最近 N 条）
  ✅ Checkpoint 修剪（只保留最近 N 个快照）
  ✅ 大对象外部存储（状态中只存引用/ID）
  ❌ 无限累积消息列表
  ❌ 把完整文档内容塞进状态

敏感数据保护：
  ✅ EncryptedSerializer（AES-EAX 加密）
  ✅ 敏感字段不放状态（用 context_schema 或环境变量）
  ❌ API 密钥明文存在 Checkpoint 中
  ❌ 用户隐私信息未加密持久化
```

## 源码层面的核心接口

```python
# 序列化协议：所有序列化器必须实现的接口
class SerializerProtocol(Protocol):
    def dumps_typed(self, obj: Any) -> tuple[str, bytes]:
        """将 Python 对象序列化为 (类型标签, 字节数据)"""
        ...
    def loads_typed(self, data: tuple[str, bytes]) -> Any:
        """将 (类型标签, 字节数据) 反序列化为 Python 对象"""
        ...
```

类型标签（type tag）是关键设计：它告诉反序列化器"这段字节是用什么格式编码的"。

```python
# 类型标签示例
("msgpack", b"\x82\xa3foo\xa3bar...")   # msgpack 编码
("bytes", b"\x89PNG\r\n...")             # 原始字节
("null", b"")                            # None 值
("msgpack+aes", b"\x10\x20...")          # msgpack 编码 + AES 加密
```

[来源: langgraph/checkpoint/serde/base.py]

## 默认序列化器的处理流程

```python
# JsonPlusSerializer.dumps_typed() 的决策逻辑
def dumps_typed(self, obj):
    if obj is None:
        return ("null", b"")           # None → 空字节
    if isinstance(obj, bytes):
        return ("bytes", obj)          # bytes → 直接返回
    if isinstance(obj, bytearray):
        return ("bytearray", bytes(obj))
    try:
        # 尝试 msgpack 编码（默认路径）
        return ("msgpack", ormsgpack.packb(obj, ...))
    except Exception:
        if self.pickle_fallback:
            # pickle 回退（默认关闭）
            return ("pickle", pickle.dumps(obj))
        raise  # 无法序列化 → 抛出异常
```

**关键洞察：** 如果你的状态字段值无法被 msgpack 编码，且 pickle_fallback 关闭（默认），序列化就会直接失败。这就是为什么"可序列化设计"是持久化准备的第一步。

[来源: langgraph/checkpoint/serde/jsonplus.py]

## Checkpoint 数据结构

每次节点执行后保存的 Checkpoint 长这样：

```python
checkpoint = {
    "v": 2,                              # 格式版本
    "id": "1ef4f797-8335-...",           # 唯一 ID（uuid6，单调递增）
    "ts": "2024-07-31T20:14:19...",      # ISO 8601 时间戳
    "channel_values": {                  # ← 这里存的就是你的状态！
        "query": "什么是向量数据库？",
        "docs": ["文档片段1", "文档片段2"],
        "answer": "向量数据库是...",
    },
    "channel_versions": {                # 每个通道的版本号
        "query": 2,
        "docs": 3,
        "answer": 4,
    },
    "versions_seen": {                   # 每个节点看到的版本
        "retrieve": {"query": 2},
        "generate": {"docs": 3},
    },
}
```

**`channel_values` 就是你的状态数据被序列化后存储的地方。** 如果这里面有不可序列化的值，整个 Checkpoint 就保存不了。如果这里面的数据太大，存储和恢复都会变慢。

[来源: langgraph/checkpoint/base/__init__.py]

## 持久化管道：三层架构

```
┌─────────────────────────────────────────────────────────┐
│                    持久化管道                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  数据层（Channel）                                       │
│  ├── checkpoint() → 产生可序列化的状态快照                │
│  └── from_checkpoint() → 从快照恢复状态                  │
│                                                         │
│  序列化层（Serializer）                                   │
│  ├── dumps_typed() → Python 对象 → (type_tag, bytes)    │
│  ├── loads_typed() → (type_tag, bytes) → Python 对象    │
│  └── [可选] encrypt/decrypt → 加密/解密字节              │
│                                                         │
│  存储层（CheckpointSaver）                               │
│  ├── put() → 存储到后端                                  │
│  ├── get_tuple() → 从后端加载                            │
│  └── list() → 列出所有 Checkpoint                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**三层各司其职：**
- **数据层**知道"什么数据需要保存"
- **序列化层**知道"怎么把数据变成字节"
- **存储层**知道"字节存到哪里"

状态持久化准备主要关注前两层：确保数据层产生的快照能被序列化层正确处理。

[来源: langgraph/checkpoint/serde/base.py, langgraph/channels/base.py, langgraph/checkpoint/base/__init__.py]

## 记忆锚点

> **状态持久化准备 = "你的状态能过安检吗？"**

三个安检关卡：
1. **X 光机**（可序列化检查）：状态里有没有"违禁品"（不可序列化对象）？
2. **称重台**（大小控制）：状态有没有"超重"（数据太大）？
3. **贵重物品柜**（敏感数据保护）：有没有"贵重物品"需要上锁（加密）？

三关全过，你的状态就能安全飞往任何目的地（存储后端）。

## 快速自检：你的状态持久化就绪了吗？

```python
from typing_extensions import TypedDict
from typing import Annotated
from operator import add

# 逐字段检查你的状态定义
class MyState(TypedDict):
    # ✅ 基础类型 → 安全
    query: str
    count: int
    score: float

    # ✅ 容器类型（值也是基础类型）→ 安全
    docs: Annotated[list[str], add]
    metadata: dict[str, str]

    # ✅ 扩展类型 → JsonPlusSerializer 支持
    # created_at: datetime        # 扩展类型代码 3
    # request_id: UUID            # 扩展类型代码 0
    # config: MyPydanticModel     # 扩展类型代码 4/5

    # ❌ 以下类型会导致序列化失败：
    # db_conn: Connection         # 数据库连接
    # callback: Callable          # 函数/Lambda
    # file: IO                    # 文件句柄
    # lock: threading.Lock        # 线程锁
    # socket: socket.socket       # 网络连接
```

**规则很简单：如果一个值能用 `json.dumps()` 或 `msgpack.packb()` 编码，它就是安全的。如果不能，要么换成可序列化的替代品，要么放到 `context_schema` 或环境变量中。**

## 三个最常见的持久化失败原因

### 1. 状态中有不可序列化对象

```python
# ❌ 失败：Lambda 函数无法序列化
class BadState(TypedDict):
    query: str
    scorer: Callable  # Lambda 或函数引用

# ✅ 修复：把函数逻辑移到节点内部，状态只存数据
class GoodState(TypedDict):
    query: str
    score_method: str  # 存方法名字符串，节点内部根据名字调用
```

### 2. 消息列表无限增长

```python
# ❌ 问题：每轮对话追加消息，Checkpoint 越来越大
class ChatState(TypedDict):
    messages: Annotated[list, add_messages]  # 只增不减！

# ✅ 修复：在节点中修剪消息
def trim_messages_node(state: ChatState) -> dict:
    """只保留最近 20 条消息"""
    messages = state["messages"]
    if len(messages) > 20:
        messages = messages[-20:]
    return {"messages": messages}
```

### 3. 敏感数据明文存储

```python
# ❌ 问题：API 密钥明文存在 Checkpoint 中
class BadState(TypedDict):
    query: str
    api_key: str  # 明文存储到数据库！

# ✅ 修复方案 A：用 context_schema（不进入 Checkpoint）
class ContextSchema(TypedDict):
    api_key: str

builder = StateGraph(State, context_schema=ContextSchema)

# ✅ 修复方案 B：用加密序列化器
from langgraph.checkpoint.serde.encrypted import EncryptedSerializer
serde = EncryptedSerializer.from_pycryptodome_aes()
checkpointer = SqliteSaver(conn, serde=serde)
```

## 与 RAG 开发的关联

在 RAG 系统中，状态持久化准备尤其重要：

| RAG 状态字段 | 持久化风险 | 解决方案 |
|-------------|-----------|---------|
| `messages: list` | 多轮对话累积，Checkpoint 膨胀 | 消息修剪（保留最近 N 条） |
| `retrieved_docs: list[str]` | 检索结果可能很大 | 只存文档 ID，不存全文 |
| `embedding_cache: dict` | 缓存数据不需要持久化 | 移到 `context_schema` 或节点局部变量 |
| `api_key: str` | 敏感信息明文存储 | 用环境变量或 `context_schema` |
| `llm_client: OpenAI` | 客户端对象不可序列化 | 在节点内部创建，不放状态 |

**核心原则：状态里只放"数据"，不放"工具"。数据可以序列化，工具不行。**
