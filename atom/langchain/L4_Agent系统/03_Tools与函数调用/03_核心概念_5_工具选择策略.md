# 核心概念 5：工具选择策略

> 当 Agent 拥有 10+ 工具时，全部塞给 LLM 会导致 Prompt 膨胀、选择准确率下降、响应变慢。工具选择策略 = 在调用 LLM 前，筛选出最相关的工具子集，让模型从"大海捞针"变成"精准点菜"。

---

## 概述

**工具选择**解决的核心问题是：Agent 有很多工具，但每次调用 LLM 时应该只暴露最相关的几个。

为什么这很重要？三个原因：

1. **Token 成本**：每个工具的 Schema 都要序列化进 Prompt，10 个工具可能占 2000+ tokens
2. **选择准确率**：工具越多，模型越容易选错——就像菜单有 200 道菜时你反而不知道点什么
3. **响应速度**：更长的 Prompt = 更慢的推理

```
没有工具选择：
用户问题 + [工具1 Schema + 工具2 Schema + ... + 工具20 Schema] → LLM → 选错概率高

有工具选择：
用户问题 → 工具选择器 → [工具3 Schema + 工具7 Schema] → LLM → 精准调用
```

LangChain 提供了从简单到智能的四层工具选择方案：

| 层级 | 方案 | 适用场景 |
|------|------|----------|
| L1 | `tool_choice` 参数 | 强制/禁止调用特定工具 |
| L2 | `render_text_description()` | 自定义工具描述格式 |
| L3 | 自定义 Middleware | 基于规则动态过滤工具 |
| L4 | `LLMToolSelectorMiddleware` | 用 LLM 智能选择工具 |

---

## 1. tool_choice 参数 — 最基础的工具选择

### 一句话定义

**`tool_choice` 是 `bind_tools()` 的参数，控制模型"是否必须调用工具"以及"调用哪个工具"。**

这是最简单的工具选择——不是筛选工具列表，而是约束模型的调用行为。

### 四种取值

| 值 | 含义 | 类比 |
|------|------|------|
| `None` / `"auto"` | 模型自行决定是否调用工具 | 自助餐，想吃什么拿什么 |
| `"any"` | 必须调用至少一个工具 | 必须点菜，不能空手 |
| `"none"` | 禁止调用任何工具 | 今天不营业，只能聊天 |
| `"工具名"` | 必须调用指定的工具 | 指定只能点"宫保鸡丁" |

### 代码示例

```python
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

@tool
def search_kb(query: str) -> str:
    """在知识库中搜索相关文档"""
    return f"知识库结果: {query}"

@tool
def search_web(query: str) -> str:
    """搜索互联网获取实时信息"""
    return f"网络结果: {query}"

model = ChatOpenAI(model="gpt-4o-mini")

# ===== auto: 模型自己决定 =====
model_auto = model.bind_tools(
    [search_kb, search_web],
    tool_choice="auto"
)

# ===== any: 强制使用工具（不允许直接回答）=====
model_must_use = model.bind_tools(
    [search_kb, search_web],
    tool_choice="any"
)

# ===== none: 禁止使用工具（只能直接回答）=====
model_no_tools = model.bind_tools(
    [search_kb, search_web],
    tool_choice="none"
)

# ===== 指定工具: 强制调用 search_kb =====
model_force_kb = model.bind_tools(
    [search_kb, search_web],
    tool_choice="search_kb"
)
```

### RAG 中的典型用法

```python
# 场景 1: 强制 Agent 先检索再回答（避免幻觉）
retriever_model = model.bind_tools(
    [search_kb],
    tool_choice="search_kb"  # 必须先查知识库
)

# 场景 2: 第一轮强制检索，后续轮次自动决定
first_turn_model = model.bind_tools([search_kb], tool_choice="search_kb")
follow_up_model = model.bind_tools([search_kb, search_web], tool_choice="auto")
```

### tool_choice 的局限性

`tool_choice` 只能控制"是否调用"和"调用哪个"，但无法解决核心问题——**当工具数量很多时，所有工具的 Schema 仍然会全部发送给 LLM**。

```
tool_choice 能做的：    约束模型的选择行为
tool_choice 做不到的：  减少发送给模型的工具数量
```

要真正减少工具数量，需要下面的方案。

## 2. 工具渲染 — 控制工具描述格式

### 源码位置

`langchain_core/tools/render.py`（68 行）

### 为什么需要工具渲染？

工具的 Schema 发送给 LLM 前，需要序列化为文本。渲染函数决定了这个文本的格式和详细程度。更精简的渲染 = 更少的 token = 更低的成本。

### 两个核心函数

#### render_text_description() — 只渲染名称和描述

```python
from langchain_core.tools import tool
from langchain_core.tools.render import render_text_description

@tool
def search_kb(query: str) -> str:
    """在知识库中搜索相关文档"""
    return f"结果: {query}"

@tool
def calculator(expression: str) -> str:
    """计算数学表达式"""
    return str(eval(expression))

# 渲染工具描述
text = render_text_description([search_kb, calculator])
print(text)
# 输出:
# search_kb: 在知识库中搜索相关文档
# calculator: 计算数学表达式
```

精简模式——只告诉模型"有什么工具"和"干什么用"，不暴露参数细节。

#### render_text_description_and_args() — 渲染名称、描述和参数

```python
from langchain_core.tools.render import render_text_description_and_args

text = render_text_description_and_args([search_kb, calculator])
print(text)
# 输出:
# search_kb: 在知识库中搜索相关文档, args: {'query': {'type': 'string'}}
# calculator: 计算数学表达式, args: {'expression': {'type': 'string'}}
```

详细模式——包含参数 Schema，模型能更准确地构造调用参数。

### 对比

| 函数 | 输出内容 | Token 消耗 | 适用场景 |
|------|----------|-----------|----------|
| `render_text_description` | 名称 + 描述 | 少 | 工具选择阶段（先选再调） |
| `render_text_description_and_args` | 名称 + 描述 + 参数 | 多 | 工具调用阶段（需要参数信息） |

### 在工具选择中的应用

渲染函数常用于两阶段策略：先用精简描述让模型选工具，再用完整 Schema 让模型调用工具。

```python
from langchain_core.tools.render import render_text_description

def two_stage_tool_selection(query: str, all_tools: list, model):
    """两阶段工具选择：先选后调"""

    # 阶段 1: 用精简描述让模型选择工具
    tool_descriptions = render_text_description(all_tools)
    selection_prompt = f"""根据用户问题，从以下工具中选择最相关的（返回工具名称列表）：

可用工具：
{tool_descriptions}

用户问题：{query}
"""
    # 用便宜的模型做选择
    selected_names = model.invoke(selection_prompt)

    # 阶段 2: 只把选中的工具绑定到模型
    selected_tools = [t for t in all_tools if t.name in selected_names]
    return selected_tools
```

## 3. 自定义 Middleware — 基于规则的动态工具过滤

### 核心思想

Middleware（中间件）是 LangChain Agent 的拦截器模式。它在 Agent 调用 LLM 之前，可以修改请求——包括动态调整传给模型的工具列表。

```
用户输入 → Agent → [Middleware 拦截] → 过滤后的工具列表 → LLM → 响应
                         ↑
                    在这里筛选工具
```

### 类方式定义 Middleware

```python
from langchain.agents import create_agent
from langchain.agents.middleware import AgentMiddleware

class ToolSelectorMiddleware(AgentMiddleware):
    """基于规则的工具选择中间件"""

    def wrap_model_call(self, request, handler):
        # 从请求中获取上下文信息
        state = request.state
        runtime = request.runtime

        # 根据规则筛选工具
        relevant_tools = self._select_tools(state, runtime)

        # 用筛选后的工具列表覆盖原始列表
        return handler(request.override(tools=relevant_tools))

    def _select_tools(self, state, runtime):
        """自定义工具选择逻辑"""
        all_tools = runtime.tools

        # 示例规则：根据对话历史中的关键词选择工具
        last_message = state.messages[-1].content if state.messages else ""

        selected = []
        for tool in all_tools:
            # 规则 1: 数学相关问题 → 包含 calculator
            if tool.name == "calculator" and any(
                kw in last_message for kw in ["计算", "多少", "加", "减", "乘", "除"]
            ):
                selected.append(tool)
            # 规则 2: 搜索相关问题 → 包含 search
            elif tool.name == "search_kb" and any(
                kw in last_message for kw in ["查找", "搜索", "什么是", "怎么"]
            ):
                selected.append(tool)
            # 规则 3: 始终包含的基础工具
            elif tool.name in ["help", "fallback"]:
                selected.append(tool)

        return selected if selected else all_tools  # 没匹配到就返回全部

# 使用
agent = create_agent(
    model="gpt-4.1",
    tools=all_tools,
    middleware=[ToolSelectorMiddleware()],
)
```

### 装饰器方式定义 Middleware

更简洁的写法，适合逻辑简单的场景：

```python
from langchain.agents.middleware import wrap_model_call

@wrap_model_call
def select_tools_by_keyword(request, handler):
    """基于关键词的工具选择"""
    last_message = request.state.messages[-1].content
    all_tools = request.runtime.tools

    # 简单的关键词匹配
    keyword_tool_map = {
        "天气": ["get_weather"],
        "计算": ["calculator"],
        "搜索": ["search_kb", "search_web"],
        "文档": ["search_kb"],
    }

    selected_names = set()
    for keyword, tool_names in keyword_tool_map.items():
        if keyword in last_message:
            selected_names.update(tool_names)

    if selected_names:
        relevant_tools = [t for t in all_tools if t.name in selected_names]
    else:
        relevant_tools = all_tools

    return handler(request.override(tools=relevant_tools))

# 使用
agent = create_agent(
    model="gpt-4.1",
    tools=all_tools,
    middleware=[select_tools_by_keyword],
)
```

### 类方式 vs 装饰器方式

| 维度 | 类方式 | 装饰器方式 |
|------|--------|-----------|
| 代码量 | 较多 | 较少 |
| 可维护性 | 高（方法拆分） | 中（单函数） |
| 状态管理 | 支持（类属性） | 不支持 |
| 适用场景 | 复杂规则、需要配置 | 简单规则、快速原型 |
| 测试友好 | 高（可单独实例化） | 中 |

## 4. LLMToolSelectorMiddleware — 用 LLM 智能选择工具

### 核心思想

规则匹配有局限——用户的表达方式千变万化，硬编码关键词很难覆盖所有情况。`LLMToolSelectorMiddleware` 的思路是：**用一个便宜的小模型先做工具选择，再把选中的工具交给主模型做实际调用**。

```
传统方式：
用户问题 + [20个工具] → GPT-4（贵） → 调用工具

LLMToolSelectorMiddleware：
用户问题 + [20个工具描述] → GPT-4.1-mini（便宜）→ 选出3个工具
用户问题 + [3个工具Schema] → GPT-4.1（贵）→ 精准调用
```

### 基本用法

```python
from langchain.agents import create_agent
from langchain.agents.middleware import LLMToolSelectorMiddleware

agent = create_agent(
    model="gpt-4.1",
    tools=[tool1, tool2, tool3, tool4, tool5,
           tool6, tool7, tool8, tool9, tool10],
    middleware=[
        LLMToolSelectorMiddleware(
            model="gpt-4.1-mini",   # 用便宜模型做选择
            max_tools=3,             # 最多选 3 个工具
            always_include=["search_kb"],  # 始终包含的工具
        ),
    ],
)
```

### 参数详解

| 参数 | 类型 | 说明 |
|------|------|------|
| `model` | str / ChatModel | 用于工具选择的模型（建议用便宜的） |
| `max_tools` | int | 每次最多选择的工具数量 |
| `always_include` | list[str] | 始终包含的工具名称列表（不参与选择） |

### 工作原理

```
步骤 1: 收集所有工具的名称和描述
步骤 2: 构造选择 Prompt:
        "根据用户问题，从以下工具中选择最相关的 {max_tools} 个：
         - search_kb: 在知识库中搜索文档
         - calculator: 计算数学表达式
         - get_weather: 查询天气
         - ..."
步骤 3: 用便宜模型（如 gpt-4.1-mini）执行选择
步骤 4: 解析选择结果，过滤工具列表
步骤 5: 合并 always_include 的工具
步骤 6: 将过滤后的工具列表传给主模型
```

### 成本对比

假设有 20 个工具，每个工具 Schema 约 100 tokens：

| 方案 | 选择模型成本 | 主模型输入 tokens | 总成本 |
|------|-------------|------------------|--------|
| 不选择 | 0 | 2000 (20 工具) | 高 |
| LLMToolSelector (max_tools=3) | ~500 tokens (mini) | 300 (3 工具) | 低 |

用便宜模型花 500 tokens 做选择，省下主模型 1700 tokens 的输入——净省约 70% 的工具相关 token 成本。

## 5. 实战场景：RAG 多知识库工具选择

### 场景描述

一个企业 RAG 系统有多个知识库，每个知识库对应一个检索工具：

```python
from langchain_core.tools import tool

@tool
def search_hr_docs(query: str) -> str:
    """搜索人事制度文档，包含考勤、请假、薪资、福利等规定"""
    return f"[HR] {query} 的相关规定..."

@tool
def search_tech_docs(query: str) -> str:
    """搜索技术文档，包含 API 文档、架构设计、开发规范等"""
    return f"[Tech] {query} 的技术文档..."

@tool
def search_finance_docs(query: str) -> str:
    """搜索财务文档，包含报销流程、预算制度、财务报表等"""
    return f"[Finance] {query} 的财务文档..."

@tool
def search_legal_docs(query: str) -> str:
    """搜索法务文档，包含合同模板、合规要求、知识产权等"""
    return f"[Legal] {query} 的法务文档..."

@tool
def search_product_docs(query: str) -> str:
    """搜索产品文档，包含需求文档、产品路线图、用户手册等"""
    return f"[Product] {query} 的产品文档..."

@tool
def calculator(expression: str) -> str:
    """计算数学表达式"""
    return str(eval(expression))

@tool
def search_web(query: str) -> str:
    """搜索互联网获取外部信息"""
    return f"[Web] {query} 的搜索结果..."

all_tools = [
    search_hr_docs, search_tech_docs, search_finance_docs,
    search_legal_docs, search_product_docs, calculator, search_web,
]
```

### 方案 1: 基于规则的 Middleware

```python
from langchain.agents.middleware import AgentMiddleware

class DomainToolSelector(AgentMiddleware):
    """基于领域关键词的工具选择"""

    # 领域关键词映射
    domain_keywords = {
        "search_hr_docs": ["请假", "考勤", "薪资", "福利", "入职", "离职", "社保"],
        "search_tech_docs": ["API", "接口", "部署", "代码", "架构", "bug", "开发"],
        "search_finance_docs": ["报销", "预算", "发票", "财务", "费用", "报表"],
        "search_legal_docs": ["合同", "合规", "知识产权", "法律", "协议", "授权"],
        "search_product_docs": ["需求", "产品", "功能", "用户", "路线图", "版本"],
    }

    def wrap_model_call(self, request, handler):
        last_message = request.state.messages[-1].content
        all_tools = request.runtime.tools

        # 匹配领域
        matched_tools = []
        for tool in all_tools:
            keywords = self.domain_keywords.get(tool.name, [])
            if any(kw in last_message for kw in keywords):
                matched_tools.append(tool)

        # 始终包含通用工具
        general_tools = [t for t in all_tools if t.name in ["calculator", "search_web"]]

        selected = matched_tools + [t for t in general_tools if t not in matched_tools]

        # 没匹配到领域 → 返回全部
        if not matched_tools:
            selected = all_tools

        return handler(request.override(tools=selected))

agent = create_agent(
    model="gpt-4.1",
    tools=all_tools,
    middleware=[DomainToolSelector()],
)
```

### 方案 2: 用 LLM 智能选择

```python
from langchain.agents import create_agent
from langchain.agents.middleware import LLMToolSelectorMiddleware

agent = create_agent(
    model="gpt-4.1",
    tools=all_tools,
    middleware=[
        LLMToolSelectorMiddleware(
            model="gpt-4.1-mini",
            max_tools=2,                    # 每次最多选 2 个知识库
            always_include=["calculator"],  # 计算器始终可用
        ),
    ],
)

# 用户问 "请假流程是什么？"
# → LLMToolSelector 选出 search_hr_docs
# → 主模型只看到 search_hr_docs + calculator
# → 精准调用 search_hr_docs("请假流程")
```

## 6. 实战场景：多租户权限控制

### 场景描述

不同用户角色能使用的工具不同——普通员工不能查财务数据，只有管理员能执行危险操作。

```python
from langchain.agents.middleware import AgentMiddleware

class PermissionMiddleware(AgentMiddleware):
    """基于用户角色的工具权限控制"""

    # 角色 → 允许的工具
    role_permissions = {
        "employee": ["search_hr_docs", "search_tech_docs", "search_product_docs", "calculator"],
        "manager": ["search_hr_docs", "search_tech_docs", "search_product_docs",
                     "search_finance_docs", "calculator"],
        "admin": None,  # None = 全部工具
        "legal": ["search_legal_docs", "search_hr_docs", "calculator"],
    }

    def wrap_model_call(self, request, handler):
        # 从运行时配置中获取用户角色
        user_role = request.runtime.config.get("user_role", "employee")
        all_tools = request.runtime.tools

        allowed_names = self.role_permissions.get(user_role)

        if allowed_names is None:
            # admin 可以使用全部工具
            return handler(request)

        # 过滤工具
        filtered_tools = [t for t in all_tools if t.name in allowed_names]
        return handler(request.override(tools=filtered_tools))

# 使用
agent = create_agent(
    model="gpt-4.1",
    tools=all_tools,
    middleware=[PermissionMiddleware()],
)

# 普通员工调用 → 只能看到 4 个工具
# agent.invoke({"input": "查看财务报表"}, config={"user_role": "employee"})
# → 没有 search_finance_docs，模型会说"我没有权限查看财务数据"

# 管理员调用 → 能看到 5 个工具
# agent.invoke({"input": "查看财务报表"}, config={"user_role": "manager"})
# → 有 search_finance_docs，正常检索
```

## 7. 组合多个 Middleware

Middleware 可以链式组合，按顺序执行：

```python
from langchain.agents import create_agent
from langchain.agents.middleware import LLMToolSelectorMiddleware

agent = create_agent(
    model="gpt-4.1",
    tools=all_tools,
    middleware=[
        # 第 1 层: 权限过滤（先根据角色去掉无权限的工具）
        PermissionMiddleware(),
        # 第 2 层: 智能选择（再从剩余工具中选最相关的）
        LLMToolSelectorMiddleware(
            model="gpt-4.1-mini",
            max_tools=3,
        ),
    ],
)
```

执行顺序：

```
全部工具 (7个)
    → PermissionMiddleware: 权限过滤 → 剩余 4 个
    → LLMToolSelectorMiddleware: 智能选择 → 剩余 2 个
    → 主模型: 从 2 个工具中精准调用
```

## 四种方案对比总结

| 维度 | tool_choice | 工具渲染 | 自定义 Middleware | LLMToolSelector |
|------|------------|---------|------------------|-----------------|
| 复杂度 | 低 | 低 | 中 | 低（开箱即用） |
| 减少工具数量 | 否 | 否 | 是 | 是 |
| 智能程度 | 无（硬编码） | 无 | 规则匹配 | LLM 语义理解 |
| 额外成本 | 无 | 无 | 无 | 小模型调用费 |
| 适用场景 | 强制/禁止调用 | 控制描述格式 | 规则明确的场景 | 工具多、规则复杂 |
| 可维护性 | 高 | 高 | 中（规则需维护） | 高（自动适应） |

### 选择决策流程

```
工具数量 ≤ 5？
    └─ YES → 不需要工具选择，直接用 tool_choice
    └─ NO → 工具数量 > 5
              │
              ├─ 有明确的领域/权限规则？
              │   └─ YES → 自定义 Middleware（规则匹配）
              │
              ├─ 规则难以穷举，需要语义理解？
              │   └─ YES → LLMToolSelectorMiddleware
              │
              └─ 两者都需要？
                  └─ YES → 组合使用（权限 Middleware + LLMToolSelector）
```

## 总结

工具选择策略是 Agent 从"玩具"到"生产"的关键一步：

1. **`tool_choice`** 是最基础的控制——约束模型是否调用工具、调用哪个，但不减少工具数量
2. **工具渲染函数** 控制工具描述的详细程度——`render_text_description()` 精简、`render_text_description_and_args()` 详细
3. **自定义 Middleware** 基于规则动态过滤——关键词匹配、权限控制、领域路由
4. **`LLMToolSelectorMiddleware`** 用便宜模型做语义级工具选择——最智能，适合工具多且规则难穷举的场景
5. **多个 Middleware 可以链式组合**——先权限过滤，再智能选择，层层收窄

核心原则：工具越少，模型越准。把"20 选 1"变成"3 选 1"，准确率和速度都会显著提升。
