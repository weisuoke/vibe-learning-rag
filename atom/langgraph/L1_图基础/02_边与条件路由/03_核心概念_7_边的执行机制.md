# 核心概念 7：边的执行机制

> 本文档详细讲解 LangGraph 边的执行机制，包括执行时机、状态传递、错误处理和编译验证

---

## 什么是边的执行机制？

**边的执行机制是指 LangGraph 如何在图执行过程中处理边的触发、状态传递、错误处理和验证的完整流程。**

理解边的执行机制对于构建可靠的状态化工作流至关重要。

[来源: reference/source_edges_01_add_edge.md, reference/source_edges_02_add_conditional_edges.md]

## 1. 边的执行时机

### 1.1 普通边的执行时机

**普通边在起始节点执行完成后立即触发。**

```python
from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict

class State(TypedDict):
    value: int

def node_a(state: State) -> dict:
    print("执行 node_a")
    return {"value": state["value"] + 1}

def node_b(state: State) -> dict:
    print("执行 node_b")
    return {"value": state["value"] * 2}

graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)

graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")  # node_a 完成后立即执行 node_b
graph.add_edge("node_b", END)

app = graph.compile()
result = app.invoke({"value": 10})
```

**执行流程**：
```
START
  ↓ (立即触发)
node_a (value: 10 → 11)
  ↓ (node_a 完成后立即触发)
node_b (value: 11 → 22)
  ↓ (node_b 完成后立即触发)
END
```

[来源: reference/source_edges_01_add_edge.md]

---

### 1.2 条件边的执行时机

**条件边在起始节点执行完成后，先调用路由函数，然后根据返回值触发目标节点。**

```python
from typing import Literal

def route_func(state: State) -> Literal["node_b", "node_c"]:
    print(f"路由决策: value={state['value']}")
    if state["value"] > 15:
        return "node_b"
    return "node_c"

def node_c(state: State) -> dict:
    print("执行 node_c")
    return {"value": state["value"] + 5}

graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.add_node("node_c", node_c)

graph.add_edge(START, "node_a")
graph.add_conditional_edges("node_a", route_func)  # node_a 完成后调用路由函数
graph.add_edge("node_b", END)
graph.add_edge("node_c", END)

app = graph.compile()
result = app.invoke({"value": 10})
```

**执行流程**：
```
START
  ↓
node_a (value: 10 → 11)
  ↓ (node_a 完成后)
route_func(state) → "node_c" (因为 11 <= 15)
  ↓
node_c (value: 11 → 16)
  ↓
END
```

**关键点**：
1. **路由函数调用时机**：在起始节点完成后立即调用
2. **路由函数输入**：接收起始节点执行后的状态
3. **目标节点触发**：根据路由函数返回值触发目标节点

[来源: reference/source_edges_02_add_conditional_edges.md]

---

### 1.3 等待边的执行时机

**等待边（多起始节点）在所有起始节点都执行完成后才触发目标节点。**

```python
def parallel_a(state: State) -> dict:
    print("执行 parallel_a")
    return {"value": state["value"] + 1}

def parallel_b(state: State) -> dict:
    print("执行 parallel_b")
    return {"value": state["value"] + 2}

def merge(state: State) -> dict:
    print(f"合并结果: value={state['value']}")
    return {"value": state["value"]}

graph = StateGraph(State)
graph.add_node("parallel_a", parallel_a)
graph.add_node("parallel_b", parallel_b)
graph.add_node("merge", merge)

graph.add_edge(START, "parallel_a")
graph.add_edge(START, "parallel_b")
graph.add_edge(["parallel_a", "parallel_b"], "merge")  # 等待两个节点都完成
graph.add_edge("merge", END)

app = graph.compile()
result = app.invoke({"value": 10})
```

**执行流程**：
```
START
  ↓
┌──────────┬──────────┐
│parallel_a│parallel_b│ (并行执行)
│(10 → 11) │(10 → 12) │
└──────────┴──────────┘
  ↓         ↓
  └─────────┘ (等待所有节点完成)
      ↓
    merge (value: 12，因为 parallel_b 的更新覆盖了 parallel_a)
      ↓
     END
```

**关键点**：
1. **AND 逻辑**：必须等待所有起始节点完成
2. **状态合并**：多个节点的状态更新会合并（后执行的覆盖先执行的）
3. **同步点**：等待边是并行执行的同步点

[来源: reference/source_edges_01_add_edge.md]

## 2. 状态传递机制

### 2.1 状态更新与传递

**每个节点执行后返回的字典会更新到状态中，然后传递给下一个节点。**

```python
class State(TypedDict):
    value: int
    message: str

def node_a(state: State) -> dict:
    # 只更新 value，不更新 message
    return {"value": state["value"] + 1}

def node_b(state: State) -> dict:
    # 更新 message，不更新 value
    return {"message": f"Value is {state['value']}"}

graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)

graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")
graph.add_edge("node_b", END)

app = graph.compile()
result = app.invoke({"value": 10, "message": ""})
print(result)  # {'value': 11, 'message': 'Value is 11'}
```

**状态传递流程**：
```
初始状态: {"value": 10, "message": ""}
  ↓
node_a 返回: {"value": 11}
  ↓
状态更新: {"value": 11, "message": ""}  (合并)
  ↓
node_b 接收: {"value": 11, "message": ""}
  ↓
node_b 返回: {"message": "Value is 11"}
  ↓
最终状态: {"value": 11, "message": "Value is 11"}  (合并)
```

**关键点**：
1. **部分更新**：节点可以只返回部分状态字段
2. **状态合并**：返回的字典会合并到当前状态中
3. **状态传递**：更新后的状态传递给下一个节点

---

### 2.2 Reducer 函数与状态合并

**使用 `Annotated` 和 reducer 函数可以自定义状态合并逻辑。**

```python
from typing import Annotated
from operator import add

class State(TypedDict):
    value: int
    results: Annotated[list[str], add]  # 使用 add reducer 合并列表

def node_a(state: State) -> dict:
    return {"results": ["A"]}

def node_b(state: State) -> dict:
    return {"results": ["B"]}

def node_c(state: State) -> dict:
    return {"results": ["C"]}

graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.add_node("node_c", node_c)

graph.add_edge(START, "node_a")
graph.add_edge(START, "node_b")
graph.add_edge(["node_a", "node_b"], "node_c")
graph.add_edge("node_c", END)

app = graph.compile()
result = app.invoke({"value": 0, "results": []})
print(result["results"])  # ['A', 'B', 'C']
```

**状态合并流程**：
```
初始状态: {"value": 0, "results": []}
  ↓
node_a 返回: {"results": ["A"]}
node_b 返回: {"results": ["B"]}
  ↓ (使用 add reducer 合并)
状态更新: {"value": 0, "results": ["A", "B"]}
  ↓
node_c 接收: {"value": 0, "results": ["A", "B"]}
  ↓
node_c 返回: {"results": ["C"]}
  ↓ (使用 add reducer 合并)
最终状态: {"value": 0, "results": ["A", "B", "C"]}
```

**常用 reducer 函数**：
- `operator.add`：列表合并、数值相加
- `operator.or_`：字典合并
- 自定义函数：任意合并逻辑

[来源: reference/source_edges_01_add_edge.md]

## 3. 错误处理机制

### 3.1 节点执行错误

**如果节点执行过程中抛出异常，图会停止执行并抛出异常。**

```python
def node_with_error(state: State) -> dict:
    if state["value"] < 0:
        raise ValueError("Value cannot be negative")
    return {"value": state["value"] + 1}

graph = StateGraph(State)
graph.add_node("node_with_error", node_with_error)
graph.add_edge(START, "node_with_error")
graph.add_edge("node_with_error", END)

app = graph.compile()

try:
    result = app.invoke({"value": -1})
except ValueError as e:
    print(f"错误: {e}")  # 错误: Value cannot be negative
```

**错误处理策略**：
1. **捕获异常**：在节点函数中使用 try-except 捕获异常
2. **错误状态**：将错误信息存储在状态中
3. **错误路由**：使用条件边路由到错误处理节点

---

### 3.2 路由函数错误

**如果路由函数抛出异常，图会停止执行并抛出异常。**

```python
def route_with_error(state: State) -> str:
    if state["value"] < 0:
        raise ValueError("Cannot route negative value")
    return "node_b" if state["value"] > 10 else "node_c"

graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.add_node("node_c", node_c)

graph.add_edge(START, "node_a")
graph.add_conditional_edges("node_a", route_with_error)
graph.add_edge("node_b", END)
graph.add_edge("node_c", END)

app = graph.compile()

try:
    result = app.invoke({"value": -5})
except ValueError as e:
    print(f"路由错误: {e}")
```

**错误处理策略**：
1. **验证输入**：在路由函数中验证状态值
2. **默认路由**：提供默认路由路径
3. **错误日志**：记录路由决策和错误信息

---

### 3.3 错误恢复模式

**使用条件边实现错误恢复和重试机制。**

```python
class State(TypedDict):
    value: int
    retry_count: int
    error: str

def process_node(state: State) -> dict:
    try:
        # 模拟可能失败的操作
        if state["value"] < 0:
            raise ValueError("Negative value")
        return {"value": state["value"] * 2, "error": ""}
    except Exception as e:
        return {"error": str(e), "retry_count": state.get("retry_count", 0) + 1}

def error_route(state: State) -> Literal["retry", "error_handler", END]:
    if state["error"]:
        if state.get("retry_count", 0) < 3:
            return "retry"  # 重试
        else:
            return "error_handler"  # 错误处理
    return END  # 成功

def retry_node(state: State) -> dict:
    print(f"重试第 {state['retry_count']} 次")
    # 修复错误（例如：将负值转为正值）
    return {"value": abs(state["value"]), "error": ""}

def error_handler(state: State) -> dict:
    print(f"错误处理: {state['error']}")
    return {"value": 0}

graph = StateGraph(State)
graph.add_node("process", process_node)
graph.add_node("retry", retry_node)
graph.add_node("error_handler", error_handler)

graph.add_edge(START, "process")
graph.add_conditional_edges("process", error_route)
graph.add_edge("retry", "process")  # 重试后回到 process
graph.add_edge("error_handler", END)

app = graph.compile()
result = app.invoke({"value": -10, "retry_count": 0, "error": ""})
```

**错误恢复流程**：
```
START
  ↓
process (失败，error="Negative value", retry_count=1)
  ↓
error_route() → "retry"
  ↓
retry (修复: value=-10 → 10)
  ↓
process (成功，value=20)
  ↓
error_route() → END
```

[来源: reference/context7_langgraph_01_edges_routing.md]

## 4. 编译验证机制

### 4.1 图编译时的验证

**图在编译时会验证边的有效性。**

```python
# ❌ 错误：END 不能作为起始节点
try:
    graph.add_edge(END, "node_a")
except ValueError as e:
    print(f"验证错误: {e}")  # 验证错误: END cannot be a start node

# ❌ 错误：START 不能作为结束节点
try:
    graph.add_edge("node_a", START)
except ValueError as e:
    print(f"验证错误: {e}")  # 验证错误: START cannot be an end node

# ❌ 错误：节点未定义
try:
    graph.add_edge("undefined_node", "node_a")
    app = graph.compile()  # 编译时会验证
except ValueError as e:
    print(f"验证错误: {e}")
```

**验证规则**：
1. **END 不能作为起始节点**
2. **START 不能作为结束节点**
3. **所有节点必须已定义**（通过 `add_node` 添加）
4. **条件边的路由函数必须有效**

[来源: reference/source_edges_01_add_edge.md]

---

### 4.2 编译后不可修改

**图编译后，添加边不会生效。**

```python
graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.add_edge(START, "node_a")
graph.add_edge("node_a", END)

app = graph.compile()  # 编译图

# ⚠️ 警告：编译后添加边不会生效
graph.add_edge("node_a", "node_b")  # 不会生效
```

**警告信息**：
```
Adding an edge to a graph that has already been compiled. This will not be reflected in the compiled graph.
```

**关键点**：
- **编译前添加**：所有边必须在编译前添加
- **不可变性**：编译后的图是不可变的
- **重新编译**：如果需要修改，必须重新编译

[来源: reference/source_edges_01_add_edge.md, reference/source_edges_02_add_conditional_edges.md]

## 5. 边的存储结构

### 5.1 普通边的存储

**普通边存储在 `self.edges` 集合中。**

```python
# StateGraph 类定义
class StateGraph:
    def __init__(self, state_schema):
        self.edges: set[tuple[str, str]] = set()
        # ...

    def add_edge(self, start_key: str, end_key: str):
        self.edges.add((start_key, end_key))
        # ...
```

**存储格式**：
```python
edges = {
    (START, "node_a"),
    ("node_a", "node_b"),
    ("node_b", END)
}
```

### 5.2 等待边的存储

## 6. 执行顺序与并发

### 6.1 顺序执行

**普通边按照定义顺序顺序执行。**

```python
graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")
graph.add_edge("node_b", "node_c")
graph.add_edge("node_c", END)

# 执行顺序: START → node_a → node_b → node_c → END
```

---

### 6.2 并行执行

**多个节点从同一个起始节点出发时，会并行执行。**

```python
graph.add_edge(START, "node_a")
graph.add_edge(START, "node_b")
graph.add_edge(START, "node_c")

# node_a, node_b, node_c 并行执行
```

---

### 6.3 同步点

**等待边是并行执行的同步点。**

```python
graph.add_edge(START, "parallel_a")
graph.add_edge(START, "parallel_b")
graph.add_edge(["parallel_a", "parallel_b"], "merge")  # 同步点

# parallel_a 和 parallel_b 并行执行
# merge 等待两者都完成后才执行
```

## 7. 最佳实践

### 7.1 明确的执行流程

```python
# ✅ 推荐：清晰的执行流程
graph.add_edge(START, "load_data")
graph.add_edge("load_data", "process_data")
graph.add_edge("process_data", "save_data")
graph.add_edge("save_data", END)

# ❌ 不推荐：复杂的执行流程
graph.add_edge(START, "node1")
graph.add_conditional_edges("node1", route1)
graph.add_conditional_edges("node2", route2)
graph.add_edge(["node3", "node4"], "node5")

```

## 总结

**边的执行机制核心要点**：

1. **执行时机**：
   - 普通边：起始节点完成后立即触发
   - 条件边：起始节点完成后调用路由函数，然后触发目标节点
   - 等待边：所有起始节点完成后才触发

2. **状态传递**：
   - 节点返回的字典会合并到当前状态
   - 使用 reducer 函数自定义合并逻辑
   - 状态传递给下一个节点

3. **错误处理**：
   - 节点或路由函数抛出异常会停止图执行
   - 使用 try-except 捕获异常
   - 使用条件边实现错误恢复和重试

4. **编译验证**：
   - 图编译时验证边的有效性
   - 编译后不可修改
   - 验证规则：END 不能作为起始节点，START 不能作为结束节点

5. **存储结构**：
   - 普通边：`self.edges`
   - 等待边：`self.waiting_edges`
   - 条件边：`self.branches`

通过理解边的执行机制，可以构建可靠、高效的状态化工作流。
