## 一、基础概念题

### Q1: 为什么 OpenClaw 选择 pnpm 而不是 npm？

**答案**:

1. **磁盘空间效率**: pnpm 使用内容寻址存储，相同版本的包只存储一次
2. **安装速度**: 硬链接比复制文件快 100 倍
3. **严格依赖**: 杜绝幽灵依赖，只能使用 package.json 中声明的依赖
4. **workspace 支持**: 更好的 monorepo 支持

**数据来源**: `reference/01_pnpm_workspace_docs.md`

---

### Q2: tsdown 和 tsc 的区别是什么？

**答案**:

| 维度 | tsc | tsdown |
|------|-----|--------|
| 编译速度 | 慢 | 快（基于 Rust） |
| 类型检查 | 严格 | 完整 |
| DTS 生成 | 支持 | 支持 |
| 使用场景 | 类型检查 | 构建 |

**推荐做法**: tsdown 用于构建，tsc 用于类型检查

**数据来源**: `reference/05_typescript_build_github.md`

---

### Q3: Node.js 22 的关键特性有哪些？

**答案**:

1. **ESM 模块系统成熟**: 完善的 ESM 和 CommonJS 互操作
2. **性能提升**: V8 引擎更新，性能显著提升
3. **LTS 版本**: 30 个月支持，稳定性保证
4. **更好的 TypeScript 集成**: 支持 NodeNext 模块解析

**数据来源**: `reference/02_nodejs_22_docs.md`

## 二、实战操作题

### Q4: 如何从零搭建 OpenClaw 开发环境？

**答案**:

```bash
# 1. 检查环境
node --version  # >= 22.12.0
pnpm --version  # = 10.23.0

# 2. 克隆仓库
git clone https://github.com/openclaw/openclaw.git
cd openclaw

# 3. 安装依赖
pnpm install

# 4. 构建
pnpm ui:build && pnpm build

# 5. 开发模式
pnpm gateway:watch
```

**数据来源**: `sourcecode/openclaw/package.json`

---

### Q5: 如何配置 watch 模式？

**答案**:

使用 chokidar 监听文件变化：

```javascript
import { watch } from 'chokidar';

const watcher = watch('src/**/*.ts', {
  ignored: /(^|[\/\\])\../,
  persistent: true
});

watcher.on('change', (path) => {
  console.log(`File ${path} changed`);
  rebuild();
});
```

**数据来源**: `sourcecode/openclaw/scripts/run-node.mjs`

## 三、原理深度题

### Q6: pnpm 的硬链接机制是如何工作的？

**答案**:

1. **全局 store**: 所有依赖存储在 `~/.pnpm-store/`
2. **硬链接**: 项目中的 `node_modules` 使用硬链接指向 store
3. **内容寻址**: 相同内容的文件只存储一次
4. **符号链接**: 包之间的依赖使用符号链接

**优势**: 节省磁盘空间，加快安装速度

**数据来源**: `reference/01_pnpm_workspace_docs.md`

---

### Q7: TypeScript 的 NodeNext 模块解析是什么？

**答案**:

NodeNext 是 TypeScript 5.0+ 引入的模块解析策略：

1. **完全支持 ESM**: 支持 Node.js 的 ESM 模块系统
2. **自动处理扩展名**: 自动处理 .mjs 和 .cjs 文件
3. **package.json 集成**: 支持 "type": "module"
4. **互操作性**: ESM 和 CommonJS 互操作

**配置**:

```json
{
  "compilerOptions": {
    "module": "NodeNext",
    "moduleResolution": "NodeNext"
  }
}
```

**数据来源**: `reference/03_typescript_5.9.3_docs.md`

## 四、故障排查题

### Q8: pnpm install 失败如何排查？

**答案**:

1. **检查 Node.js 版本**: `node --version` >= 22.12.0
2. **清理缓存**: `pnpm store prune`
3. **使用镜像**: `pnpm config set registry https://registry.npmmirror.com`
4. **重新安装**: `rm -rf node_modules pnpm-lock.yaml && pnpm install`

**数据来源**: `reference/06_openclaw_dev_setup_github_reddit.md`

---

### Q9: 编译速度慢如何优化？

**答案**:

1. **启用增量编译**: `incremental: true`
2. **使用缓存**: `clean: false`
3. **并行编译**: 使用 Turborepo
4. **减少编译范围**: 使用 `exclude` 排除不必要的文件

**数据来源**: `reference/05_typescript_build_github.md`

## 五、架构设计题

### Q10: 为什么 OpenClaw 使用三层架构？

**答案**:

三层架构实现了关注点分离：

1. **依赖管理层**: 管理所有依赖包
2. **构建工具链层**: 编译和类型检查
3. **源码结构层**: 代码组织和模块划分

**优势**:
- 职责清晰
- 易于维护
- 便于扩展

---

### Q11: monorepo 的优缺点是什么？

**答案**:

**优点**:
- 统一管理多个项目
- 代码共享容易
- 原子提交
- 统一的工具链

**缺点**:
- 仓库体积大
- 构建可能慢（可通过增量编译优化）
- 权限管理复杂（可通过 CODEOWNERS 解决）

**数据来源**: `reference/07_pnpm_workspace_best_practices.md`

## 六、最佳实践题

### Q12: 开发环境的最佳实践有哪些？

**答案**:

1. **版本管理**: 使用 asdf 或 nvm 管理 Node.js 版本
2. **依赖锁定**: 使用 pnpm-lock.yaml 锁定版本
3. **类型检查**: 启用 strict 模式
4. **代码质量**: 使用 ESLint + Prettier
5. **自动化**: 使用 watch 模式和热重载
6. **调试配置**: 配置 VSCode 调试环境

---

### Q13: 如何保证团队开发环境一致？

**答案**:

1. **锁定版本**: engines 字段 + engine-strict
2. **统一工具**: 使用 corepack 管理 pnpm
3. **配置文件**: .nvmrc, .tool-versions
4. **文档**: README 中明确环境要求
5. **CI 验证**: CI 中验证环境配置
