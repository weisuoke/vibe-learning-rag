# 实战代码 - 场景3：条件状态转换

## 场景说明

本场景演示 LangGraph 中**条件状态转换**的核心模式——根据输入条件选择不同的转换逻辑，让同一个图处理多种类型的数据。

具体场景：一个文档分类与处理管道，不同类型的文档（邮件、报告、代码）走不同的转换路径，最终汇聚到统一的后处理节点。

核心知识点：
1. 使用 `Command` API 实现节点内路由——分类与路由在同一个函数中完成
2. 多分支汇聚——不同处理路径共享同一个 `finalize` 节点
3. `Literal` 类型注解——告诉 LangGraph 所有可能的路由目标
4. `Annotated[list, add]` reducer——跨分支累积日志

[来源: reference/context7_langgraph_01.md | LangGraph 官方文档]
[来源: 03_核心概念_05_Command_API状态转换.md | Command API 详解]

## 图结构

```
                    ┌──────────────┐
                    │    START     │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   classify   │  ← 分类节点，返回 Command
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
     ┌────────▼───┐ ┌─────▼──────┐ ┌──▼──────────┐
     │process_email│ │process_report│ │process_code │
     └────────┬───┘ └─────┬──────┘ └──┬──────────┘
              │            │            │
              └────────────┼────────────┘
                           │
                    ┌──────▼───────┐
                    │   finalize   │  ← 统一后处理
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │     END      │
                    └──────────────┘
```

关键点：`classify` 节点不使用 `add_conditional_edges`，而是直接返回 `Command` 对象来决定下一步去哪个节点。三条分支最终都汇聚到 `finalize`。

## 完整代码

```python
"""
LangGraph 状态转换函数 - 条件状态转换
演示：根据输入条件选择不同转换逻辑

场景：文档分类与处理管道
- 邮件 → 提取发件人、主题、正文
- 报告 → 提取摘要和字数统计
- 代码 → 提取函数和类定义

核心技术：
- Command API 实现节点内路由
- 多分支汇聚到统一后处理
- Annotated reducer 跨分支累积日志

运行环境：Python 3.13+, langgraph
安装依赖：uv add langgraph
"""

import re
from typing import Annotated, Literal
from typing_extensions import TypedDict
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.types import Command


# ============================================================
# 1. 定义状态 Schema
# ============================================================

class DocState(TypedDict):
    """文档处理状态

    无 Reducer 字段（覆盖语义）：
    - content   → 原始文档内容
    - doc_type  → 分类结果："email" | "report" | "code"
    - processed → 处理后的格式化文本
    - metadata  → 提取的结构化元数据

    有 Reducer 字段（追加语义）：
    - logs → 使用 operator.add，每个节点的日志自动追加
    """
    content: str
    doc_type: str
    processed: str
    metadata: dict
    logs: Annotated[list[str], add]


# ============================================================
# 2. 分类节点 —— 使用 Command 实现"分类 + 路由"二合一
# ============================================================

def classify_doc(
    state: DocState,
) -> Command[Literal["process_email", "process_report", "process_code"]]:
    """根据内容自动分类文档类型，并路由到对应处理节点

    关键点：
    - 返回类型必须是 Command[Literal[...]]，列出所有可能的目标节点
    - Command 同时完成两件事：update（更新状态）+ goto（指定下一个节点）
    - 不需要 add_conditional_edges，Command 自带路由能力
    """
    content = state["content"].lower()

    # 邮件特征：包含 @ 或 subject: 头
    if "@" in content or "subject:" in content:
        return Command(
            update={"doc_type": "email", "logs": ["[classify] 分类结果: email"]},
            goto="process_email",
        )
    # 代码特征：包含 def/import/class 关键字
    elif "def " in content or "import " in content or "class " in content:
        return Command(
            update={"doc_type": "code", "logs": ["[classify] 分类结果: code"]},
            goto="process_code",
        )
    # 默认：当作报告处理
    else:
        return Command(
            update={"doc_type": "report", "logs": ["[classify] 分类结果: report"]},
            goto="process_report",
        )


# ============================================================
# 3. 三个分支处理节点 —— 各自实现不同的转换逻辑
# ============================================================

def process_email(state: DocState) -> dict:
    """处理邮件：提取发件人、主题、正文

    普通节点函数，返回 dict 即可（不需要 Command，因为出边是静态的）。
    """
    content = state["content"]
    lines = content.strip().split("\n")

    subject = ""
    sender = ""
    body_lines = []

    for line in lines:
        if line.lower().startswith("subject:"):
            subject = line.split(":", 1)[1].strip()
        elif line.lower().startswith("from:"):
            sender = line.split(":", 1)[1].strip()
        else:
            body_lines.append(line)

    body = "\n".join(body_lines).strip()

    return {
        "processed": f"[邮件] 发件人: {sender} | 主题: {subject}\n{body}",
        "metadata": {"sender": sender, "subject": subject, "type": "email"},
        "logs": [f"[process_email] 邮件处理完成: {subject}"],
    }


def process_report(state: DocState) -> dict:
    """处理报告：生成摘要和字数统计"""
    content = state["content"]

    # 简单摘要：取前 200 字符
    summary = content[:200].strip()
    if len(content) > 200:
        summary += "..."

    word_count = len(content.split())

    return {
        "processed": f"[报告摘要] ({word_count} words)\n{summary}",
        "metadata": {"word_count": word_count, "type": "report"},
        "logs": [f"[process_report] 报告处理完成: {word_count} words"],
    }


def process_code(state: DocState) -> dict:
    """处理代码：提取函数和类定义"""
    content = state["content"]

    functions = re.findall(r"def (\w+)\(", content)
    classes = re.findall(r"class (\w+)", content)

    return {
        "processed": f"[代码分析] 函数: {functions} | 类: {classes}",
        "metadata": {"functions": functions, "classes": classes, "type": "code"},
        "logs": [
            f"[process_code] 代码分析完成: {len(functions)} functions, {len(classes)} classes"
        ],
    }


# ============================================================
# 4. 通用后处理节点 —— 三条分支汇聚于此
# ============================================================

def finalize(state: DocState) -> dict:
    """最终格式化：所有分支都会经过这里

    这就是"多分支汇聚"模式——无论文档走了哪条路径，
    最终都到达同一个 finalize 节点做统一的收尾工作。
    """
    return {
        "logs": [f"[finalize] 处理完成, 文档类型: {state['doc_type']}"],
    }


# ============================================================
# 5. 构建图 —— 注意 classify 节点不需要条件边
# ============================================================

graph = StateGraph(DocState)

# 添加所有节点
graph.add_node("classify", classify_doc)
graph.add_node("process_email", process_email)
graph.add_node("process_report", process_report)
graph.add_node("process_code", process_code)
graph.add_node("finalize", finalize)

# 入口边
graph.add_edge(START, "classify")

# classify 使用 Command 路由，不需要 add_conditional_edges
# 只需要给三个处理节点添加到 finalize 的静态边
graph.add_edge("process_email", "finalize")
graph.add_edge("process_report", "finalize")
graph.add_edge("process_code", "finalize")
graph.add_edge("finalize", END)

# 编译
app = graph.compile()


# ============================================================
# 6. 测试：三种文档类型
# ============================================================

def make_input(content: str) -> DocState:
    """构造初始状态（辅助函数）"""
    return {
        "content": content,
        "doc_type": "",
        "processed": "",
        "metadata": {},
        "logs": [],
    }


print("=" * 60)
print("条件状态转换示例：文档分类与处理管道")
print("=" * 60)

# ----- 测试1：邮件 -----
print("\n--- 测试1：邮件文档 ---")
email_input = make_input(
    "From: alice@example.com\nSubject: Meeting Tomorrow\nHi team, let's meet at 3pm."
)
r1 = app.invoke(email_input)
print(f"  类型: {r1['doc_type']}")
print(f"  结果: {r1['processed']}")
print(f"  元数据: {r1['metadata']}")
print(f"  日志: {r1['logs']}")

# ----- 测试2：代码 -----
print("\n--- 测试2：代码文档 ---")
code_input = make_input(
    "import os\n\ndef hello():\n    print('hi')\n\nclass MyApp:\n    pass"
)
r2 = app.invoke(code_input)
print(f"  类型: {r2['doc_type']}")
print(f"  结果: {r2['processed']}")
print(f"  元数据: {r2['metadata']}")
print(f"  日志: {r2['logs']}")

# ----- 测试3：报告 -----
print("\n--- 测试3：报告文档 ---")
report_input = make_input(
    "本季度销售额同比增长 15%，主要得益于新产品线的推出。"
    "华东区域表现尤为突出，贡献了总收入的 40%。"
    "建议下季度加大华南区域的市场投入。"
)
r3 = app.invoke(report_input)
print(f"  类型: {r3['doc_type']}")
print(f"  结果: {r3['processed']}")
print(f"  元数据: {r3['metadata']}")
print(f"  日志: {r3['logs']}")

# ----- 使用 stream 观察执行路径 -----
print("\n--- Stream 模式：观察邮件的执行路径 ---")
for event in app.stream(email_input):
    for node_name, node_output in event.items():
        print(f"  节点 [{node_name}] → 更新字段: {list(node_output.keys())}")
```

## 关键要点

### 1. Command API 实现条件路由

传统写法需要两个函数（节点 + 边函数），Command 把它们合二为一：

```python
# ❌ 传统写法：分类和路由分开
def classify_node(state):
    doc_type = detect_type(state["content"])
    return {"doc_type": doc_type}

def route_by_type(state) -> str:
    return f"process_{state['doc_type']}"

graph.add_conditional_edges("classify", route_by_type)

# ✅ Command 写法：分类和路由在同一个函数
def classify_doc(state) -> Command[Literal["process_email", "process_report", "process_code"]]:
    if is_email(state["content"]):
        return Command(update={"doc_type": "email"}, goto="process_email")
    ...
# 不需要 add_conditional_edges
```

Command 写法更自然——分类结果和路由决策本来就是同一个思考过程。

### 2. 多分支汇聚模式

三个处理节点都通过静态边连接到同一个 `finalize` 节点：

```python
graph.add_edge("process_email", "finalize")
graph.add_edge("process_report", "finalize")
graph.add_edge("process_code", "finalize")
```

这是经典的"扇出-扇入"（fan-out / fan-in）模式。每次运行只有一条分支被激活，但图结构上三条路径都存在。`finalize` 不需要知道数据从哪条路径来——它只读取状态中已有的字段。

### 3. Literal 类型注解不能省

```python
# ❌ 缺少类型注解 → LangGraph 编译时不知道可能的路由目标
def classify_doc(state: DocState):
    return Command(goto="process_email")

# ✅ 必须声明所有可能的目标
def classify_doc(state: DocState) -> Command[Literal["process_email", "process_report", "process_code"]]:
    return Command(goto="process_email")
```

LangGraph 在编译图时需要知道所有可能的边，`Literal` 类型注解就是在告诉框架这个节点可能跳转到哪些节点。

### 4. Command 节点不要加静态出边

```python
# ❌ 危险：classify 同时有静态边和 Command
graph.add_edge("classify", "process_report")  # 静态边
# 如果 classify 返回 Command(goto="process_email")
# 结果：process_report 和 process_email 都会执行！

# ✅ 正确：classify 的出边完全由 Command 控制
graph.add_edge(START, "classify")
# classify 不添加任何静态出边
```

### 5. logs 字段的 Reducer 跨分支累积

```python
logs: Annotated[list[str], add]
```

无论文档走了哪条分支，每个节点返回的 `logs` 都会通过 `operator.add` 追加到同一个列表中。最终状态的 `logs` 完整记录了整条执行路径：`classify → process_xxx → finalize`。

## 在 RAG 中的应用

条件状态转换在 RAG 系统中非常常见——不同类型的查询需要不同的检索策略：

```python
def query_router(state) -> Command[Literal["vector_search", "keyword_search", "hybrid_search"]]:
    """根据查询特征选择检索策略"""
    query = state["query"]

    if is_semantic_query(query):       # "类似于...的文档"
        return Command(update={"strategy": "vector"}, goto="vector_search")
    elif is_exact_match(query):        # "错误码 E1001"
        return Command(update={"strategy": "keyword"}, goto="keyword_search")
    else:
        return Command(update={"strategy": "hybrid"}, goto="hybrid_search")
```

同样的模式也适用于：文档类型分流（PDF/HTML/Markdown 走不同解析器）、用户意图路由（问答/闲聊/任务执行走不同处理链）、错误处理分支（成功/重试/降级走不同路径）。
