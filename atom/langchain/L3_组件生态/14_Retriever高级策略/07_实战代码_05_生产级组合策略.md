# 实战代码 场景5：生产级组合策略

> **终极组合**：MultiQuery 查询扩展 → Ensemble 混合检索 → Rerank 重排序 → LCEL RAG 生成，四层管道覆盖生产级 RAG 检索的完整链路

---

## 场景描述

构建一个生产级 RAG 检索管道，将前四个场景的策略组合成一个完整系统：

- **查询扩展**：MultiQueryRetriever 生成多个查询变体，提高召回率
- **混合检索**：EnsembleRetriever 融合 BM25 关键词检索 + 向量语义检索
- **重排序**：ContextualCompressionRetriever + FlashRank 精排候选文档
- **RAG 生成**：LCEL 链式组合，将精选上下文注入 LLM 生成回答

这是"多路召回 + 精排 + 生成"的经典生产架构。

### Step 2: 构建混合检索层（Ensemble）

```python
"""
第二层：BM25 + 向量检索的混合检索
"""

from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever

# ===== 1. 创建 BM25 关键词检索器 =====

bm25_retriever = BM25Retriever.from_documents(
    documents,
    k=10  # 每次检索返回 10 篇
)

# ===== 2. 创建向量语义检索器 =====

vector_retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 10}
)

# ===== 3. 组合为 EnsembleRetriever =====
# 权重：向量 0.6 + BM25 0.4（语义优先，关键词补充）

ensemble_retriever = EnsembleRetriever(
    retrievers=[vector_retriever, bm25_retriever],
    weights=[0.6, 0.4],
    c=60  # RRF 常数，原始论文默认值
)

# ===== 4. 验证混合检索 =====

print("=== 混合检索测试 ===")
query = "LCEL 表达式的优势"
ensemble_docs = ensemble_retriever.invoke(query)
print(f"混合检索返回 {len(ensemble_docs)} 篇文档")
for i, doc in enumerate(ensemble_docs[:3]):
    print(f"  [{i+1}] {doc.page_content[:60]}...")
```

### Step 3: 添加重排序层（Rerank）

```python
"""
第三层：FlashRank 重排序，从候选文档中精选 Top-5
"""

from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import FlashrankRerank

# ===== 1. 创建 FlashRank 重排序器 =====
# FlashRank 是本地模型，无需 API 调用，首次运行会自动下载模型

compressor = FlashrankRerank(
    top_n=5,           # 精选 Top-5
    model="ms-marco-MiniLM-L-12-v2"  # 轻量级模型，适合生产环境
)

# ===== 2. 包装为 ContextualCompressionRetriever =====
# 装饰器模式：ensemble_retriever 粗筛 → compressor 精排

rerank_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=ensemble_retriever
)

# ===== 3. 验证重排序效果 =====

print("\n=== 重排序测试 ===")
reranked_docs = rerank_retriever.invoke(query)
print(f"重排序后返回 {len(reranked_docs)} 篇文档")
for i, doc in enumerate(reranked_docs):
    score = doc.metadata.get("relevance_score", "N/A")
    print(f"  [{i+1}] (score={score}) {doc.page_content[:60]}...")
```

### Step 5: 构建完整 LCEL RAG 管道

```python
"""
完整管道：查询扩展 → 混合检索 → 重排序 → LLM 生成
使用 LCEL 管道操作符组合
"""

# ===== 1. 定义 RAG Prompt =====

rag_prompt = ChatPromptTemplate.from_template("""基于以下上下文回答用户问题。
如果上下文中没有相关信息，请明确说明。

上下文：
{context}

问题：{question}

回答：""")

# ===== 2. 定义文档格式化函数 =====

def format_docs(docs: list) -> str:
    """将文档列表格式化为带编号的文本"""
    formatted = []
    for i, doc in enumerate(docs, 1):
        source = doc.metadata.get("source", "未知")
        formatted.append(f"[{i}] ({source}) {doc.page_content}")
    return "\n\n".join(formatted)

# ===== 3. 构建 LCEL RAG Chain =====
# RunnablePassthrough.assign 同时传递 question 并检索 context

rag_chain = (
    RunnablePassthrough.assign(
        context=lambda x: format_docs(
            multi_query_retriever.invoke(x["question"])
        )
    )
    | rag_prompt
    | llm
    | StrOutputParser()
)

# ===== 4. 端到端测试 =====

print("\n" + "=" * 60)
print("=== 端到端 RAG 测试 ===")
print("=" * 60)

test_queries = [
    "LCEL 表达式相比传统 Chain 有什么优势？",
    "如何实现混合检索？",
    "LangChain 的重排序机制是怎样的？",
]

for q in test_queries:
    print(f"\n问题: {q}")
    print("-" * 40)

    start = time.time()
    answer = rag_chain.invoke({"question": q})
    elapsed = time.time() - start

    print(f"回答: {answer}")
    print(f"耗时: {elapsed:.2f}s")
```

        "doc_count": len(final_docs)

    # 第四层：RAG 生成
    t0 = time.time()
    answer = rag_chain.invoke({"question": question})
    metrics["layers"]["rag_generation"] = {
        "time_ms": round((time.time() - t0) * 1000)
    }

    metrics["answer"] = answer
    return metrics

# ===== 2. 运行监控测试 =====

## 性能优化建议

### 1. 异步执行

MultiQueryRetriever 的异步版本会并行执行所有查询变体的检索，显著降低延迟：

```python
# 异步调用（推荐生产环境使用）
import asyncio

async def async_rag_query(question: str) -> str:
    docs = await multi_query_retriever.ainvoke(question)
    context = format_docs(docs)
    return await rag_chain.ainvoke({"question": question})

# 批量异步查询
async def batch_async_queries(questions: list[str]) -> list[str]:
    tasks = [async_rag_query(q) for q in questions]
    return await asyncio.gather(*tasks)
```

### 2. 缓存策略

对 Embedding 和检索结果做缓存，避免重复计算：

```python
from langchain.globals import set_llm_cache
from langchain_community.cache import InMemoryCache

# LLM 响应缓存（MultiQuery 生成的查询变体可复用）
set_llm_cache(InMemoryCache())

# 自定义检索缓存
from functools import lru_cache

@lru_cache(maxsize=256)
def cached_retrieve(question: str) -> tuple:
    """缓存检索结果（tuple 可哈希）"""
    docs = multi_query_retriever.invoke(question)
    return tuple(doc.page_content for doc in docs)
```

### 3. 参数调优指南

| 参数 | 默认值 | 调优方向 | 说明 |
|------|--------|----------|------|
| BM25 k | 10 | 增大提高召回 | 粗筛阶段，宁多勿少 |
| Vector k | 10 | 增大提高召回 | 与 BM25 保持一致 |
| Ensemble weights | [0.6, 0.4] | 技术文档偏向 BM25 | 关键词密集场景调高 BM25 权重 |
| RRF c | 60 | 一般不需要调 | 原始论文标准值 |
| Rerank top_n | 5 | 根据 context window 调整 | 太多浪费 token，太少丢信息 |
| MultiQuery 变体数 | 3 | 2-5 之间 | 更多变体 = 更高召回 + 更高延迟 |

## 各层策略对比

| 维度 | 纯向量检索 | + 混合检索 | + 重排序 | + 查询扩展 |
|------|-----------|-----------|---------|-----------|
| 召回率 | 中 | 高 | 高 | 很高 |
| 精确率 | 中 | 中 | 高 | 高 |
| 延迟 | ~100ms | ~150ms | ~300ms | ~2-3s |
| 复杂度 | 低 | 中 | 中 | 高 |
| 适用场景 | 原型验证 | 一般生产 | 质量敏感 | 高要求生产 |

**选型建议**：大多数生产场景用"混合检索 + 重排序"就够了。查询扩展因为多了 LLM 调用，延迟较高，适合对召回率要求极高且能接受延迟的场景。
