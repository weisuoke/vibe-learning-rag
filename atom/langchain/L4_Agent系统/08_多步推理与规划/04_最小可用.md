## 最小可用 1：Plan-and-Execute 架构

**核心思想：规划和执行分离**

```
用户输入 → Planner（大模型生成计划）
              ↓
         [步骤1, 步骤2, 步骤3]
              ↓
         Executor（小模型逐步执行）
              ↓
         Replanner（检查是否完成）
              ↓
         完成 → 输出 / 未完成 → 调整计划
```

**为什么这样做？**
- Planner 用大模型（如 GPT-4o），只调用 1-2 次
- Executor 用小模型（如 GPT-4o-mini），成本低
- 比 ReAct 的"每步都用大模型"省 60%+ 成本

## 最小可用 2：Structured Output

**让 LLM 输出结构化的计划，而不是自由文本。**

```python
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field
from typing import List

# 定义计划结构
class Plan(BaseModel):
    steps: List[str] = Field(description="按顺序执行的步骤列表")

# 使用 with_structured_output
llm = ChatOpenAI(model="gpt-4o", temperature=0)
planner = llm.with_structured_output(Plan)

# 生成计划
plan = planner.invoke(
    "帮我对比 ChromaDB 和 Milvus 在 RAG 项目中的适用性"
)

print(plan.steps)
# ['搜索 ChromaDB 的核心特性和限制',
#  '搜索 Milvus 的核心特性和限制',
#  '对比两者的性能和扩展性',
#  '分析各自适用的 RAG 场景',
#  '给出选型建议']
```

**关键点**：
- `with_structured_output()` 保证输出格式正确
- Pydantic 模型定义输出 Schema
- `Field(description=...)` 帮助 LLM 理解每个字段含义

## 最小可用 3：LangGraph 三节点实现

**最小的 Plan-and-Execute Agent，只需三个节点。**

```python
from typing import List, TypedDict, Annotated
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field
import operator

# 1. 状态定义
class State(TypedDict):
    input: str
    plan: List[str]
    step_index: int
    results: Annotated[List[tuple], operator.add]
    response: str

# 2. 模型
big_llm = ChatOpenAI(model="gpt-4o", temperature=0)
small_llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# 3. 三个节点
async def planner(state: State):
    p = big_llm.with_structured_output(Plan)
    plan = await p.ainvoke(f"制定计划：{state['input']}")
    return {"plan": plan.steps, "step_index": 0}

async def executor(state: State):
    step = state["plan"][state["step_index"]]
    result = await small_llm.ainvoke(
        f"执行：{step}\n上下文：{state['results']}"
    )
    return {
        "results": [(step, result.content)],
        "step_index": state["step_index"] + 1,
    }

async def replanner(state: State):
    if state["step_index"] >= len(state["plan"]):
        answer = await big_llm.ainvoke(
            f"目标：{state['input']}\n结果：{state['results']}\n生成最终答案。"
        )
        return {"response": answer.content}
    return {}

# 4. 路由
def route(state: State):
    if state.get("response"):
        return "end"
    if state["step_index"] < len(state["plan"]):
        return "execute"
    return "replan"

# 5. 构建图
graph = StateGraph(State)
graph.add_node("planner", planner)
graph.add_node("executor", executor)
graph.add_node("replanner", replanner)

graph.set_entry_point("planner")
graph.add_edge("planner", "executor")
graph.add_conditional_edges("executor", route,
    {"execute": "executor", "replan": "replanner", "end": END})
graph.add_conditional_edges("replanner", route,
    {"execute": "executor", "replan": "replanner", "end": END})

app = graph.compile()

# 6. 运行
result = await app.ainvoke({"input": "对比 ChromaDB 和 Milvus"})
print(result["response"])
```

## 最常犯的三个错误

### 错误1：计划步骤太多

```
❌ 分解成 15 个步骤 → 执行效率极低，LLM 调用次数多
✅ 控制在 3-7 个步骤 → 每步 1-2 次工具调用
```

### 错误2：忘记传入上下文

```python
# ❌ 每步独立执行，没有之前的结果
result = llm.invoke(f"执行：{step}")

# ✅ 带入之前的执行结果
result = llm.invoke(
    f"执行：{step}\n已有结果：{past_results}"
)
```

### 错误3：不做重规划

```
❌ 生成计划 → 盲目执行到底 → 中间出错也不管
✅ 生成计划 → 执行 → 检查 → 调整 → 继续
```
