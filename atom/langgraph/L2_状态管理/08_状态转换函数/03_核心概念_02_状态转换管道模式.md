## 1. 管道模式的定义

### 什么是状态转换管道？

**状态转换管道是多个节点按顺序串联，每个节点对状态做一步转换，最终将原始输入逐步加工为目标输出的执行模式。**

这和 Unix 管道的思想完全一致：

```bash
# Unix 管道：每个命令做一步处理
cat file.txt | grep "error" | sort | uniq -c
```

在 LangGraph 中，每个节点就是管道中的一个"工位"：

```
输入状态 → 节点A(清洗) → 节点B(转换) → 节点C(分析) → 输出状态
```

### 核心特征

1. **串联执行**: 节点按固定顺序依次执行
2. **增量转换**: 每个节点只修改状态的一部分字段
3. **状态透传**: 未修改的字段自动保留到下一步
4. **可组合**: 可以灵活添加、移除、替换管道中的节点

### 类比理解

| 类比 | 说明 |
|------|------|
| **Unix 管道** | `cat | grep | sort | uniq`，每个命令处理一步 |
| **前端中间件链** | Express `app.use(cors).use(auth).use(logger)`，请求依次经过每个中间件 |
| **工厂流水线** | 原料 → 切割工位 → 打磨工位 → 喷漆工位 → 成品 |

## 2. 线性管道（Sequential Pipeline）

线性管道是最常见的模式：节点 A → 节点 B → 节点 C，数据依次流过每个节点。

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    raw_text: str
    cleaned_text: str
    tokens: list[str]
    result: str


def clean_node(state: State) -> dict:
    """第一步：清洗原始文本"""
    text = state["raw_text"].strip().lower()
    return {"cleaned_text": text}


def tokenize_node(state: State) -> dict:
    """第二步：分词"""
    tokens = state["cleaned_text"].split()
    return {"tokens": tokens}


def analyze_node(state: State) -> dict:
    """第三步：分析"""
    result = f"Found {len(state['tokens'])} tokens"
    return {"result": result}


# 构建线性管道：START → clean → tokenize → analyze → END
graph = StateGraph(State)
graph.add_node("clean", clean_node)
graph.add_node("tokenize", tokenize_node)
graph.add_node("analyze", analyze_node)

graph.add_edge(START, "clean")
graph.add_edge("clean", "tokenize")
graph.add_edge("tokenize", "analyze")
graph.add_edge("analyze", END)

app = graph.compile()

result = app.invoke({
    "raw_text": "  Hello World  ",
    "cleaned_text": "", "tokens": [], "result": ""
})
print(result)
# {'raw_text': '  Hello World  ', 'cleaned_text': 'hello world',
#  'tokens': ['hello', 'world'], 'result': 'Found 2 tokens'}
```

**关键设计点**:
1. **每个节点只返回它修改的字段** — `clean_node` 只返回 `{"cleaned_text": ...}`
2. **节点读取上游产出的字段** — `tokenize_node` 读取 `clean_node` 产出的 `cleaned_text`
3. **框架自动保留未修改的字段** — `raw_text` 从头到尾不变，自动透传

## 3. 管道中的状态演变

理解管道的关键是看清状态在每一步如何变化。

```
初始状态:
{raw_text: "  Hello World  ", cleaned_text: "", tokens: [], result: ""}
    ↓ clean_node（只修改 cleaned_text）
{raw_text: "  Hello World  ", cleaned_text: "hello world", tokens: [], result: ""}
    ↓ tokenize_node（只修改 tokens）
{raw_text: "  Hello World  ", cleaned_text: "hello world", tokens: ["hello", "world"], result: ""}
    ↓ analyze_node（只修改 result）
{raw_text: "  Hello World  ", cleaned_text: "hello world", tokens: ["hello", "world"], result: "Found 2 tokens"}
```

每个节点执行后，框架调用 `apply_writes()` 将节点返回的部分更新合并到全局状态。对于没有 Reducer 的字段（`LastValue` Channel），新值直接覆盖旧值。

## 4. 分支管道（Branching Pipeline）

分支管道根据状态内容选择不同的转换路径，通过条件边实现。

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    input_text: str
    language: str
    output: str


def detect_language(state: State) -> dict:
    """检测语言"""
    text = state["input_text"]
    if any(ord(c) > 127 for c in text):
        return {"language": "zh"}
    return {"language": "en"}


def process_chinese(state: State) -> dict:
    return {"output": f"中文处理: {state['input_text']}"}


def process_english(state: State) -> dict:
    return {"output": f"English processing: {state['input_text']}"}


def route_by_language(state: State) -> str:
    """路由函数：根据语言选择处理节点"""
    return "process_zh" if state["language"] == "zh" else "process_en"


graph = StateGraph(State)
graph.add_node("detect", detect_language)
graph.add_node("process_zh", process_chinese)
graph.add_node("process_en", process_english)

graph.add_edge(START, "detect")
graph.add_conditional_edges("detect", route_by_language)
graph.add_edge("process_zh", END)
graph.add_edge("process_en", END)

app = graph.compile()

# 中文输入走中文分支
result = app.invoke({"input_text": "你好世界", "language": "", "output": ""})
print(result["output"])  # "中文处理: 你好世界"
```

**结构**:
```
                    ┌→ process_zh → END
START → detect ─────┤
                    └→ process_en → END
```

**类比**: 快递分拣中心。包裹先经过扫码识别（detect），然后根据目的地走不同的传送带。

## 5. 并行管道（Parallel Pipeline）

并行管道让多个节点同时处理状态的不同部分（Fan-out / Fan-in 模式）。

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
import operator

class State(TypedDict):
    document: str
    keywords: Annotated[list[str], operator.add]
    summary: str
    word_count: int

def extract_keywords(state: State) -> dict:
    """并行任务1：提取关键词"""
    return {"keywords": [w for w in state["document"].split() if len(w) > 4]}

def generate_summary(state: State) -> dict:
    """并行任务2：生成摘要"""
    text = state["document"]
    return {"summary": text[:50] + "..." if len(text) > 50 else text}

def count_words(state: State) -> dict:
    """并行任务3：统计字数"""
    return {"word_count": len(state["document"].split())}

graph = StateGraph(State)
graph.add_node("extract", extract_keywords)
graph.add_node("summarize", generate_summary)
graph.add_node("count", count_words)

# Fan-out: START 同时连接三个节点
graph.add_edge(START, "extract")
graph.add_edge(START, "summarize")
graph.add_edge(START, "count")

# Fan-in: 三个节点都连接到 END
graph.add_edge("extract", END)
graph.add_edge("summarize", END)
graph.add_edge("count", END)

app = graph.compile()
```

**结构**:
```
            ┌→ extract_keywords  ─┐
START ──────┼→ generate_summary  ─┼→ END
            └→ count_words       ─┘
```

**注意**: 如果多个并行节点修改同一个字段，必须使用 Reducer（如 `Annotated[list, operator.add]`）来合并结果，否则会抛出 `InvalidUpdateError`。

**类比**: 多个厨师同时准备不同菜品——一个切菜、一个煮汤、一个烤肉，最后一起上桌。

## 6. 管道组合：线性 + 分支 + 并行

实际项目中，三种模式经常组合使用。

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
import operator


class RAGState(TypedDict):
    query: str
    query_type: str
    docs: Annotated[list[str], operator.add]
    answer: str


def classify_query(state: RAGState) -> dict:
    """线性步骤：分类查询"""
    query_type = "factual" if ("how" in state["query"].lower() or "what" in state["query"].lower()) else "opinion"
    return {"query_type": query_type}

def search_web(state: RAGState) -> dict:
    return {"docs": [f"[Web] Result for: {state['query']}"]}

def search_db(state: RAGState) -> dict:
    return {"docs": [f"[DB] Result for: {state['query']}"]}

def generate_answer(state: RAGState) -> dict:
    return {"answer": f"Based on {len(state['docs'])} sources: {' | '.join(state['docs'])}"}

def route_query(state: RAGState) -> list[str]:
    """事实类查询走双路检索，观点类只走数据库"""
    return ["search_web", "search_db"] if state["query_type"] == "factual" else ["search_db"]


graph = StateGraph(RAGState)
graph.add_node("classify", classify_query)
graph.add_node("search_web", search_web)
graph.add_node("search_db", search_db)
graph.add_node("generate", generate_answer)

graph.add_edge(START, "classify")
graph.add_conditional_edges("classify", route_query)
graph.add_edge("search_web", "generate")
graph.add_edge("search_db", "generate")
graph.add_edge("generate", END)

app = graph.compile()
```

**结构**:
```
                         ┌→ search_web ─┐
START → classify ────────┤              ├→ generate → END
                         └→ search_db  ─┘
```

## 7. 管道设计原则

### 7.1 单一职责

每个节点只做一件事。

```python
# ❌ 坏：一个节点做太多事
def do_everything(state: State) -> dict:
    text = state["raw_text"].strip().lower()
    tokens = text.split()
    return {"cleaned_text": text, "tokens": tokens, "result": f"Found {len(tokens)} tokens"}

# ✅ 好：拆分成三个节点，各司其职
def clean_node(state: State) -> dict:
    return {"cleaned_text": state["raw_text"].strip().lower()}

def tokenize_node(state: State) -> dict:
    return {"tokens": state["cleaned_text"].split()}

def analyze_node(state: State) -> dict:
    return {"result": f"Found {len(state['tokens'])} tokens"}
```

### 7.2 状态透明

通过状态字段传递中间结果，不要用闭包或全局变量。

```python
# ❌ 坏：用全局变量传递
intermediate_result = None
def node_a(state: State) -> dict:
    global intermediate_result
    intermediate_result = process(state["input"])
    return {"output_a": intermediate_result}

# ✅ 好：通过状态字段传递
def node_a(state: State) -> dict:
    return {"intermediate": process(state["input"])}
def node_b(state: State) -> dict:
    return {"output": transform(state["intermediate"])}
```

### 7.3 可插拔

管道中的节点可以轻松添加、移除、替换。需要加一个"去重"步骤？插入一个新节点即可：

```python
# 原始管道：clean → tokenize → analyze
# 插入去重：clean → tokenize → deduplicate → analyze
graph.add_node("deduplicate", deduplicate_node)
graph.add_edge("tokenize", "deduplicate")
graph.add_edge("deduplicate", "analyze")
```

### 7.4 可观测

LangGraph 的 `stream` 模式天然支持追踪每个节点的输入输出：

```python
for event in app.stream({"raw_text": "  Hello World  ", "cleaned_text": "", "tokens": [], "result": ""}):
    print(event)
    # {'clean': {'cleaned_text': 'hello world'}}
    # {'tokenize': {'tokens': ['hello', 'world']}}
    # {'analyze': {'result': 'Found 2 tokens'}}
```

## 8. RAG 场景的管道模式

RAG 系统是管道模式的典型应用：

```
查询 → 查询改写 → 文档检索 → 结果过滤 → 上下文组装 → LLM 生成
```

用 LangGraph 实现时，每个步骤对应一个节点，通过状态字段传递中间产物：

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class RAGState(TypedDict):
    user_query: str
    rewritten_query: str
    docs: list[dict]
    filtered_docs: list[dict]
    answer: str

def rewrite(state: RAGState) -> dict:
    return {"rewritten_query": f"详细解释: {state['user_query']}"}

def retrieve(state: RAGState) -> dict:
    return {"docs": [
        {"content": "Doc1", "score": 0.95},
        {"content": "Doc2", "score": 0.45},
    ]}

def filter_docs(state: RAGState) -> dict:
    return {"filtered_docs": [d for d in state["docs"] if d["score"] > 0.7]}

def generate(state: RAGState) -> dict:
    return {"answer": f"根据 {len(state['filtered_docs'])} 篇文档生成回答"}

graph = StateGraph(RAGState)
for name, fn in [("rewrite", rewrite), ("retrieve", retrieve),
                 ("filter", filter_docs), ("generate", generate)]:
    graph.add_node(name, fn)

graph.add_edge(START, "rewrite")
graph.add_edge("rewrite", "retrieve")
graph.add_edge("retrieve", "filter")
graph.add_edge("filter", "generate")
graph.add_edge("generate", END)
```

**管道模式的优势在 RAG 中特别明显**：每个步骤都可以独立优化。检索效果不好？只改 `retrieve`。过滤太严格？只调 `filter_docs`。不需要动其他节点。

## 9. 常见问题

### Q1: 管道中的节点可以跳过某些步骤吗？

**可以**，使用条件边。比如缓存命中时跳过检索：

```python
def check_cache(state: State) -> str:
    if state.get("cached_result"):
        return "generate"   # 跳过检索
    return "retrieve"       # 正常检索

graph.add_conditional_edges("rewrite", check_cache)
```

### Q2: 管道中某个节点失败了怎么办？

节点抛出异常会中断整个管道。建议在节点内部做降级处理：

```python
def safe_retrieve(state: State) -> dict:
    try:
        docs = vector_store.search(state["query"])
        return {"docs": docs}
    except Exception as e:
        return {"docs": [], "error": str(e)}  # 降级而非崩溃
```

### Q3: 线性管道和直接写一个大函数有什么区别？

| 维度 | 一个大函数 | 管道模式 |
|------|-----------|---------|
| 可观测性 | 只能看到最终结果 | 每个节点的中间状态都可追踪 |
| 可测试性 | 只能端到端测试 | 每个节点可独立单元测试 |
| 可维护性 | 改一处可能影响全局 | 节点间通过状态解耦 |
| 持久化 | 不支持 | 配合 checkpointer 可在任意节点暂停/恢复 |

## 10. 总结

**状态转换管道模式是 LangGraph 中最基础也最强大的组合方式**：

1. **线性管道**: 节点串联，数据依次流过，适合固定流程
2. **分支管道**: 条件边实现路由，不同输入走不同路径
3. **并行管道**: Fan-out/Fan-in，多个节点同时处理不同部分
4. **组合管道**: 三种模式灵活组合，应对真实业务场景

**设计原则**: 单一职责、状态透明、可插拔、可观测。

**在 RAG 中的应用**: 查询改写 → 文档检索 → 结果过滤 → 上下文组装 → LLM 生成，每步独立可优化。
