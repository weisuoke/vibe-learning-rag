## 卡片 1：什么是 Cargo？（直觉理解）

### 🎯 核心点

**Cargo = npm + webpack + tsc 的合体。** 在 Node.js 生态里你需要 npm 管理包、webpack 打包、tsc 编译——在 Rust 里，一个 `cargo` 命令全搞定。

### npm → Cargo 命令速查

| npm | Cargo | 说明 |
|-----|-------|------|
| `npm init` | `cargo init` | 初始化项目 |
| `npm install axios` | `cargo add reqwest` | 添加依赖 |
| `npm install` | `cargo build` | 安装依赖 + 编译 |
| `npm run dev` | `cargo run` | 编译 + 运行 |
| `npm test` | `cargo test` | 运行测试 |
| `npm run build` | `cargo build --release` | 发布构建 |
| `npx eslint .` | `cargo clippy` | 代码 lint |

### ZeroClaw 应用

ZeroClaw 的整个开发流程围绕 Cargo：`cargo build` 编译 30+ 模块、`cargo test` 运行测试、`cargo build --release` 生成体积优化的二进制文件。你不需要 Makefile——Cargo 就是唯一入口。

## 卡片 2：Cargo.toml 解剖（精确定义）

### 🎯 核心点

**Cargo.toml = package.json + tsconfig.json。** 它是项目的"身份证 + 配方表"，定义了项目是谁、依赖什么、怎么编译。

### 最小 Cargo.toml

```toml
[package]
name = "my_app"          # 项目名（= package.json 的 "name"）
version = "0.1.0"        # 语义化版本（= "version"）
edition = "2024"         # Rust 版本（≈ tsconfig 的 "target"）

[dependencies]           # 运行时依赖（= "dependencies"）
serde = "1.0"
tokio = { version = "1", features = ["full"] }

[dev-dependencies]       # 测试/开发依赖（= "devDependencies"）
assert_cmd = "2.0"
```

### 三大核心部分

| 部分 | 作用 | package.json 对应 |
|------|------|-------------------|
| `[package]` | 项目元信息（名称、版本、许可证） | 顶层字段 |
| `[dependencies]` | 运行时依赖 | `"dependencies"` |
| `[features]` | 编译时开关（可选功能） | 无直接对应（≈ webpack DefinePlugin） |

### ZeroClaw 应用

ZeroClaw 的 Cargo.toml 同时定义了 `[[bin]]`（命令行工具）和 `[lib]`（库），有 15+ 个 feature flags，两个 release profile（`release` 体积优先、`release-fast` 性能优先）。所有复杂性都是从这个最小结构逐步叠加的。

## 卡片 3：依赖管理三板斧

### 🎯 核心点

**cargo add 添加依赖，版本规范控制更新范围，Cargo.lock 锁死精确版本。** 三者配合确保"能重现的构建"。

### 第一板斧：cargo add

```bash
cargo add serde                              # 最新版
cargo add serde --features derive            # 带 feature
cargo add tokio -F rt-multi-thread,macros    # -F 是 --features 缩写
cargo add --dev assert_cmd                   # 开发依赖
```

### 第二板斧：版本规范

```toml
[dependencies]
serde = "1.0"            # ≈ npm 的 "^1.0"（>=1.0.0, <2.0.0）
serde = "=1.0.193"       # 精确版本

# 三种来源
tokio = "1.0"                                # crates.io（= npmjs.com）
my_lib = { path = "../my_lib" }              # 本地路径（≈ npm link）
my_lib = { git = "https://..." }             # Git 仓库
```

### 第三板斧：Cargo.lock

**规则**：应用程序（bin）提交 Cargo.lock；库（lib）不提交。

### ZeroClaw 应用

ZeroClaw 提交了 Cargo.lock，有 60+ 个依赖。`git clone` + `cargo build` 即可获得和作者完全一致的构建。

## 卡片 4：mod 声明——Rust 模块的"入口"

### 🎯 核心点

**Rust 不会自动发现文件——你必须用 `mod` 显式声明每个模块。** 这是 Rust 和 TypeScript 最大的不同之一。

### TypeScript vs Rust 模块发现

```typescript
// TypeScript — 自动发现
// 只要文件存在，直接 import 就能用
import { User } from './user';  // ← 不需要任何"声明"
```

```rust
// Rust — 必须显式声明
// 第一步：在 lib.rs 或 mod.rs 中声明模块
mod user;         // ← 告诉编译器："去找 user.rs 或 user/mod.rs"

// 第二步：才能使用
use crate::user::User;
```

### 文件映射规则

```
src/
├── lib.rs          ← crate 根
│   mod user;       → 查找 src/user.rs 或 src/user/mod.rs
│   mod tools;      → 查找 src/tools.rs 或 src/tools/mod.rs
│
├── user.rs         ← 简单模块（单文件）
│
└── tools/          ← 复杂模块（有子模块）
    ├── mod.rs      ← tools 模块的入口
    │   mod shell;  → 查找 src/tools/shell.rs
    │   mod web;    → 查找 src/tools/web.rs
    ├── shell.rs
    └── web.rs
```

### ⚠️ 最常犯的错误

```rust
// ❌ 错误：创建了 src/helper.rs，但没有在 lib.rs 中声明 mod
// 编译器完全忽略这个文件！
// src/helper.rs 存在但从未被编译

// ✅ 正确：在 src/lib.rs 中添加声明
mod helper;  // 现在编译器才知道这个文件存在
```

**记住：** 在 Rust 里，**没有 `mod` 声明的 .rs 文件 = 不存在**。

### ZeroClaw 应用

ZeroClaw 的 `src/lib.rs` 有 **30+ 行 `pub mod` 声明**，从 `pub mod agent` 到 `pub mod util`。每一行都告诉编译器"这里有一个模块，去对应的目录找代码"。如果你新增一个模块目录但忘记在 `lib.rs` 里加 `mod` 声明，编译器会完全忽略它——没有错误，没有警告，就是不编译。

## 卡片 5：use 导入——引用其他模块

### 🎯 核心点

**`use` = TypeScript 的 `import`，但路径用 `::` 而非 `/`，且有三种起点。**

### 三种路径前缀

```rust
// 1. crate:: — 从 crate 根开始（绝对路径）
use crate::tools::ShellTool;     // ≈ import from '@/tools/shell'（@= 项目根）

// 2. super:: — 从父模块开始（相对路径）
use super::Config;               // ≈ import from '../config'

// 3. self:: — 从当前模块开始（当前目录）
use self::helper::format;        // ≈ import from './helper'
```

### 嵌套导入 + 别名

```rust
// 嵌套导入（推荐）
use std::collections::{HashMap, HashSet, BTreeMap};
use std::io::{Read, Write};

// 起别名
use std::collections::HashMap as Map;  // ≈ import { HashMap as Map }
```

### TypeScript 对照

| TypeScript | Rust |
|-----------|------|
| `import { X } from './tools'` | `use crate::tools::X;` |
| `import { X } from '../config'` | `use super::config::X;` |

### ZeroClaw 应用

ZeroClaw 的模块间大量使用 `crate::` 绝对路径导入。例如在 `src/agent/mod.rs` 中：`use crate::providers::Provider;` / `use crate::tools::Tool;` / `use crate::channels::Channel;`。绝对路径在大项目中更清晰——你一眼就知道东西来自哪个顶层模块。

## 卡片 6：pub 可见性——谁能访问什么

### 🎯 核心点

**Rust 默认一切私有——和 TypeScript 的 `export` 相反。** 你必须显式用 `pub` 标记才能让外部访问。

### 五个可见性级别

```rust
// 1. 私有（默认）— 仅当前模块 + 子模块可见
fn private_fn() {}                   // ≈ 不 export 的函数

// 2. pub — 完全公开
pub fn public_fn() {}                // ≈ export function

// 3. pub(crate) — 仅 crate 内部公开
pub(crate) fn internal_fn() {}       // ≈ 不 export 但同 package 内可用

// 4. pub(super) — 仅父模块可见
pub(super) fn parent_only_fn() {}    // ≈ 没有直接对应

// 5. pub(in path) — 指定路径可见
pub(in crate::tools) fn tools_only() {}  // ≈ 没有直接对应
```

### struct 字段的独立可见性（杀手级特性）

```rust
pub struct User {
    pub name: String,        // ✅ 外部可读写
    pub(crate) email: String, // ⚠️ 仅 crate 内部可访问
    password_hash: String,   // 🔒 完全私有，外部连碰都碰不到
}
```

TypeScript 只能控制整个 class 是否 export，无法控制单个字段的可见性。Rust 的每个字段都可以独立设置——这是**编译时强制的封装**。

### 默认私有的哲学

模块内的函数、struct 默认私有，只有加 `pub` 的才对外可见。**暴露最少的 API，隐藏实现细节**——编译器帮你强制执行。

### ZeroClaw 应用

ZeroClaw 大量使用 `pub(crate)` 来区分"给外部用户用的 API"和"crate 内部共享的工具函数"。例如 Agent 的核心循环对外暴露 `pub fn run()`，但内部的 `classify_message()`、`dispatch_tool()` 等方法都是 `pub(crate)` 或私有的——你只需要知道怎么 `run()`，不需要关心内部调度逻辑。

## 卡片 7：pub use——清洁你的公共 API

### 🎯 核心点

**`pub use` = Re-export，把深层模块的类型"提升"到顶层，让用户用更短的路径访问。** 和 TypeScript 的 barrel files（index.ts）一模一样。

### TypeScript barrel file vs Rust pub use

```typescript
// TypeScript: src/tools/index.ts（barrel file）
export { ShellTool } from './shell';
export { WebTool } from './web';
export { Tool } from './traits';

// 用户导入
import { ShellTool, Tool } from './tools';  // 不需要知道内部结构
```

```rust
// Rust: src/tools/mod.rs
mod shell;
mod web;
mod traits;

pub use shell::ShellTool;     // re-export
pub use web::WebTool;         // re-export
pub use traits::Tool;         // re-export

// 用户导入
use crate::tools::{ShellTool, Tool};  // 不需要知道内部结构
```

### 效果对比

```rust
// 没有 pub use → 暴露内部结构
use zeroclaw::tools::shell::ShellTool;         // 太长！

// 有 pub use → 干净 API
use zeroclaw::tools::ShellTool;      // 简洁！
use zeroclaw::Tool;                  // 直接从 crate 根导入！
```

### ZeroClaw 应用

ZeroClaw 的 `lib.rs` 使用选择性 re-export 构建干净的公共 API：

```rust
// lib.rs
pub use config::Config;
pub use agent::{Agent, AgentBuilder};
pub use tools::Tool;
pub use channels::Channel;
pub use providers::Provider;
```

外部使用者只需要 `use zeroclaw::Agent` 就能开始——不需要知道 Agent 实际定义在 `src/agent/builder.rs` 的第几行。这就是 `pub use` 的力量：**隐藏内部结构，暴露干净接口**。

## 卡片 8：Workspace——多 crate 协作

### 🎯 核心点

**Workspace 让多个 crate 共享依赖版本、共享编译缓存、一起测试——就像 npm/pnpm workspace。**

### 什么时候需要 Workspace？

| 场景 | 方案 |
|------|------|
| 小项目，代码 < 5000 行 | 单 crate，用 mod 组织 |
| 中型项目，编译慢了 | 拆成 workspace，多 crate 并行编译 |
| 需要独立发布子库 | 拆成 workspace，每个 crate 独立版本 |
| 需要强隔离（不同编译目标） | 独立项目（如 firmware） |

### Workspace 配置

```toml
# 根目录 Cargo.toml
[workspace]
members = [
    ".",               # 主 crate（当前目录）
    "crates/robot-kit", # 子 crate
]
resolver = "2"         # 推荐：更好的 feature 解析

# 共享依赖版本（避免版本冲突）
[workspace.dependencies]
serde = { version = "1.0", features = ["derive"] }
tokio = { version = "1", features = ["full"] }
```

```toml
# crates/robot-kit/Cargo.toml — 子 crate 继承 workspace 依赖
[package]
name = "robot-kit"
version = "0.1.0"

[dependencies]
serde = { workspace = true }    # ← 继承版本！不用重复写
tokio = { workspace = true }
```

### Workspace 目录结构

```
my_project/
├── Cargo.toml          ← workspace 根配置
├── Cargo.lock          ← 共享锁文件（所有 crate 共用）
├── target/             ← 共享构建缓存（不重复编译）
├── src/                ← 主 crate 源码
│   ├── lib.rs
│   └── main.rs
└── crates/
    └── robot-kit/      ← 子 crate
        ├── Cargo.toml
        └── src/
            └── lib.rs
```

### npm workspace 对照

| 概念 | npm/pnpm workspace | Cargo workspace |
|------|--------------------|-----------------------|
| 根配置 | 根 package.json | 根 Cargo.toml |
| 成员声明 | `"workspaces": [...]` | `members = [...]` |
| 共享依赖版本 | 根 dependencies | `workspace.dependencies` |
| 继承依赖 | 无（需手动同步） | `{ workspace = true }` |
| 共享锁文件 | ✅ pnpm-lock.yaml | ✅ Cargo.lock |
| 共享构建缓存 | ❌（各自 node_modules） | ✅ 共享 target/ |
| 运行子项目命令 | `npm -w crate1 test` | `cargo test -p crate1` |

### ZeroClaw 应用

ZeroClaw 是一个 workspace，包含主 crate（`zeroclaw`）和子 crate（`robot-kit`）。`robot-kit` 是独立的机器人控制库，有自己的 feature flags（`ros2` / `gpio` / `lidar` / `vision`）。它们共享 `Cargo.lock` 和 `target/` 目录，确保依赖版本一致且不重复编译。此外还有三个 firmware crate（ESP32 / STM32），它们因为需要嵌入式交叉编译，故意不在 workspace 中。

## 卡片 9：Feature Flags——编译时的开关

### 🎯 核心点

**Feature flags 是编译时的条件编译开关——让用户只编译需要的功能。** 不像 JS 的 `if (process.env.X)` 在运行时判断，Rust 的 feature 在编译阶段就决定了——不需要的代码根本不存在于二进制文件中。

### 定义 + 使用

```toml
# Cargo.toml — 定义 features
[features]
default = []                    # 默认不启用任何可选功能
hardware = ["dep:nusb"]         # 启用 hardware 时引入 nusb 依赖
postgres = ["dep:sqlx"]         # 启用 postgres 时引入 sqlx 依赖
full = ["hardware", "postgres"] # 组合 feature

[dependencies]
nusb = { version = "0.1", optional = true }     # optional = true！
sqlx = { version = "0.7", optional = true }
```

```rust
// 代码中 — 使用 #[cfg(feature)] 条件编译
#[cfg(feature = "hardware")]
mod hardware;                    // 只在 hardware feature 启用时编译

#[cfg(feature = "hardware")]
pub fn discover_devices() -> Vec<Device> {
    // 这个函数只在 cargo build --features hardware 时存在
    nusb::list_devices().collect()
}

// 没有启用 hardware feature？这个函数和 nusb 依赖根本不会被编译。
```

### 如何启用

```bash
cargo build                            # 只编译默认 features
cargo build --features hardware        # 启用 hardware
cargo build --features "hardware,postgres"  # 启用多个
cargo build --all-features             # 启用所有（CI 测试用）
cargo build --no-default-features      # 禁用默认
```

### ⚠️ Additive 原则——铁律

**Feature 只能增加功能，不能互斥。**

```toml
# ❌ 错误设计：两个 feature 互斥
[features]
backend-sqlite = []
backend-postgres = []
# 同时启用两个会怎样？—— 编译错误或未定义行为！

# ✅ 正确设计：additive（可叠加）
[features]
sqlite = ["dep:rusqlite"]     # 增加 SQLite 支持
postgres = ["dep:sqlx"]       # 增加 PostgreSQL 支持
# 同时启用两个？没问题，都支持！
```

为什么？因为在 Cargo 的依赖解析中，多个依赖可能请求同一个 crate 的不同 feature——Cargo 会取**并集**。如果 feature 互斥，合并就会出问题。

### ZeroClaw 的 15+ Features

```toml
# ZeroClaw Cargo.toml（简化版）
[features]
default = []                    # 最小构建

# 硬件
hardware = ["dep:nusb", "dep:tokio-serial"]
peripheral-rpi = ["dep:rppal"]
probe = ["dep:probe-rs"]

# 通信通道
channel-matrix = ["dep:matrix-sdk"]
channel-lark = ["dep:prost"]

# 存储
memory-postgres = ["dep:sqlx"]

# 可观测性
observability-otel = ["dep:opentelemetry"]

# 浏览器自动化
browser-native = ["dep:chromiumoxide"]

# 安全沙箱
sandbox-landlock = ["dep:landlock"]

# RAG
rag-pdf = ["dep:pdf-extract"]
```

每个 feature 控制一组可选依赖。普通用户 `cargo install zeroclaw` 只编译核心功能（小而快）；需要硬件控制的用户加 `--features hardware`；需要 PostgreSQL 的加 `--features memory-postgres`。**不需要的代码和依赖完全不存在于最终二进制中。**

### ZeroClaw 应用

ZeroClaw 的 feature flags 是其"模块化哲学"的核心——一个框架同时支持树莓派 GPIO、Matrix 聊天、PostgreSQL 存储、PDF 解析等 15+ 种可选能力，但任何一个用户都只需要编译自己用到的部分。这就像一辆车可以选配天窗、座椅加热、自动驾驶——你只为你选的配置付费（编译时间和二进制体积）。
