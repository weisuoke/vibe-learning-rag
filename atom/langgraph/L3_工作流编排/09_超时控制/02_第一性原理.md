# 超时控制的第一性原理

## 什么是第一性原理？

**第一性原理**：回到问题最基础、不可再分的事实，不依赖表层经验来理解系统。

对“超时控制”来说，最基础的问题不是“默认 timeout 多少秒”，而是：

> **任何工作流里的等待都不是免费的。**

等待会消耗：
- 用户耐心；
- 线程 / 协程 / 连接池；
- 平台请求窗口；
- Token 成本；
- 故障恢复空间。

所以 timeout 的本质不是“报错机制”，而是“资源预算边界”。

---

## 1. 最基础的定义

**超时控制 = 为等待设置 deadline，并在 deadline 被突破时执行预设策略。**

仅此而已。

这里面包含三个不可再分的元素：

1. **谁在等？**
   - 节点在等 HTTP；
   - 图在等当前 step 完成；
   - 前端在等后端回复；
   - 平台在等部署就绪。

2. **等多久？**
   - 1 秒、10 秒、300 秒，都是预算，不是真理。

3. **等不到怎么办？**
   - 取消；
   - 重试；
   - fallback；
   - interrupt；
   - 后台继续跑，前台改轮询。

---

## 2. 为什么需要超时控制？

### 根本问题：分布式世界里“永远等下去”是最差策略

假设你没有 timeout：

- 一个工具服务半死不活，连接一直不返回；
- 一个子图卡在某个等待上；
- 一个前端请求挂了两分钟；
- 一个多代理循环不断空转。

如果系统没有 deadline，它就无法区分：
- “暂时慢”；
- “已经坏”；
- “应该切换方案”；
- “应该把控制权交回用户”。

因此 timeout 的价值不是“更快失败”，而是：

> **让系统在不确定条件下仍然保有决策权。**

---

## 3. 超时控制的三层价值

### 价值 1：限制资源消耗

没有 timeout，一个坏掉的依赖会持续占着：
- worker；
- 连接池；
- 线程；
- event loop 时间片；
- 用户会话窗口。

加了 timeout，相当于你明确说：

> “这件事最多只值得再等 N 秒。”

### 价值 2：暴露系统边界

只要系统里开始出现 timeout，你就会被迫回答：
- 是图层慢，还是 I/O 慢？
- 是平台窗口太短，还是节点逻辑太粗？
- 是该拆步骤，还是该异步化？

timeout 是最好的架构显影剂。

### 价值 3：触发恢复流程

deadline 被突破以后，系统才能进入下一步：
- 重试；
- 走 fallback；
- 保留 checkpoint；
- 返回部分结果；
- 改成后台任务。

如果没有 timeout，恢复策略根本没有机会启动。

---

## 4. 从第一性原理推导 LangGraph 的 timeout 设计

### 推理链

```text
1. 工作流本质上是“若干等待 + 若干状态转移”
   ↓
2. 任何等待都必须有预算，否则资源会被无界占用
   ↓
3. 预算应该和等待层级匹配
   ↓
4. 图级等待要有图级 budget（step_timeout）
   ↓
5. I/O 等待要有 I/O budget（httpx.Timeout / wait_for）
   ↓
6. 用户等待要有交互层 budget（前端请求窗口 / polling）
   ↓
7. 一旦 budget 超限，就要进入恢复流程
   ↓
8. 恢复流程可以是 retry / fallback / interrupt / resume
```

所以 LangGraph 的合理超时设计一定是分层的。

---

## 5. 为什么 `step_timeout` 不是全部答案？

因为 `step_timeout` 只回答了一个问题：

> **“当前 step 整体最多等多久？”**

它没有回答：
- 单个 HTTP 请求最多读多久？
- 连接池获取最多等多久？
- 人工审批要不要继续占着同步请求？
- 前端等 2 分钟是否可接受？

这也是为什么源码层的 `step_timeout` 和官方文档里的 `interrupt` / durable execution 会同时存在。

## 6. LangGraph 超时控制的第一性结论

### 结论一：timeout 是预算，不是装饰参数

如果团队里有人说：

“先随便配个 60 秒吧。”

你要追问：
- 为什么是 60 秒？
- 这 60 秒覆盖哪一层？
- 60 秒到了以后做什么？

### 结论二：不同等待必须由不同层负责

| 等待类型 | 推荐控制层 |
|---|---|
| 当前 step 内并发任务 | `graph.step_timeout` |
| 单次网络调用 | `httpx.Timeout` |
| 单次协程逻辑 | `asyncio.wait_for()` |
| 人工等待 | `interrupt` + resume |
| 前端长请求 | 异步 run + polling |

### 结论三：正确的超时设计一定伴随恢复设计

timeout 不是终点，而是分叉点：
- 可重试？
- 可降级？
- 可中断恢复？
- 可转后台？

只有回答了这些问题，timeout 才算设计完成。

---

## 7. 用生活化方式再理解一次

你去医院看急诊，系统其实也有 timeout：

- 分诊台：最多等多久进入下一步；
- 抽血窗口：单项检查多久没出结果要重抽；
- 医生会诊：长时间等待要不要转住院；
- 你本人：等太久是否先回家，改成电话通知。

没有任何一家医院会只设置一个“全院统一等待时长”，因为等待发生在不同层级。

LangGraph 工作流也一样。

---

## 8. 一句话总结第一性原理

**超时控制的本质，是为不同层级的等待建立预算边界，并把超时后的恢复动作前置设计好。**
