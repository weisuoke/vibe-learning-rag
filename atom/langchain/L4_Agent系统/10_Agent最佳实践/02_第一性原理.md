# Agent最佳实践 - 第一性原理

---

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题。

---

## Agent 最佳实践的第一性原理

### 1. 最基础的定义

**Agent = LLM + 工具 + 循环 = 一个非确定性的自主系统**

仅此而已！

但正是这个"非确定性"，决定了 Agent 在生产环境中会面临一系列传统软件不需要担心的问题。

### 2. 为什么需要最佳实践？

**核心问题：Agent 是一个不可完全预测的系统，它在生产环境中必然会出错、可能会泄密、一定会烧钱。**

传统软件是**确定性**的——相同输入必定产生相同输出。你可以用单元测试覆盖每个分支，bug 修了就是修了。

Agent 是**非确定性**的：
- LLM 每次可能给出不同的推理路径
- 工具调用可能因网络/限流/数据变化而失败
- 多步循环中，每一步的不确定性会**累积放大**

```
传统软件的不确定性：
  f(x) = y  （几乎总是确定的）
  唯一的不确定性来自外部依赖（网络、数据库）

Agent 的不确定性：
  LLM(x) = y₁ 或 y₂ 或 y₃...  （每次可能不同）
  ↓
  Tool(y₁) = z₁  （工具也可能失败）
  ↓
  LLM(z₁) = w₁ 或 w₂...  （不确定性继续累积）
  ↓
  ...重复 N 次...
  ↓
  最终结果 = ？？？

N 步循环的不确定性 ≈ 单步不确定性 ^ N
```

### 3. 从不确定性推导出四大工程需求

#### 需求 1：可靠性（因为 Agent 一定会出错）

**推理链：**

```
1. Agent 的每一步都可能出错（LLM 幻觉、API 超时、工具异常）
   ↓
2. 多步循环中，出错概率随步数指数增长
   ↓
3. 因此，单次执行不可靠
   ↓
4. 因此，需要：
   - 重试：同一操作失败后自动再试（ModelRetryMiddleware）
   - 降级：主方案失败后切换备用方案（ModelFallbackMiddleware）
   - 限制：设置上限防止无限循环和成本失控（CallLimitMiddleware）
```

**类比：** 飞机有多引擎、多套飞控系统、燃油限制——不是因为每个组件一定会坏，而是因为**长时间飞行中出问题的概率不可忽略**。

#### 需求 2：安全性（因为 Agent 能自主操作）

**推理链：**

```
1. Agent 能自主调用工具（搜索、数据库、API、Shell）
   ↓
2. 自主操作意味着可能被恶意利用或犯错
   ↓
3. LLM 无法 100% 判断操作是否安全（可能被提示注入）
   ↓
4. 因此，需要：
   - PII 检测：防止敏感信息泄露（PIIMiddleware）
   - 人工审核：关键操作需人工确认（HumanInTheLoopMiddleware）
   - 沙箱隔离：不可信代码在受限环境执行（ShellToolMiddleware）
   - 权限控制：最小权限原则，每个 Agent 只能用必需的工具
```

**类比：** 公司新员工有入职培训（提示词）、权限限制（工具白名单）、重要决策需要领导审批（HITL）——不是不信任员工，而是**系统性的风险管理**。

#### 需求 3：上下文管理（因为 LLM 有容量限制）

**推理链：**

```
1. LLM 有固定的上下文窗口（Context Window），如 128K Token
   ↓
2. Agent 多步推理会不断积累历史消息
   ↓
3. 长对话的历史消息会逐渐填满上下文窗口
   ↓
4. 上下文溢出 → LLM 性能急剧下降或直接报错
   ↓
5. 因此，需要：
   - 对话摘要：将旧消息压缩为摘要（SummarizationMiddleware）
   - 上下文编辑：选择性保留/删除消息
   - Token 预算：设定 Token 使用上限
```

**类比：** 你的电脑内存有限（上下文窗口），运行太多程序（历史消息）会变慢甚至卡死。解决方案：关闭不用的程序（删除旧消息）、把不常用的数据存到硬盘（摘要压缩）。

#### 需求 4：可组合性（因为需求是多维的）

**推理链：**

```
1. 生产 Agent 同时需要可靠性 + 安全性 + 上下文管理
   ↓
2. 如果把所有逻辑写在 Agent 核心代码里，代码会变得臃肿不可维护
   ↓
3. 因此，需要一种架构模式来：
   - 将每个关注点独立为一个模块
   - 支持按需组合（用哪个加哪个）
   - 保持 Agent 核心代码简洁
   ↓
4. 这就是 Middleware 架构：
   middleware=[Retry, Fallback, PII, Summarization, ...]
```

**类比：** Express.js 的中间件系统——`app.use(cors())`, `app.use(auth())`, `app.use(rateLimit())`——每个中间件独立负责一个关注点，按需组合。

### 4. 完整推导链

```
Agent = LLM + 工具 + 循环 = 非确定性系统
  ↓
  ├─→ 一定会出错 ────→ 需要重试 + 降级 + 限制 ──→ 可靠性 Middleware
  ├─→ 能自主操作 ────→ 需要审核 + 脱敏 + 沙箱 ──→ 安全性 Middleware
  ├─→ 有容量限制 ────→ 需要摘要 + 编辑 + 预算 ──→ 上下文 Middleware
  └─→ 需求是多维的 ──→ 需要解耦 + 组合 + 可插拔 ─→ Middleware 架构
                                                       ↓
                                                  create_agent(
                                                    middleware=[...]
                                                  )
```

### 5. 一句话总结第一性原理

**Agent 是非确定性自主系统，这决定了它必然会出错、可能会泄密、一定会受限——Middleware 架构通过将可靠性/安全性/上下文管理解耦为独立中间件并按需组合，是应对这些挑战的工程最优解。**

---

## 对比：为什么传统软件不需要这些？

| 维度 | 传统 Web 应用 | Agent 系统 |
|------|-------------|-----------|
| **确定性** | 相同输入 → 相同输出 | 相同输入 → 可能不同输出 |
| **错误来源** | 主要是外部依赖（数据库/网络） | LLM 幻觉 + 外部依赖 + 推理错误 |
| **安全威胁** | SQL 注入、XSS | 提示注入、工具滥用、PII 泄露 |
| **资源消耗** | 可预测（CPU/内存） | 不可预测（Token 消耗随推理步数变化） |
| **测试方式** | 单元测试 + 集成测试 | 还需要对抗性测试（红队）+ LLM 评估 |

**传统软件的"最佳实践"（SOLID、设计模式、测试金字塔）并不够用。Agent 需要额外的、专门针对非确定性的工程实践。**
