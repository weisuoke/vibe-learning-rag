# 核心概念5：命名空间机制

> 每个子图在 Checkpoint 系统中拥有独立的命名空间，通过 `|` 和 `:` 两个分隔符构建层级路径——这是子图状态隔离和独立持久化的底层基础。

## 1. 为什么需要命名空间？

### 核心问题：多个子图的 Checkpoint 怎么区分？

LangGraph 的 Checkpoint 机制会在每一步保存图的完整状态。当图中有子图时，问题出现了：

```
父图执行 step 1 → 保存 checkpoint
  └─ 子图A 执行 step 1 → 保存 checkpoint
  └─ 子图A 执行 step 2 → 保存 checkpoint
父图执行 step 2 → 保存 checkpoint
  └─ 子图B 执行 step 1 → 保存 checkpoint
```

如果所有 checkpoint 都存在同一个"平面"空间里，父图和子图的 checkpoint 会混在一起，无法区分"这个 checkpoint 属于哪个图的哪一步"。

**命名空间的作用：给每个图（包括子图）一个唯一的"地址"，让 checkpoint 各归各位。**

### 一句话定义

**命名空间是一个层级路径字符串，用 `|` 分隔层级、用 `:` 分隔节点名和任务ID，为每个子图的 Checkpoint 提供独立的存储空间。**

### 双重类比

**前端类比：CSS 的命名空间 / BEM 命名规范**

```css
/* BEM: Block__Element--Modifier */
.card__header--active { }
.card__body--collapsed { }

/* LangGraph: parent_node:task_id|child_node:task_id */
"agent:abc123|researcher:def456"
```

就像 BEM 用 `__` 和 `--` 分隔不同层级，LangGraph 用 `|` 和 `:` 分隔图的层级和任务。目的一样：避免命名冲突，让每个元素都有唯一标识。

**日常生活类比：邮政地址系统**

```
中国 | 北京市 | 海淀区 | 中关村大街1号
```

邮政地址用层级结构确保每个地址唯一。LangGraph 的命名空间也是一样——从根图到最深层子图，每一层都有自己的标识，组合起来就是全局唯一的"地址"。

## 2. 两个核心常量

### 源码定义

```python
# 源码：libs/langgraph/langgraph/_internal/_constants.py

import sys

NS_SEP = sys.intern("|")   # 命名空间分隔符：分隔层级
NS_END = sys.intern(":")   # 命名空间终止符：分隔节点名和 task_id
```

[来源: sourcecode/langgraph/libs/langgraph/langgraph/_internal/_constants.py]

### 为什么用 `sys.intern()`？

`sys.intern()` 是 Python 的字符串驻留机制——确保相同内容的字符串在内存中只有一份。因为 `|` 和 `:` 在运行时会被频繁使用（每次子图执行都要拼接命名空间），驻留可以：

1. **节省内存**：所有 `"|"` 引用同一个对象
2. **加速比较**：字符串比较变成指针比较（`is` 而不是 `==`）

### 两个符号的职责

```
agent:abc123|researcher:def456
  │    │    │     │        │
  │    │    │     │        └─ task_id（任务唯一标识）
  │    │    │     └─ 子图节点名
  │    │    └─ NS_SEP（层级分隔符）
  │    └─ task_id（任务唯一标识）
  └─ 父图节点名

NS_END (:) → 分隔 "谁" 和 "哪次执行"
NS_SEP (|) → 分隔 "哪一层"
```

## 3. 命名空间的格式与示例

### 格式规则

```
{node_name}{NS_END}{task_id}[{NS_SEP}{child_node_name}{NS_END}{child_task_id}]...
```

即：`节点名:任务ID[|子节点名:子任务ID]...`

### 具体示例

| 命名空间字符串 | 含义 |
|---------------|------|
| `""` | 根图（最顶层） |
| `"agent:abc123"` | 根图中名为 `agent` 的子图节点，任务ID为 `abc123` |
| `"agent:abc123\|researcher:def456"` | `agent` 子图中名为 `researcher` 的孙子图节点 |
| `"agent:abc123\|researcher:def456\|tool:ghi789"` | 三层嵌套：agent → researcher → tool |

### 可视化

```
根图 (namespace: "")
├── node_1（普通节点，不产生子命名空间）
├── agent（子图节点）
│   namespace: "agent:abc123"
│   ├── planner（普通节点）
│   ├── researcher（子图节点）
│   │   namespace: "agent:abc123|researcher:def456"
│   │   ├── search（普通节点）
│   │   └── analyze（普通节点）
│   └── writer（普通节点）
└── node_3（普通节点）
```

## 4. 命名空间的组装过程

### 源码解析

```python
# 源码（简化）：libs/langgraph/langgraph/pregel/main.py

# 当执行一个任务（task）时，组装其命名空间
task_ns = f"{task.name}{NS_END}{task.id}"
# 例如：task_ns = "agent:abc123"

# 如果有父命名空间，拼接上去
if parent_ns:
    task_ns = f"{parent_ns}{NS_SEP}{task_ns}"
    # 例如：task_ns = "agent:abc123|researcher:def456"
```

[来源: sourcecode/langgraph/libs/langgraph/langgraph/pregel/main.py]

### 组装流程

```
第一层（根图执行子图节点 "agent"）：
  parent_ns = ""（根图没有命名空间）
  task.name = "agent"
  task.id = "abc123"
  → task_ns = "agent:abc123"

第二层（agent 子图执行子图节点 "researcher"）：
  parent_ns = "agent:abc123"
  task.name = "researcher"
  task.id = "def456"
  → task_ns = "agent:abc123|researcher:def456"

第三层（researcher 子图执行子图节点 "tool"）：
  parent_ns = "agent:abc123|researcher:def456"
  task.name = "tool"
  task.id = "ghi789"
  → task_ns = "agent:abc123|researcher:def456|tool:ghi789"
```

### task_id 的来源

task_id 是每次任务执行时生成的唯一标识（通常是 UUID）。它确保同一个子图节点在不同执行轮次中有不同的命名空间：

```
第一次执行 agent 子图：namespace = "agent:abc123"
第二次执行 agent 子图：namespace = "agent:xyz789"
```

## 5. 命名空间的命名限制

### 节点名不能包含 `|` 和 `:`

因为这两个字符是命名空间的保留分隔符，节点名中如果包含它们会导致解析混乱：

```python
# ❌ 错误：节点名包含保留字符
builder.add_node("agent|v2", some_func)    # | 是 NS_SEP
builder.add_node("agent:main", some_func)  # : 是 NS_END

# ✅ 正确：使用其他分隔符
builder.add_node("agent_v2", some_func)
builder.add_node("agent-main", some_func)
builder.add_node("agent.main", some_func)
```

### 为什么这个限制很重要？

命名空间的解析依赖于 `|` 和 `:` 的唯一性：

```python
# 解析命名空间
namespace = "agent:abc123|researcher:def456"

# 按 | 分割层级
levels = namespace.split("|")
# ["agent:abc123", "researcher:def456"]

# 按 : 分割节点名和 task_id
for level in levels:
    name, task_id = level.split(":")
    # ("agent", "abc123"), ("researcher", "def456")
```

如果节点名包含 `|` 或 `:`，上面的解析就会出错。

## 6. 相关的 Config 键

命名空间通过 Config 系统在图的执行过程中传递。以下是与命名空间相关的关键配置键：

### CONFIG_KEY_CHECKPOINT_NS

```python
# 当前子图的命名空间路径
CONFIG_KEY_CHECKPOINT_NS = "__pregel_checkpoint_ns"
```

每个子图在执行时，config 中会自动设置这个键，值就是当前的命名空间字符串。

### CONFIG_KEY_CHECKPOINT_MAP

```python
# 命名空间 → checkpoint_id 的映射
CONFIG_KEY_CHECKPOINT_MAP = "__pregel_checkpoint_map"
```

这个映射记录了每个命名空间对应的最新 checkpoint ID，用于恢复执行时定位正确的 checkpoint。

### CONFIG_KEY_CHECKPOINTER

```python
# 共享的 checkpoint 存储后端
CONFIG_KEY_CHECKPOINTER = "__pregel_checkpointer"
```

所有层级的图共享同一个 checkpointer 实例，但通过命名空间隔离各自的 checkpoint 数据。

### CONFIG_KEY_STREAM

```python
# 统一的流式输出通道
CONFIG_KEY_STREAM = "__pregel_stream"
```

所有层级的图共享同一个 stream，子图的输出会带上命名空间前缀，让消费者知道输出来自哪个子图。

### 它们如何协作

```
┌─────────────────────────────────────────────────────┐
│                  共享资源                             │
│  CONFIG_KEY_CHECKPOINTER → 同一个存储后端            │
│  CONFIG_KEY_STREAM → 同一个输出通道                   │
│                                                      │
│  ┌──────────────────┐  ┌──────────────────┐         │
│  │ 根图              │  │ 子图              │         │
│  │ NS: ""           │  │ NS: "agent:abc"  │         │
│  │ checkpoint: cp_1 │  │ checkpoint: cp_2 │         │
│  └──────────────────┘  └──────────────────┘         │
│                                                      │
│  CONFIG_KEY_CHECKPOINT_MAP:                          │
│    "" → "cp_1"                                       │
│    "agent:abc" → "cp_2"                              │
└─────────────────────────────────────────────────────┘
```

## 7. 命名空间实现的三大能力

### 能力1：独立 Checkpoint

每个子图有自己的 checkpoint 序列，互不干扰：

```
根图 checkpoints:
  cp_root_1 → cp_root_2 → cp_root_3

子图A checkpoints (ns: "agent:abc"):
  cp_a_1 → cp_a_2

子图B checkpoints (ns: "tool:def"):
  cp_b_1
```

这意味着：
- 子图可以独立回滚（时间旅行）
- 子图的 checkpoint 不会污染父图
- 父图恢复时，子图也能正确恢复到对应状态

### 能力2：状态隔离

命名空间确保不同子图的状态完全隔离：

```python
# 两个子图可以有同名的状态字段，互不冲突
class SubgraphA_State(TypedDict):
    count: int  # 子图A 的 count

class SubgraphB_State(TypedDict):
    count: int  # 子图B 的 count

# 因为命名空间不同，它们的 count 存储在不同的位置：
# "subA:xxx" → count = 10
# "subB:yyy" → count = 20
```

### 能力3：子图流式识别

通过命名空间，消费者可以区分流式输出来自哪个子图：

```python
# stream_mode="subgraphs" 时的输出格式
for namespace, chunk in graph.stream(input, stream_mode="subgraphs"):
    if namespace == ():
        print(f"根图输出: {chunk}")
    elif namespace == ("agent:abc123",):
        print(f"agent 子图输出: {chunk}")
    elif namespace == ("agent:abc123", "researcher:def456"):
        print(f"researcher 子图输出: {chunk}")
```

## 8. 代码示例：观察命名空间

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver


# ===== 子图 =====
class SubState(TypedDict):
    value: str


def sub_node(state: SubState, config: dict):
    """子图节点：打印当前命名空间"""
    ns = config.get("configurable", {}).get("checkpoint_ns", "")
    print(f"  子图命名空间: '{ns}'")
    return {"value": state["value"] + " → sub"}


sub_builder = StateGraph(SubState)
sub_builder.add_node("sub_node", sub_node)
sub_builder.add_edge(START, "sub_node")
sub_builder.add_edge("sub_node", END)
sub = sub_builder.compile()


# ===== 父图 =====
class ParentState(TypedDict):
    value: str


def parent_node(state: ParentState, config: dict):
    """父图节点：打印当前命名空间"""
    ns = config.get("configurable", {}).get("checkpoint_ns", "")
    print(f"父图命名空间: '{ns}'")
    return {"value": state["value"] + " → parent"}


parent_builder = StateGraph(ParentState)
parent_builder.add_node("parent_node", parent_node)
parent_builder.add_node("sub", sub)
parent_builder.add_edge(START, "parent_node")
parent_builder.add_edge("parent_node", "sub")
parent_builder.add_edge("sub", END)

# 使用 MemorySaver 启用 checkpoint
checkpointer = MemorySaver()
parent = parent_builder.compile(checkpointer=checkpointer)

result = parent.invoke(
    {"value": "start"},
    config={"configurable": {"thread_id": "demo"}}
)

# 输出：
# 父图命名空间: ''
#   子图命名空间: 'sub:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
```

### 观察要点

1. **父图节点**的命名空间是空字符串 `""`（根图）
2. **子图节点**的命名空间是 `"sub:任务ID"`（`sub` 是节点名，后面是自动生成的 UUID）
3. 命名空间是自动管理的，不需要手动设置

## 9. 在 RAG 开发中的应用

### 场景：多阶段 RAG Pipeline 的独立 Checkpoint

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver


# 检索子图 — 有自己的 checkpoint 命名空间
class RetrievalState(TypedDict):
    query: str
    docs: list[str]

def retrieve(state: RetrievalState):
    # 检索逻辑...
    return {"docs": ["doc1", "doc2", "doc3"]}

retrieval = StateGraph(RetrievalState)
retrieval.add_node("retrieve", retrieve)
retrieval.add_edge(START, "retrieve")
retrieval.add_edge("retrieve", END)
retrieval_graph = retrieval.compile()


# 生成子图 — 有自己的 checkpoint 命名空间
class GenerationState(TypedDict):
    docs: list[str]
    answer: str

def generate(state: GenerationState):
    # 生成逻辑...
    return {"answer": f"基于 {len(state['docs'])} 篇文档的回答"}

generation = StateGraph(GenerationState)
generation.add_node("generate", generate)
generation.add_edge(START, "generate")
generation.add_edge("generate", END)
generation_graph = generation.compile()


# 父图 — 编排检索和生成
class RAGState(TypedDict):
    query: str
    docs: list[str]
    answer: str

rag = StateGraph(RAGState)
rag.add_node("retrieval", retrieval_graph)   # ns: "retrieval:xxx"
rag.add_node("generation", generation_graph) # ns: "generation:yyy"
rag.add_edge(START, "retrieval")
rag.add_edge("retrieval", "generation")
rag.add_edge("generation", END)

app = rag.compile(checkpointer=MemorySaver())
```

### 命名空间带来的好处

```
Checkpoint 存储结构：

thread_id: "user_123"
├── namespace: ""（父图）
│   ├── cp_1: {query: "...", docs: [], answer: ""}
│   ├── cp_2: {query: "...", docs: ["doc1",...], answer: ""}
│   └── cp_3: {query: "...", docs: ["doc1",...], answer: "基于..."}
│
├── namespace: "retrieval:abc"（检索子图）
│   ├── cp_r1: {query: "...", docs: []}
│   └── cp_r2: {query: "...", docs: ["doc1","doc2","doc3"]}
│
└── namespace: "generation:def"（生成子图）
    ├── cp_g1: {docs: ["doc1",...], answer: ""}
    └── cp_g2: {docs: ["doc1",...], answer: "基于..."}
```

**实际价值：**
- 检索失败时，可以只回滚检索子图，不影响其他部分
- 生成不满意时，可以从检索完成的 checkpoint 重新生成
- 调试时，可以单独查看每个子图的状态变化历史

## 小结

| 要点 | 说明 |
|------|------|
| NS_SEP (`\|`) | 分隔命名空间的层级 |
| NS_END (`:`) | 分隔节点名和 task_id |
| 格式 | `node_name:task_id\|child_name:child_task_id` |
| 根图 | 命名空间为空字符串 `""` |
| 自动管理 | 命名空间由框架自动组装，无需手动设置 |
| 命名限制 | 节点名不能包含 `\|` 和 `:` |
| 三大能力 | 独立 checkpoint、状态隔离、子图流式识别 |
| Config 键 | CHECKPOINT_NS（当前路径）、CHECKPOINT_MAP（路径→ID映射） |
