# 节点函数约定 - 第一性原理

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而不是通过类比或经验来理解。

在物理学中，第一性原理意味着从基本物理定律推导，而不是依赖经验公式。在软件设计中，第一性原理意味着从最基本的需求和约束出发，推导出设计决策。

---

## 节点函数约定的第一性原理

### 1. 最基础的定义

**节点函数 = 状态转换函数**

这是节点函数最基础、不可再分的定义。

```python
# 最基础的形式
def node(state: State) -> State:
    """
    输入：当前状态
    输出：新状态
    """
    return new_state
```

[来源: reference/source_节点函数约定_01.md]

仅此而已！没有更基础的了。

**为什么是状态转换函数？**

因为状态化工作流的本质就是：
- **状态（State）**：工作流在某个时刻的数据快照
- **转换（Transition）**：从一个状态到另一个状态的变化
- **节点（Node）**：执行转换的单元

```
状态 A  →  [节点函数]  →  状态 B
```

这是最基本的真理，所有复杂的设计都从这里推导而来。

---

### 2. 为什么需要节点函数约定？

#### 核心问题：如何让框架自动管理状态流转？

如果没有约定，每个节点函数可以：
- 用任意方式接收输入
- 用任意方式返回输出
- 用任意方式处理错误
- 用任意方式执行异步操作

这会导致：
- ❌ 框架无法自动推断类型
- ❌ 框架无法自动管理状态
- ❌ 框架无法自动处理错误
- ❌ 框架无法自动执行异步

**解决方案：标准化接口**

通过约定函数签名和返回值，框架可以：
- ✅ 自动推断输入输出类型
- ✅ 自动合并状态更新
- ✅ 自动重试失败的节点
- ✅ 自动包装同步/异步函数

### 3. 节点函数约定的三层价值

#### 价值1：类型安全与自动推断

**第一性原理推导：**

```
前提1：Python 是动态类型语言，运行时才知道类型
   ↓
前提2：但我们希望在编写代码时就知道类型错误
   ↓
推导：需要类型注解（Type Hints）
   ↓
推导：框架需要能够读取类型注解
   ↓
结论：节点函数的第一个参数类型 = 输入 Schema
```

**LangGraph 的实现：**

```python
from typing import TypedDict

class State(TypedDict):
    x: int
    y: str

def my_node(state: State) -> dict:
    # 框架自动推断：
    # - 输入类型：State
    # - 输出类型：dict（部分更新）
    return {"x": state["x"] + 1}
```

#### 价值2：灵活的状态更新机制

**第一性原理推导：**

```
前提1：状态可能很大（包含很多字段）
   ↓
前提2：节点通常只修改部分字段
   ↓
推导：如果每次都返回完整状态，代码会很冗余
   ↓
推导：应该只返回需要更新的字段
   ↓
结论：节点函数返回部分状态更新（dict）
```

**LangGraph 的实现：**

```python
# 只返回需要更新的字段
def my_node(state: State) -> dict:
    return {"x": state["x"] + 1}  # 只更新 x，不影响其他字段
```

#### 价值3：容错性与可靠性

**第一性原理推导：**

```
前提1：分布式系统中，节点可能失败（网络错误、API 限流等）
   ↓
前提2：失败的节点应该能够重试
   ↓
推导：需要重试机制
   ↓
推导：重试需要配置（重试次数、间隔、条件）
   ↓
结论：节点函数需要支持 RetryPolicy 配置
```

**LangGraph 的实现：**

```python
from langgraph.types import RetryPolicy

builder.add_node(
    "api_call",
    unreliable_api_call,
    retry_policy=RetryPolicy(
        max_attempts=3,
        initial_interval=1.0,
        backoff_factor=2.0,
        max_interval=10.0,
        jitter=True,
        retry_on=lambda e: "rate limit" in str(e).lower()
    )
)
```

```python
def run_with_retry(
    task: PregelExecutableTask,
    retry_policy: Sequence[RetryPolicy] | None,
    configurable: dict[str, Any] | None = None,
) -> None:
    """Run a task with retries."""
    retry_policy = task.retry_policy or retry_policy
    attempts = 0
    config = task.config
    if configurable is not None:
        config = patch_configurable(config, configurable)
    while True:
        try:
            # clear any writes from previous attempts
            task.writes.clear()
            # run the task
            return task.proc.invoke(task.input, config)
        except Exception as exc:
            # ... 重试逻辑
            # 指数退避算法
            interval = min(
                matching_policy.max_interval,
                interval * (matching_policy.backoff_factor ** (attempts - 1)),
            )
            # 可选的抖动（jitter）
            sleep_time = (
                interval + random.uniform(0, 1) if matching_policy.jitter else interval
            )
            time.sleep(sleep_time)
```

### 4. 从第一性原理推导设计决策

#### 决策1：为什么支持多种返回值类型？

**推理链：**

```
1. 节点函数的基本职责：状态转换
   ↓
2. 但实际场景更复杂：
   - 有时需要更新状态
   - 有时需要控制流程（跳转到其他节点）
   - 有时不需要更新状态（只是检查）
   - 有时需要同时做多件事
   ↓
3. 如果只支持返回 dict，无法表达流程控制
   ↓
4. 需要引入 Command 对象表达流程控制
   ↓
5. 需要支持 None 表示不更新状态
   ↓
6. 需要支持 list 表示多个操作
   ↓
结论：支持 dict | Command | None | list[Command | dict]
```

**LangGraph 的实现：**

```python
# 返回 dict：部分状态更新
def update_node(state: State) -> dict:
    return {"x": state["x"] + 1}

# 返回 Command：控制流程
def router_node(state: State) -> Command[Literal["next", "end"]]:
    if state["x"] > 10:
        return Command(goto="end")
    return Command(goto="next", update={"x": state["x"] + 1})

# 返回 None：不更新状态
def check_node(state: State) -> None:
    print(f"Current x: {state['x']}")
    return None

# 返回 list：多个操作
def multi_node(state: State) -> list:
    return [
        {"x": state["x"] + 1},
        Command(goto="next")
    ]
```

[来源: reference/source_节点函数约定_01.md]

#### 决策2：为什么自动检测异步函数？

**推理链：**

```
1. 现代应用大量使用异步 API（HTTP 请求、数据库查询）
   ↓
2. 节点函数需要调用这些异步 API
   ↓
3. 如果强制所有节点都是异步，同步节点会很麻烦
   ↓
4. 如果强制所有节点都是同步，异步节点无法利用并发
   ↓
5. 需要同时支持同步和异步节点
   ↓
6. 框架需要自动检测节点是同步还是异步
   ↓
结论：自动检测 async 函数，同步函数包装成异步执行
```

**LangGraph 的实现：**

```python
# _call.py:175-192
if is_async_callable(func):
    # 异步函数：直接使用
    run = RunnableCallable(
        None, func, name=func.__name__, trace=False, recurse=False
    )
else:
    # 同步函数：包装成异步执行
    afunc = functools.update_wrapper(
        functools.partial(run_in_executor, None, func), func
    )
    run = RunnableCallable(
        func,
        afunc,
        name=func.__name__,
        trace=False,
        recurse=False,
    )
```

[来源: reference/source_节点函数约定_01.md]

**价值体现：**
- ✅ 开发者可以自由选择同步或异步
- ✅ 框架自动处理执行方式
- ✅ 同步函数在线程池中执行，不阻塞事件循环

#### 决策3：为什么支持 Config 和 Runtime 参数？

**推理链：**

```
1. 节点函数需要访问配置信息（API keys、超时设置等）
   ↓
2. 节点函数需要访问运行时上下文（用户 ID、请求 ID 等）
   ↓
3. 如果通过状态传递，会污染状态结构
   ↓
4. 需要独立的配置和上下文传递机制
   ↓
5. Config 用于静态配置，Runtime 用于动态上下文
   ↓
结论：支持可选的 Config 和 Runtime 参数
```

**LangGraph 的实现：**

```python
from langgraph.types import RunnableConfig, Runtime

# 基础签名：只有 state
def node1(state: State) -> dict:
    return {"x": state["x"] + 1}

# 带 Config：访问配置
def node2(state: State, config: RunnableConfig) -> dict:
    api_key = config.get("configurable", {}).get("api_key")
    return {"x": state["x"] + 1}

# 带 Runtime：访问运行时上下文
def node3(state: State, runtime: Runtime[Context]) -> dict:
    user_id = runtime.context.get("user_id")
    return {"x": state["x"] + 1}
```

[来源: reference/source_节点函数约定_01.md]

**价值体现：**
- ✅ 配置与状态分离
- ✅ 运行时上下文传递
- ✅ 不污染状态结构

### 5. 与状态化工作流的关系

#### 状态化工作流的本质

**状态化工作流 = 状态 + 转换规则 + 执行引擎**

```
┌─────────────────────────────────────────┐
│         状态化工作流架构                  │
├─────────────────────────────────────────┤
│                                         │
│  ┌─────────┐                            │
│  │  状态   │  ← 数据的容器               │
│  └────┬────┘                            │
│       │                                 │
│       ↓                                 │
│  ┌─────────┐                            │
│  │ 节点函数 │  ← 状态转换函数（本文主题） │
│  └────┬────┘                            │
│       │                                 │
│       ↓                                 │
│  ┌─────────┐                            │
│  │ 执行引擎 │  ← 自动管理状态流转         │
│  └─────────┘                            │
│                                         │
└─────────────────────────────────────────┘
```

**节点函数约定是连接状态和执行引擎的桥梁：**

1. **状态 → 节点函数**：通过类型注解传递状态
2. **节点函数 → 执行引擎**：通过返回值告诉引擎如何更新状态
3. **执行引擎 → 节点函数**：通过 Config 和 Runtime 传递配置和上下文

---

#### 为什么状态化工作流需要节点函数约定？

**对比：无约定 vs 有约定**

**无约定的世界：**

```python
# 每个节点函数都不一样
def node1(x, y):
    return x + y

def node2(state):
    state["x"] += 1
    return state

def node3(data, config, context):
    return {"result": data["x"] * 2}

# 框架无法自动管理
graph.add_node("node1", node1)  # 如何传参？
graph.add_node("node2", node2)  # 如何合并状态？
graph.add_node("node3", node3)  # 如何传递 config？
```

**有约定的世界：**

```python
# 所有节点函数遵循统一约定
def node1(state: State) -> dict:
    return {"x": state["x"] + 1}

def node2(state: State, config: RunnableConfig) -> dict:
    return {"y": state["y"] * 2}

def node3(state: State) -> Command[Literal["next", "end"]]:
    if state["x"] > 10:
        return Command(goto="end")
    return Command(goto="next")

# 框架自动管理
graph.add_node("node1", node1)  # 自动推断类型
graph.add_node("node2", node2)  # 自动传递 config
graph.add_node("node3", node3)  # 自动处理 Command
```

### 6. 设计哲学总结

#### 哲学1：约定优于配置（Convention over Configuration）

**核心思想：** 通过合理的约定减少配置，让开发者专注于业务逻辑。

**体现：**
- 自动推断输入输出类型
- 自动检测同步/异步函数
- 自动合并部分状态更新

#### 哲学2：声明式编程（Declarative Programming）

**核心思想：** 描述"做什么"而不是"怎么做"。

**体现：**
- 通过 Command 对象声明流程控制
- 通过 RetryPolicy 声明重试策略
- 通过类型注解声明状态结构

#### 哲学3：渐进式复杂度（Progressive Complexity）

**核心思想：** 简单场景简单用，复杂场景有能力。

**体现：**
- 最简单：`def node(state) -> dict`
- 需要配置：`def node(state, config) -> dict`
- 需要流程控制：`def node(state) -> Command`
- 需要重试：`add_node(..., retry_policy=...)`

#### 哲学4：容错优先（Fault Tolerance First）

**核心思想：** 假设失败会发生，提前设计容错机制。

**体现：**
- 内置重试机制
- 指数退避算法
- 自定义重试条件
- 错误传播和处理

**节点函数约定是从"状态转换函数"这一第一性原理出发，通过标准化接口实现类型安全、灵活更新和容错性，让框架能够自动管理状态化工作流的执行。**

## 深入理解：为什么这些约定是必要的？

### 问题1：为什么不能让节点函数直接修改状态？

**错误做法：**

```python
def bad_node(state: State) -> None:
    state["x"] += 1  # 直接修改状态
    return None
```

**为什么不好？**

1. **不可追踪**：框架无法知道状态被修改了
2. **不可回滚**：无法实现 Checkpoint 和断点续传
3. **并发问题**：多个节点同时修改状态会冲突
4. **调试困难**：无法追踪状态变化历史

**正确做法：**

```python
def good_node(state: State) -> dict:
    return {"x": state["x"] + 1}  # 返回新值，框架负责合并
```

**好处：**
- ✅ 框架可以追踪每次状态变化
- ✅ 可以实现时间旅行调试
- ✅ 可以实现 Checkpoint 和恢复
- ✅ 可以实现并发控制

---

### 问题2：为什么需要 Command 对象而不是直接返回字符串？

**对比：**

```python
# 方案1：返回字符串（简单但不灵活）
def router1(state: State) -> str:
    if state["x"] > 10:
        return "end"
    return "next"

# 方案2：返回 Command（复杂但强大）
def router2(state: State) -> Command[Literal["next", "end"]]:
    if state["x"] > 10:
        return Command(goto="end")
    return Command(goto="next", update={"x": state["x"] + 1})
```

**Command 的优势：**

1. **同时更新状态和控制流程**
   ```python
   return Command(goto="next", update={"x": state["x"] + 1})
   ```

2. **类型安全**
   ```python
   Command[Literal["next", "end"]]  # IDE 可以检查目标节点是否存在
   ```

3. **支持父图通信**
   ```python
   return Command(graph=Command.PARENT, update={"result": result})
   ```

4. **更好的错误提示**
   ```python
   # Command 指向无效节点时，错误信息更清晰
   # InvalidUpdateError: Node 'invalid_node' does not exist
   ```

### 问题3：为什么异步节点不推荐使用异步生成器？

**问题代码：**

```python
# 不推荐：异步生成器
async def bad_async_node(state: State):
    async for item in some_async_generator():
        yield {"item": item}
```

**问题：**
- 函数体可能不执行
- yield 流式输出失效
- 错误处理机制失效

[来源: reference/search_节点函数约定_01.md]

**推荐做法：**

```python
# 推荐：普通异步函数
async def good_async_node(state: State) -> dict:
    items = []
    async for item in some_async_generator():
        items.append(item)
    return {"items": items}
```

**原因：**
- ✅ 函数执行可预测
- ✅ 错误处理正常工作
- ✅ 与框架的执行模型一致

## 总结：节点函数约定的本质

节点函数约定不是任意的规则，而是从状态化工作流的第一性原理推导出来的必然结果：

1. **状态转换函数**是最基础的定义
2. **标准化接口**让框架能够自动管理
3. **类型安全**通过自动推断实现
4. **灵活更新**通过部分状态更新实现
5. **容错性**通过重试机制实现
6. **流程控制**通过 Command 对象实现
7. **异步支持**通过自动检测实现

这些约定共同构成了一个强大、灵活、可靠的状态化工作流框架。
