# 03_核心概念_5 - 结构化人机交互

> HumanInterrupt/HumanResponse 是 LangGraph 预构建的标准化协议，定义了人机交互的请求和响应格式

---

## 概念定义

**结构化人机交互是 LangGraph 在基础 `interrupt()` 之上提供的一套标准化协议，通过 `HumanInterrupt` 和 `HumanResponse` 两个 TypedDict 定义了"向人类请求什么"和"人类如何回应"的统一格式。**

为什么需要这个协议？

用基础的 `interrupt("请确认")` 虽然能工作，但存在几个问题：
1. **请求格式不统一**：每个开发者传递的 value 结构不同，前端无法通用处理
2. **响应类型不明确**：用户能做什么操作（批准？编辑？跳过？）没有标准定义
3. **UI 无法自动生成**：前端不知道该渲染什么样的交互界面

结构化协议解决了这些问题，让人机交互有了"合同"——双方都知道数据长什么样。

## HumanInterruptConfig — 定义允许的操作

### 源码定义

```python
# 来源：langgraph/prebuilt/interrupt.py
class HumanInterruptConfig(TypedDict):
    allow_ignore: bool      # 用户可以跳过/忽略这个中断
    allow_respond: bool     # 用户可以提供文本反馈
    allow_edit: bool        # 用户可以编辑请求的内容
    allow_accept: bool      # 用户可以直接批准
```

### 四种操作详解

```
┌─────────────────────────────────────────────────┐
│           HumanInterruptConfig                   │
├─────────────┬───────────────────────────────────┤
│ allow_accept│ "批准" — 同意当前操作，原样执行    │
│             │ 例：批准工具调用、确认订单          │
├─────────────┼───────────────────────────────────┤
│ allow_edit  │ "编辑" — 修改后再执行              │
│             │ 例：修改工具参数、调整金额          │
├─────────────┼───────────────────────────────────┤
│ allow_respond│ "反馈" — 提供文本指导             │
│             │ 例：告诉 Agent 换个方向            │
├─────────────┼───────────────────────────────────┤
│ allow_ignore│ "跳过" — 忽略这个中断，继续执行    │
│             │ 例：跳过非关键步骤                 │
└─────────────┴───────────────────────────────────┘
```

### 常见配置组合

```python
# 场景1：严格审批（只能批准或拒绝）
strict_config: HumanInterruptConfig = {
    "allow_accept": True,
    "allow_ignore": True,    # ignore 相当于拒绝
    "allow_respond": False,
    "allow_edit": False,
}

# 场景2：灵活审批（可以批准、编辑、反馈）
flexible_config: HumanInterruptConfig = {
    "allow_accept": True,
    "allow_edit": True,
    "allow_respond": True,
    "allow_ignore": True,
}

# 场景3：必须审批（不能跳过）
mandatory_config: HumanInterruptConfig = {
    "allow_accept": True,
    "allow_edit": True,
    "allow_respond": False,
    "allow_ignore": False,   # 不能跳过
}
```

## HumanInterrupt — 请求结构

### 源码定义

```python
# 来源：langgraph/prebuilt/interrupt.py

class ActionRequest(TypedDict):
    action: str     # 请求执行的动作名称
    args: dict      # 动作的参数

class HumanInterrupt(TypedDict):
    action_request: ActionRequest       # 请求的动作（做什么 + 参数）
    config: HumanInterruptConfig        # 允许的操作配置
    description: str | None             # 给人类的描述信息
```

### 结构可视化

```
HumanInterrupt
├── action_request: ActionRequest
│   ├── action: str          # "search_web", "send_email", "delete_file"
│   └── args: dict           # {"query": "LangGraph教程", "limit": 10}
├── config: HumanInterruptConfig
│   ├── allow_accept: bool
│   ├── allow_edit: bool
│   ├── allow_respond: bool
│   └── allow_ignore: bool
└── description: str | None  # "Agent 想要搜索网页，请确认"
```

### 使用示例

```python
from langgraph.types import interrupt

# 构造一个结构化的中断请求
human_interrupt: HumanInterrupt = {
    "action_request": {
        "action": "send_email",
        "args": {
            "to": "client@example.com",
            "subject": "项目进度报告",
            "body": "本周完成了以下工作...",
        },
    },
    "config": {
        "allow_accept": True,   # 可以直接发送
        "allow_edit": True,     # 可以修改邮件内容
        "allow_respond": True,  # 可以给出反馈
        "allow_ignore": True,   # 可以取消发送
    },
    "description": "Agent 准备发送一封邮件给客户，请审核邮件内容",
}

# 在节点中使用
def email_node(state):
    response = interrupt(human_interrupt)
    # response 是一个 HumanResponse
    return handle_response(response)
```

## HumanResponse — 响应结构

### 源码定义

```python
# 来源：langgraph/prebuilt/interrupt.py
class HumanResponse(TypedDict):
    type: Literal["accept", "ignore", "response", "edit"]
    args: None | str | ActionRequest
```

### 四种响应类型

```python
# 类型1：accept — 批准，原样执行
accept_response: HumanResponse = {
    "type": "accept",
    "args": None,  # 不需要额外参数
}

# 类型2：ignore — 跳过，不执行
ignore_response: HumanResponse = {
    "type": "ignore",
    "args": None,
}

# 类型3：response — 文本反馈
feedback_response: HumanResponse = {
    "type": "response",
    "args": "邮件语气太正式了，改成更友好的风格",  # 文本反馈
}

# 类型4：edit — 编辑后执行
edit_response: HumanResponse = {
    "type": "edit",
    "args": {  # 修改后的 ActionRequest
        "action": "send_email",
        "args": {
            "to": "client@example.com",
            "subject": "项目进度更新",       # 修改了标题
            "body": "Hi! 本周进展顺利...",   # 修改了内容
        },
    },
}
```

### 响应类型与 Config 的对应关系

| 响应类型 | 对应 Config 字段 | args 类型 | 含义 |
|---------|-----------------|----------|------|
| `"accept"` | `allow_accept` | `None` | 批准原始请求 |
| `"ignore"` | `allow_ignore` | `None` | 跳过/拒绝 |
| `"response"` | `allow_respond` | `str` | 文本反馈 |
| `"edit"` | `allow_edit` | `ActionRequest` | 修改后的请求 |

## Agent Inbox 模式

### 什么是 Agent Inbox？

Agent Inbox 是基于 HumanInterrupt/HumanResponse 协议构建的标准化人机交互 UI 模式。它的核心思想是：

```
Agent 产生的所有需要人工处理的请求
    ↓
统一进入一个"收件箱"（Inbox）
    ↓
人类在收件箱中逐一处理
    ↓
处理结果通过 HumanResponse 返回给 Agent
```

### 工作流程

```
┌──────────┐     HumanInterrupt      ┌──────────────┐
│  Agent   │ ──────────────────────→  │  Agent Inbox │
│  (图节点) │                          │  (前端 UI)   │
│          │ ←──────────────────────  │              │
└──────────┘     HumanResponse       └──────────────┘
                                           │
                                     用户看到：
                                     ┌─────────────────┐
                                     │ 📋 待处理请求     │
                                     │                  │
                                     │ Agent 想要发邮件  │
                                     │ 收件人: xxx       │
                                     │ 主题: xxx         │
                                     │                  │
                                     │ [批准] [编辑]     │
                                     │ [反馈] [跳过]     │
                                     └─────────────────┘
```

### 为什么需要标准化？

没有标准化协议时，每个 Agent 的中断格式都不同：

```python
# Agent A 的中断
interrupt("确认吗？")

# Agent B 的中断
interrupt({"type": "approval", "data": {...}})

# Agent C 的中断
interrupt({"question": "...", "options": [...]})
```

前端需要为每个 Agent 写不同的处理逻辑。有了 HumanInterrupt/HumanResponse 协议，前端只需要实现一套通用的 Inbox UI。

## 完整代码示例：工具调用审批场景

下面的示例展示了一个完整的结构化人机交互流程：Agent 想要调用工具，人类审批后决定是否执行。

```python
"""
结构化人机交互完整示例
场景：Agent 工具调用审批
"""

from typing import TypedDict, Optional, Literal, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import interrupt, Command
import operator


# ============================================================
# 1. 定义类型（模拟 HumanInterrupt/HumanResponse 协议）
# ============================================================

# 注意：实际使用时从 langgraph.prebuilt.interrupt 导入
# 这里手动定义以便理解结构

class ActionRequest(TypedDict):
    action: str
    args: dict

class HumanInterruptConfig(TypedDict):
    allow_ignore: bool
    allow_respond: bool
    allow_edit: bool
    allow_accept: bool

class HumanInterrupt(TypedDict):
    action_request: ActionRequest
    config: HumanInterruptConfig
    description: Optional[str]

class HumanResponse(TypedDict):
    type: Literal["accept", "ignore", "response", "edit"]
    args: None | str | ActionRequest


# ============================================================
# 2. 定义状态
# ============================================================

class AgentState(TypedDict):
    query: str
    tool_calls: Annotated[list[dict], operator.add]
    results: Annotated[list[str], operator.add]
    status: str


# ============================================================
# 3. 模拟工具
# ============================================================

def search_web(query: str, limit: int = 5) -> str:
    """模拟网页搜索"""
    return f"搜索 '{query}' 的前 {limit} 条结果: [结果1, 结果2, ...]"

def send_email(to: str, subject: str, body: str) -> str:
    """模拟发送邮件"""
    return f"邮件已发送给 {to}，主题: {subject}"

TOOLS = {
    "search_web": search_web,
    "send_email": send_email,
}


# ============================================================
# 4. 定义节点
# ============================================================

def plan_node(state: AgentState):
    """Agent 规划要调用的工具"""
    query = state["query"]
    print(f"[规划] 分析用户请求: {query}")

    # 模拟 Agent 决定调用哪些工具
    planned_calls = [
        {"action": "search_web", "args": {"query": query, "limit": 3}},
        {"action": "send_email", "args": {
            "to": "team@example.com",
            "subject": f"调研结果: {query}",
            "body": "请查看附件中的调研结果。",
        }},
    ]

    return {"tool_calls": planned_calls, "status": "planned"}


def approve_node(state: AgentState):
    """人工审批每个工具调用"""
    approved_calls = []

    for tool_call in state["tool_calls"]:
        # 构造结构化中断请求
        human_interrupt: HumanInterrupt = {
            "action_request": {
                "action": tool_call["action"],
                "args": tool_call["args"],
            },
            "config": {
                "allow_accept": True,
                "allow_edit": True,
                "allow_respond": True,
                "allow_ignore": True,
            },
            "description": f"Agent 想要调用工具 '{tool_call['action']}'，请审核参数",
        }

        # 中断，等待人类响应
        response: HumanResponse = interrupt(human_interrupt)

        # 处理响应
        if response["type"] == "accept":
            print(f"  [审批] ✓ 批准: {tool_call['action']}")
            approved_calls.append(tool_call)

        elif response["type"] == "edit":
            print(f"  [审批] ✎ 编辑后批准: {tool_call['action']}")
            edited = response["args"]
            approved_calls.append({
                "action": edited["action"],
                "args": edited["args"],
            })

        elif response["type"] == "response":
            print(f"  [审批] 💬 收到反馈: {response['args']}")
            # 可以根据反馈调整，这里简单跳过
            pass

        elif response["type"] == "ignore":
            print(f"  [审批] ✗ 跳过: {tool_call['action']}")
            pass

    # 用审批后的调用列表替换原始列表
    return {"tool_calls": approved_calls, "status": "approved"}


def execute_node(state: AgentState):
    """执行审批通过的工具调用"""
    results = []

    for tool_call in state["tool_calls"]:
        action = tool_call["action"]
        args = tool_call["args"]

        if action in TOOLS:
            result = TOOLS[action](**args)
            print(f"  [执行] {action} → {result}")
            results.append(result)
        else:
            results.append(f"未知工具: {action}")

    return {"results": results, "status": "completed"}


# ============================================================
# 5. 构建图
# ============================================================

builder = StateGraph(AgentState)
builder.add_node("plan", plan_node)
builder.add_node("approve", approve_node)
builder.add_node("execute", execute_node)

builder.add_edge(START, "plan")
builder.add_edge("plan", "approve")
builder.add_edge("approve", "execute")
builder.add_edge("execute", END)

checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)


# ============================================================
# 6. 运行示例
# ============================================================

config = {"configurable": {"thread_id": "tool-approval-001"}}

# --- 第一次调用：Agent 规划并请求审批 ---
print("=" * 60)
print("第一次调用：Agent 规划工具调用")
print("=" * 60)

result = graph.invoke(
    {
        "query": "LangGraph 人机循环最佳实践",
        "tool_calls": [],
        "results": [],
        "status": "init",
    },
    config,
)

# 检查中断状态
snapshot = graph.get_state(config)
print(f"\n中断节点: {snapshot.next}")
print(f"中断数量: {len(snapshot.interrupts)}")

if snapshot.interrupts:
    intr = snapshot.interrupts[0]
    print(f"第一个中断: {intr.value}")


# --- 第二次调用：审批第一个工具调用（批准搜索） ---
print("\n" + "=" * 60)
print("第二次调用：批准搜索工具")
print("=" * 60)

accept_search: HumanResponse = {
    "type": "accept",
    "args": None,
}

result = graph.invoke(Command(resume=accept_search), config)

# 检查是否还有中断（第二个工具调用）
snapshot = graph.get_state(config)
print(f"\n中断节点: {snapshot.next}")

if snapshot.interrupts:
    intr = snapshot.interrupts[0]
    print(f"下一个中断: {intr.value}")


# --- 第三次调用：审批第二个工具调用（编辑邮件后批准） ---
print("\n" + "=" * 60)
print("第三次调用：编辑邮件后批准")
print("=" * 60)

edit_email: HumanResponse = {
    "type": "edit",
    "args": {
        "action": "send_email",
        "args": {
            "to": "team@example.com",
            "subject": "调研结果: LangGraph HITL",  # 修改了标题
            "body": "团队好，附件是 LangGraph 人机循环的调研报告，请查阅。",  # 修改了内容
        },
    },
}

final_result = graph.invoke(Command(resume=edit_email), config)

print(f"\n最终状态: {final_result['status']}")
print(f"执行结果: {final_result['results']}")
```

### 预期输出

```
============================================================
第一次调用：Agent 规划工具调用
============================================================
[规划] 分析用户请求: LangGraph 人机循环最佳实践

中断节点: ('approve',)
中断数量: 1
第一个中断: {'action_request': {'action': 'search_web', ...}, 'config': {...}, ...}

============================================================
第二次调用：批准搜索工具
============================================================
  [审批] ✓ 批准: search_web

中断节点: ('approve',)
下一个中断: {'action_request': {'action': 'send_email', ...}, ...}

============================================================
第三次调用：编辑邮件后批准
============================================================
  [审批] ✎ 编辑后批准: send_email
  [执行] search_web → 搜索 'LangGraph 人机循环最佳实践' 的前 3 条结果: [...]
  [执行] send_email → 邮件已发送给 team@example.com，主题: 调研结果: LangGraph HITL

最终状态: completed
执行结果: ['搜索 ...', '邮件已发送 ...']
```

### 关键观察点

1. **多个 interrupt 按顺序触发**：`approve_node` 中有两个 `interrupt()` 调用，每次恢复只处理一个
2. **不同的响应类型**：第一个用 `accept`（直接批准），第二个用 `edit`（修改后批准）
3. **节点重新执行**：每次恢复时 `approve_node` 从头执行，已处理的 interrupt 直接返回缓存值

## 与简单 interrupt(str) 的对比

### 简单模式

```python
def simple_approval(state):
    # 简单字符串中断
    answer = interrupt("是否批准？(yes/no)")

    if answer == "yes":
        return {"status": "approved"}
    else:
        return {"status": "rejected"}

# 恢复
graph.invoke(Command(resume="yes"), config)
```

**优点**：简单直接，适合快速原型
**缺点**：
- 前端不知道该渲染什么 UI
- 没有标准的操作类型（批准/编辑/反馈/跳过）
- 每个节点的中断格式可能不同

### 结构化模式

```python
def structured_approval(state):
    response = interrupt({
        "action_request": {"action": "transfer", "args": {"amount": 500}},
        "config": {
            "allow_accept": True,
            "allow_edit": True,
            "allow_respond": False,
            "allow_ignore": True,
        },
        "description": "Agent 想要转账 $500",
    })

    if response["type"] == "accept":
        return execute_transfer(state)
    elif response["type"] == "edit":
        return execute_transfer_with_edits(response["args"])
    elif response["type"] == "ignore":
        return {"status": "cancelled"}

# 恢复
graph.invoke(
    Command(resume={"type": "accept", "args": None}),
    config,
)
```

**优点**：
- 前端可以根据 `config` 自动渲染按钮
- 响应类型明确，处理逻辑清晰
- 多个 Agent 共用同一套 UI

### 选择建议

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| 快速原型/实验 | `interrupt(str)` | 简单快速 |
| 简单确认（yes/no） | `interrupt(str)` | 不需要复杂交互 |
| 工具调用审批 | `HumanInterrupt` | 需要展示工具参数，支持编辑 |
| 多 Agent 系统 | `HumanInterrupt` | 统一的交互协议 |
| 有前端 UI 的产品 | `HumanInterrupt` | 前端可以通用处理 |
| Agent Inbox 模式 | `HumanInterrupt` | 这就是它设计的目的 |

## 自定义结构化交互

如果 HumanInterrupt 的标准格式不满足需求，可以基于同样的思路自定义：

```python
from typing import TypedDict, Literal, Optional


class CustomInterruptRequest(TypedDict):
    """自定义中断请求"""
    task_type: str                    # 任务类型
    payload: dict                     # 任务数据
    urgency: Literal["low", "medium", "high"]  # 紧急程度
    allowed_actions: list[str]        # 允许的操作
    timeout_seconds: Optional[int]    # 超时时间


class CustomInterruptResponse(TypedDict):
    """自定义中断响应"""
    action: str                       # 选择的操作
    data: Optional[dict]              # 附加数据
    comment: Optional[str]            # 备注


def custom_approval_node(state):
    request: CustomInterruptRequest = {
        "task_type": "data_export",
        "payload": {
            "table": "users",
            "format": "csv",
            "row_count": 50000,
        },
        "urgency": "high",
        "allowed_actions": ["approve", "reject", "modify_scope"],
        "timeout_seconds": 3600,
    }

    response: CustomInterruptResponse = interrupt(request)

    if response["action"] == "approve":
        return export_data(state)
    elif response["action"] == "modify_scope":
        return export_data_with_scope(response["data"])
    else:
        return {"status": "rejected", "reason": response.get("comment")}
```

**核心原则**：`interrupt()` 的 value 只要是 JSON 可序列化的就行，你可以定义任何结构。HumanInterrupt/HumanResponse 只是官方提供的一个标准方案。

## 总结

**结构化人机交互为 HITL 提供了"通信协议"**——让 Agent 和人类之间的交互有了统一的语言。

核心要点：
1. **HumanInterruptConfig** 定义四种操作权限：accept、edit、respond、ignore
2. **HumanInterrupt** 封装请求：action_request（做什么）+ config（能做什么）+ description（为什么）
3. **HumanResponse** 封装响应：type（选了什么）+ args（附加数据）
4. **Agent Inbox** 是基于此协议的标准化 UI 模式
5. 简单场景用 `interrupt(str)`，复杂场景用结构化协议
6. 这些类型已迁移到 `langchain.agents.interrupt`，但 LangGraph 中仍可用
