# 所有权与借用 -- 实战代码：场景 3 异步共享状态管理

> **一句话定位：** 用 `Arc<Mutex<T>>` 在多个异步任务间安全共享可变状态 -- Rust 并发编程的核心模式，也是 ZeroClaw 限速控制的底层原理。

---

## 1. 场景说明

**目标：** 理解 `Arc<Mutex<T>>` 在多线程/异步场景下如何安全共享可变状态。

**背景：** ZeroClaw 的 `DriveTool` 使用 `Arc<Mutex<Option<Instant>>>` 做电机限速控制 -- 多个异步命令可能同时到达，但物理电机需要至少 1 秒冷却间隔。

```rust
// ZeroClaw 源码片段 [来源: sourcecode/zeroclaw/crates/robot-kit/src/drive.rs]
pub struct DriveTool {
    backend: Arc<dyn DriveBackend>,
    last_command: Arc<Mutex<Option<std::time::Instant>>>,  // <-- 关注这行
}
```

---

## 2. 前置知识

### 为什么异步需要 Arc？

`tokio::spawn` 要求数据满足 `'static + Send`：
- **`'static`**：数据不能借用栈上局部变量（任务可能比创建它的函数活得更久）
- **`Send`**：数据可以安全地跨线程转移

`Arc`（Atomic Reference Counting）满足这两个要求 -- 堆分配、引用计数、线程安全。

**TypeScript 类比：** JS 单线程，闭包天然捕获外部变量。Rust 要求你用类型系统**证明**跨线程共享是安全的。

### std::sync::Mutex vs tokio::sync::Mutex

| 特性 | `std::sync::Mutex` | `tokio::sync::Mutex` |
|------|--------------------|--------------------|
| 锁等待 | 阻塞线程 | 异步等待（.await） |
| 适用 | 锁内操作**快速、无 await** | 锁内需要 **await** |
| 性能 | 更快 | 稍慢（异步调度开销） |

**准则：** 默认用 `std::sync::Mutex`；仅当持有锁同时 `.await` 时，用 `tokio::sync::Mutex`。

## 3. 示例 1：多线程共享计数器

> **Cargo.toml：** 纯标准库，无需额外依赖。

```rust
// Cargo.toml 无需额外依赖（纯标准库）
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    // Arc 提供共享所有权，Mutex 提供互斥访问
    let counter = Arc::new(Mutex::new(0_u32));
    let mut handles = vec![];

    for thread_id in 0..10 {
        // Arc::clone() 不是深拷贝！只增加引用计数（原子操作，极快）
        let counter_clone = Arc::clone(&counter);

        let handle = thread::spawn(move || {
            // move 闭包获取 counter_clone 的所有权
            for _ in 0..100 {
                let mut num = counter_clone.lock().unwrap();
                *num += 1;
            } // MutexGuard 在这里 drop，自动释放锁
            println!("线程 {} 完成", thread_id);
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }
    println!("最终计数: {} (预期: 1000)", counter.lock().unwrap());
}
```

**预期输出：**（线程顺序不固定，计数一定是 1000）
```
线程 0 完成
线程 3 完成
...
最终计数: 1000 (预期: 1000)
```

**所有权流转：** 每个线程拥有一个 `Arc` clone，所有 clone 指向同一个 `Mutex<u32>`。引用计数：1 -> 11（主线程+10线程） -> 逐步回到 0 -> 内存释放。

## 4. 示例 2：异步限速器（模拟 ZeroClaw DriveTool）

> **Cargo.toml：** `tokio = { version = "1", features = ["full"] }`

```rust
// Cargo.toml: tokio = { version = "1", features = ["full"] }
use std::sync::Arc;
use std::time::{Duration, Instant};
use tokio::sync::Mutex;

/// 限速器：确保操作之间有最小时间间隔
/// 直接对应 ZeroClaw DriveTool 的 last_command 字段
struct RateLimiter {
    last_call: Arc<Mutex<Option<Instant>>>,
    min_interval: Duration,
}

impl RateLimiter {
    fn new(min_interval: Duration) -> Self {
        Self {
            last_call: Arc::new(Mutex::new(None)),
            min_interval,
        }
    }

    /// 返回 Ok(()) 可执行，Err(剩余时间) 需等待
    async fn try_acquire(&self) -> Result<(), Duration> {
        let mut last = self.last_call.lock().await;
        if let Some(instant) = *last {
            let elapsed = instant.elapsed();
            if elapsed < self.min_interval {
                return Err(self.min_interval - elapsed);
            }
        }
        *last = Some(Instant::now());
        Ok(())
    } // MutexGuard drop -> 锁释放
}

async fn drive_command(id: u32, limiter: Arc<RateLimiter>) {
    match limiter.try_acquire().await {
        Ok(()) => println!("[命令 {}] 执行成功 - 电机已启动", id),
        Err(remaining) => println!(
            "[命令 {}] 被限速 - 还需等待 {:.0}ms", id, remaining.as_millis()
        ),
    }
}

#[tokio::main]
async fn main() {
    let limiter = Arc::new(RateLimiter::new(Duration::from_millis(500)));

    println!("=== 快速连续发送 3 个命令 ===");
    let mut handles = vec![];
    for i in 1..=3 {
        let limiter_clone = Arc::clone(&limiter);
        handles.push(tokio::spawn(async move {
            drive_command(i, limiter_clone).await;
        }));
    }
    for h in handles { h.await.unwrap(); }

    println!("\n=== 等待冷却后再发送 ===");
    tokio::time::sleep(Duration::from_millis(600)).await;
    drive_command(4, Arc::clone(&limiter)).await;
}
```

**预期输出：**
```
=== 快速连续发送 3 个命令 ===
[命令 1] 执行成功 - 电机已启动
[命令 2] 被限速 - 还需等待 499ms
[命令 3] 被限速 - 还需等待 499ms

=== 等待冷却后再发送 ===
[命令 4] 执行成功 - 电机已启动
```

**与 ZeroClaw 的对应：** `RateLimiter.last_call` 对应 `DriveTool.last_command`，`try_acquire()` 对应 `execute()` 中 `{}` 块内的限速检查逻辑。

## 5. 示例 3：异步配置热更新

> **Cargo.toml：** `tokio = { version = "1", features = ["full"] }`

```rust
// Cargo.toml: tokio = { version = "1", features = ["full"] }
// 灵感: ZeroClaw SecurityPolicy 的 ActionTracker
// [来源: sourcecode/zeroclaw/src/security/policy.rs]
use std::sync::Arc;
use tokio::sync::RwLock;
use tokio::time::{sleep, Duration};

#[derive(Debug, Clone)]
struct AppConfig {
    api_url: String,
    max_retries: u32,
    timeout_ms: u64,
}

async fn worker(id: u32, config: Arc<RwLock<AppConfig>>) {
    for round in 1..=3 {
        let cfg = config.read().await; // 多读者可并行持有读锁
        println!(
            "[Worker {}][轮次 {}] api={}, retries={}, timeout={}ms",
            id, round, cfg.api_url, cfg.max_retries, cfg.timeout_ms
        );
        drop(cfg); // 显式释放读锁
        sleep(Duration::from_millis(100)).await;
    }
}

async fn config_updater(config: Arc<RwLock<AppConfig>>) {
    sleep(Duration::from_millis(150)).await;
    {
        let mut cfg = config.write().await; // 写锁独占
        cfg.api_url = "https://api.example.com/v2".to_string();
        cfg.max_retries = 5;
        cfg.timeout_ms = 3000;
        println!("\n>>> 配置已更新! <<<\n");
    } // 写锁释放，后续读者看到新配置
}

#[tokio::main]
async fn main() {
    let config = Arc::new(RwLock::new(AppConfig {
        api_url: "https://api.example.com/v1".to_string(),
        max_retries: 3,
        timeout_ms: 5000,
    }));

    let mut handles = vec![];
    for id in 1..=3 {
        handles.push(tokio::spawn(worker(id, Arc::clone(&config))));
    }
    handles.push(tokio::spawn(config_updater(Arc::clone(&config))));

    for h in handles { h.await.unwrap(); }

    let final_cfg = config.read().await;
    println!("最终: api={}, retries={}", final_cfg.api_url, final_cfg.max_retries);
}
```

**预期输出：**
```
[Worker 1][轮次 1] api=https://api.example.com/v1, retries=3, timeout=5000ms
[Worker 2][轮次 1] api=https://api.example.com/v1, retries=3, timeout=5000ms
[Worker 3][轮次 1] api=https://api.example.com/v1, retries=3, timeout=5000ms

>>> 配置已更新! <<<

[Worker 1][轮次 2] api=https://api.example.com/v2, retries=5, timeout=3000ms
...
最终: api=https://api.example.com/v2, retries=5
```

**RwLock vs Mutex：** 读多写少用 `RwLock`（多读者并行）；通用场景用 `Mutex`（更简单）。

## 6. 所有权流转图

```
                Arc clone 进入每个 task
                ========================
main()
 │
 │  let data = Arc::new(Mutex::new(value));     // 引用计数 = 1
 │       │
 │       ├── Arc::clone() ──▶ tokio::spawn(task1)  // 计数 = 2
 │       ├── Arc::clone() ──▶ tokio::spawn(task2)  // 计数 = 3
 │       │
 │       │   task1 结束 → Arc drop → 计数 = 2
 │       │   task2 结束 → Arc drop → 计数 = 1
 │       │
 │       ▼   main 结束 → Arc drop → 计数 = 0 → 内存释放
```

**MutexGuard 的生命周期：**
```rust
{   let mut guard = data.lock().unwrap();   // 获取锁
    guard.push("hello".to_string());        // 独占访问
}   // guard drop → 锁自动释放（RAII）
// 其他任务现在可以获取锁了
```

## 7. 常见陷阱

### 陷阱 1：MutexGuard 跨 await 点（最常见死锁原因）

```rust
// !! 错误示范 !!
async fn bad(data: Arc<std::sync::Mutex<Vec<String>>>) {
    let mut guard = data.lock().unwrap();
    guard.push("hello".to_string());
    // 危险！持有 std::sync::Mutex 锁时 await
    // future 被挂起并移到另一个线程 → 死锁！
    tokio::time::sleep(Duration::from_secs(1)).await;
    guard.push("world".to_string());
}
```

**解决方案 A（推荐）：缩小锁作用域**

```rust
async fn good_a(data: Arc<std::sync::Mutex<Vec<String>>>) {
    {
        let mut guard = data.lock().unwrap();
        guard.push("hello".to_string());
    } // 锁在 await 之前释放

    tokio::time::sleep(Duration::from_secs(1)).await;

    {
        let mut guard = data.lock().unwrap();
        guard.push("world".to_string());
    }
}
```

这正是 ZeroClaw `DriveTool.execute()` 的模式 -- 用 `{}` 块限制 MutexGuard 作用域：

```rust
// [来源: sourcecode/zeroclaw/crates/robot-kit/src/drive.rs]
async fn execute(&self, args: Value) -> Result<ToolResult> {
    {   // <-- {} 块限制 guard 作用域
        let mut last = self.last_command.lock().await;
        if let Some(instant) = *last {
            if instant.elapsed() < Duration::from_secs(1) {
                return Ok(ToolResult { /* 限速 */ });
            }
        }
        *last = Some(std::time::Instant::now());
    }   // guard drop，锁释放
    // 安全地 await 后端操作
    self.backend.move_robot(/*...*/).await?;
}
```

**解决方案 B：** 用 `tokio::sync::Mutex`（允许跨 await，但锁持有时间变长）。

### 陷阱 2：忘记 clone Arc

```rust
// !! 编译错误 !!
let data = Arc::new(Mutex::new(0));
tokio::spawn(async move { *data.lock().unwrap() += 1; });
// println!("{}", data.lock().unwrap());  // data 已被 move！
```

**正确：先 clone 再 move**

```rust
let data = Arc::new(Mutex::new(0));
let data_clone = Arc::clone(&data);    // 先 clone
tokio::spawn(async move {
    *data_clone.lock().unwrap() += 1;  // move clone
}).await.unwrap();
println!("{}", data.lock().unwrap());  // data 仍有效
```

## 8. 与 TypeScript 对比

| 维度 | TypeScript/Node.js | Rust |
|------|-------------------|------|
| 线程模型 | 单线程事件循环 | 多线程（tokio 多线程调度） |
| 共享状态 | 直接访问（单线程天然安全） | 必须 `Arc<Mutex<T>>` 包装 |
| 数据竞争 | 不存在（单线程） | 编译器**静态禁止** |
| Worker Threads | `SharedArrayBuffer` + `Atomics` | `Arc<T>` + `Mutex<T>` |
| 锁释放 | 不需要 | 自动（RAII/Drop） |

```typescript
// TypeScript 竞态：两个 worker 同时 increment → 结果可能错误
let counter = 0;
async function increment() {
  const current = counter;       // 读
  await someAsyncOp();           // 上下文切换！
  counter = current + 1;         // 写（可能基于过期值）
}
```

```rust
// Rust：Mutex 保护"读-改-写"原子性，不可能出现竞态
async fn increment(counter: Arc<Mutex<i32>>) {
    let mut num = counter.lock().unwrap();
    *num += 1;
}
```

**核心区别：** TypeScript 把并发安全交给开发者（运行时出错才发现），Rust 交给编译器（编译不过就不让运行）。

## 速查表

```
跨任务共享数据？
  ├── 只读 → Arc<T>（T: Send + Sync）
  ├── 需修改 → Arc<Mutex<T>>（通用） 或 Arc<RwLock<T>>（读多写少）
  └── 不共享 → 直接 move 进任务

选哪个 Mutex？
  ├── 锁内无 await → std::sync::Mutex（更快）
  └── 锁内有 await → tokio::sync::Mutex

三条黄金法则：
  1. Arc::clone() 是浅拷贝（引用计数 +1）
  2. 用 {} 块缩小 MutexGuard 作用域
  3. 先 clone Arc，再 move 进 spawn
```
