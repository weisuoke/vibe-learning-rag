# Tools与函数调用 - 一句话总结

## 一句话总结

**Tool 是 LangChain 中将函数封装为标准接口的机制，通过 @tool 装饰器定义、bind_tools() 绑定、ToolCall/ToolMessage 消息流协作，让 LLM 从"只会说"变成"能做事"，是构建 Agent 系统的核心基础设施。**

---

## 关键词拆解

| 关键词 | 含义 | 重要性 |
|--------|------|--------|
| @tool 装饰器 | 将普通 Python 函数转为 LangChain Tool 的最简方式 | ⭐⭐⭐⭐⭐ |
| bind_tools() | 将 Tool 列表绑定到 LLM，让模型知道可以调用什么 | ⭐⭐⭐⭐⭐ |
| ToolCall | LLM 输出中表达"我想调用某个函数"的结构化意图 | ⭐⭐⭐⭐⭐ |
| ToolMessage | 函数执行结果回传给 LLM 的标准消息格式 | ⭐⭐⭐⭐ |
| Function Calling | 大模型原生支持的结构化输出能力，Tool 的底层依赖 | ⭐⭐⭐⭐ |
| Tool 描述 | docstring 自动转为 Tool 说明，直接影响模型调用准确率 | ⭐⭐⭐⭐ |
| args_schema | 用 Pydantic 定义参数校验，保证输入合法性 | ⭐⭐⭐ |
| Agent | Tool 的最终消费者，自主决定何时调用哪个 Tool | ⭐⭐⭐⭐⭐ |

---

## 知识点全景回顾

| 维度 | 一句话要点 |
|------|-----------|
| 30字核心 | Tool = 函数封装 + 模型绑定 + 消息流协作 |
| 第一性原理 | LLM 只能生成文本，Tool 是它与外部世界交互的桥梁 |
| 核心概念 | @tool 定义、bind_tools() 绑定、ToolCall/ToolMessage 消息循环三件套 |
| 最小可用 | 一个 @tool 函数 + bind_tools() + 手动执行 = 最小闭环 |
| 双重类比 | 前端：addEventListener 注册回调；生活：给员工发工具箱和操作手册 |
| 反直觉点 | LLM 不执行函数，只输出调用意图；描述比实现更影响效果 |
| 实战代码 | RAG 知识库检索 Tool、多 Tool 协作、错误处理模式 |
| 面试必问 | Tool 与 Function Calling 的关系、消息流转机制、调用失败处理 |
| 化骨绵掌 | 描述写不好 → 模型乱调用；忘记 tool_call_id → 消息链断裂 |
| 一句话总结 | Tool 让 LLM 从"只会说"变成"能做事" |

## 学习检查清单

- [ ] 能用 @tool 装饰器定义一个带类型注解和 docstring 的 Tool
- [ ] 理解 bind_tools() 的作用：把 Tool 信息注入 LLM 的系统提示
- [ ] 能区分 ToolCall（模型意图）和 ToolMessage（执行结果）
- [ ] 掌握完整消息流：Human → AI(ToolCall) → Tool执行 → ToolMessage → AI(最终回答)
- [ ] 知道 LLM 本身不执行函数，执行逻辑由开发者控制
- [ ] 能为 RAG 检索场景编写实用的 Tool
- [ ] 理解 Tool 描述质量对调用准确率的关键影响
- [ ] 能处理 Tool 调用失败的异常情况
