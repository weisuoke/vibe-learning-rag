# 核心概念 6：内部可变性（Interior Mutability）

## 1. 什么是内部可变性

### 一句话定义

**在只持有不可变引用 `&self` 的情况下，安全地修改内部数据。**

### 为什么需要它？

Rust 的借用规则非常严格：要修改数据，你必须持有 `&mut self`（可变引用），而且同一时刻只能存在一个可变引用。这在大多数场景下是合理的，但有些场景会让你碍手碍脚：

```rust
// 问题场景：一个缓存系统
struct Cache {
    data: HashMap<String, String>,
    hit_count: u64,  // 想在每次读取时 +1，但读取只给 &self
}

impl Cache {
    fn get(&self, key: &str) -> Option<&String> {
        self.hit_count += 1;  // 编译错误！&self 不允许修改
        self.data.get(key)
    }
}
```

你只是想记个数，但编译器不答应 —— 因为 `&self` 意味着"我承诺不修改任何东西"。

内部可变性就是 Rust 为这类场景提供的**安全逃生通道**：借用规则的检查从编译时推迟到运行时（或通过原子操作绕过），让你在 `&self` 下仍然能修改特定字段。

### TypeScript 类比

这其实很像 TypeScript 中 `const` 对象的行为：

```typescript
// TypeScript：const 只保证绑定不变，属性可以改
const user = { name: "Alice", loginCount: 0 };
user.loginCount += 1;  // 完全合法！

// Rust 默认行为：不可变就是真的全不可变
// 内部可变性让 Rust 也能实现类似的"外部不变，内部可变"
```

区别是：TypeScript 对 `const` 的"宽容"是默认行为，而 Rust 要求你**显式选择**内部可变性，并且仍然保证安全。

## 2. Cell\<T\> — 简单值的内部可变性

### 适用场景

`Cell<T>` 是最简单的内部可变性容器，适用于实现了 `Copy` trait 的类型（整数、浮点数、布尔值等）。

### 核心 API

```rust
use std::cell::Cell;

let counter = Cell::new(0);  // 创建，注意 counter 不需要 mut

counter.set(42);             // 设置新值（整个替换）
let val = counter.get();     // 获取值的副本（Copy）
println!("{}", val);         // 42
```

### 工作原理

`Cell<T>` 的策略很粗暴：**整个值替换**。你不能获取内部值的引用，只能 `get`（复制一份出来）或 `set`（整个换掉）。因为操作的是值的副本，所以不存在悬垂引用的问题。

### 限制

- **仅限 `Copy` 类型**：`String`、`Vec` 等不能用
- **线程不安全**：`Cell<T>` 不实现 `Sync`，不能跨线程共享
- **无法获取内部引用**：只能 get/set，不能 borrow

### 代码示例：计数器

```rust
use std::cell::Cell;

struct HitCounter {
    data: Vec<String>,
    hits: Cell<u64>,  // 内部可变性：通过 &self 也能修改
}

impl HitCounter {
    fn new() -> Self {
        Self {
            data: vec!["hello".into(), "world".into()],
            hits: Cell::new(0),
        }
    }

    fn get(&self, index: usize) -> Option<&String> {
        // &self 方法中修改 hits —— 编译通过！
        self.hits.set(self.hits.get() + 1);
        self.data.get(index)
    }

    fn hit_count(&self) -> u64 {
        self.hits.get()
    }
}

fn main() {
    let counter = HitCounter::new();  // 不需要 mut
    counter.get(0);
    counter.get(1);
    counter.get(0);
    println!("命中次数: {}", counter.hit_count());  // 3
}
```

## 3. RefCell\<T\> — 运行时借用检查

### 适用场景

当你需要内部可变性，但操作的是非 `Copy` 类型（如 `String`、`Vec`）时，用 `RefCell<T>`。

### 核心 API

```rust
use std::cell::RefCell;

let data = RefCell::new(vec![1, 2, 3]);

// borrow() 返回 Ref<T> —— 不可变借用（运行时检查）
let borrowed = data.borrow();
println!("{:?}", *borrowed);  // [1, 2, 3]
drop(borrowed);  // 主动释放借用

// borrow_mut() 返回 RefMut<T> —— 可变借用（运行时检查）
let mut borrowed_mut = data.borrow_mut();
borrowed_mut.push(4);
println!("{:?}", *borrowed_mut);  // [1, 2, 3, 4]
```

### 编译时 vs 运行时：借用规则并没有消失

`RefCell<T>` 没有取消 Rust 的借用规则，只是把检查时机从**编译时**推迟到了**运行时**。规则仍然是：

- 同一时刻，多个 `borrow()` 可以共存（多个不可变借用）
- 同一时刻，只能有一个 `borrow_mut()`（唯一可变借用）
- `borrow()` 和 `borrow_mut()` 不能同时存在

违反规则会导致 **panic**，不是编译错误。

### Panic 示例：运行时借用冲突

```rust
use std::cell::RefCell;

let data = RefCell::new(vec![1, 2, 3]);

let borrow1 = data.borrow();      // 第一个不可变借用 - OK
let borrow2 = data.borrow();      // 第二个不可变借用 - OK（多读允许）

// 下面这行会 panic！因为还有不可变借用存在
// thread 'main' panicked at 'already borrowed: BorrowMutError'
let borrow_mut = data.borrow_mut();  // PANIC!
```

### Rc\<RefCell\<T\>\> — 最常见的单线程共享可变模式

当多个所有者需要共享且修改同一份数据时：

```rust
use std::cell::RefCell;
use std::rc::Rc;

// 多个组件共享同一份配置
let shared_config = Rc::new(RefCell::new(vec!["default".to_string()]));

// 克隆引用计数（不克隆数据）
let config_for_a = Rc::clone(&shared_config);
let config_for_b = Rc::clone(&shared_config);

// 组件 A 修改
config_for_a.borrow_mut().push("from_a".to_string());

// 组件 B 读取 —— 能看到 A 的修改
println!("{:?}", config_for_b.borrow());
// ["default", "from_a"]
```

### Cell vs RefCell 对比

| 特性 | Cell\<T\> | RefCell\<T\> |
|------|-----------|--------------|
| 适用类型 | `Copy` 类型 | 任意类型 |
| 获取方式 | `get()` 返回值的副本 | `borrow()` 返回引用 |
| 修改方式 | `set()` 整个替换 | `borrow_mut()` 获取可变引用 |
| 性能 | 零开销 | 有运行时借用计数 |
| 违规后果 | 编译错误（不让你借引用） | 运行时 panic |
| 线程安全 | 否 | 否 |

## 4. Mutex\<T\> — 线程安全的内部可变性

### 为什么需要 Mutex

`Cell` 和 `RefCell` 都是**单线程**方案，不能跨线程共享。在多线程环境下，你需要 `Mutex<T>`（互斥锁）。

### 核心 API

```rust
use std::sync::Mutex;

let data = Mutex::new(0);

// lock() 获取 MutexGuard —— 类似 RefCell::borrow_mut()
{
    let mut guard = data.lock().unwrap();  // 获取锁
    *guard += 1;                           // 修改数据
    // guard 在这里离开作用域，自动解锁（Drop trait）
}

// 再次获取
let guard = data.lock().unwrap();
println!("{}", *guard);  // 1
```

### 自动解锁：Drop trait 的妙用

`lock()` 返回一个 `MutexGuard`，它实现了 `Drop` trait。当 `MutexGuard` 离开作用域时，锁会自动释放：

```rust
use std::sync::Mutex;

let data = Mutex::new(vec![1, 2, 3]);

{
    let mut guard = data.lock().unwrap();
    guard.push(4);
    // guard 在这里被 drop，锁自动释放
}
// 此时锁已释放，其他线程可以获取

// 常见陷阱：不要让 guard 活得太长！
let mut guard = data.lock().unwrap();
guard.push(5);
// 如果下面还有很多代码，锁会一直被持有
// 最佳实践：用花括号限制 guard 的作用域
```

### 死锁风险和预防

```rust
use std::sync::Mutex;

let lock_a = Mutex::new(1);
let lock_b = Mutex::new(2);

// 死锁示例（不要这样做！）：
// 线程 1：先锁 A，再锁 B
// 线程 2：先锁 B，再锁 A
// 两者互相等待 -> 死锁！

// 预防措施：
// 1. 始终按相同顺序获取锁
// 2. 尽量减少持锁时间
// 3. 用 try_lock() 尝试获取，失败就放弃
// 4. 考虑用通道（channel）代替共享状态

let guard_a = lock_a.lock().unwrap();
if let Ok(guard_b) = lock_b.try_lock() {
    // 两个锁都拿到了
    println!("{} {}", *guard_a, *guard_b);
} else {
    // 拿不到就放弃，避免死锁
    println!("无法获取 lock_b，放弃");
}
```

### parking\_lot::Mutex vs std::sync::Mutex

ZeroClaw 项目实际使用的是 `parking_lot::Mutex` 而非标准库版本：

```rust
// std::sync::Mutex
use std::sync::Mutex;
let m = Mutex::new(0);
let guard = m.lock().unwrap();  // 返回 Result，需要 unwrap

// parking_lot::Mutex（ZeroClaw 使用的版本）
use parking_lot::Mutex;
let m = Mutex::new(0);
let guard = m.lock();  // 直接返回 MutexGuard，无需 unwrap
```

| 特性 | std::sync::Mutex | parking\_lot::Mutex |
|------|-----------------|-------------------|
| lock() 返回值 | `Result<MutexGuard>` | `MutexGuard`（不会中毒） |
| 性能 | 标准 | 通常更快（自旋 + 阻塞混合策略） |
| 大小 | 较大（含中毒标志） | 更小（1 个字节状态） |
| 中毒机制 | 有（线程 panic 后锁被"中毒"） | 无 |
| 生态地位 | 标准库自带 | 需要额外依赖，但业界广泛使用 |

### 代码示例：多线程共享计数器

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            let mut num = counter.lock().unwrap();
            *num += 1;
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("最终结果: {}", *counter.lock().unwrap());  // 10
}
```

## 5. RwLock\<T\> — 读写锁

### 多读单写

`Mutex` 的问题是：即使你只是读数据，也要独占锁。`RwLock` 允许**多个读者同时读**，但写的时候必须独占：

```rust
use std::sync::RwLock;

let config = RwLock::new(String::from("default"));

// 多个读者可以同时读
{
    let reader1 = config.read().unwrap();
    let reader2 = config.read().unwrap();  // OK！多读并行
    println!("{} {}", *reader1, *reader2);
}

// 写必须独占
{
    let mut writer = config.write().unwrap();
    *writer = String::from("updated");
    // 此时没有任何读者可以读
}
```

### 适用场景

- **读多写少**：配置中心、缓存系统、路由表
- **读写比例悬殊**：如 100 次读 1 次写
- **不适用场景**：读写频率差不多时，RwLock 开销可能比 Mutex 更大

### ZeroClaw 中的 RwLock 使用

```rust
// [来源: sourcecode/zeroclaw/src/config/schema.rs]
// 运行时代理配置使用 OnceLock<RwLock<ProxyConfig>>
// 读多写少的典型场景 —— 配置被频繁读取，偶尔更新
static RUNTIME_PROXY_CONFIG: OnceLock<RwLock<ProxyConfig>> = OnceLock::new();
static RUNTIME_PROXY_CLIENT_CACHE: OnceLock<RwLock<HashMap<String, reqwest::Client>>> =
    OnceLock::new();

fn runtime_proxy_state() -> &'static RwLock<ProxyConfig> {
    RUNTIME_PROXY_CONFIG.get_or_init(|| RwLock::new(ProxyConfig::default()))
}
```

## 6. Tokio Mutex vs std Mutex

### 为什么需要区分？

在异步（async）代码中，`std::sync::Mutex` 和 `tokio::sync::Mutex` 有本质区别：

```rust
// std::sync::Mutex —— 阻塞当前线程
let guard = std_mutex.lock().unwrap();
// 如果锁被占用，当前线程完全阻塞，无法执行其他任务

// tokio::sync::Mutex —— 异步等待
let guard = tokio_mutex.lock().await;
// 如果锁被占用，当前任务挂起，线程可以执行其他任务
```

### 什么时候用哪个？

| 场景 | 选择 | 原因 |
|------|------|------|
| 锁持有时间很短，不跨 `.await` | `std::sync::Mutex` | 性能更好，无异步开销 |
| 锁内需要 `.await`（如异步 IO） | `tokio::sync::Mutex` | 不会阻塞整个线程 |
| 锁持有时间较长 | `tokio::sync::Mutex` | 避免线程长时间阻塞 |
| 非异步代码 | `std::sync::Mutex` | 无需异步运行时 |

### 关键规则

**永远不要在异步代码中长时间持有 `std::sync::Mutex`**，否则会阻塞 Tokio 工作线程：

```rust
// 危险！在异步代码中持有 std Mutex 跨越 await 点
async fn bad_example(data: Arc<std::sync::Mutex<Vec<u8>>>) {
    let mut guard = data.lock().unwrap();
    // 下面的 await 会让当前任务挂起，但 std Mutex 的 guard 不会释放
    // 其他任务如果在同一线程上运行，会被完全阻塞
    tokio::time::sleep(Duration::from_secs(1)).await;  // 危险！
    guard.push(42);
}

// 正确做法 1：用 tokio::sync::Mutex
async fn good_example_1(data: Arc<tokio::sync::Mutex<Vec<u8>>>) {
    let mut guard = data.lock().await;  // 异步等待
    tokio::time::sleep(Duration::from_secs(1)).await;
    guard.push(42);
}

// 正确做法 2：缩短 std Mutex 持锁范围，不跨越 await
async fn good_example_2(data: Arc<std::sync::Mutex<Vec<u8>>>) {
    {
        let mut guard = data.lock().unwrap();
        guard.push(42);
    }  // guard 在这里释放
    tokio::time::sleep(Duration::from_secs(1)).await;  // 安全！锁已释放
}
```

### ZeroClaw 的选择

```rust
// [来源: sourcecode/zeroclaw/crates/robot-kit/src/drive.rs]
// DriveTool 使用 tokio::sync::Mutex，因为 execute() 是 async 方法
use tokio::sync::Mutex;

pub struct DriveTool {
    config: RobotConfig,
    backend: Arc<dyn DriveBackend>,
    last_command: Arc<Mutex<Option<std::time::Instant>>>,  // tokio Mutex
}

// 在 async fn execute 中使用 .await 获取锁
async fn execute(&self, args: Value) -> Result<ToolResult> {
    let mut last = self.last_command.lock().await;  // 异步等待
    if let Some(instant) = *last {
        if instant.elapsed() < Duration::from_secs(1) {
            return Ok(ToolResult {
                success: false,
                output: String::new(),
                error: Some("Rate limited: wait 1 second between drive commands".to_string()),
            });
        }
    }
    *last = Some(std::time::Instant::now());
}
```

## 7. Arc\<Mutex\<T\>\> 模式

### 最常见的线程安全共享可变状态模式

当你需要在多个线程（或异步任务）之间共享可变数据时，`Arc<Mutex<T>>` 是默认选择：

- `Arc`：原子引用计数，让多个所有者安全共享同一份堆数据
- `Mutex`：互斥锁，保证同一时刻只有一个线程修改数据

```
Arc<Mutex<T>> = 共享所有权 + 互斥修改

TypeScript 类比：
const sharedState = new SharedWorkerState();  // 多个 Worker 共享
// JS 是单线程的所以不需要锁，但 Rust 多线程必须保护
```

### 完整代码示例

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;

#[derive(Debug)]
struct SharedState {
    messages: Vec<String>,
    active_users: u32,
}

fn main() {
    // 创建共享状态
    let state = Arc::new(Mutex::new(SharedState {
        messages: Vec::new(),
        active_users: 0,
    }));

    let mut handles = vec![];

    // 启动 5 个"用户"线程
    for i in 0..5 {
        let state = Arc::clone(&state);
        let handle = thread::spawn(move || {
            // 用户登录
            {
                let mut s = state.lock().unwrap();
                s.active_users += 1;
                s.messages.push(format!("用户 {} 登录", i));
            }  // 释放锁

            // 模拟工作
            thread::sleep(Duration::from_millis(100));

            // 用户操作
            {
                let mut s = state.lock().unwrap();
                s.messages.push(format!("用户 {} 完成操作", i));
            }

            // 用户登出
            {
                let mut s = state.lock().unwrap();
                s.active_users -= 1;
                s.messages.push(format!("用户 {} 登出", i));
            }
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    // 最终状态
    let final_state = state.lock().unwrap();
    println!("活跃用户: {}", final_state.active_users);      // 0
    println!("消息总数: {}", final_state.messages.len());     // 15
    println!("所有消息:");
    for msg in &final_state.messages {
        println!("  - {}", msg);
    }
}
```

### 性能考虑

| 考量点 | 建议 |
|--------|------|
| 锁粒度 | 尽量细粒度 —— 用多个小 Mutex 代替一个大 Mutex |
| 持锁时间 | 尽量短 —— 拿到锁后立即操作，操作完立即释放 |
| 读写比 | 读多写少时用 `Arc<RwLock<T>>` 代替 |
| 无锁替代 | 纯计数器考虑 `AtomicU64`；消息传递考虑 channel |
| 克隆策略 | 数据量小时可以 `clone()` 后释放锁，避免长时间持锁 |

## 8. 选择决策表

一张表搞定"该用哪个"：

| 场景 | 选择 | 理由 |
|------|------|------|
| Copy 类型 + 单线程 | `Cell<T>` | 最简单，零开销 |
| 非 Copy + 单线程 | `RefCell<T>` | 支持引用，运行时检查 |
| 共享 + 单线程 | `Rc<RefCell<T>>` | 多所有者 + 内部可变 |
| 多线程同步 | `Mutex<T>` | 互斥锁，简单直接 |
| 多线程读多写少 | `RwLock<T>` | 允许并行读 |
| 多线程共享 | `Arc<Mutex<T>>` | 共享所有权 + 互斥 |
| 多线程异步 | `tokio::sync::Mutex<T>` | 异步等待，不阻塞线程 |
| 异步 + 共享 | `Arc<tokio::sync::Mutex<T>>` | 跨异步任务共享 |
| 全局配置（读多写少） | `OnceLock<RwLock<T>>` | 延迟初始化 + 并行读 |
| 简单原子操作 | `AtomicU64` / `AtomicBool` | 无锁，性能最高 |

### 决策流程图（文字版）

```
需要内部可变性吗？
├── 否 → 用普通 &mut
└── 是 → 跨线程吗？
    ├── 否（单线程）
    │   ├── Copy 类型？ → Cell<T>
    │   └── 非 Copy？  → RefCell<T>
    │       └── 需要多个所有者？ → Rc<RefCell<T>>
    └── 是（多线程）
        ├── 纯计数/标志？ → Atomic 类型
        ├── 读多写少？   → RwLock<T>
        ├── 异步上下文？  → tokio::sync::Mutex<T>
        └── 通用场景？   → Mutex<T>
            └── 需要多个所有者？ → Arc<Mutex<T>>
```

## 9. ZeroClaw 实战实例

### 实例 1：ActionTracker — 通过 &self 记录操作时间

```rust
// [来源: sourcecode/zeroclaw/src/security/policy.rs]
// 滑动窗口操作追踪器，用于速率限制

use parking_lot::Mutex;
use std::time::Instant;

pub struct ActionTracker {
    /// 最近操作的时间戳（保留最近一小时内的）
    actions: Mutex<Vec<Instant>>,  // parking_lot::Mutex 实现内部可变性
}

impl ActionTracker {
    pub fn new() -> Self {
        Self {
            actions: Mutex::new(Vec::new()),
        }
    }

    /// 记录一次操作，返回窗口期内的操作总数
    pub fn record(&self) -> usize {       // 注意：&self，不是 &mut self！
        let mut actions = self.actions.lock();  // parking_lot 不需要 unwrap
        let cutoff = Instant::now()
            .checked_sub(std::time::Duration::from_secs(3600))
            .unwrap_or_else(Instant::now);
        actions.retain(|t| *t > cutoff);   // 清理过期记录
        actions.push(Instant::now());      // 添加新记录
        actions.len()                      // 返回当前计数
    }
}
```

**为什么用 Mutex 而非 &mut self？**

`ActionTracker` 被嵌入 `SecurityPolicy` 结构体中，而 `SecurityPolicy` 可能被多个组件共享引用。如果 `record()` 需要 `&mut self`，那调用者就必须持有可变引用，这在共享场景下不可能。`Mutex` 让 `record()` 只需要 `&self`，调用者无感知。

### 实例 2：DriveTool — Arc\<Mutex\<...\>\> 限速

```rust
// [来源: sourcecode/zeroclaw/crates/robot-kit/src/drive.rs]
// 电机驱动工具，使用异步 Mutex 实现命令限速

use tokio::sync::Mutex;
use std::sync::Arc;

pub struct DriveTool {
    config: RobotConfig,
    backend: Arc<dyn DriveBackend>,
    // 记录最后一次命令时间，用于限速（1秒间隔）
    last_command: Arc<Mutex<Option<std::time::Instant>>>,
}

// 在 execute 方法中：
async fn execute(&self, args: Value) -> Result<ToolResult> {
    {
        let mut last = self.last_command.lock().await;
        if let Some(instant) = *last {
            if instant.elapsed() < Duration::from_secs(1) {
                return Ok(ToolResult {
                    success: false,
                    error: Some("Rate limited".to_string()),
                    ..
                });
            }
        }
        *last = Some(std::time::Instant::now());
    }
    // ... 执行驱动命令
}
```

**为什么选 `tokio::sync::Mutex`？** 因为 `execute()` 是 `async fn`，整个方法在异步运行时中执行。使用 `tokio::sync::Mutex` 可以在等待锁时让出线程给其他任务。

**为什么用 `Arc`？** 因为 `DriveTool` 可能被 `clone` 或跨任务共享，`Arc` 确保多个副本指向同一个 `Mutex`。

### 实例 3：Config 全局状态 — OnceLock\<RwLock\<T\>\>

```rust
// [来源: sourcecode/zeroclaw/src/config/schema.rs]
// 运行时代理配置 —— 全局单例 + 读写锁

use std::sync::{OnceLock, RwLock};
use std::collections::HashMap;

// OnceLock：线程安全的延迟初始化（只写一次）
// RwLock：允许多个读者并行读取配置，写入时独占
static RUNTIME_PROXY_CONFIG: OnceLock<RwLock<ProxyConfig>> = OnceLock::new();
static RUNTIME_PROXY_CLIENT_CACHE: OnceLock<RwLock<HashMap<String, reqwest::Client>>> =
    OnceLock::new();

fn runtime_proxy_state() -> &'static RwLock<ProxyConfig> {
    // get_or_init：第一次调用时初始化，后续调用直接返回
    RUNTIME_PROXY_CONFIG.get_or_init(|| RwLock::new(ProxyConfig::default()))
}

fn runtime_proxy_client_cache() -> &'static RwLock<HashMap<String, reqwest::Client>> {
    RUNTIME_PROXY_CLIENT_CACHE.get_or_init(|| RwLock::new(HashMap::new()))
}
```

**这个模式的三层结构：**
1. `OnceLock` —— 保证全局变量只初始化一次（线程安全的 lazy init）
2. `RwLock` —— 配置被频繁读取、偶尔更新，读写锁完美契合
3. `ProxyConfig` / `HashMap` —— 实际的业务数据

**为什么不用 `Mutex`？** 配置是典型的**读多写少**场景。用 `RwLock` 可以让多个请求处理器同时读取配置，只在更新配置时才独占锁。

---

## 总结

内部可变性是 Rust 在**安全性**和**灵活性**之间的精心平衡：

- 它没有破坏 Rust 的安全保证，只是把检查时机从编译时推迟到了运行时（或通过同步原语保证）
- 从 `Cell`（最简单）到 `Arc<tokio::sync::Mutex>`（最强大），根据场景选择复杂度最低的方案
- ZeroClaw 项目展示了真实世界中这些工具的使用：速率限制用 `Mutex`、全局配置用 `OnceLock<RwLock>`、异步场景用 `tokio::sync::Mutex`
- **黄金法则：能用编译时检查就用编译时检查，只在必要时才使用内部可变性**
