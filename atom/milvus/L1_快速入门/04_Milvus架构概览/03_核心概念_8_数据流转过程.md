## 一、原理讲解

### 1.1 什么是数据流转过程？

**数据流转过程（Data Flow）** 是指数据在 Milvus 分布式架构中从写入到查询的完整生命周期，包括插入、索引构建、查询执行、删除等操作。

**核心流程：**
```
数据生命周期：
Client → Proxy → Streaming Node → Data Node → Object Storage
                                              ↓
                                         Index Node
                                              ↓
                                         Query Node → Client
```

### 1.2 数据流转的三大核心流程

#### 1.2.1 插入流程（Insert Flow）

```
插入流程（Milvus 2.6）：
┌──────────┐
│  Client  │ ← 1. 发起插入请求
└──────────┘
     ↓ insert(vectors)
┌──────────────┐
│ Proxy Node   │ ← 2. 路由到 VChannel
└──────────────┘
     ↓ 3. 写入 WAL
┌──────────────┐
│Streaming Node│ ← 4. Woodpecker 零磁盘 WAL
│ (Woodpecker) │    直接写入对象存储
└──────────────┘
     ↓ 5. 通知订阅者
┌──────────────┐
│  Data Node   │ ← 6. 消费 WAL，缓冲数据
└──────────────┘
     ↓ 7. Flush（达到阈值）
┌──────────────┐
│Object Storage│ ← 8. 持久化 Binlog
│ (Binlog)     │
└──────────────┘
     ↓ 9. 通知 DataCoord
┌──────────────┐
│  DataCoord   │ ← 10. 调度索引构建
└──────────────┘
     ↓ 11. 分配任务
┌──────────────┐
│ Index Node   │ ← 12. 构建索引
└──────────────┘
     ↓ 13. 上传索引
┌──────────────┐
│Object Storage│ ← 14. 索引文件
│ (Index)      │
└──────────────┘
```

**关键步骤详解：**

1. **Client 发起请求**：客户端调用 `insert()` API
2. **Proxy 路由**：根据主键哈希分配到 VChannel
3. **Streaming Node 写入 WAL**：零磁盘架构，直接写入对象存储
4. **Data Node 消费**：订阅 WAL，在内存中缓冲数据
5. **Flush 持久化**：达到阈值后，将数据写入对象存储（Binlog）
6. **索引构建**：Index Node 异步构建索引
7. **索引上传**：索引文件上传到对象存储

#### 1.2.2 查询流程（Search Flow）

```
查询流程（Milvus 2.6）：
┌──────────┐
│  Client  │ ← 1. 发起查询请求
└──────────┘
     ↓ search(vector, top_k)
┌──────────────┐
│ Proxy Node   │ ← 2. 路由到 Query Node
└──────────────┘
     ↓ 3. 分发查询
┌──────────────┐
│ Query Node 1 │ ← 4. 查询历史数据（Sealed Segment）
└──────────────┘    从对象存储加载索引
     ↓ 5. 返回 Top-K
┌──────────────┐
│Streaming Node│ ← 6. 查询实时数据（Growing Segment）
│ (Woodpecker) │    从 WAL 读取未 Flush 的数据
└──────────────┘
     ↓ 7. 返回 Top-K
┌──────────────┐
│ Proxy Node   │ ← 8. 聚合结果，全局 Top-K
└──────────────┘
     ↓ 9. 返回结果
┌──────────┐
│  Client  │
└──────────┘
```

**关键步骤详解：**

1. **Client 发起请求**：客户端调用 `search()` API
2. **Proxy 路由**：根据 Collection 分配到 Query Node
3. **Query Node 查询历史数据**：在已索引的 Sealed Segment 中查询
4. **Streaming Node 查询实时数据**：在未 Flush 的 Growing Segment 中查询
5. **Proxy 聚合结果**：合并所有结果，返回全局 Top-K

#### 1.2.3 删除流程（Delete Flow）

```
删除流程（逻辑删除 + 异步清理）：
┌──────────┐
│  Client  │ ← 1. 发起删除请求
└──────────┘
     ↓ delete(expr)
┌──────────────┐
│ Proxy Node   │ ← 2. 路由到 VChannel
└──────────────┘
     ↓ 3. 写入 Delete Log
┌──────────────┐
│Streaming Node│ ← 4. 记录删除操作到 WAL
└──────────────┘
     ↓ 5. 通知订阅者
┌──────────────┐
│  Data Node   │ ← 6. 消费 Delete Log
└──────────────┘
     ↓ 7. 标记删除（逻辑删除）
┌──────────────┐
│Object Storage│ ← 8. 持久化 Delete Log
│ (Delete Log) │
└──────────────┘
     ↓ 9. 后台 Compaction
┌──────────────┐
│  Data Node   │ ← 10. 物理删除数据
└──────────────┘
     ↓ 11. 生成新 Segment
┌──────────────┐
│Object Storage│ ← 12. 新 Binlog（已清理）
└──────────────┘
```

**关键概念：**

- **逻辑删除**：只标记删除，不立即清理数据
- **Delete Log**：记录删除的主键列表
- **Compaction**：后台异步清理已删除的数据
- **物理删除**：Compaction 时真正删除数据

### 1.3 Segment 的生命周期

```
Segment 状态转换：
┌─────────────┐
│  Growing    │ ← 正在写入，数据在内存中
└─────────────┘
      ↓ 达到大小阈值或时间阈值
┌─────────────┐
│   Sealed    │ ← 封闭，不再接受写入
└─────────────┘
      ↓ Flush 操作
┌─────────────┐
│  Flushed    │ ← 持久化到对象存储（Binlog）
└─────────────┘
      ↓ 索引构建
┌─────────────┐
│  Indexed    │ ← 索引构建完成，可高效查询
└─────────────┘
```

**状态详解：**

1. **Growing（增长中）**
   - 数据在 Data Node 内存中
   - 持续接受新的写入
   - 查询需要扫描全部数据（无索引）

2. **Sealed（已封闭）**
   - 不再接受新的写入
   - 等待 Flush 操作
   - 仍在内存中

3. **Flushed（已持久化）**
   - 数据已写入对象存储（Binlog）
   - 可以从内存中释放
   - 等待索引构建

4. **Indexed（已索引）**
   - 索引构建完成
   - Query Node 可以加载并高效查询
   - 最终状态

### 1.4 数据平面 vs 控制平面

```
Milvus 架构分层：
┌─────────────────────────────────────────────────────────┐
│                    控制平面（Control Plane）             │
│  ┌────────────────────────────────────────────────┐    │
│  │  MixCoord (Coordinator 层)                     │    │
│  │  - 元数据管理                                   │    │
│  │  - 任务调度                                     │    │
│  │  - 资源分配                                     │    │
│  └────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
                          ↓ 调度指令
┌─────────────────────────────────────────────────────────┐
│                    数据平面（Data Plane）                │
│  ┌────────────────────────────────────────────────┐    │
│  │  Worker 层                                      │    │
│  │  - Data Node: 数据写入和持久化                  │    │
│  │  - Query Node: 查询执行                         │    │
│  │  - Index Node: 索引构建                         │    │
│  │  - Streaming Node: 实时数据流                   │    │
│  └────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
                          ↓ 数据存储
┌─────────────────────────────────────────────────────────┐
│                    存储层（Storage Layer）               │
│  - Object Storage: 向量数据、索引、WAL                  │
│  - Meta Storage (etcd): 元数据                          │
└─────────────────────────────────────────────────────────┘
```

**职责分离：**

- **控制平面**：决策和调度，不处理实际数据
- **数据平面**：执行数据处理任务
- **存储层**：持久化所有数据

## 二、手写实现：简化版数据流转

下面我们用 Python 实现一个简化版的数据流转过程，展示插入、查询、删除的完整流程。

### 2.1 基础数据结构

```python
from dataclasses import dataclass
from typing import List, Dict, Optional
from enum import Enum
import numpy as np
import time

class SegmentState(Enum):
    """Segment 状态"""
    GROWING = "growing"
    SEALED = "sealed"
    FLUSHED = "flushed"
    INDEXED = "indexed"

@dataclass
class Vector:
    """向量数据"""
    id: int
    embedding: np.ndarray
    metadata: Dict
    deleted: bool = False  # 逻辑删除标记

@dataclass
class Segment:
    """Segment"""
    segment_id: int
    state: SegmentState
    vectors: List[Vector]
    max_size: int = 1000

    def is_full(self) -> bool:
        return len(self.vectors) >= self.max_size

    def seal(self):
        self.state = SegmentState.SEALED

    def flush(self):
        self.state = SegmentState.FLUSHED

    def build_index(self):
        self.state = SegmentState.INDEXED
```

### 2.2 插入流程实现

```python
class InsertFlow:
    """插入流程"""

    def __init__(self):
        self.segments: Dict[int, Segment] = {}
        self.next_segment_id = 1
        self.current_segment: Optional[Segment] = None

    def insert(self, vectors: List[Vector]) -> List[int]:
        """插入向量"""
        print(f"\n[InsertFlow] Inserting {len(vectors)} vectors")

        inserted_ids = []

        for vector in vectors:
            # 1. 检查是否需要创建新 Segment
            if not self.current_segment or self.current_segment.is_full():
                self._create_new_segment()

            # 2. 插入到 Growing Segment
            self.current_segment.vectors.append(vector)
            inserted_ids.append(vector.id)

            print(f"  Inserted vector {vector.id} to segment {self.current_segment.segment_id}")

            # 3. 检查是否需要 Seal
            if self.current_segment.is_full():
                self._seal_segment(self.current_segment.segment_id)

        return inserted_ids

    def _create_new_segment(self):
        """创建新 Segment"""
        segment_id = self.next_segment_id
        self.next_segment_id += 1

        segment = Segment(
            segment_id=segment_id,
            state=SegmentState.GROWING,
            vectors=[]
        )

        self.segments[segment_id] = segment
        self.current_segment = segment

        print(f"[InsertFlow] Created new segment {segment_id} (Growing)")

    def _seal_segment(self, segment_id: int):
        """Seal Segment"""
        segment = self.segments[segment_id]
        segment.seal()

        print(f"[InsertFlow] Sealed segment {segment_id}")

        # 触发 Flush
        self._flush_segment(segment_id)

    def _flush_segment(self, segment_id: int):
        """Flush Segment 到对象存储"""
        segment = self.segments[segment_id]

        print(f"[InsertFlow] Flushing segment {segment_id} to object storage...")
        time.sleep(0.1)  # 模拟 I/O

        segment.flush()
        print(f"[InsertFlow] Segment {segment_id} flushed (Binlog created)")

        # 触发索引构建
        self._build_index(segment_id)

    def _build_index(self, segment_id: int):
        """构建索引"""
        segment = self.segments[segment_id]

        print(f"[InsertFlow] Building index for segment {segment_id}...")
        time.sleep(0.2)  # 模拟索引构建

        segment.build_index()
        print(f"[InsertFlow] Index built for segment {segment_id}")

### 2.3 查询流程实现

```python
class SearchFlow:
    """查询流程"""

    def __init__(self, insert_flow: InsertFlow):
        self.insert_flow = insert_flow

    def search(self, query_vector: np.ndarray, top_k: int = 10) -> List[Dict]:
        """查询向量"""
        print(f"\n[SearchFlow] Searching with top_k={top_k}")

        all_results = []

        # 1. 查询所有 Segment（包括 Growing 和 Indexed）
        for segment_id, segment in self.insert_flow.segments.items():
            print(f"  Searching segment {segment_id} ({segment.state.value})")

            # 过滤已删除的向量
            active_vectors = [v for v in segment.vectors if not v.deleted]

            if not active_vectors:
                continue

            # 计算相似度（简化：使用 L2 距离）
            embeddings = np.array([v.embedding for v in active_vectors])
            distances = np.linalg.norm(embeddings - query_vector, axis=1)

            # 获取 Top-K
            top_k_indices = np.argsort(distances)[:top_k]

            for idx in top_k_indices:
                vector = active_vectors[idx]
                all_results.append({
                    "segment_id": segment_id,
                    "vector_id": vector.id,
                    "distance": float(distances[idx]),
                    "metadata": vector.metadata
                })

        # 2. 全局排序并返回 Top-K
        all_results.sort(key=lambda x: x["distance"])
        final_results = all_results[:top_k]

        print(f"[SearchFlow] Found {len(final_results)} results")
        return final_results

### 2.4 删除流程实现

```python
class DeleteFlow:
    """删除流程"""

    def __init__(self, insert_flow: InsertFlow):
        self.insert_flow = insert_flow

    def delete(self, vector_ids: List[int]):
        """删除向量（逻辑删除）"""
        print(f"\n[DeleteFlow] Deleting {len(vector_ids)} vectors (logical deletion)")

        deleted_count = 0

        # 1. 逻辑删除：标记为已删除
        for segment in self.insert_flow.segments.values():
            for vector in segment.vectors:
                if vector.id in vector_ids:
                    vector.deleted = True
                    deleted_count += 1
                    print(f"  Marked vector {vector.id} as deleted in segment {segment.segment_id}")

        print(f"[DeleteFlow] Logically deleted {deleted_count} vectors")

        # 2. 触发 Compaction（异步清理）
        self._trigger_compaction()

    def _trigger_compaction(self):
        """触发 Compaction（物理删除）"""
        print(f"\n[DeleteFlow] Triggering compaction (physical deletion)...")

        for segment_id, segment in list(self.insert_flow.segments.items()):
            # 只对 Flushed 或 Indexed 的 Segment 进行 Compaction
            if segment.state in [SegmentState.FLUSHED, SegmentState.INDEXED]:
                deleted_count = sum(1 for v in segment.vectors if v.deleted)

                if deleted_count > 0:
                    print(f"  Compacting segment {segment_id}: {deleted_count} deleted vectors")

                    # 物理删除
                    segment.vectors = [v for v in segment.vectors if not v.deleted]

                    print(f"  Segment {segment_id} compacted: {len(segment.vectors)} vectors remaining")

### 2.5 完整示例

```python
def data_flow_demo():
    """演示完整的数据流转过程"""

    print("=" * 60)
    print("Data Flow Demo: Insert → Search → Delete")
    print("=" * 60)

    # 1. 初始化流程
    insert_flow = InsertFlow()
    search_flow = SearchFlow(insert_flow)
    delete_flow = DeleteFlow(insert_flow)

    # 2. 插入数据
    print("\n[Step 1] Inserting Data")
    print("-" * 60)

    vectors = [
        Vector(id=i, embedding=np.random.rand(128), metadata={"text": f"doc_{i}"})
        for i in range(1500)  # 超过 Segment 大小限制，触发多个 Segment
    ]

    insert_flow.insert(vectors)

    # 3. 查询数据
    print("\n[Step 2] Searching Data")
    print("-" * 60)

    query_vector = np.random.rand(128)
    results = search_flow.search(query_vector, top_k=5)

    print("\nSearch Results:")
    for i, result in enumerate(results):
        print(f"  {i+1}. Vector {result['vector_id']}, Distance: {result['distance']:.4f}, Segment: {result['segment_id']}")

    # 4. 删除数据
    print("\n[Step 3] Deleting Data")
    print("-" * 60)

    delete_ids = [0, 1, 2, 3, 4]  # 删除前 5 个向量
    delete_flow.delete(delete_ids)

    # 5. 再次查询（验证删除）
    print("\n[Step 4] Searching After Deletion")
    print("-" * 60)

    results_after_delete = search_flow.search(query_vector, top_k=5)

    print("\nSearch Results After Deletion:")
    for i, result in enumerate(results_after_delete):
        print(f"  {i+1}. Vector {result['vector_id']}, Distance: {result['distance']:.4f}")

    # 验证已删除的向量不在结果中
    deleted_in_results = any(r['vector_id'] in delete_ids for r in results_after_delete)
    if not deleted_in_results:
        print("\n✅ Deleted vectors are not in search results")
    else:
        print("\n❌ Deleted vectors still in search results")

    print("\n" + "=" * 60)
    print("Data Flow Demo Completed")
    print("=" * 60)


if __name__ == "__main__":
    data_flow_demo()
```

## 三、RAG 相关应用场景

### 3.1 场景 1：RAG 系统的实时文档更新

**问题：** RAG 系统需要实时插入新文档，如何保证数据流转高效？

**数据流转的作用：**
- **插入流程**：批量插入，减少 Flush 次数
- **Segment 管理**：自动 Seal 和 Flush，无需人工干预
- **索引构建**：后台异步构建，不阻塞插入

**实际代码示例：**

```python
from pymilvus import connections, Collection
import time

connections.connect(host="localhost", port="19530")

def realtime_document_update(collection_name: str, documents: list):
    """实时更新 RAG 文档库"""

    collection = Collection(collection_name)

    # 1. 批量插入文档
    print(f"Inserting {len(documents)} documents...")
    start_time = time.time()

    texts = [doc["text"] for doc in documents]
    embeddings = [doc["embedding"] for doc in documents]

    # 插入数据（触发插入流程）
    insert_result = collection.insert([texts, embeddings])

    elapsed = time.time() - start_time
    print(f"✅ Inserted in {elapsed:.2f}s")

    # 2. 数据流转过程（自动）
    # - Proxy 路由到 VChannel
    # - Streaming Node 写入 WAL
    # - Data Node 消费并缓冲
    # - 达到阈值后自动 Flush
    # - Index Node 异步构建索引

    # 3. 可选：手动 Flush（强制持久化）
    collection.flush()
    print(f"✅ Data flushed to object storage")

    # 4. 验证数据可查询
    query_vector = embeddings[0]
    results = collection.search(
        data=[query_vector],
        anns_field="embedding",
        param={"metric_type": "L2", "params": {"nprobe": 10}},
        limit=5
    )

    print(f"✅ Query successful: {len(results[0])} results")

# 示例
documents = [
    {"text": f"Document {i}", "embedding": np.random.rand(768).tolist()}
    for i in range(1000)
]
realtime_document_update("rag_knowledge_base", documents)
```

### 3.3 场景 3：RAG 系统的文档删除

**问题：** RAG 系统需要删除过时或错误的文档

**数据流转的作用：**
- **逻辑删除**：立即标记删除，不影响性能
- **Compaction**：后台异步清理，释放存储空间
- **查询过滤**：自动过滤已删除的文档

**实际代码示例：**

```python
def delete_outdated_documents(collection_name: str, cutoff_timestamp: int):
    """删除过时的文档"""

    collection = Collection(collection_name)

    # 1. 查询需要删除的文档
    expr = f"timestamp < {cutoff_timestamp}"
    outdated_docs = collection.query(expr=expr, output_fields=["id"])

    print(f"Found {len(outdated_docs)} outdated documents")

    # 2. 删除文档（逻辑删除）
    if outdated_docs:
        ids_to_delete = [doc["id"] for doc in outdated_docs]

        collection.delete(expr=f"id in {ids_to_delete}")
        print(f"✅ Logically deleted {len(ids_to_delete)} documents")

    # 3. 数据流转过程（自动）
    # - Proxy 路由删除请求
    # - Streaming Node 写入 Delete Log
    # - Data Node 标记为已删除
    # - Compaction 后台物理删除

    # 4. 触发 Compaction（可选）
    collection.compact()
    print(f"✅ Compaction triggered (physical deletion)")

    # 5. 验证删除
    remaining_docs = collection.query(expr=expr, output_fields=["id"])
    print(f"✅ Remaining outdated documents: {len(remaining_docs)}")

# 示例
cutoff_timestamp = int(time.time()) - (30 * 24 * 3600)  # 30 天前
delete_outdated_documents("rag_knowledge_base", cutoff_timestamp)
```

### 3.4 场景 4：RAG 系统的数据流监控

**问题：** 如何监控 RAG 系统的数据流转状态？

**数据流转的作用：**
- **Segment 状态**：监控 Growing/Sealed/Flushed/Indexed
- **数据延迟**：监控插入到可查询的延迟
- **Compaction 进度**：监控删除清理进度

**实际代码示例：**

```python
from pymilvus import utility

def monitor_data_flow(collection_name: str):
    """监控数据流转状态"""

    print(f"Monitoring data flow for: {collection_name}")
    print("=" * 60)

    # 1. 获取 Segment 信息
    segments = utility.get_query_segment_info(collection_name)

    segment_stats = {
        "growing": 0,
        "sealed": 0,
        "flushed": 0,
        "indexed": 0
    }

    total_rows = 0

    for seg in segments:
        # 根据 Segment 状态分类
        if seg.state == "Growing":
            segment_stats["growing"] += 1
        elif seg.state == "Sealed":
            segment_stats["sealed"] += 1
        elif seg.state == "Flushed":
            segment_stats["flushed"] += 1
        elif seg.state == "Indexed":
            segment_stats["indexed"] += 1

        total_rows += seg.num_rows

    # 2. 显示统计信息
    print(f"\nSegment Statistics:")
    print(f"  Growing: {segment_stats['growing']} segments")
    print(f"  Sealed: {segment_stats['sealed']} segments")
    print(f"  Flushed: {segment_stats['flushed']} segments")
    print(f"  Indexed: {segment_stats['indexed']} segments")
    print(f"  Total rows: {total_rows}")

    # 3. 计算数据流转健康度
    indexed_ratio = segment_stats["indexed"] / max(sum(segment_stats.values()), 1)

    print(f"\nData Flow Health:")
    print(f"  Indexed ratio: {indexed_ratio * 100:.1f}%")

    if indexed_ratio > 0.8:
        print(f"  Status: ✅ Healthy (most data indexed)")
    elif indexed_ratio > 0.5:
        print(f"  Status: ⚠️  Warning (some data not indexed)")
    else:
        print(f"  Status: ❌ Unhealthy (too much unindexed data)")

# 示例
monitor_data_flow("rag_knowledge_base")
```

## 四、总结

### 4.1 数据流转的核心价值

1. **完整的生命周期管理**：从插入到查询到删除的全流程
2. **自动化流程**：Seal、Flush、索引构建自动触发
3. **高效的查询**：Growing + Indexed Segment 混合查询
4. **逻辑删除 + 异步清理**：删除不影响性能

### 4.2 Milvus 2.6 的创新

- **Streaming Node**：统一管理实时数据流
- **零磁盘 WAL**：直接写入对象存储
- **流批分离**：实时数据和历史数据分离处理

### 4.3 对 RAG 开发的影响

- **更快的数据更新**：实时插入，立即可查询
- **更高的查询性能**：自动索引构建，优化查询
- **更灵活的数据管理**：逻辑删除 + 异步清理
- **更简单的运维**：自动化流程，无需人工干预

### 4.4 类比总结

**前端类比：**
- Insert Flow = 数据提交（Form Submit）
- Search Flow = 数据查询（API Request）
- Delete Flow = 数据删除（Soft Delete）

- Insert Flow = 图书入库（登记、上架、编目）
