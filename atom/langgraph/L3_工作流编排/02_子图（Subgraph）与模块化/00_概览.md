## 一、知识点总览

### 1.1 核心定位

**子图（Subgraph）是什么？**

子图是 LangGraph 中将一个编译后的 StateGraph 作为另一个图的节点来使用的机制。它允许你将复杂的工作流拆分为独立的、可复用的子模块，每个子模块拥有自己的状态定义、节点和边，然后通过 `add_node()` 嵌入到父图中。

**关键特征**:
- **编译后嵌入**: 子图必须先 `compile()` 成 `CompiledStateGraph`，再通过 `add_node()` 添加到父图
- **状态隔离与共享**: 子图有独立的 State Schema，通过共享键或转换函数与父图通信
- **递归嵌套**: 子图可以包含子图，支持多层嵌套
- **透明执行**: 父图调用子图时，子图内部的节点逐步执行，支持流式输出和调试
- **独立可测试**: 每个子图可以独立编译、独立测试、独立运行

### 1.2 为什么需要子图？

**核心问题**: 当工作流变得复杂（几十个节点、多种逻辑分支），如何保持代码可维护、可复用、可测试？

**没有子图的痛点**:
- 所有节点平铺在一个图中，难以理解整体逻辑
- 重复的工作流模式无法复用，导致代码冗余
- 团队协作困难，多人修改同一个大图容易冲突
- 测试粒度太粗，无法单独验证某个子流程
- 不同场景需要不同的状态结构，单一 Schema 难以适配

**子图的解决方案**:
- **模块化拆分**: 将复杂图拆分为语义清晰的子图，每个子图负责一个明确的职责
- **复用组合**: 同一个子图可以在多个父图中复用，类似函数调用
- **独立开发**: 团队成员可以独立开发和测试各自负责的子图
- **状态隔离**: 每个子图有独立的状态空间，避免全局状态污染
- **灵活集成**: 支持共享键自动传递和手动转换两种集成方式

### 1.3 适用场景

**最佳实践场景**:
1. **多代理系统**: 每个代理作为独立子图，通过 Supervisor 或 Handoff 模式协调
2. **复杂文档处理**: 文档解析、分块、向量化等步骤各自独立为子图
3. **多阶段审批流程**: 每个审批阶段作为独立子图，支持不同审批逻辑
4. **并行任务处理**: 使用 Send 将多个任务分发到同一个子图并行执行（Map-Reduce）
5. **可插拔功能模块**: 如 RAG 检索子图、工具调用子图，按需组合

## 三、核心概念列表

本知识点包含 **6 个核心概念**，每个概念都有独立的详细文档：

### 概念 1: 子图定义与编译

**核心要点**:
- 子图就是一个普通的 StateGraph，通过 `compile()` 编译后成为可嵌入的节点
- `input_schema` 定义子图接受的输入字段子集（过滤父图传入的无关字段）
- `output_schema` 定义子图返回的输出字段子集（控制回传给父图的内容）
- 子图编译后通过 `parent_graph.add_node("subgraph_name", compiled_subgraph)` 嵌入父图

**关键代码**:
```python
# 子图定义
subgraph_builder = StateGraph(SubgraphState)
subgraph_builder.add_node(...)
subgraph_builder.add_edge(...)
subgraph = subgraph_builder.compile()

# 嵌入父图
parent_builder = StateGraph(ParentState)
parent_builder.add_node("my_subgraph", subgraph)
```

**详细文档**: `03_核心概念_1_子图定义与编译.md`

---

### 概念 2: 共享状态键集成

**核心要点**:
- 当父图和子图的 State Schema 共享相同的键名时，LangGraph 自动在父子图之间传递这些键的值
- 子图只能看到与自己 Schema 匹配的键，父图的其他键对子图不可见
- 这是最简单的集成方式，适用于父子图有相同语义字段的场景
- 子图更新的共享键值会自动回传到父图状态

**适用条件**: 父子图有共同的键名且语义一致

**详细文档**: `03_核心概念_2_共享状态键集成.md`

---

### 概念 3: 状态转换集成

**核心要点**:
- 当父图和子图的 State Schema 不同（键名不同或需要数据转换）时，使用包装函数手动转换
- 包装函数接收父图状态，转换为子图输入；接收子图输出，转换为父图状态更新
- 这种方式更灵活，适用于集成第三方子图或不同团队开发的子图
- 包装函数作为普通节点添加到父图

**关键模式**:
```python
def call_subgraph(state: ParentState) -> dict:
    # 1. 转换：父图状态 -> 子图输入
    subgraph_input = {"sub_key": state["parent_key"]}
    # 2. 调用子图
    result = subgraph.invoke(subgraph_input)
    # 3. 转换：子图输出 -> 父图状态更新
    return {"parent_key": result["sub_key"]}

parent_builder.add_node("call_subgraph", call_subgraph)
```

**详细文档**: `03_核心概念_3_状态转换集成.md`

---

### 概念 4: Command.PARENT 跨图通信

**核心要点**:
- `Command(goto="node_name", graph=Command.PARENT)` 允许子图节点向父图发送路由指令
- 子图可以决定父图接下来执行哪个节点，实现跨图的动态路由
- 这是多代理 Handoff 模式的核心机制：一个代理子图可以将控制权转交给另一个代理
- `Command` 可以同时携带状态更新（`update` 参数）和路由目标（`goto` 参数）

**典型场景**: 多代理系统中，Agent A 发现需要 Agent B 处理，通过 `Command.PARENT` 通知父图路由到 Agent B

**详细文档**: `03_核心概念_4_Command跨图通信.md`

---

### 概念 5: Send 动态分发

**核心要点**:
- `Send(node_name, input_data)` 允许在条件路由中动态创建多个子图/节点调用，每个调用接收不同的输入
- 这是 LangGraph 实现 Map-Reduce 模式的核心机制
- Map 阶段：通过返回多个 `Send` 对象，将不同的输入并行分发到同一个节点/子图
- Reduce 阶段：所有并行执行的结果通过 Reducer 函数（如 `operator.add`）合并到父图状态
- 适用场景：批量文档处理、多查询并行搜索、多角度分析等

**关键模式**:
```python
def distribute_tasks(state: State) -> list[Send]:
    return [Send("process", {"item": item}) for item in state["items"]]

builder.add_conditional_edges("start", distribute_tasks)
```

**详细文档**: `03_核心概念_5_Send动态分发.md`

---

### 概念 6: 子图 Checkpointer 与持久化

**核心要点**:
- 父图设置 Checkpointer 后，子图默认继承父图的 Checkpointer
- 子图的每个节点执行后，状态变更会作为父图 checkpoint 的一部分被保存
- 子图也可以拥有独立的 Checkpointer，用于维护独立的对话记忆或长期状态
- 嵌套子图的 checkpoint 结构是递归的，每层子图的 `thread_id` 自动派生
- 使用 `get_state()` 可以递归查看所有子图的状态快照

**注意事项**: 子图的独立 Checkpointer 与父图的 Checkpointer 是不同的概念，前者用于子图内部的跨调用持久化

**详细文档**: `03_核心概念_6_子图Checkpointer.md`

## 四、实战场景列表

本知识点包含 **4 个实战场景**，从简单到复杂逐步递进：

### 场景 1: 基础子图创建与调用

**目标**: 创建最简单的子图，理解父子图状态传递的基本机制
**内容**: 子图定义、compile、add_node 嵌入父图、共享键传递、invoke 执行
**难度**: ⭐⭐
**核心技能**: 子图创建的完整流程

**代码示例**:
```python
from langgraph.graph import START, END, StateGraph
from typing_extensions import TypedDict

class State(TypedDict):
    message: str

# 定义子图
def sub_node(state: State):
    return {"message": state["message"] + " -> SubNode"}

sub_builder = StateGraph(State)
sub_builder.add_node("sub_node", sub_node)
sub_builder.add_edge(START, "sub_node")
sub_builder.add_edge("sub_node", END)
subgraph = sub_builder.compile()

# 嵌入父图
def parent_node(state: State):
    return {"message": state["message"] + " -> ParentNode"}

parent_builder = StateGraph(State)
parent_builder.add_node("parent_node", parent_node)
parent_builder.add_node("my_subgraph", subgraph)
parent_builder.add_edge(START, "parent_node")
parent_builder.add_edge("parent_node", "my_subgraph")
parent_builder.add_edge("my_subgraph", END)
graph = parent_builder.compile()

result = graph.invoke({"message": "Start"})
# {'message': 'Start -> ParentNode -> SubNode'}
```

### 场景 2: 多代理 Handoff 系统

**目标**: 使用 Command.PARENT 实现多个代理子图之间的动态切换
**内容**: 多个代理子图定义、Command.PARENT 跨图路由、Supervisor 调度、状态传递
**难度**: ⭐⭐⭐⭐
**核心技能**: 跨图通信与多代理协作

```python
def agent_node(state: AgentState):

```

### 场景 3: Map-Reduce 并行子图

**目标**: 使用 Send 将多个任务并行分发到子图，收集并合并结果
**内容**: Send 创建、并行分发、Reducer 合并、结果聚合
**难度**: ⭐⭐⭐⭐
**核心技能**: 并行处理与结果聚合

**典型场景**: 将一篇长文档的多个段落并行发送给同一个摘要子图处理，最后合并所有摘要

**核心模式**:
```python
from langgraph.types import Send

def distribute(state: State) -> list[Send]:
    return [
        Send("summarize_subgraph", {"chunk": chunk})
        for chunk in state["chunks"]
    ]

builder.add_conditional_edges("split", distribute)
```

### 5.3 常见学习陷阱

**陷阱 1: 忘记编译子图**
- 问题：直接将 StateGraph（Builder）作为节点添加，而不是编译后的 CompiledStateGraph
- 解决：记住子图必须先 `compile()` 再通过 `add_node()` 添加

**陷阱 2: 状态键名不匹配**
- 问题：以为父子图会自动传递所有状态，实际上只传递同名的键
- 解决：明确父子图的共享键，或使用状态转换函数

**陷阱 3: 混淆 Command.PARENT 的作用域**
- 问题：在父图节点中使用 `graph=Command.PARENT`，导致错误
- 解决：`Command.PARENT` 只在子图节点中使用，表示向上一层父图发送命令

**陷阱 4: Send 缺少 Reducer**
- 问题：使用 Send 并行分发后，多个结果无法合并
- 解决：确保接收结果的状态字段配置了 Reducer（如 `Annotated[list, operator.add]`）

**陷阱 5: 子图 Checkpointer 配置混乱**
- 问题：不清楚子图是继承父图的 Checkpointer 还是使用独立的
- 解决：默认继承父图，只有显式传入独立 Checkpointer 时才独立

## 七、快速参考

### 7.1 核心 API 速查

```python
from langgraph.graph import START, END, StateGraph
from langgraph.types import Command, Send

# 1. 创建子图
sub_builder = StateGraph(SubState)
sub_builder.add_node("node", node_fn)
sub_builder.add_edge(START, "node")
sub_builder.add_edge("node", END)
subgraph = sub_builder.compile()

# 2. 嵌入父图（共享键方式）
parent_builder = StateGraph(ParentState)
parent_builder.add_node("subgraph", subgraph)

# 3. 嵌入父图（状态转换方式）
def wrapper(state: ParentState) -> dict:
    result = subgraph.invoke({"sub_key": state["parent_key"]})
    return {"parent_key": result["sub_key"]}
parent_builder.add_node("subgraph", wrapper)

# 4. Command.PARENT（子图内部使用）
def sub_node(state):
    return Command(goto="target", graph=Command.PARENT, update={...})

# 5. Send 动态分发
def distribute(state) -> list[Send]:
    return [Send("node", {"data": item}) for item in state["items"]]
builder.add_conditional_edges("source", distribute)

# 6. 流式监控嵌套子图
for ns, chunk in graph.stream(input, subgraphs=True):
    print(ns, chunk)
```

### 7.2 集成方式决策表

| 场景 | 推荐方式 | 理由 |
|------|----------|------|
| 父子图有相同的键名和语义 | 共享状态键 | 最简单，零额外代码 |
| 父子图 Schema 完全不同 | 状态转换函数 | 灵活，完全控制数据映射 |
| 集成第三方/团队他人的子图 | 状态转换函数 | 不需要修改子图代码 |
| 子图需要控制父图路由 | Command.PARENT | 跨图通信的唯一方式 |
| 需要并行处理多个输入 | Send 动态分发 | Map-Reduce 标准模式 |
