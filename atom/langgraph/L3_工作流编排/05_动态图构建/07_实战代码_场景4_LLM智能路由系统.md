# 实战场景4：LLM 智能路由系统 - 完整的动态决策流

> 构建一个「智能任务处理系统」：用户输入自然语言任务描述，LLM 分析任务类型并路由到对应处理器（代码生成、文档写作、数据分析、问答对话），低置信度任务走人工审核。

---

## 场景描述

你在构建一个 AI 任务调度中心。用户用自然语言描述需求，系统需要：

1. **路由节点**：LLM 分析任务类型，输出结构化路由决策（含置信度）
2. **4 个专用处理节点**：代码生成、文档写作、数据分析、问答对话
3. **人工审核节点**：置信度低于阈值时，转人工确认
4. **汇总节点**：收集处理结果，输出最终报告

### 双重类比

**前端类比：** 就像一个智能表单系统——用户在输入框里随便写需求，前端不是靠 `switch(type)` 硬编码分发，而是用一个 AI 分类器理解语义后，动态渲染对应的表单组件（代码编辑器、富文本编辑器、图表配置器、聊天界面）。

**日常生活类比：** 就像一个智能前台——客人走进来说"我想做个网站"，前台不是查关键词表，而是理解意图后说"您需要技术部门"，然后把客人引导过去。如果客人说了一句模棱两可的话，前台会说"我不太确定，让我请经理来帮您"。

### 工作流图示

```
                          ┌──→ [代码生成]   ──┐
                          │                    │
                          ├──→ [文档写作]   ──┤
[START] → [LLM路由] ────┤                    ├──→ [汇总] → [END]
                          ├──→ [数据分析]   ──┤
                          │                    │
                          ├──→ [问答对话]   ──┤
                          │                    │
                          └──→ [人工审核]   ──┘
                         (置信度 < 0.7)
```

## 完整可运行代码

```python
"""
实战场景4：LLM 智能路由系统 - 完整的动态决策流
演示：模拟 LLM 结构化输出 + 置信度阈值 + 条件边路由

运行要求：pip install langgraph pydantic
无需外部 API，完全本地运行（使用模拟 LLM）
"""
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph, START, END


# ============================================================
# 第一部分：定义路由 Schema（结构化输出的"契约"）
# ============================================================

class TaskRoute(BaseModel):
    """LLM 路由决策的结构化输出

    这是整个系统的核心数据结构。在真实项目中，
    LLM 通过 with_structured_output(TaskRoute) 返回这个对象。

    Literal 类型约束了 LLM 只能从预定义的类别中选择，
    避免了自由文本输出带来的不确定性。
    """
    task_type: Literal["code", "writing", "analysis", "chat"] = Field(
        description="任务类型分类"
    )
    confidence: float = Field(
        description="分类置信度，0.0 到 1.0 之间",
        ge=0.0,
        le=1.0,
    )
    reasoning: str = Field(
        description="分类理由，用于日志和调试"
    )
    complexity: Literal["low", "medium", "high"] = Field(
        description="任务复杂度评估"
    )


# ============================================================
# 第二部分：定义工作流状态
# ============================================================

class TaskState(TypedDict):
    """智能任务处理系统的状态

    Attributes:
        user_input: 用户的自然语言任务描述
        task_type: LLM 判断的任务类型
        confidence: LLM 的置信度
        complexity: 任务复杂度
        process_log: 处理日志（reducer 追加模式）
        output: 处理节点的输出结果
        needs_review: 是否需要人工审核
        final_report: 最终汇总报告
    """
    user_input: str
    task_type: str
    confidence: float
    complexity: str
    process_log: Annotated[list[str], operator.add]
    output: str
    needs_review: bool
    final_report: str


# ============================================================
# 第三部分：模拟 LLM 路由函数
# ============================================================

# 置信度阈值：低于此值走人工审核
CONFIDENCE_THRESHOLD = 0.7

def mock_llm_route(user_input: str) -> TaskRoute:
    """模拟 LLM 的结构化输出路由

    在真实项目中，替换为：
        from langchain_openai import ChatOpenAI
        llm = ChatOpenAI(model="gpt-4o-mini")
        router = llm.with_structured_output(TaskRoute)
        decision = router.invoke([
            SystemMessage(content="你是一个任务分类器..."),
            HumanMessage(content=user_input),
        ])

    这里用关键词匹配模拟 LLM 的语义理解能力。
    真实 LLM 能处理更复杂的语义，比如：
    - "帮我搞个脚本" → code（没有"代码"关键词但语义匹配）
    - "把这段话润色一下" → writing（理解"润色"的含义）
    """
    text = user_input.lower()

    # 代码生成类关键词
    code_keywords = ["代码", "函数", "脚本", "编程", "实现", "bug", "调试", "api"]
    # 文档写作类关键词
    writing_keywords = ["写", "文档", "报告", "文章", "总结", "邮件", "方案"]
    # 数据分析类关键词
    analysis_keywords = ["分析", "数据", "统计", "图表", "趋势", "对比", "指标"]
    # 问答对话类关键词
    chat_keywords = ["什么是", "为什么", "怎么理解", "解释", "区别"]

    # 计算每个类别的匹配分数
    scores = {
        "code": sum(1 for kw in code_keywords if kw in text),
        "writing": sum(1 for kw in writing_keywords if kw in text),
        "analysis": sum(1 for kw in analysis_keywords if kw in text),
        "chat": sum(1 for kw in chat_keywords if kw in text),
    }

    max_score = max(scores.values())
    total_keywords = sum(scores.values())

    # 没有匹配到任何关键词 → 低置信度
    if max_score == 0:
        return TaskRoute(
            task_type="chat",
            confidence=0.35,
            reasoning="未匹配到明确的任务类型关键词，默认归类为对话",
            complexity="low",
        )

    # 找到最高分的类别
    best_type = max(scores, key=scores.get)

    # 置信度计算：最高分占总分的比例 × 基础置信度
    if total_keywords > 0:
        dominance = max_score / total_keywords  # 最高类别的"统治力"
    else:
        dominance = 0.0
    confidence = round(min(0.5 + dominance * 0.45, 0.98), 2)

    # 复杂度评估：匹配关键词越多，任务越复杂
    if total_keywords >= 3:
        complexity = "high"
    elif total_keywords >= 2:
        complexity = "medium"
    else:
        complexity = "low"

    return TaskRoute(
        task_type=best_type,
        confidence=confidence,
        reasoning=f"匹配到 {best_type} 类关键词 {max_score} 个，置信度 {confidence}",
        complexity=complexity,
    )


# ============================================================
# 第四部分：定义所有节点
# ============================================================

def router_node(state: TaskState) -> dict:
    """路由节点：调用 LLM 分析任务类型

    这是整个工作流的"大脑"。它不做实际处理，
    只负责理解用户意图并做出路由决策。
    """
    decision = mock_llm_route(state["user_input"])

    print(f"[Router] 任务类型: {decision.task_type}")
    print(f"  置信度: {decision.confidence:.2f}")
    print(f"  复杂度: {decision.complexity}")
    print(f"  理由: {decision.reasoning}")

    # 置信度低于阈值 → 标记需要人工审核
    needs_review = decision.confidence < CONFIDENCE_THRESHOLD
    if needs_review:
        print(f"  ⚠ 置信度低于阈值 {CONFIDENCE_THRESHOLD}，转人工审核")

    return {
        "task_type": decision.task_type,
        "confidence": decision.confidence,
        "complexity": decision.complexity,
        "needs_review": needs_review,
        "process_log": [
            f"[路由] 类型={decision.task_type}, "
            f"置信度={decision.confidence:.2f}, "
            f"复杂度={decision.complexity}"
        ],
    }


def code_generator(state: TaskState) -> dict:
    """代码生成处理节点

    在真实系统中，这里会调用专门的代码生成 LLM，
    可能还会包含代码执行、测试、lint 等步骤。
    """
    user_input = state["user_input"]
    complexity = state["complexity"]
    print(f"[Code] 正在生成代码...")
    print(f"  任务: {user_input}")
    print(f"  复杂度: {complexity}")

    # 模拟不同复杂度的处理
    if complexity == "high":
        output = (
            f"[代码生成结果]\n"
            f"任务: {user_input}\n"
            f"生成了 3 个文件，包含完整的模块结构、单元测试和文档字符串\n"
            f"代码行数: ~150 行"
        )
    elif complexity == "medium":
        output = (
            f"[代码生成结果]\n"
            f"任务: {user_input}\n"
            f"生成了 1 个文件，包含核心函数和基础注释\n"
            f"代码行数: ~50 行"
        )
    else:
        output = (
            f"[代码生成结果]\n"
            f"任务: {user_input}\n"
            f"生成了一个代码片段\n"
            f"代码行数: ~15 行"
        )

    return {
        "output": output,
        "process_log": [f"[代码生成] 复杂度={complexity}, 已完成"],
    }


def doc_writer(state: TaskState) -> dict:
    """文档写作处理节点"""
    user_input = state["user_input"]
    print(f"[Writing] 正在撰写文档...")
    print(f"  任务: {user_input}")

    output = (
        f"[文档写作结果]\n"
        f"任务: {user_input}\n"
        f"生成了结构化文档，包含标题、正文、总结三部分\n"
        f"字数: ~800 字"
    )
    return {
        "output": output,
        "process_log": [f"[文档写作] 已完成"],
    }


def data_analyst(state: TaskState) -> dict:
    """数据分析处理节点"""
    user_input = state["user_input"]
    print(f"[Analysis] 正在分析数据...")
    print(f"  任务: {user_input}")

    output = (
        f"[数据分析结果]\n"
        f"任务: {user_input}\n"
        f"生成了分析报告，包含 3 个图表和关键指标摘要\n"
        f"发现 2 个显著趋势"
    )
    return {
        "output": output,
        "process_log": [f"[数据分析] 已完成"],
    }


def chat_responder(state: TaskState) -> dict:
    """问答对话处理节点"""
    user_input = state["user_input"]
    print(f"[Chat] 正在回答问题...")
    print(f"  问题: {user_input}")

    output = (
        f"[问答结果]\n"
        f"问题: {user_input}\n"
        f"提供了详细解答，包含定义、示例和延伸阅读链接"
    )
    return {
        "output": output,
        "process_log": [f"[问答对话] 已完成"],
    }


def human_review(state: TaskState) -> dict:
    """人工审核节点

    当 LLM 路由的置信度低于阈值时触发。
    在真实系统中，这里会：
    1. 发送通知给人工审核员
    2. 等待人工确认或修改路由决策
    3. 记录审核日志用于后续模型优化

    这里模拟人工审核直接放行的场景。
    """
    user_input = state["user_input"]
    task_type = state["task_type"]
    confidence = state["confidence"]

    print(f"[Human Review] 人工审核中...")
    print(f"  原始输入: {user_input}")
    print(f"  LLM 建议: {task_type} (置信度: {confidence:.2f})")
    print(f"  审核结果: 确认为 '{task_type}' 类型，放行处理")

    output = (
        f"[人工审核结果]\n"
        f"输入: {user_input}\n"
        f"LLM 建议类型: {task_type} (置信度: {confidence:.2f})\n"
        f"审核决定: 确认放行，按 '{task_type}' 类型处理\n"
        f"备注: 低置信度任务，已记录用于模型优化"
    )
    return {
        "output": output,
        "process_log": [
            f"[人工审核] LLM建议={task_type}, "
            f"置信度={confidence:.2f}, 审核结果=放行"
        ],
    }


def summarize(state: TaskState) -> dict:
    """汇总节点：收集处理结果，生成最终报告"""
    print(f"\n[Summary] 生成最终报告")
    print(f"  任务类型: {state['task_type']}")
    print(f"  置信度: {state['confidence']:.2f}")
    print(f"  经过人工审核: {'是' if state['needs_review'] else '否'}")
    print(f"  处理步骤: {len(state['process_log'])} 步")

    final_report = (
        f"{'=' * 50}\n"
        f"任务处理报告\n"
        f"{'=' * 50}\n"
        f"输入: {state['user_input']}\n"
        f"类型: {state['task_type']} | "
        f"置信度: {state['confidence']:.2f} | "
        f"复杂度: {state['complexity']}\n"
        f"人工审核: {'是' if state['needs_review'] else '否'}\n"
        f"{'─' * 50}\n"
        f"{state['output']}\n"
        f"{'─' * 50}\n"
        f"处理日志: {' → '.join(state['process_log'])}\n"
        f"{'=' * 50}"
    )
    return {
        "final_report": final_report,
        "process_log": ["[汇总] 报告生成完成"],
    }


# ============================================================
# 第五部分：路由函数与图构建
# ============================================================

def route_task(state: TaskState) -> str:
    """条件边的路由函数

    核心逻辑：
    1. 先检查置信度 → 低于阈值走人工审核
    2. 置信度足够 → 按任务类型分发到对应处理节点

    返回值是 path_map 的 key，不是节点名。
    """
    if state["needs_review"]:
        return "review"
    return state["task_type"]


def build_smart_router():
    """构建 LLM 智能路由工作流"""
    graph = StateGraph(TaskState)

    # --- 添加所有节点 ---
    graph.add_node("router", router_node)
    graph.add_node("code_generator", code_generator)
    graph.add_node("doc_writer", doc_writer)
    graph.add_node("data_analyst", data_analyst)
    graph.add_node("chat_responder", chat_responder)
    graph.add_node("human_review", human_review)
    graph.add_node("summarize", summarize)

    # --- 入口 → 路由 ---
    graph.add_edge(START, "router")

    # --- 路由 → 处理节点（条件边，核心！）---
    graph.add_conditional_edges(
        source="router",
        path=route_task,
        path_map={
            "code": "code_generator",       # 代码生成
            "writing": "doc_writer",         # 文档写作
            "analysis": "data_analyst",      # 数据分析
            "chat": "chat_responder",        # 问答对话
            "review": "human_review",        # 人工审核（低置信度）
        },
    )

    # --- 所有处理节点 → 汇总 ---
    graph.add_edge("code_generator", "summarize")
    graph.add_edge("doc_writer", "summarize")
    graph.add_edge("data_analyst", "summarize")
    graph.add_edge("chat_responder", "summarize")
    graph.add_edge("human_review", "summarize")

    # --- 汇总 → 结束 ---
    graph.add_edge("summarize", END)

    return graph.compile()


# ============================================================
# 第六部分：运行测试
# ============================================================

if __name__ == "__main__":
    app = build_smart_router()

    # 测试用例：覆盖所有分支 + 低置信度场景
    test_cases = [
        "帮我写一个 Python 函数，实现快速排序算法",       # → code（高置信度）
        "写一份项目总结报告，包含本月的工作成果",          # → writing（高置信度）
        "分析一下最近三个月的销售数据趋势",               # → analysis（高置信度）
        "什么是 RAG？为什么需要它？",                     # → chat（高置信度）
        "帮我处理一下这个事情",                           # → review（低置信度）
        "我需要一个脚本来统计数据并生成报告文档",          # → 多类别交叉
    ]

    for i, task in enumerate(test_cases, 1):
        print(f"\n{'#' * 60}")
        print(f"测试 {i}: {task}")
        print(f"{'#' * 60}")

        result = app.invoke({
            "user_input": task,
            "task_type": "",
            "confidence": 0.0,
            "complexity": "",
            "process_log": [],
            "output": "",
            "needs_review": False,
            "final_report": "",
        })

        print(f"\n{result['final_report']}")
```

## 代码逐行解析

### 1. 结构化输出的路由模式（核心设计）

```python
class TaskRoute(BaseModel):
    task_type: Literal["code", "writing", "analysis", "chat"]
    confidence: float = Field(ge=0.0, le=1.0)
    reasoning: str
    complexity: Literal["low", "medium", "high"]
```

这是整个系统最关键的设计决策。用 Pydantic BaseModel 定义路由 Schema 有三个好处：

1. `Literal` 类型约束了 LLM 只能从预定义选项中选择，消除了自由文本的不确定性
2. `Field(ge=0.0, le=1.0)` 对置信度做了范围校验，防止 LLM 输出异常值
3. `reasoning` 字段让路由决策可解释，方便调试和审计

**在真实项目中**，这个 Schema 会通过 `llm.with_structured_output(TaskRoute)` 绑定到 LLM：

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")
router = llm.with_structured_output(TaskRoute)

# LLM 返回的就是 TaskRoute 实例，不是自由文本
decision = router.invoke([
    SystemMessage(content="你是一个任务分类器。根据用户输入判断任务类型。"),
    HumanMessage(content="帮我写个排序算法"),
])
# decision.task_type == "code"
# decision.confidence == 0.92
```

[来源: Context7 官方文档 - LLM 驱动的路由模式]

### 2. 置信度阈值的设计

```python
CONFIDENCE_THRESHOLD = 0.7

def route_task(state: TaskState) -> str:
    if state["needs_review"]:
        return "review"          # 低置信度 → 人工审核
    return state["task_type"]    # 高置信度 → 直接分发
```

置信度阈值是一个**精度 vs 召回率的权衡**：

| 阈值 | 效果 | 适用场景 |
|------|------|---------|
| 0.9+ | 极少自动处理，大量人工审核 | 金融、医疗等高风险场景 |
| 0.7 | 平衡自动化和安全性 | 通用业务场景（推荐） |
| 0.5 | 大量自动处理，偶尔出错 | 低风险、容错性高的场景 |

路由函数 `route_task` 的逻辑很简单：先看 `needs_review` 标志（在 router_node 中根据置信度设置），低置信度走 "review"，否则按 `task_type` 分发。

**为什么不在路由函数里直接判断置信度？** 因为 `needs_review` 标志是在 router_node 中设置的，路由函数只需要读取状态。这遵循了 LangGraph 的设计原则：**节点负责计算和状态更新，路由函数只负责读取状态做决策**。

### 3. 从模拟切换到真实 LLM

切换只需要修改一个函数——`mock_llm_route`：

```python
# ===== 模拟版本（当前） =====
def mock_llm_route(user_input: str) -> TaskRoute:
    text = user_input.lower()
    # ... 关键词匹配逻辑 ...
    return TaskRoute(task_type=..., confidence=..., ...)


# ===== 真实 LLM 版本（替换） =====
from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage, HumanMessage

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
router_llm = llm.with_structured_output(TaskRoute)

SYSTEM_PROMPT = """你是一个智能任务分类器。根据用户的任务描述，判断任务类型。

分类规则：
- code: 涉及编程、代码编写、调试、脚本开发
- writing: 涉及文档撰写、报告、邮件、文案
- analysis: 涉及数据分析、统计、图表、趋势
- chat: 一般性问答、概念解释、闲聊

同时评估置信度（0-1）和复杂度（low/medium/high）。"""

def real_llm_route(user_input: str) -> TaskRoute:
    return router_llm.invoke([
        SystemMessage(content=SYSTEM_PROMPT),
        HumanMessage(content=user_input),
    ])
```

整个工作流的其他部分（节点、边、图结构）完全不需要改动。这就是**关注点分离**的好处——路由决策的实现细节被封装在一个函数里。

### 4. 条件边的连接方式

```python
graph.add_conditional_edges(
    source="router",
    path=route_task,
    path_map={
        "code": "code_generator",
        "writing": "doc_writer",
        "analysis": "data_analyst",
        "chat": "chat_responder",
        "review": "human_review",
    },
)
```

`path_map` 把路由函数的返回值（业务语义）映射到实际节点名（技术命名）。这样做的好处：

- 路由函数返回 `"code"` 而不是 `"code_generator"`，语义更清晰
- 如果要重命名节点，只需改 `path_map`，不需要改路由函数
- `path_map` 的 key 集合就是所有可能的路由路径，一目了然

**对比 Command 方式：** 如果用 Command API 实现同样的路由，代码会更紧凑但路由逻辑和状态更新耦合在一起：

```python
# Command 方式（路由 + 状态更新一体化）
def router_node(state: TaskState) -> Command[Literal[
    "code_generator", "doc_writer", "data_analyst",
    "chat_responder", "human_review"
]]:
    decision = mock_llm_route(state["user_input"])
    target = "human_review" if decision.confidence < 0.7 else {
        "code": "code_generator",
        "writing": "doc_writer",
        "analysis": "data_analyst",
        "chat": "chat_responder",
    }[decision.task_type]

    return Command(
        update={"task_type": decision.task_type, ...},
        goto=target,
    )
```

## 进阶扩展

### 扩展1：多级路由（先粗分类再细分类）

当任务类型很多时（比如 15+ 种），一次性让 LLM 从所有选项中选择效果会下降。更好的做法是分层路由：

```python
# 第一级：粗分类（3 个大类）
class CoarseRoute(BaseModel):
    category: Literal["development", "content", "data"]

# 第二级：细分类（每个大类下 3-5 个子类）
class DevRoute(BaseModel):
    sub_type: Literal["backend", "frontend", "devops", "debugging"]

class ContentRoute(BaseModel):
    sub_type: Literal["writing", "translation", "editing"]

# 工作流结构：
# 用户输入 → [粗分类] → development → [细分类] → backend/frontend/devops
#                      → content    → [细分类] → writing/translation/editing
#                      → data       → [细分类] → analysis/visualization/cleaning
```

**为什么分层更好？** 每一级 LLM 只需要从 3-5 个选项中选择，准确率远高于从 15 个选项中选。这和数据库索引的 B-Tree 思路一样——分层缩小搜索范围。

### 扩展2：路由结果缓存

相同或相似的输入不需要每次都调用 LLM：

```python
from functools import lru_cache
import hashlib

# 简单缓存：完全相同的输入直接返回缓存结果
route_cache: dict[str, TaskRoute] = {}

def cached_llm_route(user_input: str) -> TaskRoute:
    cache_key = hashlib.md5(user_input.encode()).hexdigest()

    if cache_key in route_cache:
        print(f"[Cache Hit] 命中缓存: {user_input[:30]}...")
        return route_cache[cache_key]

    decision = real_llm_route(user_input)  # 调用真实 LLM
    route_cache[cache_key] = decision
    return decision
```

**生产环境建议：** 用 Redis 做分布式缓存，设置 TTL（比如 1 小时），定期清理过期条目。

### 扩展3：路由日志与监控

记录每次路由决策，用于分析和优化：

```python
import json
from datetime import datetime

# 路由日志收集器
route_logs: list[dict] = []

def router_node_with_logging(state: TaskState) -> dict:
    decision = mock_llm_route(state["user_input"])

    # 记录路由日志
    log_entry = {
        "timestamp": datetime.now().isoformat(),
        "input": state["user_input"],
        "task_type": decision.task_type,
        "confidence": decision.confidence,
        "complexity": decision.complexity,
        "needs_review": decision.confidence < CONFIDENCE_THRESHOLD,
    }
    route_logs.append(log_entry)

    # 定期分析（每 100 条）
    if len(route_logs) % 100 == 0:
        avg_confidence = sum(l["confidence"] for l in route_logs) / len(route_logs)
        review_rate = sum(1 for l in route_logs if l["needs_review"]) / len(route_logs)
        print(f"[Monitor] 平均置信度: {avg_confidence:.2f}, 人工审核率: {review_rate:.1%}")

    return {
        "task_type": decision.task_type,
        "confidence": decision.confidence,
        # ...
    }
```

**监控指标建议：**
- 平均置信度：如果持续下降，说明用户输入模式在变化，需要更新 prompt
- 人工审核率：如果超过 30%，说明阈值设置过高或分类能力不足
- 各类别分布：如果某个类别占比异常，可能需要拆分或合并类别

## 关键收获

| 概念 | 本场景中的体现 |
|------|---------------|
| Pydantic Schema | `TaskRoute` 定义了 LLM 路由输出的结构化契约 |
| `Literal` 类型约束 | 限制 `task_type` 只能是 4 种预定义类型 |
| 置信度阈值 | `CONFIDENCE_THRESHOLD = 0.7`，低于此值走人工审核 |
| `add_conditional_edges` | 路由节点根据任务类型 + 置信度动态分发 |
| `path_map` 映射 | 业务语义（"code"）→ 技术节点名（"code_generator"） |
| 人工审核兜底 | 低置信度任务不直接处理，转人工确认 |
| 模拟 → 真实切换 | 只需替换 `mock_llm_route` 函数，其他代码不变 |
| `Annotated` reducer | `process_log` 使用 `operator.add` 追加日志 |
