# 核心概念 3：记忆压缩策略

> Summary / SummarizationMiddleware / RemoveMessage — 让对话永不超限

---

## 一句话定义

**记忆压缩策略是在保留关键信息的前提下减少对话历史的 Token 消耗，是解决「对话越长 → 上下文越大 → 成本越高 / 超出窗口」问题的核心技术。**

---

## 为什么需要记忆压缩？

### 问题：对话越长越贵，而且会「爆」

```
第 1 轮: 用户 100 tokens + AI 200 tokens = 300 tokens
第 2 轮: 历史 300 + 新 300 = 600 tokens
第 3 轮: 历史 600 + 新 300 = 900 tokens
   ...
第 50 轮: 历史 ~15000 tokens ← 超出 context window！

成本也在指数增长：
第 1 轮输入费: $0.001
第 50 轮输入费: $0.05 ← 50倍！
```

### 解决思路

```
方案 A: 直接删除旧消息 ← 简单粗暴，会丢信息
方案 B: 摘要旧消息     ← 保留关键信息，减少 Token
方案 C: 混合策略       ← 新消息保留原文 + 旧消息变摘要（推荐）
```

---

## 策略一：直接删除旧消息（RemoveMessage）

### 原理

最简单的压缩 —— 直接移除不需要的消息：

```
压缩前（10条消息，3000 tokens）:
  msg1 → msg2 → msg3 → msg4 → msg5 → msg6 → msg7 → msg8 → msg9 → msg10

压缩后（保留最近5条，1500 tokens）:
  删除 ←──────────────────→  msg6 → msg7 → msg8 → msg9 → msg10
```

### 现代方案：Middleware 自动删除

```python
from langchain.agents import create_agent, AgentState
from langchain.agents.middleware import after_model
from langchain.messages import RemoveMessage
from langgraph.checkpoint.memory import InMemorySaver

@after_model
def trim_messages(state: AgentState, runtime) -> dict | None:
    """每次模型回复后，检查并裁剪消息列表"""
    messages = state["messages"]
    max_messages = 20  # 保留最近 20 条

    if len(messages) > max_messages:
        # 标记要删除的旧消息
        to_remove = messages[:-max_messages]
        return {
            "messages": [RemoveMessage(id=m.id) for m in to_remove]
        }
    return None  # 不需要裁剪

agent = create_agent(
    model="gpt-4.1",
    tools=[],
    middleware=[trim_messages],
    checkpointer=InMemorySaver(),
)
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 实现简单，零成本 | 旧信息完全丢失 |
| 不需要额外 LLM 调用 | 用户提到很久前的事无法处理 |
| 延迟最低 | 可能丢失关键上下文 |

## 策略二：LLM 摘要压缩（Summary）

### 原理

用一个**低成本 LLM** 把旧消息压缩成一段摘要：

```
压缩前（20条消息，6000 tokens）:
  Human: 我叫张三
  AI: 你好张三！
  Human: 我在北京工作
  AI: 北京是个好地方！
  Human: 我是做前端开发的
  AI: 前端开发很有前途！
  ... 更多消息 ...

         ↓ LLM 摘要 ↓

压缩后（1条摘要 + 最近5条，2000 tokens）:
  System: [对话摘要] 用户名张三，在北京做前端开发，对AI技术感兴趣...
  Human: 最近的消息1
  AI: 最近的回复1
  ... 最近5条 ...
```

### 现代方案：SummarizationMiddleware（推荐）

```python
from langchain.agents import create_agent
from langchain.agents.middleware import SummarizationMiddleware
from langgraph.checkpoint.memory import InMemorySaver

agent = create_agent(
    model="gpt-4.1",
    tools=[],
    middleware=[
        SummarizationMiddleware(
            model="gpt-4.1-mini",       # 用便宜的模型做摘要
            trigger=("tokens", 4000),    # 当消息超过 4000 tokens 时触发
            keep=("messages", 20),       # 保留最近 20 条原始消息
        )
    ],
    checkpointer=InMemorySaver(),
)

# 使用方式不变
config = {"configurable": {"thread_id": "user-1"}}
for i in range(50):
    agent.invoke(
        {"messages": [{"role": "user", "content": f"问题 {i+1}"}]},
        config
    )
# 第 25 轮左右自动触发摘要，之后每隔一段触发
# 用户无感知，消息列表始终可控
```

[来源: atom/langchain/L4_Agent系统/07_Agent Memory集成/reference/context7_langchain_memory_01.md]

### SummarizationMiddleware 参数详解

```python
SummarizationMiddleware(
    model="gpt-4.1-mini",        # 摘要模型（推荐用便宜的）
    trigger=("tokens", 4000),     # 触发条件：
                                  #   ("tokens", N) - Token 超过 N 时
                                  #   ("messages", N) - 消息超过 N 条时
    keep=("messages", 20),        # 保留策略：
                                  #   ("messages", N) - 保留最近 N 条
                                  #   ("tokens", N) - 保留最近 N tokens
)
```

### 经典方案：ConversationSummaryMemory

```python
from langchain.memory import ConversationSummaryMemory
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4.1-mini")

memory = ConversationSummaryMemory(
    llm=llm,                      # 用于生成摘要的 LLM
    memory_key="chat_history",
    return_messages=True,
)

# 每次 save_context 都会更新摘要
memory.save_context(
    {"input": "我叫张三，在北京做前端开发"},
    {"output": "你好张三！前端开发很有前途。"}
)

# 加载的是摘要，不是原始消息
vars = memory.load_memory_variables({})
print(vars["chat_history"])
# → "用户名张三，北京前端开发者。"（摘要形式）
```

[来源: sourcecode/langchain/libs/langchain/langchain_classic/memory/summary.py]

### 优缺点

| 优点 | 缺点 |
|------|------|
| 保留关键信息 | 需要额外 LLM 调用（有成本） |
| Token 大幅减少 | 摘要可能丢失细节 |
| 支持超长对话 | 引入额外延迟 |

## 策略三：混合策略（Summary + Buffer）

### 原理

**最佳实践** —— 结合两种策略的优势：

```
┌──────────────────────────────────────────────┐
│                 记忆结构                       │
│                                              │
│  ┌──────────────────────────────┐            │
│  │  摘要区（旧消息的 LLM 摘要）    │ ← 低 Token │
│  │  "用户张三，北京前端开发者..."   │            │
│  └──────────────────────────────┘            │
│  ┌──────────────────────────────┐            │
│  │  缓冲区（最近 N 条原始消息）    │ ← 完整信息 │
│  │  msg1, msg2, msg3, ...       │            │
│  └──────────────────────────────┘            │
└──────────────────────────────────────────────┘

当缓冲区超过 Token 限制时：
  1. 将最旧的消息移出缓冲区
  2. 用 LLM 将这些消息合并到摘要中
  3. 缓冲区变小，摘要更新
```

### 现代方案：SummarizationMiddleware（就是混合策略）

SummarizationMiddleware 实际上就是混合策略：

```python
agent = create_agent(
    model="gpt-4.1",
    tools=[],
    middleware=[
        SummarizationMiddleware(
            model="gpt-4.1-mini",
            trigger=("tokens", 4000),   # 摘要触发点
            keep=("messages", 20),       # 缓冲区大小
            # 超过 4000 tokens 时：
            #   1. 保留最近 20 条（缓冲区）
            #   2. 其余用 LLM 摘要（摘要区）
        )
    ],
    checkpointer=InMemorySaver(),
)
```

### 经典方案：ConversationSummaryBufferMemory

```python
from langchain.memory import ConversationSummaryBufferMemory
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4.1-mini")

memory = ConversationSummaryBufferMemory(
    llm=llm,
    max_token_limit=2000,         # 缓冲区 Token 上限
    memory_key="chat_history",
    return_messages=True,
)

# 正常保存对话
for i in range(20):
    memory.save_context(
        {"input": f"问题{i+1}，这是一段较长的文本..."},
        {"output": f"回答{i+1}，这也是一段较长的回复..."}
    )

# 自动生成的记忆包含：摘要 + 最近的原始消息
vars = memory.load_memory_variables({})
# 旧消息被摘要，新消息保持原文
```

## 策略四：自定义删除逻辑

### 按消息类型删除

```python
from langchain.agents.middleware import after_model
from langchain.messages import RemoveMessage

@after_model
def smart_trim(state, runtime):
    """智能裁剪：保留 System 消息，删除旧的 Tool 调用消息"""
    messages = state["messages"]
    if len(messages) <= 30:
        return None

    to_remove = []
    for msg in messages[:-20]:  # 只处理旧消息
        # 保留 System 消息（含重要指令）
        if msg.type == "system":
            continue
        # 优先删除 Tool 相关消息（占 Token 最多）
        if msg.type in ("tool", "tool_use"):
            to_remove.append(RemoveMessage(id=msg.id))
        # 其次删除旧对话
        elif msg.type in ("human", "ai"):
            to_remove.append(RemoveMessage(id=msg.id))

    return {"messages": to_remove} if to_remove else None
```

### 按重要性评分删除

```python
@after_model
def importance_trim(state, runtime):
    """基于重要性评分裁剪消息"""
    messages = state["messages"]
    if len(messages) <= 30:
        return None

    to_remove = []
    for msg in messages[:-10]:
        # 短消息（打招呼等）优先删除
        if len(msg.content) < 20:
            to_remove.append(RemoveMessage(id=msg.id))
        # Tool 错误结果优先删除
        elif msg.type == "tool" and "error" in msg.content.lower():
            to_remove.append(RemoveMessage(id=msg.id))

    return {"messages": to_remove} if to_remove else None
```

## 四种压缩策略对比

| 维度 | 直接删除 | LLM 摘要 | 混合策略 | 自定义逻辑 |
|------|----------|----------|----------|------------|
| **信息保留** | ❌ 完全丢失 | ✅ 摘要保留 | ✅✅ 最优 | 看实现 |
| **额外成本** | 零 | 有（LLM费） | 有（LLM费） | 零~有 |
| **延迟** | 无 | 有（摘要耗时） | 有（摘要耗时） | 看实现 |
| **实现难度** | ⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **推荐场景** | 原型/不重要 | 信息密集对话 | **生产环境** | 特殊需求 |

```python
# 积极压缩，降低成本
@after_model
def aggressive_trim(state, runtime):
    messages = state["messages"]
    if len(messages) > 10:
        return {"messages": [RemoveMessage(id=m.id) for m in messages[:-10]]}
```

## 与 RAG 开发的关系

| RAG 场景 | 压缩需求 | 推荐策略 |
|----------|----------|----------|
| **多轮文档问答** | 保留检索结果和查询历史 | 混合策略（摘要旧问答，保留近期） |
| **长文档分析** | Tool 输出可能很长 | 自定义删除（压缩 Tool 输出） |
| **智能客服** | 高频短对话 | 直接删除（保留最近 10 条） |
| **研究助手** | 需要完整上下文 | 混合策略（高阈值触发） |

---

## 本节小结

> **记忆压缩 = 在 Token 成本和信息保留之间找平衡。**
> - 直接删除：简单零成本，但会丢信息
> - LLM 摘要：保留信息，但有额外成本和延迟
> - 混合策略：**生产首选**，近期完整 + 远期摘要
> - 自定义：特殊需求时按消息类型/重要性裁剪
>
> **新项目推荐 `SummarizationMiddleware`，开箱即用的混合策略。**
