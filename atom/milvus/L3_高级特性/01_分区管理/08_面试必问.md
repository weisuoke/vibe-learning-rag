## 问题 1："Milvus 的 Partition 和 Partition Key 有什么区别？"

### 普通回答（不出彩）

"Partition 是手动创建的分区，Partition Key 是自动分区。"

### 出彩回答（推荐）

> **Partition 和 Partition Key 是 Milvus 提供的两种分区机制，它们在管理方式、可扩展性和数据隔离上有本质区别：**
>
> **1. 管理方式**：
> - **手动分区（Partition）**：需要手动创建分区，手动指定数据插入到哪个分区，最多支持 1,024 个分区
> - **Partition Key**：在 Schema 中标记字段为 `is_partition_key=True`，Milvus 自动根据该字段值将数据路由到 16 个物理分区（默认），支持数百万个逻辑租户
>
> **2. 数据隔离**：
> - **手动分区**：物理隔离，每个分区独立存储，可以独立加载和释放
> - **Partition Key**：逻辑隔离，多个租户可能共享一个物理分区，但数据在逻辑上是分离的
>
> **3. 适用场景**：
> - **手动分区**：适用于中等数量租户（< 1,024 个），需要物理隔离和热冷数据管理的场景
> - **Partition Key**：适用于大规模多租户（百万级），所有租户共享相同 Schema 的场景
>
> **4. 性能优化**：
> - 两者都能避免全表扫描，提升检索性能
> - Partition Key 可以启用隔离特性（`partitionkey.isolation=True`），进一步提升性能（仅支持 HNSW 索引）
>
> **在实际工作中的应用**：
> - 我们的 SaaS 应用有 10 万个租户，使用 Partition Key 实现多租户隔离，配置了 100 个物理分区，搜索延迟从 1 秒降至 10 毫秒以内
> - 对于时间序列数据，我们使用手动分区按月份分区，只加载最近 3 个月的热数据，节省了 75% 的内存

### 为什么这个回答出彩？

1. 从多个维度对比（管理方式、数据隔离、适用场景、性能优化）
2. 提供具体的数据和限制（1,024 个分区、16 个物理分区、百万级租户）
3. 结合实际工作场景，展示实践经验
4. 量化性能提升（延迟从 1 秒降至 10 毫秒、节省 75% 内存）

## 问题 2："如何选择合适的多租户策略？"

### 普通回答（不出彩）

"根据租户数量选择，租户少用 Collection-level，租户多用 Partition Key-level。"

### 出彩回答（推荐）

> **多租户策略的选择需要综合考虑四个维度：租户数量、数据隔离需求、Schema 灵活性和 RBAC 需求：**
>
> **1. 四种多租户策略对比**：
> - **Database-level**：最强隔离，支持少量租户（< 10），支持 RBAC
> - **Collection-level**：强隔离，支持中等数量租户（< 100），支持独立 Schema 和 RBAC
> - **Partition-level**：中等隔离，支持最多 1,024 个租户，共享 Schema，支持热冷数据管理
> - **Partition Key-level**：弱隔离，支持百万级租户，共享 Schema，自动管理
>
> **2. 选择决策树**：
> - 租户数量 < 10 且需要 RBAC → Database-level
> - 租户数量 < 100 且需要独立 Schema → Collection-level
> - 租户数量 < 1,024 且需要热冷数据管理 → Partition-level
> - 租户数量 > 1,024 或需要自动管理 → Partition Key-level
>
> **3. 性能考虑**：
> - Partition Key-level 通过自动路由避免 99% 无效扫描，延迟从秒级降至毫秒级
> - 启用 `partitionkey.isolation=True` 可进一步提升性能（仅支持 HNSW 索引）
> - 配置合理的物理分区数（推荐 100-200 个）
>
> **在实际工作中的应用**：
> - 我们的 SaaS 应用预计有 10 万个租户，选择了 Partition Key-level 策略
> - 配置了 100 个物理分区，启用了 Partition Key 隔离
> - 使用 HNSW 索引，搜索延迟稳定在 10 毫秒以内
> - 通过监控发现，99% 的查询只扫描了 1% 的数据，性能提升显著

1. 提供完整的决策框架（四个维度、决策树）
2. 对比四种策略的优缺点和适用场景

## 问题 3："Partition Key 是如何实现自动路由的？"

### 普通回答（不出彩）

"Partition Key 使用哈希算法将数据分配到不同分区。"

### 出彩回答（推荐）

> **Partition Key 的自动路由机制包含三个核心步骤：字段标记、哈希分布和自动路由：**
>
> **1. 字段标记**：
> - 在 Schema 中标记字段为 `is_partition_key=True`
> - 支持的数据类型：VARCHAR（推荐）、INT64
> - 示例：`FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, is_partition_key=True)`
>
> **2. 哈希分布**：
> - Milvus 使用哈希算法将 Partition Key 值映射到物理分区
> - 默认 16 个物理分区，可配置为 100-200 个（推荐）
> - 哈希算法确保数据均匀分布，避免数据倾斜
> - 伪代码：`partition_id = hash(partition_key_value) % num_partitions`
>
> **3. 自动路由**：
> - **插入时**：Milvus 根据 Partition Key 值自动将数据路由到相应分区
> - **搜索时**：通过 `expr` 指定 Partition Key 值，Milvus 自动只搜索相关分区
> - **性能优化**：避免 99% 无效扫描，延迟从秒级降至毫秒级
>
> **4. Partition Key 隔离特性**：
> - 启用 `partitionkey.isolation=True` 后，Milvus 为每个 Partition Key 值创建独立的 HNSW 图
> - 搜索时只遍历特定 Partition Key 对应的 HNSW 图，进一步提升性能
> - 限制：仅支持 HNSW 索引，搜索时必须包含单个 Partition Key 值
>
> **在实际工作中的应用**：
> - 我们的多租户系统有 10 万个租户，使用 `tenant_id` 作为 Partition Key
> - 配置了 100 个物理分区，每个物理分区大约包含 1000 个租户的数据
> - 搜索租户 A 的数据时，Milvus 只扫描 1/100 的数据，性能提升 100 倍
