# 核心概念 1：函数签名约定

> 本文档详细讲解 LangGraph 节点函数的签名约定，包括参数类型、返回值类型和类型推断机制

---

## 概述

节点函数签名约定是 LangGraph 框架的核心设计之一。它定义了节点函数如何接收输入、如何返回输出，以及框架如何自动推断类型信息。

**为什么需要签名约定？**

1. **类型安全**：通过类型注解，框架可以在运行前检查类型错误
2. **自动推断**：框架自动推断输入输出类型，减少样板代码
3. **灵活配置**：支持多种参数组合，满足不同场景需求
4. **统一接口**：所有节点遵循相同的签名约定，易于理解和维护

[来源: reference/source_节点函数约定_01.md]

## 一、参数类型详解

### 1.1 State 参数（必需）

**定义**：State 是节点函数的第一个参数，表示当前图的状态。

```python
from typing import TypedDict

class State(TypedDict):
    messages: list[str]
    count: int

def my_node(state: State) -> dict:
    """最基础的节点函数签名"""
    return {"count": state["count"] + 1}
```

**关键点**：
- State 必须是第一个参数
- 通常使用 TypedDict 定义状态结构
- 框架会自动从第一个参数推断 input_schema

[来源: reference/source_节点函数约定_01.md]

**前端类比**：
```javascript
// React 组件的 props
function MyComponent(props) {
  return <div>{props.count + 1}</div>
}
```

**日常生活类比**：
工作站的输入材料 - 工人从传送带上拿到原材料（State），然后加工处理。

---

### 1.2 Config 参数（可选）

**定义**：Config 参数提供运行时配置信息，如 checkpoint、metadata 等。

```python
from langchain_core.runnables import RunnableConfig

def my_node(state: State, config: RunnableConfig) -> dict:
    """带 Config 参数的节点函数"""
    # 访问配置信息
    run_id = config.get("run_id")
    metadata = config.get("metadata", {})

    print(f"Running node with ID: {run_id}")
    return {"count": state["count"] + 1}
```

**Config 包含的信息**：
- `run_id`: 当前运行的唯一标识
- `metadata`: 元数据字典
- `configurable`: 可配置参数
- `callbacks`: 回调处理器
- `recursion_limit`: 递归限制

[来源: reference/source_节点函数约定_01.md]

**使用场景**：
- 需要访问运行时信息
- 需要使用回调函数
- 需要访问 checkpoint 信息
- 需要传递自定义配置

**前端类比**：
```javascript
// React 的 context
function MyComponent(props, context) {
  const theme = context.theme
  return <div style={{color: theme.color}}>{props.count}</div>
}
```

---

### 1.3 Runtime 参数（可选）

**定义**：Runtime 参数提供运行时上下文，支持泛型类型。

```python
from langgraph.types import Runtime
from typing import TypedDict

class Context(TypedDict):
    user_id: str
    session_id: str

def my_node(state: State, runtime: Runtime[Context]) -> dict:
    """带 Runtime 参数的节点函数"""
    # 访问运行时上下文
    user_id = runtime.context.get("user_id")
    session_id = runtime.context.get("session_id")

    print(f"Processing for user: {user_id}")
    return {"count": state["count"] + 1}
```

**Runtime 的优势**：
- 类型安全的上下文访问
- 支持泛型类型定义
- 与 Config 分离，职责更清晰

[来源: reference/source_节点函数约定_01.md]

**使用场景**：
- 需要访问用户信息
- 需要访问会话信息
- 需要访问自定义上下文

## 二、返回值类型详解

### 2.1 dict 返回值（部分更新）

**定义**：返回字典表示部分状态更新，只需返回需要更新的字段。

```python
def my_node(state: State) -> dict:
    """返回部分状态更新"""
    # 只返回需要更新的字段
    return {"count": state["count"] + 1}
```

**关键点**：
- 不需要返回完整状态
- 返回的字段会与现有状态合并
- 使用 Reducer 函数处理合并逻辑

[来源: reference/context7_langgraph_01.md]

**示例：多字段更新**

```python
def retrieve(state: State) -> dict:
    """检索文档并更新多个字段"""
    question = state["question"]
    documents = retriever.invoke(question)

    return {
        "documents": documents,
        "question": question,
        "retrieved_at": datetime.now()
    }
```

[来源: reference/context7_langgraph_01.md]

---

### 2.2 Command 返回值（控制流程）

**定义**：Command 对象用于控制图的执行流程，支持跳转、更新等操作。

```python
from langgraph.types import Command
from typing import Literal

def my_node(state: State) -> Command[Literal["next", "end"]]:
    """使用 Command 控制流程"""
    if state["count"] > 10:
        return Command(goto="end")

    return Command(
        goto="next",
        update={"count": state["count"] + 1}
    )
```

**Command 的参数**：
- `goto`: 跳转到指定节点
- `update`: 更新状态（可选）
- `graph`: 目标图（用于子图通信）

[来源: reference/source_节点函数约定_01.md]

**使用场景**：
- 动态路由决策
- 条件跳转
- 父子图通信

**前端类比**：
```javascript
// React Router 的编程式导航
function MyComponent() {
  const navigate = useNavigate()

  if (count > 10) {
    navigate('/end')
  } else {
    navigate('/next')
  }
}
```

---

### 2.3 None 返回值（不更新状态）

**定义**：返回 None 表示不更新状态，常用于副作用操作。

```python
def log_node(state: State) -> None:
    """只执行副作用，不更新状态"""
    print(f"Current count: {state['count']}")
    # 不返回任何值
```

**使用场景**：
- 日志记录
- 发送通知
- 触发外部 API
- 数据验证（不修改状态）

[来源: reference/source_节点函数约定_01.md]

---

### 2.4 list 返回值（多个更新）

**定义**：返回列表表示多个更新操作，可以混合 dict 和 Command。

```python
def my_node(state: State) -> list[dict | Command]:
    """返回多个更新"""
    updates = []

    # 添加状态更新
    updates.append({"count": state["count"] + 1})

    # 添加另一个更新
    updates.append({"timestamp": datetime.now()})

    # 可选：添加 Command
    if state["count"] > 10:
        updates.append(Command(goto="end"))

    return updates
```

**关键点**：
- 列表中的更新按顺序应用
- 可以混合 dict 和 Command
- Command 会影响后续流程

[来源: reference/source_节点函数约定_01.md]

## 三、类型推断机制

### 3.1 自动类型推断

LangGraph 会自动从节点函数的类型注解推断输入输出类型。

```python
from typing import get_type_hints

# 框架内部的类型推断逻辑
if isfunction(action) or ismethod(action):
    hints = get_type_hints(action)

    # 推断输入类型（第一个参数）
    first_parameter_name = next(iter(
        inspect.signature(action).parameters.keys()
    ))

    if input_hint := hints.get(first_parameter_name):
        if isinstance(input_hint, type) and get_type_hints(input_hint):
            inferred_input_schema = input_hint
```

[来源: reference/source_节点函数约定_01.md]

**推断规则**：
1. 第一个参数的类型作为 input_schema
2. 返回值类型用于验证输出
3. 支持 TypedDict、dataclass 等类型

---

### 3.2 手动指定类型

可以在 `add_node` 时手动指定 input_schema。

```python
from typing import TypedDict

class NodeInput(TypedDict):
    x: int
    y: int

def my_node(state) -> dict:
    """没有类型注解的节点函数"""
    return {"result": state["x"] + state["y"]}

# 手动指定 input_schema
builder.add_node(
    "my_node",
    my_node,
    input_schema=NodeInput
)
```

**使用场景**：
- 节点函数没有类型注解
- 需要覆盖自动推断的类型
- 使用动态类型

[来源: reference/source_节点函数约定_01.md]

## 四、完整代码示例

### 4.1 基础节点函数

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    messages: list[str]
    count: int

def process_node(state: State) -> dict:
    """基础节点函数：处理消息"""
    messages = state["messages"]
    count = state["count"]

    # 处理逻辑
    new_message = f"Processed {count} messages"

    return {
        "messages": messages + [new_message],
        "count": count + 1
    }

# 构建图
builder = StateGraph(State)
builder.add_node("process", process_node)
builder.add_edge(START, "process")
builder.add_edge("process", END)

graph = builder.compile()

# 运行
result = graph.invoke({
    "messages": ["Hello"],
    "count": 0
})
print(result)
# {'messages': ['Hello', 'Processed 0 messages'], 'count': 1}
```

[来源: reference/context7_langgraph_01.md]

---

### 4.2 异步节点函数

```python
import asyncio
from typing import TypedDict

class State(TypedDict):
    query: str
    result: str

async def async_api_call(state: State) -> dict:
    """异步节点函数：调用外部 API"""
    query = state["query"]

    # 模拟异步 API 调用
    await asyncio.sleep(1)
    result = f"API result for: {query}"

    return {"result": result}

# 构建图
builder = StateGraph(State)
builder.add_node("api_call", async_api_call)
builder.add_edge(START, "api_call")
builder.add_edge("api_call", END)

graph = builder.compile()

# 异步运行
result = await graph.ainvoke({"query": "test", "result": ""})
print(result)
# {'query': 'test', 'result': 'API result for: test'}
```

**关键点**：
- 使用 `async def` 定义异步函数
- 框架自动检测并处理异步节点
- 使用 `ainvoke` 异步调用图

[来源: reference/source_节点函数约定_01.md]

### 4.3 Command 控制流程

```python
from langgraph.types import Command
from typing import Literal, TypedDict

class State(TypedDict):
    count: int
    status: str

def decision_node(state: State) -> Command[Literal["continue", "end"]]:
    """使用 Command 进行条件路由"""
    count = state["count"]

    if count >= 10:
        return Command(
            goto="end",
            update={"status": "completed"}
        )

    return Command(
        goto="continue",
        update={
            "count": count + 1,
            "status": "processing"
        }
    )

def continue_node(state: State) -> dict:
    """继续处理"""
    return {"count": state["count"] + 1}

# 构建图
builder = StateGraph(State)
builder.add_node("decision", decision_node)
builder.add_node("continue", continue_node)
builder.add_edge(START, "decision")
builder.add_edge("continue", "decision")

graph = builder.compile()

# 运行
result = graph.invoke({"count": 0, "status": "init"})
print(result)
# {'count': 10, 'status': 'completed'}
```

[来源: reference/context7_langgraph_01.md]

### 4.4 带 Config 的节点函数

```python
from langchain_core.runnables import RunnableConfig
from typing import TypedDict

class State(TypedDict):
    data: str

def node_with_config(state: State, config: RunnableConfig) -> dict:
    """使用 Config 参数访问运行时信息"""
    # 访问 run_id
    run_id = config.get("run_id", "unknown")

    # 访问自定义配置
    custom_param = config.get("configurable", {}).get("custom_param", "default")

    print(f"Run ID: {run_id}")
    print(f"Custom param: {custom_param}")

    return {"data": f"Processed with {custom_param}"}

# 构建图
builder = StateGraph(State)
builder.add_node("process", node_with_config)
builder.add_edge(START, "process")
builder.add_edge("process", END)

graph = builder.compile()

# 运行时传递配置
result = graph.invoke(
    {"data": "input"},
    config={
        "configurable": {"custom_param": "special_mode"}
    }
)
print(result)
# {'data': 'Processed with special_mode'}
```

## 五、最佳实践

### 5.1 使用类型注解

```python
# ✅ 推荐：使用类型注解
def my_node(state: State) -> dict:
    return {"count": state["count"] + 1}

# ❌ 不推荐：没有类型注解
def my_node(state):
    return {"count": state["count"] + 1}
```

**原因**：
- 类型注解帮助框架自动推断类型
- 提供更好的 IDE 支持
- 减少运行时错误

---

### 5.2 返回部分状态更新

```python
# ✅ 推荐：只返回需要更新的字段
def my_node(state: State) -> dict:
    return {"count": state["count"] + 1}

# ❌ 不推荐：返回完整状态
def my_node(state: State) -> dict:
    return {
        "messages": state["messages"],
        "count": state["count"] + 1,
        "other_field": state["other_field"]
    }
```

**原因**：
- 减少代码冗余
- 避免意外覆盖其他字段
- 更清晰地表达意图

---

### 5.3 使用 Command 进行动态路由

```python
# ✅ 推荐：使用 Command 进行动态路由
def my_node(state: State) -> Command[Literal["next", "end"]]:
    if state["count"] > 10:
        return Command(goto="end")
    return Command(goto="next", update={"count": state["count"] + 1})

# ❌ 不推荐：在条件边中硬编码逻辑
def route_function(state: State) -> str:
    if state["count"] > 10:
        return "end"
    return "next"
```

**原因**：
- Command 可以同时更新状态和控制流程
- 更灵活的路由控制
- 支持父子图通信

---

### 5.4 避免使用异步生成器

```python
# ✅ 推荐：使用普通异步函数
async def my_node(state: State) -> dict:
    result = await some_async_api()
    return {"result": result}

# ❌ 不推荐：使用异步生成器（可能有问题）
async def my_node(state: State):
    async for item in some_async_generator():
        yield {"item": item}
```

**原因**：
- 异步生成器节点可能不执行
- 错误处理机制失效
- 社区已知问题

[来源: reference/search_节点函数约定_01.md]

## 六、常见误区

### 6.1 误区：必须返回完整状态

```python
# ❌ 错误理解
def my_node(state: State) -> State:
    # 以为必须返回完整状态
    return {
        "messages": state["messages"],
        "count": state["count"] + 1,
        "all_other_fields": state["all_other_fields"]
    }

# ✅ 正确做法
def my_node(state: State) -> dict:
    # 只返回需要更新的字段
    return {"count": state["count"] + 1}
```

---

### 6.2 误区：Config 和 Runtime 是必需的

```python
# ❌ 错误理解：以为必须使用 Config
def my_node(state: State, config: RunnableConfig) -> dict:
    # 即使不使用 config 也添加参数
    return {"count": state["count"] + 1}

# ✅ 正确做法：只在需要时添加
def my_node(state: State) -> dict:
    # 不需要 config 就不添加
    return {"count": state["count"] + 1}
```

---

### 6.3 误区：Command 只能用于条件边

```python
# ❌ 错误理解：以为 Command 只能在条件边使用
def route_function(state: State) -> str:
    if state["count"] > 10:
        return "end"
    return "next"

# ✅ 正确做法：在普通节点中使用 Command
def my_node(state: State) -> Command[Literal["next", "end"]]:
    if state["count"] > 10:
        return Command(goto="end", update={"status": "done"})
    return Command(goto="next", update={"count": state["count"] + 1})
```

## 七、总结

LangGraph 节点函数签名约定的核心要点：

1. **参数类型**：
   - State（必需）：当前图状态
   - Config（可选）：运行时配置
   - Runtime（可选）：运行时上下文

2. **返回值类型**：
   - dict：部分状态更新
   - Command：控制流程跳转
   - None：不更新状态
   - list：多个更新操作

3. **类型推断**：
   - 自动从第一个参数推断 input_schema
   - 支持手动指定类型
   - 使用类型注解提高类型安全

4. **最佳实践**：
   - 使用类型注解
   - 返回部分状态更新
   - 使用 Command 进行动态路由
   - 避免异步生成器

5. **常见误区**：
   - 不需要返回完整状态
   - Config 和 Runtime 是可选的
   - Command 可以在普通节点中使用
