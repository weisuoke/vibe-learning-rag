# OpenClaw 安装与配置 - 反直觉点

**核心理念：** 识别并纠正常见误区，避免走弯路。

本文档揭示 OpenClaw 安装与配置中的 3 个反直觉点，帮助你避免常见错误。

---

## 反直觉点 1：全局安装不会污染项目依赖

### ❌ 错误认知

**"全局安装 OpenClaw 会污染我的项目依赖，影响项目的 package.json"**

很多开发者认为：
- 全局安装会修改项目的 `package.json`
- 全局安装会影响项目的 `node_modules`
- 全局安装会导致版本冲突

### 为什么会有这个误区？

**原因 1：混淆了全局安装和本地安装**
```bash
# 本地安装（项目依赖）
npm install openclaw
# 会修改 package.json 和 node_modules

# 全局安装（系统工具）
npm install -g openclaw
# 不会修改项目文件
```

**原因 2：对 npm 全局安装机制不了解**
- 全局安装的包存储在系统级目录（如 `/usr/local/lib/node_modules/`）
- 全局安装只创建可执行文件的符号链接到 PATH 目录
- 全局安装与项目依赖完全隔离

**原因 3：过去的不良经验**
- 某些工具（如旧版 `create-react-app`）推荐全局安装，但后来改为 `npx` 方式
- 导致开发者对全局安装产生负面印象

### ✅ 正确理解

**全局安装和本地安装是完全独立的两个系统：**

```typescript
// ===== 全局安装的文件位置 =====
// macOS/Linux:
// /usr/local/lib/node_modules/openclaw/  (包文件)
// /usr/local/bin/openclaw                (可执行文件符号链接)

// Windows:
// C:\Users\<user>\AppData\Roaming\npm\node_modules\openclaw\
// C:\Users\<user>\AppData\Roaming\npm\openclaw.cmd

// ===== 项目本地安装的文件位置 =====
// ./node_modules/openclaw/  (包文件)
// ./node_modules/.bin/openclaw  (可执行文件符号链接)

// ===== 两者互不影响 =====
// 全局安装：用于 CLI 命令
// 本地安装：用于项目代码 import
```

**验证代码：**

```typescript
import { execSync } from 'child_process';
import * as path from 'path';

/**
 * 验证全局安装不影响项目依赖
 */
function verifyGlobalInstallIsolation(): void {
  console.log('=== 验证全局安装隔离性 ===\n');

  // 1. 检查全局安装位置
  try {
    const globalPath = execSync('npm root -g', { encoding: 'utf-8' }).trim();
    console.log(`✅ 全局包目录: ${globalPath}`);
    console.log(`   示例: ${path.join(globalPath, 'openclaw')}`);
  } catch (error) {
    console.error('❌ 无法获取全局包目录');
  }

  // 2. 检查项目本地安装位置
  try {
    const localPath = execSync('npm root', { encoding: 'utf-8' }).trim();
    console.log(`\n✅ 项目包目录: ${localPath}`);
    console.log(`   示例: ${path.join(localPath, 'openclaw')}`);
  } catch (error) {
    console.error('❌ 无法获取项目包目录');
  }

  // 3. 检查 package.json
  const packageJsonPath = path.join(process.cwd(), 'package.json');
  try {
    const fs = require('fs');
    if (fs.existsSync(packageJsonPath)) {
      const packageJson = JSON.parse(fs.readFileSync(packageJsonPath, 'utf-8'));
      const hasOpenClaw =
        (packageJson.dependencies && packageJson.dependencies.openclaw) ||
        (packageJson.devDependencies && packageJson.devDependencies.openclaw);

      if (hasOpenClaw) {
        console.log('\n⚠️  项目 package.json 中包含 openclaw 依赖');
        console.log('   这是本地安装，与全局安装无关');
      } else {
        console.log('\n✅ 项目 package.json 中没有 openclaw 依赖');
        console.log('   全局安装不会修改 package.json');
      }
    }
  } catch (error) {
    console.log('\n⚠️  当前目录没有 package.json');
  }

  // 4. 总结
  console.log('\n=== 结论 ===');
  console.log('✅ 全局安装和本地安装完全隔离');
  console.log('✅ 全局安装不会修改项目文件');
  console.log('✅ 可以同时存在全局和本地安装');
}

verifyGlobalInstallIsolation();
```

**预期输出：**
```
=== 验证全局安装隔离性 ===

✅ 全局包目录: /usr/local/lib/node_modules
   示例: /usr/local/lib/node_modules/openclaw

✅ 项目包目录: /Users/username/project/node_modules
   示例: /Users/username/project/node_modules/openclaw

✅ 项目 package.json 中没有 openclaw 依赖
   全局安装不会修改 package.json

=== 结论 ===
✅ 全局安装和本地安装完全隔离
✅ 全局安装不会修改项目文件
✅ 可以同时存在全局和本地安装
```

## 反直觉点 2：配置文件不应该提交到 Git

### ❌ 错误认知

**"~/.openclaw/settings.json 应该提交到 Git，方便团队共享配置"**

很多开发者认为：
- 配置文件应该版本控制
- 团队成员应该使用相同的配置
- 提交配置文件方便新成员快速上手

### 为什么会有这个误区？

**原因 1：混淆了配置模板和实际配置**
```bash
# ❌ 错误：提交实际配置（包含敏感数据）
git add ~/.openclaw/settings.json

# ✅ 正确：提交配置模板
git add .openclaw.example.json
```

**原因 2：不了解配置文件的敏感性**
- `settings.json` 包含 API keys（敏感数据）
- API keys 泄露会导致安全风险和费用损失
- 配置文件应该在 `.gitignore` 中

**原因 3：对用户级配置和项目级配置的混淆**
- `~/.openclaw/settings.json` 是用户级配置（个人的）
- 项目级配置（如 `.openclawrc`）可以提交（不包含敏感数据）

### ✅ 正确理解

**配置文件分为三类：**

```typescript
// ===== 1. 用户级配置（不提交） =====
// ~/.openclaw/settings.json
{
  "gateway": { "port": 3000 },
  "channels": {
    "claude": {
      "apiKey": "sk-ant-api03-...",  // 敏感数据！
      "enabled": true
    }
  }
}

// ===== 2. 配置模板（提交） =====
// .openclaw.example.json
{
  "gateway": { "port": 3000 },
  "channels": {
    "claude": {
      "apiKey": "YOUR_CLAUDE_API_KEY_HERE",  // 占位符
      "enabled": true
    }
  }
}

// ===== 3. 项目级配置（提交） =====
// .openclawrc
{
  "gateway": { "port": 3000 },
  "defaultModel": "claude-3-5-sonnet-20241022"
  // 不包含 API keys
}
```

**正确的 Git 配置：**

```bash
# .gitignore
# OpenClaw 配置文件
.openclaw/
settings.json
*.local.json

# 环境变量文件
.env
.env.local
.env.*.local

# 日志文件
*.log
gateway.log
```

**验证代码：**

```typescript
import * as fs from 'fs';
import * as path from 'path';

/**
 * 检查配置文件是否被 Git 跟踪
 */
function checkConfigInGit(): void {
  console.log('=== 检查配置文件 Git 状态 ===\n');

  const sensitiveFiles = [
    path.join(process.env.HOME!, '.openclaw', 'settings.json'),
    '.env',
    '.env.local'
  ];

  for (const file of sensitiveFiles) {
    if (fs.existsSync(file)) {
      try {
        const { execSync } = require('child_process');
        const gitStatus = execSync(`git ls-files ${file}`, {
          encoding: 'utf-8',
          stdio: ['pipe', 'pipe', 'ignore']
        }).trim();

        if (gitStatus) {
          console.log(`❌ 警告: ${file} 被 Git 跟踪！`);
          console.log(`   这是安全风险，应该立即移除：`);
          console.log(`   git rm --cached ${file}`);
          console.log(`   并添加到 .gitignore\n`);
        } else {
          console.log(`✅ ${file} 未被 Git 跟踪\n`);
        }
      } catch {
        console.log(`✅ ${file} 未被 Git 跟踪\n`);
      }
    } else {
      console.log(`⚠️  ${file} 不存在\n`);
    }
  }

  // 检查 .gitignore
  const gitignorePath = path.join(process.cwd(), '.gitignore');
  if (fs.existsSync(gitignorePath)) {
    const gitignore = fs.readFileSync(gitignorePath, 'utf-8');
    const hasOpenClawIgnore = gitignore.includes('.openclaw') ||
                               gitignore.includes('settings.json');

    if (hasOpenClawIgnore) {
      console.log('✅ .gitignore 包含 OpenClaw 配置文件规则');
    } else {
      console.log('❌ .gitignore 缺少 OpenClaw 配置文件规则');
      console.log('   建议添加：');
      console.log('   .openclaw/');
      console.log('   settings.json');
    }
  }
}

checkConfigInGit();
```

### 实际应用场景

**场景 1：团队协作**

## 反直觉点 3：Node.js 22+ 不仅仅是版本号要求

### ❌ 错误认知

**"Node.js 22+ 只是一个版本号要求，用 Node.js 20 也能运行 OpenClaw"**

很多开发者认为：
- 版本号只是建议，不是硬性要求
- 低版本 Node.js 可能会有警告，但不会影响功能
- 可以通过 polyfill 或 transpile 解决兼容性问题

### 为什么会有这个误区？

**原因 1：对 Node.js 版本特性不了解**
- 不知道 Node.js 22 引入了哪些关键特性
- 认为 Node.js 版本之间差异不大

**原因 2：过去的经验**
- 某些库的版本要求不严格，低版本也能运行
- 导致开发者忽视版本要求

**原因 3：npm 的宽松检查**
- npm 默认只警告版本不匹配，不会阻止安装
- 开发者可能忽略警告继续使用

### ✅ 正确理解

**Node.js 22+ 是硬性要求，原因如下：**

```typescript
// ===== Node.js 22+ 的关键特性 =====

// 1. ESM 同步 require()（稳定）
// Node.js 20: 需要 --experimental-require-module flag
// Node.js 22: 默认支持
const esmModule = require('./esm-module.mjs');

// 2. 内置 WebSocket 客户端（稳定）
// Node.js 20: 需要 --experimental-websocket flag
// Node.js 22: 默认启用
const ws = new WebSocket('ws://localhost:3000');

// 3. Maglev 编译器（默认启用）
// Node.js 20: 未启用
// Node.js 22: 默认启用，提升 CLI 启动速度 30-50%

// 4. V8 引擎 12.4
// Node.js 20: V8 11.3
// Node.js 22: V8 12.4（新特性：Array.fromAsync, Set methods）

// ===== OpenClaw 依赖这些特性 =====
// 如果使用 Node.js 20，会遇到以下问题：

// 问题 1：ESM 模块加载失败
// Error: Cannot find module './esm-module.mjs'

// 问题 2：WebSocket 连接失败
// Error: WebSocket is not defined

// 问题 3：性能下降
// CLI 启动时间增加 30-50%

// 问题 4：语法错误
// SyntaxError: Unexpected token (使用了 V8 12.4 的新语法)
```

**验证代码：**


## 反直觉点总结

| 误区 | 正确理解 | 关键要点 |
|------|----------|----------|
| **全局安装污染项目** | 全局安装和本地安装完全隔离 | 全局安装不会修改 package.json |
| **配置文件应该提交** | 配置文件包含敏感数据，不应提交 | 提交配置模板，不提交实际配置 |
| **版本号只是建议** | Node.js 22+ 是硬性要求 | 依赖关键特性，低版本无法运行 |
