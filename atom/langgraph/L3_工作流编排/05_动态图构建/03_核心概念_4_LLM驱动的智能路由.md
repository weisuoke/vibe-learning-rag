# 核心概念4：LLM 驱动的智能路由

## 一句话定义

**用 LLM 的理解能力替代硬编码规则，实现基于语义的动态路由决策——让 AI 来判断"这个请求应该交给谁处理"。**

---

## 双重类比

### 前端类比：AI 驱动的 A/B 测试路由

传统前端 A/B 测试用硬编码规则分流（用户 ID 奇偶、地区判断）。LLM 驱动的路由就像一个"智能分流器"——它不看固定规则，而是理解用户的意图后再决定走哪条路径。

```javascript
// 前端传统路由：硬编码规则
if (url.includes("/api")) return <ApiDocs />;
if (url.includes("/guide")) return <Guide />;

// LLM 驱动路由的等价思路：理解语义再路由
// "我想知道怎么调用接口" → 虽然没有 "/api" 关键词，但语义上应该去 ApiDocs
```

### 日常生活类比：智能导航推荐路线

硬编码路由就像固定路线的公交车——只认站牌名。LLM 路由就像智能导航——你说"我想去吃火锅"，它理解你的意图，推荐最近的火锅店并规划路线。你不需要告诉它具体地址，它能从语义中推断出目的地。

---

## 核心模式：LLM + 结构化输出 + 条件边

LLM 驱动路由的核心思路是三步走：

```
用户输入 → LLM 理解意图 → 结构化输出（路由决策） → 条件边分发到目标节点
```

### 为什么需要结构化输出？

LLM 默认返回自由文本，但路由需要精确的、可编程的决策结果。通过 Pydantic BaseModel 定义路由 Schema，让 LLM 的输出变成可靠的结构化数据：

```python
from pydantic import BaseModel
from typing import Literal

# 第一步：定义路由 Schema
class Route(BaseModel):
    """路由决策的结构化输出"""
    step: Literal["faq", "technical", "billing", "human"]

# 第二步：让 LLM 输出结构化决策
router_llm = llm.with_structured_output(Route)

# 第三步：路由节点解析 LLM 输出
def llm_router(state: State):
    decision = router_llm.invoke([
        SystemMessage(content="根据用户问题分类意图..."),
        HumanMessage(content=state["input"]),
    ])
    return {"route_decision": decision.step}

# 第四步：条件边根据路由结果分发
def route_decision(state: State) -> str:
    return state["route_decision"]

graph.add_conditional_edges("llm_router", route_decision, {
    "faq": "faq_handler",
    "technical": "tech_handler",
    "billing": "billing_handler",
    "human": "human_handler",
})
```

[来源: Context7 官方文档 - LLM 驱动的路由]

### 四步模式总结

| 步骤 | 做什么 | 用什么 |
|------|--------|--------|
| 1. 定义 Schema | 约束 LLM 输出格式 | Pydantic BaseModel + Literal |
| 2. 绑定结构化输出 | 让 LLM 返回 Schema 实例 | `llm.with_structured_output(Route)` |
| 3. 路由节点 | 调用 LLM 并解析结果 | 普通节点函数 |
| 4. 条件边分发 | 根据决策结果路由 | `add_conditional_edges` |

## 完整代码示例：智能客服路由系统

```python
"""
LLM 驱动的智能路由实战
场景：智能客服系统，用 LLM 判断用户意图并路由到对应处理节点
"""

import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph, START, END

# 注意：实际使用时需要配置 LLM
# from langchain_openai import ChatOpenAI
# llm = ChatOpenAI(model="gpt-4o-mini")


# ===== 1. 定义路由 Schema =====

class RouteDecision(BaseModel):
    """LLM 路由决策的结构化输出"""
    intent: Literal["faq", "technical", "billing", "human"] = Field(
        description="用户意图分类"
    )
    confidence: float = Field(
        description="置信度 0-1",
        ge=0.0, le=1.0
    )
    reasoning: str = Field(
        description="分类理由"
    )


# ===== 2. 定义状态 =====

class State(TypedDict):
    user_input: str
    intent: str
    confidence: float
    messages: Annotated[list[str], operator.add]
    resolved: bool


# ===== 3. 模拟 LLM 路由（实际项目替换为真实 LLM） =====

def mock_llm_classify(user_input: str) -> RouteDecision:
    """
    模拟 LLM 的结构化输出
    实际项目中替换为：
        router_llm = llm.with_structured_output(RouteDecision)
        decision = router_llm.invoke([...messages...])
    """
    text = user_input.lower()
    if any(w in text for w in ["价格", "多少钱", "费用", "套餐"]):
        return RouteDecision(intent="billing", confidence=0.92, reasoning="涉及费用相关")
    elif any(w in text for w in ["报错", "bug", "崩溃", "无法连接"]):
        return RouteDecision(intent="technical", confidence=0.88, reasoning="技术问题")
    elif any(w in text for w in ["怎么", "如何", "什么是"]):
        return RouteDecision(intent="faq", confidence=0.85, reasoning="常见问题")
    else:
        return RouteDecision(intent="human", confidence=0.60, reasoning="无法确定意图")


# ===== 4. 定义节点 =====

def llm_router(state: State) -> dict:
    """
    路由节点：调用 LLM 判断意图
    实际项目中这里会调用 llm.with_structured_output(RouteDecision)
    """
    decision = mock_llm_classify(state["user_input"])
    print(f"[LLM Router] 意图: {decision.intent}, 置信度: {decision.confidence:.2f}")
    print(f"  理由: {decision.reasoning}")
    return {
        "intent": decision.intent,
        "confidence": decision.confidence,
        "messages": [f"[路由] 意图={decision.intent}, 置信度={decision.confidence:.2f}"],
    }


def faq_handler(state: State) -> dict:
    """FAQ 处理节点"""
    answer = f"FAQ 回答：关于「{state['user_input']}」，请参考帮助文档。"
    print(f"[FAQ] {answer}")
    return {"messages": [f"[FAQ] {answer}"], "resolved": True}


def tech_handler(state: State) -> dict:
    """技术支持处理节点"""
    answer = f"技术支持：关于「{state['user_input']}」，建议检查日志并重启服务。"
    print(f"[Tech] {answer}")
    return {"messages": [f"[Tech] {answer}"], "resolved": True}


def billing_handler(state: State) -> dict:
    """计费处理节点"""
    answer = f"计费回答：关于「{state['user_input']}」，请查看定价页面。"
    print(f"[Billing] {answer}")
    return {"messages": [f"[Billing] {answer}"], "resolved": True}


def human_handler(state: State) -> dict:
    """人工客服处理节点"""
    print(f"[Human] 转接人工客服处理: {state['user_input']}")
    return {"messages": ["[Human] 已转接人工客服，请稍候..."], "resolved": False}


# ===== 5. 路由函数 =====

def route_by_intent(state: State) -> str:
    """条件边的路由函数：根据 LLM 判断的意图分发"""
    return state["intent"]


# ===== 6. 构建图 =====

builder = StateGraph(State)

# 添加节点
builder.add_node("llm_router", llm_router)
builder.add_node("faq_handler", faq_handler)
builder.add_node("tech_handler", tech_handler)
builder.add_node("billing_handler", billing_handler)
builder.add_node("human_handler", human_handler)

# 入口
builder.add_edge(START, "llm_router")

# LLM 路由的条件边
builder.add_conditional_edges("llm_router", route_by_intent, {
    "faq": "faq_handler",
    "technical": "tech_handler",
    "billing": "billing_handler",
    "human": "human_handler",
})

# 所有处理节点 → 结束
for handler in ["faq_handler", "tech_handler", "billing_handler", "human_handler"]:
    builder.add_edge(handler, END)

# 编译
graph = builder.compile()


# ===== 7. 运行测试 =====

if __name__ == "__main__":
    print("=== LLM 驱动的智能路由示例 ===\n")

    test_cases = [
        "你们的套餐多少钱？",
        "服务报错了，无法连接数据库",
        "怎么修改我的密码？",
        "我想和你们谈一个合作方案",
    ]

    for query in test_cases:
        print(f"--- 用户: {query} ---")
        result = graph.invoke({
            "user_input": query,
            "intent": "",
            "confidence": 0.0,
            "messages": [],
            "resolved": False,
        })
        print(f"已解决: {result['resolved']}")
        print()
```


## 进阶模式

### 模式1：多级路由（先粗分类再细分类）

当分类维度很多时，一次性让 LLM 从 20 个选项中选择效果不好。更好的做法是分层路由：

```python
# 第一级：粗分类
class CoarseRoute(BaseModel):
    category: Literal["sales", "support", "general"]

# 第二级：细分类（仅在 support 分支内）
class SupportRoute(BaseModel):
    sub_category: Literal["technical", "billing", "account"]

```

执行流程：

```
用户输入 → [LLM 粗分类] → support → [LLM 细分类] → technical/billing/account
                        → sales → 销售节点
                        → general → 通用节点
```

**前端类比：** 就像嵌套路由——先匹配 `/app/support/*`，再在 support 内部匹配 `/app/support/technical`。

### 模式2：带置信度的路由（低置信度走人工）

LLM 不是万能的，当它"不确定"时应该有兜底策略：

```python
class RouteWithConfidence(BaseModel):
    intent: Literal["faq", "technical", "billing"]
    confidence: float = Field(ge=0.0, le=1.0)

def smart_router(state: State) -> dict:
    decision = router_llm.invoke([...])
    # 低置信度时走人工审核
    if decision.confidence < 0.7:
        return {"intent": "human_review", "confidence": decision.confidence}
    return {"intent": decision.intent, "confidence": decision.confidence}
```

**日常生活类比：** 就像医院的分诊台——护士能判断的直接分科室，拿不准的就请主任来看。

### 模式3：路由 + 状态更新（Command 方式）

结合前面学过的 Command API，可以在路由的同时更新状态：

```python
from langgraph.types import Command

def llm_router_with_command(state: State) -> Command[Literal["faq", "tech", "human"]]:
    """用 Command 实现 LLM 路由：路由 + 状态更新一步到位"""
    decision = router_llm.invoke([...])

    return Command(
        update={
            "intent": decision.intent,
            "confidence": decision.confidence,
            "messages": [f"[路由] {decision.intent} (置信度: {decision.confidence})"],
        },
        goto=decision.intent,
    )

# 使用 Command 时不需要 add_conditional_edges
builder.add_node("llm_router", llm_router_with_command)
# Command 自带路由，不需要额外的条件边
```

## 与规则路由的对比

| 维度 | 规则路由（硬编码） | LLM 驱动路由 |
|------|-------------------|-------------|
| 决策方式 | `if "退款" in text` | LLM 理解语义后判断 |
| 灵活性 | 只能匹配预定义的关键词 | 理解同义词、上下文、隐含意图 |
| 维护成本 | 规则越多越难维护 | 只需调整 prompt |
| 延迟 | 微秒级 | 毫秒~秒级（LLM 调用） |
| 成本 | 几乎为零 | 每次调用消耗 token |
| 准确率 | 简单场景高，复杂场景低 | 复杂场景显著更高 |
| 可解释性 | 完全透明 | 可通过 reasoning 字段解释 |

### 什么时候用规则路由？

- 分类维度少且明确（2-3 个分支）
- 关键词匹配就够用
- 对延迟要求极高（< 10ms）
- 预算有限，不想为路由付 LLM 费用

### 什么时候用 LLM 路由？

- 用户输入多样，关键词匹配不够
- 需要理解语义和上下文
- 分类维度多（5+ 个分支）
- 需要处理模糊、歧义的输入

### 混合策略（推荐）

实际项目中，最佳实践是混合使用：

```python
def hybrid_router(state: State) -> dict:
    """混合路由：先用规则快速匹配，匹配不到再用 LLM"""
    text = state["user_input"].lower()

    # 第一层：规则快速匹配（零成本）
    if "退款" in text:
        return {"intent": "refund", "confidence": 1.0}
    if "投诉" in text:
        return {"intent": "complaint", "confidence": 1.0}

    # 第二层：规则匹配不到，用 LLM 理解语义
    decision = router_llm.invoke([...])
    return {"intent": decision.intent, "confidence": decision.confidence}
```

## 实际应用场景

### 场景1：RAG 查询路由

根据用户问题类型，决定走哪种检索策略：

```python
class RAGRoute(BaseModel):
    strategy: Literal["vector_search", "keyword_search", "hybrid", "no_retrieval"]

# "什么是量子计算" → vector_search（语义检索）
# "错误码 E1001" → keyword_search（精确匹配）
# "量子计算的错误码" → hybrid（混合检索）
# "你好" → no_retrieval（不需要检索）
```

### 场景2：多工具选择

Agent 根据用户需求选择合适的工具：

```python
class ToolRoute(BaseModel):
    tool: Literal["calculator", "web_search", "code_executor", "file_reader"]

# "帮我算一下 15% 的税" → calculator
# "最新的 Python 3.13 有什么新特性" → web_search
# "运行这段代码看看结果" → code_executor
```

### 场景3：对话管理

根据对话状态决定下一步动作：

```python
class DialogRoute(BaseModel):
    action: Literal["ask_clarification", "provide_answer", "escalate", "end_conversation"]

# 用户问题模糊 → ask_clarification
# 信息充足 → provide_answer
# 超出能力范围 → escalate
# 用户说"谢谢再见" → end_conversation
```

## 总结

LLM 驱动的智能路由是动态图构建中最强大的模式：通过 Pydantic Schema 约束 LLM 输出、`with_structured_output` 获取结构化决策、条件边分发到目标节点。它让路由从"关键词匹配"升级为"语义理解"，特别适合 RAG 查询路由、多工具选择、对话管理等场景。实际项目中推荐混合策略——规则快速匹配 + LLM 兜底，兼顾性能和准确率。
