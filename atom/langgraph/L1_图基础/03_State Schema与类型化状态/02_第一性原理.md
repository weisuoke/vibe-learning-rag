# 第一性原理

> 从底层原理理解 State Schema 与类型化状态

## 第一性原理推导

### 问题1：为什么图需要状态？

**推导链**：

```
图 = 节点 + 边
节点 = 计算单元
计算需要输入 → 需要数据
计算产生输出 → 需要传递数据
多个节点协作 → 需要共享数据
→ 需要状态（State）
```

**本质**：状态是图中数据流转的载体。

### 问题2：为什么需要定义状态结构？

**推导链**：

```
状态 = 数据容器
数据有类型（字符串、数字、列表等）
节点需要知道数据类型才能正确处理
错误的类型会导致运行时崩溃
→ 需要定义状态结构（Schema）
```

**本质**：Schema 是数据的契约，确保类型安全。

### 问题3：为什么需要 Reducer？

**推导链**：

```
多个节点可能修改同一个状态字段
节点1返回：{"messages": ["hello"]}
节点2返回：{"messages": ["world"]}
如何合并？
  - 覆盖？只保留最后一个
  - 追加？合并成 ["hello", "world"]
  - 其他？自定义逻辑
→ 需要 Reducer 定义合并规则
```

**本质**：Reducer 是状态聚合的规则。

### 问题4：为什么使用 Annotated？

**推导链**：

```
Reducer 是元数据（描述如何处理数据）
不是数据本身
Python 类型系统需要区分类型和元数据
Annotated[T, metadata] 是标准方式
→ 使用 Annotated 附加 Reducer
```

**本质**：Annotated 是类型注解的元数据容器。

### 问题5：为什么需要 Channel？

**推导链**：

```
状态字段有不同的更新策略
  - 简单字段：直接覆盖
  - 列表字段：追加
  - 字典字段：合并
需要统一的抽象来处理不同策略
→ Channel 是状态管理的抽象层
```

**本质**：Channel 封装了状态更新的底层机制。

## 核心原理

### 原理1：类型化状态的本质

**定义**：类型化状态 = 数据结构 + 类型约束 + 更新规则

```python
from typing import TypedDict, Annotated
import operator

class State(TypedDict):
    # 数据结构：定义字段
    # 类型约束：str, list, int
    # 更新规则：operator.add
    messages: Annotated[list, operator.add]
    user_id: str
    count: int
```

**为什么这样设计**：
- 数据结构：明确图需要哪些数据
- 类型约束：编译时检查，减少错误
- 更新规则：自动化状态合并，简化开发

### 原理2：Reducer 的执行时机

**定义**：Reducer 在状态需要合并时执行

```python
# 时机1：节点返回后
def node(state: State) -> dict:
    return {"messages": ["new"]}
# StateGraph 调用 reducer: old_messages + ["new"]

# 时机2：并行执行后
# node1 返回 {"messages": ["a"]}
# node2 返回 {"messages": ["b"]}
# StateGraph 调用 reducer: [] + ["a"] + ["b"]

# 时机3：手动更新时
graph.update_state({"messages": ["manual"]})
# StateGraph 调用 reducer: old_messages + ["manual"]
```

**为什么这样设计**：
- 自动化：无需手动调用 reducer
- 一致性：所有更新都经过 reducer
- 并发安全：reducer 确保并行更新正确

### 原理3：Channel 的分层设计

**定义**：Channel = 存储 + 更新策略 + 序列化

```python
class Channel:
    def update(self, value):  # 更新策略
        pass

    def get(self):  # 获取值
        pass

    def checkpoint(self):  # 序列化
        pass

    def from_checkpoint(self, data):  # 反序列化
        pass
```

**为什么这样设计**：
- 存储：管理状态值
- 更新策略：封装 reducer 逻辑
- 序列化：支持状态持久化

### 原理4：TypedDict vs Pydantic

**对比**：

| 维度 | TypedDict | Pydantic |
|------|-----------|----------|
| 类型检查 | 静态（mypy） | 运行时 |
| 验证 | 无 | 自动验证 |
| 性能 | 更快 | 稍慢 |
| 依赖 | 无 | pydantic |

**为什么推荐 TypedDict**：
- 轻量级：无额外依赖
- 性能好：无运行时开销
- 够用：大多数场景不需要验证

**何时用 Pydantic**：
- 需要验证：用户输入、外部 API
- 复杂默认值：工厂函数、计算值
- 序列化需求：JSON Schema、文档生成

### 原理5：状态更新的原子性

**定义**：状态更新是原子操作

```python
# 节点返回
def node(state: State) -> dict:
    return {
        "messages": ["a"],
        "count": 1
    }

# StateGraph 原子更新
# 1. 应用所有 reducer
# 2. 更新所有字段
# 3. 创建新状态
# 不会出现部分更新的中间状态
```

**为什么这样设计**：
- 一致性：避免中间状态
- 并发安全：原子操作不会冲突
- 可回滚：失败时状态不变

## 设计哲学

### 哲学1：声明式优于命令式

**声明式**（LangGraph）：

```python
class State(TypedDict):
    messages: Annotated[list, operator.add]

# 声明：messages 使用 add 聚合
# StateGraph 自动执行
```

**命令式**（传统方式）：

```python
def merge_state(old_state, new_state):
    old_state["messages"] = old_state["messages"] + new_state["messages"]
    return old_state

# 手动调用合并逻辑
```

**为什么声明式更好**：
- 简洁：无需手动编写合并逻辑
- 可靠：框架保证正确执行
- 可维护：规则集中定义

### 哲学2：类型安全优于运行时检查

**类型安全**（编译时）：

```python
class State(TypedDict):
    count: int

def node(state: State) -> dict:
    return {"count": "not an int"}  # mypy 报错
```

**运行时检查**：

```python
def node(state):
    return {"count": "not an int"}  # 运行时崩溃
```

**为什么类型安全更好**：
- 早发现：编译时而非运行时
- 更安全：类型检查器保证
- 更快：无运行时开销

### 哲学3：组合优于继承

**组合**（LangGraph）：

```python
# 多个 Schema 组合
class InputState(TypedDict):
    query: str

class OutputState(TypedDict):
    answer: str

class OverallState(InputState, OutputState):
    internal_data: dict
```

**继承**（传统方式）：

```python
class BaseState:
    pass

class ExtendedState(BaseState):
    pass
```

**为什么组合更好**：
- 灵活：可以组合多个 Schema
- 清晰：明确输入输出
- 解耦：Schema 之间独立

## 底层实现

### 实现1：StateGraph 如何创建 Channel

```python
def __init__(self, state_schema: Type):
    # 1. 提取类型提示
    hints = get_type_hints(state_schema, include_extras=True)

    # 2. 为每个字段创建 Channel
    self.channels = {}
    for name, type_hint in hints.items():
        # 检查是否有 Annotated
        if hasattr(type_hint, '__metadata__'):
            reducer = type_hint.__metadata__[0]
            self.channels[name] = BinaryOperatorAggregate(reducer)
        else:
            self.channels[name] = LastValue()
```

### 实现2：状态更新流程

```python
def update_state(self, updates: dict):
    # 1. 遍历更新
    for key, value in updates.items():
        channel = self.channels[key]
        channel.update(value)

    # 2. 聚合所有 Channel
    new_state = {}
    for key, channel in self.channels.items():
        new_state[key] = channel.get()

    # 3. 返回新状态
    return new_state
```

### 实现3：并行执行的状态聚合

```python
def execute_parallel_nodes(self, nodes):
    # 1. 并行执行所有节点
    results = [node(self.state) for node in nodes]

    # 2. 收集所有更新
    all_updates = {}
    for result in results:
        for key, value in result.items():
            if key not in all_updates:
                all_updates[key] = []
            all_updates[key].append(value)

    # 3. 应用 reducer 聚合
    final_updates = {}
    for key, values in all_updates.items():
        channel = self.channels[key]
        for value in values:
            channel.update(value)
        final_updates[key] = channel.get()

    # 4. 更新状态
    self.update_state(final_updates)
```

## 关键洞察

### 洞察1：状态是图的记忆

图的执行是有状态的：
- 节点读取状态（记忆）
- 节点修改状态（更新记忆）
- 下一个节点继续使用状态（延续记忆）

### 洞察2：Reducer 是聚合的本质

Reducer 不仅仅是合并：
- 列表追加：`operator.add`
- 字典合并：`operator.or_`
- 数值累加：`lambda x, y: x + y`
- 自定义逻辑：任何二元函数

### 洞察3：Channel 是状态管理的抽象

Channel 封装了复杂性：
- 用户只需定义 Schema
- Channel 自动处理更新
- 支持持久化和恢复

### 洞察4：类型化状态是契约

Schema 是节点之间的契约：
- 输入契约：节点期望什么数据
- 输出契约：节点返回什么数据
- 更新契约：如何合并数据
