# Agent最佳实践 - 核心概念 2：可靠性设计模式

> 可靠性设计 = 重试（ModelRetry）+ 降级（ModelFallback）+ 限制（CallLimit）三重保护，让 Agent 在 API 超时、模型不可用、无限循环等异常情况下依然能正常工作。

---

## 为什么 Agent 需要可靠性设计？

Agent 在生产中面临三类可靠性挑战：

| 挑战 | 场景 | 后果 |
|------|------|------|
| **瞬时故障** | API 超时、限流（429）、网络抖动 | 单次失败 → 用户看到错误 |
| **服务不可用** | 模型服务宕机、供应商维护 | 整个 Agent 不可用 |
| **无限循环** | LLM 反复调用同一工具、推理陷入死循环 | Token 飙升、响应超时 |

```
没有可靠性设计：
  API 超时 → Agent 崩溃 → 用户看到 500 错误 💀

有可靠性设计：
  API 超时 → 自动重试（等 1s）→ 再试一次 → 成功 ✅
  API 持续超时 → 重试 3 次都失败 → 降级到备用模型 → 成功 ✅
  Agent 循环 25 次 → 触发调用限制 → 优雅结束 ✅
```

## 第一重保护：ModelRetryMiddleware（模型重试）

### 核心机制

当 LLM 调用因网络问题或限流而失败时，自动重试。

```python
from langchain.agents.middleware import ModelRetryMiddleware

retry = ModelRetryMiddleware(
    max_retries=3,           # 最多重试 3 次
    retry_on=(               # 只对这些异常重试
        "ConnectionError",
        "TimeoutError",
        "RateLimitError",
    ),
    on_failure="continue",   # 所有重试都失败后的行为
    backoff_factor=2.0,      # 退避因子（每次等待时间翻倍）
    initial_delay=1.0,       # 首次重试等待 1 秒
    max_delay=30.0,          # 最大等待时间 30 秒
    jitter=True,             # 启用随机抖动
)
```

### 指数退避 + 随机抖动

这是分布式系统中的经典重试策略：

```
第 1 次重试：等待 1s × 2^0 = 1.0s ± 抖动
第 2 次重试：等待 1s × 2^1 = 2.0s ± 抖动
第 3 次重试：等待 1s × 2^2 = 4.0s ± 抖动

抖动范围：±25%（默认值）

实际等待时间示例：
  第 1 次：0.75s ~ 1.25s
  第 2 次：1.50s ~ 2.50s
  第 3 次：3.00s ~ 5.00s
```

**为什么需要随机抖动？** 防止**雷群效应（Thundering Herd）**——当 API 恢复后，如果所有客户端同时重试，会再次压垮服务器。抖动让不同客户端的重试时间错开。

### 源码解析

```python
# langchain_v1/agents/middleware/model_retry.py — 核心重试逻辑

class ModelRetryMiddleware(AgentMiddleware):
    def wrap_model_call(self, request, call_next):
        last_error = None
        for attempt in range(self.max_retries + 1):
            try:
                return call_next(request)
            except Exception as e:
                # 检查是否应该重试此类错误
                if not self._should_retry(e):
                    raise
                last_error = e
                if attempt < self.max_retries:
                    # 计算等待时间（指数退避 + 抖动）
                    delay = min(
                        self.initial_delay * (self.backoff_factor ** attempt),
                        self.max_delay,
                    )
                    if self.jitter:
                        delay *= random.uniform(0.75, 1.25)  # ±25% 抖动
                    time.sleep(delay)

        # 所有重试都失败
        if self.on_failure == "error":
            raise last_error
        elif self.on_failure == "continue":
            return None  # 让 Agent 继续执行（跳过此次 LLM 调用）
        elif callable(self.on_failure):
            return self.on_failure(last_error, request)
```

### on_failure 三种策略

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `"error"` | 抛出异常，终止 Agent | 对可靠性要求极高，宁可失败也不要错误答案 |
| `"continue"` | 跳过本次调用，继续循环 | 允许部分步骤失败的场景 |
| `callable` | 自定义处理函数 | 需要记录日志、发送告警等自定义逻辑 |

## 第二重保护：ModelFallbackMiddleware（模型降级）

### 核心机制

主模型不可用时，按优先级依次尝试备用模型。

```python
from langchain.agents.middleware import ModelFallbackMiddleware

fallback = ModelFallbackMiddleware(
    fallbacks=[
        "gpt-4.1-mini",       # 首选降级：同供应商的小模型
        "claude-3.5-sonnet",   # 次选降级：跨供应商
    ],
)
```

### 降级链路

```
用户请求 → 主模型 (gpt-4.1)
              ↓ 失败
           备用模型 1 (gpt-4.1-mini)
              ↓ 失败
           备用模型 2 (claude-3.5-sonnet)
              ↓ 失败
           最终失败 → 抛出异常
```

### 源码解析

```python
# langchain_v1/agents/middleware/model_fallback.py — 降级逻辑

class ModelFallbackMiddleware(AgentMiddleware):
    def wrap_model_call(self, request, call_next):
        # 1. 先尝试主模型
        try:
            return call_next(request)
        except Exception as primary_error:
            pass

        # 2. 依次尝试备用模型
        for fallback_model in self.fallbacks:
            try:
                # 关键：使用 request.override() 替换模型
                fallback_request = request.override(model=fallback_model)
                return call_next(fallback_request)
            except Exception:
                continue

        # 3. 所有模型都失败
        raise primary_error
```

**关键设计**：`request.override(model=fallback_model)` 只替换模型，保留其他所有设置（工具、提示词、参数）。

### 降级策略选择

| 降级策略 | 优点 | 缺点 | 示例 |
|---------|------|------|------|
| **同供应商小模型** | 延迟低、成本低 | 能力可能不足 | gpt-4.1 → gpt-4.1-mini |
| **跨供应商模型** | 供应商级别的容灾 | 可能需要不同的 API Key | gpt-4.1 → claude-3.5-sonnet |
| **本地模型** | 不依赖网络 | 能力有限 | gpt-4.1 → ollama/llama-3 |

## 第三重保护：调用限制 Middleware

### ModelCallLimitMiddleware（模型调用限制）

防止 Agent 无限循环调用 LLM，导致 Token 消耗失控。

```python
from langchain.agents.middleware import ModelCallLimitMiddleware

limit = ModelCallLimitMiddleware(
    thread_limit=100,        # 线程级别：整个会话最多 100 次 LLM 调用
    run_limit=25,            # 运行级别：单次执行最多 25 次 LLM 调用
    exit_behavior="end",     # 超限时行为："end"（优雅结束）或 "error"（抛异常）
)
```

### 两种限制维度

```
thread_limit = 100（持久化，跨多次 invoke）
  ┌───────────────────────────────────────┐
  │ invoke #1: 用了 10 次                   │
  │ invoke #2: 用了 15 次                   │
  │ invoke #3: 用了 20 次                   │
  │ ...                                     │
  │ 累计达到 100 次 → 线程结束               │
  └───────────────────────────────────────┘

run_limit = 25（每次 invoke 独立计数）
  ┌───────────────┐  ┌───────────────┐
  │ invoke #1     │  │ invoke #2     │
  │ 计数: 0→25    │  │ 计数: 0→25    │  ← 每次独立
  │ 达到限制→结束  │  │ 达到限制→结束  │
  └───────────────┘  └───────────────┘
```

### ToolCallLimitMiddleware（工具调用限制）

更细粒度的控制——按工具限制调用次数。

```python
from langchain.agents.middleware import ToolCallLimitMiddleware

tool_limit = ToolCallLimitMiddleware(
    allowed={
        "search": 10,          # search 工具最多调用 10 次
        "calculator": 5,       # calculator 最多 5 次
    },
    blocked=["dangerous_tool"], # 直接禁止某些工具
    exit_behavior="continue",   # 超限后继续执行但跳过该工具
)
```

### exit_behavior 的三种模式

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| `"end"` | 优雅结束 Agent 执行 | 超限意味着任务异常，应该停止 |
| `"error"` | 抛出异常 | 需要上层捕获并处理 |
| `"continue"` | 跳过受限的调用，继续执行 | 允许部分工具受限但 Agent 继续工作 |

## 三重保护的组合使用

### 推荐组合模式

```python
from langchain.agents import create_agent
from langchain.agents.middleware import (
    ModelCallLimitMiddleware,
    ModelRetryMiddleware,
    ModelFallbackMiddleware,
    ToolRetryMiddleware,
    ToolCallLimitMiddleware,
)

agent = create_agent(
    "gpt-4.1",
    tools=tools,
    middleware=[
        # 第 1 层：全局限制（最外层）
        ModelCallLimitMiddleware(run_limit=25),
        ToolCallLimitMiddleware(allowed={"search": 10}),

        # 第 2 层：模型可靠性
        ModelRetryMiddleware(max_retries=3, backoff_factor=2.0),
        ModelFallbackMiddleware(fallbacks=["gpt-4.1-mini"]),

        # 第 3 层：工具可靠性
        ToolRetryMiddleware(max_retries=2, tools=["search"]),
    ],
)
```

### 执行流程示例

```
用户请求："帮我查今天北京的天气"

1. ModelCallLimitMiddleware 检查：第 1 次调用，未超限 ✅
2. ModelRetryMiddleware 准备重试逻辑
3. ModelFallbackMiddleware 准备降级逻辑
4. LLM 调用 (gpt-4.1)
   → 成功！返回：调用 search 工具
5. ToolCallLimitMiddleware 检查：search 第 1 次，未超限 ✅
6. ToolRetryMiddleware 准备重试逻辑
7. search 工具执行
   → API 超时！
   → ToolRetryMiddleware 重试 #1
   → 成功！返回天气数据

8. ModelCallLimitMiddleware 检查：第 2 次调用，未超限 ✅
9. LLM 调用 (gpt-4.1)
   → 成功！返回最终答案

总计：2 次 LLM 调用 + 2 次工具调用（含 1 次重试）
```

## 经典错误处理对比（AgentExecutor vs Middleware）

### 旧方式（AgentExecutor 经典模式）

```python
# langchain_classic — 错误处理分散在多个地方
agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=15,                    # 限制
    max_execution_time=30.0,              # 时间限制
    handle_parsing_errors=True,           # 解析错误处理
    early_stopping_method="force",        # 早停
)

# 工具级别的错误处理
for tool in tools:
    tool.handle_tool_error = True
    tool.handle_validation_error = True
```

### 新方式（Middleware 模式）

```python
# langchain_v1 — 所有保护逻辑通过 Middleware 统一管理
agent = create_agent(
    "gpt-4.1",
    tools=tools,
    middleware=[
        ModelCallLimitMiddleware(run_limit=15),
        ModelRetryMiddleware(max_retries=3),
        ModelFallbackMiddleware(fallbacks=["gpt-4.1-mini"]),
        ToolRetryMiddleware(max_retries=2),
    ],
)
```

| 对比维度 | AgentExecutor | Middleware |
|---------|---------------|-----------|
| **配置位置** | 分散在 Executor + Tool 上 | 统一在 middleware 列表中 |
| **可组合性** | 低（硬编码的配置参数） | 高（按需添加 Middleware） |
| **自定义能力** | 受限（bool/str/callable 三种模式） | 强（可以完全自定义钩子逻辑） |
| **降级支持** | 无 | ModelFallbackMiddleware |
| **重试策略** | 无内置支持 | 指数退避 + 随机抖动 |

## 在 RAG 和 AI Agent 开发中的应用

| 应用场景 | 可靠性需求 | 推荐 Middleware 组合 |
|---------|-----------|-------------------|
| **RAG 文档问答** | 检索 API 偶尔超时 | Retry(3) + Fallback(mini) |
| **智能客服** | 高可用，不能崩 | Retry(3) + Fallback(跨供应商) + Limit(20) |
| **数据分析 Agent** | SQL 执行可能超时 | ToolRetry(2) + Limit(50) |
| **代码生成 Agent** | 多步推理，易循环 | Limit(25) + Retry(2) |
| **金融交易 Agent** | 极高可用性要求 | Retry(5) + Fallback(3个备用) + Limit(15) |
