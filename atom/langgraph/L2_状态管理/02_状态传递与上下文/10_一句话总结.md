## 核心定义（一句话）

**LangGraph 的状态传递与上下文是通过 Channel 机制实现节点间可变数据流转，通过 RunnableConfig 和 Runtime Context 传递不可变配置，两者共同构成了可靠、可恢复、可追溯的状态化工作流基础。**

### 2. 第一性原理

**本质定义**：状态传递 = 数据在节点间的流动 + 上下文的保持

**三层价值**：
1. **数据连续性**：确保信息在工作流的每个步骤都可访问
2. **上下文保持**：在整个执行过程中保持环境信息和配置
3. **可恢复性**：工作流可以在任意点暂停、恢复、回溯

**核心问题**：多步骤工作流如何保持信息的连续性？

**解决方案**：
- **外部状态存储**：不依赖 LLM 的上下文窗口
- **选择性暴露**：只在需要时访问特定信息
- **按需访问**：减少每次调用的 token 数量

### 3. 核心概念速查

#### 3.1 Channel 机制

**定义**：状态存储的基本单元，类似于消息队列

**关键特性**：
- 每个状态字段对应一个 Channel
- 支持不同的更新策略（Reducer）
- 触发节点执行（triggers 机制）

**代码示例**：
```python
from langgraph.channels import LastValue, BinaryOperator

# LastValue：覆盖策略
channel1 = LastValue(str)

# BinaryOperator：累加策略
channel2 = BinaryOperator(operator.add)
```

---

#### 3.2 ChannelRead 读取机制

**定义**：节点如何从 Channel 读取状态

**关键特性**：
- 通过 CONFIG_KEY_READ 注入读取函数
- fresh 参数控制是否读取最新值
- mapper 参数支持数据转换

**代码示例**：
```python
from langgraph.pregel import ChannelRead

# 读取最新值
read_messages = ChannelRead("messages", fresh=True)

# 读取并转换
read_with_mapper = ChannelRead("data", mapper=lambda x: x.upper())
```

---

#### 3.3 ChannelWrite 写入机制

**定义**：节点如何向 Channel 写入状态

**关键特性**：
- 通过 CONFIG_KEY_SEND 注入写入函数
- 支持批量写入
- PASSTHROUGH 模式直接传递节点返回值

**代码示例**：
```python
from langgraph.pregel import ChannelWrite, PASSTHROUGH

# 写入固定值
write_result = ChannelWrite([("result", "processed")])

# 写入节点返回值
write_passthrough = ChannelWrite([("output", PASSTHROUGH)])
```

---

#### 3.4 RunnableConfig 上下文传递

**定义**：LangGraph 传递运行时配置的标准方式

**常用配置项**：
- `thread_id`：线程标识（用于持久化）
- `tags`：追踪标签
- `configurable`：自定义配置

**代码示例**：
```python
from langchain_core.runnables import RunnableConfig

def node(state: State, config: RunnableConfig) -> State:
    thread_id = config.get("configurable", {}).get("thread_id")
    tags = config.get("tags", [])
    return {"processed": True}
```

---

#### 3.5 Runtime Context 机制

**定义**：运行时上下文管理，传递不可变配置

**关键特性**：
- 通过 context_schema 定义
- 通过 Runtime[Context] 访问
- 不可变性原则

**代码示例**：
```python
from dataclasses import dataclass
from langgraph.runtime import Runtime

@dataclass
class Context:
    api_key: str
    model_name: str

def node(state: State, runtime: Runtime[Context]) -> State:
    api_key = runtime.context.api_key
    model = runtime.context.model_name
    return {"processed": True}
```

---

#### 3.6 多状态 Schema 设计

**定义**：不同状态 Schema 的使用，隔离内部和外部状态

**基础模式**：
- **InputState**：定义输入接口
- **OutputState**：定义输出接口
- **OverallState**：内部完整状态
- **PrivateState**：私有状态

**代码示例**：
```python
class InputState(TypedDict):
    user_query: str

class OutputState(TypedDict):
    answer: str

class OverallState(TypedDict):
    user_query: str
    processed_query: str
    search_results: list
    answer: str

builder = StateGraph(
    OverallState,
    input_schema=InputState,
    output_schema=OutputState
)
```

### 6. 三大反直觉点

#### 误区1：Reducer 只在有新值时才调用 ❌

**正确理解**：
- 未返回字段：状态保持不变，Reducer 不会被调用
- 返回 None：Reducer 会被调用，传入 None 作为新值
- 返回空列表：Reducer 会被调用，传入空列表

---

#### 误区2：大上下文窗口可以替代状态管理 ❌

**正确理解**：
- **Context Poisoning**：幻觉内容污染上下文
- **Context Distraction**：过多信息干扰注意力
- **Context Confusion**：不同类型信息混淆
- **Context Clash**：矛盾信息冲突

**解决方案**：使用上下文工程策略（Write, Select, Compress, Isolate）

---

#### 误区3：State 和 Context 可以互换使用 ❌

**正确理解**：
- **State**：可变（Mutable），节点间共享和更新
- **Context**：不可变（Immutable），只读访问

**决策树**：
```
数据是否会在执行过程中改变？
├── 是 → 使用 State
└── 否 → 使用 Context
```

## 核心设计模式

### 1. 依赖注入模式

**问题**：节点如何访问状态？

**解决方案**：通过 RunnableConfig 注入读写函数

```python
# 框架内部实现（简化版）
def execute_node(node, config):
    config[CONF][CONFIG_KEY_READ] = lambda key: channels[key].get()
    config[CONF][CONFIG_KEY_SEND] = lambda updates: apply_updates(updates)
    result = node.invoke(config)
    return result
```

---

### 2. 策略模式

**问题**：不同的状态字段需要不同的更新策略

**解决方案**：使用 Reducer 函数定义更新策略

```python
from typing import Annotated
from operator import add

class State(TypedDict):
    last_result: str                        # 策略1：覆盖（默认）
    messages: Annotated[list, add_messages] # 策略2：追加
    counter: Annotated[int, add]            # 策略3：累加
    scores: Annotated[list, custom_merge]   # 策略4：自定义
```

---

### 3. 观察者模式

**问题**：如何触发节点执行？

**解决方案**：通过 triggers 机制监听 Channel 的变化

```python
node = PregelNode(
    channels=["input"],              # 读取的 channel
    triggers=["input", "docs"],      # 触发条件
    writers=[ChannelWrite([("output", PASSTHROUGH)])],
    bound=process_function
)
```

## 总结

**状态传递与上下文是 LangGraph 的核心机制**，它通过 Channel 机制实现节点间可变数据流转，通过 RunnableConfig 和 Runtime Context 传递不可变配置，两者共同构成了可靠、可恢复、可追溯的状态化工作流基础。

**核心价值**：
- **数据连续性**：确保信息在工作流的每个步骤都可访问
- **上下文保持**：在整个执行过程中保持环境信息和配置
- **可恢复性**：工作流可以在任意点暂停、恢复、回溯

**设计原则**：
- **外部存储**：不依赖 LLM 的上下文窗口
- **选择性暴露**：只在需要时访问特定信息
- **按需访问**：减少每次调用的 token 数量

**实现机制**：
- **Channel**：状态存储的基本单元
- **ChannelRead/ChannelWrite**：读写机制
- **RunnableConfig**：上下文传递的载体
- **Runtime Context**：运行时配置
- **多状态 Schema**：清晰的输入输出接口

**关键区别**：
- **State**：可变（Mutable），节点间共享和更新
- **Context**：不可变（Immutable），只读访问

掌握状态传递与上下文机制，是构建复杂、可靠、可维护的 LangGraph 工作流的基础。
