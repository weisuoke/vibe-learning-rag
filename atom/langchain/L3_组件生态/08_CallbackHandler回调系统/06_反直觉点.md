# CallbackHandler回调系统 - 反直觉点

## 什么是反直觉点？

**反直觉点**：与直觉相反的知识点，容易导致错误理解和使用。

在使用 CallbackHandler 时，有一些看似合理但实际错误的做法，这些错误往往源于对回调系统内部机制的误解。

---

## 误区1：并发环境回调管理 ❌

### 错误观点

**"回调处理器可以在多个请求之间重用，就像单例模式一样"**

```python
# ❌ 错误做法：全局单例回调处理器
from langfuse.langchain import CallbackHandler

# 全局初始化一个回调处理器
GLOBAL_HANDLER = CallbackHandler()

def reconciliation(request):
    # 多个请求共享同一个回调处理器
    chain = build_chain("get_python", tenant_id)
    response = chain.invoke(
        input={...},
        config={"callbacks": [GLOBAL_HANDLER]}  # ❌ 重用全局实例
    )
    trace_id = GLOBAL_HANDLER.last_trace_id  # ❌ trace_id 可能混乱
    return JsonResponse({"trace_id": trace_id, "response": response})
```

### 为什么错？

**核心问题：回调处理器包含状态，并发请求会导致状态冲突**

1. **状态污染**：
   - `last_trace_id` 会被多个请求覆盖
   - 请求 A 的 trace_id 可能被请求 B 覆盖
   - 最终返回的 trace_id 不是当前请求的

2. **追踪混乱**：
   - 多个请求的事件可能被记录到同一个 trace
   - 无法区分哪些事件属于哪个请求
   - 可观测性平台（Langfuse、LangSmith）显示错误的调用链

3. **数据丢失**：
   - `flush()` 可能在请求完成前被其他请求触发
   - 部分请求的数据可能未被持久化

### 为什么人们容易这样错？

**心理/认知原因**：

1. **单例模式的惯性思维**：
   - 在传统开发中，数据库连接、日志记录器等通常使用单例模式
   - 人们自然地认为回调处理器也应该是单例

2. **性能优化的误解**：
   - 认为每次创建新实例会影响性能
   - 实际上回调处理器的创建成本很低

3. **状态不可见**：
   - 回调处理器的内部状态（如 `last_trace_id`）不明显
   - 没有意识到状态会在并发环境中冲突

### 正确理解

**每个请求都应该创建新的 CallbackHandler 实例**

```python
from langfuse import get_client
from langfuse.langchain import CallbackHandler

def reconciliation(request):
    # ✅ 每个请求创建新的回调处理器实例
    langfuse_handler = CallbackHandler()

    chain = build_chain("get_python", tenant_id)

    response = chain.invoke(
        input={
            "source_csv": source_csv,
            "target_csv": target_csv,
            "metadata_csv": metadata_csv
        },
        config={
            "callbacks": [langfuse_handler],  # ✅ 使用独立实例
            "metadata": {
                "langfuse_user_id": user_name,
                "langfuse_session_id": "session_456",
                "langfuse_tags": ["django-request"]
            }
        }
    )

    # ✅ 从当前请求的处理器获取 trace_id
    trace_id = langfuse_handler.last_trace_id

    # ✅ 确保数据持久化（短生命周期环境）
    langfuse = get_client()
    langfuse.flush()

    return JsonResponse({"trace_id": trace_id, "response": response})
```

**关键要点**：

1. **每个请求新建实例**：避免状态冲突
2. **使用 `last_trace_id`**：从处理器实例获取 trace ID（不是从全局客户端）
3. **调用 `flush()`**：确保短生命周期环境（Django/FastAPI）的数据持久化
4. **传递 metadata**：通过 `config` 传递用户 ID、会话 ID、标签等

**FastAPI 异步环境示例**：

```python
from fastapi import FastAPI
from langfuse.langchain import CallbackHandler

app = FastAPI()

@app.post("/chat")
async def chat(request: ChatRequest):
    # ✅ 每个请求创建新实例
    langfuse_handler = CallbackHandler()

    # 异步执行 chain
    response = await chain.ainvoke(
        input={"query": request.query},
        config={"callbacks": [langfuse_handler]}
    )

    # ✅ 获取当前请求的 trace_id
    trace_id = langfuse_handler.last_trace_id

    # ✅ 异步 flush
    langfuse = get_client()
    await langfuse.aflush()

    return {"trace_id": trace_id, "response": response}
```

## 误区2：LangGraph trace 链接问题 ❌

### 错误观点

**"LangGraph 会自动正确链接 trace，不需要额外配置"**

```python
# ❌ 错误做法：期望自动链接 prompt
from langgraph.graph import StateGraph, MessagesState, START
from langfuse import Langfuse
from langfuse.callback import CallbackHandler

langfuse_handler = CallbackHandler()
langfuse = Langfuse()

def thinker(state: MessagesState):
    """LangGraph node"""
    # ❌ 直接获取 prompt，期望自动链接到 trace
    prompt = langfuse.get_prompt("test-prompt")
    return Command(goto="__end__", update={"messages": [prompt.prompt]})

# Build Graph
graph = StateGraph(MessagesState)
graph.add_node("thinker", thinker)
graph.add_edge(START, "thinker")
agent = graph.compile()

# Start flow
config = {"callbacks": [langfuse_handler]}
response = agent.invoke({"messages": [user_input]}, config)
# ❌ 结果：prompt 和 trace 是分离的，无法关联
```

### 为什么错？

**核心问题：LangGraph 的自动生成 trace 不会自动链接 prompt**

1. **上下文丢失**：
   - `langfuse.get_prompt()` 在 LangGraph node 内部调用
   - `langfuse_context` 不知道当前有 CallbackHandler 创建的 trace
   - 导致 prompt 获取创建了新的独立 trace

2. **Prompt 链接缺失**：
   - Langfuse 无法将 prompt 版本与 LLM 调用关联
   - 无法追踪哪个 prompt 版本产生了哪个输出
   - 评估和 A/B 测试失效

3. **Span 嵌套错误**：
   - 自动生成的 span 可能不在正确的父 span 下
   - 追踪树结构混乱

### 为什么人们容易这样错？

**心理/认知原因**：

1. **自动化的期望**：
   - LangChain 的 CallbackHandler 通常"自动工作"
   - 期望 LangGraph 也能自动处理所有追踪

2. **上下文传播的隐式性**：
   - 不理解 Python 的上下文变量（contextvars）机制
   - 不知道 LangGraph node 内部需要显式传播上下文

3. **文档不足**：
   - LangGraph + Langfuse 的集成文档较少
   - 容易忽略 `@observe` 装饰器的作用

### 正确理解

**使用 `@observe` 装饰器显式链接 prompt 和 trace**

```python
from typing import Literal
from dotenv import load_dotenv
from langgraph.types import Command
from langgraph.graph import StateGraph, MessagesState, START
from langfuse import Langfuse
from langfuse.callback import CallbackHandler
from langfuse.decorators import observe, langfuse_context

load_dotenv()

# Create LangFuse classes
langfuse_handler = CallbackHandler()
langfuse = Langfuse()

# ✅ 使用 @observe 装饰器链接 prompt
@observe(as_type="generation")
def get_prompt(name: str) -> str:
    """Get the prompt from LangFuse by name"""
    langfuse_prompt = langfuse.get_prompt(name)

    # ✅ 更新当前 observation，链接 prompt
    langfuse_context.update_current_observation(
        prompt=langfuse_prompt,
    )

    return langfuse_prompt.prompt

# Create LangGraph node
def thinker(state: MessagesState) -> Command[Literal["__end__"]]:
    """LangGraph node"""
    # ✅ 调用带 @observe 的函数
    prompt = get_prompt("test-prompt")
    return Command(goto="__end__", update={"messages": [prompt]})

# Build Graph
graph = StateGraph(MessagesState)
graph.add_node("thinker", thinker)
graph.add_edge(START, "thinker")
agent = graph.compile()

# Start flow
config = {"callbacks": [langfuse_handler]}
user_input = {"role": "user", "content": input("\n[You] ")}
response = agent.invoke({"messages": [user_input], "next": []}, config)
print(f"\n[Agent]\n{response['messages'][-1].content}\n")
```

**关键要点**：

1. **使用 `@observe` 装饰器**：
   - 将 prompt 获取包装在 observation 中
   - 自动链接到当前 trace

2. **调用 `langfuse_context.update_current_observation()`**：
   - 显式更新 observation 的 prompt 信息
   - 确保 prompt 版本被记录

3. **分离 prompt 获取逻辑**：
   - 将 prompt 获取封装成独立函数
   - 便于添加 `@observe` 装饰器

**高级用法：使用 `start_as_current_span`**

```python
from langfuse import get_client

def thinker_with_span(state: MessagesState):
    """LangGraph node with explicit span management"""
    langfuse = get_client()

    # ✅ 显式创建 span
    with langfuse.start_as_current_observation(
        as_type="span",
        name="thinker-node"
    ) as span:
        # 获取 prompt
        prompt = langfuse.get_prompt("test-prompt")

        # ✅ 更新 span 的 prompt 信息
        span.update(prompt=prompt)

        # 执行逻辑
        result = process_with_prompt(prompt)

        return Command(goto="__end__", update={"messages": [result]})
```

## 误区3：工具调用数据获取 ❌

### 错误观点

**"默认的回调方法会自动返回工具参数，不需要额外处理"**

```python
# ❌ 错误做法：期望默认回调返回工具参数
from langchain.callbacks import BaseCallbackHandler

class MyCallbackHandler(BaseCallbackHandler):
    def on_tool_start(self, serialized, input_str, **kwargs):
        # ❌ 期望 input_str 包含工具参数
        print(f"Tool started: {input_str}")
        # 实际上 input_str 可能只是字符串，不包含结构化参数

# 使用 Agent
agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    callbacks=[MyCallbackHandler()]
)

# ❌ 结果：无法获取工具的实际参数（如函数参数）
```

### 为什么错？

**核心问题：默认的 `on_tool_start` 实现不返回工具的结构化参数**

1. **参数格式不一致**：
   - `input_str` 通常是字符串形式的输入
   - 对于 OpenAI function call，实际参数在 `kwargs` 中
   - 需要手动从 `kwargs` 提取

2. **Agent 类型差异**：
   - 不同类型的 Agent（OpenAI function call、ReAct）参数传递方式不同
   - 默认实现无法处理所有情况

3. **序列化信息不完整**：
   - `serialized` 参数包含工具的元数据，但不包含调用参数
   - 需要从其他地方获取

### 为什么人们容易这样错？

**心理/认知原因**：

1. **API 命名的误导**：
   - `on_tool_start(serialized, input_str, **kwargs)` 的命名暗示 `input_str` 包含所有输入
   - 实际上关键信息在 `kwargs` 中

2. **文档不足**：
   - 官方文档对 `kwargs` 的内容说明不够详细
   - 需要查看源码或实验才能发现

3. **类型多样性**：
   - 不同 Agent 类型的 `kwargs` 结构不同
   - 容易忽略特定类型的处理

### 正确理解

**重写 `on_tool_start` 方法，从 `kwargs` 中提取工具参数**

```python
from typing import Any, Dict
from langchain.callbacks import BaseCallbackHandler
from langchain.agents import AgentExecutor, OpenAIFunctionsAgent, tool
from langchain_openai import ChatOpenAI

class ToolCallbackHandler(BaseCallbackHandler):
    """自定义回调处理器，追踪工具调用参数"""

    def on_tool_start(
        self,
        serialized: Dict[str, Any],
        input_str: str,
        **kwargs: Any
    ) -> None:
        """Tool 开始时触发"""
        # ✅ 从 serialized 获取工具名称
        tool_name = serialized.get("name", "unknown")

        # ✅ 从 kwargs 获取工具参数（OpenAI function call）
        tool_input = kwargs.get("tool_input", {})

        # ✅ 获取运行 ID
        run_id = kwargs.get("run_id")

        print(f"🔧 Tool Started: {tool_name}")
        print(f"   Run ID: {run_id}")
        print(f"   Input String: {input_str}")
        print(f"   Tool Input: {tool_input}")
        print(f"   Full kwargs keys: {list(kwargs.keys())}")

    def on_tool_end(
        self,
        output: str,
        **kwargs: Any
    ) -> None:
        """Tool 结束时触发"""
        run_id = kwargs.get("run_id")
        print(f"✅ Tool Ended: {run_id}")
        print(f"   Output: {output[:100]}...")  # 只显示前100字符

    def on_tool_error(
        self,
        error: BaseException,
        **kwargs: Any
    ) -> None:
        """Tool 错误时触发"""
        run_id = kwargs.get("run_id")
        print(f"❌ Tool Error: {run_id}")
        print(f"   Error: {error}")

# 定义工具
@tool
def get_word_length(word: str) -> int:
    """Returns the length of a word."""
    return len(word)

@tool
def multiply(a: int, b: int) -> int:
    """Multiply two numbers."""
    return a * b

tools = [get_word_length, multiply]

# 创建 Agent
llm = ChatOpenAI(temperature=0, model="gpt-4")
handler = ToolCallbackHandler()

from langchain.messages import SystemMessage

prompt = OpenAIFunctionsAgent.create_prompt(
    system_message=SystemMessage(
        content="You are a helpful assistant."
    )
)

agent = OpenAIFunctionsAgent(llm=llm, tools=tools, prompt=prompt)

agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    verbose=True,
    callbacks=[handler]  # ✅ 传入自定义回调
)

# 执行
result = agent_executor.run(
    "What is the length of the word 'education'? Then multiply it by 3."
)
print(f"\nFinal Result: {result}")
```

**输出示例**：

```

   Run ID: 12345678-1234-1234-1234-123456789abc
   Input String: education
   Tool Input: {'word': 'education'}
   Full kwargs keys: ['run_id', 'parent_run_id', 'tags', 'metadata', 'tool_input']
✅ Tool Ended: 12345678-1234-1234-1234-123456789abc
   Output: 9

🔧 Tool Started: multiply
   Run ID: 87654321-4321-4321-4321-cba987654321
   Input String: 9, 3
   Tool Input: {'a': 9, 'b': 3}
   Full kwargs keys: ['run_id', 'parent_run_id', 'tags', 'metadata', 'tool_input']

```

**关键要点**：

1. **从 `kwargs` 获取参数**：
   - `kwargs.get("tool_input", {})` 包含工具的实际参数
   - 对于 OpenAI function call，这是一个字典

2. **使用 `run_id` 追踪**：
   - `kwargs.get("run_id")` 唯一标识每次工具调用
   - 可以关联 `on_tool_start` 和 `on_tool_end`

3. **处理不同 Agent 类型**：
   - OpenAI function call：参数在 `tool_input`
   - ReAct Agent：参数可能在 `input_str`
   - 需要根据 Agent 类型调整

**高级用法：记录工具调用到数据库**

```python
import json
from datetime import datetime

class ToolTrackingHandler(BaseCallbackHandler):
    """追踪工具调用并记录到数据库"""

    def __init__(self):
        self.tool_calls = []

    def on_tool_start(self, serialized, input_str, **kwargs):
        tool_call = {
            "tool_name": serialized.get("name"),
            "run_id": str(kwargs.get("run_id")),
            "input": kwargs.get("tool_input", {}),
            "start_time": datetime.now().isoformat(),
            "status": "started"
        }
        self.tool_calls.append(tool_call)

    def on_tool_end(self, output, **kwargs):
        run_id = str(kwargs.get("run_id"))
        # 找到对应的工具调用
        for call in self.tool_calls:
            if call["run_id"] == run_id:
                call["output"] = output
                call["end_time"] = datetime.now().isoformat()
                call["status"] = "completed"
                break

    def on_tool_error(self, error, **kwargs):
        run_id = str(kwargs.get("run_id"))
        for call in self.tool_calls:
            if call["run_id"] == run_id:
                call["error"] = str(error)
                call["end_time"] = datetime.now().isoformat()
                call["status"] = "error"
                break

    def get_report(self):
        """生成工具调用报告"""
        return json.dumps(self.tool_calls, indent=2)

# 使用
handler = ToolTrackingHandler()
agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    callbacks=[handler]
)

result = agent_executor.run("What is 5 times 7?")

# 获取报告
print("\n=== Tool Call Report ===")
print(handler.get_report())
```

## 反直觉点总结

### 核心要点对比表

| 误区 | 错误观点 | 正确理解 | 关键操作 |
|------|---------|---------|---------|
| **并发环境回调管理** | 回调处理器可以重用 | 每个请求新建实例 | `CallbackHandler()` + `last_trace_id` + `flush()` |
| **LangGraph trace 链接** | 自动链接 prompt | 需要显式链接 | `@observe` + `langfuse_context.update_current_observation()` |
| **工具调用数据获取** | 默认返回参数 | 需要从 kwargs 提取 | `kwargs.get("tool_input", {})` |

### 记忆技巧

**并发环境**：
- **一请求一实例**：每个请求都是独立的故事，需要独立的记录员
- **last_trace_id**：从记录员那里拿 ID，不是从总部
- **flush()**：短命环境要及时保存，不然数据丢失

**LangGraph trace**：
- **@observe 装饰器**：给 prompt 获取加上"追踪标签"
- **显式链接**：LangGraph 不会读心术，要明确告诉它

**工具调用**：
- **kwargs 是宝藏**：真正的参数藏在 kwargs 里
- **tool_input 字典**：OpenAI function call 的参数在这里
