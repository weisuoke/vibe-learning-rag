# 02_第一性原理

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而不是通过类比或经验来理解。

在物理学中，第一性原理意味着从基本的物理定律推导出所有现象。在软件工程中，第一性原理意味着从最基础的概念出发，理解系统的本质。

---

## 边与条件路由的第一性原理

### 1. 最基础的定义

**边 = 有向连接 + 执行顺序**

仅此而已！没有更基础的了。

**拆解**：
- **有向连接**：从节点 A 指向节点 B 的箭头
- **执行顺序**：A 完成后，B 才能开始

**条件边 = 边 + 决策函数**

**拆解**：
- **边**：基础的有向连接
- **决策函数**：根据状态值动态选择目标节点

### 2. 为什么需要边？

**核心问题：如何控制多个操作的执行顺序？**

#### 问题场景

假设你有3个操作：
1. 从数据库检索文档
2. 评估文档质量
3. 生成答案

**没有边的世界**：
- 所有操作同时执行（混乱）
- 无法保证顺序（检索还没完成就开始评估）
- 无法根据结果做决策（文档质量差也继续生成）

**有边的世界**：
- 明确的执行顺序（检索 → 评估 → 生成）
- 可靠的依赖关系（评估等待检索完成）
- 动态的决策能力（根据质量选择路径）

#### 根本原因

**计算机程序 = 数据 + 控制流**

- **数据**：节点处理的状态
- **控制流**：边决定的执行顺序

边是控制流的具体实现，没有边就没有控制流，程序就无法按预期执行。

### 3. 边的三层价值

#### 价值1：顺序保证（Ordering Guarantee）

**本质**：确保操作按正确的顺序执行

**示例**：
```python
# 没有边：无法保证顺序
async def process():
    task1 = retrieve_docs()
    task2 = grade_docs()
    task3 = generate_answer()
    # 三个任务可能同时执行，顺序混乱

# 有边：明确的顺序
graph.add_edge("retrieve", "grade")
graph.add_edge("grade", "generate")
# retrieve 完成 → grade 开始 → generate 开始
```

**为什么重要**：
- 避免数据竞争（race condition）
- 确保依赖关系（dependency）
- 保证结果正确性（correctness）

[来源: reference/source_edges_01_add_edge.md]

#### 价值2：决策能力（Decision Making）

**本质**：根据运行时状态动态选择执行路径

**示例**：
```python
def route_by_quality(state):
    """根据文档质量决定下一步"""
    if state["doc_quality"] > 0.8:
        return "generate"  # 质量好，直接生成
    else:
        return "rewrite_query"  # 质量差，改写查询

graph.add_conditional_edges(
    "grade",
    route_by_quality,
    {
        "generate": "generate",
        "rewrite_query": "rewrite_query"
    }
)
```

**为什么重要**：
- 适应不同场景（adaptability）
- 优化执行路径（optimization）
- 实现智能决策（intelligence）

[来源: reference/context7_langgraph_01_edges_routing.md]

#### 价值3：模块化与可维护性（Modularity & Maintainability）

**本质**：将复杂流程拆分成独立的节点和边

**示例**：
```python
# 没有边：所有逻辑耦合在一起
def monolithic_rag(query):
    docs = retrieve(query)
    if quality(docs) > 0.8:
        return generate(docs, query)
    else:
        new_query = rewrite(query)
        docs = retrieve(new_query)
        return generate(docs, new_query)

# 有边：模块化设计
graph.add_node("retrieve", retrieve)
graph.add_node("grade", grade_docs)
graph.add_node("generate", generate)
graph.add_node("rewrite", rewrite_query)

graph.add_edge("retrieve", "grade")
graph.add_conditional_edges("grade", route_func, {...})
graph.add_edge("rewrite", "retrieve")
```

**为什么重要**：
- 易于理解（readability）
- 易于测试（testability）
- 易于修改（maintainability）

### 4. 从第一性原理推导 LangGraph 的边设计

**推理链：**

```
1. 【前提】工作流 = 多个操作 + 执行顺序
   ↓
2. 【推导】需要一种机制来表示"操作之间的连接"
   → 引入"边"（Edge）的概念
   ↓
3. 【推导】有些连接是固定的（A 总是到 B）
   → 引入"普通边"（add_edge）
   ↓
4. 【推导】有些连接需要根据状态动态决定（A 可能到 B 或 C）
   → 引入"条件边"（add_conditional_edges）
   ↓
5. 【推导】条件边需要一个函数来做决策
   → 引入"路由函数"（routing function）
   ↓
6. 【推导】路由函数的返回值需要映射到实际节点
   → 引入"路径映射"（path_map）
   ↓
7. 【推导】为了类型安全和可视化，需要明确可能的路由目标
   → 引入"类型提示"（Literal type hints）
   ↓
8. 【推导】有些场景需要等待多个节点完成
   → 引入"多起始节点边"（waiting edges）
   ↓
9. 【推导】需要验证边的合法性（避免无效连接）
   → 引入"编译时验证"（compile-time validation）
   ↓
10. 【结论】LangGraph 的边系统 = 普通边 + 条件边 + 路由函数 + 路径映射 + 类型提示 + 验证机制
```

[来源: reference/source_edges_01_add_edge.md, reference/source_edges_02_add_conditional_edges.md]

### 5. 从第一性原理理解边的存储结构

**推理链：**

```
1. 【前提】边需要被存储和管理
   ↓
2. 【推导】普通边是固定连接，可以用简单的元组表示
   → self.edges: set[tuple[str, str]]
   → 格式：(start_key, end_key)
   ↓
3. 【推导】条件边包含决策逻辑，需要更复杂的结构
   → self.branches: defaultdict[str, dict[str, BranchSpec]]
   → 格式：{起始节点: {条件名: BranchSpec对象}}
   ↓
4. 【推导】多起始节点边需要等待所有节点完成
   → self.waiting_edges: set[tuple[tuple[str, ...], str]]
   → 格式：((start1, start2, ...), end_key)
   ↓
5. 【结论】三种存储结构对应三种边的语义
```

[来源: reference/source_edges_01_add_edge.md]

### 6. 从第一性原理理解条件边的执行机制

**推理链：**

```
1. 【前提】条件边需要在运行时动态决定目标节点
   ↓
2. 【推导】需要一个函数来做决策
   → 路由函数：接受状态，返回目标节点名称
   ↓
3. 【推导】路由函数的返回值可能是字符串或多个字符串
   → 返回类型：Hashable | Sequence[Hashable]
   ↓
4. 【推导】返回值需要映射到实际节点名称
   → path_map: dict[Hashable, str] | list[str] | None
   ↓
5. 【推导】路由函数需要被转换为可执行的对象
   → coerce_to_runnable(path, name=None, trace=True)
   ↓
6. 【推导】条件边需要被封装成规范对象
   → BranchSpec.from_path(path, path_map, infer_schema=True)
   ↓
7. 【结论】条件边的执行 = 路由函数 + 路径映射 + BranchSpec 封装
```

[来源: reference/source_edges_02_add_conditional_edges.md]

---

### 7. 从第一性原理理解边的验证规则

**推理链：**

```
1. 【前提】边连接节点，需要确保节点存在
   ↓
2. 【推导】起始节点必须已经添加（除了 START）
   → if start not in self.nodes: raise ValueError
   ↓
3. 【推导】结束节点必须已经添加（除了 END）
   → if end_key != END and end_key not in self.nodes: raise ValueError
   ↓
4. 【推导】END 不能作为起始节点（没有后续节点）
   → if start_key == END: raise ValueError
   ↓
5. 【推导】START 不能作为结束节点（不能被其他节点指向）
   → if end_key == START: raise ValueError
   ↓
6. 【推导】非 StateGraph 不允许重复边（避免状态冲突）
   → if not hasattr(self, "channels") and start_key in edges: raise ValueError
   ↓
7. 【结论】验证规则确保图的结构合法性
```

[来源: reference/source_edges_01_add_edge.md]

### 8. 从第一性原理理解路由函数的设计

**推理链：**

```
1. 【前提】路由函数需要根据状态做决策
   ↓
2. 【推导】路由函数必须接受状态作为输入
   → def route_func(state: State) -> ...
   ↓
3. 【推导】路由函数必须返回目标节点名称
   → return "node_name" 或 ["node1", "node2"]
   ↓
4. 【推导】为了类型安全，应该使用类型提示
   → -> Literal["node_a", "node_b", END]
   ↓
5. 【推导】路由逻辑应该简单清晰
   → 避免复杂的嵌套条件
   ↓
6. 【推导】路由函数应该处理边界情况
   → 检查缺失值、异常值
   ↓
7. 【结论】路由函数 = 状态输入 + 清晰逻辑 + 类型提示 + 边界处理
```

[来源: reference/context7_langgraph_01_edges_routing.md]

## 第一性原理的实际应用

### 应用1：设计 RAG 工作流

**从第一性原理出发**：

1. **目标**：构建一个自适应的 RAG 系统
2. **拆解**：
   - 检索文档（retrieve）
   - 评估质量（grade）
   - 生成答案（generate）
   - 改写查询（rewrite）
3. **连接**：
   - 固定连接：retrieve → grade（总是需要评估）
   - 条件连接：grade → generate/rewrite（根据质量决定）
   - 循环连接：rewrite → retrieve（重新检索）
4. **实现**：

```python
from langgraph.graph import StateGraph, START, END
from typing import Literal, TypedDict

class State(TypedDict):
    query: str
    documents: list[str]
    doc_quality: float
    answer: str
    retry_count: int

def retrieve(state: State) -> dict:
    """检索文档"""
    # 模拟向量检索
    docs = ["doc1", "doc2", "doc3"]
    return {"documents": docs}

def grade_documents(state: State) -> dict:
    """评估文档质量"""
    # 模拟质量评分
    quality = 0.85
    return {"doc_quality": quality}

def generate(state: State) -> dict:
    """生成答案"""
    answer = f"Answer based on {len(state['documents'])} documents"
    return {"answer": answer}

def rewrite_query(state: State) -> dict:
    """改写查询"""
    new_query = f"Rewritten: {state['query']}"
    return {"query": new_query, "retry_count": state.get("retry_count", 0) + 1}

def route_after_grade(state: State) -> Literal["generate", "rewrite", END]:
    """根据文档质量路由"""
    # 质量好，直接生成
    if state["doc_quality"] > 0.8:
        return "generate"

    # 重试次数过多，结束
    if state.get("retry_count", 0) >= 3:
        return END

    # 质量差，改写查询
    return "rewrite"

# 构建图
graph = StateGraph(State)

# 添加节点
graph.add_node("retrieve", retrieve)
graph.add_node("grade", grade_documents)
graph.add_node("generate", generate)
graph.add_node("rewrite", rewrite_query)

# 添加边
graph.add_edge(START, "retrieve")
graph.add_edge("retrieve", "grade")

# 条件边：根据质量决定路径
graph.add_conditional_edges(
    "grade",
    route_after_grade,
    {
        "generate": "generate",
        "rewrite": "rewrite",
        END: END
    }
)

# 循环边：改写后重新检索
graph.add_edge("rewrite", "retrieve")
graph.add_edge("generate", END)

# 编译
app = graph.compile()

# 测试
result = app.invoke({
    "query": "What is LangGraph?",
    "documents": [],
    "doc_quality": 0.0,
    "answer": "",
    "retry_count": 0
})
print(result)
```

**第一性原理体现**：
- ✅ 从基础概念（连接 + 顺序）出发
- ✅ 明确每个节点的职责
- ✅ 使用普通边表示固定流程
- ✅ 使用条件边实现动态决策
- ✅ 添加循环边支持重试机制
- ✅ 使用类型提示确保类型安全

[来源: reference/context7_langgraph_01_edges_routing.md]

### 应用2：设计审批流程

**从第一性原理出发**：

1. **目标**：构建一个多级审批系统
2. **拆解**：
   - 提交申请（submit）
   - 初审（initial_review）
   - 终审（final_review）
   - 批准（approve）
   - 拒绝（reject）
   - 人工审核（human_review）
3. **连接**：
   - 固定连接：submit → initial_review
   - 条件连接：initial_review → approve/reject/final_review
   - 条件连接：final_review → approve/reject/human_review
4. **实现**：

```python
from typing import Literal, TypedDict

class ApprovalState(TypedDict):
    application: dict
    initial_score: float
    final_score: float
    status: str
    reason: str

def submit_application(state: ApprovalState) -> dict:
    """提交申请"""
    return {"status": "submitted"}

def initial_review(state: ApprovalState) -> dict:
    """初审"""
    # 模拟评分
    score = 0.75
    return {"initial_score": score}

def final_review(state: ApprovalState) -> dict:
    """终审"""
    # 模拟深度评分
    score = 0.65
    return {"final_score": score}

def approve(state: ApprovalState) -> dict:
    """批准"""
    return {"status": "approved", "reason": "Meets all criteria"}

def reject(state: ApprovalState) -> dict:
    """拒绝"""
    return {"status": "rejected", "reason": "Does not meet criteria"}

def human_review(state: ApprovalState) -> dict:
    """人工审核"""
    return {"status": "pending_human"}

def route_after_initial(state: ApprovalState) -> Literal["approve", "reject", "final_review"]:
    """初审后路由"""
    score = state["initial_score"]
    if score > 0.9:
        return "approve"  # 分数很高，直接批准
    elif score < 0.3:
        return "reject"   # 分数很低，直接拒绝
    else:
        return "final_review"  # 需要终审

def route_after_final(state: ApprovalState) -> Literal["approve", "reject", "human_review"]:
    """终审后路由"""
    score = state["final_score"]
    if score > 0.7:
        return "approve"
    elif score < 0.4:
        return "reject"
    else:
        return "human_review"  # 边界情况，需要人工审核

# 构建图
graph = StateGraph(ApprovalState)

# 添加节点
graph.add_node("submit", submit_application)
graph.add_node("initial_review", initial_review)
graph.add_node("final_review", final_review)
graph.add_node("approve", approve)
graph.add_node("reject", reject)
graph.add_node("human_review", human_review)

# 添加边
graph.add_edge(START, "submit")
graph.add_edge("submit", "initial_review")

# 初审后的条件路由
graph.add_conditional_edges(
    "initial_review",
    route_after_initial,
    {
        "approve": "approve",
        "reject": "reject",
        "final_review": "final_review"
    }
)

# 终审后的条件路由
graph.add_conditional_edges(
    "final_review",
    route_after_final,
    {
        "approve": "approve",
        "reject": "reject",
        "human_review": "human_review"
    }
)

# 结束边
graph.add_edge("approve", END)
graph.add_edge("reject", END)
graph.add_edge("human_review", END)

# 编译
app = graph.compile()
```

**第一性原理体现**：
- ✅ 多层决策（初审 → 终审）
- ✅ 清晰的路由逻辑（分数阈值）
- ✅ 人机协作（人工审核节点）
- ✅ 类型安全（Literal 类型提示）

## 核心洞察

### 洞察1：边是控制流的具体化

**传统编程**：
```python
if condition:
    func_a()
else:
    func_b()
```

**LangGraph**：
```python
graph.add_conditional_edges(
    "check",
    lambda state: "a" if state["condition"] else "b",
    {"a": "func_a", "b": "func_b"}
)
```

**区别**：
- 传统编程：控制流隐藏在代码中
- LangGraph：控制流显式表示为边

**优势**：
- 可视化（能看到流程图）
- 可分析（能分析执行路径）
- 可优化（能优化执行顺序）

### 洞察2：条件边是策略模式的应用

**策略模式**：定义一系列算法，把它们封装起来，并且使它们可以互相替换。

**条件边**：定义一系列路由策略，把它们封装在路由函数中，根据状态选择策略。

### 洞察3：边的组合产生复杂行为

**简单边 + 简单边 = 复杂工作流**

**示例**：
- 线性流程（A → B → C）
- 分支流程（A → B/C）
- 循环流程（A → B → A）
- 并行流程（A → B + C → D）
- 多层决策（A → B → C/D → E/F）

**关键**：通过组合简单的边，构建复杂的工作流。
