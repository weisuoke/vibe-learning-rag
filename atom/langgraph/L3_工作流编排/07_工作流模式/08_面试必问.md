## 问题1："LangGraph 有哪些工作流模式？如何选择？"

### 普通回答（❌ 不出彩）

"LangGraph 支持顺序执行、并行执行和条件分支。根据需求选择就行。"

### 出彩回答（✅ 推荐）

> **LangGraph 官方定义了六大工作流模式，我按复杂度递增来说：**
>
> 1. **顺序链式（Prompt Chaining）**：最基础，A→B→C 线性执行，适合有严格先后依赖的任务，比如 RAG 的"检索→重排→生成"管道。
>
> 2. **路由（Routing）**：根据输入动态选路径，用 `add_conditional_edges` 实现。适合需要分类处理的场景，比如客服分流。
>
> 3. **并行（Parallelization）**：Fan-out/Fan-in 模式，同一 superstep 内的节点自动并行。需要 `Annotated[list, operator.add]` 做 reducer 合并结果。
>
> 4. **Map-Reduce**：用 `Send` API 在运行时动态创建并行任务。和普通并行的关键区别是分支数量不是写死的，而是数据驱动的。
>
> 5. **编排者-工作者（Orchestrator-Worker）**：LLM 动态分解任务 + 并行执行。本质是"智能版 Map-Reduce"，Splitter 由 LLM 驱动。
>
> 6. **评估者-优化者（Evaluator-Optimizer）**：生成→评估→反馈循环。关键是结构化反馈和终止条件设计。
>
> **选择原则很简单：从最简单的模式开始，只有当简单模式明显不够时才升级。** 实际项目中，80% 的场景用顺序链式 + 路由就够了。

### 为什么这个回答出彩？

1. ✅ 覆盖了全部六大模式，展示知识广度
2. ✅ 按复杂度递增排列，体现系统性思考
3. ✅ 每个模式都有具体的 API 和场景，不是空谈
4. ✅ 最后给出选择原则，展示工程判断力

## 问题2："Send API 和普通并行有什么区别？什么时候用 Send？"

### 普通回答（❌ 不出彩）

"Send 可以动态创建并行任务，普通并行是固定的。"

### 出彩回答（✅ 推荐）

> **这是一个很好的问题，它们看起来效果一样，但底层机制完全不同：**
>
> **普通并行**是在图编译时就确定了分支数量。比如 `add_edge("a", "b")` + `add_edge("a", "c")`，永远是 2 个并行分支。所有分支共享全局 State，通过 reducer 合并结果。底层是 PULL 类型任务——节点被 channel 更新触发。
>
> **Send API** 是在运行时动态创建任务。`Send("worker", {"item": x})` 为每个数据项创建独立的任务实例。每个 Worker 收到的是独立的输入状态，互不干扰。底层是 PUSH 类型任务——通过 Topic channel 路由。
>
> **选择标准**：
> - 分支数量编译时就知道（比如"同时做情感分析和关键词提取"）→ 普通并行
> - 分支数量取决于运行时数据（比如"对每个文档做处理"）→ Send API
>
> **一个实际例子**：RAG 系统中，"同时查文档库和 FAQ 库"用普通并行（固定 2 个源）；"对用户上传的 N 个文档做向量化"用 Send（文档数量不确定）。

### 为什么这个回答出彩？

1. ✅ 从底层机制（PULL vs PUSH）解释区别，展示深度
2. ✅ 给出明确的选择标准
3. ✅ 用实际 RAG 场景举例，展示应用能力

## 问题3："如何避免评估者-优化者模式的无限循环？"

### 普通回答（❌ 不出彩）

"设置最大循环次数就行了。"

### 出彩回答（✅ 推荐）

> **防止无限循环需要多层保护，不能只靠最大次数：**
>
> 1. **质量阈值**：评估器输出结构化评分，达到阈值就终止。用 `with_structured_output` 确保评分格式可靠。
>
> 2. **最大次数兜底**：即使质量没达标，超过 N 次也强制终止。这是最后一道防线。
>
> 3. **改进停滞检测**：如果连续两次评分没有提升，说明已经到了模型能力上限，继续循环也没用。
>
> 4. **评估器可靠性**：这是最容易被忽略的。如果用同一个 LLM 既生成又评估，容易"自我欺骗"。最好用更强的模型做评估，或者用确定性指标（比如代码能否通过语法检查、测试是否通过）。
>
> ```python
> def should_continue(state):
>     if state["grade"] == "pass":
>         return "done"
>     if state["attempts"] >= 3:
>         return "done"  # 最大次数兜底
>     if state.get("prev_score") and state["score"] <= state["prev_score"]:
>         return "done"  # 改进停滞
>     return "retry"
> ```
>
> **核心原则：评估器要比生成器更可靠。能用确定性检查就不用 LLM 评估。**

### 为什么这个回答出彩？

1. ✅ 给出多层保护策略，不是单一方案
2. ✅ 指出"评估器可靠性"这个容易被忽略的关键点
3. ✅ 附带代码示例，展示实操能力
4. ✅ 总结出核心原则
