## 误区清单

### 误区 1：异步生成器可以作为节点函数

**错误认知**：
```python
# ❌ 错误：认为异步生成器可以作为节点函数
async def my_node(state: State):
    async for item in some_async_generator():
        yield {"item": item}
```

**为什么会产生这个误区**：
1. **Python 语法支持**：Python 支持 `async function*` 语法，看起来很自然
2. **流式输出需求**：开发者希望实现流式输出，认为生成器是最佳选择
3. **同步生成器经验**：在其他框架中使用过同步生成器，认为异步版本也应该工作

[来源: reference/search_节点函数约定_01.md]

**真实案例**：
```
GitHub Issue #1158: StateGraph中使用async function*作为节点时，函数体不执行且无yield流式，错误处理失效
```

**正确理解**：
- LangGraph 的节点函数 **不支持异步生成器**
- 函数体可能不执行，yield 流式输出失效
- 错误处理机制失效

**正确做法**：
```python
# ✅ 正确：使用普通异步函数
async def my_node(state: State) -> dict:
    result = await some_async_api()
    return {"result": result}

# ✅ 正确：如果需要流式输出，使用 astream
async for event in graph.astream(input):
    print(event)
```

[来源: reference/search_节点函数约定_01.md]

### 误区 2：节点函数必须返回完整状态

**错误认知**：
```python
# ❌ 错误：认为必须返回完整状态
def my_node(state: State) -> State:
    # 复制整个状态
    new_state = state.copy()
    new_state["x"] = state["x"] + 1
    new_state["y"] = state["y"] * 2
    new_state["z"] = state["z"]  # 即使没有修改也要返回
    return new_state
```

**为什么会产生这个误区**：
1. **传统函数式编程经验**：在函数式编程中，通常返回完整的新状态
2. **类型注解误导**：看到 `-> State` 的类型注解，认为必须返回完整状态
3. **不了解 Reducer 机制**：不知道 LangGraph 会自动合并状态

[来源: reference/source_节点函数约定_01.md, reference/context7_langgraph_01.md]

**正确理解**：
- 节点函数只需返回 **需要更新的字段**
- LangGraph 会自动将返回值与现有状态合并
- 这是 **部分状态更新** 机制

**正确做法**：
```python
# ✅ 正确：只返回需要更新的字段
def my_node(state: State) -> dict:
    return {"x": state["x"] + 1}  # 只更新 x，其他字段保持不变
```

[来源: reference/context7_langgraph_01.md]

**源码证据**：
```python
# state.py:1206-1242
def _get_updates(input: None | dict | Any) -> Sequence[tuple[str, Any]] | None:
    if input is None:
        return None
    elif isinstance(input, dict):
        return [(k, v) for k, v in input.items() if k in output_keys]
    # ...
```

[来源: reference/source_节点函数约定_01.md]

### 误区 3：所有异常都会自动重试

**错误认知**：
```python
# ❌ 错误：认为所有异常都会自动重试
async def my_node(state: State) -> dict:
    result = await unreliable_api()  # 可能抛出异常
    return {"result": result}

# 认为异常会自动重试，不需要配置
builder.add_node("api_call", my_node)
```

**为什么会产生这个误区**：
1. **框架自动化期望**：认为现代框架应该自动处理重试
2. **不了解 RetryPolicy**：不知道需要显式配置重试策略
3. **误解错误处理机制**：认为 try-except 就足够了

[来源: reference/search_节点函数约定_01.md]

**真实案例**：
```
GitHub Issue #6170: 社区需求 - 为节点提供重试策略、可配置错误处理钩子或中间件，改善异步节点异常管理
```

**正确理解**：
- 默认情况下，异常 **不会自动重试**
- 需要显式配置 `RetryPolicy`
- 可以自定义重试条件

**正确做法**：
```python
# ✅ 正确：配置 RetryPolicy
from langgraph.types import RetryPolicy

builder.add_node(
    "api_call",
    my_node,
    retry_policy=RetryPolicy(
        max_attempts=3,
        initial_interval=1.0,
        backoff_factor=2.0,
        retry_on=lambda e: isinstance(e, (ConnectionError, TimeoutError))
    )
)
```

[来源: reference/context7_langgraph_01.md, reference/source_节点函数约定_02.md]

### 误区 4：streamEvents 中的错误会自动传播

**错误认知**：
```python
# ❌ 错误：认为 try-catch 在外层就能捕获所有错误
try:
    async for event in graph.astream_events(input):
        process_event(event)
except Exception as e:
    handle_error(e)  # 认为这里能捕获所有错误
```

**为什么会产生这个误区**：
1. **传统异步编程经验**：在普通异步代码中，外层 try-catch 能捕获内层异常
2. **不了解流式错误处理**：不知道流式执行中的错误处理机制不同
3. **ToolNode 错误被忽略**：ToolNode 抛出的错误未通过 on_tool_end 通知

[来源: reference/search_节点函数约定_01.md]

**真实案例**：
```
GitHub Issue #1831: ToolNode抛出错误未通过on_tool_end通知，影响异步流式错误处理机制
GitHub Issue #812: try-catch内for await循环处理streamEvents失效，需在循环内处理异步节点错误
```

**正确理解**：
- streamEvents 中的错误 **不会自动传播** 到外层
- 需要在循环 **内部** 处理错误
- 使用 error 事件监听

**正确做法**：
```python
# ✅ 正确：在循环内处理错误
async for event in graph.astream_events(input):
    try:
        if event["event"] == "on_chain_error":
            handle_error(event["data"])
        else:
            process_event(event)
    except Exception as e:
        logger.error(f"Error in stream: {e}")
```

[来源: reference/search_节点函数约定_01.md]

### 误区 5：CancelledError 会被正常捕获

**错误认知**：
```python
# ❌ 错误：认为 CancelledError 会被 except Exception 捕获
async def my_node(state: State) -> dict:
    try:
        result = await some_async_api()
        return {"result": result}
    except Exception as e:  # 认为这里能捕获 CancelledError
        return {"error": str(e)}
```

**为什么会产生这个误区**：
1. **不了解 CancelledError 的特殊性**：CancelledError 继承自 BaseException，不是 Exception
2. **异步取消机制不熟悉**：不知道异步任务取消的特殊处理
3. **工具消息历史无效状态**：取消错误导致状态不一致

[来源: reference/search_节点函数约定_01.md]

**真实案例**：
```
GitHub Issue #6726: ToolNode未捕获CancelledError，异步取消错误绕过异常捕获，导致工具消息历史无效状态
```

**正确理解**：
- `CancelledError` 继承自 `BaseException`，不是 `Exception`
- 需要 **单独处理** CancelledError
- 取消后需要重新抛出以确保正确取消

**正确做法**：
```python
# ✅ 正确：单独处理 CancelledError
import asyncio

async def my_node(state: State) -> dict:
    try:
        result = await some_async_api()
        return {"result": result}
    except asyncio.CancelledError:
        # 特殊处理取消错误
        logger.info("Task cancelled")
        raise  # 重新抛出以确保正确取消
    except Exception as e:
        # 处理其他错误
        return {"error": str(e)}
```

[来源: reference/search_节点函数约定_01.md]

### 误区 6：Command 指向无效节点会静默失败

**错误认知**：
```python
# ❌ 错误：认为 Command 指向无效节点会静默失败
def my_node(state: State) -> Command:
    if state["x"] > 10:
        return Command(goto="non_existent_node")  # 认为这里会静默失败
    return Command(goto="next")
```

**为什么会产生这个误区**：
1. **早期版本错误提示不明显**：旧版本的错误信息模糊
2. **动态路由经验**：在其他框架中，无效路由可能被忽略
3. **不了解 InvalidUpdateError**：不知道会抛出明确的错误

[来源: reference/search_节点函数约定_01.md]

**真实案例**：
```
GitHub Issue #5556: Command指向无效节点时错误模糊，已修复为清晰InvalidUpdateError消息
```

**正确理解**：
- Command 指向无效节点会抛出 `InvalidUpdateError`
- 错误信息包含具体的节点名称和原因
- 这是 **运行时错误**，不是静默失败

**正确做法**：
```python
# ✅ 正确：使用 Literal 类型注解确保类型安全
from typing import Literal

def my_node(state: State) -> Command[Literal["next", "end"]]:
    if state["x"] > 10:
        return Command(goto="end")  # 类型检查会确保节点存在
    return Command(goto="next")

# ✅ 正确：在添加节点时验证目标节点
builder.add_node(
    "my_node",
    my_node,
    destinations=["next", "end"]  # 显式声明目标节点
)
```

[来源: reference/search_节点函数约定_01.md, reference/source_节点函数约定_01.md]

### 误区 7：不理解自动类型推断机制

**错误认知**：
```python
# ❌ 错误：认为必须显式指定 input_schema
def my_node(state: State) -> dict:
    return {"x": state["x"] + 1}

builder.add_node(
    "my_node",
    my_node,
    input_schema=State  # 认为必须显式指定
)
```

**为什么会产生这个误区**：
1. **不了解类型推断**：不知道 LangGraph 会自动推断类型
2. **显式配置习惯**：习惯显式配置所有参数
3. **类型注解不熟悉**：不知道类型注解的作用

[来源: reference/source_节点函数约定_01.md]

**正确理解**：
- LangGraph 会 **自动推断** 第一个参数的类型作为 input_schema
- 支持 TypedDict 类型推断
- 支持 Command 返回值类型推断

**源码证据**：
```python
# state.py:698-743
if (
    isfunction(action)
    or ismethod(action)
    or ismethod(getattr(action, "__call__", None))
) and (
    hints := get_type_hints(getattr(action, "__call__"))
    or get_type_hints(action)
):
    if input_schema is None:
        first_parameter_name = next(
            iter(
                inspect.signature(
                    cast(FunctionType, action)
                ).parameters.keys()
            )
        )
        if input_hint := hints.get(first_parameter_name):
            if isinstance(input_hint, type) and get_type_hints(input_hint):
                inferred_input_schema = input_hint
```

[来源: reference/source_节点函数约定_01.md]

**正确做法**：
```python
# ✅ 正确：依赖自动类型推断
def my_node(state: State) -> dict:
    return {"x": state["x"] + 1}

builder.add_node("my_node", my_node)  # 不需要显式指定 input_schema
```

### 误区 8：同步函数不能作为节点函数

**错误认知**：
```python
# ❌ 错误：认为必须使用异步函数
async def my_node(state: State) -> dict:
    # 即使是同步操作也要用 async
    result = sync_operation()
    return {"result": result}
```

**为什么会产生这个误区**：
1. **异步框架经验**：在其他异步框架中，所有函数都必须是异步的
2. **不了解自动包装**：不知道 LangGraph 会自动包装同步函数
3. **性能担忧**：担心同步函数会阻塞事件循环

[来源: reference/source_节点函数约定_01.md]

**正确理解**：
- 同步函数 **可以** 作为节点函数
- LangGraph 会自动将同步函数包装成异步执行
- 使用 `run_in_executor` 在线程池中执行同步函数

**源码证据**：
```python
# _call.py:175-192
if is_async_callable(func):
    run = RunnableCallable(
        None, func, name=func.__name__, trace=False, recurse=False
    )
else:
    afunc = functools.update_wrapper(
        functools.partial(run_in_executor, None, func), func
    )
    run = RunnableCallable(
        func,
        afunc,
        name=func.__name__,
        trace=False,
        recurse=False,
    )
```

[来源: reference/source_节点函数约定_01.md]

**正确做法**：
```python
# ✅ 正确：同步函数可以直接使用
def my_sync_node(state: State) -> dict:
    result = sync_operation()
    return {"result": result}

# ✅ 正确：异步函数用于异步操作
async def my_async_node(state: State) -> dict:
    result = await async_operation()
    return {"result": result}

builder.add_node("sync_node", my_sync_node)
builder.add_node("async_node", my_async_node)
```

## 最佳实践建议

### 1. 节点函数定义

```python
# ✅ 推荐：使用类型注解
def my_node(state: State) -> dict:
    return {"x": state["x"] + 1}

# ✅ 推荐：异步操作使用 async
async def my_async_node(state: State) -> dict:
    result = await some_async_api()
    return {"result": result}

# ❌ 不推荐：使用异步生成器
async def my_generator_node(state: State):
    async for item in some_async_generator():
        yield {"item": item}
```

### 2. 错误处理

```python
# ✅ 推荐：配置 RetryPolicy
builder.add_node(
    "api_call",
    my_node,
    retry_policy=RetryPolicy(
        max_attempts=3,
        initial_interval=1.0,
        backoff_factor=2.0,
        retry_on=lambda e: isinstance(e, (ConnectionError, TimeoutError))
    )
)

# ✅ 推荐：在节点内部处理预期异常
async def my_node(state: State) -> dict:
    try:
        result = await some_async_api()
        return {"result": result, "error": None}
    except asyncio.CancelledError:
        logger.info("Task cancelled")
        raise
    except Exception as e:
        return {"result": None, "error": str(e)}
```

### 3. 流式错误处理

```python
# ✅ 推荐：在循环内处理错误
async for event in graph.astream_events(input):
    try:
        if event["event"] == "on_chain_error":
            handle_error(event["data"])
        else:
            process_event(event)
    except Exception as e:
        logger.error(f"Error in stream: {e}")
```

### 4. Command 使用

```python
# ✅ 推荐：使用 Literal 类型注解
from typing import Literal

def my_node(state: State) -> Command[Literal["next", "end"]]:
    if state["x"] > 10:
        return Command(goto="end")
    return Command(goto="next")

# ✅ 推荐：显式声明目标节点
builder.add_node(
    "my_node",
    my_node,
    destinations=["next", "end"]
)
```
