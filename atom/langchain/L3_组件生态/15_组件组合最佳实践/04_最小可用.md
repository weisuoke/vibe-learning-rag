## 4.1 掌握三种基础组合模式

这三种模式覆盖了 90% 以上的 LangChain 应用场景。先把它们吃透，再去碰高级用法。

### 串行（最常用）

串行是 LCEL 的核心——用管道操作符 `|` 把组件一个接一个串起来，前一个的输出就是后一个的输入。

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI

# 最基础的 LCEL 链
chain = (
    ChatPromptTemplate.from_template("回答问题：{question}")
    | ChatOpenAI(model="gpt-4o-mini")
    | StrOutputParser()
)
result = chain.invoke({"question": "什么是RAG？"})
```

串行要点：
- `|` 操作符是语法糖，底层调用的是 `RunnableSequence`
- 每个组件必须实现 `Runnable` 接口（`invoke`/`stream`/`batch`）
- 数据从左到右流动，类型必须匹配：前一个的输出类型 = 后一个的输入类型

### 并行（RAG 必备）

用字典语法让多个组件同时执行，各自处理不同的数据，最后合并成一个字典传给下游。

```python
from langchain_core.runnables import RunnablePassthrough

# RAG 链的标准模式
chain = (
    {
        "question": RunnablePassthrough(),
        "context": retriever | (lambda docs: "\n".join(d.page_content for d in docs)),
    }
    | prompt
    | model
    | StrOutputParser()
)
```

并行要点：
- 字典的每个 key 对应一个并行分支
- `RunnablePassthrough()` 原样透传输入，是并行模式的常客
- 所有分支共享同一个输入，各自独立执行
- 输出是一个字典，key 就是你定义的那些 key

### 条件路由

根据输入内容动态选择不同的处理链。比如：问答走 QA 链，闲聊走 Chat 链。

```python
from langchain_core.runnables import RunnableBranch

chain = RunnableBranch(
    (lambda x: x["type"] == "qa", qa_chain),
    (lambda x: x["type"] == "chat", chat_chain),
    default_chain,
)
```

条件路由要点：
- 条件按顺序判断，第一个匹配的就执行
- 最后一个参数是默认链（没有条件），必须提供
- 条件函数接收完整输入，返回布尔值
- 适合意图分类、多模型路由等场景

## 4.2 添加基础错误处理

LLM 调用天然不稳定——网络超时、API 限流、模型过载都是家常便饭。不加错误处理的链在生产环境活不过一天。

### 模型降级

主模型失败时自动切换到备选模型，用户无感知。

```python
# 模型降级：主模型失败时切换备选
model = ChatOpenAI(model="gpt-4o").with_fallbacks(
    [ChatOpenAI(model="gpt-4o-mini")]
)
```

### 自动重试

网络抖动导致的偶发失败，重试几次通常就好了。

```python
# 重试：网络问题自动重试
model = ChatOpenAI(model="gpt-4o").with_retry(
    stop_after_attempt=3,
    wait_exponential_jitter=True,
)
```

### 组合使用（推荐）

先重试，重试都失败了再降级——这是生产环境的标准做法。

```python
# 组合：先重试再降级
model = (
    ChatOpenAI(model="gpt-4o")
    .with_retry(stop_after_attempt=3)
    .with_fallbacks([ChatOpenAI(model="gpt-4o-mini")])
)
```

错误处理要点：
- `with_retry` 处理瞬时故障（网络抖动、限流）
- `with_fallbacks` 处理持续故障（模型宕机、配额耗尽）
- 链式调用的顺序很重要：先 retry 再 fallback
- fallbacks 列表可以放多个备选，按顺序尝试

## 4.3 使用流式输出

LLM 生成文本是逐 token 的，流式输出让用户不用等完整响应就能看到内容，体验提升巨大。

### 同步流式

最简单的用法，适合脚本和命令行工具。

```python
# 同步流式
for chunk in chain.stream({"question": "什么是RAG？"}):
    print(chunk, end="", flush=True)
```

### 异步流式（FastAPI 场景）

Web 应用的标准做法——用 `astream` 配合 `StreamingResponse`。

```python
# 异步流式（FastAPI 场景）
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.post("/chat")
async def chat(question: str):
    async def generate():
        async for chunk in chain.astream({"question": question}):
            yield chunk

    return StreamingResponse(generate(), media_type="text/plain")
```

流式输出要点：
- `stream()` 和 `astream()` 是所有 Runnable 的内置方法
- 只有支持流式的组件（如 LLM）才会真正逐块输出
- `StrOutputParser` 会透传流式 chunk
- 中间的 prompt 模板等组件会一次性输出，不影响整体流式效果

## 4.4 批量处理

一次处理多个输入，LangChain 会自动并发执行，比循环调用快得多。

### 基础批量

```python
# 批量处理多个输入
questions = [{"question": q} for q in question_list]
results = chain.batch(questions, config={"max_concurrency": 5})
```

### 带异常处理的批量

单个失败不影响其他请求——批量任务的必备选项。

```python
# 带异常处理的批量
results = chain.batch(questions, return_exceptions=True)

# 处理结果
for i, result in enumerate(results):
    if isinstance(result, Exception):
        print(f"第 {i} 个请求失败: {result}")
    else:
        print(f"第 {i} 个请求成功: {result[:50]}...")
```

批量处理要点：
- `max_concurrency` 控制并发数，避免打爆 API 限流
- `return_exceptions=True` 让失败的请求返回异常对象而不是抛出
- 异步版本是 `abatch()`，用法完全一样
- 批量 + 重试 + 降级组合使用，是处理大量请求的最佳实践

## 4.5 基础可观测性

链跑起来之后，你需要知道每一步发生了什么——哪里慢了、哪里出错了、token 花了多少。

### LangSmith 追踪（最简方案）

设置两个环境变量就能开启全链路追踪，零代码侵入。

```python
import os
# 开启 LangSmith 追踪（最简单的可观测性方案）
os.environ["LANGSMITH_TRACING"] = "true"
os.environ["LANGSMITH_API_KEY"] = "your-key"

# 添加 run name 和 tags，方便在 LangSmith 面板中筛选
result = chain.invoke(
    {"question": "test"},
    config={"run_name": "qa_v1", "tags": ["production"]}
)
```

### 本地调试（不依赖外部服务）

```python
# 开启详细日志
import langchain
langchain.debug = True  # 打印每个组件的输入输出

# 或者用 verbose 模式
langchain.verbose = True  # 只打印关键信息
```

可观测性要点：
- LangSmith 是官方推荐的可观测性方案，免费版够用
- `run_name` 和 `tags` 是追踪的最小配置，一定要加
- 本地调试用 `langchain.debug = True`，上线前记得关掉
- 每次调用的 token 用量、延迟、错误都会自动记录

## 4.6 完整示例：把五个技能组合起来

下面这个例子把串行、并行、错误处理、流式、可观测性全部用上，构建一个生产可用的 RAG 问答链。

```python
import os
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

# 0. 可观测性
os.environ["LANGSMITH_TRACING"] = "true"
os.environ["LANGSMITH_API_KEY"] = "your-key"

# 1. 向量检索器
vectorstore = Chroma(
    collection_name="docs",
    embedding_function=OpenAIEmbeddings(),
)
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# 2. 带容错的模型
model = (
    ChatOpenAI(model="gpt-4o", temperature=0)
    .with_retry(stop_after_attempt=3)
    .with_fallbacks([ChatOpenAI(model="gpt-4o-mini", temperature=0)])
)

# 3. Prompt 模板
prompt = ChatPromptTemplate.from_template(
    "根据以下上下文回答问题。如果上下文中没有答案，说"我不确定"。\n\n"
    "上下文：{context}\n\n"
    "问题：{question}"
)

# 4. 组装完整链（串行 + 并行）
chain = (
    {
        "question": RunnablePassthrough(),
        "context": retriever | (lambda docs: "\n\n".join(d.page_content for d in docs)),
    }
    | prompt
    | model
    | StrOutputParser()
)

# 5. 流式调用
for chunk in chain.stream(
    "什么是向量检索？",
    config={"run_name": "rag_qa", "tags": ["production"]},
):
    print(chunk, end="", flush=True)
```

这段代码不到 40 行，但已经具备了：串行管道、并行数据准备、自动重试、模型降级、流式输出、LangSmith 追踪。这就是最小可用的力量。

## 从最小可用到生产级的路径

```
最小可用（你现在在这里）
  │
  ├── 串行 + 并行 + 条件路由
  ├── retry + fallback
  ├── stream + batch
  └── LangSmith 基础追踪
  │
  ▼
下一步（核心概念详解）
  │
  ├── RunnablePassthrough.assign() 高级数据传递
  ├── RunnableLambda 自定义组件
  ├── configurable_fields 运行时配置
  ├── 缓存策略（InMemoryCache / RedisCache）
  └── 自定义 Callback Handler
```

先用最小可用跑通业务，遇到具体问题再去学对应的高级特性。不要一开始就追求完美架构。

## 常见问题速查

| 问题 | 解决方案 |
|------|----------|
| 链的中间结果怎么看？ | `langchain.debug = True` 或 LangSmith |
| 并行分支需要共享中间结果？ | 用 `RunnablePassthrough.assign()` |
| 想在链中插入自定义逻辑？ | 用 `RunnableLambda(fn)` 包装函数 |
| 流式输出只有最后才出内容？ | 检查中间组件是否阻塞了流式传递 |
| batch 太慢？ | 调高 `max_concurrency`，注意 API 限流 |
| fallback 模型也挂了？ | fallbacks 列表多加几个备选 |
