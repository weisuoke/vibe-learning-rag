# 核心概念 6：时间旅行机制

## 概述

时间旅行（Time Travel）是 LangGraph 基于 checkpoint 系统提供的高级能力，让你可以**回到图执行历史中的任意时间点**，重放过去的执行或从某个历史状态分叉出新的执行路径。它包含两大操作：**Replay（重放）** 和 **Fork（分叉）**。这不是科幻概念——它是调试 AI Agent、优化 RAG 系统的实用工具。

[来源: reference/context7_langgraph_01.md | LangGraph 官方文档]

---

## 为什么 AI 系统需要时间旅行？

### LLM 的非确定性问题

传统软件是确定性的：相同输入总是产生相同输出。但 LLM 不同——即使相同的 prompt，每次调用可能返回不同的结果。这让调试变得极其困难。

```
传统软件：input=5 → 总是输出 25
LLM 系统：prompt="写个笑话" → 每次输出不同的笑话
```

时间旅行解决了这个问题：你可以**精确回到出问题的那个状态**，而不是重新运行整个流程（可能因为 LLM 的非确定性得到不同的结果）。

### Checkpointer 的三大能力

LangGraph 的 checkpointer 提供三大能力，时间旅行是其中两个：

| 能力 | 说明 | 对应操作 |
|------|------|---------|
| **持久化** | 跨交互保持状态 | 自动保存 checkpoint |
| **时间旅行** | 重放过去的执行 | Replay |
| **分叉** | 探索替代执行路径 | Fork |

[来源: reference/search_状态快照_01.md | 社区实践]

---

## Replay 重放：回到过去重新执行

### 什么是 Replay？

Replay 是指**从某个历史 checkpoint 恢复状态，然后重新执行后续步骤**。LangGraph 会智能地跳过已经执行过的步骤，只重新执行 checkpoint 之后的部分。

### 工作原理

```
原始执行：
  input → nodeA → nodeB → nodeC → 完成
  cp0      cp1     cp2     cp3

Replay 从 cp1 开始：
  cp1 → [跳过 nodeA，已执行] → 重新执行 nodeB → 重新执行 nodeC → 完成
```

**关键点**：LangGraph 不是简单地"从头跑一遍"。它会：
1. 恢复到指定 checkpoint 的状态
2. 智能识别哪些步骤已经执行过
3. 重放已执行的步骤（使用保存的结果，不重新计算）
4. 重新执行后续步骤

### 代码示例

```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver

class State(TypedDict):
    topic: str
    steps: Annotated[list[str], add]

def research_node(state: State):
    print(f"  [research] 正在研究: {state['topic']}")
    return {"steps": [f"研究了{state['topic']}"]}

def write_node(state: State):
    print(f"  [write] 正在撰写关于: {state['topic']}")
    return {"steps": [f"撰写了{state['topic']}文章"]}

def review_node(state: State):
    print(f"  [review] 正在审核")
    return {"steps": ["审核完成"]}

# 构建图
builder = StateGraph(State)
builder.add_node("research", research_node)
builder.add_node("write", write_node)
builder.add_node("review", review_node)
builder.add_edge(START, "research")
builder.add_edge("research", "write")
builder.add_edge("write", "review")
builder.add_edge("review", END)

memory = MemorySaver()
graph = builder.compile(checkpointer=memory)

# ===== 原始执行 =====
print("=== 原始执行 ===")
config = {"configurable": {"thread_id": "replay-demo"}}
result = graph.invoke({"topic": "AI", "steps": []}, config=config)
print(f"结果: {result['steps']}")

# ===== 查看历史，找到 research 执行后的 checkpoint =====
print("\n=== 历史快照 ===")
states = list(graph.get_state_history(config))
for i, s in enumerate(states):
    step = s.metadata.get("step", "?")
    source = s.metadata.get("source", "?")
    cp_id = s.config["configurable"]["checkpoint_id"]
    print(f"  [{i}] step={step}, source={source}, next={s.next}, id={cp_id[:12]}...")

# 找到 research 执行后的状态（next 包含 write）
replay_state = None
for s in states:
    if "write" in s.next:
        replay_state = s
        break

# ===== Replay：从 research 之后重新执行 =====
print(f"\n=== Replay 从 checkpoint {replay_state.config['configurable']['checkpoint_id'][:12]}... ===")
replay_config = {
    "configurable": {
        "thread_id": "replay-demo",
        "checkpoint_id": replay_state.config["configurable"]["checkpoint_id"]
    }
}
replay_result = graph.invoke(None, config=replay_config)
print(f"Replay 结果: {replay_result['steps']}")
```

**预期输出**：

```
=== 原始执行 ===
  [research] 正在研究: AI
  [write] 正在撰写关于: AI
  [review] 正在审核
结果: ['研究了AI', '撰写了AI文章', '审核完成']

=== 历史快照 ===
  [0] step=3, source=loop, next=(), id=1ef...003...
  [1] step=2, source=loop, next=('review',), id=1ef...002...
  [2] step=1, source=loop, next=('write',), id=1ef...001...
  [3] step=-1, source=input, next=('research',), id=1ef...000...

=== Replay 从 checkpoint 1ef...001... ===
  [write] 正在撰写关于: AI
  [review] 正在审核
Replay 结果: ['研究了AI', '撰写了AI文章', '审核完成']
```

**注意**：Replay 时 research 节点没有重新执行（没有打印 `[research]`），因为它的结果已经保存在 checkpoint 中了。

### Replay 的适用场景

1. **调试**：重现某个执行路径，观察中间状态
2. **复现问题**：精确回到出问题的状态，不受 LLM 非确定性影响
3. **性能分析**：重放执行过程，测量各节点耗时

[来源: reference/context7_langgraph_01.md | LangGraph 官方文档]

## Fork 分叉：探索替代路径

### 什么是 Fork？

Fork 是指**从某个历史 checkpoint 出发，修改状态后走一条不同的执行路径**。它是 `update_state()` + `invoke()` 的组合操作。

### 工作原理

```
原始执行：
  input → nodeA → nodeB → nodeC → 结果1
  cp0      cp1     cp2     cp3

Fork 从 cp1 分叉：
  cp1 → update_state(修改状态) → cp1' → nodeB' → nodeC' → 结果2
                                   ↑
                              source="fork"
```

### 代码示例

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver

class State(TypedDict):
    query: str
    search_engine: str
    result: str

def search_node(state: State):
    engine = state["search_engine"]
    query = state["query"]
    # 模拟不同搜索引擎的结果
    results = {
        "google": f"Google 搜索 '{query}' 的结果：找到 100 条",
        "bing": f"Bing 搜索 '{query}' 的结果：找到 80 条",
        "duckduckgo": f"DuckDuckGo 搜索 '{query}' 的结果：找到 60 条（隐私优先）",
    }
    return {"result": results.get(engine, "未知搜索引擎")}

def summarize_node(state: State):
    return {"result": f"摘要: {state['result']}"}

builder = StateGraph(State)
builder.add_node("search", search_node)
builder.add_node("summarize", summarize_node)
builder.add_edge(START, "search")
builder.add_edge("search", "summarize")
builder.add_edge("summarize", END)

memory = MemorySaver()
graph = builder.compile(checkpointer=memory)

# ===== 原始执行：使用 Google =====
config = {"configurable": {"thread_id": "fork-demo"}}
result1 = graph.invoke(
    {"query": "LangGraph 教程", "search_engine": "google"},
    config=config
)
print(f"原始结果: {result1['result']}")

# ===== 找到输入状态 =====
states = list(graph.get_state_history(config))
input_state = [s for s in states if s.metadata.get("source") == "input"][0]

# ===== Fork 1：改用 Bing =====
fork1_config = graph.update_state(
    input_state.config,
    values={"search_engine": "bing"},
    as_node="__start__"
)
result2 = graph.invoke(None, config=fork1_config)
print(f"Fork 1 (Bing): {result2['result']}")

# ===== Fork 2：改用 DuckDuckGo =====
fork2_config = graph.update_state(
    input_state.config,
    values={"search_engine": "duckduckgo"},
    as_node="__start__"
)
result3 = graph.invoke(None, config=fork2_config)
print(f"Fork 2 (DuckDuckGo): {result3['result']}")

# ===== 对比三条路径 =====
print("\n=== A/B/C 测试结果 ===")
print(f"  Google:      {result1['result']}")
print(f"  Bing:        {result2['result']}")
print(f"  DuckDuckGo:  {result3['result']}")
```

**预期输出**：

```
原始结果: 摘要: Google 搜索 'LangGraph 教程' 的结果：找到 100 条
Fork 1 (Bing): 摘要: Bing 搜索 'LangGraph 教程' 的结果：找到 80 条
Fork 2 (DuckDuckGo): 摘要: DuckDuckGo 搜索 'LangGraph 教程' 的结果：找到 60 条（隐私优先）

=== A/B/C 测试结果 ===
  Google:      摘要: Google 搜索 'LangGraph 教程' 的结果：找到 100 条
  Bing:        摘要: Bing 搜索 'LangGraph 教程' 的结果：找到 80 条
  DuckDuckGo:  摘要: DuckDuckGo 搜索 'LangGraph 教程' 的结果：找到 60 条（隐私优先）
```

### Fork 的适用场景

1. **A/B 测试**：从同一状态出发，尝试不同的参数
2. **错误修复**：回到出错前的状态，修正后重新执行
3. **探索性调试**：尝试不同的输入，观察对结果的影响

[来源: reference/context7_langgraph_01.md | LangGraph 官方文档]

## Replay vs Fork：对比理解

| 特性 | Replay 重放 | Fork 分叉 |
|------|------------|----------|
| 目的 | 重新执行相同路径 | 探索不同路径 |
| 是否修改状态 | 不修改 | 修改后再执行 |
| API 调用 | `invoke(None, config)` | `update_state()` + `invoke(None, config)` |
| source 标记 | 无新标记 | `"fork"` |
| 类比 | 重看一部电影 | 选择不同的剧情分支 |

### 前端类比

- **Replay** 像浏览器的"后退"按钮 + "前进"按钮——回到之前的页面状态，然后重新走一遍相同的路径
- **Fork** 像 Git 的 `git branch` + `git checkout`——从某个 commit 创建新分支，走不同的开发路径

### 日常生活类比

- **Replay** 像重看一盘棋的录像——每一步都和原来一样
- **Fork** 像在棋局的某一步选择不同的走法——"如果当时走马而不是走车，结果会怎样？"

## 完整的时间旅行工作流

下面是一个完整的示例，展示从执行到调试到分叉的完整工作流。

```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver

# ===== 1. 定义 RAG 状态 =====
class RAGState(TypedDict):
    query: str
    documents: Annotated[list[str], add]
    answer: str
    confidence: float

# ===== 2. 定义节点 =====
def retrieve_node(state: RAGState):
    query = state["query"]
    # 模拟检索
    docs = [f"文档A: {query}的定义", f"文档B: {query}的应用"]
    return {"documents": docs, "confidence": 0.7}

def generate_node(state: RAGState):
    docs = state["documents"]
    context = " | ".join(docs)
    answer = f"基于 {len(docs)} 篇文档回答: {context}"
    return {"answer": answer, "confidence": state["confidence"] + 0.1}

# ===== 3. 构建图 =====
builder = StateGraph(RAGState)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

memory = MemorySaver()
graph = builder.compile(checkpointer=memory)

# ===== 4. 原始执行 =====
print("=" * 60)
print("步骤 1: 原始执行")
print("=" * 60)
config = {"configurable": {"thread_id": "rag-time-travel"}}
result = graph.invoke(
    {"query": "向量数据库", "documents": [], "confidence": 0.0},
    config=config
)
print(f"答案: {result['answer']}")
print(f"置信度: {result['confidence']}")

# ===== 5. 查看完整历史 =====
print(f"\n{'=' * 60}")
print("步骤 2: 查看执行历史")
print("=" * 60)
states = list(graph.get_state_history(config))
for i, s in enumerate(states):
    step = s.metadata.get("step", "?")
    source = s.metadata.get("source", "?")
    docs_count = len(s.values.get("documents", []))
    print(f"  快照[{i}]: step={step}, source={source}, "
          f"docs={docs_count}, next={s.next}")

# ===== 6. Replay：从 retrieve 之后重放 =====
print(f"\n{'=' * 60}")
print("步骤 3: Replay（重放）")
print("=" * 60)

# 找到 retrieve 执行后的状态
after_retrieve = [s for s in states if "generate" in s.next][0]
replay_config = {
    "configurable": {
        "thread_id": "rag-time-travel",
        "checkpoint_id": after_retrieve.config["configurable"]["checkpoint_id"]
    }
}
replay_result = graph.invoke(None, config=replay_config)
print(f"Replay 答案: {replay_result['answer']}")

# ===== 7. Fork：修改检索结果后重新生成 =====
print(f"\n{'=' * 60}")
print("步骤 4: Fork（分叉）- 添加更多文档")
print("=" * 60)

# 基于 retrieve 之后的状态分叉，添加额外文档
fork_config = graph.update_state(
    after_retrieve.config,
    values={"documents": ["文档C: 向量数据库的性能优化"]},  # 追加（因为有 add reducer）
    as_node="retrieve"
)

# 从分叉点继续执行
fork_result = graph.invoke(None, config=fork_config)
print(f"Fork 答案: {fork_result['answer']}")
print(f"Fork 文档数: {len(fork_result['documents'])}")

# ===== 8. 对比结果 =====
print(f"\n{'=' * 60}")
print("步骤 5: 对比结果")
print("=" * 60)
print(f"原始 ({len(result['documents'])} 篇文档): {result['answer'][:60]}...")
print(f"Fork  ({len(fork_result['documents'])} 篇文档): {fork_result['answer'][:60]}...")
```

**预期输出**：

```
============================================================
步骤 1: 原始执行
============================================================
答案: 基于 2 篇文档回答: 文档A: 向量数据库的定义 | 文档B: 向量数据库的应用
置信度: 0.8

============================================================
步骤 2: 查看执行历史
============================================================
  快照[0]: step=2, source=loop, docs=2, next=()
  快照[1]: step=1, source=loop, docs=2, next=('generate',)
  快照[2]: step=-1, source=input, docs=0, next=('retrieve',)

============================================================
步骤 3: Replay（重放）
============================================================
Replay 答案: 基于 2 篇文档回答: 文档A: 向量数据库的定义 | 文档B: 向量数据库的应用

============================================================
步骤 4: Fork（分叉）- 添加更多文档
============================================================
Fork 答案: 基于 3 篇文档回答: 文档A: 向量数据库的定义 | 文档B: 向量数据库的应用 | 文档C: 向量数据库的性能优化

============================================================
步骤 5: 对比结果
============================================================

Fork  (3 篇文档): 基于 3 篇文档回答: 文档A: 向量数据库的定义 | 文档B: 向量数据库...
```

## 与 Human-in-the-loop 的结合

时间旅行最常见的实际应用是与 Human-in-the-loop（人工干预）结合。图在关键节点暂停，人工审核后决定是继续、修改还是回退。

### 工作流程

```
用户输入 → 检索节点 → [interrupt] → 人工审核
                                       ↓
                              ┌─── 满意 ───→ 继续执行 → 生成节点 → 输出
                              │
                              ├─── 不满意 ──→ update_state(修改检索结果)
                              │                    ↓
                              │              继续执行 → 生成节点 → 输出
                              │
                              └─── 重来 ───→ Fork 到输入状态
                                                   ↓
                                             修改查询 → 重新执行
```

### 代码示例

```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver

class State(TypedDict):
    query: str
    documents: Annotated[list[str], add]
    approved: bool
    answer: str

def retrieve_node(state: State):
    docs = [f"检索结果: {state['query']}相关文档"]
    return {"documents": docs}

def generate_node(state: State):
    return {"answer": f"基于 {len(state['documents'])} 篇文档的回答"}

def should_continue(state: State):
    # 在实际应用中，这里会检查人工审核结果
    if state.get("approved", False):
        return "generate"
    return END

builder = StateGraph(State)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "retrieve")
builder.add_conditional_edges("retrieve", should_continue, ["generate", END])
builder.add_edge("generate", END)

memory = MemorySaver()
graph = builder.compile(checkpointer=memory)

# 执行（未审核，会在 retrieve 后停止）
config = {"configurable": {"thread_id": "hitl-demo"}}
result = graph.invoke(
    {"query": "RAG 最佳实践", "documents": [], "approved": False},
    config=config
)
print(f"第一次执行结果: {result}")

# 人工审核后，批准并继续
states = list(graph.get_state_history(config))
after_retrieve = [s for s in states if s.metadata.get("step") == 1][0]

# 修改状态：标记为已审核
new_config = graph.update_state(
    after_retrieve.config,
    values={"approved": True},
    as_node="retrieve"
)

# 从审核后的状态继续执行
final_result = graph.invoke(None, config=new_config)
print(f"审核后结果: {final_result}")
```

## 在 RAG 系统中的应用

### 场景 1：调试检索质量

```
问题：RAG 系统返回了不相关的答案
调试流程：
  1. get_state_history() 查看执行历史
  2. 找到 retrieve 节点执行后的状态
  3. 检查 documents 字段，发现检索结果不相关
  4. Fork：修改 documents 为正确的文档
  5. 继续执行 generate 节点
  6. 验证：如果答案正确，说明问题在检索环节
```

### 场景 2：优化 Prompt

```
问题：检索结果正确，但生成的答案质量不高
调试流程：
  1. 找到 generate 节点执行前的状态
  2. Fork：修改 prompt 模板或参数
  3. 重新执行 generate 节点
  4. 对比不同 prompt 的生成质量
```

### 场景 3：多轮对话回退

```
问题：对话式 RAG 中，用户想回到之前的某个问题重新回答
操作流程：
  1. get_state_history() 找到目标轮次的状态
  2. Fork：修改用户的问题或上下文
  3. 从该轮次重新执行
```

## 常见误区

### 误区 1："Replay 会重新调用 LLM" ❌

**事实**：Replay 对于已执行的步骤，会使用保存在 checkpoint 中的结果，不会重新调用 LLM。只有 checkpoint 之后的步骤才会重新执行。这意味着 Replay 不会产生额外的 API 费用（对于已执行的部分）。

### 误区 2："Fork 会影响原始执行历史" ❌

**事实**：Fork 创建的是全新的 checkpoint 链，原始执行历史完全不受影响。你可以随时回到原始路径。

```
原始：cp0 → cp1 → cp2 → cp3
Fork：cp0 → cp1 → cp1'(fork) → cp2' → cp3'

两条路径独立存在，互不影响
```

### 误区 3："时间旅行只能用于调试" ❌

**事实**：时间旅行在生产环境中也有重要应用：
- **Human-in-the-loop**：人工审核和干预
- **错误恢复**：从失败点重试，而不是从头开始
- **A/B 测试**：在生产环境中对比不同策略的效果
- **审计合规**：完整的执行轨迹记录

---

## 总结

### 核心要点

1. 时间旅行包含两大操作：**Replay（重放）** 和 **Fork（分叉）**
2. Replay 使用 `checkpoint_id` 恢复状态，智能跳过已执行步骤
3. Fork 通过 `update_state()` + `invoke(None)` 组合实现
4. 时间旅行依赖 checkpointer，必须在编译时配置
5. 在 RAG 系统中，时间旅行是调试检索质量和优化 prompt 的利器

### API 速查

```python
# Replay：从特定 checkpoint 重放
config = {"configurable": {"thread_id": "1", "checkpoint_id": "xxx"}}
result = graph.invoke(None, config=config)

# Fork：修改历史状态后执行
new_config = graph.update_state(old_config, values={...}, as_node="node_name")
result = graph.invoke(None, config=new_config)

# 查看历史（时间旅行的基础）
for state in graph.get_state_history(config):
    print(state.config, state.values, state.next)
```
