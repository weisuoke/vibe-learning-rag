# 状态验证 - 第一性原理

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题。

不是"别人怎么做我就怎么做"，而是"这件事最本质的需求是什么？从这个需求出发，最合理的解决方案是什么？"

亚里士多德说："在每一个系统探索中，都存在第一性原理——它是一个最基本的命题或假设，不能被省略或删除，也不能被违反。"

对于状态验证，我们要问的第一性问题是：

> **为什么数据在系统中流转时，需要被检查？**

---

## 状态验证的第一性原理

### 1. 最基础的定义

**状态验证 = 在数据进入系统前，检查它是否符合预期格式和规则**

仅此而已！没有更基础的了。

就像快递分拣中心：每个包裹进来时，先检查地址是否完整、重量是否超标、
是否是违禁品。不合格的包裹在入口就被拦截，而不是等到送到客户手里才发现问题。

再换一个角度理解——你去银行办业务，柜员会先检查你的身份证、表格填写是否完整。
如果身份证过期了，柜员会当场告诉你，而不是等你排了两小时队、办到一半才说"不好意思，你的证件无效"。

**状态验证就是这个"柜员检查"环节。**

把这个概念放到 LangGraph 中：

```
用户输入 / LLM 输出 / API 返回值
        ↓
   【状态验证关卡】 ← 在这里检查数据
        ↓
   进入图的状态，被各节点使用
```

---

### 2. 为什么需要状态验证？

**核心问题：如何确保在复杂的多节点图中，数据始终保持正确？**

在 LangGraph 中，状态在多个节点之间传递。每个节点都可能修改状态。
如果没有验证：

- LLM 可能返回意外格式的数据（比如你期望 JSON，它返回了纯文本）
- 外部 API 可能返回缺失字段（比如天气 API 没返回温度字段）
- 用户输入可能包含非法值（比如年龄填了 -5）
- 节点可能返回错误类型的更新（比如应该返回列表，却返回了字符串）

这些错误会在图中传播，越来越难以追踪。

**类比：生产线上的质检**

想象一条生产线，如果第一道工序就用了错误的原材料，
后面所有工序都在错误的基础上继续，最终产品必然有问题。
状态验证就是在每道工序前加一个质检环节。

**类比：前端开发中的表单验证**

如果你做过前端开发，你一定写过表单验证：

```javascript
// 前端表单验证 —— 你一定见过这种代码
if (!email.includes('@')) {
  showError('邮箱格式不正确');
  return;
}
```

状态验证和表单验证的本质完全一样——在数据被使用之前，先确认它是对的。

**没有验证的后果：**

```python
# 没有状态验证的 LangGraph 节点
def process_node(state):
    # state["age"] 可能是 "twenty" 而不是 20
    # state["messages"] 可能是 None 而不是 []
    # state["config"] 可能缺少必要的 key

    result = state["age"] * 2  # 💥 TypeError: can't multiply str by int
    # 这个错误可能在第 5 个节点才爆发，但根因在第 1 个节点
```

**有验证的效果：**

```python
# 有状态验证的 LangGraph 状态
from pydantic import BaseModel, Field

class AgentState(BaseModel):
    age: int = Field(ge=0, le=150)        # 自动检查类型和范围
    messages: list[str] = Field(default_factory=list)  # 确保是列表
    config: dict = Field(min_length=1)     # 确保不为空

# 如果 age="twenty"，在状态创建时就会报错，而不是在第 5 个节点
```

### 3. 状态验证的三层价值

#### 价值1：输入安全（防御性编程）

在图的入口处验证输入数据，防止非法数据进入系统。

- **类型检查**：确保 `age` 是 `int` 而不是 `"twenty"`
- **必填检查**：确保必要字段不缺失
- **格式检查**：确保 `email` 格式正确

```python
from pydantic import BaseModel, EmailStr

class UserInput(BaseModel):
    name: str                    # 必填，必须是字符串
    age: int                     # 必填，必须是整数
    email: EmailStr              # 必填，必须是合法邮箱格式

# ✅ 合法输入
user = UserInput(name="Alice", age=25, email="alice@example.com")

# ❌ 非法输入 —— 立即报错
user = UserInput(name="Alice", age="twenty", email="not-an-email")
# ValidationError: 2 validation errors
#   age: Input should be a valid integer
#   email: value is not a valid email address
```

**RAG 场景中的输入安全：**

```python
class RAGQuery(BaseModel):
    """RAG 查询的输入验证"""
    question: str = Field(min_length=1, max_length=1000)  # 问题不能为空，不能太长
    top_k: int = Field(default=5, ge=1, le=50)            # 检索数量有合理范围
    temperature: float = Field(default=0.7, ge=0.0, le=2.0)  # 温度有合理范围
```

#### 价值2：自动类型转换（容错性）

Pydantic 不仅验证，还能自动转换兼容类型：

- `"42"` → `42`（字符串转整数）
- `"true"` → `True`（字符串转布尔）
- `{"lat": 1.0, "lng": 2.0}` → `Location(lat=1.0, lng=2.0)`（字典转模型）

这在处理 LLM 输出时特别有用，因为 LLM 经常返回字符串格式的数字。

```python
from pydantic import BaseModel

class LLMOutput(BaseModel):
    confidence: float
    num_sources: int
    is_relevant: bool

# LLM 返回的原始数据（全是字符串）
raw_output = {
    "confidence": "0.95",    # 字符串
    "num_sources": "3",      # 字符串
    "is_relevant": "true"    # 字符串
}

# Pydantic 自动转换 ✅
result = LLMOutput(**raw_output)
print(result.confidence)    # 0.95 (float)
print(result.num_sources)   # 3 (int)
print(result.is_relevant)   # True (bool)
```

**为什么这很重要？**

在 RAG 系统中，LLM 的输出是不可预测的。你让它返回一个数字，
它可能返回 `"5"`、`"五"` 或 `"approximately 5"`。
Pydantic 的自动转换能处理前两种情况，第三种则会触发验证错误——
这正是我们想要的行为：能容错的就容错，不能容错的就明确报错。

#### 价值3：业务约束（语义正确性）

超越类型检查，验证业务规则：

- 年龄必须在 0-150 之间
- 开始日期必须早于结束日期
- 如果选择了 "高级模式"，则必须提供 API key

```python
from pydantic import BaseModel, Field, model_validator

class SearchConfig(BaseModel):
    """检索配置的业务约束验证"""
    mode: str = Field(pattern=r"^(basic|advanced)$")  # 只能是 basic 或 advanced
    api_key: str | None = None
    max_results: int = Field(ge=1, le=100)

    @model_validator(mode="after")
    def check_advanced_mode(self):
        """高级模式必须提供 API key"""
        if self.mode == "advanced" and not self.api_key:
            raise ValueError("高级模式必须提供 api_key")
        return self

# ✅ 基础模式，不需要 API key
config = SearchConfig(mode="basic", max_results=10)

# ❌ 高级模式但没有 API key
config = SearchConfig(mode="advanced", max_results=10)
# ValidationError: 高级模式必须提供 api_key
```

**三层价值的关系：**

```
第1层：类型安全     → "这个值是正确的类型吗？"
第2层：自动转换     → "这个值能被转换成正确的类型吗？"
第3层：业务约束     → "这个值在业务上有意义吗？"

三层层层递进，从基础到高级，构成完整的验证体系。
```

### 4. 从第一性原理推导 LangGraph 状态验证

**推理链：**

```
1. LangGraph 的状态在多个节点间共享
   ↓
2. 每个节点都可能修改状态
   ↓
3. 错误的状态修改会在图中传播
   ↓
4. 需要在数据进入时进行验证
   ↓
5. Python 的 TypedDict 只有静态类型检查（IDE 提示），无运行时验证
   ↓
6. Pydantic BaseModel 提供运行时验证能力
   ↓
7. LangGraph 集成 Pydantic，在 _coerce_state 中触发验证
   ↓
8. 通过 field_validator 和 model_validator 实现自定义验证逻辑
   ↓
9. 验证错误通过 ValidationError 提供结构化错误信息
   ↓
10. 在图的边界处使用 Pydantic 验证，内部使用 TypedDict 保持性能
```

让我们逐步展开这条推理链：

**第 1-3 步：问题的本质**

```python
# LangGraph 中，状态像接力棒一样在节点间传递
#
#   节点A → 状态 → 节点B → 状态 → 节点C
#
# 如果节点A把状态搞坏了，节点B和节点C都会受影响

def node_a(state):
    # 本应返回 {"count": 1}，但写错了
    return {"count": "one"}  # 💥 字符串而不是整数

def node_b(state):
    # 期望 count 是整数
    return {"count": state["count"] + 1}  # 💥 "one" + 1 = TypeError
```

**第 4-6 步：解决方案的选择**

Python 提供了两种定义结构化数据的方式：

```python
# 方式1：TypedDict —— 只有静态检查，没有运行时验证
from typing import TypedDict

class State(TypedDict):
    count: int
    name: str

state: State = {"count": "not_a_number", "name": 123}
# ⚠️ IDE 会画波浪线警告，但代码照样运行！不会报错！


# 方式2：Pydantic BaseModel —— 有运行时验证
from pydantic import BaseModel

class State(BaseModel):
    count: int
    name: str

state = State(count="not_a_number", name=123)
# ❌ ValidationError: Input should be a valid integer
# 运行时直接报错！
```

**第 7 步：LangGraph 的集成方式**

LangGraph 在内部通过 `_coerce_state` 函数实现了与 Pydantic 的集成：

```python
# 简化的原理展示
def _coerce_state(schema, input_data):
    """当 schema 是 Pydantic BaseModel 时，触发验证"""
    if is_pydantic_model(schema):
        return schema(**input_data)  # Pydantic 构造函数会自动验证
    else:
        return input_data  # TypedDict 直接透传，不验证
```

**第 8 步：自定义验证逻辑**

```python
from pydantic import BaseModel, field_validator, model_validator

class AgentState(BaseModel):
    messages: list[str]
    retry_count: int = 0

    @field_validator("messages")
    @classmethod
    def messages_not_empty(cls, v):
        """单字段验证：消息列表不能为空"""
        if len(v) == 0:
            raise ValueError("消息列表不能为空")
        return v

    @model_validator(mode="after")
    def check_retry_limit(self):
        """跨字段验证：重试次数不能超过消息数量"""
        if self.retry_count > len(self.messages):
            raise ValueError("重试次数不能超过消息数量")
        return self
```

**第 9 步：结构化错误信息**

```python
from pydantic import ValidationError

try:
    state = AgentState(messages=[], retry_count=5)
except ValidationError as e:
    print(e.error_count())  # 错误数量
    print(e.errors())       # 结构化错误列表
    # [
    #   {
    #     'type': 'value_error',
    #     'loc': ('messages',),
    #     'msg': 'Value error, 消息列表不能为空',
    #     'input': [],
    #   }
    # ]
```

**第 10 步：性能与安全的平衡**

```
                    ┌─────────────────────────────┐
                    │     图的边界（入口/出口）      │
                    │   使用 Pydantic BaseModel     │
                    │   ✅ 完整验证                  │
                    │   ⚠️ 有性能开销               │
                    └──────────┬──────────────────┘
                               │
                    ┌──────────▼──────────────────┐
                    │     图的内部（节点间传递）      │
                    │   使用 TypedDict              │
                    │   ✅ 零性能开销               │
                    │   ⚠️ 无运行时验证             │
                    └─────────────────────────────┘

这就像机场安检：
- 入口处严格安检（Pydantic 验证）
- 进入候机区后自由活动（TypedDict 无验证）
- 登机口再次检查（出口验证）
```

### 5. TypedDict vs Pydantic：本质区别

理解状态验证的第一性原理，必须理解这两者的本质区别：

| 维度 | TypedDict | Pydantic BaseModel |
|------|-----------|-------------------|
| 验证时机 | 静态（IDE/mypy） | 运行时 |
| 类型转换 | 无 | 自动 |
| 自定义验证 | 无 | field_validator / model_validator |
| 性能开销 | 零 | 有（构造时验证） |
| 错误信息 | 无 | 结构化 ValidationError |
| 适用场景 | 图内部传递 | 图边界验证 |

```python
# TypedDict：像一张便签纸，写什么都行
from typing import TypedDict

class NoteState(TypedDict):
    content: str

state: NoteState = {"content": 12345}  # 不会报错！Python 不管你写了什么


# Pydantic：像一份正式合同，格式必须对
from pydantic import BaseModel

class ContractState(BaseModel):
    content: str

state = ContractState(content=12345)  # content 会被转换为 "12345"
```

---

### 6. 一句话总结第一性原理

**状态验证是在数据进入图之前的"质检关卡"，通过 Pydantic 的运行时验证能力，确保 LangGraph 状态始终符合预期的类型和业务规则。**

记住这个核心公式：

```
状态验证 = 运行时类型检查 + 自动类型转换 + 业务规则约束
         = Pydantic BaseModel + field_validator + model_validator
         = 在图的边界处，用最小的性能代价，换取最大的数据安全
```

## 源码印证

LangGraph 源码中，状态验证的核心在 `state.py` 的 `_coerce_state` 函数：

```python
# langgraph/graph/state.py（简化展示）
def _coerce_state(schema: type[Any], input: dict[str, Any]) -> dict[str, Any]:
    return schema(**input)  # 当 schema 是 Pydantic BaseModel 时，触发完整验证
```

这一行代码就是状态验证的"第一性原理"实现：
将输入字典传给 Pydantic 模型构造函数，Pydantic 自动完成类型检查、类型转换和自定义验证。

**验证触发的完整链路：**

```
graph.invoke(input_data)
    ↓
StateGraph._invoke()
    ↓
_coerce_state(self.schema, input_data)
    ↓
PydanticModel(**input_data)    ← Pydantic 在这里执行所有验证
    ↓
验证通过 → 返回验证后的状态
验证失败 → 抛出 ValidationError
```

[来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py]

3. **三层价值**：类型安全 → 自动转换 → 业务约束
