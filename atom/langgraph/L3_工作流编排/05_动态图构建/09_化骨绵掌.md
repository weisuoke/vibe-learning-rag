## 卡片 1/10：动态图的直觉理解

### 核心观点

**动态图 = 运行时才决定走哪条路的工作流。**

### 日常类比

想象你用导航 App 开车：

```
出发前：导航计算了 3 条可能的路线（编译时定义可能性）
    ↓
行驶中：前方堵车，导航自动切换到路线 B（运行时选择路径）
    ↓
到达：不管走哪条路，都到了目的地（图执行完成）
```

关键点：路线在出发前就规划好了所有可能性，但实际走哪条取决于实时路况。LangGraph 的动态图也是这样——`compile()` 时定义所有可能的路径，`invoke()` 时根据状态选择走哪条。

### 在 LangGraph 中

```python
from langgraph.graph import StateGraph, START, END

# 编译时：定义了 3 条可能的路径
builder = StateGraph(State)
builder.add_conditional_edges("router", route_function)
graph = builder.compile()  # 结构冻结，但路径是动态的

# 运行时：根据输入走不同路径
graph.invoke({"query": "天气"})   # → weather_handler
graph.invoke({"query": "新闻"})   # → news_handler
```

### 关键记忆点

"动态"不是修改图结构，而是在已有的图中选择路径。就像导航不会修改道路，只是选择走哪条。

## 卡片 2/10：条件边的精确定义

### 核心观点

**`add_conditional_edges(source, path_fn, path_map)` 三个参数，搞定 80% 的动态路由。**

### 三个参数拆解

```python
builder.add_conditional_edges(
    source,      # 从哪个节点出发
    path,        # 路由函数：接收 state，返回目标节点名
    path_map,    # 可选：返回值 → 节点名的映射字典
)
```

### 两种写法对比

```python
# 写法 1：不用 path_map（返回值 = 节点名）
def route(state):
    if state["score"] > 80:
        return "excellent_handler"  # 必须和节点名完全一致
    return "normal_handler"

builder.add_conditional_edges("evaluator", route)

# 写法 2：用 path_map（返回语义化的值，映射到节点名）
def route(state):
    if state["score"] > 80:
        return "excellent"  # 语义化返回值
    return "normal"

builder.add_conditional_edges("evaluator", route, {
    "excellent": "excellent_handler",  # 映射到实际节点名
    "normal": "normal_handler",
})
```

### 前端类比

条件边就像 React Router 的路由守卫：

```javascript
// React Router 守卫
<Route path="/dashboard" element={
  isLoggedIn ? <Dashboard /> : <Navigate to="/login" />
} />
```

根据用户状态（登录/未登录）跳转不同页面，和条件边根据 state 选择目标节点是一个道理。

### 关键记忆点

路由函数是纯函数：接收 state，返回字符串。不要在里面做副作用操作。

## 卡片 3/10：Send 的核心机制

### 核心观点

**`Send(node, arg)` 在运行时创建动态并行任务，任务数量编译时未知。**

### 为什么需要 Send

普通并行（fan-out）在编译时就确定了分支数量：

```python
# 静态并行：编译时就知道有 2 个分支
builder.add_edge(START, "worker_a")
builder.add_edge(START, "worker_b")
# 永远是 2 个并行任务，不多不少
```

但如果任务数量取决于运行时数据呢？比如用户上传了 5 个文档，你需要并行处理 5 个；下次上传 100 个，就需要 100 个并行任务。这就是 Send 的用武之地。

### 代码示例

```python
from langgraph.types import Send

def fan_out_to_workers(state):
    """运行时决定并行任务数量"""
    return [
        Send("process_doc", {"doc": doc, "index": i})
        for i, doc in enumerate(state["documents"])
        # 3 个文档 → 3 个 Send → 3 个并行任务
        # 100 个文档 → 100 个 Send → 100 个并行任务
    ]

builder.add_conditional_edges("splitter", fan_out_to_workers)
```

### 日常类比

Send 就像外卖平台的派单系统：

```
中午 12 点：收到 50 个订单 → 派 50 个骑手
下午 3 点：收到 5 个订单 → 派 5 个骑手
骑手数量完全取决于当前订单量（运行时数据）
```

### 关键记忆点

Send 的两个参数：`Send("目标节点", {传递的状态})`。每个 Send 实例携带独立状态，互不干扰。

## 卡片 4/10：Command 的统一接口

### 核心观点

**`Command(goto=, update=, graph=)` 一步完成三件事：去哪里 + 改什么 + 在哪个图。**

### 为什么需要 Command

条件边把路由逻辑和业务逻辑分开了——节点负责处理数据，路由函数负责选路径。但有时候，路由决策和业务处理是紧密耦合的：

```python
# 条件边方式：路由逻辑和业务逻辑分离
def evaluate(state):
    score = calculate_score(state)
    return {"score": score}  # 只能更新状态

def route(state):
    if state["score"] > 0.8:
        return "approve"     # 路由在另一个函数里
    return "reject"

# Command 方式：一步到位
def evaluate(state) -> Command[Literal["approve", "reject"]]:
    score = calculate_score(state)
    if score > 0.8:
        return Command(update={"score": score}, goto="approve")
    return Command(update={"score": score}, goto="reject")
```

### Command 的三个字段

```python
Command(
    update={"key": "value"},    # 更新状态（等价于节点返回值）
    goto="next_node",           # 指定下一个节点
    graph=Command.PARENT,       # 可选：导航到父图
)
```

### 前端类比

Command 就像 Redux 的 `dispatch` + React Router 的 `navigate` 合体：

```javascript
// 前端：需要两步
dispatch({ type: 'UPDATE_SCORE', payload: 0.9 });  // 改状态
navigate('/approve');                                 // 跳页面

// Command：一步搞定
return Command(update={"score": 0.9}, goto="approve")
```

### 关键记忆点

用 Command 的节点不需要 `add_conditional_edges`，也不需要 `add_edge`。Command 自带路由能力。

## 卡片 5/10：三种机制的选择指南

### 核心观点

**条件边 = 固定分支选择，Send = 动态并行创建，Command = 节点自主路由。**

### 决策表

```
你的需求是什么？
│
├─ 根据状态走不同分支（if/else）
│  └─ → 条件边（最简单，最常用）
│
├─ 并行任务数量运行时才知道
│  └─ → Send API（map-reduce 模式）
│
├─ 节点内部需要同时改状态 + 选路径
│  └─ → Command API
│
├─ 子图需要跳回父图
│  └─ → Command + graph=Command.PARENT（唯一选择）
│
└─ 不确定？
   └─ → 先用条件边，不够再升级
```

### 对比速查

| 维度 | 条件边 | Send | Command |
|------|--------|------|---------|
| 复杂度 | 低 | 中 | 高 |
| 路由决策在哪 | 独立函数 | 独立函数 | 节点内部 |
| 能否更新状态 | 不能 | 每个 Send 独立状态 | update 字段 |
| 并行支持 | 返回列表 | 天然支持 | goto 可以是列表 |
| 跨图导航 | 不支持 | 不支持 | 支持 |
| 使用频率 | 80% | 15% | 5% |

### 日常类比

```
条件边 → 红绿灯：根据信号走不同方向，规则固定
Send   → 快递分拣：包裹数量每天不同，动态分配分拣员
Command → 项目经理：自己做完评估后，直接决定下一步谁来接手
```

### 关键记忆点

从简单到复杂：条件边 → Send → Command。大多数场景条件边就够了，别过度设计。

## 卡片 6/10：LLM 路由模式

### 核心观点

**LLM + 结构化输出 + 条件边 = 让 AI 自己决定工作流走向。**

### 核心模式

```python
from pydantic import BaseModel
from typing import Literal

# 第一步：定义路由选项（结构化输出）
class RouteDecision(BaseModel):
    """LLM 必须从这些选项中选一个"""
    next_step: Literal["search", "calculate", "respond"]
    reason: str  # 可选：让 LLM 解释为什么选这个

# 第二步：创建结构化输出的 LLM
router_llm = llm.with_structured_output(RouteDecision)

# 第三步：路由函数调用 LLM 做决策
def llm_router(state):
    decision = router_llm.invoke([
        SystemMessage(content="根据用户问题选择处理方式"),
        HumanMessage(content=state["query"]),
    ])
    return decision.next_step  # 返回 "search" / "calculate" / "respond"

# 第四步：注册为条件边
builder.add_conditional_edges("planner", llm_router)
```

### 为什么用结构化输出

```python
# ❌ 不用结构化输出：LLM 可能返回任意文本
response = llm.invoke("选择下一步")
# 可能返回 "我觉得应该搜索一下" → 无法直接用作路由值

# ✅ 用结构化输出：LLM 只能返回预定义的选项
decision = llm.with_structured_output(RouteDecision).invoke(...)
# 只能返回 "search" / "calculate" / "respond" → 直接用作路由值
```

### 日常类比

LLM 路由就像智能客服的 IVR 系统：

```
传统 IVR：按 1 查余额，按 2 转人工（用户自己选）
智能 IVR：AI 听懂你说的话，自动转接对应部门（AI 帮你选）
```

### 关键记忆点

结构化输出是 LLM 路由的关键——它确保 LLM 返回的值一定是有效的路由选项，不会出现"意外值"。

## 卡片 7/10：图编译的两阶段

### 核心观点

**`StateGraph`(构建) → `compile()`(编译) → `Pregel`(执行)，编译后结构冻结，运行时路径动态。**

### 两阶段拆解

```
┌─────────────────────────────────────────────────┐
│ 阶段 1：构建（Builder Phase）                      │
│                                                   │
│ builder = StateGraph(State)                       │
│ builder.add_node("a", func_a)     # 添加节点      │
│ builder.add_edge(START, "a")      # 添加静态边     │
│ builder.add_conditional_edges(...) # 添加条件边    │
│                                                   │
│ → 可以随意修改，添加/删除节点和边                    │
└──────────────────────┬──────────────────────────┘
                       │ compile()
                       ▼
┌─────────────────────────────────────────────────┐
│ 阶段 2：执行（Runtime Phase）                      │
│                                                   │
│ graph = builder.compile()                         │
│ graph.invoke(state)               # 运行图        │
│                                                   │
│ → 结构不可变，但路径是动态的                         │
│ → 内部生成 Pregel 执行引擎                          │
└─────────────────────────────────────────────────┘
```

### 前端类比

```
StateGraph + compile() → 就像 Webpack 打包
├── 开发时（构建阶段）：随意修改源码、添加模块
├── webpack build（编译）：打包成 bundle，结构固定
└── 浏览器运行（执行阶段）：根据用户操作动态渲染不同页面
```

### compile() 的常用选项

```python
graph = builder.compile(
    checkpointer=MemorySaver(),        # 状态持久化（HITL 必需）
    interrupt_before=["risky_node"],   # 节点前暂停
    interrupt_after=["check_node"],    # 节点后暂停
)
```

### 关键记忆点

`compile()` 是构建和执行的分水岭。编译前随意改，编译后只能跑。

## 卡片 8/10：Reducer 与并行结果合并

### 核心观点

**`Annotated[list, operator.add]` 让并行结果合并而非覆盖，是所有并行场景的必备配置。**

### 问题：没有 Reducer 会怎样

```python
# ❌ 没有 Reducer
class State(TypedDict):
    results: list  # 普通列表

# Worker A 返回 {"results": ["A的结果"]}
# Worker B 返回 {"results": ["B的结果"]}
# 最终 state["results"] = ["B的结果"]  ← A 的结果被覆盖了！
```

### 解决方案

```python
import operator
from typing import Annotated

# ✅ 有 Reducer
class State(TypedDict):
    results: Annotated[list, operator.add]  # 用 operator.add 合并

# Worker A 返回 {"results": ["A的结果"]}
# Worker B 返回 {"results": ["B的结果"]}
# 最终 state["results"] = ["A的结果", "B的结果"]  ← 都保留了！
```

### 常用 Reducer

```python
import operator

# 列表拼接（最常用）
results: Annotated[list, operator.add]

# 自定义：取最大值
def max_reducer(current: int, new: int) -> int:
    return max(current, new)

best_score: Annotated[int, max_reducer]

# 自定义：计数器
def counter(current: int, new: int) -> int:
    return current + new

total_count: Annotated[int, counter]
```

### 日常类比

Reducer 就像会议记录员：

```
没有记录员：每个人发言后，只记住最后一个人说的话
有记录员：  每个人的发言都被记录下来，最后汇总成完整的会议纪要
```

### 关键记忆点

只要有并行（Send 或 fan-out），写同一个字段就必须加 Reducer。记住 `Annotated[list, operator.add]` 这个固定搭配。

## 卡片 9/10：父图导航与多代理切换

### 核心观点

**`Command(graph=Command.PARENT)` 让子图跳回父图，是多代理协作系统的关键机制。**

### 场景：多代理协作

```
┌─────────── 父图（协调器）───────────┐
│                                      │
│  ┌──────────┐    ┌──────────┐       │
│  │ 研究 Agent │ → │ 写作 Agent │      │
│  │ （子图）   │    │ （子图）   │      │
│  └──────────┘    └──────────┘       │
│       ↑                ↑             │
│       └── Command.PARENT ──┘        │
│           子图完成后跳回父图          │
└──────────────────────────────────────┘
```

### 代码示例

```python
from langgraph.types import Command
from typing import Literal

# 子图中的节点：完成后跳回父图
def research_agent(state) -> Command[Literal["writer_agent"]]:
    """研究完成后，把结果交给写作 Agent"""
    research_result = do_research(state["topic"])
    return Command(
        update={"research": research_result},  # 更新状态
        goto="writer_agent",                   # 目标：父图中的另一个子图
        graph=Command.PARENT,                  # 关键：导航到父图
    )
```

### 没有 Command.PARENT 会怎样

```python
# ❌ 不用 Command.PARENT：goto 只能跳到当前图的节点
return Command(goto="writer_agent")
# 报错！因为 writer_agent 不在当前子图中

# ✅ 用 Command.PARENT：跳到父图，再由父图路由到 writer_agent
return Command(goto="writer_agent", graph=Command.PARENT)
```

### 日常类比

```
公司组织架构：
├── CEO（父图/协调器）
│   ├── 研发部（子图 A）
│   │   └── 研发完成后，报告给 CEO，CEO 安排市场部跟进
│   └── 市场部（子图 B）
│       └── 收到 CEO 的指令，开始推广

Command.PARENT = 向上级汇报，由上级协调下一步
```

### 关键记忆点

`Command.PARENT` 是子图到父图的唯一通道。没有它，子图就是一个封闭的黑盒，无法主动影响外部流程。

## 卡片 10/10：动态图设计原则

### 核心观点

**简单用条件边，并行用 Send，自主路由用 Command——从简单开始，按需升级。**

### 设计决策树

```
需要动态路由吗？
│
├─ 不需要 → 用静态边 add_edge()
│
└─ 需要
   │
   ├─ 分支数量编译时已知？
   │  ├─ 是 → 条件边 add_conditional_edges()
   │  └─ 否 → Send API（动态并行）
   │
   ├─ 路由和业务逻辑紧耦合？
   │  ├─ 是 → Command API
   │  └─ 否 → 条件边（路由逻辑分离更清晰）
   │
   ├─ 需要跨子图通信？
   │  └─ → Command + graph=Command.PARENT
   │
   └─ 需要 AI 决策？
      └─ → LLM + 结构化输出 + 条件边
```

### 生产级系统的三条原则

```python
# 原则 1：最小权力原则
# 能用条件边就不用 Command，能用静态边就不用条件边
builder.add_edge("a", "b")                    # 优先：静态边
builder.add_conditional_edges("a", route)     # 其次：条件边
# Command                                     # 最后：Command

# 原则 2：路由函数保持纯净
def route(state) -> str:
    # ✅ 只读取状态，返回字符串
    return "node_a" if state["flag"] else "node_b"

    # ❌ 不要在路由函数里做副作用
    # send_email()  # 不要！
    # db.write()    # 不要！

# 原则 3：类型注解帮助可视化
def node(state) -> Command[Literal["a", "b"]]:  # ✅ 标注可能的目标
    ...
# 这样 graph.get_graph().draw_mermaid() 能正确画出所有可能的边
```

### 速记口诀

```
动态图三板斧，场景不同选不同：
条件边最常用，if/else 分支走；
Send 开并行路，数量运行时才有；
Command 管全局，改状态定方向。
编译定可能，运行选路径，
简单到复杂，按需再升级。
```

### 关键记忆点

动态图构建的核心就一句话：**编译时定义所有可能的路径，运行时根据状态选择走哪条。** 三种机制只是实现这个目标的不同工具。
