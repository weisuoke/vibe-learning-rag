## 核心价值

### 为什么需要动态Schema？

在实际的RAG系统和向量检索应用中，我们经常遇到以下场景：

**场景1：元数据需求不确定**
```python
# 初始设计：只有标题和内容
{
    "id": 1,
    "vector": [0.1, 0.2, ...],
    "title": "Document 1",
    "content": "..."
}

# 业务演化：需要添加作者、分类、标签
{
    "id": 1,
    "vector": [0.1, 0.2, ...],
    "title": "Document 1",
    "content": "...",
    "author": "Alice",      # 新增字段
    "category": "tech",     # 新增字段
    "tags": ["AI", "ML"]    # 新增字段
}
```

**传统方案的问题**：
- ❌ 需要重建Collection
- ❌ 数据迁移成本高
- ❌ 影响系统可用性

**动态Schema的优势**：
- ✅ 无需修改Schema即可添加字段
- ✅ 零停机时间
- ✅ 灵活应对业务变化

---

**场景2：多租户系统**
```python
# 租户A：电商文档
{
    "id": 1,
    "vector": [...],
    "title": "Product 1",
    "price": 99.99,        # 租户A特有字段
    "stock": 100           # 租户A特有字段
}

# 租户B：技术文档
{
    "id": 2,
    "vector": [...],
    "title": "API Doc",
    "version": "v1.0",     # 租户B特有字段
    "language": "Python"   # 租户B特有字段
}
```

**动态Schema的价值**：
- ✅ 不同租户可以有不同的元数据字段
- ✅ 无需为每个租户创建独立Collection
- ✅ 降低管理复杂度

---

**场景3：大规模向量存储**
```python
# 使用Float32向量：内存占用100%
vector_float32 = [0.123456, 0.234567, ...]  # 4 bytes/dim

# 使用Int8向量：内存占用25%
vector_int8 = [12, 23, ...]  # 1 byte/dim

# 节省75%内存，可存储4倍数据量
```

**Int8向量的价值**：
- ✅ 75%内存节省（生产案例验证）
- ✅ 4x查询性能提升
- ✅ 5-10%召回率下降（可接受）

#### 5. Float16/BFloat16向量
- Float16：半精度浮点向量，内存占用为Float32的1/2
- BFloat16：Brain Float16向量，保留Float32的指数范围
- 应用场景：平衡精度和内存
- 精度说明：Float16适合带宽受限场景，BFloat16适合深度学习

**向量类型对比**：
| 类型 | 内存占用 | 精度 | 适用场景 |
|------|---------|------|---------|
| Float32 | 100% | 高 | 高精度需求 |
| Float16 | 50% | 中 | 带宽受限 |
| BFloat16 | 50% | 中 | 深度学习 |
| Int8 | 25% | 低 | 大规模存储 |

## 常见问题

### Q1: 动态Schema会影响性能吗？

**A**: 会有一定影响，但可以优化：

**性能影响**：
- 动态字段的group_by可能有bug（Issue #47438）
- nullable字段聚合慢4倍（Issue #47566）
- 动态字段无法创建索引

**优化建议**：
- 核心字段使用固定Schema
- 业务字段使用动态字段
- 高频查询字段避免使用nullable
- 在应用层处理缺失字段

---

### Q2: Int8向量的精度损失可以接受吗？

**A**: 对于大多数场景是可以接受的：

**精度损失**：
- 5-10%召回率下降（生产案例）
- 对于Top-K检索影响较小
- 可以通过ReRank补偿

**适用场景**：
- 大规模向量存储（百万级以上）
- 内存受限的环境
- 对检索质量要求不是极高的场景

**不适用场景**：
- 小规模数据（几万条）
- 内存充足的环境
- 对检索质量要求极高的场景

---

### Q3: AddCollectionField会阻塞查询吗？

**A**: 不会阻塞，但有注意事项：

**操作特性**：
- 字段添加后几乎立即可用
- 不会阻塞查询操作
- 已存在数据的新字段值为NULL或默认值

**注意事项**：
- 避免与并发insert操作冲突
- 添加字段后重新加载Collection
- 测试所有查询类型（特别是HybridSearch）
- 更新数据导入流程

---

### Q4: 如何选择多租户隔离策略？

**A**: 根据租户数量选择：

**小规模（<100租户）**：
- 使用Collection级别隔离
- 每个租户独立的Collection
- 完全隔离，易于管理

**中规模（100-10K租户）**：
- 使用Partition级别隔离
- 共享Collection，独立Partition
- 平衡隔离性和资源开销

**大规模（>10K租户）**：
- 使用元数据过滤隔离
- 共享Collection，使用租户ID过滤
- 最小资源开销，需要严格权限控制
