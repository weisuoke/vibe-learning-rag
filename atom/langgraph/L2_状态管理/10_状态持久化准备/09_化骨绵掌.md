## 卡片1：什么是状态持久化准备

**一句话**：状态持久化准备就是在设计 LangGraph 状态时，从一开始就考虑"这个状态能不能被安全、高效地保存和恢复"。

**类比**：就像搬家前的打包准备。你不能等到搬家那天才发现有些东西装不进箱子（不可序列化）、有些东西太大搬不动（状态过大）、有些东西不能让搬家公司看到（敏感数据）。好的搬家准备是提前规划：什么能装箱、怎么装、贵重物品怎么保护。

**为什么重要？**

LangGraph 在每个节点执行后自动保存状态（checkpoint）。如果状态中有不可序列化的对象，运行时直接报错；如果状态太大，每次 checkpoint 都很慢；如果敏感数据没加密，数据库里全是明文。这些问题在开发阶段用 `InMemorySaver` 时不会暴露，一上生产就炸。

**持久化解锁的四大能力**：
- 中断后恢复执行（容错）
- 时间旅行调试（回溯任意节点）
- 人机协作工作流（等待人类审批）
- 跨会话记忆（长期对话）

**2 分钟记住**：状态持久化准备 = 从第一天就为"保存和恢复"设计状态，避免上生产才发现问题。

## 卡片2：可序列化类型清单

**一句话**：LangGraph 的 `JsonPlusSerializer` 能处理大部分 Python 内置类型和常见库类型，但不是所有对象都能序列化。

**支持的类型清单**：

```
✅ 基础类型：str, int, float, bool, None, bytes, bytearray
✅ 容器类型：list, dict, tuple, set, frozenset, deque
✅ 标准库：UUID, Decimal, Path, datetime, date, time, timedelta, timezone
✅ 正则表达式：re.Pattern
✅ 网络类型：IPv4Address, IPv4Network 等
✅ 枚举：Enum 子类
✅ 数据类：dataclass, NamedTuple
✅ Pydantic：BaseModel (v1 和 v2)
✅ NumPy：ndarray（通过 memoryview 零拷贝）
✅ LangGraph：Send 对象
```

**不支持的类型**（默认配置下）：

```
❌ 自定义类（非 dataclass/Pydantic/NamedTuple）
❌ 函数、lambda、闭包
❌ 文件句柄、数据库连接
❌ 线程锁、信号量
❌ 生成器、协程对象
```

**解决不可序列化类型的方法**：

```python
# ❌ 自定义类，序列化会失败
class SearchResult:
    def __init__(self, text, score):
        self.text = text
        self.score = score

# ✅ 方案1：改用 Pydantic BaseModel（推荐）
from pydantic import BaseModel

class SearchResult(BaseModel):
    text: str
    score: float

# ✅ 方案2：改用 dataclass
from dataclasses import dataclass

@dataclass
class SearchResult:
    text: str
    score: float

# ✅ 方案3：改用 TypedDict（最轻量）
from typing_extensions import TypedDict

class SearchResult(TypedDict):
    text: str
    score: float
```

**2 分钟记住**：Pydantic/dataclass/TypedDict/NamedTuple 都能序列化，普通自定义类不行。遇到序列化问题，首选改用 Pydantic。

## 卡片3：JsonPlusSerializer 的秘密——它用的是 msgpack

**一句话**：尽管名字叫 "JsonPlus"，LangGraph 的默认序列化器实际主要使用 msgpack 二进制格式，而不是 JSON。

**为什么叫 JsonPlus 却用 msgpack？**

历史原因。早期版本确实用 JSON，后来为了性能升级到 msgpack，但类名保留了。这是一个典型的"名字会骗人"的例子。

**msgpack vs JSON 对比**：

| 维度 | JSON | msgpack |
|------|------|---------|
| 格式 | 文本（UTF-8） | 二进制 |
| 大小 | 较大 | 更小（通常小 30-50%） |
| 速度 | 较慢 | 更快（编解码都快） |
| 可读性 | 人类可读 | 不可读（需要工具） |
| 类型支持 | 有限（string/number/bool/null/array/object） | 支持 ExtType 扩展 |

**序列化流程**：

```python
# JsonPlusSerializer.dumps_typed() 的决策树
def dumps_typed(self, obj):
    if obj is None:
        return ("null", b"")
    if isinstance(obj, bytes):
        return ("bytes", obj)
    if isinstance(obj, bytearray):
        return ("bytearray", bytes(obj))
    # 默认路径：尝试 msgpack
    try:
        encoded = ormsgpack.packb(obj, default=self._default)
        return ("msgpack", encoded)
    except Exception:
        if self.pickle_fallback:
            return ("pickle", pickle.dumps(obj))
        raise  # 无法序列化，抛出异常
```

**type_tag 的作用**：返回的元组第一个元素是类型标签，告诉反序列化器"这些字节应该用什么方式解码"。这是一个自描述格式——数据自己携带解码方法。

**2 分钟记住**：JsonPlusSerializer 的名字是历史遗留，实际用 msgpack。记住 `dumps_typed` 返回 `(type_tag, bytes)` 这个核心接口。

## 卡片4：类型标签系统——7 种 ExtType

**一句话**：JsonPlusSerializer 用 7 种 msgpack 扩展类型码（ExtType 0-6）处理 Python 的复杂类型，每种对应不同的对象重建策略。

**7 种 ExtType 速查表**：

| ExtType | 覆盖类型 | 重建策略 | 示例 |
|---------|----------|----------|------|
| 0 | UUID, Decimal, set, frozenset, deque, Enum, IP | `Class(single_arg)` | `UUID("abc-123")` |
| 1 | Path, regex, timedelta, date, timezone, Send | `Class(*args)` | `Path("/tmp", "file.txt")` |
| 2 | NamedTuple, time, dataclass, Item | `Class(**kwargs)` | `MyData(name="x", value=1)` |
| 3 | datetime | `Class.method(arg)` | `datetime.fromisoformat("2024-01-01")` |
| 4 | Pydantic v1 模型 | `Class(**dict)` | `MyModel(**{"name": "x"})` |
| 5 | Pydantic v2 模型 | `Class(**dict)` | `MyModel(**{"name": "x"})` |
| 6 | NumPy ndarray | `np.frombuffer + reshape` | 零拷贝恢复数组 |

**编码格式**：

```python
# 每个 ExtType 的编码内容是一个 msgpack 数组：
# [module_path, class_name, method_name (可选), constructor_args]

# 例如 Pydantic v2 模型：
# ExtType(5, msgpack.packb(["myapp.models", "SearchResult", None, {"text": "hello", "score": 0.9}]))

# 反序列化时：
# 1. 导入 myapp.models.SearchResult
# 2. 调用 SearchResult(**{"text": "hello", "score": 0.9})
```

**安全机制**：反序列化时导入模块受 `allowed_json_modules` 白名单控制，防止恶意数据触发危险模块的导入。

**2 分钟记住**：7 种 ExtType = 7 种"重建配方"。Pydantic 用 ExtType 4/5，dataclass 用 ExtType 2，NumPy 用 ExtType 6。

## 卡片5：状态大小的隐形杀手

**一句话**：状态大小爆炸是 LangGraph 生产环境的 #1 问题，主要元凶是消息历史和工具调用输出。

**为什么会爆炸？**

LangGraph 在每个节点执行后自动 checkpoint。如果你的 Agent 有 10 轮对话，每轮调用 3 个工具，每个工具返回 2KB 数据：

```
10 轮 × (用户消息 + AI 消息 + 3 个工具调用 + 3 个工具结果) = 80 条消息
80 条消息 × ~2KB = ~160KB 每个 checkpoint
每轮对话产生多个 checkpoint → 总存储快速膨胀
```

**真实案例**（来自 Reddit 社区）：

> "我的 Agent 状态因为工具使用快速增长到几 MB，导致每次 checkpoint 都很慢，数据库也在膨胀。" —— r/LangChain

**大小优化策略**：

```
策略1：消息修剪 → 只保留最近 N 条消息
策略2：外部存储卸载 → 大对象（文档、图片）存 S3，状态只存引用
策略3：摘要替代 → 用 LLM 将旧消息压缩为摘要
策略4：checkpoint 压缩 → AWS DynamoDB 的 enable_checkpoint_compression
策略5：定期清理 → prune() 方法删除旧 checkpoint
```

**快速检查**：

```python
import sys

def check_state_size(state: dict) -> None:
    """检查状态大小，超过阈值发出警告"""
    size = sys.getsizeof(str(state))  # 粗略估算
    if size > 100_000:  # 100KB
        print(f"⚠️ 状态大小 {size/1024:.1f}KB，建议优化")
    if size > 1_000_000:  # 1MB
        print(f"🚨 状态大小 {size/1024:.1f}KB，必须优化！")
```

**2 分钟记住**：消息历史 + 工具输出 = 状态大小的两大杀手。生产环境必须有修剪策略。

## 卡片6：消息修剪策略

**一句话**：消息修剪是控制状态大小最直接的手段，核心是在保留足够上下文和控制大小之间找平衡。

**三种修剪策略**：

**策略1：保留最近 N 条**

```python
def trim_messages_by_count(state):
    """保留最近 N 条消息"""
    messages = state["messages"]
    max_messages = 20
    if len(messages) > max_messages:
        # 保留系统消息 + 最近的消息
        system_msgs = [m for m in messages if m.type == "system"]
        recent_msgs = messages[-max_messages:]
        return {"messages": system_msgs + recent_msgs}
    return {}
```

**策略2：按 Token 数修剪**

```python
def trim_messages_by_tokens(state):
    """按 Token 数修剪，确保不超过上下文窗口"""
    messages = state["messages"]
    max_tokens = 4000
    total_tokens = 0
    kept = []
    # 从最新的消息开始保留
    for msg in reversed(messages):
        msg_tokens = len(msg.content) // 4  # 粗略估算
        if total_tokens + msg_tokens > max_tokens:
            break
        kept.insert(0, msg)
        total_tokens += msg_tokens
    return {"messages": kept}
```

**策略3：摘要压缩**

```python
def summarize_old_messages(state):
    """将旧消息压缩为摘要"""
    messages = state["messages"]
    if len(messages) > 30:
        old_messages = messages[:-10]
        recent_messages = messages[-10:]
        # 用 LLM 生成摘要
        summary = llm.invoke(f"请总结以下对话要点：{old_messages}")
        summary_msg = SystemMessage(content=f"之前的对话摘要：{summary}")
        return {"messages": [summary_msg] + recent_messages}
    return {}
```

**什么时候用哪种？**

| 场景 | 推荐策略 | 原因 |
|------|----------|------|
| 简单问答 | 保留最近 N 条 | 简单高效 |
| 长对话 Agent | 按 Token 数修剪 | 精确控制上下文窗口 |
| 需要长期记忆 | 摘要压缩 | 保留关键信息 |

**2 分钟记住**：简单场景用数量限制，精确场景用 Token 限制，需要记忆用摘要压缩。

## 卡片7：EncryptedSerializer 加密链

**一句话**：`EncryptedSerializer` 是一个装饰器，包装任何序列化器并在序列化之后、存储之前插入加密步骤。

**加密流程**：

```
写入（dumps_typed）：
obj → serde.dumps_typed() → ("msgpack", bytes)
    → cipher.encrypt(bytes) → ("msgpack+aes", ciphertext)

读取（loads_typed）：
("msgpack+aes", ciphertext) → cipher.decrypt() → ("msgpack", bytes)
    → serde.loads_typed() → obj
```

**type_tag 的变化**：

```python
# 加密前
("msgpack", b"\x92\xa5hello\xa5world")

# 加密后（+aes 后缀）
("msgpack+aes", b"<nonce:16B><tag:16B><ciphertext>")
```

**向后兼容设计**：

```python
# 反序列化时检查 type_tag
def loads_typed(self, data):
    type_tag, payload = data
    if "+" in type_tag:
        # 有 + 号 → 需要解密
        base_type, cipher_name = type_tag.rsplit("+", 1)
        decrypted = self.cipher.decrypt(payload)
        return self.serde.loads_typed((base_type, decrypted))
    else:
        # 没有 + 号 → 未加密，直接反序列化
        return self.serde.loads_typed(data)
```

这意味着你可以在已有的未加密 checkpoint 上启用加密——旧数据照常读取，新数据加密存储。

**使用方式**：

```python
from langgraph.checkpoint.serde.encrypted import EncryptedSerializer

# 从环境变量 LANGGRAPH_AES_KEY 读取密钥
serde = EncryptedSerializer.from_pycryptodome_aes()

# 传给任何 checkpointer
checkpointer = SqliteSaver(conn, serde=serde)
```

**2 分钟记住**：EncryptedSerializer = 装饰器模式 + AES-EAX 加密 + `+` 分隔符实现向后兼容。

## 卡片8：SecretStr 的陷阱

**一句话**：Pydantic 的 `SecretStr` 只防止日志泄露，不防止存储泄露——在 LangGraph 序列化时会被提取为明文。

**陷阱演示**：

```python
from pydantic import BaseModel, SecretStr

class AgentState(BaseModel):
    api_key: SecretStr
    query: str

state = AgentState(api_key="sk-secret-123", query="什么是 RAG？")

# ✅ 日志安全：print 和 repr 不会泄露
print(state)
# → api_key=SecretStr('**********') query='什么是 RAG？'

# ❌ 序列化不安全：JsonPlusSerializer 会提取明文
# 源码中的处理逻辑（简化）：
# if isinstance(obj, SecretStr):
#     return obj.get_secret_value()  # 返回明文 "sk-secret-123"
```

**SecretStr 的设计意图**：

SecretStr 是为了防止在日志、调试输出、错误报告中意外泄露敏感信息。它不是加密方案，不提供存储保护。

**正确的做法**：

```python
# 层次1：SecretStr 防日志泄露（必要但不充分）
class State(BaseModel):
    api_key: SecretStr

# 层次2：EncryptedSerializer 防存储泄露（关键）
serde = EncryptedSerializer.from_pycryptodome_aes()
checkpointer = SqliteSaver(conn, serde=serde)

# 层次3：源头控制——API key 根本不该进入状态
# 用 context_schema 传递，不创建 Channel，不被持久化
class ContextSchema(TypedDict):
    api_key: str  # 走 Runtime，不走 Channel
```

**记忆口诀**：

```
SecretStr  → 防"看到"（日志、repr）
Encrypted  → 防"存到"（数据库、文件）
不放状态   → 防"进入"（根本不持久化）
```

**2 分钟记住**：SecretStr 只是"遮羞布"，不是"保险箱"。真正的安全需要 EncryptedSerializer 或者根本不把敏感数据放进状态。

## 卡片9：三层持久化架构

**一句话**：LangGraph 的持久化是三层关注点分离——数据层产生快照、序列化层转换格式、存储层写入后端。

**三层架构图**：

```
┌─────────────────────────────────────────────┐
│  第一层：数据层（BaseChannel）                │
│  职责：知道如何产生和消费可序列化的状态快照     │
│  关键方法：checkpoint() / from_checkpoint()   │
│  优化：空通道返回 MISSING，不进入快照          │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│  第二层：序列化层（SerializerProtocol）        │
│  职责：将 Python 对象转换为字节并返回          │
│  默认：JsonPlusSerializer（msgpack 编码）      │
│  可选：EncryptedSerializer（加密装饰器）       │
│  接口：dumps_typed() → (type_tag, bytes)      │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│  第三层：存储层（BaseCheckpointSaver）         │
│  职责：将序列化字节持久化到后端               │
│  实现：InMemorySaver / SqliteSaver /          │
│        AsyncPostgresSaver / Redis / DynamoDB  │
│  功能：CRUD + 版本管理 + prune               │
└─────────────────────────────────────────────┘
```

**为什么要分三层？**

```
分离的好处：
1. 换存储不影响序列化 → 从 SQLite 切到 PostgreSQL，序列化代码不变
2. 换序列化不影响存储 → 加上加密层，存储代码不变
3. 换数据结构不影响其他 → 新增 Channel 类型，序列化和存储不变
```

**Checkpoint 数据结构**：

```python
checkpoint = {
    "v": 2,                          # 格式版本
    "id": "1ef4f797-8335-...",       # uuid6，单调递增
    "ts": "2024-07-31T20:14:19...",  # ISO 8601 时间戳
    "channel_values": {...},         # 序列化的通道快照
    "channel_versions": {...},       # 每个通道的版本跟踪
    "versions_seen": {...},          # 每个节点看到的版本
}
```

**2 分钟记住**：三层分离 = 数据（Channel）+ 格式（Serializer）+ 存储（Saver），各换各的互不影响。

## 卡片10：生产环境检查清单

**一句话**：上生产前，用这个检查清单确保你的状态设计已经为持久化做好准备。

**检查清单**：

```
═══ 可序列化检查 ═══
□ 状态中所有字段都是可序列化类型？
  → str/int/float/bool/list/dict/Pydantic/dataclass/TypedDict
□ 没有函数、lambda、文件句柄、数据库连接在状态中？
□ 自定义类已改为 Pydantic BaseModel 或 dataclass？
□ pickle_fallback 保持关闭（默认）？

═══ 大小控制检查 ═══
□ 消息历史有修剪策略？（数量限制/Token限制/摘要压缩）
□ 大对象（文档、图片、音频）存外部存储，状态只存引用？
□ 工具调用结果有大小限制？（截断过长的输出）
□ 监控了 checkpoint 大小的增长趋势？
□ 配置了 checkpoint prune 策略？

═══ 敏感数据检查 ═══
□ API keys 和密码不在状态中？（用 context_schema）
□ 用户 PII 已脱敏或只存引用 ID？
□ 生产环境启用了 EncryptedSerializer？
□ LANGGRAPH_AES_KEY 安全存储（不在代码中硬编码）？
□ 密钥长度至少 32 字节（AES-256）？

═══ 后端选择检查 ═══
□ 开发环境：InMemorySaver 或 SqliteSaver？
□ 生产环境：AsyncPostgresSaver 或 Redis？
□ 配置了连接池？
□ 设置了 checkpoint TTL（过期清理）？

═══ 测试检查 ═══
□ 状态的序列化/反序列化往返测试通过？
□ 加密/解密往返测试通过？
□ 大状态的性能测试通过？（checkpoint 延迟 < 100ms）
□ 故障恢复测试通过？（kill 进程后能从 checkpoint 恢复）
```

**快速验证脚本**：

```python
from langgraph.checkpoint.serde.jsonplus import JsonPlusSerializer

serde = JsonPlusSerializer()

def verify_serializable(state: dict) -> bool:
    """验证状态是否可序列化"""
    try:
        for key, value in state.items():
            type_tag, data = serde.dumps_typed(value)
            restored = serde.loads_typed((type_tag, data))
            print(f"  ✅ {key}: {type_tag} ({len(data)} bytes)")
        return True
    except Exception as e:
        print(f"  ❌ 序列化失败: {e}")
        return False

# 使用示例
test_state = {
    "query": "什么是 RAG？",
    "messages": [{"role": "user", "content": "hello"}],
    "score": 0.95,
}
verify_serializable(test_state)
```

**2 分钟记住**：上生产前过一遍检查清单——可序列化、大小可控、敏感数据安全、后端合适、测试通过。

| 7 | EncryptedSerializer | 装饰器模式 + AES-EAX + `+` 分隔符向后兼容 |
| 8 | SecretStr 陷阱 | 只防日志泄露，不防存储泄露 |
| 9 | 三层架构 | 数据(Channel) + 格式(Serializer) + 存储(Saver) |
| 10 | 生产检查清单 | 可序列化 + 大小可控 + 敏感安全 + 后端合适 |
