# Agent最佳实践 - 面试必问

> 这些问题是 Agent 最佳实践领域面试中的高频考点，掌握它们不仅帮你通过面试，更能检验你对 Agent 系统的理解深度。

---

## Q1：生产环境中 Agent 最常见的三个问题是什么？如何解决？

### 考察点

对 Agent 在生产环境中面临的实际挑战的理解。

### 标准回答

**三个最常见问题：无限循环、API 不稳定、上下文溢出。**

```
问题 1：无限循环（Agent 反复调用同一工具）
  原因：LLM 推理卡在局部最优，反复尝试无效操作
  解决：ModelCallLimitMiddleware(run_limit=25)
        + ToolCallLimitMiddleware(per-tool limit)
  原理：设置调用上限，到达后优雅结束

问题 2：API 不稳定（超时、限流、服务宕机）
  原因：LLM API 受网络、负载等影响
  解决：ModelRetryMiddleware(max_retries=3, 指数退避+抖动)
        + ModelFallbackMiddleware(备用模型链)
  原理：瞬时故障自动重试，持续故障自动降级

问题 3：上下文溢出（长对话超出 Token 限制）
  原因：多轮对话累积的消息越来越多
  解决：SummarizationMiddleware(trigger=60%, keep=5)
  原理：旧消息自动压缩为摘要，保留最近 N 条完整消息
```

### 加分回答

在 LangChain 1.0 中，这三个问题都通过 **Middleware 架构** 统一解决，而不是分散在各处配置——这是 Middleware 模式相比 AgentExecutor 的最大优势。

## Q2：Middleware 的执行顺序重要吗？为什么？

### 考察点

对 Middleware 洋葱模型（Onion Model）的理解。

### 标准回答

**非常重要！顺序决定行为。**

```python
# ❌ 错误顺序：重试在限制外面
middleware=[
    ModelRetryMiddleware(max_retries=3),     # 外层
    ModelCallLimitMiddleware(run_limit=25),   # 内层
]
# 后果：每次重试不计入限制！3×25 = 可能调用 75 次

# ✅ 正确顺序：限制在重试外面
middleware=[
    ModelCallLimitMiddleware(run_limit=25),   # 外层
    ModelRetryMiddleware(max_retries=3),      # 内层
]
# 重试也受限制管控，总调用不超过 25 次
```

**记忆口诀**："限制在外（保底），重试在中（容错），安全在内（靠近数据）"

**原理**：Middleware 采用洋葱模型——声明越靠前的越靠外，外层 Middleware 包裹内层，外层先执行检查，内层的行为都在外层的管控范围内。

## Q3：ModelRetryMiddleware 的 max_retries 应该设多少？为什么？

### 考察点

对重试策略的工程判断力。

### 标准回答

**一般设 3 次，最多不超过 5 次。**

```
max_retries=3 的总等待时间（指数退避）：
  1s + 2s + 4s = 约 7 秒 → 用户可接受

max_retries=10 的总等待时间：
  1+2+4+8+16+30+30+30+30+30 = 约 181 秒 = 3 分钟 → 用户早走了

关键认知：如果 3 次重试都失败了，问题大概率不是瞬时故障，
          应该降级到备用模型（ModelFallbackMiddleware），
          而不是继续重试同一个不可用的服务。
```

### 加分回答

重试配合降级才是完整的可靠性方案：`Retry(3)` 处理瞬时故障，`Fallback(备用模型)` 处理持久故障。两者缺一不可。

## Q4：PII 脱敏为什么要同时处理输入和输出？

### 考察点

对 LLM 数据安全风险的全面理解。

### 标准回答

**因为 PII 泄露有两个方向：用户输入泄露 + LLM 输出泄露。**

```
方向 1：用户输入包含 PII → 需要 before_model 脱敏
  用户：「帮我查 13812345678 的账户」
  → before_model 脱敏为 138****5678 再传给 LLM

方向 2：LLM 输出包含 PII → 需要 after_model 脱敏
  场景 a：LLM 从训练数据中「回忆」出真实手机号
  场景 b：工具返回的数据包含 PII，LLM 直接透传给用户
  → after_model 检查输出，发现 PII 自动脱敏

只做一个方向 = 只锁了前门没锁后门
```

PIIMiddleware 的设计正好对应这两个方向：`before_model()` 钩子处理输入，`after_model()` 钩子处理输出。

## Q5：Agent 的调用限制设 25 会不会太少？

### 考察点

对 Agent 运行模式的理解。

### 标准回答

**对大多数场景来说，25 次足够了。**

```
正常 Agent 执行：
  步骤 1: 理解问题           ← 1 次 LLM 调用
  步骤 2: 调用工具            ← 1 次工具调用
  步骤 3: 分析工具返回的结果  ← 1 次 LLM 调用
  步骤 4: 生成最终答案       ← 1 次 LLM 调用
  → 4 步完成，远低于 25 次

异常 Agent 执行：
  步骤 1-25: 反复搜索同一个关键词...
  → 大概率已经陷入死循环

关键认知：
  如果 Agent 需要 25 步以上，要么是任务太复杂（应该拆分），
  要么是 Agent 陷入了循环（应该终止）。
```

### 加分回答

不同场景可以调整：

| 场景 | 推荐 run_limit |
|------|---------------|
| 简单问答 | 15 |
| 客服对话 | 25 |
| 数据分析 | 50 |
| 研究报告 | 80 |

## Q6：Callback 和 Middleware 有什么区别？各自适用什么场景？

### 考察点

对 LangChain 两种扩展机制的理解。

### 标准回答

```
Callback = 观察者（只看不动）
  • 只能观察事件，不能修改数据
  • 适用：日志、监控、追踪（LangSmith Tracing）
  • 不影响执行流程
  • 即使 Callback 报错也不会中断 Agent（raise_error=False）

Middleware = 拦截者（能看能改能拦）
  • 可以修改请求和响应
  • 适用：重试、降级、PII 脱敏、人工审核、上下文管理
  • 直接影响执行流程
  • Middleware 出错会影响 Agent 执行

选择标准：
  想记录发生了什么 → Callback
  想改变会发生什么 → Middleware
```

### 加分回答

类比：

| 机制 | 类比 | 权限 |
|------|------|------|
| Callback | 安保摄像头 | 只能看，不能干预 |
| Middleware | 安保人员 | 可以拦人、检查、拒绝进入 |

## Q7：工具的 description 怎么写才能让 LLM 正确使用？

### 考察点

对 LLM 工具选择机制的理解。

### 标准回答

**使用黄金公式：用途 + 输入 + 输出 + 限制。**

```python
# ❌ 模糊的 description
description = "A useful tool"  # LLM 完全不知道该不该用

# ❌ 只有用途
description = "搜索知识库"  # 缺少输入输出信息

# ✅ 黄金公式
description = """搜索客服知识库获取产品信息和政策文档。
输入：用户问题的关键词。
输出：最相关的知识库文章摘要。
限制：只能搜索已有知识库内容，不能回答实时信息。"""
```

**为什么 description 比 System Prompt 更重要？**

LLM 选择工具的决策链：
1. 看到用户的问题
2. **扫描所有工具的 description**（主要依据）
3. 根据 description 判断哪个工具匹配
4. System Prompt 只提供整体引导，不影响具体工具选择

## Q8：什么时候该用多 Agent？什么时候不该用？

### 考察点

对 Agent 架构选型的工程判断。

### 标准回答

```
需要多 Agent 的信号：
  ✅ 工具超过 15 个 → LLM 选择准确率下降
  ✅ 需要不同的权限级别 → 不同 Agent 不同权限
  ✅ 需要不同的专业领域 → 不同 Agent 不同 System Prompt
  ✅ 需要并行处理 → 多 Agent 可以并行执行

不需要多 Agent 的信号：
  ❌ 单 Agent + 少量工具就够了 → 不要过度工程化
  ❌ 任务是线性的 → 不需要分支和并行
  ❌ 多 Agent 的复杂度 > 它解决的问题

多 Agent 的隐藏成本：
  1. 调试困难（错误在哪个 Agent？消息传递是否正确？）
  2. 延迟翻倍（Agent 间通信需要额外 LLM 调用）
  3. 上下文割裂（每个 Agent 只看到自己的上下文）
  4. 成本倍增（多个 Agent = 多倍 Token 消耗）
```

### 加分回答

LangChain 官方定义了四种多 Agent 模式：Subagents（子代理）、Handoffs（切换）、Skills（技能）、Router（路由器）。选择哪种取决于任务结构——独立子任务用 Subagents，对话角色切换用 Handoffs。

## Q9：AgentExecutor 和 create_agent 有什么区别？为什么要迁移？

### 考察点

对 LangChain 架构演进的理解。

### 标准回答

```
AgentExecutor（旧，已弃用）：
  • 错误处理分散在多个配置项
  • 没有内置的重试、降级、PII、人工审核
  • 可扩展性受限（只能用 Callback 观察，不能拦截修改）

create_agent（新，LangChain 1.0 推荐）：
  • 统一的 Middleware 架构
  • 内置 14 个 Middleware（重试/降级/PII/HITL/摘要等）
  • 基于 LangGraph，支持持久化、流式、人机协作
  • 配置集中在 middleware=[] 列表中

迁移收益：
  • 代码更简洁：安全防护从「自己写」变成「加一行 Middleware」
  • 更可靠：指数退避、模型降级等都是开箱即用
  • 更安全：PII 脱敏、人工审核等内置支持
  • 更易维护：所有配置集中管理
```

### 加分回答

关键代码变化：

```python
# 旧：AgentExecutor（分散配置）
agent_executor = AgentExecutor(
    agent=agent, tools=tools,
    max_iterations=15, handle_parsing_errors=True,
)

# 新：create_agent（Middleware 集中管理）
agent = create_agent(
    "gpt-4.1", tools=tools,
    middleware=[
        ModelCallLimitMiddleware(run_limit=25),
        ModelRetryMiddleware(max_retries=3),
        ModelFallbackMiddleware(fallbacks=["gpt-4.1-mini"]),
    ],
)
```

## Q10：如何测试一个 Agent？测试策略是什么？

### 考察点

对 Agent 质量保障体系的理解。

### 标准回答

**Agent 测试分五层（测试金字塔）：**

```
        /\
       /  \  红队测试
      /    \  - 提示注入攻击
     /------\  - 越狱尝试
    /        \
   / 端到端测试 \  - 完整对话流程
  /            \  - 多步骤任务
 /--------------\
/   集成测试      \  - Agent + 真实工具
/                  \  - 错误处理和重试
/--------------------\
/     单元测试         \  - 工具函数逻辑
/                       \  - 条件路由
/-------------------------\  - Middleware 行为
```

```
1. 单元测试：测试每个工具函数的输入输出，Mock LLM 调用
2. 集成测试：测试 Agent 与真实工具的交互，验证错误处理
3. 端到端测试：完整对话流程，验证多步骤任务完成
4. 红队测试：对抗性攻击测试（提示注入、越狱）
5. 评估(Eval)：使用 LangSmith 定义评估指标（工具选择准确率、任务完成率）
```

### 加分回答

Agent 测试的特殊性在于 LLM 输出不确定性——因此：
- 单元测试要 Mock LLM（确保确定性）
- 集成/E2E 测试要多次运行取统计结果
- 评估指标比精确匹配更重要（"意思对了"比"字完全一样"更有意义）
