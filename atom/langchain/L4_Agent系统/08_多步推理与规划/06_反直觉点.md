# 多步推理与规划 - 反直觉点

> LangChain Agent 系统 · 第 8 个知识点 · 维度 6/10

---

## 反直觉点 1：规划不是越详细越好

### 直觉想法

"既然要规划，那就规划得越详细越好，把每一步都想清楚。"

### 为什么是错的

```
过度规划的问题：
  目标："对比 FastAPI 和 Django"

  过度规划（20步）：
  1. 搜索 FastAPI 官网
  2. 记录 FastAPI 版本号
  3. 搜索 FastAPI 异步特性
  4. 记录 FastAPI 异步特性
  5. 搜索 FastAPI 性能数据
  ...
  20. 格式化最终报告

  问题：
  - 20 次执行器调用 → 成本高、速度慢
  - 步骤过于机械 → LLM 的灵活性被浪费
  - 早期步骤的结果可能让后续步骤变得不合理
```

### 正确理解

```
适度规划（5步）：
  1. 分别搜索 FastAPI 和 Django 的核心特性
  2. 对比两者的性能和架构
  3. 分析各自的适用场景
  4. 综合分析给出建议
  5. 生成对比报告

  优点：
  - 每步有足够灵活度 → 执行器可自主决定具体动作
  - 3-7 步是最佳范围
  - 粗粒度规划 + 细粒度执行 = 最优策略
```

**核心原则：规划负责"做什么"（What），执行负责"怎么做"（How）。不要在规划层面决定执行细节。**

## 反直觉点 2：Plan-and-Execute 不一定比 ReAct 好

### 直觉想法

"Plan-and-Execute 更高级，应该总是比 ReAct 好。"

### 为什么是错的

```
场景1：用户问 "今天天气怎么样？"
  ReAct：Thought→search(天气)→回答 ← 1次LLM + 1次工具 ✅
  P&E：Plan[搜索天气]→Execute[搜索]→Replan→回答 ← 3次LLM + 1次工具 ❌

场景2：用户问 "帮我查一下快递到哪了"
  ReAct：Thought→query(快递号)→回答 ← 快速完成 ✅
  P&E：Plan→Execute→Replan ← 过度设计 ❌

场景3：探索性任务 "帮我研究一下最新的 AI 发展"
  ReAct：每步动态决策 → 灵活适应新发现 ✅
  P&E：预先规划 → 可能错过执行中发现的新方向 ❌（需要频繁重规划）
```

### 正确理解

| 任务类型 | 推荐 | 原因 |
|---------|------|------|
| 1-2步简单任务 | ReAct | P&E 的规划开销大于收益 |
| 3步以上复杂任务 | P&E | 全局规划的优势开始显现 |
| 完全探索性任务 | ReAct 或迭代分解 | 无法预先规划 |
| 可预期的工作流 | P&E | 计划质量高 |

**核心原则：简单任务用 ReAct，复杂任务用 P&E，不是"谁更高级"的问题，而是"谁更合适"的问题。**

## 反直觉点 3：重规划不是失败，是特性

### 直觉想法

"好的规划应该一次到位，需要重规划说明规划失败了。"

### 为什么是错的

```
现实世界的不确定性：
  - 搜索结果可能与预期不同（搜索"Python性能" → 结果是Python教程）
  - 依赖的服务可能不可用
  - 用户可能中途改变需求
  - 早期步骤的结果可能改变后续方向

完美规划的幻觉：
  "先搜索A再搜索B" → 搜索A后发现，A已被C替代
  → 如果不能重规划，Agent 会继续搜索已过时的A的细节

重规划的价值：
  → 重规划不是"失败后的补救"
  → 而是"根据新信息做出更好决策"
  → 就像GPS导航：走错路了会重新规划路线，这是特性，不是bug
```

### 正确理解

```
好的 P&E Agent = 合理初始规划 + 适时重规划

设计哲学：
  1. 初始计划只需"大致正确"（80%准确度就够）
  2. 执行过程中收集信息
  3. 根据新信息动态调整
  4. 最终路径 ≠ 初始计划，但结果更好

重规划次数的经验值：
  0 次 → 任务太简单，可能不需要 P&E
  1-2 次 → 正常范围，说明有适度调整
  3+ 次 → 初始规划质量可能需要提升
  5+ 次 → 规划器有问题，需要优化 Prompt
```

**核心原则：规划是"起点"不是"终点"。好的 Agent 不是不需要重规划，而是能够优雅地重规划。**

## 三个反直觉点总结

| # | 直觉 | 现实 |
|---|------|------|
| 1 | 规划越详细越好 | 3-7步粗粒度规划最优 |
| 2 | P&E 总是比 ReAct 好 | 简单任务用 ReAct 更合适 |
| 3 | 重规划 = 规划失败 | 重规划是应对不确定性的核心特性 |

---

## 额外洞察：Agent 规划的"人类偏见"

```
LLM 规划的一个隐藏问题：

人类写 Prompt："请分解这个任务"
LLM 倾向于：
  - 生成 5-10 个步骤（因为训练数据中的列表通常是这个长度）
  - 按线性顺序排列（不擅长识别可并行的任务）
  - 遗漏"验证"步骤（人类也经常忘记）

解决方案：
  1. 在 Prompt 中明确限制步骤数
  2. 要求 LLM 识别可并行的步骤
  3. 强制要求最后一步是"验证/审查"
```
