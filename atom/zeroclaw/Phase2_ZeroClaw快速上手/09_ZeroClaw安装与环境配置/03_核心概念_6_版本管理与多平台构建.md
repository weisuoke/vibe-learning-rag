# 核心概念 6：版本管理与多平台构建

> **一句话定义：** 版本管理通过 `rustup` + `rust-toolchain.toml` 固定 Rust 工具链版本（MSRV 1.87），多平台构建通过 CI/CD 流水线将同一份源码交叉编译为 4+ 个平台的二进制产物，确保 ZeroClaw v0.1.7 在 Linux/macOS/Windows、x86_64/aarch64 上都能一致运行。

---

## 1. Rust 版本管理

### 1.1 为什么需要版本管理？

Rust 语言每 6 周发布一个新版本（比 Node.js 频繁得多），新版本可能引入新语法或改变行为。如果团队成员用不同版本编译，可能出现"你的机器能编译，我的不行"的问题。

```
Node.js 世界：                      Rust 世界：
nvm install 20                      rustup install 1.92.0
nvm use 20                          rustup default 1.92.0
.nvmrc → "20"                       rust-toolchain.toml → "1.92.0"
package.json.engines.node            Cargo.toml.rust-version (MSRV)
```

> **前端类比：** `rustup` 就是 Rust 的 `nvm`——管理多个版本、一键切换。`rust-toolchain.toml` 就是 `.nvmrc`——打开项目目录自动切换到正确版本。

> **日常生活类比：** 版本管理像烹饪学校要求所有学生用同一版本的教材——每个人翻到第 3 章讲的都是同一道菜，不会因为版本不同而做出不同的东西。

### 1.2 rustup 核心命令

```bash
# ===== 安装管理 =====

# 安装 rustup（Rust 版本管理器）
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# 查看已安装的工具链
rustup toolchain list
# 输出示例：
# stable-aarch64-apple-darwin (default)
# 1.87.0-aarch64-apple-darwin
# 1.92.0-aarch64-apple-darwin
# nightly-aarch64-apple-darwin

# ===== 版本切换 =====

# 安装特定版本
rustup toolchain install 1.92.0

# 设置全局默认版本
rustup default 1.92.0

# 仅在当前目录使用特定版本（目录级覆盖）
rustup override set 1.87.0

# 查看当前活跃版本
rustup show
# 或
rustc --version
```

### 1.3 Rust 的三个 Channel（发布通道）

```
stable ─────── 每 6 周发布，生产环境用这个
  │              例：1.87.0, 1.88.0, 1.92.0
  │
beta ────────── 下一个 stable 的候选版本
  │              例：测试新功能是否稳定
  │
nightly ─────── 每天发布，包含实验性功能
                 例：使用还没稳定的新语法
```

| Channel | 前端类比 | 适用场景 |
|---------|---------|---------|
| stable | Node.js LTS（18, 20） | 生产代码、日常开发 |
| beta | Node.js Current | 测试即将发布的功能 |
| nightly | Chrome Canary | 实验性功能、编译器开发 |

> ZeroClaw 只使用 **stable** channel，不依赖任何 nightly 特性。

### 1.4 rust-toolchain.toml（项目级版本锁定）

ZeroClaw 仓库根目录有一个 `rust-toolchain.toml` 文件：

```toml
# rust-toolchain.toml — 项目级 Rust 版本锁定
[toolchain]
channel = "1.92.0"                        # 固定版本号
targets = ["x86_64-unknown-linux-gnu"]    # 需要的编译目标（可选）
components = ["rustfmt", "clippy"]         # 附带的工具（可选）
```

**工作原理：**

```
你 cd 到 ZeroClaw 目录
  │
  ├── rustup 检测到 rust-toolchain.toml
  │
  ├── 本地有 1.92.0？
  │   ├── 有 → 自动切换到 1.92.0
  │   └── 没有 → 自动下载 1.92.0 → 切换
  │
  └── 所有 cargo 命令自动使用 1.92.0
      不影响其他项目的 Rust 版本
```

> **前端类比：** 这和 `.nvmrc` + `nvm auto` 完全一样——进入项目目录，自动切换到项目指定的版本。VSCode 的 rust-analyzer 也会读取这个文件。

## 2. MSRV 策略

### 2.1 什么是 MSRV？

**MSRV = Minimum Supported Rust Version**（最低支持的 Rust 版本）

```toml
# Cargo.toml
[package]
name = "zeroclaw"
version = "0.1.7"
rust-version = "1.87"    # ← MSRV：用 1.87 或更高版本都能编译
```

### 2.2 MSRV 的意义

```
Rust 1.80    1.85    1.87(MSRV)    1.90    1.92(CI)    1.95
  │          │        │             │        │           │
  ╳          ╳        ✅ ←──────── ✅ ───── ✅ ──────── ✅
  不支持     不支持   最低版本      支持      CI 使用     支持
                      保证编译                 最新稳定
```

| 概念 | 值 | 说明 |
|------|---|------|
| MSRV | 1.87 | 最低要求：低于此版本无法编译 |
| CI 工具链 | 1.92.0 | CI/CD 实际使用的版本 |
| rust-toolchain.toml | 1.92.0 | 推荐开发者使用的版本 |

### 2.3 为什么 MSRV 是 1.87 而不是最新版？

```
三个核心理由：

1. 兼容性：
   └── 有些用户/系统的 Rust 版本可能不是最新的
       例：Debian stable 的包管理器提供的 Rust 可能滞后

2. 依赖兼容：
   └── 上游依赖（crate）也有自己的 MSRV
       如果 ZeroClaw 要求最新版，可能和某些依赖冲突

3. 稳定性承诺：
   └── "只要你的 Rust ≥ 1.87，ZeroClaw 就一定能编译"
       这是对用户的一个可靠承诺
```

> **前端类比：** MSRV 就像 `package.json` 中的 `engines: { "node": ">=18" }`——告诉用户"我们支持 Node 18 及以上"。你可能用 Node 22 开发，但保证 18 也能跑。

> **日常生活类比：** MSRV 就像一个蛋糕配方上写的"烤箱温度 180°C 以上"——你可以用 200°C 的烤箱（更新的版本），但至少要 180°C（MSRV）才能烤好。

### 2.4 MSRV 的验证

```bash
# 在 CI 中验证 MSRV 兼容性
rustup toolchain install 1.87.0
cargo +1.87.0 check          # 用 MSRV 版本检查是否能编译

# 如果某个新语法只在 1.88+ 可用，这里就会报错
# 迫使开发者要么放弃那个语法，要么提升 MSRV
```

## 3. ZeroClaw 的版本号（语义化版本）

### 3.1 版本号格式

```
ZeroClaw v0.1.7
           │ │ │
           │ │ └── Patch（补丁）：修 bug、小优化
           │ └──── Minor（次版本）：新功能，向后兼容
           └────── Major（主版本）：0 = 还在快速迭代期
```

| 版本段 | 含义 | 前端类比 |
|--------|------|---------|
| Major = 0 | API 可能会变（0.x 不保证稳定） | React 0.x 时代，API 经常大改 |
| Minor = 1 | 第一组功能集 | 第一个 feature release |
| Patch = 7 | 第 7 次修复/改进 | 第 7 次 bugfix |

> **注意：** Major 版本为 0 意味着 ZeroClaw 还在**快速迭代期**。API 可能在 0.2.0 发生不兼容变更。当 Major 升到 1.0.0 时，表示 API 稳定了。

### 3.2 Cargo.toml 中的版本

```toml
[package]
name = "zeroclaw"
version = "0.1.7"          # ZeroClaw 自身版本
edition = "2024"           # Rust Edition（语言特性集合）
rust-version = "1.87"      # MSRV
```

| 字段 | 作用 | 前端类比 |
|------|------|---------|
| `version` | ZeroClaw 的发布版本 | `package.json` 的 `version` |
| `edition` | Rust 语言特性集合（2015/2018/2021/2024） | `tsconfig.json` 的 `target` |
| `rust-version` | 最低 Rust 编译器版本 | `engines.node` |

### 4.3 构建矩阵（Build Matrix）

```yaml
# .github/workflows/release.yml（简化示意）
strategy:
  matrix:
    include:
      - target: x86_64-unknown-linux-gnu
        os: ubuntu-latest
        linker: mold
      - target: aarch64-unknown-linux-gnu
        os: ubuntu-latest
        linker: mold
        cross: true                        # 需要交叉编译
      - target: aarch64-apple-darwin
        os: macos-latest
      - target: x86_64-pc-windows-msvc
        os: windows-latest
```

> **前端类比：** Build Matrix 就像 GitHub Actions 中测试多个 Node 版本：
> ```yaml
> strategy:
>   matrix:
>     node-version: [18, 20, 22]
> ```
> 只不过 Rust 的矩阵维度是"目标平台"而不是"运行时版本"。

### 4.4 Linker 选择

| Linker | 平台 | 特点 | 前端类比 |
|--------|------|------|---------|
| `mold` | Linux | 极速链接（比 `ld` 快 5-10x） | `esbuild` 替代 `webpack` |
| `clang/lld` | macOS | Apple 平台默认 | Xcode 自带工具链 |
| `link.exe` | Windows | MSVC 默认 | Visual Studio 自带 |

```bash
# 安装 mold（Linux）
sudo apt install mold

# 在 .cargo/config.toml 中配置使用 mold
# [target.x86_64-unknown-linux-gnu]
# linker = "clang"
# rustflags = ["-C", "link-arg=-fuse-ld=mold"]
```

> **为什么选 mold？** Rust 的编译瓶颈之一是链接阶段。mold 是最快的开源链接器，可以将链接时间从几十秒降到几秒，在 CI/CD 中节省大量时间。

## 5. 交叉编译

### 5.1 什么是交叉编译？

**交叉编译（Cross Compilation）**= 在一台机器上编译出另一台机器（不同 CPU / 操作系统）能运行的程序。

```
你的开发机（macOS M2）          目标设备（Linux x86_64 服务器）
┌──────────────┐                ┌──────────────┐
│ aarch64      │  交叉编译 →    │ x86_64       │
│ Apple Darwin │  ─────────→    │ Linux GNU    │
│              │                │              │
│ 编译在这里做  │                │ 程序在这里跑  │
└──────────────┘                └──────────────┘
```

> **前端类比：** 前端开发者其实一直在做"交叉编译"——你在 macOS 上写 JS 代码，但产物（HTML/CSS/JS）在用户的 Windows Chrome、iOS Safari 上运行。Rust 只是把这个过程显式化了。

> **日常生活类比：** 交叉编译就像在自家厨房（你的开发机）做便当（编译二进制），带到办公室（目标服务器）吃。你不需要在办公室安装厨房设备（不需要在服务器上装 Rust 编译器）。

### 5.2 Target Triple 格式解析

每个编译目标用一个 **target triple** 字符串描述：

```
aarch64-unknown-linux-gnu
   │       │       │    │
   │       │       │    └── ABI（应用二进制接口）
   │       │       │        gnu = GNU libc
   │       │       │        musl = musl libc（静态链接）
   │       │       │        msvc = Microsoft Visual C++
   │       │       │
   │       │       └── 操作系统
   │       │           linux, windows, darwin(macOS)
   │       │
   │       └── 厂商
   │           unknown = 通用
   │           apple = Apple
   │           pc = 通用 PC
   │
   └── CPU 架构
       x86_64 = Intel/AMD 64位
       aarch64 = ARM 64位（M1/M2、树莓派 4+）
       armv7 = ARM 32位（树莓派 3）
```

### 5.3 ZeroClaw 支持的 Target Triples

| Target Triple | CPU | 系统 | 常见设备 |
|--------------|-----|------|---------|
| `x86_64-unknown-linux-gnu` | Intel/AMD 64 | Linux | 服务器、PC |
| `aarch64-unknown-linux-gnu` | ARM 64 | Linux | ARM 服务器、树莓派 4+ |
| `aarch64-apple-darwin` | Apple Silicon | macOS | M1/M2/M3/M4 Mac |
| `x86_64-pc-windows-msvc` | Intel/AMD 64 | Windows | Windows PC |

### 5.4 交叉编译实操

#### 方式 1：rustup 添加 target

```bash
# 查看已安装的 target
rustup target list --installed

# 添加新的编译目标
rustup target add aarch64-unknown-linux-gnu

# 交叉编译
cargo build --release --target aarch64-unknown-linux-gnu
# ⚠️ 需要对应平台的链接器和系统库（见下方）
```

#### 方式 2：cross 工具（推荐）

[`cross`](https://github.com/cross-rs/cross) 是一个交叉编译辅助工具，自动使用 Docker 容器提供目标平台的工具链：

```bash
# 安装 cross
cargo install cross

# 交叉编译（自动拉取对应的 Docker 镜像）
cross build --release --target aarch64-unknown-linux-gnu
# cross 自动处理：
# 1. 拉取包含 aarch64 工具链的 Docker 镜像
# 2. 在容器中编译
# 3. 输出到本地 target/ 目录
```

```
使用 cross 的流程：

你的 macOS/Linux 开发机
│
├── cross build --target aarch64-unknown-linux-gnu
│   │
│   ├── Docker 拉取 cross 镜像
│   │   └── 镜像包含：aarch64 GCC + 系统库 + Rust target
│   │
│   ├── 在容器中编译
│   │
│   └── 产物输出到 target/aarch64-unknown-linux-gnu/release/zeroclaw
│
└── scp zeroclaw user@arm-server:~/
    └── 直接在 ARM 服务器上运行
```

> **前端类比：** `cross` 就像 `docker run -v $(pwd):/app node:20 npm run build`——在标准化容器中构建，保证环境一致。

#### 方式 3：Docker 多阶段构建中的交叉编译

```dockerfile
# 在 x86_64 的 CI 服务器上构建 ARM64 版本
FROM --platform=linux/amd64 rust:1.92-slim AS builder

# 安装交叉编译工具链
RUN apt-get update && apt-get install -y \
    gcc-aarch64-linux-gnu \
    libc6-dev-arm64-cross

# 添加 target
RUN rustup target add aarch64-unknown-linux-gnu

# 设置链接器
ENV CARGO_TARGET_AARCH64_UNKNOWN_LINUX_GNU_LINKER=aarch64-linux-gnu-gcc

COPY . .
RUN cargo build --release --target aarch64-unknown-linux-gnu

# 运行阶段
FROM --platform=linux/arm64 debian:bookworm-slim
COPY --from=builder \
    /app/target/aarch64-unknown-linux-gnu/release/zeroclaw \
    /usr/local/bin/
CMD ["zeroclaw"]
```

### 5.5 交叉编译常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `linker 'aarch64-linux-gnu-gcc' not found` | 没装交叉编译器 | `apt install gcc-aarch64-linux-gnu` |
| OpenSSL 链接错误 | 缺少目标平台的 OpenSSL | ZeroClaw 用 rustls（纯 Rust TLS），不依赖 OpenSSL |
| `cross` 构建失败 | Docker 未运行 | 启动 Docker Desktop |
| 目标平台系统库缺失 | libc 版本不匹配 | 使用 `musl` target（静态链接） |

### 6.2 与前端 CI/CD 的对比

| 步骤 | ZeroClaw (Rust) | 前端 (TypeScript/React) |
|------|-----------------|----------------------|
| 格式检查 | `cargo fmt --check` | `prettier --check` |
| Lint | `cargo clippy` | `eslint .` |
| 类型检查 | 编译器自带 | `tsc --noEmit` |
| 测试 | `cargo test` | `jest` / `vitest` |
| 构建 | `cargo build --release` | `npm run build` |
| 多平台 | Build Matrix（4 targets） | 通常不需要（JS 跨平台） |
| 发布 | GitHub Releases + crates.io | npm publish |

> **关键差异：** 前端项目通常只需一次构建（JS 跨平台运行），Rust 项目需要为每个目标平台单独编译。这就是 Build Matrix 存在的原因。

```bash

cargo build
# → 使用 rust-toolchain.toml 指定的版本
# → 不需要手动 rustup install / rustup override

# 4. 运行测试
cargo test
```

### 7.2 版本管理最佳实践

| 做法 | ✅ 推荐 | ❌ 避免 |
|------|--------|--------|
| 指定版本 | `rust-toolchain.toml` 锁定到具体版本 | 只说"用 stable" |
| MSRV 策略 | 定期评估并谨慎提升 | 每次发布都要求最新 Rust |
| CI 验证 | 同时测试 MSRV 和最新版 | 只测最新版 |
| 依赖管理 | `cargo update` + 测试后再提交 `Cargo.lock` | 随意 `cargo update` |

### 7.3 多平台构建的意义

```
ZeroClaw 的定位：在任何地方运行的 AI Agent 框架

          ┌── 云服务器 (x86_64-linux-gnu)
          │      └── 24/7 知识库助手
          │
          ├── 办公电脑 (x86_64-windows-msvc / aarch64-apple-darwin)
ZeroClaw ─┤      └── 本地开发 & 个人助手
          │
          ├── ARM 服务器 (aarch64-linux-gnu)
          │      └── AWS Graviton / 边缘节点
          │
          └── 树莓派 (aarch64-linux-gnu / armv7)
                 └── IoT 智能家居控制中心

一份源码 → 4+ 个平台 → 统一的 ZeroClaw 体验
```

### 7.4 开发者需要知道的

| 你的角色 | 需要关心什么 |
|---------|------------|
| **使用者** | 下载对应平台的预编译二进制即可，不需要了解交叉编译 |
| **贡献者** | 确保代码在 MSRV 1.87 下编译通过，CI 会自动验证 |
| **维护者** | 管理 Build Matrix、更新 MSRV、维护 Release 流水线 |
| **部署工程师** | 选择正确的 target triple，配置 Docker 多阶段构建 |

## 8. 类比总结表

| ZeroClaw 概念 | 前端类比 | 日常生活类比 |
|--------------|---------|------------|
| rustup | nvm（Node 版本管理） | 教材版本管理（指定用第几版） |
| rust-toolchain.toml | .nvmrc | 课程指定教材（打开目录自动匹配） |
| stable/beta/nightly | LTS/Current/Canary | 正式版/测试版/实验版软件 |
| MSRV 1.87 | `engines.node >= 18` | 烤箱最低温度要求 |
| Target Triple | — | 快递地址格式（国家-省-市-区） |
| 交叉编译 | — （JS 天然跨平台） | 在自家厨房做便当带去办公室 |
| cross 工具 | Docker + node 多版本构建 | 租用标准化厨房做便当 |
| Build Matrix | `matrix: { node: [18,20] }` | 同一配方适配不同烤箱 |
| mold linker | esbuild 替代 webpack | 高速公路 vs 普通公路 |
| `cargo fmt` | `prettier` | 统一排版格式 |
| `cargo clippy` | `eslint` | 语法审查员 |
| 语义化版本 0.1.7 | `package.json` version | 软件迭代日记 |
