## 反直觉点 1：max_attempts=3 不是重试 3 次

### ❌ 错误认知

"设了 `max_attempts=3`，失败后会重试 3 次，总共执行 4 次。"

### ✅ 真相

**`max_attempts=3` 表示总共尝试 3 次（含首次执行），只重试 2 次。**

```
max_attempts=3 的实际执行：
  尝试1（首次执行）→ 失败
  尝试2（第1次重试）→ 失败
  尝试3（第2次重试）→ 失败 → 抛出异常

总共 3 次尝试，2 次重试！
```

**源码验证：**

```python
# 来自 pregel/_retry.py
if attempt >= retry_policy.max_attempts:
    raise  # attempt 从 1 开始计数
```

**类比：** "你最多可以考 3 次" 意味着第一次考试 + 最多 2 次补考。

## 反直觉点 2：不传 retry_policy 不等于"默认重试"

### ❌ 错误认知

"LangGraph 应该默认就有重试机制，毕竟框架已经内置了。"

### ✅ 真相

**不传 `retry_policy` 参数，节点完全不会重试，任何异常直接抛出。**

```python
# ❌ 没有重试！
builder.add_node("my_node", my_func)

# ✅ 有重试！必须显式传入
builder.add_node("my_node", my_func, retry_policy=RetryPolicy())
```

**设计原因：** 重试是有成本的（时间、资源），不应该默认开启。某些节点（纯计算、确定性操作）根本不需要重试。

## 反直觉点 3：GraphBubbleUp 不是"错误"，但用 Exception 实现

### ❌ 错误认知

"所有 Exception 子类都是错误，catch 到就应该处理。"

### ✅ 真相

**GraphBubbleUp 继承自 Exception，但它是"流程控制信号"，不是错误。**

```python
# 它是故意抛出的"暂停信号"
from langgraph.types import interrupt

def human_review(state):
    decision = interrupt("请审核")  # ← 这会引发 GraphInterrupt（GraphBubbleUp 子类）
    return {"approved": decision}
```

**LangGraph 在重试逻辑中特意跳过它：**

```python
try:
    execute_node()
except GraphBubbleUp:
    raise  # ← 不重试！直接向上传播
except Exception:
    # 只有真正的"异常"才进入重试逻辑
```

**类比：** 红灯停车不是"汽车故障"，虽然都让车停下来，但处理方式完全不同。

## 反直觉点 4：默认重试策略会重试"未知异常"

### ❌ 错误认知

"默认策略只重试明确列出的异常类型（如 ConnectionError），其他不重试。"

### ✅ 真相

**默认的 `default_retry_on` 对无法识别的异常会选择重试（保守策略）。**

```python
def default_retry_on(exc: Exception) -> bool:
    if isinstance(exc, ConnectionError): return True      # 明确重试
    if isinstance(exc, (ValueError, TypeError, ...)): return False  # 明确不重试
    return True  # ← 未知异常？默认重试！
```

**设计哲学：**

```
两种策略的对比：

激进策略：未知异常不重试 → 可能因为一个可恢复的异常导致系统崩溃
保守策略：未知异常重试   → 最坏情况是多浪费几秒钟

LangGraph 选择了保守策略：宁可多试几次，也不要轻易放弃。
```

## 反直觉点 5：重试前会清除上次的状态写入

### ❌ 错误认知

"重试只是简单地再执行一次函数，之前的中间结果还在。"

### ✅ 真相

**每次重试前，LangGraph 会清除上次执行的所有写入（`task.writes.clear()`），确保重试是"干净"的。**

```python
# 来自 pregel/_retry.py 的 run_with_retry()
while True:
    try:
        task.writes.clear()  # ← 关键！清除上次的写入
        task.proc.invoke(task.input, config)
        break
    except Exception:
        # 重试逻辑...
```

**为什么重要？**

```
假设节点执行到一半写了部分数据然后失败：
  节点写入 {"count": 1} → 失败

如果不清除，重试时：
  节点又写入 {"count": 1} → 可能变成 {"count": 2}（reducer 累加了！）

清除后，重试时从干净状态开始：
  节点写入 {"count": 1} → 正确！
```

## 反直觉点 6：429 状态码默认不被 LangGraph 重试

### ❌ 错误认知

"429 (Too Many Requests) 是最典型的'应该重试'的错误，LangGraph 肯定默认处理了。"

### ✅ 真相

**`default_retry_on` 只重试 5xx（500-599），429 属于 4xx，默认不重试。**

```python
# 源码
if isinstance(exc, httpx.HTTPStatusError):
    return 500 <= exc.response.status_code < 600  # 429 不在范围内！
```

**但这是合理的！原因：**

```
1. OpenAI SDK 自带 429 重试（带 Retry-After 解析）
2. Anthropic SDK 也自带 429 重试
3. LangGraph 层面再重试会导致"双重重试"

如果你确实需要，自定义 retry_on：
```

```python
def retry_with_rate_limit(exc):
    if isinstance(exc, httpx.HTTPStatusError):
        return exc.response.status_code in [429, 500, 502, 503, 504]
    return default_retry_on(exc)
```

## 反直觉点 7：jitter 只加 0~1 秒，看似微不足道但至关重要

### ❌ 错误认知

"随机加 0~1 秒有什么用？跟退避间隔比起来可以忽略不计。"

### ✅ 真相

**Jitter 的作用不在于"让单个客户端等更久"，而在于"让多个客户端错开时间"。**

```
100 个客户端同时遇到故障：

无 Jitter → 第一次全在 0.5s 重试，第二次全在 1.0s 重试
  → 服务器每次收到 100 个请求的"波峰" → 反复过载

有 Jitter → 第一次在 0.5~1.5s 分散重试
  → 服务器收到的是均匀分布的请求 → 逐个处理 → 快速恢复
```

**类比：** 超市只有 2 个收银台，50 个人排队。
- ❌ 50 人同时涌上去 → 混乱
- ✅ 每人随机等 0~30 秒再排 → 有序

## 反直觉点 8：多个重试策略不是"叠加"，是"优先匹配"

### ❌ 错误认知

"传多个 RetryPolicy，会每个都试一遍，重试次数是累加的。"

### ✅ 真相

**多策略是"短路匹配"：找到第一个匹配的策略就用它，其他的完全忽略。**

```python
builder.add_node("my_node", func,
    retry_policy=(
        RetryPolicy(retry_on=ConnectionError, max_attempts=5),  # 策略A
        RetryPolicy(retry_on=ValueError, max_attempts=2),       # 策略B
    )
)

# 遇到 ConnectionError → 用策略A，最多5次
# 遇到 ValueError → 用策略B，最多2次
# 遇到 TypeError → 两个都不匹配 → 直接抛出！
# 不会用 5+2=7 次！
```

## 总结速查

| 误区 | 真相 |
|------|------|
| max_attempts=3 重试 3 次 | 总共 3 次尝试，重试 2 次 |
| 节点默认有重试 | 必须显式传 `retry_policy` |
| GraphBubbleUp 是错误 | 是流程控制信号 |
| 默认只重试已知异常 | 未知异常也重试（保守策略）|
| 重试保留上次状态 | 每次重试前清除写入 |
| 429 默认被重试 | 默认不重试（SDK 自己处理）|
| Jitter 作用小 | 防止多客户端同时重试 |
| 多策略叠加重试次数 | 优先匹配，只用第一个 |
