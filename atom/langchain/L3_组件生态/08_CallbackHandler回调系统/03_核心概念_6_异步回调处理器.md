# 异步回调处理器

## 1. 【30字核心】

**AsyncCallbackHandler 是 LangChain 的异步回调处理器，通过 async/await 实现非阻塞回调，是构建高性能异步 RAG 系统的核心组件。**

## 2. 【第一性原理】

### 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题

### 异步回调处理器的第一性原理

#### 1. 最基础的定义

**AsyncCallbackHandler = 异步回调接口 + 非阻塞执行**

仅此而已！没有更基础的了。

#### 2. 为什么需要异步回调处理器？

**核心问题：同步回调会阻塞主线程，导致性能瓶颈**

在 RAG 系统中，回调处理器需要执行各种操作：
- 记录日志到数据库
- 发送追踪数据到监控平台
- 流式输出到客户端
- 计算 token 成本

如果这些操作是同步的，每个操作都会阻塞主线程，导致：
- LLM 调用变慢
- 用户体验变差
- 系统吞吐量下降

#### 3. 异步回调处理器的三层价值

##### 价值1：非阻塞执行

**同步回调**：
```python
def on_llm_end(self, response):
    # 阻塞主线程，等待数据库写入完成
    db.save(response)  # 可能需要 100ms
```

**异步回调**：
```python
async def on_llm_end(self, response):
    # 不阻塞主线程，立即返回
    await db.save(response)  # 异步执行
```

##### 价值2：高并发支持

在 FastAPI 等异步框架中，异步回调处理器可以：
- 同时处理多个请求
- 不会因为回调操作阻塞其他请求
- 提高系统吞吐量

##### 价值3：流式输出

异步回调处理器是实现流式输出的关键：
- 实时接收 LLM 生成的 token
- 通过异步队列传递给客户端
- 用户可以立即看到生成的内容

#### 4. 从第一性原理推导 FastAPI 流式回调

**推理链：**
```
1. FastAPI 是异步框架，使用 async/await
   ↓
2. 回调处理器需要与 FastAPI 集成
   ↓
3. 回调处理器必须是异步的（AsyncCallbackHandler）
   ↓
4. 流式输出需要实时传递 token
   ↓
5. 使用异步队列（asyncio.Queue）传递 token
   ↓
6. 使用 StreamingResponse 返回流式响应
   ↓
7. 最终实现：AsyncIteratorCallbackHandler + FastAPI
```

#### 5. 一句话总结第一性原理

**异步回调处理器是通过 async/await 实现非阻塞回调执行，解决同步回调的性能瓶颈，是构建高性能异步 RAG 系统的核心基础。**

## 3. 【核心概念】

### 核心概念1：AsyncCallbackHandler 基础

**AsyncCallbackHandler 是 BaseCallbackHandler 的异步版本，所有回调方法都是 async def，支持非阻塞执行。**

```python
from langchain_core.callbacks import AsyncCallbackHandler
from typing import Any

class MyAsyncCallbackHandler(AsyncCallbackHandler):
    """自定义异步回调处理器"""

    async def on_llm_start(
        self,
        serialized: dict[str, Any],
        prompts: list[str],
        **kwargs: Any
    ) -> None:
        """LLM 开始时触发（异步）"""
        print(f"LLM 开始处理 {len(prompts)} 个 prompt")
        # 可以执行异步操作，如数据库写入
        # await db.save_log(...)

    async def on_llm_end(
        self,
        response: Any,
        **kwargs: Any
    ) -> None:
        """LLM 结束时触发（异步）"""
        print(f"LLM 完成，生成了 {len(response.generations)} 个结果")
        # 异步保存结果
        # await db.save_result(...)

    async def on_llm_error(
        self,
        error: BaseException,
        **kwargs: Any
    ) -> None:
        """LLM 错误时触发（异步）"""
        print(f"LLM 错误: {error}")
        # 异步记录错误
        # await error_tracker.log(error)
```

**详细解释**：

AsyncCallbackHandler 继承自 BaseCallbackHandler，但所有方法都是异步的：
- 方法签名使用 `async def` 而非 `def`
- 可以在方法内使用 `await` 调用其他异步函数
- 不会阻塞主线程，适合高并发场景

**在 RAG 开发中的应用**：

在 RAG 系统中，异步回调处理器用于：
- 异步记录 LLM 调用日志到数据库
- 异步发送追踪数据到监控平台
- 异步计算和保存 token 成本
- 异步流式输出到客户端

### 核心概念2：异步方法实现

**异步回调方法通过 async/await 实现非阻塞执行，支持并发处理多个请求。**

```python
import asyncio
from langchain_core.callbacks import AsyncCallbackHandler
from typing import Any

class TokenCounterCallback(AsyncCallbackHandler):
    """异步 Token 计数回调"""

    def __init__(self):
        self.total_tokens = 0
        self.lock = asyncio.Lock()  # 用于并发安全

    async def on_llm_end(
        self,
        response: Any,
        **kwargs: Any
    ) -> None:
        """异步统计 token 使用量"""
        # 从响应中提取 token 数量
        tokens = response.llm_output.get("token_usage", {}).get("total_tokens", 0)

        # 使用锁保证并发安全
        async with self.lock:
            self.total_tokens += tokens

        # 异步保存到数据库
        await self._save_to_db(tokens)

    async def _save_to_db(self, tokens: int):
        """异步保存到数据库"""
        # 模拟异步数据库操作
        await asyncio.sleep(0.1)
        print(f"保存 {tokens} tokens 到数据库")
```

**详细解释**：

异步方法实现的关键点：
1. **async def**：方法定义使用 `async def`
2. **await**：调用其他异步函数时使用 `await`
3. **并发安全**：使用 `asyncio.Lock()` 保护共享状态
4. **非阻塞**：异步操作不会阻塞主线程

**在 RAG 开发中的应用**：

在 RAG 系统中，异步方法实现用于：
- 并发处理多个用户请求
- 异步写入日志和监控数据
- 异步计算成本和性能指标
- 异步流式输出到多个客户端

### 核心概念3：FastAPI 流式回调集成

**FastAPI 流式回调集成通过 AsyncIteratorCallbackHandler 实现实时流式输出。**

```python
import asyncio
from typing import AsyncIterable
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from langchain.callbacks import AsyncIteratorCallbackHandler
from langchain_openai import ChatOpenAI
from langchain.schema import HumanMessage

app = FastAPI()

async def send_message(message: str) -> AsyncIterable[str]:
    """发送消息并流式返回结果"""
    # 创建异步迭代器回调处理器
    callback = AsyncIteratorCallbackHandler()

    # 创建 LLM，传入回调处理器
    model = ChatOpenAI(
        streaming=True,
        callbacks=[callback],
    )

    # 包装异步任务
    async def wrap_done(fn, event):
        try:
            await fn
        except Exception as e:
            print(f"错误: {e}")
        finally:
            event.set()  # 通知迭代器结束

    # 启动后台任务
    task = asyncio.create_task(
        wrap_done(
            model.agenerate(messages=[[HumanMessage(content=message)]]),
            callback.done
        )
    )

    # 流式输出 token
    async for token in callback.aiter():
        yield f"data: {token}\n\n"

    await task

@app.post("/stream")
async def stream(message: str):
    """流式响应端点"""
    return StreamingResponse(
        send_message(message),
        media_type="text/event-stream"
    )
```

**详细解释**：

FastAPI 流式回调集成的关键组件：
1. **AsyncIteratorCallbackHandler**：内置的异步迭代器回调处理器
2. **aiter()**：异步迭代器，逐个返回 token
3. **done**：事件标志，用于通知迭代器结束
4. **StreamingResponse**：FastAPI 的流式响应类

**在 RAG 开发中的应用**：

在 RAG 系统中，FastAPI 流式回调集成用于：
- 实时流式输出 RAG 生成的答案
- 提升用户体验（立即看到生成内容）
- 支持长文本生成（避免超时）
- 实现类似 ChatGPT 的打字效果

## 5. 【双重类比】

### 类比1：AsyncCallbackHandler vs 同步回调

**前端类比：** Promise vs 回调函数

同步回调就像传统的回调函数，会阻塞执行：
```javascript
// 同步回调（阻塞）
function processData(callback) {
    const result = heavyComputation(); // 阻塞
    callback(result);
}
```

异步回调就像 Promise，不会阻塞：
```javascript
// 异步回调（非阻塞）
async function processData(callback) {
    const result = await heavyComputation(); // 不阻塞
    await callback(result);
}
```

**日常生活类比：** 餐厅点餐

同步回调：你在柜台点餐，厨师做完菜才能点下一单（阻塞）
异步回调：你拿号码牌回座位，厨师做好了叫号（非阻塞）

```python
# Python 同步回调
class SyncCallback(BaseCallbackHandler):
    def on_llm_end(self, response, **kwargs):
        db.save(response)  # 阻塞主线程

# Python 异步回调
class AsyncCallback(AsyncCallbackHandler):
    async def on_llm_end(self, response, **kwargs):
        await db.save(response)  # 不阻塞主线程
```

## 6. 【反直觉点】

### 误区1：异步回调处理器可以在多个请求之间重用 ❌

**为什么错？**
- 异步回调处理器可能包含状态（如计数器、队列）
- 多个请求并发执行时，状态会混乱
- 导致数据错乱、trace_id 混淆

**为什么人们容易这样错？**
因为在单线程同步代码中，重用对象是常见的优化手段。人们习惯性地认为创建对象开销大，应该重用。但在异步并发环境中，状态共享会导致竞态条件。

**正确理解：**
```python
# ❌ 错误：重用回调处理器
callback = AsyncIteratorCallbackHandler()  # 全局变量

@app.post("/stream1")
async def stream1(message: str):
    # 多个请求共享同一个 callback，状态混乱
    model = ChatOpenAI(streaming=True, callbacks=[callback])
    async for token in callback.aiter():
        yield token

# ✅ 正确：每个请求新建回调处理器
@app.post("/stream2")
async def stream2(message: str):
    # 每个请求新建 callback，状态隔离
    callback = AsyncIteratorCallbackHandler()
    model = ChatOpenAI(streaming=True, callbacks=[callback])
    async for token in callback.aiter():
        yield token
```

---

### 误区2：异步回调方法会自动并发执行 ❌

**为什么错？**
- 异步方法需要显式使用 `await` 才会执行
- 回调方法是按顺序调用的，不是并发的
- 如果回调方法耗时长，仍然会阻塞

**为什么人们容易这样错？**
因为"异步"这个词容易让人联想到"并发"。人们误以为只要方法是 `async def`，就会自动并发执行。但实际上，异步只是提供了非阻塞的能力，并不等于并发。

**正确理解：**
```python
# ❌ 错误理解：认为这些方法会并发执行
class MyCallback(AsyncCallbackHandler):
    async def on_llm_start(self, serialized, prompts, **kwargs):
        await asyncio.sleep(1)  # 这会阻塞 1 秒

    async def on_llm_end(self, response, **kwargs):
        await asyncio.sleep(1)  # 这也会阻塞 1 秒

# 总耗时：2 秒（顺序执行）

# ✅ 正确理解：回调方法是顺序调用的
# 如果需要并发，使用 asyncio.create_task()
class MyCallback(AsyncCallbackHandler):
    async def on_llm_start(self, serialized, prompts, **kwargs):
        # 创建后台任务，不阻塞
        asyncio.create_task(self._log_to_db(prompts))

    async def _log_to_db(self, data):
        await asyncio.sleep(1)
```

---

### 误区3：AsyncIteratorCallbackHandler 会自动处理流式输出结束 ❌

**为什么错？**
- AsyncIteratorCallbackHandler 需要显式调用 `done.set()` 来结束迭代
- 如果不调用，`aiter()` 会一直等待，导致挂起
- 需要使用 `wrap_done()` 包装异步任务

**为什么人们容易这样错？**
因为在同步代码中，迭代器会自动结束（如 `for` 循环）。人们误以为异步迭代器也会自动检测结束。但实际上，异步迭代器需要显式信号来结束。

**正确理解：**
```python
# ❌ 错误：忘记设置 done 事件
callback = AsyncIteratorCallbackHandler()
model = ChatOpenAI(streaming=True, callbacks=[callback])

# 启动任务但不设置 done
task = asyncio.create_task(model.agenerate(...))

# 这会一直等待，永远不会结束
async for token in callback.aiter():
    print(token)

# ✅ 正确：使用 wrap_done 包装任务
async def wrap_done(fn, event):
    try:
        await fn
    except Exception as e:
        print(f"错误: {e}")
    finally:
        event.set()  # 确保设置 done 事件

callback = AsyncIteratorCallbackHandler()
model = ChatOpenAI(streaming=True, callbacks=[callback])

task = asyncio.create_task(
    wrap_done(
        model.agenerate(...),
        callback.done  # 传入 done 事件
    )
)

# 现在会正常结束
async for token in callback.aiter():
    print(token)

await task
```

## 7. 【实战代码】

```python
"""
异步回调处理器实战示例
演示：FastAPI 流式回调集成
"""

import asyncio
import os
from typing import AsyncIterable
from dotenv import load_dotenv
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from langchain.callbacks import AsyncIteratorCallbackHandler
from langchain_openai import ChatOpenAI
from langchain.schema import HumanMessage
from pydantic import BaseModel

# 加载环境变量
load_dotenv()

app = FastAPI()

# ===== 1. 定义流式消息处理函数 =====
async def send_message(message: str) -> AsyncIterable[str]:
    """
    发送消息并流式返回 LLM 生成的结果

    Args:
        message: 用户输入的消息

    Yields:
        str: 流式输出的 token
    """
    # 创建异步迭代器回调处理器
    callback = AsyncIteratorCallbackHandler()

    # 创建 LLM，启用流式输出并传入回调处理器
    model = ChatOpenAI(
        streaming=True,
        verbose=True,
        callbacks=[callback],
    )

    # 定义包装函数，确保任务完成后设置 done 事件
    async def wrap_done(fn: asyncio.Future, event: asyncio.Event):
        """
        包装异步任务，确保完成后设置事件

        Args:
            fn: 异步任务
            event: 完成事件
        """
        try:
            await fn
        except Exception as e:
            print(f"错误: {e}")
        finally:
            # 通知迭代器结束
            event.set()

    # 启动后台任务生成 LLM 响应
    task = asyncio.create_task(
        wrap_done(
            model.agenerate(messages=[[HumanMessage(content=message)]]),
            callback.done
        )
    )

    # 流式输出 token（Server-Sent Events 格式）
    async for token in callback.aiter():
        yield f"data: {token}\n\n"

@app.post("/stream_with_logging")

    """
    带日志记录的流式响应端点

    Args:
        body: 请求体，包含用户消息

    Returns:
        StreamingResponse: 流式响应
    """
    async def send_with_logging(message: str) -> AsyncIterable[str]:
        # 使用自定义回调处理器
        callback = CustomAsyncCallback()

        model = ChatOpenAI(
            streaming=True,
            callbacks=[callback],
        )

        async def wrap_done(fn, event):
            try:
                await fn
            except Exception as e:
                print(f"错误: {e}")
            finally:
                event.set()

        task = asyncio.create_task(
            wrap_done(
                model.agenerate(messages=[[HumanMessage(content=message)]]),
                callback.done
            )
        )

        async for token in callback.aiter():
            yield f"data: {token}\n\n"

        await task

    return StreamingResponse(
        send_with_logging(body.message),
        media_type="text/event-stream"
    )
```

## 8. 【面试必问】

### 问题："AsyncCallbackHandler 和 BaseCallbackHandler 有什么区别？在什么场景下应该使用异步回调？"

**普通回答（不出彩）：**
"AsyncCallbackHandler 是异步的，BaseCallbackHandler 是同步的。异步回调不会阻塞，性能更好。"

**出彩回答（推荐）：**

> **AsyncCallbackHandler 和 BaseCallbackHandler 有三层区别：**
>
> 1. **方法签名层面**：AsyncCallbackHandler 的所有回调方法都是 `async def`，可以使用 `await` 调用其他异步函数；BaseCallbackHandler 的方法是普通的 `def`，只能同步执行。
>
> 2. **执行模型层面**：AsyncCallbackHandler 基于事件循环（event loop），支持非阻塞 I/O；BaseCallbackHandler 是阻塞式执行，每个操作都会等待完成。
>
> 3. **并发能力层面**：AsyncCallbackHandler 可以在单线程中并发处理多个请求；BaseCallbackHandler 需要多线程或多进程才能实现并发。
>
> **应该使用异步回调的场景**：
>
> - **FastAPI/Starlette 等异步框架**：框架本身是异步的，使用异步回调可以充分利用框架的并发能力
> - **流式输出场景**：需要实时传递 token 到客户端，异步回调配合 AsyncIteratorCallbackHandler 可以实现流式响应
> - **高并发场景**：需要同时处理多个用户请求，异步回调可以在单线程中实现高并发
> - **I/O 密集型操作**：回调中需要执行数据库写入、网络请求等 I/O 操作，异步回调可以避免阻塞
>
> **不应该使用异步回调的场景**：
>
> - **同步框架**：如 Flask（非异步模式），使用异步回调反而会增加复杂度
> - **CPU 密集型操作**：如果回调中主要是计算操作，异步回调不会带来性能提升
> - **简单脚本**：如果只是本地测试或简单脚本，同步回调更简单直接
>
> **在 RAG 开发中的实践**：
>
> 在生产环境的 RAG 系统中，我们通常使用 FastAPI + AsyncCallbackHandler 的组合：
> - FastAPI 提供异步 API 端点
> - AsyncIteratorCallbackHandler 实现流式输出
> - 自定义 AsyncCallbackHandler 异步记录日志和监控数据
> - 这样可以在单个服务器上支持数百个并发请求

**为什么这个回答出彩？**
1. 从三个层面（方法签名、执行模型、并发能力）深入解释了区别
2. 明确列出了应该和不应该使用异步回调的场景
3. 结合 RAG 开发的实际应用，展示了对生产环境的理解
4. 体现了对异步编程模型的深刻理解

## 9. 【化骨绵掌】

### 卡片1：直觉理解 - 异步回调是什么

**一句话：** 异步回调处理器是不阻塞主线程的回调函数，就像餐厅的号码牌系统。

**举例：**
同步回调：你在柜台等厨师做完菜（阻塞）
异步回调：你拿号码牌回座位，厨师做好了叫号（非阻塞）

**应用：** 在 RAG 系统中，异步回调让你可以同时处理多个用户请求，不会因为一个请求的日志记录而阻塞其他请求。

---

### 卡片2：形式化定义 - 精确表述

**一句话：** AsyncCallbackHandler 是继承自 BaseCallbackHandler 的异步回调接口，所有方法都是 `async def`，支持 `await` 调用。

**举例：**
```python
class AsyncCallbackHandler(BaseCallbackHandler):
    async def on_llm_start(self, serialized, prompts, **kwargs):
        await self._async_operation()  # 可以使用 await
```

**应用：** 在 FastAPI 等异步框架中，必须使用 AsyncCallbackHandler 才能充分利用框架的并发能力。

---

### 卡片3：关键概念1 - AsyncIteratorCallbackHandler

**一句话：** AsyncIteratorCallbackHandler 是内置的异步迭代器回调处理器，用于流式输出。

**举例：**
```python
callback = AsyncIteratorCallbackHandler()
async for token in callback.aiter():
    print(token, end="")  # 实时输出每个 token
```

**应用：** 在 RAG 系统中，用于实现类似 ChatGPT 的打字效果，用户可以立即看到生成的内容。

---

### 卡片4：关键概念2 - 异步队列

**一句话：** 异步回调使用 asyncio.Queue 传递数据，生产者和消费者互不阻塞。

**举例：**
```python
queue = asyncio.Queue()
await queue.put(token)  # 生产者
token = await queue.get()  # 消费者
```

**应用：** 在流式输出中，LLM 生成 token 放入队列（生产者），FastAPI 从队列取出 token 返回给客户端（消费者）。

---

### 卡片5：编程实现 - FastAPI 集成

**一句话：** FastAPI 流式回调集成需要三个组件：AsyncIteratorCallbackHandler、wrap_done、StreamingResponse。

**举例：**
```python

task = asyncio.create_task(wrap_done(model.agenerate(...), callback.done))
return StreamingResponse(callback.aiter(), media_type="text/event-stream")
```

**应用：** 这是构建生产级 RAG API 的标准模式，支持流式输出和高并发。

---

### 卡片6：对比区分 - 异步 vs 同步

**一句话：** 异步回调不阻塞主线程，同步回调会阻塞主线程。

**举例：**
```python
# 同步：阻塞 100ms
def on_llm_end(self, response):
    db.save(response)  # 阻塞

# 异步：不阻塞
async def on_llm_end(self, response):
    await db.save(response)  # 不阻塞
```

**应用：** 在高并发场景下，异步回调可以提升 10 倍以上的吞吐量。

---

### 卡片7：进阶理解 - 并发安全

**一句话：** 异步回调处理器需要使用 asyncio.Lock 保护共享状态，避免竞态条件。

**举例：**
```python
class SafeCallback(AsyncCallbackHandler):
    def __init__(self):
        self.lock = asyncio.Lock()
        self.count = 0

    async def on_llm_end(self, response, **kwargs):
        async with self.lock:
            self.count += 1
```

**应用：** 在多请求并发处理时，必须使用锁保护共享状态，否则会导致数据错乱。
