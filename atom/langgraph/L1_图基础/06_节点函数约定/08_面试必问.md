## 基础层次（5题）

### 问题1：节点函数的基本签名约定是什么？

**考察点**：
- 节点函数的参数类型
- 返回值约定
- 类型注解的使用

**标准答案**：

LangGraph 节点函数有三种基本签名：

1. **基础签名**：`def node(state: State) -> dict`
   - 接收状态对象，返回部分状态更新

2. **带 Config 参数**：`def node(state: State, config: RunnableConfig) -> dict`
   - 可以访问运行时配置信息

3. **带 Runtime 参数**：`def node(state: State, runtime: Runtime[Context]) -> dict`
   - 可以访问运行时上下文

[来源: reference/source_节点函数约定_01.md]

**代码示例**：

```python
from typing_extensions import TypedDict
from langgraph.types import RunnableConfig, Runtime

class State(TypedDict):
    messages: list[str]
    count: int

# 基础签名
def basic_node(state: State) -> dict:
    return {"count": state["count"] + 1}

# 带 Config 参数
def config_node(state: State, config: RunnableConfig) -> dict:
    user_id = config.get("configurable", {}).get("user_id")
    return {"messages": [f"User {user_id} processed"]}

# 带 Runtime 参数
def runtime_node(state: State, runtime: Runtime) -> dict:
    context_value = runtime.context.get("key", "default")
    return {"messages": [f"Context: {context_value}"]}
```

**追问**：
- Q: 为什么节点函数只需要返回部分状态更新？
- A: 这是 LangGraph 的设计理念，通过 Reducer 函数自动合并状态，简化节点实现。

- Q: 如果节点函数不需要更新状态怎么办？
- A: 可以返回 `None` 或空字典 `{}`，状态不会改变。

[来源: reference/context7_langgraph_01.md]

### 问题2：节点函数可以返回哪些类型的值？

**考察点**：
- 返回值类型的多样性
- Command 对象的使用
- 类型推断机制

**标准答案**：

节点函数支持以下返回值类型：

1. **dict**：部分状态更新
2. **Command**：控制流程跳转
3. **None**：不更新状态
4. **list[Command | dict]**：多个更新或命令
5. **带 Annotated 的自定义类型**：自定义状态更新

[来源: reference/source_节点函数约定_01.md]

**代码示例**：

```python
from langgraph.types import Command
from typing import Literal

# 返回 dict
def update_state(state: State) -> dict:
    return {"count": state["count"] + 1}

# 返回 Command
def route_node(state: State) -> Command[Literal["next", "end"]]:
    if state["count"] > 10:
        return Command(goto="end")
    return Command(goto="next", update={"count": state["count"] + 1})

# 返回 None
def no_update(state: State) -> None:
    print("Processing without state update")
    return None

# 返回 list
def multi_update(state: State) -> list:
    return [
        {"count": state["count"] + 1},
        Command(goto="next")
    ]
```

**追问**：
- Q: Command 对象和普通 dict 返回值有什么区别？
- A: Command 可以控制流程跳转（goto），而 dict 只能更新状态。Command 还可以同时更新状态和跳转。

- Q: 如果返回了无效的类型会怎样？
- A: 会抛出 `InvalidUpdateError` 异常，提示返回值类型不符合约定。

[来源: reference/source_节点函数约定_01.md]

### 问题3：什么是部分状态更新？为什么重要？

**考察点**：
- 状态更新机制
- Reducer 函数的作用
- 设计理念理解

**标准答案**：

**部分状态更新**是指节点函数只需要返回需要修改的状态字段，而不是完整的状态对象。

**重要性**：
1. **简化节点实现**：节点不需要关心整个状态结构
2. **提高可维护性**：状态字段变化不影响节点实现
3. **支持并行更新**：多个节点可以同时更新不同字段
4. **通过 Reducer 控制合并逻辑**：灵活定义状态更新规则

[来源: reference/context7_langgraph_01.md]

**代码示例**：

```python
from typing import Annotated
from operator import add

class State(TypedDict):
    messages: Annotated[list[str], add]  # 使用 add reducer
    count: int  # 默认覆盖
    metadata: dict

# 节点只返回需要更新的字段
def process_node(state: State) -> dict:
    # 只更新 messages 和 count，不需要返回 metadata
    return {
        "messages": ["New message"],  # 会追加到列表
        "count": state["count"] + 1   # 会覆盖原值
    }

# 完整示例
from langgraph.graph import StateGraph

builder = StateGraph(State)
builder.add_node("process", process_node)

# 执行
initial_state = {
    "messages": ["Hello"],
    "count": 0,
    "metadata": {"user": "alice"}
}

graph = builder.compile()
result = graph.invoke(initial_state)

print(result)
# {
#   "messages": ["Hello", "New message"],  # 追加
#   "count": 1,                             # 覆盖
#   "metadata": {"user": "alice"}          # 保持不变
# }
```

**追问**：
- Q: 如果两个节点同时更新同一个字段会怎样？
- A: 取决于 Reducer 函数。默认是覆盖，但可以自定义合并逻辑（如 add 追加列表）。

- Q: 部分更新和完整更新哪个性能更好？
- A: 部分更新性能更好，因为只需要序列化和传输修改的字段。

[来源: reference/source_节点函数约定_01.md]

### 问题4：同步节点和异步节点有什么区别？

**考察点**：
- 异步编程理解
- 节点执行机制
- 性能优化

**标准答案**：

**同步节点**：
- 使用普通函数定义：`def node(state: State) -> dict`
- 在线程池中执行（`run_in_executor`）
- 适合 CPU 密集型任务

**异步节点**：
- 使用 async 函数定义：`async def node(state: State) -> dict`
- 在事件循环中执行
- 适合 I/O 密集型任务（API 调用、数据库查询）

**LangGraph 自动检测**：
- 自动识别 `async` 函数
- 同步函数自动包装成异步执行

[来源: reference/source_节点函数约定_01.md, reference/search_节点函数约定_01.md]

**代码示例**：

```python
import asyncio
import time
from openai import AsyncOpenAI

# 同步节点（CPU 密集型）
def sync_node(state: State) -> dict:
    # 模拟 CPU 密集型计算
    result = sum(range(1000000))
    return {"count": result}

# 异步节点（I/O 密集型）
async def async_node(state: State) -> dict:
    # 异步 API 调用
    client = AsyncOpenAI()
    response = await client.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": state["messages"][-1]}]
    )
    return {"messages": [response.choices[0].message.content]}

# 混合使用
from langgraph.graph import StateGraph, START, END

builder = StateGraph(State)
builder.add_node("sync", sync_node)      # 同步节点
builder.add_node("async", async_node)    # 异步节点
builder.add_edge(START, "sync")
builder.add_edge("sync", "async")
builder.add_edge("async", END)

graph = builder.compile()
```

**追问**：
- Q: 可以在同步节点中调用异步函数吗？
- A: 不推荐。如果必须调用，需要使用 `asyncio.run()`，但这会创建新的事件循环，影响性能。

- Q: 异步节点的执行顺序是怎样的？
- A: 按照图的拓扑顺序执行，但多个独立的异步节点可以并行执行。

[来源: reference/source_节点函数约定_01.md]

### 问题5：如何在节点函数中访问配置信息？

**考察点**：
- RunnableConfig 的使用
- 配置传递机制
- 实际应用场景

**标准答案**：

通过 `RunnableConfig` 参数访问配置信息：

1. **添加 config 参数**：`def node(state: State, config: RunnableConfig) -> dict`
2. **访问配置字段**：
   - `config.get("configurable", {})` - 用户自定义配置
   - `config.get("metadata", {})` - 元数据
   - `config.get("tags", [])` - 标签
   - `config.get("run_name")` - 运行名称

**常见用途**：
- 多租户系统（user_id, tenant_id）
- 环境配置（dev, prod）
- 功能开关（feature flags）
- 调试信息（debug mode）

[来源: reference/source_节点函数约定_01.md, reference/context7_langgraph_01.md]

**代码示例**：

```python
from langgraph.types import RunnableConfig

def multi_tenant_node(state: State, config: RunnableConfig) -> dict:
    # 获取租户 ID
    tenant_id = config.get("configurable", {}).get("tenant_id")

    # 获取调试模式
    debug = config.get("configurable", {}).get("debug", False)

    # 根据配置执行不同逻辑
    if debug:
        print(f"Processing for tenant: {tenant_id}")

    # 使用租户 ID 查询数据
    data = fetch_tenant_data(tenant_id)

    return {"data": data, "tenant_id": tenant_id}

# 使用配置调用
graph = builder.compile()
result = graph.invoke(
    {"messages": ["Hello"]},
    config={
        "configurable": {
            "tenant_id": "tenant_123",
            "debug": True
        }
    }
)
```

**追问**：
- Q: 配置信息会传递给所有节点吗？
- A: 是的，config 会自动传递给图中的所有节点，无需手动传递。

- Q: 可以在节点中修改配置吗？
- A: 不推荐。配置应该是只读的，如果需要传递信息，应该通过状态更新。

[来源: reference/source_节点函数约定_01.md]

### 问题6：RetryPolicy 的工作原理是什么？如何配置？

**考察点**：
- 重试机制理解
- 指数退避算法
- 错误处理策略

**标准答案**：

**RetryPolicy 工作原理**：

1. **捕获异常**：节点执行失败时捕获异常
2. **匹配策略**：检查异常是否符合重试条件
3. **计算延迟**：使用指数退避算法计算等待时间
4. **重试执行**：清除之前的写入，重新执行节点
5. **达到上限**：超过最大重试次数后抛出异常

**配置参数**：
- `max_attempts`: 最大重试次数
- `initial_interval`: 初始重试间隔（秒）
- `backoff_factor`: 退避因子（指数增长）
- `max_interval`: 最大重试间隔（秒）
- `jitter`: 是否添加随机抖动
- `retry_on`: 重试条件（异常类型或自定义函数）

[来源: reference/source_节点函数约定_02.md, reference/context7_langgraph_01.md]

**代码示例**：

```python
from langgraph.types import RetryPolicy
from langgraph.graph import StateGraph
import random

# 自定义重试条件
def retry_on_rate_limit(error: Exception) -> bool:
    return "rate limit" in str(error).lower()

# 模拟不稳定的 API
def unreliable_api_call(state: State) -> dict:
    if random.random() < 0.5:  # 50% 失败率
        raise Exception("Rate limit exceeded")
    return {"result": "Success!"}

# 配置重试策略
builder = StateGraph(State)
builder.add_node(
    "api_call",
    unreliable_api_call,
    retry_policy=RetryPolicy(
        initial_interval=0.5,      # 初始等待 0.5 秒
        backoff_factor=2.0,        # 每次翻倍
        max_interval=10.0,         # 最多等待 10 秒
        max_attempts=5,            # 最多重试 5 次
        jitter=True,               # 添加随机抖动
        retry_on=retry_on_rate_limit  # 只重试 rate limit 错误
    )
)

# 指数退避计算示例
# 第1次重试: 0.5 * (2.0 ^ 0) = 0.5 秒
# 第2次重试: 0.5 * (2.0 ^ 1) = 1.0 秒
# 第3次重试: 0.5 * (2.0 ^ 2) = 2.0 秒
# 第4次重试: 0.5 * (2.0 ^ 3) = 4.0 秒
# 第5次重试: 0.5 * (2.0 ^ 4) = 8.0 秒
```

**追问**：
- Q: 为什么需要 jitter（抖动）？
- A: 防止"惊群效应"（thundering herd）。多个客户端同时重试会导致服务器压力激增，随机抖动可以分散重试时间。

- Q: 重试会清除之前的写入吗？
- A: 是的，每次重试前会调用 `task.writes.clear()`，确保不会重复写入。

[来源: reference/source_节点函数约定_02.md]

### 问题7：Command 对象的作用是什么？有哪些使用场景？

**考察点**：
- Command 对象理解
- 流程控制机制
- 高级路由场景

**标准答案**：

**Command 对象的作用**：

1. **控制流程跳转**：指定下一个执行的节点
2. **同时更新状态**：在跳转的同时更新状态
3. **父图通信**：子图向父图发送命令
4. **动态路由**：根据运行时条件决定路由

**使用场景**：
- 条件分支（if-else 逻辑）
- 循环控制（while 循环）
- 错误恢复（重试或跳过）
- 子图协作（父子图通信）

[来源: reference/source_节点函数约定_01.md, reference/context7_langgraph_01.md]

**代码示例**：

```python
from langgraph.types import Command
from typing import Literal

# 场景1：条件分支
def decision_node(state: State) -> Command[Literal["process", "skip", "end"]]:
    if state["count"] > 10:
        return Command(goto="end")
    elif state["error"]:
        return Command(goto="skip", update={"error": None})
    else:
        return Command(goto="process", update={"count": state["count"] + 1})

# 场景2：循环控制
def loop_node(state: State) -> Command[Literal["process", "end"]]:
    if state["iterations"] < state["max_iterations"]:
        return Command(
            goto="process",
            update={"iterations": state["iterations"] + 1}
        )
    return Command(goto="end")

# 场景3：父图通信
def subgraph_node(state: State) -> Command:
    if state["need_parent_help"]:
        # 向父图发送命令
        return Command(
            graph=Command.PARENT,
            update={"request": "help_needed"}
        )
    return Command(goto="next")

# 场景4：错误恢复
def error_handler(state: State) -> Command[Literal["retry", "fallback", "end"]]:
    if state["retry_count"] < 3:
        return Command(
            goto="retry",
            update={"retry_count": state["retry_count"] + 1}
        )
    elif state["has_fallback"]:
        return Command(goto="fallback")
    else:
        return Command(goto="end", update={"error": "Max retries exceeded"})
```

**追问**：
- Q: Command 和 add_conditional_edges 有什么区别？
- A: Command 是运行时动态决定路由，add_conditional_edges 是编译时定义路由规则。Command 更灵活，但 add_conditional_edges 更清晰。

- Q: 如果 Command 指向不存在的节点会怎样？
- A: 会抛出 `InvalidUpdateError`，错误信息会明确指出节点不存在。

[来源: reference/search_节点函数约定_01.md]

### 问题8：节点函数的类型推断机制是如何工作的？

**考察点**：
- 类型系统理解
- 源码实现细节
- 类型安全

**标准答案**：

**类型推断机制**：

1. **检查函数类型**：判断是否为函数或方法
2. **获取类型提示**：使用 `get_type_hints()` 获取参数类型
3. **推断 input_schema**：从第一个参数的类型推断输入 schema
4. **推断 output_schema**：从返回值类型推断输出 schema
5. **支持 TypedDict**：自动识别 TypedDict 类型
6. **支持 Command**：识别 Command 返回值类型

**推断规则**：
- 第一个参数类型 → input_schema
- 返回值类型 → output_schema
- 如果没有类型注解，使用默认 schema

[来源: reference/source_节点函数约定_01.md]

**代码示例**：

```python
from typing_extensions import TypedDict
from typing import Literal
from langgraph.types import Command

# 示例1：自动推断 input_schema
class InputState(TypedDict):
    messages: list[str]

class OutputState(TypedDict):
    result: str

def typed_node(state: InputState) -> OutputState:
    # LangGraph 自动推断：
    # - input_schema = InputState
    # - output_schema = OutputState
    return {"result": "processed"}

# 示例2：推断 Command 类型
def command_node(state: InputState) -> Command[Literal["next", "end"]]:
    # LangGraph 自动推断：
    # - 返回值是 Command
    # - 可能的目标节点：["next", "end"]
    return Command(goto="next")

# 示例3：显式指定 input_schema
from langgraph.graph import StateGraph

builder = StateGraph(InputState)

# 自动推断（推荐）
builder.add_node("auto", typed_node)

# 显式指定（覆盖推断）
builder.add_node(
    "manual",
    typed_node,
    input_schema=InputState  # 显式指定
)

# 源码实现（简化版）
def infer_input_schema(func):
    hints = get_type_hints(func)
    first_param = next(iter(inspect.signature(func).parameters.keys()))
    if first_param in hints:
        input_hint = hints[first_param]
        if isinstance(input_hint, type) and get_type_hints(input_hint):
            return input_hint
    return None
```

**追问**：
- Q: 如果不使用类型注解会怎样？
- A: LangGraph 会使用默认的 schema，但失去了类型检查的好处，容易出错。

- Q: 类型推断会影响性能吗？
- A: 类型推断只在图编译时执行一次，不影响运行时性能。

[来源: reference/source_节点函数约定_01.md]

### 问题9：如何实现自定义重试条件？

**考察点**：
- RetryPolicy 高级用法
- 自定义重试逻辑
- 错误分类处理

**标准答案**：

自定义重试条件有三种方式：

1. **单个异常类型**：`retry_on=ValueError`
2. **多个异常类型**：`retry_on=[ValueError, TypeError, ConnectionError]`
3. **自定义函数**：`retry_on=lambda e: custom_logic(e)`

**自定义函数的优势**：
- 可以检查异常消息内容
- 可以根据异常属性决定是否重试
- 可以实现复杂的重试逻辑

[来源: reference/source_节点函数约定_02.md]

**代码示例**：

```python
from langgraph.types import RetryPolicy

# 方式1：基于异常消息的重试条件
def retry_on_rate_limit(error: Exception) -> bool:
    """只重试 rate limit 相关的错误"""
    error_msg = str(error).lower()
    return any(keyword in error_msg for keyword in [
        "rate limit", "too many requests", "quota exceeded"
    ])

# 方式2：基于异常属性的重试条件
def retry_on_transient_error(error: Exception) -> bool:
    """只重试临时性错误"""
    if isinstance(error, (ConnectionError, TimeoutError)):
        return True
    if hasattr(error, 'status_code'):
        return 500 <= error.status_code < 600
    return False

# 使用自定义重试条件
builder.add_node(
    "api_call",
    unreliable_api_call,
    retry_policy=RetryPolicy(
        max_attempts=5,
        initial_interval=1.0,
        backoff_factor=2.0,
        retry_on=retry_on_rate_limit
    )
)
```

**追问**：
- Q: 自定义重试函数可以访问节点状态吗？
- A: 不能。重试函数只接收异常对象，无法访问节点状态。

- Q: 如何实现不同异常类型的不同重试策略？
- A: 可以配置多个 RetryPolicy，每个 policy 针对不同的异常类型。

[来源: reference/source_节点函数约定_02.md]

### 问题10：异步节点的错误处理有哪些注意事项？

**考察点**：
- 异步编程错误处理
- CancelledError 处理
- 流式错误处理

**标准答案**：

异步节点的错误处理注意事项：

1. **CancelledError 需要特殊处理**
   - 不应该被普通的 except 捕获
   - 需要重新抛出以确保正确取消

2. **流式错误处理**
   - streamEvents 中的错误可能被忽略
   - 需要在循环内处理错误

3. **异步生成器不推荐**
   - 可能导致函数体不执行
   - 错误处理机制失效

[来源: reference/search_节点函数约定_01.md]

**代码示例**：

```python
import asyncio

# 正确的异步错误处理
async def async_node_with_error_handling(state: State) -> dict:
    try:
        result = await asyncio.wait_for(some_async_api(), timeout=30.0)
        return {"result": result, "error": None}
    except asyncio.CancelledError:
        print("Task cancelled, cleaning up...")
        await cleanup_resources()
        raise  # 重新抛出以确保正确取消
    except asyncio.TimeoutError:
        return {"result": None, "error": "Timeout"}
    except Exception as e:
        return {"result": None, "error": str(e)}

# 流式错误处理
async def process_with_stream(graph, input_data):
    try:
        async for event in graph.astream_events(input_data):
            try:
                process_event(event)
            except Exception as e:
                if is_critical_error(e):
                    raise
    except Exception as e:
        print(f"Stream error: {e}")
```

**追问**：
- Q: 为什么 CancelledError 需要重新抛出？
- A: 因为 CancelledError 是 asyncio 的取消机制，如果不重新抛出，会导致任务无法正确取消，可能造成资源泄漏。

### 问题11：如何实现 Human-in-the-loop 模式？

**考察点**：
- interrupt 函数使用
- Checkpoint 机制
- 状态恢复

**标准答案**：

Human-in-the-loop 模式通过 `interrupt()` 函数实现：

1. **暂停执行**：调用 `interrupt()` 暂停图执行
2. **保存状态**：需要配置 Checkpointer 保存状态
3. **等待输入**：等待用户提供输入
4. **恢复执行**：使用 `Command(resume=...)` 恢复执行

[来源: reference/context7_langgraph_01.md]

**代码示例**：

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import interrupt, Command
from langgraph.checkpoint.memory import MemorySaver

def collect_info(state: FormState) -> dict:
    """使用 interrupt 收集用户输入"""
    name = interrupt("Please enter your name:")
    email = interrupt("Please enter your email:")
    return {"name": name, "email": email}

# 必须配置 checkpointer
checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 使用示例
config = {"configurable": {"thread_id": "user_123"}}
result = graph.invoke({"name": "", "email": ""}, config)
# 恢复执行
result = graph.invoke(Command(resume="Alice"), config)
```

**追问**：
- Q: 如果不配置 Checkpointer 会怎样？
- A: 会抛出异常，因为 interrupt 需要保存状态才能恢复执行。

### 问题12：在生产环境中如何处理节点函数的异常？

**考察点**：
- 生产环境最佳实践
- 错误监控
- 降级策略

**标准答案**：

生产环境的异常处理策略：

1. **分层错误处理**
   - 节点内部：捕获预期异常，返回错误状态
   - RetryPolicy：处理临时性错误
   - 图层面：捕获未处理的异常，记录日志

2. **错误监控**
   - 使用 LangSmith 追踪执行
   - 记录详细的错误日志
   - 设置告警阈值

3. **降级策略**
   - 提供默认值或缓存结果
   - 跳过非关键节点
   - 使用备用服务

[来源: reference/context7_langgraph_01.md, reference/search_节点函数约定_01.md]

**代码示例**：

```python
import logging

logger = logging.getLogger(__name__)

def robust_api_node(state: State) -> dict:
    """生产级节点实现"""
    try:
        result = call_primary_api(state["query"])
        return {"result": result, "error": None, "source": "primary"}
    except RateLimitError as e:
        logger.warning(f"Rate limit hit: {e}")
        cached_result = get_from_cache(state["query"])
        if cached_result:
            return {"result": cached_result, "error": None, "source": "cache"}
        raise
    except ServiceUnavailableError as e:
        logger.error(f"Primary service unavailable: {e}")
        try:
            result = call_fallback_api(state["query"])
            return {"result": result, "error": None, "source": "fallback"}
        except Exception as fallback_error:
            return {"result": None, "error": "All services unavailable"}
    except Exception as e:
        logger.exception(f"Unexpected error: {e}")
        return {"result": None, "error": str(e)}
```

**追问**：
- Q: 如何实现节点的幂等性？
- A: 使用唯一的请求 ID，在节点开始时检查是否已执行过，避免重复执行。
