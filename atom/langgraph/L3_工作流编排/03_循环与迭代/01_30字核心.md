# 循环与迭代 - 30字核心

**循环与迭代是让节点的输出重新触发已执行节点的机制，是 AI Agent 实现反复推理、自我修正的基础能力。**

---

## 核心要素拆解

### 1. 循环的本质

**定义**: 边（edge）指向图中已经存在的节点，形成有向环

**底层机制**:
- 节点执行后产生 writes（写入）
- writes 被 `apply_writes()` 应用到 channels
- 如果写入了 `branch:to:{node}` channel，目标节点在下一个超步被触发
- 如果目标节点已经执行过，就形成了循环

**关键洞察**: 无论使用哪种 API 构建循环（`add_edge`、`add_conditional_edges`、`Command`、`Send`），底层都归结为同一个机制——节点写入触发 `branch:to:{node}` channel，激活目标节点。

---

### 2. 为什么 Agent 需要循环

**单次执行的局限**:
- LLM 的输出不确定，一次可能不够好
- 工具调用需要根据返回结果继续推理
- 复杂任务需要分步完成，每步依赖上一步的结果

**循环解决的核心问题**:
- **ReAct 模式**: 思考 → 行动 → 观察 → 再思考（直到找到答案）
- **自我修正**: 生成 → 检查 → 修正（直到质量达标）
- **迭代检索**: 检索 → 评估 → 改写查询（直到找到相关文档）

没有循环的 Agent 就像一个只能回答一次的问答机器——无法追问、无法修正、无法深入思考。

---

### 3. 循环的安全保障

**递归限制**: 默认 25 步，防止无限循环

```python
# 被动保护：超出限制抛异常
graph.invoke(inputs, config={"recursion_limit": 50})

# 主动保护：感知剩余步数，优雅退出（推荐）
from langgraph.managed.is_last_step import RemainingSteps
```

**设计原则**: 每个循环都必须有明确的终止条件，递归限制只是最后的安全网。

---

## 一句话记忆

**循环 = 回边触发已执行节点，迭代 = 每轮累积状态直到满足终止条件。**

---

## 快速示例

### 最基础的循环模式

```python
import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# 1. 定义状态（使用 reducer 累积每轮结果）
class State(TypedDict):
    aggregate: Annotated[list, operator.add]

# 2. 定义节点
def node_a(state: State):
    print(f'Node A sees {state["aggregate"]}')
    return {"aggregate": ["A"]}

def node_b(state: State):
    print(f'Node B sees {state["aggregate"]}')
    return {"aggregate": ["B"]}

# 3. 定义终止条件
def route(state: State) -> Literal["b", "__end__"]:
    if len(state["aggregate"]) < 7:
        return "b"       # 继续循环
    return END           # 终止

# 4. 构建图（包含回边）
builder = StateGraph(State)
builder.add_node("a", node_a)
builder.add_node("b", node_b)

builder.add_edge(START, "a")           # 入口
builder.add_conditional_edges("a", route)  # 条件：继续或终止
builder.add_edge("b", "a")            # 回边：b → a 形成循环！

# 5. 编译并运行
graph = builder.compile()
result = graph.invoke({"aggregate": []})
print(result)
# {'aggregate': ['A', 'B', 'A', 'B', 'A', 'B', 'A']}
```

**执行过程**:
```
超步 1: a 执行 → aggregate = ['A']        → route 返回 "b"（继续）
超步 2: b 执行 → aggregate = ['A', 'B']   → 回到 a
超步 3: a 执行 → aggregate = ['A', 'B', 'A'] → route 返回 "b"（继续）
超步 4: b 执行 → aggregate = ['A', 'B', 'A', 'B']
超步 5: a 执行 → aggregate = ['A', 'B', 'A', 'B', 'A']
超步 6: b 执行 → aggregate = ['A', 'B', 'A', 'B', 'A', 'B']
超步 7: a 执行 → aggregate = ['A', 'B', 'A', 'B', 'A', 'B', 'A'] → route 返回 END（终止！）
```

## 循环结构图

### 基础循环

```
                    ┌──────────────┐
                    │    START     │
                    └──────┬───────┘
                           │
                           ▼
                    ┌──────────────┐
            ┌──────│   Node A     │◄─────────┐
            │      └──────┬───────┘          │
            │             │                  │
            │      ┌──────▼───────┐          │
            │      │  条件路由     │          │
            │      │ (route函数)  │          │
            │      └──┬───────┬──┘          │
            │         │       │              │
            │    继续  │       │ 终止         │
            │         ▼       ▼              │
            │  ┌──────────┐  ┌─────┐        │
            │  │  Node B   │  │ END │        │
            │  └──────┬────┘  └─────┘        │
            │         │                      │
            │         └──────────────────────┘
            │                回边 (b → a)
            │
            │  这就是循环！Node B 执行完后
            │  回到 Node A，形成有向环
            └─────────────────────────────────
```

### 执行时间线

```
时间 ──────────────────────────────────────────────────►

超步1    超步2    超步3    超步4    超步5    超步6    超步7
 [A] ──► [B] ──► [A] ──► [B] ──► [A] ──► [B] ──► [A] ──► END
  │       │       │       │       │       │       │
  ▼       ▼       ▼       ▼       ▼       ▼       ▼
 ['A']  ['A','B'] ...                           len=7, 终止!

每个超步都会保存 checkpoint（可回溯、可恢复）
```

## 核心对比

### 四种构建循环的方式

| 方式 | 代码示例 | 适用场景 | 特点 |
|------|---------|---------|------|
| 静态边 + 条件边 | `add_edge("b", "a")` + `add_conditional_edges("a", route)` | 标准循环 | 最常用，结构清晰 |
| 纯条件边 | `add_conditional_edges("a", route)` 返回 "b" 或 END | 简单循环 | 路由逻辑集中 |
| Command | `return Command(goto="b")` | 需要同时更新状态和路由 | 适合 Agent 交接 |
| Send | `return [Send("process", {...})]` | 动态并行迭代 | Map-Reduce 模式 |

### 循环 vs 单次执行

| 特性 | 单次执行 | 循环迭代 |
|------|---------|---------|
| 执行次数 | 每个节点最多执行一次 | 节点可以反复执行 |
| 结果质量 | 取决于单次 LLM 输出 | 可以逐步优化 |
| 状态管理 | 简单的线性流 | 需要 Reducer 累积状态 |
| 安全性 | 天然终止 | 需要终止条件 + 递归限制 |
| 适用场景 | 简单的数据处理管道 | Agent、自我修正、迭代检索 |

### 终止策略对比

| 策略 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| 条件终止 | `route` 返回 `END` | 精确控制，业务语义清晰 | 需要设计好终止条件 |
| RemainingSteps | 检查剩余步数 | 优雅降级，不抛异常 | 需要额外状态字段 |
| recursion_limit | 配置参数 | 兜底保护，零代码 | 粗暴终止，抛异常 |

## 关键类比

**前端类比**: 就像 React 的 `useEffect` + 状态更新触发重渲染的循环——组件渲染 → 副作用执行 → 状态更新 → 触发重渲染 → 直到状态稳定。LangGraph 的循环也是：节点执行 → 状态更新 → 触发下一个节点 → 直到条件满足。

**日常生活类比**: 就像改论文——写初稿 → 导师审阅 → 根据反馈修改 → 再次审阅 → 再修改... 直到导师说"可以了"。每一轮修改都基于上一轮的反馈（状态累积），导师的"可以了"就是终止条件。

---

## 核心公式

```
循环执行 = while tick():
               执行当前超步的所有任务
               apply_writes() → 更新 channels
               step += 1
               if step > stop: raise GraphRecursionError

终止条件 = route 函数返回 END
         OR remaining_steps <= 阈值
         OR step > recursion_limit + 1

状态累积 = Annotated[list, operator.add]  → 每轮追加
         普通字段                         → 每轮覆盖
```

## 核心机制

### 回边 (Back Edge)

```python
# 回边是循环的关键：让执行流回到之前的节点
builder.add_edge("b", "a")  # b 完成后，触发 a 重新执行
```

### 条件终止 (Conditional Termination)

```python
# 条件边决定是继续循环还是终止
def route(state: State) -> Literal["b", "__end__"]:
    if should_continue(state):
        return "b"    # 继续
    return END        # 终止
```

### 状态累积 (State Accumulation)

```python
# Reducer 确保每轮迭代的结果被保留
class State(TypedDict):
    messages: Annotated[list, operator.add]  # 追加，不覆盖
```

## 核心价值

1. **Agent 能力**: 让 AI 能够反复思考、调用工具、自我修正
2. **质量提升**: 通过迭代逐步优化输出质量
3. **安全可控**: 递归限制 + RemainingSteps 防止失控
4. **可观测**: 每步 checkpoint，可回溯、可调试
5. **灵活组合**: 四种构建方式适配不同场景
