# 核心概念 4：推理提示工程

> Prompt 是 Agent 推理的"剧本"——它告诉 LLM 扮演什么角色、有哪些工具可用、按什么格式输出推理过程

---

## 为什么 Prompt 设计如此重要？

LLM 本身不知道什么是"推理"，也不知道什么是"Agent"。**是 Prompt 告诉它：你是一个 Agent，你可以使用这些工具，你必须按这个格式思考和行动。**

```
没有推理 Prompt:  LLM 直接回答问题（可能幻觉）
有了推理 Prompt:  LLM 按 Thought→Action→Observation 格式逐步推理
```

**前端类比：** Prompt 就像 React 组件的 Props。同一个 LLM（组件），传入不同的 Prompt（Props），行为完全不同。ReAct Prompt 让它变成推理 Agent，Chat Prompt 让它变成聊天机器人。

**日常生活类比：** Prompt 就像给新员工的"工作手册"。手册写得好，员工知道遇到什么情况该怎么处理；手册写得差，员工就会乱来。

## MRKL Prompt 四段式结构

MRKL（Modular Reasoning, Knowledge and Language）定义了 Agent Prompt 的经典结构，几乎所有 LangChain Agent 都遵循这个模式：

```
┌─────────────────────────────────────────────┐
│              MRKL 四段式 Prompt               │
│                                              │
│  ┌────────────────────────────────────────┐ │
│  │ 1. PREFIX（角色定义 + 任务说明）        │ │
│  │    "你是一个有用的助手，可以使用工具"    │ │
│  └────────────────────────────────────────┘ │
│                    ↓                         │
│  ┌────────────────────────────────────────┐ │
│  │ 2. TOOLS（可用工具列表）                │ │
│  │    "你可以使用以下工具：                 │ │
│  │     search: 搜索互联网                  │ │
│  │     calculator: 数学计算"               │ │
│  └────────────────────────────────────────┘ │
│                    ↓                         │
│  ┌────────────────────────────────────────┐ │
│  │ 3. FORMAT_INSTRUCTIONS（输出格式要求）  │ │
│  │    "请按以下格式输出：                   │ │
│  │     Thought: 你的思考                   │ │
│  │     Action: 工具名                      │ │
│  │     Action Input: 工具输入"             │ │
│  └────────────────────────────────────────┘ │
│                    ↓                         │
│  ┌────────────────────────────────────────┐ │
│  │ 4. SUFFIX（用户输入 + Scratchpad）      │ │
│  │    "问题：{input}                       │ │
│  │     {agent_scratchpad}"                 │ │
│  └────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
```

### 源码中的实际实现

```python
# langchain/agents/mrkl/prompt.py（简化版）

PREFIX = """Answer the following questions as best you can. You have access to the following tools:"""

FORMAT_INSTRUCTIONS = """Use the following format:

Question: the input question you must answer
Thought: you should always think about what to do
Action: the action to take, should be one of [{tool_names}]
Action Input: the input to the action
Observation: the result of the action
... (this Thought/Action/Action Input/Observation can repeat N times)
Thought: I now know the final answer
Final Answer: the final answer to the original input question"""

SUFFIX = """Begin!

Question: {input}
Thought:{agent_scratchpad}"""
```

### 四段式的组装过程

```python
from langchain.agents import ZeroShotAgent

# 工具描述自动生成
tools_description = "\n".join([
    f"{tool.name}: {tool.description}"
    for tool in tools
])
# 结果：
# "search: 搜索互联网获取最新信息"
# "calculator: 进行数学计算"

# 工具名列表
tool_names = ", ".join([tool.name for tool in tools])
# 结果："search, calculator"

# 最终 Prompt = PREFIX + tools_description + FORMAT_INSTRUCTIONS + SUFFIX
# FORMAT_INSTRUCTIONS 中的 {tool_names} 会被替换为实际工具名
```

## ReAct Prompt 格式详解

### 标准 ReAct 格式

ReAct Prompt 定义了 LLM 必须遵循的推理格式：

```
Question: {用户问题}
Thought: {LLM 的思考过程}
Action: {要调用的工具名}
Action Input: {传给工具的参数}
Observation: {工具返回的结果}
... (重复 Thought/Action/Action Input/Observation)
Thought: I now know the final answer
Final Answer: {最终答案}
```

**每个字段的作用：**

| 字段 | 谁生成 | 作用 |
|------|--------|------|
| Question | 系统（从用户输入填充） | 定义要解决的问题 |
| Thought | LLM | 展示推理过程，决定下一步 |
| Action | LLM | 选择要调用的工具 |
| Action Input | LLM | 指定工具的输入参数 |
| Observation | 系统（工具返回值） | 提供真实信息反馈 |
| Final Answer | LLM | 给出最终答案，结束循环 |

### JSON 格式 ReAct Prompt

有些场景下，用 JSON 格式替代纯文本格式更可靠：

```
Question: {用户问题}
Thought: {LLM 的思考过程}
Action:
```json
{
    "action": "工具名",
    "action_input": "工具输入"
}
```
Observation: {工具返回的结果}
... (重复)
Thought: I now know the final answer
Final Answer: {最终答案}
```

**JSON 格式的优势：**
- 参数结构更清晰，支持复杂参数（嵌套对象、数组）
- 解析更可靠（JSON.parse vs 正则匹配）
- 适合需要多参数的工具调用

### XML 格式 Prompt

特别适合 Claude 模型：

```
Question: {用户问题}

{LLM 的思考过程}

<tool>工具名</tool>
<tool_input>工具输入</tool_input>

<observation>{工具返回的结果}</observation>

... (重复)

<final_answer>{最终答案}</final_answer>
```

**为什么 Claude 适合 XML？**
- Claude 在训练数据中大量接触 XML 格式
- XML 标签有明确的开始/结束标记，生成更稳定
- 不需要严格的换行和缩进约束

## Stop Sequence：防止 LLM 幻觉的关键

### 问题：LLM 会"编造" Observation

没有 Stop Sequence 时，LLM 可能在输出 `Action Input` 后继续生成，自己编造一个 Observation：

```
Thought: 我需要查天气
Action: weather_tool
Action Input: 北京
Observation: 北京今天25°C，晴天  ← LLM 自己编的！不是真实数据！
Thought: 好的，天气不错
Final Answer: 北京今天25°C，晴天  ← 基于幻觉的错误答案
```

### 解决方案：绑定 Stop Sequence

```python
# create_react_agent 源码中的关键一行：
llm_with_stop = llm.bind(stop=["\nObservation"])

# 效果：LLM 生成到 "\nObservation" 时自动停止
# 输出变成：
# "Thought: 我需要查天气\nAction: weather_tool\nAction Input: 北京"
# ← 到这里就停了！不会继续生成 Observation
```

### 工作流程

```
┌──────────┐    ┌─────────────────┐    ┌──────────────┐
│ LLM 生成 │───→│ 遇到 stop token │───→│ 停止生成     │
│ Thought  │    │ "\nObservation" │    │ 返回已生成   │
│ Action   │    │                 │    │ 的文本       │
│ Action   │    │                 │    │              │
│ Input    │    │                 │    │              │
└──────────┘    └─────────────────┘    └──────┬───────┘
                                              │
                                              ▼
                                    ┌──────────────────┐
                                    │ OutputParser 解析 │
                                    │ 提取 Action 信息  │
                                    └──────────────────┘
                                              │
                                              ▼
                                    ┌──────────────────┐
                                    │ 执行真实工具      │
                                    │ 获取真实 Observe  │
                                    └──────────────────┘
```

### 不同推理策略的 Stop Sequence

| 推理策略 | Stop Sequence | 原因 |
|----------|--------------|------|
| ReAct 文本 | `["\nObservation"]` | 防止 LLM 编造 Observation |
| ReAct JSON | `["\nObservation"]` | 同上 |
| XML | `["</tool_input>"]` 或无 | XML 标签自带边界 |
| Tool Calling | 无需 | 模型原生支持，不会编造 |

## Few-Shot 示例：教 LLM 怎么推理

### 为什么需要 Few-Shot？

Zero-Shot（不给示例）时，LLM 可能不严格遵循格式。Few-Shot（给几个示例）能显著提升格式遵循率和推理质量。

### wiki_prompt 中的 Few-Shot 示例分析

LangChain 的 `hwchase17/react` prompt hub 包含 6 个完整推理示例。以下是其中一个的结构分析：

```python
# Few-Shot 示例（简化版）
example = """
Question: What is the elevation range for the area that the eastern sector
of the Colorado orogeny extends into?

Thought: I need to search Colorado orogeny, find the area that the eastern
sector extends into, then find the elevation range of that area.

Action: Search
Action Input: Colorado orogeny

Observation: The Colorado orogeny was an episode of mountain building...
The eastern sector extends into the High Plains.

Thought: The eastern sector extends into the High Plains. I need to search
High Plains and find its elevation range.

Action: Search
Action Input: High Plains (United States)

Observation: The High Plains are a subregion of the Great Plains...
elevation from around 1,800 to 7,000 ft.

Thought: High Plains rise in elevation from around 1,800 to 7,000 ft,
so the answer is 1,800 to 7,000 ft.

Final Answer: 1,800 to 7,000 ft
"""
```

### Few-Shot 示例的设计原则

**1. 展示完整循环**

每个示例都要包含完整的 Thought→Action→Observation→...→Final Answer 循环，让 LLM 看到"从头到尾"的推理过程。

**2. 展示多步推理**

至少有一个示例包含 2-3 轮循环，让 LLM 知道"可以多次调用工具"。

**3. 展示推理的"转折"**

好的示例会展示"第一次搜索没找到想要的信息，需要换个关键词再搜"的场景，教 LLM 如何应对不完美的工具返回。

**4. 展示终止判断**

最后一个 Thought 要明确说"我现在知道答案了"，教 LLM 何时该停止循环。

## 工具描述的最佳实践

工具描述是 Prompt 的重要组成部分，直接影响 LLM 的工具选择准确率。

### 好的工具描述 vs 差的工具描述

```python
from langchain_core.tools import tool

# 差的描述：太模糊，LLM 不知道什么时候该用
@tool
def search(query: str) -> str:
    """搜索工具"""  # ← 太简单了！
    return do_search(query)

# 好的描述：明确说明用途、输入要求、返回内容
@tool
def search(query: str) -> str:
    """搜索互联网获取最新信息。当你需要查询实时数据、
    最新新闻或你不确定的事实时使用此工具。
    输入应该是一个搜索查询字符串。
    返回搜索结果的摘要文本。"""
    return do_search(query)
```

### 工具描述的四要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **做什么** | 工具的核心功能 | "搜索互联网获取最新信息" |
| **何时用** | 什么场景下应该选择这个工具 | "当需要查询实时数据时" |
| **输入要求** | 参数的格式和内容要求 | "输入应该是搜索查询字符串" |
| **返回内容** | 工具会返回什么 | "返回搜索结果的摘要文本" |

### 工具描述在 Prompt 中的渲染

```python
# LangChain 自动将工具列表渲染为文本
# render_text_description 函数的输出格式：

"""
search: 搜索互联网获取最新信息。当你需要查询实时数据时使用此工具。
calculator: 进行数学计算。当你需要精确计算数字时使用此工具。
retriever: 从知识库中检索相关文档。当你需要查询内部文档时使用此工具。
"""

# 格式：tool_name: tool_description
# 这段文本会被插入到 MRKL Prompt 的 TOOLS 段落中
```

## 工具调用模式的 Prompt 差异

使用原生函数调用（Tool Calling）时，Prompt 结构有本质不同：

### 文本推理 vs 工具调用的 Prompt 对比

```python
# 文本推理（ReAct）的 Prompt：
react_prompt = """You have access to the following tools:
search: 搜索互联网
calculator: 数学计算

Use the following format:
Thought: ...
Action: the action to take, should be one of [search, calculator]
Action Input: ...
Observation: ...

Question: {input}
Thought:{agent_scratchpad}"""

# 工具调用（Tool Calling）的 Prompt：
tool_calling_prompt = """You are a helpful assistant.

{input}"""
# 工具信息不在 Prompt 文本中！
# 而是通过 llm.bind_tools(tools) 传递给模型的 API
```

**关键区别：**

| 维度 | 文本推理（ReAct） | 工具调用（Tool Calling） |
|------|-------------------|------------------------|
| 工具信息位置 | 在 Prompt 文本中 | 在 API 参数中（bind_tools） |
| 格式约束 | Prompt 中定义格式 | 模型原生支持，无需格式约束 |
| Scratchpad | 文本格式的推理历史 | ToolMessage 消息列表 |
| Stop Sequence | 需要（防幻觉） | 不需要 |

### format_log_to_str vs format_to_tool_messages

两种推理策略使用不同的 Scratchpad 格式化方式：

```python
# 文本推理：format_log_to_str
# 将 intermediate_steps 格式化为字符串
def format_log_to_str(intermediate_steps):
    """将推理步骤转换为文本格式的 scratchpad"""
    log = ""
    for action, observation in intermediate_steps:
        log += action.log  # "Thought: ...\nAction: ...\nAction Input: ..."
        log += f"\nObservation: {observation}\nThought: "
    return log

# 结果示例：
# "Thought: 我需要查天气\nAction: weather\nAction Input: 北京
#  \nObservation: 5°C\nThought: "

# 工具调用：format_to_tool_messages
# 将 intermediate_steps 转换为消息列表
def format_to_tool_messages(intermediate_steps):
    """将推理步骤转换为 ToolMessage 列表"""
    messages = []
    for action, observation in intermediate_steps:
        messages.append(AIMessage(
            content="",
            tool_calls=[{"name": action.tool, "args": action.tool_input, "id": action.tool_call_id}]
        ))
        messages.append(ToolMessage(
            content=str(observation),
            tool_call_id=action.tool_call_id
        ))
    return messages
```

## Prompt 设计的常见陷阱

### 1. 工具描述过于模糊

```python
# 错误：LLM 无法区分两个工具
tools = [
    Tool(name="search1", description="搜索"),      # 搜索什么？
    Tool(name="search2", description="另一个搜索"),  # 和 search1 有什么区别？
]

# 正确：明确区分使用场景
tools = [
    Tool(name="web_search", description="搜索互联网获取实时信息，如新闻、天气"),
    Tool(name="doc_search", description="搜索内部知识库文档，如公司政策、产品手册"),
]
```

### 2. 忘记 Stop Sequence

```python
# 错误：没有 stop sequence，LLM 可能编造 Observation
llm_chain = prompt | llm | parser

# 正确：绑定 stop sequence
llm_with_stop = llm.bind(stop=["\nObservation"])
llm_chain = prompt | llm_with_stop | parser
```

### 3. Scratchpad 格式不匹配

```python
# 错误：用文本推理的 Prompt 但用消息格式的 Scratchpad
# 或者反过来

# 正确：
# 文本推理 → format_log_to_str → 字符串 scratchpad
# 工具调用 → format_to_tool_messages → 消息列表 scratchpad
```

### 4. Few-Shot 示例与实际工具不匹配

```python
# 错误：示例中用了 "Search" 工具，但实际工具叫 "web_search"
# LLM 可能会输出 "Action: Search"，导致找不到工具

# 正确：确保示例中的工具名与实际工具名一致
# 或者使用动态 few-shot，根据实际工具生成示例
```

## 实战：构建自定义推理 Prompt

```python
from langchain_core.prompts import PromptTemplate

# 自定义 ReAct Prompt（中文版）
custom_react_prompt = PromptTemplate.from_template("""你是一个智能助手，可以使用以下工具来回答问题：

{tools}

请严格按照以下格式输出：

问题：你需要回答的问题
思考：分析问题，决定下一步行动
工具：要使用的工具名，必须是 [{tool_names}] 之一
输入：传给工具的参数
观察：工具返回的结果
...（思考/工具/输入/观察 可以重复多次）
思考：我现在知道最终答案了
最终答案：对原始问题的完整回答

开始！

问题：{input}
思考：{agent_scratchpad}""")

# 使用自定义 Prompt 创建 Agent
from langchain.agents import create_react_agent

agent = create_react_agent(
    llm=llm,
    tools=tools,
    prompt=custom_react_prompt
)
```

## 小结

| 要点 | 内容 |
|------|------|
| MRKL 四段式 | PREFIX + TOOLS + FORMAT_INSTRUCTIONS + SUFFIX |
| ReAct 格式 | Thought→Action→Action Input→Observation 循环 |
| Stop Sequence | `["\nObservation"]` 防止 LLM 编造工具返回值 |
| Few-Shot | 提供完整推理示例，提升格式遵循率和推理质量 |
| 工具描述 | 四要素：做什么、何时用、输入要求、返回内容 |
| 文本 vs 工具调用 | 文本推理靠 Prompt 约束格式，工具调用靠 API 原生支持 |
