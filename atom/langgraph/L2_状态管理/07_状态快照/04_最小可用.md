## 4.1 配置 Checkpointer（前提条件）

**一句话：** 没有 Checkpointer，就没有状态快照——这是一切的前提

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from typing_extensions import TypedDict
from typing import Annotated
import operator

# 1. 定义状态
class State(TypedDict):
    messages: Annotated[list, operator.add]
    current_step: str

# 2. 定义节点
def step_a(state):
    return {"messages": ["步骤A完成"], "current_step": "a"}

def step_b(state):
    return {"messages": ["步骤B完成"], "current_step": "b"}

# 3. 构建图
builder = StateGraph(State)
builder.add_node("step_a", step_a)
builder.add_node("step_b", step_b)
builder.add_edge(START, "step_a")
builder.add_edge("step_a", "step_b")
builder.add_edge("step_b", END)

# 4. 关键：编译时传入 checkpointer
memory = InMemorySaver()
graph = builder.compile(checkpointer=memory)

# 5. 关键：调用时传入 thread_id
config = {"configurable": {"thread_id": "user-001"}}
result = graph.invoke({"messages": ["开始"]}, config)
```

**关键点：**
- `InMemorySaver` 是最简单的 checkpointer，数据存在内存中，进程结束就丢失
- `thread_id` 是必须的，它标识一个独立的对话/执行线程
- 没有 checkpointer 时调用 `get_state()` 会抛出 `ValueError`
- 生产环境应使用持久化 checkpointer（如 `SqliteSaver`、`PostgresSaver`）

**常见错误：**

```python
# ❌ 忘记配置 checkpointer
graph = builder.compile()  # 没有 checkpointer
graph.get_state(config)    # ValueError!

# ❌ 忘记传 thread_id
config = {}  # 没有 thread_id
graph.invoke({"messages": ["开始"]}, config)  # 报错！
```

## 4.2 获取当前状态快照

**一句话：** `get_state(config)` 返回一个 StateSnapshot，包含当前状态的完整信息

```python
# 获取当前状态快照
snapshot = graph.get_state(config)

# StateSnapshot 是 NamedTuple，包含 8 个字段
print(snapshot.values)       # 当前状态值 {'messages': [...], 'current_step': 'b'}
print(snapshot.next)         # 下一步要执行的节点，() 表示已结束
print(snapshot.config)       # 当前 checkpoint 的配置（含 checkpoint_id）
print(snapshot.metadata)     # 元数据：step、source、writes、parents
print(snapshot.created_at)   # 创建时间（ISO 格式字符串）
print(snapshot.parent_config)  # 父 checkpoint 的配置（用于回溯）
print(snapshot.tasks)        # 待执行的任务列表
print(snapshot.interrupts)   # 中断信息（human-in-the-loop 场景）
```

**最常用的 3 个字段：**

```python
snapshot = graph.get_state(config)

# 1. values —— 查看当前状态
print(f"消息数: {len(snapshot.values['messages'])}")
print(f"当前步骤: {snapshot.values['current_step']}")

# 2. next —— 判断图是否执行完毕
if snapshot.next:
    print(f"下一步: {snapshot.next}")  # ('step_b',)
else:
    print("图已执行完毕")  # next 为空元组 ()

# 3. config —— 获取 checkpoint_id（用于时间旅行）
checkpoint_id = snapshot.config["configurable"]["checkpoint_id"]
print(f"Checkpoint ID: {checkpoint_id}")
```

**关键点：**
- `snapshot.next` 为空元组 `()` 表示图已执行完毕
- `snapshot.next` 非空表示图在某个节点暂停（如 interrupt）
- `snapshot.config` 中的 `checkpoint_id` 是时间旅行的关键

## 4.3 遍历状态历史

**一句话：** `get_state_history(config)` 按逆时间顺序返回所有历史快照

```python
# 遍历所有历史快照（最新的在前）
print("=== 状态历史 ===")
for state in graph.get_state_history(config):
    step = state.metadata.get("step", "?")
    source = state.metadata.get("source", "?")
    checkpoint_id = state.config["configurable"]["checkpoint_id"]
    print(f"Step {step} | Source: {source} | ID: {checkpoint_id}")
    print(f"  values: {state.values}")
    print(f"  next: {state.next}")
    print()

# 输出示例（逆时间顺序）：
# Step 2 | Source: loop | ID: xxx-003
#   values: {'messages': ['开始', '步骤A完成', '步骤B完成'], 'current_step': 'b'}
#   next: ()
#
# Step 1 | Source: loop | ID: xxx-002
#   values: {'messages': ['开始', '步骤A完成'], 'current_step': 'a'}
#   next: ('step_b',)
#
# Step 0 | Source: input | ID: xxx-001
#   values: {'messages': ['开始']}
#   next: ('step_a',)
```

**使用 filter 和 limit 精确查询：**

```python
# 只获取最近 3 条历史
for state in graph.get_state_history(config, limit=3):
    print(state.metadata["step"])

# 按元数据过滤：只看 source 为 "loop" 的（节点执行产生的）
for state in graph.get_state_history(config, filter={"source": "loop"}):
    print(state.metadata["step"])
```

**关键点：**
- 返回顺序是**逆时间顺序**（最新的在前），不是正序
- `metadata["source"]` 的值：`"input"` 表示初始输入，`"loop"` 表示节点执行
- `metadata["step"]` 是执行步骤编号（从 0 开始）
- `limit` 参数限制返回数量，避免历史过长时的性能问题

[来源: reference/context7_langgraph_01.md | get_state_history 文档]

## 4.4 修改状态并恢复执行

**一句话：** `update_state(config, values)` 创建一个新的 checkpoint，可以从修改后的状态继续执行

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from typing_extensions import TypedDict
from typing import Annotated
import operator

class State(TypedDict):
    messages: Annotated[list, operator.add]
    answer: str

def search(state):
    return {"messages": ["搜索结果: RAG 是检索增强生成"]}

def generate(state):
    context = state["messages"][-1]
    return {"answer": f"基于上下文的回答: {context}"}

builder = StateGraph(State)
builder.add_node("search", search)
builder.add_node("generate", generate)
builder.add_edge(START, "search")
builder.add_edge("search", "generate")
builder.add_edge("generate", END)

memory = InMemorySaver()
graph = builder.compile(checkpointer=memory)
config = {"configurable": {"thread_id": "demo-1"}}

# 正常执行
result = graph.invoke({"messages": ["什么是 RAG？"], "answer": ""}, config)

# 修改状态：注入新的搜索结果
# 注意：messages 有 Reducer (operator.add)，所以新值会追加
new_config = graph.update_state(
    config,
    {"messages": ["[人工修正] RAG = Retrieval-Augmented Generation"]}
)

# 查看修改后的状态
snapshot = graph.get_state(new_config)
print(snapshot.values["messages"])
# ['什么是 RAG？', '搜索结果: RAG 是检索增强生成',
#  '[人工修正] RAG = Retrieval-Augmented Generation']
```

**指定 as_node 模拟特定节点的输出：**

```python
# 模拟 search 节点的输出，这样 generate 节点会被标记为下一步
new_config = graph.update_state(
    config,
    {"messages": ["[替换的搜索结果]"]},
    as_node="search"  # 假装这是 search 节点的输出
)

# 从修改后的状态继续执行（会从 generate 节点开始）
result = graph.invoke(None, new_config)
```

**关键点：**
- `update_state` 不会修改原有 checkpoint，而是创建一个全新的 checkpoint
- 有 Reducer 的字段（如 `Annotated[list, operator.add]`）会按 Reducer 规则合并
- 没有 Reducer 的字段会直接覆盖
- `as_node` 参数决定了"下一步该执行哪个节点"
- 返回的 `new_config` 包含新 checkpoint 的 ID

## 4.5 时间旅行

**一句话：** 通过指定 `checkpoint_id` 回到历史中的任意时刻，可以重放或分叉

```python
# 第一步：找到目标 checkpoint
for state in graph.get_state_history(config):
    print(f"Step {state.metadata['step']} | ID: {state.config['configurable']['checkpoint_id']}")
    print(f"  next: {state.next}")

# 假设我们想回到 Step 1（search 执行完、generate 还没执行的时刻）
# 从历史中找到对应的 checkpoint_id
target_config = None
for state in graph.get_state_history(config):
    if state.metadata.get("step") == 1:
        target_config = state.config
        break

# 方式1：Replay（重放）—— 从历史 checkpoint 原样继续执行
if target_config:
    result = graph.invoke(None, target_config)
    print("重放结果:", result)

# 方式2：Fork（分叉）—— 先修改状态，再从修改后的状态继续
if target_config:
    # 修改历史状态
    forked_config = graph.update_state(
        target_config,
        {"messages": ["[分叉] 使用不同的搜索结果"]}
    )
    # 从分叉点继续执行
    result = graph.invoke(None, forked_config)
    print("分叉结果:", result)
```

**Replay vs Fork 的区别：**

```
原始执行：  input → search → generate → END
                      ↑
                  checkpoint_id = "xxx-002"

Replay（重放）：从 xxx-002 原样继续
  xxx-002 → generate → END（结果和原来一样）

Fork（分叉）：先修改 xxx-002 的状态，再继续
  xxx-002 → [update_state 修改] → xxx-004（新 checkpoint）
  xxx-004 → generate → END（结果可能不同）
```

**关键点：**
- Replay 传入历史 `checkpoint_id`，用 `graph.invoke(None, config)` 继续
- Fork 先用 `update_state` 修改历史状态，再用返回的新 config 继续
- `graph.invoke(None, config)` 中的 `None` 表示"不提供新输入，从已有状态继续"
- 原始的 checkpoint 链不受影响，Fork 创建的是新的分支

## 这些知识足以

掌握以上 5 个要点，你就能：

1. **查看图的执行状态** — 用 `get_state()` 获取当前快照的 values、next、metadata
2. **调试 Agent 行为** — 用 `get_state_history()` 回溯每一步的状态变化
3. **实现断点续传** — 用 `update_state()` 修改状态后从中断处继续
