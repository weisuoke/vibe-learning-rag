# 02_第一性原理 - 条件分支策略

> 回到最基础的真理，从源头理解 LangGraph 的条件分支机制

---

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而不是通过类比或经验来理解。

在物理学中，第一性原理意味着从基本的物理定律出发推导现象。在工程中，意味着从最基础的需求和约束出发设计系统。

对于 LangGraph 的条件分支策略，我们要问：**什么是条件分支的本质？为什么需要它？如何从零构建它？**

---

## 条件分支策略的第一性原理

### 1. 最基础的定义

**条件分支 = 根据当前状态，在运行时选择不同的执行路径**

仅此而已！没有更基础的了。

这个定义包含三个关键要素：
1. **当前状态**：决策的依据是运行时的数据，不是编译时的配置
2. **选择**：从多个可能的路径中挑选一个或多个
3. **执行路径**：选择的结果是"接下来执行哪个节点"

用最简单的代码表达：

```python
# 条件分支的本质就是一个函数
def route(state) -> str:
    if state["score"] > 0.8:
        return "accept"    # 路径 A
    else:
        return "retry"     # 路径 B
```

**与普通边的区别**：

```python
# 普通边：无条件，A 执行完一定去 B
builder.add_edge("A", "B")

# 条件边：有条件，A 执行完根据状态决定去哪
builder.add_conditional_edges("A", route)
```

### 2. 为什么需要条件分支？

**核心问题：工作流不是线性的，需要根据中间结果做出决策。**

让我们从最基本的场景思考：

#### 场景 1：线性工作流的局限

```python
# 线性工作流：每一步都是固定的
START → 检索文档 → 生成回答 → END

# 问题：如果检索到的文档质量很差怎么办？
# 问题：如果用户的问题需要不同的处理策略怎么办？
# 问题：如果 LLM 生成的回答不合格怎么办？
```

**线性工作流的三大局限**：
- **无法适应变化**：不管输入是什么，都走同一条路
- **无法处理异常**：某一步失败了，没有备选方案
- **无法优化效率**：简单问题和复杂问题走同样的流程

#### 场景 2：现实世界的决策是动态的

想象你去医院看病：

```
挂号 → 医生问诊 → ???

医生不会对每个病人都做同样的事：
- 感冒 → 开药 → 回家
- 骨折 → 拍片 → 手术 → 住院
- 体检 → 抽血 + 心电图 + B超（并行！）→ 汇总报告
```

**AI Agent 也是一样**：

```python
# Agent 需要根据 LLM 的输出决定下一步
用户提问 → LLM 思考 → ???

LLM 的输出可能是：
- 直接回答 → 返回给用户（END）
- 需要搜索 → 调用搜索工具 → 再次思考
- 需要计算 → 调用计算工具 → 再次思考
- 需要多个工具 → 并行调用多个工具 → 再次思考
```

这就是为什么需要条件分支——**让工作流具备"思考"和"决策"的能力**。

#### 场景 3：不同输入需要不同处理流程

```python
# RAG 系统中的条件分支
用户查询 → 意图分类 → ???

意图分类结果：
- "事实查询" → 向量检索 → 生成回答
- "闲聊" → 直接 LLM 回答（跳过检索）
- "代码问题" → 代码库检索 → 代码生成
- "多步推理" → 检索 → 推理 → 验证 → 可能重试
```

### 3. 条件分支的三层价值

#### 价值 1：动态决策能力 — 运行时路由

**从"写死的流程"到"会思考的流程"**

```python
# ❌ 没有条件分支：写死的流程
START → retrieve → generate → END
# 不管检索质量如何，都直接生成

# ✅ 有条件分支：会思考的流程
START → retrieve → evaluate → (条件判断)
                                ↓ 质量好      ↓ 质量差
                              generate      rewrite_query → retrieve（重试）
```

**在 LangGraph 中的体现**：

```python
def evaluate_route(state) -> Literal["generate", "rewrite_query"]:
    if state["relevance_score"] > 0.7:
        return "generate"       # 质量好，直接生成
    return "rewrite_query"      # 质量差，改写查询重试

builder.add_conditional_edges("evaluate", evaluate_route)
```

**实际价值**：
- ReAct Agent：根据 LLM 输出决定是否调用工具
- Self-RAG：根据检索质量决定是否重试
- 多 Agent 系统：根据任务类型路由到不同的专家 Agent

#### 价值 2：复杂流程编排 — 多路分支、并行、回退

**从"二选一"到"多路并行"**

条件分支不只是 if-else，它支持三种模式：

```python
# 模式 1：单路选择（最常见）
def route(state) -> Literal["a", "b", "c"]:
    if condition_a:
        return "a"
    elif condition_b:
        return "b"
    return "c"

# 模式 2：多路并行（Send）
def route(state):
    # 同时去多个节点！
    return [Send("analyst_1", state), Send("analyst_2", state)]

# 模式 3：路由 + 状态更新（Command）
def node(state) -> Command[Literal["next"]]:
    return Command(
        update={"processed": True},  # 更新状态
        goto="next"                   # 同时路由
    )
```

**在 LangGraph 中的体现**：
- `add_conditional_edges()`：基础条件路由
- `Send`：动态并行分支
- `Command`：路由与状态更新一体化

#### 价值 3：系统韧性 — 错误处理、降级策略

**从"一条路走到黑"到"有备选方案"**

```python
def resilient_route(state) -> Literal["primary", "fallback", "__end__"]:
    # 主路径
    if state.get("primary_available"):
        return "primary"
    # 降级路径
    if state.get("fallback_available"):
        return "fallback"
    # 兜底：直接结束
    return END

# 重试机制
def retry_route(state) -> Literal["retry", "__end__"]:
    if state["attempt"] < 3 and not state["success"]:
        return "retry"
    return END
```

**实际价值**：
- API 调用失败时自动切换备用服务
- LLM 生成质量不达标时自动重试
- 超过最大重试次数时优雅降级

### 4. 从第一性原理推导

让我们从最基础的需求出发，逐步推导出 LangGraph 的条件分支机制：

**推理链：**

```
1. 【基础需求】工作流需要决策能力
   ↓
   问题：如何在图中表达"根据条件选择路径"？

2. 【条件判断】需要一个函数来做决策
   ↓
   解决方案：路由函数 path(state) → 目标节点名

3. 【状态依赖】决策需要基于当前状态
   ↓
   解决方案：路由函数接收完整的 state 作为输入
   → 这就是 add_conditional_edges(source, path)

4. 【静态路由不够】有时需要动态创建并行任务
   ↓
   问题：路由函数只能返回一个节点名？
   解决方案：允许返回多个目标 → Send 对象列表
   → Send("node", custom_state) 实现动态并行

5. 【路由与更新耦合】有时路由决策和状态更新是一体的
   ↓
   问题：路由函数只负责路由，状态更新在节点里？
   解决方案：Command(goto=..., update=...) 同时做两件事
   → 节点返回 Command 实现路由+更新一体化

6. 【路由可能失败】路由函数返回无效目标怎么办？
   ↓
   解决方案：
   - 编译时验证：path_map 限定合法目标
   - 运行时验证：不能返回 None 或 START
   - 兜底机制：recursion_limit 防止无限循环

7. 【可视化需求】需要在图可视化中看到所有可能的分支
   ↓
   解决方案：
   - Literal 类型注解自动推断分支
   - path_map 显式声明分支
   → 编译时就知道所有可能的路径
```

**关键洞察**：

1. **add_conditional_edges 是基础**
   - 最核心的 API，覆盖 80% 的条件分支场景
   - 路由函数 + 可选的 path_map = 完整的条件路由

2. **Send 是并行扩展**
   - 当路由函数需要返回多个目标时使用
   - 每个 Send 可以携带不同的状态
   - 典型场景：Map-Reduce、多工具并行调用

3. **Command 是高级控制**
   - 当路由和状态更新需要原子化时使用
   - 支持跨图导航（子图 → 父图）
   - 典型场景：Agent handoff、中断恢复

4. **三者的关系**
   ```
   add_conditional_edges  ← 图级别的路由声明
        ↓ 路由函数可以返回
   字符串节点名           ← 最简单，单路选择
   Send 对象列表          ← 动态并行

   Command               ← 节点级别的路由声明（不需要条件边）
        ↓ 包含
   goto + update          ← 路由 + 状态更新一体化
   ```

### 5. 双重类比

#### 前端开发类比：React Router

条件分支就像前端的路由系统：

```javascript
// 前端路由：根据 URL 决定渲染哪个组件
<Route path="/home" component={Home} />
<Route path="/about" component={About} />

// LangGraph 条件分支：根据状态决定执行哪个节点
builder.add_conditional_edges("classifier", route_fn)
// route_fn 返回 "home_handler" 或 "about_handler"
```

| 前端路由概念 | LangGraph 条件分支 |
|-------------|-------------------|
| URL 路径 | 路由函数返回值 |
| Route 组件 | add_conditional_edges |
| 路由守卫 (beforeEnter) | 路由函数中的条件判断 |
| 动态路由 (/user/:id) | Send 动态路由 |
| 嵌套路由 | Command 跨图导航 |
| 404 页面 | END 节点（兜底路径） |
| 路由懒加载 | 条件边的延迟执行 |

**关键相似点**：
- 都是根据"当前状态"决定"下一步去哪"
- 都支持静态声明和动态决策
- 都需要兜底路径（404 / END）

#### 日常生活类比：导航 App

条件分支就像开车时的导航 App：

```
你在开车，导航 App 实时计算路线：

1. 简单路由（add_conditional_edges）
   "前方路口，左转去公司，右转去超市"
   → 根据你的目的地（状态）选择方向

2. 动态并行（Send）
   "你需要同时去三个地方取快递"
   → 导航规划三条并行路线，分别派三个人去

3. 路由 + 更新（Command）
   "前方施工，已为你重新规划路线并更新预计到达时间"
   → 同时改变路线（goto）和更新状态（update）

4. 回退机制
   "前方道路封闭，正在寻找替代路线..."
   → 主路径不通时自动切换备选路径
```

| 导航概念 | LangGraph 条件分支 |
|---------|-------------------|
| 目的地 | 目标节点 |
| 实时路况 | 当前 state |
| 路线选择 | 路由函数 |
| 多目的地导航 | Send 并行路由 |
| 重新规划路线 | Command(goto=...) |
| 替代路线 | 回退/降级路径 |
| 到达目的地 | END 节点 |

### 6. 条件分支的底层统一模型

无论使用哪种条件分支方式，底层都归结为同一个机制：

```
┌─────────────────────────────────────────────────────────────┐
│                    底层统一模型                                │
│                                                             │
│  add_conditional_edges("A", route_fn)                       │
│    → route_fn 返回 "B"                                      │
│    → ChannelWrite("branch:to:B", None)                      │
│                                                             │
│  add_conditional_edges("A", route_fn)                       │
│    → route_fn 返回 [Send("B", state1), Send("C", state2)]  │
│    → 创建两个并行任务，写入 TASKS channel                     │
│                                                             │
│  Command(goto="B", update={...})                            │
│    → _control_branch 处理                                    │
│    → ChannelWrite("branch:to:B", None) + 状态更新            │
│                                                             │
│  ─────────────────────────────────────                      │
│  最终都是：写入 branch:to:{node} channel                     │
│  → 目标节点在下一个 superstep 被触发                          │
└─────────────────────────────────────────────────────────────┘
```

**关键机制**：每个节点有一个 `branch:to:{node_name}` 的 `EphemeralValue` channel 作为触发器。路由的本质就是向目标节点的触发 channel 写入信号。

### 7. 三种机制的选择指南

```
你需要条件分支吗？
    │
    ├── 是：路由逻辑独立于节点逻辑？
    │       │
    │       ├── 是 → add_conditional_edges()
    │       │       │
    │       │       ├── 需要并行？ → 返回 Send 列表
    │       │       └── 不需要？  → 返回字符串节点名
    │       │
    │       └── 否（路由和状态更新耦合）→ Command
    │               │
    │               ├── 需要跨图？ → Command(graph=PARENT)
    │               └── 当前图内？ → Command(goto="node")
    │
    └── 否 → add_edge()（普通边）
```

## 核心洞察

### 洞察 1：条件分支是 Agent 智能的基础

没有条件分支的工作流只是一个固定的流水线。有了条件分支，工作流才能：
- **感知**：通过状态了解当前情况
- **决策**：通过路由函数选择最优路径
- **适应**：根据中间结果调整执行策略

这正是 AI Agent 区别于传统程序的核心能力。

### 洞察 2：三种机制不是替代关系，而是互补关系

```
add_conditional_edges  → 图的"交通规则"（在哪个路口可以转弯）
Send                   → 图的"分身术"（一个人变多个人并行走）
Command                → 图的"瞬移术"（节点自己决定跳到哪里）
```

它们解决不同层面的问题：
- `add_conditional_edges`：**声明式**路由，在图构建时定义
- `Send`：**动态并行**，在运行时创建多个执行分支
- `Command`：**命令式**路由，在节点执行时决定下一步

### 洞察 3：路由函数是条件分支的灵魂

路由函数的质量直接决定了工作流的智能程度：

```python
# 低质量路由：硬编码条件
def route(state):
    if state["type"] == "A":
        return "handler_a"
    return "handler_b"

# 高质量路由：LLM 驱动的智能路由
def route(state):
    result = llm.invoke(f"分析这个查询的意图: {state['query']}")
    return result.intent  # LLM 决定路由目标
```

## 从第一性原理到实践的映射

### 基础概念映射

| 第一性原理概念 | LangGraph 实现 | 代码体现 |
|----------------|----------------|----------|
| 条件判断 | 路由函数 | `def route(state) -> str` |
| 路径选择 | add_conditional_edges | `builder.add_conditional_edges(source, path)` |
| 动态并行 | Send 对象 | `return [Send("node", state)]` |
| 路由+更新 | Command | `Command(goto="node", update={...})` |
| 兜底路径 | END 节点 | `return END` |
| 循环守卫 | recursion_limit | `graph.invoke(input, {"recursion_limit": 25})` |

### 实践模式映射

| 应用场景 | 第一性原理 | LangGraph 实现 |
|----------|------------|----------------|
| ReAct Agent | 根据 LLM 输出决策 | 条件边判断 tool_calls |
| Self-RAG | 根据检索质量决策 | 条件边判断 relevance_score |
| 多 Agent 路由 | 根据任务类型分发 | 条件边 + Supervisor 模式 |
| Map-Reduce | 动态创建并行任务 | Send 对象列表 |
| Agent Handoff | 路由 + 状态传递 | Command(goto=..., update=...) |
| 错误重试 | 失败时回退重试 | 条件边 + 计数器 |

---

## 一句话总结第一性原理

**条件分支策略是通过路由函数在运行时动态选择执行路径的机制，LangGraph 提供 add_conditional_edges（声明式路由）、Send（动态并行）、Command（命令式路由+状态更新）三种互补工具，使工作流从固定流水线进化为具备感知、决策、适应能力的智能系统。**
