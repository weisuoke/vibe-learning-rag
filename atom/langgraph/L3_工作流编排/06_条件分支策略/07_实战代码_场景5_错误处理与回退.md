# 实战代码 - 场景5：错误处理与回退

> 本文档演示 LangGraph 条件分支策略中的三级回退模式，构建生产级容错数据处理管道

---

## 场景描述

**场景**：构建一个数据处理管道，包含完整的三级回退机制。

**核心挑战**：在 Agent 工作流中，LLM 调用、外部 API、工具执行都可能失败。单纯的重试不够，需要一套分层降级策略。

**三级回退模式**：

```
第一级：重试（Retry）
  → 相同操作，最多 N 次，带指数退避
  → 适用于：网络抖动、API 限流等临时性错误

第二级：降级（Fallback）
  → 使用备用方案（更简单的模型、缓存结果、简化处理）
  → 适用于：主服务持续不可用

第三级：人工兜底（Manual）
  → 记录错误上下文，返回安全默认值，标记需人工处理
  → 适用于：所有自动化手段都失败
```

**与核心概念6的关系**：核心概念6讲解了回退机制的原理，本场景将其组合成完整的三级回退管道。

## 完整代码

```python
"""
条件分支策略 - 实战场景5：错误处理与回退
演示：三级回退的数据处理管道

三级回退模式：
  第一级 → 重试（相同操作，最多3次，带指数退避）
  第二级 → 降级（使用备用方案）
  第三级 → 人工兜底（记录错误，返回默认结果）
"""

import random
import time
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END


# ============================================================
# 1. 状态定义
# ============================================================

class PipelineState(TypedDict):
    """数据处理管道状态"""
    input_data: str              # 输入数据
    result: str                  # 处理结果
    status: str                  # pending / success / error / degraded / manual
    error_count: int             # 当前错误次数
    max_retries: int             # 最大重试次数（第一级上限）
    error_message: str           # 最近一次错误信息
    processing_log: list[str]    # 完整处理日志
    fallback_attempted: bool     # 是否已尝试降级


# ============================================================
# 2. 第一级：主处理节点（带模拟失败）
# ============================================================

def process_data(state: PipelineState) -> dict:
    """
    主处理节点 —— 模拟一个可能失败的数据处理操作。

    实际场景中，这里可能是：
    - 调用 LLM 生成摘要
    - 调用外部 API 获取数据
    - 执行复杂的数据转换
    """
    attempt = state.get("error_count", 0) + 1
    log = list(state.get("processing_log", []))

    print(f"  [处理] 第 {attempt} 次尝试...")

    # 模拟 60% 失败率（实际项目中替换为真实逻辑）
    if random.random() < 0.6:
        error_msg = random.choice([
            "LLM 响应超时",
            "API 返回 429 限流",
            "数据格式解析失败",
        ])
        print(f"  [处理] 失败: {error_msg}")
        return {
            "status": "error",
            "error_count": attempt,
            "error_message": error_msg,
            "processing_log": log + [f"尝试#{attempt}: 失败 - {error_msg}"],
        }

    # 成功
    result = f"处理完成: {state['input_data'].upper()}"
    print(f"  [处理] 成功: {result}")
    return {
        "status": "success",
        "result": result,
        "error_count": attempt,
        "processing_log": log + [f"尝试#{attempt}: 成功"],
    }


# ============================================================
# 3. 第二级：降级处理节点
# ============================================================

def fallback_process(state: PipelineState) -> dict:
    """
    降级处理 —— 使用更简单但更可靠的备用方案。

    实际场景中，这里可能是：
    - 使用更小的模型（GPT-4 → GPT-3.5）
    - 返回缓存结果
    - 使用规则引擎代替 LLM
    """
    log = list(state.get("processing_log", []))
    print(f"  [降级] 启动备用方案...")

    # 模拟降级处理（20% 失败率，比主处理可靠得多）
    if random.random() < 0.2:
        print(f"  [降级] 备用方案也失败了")
        return {
            "status": "error",
            "error_message": "降级方案也失败",
            "fallback_attempted": True,
            "processing_log": log + ["降级处理: 失败"],
        }

    # 降级成功（结果质量可能不如主处理）
    result = f"[降级结果] {state['input_data'][:20]}..."
    print(f"  [降级] 成功: {result}")
    return {
        "status": "degraded",
        "result": result,
        "fallback_attempted": True,
        "processing_log": log + ["降级处理: 成功"],
    }


# ============================================================
# 4. 第三级：人工兜底节点
# ============================================================

def manual_handler(state: PipelineState) -> dict:
    """
    人工兜底 —— 所有自动化手段都失败后的最终保障。

    实际场景中，这里可能是：
    - 发送告警到 Slack/钉钉
    - 创建工单到 Jira
    - 将任务放入人工审核队列
    - 返回安全的默认值
    """
    log = list(state.get("processing_log", []))
    print(f"  [人工] 转入人工处理队列")
    print(f"  [人工] 错误历史: {state.get('error_message', '未知')}")

    return {
        "status": "manual",
        "result": f"[需人工处理] 输入: {state['input_data']}, 错误: {state.get('error_message', '未知')}",
        "processing_log": log + [
            f"转人工处理 (共尝试{state.get('error_count', 0)}次)",
            f"最后错误: {state.get('error_message', '未知')}",
        ],
    }


# ============================================================
# 5. 路由函数：三级回退决策
# ============================================================

def route_after_process(state: PipelineState) -> Literal["done", "retry", "fallback", "manual"]:
    """
    主处理后的路由决策。

    决策逻辑：
      成功 → done
      失败 + 未超重试上限 → retry（第一级）
      失败 + 超过重试上限 → fallback（第二级）
    """
    status = state.get("status", "error")
    error_count = state.get("error_count", 0)
    max_retries = state.get("max_retries", 3)

    if status == "success":
        print(f"  [路由] → done (处理成功)")
        return "done"

    if error_count < max_retries:
        print(f"  [路由] → retry (第{error_count}次失败，最多{max_retries}次)")
        return "retry"

    print(f"  [路由] → fallback (已失败{error_count}次，启动降级)")
    return "fallback"


def route_after_fallback(state: PipelineState) -> Literal["done", "manual"]:
    """
    降级处理后的路由决策。

    决策逻辑：
      降级成功 → done
      降级也失败 → manual（第三级）
    """
    if state.get("status") == "degraded":
        print(f"  [路由] → done (降级成功)")
        return "done"

    print(f"  [路由] → manual (降级失败，转人工)")
    return "manual"


# ============================================================
# 6. 终止节点
# ============================================================

def done_node(state: PipelineState) -> dict:
    """终止节点 —— 输出最终结果"""
    log = list(state.get("processing_log", []))
    status = state.get("status", "unknown")
    print(f"  [完成] 最终状态: {status}")
    return {
        "processing_log": log + [f"流程结束: {status}"],
    }


# ============================================================
# 7. 构建图
# ============================================================

def build_pipeline() -> StateGraph:
    """构建三级回退管道"""
    builder = StateGraph(PipelineState)

    # 添加节点
    builder.add_node("process", process_data)
    builder.add_node("fallback", fallback_process)
    builder.add_node("manual", manual_handler)
    builder.add_node("done", done_node)

    # 入口 → 主处理
    builder.add_edge(START, "process")

    # 主处理后：条件路由（三级回退的核心）
    builder.add_conditional_edges(
        "process",
        route_after_process,
        {
            "done": "done",         # 成功 → 结束
            "retry": "process",     # 第一级：重试（回到自身）
            "fallback": "fallback", # 第二级：降级
            "manual": "manual",     # 直接转人工（预留）
        },
    )

    # 降级后：条件路由
    builder.add_conditional_edges(
        "fallback",
        route_after_fallback,
        {
            "done": "done",     # 降级成功 → 结束
            "manual": "manual", # 第三级：人工兜底
        },
    )

    # 人工兜底 → 结束
    builder.add_edge("manual", "done")

    # 终止节点 → END
    builder.add_edge("done", END)

    return builder.compile()


# ============================================================
# 8. 测试运行
# ============================================================

def main():
    """测试三级回退管道"""
    print("=" * 60)
    print("条件分支策略 - 场景5：三级回退数据处理管道")
    print("=" * 60)

    graph = build_pipeline()

    # 初始状态
    initial_state: PipelineState = {
        "input_data": "hello world, this is a test message",
        "result": "",
        "status": "pending",
        "error_count": 0,
        "max_retries": 3,
        "error_message": "",
        "processing_log": [],
        "fallback_attempted": False,
    }

    # 运行多次观察不同路径
    for i in range(3):
        print(f"\n{'─' * 60}")
        print(f"第 {i + 1} 次运行 (seed={42 + i})")
        print(f"{'─' * 60}")

        random.seed(42 + i)
        result = graph.invoke(initial_state)

        print(f"\n  最终状态: {result['status']}")
        print(f"  结果: {result['result']}")
        print(f"  尝试次数: {result['error_count']}")
        print(f"  处理日志:")
        for entry in result["processing_log"]:
            print(f"    - {entry}")


if __name__ == "__main__":
    main()
```

## 代码详解

### 1. 三级回退的图结构

```
                    ┌─────────────┐
                    │    START    │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
              ┌────→│   process   │←────┐
              │     └──────┬──────┘     │
              │            │            │
              │     ┌──────▼──────┐     │
              │     │  路由决策    │     │
              │     └──┬───┬───┬──┘     │
              │        │   │   │        │
              │   成功  │  失败 │ 超限    │
              │        │   │   │        │
              │        │   │   │        │
              │        ▼   │   ▼        │
              │     done   │  fallback  │
              │        │   │   │        │
              │        │   └───┘        │
              │        │  retry         │
              │        │   (回到process) │
              │        │               │
              │        │        ┌──────▼──────┐

              │        │           │       │

              │     ┌──────────────────────▼─┐
              │     │           END           │

              │

```

### 2. 路由函数是三级回退的核心

```python
# 第一级路由：主处理后
def route_after_process(state) -> Literal["done", "retry", "fallback", "manual"]:
    if status == "success":   return "done"      # 成功 → 结束
    if error_count < max:     return "retry"     # 第一级：重试
    else:                     return "fallback"  # 第二级：降级

# 第二级路由：降级后
def route_after_fallback(state) -> Literal["done", "manual"]:
    if status == "degraded":  return "done"      # 降级成功 → 结束
    else:                     return "manual"    # 第三级：人工
```

**关键设计**：每一级都有独立的路由函数，职责清晰。不要把所有逻辑塞进一个路由函数。

### 3. 状态字段设计

```python
class PipelineState(TypedDict):
    # 业务数据
    input_data: str           # 输入
    result: str               # 输出

    # 控制字段（驱动条件分支）
    status: str               # 当前状态 → 路由依据
    error_count: int          # 错误计数 → 循环守卫
    max_retries: int          # 重试上限 → 可配置

    # 可观测性
    error_message: str        # 错误详情
    processing_log: list[str] # 完整日志
    fallback_attempted: bool  # 降级标记
```

**设计原则**：状态中同时包含「业务数据」和「控制字段」。控制字段驱动条件分支的路由决策。

## 扩展：添加指数退避

在生产环境中，重试之间应该加入指数退避，避免对下游服务造成更大压力。

```python
"""
扩展：带指数退避的重试节点
"""

import time


def process_data_with_backoff(state: PipelineState) -> dict:
    """带指数退避的主处理节点"""
    attempt = state.get("error_count", 0)
    log = list(state.get("processing_log", []))

    # ✅ 指数退避：第1次等1秒，第2次等2秒，第3次等4秒
    if attempt > 0:
        wait_time = min(2 ** (attempt - 1), 8)  # 上限 8 秒
        print(f"  [退避] 等待 {wait_time} 秒后重试...")
        time.sleep(wait_time)

    print(f"  [处理] 第 {attempt + 1} 次尝试...")

    # 模拟处理逻辑（同上）
    if random.random() < 0.6:
        error_msg = "处理失败"
        return {
            "status": "error",
            "error_count": attempt + 1,
            "error_message": error_msg,
            "processing_log": log + [
                f"尝试#{attempt + 1}: 失败 (退避{2**attempt if attempt > 0 else 0}s)"
            ],
        }

    return {
        "status": "success",
        "result": f"处理完成: {state['input_data'].upper()}",
        "error_count": attempt + 1,
        "processing_log": log + [f"尝试#{attempt + 1}: 成功"],
    }


# 使用方式：替换原来的 process_data 节点
# builder.add_node("process", process_data_with_backoff)
```

### 指数退避的退避时间表

```
重试次数 │ 等待时间 │ 累计等待
─────────┼──────────┼─────────
  第1次  │   0 秒   │   0 秒
  第2次  │   1 秒   │   1 秒
  第3次  │   2 秒   │   3 秒
  第4次  │   4 秒   │   7 秒
  第5次  │   8 秒   │  15 秒（上限）
```

## 扩展：带抖动的指数退避

纯指数退避在多个客户端同时重试时会产生「惊群效应」。加入随机抖动可以分散重试时间。

```python
def calculate_backoff(attempt: int, base: float = 1.0, max_wait: float = 8.0) -> float:
    """
    计算带抖动的指数退避时间。

    公式：min(base * 2^attempt + random_jitter, max_wait)
    """
    if attempt <= 0:
        return 0.0

    # 指数退避
    wait = base * (2 ** (attempt - 1))

    # 加入 ±25% 的随机抖动
    jitter = wait * 0.25 * (2 * random.random() - 1)
    wait += jitter

    # 上限
    return min(wait, max_wait)


# 使用示例
for i in range(6):
    print(f"  第{i}次重试: 等待 {calculate_backoff(i):.2f} 秒")
```

def generate_answer(state):
    try:
        response = openai.chat.completions.create(
            model="gpt-4",
            messages=[{"role": "user", "content": state["prompt"]}],
            timeout=30,
        )
        return {"answer": response.choices[0].message.content, "status": "success"}

def fallback_generate(state):

def manual_fallback(state):

## 最佳实践

### 1. 每一级都要有明确的退出条件

```python
# ✅ 好：每一级都有退出条件
route_after_process:   error_count < max_retries → retry, else → fallback
route_after_fallback:  status == "degraded" → done, else → manual
manual_handler:        无条件 → done（最终兜底，必须结束）

# ❌ 坏：降级也能重试，可能形成嵌套循环
route_after_fallback:  retry_count < 3 → retry_fallback  # 危险！
```

### 2. 日志要贯穿全流程

```python
# 每个节点都追加日志，而不是覆盖
"processing_log": log + [f"尝试#{attempt}: 失败 - {error_msg}"]
```

### 3. 状态字段要区分「业务数据」和「控制数据」

```python
# 业务数据：input_data, result
# 控制数据：status, error_count, max_retries, fallback_attempted
# 可观测性：error_message, processing_log
```

### 4. 人工兜底必须是终止节点

```python
# 人工兜底后不应该再有条件分支
builder.add_edge("manual", "done")  # 直接到终止
builder.add_edge("done", END)
```

---

## 总结

**三级回退模式的核心**：

1. 每一级对应一个条件路由函数，职责单一
2. 循环守卫（error_count < max_retries）防止无限重试
3. 降级方案要比主方案更可靠（更简单 = 更稳定）
4. 人工兜底是最终保障，必须无条件终止

**适用场景**：LLM 调用、外部 API、数据处理管道、RAG 生成环节——任何存在不确定性的 Agent 工作流节点。
