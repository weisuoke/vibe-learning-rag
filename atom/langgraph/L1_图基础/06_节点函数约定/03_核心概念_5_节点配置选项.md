# 核心概念 5：节点配置选项

## 概述

在 LangGraph 中，节点函数不仅可以定义执行逻辑，还可以通过配置选项来控制其行为。这些配置选项提供了强大的功能，包括重试策略、缓存策略、元数据管理、延迟执行和目标节点定义。

[来源: reference/source_节点函数约定_01.md]

### 配置选项列表

LangGraph 的 `add_node` 方法支持以下配置选项：

```python
def add_node(
    self,
    node: str | StateNode[NodeInputT, ContextT],
    action: StateNode[NodeInputT, ContextT] | None = None,
    *,
    defer: bool = False,
    metadata: dict[str, Any] | None = None,
    input_schema: type[NodeInputT] | None = None,
    retry_policy: RetryPolicy | Sequence[RetryPolicy] | None = None,
    cache_policy: CachePolicy | None = None,
    destinations: dict[str, str] | tuple[str, ...] | None = None,
) -> Self:
```

[来源: reference/source_节点函数约定_01.md]

## 1. retry_policy - 重试策略

### 基本概念

`retry_policy` 允许你为节点配置自动重试机制，当节点执行失败时，LangGraph 会根据策略自动重试。这对于处理不稳定的外部 API 调用非常有用。

[来源: reference/source_节点函数约定_02.md]

### RetryPolicy 参数

```python
class RetryPolicy:
    max_attempts: int              # 最大重试次数
    initial_interval: float        # 初始重试间隔（秒）
    backoff_factor: float          # 退避因子（指数退避）
    max_interval: float            # 最大重试间隔（秒）
    jitter: bool                   # 是否添加随机抖动
    retry_on: type[Exception] | Sequence[type[Exception]] | Callable[[Exception], bool]
```

[来源: reference/source_节点函数约定_02.md]

### 重试算法

LangGraph 使用指数退避算法计算重试间隔：

```python
# 计算重试间隔
interval = initial_interval * (backoff_factor ** (attempts - 1))
interval = min(max_interval, interval)

# 添加抖动（如果启用）
if jitter:
    sleep_time = interval + random.uniform(0, 1)
else:
    sleep_time = interval
```

[来源: reference/source_节点函数约定_02.md]

### 基础示例

```python
from langgraph.graph import StateGraph
from langgraph.types import RetryPolicy
from typing_extensions import TypedDict

class State(TypedDict):
    result: str
    attempts: int

def unreliable_api_call(state: State) -> dict:
    """模拟不稳定的 API 调用"""
    import random
    attempts = state.get("attempts", 0) + 1

    if random.random() < 0.5:  # 50% 失败率
        raise Exception("API rate limit exceeded")

    return {"result": "Success!", "attempts": attempts}

# 配置重试策略
builder = StateGraph(State)
builder.add_node(
    "api_call",
    unreliable_api_call,
    retry_policy=RetryPolicy(
        initial_interval=0.5,      # 初始等待 0.5 秒
        backoff_factor=2.0,        # 每次翻倍
        max_interval=10.0,         # 最多等待 10 秒
        max_attempts=5,            # 最多重试 5 次
        jitter=True,               # 添加随机抖动
        retry_on=Exception         # 所有异常都重试
    )
)
```

[来源: reference/context7_langgraph_01.md]

### 自定义重试条件

你可以使用自定义函数来决定是否重试：

```python
def retry_on_rate_limit(error: Exception) -> bool:
    """只在遇到速率限制时重试"""
    return "rate limit" in str(error).lower()

builder.add_node(
    "api_call",
    unreliable_api_call,
    retry_policy=RetryPolicy(
        initial_interval=0.5,
        backoff_factor=2.0,
        max_interval=10.0,
        max_attempts=5,
        jitter=True,
        retry_on=retry_on_rate_limit  # 自定义重试条件
    )
)
```

[来源: reference/context7_langgraph_01.md]

### 多异常类型重试

```python
# 只重试特定异常类型
retry_policy = RetryPolicy(
    max_attempts=3,
    initial_interval=1.0,
    backoff_factor=2.0,
    max_interval=10.0,
    jitter=False,
    retry_on=[ValueError, TypeError, ConnectionError]
```

## 2. cache_policy - 缓存策略

### 基本概念

`cache_policy` 允许你为节点配置缓存策略，避免重复执行相同的计算。这对于昂贵的计算或 API 调用非常有用。

[来源: reference/source_节点函数约定_01.md]

### 使用场景

- **昂贵的 API 调用**：避免重复调用相同的 API
- **计算密集型操作**：缓存计算结果
- **幂等操作**：相同输入总是产生相同输出的操作

### 基础示例

```python
from langgraph.types import CachePolicy

def expensive_computation(state: State) -> dict:
    """昂贵的计算操作"""
    import time
    time.sleep(2)  # 模拟耗时操作
    result = state["input"] * 2
    return {"result": result}

builder.add_node(
    "compute",
    expensive_computation,
    cache_policy=CachePolicy(
        ttl=300,  # 缓存 5 分钟
        key_fn=lambda state: state["input"]  # 缓存键函数
    )
)
```

### 注意事项

- 缓存基于输入状态的哈希值
- 需要确保节点函数是幂等的
- 缓存可能占用内存，需要合理设置 TTL

## 3. metadata - 元数据

### 基本概念

`metadata` 允许你为节点附加任意元数据，用于调试、监控或文档化。

[来源: reference/source_节点函数约定_01.md]

### 使用场景

- **文档化**：描述节点的用途
- **监控**：添加监控标签
- **调试**：添加调试信息
- **版本控制**：记录节点版本

### 基础示例

```python
builder.add_node(
    "process_data",
    process_function,
    metadata={
        "description": "处理用户输入数据",
        "version": "1.0.0",
        "author": "team@example.com",
        "tags": ["data-processing", "critical"],
        "sla_ms": 1000  # 期望响应时间
    }
)
```

### 访问元数据

元数据可以在运行时通过图的配置访问：

```python
def my_node(state: State, config: RunnableConfig) -> dict:
    # 访问节点元数据
    metadata = config.get("metadata", {})
    print(f"Node description: {metadata.get('description')}")
    return {"result": "done"}
```

## 4. defer - 延迟执行

### 基本概念

`defer` 选项控制节点是否延迟执行。当设置为 `True` 时，节点不会立即执行，而是等待特定条件触发。

[来源: reference/source_节点函数约定_01.md]

### 使用场景

- **条件执行**：只在满足特定条件时执行
- **懒加载**：延迟加载资源
- **优化性能**：避免不必要的计算

### 基础示例

```python
def expensive_initialization(state: State) -> dict:
    """昂贵的初始化操作"""
    # 只在需要时才执行
    return {"initialized": True}

builder.add_node(
    "init",
    expensive_initialization,
    defer=True  # 延迟执行
)
```

### 注意事项

- 延迟执行的节点需要显式触发
- 适用于可选的或条件性的操作
- 可以与条件边结合使用

## 5. destinations - 目标节点

### 基本概念

`destinations` 定义了节点可以跳转到的目标节点列表。这主要用于 `Command` 对象的类型检查和验证。

[来源: reference/source_节点函数约定_01.md]

### 使用场景

- **类型安全**：确保 Command 指向有效的节点
- **文档化**：明确节点的可能跳转目标
- **验证**：在编译时检查路由逻辑

### 基础示例

```python
from langgraph.types import Command
from typing import Literal

def router_node(state: State) -> Command[Literal["process", "skip", "end"]]:
    """路由节点，根据条件跳转"""
    if state["value"] > 10:
        return Command(goto="process")
    elif state["value"] > 0:
        return Command(goto="skip")
    else:
        return Command(goto="end")

builder.add_node(
    "router",
    router_node,
    destinations=("process", "skip", "end")  # 定义可能的目标
)
```

[来源: reference/search_节点函数约定_01.md]

### 字典形式

你也可以使用字典形式提供更多信息：

```python
builder.add_node(
    "router",
    router_node,
    destinations={
        "process": "处理节点",
        "skip": "跳过节点",
        "end": "结束节点"
    }
)
```

## 完整代码示例

### 场景：带重试和缓存的 API 调用系统

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy, CachePolicy, Command
from typing_extensions import TypedDict, Literal
import random
import time

# 定义状态
class APIState(TypedDict):
    query: str
    result: str | None
    error: str | None
    attempts: int
    cached: bool

# 1. API 调用节点（带重试）
def call_external_api(state: APIState) -> dict:
    """调用外部 API，可能失败"""
    attempts = state.get("attempts", 0) + 1

    # 模拟 50% 失败率
    if random.random() < 0.5:
        raise ConnectionError("API connection failed")

    # 模拟 API 响应
    time.sleep(0.1)
    result = f"API result for: {state['query']}"

    return {
        "result": result,
        "error": None,
        "attempts": attempts,
        "cached": False
    }

# 2. 数据处理节点（带缓存）
def process_data(state: APIState) -> dict:
    """处理 API 返回的数据"""
    if not state.get("result"):
        return {"error": "No data to process"}

    # 模拟耗时处理
    time.sleep(0.5)
    processed = state["result"].upper()

    return {
        "result": processed,
        "cached": True
    }

# 3. 错误处理节点
def handle_error(state: APIState) -> dict:
    """处理错误情况"""
    return {
        "result": None,
        "error": f"Failed after {state.get('attempts', 0)} attempts"
    }

# 4. 路由节点
def route_result(state: APIState) -> Command[Literal["process", "error", "end"]]:
    """根据结果路由到不同节点"""
    if state.get("error"):
        return Command(goto="error")
    elif state.get("result"):
        return Command(goto="process")
    else:
        return Command(goto="end")

# 构建图
builder = StateGraph(APIState)

# 添加 API 调用节点（带重试策略）
builder.add_node(
    "api_call",
    call_external_api,
    retry_policy=RetryPolicy(
        initial_interval=0.5,
        backoff_factor=2.0,
        max_interval=5.0,
        max_attempts=3,
        jitter=True,
        retry_on=ConnectionError
    ),
    metadata={
        "description": "调用外部 API",
        "critical": True,
        "timeout_ms": 5000
    }
)

# 添加数据处理节点（带缓存策略）
builder.add_node(
    "process",
    process_data,
    cache_policy=CachePolicy(
        ttl=300,  # 缓存 5 分钟
        key_fn=lambda state: state.get("result", "")
    ),
    metadata={
        "description": "处理 API 数据",
        "expensive": True
    }
)

# 添加错误处理节点
builder.add_node(
    "error",
    handle_error,
    metadata={
        "description": "处理错误情况"
    }
)

# 添加路由节点
builder.add_node(
    "router",
    route_result,
    destinations=("process", "error", "end"),
    metadata={
        "description": "路由到不同处理节点"
    }
)

# 添加边
builder.add_edge(START, "api_call")
builder.add_edge("api_call", "router")
builder.add_edge("process", END)
builder.add_edge("error", END)

# 编译图
graph = builder.compile()

# 测试
result = graph.invoke({
    "query": "test query",
    "result": None,
    "error": None,
    "attempts": 0,
    "cached": False
})

print(f"Result: {result}")
```

## 最佳实践

### 1. 重试策略

- **合理设置重试次数**：避免无限重试
- **使用指数退避**：减少服务器压力
- **添加抖动**：避免雷鸣群效应
- **精确的重试条件**：只重试可恢复的错误

```python
# 好的实践
retry_policy=RetryPolicy(
    max_attempts=3,
    initial_interval=1.0,
    backoff_factor=2.0,
    max_interval=10.0,
    jitter=True,
    retry_on=lambda e: isinstance(e, (ConnectionError, TimeoutError))
)

# 避免
retry_policy=RetryPolicy(
    max_attempts=100,  # 太多
    retry_on=Exception  # 太宽泛
)
```

[来源: reference/source_节点函数约定_02.md]

### 2. 缓存策略

- **确保幂等性**：缓存的节点必须是幂等的
- **合理设置 TTL**：根据数据更新频率设置
- **注意内存使用**：大量缓存可能占用内存

### 3. 元数据

- **标准化字段**：团队内统一元数据字段
- **包含关键信息**：描述、版本、负责人
- **用于监控**：添加监控和告警相关信息

### 4. 延迟执行

- **明确触发条件**：文档化延迟节点的触发条件
- **避免过度使用**：只在必要时使用

### 5. 目标节点

- **使用类型注解**：利用 Literal 类型提供类型安全
- **验证路由逻辑**：确保所有可能的路径都有定义

## 常见问题

### Q1: 重试策略会影响性能吗？

是的，重试会增加执行时间。建议：
- 合理设置 `max_attempts`
- 使用精确的 `retry_on` 条件
- 监控重试次数和成功率

[来源: reference/source_节点函数约定_02.md]

### Q2: 缓存策略如何选择缓存键？

缓存键应该基于：
- 输入状态的关键字段
- 影响输出的所有因素
- 避免包含时间戳等变化字段

### Q3: 元数据会影响执行吗？

不会，元数据只用于文档化和监控，不影响节点执行逻辑。

### Q4: defer 和条件边有什么区别？

- `defer`：控制节点是否执行
- 条件边：控制执行流程的路由

两者可以结合使用。

### Q5: destinations 是必需的吗？

不是必需的，但强烈推荐使用，因为它提供：
- 类型安全
- 编译时验证
- 更好的文档化

[来源: reference/search_节点函数约定_01.md]

## 总结

节点配置选项为 LangGraph 提供了强大的控制能力：

1. **retry_policy**：自动重试失败的操作，提高系统鲁棒性
2. **cache_policy**：缓存计算结果，优化性能
3. **metadata**：文档化和监控节点
4. **defer**：延迟执行，优化资源使用
5. **destinations**：类型安全的路由定义

合理使用这些配置选项，可以构建更加健壮、高效和可维护的工作流系统。
