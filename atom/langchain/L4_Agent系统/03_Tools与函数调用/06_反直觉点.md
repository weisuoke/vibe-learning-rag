# 反直觉点

> 揭示 Tools 与函数调用中最常见的 3 个误区——description 的重要性被严重低估、工具数量与 Agent 能力不成正比、Tool 远不止是函数包装器。

---

## 误区 1："Tool 的 description 不重要，随便写就行" ❌

### 为什么错？

**错误观点**："函数名已经够清楚了，description 随便写个注释就行"

**正确理解**：

description 是 LLM 选择工具的**唯一依据**。模型不会看你的函数实现代码，它只能看到 name + description + args_schema。description 写得模糊，模型就会选错工具或干脆不选。

```python
from langchain_core.tools import tool

# ❌ 模糊的 description：LLM 不知道什么时候该用这个工具
@tool
def search(query: str) -> str:
    """搜索"""
    return do_search(query)

# ❌ 太简短：LLM 无法区分这个搜索和其他搜索工具
@tool
def search_kb(query: str) -> str:
    """搜索知识库"""
    return search_knowledge_base(query)

# ✅ 精确的 description：LLM 能准确判断何时使用
@tool
def search_kb(query: str) -> str:
    """在公司内部知识库中搜索技术文档。

    适用场景：用户询问公司内部的技术规范、API 文档、架构设计等问题。
    不适用：外部公开信息、实时新闻、个人问题。
    返回：最相关的 3 个文档片段及其相似度分数。
    """
    return search_knowledge_base(query)
```

**实测对比**——用户问"公司的请假制度是什么？"：

```python
# 模糊描述：LLM 可能随机选一个，甚至直接回答（不调用工具）
@tool
def tool_a(query: str) -> str:
    """搜索文档"""
    ...

@tool
def tool_b(query: str) -> str:
    """查询信息"""
    ...

# 精确描述：LLM 能准确选择 search_hr_docs，因为 description 明确包含"请假"
@tool
def search_hr_docs(query: str) -> str:
    """搜索人事制度文档，包含请假、考勤、薪资、福利、入职离职等规定。
    当用户询问公司人事相关政策时使用此工具。"""
    ...

@tool
def search_tech_docs(query: str) -> str:
    """搜索技术文档，包含 API 文档、架构设计、开发规范、部署流程等。
    当用户询问技术实现相关问题时使用此工具。"""
    ...
```

### 为什么人们容易这样错？

**心理原因**：
- 程序员习惯写代码注释给**人**看，不习惯写给 **AI** 看
- 觉得函数名 `search_hr_docs` 已经够清楚了——对人来说确实够了，但 LLM 需要更多上下文
- 把 description 当成可选的代码注释，而不是**核心功能参数**

**认知偏差**：
- "代码自文档化"的习惯——在传统开发中是好习惯，但 Tool 的 description 不是给开发者看的
- "名字说明一切"——`get_weather` 对人类很直观，但 LLM 需要知道"查询哪些城市？返回什么格式？实时还是预报？"

### 正确理解

**description 要像写 API 文档一样精确**，包含四个要素：

```python
@tool
def rag_search(query: str, top_k: int = 5) -> str:
    """在 RAG 知识库中检索相关文档片段。          ← 1. 功能：做什么

    适用场景：用户提出需要查阅知识库才能回答的问题，  ← 2. 适用场景：什么时候用
    如技术文档查询、产品规格确认、历史记录检索等。
    不适用于闲聊、数学计算、实时信息查询。           ← 2b. 不适用场景（同样重要）

    Args:
        query: 自然语言搜索查询，描述要查找的信息     ← 3. 参数含义
        top_k: 返回的文档片段数量，默认 5

    Returns:
        最相关的文档片段列表，包含文本内容和相似度分数  ← 4. 返回值
    """
    return search_vector_store(query, top_k=top_k)
```

**description 质量检查清单**：

| 检查项 | 差 | 好 |
|--------|----|----|
| 功能描述 | "搜索" | "在公司内部知识库中搜索技术文档" |
| 适用场景 | 无 | "当用户询问公司技术规范时使用" |
| 不适用场景 | 无 | "不适用于实时新闻、个人问题" |
| 返回值 | 无 | "返回最相关的 3 个文档片段" |

**RAG 实战建议**：当你有多个检索工具，description 中的**领域关键词**是 LLM 区分它们的关键。"包含请假、考勤、薪资"这几个词，就是 LLM 把"请假制度"路由到 HR 工具的依据。

## 误区 2："工具越多，Agent 越强大" ❌

### 为什么错？

**错误观点**："给 Agent 配 20 个工具，它就能处理 20 种任务"

**正确理解**：

工具过多会导致三个问题：选择困难（LLM 面对 20 个工具选错概率远高于 3 个）、Token 成本飙升（每个工具 Schema 约 100-200 tokens）、响应变慢（更长的 Prompt = 更慢的推理）。

```
工具数量 vs 准确率（经验数据）：

工具数量    选择准确率    额外 Token 成本
  3 个       ~95%          ~300 tokens
  5 个       ~90%          ~500 tokens
 10 个       ~80%         ~1000 tokens
 20 个       ~60%         ~2000 tokens
 50 个       ~40%         ~5000 tokens
```

**类比**：就像餐厅菜单——3 道菜的精选菜单，你能快速决定；200 道菜的大菜单，你反而不知道点什么，还可能点到不好吃的。

### 为什么人们容易这样错？

**心理原因**：
- "多多益善"的直觉——在传统编程中，提供更多 API 通常是好事
- 没有意识到 LLM 的工具选择是概率性的，不是确定性的

**认知偏差**：
- "能力 = 工具数量"——实际上能力 = 工具质量 x 选择准确率

### 正确理解

**核心原则：精选 3-5 个核心工具，用策略动态扩展。**

**策略 1：精选核心工具集**

```python
# ❌ 错误：一股脑塞 15 个工具 → 模型选择准确率大幅下降
all_tools = [search_hr, search_tech, search_finance, search_legal,
             search_product, calculator, weather, translator, ...]

# ✅ 正确：精选核心工具 → 准确率 ~95%
core_tools = [
    search_kb,      # 知识库检索（覆盖 80% 的查询需求）
    calculator,     # 数学计算
    web_search,     # 外部信息补充
]
```

**策略 2：用 Middleware 动态过滤**

当确实需要很多工具时，用 `LLMToolSelectorMiddleware` 根据上下文动态筛选：

```python
from langchain.agents import create_agent
from langchain.agents.middleware import LLMToolSelectorMiddleware

agent = create_agent(
    model="gpt-4.1",
    tools=all_tools,  # 注册 15 个工具
    middleware=[
        LLMToolSelectorMiddleware(
            model="gpt-4.1-mini",  # 用便宜模型做预筛选
            max_tools=3,            # 每次最多暴露 3 个给主模型
        ),
    ],
)
# 用户问 "请假制度是什么？"
# → mini 从 15 个中选出 search_hr → 主模型精准调用
```

**策略 3：工具分组 + 路由**

```python
# 按领域分组，先路由到组（3 选 1），再在组内选择（3 选 1）
# 两步都是 3 选 1，比 9 选 1 准确率高得多
tool_groups = {
    "人事": [search_hr, leave_request, attendance],
    "技术": [search_tech, code_runner, deploy],
    "财务": [search_finance, expense_report, budget],
}
```

**决策流程**：

```
你的 Agent 需要多少工具？
├─ ≤ 5 个 → 直接全部暴露，不需要选择策略
├─ 6-15 个 → 用 LLMToolSelectorMiddleware 动态筛选
└─ > 15 个 → 工具分组 + 路由，或者拆分成多个专用 Agent
```

## 误区 3："Tool 只能是普通函数" ❌

### 为什么错？

**错误观点**："Tool 就是用 @tool 包装一个 Python 函数"

**正确理解**：

Tool 是一个**标准接口**（`BaseTool`），任何实现了 `invoke()` 的对象都可以成为 Tool——包括 Chain、Retriever、甚至另一个 Agent。

```python
# ❌ 误区：以为 Tool 只能这样定义
@tool
def search(query: str) -> str:
    """搜索"""
    return do_search(query)

# ✅ 实际上 Tool 可以是：普通函数、Retriever、Chain、另一个 Agent、任何 Runnable
```

### 为什么人们容易这样错？

**心理原因**：
- 教程中 90% 的示例都是 `@tool` 装饰器包装函数，形成了思维定式
- "Tool"这个词本身暗示"工具 = 函数"

**认知偏差**：
- "示例 = 全部用法"——看到的都是函数示例，就以为只能用函数

### 正确理解

**Tool 是一个标准接口，不限于函数。** 以下是四种常见的非函数 Tool：

#### 用法 1：Retriever 转 Tool

RAG 中最常用的模式——把向量检索器包装成 Agent 可调用的工具。

```python
from langchain_core.tools import create_retriever_tool
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

vectorstore = Chroma(collection_name="knowledge_base", embedding_function=OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# 将 Retriever 转为 Tool — 本质是把 retriever.invoke(query) 包装成标准 Tool
retriever_tool = create_retriever_tool(
    retriever=retriever,
    name="search_knowledge_base",
    description="在知识库中搜索相关文档。当用户提出需要查阅资料才能回答的问题时使用。",
)

print(type(retriever_tool))  # <class 'langchain_core.tools.structured.StructuredTool'>
result = retriever_tool.invoke({"query": "什么是 RAG？"})
```

#### 用法 2：Chain 转 Tool

把一个 LCEL Chain 包装成 Tool，让 Agent 可以调用复杂的处理流程。

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

# 定义一个摘要 Chain
summarize_chain = (
    ChatPromptTemplate.from_template("请用 3 句话总结以下内容：\n\n{text}")
    | ChatOpenAI(model="gpt-4o-mini")
)

# 方式 1: 用 @tool 包装
@tool
def summarize(text: str) -> str:
    """将长文本总结为 3 句话的摘要。当需要压缩长文档内容时使用。"""
    return summarize_chain.invoke({"text": text}).content

# 方式 2: 用 as_tool() 直接转换（LangChain 0.3+）
summarize_tool = summarize_chain.as_tool(
    name="summarize",
    description="将长文本总结为 3 句话的摘要。",
)
```

#### 用法 3：任意 Runnable 转 Tool

LangChain 的 `as_tool()` 方法可以把任何 Runnable 转为 Tool：

```python
from langchain_core.runnables import RunnableLambda

def format_as_markdown(data: dict) -> str:
    """将字典数据格式化为 Markdown 表格"""
    headers = list(data.keys())
    header_row = "| " + " | ".join(headers) + " |"
    separator = "| " + " | ".join(["---"] * len(headers)) + " |"
    value_row = "| " + " | ".join(str(v) for v in data.values()) + " |"
    return f"{header_row}\n{separator}\n{value_row}"

markdown_tool = RunnableLambda(format_as_markdown).as_tool(
    name="format_markdown",
    description="将结构化数据格式化为 Markdown 表格。",
)
```

#### 用法 4：Agent 作为 Tool（Agent 嵌套）

```python
from langchain.agents import create_agent

# 创建专门做数据分析的子 Agent
data_agent = create_agent(model="gpt-4.1", tools=[calculator, chart_generator])

# 把子 Agent 包装成 Tool → 主 Agent 可以在需要时调用
@tool
def analyze_data(question: str) -> str:
    """调用数据分析专家来处理复杂的数据分析任务。
    适用于需要多步计算、数据可视化、统计分析的问题。"""
    result = data_agent.invoke({"input": question})
    return result["output"]

# 主 Agent 的工具列表中，analyze_data 背后是另一个完整的 Agent
main_agent = create_agent(
    model="gpt-4.1",
    tools=[search_kb, analyze_data, web_search],
)
```

### 四种 Tool 来源对比

| 来源 | 转换方式 | 适用场景 | 复杂度 |
|------|----------|----------|--------|
| 普通函数 | `@tool` 装饰器 | 简单操作（API 调用、计算） | 低 |
| Retriever | `create_retriever_tool()` | RAG 检索 | 低 |
| Chain | `chain.as_tool()` 或 `@tool` 包装 | 多步处理流程 | 中 |
| Agent | `@tool` 包装 Agent 调用 | 多 Agent 协作 | 高 |

### BaseTool 接口的本质

所有 Tool 最终都实现了同一个接口——`BaseTool`（继承自 `RunnableSerializable`），包含 `name`、`description`、`args_schema`、`invoke()`。无论底层是函数、Retriever、Chain 还是 Agent，对 LLM 来说都是同一个接口。

这就是为什么 Tool 不限于函数——它是一个**协议**，任何满足协议的对象都可以成为 Tool。

## 误区总结

| 误区 | 错误观点 | 正确理解 |
|------|----------|----------|
| **误区 1** | description 不重要，随便写 | description 是 LLM 选择工具的唯一依据，要像写 API 文档一样精确 |
| **误区 2** | 工具越多，Agent 越强大 | 工具过多导致选择困难、成本飙升、准确率下降；精选 3-5 个 + 动态过滤 |
| **误区 3** | Tool 只能是普通函数 | Tool 是标准接口，Retriever、Chain、Agent 都可以转为 Tool |

---

## 避免误区的建议

### 建议 1：把 description 当成产品文档来写

每个 Tool 的 description 至少包含：做什么、什么时候用、什么时候不用、返回什么。写完后自问："如果我是 LLM，只看这段描述，能准确判断何时使用这个工具吗？"

### 建议 2：遵循"少即是多"原则

从 3 个核心工具开始，只在确实需要时才增加。如果工具超过 5 个，引入 `LLMToolSelectorMiddleware` 做动态筛选。监控工具选择的准确率——如果低于 85%，先减少工具数量再排查原因。

### 建议 3：善用 Runnable 转 Tool

在 RAG 场景中，`create_retriever_tool()` 是最常用的模式。不要手动写检索逻辑再用 `@tool` 包装——直接把 Retriever 转为 Tool，代码更简洁，维护更方便。

### 建议 4：测试工具选择行为

定期用典型问题测试 LLM 的工具选择行为，确保 description 的质量：

```python
model_with_tools = ChatOpenAI(model="gpt-4o").bind_tools(tools)
response = model_with_tools.invoke("公司的请假制度是什么？")

if response.tool_calls:
    for tc in response.tool_calls:
        print(f"选择了: {tc['name']}, 参数: {tc['args']}")
else:
    print("没有选择任何工具（description 可能不够明确）")
```

---

**记住**：Tool 的威力不在于数量多，而在于 description 精确、数量精简、形式灵活。一个 description 写得好的 Retriever Tool，比 10 个描述模糊的函数 Tool 更有用。
