## 卡片 1：async fn 的本质——惰性状态机

### 🎯 核心点

**调用 async fn 不执行代码，只创建一个 Future 值。** 这是 Rust 与 JavaScript 最根本的区别。

### 代码对比

```typescript
// JavaScript — 调用即执行（eager）
const promise = fetchUser("123");  // ← HTTP 请求已经发出了！
// 即使你永远不 await，请求也已经在进行中
```

```rust
// Rust — 惰性的（lazy）
let future = fetch_user("123");    // ← 什么都没发生！只创建了一个值
// future 只是一个状态机实例，没有任何副作用
future.await;                       // ← 现在才真正开始执行
```

### 类比

JavaScript 的 `async` 像在线点外卖——下单那一刻厨房就开火了。Rust 的 `async` 像写了一张食谱卡——你可以传递它、丢掉它，只有厨师（运行时）拿起它才会开火。

### ZeroClaw 应用

ZeroClaw 有 **1939 个 async fn**，它们全部是惰性的。当 Agent 构建一个 LLM 调用链时，`provider.chat(messages)` 不会立即发起网络请求——只有链条末端的 `.await` 才会触发整个执行过程。这意味着你可以安全地构造、组合、甚至丢弃 Future，而不产生任何副作用。

## 卡片 2：Future trait——poll 驱动的异步模型

### 🎯 核心点

**Future 是一个可以被反复 poll 的状态机——每次 poll 要么返回"好了"，要么返回"还没好"。**

### 核心接口

```rust
trait Future {
    type Output;

    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

enum Poll<T> {
    Ready(T),   // "好了，结果在这"
    Pending,    // "还没好，好了会通知你"
}
```

### 类比：自助餐叫号器

去自助餐厅点餐，柜台给你一个叫号器（Future）：
- 你按一下查看（poll）→ 叫号器没响（`Pending`）→ 你去做别的事
- 你再按一下（poll）→ 叫号器响了（`Ready(食物)`）→ 取餐

关键在于：**叫号器没响时你不会傻站着等**——你可以去拿饮料、找座位。这就是异步的本质：等待时不阻塞，把 CPU 让给别的任务。

### 你需要手写 poll 吗？

**几乎不需要。** `async fn` 和 `.await` 是语法糖，编译器自动生成 poll 逻辑。ZeroClaw 1939 个 async fn 中没有一个手写 `poll`。理解 poll 是为了**读懂错误信息**和**理解取消行为**，而非手动实现。

## 卡片 3：.await——让出而非等待

### 🎯 核心点

**`.await` 不阻塞线程，而是让出 CPU——"我现在没事做，让别人先用"。**

### 串行 vs 并发

```rust
// ❌ 串行：一个做完再做下一个（总耗时 = A + B）
let user = fetch_user("123").await;       // 等 100ms
let order = fetch_order("456").await;     // 再等 100ms
// 总共 200ms

// ✅ 并发：两个同时进行（总耗时 = max(A, B)）
let (user, order) = tokio::join!(
    fetch_user("123"),       // 同时等
    fetch_order("456"),      // 同时等
);
// 总共 ~100ms
```

### 类比：叫号牌 vs 排队死等

- `.await` = **叫号牌**：领了号牌就可以去做别的事，叫到号再回来取
- `thread::sleep` = **排队死等**：站在柜台前不走，后面的人全部等着

### TypeScript 对照

```typescript
// TS 的 Promise.all() ≈ Rust 的 tokio::join!
const [user, order] = await Promise.all([
    fetchUser("123"),
    fetchOrder("456"),
]);
```

区别在于：JS 的 `Promise.all` 里的 Promise 在传入之前就已经开始执行了（eager），而 Rust 的 `join!` 是**同时开始 poll** 所有传入的 Future（lazy）。

## 卡片 4：Tokio 运行时——异步调度引擎

### 🎯 核心点

**Tokio 是 Future 的"厨房经理"——它决定哪个 Future 该被 poll、什么时候 poll、在哪个线程 poll。**

### #[tokio::main] 做了什么

```rust
// 你写的
#[tokio::main]
async fn main() {
    println!("Hello async!");
}

// 编译器展开为
fn main() {
    tokio::runtime::Builder::new_multi_thread()
        .enable_all()
        .build()
        .unwrap()
        .block_on(async {
            println!("Hello async!");
        });
}
```

`#[tokio::main]` 创建一个多线程运行时，然后在上面执行你的 async 代码。没有运行时，Future 就是一张没人看的食谱卡——永远不会被执行。

### 多线程工作窃取

```
Tokio 多线程调度器（默认 = CPU 核心数个线程）

线程 1: [任务A] [任务C] [任务E] ...
线程 2: [任务B] [任务D] [      ] ← 空闲了！
                                    ↓ 偷一个
线程 2: [任务B] [任务D] [任务F] ← 从线程 3 偷来的
线程 3: [任务F] [任务G] → [任务G]
```

当一个线程闲置时，它会从其他繁忙线程的队列里"偷"任务来执行——这就是**工作窃取（work-stealing）**。无需你手动分配，Tokio 自动负载均衡。

### ZeroClaw 应用

ZeroClaw 使用 `#[tokio::main]` 作为入口，启用了几乎所有 Tokio feature（`rt-multi-thread` / `macros` / `time` / `net` / `sync` / `signal` 等）。整个框架——Agent 循环、Channel 监听、Tool 执行——全部运行在 Tokio 调度器上。

## 卡片 5：tokio::spawn——后台任务

### 🎯 核心点

**`tokio::spawn` 创建一个独立的异步任务，在后台并发运行，不影响当前任务。**

```rust
// 生成后台任务，立即返回 JoinHandle
let handle = tokio::spawn(async {
    // 这段代码在后台独立运行
    do_heavy_work().await;
    42  // 返回值
});

// 当前任务继续执行...
do_other_stuff().await;

// 需要结果时再 await handle
let result = handle.await?;  // result = 42
```

### Send + 'static 约束

`tokio::spawn` 的任务可能被移动到**任何线程**执行，因此有两个关键约束：

| 约束 | 含义 | 类比 |
|------|------|------|
| `Send` | 数据可以安全地跨线程发送 | 包裹可以寄到别的城市 |
| `'static` | 不借用当前栈上的短命数据 | 包裹里不能放"借来的东西" |

```rust
// ❌ 编译错误：&local_ref 不是 'static
let local = String::from("hello");
tokio::spawn(async {
    println!("{}", &local);  // 借用了栈上的 local！
});

// ✅ 正确：move 将所有权转移进任务
let local = String::from("hello");
tokio::spawn(async move {
    println!("{}", local);   // local 被 move 进来了，任务拥有它
});
```

### TypeScript 对照

`tokio::spawn` ≈ Web Worker 的简化版——都是把任务丢到"别的地方"独立运行。但 Web Worker 需要序列化消息传递，而 `tokio::spawn` 可以直接移动 Rust 值（只要满足 `Send + 'static`）。

### ZeroClaw 应用

ZeroClaw 有 **14 个 `tokio::spawn`**，主要用于：组件监控（`spawn_component_supervisor`）、Channel 后台监听、心跳检测。每个 spawn 的任务都独立运行，崩溃不影响主流程。

## 卡片 6：tokio::select!——异步竞赛

### 🎯 核心点

**`select!` 同时等多个 Future，第一个完成的赢，其余被取消。**

```rust
tokio::select! {
    msg = rx.recv() => {
        // 收到消息了，处理消息
        handle_message(msg).await;
    }
    _ = tokio::time::sleep(Duration::from_secs(30)) => {
        // 30 秒超时了，没收到消息
        println!("超时！");
    }
    _ = cancellation_token.cancelled() => {
        // 收到取消信号，优雅退出
        println!("收到取消，退出循环");
        break;
    }
}
```

### 超时模式——最常用

```rust
// 给任何操作加超时
let result = tokio::select! {
    res = do_work() => res,                              // 工作完成
    _ = tokio::time::sleep(Duration::from_secs(10)) => { // 超时
        Err(anyhow!("操作超时"))
    }
};
```

### TypeScript 对照

```typescript
// select! ≈ Promise.race()
const result = await Promise.race([
    doWork(),
    timeout(10000),
]);
```

### ⚠️ 取消安全陷阱

当 `select!` 选中一个分支时，**其余分支的 Future 会被 drop（取消）**。如果一个 Future 已经读了一半数据就被取消了，数据就丢了——这叫**取消不安全**。

```rust
// ❌ 危险：每次循环都创建新的 recv future，之前读了一半的数据可能丢失
loop {
    tokio::select! {
        data = stream.next() => { /* ... */ }  // stream.next() 可能不是取消安全的
        _ = cancel.cancelled() => break,
    }
}
```

解决方案：使用 `tokio::pin!` 或在循环外创建 Future，确保不会重复创建。

### ZeroClaw 应用

ZeroClaw 有 **7 个 `tokio::select!`**，核心用途是 Agent 消息循环：同时等待新消息 (`rx.recv()`) 和取消信号 (`CancellationToken.cancelled()`)。当用户发送 `/stop` 命令时，取消信号触发，Agent 优雅退出。

## 卡片 7：mpsc 通道——任务间通信

### 🎯 核心点

**通过消息传递而非共享内存通信——这是 Rust 并发的哲学。** mpsc = Multi-Producer, Single-Consumer（多生产者，单消费者）。

```rust
use tokio::sync::mpsc;

// 创建有界通道（容量 100）
let (tx, mut rx) = mpsc::channel::<String>(100);

// 生产者（可以 clone 多份 tx）
let tx2 = tx.clone();
tokio::spawn(async move {
    tx.send("来自任务1".into()).await.unwrap();
});
tokio::spawn(async move {
    tx2.send("来自任务2".into()).await.unwrap();
});

// 消费者（只有一个 rx）
while let Some(msg) = rx.recv().await {
    println!("收到: {}", msg);
}
```

### 有界 vs 无界

| 类型 | 创建方式 | 满了怎么办 | 适用场景 |
|------|---------|-----------|---------|
| **有界** | `channel(100)` | `send().await` 会等待（背压） | ✅ 生产环境默认选择 |
| **无界** | `unbounded_channel()` | 无限堆积（OOM 风险） | ⚠️ 仅用于确定不会积压 |

**经验法则：永远用有界通道。** 无界通道在生产者速度 > 消费者速度时会导致内存无限增长。

### ZeroClaw Channel 架构

```
┌──────────────┐    mpsc::channel     ┌──────────────┐
│  Telegram     │───── tx.send() ─────→│              │
│  Channel      │                      │   Agent      │
└──────────────┘                      │              │
                                       │  rx.recv()   │
┌──────────────┐    mpsc::channel     │   .await     │
│  Discord      │───── tx.send() ─────→│              │
│  Channel      │                      │  (单消费者)   │
└──────────────┘                      └──────────────┘
```

ZeroClaw 有 **31 个 mpsc 通道使用**。每个 Channel（Telegram / Discord / CLI）持有 `tx`（发送端），Agent 持有 `rx`（接收端）。多个 Channel 可以同时向同一个 Agent 发送消息——这就是"多生产者，单消费者"的经典应用。

## 卡片 8：#[async_trait]——让 Trait Object 支持 async

### 🎯 核心点

**`#[async_trait]` 将 async fn 转为返回 `Pin<Box<dyn Future>>` 的普通方法，从而让 `dyn Trait` 支持异步。**

### 为什么需要它？

```rust
// ❌ 编译错误：async fn 返回匿名类型，不满足对象安全
trait Provider {
    async fn chat(&self) -> Response;  // 返回类型是 impl Future<...>，每个实现不同
}
// 无法创建 Box<dyn Provider>！

// ✅ 用 #[async_trait] 解决
#[async_trait]
trait Provider: Send + Sync {
    async fn chat(&self) -> Response;
    // 宏展开为：
    // fn chat(&self) -> Pin<Box<dyn Future<Output = Response> + Send + '_>>
    // Pin<Box<dyn Future>> 是固定大小，满足对象安全！
}
```

### ZeroClaw 四大核心 Trait

```rust
#[async_trait]
trait Provider: Send + Sync {           // LLM 调用
    async fn chat(&self, msgs: &[Message]) -> Result<Response>;
}

#[async_trait]
trait Channel: Send + Sync {            // 消息收发
    async fn listen(&self, tx: Sender<Message>) -> Result<()>;
    async fn send(&self, msg: &str) -> Result<()>;
}

#[async_trait]
trait Tool: Send + Sync {               // 工具执行
    async fn execute(&self, input: &str) -> Result<String>;
}

#[async_trait]
trait Memory: Send + Sync {             // 记忆存取
    async fn store(&self, entry: &Entry) -> Result<()>;
    async fn recall(&self, query: &str) -> Result<Vec<Entry>>;
}
```

ZeroClaw 有 **124 个 `#[async_trait]`** 使用。所有核心 Trait 都是异步的——因为它们的操作（网络请求、磁盘 IO、数据库查询）本质上都是 IO 密集型的。

### 何时可以不用？

Rust 1.75+ 支持 `async fn` in trait（AFIT），但仅限**静态分发**（`impl Trait`）。如果你需要 `dyn Trait`（动态分发），仍然需要 `#[async_trait]`。ZeroClaw 的架构依赖动态分发，所以全部使用 `#[async_trait]`。

## 卡片 9：阻塞代码的正确处理

### 🎯 核心点

**永远不要在 async 上下文中阻塞线程——这会"饿死"其他所有异步任务。**

### 问题

```rust
// ❌ 致命错误：阻塞了 Tokio 工作线程！
async fn bad_handler() {
    std::thread::sleep(Duration::from_secs(5));  // 阻塞！
    // 这 5 秒内，这个线程上的所有其他异步任务都无法推进
}
```

Tokio 默认只有几个工作线程（= CPU 核心数）。如果你用 `std::thread::sleep` 阻塞一个线程 5 秒，意味着系统 1/N 的处理能力被冻结了。

### 两种正确方案

```rust
// ✅ 方案 1：用 tokio 异步版本（推荐）
async fn good_handler() {
    tokio::time::sleep(Duration::from_secs(5)).await;  // 让出线程！
    // 等待期间其他任务正常运行
}

// ✅ 方案 2：用 spawn_blocking 隔离阻塞代码
async fn compute_heavy() -> i32 {
    // 把阻塞操作扔到专用的阻塞线程池
    tokio::task::spawn_blocking(|| {
        // 这里可以安全地做 CPU 密集计算或调用阻塞 API
        heavy_sync_computation()
    }).await.unwrap()
}
```

### 常见阻塞陷阱

| ❌ 阻塞版本 | ✅ 异步版本 |
|------------|-----------|
| `std::thread::sleep()` | `tokio::time::sleep().await` |
| `std::fs::read()` | `tokio::fs::read().await` |
| `reqwest::blocking::get()` | `reqwest::get().await` |
| CPU 密集计算 | `spawn_blocking(|| ...)` |
| 同步数据库查询 | 使用异步数据库驱动 |

### ZeroClaw 应用

ZeroClaw 有 **48 个 `tokio::time` 使用**，全部是异步的 sleep / timeout / interval。在组件监控中，指数退避重启使用 `tokio::time::sleep`（而非 `std::thread::sleep`），确保等待期间其他组件正常运行。

## 卡片 10：ZeroClaw 异步架构速览

### 🎯 核心点

**把卡片 1-9 的知识组合起来，看 ZeroClaw 如何构建完整的异步架构。**

### 完整调用链

```
#[tokio::main]                          ← 卡片 4: Tokio 运行时
  │
  ├── Daemon::run()                     ← async fn（卡片 1）
  │     │
  │     ├── tokio::spawn(supervisor)    ← 卡片 5: 后台任务
  │     │     │
  │     │     ├── channel.listen(tx)    ← #[async_trait]（卡片 8）
  │     │     │     │
  │     │     │     └── tx.send(msg)    ← 卡片 7: mpsc 通道
  │     │     │
  │     │     └── 失败时指数退避重启      ← tokio::time::sleep（卡片 9）
  │     │
  │     └── Agent::run(rx)              ← async fn
  │           │
  │           ├── tokio::select! {      ← 卡片 6: 异步竞赛
  │           │     rx.recv()           ← 收消息
  │           │     cancel.cancelled()  ← 取消信号
  │           │   }
  │           │
  │           ├── provider.chat()       ← #[async_trait]（卡片 8）
  │           │     .await              ← 卡片 3: 让出
  │           │
  │           ├── tool.execute()        ← #[async_trait]（卡片 8）
  │           │     .await              ← Future 惰性执行（卡片 1-2）
  │           │
  │           └── memory.store()        ← #[async_trait]（卡片 8）
  │                 .await
  │
  └── signal::ctrl_c().await            ← 优雅关闭
```

### 组件监控模式

```rust
// ZeroClaw 的核心可靠性模式
async fn spawn_component_supervisor(name: &str, factory: impl Fn() -> Component) {
    let mut backoff = Duration::from_secs(1);

    loop {
        let component = factory();
        match component.run().await {                // ← 运行组件
            Ok(_) => break,                          // 正常退出
            Err(e) => {
                log::error!("{} 崩溃: {}, {}s 后重启", name, e, backoff.as_secs());
                tokio::time::sleep(backoff).await;   // ← 异步等待，不阻塞
                backoff = (backoff * 2).min(Duration::from_secs(60)); // 指数退避
            }
        }
    }
}
```

### 优雅关闭

```
用户按 Ctrl+C
     │
     ▼
signal::ctrl_c().await 触发
     │
     ▼
CancellationToken.cancel()         ← 广播取消信号
     │
     ├──→ Agent select! 收到 → 保存状态 → 退出循环
     ├──→ Channel select! 收到 → 断开连接 → 退出
     └──→ Supervisor 收到 → 不再重启 → 退出
     │
     ▼
所有 JoinHandle.await 完成         ← 等待所有任务结束
     │
     ▼
程序退出
```
