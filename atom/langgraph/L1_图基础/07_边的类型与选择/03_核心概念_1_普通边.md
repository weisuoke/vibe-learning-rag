# 核心概念 1：普通边（Normal Edge）

## 概念定义

**普通边（Normal Edge）是 LangGraph 中最基础的边类型，通过 `add_edge()` 方法添加，用于在两个节点之间建立固定的、确定性的连接关系。**

当一个节点执行完成后，普通边会自动将执行流程传递到下一个节点，无需任何条件判断或动态决策。

---

## API 签名

```python
def add_edge(self, start_key: str | list[str], end_key: str) -> Self:
    """Add a directed edge from the start node (or list of start nodes) to the end node.

    When a single start node is provided, the graph will wait for that node to complete
    before executing the end node. When multiple start nodes are provided,
    the graph will wait for ALL of the start nodes to complete before executing the end node.

    Args:
        start_key: 起始节点名称（单个）或起始节点列表（多个）
        end_key: 目标节点名称

    Returns:
        Self: 返回 StateGraph 实例，支持链式调用
    """
```

**两种使用形式：**

1. **单起始节点**：`add_edge("A", "B")` - 节点 A 完成后执行节点 B
2. **多起始节点**：`add_edge(["A", "B"], "C")` - 节点 A 和 B 都完成后才执行节点 C（等待边）

---

## 详细解释

### 1. 工作原理

普通边在 LangGraph 中的工作机制：

```
节点 A 执行完成
    ↓
触发普通边
    ↓
写入目标节点的触发通道
    ↓
节点 B 开始执行
```

**内部实现机制**（基于源码分析）：

```python
# 编译时，普通边被转换为通道写入操作
# sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py:1306-1321

def attach_edge(self, starts: str | Sequence[str], end: str) -> None:
    if isinstance(starts, str):
        # 单起始节点：直接写入目标节点的触发通道
        if end != END:
            self.nodes[starts].writers.append(
                ChannelWrite((ChannelWriteEntry(_CHANNEL_BRANCH_TO.format(end), None),))
            )
```

**关键特征：**
- **确定性**：执行路径固定，不依赖状态或条件
- **自动触发**：起始节点完成后自动触发
- **单向传递**：只能从起始节点传递到目标节点
- **状态继承**：目标节点接收起始节点更新后的状态

### 2. 存储机制

普通边在 StateGraph 中的存储方式：

```python
# 单起始节点的边存储在 self.edges 集合中
self.edges.add((start_key, end_key))

# 多起始节点的边存储在 self.waiting_edges 集合中
self.waiting_edges.add((tuple(start_key), end_key))
```

### 3. 使用场景

**适用场景：**
- ✅ 线性流程：节点按固定顺序执行
- ✅ 确定性工作流：不需要条件判断
- ✅ 简单的前后依赖关系
- ✅ 并行任务的同步点（多起始节点）

**不适用场景：**
- ❌ 需要根据状态动态决策下一个节点
- ❌ 需要条件分支
- ❌ 需要动态并行执行

## 代码示例

### 示例 1：基础线性流程

```python
"""
普通边示例 1：基础线性流程
演示：使用普通边构建简单的顺序执行流程
"""

from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict


# ===== 1. 定义状态 =====
class State(TypedDict):
    input: str
    step1_result: str
    step2_result: str
    final_result: str


# ===== 2. 定义节点函数 =====
def step1(state: State) -> dict:
    """第一步：处理输入"""
    print(f"Step 1: 处理输入 '{state['input']}'")
    return {"step1_result": f"已处理: {state['input']}"}


def step2(state: State) -> dict:
    """第二步：转换结果"""
    print(f"Step 2: 转换 '{state['step1_result']}'")
    return {"step2_result": f"已转换: {state['step1_result']}"}


def step3(state: State) -> dict:
    """第三步：生成最终结果"""
    print(f"Step 3: 生成最终结果")
    return {"final_result": f"完成: {state['step2_result']}"}


# ===== 3. 构建图 =====
builder = StateGraph(State)

# 添加节点
builder.add_node("step1", step1)
builder.add_node("step2", step2)
builder.add_node("step3", step3)

# 添加普通边：构建线性流程
builder.add_edge(START, "step1")      # 开始 -> 步骤1
builder.add_edge("step1", "step2")    # 步骤1 -> 步骤2
builder.add_edge("step2", "step3")    # 步骤2 -> 步骤3
builder.add_edge("step3", END)        # 步骤3 -> 结束

# 编译图
graph = builder.compile()


# ===== 4. 执行图 =====
print("=== 执行线性流程 ===\n")
result = graph.invoke({"input": "Hello LangGraph"})

print("\n=== 最终状态 ===")
print(f"输入: {result['input']}")
print(f"步骤1结果: {result['step1_result']}")
print(f"步骤2结果: {result['step2_result']}")
print(f"最终结果: {result['final_result']}")
```

**运行输出：**
```
=== 执行线性流程 ===

Step 1: 处理输入 'Hello LangGraph'
Step 2: 转换 '已处理: Hello LangGraph'
Step 3: 生成最终结果

=== 最终状态 ===
输入: Hello LangGraph
步骤1结果: 已处理: Hello LangGraph
步骤2结果: 已转换: 已处理: Hello LangGraph
最终结果: 完成: 已转换: 已处理: Hello LangGraph
```

### 示例 2：多起始节点（等待边）

```python
"""
普通边示例 2：多起始节点（等待边）
演示：等待多个并行任务完成后再执行下一步
"""

from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict
import time


# ===== 1. 定义状态 =====
class ParallelState(TypedDict):
    task_a_result: str
    task_b_result: str
    task_c_result: str
    merged_result: str


# ===== 2. 定义并行任务节点 =====
def task_a(state: ParallelState) -> dict:
    """任务 A：模拟耗时操作"""
    print("Task A: 开始执行...")
    time.sleep(0.1)
    print("Task A: 完成")
    return {"task_a_result": "A完成"}


def task_b(state: ParallelState) -> dict:
    """任务 B：模拟耗时操作"""
    print("Task B: 开始执行...")
    time.sleep(0.15)
    print("Task B: 完成")
    return {"task_b_result": "B完成"}


def task_c(state: ParallelState) -> dict:
    """任务 C：模拟耗时操作"""
    print("Task C: 开始执行...")
    time.sleep(0.12)
    print("Task C: 完成")
    return {"task_c_result": "C完成"}


def merge_results(state: ParallelState) -> dict:
    """合并所有任务的结果"""
    print("\nMerge: 合并所有任务结果...")
    merged = f"{state['task_a_result']}, {state['task_b_result']}, {state['task_c_result']}"
    return {"merged_result": merged}


# ===== 3. 构建图 =====
builder = StateGraph(ParallelState)

# 添加节点
builder.add_node("task_a", task_a)
builder.add_node("task_b", task_b)
builder.add_node("task_c", task_c)
builder.add_node("merge", merge_results)

# 添加普通边：并行启动三个任务
builder.add_edge(START, "task_a")
builder.add_edge(START, "task_b")
builder.add_edge(START, "task_c")

# 添加等待边：等待所有任务完成后再合并
# 这是普通边的特殊用法：多起始节点
builder.add_edge(["task_a", "task_b", "task_c"], "merge")

# 合并后结束
builder.add_edge("merge", END)

# 编译图
graph = builder.compile()


# ===== 4. 执行图 =====
print("=== 执行并行任务 ===\n")
start_time = time.time()
result = graph.invoke({})
end_time = time.time()

print("\n=== 执行结果 ===")
print(f"任务A结果: {result['task_a_result']}")
print(f"任务B结果: {result['task_b_result']}")
print(f"任务C结果: {result['task_c_result']}")
print(f"合并结果: {result['merged_result']}")
print(f"\n总耗时: {end_time - start_time:.2f}秒")
print("注意：并行执行，总耗时约等于最慢任务的耗时")
```

**运行输出：**
```
=== 执行并行任务 ===

Task A: 开始执行...
Task B: 开始执行...
Task C: 开始执行...
Task A: 完成
Task C: 完成
Task B: 完成

Merge: 合并所有任务结果...

=== 执行结果 ===
任务A结果: A完成
任务B结果: B完成
任务C结果: C完成
合并结果: A完成, B完成, C完成

总耗时: 0.15秒
注意：并行执行，总耗时约等于最慢任务的耗时
```

### 示例 3：RAG 工作流中的普通边

```python
"""
普通边示例 3：RAG 工作流中的应用
演示：在 RAG 系统中使用普通边构建检索-生成流程
"""

from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict
from typing import List


# ===== 1. 定义 RAG 状态 =====
class RAGState(TypedDict):
    query: str
    documents: List[str]
    context: str
    answer: str


# ===== 2. 定义 RAG 节点 =====
def retrieve_documents(state: RAGState) -> dict:
    """检索相关文档"""
    query = state["query"]
    print(f"Retrieve: 检索与 '{query}' 相关的文档...")

    # 模拟检索结果
    documents = [
        f"文档1: {query}的定义是...",
        f"文档2: {query}的应用场景包括...",
        f"文档3: {query}的最佳实践是..."
    ]

    print(f"Retrieve: 找到 {len(documents)} 个相关文档")
    return {"documents": documents}


def build_context(state: RAGState) -> dict:
    """构建上下文"""
    documents = state["documents"]
    print(f"\nBuild Context: 构建上下文...")

    # 合并文档内容
    context = "\n".join(documents)

    print(f"Build Context: 上下文长度 {len(context)} 字符")
    return {"context": context}


def generate_answer(state: RAGState) -> dict:
    """生成答案"""
    query = state["query"]
    context = state["context"]

    print(f"\nGenerate: 基于上下文生成答案...")

    # 模拟 LLM 生成
    answer = f"根据检索到的文档，{query}是一个重要的概念，具有广泛的应用场景。"

    print(f"Generate: 答案生成完成")
    return {"answer": answer}


# ===== 3. 构建 RAG 图 =====
builder = StateGraph(RAGState)

# 添加节点
builder.add_node("retrieve", retrieve_documents)
builder.add_node("build_context", build_context)
builder.add_node("generate", generate_answer)

# 添加普通边：构建 RAG 流程
# START -> 检索 -> 构建上下文 -> 生成答案 -> END
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "build_context")
builder.add_edge("build_context", "generate")
builder.add_edge("generate", END)

# 编译图
graph = builder.compile()


# ===== 4. 执行 RAG 查询 =====
print("=== 执行 RAG 查询 ===\n")
result = graph.invoke({"query": "LangGraph"})

print("\n=== RAG 结果 ===")
print(f"查询: {result['query']}")
print(f"\n检索到的文档:")
for i, doc in enumerate(result['documents'], 1):
    print(f"  {i}. {doc}")
print(f"\n生成的答案:")
print(f"  {result['answer']}")
```

**运行输出：**
```
=== 执行 RAG 查询 ===

Retrieve: 检索与 'LangGraph' 相关的文档...
Retrieve: 找到 3 个相关文档

Build Context: 构建上下文...
Build Context: 上下文长度 90 字符

Generate: 基于上下文生成答案...
Generate: 答案生成完成

=== RAG 结果 ===
查询: LangGraph

检索到的文档:
  1. 文档1: LangGraph的定义是...
  2. 文档2: LangGraph的应用场景包括...
  3. 文档3: LangGraph的最佳实践是...

生成的答案:
  根据检索到的文档，LangGraph是一个重要的概念，具有广泛的应用场景。
```

## 在 LangGraph 工作流中的应用

### 1. 与 START 和 END 的配合

```python
# 入口边：从 START 到第一个节点
builder.add_edge(START, "first_node")

# 出口边：从最后一个节点到 END
builder.add_edge("last_node", END)
```

**注意事项：**
- START 只能作为起始节点，不能作为目标节点
- END 只能作为目标节点，不能作为起始节点

### 2. 与条件边的配合

普通边和条件边可以混合使用：

```python
# 普通边：固定路由
builder.add_edge(START, "process")

# 条件边：动态路由
builder.add_conditional_edges(
    "process",
    route_function,
    {"path_a": "node_a", "path_b": "node_b"}
)

# 普通边：汇聚到结束
builder.add_edge("node_a", END)
builder.add_edge("node_b", END)
```

### 3. 链式调用

`add_edge()` 返回 `Self`，支持链式调用：

```python
builder = (
    StateGraph(State)
    .add_node("step1", step1)
    .add_node("step2", step2)
    .add_node("step3", step3)
    .add_edge(START, "step1")
    .add_edge("step1", "step2")
    .add_edge("step2", "step3")
    .add_edge("step3", END)
)
```

## 核心要点总结

### 优势

1. **简单直观**：最容易理解和使用的边类型
2. **性能高效**：无需条件判断，执行效率高
3. **可预测性**：执行路径固定，易于调试和维护
4. **支持并行同步**：多起始节点可以实现并行任务的同步

### 限制

1. **缺乏灵活性**：无法根据状态动态决策
2. **不支持条件分支**：需要条件分支时必须使用条件边
3. **固定路由**：一旦定义，执行路径不可改变

### 最佳实践

1. **优先使用普通边**：对于确定性流程，优先使用普通边而非条件边
2. **合理使用等待边**：需要同步多个并行任务时，使用多起始节点的普通边
3. **避免过度复杂**：如果流程需要大量条件判断，考虑重构节点逻辑
4. **清晰命名**：节点名称要清晰，便于理解边的连接关系
