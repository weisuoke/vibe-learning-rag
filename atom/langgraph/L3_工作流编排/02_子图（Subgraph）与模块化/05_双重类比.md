# 【双重类比】LangGraph 子图（Subgraph）与模块化

> 用你熟悉的概念，秒懂 LangGraph 子图设计

---

## 类比1：子图 = 封装与复用

### 前端类比：React 子组件

子图就像 React 中的子组件。你不会把所有 UI 逻辑写在一个巨大的 `App.tsx` 里，而是拆成 `<Header>`、`<Sidebar>`、`<ChatWindow>` 等子组件。每个子组件有自己的内部状态和逻辑，父组件只需要引用它、传入 props、接收回调。

LangGraph 的子图也一样：一个编译后的 `StateGraph` 可以直接作为父图的一个节点。子图内部有自己的节点和边，父图不需要关心子图内部怎么运行，只关心输入什么、输出什么。

### 日常生活类比：公司部门

一家公司不会让 CEO 亲自处理每一件事，而是设立财务部、技术部、市场部等部门。每个部门内部有自己的流程和分工，CEO 只需要下达任务、接收汇报。子图就是这些独立运作的部门——内部自治，对外提供标准化的输入输出接口。

---

## 类比2：input_schema / output_schema = 接口契约

### 前端类比：TypeScript Props 接口

在 TypeScript 中，你会用 `interface Props { title: string; onSubmit: (data: FormData) => void }` 来定义组件接口。调用者必须按接口传参，组件内部也只暴露约定好的输出。这就是契约。

LangGraph 子图的 `input_schema` 和 `output_schema` 作用完全相同。`input_schema` 定义父图传给子图的数据结构，`output_schema` 定义子图返回给父图的数据结构。不在 schema 里的字段，对方看不到。

### 日常生活类比：部门间交接单

部门之间协作靠的是标准化的交接单。市场部给技术部提需求，要填《需求申请表》（input_schema）；技术部交付成果，要填《验收报告》（output_schema）。表格之外的信息（比如技术部内部的代码评审记录），市场部无权也无需知道。

---

## 类比3：共享状态键 = 自动共享机制

### 前端类比：React Context API

React Context 让深层嵌套的组件无需逐层传 props 就能共享数据。只要子组件和父组件使用同一个 Context，数据自动流通。

LangGraph 的共享状态键机制类似：如果子图和父图的 State 中有同名的 key（比如都有 `messages`），数据会自动在父子图之间传递，无需手动转换。这是最简单的集成方式——命名一致，数据自通。

### 日常生活类比：公司公告板

公司大厅有一块公告板，所有部门都能看到上面的信息（共享 key）。HR 发布了新的考勤制度，技术部、市场部、财务部自动就知道了，不需要 HR 挨个部门去通知。共享状态键就是这块公告板——同名即共享，自动同步。

---

## 类比4：Command.PARENT = 反向通信

### 前端类比：子组件 emit 事件给父组件

在 Vue 中子组件通过 `$emit('update', data)` 通知父组件；在 React 中通过调用父组件传入的回调函数 `props.onUpdate(data)` 实现。核心思想是：子组件主动向上传递信息。

LangGraph 的 `Command(goto="some_node", graph=Command.PARENT)` 做的就是同一件事。子图中的某个节点可以主动向父图发送指令："跳转到父图的某个节点"，并附带需要更新的状态数据。这打破了单向的自顶向下控制流。

### 日常生活类比：员工向上级汇报

正常流程是 CEO 下达任务给部门，部门执行后汇报。但遇到紧急情况，员工可以越级汇报——直接告诉上级"这件事需要你来决策"。`Command.PARENT` 就是这种反向通信机制：子图不只是被动执行，它可以主动影响父图的执行流程。

---

## 类比5：Send 动态分发 = 并行执行

### 前端类比：Promise.all()

当你需要同时请求多个 API 时，用 `Promise.all([fetchA(), fetchB(), fetchC()])` 可以并行发出请求，等全部完成后再汇总结果。每个 Promise 独立执行，互不干扰。

LangGraph 的 `Send("process_item", item)` 机制类似。父图可以动态创建多个子图实例，每个实例处理不同的数据（比如对 10 个文档分别做摘要），最后汇总结果。这就是经典的 map-reduce 模式。

### 日常生活类比：CEO 同时派发任务给多个部门

CEO 在年会上同时给五个部门布置不同的任务："市场部做用户调研、技术部做性能优化、财务部做年度预算……" 五个部门同时开工，各干各的，最后在季度会议上汇总成果。Send 就是这个"同时派发、各自执行、统一汇总"的过程。

---

## 总结对照表

| LangGraph 子图概念 | 前端类比 | 日常生活类比 | 核心要点 |
|---|---|---|---|
| 子图（Subgraph） | React 子组件 | 公司部门 | 封装内部逻辑，对外标准接口 |
| input/output_schema | TypeScript Props 接口 | 部门间交接单 | 明确输入输出契约，隔离内部细节 |
| 共享状态键 | React Context API | 公司公告板 | 同名 key 自动传递，零配置共享 |
| Command.PARENT | 子组件 emit 事件 | 员工向上级汇报 | 子图主动影响父图执行流程 |
| Send 动态分发 | Promise.all() | CEO 同时派发任务 | 并行创建多实例，map-reduce 汇总 |

> **一句话记忆：** 子图就是"独立运作的部门"，通过交接单（schema）协作，通过公告板（共享key）同步，必要时可以向上汇报（Command.PARENT），也可以并行派发（Send）。
