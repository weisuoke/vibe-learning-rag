## 场景概述

**目标**：构建一个多轮对话管理系统，演示循环在对话场景中的自然应用

**核心技术**：
- `add_messages` reducer 自动累积对话历史
- 条件边实现对话循环（继续 / 结束）
- 消息修剪（message pruning）防止 context window 溢出
- 对话摘要压缩长历史

**实际应用**：
- 智能客服多轮对话
- RAG 对话式问答
- Agent 交互式任务执行
- 面试机器人

**为什么对话是最自然的循环场景？**

对话本身就是一个循环：用户说 → 系统回 → 用户再说 → 系统再回 → ... → 用户说"再见"。这和 LangGraph 的条件边循环完美对应：路由函数判断用户是否要结束对话，决定继续循环还是走向 END。

## 核心知识：add_messages Reducer

在写代码之前，先理解 LangGraph 内置的消息 reducer。

### 什么是 add_messages？

```python
from langgraph.graph.message import add_messages
```

`add_messages` 是 LangGraph 专门为对话场景设计的 reducer 函数。它比 `operator.add` 更智能：

| 特性 | `operator.add` | `add_messages` |
|------|----------------|----------------|
| 基本行为 | 简单列表拼接 | 智能消息合并 |
| 去重 | 不支持 | 按 message ID 去重 |
| 更新 | 不支持 | 同 ID 消息会被更新而非追加 |
| 删除 | 不支持 | 支持 `RemoveMessage` 操作 |
| 适用场景 | 通用列表累积 | 对话消息管理 |

### 源码关键逻辑

```python
# [来源: sourcecode/langgraph/graph/message.py]
# add_messages 的核心逻辑：
# 1. 如果新消息的 ID 已存在于旧列表中 → 更新（替换）
# 2. 如果新消息是 RemoveMessage → 从列表中删除对应消息
# 3. 否则 → 追加到列表末尾
```

这意味着在对话循环中，每轮迭代追加的消息会自动累积，同时支持精确的消息管理操作。

## 完整代码示例 1：基础多轮对话循环

```python
"""
LangGraph 多轮对话管理 - 基础版
演示：对话循环 + add_messages + 条件终止

图结构：
  START → process_message → [路由判断]
                              ├─ "bye" → END
                              └─ 继续 → wait_for_input → process_message
"""

import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages


# ===== 1. 状态定义 =====
class ConversationState(TypedDict):
    messages: Annotated[list, add_messages]  # 对话历史（智能累积）
    turn_count: int                          # 对话轮次
    summary: str                             # 对话摘要（用于长对话压缩）


# ===== 2. 节点实现 =====
def process_message_node(state: ConversationState) -> dict:
    """
    处理用户消息，生成回复（模拟 LLM）

    在真实场景中，这里会调用 LLM API：
      response = llm.invoke(state["messages"])

    这里用规则模拟，让代码可以直接运行。
    """
    messages = state["messages"]
    turn_count = state.get("turn_count", 0) + 1

    # 获取最后一条用户消息
    last_message = messages[-1] if messages else None
    user_text = ""
    if last_message:
        # 兼容 dict 和 BaseMessage 两种格式
        if isinstance(last_message, dict):
            user_text = last_message.get("content", "")
        else:
            user_text = getattr(last_message, "content", str(last_message))

    # 模拟 LLM 回复逻辑
    user_lower = user_text.lower().strip()
    if "天气" in user_text or "weather" in user_lower:
        reply = f"今天天气不错！适合出门散步。（第{turn_count}轮）"
    elif "rag" in user_lower:
        reply = f"RAG 是检索增强生成的缩写，结合了检索和生成两个步骤。（第{turn_count}轮）"
    elif "你好" in user_text or "hello" in user_lower or "hi" in user_lower:
        reply = f"你好！有什么我可以帮你的吗？（第{turn_count}轮）"
    elif "谢谢" in user_text or "thanks" in user_lower:
        reply = f"不客气！还有其他问题吗？（第{turn_count}轮）"
    else:
        reply = f"收到你的消息：'{user_text}'。我会尽力帮助你！（第{turn_count}轮）"

    print(f"  [轮次 {turn_count}] 用户: {user_text}")
    print(f"  [轮次 {turn_count}] 助手: {reply}")

    return {
        "messages": [{"role": "assistant", "content": reply}],
        "turn_count": turn_count,
    }


# ===== 3. 路由函数 =====
def should_continue(state: ConversationState) -> Literal["wait_for_input", "__end__"]:
    """
    判断对话是否结束：
    - 用户说 "bye"/"再见"/"退出" → 结束
    - 否则 → 继续等待下一轮输入
    """
    messages = state["messages"]
    if not messages:
        return END

    last_message = messages[-1]
    if isinstance(last_message, dict):
        content = last_message.get("content", "")
    else:
        content = getattr(last_message, "content", "")

    # 检查结束关键词
    end_keywords = ["bye", "再见", "退出", "结束", "quit", "exit"]
    if any(kw in content.lower() for kw in end_keywords):
        print("  [路由] → 对话结束")
        return END

    print("  [路由] → 继续对话")
    return "wait_for_input"


def wait_for_input_node(state: ConversationState) -> dict:
    """
    等待用户输入的占位节点。

    在真实应用中，这里会使用 interrupt() 暂停图执行，
    等待外部系统（如 Web API）注入用户的下一条消息。

    这里为了演示，直接返回空更新。
    """
    return {}


# ===== 4. 构建图 =====
builder = StateGraph(ConversationState)

# 添加节点
builder.add_node("process_message", process_message_node)
builder.add_node("wait_for_input", wait_for_input_node)

# 添加边
builder.add_edge(START, "process_message")
builder.add_conditional_edges("process_message", should_continue)
builder.add_edge("wait_for_input", "process_message")

graph = builder.compile()


# ===== 5. 模拟多轮对话 =====
print("=" * 60)
print("场景1：基础多轮对话循环")
print("=" * 60)

# 模拟用户的多轮输入
user_inputs = [
    "你好",
    "今天天气怎么样？",
    "RAG 是什么？",
    "谢谢",
    "bye",
]

# 逐轮执行对话
state = {"messages": [], "turn_count": 0, "summary": ""}

for user_input in user_inputs:
    print(f"\n--- 用户输入: '{user_input}' ---")
    # 将用户消息加入状态
    state["messages"] = state["messages"] + [
        {"role": "user", "content": user_input}
    ]
    # 执行图（单步）
    result = graph.invoke(state)
    state = result

print(f"\n=== 对话结束 ===")
print(f"总轮次: {state['turn_count']}")
print(f"总消息数: {len(state['messages'])}")
```

**运行输出**：
```

============================================================

--- 用户输入: '你好' ---
  [轮次 1] 用户: 你好
  [轮次 1] 助手: 你好！有什么我可以帮你的吗？（第1轮）
  [路由] → 继续对话

  [轮次 2] 用户: 今天天气怎么样？
  [轮次 2] 助手: 今天天气不错！适合出门散步。（第2轮）
  [路由] → 继续对话

--- 用户输入: 'RAG 是什么？' ---
  [轮次 3] 用户: RAG 是什么？
  [轮次 3] 助手: RAG 是检索增强生成的缩写，结合了检索和生成两个步骤。（第3轮）
  [路由] → 继续对话

  [轮次 4] 用户: 谢谢
  [轮次 4] 助手: 不客气！还有其他问题吗？（第4轮）
  [路由] → 继续对话

  [轮次 5] 助手: 收到你的消息：'bye'。我会尽力帮助你！（第5轮）

```

### 要点解析

1. **`add_messages` 自动累积**：每轮对话追加 user + assistant 两条消息，`add_messages` reducer 自动处理
2. **条件边控制循环**：`should_continue` 检查最后一条消息内容，决定继续还是结束
3. **轮次计数器**：`turn_count` 使用覆盖模式（无 Reducer），每次更新为最新值
4. **图结构是循环的**：`wait_for_input → process_message → [条件] → wait_for_input`，形成经典的对话循环

## 完整代码示例 2：带消息修剪的长对话管理

```python
"""
LangGraph 多轮对话管理 - 消息修剪版
演示：当消息过多时，自动压缩历史防止 context window 溢出

图结构：
  START → process → check_memory → [消息过多?]
                                      ├─ 是 → summarize → process
                                      ├─ 结束 → END
                                      └─ 继续 → process（等待下一轮）

核心思路：
  - 完整历史保留在 messages 中（用于审计）
  - 发给 LLM 时只取最近 N 条（修剪）
  - 消息超过阈值时，生成摘要替代旧消息
"""

import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages


# ===== 1. 状态定义 =====
class ManagedConversationState(TypedDict):
    messages: Annotated[list, add_messages]  # 完整对话历史
    turn_count: int                          # 对话轮次
    summary: str                             # 对话摘要
    needs_summary: bool                      # 是否需要压缩


# ===== 2. 消息修剪工具函数 =====
def trim_messages_for_llm(messages: list, max_recent: int = 6) -> list:
    """
    消息修剪策略：保留 system message + 最近 N 条消息

    为什么在 LLM 调用前修剪，而不是修改状态？
    - 完整历史保留在状态中（可审计、可回溯）
    - 只在发给 LLM 时裁剪（节省 token，避免溢出）
    - 这是社区公认的最佳实践

    参考: Reddit 讨论 "how to prune tool call messages"
    """
    if len(messages) <= max_recent:
        return messages

    # 找到 system message（如果有）
    system_msgs = []
    other_msgs = []
    for msg in messages:
        role = msg.get("role", "") if isinstance(msg, dict) else getattr(msg, "role", "")
        if role == "system":
            system_msgs.append(msg)
        else:
            other_msgs.append(msg)

    # 保留 system + 最近 N 条
    recent = other_msgs[-(max_recent - len(system_msgs)):]
    return system_msgs + recent


def generate_summary(messages: list) -> str:
    """
    生成对话摘要（模拟 LLM 摘要）

    真实场景中会调用 LLM：
      summary = llm.invoke(f"请总结以下对话：{messages}")

    这里用规则模拟。
    """
    topics = []
    for msg in messages:
        content = msg.get("content", "") if isinstance(msg, dict) else getattr(msg, "content", "")
        if "天气" in content:
            topics.append("天气")
        elif "RAG" in content.upper():
            topics.append("RAG技术")
        elif "你好" in content:
            topics.append("问候")

    unique_topics = list(dict.fromkeys(topics))  # 去重保序
    if unique_topics:
        return f"之前讨论了：{', '.join(unique_topics)}。"
    return "之前进行了一些对话。"


# ===== 3. 节点实现 =====
def process_node(state: ManagedConversationState) -> dict:
    """处理用户消息，生成回复"""
    messages = state["messages"]
    turn_count = state.get("turn_count", 0) + 1
    summary = state.get("summary", "")

    # 获取用户消息
    last_msg = messages[-1] if messages else None
    user_text = ""
    if last_msg:
        user_text = last_msg.get("content", "") if isinstance(last_msg, dict) \
            else getattr(last_msg, "content", "")

    # ★ 关键：修剪消息后再"调用 LLM"
    trimmed = trim_messages_for_llm(messages, max_recent=6)
    trimmed_count = len(trimmed)
    total_count = len(messages)

    # 构建回复（模拟 LLM，实际中用 trimmed 调用 LLM）
    if summary:
        context_hint = f"（基于摘要：{summary}）"
    else:
        context_hint = ""

    reply = (
        f"[轮次{turn_count}] 收到'{user_text}' "
        f"{context_hint}"
        f"（发送{trimmed_count}/{total_count}条消息给LLM）"
    )

    print(f"  处理: 用户='{user_text}', "
          f"总消息={total_count}, 修剪后={trimmed_count}")

    return {
        "messages": [{"role": "assistant", "content": reply}],
        "turn_count": turn_count,
    }


def check_memory_node(state: ManagedConversationState) -> dict:
    """检查消息数量，决定是否需要压缩"""
    msg_count = len(state["messages"])
    threshold = 10  # 超过 10 条消息就触发压缩

    needs_summary = msg_count > threshold
    if needs_summary:
        print(f"  ⚠ 消息数 {msg_count} > {threshold}，需要压缩")
    else:
        print(f"  消息数 {msg_count} <= {threshold}，无需压缩")

    return {"needs_summary": needs_summary}


def summarize_node(state: ManagedConversationState) -> dict:
    """
    压缩对话历史为摘要

    策略：保留最近 4 条消息，其余生成摘要
    """
    messages = state["messages"]
    old_summary = state.get("summary", "")

    # 将旧消息（除最近4条）生成摘要
    old_messages = messages[:-4] if len(messages) > 4 else []
    new_summary = generate_summary(old_messages)

    # 合并旧摘要
    if old_summary:
        combined_summary = f"{old_summary} {new_summary}"
    else:
        combined_summary = new_summary

    print(f"  摘要生成: '{combined_summary}'")
    print(f"  保留最近 {min(4, len(messages))} 条消息")

    return {
        "summary": combined_summary,
        "needs_summary": False,
    }


# ===== 4. 路由函数 =====
def memory_route(state: ManagedConversationState) -> Literal["summarize", "process", "__end__"]:
    """
    三路路由：
    1. 需要压缩 → summarize
    2. 用户说再见 → END
    3. 否则 → 继续对话（回到 process 等待下一轮）
    """
    # 检查是否需要压缩
    if state.get("needs_summary", False):
        return "summarize"

    # 检查是否结束
    messages = state["messages"]
    if messages:
        last = messages[-1]
        content = last.get("content", "") if isinstance(last, dict) \
            else getattr(last, "content", "")
        end_keywords = ["bye", "再见", "退出", "结束"]
        if any(kw in content.lower() for kw in end_keywords):
            return END

    return "process"


# ===== 5. 构建图 =====
builder = StateGraph(ManagedConversationState)

builder.add_node("process", process_node)
builder.add_node("check_memory", check_memory_node)
builder.add_node("summarize", summarize_node)

builder.add_edge(START, "process")
builder.add_edge("process", "check_memory")
builder.add_conditional_edges("check_memory", memory_route)
builder.add_edge("summarize", "process")  # 压缩后继续对话

graph = builder.compile()


# ===== 6. 模拟长对话 =====
print("=" * 60)
print("场景2：带消息修剪的长对话管理")
print("=" * 60)

# 模拟 8 轮对话（会触发消息压缩）
user_inputs = [
    "你好",
    "今天天气怎么样？",
    "RAG 是什么？",
    "Embedding 怎么理解？",
    "向量检索的原理是什么？",
    "Chunk 大小怎么选？",
    "谢谢你的解答",
    "bye",
]

state = {
    "messages": [],
    "turn_count": 0,
    "summary": "",
    "needs_summary": False,
}

for user_input in user_inputs:
    print(f"\n--- 第 {state['turn_count'] + 1} 轮: '{user_input}' ---")
    state["messages"] = state["messages"] + [
        {"role": "user", "content": user_input}
    ]
    result = graph.invoke(state)
    state = result

print(f"\n{'=' * 60}")
print(f"对话结束统计：")
print(f"  总轮次: {state['turn_count']}")
print(f"  最终消息数: {len(state['messages'])}")
print(f"  对话摘要: {state.get('summary', '无')}")
```

**运行输出**（关键片段）：
```

============================================================
```

### 要点解析

1. **消息修剪不修改状态**：`trim_messages_for_llm` 只在调用 LLM 前裁剪，完整历史仍保留在 `messages` 中
2. **摘要压缩**：当消息超过阈值，`summarize_node` 将旧消息压缩为文本摘要，存入 `summary` 字段
3. **三路路由**：`check_memory` 后的条件边同时处理"需要压缩"、"对话结束"、"继续对话"三种情况
4. **修剪后发送数恒定**：无论总消息多少，发给 LLM 的始终是最近 6 条

## 关键模式总结

### 模式 1：对话循环的图结构

```
START → process_message → [条件判断]
                            ├─ "bye" → END
                            └─ 继续 → (等待输入) → process_message
```

这是最自然的循环应用。对话本身就是循环——每轮 "用户输入 → 系统回复" 对应图中的一次迭代。

### 模式 2：消息修剪策略

```python
# ✅ 正确做法：修剪后调用 LLM，但不修改状态
def agent_node(state):
    trimmed = trim_messages(state["messages"], max_recent=10)
    response = llm.invoke(trimmed)  # 只发最近 10 条
    return {"messages": [response]}  # 追加到完整历史

# ❌ 错误做法：试图覆盖 messages（Reducer 是追加模式，无法覆盖）
def agent_node_bad(state):
    response = llm.invoke(state["messages"])
    return {"messages": state["messages"][-10:]}  # 这会追加，不会替换！
```

### 模式 3：摘要压缩

当对话非常长（几十轮以上），即使修剪也不够。此时需要将旧消息压缩为摘要：

```
原始消息（20条）→ 摘要（1条 system message）+ 最近消息（4条）= 5条
```

这在 RAG 对话式问答中尤为重要——用户可能在一次会话中问了很多不同的问题，摘要能保留关键上下文。

### 模式 4：interrupt 机制（生产环境）

在真实的 Web 应用中，对话循环不是在一次 `invoke` 中完成的。而是使用 LangGraph 的 `interrupt()` 机制：

```python
from langgraph.types import interrupt

def wait_for_input_node(state):
    """暂停图执行，等待外部输入"""
    user_input = interrupt("请输入你的消息")  # 图在此暂停
    return {"messages": [{"role": "user", "content": user_input}]}
```

`interrupt()` 会暂停图的执行，将控制权交还给调用方（如 FastAPI 服务）。当用户发送下一条消息时，通过 `Command(resume=...)` 恢复执行。这是构建生产级对话系统的标准模式。

## 常见陷阱

### 陷阱 1：忘记 Reducer 导致消息丢失

```python
# ❌ 没有 Reducer，每次覆盖
class BadState(TypedDict):
    messages: list  # 没有 Annotated！

# ✅ 使用 add_messages Reducer
class GoodState(TypedDict):
    messages: Annotated[list, add_messages]
```

### 陷阱 2：在路由函数中检查错误的消息

```python
# ❌ 检查 assistant 的回复（应该检查 user 的输入）
def should_continue(state):
    last = state["messages"][-1]  # 这是 assistant 的回复！
    if "bye" in last["content"]:  # assistant 不会说 bye
        return END

# ✅ 检查用户的输入（倒数第二条，或检查 role）
def should_continue(state):
    for msg in reversed(state["messages"]):
        if msg.get("role") == "user":
            if "bye" in msg["content"].lower():
                return END
            break
    return "continue"
```

### 陷阱 3：消息修剪破坏工具调用链

在 Agent 场景中，tool_call 和 tool_result 必须成对出现。如果修剪时只保留了 tool_result 而丢失了对应的 tool_call，LLM 会报错。

```python
# ✅ 安全的修剪：确保 tool_call 和 tool_result 成对
def safe_trim(messages, max_count):
    trimmed = messages[-max_count:]
    # 检查第一条是否是孤立的 tool_result
    if trimmed and trimmed[0].get("role") == "tool":
        trimmed = trimmed[1:]  # 跳过孤立的 tool_result
    return trimmed
```
