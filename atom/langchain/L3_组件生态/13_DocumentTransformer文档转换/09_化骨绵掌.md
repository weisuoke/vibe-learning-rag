### 卡片1：什么是 DocumentTransformer

**一句话：** DocumentTransformer 是 LangChain 中对文档序列进行变换的统一抽象——输入一组文档，输出一组文档，中间可以做任何事。

**举例：**

想象一条食品加工流水线：

```
原料（生菜、肉、面包）
  → 清洗站（去泥、去杂质）        ← Html2TextTransformer
  → 切割站（切片、切丁）          ← RecursiveCharacterTextSplitter
  → 质检站（去掉坏的）            ← EmbeddingsRedundantFilter
  → 摆盘站（调整摆放顺序）        ← LongContextReorder
  → 成品（汉堡）
```

每个"站"就是一个 DocumentTransformer。它们的共同特点是：接收食材，输出食材，但食材的状态发生了变化。

**在 RAG 中的应用：** RAG 系统的文档处理管道就是由多个 DocumentTransformer 串联而成的。原始文档经过清洗、分块、过滤、重排序后，才能被高效检索和利用。没有这条管道，LLM 拿到的就是"带泥的生菜"——充满噪音的原始文档。

### 卡片2：BaseDocumentTransformer 接口

**一句话：** BaseDocumentTransformer 只定义了一个核心方法 `transform_documents`，接收 `Sequence[Document]`，返回 `Sequence[Document]`——极简到不能再简。

**举例：**

```python
from abc import ABC, abstractmethod
from typing import Any, Sequence
from langchain_core.documents import Document

class BaseDocumentTransformer(ABC):
    """文档转换器的抽象基类"""

    @abstractmethod
    def transform_documents(
        self, documents: Sequence[Document], **kwargs: Any
    ) -> Sequence[Document]:
        """转换文档列表"""

    async def atransform_documents(
        self, documents: Sequence[Document], **kwargs: Any
    ) -> Sequence[Document]:
        """异步版本，默认用线程池包装同步方法"""
        return await run_in_executor(
            None, self.transform_documents, documents, **kwargs
        )
```

关键设计决策：
- 输入输出类型一致（`Sequence[Document] → Sequence[Document]`），所以转换器可以任意串联
- 异步方法有默认实现，子类不必重写
- 纯函数式：不修改原始文档，返回新序列

**在 RAG 中的应用：** 这个统一接口是 DocumentCompressorPipeline 能够工作的基础。正因为所有转换器"说同一种语言"，才能像乐高积木一样自由组合。

### 卡片3：HTML 清洗转换器

**一句话：** BeautifulSoupTransformer 和 Html2TextTransformer 是 RAG 系统处理网页数据的第一道关卡——把 HTML 标签、导航栏、广告脚本统统去掉，只留纯净文本。

**举例：**

```python
from langchain_community.document_transformers import (
    BeautifulSoupTransformer,
    Html2TextTransformer,
)
from langchain_core.documents import Document

# 原始 HTML 文档
html_doc = Document(page_content="""
<html>
<head><title>RAG 教程</title></head>
<body>
  <nav>首页 | 关于我们 | 联系方式</nav>
  <article>
    <h1>什么是 RAG？</h1>
    <p>RAG 是检索增强生成的缩写...</p>
  </article>
  <footer>版权所有 2026</footer>
</body>
</html>
""")

# 方式 1：BeautifulSoupTransformer（精细控制）
bs_transformer = BeautifulSoupTransformer()
clean_docs = bs_transformer.transform_documents(
    [html_doc],
    tags_to_extract=["article"],  # 只提取 article 标签内容
)

# 方式 2：Html2TextTransformer（简单快速）
h2t_transformer = Html2TextTransformer()
clean_docs = h2t_transformer.transform_documents([html_doc])
```

两者的区别：
- `BeautifulSoupTransformer`：可以指定提取哪些标签，精细控制
- `Html2TextTransformer`：一键转换，将 HTML 转为 Markdown 格式的纯文本

**在 RAG 中的应用：** 当你用 WebBaseLoader 爬取网页构建知识库时，原始 HTML 中的导航栏、页脚、广告会严重干扰 embedding 质量。先用 HTML 清洗转换器处理，检索准确率通常能提升 15-25%。

### 卡片4：冗余文档过滤

**一句话：** EmbeddingsRedundantFilter 通过比较文档的 embedding 向量相似度，自动去除语义重复的文档——让每一份送入 LLM 的文档都"物有所值"。

**举例：**

```python
from langchain_community.document_transformers import (
    EmbeddingsRedundantFilter,
)
from langchain_openai import OpenAIEmbeddings

# 创建冗余过滤器
filter_embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
redundant_filter = EmbeddingsRedundantFilter(
    embeddings=filter_embeddings,
    similarity_threshold=0.95,  # 相似度 > 95% 视为冗余
)

# 假设检索返回了 5 个文档，其中 2 个高度相似
docs = [doc1, doc2, doc3, doc4_similar_to_doc1, doc5_similar_to_doc2]

# 过滤后只保留 3 个不重复的文档
filtered_docs = redundant_filter.transform_documents(docs)
# 结果：[doc1, doc2, doc3]
```

为什么需要去重？因为向量检索的 top-k 结果经常包含语义高度相似的文档（比如同一篇文章的不同段落说了类似的话）。这些重复文档会浪费宝贵的 context window 空间。

**在 RAG 中的应用：** 假设你的 context window 只能放 5 个文档，如果其中 3 个说的是同一件事，LLM 实际上只获得了 3 个文档的信息量。去重后，5 个文档 = 5 份不同的信息，信息密度提升 30-40%。

### 卡片5：文档聚类过滤

**一句话：** EmbeddingsClusteringFilter 将文档按语义聚类，从每个簇中选出最具代表性的文档——既保证了多样性，又控制了数量。

**举例：**

```python
from langchain_community.document_transformers import (
    EmbeddingsClusteringFilter,
)
from langchain_openai import OpenAIEmbeddings

filter_embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 创建聚类过滤器
clustering_filter = EmbeddingsClusteringFilter(
    embeddings=filter_embeddings,
    num_clusters=10,    # 将文档聚成 10 个语义簇
    num_closest=1,      # 每个簇选 1 个最接近簇心的文档
    sorted=True,        # 按原始检索分数排序
)

# 假设检索返回了 50 个文档
# 聚类后只保留 10 个代表性文档
selected_docs = clustering_filter.transform_documents(retrieved_docs)
```

聚类过滤 vs 冗余过滤的区别：
- **冗余过滤**：去掉"几乎一样"的文档（相似度阈值）
- **聚类过滤**：从"不同话题"中各选一个代表（K-Means 聚类）

**在 RAG 中的应用：** 当用户问"RAG 系统有哪些优化方向？"时，检索结果可能集中在某一个方向（比如分块优化）。聚类过滤能确保结果覆盖多个方向（分块、检索、生成、评估），让 LLM 的回答更全面。

### 卡片6：元数据自动标注

**一句话：** OpenAIMetadataTagger 利用 LLM 的理解能力，自动为文档标注结构化元数据（类别、难度、主题等），让后续检索可以精确过滤。

**举例：**

```python
from langchain_community.document_transformers.openai_functions import (
    create_metadata_tagger,
)
from langchain_openai import ChatOpenAI
from langchain_core.documents import Document

# 定义元数据 schema
schema = {
    "properties": {
        "category": {
            "type": "string",
            "enum": ["教程", "API文档", "最佳实践", "故障排查"],
            "description": "文档类别",
        },
        "language": {
            "type": "string",
            "enum": ["Python", "JavaScript", "Go", "通用"],
            "description": "涉及的编程语言",
        },
    },
    "required": ["category", "language"],
}

# 创建标注器
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
tagger = create_metadata_tagger(metadata_schema=schema, llm=llm)

# 自动标注
doc = Document(page_content="使用 Python 的 RecursiveCharacterTextSplitter 进行文本分块...")
tagged_docs = tagger.transform_documents([doc])

# 结果：doc.metadata = {"category": "教程", "language": "Python"}
```

**在 RAG 中的应用：** 纯向量检索只能捕捉语义相似度，但用户可能需要"只看 Python 教程"或"只看故障排查文档"。有了元数据标注，检索时可以先按元数据过滤，再做向量检索，精确率提升 20-30%。

### 卡片7：上下文重排序

**一句话：** LongContextReorder 针对 "Lost in the Middle" 问题设计——将最相关的文档放在上下文的开头和结尾，因为 LLM 对中间位置的信息关注度最低。

**举例：**

```python
from langchain_community.document_transformers import LongContextReorder

reordering = LongContextReorder()

# 假设文档按相关性排序：[最相关, 次相关, 第三, 第四, 最不相关]
# 重排序后：[最相关, 第三, 最不相关, 第四, 次相关]
# 最相关的在开头，次相关的在结尾，不太相关的在中间
reordered_docs = reordering.transform_documents(docs)
```

"Lost in the Middle" 是什么？

2023 年 Stanford 的研究发现，当 LLM 处理长上下文时，对信息的关注度呈 U 型分布：

```
关注度
  高 ┃ ██                              ██
     ┃ ████                          ████
     ┃ ██████                      ██████
  低 ┃ ████████████████████████████████
     ┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
       开头        中间          结尾
```

开头和结尾的信息被充分利用，中间的信息容易被"遗忘"。

**在 RAG 中的应用：** 当你向 LLM 提供 10 个检索文档时，如果最相关的文档恰好排在第 5 位（中间），LLM 可能会忽略它。LongContextReorder 通过调整顺序，确保关键信息出现在 LLM 最"关注"的位置，准确率提升 10-15%。

### 卡片8：Pipeline 组合模式

**一句话：** DocumentCompressorPipeline 将多个转换器串联成管道，数据像流水一样依次经过每个转换器——这是 LangChain 中实现复杂文档处理逻辑的标准模式。

**举例：**

```python
from langchain.retrievers.document_compressors import (
    DocumentCompressorPipeline,
)
from langchain.retrievers import ContextualCompressionRetriever
from langchain_community.document_transformers import (
    EmbeddingsRedundantFilter,
    EmbeddingsClusteringFilter,
    LongContextReorder,
)
from langchain_openai import OpenAIEmbeddings

filter_embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 构建三步优化管道
pipeline = DocumentCompressorPipeline(
    transformers=[
        # 第一步：去除冗余文档
        EmbeddingsRedundantFilter(embeddings=filter_embeddings),
        # 第二步：聚类选择代表性文档
        EmbeddingsClusteringFilter(
            embeddings=filter_embeddings,
            num_clusters=5,
            num_closest=1,
        ),
        # 第三步：重排序优化 LLM 关注度
        LongContextReorder(),
    ]
)

# 集成到检索器
compression_retriever = ContextualCompressionRetriever(
    base_compressor=pipeline,
    base_retriever=vector_retriever,
)

# 一行代码完成：检索 → 去重 → 聚类 → 重排序
results = compression_retriever.invoke("什么是 RAG？")
```

Pipeline 的核心优势：
- **可组合**：任意添加、删除、调整转换器顺序
- **可复用**：同一个 Pipeline 可以用于不同的检索器
- **可测试**：每个转换器可以独立测试

**在 RAG 中的应用：** 生产级 RAG 系统几乎都会使用 Pipeline 模式。不同的业务场景可以配置不同的 Pipeline——知识库问答侧重去重和重排序，多文档摘要侧重聚类和多样性。

### 卡片9：自定义转换器

**一句话：** 继承 BaseDocumentTransformer 并实现 `transform_documents` 方法，就能创建自己的转换器——比如敏感信息脱敏、语言检测、质量评分等。

**举例：**

```python
from langchain_core.documents import Document
from langchain_core.documents.transformers import BaseDocumentTransformer
from typing import Any, Sequence
import re

class SensitiveInfoMasker(BaseDocumentTransformer):
    """敏感信息脱敏转换器

    自动将文档中的手机号、邮箱、身份证号替换为占位符
    """

    def transform_documents(
        self, documents: Sequence[Document], **kwargs: Any
    ) -> Sequence[Document]:
        masked_docs = []
        for doc in documents:
            masked_content = doc.page_content
            # 脱敏手机号
            masked_content = re.sub(
                r'1[3-9]\d{9}',
                '[PHONE]',
                masked_content,
            )
            # 脱敏邮箱
            masked_content = re.sub(
                r'[\w.-]+@[\w.-]+\.\w+',
                '[EMAIL]',
                masked_content,
            )
            masked_docs.append(
                Document(
                    page_content=masked_content,
                    metadata={**doc.metadata, "masked": True},
                )
            )
        return masked_docs

# 使用
masker = SensitiveInfoMasker()
docs = [Document(page_content="联系我：13800138000，邮箱 test@example.com")]
masked = masker.transform_documents(docs)
print(masked[0].page_content)
# 输出：联系我：[PHONE]，邮箱 [EMAIL]
```

自定义转换器的关键原则：
- 不修改原始文档，返回新的 Document 对象
- 保留并可以扩展 metadata
- 实现 `transform_documents` 即可，异步方法有默认实现

**在 RAG 中的应用：** 企业级 RAG 系统经常需要自定义转换器来满足合规要求（如 GDPR 数据脱敏）、业务需求（如添加来源标注）或质量控制（如过滤低质量文档）。

### 卡片10：RAG 中的最佳实践

**一句话：** 在 RAG 系统中使用 DocumentTransformer 的核心原则是"先清洗、再分块、后优化"，转换器的选择和排序直接决定检索质量。

**举例：**

**推荐的 RAG 文档处理管道：**

```
阶段 1：文档清洗（检索前）
┌─────────────────────────────────────────┐
│ 原始文档                                 │
│   ↓                                      │
│ Html2TextTransformer（去 HTML 标签）      │
│   ↓                                      │
│ SensitiveInfoMasker（敏感信息脱敏）       │
│   ↓                                      │
│ 干净文档                                 │
└─────────────────────────────────────────┘

阶段 2：文档分块（检索前）
┌─────────────────────────────────────────┐
│ 干净文档                                 │
│   ↓                                      │
│ RecursiveCharacterTextSplitter           │
│   (chunk_size=500, chunk_overlap=50)     │
│   ↓                                      │
│ 文档块 → 存入向量数据库                   │
└─────────────────────────────────────────┘

阶段 3：检索后优化
┌─────────────────────────────────────────┐
│ 向量检索返回 top-20 文档                  │
│   ↓                                      │
│ EmbeddingsRedundantFilter（去重）         │
│   ↓                                      │
│ EmbeddingsClusteringFilter（聚类选择）    │
│   ↓                                      │
│ LongContextReorder（重排序）              │
│   ↓                                      │
│ 优化后的 5-10 个文档 → 送入 LLM           │
└─────────────────────────────────────────┘
```

**转换器选择决策表：**

| 场景 | 推荐转换器 | 优先级 |
|------|-----------|--------|
| 网页数据源 | Html2TextTransformer | 必须 |
| 文档量大、有重复 | EmbeddingsRedundantFilter | 高 |
| 需要结果多样性 | EmbeddingsClusteringFilter | 中 |
| 上下文较长（>5个文档） | LongContextReorder | 高 |
| 需要精确过滤 | OpenAIMetadataTagger | 中 |
| 合规要求 | 自定义脱敏转换器 | 视需求 |

**排序原则：**
1. 清洗类转换器放最前面（垃圾进垃圾出）
2. 过滤类转换器放中间（减少后续处理量）
3. 重排序类转换器放最后（优化最终输出顺序）

**在 RAG 中的应用：** 不同业务场景需要不同的 Pipeline 配置。知识库问答侧重去重和重排序；多文档摘要侧重聚类和多样性；客服系统侧重元数据过滤和精确匹配。没有"万能配置"，需要根据实际效果调优。
