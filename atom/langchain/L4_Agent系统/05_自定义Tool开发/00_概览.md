# 自定义Tool开发 - 概览

> 为 AI Agent 赋予能力的核心机制

---

## 知识点概述

在 LangChain 的 Agent 系统中，Tool（工具）是 Agent 与外部世界交互的桥梁。自定义 Tool 开发是构建实用 AI Agent 的核心技能，它让 Agent 能够调用 API、查询数据库、执行计算、访问文件系统等。

本知识点深入讲解 LangChain 中三种主流的工具创建方式，帮助你根据不同场景选择最合适的方法，并掌握工具开发的最佳实践。

---

## 为什么这个知识点重要？

### 对 AI Agent 开发的价值

1. **扩展 Agent 能力边界**
   - LLM 本身只能生成文本，无法直接操作外部系统
   - 通过自定义工具，Agent 可以调用任何 API、执行任何代码
   - 工具是 Agent 从"聊天机器人"进化为"行动者"的关键

2. **实现业务逻辑集成**
   - 将企业内部系统（CRM、ERP、数据库）暴露给 Agent
   - 封装复杂的业务规则和验证逻辑
   - 提供统一的接口供 Agent 调用

3. **控制 Agent 行为**
   - 通过工具描述引导 Agent 何时调用
   - 通过参数验证确保调用安全性
   - 通过错误处理提升系统鲁棒性

4. **提升开发效率**
   - 复用现有代码和 API
   - 快速原型开发（@tool 装饰器）
   - 渐进式优化（从简单到复杂）

### 实际应用场景

- **客服机器人**：查询订单、修改地址、退款处理
- **数据分析 Agent**：执行 SQL 查询、生成图表、导出报告
- **自动化运维**：重启服务、检查日志、部署代码
- **内容生成**：调用图像生成 API、视频剪辑工具
- **知识库问答**：检索文档、搜索网页、查询向量数据库

## 三种工具创建方式对比

LangChain 提供了三种创建自定义工具的方式，它们在易用性、灵活性和控制力上各有侧重。

### 1. @tool 装饰器（推荐方式）

**特点**：
- 最简洁的语法，一行代码即可创建工具
- 自动推断参数 schema（基于类型提示）
- 支持 Google-style docstring 解析
- 适合 80% 的工具开发场景

**适用场景**：
- 简单的函数包装
- 快速原型开发
- 无状态的工具
- 标准的同步/异步函数

**示例**：
```python
from langchain.tools import tool

@tool
def search_database(query: str, limit: int = 10) -> str:
    """Search the product database.

    Args:
        query: Search keywords
        limit: Maximum number of results
    """
    # 实现逻辑
    return f"Found {limit} results for '{query}'"
```

**来源**：[源码分析 convert.py | @tool 装饰器实现]

---

### 2. StructuredTool.from_function()（灵活方式）

**特点**：
- 显式控制工具名称、描述和 schema
- 支持包装现有函数或 API 客户端
- 可以分离函数实现和工具定义
- 适合需要精细控制的场景

**适用场景**：
- 包装第三方 API 客户端
- 需要自定义工具元数据
- 复杂的参数验证需求
- 动态生成工具

**示例**：
```python
from langchain.tools import StructuredTool
from pydantic import BaseModel, Field

class EmailInput(BaseModel):
    to: str = Field(description="Recipient email")
    subject: str = Field(description="Email subject")
    body: str = Field(description="Email body")

email_tool = StructuredTool.from_function(
    name="send_email",
    description="Send email via SMTP",
    func=email_client.send,
    args_schema=EmailInput,
)
```

**来源**：[Context7 文档 | StructuredTool 用法]

### 3. BaseTool 类继承（最大控制）

**特点**：
- 完全控制工具的生命周期
- 支持复杂的状态管理
- 可以重写更多方法（初始化、清理、错误处理）
- 适合企业级复杂工具

**适用场景**：
- 需要维护连接池或会话状态
- 复杂的初始化和清理逻辑
- 需要重写多个生命周期方法
- 工具需要访问 Agent 状态（注意：LangGraph 中有限制）

**示例**：
```python
from langchain.tools import BaseTool
from pydantic import BaseModel, Field

class DatabaseInput(BaseModel):
    query: str = Field(description="SQL query")

class DatabaseTool(BaseTool):
    name = "database_query"
    description = "Execute SQL queries"
    args_schema = DatabaseInput

    # 工具状态
    connection_pool: Any = None

    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self.connection_pool = create_pool()

    def _run(self, query: str, config: RunnableConfig) -> str:
        """Synchronous execution."""
        with self.connection_pool.get_connection() as conn:
            result = conn.execute(query)
            return str(result.fetchall())

    async def _arun(self, query: str, config: RunnableConfig) -> str:
        """Asynchronous execution."""
        async with self.connection_pool.get_async_connection() as conn:
            result = await conn.execute(query)
            return str(await result.fetchall())
```

## 选择决策树

```
需要创建自定义工具？
    ↓
是否需要维护状态（连接池、会话）？
    ├─ 是 → 使用 BaseTool 继承
    └─ 否 → 继续
        ↓
是否需要包装现有 API 客户端？
    ├─ 是 → 使用 StructuredTool.from_function()
    └─ 否 → 继续
        ↓
是否需要自定义工具名称或复杂 schema？
    ├─ 是 → 使用 @tool(args_schema=...)
    └─ 否 → 使用 @tool 装饰器（最简单）
```

## 核心概念预览

本知识点将深入讲解以下 8 个核心概念：

### 1. @tool 装饰器方法
- 三种使用模式（无参、带参、函数调用）
- 自动 schema 推断机制
- Google-style docstring 解析
- 与 StructuredTool 的关系

### 2. StructuredTool 创建
- `from_function()` 类方法
- 显式 schema 定义
- 同步和异步函数支持
- response_format 配置

### 3. BaseTool 类继承
- 抽象基类接口
- 必须实现的方法（`_run`）
- 可选实现的方法（`_arun`）
- 状态管理和初始化

### 4. Schema 定义与验证
- Pydantic BaseModel 定义
- Field 描述和约束
- Literal 类型限制
- 自动推断 vs 显式定义

### 5. 参数注入机制
- InjectedToolArg 标记
- InjectedToolCallId 注入
- RunnableConfig 注入
- 过滤机制（FILTERED_ARGS）

### 6. 异步工具支持
- coroutine 参数（StructuredTool）
- `_arun` 方法（BaseTool）
- `run_in_executor` 自动回退
- 性能优化建议

### 7. 响应格式控制
- "content" 模式（默认）
- "content_and_artifact" 模式
- ToolMessage 结构
- 使用场景对比

### 8. 错误处理模式
- ToolException 特殊异常
- handle_tool_error 配置
- 错误传播 vs 错误捕获
- Agent 循环中的错误处理

## 2025-2026 最新趋势

### 技术演进
1. **@tool 装饰器成为主流**
   - 更简洁的 API 设计
   - 更好的 IDE 支持
   - 社区推荐的首选方式

2. **LangGraph 集成深化**
   - 工具与状态管理紧密结合
   - 注意：BaseTool 在 LangGraph 中的状态访问限制
   - @tool 装饰器对 LangGraph 支持更好

3. **大规模工具管理**
   - LangGraph BigTool 库（支持数百工具）
   - 工具检索和过滤机制
   - 动态工具加载

4. **本地模型优化**
   - Ollama 等本地模型的工具调用优化
   - 更清晰的工具描述要求
   - 格式兼容性测试

### 常见陷阱（2025-2026）
1. **参数描述丢失**
   - 问题：create_react_agent 中工具参数描述可能丢失
   - 解决：使用 `parse_docstring=True` 参数

2. **状态访问限制**
   - 问题：BaseTool 子类在 LangGraph 中无法访问状态
   - 解决：优先使用 @tool 装饰器

3. **版本兼容性**
   - 问题：LangGraph 1.0.2+ 破坏性变更
   - 解决：固定依赖版本（使用 `==` 而非 `>=`）

4. **ToolRuntime 参数**
   - 问题：@tool 中的 ToolRuntime 参数在 ToolNode 中不被识别
   - 解决：使用 InjectedToolArg 替代

## 快速参考

### 三种方式的代码模板

#### @tool 装饰器
```python
from langchain.tools import tool

@tool
def my_tool(param: str) -> str:
    """Tool description."""
    return f"Result: {param}"
```

#### StructuredTool
```python
from langchain.tools import StructuredTool
from pydantic import BaseModel, Field

class MyInput(BaseModel):
    param: str = Field(description="Parameter description")

my_tool = StructuredTool.from_function(
    name="my_tool",
    description="Tool description",
    func=my_function,
    args_schema=MyInput,
)
```

#### BaseTool
```python
from langchain.tools import BaseTool
from pydantic import BaseModel, Field

class MyInput(BaseModel):
    param: str = Field(description="Parameter description")

class MyTool(BaseTool):
    name = "my_tool"
    description = "Tool description"
    args_schema = MyInput

    def _run(self, param: str, config: RunnableConfig) -> str:
        return f"Result: {param}"
```

## 总结

自定义 Tool 开发是 LangChain Agent 系统的核心技能。通过掌握三种创建方式（@tool、StructuredTool、BaseTool），你可以根据不同场景选择最合适的方法，快速构建功能强大的 AI Agent。

**关键要点**：
- 优先使用 @tool 装饰器（简洁高效）
- 需要显式控制时使用 StructuredTool
- 复杂状态管理时继承 BaseTool
- 注意 2025-2026 的版本兼容性问题
- 充分利用 schema 定义和错误处理机制
