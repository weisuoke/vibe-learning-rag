# 核心概念 1：子图定义与编译

> 子图是通过 StateGraph 定义并编译的独立图，可作为父图的节点使用

---

## 概述

**子图定义与编译是什么？**

子图（Subgraph）是 LangGraph 模块化设计的核心。它本质上就是一个普通的 StateGraph，经过 `compile()` 编译后，通过 `add_node()` 作为另一个图的节点嵌入使用。子图拥有独立的状态定义、节点和边，是构建复杂多代理系统的基础。

**为什么需要理解子图定义与编译？**

- 当工作流逻辑变复杂，需要拆分为可管理的模块
- 当多个工作流共享相同的子流程，需要复用
- 当需要让不同团队独立开发各自的子模块
- 当需要对子模块进行独立测试和调试

---

## 核心概念：StateGraph 的三个 Schema 参数

### 一句话定义

**StateGraph 通过 `state_schema`、`input_schema`、`output_schema` 三个参数分别定义内部状态、输入接口和输出接口，实现子图的封装与隔离。**

### 详细解释

当我们创建一个 StateGraph 时，构造函数接受以下参数：

```python
class StateGraph:
    def __init__(
        self,
        state_schema: type,              # 必填：内部状态定义
        context_schema: type | None = None,
        input_schema: type | None = None,  # 可选：输入接口
        output_schema: type | None = None, # 可选：输出接口
    ):
        self.input_schema = input_schema or state_schema   # 默认等于 state_schema
        self.output_schema = output_schema or state_schema # 默认等于 state_schema
```

**三个 Schema 的关系：**

```
            ┌─────────────────────────────────────────────┐
            │              子图 (Subgraph)                  │
            │                                             │
 input_schema     state_schema（内部状态）    output_schema
 ─────────→ │  foo, bar, baz, internal_data  │ ─────────→
 {foo, bar}  │  （可以比输入输出更复杂）       │  {foo, result}
            │                                             │
            └─────────────────────────────────────────────┘
```

#### 1. state_schema（内部状态）

这是子图**内部运行时**使用的完整状态定义。子图内部的所有节点都可以读写 state_schema 中定义的所有字段。

```python
from typing import TypedDict

class SubgraphInternalState(TypedDict):
    """子图内部状态 - 包含所有工作字段"""
    query: str              # 输入：用户查询
    documents: list[str]    # 中间结果：检索到的文档
    ranked_docs: list[str]  # 中间结果：排序后的文档
    answer: str             # 输出：最终答案
```

#### 2. input_schema（输入接口）

定义子图**接收**的数据结构。当父图调用子图时，只有 input_schema 中定义的字段会被传入。

- **默认值**：如果不指定，等于 state_schema
- **用途**：限制子图暴露的输入接口，隐藏内部细节

```python
class SubgraphInput(TypedDict):
    """子图输入接口 - 只暴露必要的输入"""
    query: str  # 只需要用户查询
```

#### 3. output_schema（输出接口）

定义子图**返回**的数据结构。子图执行完毕后，只有 output_schema 中定义的字段会返回给父图。

- **默认值**：如果不指定，等于 state_schema
- **用途**：限制子图暴露的输出接口，避免内部状态泄露

```python
class SubgraphOutput(TypedDict):
    """子图输出接口 - 只返回最终结果"""
    answer: str  # 只返回答案
```

### 为什么需要区分三个 Schema？

**核心原因：子图内部状态往往比输入输出更复杂。**

想象一个 RAG 检索子图：
- **输入**：只需要一个 `query` 字符串
- **内部**：需要 `query`、`documents`、`embeddings`、`ranked_docs`、`scores` 等多个中间字段
- **输出**：只需要返回 `answer` 和 `sources`

如果不区分三个 schema，父图就会看到子图的所有内部字段，这违背了封装原则。

```
不区分 schema（糟糕）：
  父图看到：query, documents, embeddings, ranked_docs, scores, answer, sources
  → 父图和子图紧耦合，修改子图内部逻辑可能破坏父图

区分 schema（推荐）：
  父图看到：query (输入) → answer, sources (输出)
  → 干净的接口，子图内部可以自由重构
```

### 类比理解

**前端类比**：
- `state_schema` = React 组件的内部 state（useState 管理的所有状态）
- `input_schema` = 组件的 Props 接口（从父组件接收的属性）
- `output_schema` = 组件的回调/事件接口（通过 onChange 等回调返回的数据）

**日常生活类比**：
- `state_schema` = 餐厅厨房里的所有工作流程（洗菜、切菜、烹饪、摆盘）
- `input_schema` = 餐厅菜单上的点餐接口（顾客只需要说"来一份宫保鸡丁"）
- `output_schema` = 端上桌的最终菜品（顾客不需要知道厨房里发生了什么）

## 子图的完整生命周期

### 第一步：定义子图

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END

# ===== 1. 定义 Schema =====

class RAGInternalState(TypedDict):
    """子图内部状态"""
    query: str
    documents: list[str]
    answer: str

class RAGInput(TypedDict):
    """子图输入接口"""
    query: str

class RAGOutput(TypedDict):
    """子图输出接口"""
    answer: str

# ===== 2. 定义节点函数 =====

def retrieve(state: RAGInternalState) -> dict:
    """检索文档"""
    query = state["query"]
    # 模拟检索
    docs = [f"文档关于 {query} 的内容1", f"文档关于 {query} 的内容2"]
    return {"documents": docs}

def generate(state: RAGInternalState) -> dict:
    """基于文档生成答案"""
    docs = state["documents"]
    query = state["query"]
    answer = f"根据 {len(docs)} 篇文档，关于 '{query}' 的答案是：..."
    return {"answer": answer}

# ===== 3. 构建子图 =====

rag_subgraph = StateGraph(
    RAGInternalState,
    input_schema=RAGInput,
    output_schema=RAGOutput,
)

rag_subgraph.add_node("retrieve", retrieve)
rag_subgraph.add_node("generate", generate)

rag_subgraph.add_edge(START, "retrieve")
rag_subgraph.add_edge("retrieve", "generate")
rag_subgraph.add_edge("generate", END)
```

### 第二步：编译子图

```python
# 编译子图 → 得到 CompiledStateGraph
compiled_rag = rag_subgraph.compile()

# 编译后的子图可以独立测试
result = compiled_rag.invoke({"query": "什么是向量检索？"})
print(result)
# 输出: {"answer": "根据 2 篇文档，关于 '什么是向量检索？' 的答案是：..."}
# 注意：只返回 output_schema 中定义的字段（answer），不返回 documents
```

### 第三步：嵌入父图

```python
from typing import Annotated
import operator

class ParentState(TypedDict):
    """父图状态"""
    user_question: str
    final_answer: str

# 需要包装函数来转换状态
def call_rag(state: ParentState) -> dict:
    """调用 RAG 子图"""
    result = compiled_rag.invoke({"query": state["user_question"]})
    return {"final_answer": result["answer"]}

# 构建父图
parent_graph = StateGraph(ParentState)
parent_graph.add_node("rag_pipeline", call_rag)
parent_graph.add_edge(START, "rag_pipeline")
parent_graph.add_edge("rag_pipeline", END)

# 编译并运行
app = parent_graph.compile()
result = app.invoke({"user_question": "什么是 RAG？"})
print(result)
# {"user_question": "什么是 RAG？", "final_answer": "根据 2 篇文档..."}
```

## 三种子图定义模式

### 模式一：最简子图（不指定 input/output_schema）

当子图状态简单，或者父子图状态完全一致时使用。

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class SimpleState(TypedDict):
    message: str
    processed: bool

def process_node(state: SimpleState) -> dict:
    return {
        "message": state["message"].upper(),
        "processed": True
    }

# 不指定 input/output_schema → 默认都等于 state_schema
simple_sub = StateGraph(SimpleState)
simple_sub.add_node("process", process_node)
simple_sub.add_edge(START, "process")
simple_sub.add_edge("process", END)

compiled_simple = simple_sub.compile()

# 调用时需要传入所有 state_schema 字段
result = compiled_simple.invoke({"message": "hello", "processed": False})
print(result)
# {"message": "HELLO", "processed": True}
```

**适用场景**：简单的工具调用子图、数据转换子图。

### 模式二：带 input/output_schema 的子图

当子图内部有复杂的中间状态，但只想暴露简洁接口时使用。

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

# 内部状态：包含中间处理字段
class AnalysisState(TypedDict):
    text: str                # 输入
    tokens: list[str]        # 中间结果：分词
    sentiment_score: float   # 中间结果：情感分数
    summary: str             # 输出

# 输入接口：只接收文本
class AnalysisInput(TypedDict):
    text: str

# 输出接口：只返回摘要
class AnalysisOutput(TypedDict):
    summary: str

def tokenize(state: AnalysisState) -> dict:
    """分词"""
    tokens = state["text"].split()
    return {"tokens": tokens}

def analyze_sentiment(state: AnalysisState) -> dict:
    """情感分析"""
    # 简单模拟
    score = 0.8 if "good" in state["tokens"] else 0.3
    return {"sentiment_score": score}

def summarize(state: AnalysisState) -> dict:
    """生成摘要"""
    score = state.get("sentiment_score", 0.5)
    sentiment = "正面" if score > 0.5 else "负面"
    summary = f"文本包含 {len(state['tokens'])} 个词，情感倾向 {sentiment}（{score:.1f}）"
    return {"summary": summary}

# 带完整 schema 定义的子图
analysis_sub = StateGraph(
    AnalysisState,
    input_schema=AnalysisInput,
    output_schema=AnalysisOutput,
)

analysis_sub.add_node("tokenize", tokenize)
analysis_sub.add_node("sentiment", analyze_sentiment)
analysis_sub.add_node("summarize", summarize)

analysis_sub.add_edge(START, "tokenize")
analysis_sub.add_edge("tokenize", "sentiment")
analysis_sub.add_edge("sentiment", "summarize")
analysis_sub.add_edge("summarize", END)

compiled_analysis = analysis_sub.compile()

# 调用：只传 input_schema 中的字段
result = compiled_analysis.invoke({"text": "this is a good example"})
print(result)
# {"summary": "文本包含 5 个词，情感倾向 正面（0.8）"}
# 注意：tokens 和 sentiment_score 不会返回
```

**适用场景**：RAG 管道、多步骤处理链、需要封装的复杂逻辑。

### 模式三：编译后直接添加到父图（共享状态键方式）

当父子图有共享 key 时，可以直接将编译后的子图作为节点添加。

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

# 父图状态
class ParentState(TypedDict):
    foo: str
    result: str

# 子图状态 - foo 与父图共享，internal 是私有的
class ChildState(TypedDict):
    foo: str       # 共享 key：自动从父图传入
    internal: str  # 私有 key：不影响父图

def child_node(state: ChildState) -> dict:
    """子图节点"""
    return {
        "foo": state["foo"] + " -> processed by child",
        "internal": "child internal data"
    }

# 构建并编译子图
child_builder = StateGraph(ChildState)
child_builder.add_node("work", child_node)
child_builder.add_edge(START, "work")
child_builder.add_edge("work", END)
compiled_child = child_builder.compile()

# 直接添加到父图
def parent_entry(state: ParentState) -> dict:
    return {"foo": "start"}

def parent_exit(state: ParentState) -> dict:
    return {"result": f"Final: {state['foo']}"}

parent_builder = StateGraph(ParentState)
parent_builder.add_node("entry", parent_entry)
parent_builder.add_node("child_subgraph", compiled_child)  # 直接添加编译后的子图
parent_builder.add_node("exit", parent_exit)

parent_builder.add_edge(START, "entry")
parent_builder.add_edge("entry", "child_subgraph")
parent_builder.add_edge("child_subgraph", "exit")
parent_builder.add_edge("exit", END)

app = parent_builder.compile()
result = app.invoke({"foo": "", "result": ""})
print(result)
# {"foo": "start -> processed by child", "result": "Final: start -> processed by child"}
```

**适用场景**：父子图有部分重叠的状态字段。

## 在 LangGraph 中的应用

### 应用一：多代理系统

每个代理作为独立子图，通过父图编排：

```python
# 研究代理子图
research_agent = StateGraph(ResearchState, input_schema=AgentInput, output_schema=AgentOutput)
# ... 添加节点和边 ...
compiled_research = research_agent.compile()

# 写作代理子图
writing_agent = StateGraph(WritingState, input_schema=AgentInput, output_schema=AgentOutput)
# ... 添加节点和边 ...
compiled_writing = writing_agent.compile()

# 编排父图
orchestrator = StateGraph(OrchestratorState)
orchestrator.add_node("research", compiled_research)
orchestrator.add_node("writing", compiled_writing)
```

### 应用二：RAG 管道模块化

将 RAG 流程拆分为独立子图：

```
父图（主流程）
  ├── 子图1：文档检索（query → documents）
  ├── 子图2：重排序（documents → ranked_docs）
  └── 子图3：答案生成（ranked_docs + query → answer）
```

### 应用三：递归嵌套

子图可以包含子子图，支持任意深度的嵌套：

```python
# 第三层：工具调用子图
tool_sub = StateGraph(ToolState)
compiled_tool = tool_sub.compile()

# 第二层：代理子图（包含工具子图）
agent_sub = StateGraph(AgentState)
agent_sub.add_node("use_tool", compiled_tool)
compiled_agent = agent_sub.compile()

# 第一层：编排父图（包含代理子图）
orchestrator = StateGraph(OrchestratorState)
orchestrator.add_node("agent", compiled_agent)
```

## 源码关键细节

### add_node 如何处理子图

当 `add_node()` 接收到一个 `CompiledStateGraph` 时，LangGraph 会自动处理状态映射：

1. 从父图状态中提取子图 `input_schema` 定义的字段
2. 将这些字段作为输入传递给子图
3. 子图执行完毕后，从子图输出中提取 `output_schema` 定义的字段
4. 将这些字段合并回父图状态

```python
# 源码中的关键逻辑（简化版）
# libs/langgraph/langgraph/graph/state.py

def add_node(self, node_name, node, *, input_schema=None):
    # 如果 node 是 CompiledStateGraph，自动提取 input_schema
    if isinstance(node, CompiledStateGraph):
        input_schema = node.input_schema
    # ...
```

### compile() 做了什么

编译过程将 StateGraph 转换为 CompiledStateGraph：

1. **验证图结构**：检查所有节点是否连通，是否有孤立节点
2. **解析 Channel**：为每个状态字段创建对应的 Channel
3. **绑定 Schema**：设置 input_schema 和 output_schema
4. **生成执行计划**：确定节点执行顺序（基于 Pregel 算法）
5. **返回可执行对象**：CompiledStateGraph 实现了 Runnable 接口

```python
# 编译后的子图是一个 Runnable
compiled = subgraph.compile()

# 可以像普通函数一样调用
result = compiled.invoke(input_data)

# 也支持流式输出
for chunk in compiled.stream(input_data):
    print(chunk)
```

## 常见错误与注意事项

### 错误一：忘记编译子图

```python
# 错误：传入未编译的 StateGraph
parent.add_node("sub", subgraph)  # TypeError!

# 正确：传入编译后的 CompiledStateGraph
parent.add_node("sub", subgraph.compile())
```

### 错误二：input_schema 字段与 state_schema 不一致

```python
# 错误：input_schema 中的字段必须存在于 state_schema 中
class State(TypedDict):
    foo: str

class Input(TypedDict):
    bar: str  # state_schema 中没有 bar！

# 这会导致运行时错误
graph = StateGraph(State, input_schema=Input)
```

### 错误三：忽略 output_schema 导致状态泄露

```python
# 不设置 output_schema → 返回所有内部状态
# 如果内部状态包含敏感信息（API key、中间计算结果等），会泄露给父图
# 建议：始终显式设置 output_schema
```

## 小结

| 要素 | 说明 |
|------|------|
| **state_schema** | 子图内部的完整状态定义，所有节点都能访问 |
| **input_schema** | 子图接收的输入接口，默认等于 state_schema |
| **output_schema** | 子图返回的输出接口，默认等于 state_schema |
| **compile()** | 将 StateGraph 编译为可执行的 CompiledStateGraph |
| **add_node()** | 将编译后的子图作为节点添加到父图 |
| **最佳实践** | 始终显式设置 input/output_schema，保持接口清晰 |
