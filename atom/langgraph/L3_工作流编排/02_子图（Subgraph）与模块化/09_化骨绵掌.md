## 卡片 1：子图的本质

**一句话：** 子图就是编译后的 StateGraph 作为父图的一个节点，父图像调度普通函数一样调度它。

**举例：**

```python
from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict

class State(TypedDict):
    value: str

def process(state: State):
    return {"value": state["value"] + " -> sub"}

# 1. 定义子图（和普通图完全一样）
sub_builder = StateGraph(State)
sub_builder.add_node(process)
sub_builder.add_edge(START, "process")
sub_builder.add_edge("process", END)
child = sub_builder.compile()  # 编译

# 2. 嵌入父图（像普通节点一样 add_node）
parent_builder = StateGraph(State)
parent_builder.add_node("child", child)
parent_builder.add_edge(START, "child")
parent_builder.add_edge("child", END)
graph = parent_builder.compile()

print(graph.invoke({"value": "hello"}))
# {'value': 'hello -> sub'}
```

**在 LangGraph 中的应用：** 任何超过 5 个节点的图都应该考虑拆分为子图。子图可以独立调用 `child.invoke()` 进行单元测试，也可以嵌入父图参与完整工作流。编译后的 `CompiledStateGraph` 既是独立可执行体，也是父图的合法节点。

## 卡片 2：input_schema 与 output_schema

**一句话：** `input` 和 `output` 参数定义子图的"接口契约"，控制子图从父图接收哪些字段、向父图返回哪些字段。

**举例：**

```python
# 接口定义
class InputSchema(TypedDict):
    query: str

class OutputSchema(TypedDict):
    answer: str

# 内部状态（比接口更丰富）
class InternalState(TypedDict):
    query: str
    answer: str
    intermediate_steps: list   # 私有，不暴露
    confidence: float          # 私有，不暴露

# 用 input/output 构建子图
sub = StateGraph(InternalState, input=InputSchema, output=OutputSchema)
```

**在 LangGraph 中的应用：** 这就像函数签名 -- `input` 是参数列表，`output` 是返回值类型。内部状态可以有 10 个字段，但接口只暴露 2 个。父图只能看到 `output` 中定义的字段。这是实现子图解耦的关键手段：修改子图内部逻辑不会影响父图，只要接口不变。

## 卡片 3：共享状态键

**一句话：** 父图和子图 Schema 中同名的键会自动双向传递，无需任何额外代码。

**举例：**

```python
class ParentState(TypedDict):
    query: str       # 同名 -> 自动传给子图
    result: str      # 同名 -> 自动从子图接收
    metadata: dict   # 子图没有 -> 保留不变

class ChildState(TypedDict):
    query: str       # 同名 -> 自动从父图接收
    result: str      # 同名 -> 自动传回父图
    temp: int        # 父图没有 -> 子图私有

# 传递流程：
# 父图 {query:"hello", result:"", metadata:{}}
#   -> 子图收到 {query:"hello", result:""}
#   -> 子图返回 {query:"hello", result:"done"}
#   -> 父图变为 {query:"hello", result:"done", metadata:{}}
```

**在 LangGraph 中的应用：** 共享键是最简单的集成方式，适合父子图由同一团队开发的场景。需要注意的是：如果子图修改了一个同名键，修改会自动传回父图。如果不希望子图的内部字段"泄漏"，使用 `output` 参数限制返回字段。

## 卡片 4：状态转换

**一句话：** 当父子图 Schema 不同时，用包装函数手动完成状态映射，把子图当作"黑盒"调用。

**举例：**

```python
class ParentState(TypedDict):
    user_input: str    # 父图叫 user_input
    final_output: str

class ChildState(TypedDict):
    query: str         # 子图叫 query
    result: str

# 编译子图
child = StateGraph(ChildState)
# ... 添加节点和边 ...
compiled_child = child.compile()

# 包装函数：手动做状态映射
def call_child(state: ParentState):
    # 父图状态 -> 子图输入
    child_input = {"query": state["user_input"]}
    # 调用子图
    child_output = compiled_child.invoke(child_input)
    # 子图输出 -> 父图状态
    return {"final_output": child_output["result"]}

# 在父图中使用包装函数（而非直接添加子图）
parent = StateGraph(ParentState)
parent.add_node("child", call_child)  # 函数节点，不是子图节点
```

**在 LangGraph 中的应用：** 状态转换适合集成第三方子图或跨团队复用的场景。代价是需要手动写映射代码，但换来的是完全解耦 -- 父子图可以独立演进 Schema。注意：用函数包装后，子图的中间步骤不会出现在父图的流式输出中，除非手动透传。

## 卡片 5：Command.PARENT

**一句话：** 子图通过 `Command(goto="target", graph=Command.PARENT)` 向父图发送路由指令，是唯一让子图"控制"父图行为的机制。

**举例：**

```python
from langgraph.types import Command

class ChildState(TypedDict):
    messages: list
    task: str

def child_node(state: ChildState):
    # 子图判断需要转交给另一个子图
    if state["task"] == "billing":
        return Command(
            goto="billing_agent",        # 父图中的节点名
            graph=Command.PARENT,         # 路由指令发给父图
            update={"messages": [("assistant", "转交计费部门")]}
        )
    # 正常处理
    return Command(
        update={"messages": [("assistant", "已处理")]}
    )
```

**在 LangGraph 中的应用：** Command.PARENT 是多代理 Handoff 的核心。普通的子图只能影响自己的状态，而 Command.PARENT 让子图能说"请父图接下来去执行某个节点"。`update` 字段中的数据会直接更新到父图状态。注意父图状态中被更新的字段需要有合适的 reducer（如 `add_messages`），否则会被覆盖。

## 卡片 6：Send 动态分发

**一句话：** `Send` 在运行时动态创建多个子图实例并行执行，实现 map-reduce 模式。

**举例：**

```python
from langgraph.types import Send
from langgraph.graph import StateGraph, START, END
from typing import Annotated
import operator

class State(TypedDict):
    topics: list[str]
    summaries: Annotated[list[str], operator.add]

class WorkerState(TypedDict):
    topic: str
    summaries: Annotated[list[str], operator.add]

def fan_out(state: State):
    # 动态创建 N 个并行任务
    return [Send("summarize", {"topic": t}) for t in state["topics"]]

def summarize(state: WorkerState):
    return {"summaries": [f"Summary of {state['topic']}"]}

builder = StateGraph(State)
builder.add_node(summarize)
builder.add_conditional_edges(START, fan_out, ["summarize"])
builder.add_edge("summarize", END)
graph = builder.compile()

result = graph.invoke({"topics": ["AI", "ML", "NLP"], "summaries": []})
# summaries: ["Summary of AI", "Summary of ML", "Summary of NLP"]
```

**在 LangGraph 中的应用：** Send 适合"对多个文档分别处理再汇总"的场景，比如 RAG 中对检索到的 N 个文档分别做摘要、评分，然后汇总结果。每个 Send 创建一个独立的执行实例，共享键通过 reducer（如 `operator.add`）合并结果。

## 卡片 7：Checkpointer 继承

**一句话：** 子图的 checkpointer 有三种模式：继承父图（默认）、使用独立 checkpointer、显式禁用。

**举例：**

```python
from langgraph.checkpoint.memory import MemorySaver

# 模式 1：继承（默认）
# 子图共享父图的 checkpointer，状态统一持久化
parent_graph = parent_builder.compile(checkpointer=MemorySaver())
# 子图自动继承，无需额外配置

# 模式 2：独立
# 子图有自己的 checkpointer，独立管理状态
child_graph = child_builder.compile(checkpointer=MemorySaver())
parent_builder.add_node("child", child_graph)
# 子图的检查点独立于父图

# 模式 3：禁用
# 子图不做任何持久化
child_graph = child_builder.compile(checkpointer=False)
parent_builder.add_node("child", child_graph)
# 子图运行不产生检查点
```

**在 LangGraph 中的应用：** 继承模式最常用，适合需要统一恢复的场景（如整个对话从断点恢复）。独立模式适合子图有自己的会话管理需求（如每个代理维护独立的对话历史）。禁用模式适合无状态的工具调用子图（如计算器、API 调用），节省存储开销。

## 卡片 8：多层嵌套

**一句话：** 子图可以嵌套子图，形成 parent -> child -> grandchild 的多层结构，通过 `subgraphs=True` 流式监控所有层级。

**举例：**

```python
# 第三层：grandchild
grandchild_builder = StateGraph(State)
grandchild_builder.add_node("gc_node", gc_func)
grandchild_builder.add_edge(START, "gc_node")
grandchild = grandchild_builder.compile()

# 第二层：child（嵌套 grandchild）
child_builder = StateGraph(State)
child_builder.add_node("grandchild", grandchild)
child_builder.add_node("c_node", c_func)
child_builder.add_edge(START, "grandchild")
child_builder.add_edge("grandchild", "c_node")
child = child_builder.compile()

# 第一层：parent（嵌套 child）
parent_builder = StateGraph(State)
parent_builder.add_node("child", child)
parent_builder.add_edge(START, "child")
parent = parent_builder.compile()

# 流式监控所有层级
for namespace, chunk in parent.stream(
    {"value": "start"},
    subgraphs=True
):
    print(f"层级: {namespace}, 数据: {chunk}")
# 层级: (), 数据: ...             (parent)
# 层级: ('child',), 数据: ...     (child)
# 层级: ('child', 'grandchild'), 数据: ...  (grandchild)
```

**在 LangGraph 中的应用：** 多层嵌套适合大型系统：顶层是调度器，中间层是功能模块，底层是具体任务。`stream(subgraphs=True)` 返回的 `namespace` 元组表示层级路径，用于日志追踪和调试。生产环境中建议嵌套不超过 3 层，否则调试和状态追踪会变得困难。

## 卡片 9：多代理 Handoff

**一句话：** 每个代理是独立子图，通过 Command.PARENT 实现代理间切换，父图用条件边路由到目标代理。

**举例：**

```python
from langgraph.types import Command
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from typing import Annotated, Literal

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

# 代理 A：通用客服
def agent_a_node(state: AgentState):
    # 判断是否需要转交
    last_msg = state["messages"][-1].content
    if "退款" in last_msg:
        return Command(
            goto="agent_b",
            graph=Command.PARENT,
            update={"messages": [("assistant", "转交退款专员...")]}
        )
    return Command(update={"messages": [("assistant", "通用回复")]})

agent_a = StateGraph(AgentState)
agent_a.add_node("respond", agent_a_node)
agent_a.add_edge(START, "respond")
compiled_a = agent_a.compile()

# 代理 B：退款专员（类似结构）
# ...
compiled_b = ...

# 父图：调度器
class RouterState(TypedDict):
    messages: Annotated[list, add_messages]
    active_agent: str

parent = StateGraph(RouterState)
parent.add_node("agent_a", compiled_a)
parent.add_node("agent_b", compiled_b)

def route(state: RouterState) -> Literal["agent_a", "agent_b"]:
    return state.get("active_agent", "agent_a")

parent.add_conditional_edges(START, route)
```

**在 LangGraph 中的应用：** Handoff 模式是构建多代理系统的标准做法。每个代理独立开发和测试，通过 Command.PARENT 松耦合通信。LangGraph 还提供 `create_swarm()` 工具函数自动生成这种结构，适合标准场景；复杂场景（如分层 Handoff、有审批流程）需要手动实现。

## 卡片 10：子图设计原则

**一句话：** 好的子图设计遵循三个原则：单一职责、清晰接口、状态隔离。

**原则详解：**

| 原则 | 做法 | 反例 |
|------|------|------|
| **单一职责** | 一个子图只做一件事（如"检索"或"生成"） | 一个子图既检索又生成又评估 |
| **清晰接口** | 用 input/output_schema 显式定义边界 | 依赖同名键的隐式传递 |
| **状态隔离** | 子图内部状态对父图不可见 | 子图的中间变量泄漏到父图 |

**设计检查清单：**

```
1. 子图能否独立测试？
   -> 能用 child.invoke() 单独跑通

2. 修改子图内部实现会不会影响父图？
   -> 只要接口不变就不影响

3. 子图能否在其他父图中复用？
   -> 好的子图应该是可移植的

4. 子图的命名是否反映其职责？
   -> "retrieval_agent" 比 "sub1" 好

5. 嵌套层数是否合理？
   -> 建议不超过 3 层
```

**在 LangGraph 中的应用：** 遵循这些原则可以让 LangGraph 项目保持可维护性。一个常见的 RAG 系统划分方式是：`query_rewrite_sub`（查询改写子图）、`retrieval_sub`（检索子图）、`generation_sub`（生成子图）、`evaluation_sub`（评估子图），每个子图通过 input/output_schema 定义清晰边界，父图只负责编排和路由。
