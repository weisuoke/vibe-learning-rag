# 核心概念 5：参数注入机制

## 一句话定义

**参数注入机制是 LangChain 工具系统的高级特性，允许工具在运行时自动接收上下文信息（如配置、回调、工具调用 ID），而无需 LLM 显式传递这些参数。**

---

## 详细解释

### 什么是参数注入？

在传统的函数调用中，所有参数都需要调用者显式传递。但在 Agent 系统中，有些参数（如用户 ID、会话状态、配置信息）是**运行时上下文**，不应该由 LLM 决定。

**参数注入机制**解决了这个问题：
- **LLM 只需要传递业务参数**（如搜索关键词、计算表达式）
- **系统自动注入上下文参数**（如用户 ID、配置、回调）

**类比**：
- **前端类比**：就像 React 的 Context API，子组件可以访问全局状态而无需层层传递 props
- **日常生活类比**：就像餐厅服务员自动知道你的桌号，你只需要点菜，不需要每次都报桌号

---

### 为什么需要参数注入？

#### 问题场景

假设你有一个工具需要访问用户 ID：

```python
# ❌ 不好的设计：让 LLM 传递用户 ID
@tool
def get_user_data(user_id: str, query: str) -> str:
    """获取用户数据"""
    # 问题：LLM 怎么知道 user_id？
    return f"User {user_id}: {query}"

# LLM 调用时会困惑：
# "我应该传什么 user_id？我不知道当前用户是谁！"
```

#### 解决方案：参数注入

```python
# ✅ 好的设计：系统自动注入用户 ID
from langchain_core.tools import InjectedToolArg
from langchain_core.runnables import RunnableConfig
from typing import Annotated

@tool
def get_user_data(
    query: str,  # LLM 传递的业务参数
    config: Annotated[RunnableConfig, InjectedToolArg]  # 系统注入的上下文
) -> str:
    """获取用户数据"""
    user_id = config.get("configurable", {}).get("user_id")
    return f"User {user_id}: {query}"

# LLM 只需要传递 query，user_id 由系统注入
```

## 支持的注入类型

### 1. RunnableConfig 注入

**用途**：访问运行时配置，如用户 ID、会话 ID、元数据等

```python
from langchain_core.tools import tool, InjectedToolArg
from langchain_core.runnables import RunnableConfig
from typing import Annotated

@tool
def personalized_search(
    query: str,
    config: Annotated[RunnableConfig, InjectedToolArg]
) -> str:
    """个性化搜索工具"""
    # 从 config 中提取上下文信息
    user_id = config.get("configurable", {}).get("user_id")
    session_id = config.get("configurable", {}).get("session_id")
    language = config.get("configurable", {}).get("language", "zh")

    return f"为用户 {user_id} 搜索 '{query}' (会话: {session_id}, 语言: {language})"

# 使用时传递 config
result = personalized_search.invoke(
    {"query": "LangChain 教程"},
    config={
        "configurable": {
            "user_id": "user123",
            "session_id": "session456",
            "language": "zh"
        }
    }
)
```

**RunnableConfig 结构**：
```python
{
    "configurable": {
        "user_id": "...",
        "session_id": "...",
        # 自定义字段
    },
    "callbacks": [...],  # 回调处理器
    "tags": [...],       # 标签
    "metadata": {...},   # 元数据
}
```

### 2. InjectedToolCallId 注入

**用途**：获取当前工具调用的唯一 ID，用于追踪和日志记录

```python
from langchain_core.tools import tool, InjectedToolArg, InjectedToolCallId
from typing import Annotated

@tool
def tracked_search(
    query: str,
    tool_call_id: Annotated[str, InjectedToolCallId]
) -> str:
    """带追踪的搜索工具"""
    print(f"[工具调用 ID: {tool_call_id}] 搜索: {query}")

    # 记录到日志系统
    log_tool_call(tool_call_id, "search", {"query": query})

    return f"搜索结果 (ID: {tool_call_id})"

# 使用时自动注入 tool_call_id
result = tracked_search.invoke({"query": "LangChain"})
# 输出: [工具调用 ID: call_abc123] 搜索: LangChain
```

**应用场景**：
- 分布式追踪（Distributed Tracing）
- 日志关联
- 性能监控
- 错误排查

### 3. Callbacks 注入

**用途**：访问回调处理器，用于日志、监控、流式输出等

```python
from langchain_core.tools import tool, InjectedToolArg
from langchain_core.callbacks import CallbackManagerForToolRun
from typing import Annotated, Optional

@tool
def streaming_search(
    query: str,
    callbacks: Annotated[Optional[CallbackManagerForToolRun], InjectedToolArg] = None
) -> str:
    """支持流式输出的搜索工具"""
    # 模拟搜索过程
    steps = [
        "正在连接搜索引擎...",
        "正在检索相关文档...",
        "正在排序结果...",
        "搜索完成！"
    ]

    for step in steps:
        if callbacks:
            # 通过回调发送中间状态
            callbacks.on_tool_start({"name": "streaming_search"}, step)

    return f"找到 10 条关于 '{query}' 的结果"

# 使用时传递回调
from langchain.callbacks import StdOutCallbackHandler

result = streaming_search.invoke(
    {"query": "LangChain"},
    config={"callbacks": [StdOutCallbackHandler()]}
)
```

## 完整示例：多租户 RAG 系统

### 场景描述

构建一个多租户知识库系统，每个用户只能访问自己的文档。

```python
from langchain_core.tools import tool, InjectedToolArg, InjectedToolCallId
from langchain_core.runnables import RunnableConfig
from langchain_community.vectorstores import Chroma
from typing import Annotated
import logging

# 配置日志
logger = logging.getLogger(__name__)

# 假设有多个租户的向量数据库
tenant_vectorstores = {
    "tenant_a": Chroma(collection_name="tenant_a_docs"),
    "tenant_b": Chroma(collection_name="tenant_b_docs"),
}

@tool
def search_knowledge_base(
    query: str,
    top_k: int = 3,
    # 注入参数（不会出现在 schema 中）
    config: Annotated[RunnableConfig, InjectedToolArg] = None,
    tool_call_id: Annotated[str, InjectedToolCallId] = None,
) -> str:
    """从用户的知识库中搜索相关文档

    Args:
        query: 搜索关键词
        top_k: 返回文档数量（1-10）
    """
    # 1. 从 config 中提取租户信息
    if not config:
        raise ValueError("缺少配置信息")

    tenant_id = config.get("configurable", {}).get("tenant_id")
    user_id = config.get("configurable", {}).get("user_id")

    if not tenant_id:
        raise ValueError("缺少 tenant_id")

    # 2. 记录日志（使用 tool_call_id）
    logger.info(
        f"[{tool_call_id}] 租户 {tenant_id} 的用户 {user_id} 搜索: {query}"
    )

    # 3. 获取租户的向量数据库
    vectorstore = tenant_vectorstores.get(tenant_id)
    if not vectorstore:
        return f"错误：租户 {tenant_id} 的知识库不存在"

    # 4. 执行搜索
    docs = vectorstore.similarity_search(query, k=top_k)

    # 5. 格式化结果
    if not docs:
        return f"未找到关于 '{query}' 的相关文档"

    results = []
    for i, doc in enumerate(docs, 1):
        results.append(f"{i}. {doc.page_content[:200]}...")

    return "\n\n".join(results)

# ===== 使用示例 =====

# 租户 A 的用户搜索
result_a = search_knowledge_base.invoke(
    {"query": "如何使用 LangChain", "top_k": 3},
    config={
        "configurable": {
            "tenant_id": "tenant_a",
            "user_id": "user_a1"
        }
    }
)
print(f"租户 A 结果:\n{result_a}\n")

# 租户 B 的用户搜索（自动隔离数据）
result_b = search_knowledge_base.invoke(
    {"query": "如何使用 LangChain", "top_k": 3},
    config={
        "configurable": {
            "tenant_id": "tenant_b",
            "user_id": "user_b1"
        }
    }
)
print(f"租户 B 结果:\n{result_b}\n")
```

**关键优势**：
1. **安全性**：LLM 无法伪造 tenant_id，数据隔离由系统保证
2. **简洁性**：LLM 只需要传递 query 和 top_k
3. **可追踪**：通过 tool_call_id 记录每次调用
4. **灵活性**：可以在 config 中传递任意上下文信息

## 源码实现原理

### 参数过滤机制

```python
# 源码位置: langchain_core/tools/base.py

# 1. 定义需要过滤的参数
FILTERED_ARGS = ("run_manager", "callbacks")

# 2. 检测注入参数
def _is_injected_arg_type(type_: type) -> bool:
    """检查参数是否为注入类型"""
    return (
        type_ is InjectedToolArg
        or getattr(type_, "__origin__", None) is Annotated
        and any(_is_injected_arg_type(arg) for arg in type_.__args__)
    )

# 3. 过滤参数
def _get_filtered_args(
    inferred_model: type[BaseModel],
    func: Callable,
    *,
    filter_args: Sequence[str],
    include_injected: bool = True,
) -> dict:
    """获取过滤后的参数"""
    schema = inferred_model.model_json_schema()
    valid_keys = set(schema["properties"])

    for param_name, param in inspect.signature(func).parameters.items():
        # 检查是否为注入参数
        if not include_injected and _is_injected_arg_type(param.annotation):
            filter_args_.append(param_name)

    # 从 schema 中移除注入参数
    return {k: v for k, v in schema["properties"].items() if k not in filter_args_}
```

**工作流程**：
1. 使用 `inspect.signature()` 获取函数签名
2. 检测哪些参数标记了 `InjectedToolArg`
3. 从生成的 JSON Schema 中移除这些参数
4. LLM 只能看到业务参数

## 最佳实践

### 1. 始终提供默认值

```python
# ✅ 推荐：提供默认值
@tool
def my_tool(
    query: str,
    config: Annotated[RunnableConfig, InjectedToolArg] = None,  # 默认 None
) -> str:
    if config:
        user_id = config.get("configurable", {}).get("user_id")
    else:
        user_id = "anonymous"
    return f"User {user_id}: {query}"

# ❌ 不推荐：不提供默认值
@tool
def my_tool(
    query: str,
    config: Annotated[RunnableConfig, InjectedToolArg],  # 缺少默认值
) -> str:
    # 如果调用时没有传递 config，会报错
    user_id = config.get("configurable", {}).get("user_id")
    return f"User {user_id}: {query}"
```

### 2. 使用类型提示

```python
from typing import Annotated, Optional

# ✅ 推荐：明确类型
@tool
def my_tool(
    query: str,
    config: Annotated[Optional[RunnableConfig], InjectedToolArg] = None,
) -> str:
    # 类型检查器知道 config 可能为 None
    if config:
        user_id = config.get("configurable", {}).get("user_id")
    return f"Query: {query}"
```

### 3. 安全地访问嵌套字段

```python
# ✅ 推荐：使用 .get() 链式调用
user_id = config.get("configurable", {}).get("user_id", "default_user")

# ❌ 不推荐：直接访问可能抛出 KeyError
user_id = config["configurable"]["user_id"]
```

### 4. 文档注释中不要提及注入参数

```python
@tool
def my_tool(
    query: str,
    config: Annotated[RunnableConfig, InjectedToolArg] = None,
) -> str:
    """搜索工具

    Args:
        query: 搜索关键词
        # ❌ 不要写：config: 运行时配置（LLM 看不到这个参数）

    Returns:
        搜索结果
    """
    return f"Results for {query}"
```

## 常见陷阱

### 陷阱 1：忘记标记 InjectedToolArg

```python
# ❌ 错误：没有标记 InjectedToolArg
@tool
def my_tool(
    query: str,
    config: RunnableConfig,  # 缺少 InjectedToolArg 标记
) -> str:
    return f"Query: {query}"

# 结果：LLM 会尝试传递 config 参数，导致错误

# ✅ 正确：使用 Annotated 标记
@tool
def my_tool(
    query: str,
    config: Annotated[RunnableConfig, InjectedToolArg],
) -> str:
    return f"Query: {query}"
```

### 陷阱 2：在 StructuredTool 中使用注入参数

```python
from langchain.tools import StructuredTool
from pydantic import BaseModel, Field

class MySchema(BaseModel):
    query: str = Field(description="搜索关键词")
    # ❌ 错误：Pydantic schema 中不能使用 InjectedToolArg
    config: Annotated[RunnableConfig, InjectedToolArg] = None

# ✅ 正确：在函数签名中使用，不在 schema 中
class MySchema(BaseModel):
    query: str = Field(description="搜索关键词")

def my_function(
    query: str,
    config: Annotated[RunnableConfig, InjectedToolArg] = None,
) -> str:
    return f"Query: {query}"

tool = StructuredTool.from_function(
    func=my_function,
    name="my_tool",
    description="我的工具",
    args_schema=MySchema,
)
```

### 陷阱 3：混淆 config 和 kwargs

```python
# ❌ 错误：将 config 作为普通参数传递
result = my_tool.invoke({"query": "test", "config": {...}})

# ✅ 正确：config 作为 invoke 的第二个参数
result = my_tool.invoke(
    {"query": "test"},
    config={...}  # 作为独立参数
)
```

## 与其他框架的对比

### LangChain vs LlamaIndex

| 特性 | LangChain | LlamaIndex |
|------|-----------|------------|
| **参数注入** | ✅ InjectedToolArg | ⚠️ 有限支持 |
| **配置传递** | RunnableConfig | ServiceContext |
| **回调机制** | CallbackManager | CallbackManager |
| **灵活性** | ✅ 高 | ⚠️ 中等 |

### LangChain vs AutoGPT

| 特性 | LangChain | AutoGPT |
|------|-----------|---------|
| **上下文传递** | ✅ 自动注入 | ⚠️ 手动传递 |
| **类型安全** | ✅ 类型提示 | ❌ 无类型检查 |
| **学习曲线** | ⚠️ 中等 | ✅ 简单 |

## 高级用法：自定义注入类型

虽然 LangChain 提供了内置的注入类型，你也可以创建自定义注入逻辑：

```python
from typing import Annotated, Any
from langchain_core.tools import tool, InjectedToolArg
from langchain_core.runnables import RunnableConfig

# 自定义注入类型标记
class InjectedDatabase(InjectedToolArg):
    """标记数据库连接需要注入"""
    pass

@tool
def query_database(
    sql: str,
    db: Annotated[Any, InjectedDatabase] = None,
) -> str:
    """执行 SQL 查询"""
    if not db:
        raise ValueError("数据库连接未注入")

    result = db.execute(sql)
    return str(result)

# 使用时需要自定义 ToolNode 来处理注入
# （这是高级用法，通常不需要）
```
