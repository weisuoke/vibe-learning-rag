## 卡片1：架构全景 - 三层设计

**一句话：** Milvus 2.6 采用存算分离的三层架构：Coordinator（协调）、Worker（执行）、Storage（存储）

**举例：**
```
类比餐厅：
- Coordinator = 管理层（经理、主管）
- Worker = 执行层（服务员、厨师）
- Storage = 仓库（食材存储）

优势：
- 管理层可以独立优化（换经理不影响厨师）
- 执行层可以独立扩展（客人多了加服务员）
- 仓库可以独立扩展（食材多了扩展仓库）
```

**应用：** 在 RAG 系统中，查询高峰时只扩展 Query Node（服务员），文档增长时只扩展 Object Storage（仓库），成本优化 60%

## 卡片2：存算分离 - 核心理念

**一句话：** 计算和存储独立部署、独立扩展，是 Milvus 2.6 架构的核心理念

**举例：**
```
传统架构（存算耦合）：
  计算 + 存储 = 一台机器
  问题：扩展时必须一起扩展，资源浪费

存算分离：
  计算 = Worker Node（无状态）
  存储 = Object Storage（S3/MinIO）
  优势：独立扩展，按需付费
```

**应用：** RAG 系统白天查询多（扩展 Query Node），夜间文档入库多（扩展 Data Node），存储始终共享，成本降低 65%

---

## 卡片3：Streaming Node - 2.6 核心创新

**一句话：** Streaming Node 是 Milvus 2.6 的核心组件，统一处理所有流式数据，替代外部消息队列

**举例：**
```
2.5 架构：
  数据 → Pulsar/Kafka → Data Node
  问题：依赖外部组件，延迟高，成本高

2.6 架构：
  数据 → Streaming Node → Data Node
  优势：内置处理，延迟降低 50%，成本降低 30%
```

**应用：** RAG 系统实时文档入库，用户上传后 < 1秒可搜索（得益于 Streaming Node 的低延迟）

---

## 卡片4：Woodpecker WAL - 零磁盘基石

**一句话：** Woodpecker WAL 将预写日志直接写入对象存储，实现零磁盘架构

**举例：**
```
传统 WAL：
  数据 → 本地磁盘 WAL → 对象存储
  问题：需要本地磁盘，成本高

Woodpecker WAL：
  数据 → 对象存储 WAL → 对象存储
  优势：零磁盘，成本降低 90%
```

**应用：** RAG 系统 100TB 向量数据，使用 Woodpecker WAL 节省 $20,000/月的本地磁盘成本

## 卡片5：Coordinator 层 - 大脑中枢

**一句话：** Coordinator 层包含 Root/Data/Query Coord，负责协调任务，不执行计算

**举例：**
```
三个 Coordinator：
- Root Coord：管理元数据（Collection、Partition）
- Data Coord：协调数据写入和索引构建
- Query Coord：协调查询任务和负载均衡

2.6 新特性：Coord Merge
- 小规模部署时可以合并为一个进程
- 降低资源消耗
```

**应用：** RAG 系统小规模部署（< 10GB 数据）使用 Coord Merge，节省 2 个进程的资源

---

## 卡片6：Worker 层 - 无状态执行者

**一句话：** Worker 层包含 Query/Data/Index/Streaming Node，执行具体任务，完全无状态

**举例：**
```
四个 Worker：
- Query Node：执行向量检索
- Data Node：处理数据写入
- Index Node：构建向量索引
- Streaming Node：处理流式数据（2.6 新增）

无状态的优势：
- 快速启动（秒级）
- 快速替换（故障恢复）
- 灵活调度（任意节点处理）
```

**应用：** RAG 系统查询高峰时，Kubernetes 自动扩展 Query Node 从 2 个到 10 个，2 分钟内完成

---

## 卡片7：Storage 层 - 持久化保障

**一句话：** Storage 层包含 etcd（元数据）、Object Storage（数据）、Woodpecker WAL（日志）

**举例：**
```
三个存储：
- etcd：存储 Collection/Partition 元数据
- Object Storage：存储向量数据和索引文件
- Woodpecker WAL：存储预写日志

可靠性：
- etcd：99.99%（4 个 9）
- Object Storage：99.999999999%（11 个 9）
- Woodpecker WAL：99.999999999%（11 个 9）
```

**应用：** RAG 系统 1 年内 0 次数据丢失（得益于 Storage 层的高可靠性）

## 卡片8：零磁盘架构 - 成本革命

**一句话：** Worker 节点完全无状态，不需要本地磁盘，所有数据持久化在对象存储

**举例：**
```
成本对比（100TB 数据）：
传统架构：
  本地 SSD: $20,000/月
  对象存储: $2,300/月
  总成本: $22,300/月

零磁盘架构：
  对象存储: $2,300/月
  节省: $20,000/月（90%）
```

**应用：** RAG 系统大规模部署（100TB+）使用零磁盘架构，每年节省 $240,000

## 卡片9：数据流转 - 端到端流程

**一句话：** 数据从 Client 写入到可搜索，经过 Streaming Node → Data Node → Index Node → Query Node

**举例：**
```
完整流程：
1. Client 写入数据
   ↓
2. Streaming Node 接收并写入 WAL
   ↓
3. Data Node 持久化到对象存储
   ↓
4. Index Node 构建索引
   ↓
5. Query Node 加载索引到内存
   ↓
6. Client 查询数据

关键路径：
- 写入路径：Client → Streaming Node → Data Node → Object Storage
- 查询路径：Client → Query Node（内存检索）→ Client
```

**应用：** RAG 系统理解数据流转，优化写入和查询性能，延迟降低 50%

## 卡片10：架构权衡 - 设计哲学

**一句话：** Milvus 2.6 的架构设计是在性能、成本、复杂度之间的权衡

**举例：**
```
权衡1：存算分离 vs 单体架构
- 牺牲：部署复杂度
- 换取：可扩展性、成本优化

权衡2：零磁盘 vs 本地磁盘
- 牺牲：微秒级延迟
- 换取：90% 成本节省

权衡3：微服务 vs 单体服务
- 牺牲：运维复杂度
- 换取：故障隔离、独立优化
```

**应用：** RAG 系统根据规模选择架构：小规模用 Coord Merge 简化部署，大规模用完整微服务架构
