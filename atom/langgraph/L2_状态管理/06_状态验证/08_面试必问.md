# 状态验证 - 面试必问

> 2 个高频面试问题，帮你从"知道"到"说得出彩"

---

## 问题1："LangGraph 中 TypedDict 和 Pydantic BaseModel 定义状态有什么区别？什么时候用哪个？"

### 普通回答（❌ 不出彩）

"TypedDict 是 Python 的类型提示，Pydantic 是数据验证库。Pydantic 可以做运行时验证，TypedDict 不行。"

**问题**：只说了表面差异，没有深度，面试官会追问"那生产环境怎么选？"

### 出彩回答（✅ 推荐）

> **这个问题涉及三个层面：**
>
> **1. 类型安全层面**：TypedDict 提供静态类型检查（IDE 和 mypy），Pydantic 提供运行时验证。两者不是替代关系，而是互补。
>
> **2. 性能层面**：TypedDict 几乎零开销，Pydantic 有验证开销。在 LangGraph 中，Pydantic 验证仅在首个节点输入时触发（通过 `_coerce_state`），所以性能影响有限。
>
> **3. 架构层面**：生产环境推荐混合架构——`input_schema` 用 Pydantic 验证外部输入，内部 `state_schema` 用 TypedDict + Reducer 保持轻量。这是社区的最佳实践共识。
>
> **具体选型**：
> - 处理不可信输入（用户输入、LLM 输出）→ Pydantic
> - 内部节点间传递 + 需要 Reducer → TypedDict
> - 需要自动类型转换（LLM 返回字符串数字）→ Pydantic
>
> **在实际项目中**，我通常用 `StateGraph(InternalState, input=InputState)` 的混合模式，InputState 是 Pydantic，InternalState 是 TypedDict。

### 为什么这个回答出彩？

1. ✅ **多层次分析**（类型安全、性能、架构）—— 展示系统思维
2. ✅ **提到源码细节**（`_coerce_state`）—— 证明读过源码
3. ✅ **给出具体的架构建议** —— 展示工程判断力
4. ✅ **展示实际项目经验** —— 不是纸上谈兵

### 追问预备

**Q: TypedDict 能不能也做运行时验证？**

> 不能。TypedDict 在运行时就是普通 dict，类型注解只供静态分析工具使用。如果你传入错误类型的值，Python 不会报错，只有 mypy 等工具会提示。这就是为什么处理外部输入时必须用 Pydantic。

**Q: Pydantic 验证在 LangGraph 中具体什么时候触发？**

> 在 `graph.invoke()` 或 `graph.stream()` 调用时，LangGraph 内部通过 `_coerce_state` 方法将输入数据传给 Pydantic 模型进行验证。验证只发生在图的入口处，节点之间传递状态时不会重复验证。

**Q: 为什么不全用 Pydantic？**

> 两个原因：一是 Pydantic BaseModel 实例是不可变的（默认行为），而 LangGraph 的 Reducer 机制需要对状态进行增量更新，TypedDict（本质是 dict）天然支持这种操作；二是内部节点间的数据已经经过入口验证，重复验证是浪费。

## 问题2："Pydantic 的 field_validator 和 model_validator 有什么区别？在 LangGraph 状态中怎么用？"

### 普通回答（❌ 不出彩）

"field_validator 验证单个字段，model_validator 验证整个模型。"

**问题**：太笼统，没有说清执行时机和实际应用场景。

### 出彩回答（✅ 推荐）

> **两者的核心区别在于验证粒度和执行时机：**
>
> **1. field_validator**：字段级验证，每个字段独立验证。适合：格式清洗（去空格）、范围检查（1-100）、格式验证（邮箱正则）。
>
> **2. model_validator**：模型级验证，可以访问所有字段。适合：跨字段约束（开始日期 < 结束日期）、条件依赖（高级模式需要 API key）、互斥字段检查。
>
> **执行顺序**：`field_validator(before)` → 类型转换 → `field_validator(after)` → `model_validator(after)`
>
> **在 LangGraph 中的典型用法**：
>
> ```python
> class RAGState(BaseModel):
>     query: str
>     top_k: int = 5
>     threshold: float = 0.7
>
>     @field_validator('query')
>     @classmethod
>     def clean_query(cls, v):
>         return v.strip()
>
>     @model_validator(mode='after')
>     def validate_params(self):
>         if self.top_k > 20 and self.threshold < 0.5:
>             raise ValueError('参数组合不合理：top_k 过大时 threshold 不能过低')
>         return self
> ```
>
> **注意**：在 LangGraph 中，这些验证器只在图的输入阶段触发，不会在每个节点间重复执行。所以验证器应该聚焦于"入口数据质量"，而不是"中间状态一致性"。

### 为什么这个回答出彩？

1. ✅ **明确执行顺序** —— 展示对 Pydantic 内部机制的理解
2. ✅ **给出具体代码** —— 不是空谈概念
3. ✅ **结合 LangGraph 场景** —— 说明验证触发时机
4. ✅ **指出实际注意事项** —— 展示踩坑经验

### 追问预备

**Q: field_validator 的 mode='before' 和 mode='after' 有什么区别？**

> `before` 模式在类型转换之前执行，接收的是原始输入值（可能是任意类型）；`after` 模式在类型转换之后执行，接收的是已经转换好的目标类型值。一般用 `before` 做原始数据清洗，用 `after` 做业务逻辑验证。

**Q: model_validator 的 mode='before' 接收什么参数？**

> 接收原始输入数据，通常是一个 dict。此时字段还没有被解析和验证，适合做数据预处理，比如从一个字段派生出另一个字段的默认值。

**Q: 验证失败了怎么处理？**

> Pydantic 会抛出 `ValidationError`，包含结构化的错误信息。在 LangGraph 中，建议在 `graph.invoke()` 外层用 try-except 捕获，将错误转换为用户友好的提示，而不是让异常直接传播。

## 面试加分技巧

### 回答框架：STAR-T

| 步骤 | 含义 | 示例 |
|------|------|------|
| **S** | Situation（场景） | "在 RAG 系统中..." |
| **T** | Task（任务） | "需要验证用户查询输入..." |
| **A** | Action（行动） | "我用 Pydantic 定义了 InputState..." |
| **R** | Result（结果） | "拦截了 30% 的无效请求..." |
| **T** | Takeaway（总结） | "所以混合架构是最佳实践..." |

### 关键词清单

面试中提到这些关键词会加分：

- `_coerce_state`（源码级理解）
- 混合架构 / `input_schema`（工程实践）
- 验证触发时机（框架理解）
- `ValidationError.errors()`（错误处理）
- `before` / `after` 模式（Pydantic 深度）

### 常见陷阱

面试中容易踩的坑：

1. **只说概念不说场景** —— 面试官想听的是"你在什么场景下用过"，不是"它是什么"
2. **把 TypedDict 和 Pydantic 说成对立关系** —— 它们是互补的，不是二选一
3. **忽略性能话题** —— 提到"验证只在入口触发"会让面试官觉得你真正用过
4. **不提错误处理** —— 生产环境中 ValidationError 的处理方式是加分项
