# Agent最佳实践 - 反直觉点

> 这些关于 Agent 最佳实践的常见误解，可能正在让你的 Agent 在生产环境中"裸奔"。

---

## 反直觉点 1：Middleware 的顺序不是随便排的

### 直觉 ❌

"Middleware 就是一个列表，顺序不重要，反正都会执行。"

### 真相 ✅

**Middleware 的声明顺序直接决定执行行为！** 错误的顺序会导致保护失效。

```python
# ❌ 错误顺序：重试在限制外面
middleware=[
    ModelRetryMiddleware(max_retries=3),         # 外层：重试
    ModelCallLimitMiddleware(run_limit=25),        # 内层：限制
]
# 后果：每次重试不计入限制！3 次重试 × 25 次限制 = 实际可能调用 75 次！

# ✅ 正确顺序：限制在重试外面
middleware=[
    ModelCallLimitMiddleware(run_limit=25),        # 外层：限制
    ModelRetryMiddleware(max_retries=3),            # 内层：重试
]
# 重试也受限制管控，总调用不超过 25 次
```

**记忆口诀：** "限制在外（保底），重试在中（容错），安全在内（靠近数据）"

## 反直觉点 2：重试不是越多越好

### 直觉 ❌

"max_retries 设大一点，比如 10 次，成功率不就更高了？"

### 真相 ✅

过多的重试会**加剧问题**而不是解决问题：

```
max_retries=10 的真实代价：

时间成本（指数退避）：
  第 1 次重试：等 1s
  第 2 次重试：等 2s
  第 3 次重试：等 4s
  第 4 次重试：等 8s
  第 5 次重试：等 16s
  第 6 次重试：等 30s（已达上限）
  第 7-10 次：各等 30s
  ─────────────────
  总等待：~181s = 3 分钟！用户早走了。

  而 max_retries=3 的总等待：~7s，用户可以接受。
```

**最佳实践：** 一般设 `max_retries=3`，配合 `ModelFallbackMiddleware` 降级到备用模型。

## 反直觉点 3：PII 检测不能只管输入，不管输出

### 直觉 ❌

"只要脱敏用户的输入就行了，LLM 不会自己编造 PII。"

### 真相 ✅

**LLM 会在输出中"回忆"训练数据中的真实 PII！**

```
场景：
  用户问："张三的电话号码是多少？"
  LLM 可能回答："张三的电话是 138xxxx5678"  ← 从训练数据中"回忆"出来的！

  更危险的场景：
  LLM 在多步推理中，把前面工具返回的真实数据直接暴露给用户

这就是 PIIMiddleware 需要同时使用 before_model 和 after_model 的原因：
  before_model → 脱敏用户输入（防止输入泄露）
  after_model  → 脱敏 LLM 输出（防止输出泄露）
```

## 反直觉点 4：Agent 调用限制不是"限制能力"，而是"保命"

### 直觉 ❌

"`run_limit=25` 太小了吧？复杂任务可能需要更多步骤，限制太紧会影响效果。"

### 真相 ✅

**如果 Agent 需要 25 步以上还没完成任务，大概率它已经陷入了死循环。**

```
正常 Agent 执行：
  步骤 1: 理解问题
  步骤 2: 调用搜索
  步骤 3: 分析结果
  步骤 4: 生成答案
  → 4 步完成 ✅

死循环 Agent 执行：
  步骤 1: 调用搜索
  步骤 2: 结果不满意，再搜一次
  步骤 3: 还是不满意，换个关键词搜
  步骤 4: 搜到了，但觉得不够，再搜
  ...
  步骤 25: 还在搜... → 触发限制，强制结束 ✅

  没有限制 →
  步骤 100: 还在搜... Token 费用已经 $50 了 💀
```

**最佳实践：** 大多数场景 `run_limit=25` 足够。数据分析等需要多步的场景可以设 50。

## 反直觉点 5：工具的 description 比 Agent 的 System Prompt 更重要

### 直觉 ❌

"只要 System Prompt 写得好，Agent 就能正确使用工具。"

### 真相 ✅

**LLM 选择工具时，主要看的是工具的 `description`，不是 System Prompt。**

```python
# System Prompt 写得再好也没用
system_prompt = """你是一个专业的客服助手，请使用合适的工具回答问题。
当用户问订单相关问题时，请使用订单查询工具。
当用户问天气时，请使用天气查询工具。"""

# 但如果工具 description 模糊...
class Tool1(BaseTool):
    name = "tool_a"
    description = "A useful tool"  # ← LLM 看到这个，完全不知道该不该用！

# LLM 的工具选择决策链：
# 1. 看到用户问"查订单"
# 2. 扫描所有工具的 description
# 3. 发现 "A useful tool" — 没法判断是不是订单工具
# 4. 随机选一个 → 选错了 ❌
```

**记住：** System Prompt 告诉 Agent "你是谁"，工具 description 告诉 Agent "这个工具能做什么"。两者缺一不可，但工具 description 对工具选择的影响更大。

## 反直觉点 6：Callback 和 Middleware 不是一回事

### 直觉 ❌

"Callback 和 Middleware 都能在执行过程中插入逻辑，应该差不多吧？"

### 真相 ✅

```
Callback = 观察者（看但不碰）
  → 只能观察事件，不能修改数据
  → 用于日志、监控、追踪
  → 类比：安保摄像头

Middleware = 拦截者（可以改、可以拦、可以重试）
  → 可以修改请求和响应
  → 用于重试、安全、上下文管理
  → 类比：安保人员（可以拦人、检查、拒绝进入）
```

如果你想**记录日志**，用 Callback。如果你想**修改行为**，用 Middleware。用错了工具，要么功能实现不了，要么引入不必要的复杂度。

## 反直觉点 7：单 Agent 往往比多 Agent 更好

### 直觉 ❌

"多 Agent 系统更强大、更灵活，应该尽量使用多 Agent 架构。"

### 真相 ✅

**多 Agent 带来的复杂度往往大于它解决的问题。**

```
多 Agent 的隐藏成本：
  1. 调试困难：错误在哪个 Agent？消息传递是否正确？
  2. 延迟翻倍：Agent 之间的通信需要额外的 LLM 调用
  3. 上下文割裂：每个 Agent 只看到自己的上下文，可能遗漏信息
  4. 成本倍增：多个 Agent = 多倍 Token 消耗

只有当这些条件出现时才需要多 Agent：
  • 工具超过 15 个
  • 需要不同的权限级别
  • 需要不同的专业领域
  • 需要并行处理
```

**记住：** 从单 Agent 开始，只在遇到瓶颈时才升级到多 Agent。

## 反直觉总结表

| # | 直觉 ❌ | 真相 ✅ |
|---|---------|---------|
| 1 | Middleware 顺序不重要 | 顺序决定行为，限制必须在重试外面 |
| 2 | 重试越多越好 | 3 次足够，多了反而加剧问题 |
| 3 | 只脱敏输入就行 | LLM 输出也会泄露 PII |
| 4 | 调用限制太紧影响效果 | 25 步还没完成大概率是死循环 |
| 5 | System Prompt 比工具描述重要 | 工具 description 对工具选择影响更大 |
| 6 | Callback 和 Middleware 差不多 | 观察者 vs 拦截者，能力完全不同 |
| 7 | 多 Agent 更强大 | 单 Agent 足够时不要过度工程化 |
