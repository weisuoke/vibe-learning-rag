# 状态持久化准备 - 第一性原理

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题。

不是"别人怎么做我就怎么做"，而是"这件事最本质的需求是什么？从这个需求出发，最合理的解决方案是什么？"

对于状态持久化准备，我们要问的第一性问题是：

> **图的状态是活在进程内存里的 Python 对象——当进程结束、机器重启、服务迁移时，这些对象就消失了。如何让状态"活"过这些边界？**

---

## 状态持久化准备的第一性原理

### 1. 最基础的定义

**状态持久化准备 = 在设计状态时，确保状态数据能被安全、高效地转换为可存储的字节序列**

```python
# 一句话版本：
# 你的状态必须能变成字节，字节必须能变回状态，而且这个过程要快、要安全

# 源码层面：这就是 SerializerProtocol 的核心契约
class SerializerProtocol(Protocol):
    def dumps_typed(self, obj: Any) -> tuple[str, bytes]:
        """Python 对象 → (类型标签, 字节)"""
        ...
    def loads_typed(self, data: tuple[str, bytes]) -> Any:
        """(类型标签, 字节) → Python 对象"""
        ...
```

仅此而已！状态持久化准备的本质就是**确保你的状态能完成这个往返转换**。

想象你要搬家。搬家公司（Checkpointer）负责运输，但你得先把东西打包好：
- 衣服、书籍可以直接装箱（基础类型：str, int, list）
- 花瓶要用泡沫纸包好（特殊类型：datetime, UUID, Pydantic 需要特殊编码）
- 鱼缸里的活鱼不能装箱（不可序列化：数据库连接、文件句柄）
- 箱子不能太重搬不动（状态大小控制）
- 贵重首饰要放保险箱（敏感数据加密）

**状态持久化准备就是"打包"的过程——在搬家公司来之前，确保所有东西都能安全运输。**

---

### 2. 为什么需要状态持久化准备？

**核心问题：图的状态是内存中的 Python 对象，而内存是易失的。要让状态跨越进程边界存活，就必须将其转换为可存储的格式。**

想象一下没有持久化准备的世界：

```python

# ❌ 没有考虑持久化的状态设计
