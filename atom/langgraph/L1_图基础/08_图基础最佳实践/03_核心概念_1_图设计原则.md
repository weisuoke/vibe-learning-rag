# 图基础最佳实践 - 核心概念：图设计原则

## 概述

图设计原则是构建稳定、可维护、高性能 LangGraph 应用的基础。本文档介绍 7 个核心设计原则，每个原则都基于实际生产环境的经验总结。

## 核心概念1：单一职责原则

**一句话定义**：每个节点只负责一个明确的功能，避免在单个节点中混合多个职责。

### 代码示例

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    query: str
    search_results: list
    is_valid: bool
    formatted_results: str

# ❌ 错误示例：节点做太多事情
def bad_process_node(state: State):
    """搜索 + 验证 + 格式化 + 日志 + 缓存"""
    # 1. 搜索
    results = search_api(state["query"])

    # 2. 验证
    is_valid = len(results) > 0

    # 3. 格式化
    formatted = "\n".join([r["title"] for r in results])

    # 4. 日志
    print(f"Processed {len(results)} results")

    # 5. 缓存
    cache.set(state["query"], formatted)

    return {
        "search_results": results,
        "is_valid": is_valid,
        "formatted_results": formatted
    }

# ✅ 正确示例：每个节点单一职责
def search_node(state: State):
    """只负责搜索"""
    results = search_api(state["query"])
    return {"search_results": results}

def validate_node(state: State):
    """只负责验证"""
    is_valid = len(state["search_results"]) > 0
    return {"is_valid": is_valid}

def format_node(state: State):
    """只负责格式化"""
    formatted = "\n".join([r["title"] for r in state["search_results"]])
    return {"formatted_results": formatted}

# 构建图
graph = StateGraph(State)
graph.add_node("search", search_node)
graph.add_node("validate", validate_node)
graph.add_node("format", format_node)

graph.add_edge(START, "search")
graph.add_edge("search", "validate")
graph.add_edge("validate", "format")
graph.add_edge("format", END)

app = graph.compile()
```

### 详细解释

**为什么需要单一职责？**

1. **易于测试**：每个节点功能单一，测试用例简单明确
2. **易于调试**：问题定位快速，不需要在复杂逻辑中排查
3. **易于复用**：单一职责的节点可以在不同图中复用
4. **易于维护**：修改一个功能不会影响其他功能

**如何判断节点职责是否单一？**

- 节点名称能否用一个动词清晰描述？（搜索、验证、格式化）
- 节点函数是否可以在 20 行代码内完成？
- 节点是否只修改状态中的 1-2 个字段？
- 节点是否可以独立测试？

### 在实际应用中的使用

**RAG 系统示例**：

```python
# 将 RAG 流程拆分为多个单一职责节点
def load_documents_node(state):
    """只负责加载文档"""
    docs = load_documents(state["file_path"])
    return {"documents": docs}

def chunk_documents_node(state):
    """只负责分块"""
    chunks = chunk_text(state["documents"])
    return {"chunks": chunks}

def embed_chunks_node(state):
    """只负责向量化"""
    embeddings = embed_text(state["chunks"])
    return {"embeddings": embeddings}

def store_vectors_node(state):
    """只负责存储"""
    vector_store.add(state["chunks"], state["embeddings"])
    return {"stored": True}
```

## 核心概念2：状态最小化原则

**一句话定义**：状态中只存储必要的数据，大对象存储在外部，减少序列化开销。

### 代码示例

```python
from typing_extensions import TypedDict

# ❌ 错误示例：在状态中存储大对象
class BadState(TypedDict):
    documents: list[str]  # 可能包含数千个文档，每个几KB
    embeddings: list[list[float]]  # 每个文档的 embedding，768维
    search_results: list[dict]  # 完整的搜索结果

# ✅ 正确示例：只存储引用
class GoodState(TypedDict):
    document_ids: list[str]  # 只存储 ID
    query: str
    top_k_ids: list[str]  # 只存储 top-k 的 ID

# 大对象存储在外部
document_store = {}  # 或使用数据库
embedding_store = {}

def search_node(state: GoodState):
    # 从外部存储加载文档
    documents = [document_store[id] for id in state["document_ids"]]
    embeddings = [embedding_store[id] for id in state["document_ids"]]

    # 执行搜索
    top_k_ids = search(state["query"], documents, embeddings)

    # 只返回 ID
    return {"top_k_ids": top_k_ids}
```

### 详细解释

**为什么需要状态最小化？**

1. **性能优化**：状态需要序列化保存到 Checkpoint，大对象导致序列化慢
2. **内存优化**：减少内存占用，避免 OOM
3. **网络优化**：分布式部署时减少网络传输开销
4. **存储优化**：Checkpoint 存储空间更小

**状态最小化策略**：

1. **只存储 ID**：大对象存储在外部，状态中只存储引用
2. **只存储必要字段**：避免存储中间计算结果
3. **使用外部存储**：数据库、缓存、文件系统
4. **定期清理**：删除不再需要的状态字段

### 在实际应用中的使用

**生产环境示例**：

```python
import redis
from typing_extensions import TypedDict

# 使用 Redis 存储大对象
redis_client = redis.Redis(host='localhost', port=6379, db=0)

class State(TypedDict):
    session_id: str
    document_ids: list[str]
    query: str

def process_node(state: State):
    # 从 Redis 加载大对象
    documents = []
    for doc_id in state["document_ids"]:
        doc = redis_client.get(f"doc:{doc_id}")
        documents.append(doc)

    # 处理文档
    results = process(documents, state["query"])

    # 将结果存储到 Redis
    result_id = f"result:{state['session_id']}"
    redis_client.set(result_id, results)

    # 状态中只返回 ID
    return {"result_id": result_id}
```

## 核心概念3：确定性原则

**一句话定义**：工作流的执行路径应该是确定的，避免使用非确定性因素（如当前时间）决定执行路径。

### 代码示例

```python
import time
from langgraph.func import entrypoint
from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()

# ❌ 错误示例：非确定性工作流
@entrypoint(checkpointer=checkpointer)
def bad_workflow(inputs: dict) -> int:
    t0 = inputs["t0"]
    t1 = time.time()  # 非确定性！

    delta_t = t1 - t0

    if delta_t > 1:
        result = slow_task(1).result()
    else:
        result = slow_task(2).result()

    return result

# ✅ 正确示例：确定性工作流
@entrypoint(checkpointer=checkpointer)
def good_workflow(inputs: dict) -> int:
    # 将时间戳作为输入参数传递
    t0 = inputs["t0"]
    t1 = inputs["t1"]  # 确定性！

    delta_t = t1 - t0

    if delta_t > 1:
        result = slow_task(1).result()
    else:
        result = slow_task(2).result()

    return result
```

### 详细解释

**为什么需要确定性？**

1. **可重现性**：相同输入产生相同输出，便于调试
2. **可恢复性**：工作流中断后恢复时行为一致
3. **可测试性**：测试结果稳定可靠
4. **可追踪性**：执行路径可预测

**常见非确定性因素**：

- `time.time()` - 当前时间
- `random.random()` - 随机数
- `uuid.uuid4()` - 随机 UUID
- 外部 API 调用（返回值可能变化）
- 文件系统状态（文件可能被修改）

**如何保证确定性？**

1. **输入参数化**：将时间戳、随机种子作为输入参数
2. **使用固定种子**：`random.seed(42)`
3. **缓存外部调用**：使用 Checkpoint 缓存 API 结果
4. **版本控制**：记录外部资源的版本号

### 在实际应用中的使用

**确定性 RAG 流程**：

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph

class State(TypedDict):
    query: str
    timestamp: float  # 作为输入参数
    random_seed: int  # 作为输入参数
    results: list

def search_node(state: State):
    # 使用固定种子保证确定性
    import random
    random.seed(state["random_seed"])

    # 搜索逻辑
    results = search(state["query"])

    return {"results": results}

graph = StateGraph(State)
graph.add_node("search", search_node)
app = graph.compile()

# 调用时传入时间戳和随机种子
result = app.invoke({
    "query": "LangGraph",
    "timestamp": 1234567890.0,
    "random_seed": 42
})
```

## 核心概念4：副作用封装原则

**一句话定义**：将副作用（写文件、发邮件）封装在 @task 函数中，确保只执行一次。

### 代码示例

```python
from langgraph.func import task, entrypoint, interrupt
from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()

# ❌ 错误示例：副作用在 interrupt 前执行
@entrypoint(checkpointer=checkpointer)
def bad_workflow(inputs: dict) -> int:
    # 这段代码在每次恢复时都会执行！
    send_email(inputs["email"], "Processing started")

    value = interrupt("Continue?")
    return value

# ✅ 正确示例：使用 @task 封装副作用
@task
def send_notification(email: str):
    send_email(email, "Processing started")

@entrypoint(checkpointer=checkpointer)
def good_workflow(inputs: dict) -> int:
    # 副作用封装在 task 中，只执行一次
    send_notification(inputs["email"]).result()

    value = interrupt("Continue?")
    return value
```

### 详细解释

**为什么需要封装副作用？**

1. **避免重复执行**：工作流恢复时不会重复执行副作用
2. **保证一致性**：副作用只执行一次，状态一致
3. **易于测试**：可以 mock @task 函数进行测试
4. **易于监控**：可以单独监控副作用的执行

**常见副作用**：

- 写文件
- 发送邮件
- 发送 HTTP 请求
- 写数据库
- 发送消息队列

### 在实际应用中的使用

**生产环境示例**：

```python
from langgraph.func import task, entrypoint
from langgraph.checkpoint.postgres import PostgresSaver

checkpointer = PostgresSaver(connection_string="postgresql://...")

@task
def write_to_database(data: dict):
    """副作用：写数据库"""
    db.insert("results", data)

@task
def send_webhook(url: str, data: dict):
    """副作用：发送 webhook"""
    requests.post(url, json=data)

@entrypoint(checkpointer=checkpointer)
def workflow(inputs: dict) -> dict:
    # 处理数据
    result = process(inputs["data"])

    # 封装副作用
    write_to_database(result).result()
    send_webhook(inputs["webhook_url"], result).result()

    return result
```

## 核心概念5：错误处理原则

**一句话定义**：使用 RetryPolicy 处理临时性错误，让意外错误向上传播。

### 代码示例

```python
from langgraph.graph import StateGraph
from langgraph.types import RetryPolicy
from typing_extensions import TypedDict

class State(TypedDict):
    query: str
    results: list

# 配置重试策略
retry_policy = RetryPolicy(
    max_attempts=3,
    initial_interval=1.0,
    backoff_factor=2.0,
    retry_on=ConnectionError  # 只重试连接错误
)

def api_call_node(state: State):
    """调用外部 API"""
    try:
        results = external_api.search(state["query"])
        return {"results": results}
    except ValueError:
        # 编程错误，不应该重试
        raise
    except ConnectionError:
        # 临时性错误，会被 RetryPolicy 重试
        raise

# 为节点配置重试策略
graph = StateGraph(State)
graph.add_node("api_call", api_call_node, retry_policy=retry_policy)
app = graph.compile()
```

### 详细解释

**错误分类**：

1. **临时性错误**（应该重试）：
   - 网络错误（ConnectionError）
   - 服务暂时不可用（503 错误）
   - 数据库锁超时

2. **编程错误**（不应该重试）：
   - ValueError, TypeError
   - KeyError, AttributeError
   - 逻辑错误

**RetryPolicy 配置**：

- `max_attempts`: 最大重试次数
- `initial_interval`: 初始重试间隔（秒）
- `backoff_factor`: 退避因子（指数退避）
- `jitter`: 是否添加随机抖动（避免雷鸣群效应）
- `retry_on`: 重试的异常类型

### 在实际应用中的使用

**生产环境示例**：

```python
import sqlite3
from langgraph.types import RetryPolicy

# 数据库查询节点：重试 sqlite3.OperationalError
db_retry_policy = RetryPolicy(
    max_attempts=5,
    initial_interval=0.5,
    backoff_factor=2.0,
    jitter=True,  # 生产环境启用
    retry_on=sqlite3.OperationalError
)

# LLM 调用节点：重试 5xx 错误
llm_retry_policy = RetryPolicy(
    max_attempts=3,
    initial_interval=1.0,
    backoff_factor=2.0,
    retry_on=lambda exc: isinstance(exc, HTTPError) and exc.status_code >= 500
)

graph = StateGraph(State)
graph.add_node("query_db", query_database, retry_policy=db_retry_policy)
graph.add_node("call_llm", call_model, retry_policy=llm_retry_policy)
```

## 核心概念6：模块化原则

**一句话定义**：使用子图组织相关节点，保持单个图的简洁性。

### 代码示例

```python
from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict

class State(TypedDict):
    query: str
    documents: list
    results: str

# 子图1：文档处理
def create_document_processing_subgraph():
    subgraph = StateGraph(State)
    subgraph.add_node("load", load_documents)
    subgraph.add_node("chunk", chunk_documents)
    subgraph.add_node("embed", embed_documents)
    subgraph.add_edge(START, "load")
    subgraph.add_edge("load", "chunk")
    subgraph.add_edge("chunk", "embed")
    subgraph.add_edge("embed", END)
    return subgraph.compile()

# 子图2：搜索处理
def create_search_subgraph():
    subgraph = StateGraph(State)
    subgraph.add_node("search", search_documents)
    subgraph.add_node("rerank", rerank_results)
    subgraph.add_edge(START, "search")
    subgraph.add_edge("search", "rerank")
    subgraph.add_edge("rerank", END)
    return subgraph.compile()

# 主图：组合子图
main_graph = StateGraph(State)
main_graph.add_node("process_docs", create_document_processing_subgraph())
main_graph.add_node("search", create_search_subgraph())
main_graph.add_node("generate", generate_response)

main_graph.add_edge(START, "process_docs")
main_graph.add_edge("process_docs", "search")
main_graph.add_edge("search", "generate")
main_graph.add_edge("generate", END)

app = main_graph.compile()
```

### 详细解释

**为什么需要模块化？**

1. **降低复杂度**：单个图保持简洁，易于理解
2. **提高复用性**：子图可以在不同场景中复用
3. **便于测试**：可以单独测试子图
4. **便于维护**：修改子图不影响主图

### 在实际应用中的使用

**RAG 系统模块化**：

```python
# 文档处理子图
doc_subgraph = create_document_processing_subgraph()

# 检索子图
retrieval_subgraph = create_retrieval_subgraph()

# 生成子图
generation_subgraph = create_generation_subgraph()

# 主图组合
main_graph = StateGraph(State)
main_graph.add_node("docs", doc_subgraph)
main_graph.add_node("retrieval", retrieval_subgraph)
main_graph.add_node("generation", generation_subgraph)
```

## 核心概念7：清晰命名原则

**一句话定义**：使用描述性的节点名称和变量名称，提高代码可读性。

### 代码示例

```python
# ❌ 错误示例：命名不清晰
def n1(s):
    r = api(s["q"])
    return {"r": r}

def n2(s):
    v = check(s["r"])
    return {"v": v}

# ✅ 正确示例：命名清晰
def search_documents_node(state: State):
    """搜索文档"""
    search_results = search_api(state["query"])
    return {"search_results": search_results}

def validate_results_node(state: State):
    """验证搜索结果"""
    is_valid = validate(state["search_results"])
    return {"is_valid": is_valid}
```

### 详细解释

**命名规范**：

1. **节点名称**：使用动词 + 名词（search_documents, validate_results）
2. **变量名称**：使用完整单词（search_results 而非 sr）
3. **状态字段**：使用描述性名称（is_valid 而非 v）
4. **函数名称**：使用动词开头（load_documents, process_query）

### 在实际应用中的使用

**生产环境示例**：

## 总结

7 个核心图设计原则：

1. **单一职责原则**：每个节点只做一件事
2. **状态最小化原则**：只存储必要数据
3. **确定性原则**：避免非确定性因素
4. **副作用封装原则**：使用 @task 封装副作用
5. **错误处理原则**：使用 RetryPolicy 处理临时性错误
6. **模块化原则**：使用子图组织相关节点
7. **清晰命名原则**：使用描述性名称

遵循这些原则，可以构建稳定、可维护、高性能的 LangGraph 应用。
