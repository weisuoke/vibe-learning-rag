# 图基础最佳实践 - 核心概念：错误处理模式

## 概述

错误处理模式是构建稳定、可靠 LangGraph 应用的关键。本文档介绍 LangGraph 中的错误处理机制，包括 RetryPolicy、错误传播、副作用处理等核心概念。

---

## 核心概念1：RetryPolicy 重试策略

**一句话定义**：RetryPolicy 是 LangGraph 提供的声明式重试机制，用于处理临时性错误（网络波动、服务暂时不可用）。

### 代码示例

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy

class State(TypedDict):
    result: str
    attempt_count: int

# 创建重试策略
retry_policy = RetryPolicy(
    max_attempts=3,              # 最大重试次数
    initial_interval=0.1,        # 初始重试间隔(秒)
    backoff_factor=2.0,          # 退避因子(指数退避)
    jitter=True,                 # 添加随机抖动(生产环境推荐)
    retry_on=ValueError,         # 重试的异常类型
)

# 可能失败的节点
def api_call_node(state: State):
    """调用外部 API，可能失败"""
    import random
    if random.random() < 0.7:  # 70% 概率失败
        raise ValueError("API call failed")
    return {"result": "success"}

# 构建图
graph = StateGraph(State)
graph.add_node("api_call", api_call_node, retry_policy=retry_policy)
graph.add_edge(START, "api_call")
graph.add_edge("api_call", END)

app = graph.compile()

# 执行
result = app.invoke({"result": "", "attempt_count": 0})
print(f"Result: {result}")
```

### 详细解释

**RetryPolicy 参数说明**：

1. **max_attempts**：最大重试次数（包括首次尝试）
   - 默认值：3
   - 建议：API 调用 3-5 次，数据库查询 2-3 次

2. **initial_interval**：初始重试间隔（秒）
   - 默认值：0.5
   - 建议：快速操作 0.1-0.5 秒，慢速操作 1-2 秒

3. **backoff_factor**：退避因子（指数退避）
   - 默认值：2.0
   - 计算公式：`interval = initial_interval * (backoff_factor ** (attempt - 1))`
   - 示例：0.1s → 0.2s → 0.4s → 0.8s

4. **jitter**：是否添加随机抖动
   - 默认值：False
   - 生产环境推荐：True（避免雷鸣群效应）

5. **retry_on**：重试条件
   - 单个异常：`ValueError`
   - 多个异常：`(ValueError, KeyError)`
   - 自定义函数：`lambda exc: isinstance(exc, ValueError) and "retry" in str(exc)`

**默认重试策略**：

```python
# 默认会重试：
- ConnectionError (连接错误)
- httpx.HTTPStatusError (5xx 状态码)
- requests.HTTPError (5xx 状态码或无响应)

# 默认不会重试：
- ValueError, TypeError, ArithmeticError (编程错误)
- ImportError, NameError, SyntaxError (语法错误)
- httpx.HTTPStatusError (4xx 状态码)
```

### 在实际应用中的使用

**场景1：LLM API 调用**

```python
from langgraph.types import RetryPolicy
import httpx

# LLM API 调用重试策略
llm_retry_policy = RetryPolicy(
    max_attempts=5,
    initial_interval=1.0,
    backoff_factor=2.0,
    jitter=True,
    retry_on=(httpx.HTTPStatusError, ConnectionError),
)

def call_llm_node(state: State):
    """调用 LLM API"""
    response = llm_client.chat.completions.create(
        model="gpt-4",
        messages=state["messages"]
    )
    return {"response": response.choices[0].message.content}

graph.add_node("call_llm", call_llm_node, retry_policy=llm_retry_policy)
```

**场景2：数据库查询**

```python
import sqlite3

# 数据库查询重试策略
db_retry_policy = RetryPolicy(
    max_attempts=3,
    initial_interval=0.5,
    retry_on=sqlite3.OperationalError,
)

def query_db_node(state: State):
    """查询数据库"""
    conn = sqlite3.connect("data.db")
    cursor = conn.execute("SELECT * FROM documents WHERE id = ?", (state["doc_id"],))
    result = cursor.fetchone()
    conn.close()
    return {"document": result}

graph.add_node("query_db", query_db_node, retry_policy=db_retry_policy)
```

## 核心概念2：错误传播与捕获

**一句话定义**：错误传播是指节点中的异常如何向上传播，错误捕获是指在合适的层级处理错误。

### 代码示例

```python
from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict

class State(TypedDict):
    input: str
    result: str
    error: str | None

# ❌ 错误示例：抑制所有错误
def bad_node(state: State):
    try:
        result = risky_operation(state["input"])
        return {"result": result}
    except Exception:
        pass  # 静默忽略错误
    return {"result": ""}

# ✅ 正确示例：让意外错误向上传播
def good_node(state: State):
    try:
        result = risky_operation(state["input"])
        return {"result": result}
    except ValueError as e:
        # 只捕获预期的错误
        return {"error": f"Invalid input: {e}"}
    # 其他错误向上传播

# ✅ 正确示例：使用 RetryPolicy 处理临时性错误
retry_policy = RetryPolicy(retry_on=ConnectionError)

def api_node(state: State):
    """不需要 try-except，RetryPolicy 会处理"""
    response = api_call(state["input"])
    return {"result": response}

graph = StateGraph(State)
graph.add_node("api", api_node, retry_policy=retry_policy)
graph.add_edge(START, "api")
graph.add_edge("api", END)

app = graph.compile()
```

### 详细解释

**错误处理的三个层级**：

1. **节点层级**：处理预期的业务错误
   ```python
   def validate_node(state: State):
       try:
           data = parse_input(state["input"])
           return {"data": data}
       except ValueError as e:
           # 预期的业务错误
           return {"error": f"Invalid format: {e}"}
   ```

2. **RetryPolicy 层级**：处理临时性错误
   ```python
   retry_policy = RetryPolicy(retry_on=ConnectionError)
   graph.add_node("api", api_node, retry_policy=retry_policy)
   ```

3. **图层级**：处理意外错误
   ```python
   try:
       result = app.invoke(input_data)
   except Exception as e:
       # 记录日志、发送告警
       logger.error(f"Graph execution failed: {e}")
       raise
   ```

**最佳实践**：

- ✅ 只捕获预期的特定错误
- ✅ 使用 RetryPolicy 处理临时性错误
- ✅ 让意外错误向上传播
- ❌ 不要使用 `except Exception: pass`
- ❌ 不要在节点中处理所有错误

### 在实际应用中的使用

**RAG 系统错误处理**：

```python
class RAGState(TypedDict):
    query: str
    documents: list
    response: str
    error: str | None

def retrieve_node(state: RAGState):
    """检索文档，处理预期错误"""
    try:
        docs = vector_store.similarity_search(state["query"], k=5)
        if not docs:
            return {"error": "No relevant documents found"}
        return {"documents": docs}
    except ValueError as e:
        # 预期的业务错误
        return {"error": f"Invalid query: {e}"}
    # ConnectionError 由 RetryPolicy 处理

# 配置重试策略
retrieve_retry_policy = RetryPolicy(
    max_attempts=3,
    retry_on=ConnectionError,
)

graph.add_node("retrieve", retrieve_node, retry_policy=retrieve_retry_policy)
```

## 核心概念3：副作用处理与幂等性

**一句话定义**：副作用是指修改外部状态的操作（写文件、发送邮件），幂等性是指多次执行产生相同结果。

### 代码示例

```python
from langgraph.func import task, entrypoint
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()

# ❌ 错误示例：副作用在 interrupt 前执行
@entrypoint(checkpointer=checkpointer)
def bad_workflow(inputs: dict) -> int:
    # 这段代码在恢复工作流时会再次执行
    with open("output.txt", "a") as f:
        f.write("Processing...\n")  # 每次恢复都会追加
    
    value = interrupt("Approve?")
    return value

# ✅ 正确示例：将副作用封装在 task 中
@task
def write_to_file():
    """副作用封装在 task 中，只执行一次"""
    with open("output.txt", "w") as f:
        f.write("Processing completed\n")

@entrypoint(checkpointer=checkpointer)
def good_workflow(inputs: dict) -> int:
    # 副作用在 interrupt 之后执行
    value = interrupt("Approve?")
    
    # 使用 task 确保副作用只执行一次
    write_to_file().result()
    
    return value
```

### 详细解释

**副作用的三种类型**：

1. **文件操作**：写文件、删除文件
2. **网络操作**：发送邮件、调用 webhook
3. **数据库操作**：插入记录、更新状态

**副作用处理的最佳实践**：

1. **使用 @task 装饰器**：
   ```python
   @task
   def send_email():
       email_service.send(to="user@example.com", subject="Notification")
   
   @entrypoint(checkpointer=checkpointer)
   def workflow(inputs: dict):
       value = interrupt("Approve?")
       send_email().result()  # 只执行一次
       return value
   ```

2. **在 interrupt 之后执行副作用**：
   ```python
   @entrypoint(checkpointer=checkpointer)
   def workflow(inputs: dict):
       # ❌ 错误：在 interrupt 前执行副作用
       # log_to_db("approval_requested")
       
       value = interrupt("Approve?")
       
       # ✅ 正确：在 interrupt 后执行副作用
       log_to_db("approval_granted")
       
       return value
   ```

3. **使用幂等操作**：
   ```python
   def idempotent_insert(record_id, data):
       """幂等的插入操作"""
       # 使用 UPSERT 或检查是否存在
       existing = db.get(record_id)
       if not existing:
           db.insert(record_id, data)
   ```

**常见反模式**：

```python
# ❌ 反模式1：在 interrupt 前追加列表
@entrypoint(checkpointer=checkpointer)
def bad_workflow(inputs: dict):
    db.append_to_history(inputs["user_id"], "approval_requested")  # 每次恢复都会追加
    value = interrupt("Approve?")
    return value

# ❌ 反模式2：非确定性工作流
@entrypoint(checkpointer=checkpointer)
def bad_workflow(inputs: dict):
    t0 = inputs["t0"]
    t1 = time.time()  # 非确定性
    
    if t1 - t0 > 1:
        result = slow_task(1).result()
    else:
        result = slow_task(2).result()
    
    value = interrupt("question")
    return {"result": result, "value": value}
```

### 在实际应用中的使用

**场景：审批流程**

```python
from langgraph.func import task, entrypoint
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()

@task
def send_approval_email(user_email: str):
    """发送审批邮件（副作用）"""
    email_service.send(
        to=user_email,
        subject="Approval Granted",
        body="Your request has been approved."
    )

@task
def log_approval(user_id: str, decision: str):
    """记录审批日志（副作用）"""
    db.insert("approval_logs", {
        "user_id": user_id,
        "decision": decision,
        "timestamp": datetime.now()
    })

@entrypoint(checkpointer=checkpointer)
def approval_workflow(inputs: dict) -> dict:
    user_id = inputs["user_id"]
    user_email = inputs["user_email"]
    
    # 等待人工审批
    decision = interrupt("Approve this request?")
    
    # 副作用在 interrupt 之后执行
    if decision == "approved":
        send_approval_email(user_email).result()
        log_approval(user_id, "approved").result()
    else:
        log_approval(user_id, "rejected").result()
    
    return {"decision": decision}
```

## 最佳实践总结

### 1. RetryPolicy 使用指南

**何时使用 RetryPolicy**：
- ✅ API 调用（LLM、搜索引擎、数据库）
- ✅ 网络请求（HTTP、gRPC）
- ✅ 临时性错误（连接超时、服务暂时不可用）

**何时不使用 RetryPolicy**：
- ❌ 编程错误（ValueError、TypeError）
- ❌ 业务逻辑错误（权限不足、数据不存在）
- ❌ 4xx HTTP 错误（客户端错误）

**配置建议**：

| 场景 | max_attempts | initial_interval | backoff_factor | jitter |
|------|--------------|------------------|----------------|--------|
| LLM API | 5 | 1.0s | 2.0 | True |
| 数据库查询 | 3 | 0.5s | 2.0 | False |
| 文件操作 | 2 | 0.1s | 1.5 | False |
| 外部 API | 3 | 1.0s | 2.0 | True |

### 2. 错误处理层级

```python
# 层级1：节点层级 - 处理预期的业务错误
def node(state):
    try:
        result = business_logic(state)
        return {"result": result}
    except ValueError as e:
        return {"error": str(e)}

# 层级2：RetryPolicy - 处理临时性错误
retry_policy = RetryPolicy(retry_on=ConnectionError)
graph.add_node("node", node, retry_policy=retry_policy)

# 层级3：图层级 - 处理意外错误
try:
    result = app.invoke(input_data)
except Exception as e:
    logger.error(f"Unexpected error: {e}")
    raise
```

### 3. 副作用处理清单

- [ ] 副作用封装在 @task 函数中
- [ ] 副作用在 interrupt 之后执行
- [ ] 使用幂等操作（UPSERT、检查是否存在）
- [ ] 避免非确定性操作（time.time()、random）
- [ ] 使用 checkpointer 确保副作用只执行一次

## 完整示例：RAG 系统错误处理

```python
from typing_extensions import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy
from langgraph.func import task
import operator

class RAGState(TypedDict):
    query: str
    documents: Annotated[list, operator.add]
    response: str
    error: str | None

# 配置重试策略
retrieve_retry = RetryPolicy(
    max_attempts=3,
    initial_interval=0.5,
    retry_on=ConnectionError,
)

llm_retry = RetryPolicy(
    max_attempts=5,
    initial_interval=1.0,
    backoff_factor=2.0,
    jitter=True,
)

# 节点定义
def retrieve_node(state: RAGState):
    """检索文档"""
    try:
        docs = vector_store.similarity_search(state["query"], k=5)
        if not docs:
            return {"error": "No relevant documents found"}
        return {"documents": docs}
    except ValueError as e:
        return {"error": f"Invalid query: {e}"}

def generate_node(state: RAGState):
    """生成回答"""
    if state.get("error"):
        return {"response": f"Error: {state['error']}"}
    
    try:
        context = "\n".join([doc.page_content for doc in state["documents"]])
        response = llm.invoke(f"Context: {context}\n\nQuestion: {state['query']}")
        return {"response": response}
    except ValueError as e:
        return {"error": f"Generation failed: {e}"}

@task
def log_query(query: str, response: str):
    """记录查询日志（副作用）"""
    db.insert("query_logs", {
        "query": query,
        "response": response,
        "timestamp": datetime.now()
    })

def log_node(state: RAGState):
    """记录日志"""
    log_query(state["query"], state["response"]).result()
    return {}

# 构建图
graph = StateGraph(RAGState)
graph.add_node("retrieve", retrieve_node, retry_policy=retrieve_retry)
graph.add_node("generate", generate_node, retry_policy=llm_retry)
graph.add_node("log", log_node)

graph.add_edge(START, "retrieve")
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", "log")
graph.add_edge("log", END)

app = graph.compile()

# 执行
try:
    result = app.invoke({"query": "What is LangGraph?", "documents": []})
    print(f"Response: {result['response']}")
except Exception as e:
    logger.error(f"RAG system failed: {e}")
    raise
```

## 常见错误与解决方案

### 错误1：过度使用 try-except

**问题**：
```python
def node(state):
    try:
        result = operation(state)
        return {"result": result}
    except Exception:
        return {"result": ""}  # 静默忽略所有错误
```

**解决方案**：
```python
def node(state):
    try:
        result = operation(state)
        return {"result": result}
    except ValueError as e:
        # 只捕获预期的错误
        return {"error": str(e)}
    # 其他错误向上传播
```

### 错误2：在 interrupt 前执行副作用

**问题**：
```python
@entrypoint(checkpointer=checkpointer)
def workflow(inputs):
    db.append("logs", "processing")  # 每次恢复都会追加
    value = interrupt("Approve?")
    return value
```

**解决方案**：
```python
@task
def log_processing():
    db.append("logs", "processing")

@entrypoint(checkpointer=checkpointer)
def workflow(inputs):
    value = interrupt("Approve?")
    log_processing().result()  # 只执行一次
    return value
```

### 错误3：重试编程错误

**问题**：
```python
retry_policy = RetryPolicy(retry_on=Exception)  # 重试所有错误
```

**解决方案**：
```python
retry_policy = RetryPolicy(
    retry_on=(ConnectionError, TimeoutError)  # 只重试临时性错误
)
```

## 总结

LangGraph 的错误处理模式包括三个核心概念：

1. **RetryPolicy**：声明式重试机制，处理临时性错误
2. **错误传播与捕获**：在合适的层级处理错误，让意外错误向上传播
3. **副作用处理**：使用 @task 装饰器和 checkpointer 确保副作用只执行一次

**关键原则**：
- 使用 RetryPolicy 处理临时性错误
- 只捕获预期的特定错误
- 让意外错误向上传播
- 将副作用封装在 @task 中
- 在 interrupt 之后执行副作用
- 使用幂等操作避免重复执行

通过遵循这些最佳实践，可以构建稳定、可靠的 LangGraph 应用。
