# 第一性原理

> 回到事物最基本的真理，从源头思考多向量检索的本质

---

## 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题，而不是通过类比或经验。

**应用到多向量检索**：
- 不是"多向量检索像多条件搜索"
- 而是"多向量检索是什么？为什么需要它？它解决了什么根本问题？"

---

## 多向量检索的第一性原理

### 1. 最基础的定义

**多向量检索 = 在多个向量字段上同时进行检索，并融合结果**

仅此而已！没有更基础的了。

**形式化定义**：
```
Collection C 包含多个向量字段：{V₁, V₂, ..., Vₙ}
查询向量：{Q₁, Q₂, ..., Qₙ}
多向量检索：Search(Q₁, V₁) ⊕ Search(Q₂, V₂) ⊕ ... ⊕ Search(Qₙ, Vₙ)
其中 ⊕ 表示结果融合操作
```

**核心特征**：
- 一个 Collection 包含多个向量字段
- 每个向量字段独立检索
- 检索结果需要融合

---

### 2. 为什么需要多向量检索？

**核心问题：单向量表示的局限性**

#### 问题场景 1：多模态数据

假设你在构建一个电商搜索系统：
- 商品有图片
- 商品有文本描述
- 用户可能用图片搜索，也可能用文字搜索

**不使用多向量的问题**：
```python
# 方案 1：只用图片向量
schema = CollectionSchema([
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="image_vector", dtype=DataType.FLOAT_VECTOR, dim=512),
    FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=1000)
])

# 问题：用户输入文字查询时，无法利用文本语义
results = collection.search(
    data=[image_query_vector],  # 只能用图片向量
    anns_field="image_vector",
    limit=10
)
# 结果：文本相关性差，召回率低
```

**问题分析**：
1. **信息丢失**：只用图片向量，丢失了文本语义信息
2. **召回不全**：文本相关的商品无法被召回
3. **用户体验差**：用户输入文字时，搜索效果不佳

**根本原因**：
- 单一向量无法同时表示多种模态的信息
- 图片向量擅长视觉特征，文本向量擅长语义理解
- 强行用一个向量表示所有信息会导致信息压缩和丢失

---

#### 问题场景 2：语义 + 关键词匹配

假设你在构建一个文档问答系统：
- 用户问题："如何配置 Kubernetes 的 ingress？"
- 文档库包含大量技术文档

**不使用多向量的问题**：
```python
# 只用稠密向量（语义理解）
results = collection.search(
    data=[dense_query_vector],
    anns_field="dense_vector",
    limit=10
)
# 问题：可能召回语义相关但不包含关键词的文档
# 例如：召回了关于"网络配置"的文档，但没有提到"ingress"
```

**问题分析**：
1. **语义漂移**：稠密向量可能召回语义相关但不精确的文档
2. **关键词缺失**：用户明确提到的关键词（如"ingress"）可能被忽略
3. **精准度不足**：无法保证召回的文档包含用户关心的关键词

**根本原因**：
- 稠密向量擅长语义理解，但可能忽略精确关键词
- 稀疏向量（如 BM25）擅长关键词匹配，但缺乏语义理解
- 单一向量类型无法兼顾语义和精确匹配

---

### 3. 多向量检索的三层价值

#### 价值 1：多模态信息融合

**原理**：不同模态的信息用不同的向量表示，检索时综合考虑

**示例**：
```python
from pymilvus import CollectionSchema, FieldSchema, DataType

# 定义多向量字段
schema = CollectionSchema([
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="image_vector", dtype=DataType.FLOAT_VECTOR, dim=512),
    FieldSchema(name="text_vector", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=200)
])

# 多向量检索
from pymilvus import AnnSearchRequest, RRFRanker

image_req = AnnSearchRequest(
    data=[image_query_vector],
    anns_field="image_vector",
    param={"metric_type": "IP"},
    limit=10
)

text_req = AnnSearchRequest(
    data=[text_query_vector],
    anns_field="text_vector",
    param={"metric_type": "IP"},
    limit=10
)

# 融合结果
results = client.hybrid_search(
    collection_name="products",
    reqs=[image_req, text_req],
    ranker=RRFRanker(),
    limit=10
)
```

**价值量化**：
- 召回率提升：30-50%（同时利用图片和文本信息）
- 用户满意度提升：40%（更准确的搜索结果）

---

#### 价值 2：语义 + 精确匹配

**原理**：稠密向量捕获语义，稀疏向量捕获关键词，两者互补

**示例**：
```python
# 定义稠密向量 + 稀疏向量
schema = CollectionSchema([
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="dense_vector", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="sparse_vector", dtype=DataType.SPARSE_FLOAT_VECTOR),
    FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=2000)
])

# BM25 混合检索
sparse_req = AnnSearchRequest(
    data=["Kubernetes ingress configuration"],  # 直接传入文本
    anns_field="sparse_vector",
    param={"metric_type": "BM25"},
    limit=10
)

dense_req = AnnSearchRequest(
    data=[dense_query_vector],
    anns_field="dense_vector",
    param={"metric_type": "IP"},
    limit=10
)

# 融合结果
results = client.hybrid_search(
    collection_name="docs",
    reqs=[sparse_req, dense_req],
    ranker=RRFRanker(),
    limit=10
)
```

**价值量化**：
- 精准度提升：25-40%（关键词匹配 + 语义理解）
- 召回率提升：20-30%（兼顾精确和模糊匹配）

---

#### 价值 3：多粒度信息表示

**原理**：同一对象的不同粒度信息用不同向量表示

**示例**：
```python
# 定义标题向量 + 内容向量
schema = CollectionSchema([
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="title_vector", dtype=DataType.FLOAT_VECTOR, dim=384),
    FieldSchema(name="content_vector", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=200),
    FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=5000)
])

# 多粒度检索
title_req = AnnSearchRequest(
    data=[query_vector],
    anns_field="title_vector",
    param={"metric_type": "IP"},
    limit=10
)

content_req = AnnSearchRequest(
    data=[query_vector],
    anns_field="content_vector",
    param={"metric_type": "IP"},
    limit=10
)

# 加权融合（标题权重更高）
from pymilvus import WeightedRanker

results = client.hybrid_search(
    collection_name="articles",
    reqs=[title_req, content_req],
    ranker=WeightedRanker(2.0, 1.0),  # 标题权重 2.0，内容权重 1.0
    limit=10
)
```

**价值量化**：
- 精准度提升：15-25%（标题匹配更精准）
- 用户体验提升：30%（更符合用户意图）

### 4. 从第一性原理推导结果融合

**推理链**：

```
1. 前提：多个向量字段独立检索，产生多个结果列表
   ↓
2. 推导：需要将多个结果列表合并为一个最终结果
   ↓
3. 推导：不同向量字段的相似度分数不可直接比较
   （例如：图片向量的分数 0.8 vs 文本向量的分数 0.7，哪个更好？）
   ↓
4. 推导：需要一个统一的融合策略来合并结果
   ↓
5. 推导：融合策略需要考虑两个问题：
   a. 如何归一化不同向量字段的分数？
   b. 如何平衡不同向量字段的重要性？
   ↓
6. 推导：两种融合策略
   a. RRFRanker：基于排名的融合（不依赖分数）
   b. WeightedRanker：基于加权的融合（需要归一化分数）
   ↓
7. 最终：Ranker 机制
```

**Ranker 的本质**：
- **归一化**：将不同向量字段的分数映射到统一的尺度
- **加权**：根据向量字段的重要性调整权重
- **排序**：根据融合后的分数重新排序

---

### 5. 从第一性原理推导 RRF 和 WeightedRanker

#### RRF（Reciprocal Rank Fusion）的推导

**问题**：不同向量字段的分数不可比较

**推理**：
```
1. 前提：分数不可比较，但排名是可比较的
   ↓
2. 推导：使用排名而非分数来融合结果
   ↓
3. 推导：排名越靠前，重要性越高
   ↓
4. 推导：使用倒数排名作为分数：score = 1 / (k + rank)
   （k 是平滑参数，避免除零）
   ↓
5. 推导：对所有向量字段的倒数排名求和
   ↓
6. 最终：RRF 算法
```

**RRF 公式**：
```
score(doc) = Σ [1 / (k + rank_i(doc))]
其中：
- rank_i(doc) 是文档在第 i 个向量字段检索结果中的排名
- k 是平滑参数（默认 60）
```

**RRF 的优势**：
- **无需归一化**：不依赖分数，只依赖排名
- **自动平衡**：不需要手动设置权重
- **鲁棒性强**：对异常分数不敏感

**RRF 的劣势**：
- **无法精确控制权重**：所有向量字段权重相同
- **排名信息丢失**：只考虑排名，不考虑分数差异

---

#### WeightedRanker 的推导

**问题**：不同向量字段的重要性不同

**推理**：
```
1. 前提：某些向量字段比其他字段更重要
   （例如：标题向量比内容向量更重要）
   ↓
2. 推导：需要为每个向量字段设置权重
   ↓
3. 推导：使用加权求和来融合分数
   ↓
4. 推导：score = w₁ × score₁ + w₂ × score₂ + ... + wₙ × scoreₙ
   ↓
5. 最终：WeightedRanker 算法
```

**WeightedRanker 公式**：
```
score(doc) = Σ [w_i × score_i(doc)]
其中：
- score_i(doc) 是文档在第 i 个向量字段的相似度分数
- w_i 是第 i 个向量字段的权重
```

**WeightedRanker 的优势**：
- **精确控制权重**：可以根据业务需求调整权重
- **保留分数信息**：考虑分数差异，更精细

**WeightedRanker 的劣势**：
- **需要归一化**：不同向量字段的分数需要在同一尺度
- **权重调优复杂**：需要实验确定最佳权重

---

### 6. 一句话总结第一性原理

**多向量检索是通过在多个向量字段上独立检索并融合结果，来克服单向量表示的局限性，实现多模态、多粒度、语义+精确的综合检索。**

## 从第一性原理推导 Ranker 选择策略

### 推理链

```
1. 前提：需要融合多个向量字段的检索结果
   ↓
2. 推导：需要选择合适的 Ranker
   ↓
3. 推导：Ranker 选择取决于两个因素：
   a. 是否知道各向量字段的重要性？
   b. 是否需要精确控制权重？
   ↓
4. 推导：两种场景
   a. 不确定权重 → RRFRanker
   b. 确定权重 → WeightedRanker
   ↓
5. 最终：Ranker 选择策略
```

### Ranker 选择策略

| 场景 | 推荐 Ranker | 原因 |
|------|-------------|------|
| 多模态检索（图片+文本） | RRFRanker | 不确定图片和文本的相对重要性 |
| BM25 + 稠密向量 | RRFRanker | 语义和关键词同等重要 |
| 标题 + 内容向量 | WeightedRanker | 标题通常比内容更重要 |
| 多语言检索 | RRFRanker | 不同语言的重要性相同 |
| 用户画像 + 内容向量 | WeightedRanker | 可根据业务调整权重 |

**选择建议**：
- **默认使用 RRFRanker**：简单、鲁棒、无需调参
- **需要精确控制时使用 WeightedRanker**：业务明确知道权重
- **先用 RRF 验证效果，再用 Weighted 优化**：渐进式优化

## 核心洞察

### 1. 多向量检索的本质

**多向量检索不是技术实现，而是一种信息表示的思维方式**：
- 单向量 = 单一视角
- 多向量 = 多维度视角
- 融合 = 综合决策

### 2. Ranker 的创新

**Ranker 是 Milvus 多向量检索的核心创新**：
- **RRFRanker**：基于排名的民主投票
- **WeightedRanker**：基于权重的专家决策
- **可扩展**：支持自定义 Ranker（如 Decay Ranker）

### 3. 检索质量的根本

**检索质量的根本是信息的完整性**：
- 不是优化算法（虽然也重要）
- 而是捕获更多维度的信息
- 多向量检索通过多维度表示实现这一目标
