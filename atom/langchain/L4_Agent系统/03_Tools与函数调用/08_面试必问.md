# 面试必问

> 掌握 LangChain Tools 与函数调用的 2 道高频面试题——从"能说出来"到"说得出彩"，关键在于分层思维、源码理解和实战经验。

---

## 问题 1: "LangChain 中 Tool 的函数调用流程是怎样的？"

### 普通回答（❌ 不出彩）

"就是定义一个工具，然后让模型调用它，返回结果。"

**问题**：
- 太笼统，没有体现对流程的理解
- 没有区分"模型决策"和"实际执行"
- 没有提到消息协议和关联机制
- 面试官会追问："模型是怎么知道有哪些工具的？调用结果怎么回传？"

### 出彩回答（✅ 推荐）

> **Tool 的函数调用流程分三个层次：定义层、协议层、执行层。**
>
> **1. 定义层——告诉系统"有哪些工具"**
>
> LangChain 提供三种定义 Tool 的方式，复杂度递增：
> - `@tool` 装饰器：最简洁，90% 场景首选。自动从函数名提取 `name`，从 docstring 提取 `description`，从类型注解推断 `args_schema`
> - `StructuredTool.from_function()`：需要自定义 Pydantic Schema 时使用，比如参数要加枚举约束或范围验证
> - `BaseTool` 子类：需要内部状态管理（连接池、缓存、调用计数）时使用
>
> 三种方式本质相同，核心三要素是 `name` + `description` + `args_schema`。其中 **description 是 LLM 选择工具的唯一依据**——模型看不到你的函数实现代码，它只能看到这三个字段。
>
> ```python
> from langchain_core.tools import tool
>
> @tool
> def search_kb(query: str, top_k: int = 5) -> str:
>     """在公司知识库中搜索技术文档。
>     适用：技术规范、API文档、架构设计等问题。
>     不适用：实时新闻、个人问题。"""
>     return search_vector_store(query, top_k=top_k)
>
> # 装饰器自动提取：
> # name = "search_kb"
> # description = "在公司知识库中搜索技术文档..."
> # args_schema = {"query": str, "top_k": int}
> ```
>
> **2. 协议层——告诉模型"你可以调用什么"**
>
> `bind_tools()` 将 Tool 的 JSON Schema 注入模型的上下文。底层调用 `convert_to_openai_tool()` 做跨提供商格式转换（OpenAI function calling / Anthropic tool use / Google tool config），开发者不需要关心不同提供商的格式差异。
>
> 模型返回的是 `AIMessage.tool_calls`——一个结构化的 JSON 列表，**不是直接执行函数**。这是"声明式"设计：模型只说"我想调用 search_kb，参数是 {'query': 'RAG架构', 'top_k': 3}"，但它自己不执行。
>
> ```python
> from langchain_openai import ChatOpenAI
>
> model = ChatOpenAI(model="gpt-4o")
> model_with_tools = model.bind_tools([search_kb])
>
> response = model_with_tools.invoke("公司的微服务架构是怎样的？")
> # response.tool_calls = [{
> #     "name": "search_kb",
> #     "args": {"query": "微服务架构", "top_k": 5},
> #     "id": "call_abc123"
> # }]
> # 注意：此时函数还没有被执行！
> ```
>
> **3. 执行层——外部系统执行函数，结果回传模型**
>
> 外部系统（Agent 循环或开发者代码）根据 `tool_calls` 执行对应的 Tool，将结果封装为 `ToolMessage`。关键是 `tool_call_id` 字段——它将执行结果与模型的调用请求一一关联，模型据此继续推理或生成最终回答。
>
> ```python
> from langchain_core.messages import ToolMessage
>
> # 执行 Tool
> for tc in response.tool_calls:
>     result = search_kb.invoke(tc["args"])
>     # 构造 ToolMessage，tool_call_id 关联请求和响应
>     tool_msg = ToolMessage(content=result, tool_call_id=tc["id"])
> ```
>
> 这三步形成了 Agent 的核心循环：Think（模型决策调用哪个工具）→ Act（外部执行工具）→ Observe（结果回传模型）。模型可以根据工具结果决定继续调用其他工具，或者生成最终回答。
>
> **源码层面**，`convert_to_openai_tool()`（位于 `langchain_core/utils/function_calling.py`，813 行）负责将 LangChain 的 Tool 定义转换为各提供商的格式，这是跨模型兼容的关键。

### 为什么这个回答出彩？

| 维度 | 分析 |
|------|------|
| **分层思维** | 定义层→协议层→执行层，结构清晰，面试官一听就知道你理解了全貌 |
| **关键洞察** | 强调"模型不执行函数，只声明意图"——这是很多人搞混的点 |
| **源码理解** | 提到 `convert_to_openai_tool()` 和跨提供商格式转换，体现深度 |
| **实战细节** | `tool_call_id` 关联机制、description 是唯一依据——这些是踩过坑才知道的 |
| **代码佐证** | 每一层都有代码示例，不是空谈概念 |

### 面试官可能的追问

**追问 1："如果模型一次返回多个 tool_calls 怎么办？"**

> 模型可以在一次响应中返回多个 tool_calls（并行调用）。每个 tool_call 有独立的 `id`，执行后各自构造 ToolMessage，通过 `tool_call_id` 一一对应。Agent 循环会把所有 ToolMessage 一起发回模型。
>
> ```python
> # 模型可能同时请求调用两个工具
> # response.tool_calls = [
> #     {"name": "search_kb", "args": {"query": "Q3营收"}, "id": "call_001"},
> #     {"name": "search_kb", "args": {"query": "Q2营收"}, "id": "call_002"},
> # ]
>
> tool_messages = []
> for tc in response.tool_calls:
>     result = search_kb.invoke(tc["args"])
>     tool_messages.append(ToolMessage(content=result, tool_call_id=tc["id"]))
>
> # 所有 ToolMessage 一起发回模型
> final_response = model_with_tools.invoke(
>     [human_msg, response] + tool_messages
> )
> ```

**追问 2："bind_tools() 底层做了什么？"**

> `bind_tools()` 做了两件事：
> 1. 将每个 Tool 的 `name`、`description`、`args_schema` 转换为 JSON Schema 格式
> 2. 将这些 Schema 作为 `tools` 参数绑定到模型的请求配置中
>
> 本质上是 `model.bind(tools=[...])` 的语法糖。每次调用模型时，这些 Schema 会作为系统级参数发送给 API，告诉模型"你有这些工具可以用"。

**追问 3："tool_call_id 如果对不上会怎样？"**

> 模型会报错或产生混乱的推理。`tool_call_id` 是模型区分多个并行工具调用结果的唯一标识。如果 id 对不上，模型无法将结果与对应的调用请求关联，可能导致张冠李戴——把 A 工具的结果当成 B 工具的结果来推理。

## 问题 2: "@tool 装饰器和 BaseTool 子类有什么区别？什么时候用哪个？"

### 普通回答（❌ 不出彩）

"@tool 简单，BaseTool 复杂。简单的用 @tool，复杂的用 BaseTool。"

**问题**：
- 没有说清楚"简单"和"复杂"的具体含义
- 没有提到 StructuredTool（中间方案）
- 没有揭示三者的本质关系
- 面试官会追问："什么叫复杂？给个具体例子？"

### 出彩回答（✅ 推荐）

> **三种方式本质都创建 BaseTool 实例，区别在灵活度和代码量的权衡。**
>
> 先说本质关系：`@tool` 装饰器内部调用的就是 `StructuredTool.from_function()`，而 `StructuredTool` 继承自 `BaseTool`。所以 `@tool` 是语法糖，三种方式最终都是 BaseTool 实例。
>
> ```
> 源码继承关系：
> RunnableSerializable
>     └── BaseTool                    # tools/base.py（抽象基类）
>             └── StructuredTool      # tools/structured.py
>                     └── @tool 装饰器内部调用 StructuredTool.from_function()
> ```
>
> **1. @tool 装饰器（90% 场景）**
>
> 一行代码搞定，自动从函数签名推断 Schema。适合逻辑简单、参数类型注解清晰的场景。
>
> ```python
> from langchain_core.tools import tool
>
> @tool
> def get_weather(city: str) -> str:
>     """查询指定城市的实时天气信息。
>     适用于用户询问天气相关问题。"""
>     return call_weather_api(city)
>
> # 自动推断：name="get_weather", description=docstring, args_schema 从类型注解生成
> ```
>
> **优势**：代码量最少，3-5 行搞定
> **局限**：参数描述依赖 docstring 或 `parse_docstring=True`，无法添加复杂验证规则
>
> **2. StructuredTool.from_function()（约 10% 场景）**
>
> 当你需要精确控制参数 Schema 时使用——比如参数要加枚举约束、范围验证、详细描述，或者需要同时提供同步和异步实现。
>
> ```python
> from langchain_core.tools import StructuredTool
> from pydantic import BaseModel, Field
> from enum import Enum
>
> class SearchType(str, Enum):
>     VECTOR = "vector"
>     KEYWORD = "keyword"
>     HYBRID = "hybrid"
>
> class RAGSearchInput(BaseModel):
>     query: str = Field(description="搜索查询文本")
>     search_type: SearchType = Field(
>         default=SearchType.VECTOR,
>         description="检索类型：vector/keyword/hybrid"
>     )
>     top_k: int = Field(default=5, ge=1, le=20, description="返回数量，1-20")
>
> rag_tool = StructuredTool.from_function(
>     func=rag_search,
>     name="rag_search",
>     description="在 RAG 知识库中执行检索",
>     args_schema=RAGSearchInput,  # 精确的参数描述和验证
> )
> ```
>
> **优势**：Pydantic Field 支持枚举、范围、正则等丰富验证；参数描述更精确，LLM 生成参数的准确率更高
> **局限**：不支持内部状态管理
>
> **3. BaseTool 子类（极少场景，< 5%）**
>
> 只在需要**内部状态**或**完全自定义行为**时使用。典型场景：连接池管理、调用计数与去重、自定义错误处理、权限检查。
>
> ```python
> from langchain_core.tools import BaseTool, ToolException
> from pydantic import BaseModel, Field
> from typing import Type
>
> class QueryInput(BaseModel):
>     query: str = Field(description="搜索查询")
>
> class CachedSearchTool(BaseTool):
>     """带缓存和调用统计的搜索工具"""
>     name: str = "cached_search"
>     description: str = "搜索知识库，自动缓存重复查询"
>     args_schema: Type[BaseModel] = QueryInput
>
>     # 内部状态——这是 @tool 和 StructuredTool 做不到的
>     cache: dict = {}
>     call_count: int = 0
>
>     def _run(self, query: str) -> str:
>         self.call_count += 1
>         if query in self.cache:
>             return f"[缓存] {self.cache[query]}"
>         result = do_search(query)
>         self.cache[query] = result
>         return result
> ```
>
> **优势**：完全控制——状态管理、错误处理、初始化逻辑
> **局限**：代码量最多（15-30 行），大多数场景是过度设计
>
> **选择原则：从最简单开始，只在需要时升级。**
>
> ```
> 需要定义一个 Tool
>     │
>     ├─ 函数逻辑简单，类型注解清晰？ → @tool ✅
>     │
>     ├─ 需要枚举/范围约束/精确参数描述？ → StructuredTool ✅
>     │
>     └─ 需要内部状态或自定义错误处理？ → BaseTool 子类 ✅
> ```
>
> 实际项目中，一个 Agent 的工具集经常混合使用三种方式：简单的搜索和计算用 `@tool`，RAG 检索工具用 `StructuredTool`（因为参数多且需要精确描述），数据库连接工具用 `BaseTool`（因为要管理连接池）。

### 为什么这个回答出彩？

| 维度 | 分析 |
|------|------|
| **揭示本质** | 指出 @tool 内部就是调用 StructuredTool.from_function()，三者是同一棵继承树 |
| **具体比例** | 90% / 10% / <5% 的使用比例，体现实战经验而非纸上谈兵 |
| **代码对比** | 三种方式各给一个代码示例，面试官一目了然 |
| **决策流程** | 给出清晰的选择标准，而不是模糊的"简单用这个，复杂用那个" |
| **设计哲学** | "从最简单开始，只在需要时升级"——体现工程思维 |

### 面试官可能的追问

**追问 1："@tool 装饰器的 parse_docstring 参数是做什么的？"**

> `parse_docstring=True` 让 `@tool` 自动解析 Google-style docstring，提取每个参数的描述写入 Schema。这样不需要单独定义 Pydantic BaseModel，就能让 LLM 看到参数的详细说明。
>
> ```python
> @tool(parse_docstring=True)
> def create_doc(title: str, content: str, tags: list[str]) -> str:
>     """创建新文档。
>
>     Args:
>         title: 文档标题，不超过100字符
>         content: 文档正文内容
>         tags: 标签列表，用于分类检索
>     """
>     return f"文档 '{title}' 已创建"
>
> # Schema 中自动包含参数描述：
> # title.description = "文档标题，不超过100字符"
> # content.description = "文档正文内容"
> # tags.description = "标签列表，用于分类检索"
> ```
>
> 这是 `@tool` 和 `StructuredTool` 之间的折中方案——不需要写 Pydantic Model，但能获得精确的参数描述。

**追问 2："BaseTool 的 handle_tool_error 是怎么工作的？"**

> `handle_tool_error=True` 时，Tool 执行中抛出的 `ToolException` 不会中断 Agent 循环，而是将错误信息作为 ToolMessage 的 content 回传给模型。模型可以根据错误信息决定重试、换个参数、或者换个工具。
>
> ```python
> class SafeDBTool(BaseTool):
>     handle_tool_error: bool = True  # 启用错误处理
>
>     def _run(self, sql: str) -> str:
>         if "DROP" in sql.upper():
>             raise ToolException("禁止执行 DROP 操作")
>         return execute_query(sql)
>
> # 模型收到的 ToolMessage.content = "禁止执行 DROP 操作"
> # 模型可以据此调整策略，而不是整个 Agent 崩溃
> ```
>
> 也可以传入一个函数 `handle_tool_error=lambda e: f"工具出错：{e}"`，自定义错误消息的格式。

**追问 3："能不能把一个 Retriever 或 Chain 变成 Tool？"**

> 可以。Tool 不限于普通函数——任何实现了 Runnable 接口的对象都可以转为 Tool：
>
> - **Retriever → Tool**：用 `create_retriever_tool(retriever, name, description)`，这是 RAG 中最常用的模式
> - **Chain → Tool**：用 `chain.as_tool(name, description)`（LangChain 0.3+）
> - **任意 Runnable → Tool**：用 `runnable.as_tool(name, description)`
> - **Agent → Tool**：用 `@tool` 包装 Agent 的 invoke 调用，实现 Agent 嵌套
>
> 本质上 Tool 是一个协议（BaseTool 接口），不是一种特定的实现。任何满足 `name` + `description` + `args_schema` + `invoke()` 的对象都可以成为 Tool。

## 面试准备建议

### 理解层次对照

| 层次 | 表现 | 面试官评价 |
|------|------|-----------|
| **表面** | "定义工具，模型调用，返回结果" | 不及格——没有区分声明和执行 |
| **中等** | 能说出 bind_tools → tool_calls → ToolMessage 流程 | 及格——知道流程但缺乏深度 |
| **深入** | 分层讲解 + 源码理解 + 跨提供商格式转换 | 良好——有技术深度 |
| **出彩** | 分层 + 源码 + 实战经验 + 设计哲学 | 优秀——能独立设计 Tool 系统 |

### 准备实际案例

面试前准备一个你用 Tool 系统解决实际问题的案例，包含：

1. **场景**：你在什么项目中用了 Tool？（如 RAG 知识库问答）
2. **选型**：为什么选择某种定义方式？（如用 StructuredTool 因为需要枚举参数）
3. **踩坑**：遇到过什么问题？（如 description 写得模糊导致工具选择错误）
4. **优化**：如何解决的？（如重写 description，加入适用/不适用场景说明）

**示例**：

> "我在 RAG 项目中构建了一个多工具 Agent，包含知识库检索、网络搜索和计算器三个工具。最初用 @tool 定义所有工具，但发现模型经常把'公司内部问题'路由到网络搜索。排查后发现是 description 写得太模糊——知识库工具只写了'搜索文档'，没有说明适用范围。重写 description 后加入'适用于公司内部技术规范、API 文档'等关键词，工具选择准确率从 70% 提升到 95%。"

### 关键数据速记

| 数据点 | 用途 |
|--------|------|
| @tool 覆盖 90% 场景 | 说明选型时优先用最简单的方式 |
| description 是 LLM 选择工具的唯一依据 | 强调 description 的重要性 |
| 工具 3-5 个时选择准确率 ~90-95% | 说明工具数量要精简 |
| 每个工具 Schema 约 100-200 tokens | 说明工具过多的成本问题 |
| `convert_to_openai_tool()` 813 行 | 体现对源码的了解 |

## 快速复习卡片

### 问题 1 核心要点

```
函数调用三层流程：
├── 定义层：@tool / StructuredTool / BaseTool → name + description + args_schema
├── 协议层：bind_tools() → JSON Schema 注入 → 模型返回 tool_calls（声明式，不执行）
└── 执行层：外部执行 → ToolMessage(tool_call_id) → 回传模型继续推理
```

### 问题 2 核心要点

```
三种方式的本质关系：
├── @tool 装饰器 → 内部调用 StructuredTool.from_function()（语法糖）
├── StructuredTool → 继承自 BaseTool（支持自定义 Schema）
└── BaseTool → 抽象基类（支持状态管理）

选择原则：从最简单开始，只在需要时升级
├── 90% → @tool
├── 10% → StructuredTool（需要精确 Schema）
└── <5% → BaseTool（需要内部状态）
```
