## 4.1 Schema 设计三原则

**一句话：** TypedDict 定义结构，Annotated 绑定 Reducer，Pydantic 做运行时验证

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph.message import add_messages
from langchain_core.messages import AnyMessage
import operator

# 原则 1：每个字段都有明确类型注解
# 原则 2：用 Annotated 声明合并策略
# 原则 3：字段命名体现用途，不要用 data、info 这种模糊名
class AgentState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]  # 对话历史，按 ID 合并
    search_results: Annotated[list[str], operator.add]   # 检索结果，累积追加
    current_query: str                                    # 当前查询，覆盖更新
    iteration: int                                        # 迭代次数，覆盖更新
```

**什么时候用 Pydantic？** 当你需要运行时校验（比如外部输入、用户提交的数据）时：

```python
from pydantic import BaseModel, field_validator

class ValidatedState(BaseModel):
    query: str
    max_results: int = 5

    @field_validator("max_results")
    @classmethod
    def check_range(cls, v):
        if not 1 <= v <= 100:
            raise ValueError("max_results 必须在 1-100 之间")
        return v
```

**简单工作流用 TypedDict（零开销），对外接口用 Pydantic（有校验）。**

## 4.2 Reducer 选型速查

**一句话：** 根据字段的更新语义选择 Reducer，选错了数据会丢失或重复

| 场景 | 写法 | 效果 |
|------|------|------|
| 覆盖更新 | `field: str` | 新值替换旧值 |
| 列表累积 | `field: Annotated[list, operator.add]` | 新列表追加到旧列表 |
| 消息管理 | `field: Annotated[list, add_messages]` | 按 ID 合并，相同 ID 覆盖 |
| 字典合并 | `field: Annotated[dict, operator.or_]` | 新键值对合并到旧字典 |
| 自定义逻辑 | `field: Annotated[T, my_reducer]` | 你说了算 |

```python
import operator
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph.message import add_messages

class RAGState(TypedDict):
    messages: Annotated[list, add_messages]        # 消息：按 ID 合并
    documents: Annotated[list[str], operator.add]  # 文档：累积追加
    metadata: Annotated[dict, operator.or_]        # 元数据：字典合并
    status: str                                     # 状态：直接覆盖
```

## 4.3 不可变更新模式

**一句话：** 永远不要直接修改 state 对象，返回新字典作为更新

```python
# ❌ 错误：直接修改 state（会引发隐蔽 bug，尤其在并发和检查点场景）
def bad_node(state: AgentState) -> dict:
    state["messages"].append(new_msg)  # 直接修改了原始状态！
    state["iteration"] += 1
    return state

# ✅ 正确：返回新的字典，只包含要更新的字段
def good_node(state: AgentState) -> dict:
    return {
        "messages": [new_msg],              # Reducer 会自动合并
        "iteration": state["iteration"] + 1  # 返回新值，不修改原值
    }
```

**为什么？** LangGraph 的检查点和状态快照依赖不可变性。直接修改 state 会导致：
- 检查点保存的是被污染的状态
- 时间旅行（回退到历史节点）失效
- 并行节点之间互相干扰

## 4.4 状态最小化

**一句话：** 只存必要数据，派生数据运行时计算，临时数据用完即弃

```python
from typing import Annotated
from typing_extensions import TypedDict
import operator

# ❌ 臃肿：存了大量可派生的冗余数据
class BloatedState(TypedDict):
    documents: Annotated[list[str], operator.add]
    document_count: int          # 冗余：len(documents) 就能算
    all_text: str                # 冗余："\n".join(documents) 就能算
    raw_api_response: dict       # 临时数据：用完就不需要了

# ✅ 精简：只存核心数据
class LeanState(TypedDict):
    messages: Annotated[list, operator.add]
    documents: Annotated[list[str], operator.add]
    query: str
```

**消息列表要有上限**——这是最常见的状态膨胀来源：

```python
def trim_messages_reducer(left: list, right: list) -> list:
    """保留最近 20 条消息，防止状态无限增长"""
    combined = left + right
    return combined[-20:]  # 只保留最近 20 条

class ChatState(TypedDict):
    messages: Annotated[list, trim_messages_reducer]
```

## 4.5 Schema 演进

**一句话：** 新字段用 Optional + 默认值，保持向后兼容

```python
from typing import Optional

# v1：初始版本
class StateV1(TypedDict):
    messages: Annotated[list, add_messages]
    query: str

# v2：新增字段，旧检查点依然能加载
class StateV2(TypedDict):
    messages: Annotated[list, add_messages]
    query: str
    language: str            # 新字段：有默认值的通道不会破坏旧数据
    max_tokens: int          # 新字段：同上
```

**关键规则：**
- 新增字段：安全，旧检查点中该字段为空，节点里做好空值处理即可
- 删除字段：危险，旧检查点中仍有该字段的数据，可能报错
- 修改类型：危险，反序列化会失败

## 这些知识足以

掌握以上 5 个要点，你就能：

1. **设计清晰的状态 Schema** — TypedDict + Annotated，字段职责分明
2. **选择合适的 Reducer** — 覆盖、累积、合并，一张表搞定
3. **避免状态突变 bug** — 返回新字典，不碰原始 state
4. **控制状态大小** — 只存必要数据，消息列表设上限
5. **安全演进 Schema** — 新字段加默认值，不破坏旧检查点
