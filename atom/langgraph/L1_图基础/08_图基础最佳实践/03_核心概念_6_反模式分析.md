# 核心概念 6：反模式分析

> 本文档分析 LangGraph 图基础开发中的常见反模式，帮助你避开这些陷阱

---

## 什么是反模式？

**反模式（Anti-Pattern）** 是看似合理但实际会导致问题的解决方案。

在 LangGraph 开发中，反模式通常源于：
- 对框架机制的误解
- 从其他框架迁移时的惯性思维
- 忽视状态化工作流的特殊性
- 缺少生产环境经验

---

## 反模式分类

### 按影响范围分类

| 类别 | 影响 | 典型反模式 |
|------|------|-----------|
| **状态管理** | 数据一致性、性能 | 状态中存储大对象、直接修改状态 |
| **副作用处理** | 重复执行、数据污染 | Interrupt 前执行副作用 |
| **错误处理** | 稳定性、调试困难 | 抑制意外错误、缺少重试策略 |
| **图设计** | 可维护性、性能 | 过度复杂、节点粒度不当 |
| **生产环境** | 可靠性、成本 | 缺少监控、不清理 Checkpoint |

---

## 核心反模式详解

### 反模式 1：在 Interrupt 前执行副作用 ❌

#### 问题描述

在 `interrupt()` 之前执行副作用（写文件、发送邮件、数据库写入）会导致恢复工作流时重复执行。

#### 错误示例

```python
from langgraph.func import entrypoint, interrupt
from langgraph.checkpoint.memory import InMemorySaver

@entrypoint(checkpointer=InMemorySaver())
def approval_workflow(inputs: dict) -> dict:
    # ❌ 错误：在 interrupt 前写入数据库
    db.append_to_history(inputs["user_id"], "approval_requested")

    # 等待人工审批
    approved = interrupt("请审批此操作")

    if approved:
        return {"status": "approved"}
    return {"status": "rejected"}

# 问题：每次恢复工作流时，都会再次写入 "approval_requested"
```

**运行结果**：
```
# 第一次执行
db.history = ["approval_requested"]

# 恢复工作流后
db.history = ["approval_requested", "approval_requested"]  # 重复了！
```

#### 为什么错？

LangGraph 的 Checkpoint 机制会在 `interrupt()` 处保存状态。恢复时，会从 `interrupt()` 之前重新执行代码，导致副作用重复执行。

#### 为什么人们容易这样错？

- **惯性思维**：在传统编程中，代码只执行一次
- **误解 Checkpoint**：以为 Checkpoint 会记录所有副作用
- **缺少测试**：没有测试恢复场景

#### 正确做法

**方案 1：使用 @task 装饰器**

```python
from langgraph.func import entrypoint, interrupt, task
from langgraph.checkpoint.memory import InMemorySaver

@task
def write_to_db(user_id: str, message: str):
    """副作用封装在 task 中，只执行一次"""
    db.append_to_history(user_id, message)

@entrypoint(checkpointer=InMemorySaver())
def approval_workflow(inputs: dict) -> dict:
    # ✅ 正确：副作用封装在 task 中
    write_to_db(inputs["user_id"], "approval_requested").result()

    approved = interrupt("请审批此操作")

    if approved:
        return {"status": "approved"}
    return {"status": "rejected"}
```

**方案 2：在 interrupt 之后执行副作用**

```python
@entrypoint(checkpointer=InMemorySaver())
def approval_workflow(inputs: dict) -> dict:
    # 先 interrupt
    approved = interrupt("请审批此操作")

    # ✅ 正确：在 interrupt 之后执行副作用
    if approved:
        db.append_to_history(inputs["user_id"], "approved")
        return {"status": "approved"}
    else:
        db.append_to_history(inputs["user_id"], "rejected")
        return {"status": "rejected"}
```

#### 实际影响

- **数据重复**：数据库中出现重复记录
- **成本增加**：重复发送邮件、API 调用
- **状态不一致**：业务逻辑错误

### 反模式 2：非确定性工作流 ❌

#### 问题描述

使用 `time.time()`、`random.random()` 等非确定性因素决定执行路径，导致恢复时行为不一致。

#### 错误示例

```python
import time
from langgraph.func import entrypoint, interrupt
from langgraph.checkpoint.memory import InMemorySaver

@entrypoint(checkpointer=InMemorySaver())
def time_based_workflow(inputs: dict) -> dict:
    t0 = inputs["start_time"]
    t1 = time.time()  # ❌ 非确定性

    delta_t = t1 - t0

    if delta_t > 1:
        result = slow_task()
        value = interrupt("慢任务完成，请确认")
    else:
        result = fast_task()
        value = interrupt("快任务完成，请确认")

    return {"result": result, "value": value}
```

**问题场景**：
```
# 第一次执行（delta_t = 0.5）
执行 fast_task()

# 恢复工作流（delta_t = 2.0）
执行 slow_task()  # 路径改变了！
```

#### 为什么错？

恢复工作流时，`time.time()` 返回的是当前时间，不是原始执行时的时间，导致执行路径改变。

#### 为什么人们容易这样错？

- **忽视恢复场景**：只考虑首次执行
- **误用时间戳**：把时间戳当作业务逻辑的一部分
- **缺少确定性意识**：不了解 Checkpoint 的重放机制

#### 正确做法

```python
from langgraph.func import entrypoint, interrupt
from langgraph.checkpoint.memory import InMemorySaver

@entrypoint(checkpointer=InMemorySaver())
def deterministic_workflow(inputs: dict) -> dict:
    # ✅ 正确：将时间戳作为输入参数
    t0 = inputs["start_time"]
    t1 = inputs["current_time"]  # 从输入获取，而不是 time.time()

    delta_t = t1 - t0

    if delta_t > 1:
        result = slow_task()
        value = interrupt("慢任务完成，请确认")
    else:
        result = fast_task()
        value = interrupt("快任务完成，请确认")

    return {"result": result, "value": value}

# 调用时传入时间戳
result = deterministic_workflow.invoke({
    "start_time": 1234567890,
    "current_time": 1234567891
})
```

#### 实际影响

- **行为不一致**：恢复后执行不同的路径
- **调试困难**：无法复现问题
- **测试失败**：单元测试不稳定

### 反模式 3：抑制意外错误 ❌

#### 问题描述

捕获并忽略所有异常，隐藏关键问题，使调试困难。

#### 错误示例

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    email: str
    status: str

def send_email(state: State):
    try:
        email_service.send(state["email"])
        return {"status": "sent"}
    except Exception:
        pass  # ❌ 错误：忽略所有异常
        return {"status": "unknown"}

graph = StateGraph(State)
graph.add_node("send_email", send_email)
graph.add_edge(START, "send_email")
graph.add_edge("send_email", END)
app = graph.compile()
```

**问题**：
- 网络错误被忽略
- 配置错误被忽略
- 无法知道失败原因

#### 为什么错？

- **隐藏问题**：错误被吞掉，无法发现根本原因
- **状态不一致**：返回 "unknown" 但没有记录错误
- **调试困难**：生产环境出问题时无法定位

#### 为什么人们容易这样错？

- **过度防御**：担心程序崩溃
- **懒惰处理**：不想处理具体错误
- **误解异常**：以为所有异常都应该捕获

#### 正确做法

**方案 1：让意外错误向上传播**

```python
def send_email(state: State):
    try:
        email_service.send(state["email"])
        return {"status": "sent"}
    except EmailServiceError as e:
        # ✅ 只捕获预期的错误
        print(f"邮件发送失败: {e}")
        return {"status": "failed", "error": str(e)}
    # ✅ 其他错误向上传播
```

**方案 2：使用 RetryPolicy 处理临时性错误**

```python
from langgraph.types import RetryPolicy

retry_policy = RetryPolicy(
    max_attempts=3,
    initial_interval=1.0,
    retry_on=EmailServiceError  # 只重试特定错误
)

graph = StateGraph(State)
graph.add_node("send_email", send_email, retry_policy=retry_policy)
graph.add_edge(START, "send_email")
graph.add_edge("send_email", END)
app = graph.compile()
```

#### 实际影响

- **问题隐藏**：生产环境出错但不知道
- **调试困难**：无法追踪错误原因
- **用户体验差**：错误没有反馈

### 反模式 4：在状态中存储大对象 ❌

#### 问题描述

在状态中存储大对象（文件内容、模型权重）导致序列化失败和性能问题。

#### 错误示例

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    file_content: bytes  # ❌ 存储大文件内容
    model: object        # ❌ 存储模型对象

def load_file(state: State):
    # ❌ 将整个文件内容存入状态
    with open("large_file.pdf", "rb") as f:
        content = f.read()  # 可能是几百 MB
    return {"file_content": content}

def process_file(state: State):
    content = state["file_content"]
    # 处理文件...
    return {"status": "processed"}
```

**问题**：
- Checkpoint 序列化失败（对象太大）
- 内存占用高
- 状态传递慢

#### 为什么错？

LangGraph 会序列化状态到 Checkpoint，大对象会导致：
- 序列化时间长
- 存储空间大
- 反序列化失败

#### 为什么人们容易这样错？

- **方便性**：直接存储最简单
- **误解状态**：以为状态可以存储任何东西
- **缺少性能意识**：没有考虑序列化成本

#### 正确做法

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    file_path: str    # ✅ 只存储文件路径
    file_hash: str    # ✅ 存储文件哈希
    status: str

def load_file(state: State):
    # ✅ 只存储引用，不存储内容
    file_path = "large_file.pdf"
    file_hash = compute_hash(file_path)
    return {
        "file_path": file_path,
        "file_hash": file_hash
    }

def process_file(state: State):
    # ✅ 从文件路径读取内容
    with open(state["file_path"], "rb") as f:
        content = f.read()
    # 处理文件...
    return {"status": "processed"}
```

**最佳实践**：
- ✅ 状态中只存储引用（路径、ID、URL）
- ✅ 大对象存储在外部（文件系统、数据库、对象存储）
- ✅ 使用哈希验证数据完整性

#### 实际影响

- **序列化失败**：Checkpoint 保存失败
- **内存溢出**：OOM 错误
- **性能下降**：状态传递慢

### 反模式 5：过度复杂的图设计 ❌

#### 问题描述

图结构过于复杂，包含大量节点和条件分支，难以理解和维护。

#### 错误示例

```python
from langgraph.graph import StateGraph, START, END

# ❌ 过度复杂的图
graph = StateGraph(State)

# 添加 20 个节点
for i in range(20):
    graph.add_node(f"node_{i}", lambda s: s)

# 添加复杂的条件路由
def complex_router(state: State):
    if state["x"] > 10:
        if state["y"] < 5:
            if state["z"] == "a":
                return "node_1"
            elif state["z"] == "b":
                return "node_2"
            else:
                return "node_3"
        else:
            # 更多嵌套条件...
            pass
    # 更多条件...
    pass

graph.add_conditional_edges("start", complex_router)
```

**问题**：
- 难以理解执行流程
- 难以测试
- 难以维护

#### 为什么错？

- **可读性差**：无法一眼看出流程
- **测试困难**：需要覆盖大量分支
- **维护成本高**：修改一处影响多处

#### 为什么人们容易这样错？

- **一次性设计**：把所有逻辑放在一个图中
- **缺少模块化**：不知道如何拆分
- **过度优化**：追求性能而牺牲可读性

#### 正确做法

**使用子图模块化**

```python
from langgraph.graph import StateGraph, START, END

# ✅ 拆分成多个子图
def create_validation_subgraph():
    """验证子图"""
    subgraph = StateGraph(State)
    subgraph.add_node("check_format", check_format)
    subgraph.add_node("check_content", check_content)
    subgraph.add_edge(START, "check_format")
    subgraph.add_edge("check_format", "check_content")
    subgraph.add_edge("check_content", END)
    return subgraph.compile()

def create_processing_subgraph():
    """处理子图"""
    subgraph = StateGraph(State)
    subgraph.add_node("transform", transform)
    subgraph.add_node("enrich", enrich)
    subgraph.add_edge(START, "transform")
    subgraph.add_edge("transform", "enrich")
    subgraph.add_edge("enrich", END)
    return subgraph.compile()

# ✅ 主图简洁清晰
main_graph = StateGraph(State)
main_graph.add_node("validate", create_validation_subgraph())
main_graph.add_node("process", create_processing_subgraph())
main_graph.add_edge(START, "validate")
main_graph.add_edge("validate", "process")
main_graph.add_edge("process", END)
app = main_graph.compile()
```

**最佳实践**：
- ✅ 单个图不超过 10 个节点
- ✅ 条件分支不超过 3 层嵌套
- ✅ 使用子图实现模块化
- ✅ 每个节点职责单一

#### 实际影响

- **开发效率低**：新人难以理解
- **Bug 多**：复杂逻辑容易出错
- **重构困难**：牵一发而动全身

### 反模式 6：不清理过期 Checkpoint ❌

#### 问题描述

不定期清理过期的 Checkpoint，导致存储爆炸。

#### 错误示例

```python
from langgraph.checkpoint.postgres import PostgresSaver

# ❌ 只创建 Checkpoint，从不清理
checkpointer = PostgresSaver(
    connection_string="postgresql://user:pass@localhost/dbname"
)

graph = StateGraph(State)
# ... 添加节点 ...
app = graph.compile(checkpointer=checkpointer)

# 运行数千次，Checkpoint 表越来越大
for i in range(10000):
    app.invoke({"x": i}, config={"configurable": {"thread_id": f"thread_{i}"}})
```

**问题**：
- 数据库存储快速增长
- 查询性能下降
- 成本增加

#### 为什么错？

每次执行都会创建新的 Checkpoint，不清理会导致：
- 存储空间耗尽
- 数据库性能下降
- 备份时间增加

#### 为什么人们容易这样错？

- **忽视清理**：只关注功能开发
- **缺少监控**：不知道存储增长速度
- **误解持久化**：以为所有 Checkpoint 都需要保留

#### 正确做法

```python
from langgraph.checkpoint.postgres import PostgresSaver
from datetime import datetime, timedelta

checkpointer = PostgresSaver(
    connection_string="postgresql://user:pass@localhost/dbname"
)

graph = StateGraph(State)
# ... 添加节点 ...
app = graph.compile(checkpointer=checkpointer)

# ✅ 定期清理过期 Checkpoint
def cleanup_old_checkpoints():
    """清理 7 天前的 Checkpoint"""
    cutoff_time = datetime.now() - timedelta(days=7)

    # 使用 SQL 清理
    with checkpointer.conn.cursor() as cur:
        cur.execute("""
            DELETE FROM checkpoints
            WHERE created_at < %s
        """, (cutoff_time,))
        checkpointer.conn.commit()

    print(f"清理了 {cur.rowcount} 个过期 Checkpoint")

# ✅ 使用定时任务清理
import schedule
schedule.every().day.at("02:00").do(cleanup_old_checkpoints)
```

**最佳实践**：
- ✅ 设置 Checkpoint 保留策略（如 7 天）
- ✅ 使用定时任务自动清理
- ✅ 监控存储使用情况
- ✅ 对重要会话单独备份

#### 实际影响

- **存储爆炸**：数据库空间耗尽
- **性能下降**：查询变慢
- **成本增加**：存储费用上升

## 反模式速查表

| 反模式 | 典型错误 | 正确做法 | 影响 |
|--------|---------|---------|------|
| **Interrupt 前副作用** | 直接写数据库 | 使用 @task 或 interrupt 后执行 | 数据重复 |
| **非确定性工作流** | 使用 time.time() | 时间戳作为输入参数 | 行为不一致 |
| **抑制意外错误** | except Exception: pass | 只捕获预期错误 + RetryPolicy | 问题隐藏 |
| **状态存储大对象** | 存储文件内容 | 只存储引用（路径、ID） | 序列化失败 |
| **过度复杂图** | 20+ 节点单图 | 使用子图模块化 | 难以维护 |
| **不清理 Checkpoint** | 从不清理 | 定期清理 + 保留策略 | 存储爆炸 |

---

## 如何避免反模式？

### 1. 开发阶段

- ✅ 阅读官方文档和最佳实践
- ✅ 参考社区案例
- ✅ 进行代码审查
- ✅ 编写单元测试

### 2. 测试阶段

- ✅ 测试恢复场景（Checkpoint + Resume）
- ✅ 测试错误场景
- ✅ 性能测试（状态大小、序列化时间）
- ✅ 压力测试（并发、长时间运行）

### 3. 生产阶段

- ✅ 监控关键指标（错误率、响应时间、存储使用）
- ✅ 设置告警
- ✅ 定期审查日志
- ✅ 建立清理策略

---

## 总结

**反模式的本质**：看似合理但实际有害的做法

**避免反模式的关键**：
1. 理解 LangGraph 的 Checkpoint 机制
2. 重视状态设计和副作用处理
3. 充分测试恢复场景
4. 建立生产环境最佳实践

**记住**：反模式不是错误，而是经验教训。学习反模式可以帮助你更快地掌握 LangGraph 的正确用法。
