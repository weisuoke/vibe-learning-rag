# SDK 与平台运行超时

> **核心概念 6/6**
> **关键词**：client timeout、RunStatus timeout、APITimeoutError、platform budget

---

## 为什么很多“LangGraph 超时”其实根本不是图层问题？

因为用户感知到的 timeout 往往发生在更外层：

- SDK 请求超时；
- 平台请求 / 部署超时；
- 前端 fetch / SSE 断开；
- 反向代理窗口耗尽。

如果你把这些全部算成“图超时”，排障就会彻底混乱。

---

## 一句话定义

**SDK 与平台运行超时，指的是发生在图执行器之外、但会直接影响调用方体验与 run 生命周期的 timeout 机制。**

---

## SDK timeout 是什么？

在 `langgraph_sdk` 的 sync / async client 里，都能看到：

```python
timeout=(
    httpx.Timeout(timeout)
    if timeout is not None
    else httpx.Timeout(connect=5, read=300, write=300, pool=5)
)
```

这说明 SDK 的 `timeout` 本质上是：

- HTTP client budget；
- 面向请求生命周期；
- 不是面向 graph step。

默认值也很值得记：
- connect=5s
- read=300s
- write=300s
- pool=5s

[来源: sourcecode/langgraph/libs/sdk-py/langgraph_sdk/_sync/client.py]

## `APITimeoutError` 是什么？

SDK 还定义了：

```python
class APITimeoutError(APIConnectionError):
    def __init__(self, request: httpx.Request) -> None:
        super().__init__(message="Request timed out.", request=request)
```

它表达的是：

> “客户端请求 LangGraph API 时超时了。”

不是：
- 某个 node 超时；
- 某个 step 超时；
- 子图超时。

这是非常典型的“外层 transport timeout”。

---

## 平台运行状态里的 `timeout`

SDK schema 中还有一个更高层语义：

```python
RunStatus = Literal[
    "pending", "running", "error", "success", "timeout", "interrupted"
]
```

并且文档注释写得很明确：

```python
- "timeout": The run exceeded its time limit.
```

这说明到了平台化运行阶段，“timeout” 不再只是进程内抛错，而是 run 生命周期的一种显式状态。

这很像：
- 进程内异常 = 低层实现细节
- run status = 对外产品语义

---

## 为什么这很重要？

因为平台环境里你经常拿不到 Python traceback，只能拿到 run status。

此时：
- `TimeoutError` 更像内部事件；
- `RunStatus = timeout` 才是调用方真正消费的结果。

如果文档里不区分这两层，团队很容易把排障方向带偏。

## 生产环境常见错位

### 错位 1：图没超时，SDK 先超时

例如：
- graph step 预算 30 秒；
- 整体 run 需要 90 秒；
- SDK read timeout 60 秒；

结果图可能还在正常推进，但客户端已经报 `APITimeoutError`。

### 错位 2：图没超时，前端先断了

这在长运行任务里非常常见。

### 错位 3：部署超时与执行超时混淆

搜索结果里还有 2025 年 Platform 部署 300 秒未就绪的讨论。

这根本不是图 step budget，而是平台准备窗口。

[来源: atom/langgraph/L3_工作流编排/09_超时控制/reference/search_生产部署超时_02.md]

## 那么长任务应该怎么办？

如果一个 run 天生就可能很长，通常不要让调用方一直同步等待。

更合理的模式是：

1. 启动 run；
2. 返回 `run_id`；
3. 前端轮询或订阅状态；
4. 读 `pending / running / timeout / success / interrupted`；
5. 必要时 resume 或查看 checkpoint。

这类模式和官方 durable execution 路线是高度一致的。

---

## 一个简单分层图

```text
前端 / 调用方
  ↓
SDK timeout / APITimeoutError
  ↓
平台 run status timeout
  ↓
graph step_timeout
  ↓
node I/O timeout
```

越往里越接近根因；
越往外越接近用户体验。

## 实战建议

### 1. SDK timeout 要和产品交互模型匹配

如果前端本来就是同步按钮点击，那么 SDK timeout 不应无限大；
如果是后台任务模型，SDK timeout 可以更短，因为客户端不必一直等完整结果。

### 2. 对长任务优先用 async run + status polling

不要试图用更长的 client read timeout 来拯救不合适的同步交互。

### 3. 把 timeout 状态变成产品语义

不要只显示“网络错误”，最好区分：
- 连接超时；
- 平台 run timeout；
- 图执行超时；
- 用户中断。

### 4. 文档和日志里明确标注层级

记录 timeout 时，建议带上标签：
- `layer=node`
- `layer=step`
- `layer=sdk`
- `layer=platform`

**LangGraph 生态里的“timeout”至少有四层含义：node、step、SDK、platform。只有把它们拆开，生产排障才会真正变简单。**
