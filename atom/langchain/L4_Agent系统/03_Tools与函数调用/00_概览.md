# Tools与函数调用

> Tool 是 Agent 的"手"——LLM 负责思考决策，Tool 负责执行动作。通过 Tool 定义、函数调用协议和工具选择策略，Agent 能够调用搜索引擎、数据库、API 等外部工具来完成任务。

---

## 知识点定位

- **所属层级**: L4_Agent系统 - 第 3 个知识点
- **重要程度**: 极高（Agent 与外部世界交互的唯一通道）
- **核心文件**: `langchain_core/tools/` 目录（base.py、convert.py、structured.py、simple.py、render.py）
- **消息协议**: `langchain_core/messages/tool.py`（ToolCall、ToolMessage）
- **格式转换**: `langchain_core/utils/function_calling.py`（OpenAI/Anthropic 格式转换）

### 为什么这个知识点重要？

LLM 本身只能生成文本，不能查天气、不能查数据库、不能调 API。**Tool 是 LLM 从"只会说"变成"能做事"的关键桥梁。**

一个 Agent 的能力上限，不取决于 LLM 有多聪明，而取决于它能调用多少、多好的 Tool。

理解 Tool 系统，你就能：
1. 让 Agent 调用任意外部服务（搜索、数据库、API）
2. 控制 Agent 能做什么、不能做什么（权限边界）
3. 优化 Agent 的工具选择准确率（Schema 设计）
4. 构建生产级的 Tool 调用链路（参数验证、错误处理、结果回传）

### 与前两个知识点的关系

```
01_create_agent标准抽象  →  定义 Agent 的创建方式和 Middleware 系统
02_AgentExecutor执行循环  →  定义 Agent 的 Think→Act→Observe 循环
03_Tools与函数调用       →  定义 Agent "Act" 阶段的具体执行能力 ← 你在这里
```

Agent 循环中的 "Act" 步骤，就是调用 Tool。没有 Tool，Agent 的循环就是空转。

│       └──────────────────┴────────────────────┘         │

### 函数调用完整流程

```
用户: "北京今天天气怎么样？"
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 1. Tool 定义                                         │
│    @tool                                             │
│    def get_weather(city: str) -> str:                │
│        """查询城市天气"""                              │
│        return call_weather_api(city)                  │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│ 2. 绑定工具到模型                                     │
│    model_with_tools = model.bind_tools([get_weather]) │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│ 3. 模型生成 tool_calls（不是直接回答）                 │
│    AIMessage(                                        │
│      content="",                                     │
│      tool_calls=[{                                   │
│        "name": "get_weather",                        │
│        "args": {"city": "北京"},                      │
│        "id": "call_abc123"                           │
│      }]                                              │
│    )                                                 │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│ 4. 执行 Tool，获取结果                                │
│    result = get_weather.invoke({"city": "北京"})      │
│    → "北京今天晴，25°C"                               │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│ 5. 构造 ToolMessage 回传给模型                        │
│    ToolMessage(                                      │
│      content="北京今天晴，25°C",                      │
│      tool_call_id="call_abc123"                      │
│    )                                                 │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│ 6. 模型基于工具结果生成最终回答                        │
│    AIMessage("北京今天天气晴朗，气温25°C，适合出行。")  │
└─────────────────────────────────────────────────────┘
```

这个 6 步流程是所有 Tool 调用的核心链路。理解它，就理解了 Agent 如何"动手做事"。

## 核心概念速览

本知识点包含 **6 个核心概念**，覆盖 Tool 系统的完整知识面：

### 概念 1: Tool 定义三种方式

LangChain 提供三种定义 Tool 的方式，复杂度递增：

| 方式 | 适用场景 | 代码量 | 灵活性 |
|------|----------|--------|--------|
| `@tool` 装饰器 | 简单函数包装（推荐） | 最少 | 中等 |
| `StructuredTool.from_function()` | 需要自定义 Schema | 中等 | 较高 |
| `BaseTool` 子类 | 复杂工具、需要状态管理 | 最多 | 最高 |

```python
# 方式 1: @tool 装饰器（90% 场景用这个就够了）
@tool
def search(query: str) -> str:
    """搜索互联网内容"""
    return do_search(query)

# 方式 2: StructuredTool（需要自定义参数 Schema）
from langchain_core.tools import StructuredTool
tool = StructuredTool.from_function(
    func=search_func,
    name="search",
    description="搜索互联网",
    args_schema=SearchInput,  # Pydantic BaseModel
)

# 方式 3: BaseTool 子类（需要完全控制）
from langchain_core.tools import BaseTool
class SearchTool(BaseTool):
    name = "search"
    description = "搜索互联网"
    def _run(self, query: str) -> str:
        return do_search(query)
```

### 概念 2: 函数调用协议

`bind_tools()` 将 Tool 的 Schema 转换为模型能理解的格式（OpenAI function calling / Anthropic tool use），模型返回 `tool_calls` 而非直接文本回答。

### 概念 3: ToolCall 与 ToolMessage 消息流

完整的消息链路：`HumanMessage → AIMessage(tool_calls) → ToolMessage → AIMessage(最终回答)`。每个 ToolMessage 必须通过 `tool_call_id` 与对应的 ToolCall 关联。

### 概念 4: Tool Schema 与参数验证

Tool 的参数通过 Pydantic BaseModel 或 JSON Schema 定义。LangChain 支持从函数签名自动推断 Schema，也支持 `parse_docstring=True` 从 Google-style docstring 提取参数描述。

### 概念 5: 工具选择策略

当 Agent 拥有大量工具时，需要动态选择相关工具。LangChain 1.0 提供 `LLMToolSelectorMiddleware`，用一个小模型预筛选工具，减少主模型的选择负担。

### 概念 6: 高级 Tool 特性

- `response_format="content_and_artifact"`: Tool 同时返回内容和工件（如文件、图表）
- `InjectedToolArg`: 自动注入运行时参数（如用户 ID），不暴露给 LLM
- `extras`: 提供商特定字段（如 Anthropic 的 `cache_control`）
- `return_direct`: Tool 结果直接返回用户，跳过 LLM 二次处理

## 与 RAG 和 AI Agent 的关系

### RAG 场景中的 Tool

在 RAG 系统中，Tool 让检索和生成变得可编程：

```
用户: "帮我查一下公司Q3财报中的营收数据，并和Q2做对比"
    │
    ▼
Agent Think: 需要两个工具——文档检索 + 数据计算
    │
    ▼
Agent Act: 调用 retriever_tool("Q3财报 营收")
    → 返回: "Q3营收 5.2亿元"
    │
    ▼
Agent Act: 调用 retriever_tool("Q2财报 营收")
    → 返回: "Q2营收 4.8亿元"
    │
    ▼
Agent Act: 调用 calculator_tool("(5.2-4.8)/4.8*100")
    → 返回: "环比增长 8.33%"
    │
    ▼
Agent Finish: "Q3营收5.2亿元，较Q2的4.8亿元环比增长8.33%。"
```

Tool 让 RAG 从"只能检索"升级为"检索 + 计算 + 多步推理"。

### AI Agent 开发中的核心地位

Tool 是所有 Agent 框架的共同基础设施：

| 框架 | Tool 抽象 | 函数调用方式 |
|------|-----------|-------------|
| LangChain | BaseTool / @tool | bind_tools() + tool_calls |
| LlamaIndex | FunctionTool | tool_spec |
| AutoGPT | Command | function_calling |
| CrewAI | Tool | agent.tools |
| OpenAI Assistants | function | tools parameter |

**掌握 LangChain 的 Tool 系统，就掌握了所有 Agent 框架的工具调用模式。**

## 关键数据类型

Tool 系统围绕以下核心类型运转：

```
BaseTool                    ToolCall                    ToolMessage
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│ name: str        │       │ name: str        │       │ content: str     │
│ description: str │       │ args: dict       │       │ tool_call_id: str│
│ args_schema:     │       │ id: str          │       │ artifact: Any    │
│   BaseModel      │       │ type: "tool_call"│       │ status: str      │
│ return_direct:   │       └──────────────────┘       └──────────────────┘
│   bool           │        模型说"我要调用这个"         工具说"结果在这里"
│ response_format: │
│   str            │
│ extras: dict     │
└──────────────────┘
  开发者定义的工具
```

- **BaseTool**: 工具的定义（名称、描述、参数 Schema、执行逻辑）
- **ToolCall**: 模型生成的工具调用请求（在 AIMessage.tool_calls 中）
- **ToolMessage**: 工具执行结果，回传给模型用于生成最终回答

## 关键术语速查

| 术语 | 含义 |
|------|------|
| `BaseTool` | 所有 Tool 的抽象基类，继承自 RunnableSerializable |
| `@tool` | 装饰器，将普通函数转换为 Tool 对象 |
| `StructuredTool` | 支持多参数和自定义 Schema 的 Tool 实现 |
| `bind_tools()` | 将 Tool 列表绑定到 ChatModel，启用函数调用 |
| `tool_calls` | AIMessage 中的字段，表示模型请求调用的工具列表 |
| `ToolCall` | TypedDict，包含 name、args、id 三个字段 |
| `ToolMessage` | 工具执行结果消息，通过 tool_call_id 关联请求 |
| `args_schema` | Tool 的参数 Schema，Pydantic BaseModel 或 dict |
| `parse_docstring` | 从 Google-style docstring 自动提取参数描述 |
| `InjectedToolArg` | 标记参数为运行时注入，不暴露给 LLM |
| `response_format` | Tool 返回格式："content" 或 "content_and_artifact" |
| `return_direct` | 设为 True 时，Tool 结果直接返回用户，跳过 LLM |
| `extras` | 提供商特定的额外字段（如 Anthropic cache_control） |
| `LLMToolSelectorMiddleware` | 用小模型预筛选工具的内置 Middleware |
| `convert_to_openai_tool()` | 将 Tool 转换为 OpenAI function calling 格式 |
