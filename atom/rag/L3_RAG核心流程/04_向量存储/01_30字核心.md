# 30字核心

**向量存储是将 Embedding 向量持久化并建立索引的系统，是 RAG 实现毫秒级语义检索的基础设施。**

---

## 拆解理解

| 关键词 | 含义 |
|-------|------|
| **向量存储（Vector Store）** | 专门存储和检索高维向量的数据库系统 |
| **持久化** | 向量数据保存到磁盘，重启后不丢失 |
| **索引** | 加速搜索的数据结构，避免暴力遍历 |
| **毫秒级检索** | 百万级数据仍能快速响应（<100ms） |
| **基础设施** | RAG 系统的核心组件，不可或缺 |

---

## 为什么这句话重要？

理解这句话，你就理解了：

1. **向量存储不是普通数据库** → 专门为高维向量优化，支持相似度搜索
2. **索引是性能关键** → 没有索引，百万数据搜索需要几秒；有索引，只需几毫秒
3. **持久化避免重复计算** → Embedding 计算成本高，存储后可重复使用

---

## 向量存储的一句话公式

```
向量存储 = 持久化 + 索引 + 相似度搜索
```

用前端的话说：

```
向量存储 = IndexedDB + 搜索索引 + 模糊匹配
```

## 向量存储 vs 传统数据库

| 需求 | 传统数据库（MySQL） | 向量存储（ChromaDB/Milvus） |
|------|-------------------|---------------------------|
| 精确匹配 | ✅ 擅长 `WHERE id = 1` | ❌ 不是设计目标 |
| 语义相似 | ❌ 无法实现 | ✅ 核心能力 |
| 高维向量 | ❌ 效率极低 | ✅ 专门优化 |
| 近似搜索 | ❌ 不支持 | ✅ ANN 算法 |
| 元数据过滤 | ✅ 擅长 | ✅ 支持（联合查询） |

---

**记住这句话：** 向量存储 = 持久化 + 索引 + 相似度搜索，三位一体支撑 RAG 检索。
