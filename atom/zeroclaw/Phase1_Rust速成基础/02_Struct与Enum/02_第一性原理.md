# 02_第一性原理：Struct 与 Enum

> **第一性原理**：回到事物最基本的真理，从源头思考问题
>
> 本文从"程序如何组织数据"这一根本问题出发，推导出 Rust 的 struct（积类型）和 enum（和类型）为何是构建 ZeroClaw 这类 AI Agent 系统的理想基石。

---

## 起点问题

**程序需要组织数据，如何在编译期保证数据结构的正确性？**

这个问题看似简单，却是所有编程语言类型系统设计的核心挑战。让我们从零开始推理。

---

## 推理链：5 步推导 Struct 与 Enum

### Step 1：为什么需要自定义类型？

**原始类型不够用。**

每门语言都有原始类型（数字、字符串、布尔值），但真实世界的数据是**复合**的：

```typescript
// TypeScript：一个 Agent 需要多少信息？
const agentName: string = "assistant";
const agentModel: string = "gpt-4";
const agentTemperature: number = 0.7;
const agentMaxTokens: number = 4096;
// ... 还有 20+ 个字段散落各处
```

问题立刻出现：

| 问题 | 说明 |
|------|------|
| **数据散落** | 20 个变量分布在代码各处，无法作为整体传递 |
| **无约束** | 任何 string 都能赋给 agentModel，拼写错误无人发现 |
| **无结构** | 函数签名变成 `fn(string, string, number, number, ...)` |
| **无保护** | 谁都可以随意修改任何变量 |

**结论**：我们需要一种方式，把相关数据**打包**成一个有名字、有结构、有约束的自定义类型。

#### 日常生活类比

想象你搬家。你不会把所有物品散放在卡车里——你会用**纸箱**分类打包：一箱厨房用品、一箱书籍、一箱衣服。每个箱子有标签（类型名），里面的物品有固定种类（字段类型）。

#### 前端开发类比

就像 TypeScript 的 `interface`——你不会让 API 返回 10 个散乱的变量，而是定义一个 `interface UserResponse` 把它们组织起来。

---

### Step 2：数据有两种根本的组合方式

这是类型理论中最核心的洞察：

> **所有复合数据类型，归根到底只有两种组合逻辑：AND 和 OR。**

#### AND 组合（积类型 / Product Type）

"这个数据需要**所有这些字段同时存在**。"

```
Agent = name AND model AND temperature AND max_tokens AND ...
```

一个 Agent 必须**同时**拥有名字、模型、温度等所有属性。缺一不可。

**为什么叫"积类型"？** 因为可能的值的数量是各字段值数量的**乘积**。

```
// 如果 bool 有 2 种值，u8 有 256 种值
struct Pair { a: bool, b: u8 }
// 可能的值 = 2 × 256 = 512 种
```

#### OR 组合（和类型 / Sum Type）

"这个数据是**这些可能中的某一种**。"

```
Message = TextMessage OR ToolCall OR ToolResult OR SystemPrompt
```

一条消息在任何时刻只能是其中**一种**。不可能同时是文本消息和工具调用。

**为什么叫"和类型"？** 因为可能的值的数量是各变体值数量的**总和**。

```
// 如果 TextMessage 有 N 种, ToolCall 有 M 种
// Message 的可能值 = N + M 种
```

#### TypeScript 对照

```typescript
// AND -> interface（所有字段同时需要）
interface Agent {
  name: string;
  model: string;
  temperature: number;
}

// OR -> union type（选其一）
type Message = TextMessage | ToolCall | ToolResult;
```

#### 日常生活类比

- **AND（积类型）**：一份简历 = 姓名 + 学历 + 工作经历 + 技能。每项都要有。
- **OR（和类型）**：支付方式 = 现金 | 信用卡 | 微信 | 支付宝。每次只选一种。

---

### Step 3：AND -> Struct（所有字段都需要）

Rust 用 `struct` 表达 AND 组合：

```rust
// Rust struct = TypeScript interface
struct Agent {
    name: String,
    model: String,
    temperature: f64,
    max_tokens: u32,
}
```

```typescript
// TypeScript 等价物
interface Agent {
  name: string;
  model: string;
  temperature: number;
  maxTokens: number;
}
```

**关键差异：Rust struct 在编译期做得更多**

| 特性 | TypeScript interface | Rust struct |
|------|---------------------|-------------|
| 类型检查 | 编译时（tsc） | 编译时（rustc） |
| 运行时存在 | 否（擦除） | 是（真实内存布局） |
| 内存布局 | V8 决定 | 编译器优化，开发者可控 |
| 字段可见性 | 无原生支持 | pub / private |
| 所有权语义 | 无（GC） | 每个字段有明确所有权 |
| 默认可变性 | 可变 | 不可变（需 `mut`） |

TypeScript 的 interface 在编译后消失，只是给开发者看的"注释"。Rust 的 struct 是**真实存在于内存中的数据布局**，编译器会根据它优化内存分配、检查生命周期、管理所有权。

---

### Step 4：OR -> Enum（选其一）= 比 Union Type 更强

Rust 用 `enum` 表达 OR 组合。但 Rust 的 enum 远比 TypeScript 的 union type 强大：

```rust
// Rust enum：每个变体可以携带不同类型和数量的数据
enum ChatMessage {
    System(String),                          // 系统消息：只有文本
    User { content: String, name: Option<String> },  // 用户消息：文本 + 可选名字
    Assistant {
        content: Option<String>,             // 助手回复：可能有文本
        tool_calls: Vec<ToolCall>,           // 也可能有工具调用
    },
    Tool { tool_call_id: String, output: String },   // 工具结果
}
```

```typescript
// TypeScript union type：需要手动管理判别
type ChatMessage =
  | { role: "system"; content: string }
  | { role: "user"; content: string; name?: string }
  | { role: "assistant"; content?: string; tool_calls: ToolCall[] }
  | { role: "tool"; tool_call_id: string; output: string };
```

**Rust enum 比 TypeScript union type 强在哪里？**

| 方面 | TypeScript Union | Rust Enum |
|------|-----------------|-----------|
| 判别方式 | 手动加 `role` 字段 | 编译器自动管理判别符 |
| 穷尽性检查 | 需要 `never` 技巧 | 原生支持（Step 5 详解） |
| 数据关联 | 每个变体是独立 interface | 变体直接内嵌数据 |
| 方法定义 | 无法直接在 union 上定义 | `impl` 块直接定义方法 |
| 内存效率 | 每个对象独立分配 | 编译器优化对齐和大小 |
| 运行时安全 | 可能拼错 `role` | 不可能——变体是类型系统的一部分 |

#### 日常生活类比

TypeScript union type 像是一个纸质表格，你在"类型"一栏手写"现金"或"信用卡"——写错了没人管。

Rust enum 像是一台 ATM 机上的按钮——只有那几个固定选项，你不可能按到不存在的按钮。

---

### Step 5：match 穷尽性 -> 编译器保证每种可能都被处理

这是 Rust 类型系统最令人安心的特性：**穷尽性检查（Exhaustiveness Checking）**。

```rust
fn process_message(msg: &ChatMessage) -> String {
    match msg {
        ChatMessage::System(text) => format!("System: {}", text),
        ChatMessage::User { content, .. } => format!("User: {}", content),
        ChatMessage::Assistant { content, tool_calls } => {
            if let Some(text) = content {
                format!("Assistant: {}", text)
            } else {
                format!("Assistant called {} tools", tool_calls.len())
            }
        }
        // 如果忘记处理 ChatMessage::Tool？
        // 编译错误！编译器会告诉你缺少了哪个变体
    }
}
```

**编译器报错：**

```
error[E0004]: non-exhaustive patterns: `ChatMessage::Tool { .. }` not covered
 --> src/main.rs:10:11
  |
  = help: ensure that all possible cases are being handled
```

这意味着：

1. **新增变体时自动发现所有需要修改的地方**。给 ChatMessage 加一个 `Image` 变体？编译器会标出每一处 `match` 让你补全。
2. **不可能遗漏边界情况**。这在 TypeScript 中需要 `switch` + `default` + `never` 的技巧组合才能部分实现。
3. **重构时零恐惧**。改变 enum 结构，编译器就是你的检查清单。

#### 前端开发类比

想象你有一个 Redux reducer，处理多种 action type：

```typescript
// TypeScript：忘记处理某个 action？
// 运行时才发现 bug（如果运气好的话）
switch (action.type) {
  case "FETCH_START": return { ...state, loading: true };
  case "FETCH_SUCCESS": return { ...state, data: action.payload };
  // 忘了 "FETCH_ERROR"？TypeScript 不会报错！
  default: return state;
}
```

Rust 的 `match` 就像一个**没有 default 分支的 switch，但编译器保证你处理了所有情况**。

#### 日常生活类比

快递分拣中心有 4 个出口：北京、上海、广州、深圳。传送带上每个包裹都有目的地标签。Rust 的 match 就是保证：**没有任何包裹会掉到地上**——每个目的地都有对应的出口。如果新增了"成都"出口，系统会立刻提醒你把传送带接上去。

## 为什么这个推理链对 ZeroClaw 至关重要？

ZeroClaw 是一个 AI Agent 运行时系统。让我们看看这两种组合方式如何在其中发挥作用：

### AND = Struct：Agent 需要同时具备所有能力

```rust
// ZeroClaw 源码中的 Agent 结构体有 20+ 字段
// 所有字段必须同时存在，Agent 才能运行
struct Agent {
    // 核心配置（AND：全部需要）
    provider: Box<dyn Provider>,        // LLM 提供者
    tools: Vec<Arc<dyn Tool>>,          // 可用工具
    memory: Option<Box<dyn Memory>>,    // 记忆系统

    // 安全策略（AND：全部需要）
    security_policy: SecurityPolicy,     // 安全限制
    autonomy_level: AutonomyLevel,       // 自主权级别

    // 运行时状态（AND：全部需要）
    conversation: Vec<ConversationMessage>,  // 对话历史
    system_prompt: String,                   // 系统提示词
    // ... 更多字段
}
```

struct 保证：**一个 Agent 实例必须拥有全部这些组件**。编译器不允许你创建一个"没有 provider"的 Agent。

### OR = Enum：消息只能是某一种类型

```rust
// ZeroClaw 源码中的 ConversationMessage
// 每条消息只能是其中一种（OR）
enum ConversationMessage {
    Chat(ChatMessage),         // 普通对话消息
    ToolCalls(Vec<ToolCall>),  // 工具调用请求
    ToolResults(Vec<ToolResult>), // 工具返回结果
}

// 处理消息时，match 保证不遗漏
fn handle_message(msg: &ConversationMessage) {
    match msg {
        ConversationMessage::Chat(chat) => { /* 处理对话 */ }
        ConversationMessage::ToolCalls(calls) => { /* 执行工具 */ }
        ConversationMessage::ToolResults(results) => { /* 使用结果 */ }
        // 编译器保证：不会有第四种情况悄悄溜过
    }
}
```

### AND + OR 组合：真实系统的复杂性

ZeroClaw 中 struct 和 enum 互相嵌套，构建出精确的类型模型：

```rust
// SecurityPolicy（struct）中包含 AutonomyLevel（enum）
struct SecurityPolicy {
    pub autonomy_level: AutonomyLevel,  // enum：选择一种级别
    pub max_actions_per_turn: usize,    // 与其他字段同时存在
    pub require_approval: bool,
}

// AutonomyLevel（enum）定义了几种可能的安全级别
enum AutonomyLevel {
    Full,           // 完全自主
    Supervised,     // 有监督
    Restricted,     // 受限
    Manual,         // 手动
}
```

**类比**：SecurityPolicy 是一份安全协议文件（struct = AND，所有条款都要有），其中"自主权级别"这一栏是一个**单选题**（enum = OR，只能选一个）。

## 与其他语言的根本差异

### C 语言：有 struct，但 enum 太弱

```c
// C enum 只是整数常量，不能携带数据
enum Color { RED, GREEN, BLUE };  // RED = 0, GREEN = 1, BLUE = 2

// 想表达"消息可以是文本或工具调用"？
// 需要手动用 union + tag，极易出错
struct Message {
    enum MessageType type;
    union {
        char* text;
        struct ToolCall tool_call;
    } data;
};
// 没有编译器帮你检查 type 和 data 是否匹配！
```

### Java / TypeScript：有 class，但无真正的 Sum Type

```typescript
// TypeScript union type 是"结构化"的——基于字段内容区分
// 不是"名义化"的——没有编译器管理的判别符
type Shape = Circle | Rectangle;  // 怎么区分？靠 "kind" 字段

// Java sealed class（Java 17+）接近但更繁琐
sealed interface Shape permits Circle, Rectangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double w, double h) implements Shape {}
```

### Rust：两者都在类型系统中原生支持

```rust
// struct（AND）和 enum（OR）是语言的一等公民
// 编译器同时理解两者，提供穷尽性检查、内存优化、所有权管理

struct Circle { radius: f64 }
struct Rectangle { width: f64, height: f64 }

enum Shape {
    Circle(Circle),
    Rectangle(Rectangle),
}

fn area(shape: &Shape) -> f64 {
    match shape {
        Shape::Circle(c) => std::f64::consts::PI * c.radius * c.radius,
        Shape::Rectangle(r) => r.width * r.height,
        // 编译器保证穷尽
    }
}
```

### 对比总结

| 语言 | Product Type (AND) | Sum Type (OR) | 穷尽性检查 |
|------|--------------------|---------------|-----------|
| C | struct（有） | enum（只有整数） | 无 |
| Java | class（有） | sealed interface（Java 17+，繁琐） | 部分 |
| TypeScript | interface（有） | union type（结构化，非名义化） | 需 never 技巧 |
| **Rust** | **struct（原生）** | **enum（原生，携带数据）** | **编译器强制** |

## 第一性原理总结

```
程序需要组织复合数据
       ↓
复合数据只有两种组合逻辑
       ↓
   ┌───────────────┬──────────────────┐
   │               │                  │
 AND（积类型）    OR（和类型）        │
   │               │                  │
 struct           enum               │
 所有字段同时存在  只能是某一种变体    │
   │               │                  │
   └───────────────┴──────────────────┘
                   ↓
         match 穷尽性检查
                   ↓
     编译器保证每种可能都被处理
                   ↓
     零运行时开销的类型安全
```

**一句话总结：**

**Struct（AND）和 Enum（OR）是数据组合的两种根本方式。Rust 在类型系统中原生支持两者，并通过 match 穷尽性检查，在编译期保证每种数据可能都被正确处理——这就是 ZeroClaw 能安全、高效地管理 Agent 复杂状态的根本原因。**

## 思考题

在理解了 AND 和 OR 的本质之后，思考这些问题：

1. **ZeroClaw 的 Agent 有 20+ 个字段（AND），但很多是可选的。如何处理？** 提示：`Option<T>` 本身就是一个 enum（`Some(T) | None`），所以"可选字段"实际上是 AND 内嵌 OR。

2. **如果 ZeroClaw 需要支持新的消息类型（比如 Image），需要做什么？** 提示：给 enum 加一个变体，然后编译器会告诉你所有需要更新的 match。

3. **TypeScript 中 `string | number` 和 Rust 中 `enum StringOrNumber { S(String), N(i64) }` 有什么本质区别？** 提示：TypeScript 的 union 是"结构化的"（看形状），Rust 的 enum 是"名义化的"（看名字）。
