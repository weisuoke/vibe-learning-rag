# 所有权与借用 —— 化骨绵掌

> **一句话定位：** 10 张知识卡片，每张 2 分钟，从直觉到融会贯通，把所有权系统"化"进你的编程本能。

---

### 卡片 1：所有权的直觉——钥匙模型

**一句话：** 每个值是一把钥匙，同一时刻只能在一个人手里。

**举例：**
```rust
let key = String::from("房间钥匙");
let bob = key;          // 钥匙交给 Bob
// println!("{}", key);  // 你手里已经没钥匙了！
```

TypeScript 里 `const b = a` 是"配了一把新钥匙"（共享引用），Rust 里是"把钥匙递过去"（移动所有权）。

**应用：** ZeroClaw 中 `DelegateTool::new(agents)` 把 HashMap 的"钥匙"交给结构体，调用者之后不能再用原始 HashMap。

---

### 卡片 2：三条铁律——所有权的完整规则

**一句话：** 一值一主人，转手即无权，出域即归零。

**规则表：**

| 规则 | 含义 | 触发时机 |
|------|------|---------|
| 唯一所有者 | 每个值只有一个变量拥有它 | 始终 |
| 移动语义 | 赋值/传参转移所有权，原变量失效 | `let b = a;` |
| 自动释放 | 所有者离开作用域，值被 `drop()` | `}` |

**应用：** 这三条规则让 Rust 在编译期就确定每块内存何时释放，零运行时开销。ZeroClaw 空载内存 < 5MB，等价 Node.js 实现需要 50-200MB。

---

### 卡片 3：借用——不转移所有权的访问

**一句话：** `&T` 是"借来看看"，`&mut T` 是"借来改改"，用完还回去。

**举例：**
```rust
fn print_name(name: &str) {       // 借来看看，不拿走
    println!("{}", name);
}
let name = String::from("ZeroClaw");
print_name(&name);                 // 借出去
println!("{}", name);              // 还回来了，仍然有效
```

**核心规则：**
- `&T`：可以同时借给多人看（多个不可变借用共存）
- `&mut T`：一次只能借给一个人改（独占）
- `&T` 和 `&mut T` 不能同时存在

**应用：** ZeroClaw 80% 的函数参数用 `&T`——`create_memory(config: &MemoryConfig, workspace_dir: &Path)`。

---

### 卡片 4：Copy vs Clone vs Move——三种数据传递方式

**一句话：** 小数据自动复制（Copy），大数据默认移动（Move），需要副本时显式克隆（Clone）。

| 方式 | 触发条件 | 开销 | 例子 |
|------|---------|------|------|
| Copy | 类型实现了 Copy trait | 栈上按位复制 | `i32`, `f64`, `bool`, `char` |
| Move | 默认行为（非 Copy 类型） | 复制 24 字节栈元数据 | `String`, `Vec<T>`, `HashMap` |
| Clone | 显式调用 `.clone()` | 深拷贝（分配新堆内存） | 任何实现 Clone 的类型 |

```rust
let a: i32 = 42;
let b = a;              // Copy：a 仍然有效
let s1 = String::from("hello");
let s2 = s1;            // Move：s1 失效
let s3 = s2.clone();    // Clone：s2 和 s3 都有效，各自拥有独立数据
```

**应用：** ZeroClaw 的 `SecurityPolicy` 枚举实现了 `#[derive(Copy, Clone)]`，因为它是小型枚举，复制开销极低。

---

### 卡片 5：生命周期——引用的有效期

**一句话：** 生命周期注解 `'a` 告诉编译器"这个引用至少活这么久"，防止悬垂引用。

**举例：**
```rust
// 返回值的生命周期 = 两个输入中较短的那个
fn longer<'a>(s1: &'a str, s2: &'a str) -> &'a str {
    if s1.len() > s2.len() { s1 } else { s2 }
}
```

**省略规则（90% 的情况不用手写）：**
1. 每个引用参数获得独立生命周期
2. 只有一个输入引用 → 输出生命周期 = 输入生命周期
3. 方法中有 `&self` → 输出生命周期 = `self` 的生命周期

**应用：** ZeroClaw 中 `convert_tools<'a>(tools: Option<&'a [ToolSpec]>) -> Option<Vec<NativeToolSpec<'a>>>` 显式标注，保证返回值不超过输入数据的存活时间。

### 卡片 6：智能指针——突破"一个所有者"的限制

**一句话：** `Box` 放堆上，`Rc` 单线程共享，`Arc` 多线程共享。

| 指针 | 所有者数量 | 线程安全 | 典型场景 |
|------|-----------|---------|---------|
| `Box<T>` | 1 个 | 是 | Trait 对象 `Box<dyn Tool>` |
| `Rc<T>` | 多个 | 否 | 单线程图结构 |
| `Arc<T>` | 多个 | 是 | 跨线程共享配置 |

```rust
let policy = Arc::new(SecurityPolicy::default());
let p2 = Arc::clone(&policy);  // 引用计数 +1，约 1 纳秒，零数据拷贝
// policy 和 p2 指向同一份数据
```

**应用：** ZeroClaw 所有工具通过 `Arc<SecurityPolicy>` 共享安全策略，`Arc::clone()` 开销几乎为零。

---

### 卡片 7：内部可变性——用 `&self` 修改数据

**一句话：** `RefCell`（单线程）和 `Mutex`（多线程）让你通过不可变引用修改内部数据，把借用检查从编译期推迟到运行时。

```rust
use std::sync::Mutex;

struct Counter {
    count: Mutex<u32>,
}
impl Counter {
    fn increment(&self) {           // 注意：&self 不是 &mut self
        *self.count.lock().unwrap() += 1;
    }
}
```

**为什么需要？** 当多个地方持有 `Arc<T>` 共享同一个对象时，你拿到的是 `&T`（不可变引用）。要修改内部状态，就需要内部可变性。

**应用：** ZeroClaw 的 `ActionTracker` 用 `Mutex<Vec<Instant>>` 记录操作时间，通过 `&self` 就能追加记录。

---

### 卡片 8：`Arc<Mutex<T>>`——异步共享可变状态的标准模式

**一句话：** `Arc` 负责跨线程共享，`Mutex` 负责安全修改，组合起来就是异步编程的"万能钥匙"。

```rust
use std::sync::Arc;
use tokio::sync::Mutex;

let state = Arc::new(Mutex::new(Vec::new()));
let state_clone = Arc::clone(&state);

tokio::spawn(async move {
    state_clone.lock().await.push("from task");
});
```

**注意事项：**
- 异步代码用 `tokio::sync::Mutex`（支持 `.await`）
- 同步代码用 `std::sync::Mutex`
- 锁的持有时间尽量短，避免死锁

**应用：** ZeroClaw 的 `DriveTool` 用 `Arc<Mutex<Option<Instant>>>` 跟踪最后一次命令时间，多个异步任务安全共享。

### 卡片 9：所有权决策流程——实战选型

**一句话：** 遵循最小权限原则，从 `&T` 开始，按需升级。

```
需要传递数据？
├── 只读 ──────────────→ &T          （80% 的情况）
├── 需要修改 ──────────→ &mut T      （就地修改）
├── 需要拥有 ──────────→ T           （构造函数、跨线程发送）
│
需要多处共享？
├── 单线程 ────────────→ Rc<T>
├── 多线程只读 ────────→ Arc<T>
├── 多线程读写 ────────→ Arc<Mutex<T>>
│
以上都不行？ ──────────→ clone()     （最后手段）
```

**应用：** ZeroClaw 的函数签名完美体现了这个决策链——`create_memory` 用 `&T` 借用配置，`DelegateTool::new` 用 `T` 获取所有权，工具列表用 `Arc<Vec<Arc<dyn Tool>>>` 多层共享。

---

### 卡片 10：从 TypeScript 到 Rust——思维转换总结

**一句话：** 从"数据默认共享"转变为"数据默认独占，共享需要显式声明"。

| TypeScript 思维 | Rust 思维 |
|----------------|-----------|
| GC 会处理内存 | 谁拥有这个值？什么时候释放？ |
| 赋值 = 共享引用 | 赋值 = 转移所有权 |
| 随时修改对象 | `&mut` 是独占的 |
| 不用想生命周期 | 引用不能活过被引用的数据 |
| clone 解决一切 | 先 `&` 借用，再 `Arc` 共享，最后 `clone` |
| 并发安全靠自觉 | 编译器强制并发安全 |

**最终心法：** 所有权系统不是限制，而是编译器帮你提前发现 bug 的工具。前期多花 10% 时间和编译器"对话"，换来运行时零内存 bug、零数据竞争、零 GC 暂停。
