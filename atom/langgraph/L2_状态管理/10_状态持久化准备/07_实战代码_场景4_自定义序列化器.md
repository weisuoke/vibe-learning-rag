# 实战代码 - 场景4：自定义序列化器

## 场景说明

本场景演示如何实现自定义序列化器——理解 `SerializerProtocol` 接口，并动手实现自己的序列化逻辑。这是深入理解 LangGraph 持久化架构的关键一步。

核心要点：
1. **SerializerProtocol 只有两个方法**：`dumps_typed` 和 `loads_typed`，返回 `(type_tag, bytes)` 元组
2. **类型标签是关键**：告诉反序列化器如何解码字节（如 "json"、"msgpack"、"pickle"）
3. **装饰器模式**：可以像 EncryptedSerializer 一样包装现有序列化器，添加额外功能
4. **实际应用**：压缩、日志记录、指标收集、格式转换等

[来源: reference/source_状态持久化准备_01.md | LangGraph 源码分析]
[来源: reference/context7_langgraph_01.md | LangGraph 官方文档]

---

## 核心知识点

| 知识点 | 说明 |
|--------|------|
| `SerializerProtocol` | 现代序列化接口，定义 `dumps_typed` / `loads_typed` |
| `dumps_typed(obj)` | 将 Python 对象序列化为 `(type_tag: str, data: bytes)` |
| `loads_typed((tag, data))` | 将 `(type_tag, bytes)` 反序列化为 Python 对象 |
| 类型标签 (type_tag) | 字符串标识，告诉反序列化器数据格式（如 "json", "msgpack"） |
| 装饰器模式 | 包装现有序列化器，在序列化/反序列化前后添加逻辑 |
| `JsonPlusSerializer` | LangGraph 默认序列化器，主要使用 msgpack 格式 |

class JsonGzipSerializer:
    """自定义序列化器：JSON 编码 + Gzip 压缩

    适用场景：
    - 状态数据较大（长文本、大列表）
    - 存储空间有限
    - 网络传输需要压缩

    类型标签：
    - "json.gz" → JSON + Gzip 压缩
    - "null"    → None 值（不压缩）
    - "bytes.gz" → 原始字节 + Gzip 压缩
    """

    def __init__(self, compress_level: int = 6):
        """
        Args:
            compress_level: Gzip 压缩级别 (1-9)，默认 6
                1 = 最快压缩，压缩率低
                9 = 最慢压缩，压缩率高
        """
        self.compress_level = compress_level

    def dumps_typed(self, obj: Any) -> tuple[str, bytes]:
        """序列化：Python 对象 → (类型标签, 压缩字节)"""
        if obj is None:
            return ("null", b"")

        if isinstance(obj, bytes):
            compressed = gzip.compress(obj, compresslevel=self.compress_level)
            return ("bytes.gz", compressed)

        # 其他类型：JSON 编码 → UTF-8 字节 → Gzip 压缩
        json_str = json.dumps(obj, ensure_ascii=False, default=str)
        json_bytes = json_str.encode("utf-8")
        compressed = gzip.compress(json_bytes, compresslevel=self.compress_level)
        return ("json.gz", compressed)

    def loads_typed(self, data: tuple[str, bytes]) -> Any:
        """反序列化：(类型标签, 压缩字节) → Python 对象"""
        type_tag, payload = data

        if type_tag == "null":
            return None

        if type_tag == "bytes.gz":
            return gzip.decompress(payload)

        if type_tag == "json.gz":
            decompressed = gzip.decompress(payload)
            return json.loads(decompressed.decode("utf-8"))

        raise ValueError(f"未知的类型标签: {type_tag!r}")

class LoggingSerializer:
    """日志序列化器：包装任何序列化器，添加日志和指标收集

    这是 EncryptedSerializer 使用的同一种设计模式——装饰器模式：
    - 不改变原始序列化器的行为
    - 在序列化/反序列化前后添加额外逻辑
    - 可以嵌套多层（日志 → 加密 → 压缩 → 基础序列化）

    实际应用场景：
    - 调试：记录每次序列化/反序列化的详情
    - 监控：收集序列化耗时、数据大小等指标
    - 审计：记录谁在什么时候访问了什么数据
    """

    def __init__(self, inner_serde, name: str = "LoggingSerde"):
        """
        Args:
            inner_serde: 被包装的序列化器（任何实现 SerializerProtocol 的对象）
            name: 日志前缀名称
        """
        self.inner = inner_serde
        self.name = name
        # 指标收集
        self.stats = {
            "dumps_count": 0,
            "loads_count": 0,
            "total_bytes_out": 0,
            "total_bytes_in": 0,
            "total_dumps_time_ms": 0.0,
            "total_loads_time_ms": 0.0,
        }

    def dumps_typed(self, obj: Any) -> tuple[str, bytes]:
        """序列化 + 日志记录"""
        start = time.perf_counter()

        # 委托给内部序列化器
        type_tag, data = self.inner.dumps_typed(obj)

        elapsed_ms = (time.perf_counter() - start) * 1000

        # 更新指标
        self.stats["dumps_count"] += 1
        self.stats["total_bytes_out"] += len(data)
        self.stats["total_dumps_time_ms"] += elapsed_ms

        # 输出日志
        obj_type = type(obj).__name__
        print(f"  [{self.name}] dumps: {obj_type} → "
              f"tag={type_tag!r}, size={len(data)}B, "
              f"time={elapsed_ms:.2f}ms")

        return (type_tag, data)

    def loads_typed(self, data: tuple[str, bytes]) -> Any:
        """反序列化 + 日志记录"""
        type_tag, payload = data
        start = time.perf_counter()

        # 委托给内部序列化器
        obj = self.inner.loads_typed(data)

        elapsed_ms = (time.perf_counter() - start) * 1000

        # 更新指标
        self.stats["loads_count"] += 1
        self.stats["total_bytes_in"] += len(payload)
        self.stats["total_loads_time_ms"] += elapsed_ms

        obj_type = type(obj).__name__
        print(f"  [{self.name}] loads: tag={type_tag!r}, "
              f"size={len(payload)}B → {obj_type}, "
              f"time={elapsed_ms:.2f}ms")

        return obj

    def print_stats(self):
        """打印收集的指标"""
        print(f"\n  [{self.name}] 统计:")
        print(f"    序列化次数: {self.stats['dumps_count']}")
        print(f"    反序列化次数: {self.stats['loads_count']}")
        print(f"    总输出字节: {self.stats['total_bytes_out']:,}")
        print(f"    总输入字节: {self.stats['total_bytes_in']:,}")
        print(f"    总序列化耗时: {self.stats['total_dumps_time_ms']:.2f}ms")
        print(f"    总反序列化耗时: {self.stats['total_loads_time_ms']:.2f}ms")

# ----- 1. 定义状态 -----

class DocState(TypedDict):
    """文档处理状态"""
    query: str
    documents: Annotated[list[str], add]
    summary: str


# ----- 2. 定义节点 -----

def search_node(state: DocState) -> dict:
    """搜索节点：模拟文档检索"""
    query = state["query"]
    docs = [
        f"文档A: {query}的基础概念和定义...",
        f"文档B: {query}的实际应用案例...",
        f"文档C: {query}的最佳实践指南...",
    ]
    print(f"  search: 检索到 {len(docs)} 篇文档")
    return {"documents": docs}


def summarize_node(state: DocState) -> dict:
    """摘要节点：生成文档摘要"""
    docs = state.get("documents", [])
    summary = f"综合 {len(docs)} 篇文档的摘要: {state['query']}相关内容"
    print(f"  summarize: {summary}")
    return {"summary": summary}


# ----- 3. 构建图 -----

builder = StateGraph(DocState)
builder.add_node("search", search_node)
builder.add_node("summarize", summarize_node)
builder.add_edge(START, "search")
builder.add_edge("search", "summarize")
builder.add_edge("summarize", END)

# ----- 4. 使用日志序列化器 -----

# 创建带日志的序列化器
monitored_serde = LoggingSerializer(JsonPlusSerializer(), name="Checkpoint")

# 创建 SqliteSaver 并传入自定义序列化器
conn = sqlite3.connect(":memory:", check_same_thread=False)
checkpointer = SqliteSaver(conn, serde=monitored_serde)
checkpointer.setup()

# 编译图
graph = builder.compile(checkpointer=checkpointer)

# ----- 5. 执行并观察序列化日志 -----

config = {"configurable": {"thread_id": "custom-serde-1"}}

print("\n--- 执行图（观察序列化日志）---")
result = graph.invoke(
    {"query": "LangGraph 状态管理", "documents": [], "summary": ""},
    config,
)
print(f"\n结果: {result['summary']}")

# 打印序列化统计
monitored_serde.print_stats()

# ----- 6. 恢复状态（观察反序列化日志）-----

print("\n--- 恢复状态（观察反序列化日志）---")
state = graph.get_state(config)
print(f"恢复的查询: {state.values['query']}")
print(f"恢复的文档数: {len(state.values['documents'])}")

monitored_serde.print_stats()

conn.close()

## 代码解析

### 1. SerializerProtocol 的极简设计

```python
class SerializerProtocol(Protocol):
    def dumps_typed(self, obj: Any) -> tuple[str, bytes]: ...
    def loads_typed(self, data: tuple[str, bytes]) -> Any: ...
```

只有两个方法，但设计精妙：
- `tuple[str, bytes]` 中的 `str` 是类型标签——自描述格式
- 类型标签让不同序列化器可以共存（同一个数据库中可以有 msgpack 和 json 格式的数据）
- Protocol 模式（鸭子类型）：不需要继承，只要实现这两个方法就行

### 2. 装饰器模式的层次结构

```
                    ┌─────────────────────────────────┐
                    │     LoggingSerializer            │
                    │  ┌───────────────────────────┐   │
                    │  │   EncryptedSerializer      │   │
                    │  │  ┌─────────────────────┐   │   │
                    │  │  │  JsonPlusSerializer  │   │   │
                    │  │  │  (实际序列化)        │   │   │
                    │  │  └─────────────────────┘   │   │
                    │  │  (加密/解密)               │   │
                    │  └───────────────────────────┘   │
                    │  (日志/指标)                      │
                    └─────────────────────────────────┘

数据流（序列化）：
  obj → 日志记录 → 加密 → msgpack 编码 → (tag, bytes)

数据流（反序列化）：
  (tag, bytes) → msgpack 解码 → 解密 → 日志记录 → obj
```

每一层只关心自己的职责，互不干扰。这就是关注点分离的力量。

### 3. 自定义序列化器的取舍

```
JsonPlusSerializer (默认)     JsonGzipSerializer (自定义)
─────────────────────────     ──────────────────────────
✓ 类型保持完美                 ✗ datetime/enum 等类型丢失
✓ msgpack 高效二进制           ✓ Gzip 压缩率更高
✓ 支持 Pydantic/NumPy         ✗ 只支持 JSON 可序列化类型
✗ 无内置压缩                   ✓ 内置 Gzip 压缩
```

选择序列化器时要考虑：
- 状态中是否有特殊类型（datetime, Enum, Pydantic）？→ 用默认
- 状态数据是否很大，需要压缩？→ 考虑自定义压缩层
- 是否需要人类可读的存储格式？→ 考虑 JSON 序列化器

### 4. 为什么 LangGraph 用 Protocol 而不是抽象基类

```python
# Protocol 模式（LangGraph 的选择）
class SerializerProtocol(Protocol):
    def dumps_typed(self, obj: Any) -> tuple[str, bytes]: ...
    def loads_typed(self, data: tuple[str, bytes]) -> Any: ...

# 你的类不需要继承任何东西
class MySerializer:  # 没有继承！
    def dumps_typed(self, obj): ...
    def loads_typed(self, data): ...
# MySerializer 自动满足 SerializerProtocol（鸭子类型）

# 对比：抽象基类模式
class AbstractSerializer(ABC):
    @abstractmethod
    def dumps_typed(self, obj): ...

class MySerializer(AbstractSerializer):  # 必须继承
    def dumps_typed(self, obj): ...
```

Protocol 的优势：
- 不需要继承，降低耦合
- 第三方库的类也能满足协议（无需修改源码）
- 更 Pythonic（鸭子类型的精神）

## 关键要点

1. **SerializerProtocol 只有两个方法**：`dumps_typed` 和 `loads_typed`，返回 `(type_tag, bytes)` 元组
2. **类型标签是自描述的**：让反序列化器知道如何解码，也让不同格式的数据可以共存
3. **装饰器模式是扩展序列化的标准方式**：日志、加密、压缩都可以作为独立的装饰层
4. **自定义序列化器要注意类型保持**：JSON 会丢失 datetime/enum 等类型，msgpack + 扩展类型系统更完善
5. **Protocol 模式降低耦合**：不需要继承，只要实现方法签名就行
6. **生产环境建议用默认 + 装饰器**：在 JsonPlusSerializer 外面包装日志/加密/压缩层

## 扩展思考

1. **能否实现一个版本化序列化器？** 可以。在类型标签中加入版本号（如 "json.v2"），loads_typed 根据版本号选择不同的反序列化逻辑。这对于状态 Schema 演进很有用。

2. **压缩序列化器在什么场景下值得用？** 当状态包含大量文本（RAG 的文档片段、长对话历史）时，Gzip 压缩可以节省 60-90% 的存储空间。但对于小状态（几百字节），压缩反而会增大体积。

3. **如何实现选择性序列化？** 可以在装饰器中检查对象类型或大小，对大对象使用压缩，对小对象直接透传。类型标签可以标记是否压缩（如 "msgpack" vs "msgpack.gz"）。

4. **LoggingSerializer 在生产环境有什么用？** 可以用来监控 checkpoint 的大小增长趋势、序列化耗时异常、以及定位性能瓶颈。建议在开发/测试环境启用，生产环境按需开启。
