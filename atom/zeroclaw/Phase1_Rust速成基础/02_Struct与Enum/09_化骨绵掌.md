# 化骨绵掌：Struct 与 Enum 的深层内功

> **一句话定位：** 超越语法层面，理解 Rust Struct/Enum 在内存布局、编译器优化、类型状态机等方面的深层设计——这些是写出高性能、零 bug 代码的内功心法。

---

## 内功 1：Enum 的内存布局与 Niche Optimization

### 表面理解

"Enum 大小 = 标签 + 最大变体"

### 深层真相

编译器会利用**niche（空位）**优化，让某些 Option 零开销：

```rust
use std::mem::size_of;

// 普通 enum：标签(1字节) + 最大变体
enum Foo { A(u8), B(u16) }
// size = 4 (1 tag + 1 padding + 2 data)

// Niche 优化：Option<&T> 不额外占空间！
assert_eq!(size_of::<&str>(), size_of::<Option<&str>>());  // 都是 8 字节
assert_eq!(size_of::<Box<i32>>(), size_of::<Option<Box<i32>>>());  // 都是 8 字节

// 原理：引用/Box 永远不为 null，所以编译器用 0x0 表示 None
// 这就是 "niche"——类型中不可能出现的值被借用来表示 None
```

**受益类型：** `&T`、`Box<T>`、`NonZeroU32`、函数指针——任何有"不可能值"的类型。

### 实际影响

```rust
// 链表节点：Option<Box<Node>> 和 Box<Node> 一样大
struct Node {
    value: i32,
    next: Option<Box<Node>>,  // 8 字节，不是 16 字节
}
```

## 内功 2：Zero-Sized Types (ZST) 与类型标记

### 表面理解

"空 struct 没什么用"

### 深层真相

零大小类型（ZST）在编译期携带信息，运行时零开销：

```rust
use std::marker::PhantomData;

struct Draft;
struct Published;

struct Article<State> {
    title: String,
    content: String,
    _state: PhantomData<State>,  // 零大小，不占内存
}

impl Article<Draft> {
    fn new(title: &str) -> Self {
        Article { title: title.into(), content: String::new(), _state: PhantomData }
    }

    fn edit(&mut self, content: &str) { self.content = content.into(); }

    // 只有 Draft 能 publish，返回 Published 状态
    fn publish(self) -> Article<Published> {
        Article { title: self.title, content: self.content, _state: PhantomData }
    }
}

impl Article<Published> {
    fn view(&self) -> &str { &self.content }
    // 没有 edit() 方法——已发布的文章不能编辑！编译期保证
}

fn main() {
    let mut draft = Article::<Draft>::new("Rust ZST");
    draft.edit("ZST are powerful");
    let published = draft.publish();
    println!("{}", published.view());
    // published.edit("hack");  // ❌ 编译错误！Published 没有 edit 方法
}
```

**核心价值：** 状态转换的合法性由类型系统保证，不需要运行时检查。

## 内功 3：Enum 的 `#[repr]` 与 FFI

### 表面理解

"Enum 的内存布局是编译器自动决定的"

### 深层真相

默认布局下编译器可以自由优化（重排字段、压缩标签）。但与 C 交互时需要固定布局：

```rust
// 默认：编译器自由优化
enum Color { Red, Green, Blue }

// 指定底层类型：与 C 兼容
#[repr(u8)]
enum StatusCode {
    Ok = 0,
    Error = 1,
    Timeout = 2,
}

// 指定 C 布局：FFI 安全
#[repr(C)]
enum CEvent {
    None,
    Click { x: i32, y: i32 },
    Key(u32),
}
```

| repr | 用途 | 标签大小 |
|------|------|---------|
| 默认 | 纯 Rust 代码 | 编译器决定（可能 0） |
| `#[repr(u8/u16/...)]` | 简单 C enum | 指定整数类型 |
| `#[repr(C)]` | C FFI 交互 | 平台相关 |

## 内功 4：match 的编译优化——跳转表

### 表面理解

"match 就是一串 if-else"

### 深层真相

编译器会把 match 优化为**跳转表（jump table）**或**二分查找**，而不是线性比较：

```rust
// 这个 match 不是 11 次 if-else
// 编译器生成跳转表：O(1) 直接跳到对应分支
match event {
    ObserverEvent::AgentStarted { .. } => { /* ... */ }
    ObserverEvent::AgentCompleted { .. } => { /* ... */ }
    // ... 9 more variants
}

// 等价伪代码：
// jump_table[event.tag]()  // 一次索引，直接跳转
```

**对比 TypeScript：**
```typescript
// switch 在 V8 中可能优化为跳转表，也可能不
// 字符串 switch 几乎不可能优化
switch (event.type) {  // 字符串比较，O(n)
    case "AgentStarted": ...
}
```

## 内功 5：Struct 的字段重排与对齐

### 表面理解

"Struct 字段按声明顺序排列"

### 深层真相

Rust 默认会**重排字段**以减少 padding（内存对齐填充）：

```rust
// 你写的顺序
struct Bad { a: u8, b: u64, c: u8 }
// 如果按声明顺序：1 + 7padding + 8 + 1 + 7padding = 24 字节

// 编译器重排后：b(8) + a(1) + c(1) + 6padding = 16 字节
// 节省 8 字节！

// 如果需要固定顺序（FFI）：
#[repr(C)]
struct Fixed { a: u8, b: u64, c: u8 }  // 强制按声明顺序，24 字节
```

**实际建议：** 纯 Rust 代码不要加 `#[repr(C)]`，让编译器自由优化。只在 FFI 边界才需要固定布局。

## 内功 6：Enum 变体大小不均——Box 优化

### 表面理解

"Enum 就是 Enum，不需要特别处理"

### 深层真相

一个超大变体会拖累整个 enum 的大小：

```rust
use std::mem::size_of;

enum Bloated {
    Small(u8),                    // 1 字节
    Big([u8; 4096]),              // 4096 字节
}
// size_of::<Bloated>() ≈ 4104——即使 99% 的时候是 Small！

enum Optimized {
    Small(u8),
    Big(Box<[u8; 4096]>),         // 8 字节指针
}
// size_of::<Optimized>() ≈ 16——Box 把大数据放到堆上
```

**经验法则：** 如果某个变体比其他变体大 10 倍以上，用 `Box` 包装它。Clippy 的 `large_enum_variant` lint 会自动提醒你。

## 内功 7：穷尽性 + `#[non_exhaustive]` 的库设计

### 表面理解

"match 必须写全所有变体"

### 深层真相

库作者用 `#[non_exhaustive]` 预留扩展空间：

```rust
// 库代码（crate A）
#[non_exhaustive]
pub enum ApiError {
    NotFound,
    Unauthorized,
    RateLimit,
}

// 用户代码（crate B）
fn handle(err: ApiError) {
    match err {
        ApiError::NotFound => { /* ... */ }
        ApiError::Unauthorized => { /* ... */ }
        ApiError::RateLimit => { /* ... */ }
        _ => { /* 必须有！因为库可能新增变体 */ }
    }
}
```

**设计决策：**
- 内部 enum（不导出）：不加，享受穷尽检查
- 公开 enum（库 API）：考虑加，保留向后兼容性
- ZeroClaw 的 `ObserverEvent`：如果作为库发布，应该加 `#[non_exhaustive]`

## 速查卡

```
深层内功：
  1. Niche 优化        Option<&T> 零开销，借用不可能值表示 None
  2. ZST 类型标记      PhantomData<State> 编译期状态机，零运行时开销
  3. #[repr] 控制布局   默认自由优化，repr(C) 用于 FFI
  4. match 跳转表       O(1) 分发，不是 if-else 链
  5. 字段重排           编译器自动减少 padding，别加 repr(C)
  6. Box 大变体         大变体拖累整个 enum，Box 放堆上
  7. #[non_exhaustive]  库 API 预留扩展，强制下游写 _ 分支
```
