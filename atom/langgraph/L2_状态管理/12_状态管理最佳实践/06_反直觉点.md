# 状态管理最佳实践 - 反直觉点

> 3 个最常见的误区，每个都是"看起来对，跑起来炸"的典型

---

## 误区1："状态越详细越好，把所有信息都存进去" ❌

**为什么错？**
- 状态每次节点执行都要序列化/反序列化，字段越多成本越高
- 检查点保存完整状态快照，状态越大存储和恢复越慢
- 派生数据存入状态会导致一致性问题——源数据改了，派生数据忘了同步

**为什么人们容易这样错？**
- 数据库思维：习惯把所有字段都建好，觉得"多存点总没错"
- 安全感：怕后面要用的时候没有，不如先存着

**正确理解：只存"源数据"，派生数据运行时计算。**

```python
from typing import TypedDict, Annotated
import operator

# ❌ 错误：把派生数据也塞进状态
class BadState(TypedDict):
    docs: Annotated[list[str], operator.add]
    doc_count: int          # 派生：len(docs) 就能算
    has_docs: bool          # 派生：bool(docs) 就能算
    doc_summary: str        # 派生：可以在需要时生成

# ✅ 正确：只存源数据，派生值用函数计算
class GoodState(TypedDict):
    docs: Annotated[list[str], operator.add]
    query: str

def get_doc_count(state: GoodState) -> int:
    return len(state["docs"])  # 需要时再算，永远和 docs 一致
```

## 误区2："Reducer 只是简单的合并函数，随便写就行" ❌

**为什么错？**
- 并行节点的更新到达 Reducer 的顺序不确定
- Reducer 不满足结合律和交换律时，不同执行顺序会产生不同结果
- 有副作用的 Reducer（写日志、用随机数）会导致不可复现的行为

**为什么人们容易这样错？**
- 单节点测试时随便写确实能跑，问题只在并发时暴露
- "合并两个值"听起来太简单了，不觉得需要数学约束

**正确理解：Reducer 是状态更新的"合约"，必须是纯函数且满足数学性质。**

```python
# ❌ 错误：减法不满足结合律，并发结果不确定
def bad_reducer(current: int, new: int) -> int:
    return current - new
# (10 - 3) - 2 = 5  vs  10 - (3 - 2) = 9  → 结果不一致！

# ❌ 错误：Reducer 里搞副作用
def bad_reducer_2(current: list, new: list) -> list:
    import random
    combined = current + new
    random.shuffle(combined)  # 每次结果不同！
    return combined

# ✅ 正确：纯函数 + 满足结合律和交换律
def safe_merge(current: set, new: set) -> set:
    return current | new  # 集合并集，顺序无关，结果确定
```

## 误区3："直接修改状态对象更高效" ❌

**为什么错？**
- LangGraph 通过 `BinaryOperatorAggregate.update()` 的返回值判断状态是否变化，直接修改绕过了这个检测
- `StateSnapshot` 是不可变的 `NamedTuple`，框架设计上就假定状态不会被原地修改
- 并发节点拿到的是浅拷贝，直接 `append` 会污染其他节点看到的数据

**为什么人们容易这样错？**
- Python 中 `list.append()` 和 `dict["key"] = val` 是最自然的操作
- 创建新对象看起来"浪费内存"，直觉上原地修改更快

**正确理解：返回增量更新字典，让框架通过 Reducer 处理合并。**

```python
from typing import TypedDict, Annotated
import operator

class State(TypedDict):
    items: Annotated[list[str], operator.add]
    counter: Annotated[int, operator.add]

# ❌ 错误：直接修改输入状态
def bad_node(state: State) -> dict:
    state["items"].append("new")   # 污染共享引用！
    state["counter"] += 1          # 绕过 Reducer！
    return state

# ✅ 正确：返回增量，让 Reducer 负责合并
def good_node(state: State) -> dict:
    return {
        "items": ["new"],   # Reducer(operator.add) 负责拼接
        "counter": 1        # Reducer(operator.add) 负责累加
    }
```

---

## 一句话记住

> 状态要精简不要贪多，Reducer 要纯粹不要随意，更新要返回不要修改。
