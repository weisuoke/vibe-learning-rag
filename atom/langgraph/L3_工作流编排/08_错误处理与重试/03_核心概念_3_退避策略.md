## 什么是退避策略？

**退避策略 = 每次重试之间等待时间的增长规则。**

核心思想：**每次重试都比上次等更久**，给故障服务恢复的时间。

```
❌ 固定间隔重试：    0.5s → 0.5s → 0.5s → 0.5s  （持续施压）
✅ 指数退避重试：    0.5s → 1.0s → 2.0s → 4.0s  （逐渐放松）
✅ 指数退避+抖动：   0.7s → 1.3s → 2.8s → 4.1s  （分散压力）
```

---

## 源码解析：退避计算

```python
# 来自 pregel/_retry.py 的 run_with_retry()
interval = min(
    retry_policy.max_interval,                          # 上限保护
    retry_policy.initial_interval                       # 基础间隔
    * retry_policy.backoff_factor ** (attempt - 1),     # 指数增长
)
if retry_policy.jitter:
    interval += random.uniform(0, 1)                    # 随机抖动
time.sleep(interval)                                    # 等待
```

**数学公式：**

```
interval = min(max_interval, initial_interval × backoff_factor^(attempt-1))

如果 jitter=True:
    interval += random(0, 1)
```

## 指数退避详解

### 默认参数下的退避序列

```
参数：initial_interval=0.5, backoff_factor=2.0, max_interval=128.0

尝试次数    计算过程                    等待时间（无抖动）
─────────────────────────────────────────────────────
第1次重试   0.5 × 2.0^0 = 0.5         0.5 秒
第2次重试   0.5 × 2.0^1 = 1.0         1.0 秒
第3次重试   0.5 × 2.0^2 = 2.0         2.0 秒
第4次重试   0.5 × 2.0^3 = 4.0         4.0 秒
第5次重试   0.5 × 2.0^4 = 8.0         8.0 秒
第6次重试   0.5 × 2.0^5 = 16.0        16.0 秒
第7次重试   0.5 × 2.0^6 = 32.0        32.0 秒
第8次重试   0.5 × 2.0^7 = 64.0        64.0 秒
第9次重试   0.5 × 2.0^8 = 128.0       128.0 秒 ← 触及上限
第10次重试  0.5 × 2.0^9 = 256.0       128.0 秒 ← 被 max_interval 限制
```

**总等待时间累计：**
- 前 3 次重试：0.5 + 1.0 + 2.0 = **3.5 秒**（默认 max_attempts=3 会在此结束）
- 前 5 次重试：0.5 + 1.0 + 2.0 + 4.0 + 8.0 = **15.5 秒**
- 前 8 次重试：约 **127.5 秒**（约 2 分钟）

### 可视化退避增长

```
时间轴（默认参数，max_attempts=5）：

|--0.5s--|---1.0s---|-----2.0s-----|----------4.0s----------|
执行1     执行2      执行3          执行4                     执行5
(失败)    (失败)     (失败)         (失败)                    (成功或放弃)

总等待：约 7.5 秒
```

## 抖动（Jitter）详解

### 什么是抖动？

**抖动 = 在退避间隔上添加一个小的随机值（0~1秒）。**

```python
# LangGraph 源码实现
if retry_policy.jitter:
    interval += random.uniform(0, 1)  # 添加 0~1 秒随机值
```

### 为什么需要抖动？

**防止"雷群效应"（Thundering Herd Problem）。**

```
场景：LLM API 临时过载，100 个 Agent 同时遇到 429 错误

❌ 无抖动 — 同步重试风暴：
  t=0.0s: 100个请求全部失败
  t=0.5s: 100个请求同时重试 → API 再次过载
  t=1.0s: 100个请求同时重试 → API 再次过载
  ↓ 死循环，API 永远无法恢复

✅ 有抖动 — 分散重试：
  t=0.0s: 100个请求全部失败
  t=0.5~1.5s: 请求分散在 1 秒窗口内到达
  t=1.0~2.0s: 请求分散在 1 秒窗口内到达
  ↓ API 逐个处理，负载平稳，逐步恢复
```

### 抖动效果对比

```
3 个客户端同时失败，backoff_factor=2.0, initial_interval=0.5

无抖动：
  客户端A: |0.5s|1.0s|2.0s|      所有客户端
  客户端B: |0.5s|1.0s|2.0s|  ←── 完全同步！
  客户端C: |0.5s|1.0s|2.0s|

有抖动（+random(0,1)）：
  客户端A: |0.7s|1.3s|2.8s|      每个客户端
  客户端B: |1.2s|1.9s|2.1s|  ←── 时间不同！
  客户端C: |0.9s|1.5s|2.6s|
```

### 日常生活类比

```
❌ 无抖动：停电恢复后，整个小区的空调同时启动
   → 电网瞬间过载 → 再次跳闸

✅ 有抖动：停电恢复后，每户随机延迟 0~60 秒启动
   → 电网负载平稳上升 → 正常运行
```

## 不同退避策略的对比

### 1. 固定间隔（不推荐）

```python
# backoff_factor=1.0 → 固定间隔
RetryPolicy(initial_interval=1.0, backoff_factor=1.0)
# 等待序列：1.0s → 1.0s → 1.0s → ...
```

**问题：** 持续给服务器施加恒定压力，不给恢复喘息空间。

### 2. 线性退避

```python
# LangGraph 不直接支持线性退避，但可以用自定义逻辑模拟
# 等待序列：1.0s → 2.0s → 3.0s → 4.0s → ...
```

**问题：** 增长太慢，对严重故障场景等待不够。

### 3. 指数退避（LangGraph 默认）

```python
RetryPolicy(initial_interval=0.5, backoff_factor=2.0)
# 等待序列：0.5s → 1.0s → 2.0s → 4.0s → 8.0s → ...
```

**优势：** 增长快，快速给服务器足够的恢复时间。

### 4. 指数退避 + 抖动（LangGraph 默认 + 推荐）

```python
RetryPolicy(initial_interval=0.5, backoff_factor=2.0, jitter=True)
# 等待序列：0.5+r → 1.0+r → 2.0+r → 4.0+r → ... （r 为 0~1 随机值）
```

**最优选择：** 既给恢复时间，又避免雷群效应。

### 对比可视化

```
服务器负载视角（10 个客户端同时重试）：

固定间隔:     |||||||||| |||||||||| |||||||||| ||||||||||
              (全部堆积)  (全部堆积)  (全部堆积)  (全部堆积)

指数退避:     |||||||||| |||||||||| ||||||||||  ||||||||||
              (堆积)     (堆积)     (间隔更大但仍堆积)

指数+抖动:    | || ||| |   ||| | ||   || | |||   | ||| ||
              (分散)      (分散)      (分散)      (分散)
              ← 这才是服务器喜欢的 ✅
```

## 异步退避

LangGraph 同时提供同步和异步版本：

```python
# 同步版本 — run_with_retry()
time.sleep(interval)

# 异步版本 — arun_with_retry()
await asyncio.sleep(interval)
```

**关键区别：**
- `time.sleep()` 阻塞线程
- `asyncio.sleep()` 不阻塞事件循环，其他任务可以继续执行

在 LangGraph 的异步工作流中，退避等待期间其他节点可以并行执行，不浪费资源。

## 实际场景的推荐配置

| 场景 | initial_interval | backoff_factor | max_interval | 说明 |
|------|------------------|----------------|--------------|------|
| LLM API (OpenAI) | 1.0 | 2.0 | 60 | 速率限制恢复需时间 |
| 向量数据库 | 0.3 | 2.0 | 10 | 通常快速恢复 |
| 外部搜索 API | 2.0 | 2.0 | 30 | 第三方服务不可控 |
| 内部微服务 | 0.1 | 2.0 | 5 | 内部网络延迟低 |
| 测试环境 | 0.01 | 2.0 | 1 | 加速测试 |

## 前端类比

```
指数退避 ≈ 前端轮询的策略

// ❌ 固定间隔轮询
setInterval(() => fetch('/api/status'), 1000);

// ✅ 指数退避轮询
let delay = 1000;
function poll() {
    fetch('/api/status')
        .then(handle)
        .catch(() => {
            delay = Math.min(delay * 2, 30000);  // 指数增长，上限 30s
            setTimeout(poll, delay + Math.random() * 1000);  // + 抖动
        });
}
```
