# 核心概念 02：节点级状态分组

> 每个节点可以通过类型注解"只看到"状态的一个子集——不同节点接收不同的状态视图，实现关注点分离。

## 一句话定义

**节点级状态分组是指每个节点可以声明自己只接收状态的某些字段，LangGraph 通过函数参数的类型注解自动推断节点的输入 Schema，实现"每个节点只看到自己需要的数据"。**

---

## 为什么需要这个？

### 问题场景

一个复杂的 RAG 系统，状态可能有十几个字段：

```python
class RAGState(TypedDict):
    query: str
    chat_history: list
    retrieved_docs: list[str]
    rerank_scores: list[float]
    selected_docs: list[str]
    prompt_template: str
    llm_response: str
    answer: str
    metadata: dict
    token_count: int
    latency_ms: float
    error_log: list[str]
```

问题来了：
- **检索节点**只需要 `query`，为什么要看到 `rerank_scores`、`token_count`？
- **生成节点**只需要 `query` + `selected_docs`，为什么要看到 `error_log`？
- 当状态字段越来越多，每个节点函数里都要面对一个"巨大的"状态对象

这就像一个公司里，每个部门都能看到所有部门的数据——财务数据、人事数据、客户数据全部混在一起。

### 解决方案

让每个节点只声明自己需要的字段：

```python
class QueryOnly(TypedDict):
    query: str

class GenerateInput(TypedDict):
    query: str
    selected_docs: list[str]

def search_node(state: QueryOnly):        # 只看到 query
    return {"retrieved_docs": ["doc1", "doc2"]}

def generate_node(state: GenerateInput):  # 只看到 query + selected_docs
    return {"answer": f"基于 {len(state['selected_docs'])} 篇文档的回答"}
```

### 前端类比：组件 Props 解构

```
React 组件                              LangGraph 节点
─────────────                           ─────────────
// 全局 Store 有很多字段                  // OverallState 有很多字段

  user, cart, products,                     query, docs, scores,
  orders, settings, ...                     answer, metadata, ...
}

// 组件只取需要的字段                     // 节点只声明需要的字段

  return <Badge count={cart.length}/>       return {"docs": [...]}

function ProductList({ products }) {    def generate(state: GenerateInput):
  return products.map(...)                  return {"answer": "..."}

```

- React 组件通过 Props 接口声明需要的数据
- LangGraph 节点通过函数参数的类型注解声明需要的数据
- 两者都实现了"按需获取"——组件/节点不需要知道全局状态的全貌

### 日常生活类比：医院分诊

去医院看病：

- **主治医生**需要你的症状 + 检查结果（不需要知道你的缴费记录）

每个科室只接收与自己职责相关的信息，而不是你的完整病历。

## 详细解释

### 1. 两种设置节点 input_schema 的方式

#### 方式一：显式指定（通过 `add_node` 参数）

```python
builder.add_node("search", search_node, input_schema=QueryOnly)
```

#### 方式二：自动推断（通过函数参数类型注解）——推荐

```python
def search_node(state: QueryOnly):  # LangGraph 自动推断 input_schema = QueryOnly
    ...

builder.add_node("search", search_node)  # 不需要显式指定
```

LangGraph 会检查函数的第一个参数的类型注解，如果它是一个带有 type hints 的类（如 TypedDict、Pydantic BaseModel），就自动将其作为该节点的 `input_schema`。

### 2. 自动推断的源码机制

```python
# 来源: libs/langgraph/langgraph/graph/state.py（简化版）
def add_node(self, node, action=None, *, input_schema=None, ...):
    # 获取节点函数
    func = action or node

    # 如果没有显式指定 input_schema，尝试自动推断
    if input_schema is None:
        # 获取函数签名的类型提示
        hints = get_type_hints(func)
        # 获取第一个参数的类型
        first_param = list(hints.values())[0] if hints else None

        if first_param is not None and hasattr(first_param, '__annotations__'):
            # 第一个参数是带类型注解的类 → 用作 input_schema
            inferred_input_schema = first_param
        else:
            inferred_input_schema = None

    if input_schema is not None:
        self.nodes[node] = StateNodeSpec(
            action, metadata, input_schema=input_schema, ...
        )
    elif inferred_input_schema is not None:
        self.nodes[node] = StateNodeSpec(
            action, metadata, input_schema=inferred_input_schema, ...
        )
```

**推断条件**：
1. 函数第一个参数有类型注解
2. 该类型注解是一个"带 type hints 的类"（TypedDict、dataclass、Pydantic BaseModel 等）
3. 如果是普通的 `dict` 或 `Any`，则不推断

### 3. 节点 input_schema 的作用

节点的 `input_schema` 影响的是 **ChannelRead 的行为**——框架在调用节点前，会根据 input_schema 从 Channel 池中只读取对应的字段，然后将这些字段组装成 input_schema 类型的对象传给节点。

```
Channel 池（完整状态）                    节点接收的状态
┌──────────────────────┐
│ query: "什么是RAG？"  │ ──→ ┌─────────────────────┐
│ docs: ["doc1", ...]  │     │ query: "什么是RAG？"  │  ← search_node 只看到这个
│ scores: [0.9, ...]   │     └─────────────────────┘
│ answer: "..."        │
│ metadata: {...}      │ ──→ ┌─────────────────────────────┐
│ token_count: 150     │     │ query: "什么是RAG？"          │  ← generate_node 看到这两个
└──────────────────────┘     │ selected_docs: ["doc1", ...] │
                             └─────────────────────────────┘
```

### 4. 写入不受 input_schema 限制

关键点：**节点的 input_schema 只限制"读取"，不限制"写入"**。

节点可以返回任何在 OverallState 中定义的字段，即使这些字段不在节点的 input_schema 中：

```python
class QueryOnly(TypedDict):
    query: str

def search_node(state: QueryOnly):
    # 只读取 query（受 input_schema 限制）
    query = state["query"]

    # 但可以写入任何 OverallState 中的字段（不受限制）
    return {
        "retrieved_docs": search(query),    # 不在 QueryOnly 中，但可以写入
        "metadata": {"search_time": 0.5},   # 不在 QueryOnly 中，但可以写入
    }
```

### 5. Mapper 机制：类型转换

当节点的 input_schema 是 Pydantic BaseModel 时，框架会自动将 Channel 中的原始值转换（coerce）为 Pydantic 对象：

```python
from pydantic import BaseModel

class QueryInput(BaseModel):
    query: str
    max_results: int = 10  # 带默认值

def search_node(state: QueryInput):
    # state 是一个 Pydantic 对象，可以用属性访问
    print(state.query)        # 属性访问
    print(state.max_results)  # 默认值自动填充
```

框架内部通过 `mapper` 函数实现这个转换：

```python
# 来源: libs/langgraph/langgraph/graph/state.py（简化版）
# 如果 input_schema 是 Pydantic BaseModel
mapper = lambda data: input_schema(**data)  # 将 dict 转换为 Pydantic 对象
```

## 代码示例

### 示例 1：基础节点级状态分组

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# ========== 完整状态 ==========
class FullState(TypedDict):
    query: str
    context: list[str]
    answer: str
    metadata: dict

# ========== 节点级子状态 ==========
class QueryState(TypedDict):
    """检索节点只需要 query"""
    query: str

class ContextState(TypedDict):
    """生成节点需要 query + context"""
    query: str
    context: list[str]

# ========== 节点函数 ==========
def search_node(state: QueryState):
    """自动推断 input_schema = QueryState"""
    print(f"search_node 收到的 state 类型: {type(state)}")
    print(f"search_node 收到的字段: {list(state.keys())}")
    # 只能看到 query
    return {"context": [f"关于'{state['query']}'的文档1", f"关于'{state['query']}'的文档2"]}

def answer_node(state: ContextState):
    """自动推断 input_schema = ContextState"""
    print(f"answer_node 收到的 state 类型: {type(state)}")
    print(f"answer_node 收到的字段: {list(state.keys())}")
    # 能看到 query 和 context
    return {
        "answer": f"基于 {len(state['context'])} 篇文档回答: {state['query']}",
        "metadata": {"doc_count": len(state["context"])},
    }

# ========== 构建图 ==========
builder = StateGraph(FullState)
builder.add_node("search", search_node)   # 自动推断: QueryState
builder.add_node("answer", answer_node)   # 自动推断: ContextState
builder.add_edge(START, "search")
builder.add_edge("search", "answer")
builder.add_edge("answer", END)

graph = builder.compile()

# ========== 执行 ==========
result = graph.invoke({
    "query": "什么是RAG？",
    "context": [],
    "answer": "",
    "metadata": {},
})

print(f"\n最终结果: {result}")
# search_node 收到的字段: ['query']
# answer_node 收到的字段: ['query', 'context']
# 最终结果: {'query': '什么是RAG？', 'context': [...], 'answer': '基于 2 篇文档回答: 什么是RAG？', 'metadata': {'doc_count': 2}}
```

### 示例 2：结合 Input/Output Schema 的完整分组

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
import operator

# ========== 图级 Schema ==========
class UserInput(TypedDict):
    question: str

class FinalOutput(TypedDict):
    answer: str

# ========== 完整内部状态 ==========
class InternalState(TypedDict):
    question: str
    answer: str
    raw_docs: list[str]
    filtered_docs: list[str]
    confidence: float

# ========== 节点级子状态 ==========
class RetrieveInput(TypedDict):
    question: str

class FilterInput(TypedDict):
    raw_docs: list[str]

class GenerateInput(TypedDict):
    question: str
    filtered_docs: list[str]

# ========== 节点函数 ==========
def retrieve(state: RetrieveInput):
    """检索节点：只看到 question"""
    return {"raw_docs": [f"doc about {state['question']}"]}

def filter_docs(state: FilterInput):
    """过滤节点：只看到 raw_docs"""
    return {
        "filtered_docs": [d for d in state["raw_docs"] if len(d) > 0],
        "confidence": 0.85,
    }

def generate(state: GenerateInput):
    """生成节点：只看到 question + filtered_docs"""
    docs = "\n".join(state["filtered_docs"])
    return {"answer": f"回答 '{state['question']}': 基于 {docs}"}

# ========== 构建图 ==========
builder = StateGraph(
    InternalState,
    input_schema=UserInput,
    output_schema=FinalOutput,
)
builder.add_node("retrieve", retrieve)
builder.add_node("filter", filter_docs)
builder.add_node("generate", generate)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "filter")
builder.add_edge("filter", "generate")
builder.add_edge("generate", END)

graph = builder.compile()

result = graph.invoke({"question": "向量数据库是什么？"})
print(result)
# {'answer': "回答 '向量数据库是什么？': 基于 doc about 向量数据库是什么？"}
```

### 示例 3：显式指定 vs 自动推断

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class FullState(TypedDict):
    x: int
    y: int
    z: int

class XOnly(TypedDict):
    x: int

class XYOnly(TypedDict):
    x: int
    y: int

# 方式一：自动推断（推荐）
def node_a(state: XOnly):
    return {"y": state["x"] * 2}

# 方式二：函数用 dict 注解，显式指定 input_schema
def node_b(state: dict):
    return {"z": state["x"] + state["y"]}

builder = StateGraph(FullState)
builder.add_node("a", node_a)                              # 自动推断 XOnly
builder.add_node("b", node_b, input_schema=XYOnly)         # 显式指定 XYOnly
builder.add_edge(START, "a")
builder.add_edge("a", "b")
builder.add_edge("b", END)

graph = builder.compile()
result = graph.invoke({"x": 5, "y": 0, "z": 0})
print(result)  # {'x': 5, 'y': 10, 'z': 15}
```

## 源码解析

### 关键源码 1：函数签名检查

LangGraph 在 `add_node()` 时检查函数的第一个参数类型：

```python
# 来源: libs/langgraph/langgraph/graph/state.py（简化版）
from typing import get_type_hints
import inspect

def _infer_input_schema(func):
    """从函数签名推断 input_schema"""
    try:
        hints = get_type_hints(func)
    except Exception:
        return None

    # 获取第一个参数名
    sig = inspect.signature(func)
    params = list(sig.parameters.keys())
    if not params:
        return None

    first_param_name = params[0]
    first_param_type = hints.get(first_param_name)

    if first_param_type is None:
        return None

    # 检查是否是带 type hints 的类
    try:
        type_hints = get_type_hints(first_param_type)
        if type_hints:
            return first_param_type
    except Exception:
        pass

    return None
```

### 关键源码 2：ChannelRead 的 Schema 过滤

节点执行前，ChannelRead 根据节点的 input_schema 决定读取哪些 Channel：

```python
# 来源: libs/langgraph/langgraph/graph/state.py（简化版）
# 编译时为每个节点生成 ChannelRead
if node_spec.input_schema:
    # 只读取 input_schema 中定义的字段
    read_channels = list(get_type_hints(node_spec.input_schema).keys())
else:
    # 读取所有 state_schema 的字段
    read_channels = list(get_type_hints(self.state_schema).keys())

# 生成 ChannelRead 操作
channel_read = ChannelRead(
    channel=read_channels,
    mapper=lambda data: node_spec.input_schema(**data) if is_pydantic else data,
)
```

### 关键源码 3：StateNodeSpec 数据结构

每个节点在内部表示为 `StateNodeSpec`，包含 input_schema 信息：

```python
# 来源: libs/langgraph/langgraph/graph/state.py
@dataclass
class StateNodeSpec:
    """节点的内部表示"""
    runnable: Runnable          # 节点函数
    metadata: dict              # 元数据
    input_schema: type | None   # 节点级 input_schema
    retry_policy: RetryPolicy | None
    # ...
```

## 在 LangGraph 开发中的应用

### 场景 1：大型 RAG Pipeline 的关注点分离

```python
# 每个阶段只关心自己需要的数据
class EmbeddingInput(TypedDict):
    chunks: list[str]

class SearchInput(TypedDict):
    query_embedding: list[float]
    top_k: int

class RerankInput(TypedDict):
    query: str
    candidates: list[str]

class GenerateInput(TypedDict):
    query: str
    context: list[str]
    system_prompt: str
```

好处：
- 每个节点函数的参数清晰明了
- 修改一个阶段不影响其他阶段
- 代码审查时一眼就能看出每个节点的数据依赖

### 场景 2：团队协作

不同开发者负责不同节点时，节点级状态分组让每个人只需要关心自己的"接口"：

```python
# 开发者 A 负责检索模块
class RetrieverInput(TypedDict):
    query: str
    filters: dict

# 开发者 B 负责生成模块
class GeneratorInput(TypedDict):
    query: str
    context: list[str]
    temperature: float
```

### 场景 3：测试隔离

节点级状态分组让单元测试更简单——只需要构造节点需要的字段：

```python
# 测试 search_node 时，不需要构造完整的 FullState
def test_search_node():
    # 只需要 QueryState 的字段

    assert "context" in result
    assert len(result["context"]) > 0
```

## 常见问题

### Q1：自动推断什么时候不生效？

以下情况不会自动推断：
- 函数参数没有类型注解：`def node(state):`
- 类型注解是 `dict`：`def node(state: dict):`
- 类型注解是 `Any`：`def node(state: Any):`
- 类型注解是没有 `__annotations__` 的类型

### Q2：节点的 input_schema 字段必须是 OverallState 的子集吗？

是的。节点 input_schema 中的每个字段都必须在 OverallState（或已注册的某个 Schema）中有对应的 Channel。否则框架无法从 Channel 池中读取该字段。

### Q3：两个节点可以用相同的子状态类型吗？

当然可以。多个节点可以共享同一个子状态类型：

```python
class QueryState(TypedDict):
    query: str

def node_a(state: QueryState): ...
def node_b(state: QueryState): ...  # 和 node_a 用同一个类型
```

### Q4：节点级 input_schema 和图级 input_schema 有什么区别？

- **图级 input_schema**：控制图的外部接口——调用方传入什么
- **节点级 input_schema**：控制节点的内部视图——节点看到什么

两者是不同层级的概念，互不冲突。

---

## 小结

1. **核心机制**：每个节点可以通过函数参数的类型注解声明自己需要的状态子集
2. **自动推断**：LangGraph 检查函数第一个参数的类型，自动设置节点的 input_schema
3. **读取受限，写入自由**：input_schema 只限制节点"看到"的字段，不限制节点"写入"的字段
4. **关注点分离**：不同节点只关心自己需要的数据，降低耦合度
5. **实际价值**：让大型图的代码更清晰、更易测试、更适合团队协作
