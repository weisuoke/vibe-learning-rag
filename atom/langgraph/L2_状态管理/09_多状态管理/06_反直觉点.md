## 误区1："子图可以直接访问父图的所有状态" ❌

### 错误观点

"子图是父图的一部分，运行在同一个程序里，所以子图的节点应该能直接读取父图的状态字段。"

```python
class ParentState(TypedDict):
    query: str
    user_id: str
    config: dict

class SubgraphState(TypedDict):
    bar: str

def subgraph_node(state: SubgraphState) -> dict:
    # ❌ 错误预期：能直接访问父图的 query
    query = state["query"]  # KeyError! SubgraphState 里没有 query
    return {"bar": f"处理了 {query}"}
```

### 为什么错？

**核心事实：** 子图有完全独立的 Channel 集合。子图的 `SubgraphState` 定义了哪些 Channel 存在，父图的 `ParentState` 定义了另一套 Channel。两套 Channel 之间没有任何自动桥接。

子图和父图的关系不是"内部函数访问外部变量"，而是"两个独立服务通过 API 通信"。

**源码层面：** 每个 `StateGraph` 实例维护自己的 `self.channels` 字典。子图编译后是一个独立的 `CompiledGraph`，有自己的 Channel 注册表。父图调用子图时，走的是 `invoke()` 接口——和你从外部调用一个图完全一样。

```python
# 子图编译后的本质
subgraph = subgraph_builder.compile()

# 父图调用子图 = 外部调用子图，没有任何区别
# 以下两种调用方式在底层是等价的：

# 方式1：在父图节点中调用
def parent_node(state: ParentState) -> dict:
    result = subgraph.invoke({"bar": state["query"]})
    return {"query": result["bar"]}

# 方式2：在任意 Python 代码中调用
result = subgraph.invoke({"bar": "hello"})
```

**实际验证：**

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# === 子图 ===
class SubgraphState(TypedDict):
    bar: str

def sub_node(state: SubgraphState) -> dict:
    # 尝试访问父图的字段
    try:
        query = state["query"]  # 父图有 query，但子图没有
    except KeyError as e:
        print(f"KeyError: {e}")  # 会打印 KeyError: 'query'
    return {"bar": state["bar"] + " processed"}

subgraph_builder = StateGraph(SubgraphState)
subgraph_builder.add_node("sub", sub_node)
subgraph_builder.add_edge(START, "sub")
subgraph_builder.add_edge("sub", END)
subgraph = subgraph_builder.compile()

# === 父图 ===
class ParentState(TypedDict):
    query: str
    user_id: str

def call_sub(state: ParentState) -> dict:
    # 必须显式传递数据给子图
    result = subgraph.invoke({"bar": state["query"]})
    return {"query": result["bar"]}

builder = StateGraph(ParentState)
builder.add_node("call_sub", call_sub)
builder.add_edge(START, "call_sub")
builder.add_edge("call_sub", END)
graph = builder.compile()

result = graph.invoke({"query": "hello", "user_id": "u123"})
# 子图完全看不到 user_id，只能看到通过 invoke 传入的 bar
```

### 为什么人们容易这样错？

- **闭包思维**：Python 中内部函数可以访问外部变量，直觉上觉得子图也应该能访问父图
- **同一进程的错觉**：子图和父图在同一个 Python 进程中运行，感觉应该共享内存
- **组件树经验**：React 中子组件可以通过 Context 访问祖先的数据，但 LangGraph 子图不行

```
Python 闭包                    LangGraph 子图
──────────                    ──────────────
内部函数自动捕获外部变量        子图不能自动访问父图状态
共享同一个作用域               完全独立的 Channel 集合
无需显式传递                   必须通过 invoke 或共享键传递
```

### 正确理解

**子图是独立的"微服务"，不是"内部函数"。** 父子图通信只有两条路：

```python
# ✅ 方式1：函数包装 + 显式传参（Schema 不同时）
def call_subgraph(state: ParentState) -> dict:
    # 手动把父图字段映射到子图字段
    result = subgraph.invoke({
        "bar": state["query"],  # 显式传递
    })
    return {"query": result["bar"]}  # 显式接收

# ✅ 方式2：共享键 + 直接添加子图（有同名字段时）
class ParentState(TypedDict):
    foo: str  # 共享键

class SubgraphState(TypedDict):
    foo: str  # 同名 = 自动共享
    bar: str  # 私有

# 直接添加编译后的子图，foo 自动在父子图之间传递
builder.add_node("subgraph", compiled_subgraph)
```

**如果子图需要父图的多个字段，要么：**
1. 在共享键中包含这些字段
2. 在函数包装中显式传递
3. 把需要的数据打包成一个字段传入

```python
# ✅ 打包传递多个字段
def call_subgraph(state: ParentState) -> dict:
    result = subgraph.invoke({
        "input_data": {
            "query": state["query"],
            "user_id": state["user_id"],
            "config": state["config"],
        }
    })
    return {"result": result["output"]}
```

**记忆口诀：** "子图是独立服务，不是内部函数——要数据？走接口"

## 误区2："Input/Output Schema 会创建额外的状态字段" ❌

### 错误观点

"定义 `InputState`、`OutputState`、`OverallState` 三个 Schema，LangGraph 会创建三套独立的状态字段。InputState 的 `question` 和 OverallState 的 `question` 是两个不同的变量。"

```python
class InputState(TypedDict):
    question: str       # ❌ 错误理解：这是 InputState 的 question

class OutputState(TypedDict):
    answer: str         # ❌ 错误理解：这是 OutputState 的 answer

class OverallState(TypedDict):
    question: str       # ❌ 错误理解：这是 OverallState 的 question（另一个变量）
    answer: str         # ❌ 错误理解：这是 OverallState 的 answer（另一个变量）
    context: str

# ❌ 错误预期：InputState.question 和 OverallState.question 是两个独立的存储
```

### 为什么错？

**核心事实：** 所有 Schema 共享同一个 Channel 池。`InputState` 的 `question`、`OverallState` 的 `question` 指向的是同一个 Channel。Schema 不创建存储，Schema 只是 Channel 池的"视图"（过滤器）。

**源码层面：** `_add_schema()` 方法将所有 Schema 的字段合并到统一的 `self.channels` 字典中：

```python
# StateGraph 内部（简化）
class StateGraph:
    def __init__(self, state_schema, input_schema=None, output_schema=None, ...):
        self.channels = {}   # 统一的 Channel 池
        self.schemas = {}    # Schema → 字段映射

        # 所有 Schema 的字段都合并到同一个 channels 字典
        self._add_schema(state_schema)    # question, answer, context → channels
        self._add_schema(input_schema)    # question → 已存在，跳过
        self._add_schema(output_schema)   # answer → 已存在，跳过

    def _add_schema(self, schema):
        channels, _, _ = _get_channels(schema)
        for key, channel in channels.items():
            if key in self.channels:
                # 同名字段：检查兼容性，不创建新 Channel
                pass
            else:
                # 新字段：注册到 Channel 池
                self.channels[key] = channel
```

**关键机制：**
- `input_schema` 的作用：START 节点只写入 `input_schema` 中定义的字段
- `output_schema` 的作用：图的返回值只包含 `output_schema` 中定义的字段
- 它们是**过滤器**，不是**存储**

**实际验证：**

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class InputState(TypedDict):
    question: str

class OutputState(TypedDict):
    answer: str

class OverallState(TypedDict):
    question: str
    answer: str
    context: str

def node_1(state: OverallState) -> dict:
    # InputState 的 question 和 OverallState 的 question 是同一个值
    print(f"question = {state['question']}")  # "你好"
    return {"context": "检索结果", "answer": "回答"}

builder = StateGraph(OverallState, input_schema=InputState, output_schema=OutputState)
builder.add_node("node_1", node_1)
builder.add_edge(START, "node_1")
builder.add_edge("node_1", END)
graph = builder.compile()

# 输入 question = "你好"
result = graph.invoke({"question": "你好"})

# 输出只有 answer（OutputState 过滤）
print(result)  # {'answer': '回答'}
# question 和 context 存在于 Channel 池中，但被 OutputState 过滤掉了
```

### 为什么人们容易这样错？

- **面向对象思维**：三个 TypedDict 类 = 三个独立的对象实例
- **数据库表经验**：不同的表有不同的存储，即使列名相同
- **命名空间直觉**：`InputState.question` 和 `OverallState.question` 看起来像不同命名空间下的变量

```
面向对象思维                    LangGraph 实际机制
──────────────                 ──────────────────
三个类 = 三套存储               三个 Schema = 一个 Channel 池的三个视图
InputState.question ≠          InputState.question =
OverallState.question          OverallState.question（同一个 Channel）
```

### 正确理解

**Schema 是 Channel 池的"窗口"，不是"房间"。**

```python
# 正确的心智模型：

# Channel 池（唯一的存储）
channels = {
    "question": LastValue[str],    # 一个 Channel
    "answer": LastValue[str],      # 一个 Channel
    "context": LastValue[str],     # 一个 Channel
}

# InputState = 只看到 question 的窗口
# OutputState = 只看到 answer 的窗口
# OverallState = 看到所有字段的窗口

# 三个窗口看到的是同一个房间里的同一批家具
```

**这意味着：**
- 同名字段必须有兼容的类型和 Reducer
- 不能在 `InputState` 中定义 `question: str`，又在 `OverallState` 中定义 `question: int`
- 不能在一个 Schema 中给 `items` 加 `operator.add` Reducer，在另一个 Schema 中不加

```python
# ❌ 会报错：同名字段 Reducer 不兼容
class SchemaA(TypedDict):
    items: Annotated[list, operator.add]  # 有 Reducer

class SchemaB(TypedDict):
    items: Annotated[list, custom_reducer]  # 不同的 Reducer → 冲突！
```

**记忆口诀：** "Schema 是窗口不是房间——三个窗口看的是同一个 Channel 池"

## 误区3："context_schema 的数据会被 checkpoint 保存" ❌

### 错误观点

"context_schema 定义的字段和普通状态字段一样，会被 checkpoint 保存。所以不能把 API key 放在 context 里，因为它会被持久化到数据库。"

```python
class MyContext(TypedDict):
    api_key: str
    model_name: str

# ❌ 错误担心：api_key 会被保存到 checkpoint 数据库
builder = StateGraph(State, context_schema=MyContext)
```

### 为什么错？

**核心事实：** `context_schema` 不创建 Channel。它的数据通过 `Runtime` 对象在运行时传递，完全不参与 Channel 系统，因此不会被 checkpoint 保存。

**源码层面：** 对比 `state_schema` 和 `context_schema` 的处理方式：

```python
# StateGraph.__init__() 内部（简化）

# state_schema → 创建 Channel → 参与 checkpoint
self._add_schema(state_schema)
# _add_schema 调用 _get_channels()，为每个字段创建 Channel
# Channel 的值会被 checkpointer 序列化保存

# context_schema → 不创建 Channel → 不参与 checkpoint
self.context_schema = context_schema
# 只是保存了类型定义，没有调用 _add_schema()
# 没有 Channel = 没有持久化
```

**数据流对比：**

```
state_schema 的数据流：
  invoke(input) → Channel 写入 → 节点读取 → Channel 更新 → checkpoint 保存
                                                              ↑ 会被持久化

context_schema 的数据流：
  invoke(input, context={...}) → Runtime 对象 → 节点读取
                                                  ↑ 只在内存中，不持久化
```

**实际验证：**

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.runtime import Runtime
from langgraph.checkpoint.memory import MemorySaver

class MyContext(TypedDict):
    api_key: str

class State(TypedDict):
    query: str
    answer: str

def my_node(state: State, runtime: Runtime[MyContext]) -> dict:
    key = runtime.context["api_key"]
    return {"answer": f"使用 key={key[:4]}*** 回答了 {state['query']}"}

builder = StateGraph(State, context_schema=MyContext)
builder.add_node("node", my_node)
builder.add_edge(START, "node")
builder.add_edge("node", END)

# 使用 MemorySaver 作为 checkpointer
checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 调用时传入 context
config = {"configurable": {"thread_id": "test-1"}}
result = graph.invoke(
    {"query": "什么是 RAG？"},
    config=config,
    context={"api_key": "sk-secret-key-12345"},
)
print(result["answer"])
# 使用 key=sk-s*** 回答了 什么是 RAG？

# 检查 checkpoint 中保存了什么
state = graph.get_state(config)
print(state.values)
# {'query': '什么是 RAG？', 'answer': '使用 key=sk-s*** 回答了 什么是 RAG？'}
# 注意：api_key 不在 checkpoint 中！
```

### 为什么人们容易这样错？

- **"传给图的数据都会被保存"的直觉**：`invoke` 的 `input` 会被保存，所以 `context` 也应该被保存
- **混淆 config 和 context**：`config` 中的 `configurable` 确实会影响 checkpoint 的存储键，容易和 `context` 混淆
- **安全意识过度泛化**：知道"不要把敏感数据放在会被持久化的地方"，但不确定 context 是否会被持久化

```
invoke 的 input                context 参数
──────────────                ──────────────
写入 Channel                  传入 Runtime 对象
被 checkpoint 保存            不被 checkpoint 保存
参与状态更新                  只读，不参与状态更新
```

### 正确理解

**context 是"运行时注入"，不是"状态字段"。** 它的生命周期只在一次 `invoke` 调用中。

```python
# ✅ 适合放在 context 中的数据：
context = {
    "api_key": "sk-...",           # 敏感凭证（不会被持久化）
    "model_name": "gpt-4",        # 运行时配置（每次调用可能不同）
    "user_role": "admin",         # 用户身份（不需要保存在状态中）
    "temperature": 0.7,           # 模型参数（运行时决定）
}

# ❌ 不适合放在 context 中的数据：
# 需要在节点间传递和修改的 → 放在 State 中
# 需要被 checkpoint 恢复的 → 放在 State 中
# 需要被 Reducer 处理的 → 放在 State 中
```

**context 的三个特性：**

| 特性 | 说明 |
|------|------|
| 只读 | 节点不能修改 context，只能读取 |
| 不持久化 | 不创建 Channel，不被 checkpoint 保存 |
| 每次调用独立 | 每次 `invoke` 可以传不同的 context |

**这正是 context 的设计意图：** 提供一个安全的通道，让敏感数据和运行时配置能够到达节点，但不会被意外持久化或泄露到状态快照中。

**记忆口诀：** "context 是临时工牌，不是员工档案——用完即走，不留记录"

## 额外提醒：容易忽略的细节

### 细节1：节点的 input schema 只控制"读"，不控制"写"

```python
class QueryState(TypedDict):
    query: str

class OverallState(TypedDict):
    query: str
    answer: str
    context: str

def my_node(state: QueryState) -> dict:
    # 读：只能读到 QueryState 的字段（query）
    q = state["query"]

    # 写：可以写入任何 Channel 的字段！
    return {
        "answer": "回答",     # ✅ 可以写入 OverallState 的字段
        "context": "上下文",   # ✅ 可以写入 OverallState 的字段
    }
```

input schema 是"读取过滤器"，不是"写入限制器"。节点可以向 Channel 池中的任何字段写入。

### 细节2：同名字段的 Reducer 必须一致

```python
class SchemaA(TypedDict):
    items: Annotated[list, operator.add]

class SchemaB(TypedDict):
    items: list  # 没有 Reducer

# 这两个 Schema 可以共存吗？
# ✅ 可以！LastValue（无 Reducer）与任何 Channel 兼容
# LangGraph 会使用 SchemaA 中定义的 operator.add

# 但如果两个 Schema 都有 Reducer 且不同：
class SchemaC(TypedDict):
    items: Annotated[list, operator.add]

class SchemaD(TypedDict):
    items: Annotated[list, custom_reducer]

# ❌ 会报错：同名字段的 Reducer 冲突
```

### 细节3：子图的 checkpoint 命名空间是自动隔离的

```python
# 命名空间格式
""                              # 根图
"agent:abc123"                  # 一级子图
"agent:abc123|researcher:def456"  # 二级子图

# 每层子图的 checkpoint 存储在不同的命名空间下
# 即使父子图有同名字段，checkpoint 也不会冲突
```

## 误区总结表

| 误区 | 错误观点 | 正确理解 | 记忆口诀 |
|------|----------|----------|----------|
| 误区1 | 子图能直接访问父图状态 | 子图有独立的 Channel 集合，必须通过接口通信 | "子图是独立服务，不是内部函数" |
| 误区2 | 多个 Schema 创建多套存储 | 所有 Schema 共享一个 Channel 池，Schema 只是过滤器 | "Schema 是窗口不是房间" |
| 误区3 | context 数据会被 checkpoint 保存 | context 不创建 Channel，不参与持久化 | "context 是临时工牌，不留记录" |

---

## 避免误区的检查清单

- [ ] 子图需要父图数据时，通过共享键或函数包装显式传递，不依赖"自动访问"
- [ ] 理解 Input/Output Schema 是过滤器，不是独立存储——同名字段指向同一个 Channel
- [ ] 同名字段的 Reducer 定义保持一致，避免冲突
- [ ] 敏感数据（API key、token）放在 context 中，而非 State 中
- [ ] 需要被 checkpoint 恢复的数据放在 State 中，而非 context 中
- [ ] 节点的 input schema 只限制读取范围，不限制写入范围
