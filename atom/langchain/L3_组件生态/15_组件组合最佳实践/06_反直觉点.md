# 组件组合最佳实践 - 反直觉点

## 3个最常见的误区

### 误区1：重试应该包装整个链 ❌

**错误观点：** "为了安全，应该对整个链添加重试"

```python
# ❌ 错误做法
chain = (prompt | model | parser).with_retry(stop_after_attempt=3)
```

**为什么错？**
- 整个链重试意味着 prompt 构建也会重复执行（浪费）
- 如果 parser 失败，model 也会重新调用（浪费 token 和钱）
- 重试范围越大，副作用越不可控

**为什么人们容易这样错？**
因为在传统 Web 开发中，我们习惯在最外层添加重试（比如 HTTP 请求重试）。但 LLM 链不同，每个步骤的成本和失败概率差异很大。

**正确理解：**
```python
# ✅ 正确做法：只在可能失败的组件上重试
chain = prompt | model.with_retry(stop_after_attempt=3) | parser

# ✅ 更精细：不同组件不同策略
chain = (
    prompt
    | model.with_retry(
        retry_if_exception_type=(RateLimitError, APITimeoutError),
        stop_after_attempt=3,
    )
    | parser
)
```

**源码佐证：** retry.py 的文档明确说明：
> "it's best practice to keep the scope of the retry as small as possible. For example, if you have a chain of Runnables, you should only retry the Runnable that is likely to fail, not the entire chain."

### 误区2：RunnableLambda 和其他组件一样支持流式 ❌

**错误观点：** "用 `|` 连接的所有组件都自动支持流式传输"

```python
# ❌ 以为这样就能流式
chain = prompt | model | RunnableLambda(lambda x: x.content.upper())
for chunk in chain.stream(input):  # lambda 会阻断流式！
    print(chunk)
```

**为什么错？**
- RunnableLambda 默认不实现 `transform` 方法
- 流式传播需要每个组件都支持 transform
- RunnableLambda 会等待上游完整输出后才执行，阻断了流式传播

**为什么人们容易这样错？**
因为 `|` 操作符让所有组件看起来是"平等"的，人们自然以为它们的行为也一样。但实际上，流式传播是一个需要每个组件主动支持的特性。

**正确理解：**
```python
# ✅ 使用 RunnableGenerator 保持流式
from langchain_core.runnables import RunnableGenerator

def upper_stream(input_stream):
    for chunk in input_stream:
        yield chunk.upper()

chain = prompt | model | RunnableGenerator(upper_stream)

# ✅ 或者把 lambda 放在流式不重要的位置
chain = RunnableLambda(preprocess) | prompt | model | parser
# preprocess 在最前面，不影响 model 的流式输出
```

**源码佐证：** base.py RunnableSequence 的文档：
> "RunnableLambdas do not support transform by default! So if you need to use a RunnableLambdas be careful about where you place them in a RunnableSequence"

### 误区3：简单任务也应该用 LangChain ❌

**错误观点：** "既然学了 LangChain，所有 LLM 调用都应该用它"

```python
# ❌ 过度使用 LangChain
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

chain = (
    ChatPromptTemplate.from_template("{text}")
    | ChatOpenAI(model="gpt-4o-mini")
    | StrOutputParser()
)
result = chain.invoke({"text": "Hello"})
```

**为什么错？**
- 简单的单次 LLM 调用，LangChain 增加了不必要的抽象层
- 额外的依赖和复杂度
- 调试更困难（多了一层封装）
- 性能开销（虽然很小）

**为什么人们容易这样错？**
学了新工具就想到处用，这是"锤子综合症"（手里有锤子，看什么都像钉子）。

**正确理解：**
```python
# ✅ 简单任务直接用 SDK
from openai import OpenAI
client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Hello"}],
)
print(response.choices[0].message.content)

# ✅ 什么时候用 LangChain？
# - 多步骤管道（prompt → model → parser → ...）
# - 需要错误处理（fallback, retry）
# - 需要流式/批处理
# - 需要可观测性（tracing, callbacks）
# - 需要组件复用和替换
```

**判断标准：**

| 场景 | 推荐 |
|------|------|
| 单次 LLM 调用，固定 prompt | 直接 SDK |
| 多步骤管道 | LangChain LCEL |
| 需要 fallback/retry | LangChain |
| 需要条件路由 | LangChain |
| 需要循环/状态 | LangGraph |
| 需要可观测性 | LangChain + LangSmith |

## 额外反直觉点

### 反直觉4：batch 不一定比循环 invoke 快

batch 的优势在于 IO 并行，但如果组件本身是 CPU 密集型的，batch 可能不会更快（受 GIL 限制）。对于 LLM API 调用（IO 密集型），batch 确实更快。

### 反直觉5：Fallback 的 batch 不会重复成功的输入

很多人以为 fallback 的 batch 会对所有输入重新执行 fallback。实际上，源码中 fallback 的 batch 实现只对失败的输入尝试 fallback，已成功的不会重复执行。
