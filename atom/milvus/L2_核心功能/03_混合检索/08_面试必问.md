## 问题1：请解释Milvus混合检索的原理，以及为什么需要混合检索？

**问题：**

### 优秀回答（95分）

**回答：**

"混合检索是Milvus 2.5+引入的核心特性，它通过组合向量检索和BM25全文检索，利用RRF算法融合结果，从而兼顾语义理解和关键词匹配。

**为什么需要混合检索？**

单一检索方式存在局限性：

1. **纯向量检索的问题**：
   - 语义相似 ≠ 精确匹配
   - 例如搜索'Python 3.11新特性'，可能匹配到'Python 3.10新特性'（语义相似但版本不对）
   - 可能漏掉精确关键词（如版本号、产品型号）

2. **纯BM25检索的问题**：
   - 关键词匹配 ≠ 语义理解
   - 例如搜索'如何学习Python'，可能只匹配到'Python'和'学习'，但不理解'如何'的意图
   - 无法处理同义词（'怎样'、'方法'）

**混合检索的原理：**

```
1. 并行检索
   ├─ 稠密向量检索（HNSW索引，COSINE相似度）
   └─ 稀疏向量检索（倒排索引，BM25算法）

2. RRF重排序
   score_rrf(d) = Σ(1 / (k + rank_i(d)))
   - 基于排名位置而非分数
   - k=60（默认平滑参数）
   - 避免量纲问题

3. 返回TopK结果
```

**实际效果：**

在RAG文档检索场景中：
- Recall@10: 0.85 (纯向量) → 0.92 (混合) (+8%)
- Precision@10: 0.80 (纯向量) → 0.88 (混合) (+10%)
- 延迟: 45ms (纯向量) → 65ms (混合) (+44%)

**适用场景：**
- ✅ RAG文档问答系统
- ✅ 企业知识库检索
- ✅ 电商商品搜索
- ❌ 纯语义推荐（如音乐推荐）
- ❌ 延迟敏感场景（如实时推荐）

**技术优势（Milvus 2.6）：**
- 统一架构：无需Elasticsearch，降低运维成本
- 性能优化：4x于Elasticsearch，72%内存节省
- 内置BM25：自动生成稀疏向量，无需手动处理"

## 问题2：在混合检索中，如何选择Pre-filter和Post-filter策略？

### 普通回答（60分）

**回答：**
"Pre-filter是在检索前过滤，Post-filter是在检索后过滤。Pre-filter更快，所以一般用Pre-filter。"

**问题：**
- ❌ 过于简化，没有说明选择依据
- ❌ "Pre-filter更快"不一定正确
- ❌ 没有提到选择性的概念
- ❌ 缺少实际案例
- ❌ 没有性能数据支撑

---

### 优秀回答（95分）

**回答：**

"Pre-filter和Post-filter的选择取决于过滤条件的选择性（Selectivity），而不是简单的'Pre-filter更快'。

**核心概念：选择性**

```
选择性 = 过滤后文档数 / 总文档数

例如：
- 100万文档，过滤后剩5万 → 选择性 = 5%（高选择性）
- 100万文档，过滤后剩80万 → 选择性 = 80%（低选择性）
```

**决策规则：**

```
选择性 < 20%？
├─ 是 → Pre-filter（先过滤，再检索）
│   - 减少检索空间
│   - 提升检索速度
│   - 示例：category == 'rare_category'（只有5%的文档）
│
└─ 否 → 选择性 > 50%？
    ├─ 是 → Post-filter（先检索，再过滤）
    │   - 保证召回率
    │   - 避免扫描大量标量索引
    │   - 示例：price < 1000（80%的文档）
    │
    └─ 否 → 两者性能相近，优先Pre-filter
```

**为什么选择性决定策略？**

**场景1：高选择性（5%）**

```python
# Pre-filter（快）
dense_req = AnnSearchRequest(
    data=[query_vector],
    anns_field="dense_vector",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    limit=20,
    expr="category == 'rare_category'"  # 只扫描5%的数据
)
# 延迟: 15ms

# Post-filter（慢）
dense_req = AnnSearchRequest(
    data=[query_vector],
    anns_field="dense_vector",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    limit=100  # 需要召回更多候选
)
# 应用层过滤
filtered = [hit for hit in results if hit.entity.get("category") == "rare_category"][:20]
# 延迟: 45ms + 应用层过滤开销
```

**场景2：低选择性（80%）**

```python
# Pre-filter（慢）
dense_req = AnnSearchRequest(
    data=[query_vector],
    anns_field="dense_vector",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    limit=20,
    expr="price < 1000"  # 需要扫描80%的标量索引
)
# 延迟: 85ms（标量索引扫描开销大）

# Post-filter（快）
dense_req = AnnSearchRequest(
    data=[query_vector],
    anns_field="dense_vector",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    limit=100  # 召回100个候选
)
# 应用层过滤
filtered = [hit for hit in results if hit.entity.get("price") < 1000][:20]
# 延迟: 45ms + 5ms（应用层过滤） = 50ms
```

**性能对比（100万文档）：**

| 选择性 | Pre-filter延迟 | Post-filter延迟 | 推荐策略 |
|-------|---------------|----------------|---------|
| 5% | 15ms | 50ms | Pre-filter（3.3x快） |
| 20% | 35ms | 55ms | Pre-filter（1.6x快） |
| 50% | 65ms | 60ms | Post-filter（1.1x快） |
| 80% | 85ms | 50ms | Post-filter（1.7x快） |

**实际案例：电商商品搜索**

```python
# 案例1：稀有类别（选择性5%）→ Pre-filter
filter1 = "category == 'luxury_watches' AND brand == 'Rolex'"
# 推荐：Pre-filter

# 案例2：常见价格范围（选择性80%）→ Post-filter
filter2 = "price < 1000"
# 推荐：Post-filter

# 案例3：复合条件（选择性30%）→ Pre-filter
filter3 = "category == 'electronics' AND price > 500 AND rating >= 4.5"
# 推荐：Pre-filter（选择性适中，但复合条件可以利用索引）
```

**优化建议：**

1. **创建标量索引**：
   ```python
   index_params.add_index(field_name="category", index_type="INVERTED")
   index_params.add_index(field_name="price", index_type="STL_SORT")
   ```

2. **选择性高的条件放前面**：
   ```python
   # 好：tenant_id选择性高（1%）
   filter = "tenant_id == 'tenant_a' AND category == 'electronics'"
   
   # 差：category选择性低（50%）
   filter = "category == 'electronics' AND tenant_id == 'tenant_a'"
   ```

3. **使用IN代替多个OR**：
   ```python
   # 好：使用IN
   filter = "category IN ['electronics', 'computers']"
   
   # 差：使用OR
   filter = "category == 'electronics' OR category == 'computers'"
   ```

**总结：**
- Pre-filter适合高选择性（<20%）
- Post-filter适合低选择性（>50%）
- 选择性是决定因素，不是固定规则
- 需要根据实际数据分布和查询模式调整策略"

## 面试技巧总结

### 回答框架

**STAR法则（适用于技术问题）：**

1. **Situation（背景）**：为什么需要这个技术？
2. **Task（任务）**：这个技术要解决什么问题？
3. **Action（行动）**：技术原理和实现方式
4. **Result（结果）**：实际效果和性能数据

**示例：**
- Situation: 单一检索方式的局限性
- Task: 兼顾语义理解和关键词匹配
- Action: 向量检索 + BM25 + RRF重排序
- Result: Recall提升8%，Precision提升10%

## 延伸问题

### 可能的追问

1. **RRF算法的k参数如何选择？**
   - 默认k=60
   - k越大，排名靠后的文档权重越低
   - 一般不需要调整，除非有特殊需求

2. **如何评估混合检索的效果？**
   - Recall@K：召回率
   - Precision@K：精确率
   - NDCG@K：排序质量
   - 延迟和QPS

3. **混合检索和ReRank的区别？**
   - 混合检索：多路召回 + RRF融合
   - ReRank：单路召回 + 精排模型
   - 可以组合使用：混合检索 + ReRank

4. **如何优化混合检索的性能？**
   - 创建标量索引
   - 选择合适的过滤策略
   - 使用批量检索
   - 调整召回数量
