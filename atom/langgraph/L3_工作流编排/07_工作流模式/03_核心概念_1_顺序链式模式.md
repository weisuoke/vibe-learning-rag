# 03_核心概念_1 - 顺序链式模式（Prompt Chaining）

> 将多个 LLM 调用按线性顺序串联，前一步输出作为后一步输入，可在步骤间插入 Gate 检查点的最基础工作流模式

---

## 概念定义

**顺序链式模式是将多个 LLM 调用按线性顺序串联，前一步输出作为后一步输入的最基础工作流模式。**

这是六大工作流模式中最简单的一个，也是所有其他模式的基础。就像工厂流水线一样，每个工位完成自己的工序，半成品依次传递，最终产出成品。

---

## 模式原理

### 核心结构

```
START → A → B → C → END
```

每个节点处理并转换 State，结果自动传递给下一个节点。整个流程严格按顺序执行，没有分支、没有并行、没有循环。

### 带 Gate 检查的结构

```
START → A → [Gate] → B → C → END
              ↓
             END（不合格直接终止）
```

Gate 是步骤间的质量检查点。如果中间结果不合格，可以提前终止流程，避免浪费后续的 LLM 调用。

### 三个核心特征

| 特征 | 说明 |
|------|------|
| 线性执行流 | 严格按 A → B → C 顺序执行，不跳步、不回退 |
| 状态传递 | 前一步的输出自动合并到 State，后一步可以读取 |
| Gate 检查 | 可在任意步骤后插入条件判断，决定继续还是终止 |

## LangGraph 三种实现方式

### 方式 1：Graph API — `add_edge` + `add_conditional_edges`

**最基础的方式，显式定义每条边，适合需要 Gate 检查的场景。**

```python
"""
顺序链式模式 - Graph API 实现
场景：文章生成管道（生成 → 质量检查 → 审核 → 润色）

图拓扑：
    START → generate → [check_quality]
                          ↓ pass        ↓ fail
                        review → polish → END
                                         END
"""
from typing import Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class ArticleState(TypedDict):
    topic: str       # 输入主题
    draft: str       # 初稿
    review: str      # 审核意见
    final: str       # 最终稿
    quality: str     # 质量评级


def generate(state: ArticleState) -> dict:
    """第一步：根据主题生成初稿"""
    topic = state["topic"]
    # 实际项目中这里调用 LLM
    draft = f"这是一篇关于「{topic}」的文章初稿。包含引言、正文和结论三个部分。"
    print(f"[generate] 生成初稿: {draft[:30]}...")
    return {"draft": draft, "quality": "pending"}


def check_quality(state: ArticleState) -> Literal["review", "__end__"]:
    """Gate 检查：判断初稿质量是否值得继续处理"""
    draft = state["draft"]
    # 实际项目中可以用 LLM 做质量评估
    if len(draft) < 10:
        print("[gate] 初稿太短，终止流程")
        return END
    print("[gate] 初稿质量合格，继续审核")
    return "review"


def review(state: ArticleState) -> dict:
    """第二步：审核初稿，给出修改意见"""
    draft = state["draft"]
    review_comment = f"审核意见：文章结构清晰，建议补充具体案例和数据支撑。"
    print(f"[review] {review_comment}")
    return {"review": review_comment, "quality": "reviewed"}


def polish(state: ArticleState) -> dict:
    """第三步：根据审核意见润色终稿"""
    draft = state["draft"]
    review_comment = state["review"]
    final = f"{draft}\n\n[根据审核意见润色] {review_comment}\n补充了案例和数据后的最终版本。"
    print(f"[polish] 生成终稿")
    return {"final": final, "quality": "polished"}


# --- 构建图 ---
builder = StateGraph(ArticleState)

# 添加节点
builder.add_node("generate", generate)
builder.add_node("review", review)
builder.add_node("polish", polish)

# 添加边：定义执行顺序
builder.add_edge(START, "generate")
# Gate 检查：generate 之后用条件边决定是否继续
builder.add_conditional_edges("generate", check_quality)
builder.add_edge("review", "polish")
builder.add_edge("polish", END)

# 编译并执行
graph = builder.compile()
result = graph.invoke({
    "topic": "LangGraph 工作流模式",
    "draft": "",
    "review": "",
    "final": "",
    "quality": ""
})
print(f"\n最终结果: {result['quality']}")
# [generate] 生成初稿: 这是一篇关于「LangGraph 工作流模式」的文章初...
# [gate] 初稿质量合格，继续审核
# [review] 审核意见：文章结构清晰，建议补充具体案例和数据支撑。
# [polish] 生成终稿
# 最终结果: polished
```

#### 源码层面发生了什么

每个 `add_edge(A, B)` 在编译时被转化为 channel 写入：

```python
# [来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py L1297-L1305]
def attach_edge(self, starts, end):
    if isinstance(starts, str):
        if end != END:
            # A 执行完后，向 B 的触发 channel 写入
            self.nodes[starts].writers.append(
                ChannelWrite(
                    (ChannelWriteEntry(
                        f"branch:to:{end}", None
                    ),)
                )
            )
```

执行流程：generate 完成 → 写入 `branch:to:review` channel → review 被触发 → 写入 `branch:to:polish` channel → polish 被触发 → 写入 END channel → 流程结束。

### 方式 2：`add_sequence` 快捷方式

**最简洁的方式，一行代码定义整个顺序链。适合不需要 Gate 检查的纯线性流程。**

```python
"""
顺序链式模式 - add_sequence 快捷方式
场景：数据处理管道（提取 → 转换 → 加载）

注意：add_sequence 不支持中间插入条件边
"""
from typing_extensions import TypedDict
from langgraph.graph import StateGraph


class ETLState(TypedDict):
    raw_data: str        # 原始数据
    extracted: str       # 提取结果
    transformed: str     # 转换结果
    loaded: str          # 加载结果


def extract(state: ETLState) -> dict:
    """提取：从原始数据中提取关键信息"""
    raw = state["raw_data"]
    extracted = f"从 '{raw}' 中提取的结构化数据"
    print(f"[extract] {extracted}")
    return {"extracted": extracted}


def transform(state: ETLState) -> dict:
    """转换：对提取的数据进行格式转换"""
    data = state["extracted"]
    transformed = f"转换后: {data} → 标准化格式"
    print(f"[transform] {transformed}")
    return {"transformed": transformed}


def load(state: ETLState) -> dict:
    """加载：将转换后的数据写入目标"""
    data = state["transformed"]
    loaded = f"已加载: {data} → 向量数据库"
    print(f"[load] {loaded}")
    return {"loaded": loaded}


# --- 一行代码定义顺序链 ---
builder = StateGraph(ETLState)
builder.add_sequence([extract, transform, load])
# 等价于：
# builder.add_node("extract", extract)
# builder.add_node("transform", transform)
# builder.add_node("load", load)
# builder.add_edge(START, "extract")
# builder.add_edge("extract", "transform")
# builder.add_edge("transform", "load")
# builder.add_edge("load", END)

graph = builder.compile()
result = graph.invoke({
    "raw_data": "用户上传的 PDF 文档",
    "extracted": "",
    "transformed": "",
    "loaded": ""
})
print(f"\n最终: {result['loaded']}")
```

### 方式 3：Functional API — `@entrypoint` + `@task`

**最 Pythonic 的方式，用普通函数调用的写法定义工作流。适合熟悉 Python 异步编程的开发者。**

```python
"""
顺序链式模式 - Functional API 实现
场景：RAG 查询链（改写 → 检索 → 生成 → 验证）
"""
from langgraph.func import entrypoint, task


@task
def rewrite_query(query: str) -> str:
    """改写用户查询，提升检索效果"""
    rewritten = f"优化后的查询: {query} + 相关同义词扩展"
    print(f"[rewrite] {rewritten}")
    return rewritten


@task
def retrieve(query: str) -> list[str]:
    """从向量数据库检索相关文档"""
    docs = [
        f"文档1: 关于 {query} 的核心概念",
        f"文档2: {query} 的最佳实践",
        f"文档3: {query} 的常见问题",
    ]
    print(f"[retrieve] 检索到 {len(docs)} 篇文档")
    return docs


@task
def generate_answer(query: str, docs: list[str]) -> str:
    """基于检索结果生成回答"""
    context = "\n".join(docs)
    answer = f"基于 {len(docs)} 篇文档生成的回答：{query} 的核心要点是..."
    print(f"[generate] {answer[:40]}...")
    return answer


@task
def validate(answer: str) -> dict:
    """验证回答质量"""
    is_valid = len(answer) > 20
    result = {"answer": answer, "valid": is_valid}
    print(f"[validate] 验证结果: {'通过' if is_valid else '不通过'}")
    return result


@entrypoint()
def rag_chain(query: str) -> dict:
    """RAG 查询链：改写 → 检索 → 生成 → 验证"""
    # 每个 task 调用 .result() 获取结果，形成顺序链
    rewritten = rewrite_query(query).result()
    docs = retrieve(rewritten).result()
    answer = generate_answer(rewritten, docs).result()
    validated = validate(answer).result()
    return validated


# 执行
result = rag_chain.invoke("LangGraph 工作流模式有哪些")
print(f"\n最终结果: {result}")
```

#### Functional API 的特点

| 特性 | 说明 |
|------|------|
| `@task` | 标记一个函数为工作流中的一个步骤 |
| `@entrypoint()` | 标记入口函数，自动编译为可执行图 |
| `.result()` | 阻塞等待 task 完成并获取返回值 |
| 自动推断 | LangGraph 根据 `.result()` 调用顺序自动推断图结构 |

## 三种方式对比

| 维度 | Graph API | add_sequence | Functional API |
|------|-----------|--------------|----------------|
| 代码量 | 多（显式定义每条边） | 少（一行搞定） | 中（普通函数写法） |
| Gate 检查 | 支持（条件边） | 不支持 | 支持（if 语句） |
| 可视化 | 支持 | 支持 | 支持 |
| 灵活性 | 最高 | 最低 | 高 |
| 适用场景 | 需要 Gate 的复杂链 | 纯线性简单链 | 偏好函数式编程 |
| 学习成本 | 中 | 低 | 中 |

**选择建议**：
- 纯线性流程，没有条件判断 → `add_sequence`
- 需要中间 Gate 检查 → Graph API
- 喜欢 Pythonic 写法 → Functional API

## 适用场景

| 场景 | 流程 | Gate 位置 |
|------|------|-----------|
| 文档生成管道 | 生成初稿 → 审核 → 润色 | 初稿质量检查 |
| RAG 查询链 | 改写 → 检索 → ReRank → 生成 → 验证 | 检索结果/幻觉验证 |
| 数据处理 ETL | 提取 → 转换 → 清洗 → 向量化 → 入库 | 数据质量检查 |
| 多步推理 | 分解 → 求解 → 合成 → 一致性检查 | 子答案验证 |

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| 简单直观，易于理解和实现 | 延迟累加（总时间 = 各步骤时间之和） |
| 调试方便，每步状态可追踪 | 无法并行，即使步骤间无依赖 |
| 状态流向清晰，不会出现竞态 | 单点故障（任一步骤失败则整体失败） |
| Gate 检查可以提前终止，节省资源 | 不适合需要迭代优化的场景 |

### 延迟累加问题

```
步骤 A: 2秒 → 步骤 B: 3秒 → 步骤 C: 1秒
总延迟 = 2 + 3 + 1 = 6秒

如果 B 和 C 可以并行：
总延迟 = 2 + max(3, 1) = 5秒（需要用并行模式）
```

当步骤之间没有数据依赖时，应该考虑切换到并行模式来降低延迟。

## 与其他模式的关系

### 顺序链式是所有模式的基础

```
顺序链式 ──→ + 条件分支 ──→ 路由模式
         ──→ + 多分支并行 ──→ 并行模式
         ──→ + 循环反馈 ──→ 评估者-优化者模式
         ──→ + 动态任务分解 ──→ 编排者-工作者模式
         ──→ + Send 动态并行 ──→ Map-Reduce 模式
```

顺序链式可以嵌入到其他模式中。比如并行模式的每个分支内部可以是顺序链，评估者-优化者模式本质上是顺序链 + 循环反馈。

## 在 RAG 中的典型应用

RAG 系统天然适合顺序链式模式。典型的五步 RAG 链：

```
Query 改写 → 向量检索 → [Gate: 有文档?] → ReRank → 生成 → [Gate: 幻觉检查?] → 输出
```

Gate 检查是 RAG 链的关键优化点。在检索后加 Gate，如果没有相关文档就提前终止，避免浪费后续的 Rerank 和 Generation 调用：

```python
def check_retrieval(state) -> Literal["rerank", "__end__"]:
    """Gate：检查检索结果是否为空"""
    if not state["documents"]:
        return END      # 无文档，提前终止
    return "rerank"     # 有文档，继续处理

# 用条件边替代 retrieve → rerank 的直连边
builder.add_conditional_edges("retrieve", check_retrieval)
```

方式 3 的 Functional API 示例已经展示了完整的 RAG 顺序链实现（改写 → 检索 → 生成 → 验证），可直接参考。

## 最佳实践

### 1. 每个节点职责单一

```python
# 推荐：每个节点只做一件事
def rewrite(state): ...    # 只负责查询改写
def retrieve(state): ...   # 只负责检索
def generate(state): ...   # 只负责生成

# 不推荐：一个节点做太多事
def do_everything(state):
    rewritten = rewrite_query(state["query"])
    docs = search_docs(rewritten)
    answer = generate_answer(docs)
    return {"answer": answer}
```

### 2. 在关键步骤后加 Gate

```python
# 在耗时/耗费 Token 的步骤前加 Gate
builder.add_conditional_edges("retrieve", check_has_docs)
builder.add_conditional_edges("generate", check_answer_quality)
```

Gate 检查可以节省大量不必要的 LLM 调用开销。

### 3. State 设计要包含中间结果

每步的输出都应该有独立的 State 字段（如 `rewritten`、`documents`、`answer`），而不是只保留一个 `result` 字段。这样方便调试和追踪中间过程。

### 4. 纯线性链优先用 `add_sequence`

没有 Gate 检查时用 `add_sequence` 最简洁；有 Gate 检查时用 `add_edge` + `add_conditional_edges`。

## 常见误区

### 误区 1："顺序链只能用 add_edge"

实际上有三种实现方式（Graph API、add_sequence、Functional API），根据场景选择最合适的。

### 误区 2："Gate 检查只能终止流程"

Gate 也可以跳过某些步骤：

```python
def gate(state) -> Literal["rerank", "generate"]:
    if state["documents_count"] < 5:
        return "generate"   # 文档少时跳过 rerank
    return "rerank"         # 文档多时先 rerank
```

### 误区 3："顺序链不需要错误处理"

每个节点都可能失败（LLM 超时、检索为空等），应该在 State 中记录错误信息，并在 Gate 中处理。
