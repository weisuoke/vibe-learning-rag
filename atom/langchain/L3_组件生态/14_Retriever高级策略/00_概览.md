# Retriever高级策略 - 学习导航

> **知识点定位**：LangChain 组件生态 - 高级 Retriever 策略的原理、组合与生产级应用

---

## 一、知识点概述

### 核心价值

基础 Retriever（`as_retriever()`）只能做单路向量检索，面对真实业务场景远远不够。Retriever 高级策略通过混合检索、重排序、查询扩展、父文档检索和自查询五大武器，将基础检索器组合成生产级高精度检索管道。

本知识点是 `01_Retriever与VectorStore集成` 的进阶篇。如果说基础集成解决了"能检索"的问题，高级策略解决的是"检索得准、检索得全、检索得智能"的问题。

### 30 字核心

**Retriever高级策略通过混合检索、重排序和多路召回，将基础检索器组合成生产级高精度检索管道。**

### 适用场景

1. **企业知识库问答**：文档量大、查询多样，需要混合检索 + 重排序保证精度
2. **多模态文档检索**：长文档需要父文档检索，小块匹配、大块返回
3. **用户意图模糊**：查询扩展自动生成多角度查询，提高召回率
4. **结构化过滤**：自查询将自然语言转为结构化过滤条件，精准定位
5. **生产级 RAG 管道**：组合多种策略构建高可用检索服务

## 二、学习目标

### 知识目标

1. **理解 EnsembleRetriever 混合检索**：掌握 RRF 融合算法，理解 BM25 + 向量检索的互补原理
2. **掌握 ContextualCompression 重排序**：理解上下文压缩与重排序管道的装饰器模式
3. **熟悉 MultiQueryRetriever 查询扩展**：理解 LLM 驱动的多查询生成与去重合并机制
4. **理解 ParentDocumentRetriever 父文档检索**：掌握"小块搜索、大块返回"的两层/三层存储架构
5. **掌握 SelfQueryRetriever 自查询**：理解 LLM 结构化查询生成与访问者模式翻译

### 技能目标

1. **混合检索实战**：能够配置 BM25 + 向量检索的权重和融合参数
2. **重排序管道搭建**：能够使用 CrossEncoder 或 FlashRank 构建重排序管道
3. **查询扩展应用**：能够自定义查询生成提示词并控制扩展数量
4. **父文档检索配置**：能够设计两层/三层文档层次结构
5. **组合策略设计**：能够将多种策略组合成生产级 RAG 检索管道

### 五大策略的关系

| 策略 | 解决的问题 | 作用阶段 | 核心思路 |
|------|-----------|---------|---------|
| EnsembleRetriever | 单一检索召回不全 | 检索阶段 | 多路召回 + RRF 融合 |
| ContextualCompression | 召回结果噪声多 | 后处理阶段 | 重排序 + 压缩过滤 |
| MultiQueryRetriever | 用户查询角度单一 | 查询阶段 | LLM 生成多角度查询 |
| ParentDocumentRetriever | 小块匹配但上下文不足 | 存储+检索阶段 | 小块搜索、大块返回 |
| SelfQueryRetriever | 自然语言无法精确过滤 | 查询阶段 | LLM 解析为结构化查询 |

## 八、常见问题

### Q1: 五种策略应该怎么选？

**A**: 根据你的核心痛点选择：
- 召回不全 → EnsembleRetriever（多路召回）
- 结果不准 → ContextualCompression（重排序）
- 查询太窄 → MultiQueryRetriever（查询扩展）
- 上下文不够 → ParentDocumentRetriever（父文档）
- 需要精确过滤 → SelfQueryRetriever（自查询）

### Q2: 生产环境一般用哪些组合？

**A**: 最常见的生产组合是 **Ensemble + Rerank**：
```python
# 经典生产组合
ensemble = EnsembleRetriever(retrievers=[bm25, vector], weights=[0.4, 0.6])
pipeline = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=ensemble
)
```

### Q3: 这些策略和基础 as_retriever() 是什么关系？

**A**: 基础 `as_retriever()` 返回的 `VectorStoreRetriever` 是最底层的检索器。高级策略都是在它之上的包装和增强：
- EnsembleRetriever 组合多个基础检索器
- ContextualCompression 包装任意检索器做后处理
- MultiQuery 包装任意检索器做查询扩展
- ParentDocument 替换基础检索器的存储和返回逻辑
- SelfQuery 替换基础检索器的查询解析逻辑

### Q4: 策略越多越好吗？

**A**: 不是。每增加一个策略都会增加延迟和复杂度。生产环境建议：
- **起步**：单路向量检索（基线）
- **优化**：加 Rerank（性价比最高）
- **进阶**：换成 Ensemble + Rerank（召回+精度双提升）
- **按需**：根据具体场景加 MultiQuery 或 ParentDocument
