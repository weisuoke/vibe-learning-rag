# 状态调试技巧（State Debugging Techniques）- 知识点概览

> **当 AI Agent 的行为不符合预期时，你需要的不是猜测，而是一套系统化的调试手段——从流式观察每一步状态变化，到回溯历史定位问题根源，再到生产环境的全链路追踪。**

---

## 知识点简介

**状态调试技巧是 LangGraph 提供的一整套工具和方法，用于在开发和生产环境中观察、检查、追溯图执行过程中的状态变化，帮助开发者快速定位和解决问题。**

LangGraph 的图执行是多步骤、非确定性的——每个节点可能调用 LLM、执行检索、做出路由决策。当最终结果不对时，问题可能出在任何一步。没有调试工具，你只能看到输入和输出，中间发生了什么完全是黑盒。

状态调试技巧就是把这个黑盒打开的钥匙。它回答三个核心问题：

| 问题 | 调试手段 | 说明 |
|------|----------|------|
| **现在发生了什么？** | `stream_mode` 流式观察 | 实时看到每个节点的输入、输出、状态变化 |
| **刚才发生了什么？** | `get_state` / `get_state_history` | 执行后检查当前状态和历史快照 |
| **生产环境出了什么问题？** | LangSmith + 自定义日志 | 全链路追踪、元数据标签、结构化日志 |

### 调试的四个层次

```
┌─────────────────────────────────────────────────────────┐
│  Level 4: 生产级监控                                      │
│  LangSmith 追踪 · 元数据标签 · 告警                       │
├─────────────────────────────────────────────────────────┤
│  Level 3: 历史追溯                                        │
│  get_state_history · checkpoint_id 回溯 · 时间旅行        │
├─────────────────────────────────────────────────────────┤
│  Level 2: 状态检查                                        │
│  get_state · StateSnapshot 字段解读                       │
├─────────────────────────────────────────────────────────┤
│  Level 1: 流式观察                                        │
│  stream_mode · print · logging                           │
└─────────────────────────────────────────────────────────┘
```

从 Level 1 到 Level 4，调试能力逐步增强。初学者从 Level 1 开始就能解决大部分问题，生产环境则需要 Level 3-4 的能力。

## 为什么需要状态调试？

```python
# ❌ 没有调试工具：只能看到最终结果，中间过程是黑盒
result = graph.invoke({"query": "什么是向量数据库？"})
# {'answer': '我不知道'} — 问题出在检索？Rerank？生成？完全不知道。

# ✅ 有了调试工具：实时观察每一步
for chunk in graph.stream({"query": "什么是向量数据库？"},
                          config, stream_mode="updates"):
    print(chunk)
# ('retrieve', {'docs': ['向量数据库是...', 'Milvus 是...']})  ← 检索正常
# ('rerank', {'docs': ['Milvus 是...']})                       ← Rerank 只留了1条
# ('generate', {'answer': '我不知道'})                          ← 上下文太少
# → 问题定位：Rerank 过滤太激进
```

## 核心概念预览

本知识点包含 5 个核心概念：

| 编号 | 概念 | 一句话说明 | 重要程度 |
|------|------|-----------|----------|
| 1 | stream_mode 调试模式 | 6 种流式模式（values/updates/debug/checkpoints/tasks/custom）实时观察执行过程 | ★★★★★ |
| 2 | get_state 状态检查 | 运行后获取 StateSnapshot，检查 values/next/metadata 等 8 个字段 | ★★★★★ |
| 3 | get_state_history 历史追溯 | 逆时间顺序遍历所有历史快照，定位问题发生的精确步骤 | ★★★★☆ |
| 4 | 自定义调试工具 | 节点内 print/logging、StreamWriter 自定义流式输出 | ★★★★☆ |
| 5 | LangSmith 可观测性 | 追踪配置、元数据标签、生产级全链路监控 | ★★★☆☆ |

### 概念 1：stream_mode 调试模式

LangGraph 的 `stream()` 方法支持 6 种流式模式，每种模式输出不同粒度的信息：

| 模式 | 输出内容 | 适用场景 | 信息量 |
|------|----------|----------|--------|
| `"values"` | 每步后的完整状态 | 查看状态全貌 | ★★☆ |
| `"updates"` | 仅节点名 + 本次更新 | 追踪增量变化（最常用） | ★★☆ |
| `"debug"` | checkpoints + tasks 组合 | 全面调试 | ★★★ |
| `"checkpoints"` | 检查点事件（CheckpointPayload） | 持久化调试 | ★★★ |
| `"tasks"` | 任务开始/完成事件（TaskPayload） | 执行流程调试 | ★★★ |
| `"custom"` | 开发者自定义数据 | 特定业务场景 | 自定义 |

源码揭秘：`stream_mode="debug"` 并不是一个独立的模式，而是在 `_loop.py` 中被重映射为 `"checkpoints"` + `"tasks"` 的组合。

```python
# 最常用的调试模式
for chunk in graph.stream(input, config, stream_mode="updates"):
    node_name, updates = chunk  # 解构：节点名 + 更新内容
    print(f"[{node_name}] → {updates}")
```

### 概念 2：get_state 状态检查

`get_state()` 返回一个 `StateSnapshot` 对象，包含执行的完整上下文。这是 `PregelProtocol` 定义的核心调试方法。

StateSnapshot 的 8 个字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `values` | dict | 当前所有通道的值 |
| `next` | tuple[str, ...] | 下一步要执行的节点（空元组 = 已完成） |
| `config` | RunnableConfig | 获取此快照的配置（含 thread_id、checkpoint_id） |
| `metadata` | CheckpointMetadata | 元数据（source、step、writes、parents） |
| `created_at` | str | ISO 格式时间戳 |
| `parent_config` | RunnableConfig \| None | 父快照的配置（用于回溯） |
| `tasks` | tuple[PregelTask, ...] | 待执行/已执行的任务详情 |
| `interrupts` | tuple[Interrupt, ...] | 中断信息 |

```python
snapshot = graph.get_state(config)
print(snapshot.values)   # 当前状态值
print(snapshot.next)     # 下一步节点
print(snapshot.metadata) # {'source': 'loop', 'step': 2, ...}
```

### 概念 3：get_state_history 历史追溯

`get_state_history()` 返回一个迭代器，按逆时间顺序遍历所有历史快照。当你知道"结果不对"但不知道"哪一步出了问题"时，这个方法就是你的时间机器。

```python
# 遍历所有历史状态
for snapshot in graph.get_state_history(config):
    print(f"Step {snapshot.metadata['step']}: "
          f"next={snapshot.next}, "
          f"values={list(snapshot.values.keys())}")
```

配合 `checkpoint_id`，你可以从任意历史状态重新执行（时间旅行调试）。

### 概念 4：自定义调试工具

当内置的 stream_mode 不够用时，你可以在节点内部添加自定义调试逻辑：

- **print / logging**：最简单直接，适合开发阶段
- **StreamWriter**：通过 `get_stream_writer()` 向流式输出中注入自定义数据，配合 `stream_mode="custom"` 使用
- **Python logging 模块**：结构化日志，适合生产环境

```python
from langgraph.config import get_stream_writer

def retrieve_node(state: State):
    writer = get_stream_writer()
    # 向流式输出注入自定义调试信息
    writer({"debug_info": "检索开始", "query": state["query"]})
    docs = do_retrieval(state["query"])
    writer({"debug_info": "检索完成", "doc_count": len(docs)})
    return {"docs": docs}
```

### 概念 5：LangSmith 可观测性

LangSmith 是 LangChain 团队提供的可观测性平台，为 LangGraph 提供生产级的追踪和监控能力。它不是调试的必需品，但在生产环境中几乎不可替代。

核心能力：
- **追踪配置**：通过环境变量启用，零代码侵入
- **元数据标签**：为每次执行打标签（user_id、session_id、environment）
- **全链路可视化**：在 Web UI 中查看完整的执行链路、每步耗时、Token 消耗

```python
# 通过 config 添加追踪元数据
result = graph.invoke(input, config={
    "configurable": {"thread_id": "1"},
    "tags": ["production", "rag-v2"],
    "metadata": {"user_id": "user_123", "environment": "prod"}
})
```

### 场景 1 预览：基础状态打印与检查

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from typing_extensions import TypedDict
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("rag_debug")

class State(TypedDict):
    query: str
    docs: list[str]
    answer: str

def retrieve_node(state: State) -> dict:
    docs = [f"文档片段: 关于 '{state['query']}' 的内容"]
    logger.info(f"[retrieve] 检索到 {len(docs)} 条文档")
    return {"docs": docs}

def generate_node(state: State) -> dict:
    answer = f"基于 {len(state['docs'])} 条文档的回答"
    logger.info(f"[generate] 生成回答完成")
    return {"answer": answer}

builder = StateGraph(State)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

graph = builder.compile(checkpointer=MemorySaver())
config = {"configurable": {"thread_id": "debug-001"}}

# stream_mode="updates" 实时观察
for chunk in graph.stream({"query": "什么是 RAG？", "docs": [], "answer": ""}, config, stream_mode="updates"):
    print(f"节点更新: {chunk}")

# 执行后检查状态快照
snapshot = graph.get_state(config)
print(f"最终状态: {snapshot.values}")
print(f"是否完成: {snapshot.next == ()}")
```

对同一个图分别使用 `"values"`、`"updates"`、`"debug"` 三种模式，对比输出差异，理解每种模式的适用场景。重点展示 `"debug"` 模式输出的 `TaskPayload` 和 `CheckpointPayload` 结构。

### 场景 3 预览：时间旅行调试

利用 `get_state_history()` 找到问题步骤的 `checkpoint_id`，然后通过 `graph.invoke(None, checkpoint_config)` 从该历史状态重新执行，或通过 `update_state()` 修改历史状态后分叉执行。

结合 LangSmith 追踪配置、StreamWriter 自定义输出、结构化 logging，构建一个可用于生产环境的完整调试方案。包含错误捕获、性能监控、用户会话追踪等实战模式。

### 核心数据流

```
graph.stream(input, config, stream_mode="debug")
    ↓
_loop.py: "debug" 重映射为 "checkpoints" + "tasks"
    ├─ 节点执行前 → map_debug_tasks()      → TaskPayload
    ├─ 节点执行后 → map_debug_task_results() → TaskResultPayload
    └─ checkpoint 保存后 → map_debug_checkpoint() → CheckpointPayload

graph.get_state(config)
    ↓
checkpointer.get_tuple() → _prepare_state_snapshot() → StateSnapshot
```

## 常见问题

**Q：get_state 和 stream_mode 有什么区别？**
`stream_mode` 是实时的（执行过程中看每步变化），`get_state` 是事后的（执行完毕后检查快照）。两者互补：stream_mode 帮你"看过程"，get_state 帮你"查结果"。

**Q：必须配置 Checkpointer 才能调试吗？**
`stream_mode` 不需要 Checkpointer。但 `get_state()` 和 `get_state_history()` 必须配置 Checkpointer。建议开发阶段始终配置 `MemorySaver()`。

**Q：stream_mode 可以同时使用多种模式吗？**
可以。传入列表即可：`stream_mode=["updates", "custom"]`。
