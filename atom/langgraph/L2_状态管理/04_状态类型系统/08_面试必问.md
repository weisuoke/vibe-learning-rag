# 面试必问

## 问题 1：TypedDict 和 Pydantic 在 LangGraph 中的区别是什么？什么时候用哪个？

### 核心答案

**TypedDict**：
- 只提供静态类型提示，无运行时验证
- 零性能开销
- 官方推荐用于内部状态

**Pydantic**：
- 提供运行时验证和类型转换
- 有性能开销
- 推荐用于 API 边界验证

[来源: reference/search_状态类型系统_01.md, reference/context7_pydantic_01.md]

### 详细解释

```python
from typing_extensions import TypedDict
from pydantic import BaseModel, Field

# TypedDict：轻量快速
class InternalState(TypedDict):
    messages: list
    user_id: str

# Pydantic：严格验证
class APIInput(BaseModel):
    query: str = Field(..., min_length=1, max_length=1000)
    user_id: str = Field(..., pattern=r'^[a-zA-Z0-9_-]+$')

# 使用场景
# 1. API 输入 -> Pydantic 验证
api_input = APIInput(query="test", user_id="user_123")

# 2. 转换为内部状态 -> TypedDict
internal_state: InternalState = {
    "messages": [api_input.query],
    "user_id": api_input.user_id
}

# 3. 图内部处理 -> TypedDict（高性能）
# 4. API 输出 -> Pydantic 验证
```

### 追问点

**Q: 为什么不全部使用 Pydantic？**
A: 内部状态更新频繁，Pydantic 的验证开销会累积。边界验证次数少，安全性优先。

**Q: TypedDict 完全没有验证吗？**
A: 运行时没有，但可以用 mypy 等工具进行静态检查。

[来源: reference/search_状态类型系统_01.md]

## 问题 2：Annotated 和 Reducer 函数是如何工作的？

### 核心答案

`Annotated[type, reducer]` 将 Reducer 函数绑定到字段，Reducer 必须是二元操作符 `(current, new) -> merged`。

[来源: reference/source_状态类型系统_01.md]

### 详细解释

```python
from typing_extensions import Annotated, TypedDict

# 定义 Reducer 函数（必须是二元操作符）
def add_messages(left: list, right: list) -> list:
    """累积消息"""
    return left + right

# 绑定到字段
class State(TypedDict):
    messages: Annotated[list, add_messages]

# 工作原理
# 1. LangGraph 提取 __metadata__
# 2. 检查函数签名（必须有 2 个参数）
# 3. 调用 reducer(current_value, new_value)

# 示例
current_state = {"messages": ["A", "B"]}
update = {"messages": ["C"]}
# 结果: {"messages": ["A", "B", "C"]}
```

### 源码实现

```python
# 来自 state.py
def _is_field_binop(typ: type[Any]) -> BinaryOperatorAggregate | None:
    if hasattr(typ, "__metadata__"):
        meta = typ.__metadata__
        if len(meta) >= 1 and callable(meta[-1]):
            sig = signature(meta[-1])
            params = list(sig.parameters.values())
            # 必须恰好有 2 个位置参数
            if sum(p.kind in (p.POSITIONAL_ONLY, p.POSITIONAL_OR_KEYWORD) for p in params) == 2:
                return BinaryOperatorAggregate(typ, meta[-1])
    return None
```

[来源: reference/source_状态类型系统_01.md]

### 追问点

**Q: Reducer 函数可以有 3 个参数吗？**
A: 不可以，必须恰好 2 个参数，否则不会被识别为 Reducer。

**Q: 可以用 lambda 吗？**
A: 可以，`Annotated[list, lambda x, y: x + y]`

## 问题 3：LangGraph 的泛型类型系统有哪些参数？各自的作用是什么？

### 核心答案

4 个泛型参数：
- **StateT**：内部状态类型
- **ContextT**：不可变上下文类型
- **InputT**：图输入类型（默认 = StateT）
- **OutputT**：图输出类型（默认 = StateT）

[来源: reference/source_状态类型系统_01.md]

### 详细解释

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph

# 定义类型
class InternalState(TypedDict):
    messages: list
    context: dict
    metadata: dict

class Context(TypedDict):
    user_id: str
    config: dict

class Input(TypedDict):
    query: str

class Output(TypedDict):
    answer: str

# 创建图
graph = StateGraph(
    state_schema=InternalState,      # StateT
    context_schema=Context,           # ContextT
    input_schema=Input,               # InputT
    output_schema=Output              # OutputT
)

# 使用场景
# 1. API 接收 Input
# 2. 内部使用 InternalState 处理
# 3. Context 提供不可变配置
# 4. 返回 Output
```

### 追问点

**Q: 为什么需要分离输入输出类型？**
A: API 边界通常只需要部分字段，内部状态包含更多中间结果。

**Q: Context 和 State 的区别？**
A: Context 不可变，State 可变。Context 用于配置、用户信息等。

[来源: reference/source_状态类型系统_01.md]

## 问题 4：Pydantic 的 model_fields_set 在 LangGraph 中如何使用？

### 核心答案

`model_fields_set` 跟踪显式设置的字段，LangGraph 只更新这些字段，避免覆盖未修改的字段。

[来源: reference/source_状态类型系统_01.md, reference/context7_pydantic_01.md]

### 详细解释

```python
from pydantic import BaseModel

class State(BaseModel):
    x: int = 1
    y: str = "default"

# 只设置 x
state = State(x=2)
print(state.model_fields_set)  # {'x'}

# LangGraph 的处理逻辑
def get_update_as_tuples(input: BaseModel, keys: list[str]) -> list[tuple[str, Any]]:
    keep = input.model_fields_set  # {'x'}
    defaults = {k: v.default for k, v in type(input).model_fields.items()}
    
    return [
        (k, value)
        for k in keys
        if (value := getattr(input, k, MISSING)) is not MISSING
        and (
            value is not None
            or defaults.get(k, MISSING) is not None
            or (keep is not None and k in keep)  # 只更新显式设置的字段
        )
    ]

# 结果：只更新 x，不更新 y
```

[来源: reference/source_状态类型系统_01.md]

### 追问点

**Q: 如果想更新所有字段怎么办？**
A: 显式设置所有字段，或使用 TypedDict。

**Q: 默认值为 None 的字段如何处理？**
A: 特殊处理：如果默认值不是 None，则更新。

## 问题 5：类型修饰符 Required、NotRequired、ReadOnly 的作用和优先级？

### 核心答案

- **Required**：标记必需字段
- **NotRequired**：标记可选字段
- **ReadOnly**：标记只读字段（PEP 705）

**优先级**：`Required/NotRequired` > `__optional_keys__` (total=False) > 类型注解

[来源: reference/context7_typing_extensions_01.md, reference/source_状态类型系统_01.md]

### 详细解释

```python
from typing_extensions import TypedDict, Required, NotRequired, ReadOnly

# 示例 1：Required/NotRequired
class State1(TypedDict, total=False):
    x: Required[int]  # 必需（覆盖 total=False）
    y: str            # 可选（继承 total=False）

# 示例 2：ReadOnly
class State2(TypedDict):
    config: ReadOnly[dict]  # 只读（静态检查）
    data: dict              # 可变

# 优先级示例
class State3(TypedDict, total=False):
    a: Required[int]     # 必需（Required 优先级最高）
    b: NotRequired[str]  # 可选（显式 NotRequired）
    c: str               # 可选（继承 total=False）

# 字段默认值推断逻辑
def get_field_default(name: str, type_: Any, schema: type[Any]) -> Any:
    optional_keys = getattr(schema, "__optional_keys__", set())
    irq = _is_required_type(type_)
    
    if name in optional_keys:
        if irq:  # Required 覆盖
            return ...
        return None
    
    if irq is not None:
        if irq:
            return ...  # Required
        return None  # NotRequired
    
    # 其他情况...
```

[来源: reference/source_状态类型系统_01.md]

### 追问点

**Q: ReadOnly 会在运行时强制执行吗？**
A: 不会，只是静态检查。运行时可以修改。

**Q: Python 版本兼容性？**
A: Required/NotRequired 需要 3.11+，ReadOnly 需要 3.13+。使用 typing_extensions 可以支持旧版本。

## 问题 6：类型推断机制是如何工作的？推断失败时会怎样？

### 核心答案

从函数签名的第一个参数推断输入类型。推断失败时回退到 `state_schema`。

**优先级**：显式 `input_schema` > 推断类型 > `state_schema`

[来源: reference/source_状态类型系统_01.md]

### 详细解释

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph

class State(TypedDict):
    x: int

class NodeInput(TypedDict):
    query: str

# 情况 1：类型推断成功
def node1(state: NodeInput) -> State:
    return {"x": len(state["query"])}

graph = StateGraph(State)
graph.add_node("node1", node1)  # 推断 input_schema=NodeInput

# 情况 2：推断失败（无类型注解）
def node2(state):  # 无类型注解
    return {"x": 1}

graph.add_node("node2", node2)  # 回退到 state_schema=State

# 情况 3：显式指定（最高优先级）
graph.add_node("node3", node2, input_schema=NodeInput)  # 使用显式指定

# 推断逻辑（源码）
try:
    if isfunction(action) and (hints := get_type_hints(action)):
        first_parameter_name = next(iter(inspect.signature(action).parameters.keys()))
        if input_hint := hints.get(first_parameter_name):
            if isinstance(input_hint, type) and get_type_hints(input_hint):
                inferred_input_schema = input_hint
except (NameError, TypeError, StopIteration):
    pass  # 推断失败
```

[来源: reference/source_状态类型系统_01.md]

### 追问点

**Q: 为什么推断失败不报错？**
A: 提供回退机制，保证总是有类型。

**Q: 如何确保推断成功？**
A: 始终为节点函数添加类型注解。

## 问题 7：StateLike 协议是什么？为什么使用 Protocol 而非继承？

### 核心答案

`StateLike` 是类型协议，支持 TypedDict、Pydantic、dataclass。使用 Protocol 实现鸭子类型，更灵活。

[来源: reference/source_状态类型系统_01.md]

### 详细解释

```python
from typing import Protocol, ClassVar
from pydantic import BaseModel

# Protocol 定义
class TypedDictLike(Protocol):
    __required_keys__: ClassVar[frozenset[str]]
    __optional_keys__: ClassVar[frozenset[str]]

class DataclassLike(Protocol):
    __dataclass_fields__: ClassVar[dict]

StateLike = TypedDictLike | DataclassLike | BaseModel

# 为什么使用 Protocol？
# 1. 鸭子类型：只要满足协议即可
# 2. 不强制继承：用户可以使用任何类型
# 3. 向后兼容：不破坏现有代码

# 示例：所有这些都是有效的状态类型
from typing_extensions import TypedDict
from dataclasses import dataclass

class State1(TypedDict):  # 满足 TypedDictLike
    x: int

@dataclass
class State2:  # 满足 DataclassLike
    x: int

class State3(BaseModel):  # 满足 BaseModel
    x: int

# 都可以用于 StateGraph
graph1 = StateGraph(State1)
graph2 = StateGraph(State2)
graph3 = StateGraph(State3)
```

[来源: reference/source_状态类型系统_01.md]

### 追问点

**Q: Protocol 和 ABC 的区别？**
A: Protocol 是结构化类型（鸭子类型），ABC 是名义类型（需要继承）。

**Q: 如何自定义状态类型？**
A: 只要满足 Protocol 定义的属性即可。

## 问题 8：性能优化：如何在保证类型安全的同时优化性能？

### 核心答案

**分层验证策略**：
- API 边界：Pydantic 严格验证
- 内部状态：TypedDict 零开销
- 输出边界：Pydantic 验证

[来源: reference/search_状态类型系统_01.md]

### 详细解释

```python
from pydantic import BaseModel, Field
from typing_extensions import TypedDict

# 1. 边界验证
class APIInput(BaseModel):
    query: str = Field(..., min_length=1)
    user_id: str

# 2. 内部状态（高性能）
class InternalState(TypedDict):
    messages: list
    context: dict

# 3. 转换层
def convert_input(api_input: APIInput) -> InternalState:
    # 验证后转换为 TypedDict
    return {
        "messages": [api_input.query],
        "context": {"user_id": api_input.user_id}
    }

# 4. 性能优化技巧
# 技巧 1：避免深拷贝
def fast_convert(api_input: APIInput) -> InternalState:
    return {
        "messages": [api_input.query],
        "context": {"user_id": api_input.user_id}  # 直接引用
    }

# 技巧 2：使用 model_construct 跳过验证
from pydantic import BaseModel

class APIOutput(BaseModel):
    result: str

def fast_output(state: InternalState) -> APIOutput:
    return APIOutput.model_construct(result=state["messages"][0])

# 技巧 3：验证缓存
from functools import lru_cache

@lru_cache(maxsize=1000)
def validate_user_id(user_id: str) -> bool:
    return user_id.startswith("user_")
```

[来源: reference/context7_pydantic_01.md, reference/search_状态类型系统_01.md]

### 追问点

**Q: 性能差距有多大？**
A: TypedDict 零开销，Pydantic 有验证成本（约 10-100 倍）。

**Q: 什么时候必须用 Pydantic？**
A: API 边界、外部数据、需要递归验证的复杂结构。

## 问题 9：常见错误：TypedDict 无运行时验证导致的问题如何避免？

### 核心答案

**策略**：
1. 开发阶段：使用 mypy 静态检查
2. 生产阶段：边界使用 Pydantic 验证
3. 测试阶段：编写类型测试

[来源: reference/search_状态类型系统_01.md]

### 详细解释

```python
from typing_extensions import TypedDict
from pydantic import BaseModel, Field

# 问题：TypedDict 无运行时验证
class State(TypedDict):
    x: int
    y: str

# 这不会报错！
state: State = {"x": "not an int", "y": 123}

# 解决方案 1：静态检查
# $ mypy your_script.py

# 解决方案 2：边界验证
class InputState(BaseModel):
    x: int = Field(..., gt=0)
    y: str = Field(..., min_length=1)

def process_input(api_input: InputState) -> State:
    # Pydantic 验证后转换为 TypedDict
    return {
        "x": api_input.x,
        "y": api_input.y
    }

# 解决方案 3：运行时断言（开发环境）
def validate_state(state: State):
    assert isinstance(state["x"], int), "x must be int"
    assert isinstance(state["y"], str), "y must be str"

# 解决方案 4：类型测试
def test_state_types():
    state: State = {"x": 1, "y": "test"}
    assert isinstance(state["x"], int)
    assert isinstance(state["y"], str)
```

[来源: reference/search_状态类型系统_01.md]

### 追问点

**Q: 为什么不全部用 Pydantic？**
A: 性能考虑。内部状态更新频繁，验证开销会累积。

**Q: mypy 能发现所有类型错误吗？**
A: 只能发现静态可检查的错误，运行时动态类型错误无法发现。

## 问题 10：企业级应用：如何设计多租户系统的类型架构？

### 核心答案

**分层设计**：
- 租户配置：Pydantic 验证
- 租户状态：TypedDict 高性能
- 配额管理：独立模块
- 缓存策略：键值隔离

[来源: reference/search_状态类型系统_01.md]

### 详细解释

```python
from pydantic import BaseModel, Field, root_validator
from typing_extensions import TypedDict, Annotated
from enum import Enum

# 1. 租户配置（Pydantic 验证）
class TenantTier(str, Enum):
    FREE = "free"
    PRO = "pro"
    ENTERPRISE = "enterprise"

class TenantConfig(BaseModel):
    tenant_id: str = Field(..., pattern=r'^tenant_[a-zA-Z0-9_-]+$')
    tier: TenantTier
    max_queries_per_day: int = Field(..., gt=0)
    
    @root_validator
    def validate_tier_limits(cls, values):
        tier = values.get('tier')
        max_queries = values.get('max_queries_per_day')
        
        if tier == TenantTier.FREE and max_queries > 100:
            raise ValueError("免费版每天最多 100 次查询")
        
        return values

# 2. 租户状态（TypedDict 高性能）
def add_usage(left: dict, right: dict) -> dict:
    result = left.copy()
    for key, value in right.items():
        result[key] = result.get(key, 0) + value
    return result

class TenantState(TypedDict):
    tenant_id: str
    user_id: str
    query: str
    usage: Annotated[dict, add_usage]
    answer: str

# 3. 配额管理
class QuotaManager:
    def check_quota(self, tenant_id: str, resource: str, amount: int = 1) -> bool:
        # 检查配额
        pass
    
    def consume_quota(self, tenant_id: str, resource: str, amount: int = 1):
        # 消耗配额
        pass

# 4. 完整流程
def process_tenant_request(config: TenantConfig, query: str) -> dict:
    # 1. 验证配置（Pydantic 自动验证）
    # 2. 检查配额
    quota_manager = QuotaManager()
    if not quota_manager.check_quota(config.tenant_id, "query"):
        raise ValueError("配额已用完")
    
    # 3. 处理请求（TypedDict 高性能）
    state: TenantState = {
        "tenant_id": config.tenant_id,
        "user_id": "user_123",
        "query": query,
        "usage": {"query": 1},
        "answer": "处理结果"
    }
    
    # 4. 消耗配额
    quota_manager.consume_quota(config.tenant_id, "query")
    
    return state
```

[来源: reference/search_状态类型系统_01.md]

### 追问点

**Q: 如何实现租户隔离？**
A: 使用 namespace、分区、独立缓存键。

**Q: 配额如何持久化？**
A: 使用 Redis、数据库等外部存储。
