## 卡片 1/10：什么是人机循环？

### 核心观点

**人机循环 = AI 干活到一半，停下来问人类"这样行不行？"，得到回答后继续干。**

### 日常类比

想象你在餐厅点餐：

```
你：来一份牛排（发起任务）
厨师：开始准备...
厨师：牛排要几分熟？（暂停，等待人类输入）  ← 这就是 interrupt
你：七分熟（人类回答）                       ← 这就是 resume
厨师：好的，继续做...（恢复执行）
厨师：做好了！（任务完成）
```

没有人机循环的 AI 就像一个不问你几分熟就直接做的厨师——可能做出来的东西不是你想要的。

### 在 LangGraph 中

```python
from langgraph.types import interrupt, Command

def chef_node(state):
    # 厨师问你几分熟
    doneness = interrupt("牛排要几分熟？")
    # 得到回答后继续做
    return {"result": f"七分熟牛排，完成！"}

# 你回答：七分熟
graph.invoke(Command(resume="七分熟"), config)
```

## 卡片 2/10：interrupt() 函数的本质

### 核心观点

**interrupt() 做三件事：暂停执行、把信息传给人类、等待人类回答。**

### 拆解

```python
answer = interrupt(value)
```

这一行代码背后发生了三件事：

1. **暂停**：抛出 `GraphInterrupt` 异常，整个节点函数终止
2. **传值**：`value` 被保存到 checkpoint，客户端可以通过 `__interrupt__` 读取
3. **等待**：图停在这里，直到有人调用 `Command(resume=...)` 恢复

`value` 可以是任何 JSON 可序列化的东西——字符串、字典、列表都行：

```python
# 简单问题
answer = interrupt("你确定吗？")

# 丰富的上下文信息
answer = interrupt({
    "question": "请审批这笔转账",
    "amount": 5000,
    "recipient": "张三",
    "options": ["approve", "reject"],
})
```

### 关键记忆点

`interrupt()` 的本质是一个**可恢复的异常**。第一次调用时抛异常暂停，恢复后返回人类给的值。

## 卡片 3/10：Command(resume=value) 的工作方式

### 核心观点

**Command(resume=value) 是人类回答 AI 问题的方式——把答案传回去，让图继续跑。**

### 使用方式

```python
# AI 问了一个问题（interrupt）
# 人类通过 Command 回答

# 最基本的恢复
graph.invoke(Command(resume="approve"), config)

# Command 还能同时做其他事
graph.invoke(Command(
    resume="approve",              # 回答问题
    goto="special_node",           # 顺便指定下一步去哪
    update={"priority": "high"},   # 顺便更新状态
), config)
```

### 类比

如果 `interrupt()` 是发出去的微信消息"你觉得怎么样？"，那 `Command(resume=)` 就是对方回复的那条消息。`config` 里的 `thread_id` 就是聊天窗口——你得在同一个窗口里回复，对方才能收到。

### 关键记忆点

`Command` 不只是传值，它是一个"复合指令"：resume（回答）+ goto（路由）+ update（改状态），三合一。

## 卡片 4/10：为什么必须有 Checkpointer

### 核心观点

**没有 Checkpointer，interrupt 就像断电后没保存的 Word 文档——全丢了。**

### 为什么需要

`interrupt()` 暂停后，图的状态需要保存在某个地方：

```
interrupt() 暂停
    ↓
需要保存：当前状态值、执行到哪个节点、中断信息
    ↓
保存到哪？→ Checkpointer！
    ↓
恢复时：从 Checkpointer 加载状态，继续执行
```

没有 Checkpointer，这些信息无处存放，恢复时不知道从哪继续。

### 代码对比

```python
# ❌ 没有 checkpointer → 报错
graph = builder.compile()  # 缺少 checkpointer
graph.invoke(input, config)  # ValueError!

# ✅ 有 checkpointer → 正常工作
graph = builder.compile(checkpointer=MemorySaver())
graph.invoke(input, config)  # 正常暂停和恢复
```

### 选型速查

| 场景 | 推荐 |
|------|------|
| 开发/测试 | `MemorySaver`（内存，重启丢失） |
| 生产环境 | `PostgresSaver`（持久化，重启不丢失） |

## 卡片 5/10：恢复时的重新执行机制（最关键的反直觉点）

### 核心观点

**恢复时，节点从头重新执行整个函数，不是从 interrupt() 那一行继续。**

### 这是最容易踩的坑

很多人以为 `interrupt()` 像 Python 的 `yield`——暂停在那一行，恢复后从那一行继续。实际上：

```python
def my_node(state):
    print("A: 我会执行")          # ← 恢复时也会执行！
    print("B: 我也会执行")        # ← 恢复时也会执行！
    answer = interrupt("问题")    # ← 恢复时直接返回 resume 值
    print("C: 只有恢复后才到这")  # ← 只在恢复时执行
    return {"answer": answer}
```

执行顺序：
```
第一次调用：A → B → interrupt（抛异常，函数终止）
恢复执行：  A → B → interrupt（返回 resume 值）→ C → return
```

### 为什么这很重要

如果你在 `interrupt()` 之前做了有副作用的操作（发邮件、扣款、写数据库），恢复时会再做一次！

```python
# ❌ 危险：恢复时会再发一次邮件
def bad_node(state):
    send_email("开始处理")   # 恢复时重复执行！
    answer = interrupt("确认？")
    return {"result": answer}

# ✅ 安全：副作用放在 interrupt 之后
def good_node(state):
    answer = interrupt("确认？")
    send_email("开始处理")   # 只在恢复后执行一次
    return {"result": answer}
```

### 关键记忆点

`interrupt()` 不是 `yield`。恢复 = 重新执行整个函数，只是 `interrupt()` 会直接返回缓存值而不再暂停。

## 卡片 6/10：静态断点 interrupt_before/after

### 核心观点

**interrupt_before/after 是"固定路障"——不管什么情况，到了这个节点就停。**

### 使用方式

```python
# 在 compile 时配置，不需要改节点代码
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["risky_node"],   # risky_node 执行前暂停
    interrupt_after=["check_node"],    # check_node 执行后暂停
)

# 也可以在运行时临时指定
for chunk in graph.stream(input, config, interrupt_before=["node_a"]):
    print(chunk)
```

### 和 interrupt() 的区别

| | interrupt_before/after | interrupt() |
|---|---|---|
| 比喻 | 固定路障 | 智能红绿灯 |
| 条件判断 | 不支持 | 支持 |
| 数据传递 | 不能 | 能 |
| 改代码 | 不需要 | 需要 |

### 最佳使用场景

```python
# 场景：调试时想在每个节点前暂停查看状态
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["*"],  # 所有节点前暂停（调试神器）
)
```

### 关键记忆点

调试用 `interrupt_before/after`（不改代码），业务逻辑用 `interrupt()`（灵活可控）。

## 卡片 7/10：单节点多 interrupt 的顺序匹配

### 核心观点

**一个节点里可以有多个 interrupt()，恢复时按顺序一一匹配 resume 值。**

### 工作原理

```python
def multi_step_node(state):
    name = interrupt("请输入姓名")      # 第 1 个 interrupt
    email = interrupt("请输入邮箱")     # 第 2 个 interrupt
    confirm = interrupt(f"确认：{name}, {email}？")  # 第 3 个 interrupt
    return {"name": name, "email": email}
```

恢复过程（需要三次 resume）：

```
第 1 次 invoke → 遇到第 1 个 interrupt，暂停
Command(resume="张三") → 节点重执行，第 1 个 interrupt 返回 "张三"，遇到第 2 个 interrupt，暂停
Command(resume="zhang@example.com") → 节点重执行，前两个 interrupt 返回缓存值，遇到第 3 个，暂停
Command(resume="yes") → 节点重执行，三个 interrupt 都返回缓存值，函数正常完成
```

### 关键细节

每次恢复，节点都从头执行。已经匹配过的 `interrupt()` 直接返回之前的 resume 值，不会再暂停。只有下一个未匹配的 `interrupt()` 才会触发新的暂停。

### 关键记忆点

多个 interrupt 就像排队——第 1 个 resume 给第 1 个 interrupt，第 2 个给第 2 个，严格按顺序。

## 卡片 8/10：结构化交互——HumanInterrupt 协议

### 核心观点

**HumanInterrupt 是一套标准化的"人机对话协议"，定义了人类可以做哪些操作。**

### 为什么需要协议

直接用 `interrupt("你确定吗？")` 传字符串，前端不知道该展示什么 UI。HumanInterrupt 协议解决了这个问题：

```python
from langgraph.prebuilt.interrupt import HumanInterrupt, HumanResponse

# AI 端：告诉前端"用户可以做什么"
interrupt_value = HumanInterrupt(
    action_request={"action": "delete_file", "args": {"path": "/data/old.csv"}},
    config={
        "allow_accept": True,    # 可以批准
        "allow_reject": True,    # 可以拒绝（通过 ignore）
        "allow_respond": True,   # 可以回复文字
        "allow_edit": True,      # 可以编辑参数
    },
    description="AI 想要删除文件 /data/old.csv",
)

decision = interrupt(interrupt_value)
```

### 前端可以返回四种响应

```python
# 批准：直接执行
HumanResponse(type="accept", args=None)

# 跳过：忽略这个操作
HumanResponse(type="ignore", args=None)

# 回复：给 AI 文字反馈
HumanResponse(type="response", args="请先备份再删除")

# 编辑：修改操作参数
HumanResponse(type="edit", args={"action": "delete_file", "args": {"path": "/data/backup/old.csv"}})
```

### 关键记忆点

HumanInterrupt 让人机交互从"自由文本"升级为"结构化协议"，前端可以根据 config 自动渲染对应的 UI 按钮。

## 卡片 9/10：生产实践——从 MemorySaver 到 PostgresSaver

### 核心观点

**MemorySaver 是开发用的"便签纸"，PostgresSaver 是生产用的"保险柜"。**

### 开发 vs 生产

```python
# === 开发环境 ===
from langgraph.checkpoint.memory import MemorySaver

graph = builder.compile(checkpointer=MemorySaver())
# 优点：零配置，开箱即用
# 缺点：进程重启后所有中断状态丢失

# === 生产环境 ===
from langgraph.checkpoint.postgres import PostgresSaver

graph = builder.compile(
    checkpointer=PostgresSaver(
        conn_string="postgresql://user:pass@host:5432/langgraph"
    )
)
# 优点：持久化，重启不丢失，支持多实例共享
# 缺点：需要维护数据库
```

### 迁移很简单

从 MemorySaver 切换到 PostgresSaver，只需要改一行代码——`compile()` 的 `checkpointer` 参数。节点代码、interrupt 调用、Command 恢复，全部不用改。

### 生产环境还要考虑

```python
# 1. 过期 checkpoint 清理（避免数据库无限膨胀）
# 2. 连接池配置（高并发场景）
# 3. 读写分离（大规模部署）
# 4. 监控告警（checkpoint 写入失败时及时发现）
```

### 关键记忆点

Checkpointer 是可插拔的。开发用 MemorySaver 快速迭代，上线前换成 PostgresSaver，一行代码的事。

## 卡片 10/10：interrupt() vs 静态断点的决策树

### 核心观点

**90% 的场景用 interrupt()，只有调试和快速原型才考虑 interrupt_before/after。**

### 决策树

```
你需要人机中断吗？
│
├─ 只是调试/查看中间状态
│  └─ → interrupt_before/after（不用改代码）
│
├─ 需要根据条件决定是否中断？
│  └─ → interrupt()（只有它支持 if/else）
│
├─ 需要传递信息给前端？
│  └─ → interrupt()（可传任意 JSON）
│
├─ 需要一个节点内多次交互？
│  └─ → interrupt()（支持多中断点）
│
├─ 需要根据人类回答路由到不同节点？
│  └─ → interrupt() + Command(goto=)
│
└─ 不确定？
   └─ → interrupt()（官方推荐，功能更全）
```

### 速查表

| 我想要... | 用什么 |
|-----------|--------|
| 调试时暂停查看状态 | `interrupt_before=["*"]` |
| 大额支付才需要审批 | `interrupt()`（条件判断） |
| 收集用户多步输入 | `interrupt()`（多中断点） |
| 展示审批详情给前端 | `interrupt({"details": ...})`（数据传递） |
| 人类决定走哪条路 | `interrupt()` + `Command(goto=)` |
| 快速原型，不想改节点 | `interrupt_before=["node"]` |

### 关键记忆点

`interrupt()` 是瑞士军刀，`interrupt_before/after` 是螺丝刀。螺丝刀能干的活瑞士军刀都能干，但拧螺丝的时候螺丝刀更顺手。
