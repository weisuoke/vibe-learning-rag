# Reducer函数与状态更新 - 知识点概览

> LangGraph 状态管理的核心机制：如何优雅地合并多个节点的状态更新

---

## 知识点简介

在 LangGraph 中，多个节点可能同时或先后更新同一个状态字段。**Reducer函数**定义了如何合并这些更新值，是实现累积型状态管理（如消息历史、文档列表、计数器）的核心机制。

**核心问题**：当节点 A 返回 `{"messages": ["Hello"]}`，节点 B 返回 `{"messages": ["World"]}`，最终状态的 `messages` 应该是什么？

- **没有 Reducer**：`["World"]`（直接替换）
- **有 Reducer（如 `operator.add`）**：`["Hello", "World"]`（合并）

本知识点将深入讲解 Reducer 函数的定义、使用、内置实现、自定义策略，以及与部分状态更新的交互。

## 常见问题

### Q1: Reducer 和 Redux 的 Reducer 有什么区别？
**A**: 概念相似但实现不同。Redux Reducer 是 `(State, Action) -> State`，而 LangGraph Reducer 是 `(Value, Value) -> Value`。LangGraph Reducer 只关注单个字段的合并，而 Redux Reducer 处理整个状态树。

### Q2: 什么时候需要使用 Reducer？
**A**: 当状态字段需要累积更新（而非替换）时使用 Reducer。典型场景：消息历史、文档列表、计数器、配置合并。

### Q3: 可以在运行时动态修改 Reducer 吗？
**A**: 不可以。Reducer 在定义 StateGraph 时绑定到状态字段，运行时不可修改。如需动态行为，可以在 Reducer 函数内部实现条件逻辑。

### Q4: Reducer 会影响性能吗？
**A**: 影响很小。Reducer 只在状态更新时调用，且通常是简单的合并操作。对于大型列表，建议使用 `operator.add` 而非自定义循环。

### Q5: 如何调试 Reducer 的执行？
**A**: 在 Reducer 函数内部添加 `print()` 语句，或使用 LangSmith 追踪状态变化。
