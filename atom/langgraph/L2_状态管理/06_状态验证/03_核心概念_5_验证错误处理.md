# 核心概念 5：验证错误处理

## 概述

验证错误处理是捕获、分析和响应状态验证失败的机制。在 LangGraph 中，当你用 Pydantic BaseModel 定义状态 schema 后，任何不符合约束的输入都会抛出 `ValidationError`。如果不处理这些错误，图会直接崩溃；如果处理得当，系统可以给出清晰的错误提示、自动降级、甚至自我修复。

验证错误处理的核心目标：**让系统在遇到坏数据时优雅地失败，而不是静默地产生错误结果。**

[来源: reference/context7_pydantic_01.md | Pydantic 官方文档]

## 第一性原理

### 为什么需要专门处理验证错误？

一个 RAG 系统的输入来源多种多样——用户输入、LLM 输出、外部 API 返回值、数据库查询结果。这些数据都可能不符合预期格式。

不处理验证错误会导致两种糟糕的结果：

1. **程序崩溃**：未捕获的 `ValidationError` 直接终止整个图的执行
2. **错误传播**：如果绕过验证，坏数据会在节点间传递，最终产生不可预测的输出

验证错误处理的本质是在**数据边界**设置守卫：

```
用户输入 → [验证守卫] → 图执行 → [验证守卫] → 节点输出
              ↓                          ↓
         捕获错误                    捕获错误
         友好提示                    优雅降级
```

## Pydantic ValidationError 结构

### 触发验证错误

当输入数据违反 Pydantic 约束时，会抛出 `ValidationError`。一次验证可以同时发现多个错误：

```python
from pydantic import BaseModel, ValidationError, Field

class AgentState(BaseModel):
    query: str
    max_tokens: int = Field(ge=1, le=4096)
    temperature: float = Field(ge=0.0, le=2.0)

try:
    state = AgentState(query="", max_tokens=-1, temperature=5.0)
except ValidationError as e:
    print(e)                    # 人类可读格式
    print(e.errors())           # 结构化错误列表
    print(e.error_count())      # 错误数量
    print(e.json())             # JSON 格式
```

**输出示例**（人类可读格式）：

```
2 validation errors for AgentState
max_tokens
  Input should be greater than or equal to 1 [type=greater_than_equal, ...]
temperature
  Input should be less than or equal to 2.0 [type=less_than_equal, ...]
```

注意 `query=""` 没有报错——因为空字符串仍然是合法的 `str` 类型。如果你想禁止空字符串，需要加 `Field(min_length=1)` 或自定义 `field_validator`。

### ValidationError 的四个访问方式

| 方法 | 返回类型 | 用途 |
|------|---------|------|
| `str(e)` 或 `print(e)` | 字符串 | 给开发者看的可读格式 |
| `e.errors()` | `list[dict]` | 程序化处理错误 |
| `e.error_count()` | `int` | 快速判断错误数量 |
| `e.json()` | `str` | 序列化传输（如 API 响应） |

## 错误结构详解

### 单个错误的字段

`e.errors()` 返回一个列表，每个元素是一个字典，包含以下字段：

```python
# 每个错误包含：
{
    'type': 'greater_than_equal',  # 错误类型（机器可读的标识符）
    'loc': ('max_tokens',),        # 错误位置（字段路径，支持嵌套）
    'msg': 'Input should be >= 1', # 错误消息（人类可读）
    'input': -1,                   # 原始输入值
    'ctx': {'ge': 1},              # 上下文信息（约束参数）
    'url': 'https://errors.pydantic.dev/2.x/v/greater_than_equal'  # 文档链接
}
```

### 各字段的作用

| 字段 | 说明 | 使用场景 |
|------|------|---------|
| `type` | 错误类型标识符 | 程序化判断错误种类，做分支处理 |
| `loc` | 字段路径元组 | 定位哪个字段出错，嵌套模型会有多级路径 |
| `msg` | 人类可读消息 | 直接展示给用户或写入日志 |
| `input` | 原始输入值 | 调试时查看用户到底传了什么 |
| `ctx` | 约束上下文 | 了解具体的约束条件是什么 |
| `url` | Pydantic 文档链接 | 快速查阅该错误类型的详细说明 |

### 嵌套模型的错误定位

当状态包含嵌套模型时，`loc` 会包含完整路径：

```python
from pydantic import BaseModel, Field, ValidationError

class SearchResult(BaseModel):
    title: str = Field(min_length=1)
    score: float = Field(ge=0.0, le=1.0)

class RAGState(BaseModel):
    query: str
    results: list[SearchResult] = []

try:
    RAGState(query="测试", results=[{"title": "", "score": 1.5}])
except ValidationError as e:
    for error in e.errors():
        print(f"位置: {error['loc']}, 消息: {error['msg']}")
        # 位置: ('results', 0, 'title'), 消息: String should have at least 1 character
        # 位置: ('results', 0, 'score'), 消息: Input should be less than or equal to 1.0
```

`('results', 0, 'title')` 表示 `results` 列表第 0 个元素的 `title` 字段。

## 在 LangGraph 中捕获验证错误

### 基础捕获模式

LangGraph 的 `graph.invoke()` 在接收输入时会触发 Pydantic 验证。验证失败时 `ValidationError` 会直接抛出，你需要在调用处捕获：

```python
from pydantic import BaseModel, Field, ValidationError
from langgraph.graph import StateGraph, START, END

class AgentState(BaseModel):
    query: str = Field(min_length=1, max_length=500)
    max_tokens: int = Field(default=1000, ge=1, le=4096)
    temperature: float = Field(default=0.7, ge=0.0, le=2.0)

def process_node(state: AgentState):
    return {"query": state.query}

builder = StateGraph(AgentState)
builder.add_node("process", process_node)
builder.add_edge(START, "process")
builder.add_edge("process", END)
graph = builder.compile()

def safe_invoke(graph, input_data):
    """安全调用图，捕获验证错误"""
    try:
        return graph.invoke(input_data)
    except ValidationError as e:
        print(f"输入验证失败，共 {e.error_count()} 个错误：")
        for error in e.errors():
            field = '.'.join(str(loc) for loc in error['loc'])
            print(f"  - {field}: {error['msg']}")
        return None

result = safe_invoke(graph, {"query": "什么是 RAG？"})       # ✅ 正常
result = safe_invoke(graph, {"query": "", "temperature": 5.0})  # ❌ 捕获错误，不崩溃
# 输入验证失败，共 2 个错误：
#   - query: String should have at least 1 character
#   - temperature: Input should be less than or equal to 2.0
```

## LangGraph 内置错误类型

除了 Pydantic 的 `ValidationError`，LangGraph 自身也定义了一组错误类型。

### 错误类型一览

```python
from langgraph.errors import InvalidUpdateError, GraphRecursionError

# InvalidUpdateError - 节点返回的数据无法合并到状态中时抛出
# GraphRecursionError - 图执行步数超过 recursion_limit（默认 25 步）时抛出
```

### ErrorCode 枚举

```python
# ErrorCode.INVALID_GRAPH_NODE_RETURN_VALUE - 节点返回值格式不正确
# ErrorCode.INVALID_CONCURRENT_GRAPH_UPDATE - 并发执行时多个节点同时更新同一字段
```

[来源: sourcecode/langgraph/libs/langgraph/langgraph/errors.py]

### 完整的错误捕获模板

在生产环境中，你需要同时处理 Pydantic 验证错误和 LangGraph 运行时错误：

```python
from pydantic import ValidationError
from langgraph.errors import InvalidUpdateError, GraphRecursionError

def robust_invoke(graph, input_data: dict) -> dict:
    """生产级的图调用函数，处理所有已知错误类型"""
    try:
        return {"success": True, "data": graph.invoke(input_data)}
    except ValidationError as e:
        return {"success": False, "error_type": "validation",
                "message": f"输入验证失败：{e.error_count()} 个错误", "details": e.errors()}
    except InvalidUpdateError as e:
        return {"success": False, "error_type": "invalid_update", "message": str(e)}
    except GraphRecursionError:
        return {"success": False, "error_type": "recursion_limit",
                "message": "图执行步数超过限制，可能存在无限循环"}
    except Exception as e:
        return {"success": False, "error_type": "unexpected", "message": str(e)}
```

## 验证错误 vs 运行时错误

这是一个关键区分。两类错误的性质不同，处理策略也不同。

### 对比表

| 维度 | 验证错误 (ValidationError) | 运行时错误 (RuntimeError 等) |
|------|--------------------------|---------------------------|
| 发生时机 | 数据进入图之前 / 状态更新时 | 节点执行过程中 |
| 可预期性 | **可预期**——你知道哪些约束可能被违反 | **不可预期**——网络超时、API 限流、内存不足 |
| 错误信息 | 结构化的，包含字段路径和约束详情 | 通常是非结构化的异常消息 |
| 根本原因 | 输入数据不符合 schema | 外部依赖故障或逻辑 bug |
| 处理策略 | 拒绝输入 + 返回具体错误提示 | 重试 / 降级 / 熔断 |
| 用户感知 | "你的输入有问题，请修改" | "系统暂时不可用，请稍后重试" |

### 代码示例：区分处理

```python
from pydantic import BaseModel, Field, ValidationError

class AgentState(BaseModel):
    query: str = Field(min_length=1)
    model: str = "gpt-4"

def handle_request(input_data: dict) -> str:
    # 第一层：验证错误处理
    try:
        state = AgentState(**input_data)
    except ValidationError as e:
        errors = "; ".join(
            f"{'.'.join(str(l) for l in err['loc'])}: {err['msg']}"
            for err in e.errors()
        )
        return f"输入有误：{errors}"

    # 第二层：运行时错误处理
    try:
        if state.model != "gpt-4":
            raise ConnectionError("API 连接超时")
        return f"回答：{state.query}"
    except ConnectionError:
        return "服务暂时不可用，请稍后重试"

print(handle_request({"query": ""}))              # 输入有误：query: String should have at least 1 character
print(handle_request({"query": "什么是 RAG？"}))   # 回答：什么是 RAG？
print(handle_request({"query": "测试", "model": "x"}))  # 服务暂时不可用，请稍后重试
```

## 优雅降级策略

当验证失败时，直接拒绝请求并不总是最佳选择。使用默认值替代无效输入、记录日志并继续执行，能提供更好的用户体验。

### 策略 1：使用默认值替代无效输入

```python
from pydantic import BaseModel, Field, ValidationError

class AgentState(BaseModel):
    query: str = Field(min_length=1)
    max_tokens: int = Field(default=1000, ge=1, le=4096)
    temperature: float = Field(default=0.7, ge=0.0, le=2.0)

def create_state_with_fallback(input_data: dict) -> AgentState:
    """尝试创建状态，对无效字段使用默认值"""
    try:
        return AgentState(**input_data)
    except ValidationError as e:
        bad_fields = {err['loc'][0] for err in e.errors() if err['loc']}
        cleaned = {k: v for k, v in input_data.items() if k not in bad_fields}
        try:
            state = AgentState(**cleaned)
            print(f"已自动修正字段：{bad_fields}，使用默认值")
            return state
        except ValidationError:
            raise  # 必填字段无效时无法恢复

state = create_state_with_fallback({
    "query": "什么是 RAG？",
    "max_tokens": -1,        # 无效 → 默认值 1000
    "temperature": 5.0       # 无效 → 默认值 0.7
})
print(f"max_tokens={state.max_tokens}, temperature={state.temperature}")
# 已自动修正字段：{'max_tokens', 'temperature'}，使用默认值
# max_tokens=1000, temperature=0.7
```

### 策略 2：记录错误日志并继续执行

```python
import logging

logger = logging.getLogger("langgraph.validation")

def safe_invoke_with_logging(graph, input_data: dict):
    """带日志记录的安全调用"""
    try:
        return graph.invoke(input_data)
    except ValidationError as e:
        logger.warning(
            "状态验证失败",
            extra={
                "error_count": e.error_count(),
                "errors": [
                    {
                        "field": '.'.join(str(loc) for loc in err['loc']),
                        "type": err['type'],
                        "message": err['msg']
                    }
                    for err in e.errors()
                ]
            }
        )
        return None
```

### 策略 3：返回友好的错误消息给用户

```python
ERROR_MESSAGES = {
    "string_too_short": "输入内容太短，请提供更详细的描述",
    "string_too_long": "输入内容超过长度限制，请精简后重试",
    "greater_than_equal": "数值太小，请检查参数范围",
    "less_than_equal": "数值太大，请检查参数范围",
    "missing": "缺少必填参数",
}

def format_user_friendly_errors(e: ValidationError) -> list[str]:
    """将 ValidationError 转换为用户友好的中文消息"""
    return [
        f"「{'.'.join(str(loc) for loc in err['loc'])}」"
        f"{ERROR_MESSAGES.get(err['type'], err['msg'])}"
        for err in e.errors()
    ]

try:
    AgentState(query="", max_tokens=-1)
except ValidationError as e:
    for msg in format_user_friendly_errors(e):
        print(msg)
    # 「query」输入内容太短，请提供更详细的描述
    # 「max_tokens」数值太小，请检查参数范围
```

## 生产环境最佳实践

### 1. 在图的入口处统一处理验证错误

不要在每个节点内部分别处理验证错误。把验证逻辑集中在图的入口处，保持节点函数的简洁：

```python
from pydantic import BaseModel, Field, ValidationError
from langgraph.graph import StateGraph, START, END

class RAGState(BaseModel):
    query: str = Field(min_length=1, max_length=500)
    top_k: int = Field(default=5, ge=1, le=20)
    response: str = ""

def retrieve_node(state: RAGState):
    return {"response": f"检索到 {state.top_k} 条结果"}

builder = StateGraph(RAGState)
builder.add_node("retrieve", retrieve_node)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", END)
graph = builder.compile()

# 统一入口：封装为服务类
class RAGService:
    def __init__(self, graph):
        self.graph = graph

    def query(self, input_data: dict) -> dict:
        try:
            result = self.graph.invoke(input_data)
            return {"success": True, "data": result}
        except ValidationError as e:
            return {
                "success": False,
                "errors": [
                    {"field": '.'.join(str(l) for l in err['loc']), "message": err['msg']}
                    for err in e.errors()
                ]
            }
        except Exception as e:
            return {"success": False, "message": str(e)}

service = RAGService(graph)
print(service.query({"query": "什么是 RAG？"}))  # 成功
print(service.query({"query": ""}))              # 验证失败，返回错误详情
```

### 2. 为不同类型的错误设计不同的恢复策略

核心思路：区分"可恢复错误"（可选字段超范围）和"致命错误"（必填字段缺失），分别处理。

```python
from pydantic import ValidationError

def smart_error_handler(e: ValidationError, input_data: dict) -> dict:
    """根据错误类型选择不同的恢复策略"""
    recoverable = set()
    fatal = set()

    for err in e.errors():
        field = err['loc'][0] if err['loc'] else 'unknown'
        err_type = err['type']

        if err_type in ('greater_than_equal', 'less_than_equal',
                        'greater_than', 'less_than'):
            recoverable.add(field)  # 范围错误 → 可用默认值替代
        else:
            fatal.add(field)        # 其他错误 → 无法自动修复

    if fatal:
        return {"action": "reject", "message": f"无法自动修复：{fatal}"}

    cleaned = {k: v for k, v in input_data.items() if k not in recoverable}
    return {"action": "retry_with_defaults", "cleaned_input": cleaned}
```

## 常见错误类型速查表

| 错误类型 (type) | 触发条件 | 示例 |
|----------------|---------|------|
| `missing` | 必填字段未提供 | `AgentState()` 缺少 `query` |
| `string_type` | 需要字符串但传入其他类型 | `query=123` 且无法转换 |
| `int_type` | 需要整数但传入其他类型 | `max_tokens="abc"` |
| `greater_than_equal` | 值小于最小约束 | `max_tokens=-1` (ge=1) |
| `less_than_equal` | 值大于最大约束 | `temperature=5.0` (le=2.0) |
| `string_too_short` | 字符串短于最小长度 | `query=""` (min_length=1) |
| `string_pattern_mismatch` | 不匹配正则表达式 | 不符合 pattern 约束 |
| `value_error` | 自定义 validator 抛出 ValueError | `field_validator` 中的自定义检查 |

## 小结

验证错误处理的核心要点：

1. **ValidationError** 提供结构化错误信息（字段路径、错误类型、原始输入），方便程序化处理
2. **在图的入口处统一捕获**，而不是在每个节点内部分散处理
3. **区分验证错误和运行时错误**——前者拒绝并提示，后者重试或降级
4. **LangGraph 内置错误**（`InvalidUpdateError`、`GraphRecursionError`）也需要纳入处理体系
5. **优雅降级**——可选字段用默认值替代，必填字段返回清晰错误提示
