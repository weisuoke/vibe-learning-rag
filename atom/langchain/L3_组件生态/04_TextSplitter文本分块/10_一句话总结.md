# 一句话总结：TextSplitter 文本分块

> 9 个核心概念的精华提炼与记忆口诀

---

## 核心概念速查表

### 1. TextSplitter 基类架构

**一句话总结**：所有分块器的抽象基类，定义统一接口和核心算法

**记忆口诀**：基类定骨架，子类填血肉

**使用场景**：理解分块器架构，自定义分块器开发

---

### 2. CharacterTextSplitter

**一句话总结**：基于单个分隔符的简单文本分块器

**记忆口诀**：单一分隔，简单高效

**使用场景**：结构简单、分隔符明确的文本

---

### 3. RecursiveCharacterTextSplitter

**一句话总结**：递归尝试多个分隔符，优先保留大语义单元

**记忆口诀**：递归分割，语义优先

**使用场景**：通用文本、代码文件（LangChain 默认推荐）

**推荐指数**：⭐⭐⭐⭐⭐

---

### 4. TokenTextSplitter

**一句话总结**：基于 token 计数精确控制块大小和成本

**记忆口诀**：Token 计数，成本可控

**使用场景**：成本敏感、多语言文本、Context Window 受限

---

### 5. HTMLHeaderTextSplitter

**一句话总结**：基于 HTML 标题标签分块，保留文档层级结构

**记忆口诀**：HTML 标签，结构保留

**使用场景**：网页内容、技术文档、在线文档 RAG

---

### 6. MarkdownHeaderTextSplitter

**一句话总结**：基于 Markdown 标题分块，代码块感知

**记忆口诀**：Markdown 标题，代码感知

**使用场景**：技术文档、README 文件、博客文章

---

### 7. PythonCodeTextSplitter

**一句话总结**：语法感知的代码分块器，支持 20+ 种编程语言

**记忆口诀**：语法感知，代码完整

**使用场景**：代码仓库分析、代码问答系统、代码文档生成

---

### 8. RecursiveJsonSplitter

**一句话总结**：基于 JSON 结构分块，保留键值对完整性

**记忆口诀**：JSON 结构，键值完整

**使用场景**：API 响应、配置文件、结构化数据检索

---

### 9. 语义分块与 NLP 分块器

**一句话总结**：基于 NLP 技术识别句子边界，保持语义完整性

**记忆口诀**：句子边界，语义完整

**使用场景**：学术论文、新闻文章、多语言文档、高质量 RAG

**三大核心分块器**：
- **NLTKTextSplitter**：轻量级，通用文本
- **SpacyTextSplitter**：高准确度，复杂文本
- **SentenceTransformersTokenTextSplitter**：精确 token 控制，Embedding 场景

## 最佳实践总结

### 1. 通用配置（推荐）

```python
RecursiveCharacterTextSplitter(
    chunk_size=1000,      # 1000 characters
    chunk_overlap=200,    # 20% 重叠
    add_start_index=True  # 便于溯源
)
```

### 2. 成本优化配置

```python
TokenTextSplitter.from_tiktoken_encoder(
    model_name="gpt-4",
    chunk_size=512,       # 512 tokens
    chunk_overlap=50
)
```

### 3. 高质量 RAG 配置

```python
SpacyTextSplitter(
    chunk_size=1000,
    chunk_overlap=100,
    pipeline="en_core_web_sm"
)
```

### 4. 代码分块配置

```python
RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=1000,
    chunk_overlap=200
)
```

## 常见误区

### 误区 1：chunk_size 越大越好
**正确理解**：需要根据 LLM 的 context window 和检索质量平衡
- 推荐：1000 characters 或 512 tokens

### 误区 2：chunk_overlap 是浪费
**正确理解**：chunk_overlap 是保持语义连续性的必要投资
- 推荐：chunk_size 的 15-20%

### 误区 3：所有文本都应该用相同的分块器
**正确理解**：不同文本类型需要不同的分块策略
- 通用文本 → RecursiveCharacterTextSplitter
- 代码文件 → PythonCodeTextSplitter
- 网页内容 → HTMLHeaderTextSplitter

### 误区 4：字符数 = Token 数
**正确理解**：不同语言的字符数和 token 数差异很大
- 英文："Hello world" = 2 tokens
- 中文："你好世界" = 4 tokens
- 使用 TokenTextSplitter 精确控制

### 误区 5：RecursiveCharacterTextSplitter 比 CharacterTextSplitter 慢
**正确理解**：性能差异很小，语义完整性提升显著
- 优先使用 RecursiveCharacterTextSplitter

## 一句话精华总结

**TextSplitter 的本质**：
TextSplitter 是应对 LLM context window 物理约束的必然产物，通过将长文本分割成固定大小的块，在满足硬件限制的同时最大化语义完整性和检索精度。

**核心价值**：
1. 适配物理约束（LLM context window 限制）
2. 提高检索精度（小块更容易匹配查询）
3. 优化成本（只处理相关块）

**推荐配置**：
- RecursiveCharacterTextSplitter（默认选择）
- chunk_size=1000 characters / 512 tokens
- chunk_overlap=200 characters / 50 tokens
- add_start_index=True

**记忆口诀**：
```
TextSplitter 分文本，
递归分隔是首选，
千字两百好配置，
Token 计数省成本，
结构保留用专用，
RAG 开发必备器。
```
