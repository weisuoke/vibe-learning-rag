# 核心概念 3：循环控制机制

> Agent 的执行循环就像一辆高速行驶的汽车——没有刹车系统，再强的引擎也只会带来灾难。循环控制机制就是 AgentExecutor 的刹车、油门和仪表盘。

---

## 为什么需要循环控制？

AgentExecutor 的核心是一个 `while True` 循环。没有控制机制，这个循环可能：

1. **无限运行**：LLM 反复调用工具却永远不给出最终答案
2. **烧光预算**：每次循环都消耗 LLM tokens + 工具调用费用
3. **阻塞系统**：一个失控的 Agent 占用资源，影响其他请求
4. **用户等待**：用户发了一个问题，等了 5 分钟还没响应

```python
# 没有控制的危险循环
while True:
    action = llm.plan(question, history)  # 每次调用花钱
    if action.type == "finish":
        return action.answer
    result = tool.run(action)             # 每次调用花时间
    history.append(result)
    # 如果 LLM 永远不返回 "finish" 呢？💸💸💸
```

AgentExecutor 提供了四个控制参数来解决这些问题：

| 参数 | 类型 | 默认值 | 作用 |
|------|------|--------|------|
| `max_iterations` | `int` | 15 | 最大循环次数 |
| `max_execution_time` | `float \| None` | `None` | 最大执行时间（秒） |
| `early_stopping_method` | `str` | `"force"` | 达到上限后的处理方式 |
| `trim_intermediate_steps` | `int \| Callable` | `-1` | 中间步骤裁剪（详见核心概念 5） |

## max_iterations：迭代次数上限

### 源码定义（agent.py:1012）

```python
class AgentExecutor(Chain):
    max_iterations: int | None = 15
```

### 为什么默认是 15？

这个数字不是随便选的，背后有实际考量：

```
常见 Agent 任务的迭代次数分布：

简单问答（1个工具调用）：    1-2 次   ████
中等任务（2-3个工具调用）：  3-5 次   ████████
复杂任务（多步推理）：       5-10 次  ████████████
极端任务（研究型）：         10-15 次 ████████████████

15 次覆盖了 99% 的正常场景。
超过 15 次，大概率是 Agent 陷入了死循环。
```

**实际经验：**
- 大多数 ReAct Agent 在 3-5 次迭代内完成任务
- Function Calling Agent 通常更快，2-3 次
- 如果超过 10 次还没完成，通常说明 Prompt 设计有问题或工具不匹配

### 如何调优 max_iterations

```python
from langchain.agents import AgentExecutor

# 场景 1：简单工具调用（查天气、算数学）
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=5,  # 5 次足够
)

# 场景 2：多步研究任务（搜索 + 分析 + 对比）
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,  # 给更多空间
)

# 场景 3：复杂数据分析（多次查询数据库 + 计算）
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=20,  # 需要更多步骤
)

# 场景 4：不限制（危险！仅用于调试）
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=None,  # 无限循环，生产环境绝对不要这样做
)
```

### 调优原则

```
max_iterations 的设置公式：

推荐值 = 预期步骤数 × 2 + 缓冲量

例如：
  预期 3 步完成 → max_iterations = 3 × 2 + 2 = 8
  预期 5 步完成 → max_iterations = 5 × 2 + 2 = 12

为什么乘以 2？
  因为 Agent 可能会犯错重试，需要额外的迭代空间。
```

## max_execution_time：墙钟时间限制

### 源码定义（agent.py:1012）

```python
class AgentExecutor(Chain):
    max_execution_time: float | None = None  # 单位：秒
```

### 为什么默认是 None？

默认不设置时间限制，因为不同工具的执行时间差异巨大：

| 工具类型 | 典型耗时 | 说明 |
|----------|----------|------|
| 数学计算 | < 1ms | 纯 CPU 计算 |
| 向量检索 | 10-100ms | 本地向量库 |
| API 调用 | 100ms-2s | 网络延迟 |
| 网页搜索 | 1-5s | 搜索引擎 API |
| LLM 调用 | 1-30s | 取决于模型和 token 数 |
| 数据库查询 | 100ms-60s | 取决于查询复杂度 |

一个 5 步的 Agent，如果每步包含一次 LLM 调用（~3s）+ 一次工具调用（~1s），总耗时约 20s。但如果某个工具卡住了，可能等 5 分钟都没响应。

### 使用方式

```python
# 设置 30 秒超时
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_execution_time=30.0,  # 30 秒后强制停止
)

# 结合 max_iterations 使用（双重保险）
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,
    max_execution_time=60.0,  # 哪个先到就停
)
```

### max_iterations vs max_execution_time

```
场景对比：

场景 A：Agent 快速循环但步骤多
  每步耗时 1 秒，循环 20 次 = 20 秒
  max_iterations=15 → 第 15 次停止（15 秒）  ← 先触发
  max_execution_time=60 → 不会触发

场景 B：Agent 步骤少但某步很慢
  第 1 步：1 秒
  第 2 步：调用外部 API，卡住 120 秒
  max_iterations=15 → 不会触发（才第 2 步）
  max_execution_time=60 → 60 秒时停止          ← 先触发

结论：两个参数互补，建议同时设置。
```

## _should_continue：循环守卫条件

### 源码实现（agent.py:1235）

这是控制循环是否继续的核心方法：

```python
def _should_continue(self, iterations: int, time_elapsed: float) -> bool:
    # 检查 1：迭代次数是否超限
    if self.max_iterations is not None and iterations >= self.max_iterations:
        return False

    # 检查 2：执行时间是否超限
    if self.max_execution_time is not None and time_elapsed >= self.max_execution_time:
        return False

    return True
```

逻辑非常简洁——两个条件，任一触发就停止。

### 在主循环中的位置

```python
# agent.py:1570 _call 方法（简化）
def _call(self, inputs, run_manager=None):
    intermediate_steps = []
    iterations = 0
    time_elapsed = 0.0
    start_time = time.time()

    # ★ _should_continue 是循环的守卫条件
    while self._should_continue(iterations, time_elapsed):
        # 执行一步
        next_step_output = self._take_next_step(
            name_to_tool_map, color_mapping,
            inputs, intermediate_steps, run_manager
        )

        # Agent 主动结束
        if isinstance(next_step_output, AgentFinish):
            return self._return(next_step_output, intermediate_steps, run_manager)

        intermediate_steps.extend(next_step_output)
        iterations += 1
        time_elapsed = time.time() - start_time

    # ★ 循环结束但 Agent 没有主动 finish → 触发 early stopping
    output = self.agent.return_stopped_response(
        self.early_stopping_method, intermediate_steps, **inputs
    )
    return self._return(output, intermediate_steps, run_manager)
```

### 执行流程图

```
开始执行
│
▼
iterations = 0, start_time = now()
│
▼
┌─────────────────────────────────────────┐
│  _should_continue(iterations, elapsed)  │
│                                         │
│  iterations >= max_iterations?  ──Yes──→ 跳出循环
│  elapsed >= max_execution_time? ──Yes──→ 跳出循环
│  否则 → 继续                            │
└─────────────────┬───────────────────────┘
                  │ 继续
                  ▼
          _take_next_step()
                  │
          ┌───────┴───────┐
          │               │
     AgentFinish      AgentStep
          │               │
     _return()      intermediate_steps.extend()
     (正常结束)     iterations += 1
                    time_elapsed = now() - start
                          │
                          ▼
                    回到 _should_continue
                          │
                    (如果跳出循环)
                          ▼
              early_stopping_method 处理
              (非正常结束，需要善后)
```

## early_stopping_method：优雅退出策略

当 `_should_continue` 返回 `False` 时，Agent 没有主动给出最终答案。这时候怎么办？

### 两种策略

```python
class AgentExecutor(Chain):
    early_stopping_method: str = "force"  # "force" 或 "generate"
```

#### 策略一："force"（默认）

直接返回一个固定字符串，不再调用 LLM：

```python
# BaseSingleActionAgent.return_stopped_response（简化）
def return_stopped_response(self, early_stopping_method, intermediate_steps, **kwargs):
    if early_stopping_method == "force":
        return AgentFinish(
            return_values={"output": "Agent stopped due to iteration limit or time limit."},
            log=""
        )
```

**优点：** 零额外成本，立即返回
**缺点：** 用户看到的是一个冷冰冰的错误信息，没有任何有用内容

#### 策略二："generate"

再调用一次 LLM，让它基于已有的中间步骤生成一个"尽力而为"的答案：

```python
def return_stopped_response(self, early_stopping_method, intermediate_steps, **kwargs):
    if early_stopping_method == "generate":
        # 把所有中间步骤传给 LLM，让它总结出一个答案
        thoughts = ""
        for action, observation in intermediate_steps:
            thoughts += action.log
            thoughts += f"\nObservation: {observation}\nThought: "

        # 构造一个特殊的 prompt，要求 LLM 给出最终答案
        thoughts += (
            "\n\nI need to give a final answer based on "
            "the previous observations. Let me summarize what I found.\n"
        )
        # 调用 LLM 生成最终答案
        output = self.agent.llm_chain.predict(
            intermediate_steps=intermediate_steps, **kwargs
        )
        return AgentFinish(return_values={"output": output}, log=thoughts)
```

**优点：** 用户能得到一个有意义的回答（虽然可能不完整）
**缺点：** 额外消耗一次 LLM 调用的 tokens 和时间

### 如何选择？

```python
# 场景 1：内部工具，用户能理解错误信息
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,
    early_stopping_method="force",  # 快速失败
)

# 场景 2：面向终端用户，需要友好体验
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,
    early_stopping_method="generate",  # 尽力给出答案
)
```

### 对比表

| 维度 | "force" | "generate" |
|------|---------|------------|
| 额外 LLM 调用 | 0 次 | 1 次 |
| 响应速度 | 立即返回 | 多等 1-5 秒 |
| 用户体验 | 差（固定错误信息） | 较好（有内容的回答） |
| 额外成本 | 无 | 一次 LLM 调用费用 |
| 适用场景 | 内部系统、调试 | 面向用户的产品 |
| 回答质量 | 无回答 | 基于已有信息的总结 |

## 异步超时处理

AgentExecutor 同时支持同步和异步执行。异步模式下的超时处理有所不同。

### 异步主循环（agent.py 中的 _acall）

```python
import asyncio

async def _acall(self, inputs, run_manager=None):
    intermediate_steps = []
    iterations = 0
    time_elapsed = 0.0
    start_time = time.time()

    while self._should_continue(iterations, time_elapsed):
        # 异步执行一步
        next_step_output = await self._atake_next_step(
            name_to_tool_map, color_mapping,
            inputs, intermediate_steps, run_manager
        )

        if isinstance(next_step_output, AgentFinish):
            return await self._areturn(next_step_output, intermediate_steps, run_manager)

        intermediate_steps.extend(next_step_output)
        iterations += 1
        time_elapsed = time.time() - start_time

    # early stopping（同步逻辑一样）
    output = self.agent.return_stopped_response(
        self.early_stopping_method, intermediate_steps, **inputs
    )
    return await self._areturn(output, intermediate_steps, run_manager)
```

### 异步超时的注意事项

```python
# 方式 1：使用 AgentExecutor 自带的 max_execution_time
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_execution_time=30.0,
)
result = await executor.ainvoke({"input": "..."})

# 方式 2：使用 asyncio.wait_for 做外部超时控制
try:
    result = await asyncio.wait_for(
        executor.ainvoke({"input": "..."}),
        timeout=30.0
    )
except asyncio.TimeoutError:
    result = {"output": "请求超时，请稍后重试"}
```

**两种方式的区别：**

```
max_execution_time（内部超时）：
  - 在每轮循环开始时检查
  - 如果某个工具调用本身耗时 60 秒，这 60 秒内不会被中断
  - 触发后走 early_stopping_method 流程

asyncio.wait_for（外部超时）：
  - 真正的硬超时，到时间直接取消协程
  - 不走 early_stopping_method，直接抛 TimeoutError
  - 更适合 API 服务场景（需要严格的响应时间保证）
```

### 生产环境推荐：双重超时

```python
async def safe_agent_call(executor, question, soft_timeout=30, hard_timeout=45):
    """双重超时保护"""
    try:
        # 外部硬超时
        result = await asyncio.wait_for(
            executor.ainvoke({"input": question}),
            timeout=hard_timeout
        )
        return result
    except asyncio.TimeoutError:
        return {"output": "抱歉，处理时间过长，请简化您的问题后重试。"}

# executor 内部有 soft_timeout（走 early_stopping）
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,
    max_execution_time=30.0,        # 软超时：30 秒
    early_stopping_method="generate",
)

# 外部有 hard_timeout（兜底）
result = await safe_agent_call(executor, "分析这份报告", soft_timeout=30, hard_timeout=45)
```

## 控制参数组合的最佳实践

### 开发环境

```python
dev_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=25,              # 宽松，方便调试
    max_execution_time=120.0,       # 2 分钟
    early_stopping_method="force",  # 快速看到是否超限
    verbose=True,                   # 打印每一步
    return_intermediate_steps=True, # 返回推理过程
)
```

### 生产环境（API 服务）

```python
prod_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10,              # 严格限制
    max_execution_time=30.0,        # 30 秒 SLA
    early_stopping_method="generate",  # 给用户有意义的回答
    verbose=False,
    return_intermediate_steps=False,
    handle_parsing_errors=True,     # 自动处理解析错误
)
```

### 批处理场景

```python
batch_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=20,              # 允许更多步骤
    max_execution_time=300.0,       # 5 分钟（批处理不急）
    early_stopping_method="force",  # 失败就跳过，处理下一个
    verbose=False,
)
```

### 参数组合速查表

| 场景 | max_iterations | max_execution_time | early_stopping | 说明 |
|------|---------------|-------------------|----------------|------|
| 简单问答 | 5 | 15s | force | 快进快出 |
| 多步研究 | 10 | 60s | generate | 给足空间 |
| API 服务 | 8 | 30s | generate | 严格 SLA |
| 批处理 | 20 | 300s | force | 宽松但不卡住 |
| 调试 | 25 | 120s | force | 观察行为 |
| 演示 | 5 | 20s | generate | 快速响应 |

## 与 RAG 的关联

在 RAG Agent 中，循环控制直接影响检索质量和用户体验：

```python
# RAG Agent 的典型执行流程
# 第 1 轮：检索文档（~2s）
# 第 2 轮：发现不够相关，改写查询（~3s）
# 第 3 轮：重新检索（~2s）
# 第 4 轮：生成答案（~3s）
# 总计：~10s，4 次迭代

rag_executor = AgentExecutor(
    agent=rag_agent,
    tools=[retriever_tool, rewrite_tool],
    max_iterations=8,           # RAG 通常 4-6 步完成
    max_execution_time=30.0,    # 用户等待极限
    early_stopping_method="generate",  # 即使没检索到最佳结果，也给个答案
)
```

**RAG 场景的特殊考量：**

1. **检索工具耗时稳定**：向量检索通常 100ms 以内，时间主要花在 LLM 调用上
2. **迭代次数可预测**：检索 → 判断 → 可能改写 → 再检索 → 生成，通常 3-6 步
3. **early_stopping 用 "generate" 更好**：即使检索不完美，LLM 也能基于已有信息给出部分答案
4. **max_execution_time 要考虑用户耐心**：Web 应用通常 30 秒是极限

## 小结

循环控制机制是 AgentExecutor 的安全网，由四个参数协同工作：

- `max_iterations`（默认 15）：防止无限循环，按预期步骤数的 2 倍设置
- `max_execution_time`（默认 None）：防止单步卡死，生产环境必须设置
- `_should_continue`：每轮循环开始时检查两个条件，任一触发就停止
- `early_stopping_method`：超限后的善后策略，"force" 快速失败，"generate" 尽力回答

一句话：**max_iterations 管"做了多少步"，max_execution_time 管"花了多少时间"，early_stopping_method 管"超限了怎么收场"——三者配合，让 Agent 既能充分工作，又不会失控。**
