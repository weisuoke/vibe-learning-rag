# Agent类型与选择 - 一句话总结

## 核心总结（30-50字）

**根据模型能力（函数调用 vs 纯文本）和工具复杂度（简单 vs 多输入）选择合适的Agent类型，2026年优先使用create_agent()统一API创建Tool Calling Agent。**

---

## 为什么这句话重要？

这句话浓缩了Agent类型选择的三个核心维度：

### 1. 模型能力（第一决策层）

**函数调用 vs 纯文本**
- 支持函数调用（OpenAI, Anthropic）→ 可选择 OpenAI Functions / Structured Chat
- 不支持函数调用（开源模型）→ 只能选择 ReAct

**关键洞察**：模型能力是硬约束，决定了可选范围。

---

### 2. 工具复杂度（第二决策层）

**简单 vs 多输入**
- 简单工具（1-3个参数）→ OpenAI Functions Agent（最优）
- 复杂工具（多参数/嵌套结构）→ Structured Chat Agent（必需）

**关键洞察**：工具复杂度决定了技术选型，避免过度工程或能力不足。

---

### 3. 2026最佳实践（推荐方向）

**create_agent() 统一API**
- 自动选择最佳Agent类型
- 简化决策成本
- 官方推荐方向

**关键洞察**：新项目优先使用统一API，除非有特殊需求。

---

## 实战应用

### 场景1：RAG知识库问答

**需求**：OpenAI gpt-4 + 向量检索工具（简单参数）

**选择**：
```python
from langchain.agents import create_agent
agent = create_agent(llm, [vector_search_tool])
```

**原因**：模型支持函数调用 + 工具简单 → 使用2026推荐API

---

### 场景2：开源模型应用

**需求**：Llama-3 + 搜索工具

**选择**：
```python
from langchain.agents import create_react_agent
agent = create_react_agent(llm, tools, prompt)
```

**原因**：模型不支持函数调用 → 只能用ReAct

---

### 场景3：数据分析系统

**需求**：OpenAI gpt-4 + 数据库查询工具（多参数+过滤条件）

**选择**：
```python
from langchain.agents import create_structured_chat_agent
agent = create_structured_chat_agent(llm, [db_query_tool], prompt)
```

**原因**：模型支持函数调用 + 工具复杂 → 使用Structured Chat

## 延伸思考

### 问题1：为什么不是"总是用最新的"？

**答案**：因为不同场景有不同需求
- 开源模型场景 → ReAct仍是最佳选择
- 复杂工具场景 → Structured Chat不可替代
- 调试场景 → ReAct的可解释性更好

**关键洞察**：技术选型要基于实际需求，而不是追新。

---

### 问题2：为什么强调"2026年"？

**答案**：因为技术在演进
- 2023年：主要用initialize_agent()（已弃用）
- 2024-2025年：主要用create_*_agent()
- 2026年：推荐用create_agent()统一API

**关键洞察**：保持对最新最佳实践的关注。

---

### 问题3：如果选错了怎么办？

**答案**：轻松切换，成本很低
```python
# 从OpenAI Functions切换到ReAct
# agent = create_openai_functions_agent(llm, tools, prompt)
agent = create_react_agent(llm, tools, prompt)  # 只改这一行
```

**关键洞察**：Agent类型不是架构级决策，可以灵活调整。
