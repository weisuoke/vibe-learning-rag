# 实战代码 - 场景4：RAG 场景状态转换

## 场景说明

本场景演示如何用 LangGraph 构建一个完整的 RAG（检索增强生成）管道，重点展示**每个节点如何专注于一步状态转换**，将用户查询逐步转化为最终答案。

具体流程：用户查询 → 查询改写 → 文档检索 → 结果过滤清洗 → 上下文组装 → 答案生成，每一步都是一个独立的状态转换函数。

核心知识点：
1. RAG 管道的状态设计——如何用 TypedDict 描述检索增强生成的全流程数据
2. 单一职责节点——每个节点只做一步转换（改写、检索、过滤、组装、生成）
3. Reducer 累积日志——`Annotated[list, add]` 跨节点追踪处理过程
4. 模拟 LLM 调用——不依赖 API Key，本地即可运行验证管道逻辑
5. stream 模式观察——逐步查看 RAG 管道中每个节点的状态变化

[来源: sourcecode/langgraph/libs/langgraph/langgraph/graph/state.py]
[来源: reference/context7_langgraph_01.md | LangGraph 官方文档]

         └──────┬───────┘
                │
         ┌──────▼───────┐
         │    filter    │  ← 过滤 + 清洗
         └──────┬───────┘
                │
         ┌──────▼───────┐
         │    format    │  ← 上下文组装
         └──────┬───────┘
                │
         ┌──────▼───────┐
         │   generate   │  ← 答案生成（模拟）
         └──────┬───────┘
                │
         ┌──────▼───────┐
         │     END      │
         └──────────────┘

关键点：这是一个纯线性管道，每个节点只读取上游产出的字段、处理后写入下游需要的字段。`logs` 字段使用 `Annotated[list, add]` Reducer，跨所有节点自动累积处理日志。

## 完整代码

```python
"""
LangGraph 状态转换函数 - RAG 场景状态转换
演示：文档检索结果的清洗、格式化与注入

场景：完整 RAG 管道
- 查询改写 → 优化用户原始查询
- 文档检索 → 模拟向量数据库检索
- 结果过滤 → 去除低质量 / 空文档
- 上下文组装 → 格式化为 LLM 可用的上下文
- 答案生成 → 模拟 LLM 生成最终回答

核心技术：
- RAG 状态设计（TypedDict 描述全流程数据）
- 单一职责节点（每个节点只做一步转换）
- Reducer 累积日志（跨节点追踪处理过程）
- 模拟 LLM（不依赖 API Key，本地可运行）

运行环境：Python 3.13+, langgraph
安装依赖：uv add langgraph
"""

from typing import Annotated
from typing_extensions import TypedDict
from operator import add
from langgraph.graph import StateGraph, START, END


# ============================================================
# 1. RAG 状态定义
# ============================================================

class RAGState(TypedDict):
    """RAG 管道状态

    无 Reducer 字段（LastValue Channel）：
    - original_query, rewritten_query → 查询相关，直接覆盖
    - documents, filtered_docs → 文档列表，直接覆盖
    - context → 组装后的上下文字符串，直接覆盖
    - answer → 最终答案，直接覆盖
    - metadata → 元数据字典，直接覆盖

    有 Reducer 字段（BinaryOperatorAggregate Channel）：
    - logs → 使用 operator.add，每个节点返回的日志列表会追加
    """
    # 无 Reducer：直接覆盖
    original_query: str          # 用户原始查询
    rewritten_query: str         # 改写后的查询
    documents: list[dict]        # 检索到的原始文档
    filtered_docs: list[dict]    # 过滤清洗后的文档
    context: str                 # 组装的上下文字符串
    answer: str                  # 最终生成的答案
    metadata: dict               # 管道元数据

    # 有 Reducer：跨节点累积追加
    logs: Annotated[list[str], add]


# ============================================================
# 2. 模拟数据——替代真实向量数据库
# ============================================================

MOCK_DOCUMENTS = [
    {
        "content": "LangGraph 是一个用于构建有状态多 Agent 应用的框架",
        "source": "langgraph_intro.md",
        "score": 0.95,
    },
    {
        "content": "状态转换函数是节点处理数据的核心机制",
        "source": "state_transform.md",
        "score": 0.88,
    },
    {
        "content": "今天天气很好，适合出去玩",
        "source": "noise.md",
        "score": 0.30,
    },
    {
        "content": "Reducer 函数定义了状态字段如何合并更新",
        "source": "reducer_guide.md",
        "score": 0.82,
    },
    {
        "content": "",
        "source": "empty.md",
        "score": 0.10,
    },
    {
        "content": "Python 是一种编程语言",
        "source": "generic.md",
        "score": 0.45,
    },
]


# ============================================================
# 3. RAG 节点函数——每个节点只做一步转换
# ============================================================

def rewrite_query(state: RAGState) -> dict:
    """节点1：查询改写

    职责：优化用户的原始查询，补充关键词提高检索召回率
    - 去除首尾空白
    - 追加领域关键词（模拟 LLM 改写）

    读取：original_query
    写入：rewritten_query, logs
    """
    query = state["original_query"].strip()

    # 模拟 LLM 查询改写（实际中会调用 OpenAI API）
    rewritten = f"{query} LangGraph 状态管理 最佳实践"

    return {
        "rewritten_query": rewritten,
        "logs": [f"[rewrite] '{query}' → '{rewritten}'"],
    }


def retrieve_docs(state: RAGState) -> dict:
    """节点2：文档检索

    职责：根据改写后的查询，从向量数据库检索相关文档
    - 这里使用 MOCK_DOCUMENTS 模拟向量检索结果
    - 实际场景中会调用 ChromaDB / Milvus / FAISS

    读取：rewritten_query
    写入：documents, logs
    """
    query = state["rewritten_query"]

    # 模拟向量检索（实际中：docs = vector_store.similarity_search(query)）
    docs = MOCK_DOCUMENTS.copy()

    return {
        "documents": docs,
        "logs": [f"[retrieve] 检索到 {len(docs)} 篇文档"],
    }


def filter_and_clean(state: RAGState) -> dict:
    """节点3：过滤与清洗

    职责：去除低质量文档，清洗文档内容
    过滤规则：
    - 空内容文档 → 移除
    - 相关度分数 < 0.5 → 移除
    - 内容长度 < 10 字符 → 移除
    清洗操作：
    - 去除首尾空白
    - 标准化字段格式
    - 按分数降序排列

    读取：documents
    写入：filtered_docs, logs
    """
    docs = state["documents"]

    filtered = []
    removed_count = 0

    for doc in docs:
        # 过滤条件1：空内容
        if not doc.get("content") or not doc["content"].strip():
            removed_count += 1
            continue
        # 过滤条件2：低相关度
        if doc.get("score", 0) < 0.5:
            removed_count += 1
            continue
        # 过滤条件3：内容过短
        if len(doc["content"].strip()) < 10:
            removed_count += 1
            continue

        # 清洗：标准化字段
        cleaned_doc = {
            "content": doc["content"].strip(),
            "source": doc.get("source", "unknown"),
            "score": round(float(doc.get("score", 0)), 2),
        }
        filtered.append(cleaned_doc)

    # 按相关度分数降序排列
    filtered.sort(key=lambda x: x["score"], reverse=True)

    return {
        "filtered_docs": filtered,
        "logs": [
            f"[filter] {len(docs)} → {len(filtered)} 篇 (移除 {removed_count})"
        ],
    }


def format_context(state: RAGState) -> dict:
    """节点4：上下文组装

    职责：将过滤后的文档格式化为 LLM 可用的上下文字符串
    - 按 [文档N] 格式编号
    - 附带来源和相关度信息
    - 截断过长上下文（模拟 token 限制）
    - 记录元数据（文档数、上下文长度、最高分数）

    读取：filtered_docs
    写入：context, metadata, logs
    """
    docs = state["filtered_docs"]

    # 格式化每篇文档
    context_parts = []
    for i, doc in enumerate(docs, 1):
        part = (
            f"[文档{i}] (来源: {doc['source']}, 相关度: {doc['score']})\n"
            f"{doc['content']}"
        )
        context_parts.append(part)

    context = "\n\n".join(context_parts)

    # 截断过长上下文（模拟 token 限制）
    max_length = 2000
    truncated = False
    if len(context) > max_length:
        context = context[:max_length] + "\n...(已截断)"
        truncated = True

    return {
        "context": context,
        "metadata": {
            "doc_count": len(docs),
            "context_length": len(context),
            "top_score": docs[0]["score"] if docs else 0,
            "truncated": truncated,
        },
        "logs": [
            f"[format] 组装 {len(docs)} 篇文档, {len(context)} chars"
        ],
    }


def generate_answer(state: RAGState) -> dict:
    """节点5：答案生成

    职责：基于查询和上下文，生成最终答案
    - 模拟 LLM 生成（实际中会调用 OpenAI API）
    - 引用检索到的文档数量
    - 综合上下文信息生成回答

    读取：original_query, context, metadata
    写入：answer, logs
    """
    query = state["original_query"]
    doc_count = state["metadata"].get("doc_count", 0)

    # 模拟 LLM 生成（实际中：response = openai.chat.completions.create(...)）
    answer = (
        f"根据检索到的 {doc_count} 篇相关文档，"
        f"关于「{query}」的回答如下：\n\n"
        f"LangGraph 的状态转换函数是节点处理数据的核心机制。"
        f"每个节点接收完整状态，执行特定转换逻辑，返回部分更新。"
        f"通过 Reducer 函数定义状态合并策略，"
        f"实现安全可控的状态管理。"
    )

    return {
        "answer": answer,
        "logs": [f"[generate] 答案生成完成 ({len(answer)} chars)"],
    }


# ============================================================
# 4. 构建 RAG 管道——线性串联
# ============================================================

graph = StateGraph(RAGState)

# 添加节点
graph.add_node("rewrite", rewrite_query)
graph.add_node("retrieve", retrieve_docs)
graph.add_node("filter", filter_and_clean)
graph.add_node("format", format_context)
graph.add_node("generate", generate_answer)

# 线性串联：rewrite → retrieve → filter → format → generate
graph.add_edge(START, "rewrite")
graph.add_edge("rewrite", "retrieve")
graph.add_edge("retrieve", "filter")
graph.add_edge("filter", "format")
graph.add_edge("format", "generate")
graph.add_edge("generate", END)

# 编译
rag_pipeline = graph.compile()


# ============================================================
# 5. 测试用例
# ============================================================

def make_initial_state(query: str) -> RAGState:
    """构造初始状态——所有字段都需要初始值"""
    return {
        "original_query": query,
        "rewritten_query": "",
        "documents": [],
        "filtered_docs": [],
        "context": "",
        "answer": "",
        "metadata": {},
        "logs": ["RAG 管道启动"],
    }


# --- 测试1：invoke 模式——一次性执行，返回最终状态 ---
print("=" * 60)
print("=== 测试1：invoke 模式 - 完整 RAG 管道 ===")
print("=" * 60)

result = rag_pipeline.invoke(
    make_initial_state("LangGraph 状态转换函数怎么用")
)

print(f"\n原始查询: {result['original_query']}")
print(f"改写查询: {result['rewritten_query']}")
print(f"检索文档: {len(result['documents'])} 篇")
print(f"过滤后:   {len(result['filtered_docs'])} 篇")
print(f"\n--- 过滤后的文档 ---")
for doc in result["filtered_docs"]:
    print(f"  [{doc['score']}] {doc['source']}: {doc['content']}")
print(f"\n--- 上下文 ---\n{result['context']}")
print(f"\n--- 答案 ---\n{result['answer']}")
print(f"\n--- 元数据 ---\n{result['metadata']}")
print(f"\n--- 处理日志 ---")
for log in result["logs"]:
    print(f"  {log}")


# --- 测试2：stream 模式——逐步观察每个节点的状态变化 ---
print()
print("=" * 60)
print("=== 测试2：stream 模式 - 逐步观察状态变化 ===")
print("=" * 60)

for step in rag_pipeline.stream(
    make_initial_state("什么是 Reducer")
):
    # step 格式: {"node_name": {节点返回的部分更新}}
    for node_name, node_output in step.items():
        print(f"\n--- 节点: {node_name} ---")
        for key, value in node_output.items():
            # 截断过长的值
            display = str(value)
            if len(display) > 100:
                display = display[:100] + "..."
            print(f"  {key}: {display}")
```

## 关键知识点解析

### 1. RAG 状态设计原则

| 设计决策 | 选择 | 原因 |
|----------|------|------|
| `documents` 用覆盖模式 | 无 Reducer | 检索结果每次全量替换，不需要累积 |
| `filtered_docs` 用覆盖模式 | 无 Reducer | 过滤结果是 documents 的子集，全量替换 |
| `context` 用覆盖模式 | 无 Reducer | 上下文每次重新组装，不需要追加 |
| `logs` 用追加模式 | `Annotated[list, add]` | 每个节点的日志需要跨节点累积 |
| `metadata` 用覆盖模式 | 无 Reducer | 元数据由 format 节点一次性生成 |

### 2. 每个节点的读写映射

```
rewrite:   读 original_query        → 写 rewritten_query, logs
retrieve:  读 rewritten_query       → 写 documents, logs
filter:    读 documents             → 写 filtered_docs, logs
format:    读 filtered_docs         → 写 context, metadata, logs
generate:  读 original_query,       → 写 answer, logs
               context, metadata
```

这就是状态转换的核心：每个节点只关心自己需要的输入字段，只产出自己负责的输出字段。

### 3. 过滤节点的三层防御

```python
# 第1层：空内容检查
if not doc.get("content") or not doc["content"].strip():
    continue

# 第2层：相关度阈值
if doc.get("score", 0) < 0.5:
    continue

# 第3层：最小长度检查
if len(doc["content"].strip()) < 10:
    continue
```

在 RAG 系统中，过滤节点是保证生成质量的关键防线。低质量文档注入上下文会导致 LLM 产生幻觉或偏离主题。

### 4. stream vs invoke 在 RAG 调试中的价值

```python
# invoke：适合生产环境，一次性拿到最终答案
result = rag_pipeline.invoke(initial_state)
print(result["answer"])

# stream：适合开发调试，逐步检查每个节点的输出
for step in rag_pipeline.stream(initial_state):
    for node_name, output in step.items():
        print(f"{node_name}: {output}")
```

RAG 管道调试时，stream 模式可以帮你定位问题出在哪个环节：
- 检索结果不好？→ 看 `retrieve` 节点输出
- 过滤太激进？→ 看 `filter` 节点输出
- 上下文格式有问题？→ 看 `format` 节点输出

## 在 RAG 系统中的应用

本场景直接就是一个 RAG 系统的骨架。在生产环境中的扩展方向：

- **查询改写节点**：接入 LLM 做 HyDE（假设文档嵌入）、多查询扩展、意图识别
- **检索节点**：接入 ChromaDB / Milvus / FAISS，支持混合检索（向量 + 关键词）
- **过滤节点**：增加 ReRank 重排序（用交叉编码器对候选文档精排）
