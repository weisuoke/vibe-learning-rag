# 核心概念 3：环境变量体系

> **一句话定义：** 环境变量体系是 ZeroClaw 的"外部旋钮面板"——通过 `ZEROCLAW_*` 前缀的环境变量，在不修改配置文件的情况下控制 API Key、Provider、模型选择、Gateway 端口、代理、存储等所有运行时行为，并且遵循"环境变量 → 配置文件 → 代码默认值"的优先级链。

---

## 1. 什么是环境变量？为什么 ZeroClaw 需要它？

### 1.1 基础概念

**环境变量** = 操作系统级别的键值对，任何进程都可以读取。

```bash
# 设置环境变量
export ZEROCLAW_API_KEY="sk-or-v1-abc123"

# 读取环境变量
echo $ZEROCLAW_API_KEY   # → sk-or-v1-abc123
```

它和"写死在配置文件里"的区别是：**环境变量在进程外部设置，不需要改代码或配置文件**。

### 1.2 为什么不全写在配置文件里？

| 场景 | 配置文件（config.toml） | 环境变量 |
|------|----------------------|---------|
| 本地开发 | ✅ 方便，改一次就行 | ⚠️ 每次开终端要重新 export |
| Docker 部署 | ❌ 需要挂载文件 | ✅ `-e KEY=val` 一行搞定 |
| CI/CD | ❌ 不能把密钥提交到仓库 | ✅ 用 GitHub Secrets 注入 |
| 多环境切换 | ❌ 要维护多个配置文件 | ✅ 不同环境设不同变量 |
| 安全性 | ❌ 文件可能被泄露 | ✅ 不落盘，进程退出即消失 |

> **前端类比：** 你的 React 项目为什么用 `.env` + `process.env.REACT_APP_API_URL` 而不是硬编码？原因完全一样——不同环境（开发 / 测试 / 生产）需要不同的值，密钥不能提交到 Git。ZeroClaw 的环境变量体系就是 Rust 世界的 `.env` + `process.env`。

> **日常生活类比：** 配置文件像家里固定安装的灯开关——装修时决定了位置。环境变量像遥控器——你随时可以调亮度、切模式，不需要重新装修。

## 2. 环境变量分类体系

ZeroClaw 的环境变量按**功能域**分为 7 大类。

[来源: sourcecode/zeroclaw/src/config/mod.rs]

### 2.1 核心运行时（最常用 ⭐）

控制 ZeroClaw "连谁、用什么模型、在哪工作"。

| 变量 | 用途 | 默认值 | 示例 |
|------|------|-------|------|
| `ZEROCLAW_API_KEY` | Provider API Key | 无（必填） | `sk-or-v1-abc...` |
| `API_KEY` | 通用回退 API Key | 无 | `sk-...` |
| `ZEROCLAW_PROVIDER` | 默认 Provider | `openrouter` | `openai`, `ollama` |
| `PROVIDER` | 替代 Provider 变量 | 无 | `openrouter` |
| `ZEROCLAW_MODEL` | 默认模型 | 无 | `anthropic/claude-sonnet-4-6` |
| `MODEL` | 替代模型变量 | 无 | `gpt-4` |
| `ZEROCLAW_TEMPERATURE` | 温度 (0.0-2.0) | `0.7` | `0.5` |
| `ZEROCLAW_WORKSPACE` | 工作区目录 | `~/.zeroclaw` | `/opt/zeroclaw/ws` |
| `ZEROCLAW_CONFIG_DIR` | 配置目录 | `~/.zeroclaw` | `/etc/zeroclaw` |

```bash
# 最小启动配置（只需这两个就能跑）
export ZEROCLAW_API_KEY="sk-or-v1-your-key-here"
export ZEROCLAW_PROVIDER="openrouter"
```

### 2.2 Gateway 配置

控制 ZeroClaw 内置 HTTP 网关的监听行为。

| 变量 | 用途 | 默认值 | 说明 |
|------|------|-------|------|
| `ZEROCLAW_GATEWAY_PORT` | 网关端口 | `42617` | 本地 API 服务端口 |
| `PORT` | 替代端口变量 | 无 | 兼容 Heroku/Railway |
| `ZEROCLAW_GATEWAY_HOST` | 绑定地址 | `[::]` | IPv6 全地址 |
| `HOST` | 替代地址变量 | 无 | 兼容云平台 |
| `ZEROCLAW_ALLOW_PUBLIC_BIND` | 允许公网绑定 | `false` | 安全开关 |

```bash
# 开发：默认就好
# 不需要设置任何 Gateway 变量

# 生产：自定义端口 + 允许公网
export ZEROCLAW_GATEWAY_PORT=8080
export ZEROCLAW_ALLOW_PUBLIC_BIND=true
```

> **前端类比：** `PORT=3000 npm start` — 你改过 React dev server 的端口吧？`ZEROCLAW_GATEWAY_PORT` 完全一样。`HOST=0.0.0.0` 让 dev server 局域网可访问，也对应 `ZEROCLAW_GATEWAY_HOST` + `ZEROCLAW_ALLOW_PUBLIC_BIND`。

### 2.3 存储与数据库

控制 ZeroClaw 的持久化存储。

| 变量 | 用途 | 默认值 | 说明 |
|------|------|-------|------|
| `ZEROCLAW_STORAGE_PROVIDER` | 存储后端 | `sqlite` | 可选: `sqlite`, `postgres` |
| `ZEROCLAW_STORAGE_DB_URL` | 数据库连接 URL | 无 | PostgreSQL 连接串 |
| `ZEROCLAW_STORAGE_CONNECT_TIMEOUT_SECS` | 连接超时 | `5` | 秒数 |

```bash
# 本地开发：SQLite（零配置，开箱即用）
# 不需要设置任何存储变量

# 生产部署：PostgreSQL
export ZEROCLAW_STORAGE_PROVIDER="postgres"
export ZEROCLAW_STORAGE_DB_URL="postgresql://user:pass@db-host:5432/zeroclaw"
export ZEROCLAW_STORAGE_CONNECT_TIMEOUT_SECS=10
```

### 2.4 代理配置

在受限网络环境中让 ZeroClaw 正常访问外部 API。

| 变量 | 用途 | 默认值 | 说明 |
|------|------|-------|------|
| `ZEROCLAW_PROXY_ENABLED` | 启用代理 | `false` | 总开关 |
| `ZEROCLAW_HTTP_PROXY` | HTTP 代理 | 无 | `http://proxy:8080` |
| `ZEROCLAW_HTTPS_PROXY` | HTTPS 代理 | 无 | `http://proxy:8080` |
| `ZEROCLAW_ALL_PROXY` | SOCKS 代理 | 无 | `socks5://proxy:1080` |
| `ZEROCLAW_NO_PROXY` | 排除域名 | 无 | `localhost,127.0.0.1` |
| `ZEROCLAW_PROXY_SCOPE` | 代理范围 | `zeroclaw` | `zeroclaw` 或 `system` |
| `ZEROCLAW_PROXY_SERVICES` | 代理服务列表 | 无 | 指定走代理的服务 |

```bash
# 公司网络需要代理
export ZEROCLAW_PROXY_ENABLED=true
export ZEROCLAW_HTTPS_PROXY="http://corp-proxy.internal:8080"
export ZEROCLAW_NO_PROXY="localhost,127.0.0.1,.internal"
```

> **前端类比：** 公司里 `npm config set proxy http://proxy:8080` 设置 npm 代理——ZeroClaw 的代理配置逻辑完全一样，只是用 `ZEROCLAW_` 前缀的环境变量来设置。

### 2.5 Skills 与功能

控制 ZeroClaw 的扩展能力。

| 变量 | 用途 | 默认值 | 说明 |
|------|------|-------|------|
| `ZEROCLAW_OPEN_SKILLS_ENABLED` | 开放 Skills 加载 | `false` | 允许加载外部 Skills |
| `ZEROCLAW_OPEN_SKILLS_DIR` | Skills 目录 | 无 | 自定义 Skills 路径 |
| `ZEROCLAW_SKILLS_PROMPT_MODE` | Skills 提示模式 | `full` | `full` / `minimal` |
| `ZEROCLAW_REASONING_ENABLED` | 扩展思维 | `false` | 启用链式推理 |

```bash
# 启用自定义 Skills
export ZEROCLAW_OPEN_SKILLS_ENABLED=true
export ZEROCLAW_OPEN_SKILLS_DIR="$HOME/.zeroclaw/skills"
```

### 2.6 Web 搜索

控制 ZeroClaw 的联网搜索能力。

| 变量 | 用途 | 默认值 | 说明 |
|------|------|-------|------|
| `WEB_SEARCH_ENABLED` | 启用搜索 | `true` | 总开关 |
| `WEB_SEARCH_PROVIDER` | 搜索提供者 | `duckduckgo` | `duckduckgo`, `brave` |
| `WEB_SEARCH_MAX_RESULTS` | 最大结果数 | `5` | 每次搜索返回条数 |
| `BRAVE_API_KEY` | Brave 搜索 Key | 无 | Brave Search API |

```bash

export WEB_SEARCH_PROVIDER="brave"

export WEB_SEARCH_MAX_RESULTS=10
```

### 2.7 Provider API Keys

每个 LLM 提供者有自己专用的 API Key 变量：

```bash
# 按需设置你使用的 Provider 的 Key

MISTRAL_API_KEY=...                  # Mistral

OLLAMA_API_KEY=...                   # Ollama（本地模型，通常不需要 Key）
```

## 3. 命名规范解析

### 3.1 ZEROCLAW_ 前缀 + 功能域 + 具体项

ZeroClaw 环境变量遵循**三段式命名**：

```
ZEROCLAW_GATEWAY_PORT
│         │        │
│         │        └── 具体项（Port / Host / Enabled）
│         └── 功能域（GATEWAY / STORAGE / PROXY）
└── 命名空间前缀（区分其他工具的变量）
```

#### 功能域速查

| 功能域 | 前缀 | 控制范围 |
|-------|------|---------|
| 核心运行时 | `ZEROCLAW_` | API Key、Provider、模型 |
| Gateway | `ZEROCLAW_GATEWAY_` | HTTP 网关端口、地址 |
| 存储 | `ZEROCLAW_STORAGE_` | 数据库、持久化 |
| 代理 | `ZEROCLAW_PROXY_` / `ZEROCLAW_HTTP_PROXY` 等 | 网络代理 |
| Skills | `ZEROCLAW_OPEN_SKILLS_` / `ZEROCLAW_SKILLS_` | 扩展功能 |
| Bootstrap | `ZEROCLAW_BOOTSTRAP_` | 安装脚本 |
| 容器 | `ZEROCLAW_CONTAINER_` / `ZEROCLAW_DOCKER_` | 容器化 |

### 3.2 为什么需要 ZEROCLAW_ 前缀？

```bash
# ❌ 没有前缀 → 可能和其他工具冲突
export API_KEY="sk-..."        # 是给 ZeroClaw 的？还是给其他工具的？
export PORT=8080               # Docker? Node? ZeroClaw?

# ✅ 有前缀 → 一目了然
export ZEROCLAW_API_KEY="sk-..."
export ZEROCLAW_GATEWAY_PORT=8080
```

> **前端类比：** React 项目中自定义环境变量必须以 `REACT_APP_` 开头（`REACT_APP_API_URL`），Vue 必须以 `VITE_` 开头。原因一样——框架用前缀来识别"这是给我的"，避免和系统变量冲突。ZeroClaw 用 `ZEROCLAW_` 前缀，道理完全相同。

> **日常生活类比：** 这像快递包裹上写"XX公司 - 技术部 - 张三"。`ZEROCLAW` 是公司名，`GATEWAY` 是部门名，`PORT` 是收件人——三段式确保快递不会送错。

## 4. 回退机制详解（优先级链）

这是环境变量体系中**最重要的概念**——ZeroClaw 读取配置时不只看一个地方，而是按优先级链依次查找。

### 4.1 核心优先级链

```
最高优先级                                              最低优先级
    │                                                      │
    ▼                                                      ▼
ZEROCLAW_API_KEY  →  API_KEY  →  config.toml 中的 api_key  →  无（报错）
    │                   │              │
    │                   │              └── 配置文件中的值
    │                   └── 通用回退变量
    └── 专用环境变量（最高优先级）
```

### 4.2 具体示例：API Key 查找过程

用 Rust 伪代码解释 ZeroClaw 内部的查找逻辑：

```rust
// [来源: sourcecode/zeroclaw/src/config/mod.rs]
// 简化后的 API Key 解析逻辑

fn resolve_api_key() -> Option<String> {
    // 第1优先：ZEROCLAW_API_KEY 环境变量
    if let Ok(key) = std::env::var("ZEROCLAW_API_KEY") {
        return Some(key);  // ← 找到了，直接用
    }

    // 第2优先：通用 API_KEY 环境变量（回退）
    if let Ok(key) = std::env::var("API_KEY") {
        return Some(key);  // ← 找到了，用这个
    }

    // 第3优先：config.toml 中的 api_key 字段
    if let Some(key) = config_file.api_key {
        return Some(key);  // ← 找到了，用配置文件的
    }

    // 全部没找到 → 返回 None（后续会报错提示用户设置）
    None
}
```

### 4.3 Provider 和模型的回退链

同样的逻辑也适用于 Provider 和 Model：

```
Provider 优先级：
ZEROCLAW_PROVIDER → PROVIDER → config.toml.default_provider → "openrouter"（硬编码默认）

Model 优先级：
ZEROCLAW_MODEL → MODEL → config.toml.default_model → 无（由 Provider 决定）

Port 优先级：
ZEROCLAW_GATEWAY_PORT → PORT → config.toml.gateway.port → 42617（硬编码默认）
```

### 4.4 为什么要有回退变量？

| 变量 | 回退变量 | 为什么需要回退 |
|------|---------|-------------|
| `ZEROCLAW_API_KEY` | `API_KEY` | 兼容其他工具共享同一 Key |
| `ZEROCLAW_PROVIDER` | `PROVIDER` | 简写，减少输入 |
| `ZEROCLAW_GATEWAY_PORT` | `PORT` | 兼容 Heroku/Railway 等自动注入 `PORT` 的云平台 |
| `ZEROCLAW_GATEWAY_HOST` | `HOST` | 同上，兼容云平台 |

> **前端类比：** Next.js 读端口的优先级是 `PORT` 环境变量 → `next.config.js` 中的配置 → 默认 `3000`。ZeroClaw 完全一样，只是多了一层 `ZEROCLAW_` 前缀的专用变量。

> **日常生活类比：** 你联系一个人：先试手机 → 手机没通试微信 → 微信没回试邮件 → 都不行就去办公室找。每一步是"回退"，优先级从高到低。环境变量的回退机制就是这个道理。

## 5. 安全最佳实践

### 5.1 绝对禁止：硬编码 API Key

```bash
# ❌ 最危险的做法：写在代码里
# 一旦提交到 Git，全世界都能看到你的 Key
zeroclaw agent --api-key "sk-or-v1-real-key-here"

# ❌ 危险：写在 shell 历史记录中
export ZEROCLAW_API_KEY="sk-or-v1-real-key-here"  # 这条命令会被记录在 ~/.bash_history

# ✅ 安全：使用 .env 文件（记得加入 .gitignore）
# .env 文件（不提交到 Git）
echo 'ZEROCLAW_API_KEY=sk-or-v1-real-key-here' >> ~/.zeroclaw/.env

# ✅ 更安全：使用 shell 配置文件
echo 'export ZEROCLAW_API_KEY="sk-or-v1-real-key-here"' >> ~/.bashrc
```

### 5.2 .env 文件的正确用法

```bash
# 1. 创建 .env 文件
cat > ~/project/.env << 'EOF'
ZEROCLAW_API_KEY=sk-or-v1-your-key
ZEROCLAW_PROVIDER=openrouter
ZEROCLAW_MODEL=anthropic/claude-sonnet-4-6
ZEROCLAW_TEMPERATURE=0.5
EOF

# 2. 确保 .gitignore 包含 .env
echo ".env" >> ~/project/.gitignore

# 3. 加载（方式一：手动 source）
set -a && source ~/project/.env && set +a

# 3. 加载（方式二：在 shell 配置中自动加载）
# 在 ~/.bashrc 或 ~/.zshrc 中添加：
# [ -f ~/project/.env ] && set -a && source ~/project/.env && set +a
```

### 5.3 各环境的安全配置

```
本地开发                    CI/CD                      Docker 部署
┌──────────────┐     ┌──────────────────┐     ┌──────────────────────┐
│ ~/.bashrc    │     │ GitHub Secrets   │     │ docker run           │
│ 或 .env 文件  │     │ 或 CI 变量面板     │     │   -e ZEROCLAW_API_KEY│
│              │     │                  │     │   =sk-...            │
│ 不提交 Git    │     │ 平台加密存储      │     │   或 --env-file .env │
└──────────────┘     └──────────────────┘     └──────────────────────┘
```

> **前端类比：** 你一定写过 `.env.local`（不提交 Git）和 `.env.production`（CI/CD 中注入）。ZeroClaw 的安全策略完全一样——开发用文件、生产用注入、密钥永远不提交。

## 6. 常见配置场景

### 场景 1：本地开发（最简配置）

```bash
# ~/.bashrc 或 ~/.zshrc 中添加
export ZEROCLAW_API_KEY="sk-or-v1-your-key"
export ZEROCLAW_PROVIDER="openrouter"

# 可选：指定默认模型
export ZEROCLAW_MODEL="anthropic/claude-sonnet-4-6"

# 重新加载
source ~/.bashrc
```

验证：

```bash
# 检查变量是否生效
echo $ZEROCLAW_API_KEY        # 应该输出你的 key（部分）
echo $ZEROCLAW_PROVIDER       # 应该输出 openrouter

# 测试 ZeroClaw
zeroclaw agent --message "Hello!"
```

### 场景 2：Docker 部署

```bash
# 方式 A：命令行传入
docker run -it \
  -e ZEROCLAW_API_KEY="sk-or-v1-your-key" \
  -e ZEROCLAW_PROVIDER="openrouter" \
  -e ZEROCLAW_GATEWAY_PORT=8080 \
  -e ZEROCLAW_ALLOW_PUBLIC_BIND=true \
  -p 8080:8080 \
  zeroclaw-bootstrap:local

# 方式 B：env-file 传入（推荐，更整洁）
docker run -it \
  --env-file .env.production \
  -p 8080:8080 \
  zeroclaw-bootstrap:local
```

Docker Compose 方式：

```yaml
# docker-compose.yml
services:
  zeroclaw:
    build: { context: ., target: release }
    env_file: .env.production
    environment:
      # 可以覆盖 env_file 中的值
      - ZEROCLAW_GATEWAY_PORT=8080
      - ZEROCLAW_ALLOW_PUBLIC_BIND=true
      - ZEROCLAW_STORAGE_PROVIDER=postgres
      - ZEROCLAW_STORAGE_DB_URL=postgresql://user:pass@db:5432/zeroclaw
    ports: ["8080:8080"]
    volumes: [zeroclaw-data:/root/.zeroclaw]
    restart: unless-stopped
    depends_on: [db]

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: zeroclaw
    volumes: [pg-data:/var/lib/postgresql/data]

volumes:
  zeroclaw-data:
  pg-data:
```

### 场景 3：CI/CD（GitHub Actions）

```yaml
# .github/workflows/test.yml
name: ZeroClaw Integration Test
on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install ZeroClaw
        run: |
          curl -fsSL https://zeroclawlabs.ai/install.sh | bash

      - name: Run Agent Test
        env:
          # 从 GitHub Secrets 注入（绝不硬编码）
          ZEROCLAW_API_KEY: ${{ secrets.ZEROCLAW_API_KEY }}
          ZEROCLAW_PROVIDER: openrouter
          ZEROCLAW_MODEL: anthropic/claude-sonnet-4-6
          ZEROCLAW_TEMPERATURE: "0"  # CI 中用 0 温度保证确定性输出
        run: |
          zeroclaw agent --message "Run tests"
```

```bash
# Ollama 默认在 localhost:11434，无需设置代理

# 禁用联网搜索
export WEB_SEARCH_ENABLED=false
```

## 9. 在 ZeroClaw 开发中的应用

### 9.1 不同开发阶段的环境变量策略

| 阶段 | 关键变量 | 策略 |
|------|---------|------|
| **初次体验** | `ZEROCLAW_API_KEY` + `ZEROCLAW_PROVIDER` | 最简配置，`zeroclaw onboard` 自动生成 |
| **日常开发** | + `ZEROCLAW_MODEL` + `ZEROCLAW_TEMPERATURE` | 写入 `~/.bashrc` 持久化 |
| **功能测试** | + `ZEROCLAW_REASONING_ENABLED` 等 | 按需开关功能 |
| **Docker 部署** | 全部通过 `--env-file` 注入 | 不改镜像，只改变量 |
| **CI/CD** | 通过 Secrets 注入 + `TEMPERATURE=0` | 确定性输出，密钥安全 |
| **源码贡献** | + `ZEROCLAW_CONFIG_DIR`（隔离测试配置） | 避免污染日常配置 |

### 9.2 环境变量与配置文件的分工

```
环境变量擅长的事：                    配置文件（config.toml）擅长的事：
├── 敏感信息（API Key）              ├── 复杂结构化配置
├── 跨环境变化的值（端口、代理）       ├── 不常变化的默认值
├── CI/CD 注入                      ├── 团队共享的配置
├── Docker 部署传参                  ├── 多层嵌套配置（autonomy / security）
└── 临时覆盖测试                     └── 需要注释说明的配置
```

> **核心原则：** 环境变量用于"外部注入 + 临时覆盖"，配置文件用于"稳定基线 + 复杂结构"。两者配合，覆盖所有场景。

## 10. 常见问题速查

| 问题 | 解决方案 |
|------|---------|
| 设置了环境变量但不生效 | 确认 `export` 了（不是只 `VAR=val`），新终端需要重新加载 |
| 环境变量和 config.toml 冲突 | 环境变量优先级更高，会覆盖 config.toml |
| 不知道当前环境变量值 | `env \| grep ZEROCLAW` 查看所有相关变量 |
| Docker 中环境变量不生效 | 确认 `-e` 或 `--env-file` 参数正确 |
| CI 中 API Key 泄露 | 使用平台 Secrets，不要 `echo` 打印 Key |
| 代理配置了但还是超时 | 检查 `ZEROCLAW_PROXY_ENABLED=true` 是否设置 |
| 想临时用不同模型 | `ZEROCLAW_MODEL="gpt-4" zeroclaw agent ...`（单次覆盖） |

### 单次覆盖技巧

```bash
# 只对这一次命令生效，不影响后续
ZEROCLAW_MODEL="gpt-4o" ZEROCLAW_TEMPERATURE=0 zeroclaw agent --message "Be precise"

# 等价于前端的：
# PORT=4000 npm start    ← 只对这次启动生效
```
