## 卡片1：什么是多状态管理？

**一句话**：多状态管理就是为图的不同层级和不同阶段划定独立的状态边界，控制"谁能看到什么、谁能改什么"。

**类比**：就像一家公司的权限管理系统。公司有一个巨大的数据库（完整状态），但前台只能看访客登记（input_schema），客户只能拿到最终报告（output_schema），各部门有自己的内部文档（节点级状态分组），子公司有独立的账本（子图状态隔离）。

**代码直觉**：

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph

class InputState(TypedDict):
    question: str                    # 外部只需传这一个

class OutputState(TypedDict):
    answer: str                      # 外部只能拿到这一个

class OverallState(InputState, OutputState):
    raw_chunks: list[str]            # 内部中间状态，对外不可见
    rerank_scores: list[float]

graph = StateGraph(
    OverallState,
    input_schema=InputState,         # 入口白名单
    output_schema=OutputState,       # 出口白名单
)
```

**核心原则**：不是所有节点都需要看到所有状态，不是所有状态都需要暴露给外部。多状态管理就是把这些边界定义清楚。

**2 分钟记住**：多状态管理 = 图的权限管理系统，控制数据的可见性和可修改性。

## 卡片2：StateGraph 的四个 Schema 参数

**一句话**：StateGraph 构造函数接受四个 Schema 参数，分别控制内部状态、输入接口、输出接口和不可变上下文。

**四个参数**：

```python
graph = StateGraph(
    state_schema=OverallState,      # 内部完整状态（所有字段的超集）
    input_schema=InputState,        # 外部输入接口（入口白名单）
    output_schema=OutputState,      # 外部输出接口（出口白名单）
    context_schema=ContextSchema,   # 运行时不可变上下文（不创建 Channel）
)
```

**源码签名**：

```python
# libs/langgraph/langgraph/graph/state.py
class StateGraph:
    def __init__(
        self,
        state_schema: type[StateT],
        context_schema: type[ContextT] | None = None,
        *,
        input_schema: type[InputT] | None = None,   # 默认 = state_schema
        output_schema: type[OutputT] | None = None,  # 默认 = state_schema
    ) -> None: ...
```

**记忆口诀**：state 管内部，input 管入口，output 管出口，context 管配置。

**2 分钟记住**：四个 Schema = 四道门，每道门控制不同方向的数据流动。

## 卡片3：Input/Output Schema 的默认行为

**一句话**：不设置 input_schema 和 output_schema 时，它们默认等于 state_schema，意味着没有任何边界控制。

**举例**：

```python
# 不设置 input/output schema
graph = StateGraph(OverallState)
# 等价于：
graph = StateGraph(
    OverallState,
    input_schema=OverallState,     # 外部可以传入所有字段
    output_schema=OverallState,    # 外部可以拿到所有字段（包括中间状态）
)

# 设置后：
graph = StateGraph(OverallState, input_schema=InputState, output_schema=OutputState)
# 外部只能传 InputState 的字段，只能拿 OutputState 的字段
# 中间状态（raw_chunks、rerank_scores）对外完全不可见
```

**什么时候需要设置？**

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 简单原型 | 不需要 | 快速迭代，不用管边界 |
| 对外暴露 API | 需要 | 隐藏内部实现细节 |
| 多人协作 | 需要 | 明确接口契约 |
| 子图复用 | 需要 | 减少耦合 |

**2 分钟记住**：默认不设置 = 全部暴露。生产环境建议显式设置，隐藏内部细节。

## 卡片4：节点如何自动推断 input_schema

**一句话**：LangGraph 通过检查节点函数第一个参数的类型注解，自动推断该节点的 input_schema。

**举例**：

```python
class OverallState(TypedDict):
    foo: str
    user_input: str
    graph_output: str

class PrivateState(TypedDict):
    bar: str

# node_1 的第一个参数是 InputState → 自动推断 input_schema = InputState
def node_1(state: InputState) -> OverallState:
    return {"foo": state["user_input"] + " name"}

# node_3 的第一个参数是 PrivateState → 自动推断 input_schema = PrivateState
def node_3(state: PrivateState) -> OutputState:
    return {"graph_output": state["bar"] + " Lance"}
```

**源码机制**：

```python
# add_node() 内部逻辑（简化）
def add_node(self, node, action=None, *, input_schema=None, ...):
    if input_schema is not None:
        # 显式指定，直接使用
        ...
    elif inferred_input_schema is not None:
        # 从函数第一个参数的类型提示推断
        ...
```

**注意**：推断出的 Schema 中的字段必须在图的 Channel 池中存在（即在某个已注册的 Schema 中定义过），否则会报错。

**2 分钟记住**：函数参数类型注解 = 节点的"我只需要这些字段"声明，框架自动识别。

## 卡片5：子图的两种添加方式

**一句话**：子图可以作为编译图直接添加（共享键自动传递），也可以用函数包装后添加（手动状态转换）。

**方式1：直接添加编译图**

```python
subgraph = subgraph_builder.compile()

# 直接传入编译后的子图
parent_builder.add_node("sub", subgraph)
# 同名键自动在父子图之间传递
```

**方式2：函数包装**

```python
subgraph = subgraph_builder.compile()

def call_subgraph(state: ParentState):
    # 手动转换状态
    response = subgraph.invoke({"bar": state["query"]})
    return {"result": response["bar"]}

parent_builder.add_node("sub", call_subgraph)
```

**怎么选？**

| 条件 | 选择 | 原因 |
|------|------|------|
| 父子图有共同字段 | 直接添加 | 自动传递，代码更简洁 |
| Schema 完全不同 | 函数包装 | 需要手动做字段映射 |
| 需要预处理/后处理 | 函数包装 | 可以在调用前后加逻辑 |

**2 分钟记住**：有共同字段用直接添加，Schema 不同用函数包装。

## 卡片6：共享键 vs 私有键

**一句话**：父子图中同名的键是共享键（自动传递），只在子图中定义的键是私有键（父图看不到）。

**举例**：

```python
class ParentState(TypedDict):
    foo: str          # 父图有 foo

class SubgraphState(TypedDict):
    foo: str          # 子图也有 foo → 共享键，自动传递
    bar: str          # 只有子图有 bar → 私有键，父图看不到

# 执行流程：
# 1. 父图的 foo="hello" 自动传入子图
# 2. 子图内部处理，可以读写 foo 和 bar
# 3. 子图结束，foo 的最终值自动传回父图
# 4. bar 留在子图内部，父图完全不知道它的存在
```

**RAG 场景应用**：

```python
class ParentState(TypedDict):
    query: str        # 共享：父子图都需要
    answer: str       # 共享：子图生成，父图使用

class RetrievalSubgraphState(TypedDict):
    query: str        # 共享键
    answer: str       # 共享键
    chunks: list      # 私有键：检索中间结果
    scores: list      # 私有键：排序分数
```

**2 分钟记住**：同名 = 共享（自动传递），独有 = 私有（互不可见）。

## 卡片7：命名空间格式解析

**一句话**：LangGraph 用命名空间为每个子图分配独立的 checkpoint 空间，格式是 `{node}:{task_id}|{child}:{child_task_id}`。

**源码常量**：

```python
# libs/langgraph/langgraph/_internal/_constants.py
NS_SEP = "|"   # 分隔层级
NS_END = ":"   # 分隔命名空间和 task_id
```

**命名空间示例**：

```
""                                    # 根图
"agent:abc123"                        # 一级子图（agent 节点，task_id=abc123）
"agent:abc123|researcher:def456"      # 二级子图（agent 下的 researcher 节点）
"agent:abc123|researcher:def456|tool:ghi789"  # 三级子图
```

**为什么需要命名空间？** 每个子图需要独立的 checkpoint 来保存自己的状态快照。命名空间确保父图和子图的 checkpoint 互不干扰，即使它们共享同一个 checkpointer 实例。

**类比**：就像文件系统的目录结构——`/root/agent/researcher/tool`，每个目录有自己的文件，互不干扰。

**2 分钟记住**：`|` 分层级，`:` 分 task_id。命名空间 = 子图的 checkpoint 地址。

## 卡片8：context_schema vs state_schema

**一句话**：`state_schema` 定义可变的图状态（创建 Channel），`context_schema` 定义不可变的运行时上下文（不创建 Channel）。

**对比**：

| 维度 | state_schema | context_schema |
|------|-------------|----------------|
| 是否创建 Channel | 是 | 否 |
| 是否可变 | 节点可以更新 | 不可变，只读 |
| 访问方式 | `state["key"]` | `runtime.context["key"]` |
| 典型用途 | 工作流数据 | API keys、配置参数 |
| 是否持久化 | 是（checkpoint） | 否 |

**代码示例**：

```python
from langgraph.runtime import Runtime

class ContextSchema(TypedDict):
    api_key: str
    model_name: str

class State(TypedDict):
    query: str
    answer: str

def generate_node(state: State, runtime: Runtime[ContextSchema]):
    model = runtime.context["model_name"]    # 读取不可变上下文
    key = runtime.context["api_key"]
    answer = call_llm(state["query"], model=model, api_key=key)
    return {"answer": answer}                # 更新可变状态

graph = StateGraph(State, context_schema=ContextSchema)
# ...
compiled.invoke({"query": "什么是 RAG？"}, context={"api_key": "sk-...", "model_name": "gpt-4"})
```

**2 分钟记住**：state 是可变的工作数据，context 是不可变的运行配置。一个走 Channel，一个走 Runtime。

## 卡片9：Channel 池的合并机制

**一句话**：`_add_schema()` 将多个 Schema 的字段合并到统一的 Channel 池，Schema 只是 Channel 的过滤视图。

**合并规则**：

```
OverallState: {query: LastValue, answer: LastValue, chunks: BinaryOp(add)}
InputState:   {query: LastValue}
OutputState:  {answer: LastValue}

合并后的 Channel 池:
self.channels = {
    "query":  LastValue,        # 来自 OverallState + InputState
    "answer": LastValue,        # 来自 OverallState + OutputState
    "chunks": BinaryOp(add),   # 来自 OverallState
}

self.schemas = {
    OverallState: {"query", "answer", "chunks"},  # 看到全部
    InputState:   {"query"},                       # 只看到 query
    OutputState:  {"answer"},                      # 只看到 answer
}
```

**兼容性规则**：
- 同名字段的 Channel 类型必须兼容
- LastValue 与任何 Channel 兼容（会被更具体的 Channel 覆盖）
- 两个不同 Reducer 的同名字段会抛出错误

```python
# ✅ 兼容：LastValue + BinaryOp → 使用 BinaryOp
class A(TypedDict):
    items: list              # LastValue

class B(TypedDict):
    items: Annotated[list, operator.add]  # BinaryOp(add)

# ❌ 不兼容：两个不同的 Reducer
class C(TypedDict):
    items: Annotated[list, operator.add]     # add

class D(TypedDict):
    items: Annotated[list, custom_reducer]   # custom_reducer ≠ add → 报错
```

**2 分钟记住**：多个 Schema 合并成一个 Channel 池，Schema 只是"看哪些 Channel"的过滤器。

## 卡片10：多状态管理的设计原则

**一句话**：好的多状态管理遵循最小暴露、单一职责、显式通信三个原则。

**三条核心原则**：

```
原则1：最小暴露
  → input_schema 只包含调用者需要传的字段
  → output_schema 只包含调用者需要拿的字段
  → 内部中间状态不要暴露给外部

原则2：单一职责
  → 每个子图管理自己的状态空间
  → 不要让子图依赖父图的内部字段
  → 共享键越少越好

原则3：显式通信
  → 父子图之间的数据传递要明确可见
  → 优先用共享键（简单场景）或函数包装（复杂场景）
  → Command.PARENT 谨慎使用，它是"越级汇报"
```

**检查清单——设计多状态图时问自己**：

```
□ 是否设置了 input_schema 和 output_schema？
□ 内部中间状态是否对外不可见？
□ 每个节点是否只接收自己需要的字段？
□ 子图的共享键是否最少化？
□ 父子图的通信方式是否明确？
□ context_schema 是否用于不可变配置？
```

**RAG 系统的典型状态分层**：

```python
# 外部接口层
class InputState(TypedDict):
    question: str

class OutputState(TypedDict):
    answer: str
    sources: list[str]

# 内部工作层
class OverallState(InputState, OutputState):
    cleaned_query: str
    raw_chunks: list[str]
    rerank_scores: list[float]
    prompt: str

# 子图私有层
class RetrievalState(TypedDict):
    question: str          # 共享键
    raw_chunks: list[str]  # 共享键
    embedding_cache: dict  # 私有键
```

**2 分钟记住**：最小暴露、单一职责、显式通信——三条原则让你的多状态图清晰、可维护、可复用。
