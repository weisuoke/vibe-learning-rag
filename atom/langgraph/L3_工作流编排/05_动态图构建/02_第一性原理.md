# 动态图构建 - 第一性原理

### 什么是第一性原理？

**第一性原理**：回到事物最基本的真理，从源头思考问题

不是"别人怎么做我就怎么做"，而是问自己：**最底层的事实是什么？从这些事实能推导出什么？**

### 动态图构建的第一性原理

#### 1. 最基础的定义

**动态图构建 = 让工作流在运行时根据数据自主决定执行路径和并行任务数量**

仅此而已！没有更基础的了。

把这句话拆开：

- **运行时**：不是写代码时决定，而是程序跑起来后才决定
- **根据数据**：决策依据是当前的状态数据，不是硬编码的规则
- **自主决定**：图自己选择走哪条路，不需要外部干预
- **执行路径**：下一步去哪个节点
- **并行任务数量**：同时启动多少个任务

用一个公式表达：

```
静态图：编译时确定所有路径 → 运行时按固定路线走
动态图：编译时定义可能的路径 → 运行时根据数据选择实际路径
```

#### 2. 为什么需要动态图构建？

**核心问题：真实世界的工作流充满不确定性，静态图无法应对**

想象你在开发一个 AI 客服系统：

```
用户问题 → ???
```

这个 `???` 在写代码时你不知道是什么。用户可能问产品问题、投诉、退款、技术支持……每种情况需要不同的处理流程。

**静态图的三大局限：**

##### 局限1：路径在编译时固定

```python
# 静态图：写死了 A → B → C
graph.add_edge("A", "B")
graph.add_edge("B", "C")
# 如果运行时发现应该跳过 B 直接到 C？做不到！
```

这就像导航软件只有一条固定路线——不管前方堵不堵车，都走同一条路。

##### 局限2：并行任务数量固定

```python
# 静态图：写死了 3 个并行任务
graph.add_edge(START, "task_1")
graph.add_edge(START, "task_2")
graph.add_edge(START, "task_3")
# 如果运行时需要处理 10 个任务？或者只需要 1 个？做不到！
```

这就像餐厅只有 3 个厨师的工位——不管今天来了多少订单，都只能同时做 3 道菜。

##### 局限3：节点无法自主控制流向

```python
# 静态图：节点只能返回数据，不能决定下一步去哪
def my_node(state):
    return {"result": "done"}  # 只能更新状态
    # 想说"接下来去 node_X"？做不到！
```

这就像员工只能完成任务交报告，不能自己决定下一步该找谁——所有流程都要提前写在规章制度里。

**动态图解决了这三个问题：**

| 问题 | 静态图 | 动态图 |
|------|--------|--------|
| 路径选择 | 编译时固定 | 运行时根据状态决定 |
| 并行数量 | 编译时固定 | 运行时动态创建 |
| 节点自主性 | 只能返回数据 | 可以同时更新状态 + 控制路由 |

#### 3. 动态图构建的三层价值

##### 价值1：运行时灵活性

**条件边（Conditional Edges）让图在运行时选择路径。**

```python
from langgraph.graph import StateGraph, END

# 路由函数：运行时根据状态决定去哪
def route_by_intent(state):
    """根据用户意图路由到不同处理节点"""
    if state["intent"] == "question":
        return "answer_node"
    elif state["intent"] == "complaint":
        return "complaint_node"
    else:
        return "default_node"

# 添加条件边：编译时定义可能的路径，运行时选择实际路径
graph.add_conditional_edges("classifier", route_by_intent)
```

[来源: sourcecode/langgraph/graph/state.py - add_conditional_edges 方法]

**前端类比：** 就像 React Router 的动态路由——URL 是运行时才知道的，路由器根据 URL 决定渲染哪个组件。

```javascript
// 前端动态路由（类比）
<Route path="/user/:id" component={UserPage} />
// :id 是运行时才知道的，路由器动态匹配
```

**日常生活类比：** 就像医院的分诊台——护士根据你的症状（运行时数据）决定你去哪个科室（执行路径），而不是所有人都走同一条路。

##### 价值2：数据驱动的工作流

**Send API 让并行任务数量由数据决定。**

```python
from langgraph.types import Send

def fan_out_to_workers(state):
    """根据数据动态创建并行任务"""
    # 任务数量在运行时才知道！
    subjects = state["subjects"]  # 可能是 2 个，也可能是 20 个
    return [Send("process_subject", {"subject": s}) for s in subjects]

graph.add_conditional_edges("splitter", fan_out_to_workers)
```

[来源: sourcecode/langgraph/types.py - Send 类（行 289-362）]

**前端类比：** 就像 `Promise.all()` 处理动态数量的 API 请求——你不知道要发多少个请求，取决于用户选了多少项。

```javascript
// 前端动态并行（类比）
const urls = userSelectedItems.map(item => `/api/item/${item.id}`);
const results = await Promise.all(urls.map(url => fetch(url)));
// 并行请求数量由用户选择决定
```

**日常生活类比：** 就像快递分拣中心——今天来了 100 个包裹就开 100 条分拣线，明天来了 10 个就开 10 条。工人数量跟着包裹数量走，不是提前固定的。

##### 价值3：可组合的复杂系统

**Command API 让节点同时更新状态和控制流向。**

```python
from langgraph.types import Command
from typing import Literal

def smart_node(state) -> Command[Literal["next_a", "next_b"]]:
    """节点自主决定：更新什么数据 + 下一步去哪"""
    if state["score"] > 0.8:
        return Command(
            update={"quality": "high"},  # 更新状态
            goto="next_a"                # 控制路由
        )
    else:
        return Command(
            update={"quality": "low"},
            goto="next_b"
        )
```

[来源: sourcecode/langgraph/types.py - Command 类（行 367-418）]

**前端类比：** 就像 Redux 的 action 同时携带数据更新和副作用——一个 dispatch 既改了 store，又触发了页面跳转。

```javascript
// 前端类比：action 同时更新状态和导航
dispatch({ type: 'SUBMIT_FORM', payload: data });
// reducer 更新 store + middleware 触发路由跳转
```

**日常生活类比：** 就像项目经理开完会后同时做两件事：更新项目进度表（状态更新）+ 通知下一个团队开始工作（路由控制）。不需要分两步，一个动作搞定。

#### 4. 从第一性原理推导 LangGraph 动态图

**推理链：**

```
1. 工作流需要根据数据做决策
   → 需要条件边（add_conditional_edges）
   ↓
2. 条件边需要一个函数来做决策
   → 需要路由函数（path function）
   ↓
3. 并行任务数量运行时才知道
   → 需要 Send API（动态创建并行节点调用）
   ↓
4. 节点需要同时更新状态和控制流向
   → 需要 Command API（update + goto 合一）
   ↓
5. AI 决策需要结构化输出
   → 需要 LLM 驱动路由（structured output + 条件边）
   ↓
6. 所有这些需要统一的构建方式
   → 需要 StateGraph builder（两阶段：构建 → 编译）
   ↓
7. 编译后需要高效执行
   → 需要 Pregel 执行引擎（BSP 模型：计划 → 执行 → 更新）
```

#### 5. 动态图的执行模型

理解了"为什么需要"，还要理解"怎么执行"。

LangGraph 使用 **BSP（批量同步并行）** 执行模型：

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│  计划阶段  │ →  │  执行阶段  │ →  │  更新阶段  │
│  Plan     │    │  Execute  │    │  Update   │
└──────────┘    └──────────┘    └──────────┘
     │                │                │
     │                │                │
  准备下一步        并行执行          应用写入
  任务列表         所有就绪节点       到状态通道
```

[来源: sourcecode/langgraph/pregel/_algo.py - BSP 执行模型]

每一步（superstep）都经历这三个阶段：

1. **计划**：`prepare_next_tasks()` 检查哪些节点被触发（包括 Send 创建的动态任务）
2. **执行**：并行运行所有就绪的节点
3. **更新**：`apply_writes()` 将节点输出写入状态通道

**这就是为什么动态图能工作的底层原因**——Send 对象在计划阶段被处理为 PUSH 任务，条件边在计划阶段决定 PULL 任务，Command 在执行阶段同时产生状态更新和路由指令。

#### 6. 动态图 vs 静态图：本质区别

从第一性原理看，静态图和动态图的区别不在于"功能多少"，而在于**决策时机**：

```python
# ===== 静态图：所有决策在编译时完成 =====
graph = StateGraph(State)
graph.add_node("A", node_a)
graph.add_node("B", node_b)
graph.add_edge("A", "B")        # 编译时就确定：A 之后一定是 B
app = graph.compile()            # 编译后路径固定

# ===== 动态图：决策推迟到运行时 =====
graph = StateGraph(State)
graph.add_node("A", node_a)
graph.add_node("B", node_b)
graph.add_node("C", node_c)
graph.add_conditional_edges(     # 编译时定义可能的路径
    "A",
    route_function,              # 运行时才执行这个函数
    {"go_b": "B", "go_c": "C"}  # 运行时才选择实际路径
)
app = graph.compile()            # 编译后路径仍然灵活
```

| 维度 | 静态图 | 动态图 |
|------|--------|--------|
| 决策时机 | 编译时 | 运行时 |
| 路径确定性 | 100% 确定 | 取决于数据 |
| 并行任务数 | 固定 | 可变 |
| 节点自主性 | 无（只返回数据） | 有（Command 控制流向） |
| 适用场景 | 固定流程 | 智能决策、AI Agent |
| 类比 | 流水线 | 智能调度系统 |

#### 7. 三种动态机制的协作关系

三种动态机制不是互相替代的，而是解决不同层面的问题：

```
┌─────────────────────────────────────────────────┐
│                 动态图构建                        │
│                                                  │
│  ┌──────────────┐  解决"去哪"的问题               │
│  │  条件边       │  → 基于状态选择路径              │
│  │  + 路由函数   │  → 最基础的动态能力              │
│  └──────────────┘                                │
│                                                  │
│  ┌──────────────┐  解决"去多少"的问题              │
│  │  Send API    │  → 动态创建并行任务              │
│  │              │  → Map-Reduce 模式              │
│  └──────────────┘                                │
│                                                  │
│  ┌──────────────┐  解决"去哪 + 带什么"的问题       │
│  │  Command API │  → 状态更新 + 路由控制合一       │
│  │              │  → 支持跨图导航                  │
│  └──────────────┘                                │
└─────────────────────────────────────────────────┘
```

[来源: sourcecode/langgraph - 三大核心机制综合分析]

它们可以组合使用：

```python
from langgraph.types import Send, Command
from typing import Literal

# 条件边 + Send：动态数量的并行任务
def dynamic_fan_out(state):
    """条件边的路由函数返回 Send 列表"""
    return [Send("worker", {"task": t}) for t in state["tasks"]]

# Command：节点内部同时更新状态和路由
def smart_worker(state) -> Command[Literal["aggregator", "retry"]]:
    result = process(state["task"])
    if result["success"]:
        return Command(update={"results": [result]}, goto="aggregator")
    else:
        return Command(update={"errors": [result]}, goto="retry")
```

#### 8. 一句话总结第一性原理

**动态图构建的本质是将工作流的决策权从编译时转移到运行时，通过条件边选择路径、Send 动态创建并行任务、Command 统一状态更新与路由控制，让 AI 工作流能像人类一样根据实际情况灵活应对。**
