## 知识点拆解

### 核心概念识别

从源码分析、官方文档和社区讨论中，识别出以下核心概念：

1. **普通边（Normal Edge）**
   - 定义：通过 `add_edge()` 添加的固定路由边
   - 特征：直接连接两个节点，确定性流程
   - 实现：存储在 `self.edges` 集合中
   - 应用：简单的线性流程

2. **条件边（Conditional Edge）**
   - 定义：通过 `add_conditional_edges()` 添加的动态路由边
   - 特征：根据路由函数返回值决定下一个节点
   - 实现：存储在 `self.branches[source][name]` 中，使用 `BranchSpec` 类
   - 应用：需要根据状态进行决策的场景

3. **等待边（Waiting Edge）**
   - 定义：通过 `add_edge([node1, node2], target)` 添加的同步边
   - 特征：等待多个节点全部完成后才执行目标节点
   - 实现：使用 `NamedBarrierValue` 通道同步
   - 应用：需要等待多个并行任务完成的场景

4. **序列边（Sequence Edge）**
   - 定义：通过 `add_sequence()` 添加的语法糖
   - 特征：自动添加节点和边，简化线性流程定义
   - 实现：内部调用 `add_node()` 和 `add_edge()`
   - 应用：简化顺序节点的添加

5. **动态路由（Send）**
   - 定义：条件边返回 `Send` 对象实现动态并行执行
   - 特征：可以向目标节点发送自定义状态
   - 实现：`Send(node, state)` 对象
   - 应用：动态并行执行（fan-out）

### 边的选择策略

从社区讨论和最佳实践中，总结出以下选择策略：

1. **简单流程** → 普通边
2. **条件分支** → 条件边
3. **并行等待** → 等待边
4. **线性流程** → 序列边
5. **动态并行** → 条件边 + Send

### 常见误区

从 GitHub issues 和社区讨论中，识别出以下常见误区：

1. **误区1**：认为条件边和普通边可以随意混用
   - **正确理解**：混用可能导致节点重复执行

2. **误区2**：认为 Command 可以覆盖所有预定义边
   - **正确理解**：Command 无法覆盖通过 `add_edge` 显式定义的边

3. **误区3**：认为条件边的路由函数总是接收完整状态
   - **正确理解**：路由函数接收的状态类型取决于 input_schema

4. **误区4**：认为 defer 节点可以与任何边类型组合
   - **正确理解**：defer 节点与条件边组合时需要特别注意

5. **误区5**：认为条件边的可视化会自动正确
   - **正确理解**：需要类型注解或 path_map 才能正确可视化
