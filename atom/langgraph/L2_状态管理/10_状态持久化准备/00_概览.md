# 状态持久化准备（State Persistence Preparation）- 知识点概览

> **图的状态要跨进程、跨重启存活，就必须在设计阶段考虑：状态能不能序列化？状态会不会太大？敏感数据怎么保护？这就是持久化准备。**

---

## 知识点简介

**状态持久化准备是在设计 LangGraph 状态时，为后续的 Checkpoint 持久化做好三方面准备：可序列化设计、状态大小控制、敏感数据保护。**

当你给 StateGraph 配上 Checkpointer 后，LangGraph 会在每个节点执行后自动保存状态快照（Checkpoint）。这意味着你的状态数据要经历一条完整的管道：

```
Channel.checkpoint() → Serializer.dumps_typed() → [加密] → Saver.put()
```

如果你的状态里有不可序列化的对象（比如数据库连接、文件句柄），序列化就会失败。如果状态太大（比如累积了几百条消息），存储和恢复都会变慢。如果状态里有 API 密钥等敏感信息，明文存储就是安全隐患。

**状态持久化准备就是在"写状态定义"的阶段，提前解决这三个问题。**

### 场景 1 预览：可序列化状态设计

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from operator import add

# ✅ 持久化友好的状态设计
class RAGState(TypedDict):
    query: str                              # 基础类型 → 直接序列化
    docs: Annotated[list[str], add]         # 列表 + Reducer → 可序列化
    answer: str                             # 基础类型 → 直接序列化
    metadata: dict[str, str]                # 字典 → 可序列化

# ❌ 持久化不友好的状态设计
# class BadState(TypedDict):
#     query: str
#     db_conn: Any           # 数据库连接 → 不可序列化！
#     callback: Callable     # 函数对象 → 不可序列化！
#     file_handle: IO        # 文件句柄 → 不可序列化！

def retrieve_node(state: RAGState) -> dict:
    return {"docs": [f"关于 '{state['query']}' 的文档片段"]}

def generate_node(state: RAGState) -> dict:
    context = "\n".join(state["docs"])
    return {"answer": f"基于 {context} 的回答"}

# 构建图 + 配置 Checkpointer
builder = StateGraph(RAGState)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 执行：每个节点后自动保存 Checkpoint
config = {"configurable": {"thread_id": "rag-001"}}
result = graph.invoke({"query": "什么是向量数据库？", "docs": [], "metadata": {}}, config)
print(result["answer"])
```

### 核心数据流（持久化管道）

```
保存方向（每个节点执行后）：
Channel.checkpoint()              → 产生可序列化的状态快照
    ↓
SerializerProtocol.dumps_typed()  → 序列化为 (type_tag, bytes)
    ↓
[可选] CipherProtocol.encrypt()   → 加密字节数据
    ↓
CheckpointSaver.put()             → 存储到后端（内存/SQLite/Postgres）

恢复方向（中断后恢复时）：
CheckpointSaver.get_tuple()       → 从存储加载
    ↓
[可选] CipherProtocol.decrypt()   → 解密字节数据
    ↓
SerializerProtocol.loads_typed()  → 反序列化为 Python 对象
    ↓
Channel.from_checkpoint()         → 恢复通道状态
```

## 常见问题

### Q1：什么时候需要关心持久化准备？

当你给 StateGraph 配了 Checkpointer 的时候。如果你只用 `graph = builder.compile()`（不传 checkpointer），状态只在内存中存在，不需要序列化。但一旦你用了 `compile(checkpointer=InMemorySaver())` 或任何其他 Checkpointer，状态就必须能被序列化。

### Q2：TypedDict 和 Pydantic 哪个更适合持久化？

两者都可以。TypedDict 更轻量，Pydantic 提供运行时验证。对于持久化来说，关键不是用哪个定义状态，而是状态字段的值是否可序列化。一个 TypedDict 状态如果字段值是数据库连接，一样会失败。

### Q3：pickle_fallback 应该开启吗？

默认关闭，建议保持关闭。pickle 反序列化存在安全风险（可以执行任意代码），而且 pickle 序列化的数据不跨语言、不跨版本。如果你的状态中有 msgpack 无法处理的类型，更好的做法是重新设计状态结构，而不是开启 pickle 回退。

### Q4：Checkpoint 大小有上限吗？

LangGraph 本身没有硬性限制，但存储后端有。比如 SQLite 单行默认最大 1GB，Postgres 的 TOAST 机制可以处理大字段但会影响性能。实践中，建议单个 Checkpoint 控制在几 MB 以内。如果状态中有大对象（如完整文档内容），应该存到外部存储（如 S3），状态中只保留引用。
