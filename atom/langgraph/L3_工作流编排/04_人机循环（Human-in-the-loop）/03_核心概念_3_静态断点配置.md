# 核心概念3：静态断点配置（interrupt_before / interrupt_after）

> 静态断点是在 compile() 时配置的固定中断点，在指定节点执行前/后自动暂停图的运行

---

## 什么是静态断点？

**静态断点**是 LangGraph 中一种在编译阶段就确定好的中断机制。你在调用 `graph.compile()` 时通过 `interrupt_before` 或 `interrupt_after` 参数指定哪些节点需要暂停，图在运行到这些节点时会自动中断。

与 `interrupt()` 函数（动态中断）不同，静态断点不需要修改节点代码，也不能传递数据给客户端。它更像是在代码中打断点调试——简单直接，但灵活性有限。

> **官方定位**：静态断点是遗留方式（legacy approach），官方推荐使用 `interrupt()` 函数。
> 但在简单调试和全节点暂停场景下，静态断点仍然很实用。

---

## compile() 参数说明

从源码 `langgraph/graph/state.py` 中，`compile()` 方法的中断相关参数：

```python
# 源码位置：libs/langgraph/langgraph/graph/state.py
def compile(
    self,
    checkpointer: Checkpointer = None,
    *,
    cache: BaseCache | None = None,
    store: BaseStore | None = None,
    interrupt_before: All | list[str] | None = None,   # 节点执行前中断
    interrupt_after: All | list[str] | None = None,     # 节点执行后中断
    debug: bool = False,
    name: str | None = None,
) -> CompiledStateGraph[StateT, ContextT, InputT, OutputT]:
```

### 参数类型

| 参数 | 类型 | 说明 |
|------|------|------|
| `interrupt_before` | `All \| list[str] \| None` | 在指定节点执行**前**暂停 |
| `interrupt_after` | `All \| list[str] \| None` | 在指定节点执行**后**暂停 |

其中 `All` 是一个特殊类型，等价于 `"*"` 通配符，表示所有节点。

### 三种配置方式

```python
from langgraph.graph import StateGraph
from langgraph.checkpoint.memory import MemorySaver

builder = StateGraph(State)
# ... 添加节点和边 ...

# 方式1：指定具体节点列表
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["review", "execute"],  # 在 review 和 execute 节点前暂停
)

# 方式2：所有节点都暂停（使用 "*" 通配符）
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before="*",  # 每个节点执行前都暂停
)

# 方式3：组合使用 before 和 after
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["dangerous_action"],  # 危险操作前暂停
    interrupt_after=["generate_report"],    # 报告生成后暂停（供人工检查）
)
```

### interrupt_before vs interrupt_after 的区别

```
节点执行流程：

    ┌─ interrupt_before 检查点 ─┐
    │                           │
    │   如果匹配 → 暂停         │
    │   如果不匹配 → 继续       │
    │                           │
    ↓                           │
  节点执行                      │
    │                           │
    ↓                           │
    ┌─ interrupt_after 检查点 ──┐
    │                           │
    │   如果匹配 → 暂停         │
    │   如果不匹配 → 继续       │
    │                           │
    ↓
  下一个节点...
```

- **interrupt_before**：节点还没执行就暂停了。适合"执行前确认"场景。
- **interrupt_after**：节点已经执行完了才暂停。适合"执行后检查"场景。

## 运行时覆盖

除了在 `compile()` 时配置，你还可以在 `stream()` / `invoke()` 调用时动态覆盖断点配置：

```python
# 源码位置：libs/langgraph/langgraph/pregel/protocol.py
def stream(
    self,
    input: InputT | Command | None,
    config: RunnableConfig | None = None,
    *,
    interrupt_before: All | Sequence[str] | None = None,  # 运行时覆盖
    interrupt_after: All | Sequence[str] | None = None,    # 运行时覆盖
    ...
) -> Iterator[dict[str, Any] | Any]: ...
```

**使用示例**：

```python
# 编译时没有配置断点
graph = builder.compile(checkpointer=MemorySaver())

config = {"configurable": {"thread_id": "debug-1"}}

# 运行时临时添加断点（不影响编译配置）
for chunk in graph.stream(
    {"input": "test"},
    config=config,
    interrupt_before=["risky_node"],  # 临时在 risky_node 前暂停
):
    print(chunk)
```

这在调试时特别有用——你不需要重新编译图，就可以临时在某个节点前加断点。

## should_interrupt() 算法解析

静态断点的核心逻辑在 `_algo.py` 的 `should_interrupt()` 函数中。理解这个算法有助于理解断点为什么会（或不会）触发。

### 源码分析

```python
# 源码位置：libs/langgraph/langgraph/pregel/_algo.py
def should_interrupt(
    checkpoint: Checkpoint,
    interrupt_nodes: All | Sequence[str],
    tasks: Iterable[PregelExecutableTask],
) -> list[PregelExecutableTask]:
    """检查图是否应该中断"""

    # 1. 获取版本类型
    version_type = type(next(iter(checkpoint["channel_versions"].values()), None))
    null_version = version_type()

    # 2. 获取上次中断时的版本快照
    seen = checkpoint["versions_seen"].get(INTERRUPT, {})

    # 3. 核心判断：自上次中断以来是否有通道更新
    any_updates_since_prev_interrupt = any(
        version > seen.get(chan, null_version)
        for chan, version in checkpoint["channel_versions"].items()
    )

    # 4. 如果有更新，返回需要中断的任务列表
    return (
        [
            task for task in tasks
            if (
                # "*" 通配符：所有非隐藏节点
                (not task.config or TAG_HIDDEN not in task.config.get("tags", EMPTY_SEQ))
                if interrupt_nodes == "*"
                # 具体列表：检查节点名是否在列表中
                else task.name in interrupt_nodes
            )
        ]
        if any_updates_since_prev_interrupt
        else []
    )
```

### 算法拆解

这个函数做了三件事：

**第一步：检查是否有新的状态变化**

```python
any_updates_since_prev_interrupt = any(
    version > seen.get(chan, null_version)
    for chan, version in checkpoint["channel_versions"].items()
)
```

- `channel_versions`：记录每个状态通道的当前版本号
- `versions_seen[INTERRUPT]`：记录上次中断时各通道的版本号
- 如果任何通道的版本号比上次中断时更大，说明有新的状态变化

**为什么需要这个检查？** 防止重复中断。如果图恢复后没有任何状态变化就又到了同一个断点，不应该再次中断。

**第二步：匹配需要中断的节点**

```python
if interrupt_nodes == "*":
    # 通配符模式：所有非隐藏节点都中断
    (not task.config or TAG_HIDDEN not in task.config.get("tags", EMPTY_SEQ))
else:
    # 列表模式：只中断列表中的节点
    task.name in interrupt_nodes
```

- `"*"` 通配符会匹配所有节点，但排除带 `TAG_HIDDEN` 标签的内部节点
- 具体列表则精确匹配节点名称

**第三步：返回结果**

- 如果有状态变化且有匹配的节点 → 返回需要中断的任务列表（非空 = 触发中断）
- 如果没有状态变化 → 返回空列表（不中断）

### PregelLoop 中的调用

`should_interrupt()` 在执行循环中被调用两次：

```python
# 源码位置：libs/langgraph/langgraph/pregel/_loop.py

class PregelLoop:
    status: Literal[
        "input", "pending", "done",
        "interrupt_before", "interrupt_after", "out_of_steps"
    ]

    # 节点执行前检查
    if self.interrupt_before and should_interrupt(
        self.checkpoint, self.interrupt_before, self.tasks.values()
    ):
        self.status = "interrupt_before"
        raise GraphInterrupt()

    # ... 执行节点 ...

    # 节点执行后检查
    if self.interrupt_after and should_interrupt(
        self.checkpoint, self.interrupt_after, self.tasks.values()
    ):
        self.status = "interrupt_after"
        raise GraphInterrupt()
```

## 与动态 interrupt() 的对比

这是选择中断方式时最重要的参考表：

| 特性 | `interrupt()` 函数 | `interrupt_before/after` |
|------|-------------------|-------------------------|
| **类型** | 动态中断 | 静态断点 |
| **配置位置** | 节点代码内部 | `compile()` 参数 |
| **中断位置** | 节点内任意位置 | 节点执行前/后 |
| **灵活性** | 高（支持条件逻辑） | 低（固定节点列表） |
| **数据传递** | 支持（传递任意 JSON 值） | 不支持 |
| **条件中断** | 支持（`if condition: interrupt()`） | 不支持（总是中断） |
| **推荐度** | 官方推荐 | 遗留方式 |
| **修改节点代码** | 需要 | 不需要 |
| **运行时覆盖** | N/A | 支持 |

### 什么时候用静态断点？

- 快速调试：不想改节点代码，临时加个断点看看状态
- 全节点暂停：`interrupt_before="*"` 实现单步执行
- 简单场景：不需要向客户端传递数据，只需要暂停/恢复

### 什么时候用 interrupt()？

- 需要向客户端传递信息（如审批详情）
- 需要条件中断（只在特定条件下暂停）
- 需要在节点中间暂停（不是开头或结尾）
- 生产环境的人机交互

## 适用场景

### 场景1：调试断点

在开发阶段，你想逐步检查图的执行过程：

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from typing import TypedDict

class DebugState(TypedDict):
    data: str
    step: int

def step_a(state: DebugState):
    print(f"[Step A] 处理数据: {state['data']}")
    return {"data": state["data"] + " -> A处理完毕", "step": 1}

def step_b(state: DebugState):
    print(f"[Step B] 处理数据: {state['data']}")
    return {"data": state["data"] + " -> B处理完毕", "step": 2}

def step_c(state: DebugState):
    print(f"[Step C] 处理数据: {state['data']}")
    return {"data": state["data"] + " -> C处理完毕", "step": 3}

builder = StateGraph(DebugState)
builder.add_node("step_a", step_a)
builder.add_node("step_b", step_b)
builder.add_node("step_c", step_c)
builder.add_edge(START, "step_a")
builder.add_edge("step_a", "step_b")
builder.add_edge("step_b", "step_c")
builder.add_edge("step_c", END)

# 在每个节点后暂停，方便检查中间状态
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_after="*"  # 所有节点执行后都暂停
)

config = {"configurable": {"thread_id": "debug-session-1"}}

# 第一步：执行 step_a 后暂停
result = graph.invoke({"data": "原始数据", "step": 0}, config=config)
snapshot = graph.get_state(config)
print(f"当前状态: step={snapshot.values['step']}, data={snapshot.values['data']}")
print(f"下一个节点: {snapshot.next}")
# 输出: 当前状态: step=1, data=原始数据 -> A处理完毕
# 输出: 下一个节点: ('step_b',)

# 检查完毕，继续执行（传入 None 恢复）
result = graph.invoke(None, config=config)
snapshot = graph.get_state(config)
print(f"当前状态: step={snapshot.values['step']}, data={snapshot.values['data']}")
print(f"下一个节点: {snapshot.next}")
# 输出: 当前状态: step=2, data=原始数据 -> A处理完毕 -> B处理完毕
# 输出: 下一个节点: ('step_c',)

# 继续执行到完成
result = graph.invoke(None, config=config)
print(f"最终结果: {result}")
```

> **注意**：静态断点恢复时传入 `None`（而不是 `Command(resume=value)`），因为静态断点不支持数据传递。

### 场景2：危险操作前确认

在执行不可逆操作前自动暂停：

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from typing import TypedDict, Optional

class DeployState(TypedDict):
    service: str
    version: str
    status: Optional[str]

def validate_node(state: DeployState):
    """验证部署参数"""
    print(f"[Validate] 验证服务 {state['service']} v{state['version']}")
    return {"status": "validated"}

def deploy_node(state: DeployState):
    """执行部署（危险操作）"""
    print(f"[Deploy] 部署 {state['service']} v{state['version']} 到生产环境")
    return {"status": "deployed"}

def notify_node(state: DeployState):
    """发送通知"""
    print(f"[Notify] 部署完成通知已发送")
    return {"status": "notified"}

builder = StateGraph(DeployState)
builder.add_node("validate", validate_node)
builder.add_node("deploy", deploy_node)
builder.add_node("notify", notify_node)
builder.add_edge(START, "validate")
builder.add_edge("validate", "deploy")
builder.add_edge("deploy", "notify")
builder.add_edge("notify", END)

# 只在 deploy 节点前暂停（危险操作前确认）
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["deploy"]
)

config = {"configurable": {"thread_id": "deploy-001"}}

# 运行到 deploy 前暂停
result = graph.invoke(
    {"service": "user-api", "version": "2.0.0"},
    config=config
)

# 检查当前状态
snapshot = graph.get_state(config)
print(f"即将执行: {snapshot.next}")  # ('deploy',)
print(f"当前状态: {snapshot.values}")

# 人工确认后继续
input("确认部署？按回车继续...")
result = graph.invoke(None, config=config)
print(f"最终状态: {result['status']}")  # "notified"
```

## 完整实战示例：单步执行调试器

```python
"""
完整示例：使用静态断点实现图的单步执行调试器
类似 IDE 的 Step Over 功能
"""

from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from typing import TypedDict, Annotated
import operator


# ===== 1. 定义状态 =====
class PipelineState(TypedDict):
    input_text: str
    processed: str
    tokens: list[str]
    summary: str
    execution_log: Annotated[list[str], operator.add]

# ===== 2. 定义节点 =====
def preprocess_node(state: PipelineState):
    """预处理：清洗文本"""
    text = state["input_text"].strip().lower()
    print(f"  [Preprocess] 输入: '{state['input_text']}' → 输出: '{text}'")
    return {
        "processed": text,
        "execution_log": ["preprocess: 文本清洗完成"]
    }

def tokenize_node(state: PipelineState):
    """分词"""
    tokens = state["processed"].split()
    print(f"  [Tokenize] 输入: '{state['processed']}' → 输出: {tokens}")
    return {
        "tokens": tokens,
        "execution_log": [f"tokenize: 分词完成，共 {len(tokens)} 个 token"]
    }

def analyze_node(state: PipelineState):
    """分析"""
    token_count = len(state["tokens"])
    unique_count = len(set(state["tokens"]))
    summary = f"共 {token_count} 个词，{unique_count} 个不重复"
    print(f"  [Analyze] 分析结果: {summary}")
    return {
        "summary": summary,
        "execution_log": [f"analyze: {summary}"]
    }

# ===== 3. 构建图 =====
builder = StateGraph(PipelineState)
builder.add_node("preprocess", preprocess_node)
builder.add_node("tokenize", tokenize_node)
builder.add_node("analyze", analyze_node)
builder.add_edge(START, "preprocess")
builder.add_edge("preprocess", "tokenize")
builder.add_edge("tokenize", "analyze")
builder.add_edge("analyze", END)

# 使用 interrupt_after="*" 实现单步执行
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_after="*"  # 每个节点执行后都暂停
)

# ===== 4. 单步执行调试器 =====
def step_debugger(graph, initial_input, thread_id="debug"):
    """单步执行调试器"""
    config = {"configurable": {"thread_id": thread_id}}
    step = 0

    print("=" * 60)
    print("LangGraph 单步执行调试器")
    print("=" * 60)

    # 第一步
    print(f"\n--- Step {step}: 初始执行 ---")
    result = graph.invoke(initial_input, config=config)
    step += 1

    while True:
        # 检查状态
        snapshot = graph.get_state(config)

        if not snapshot.next:
            print(f"\n--- 执行完成 ---")
            print(f"最终状态:")
            for key, value in snapshot.values.items():
                print(f"  {key}: {value}")
            break

        print(f"\n  当前状态快照:")
        print(f"    下一个节点: {snapshot.next}")
        print(f"    执行日志: {snapshot.values.get('execution_log', [])}")

        # 等待用户确认
        user_input = input(f"\n--- Step {step}: 按回车继续，输入 'q' 退出 --- ")
        if user_input.lower() == 'q':
            print("调试中止。")
            break

        # 继续执行下一步
        result = graph.invoke(None, config=config)
        step += 1

    return snapshot.values

# 运行调试器
final_state = step_debugger(
    graph,
    {
        "input_text": "  Hello World Hello LangGraph  ",
        "processed": "",
        "tokens": [],
        "summary": "",
        "execution_log": []
    }
)
```

**运行输出**：

  当前状态快照:

## 执行流程图

静态断点在 PregelLoop 执行循环中的位置：

```
graph.invoke(input, config)
    ↓
PregelLoop 初始化
    ↓
┌─→ 检查 interrupt_before
│   ├─ should_interrupt() 返回非空 → status="interrupt_before"
│   │   → 保存 checkpoint → 抛出 GraphInterrupt → 返回给客户端
│   └─ 返回空 → 继续
│       ↓
│   执行当前节点
│   ├─ 节点内调用 interrupt() → 抛出 GraphInterrupt（动态中断）
│   └─ 正常完成
│       ↓
│   检查 interrupt_after
│   ├─ should_interrupt() 返回非空 → status="interrupt_after"
│   │   → 保存 checkpoint → 抛出 GraphInterrupt → 返回给客户端
│   └─ 返回空 → 继续
│       ↓
│   检查是否有下一步
│   ├─ 有 → 回到循环开头
│   └─ 无 → status="done"
└───────┘
```

## 与前端开发的类比

| 静态断点概念 | 前端类比 | 说明 |
|-------------|----------|------|
| `interrupt_before` | `beforeEach` 路由守卫 | 进入页面前拦截 |
| `interrupt_after` | `afterEach` 路由守卫 | 离开页面后触发 |
| `"*"` 通配符 | 全局中间件 | 所有路由都拦截 |
| 运行时覆盖 | 条件中间件 | 特定请求才启用 |
| `should_interrupt()` | 守卫函数的判断逻辑 | 决定是否放行 |
| `versions_seen` | `ETag` / `If-Modified-Since` | 避免重复处理 |

**前端类比示例**：
```javascript
// interrupt_before 就像 Vue Router 的 beforeEach
router.beforeEach((to, from, next) => {
  if (to.name === 'deploy') {
    // 在进入 deploy 页面前暂停，等待确认
    const confirmed = await confirm('确认部署？');
    if (confirmed) next();
    else next(false);
  } else {
    next();
  }
});

// interrupt_after 就像 afterEach
router.afterEach((to, from) => {
  // 页面加载完成后，记录日志
  logPageView(to.name);
});
```

## 日常生活类比

**静态断点就像高速公路的收费站**：

- **interrupt_before**：收费站在入口处。你必须先停车缴费，才能进入高速公路（节点执行前暂停）。
- **interrupt_after**：收费站在出口处。你开完高速后，在出口停车结算（节点执行后暂停）。
- **`"*"` 通配符**：每个出入口都有收费站（所有节点都暂停）。
- **versions_seen**：ETC 自动扣费记录。如果你刚刚通过了这个收费站，不会让你再停一次（避免重复中断）。

**与 interrupt() 的区别**：
- 静态断点 = 固定位置的收费站（建好了就在那里）
- interrupt() = 交警临时设卡（可以在任何地方、任何时候设置，还能告诉你为什么拦你）

## 常见误区

### 误区1：静态断点可以传递数据给客户端 ❌

**错误理解**：`interrupt_before` 暂停时可以像 `interrupt(value)` 一样传递信息。

**正确理解**：静态断点只是暂停，不传递任何数据。如果需要传递数据，请使用 `interrupt()` 函数。

```python
# 静态断点：只能暂停，不能传数据
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["review"]
)
# 恢复时只能传 None
graph.invoke(None, config=config)

# 动态中断：可以传数据
def review_node(state):
    decision = interrupt({"question": "请审核", "details": state})  # 传递数据
    return {"decision": decision}
# 恢复时传 Command(resume=value)
graph.invoke(Command(resume="approved"), config=config)
```

### 误区2：interrupt_before 和 interrupt_after 不能同时用 ❌

**正确理解**：完全可以同时使用，甚至可以对同一个节点同时配置 before 和 after。

```python
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["deploy"],    # 部署前确认
    interrupt_after=["deploy"],     # 部署后检查
)
```

### 误区3：恢复静态断点需要用 Command ❌

**正确理解**：静态断点恢复时传入 `None` 即可，不需要 `Command`。

```python
# 静态断点恢复
graph.invoke(None, config=config)  # 正确：传 None

# 动态中断恢复
graph.invoke(Command(resume="value"), config=config)  # 正确：传 Command
```

### 误区4：`"*"` 会中断所有节点包括内部节点 ❌

**正确理解**：`"*"` 通配符会排除带 `TAG_HIDDEN` 标签的内部节点（如 LangGraph 内部使用的辅助节点）。

## 总结

**静态断点是 LangGraph 中最简单的中断机制**：

1. **配置方式**：`compile(interrupt_before=[...], interrupt_after=[...])`
2. **三种粒度**：具体节点列表、`"*"` 通配符、`None`（不中断）
3. **运行时覆盖**：`stream()` / `invoke()` 也支持 interrupt_before/after 参数
4. **核心算法**：`should_interrupt()` 通过版本号比较避免重复中断
5. **恢复方式**：传入 `None` 即可，不支持数据传递
6. **适用场景**：调试断点、危险操作前确认、单步执行
7. **官方建议**：新项目优先使用 `interrupt()` 函数，静态断点作为补充
