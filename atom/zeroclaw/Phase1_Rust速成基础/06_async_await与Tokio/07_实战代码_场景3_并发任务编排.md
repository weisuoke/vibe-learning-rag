## 概述

异步编程最强大的地方不是"能等"，而是"能同时等多个"。Rust/Tokio 提供了三种核心并发模式：

- **join!** — 并行等待全部完成（像 `Promise.all()`）
- **select!** — 竞争/超时，取第一个完成的（像 `Promise.race()`）
- **spawn** — 后台任务，不阻塞当前流程（像不 await 的 `Promise`）

本节通过 5 个完整可运行场景，带你掌握这三种模式以及它们的组合使用。

## 1. 三种并发模式速查表

```
┌──────────────┬──────────────────────┬────────────────────────┐
│    模式       │       用途           │     JS 类比            │
├──────────────┼──────────────────────┼────────────────────────┤
│ join!        │ 并行等全部完成       │ Promise.all()          │
│ try_join!    │ 并行等全部，任一失败  │ Promise.all() 带 catch │
│              │ 立即返回错误         │                        │
│ select!      │ 竞争取第一个完成     │ Promise.race()         │
│ spawn        │ 后台任务独立运行     │ 不 await 的 Promise    │
│ spawn + Vec  │ 动态数量的并发任务   │ Promise.all(arr.map()) │
└──────────────┴──────────────────────┴────────────────────────┘
```

### 什么时候用哪个？决策树

```
需要并发？
│
├─ 任务数量固定（编译时已知）？
│   ├─ 等全部完成 → tokio::join!
│   │   └─ 需要错误提前返回？ → tokio::try_join!
│   └─ 只要第一个 → tokio::select!
│       └─ 超时控制？ → select! + tokio::time::sleep
│
├─ 任务数量动态（运行时才知道）？
│   ├─ 等全部完成 → spawn + Vec<JoinHandle> + join_all
│   └─ 后台运行不等待 → spawn（fire and forget）
│
└─ 组合场景？
    └─ select! 包 join!，或 spawn 里用 select! → 灵活组合
```

## 2. 场景1：join! 并行获取多个数据源

**场景**：同时调用 3 个 API，等全部返回后合并结果。

### Cargo.toml

```toml
[package]
name = "concurrency-patterns"
version = "0.1.0"
edition = "2021"

[dependencies]
tokio = { version = "1", features = ["full"] }
anyhow = "1"
```

### 代码

```rust
use std::time::{Duration, Instant};
use anyhow::Result;

// ─── 模拟三个不同的 API 调用 ───

async fn fetch_users() -> Vec<String> {
    tokio::time::sleep(Duration::from_millis(300)).await; // 模拟网络延迟
    println!("  ✅ fetch_users 完成（300ms）");
    vec!["Alice".into(), "Bob".into()]
}

async fn fetch_posts() -> Vec<String> {
    tokio::time::sleep(Duration::from_millis(500)).await;
    println!("  ✅ fetch_posts 完成（500ms）");
    vec!["Post 1".into(), "Post 2".into(), "Post 3".into()]
}

async fn fetch_comments() -> Vec<String> {
    tokio::time::sleep(Duration::from_millis(200)).await;
    println!("  ✅ fetch_comments 完成（200ms）");
    vec!["Great!".into(), "Nice post".into()]
}

#[tokio::main]
async fn main() {
    println!("═══ 场景1：join! 并行获取多个数据源 ═══\n");

    // ─── 串行方式（对照） ───
    let start = Instant::now();
    let users = fetch_users().await;
    let posts = fetch_posts().await;
    let comments = fetch_comments().await;
    println!(
        "\n串行耗时: {:?}（300 + 500 + 200 = 1000ms）",
        start.elapsed()
    );
    println!("  users: {:?}, posts: {:?}, comments: {:?}\n", users, posts, comments);

    // ─── 并行方式（join!） ───
    let start = Instant::now();
    let (users, posts, comments) = tokio::join!(
        fetch_users(),
        fetch_posts(),
        fetch_comments(),
    );
    println!(
        "\n并行耗时: {:?}（取最慢的 ≈ 500ms）",
        start.elapsed()
    );
    println!("  users: {:?}, posts: {:?}, comments: {:?}", users, posts, comments);
}
```

### 输出效果

```
═══ 场景1：join! 并行获取多个数据源 ═══

  ✅ fetch_users 完成（300ms）
  ✅ fetch_posts 完成（500ms）
  ✅ fetch_comments 完成（200ms）

串行耗时: 1.001s（300 + 500 + 200 = 1000ms）
  users: ["Alice", "Bob"], posts: ["Post 1", ...], comments: ["Great!", ...]

  ✅ fetch_comments 完成（200ms）     ← 最快的先完成
  ✅ fetch_users 完成（300ms）
  ✅ fetch_posts 完成（500ms）

并行耗时: 501ms（取最慢的 ≈ 500ms）   ← 快了一倍！
  users: ["Alice", "Bob"], posts: ["Post 1", ...], comments: ["Great!", ...]
```

### 关键理解

```
串行：   |--- users 300ms ---|--- posts 500ms ---|--- comments 200ms ---|  = 1000ms
并行：   |--- users 300ms ---|
         |------- posts 500ms -------|                                     = 500ms
         |-- comments 200ms --|
                总耗时 = max(300, 500, 200) = 500ms
```

**join! 的本质**：把多个 Future 打包成一个大 Future，Tokio 调度器在每个 `.await` 点轮流推进它们。

## 3. 场景2：try_join! 并行带错误处理

**场景**：三个 API 调用中任何一个失败就立即返回错误，不再等其他的。

```rust
use std::time::Duration;
use anyhow::{Result, bail};

async fn fetch_user_profile(user_id: u32) -> Result<String> {
    tokio::time::sleep(Duration::from_millis(300)).await;
    Ok(format!("用户{}的资料", user_id))
}

async fn fetch_user_settings(user_id: u32) -> Result<String> {
    tokio::time::sleep(Duration::from_millis(200)).await;
    if user_id == 42 {
        bail!("设置服务不可用")  // 模拟失败
    }
    Ok(format!("用户{}的设置", user_id))
}

async fn fetch_user_history(user_id: u32) -> Result<String> {
    tokio::time::sleep(Duration::from_millis(500)).await;
    println!("  ✅ history 完成（但如果 try_join 已经失败，这行不会打印）");
    Ok(format!("用户{}的历史", user_id))
}

#[tokio::main]
async fn main() {
    println!("═══ 场景2：try_join! 带错误处理 ═══\n");

    // ─── 成功场景 ───
    println!("--- 用户 1（全部成功）---");
    match tokio::try_join!(
        fetch_user_profile(1),
        fetch_user_settings(1),
        fetch_user_history(1),
    ) {
        Ok((profile, settings, history)) => {
            println!("  全部成功: {}, {}, {}", profile, settings, history);
        }
        Err(e) => {
            println!("  ❌ 出错: {}", e);
        }
    }

    // ─── 失败场景 ───
    println!("\n--- 用户 42（settings 会失败）---");
    match tokio::try_join!(
        fetch_user_profile(42),
        fetch_user_settings(42),   // ← 这个会失败
        fetch_user_history(42),    // ← 这个不会等到完成
    ) {
        Ok((profile, settings, history)) => {
            println!("  全部成功: {}, {}, {}", profile, settings, history);
        }
        Err(e) => {
            println!("  ❌ 出错: {}（500ms 的 history 被提前取消了）", e);
        }
    }
}
```

### join! vs try_join! 对比

```
┌────────────┬─────────────────────────┬────────────────────────────┐
│            │ join!                   │ try_join!                  │
├────────────┼─────────────────────────┼────────────────────────────┤
│ 返回类型   │ (T1, T2, T3)            │ Result<(T1, T2, T3)>       │
│ 错误处理   │ 全部完成，各自处理错误  │ 任一失败立即返回 Err       │
│ 取消行为   │ 不取消，全跑完          │ 一个失败，其余被取消       │
│ 适用场景   │ 独立操作，各自有降级    │ 全有或全无（事务性）       │
│ JS 类比    │ Promise.allSettled()    │ Promise.all()              │
└────────────┴─────────────────────────┴────────────────────────────┘
```

## 4. 场景3：select! 超时控制

**场景**：给 API 调用设置超时——要么在 3 秒内返回结果，要么超时报错。

```rust
use std::time::Duration;
use anyhow::Result;

/// 模拟一个可能很慢的 API 调用
async fn slow_api_call(delay_ms: u64) -> Result<String> {
    tokio::time::sleep(Duration::from_millis(delay_ms)).await;
    Ok(format!("API 返回结果（耗时 {}ms）", delay_ms))
}

#[tokio::main]
async fn main() {
    println!("═══ 场景3：select! 超时控制 ═══\n");

    // ─── 方法1：用 select! 手动实现超时 ───
    println!("--- 方法1：select! 手动实现超时 ---");

    // 快速 API（200ms < 1000ms 超时）
    let result = tokio::select! {
        result = slow_api_call(200) => {
            // API 先完成
            match result {
                Ok(data) => format!("✅ 成功: {}", data),
                Err(e) => format!("❌ API 错误: {}", e),
            }
        }
        _ = tokio::time::sleep(Duration::from_secs(1)) => {
            // 超时先触发
            "⏰ 超时！API 调用超过 1 秒".to_string()
        }
    };
    println!("  {}\n", result);

    // 慢速 API（2000ms > 1000ms 超时）
    let result = tokio::select! {
        result = slow_api_call(2000) => {
            match result {
                Ok(data) => format!("✅ 成功: {}", data),
                Err(e) => format!("❌ API 错误: {}", e),
            }
        }
        _ = tokio::time::sleep(Duration::from_secs(1)) => {
            "⏰ 超时！API 调用超过 1 秒".to_string()
        }
    };
    println!("  {}\n", result);

    // ─── 方法2：tokio::time::timeout（推荐！） ───
    println!("--- 方法2：tokio::time::timeout（推荐） ---");

    // 三层 match 解析：
    // 外层：timeout 是否超时
    //   中层：API 调用是否成功
    match tokio::time::timeout(
        Duration::from_secs(1),
        slow_api_call(200),
    ).await {
        Ok(Ok(data))  => println!("  ✅ 成功: {}", data),        // API 成功
        Ok(Err(e))    => println!("  ❌ API 错误: {}", e),       // API 失败
        Err(_)        => println!("  ⏰ 超时！超过 1 秒"),       // 超时
    }

    match tokio::time::timeout(
        Duration::from_secs(1),
        slow_api_call(2000),
    ).await {
        Ok(Ok(data))  => println!("  ✅ 成功: {}", data),
        Ok(Err(e))    => println!("  ❌ API 错误: {}", e),
        Err(_)        => println!("  ⏰ 超时！超过 1 秒"),
    }
}
```

### select! 的核心行为

```
select! {
    result = future_a => { ... }     ← 如果 A 先完成，执行这个分支
    result = future_b => { ... }     ← 如果 B 先完成，执行这个分支
}

关键点：
  1. 同时启动所有分支的 Future
  2. 第一个完成的分支被选中
  3. 其余分支被 DROP（取消）
  4. 只执行选中分支的代码块
```

**注意**：被取消的 Future 会在下一个 `.await` 点停止。如果 Future 内部持有需要清理的资源，需要考虑取消安全性（cancel safety）。

## 5. 场景4：spawn 并行任务 + 收集结果

**场景**：任务数量在运行时才知道（比如批量处理 N 个 URL），需要动态并发。

```rust
use std::time::{Duration, Instant};
use anyhow::Result;

/// 模拟处理单个 URL
async fn process_url(url: String) -> Result<String> {
    // 模拟不同 URL 不同的处理时间
    let delay = (url.len() as u64 * 50) % 500 + 100;
    tokio::time::sleep(Duration::from_millis(delay)).await;
    Ok(format!("{} → 处理完成（{}ms）", url, delay))
}

#[tokio::main]
async fn main() -> Result<()> {
    println!("═══ 场景4：spawn 动态数量的并发任务 ═══\n");

    // 动态的 URL 列表（运行时才知道数量）
    let urls = vec![
        "https://api.example.com/users".to_string(),
        "https://api.example.com/posts".to_string(),
        "https://api.example.com/comments".to_string(),
        "https://api.example.com/likes".to_string(),
        "https://api.example.com/shares".to_string(),
    ];

    println!("处理 {} 个 URL...\n", urls.len());
    let start = Instant::now();

    // ─── 方法1：spawn + collect JoinHandles ───
    // 类似 JS 的 Promise.all(urls.map(url => fetch(url)))
    let mut handles = Vec::new();

    for url in urls {
        // 每个 URL 在独立的 spawn 任务中处理
        let handle = tokio::spawn(async move {
            process_url(url).await
        });
        handles.push(handle);
    }

    // 收集所有结果
    let mut results = Vec::new();
    for handle in handles {
        match handle.await {
            Ok(Ok(result)) => {
                println!("  ✅ {}", result);
                results.push(result);
            }
            Ok(Err(e)) => {
                println!("  ❌ 任务错误: {}", e);
            }
            Err(e) => {
                println!("  💥 任务 panic: {}", e);
            }
        }
    }

    println!("\n总耗时: {:?}（并行 ≈ 最慢的那个）", start.elapsed());
    println!("成功处理: {}/{}", results.len(), 5);

    // ─── 方法2：futures::future::join_all（更简洁） ───
    // 需要在 Cargo.toml 添加 futures = "0.3"
    // 这里用 tokio::spawn + collect 代替演示
    println!("\n--- 方法2 思路：join_all ---");
    println!("  futures::future::join_all(handles).await");
    println!("  一行代码等待所有 handle 完成\n");

    Ok(())
}
```

### JoinHandle 的三层结果

```rust
// spawn 返回 JoinHandle<T>
// handle.await 返回 Result<T, JoinError>
//
// 所以如果内部函数返回 Result<String>：
// handle.await 的类型是 Result<Result<String>, JoinError>

match handle.await {
    Ok(Ok(data))  => { /* 任务正常完成，函数返回 Ok */ }
    Ok(Err(e))    => { /* 任务正常完成，但函数返回 Err */ }
    Err(join_err) => { /* 任务本身出问题了：panic 或被 abort */ }
}

// 对比 TypeScript：
// const result = await promise;  ← 只有成功/失败两种
// Rust 多了一层"任务本身是否正常运行完毕"
```

## 6. 场景5：组合使用 —— 真实业务场景

**场景**：一个 AI Agent 需要同时查询多个数据源，设置总超时，后台记录日志。

```rust
use std::time::Duration;
use anyhow::Result;

// ─── 模拟多个数据源 ───

async fn query_database(query: &str) -> Result<String> {
    tokio::time::sleep(Duration::from_millis(400)).await;
    Ok(format!("[DB] {} 的结果: 3 条记录", query))
}

async fn search_vector_store(query: &str) -> Result<String> {
    tokio::time::sleep(Duration::from_millis(300)).await;
    Ok(format!("[向量] {} 的语义搜索: 5 个匹配", query))
}

async fn call_external_api(query: &str) -> Result<String> {
    tokio::time::sleep(Duration::from_millis(600)).await;
    Ok(format!("[API] {} 的外部数据: 天气晴朗", query))
}

async fn log_to_analytics(event: String) {
    tokio::time::sleep(Duration::from_millis(100)).await;
    println!("  📊 [后台] 日志已记录: {}", event);
}

/// 组合模式：join! + select! + spawn
async fn handle_user_query(query: &str) -> Result<String> {
    println!("\n🔍 处理查询: \"{}\"", query);

    // ─── spawn：后台记录日志（fire and forget，不等待结果） ───
    let event = format!("用户查询: {}", query);
    tokio::spawn(async move {
        log_to_analytics(event).await;
    });
    // 注意：spawn 后我们不 .await handle，日志在后台异步完成

    // ─── select! + join!：并行查询 + 总超时控制 ───
    let query_owned = query.to_string();

    let result = tokio::select! {
        // 分支1：所有数据源并行查询（join!）
        results = async {
            let q = query_owned.as_str();
            tokio::join!(
                query_database(q),
                search_vector_store(q),
                call_external_api(q),
            )
        } => {
            let (db_result, vector_result, api_result) = results;

            // 收集成功的结果，失败的记录错误
            let mut combined = Vec::new();
            if let Ok(data) = db_result {
                println!("  ✅ {}", data);
                combined.push(data);
            }
            if let Ok(data) = vector_result {
                println!("  ✅ {}", data);
                combined.push(data);
            }
            if let Ok(data) = api_result {
                println!("  ✅ {}", data);
                combined.push(data);
            }

            format!("综合结果 ({} 个数据源): {}", combined.len(), combined.join(" | "))
        }

        // 分支2：总超时 2 秒
        _ = tokio::time::sleep(Duration::from_secs(2)) => {
            "⏰ 查询超时，请稍后重试".to_string()
        }
    };

    Ok(result)
}

#[tokio::main]
async fn main() -> Result<()> {
    println!("═══ 场景5：组合使用 join! + select! + spawn ═══");

    let result = handle_user_query("今天天气怎么样").await?;
    println!("\n📋 最终结果: {}", result);

    // 给后台任务一点时间完成（实际项目中不需要这样做）
    tokio::time::sleep(Duration::from_millis(200)).await;

    Ok(())
}
```

### 组合模式图解

```
handle_user_query("今天天气")
│
├── spawn(log_to_analytics)          ← 后台运行，不等待
│       └── 独立任务，不影响主流程
│
└── select! {                        ← 超时控制
        ├── join! {                  ← 并行查询
        │     ├── query_database     ── 400ms
        │     ├── search_vector_store── 300ms
        │     └── call_external_api  ── 600ms  ← 总耗时 ≈ 600ms
        │   } => 合并结果
        │
        └── sleep(2s) => "超时"      ← 2s 兜底
    }

时间线（成功场景）：
0ms     300ms   400ms   600ms                     2000ms
|-------|-------|-------|--------------------------|
  vector  DB     API                              超时线（未触发）
  完成    完成    完成
                  ↑
                  select! 选中 join! 分支
```

### Rust vs TypeScript 关键差异

| 维度 | Rust (Tokio) | TypeScript |
|------|-------------|------------|
| 并行等全部 | `tokio::join!` | `Promise.all()` |
| 带错误的并行 | `tokio::try_join!` | `Promise.all()` (默认就是 fail-fast) |
| 竞争/超时 | `tokio::select!` | `Promise.race()` |
| 后台任务 | `tokio::spawn` | 不 `await` 的 Promise |
| 动态并发 | `spawn` + `Vec<JoinHandle>` | `Promise.all(arr.map())` |
| 取消机制 | select! 自动 drop 未完成分支 | Promise.race() **不会取消**其他 Promise！ |
| 结果收集 | 需要处理 `JoinError` | 直接 `await` 得到结果 |
| 线程安全 | `Send + 'static` 约束 | 单线程，无需考虑 |

**最大区别**：`Promise.race()` 不会取消失败的 Promise（它们继续在后台运行！），而 `select!` 会 drop 未选中的 Future（真正取消）。

```typescript
// ⚠️ TypeScript 的 Promise.race 不取消：
const result = await Promise.race([
  fetch('/slow'),    // 即使 race 结束了，这个请求还在进行！
  timeout(1000),
]);
// /slow 的请求不会被取消，资源浪费

// ✅ Rust 的 select! 真正取消：
// select! {
//     data = fetch_slow() => { ... }
//     _ = sleep(1s) => { ... }   ← 超时触发后，fetch_slow() 的 Future 被 drop
// }                                   → HTTP 连接被关闭，资源释放
```

## 8. 常见错误与陷阱

### 陷阱1：join! 只能用于固定数量的 Future

```rust
// ❌ 错误：不能用 join! 处理动态数量
let urls = vec!["a", "b", "c"];
// tokio::join!(urls.iter().map(process)); // 编译错误！join! 是宏，需要固定参数数量

// ✅ 正确：用 spawn + collect
let handles: Vec<_> = urls.into_iter()
    .map(|url| tokio::spawn(async move { process(url).await }))
    .collect();
```

### 陷阱2：select! 中的引用借用问题

```rust
// ❌ 错误：两个分支同时借用 &mut data
let mut data = vec![1, 2, 3];
tokio::select! {
    _ = modify_a(&mut data) => {}  // ← &mut data
    _ = modify_b(&mut data) => {}  // ← &mut data 编译错误！
}

// ✅ 正确：用不可变引用，或者只在一个分支中修改
tokio::select! {
    result = read_a(&data) => { /* 在这里修改 data */ }
    result = read_b(&data) => { /* 在这里修改 data */ }
}
```

### 陷阱3：spawn 的 move 语义

```rust
// ❌ 错误：闭包借用了局部变量
let name = String::from("test");
tokio::spawn(async {
    println!("{}", name);  // 编译错误：name 可能在 spawn 任务完成前被 drop
});

// ✅ 正确：move 把 name 移进闭包
let name = String::from("test");
tokio::spawn(async move {
    println!("{}", name);  // OK：name 被移进了闭包
});
// 注意：这里之后不能再用 name（已被 move）
```

## 9. 并发模式速查表

```
┌─────────────────┬──────────────────┬───────────────┬─────────────────────┐
│ 模式             │ 何时用            │ 取消行为       │ 错误处理             │
├─────────────────┼──────────────────┼───────────────┼─────────────────────┤
│ join!(a, b)     │ 等全部完成        │ 不取消        │ 每个独立处理          │
│ try_join!(a, b) │ 全有或全无        │ 一个失败取消其 │ 第一个 Err 返回       │
│                 │                  │ 余分支        │                     │
│ select! { a, b }│ 取最快的/超时     │ 自动取消未选中 │ 每个分支独立处理      │
│ spawn(a)        │ 后台独立运行      │ 手动 abort    │ JoinHandle + 内部 Err│
│ spawn + Vec     │ 动态数量并发      │ 遍历 abort    │ 双层 Result          │
└─────────────────┴──────────────────┴───────────────┴─────────────────────┘
```

### 一句话总结

> **join! = Promise.all（等全部），select! = Promise.race（取最快），spawn = 后台 Promise（不等待）**——这三种模式覆盖了 95% 的异步并发需求。记住：Rust 的 select! 会真正取消未选中分支，这比 JS 的 Promise.race 更安全也更高效。
