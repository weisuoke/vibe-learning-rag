## 一句话定义

**? 运算符是 Rust 的错误传播语法糖，遇到 Err 自动返回错误，遇到 Ok 解包值继续执行。**

一个字符 `?`，替代了十几行 match 嵌套。它是 Rust 错误处理能写得简洁优雅的核心原因。

[来源: Comprehensive Rust (Google) - ? 运算符]

---

## 没有 ? 的世界（手动 match）

假设你要：读取文件 → 解析为数字 → 乘以 2 返回。每一步都可能失败。

```rust
use std::fs;
use std::io;
use std::num::ParseIntError;

#[derive(Debug)]
enum MyError {
    Io(io::Error),
    Parse(ParseIntError),
}

fn read_and_double(path: &str) -> Result<i64, MyError> {
    // 第一步：读文件
    let content = match fs::read_to_string(path) {
        Ok(c) => c,
        Err(e) => return Err(MyError::Io(e)),
    };

    // 第二步：解析数字
    let number = match content.trim().parse::<i64>() {
        Ok(n) => n,
        Err(e) => return Err(MyError::Parse(e)),
    };

    // 第三步：计算
    Ok(number * 2)
}
```

三步操作，写了 15 行。每一步都是同样的模式：Ok 就继续，Err 就返回。**重复、冗长、噪音大。**

[来源: Comprehensive Rust (Google)]

## 有了 ? 的世界

```rust
use std::fs;

fn read_and_double(path: &str) -> Result<i64, MyError> {
    let content = fs::read_to_string(path).map_err(MyError::Io)?;
    let number = content.trim().parse::<i64>().map_err(MyError::Parse)?;
    Ok(number * 2)
}
```

三行。逻辑完全一样。

如果用 `anyhow`（ZeroClaw 的选择），更简洁：

```rust
use anyhow::Result;
use std::fs;

fn read_and_double(path: &str) -> Result<i64> {
    let content = fs::read_to_string(path)?;
    let number = content.trim().parse::<i64>()?;
    Ok(number * 2)
}
```

`?` 做了什么？
- `Ok(value)` → 解包，把 `value` 赋给左边的变量
- `Err(e)` → 自动调用 `From::from(e)` 转换错误类型，然后 `return Err(converted)`

[来源: Comprehensive Rust (Google) - ? 运算符]

## TypeScript 对照

### throw + try/catch vs ? 运算符

```typescript
// TypeScript：错误通过 throw 抛出，try/catch 捕获
function readAndDouble(path: string): number {
    try {
        const content = fs.readFileSync(path, "utf-8");  // 可能 throw
        const number = parseInt(content.trim());          // 可能 NaN
        if (isNaN(number)) throw new Error("Parse failed");
        return number * 2;
    } catch (e) {
        throw e;  // 重新抛出（手动传播）
    }
}
```

```rust
// Rust：错误通过 Result 返回，? 自动传播
fn read_and_double(path: &str) -> Result<i64> {
    let content = fs::read_to_string(path)?;          // 失败自动返回 Err
    let number = content.trim().parse::<i64>()?;      // 失败自动返回 Err
    Ok(number * 2)
}
```

**关键区别**：

| 维度 | TypeScript throw/catch | Rust ? 运算符 |
|------|----------------------|---------------|
| 错误可见性 | 函数签名看不出会不会 throw | 返回 `Result` 一目了然 |
| 强制处理 | 不 catch 也能编译 | 不处理 Result 编译器警告 |
| 传播方式 | 自动冒泡（隐式） | `?` 显式标记（你知道哪行可能失败） |
| 错误类型 | any（运行时才知道） | 编译时确定的类型 |

### async/await 的 .catch() 链 vs ? 链

```typescript
// TypeScript：Promise 链式错误处理
const data = await fetch(url)
    .then(res => res.json())
    .catch(err => { throw new Error(`Fetch failed: ${err}`) });
```

```rust
// Rust：? 链式错误传播
let data = reqwest::get(url).await?
    .json::<Value>().await?;
```

Rust 的 `?` 在 async 函数中同样有效，而且每个 `?` 都精确标记了可能失败的点。

[来源: search_error_handling_01.md - TypeScript 对比]

## ? 的工作原理

`?` 不是魔法，它是编译器的语法糖。编译器把 `expression?` 展开为：

```rust
// 你写的：
let value = some_operation()?;

// 编译器展开为（对 Result）：
let value = match some_operation() {
    Ok(v) => v,
    Err(e) => return Err(From::from(e)),  // 注意 From::from 自动转换
};

// 编译器展开为（对 Option）：
let value = match some_operation() {
    Some(v) => v,
    None => return None,
};
```

三个关键行为：
1. **Ok(value) / Some(value)** → 解包，返回内部的 value，继续执行
2. **Err(e)** → 调用 `From::from(e)` 转换错误类型，然后 `return Err(converted)`
3. **None** → 直接 `return None`

**`From::from(e)` 是什么？** 这是 Rust 的类型转换 trait。如果函数返回 `Result<T, BigError>`，而 `?` 遇到的是 `SmallError`，只要实现了 `impl From<SmallError> for BigError`，`?` 就会自动转换。anyhow 对所有实现了 `std::error::Error` 的类型都实现了 From，所以用 anyhow 时 `?` 几乎可以用在任何 Result 上。

[来源: Comprehensive Rust (Google) - 自定义错误类型 + From trait]

## ? 用于 Result

### 基础示例

```rust
use std::fs;
use std::io;

fn read_first_line(path: &str) -> Result<String, io::Error> {
    let content = fs::read_to_string(path)?;  // io::Error 直接传播
    let first_line = content.lines().next().unwrap_or("").to_string();
    Ok(first_line)
}
```

### 链式 ?（多步操作）

```rust
use anyhow::Result;
use std::fs;

fn load_config_value(path: &str, key: &str) -> Result<String> {
    let content = fs::read_to_string(path)?;           // 步骤1：读文件
    let config: serde_json::Value = serde_json::from_str(&content)?;  // 步骤2：解析 JSON
    let value = config.get(key)                         // 步骤3：取值
        .and_then(|v| v.as_str())
        .ok_or_else(|| anyhow::anyhow!("Key '{}' not found", key))?;
    Ok(value.to_string())
}
```

每个 `?` 都是一个"可能提前返回"的点。读代码时，看到 `?` 就知道"这里可能失败并退出函数"。

[来源: Comprehensive Rust (Google)]

## ? 用于 Option

`?` 对 Option 也有效，但只能在返回 Option 的函数中使用。

```rust
fn get_middle_name(full_name: &str) -> Option<&str> {
    let mut parts = full_name.split_whitespace();
    let _first = parts.next()?;   // None → 直接 return None
    let middle = parts.next()?;   // None → 直接 return None
    let _last = parts.next()?;    // None → 直接 return None（确保有三个部分）
    Some(middle)
}

assert_eq!(get_middle_name("John Michael Smith"), Some("Michael"));
assert_eq!(get_middle_name("John Smith"), None);  // 没有中间名
assert_eq!(get_middle_name(""), None);             // 空字符串
```

**注意**：Option 的 `?` 不做任何转换，遇到 None 就直接 `return None`。

## ? 的自动错误转换（From trait）

这是 `?` 最强大的特性：自动把一种错误类型转换为另一种。

```rust
use std::io;
use std::num::ParseIntError;

// 自定义错误类型
#[derive(Debug)]
enum AppError {
    Io(io::Error),
    Parse(ParseIntError),
}

// 实现 From trait：io::Error → AppError
impl From<io::Error> for AppError {
    fn from(e: io::Error) -> Self {
        AppError::Io(e)
    }
}

// 实现 From trait：ParseIntError → AppError
impl From<ParseIntError> for AppError {
    fn from(e: ParseIntError) -> Self {
        AppError::Parse(e)
    }
}

fn process(path: &str) -> Result<i64, AppError> {
    let content = std::fs::read_to_string(path)?;  // io::Error → AppError::Io（自动）
    let number = content.trim().parse::<i64>()?;    // ParseIntError → AppError::Parse（自动）
    Ok(number)
}
```

**你不需要手动写 `map_err`**——只要实现了 From，`?` 就自动转换。

**ZeroClaw 为什么不需要手动写 From？** 因为 ZeroClaw 用 `anyhow::Result<T>`，anyhow 已经为所有 `std::error::Error` 实现了 From。所以任何标准错误类型遇到 `?` 都能自动转换为 `anyhow::Error`。

> **双重类比**
> - **前端类比**：From trait 就像 TypeScript 的类型转换函数。`?` 遇到类型不匹配时自动调用转换，就像 JS 引擎在 `==` 比较时自动做类型转换——但 Rust 的转换是显式定义的、编译时检查的。
> - **日常生活类比**：From trait 就像货币兑换。你在日本（函数 A）拿到美元（错误类型 A），`?` 自动帮你按汇率（From 实现）换成日元（错误类型 B），你不需要自己跑银行。

[来源: Comprehensive Rust (Google) - 自定义错误类型 + From trait]

## ZeroClaw 中的 ? 使用

### 加密操作中的链式 ?（security/secrets.rs）

```rust
// 来自 ZeroClaw security/secrets.rs
pub fn encrypt(&self, plaintext: &str) -> anyhow::Result<String> {
    let key_bytes = self.load_or_create_key()?;  // 读取密钥，失败则传播

    // 加密操作——cipher.encrypt 返回的不是标准 Error，需要 map_err 转换
    let ciphertext = cipher.encrypt(&nonce, plaintext.as_bytes())
        .map_err(|e| anyhow::anyhow!("Encryption failed: {e}"))?;

    Ok(base64_encode(&ciphertext))
}
```

**为什么这里用 `map_err` 而不是直接 `?`？** 因为加密库的错误类型没有实现 `std::error::Error`，anyhow 的 From 转换不适用。`map_err` 手动把它包装成 `anyhow::Error`，然后 `?` 再传播。

[来源: source_error_handling_01.md - security/secrets.rs 分析]

### 文件操作 + context（security/secrets.rs）

```rust
// 来自 ZeroClaw security/secrets.rs
fn load_key(&self) -> anyhow::Result<Vec<u8>> {
    let key_str = fs::read_to_string(&self.key_path)
        .context("Failed to read secret key file")?;
    // ...
}
```

**`.context("...")？` 是 ZeroClaw 中最常见的模式。** 拆解：
1. `fs::read_to_string()` 返回 `Result<String, io::Error>`
2. `.context("...")` 给错误附加上下文描述，返回 `Result<String, anyhow::Error>`
3. `?` 如果是 Err，传播带上下文的错误；如果是 Ok，解包字符串

最终的错误信息会是：`Failed to read secret key file: No such file or directory (os error 2)`——既有你写的上下文，也保留了原始错误。

[来源: source_error_handling_01.md, context7_anyhow_01.md - Context trait]

### 参数提取链（tools/web_fetch.rs）

```rust
// 来自 ZeroClaw tools/web_fetch.rs
let url = args.get("url")
    .and_then(|v| v.as_str())
    .ok_or_else(|| anyhow::anyhow!("Missing 'url' parameter"))?;
```

这是 Option 链 + `ok_or_else` 转 Result + `?` 传播的经典组合：
1. `args.get("url")` → `Option<&Value>`
2. `.and_then(|v| v.as_str())` → `Option<&str>`
3. `.ok_or_else(|| anyhow!(...))` → `Result<&str, anyhow::Error>`
4. `?` → 如果是 Err，传播错误；如果是 Ok，得到 `&str`

**这个模式在 ZeroClaw 的每个 Tool 实现中都会出现。** 记住它。

[来源: source_error_handling_01.md - tools/web_fetch.rs 分析]

### bail! 提前退出（tools/web_fetch.rs）

```rust
// 来自 ZeroClaw tools/web_fetch.rs
if url_str.is_empty() {
    anyhow::bail!("URL cannot be empty");
}
if !url_str.starts_with("http://") && !url_str.starts_with("https://") {
    anyhow::bail!("Only http:// and https:// URLs are allowed");
}
```

`bail!("msg")` 等价于 `return Err(anyhow!("msg"))`。它是 `?` 的"主动版"——不是等操作失败再传播，而是你主动判断条件然后返回错误。

[来源: source_error_handling_01.md, context7_anyhow_01.md - bail! 宏]

### ensure! 条件断言（security/secrets.rs）

```rust
// 来自 ZeroClaw security/secrets.rs
anyhow::ensure!(blob.len() > NONCE_LEN, "Encrypted value too short");
```

`ensure!(condition, "msg")` 等价于 `if !condition { bail!("msg") }`。更简洁的条件检查。

[来源: source_error_handling_01.md, context7_anyhow_01.md - ensure! 宏]

## ? 的限制

### 限制1：只能在返回 Result 或 Option 的函数中使用

```rust
// 编译错误！main 默认返回 ()，不是 Result
fn main() {
    let content = fs::read_to_string("file.txt")?;  // 错误！
}

// 修复：让 main 返回 Result
fn main() -> anyhow::Result<()> {
    let content = fs::read_to_string("file.txt")?;  // 现在可以了
    println!("{}", content);
    Ok(())
}
```

### 限制2：不能混用 Result 和 Option 的 ?

```rust
// 编译错误！函数返回 Result，但 Option 的 ? 想返回 None
fn process(data: &HashMap<String, String>) -> anyhow::Result<String> {
    let value = data.get("key")?;  // 错误！Option 的 ? 返回 None，但函数要求 Result
    Ok(value.clone())
}

// 修复：用 ok_or_else 把 Option 转成 Result
fn process(data: &HashMap<String, String>) -> anyhow::Result<String> {
    let value = data.get("key")
        .ok_or_else(|| anyhow::anyhow!("Key 'key' not found"))?;  // 现在是 Result 的 ?
    Ok(value.clone())
}
```

### 限制3：? 不能用在闭包中传播到外层函数

```rust
fn process_files(paths: &[&str]) -> anyhow::Result<Vec<String>> {
    // 修复：collect 到 Result<Vec<String>, _>，而不是在闭包里用 ?
    let contents: Result<Vec<String>, _> = paths.iter()
        .map(|p| fs::read_to_string(p))
        .collect();
    contents.map_err(|e| anyhow::anyhow!(e))
}
```

## 常见错误与解决

### 错误1：在非 Result/Option 返回的函数中使用 ?

```
error[E0277]: the `?` operator can only be used in a function
that returns `Result` or `Option`
```

**修复**：改函数签名，加上 `-> Result<T, E>` 或 `-> Option<T>`。

### 错误2：错误类型不匹配

```
error[E0277]: `?` couldn't convert the error to `MyError`
  the trait `From<io::Error>` is not implemented for `MyError`
```

**修复方案（三选一）**：
1. 实现 `From<io::Error> for MyError`
2. 用 `.map_err(|e| MyError::Io(e))?` 手动转换
3. 改用 `anyhow::Result`（推荐，ZeroClaw 的选择）

### 错误3：忘记在函数末尾返回 Ok(())

```rust
fn setup() -> anyhow::Result<()> {
    let config = load_config()?;
    apply_config(&config)?;
    // 编译错误！函数体最后一个表达式是 ()，不是 Result<()>

    // 修复：加上 Ok(())
    Ok(())
}
```

**Rust 函数的最后一个表达式就是返回值。** 如果函数返回 `Result<()>`，最后必须是 `Ok(())`。

## ? 运算符速查表

```
表达式                              ? 的行为
─────────────────────────────────────────────────────────
Ok(value)?                        → value（继续执行）
Err(e)?                           → return Err(From::from(e))
Some(value)?                      → value（继续执行）
None?                             → return None

expr.context("msg")?              → 给错误附加上下文后传播
expr.map_err(|e| anyhow!(...))?   → 手动转换错误后传播
expr.ok_or_else(|| anyhow!(...))?  → Option → Result 后传播
```

## 小结

| 要点 | 说明 |
|------|------|
| ? 是什么 | 语法糖：Ok 解包，Err 自动返回 |
| 核心价值 | 把 10 行 match 压缩成 1 个字符 |
| 自动转换 | 通过 From trait 自动转换错误类型 |
| 使用前提 | 函数必须返回 Result 或 Option |
| 不能混用 | Result 函数中不能直接对 Option 用 ?，需要 ok_or_else 转换 |
| ZeroClaw 三件套 | `?` 传播 + `.context()` 加上下文 + `bail!()` 主动退出 |
| 最常见模式 | `args.get("x").and_then(\|v\| v.as_str()).ok_or_else(\|\| anyhow!("..."))?` |

**记住**：看到 `?` 就想"这里可能失败并提前返回"。它让错误处理从"被迫写 match"变成"在正常逻辑上标记风险点"。
