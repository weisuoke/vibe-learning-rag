## 核心理念

**20% 的核心知识，解决 80% 的状态调试问题**

调试 LangGraph 工作流，本质上就是回答三个问题：

1. **现在状态是什么？** — 用 `get_state()` 查看
2. **状态怎么变成这样的？** — 用 `stream_mode="updates"` 追踪
3. **哪一步出了问题？** — 用 `get_state_history()` 定位

掌握以下 5 个要点，你就能定位和解决绝大多数 LangGraph 状态问题：

## 4.1 三行代码查看当前状态

**一句话：** `get_state()` 是你的第一调试工具——一个调用，拿到图执行的完整快照

调试的第一步永远是"看看现在是什么情况"。`get_state()` 返回一个 `StateSnapshot` 对象，包含当前状态值、下一步节点、元数据等所有你需要的信息。

**前提条件：** 图必须配置了 checkpointer，否则无法记录状态。

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from typing_extensions import TypedDict
from typing import Annotated
from operator import add

# ===== 构建一个简单的图 =====
class State(TypedDict):
    messages: Annotated[list[str], add]
    count: int

def greet(state: State) -> dict:
    return {"messages": ["你好！"], "count": state["count"] + 1}

def farewell(state: State) -> dict:
    return {"messages": ["再见！"], "count": state["count"] + 1}

builder = StateGraph(State)
builder.add_node("greet", greet)
builder.add_node("farewell", farewell)
builder.add_edge(START, "greet")
builder.add_edge("greet", "farewell")
builder.add_edge("farewell", END)

# 关键：配置 checkpointer
checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# ===== 执行图 =====
config = {"configurable": {"thread_id": "1"}}
result = graph.invoke({"messages": ["开始"], "count": 0}, config)

# ===== 调试：查看当前状态 =====
snapshot = graph.get_state(config)

# 最常用的两个字段
print(snapshot.values)   # 当前状态值
# {'messages': ['开始', '你好！', '再见！'], 'count': 2}

print(snapshot.next)     # 下一步节点
# ()  ← 空元组表示图已执行完毕
```

**`snapshot` 中最有用的字段：**

| 字段 | 含义 | 调试用途 |
|------|------|----------|
| `values` | 当前状态的所有字段值 | 检查数据是否正确 |
| `next` | 下一步要执行的节点 | 判断图是否执行完毕、卡在哪 |
| `metadata["step"]` | 执行步骤编号 | 知道跑到了第几步 |
| `metadata["source"]` | 快照来源（input/loop/update） | 区分是用户输入还是节点执行 |

```python
# 快速判断图的执行状态
snapshot = graph.get_state(config)
if not snapshot.next:
    print("图已执行完毕")
else:
    print(f"图还在运行，下一步: {snapshot.next}")

# 查看执行进度
step = snapshot.metadata.get("step", "?")
source = snapshot.metadata.get("source", "?")
print(f"当前在第 {step} 步，来源: {source}")
```

## 4.2 使用 stream_mode="updates" 追踪每个节点的变化

**一句话：** `stream_mode="updates"` 让你看到每个节点具体改了什么，是追踪状态变化的最佳方式

`get_state()` 只能看到最终结果，但你经常需要知道"每一步都发生了什么"。`stream_mode="updates"` 会在每个节点执行完毕后，输出该节点对状态的增量更新。

```python
# 用 stream 代替 invoke，追踪每一步的变化
config = {"configurable": {"thread_id": "2"}}

for chunk in graph.stream(
    {"messages": ["开始"], "count": 0},
    config,
    stream_mode="updates",
):
    # chunk 格式: {"节点名": {更新的字段}}
    node_name = list(chunk.keys())[0]
    update = chunk[node_name]
    print(f"[{node_name}] 更新了: {update}")
```

**预期输出：**
```
[greet] 更新了: {'messages': ['你好！'], 'count': 1}
[farewell] 更新了: {'messages': ['再见！'], 'count': 2}
```

**为什么 `updates` 比 `values` 更适合调试？**

| stream_mode | 输出内容 | 适合场景 |
|-------------|----------|----------|
| `"values"` | 每步后的完整状态 | 查看状态全貌 |
| `"updates"` | 仅该节点的增量更新 | 追踪"谁改了什么" |

```python
# 对比：stream_mode="values" 的输出
config = {"configurable": {"thread_id": "3"}}

for chunk in graph.stream(
    {"messages": ["开始"], "count": 0},
    config,
    stream_mode="values",
):
    print(f"完整状态: {chunk}")

# 输出：
# 完整状态: {'messages': ['开始'], 'count': 0}
# 完整状态: {'messages': ['开始', '你好！'], 'count': 1}
# 完整状态: {'messages': ['开始', '你好！', '再见！'], 'count': 2}
```

**实用技巧：** 同时使用多种 stream_mode

```python
# 可以传入列表，同时获取多种模式的输出
config = {"configurable": {"thread_id": "4"}}

for mode, chunk in graph.stream(
    {"messages": ["开始"], "count": 0},
    config,
    stream_mode=["updates", "values"],
):
    print(f"[{mode}] {chunk}")
```

## 4.3 遍历历史定位问题步骤

**一句话：** `get_state_history()` 返回完整的执行历史，让你像 Git log 一样回溯每一步

当你发现最终状态不对，但不知道是哪一步出了问题时，`get_state_history()` 就是你的"时间机器"。它返回按时间倒序排列的所有状态快照。

```python
config = {"configurable": {"thread_id": "1"}}

# 遍历完整执行历史（最新的在前）
for snapshot in graph.get_state_history(config):
    step = snapshot.metadata.get("step", "?")
    source = snapshot.metadata.get("source", "?")
    next_nodes = snapshot.next
    values = snapshot.values
    print(f"Step {step} [{source}]: next={next_nodes}")
    print(f"  values={values}")
    print("---")
```

**预期输出：**
```
Step 2 [loop]: next=()
  values={'messages': ['开始', '你好！', '再见！'], 'count': 2}
---
Step 1 [loop]: next=('farewell',)
  values={'messages': ['开始', '你好！'], 'count': 1}
---
Step 0 [loop]: next=('greet',)
  values={'messages': ['开始'], 'count': 0}
---
Step -1 [input]: next=('greet',)
  values={'messages': ['开始'], 'count': 0}
---
```

**读懂历史记录的关键：**

| step 值 | source | 含义 |
|---------|--------|------|
| `-1` | `input` | 用户输入刚进来，还没执行任何节点 |
| `0` | `loop` | 第一个节点执行完毕 |
| `1` | `loop` | 第二个节点执行完毕 |
| `N` | `update` | 手动调用了 `update_state()` |

**实用技巧：** 限制历史数量和按条件筛选

```python
# 只看最近 5 条历史
for snapshot in graph.get_state_history(config, limit=5):
    print(snapshot.metadata.get("step"))

# 转为列表，方便随机访问
history = list(graph.get_state_history(config))
print(f"共 {len(history)} 个快照")

# 对比两个步骤的状态差异
if len(history) >= 2:
    current = history[0].values
    previous = history[1].values
    for key in current:
        if current.get(key) != previous.get(key):
            print(f"字段 '{key}' 发生变化:")
            print(f"  之前: {previous.get(key)}")
            print(f"  之后: {current.get(key)}")
```

## 4.4 节点内添加调试日志

**一句话：** 最简单直接的调试方法——在节点函数里加 print，看输入和输出

有时候你不需要复杂的工具，一个 `print` 就够了。在节点函数的入口和出口打印状态，就能快速定位问题。

### 基础版：print 调试

```python
def my_node(state: State) -> dict:
    # 入口：看看节点收到了什么
    print(f"[my_node] 输入状态: {state}")

    # 业务逻辑
    result = {"messages": ["处理完成"], "count": state["count"] + 1}

    # 出口：看看节点要返回什么
    print(f"[my_node] 输出更新: {result}")
    return result
```

### 进阶版：logging 模块

```python
import logging

# 配置日志（在程序入口处设置一次）
logging.basicConfig(
    level=logging.DEBUG,
    format="%(asctime)s [%(name)s] %(levelname)s: %(message)s",
)
logger = logging.getLogger("my_graph")

def my_node(state: State) -> dict:
    logger.debug(f"输入: messages={state.get('messages')}, count={state.get('count')}")

    result = {"messages": ["处理完成"], "count": state["count"] + 1}

    logger.info(f"输出: {result}")
    return result
```

**logging vs print：**

| 特性 | print | logging |
|------|-------|---------|
| 配置复杂度 | 零配置 | 需要初始化 |
| 级别控制 | 无 | DEBUG/INFO/WARNING/ERROR |
| 生产环境 | 需要手动删除 | 调整级别即可关闭 |
| 推荐场景 | 快速排查 | 长期保留的调试代码 |

### 高级版：StreamWriter 自定义输出

```python
from langgraph.config import get_stream_writer

def my_node(state: State) -> dict:
    writer = get_stream_writer()

    # 发送自定义调试信息到流式输出
    writer({"debug": "开始处理", "input_count": state["count"]})

    result = {"messages": ["处理完成"], "count": state["count"] + 1}

    writer({"debug": "处理完成", "output_count": result["count"]})
    return result

# 接收自定义调试信息
config = {"configurable": {"thread_id": "5"}}
for chunk in graph.stream(
    {"messages": ["开始"], "count": 0},
    config,
    stream_mode="custom",
):
    print(f"[自定义] {chunk}")
```

## 4.5 使用 stream_mode="debug" 获取最详细的执行信息

**一句话：** `stream_mode="debug"` 是调试的"核武器"——输出任务事件和检查点事件的完整信息

当其他方法都不够用时，`stream_mode="debug"` 会给你最细粒度的执行信息。它实际上是 `"checkpoints"` + `"tasks"` 的组合，包含每个任务的输入、输出、错误、中断等全部细节。

```python
config = {"configurable": {"thread_id": "6"}}

for event in graph.stream(
    {"messages": ["开始"], "count": 0},
    config,
    stream_mode="debug",
):
    event_type = event.get("type", "unknown")

    if event_type == "task":
        # 任务开始事件
        payload = event["payload"]
        print(f"[任务开始] 节点: {payload['name']}")
        print(f"  输入: {payload.get('input')}")

    elif event_type == "task_result":
        # 任务完成事件
        payload = event["payload"]
        print(f"[任务完成] 节点: {payload['name']}")
        if payload.get("error"):
            print(f"  错误: {payload['error']}")
        else:
            print(f"  结果: {payload.get('result')}")

    elif event_type == "checkpoint":
        # 检查点事件
        payload = event["payload"]
        print(f"[检查点] 值: {payload['values']}")
        print(f"  下一步: {payload['next']}")

    print("---")
```

**debug 模式输出的三种事件：**

| 事件类型 | 触发时机 | 包含信息 |
|----------|----------|----------|
| `task` | 节点开始执行前 | 节点名、输入数据、触发器 |
| `task_result` | 节点执行完毕后 | 节点名、结果、错误、中断 |
| `checkpoint` | 状态保存时 | 完整状态值、下一步节点、元数据 |

**什么时候用 debug 模式？**

- 其他 stream_mode 信息不够时
- 需要看到节点的原始输入（不只是状态）
- 需要捕获错误和中断的详细信息
- 需要同时看到任务执行和状态变化

## 调试决策树

遇到问题时，按这个顺序排查：

```
图的输出不对
  │
  ├─ 第1步：graph.get_state(config)
  │   └─ 看 values 和 next，确认最终状态
  │
  ├─ 第2步：stream_mode="updates"
  │   └─ 追踪每个节点的更新，找到"从哪一步开始不对"
  │
  ├─ 第3步：get_state_history(config)
  │   └─ 对比相邻步骤的状态差异，精确定位问题
  │
  ├─ 第4步：在可疑节点内加 print/logging
  │   └─ 看节点收到的输入和产生的输出
  │
  └─ 第5步：stream_mode="debug"
      └─ 查看任务级别的完整信息，包括错误和中断
```
