# Agent推理与决策 - 30字核心

> **Agent推理是LLM通过Thought→Action→Observation循环实现多步决策的机制，是Agent智能行为的核心引擎。**

---

## 关键词解析

| 关键词 | 含义 |
|--------|------|
| **Agent推理** | LLM 在约束格式下的多步思考过程，不是一次性生成答案，而是"想一步→做一步→看结果→再想" |
| **Thought→Action→Observation** | ReAct 循环的三个阶段：思考当前状态（Thought）、选择并执行工具（Action）、获取执行结果（Observation），循环往复直到问题解决 |
| **循环** | 推理不是一次性的，而是迭代的。每一轮 Observation 都会成为下一轮 Thought 的输入，让 LLM 基于新信息做出更好的决策 |
| **多步决策** | 复杂问题需要多个步骤才能解决。Agent 在每一步都要决定：继续调用工具（AgentAction）还是返回最终答案（AgentFinish） |
| **核心引擎** | 推理与决策是 Agent 的"大脑"。AgentExecutor 是"身体"（执行框架），Tools 是"手脚"（执行能力），而推理决定了"做什么"和"怎么做" |

---

## 为什么这30个字重要

### LLM 的根本局限

LLM 是一个"一次性文本生成器"。给它一个复杂问题，它只能凭训练数据"猜"一个答案。它不能分步骤思考，不能中途查资料，不能根据新信息调整方向。

### 推理机制解决了什么

推理机制通过**结构化的 Prompt + 循环执行**，把 LLM 从"一次性回答"升级为"多步推理"：

```
没有推理:  用户提问 → LLM 一次性生成答案（可能幻觉、可能遗漏）
有了推理:  用户提问 → LLM 思考 → 调用工具 → 看结果 → 再思考 → ... → 最终答案
```

这不是锦上添花，而是 Agent 能否解决复杂问题的**前提条件**。

### 决策的两种结果

Agent 每一步推理的输出，最终都会被解析为两种决策之一：

- **AgentAction**: "我还需要更多信息，让我调用这个工具" → 循环继续
- **AgentFinish**: "我已经有足够信息了，这是最终答案" → 循环结束

这个二元决策机制，就是 Agent 循环的"开关"。

## 与 RAG/Agent 开发的关系

在 RAG 系统中，推理与决策让 Agent 能够**智能编排检索流程**：

| 场景 | 推理的作用 |
|------|-----------|
| 多步检索 | 先查 A 知识库，根据结果决定是否还需要查 B 知识库 |
| 查询分解 | 把复杂问题拆成多个子问题，逐个检索后综合回答 |
| 结果验证 | 检索到信息后，推理判断是否足够回答问题，不够就继续检索 |
| 工具选择 | 根据问题类型，决定用向量检索、关键词搜索还是 SQL 查询 |

**没有推理，Agent 只是一个"固定流程的脚本"；有了推理，Agent 才是一个"能自主决策的智能体"。**
