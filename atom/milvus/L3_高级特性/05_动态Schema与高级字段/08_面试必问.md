## 问题1："Milvus的动态Schema和AddCollectionField有什么区别？分别在什么场景下使用？"

**普通回答（❌ 不出彩）：**

"动态Schema可以灵活添加字段，AddCollectionField是往已有Collection里加字段。动态Schema更灵活，AddCollectionField更正式。"

**出彩回答（✅ 推荐）：**

> **这两个机制解决的是同一个问题——Schema演化，但在机制、性能和适用场景上有本质区别：**
>
> 1. **机制层面**：动态Schema通过 `enable_dynamic_field=True` 启用后，未定义的字段自动存储在隐式的 `$meta` JSON 字段中，本质是一个大JSON。AddCollectionField 则是真正修改 Collection 的 Schema 定义，添加一个有明确类型的标量字段，必须设置 `nullable=True`，内部通过 Schema 版本号递增来管理（源码中 `Version: coll.SchemaVersion + 1`）。
>
> 2. **性能层面**：动态字段**无法创建索引**，每次查询都需要 JSON 解析，相当于全表扫描。AddCollectionField 添加的字段可以创建 INVERTED 索引，查询性能高出一个数量级。但要注意，nullable 字段的聚合操作比非 nullable 慢 4 倍（Issue #47566）。
>
> 3. **实践层面**：生产环境的最佳实践是**三层策略**——核心字段（id、vector、tenant_id）用固定 Schema 加索引；业务元数据（tags、author）用动态字段快速迭代；当某个动态字段的查询频率升高后，用 AddCollectionField 将其"转正"为固定字段并加索引。但 AddCollectionField 有并发写入冲突的风险（Issue #41858），需要在维护窗口操作。
>
> **与传统数据库的区别**：AddCollectionField 类似 `ALTER TABLE ADD COLUMN`，但限制更多——只能加标量字段、必须 nullable、添加后需要 reload Collection。动态 Schema 则没有传统数据库的对应物，更像 MongoDB 的 schemaless 模式。

**为什么这个回答出彩？**
1. ✅ 从机制、性能、实践三个层面递进分析，展示系统性思维
2. ✅ 引用了具体的源码实现和 GitHub Issue 编号，展示深度研究
3. ✅ 给出了"三层策略"的实践方案，展示工程经验
4. ✅ 与传统数据库对比，展示知识迁移能力

## 问题2："在RAG系统中，如何设计向量存储的Schema来支持多租户和元数据扩展？"

**普通回答（❌ 不出彩）：**

"给每个租户创建一个Collection，然后用动态字段存元数据就行了。"

**出彩回答（✅ 推荐）：**

> **多租户RAG系统的Schema设计需要从隔离策略、字段设计和演化策略三个维度考虑：**
>
> 1. **隔离策略选择**：根据租户规模选择不同方案。小规模（<100租户）用 Collection 级别隔离，每个租户独立 Collection，完全隔离但资源开销大；中规模（100-10K）用 Partition 隔离，共享 Collection 和索引，查询时指定 `partition_names`；大规模（>10K）用元数据过滤，所有租户共享一个 Collection，通过 `tenant_id` 字段过滤，资源开销最小但需要严格的权限控制。Milvus 2.6 支持 100K collections，所以 Collection 隔离的上限比以前高很多。
>
> 2. **Schema 字段设计**：采用"固定+动态"混合策略。固定字段包括 id、vector、text、tenant_id、created_at 这些高频过滤字段，必须创建索引。动态字段用于各租户不同的业务元数据，比如技术部门需要 api_version，法务部门需要 approval_status。如果元数据结构复杂且有嵌套，可以用 JSON 字段配合 JSON Path Index。
>
> 3. **演化策略**：初期用动态 Schema 快速迭代，当某个字段的查询频率升高后，用 AddCollectionField 转为固定字段并加索引。在元数据中记录 schema_version，支持多版本数据共存。对于大规模场景，还可以用 Int8 向量替代 Float32，节省 75% 内存——Reddit 上有 RAG 创业公司用 RaBitQ+SQ8 组合量化，成本直降 75%。
>
> **实际项目中的坑**：动态字段的 group_by 对缺失字段处理有 bug（Issue #47438），所以聚合查询一定要用固定字段。AddCollectionField 与并发 insert 有冲突（Issue #41858），添加字段时必须暂停写入。

**为什么这个回答出彩？**
1. ✅ 给出了按租户规模分级的具体方案，而不是一刀切
2. ✅ "固定+动态"混合策略展示了工程权衡能力
3. ✅ 提到了 Int8 量化和具体的成本数据，展示成本意识
4. ✅ 主动提到生产环境的坑和 Issue 编号，展示实战经验
