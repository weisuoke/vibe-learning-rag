## 概述

长运行服务中，组件不可避免会崩溃：网络断开、数据库超时、外部 API 报错……你不能每次都手动重启。

ZeroClaw 的 Daemon 模式用一个精巧的模式解决了这个问题：**`spawn_component_supervisor`**——通用的异步组件监控器，支持自动重启、指数退避、健康追踪、优雅关闭。

本节完整复刻这个模式，代码可直接 `cargo run`。

## 1. 问题场景

想象你在运行一个 AI Agent 服务：

```
┌─────────────────────────────────────────┐
│              Daemon 进程                 │
│                                         │
│   ┌──────────┐  ┌──────────┐           │
│   │ Gateway  │  │ Channel  │  ← 这些组件都可能崩溃！  │
│   │ (HTTP)   │  │ (Discord)│           │
│   └──────────┘  └──────────┘           │
│   ┌──────────┐                          │
│   │Scheduler │  ← 定时任务也可能出错    │
│   └──────────┘                          │
└─────────────────────────────────────────┘
```

**如果不做监控**：
- Gateway 端口被占用 → 整个服务挂掉
- Discord 断线 → 用户消息丢失
- 定时器 panic → 数据不同步

**期望行为**：
- 组件崩溃后自动重启
- 连续崩溃时增加等待时间（指数退避），避免疯狂重试
- 重启成功后重置等待时间
- 随时可查看每个组件的健康状态
- 收到 Ctrl+C 时优雅关闭所有组件

## 2. 指数退避可视化

```
组件首次启动
    │
    ▼
 ╔═══════╗
 ║  运行  ║─── 正常运行中 ───────────────────────────────────────┐
 ╚═══╤═══╝                                                      │
     │ 崩溃!                                                     │
     ▼                                                           │
  失败1 → 等待 2s  → 🔄 重启                                     │
     │ 又崩了!                                                    │
     ▼                                                           │
  失败2 → 等待 4s  → 🔄 重启        backoff × 2                  │
     │ 还是崩!                                                    │
     ▼                                                           │
  失败3 → 等待 8s  → 🔄 重启        backoff × 2                  │
     │ 继续崩...                                                  │
     ▼                                                           │
  失败4 → 等待 16s → 🔄 重启        backoff × 2                  │
     │                                                            │
     ▼                                                           │
  失败5 → 等待 32s → 🔄 重启        backoff × 2                  │
     │                                                            │
     ▼                                                           │
  失败6 → 等待 60s → 🔄 重启        cap at max_backoff!          │
     │                                                            │
     ▼                                                           │
  失败7 → 等待 60s → 🔄 重启        保持 max_backoff              │
     │                                                            │
     ▼                                                           │
  成功!  → 重置 backoff = 2s ←──────────────────────────────────┘
     │
     ▼
  继续正常运行...
```

**核心算法**：

```
backoff = initial_backoff  // 初始值，比如 2 秒
max_backoff                // 上限，比如 60 秒

loop:
    运行组件
    if 崩溃:
        等待 backoff 秒
        backoff = min(backoff × 2, max_backoff)  // 翻倍，但不超过上限
    if 成功后退出:
        backoff = initial_backoff  // 重置
```

### src/main.rs

```rust
use anyhow::Result;
use std::collections::HashMap;
use std::future::Future;
use std::sync::{Arc, Mutex};
use std::time::Duration;
use tokio::task::JoinHandle;

// ============================================================
// 1. 健康状态追踪
// 对应 ZeroClaw: src/health.rs
// ============================================================

#[derive(Debug, Clone)]
struct ComponentHealth {
    name: String,
    status: String,        // "ok" | "error"
    error_msg: String,     // 最后一次错误信息
    restart_count: u32,    // 重启次数
}

/// 全局健康状态（Arc<Mutex<>> 实现跨任务共享）
#[derive(Clone)]
struct HealthRegistry {
    components: Arc<Mutex<HashMap<String, ComponentHealth>>>,
}

impl HealthRegistry {
    fn new() -> Self {
        Self {
            components: Arc::new(Mutex::new(HashMap::new())),
        }
    }

    fn mark_ok(&self, name: &str) {
        let mut map = self.components.lock().unwrap();
        let entry = map.entry(name.to_string()).or_insert(ComponentHealth {
            name: name.to_string(),
            status: "ok".to_string(),
            error_msg: String::new(),
            restart_count: 0,
        });
        entry.status = "ok".to_string();
        entry.error_msg.clear();
    }

    fn mark_error(&self, name: &str, msg: &str) {
        let mut map = self.components.lock().unwrap();
        let entry = map.entry(name.to_string()).or_insert(ComponentHealth {
            name: name.to_string(),
            status: "error".to_string(),
            error_msg: msg.to_string(),
            restart_count: 0,
        });
        entry.status = "error".to_string();
        entry.error_msg = msg.to_string();
    }

    fn bump_restart(&self, name: &str) {
        let mut map = self.components.lock().unwrap();
        if let Some(entry) = map.get_mut(name) {
            entry.restart_count += 1;
        }
    }

    fn print_status(&self) {
        let map = self.components.lock().unwrap();
        println!("\n╔══════════════════════════════════════════════╗");
        println!("║          组件健康状态                         ║");
        println!("╠══════════════════════════════════════════════╣");
        for (_, health) in map.iter() {
            let icon = if health.status == "ok" { "✅" } else { "❌" };
            println!(
                "║ {} {:<12} | 重启: {:<3} | {}",
                icon,
                health.name,
                health.restart_count,
                if health.error_msg.is_empty() {
                    "正常运行".to_string()
                } else {
                    health.error_msg.clone()
                }
            );
        }
        println!("╚══════════════════════════════════════════════╝\n");
    }
}

// ============================================================
// 2. 通用组件监控器
// 对应 ZeroClaw: src/daemon/mod.rs → spawn_component_supervisor
// ============================================================

/// 通用的异步组件监控器
///
/// 泛型约束解析：
///   F: FnMut() -> Fut + Send + 'static
///      │  │        │     │      │
///      │  │        │     │      └─ 闭包必须拥有所有数据（不能借用栈上的值）
///      │  │        │     └─ 闭包可以安全地跨线程传递
///      │  │        └─ F 每次调用返回一个 Future
///      │  └─ 可变闭包（允许闭包内部修改捕获的变量）
///      └─ 闭包类型
///
///   Fut: Future<Output = Result<()>> + Send + 'static
///        │                             │      │
///        │                             │      └─ Future 可以跨 .await 存活
///        │                             └─ Future 可以在多线程运行时调度
///        └─ 异步操作完成后返回 Result<()>
///
fn spawn_component_supervisor<F, Fut>(
    name: &'static str,
    initial_backoff_secs: u64,
    max_backoff_secs: u64,
    health: HealthRegistry,
    mut run_component: F,
) -> JoinHandle<()>
where
    F: FnMut() -> Fut + Send + 'static,
    Fut: Future<Output = Result<()>> + Send + 'static,
{
    tokio::spawn(async move {
        // 初始化退避参数
        let mut backoff = initial_backoff_secs.max(1);
        let max_backoff = max_backoff_secs.max(backoff);

        loop {
            // 标记组件为健康状态
            health.mark_ok(name);
            println!("🟢 [{}] 启动运行...", name);

            // 运行组件
            match run_component().await {
                Ok(()) => {
                    // 组件正常退出 → 意外！长运行组件不应该正常退出
                    health.mark_error(name, "组件意外退出");
                    println!(
                        "⚠️  [{}] 意外退出（正常返回），重置 backoff → {}s",
                        name, initial_backoff_secs
                    );
                    backoff = initial_backoff_secs.max(1);
                }
                Err(e) => {
                    // 组件崩溃 → 记录错误
                    let err_msg = e.to_string();
                    health.mark_error(name, &err_msg);
                    println!("❌ [{}] 崩溃: {}", name, err_msg);
                }
            }

            // 增加重启计数
            health.bump_restart(name);

            // 等待退避时间
            println!(
                "⏳ [{}] 等待 {}s 后重启（max: {}s）...",
                name, backoff, max_backoff
            );
            tokio::time::sleep(Duration::from_secs(backoff)).await;

            // 指数退避：backoff × 2，但不超过 max_backoff
            // saturating_mul：溢出时返回 u64::MAX 而不是 panic
            backoff = backoff.saturating_mul(2).min(max_backoff);
        }
    })
}

// 3. 模拟组件（模拟随机成功/失败）

/// 模拟 Gateway 组件（HTTP 服务器）
/// 随机运行 1-3 秒后以 70% 概率崩溃
async fn run_gateway() -> Result<()> {
    let run_time = 1 + (rand_simple() % 3);
    println!("    [gateway] 模拟运行 {}s...", run_time);
    tokio::time::sleep(Duration::from_secs(run_time)).await;

    if rand_simple() % 10 < 7 {
        anyhow::bail!("端口 8080 被占用")
    }
    Ok(()) // 正常退出（也会触发重启）
}

    tokio::time::sleep(Duration::from_secs(run_time)).await;

    if rand_simple() % 10 < 6 {

/// 模拟 Scheduler 组件（定时任务）
/// 随机运行 2-4 秒后以 50% 概率崩溃
async fn run_scheduler() -> Result<()> {
    let run_time = 2 + (rand_simple() % 3);
    println!("    [scheduler] 模拟运行 {}s...", run_time);
    tokio::time::sleep(Duration::from_secs(run_time)).await;

    if rand_simple() % 10 < 5 {
        anyhow::bail!("cron 任务执行超时")
    }
    Ok(())

/// 极简随机数（不引入 rand crate，用时间戳模拟）
fn rand_simple() -> u64 {
    use std::time::SystemTime;
    SystemTime::now()
        .duration_since(SystemTime::UNIX_EPOCH)
        .unwrap()
        .subsec_nanos() as u64
}

// ============================================================
// 4. main：多组件并发管理 + 优雅关闭
// 对应 ZeroClaw: src/daemon/mod.rs → pub async fn run()
// ============================================================

#[tokio::main]
async fn main() -> Result<()> {
    println!("╔══════════════════════════════════════════╗");
    println!("║   组件监控 Demo（Ctrl+C 优雅退出）       ║");
    println!("╚══════════════════════════════════════════╝\n");

    let health = HealthRegistry::new();

    // 启动所有组件监控器
    let mut handles: Vec<JoinHandle<()>> = vec![];

    // Gateway：初始退避 2s，最大退避 30s
    handles.push(spawn_component_supervisor(
        "gateway",
        2,
        30,
        health.clone(),
        || async { run_gateway().await },
    ));

    // Channel：初始退避 2s，最大退避 60s
    handles.push(spawn_component_supervisor(
        "channel",
        2,
        60,
        health.clone(),
        || async { run_channel().await },
    ));

    // Scheduler：初始退避 3s，最大退避 30s
    handles.push(spawn_component_supervisor(
        "scheduler",
        3,
        30,
        health.clone(),
        || async { run_scheduler().await },
    ));

    // 定期打印健康状态（每 5 秒）
    let health_for_print = health.clone();
    let status_handle = tokio::spawn(async move {
        let mut interval = tokio::time::interval(Duration::from_secs(5));
        loop {
            interval.tick().await;
            health_for_print.print_status();
        }
    });

    // 等待 Ctrl+C 信号（优雅退出的关键）
    println!("📌 按 Ctrl+C 停止所有组件...\n");
    tokio::signal::ctrl_c().await?;

    println!("\n🛑 收到 Ctrl+C，正在关闭所有组件...");

    // 第一步：取消所有任务
    status_handle.abort();
    for handle in &handles {
        handle.abort(); // abort 在下一个 .await 点生效
    }

    // 第二步：等待所有任务完成清理
    for handle in handles {
        let _ = handle.await; // 忽略 JoinError（因为是我们主动 abort 的）
    }

    // 最终状态
    health.print_status();
    println!("✅ 所有组件已安全关闭");

    Ok(())
}
```

## 4. 泛型约束逐字解析

`spawn_component_supervisor` 的类型签名是理解 Rust 异步泛型的绝佳案例：

```rust
fn spawn_component_supervisor<F, Fut>(
    name: &'static str,
    initial_backoff_secs: u64,
    max_backoff_secs: u64,
    mut run_component: F,        // ← mut 因为 FnMut
) -> JoinHandle<()>
where
    F: FnMut() -> Fut + Send + 'static,
    //  │  │        │     │      │
    //  │  │        │     │      └── 'static：闭包不能借用局部变量
    //  │  │        │     │          （因为 spawn 的任务可能活很久）
    //  │  │        │     │
    //  │  │        │     └── Send：闭包可以发送到其他线程
    //  │  │        │        （Tokio 多线程运行时会跨线程调度）
    //  │  │        │
    //  │  │        └── Fut：每次调用闭包返回一个新的 Future
    //  │  │
    //  │  └── FnMut：闭包可以修改捕获的变量
    //  │      为什么不是 Fn？因为 loop 中每次调用可能需要
    //  │      修改内部状态（比如重新创建连接对象）
    //  │
    //  └── F：闭包类型（编译器自动推导）
    //
    Fut: Future<Output = Result<()>> + Send + 'static,
    //   │       │                     │      │
    //   │       │                     │      └── Future 也必须 'static
    //   │       │                     │
    //   │       │                     └── Future 可以跨线程
    //   │       │
    //   │       └── 异步操作完成后返回 Result<()>
    //   │
    //   └── Fut 是一个 Future 类型
{
    tokio::spawn(async move { ... })
}
```

### 为什么是 FnMut 而不是 Fn？

```rust
// ─── 场景：闭包内部需要修改状态 ───

// FnMut 允许这样的代码：
let mut retry_count = 0;
spawn_component_supervisor("test", 2, 60, health, move || {
    retry_count += 1;   // ← 修改捕获的变量！需要 FnMut
    let count = retry_count;
    async move {
        println!("第 {} 次尝试", count);
        run_something().await
    }
});

// 如果用 Fn，上面的 `retry_count += 1` 就编译不过
// 因为 Fn 要求闭包不修改捕获的变量
```

### ZeroClaw 中的实际调用

```rust
// ZeroClaw 源码中的调用方式：
handles.push(spawn_component_supervisor(
    "gateway",
    2,
    60,
    move || {                          // move：把 cfg 移进闭包
        let cfg = gateway_cfg.clone(); // clone 在闭包内部
        async move {                   // async move：把 clone 的 cfg 移进 Future
            crate::gateway::run_gateway(&host, port, cfg).await
        }
    },
));

// 为什么要 clone 在闭包内部？
// 因为 FnMut 会被调用多次（每次重启调一次）
// 如果在闭包外 clone，第一次调用后 cfg 就被 move 走了
// 下次调用就没有 cfg 可用了
```

### TypeScript 完全不需要这些约束

```typescript
// TypeScript 版本：不需要任何约束标注
function spawnComponentSupervisor(
  name: string,
  initialBackoff: number,
  maxBackoff: number,
  runComponent: () => Promise<void>  // 就这么简单
) {
  // JS 是单线程的，不需要 Send
  // JS 有 GC，不需要 'static
  // JS 闭包默认可变，不需要 FnMut
}
```

**这就是 Rust "零成本抽象" 的代价**：你需要明确告诉编译器数据的线程安全性和生命周期。但换来的是编译时保证不会有数据竞争和悬垂引用。

## 5. 与前端的对比

### 对比1：PM2 进程管理器

```
PM2（Node.js）                    spawn_component_supervisor（Rust/Tokio）
─────────────────                 ──────────────────────────────────────
pm2 start app.js                  spawn_component_supervisor("app", ...)
  --max-restarts 10                 loop { run_component().await }
  --restart-delay 3000              tokio::time::sleep(backoff).await
  --exp-backoff-restart-delay       backoff = backoff * 2
                                    backoff = min(backoff, max_backoff)
pm2 stop all                      handle.abort()
pm2 status                        health.print_status()
```

**关键区别**：
- PM2 是进程级管理，每个 app 是独立进程
- Tokio 是任务级管理，所有组件在同一进程内的不同异步任务中
- Tokio 版本更轻量——没有进程间通信开销

### 对比2：WebSocket reconnect with exponential backoff

这是前端最常见的指数退避场景：

```typescript
// ─── 前端 WebSocket 重连（指数退避）───
class WebSocketClient {
  private backoff = 1000;          // 初始 1s
  private maxBackoff = 30000;      // 最大 30s

  connect() {
    const ws = new WebSocket('wss://api.example.com');

    ws.onopen = () => {
      console.log('✅ 连接成功');
      this.backoff = 1000;         // 重置退避
    };

    ws.onclose = () => {
      console.log(`❌ 断开，${this.backoff/1000}s 后重连`);
      setTimeout(() => this.connect(), this.backoff);
      this.backoff = Math.min(this.backoff * 2, this.maxBackoff);
    };
  }
}
```

```rust
// ─── Rust 等价实现 ───
// ZeroClaw 的 spawn_component_supervisor 本质上就是这个模式的通用化版本！
spawn_component_supervisor("websocket", 1, 30, health, move || {
    async move {
        let ws = connect_websocket("wss://api.example.com").await?;
        // 连接成功 → 持续监听
        loop {
            let msg = ws.recv().await?;  // 断开会返回 Err
            handle_message(msg).await?;
        }
        // 如果执行到这里（连接断开） → 返回 Err → 触发重启 + 退避
    }
});
```

**模式完全一样**——只是 Rust 版本把它抽象成了通用函数，适用于任何异步组件。

### 对比3：React Query 的 retry 机制

```typescript
// React Query 也有指数退避！
const { data } = useQuery({
  queryKey: ['todos'],
  queryFn: fetchTodos,
  retry: 3,                    // 最多重试 3 次
  retryDelay: (attempt) =>     // 指数退避
    Math.min(1000 * 2 ** attempt, 30000),
});
```

```rust
// Rust spawn_component_supervisor 的退避逻辑完全一样：
backoff = backoff.saturating_mul(2).min(max_backoff);
//        └──── 2 ** attempt ────┘  └── Math.min ──┘
```

## 6. 关键模式总结

### 指数退避 Cheat Sheet

```rust
// 任何需要重试的场景都可以用这个模式：

let mut backoff: u64 = initial;   // 初始等待（秒）
let max_backoff: u64 = max;       // 最大等待（秒）

loop {
    match do_something().await {
        Ok(_) => {
            backoff = initial;     // 成功 → 重置
        }
        Err(e) => {
            log_error(e);
        }
    }
    sleep(Duration::from_secs(backoff)).await;  // 等待
    backoff = backoff.saturating_mul(2)         // × 2
                     .min(max_backoff);         // 但不超过上限
}
```

### spawn_component_supervisor 的设计精髓

| 特性 | 实现方式 | 为什么需要 |
|------|---------|-----------|
| 自动重启 | `loop { run().await }` | 长运行组件不应该停 |
| 指数退避 | `backoff *= 2; min(max)` | 避免崩溃风暴 |
| 健康追踪 | `Arc<Mutex<HashMap>>` | 随时可查组件状态 |
| 错误隔离 | `tokio::spawn` | 一个崩了不影响其他 |
| 优雅关闭 | `abort() + await` | 干净退出不留孤儿 |
| 通用性 | 泛型 `F: FnMut() -> Fut` | 适用于任何异步组件 |

### 一句话总结

> **`spawn_component_supervisor` = tokio::spawn（隔离）+ loop（自动重启）+ 指数退避（防风暴）+ 健康追踪（可观测）**——这个模式在任何需要长运行异步组件的项目中都能直接复用，就像前端的 WebSocket 重连 + PM2 进程管理的合体。
